### Вопрос 1. Чем отличается режим автодополнения (inline suggestions) GitHub Copilot от режима чата (Copilot Chat)?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [GitHub Docs — Responsible use of Copilot inline suggestions](https://docs.github.com/en/copilot/responsible-use/copilot-code-completion)

Режим inline suggestions — это синхронный режим «призрачного текста» (ghost text): система анализирует код вокруг текущей позиции курсора, историю правок и фрагменты кода из открытых вкладок редактора, отправляет промпт в языковую модель и отображает встроенное предложение прямо в редакторе. Предложение применяется только при явном принятии пользователем (например, клавишей Tab). Copilot Chat — это диалоговый интерфейс: разработчик задаёт вопросы о коде, просит объяснений, генерирует тесты или ищет баги, обмениваясь сообщениями с моделью. GitHub Docs фокусируется на inline suggestions и отмечает, что контекст формируется из «кода вокруг текущей работы вместе с контекстной информацией (например, фрагментами кода из открытых вкладок в редакторе)».

#### Ответ от Claude

Два режима GitHub Copilot решают разные задачи:

| Характеристика | Inline suggestions (autocomplete) | Copilot Chat |
|---|---|---|
| Интерфейс | Ghost text прямо в редакторе | Отдельная панель чата |
| Взаимодействие | Пассивное — Copilot предлагает, разработчик принимает/отклоняет | Активное — разработчик задаёт вопросы |
| Применение | Завершение кода по мере набора | Объяснение кода, генерация тестов, поиск багов, рефакторинг |
| Синхронность | Синхронный, встроен в поток набора | Асинхронный диалог |
| Контекст | Код вокруг курсора + открытые вкладки + история правок | Явно указанные файлы + диалог |

**Inline suggestions** лучше подходят для написания нового кода «на лету»: предложение появляется автоматически и не прерывает рабочий процесс.

**Copilot Chat** лучше подходит для понимания существующего кода, генерации тестов, рефакторинга и исследовательских задач — когда нужен развёрнутый ответ, а не просто следующая строка.

### Вопрос 2. Какой контекст использует GitHub Copilot при генерации inline-подсказок? Почему важно держать нужные файлы открытыми?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [GitHub Docs — Responsible use of Copilot inline suggestions](https://docs.github.com/en/copilot/responsible-use/copilot-code-completion)

Согласно документации GitHub, при генерации inline suggestions Copilot анализирует «код вокруг текущей работы вместе с контекстной информацией (например, фрагментами кода из открытых вкладок в редакторе)», а также «историю правок, окружающий код и позицию курсора». Это означает, что открытые вкладки редактора напрямую влияют на качество предложений: без нужных файлов модель опирается только на текущий файл и знания, полученные при обучении.

#### Ответ от Claude

Copilot формирует контекстный промпт из нескольких источников:

1. **Окружающий код** — блок кода непосредственно вокруг позиции курсора (до и после).
2. **Открытые вкладки редактора** — фрагменты кода из файлов, открытых в данный момент в IDE. Это основной способ, которым Copilot узнаёт о структуре проекта, типах, интерфейсах и соглашениях.
3. **История правок** — недавние изменения в текущем файле.
4. **Позиция курсора** — точное место ввода задаёт намерение.

**Почему открытые файлы важны:** GitHub Copilot не индексирует весь проект — он видит только то, что открыто в редакторе. Если разработчик реализует метод, использующий типы из соседнего файла, но тот файл закрыт, Copilot не знает о существующих типах и может предложить несовместимый код. Чем точнее набор открытых файлов соответствует задаче (модели данных, интерфейсы, смежные модули), тем релевантнее автодополнение.

### Вопрос 3. Что такое Next Edit Suggestions (NES) в GitHub Copilot и чем они отличаются от классического autocomplete?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [VS Code Blog — Next Edit Suggestions](https://code.visualstudio.com/blogs/2025/02/12/next-edit-suggestions)

NES — это функция, которая предлагает правки существующего кода по всему файлу или проекту, в дополнение к классическому autocomplete. Как объясняет блог VS Code, «большая часть работы по написанию кода включает редактирование существующего кода в той же мере, что и написание новых строк». Классический autocomplete (ghost text) хорошо справляется с генерацией новых строк кода. NES меняет фокус: он определяет связанные изменения, которые нужно внести в других местах кодовой базы. Ключевая идея: «лучший промпт — тот, который не нужно писать». NES проактивно предвидит каскадные правки до того, как разработчик вручную ищет связанные места. Принять предложение можно клавишей Tab.

#### Ответ от Claude

**Классический autocomplete** предлагает продолжение кода в позиции курсора — он хорошо справляется с написанием нового кода «с нуля».

**Next Edit Suggestions (NES)** решает другую задачу — отслеживает историю недавних правок и предсказывает:
- **что** нужно изменить следующим;
- **где** в файле это изменение должно произойти.

Практические сценарии, где NES особенно полезен:

| Сценарий | Поведение NES |
|---|---|
| Переименование переменной | Предлагает обновить все связанные строки в файле |
| Изменение сигнатуры метода | Указывает на места вызовов, требующие обновления |
| Регистрация команды VS Code | При добавлении в `extension.ts` предлагает синхронно обновить `package.json` |
| Исправление опечатки | Автоматически обнаруживает и исправляет аналогичные ошибки |

Это делает NES особенно ценным при рефакторинге: вместо ручного поиска по файлу по Ctrl+F разработчик просто нажимает Tab для каждого следующего изменения.

### Вопрос 4. Что такое agent mode в GitHub Copilot и чем он принципиально отличается от chat mode?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [GitHub Blog — Agent mode 101](https://github.blog/ai-and-ml/github-copilot/agent-mode-101-all-about-github-copilots-powerful-mode/)

По данным блога GitHub, agent mode — это «автономный и агентный коллаборатор в реальном времени, выполняющий многошаговые задачи по написанию кода на основе промптов на естественном языке». В отличие от chat mode (режимов ask/edit), которые реактивны и отвечают на конкретные запросы, agent mode проактивен: он автономно выполняет многошаговые рабочие процессы, обнаруживает ошибки и самостоятельно их исправляет. Разработчик «определяет результат, а агент определяет наилучший подход — запрашивая обратную связь по мере необходимости, тестируя собственные решения и улучшая свою работу в реальном времени». Agent mode может анализировать всю кодовую базу, планировать и выполнять сложные многошаговые решения, запускать команды и тесты, обнаруживать синтаксические ошибки, сбои сборки и провалы тестов — и исправлять их автоматически.

#### Ответ от Claude

Фундаментальное различие — уровень автономности:

| Аспект | Chat mode (ask/edit) | Agent mode |
|---|---|---|
| Модель взаимодействия | Реактивная: один запрос — один ответ | Проактивная: цикл планирование → изменение → тест → исправление |
| Управление | Разработчик направляет каждый шаг | Разработчик определяет цель, агент сам находит путь |
| Файлы | Работает с указанными файлами | Самостоятельно определяет, какие файлы нужно изменить |
| Терминал | Не выполняет команды | Предлагает и выполняет команды терминала |
| Итерации | Одна итерация за запрос | Автоматически итерирует до завершения или тупика |
| Исправление ошибок | Вручную, по запросу | Автоматически — синтаксические ошибки, провалы тестов, ошибки сборки |

**Когда использовать agent mode:** реализация фич, затрагивающих несколько файлов; рефакторинг; добавление тестов к существующему коду; автоматизация повторяющихся задач обслуживания.

**Важно:** агент работает синхронно в локальной IDE и требует присутствия разработчика — это отличает его от асинхронного Copilot Coding Agent в GitHub Actions.

### Вопрос 5. Как работает файл `.github/copilot-instructions.md` и зачем его использовать?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [GitHub Docs — Adding repository custom instructions for GitHub Copilot](https://docs.github.com/copilot/customizing-copilot/adding-custom-instructions-for-github-copilot)

Файл `.github/copilot-instructions.md` предоставляет Copilot инструкции, специфичные для данного репозитория, применимые ко всем запросам в этом репозитории. Содержимое должно включать: краткое описание репозитория и высокоуровневую информацию, процедуры сборки и валидации с проверенными командами, описание структуры проекта и архитектуры, расположение ключевых файлов и конфигурации. Ограничения: инструкции должны быть не длиннее 2 страниц и не должны быть задачеспецифичными. Дополнительно можно создавать файлы `.instructions.md` в `.github/instructions/` с frontmatter-полем `applyTo`, принимающим glob-паттерны (например, `"**/*.ts,**/*.tsx"`), для прицельного применения инструкций к конкретным путям файлов.

#### Ответ от Claude

`.github/copilot-instructions.md` — это механизм хранения постоянного контекста проекта в репозитории. Файл автоматически добавляется к каждому запросу к Copilot в данном репо, избавляя разработчика от необходимости объяснять контекст заново в каждом чате.

**Что стоит включать:**
- Краткое описание назначения репозитория
- Технологический стек и версии
- Команды сборки, запуска тестов и линтинга
- Структуру директорий и расположение ключевых файлов
- Архитектурные решения и соглашения

**Ограничения:**
- Не более 2 страниц текста
- Только универсальный контекст, применимый ко всем запросам — не задачеспецифичные инструкции
- Доступен в GitHub Copilot Chat (IDE и GitHub.com)

**Path-scoped инструкции:** для тонкой настройки под конкретные части кодовой базы создаются файлы `.github/instructions/<name>.instructions.md` с YAML frontmatter:

```yaml
---
applyTo: "src/**/*.ts,src/**/*.tsx"
---
```

Это позволяет задавать разные соглашения для фронтенда, бэкенда, тестов — без засорения общего файла инструкций.

### Вопрос 6. Что такое Copilot Spaces и как они помогают улучшить качество ответов Copilot на уровне команды?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [GitHub Changelog — Introducing Copilot Spaces](https://github.blog/changelog/2025-05-29-introducing-copilot-spaces-a-new-way-to-work-with-code-and-context/)

Copilot Spaces — это организационные хабы для централизации контекста проекта. Они позволяют командам «фокусировать знания Copilot на подобранном наборе конкретного кода, документов, заметок и другого». Пространство может содержать: исходный код из репозиториев, документацию и спецификации, кастомные инструкции, заметки и справочные материалы. Ключевое преимущество: «файлы и репозитории, добавленные в пространство, всегда актуализируются по мере изменений» — Copilot поддерживает актуальные знания без ручного обновления. Пространства можно делиться внутри GitHub-организации, что ускоряет онбординг новых разработчиков и делает Copilot полезным даже для тех, кто незнаком с определённой частью кодовой базы.

#### Ответ от Claude

Copilot Spaces решают проблему «глухого» контекста: без пространства Copilot знает только то, что разработчик открыл в редакторе. Spaces позволяют создать «область знаний» для конкретной задачи или продукта.

**Что можно добавить в Space:**
- Репозитории (синхронизируются автоматически при изменениях)
- Документацию и технические спецификации
- Инструкции для команды
- Заметки и справочные материалы

**Преимущества для команды:**
- Copilot становится «экспертом» по конкретному продукту или домену, а не общим ассистентом
- Новые разработчики получают контекстуально точные ответы с первого дня
- Знания команды централизованы и актуальны — нет риска устаревшего контекста
- Пространства доступны всей организации через GitHub

**Сравнение с `.github/copilot-instructions.md`:** instructions.md ограничен одним репозиторием и только текстовыми инструкциями. Spaces агрегируют несколько репозиториев, документацию и код в единую точку знаний.

### Вопрос 7. Что такое Cursor и чем его подход к AI-интеграции отличается от подхода GitHub Copilot?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Cursor — официальный сайт](https://cursor.com/)

Cursor позиционируется как «coding agent для создания амбициозного ПО». Платформа включает несколько уровней автономности: Tab completion, Cmd+K для точечных правок и полностью автономный агентный режим. Cursor поддерживает множество передовых моделей: OpenAI, Anthropic (Claude), Google Gemini, xAI, а также собственные проприетарные модели (Composer). Агенты работают в облачной инфраструктуре и могут запускаться на собственных компьютерах Cursor для сборки, тестирования и демонстрации фич. Интеграция охватывает терминал, Slack и ревью GitHub PR — не только редактор.

#### Ответ от Claude

Принципиальное различие: **GitHub Copilot — расширение для редактора**, тогда как **Cursor — полноценная IDE**, созданная как форк VS Code, где AI интегрирован на всех уровнях.

| Аспект | GitHub Copilot | Cursor |
|---|---|---|
| Архитектура | Плагин для VS Code, JetBrains, Vim и др. | IDE (форк VS Code), AI встроен в ядро |
| Модели | OpenAI / Anthropic (через Copilot API) | Мультимодельность: OpenAI, Anthropic, Gemini, xAI + собственные |
| Индексация проекта | Открытые вкладки | Семантическая индексация всего проекта (tree-sitter + векторная БД) |
| Autocomplete модель | Codex/GPT-based | Собственная Tab-модель, обученная на предсказание следующей правки |
| Агентный режим | В рамках IDE через agent mode | В IDE + Background/Cloud Agents в удалённых контейнерах |
| Переключение редактора | Нет (работает где установлен) | Да — нужно переключиться на Cursor IDE |

Cursor делает ставку на глубокое понимание кодовой базы целиком, а не просто на помощь в момент набора — это и определяет принципиальный архитектурный выбор.

### Вопрос 8. Как работают правила (Rules) в Cursor и чем `.mdc`-файлы отличаются от устаревшего `.cursorrules`?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Cursor Docs — Rules](https://cursor.com/docs/context/rules)

Правила (Rules) предоставляют системные инструкции агенту Cursor, объединяя промпты и скрипты. Поскольку «языковые модели не сохраняют память между завершениями», правила обеспечивают постоянный контекст на уровне промпта. Современный формат — файлы `.mdc` в директории `.cursor/rules/`, версионируемые вместе с проектом. Обычные `.md`-файлы в `.cursor/rules/` игнорируются — требуется расширение `.mdc`. Frontmatter-поля управляют применением правила: `alwaysApply` (булев, применять в каждой сессии), `description` (агент оценивает релевантность), `globs` (glob-паттерны для scoping по файлам). Старый файл `.cursorrules` в корне проекта поддерживается для обратной совместимости, но не рекомендуется.

#### Ответ от Claude

Rules в Cursor — аналог `CLAUDE.md` в Claude Code или `copilot-instructions.md` в GitHub Copilot, но с более гибкой системой применения.

**Типы применения правил (определяются frontmatter):**

| Тип | Поведение | Frontmatter |
|---|---|---|
| Always Apply | Применяется в каждой сессии | `alwaysApply: true` |
| Intelligent Apply | Агент сам решает по `description` | Только `description` |
| File-scoped | Активируется при работе с файлами по glob | `globs: "src/**/*.tsx"` |
| Manual | Только при явном `@rule-name` в чате | Без frontmatter активации |

**Пример .mdc-файла:**
```yaml
---
globs: "src/api/**/*.ts"
description: "Правила для API-эндпоинтов"
---
Все эндпоинты должны включать валидацию входных данных.
Используй стандартный формат ответа об ошибке.
```

**`.mdc` vs `.cursorrules`:**
- `.cursorrules` — единый файл в корне проекта, без возможности scoping; устарел
- `.mdc` в `.cursor/rules/` — модульный подход, версионируется, поддерживает path-scoping, разбивается по темам

### Вопрос 9. Как Cursor Tab отличается от GitHub Copilot autocomplete при работе с несколькими строками?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [APIdog — Cursor Tab Completion vs GitHub Copilot](https://apidog.com/blog/cursor-tab/)

Ключевые отличия Cursor Tab от GitHub Copilot согласно статье: Cursor Tab может «изменять, добавлять или удалять несколько строк вокруг курсора» — тогда как Copilot преимущественно добавляет код в позицию курсора. При предложении изменений в нескольких блоках кода Cursor Tab отображает diff-интерфейс, наглядно показывающий, что именно изменится. Инструмент использует «кастомную AI-модель, которая отслеживает историю кода, недавние правки и даже ошибки линтера», что обеспечивает более умные предложения. После принятия предложения редактор может выделить следующее логическое место правки; повторное нажатие Tab позволяет «перейти» туда и применить следующее предложение — это создаёт рабочий процесс «tab-tab-tab» для многоступенчатого рефакторинга.

#### Ответ от Claude

**GitHub Copilot autocomplete:** вставляет предложение в точку курсора. Классический ghost text — хорошо для написания нового кода.

**Cursor Tab:** переосмысляет автодополнение как инструмент редактирования существующего кода:

| Возможность | GitHub Copilot | Cursor Tab |
|---|---|---|
| Новые строки | Да | Да |
| Изменение существующих строк | Ограниченно | Да — редактирует, добавляет и удаляет |
| Удаление строк | Нет | Да |
| Визуализация изменений | Ghost text | Diff-интерфейс при мультистрочных правках |
| История правок | Нет | Да — учитывает недавние изменения |
| Ошибки линтера | Нет | Да — использует как сигнал |
| Cursor jumps (следующая правка) | Нет (NES — отдельная фича) | Да — после Tab предсказывает следующее место |

**Cursor Prediction (cursor jumps):** после принятия предложения Cursor подсвечивает следующее логически связанное место в файле, которое нужно изменить. Повторное нажатие Tab «прыгает» туда. Это создаёт «волну» каскадных изменений без ручного поиска по файлу.

Агентный режим Cursor работает через harness из трёх компонентов: инструкции (системные промпты и правила), инструменты (редактирование файлов, поиск по кодовой базе, выполнение команд терминала), модель (выбранная AI-модель). Plan Mode (Shift+Tab) позволяет переключиться в режим планирования перед генерацией кода: агент исследует файлы, задаёт уточняющие вопросы и создаёт план реализации для утверждения. - Итерирует в цикле: понять → изменить → проверить → исправить
- Требует присутствия разработчика — синхронное выполнение
- Plan Mode (Shift+Tab): сначала строит план и получает одобрение, затем реализует

**Background Agents (Cloud Agents):**
- Работают асинхронно в изолированных облачных контейнерах (Ubuntu VM)
- Клонируют репозиторий, выполняют задачу независимо
- Разработчик запускает из веб-интерфейса или Slack и продолжает другую работу
- По завершении создают черновой pull request и уведомляют разработчика
- Идеальны для параллельных задач: «пока я работаю над X, агент фиксит Y»

| Аспект | Локальный агент | Background Agent |
|---|---|---|
| Выполнение | Синхронное, в IDE | Асинхронное, в облаке |
| Присутствие разработчика | Требуется | Не требуется |
| Результат | Изменения в рабочем дереве | Черновой pull request |
| Запуск | Из чата в IDE | Из веба или Slack |

### Вопрос 12. Как CLAUDE.md и auto memory обеспечивают персистентный контекст между сессиями Claude Code?

| Характеристика | CLAUDE.md | Auto memory |
|---|---|---|

| Содержимое | Инструкции и правила | Полученные знания и паттерны |

### Вопрос 15. Почему Cursor использует семантическую индексацию всего проекта, а не просто открытые файлы, и как это влияет на качество ответов агента?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Collabnix — Cursor AI Deep Dive Technical Architecture](https://collabnix.com/cursor-ai-deep-dive-technical-architecture-advanced-features-best-practices-2025/)

Cursor использует многослойный процесс индексации. **Парсинг:** Cursor применяет tree-sitter для языко-независимого парсинга, разбивая код на структурированные компоненты, а не обрабатывая файлы как обычный текст. **Семантическая нарезка:** файлы разбиваются на смысловые единицы — функции, классы, модули — с созданием чанков по 100–500 токенов с перекрывающимися окнами для сохранения контекстных связей. **Генерация эмбеддингов:** каждый чанк преобразуется в 1536-мерный вектор через проприетарную модель эмбеддингов. **Хранение и поиск:** чанки индексируются в векторной базе данных для similarity search: запрос пользователя преобразуется в вектор, и возвращаются top-K наиболее релевантных результатов. В отличие от Copilot, который ограничен открытыми вкладками, Cursor поддерживает «граф знаний» структуры проекта, обеспечивая контекстно-зависимые дополнения для дальних связей.

#### Ответ от Claude

Ограничение подхода «только открытые вкладки» (Copilot): агент не знает о коде, который не открыт в редакторе. Для крупных проектов это означает «слепые пятна» при ответах о взаимодействии компонентов.

**Как работает семантическая индексация Cursor:**

```
Весь код проекта
        ↓
tree-sitter (парсинг в AST, языко-независимо)
        ↓
Семантические чанки (функции, классы, модули, 100-500 токенов)
        ↓
Embedding модель (1536-мерные векторы)
        ↓
Векторная БД (индекс схожести)
        ↓
Similarity search при запросе (top-K релевантных чанков)
        ↓
Приоритизированный контекст для модели
```

**Практическое влияние на качество ответов:**

| Вопрос к агенту | Copilot (только вкладки) | Cursor (индекс проекта) |
|---|---|---|
| «Как реализована аутентификация?» | Нужно открыть нужные файлы вручную | Автоматически находит релевантные компоненты |
| «Где используется этот метод?» | Ограничено открытыми файлами | Поиск по всей кодовой базе |
| «Можно ли повторно использовать X?» | Может не знать о существующем X | Находит похожие реализации |

Индексация обновляется инкрементально при изменении файлов, поэтому не требует ручного пересчёта.

### Вопрос 16. Какова архитектурная разница в подаче контекста проекта между Claude Code (CLAUDE.md), Cursor (`.cursor/rules`) и GitHub Copilot (`.github/copilot-instructions.md`)?

#### Оригинальный ответ из интернета

Документация Claude Code описывает многоуровневую систему контекста. #### Ответ от Claude

Все три инструмента решают одну задачу — дать модели постоянный контекст о проекте — но архитектурно это реализовано по-разному:

**GitHub Copilot:**
- `.github/copilot-instructions.md` — единый файл для всего репозитория
- `.github/instructions/*.instructions.md` + `applyTo: "glob"` — path-scoping

- `.cursor/rules/*.mdc` — модульные правила с YAML frontmatter

**Claude Code:**
- Иерархия CLAUDE.md (4 уровня: org / user / project / local)
- **Auto memory** — уникальная функция: Claude сам накапливает знания между сессиями
- `.claude/rules/*.md` с `paths` frontmatter — path-scoped правила
- До 200 строк на CLAUDE.md-файл; path-scoped правила экономят контекст

| Возможность | Copilot | Cursor | Claude Code |
|---|---|---|---|
| Иерархия уровней | 2 (repo / path) | 2 (project / path) | 4 (org / user / project / local) |
| Path-scoping | Да (`applyTo`) | Да (`globs`) | Да (`paths`) |
| Автонакопление | Нет | Нет | Да (auto memory) |
| Версионирование | Да (в git) | Да (в git) | CLAUDE.md — да; auto memory — нет (локальный) |
| Личные инструкции | Нет | Нет | Да (CLAUDE.local.md) |

Ключевое уникальное преимущество Claude Code — auto memory: агент сам пополняет базу знаний по мере работы, без ручного ввода.

### Вопрос 17. Какова главная ответственность разработчика при принятии предложений GitHub Copilot?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [GitHub Docs — Responsible use of GitHub Copilot code review](https://docs.github.com/en/copilot/responsible-use/code-review)

Документация GitHub явно указывает: разработчики «несут ответственность за проверку и валидацию ответов, генерируемых Copilot», и должны «всегда тщательно проверять и тестировать код, сгенерированный Copilot». Инструмент может «выделять проблемы в проверяемом коде, которые не существуют или основаны на неправильном понимании кода» — комментарии следует «тщательно обдумывать перед принятием действий». Copilot «может не идентифицировать все проблемы в коде, особенно когда изменения масштабны или сложны», поэтому его ревью «следует дополнять тщательным ручным ревью». Предложения кода «могут фактически не быть семантически или синтаксически корректными, или могут неверно решать выявленную проблему».

#### Ответ от Claude

GitHub прямо указывает: Copilot — вспомогательный инструмент, а не замена суждению разработчика. Ответственность остаётся за человеком.

**Что должен делать разработчик:**

1. **Проверять корректность** — предложения могут быть синтаксически неверными или семантически ошибочными
2. **Тестировать код** — сгенерированный код нужно запускать и проверять, как любой другой
3. **Оценивать безопасность** — Copilot не гарантирует отсутствие уязвимостей (см. Вопрос 18)
4. **Критически читать комментарии ревью** — Copilot может указывать на несуществующие проблемы (галлюцинации)
5. **Дополнять ручным ревью** — особенно при масштабных или сложных изменениях

**Чего нельзя делать:**
- Принимать предложения без понимания того, что они делают
- Считать Copilot ревью полноценной заменой ревью коллег
- Доверять Copilot определение правильности собственных предложений

**Практическое правило:** относитесь к коду от Copilot так же, как к коду нового стажёра: полезный вклад, но требует проверки перед мержем.

### Вопрос 18. Какие категории уязвимостей чаще всего встречаются в AI-сгенерированном коде и почему они возникают?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Vibe Coder Blog — A Practical Code Review Checklist for AI-Generated Code](https://blog.vibecoder.me/code-review-checklist-ai-generated-code)

По данным Veracode 2025, «45% AI-сгенерированного кода содержит хотя бы одну уязвимость безопасности». При применении структурированного процесса ревью этот показатель снижается до менее чем 10% — разница в 35 процентных пунктов. Наиболее распространённые уязвимости: захардкоженные секреты (API-ключи, токены, пароли, строки подключения), injection-уязвимости (прямая интерполяция пользовательских данных в строки запросов вместо параметризации), XSS (некорректное экранирование при рендеринге ввода в DOM или шаблоны), отсутствие аутентификации на новых эндпоинтах, пропущенная авторизация на уровне ресурсов. Причины: «AI-модели допускают ошибки, потому что галлюцинируют API, игнорируют контекст за пределами текущего промпта и оптимизируют код, который выглядит правильным, а не является правильным». Модели обучены на миллионах репозиториев, содержащих небезопасные паттерны.

#### Ответ от Claude

**Статистика:** по данным Veracode (2025) 45% AI-сгенерированного кода содержит хотя бы одну уязвимость. Структурированный процесс ревью снижает этот показатель до менее 10%.

**Топ-5 уязвимостей в AI-коде:**

| Уязвимость | Пример | Почему возникает |
|---|---|---|
| Захардкоженные секреты | `api_key = "sk-abc123..."` прямо в коде | Модель видела такой паттерн в обучающих данных |
| SQL-инъекция | `f"SELECT * FROM users WHERE id={user_id}"` | Модель оптимизирует краткость, игнорирует параметризацию |
| XSS | `innerHTML = userInput` без санитизации | Фокус на функциональности, не на безопасности вывода |
| Пропущенная аутентификация | Новый эндпоинт без `@RequiresAuth` | Модель генерирует «минимальный рабочий» код |
| Пропущенная авторизация | Аутентификация есть, но нет проверки прав на ресурс | Тонкое различие, часто упускается |

**Системная причина:** языковые модели обучены оптимизировать код, который «выглядит правильным» и функционален. Безопасность — нефункциональное требование, которое не очевидно из большинства примеров обучающих данных. Модели также не понимают контекст: что данные пришли от пользователя, насколько критичен эндпоинт.

**Меры защиты:** чеклист безопасности при ревью AI-кода, SAST-инструменты (SonarQube, Checkmarx), dependency scanning, принцип least privilege при генерации кода.

### Вопрос 19. Что такое файл CLAUDE.md и зачем он нужен при работе с Claude Code?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [HumanLayer Blog — Writing a good CLAUDE.md](https://www.humanlayer.dev/blog/writing-a-good-claude-md)

`CLAUDE.md` — конфигурационный файл, служащий основным механизмом онбординга Claude в кодовую базу. Поскольку «LLM — это функции без состояния» и не имеют персистентной памяти между сессиями, этот файл автоматически включается в каждый разговор с Claude Code для предоставления необходимого контекста. Он описывает три аспекта: **ЧТО** (технический стек, структура проекта, организация кодовой базы), **ПОЧЕМУ** (назначение проекта и функция отдельных компонентов), **КАК** (рабочие процессы разработки, процедуры тестирования, методы верификации, команды сборки). Рекомендованный объём — менее 300 строк (HumanLayer держит свой под 60 строк). Стилистические руководства лучше выносить в линтеры, а не в CLAUDE.md. Задачеспецифичный контент нужно убирать в отдельные файлы документации (`agent_docs/`), а в CLAUDE.md давать только ссылки.

#### Ответ от Claude

`CLAUDE.md` — это «инструкция по эксплуатации» вашего проекта для Claude. Каждая сессия Claude Code стартует с пустым контекстным окном; CLAUDE.md — первое, что агент читает.

**Что включать:**

```
CLAUDE.md = ЧТО + ПОЧЕМУ + КАК

ЧТО:  технологический стек, структура директорий, ключевые зависимости
ПОЧЕМУ: назначение проекта, роль основных компонентов, архитектурные решения
КАК:  команды сборки/тестирования/запуска, рабочие процессы, соглашения
```

**Практические рекомендации:**
- Целевой объём — до 200–300 строк (документация Claude Code рекомендует под 200; HumanLayer держит под 60)
- Включать только то, что нужно в каждой сессии — не задачеспецифичный контент
- Стилистические правила → в линтеры, а не в CLAUDE.md
- Детальные гайды → в отдельные файлы (`docs/`, `agent_docs/`), с упоминанием в CLAUDE.md
- Когда Claude дважды делает одну и ту же ошибку — добавить правило в CLAUDE.md

**Иерархия:** проект может иметь несколько CLAUDE.md на разных уровнях (организация, пользователь, проект, поддиректории). Команда `/init` автоматически генерирует стартовый CLAUDE.md на основе анализа кодовой базы.

### Вопрос 20. Что должен содержать файл .github/copilot-instructions.md и какие у него ограничения?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [GitHub Docs — Adding repository custom instructions for GitHub Copilot](https://docs.github.com/copilot/customizing-copilot/adding-custom-instructions-for-github-copilot)

Согласно документации GitHub, файл `.github/copilot-instructions.md` должен содержать: краткое описание репозитория и высокоуровневую информацию, процедуры сборки и валидации с проверенными командами, описание структуры проекта и архитектуры, расположение ключевых файлов и конфигурации. Ограничения: инструкции не должны превышать 2 страниц, не должны быть задачеспецифичными — только универсальный контекст. Пробелы между инструкциями игнорируются. Дополнительно: файлы `.github/instructions/*.instructions.md` с frontmatter `applyTo` поддерживают несколько glob-паттернов через запятую; при желании можно добавить `excludeAgent`, чтобы исключить cloud agent или code review из использования конкретного файла инструкций.

#### Ответ от Claude

`.github/copilot-instructions.md` автоматически добавляется к каждому запросу к Copilot в репозитории. Это «постоянная» часть контекста, которую разработчику не нужно повторять в каждом чате.

**Рекомендуемое содержание:**

```markdown
## О проекте
Краткое описание: что делает проект, для кого, на каком стеке

## Технологии
- Java 21, Spring Boot 3.3, PostgreSQL 16
- Maven, JUnit 5, Testcontainers

## Структура
src/main/java/com/example/
  api/       - REST-контроллеры
  service/   - бизнес-логика
  domain/    - сущности JPA

## Команды
- Сборка: `mvn clean package`
- Тесты: `mvn test`
- Запуск: `java -jar target/app.jar`
```

**Ограничения:**
- Не более 2 страниц (примерно 500–700 слов)
- Только универсальный контекст — не задачеспецифичные инструкции
- Нельзя задавать разное поведение для разных частей кодовой базы (для этого — `.instructions.md`)

**Path-scoped инструкции `.github/instructions/*.instructions.md`:**
```yaml
---
applyTo: "src/api/**/*.java,src/api/**/*.kt"
---
Все эндпоинты должны использовать ResponseEntity<T>.
Документируй с помощью OpenAPI-аннотаций.
```

Поддерживается для Copilot cloud agent и code review на GitHub.com. В IDE поддержка path-scoped инструкций зависит от версии расширения.
