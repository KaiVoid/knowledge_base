### Вопрос 21. В чём разница между CLAUDE.md, AGENTS.md и copilot-instructions.md?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [DeployHQ Blog — CLAUDE.md, AGENTS.md & Copilot Instructions: Configure Every AI Coding Assistant](https://www.deployhq.com/blog/ai-coding-config-files-guide)

Все три файла решают одну фундаментальную проблему: «AI-модели не имеют постоянной памяти — каждая сессия начинается с чистого листа». Конфигурационные файлы позволяют задать постоянный контекст проекта раз и навсегда.

**CLAUDE.md** — конфигурационный файл для Claude Code (CLI-инструмент Anthropic). Обнаруживается в корне проекта, в директории `~/.claude/` и в поддиректориях для scoped-инструкций. Рекомендуемый лимит — около 300 строк (LLM надёжно следуют примерно 150–200 инструкциям). Особенность: Claude Code читает `AGENTS.md` как запасной вариант, если `CLAUDE.md` не найден в текущей директории.

**AGENTS.md** — наиболее широкий кросс-платформенный стандарт, изначально созданный для Codex CLI (OpenAI). Поддерживается: Codex CLI, Cursor, Claude Code (как fallback), Continue.dev, Aider, OpenHands. Использует иерархическую систему обнаружения — обходит директории от корня проекта до текущей. Поддерживает `AGENTS.override.md` для локальных переопределений (gitignored). По охвату — «ближайший аналог универсального стандарта», принятый в 60 000+ open-source проектов.

**copilot-instructions.md** — конфигурационный файл для GitHub Copilot (Chat, CLI и дополнение кода). Располагается в директории `.github/`; scoped-версии — в `.github/instructions/` с frontmatter glob-паттернами. Уникальная возможность: поддержка правил для конкретных типов файлов через `applyTo: "**/*.tsx"` — это позволяет задавать разные инструкции для разных расширений без их централизации. Рекомендуется использовать вместе с `AGENTS.md`, а не вместо него.

#### Ответ от Claude

Каждый AI-инструмент читает собственный конфигурационный файл, который задаёт постоянный контекст проекта (стек, соглашения, ограничения) без необходимости повторять его в каждой сессии.

| Файл | AI-инструмент | Расположение |
|------|--------------|--------------|
| `CLAUDE.md` | Claude Code (Anthropic) | Корень проекта, `~/.claude/`, поддиректории |
| `AGENTS.md` | Codex CLI, Cursor, Claude Code (fallback), Aider, Continue.dev | Иерархически от корня до текущей директории |
| `.github/copilot-instructions.md` | GitHub Copilot | `.github/` (scoped-версии в `.github/instructions/`) |
| `.cursorrules` | Cursor | Корень проекта |

Ключевые различия:

- **CLAUDE.md** специфичен для Claude Code; при его отсутствии Claude Code автоматически обращается к `AGENTS.md`.
- **AGENTS.md** — де-факто кросс-платформенный стандарт; единственный файл, который поддерживают сразу несколько несвязанных инструментов.
- **copilot-instructions.md** уникален поддержкой glob-паттернов (`applyTo`), позволяющих задавать разные правила для разных типов файлов (например, строже для `.tsx`, либеральнее для тестов).

Практическая рекомендация: в монорепозитории с несколькими AI-инструментами заводят `AGENTS.md` как основу и `CLAUDE.md` / `copilot-instructions.md` для инструментно-специфичных правил.

### Вопрос 22. Почему при ревью AI-сгенерированного кода особое внимание нужно уделять логике работы с API и обработке ошибок?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Vibe Coder Blog — A Practical Code Review Checklist for AI-Generated Code](https://blog.vibecoder.me/code-review-checklist-ai-generated-code)

AI-модели имеют дату отсечения обучения и описывают API такими, какими они были на момент обучения, а API меняются. Статья называет это «наиболее частым источником логических ошибок в AI-сгенерированном коде» и рекомендует открывать актуальную документацию и сверять каждый вызов с реальностью.

Проблема с обработкой ошибок: AI генерирует `try-catch` блоки, которые обрабатывают ожидаемые типы ошибок и молча «поглощают» всё остальное (`catch (e) {}`). Это создаёт опасные сценарии, при которых «перехваченные и проигнорированные ошибки означают, что задачи прекращают выполнение без какого-либо предупреждения» — особенно критично в фоновых процессах, где сбои остаются незамеченными.

Суть проблемы: AI оптимизирован для кода, который *выглядит* корректным при тестировании «счастливого пути», но молча падает при неожиданных условиях, которые не были явно указаны в промпте.

#### Ответ от Claude

Две основные причины, по которым именно API-логика и обработка ошибок требуют повышенного внимания:

**1. Устаревшие API-вызовы**

Обучающие данные LLM имеют дату отсечения. Внешние библиотеки, REST API и SDK регулярно меняют сигнатуры методов, параметры аутентификации, URL эндпоинтов и форматы ответов. AI уверенно генерирует вызовы, которые были актуальны год-два назад. Ревьюер должен открыть официальную документацию и сверить каждый внешний вызов с текущей версией.

**2. Неполная обработка ошибок**

AI хорошо обрабатывает «счастливый путь», но систематически пропускает:
- сетевые тайм-ауты и повторные попытки;
- HTTP 4xx/5xx с различающейся семантикой;
- частичные ошибки (batch-запросы, где часть элементов не обработана);
- ошибки аутентификации и истечения токенов;
- конкурентные сбои в асинхронном коде.

Результат — молчаливые сбои, которые не порождают ни исключений, ни записей в логах. В фоновых задачах (очереди, планировщики, стриминг-пайплайны) это означает незамеченную потерю данных.

Практическое правило: каждый `catch (e) {}` или `except: pass` в AI-коде — это красный флаг, требующий явной обработки или, как минимум, логирования с контекстом.

### Вопрос 23. Как организовать тестирование AI-сгенерированного кода, чтобы избежать «слепых пятен» при использовании AI для написания тестов?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ContextQA — How to Test AI Generated Code: A QA Checklist for 2026](https://contextqa.com/blog/what-is-ai-generated-code-testing-checklist/)

Ключевая проблема сформулирована так: «когда код и тесты разделяют одни слепые пятна, у вас нет страховочной сетки — у вас эхо-камера». AI-сгенерированный код падает иначе, чем написанный человеком, и традиционные QA-процессы пропускают эти паттерны отказов.

Типичные паттерны отказов AI-кода:
- **Галлюцинированные API-вызовы** — ссылки на несуществующие методы;
- **Logic drift** — правдоподобная, но некорректная реализация;
- **Тавтологические тесты** — когда один и тот же AI пишет и код, и тесты, оба проверяют одни предположения;
- **Пропущенные граничные случаи** — недостаточно представленные в обучающих данных.

Рекомендуемый шестиуровневый подход:

1. **Фidelity требований (до генерации)** — написать описания тест-кейсов до того, как AI генерирует код. Тесты должны валидировать спецификацию, а не реализацию.
2. **Статический анализ (каждый коммит)** — линтинг, проверка типов, сканирование безопасности и верификация зависимостей (AI-специфичный шаг: ассистенты иногда ссылаются на несуществующие пакеты).
3. **Юнит-тесты (покрытие ветвей 85%+)** — порог выше, чем для кода, написанного человеком (70–80%). Приоритет: контрактное тестирование, пути исключений, граничные значения, валидация безопасности.
4. **Адверсариальное AI-ревью** — отдельный AI-промпт проверяет код на галлюцинированные API, пробелы в граничных случаях и уязвимости безопасности.
5. **Интеграционное и E2E-тестирование** — AI-компоненты тестируются в полном контексте приложения, а не изолированно.
6. **Мониторинг в продакшне** — отслеживание частоты ошибок, задержек и бизнес-метрик с сегментацией по источнику кода.

#### Ответ от Claude

Корневая проблема тестирования AI-кода AI-тестами — «эхо-камера»: оба артефакта разделяют одни и те же слепые пятна модели (недостаточно обученные граничные случаи, неправильно понятые требования, отсутствующие сценарии ошибок).

Практические меры:

**До генерации кода**
- Написать спецификацию тест-кейсов на естественном языке (что именно должен делать код).
- Зафиксировать граничные условия и негативные сценарии в отдельном документе.
- Разработчик-человек рецензирует спецификацию до запуска AI.

**При написании тестов**
- Использовать *другой* AI-инструмент или другой промпт (adversarial review) для проверки тестов.
- Установить порог покрытия ветвей выше стандартного: не 70%, а 85%+.
- Явно запрашивать у AI тесты на пути исключений и отрицательные сценарии (AI по умолчанию тяготеет к happy path).

**Проверка качества самих тестов**
- Применять мутационное тестирование (Stryker для JS, PIT для Java): если тесты не ловят намеренные ошибки в коде, они не годятся.
- Прогонять статический анализ на каждый коммит, включая проверку зависимостей (галлюцинированные пакеты).

**В продакшне**
- Сегментировать метрики ошибок по типу происхождения кода (AI vs. человек), чтобы своевременно замечать деградацию качества AI-кода.

### Вопрос 24. Как применять мутационное тестирование для проверки качества тестов, написанных на AI-сгенерированный код?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Stryker Mutator — Mutant States and Metrics](https://stryker-mutator.io/docs/mutation-testing-elements/mutant-states-and-metrics/)

Мутационное тестирование автоматически вносит изменения в код — создавая «мутантов» — и проверяет, обнаруживают ли тесты эти модификации.

**Возможные состояния мутантов:**

- **Killed (убит)**: «хотя бы один тест упал, пока этот мутант был активен» — тест успешно обнаружил изменение.
- **Survived (выжил)**: «все тесты прошли, пока этот мутант был активен» — тест не способен поймать такой класс ошибок.
- **No coverage**: мутант не покрыт ни одним тестом и выжил как следствие.
- **Timeout**: выполнение теста зависло (например, бесконечный цикл). Засчитывается как обнаруженный.
- **Runtime error / Compile error**: ошибки исполнения или компиляции — не влияют на счёт мутаций.

**Ключевые метрики:**

- **Mutation Score** = `detected / valid * 100` — «общий процент обнаруженных мутантов».
- **Coverage-based Score** = `detected / covered * 100` — фокус только на покрытом коде.

Это принципиально отличается от традиционного code coverage, который измеряет лишь выполнение строк, но не проверяет, ловят ли тесты намеренные изменения поведения. Тест может «покрывать» строку, не «убивая» мутанта — и именно это делает mutation score более строгой метрикой качества тест-сьюта, чем процент покрытия.

#### Ответ от Claude

Мутационное тестирование — наиболее надёжный способ проверить, действительно ли тесты обнаруживают ошибки, а не просто выполняют строки кода.

**Принцип работы**

Инструмент автоматически создаёт «мутантов» — варианты кода с намеренно внесёнными изменениями (замена оператора, инверсия условия, изменение константы). Хороший тест должен «убить» мутанта (упасть при его появлении). Если мутант «выживает», тест не способен обнаружить такой класс ошибок.

**Почему это особенно важно для AI-кода**

AI генерирует тесты, которые «выглядят правильно»: они компилируются, проходят и обеспечивают высокое покрытие строк. Но поскольку один и тот же AI пишет и код, и тесты, оба разделяют одни предположения об «ожидаемом» поведении. Мутационное тестирование разрывает этот порочный круг — оно не зависит от намерений автора.

**Инструменты и пороги**

| Язык | Инструмент | Минимальный mutation score |
|------|-----------|--------------------------|
| JavaScript / TypeScript | Stryker | 70% (критические пути) |
| Java | PIT (Pitest) | 70% (критические пути) |
| Python | mutmut | 50–70% |

**Рекомендуемый процесс**

1. Написать или сгенерировать тесты.
2. Запустить мутационное тестирование.
3. Собрать список выживших мутантов.
4. Передать их обратно AI с запросом «добавь тесты, которые убьют этих мутантов».
5. Повторить до достижения целевого mutation score.

Этот итеративный цикл позволяет AI-инструментам улучшать собственные слабые места в тестовом покрытии.

### Вопрос 25. По каким трём измерениям следует измерять влияние AI-инструментов на продуктивность разработчиков?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [DX — How to measure AI's impact on developer productivity](https://getdx.com/blog/ai-measurement-hub/)

DX-фреймворк разделяет измерение на три измерения:

1. **Utilization (Внедрение)** — как разработчики принимают и используют AI-инструменты: ежедневные и еженедельные активные пользователи, доля PR с AI-генерированным кодом, код, попавший в продакшн как AI-authored, задачи, делегированные автономным агентам.

2. **Impact (Влияние)** — как AI влияет на продуктивность, качество и developer experience: сэкономленные часы разработчиков в неделю (industry-standard метрика), удовлетворённость разработчиков, метрики пропускной способности PR и скорости поставки, показатели качества кода и восприятие сопровождаемости.

3. **Cost (Стоимость)** — приносит ли AI-расходование положительный ROI: общие и удельные расходы на AI, чистый выигрыш по времени минус расходы на лицензии, эквивалент почасовой ставки для автономных агентов.

Реальные результаты: зрелые внедрения достигают 3 часов 45 минут экономии в неделю на разработчика. Организации, системно измеряющие продуктивность, фиксируют 5–15% реального роста, при этом «активные пользователи» видят значительно более высокий прирост, чем эпизодические.

Для успеха необходимо зафиксировать базовые значения до внедрения AI, использовать многоуровневый сбор данных (метрики инструментов, опросы, experience sampling) и избегать «метрик тщеславия», оторванных от бизнес-результатов.

#### Ответ от Claude

DX-фреймворк предлагает трёхмерную модель оценки, которая позволяет получить полную картину — в отличие от одиночных метрик вроде «число строк кода» или «скорость коммитов».

**Utilization — Внедрение**

Отвечает на вопрос: «Используют ли разработчики инструменты достаточно для получения эффекта?»
- DAU/WAU AI-функций;
- доля PR, где использовался AI-ассистент;
- доля кода, попавшего в продакшн как AI-authored.

**Impact — Влияние**

Отвечает на вопрос: «Что меняется в метриках производительности?»
- Часы, сэкономленные разработчиком в неделю (стандартная метрика отрасли);
- пропускная способность PR;
- скорость поставки фич (time-to-production);
- оценка сопровождаемости кода разработчиками;
- Developer Experience (удовлетворённость, уровень стресса).

**Cost — Стоимость**

Отвечает на вопрос: «Окупается ли инвестиция?»
- Расходы на лицензии / API в пересчёте на разработчика;
- чистая экономия времени (сэкономленные часы × ставка) минус стоимость инструментов;
- ROI и срок окупаемости.

Важно: устанавливать базовые значения *до* внедрения. Без baseline невозможно определить, изменились ли метрики благодаря AI или по другим причинам. Данные собираются тремя методами: API инструментов (объективные данные об использовании), периодические опросы и experience sampling (субъективное восприятие).

### Вопрос 26. Что показало исследование METR 2025 об ускорении разработчиков при использовании AI-инструментов и почему результаты оказались неожиданными?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [METR — Measuring the Impact of Early-2025 AI on Experienced Open-Source Developer Productivity](https://metr.org/blog/2025-07-10-early-2025-ai-experienced-os-dev-study/)

Рандомизированное контролируемое исследование METR с 16 опытными разработчиками на 246 реальных задачах в open-source репозиториях показало: при использовании AI (Cursor Pro с Claude 3.5/3.7 Sonnet) задачи выполнялись на **19% медленнее**, чем без AI.

Разрыв между восприятием и реальностью оказался разительным:
- **До начала**: разработчики ожидали ускорения на 24%;
- **Фактический результат**: замедление на 19%;
- **После опыта работы с AI**: разработчики по-прежнему считали, что ускорились на 20%.

Авторы уточняют: это снимок для конкретного контекста (опытные open-source разработчики, начало 2025 года, конкретный набор инструментов) и предостерегают от чрезмерных обобщений на другие категории разработчиков или временные рамки.

#### Ответ от Claude

Исследование METR 2025 стало одним из наиболее методологически строгих измерений продуктивности с AI: рандомизированный контролируемый эксперимент (RCT), а не опрос или самооценка.

**Ключевые результаты**

Опытные open-source разработчики работали на 19% *медленнее* при использовании Cursor Pro с frontier-моделями, чем без AI. Это прямо противоречило как предварительным ожиданиям (+24%), так и постфактумному восприятию (+20%).

**Почему результаты неожиданны**

1. Исследование опровергло широко распространённый нарратив «AI всегда ускоряет разработку».
2. Обнаружен значительный разрыв восприятия: даже после реального замедления разработчики субъективно ощущали ускорение. Это делает самооценку ненадёжным методом измерения.
3. Замедление происходило у *опытных* разработчиков на реальных, а не синтетических задачах.

**Возможные причины замедления**

- Время на проверку и исправление AI-ошибок перевешивало выигрыш от генерации;
- когнитивные переключения между режимами «генерация → ревью → доработка»;
- специфика задач в реальных репозиториях (глубокий контекст, сложные зависимости).

**Важные ограничения**

Исследование — срез начала 2025 года для конкретного контекста. К середине 2026 года инструменты существенно изменились. Результаты не применимы к юниорам, генеративным задачам (скаффолдинг, шаблонный код) или более новым агентным системам.

### Вопрос 27. Почему традиционные DORA-метрики неточно отражают влияние AI на продуктивность команды?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Faros — DORA Report 2025 Key Takeaways: AI Impact on Dev Metrics](https://www.faros.ai/blog/key-takeaways-from-the-dora-report-2025)

Исследования Faros выявили парадокс AI-продуктивности: индивидуальные метрики резко растут, а организационные — нет. Разработчики демонстрировали «на 21% больше завершённых задач, на 98% больше смёрдженных PR», однако организационные метрики поставки — Lead Time и Deployment Frequency — оставались неизменными. Прирост продуктивности поглощается узкими местами дальше по потоку, невидимыми для традиционных метрик.

Три конкретные проблемы:
1. **Неверные единицы измерения**: «административные группировки (Jira-доски, GitHub-команды) редко отражают реальные рабочие команды», в которых и происходит влияние AI.
2. **Ошибки атрибуции**: когда разработчики меняют команды, исторические данные искажают метрики, делая невозможным выделение эффекта вмешательства.
3. **Локальная оптимизация**: команды ускоряют генерацию кода, пока ограничения смещаются к этапам ревью, интеграции и деплоя.

#### Ответ от Claude

DORA-метрики (Deployment Frequency, Lead Time for Changes, Change Failure Rate, Time to Restore) разрабатывались для измерения зрелости доставки ПО в целом — они не были рассчитаны на выявление эффектов AI на отдельных этапах цикла.

**Почему DORA вводит в заблуждение при AI-внедрении**

Когда AI генерирует 30–70% коммитов и PR:
- **Deployment Frequency** растёт, но это отражает скорость генерации, а не скорость ценной поставки.
- **Lead Time for Changes** может не меняться или расти, если ревью AI-PR занимает больше времени.
- **Change Failure Rate** может расти из-за того, что AI-код содержит больше дефектов (до 1,7× по ряду исследований), но метрика будет ухудшаться с задержкой.

**Реальный парадокс**

Индивидуальная скорость выполнения задач +21%, слитых PR +98% — но организационный Lead Time неизменен. AI усиливает существующие узкие места: если раньше тормозило написание кода, теперь тормозит ревью. Метрика Deployment Frequency маскирует это смещение.

**Что использовать вместо или в дополнение**

- Сегментировать метрики по источнику кода (AI vs. человек).
- Отслеживать время ревью AI-PR отдельно.
- Добавить метрики качества: rework rate, incident rate от AI-кода.
- Применять Value Stream Mapping для видимости сквозного потока.

### Вопрос 28. Что такое «prompt injection» для AI-агентов в разработке и как он угрожает рабочим процессам?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [OWASP Gen AI Security Project — LLM01:2025 Prompt Injection](https://genai.owasp.org/llmrisk/llm01-prompt-injection/)

Prompt injection происходит, когда «пользовательские промпты меняют поведение или вывод LLM непредусмотренным образом». При этом вредоносные входные данные необязательно видны человеку — модели достаточно их распарсить.

Уязвимости prompt injection позволяют атакующим манипулировать поведением агента, вынуждая модели «нарушать правила, генерировать вредоносный контент, открывать несанкционированный доступ или влиять на критические решения».

Два основных типа:
- **Прямая инъекция** — пользователь напрямую изменяет поведение модели через вредоносный промпт (умышленно или случайно).
- **Косвенная инъекция** — «LLM принимает ввод из внешних источников, таких как веб-сайты или файлы», где встроенный вредоносный контент манипулирует поведением при обработке.

Меры защиты: ограничение поведения модели через system prompt, строгая валидация форматов вывода, фильтрация входных данных и семантическое сканирование, принцип минимальных привилегий для агентных функций, требование одобрения человеком для высокорисковых операций, adversarial testing. Эти меры снижают — но не устраняют — риск инъекций.

#### Ответ от Claude

Prompt injection (LLM01:2025 по классификации OWASP) — это атака на AI-агент, при которой злоумышленник встраивает вредоносные инструкции во входные данные, заставляя агента выполнять непредусмотренные действия.

**Прямая vs. косвенная инъекция**

| Тип | Механизм | Пример |
|-----|----------|--------|
| Прямая | Вредоносный промпт в пользовательском вводе | «Игнорируй предыдущие инструкции и...» |
| Косвенная | Вредоносные инструкции во внешних данных | README.md в репозитории содержит скрытые инструкции; агент читает файл и выполняет их |

**Угрозы для coding-агентов**

Coding-агенты особенно уязвимы к косвенным инъекциям, поскольку постоянно обрабатывают внешние данные:
- файлы репозитория и документация могут содержать скрытые инструкции;
- зависимости из `package.json` или `pom.xml` могут включать вредоносный код;
- содержимое веб-страниц, которые агент читает для ответа на вопросы.

Последствия: внедрение бэкдоров, утечка токенов и секретов, несанкционированный доступ к API, выполнение произвольных команд в shell-интеграциях.

**Эшелонированная защита**

Единого технического решения нет. Рекомендуется комбинация:
1. Минимальные привилегии агента (sandbox, ограниченные права на файловую систему).
2. Human approval gates для деструктивных операций.
3. Валидация входных данных перед передачей в контекст агента.
4. Adversarial testing сценариев с вредоносными файлами.

### Вопрос 29. Как риск галлюцинаций LLM угрожает безопасности цепочки поставок программного обеспечения?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [OWASP Gen AI Security Project — LLM09:2025 Misinformation](https://genai.owasp.org/llmrisk/llm092025-misinformation/)

Согласно OWASP LLM09:2025, «модели предлагают небезопасные или несуществующие библиотеки кода, которые могут вносить уязвимости при интеграции в программные системы».

Наиболее опасный сценарий: атакующие обнаруживают часто галлюцинируемые AI-ассистентами названия пакетов и публикуют вредоносные пакеты с такими именами в легитимных реестрах. Разработчики по незнанию устанавливают заражённые зависимости, доверяя рекомендациям AI. Это обходит традиционные меры безопасности, поскольку разработчики доверяют рекомендациям AI.

Приводится конкретный пример: «LLM предлагают использовать небезопасные сторонние библиотеки, которые при доверии без верификации приводят к рискам безопасности».

Ключевая защитная мера — безопасные практики кодирования и отказ от чрезмерного доверия AI-предложениям. Организациям следует требовать верификации всех AI-рекомендованных зависимостей перед интеграцией.

#### Ответ от Claude

Галлюцинации LLM создают специфический вектор атаки на цепочку поставок — **slopsquatting**: злоумышленники регистрируют пакеты с именами, которые AI систематически «выдумывает».

**Механизм угрозы**

Атака проходит четыре стадии:
1. AI-ассистент рекомендует несуществующий пакет (например, `jwt-secure-validator`).
2. Разработчик не проверяет наличие пакета в реестре.
3. Злоумышленник заблаговременно регистрирует этот пакет в npm/PyPI с вредоносным кодом.
4. Последующие разработчики устанавливают пакет — бэкдор распространяется.

**Масштаб проблемы**

По данным исследований, около 19,7% пакетов, рекомендованных LLM, не существуют. При этом галлюцинации воспроизводимы: 58% галлюцинированных имён появляется при повторных запросах, 43% — стабильно в 10 из 10 идентичных запросов.

**Меры защиты**

- Верифицировать каждую новую зависимость через актуальный реестр (npm, Maven Central, PyPI) до установки.
- Использовать инструменты для проверки репутации пакетов (количество загрузок, возраст, мейнтейнеры).
- Настроить SBOM (Software Bill of Materials) и автоматическую проверку зависимостей в CI/CD.
- Применять RAG (Retrieval-Augmented Generation) для заземления AI-ответов в реальных данных реестра.
- Обучать разработчиков: «никогда не устанавливать пакет только потому, что его порекомендовал AI».

### Вопрос 30. Как правильно структурировать контекст для AI-агентов, чтобы избежать «деградации контекста» и повысить качество генерации?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Anthropic Engineering — Effective context engineering for AI agents](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)

Context engineering — это «набор стратегий для курирования и поддержки оптимального набора токенов (информации) во время инференса LLM». В отличие от prompt engineering, которое фокусируется на формулировке инструкций, context engineering решает более широкую задачу управления всей информацией, доступной агенту на протяжении нескольких ходов — включая system prompt, инструменты, внешние данные и историю сообщений.

Ключевые принципы:

1. **Контекст — конечный ресурс.** «По мере роста числа токенов в контексте способность модели точно извлекать информацию снижается» — это явление называется context rot. Необходимо приоритизировать, что попадает в «attention budget» модели.

2. **Максимизировать соотношение сигнал/шум.** Руководящий принцип — найти «наименьший возможный набор высокосигнальных токенов, максимизирующих вероятность желаемого результата».

3. **Структурировать системно.** Организовывать system prompt в разделы с XML-тегами или Markdown-заголовками (фоновая информация, инструкции, руководство по инструментам, описания вывода). Это помогает модели эффективно обрабатывать информацию.

4. **Проектировать минимальные наборы инструментов.** Инструменты должны быть самодостаточными, однозначными и без функционального перекрытия. Если человек не может уверенно выбрать между инструментами, AI-агент тоже не сможет.

5. **Использовать разнообразные примеры стратегически.** Не исчерпывающие граничные случаи, а канонические примеры, демонстрирующие ожидаемое поведение.

Техники для длительных задач: **компакция** (сжатие истории с сохранением критических решений), **структурированные заметки** (внешняя память агента), **мультиагентные архитектуры** (делегирование сфокусированных задач субагентам).

#### Ответ от Claude

Context engineering — управление информацией в контексте агента как конечным ресурсом. Ключевая идея: не «дать агенту как можно больше информации», а «дать агенту минимально достаточный набор высококачественных токенов».

**Context rot: что это и почему опасно**

При заполнении контекстного окна (context window) способность модели точно обращаться к ранней информации деградирует. Это не техническое ограничение конкретной модели — это свойство архитектуры трансформеров. Практическое следствие: длинный контекст с низким соотношением сигнал/шум хуже короткого с высоким.

**Структурирование системного промпта**

Разбить на явные секции:
- Фоновый контекст (кто агент, какова задача).
- Инструкции (что делать, чего не делать).
- Описание инструментов (когда использовать каждый).
- Формат вывода (структура ответа).

**Принцип минимального набора инструментов**

Каждый лишний инструмент добавляет «шум» в контекст и увеличивает вероятность неправильного выбора. Если два инструмента делают похожее — устранить дублирование.

**Управление длинными сессиями**

| Техника | Когда применять |
|---------|----------------|
| Компакция истории | Сессии > 50 ходов; агент суммирует историю, сохраняя ключевые решения |
| Внешние заметки (scratchpad) | Долгие задачи с промежуточными результатами, которые нужно помнить |
| Мультиагентность | Независимые подзадачи, которые можно делегировать субагентам с компактным контекстом |

**Just-in-time retrieval** — загружать документацию и код не заранее («на всякий случай»), а в момент, когда они нужны агенту.

### Вопрос 31. Как при парном программировании с AI-ассистентом применять практику «тест-первый» и почему AI не должен генерировать тесты после кода?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [ForgeCode — AI Agent Best Practices: 12 Lessons from AI Pair Programming](https://forgecode.dev/blog/ai-agent-best-practices/)

Статья рекомендует конкретный рабочий процесс: «Попросите AI написать падающий тест, который фиксирует именно то, что вы хотите. Проверьте тест самостоятельно — убедитесь, что он тестирует правильное поведение. Затем скажите AI: "Сделай так, чтобы этот тест прошёл"».

Почему AI не должен генерировать тесты после кода: «Неверный тест приведёт к коду, который удовлетворяет неверным требованиям». Если AI сначала пишет код, а затем тесты, тесты адаптируются под существующую реализацию, а не задают правильное поведение независимо.

Цикл edit-test loop:
1. Разработчик задаёт, что тест должен валидировать.
2. AI пишет падающий тест.
3. Разработчик проверяет тест на корректность.
4. AI итерирует, добиваясь прохождения теста, при необходимости запуская его несколько раз.

Человеческое суждение остаётся центральным — разработчик проектирует требования, AI выполняет их. Это гарантирует, что требования, а не предположения AI, управляют разработкой.

#### Ответ от Claude

Практика «тест-первый» с AI-ассистентом — это TDD с изменённым распределением ролей: разработчик определяет требования через тест-кейсы, AI реализует логику.

**Почему нельзя давать AI писать тесты после кода**

Если AI сначала пишет код, а потом тесты:
- тесты будут подтверждать то, что код *делает*, а не то, что код *должен делать*;
- баги и неверные предположения будут «задокументированы» в тестах как ожидаемое поведение;
- покрытие будет высоким, но смысловое качество тестов — низким.

**Правильный цикл «тест-первый» с AI**

```
1. Разработчик формулирует требование («функция должна возвращать X для входа Y»)
2. Разработчик просит AI написать падающий тест для этого требования
3. Разработчик проверяет тест: корректно ли он описывает требование?
   └── если нет: уточнить требование, вернуться к шагу 2
4. AI пишет реализацию, делающую тест зелёным
5. Проверить: тест действительно зелёный? Нет регрессий?
6. Повторить для следующего требования
```

**Роли в парном программировании с AI**

- *Разработчик* (навигатор): формулирует требования, задаёт граничные условия, проверяет тесты на смысловую корректность.
- *AI* (водитель): пишет тесты по спецификации, реализует код, исправляет до прохождения тестов.

Частые небольшие коммиты предотвращают накопление непроверенного AI-кода: каждый коммит должен иметь прошедшие тесты.

### Вопрос 32. Как подходить к архитектурным решениям в парном программировании с AI — что можно делегировать AI, а что требует обязательного участия человека?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Exceeds AI — AI Pair Programming Guide: Tools, Benefits & Best Practices](https://blog.exceeds.ai/ai-pair-programming/)

«Правило 30%» рекомендует ограничивать AI до 30% начальной генерации кода с доработкой разработчиком оставшихся 70%. Этот порог предотвращает чрезмерную зависимость от AI в сложной логике и задачах, критических с точки зрения безопасности.

Что требует обязательного участия человека:
- **Архитектура и основная доменная логика** — фундаментальные решения, определяющие структуру системы;
- **Production-механизмы защиты** — контроль безопасности и критическая инфраструктурная инфраструктура;
- **Задачи сложного рассуждения** — проблемы, требующие глубоких предметных знаний;
- **Граничные случаи и обработка ошибок** — «последние 20% работы», которые AI обычно опускает.

Проблема «распространения предположений»: модели могут неверно понять требования на раннем этапе, затем выстроить целые функции на ошибочных предпосылках — сбой, который только люди могут поймать через тщательное ревью.

AI хорошо справляется с: генерацией шаблонного кода и автодополнением, мультифайловым рефакторингом при наличии контекста, начальным скаффолдингом функций, предложениями кода для рутинных паттернов.

Статья рекомендует относиться к AI-выводу «как к коду джуниора» через итеративное ревью, проверяющее корректность, граничные случаи и security-импликации перед мёрджем.

#### Ответ от Claude

AI-ассистент берёт на себя роль «водителя» в традиционном парном программировании — выполняет рутинную реализацию, но лишён контекста бизнес-целей, архитектурных ограничений и нефункциональных требований системы.

**Что можно делегировать AI**

- Шаблонный код (DTO, геттеры/сеттеры, конструкторы, Mapper-классы).
- CRUD-операции по существующим паттернам.
- Начальный скаффолдинг сервисов и контроллеров.
- Рефакторинг по чётко сформулированному правилу (переименование, выделение метода).
- Тесты по написанным разработчиком спецификациям (см. вопрос 31).

**Что требует обязательного участия человека**

| Область | Почему нельзя делегировать |
|---------|--------------------------|
| Архитектурные паттерны | AI не знает историю решений и причины текущих ограничений |
| Безопасность и аутентификация | AI систематически пропускает «последние 20%» — граничные случаи и исключения |
| Бизнес-логика с исключениями | Правила предметной области часто неявны и не зафиксированы в коде |
| Выбор технологий и зависимостей | AI может предлагать устаревшие или галлюцинированные библиотеки |
| Компромиссы производительности | Требуют понимания нагрузочного профиля системы |

**Правило 30%**

Ограничивать долю AI-кода в начальной генерации до ~30% от объёма сессии. Остальное — доработка, проверка, адаптация к контексту. Это правило предотвращает ситуацию, когда AI «распространяет предположения»: раннее неверное понимание требования тиражируется на всю функцию.

### Вопрос 33. Как настроить процесс code review в команде для AI-ассистированной разработки: что нужно изменить в PR-шаблоне и распределении внимания ревьюеров?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Git AutoReview — Code Review Checklist for AI-Generated Code: 12 Things to Verify](https://gitautoreview.com/blog/code-review-checklist-ai-generated-code)

Статья подчёркивает: «AI не пишет плохой код — он пишет правдоподобный код», который проходит линтеры и компилируется без ошибок, но содержит семантические ошибки, которые замечает человек. Процессы ревью нуждаются в структурных обновлениях.

12-пунктный чеклист AI-специфичных проблем:
- Соответствие требованиям (соответствует ли код задаче в тикете?);
- галлюцинированные имена пакетов (верификация всех новых зависимостей);
- побочные эффекты в других файлах при переименованиях;
- захардкоженные учётные данные;
- полнота обработки ошибок;
- корректность логики за пределами синтаксиса;
- согласованность именования с существующей кодовой базой;
- мёртвый/недостижимый код;
- покрытие граничных случаев тестами;
- отладочные артефакты (`console.log`);
- уязвимости безопасности (OWASP Top 10);
- соответствие архитектурным паттернам.

**PR-шаблон** (добавить в `.github/PULL_REQUEST_TEMPLATE.md`):
```markdown
## AI-Generated Code Review Checklist
- [ ] Code fulfills original ticket requirements
- [ ] All imports verified on registries (npm/PyPI)
- [ ] Error handling covers failure scenarios
- [ ] No hardcoded secrets or credentials
```

Автоматизация берёт на себя 8 из 12 пунктов: Quick Review инструменты ловят нарушения именования, мёртвый код, отладочные артефакты; Deep Review агенты исследуют кросс-файловые эффекты и архитектурное соответствие; SAST-сканирование выявляет уязвимости OWASP. Люди остаются ответственными за соответствие требованиям, верификацию пакетов и архитектурные суждения.

#### Ответ от Claude

Процесс ревью для AI-кода отличается от стандартного: AI систематически пропускает одни классы ошибок (семантика, архитектура, безопасность) и реже делает другие (синтаксис, форматирование). Это требует перераспределения внимания ревьюера.

**Изменения в PR-шаблоне**

Добавить секцию об участии AI в `.github/PULL_REQUEST_TEMPLATE.md`:

```markdown
## AI-участие
- [ ] Код написан полностью человеком
- [ ] Код написан с AI-ассистентом (укажите инструмент): _______
- [ ] Код написан преимущественно AI с доработкой человека

## Чеклист для AI-кода (заполнить при наличии AI-участия)
- [ ] Все новые зависимости верифицированы в реестре
- [ ] Обработка ошибок покрывает не только «счастливый путь»
- [ ] Отсутствуют захардкоженные секреты и токены
- [ ] Логика соответствует требованиям задачи, а не только проходит тесты
```

**Распределение внимания ревьюеров**

На автоматику (ESLint, CodeQL, SAST, зависимости):
- форматирование и стиль;
- базовые уязвимости (XSS, SQL injection по паттернам);
- устаревшие зависимости.

На человека-ревьюера:
- соответствие логики бизнес-требованиям;
- архитектурная целостность (не «как написано», а «зачем так»);
- security patterns, которые статический анализ пропускает;
- полнота обработки ошибок и edge cases;
- верификация новых пакетов (slopsquatting).

Ключевой принцип: чем меньше размер PR, тем эффективнее ревью. AI-агенты могут генерировать большие диффы — устанавливать ограничение в ~200–300 строк на PR.

### Вопрос 34. Как оценить реальный ROI от внедрения AI-инструментов в команде разработчиков и какие сигналы указывают на неэффективное внедрение?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Swarmia — Measuring the productivity impact of AI coding tools](https://www.swarmia.com/blog/productivity-impact-of-ai-coding-tools/)

Ключевой вывод: «ускорение кодирования в 3 раза даёт лишь 2–5% роста на уровне организации» без системного редизайна процессов. AI усиливает существующие узкие места.

Системные метрики для оценки ROI:
- Cycle time по стадиям (время кодирования vs. время ревью);
- пропускная способность завершённой работы в продакшне;
- баланс инвестиций между новыми функциями и обслуживанием;
- rework rate за шестимесячные периоды.

Показатели качества: частота инцидентов в продакшне и время восстановления, паттерны обратной связи на код-ревью, надёжность CI и тренды нестабильных тестов, тренды бэклога багов.

**Тревожные сигналы неэффективного внедрения:**
1. Растущая нагрузка на ревью: PR от агентов накапливаются быстрее, чем команды успевают их ревьюировать;
2. Рост объёма технического обслуживания: если KTLO-работа растёт быстрее, чем выпуск функций, «система генерирует код быстрее, чем код, достойный сохранения»;
3. Деградация качества: AI-код содержит примерно в 1,7 раза больше проблем, чем написанный человеком;
4. Рост расходов без роста эффекта: «растущие расходы приемлемы, когда растёт и эффект» — сигнал к действию, когда расходы растут, а эффективность выходит на плато;
5. Концентрация знаний: владение кодом становится неясным, когда инженеры чувствуют отчуждённость от AI-кода, который они деплоят.

#### Ответ от Claude

Оценка ROI от AI-инструментов требует многоуровневого подхода: ни одна метрика не даёт полной картины.

**Методология оценки**

Обязательный шаг — зафиксировать baseline до внедрения. Без исходных значений невозможно определить, что изменилось благодаря AI.

Метрики распределены по трём уровням:

**Уровень 1 — Индивидуальная скорость** (легко измерить, легко исказить):
- Число PR, строк кода, завершённых задач;
- время написания кода.

**Уровень 2 — Системное качество** (сложнее измерить, надёжнее):
- Rework rate (доля кода, переписанного в течение 30 дней);
- Cycle time по стадиям (рост времени ревью — красный флаг);
- Change Failure Rate (AI-код попадает в продакшн с дефектами);
- KTLO vs. feature balance.

**Уровень 3 — Бизнес-результат** (самый надёжный индикатор):
- Time-to-market для фич;
- Production incident rate;
- Customer-visible defects.

**Сигналы неэффективного внедрения**

| Сигнал | Что означает |
|--------|-------------|
| Время ревью AI-PR выше, чем human-PR | Нет пользы: ревью поглощает экономию от генерации |
| Rework rate > 15% | AI-код переписывается чаще, чем создаётся ценность |
| Рост KTLO-работы | Технический долг накапливается быстрее, чем успевают разобраться |
| Production incidents > 10% от baseline | Качество деградирует несмотря на скорость |
| Размер PR агентов > 500 строк | Ревью нереалистично; качество будет низким |

Формула чистой экономии: `(сэкономленные часы × ставка разработчика) - стоимость лицензий - рост накладных расходов на ревью`.

### Вопрос 35. Что такое галлюцинации в AI-ассистированной разработке и как они проявляются в генерируемом коде?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [diffray.ai — LLM Hallucinations in AI Code Review](https://diffray.ai/blog/llm-hallucinations-code-review/)

Галлюцинации происходят, когда LLM генерируют «некорректные, сфабрикованные или опасные предложения кода» с тревожной частотой. Фундаментальная причина — в стимулах обучения: модели учатся тому, что «уверенные предположения лучше признания неопределённости».

Три критических проявления в коде:
1. **Уязвимости безопасности**: 29–45% AI-генерированного кода содержит эксплуатируемые security-дефекты;
2. **Выдуманные рекомендации пакетов**: 19,7% предложенных пакетов не существуют, при этом 58% галлюцинированных пакетов повторяются при последующих запросах;
3. **Несуществующие проблемы**: AI уверенно сообщает о проблемах, которых нет, провоцируя расследования продолжительностью 15–30 минут, подрывающие доверие разработчиков.

Четыре формы галлюцинаций:
1. **Фактические ошибки** — некорректная информация, высказанная с уверенностью;
2. **Сфабрикованные источники** — генерированные ссылки (точность цитирования GPT-4 составляла лишь 13,4%);
3. **Ошибки рассуждения** — логические несоответствия (около 19% галлюцинаций);
4. **Ошибки, спровоцированные промптом** — модели следуют некорректным предпосылкам вместо их исправления.

#### Ответ от Claude

Галлюцинация LLM в контексте разработки — это генерация технически правдоподобного, но фактически неверного контента с высокой степенью уверенности.

**Четыре формы галлюцинаций**

1. **Фактические ошибки** — неверные технические факты (версии API, поведение методов, сигнатуры функций). Пример: `String.format()` описывается как принимающий параметры в другом порядке.

2. **Выдуманные источники** — ссылки на несуществующие статьи, RFC, официальные документы. LLM уверенно цитирует «документацию Oracle» с несуществующим URL.

3. **Ошибки рассуждения** — логически некорректные выводы. Пример: AI генерирует корректный алгоритм сортировки, но применяет его к структуре данных, которая уже отсортирована, добавляя лишний проход.

4. **Ошибки от некорректного промпта** — если разработчик задаёт неверную предпосылку, AI не исправляет её, а строит на ней ответ.

**Специфичные для кода проявления**

- Рекомендация несуществующих пакетов (19,7% случаев) — основа атак slopsquatting.
- Код с уязвимостями безопасности, принятый за безопасный (XSS, SQL injection, раскрытие секретов).
- Ложные предупреждения в code review — расходование 15–30 минут на расследование несуществующих проблем.
- Использование устаревших API (обучающие данные с датой отсечения).

**Почему нельзя «устранить» галлюцинации**

Они являются следствием принципа обучения: максимизация правдоподобия следующего токена, а не точности. Единственная защита — верификационный слой (статический анализ, ревью, проверка зависимостей).

### Вопрос 37. Какие типы уязвимостей наиболее часто встречаются в коде, сгенерированном AI-ассистентами?

**Топ уязвимостей по частоте и критичности**

| Тип | Частота/статистика | Причина |
|-----|---------------------|---------|
| XSS | 86% провалов | AI выводит пользовательский ввод без `htmlspecialchars` / экранирования |
| Захардкоженные секреты | +40% vs human | AI вставляет тестовые токены из обучающих данных «для наглядности» |
| Privilege Escalation | +322% vs human | AI не проверяет права доступа при генерации CRUD-методов |
| Архитектурные дефекты | +153% vs human | AI не видит всей системы — генерирует компонент без учёта security-контракта |

**Почему AI системно пропускает security-контекст**

AI оптимизирован для «функционально выглядящего» кода. При генерации:
- не видит соседних компонентов и их security-гарантий;
- использует паттерны из обучающих данных, включая небезопасные примеры;
- генерирует «счастливый путь» без проверок прав доступа.

**Приоритеты при ревью AI-кода (по убыванию риска)**

1. Все точки вывода пользовательского ввода — проверить экранирование.
2. Все переменные с именами `key`, `token`, `secret`, `password` — не должны быть захардкоденными строками.
3. Все методы с изменением состояния (PUT/POST/DELETE) — проверить наличие авторизационных проверок.
4. Все новые зависимости — верифицировать в реестре.

### Вопрос 38. Что такое slopsquatting и как он угрожает цепочке поставок программного обеспечения?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Snyk — Slopsquatting: New AI Hallucination Threats & Mitigation Strategies](https://snyk.io/articles/slopsquatting-mitigation-strategies/)

Slopsquatting — новый вектор атаки, эксплуатирующий склонность AI генерировать неточные рекомендации пакетов. По данным исследования, «19,7% генерируемых пакетов в протестированных LLM были галлюцинированными» — AI-ассистенты часто рекомендуют зависимости, которые звучат правдоподобно, но на самом деле не существуют.

Атакующие используют это, регистрируя выдуманные имена пакетов в публичных реестрах и создавая вредоносные ловушки для разработчиков, доверяющих AI-предложениям без верификации. Пакеты часто включают функциональный код вместе с вредоносной нагрузкой, что затрудняет обнаружение при первоначальном тестировании.

Четыре стадии атаки:
1. **Предложение AI** — ассистент рекомендует реалистично звучащий, но несуществующий пакет вроде «jwt-secure-validator»;
2. **Доверие разработчика** — разработчик принимает рекомендацию без проверки существования пакета;
3. **Регистрация злоумышленником** — злоумышленник регистрирует галлюцинированное имя пакета в PyPI, npm или аналогах;
4. **Установка** — выполнение предложенного кода устанавливает скомпрометированный пакет.

Атака эксплуатирует психологическую склонность разработчиков доверять AI-рекомендациям, превращая инструменты продуктивности в векторы атаки.

#### Ответ от Claude

Slopsquatting — атака на цепочку поставок (software supply chain), специфичная для эпохи AI-ассистентов. Она эксплуатирует предсказуемость галлюцинаций LLM, а не случайные опечатки разработчиков.

**Механизм атаки**

```
AI рекомендует «python-jwt-validator»
        ↓
Разработчик не проверяет наличие пакета
        ↓
Злоумышленник регистрирует «python-jwt-validator» в PyPI
        ↓
pip install python-jwt-validator → установка вредоносного кода
```

**Чем slopsquatting отличается от typosquatting**

| Аспект | Typosquatting | Slopsquatting |
|--------|--------------|---------------|
| Источник имени | Реальный пакет + опечатка человека | Выдуманное AI имя |
| Предсказуемость | Низкая (зависит от опечаток) | Высокая (галлюцинации воспроизводимы) |
| Масштаб атаки | Ограничен популярными пакетами | Зависит от частоты использования AI |

**Масштаб угрозы**

19,7% рекомендованных LLM пакетов не существует. Имена воспроизводимы: 43% появляются стабильно в 10 из 10 идентичных запросов — это создаёт надёжную поверхность атаки.

**Защита**

- Всегда проверять наличие пакета в реестре (npm, PyPI, Maven Central) перед установкой.
- Настроить SBOM и автоматическую верификацию зависимостей в CI.
- Использовать `npm audit` / `pip-audit` / OWASP Dependency-Check в пайплайне.
- Обучать команду: «любой новый пакет от AI = верификация до `install`».

### Вопрос 39. Почему hallucinated (выдуманные) имена пакетов в AI-коде особенно удобны для атакующих?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [FOSSA — Slopsquatting: AI Hallucinations and the New Software Supply Chain Risk](https://fossa.com/blog/slopsquatting-ai-hallucinations-new-software-supply-chain-risk/)

Выдуманные AI пакеты особенно удобны для атакующих, поскольку они системны, а не случайны. «Большинство галлюцинаций — не просто случайный шум, а воспроизводимые артефакты» того, как AI-модели реагируют на промпты, что «увеличивает их ценность для атакующих». Эта предсказуемость позволяет злоумышленникам идентифицировать жизнеспособные цели без случайного перебора.

Показатели воспроизводимости: более 58% галлюцинированных имён пакетов появлялись при нескольких прогонках, а 43% воспроизводились стабильно при десяти идентичных запросах. Эта согласованность создаёт устойчивую поверхность атаки.

Ключевое отличие от typosquatting: традиционный typosquatting эксплуатирует распространённые опечатки реальных пакетов (например, регистрация `requestss` для перехвата опечаток `requests`). Slopsquatting нацелен на полностью вымышленные имена, которые создаёт только AI, — создавая поверхность уязвимости, которой прежде не существовало в цепочке поставок ПО.

#### Ответ от Claude

Удобство галлюцинированных имён для атакующих определяется ключевым свойством — **воспроизводимостью**. В отличие от случайных ошибок, галлюцинации LLM систематичны и предсказуемы.

**Почему это важно для атакующих**

При typosquatting злоумышленник угадывает, какую опечатку допустит разработчик. При slopsquatting:
1. Атакующий делает тот же запрос к LLM, что и разработчики.
2. Модель воспроизводимо галлюцинирует одни и те же имена (43% — стабильно в 10/10 запросах).
3. Атакующий заблаговременно регистрирует эти имена.
4. К моменту, когда разработчик пробует установить пакет, ловушка уже готова.

**Количественные характеристики**

- 19,7% рекомендованных пакетов не существует.
- 58% галлюцинированных имён повторяются при повторных запросах.
- 43% появляются стабильно в 10 из 10 идентичных сессий.

Это означает: злоумышленник может автоматизировать составление списка «надёжных» целей, просто многократно отправляя один и тот же промпт разным моделям.

**Асимметрия атаки**

Регистрация пакета в npm/PyPI занимает минуты и стоит ничего. Атака масштабируема: один раз зарегистрированный пакет потенциально устанавливается тысячами разработчиков, следующих одному и тому же AI-совету.

### Вопрос 40. Какие из десяти рисков OWASP LLM Top 10 наиболее критичны при использовании AI для генерации кода?

**Категория:** Разработка с AI-ассистентами · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Sonar — OWASP LLM Top 10: How it Applies to Code Generation](https://www.sonarsource.com/resources/library/owasp-llm-code-generation/)

Пять из десяти рисков OWASP LLM Top 10 непосредственно пересекаются с генерацией кода:

**LLM01: Prompt Injection** — атакующие создают вредоносные входные данные для компрометации вывода LLM. «Как SQLi для чат-ботов». Разработчики должны строго валидировать входные данные и поддерживать человеческий надзор за генерируемым кодом.

**LLM02: Insecure Output Handling** — «LLM-версия ненадлежащего кодирования или экранирования вывода». Когда LLM генерирует код с SQL-инъекциями или раскрытыми API-ключами, непроверенный код может вносить критические уязвимости.

**LLM03: Training Data Poisoning** — вредоносный код в обучающих датасетах заставляет LLM воспроизводить уязвимости. «Валидируйте качество любого кода, сгенерированного AI» — нельзя предполагать надёжность.

**LLM06: Sensitive Information Disclosure** — LLM могут непреднамеренно включать учётные данные или проприетарные данные в генерируемый код. Команды должны «сканировать код на секреты» перед деплоем.

**LLM07: Insecure Plugin Design** — некорректно написанные плагины создают бреши в безопасности. Реализация требует безопасных практик кодирования и осведомлённости об API-безопасности.

Ключевая мера защиты для всех пяти рисков: **инструменты автоматизированного анализа кода обнаруживают уязвимости в AI-сгенерированном коде до деплоя в продакшн**.

#### Ответ от Claude

OWASP LLM Top 10 описывает риски LLM-систем в целом, но пять из них особенно критичны именно для сценариев генерации кода.

**Топ-5 рисков для code generation**

| Риск | Как проявляется при генерации кода |
|------|------------------------------------|
| **LLM01 — Prompt Injection** | Злоумышленник встраивает в README или тикет инструкцию «добавь backdoor», AI выполняет её |
| **LLM02 — Insecure Output Handling** | AI генерирует код с захардкоженными токенами, SQL без параметризации, вывод без экранирования |
| **LLM03 — Training Data Poisoning** | Уязвимый open-source код в обучающих данных воспроизводится в генерируемом коде |
| **LLM06 — Sensitive Information Disclosure** | AI «вспоминает» API-ключи или внутренние URL из обучающих данных и вставляет их в код |
| **LLM07 — Insecure Plugin Design** | IDE-расширения и MCP-серверы с широкими правами доступа становятся точкой входа |

**Специфика каждого риска**

**LLM01 (Prompt Injection)** в coding-агентах особенно опасен косвенной формой: вредоносные инструкции в файлах проекта (issues, документация, `package.json`) обрабатываются агентом и выполняются.

**LLM02 (Insecure Output Handling)** — наиболее часто реализующийся риск: 86% провалов на XSS, захардкоженные секреты в 40% случаев чаще, чем у human-разработчиков.

**LLM03 (Training Data Poisoning)** — труднее всего обнаружить. Уязвимость «встроена» в модель и воспроизводится систематически, а не случайно.

**Единая линия защиты для всех пяти рисков**

Автоматизированный анализ кода (SAST) обязателен на каждый коммит: он перехватывает большинство артефактов LLM02 и частично LLM06. Human review остаётся незаменимым для LLM01 и архитектурных дефектов.
