### Вопрос 1. Что такое нотация Big-O? Какова временная сложность операций поиска, вставки и удаления для массива, связного списка и хэш-таблицы?

**Категория:** Алгоритмы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Big-O Algorithm Complexity Cheat Sheet](https://www.bigocheatsheet.com/)

Нотация Big-O — это способ описания производительности или сложности алгоритма. Она показывает, как время выполнения или использование памяти растут по мере увеличения размера входных данных.

Временная сложность операций для основных структур данных (средний случай / худший случай):

| Структура данных | Поиск | Вставка | Удаление |
|---|---|---|---|
| Массив (Array) | O(n) / O(n) | O(n) / O(n) | O(n) / O(n) |
| Односвязный список (Singly-Linked List) | O(n) / O(n) | O(1) / O(1) | O(1) / O(1) |
| Хэш-таблица (Hash Table) | O(1) / O(n) | O(1) / O(n) | O(1) / O(n) |

Хэш-таблицы обеспечивают константное время в среднем случае для всех операций, но деградируют до линейного в худшем случае (при большом количестве коллизий). Массивы требуют последовательного сканирования при поиске. Связные списки достигают константной вставки и удаления при наличии указателя, но нуждаются в линейном обходе для поиска нужного узла.

#### Ответ от Claude

**Что такое нотация Big-O**

Big-O — это математическая нотация, описывающая асимптотическую верхнюю границу роста времени выполнения (или потребления памяти) алгоритма в зависимости от размера входных данных `n`. Она отвечает на вопрос: «Как ухудшится производительность, если входные данные вырастут?» Константные множители и слагаемые меньшего порядка игнорируются — важен только доминирующий член.

Основные классы сложности (от лучшего к худшему):
- **O(1)** — константное время (независимо от размера данных)
- **O(log n)** — логарифмическое (бинарный поиск)
- **O(n)** — линейное (обход массива)
- **O(n log n)** — линейно-логарифмическое (эффективные сортировки)
- **O(n²)** — квадратичное (вложенные циклы)
- **O(2^n)** — экспоненциальное

**Временная сложность операций**

Важно разграничивать: «поиск» может означать поиск по индексу (access by index) или поиск по значению (search by value). Ниже — поиск по значению и стандартные операции вставки/удаления:

**Массив (Array)**
- **Поиск по значению:** O(n) — нужно обойти все элементы в худшем случае.
- **Доступ по индексу:** O(1) — прямое вычисление адреса ячейки памяти.
- **Вставка в конец:** O(1) амортизированно (для динамического массива, например `ArrayList`), O(n) при необходимости перераспределения памяти.
- **Вставка в середину/начало:** O(n) — требуется сдвиг всех последующих элементов.
- **Удаление:** O(n) — требуется сдвиг элементов для заполнения пустоты.

**Связный список (Linked List)**
- **Поиск по значению:** O(n) — необходим последовательный обход от головы.
- **Вставка в начало (с указателем):** O(1) — создаётся новый узел, перенаправляется указатель head.
- **Вставка в конец (без tail-указателя):** O(n) — нужен обход до конца; если есть `tail`, то O(1).
- **Вставка в произвольную позицию (по указателю на узел):** O(1) — только переключение указателей.
- **Удаление (по указателю на узел):** O(1); поиск самого узла — O(n).

**Хэш-таблица (HashMap/HashSet)**
- **Поиск, вставка, удаление — средний случай:** O(1) — вычисление хэша + прямой доступ к «корзине» (bucket).
- **Худший случай:** O(n) — возникает при многочисленных коллизиях, когда все элементы попадают в одну корзину. В Java 8+ `HashMap` переходит с цепочки (`LinkedList`) на красно-чёрное дерево при длине цепочки ≥ 8, что улучшает худший случай до O(log n).

**Сводная таблица (средний / худший случай)**

| Структура данных | Поиск (по значению) | Вставка | Удаление |
|---|---|---|---|
| Массив | O(n) / O(n) | O(n) / O(n) | O(n) / O(n) |
| Связный список | O(n) / O(n) | O(1) / O(1) | O(1) / O(1) |
| Хэш-таблица | O(1) / O(n) | O(1) / O(n) | O(1) / O(n) |

Примечание: вставка/удаление в связном списке — O(1) при наличии указателя на нужный узел. Если узел сначала нужно найти, суммарная операция займёт O(n).

### Вопрос 2. В чём разница между O(n log n) и O(n²)? Приведите примеры алгоритмов сортировки для каждой сложности.

**Категория:** Алгоритмы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Sorting algorithm — Wikipedia](https://en.wikipedia.org/wiki/Sorting_algorithm)

Практические алгоритмы сортировки общего назначения почти всегда основаны на алгоритмах со средней временной сложностью O(n log n), наиболее распространёнными из которых являются heapsort, merge sort и quicksort.

**O(n log n) — эффективные алгоритмы:**
- **Merge Sort** — использует преимущество лёгкого слияния уже отсортированных списков в новый отсортированный список. Гарантирует O(n log n) во всех случаях.
- **Heapsort** — использует структуру данных «куча» и эффективно находит наибольший элемент за O(log n) вместо O(n).
- **Quicksort** — достигает O(n log n) в среднем через разбиение, однако в худшем случае при плохом выборе опорного элемента деградирует до O(n²).

**O(n²) — простые, но медленные алгоритмы:**
- **Insertion Sort** — обрабатывает элементы по одному, вставляя каждый на нужную позицию. Эффективен для маленьких списков, но квадратичен на больших.
- **Selection Sort** — многократно находит минимальные значения и ставит их в начало, требуя O(n²) сравнений в целом.
- **Bubble Sort** — сравнивает первые два элемента, и если первый больше второго — меняет их местами, продолжая проходы с квадратичной сложностью.

#### Ответ от Claude

**Количественная разница**

Пусть n = 1 000 000 элементов:
- O(n log n): примерно 1 000 000 × 20 = **20 000 000** операций
- O(n²): примерно 1 000 000 × 1 000 000 = **1 000 000 000 000** операций

Разница — в 50 000 раз. На практике это означает разницу между миллисекундами и часами.

При n = 10 000:
- O(n log n) ≈ 133 000 операций
- O(n²) = 100 000 000 операций

Для совсем маленьких n (например, n < 20) O(n²) алгоритмы могут работать быстрее из-за меньших константных накладных расходов — именно поэтому `Arrays.sort()` в Java использует Dual-Pivot Quicksort с fallback на Insertion Sort при размере подмассива ≤ 47 элементов.

**Алгоритмы O(n log n)**

| Алгоритм | Лучший | Средний | Худший | Доп. память |
|---|---|---|---|---|
| Merge Sort | O(n log n) | O(n log n) | O(n log n) | O(n) |
| Heapsort | O(n log n) | O(n log n) | O(n log n) | O(1) |
| Quicksort | O(n log n) | O(n log n) | O(n²) | O(log n) |
| Timsort (Java для объектов) | O(n) | O(n log n) | O(n log n) | O(n) |

- **Merge Sort**: делит массив пополам рекурсивно (log n уровней), на каждом уровне слияние занимает O(n) — итого O(n log n). Стабильный, но требует дополнительную память O(n).
- **Heapsort**: строит max-heap за O(n), затем n раз извлекает максимум за O(log n). Нестабильный, но in-place.
- **Quicksort**: выбирает опорный элемент, разбивает массив на две части, рекурсивно сортирует их. Средний случай — O(n log n), худший — O(n²) при упорядоченных данных и наивном выборе опорного. Реализации используют randomized pivot или median-of-three для избежания худшего случая.

**Алгоритмы O(n²)**

| Алгоритм | Лучший | Средний | Худший | Доп. память |
|---|---|---|---|---|
| Bubble Sort | O(n) | O(n²) | O(n²) | O(1) |
| Insertion Sort | O(n) | O(n²) | O(n²) | O(1) |
| Selection Sort | O(n²) | O(n²) | O(n²) | O(1) |

- **Bubble Sort**: попарно сравнивает соседние элементы и меняет их местами, если они в неправильном порядке. Простейший, но наименее эффективный.
- **Insertion Sort**: берёт каждый элемент и вставляет его в правильную позицию среди уже отсортированных. Эффективен на почти отсортированных данных (лучший случай O(n)).
- **Selection Sort**: на каждом шаге находит минимальный элемент оставшейся части и ставит его на нужную позицию. В отличие от Bubble Sort — нестабильный.

**Нижняя граница сложности сортировки сравнением**

Доказано, что любой алгоритм сортировки, основанный на сравнениях, требует не менее O(n log n) операций в худшем случае. Это теоретический оптимум для comparison-based сортировок. Линейные алгоритмы (Counting Sort, Radix Sort, Bucket Sort) обходят это ограничение за счёт знаний о природе данных (например, ограниченный диапазон целых чисел).

### Вопрос 3. Чем стек отличается от очереди? Приведите примеры задач, где каждая из этих структур данных уместна.

**Категория:** Алгоритмы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Stack vs. Queue — happycoders.eu](https://www.happycoders.eu/algorithms/stack-vs-queue/)

**Стек (Stack)** — линейная структура данных, в которой элементы вставляются и удаляются по принципу LIFO («последним пришёл — первым ушёл»). Самый последний добавленный элемент удаляется первым.

**Очередь (Queue)** — линейная структура данных, в которой элементы вставляются и удаляются по принципу FIFO («первым пришёл — первым ушёл»). Элементы выходят в том порядке, в котором вошли.

Таблица операций:

| Операция | Стек | Очередь |
|---|---|---|
| Вставка | Push (в вершину) | Enqueue (в хвост) |
| Удаление | Pop (из вершины) | Dequeue (из головы) |

В Java класс `Stack` предоставляет конкретную реализацию с синхронизированными методами, что делает его потокобезопасным, но потенциально менее эффективным. Интерфейс `Queue` предлагает гибкость через множество реализаций: непотокобезопасные (например, `ArrayDeque`), с пессимистической блокировкой (`LinkedBlockingQueue`) и оптимистичные (`ConcurrentLinkedQueue`). Статья рекомендует избегать устаревшего класса `Stack` в пользу реализаций интерфейса `Deque`.

#### Ответ от Claude

**Принципиальное отличие**

| Характеристика | Стек (Stack) | Очередь (Queue) |
|---|---|---|
| Принцип работы | LIFO — Last In, First Out | FIFO — First In, First Out |
| Аналог из жизни | Стопка тарелок | Очередь в кассу |
| Добавление | Push — в вершину (top) | Enqueue — в хвост (tail/rear) |
| Извлечение | Pop — из вершины (top) | Dequeue — из головы (head/front) |
| Просмотр без удаления | Peek — вершина | Peek — голова |
| Сложность операций | O(1) для push/pop | O(1) для enqueue/dequeue |

**Java API**

Стек:
- Устаревший класс `java.util.Stack` (наследник `Vector`, синхронизирован, не рекомендуется).
- Предпочтительная замена: `Deque<T> stack = new ArrayDeque<>()` с методами `push()` / `pop()` / `peek()`.

Очередь:
- Интерфейс `java.util.Queue<T>` с методами `offer()` / `poll()` / `peek()`.
- Реализации: `LinkedList`, `ArrayDeque`, `PriorityQueue` (с упорядочиванием по приоритету).
- Для потокобезопасности: `LinkedBlockingQueue`, `ConcurrentLinkedQueue`, `ArrayBlockingQueue`.

**Когда использовать стек**

1. **Управление вызовами функций (Call Stack)** — JVM автоматически использует стек вызовов: при входе в метод его фрейм помещается на стек, при выходе — снимается.
2. **Алгоритм обхода дерева/графа в глубину (DFS)** — явный стек заменяет рекурсию, сохраняя узлы для дальнейшего посещения.
3. **Вычисление выражений и разбор скобок** — алгоритм сортировочной станции (Shunting-yard), проверка корректности скобок `(){}[]`.
4. **Операция Undo/Redo** — каждое действие помещается на стек; отмена извлекает последнее действие.
5. **Навигация в браузере** — кнопка «Назад» работает со стеком посещённых страниц.
6. **Разворот строки** — помещаем символы в стек по одному, затем извлекаем в обратном порядке.

**Когда использовать очередь**

1. **Алгоритм обхода графа в ширину (BFS)** — узлы текущего уровня добавляются в хвост; обработка ведётся с головы, гарантируя уровневый обход.
2. **Планировщики задач (Task Scheduling)** — ОС и JVM используют очереди задач, обрабатывая их в порядке поступления.
3. **Паттерн Producer-Consumer** — производитель кладёт задачи в очередь, потребитель(и) берут из неё; `BlockingQueue` обеспечивает потокобезопасность.
4. **Буферизация I/O и сетевые пакеты** — входящие сообщения обрабатываются в порядке поступления.
5. **Очереди печати** — задания отправляются на принтер в порядке добавления.
6. **Rate limiting / дросселирование запросов** — запросы помещаются в очередь и обрабатываются с заданной скоростью.

**Специализированные варианты**

- **Deque (двусторонняя очередь)** — `ArrayDeque` в Java: добавление/удаление с обоих концов; можно использовать как стек и как очередь одновременно. Предпочтительнее `Stack` и `LinkedList` по производительности.
- **PriorityQueue** — элементы извлекаются не в порядке FIFO, а по приоритету (естественный порядок или `Comparator`). Используется в алгоритме Дейкстры, планировщиках с приоритетами.

### Вопрос 4. Как найти средний элемент односвязного списка за один проход? Какова временная и пространственная сложность вашего решения?

**Категория:** Алгоритмы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Find the Middle of a Linked List Using Two Pointers and its Time Complexity — Dev Genius / Medium](https://blog.devgenius.io/find-the-middle-of-a-linked-list-using-two-pointers-and-its-time-complexity-cc8f69a67d60)

Используются два указателя: «медленный» (slow) и «быстрый» (fast), оба стартуют с головы списка. На каждой итерации slow продвигается на один узел вперёд, а fast — на два. Когда fast достигает конца списка (fast == null или fast.next == null), slow указывает ровно на середину. Метод работает за один проход и использует константный объём памяти, поскольку хранит лишь два указателя. Временная сложность — O(n), пространственная — O(1).

```java
public ListNode findMiddle(ListNode head) {
    ListNode slow = head, fast = head;
    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
    }
    return slow;
}
```

Для списка с чётным числом узлов метод возвращает второй из двух средних узлов (LeetCode-семантика задачи 876).

#### Ответ от Claude

**Идея алгоритма**

Алгоритм «черепаха и заяц» использует два указателя, движущихся с разной скоростью:
- `slow` — перемещается по одному узлу за итерацию;
- `fast` — перемещается по два узла за итерацию.

Инвариант: когда `fast` прошёл расстояние `2k`, `slow` прошёл ровно `k`. Значит, в момент достижения `fast` конца списка `slow` находится ровно посередине.

**Реализация на Java**

```java
public ListNode middleNode(ListNode head) {
    ListNode slow = head;
    ListNode fast = head;
    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
    }
    return slow;
}
```

**Граничные случаи**

| Длина списка | Что возвращает slow |
|---|---|
| 1 (один узел) | единственный узел |
| 2 | второй узел |
| нечётная n | узел (n+1)/2 — истинная середина |
| чётная n | узел n/2 + 1 — «правая» середина |

Если нужен «левый» средний узел в чётном списке, условие цикла меняется на `fast.next != null && fast.next.next != null`.

**Сложность**

- Временная: O(n) — список обходится ровно один раз.
- Пространственная: O(1) — используются только два указателя, объём памяти не зависит от длины списка.

**Почему это лучше наивного подхода**

Наивный подход «сначала посчитать длину, потом пройти n/2 шагов» требует двух проходов — O(2n) по числу итераций. Алгоритм двух указателей решает задачу строго за один проход, что важно для потоков (stream), где второй проход невозможен.

### Вопрос 5. Как определить, содержит ли связный список цикл? Опишите алгоритм «двух указателей» (черепаха и заяц).

**Категория:** Алгоритмы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Tortoise and Hare Algorithm — cp-algorithms.com](https://cp-algorithms.com/others/tortoise_and_hare.html)

Алгоритм Флойда обнаруживает цикл, инициализируя два указателя (`slow` и `fast`) у головы списка. `slow` сдвигается на один узел, `fast` — на два узла за итерацию. Если цикла нет, `fast` достигнет null. Если цикл существует, `fast` «обгонит» `slow` внутри цикла и в итоге они встретятся в одном узле. Чтобы найти точку входа в цикл, `slow` сбрасывается к голове; затем оба указателя двигаются по одному шагу — их следующая встреча произойдёт в начале цикла. Сложность: O(n) по времени, O(1) по памяти.

```java
public boolean hasCycle(ListNode head) {
    ListNode slow = head, fast = head;
    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
        if (slow == fast) return true;
    }
    return false;
}
```

#### Ответ от Claude

**Алгоритм Флойда (Tortoise and Hare)**

Роберт Флойд предложил этот алгоритм в 1960-х годах. Он требует только двух указателей и не использует дополнительных структур данных (в отличие от подхода с HashSet).

**Фаза 1 — обнаружение цикла**

```java
public boolean hasCycle(ListNode head) {
    ListNode slow = head;
    ListNode fast = head;
    while (fast != null && fast.next != null) {
        slow = slow.next;       // шаг +1
        fast = fast.next.next;  // шаг +2
        if (slow == fast) {
            return true;        // указатели совпали — цикл есть
        }
    }
    return false;               // fast достиг конца — цикла нет
}
```

**Почему они обязательно встретятся?**

Пусть список состоит из линейной части длиной `F` и цикла длиной `C`. Когда `slow` входит в цикл, `fast` уже находится где-то внутри. Разрыв между ними сокращается ровно на 1 за каждую итерацию (fast сокращает дистанцию на 2, slow увеличивает на 1 — относительное сближение = 1). Значит, через не более чем `C` итераций они встретятся.

**Фаза 2 — нахождение точки входа в цикл (бонус)**

После встречи указателей сбросим `slow` к голове и будем двигать оба по одному шагу:

```java
// после обнаружения цикла: slow == fast (внутри цикла)
slow = head;
while (slow != fast) {
    slow = slow.next;
    fast = fast.next;
}
// slow == fast == начало цикла
```

Математическое обоснование: расстояние от головы до входа в цикл равно расстоянию от точки встречи до входа в цикл (при движении по циклу вперёд). Это следует из равенства `2 * dist(slow) = dist(fast)` в момент встречи.

**Сравнение с альтернативным подходом через HashSet**

| Критерий | Floyd (два указателя) | HashSet |
|---|---|---|
| Время | O(n) | O(n) |
| Память | O(1) | O(n) |
| Сложность кода | средняя | низкая |

**Сложность алгоритма Флойда**

- Временная: O(n) — в худшем случае `fast` делает не более n итераций до встречи или до конца.
- Пространственная: O(1) — только два указателя.

### Вопрос 6. Объясните принцип работы хэш-таблицы. Что такое коллизия и какие методы её разрешения существуют?

**Категория:** Алгоритмы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [3.4 Hash Tables — algs4.cs.princeton.edu](https://algs4.cs.princeton.edu/34hash/)

Хэш-таблица хранит пары ключ-значение в массиве. Хэш-функция превращает ключ в индекс массива: «мы обращаемся к парам ключ-значение через массив, выполняя арифметические операции для преобразования ключей в индексы». Идеальная хэш-функция отображала бы каждый ключ на уникальный индекс, но на практике разные ключи могут отображаться на один и тот же индекс — это называется коллизией.

Два основных метода разрешения коллизий:

1. **Метод цепочек (Separate Chaining)** — каждый слот массива хранит связный список пар. При коллизии новый элемент добавляется в список соответствующего слота. Для таблицы с M слотами и N ключами среднее число сравнений при поиске и вставке пропорционально N/M.

2. **Открытая адресация (Open Addressing, Linear Probing)** — при коллизии алгоритм последовательно проверяет следующие слоты массива до нахождения свободного. Среднее число проверок при поиске: `~1/2 * (1 + 1/(1-a))` для успешного поиска и `~1/2 * (1 + 1/(1-a)^2)` для неуспешного, где `a = N/M` — коэффициент загрузки.

#### Ответ от Claude

**Принцип работы хэш-таблицы**

Хэш-таблица — структура данных, обеспечивающая вставку, удаление и поиск за O(1) в среднем случае. Работа состоит из двух этапов:

1. **Хэширование**: хэш-функция `h(key)` отображает ключ в целочисленный индекс `[0, M-1]`, где M — размер внутреннего массива.
2. **Разрешение коллизий**: обработка ситуации, когда `h(k1) == h(k2)` при `k1 != k2`.

**Что такое коллизия**

Коллизия — ситуация, когда два различных ключа дают одинаковый хэш-индекс. Это неизбежно при конечном размере массива и потенциально бесконечном множестве ключей (принцип Дирихле / «ящичный» принцип).

**Методы разрешения коллизий**

**1. Метод цепочек (Separate Chaining)**

Каждый слот массива содержит связный список (или другую структуру: дерево, список пропусков). При коллизии новый элемент добавляется в список слота.

```
index 2 -> [("cat", 1)] -> [("act", 2)] -> null
index 5 -> [("dog", 3)] -> null
```

- Сложность поиска: O(1 + n/m) в среднем, O(n) в худшем (все ключи в одном слоте).
- Java HashMap использует этот подход: начиная с Java 8, при превышении 8 элементов в бакете связный список заменяется красно-чёрным деревом, что снижает худший случай до O(log n).

**2. Открытая адресация (Open Addressing)**

При коллизии алгоритм ищет следующий свободный слот в самой таблице. Все элементы хранятся непосредственно в массиве.

Варианты зондирования (probing):

- **Линейное зондирование (Linear Probing)**: `h(k, i) = (h(k) + i) mod M`. Простое, но вызывает «первичную кластеризацию» — длинные цепочки заполненных слотов.
- **Квадратичное зондирование (Quadratic Probing)**: `h(k, i) = (h(k) + c1*i + c2*i^2) mod M`. Снижает первичную кластеризацию, но вызывает вторичную.
- **Двойное хэширование (Double Hashing)**: `h(k, i) = (h1(k) + i * h2(k)) mod M`. Лучшее распределение, практически устраняет кластеризацию.

**3. Robin Hood Hashing**

Разновидность открытой адресации: при вставке новый элемент может «забрать» слот у элемента с меньшим смещением от идеальной позиции, что выравнивает длины цепочек зондирования.

**Коэффициент загрузки (load factor)**

`a = N / M` (число элементов / размер массива). При достижении порогового значения таблица перестраивается (rehashing) с увеличением массива. В Java HashMap порог по умолчанию = 0.75.

**Сравнение методов**

| Критерий | Separate Chaining | Open Addressing |
|---|---|---|
| Память | дополнительная (узлы списка) | только массив |
| Load factor | может быть > 1 | должен быть < 1 |
| Кэш-дружелюбность | низкая (указатели) | высокая (массив) |
| Удаление | простое | сложное (нужен маркер) |
| Худший случай | O(n) / O(log n) с деревом | O(n) при высоком a |

**Сложность операций хэш-таблицы**

| Операция | Среднее | Худший случай |
|---|---|---|
| Вставка | O(1) | O(n) |
| Поиск | O(1) | O(n) |
| Удаление | O(1) | O(n) |

Хорошая хэш-функция (равномерное распределение) и контроль коэффициента загрузки — ключевые условия для сохранения O(1) на практике.

### Вопрос 7. Реализуйте быструю сортировку (quicksort). Какова её средняя и худшая временная сложность и почему?

**Категория:** Алгоритмы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Quicksort – Algorithm, Source Code, Time Complexity (happycoders.eu)](https://www.happycoders.eu/algorithms/quicksort/)

Quicksort реализует стратегию «разделяй и властвуй»: выбирается опорный элемент (pivot), массив разбивается так, чтобы все элементы меньше pivot оказались слева, а больше — справа. Затем обе части сортируются рекурсивно.

**Средняя сложность — O(n log n).** Когда pivot делит массив примерно пополам, получается log₂(n) уровней разбиения. На каждом уровне суммарно обрабатываются все n элементов. Итог: n × log₂(n).

**Худшая сложность — O(n²).** Если pivot всегда оказывается наименьшим или наибольшим элементом (типичная ситуация для уже отсортированного массива при стратегии «крайний правый элемент»), получается n уровней с убывающим числом сравнений: n + (n−1) + (n−2) + … ≈ n²/2. На практике это приводит к `StackOverflowException` при больших входных данных.

Улучшения: выбор среднего из трёх (median-of-three) или случайного pivot снижает вероятность худшего случая. Комбинация с Insertion Sort для подмассивов размером < 48 элементов даёт прирост производительности ~15%.

#### Ответ от Claude

**Принцип работы.** Quicksort — алгоритм сортировки «разделяй и властвуй». На каждом шаге выбирается опорный элемент (pivot), массив перекраивается так, что элементы меньше pivot уходят влево, больше — вправо. После этого левая и правая части рекурсивно сортируются независимо.

**Реализация на Java (классический вариант с последним элементом как pivot):**

```java
public class QuickSort {

    public static void quickSort(int[] arr, int low, int high) {
        if (low < high) {
            int pi = partition(arr, low, high);
            quickSort(arr, low, pi - 1);
            quickSort(arr, pi + 1, high);
        }
    }

    private static int partition(int[] arr, int low, int high) {
        int pivot = arr[high]; // выбираем последний элемент как опорный
        int i = low - 1;      // индекс меньшего элемента

        for (int j = low; j < high; j++) {
            if (arr[j] < pivot) {
                i++;
                // меняем местами arr[i] и arr[j]
                int temp = arr[i];
                arr[i] = arr[j];
                arr[j] = temp;
            }
        }
        // ставим pivot на правильное место
        int temp = arr[i + 1];
        arr[i + 1] = arr[high];
        arr[high] = temp;
        return i + 1;
    }

    public static void main(String[] args) {
        int[] arr = {10, 7, 8, 9, 1, 5};
        quickSort(arr, 0, arr.length - 1);
        // результат: 1 5 7 8 9 10
    }
}
```

**Временная сложность:**

| Случай | Сложность | Причина |
|--------|-----------|---------|
| Лучший | O(n log n) | Pivot всегда делит массив ровно пополам → log n уровней × n операций на уровень |
| Средний | O(n log n) | В среднем pivot делит массив на части, близкие к равным |
| Худший | O(n²) | Pivot всегда минимальный или максимальный → n уровней, на каждом обрабатывается на 1 элемент меньше: n + (n-1) + … + 1 = n(n+1)/2 |

**Пространственная сложность:** O(log n) в среднем (глубина стека рекурсии), O(n) в худшем случае.

**Почему худший случай возникает на отсортированных данных.** Если брать крайний элемент как pivot и массив уже отсортирован, каждый шаг разбиения создаёт одну пустую часть и одну часть из n-1 элементов. Рекурсия углубляется на n уровней вместо log n.

**Как избежать худшего случая:**
- Случайный выбор pivot (рандомизированный quicksort).
- Median-of-three: pivot = медиана из первого, среднего и последнего элементов.
- В JDK (`Arrays.sort` для примитивов) используется Dual-Pivot Quicksort (два опорных элемента), что снижает среднее число сравнений.

**Quicksort НЕ является стабильной сортировкой** — равные элементы могут поменять относительный порядок.

### Вопрос 8. Как работает сортировка слиянием (merge sort)? В чём её преимущество перед quicksort по гарантиям сложности?

**Категория:** Алгоритмы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Merge Sort – GeeksforGeeks](https://www.geeksforgeeks.org/dsa/merge-sort/)

Merge sort — алгоритм «разделяй и властвуй», работающий в два этапа.

**Этап разбиения (Divide):** массив рекурсивно делится на два подмассива до тех пор, пока каждый не содержит единственный элемент (одноэлементный массив тривиально отсортирован).

**Этап слияния (Merge):** два отсортированных подмассива объединяются: элементы сравниваются попарно и записываются во временный массив в правильном порядке. Это повторяется снизу вверх до получения целого отсортированного массива.

**Временная сложность:** O(n log n) для лучшего, среднего и худшего случаев — в отличие от quicksort, у которого худший случай O(n²).

**Пространственная сложность:** O(n) — для временных массивов при слиянии.

**Преимущества перед quicksort:**
1. **Стабильность:** равные элементы сохраняют исходный относительный порядок.
2. **Гарантированная производительность:** O(n log n) в любом случае независимо от входных данных.
3. **Эффективен для связных списков:** не требует произвольного доступа к элементам.

#### Ответ от Claude

**Принцип работы.**

Merge sort рекурсивно разбивает массив пополам до единичных элементов, а затем сливает отсортированные части обратно. Ключевая операция — `merge`: слияние двух отсортированных подмассивов в один отсортированный.

**Реализация на Java:**

```java
public class MergeSort {

    public static void mergeSort(int[] arr, int left, int right) {
        if (left < right) {
            int mid = left + (right - left) / 2; // избегаем переполнения
            mergeSort(arr, left, mid);
            mergeSort(arr, mid + 1, right);
            merge(arr, left, mid, right);
        }
    }

    private static void merge(int[] arr, int left, int mid, int right) {
        int n1 = mid - left + 1;
        int n2 = right - mid;

        // временные массивы
        int[] L = new int[n1];
        int[] R = new int[n2];

        System.arraycopy(arr, left, L, 0, n1);
        System.arraycopy(arr, mid + 1, R, 0, n2);

        int i = 0, j = 0, k = left;
        while (i < n1 && j < n2) {
            // <= обеспечивает стабильность: левый элемент берётся первым при равенстве
            if (L[i] <= R[j]) {
                arr[k++] = L[i++];
            } else {
                arr[k++] = R[j++];
            }
        }
        while (i < n1) arr[k++] = L[i++];
        while (j < n2) arr[k++] = R[j++];
    }

    public static void main(String[] args) {
        int[] arr = {38, 27, 43, 3, 9, 82, 10};
        mergeSort(arr, 0, arr.length - 1);
        // результат: 3 9 10 27 38 43 82
    }
}
```

**Как работает пошагово (пример: [38, 27, 43, 3]):**

```
Разбиение:
[38, 27, 43, 3]
  [38, 27]   [43, 3]
  [38] [27]  [43] [3]

Слияние:
  [38] + [27] → [27, 38]
  [43] + [3]  → [3, 43]
  [27, 38] + [3, 43] → [3, 27, 38, 43]
```

**Временная сложность:**

| Случай | Сложность | Причина |
|--------|-----------|---------|
| Лучший | O(n log n) | log n уровней разбиения, n операций слияния на каждом |
| Средний | O(n log n) | То же — структура алгоритма не зависит от входных данных |
| Худший | O(n log n) | Даже при самых «плохих» данных: всегда log n уровней |

**Пространственная сложность:** O(n) — требуются временные массивы для слияния.

**Сравнение с quicksort:**

| Критерий | Merge sort | Quicksort |
|----------|-----------|-----------|
| Худший случай | **O(n log n)** | O(n²) |
| Стабильность | **Стабильный** | Нестабильный |
| Память | O(n) | O(log n) |
| Скорость на практике | Чуть медленнее | Быстрее (кэш-эффективность) |
| Linked lists | Отлично | Плохо |
| Внешняя сортировка | Да | Нет |

**Главное преимущество merge sort** — гарантированный O(n log n) в любом случае. Quicksort на предсортированных или специально подобранных данных деградирует до O(n²), merge sort — никогда.

**В Java на практике:** `Collections.sort()` и `Arrays.sort()` для объектов используют TimSort — гибридный алгоритм на основе merge sort, обеспечивающий стабильность. `Arrays.sort()` для примитивов использует Dual-Pivot Quicksort (нет нужды в стабильности, приоритет — скорость).

### Вопрос 9. Что такое рекурсия? Как преобразовать рекурсивный алгоритм в итеративный с помощью стека? Приведите пример.

**Категория:** Алгоритмы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Implementing Recursion with a Stack – CS3 Data Structures & Algorithms (OpenDSA, Virginia Tech)](https://opendsa-server.cs.vt.edu/ODSA/Books/CS3/html/StackRecur.html)

Вызов подпрограммы реализуется через стек: при каждом вызове на системный стек кладётся «запись активации» (activation record) с адресом возврата, параметрами и локальными переменными. Рекурсия — это частный случай, когда подпрограмма вызывает саму себя.

Любой рекурсивный алгоритм можно преобразовать в итеративный, вручную управляя стеком: вместо рекурсивного вызова состояние (параметры, локальные переменные, метка возврата) кладётся на явный стек, а обработка продолжается в цикле `while (!stack.isEmpty())`.

Пример — факториал: вместо рекурсивного вызова `factorial(n)` значения n кладутся на стек до достижения базового случая, затем снимаются и перемножаются. Для более сложных алгоритмов (например, Ханойские башни) создаётся класс-дескриптор операции, экземпляры которого помещаются на стек.

Авторы отмечают, что полная симуляция системного стека «почти всегда избыточна» — для конкретных алгоритмов обычно достаточно упрощённого стека лишь с теми данными, которые реально меняются между вызовами.

#### Ответ от Claude

**Что такое рекурсия.**

Рекурсия — это техника программирования, при которой метод вызывает сам себя для решения подзадачи того же типа, но меньшего размера. Каждый рекурсивный алгоритм содержит:
- **Базовый случай (base case)** — условие остановки, при котором рекурсия прекращается.
- **Рекурсивный случай (recursive case)** — вызов себя с уменьшенным входом, приближающий к базовому случаю.

Пример рекурсивного вычисления факториала:

```java
// Рекурсивный вариант
public static int factorialRecursive(int n) {
    if (n <= 1) return 1;           // базовый случай
    return n * factorialRecursive(n - 1); // рекурсивный случай
}
```

При каждом вызове JVM кладёт на стек вызовов (call stack) новый фрейм с локальными переменными и адресом возврата. При глубокой рекурсии это приводит к `StackOverflowError`.

**Как преобразовать рекурсию в итерацию с явным стеком.**

Системный стек JVM делает то, что мы делаем вручную: хранит состояние между «вызовами». Идея преобразования — заменить системный стек явным объектом `Deque`/`Stack` и обернуть логику в цикл.

**Общий алгоритм преобразования:**
1. Создать явный стек для хранения параметров/состояния.
2. Положить начальные параметры на стек.
3. В цикле `while (!stack.isEmpty())` снимать верхний элемент и обрабатывать.
4. Вместо рекурсивного вызова класть новые параметры на стек.

**Пример: обход бинарного дерева в глубину (inorder).**

Это классический и практически важный пример, где итеративная версия со стеком стандартно используется в собеседованиях.

```java
class TreeNode {
    int val;
    TreeNode left, right;
    TreeNode(int val) { this.val = val; }
}

// Рекурсивный вариант
public static void inorderRecursive(TreeNode root) {
    if (root == null) return;
    inorderRecursive(root.left);
    System.out.print(root.val + " ");
    inorderRecursive(root.right);
}

// Итеративный вариант с явным стеком
public static void inorderIterative(TreeNode root) {
    Deque<TreeNode> stack = new ArrayDeque<>();
    TreeNode current = root;

    while (current != null || !stack.isEmpty()) {
        // идём как можно левее, кладём узлы на стек
        while (current != null) {
            stack.push(current);
            current = current.left;
        }
        // обрабатываем узел
        current = stack.pop();
        System.out.print(current.val + " ");
        // переходим в правое поддерево
        current = current.right;
    }
}
```

**Ещё пример: вычисление n-го числа Фибоначчи итеративно.**

```java
// Рекурсивный — экспоненциальная сложность O(2^n)
public static int fibRecursive(int n) {
    if (n <= 1) return n;
    return fibRecursive(n - 1) + fibRecursive(n - 2);
}

// Итеративный — O(n) времени, O(1) памяти
public static int fibIterative(int n) {
    if (n <= 1) return n;
    int prev = 0, curr = 1;
    for (int i = 2; i <= n; i++) {
        int next = prev + curr;
        prev = curr;
        curr = next;
    }
    return curr;
}
```

**Когда и зачем преобразовывать рекурсию в итерацию:**

| Причина | Пояснение |
|---------|-----------|
| `StackOverflowError` | Глубокая рекурсия (n > ~10 000) исчерпывает стек JVM (по умолчанию 512 КБ — 1 МБ) |
| Производительность | Итеративный вариант не создаёт новых фреймов стека — нет накладных расходов на вызовы |
| Хвостовая рекурсия | Java (в отличие от Scala/Kotlin) не оптимизирует хвостовые вызовы, поэтому итерация предпочтительна |

**Важное замечание.** Не всякую рекурсию нужно заменять стеком. Если рекурсия хвостовая (результат рекурсивного вызова напрямую возвращается), она легко переписывается простым циклом без явного стека. Явный стек нужен только тогда, когда после возврата из рекурсивного вызова выполняются дополнительные действия (например, `n * factorial(n-1)` — умножение после возврата).

### Вопрос 10. В массиве целых чисел от 1 до 100 одно число пропущено. Как найти его за O(n) по времени и O(1) по памяти?

**Категория:** Алгоритмы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Find The Missing Number — InterviewBit](https://www.interviewbit.com/blog/find-the-missing-number/)

Метод арифметической суммы: вычисляется ожидаемая сумма чисел от 1 до N по формуле N×(N+1)/2, затем из неё вычитается реальная сумма элементов массива. Разность и есть пропущенное число.

```java
int MissingNo(int[] arr) {
    int n = arr.length;
    int total = (n + 1) * (n + 2) / 2;
    for (int i = 0; i < n; i++)
        total -= arr[i];
    return total;
}
```

Временная сложность — O(n), пространственная — O(1). При больших значениях n промежуточное умножение может вызвать переполнение int; в этом случае следует использовать тип long.

#### Ответ от Claude

**Суть задачи:** массив содержит 99 уникальных чисел из диапазона [1..100], одно пропущено. Нужно найти его, не используя дополнительной памяти (O(1)) за один-два прохода (O(n)).

**Подход 1 — формула суммы арифметической прогрессии**

Сумма чисел от 1 до n равна n×(n+1)/2. Вычитаем из неё реальную сумму элементов массива:

```java
public static int findMissing(int[] arr) {
    int n = 100;
    long expected = (long) n * (n + 1) / 2; // приводим к long, чтобы избежать переполнения
    long actual = 0;
    for (int x : arr) actual += x;
    return (int)(expected - actual);
}
```

Временная сложность: O(n) — один проход по массиву.
Пространственная сложность: O(1) — только две переменные.

Важно: при n=100 максимальная сумма = 5050, что не переполняет int (max ~2.1×10^9), но привычка использовать long при суммировании защищает от ошибок при масштабировании задачи.

**Подход 2 — XOR**

Используем свойства XOR: a ^ a = 0 и a ^ 0 = a. XOR всех чисел от 1 до n и XOR всех элементов массива дадут в результате пропущенное число (все присутствующие числа взаимно уничтожатся):

```java
public static int findMissingXOR(int[] arr) {
    int xorAll = 0;
    int xorArr = 0;
    for (int i = 1; i <= 100; i++) xorAll ^= i;
    for (int x : arr) xorArr ^= x;
    return xorAll ^ xorArr;
}
```

Временная сложность: O(n), пространственная: O(1).
Преимущество XOR: полностью исключает риск переполнения, поскольку не производит сложения.

**Сравнение подходов:**

| Критерий | Формула суммы | XOR |
|---|---|---|
| Время | O(n) | O(n) |
| Память | O(1) | O(1) |
| Риск переполнения | есть (решается через long) | отсутствует |
| Читаемость | выше | ниже для начинающих |

**Обоснование O(1) по памяти:** оба алгоритма используют фиксированное число переменных независимо от размера входных данных — это и есть константная память.

### Вопрос 11. Чем BFS отличается от DFS? На каких структурах данных они основаны и в каких задачах каждый из них предпочтительнее?

**Категория:** Алгоритмы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [BFS vs DFS – Difference Between Them — Guru99](https://www.guru99.com/difference-between-bfs-and-dfs.html)

BFS (обход в ширину) исследует граф или дерево «уровень за уровнем» — сначала все соседи текущего узла, затем соседи соседей. DFS (обход в глубину) уходит как можно глубже по одной ветви, прежде чем возвращаться назад.

Структуры данных: BFS использует очередь (FIFO), DFS — стек (LIFO, либо стек вызовов при рекурсии).

BFS гарантирует нахождение кратчайшего пути в невзвешенном графе и не может зациклиться на бесконечных ветвях. DFS требует меньше памяти, так как хранит только путь от корня до текущего узла. DFS подходит для топологической сортировки, поиска компонент связности, обнаружения циклов, решения головоломок с единственным решением.

#### Ответ от Claude

**Ключевое различие в стратегии обхода:**

- BFS (Breadth-First Search, обход в ширину) — посещает все узлы на текущем «уровне» (расстоянии от стартового), прежде чем перейти к следующему уровню.
- DFS (Depth-First Search, обход в глубину) — уходит максимально глубоко по одной ветви, затем возвращается (backtracking) и исследует другую.

**Используемые структуры данных:**

| Алгоритм | Структура | Принцип |
|---|---|---|
| BFS | Очередь (Queue, FIFO) | Первым добавлен — первым обработан |
| DFS | Стек (Stack, LIFO) или рекурсия | Последним добавлен — первым обработан |

**Сложность (для графа с V вершинами и E рёбрами):**

Оба алгоритма: время O(V + E), но память различается:
- BFS: O(V) в худшем случае (широкий граф — очередь хранит целый уровень)
- DFS: O(h), где h — высота дерева/глубина рекурсии (при рекурсии — стек вызовов)

**Когда предпочтителен BFS:**

1. Поиск кратчайшего пути в невзвешенном графе (BFS гарантированно находит минимальный путь).
2. Обход по уровням (level-order traversal) дерева.
3. Поиск ближайшего элемента (например, ближайший друг в социальной сети).
4. Сетевое вещание (broadcast) — обход всех достижимых узлов.
5. Когда решение заведомо находится близко к корню.

**Когда предпочтителен DFS:**

1. Топологическая сортировка (порядок зависимостей).
2. Обнаружение циклов в графе.
3. Поиск всех путей между двумя вершинами.
4. Решение головоломок и задач с возвратом (sudoku, N ферзей).
5. Когда решение находится глубоко и памяти мало.
6. Поиск компонент связности, поиск мостов и точек сочленения (алгоритмы Тарьяна/Косарайю).

**Простой пример на Java (BFS с очередью):**

```java
void bfs(int start, List<List<Integer>> adj, int n) {
    boolean[] visited = new boolean[n];
    Queue<Integer> queue = new LinkedList<>();
    visited[start] = true;
    queue.offer(start);
    while (!queue.isEmpty()) {
        int node = queue.poll();
        System.out.print(node + " ");
        for (int neighbor : adj.get(node)) {
            if (!visited[neighbor]) {
                visited[neighbor] = true;
                queue.offer(neighbor);
            }
        }
    }
}
```

**Итог:** главное правило выбора — BFS, если нужен кратчайший путь или обход по уровням; DFS, если нужна экономия памяти или исследование всех ветвей (полный перебор, топологический порядок).

### Вопрос 12. Что такое двоичное дерево поиска (BST)? Какова сложность поиска, вставки и удаления в лучшем и худшем случае? Когда BST деградирует до O(n)?

**Категория:** Алгоритмы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Binary search tree — Wikipedia](https://en.wikipedia.org/wiki/Binary_search_tree)

Двоичное дерево поиска (BST) — корневое двоичное дерево, в котором ключ каждого внутреннего узла больше всех ключей в левом поддереве и меньше всех ключей в правом поддереве. Это позволяет эффективно выполнять операции поиска, вставки и удаления.

Средняя временная сложность операций: Θ(log n). Худший случай: O(n). Дерево деградирует до O(n), когда элементы вставляются в отсортированном или почти отсортированном порядке — дерево становится похоже на связный список (скошенное/skewed дерево). Производительность зависит от порядка вставки и удаления узлов. Для предотвращения деградации используются самобалансирующиеся деревья: AVL-дерево, красно-чёрное дерево.

#### Ответ от Claude

**Определение BST:**

Двоичное дерево поиска (Binary Search Tree) — это двоичное дерево, в котором для каждого узла выполняется инвариант:
- все ключи в **левом** поддереве **меньше** ключа узла;
- все ключи в **правом** поддереве **больше** ключа узла.

Это позволяет исключать половину оставшихся узлов на каждом шаге поиска (при сбалансированном дереве).

**Сложность операций:**

| Операция | Лучший случай | Средний случай | Худший случай |
|---|---|---|---|
| Поиск (Search) | O(1)* | O(log n) | O(n) |
| Вставка (Insert) | O(log n) | O(log n) | O(n) |
| Удаление (Delete) | O(log n) | O(log n) | O(n) |

*O(1) — если искомый элемент находится в корне.

Все операции зависят от высоты дерева h: реальная сложность — O(h). При сбалансированном дереве h = O(log n), при вырожденном — h = O(n).

**Когда BST деградирует до O(n):**

BST вырождается (становится скошенным — skewed tree), когда элементы вставляются в отсортированном или обратно отсортированном порядке:

```
// Вставка: 1, 2, 3, 4, 5 — правоскошенное дерево
1
 \
  2
   \
    3
     \
      4
       \
        5
```

В этом случае дерево превращается в аналог связного списка. Каждый поиск вынужден пройти все n узлов — O(n).

**Другие причины деградации:**
- Вставка в обратном порядке (5, 4, 3, 2, 1) — левоскошенное дерево.
- Чередующиеся вставки и удаления без балансировки — постепенный перекос структуры.

**Решение — самобалансирующиеся BST:**

| Структура | Гарантированная высота | Сложность операций |
|---|---|---|
| AVL-дерево | O(log n) | O(log n) |
| Красно-чёрное дерево | O(log n) | O(log n) |
| Treap | O(log n) ожидаемая | O(log n) ожидаемая |

В Java стандартная библиотека использует красно-чёрное дерево в реализациях TreeMap и TreeSet, что гарантирует O(log n) для всех операций.

**Операция удаления — три случая:**

1. Удаляемый узел — лист: просто удаляем.
2. Удаляемый узел имеет одного потомка: заменяем его потомком.
3. Удаляемый узел имеет двух потомков: заменяем его **in-order преемником** (наименьшим элементом правого поддерева) или in-order предшественником, затем удаляем преемника.

**Итог:** BST эффективен при случайном порядке вставки (O(log n)), но без механизма балансировки уязвим к деградации до O(n) при упорядоченных данных. Для продакшн-кода всегда предпочтительны сбалансированные варианты (TreeMap/TreeSet в Java).
### Вопрос 13. Объясните разницу между обходами дерева: inorder, preorder, postorder. Какой из них для BST возвращает элементы в отсортированном порядке и почему?

**Категория:** Алгоритмы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Binary Search Tree Traversal – Inorder, Preorder, Post Order for BST (freeCodeCamp)](https://www.freecodecamp.org/news/binary-search-tree-traversal-inorder-preorder-post-order-for-bst/)

Существуют три основных обхода дерева в глубину:

- **Inorder (Left → Root → Right):** Сначала обходится левое поддерево, затем корень, затем правое поддерево. Для дерева с корнем A результат: D, B, E, A, F, C, G.
- **Preorder (Root → Left → Right):** Сначала корень, затем левое поддерево, затем правое. Результат: A, B, D, E, C, F, G.
- **Postorder (Left → Right → Root):** Сначала левое поддерево, затем правое, корень — последний. Результат: D, E, B, F, G, C, A.

Именно **inorder**-обход возвращает элементы BST в отсортированном (возрастающем) порядке, поскольку алгоритм посещает узлы последовательно по возрастанию: сначала все меньшие значения слева, затем корень, затем большие значения справа.

#### Ответ от Claude

**Ключевое свойство BST:** В бинарном дереве поиска (BST) для каждого узла выполняется инвариант: все элементы левого поддерева меньше узла, все элементы правого поддерева — больше. Именно этот инвариант определяет, какой обход даёт отсортированный результат.

**Три вида обхода в глубину (DFS):**

| Обход | Порядок посещения | Применение |
|-------|------------------|------------|
| Inorder | Левое → Узел → Правое | Извлечение отсортированных данных из BST |
| Preorder | Узел → Левое → Правое | Копирование/сериализация дерева, префиксные выражения |
| Postorder | Левое → Правое → Узел | Удаление дерева, вычисление постфиксных выражений |

**Реализация на Java (рекурсивно):**

```java
class Node {
    int data;
    Node left, right;
}

// Inorder: Left -> Root -> Right
void inorder(Node node) {
    if (node == null) return;
    inorder(node.left);
    System.out.print(node.data + " ");
    inorder(node.right);
}

// Preorder: Root -> Left -> Right
void preorder(Node node) {
    if (node == null) return;
    System.out.print(node.data + " ");
    preorder(node.left);
    preorder(node.right);
}

// Postorder: Left -> Right -> Root
void postorder(Node node) {
    if (node == null) return;
    postorder(node.left);
    postorder(node.right);
    System.out.print(node.data + " ");
}
```

**Почему inorder даёт отсортированный порядок для BST:**

Пусть BST содержит узлы 3, 9, 10, 15, 16, 52, 54, 56, 61 с корнем 52. Inorder-обход посетит: сначала всё левое поддерево (значения < 52), затем 52, затем всё правое поддерево (значения > 52). Рекурсивно это свойство применяется к каждому поддереву, поэтому результат всегда строго возрастающий: `3, 9, 10, 15, 16, 52, 54, 56, 61`.

**Сложность всех трёх обходов:**
- Время: O(n) — каждый узел посещается ровно один раз.
- Память: O(h), где h — высота дерева (глубина стека рекурсии). В худшем случае (вырожденное дерево) O(n), в лучшем (сбалансированное дерево) O(log n).

**Практическое применение:**
- **Inorder** — проверка корректности BST, извлечение отсортированного списка.
- **Preorder** — сериализация дерева (порядок вставки узлов для восстановления той же структуры).
- **Postorder** — корректное удаление дерева из памяти, вычисление выражений в постфиксной нотации.

### Вопрос 14. Что такое динамическое программирование? Объясните разницу между мемоизацией (top-down) и табуляцией (bottom-up) на примере задачи о числах Фибоначчи.

**Категория:** Алгоритмы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Dynamic programming vs memoization vs tabulation (programming.guide)](https://programming.guide/dynamic-programming-vs-memoization-vs-tabulation.html)

**Динамическое программирование (DP)** — техника решения задач рекурсивным способом при наличии перекрывающихся подзадач. Вместо повторного вычисления одних и тех же значений DP сворачивает избыточные вычисления в ориентированный ациклический граф, сокращая сложность с O(2^n) до O(n) для задачи Фибоначчи.

**Мемоизация (top-down):** рекурсивный подход с кешированием. «Оригинальная задача находится наверху в вычислении» — отсюда название top-down.

```
fib_mem(n) {
    if (mem[n] is not set)
        if (n < 2) result = n
        else result = fib_mem(n-2) + fib_mem(n-1)
        mem[n] = result
    return mem[n]
}
```

**Табуляция (bottom-up):** итеративный подход, начинающий с базовых случаев и строящий решение снизу вверх.

```
fib_tab(n) {
    mem[0] = 0
    mem[1] = 1
    for i = 2...n
        mem[i] = mem[i-2] + mem[i-1]
    return mem[n]
}
```

Табуляция предпочтительна, когда необходимо решить все подзадачи; мемоизация — когда нужны лишь некоторые из них (ленивые вычисления).

#### Ответ от Claude

**Что такое динамическое программирование:**

Динамическое программирование (DP) — метод решения задач оптимизации и подсчёта, применяемый когда задача обладает двумя свойствами:
1. **Оптимальная подструктура** — решение задачи выражается через решения подзадач меньшего размера.
2. **Перекрывающиеся подзадачи** — одни и те же подзадачи решаются многократно (в отличие от метода «разделяй и властвуй»).

**Наивная рекурсия Фибоначчи — проблема:**

```java
// Наивная рекурсия — O(2^n), катастрофически медленно
int fib(int n) {
    if (n <= 1) return n;
    return fib(n - 1) + fib(n - 2);
}
```

При вычислении `fib(5)` значение `fib(3)` вычисляется дважды, `fib(2)` — трижды. Для `fib(50)` количество вызовов превышает 10^10.

---

**Мемоизация (Top-Down DP):**

Рекурсивный подход с кешированием: перед вычислением проверяем, не был ли результат уже посчитан.

```java
// Java: мемоизация через HashMap
Map<Integer, Long> memo = new HashMap<>();

long fibMemo(int n) {
    if (n <= 1) return n;
    if (memo.containsKey(n)) return memo.get(n);  // берём из кеша
    long result = fibMemo(n - 1) + fibMemo(n - 2);
    memo.put(n, result);  // сохраняем в кеш
    return result;
}
```

- **Сложность по времени:** O(n) — каждое значение вычисляется ровно один раз.
- **Сложность по памяти:** O(n) — кеш + стек рекурсии глубиной n.
- **Риск:** StackOverflowError при больших n из-за глубокой рекурсии.

---

**Табуляция (Bottom-Up DP):**

Итеративный подход: заполняем таблицу начиная с базовых случаев, поднимаясь к целевому значению.

```java
// Java: табуляция через массив
long fibTab(int n) {
    if (n <= 1) return n;
    long[] dp = new long[n + 1];
    dp[0] = 0;
    dp[1] = 1;
    for (int i = 2; i <= n; i++) {
        dp[i] = dp[i - 1] + dp[i - 2];
    }
    return dp[n];
}

// Оптимизация памяти до O(1):
long fibOptimized(int n) {
    if (n <= 1) return n;
    long prev2 = 0, prev1 = 1;
    for (int i = 2; i <= n; i++) {
        long curr = prev1 + prev2;
        prev2 = prev1;
        prev1 = curr;
    }
    return prev1;
}
```

- **Сложность по времени:** O(n).
- **Сложность по памяти:** O(n) для таблицы, или O(1) при оптимизации (хранить только два последних значения).
- **Нет риска StackOverflow** — полностью итеративный.

---

**Сравнение подходов:**

| Критерий | Мемоизация (top-down) | Табуляция (bottom-up) |
|----------|----------------------|----------------------|
| Стиль | Рекурсивный | Итеративный |
| Простота реализации | Проще (следует структуре задачи) | Требует продумать порядок заполнения |
| Риск StackOverflow | Есть при больших n | Отсутствует |
| Скорость на практике | Медленнее (overhead вызовов) | Быстрее (нет overhead рекурсии) |
| Когда вычисляет | Лениво (только нужные подзадачи) | Жадно (все подзадачи) |
| Оптимизация памяти | Затруднена | Часто можно сократить до O(1) |
| Временная сложность | O(n) | O(n) |
| Пространственная сложность | O(n) | O(n) или O(1) |

**Вывод:** оба подхода дают одинаковую асимптотику O(n), но табуляция обычно предпочтительнее на практике из-за отсутствия накладных расходов рекурсии и возможности оптимизировать память.

### Вопрос 15. Как найти все пары элементов в массиве, сумма которых равна заданному числу? Предложите решение за O(n) с использованием хэш-таблицы.

**Категория:** Алгоритмы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [2 Sum – All distinct pairs with given sum (GeeksforGeeks)](https://www.geeksforgeeks.org/dsa/print-all-pairs-with-given-sum/)

Подход с HashMap отслеживает частоту встреченных элементов. Для каждого элемента проверяется наличие дополнения (target - element) в карте.

Ключевая логика: «Обходим все элементы и для каждого arr[i] проверяем, существует ли дополнение (target - arr[i]) уже в map; если существует — пара найдена».

Для избежания дубликатов:
- Если дополнение совпадает с элементом: включаем пару только если этот номер встречался ровно один раз.
- Если они различаются: включаем пару только если дополнение было встречено ранее, а текущий элемент ещё не обрабатывался.

```java
static List<List<Integer>> distinctPairs(int[] arr, int target) {
    List<List<Integer>> res = new ArrayList<>();
    Map<Integer, Integer> freq = new HashMap<>();

    for (int i = 0; i < arr.length; i++) {
        int complement = target - arr[i];

        if (complement == arr[i]) {
            if (freq.getOrDefault(complement, 0) == 1)
                res.add(Arrays.asList(arr[i], arr[i]));
        } else if (freq.getOrDefault(complement, 0) > 0
                   && freq.getOrDefault(arr[i], 0) == 0) {
            int first = Math.min(arr[i], complement);
            int second = Math.max(arr[i], complement);
            res.add(Arrays.asList(first, second));
        }
        freq.put(arr[i], freq.getOrDefault(arr[i], 0) + 1);
    }
    return res;
}
```

- **Временная сложность:** O(n) — один проход по массиву с O(1) операциями HashMap.
- **Пространственная сложность:** O(n) — хранение частотной карты.

#### Ответ от Claude

**Постановка задачи:** Дан массив `arr[]` и целевое число `target`. Найти все уникальные пары `(a, b)` такие, что `a + b == target`.

**Сравнение подходов:**

| Подход | Временная сложность | Пространственная сложность |
|--------|--------------------|-----------------------------|
| Грубая сила (два вложенных цикла) | O(n^2) | O(1) |
| Сортировка + два указателя | O(n log n) | O(1) |
| Хэш-таблица | O(n) | O(n) |

**Решение 1 — простой поиск первой пары через HashSet:**

```java
// Возвращает true, если хотя бы одна пара существует
boolean hasPairWithSum(int[] arr, int target) {
    Set<Integer> seen = new HashSet<>();
    for (int num : arr) {
        int complement = target - num;
        if (seen.contains(complement)) {
            return true; // найдена пара (complement, num)
        }
        seen.add(num);
    }
    return false;
}
```

**Принцип:** для каждого элемента `num` проверяем, был ли уже встречен `target - num`. Если да — пара найдена. Если нет — добавляем `num` в множество.

---

**Решение 2 — все уникальные пары через HashMap (обработка дубликатов):**

```java
import java.util.*;

List<int[]> findAllPairs(int[] arr, int target) {
    List<int[]> result = new ArrayList<>();
    Map<Integer, Integer> freq = new HashMap<>();

    for (int num : arr) {
        int complement = target - num;

        if (complement == num) {
            // Пара вида (x, x): нужно ровно два экземпляра x
            if (freq.getOrDefault(num, 0) == 1) {
                result.add(new int[]{num, num});
            }
        } else {
            // Обычная пара: добавляем, только когда видим второй элемент
            if (freq.getOrDefault(complement, 0) > 0
                    && freq.getOrDefault(num, 0) == 0) {
                result.add(new int[]{Math.min(num, complement),
                                     Math.max(num, complement)});
            }
        }
        freq.merge(num, 1, Integer::sum);
    }
    return result;
}
```

**Пример работы:**

```
arr = [1, 5, 3, 2, 4, 1, 3], target = 6

Шаг 1: num=1, complement=5, freq={} -> добавляем 1. freq={1:1}
Шаг 2: num=5, complement=1, freq={1:1} -> freq[1]=1>0, freq[5]=0 -> пара (1,5). freq={1:1,5:1}
Шаг 3: num=3, complement=3, freq={1:1,5:1} -> freq[3]=0, не равно 1 -> пропуск. freq={1:1,5:1,3:1}
Шаг 4: num=2, complement=4, freq={...} -> freq[4]=0 -> пропуск. freq={...,2:1}
Шаг 5: num=4, complement=2, freq={...,2:1} -> freq[2]=1>0, freq[4]=0 -> пара (2,4). freq={...,4:1}
Шаг 6: num=1 (дубль), freq[1]=1, это не первое вхождение -> пропуск. freq={...,1:2}
Шаг 7: num=3 (дубль), complement=3, freq[3]=1 -> равно 1 -> пара (3,3). freq={...,3:2}

Результат: [(1,5), (2,4), (3,3)]
```

---

**Важные нюансы:**

1. **Дубликаты в массиве:** наивный HashSet даст неверный результат для пары `(3,3)` из `[3,3,7]` — нужен HashMap с частотами.
2. **Порядок добавления:** элемент добавляется в карту ПОСЛЕ проверки, чтобы не использовать один элемент дважды для образования пары с самим собой.
3. **Отрицательные числа:** алгоритм корректно работает с отрицательными числами без изменений.

**Сложность итогового решения:**
- **Время:** O(n) — один линейный проход, все операции HashMap — O(1) амортизированно.
- **Память:** O(n) — в худшем случае все элементы уникальны и попадают в карту.

### Вопрос 16. Как реализовать обход графа в ширину (BFS) для поиска кратчайшего пути в невзвешенном графе? Какова сложность по времени и памяти?

**Категория:** Алгоритмы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Breadth First Search — Algorithms for Competitive Programming](https://cp-algorithms.com/graph/breadth-first-search.html)

BFS — алгоритм обхода графа, который исследует вершины уровень за уровнем. Ключевое свойство: «путь, найденный BFS до любой вершины, является кратчайшим путём до этой вершины, то есть путём с наименьшим числом рёбер в невзвешенном графе».

Алгоритм использует очередь и три вспомогательных массива:
- `used[]` — отметка посещённых вершин;
- `d[]` — расстояние от источника до каждой вершины;
- `p[]` — массив предков для восстановления пути.

Шаги:
1. Инициализировать очередь, добавить исходную вершину `s`, пометить `used[s] = true`, установить `d[s] = 0`.
2. Пока очередь не пуста: извлечь вершину `v` из начала очереди.
3. Для каждого непосещённого соседа `u` вершины `v`: пометить `used[u] = true`, установить `d[u] = d[v] + 1`, записать `p[u] = v`, добавить `u` в очередь.
4. Для восстановления пути — обратный проход по массиву предков от цели к источнику.

Сложность по времени: O(n + m), где n — количество вершин, m — количество рёбер.
Сложность по памяти: O(n) — для массивов `used`, `d`, `p` и самой очереди.

#### Ответ от Claude

**Идея алгоритма**

BFS (обход в ширину) гарантирует нахождение кратчайшего пути в невзвешенном графе, потому что он исследует вершины строго в порядке возрастания расстояния от источника: сначала все вершины на расстоянии 1, затем на расстоянии 2 и т. д. Первый раз, когда BFS достигает целевой вершины, это происходит по кратчайшему маршруту — пропустить уровень невозможно по самой природе обхода через очередь (FIFO).

**Реализация на Java**

```java
import java.util.*;

public class BfsShortestPath {

    // graph — список смежности: graph.get(v) содержит соседей вершины v
    public static List<Integer> shortestPath(List<List<Integer>> graph, int src, int dst) {
        int n = graph.size();
        boolean[] visited = new boolean[n];
        int[]     parent  = new int[n];
        int[]     dist    = new int[n];

        Arrays.fill(parent, -1);
        Arrays.fill(dist,   Integer.MAX_VALUE);

        Queue<Integer> queue = new LinkedList<>();
        visited[src] = true;
        dist[src]    = 0;
        queue.add(src);

        while (!queue.isEmpty()) {
            int v = queue.poll();
            if (v == dst) break;                   // нашли цель — ранний выход
            for (int u : graph.get(v)) {
                if (!visited[u]) {
                    visited[u] = true;
                    dist[u]    = dist[v] + 1;
                    parent[u]  = v;
                    queue.add(u);
                }
            }
        }

        if (!visited[dst]) return Collections.emptyList(); // пути нет

        // Восстановление пути обратным проходом по массиву предков
        LinkedList<Integer> path = new LinkedList<>();
        for (int cur = dst; cur != -1; cur = parent[cur]) {
            path.addFirst(cur);
        }
        return path;
    }
}
```

**Почему BFS, а не DFS?**

DFS уходит вглубь по одному пути и может найти длинный маршрут раньше короткого. BFS «расширяет фронт» равномерно, поэтому первое достижение цели всегда соответствует минимальному числу рёбер.

**Сложность**

| Ресурс | Сложность | Объяснение |
|---|---|---|
| Время | O(V + E) | Каждая вершина добавляется в очередь и извлекается из неё ровно один раз; каждое ребро просматривается ровно один раз |
| Память | O(V) | Очередь в худшем случае (полный граф) хранит O(V) вершин; массивы `visited`, `dist`, `parent` — тоже O(V) |

**Представление графа** влияет на сложность:
- Список смежности: O(V + E) по времени и O(V + E) по памяти на хранение самого графа.
- Матрица смежности: O(V²) по времени, т. к. проверка всех соседей вершины занимает O(V).

**Важные детали реализации**
- Помечать вершину как посещённую нужно **в момент добавления** в очередь, а не при извлечении — иначе одна и та же вершина может быть добавлена несколько раз, что ломает сложность.
- Для невзвешенного орграфа алгоритм работает идентично; для взвешенного графа нужен алгоритм Дейкстры.

### Вопрос 17. Как определить наличие цикла в ориентированном графе с помощью DFS? Опишите алгоритм с использованием цветовой разметки вершин.

**Категория:** Алгоритмы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Detect Cycle in a directed graph using colors — GeeksforGeeks](https://www.geeksforgeeks.org/dsa/detect-cycle-direct-graph-using-colors/)

Алгоритм использует DFS с тремя цветами для отслеживания состояния каждой вершины:

- **Белый (0)** — вершина ещё не посещена.
- **Серый (1)** — вершина находится в текущем стеке рекурсии (обрабатывается прямо сейчас).
- **Чёрный (2)** — вершина полностью обработана (все её потомки исследованы).

Шаги алгоритма:
1. Инициализировать массив цветов: все вершины — белые.
2. Для каждой белой вершины запустить DFS.
3. При входе в вершину пометить её серым.
4. Рекурсивно обойти всех соседей. Если среди соседей встречается серая вершина — найдено обратное ребро, а значит, есть цикл.
5. По завершении обработки всех соседей пометить вершину чёрной.

Цикл существует тогда и только тогда, когда DFS находит ребро, ведущее к серой вершине (back edge).

Сложность по времени: O(V + E).
Сложность по памяти: O(V) — массив цветов и стек рекурсии.

#### Ответ от Claude

**Зачем нужны три цвета, а не два?**

В неориентированном графе для поиска цикла достаточно булевого массива `visited`. Но в **ориентированном** графе это недостаточно: ребро `(u, v)`, где `v` уже посещена, не обязательно образует цикл — `v` могла быть посещена в другой ветви DFS (это называется *cross edge* или *forward edge*). Цикл образует только *back edge* — ребро к вершине, которая всё ещё находится в текущем стеке рекурсии. Именно для различения «в стеке» (серый) и «уже завершён» (чёрный) нужен третий цвет.

**Алгоритм пошагово**

```
COLOR = {WHITE=0, GRAY=1, BLACK=2}

hasCycle(graph):
    color[v] = WHITE  для всех v
    для каждой вершины v:
        если color[v] == WHITE:
            если dfs(v, color, graph) == true:
                вернуть true
    вернуть false

dfs(v, color, graph):
    color[v] = GRAY                     // входим в вершину
    для каждого соседа u вершины v:
        если color[u] == GRAY:
            вернуть true                // back edge → цикл!
        если color[u] == WHITE:
            если dfs(u, color, graph):
                вернуть true
    color[v] = BLACK                    // покидаем вершину
    вернуть false
```

**Реализация на Java**

```java
import java.util.*;

public class DirectedCycleDetector {

    private static final int WHITE = 0, GRAY = 1, BLACK = 2;

    public static boolean hasCycle(List<List<Integer>> graph) {
        int n = graph.size();
        int[] color = new int[n];   // все WHITE по умолчанию
        for (int v = 0; v < n; v++) {
            if (color[v] == WHITE && dfs(v, color, graph)) {
                return true;
            }
        }
        return false;
    }

    private static boolean dfs(int v, int[] color, List<List<Integer>> graph) {
        color[v] = GRAY;
        for (int u : graph.get(v)) {
            if (color[u] == GRAY)  return true;   // back edge → цикл
            if (color[u] == WHITE && dfs(u, color, graph)) return true;
        }
        color[v] = BLACK;
        return false;
    }
}
```

**Типы рёбер в DFS ориентированного графа**

| Тип ребра | Куда ведёт | Цикл? |
|---|---|---|
| Tree edge | в WHITE вершину | нет |
| Back edge | в GRAY вершину | **да** |
| Forward edge | в BLACK вершину-потомка | нет |
| Cross edge | в BLACK вершину из другой ветви | нет |

**Практическое применение**
- Проверка топологической сортировки (граф DAG — ациклический).
- Обнаружение циклических зависимостей в сборочных системах (Maven, Gradle).
- Обнаружение дедлоков в системах управления ресурсами.

**Сложность**

| Ресурс | Сложность | Объяснение |
|---|---|---|
| Время | O(V + E) | Каждая вершина получает цвет GRAY и BLACK ровно по одному разу; каждое ребро проверяется один раз |
| Память | O(V) | Массив цветов + стек рекурсии глубиной до V |

**Замечание о стеке рекурсии:** в худшем случае (граф-цепочка) глубина рекурсии достигает V, что может привести к StackOverflowError в Java для очень больших графов. В таком случае рекурсию нужно заменить явным стеком (`Deque<Integer>`).

### Вопрос 18. Как решить задачу о наибольшей общей подпоследовательности (LCS) с помощью динамического программирования? Какова временная и пространственная сложность решения?

**Категория:** Алгоритмы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Longest Common Subsequence (LCS) — GeeksforGeeks](https://www.geeksforgeeks.org/dsa/longest-common-subsequence-dp-4/)

Задача LCS состоит в нахождении наибольшей подпоследовательности, общей для двух строк. Подпоследовательность — это последовательность символов, которая встречается в той же относительной позиции, что и в исходной строке, но не обязана быть непрерывной.

Рекуррентное соотношение:
- Если символы совпадают: `dp[i][j] = 1 + dp[i-1][j-1]`
- Если не совпадают: `dp[i][j] = max(dp[i-1][j], dp[i][j-1])`
- Базовый случай: если одна из строк пуста — LCS равна 0.

Алгоритм строит таблицу размером (m+1) × (n+1), где m и n — длины строк. Таблица заполняется итеративно слева направо, сверху вниз. Ответ находится в ячейке `dp[m][n]`.

Временная сложность: O(m × n).
Пространственная сложность: O(m × n) для стандартного подхода, O(n) при оптимизации до двух строк.

Пример: LCS("AGGTAB", "GXTXAYB") = 4, подпоследовательность "GTAB".

#### Ответ от Claude

**Постановка задачи**

Дано две строки `X` длиной `m` и `Y` длиной `n`. Найти длину наибольшей общей подпоследовательности (LCS). Подпоследовательность сохраняет относительный порядок символов, но не требует их смежности. Например, для "ABCBDAB" и "BDCABA" LCS = "BCBA" длиной 4.

**Рекуррентное соотношение**

Обозначим `dp[i][j]` — длину LCS для префиксов `X[0..i-1]` и `Y[0..j-1]`.

```
dp[i][j] = 0                             если i = 0 или j = 0 (базовый случай)
dp[i][j] = dp[i-1][j-1] + 1             если X[i-1] == Y[j-1]
dp[i][j] = max(dp[i-1][j], dp[i][j-1]) если X[i-1] != Y[j-1]
```

Интуиция: если последние символы совпадают — они входят в LCS, задача сводится к LCS оставшихся префиксов. Если не совпадают — отбрасываем последний символ одной из строк и берём лучший результат.

**Реализация на Java**

```java
public class LCS {

    // Возвращает длину LCS
    public static int lcsLength(String x, String y) {
        int m = x.length(), n = y.length();
        int[][] dp = new int[m + 1][n + 1];
        // dp[0][*] и dp[*][0] уже равны 0 (Java инициализирует int[] нулями)

        for (int i = 1; i <= m; i++) {
            for (int j = 1; j <= n; j++) {
                if (x.charAt(i - 1) == y.charAt(j - 1)) {
                    dp[i][j] = dp[i - 1][j - 1] + 1;
                } else {
                    dp[i][j] = Math.max(dp[i - 1][j], dp[i][j - 1]);
                }
            }
        }
        return dp[m][n];
    }

    // Восстановление самой подпоследовательности
    public static String lcsString(String x, String y) {
        int m = x.length(), n = y.length();
        int[][] dp = new int[m + 1][n + 1];
        for (int i = 1; i <= m; i++) {
            for (int j = 1; j <= n; j++) {
                if (x.charAt(i - 1) == y.charAt(j - 1)) dp[i][j] = dp[i-1][j-1] + 1;
                else dp[i][j] = Math.max(dp[i-1][j], dp[i][j-1]);
            }
        }
        // Обратный проход для восстановления строки
        StringBuilder sb = new StringBuilder();
        int i = m, j = n;
        while (i > 0 && j > 0) {
            if (x.charAt(i - 1) == y.charAt(j - 1)) {
                sb.append(x.charAt(i - 1));
                i--; j--;
            } else if (dp[i - 1][j] > dp[i][j - 1]) {
                i--;
            } else {
                j--;
            }
        }
        return sb.reverse().toString();
    }
}
```

**Пример заполнения таблицы dp**

Для X = "ABCB", Y = "BCB":

```
     ""  B  C  B
""  [ 0  0  0  0 ]
A   [ 0  0  0  0 ]
B   [ 0  1  1  1 ]
C   [ 0  1  2  2 ]
B   [ 0  1  2  3 ]
```

LCS = "BCB", длина 3.

**Сложность**

| Вариант реализации | Время | Память |
|---|---|---|
| Наивная рекурсия (без мемоизации) | O(2^min(m,n)) | O(min(m,n)) — стек |
| Мемоизация (top-down DP) | O(m × n) | O(m × n) — таблица + стек |
| Табуляция (bottom-up DP) | O(m × n) | O(m × n) — таблица |
| Оптимизация двумя строками | O(m × n) | O(min(m, n)) |

**Оптимизация памяти до O(min(m, n))**

Для каждой строки `i` значения `dp[i][j]` зависят только от строки `i-1`. Поэтому достаточно хранить лишь две строки. При этом нельзя восстановить саму подпоследовательность без полной таблицы — для восстановления строки при ограниченной памяти используется алгоритм Хиршберга (Hirschberg's algorithm), который работает за O(m × n) по времени и O(min(m, n)) по памяти.

```java
// Оптимизированная версия: только длина LCS, O(min(m,n)) памяти
public static int lcsOptimized(String x, String y) {
    if (x.length() < y.length()) { String t = x; x = y; y = t; } // y — короткая
    int n = y.length();
    int[] prev = new int[n + 1];
    int[] curr = new int[n + 1];

    for (int i = 1; i <= x.length(); i++) {
        for (int j = 1; j <= n; j++) {
            if (x.charAt(i - 1) == y.charAt(j - 1)) curr[j] = prev[j - 1] + 1;
            else curr[j] = Math.max(prev[j], curr[j - 1]);
        }
        int[] tmp = prev; prev = curr; curr = tmp;
        Arrays.fill(curr, 0);
    }
    return prev[n];
}
```

**Практические применения LCS**
- `diff` в системах контроля версий (Git) — сравнение версий файлов.
- Биоинформатика — выравнивание последовательностей ДНК/белков.
- Проверка плагиата — поиск общих фрагментов текстов.
- Инструменты слияния (merge) в редакторах и IDE.

### Вопрос 19. В чём разница между ArrayList и LinkedList в Java с точки зрения Big-O сложности операций доступа по индексу, вставки в начало и вставки в конец?

**Категория:** Алгоритмы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [List Implementations (The Java™ Tutorials > Collections > Implementations)](https://docs.oracle.com/javase/tutorial/collections/implementations/list.html)

Доступ по позиции требует линейного времени в LinkedList и константного времени в ArrayList. Если вы часто добавляете элементы в начало списка или итерируетесь по нему для удаления элементов изнутри, лучше рассмотреть использование LinkedList. Эти операции требуют константного времени в LinkedList и линейного в ArrayList. Однако вы платите большую цену в производительности. Позиционный доступ требует линейного времени в LinkedList. Кроме того, константный множитель у LinkedList значительно хуже. Если вы думаете использовать LinkedList — измерьте производительность вашего приложения с обоими вариантами перед принятием решения; ArrayList обычно быстрее.

#### Ответ от Claude

Сравнение по ключевым операциям:

| Операция | ArrayList | LinkedList |
|---|---|---|
| Доступ по индексу `get(i)` | **O(1)** | **O(n)** |
| Вставка в начало `add(0, e)` | **O(n)** | **O(1)** |
| Вставка в конец `add(e)` | **O(1) амортизированно** | **O(1)** |

**Доступ по индексу:**
- ArrayList хранит данные в массиве, поэтому `get(i)` — прямое обращение по смещению в памяти, O(1).
- LinkedList — двусвязный список: чтобы добраться до i-го элемента, нужно пройти по цепочке ссылок от head или tail, O(n).

**Вставка в начало:**
- ArrayList требует сдвига всех существующих элементов на одну позицию вправо (`System.arraycopy`), что занимает O(n).
- LinkedList просто создаёт новый узел и переставляет ссылку head, O(1).

**Вставка в конец:**
- ArrayList: в большинстве случаев O(1) — просто записывается в следующую свободную ячейку массива. Когда массив заполнен, происходит расширение (создаётся новый массив вдвое больше и копируются все элементы), что O(n), но амортизированно по всем n операциям это O(1) на операцию.
- LinkedList хранит ссылку на tail, поэтому вставка в конец — всегда O(1) без амортизации.

**Практическая рекомендация:**
На практике ArrayList почти всегда предпочтительнее LinkedList из-за лучшей локальности кэша CPU. LinkedList имеет гораздо худший константный множитель: каждый узел содержит объект-обёртку с двумя ссылками (prev/next), что создаёт значительный overhead по памяти и приводит к промахам кэша. Oracle прямо предупреждает: «LinkedList is usually slower» — сначала измерьте, потом выбирайте.

### Вопрос 20. Как реализовать LRU-кэш (Least Recently Used) с операциями get и put за O(1)? Какие структуры данных для этого используются?

**Категория:** Алгоритмы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Building a True O(1) LRU Cache in Java Using a HashMap and a Doubly Linked List](https://medium.com/@abhishek.bisht1/building-a-true-o-1-lru-cache-in-java-using-a-hashmap-and-a-doubly-linked-list-ed65d0af5374)

Наивный подход с HashMap + обычным LinkedList не работает для O(1), потому что удаление элемента из LinkedList по значению требует O(n) обхода. Решение требует «прямого доступа к узлам (а не только к значениям)» и «вставки и удаления за константное время». Архитектура из двух структур: HashMap<Integer, Node> отображает ключи кэша непосредственно на узлы связного списка, устраняя необходимость обхода. Кастомный двусвязный список поддерживает порядок использования: узлы MRU (Most Recently Used) ближе к началу, LRU-узлы ближе к концу. Каждый узел хранит ключ (нужен при вытеснении), значение и двунаправленные указатели. Фиктивные sentinel-узлы head и tail устраняют граничные случаи. get(key): извлекает значение и перемещает узел в начало через moveToFront() — O(1). put(key, value): обновляет или вставляет; при превышении ёмкости вытесняет tail.prev (LRU-узел) — O(1). Прямые ссылки на узлы через HashMap устраняют стоимость O(n) обхода.

#### Ответ от Claude

**Суть задачи:**
LRU-кэш вытесняет наименее недавно использованный элемент при переполнении. Нужны O(1) для `get` (чтение) и `put` (запись/обновление).

**Почему нужны две структуры данных:**

Одного HashMap недостаточно — он не отслеживает порядок использования. Одного двусвязного списка недостаточно — поиск по ключу занимает O(n).

Решение: **HashMap + двусвязный список** используются совместно:

- **HashMap<Key, Node>** — даёт O(1) доступ к узлу по ключу;
- **Двусвязный список** — поддерживает порядок: MRU-узел у head, LRU-узел у tail; O(1) вставка и удаление за счёт двух указателей (prev/next) в каждом узле.

**Ключевой инсайт:** HashMap хранит не значение, а ссылку на узел списка. Это позволяет:
1. Найти узел за O(1) через HashMap;
2. Удалить/переместить узел за O(1) через prev/next без обхода.

**Схема реализации на Java:**

```java
class LRUCache {
    private final int capacity;
    private final Map<Integer, Node> map = new HashMap<>();
    private final Node head = new Node(0, 0); // sentinel
    private final Node tail = new Node(0, 0); // sentinel

    LRUCache(int capacity) {
        this.capacity = capacity;
        head.next = tail;
        tail.prev = head;
    }

    public int get(int key) {
        if (!map.containsKey(key)) return -1;
        Node node = map.get(key);
        moveToFront(node); // O(1)
        return node.value;
    }

    public void put(int key, int value) {
        if (map.containsKey(key)) {
            Node node = map.get(key);
            node.value = value;
            moveToFront(node); // O(1)
        } else {
            if (map.size() == capacity) {
                Node lru = tail.prev;
                removeNode(lru); // O(1)
                map.remove(lru.key);
            }
            Node node = new Node(key, value);
            addToFront(node); // O(1)
            map.put(key, node);
        }
    }
}
```

**Альтернатива через LinkedHashMap:**
В Java можно использовать `LinkedHashMap` в режиме access-order (третий параметр конструктора `true`) с переопределением `removeEldestEntry`. Это проще в коде, но менее гибко:

```java
class LRUCache extends LinkedHashMap<Integer, Integer> {
    private final int capacity;
    LRUCache(int capacity) {
        super(capacity, 0.75f, true); // accessOrder = true
        this.capacity = capacity;
    }
    @Override
    protected boolean removeEldestEntry(Map.Entry<Integer, Integer> eldest) {
        return size() > capacity;
    }
}
```

`LinkedHashMap` внутри сам реализован на основе HashMap + двусвязного списка — та же идея, но готовая из JDK.

**Итог:** O(1) для обеих операций достигается за счёт того, что HashMap хранит прямые ссылки на узлы двусвязного списка, а двусвязный список обеспечивает O(1) удаление и перемещение узла без обхода всей структуры.
