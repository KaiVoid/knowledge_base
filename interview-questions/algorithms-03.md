### Вопрос 41. Что такое куча (heap) и как она устроена внутри? Какова временная сложность вставки, извлечения минимума/максимума и построения кучи из n элементов (heapify)? Почему build-heap работает за O(n), а не за O(n log n)?

**Категория:** Алгоритмы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Heap (data structure) — Wikipedia](https://en.wikipedia.org/wiki/Heap_(data_structure))

Куча — это древовидная структура данных, реализованная в виде массива, где отношения родитель-потомок определяются арифметически по индексам. Для бинарной кучи с корнем в индексе 0: левый потомок узла `i` — `2i+1`, правый — `2i+2`, родитель — `⌊(i−1)/2⌋`.

Куча всегда является полным бинарным деревом, поэтому она хранится в массиве компактно — указатели не нужны.

**Временная сложность операций:**

| Операция | Среднее | Худший случай |
|---|---|---|
| Вставка | O(1) | O(log n) |
| Find-min/Find-max | O(1) | O(1) |
| Извлечение min/max | O(log n) | O(log n) |
| Build-heap | — | O(n) |
| Поиск произвольного элемента | — | O(n) |

**Почему build-heap — O(n), а не O(n log n)**

Алгоритм Флойда строит кучу снизу вверх. Ключевое наблюдение: на высоте `h` находится примерно `n / 2^(h+1)` узлов, и каждый из них требует не более `O(h)` работы при просеивании вниз.

Суммарная стоимость:

```
T(n) = sum_{h=0}^{log n} (n / 2^(h+1)) * h
     = (n/2) * sum_{h=0}^{log n} h / 2^h
```

Ряд `∑ h/2^h` сходится к 2, поэтому `T(n) = O(n)`.

Для сравнения, при n последовательных вставках каждый элемент просеивается вверх до высоты `O(log n)`, что даёт суммарно O(n log n). Алгоритм Флойда экономит работу за счёт того, что большинство узлов находятся на нижних уровнях дерева и требуют малого числа перестановок.

#### Ответ от Claude

**Внутреннее устройство**

Бинарная куча — полное бинарное дерево (все уровни заполнены, кроме последнего, заполненного слева направо), хранящееся в массиве без явных указателей. Для узла с индексом `i` (нумерация с 0):
- Левый потомок: `2i + 1`
- Правый потомок: `2i + 2`
- Родитель: `⌊(i - 1) / 2⌋`

**Свойство кучи (heap property):**
- Min-heap: `A[parent] <= A[child]` для всех узлов — минимум всегда в корне.
- Max-heap: `A[parent] >= A[child]` — максимум в корне.

**Операции и их сложность**

*Вставка — O(log n):*
Новый элемент помещается в конец массива (конец последнего уровня), затем «всплывает» вверх (sift-up / bubble-up): элемент сравнивается с родителем и меняется с ним местами, пока свойство кучи не восстановлено. Высота дерева — `⌊log₂ n⌋`, поэтому максимум `O(log n)` перестановок.

*Извлечение min/max — O(log n):*
Корень (минимум или максимум) сохраняется. На его место помещается последний элемент массива, после чего он «тонет» вниз (sift-down / heapify-down): сравнивается с меньшим (в min-heap) из потомков и меняется местами до восстановления свойства. Также O(log n) перестановок по высоте дерева.

*Просмотр min/max — O(1):*
Корень всегда доступен как `A[0]`.

**Алгоритм Флойда (build-heap) — O(n)**

Наивный подход: вставить все n элементов по одному. Каждая вставка — O(log n), итого O(n log n).

Алгоритм Флойда строит кучу за O(n):
1. Взять исходный массив как есть (форма дерева уже правильная — полное дерево).
2. Начиная с последнего внутреннего узла (индекс `⌊n/2⌋ - 1`) и двигаясь к корню, применять sift-down к каждому узлу.

Листья (индексы от `⌊n/2⌋` до `n-1`) — уже тривиальные кучи, их пропускаем.

**Почему O(n), а не O(n log n):**

На высоте `h` дерева количество узлов — примерно `n / 2^(h+1)`. Стоимость sift-down для одного узла на высоте `h` — O(h). Суммарная работа:

```
T(n) = sum_{h=0}^{⌊log n⌋} ⌈n / 2^(h+1)⌉ * O(h)
     = O(n * sum_{h=0}^{∞} h / 2^h)
     = O(n * 2)   // ряд sum h/2^h сходится к 2
     = O(n)
```

Интуитивно: листьев ~n/2, и им не нужна никакая работа. Узлов на высоте 1 ~n/4, каждый делает 1 перестановку. Узлов на высоте 2 ~n/8, каждый делает 2 перестановки. Суммарная работа сходится к O(n), потому что большинство узлов дерева находятся на нижних уровнях и требуют минимальной работы при sift-down.

**Применение в Java**

В Java куча реализована в `java.util.PriorityQueue` (min-heap по умолчанию). `Collections.sort()` использует TimSort, но `Arrays.sort()` для примитивов использует двойной quicksort. Для алгоритма heap-sort используется max-heap, построенная алгоритмом Флойда.

### Вопрос 42. Как с помощью двух куч (min-heap и max-heap) поддерживать медиану потока чисел в реальном времени с O(log n) на вставку и O(1) на получение медианы? Опишите инвариант и операцию балансировки.

**Категория:** Алгоритмы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Median of a Data Stream with Two Heaps in Java, alexanderobregon.substack.com](https://alexanderobregon.substack.com/p/median-of-a-data-stream-with-two)

Алгоритм использует две очереди с приоритетами:
- **max-heap `lower`** — хранит нижнюю половину элементов (самый большой из них доступен через `peek()`).
- **min-heap `upper`** — хранит верхнюю половину элементов (самый маленький из них доступен через `peek()`).

**Инвариант:** разница размеров куч не превышает 1; все элементы `lower` не больше всех элементов `upper`.

**Вставка и балансировка:**
1. Если новый элемент не превышает `lower.peek()` — добавить в `lower`, иначе — в `upper`.
2. Если `lower.size() > upper.size() + 1` — перенести `lower.poll()` в `upper`.
3. Если `upper.size() > lower.size() + 1` — перенести `upper.poll()` в `lower`.

**Получение медианы:**
- Если оба размера равны — среднее арифметическое `lower.peek()` и `upper.peek()`.
- Иначе — вершина большей кучи.

**Java-реализация:**

```java
public void addValue(int value) {
    if (lower.isEmpty() || value <= lower.peek()) {
        lower.add(value);
    } else {
        upper.add(value);
    }

    // Балансировка
    if (lower.size() > upper.size() + 1) {
        upper.add(lower.poll());
    } else if (upper.size() > lower.size() + 1) {
        lower.add(upper.poll());
    }
}

public double getMedian() {
    if (lower.size() == upper.size()) {
        return ((long) lower.peek() + upper.peek()) / 2.0;
    }
    return lower.size() > upper.size() ? lower.peek() : upper.peek();
}
```

**Сложность:** вставка — O(log n), получение медианы — O(1), память — O(n).

#### Ответ от Claude

**Постановка задачи**

Дан поток чисел, поступающих по одному. После каждого нового числа нужно возвращать медиану уже поступивших чисел. Медиана — это значение, делящее отсортированную последовательность пополам: при нечётном количестве n — средний элемент, при чётном — среднее двух средних.

**Почему одна структура данных недостаточна**

- Отсортированный массив: поиск медианы O(1), но вставка O(n).
- Бинарное дерево поиска / skip list: вставка O(log n), но нахождение медианы O(log n).
- Одна куча: peek() — O(1), но она даёт только min или только max, а не медиану.

**Идея двух куч**

Разделим все элементы на две половины: нижнюю (меньшую) и верхнюю (большую). Сохраним нижнюю в max-heap — тогда её максимум (потенциальная «левая медиана») всегда в корне. Сохраним верхнюю в min-heap — тогда её минимум (потенциальная «правая медиана») всегда в корне. Оба корня доступны за O(1).

**Инвариант (два условия, которые поддерживаются всегда)**

1. **Условие порядка:** максимум max-heap `<=` минимум min-heap. То есть `lower.peek() <= upper.peek()`.
2. **Условие баланса:** `|lower.size() - upper.size()| <= 1`. Допускается, что одна из куч больше другой не более чем на 1 элемент.

**Алгоритм вставки**

При поступлении нового числа `x`:

1. Определить, в какую кучу помещается `x`:
   - Если `lower` пуста или `x <= lower.peek()` — добавить в `lower` (нижнюю половину).
   - Иначе — добавить в `upper` (верхнюю половину).

2. Восстановить баланс (если нарушилось условие баланса):
   - Если `lower.size() > upper.size() + 1`: перенести `lower.poll()` в `upper`.
   - Если `upper.size() > lower.size() + 1`: перенести `upper.poll()` в `lower`.

Каждая вставка и каждый перенос — это heap add() и heap poll(), обе O(log n). Максимум одна операция переноса за вставку, поэтому итоговая сложность вставки — O(log n).

**Получение медианы — O(1)**

После вставки оба инварианта соблюдены. Медиана читается за O(1):
- Если `lower.size() == upper.size()`: медиана = `(lower.peek() + upper.peek()) / 2.0`.
- Если `lower.size() > upper.size()`: медиана = `lower.peek()`.
- Если `upper.size() > lower.size()`: медиана = `upper.peek()`.

**Полная Java-реализация (LeetCode 295)**

```java
import java.util.PriorityQueue;
import java.util.Collections;

class MedianFinder {
    // max-heap: нижняя половина
    private final PriorityQueue<Integer> lower = new PriorityQueue<>(Collections.reverseOrder());
    // min-heap: верхняя половина
    private final PriorityQueue<Integer> upper = new PriorityQueue<>();

    public void addNum(int num) {
        if (lower.isEmpty() || num <= lower.peek()) {
            lower.offer(num);
        } else {
            upper.offer(num);
        }

        // Восстановление баланса
        if (lower.size() > upper.size() + 1) {
            upper.offer(lower.poll());
        } else if (upper.size() > lower.size() + 1) {
            lower.offer(upper.poll());
        }
    }

    public double findMedian() {
        if (lower.size() == upper.size()) {
            return (lower.peek() + (double) upper.peek()) / 2.0;
        }
        return lower.size() > upper.size() ? lower.peek() : upper.peek();
    }
}
```

**Трассировка на примере:** поток `[5, 2, 4, 1, 7, 3]`

| Шаг | Добавлен | lower (max-heap) | upper (min-heap) | Медиана |
|-----|----------|------------------|------------------|---------|
| 1 | 5 | [5] | [] | 5.0 |
| 2 | 2 | [2] | [5] | 3.5 |
| 3 | 4 | [2, 4] | [5] | 4.0 |
| 4 | 1 | [1, 2] | [4, 5] | 3.0 |
| 5 | 7 | [1, 2, 4] | [5, 7] | 4.0 |
| 6 | 3 | [1, 2, 3] | [4, 5, 7] | 3.5 |

**Итоговые сложности**

| Операция | Сложность |
|---|---|
| addNum() | O(log n) |
| findMedian() | O(1) |
| Память | O(n) |

**Расхождения между источниками:** большинство источников соглашаются, что инвариант допускает одну из куч быть на 1 элемент больше. Некоторые реализации всегда хранят лишний элемент в lower (нижней) куче, другие допускают любую из двух. Оба подхода корректны при условии точного соблюдения выбранного правила в getMedian().

### Вопрос 43. Как слить k отсортированных массивов (или списков) в один за O(n log k), используя PriorityQueue в Java? Почему это эффективнее наивного попарного слияния O(nk)?

**Категория:** Алгоритмы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Merge K Sorted Arrays Using Priority Queue — procodebase.com](https://procodebase.com/article/merge-k-sorted-arrays-using-priority-queue)

Подход использует минимальную кучу (min-heap) для эффективного отслеживания наименьших элементов из нескольких отсортированных массивов.

Пошаговый алгоритм:
1. Создать PriorityQueue и вставить первый элемент каждого массива вместе с метаданными (индекс массива, позиция в массиве).
2. Повторять, пока очередь не пуста: извлечь минимальный элемент, добавить его в результат, затем вставить следующий элемент из того же массива.
3. Продолжать до обработки всех элементов.

Реализация использует вспомогательный класс `ArrayEntry`, хранящий три поля: значение элемента, номер массива-источника и позицию внутри него. Это позволяет знать, какой элемент добавить в кучу следующим.

Сложность:
- Временная: O(N log k), где N — суммарное число элементов во всех массивах, k — количество массивов. Логарифмический множитель возникает из операций вставки/извлечения из кучи размером k.
- Пространственная: O(k) — куча хранит не более k элементов одновременно.

Этот подход значительно превосходит наивное попарное слияние, при котором на каждый шаг слияния тратится O(N), а всего шагов k-1, итого O(Nk).

#### Ответ от Claude

**Постановка задачи.** Дано k отсортированных массивов (или списков) суммарным размером N. Требуется получить единый отсортированный массив.

**Алгоритм с минимальной кучей (PriorityQueue).**

```java
import java.util.PriorityQueue;

public class MergeKSorted {

    // Вспомогательный класс для хранения элемента и его координат
    private static class Entry {
        int value;
        int arrayIndex;   // из какого массива
        int elementIndex; // позиция в массиве

        Entry(int value, int arrayIndex, int elementIndex) {
            this.value = value;
            this.arrayIndex = arrayIndex;
            this.elementIndex = elementIndex;
        }
    }

    public static int[] merge(int[][] arrays) {
        // Суммируем общее количество элементов
        int n = 0;
        for (int[] arr : arrays) n += arr.length;

        int[] result = new int[n];
        int idx = 0;

        // Минимальная куча: сравниваем по значению элемента
        PriorityQueue<Entry> pq = new PriorityQueue<>(
            arrays.length, (a, b) -> a.value - b.value
        );

        // Инициализация: добавляем первый элемент каждого массива
        for (int i = 0; i < arrays.length; i++) {
            if (arrays[i].length > 0) {
                pq.offer(new Entry(arrays[i][0], i, 0));
            }
        }

        // Основной цикл
        while (!pq.isEmpty()) {
            Entry min = pq.poll();
            result[idx++] = min.value;
            // Если в том же массиве есть следующий элемент — добавляем в кучу
            int next = min.elementIndex + 1;
            if (next < arrays[min.arrayIndex].length) {
                pq.offer(new Entry(
                    arrays[min.arrayIndex][next],
                    min.arrayIndex,
                    next
                ));
            }
        }
        return result;
    }
}
```

**Анализ временной сложности.**

- Куча всегда содержит не более k элементов (по одному «фронту» от каждого массива).
- Каждый из N элементов ровно один раз извлекается из кучи (`poll`, O(log k)) и ровно один раз вставляется (`offer`, O(log k)).
- Итого: **O(N log k)**.

**Сравнение с наивными подходами.**

| Подход | Сложность | Причина |
|---|---|---|
| Куча (PriorityQueue) | O(N log k) | операции над кучей размером k |
| Попарное слияние (наивное) | O(Nk) | каждое из k-1 слияний обрабатывает растущий массив |
| Конкатенация + сортировка | O(N log N) | не использует упорядоченность исходных данных |

При k, значительно меньшем N, выигрыш подхода с кучей огромен. Например, при k=1000 и N=10^6: log k ≈ 10 против k=1000 — ускорение в 100 раз по сравнению с наивным попарным слиянием.

**Почему попарное слияние хуже.** При попарном слиянии на первом шаге сливаются два массива размером n — за O(n). На втором шаге результирующий массив размером 2n сливается со следующим — за O(2n). Суммарно: n + 2n + 3n + ... + (k-1)n = O(nk^2/2) = O(nk) в среднем (и O(nk) в наихудшем). Если же применять попарное слияние подобно merge sort (разбивая k массивов пополам рекурсивно), можно достичь O(N log k), но это уже другой, более сложный алгоритм.

**Пространственная сложность:** O(k) для кучи, O(N) для выходного массива.

### Вопрос 44. Что такое структура данных Union-Find (Disjoint Set Union)? Какова амортизированная сложность операций после применения оптимизаций «сжатие пути» (path compression) и «объединение по рангу» (union by rank)? Где применяется в алгоритмах на графах?

**Категория:** Алгоритмы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Union-Find (Disjoint Set): A Comprehensive Guide — algocademy.com](https://algocademy.com/blog/union-find-disjoint-set-a-comprehensive-guide-for-efficient-data-structure-operations/)

Union-Find (Disjoint Set) поддерживает коллекции непересекающихся множеств с двумя основными операциями: **Find** (нахождение представителя множества для элемента) и **Union** (объединение двух множеств). Эта структура данных особенно эффективна для таких операций, обеспечивая почти константную временную сложность при использовании оптимизаций.

**Сжатие пути (Path Compression):** Во время операции find структура выравнивает дерево, перенаправляя элементы напрямую к корню, уменьшая время будущих поиском.

**Объединение по рангу (Union by Rank):** Вместо произвольного слияния меньшие деревья присоединяются к большим, поддерживая сбалансированную глубину и предотвращая вырождение в линейные цепочки.

**Сложность операций:** Амортизированная сложность O(α(n)), где α — обратная функция Аккермана — значение настолько медленно растущее, что для всех практических значений n его можно считать константой.

**Применение в алгоритмах на графах:**
- Алгоритм Крускала для построения минимального остовного дерева: обнаружение циклов при добавлении рёбер путём проверки, принадлежат ли конечные вершины одному множеству.
- Обнаружение циклов: если оба конца ребра принадлежат одному множеству до объединения, добавление этого ребра создаёт цикл.
- Поиск связных компонент: операции union отслеживают, какие вершины принадлежат одной группе.

#### Ответ от Claude

**Определение.** Union-Find (DSU, Disjoint Set Union) — структура данных для управления разбиением множества из n элементов на непересекающиеся (disjoint) подмножества. Внутренне представляется как лес деревьев, где каждый узел указывает на своего родителя; корень дерева является представителем (representative) всего множества.

**Основные операции:**
- `find(x)` — возвращает представителя (корень) множества, которому принадлежит x.
- `union(x, y)` — объединяет множества, содержащие x и y.
- `connected(x, y)` — проверяет, находятся ли x и y в одном множестве (`find(x) == find(y)`).

**Базовая реализация на Java.**

```java
public class UnionFind {
    private int[] parent;
    private int[] rank;

    public UnionFind(int n) {
        parent = new int[n];
        rank = new int[n];
        for (int i = 0; i < n; i++) parent[i] = i; // каждый элемент — сам себе корень
    }

    // Find с сжатием пути (path compression)
    public int find(int x) {
        if (parent[x] != x) {
            parent[x] = find(parent[x]); // рекурсивно делаем корень прямым родителем
        }
        return parent[x];
    }

    // Union по рангу (union by rank)
    public void union(int x, int y) {
        int rx = find(x), ry = find(y);
        if (rx == ry) return; // уже в одном множестве

        if (rank[rx] < rank[ry]) {
            parent[rx] = ry;
        } else if (rank[rx] > rank[ry]) {
            parent[ry] = rx;
        } else {
            parent[ry] = rx;
            rank[rx]++;
        }
    }

    public boolean connected(int x, int y) {
        return find(x) == find(y);
    }
}
```

**Оптимизации и их эффект.**

*Сжатие пути (Path Compression).* При вызове `find(x)` алгоритм находит корень, а затем делает корень прямым родителем всех узлов на пройденном пути. После этого последующие вызовы `find` для тех же элементов работают за O(1).

*Объединение по рангу (Union by Rank).* Ранг — верхняя граница высоты поддерева. При объединении корень с меньшим рангом присоединяется к корню с большим рангом. Это предотвращает вырождение деревьев в линейные цепочки. Альтернатива — *объединение по размеру (union by size)*, где вместо ранга хранится количество элементов в поддереве; эффект аналогичен.

**Амортизированная сложность.** При совместном применении обеих оптимизаций каждая операция выполняется за амортизированное **O(α(n))**, где α — обратная функция Аккермана. Для любых практически встречающихся значений n (вплоть до 10^600) значение α(n) не превышает 4. На практике операции можно считать выполняющимися за **O(1)** амортизированно.

Без оптимизаций: find и union работают за O(n) в наихудшем случае (вырожденная цепочка). Только path compression или только union by rank дают O(log n) амортизированно.

**Применение в алгоритмах на графах.**

1. *Алгоритм Крускала (MST).* Рёбра сортируются по весу. Для каждого ребра (u, v) проверяется, принадлежат ли u и v одному компоненту (`find(u) == find(v)`). Если нет — ребро добавляется в MST и вызывается `union(u, v)`. Union-Find делает проверку цикла за O(α(n)) вместо O(V) при обходе в глубину.

2. *Обнаружение циклов в неориентированном графе.* Если при добавлении ребра (u, v) оказывается, что `find(u) == find(v)`, ребро образует цикл.

3. *Задача связных компонент.* Быстрое определение, принадлежат ли две вершины одной компоненте; подсчёт количества компонент.

4. *Алгоритм Борувки (MST).* Использует DSU для слияния компонент на каждой итерации.

5. *Задачи о сетевой связности* (network connectivity): онлайн-добавление рёбер с запросами «соединены ли вершины u и v?».

### Вопрос 45. Опишите алгоритм Дейкстры для нахождения кратчайшего пути в графе с неотрицательными весами рёбер. Какова его временная сложность при использовании PriorityQueue (бинарной кучи)? Почему Дейкстра не работает с отрицательными весами?

**Категория:** Алгоритмы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Dijkstra's algorithm — Wikipedia](https://en.wikipedia.org/wiki/Dijkstra%27s_algorithm)

Алгоритм Дейкстры находит кратчайшие пути от одной вершины-источника до всех остальных вершин в графе с неотрицательными весами рёбер.

**Временная сложность с бинарной кучей.** При использовании самобалансирующегося двоичного дерева поиска или бинарной кучи алгоритм требует Θ((|E| + |V|) log |V|) времени в наихудшем случае; для связных графов эта оценка упрощается до Θ(|E| log |V|).

**Почему алгоритм не работает с отрицательными весами.** Алгоритм Дейкстры не может корректно обрабатывать отрицательные веса рёбер, так как опирается на жадный подход. Как только вершина помечена как посещённая, алгоритм считает расстояние до неё окончательным и никогда не пересматривает его. При наличии отрицательных весов более короткий путь может существовать через непосещённую вершину по ребру с отрицательным весом, что нарушает это предположение. Для графов с отрицательными весами рёбер (но без отрицательных циклов) используется алгоритм Беллмана–Форда.

#### Ответ от Claude

**Описание алгоритма.**

Алгоритм Дейкстры решает задачу Single-Source Shortest Path (SSSP) для графа с неотрицательными весами рёбер. Он жадно выбирает на каждом шаге непосещённую вершину с наименьшим известным расстоянием от источника и «релаксирует» все исходящие из неё рёбра.

**Реализация на Java с PriorityQueue.**

```java
import java.util.*;

public class Dijkstra {

    // Граф задан списком смежности: graph[u] = список пар (v, weight)
    public static int[] dijkstra(List<List<int[]>> graph, int source) {
        int n = graph.size();
        int[] dist = new int[n];
        Arrays.fill(dist, Integer.MAX_VALUE);
        dist[source] = 0;

        // PriorityQueue: пара [расстояние, вершина], минимум по расстоянию
        PriorityQueue<int[]> pq = new PriorityQueue<>(Comparator.comparingInt(a -> a[0]));
        pq.offer(new int[]{0, source});

        while (!pq.isEmpty()) {
            int[] curr = pq.poll();
            int d = curr[0], u = curr[1];

            // Если это устаревшая запись — пропускаем
            if (d > dist[u]) continue;

            for (int[] edge : graph.get(u)) {
                int v = edge[0], w = edge[1];
                if (dist[u] + w < dist[v]) {
                    dist[v] = dist[u] + w;
                    pq.offer(new int[]{dist[v], v});
                }
            }
        }
        return dist;
    }
}
```

**Пошаговая логика.**
1. Инициализация: расстояние до источника = 0, до всех остальных = ∞.
2. Добавить источник в min-heap.
3. Извлечь вершину u с минимальным расстоянием. Если расстояние устарело (в куче могут быть дубликаты) — пропустить.
4. Для каждого соседа v вершины u: если `dist[u] + weight(u,v) < dist[v]`, обновить `dist[v]` и добавить v в кучу.
5. Повторять до опустошения кучи.

**Временная сложность с бинарной кучей.**

| Операция | Количество | Стоимость | Итого |
|---|---|---|---|
| Извлечение минимума (poll) | O(V) | O(log V) | O(V log V) |
| Вставка в кучу (offer) | O(E) | O(log V) | O(E log V) |
| **Итого** | | | **O((V + E) log V)** |

Для связного графа E >= V-1, поэтому оценка упрощается до **O(E log V)**. С кучей Фибоначчи сложность улучшается до O(V log V + E), однако на практике бинарная куча предпочтительнее из-за меньших константных множителей и простоты реализации.

**Инвариант жадного алгоритма и почему он нарушается при отрицательных весах.**

Ключевой инвариант Дейкстры: *когда вершина u извлекается из приоритетной очереди, значение `dist[u]` является окончательным кратчайшим расстоянием от источника до u*.

Доказательство инварианта опирается на то, что все веса рёбер неотрицательны. При извлечении u из кучи с расстоянием d: любой альтернативный путь к u должен пройти через некоторую непосещённую вершину w, у которой `dist[w] >= d` (так как d — минимум в куче). Добавление любых неотрицательных рёбер от w далее только увеличивает суммарное расстояние, значит никакой более короткий путь через w не существует.

При отрицательном весе ребра (w → u) может оказаться, что `dist[w] + weight(w, u) < d`, то есть существует более короткий путь, который был не рассмотрен на момент «окончательного» закрытия u. Алгоритм не пересматривает уже закрытые вершины, поэтому выдаёт неверный результат.

**Пример сбоя при отрицательном весе:**
```
A --(1)--> B --(−3)--> C
A --(5)--> C
```
Дейкстра от A: сначала закрывает B (dist=1), затем C с dist=5 (путь A→C). Но реальный кратчайший путь A→B→C = 1 + (−3) = −2. Алгоритм уже закрыл C и не пересмотрит результат.

**Альтернативы при наличии отрицательных весов:**
- *Алгоритм Беллмана–Форда* — работает за O(VE), корректно обрабатывает отрицательные веса, обнаруживает отрицательные циклы.
- *SPFA (Shortest Path Faster Algorithm)* — эвристическое ускорение Беллмана–Форда, O(VE) в худшем случае.
- *Алгоритм Джонсона* — для задачи All-Pairs Shortest Path: перевешивает граф, устраняя отрицательные веса, затем применяет Дейкстру от каждой вершины.
### Вопрос 46. Что такое монотонный стек (monotonic stack) и какие задачи он решает? Объясните, как с его помощью найти «следующий больший элемент» (Next Greater Element) для каждого элемента массива за O(n).

**Категория:** Алгоритмы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Next Greater Element in Array — GeeksforGeeks](https://www.geeksforgeeks.org/dsa/next-greater-element/)

Монотонный стек — это стек, в котором все элементы упорядочены по монотонно убывающему или монотонно возрастающему принципу. Для поиска следующего большего элемента используется монотонно убывающий стек (от дна к вершине).

Алгоритм (обход справа налево):
1. Инициализировать массив результатов значениями -1 и пустой стек.
2. Для каждого элемента (от правого конца к левому) удалять из стека все элементы, которые меньше или равны текущему — они не могут быть «следующим большим» ни для одного последующего элемента.
3. Если стек не пуст после удалений, его вершина и есть следующий больший элемент.
4. Добавить текущий элемент в стек.

```python
def nextLargerElement(arr):
    n = len(arr)
    res = [-1] * n
    stk = []
    for i in range(n - 1, -1, -1):
        while stk and arr[stk[-1]] <= arr[i]:
            stk.pop()
        if stk:
            res[i] = arr[stk[-1]]
        stk.append(i)
    return res
```

Для входа `[6, 8, 0, 1, 3]` результат: `[8, -1, 1, 3, -1]`.

Временная сложность: O(n) — каждый элемент ровно один раз помещается в стек и не более одного раза извлекается.
Пространственная сложность: O(n) — размер стека.

#### Ответ от Claude

**Что такое монотонный стек**

Монотонный стек — специализированная структура данных на основе обычного стека с дополнительным инвариантом: элементы в нём всегда упорядочены (возрастают от дна к вершине — монотонно возрастающий, или убывают — монотонно убывающий). При добавлении нового элемента нарушающие инвариант элементы удаляются из вершины. Сам факт удаления — полезное событие: именно тогда фиксируется ответ для удалённого элемента.

**Классы задач, решаемых монотонным стеком:**
- Следующий/предыдущий больший или меньший элемент (Next/Previous Greater/Smaller Element).
- Наибольший прямоугольник в гистограмме (Largest Rectangle in Histogram).
- Захват дождевой воды (Trapping Rain Water).
- Скользящий максимум/минимум окна (Sliding Window Maximum).

**Алгоритм Next Greater Element за O(n)**

Используется монотонно убывающий стек. Возможны два варианта обхода:

*Вариант 1 — справа налево (как в источнике выше):* для каждого элемента мы сразу знаем своих «кандидатов» в стеке.

*Вариант 2 — слева направо (через индексы неразрешённых элементов):*
1. Инициализировать `result[i] = -1` для всех i.
2. Для каждого элемента `arr[i]`:
   - Пока стек не пуст и `arr[стек.вершина] < arr[i]`: извлечь индекс j, присвоить `result[j] = arr[i]`.
   - Поместить i в стек.
3. Оставшиеся в стеке индексы сохраняют значение -1.

```java
public int[] nextGreaterElement(int[] arr) {
    int n = arr.length;
    int[] result = new int[n];
    Arrays.fill(result, -1);
    Deque<Integer> stack = new ArrayDeque<>();
    for (int i = 0; i < n; i++) {
        while (!stack.isEmpty() && arr[stack.peek()] < arr[i]) {
            result[stack.pop()] = arr[i];
        }
        stack.push(i);
    }
    return result;
}
```

**Почему O(n), несмотря на вложенный цикл while:**

Ключевое наблюдение: каждый элемент помещается в стек ровно один раз и удаляется из стека не более одного раза. Суммарное количество операций push и pop не превышает 2n. Внешний цикл выполняется n раз, внутренний цикл while в совокупности по всем итерациям — не более n раз. Итого: O(n) по времени, O(n) по памяти.

**Тип стека зависит от задачи:**
- Следующий больший элемент → монотонно убывающий стек (при добавлении большего элемента выталкиваем меньшие).
- Следующий меньший элемент → монотонно возрастающий стек.

### Вопрос 47. Что такое префиксное дерево (Trie)? Какова временная и пространственная сложность операций вставки, поиска слова и поиска по префиксу? Чем Trie лучше HashSet при работе с множеством строк?

**Категория:** Алгоритмы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Introduction to Trie Data Structure and Algorithm — GeeksforGeeks](https://www.geeksforgeeks.org/dsa/introduction-to-trie-data-structure-and-algorithm-tutorials/)

Trie (произносится «трай», от слова re**trie**val) — древовидная структура данных, в которой каждый узел представляет строку, а каждое ребро — символ. Пустой корневой узел соединён с дочерними узлами, формирующими префиксы хранимых слов. Каждый узел содержит массив или хеш-таблицу из 26 указателей (для строчных латинских букв) и булев флаг `isEndOfWord`.

**Временная сложность:**
- Вставка: O(m), где m — длина слова.
- Поиск слова: O(m).
- Поиск по префиксу (startsWith): O(m), где m — длина префикса.

**Пространственная сложность:** O(n × m) в худшем случае (нет общих префиксов), где n — количество слов.

**Преимущества перед HashSet:**
1. Эффективный поиск по префиксу (автодополнение): HashSet не может эффективно найти все строки с заданным префиксом.
2. Лексикографическая сортировка: обход в глубину выдаёт слова в алфавитном порядке — с хешированием это невозможно без дополнительной сортировки.
3. Нет накладных расходов на хеш-функцию и разрешение коллизий.

**Недостаток Trie:** высокое потребление памяти — каждый узел хранит 26 (или более) указателей.

#### Ответ от Claude

**Структура Trie**

Trie (префиксное дерево) — дерево поиска, в котором позиция узла определяет ключ: путь от корня до узла кодирует строку-префикс. Корень соответствует пустой строке; каждый дочерний узел добавляет один символ. Узел содержит:
- Массив/хеш-таблицу дочерних узлов (для алфавита размером σ — обычно 26 или 128).
- Флаг `isEndOfWord` (или счётчик вхождений).

**Временная сложность:**

| Операция | Сложность |
|---|---|
| insert(word) | O(m), m = длина слова |
| search(word) | O(m) |
| startsWith(prefix) | O(p), p = длина префикса |

Все три операции не зависят от количества слов в Trie — только от длины целевой строки. В этом принципиальное отличие от HashMap с потенциальными коллизиями.

**Пространственная сложность:**
- Худший случай (уникальные символы во всех словах): O(n × m × σ), где σ — размер алфавита.
- Лучший случай (все слова с общим префиксом): близко к O(суммарная длина уникальных префиксов).
- На практике при большом количестве слов с общими префиксами Trie экономит память относительно хранения полных строк по отдельности.

**Реализация узла (Java):**

```java
class TrieNode {
    TrieNode[] children = new TrieNode[26];
    boolean isEndOfWord = false;
}
```

**Trie против HashSet: когда Trie выигрывает**

1. **Поиск по префиксу (startsWith).** HashSet не имеет эффективной операции «найти все строки с префиксом P». Для этого потребовалось бы перебирать все элементы за O(n × m). Trie выполняет traversal за O(p + k), где k — количество найденных результатов.

2. **Автодополнение и spell-check.** Trie — стандартная структура данных для этих задач (поисковые строки, IDE autocomplete).

3. **Алфавитный порядок.** DFS по Trie автоматически обходит ключи в лексикографическом порядке.

4. **Детерминированное время поиска.** Поиск в Trie за O(m) не ухудшается при коллизиях или плохих хеш-функциях.

5. **Ранняя остановка при отсутствии ключа.** Если на i-м символе нет соответствующего дочернего узла, поиск завершается немедленно — не нужно вычислять хеш всей строки.

**Когда HashSet предпочтительнее Trie:**
- Нужен только поиск по точному совпадению (exact match): один вызов `hashCode()` быстрее обхода m узлов.
- Алфавит большой (Unicode): 26-элементный массив превращается в объёмную хеш-таблицу на каждом узле.
- Критична экономия памяти при отсутствии общих префиксов.

### Вопрос 48. Объясните три метода амортизированного анализа: агрегатный (aggregate), метод банкира (accounting) и метод потенциала (potential). Проиллюстрируйте на примере динамического массива (ArrayList), почему вставка в конец амортизированно O(1), несмотря на редкие O(n) расширения.

**Категория:** Алгоритмы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Introduction to Amortized Analysis — GeeksforGeeks](https://www.geeksforgeeks.org/dsa/introduction-to-amortized-analysis/)

Амортизированный анализ — метод оценки средней стоимости операции в последовательности из n операций, где отдельные операции могут быть дорогими, но редкими.

**Три метода:**

1. **Агрегатный метод (Aggregate method):** вычисляется суммарная стоимость T(n) всей последовательности из n операций, затем амортизированная стоимость одной операции = T(n) / n.

2. **Метод учёта / банкира (Accounting method):** каждой операции назначается амортизированная стоимость, отличающаяся от реальной. Дешёвые операции накапливают «кредит», который списывается при дорогих операциях. Кредит не может стать отрицательным.

3. **Метод потенциала (Potential method):** уточнение метода учёта, в котором накопленный кредит выражается через функцию потенциала Φ, зависящую от состояния структуры данных. Амортизированная стоимость = реальная стоимость + ΔΦ (изменение потенциала).

**Пример — динамический массив:**
При добавлении элемента в заполненный ArrayList создаётся массив вдвое большего размера и копируются все элементы (стоимость O(n)). Однако копирования происходят редко — после вставки элементов в объём 1, 2, 4, 8 ... При n вставках суммарные расходы на копирование образуют геометрическую прогрессию: 1 + 2 + 4 + ... ≤ 2n = O(n). Вместе с n обычными вставками общая стоимость O(n), а амортизированная стоимость одной операции O(n)/n = O(1).

#### Ответ от Claude

**Зачем нужен амортизированный анализ**

Пессимистичный анализ «worst-case per operation» даёт завышенную оценку, когда дорогая операция происходит редко и её стоимость «распределяется» по многим дешёвым операциям. Амортизированный анализ даёт гарантию на среднюю стоимость в последовательности, а не только на худший случай одной операции.

---

**Метод 1 — Агрегатный (Aggregate)**

Идея: посчитать суммарную стоимость T(n) ровно n операций, затем амортизированная стоимость = T(n) / n.

Пример с ArrayList (стратегия удвоения):
- n вставок в конец.
- Обычная вставка стоит 1.
- Расширение при заполнении на k-м элементе стоит k (копирование).
- Расширения происходят при размерах 1, 2, 4, 8, ..., 2^⌊log n⌋.
- Суммарная стоимость расширений: 1 + 2 + 4 + ... + 2^⌊log n⌋ < 2n.
- Суммарная стоимость обычных вставок: n.
- T(n) < 3n = O(n).
- Амортизированная стоимость одной вставки: O(n) / n = **O(1)**.

---

**Метод 2 — Метод банкира (Accounting / Banker's method)**

Идея: назначить каждой операции «цену» (amortized cost) так, чтобы оставался неотрицательный «кредит» на будущие дорогие операции. Главное правило: сумма амортизированных стоимостей ≥ сумма реальных стоимостей (кредит неотрицателен в любой момент).

Пример с ArrayList:
- Назначим вставке амортизированную цену = 3 (вместо реальной 1).
- 1 монета — оплата реальной вставки.
- 2 монеты — «предоплата» будущего копирования.
- При расширении от n до 2n нужно скопировать n элементов. Эти n элементов уже накопили по 2 монеты каждый = 2n монет ≥ n копирований. Кредит покрывает расходы.
- Итого: амортизированная стоимость вставки = 3 = **O(1)**.

---

**Метод 3 — Метод потенциала (Potential method)**

Идея: определить функцию потенциала Φ(D) — «запас энергии» структуры данных D в текущем состоянии. Амортизированная стоимость операции i:

```
â_i = c_i + Φ(D_i) - Φ(D_{i-1})
```

где `c_i` — реальная стоимость, `Φ(D_i) - Φ(D_{i-1})` — изменение потенциала.

Условия корректности: Φ(D_0) = 0 и Φ(D_i) ≥ 0 для всех i (потенциал не уходит ниже начального).

Сумма амортизированных стоимостей: Σ â_i = Σ c_i + Φ(D_n) - Φ(D_0) ≥ Σ c_i, что означает оценку сверху.

Пример с ArrayList (пусть n = текущее число элементов, cap = ёмкость):

Определим Φ(D) = 2·size - capacity.

- После расширения (size = capacity/2 + 1, capacity удвоилась): Φ ≈ 2·(cap/2 + 1) - cap = 2 (малый потенциал).
- По мере заполнения потенциал растёт до cap (= 2·size - size = size при size = cap).

Обычная вставка (нет расширения):
- c_i = 1, Φ увеличивается на 2·1 - 0 = 2 (size растёт на 1, capacity не меняется).
- â_i = 1 + 2 = 3.

Вставка с расширением (size = cap перед вставкой):
- c_i = cap + 1 (копирование cap элементов + одна вставка).
- После расширения capacity = 2·cap, size = cap + 1.
- ΔΦ = (2·(cap+1) - 2·cap) - (2·cap - cap) = 2 - cap.
- â_i = (cap + 1) + (2 - cap) = 3.

В обоих случаях амортизированная стоимость = 3 = **O(1)**.

---

**Сравнение методов:**

| Метод | Подход | Применимость |
|---|---|---|
| Агрегатный | Подсчёт суммы, деление | Прост, если T(n) легко вычислить |
| Банкира | Явный кредит на операцию | Нагляден для интуитивного объяснения |
| Потенциала | Функция состояния структуры | Мощнее, подходит для сложных структур (splay tree, фибоначчиева куча) |

Все три метода дают одинаковый результат для одной и той же структуры данных — это разные способы формального обоснования одного факта. Метод потенциала наиболее универсален и используется в доказательствах для AVL/splay-деревьев, фибоначчиевых куч и других самобалансирующихся структур.
### Вопрос 49. Как решить задачу «Размен монет» (Coin Change) с помощью DP за O(n·W) по времени? Почему жадный алгоритм не даёт оптимального результата в общем случае? Как считать число способов разменять сумму, а не минимальное количество монет?

**Категория:** Алгоритмы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Change-making problem — Wikipedia](https://en.wikipedia.org/wiki/Change-making_problem)

**Постановка задачи.** Дан набор номиналов монет w₁, w₂, …, wₙ и целевая сумма W. Требуется набрать W минимальным числом монет (монеты можно использовать повторно).

**DP-решение за O(n·W).** Алгоритм строит таблицу dp размером (n+1) × (W+1), где dp[i][j] — минимальное число монет из первых i номиналов для набора суммы j. Переход:

```
dp[i][j] = min(dp[i-1][j],          // не берём монету i
               dp[i][j - coin_i] + 1) // берём монету i (неограниченно)
```

Использование dp[i][j - coin_i] (а не dp[i-1][j - coin_i]) позволяет брать одну и ту же монету неограниченное число раз. Такой подход называется «Unbounded Knapsack» («неограниченный рюкзак»).

**Почему жадный алгоритм неоптимален.** Жадный выбор — всегда брать монету наибольшего достоинства — работает лишь для «канонических» систем монет (например, US: 1, 5, 10, 25). Пример провала: номиналы {5, 10, 20, 25}, сумма 40 пайс. Жадный даёт {25, 10, 5} — 3 монеты; оптимум — {20, 20} — 2 монеты. Другой пример: {1, 3, 4}, сумма 6. Жадный: {4, 1, 1} — 3 монеты; оптимум: {3, 3} — 2 монеты. Жадный алгоритм делает локально оптимальный выбор, не учитывая, что меньшие номиналы в сумме могут дать глобально лучший результат.

**Подсчёт числа способов** — это отдельный вариант задачи (LeetCode 518 «Coin Change II»). dp[j] здесь означает количество комбинаций монет, дающих сумму j. Переход:

```
dp[0] = 1  // одна комбинация для суммы 0 — пустое множество
for каждой монеты coin:
    for j от coin до W:
        dp[j] += dp[j - coin]
```

Ключевое правило: монеты перебираются во внешнем цикле, суммы — во внутреннем. Это гарантирует, что каждая комбинация считается ровно один раз (не как перестановка). Если поменять циклы местами, получим подсчёт перестановок.

#### Ответ от Claude

**Задача.** Дан массив номиналов `coins[0..n-1]` и целевая сумма `W`. Монеты можно использовать неограниченное число раз.

---

**Вариант 1: минимальное число монет**

Используется одномерный массив dp размером W+1, что является пространственной оптимизацией двумерного подхода:

```java
public int coinChange(int[] coins, int amount) {
    int[] dp = new int[amount + 1];
    Arrays.fill(dp, amount + 1);  // «бесконечность»
    dp[0] = 0;
    for (int coin : coins) {
        for (int j = coin; j <= amount; j++) {
            dp[j] = Math.min(dp[j], dp[j - coin] + 1);
        }
    }
    return dp[amount] > amount ? -1 : dp[amount];
}
```

Здесь dp[j] — минимальное число монет для суммы j. Оба порядка циклов (монеты снаружи или суммы снаружи) дают правильный результат для задачи минимума, поскольку операция min идемпотентна относительно порядка рассмотрения монет.

Сложность: время O(n · W), память O(W).

---

**Почему жадный алгоритм неоптимален в общем случае**

Жадный подход предполагает, что локально лучший выбор (самая крупная монета) приводит к глобально лучшему результату. Это верно только для «канонических» систем (US, EU). В произвольной системе номиналов жадный алгоритм не обладает свойством оптимальной подструктуры.

Контрпример — {1, 3, 4}, сумма 6:
- Жадный: 4 + 1 + 1 = 3 монеты
- Оптимум: 3 + 3 = 2 монеты

Другой контрпример — {1, 6, 10}, сумма 12:
- Жадный: 10 + 1 + 1 = 3 монеты
- Оптимум: 6 + 6 = 2 монеты

DP рассматривает все возможные комбинации и выбирает глобальный оптимум.

---

**Вариант 2: число способов разменять сумму**

```java
public int change(int amount, int[] coins) {
    int[] dp = new int[amount + 1];
    dp[0] = 1;  // одна комбинация для суммы 0
    for (int coin : coins) {          // внешний цикл — монеты
        for (int j = coin; j <= amount; j++) {  // внутренний — суммы
            dp[j] += dp[j - coin];
        }
    }
    return dp[amount];
}
```

dp[j] означает количество комбинаций (без учёта порядка) для набора суммы j.

Почему важен порядок циклов:
- Монеты снаружи, суммы внутри → каждая монета «добавляется» в уже построенные комбинации → считаются комбинации (каждый набор ровно один раз).
- Суммы снаружи, монеты внутри → считаются перестановки ({1,2} и {2,1} считаются разными вариантами).

Сложность обоих вариантов: время O(n · W), память O(W).

---

**Сводная таблица**

| Вариант | Внешний цикл | Внутренний цикл | Переход | Считает |
|---|---|---|---|---|
| Мин. монет | монеты или суммы | суммы или монеты | min | минимум монет |
| Число комбинаций | монеты | суммы | += | комбинации |
| Число перестановок | суммы | монеты | += | перестановки |

---

**Расхождения между источниками.** Некоторые источники описывают вариант подсчёта числа способов через двумерный массив dp[i][j] с явным рассмотрением i первых монет — это эквивалентно одномерному варианту, но требует O(n · W) памяти. Одномерный вариант с внешним циклом по монетам является стандартной оптимизацией и даёт те же результаты.

### Вопрос 50. Как решить задачу «Редакционное расстояние» (Edit Distance / Levenshtein Distance) с помощью DP? Что представляет ячейка dp[i][j]? Как оптимизировать пространственную сложность с O(m·n) до O(min(m,n))?

**Категория:** Алгоритмы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Edit distance — Wikipedia](https://en.wikipedia.org/wiki/Edit_distance)

**Определение.** Расстояние Левенштейна между строками a и b — минимальное число элементарных операций (вставка, удаление, замена символа), необходимых для превращения a в b.

**DP-подход.** Строится матрица d размером (m+1) × (n+1), где m = |a|, n = |b|. Ячейка d[i][j] хранит редакционное расстояние между первыми i символами строки a и первыми j символами строки b.

Базовые случаи:
- d[i][0] = i (удалить i символов из a, чтобы получить пустую строку)
- d[0][j] = j (вставить j символов в пустую строку, чтобы получить b[1..j])

Рекуррентное соотношение:
- Если a[i] = b[j]: d[i][j] = d[i-1][j-1]  (символы совпадают — операция не нужна)
- Иначе: d[i][j] = 1 + min(d[i-1][j],    // удаление из a
                             d[i][j-1],    // вставка в a
                             d[i-1][j-1]) // замена

Итоговый ответ: d[m][n].

**Пространственная оптимизация.** В любой момент алгоритм использует только текущую и предыдущую строки матрицы. Поэтому вместо матрицы O(m·n) достаточно двух массивов длины min(m, n) + 1 (помещаем более короткую строку по столбцам). Это снижает память до O(min(m, n)). Компромисс: при такой оптимизации восстановить саму последовательность операций невозможно. Для линейного по памяти восстановления последовательности используется алгоритм Хиршберга (Hirschberg's algorithm).

#### Ответ от Claude

**Постановка.** Даны строки `word1` длины m и `word2` длины n. Разрешены три операции стоимостью 1: вставка, удаление, замена символа. Найти минимальное число операций для превращения `word1` в `word2`.

---

**Смысл ячейки dp[i][j]**

dp[i][j] — минимальное число операций для превращения префикса `word1[0..i-1]` (первые i символов) в префикс `word2[0..j-1]` (первые j символов).

Примеры: dp[0][j] = j (вставить j символов в пустую строку); dp[i][0] = i (удалить i символов до пустой строки).

---

**Рекуррентное соотношение**

```
if word1[i-1] == word2[j-1]:
    dp[i][j] = dp[i-1][j-1]          // символы равны — операция не нужна
else:
    dp[i][j] = 1 + min(
        dp[i-1][j],    // удаление word1[i-1]
        dp[i][j-1],    // вставка word2[j-1]
        dp[i-1][j-1]   // замена word1[i-1] на word2[j-1]
    )
```

---

**Реализация (Java, O(m·n) памяти)**

```java
public int minDistance(String word1, String word2) {
    int m = word1.length(), n = word2.length();
    int[][] dp = new int[m + 1][n + 1];
    for (int i = 0; i <= m; i++) dp[i][0] = i;
    for (int j = 0; j <= n; j++) dp[0][j] = j;
    for (int i = 1; i <= m; i++) {
        for (int j = 1; j <= n; j++) {
            if (word1.charAt(i - 1) == word2.charAt(j - 1)) {
                dp[i][j] = dp[i - 1][j - 1];
            } else {
                dp[i][j] = 1 + Math.min(dp[i - 1][j - 1],
                                Math.min(dp[i - 1][j], dp[i][j - 1]));
            }
        }
    }
    return dp[m][n];
}
```

Сложность: время O(m · n), память O(m · n).

---

**Пример трассировки: «horse» → «ros»**

```
     ""  r  o  s
""  [ 0  1  2  3 ]
h   [ 1  1  2  3 ]
o   [ 2  2  1  2 ]
r   [ 3  2  2  2 ]
s   [ 4  3  3  2 ]
e   [ 5  4  4  3 ]
```

dp[5][3] = 3: три операции (удалить 'h', заменить 'r' на 'r' — уже совпадает, удалить 'e').

---

**Оптимизация памяти до O(min(m, n))**

Наблюдение: для вычисления строки i нужны только строка i-1 и текущая строка i. Переставляем строки так, чтобы n = min(m, n) (транспонируем задачу при необходимости). Затем работаем с двумя массивами длины n+1:

```java
public int minDistance(String word1, String word2) {
    if (word1.length() < word2.length()) {
        // Делаем word2 короткой, чтобы хранить массив меньшего размера
        return minDistance(word2, word1);
    }
    int m = word1.length(), n = word2.length();
    int[] prev = new int[n + 1];
    int[] curr = new int[n + 1];
    for (int j = 0; j <= n; j++) prev[j] = j;
    for (int i = 1; i <= m; i++) {
        curr[0] = i;
        for (int j = 1; j <= n; j++) {
            if (word1.charAt(i - 1) == word2.charAt(j - 1)) {
                curr[j] = prev[j - 1];
            } else {
                curr[j] = 1 + Math.min(prev[j - 1],
                               Math.min(prev[j], curr[j - 1]));
            }
        }
        int[] tmp = prev; prev = curr; curr = tmp;  // своп без аллокации
    }
    return prev[n];
}
```

Память: O(min(m, n)). Время остаётся O(m · n).

```java
            dp[j] = 1 + Math.min(prev, Math.min(dp[j], dp[j - 1]));
        }
        prev = temp;  // сохраняем для следующей итерации
    }
}
return dp[n];
```

---

**Ограничение оптимизации и алгоритм Хиршберга**

При оптимизации до O(min(m, n)) теряется возможность восстановить саму последовательность операций (путь в матрице dp). Если последовательность нужна, применяется алгоритм Хиршберга (Hirschberg's algorithm): он комбинирует divide-and-conquer с DP и достигает линейной памяти O(min(m, n)) при сохранении возможности восстановить оптимальное выравнивание строк.

---

**Сложность итого**

| Подход | Время | Память | Восстановление пути |
|---|---|---|---|
| 2D DP | O(m·n) | O(m·n) | Да |
| 2 строки / 1 строка | O(m·n) | O(min(m,n)) | Нет |
| Алгоритм Хиршберга | O(m·n) | O(min(m,n)) | Да |

### Вопрос 51. Опишите алгоритм Беллмана–Форда для поиска кратчайших путей. Почему он работает с отрицательными весами рёбер, тогда как Дейкстра — нет? Как алгоритм обнаруживает отрицательный цикл?

**Категория:** Алгоритмы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Bellman–Ford — finding shortest paths with negative weights — Algorithms for Competitive Programming (cp-algorithms.com)](https://cp-algorithms.com/graph/bellman_ford.html)

Беллман–Форд — алгоритм поиска кратчайших путей от одной вершины-источника в ориентированном взвешенном графе. В отличие от Дейкстры, он применим к графам с отрицательными весами рёбер.

**Алгоритм:**
1. Инициализировать расстояние до источника = 0, до всех остальных вершин = ∞.
2. Выполнить n−1 фаз релаксации: для каждого ребра (a, b) с весом c попытаться улучшить `d[b] = min(d[b], d[a] + c)`.
3. После n−1 фаз выполнить ещё одну проверочную фазу: если хотя бы одна релаксация удалась — граф содержит отрицательный цикл, достижимый из источника.

**Временная сложность:** O(n·m), где n — вершины, m — рёбра. **Пространственная сложность:** O(n + m).

Ключевое свойство: после k фаз алгоритм правильно вычисляет кратчайшие пути, использующие не более k рёбер. Через n−1 фаз любой простой путь (без циклов) уже рассмотрен, поскольку простой путь в графе содержит не более n−1 рёбер.

**Обнаружение отрицательного цикла:** если n-я фаза всё ещё даёт релаксацию, значит, существует путь длиной n рёбер, который короче пути длиной n−1 рёбер — это возможно только при наличии отрицательного цикла.

#### Ответ от Claude

**Суть алгоритма**

Беллман–Форд решает задачу Single-Source Shortest Path (SSSP) через многократную релаксацию всех рёбер графа. Он не требует, чтобы веса рёбер были неотрицательными, и способен обнаружить отрицательные циклы.

**Реализация на Java:**

```java
public class BellmanFord {
    static final int INF = Integer.MAX_VALUE / 2;

    public static int[] bellmanFord(int n, int[][] edges, int src) {
        int[] dist = new int[n];
        Arrays.fill(dist, INF);
        dist[src] = 0;

        // n−1 фаз релаксации
        for (int i = 0; i < n - 1; i++) {
            for (int[] edge : edges) {
                int u = edge[0], v = edge[1], w = edge[2];
                if (dist[u] != INF && dist[u] + w < dist[v]) {
                    dist[v] = dist[u] + w;
                }
            }
        }

        // Проверка наличия отрицательного цикла
        for (int[] edge : edges) {
            int u = edge[0], v = edge[1], w = edge[2];
            if (dist[u] != INF && dist[u] + w < dist[v]) {
                throw new RuntimeException("Граф содержит отрицательный цикл");
            }
        }
        return dist;
    }
}
```

**Почему Дейкстра не работает с отрицательными весами, а Беллман–Форд работает**

Дейкстра опирается на инвариант: как только вершина u извлечена из приоритетной очереди, её расстояние является окончательным. Этот инвариант справедлив только при неотрицательных весах. При наличии отрицательного ребра (w → u) более короткий путь может появиться позже — Дейкстра его пропустит.

Беллман–Форд не «закрывает» вершины. Он повторяет релаксацию n−1 раз: после k итераций гарантируется нахождение всех кратчайших путей, использующих не более k рёбер. Отрицательные веса допустимы, потому что алгоритм не делает жадных предположений о финальности расстояний.

**Отрицательный цикл**

Если между вершинами u и v существует путь через отрицательный цикл, расстояние `dist[v]` может быть уменьшено до −∞. Алгоритм обнаруживает это на n-й проверочной фазе: если хотя бы одна релаксация улучшает результат — цикл найден.

**Сравнение с Дейкстрой:**

| Критерий | Беллман–Форд | Дейкстра (с кучей) |
|---|---|---|
| Отрицательные веса | Да | Нет |
| Отрицательные циклы | Обнаруживает | Не применим |
| Временная сложность | O(V · E) | O((V + E) log V) |
| Применение | Общий случай | Неотрицательные веса |

**Практическое применение:** алгоритм используется в протоколе маршрутизации RIP (Routing Information Protocol), задачах арбитража на валютных рынках (обнаружение отрицательных циклов в графе курсов обмена), а также в алгоритме Джонсона для перевешивания рёбер.

### Вопрос 52. Что такое алгоритм Флойда–Уоршелла? Как он решает задачу All-Pairs Shortest Path? Как обнаружить отрицательный цикл и почему временная сложность O(V³)?

**Категория:** Алгоритмы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Floyd–Warshall — finding all shortest paths — Algorithms for Competitive Programming (cp-algorithms.com)](https://cp-algorithms.com/graph/all-pair-shortest-path-floyd-warshall.html)

Алгоритм Флойда–Уоршелла вычисляет кратчайшие пути между всеми парами вершин взвешенного ориентированного или неориентированного графа. Он работает с отрицательными весами рёбер, но требует отсутствия отрицательных циклов.

Алгоритм выполняет n фаз. На фазе k рассматривается вопрос: «улучшится ли путь от i до j, если он пройдёт через промежуточную вершину k?»

Рекуррентное соотношение: `d[i][j] = min(d[i][j], d[i][k] + d[k][j])`.

**Временная сложность:** O(n³) — три вложенных цикла по n вершинам. **Пространственная сложность:** O(n²).

**Обнаружение отрицательного цикла:** после завершения алгоритма, если хотя бы одна диагональная ячейка `d[t][t] < 0` — граф содержит отрицательный цикл, проходящий через вершину t. Любая пара вершин (i, j), путь которой проходит через такой t, имеет кратчайшее расстояние −∞.

#### Ответ от Claude

**Постановка задачи**

All-Pairs Shortest Path (APSP) — задача нахождения кратчайших расстояний между всеми парами вершин графа. В отличие от Single-Source Shortest Path (Дейкстра, Беллман–Форд), APSP требует ответа для каждой пары (i, j), а не только от одного источника.

**Алгоритм Флойда–Уоршелла**

Инициализация: `d[i][j] = weight(i, j)` если ребро существует; `d[i][j] = ∞` если ребра нет; `d[i][i] = 0`.

На каждой фазе k (от 0 до n−1) обновляем все пары:

```java
for (int k = 0; k < n; k++) {
    for (int i = 0; i < n; i++) {
        for (int j = 0; j < n; j++) {
            if (dist[i][k] + dist[k][j] < dist[i][j]) {
                dist[i][j] = dist[i][k] + dist[k][j];
            }
        }
    }
}
```

**Полная реализация на Java:**

```java
public class FloydWarshall {
    static final int INF = Integer.MAX_VALUE / 2;

    public static int[][] floydWarshall(int[][] graph) {
        int n = graph.length;
        int[][] dist = new int[n][n];
        for (int i = 0; i < n; i++)
            dist[i] = Arrays.copyOf(graph[i], n);

        for (int k = 0; k < n; k++)
            for (int i = 0; i < n; i++)
                for (int j = 0; j < n; j++)
                    if (dist[i][k] != INF && dist[k][j] != INF)
                        dist[i][j] = Math.min(dist[i][j], dist[i][k] + dist[k][j]);

        // Проверка отрицательных циклов
        for (int t = 0; t < n; t++)
            if (dist[t][t] < 0)
                throw new RuntimeException("Отрицательный цикл через вершину " + t);

        return dist;
    }
}
```

**Почему O(V³)**

Три вложенных цикла по V вершинам: для каждого из V значений промежуточной вершины k, алгоритм проверяет V² пар (i, j). Суммарно: V × V × V = V³ операций.

**Смысл рекуррентного соотношения**

Ключевая идея: если мы уже знаем кратчайшие пути, использующие только вершины из множества {0, 1, ..., k−1}, то добавив вершину k как возможную промежуточную, можем получить более короткий путь: `i → k → j`. После n фаз все вершины рассмотрены как промежуточные — расстояния оптимальны.

**Обнаружение отрицательного цикла**

Если вершина t входит в отрицательный цикл, то алгоритм уменьшит `d[t][t]` ниже 0. Пара (i, j) недостижима с конечным минимальным расстоянием, если любой путь от i до j проходит через вершину с `d[t][t] < 0`.

**Когда использовать Флойда–Уоршелла vs Дейкстру:**

| Критерий | Флойд–Уоршелл | Дейкстра (n раз) |
|---|---|---|
| Задача | All-Pairs | Single-Source (или All-Pairs) |
| Отрицательные веса | Да (без циклов) | Нет |
| Сложность (плотный граф) | O(V³) | O(V² · V) = O(V³) — аналогично |
| Сложность (разреженный граф) | O(V³) | O(V · E log V) — лучше |
| Простота реализации | Очень простая | Сложнее |

Флойд–Уоршелл предпочтителен для плотных графов, где число рёбер E близко к V², и для задач, требующих расстояний между всеми парами. Для разреженных графов выгоднее запускать Дейкстру из каждой вершины (алгоритм Джонсона).

### Вопрос 53. Опишите алгоритм Крускала для построения минимального остовного дерева (MST). Почему для обнаружения циклов используется Union-Find, а не DFS? Какова итоговая временная сложность?

**Категория:** Алгоритмы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Minimum Spanning Tree — Kruskal with Disjoint Set Union — Algorithms for Competitive Programming (cp-algorithms.com)](https://cp-algorithms.com/graph/mst_kruskal_with_dsu.html)

Алгоритм Крускала строит минимальное остовное дерево (MST) жадным методом: рёбра сортируются по весу, затем добавляются по одному в порядке возрастания. Ребро добавляется в MST только тогда, когда его концы принадлежат разным компонентам связности (то есть добавление не создаёт цикл).

**Шаги алгоритма:**
1. Отсортировать все рёбра по весу в порядке невозрастания их значений.
2. Инициализировать DSU: каждая вершина — отдельное множество.
3. Для каждого ребра (u, v) вызвать `find_set(u)` и `find_set(v)`: если результаты различны, добавить ребро в MST и вызвать `union_sets(u, v)`.
4. Остановиться, когда добавлено V−1 рёбер (MST готово) или рёбра закончились.

**Временная сложность:** O(m log n), где m — число рёбер, n — число вершин. Сортировка рёбер занимает O(m log m), а операции DSU — почти константное время (O(α(n))). Поскольку m ≤ n², то log m ≤ 2 log n, итого O(m log n).

#### Ответ от Claude

**Что такое MST**

Минимальное остовное дерево (MST) — это остовное дерево (подграф, соединяющий все вершины без циклов), суммарный вес рёбер которого минимален. MST существует тогда и только тогда, когда граф связный.

**Алгоритм Крускала**

Жадный подход: добавляем рёбра в MST в порядке возрастания веса, пропуская те, что замыкают цикл.

```java
import java.util.*;

public class Kruskal {
    static int[] parent, rank;

    static int find(int x) {
        if (parent[x] != x) parent[x] = find(parent[x]); // сжатие пути
        return parent[x];
    }

    static boolean union(int x, int y) {
        int rx = find(x), ry = find(y);
        if (rx == ry) return false; // уже в одном компоненте — цикл
        if (rank[rx] < rank[ry]) { int t = rx; rx = ry; ry = t; }
        parent[ry] = rx;
        if (rank[rx] == rank[ry]) rank[rx]++;
        return true;
    }

    public static int kruskal(int n, int[][] edges) {
        // edges[i] = {weight, u, v}
        Arrays.sort(edges, Comparator.comparingInt(e -> e[0]));
        parent = new int[n]; rank = new int[n];
        for (int i = 0; i < n; i++) parent[i] = i;

        int mstWeight = 0, edgesAdded = 0;
        for (int[] e : edges) {
            if (union(e[1], e[2])) {
                mstWeight += e[0];
                if (++edgesAdded == n - 1) break; // MST готово
            }
        }
        return mstWeight;
    }
}
```

**Почему Union-Find, а не DFS для обнаружения циклов**

DFS для проверки цикла при добавлении ребра (u, v) потребует обхода всего текущего дерева за O(V) в худшем случае. При m рёбрах суммарно это даёт O(V · m) — слишком медленно.

Union-Find (с path compression и union by rank) проверяет, принадлежат ли u и v одному компоненту, за амортизированное O(α(n)) — практически O(1). Это делает суммарную работу над DSU-операциями незначительной по сравнению с сортировкой.

**Итоговая сложность:**

| Этап | Сложность |
|---|---|
| Сортировка рёбер | O(E log E) |
| DSU-операции (суммарно) | O(E · α(V)) ≈ O(E) |
| Итого | O(E log E) = O(E log V) |

**Сравнение с алгоритмом Прима:**

| Критерий | Крускал | Прим (с кучей) |
|---|---|---|
| Сложность | O(E log E) | O((V + E) log V) |
| Лучший для | Разреженных графов | Плотных графов |
| Подход | Глобальный (сортировка рёбер) | Локальный (рост от одной вершины) |

**Применение MST:** алгоритмы кластеризации, сети электропередачи и водоснабжения, дорожные сети, алгоритм аппроксимации задачи коммивояжёра (TSP 2-approximation).

### Вопрос 54. Как работает алгоритм Кнута–Морриса–Пратта (KMP) для поиска образца в строке? Что такое префикс-функция (LPS-массив) и зачем она нужна? Какова временная и пространственная сложность?

**Категория:** Алгоритмы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [KMP Algorithm for Pattern Searching — InterviewBit](https://www.interviewbit.com/blog/kmp-algorithm-pattern-search/)

KMP (Knuth–Morris–Pratt) — алгоритм поиска образца в тексте, достигающий линейного времени O(n + m), где n — длина текста, m — длина образца. Ключевое отличие от наивного подхода: при несовпадении символов алгоритм не возвращается к началу образца, а использует предварительно вычисленную информацию о самом образце.

**LPS-массив (Longest Proper Prefix which is also Suffix):** для каждой позиции i в образце хранит длину наибольшего собственного префикса образца `pattern[0..i]`, который одновременно является его суффиксом. Например, для "abcabd": LPS = [0, 0, 0, 1, 2, 0].

**Как LPS используется при поиске:** при несовпадении `text[i] != pattern[j]` вместо возврата к `j=0` (наивный подход) мы устанавливаем `j = LPS[j-1]`. Это означает: «у нас уже есть совпадение подстроки длиной LPS[j-1], начинать заново не нужно». Каждый символ текста обрабатывается не более одного раза.

**Сложность:** время — O(n + m), где O(m) тратится на построение LPS, O(n) — на поиск. Память — O(m) для хранения LPS.

#### Ответ от Claude

**Проблема наивного поиска**

Наивный алгоритм: для каждой позиции i в тексте начинать попытку совпадения с начала образца. Сложность O(n · m) — в худшем случае (например, текст `"aaaaab"`, образец `"aaab"`) мы многократно сравниваем уже проверенные символы.

**Идея KMP: не выбрасывать информацию о совпадении**

После частичного совпадения `pattern[0..j-1]` с `text[i-j..i-1]` и несовпадения на позиции j мы уже знаем, какие символы текста и образца были сопоставлены. LPS-массив позволяет «перепрыгнуть» к следующей потенциальной позиции.

**Построение LPS-массива:**

```java
// LPS[i] = длина наибольшего собственного префикса pattern[0..i],
//           который также является суффиксом pattern[0..i]
int[] buildLPS(String pattern) {
    int m = pattern.length();
    int[] lps = new int[m];
    int len = 0, i = 1;
    while (i < m) {
        if (pattern.charAt(i) == pattern.charAt(len)) {
            lps[i++] = ++len;
        } else if (len != 0) {
            len = lps[len - 1]; // не сбрасываем i
        } else {
            lps[i++] = 0;
        }
    }
    return lps;
}
```

**Поиск образца с KMP:**

```java
List<Integer> kmpSearch(String text, String pattern) {
    List<Integer> result = new ArrayList<>();
    int n = text.length(), m = pattern.length();
    int[] lps = buildLPS(pattern);
    int i = 0, j = 0;
    while (i < n) {
        if (text.charAt(i) == pattern.charAt(j)) {
            i++; j++;
        }
        if (j == m) {
            result.add(i - j); // нашли вхождение
            j = lps[j - 1];   // продолжаем поиск
        } else if (i < n && text.charAt(i) != pattern.charAt(j)) {
            if (j != 0) j = lps[j - 1];
            else i++;
        }
    }
    return result;
}
```

**Пример работы LPS для образца "AAACAAAA":**

| i | 0 | 1 | 2 | 3 | 4 | 5 | 6 | 7 |
|---|---|---|---|---|---|---|---|---|
| Символ | A | A | A | C | A | A | A | A |
| LPS | 0 | 1 | 2 | 0 | 1 | 2 | 3 | 3 |

При несовпадении на позиции j=6, KMP переходит к j = LPS[5] = 2, сохраняя уже известное совпадение "AA".

**Сложность:**

| | Наивный поиск | KMP |
|---|---|---|
| Время | O(n · m) | O(n + m) |
| Память | O(1) | O(m) для LPS |

**Расхождение между источниками:** некоторые источники называют массив «failure function», «partial match table» или «pi-функция» — все эти термины описывают одно и то же: LPS[i] = длина наибольшего совпадающего префикса-суффикса. Различия только в названии, не в содержании.

### Вопрос 55. Что такое алгоритм Рабина–Карпа и как работает «скользящий хеш» (rolling hash)? В чём преимущество перед KMP? Каковы временная сложность в среднем и в худшем случае?

**Категория:** Алгоритмы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Rabin–Karp string search algorithm — Algorithms for Competitive Programming (cp-algorithms.com)](https://cp-algorithms.com/string/rabin-karp.html)

Алгоритм Рабина–Карпа (1987) использует хеширование для поиска образца в тексте. Вместо посимвольного сравнения строк вычисляются их хеши: хеш образца сравнивается с хешем каждого окна текста той же длины.

**Скользящий хеш (rolling hash):** применяется полиномиальное хеширование с основанием p и модулем m (обычно p=31, m=10⁹+9). Хеш подстроки `text[i..i+|pattern|−1]` вычисляется за O(1) через префиксные хеши: `hash(i..j) = (prefix[j+1] − prefix[i] · p^(j−i+1)) mod m`.

**Среднее время:** O(|text| + |pattern|). **Худший случай:** O(|text| · |pattern|) — при многочисленных коллизиях хешей алгоритм вынужден делать посимвольное сравнение.

**Преимущество перед KMP:** Рабин–Карп легко обобщается на поиск нескольких образцов одновременно — хеши всех образцов добавляются в HashSet, и каждое окно текста проверяется за O(1). KMP для k образцов потребует либо k отдельных запусков, либо алгоритма Ахо–Корасик.

#### Ответ от Claude

**Центральная идея: сравнивать хеши, а не строки**

Наивный подход требует O(m) для сравнения каждого окна с образцом. Если вычислить хеши за O(1) с помощью скользящего окна, то сравнение хешей займёт O(1). Посимвольное сравнение происходит только при совпадении хешей (для подтверждения, что это не коллизия).

**Полиномиальный хеш и скользящее окно:**

Хеш строки s длиной m: `H(s) = s[0]·p^(m-1) + s[1]·p^(m-2) + ... + s[m-1]·p^0`.

При сдвиге окна на 1 вправо:
- Убираем левый символ: вычитаем `s[0] · p^(m-1)`.
- Добавляем правый символ: добавляем `s[m] · p^0`.

Пересчёт за O(1) вместо O(m) — это и есть rolling hash.

**Реализация на Java через префиксные хеши:**

```java
public class RabinKarp {
    static final long BASE = 31L;
    static final long MOD = 1_000_000_007L;

    public static List<Integer> search(String text, String pattern) {
        int n = text.length(), m = pattern.length();
        List<Integer> result = new ArrayList<>();
        if (m > n) return result;

        // Предвычисление степеней BASE и префиксных хешей текста
        long[] pow = new long[n + 1];
        long[] prefHash = new long[n + 1];
        pow[0] = 1;
        for (int i = 1; i <= n; i++) {
            pow[i] = pow[i - 1] * BASE % MOD;
            prefHash[i] = (prefHash[i - 1] * BASE + (text.charAt(i - 1) - 'a' + 1)) % MOD;
        }

        // Хеш образца
        long patHash = 0;
        for (int i = 0; i < m; i++)
            patHash = (patHash * BASE + (pattern.charAt(i) - 'a' + 1)) % MOD;

        // Поиск: проверяем каждое окно
        for (int i = 0; i <= n - m; i++) {
            long windowHash = (prefHash[i + m] - prefHash[i] * pow[m] % MOD + MOD * MOD) % MOD;
            if (windowHash == patHash) {
                // Подтверждаем посимвольно (избегаем ложных срабатываний)
                if (text.regionMatches(i, pattern, 0, m))
                    result.add(i);
            }
        }
        return result;
    }
}
```

**Временная сложность:**

| Случай | Сложность | Причина |
|---|---|---|
| Среднее | O(n + m) | Мало коллизий; хеши сравниваются за O(1) |
| Худший | O(n · m) | Все окна дают коллизию → посимвольное сравнение каждый раз |

Для снижения вероятности коллизий применяют двойное хеширование (два разных основания и модуля) или выбирают большой простой модуль.

**Главное преимущество перед KMP: поиск k образцов**

KMP: O((n + m) · k) — k отдельных запусков.
Рабин–Карп: O(n + m₁ + m₂ + ... + mₖ) — вычисляем хеши k образцов, кладём их в HashSet, а текст обходим один раз. Подходит для задач типа «найти все слова из словаря в тексте».

### Вопрос 56. Что такое задача о наибольшей возрастающей подпоследовательности (LIS)? Как решить её за O(n²) через DP и за O(n log n) с помощью бинарного поиска? В чём смысл массива «хвостов»?

**Категория:** Алгоритмы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Longest increasing subsequence — Algorithms for Competitive Programming (cp-algorithms.com)](https://cp-algorithms.com/dynamic_programming/longest_increasing_subsequence.html)

Задача LIS (Longest Increasing Subsequence): найти длину наибольшей строго возрастающей подпоследовательности данного массива. Подпоследовательность — элементы в исходном относительном порядке, не обязательно непрерывные.

**O(n²) DP:** массив `d[i]` хранит длину LIS, оканчивающейся на элементе с индексом i. Переход: `d[i] = max(d[j] + 1)` для всех j < i, где `a[j] < a[i]`. Ответ — максимум по всем `d[i]`.

**O(n log n) через бинарный поиск:** переопределяем `d`: теперь `d[l]` — наименьший элемент, которым может оканчиваться возрастающая подпоследовательность длиной l. Ключевое свойство: «массив d всегда будет отсортирован» — потому что из каждой LIS длиной l можно получить LIS длиной l−1 с не бо́льшим последним элементом. Это позволяет применять бинарный поиск: для нового элемента `a[i]` находим первую позицию в d, которая больше или равна `a[i]`, и обновляем её. Итог: O(n log n).

#### Ответ от Claude

**Постановка задачи**

Дан массив `a[0..n-1]`. Найти длину наибольшей подпоследовательности, в которой все элементы строго возрастают. Например, для `[3, 1, 4, 1, 5, 9, 2, 6]` LIS = `[1, 4, 5, 9]` или `[1, 4, 5, 6]` длиной 4.

**Решение 1 — DP за O(n²):**

```java
public int lisDP(int[] a) {
    int n = a.length;
    int[] dp = new int[n];
    Arrays.fill(dp, 1); // каждый элемент — LIS длиной 1
    int maxLen = 1;
    for (int i = 1; i < n; i++) {
        for (int j = 0; j < i; j++) {
            if (a[j] < a[i]) {
                dp[i] = Math.max(dp[i], dp[j] + 1);
            }
        }
        maxLen = Math.max(maxLen, dp[i]);
    }
    return maxLen;
}
```

**Решение 2 — O(n log n) через массив хвостов:**

Массив `tails` (хвостов): `tails[l]` — наименьший возможный последний элемент LIS длиной `l+1`. Этот массив всегда отсортирован, что позволяет применять бинарный поиск.

Для каждого нового элемента `a[i]`:
- Если `a[i] > tails[len−1]`: добавляем `a[i]` в конец → LIS стала длиннее на 1.
- Иначе: находим бинарным поиском наименьший элемент в `tails`, который `>= a[i]`, и заменяем его на `a[i]` (уменьшаем «хвост», открывая больше возможностей для будущих элементов).

```java
public int lisOptimal(int[] a) {
    int n = a.length;
    int[] tails = new int[n];
    int len = 0;
    for (int x : a) {
        // Бинарный поиск: первая позиция в tails[0..len-1], где tails[pos] >= x
        int lo = 0, hi = len;
        while (lo < hi) {
            int mid = (lo + hi) / 2;
            if (tails[mid] < x) lo = mid + 1;
            else hi = mid;
        }
        tails[lo] = x;     // заменяем или расширяем
        if (lo == len) len++;
    }
    return len;
}
```

**Трассировка на примере `[3, 1, 4, 1, 5, 9, 2, 6]`:**

| Элемент | tails после | len |
|---|---|---|
| 3 | [3] | 1 |
| 1 | [1] | 1 |
| 4 | [1, 4] | 2 |
| 1 | [1, 4] | 2 |
| 5 | [1, 4, 5] | 3 |
| 9 | [1, 4, 5, 9] | 4 |
| 2 | [1, 2, 5, 9] | 4 |
| 6 | [1, 2, 5, 6] | 4 |

Ответ: len = 4.

**Важное уточнение:** массив `tails` не является самой LIS — это вспомогательная структура. Для восстановления конкретной подпоследовательности нужно хранить дополнительный массив предшественников.

**Сравнение подходов:**

| | O(n²) DP | O(n log n) |
|---|---|---|
| Сложность времени | O(n²) | O(n log n) |
| Память | O(n) | O(n) |
| Восстановление LIS | Легко через predecessors | Требует дополнительной работы |

### Вопрос 57. Что такое дерево отрезков (Segment Tree)? Как оно обрабатывает запросы на отрезке (range query) и точечные обновления (point update) за O(log n)? Какова пространственная сложность и зачем выделяют массив размером 4n?

**Категория:** Алгоритмы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Segment Trees Tutorials & Notes — HackerEarth](https://www.hackerearth.com/practice/data-structures/advanced-data-structures/segment-trees/tutorial/)

Дерево отрезков — бинарное дерево, позволяющее эффективно выполнять запросы на произвольных подотрезках массива и обновления элементов. В отличие от наивного подхода O(n) на запрос, оно обрабатывает как запросы, так и обновления за O(log n).

**Основные операции:**
- **Build:** построение дерева по исходному массиву за O(n).
- **Query:** получение агрегированного значения (сумма, минимум, максимум) на отрезке [l, r] за O(log n).
- **Update:** изменение одного элемента массива с пересчётом дерева за O(log n).

**Пространственная сложность:** O(n) — в дереве 2n−1 внутренних узлов. Рекурсивная реализация традиционно выделяет массив размером 4n для удобства адресации узлов без вычисления точных границ.

**Применение:** запросы суммы на отрезке с модификациями, поиск минимума/максимума (RMQ), задачи с диапазонными обновлениями через lazy propagation.

#### Ответ от Claude

**Что такое дерево отрезков**

Дерево отрезков (Segment Tree) — полное бинарное дерево, каждый узел которого хранит агрегированное значение (сумму, минимум, максимум и т. д.) для подотрезка исходного массива. Корень хранит агрегат всего массива, листья — отдельные элементы.

Для массива размером n дерево имеет высоту ⌈log₂ n⌉. На каждом уровне родительский узел для отрезка [l, r] имеет двух потомков: для [l, mid] и [mid+1, r], где mid = (l + r) / 2.

**Реализация на Java (сумма на отрезке):**

```java
public class SegmentTree {
    private int[] tree;
    private int n;

    public SegmentTree(int[] arr) {
        n = arr.length;
        tree = new int[4 * n]; // 4n для безопасной адресации
        build(arr, 0, 0, n - 1);
    }

    private void build(int[] arr, int node, int start, int end) {
        if (start == end) {
            tree[node] = arr[start];
        } else {
            int mid = (start + end) / 2;
            build(arr, 2 * node + 1, start, mid);
            build(arr, 2 * node + 2, mid + 1, end);
            tree[node] = tree[2 * node + 1] + tree[2 * node + 2];
        }
    }

    // Точечное обновление: arr[idx] += val
    public void update(int node, int start, int end, int idx, int val) {
        if (start == end) {
            tree[node] += val;
        } else {
            int mid = (start + end) / 2;
            if (idx <= mid) update(2 * node + 1, start, mid, idx, val);
            else             update(2 * node + 2, mid + 1, end, idx, val);
            tree[node] = tree[2 * node + 1] + tree[2 * node + 2];
        }
    }

    // Запрос суммы на отрезке [l, r]
    public int query(int node, int start, int end, int l, int r) {
        if (r < start || end < l) return 0;     // нет пересечения
        if (l <= start && end <= r) return tree[node]; // полное покрытие
        int mid = (start + end) / 2;
        return query(2 * node + 1, start, mid, l, r)
             + query(2 * node + 2, mid + 1, end, l, r);
    }
}
```

**Почему O(log n) на запрос и обновление**

При запросе на [l, r] узел либо полностью покрыт запросом (возвращает значение сразу), либо частично — делится на двух потомков. На каждом уровне дерева обрабатывается не более 4 узлов. При высоте ⌈log₂ n⌉ суммарно O(log n) узлов.

При обновлении элемента изменяется цепочка от листа до корня — ровно ⌈log₂ n⌉ узлов.

**Почему 4n, а не 2n?**

При нечётном n или при рекурсивном разбиении «правый» последний уровень может иметь пустые узлы. Математически дерево с n листьями может иметь до 4n − 5 узлов в рекурсивной индексации через `2*node+1` и `2*node+2`. Размер 4n — это безопасный верхний предел, позволяющий не вычислять точный размер перед построением.

**Сравнение с альтернативами:**

| Структура | Build | Query | Update | Память |
|---|---|---|---|---|
| Наивный массив | O(n) | O(n) | O(1) | O(n) |
| Префиксные суммы | O(n) | O(1) | O(n) | O(n) |
| Дерево отрезков | O(n) | O(log n) | O(log n) | O(n) |
| Дерево Фенвика (BIT) | O(n log n) | O(log n) | O(log n) | O(n) |

Дерево отрезков предпочтительнее при необходимости диапазонных обновлений (lazy propagation), нестандартных агрегатных функциях или произвольных запросах.

### Вопрос 58. Что такое дерево Фенвика (Binary Indexed Tree / Fenwick Tree)? Как работают операции update и prefixSum через бит-манипуляции с lowbit? Чем оно проще дерева отрезков при одинаковой сложности O(log n)?

**Категория:** Алгоритмы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Binary Indexed Tree or Fenwick Tree — GeeksforGeeks](https://www.geeksforgeeks.org/dsa/binary-indexed-tree-or-fenwick-tree-2/)

Дерево Фенвика (Binary Indexed Tree, BIT) — структура данных, реализованная в виде обычного массива с 1-индексацией, позволяющая выполнять префиксные суммы и точечные обновления за O(log n).

Каждый индекс `i` в BIT-массиве хранит сумму группы элементов, размер которой определяется наименьшим установленным битом (lowbit) числа i: `lowbit(i) = i & (-i)`.

**Операция prefixSum(i):** суммирует элементы, двигаясь по индексам, вычитая lowbit: `i → i − lowbit(i)` до i = 0. Максимальный число шагов — O(log n).

**Операция update(i, delta):** добавляет delta к позиции i и всем родительским узлам, прибавляя lowbit: `i → i + lowbit(i)` до выхода за границы. Тоже O(log n).

Дерево Фенвика требует O(n) памяти и проще дерева отрезков в реализации: всего 10–15 строк кода вместо 50–80.

#### Ответ от Claude

**Идея BIT: хранить частичные суммы с умными границами**

Вместо хранения всего префикса или отдельного элемента, каждый индекс i в BIT хранит сумму блока элементов. Размер блока — `lowbit(i) = i & (-i)` (выделение наименьшего установленного бита):

- `BIT[4]` (100₂) хранит сумму `a[1..4]` — блок из 4 элементов.
- `BIT[6]` (110₂) хранит сумму `a[5..6]` — блок из 2 элементов.
- `BIT[7]` (111₂) хранит `a[7]` — блок из 1 элемента.

**Операция prefixSum(i) — сумма a[1..i]:**

Собираем блоки, двигаясь от i к 0, каждый раз вычитая lowbit:

```java
int prefixSum(int[] bit, int i) {
    int sum = 0;
    for (; i > 0; i -= i & (-i)) {
        sum += bit[i];
    }
    return sum;
}
```

Для i = 7 (111₂): bit[7] + bit[6] + bit[4] = a[7] + a[5..6] + a[1..4] = a[1..7].

**Операция update(i, delta):**

Обновляем блок i и все родительские блоки, прибавляя lowbit:

```java
void update(int[] bit, int n, int i, int delta) {
    for (; i <= n; i += i & (-i)) {
        bit[i] += delta;
    }
}
```

**Запрос суммы на произвольном отрезке [l, r]:**

`rangeSum(l, r) = prefixSum(r) − prefixSum(l − 1)`.

**Полная реализация:**

```java
public class FenwickTree {
    private final int[] bit;
    private final int n;

    public FenwickTree(int[] arr) {
        n = arr.length;
        bit = new int[n + 1];           // 1-индексация
        for (int i = 0; i < n; i++)
            update(i + 1, arr[i]);      // строим BIT за O(n log n)
    }

    public void update(int i, int delta) {
        for (; i <= n; i += i & (-i)) bit[i] += delta;
    }

    public int prefixSum(int i) {
        int sum = 0;
        for (; i > 0; i -= i & (-i)) sum += bit[i];
        return sum;
    }

    public int rangeSum(int l, int r) {
        return prefixSum(r) - prefixSum(l - 1);
    }
}
```

**Почему проще дерева отрезков:**

| Критерий | Дерево Фенвика | Дерево отрезков |
|---|---|---|
| Размер кода | ~15 строк | ~50–80 строк |
| Память | O(n) — один массив | O(4n) — четыре массива или рекурсия |
| Произвольная агрегатная функция | Нет (только обратимые: сумма, XOR) | Да (min, max, GCD и т. д.) |
| Диапазонные обновления | Не напрямую | Да (lazy propagation) |

**Применение BIT на практике:** подсчёт инверсий в массиве, задача «число элементов, меньших x, в prefix[0..i]», онлайн-суммы с обновлениями, задачи порядковой статистики.

### Вопрос 59. В чём принципиальное отличие неосновных (non-comparison-based) сортировок от сортировок сравнением? Опишите Counting Sort, Radix Sort и Bucket Sort: когда применяется каждый, каковы их временные сложности и ограничения?

**Категория:** Алгоритмы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Counting sort — Wikipedia](https://en.wikipedia.org/wiki/Counting_sort)

Сортировки, не основанные на сравнении, обходят теоретический нижний предел O(n log n) для сортировок сравнением, используя дополнительную информацию о природе ключей (например, ограниченный диапазон целых чисел).

**Counting Sort:** создаёт массив счётчиков размером k (диапазон ключей), подсчитывает вхождения, затем перестраивает результат. Временная сложность: Θ(n + k), пространственная: O(n + k). Алгоритм стабилен. Применим при ключах в небольшом диапазоне [0, k).

**Radix Sort:** сортирует по отдельным «разрядам» ключа (цифрам числа по основанию b) от наименее значимого к наиболее значимому (LSD-вариант). В каждом проходе применяется стабильная сортировка (обычно Counting Sort). Временная сложность: Θ(d · (n + b)), где d — количество разрядов, b — основание системы счисления. При фиксированном b и d = O(log_b(max_key)) даёт O(n).

**Bucket Sort:** равномерно распределяет элементы по «вёдрам», каждое ведро сортируется отдельно (обычно сортировкой вставками). Эффективен при равномерно распределённых данных: O(n) в среднем, O(n²) в худшем.

#### Ответ от Claude

**Нижний предел сортировок сравнением и как его обойти**

Доказано, что любой алгоритм сортировки, основанный на попарном сравнении элементов, требует не менее Ω(n log n) операций в худшем случае. Это следует из того, что дерево решений имеет n! листьев и высоту не меньше log₂(n!) ≈ n log₂ n.

Non-comparison сортировки не сравнивают элементы между собой — они используют числовые свойства ключей, что позволяет работать за O(n) при определённых условиях.

---

**Counting Sort — сортировка подсчётом**

Условие применимости: ключи — целые числа в диапазоне [0, k).

```java
int[] countingSort(int[] arr, int k) {
    int[] count = new int[k];
    for (int x : arr) count[x]++;           // подсчёт вхождений
    for (int i = 1; i < k; i++) count[i] += count[i - 1]; // префиксные суммы
    int[] out = new int[arr.length];
    for (int i = arr.length - 1; i >= 0; i--) // обход справа налево для стабильности
        out[--count[arr[i]]] = arr[i];
    return out;
}
```

- Время: Θ(n + k). Если k = O(n) — линейное.
- Память: O(n + k).
- Стабилен: равные элементы сохраняют исходный порядок.
- Ограничение: при большом k (например, k = 10⁹) расходует огромную память.

---

**Radix Sort — поразрядная сортировка**

Сортирует числа по разрядам, от наименее значимого к наиболее (LSD), применяя на каждом шаге стабильную сортировку (Counting Sort по одному разряду).

```java
void radixSort(int[] arr) {
    int max = Arrays.stream(arr).max().getAsInt();
    for (int exp = 1; max / exp > 0; exp *= 10)
        countByDigit(arr, exp); // Counting Sort по разряду exp
}
```

- Время: Θ(d · (n + b)), где d — число разрядов, b — основание (обычно 10 или 256).
- При числах до 10^9 (d = 9, b = 10): 9 проходов.
- Стабилен (LSD-вариант).
- Ограничение: применим только к ключам с фиксированной разрядностью (целые числа, строки фиксированной длины).

---

**Bucket Sort — сортировка корзинами**

Элементы распределяются по равномерным «вёдрам» по диапазону значений, каждое ведро сортируется отдельно.

```java
void bucketSort(float[] arr, int n) {
    List<List<Float>> buckets = new ArrayList<>();
    for (int i = 0; i < n; i++) buckets.add(new ArrayList<>());
    for (float x : arr) buckets.get((int)(x * n)).add(x);
    int idx = 0;
    for (List<Float> bucket : buckets) {
        Collections.sort(bucket); // вставочная сортировка внутри ведра
        for (float x : bucket) arr[idx++] = x;
    }
}
```

- Среднее время: O(n) при равномерном распределении.
- Худшее время: O(n²) — все элементы в одном ведре.
- Применим для вещественных чисел в [0, 1) или при известном равномерном распределении ключей.

| Алгоритм | Время (среднее) | Время (худшее) | Память | Стабильность | Условие применения |
|---|---|---|---|---|---|
| Counting Sort | Θ(n + k) | Θ(n + k) | O(n + k) | Да | Целые числа, малый диапазон k |
| Radix Sort | Θ(d·(n+b)) | Θ(d·(n+b)) | O(n + b) | Да (LSD) | Фиксированная разрядность |

| Merge Sort | O(n log n) | O(n log n) | O(n) | Да | Общий случай |
| Quick Sort | O(n log n) | O(n²) | O(log n) | Нет | Общий случай |

### Вопрос 60. Что такое массив префиксных сумм (prefix sum array)? Как с его помощью отвечать на запросы суммы на отрезке [l, r] за O(1)? Какова сложность построения и почему эта структура не поддерживает эффективные обновления?

**Категория:** Алгоритмы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Prefix Sum Array — GeeksforGeeks](https://www.geeksforgeeks.org/dsa/prefix-sum-array-implementation-applications-competitive-programming/)

Массив префиксных сумм `prefix[i]` — вспомогательный массив, где каждый элемент хранит сумму всех элементов исходного массива от индекса 0 до i включительно: `prefix[i] = arr[0] + arr[1] + ... + arr[i]`.

Сумма на произвольном отрезке [l, r] вычисляется за O(1): `rangeSum(l, r) = prefix[r] − prefix[l−1]` (при l > 0; если l = 0, то просто `prefix[r]`).

**Построение:** O(n) — один проход по массиву с накоплением суммы. **Запрос:** O(1). **Пространство:** O(n).

**Ограничение:** при изменении одного элемента `arr[i]` необходимо пересчитать все `prefix[i], prefix[i+1], ..., prefix[n-1]` — O(n) на каждое обновление. Поэтому для задач с частыми обновлениями используют Дерево Фенвика или Дерево отрезков.

#### Ответ от Claude

**Что такое префиксная сумма и как она строится**

Массив префиксных сумм — это предвычисленный массив, позволяющий отвечать на запросы суммы подотрезка за O(1) вместо O(n).

```java
int[] buildPrefix(int[] arr) {
    int n = arr.length;
    int[] prefix = new int[n + 1]; // prefix[0] = 0 — страж
    for (int i = 0; i < n; i++) {
        prefix[i + 1] = prefix[i] + arr[i];
    }
    return prefix;
}
```

После построения: `prefix[i]` = сумма `arr[0] + arr[1] + ... + arr[i-1]` (при 1-индексации с нулевым стражом).

**Запрос суммы на отрезке [l, r] за O(1):**

```java
int rangeSum(int[] prefix, int l, int r) {
    // l и r — 0-индексированные границы в arr
    return prefix[r + 1] - prefix[l];
}
```

Формула работает потому что: `prefix[r+1] = arr[0]+...+arr[r]`, `prefix[l] = arr[0]+...+arr[l-1]`. Разность — в точности `arr[l]+...+arr[r]`.

**Пример:**

```
arr    = [3, 1, 4, 1, 5, 9, 2, 6]
prefix = [0, 3, 4, 8, 9,14,23,25,31]

rangeSum(2, 5) = prefix[6] − prefix[2] = 23 − 4 = 19
// arr[2]+arr[3]+arr[4]+arr[5] = 4+1+5+9 = 19 ✓
```

**Двумерный вариант (2D prefix sum):**

Для матриц применяется 2D префиксная сумма для запросов суммы в прямоугольнике за O(1):

```java
prefix[i][j] = arr[i][j] + prefix[i-1][j] + prefix[i][j-1] - prefix[i-1][j-1];
// Сумма прямоугольника (r1,c1)-(r2,c2):
int rectSum = prefix[r2][c2] - prefix[r1-1][c2] - prefix[r2][c1-1] + prefix[r1-1][c1-1];
```

**Почему обновления дороги**

Массив prefix хранит накопленные суммы: `prefix[k] = sum(arr[0..k-1])`. При изменении `arr[i]` на delta каждое `prefix[k]` для k > i увеличивается на delta — все правые элементы зависят от изменённого. Пересчёт: O(n).

**Сравнение структур для задачи «сумма с обновлениями»:**

| Структура | Build | Query [l,r] | Update |
|---|---|---|---|
| Префиксная сумма | O(n) | O(1) | O(n) |
| Дерево Фенвика | O(n log n) | O(log n) | O(log n) |
| Дерево отрезков | O(n) | O(log n) | O(log n) |

**Вывод:** префиксная сумма оптимальна для статических данных (нет или редкие обновления) с частыми запросами суммы. При частых обновлениях следует использовать Дерево Фенвика или Дерево отрезков.
