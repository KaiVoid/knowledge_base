### Вопрос 1. Что такое координаты артефакта в Maven? Из каких обязательных элементов они состоят и зачем нужен каждый?

**Категория:** Maven/Gradle · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Maven Artifacts – Apache Maven](https://maven.apache.org/repositories/artifacts.html)

Координаты Maven-артефакта однозначно идентифицируют артефакт в репозитории. Они чаще всего записываются в форме `groupId:artifactId:version` (сокращённо — GAV). Каждый артефакт должен иметь уникальную комбинацию этих трёх идентификаторов.

- **groupId** — идентифицирует группу (организацию/пространство имён), к которой принадлежит проект. По соглашению используется обратная нотация домена: `org.apache.maven`.
- **artifactId** — уникальное имя артефакта внутри группы. Является основой для имени файла.
- **version** — указывает конкретную версию артефакта. Рекомендуется следовать Semantic Versioning (`MAJOR.MINOR.PATCH`).

Дополнительно существуют необязательные элементы:
- **classifier** — вариант артефакта (например, `sources`, `javadoc`, `tests`).
- **extension/type** — тип файла (по умолчанию `jar`).

#### Ответ от Claude

**Координаты артефакта (GAV-координаты)** — это набор идентификаторов, позволяющих Maven однозначно найти, скачать или опубликовать артефакт в репозитории. Аббревиатура GAV расшифровывается как **G**roupId · **A**rtifactId · **V**ersion.

В `pom.xml` все три элемента обязательны:

```xml
<groupId>com.example</groupId>
<artifactId>my-app</artifactId>
<version>1.0.0</version>
```

**groupId**
- Идентифицирует организацию или проектную группу.
- По соглашению записывается в обратной доменной нотации: `com.google.guava`, `org.springframework`.
- Должен следовать правилам именования Java-пакетов.
- Позволяет группировать несколько связанных артефактов (например, все модули Spring Framework имеют `org.springframework`).

**artifactId**
- Уникальное имя конкретного артефакта внутри группы.
- По соглашению — только строчные буквы, цифры и дефисы: `spring-core`, `commons-math`.
- Используется как основа имени результирующего файла: `my-app-1.0.0.jar`.

**version**
- Определяет конкретный выпуск артефакта.
- Рекомендуется формат Semantic Versioning: `MAJOR.MINOR.PATCH` (например, `2.5.3`).
- Суффикс `-SNAPSHOT` означает нестабильную разрабатываемую версию; Maven хранит снэпшоты отдельно и может их обновлять.

**Необязательные элементы:**
- **packaging** (тип, по умолчанию `jar`) — определяет формат результирующего артефакта (`jar`, `war`, `pom`, `ear`).
- **classifier** — позволяет создавать варианты одного артефакта: `my-app-1.0.0-sources.jar`, `my-app-1.0.0-javadoc.jar`.

**Зачем нужны координаты:**
1. Уникальная идентификация в репозитории — Maven может находить артефакт среди тысяч других.
2. Управление зависимостями — разрешение транзитивных зависимостей и предотвращение конфликтов версий.
3. Предсказуемый путь в репозитории: `~/.m2/repository/com/example/my-app/1.0.0/my-app-1.0.0.jar`.

### Вопрос 2. Перечислите фазы жизненного цикла default (build) в Maven в правильном порядке. Что происходит при запуске команды `mvn package`?

**Категория:** Maven/Gradle · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Introduction to the Build Lifecycle – Apache Maven](https://maven.apache.org/guides/introduction/introduction-to-the-lifecycle.html)

Жизненный цикл default содержит 23 фазы в следующем порядке:

1. `validate` — проверяет корректность проекта и наличие всей необходимой информации.
2. `initialize` — инициализирует состояние сборки (устанавливает свойства, создаёт директории).
3. `generate-sources` — генерирует исходный код для включения в компиляцию.
4. `process-sources` — обрабатывает исходный код (например, фильтрует значения).
5. `generate-resources` — генерирует ресурсы для включения в пакет.
6. `process-resources` — копирует и обрабатывает ресурсы в целевую директорию.
7. `compile` — компилирует исходный код проекта.
8. `process-classes` — постобработка скомпилированных файлов.
9. `generate-test-sources` — генерирует тестовый исходный код.
10. `process-test-sources` — обрабатывает тестовый исходный код.
11. `generate-test-resources` — создаёт ресурсы для тестирования.
12. `process-test-resources` — копирует ресурсы в тестовую директорию.
13. `test-compile` — компилирует тестовый исходный код.
14. `process-test-classes` — постобработка скомпилированных тестовых файлов.
15. `test` — запускает тесты подходящим тестовым фреймворком.
16. `prepare-package` — выполняет операции, необходимые перед упаковкой.
17. `package` — упаковывает скомпилированный код в дистрибутивный формат (JAR и т.д.).
18. `pre-integration-test` — выполняет действия перед интеграционными тестами.
19. `integration-test` — разворачивает и запускает интеграционные тесты.
20. `post-integration-test` — выполняет действия после интеграционных тестов.
21. `verify` — проверяет, что пакет валиден и соответствует критериям качества.
22. `install` — устанавливает пакет в локальный репозиторий.
23. `deploy` — публикует финальный пакет в удалённый репозиторий.

При запуске `mvn package` Maven последовательно выполняет все фазы от `validate` до `package` включительно. Фазы интеграционных тестов (`pre-integration-test`, `integration-test`, `post-integration-test`, `verify`) при этом **не** выполняются.

#### Ответ от Claude

**Жизненный цикл default** описывает полный процесс сборки и развёртывания проекта. Он содержит 23 фазы; ключевые из них (на которые чаще всего привязываются плагины при packaging=jar):

| # | Фаза | Что делает |
|---|------|-----------|
| 1 | `validate` | Проверяет корректность POM и наличие всех необходимых данных |
| 2 | `initialize` | Инициализирует свойства и создаёт директории |
| 3 | `generate-sources` | Генерирует исходный код (например, из WSDL или аннотаций) |
| 4 | `process-sources` | Обрабатывает исходники (фильтрация плейсхолдеров) |
| 5 | `generate-resources` | Генерирует ресурсные файлы |
| 6 | `process-resources` | Копирует ресурсы в `target/classes` |
| 7 | **`compile`** | Компилирует Java-код → `target/classes` |
| 8 | `process-classes` | Постобработка .class-файлов (например, байт-код weaving) |
| 9 | `generate-test-sources` | Генерирует тестовые исходники |
| 10 | `process-test-sources` | Обрабатывает тестовые исходники |
| 11 | `generate-test-resources` | Генерирует тестовые ресурсы |
| 12 | `process-test-resources` | Копирует тестовые ресурсы в `target/test-classes` |
| 13 | `test-compile` | Компилирует тесты → `target/test-classes` |
| 14 | `process-test-classes` | Постобработка тестовых .class-файлов |
| 15 | **`test`** | Запускает unit-тесты (maven-surefire-plugin) |
| 16 | `prepare-package` | Подготовительные действия перед упаковкой |
| 17 | **`package`** | Упаковывает артефакт (JAR/WAR) в `target/` |
| 18 | `pre-integration-test` | Подготовка к интеграционным тестам |
| 19 | `integration-test` | Запуск интеграционных тестов |
| 20 | `post-integration-test` | Очистка после интеграционных тестов |
| 21 | `verify` | Проверяет корректность и качество пакета |
| 22 | **`install`** | Копирует артефакт в локальный репозиторий (`~/.m2`) |
| 23 | **`deploy`** | Публикует артефакт в удалённый репозиторий |

**Что происходит при `mvn package`:**

Maven выполняет все фазы от `validate` до `package` включительно — итого 17 фаз. Конкретно для проекта с `<packaging>jar</packaging>` это означает:
1. Проверяет POM.
2. Компилирует исходный код (`compiler:compile`).
3. Копирует ресурсы (`resources:resources`).
4. Компилирует тесты (`compiler:testCompile`).
5. Запускает unit-тесты (`surefire:test`). Если тест падает — сборка прерывается.
6. Упаковывает классы в JAR-файл (`jar:jar`), помещает его в `target/`.

Фазы `pre-integration-test`, `integration-test`, `post-integration-test`, `verify`, `install`, `deploy` при `mvn package` **не выполняются**.

> Ключевой принцип: запуск любой фазы автоматически запускает все предшествующие фазы того же жизненного цикла.

### Вопрос 3. Сколько стандартных жизненных циклов есть в Maven и чем они отличаются друг от друга?

**Категория:** Maven/Gradle · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Introduction to the Build Lifecycle – Apache Maven](https://maven.apache.org/guides/introduction/introduction-to-the-lifecycle.html)

В Maven есть три встроенных жизненных цикла: **default**, **clean** и **site**.

- **default** — отвечает за развёртывание проекта (компиляция, тестирование, упаковка, публикация). Содержит 23 фазы. Определяется без заранее привязанных плагинов на уровне самого цикла — привязки задаются через тип упаковки (`<packaging>`).
- **clean** — отвечает за очистку проекта. Содержит 3 фазы: `pre-clean`, `clean`, `post-clean`. Плагин `maven-clean-plugin` привязан к фазе `clean` по умолчанию.
- **site** — отвечает за создание документации и сайта проекта. Содержит 4 фазы: `pre-site`, `site`, `post-site`, `site-deploy`. Плагин `maven-site-plugin` привязан к фазам `site` и `site-deploy` по умолчанию.

#### Ответ от Claude

В Maven существует ровно **три** стандартных жизненных цикла. Они независимы друг от друга — выполнение одного не запускает другой автоматически.

---

**1. default (build)**

Основной цикл — управляет полным процессом сборки и развёртывания.

- Количество фаз: **23**
- Назначение: компиляция → тестирование → упаковка → публикация
- Особенность: сам цикл не несёт предустановленных привязок к плагинам; конкретные привязки зависят от значения `<packaging>` в POM (`jar`, `war`, `pom`, `ear` и т.д.)
- Запуск: `mvn deploy` (или любая промежуточная фаза: `compile`, `test`, `package`, `install`)

---

**2. clean**

Цикл очистки — удаляет артефакты предыдущей сборки.

- Количество фаз: **3**
  - `pre-clean` — подготовительные действия перед очисткой
  - `clean` — удаляет директорию `target/` (всё, что было создано предыдущей сборкой)
  - `post-clean` — финализирующие действия после очистки
- Привязка по умолчанию: `maven-clean-plugin:clean` → фаза `clean`
- Запуск: `mvn clean`

---

**3. site**

Цикл документации — генерирует сайт с документацией проекта.

- Количество фаз: **4**
  - `pre-site` — подготовка к генерации
  - `site` — генерация HTML-документации (Javadoc, отчёты о тестах, зависимостях и т.д.)
  - `post-site` — финализация перед публикацией
  - `site-deploy` — публикация сгенерированного сайта на сервер
- Привязки по умолчанию: `maven-site-plugin:site` → `site`, `maven-site-plugin:deploy` → `site-deploy`
- Запуск: `mvn site`

---

**Сравнительная таблица:**

| Цикл | Фаз | Назначение | Привязки по умолчанию |
|------|-----|-----------|----------------------|
| `default` | 23 | Компиляция, тестирование, упаковка, публикация | Зависят от `<packaging>` |
| `clean` | 3 | Удаление артефактов предыдущей сборки | `maven-clean-plugin` |
| `site` | 4 | Генерация и публикация документации | `maven-site-plugin` |

**Практическое использование:** циклы можно комбинировать в одной команде: `mvn clean package` — сначала полностью выполнится цикл `clean`, затем цикл `default` до фазы `package`. Это гарантирует, что сборка происходит с нуля, без артефактов предыдущих запусков.

### Вопрос 4. Что такое транзитивные зависимости? Приведите пример: проект A зависит от B, B зависит от C — как Maven включит C?

**Категория:** Maven/Gradle · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Introduction to the Dependency Mechanism — Apache Maven](https://maven.apache.org/guides/introduction/introduction-to-dependency-mechanism.html)

Maven поддерживает транзитивные зависимости. Это означает: достаточно объявить только те библиотеки, которые ваш проект использует напрямую — Maven автоматически подтянет все их собственные зависимости. Библиотеки, от которых зависят ваши зависимости (и так далее по цепочке), включаются без каких-либо дополнительных объявлений.

Когда несколько путей в дереве зависимостей ведут к одной библиотеке, Maven применяет правило «ближайшего определения» (nearest definition): выбирается та версия, которая находится ближе к корню дерева (т.е. к вашему проекту). Если две версии находятся на одинаковой глубине, побеждает та, что объявлена первой в pom.xml.

Пример:

```
A
├── B
│   └── C 2.0  (глубина 2)
└── D
    └── C 1.0  (глубина 2)
```

Обе версии C на одинаковой глубине — победит та, что объявлена первой. Чтобы принудительно использовать нужную версию, достаточно явно объявить C в pom.xml вашего проекта.

#### Ответ от Claude

**Транзитивные зависимости** — это зависимости ваших зависимостей. Maven подключает их автоматически без необходимости явно перечислять в pom.xml.

**Пример A → B → C:**

Если проект A объявляет зависимость на B, а B в своём pom.xml объявляет зависимость на C, то Maven:
1. Скачивает B (прямая зависимость A).
2. Читает pom.xml библиотеки B и обнаруживает зависимость на C.
3. Автоматически добавляет C в classpath проекта A — как транзитивную зависимость.

**Разрешение конфликтов версий:**

Если разные пути дерева зависимостей ведут к разным версиям одного артефакта, Maven применяет два правила:

- **Nearest definition (ближайшее определение):** побеждает версия, ближайшая к корню дерева.

  Пример: A→B→C→D:2.0 и A→E→D:1.0 — Maven выберет D:1.0, потому что путь до неё короче (2 уровня против 3).

- **First declaration wins (первое объявление побеждает):** если обе версии на одинаковой глубине, выбирается та, что идёт первой в pom.xml.

**Как принудительно задать версию:**

Явно объявить зависимость на D в pom.xml проекта A — тогда она будет на глубине 1 и гарантированно победит.

**Практические советы:**

- Используйте `mvn dependency:tree` для просмотра полного дерева зависимостей.
- Используйте `mvn dependency:analyze` для выявления неиспользуемых и недекларированных зависимостей.
- Даже если Maven подтягивает библиотеку транзитивно, рекомендуется явно объявлять в pom.xml те зависимости, классы которых вы напрямую используете в коде — это защищает от поломки сборки при изменении чужого pom.xml.

**Управление транзитивными зависимостями:**

- `<optional>true</optional>` — помечает зависимость как необязательную: потребители вашей библиотеки не получат её транзитивно.
- `<exclusions>` — позволяет явно исключить нежелательную транзитивную зависимость.

### Вопрос 5. Перечислите scope зависимостей в Maven. Чем отличается compile от provided и когда используется каждый?

**Категория:** Maven/Gradle · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Introduction to the Dependency Mechanism — Apache Maven](https://maven.apache.org/guides/introduction/introduction-to-dependency-mechanism.html)

Maven определяет 6 scope зависимостей:

- **compile** (по умолчанию) — зависимость доступна во всех classpath проекта и транзитивно передаётся зависимым проектам.
- **provided** — аналогично compile по доступности в compile- и test-classpath, но предполагается, что зависимость будет предоставлена JDK или контейнером во время выполнения. Типичный пример — Servlet API при разработке веб-приложения: контейнер (Tomcat, JBoss) сам предоставит эти классы. Зависимость не транзитивна и не попадает в итоговый пакет.
- **runtime** — зависимость не нужна при компиляции, но нужна при выполнении. Присутствует в runtime- и test-classpath.
- **test** — доступна только при компиляции и выполнении тестов. Не транзитивна.
- **system** — аналогична provided, но требует явно указать путь к JAR через `<systemPath>`. Не рекомендуется к использованию.
- **import** — применяется только в `<dependencyManagement>` для импорта managed-зависимостей из BOM (Bill of Materials) POM.

#### Ответ от Claude

Maven поддерживает 6 scope зависимостей. Scope определяет, в каких classpath доступна зависимость, попадает ли она в финальный артефакт и является ли транзитивной.

**Таблица scope:**

| Scope | Compile CP | Test CP | Runtime CP | В JAR/WAR | Транзитивна |
|-------|-----------|---------|------------|-----------|-------------|
| compile | да | да | да | да | да |
| provided | да | да | нет | нет | нет |
| runtime | нет | да | да | да | runtime |
| test | нет | да | нет | нет | нет |
| system | да | да | нет | нет | нет |
| import | — | — | — | — | — |

**compile (значение по умолчанию):**

Используется, когда зависимость нужна и при компиляции, и при выполнении приложения. Попадает во все classpath и в финальный JAR/WAR. Транзитивно передаётся потребителям вашей библиотеки.

```xml
<dependency>
  <groupId>org.apache.commons</groupId>
  <artifactId>commons-lang3</artifactId>
  <version>3.12.0</version>
  <!-- scope compile — можно не указывать явно -->
</dependency>
```

Когда использовать: для всех библиотек, классы которых вы напрямую вызываете в production-коде.

**provided:**

Используется, когда зависимость нужна для компиляции, но во время выполнения её предоставляет внешняя среда (сервер приложений, JDK, контейнер). В финальный WAR/JAR не включается, чтобы избежать конфликта с версией, уже поставляемой контейнером.

```xml
<dependency>
  <groupId>javax.servlet</groupId>
  <artifactId>javax.servlet-api</artifactId>
  <version>4.0.1</version>
  <scope>provided</scope>
</dependency>
```

Когда использовать:
- Servlet API, JSP API при разработке под Tomcat/Jetty/WildFly.
- Java EE / Jakarta EE API при деплое на application server.
- Lombok (процессор аннотаций, не нужен в runtime).

**Ключевое отличие compile от provided:**

- `compile` — зависимость входит в финальный артефакт и доступна в runtime.
- `provided` — зависимость используется только при компиляции; в runtime её поставляет контейнер, поэтому она не включается в артефакт.

**Остальные scope (кратко):**

- **runtime** — нужен только в runtime, не при компиляции. Пример: JDBC-драйвер (код компилируется против интерфейсов `java.sql.*`, а конкретный драйвер нужен только при выполнении).
- **test** — только для тестов (JUnit, Mockito, Testcontainers). Подробно — в следующем вопросе.
- **system** — устаревший, требует `<systemPath>`. Не рекомендуется: используйте корпоративный Maven-репозиторий.
- **import** — только в `<dependencyManagement>`, тип `pom`. Используется для импорта BOM (например, Spring Boot BOM).

### Вопрос 6. Для чего нужен scope test? Попадает ли зависимость с таким scope в финальный JAR/WAR?

**Категория:** Maven/Gradle · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Introduction to the Dependency Mechanism — Apache Maven](https://maven.apache.org/guides/introduction/introduction-to-dependency-mechanism.html)

Scope `test` указывает, что зависимость не требуется при стандартном выполнении приложения и нужна только для тестирования. Зависимости с таким scope присутствуют только в test-classpath (компиляция тестов и их выполнение) и не являются транзитивными. Стандартный пример использования — добавление тестовой библиотеки JUnit.

Зависимости со scope `provided` и `test` никогда не включаются в финальный пакет (JAR/WAR).

#### Ответ от Claude

**Назначение scope test:**

`test` — специальный scope для зависимостей, которые нужны исключительно при написании и запуске тестов, но не требуются в production-коде. Это позволяет:

1. Уменьшить размер финального артефакта — тестовые библиотеки не попадают в JAR/WAR.
2. Избежать засорения production-classpath тестовыми инструментами.
3. Предотвратить случайное использование тестовых утилит в основном коде.

**Classpath:**

Зависимость со scope `test` доступна только в двух classpath:
- **test-compile classpath** — при компиляции тестовых классов (src/test/java).
- **test-runtime classpath** — при выполнении тестов (mvn test, mvn verify).

Она отсутствует в compile-classpath и runtime-classpath основного кода. Это означает: если вы случайно попытаетесь использовать класс из test-зависимости в src/main/java — компиляция завершится ошибкой.

**Попадает ли в финальный JAR/WAR?**

Нет. Зависимости со scope `test` **никогда не включаются** в финальный JAR/WAR/EAR. Maven-плагины упаковки (maven-jar-plugin, maven-war-plugin) явно исключают их из итогового артефакта.

**Транзитивность:**

Scope `test` не является транзитивным. Если ваша библиотека A зависит от JUnit со scope `test`, то проекты, которые подключают A как зависимость, не получат JUnit транзитивно.

**Типичные зависимости со scope test:**

```xml
<!-- JUnit 5 -->
<dependency>
  <groupId>org.junit.jupiter</groupId>
  <artifactId>junit-jupiter</artifactId>
  <version>5.10.0</version>
  <scope>test</scope>
</dependency>

<!-- Mockito -->
<dependency>
  <groupId>org.mockito</groupId>
  <artifactId>mockito-core</artifactId>
  <version>5.4.0</version>
  <scope>test</scope>
</dependency>

<!-- Spring Boot Test -->
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-test</artifactId>
  <scope>test</scope>
</dependency>

<!-- Testcontainers -->
<dependency>
  <groupId>org.testcontainers</groupId>
  <artifactId>junit-jupiter</artifactId>
  <version>1.19.0</version>
  <scope>test</scope>
</dependency>
```

**Особый случай — test-jar:**

Если нужно передать тестовые вспомогательные классы другому модулю (например, общие фикстуры в многомодульном проекте), можно создать отдельный артефакт с classifier `tests` через maven-jar-plugin с целью `test-jar`. Такой артефакт подключается как:

```xml
<dependency>
  <groupId>com.example</groupId>
  <artifactId>module-a</artifactId>
  <version>1.0</version>
  <type>test-jar</type>
  <scope>test</scope>
</dependency>
```

Это единственный штатный способ переиспользовать тестовый код между модулями в Maven.

### Вопрос 7. Чем scope runtime отличается от compile? В каком сценарии нужно использовать runtime?

**Категория:** Maven/Gradle · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Introduction to the Dependency Mechanism — Apache Maven](https://maven.apache.org/guides/introduction/introduction-to-dependency-mechanism.html)

Официальная документация Maven описывает оба scope следующим образом:

- **compile** — область видимости по умолчанию, используемая если scope не указан явно. Зависимость доступна во всех classpath проекта. Более того, такие зависимости транзитивно распространяются на зависящие проекты.
- **runtime** — указывает, что зависимость не требуется при компиляции, но необходима при выполнении. Maven включает такую зависимость в runtime-classpath и test-classpath, но не в compile-classpath.

Типичный пример использования runtime — JDBC-драйвер: код компилируется против стандартного интерфейса `java.sql.*`, а конкретная реализация драйвера подключается только при запуске приложения.

#### Ответ от Claude

**Ключевое отличие по classpath:**

| Classpath         | compile | runtime |
|-------------------|:-------:|:-------:|
| compile-classpath | да      | нет     |
| test-classpath    | да      | да      |
| runtime-classpath | да      | да      |
| Транзитивность    | да      | нет     |

**compile** (значение по умолчанию) — зависимость нужна уже на этапе компиляции: классы из неё используются непосредственно в исходном коде. Такая зависимость присутствует везде и транзитивно наследуется дочерними проектами.

**runtime** — зависимость не участвует в компиляции (компилятор её не видит), но обязана быть в classpath во время выполнения. Транзитивно не передаётся.

**Когда использовать runtime:**

1. **JDBC-драйверы** — код пишется против интерфейса `java.sql.Connection`/`DriverManager`, а конкретный драйвер (например, `mysql-connector-java`) нужен только в рантайме:
   ```xml
   <dependency>
     <groupId>com.mysql</groupId>
     <artifactId>mysql-connector-j</artifactId>
     <version>9.3.0</version>
     <scope>runtime</scope>
   </dependency>
   ```
2. **Реализации SLF4J** — код использует только API (`slf4j-api`), а реализация (`logback-classic`) нужна только при выполнении.
3. **Любая реализация интерфейса/SPI** — когда компиляция происходит против абстракции, а конкретный провайдер подгружается через ServiceLoader или DI-контейнер.

**Практическая польза:** использование `runtime` вместо `compile` делает компиляцию защищённой от случайного прямого использования реализации (компилятор выдаст ошибку, если вы попытаетесь импортировать класс, которого нет в compile-classpath), и уменьшает количество транзитивных зависимостей у потребителей библиотеки.

### Вопрос 8. В чём разница между фазой (phase), целью (goal) и плагином (plugin) в Maven?

**Категория:** Maven/Gradle · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Introduction to the Build Lifecycle — Apache Maven](https://maven.apache.org/guides/introduction/introduction-to-the-lifecycle.html)

Официальная документация Maven даёт следующие определения:

- **Lifecycle (жизненный цикл)** — упорядоченная последовательность именованных фаз. Maven имеет три встроенных жизненных цикла: `default` (сборка и деплой), `clean` (очистка) и `site` (генерация документации).
- **Phase (фаза)** — конкретный шаг внутри жизненного цикла, отвечающий за определённый этап сборки. Фазы выполняются последовательно: запуск фазы `package` автоматически запускает все предшествующие фазы (`validate`, `compile`, `test` и т.д.).
- **Goal (цель)** — конкретная задача (более детальная, чем фаза), которая вносит вклад в сборку проекта. Цель может быть привязана к нулю или более фазам, а при необходимости — вызвана напрямую вне жизненного цикла.
- **Plugin (плагин)** — артефакт, предоставляющий набор целей. Плагин может иметь одну или несколько целей, каждая из которых представляет отдельную возможность плагина.

#### Ответ от Claude

**Иерархия понятий:**

```
Lifecycle (жизненный цикл)
  └── Phase (фаза)  ← несколько фаз в строгом порядке
        └── Goal (цель) ← привязывается к фазе плагином
              └── Plugin (плагин) ← поставляет цели
```

**Plugin (плагин)** — JAR-артефакт, содержащий исполняемый код. Примеры: `maven-compiler-plugin`, `maven-surefire-plugin`, `maven-jar-plugin`. Каждый плагин предоставляет одну или несколько *целей*.

**Goal (цель)** — единица работы, реализованная плагином. Записывается в формате `плагин:цель`, например `compiler:compile`, `surefire:test`, `jar:jar`. Цель можно вызвать напрямую: `mvn compiler:compile`.

**Phase (фаза)** — именованный шаг в жизненном цикле. Фазы жизненного цикла `default`: `validate` → `compile` → `test` → `package` → `verify` → `install` → `deploy`. Когда вы запускаете `mvn package`, Maven последовательно выполняет все фазы вплоть до `package` включительно.

**Привязка** (binding): цели плагинов привязываются к фазам. При выполнении фазы Maven запускает все цели, привязанные к ней. Стандартные привязки определяются типом упаковки проекта (jar, war, pom и т.д.). Дополнительные привязки задаются в `pom.xml`:

```xml
<plugin>
  <groupId>org.apache.maven.plugins</groupId>
  <artifactId>maven-antrun-plugin</artifactId>
  <executions>
    <execution>
      <phase>verify</phase>        <!-- фаза -->
      <goals>
        <goal>run</goal>           <!-- цель плагина -->
      </goals>
    </execution>
  </executions>
</plugin>
```

**Важный нюанс:** если к фазе не привязана ни одна цель, эта фаза пропускается без ошибки. Фаза сама по себе — лишь «слот» в расписании; реальную работу выполняют цели плагинов.

### Вопрос 9. Что такое локальный репозиторий Maven? Где он находится по умолчанию и как изменить его расположение?

**Категория:** Maven/Gradle · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Settings Reference — Apache Maven](https://maven.apache.org/settings.html)

Из официальной документации Maven по файлу `settings.xml`:

Элемент `localRepository` задаёт путь к локальному репозиторию на данной машине. Значение по умолчанию — `${user.home}/.m2/repository`. Этот элемент особенно полезен на главном сервере сборки, где все вошедшие пользователи могут использовать общий локальный репозиторий.

Расположение файла `settings.xml`:
- **Глобальные настройки** (применяются ко всем пользователям): `${maven.home}/conf/settings.xml`
- **Пользовательские настройки** (приоритетнее глобальных): `${user.home}/.m2/settings.xml`

Пример изменения пути в `settings.xml`:
```xml
<settings xmlns="http://maven.apache.org/SETTINGS/1.0.0">
  <localRepository>/path/to/local/repo/</localRepository>
</settings>
```

#### Ответ от Claude

**Что такое локальный репозиторий:**

Локальный репозиторий Maven — это директория на компьютере разработчика (или CI-сервере), в которой Maven кэширует все скачанные зависимости, плагины и артефакты собственных проектов (`mvn install`). Он исполняет роль локального кэша: при следующем запросе той же зависимости Maven берёт артефакт из него, не обращаясь в сеть.

**Расположение по умолчанию:**

| ОС           | Путь                                      |
|--------------|-------------------------------------------|
| Linux/macOS  | `~/.m2/repository`                        |
| Windows      | `C:\Users\<имя_пользователя>\.m2\repository` |

Переменная Maven: `${user.home}/.m2/repository`

**Как изменить расположение:**

*Способ 1 — через `settings.xml` (рекомендуется):*

Отредактируйте файл `~/.m2/settings.xml` (пользовательские настройки) или `${maven.home}/conf/settings.xml` (глобальные настройки):
```xml
<settings xmlns="http://maven.apache.org/SETTINGS/1.0.0">
  <localRepository>/var/maven/repository</localRepository>
</settings>
```
Путь должен быть абсолютным. Если оба файла существуют, их содержимое объединяется, при этом пользовательский `settings.xml` имеет более высокий приоритет.

*Способ 2 — через аргумент командной строки (разово):*
```bash
mvn -Dmaven.repo.local=/tmp/my-repo clean install
```

**Типичные сценарии изменения расположения:**
- Вынесение репозитория на диск с большим объёмом свободного места.
- Общий кэш на CI-сервере для всех пользователей (ускорение сборки).
- Изоляция зависимостей разных проектов (через `-Dmaven.repo.local` при каждом запуске).
- Работа в корпоративной среде с нестандартным домашним каталогом.
### Вопрос 10. Чем Gradle принципиально отличается от Maven по формату описания сборки и механизму выполнения задач?

**Категория:** Maven/Gradle · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Gradle vs Maven: Understanding the Real Difference (Especially the Lifecycle) — Medium](https://medium.com/@abhi.m.abhijit/gradle-vs-maven-understanding-the-real-difference-especially-the-lifecycle-4d0338fa7788)

Maven использует XML-файлы конфигурации, тогда как Gradle применяет современный DSL на Groovy или Kotlin. Синтаксис Gradle более лаконичен и программатичен по сравнению с декларативным подходом Maven.

Ключевое архитектурное различие — в механизме выполнения сборки:

**Maven** следует «фазовой модели сборки» с фиксированным жизненным циклом. При запуске `mvn install` Maven выполняет фазы в предопределённом порядке: `validate → compile → test → package → verify → install → deploy`. Каждая фаза автоматически запускает все предыдущие.

**Gradle** использует «задачно-ориентированную модель», строя Направленный Ациклический Граф (DAG). Вместо фиксированных фаз плагины Gradle регистрируют задачи, которые объявляют зависимости друг от друга, и Gradle выполняет только необходимые задачи на основе этого графа.

Gradle также поддерживает инкрементальные сборки (пересборка только изменившегося), кэширование сборки и параллельное выполнение задач. Maven выполняет фазы последовательно без инкрементальной оптимизации.

#### Ответ от Claude

**Формат описания сборки**

Maven использует декларативный XML (`pom.xml`): разработчик описывает *что* является проектом (структура, зависимости, метаданные), а не *как* его строить. Поведение расширяется через плагины, привязанные к фазам жизненного цикла.

Gradle использует императивный DSL — изначально на Groovy (`build.gradle`), с переходом к Kotlin DSL (`build.gradle.kts`) как рекомендованному стандарту. Kotlin DSL обеспечивает статическую типизацию и полноценную IDE-поддержку (автодополнение, навигация по типам). Скрипт Gradle — это полноценная программа, в которой можно использовать условия, циклы и любую логику.

**Механизм выполнения задач**

Maven работает по принципу фиксированного жизненного цикла. Существуют три встроенных жизненных цикла (`default`, `clean`, `site`), каждый состоит из последовательных фаз. При вызове `mvn package` Maven обязательно выполнит все предшествующие фазы (`validate`, `initialize`, `compile`, `test` и т.д.) — пропустить фазу нельзя.

Gradle строит Directed Acyclic Graph (DAG) задач. Задачи декларируют свои входные и выходные данные, а также зависимости от других задач. Gradle вычисляет минимальный набор задач для достижения цели и выполняет только их — в том числе параллельно, если задачи независимы. Если входные и выходные данные задачи не изменились, задача помечается как `UP-TO-DATE` и пропускается (инкрементальная сборка). Дополнительно поддерживается Build Cache — повторное использование результатов задач между разными машинами.

**Итоговое сравнение**

| Аспект | Maven | Gradle |
|---|---|---|
| Формат | XML (`pom.xml`) | Groovy/Kotlin DSL |
| Подход | Декларативный | Декларативный + императивный |
| Модель выполнения | Фиксированные фазы жизненного цикла | DAG задач |
| Инкрементальные сборки | Нет | Да (UP-TO-DATE, Build Cache) |
| Параллелизм | Ограниченный | Встроенный |
| Порог вхождения | Ниже | Выше |

### Вопрос 11. Как Maven разрешает конфликт версий транзитивных зависимостей по принципу «nearest wins» (ближайшая выигрывает)? Что произойдёт, если две зависимости находятся на одинаковой глубине?

**Категория:** Maven/Gradle · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Introduction to the Dependency Mechanism — Apache Maven](https://maven.apache.org/guides/introduction/introduction-to-dependency-mechanism.html)

Maven разрешает конфликты версий по принципу «ближайшее определение побеждает» (nearest definition wins): выбирается версия зависимости, ближайшая к корню дерева зависимостей.

Пример:
```
A
├── B
│   └── C
│       └── D 2.0
└── E
    └── D 1.0
```
Будет выбрана **D 1.0**, поскольку путь A → E → D короче (глубина 2), чем A → B → C → D (глубина 3).

Чтобы принудительно использовать D 2.0, нужно явно объявить её в самом проекте A — тогда прямое объявление (глубина 1) выиграет у обоих транзитивных путей.

Если две версии находятся на **одинаковой глубине**, побеждает та, что объявлена **первой** в `pom.xml`.

Для централизованного управления версиями служит секция `<dependencyManagement>`: она имеет приоритет над dependency mediation и позволяет зафиксировать версию транзитивной зависимости без её явного объявления в `<dependencies>`.

#### Ответ от Claude

**Алгоритм «nearest wins»**

Maven обходит дерево зависимостей в ширину (BFS — breadth-first search). Когда один и тот же артефакт встречается в дереве несколько раз с разными версиями, Maven выбирает ту версию, которая находится на меньшей глубине от корня (т.е. ближе к самому проекту).

Формально: если артефакт `D` встречается на пути длиной `d1` и на пути длиной `d2`, где `d1 < d2`, выбирается версия из пути `d1`.

**Пример:**
```
Проект (глубина 0)
├── A (глубина 1)
│   └── D:2.0 (глубина 2)
└── B (глубина 1)
    └── C (глубина 2)
        └── D:1.0 (глубина 3)
```
Будет выбрана **D:2.0** (глубина 2 < 3).

**Случай одинаковой глубины**

Если два конкурирующих объявления одного артефакта находятся на одинаковой глубине в дереве, Maven выбирает ту, что встретилась **первой при BFS-обходе**, что на практике соответствует порядку объявления зависимостей в `pom.xml`. То есть зависимость, объявленная выше в файле, «выиграет».

Это поведение непредсказуемо при перестановке зависимостей и считается потенциально опасным.

**Риски стратегии**

Стратегия «nearest wins» может привести к выбору **устаревшей версии**: если ваш прямой dependency A тянет D:1.0, а глубоко вложенный B тянет D:2.0, Maven выберет D:1.0 — более старую версию, что может привести к ошибкам времени выполнения, если другой код ожидает API из D:2.0.

**Инструменты диагностики и контроля**

- `mvn dependency:tree -Dverbose` — показывает дерево зависимостей и конфликты (omitted for conflict).
- `<dependencyManagement>` — раздел в POM, который задаёт версию «сверху»; имеет наивысший приоритет и переопределяет dependency mediation для всего проекта и дочерних модулей.
- `<exclusions>` в конкретной зависимости — явное исключение транзитивного артефакта.
- Явное объявление зависимости в `<dependencies>` проекта с нужной версией (глубина 1 всегда побеждает транзитивные пути).

### Вопрос 12. Как Gradle разрешает конфликты версий по умолчанию? Чем его стратегия отличается от Maven, и каковы риски каждого подхода?

**Категория:** Maven/Gradle · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Dependency Constraints and Conflict Resolution — Gradle User Manual](https://docs.gradle.org/current/userguide/dependency_constraints_conflicts.html)

По умолчанию Gradle выбирает **наибольшую версию** из всех найденных в графе зависимостей. Когда несколько зависимостей запрашивают разные версии одного модуля, Gradle автоматически выбирает наибольший номер версии — исходя из предположения, что новые версии обратно совместимы.

Gradle применяет правило наибольшей версии даже если это переопределяет одно из ваших прямых объявлений зависимостей.

Данная стратегия несёт риски:
- **Бинарная несовместимость**: наибольшая версия может содержать критические изменения API, несовместимые с зависимым кодом.
- **Непроверенные комбинации**: проект может оказаться собранным против версии, которую разработчик не тестировал.
- **Скрытые сбои**: несовместимости проявляются только в рантайме.

Для точного контроля Gradle предоставляет механизм constraints:
```groovy
dependencies {
    constraints {
        implementation("org.apache.commons:commons-lang3:3.12.0")
    }
}
```

#### Ответ от Claude

**Стратегия Gradle по умолчанию: «highest version wins»**

Gradle обходит весь граф зависимостей и собирает все запрошенные версии одного артефакта. Из них выбирается **максимальная версия** (по семантическому сравнению номеров). Это глобальное решение — Gradle не останавливается на «ближайшей» версии, а учитывает все упоминания артефакта в графе независимо от глубины.

**Сравнение со стратегией Maven**

| Критерий | Maven (nearest wins) | Gradle (highest wins) |
|---|---|---|
| Принцип выбора | Минимальная глубина в дереве | Максимальный номер версии |
| Алгоритм обхода | BFS, останавливается на первом найденном | Полный обход графа |
| Может выбрать более старую версию | Да, если прямая зависимость старее | Нет — всегда берёт новейшую |
| Может нарушить прямую зависимость | Нет | Да — транзитивная может повысить версию |

**Риски каждой стратегии**

*Риски Maven (nearest wins):*
- Выбирается **устаревшая версия**: прямая зависимость на старую версию перекрывает более новую транзитивную; могут возникнуть `NoSuchMethodError` если другой код ожидает новый API.
- Поведение зависит от **порядка объявления** в `pom.xml` при одинаковой глубине — хрупкость конфигурации.
- Разные разработчики могут получить разные версии, изменив порядок зависимостей.

*Риски Gradle (highest wins):*
- Выбирается **версия с breaking changes**: новая версия транзитивной зависимости может сломать API, на который рассчитывает другой код.
- Версия, выбранная Gradle, может быть **непроверенной** — разработчик не тестировал свой код против этой версии.
- Проблемы **проявляются в рантайме**, а не при компиляции, если изменения API затрагивают не сигнатуры, а поведение.
- Несовместимость между экосистемами: библиотека, собранная Gradle с D:10.1, при публикации в Maven Central создаёт POM, по которому Maven-потребитель может взять D:10.0 (nearest), что приводит к расхождению.

**Инструменты контроля в Gradle**

- `constraints` — фиксация версии без прямого объявления зависимости.
- `resolutionStrategy.force(...)` — принудительная фиксация конкретной версии.
- `resolutionStrategy.failOnVersionConflict()` — перевод конфликтов в ошибку сборки для явного разрешения.
- `configurations.all { resolutionStrategy { ... } }` — глобальная стратегия разрешения.
- `./gradlew dependencies` — просмотр итогового дерева зависимостей с указанием, какие версии были выбраны и почему.
### Вопрос 13. Как посмотреть дерево зависимостей в Maven и в Gradle? Какие команды для этого используются и как читать вывод при наличии конфликтов?

**Категория:** Maven/Gradle · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Maven Dependency Tree - Resolving Conflicts | DigitalOcean](https://www.digitalocean.com/community/tutorials/maven-dependency-tree-resolving-conflicts)

Команда `mvn dependency:tree` отображает все зависимости проекта в виде иерархического дерева. Запускается из корня проекта. Символы в выводе: `+-` — зависимость с вложенными элементами, `|` — продолжение ветки, `\-` — последний элемент ветки.

Версия, находящаяся ближе к корню дерева (прямая зависимость), имеет приоритет над транзитивной. Флаг `-Dverbose` показывает все конфликтующие зависимости, которые были исключены из дерева. Для разрешения конфликтов используются: явное объявление нужной версии как прямой зависимости, блок `<exclusions>` или фильтрация с `-Dincludes`/`-Dexcludes`.

|    \--- ...
\--- org.apache.httpcomponents:httpclient:4.5.13
     \--- commons-codec:commons-codec:1.7

Запись `1.6 -> 1.7` означает: была запрошена версия `1.6`, но в результате разрешения конфликтов выбрана `1.7`. Gradle по умолчанию выбирает наибольшую версию (стратегия newest-wins, в отличие от nearest-wins у Maven).

- `(*)` — повторное вхождение транзитивного поддерева (уже показано выше, не раскрывается)

**Разрешение конфликтов:**

В Maven:
```xml
<!-- 1. Явная прямая зависимость — выиграет как nearest -->
<dependency>
    <groupId>commons-collections</groupId>
    <artifactId>commons-collections</artifactId>
    <version>2.1</version>
</dependency>

<!-- 2. Управление через dependencyManagement -->
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>commons-collections</groupId>
            <artifactId>commons-collections</artifactId>
            <version>2.1</version>
        </dependency>
    </dependencies>
</dependencyManagement>

<!-- 3. Исключение транзитивной зависимости -->
<dependency>
    <groupId>some.group</groupId>
    <artifactId>some-artifact</artifactId>
    <version>1.0</version>
    <exclusions>
        <exclusion>
            <groupId>commons-collections</groupId>
            <artifactId>commons-collections</artifactId>
        </exclusion>
    </exclusions>
</dependency>
```

В Gradle:
```kotlin
// 1. Принудительная версия через constraints
dependencies {
    constraints {
        implementation("commons-codec:commons-codec:1.7")
    }
}

// 2. Принудительная замена через resolutionStrategy
configurations.all {
    resolutionStrategy.force("commons-codec:commons-codec:1.7")
}

// 3. Исключение транзитивной зависимости
dependencies {
    implementation("org.example:lib:1.0") {
        exclude(group = "commons-codec", module = "commons-codec")
    }
}
```

### Вопрос 14. Что такое инкрементальная сборка в Gradle и как она работает? За счёт чего Gradle быстрее Maven на больших проектах?

**Категория:** Maven/Gradle · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Incremental Build · Gradle User Manual](https://docs.gradle.org/current/userguide/incremental_build.html)

Инкрементальная сборка в Gradle — оптимизация, которая пропускает задачи, если их входные и выходные данные не изменились с момента последнего выполнения. При пометке `UP-TO-DATE` в консоли инкрементальная сборка работает.

Gradle реализует систему снятия отпечатков (fingerprinting): перед первым запуском создаёт снимок с путями и хэшами содержимого всех входных файлов; после успешного завершения снимает отпечатки выходных файлов. При каждом последующем запуске сравнивает новые отпечатки с сохранёнными. Если отпечатки совпадают — задача пропускается; если изменились — выполняется.

Важно: Gradle сравнивает хэши содержимого файлов, а не только временные метки. Это означает, что если генерирующая задача перезаписала файлы с тем же содержимым, зависимая задача не будет запущена повторно.

Для работы инкрементальной сборки задачи должны явно объявить свои входы и выходы через аннотации (`@Input`, `@InputFiles`, `@InputDirectory`, `@OutputDirectory`, `@OutputFile`) или через runtime API (`task.inputs`, `task.outputs`).

#### Ответ от Claude

**Что такое инкрементальная сборка в Gradle**

Инкрементальная сборка (incremental build) — механизм, при котором Gradle пропускает выполнение задачи, если её входные данные (inputs) и выходные данные (outputs) не изменились с момента последнего успешного выполнения. Задача помечается `UP-TO-DATE`.

**Как работает механизм (fingerprinting):**

1. Перед первым выполнением задачи Gradle вычисляет снимок (fingerprint): хэши содержимого каждого входного файла, значения входных параметров.
2. После успешного выполнения задачи Gradle сохраняет финпринты выходных файлов.
3. При каждом последующем запуске: сравниваются текущие финпринты входных данных с сохранёнными. Если совпадают — задача пропускается (`UP-TO-DATE`). Если изменились — задача выполняется, финпринты обновляются.

Ключевой момент: Gradle сравнивает именно хэши содержимого, а не временные метки файлов. Это исключает ложные срабатывания при регенерации файлов с одинаковым содержимым (например, при git checkout без реальных изменений).

**Аннотации для объявления входов/выходов:**

```kotlin
abstract class ProcessFilesTask : DefaultTask() {
    @get:InputDirectory
    abstract val sourceDir: DirectoryProperty

    @get:Input
    val version: String = "1.0"

    @get:Classpath          // семантически умный: игнорирует нерелевантные изменения
    abstract val compileClasspath: ConfigurableFileCollection

    @get:OutputDirectory
    abstract val outputDir: DirectoryProperty

    @TaskAction
    fun process() { /* логика задачи */ }
}
```

**Три уровня оптимизации Gradle (работают вместе):**

| Механизм | Что делает | Маркер в выводе |
|---|---|---|
| Incremental build | Пропускает задачу если inputs/outputs не изменились | `UP-TO-DATE` |
| Build Cache | Восстанавливает outputs из кэша (локального или удалённого) даже после `clean` или на другой машине | `FROM-CACHE` |
| Configuration Cache | Кэширует фазу конфигурации (построение графа задач), пропускает её при повторных запусках | отображается при старте |

**За счёт чего Gradle быстрее Maven на больших проектах:**

1. **Gradle Daemon** — долгоживущий фоновый процесс, удерживающий JVM и скомпилированные скрипты сборки в памяти. Устраняет overhead на старт JVM при каждом запуске.

2. **Инкрементальная компиляция Java** — анализирует зависимости между классами (ABI — Application Binary Interface). Если публичный API класса не изменился (только реализация), зависимые классы не перекомпилируются.

3. **Умный анализ classpath** — аннотация `@CompileClasspath` позволяет Gradle игнорировать изменения в jar-файлах, не затрагивающие публичный API. Maven перекомпилирует при любом изменении jar-файла.

4. **Параллельное выполнение** — задачи без зависимостей между собой выполняются параллельно (`--parallel`). Maven выполняет фазы жизненного цикла последовательно (без специальной настройки).

5. **Build Cache** — результаты задач кэшируются по хэшу входных данных. При переключении веток, clean-сборках или сборках на CI-сервере Gradle восстанавливает outputs из кэша вместо повторного выполнения.

6. **Configuration Cache** — кэширует результат фазы конфигурации (граф задач). При неизменных build-скриптах конфигурация пропускается полностью.

**Реальные цифры (данные gradle.org):**

- Чистая сборка: Gradle быстрее Maven в 2–10 раз
- Инкрементальные изменения: в 7–85 раз (разница растёт с числом модулей)
- С использованием build cache: в 3–30 раз
- Gradle минимум в 2 раза быстрее в каждом из сценариев

**Maven тоже имеет оптимизации**, но они менее развиты: `-T` для параллельной сборки модулей, `-pl`/`-am` для пересборки только изменившихся модулей, Takari Lifecycle для инкрементальной компиляции (требует отдельного плагина). Базовый Maven не имеет встроенного build cache уровня Gradle.

### Вопрос 15. Как настроить кастомный плагин в Maven или задачу (task) в Gradle? Чем plugin lifecycle binding в Maven отличается от явной зависимости между tasks в Gradle?

**Категория:** Maven/Gradle · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Guide to Developing Java Plugins - Apache Maven](https://maven.apache.org/guides/plugin/guide-java-plugin-development.html)

Плагин Maven состоит из одного или нескольких Mojo (Maven plain Old Java Object). Каждый Mojo представляет одну цель плагина и идентифицируется аннотацией `@Mojo`. Класс должен наследовать `AbstractMojo` и переопределить метод `execute()`.

Минимальный Mojo:
```java
@Mojo(name = "sayhi")
public class GreetingMojo extends AbstractMojo {
    @Override
    public void execute() throws MojoExecutionException {
        getLog().info("Hello, world.");
    }
}
```

POM плагина должен иметь `<packaging>maven-plugin</packaging>` и зависимости от `maven-plugin-api` и `maven-plugin-annotations`. Для привязки к фазе жизненного цикла используется либо аннотация `@Mojo(defaultPhase = LifecyclePhase.PACKAGE)`, либо элемент `<executions>/<phase>` в потребляющем POM.

Именование: плагин должен называться `<yourplugin>-maven-plugin`. Паттерн `maven-<yourplugin>-plugin` зарезервирован для официальных плагинов Apache Maven.

#### Ответ от Claude

**Кастомный плагин в Maven**

**Шаг 1. Создание проекта плагина:**

```xml
<!-- pom.xml плагина -->
<project>
    <groupId>com.example</groupId>
    <artifactId>my-tool-maven-plugin</artifactId>
    <version>1.0.0</version>
    <packaging>maven-plugin</packaging>  <!-- обязательно -->

    <dependencies>
        <dependency>
            <groupId>org.apache.maven</groupId>
            <artifactId>maven-plugin-api</artifactId>
            <version>3.9.9</version>
            <scope>provided</scope>
        </dependency>
        <dependency>
            <groupId>org.apache.maven.plugin-tools</groupId>
            <artifactId>maven-plugin-annotations</artifactId>
            <version>3.15.2</version>
            <scope>provided</scope>
        </dependency>
    </dependencies>
</project>
```

**Шаг 2. Реализация Mojo:**

```java
package com.example;

import org.apache.maven.plugin.AbstractMojo;
import org.apache.maven.plugin.MojoExecutionException;
import org.apache.maven.plugins.annotations.LifecyclePhase;
import org.apache.maven.plugins.annotations.Mojo;
import org.apache.maven.plugins.annotations.Parameter;

// defaultPhase задаёт фазу по умолчанию; её можно переопределить в pom потребителя
@Mojo(name = "greet", defaultPhase = LifecyclePhase.COMPILE)
public class GreetingMojo extends AbstractMojo {

    @Parameter(property = "greet.message", defaultValue = "Hello from plugin!")
    private String message;

    @Override
    public void execute() throws MojoExecutionException {
        getLog().info(message);
    }
}
```

**Шаг 3. Использование плагина в проекте-потребителе:**

```xml
<build>
    <plugins>
        <plugin>
            <groupId>com.example</groupId>
            <artifactId>my-tool-maven-plugin</artifactId>
            <version>1.0.0</version>
            <executions>
                <execution>
                    <id>greet-on-package</id>
                    <!-- переопределяет defaultPhase из @Mojo аннотации -->
                    <phase>package</phase>
                    <goals>
                        <goal>greet</goal>
                    </goals>
                    <configuration>
                        <message>Building artifact!</message>
                    </configuration>
                </execution>
            </executions>
        </plugin>
    </plugins>
</build>
```

После этого цель `greet` будет автоматически выполнена при запуске `mvn package` (или любой фазы, идущей после package).

**Кастомная задача в Gradle**

**Вариант 1: задача прямо в build-скрипте (простой случай):**

```kotlin
// build.gradle.kts
tasks.register("greet") {
    doLast {
        println("Hello from Gradle task!")
    }
}
```

**Вариант 2: типизированная задача (рекомендуется для сложной логики):**

```kotlin
// buildSrc/src/main/kotlin/GreetTask.kt  (или отдельный проект плагина)
import org.gradle.api.DefaultTask
import org.gradle.api.file.DirectoryProperty
import org.gradle.api.provider.Property
import org.gradle.api.tasks.*

abstract class GreetTask : DefaultTask() {

    @get:Input
    abstract val message: Property<String>

    @get:InputDirectory
    abstract val sourceDir: DirectoryProperty

    @get:OutputDirectory
    abstract val outputDir: DirectoryProperty

    @TaskAction
    fun greet() {
        println(message.get())
        // логика задачи
    }
}
```

Регистрация и настройка:
```kotlin
// build.gradle.kts
tasks.register<GreetTask>("greet") {
    message.set("Hello from typed task!")
    sourceDir.set(layout.projectDirectory.dir("src/main"))
    outputDir.set(layout.buildDirectory.dir("greet-output"))
}
```

**Установка зависимостей между задачами в Gradle:**

```kotlin
// dependsOn — жёсткая зависимость: taskB не выполнится без taskA
tasks.named("processResources") {
    dependsOn("greet")
}

// mustRunAfter — мягкий порядок: если обе задачи запускаются, A будет раньше B
// но A не запускается автоматически при запуске B
tasks.named("test") {
    mustRunAfter("greet")
}

// finalizedBy — гарантированное выполнение "после", даже при ошибке
tasks.named("integrationTest") {
    finalizedBy("stopServer")
}

// Привязка к жизненному циклу Gradle (аналог Maven phase binding):
tasks.named("build") {
    dependsOn("greet")  // greet выполнится как часть build
}
```

**Ключевые различия между Maven lifecycle binding и Gradle task dependencies:**

| Аспект | Maven | Gradle |
|---|---|---|
| Модель | Предопределённые фазы жизненного цикла (validate, compile, test, package, install, deploy). Плагин привязывается к фазе, и всё что идёт до неё — выполняется автоматически. | Явный граф зависимостей между задачами. Нет встроенных фаз; порядок определяется через `dependsOn`, `mustRunAfter`, `finalizedBy`. |
| Конвенции | Стандартизированы для всех проектов: `mvn package` всегда выполняет compile → test → package. | Конвенции встроены плагинами (java, application и др.) через те же механизмы `dependsOn`, но разработчик может полностью переопределить граф. |
| Гибкость | Ограниченная: нельзя запустить package без compile и test (только с `-DskipTests` и т.п.). | Высокая: можно запустить любую задачу изолированно; Gradle выполнит только её прямые зависимости. |
| Обнаружение порядка | Неявное: вытекает из привязки к фазе. Разработчик должен знать порядок фаз. | Явное: из объявлений `dependsOn`/`mustRunAfter` в скриптах. Граф можно визуализировать: `./gradlew tasks --all`. |
| Инкрементальность | Maven не отслеживает up-to-date состояние задач нативно; каждая фаза выполняется при каждом запуске. | Gradle пропускает задачи, чьи inputs/outputs не изменились (`UP-TO-DATE`). |
| Параллелизм | Параллельная сборка модулей через `-T`; фазы внутри модуля — последовательны. | Независимые задачи выполняются параллельно автоматически при `--parallel`. |

**Итог:** Maven использует декларативную привязку к фиксированным фазам — удобно для стандартных проектов, легко предсказуемо. Gradle даёт явный граф задач с полным контролем над порядком и условиями выполнения, что лучше масштабируется для сложных и нестандартных сценариев сборки.
### Вопрос 16. Какие scope зависимостей существуют в Maven и какой эффект каждый из них оказывает на compile classpath, runtime classpath и test classpath? Приведи конкретные примеры, когда используешь provided и когда runtime.

**Категория:** Maven/Gradle · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Introduction to the Dependency Mechanism — Apache Maven](https://maven.apache.org/guides/introduction/introduction-to-dependency-mechanism.html)

В Maven существует шесть scope зависимостей. **compile** — значение по умолчанию: зависимость доступна во всех трёх classpath (compile, runtime, test) и транзитивно передаётся зависимым проектам. **provided** — аналогичен compile, но предполагает, что зависимость будет предоставлена JDK или контейнером во время выполнения; включается только в compile и test classpath, не в runtime. Пример: Servlet API для веб-приложения — контейнер (Tomcat, Jetty) сам предоставляет эти классы. **runtime** — зависимость не нужна для компиляции, но нужна для выполнения; присутствует в runtime и test classpath, но не в compile classpath. Пример: JDBC-драйвер — код компилируется против интерфейса `java.sql.Driver`, а конкретная реализация нужна только в runtime. **test** — доступна только в test classpath (compile и execution тестов), не транзитивна. Примеры: JUnit, Mockito. **system** — аналогичен provided, но требует явного указания пути к jar через `<systemPath>`; не рекомендуется к использованию. **import** — специальный scope, применяется только к артефактам типа `pom` в секции `<dependencyManagement>` для импорта BOM.

#### Ответ от Claude

Maven определяет шесть scope зависимостей. Ниже приведена сводная таблица их присутствия в classpath:

| Scope     | Compile classpath | Runtime classpath | Test classpath | Транзитивность |
|-----------|:-----------------:|:-----------------:|:--------------:|:--------------:|
| compile   | да                | да                | да             | да             |
| provided  | да                | нет               | да             | нет            |
| runtime   | нет               | да                | да             | да             |
| test      | нет               | нет               | да             | нет            |
| system    | да                | нет               | да             | нет            |
| import    | n/a               | n/a               | n/a            | n/a            |

**compile** — используется по умолчанию, если `<scope>` не указан. Зависимость включается во все фазы и транзитивно попадает к потребителям проекта. Пример: `com.google.guava:guava`.

**provided** — применяется, когда среда исполнения (контейнер сервлетов, сервер приложений) самостоятельно предоставит библиотеку в runtime. Классический пример — Servlet API:

```xml
<dependency>
    <groupId>javax.servlet</groupId>
    <artifactId>javax.servlet-api</artifactId>
    <version>4.0.1</version>
    <scope>provided</scope>
</dependency>
```

Tomcat или Jetty кладут `servlet-api.jar` в свой classpath, поэтому включать его в WAR-архив не нужно и вредно (конфликт версий). Аналогично используется для `lombok` — он нужен только компилятору, а в runtime не требуется.

**runtime** — применяется, когда код компилируется против абстрактного API, а конкретная реализация нужна лишь в момент выполнения. Самый показательный пример — JDBC-драйвер:

```xml
<dependency>
    <groupId>org.postgresql</groupId>
    <artifactId>postgresql</artifactId>
    <version>42.7.3</version>
    <scope>runtime</scope>
</dependency>
```

Код компилируется против интерфейсов `java.sql.*` из JDK, а драйвер PostgreSQL нужен только при реальном подключении к базе. Аналогично для реализаций SLF4J (например, `logback-classic`) — код пишется против API `org.slf4j.Logger`, реализация подставляется в runtime.

**test** — зависимость видна только при компиляции и запуске тестов, не попадает в итоговый артефакт. Примеры: `junit`, `mockito-core`, `assertj-core`, `testcontainers`.

**system** — устаревший механизм для jar-файлов вне репозитория; требует `<systemPath>`. Не рекомендуется: вместо него следует использовать локальный или корпоративный Maven-репозиторий.

**import** — не scope в обычном смысле. Используется исключительно внутри `<dependencyManagement>` для зависимости с `<type>pom</type>`, чтобы импортировать управляемые версии из BOM-файла.

**Транзитивность scope** — когда зависимость A тянет зависимость B, итоговый scope B определяется комбинацией. Ключевые случаи: если A объявлена как `compile` и B — `runtime`, то для потребителя B станет `runtime`; если A объявлена как `provided` — транзитивные зависимости B отбрасываются полностью.

### Вопрос 17. В чём разница между секциями `<dependencies>` и `<dependencyManagement>` в Maven pom.xml? Что произойдёт, если объявить зависимость только в `<dependencyManagement>`, но не в `<dependencies>`?

**Категория:** Maven/Gradle · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Introduction to the Dependency Mechanism — Apache Maven](https://maven.apache.org/guides/introduction/introduction-to-dependency-mechanism.html)

Секция `<dependencyManagement>` предназначена для централизованного хранения метаданных зависимостей (версия, scope, exclusions) в родительском POM. Она не добавляет зависимость в проект сама по себе — это лишь «справочная карточка». Реальное добавление зависимости происходит только через секцию `<dependencies>`. Если дочерний модуль объявляет зависимость в своём `<dependencies>` без версии, Maven ищет версию в `<dependencyManagement>` ближайшего родителя. Если зависимость объявлена только в `<dependencyManagement>` и нигде не указана в `<dependencies>`, она не попадает ни в какой classpath и фактически игнорируется при сборке.

#### Ответ от Claude

**`<dependencies>`** — секция, определяющая реальные зависимости проекта. Каждая объявленная здесь зависимость немедленно разрешается и добавляется в соответствующий classpath согласно своему scope. Если версия не указана, Maven ищет её в `<dependencyManagement>`.

**`<dependencyManagement>`** — секция управления версиями. Она не добавляет зависимости в проект. Её назначение: централизованно задать версию, scope и exclusions так, чтобы дочерние модули и секции `<dependencies>` могли опускать эти атрибуты.

**Что будет, если зависимость объявлена только в `<dependencyManagement>`?**

Ничего. Зависимость не попадёт ни в один classpath, сборка пройдёт без неё. Это частая ошибка начинающих: перенести зависимость в `<dependencyManagement>` родительского POM и забыть добавить её в `<dependencies>` дочернего модуля — в результате получают `ClassNotFoundException` или `NoClassDefFoundError` в runtime, а `cannot find symbol` в compile-time, если scope был `compile`.

**Типичный паттерн в многомодульном проекте:**

Родительский POM (централизует версии):
```xml
<!-- parent/pom.xml -->
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-web</artifactId>
            <version>3.3.0</version>
        </dependency>
        <dependency>
            <groupId>org.junit.jupiter</groupId>
            <artifactId>junit-jupiter</artifactId>
            <version>5.10.2</version>
            <scope>test</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
```

Дочерний модуль (подключает нужные зависимости без версий):
```xml
<!-- module-a/pom.xml -->
<dependencies>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
        <!-- версия и scope унаследованы из родительского dependencyManagement -->
    </dependency>
    <dependency>
        <groupId>org.junit.jupiter</groupId>
        <artifactId>junit-jupiter</artifactId>
        <!-- scope=test унаследован -->
    </dependency>
</dependencies>
```

**Дополнительная роль `<dependencyManagement>`** — управление версиями транзитивных зависимостей. Если в дереве зависимостей встречается артефакт, чья версия указана в `<dependencyManagement>`, Maven принудительно использует именно эту версию, игнорируя «победителя» по правилу nearest-definition. Это позволяет явно закрепить версию уязвимой транзитивной библиотеки без добавления её в `<dependencies>`.

**Ключевое различие:** `<dependencyManagement>` — конфигурация, `<dependencies>` — активация. Без записи в `<dependencies>` зависимость не будет скачана и не попадёт ни в один classpath.

### Вопрос 18. Что такое BOM (Bill of Materials) в Maven? Как его подключить и зачем это нужно в многомодульном проекте? Как аналогичная задача решается в Gradle?

**Категория:** Maven/Gradle · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Introduction to the Dependency Mechanism — Apache Maven](https://maven.apache.org/guides/introduction/introduction-to-dependency-mechanism.html)

BOM (Bill of Materials) — это специальный POM-файл с `<packaging>pom</packaging>`, содержащий только секцию `<dependencyManagement>` с набором согласованных версий зависимостей. Чтобы подключить BOM, нужно добавить его в `<dependencyManagement>` со значениями `<type>pom</type>` и `<scope>import</scope>`. После импорта все версии из BOM становятся доступны в текущем проекте: их можно использовать в `<dependencies>` без явного указания версии. BOM решает проблему многомодульных проектов, где разные модули должны использовать согласованные версии одних и тех же библиотек (например, всех артефактов Jackson или Spring). Важное преимущество перед наследованием: один модуль может импортировать несколько BOM, тогда как `<parent>` может быть только один.

#### Ответ от Claude

**Что такое BOM?**

BOM (Bill of Materials, «список материалов») — это специальный Maven POM-артефакт с `<packaging>pom</packaging>`, который содержит только `<dependencyManagement>` с согласованным набором версий взаимосвязанных библиотек. BOM не содержит кода, не добавляет зависимостей — он лишь публикует «таблицу совместимых версий».

Примеры широко известных BOM: `spring-boot-dependencies`, `jackson-bom`, `junit-bom`, `quarkus-bom`.

**Структура BOM-файла:**

```xml
<!-- my-platform/pom.xml -->
<project>
    <groupId>com.example</groupId>
    <artifactId>my-platform</artifactId>
    <version>1.0.0</version>
    <packaging>pom</packaging>

    <dependencyManagement>
        <dependencies>
            <dependency>
                <groupId>com.fasterxml.jackson.core</groupId>
                <artifactId>jackson-core</artifactId>
                <version>2.17.1</version>
            </dependency>
            <dependency>
                <groupId>com.fasterxml.jackson.core</groupId>
                <artifactId>jackson-databind</artifactId>
                <version>2.17.1</version>
            </dependency>
            <dependency>
                <groupId>com.fasterxml.jackson.datatype</groupId>
                <artifactId>jackson-datatype-jsr310</artifactId>
                <version>2.17.1</version>
            </dependency>
        </dependencies>
    </dependencyManagement>
</project>
```

**Подключение BOM в потребляющем проекте:**

```xml
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>com.example</groupId>
            <artifactId>my-platform</artifactId>
            <version>1.0.0</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>

<dependencies>
    <!-- версии берутся из BOM, явно не указываются -->
    <dependency>
        <groupId>com.fasterxml.jackson.core</groupId>
        <artifactId>jackson-databind</artifactId>
    </dependency>
    <dependency>
        <groupId>com.fasterxml.jackson.datatype</groupId>
        <artifactId>jackson-datatype-jsr310</artifactId>
    </dependency>
</dependencies>
```

**Зачем это нужно в многомодульном проекте?**

1. **Согласованность версий.** Все модули проекта используют одни и те же версии библиотек, исключая конфликты (`jackson-core 2.15` и `jackson-databind 2.17` — несовместимая комбинация).
2. **Обход ограничения одного `<parent>`.** У модуля может быть только один родительский POM. BOM через `scope=import` позволяет импортировать несколько независимых наборов версий.
3. **Разделение ответственности.** BOM публикуется как отдельный артефакт: команды платформы обновляют BOM, команды продуктов подтягивают новую версию BOM и автоматически получают все согласованные обновления.
4. **Управление транзитивными зависимостями.** Версии в `<dependencyManagement>` импортированного BOM переопределяют версии, найденные в дереве транзитивных зависимостей.

**Аналог в Gradle**

Gradle решает задачу BOM двумя способами.

*Способ 1 — импорт внешнего Maven BOM* через функцию `platform()`:

```kotlin
// build.gradle.kts
dependencies {
    implementation(platform("org.springframework.boot:spring-boot-dependencies:3.3.0"))
    // версии определяются BOM, явно не указываются
    implementation("org.springframework.boot:spring-boot-starter-web")
    implementation("com.fasterxml.jackson.core:jackson-databind")
}
```

`enforcedPlatform()` работает аналогично, но принудительно перекрывает любые другие версии в графе зависимостей (аналог `<dependencyManagement>` с полным приоритетом). Его следует использовать с осторожностью в библиотеках, предназначенных для повторного использования, так как forced-версии транзитивно навязываются потребителям.

*Способ 2 — собственная платформа* через плагин `java-platform`:

```kotlin
// platform/build.gradle.kts
plugins {
    `java-platform`
}

dependencies {
    constraints {
        api("com.fasterxml.jackson.core:jackson-databind:2.17.1")
        api("com.fasterxml.jackson.datatype:jackson-datatype-jsr310:2.17.1")
        runtime("org.postgresql:postgresql:42.7.3")
    }
}
```

Потребляющий модуль:
```kotlin
// app/build.gradle.kts
dependencies {
    implementation(platform(project(":platform")))
    implementation("com.fasterxml.jackson.core:jackson-databind") // версия из платформы
}
```

При публикации через `maven-publish` Gradle-платформа генерирует BOM-совместимый POM-файл с блоком `<dependencyManagement>`, что обеспечивает совместимость с Maven-потребителями.

**Ключевые отличия Gradle-платформы от Maven BOM:** Gradle разделяет `api`-ограничения (compile + runtime) и `runtime`-ограничения (только runtime), что даёт более тонкий контроль. Maven BOM не различает эти два scope в `<dependencyManagement>`.

### Вопрос 19. Если зависимость A транзитивно тянет библиотеку C версии 1.0, а зависимость B транзитивно тянет C версии 2.0, и оба пути имеют одинаковую глубину, — какую версию выберет Maven? Как это поведение изменилось в Maven 4?

**Категория:** Maven/Gradle · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Introduction to the Dependency Mechanism — Apache Maven](https://maven.apache.org/guides/introduction/introduction-to-dependency-mechanism.html)

Maven использует алгоритм «ближайшего определения» (nearest definition / nearest-wins) для разрешения конфликтов версий транзитивных зависимостей. Правило формулируется так: выбирается та версия зависимости, которая находится ближе всего к корню дерева зависимостей.

Если два пути до одной и той же зависимости имеют **одинаковую глубину**, применяется второе правило: **выигрывает первое объявление** (first declaration wins) — то есть выбирается версия, соответствующая зависимости, объявленной раньше в файле `pom.xml`.

Пример из официальной документации:
```
A
├── B
│   └── C
│       └── D 2.0
└── E
    └── D 1.0
```
Здесь путь до D через E короче (2 перехода против 3), поэтому выбирается D 1.0.

Если же оба пути одинаковой длины:
```
A
├── B
│   └── D 1.0   (объявлено первым)
└── E
    └── D 2.0
```
Выбирается D 1.0, поскольку B объявлена в `pom.xml` раньше, чем E.

Для гарантированного выбора нужной версии рекомендуется явно объявить зависимость в самом проекте (direct dependency), что полностью переопределяет любые транзитивные решения.

#### Ответ от Claude

**Алгоритм разрешения конфликтов в Maven 3 (текущее поведение)**

Maven 3 применяет двухуровневый алгоритм медиации зависимостей:

1. **Nearest-wins (ближайшее определение).** Из всех путей, ведущих к одной и той же зависимости, выбирается та версия, чей путь от корня проекта короче. Это может приводить к **нежелательным даунгрейдам**: если прямая зависимость (глубина 1) тянет старую версию, а транзитивная зависимость глубины 3 тянет новую — Maven выберет старую.

2. **First-declaration-wins (первое объявление побеждает).** Если два пути до зависимости имеют **одинаковую глубину**, победа достаётся той версии, чья родительская зависимость объявлена **раньше** в блоке `<dependencies>` файла `pom.xml`.

**Применительно к вопросу:** если A и B являются прямыми зависимостями проекта (глубина 1), а C — их транзитивной зависимостью (глубина 2), то глубина у обоих путей одинакова. Maven выберет **ту версию C, чья родительская зависимость (A или B) объявлена первой** в `pom.xml`.

Пример: если в `pom.xml` сначала идёт зависимость A, а потом B — будет выбрана C версии 1.0.

**Как переопределить:** явно добавить C в `<dependencies>` проекта или зафиксировать версию через `<dependencyManagement>`.

**Инструмент диагностики:**
```bash
mvn dependency:tree -Dverbose
```
Флаг `-Dverbose` покажет все конфликтующие версии, включая те, которые были отброшены (строки с пометкой `omitted for conflict`).

**Изменения в Maven 4**

В Maven 4.0 алгоритм nearest-wins **официально сохранён без изменений** как поведение по умолчанию. Однако в сообществе ведётся активная дискуссия о его недостатках (задача [MNG-7852](https://issues.apache.org/jira/browse/MNG-7852) в JIRA Apache): алгоритм nearest-wins считается непредсказуемым, поскольку результат зависит от «случайных» факторов — глубины пути и порядка объявления, а не от явных требований разработчика. Реальные проблемы включают нежелательный даунгрейд `protobuf-java`, приводящий к `NoSuchMethodError` во время выполнения.

Предложения, обсуждавшиеся для Maven 4:
- Стратегия **«highest-wins»** как умолчание для нового формата POM (4.1.0).
- Полное разворачивание графа зависимостей перед выбором версии (вместо остановки на первом найденном пути).
- Конфигурируемый атрибут `directDependencyStrategy` с режимами `fixed` (текущее поведение) и `flexible`.

На момент выхода Maven 4.0.0 данные предложения **не были включены** в релиз в виде изменения дефолтного поведения. MNG-7852 по-прежнему открыт. Таким образом, практически значимые изменения для ответа на вопрос собеседования: в Maven 4 алгоритм nearest-wins и first-declaration-wins **остались такими же**, как в Maven 3.

**Рекомендуемая практика** для управления версиями в любом Maven:
- Использовать `<dependencyManagement>` для явной фиксации версий.
- Подключить плагин `maven-enforcer` с правилом `requireUpperBoundDeps` для обнаружения даунгрейдов.
- Применять плагин `dependency:tree -Dverbose` для аудита дерева зависимостей.

### Вопрос 20. Как исключить конкретную транзитивную зависимость в Maven с помощью \<exclusions\>? В каких случаях это необходимо и какие подводные камни несёт массовое использование exclusions?

**Категория:** Maven/Gradle · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Introduction to Optional Dependencies and Dependency Exclusions — Apache Maven](https://maven.apache.org/guides/introduction/introduction-to-optional-and-excludes-dependencies.html)

Официальная документация Maven объясняет механизм `<exclusions>` следующим образом.

Исключения объявляются **на уровне конкретной зависимости**, а не на уровне всего POM. Это сделано намеренно: глобальное исключение могло бы сломать другие зависимости, которым этот артефакт действительно нужен.

Синтаксис:
```xml
<dependency>
  <groupId>sample.ProjectA</groupId>
  <artifactId>Project-A</artifactId>
  <version>1.0</version>
  <exclusions>
    <exclusion>
      <groupId>sample.ProjectB</groupId>
      <artifactId>Project-B</artifactId>
    </exclusion>
  </exclusions>
</dependency>
```

Эффект: если дерево зависимостей выглядит как `Project-A -> Project-D -> Project-E, Project-F`, а Project-D исключён через exclusion, то из classpath убираются Project-D и все его транзитивные зависимости (Project-E, Project-F).

Рекомендация из документации: `<exclusions>` — это **инструмент последнего средства**, когда разработчики upstream-библиотеки не пометили зависимость как `<optional>`. Предпочтительные альтернативы — использование `<dependencyManagement>`, явные зависимости, правильные scope.

Исключения наследуются транзитивно: если проект X зависит от проекта A, а A исключает Project-D — то Project-D также будет отсутствовать в классовом пути проекта X.

#### Ответ от Claude

**Синтаксис и механизм работы**

`<exclusions>` объявляется внутри конкретной зависимости в `pom.xml`:

```xml
<dependency>
    <groupId>com.example</groupId>
    <artifactId>library-a</artifactId>
    <version>1.0.0</version>
    <exclusions>
        <exclusion>
            <groupId>commons-logging</groupId>
            <artifactId>commons-logging</artifactId>
        </exclusion>
    </exclusions>
</dependency>
```

Обратите внимание: **версия в `<exclusion>` не указывается** — исключаются все версии указанного артефакта, приходящие через данную зависимость. Начиная с Maven 3.x поддерживается wildcard-исключение: `<groupId>*</groupId><artifactId>*</artifactId>` — убирает все транзитивные зависимости данного артефакта.

**Когда exclusions необходимы**

1. **Конфликт реализаций логирования.** Классический пример: библиотека тянет `commons-logging`, но проект использует SLF4J + Logback. Нужно исключить `commons-logging` и добавить `jcl-over-slf4j`.

2. **Уязвимость в транзитивной зависимости.** Если upstream-библиотека не обновляется, а транзитивная зависимость содержит CVE — исключение и явное добавление безопасной версии.

3. **Конфликт jar-файлов с одинаковыми классами.** Например, `javax.servlet-api` vs `servlet-api` — два артефакта, содержащие одни и те же классы.

4. **Дублирующиеся или переименованные артефакты.** Примеры: `log4j` → `reload4j`, старые `com.sun.mail:mail` vs `jakarta.mail`.

5. **Зависимость не нужна в данном окружении.** Например, исключение драйвера БД, который встроен в application server.

**Алгоритм правильного применения**

```bash
# Шаг 1: найти проблемную зависимость
mvn dependency:tree -Dverbose -Dincludes=commons-logging

# Шаг 2: определить через кого она приходит
# Шаг 3: добавить exclusion на нужную зависимость
# Шаг 4: проверить результат
mvn dependency:tree -Dincludes=commons-logging
```

**Подводные камни массового использования exclusions**

1. **Скрытое нарушение функциональности.** Исключённая зависимость может быть реально нужна upstream-библиотеке. Ошибки проявятся только в runtime в виде `ClassNotFoundException` или `NoClassDefFoundError` — не во время компиляции.

2. **Дублирование исключений.** Если несколько зависимостей тянут одну и ту же транзитивную библиотеку, нужно добавить `<exclusion>` к **каждой** из них. Пропуск хотя бы одной сводит эффект к нулю. Проверить это непросто.

3. **Хрупкость при обновлении версий.** При обновлении upstream-библиотеки структура её зависимостей может измениться: нежелательная зависимость может начать приходить через другой путь, а exclusion, написанный для старой версии, перестанет работать.

4. **Трудность сопровождения.** Большое количество `<exclusions>` делает `pom.xml` трудночитаемым и сложным для понимания — теряется прозрачность дерева зависимостей.

5. **Wildcard-исключения опасны.** `<groupId>*</groupId><artifactId>*</artifactId>` удаляет **все** транзитивные зависимости, включая те, без которых библиотека не работает. Применять только если нужны исключительно API/интерфейсы без реализаций.

6. **Исключение наследуется потребителями.** Если ваша библиотека публикуется и кто-то на неё зависит — он унаследует все ваши exclusions. Это может сломать его проект, если ему та зависимость нужна.

**Предпочтительные альтернативы**

- **`<dependencyManagement>`** — фиксирует версию транзитивной зависимости без её исключения. Предпочтителен, когда нужна конкретная версия, а не полное исключение.
- **Maven Enforcer Plugin** (`requireUpperBoundDeps`, `bannedDependencies`) — явно запрещает определённые артефакты на уровне всего build и падает с ошибкой, если они появляются.
- **Правильный `<scope>`** — зависимость с `scope=provided` или `scope=test` не попадает в runtime/compile classpath.
- **Обратиться к авторам upstream-библиотеки** с просьбой пометить ненужную зависимость как `<optional>true</optional>`.
