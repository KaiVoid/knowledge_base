### Вопрос 21. В Gradle чем отличаются конфигурации api и implementation в плагине java-library? Что такое «утечка зависимости» на compile classpath потребителя и как implementation её предотвращает?

**Категория:** Maven/Gradle · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [The Java Library Plugin — Gradle User Manual](https://docs.gradle.org/current/userguide/java_library_plugin.html)

Официальная документация Gradle описывает разницу между `api` и `implementation` следующим образом.

Плагин `java-library` вводит две основные конфигурации для объявления зависимостей:

- **`api`**: зависимость «транзитивно экспонируется потребителям библиотеки» и поэтому попадает на compile classpath потребителей.
- **`implementation`**: зависимость «внутренняя для компонента» и **не экспонируется** потребителям, поэтому **не попадает** на compile classpath потребителей.

Преимущества использования `implementation`:
- Зависимости больше не «утекают» на compile classpath потребителей — невозможно случайно использовать транзитивную зависимость.
- Более быстрая компиляция за счёт уменьшения classpath.
- Меньше перекомпиляций при изменении implementation-зависимостей.
- Чище публикация артефактов.

Концепция ABI (Application Binary Interface): `api`-зависимость должна использоваться, когда хотя бы один тип из неё присутствует в публичном API библиотеки — в сигнатурах публичных методов, полях, суперклассах или реализуемых интерфейсах.

Рекомендация: **предпочитать `implementation` везде, где возможно**, используя `api` только тогда, когда типы зависимости реально присутствуют в публичном интерфейсе библиотеки.

#### Ответ от Claude

**Контекст: для чего нужен плагин java-library**

Стандартный плагин `java` не различает зависимости, нужные только для внутренней реализации, и зависимости, являющиеся частью публичного API. В результате **все** зависимости библиотеки транзитивно попадали на compile classpath её потребителей. Плагин `java-library` решает эту проблему.

**Конфигурация `api`**

Зависимость объявляется как `api`, если хотя бы один её тип **присутствует в публичном ABI библиотеки**:

```groovy
// build.gradle (модуль-библиотека)
plugins {
    id 'java-library'
}

dependencies {
    // Jackson используется в возвращаемых типах публичных методов
    api 'com.fasterxml.jackson.core:jackson-databind:2.15.0'

    // HTTP-клиент используется только внутри реализации
    implementation 'com.squareup.okhttp3:okhttp:4.12.0'
}
```

Признаки того, что зависимость должна быть `api`:
- Тип из зависимости фигурирует в **возвращаемом типе** публичного метода.
- Тип из зависимости фигурирует в **параметрах** публичного метода (включая generics).
- Тип из зависимости используется в качестве **суперкласса** или **реализуемого интерфейса**.
- Тип из зависимости является **публичным полем**.
- Тип из зависимости используется в **публичных аннотациях**.

**Конфигурация `implementation`**

Зависимость объявляется как `implementation`, если она используется **только внутри метода** (в теле) или в приватных членах класса. Потребитель библиотеки не «видит» эти типы при компиляции.

**Что такое «утечка зависимости» на compile classpath**

До появления `java-library` (и при использовании устаревшей конфигурации `compile`) все зависимости библиотеки автоматически появлялись на compile classpath её потребителей. Это создавало несколько проблем:

1. **Случайное использование транзитивных зависимостей.** Разработчик видит в IDE классы из `okhttp`, хотя не объявлял его как прямую зависимость. Если потом библиотека перейдёт на другой HTTP-клиент — код потребителя сломается.

2. **Ненужная перекомпиляция.** Изменение любой внутренней зависимости библиотеки вызывает перекомпиляцию всех потребителей, даже если публичный API не изменился.

3. **Раздутый classpath.** Потребитель получает на compile classpath все транзитивные зависимости, включая те, которые ему не нужны.

**Как `implementation` предотвращает утечку**

Gradle строго разграничивает два вида classpath для потребителей:

| Classpath | api-зависимости | implementation-зависимости |
|-----------|-----------------|---------------------------|
| Compile classpath потребителя | Присутствуют | **Отсутствуют** |
| Runtime classpath потребителя | Присутствуют | Присутствуют |

Благодаря этому:
- Потребитель **не может скомпилироваться** с использованием `implementation`-зависимости библиотеки — ошибка компиляции немедленно сигнализирует об утечке.
- При изменении `implementation`-зависимости **перекомпилируется только библиотека**, но не её потребители (при неизменном ABI).
- Опубликованный POM библиотеки содержит `implementation`-зависимости со scope `runtime`, а `api`-зависимости — со scope `compile`.

**Пример полного разграничения**

```groovy
// Модуль :awesome-lib
dependencies {
    api 'com.fasterxml.jackson.core:jackson-databind:2.15.0'
    implementation 'com.google.guava:guava:32.0-jre'
}
```

```groovy
// Модуль :app, зависящий от :awesome-lib
dependencies {
    implementation project(':awesome-lib')
}

// В коде :app:
JsonNode node = ...; // OK: jackson-databind доступен через api
ImmutableList list = ...; // ОШИБКА КОМПИЛЯЦИИ: guava не доступен через implementation
```

**Практические рекомендации**

- **По умолчанию использовать `implementation`** — это безопаснее и быстрее.
- Переходить на `api` только при явной необходимости — когда типы из зависимости реально присутствуют в публичном контракте.
- Использовать плагин `java-library` для всех модулей, публикующих артефакты; для конечных приложений (не библиотек) плагина `java` достаточно — там `implementation` делает то же самое, что `compile` раньше.
- В многомодульных проектах правильное разграничение `api`/`implementation` значительно ускоряет инкрементальную компиляцию.
### Вопрос 22. В чём разница между platform() и enforcedPlatform() при импорте BOM в Gradle? Какие побочные эффекты у enforcedPlatform и почему Spring Boot Gradle Plugin перешёл на обычный platform()?

**Категория:** Maven/Gradle · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Platforms — Gradle User Manual](https://docs.gradle.org/current/userguide/platforms.html)

**`platform()`** импортирует BOM и применяет версии зависимостей как *рекомендации*. Другие версии и ограничения в графе зависимостей могут привести к тому, что будет выбрана версия, отличная от той, что объявлена в BOM.

**`enforcedPlatform()`** рассматривает версии из BOM как *требования* и переопределяет любую другую версию в графе зависимостей. Более того, это поведение *транзитивно*: оно распространяется на всех потребителей вашего проекта. Если потребители не согласны с принудительными версиями, им придётся использовать `exclude`, чтобы обойти это ограничение.

Документация явно предупреждает: "Проявляйте осторожность при использовании `enforcedPlatform`, если ваш компонент предназначен для потребления другими проектами. Это объявление транзитивно и влияет на граф зависимостей ваших потребителей." В качестве альтернативы для переиспользуемых компонентов Gradle рекомендует применять богатые объявления версий с `strictly`.

#### Ответ от Claude

**Различие между `platform()` и `enforcedPlatform()`**

| Характеристика | `platform()` | `enforcedPlatform()` |
|---|---|---|
| Версии из BOM | Рекомендации (soft constraints) | Требования (hard constraints) |
| Можно ли переопределить явной версией | Да | Нет — BOM-версия всё равно победит |
| Транзитивность | Нет | Да — расплространяется на потребителей |
| Рекомендованная область применения | Приложения и библиотеки | Только конечные приложения |

**Как работает `platform()`**

```groovy
dependencies {
    implementation platform("org.springframework.boot:spring-boot-dependencies:3.2.0")
    implementation "org.springframework.boot:spring-boot-starter-web" // версия из BOM
    implementation "com.fasterxml.jackson.core:jackson-databind:2.16.0" // переопределяет BOM
}
```

Если в BOM прописан `jackson-databind:2.15.x`, но вы явно задаёте `2.16.0` — Gradle выберет явно указанную версию. Это стандартное поведение разрешения версий по принципу "победит самая высокая".

**Как работает `enforcedPlatform()`**

```groovy
dependencies {
    implementation enforcedPlatform("org.springframework.boot:spring-boot-dependencies:3.2.0")
    implementation "com.fasterxml.jackson.core:jackson-databind:2.16.0" // ИГНОРИРУЕТСЯ
}
```

Даже если вы явно задаёте `2.16.0`, Gradle принудительно применит версию из BOM. Escape-хэтч — только `exclude`, что делает управление зависимостями крайне неудобным для потребителей.

**Побочные эффекты `enforcedPlatform()`**

1. **Транзитивное навязывание версий.** Если ваша библиотека A публикуется с зависимостью `enforcedPlatform(someBOM)`, то все проекты, подключающие A, автоматически получают принудительно зафиксированные версии из этого BOM — даже если они хотят использовать другие версии. Это задокументировано в Gradle Issue #18454: "проекты, потребляющие такую библиотеку, получают строго принудительные версии их транзитивных зависимостей".

2. **Невозможность переопределения через `strictly`.** Даже объявление `strictly("2.16.0")` не поможет потребителю — принудительная версия из BOM перекроет это ограничение.

3. **Единственный обходной путь — `exclude`.** Потребителю придётся исключать конкретные артефакты, что разрушает преимущества BOM-управления.

**Почему Spring Boot Gradle Plugin отказался от `enforcedPlatform()`**

До Spring Boot 2.4 плагин `io.spring.dependency-management` внутренне использовал механизм, схожий с `enforcedPlatform()`. Это порождало проблемы:

- Модуль `spring-boot-configuration-metadata` транзитивно протаскивал принудительные версии через `enforcedPlatform(spring-boot-dependencies)`, что не позволяло downstream-проектам переопределить версии даже через явные constraints (Issue #27726 в spring-projects/spring-boot, исправлено в 2.4.10).
- Потребители теряли контроль над версиями зависимостей — а это противоречит принципу "BOM даёт рекомендации, приложение решает".

Начиная с актуальных версий Spring Boot, документация предлагает использовать нативный Gradle-подход с `platform()` вместо `enforcedPlatform()`:

```groovy
dependencies {
    implementation platform(org.springframework.boot.gradle.plugin.SpringBootPlugin.BOM_COORDINATES)
}
```

Это даёт: а) более быструю сборку по сравнению с `io.spring.dependency-management`, б) гибкость — явные версии и `constraints` можно переопределять, в) отсутствие транзитивного "заражения" потребителей принудительными версиями.

**Итог:** `enforcedPlatform()` полезен только в финальных приложениях (где нет потребителей), когда нужно гарантировать конкретный набор версий. В библиотеках — только `platform()`.

### Вопрос 23. Как Maven обрабатывает зависимости с scope provided при транзитивном наследовании? Если библиотека A объявляет зависимость B как provided, увидит ли потребитель A зависимость B в своём classpath?

**Категория:** Maven/Gradle · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Introduction to the Dependency Mechanism — Apache Maven](https://maven.apache.org/guides/introduction/introduction-to-dependency-mechanism.html)

Официальная документация Maven объясняет: scope `provided` означает, что зависимость требуется для компиляции и тестов, но ожидается, что JDK или контейнер предоставят её во время выполнения. Явно указано: "A dependency with this scope is added to the classpath used for compilation and test, but not the runtime classpath. **It is not transitive.**"

Таблица разрешения транзитивных зависимостей в Maven подтверждает это. Если проект A зависит от библиотеки X с scope `compile`, а X объявляет зависимость Y с scope `provided` — Y **опускается** (ячейка пустая, "—"). Аналогично: если сама X подключена как `provided` или `runtime` — все её provided-транзитивы тоже опускаются.

#### Ответ от Claude

**Как Maven транзитивно разрешает scope `provided`**

Ключевой факт: **`provided` — нетранзитивный scope**. Это значит, что потребитель библиотеки A никогда не получит зависимость B (с scope `provided`) автоматически ни в compile-, ни в runtime-, ни в test-classpath.

**Таблица разрешения транзитивных зависимостей Maven**

В документации Maven есть таблица, где строка — scope зависимости A→X (как потребитель видит X), столбец — scope X→Y (как X объявляет Y):

| Scope A→X \ Scope X→Y | compile | provided | runtime | test |
|---|---|---|---|---|
| **compile** | compile | *опущено* | runtime | *опущено* |
| **provided** | provided | *опущено* | provided | *опущено* |
| **runtime** | runtime | *опущено* | runtime | *опущено* |
| **test** | test | *опущено* | test | *опущено* |

Столбец `provided` везде пустой ("—"). Это означает: **если X объявляет Y с scope `provided`, потребитель X никогда не получит Y транзитивно** — ни при каком scope, с которым потребитель подключает X.

**Практический пример**

```
Библиотека A (ваш код)
  └── зависит на: Servlet API с scope=provided

Проект-потребитель
  └── зависит на: библиотека A с scope=compile
```

Что получает потребитель:
- Библиотека A — на compile classpath.
- Servlet API — **НЕТ**. Ни в compile, ни в runtime classpath.

Если потребителю нужен Servlet API (например, для компиляции кода, который использует типы из A), он должен объявить его **явно** в своём `pom.xml` с нужным scope.

**Зачем так устроено**

Цель scope `provided` — сигнализировать: "эта зависимость будет предоставлена окружением" (сервлет-контейнер, JDK, application server). Транзитивное распространение нарушило бы этот контракт — потребитель получил бы библиотеку, которую сам контейнер уже предоставляет, что привело бы к конфликтам версий и раздуванию classpath.

**Частая ошибка на практике**

```xml
<!-- В pom.xml библиотеки A -->
<dependency>
    <groupId>org.projectlombok</groupId>
    <artifactId>lombok</artifactId>
    <version>1.18.30</version>
    <scope>provided</scope>
</dependency>
```

Потребитель A **не получит** Lombok транзитивно. Если потребитель сам использует Lombok-аннотации — он обязан объявить его явно. Это корректное поведение: `provided` — это детали реализации библиотеки, не API-контракт.

**Отличие от scope `optional`**

`optional` — это декларативный способ сказать "эта зависимость нужна только в специфических сценариях". Она тоже не транзитивна (потребитель должен объявить явно), но семантически другая: `provided` — "контейнер предоставит", `optional` — "может быть нужна, но не обязательна".

**Итог:** Потребитель библиотеки A **не увидит** зависимость B (`provided`) в своём classpath. Это намеренное поведение Maven. Если B нужна потребителю — он должен объявить её самостоятельно в своём `pom.xml`.

### Вопрос 24. Что такое Maven Reactor в многомодульном проекте? Как Reactor определяет порядок сборки модулей и что произойдёт при наличии циклических зависимостей между модулями?

**Категория:** Maven/Gradle · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Guide to Working with Multiple Modules — Apache Maven](https://maven.apache.org/guides/mini/guide-multiple-modules.html)

Официальная документация Maven описывает Reactor как механизм ядра Maven, который:
1. Собирает все доступные модули для сборки.
2. Сортирует проекты в корректный порядок сборки.
3. Собирает выбранные проекты по порядку.

Порядок определяется по следующим правилам (в порядке приоритета): зависимость проекта на другой модуль сборки; объявление плагина, являющегося другим модулем; зависимость плагина на другой модуль; объявление build extension на другой модуль; порядок объявления в элементе `<modules>` (если другие правила не применяются). При этом важно: только "инстанциированные" ссылки учитываются — элементы `dependencyManagement` и `pluginManagement` на порядок сборки **не влияют**.

Документация не описывает поведение при цикличных зависимостях явно, однако алгоритм сортировки предполагает DAG (направленный ациклический граф).

#### Ответ от Claude

**Что такое Maven Reactor**

Maven Reactor — это подсистема ядра Maven, отвечающая за оркестрацию сборки многомодульных проектов. Без Reactor разработчику пришлось бы вручную заходить в каждый модуль и запускать `mvn install` в правильном порядке. Reactor автоматизирует весь процесс.

Reactor активируется автоматически, когда Maven находит родительский `pom.xml` с секцией `<modules>`. Специальной конфигурации не требуется.

**Алгоритм определения порядка сборки**

Reactor строит направленный ациклический граф (DAG) из модулей и их зависимостей, затем выполняет топологическую сортировку. Порядок сборки определяется по следующим правилам (приоритет сверху вниз):

1. **Зависимость `<dependency>` на другой модуль** — самый распространённый случай: модуль `service` зависит от `core`, значит `core` соберётся первым.
2. **Объявление плагина**, являющегося другим модулем в той же сборке.
3. **Зависимость плагина** на другой модуль сборки.
4. **Декларация build extension** на другой модуль.
5. **Порядок в `<modules>`** — только если ни одно из правил выше не применяется.

Критически важный нюанс: `dependencyManagement` и `pluginManagement` **не создают реальных зависимостей** между модулями и не влияют на порядок сборки. Только фактически используемые зависимости (`<dependency>`) участвуют в топологической сортировке.

**Пример типичного вывода Reactor**

```
[INFO] Reactor Build Order:
[INFO]   parent-project               [pom]
[INFO]   core-module                  [jar]
[INFO]   service-module               [jar]
[INFO]   web-module                   [war]
```

Здесь Maven автоматически определил: `core-module` не зависит ни от кого → `service-module` зависит от `core-module` → `web-module` зависит от `service-module`.

**Управление Reactor через CLI**

```bash
# Собрать только указанный модуль и все его зависимости в рамках реактора
mvn install -pl service-module --also-make

# Собрать модуль и все, кто от него зависит
mvn install -pl core-module --also-make-dependents

# Продолжить сборку с определённого модуля после сбоя
mvn install --resume-from service-module

# Продолжать сборку при ошибке, собрать все модули и показать все ошибки в конце
mvn install --fail-at-end
```

**Параллельная сборка**

Reactor поддерживает параллельную сборку независимых модулей:

```bash
mvn install -T 4        # 4 потока
mvn install -T 1C       # по одному потоку на CPU
```

При этом Reactor гарантирует: если модуль B зависит от A, то B не начнёт собираться, пока A не завершится — даже при параллельной сборке.

**Что происходит при циклических зависимостях**

При наличии цикла (A зависит от B, B зависит от A) Maven **не может построить корректный DAG** и завершает работу с ошибкой ещё на фазе анализа, до начала сборки:

```
[ERROR] The projects in the reactor contain a cyclic reference:
Edge between 'Vertex{label='com.example:module-a:1.0-SNAPSHOT'}'
and 'Vertex{label='com.example:module-b:1.0-SNAPSHOT'}'
introduces a cycle in the graph
```

Это не предупреждение — это **фатальная ошибка**. Maven 2 и Maven 3 оба не допускают цикличных зависимостей между модулями.

Варианты решения:
- **Выделить общий код** в отдельный модуль `common`, от которого зависят оба.
- **Инвертировать зависимость**: пересмотреть архитектуру — чаще всего цикл сигнализирует о нарушении принципов разделения ответственности.
- **Объединить модули**: если два модуля реально неразделимы, возможно, это один модуль.
- **Использовать интерфейсы**: выделить API-модуль с интерфейсами, который импортируют оба.

**Практическая рекомендация:** инструменты статического анализа (SonarQube, IntelliJ IDEA, `mvn dependency:analyze`) помогают выявлять потенциальные цикличные зависимости ещё до того, как они попадут в `pom.xml` в виде `<dependency>` между модулями.
### Вопрос 25. Чем Maven-профиль отличается от обычной конфигурации в pom.xml? Перечислите все способы активации профиля и объясните, что происходит с профилями, помеченными `<activeByDefault>true</activeByDefault>`, когда командной строкой активируется другой профиль.

**Категория:** Maven/Gradle · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Introduction to Build Profiles — Apache Maven](https://maven.apache.org/guides/introduction/introduction-to-profiles.html)

Профили представляют собой подмножество элементов POM, позволяющих условно модифицировать сборку в зависимости от окружения. Главное отличие от обычной конфигурации: профили срабатывают только при выполнении условия активации, тогда как обычная конфигурация всегда применяется безусловно.

Что можно менять внутри профиля в `pom.xml`: `<repositories>`, `<pluginRepositories>`, `<dependencies>`, `<plugins>`, `<properties>`, `<modules>`, `<reports>`, `<reporting>`, `<dependencyManagement>`, `<distributionManagement>`, а также подмножество `<build>` (`<defaultGoal>`, `<resources>`, `<testResources>`, `<directory>`, `<finalName>`, `<filters>`, `<pluginManagement>`, `<plugins>`). В профилях `settings.xml` доступны только `<repositories>`, `<pluginRepositories>` и `<properties>`.

Способы активации профиля:

1. **Командная строка** — флаг `-P`: `mvn install -P profile-1,profile-2`
2. **`<activeByDefault>true</activeByDefault>`** внутри секции `<activation>` профиля
3. **Версия JDK**: `<activation><jdk>11</jdk></activation>` или диапазон `<jdk>[11,17)</jdk>`
4. **Операционная система**: `<activation><os><family>Windows</family></os></activation>`
5. **Системное свойство / свойство командной строки**: `<activation><property><name>env</name><value>prod</value></property></activation>` — активируется при `-Denv=prod`
6. **Наличие или отсутствие файла**: `<activation><file><exists>src/env.properties</exists></file></activation>` / `<file><missing>...</missing></file>`
7. **`<activeProfiles>` в `settings.xml`**: профиль активируется глобально для пользователя

Поведение `activeByDefault` при явной активации другого профиля (дословная цитата из документации):

> "This profile will automatically be active for all builds unless another profile in the same POM is activated using one of the previously described methods. All profiles that are active by default are automatically deactivated when a profile in the POM is activated on the command line or through its activation config."

То есть если выполнить `mvn install -P profile-2`, то профиль с `<activeByDefault>true</activeByDefault>` автоматически деактивируется. Это касается только профилей из того же POM-файла.

Явно деактивировать профиль (включая `activeByDefault`) можно командой:
```
mvn install -P !profile-1
# или (рекомендуется из-за ошибки CLI-309 с дефисом):
mvn install -P=-profile-1
```

#### Ответ от Claude

**Чем профиль отличается от обычной конфигурации**

Обычная конфигурация в `pom.xml` применяется при каждой сборке без исключения. Профиль — это условный блок конфигурации: он модифицирует POM в момент сборки только тогда, когда выполнено условие активации. Это позволяет одному проекту иметь разные параметры для разных окружений (dev/staging/prod), разных ОС или JDK без дублирования файлов.

**Все способы активации**

| Способ | Механизм |
|---|---|
| Командная строка | `mvn ... -P profile-id` |
| `<activeByDefault>` | Автоматически при отсутствии других активных профилей из этого POM |
| Версия JDK | `<activation><jdk>11</jdk></activation>` |
| Операционная система | `<activation><os><family>unix</family></os></activation>` |
| Системное свойство | `<activation><property><name>X</name><value>Y</value></property></activation>` |
| Наличие/отсутствие файла | `<activation><file><exists>path</exists></file></activation>` |
| `settings.xml` (`<activeProfiles>`) | Глобальная активация на уровне пользователя или всей машины |

Несколько условий внутри одного `<activation>` трактуются как логическое И — профиль активируется только при выполнении всех условий одновременно.

**Поведение `activeByDefault` при запуске с `-P`**

Ключевой момент: `activeByDefault` действует как запасной вариант. Как только в том же POM активируется хотя бы один другой профиль (через командную строку, `settings.xml` или собственный триггер активации), все профили с `activeByDefault=true` из этого POM автоматически выключаются. Это поведение задокументировано в официальных источниках и подтверждается практикой.

Пример: есть профили `base` (activeByDefault=true) и `prod`.
- `mvn install` — активен только `base`
- `mvn install -P prod` — активен только `prod`, `base` деактивирован
- `mvn install -P prod,base` — активны оба (явное включение base)

**Расхождение между источниками:** В старых версиях Maven (до 3.x) поведение `activeByDefault` при наличии автоматически активируемых профилей (например, через условие `<jdk>`) могло различаться. Начиная с Maven 3, поведение унифицировано: любая активация профиля из этого POM отключает `activeByDefault`.

### Вопрос 26. Можно ли объявить профиль в родительском POM и рассчитывать, что дочерние модули унаследуют его активацию? Как Maven на самом деле обрабатывает профили из родительского POM в контексте многомодульного проекта?

**Категория:** Maven/Gradle · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Introduction to Build Profiles — Apache Maven](https://maven.apache.org/guides/introduction/introduction-to-profiles.html)

Официальная документация Apache Maven прямо поясняет механизм обработки профилей в многомодульных проектах:

> "The profiles are not inherited by child POMs. Instead, they are resolved very early by the Maven Model Builder and only the effects of active profiles are inherited (e.g. the plugins defined in the profile). Implicit profile activation only has an effect on the surrounding profile container but never on any other profile (even if it has the same id)."

Иными словами:
- Профили **не наследуются** дочерними POM.
- Профили из родительского POM **разрешаются очень рано** — до того как происходит наследование.
- Если профиль в родительском POM активен, его эффекты (плагины, зависимости, свойства) попадают в собранный родительский POM, и уже этот собранный POM передаётся дочернему модулю через механизм наследования.
- Условия `<activation>` (например, по имени файла, свойству) в профиле родителя **не переносятся** в дочерний модуль и не проверяются в его контексте.

#### Ответ от Claude

**Короткий ответ: нет, профиль не наследуется**

Рассчитывать на унаследованную активацию профиля из родительского POM нельзя. Объявить профиль в родителе — не то же самое, что объявить его в дочернем модуле.

**Как Maven реально обрабатывает профили из родительского POM**

Maven использует компонент Maven Model Builder, который обрабатывает профили на очень ранней стадии жизненного цикла сборки — до того как происходит объединение (merge) родительского и дочернего POM. Порядок следующий:

1. Maven читает родительский POM.
2. Для родительского POM проверяются условия активации профилей (командная строка, `<jdk>`, `<os>`, `<property>`, `<file>`, `activeByDefault`).
3. Активные профили применяются к родительскому POM — он «разворачивается» с учётом профилей.
4. Уже этот развёрнутый родительский POM передаётся дочернему модулю как основа для наследования.
5. Для дочернего POM профили родителя **заново не проверяются** и не переносятся.

**Практические последствия**

1. **Свойства (`<properties>`) в активном профиле родителя** — работают корректно, потому что свойства наследуются из развёрнутого родителя. Если профиль родителя активирован (например, через командную строку), дочерний модуль получит эти свойства.

2. **Плагины в профиле родителя** — частичная работа. Если профиль с плагином активен в контексте родителя (например, через `-P` при сборке из корня), плагин попадает в развёрнутый родительский POM. Но этот плагин будет применяться к родителю, а не к дочернему модулю. Для корректного распространения на дочерние модули плагин надо объявлять в `<pluginManagement>` внутри профиля в родителе и затем отдельно подключать в дочернем через `<plugins>`.

3. **Условия `<activation>` (файл, свойство, JDK) в профиле родителя** — не переносятся в контекст дочернего модуля. Условие проверяется только в контексте родителя. Это задокументировано и подтверждается баг-репортом MNG-3228 (закрыт как «Cannot Reproduce», но смежные проблемы MNG-3897, MNG-3773, MNG-5127 указывают, что поведение воспроизводится на практике).

4. **`activeByDefault` в профиле родителя** — если при сборке из корня многомодульного проекта нет других активных профилей, `activeByDefault` сработает в контексте родителя, и его эффекты (через шаг 4 выше) унаследует дочерний. Но если собирать дочерний модуль отдельно (`cd child-module && mvn install`), `activeByDefault`-профиль родителя может не активироваться предсказуемо.

**Правильный подход**

- Если профиль должен влиять на все модули — объявить его в каждом дочернем POM (или вынести общую логику в `settings.xml` / глобальные `settings.xml`).
- Зависимости в профилях родителя — через `<dependencyManagement>`, плагины — через `<pluginManagement>`, чтобы дочерние модули явно подключали их.
- Командная строка (`-P profile-id`) при сборке из корня агрегатора — единственный надёжный способ активировать профиль для всего многомодульного проекта сразу.

**Расхождение между источниками:** Блог DASHbay (2011) описывает подход с `<pluginManagement>` как рабочий обход, но признаёт его неудобство («Yes, it sucks that you have to double-declare things»). Официальная документация Maven это поведение описывает как намеренное, а не как ошибку.

### Вопрос 27. Чем агрегирующий POM (aggregator) отличается от родительского POM (parent) в многомодульном Maven-проекте? Может ли один файл `pom.xml` выполнять обе роли одновременно, и каковы последствия разделения этих ролей на два отдельных POM?

**Категория:** Maven/Gradle · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Introduction to the POM — Apache Maven](https://maven.apache.org/guides/introduction/introduction-to-the-pom.html)

Официальная документация Apache Maven разграничивает понятия следующим образом:

Агрегирующий POM использует элемент `<modules>` и определяет, какие дочерние модули собираются совместно:

> "A project with modules is known as a multi-module, or aggregator project. Modules are projects that this POM lists, and are executed as a group."

Родительский POM — источник наследуемой конфигурации (зависимости, плагины, свойства); дочерние модули ссылаются на него через `<parent>`.

Оба механизма могут сосуществовать в одном файле, и документация явно на это указывает:

> "However, an aggregator project and a parent project are both POM projects, they are not one and the same and should not be confused."

И далее:

> "A POM project may be inherited from — but does not necessarily have — any modules that it aggregates. Conversely, a POM project may aggregate projects that do not inherit from it."

То есть: можно иметь родителя без списка модулей, и можно иметь агрегатор, от которого дочерние модули не наследуются.

#### Ответ от Claude

**Различие ролей**

| Характеристика | Родительский POM (parent) | Агрегирующий POM (aggregator) |
|---|---|---|
| Ключевой элемент в дочернем | `<parent>` в дочернем POM | `<modules>` в самом агрегаторе |
| Направление ссылки | Дочерний указывает на родителя | Родитель указывает на дочерние |
| Основная функция | Общая конфигурация, наследуемая дочерними | Запуск сборки всех модулей одной командой |
| Содержимое | `<dependencyManagement>`, `<pluginManagement>`, `<properties>` | `<modules>` с перечнем подмодулей |
| Деплой в репозиторий | Да, обязателен для корректного наследования | Не обязателен |

**Packaging**

Оба типа требуют `<packaging>pom</packaging>`. Это единственное общее формальное требование.

**Может ли один файл выполнять обе роли**

Да, и это самый распространённый вариант. Большинство реальных многомодульных проектов используют один корневой `pom.xml`, который одновременно:
- содержит `<modules>` (агрегатор)
- содержит `<dependencyManagement>`, `<pluginManagement>`, `<properties>` (родитель)
- является `<parent>` для всех дочерних модулей

Пример Maven Core: весь проект Apache Maven собирается из корневого `pom.xml`, выполняющего обе роли.

**Последствия разделения на два отдельных POM**

Разделение применяется в более сложных проектах и имеет следующие последствия:

*Структура при разделении:*
```
root/
  aggregator/pom.xml     — только <modules>, packaging=pom
  parent/pom.xml         — только dependencyManagement, packaging=pom
  module-a/pom.xml       — <parent> указывает на parent/pom.xml
  module-b/pom.xml       — <parent> указывает на parent/pom.xml
```

*Преимущества разделения:*
1. **Независимость версий**: Родительский POM можно версионировать и релизить отдельно. Несколько несвязанных проектов могут использовать один корпоративный parent POM без агрегатора.
2. **Развёртывание**: Агрегатор не обязательно публиковать в Maven-репозиторий (он нужен только для локальной/CI-сборки). Родительский POM публикуется и должен быть доступен для разрешения зависимостей.
3. **Разделение ответственности**: Агрегатор управляет только структурой модулей, родитель — только конфигурацией.

*Последствия и сложности разделения:*
1. **`relativePath` в дочерних модулях**: При стандартной структуре Maven ищет родительский POM по пути `..` — туда же, где лежит агрегатор. Если агрегатор и родитель — разные файлы, в `<parent>` каждого дочернего модуля нужно явно указать `<relativePath>../parent/pom.xml</relativePath>` или `<relativePath/>` (пустой — поиск только в репозитории).
2. **Два файла для поддержки**: Изменения в структуре проекта требуют правок в агрегаторе, изменения в зависимостях — в родителе.
3. **Риск рассинхронизации версий**: Если версии агрегатора и родителя расходятся, сборка может вести себя непредсказуемо.

**Расхождение между источниками:** Блог rostislav-matl (2011) акцентирует внимание на том, что разделение — архитектурно более чистое решение для крупных проектов, тогда как официальная документация Maven нейтральна и рассматривает оба подхода как равноправные. Baeldung рекомендует объединённый вариант как более простой для большинства проектов.

### Вопрос 28. Какие цели (goals) плагинов Maven автоматически привязаны к фазам жизненного цикла default для упаковки типа `jar`? Назовите плагин и цель для фаз `process-resources`, `compile`, `test` и `package`. Что изменится, если тип упаковки поменять на `war`?

**Категория:** Maven/Gradle · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Plugin Bindings for Default Lifecycle Reference — Maven Core 3.9.9](https://maven.apache.org/ref/3.9.9/maven-core/default-bindings.html)

Для упаковки `jar` к фазам жизненного цикла `default` привязаны следующие цели плагинов:

| Фаза | Плагин | Цель |
|---|---|---|
| `process-resources` | `maven-resources-plugin:3.3.1` | `resources` |
| `compile` | `maven-compiler-plugin:3.13.0` | `compile` |
| `test` | `maven-surefire-plugin:3.2.5` | `test` |
| `package` | `maven-jar-plugin:3.4.1` | `jar` |

Для упаковки `war` все привязки идентичны, за исключением фазы `package`:

| Фаза | Плагин | Цель |
|---|---|---|
| `package` | `maven-war-plugin:3.4.0` | `war` |

Таким образом, единственное изменение при смене `jar` на `war` — замена `maven-jar-plugin:jar` на `maven-war-plugin:war` в фазе `package`.

#### Ответ от Claude

Maven реализует концепцию «соглашение вместо конфигурации»: в зависимости от значения `<packaging>` в POM-файле к определённым фазам жизненного цикла `default` автоматически привязываются стандартные цели плагинов. Эти привязки описаны в файле `default-bindings.xml` внутри `maven-core`.

**Привязки для `jar` (актуальные версии плагинов по состоянию на Maven 3.9.x):**

| Фаза | Плагин:Цель |
|---|---|
| `process-resources` | `maven-resources-plugin:resources` — копирует ресурсы из `src/main/resources` в `target/classes` |
| `compile` | `maven-compiler-plugin:compile` — компилирует исходный код из `src/main/java` в `target/classes` |
| `process-test-resources` | `maven-resources-plugin:testResources` — копирует тестовые ресурсы |
| `test-compile` | `maven-compiler-plugin:testCompile` — компилирует тестовый код |
| `test` | `maven-surefire-plugin:test` — запускает JUnit/TestNG тесты |
| `package` | `maven-jar-plugin:jar` — упаковывает скомпилированные классы в JAR-архив |
| `install` | `maven-install-plugin:install` — устанавливает артефакт в локальный репозиторий |
| `deploy` | `maven-deploy-plugin:deploy` — публикует артефакт в удалённый репозиторий |

**Что изменится при смене `jar` на `war`:**

Единственное отличие — фаза `package` вместо `maven-jar-plugin:jar` выполняет `maven-war-plugin:war`. Этот плагин:
- собирает директорию `src/main/webapp` (статика, `WEB-INF/web.xml`);
- добавляет все зависимости в `WEB-INF/lib`;
- упаковывает всё вместе в WAR-архив.

Фазы `process-resources`, `compile` и `test` имеют одинаковые привязки для обоих типов упаковки.

**Важно:** номера версий плагинов в привязках зависят от версии самого Maven (определяются в `pom.xml` суперПОМа). Явное объявление версии плагина в `<build><plugins>` всегда перекрывает привязку по умолчанию.

### Вопрос 29. В чём принципиальная разница между `<optional>true</optional>` у зависимости и `<exclusions>` на зависимость? Кто принимает решение в каждом из случаев — производитель библиотеки или её потребитель? Приведите конкретный пример, когда уместен каждый подход.

**Категория:** Maven/Gradle · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Optional Dependencies and Dependency Exclusions — Apache Maven](https://maven.apache.org/guides/introduction/introduction-to-optional-and-excludes-dependencies.html)

**Optional-зависимости** используются, когда проект нельзя разбить на подмодули: некоторые зависимости нужны только для определённых функций и не требуются, если эти функции не используются.

Пример объявления:
```xml
<dependency>
  <groupId>sample.ProjectB</groupId>
  <artifactId>Project-B</artifactId>
  <version>1.0</version>
  <optional>true</optional>
</dependency>
```

Если проект A объявляет B как `optional`, а проект X зависит от A, то B **не** добавляется в classpath X автоматически. X должен явно объявить B, если нуждается в нём.

**Exclusions** решают проблему нежелательных транзитивных зависимостей — устаревших jar-файлов с проблемами безопасности или несовместимостью с версией Java.

Пример объявления:
```xml
<dependency>
  <groupId>sample.ProjectA</groupId>
  <artifactId>Project-A</artifactId>
  <version>1.0</version>
  <exclusions>
    <exclusion>
      <groupId>sample.ProjectB</groupId>
      <artifactId>Project-B</artifactId>
    </exclusion>
  </exclusions>
</dependency>
```

Exclusions действуют на весь поддерев зависимостей ниже точки объявления.

#### Ответ от Claude

Два механизма решают противоположные задачи и задействуют разные стороны цепочки зависимостей.

**`<optional>true</optional>` — решение производителя (bottom-up)**

Производитель библиотеки сам решает пометить зависимость как необязательную. Это уместно, когда библиотека поддерживает несколько взаимоисключающих функций и каждая из них требует своей зависимости, которая не нужна всем потребителям.

Классический пример: ORM-библиотека, поддерживающая MySQL, PostgreSQL и Oracle. Она объявляет все три JDBC-драйвера как `optional`. Потребитель сам добавляет только нужный драйвер:

```xml
<!-- В POM библиотеки (производитель) -->
<dependency>
  <groupId>mysql</groupId>
  <artifactId>mysql-connector-java</artifactId>
  <optional>true</optional>
</dependency>
<dependency>
  <groupId>org.postgresql</groupId>
  <artifactId>postgresql</artifactId>
  <optional>true</optional>
</dependency>
```

```xml
<!-- В POM потребителя — добавляет только то, что нужно -->
<dependency>
  <groupId>mysql</groupId>
  <artifactId>mysql-connector-java</artifactId>
  <version>8.0.33</version>
</dependency>
```

**`<exclusions>` — решение потребителя (top-down)**

Потребитель принудительно исключает конкретный артефакт из транзитивного дерева. Применяется как крайняя мера, когда транзитивная зависимость конфликтует с другой версией того же артефакта, содержит уязвимость, или несовместима с окружением, а производитель не пометил её как `optional`.

Пример: библиотека A тянет старую версию `commons-logging`, которая конфликтует с нужной версией:

```xml
<dependency>
  <groupId>com.example</groupId>
  <artifactId>library-a</artifactId>
  <version>2.0</version>
  <exclusions>
    <exclusion>
      <groupId>commons-logging</groupId>
      <artifactId>commons-logging</artifactId>
    </exclusion>
  </exclusions>
</dependency>
```

**Сводное сравнение:**

| Аспект | `<optional>` | `<exclusions>` |
|---|---|---|
| Кто объявляет | Производитель | Потребитель |
| Направление | Bottom-up | Top-down |
| Когда уместно | Фича-ориентированные зависимости | Конфликты, уязвимости, ненужные транзитивные артефакты |
| Эффект по умолчанию | Зависимость включена в classpath производителя, но не транслируется | Зависимость полностью удаляется из поддерева |
| Рекомендация | Предпочтительный подход при проектировании библиотек | Крайняя мера для потребителя |

**Расхождение источников:** документация Maven называет `exclusions` «последним средством» (`last resort`), тогда как в реальных проектах они используются достаточно часто из-за того, что не все авторы библиотек корректно используют `optional`.

### Вопрос 30. Как Maven вычисляет итоговый scope транзитивной зависимости, когда цепочка зависимостей содержит несколько промежуточных scopes? Например, ваш проект зависит от A (scope=compile), A зависит от B (scope=runtime) — какой scope получит B в вашем проекте? Что произойдёт, если A объявит B как `provided`?

**Категория:** Maven/Gradle · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Introduction to the Dependency Mechanism — Apache Maven](https://maven.apache.org/guides/introduction/introduction-to-dependency-mechanism.html)

Maven разрешает транзитивные scope по таблице, где строки — scope прямой зависимости вашего проекта, столбцы — scope транзитивной зависимости этой библиотеки:

| Scope прямой зависимости | compile | provided | runtime | test |
|---|---|---|---|---|
| **compile** | compile | — | runtime | — |
| **provided** | provided | — | provided | — |
| **runtime** | runtime | — | runtime | — |
| **test** | test | — | test | — |

Прочерк (—) означает, что зависимость **не включается** в сборку проекта.

**Конкретные примеры:**
- Ваш проект → A (compile), A → B (runtime): B получает scope `runtime` в вашем проекте.
- Ваш проект → A (compile), A → B (provided): B **исключается** (не попадает в ваш classpath).
- Ваш проект → A (provided), A → B (compile): B получает scope `provided` в вашем проекте.
- Ваш проект → A (provided), A → B (runtime): B получает scope `provided` в вашем проекте.

Scope `provided` и `test` не являются транзитивными — они не транслируются потребителям.

#### Ответ от Claude

**Алгоритм разрешения транзитивных scope**

Maven применяет детерминированную таблицу приоритетов при разрешении scope транзитивных зависимостей. Итоговый scope зависимости B в вашем проекте определяется двумя факторами:
1. Scope, с которым ваш проект объявляет зависимость A.
2. Scope, с которым A объявляет зависимость B.

**Полная матрица разрешения:**

| Scope A (прямая) \ Scope B (транзитивная) | compile | provided | runtime | test |
|---|---|---|---|---|
| **compile** | compile | не включается | runtime | не включается |
| **provided** | provided | не включается | provided | не включается |
| **runtime** | runtime | не включается | runtime | не включается |
| **test** | test | не включается | test | не включается |

**Разбор конкретных случаев из вопроса:**

1. **Ваш проект → A (compile), A → B (runtime):**
   B получает scope `runtime`. Это логично: A нужна при компиляции, B нужна A при выполнении, значит B нужна вашему проекту при выполнении. B попадает в `runtime` classpath, но отсутствует в `compile` classpath вашего проекта.

2. **Ваш проект → A (compile), A → B (provided):**
   B **полностью исключается** из вашего проекта. Логика: A заявила, что B будет предоставлена контейнером/JDK в окружении A — это декларация об окружении библиотеки A, не вашего проекта. Maven не транслирует `provided`-зависимости транзитивно.

**Почему `provided` не транзитивен — важный нюанс:**

`provided` означает «зависимость будет в classpath контейнера». Когда библиотека A объявляет B как `provided`, она говорит: «в моём окружении (например, сервлет-контейнер) B уже есть». Потребитель A может работать в другом окружении, где B отсутствует. Поэтому Maven принимает консервативное решение и не транслирует `provided` транзитивно — это предотвращает случайное попадание «контейнерных» зависимостей в classpath downstream-проектов.

**Аналогично `test` не транзитивен:**

Тестовые зависимости нужны только при тестировании конкретного модуля и не имеют смысла в контексте зависящих от него проектов.

**Практическое следствие:**

Если вашему проекту действительно нужна B, но она оказалась `provided` или `test` у A, необходимо объявить B явно в собственном POM с нужным scope. Maven не добавит её автоматически.

**Приоритет при конфликте версий:**

Если B присутствует в нескольких путях транзитивного дерева с разными версиями, Maven применяет правило «ближайшего определения» (nearest definition): побеждает тот артефакт, который находится ближе к корню дерева зависимостей. При равном расстоянии побеждает тот, кто встречается первым в POM. Scope итоговой зависимости определяется по той же матрице, исходя из пути, по которому была выбрана «победившая» версия.
### Вопрос 31. Как настроить удалённый Build Cache в Gradle для CI/CD-пипплайна? Опишите рекомендуемую стратегию: кто должен иметь права на запись в кэш (push), а кто — только на чтение. Чем `FROM-CACHE` отличается от `UP-TO-DATE` в выводе Gradle?

**Категория:** Maven/Gradle · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Build Cache — Gradle User Manual](https://docs.gradle.org/current/userguide/build_cache.html)

Gradle поддерживает локальный и удалённый Build Cache, которые настраиваются независимо. При включённом удалённом кэше Gradle сначала ищет выходные данные задач в локальном кэше, а если там их нет — обращается к удалённому.

**Рекомендуемая стратегия push/pull:**

CI-серверы наполняют распределённый кэш из чистых сборок, а разработчики и другие CI-агенты лишь читают из него. Gradle записывает результаты задач только в те кэши, у которых `isPush = true`. По умолчанию локальный кэш имеет `push = true`, а удалённый — `push = false`.

Типичная конфигурация в `settings.gradle.kts`:

```kotlin
val isCiServer = System.getenv().containsKey("CI")

buildCache {
    remote<HttpBuildCache> {
        url = uri("https://example.com:8123/cache/")
        isPush = isCiServer  // только CI пишет в кэш
    }
}
```

При такой настройке разработчики получают скомпилированные артефакты из кэша (read-only), не тратя время на сборку.

**FROM-CACHE vs UP-TO-DATE:**

- `UP-TO-DATE` — механизм инкрементальной сборки внутри одного воркспейса. Задача пропускается, потому что её входные данные не изменились с прошлого запуска в той же рабочей директории.
- `FROM-CACHE` — механизм кэша задач (Task Output Caching). Gradle восстановил результаты задачи из кэша (локального или удалённого), минуя выполнение. Используется, например, после `clean`, смены ветки или на другой машине.

Пример вывода:

```
$ ./gradlew --build-cache assemble
:compileJava FROM-CACHE
:processResources
:classes
:jar FROM-CACHE
```

`FROM-CACHE` появляется, когда задача кэшируема и её fingerprint совпал с записью в кэше, даже если локальных выходных данных нет. `UP-TO-DATE` означает, что локальные выходные данные уже существуют и актуальны.

#### Ответ от Claude

**Архитектура удалённого Build Cache**

Gradle Build Cache работает по принципу content-addressable storage: ключ кэша — это хэш всех входных данных задачи (исходный код, компилятор, флаги и т.д.). Если ключ найден в кэше, задача не выполняется, а результаты загружаются.

Gradle предоставляет официальный Docker-образ для одноузлового Build Cache Node (бесплатно). Для production рекомендуется Develocity (ранее Gradle Enterprise) с поддержкой репликации, авторизации и метрик.

**Рекомендуемая стратегия для CI/CD:**

```kotlin
// settings.gradle.kts
val isCiServer = System.getenv().containsKey("CI")

buildCache {
    local {
        isEnabled = true  // всегда включён локально
    }
    remote<HttpBuildCache> {
        url = uri("https://build-cache.example.com/cache/")
        isPush = isCiServer          // только CI пишет
        isEnabled = true             // все читают
        credentials {
            username = System.getenv("CACHE_USER") ?: ""
            password = System.getenv("CACHE_PASSWORD") ?: ""
        }
    }
}
```

Принцип разделения прав:
- **CI (push = true):** только чистые сборки (clean build) на CI-агентах гарантируют корректность кэшируемых результатов. Сборки от разработчиков могут содержать локальные изменения, поэтому им push закрыт.
- **Разработчики (push = false):** читают кэш, ускоряя локальные сборки. Это безопасно, так как невалидные данные не попадают в общий кэш.
- **Pull Request-сборки на CI** могут иметь отдельную политику — часто они тоже только читают, чтобы не загрязнять кэш промежуточными коммитами.

**FROM-CACHE vs UP-TO-DATE — детальное сравнение:**

| Признак | UP-TO-DATE | FROM-CACHE |
|---|---|---|
| Механизм | Инкрементальная сборка | Task Output Caching |
| Условие | Выходные данные уже существуют на диске и актуальны | Gradle восстановил выходные данные из кэша |
| После `clean` | Не появится (данные удалены) | Появится (данные загружены из кэша) |
| После смены ветки | Может не появиться | Появится, если fingerprint совпал |
| Требует `--build-cache` | Нет | Да (или `org.gradle.caching=true`) |
| Работает между машинами | Нет | Да (при удалённом кэше) |

`FROM-CACHE` — более мощный механизм: он работает поверх `UP-TO-DATE` и позволяет избежать пересборки даже в ситуациях, когда инкрементальная сборка невозможна. В выводе сборки метки взаимоисключающие: если задача `FROM-CACHE`, она не будет `UP-TO-DATE`.

### Вопрос 32. Что такое Gradle Configuration Cache? Чем он отличается от Build Cache, и какие ограничения накладывает на написание build-скриптов? Какие типичные нарушения приводят к тому, что Configuration Cache не может быть применён?

**Категория:** Maven/Gradle · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Configuration Cache — Gradle User Manual](https://docs.gradle.org/current/userguide/configuration_cache.html)

Configuration Cache — это механизм, позволяющий Gradle пропустить фазу конфигурации целиком, если ничего, влияющего на конфигурацию сборки (build-скрипты, плагины), не изменилось. Gradle сохраняет результат фазы конфигурации (граф задач) в кэш и переиспользует его при следующих сборках.

**Отличие от Build Cache:**

- **Build Cache** — кэширует выходные данные задач (скомпилированные классы, jar-файлы и т.д.), то есть результаты фазы выполнения.
- **Configuration Cache** — кэширует граф задач, то есть результат фазы конфигурации. Позволяет полностью пропустить выполнение build-скриптов.

**Ограничения на build-скрипты:**

Плагины и build-скрипты не должны:
- Регистрировать слушателей (BuildListener, TaskExecutionListener), которые создаются во время конфигурации и вызываются во время выполнения.
- Напрямую обращаться к состоянию другой задачи — задачи должны быть связаны через входные/выходные данные.
- Читать файлы напрямую через Java/Kotlin/Groovy API во время конфигурации — вместо этого нужно объявлять файлы как потенциальные входные данные конфигурации через Provider API.

**Типичные нарушения:**

1. Использование `project` объекта во время выполнения задачи (вызов `Task.getProject()`).
2. Регистрация `BuildListener` или `TaskExecutionListener` в build-скрипте.
3. Прямое чтение файлов во время конфигурации без Provider API.
4. Доступ к состоянию другой задачи напрямую, а не через зависимости.

#### Ответ от Claude

**Что такое Configuration Cache**

Configuration Cache (стабилен с Gradle 8.x, в Gradle 9 включён по умолчанию) позволяет Gradle сериализовать граф задач после фазы конфигурации и восстанавливать его при последующих запусках, полностью пропуская выполнение всех build.gradle-скриптов и плагинов в фазе конфигурации. Результат — значительное ускорение: на больших проектах время до начала выполнения задач сокращается с секунд до долей секунды.

Включение:
```kotlin
// gradle.properties
org.gradle.configuration-cache=true
```

**Отличие от Build Cache:**

| | Build Cache | Configuration Cache |
|---|---|---|
| Что кэшируется | Выходные данные задач (классы, jar и т.д.) | Граф задач (результат фазы конфигурации) |
| Что пропускается | Выполнение отдельных задач | Вся фаза конфигурации |
| Инвалидация | При изменении входных данных задачи | При изменении build-скриптов, плагинов, свойств |
| Работает между машинами | Да (при удалённом кэше) | Нет (только локально) |
| Уровень оптимизации | Фаза выполнения | Фаза конфигурации |

Обе функции дополняют друг друга и могут использоваться одновременно.

**Ограничения и требования к build-скриптам:**

1. **Запрет на обращение к `Project` во время выполнения задачи.** Задача не должна вызывать `task.project` или `Task.getProject()` в `@TaskAction`. Нарушение: объект `Project` не сериализуем.

   ```kotlin
   // НАРУШЕНИЕ
   abstract class CopyTask : DefaultTask() {
       @TaskAction
       fun copy() {
           project.copy { from("src"); into("dst") }  // project недоступен при выполнении
       }
   }

   // ПРАВИЛЬНО — инжектировать FileSystemOperations
   abstract class CopyTask : DefaultTask() {
       @get:Inject abstract val fs: FileSystemOperations
       @TaskAction
       fun copy() {
           fs.copy { from("src"); into("dst") }
       }
   }
   ```

2. **Запрет на регистрацию BuildListener/TaskExecutionListener.** Эти слушатели создаются в фазе конфигурации, но вызываются в фазе выполнения — их нельзя сериализовать.

   ```kotlin
   // НАРУШЕНИЕ
   gradle.addBuildListener(object : BuildAdapter() {
       override fun buildFinished(result: BuildResult) { ... }
   })

   // ПРАВИЛЬНО — использовать Build Services
   ```

3. **Запрет на прямое чтение файлов в фазе конфигурации.** Прямой вызов `File.readText()` в скрипте — необъявленный ввод, конфигурация не знает, когда его инвалидировать.

   ```kotlin
   // НАРУШЕНИЕ
   val version = file("version.txt").readText()

   // ПРАВИЛЬНО — Provider API отслеживает изменения файла
   val version = providers.fileContents(layout.projectDirectory.file("version.txt")).asText
   ```

4. **Задача не должна читать состояние другой задачи напрямую.** Связь между задачами — только через входные/выходные данные.

   ```kotlin
   // НАРУШЕНИЕ
   val otherTask = tasks.getByName("generateCode") as GenerateTask
   tasks.register("compile") {
       doLast { println(otherTask.outputDir) }  // прямой доступ к состоянию задачи
   }

   // ПРАВИЛЬНО
   val generateTask = tasks.named<GenerateTask>("generateCode")
   tasks.register("compile") {
       val dir = generateTask.flatMap { it.outputDir }
       doLast { println(dir.get()) }
   }
   ```

5. **Захват ссылок на изменяемое состояние окружения.** Обращение к `System.getenv()` или `System.getProperties()` напрямую в теле задачи без Provider API приведёт к некорректной инвалидации кэша.

### Вопрос 33. Как в Gradle объявить `strictly`-версию зависимости и чем она отличается от `force`? Когда следует использовать `resolutionStrategy.failOnVersionConflict()` и каков практический эффект этой настройки в многомодульном проекте?

**Категория:** Maven/Gradle · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Declaring Versions and Ranges — Gradle User Manual](https://docs.gradle.org/current/userguide/dependency_versions.html)

**strictly** — это сильнейшая форма объявления версии в модели богатых версионных ограничений (Rich Version Constraints) Gradle. Она означает: «любая версия, не совпадающая с этим ограничением, должна быть отклонена». Если разрешение зависимостей не может удовлетворить `strictly`-ограничение, сборка завершается ошибкой.

Синтаксис:

```kotlin
// Полная форма (Kotlin DSL)
implementation("org.slf4j:slf4j-api") {
    version {
        strictly("1.7.15")
    }
}

// Сокращённая форма с символом !!
implementation("org.slf4j:slf4j-api:1.7.15!!")

// Strict-диапазон с предпочтительной версией
implementation("org.slf4j:slf4j-api") {
    version {
        strictly("[1.7, 1.8[")
        prefer("1.7.25")
    }
}
```

`strictly` перекрывает версии из транзитивных зависимостей. Если прямая зависимость объявляет `strictly("1.7.15")`, но транзитивная зависимость требует `1.8.x`, разрешение конфликта завершится ошибкой, а не автоматическим повышением версии.

Для авторов библиотек `strictly`-ограничения следует применять осторожно, поскольку они распространяются на потребителей библиотеки и могут вызывать конфликты у них.

#### Ответ от Claude

**Модель богатых версионных ограничений в Gradle**

Gradle поддерживает иерархию способов объявления версий (от мягкого к жёсткому):

| Ограничение | Семантика | Передаётся потребителям |
|---|---|---|
| `prefer` | Предпочтительная версия, но может быть повышена | Нет |
| `require` | Минимальная требуемая версия | Да (как обычная зависимость) |
| `strictly` | Точная версия или диапазон; всё вне него — ошибка | Да (как жёсткое ограничение) |
| `reject` | Явно исключить конкретные версии | Да |

**strictly — объявление и поведение:**

```kotlin
// Kotlin DSL — точная версия
dependencies {
    implementation("com.google.guava:guava") {
        version {
            strictly("31.1-jre")
        }
    }
}

// Groovy DSL — точная версия
dependencies {
    implementation("com.google.guava:guava") {
        version {
            strictly "31.1-jre"
        }
    }
}

// Сокращённая нотация
dependencies {
    implementation("com.google.guava:guava:31.1-jre!!")
}

// Диапазон с предпочтением
dependencies {
    implementation("org.springframework:spring-core") {
        version {
            strictly("[5.3, 6.0[")
            prefer("5.3.27")
        }
    }
}
```

**Отличие strictly от force:**

| Аспект | `strictly` | `force` (resolutionStrategy.force) |
|---|---|---|
| Подход | Декларативный, часть Rich Version model | Императивный, уровень конфигурации |
| Область действия | Конкретная зависимость и её потребители | Вся конфигурация (все зависимости) |
| Понижение версии | Может инициировать downgrade транзитивных зависимостей | Принудительно устанавливает версию |
| Поведение при конфликте | Завершает сборку ошибкой, если ограничение невыполнимо | Молча применяет версию, игнорируя конфликт |
| Публикация в метаданные | Да (через Gradle Module Metadata) | Нет |
| Рекомендован | Да, предпочтительный подход | Нет, считается устаревшим грубым решением |

```kotlin
// force — старый подход (не рекомендован)
configurations.all {
    resolutionStrategy.force("com.google.guava:guava:31.1-jre")
}

// strictly — современный подход
dependencies {
    constraints {
        implementation("com.google.guava:guava:31.1-jre!!")
    }
}
```

Ключевое различие: `force` молча побеждает любой конфликт, `strictly` объявляет намерение и завершает сборку ошибкой, если намерение невыполнимо — это делает поведение предсказуемым и явным.

**resolutionStrategy.failOnVersionConflict():**

По умолчанию Gradle разрешает конфликты версий, выбирая наибольшую из запрошенных — это называется «оптимистичная стратегия». `failOnVersionConflict()` меняет поведение: при любом конфликте (когда хотя бы два модуля запрашивают разные версии одной зависимости) сборка немедленно завершается ошибкой.

```kotlin
// Применить ко всем конфигурациям проекта
configurations.all {
    resolutionStrategy.failOnVersionConflict()
}

// Или только к конкретной конфигурации
configurations.named("runtimeClasspath") {
    resolutionStrategy.failOnVersionConflict()
}
```

**Практический эффект в многомодульном проекте:**

Предположим, проект состоит из трёх модулей:
- `module-a` зависит от `jackson-databind:2.14.0`
- `module-b` зависит от `jackson-databind:2.15.2`
- `module-c` зависит от обоих

Без `failOnVersionConflict()`: Gradle молча выберет `2.15.2` (наибольшую). Это может скрыть несовместимость API.

С `failOnVersionConflict()`: сборка завершится ошибкой:
```
FAILURE: Build failed with an exception.
> Conflict(s) found for the following module(s):
  - com.fasterxml.jackson.core:jackson-databind between versions 2.15.2 and 2.14.0
```

Это вынуждает разработчика явно указать используемую версию через `constraints` или `strictly`, что делает выбор версий задокументированным решением, а не молчаливым поведением инструмента.

**Рекомендации по использованию:**

- Используйте `failOnVersionConflict()` в многомодульных проектах-приложениях (не в библиотеках) для обнаружения скрытых конфликтов на ранней стадии.
- Комбинируйте с `dependencyConstraints` в корневом `build.gradle` для централизованного управления версиями:

```kotlin
// root build.gradle.kts
subprojects {
    configurations.all {
        resolutionStrategy.failOnVersionConflict()
    }
}

dependencies {
    constraints {
        // Единая версия для всего проекта
        implementation("com.fasterxml.jackson.core:jackson-databind:2.15.2!!")
    }
}
```

- Не применяйте `failOnVersionConflict()` в библиотеках, предназначенных для публикации: это нарушит сборку у потребителей при любом конфликте в их зависимостях.
### Вопрос 34. Что такое `buildSrc` в Gradle и для чего он используется? Чем `buildSrc` принципиально отличается от composite build (included build) с точки зрения изоляции, кэширования и масштабируемости? Когда следует предпочесть composite build?

**Категория:** Maven/Gradle · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Best Practices for Structuring Builds — Gradle User Manual](https://docs.gradle.org/current/userguide/best_practices_structuring_builds.html)

`buildSrc` — специальный каталог в корне Gradle-проекта, содержимое которого автоматически компилируется и помещается в classpath всех build-скриптов. Он используется для выноса общей логики сборки: кастомных task-классов, convention-плагинов, вспомогательных утилит — чтобы не дублировать их в каждом `build.gradle`.

**Ключевые различия `buildSrc` и composite build:**

- **Изоляция.** Composite build (included build) трактуется Gradle как полностью независимая сборка, которая не разделяет репозитории, плагины и свойства с основным проектом. `buildSrc` же встроен в classpath основной сборки и имеет более тесную связь с ней, что может приводить к неожиданным поведениям загрузчика классов.

- **Кэширование.** Любое изменение в `buildSrc` полностью инвалидирует build cache — как локальный, так и remote. Даже правка одного класса в `buildSrc` вынуждает Gradle пересобрать все задачи, которые зависят от classpath сборки. В composite build изменения в одном subproject включённого проекта инвалидируют только те задачи основного проекта, которые фактически используют результат этого subproject.

- **Масштабируемость.** Composite builds масштабируются значительно лучше: каждый included build является самостоятельным Gradle-проектом, его можно открывать, разрабатывать и собирать независимо. Это особенно важно для монорепозиториев с сотнями модулей, где инвалидация кэша из-за `buildSrc` становится серьёзной проблемой производительности.

**Когда предпочесть composite build:**
Официальная рекомендация Gradle — использовать composite build (обычно каталог с именем `build-logic`) как основной способ организации и совместного использования логики сборки. `buildSrc` удобен для быстрого старта и прототипирования, но при стабилизации логики сборки и росте проекта следует переходить на composite build.

#### Ответ от Claude

**`buildSrc`** — это специальный каталог в корне Gradle-проекта. Gradle автоматически обнаруживает его, компилирует содержащийся в нём код (Groovy/Kotlin/Java) и добавляет результат в classpath всех `build.gradle`-файлов проекта. Предназначен для хранения переиспользуемой логики сборки: кастомных task-классов, convention-плагинов, DSL-расширений.

**Composite build (included build)** — механизм включения одной полноценной Gradle-сборки в другую через директиву `includeBuild(...)` в `settings.gradle`. Включённая сборка сохраняет полную автономность и может, в свою очередь, содержать собственные subprojects.

---

**Сравнение по трём критериям:**

| Критерий | `buildSrc` | Composite build |
|---|---|---|
| Изоляция | Часть classpath основной сборки; разделяет окружение с ней | Полностью изолированная сборка; не разделяет репозитории, properties, plugins |
| Кэширование | Любое изменение в `buildSrc` инвалидирует весь build cache (локальный и remote) | Инвалидируются только задачи, зависящие от изменившегося subproject included build |
| Масштабируемость | Плохо масштабируется: рост `buildSrc` замедляет CI, невозможно открыть его в IDE как самостоятельный проект | Хорошо масштабируется: каждый included build открывается и разрабатывается независимо |

**Почему `buildSrc` так сильно влияет на кэш:** Gradle рассматривает `buildSrc` как часть самих build-скриптов. При изменении любого файла в `buildSrc` Gradle считает build-скрипты изменившимися и инвалидирует все задачи, которые используют classpath сборки. В composite build включённая сборка является внешней зависимостью — Gradle проверяет реальные inputs/outputs и инвалидирует только то, что фактически зависит от изменившегося артефакта.

**Когда предпочесть composite build:**
- Проект содержит более нескольких convention-плагинов или кастомных задач.
- Используется remote build cache (CI) — инвалидация из `buildSrc` особенно дорога на CI.
- Логику сборки нужно переиспользовать в нескольких независимых репозиториях (included build можно опубликовать в Maven-репозиторий).
- Команда хочет разрабатывать логику сборки в отдельном контексте IDE.

**Когда `buildSrc` ещё оправдан:**
- Небольшой проект, где логика сборки минимальна.
- Быстрое прототипирование: `buildSrc` не требует дополнительной конфигурации `settings.gradle`.

Официальная позиция Gradle (начиная с документации к Gradle 8): preferred location for build logic — included build с именем `build-logic`, а не `buildSrc`.

### Вопрос 35. Как работает инкрементальная компиляция Java в Gradle на основе ABI (Application Binary Interface)? Объясните, почему изменение только тела приватного метода не вызывает перекомпиляцию зависимых модулей, тогда как изменение сигнатуры публичного метода — вызывает.

**Категория:** Maven/Gradle · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [An In-depth Look at Gradle's Approach to Faster Compilation — Gradle Blog](https://blog.gradle.org/our-approach-to-faster-compilation)

ABI (Application Binary Interface) — это всё, что видно потребителям библиотеки на этапе компиляции: публичные классы, публичные методы, публичные поля. ABI **не включает** тела методов, приватные классы, приватные методы, приватные поля.

Gradle отслеживает ABI каждого модуля. Если зависимый проект изменился в ABI-совместимом режиме (изменилась только реализация — например, тело метода, — но не его сигнатура), то задача компиляции downstream-модуля остаётся up-to-date, и перекомпиляция не запускается.

Конкретный пример из блога: реализация метода `NumberUtils` была изменена с императивного цикла `for` на `Stream`-based `reduce`. Публичная сигнатура метода не изменилась — `MyApp`, зависящий от `NumberUtils`, не перекомпилируется. Gradle непосредственно сравнивает ABI с предыдущей версией, не создавая отдельных header-JAR-файлов — это эффективнее и быстрее с точки зрения дискового I/O.

#### Ответ от Claude

**ABI (Application Binary Interface) в контексте Java-компиляции** — контракт, который библиотека предоставляет своим потребителям на уровне скомпилированного байткода. Gradle использует анализ ABI для определения: достаточно ли изменения в одном модуле, чтобы перекомпилировать зависящие от него модули.

**Что входит в ABI:**
- Публичные и protected классы, интерфейсы, аннотации, перечисления.
- Сигнатуры публичных и protected методов (имя, типы параметров, возвращаемый тип, throws-декларации).
- Публичные и protected поля и их типы.
- Типы, которые используются в публичных сигнатурах (transitively).

**Что НЕ входит в ABI:**
- Тела методов (реализация).
- Приватные методы, поля, внутренние классы.
- Комментарии и форматирование.
- Имена параметров методов.

---

**Почему изменение тела приватного метода не вызывает перекомпиляцию:**

Приватный метод недоступен за пределами своего класса — javac никогда не генерирует bytecode, ссылающийся на него, в downstream-модулях. Даже если реализация метода изменилась, скомпилированный байткод downstream-модулей остаётся корректным: вызовов приватного метода в нём нет. ABI модуля не изменился, поэтому Gradle помечает задачу `compileJava` зависимого модуля как up-to-date.

**Почему изменение сигнатуры публичного метода вызывает перекомпиляцию:**

Публичная сигнатура — часть ABI. Если метод `calculate(int a)` переименовывается в `compute(int a)` или его сигнатура изменяется на `calculate(long a)`, то скомпилированный байткод downstream-модуля содержит вызов старого метода (константа в constant pool класса). Этот байткод становится некорректным — JVM не сможет разрешить такой вызов во время выполнения. Gradle обнаруживает ABI-несовместимое изменение и запускает перекомпиляцию всех зависимых модулей.

**Как Gradle это реализует технически:**

Gradle не создаёт отдельные «header JAR»-файлы (в отличие от некоторых других систем). Вместо этого он:
1. После компиляции модуля вычисляет «ABI-снимок» — хеш всех публичных сигнатур.
2. Перед следующей сборкой сравнивает текущий ABI-снимок с предыдущим.
3. Если ABI не изменился — downstream-задача компиляции помечается up-to-date без её запуска.
4. Если ABI изменился — запускается перекомпиляция только тех downstream-модулей, которые объявили зависимость на данный модуль через `api` или `implementation`.

**Разница между `api` и `implementation` в контексте ABI:**

При использовании `java-library`-плагина объявление зависимости через `implementation` означает, что она не «протекает» в ABI библиотеки. Если модуль A использует `implementation(B)`, то потребители A (модуль C) не получают B в compile classpath. Изменение ABI модуля B не вызывает перекомпиляцию C. При объявлении через `api` зависимость становится частью ABI и изменения в ней распространяются транзитивно.

**Практический эффект:** В крупных многомодульных проектах (50+ модулей) ABI-based compile avoidance кардинально сокращает время сборки: рефакторинг внутренней реализации (который происходит значительно чаще, чем изменение публичных сигнатур) не вызывает каскадной перекомпиляции.

### Вопрос 36. Как в Maven организовать многомодульный проект так, чтобы версии всех сторонних зависимостей управлялись из одного места, но при этом не нарушалась возможность выборочной сборки отдельных модулей командой `mvn install -pl module-name`? Какую роль в этом играет `<dependencyManagement>` в родительском POM и публикуемый BOM-артефакт?

**Категория:** Maven/Gradle · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Introduction to the Dependency Mechanism — Apache Maven](https://maven.apache.org/guides/introduction/introduction-to-dependency-mechanism.html)

`<dependencyManagement>` в родительском POM позволяет централизованно задавать версии зависимостей. Дочерние модули объявляют зависимости без указания версии — Maven разрешает её из `<dependencyManagement>` секции родителя. Это не добавляет зависимости автоматически: модуль получает зависимость только если явно её объявляет в `<dependencies>`.

BOM (Bill of Materials) — это POM-файл с `<packaging>pom</packaging>`, содержащий только секцию `<dependencyManagement>`. Другие проекты импортируют его через `<scope>import</scope>` в своей секции `<dependencyManagement>`. BOM решает ограничение единственного родителя: проект может импортировать несколько BOM-файлов, тогда как наследовать может только от одного родителя.

При конкуренции версий из нескольких импортированных BOM побеждает тот, который объявлен первым в `<dependencyManagement>`.

BOM-файл выглядит следующим образом:

```xml
<project>
  <groupId>com.example</groupId>
  <artifactId>my-bom</artifactId>
  <version>1.0.0</version>
  <packaging>pom</packaging>

  <dependencyManagement>
    <dependencies>
      <dependency>
        <groupId>org.springframework</groupId>
        <artifactId>spring-core</artifactId>
        <version>6.1.0</version>
      </dependency>
    </dependencies>
  </dependencyManagement>
</project>
```

Импорт в другом проекте:

```xml
<dependencyManagement>
  <dependencies>
    <dependency>
      <groupId>com.example</groupId>
      <artifactId>my-bom</artifactId>
      <version>1.0.0</version>
      <type>pom</type>
      <scope>import</scope>
    </dependency>
  </dependencies>
</dependencyManagement>
```

#### Ответ от Claude

**Архитектура централизованного управления версиями в Maven:**

В многомодульном Maven-проекте версии зависимостей управляются через `<dependencyManagement>` — либо в родительском POM, либо через отдельный BOM-артефакт (или их комбинацию).

---

**Роль `<dependencyManagement>` в родительском POM:**

`<dependencyManagement>` — это секция, которая определяет версии зависимостей (и другие атрибуты: scope, exclusions) для всего дерева модулей, наследующих от данного POM. Ключевое свойство: секция не добавляет зависимости в classpath — она лишь устанавливает версию «по умолчанию», которую модуль получит, когда сам объявит эту зависимость без `<version>`.

```xml
<!-- parent/pom.xml -->
<dependencyManagement>
  <dependencies>
    <dependency>
      <groupId>org.slf4j</groupId>
      <artifactId>slf4j-api</artifactId>
      <version>2.0.9</version>
    </dependency>
  </dependencies>
</dependencyManagement>
```

```xml
<!-- module-a/pom.xml -->
<parent>
  <groupId>com.example</groupId>
  <artifactId>parent</artifactId>
  <version>1.0.0</version>
</parent>
<dependencies>
  <dependency>
    <groupId>org.slf4j</groupId>
    <artifactId>slf4j-api</artifactId>
    <!-- версия не указана — берётся из dependencyManagement родителя -->
  </dependency>
</dependencies>
```

**Почему `mvn install -pl module-name` не ломается:**

Флаг `-pl` (project list) указывает Maven собрать только перечисленные модули. При выборочной сборке Maven читает POM-файл указанного модуля и его родителя (по цепочке `<parent>`). Поскольку `<dependencyManagement>` содержит только декларации версий (не фактические зависимости), Maven может разрешить версии из родительского POM без необходимости предварительной сборки братских модулей. Реактор просто ищет POM родителя в локальном репозитории (или в текущей структуре каталогов) и читает оттуда `<dependencyManagement>`.

Условие корректной работы: родительский POM должен быть установлен в локальный репозиторий (`~/.m2`) до запуска `-pl` сборки, или должна использоваться опция `-am` (also-make), которая автоматически включает зависимые модули.

---

**Роль публикуемого BOM-артефакта:**

BOM (Bill of Materials) — это POM-файл с `<packaging>pom</packaging>`, содержащий исключительно `<dependencyManagement>`. Он публикуется в Maven-репозиторий как самостоятельный артефакт и импортируется в `<dependencyManagement>` через `<scope>import</scope>`.

**Отличие BOM от dependencyManagement в родительском POM:**

| Характеристика | `dependencyManagement` в parent | BOM |
|---|---|---|
| Механизм передачи | Наследование (единственный parent) | Импорт (можно импортировать несколько) |
| Доступность вне проекта | Только внутри иерархии | Публикуется и используется внешними проектами |
| Ограничения | Конкурирует с наследованием Spring Boot parent и др. | Не конкурирует: импортируется через dependencyManagement |

**Типичная структура комбинированного подхода:**

```
my-project/
  pom.xml              (родительский POM — агрегатор и parent)
  bom/
    pom.xml            (BOM-артефакт: packaging=pom, только dependencyManagement)
  module-api/
    pom.xml
  module-service/
    pom.xml
  module-web/
    pom.xml
```

Родительский POM импортирует собственный BOM:

```xml
<!-- parent/pom.xml -->
<dependencyManagement>
  <dependencies>
    <dependency>
      <groupId>com.example</groupId>
      <artifactId>my-project-bom</artifactId>
      <version>${project.version}</version>
      <type>pom</type>
      <scope>import</scope>
    </dependency>
  </dependencies>
</dependencyManagement>
```

Внешние потребители (другие команды, другие проекты) импортируют только BOM — им не нужен весь родительский POM.

**Важные нюансы при выборочной сборке:**

1. Если модуль зависит от другого модуля того же проекта, `-pl module-name` без `-am` завершится ошибкой разрешения зависимостей — братский модуль должен быть установлен в `~/.m2` заранее.
2. `<dependencyManagement>` не влияет на порядок сборки в реакторе — реактор упорядочивает модули только по фактическим `<dependency>` между ними.
3. Версия, явно объявленная в дочернем POM, всегда имеет приоритет над `<dependencyManagement>` родителя.
4. При конфликте версий между несколькими импортированными BOM побеждает первый объявленный (first-declared-wins).
### Вопрос 37. В Gradle многомодульном проекте как работают `subprojects {}` и `allprojects {}` блоки в корневом `build.gradle`? В чём их недостатки по сравнению с convention plugins, и почему Gradle рекомендует отказаться от `subprojects/allprojects` в современных проектах?

**Категория:** Maven/Gradle · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Sharing Build Logic using buildSrc — Gradle User Manual](https://docs.gradle.org/current/userguide/sharing_build_logic_between_subprojects.html)

`subprojects {}` и `allprojects {}` — это DSL-конструкции корневого `build.gradle`, позволяющие «инжектировать» конфигурацию сразу во все подпроекты (или во все включая корень) без явного повторения в каждом модуле.

**Как они работают.** Gradle обходит дерево проектов и применяет переданный блок-замыкание к каждому подходящему проекту в момент конфигурации. Это называется *cross-project configuration* — конфигурация одного проекта производится из контекста другого.

**Недостатки, которые называет официальная документация:**

- **Непрозрачность конфигурации.** «С cross-project configuration логика может быть инжектирована в подпроект незаметно — это не очевидно при взгляде на его build-скрипт.» Разработчик модуля видит пустой (или почти пустой) `build.gradle`, но на самом деле на него действуют десятки правил из корня.
- **Рост сложности со временем.** «В долгосрочной перспективе cross-project configuration обычно растёт в сложности и становится обузой.»
- **Конфигурационная связанность проектов.** Блоки вводят зависимость между временем конфигурации проектов, что препятствует оптимизациям вроде *configuration-on-demand*.
- **Хрупкость структуры.** «Незначительные структурные изменения (добавление нового проекта, переименование include) могут неожиданно изменить порядок применения плагинов.»
- **Применение к пустым проектам.** Если в дереве есть «агрегирующие» модули без кода, `allprojects {}` всё равно применит к ним всё — замедляя конфигурацию и засоряя отчёты.

**Как работают convention plugins.** Вместо инжекции из родителя каждый подпроект явно применяет нужный плагин (`id("mycompany.java-conventions")`). Логика живёт в `buildSrc/src/main/kotlin/` или во включённой сборке `build-logic/`. Gradle рекомендует предпочитать `build-logic/` как более масштабируемый вариант. Подпроект «видит» всю свою конфигурацию, потому что она применяется явно — никаких скрытых зависимостей от структуры каталогов.

#### Ответ от Claude

**Как работают `subprojects {}` и `allprojects {}`.**

Оба блока размещаются в корневом `build.gradle` (или `build.gradle.kts`) и реализуют паттерн *cross-project configuration*:

- `allprojects {}` — применяет замыкание ко всем проектам сборки, включая сам корневой.
- `subprojects {}` — применяет только к дочерним подпроектам, исключая корень.

Внутри замыкания `this` — это объект `Project` каждого обходимого проекта. Можно применять плагины, добавлять зависимости, настраивать задачи:

```groovy
// корневой build.gradle (Groovy DSL)
subprojects {
    apply plugin: 'java'
    repositories {
        mavenCentral()
    }
    dependencies {
        testImplementation 'org.junit.jupiter:junit-jupiter:5.10.0'
    }
}
```

Этот код применится ко всем подмодулям в момент конфигурационной фазы сборки.

**Недостатки (по официальной документации Gradle).**

1. **Непрозрачность.** Смотря на `build.gradle` подпроекта, разработчик не видит источника применённых плагинов и зависимостей — они приходят «снаружи» из корня. Это нарушает принцип локальности конфигурации.

2. **Хрупкость структуры.** Документация прямо предупреждает: «Не полагайтесь на блоки вроде `allprojects {}`, `subprojects {}` или `afterEvaluate {}`, которые сильно зависят от структуры проекта и расположения файлов. Незначительные структурные изменения могут неожиданно изменить порядок применения плагинов.»

3. **Блокировка оптимизаций.** Cross-project configuration вводит временну́ю зависимость между конфигурацией проектов. Это несовместимо с *configuration-on-demand* — механизмом, при котором Gradle конфигурирует только те проекты, которые нужны для конкретной задачи.

4. **Применение к лишним проектам.** `allprojects {}` затрагивает даже «пустые» агрегирующие модули, замедляя конфигурацию и загромождая отчёты.

5. **Рост необслуживаемости.** По мере роста проекта условия внутри `subprojects {}` (проверки имён, типов) превращаются в запутанную логику.

**Convention plugins — рекомендованный подход.**

Convention plugin — это предкомпилированный скрипт-плагин, фиксирующий соглашения организации. Он явно применяется только там, где нужен:

```kotlin
// build-logic/src/main/kotlin/mycompany.java-conventions.gradle.kts
plugins {
    java
    checkstyle
}

java {
    toolchain.languageVersion.set(JavaLanguageVersion.of(21))
}

checkstyle {
    toolVersion = "10.12.4"
}
```

```kotlin
// модуль my-service/build.gradle.kts
plugins {
    id("mycompany.java-conventions")   // явно и локально
}
```

Преимущества:

- **Прозрачность**: конфигурация модуля видна из его же `build.gradle`.
- **Компонуемость**: один convention plugin может включать другие.
- **Совместимость с configuration-on-demand и isolated projects**.
- **Тестируемость**: плагины можно покрыть тестами через `TestKit`.

**Где размещать convention plugins.**

Gradle рекомендует включённую сборку `build-logic/` (не `buildSrc/`): изменения в `buildSrc/` инвалидируют весь кеш сборки, тогда как `build-logic/` как включённая сборка инвалидирует кеш более избирательно.

**Вывод.** `subprojects {}` и `allprojects {}` — удобный, но архаичный приём. Gradle явно причисляет их к антипаттернам в разделе *Best Practices* и рекомендует заменять convention plugins, размещёнными в `build-logic/`.

### Вопрос 38. Что такое dependency constraints в Gradle и как они соотносятся с `dependencyManagement` в Maven? Чем constraint отличается от прямого объявления зависимости, и как constraint работает при публикации артефакта через `maven-publish`?

**Категория:** Maven/Gradle · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Declaring Dependency Constraints — Gradle User Manual](https://docs.gradle.org/current/userguide/dependency_constraints.html)

**Что такое dependency constraint.**

«Constraint по зависимости задаёт версионные требования к модулю, не добавляя этот модуль в граф зависимостей как зависимость.» Иными словами, constraint влияет на выбор версии (в том числе транзитивной зависимости), но сам модуль не попадает в classpath, если на него нет прямой зависимости.

**Отличие от прямого объявления зависимости.**

«Объявления версий (включая диапазоны) влияют только на те зависимости, которые объявлены напрямую. Чтобы контролировать версии транзитивных зависимостей, используйте dependency constraints.»

Пример:

```kotlin
dependencies {
    implementation("org.apache.httpcomponents:httpclient:4.5.3")

    constraints {
        implementation("commons-codec:commons-codec:1.11") {
            because("версия, транзитивно подтягиваемая httpclient, содержит CVE")
        }
    }
}
```

`commons-codec` окажется в classpath (как транзитивная зависимость `httpclient`), но именно с версией `1.11`, а не с той, что прописана в POM httpclient.

**Публикация через `maven-publish`.**

«Dependency constraints публикуются только при использовании Gradle Module Metadata. Это означает, что в полной мере они поддерживаются, только если и публикация, и потребление выполняются через Gradle. Если модули потребляются через Maven или Ivy, constraints могут не сохраниться.»

Для совместимости с Maven-экосистемой рекомендуется использовать java-platform plugin, который позволяет публиковать platform как Maven BOM (POM с `<dependencyManagement>`).

#### Ответ от Claude

**Что такое dependency constraints.**

Dependency constraint — механизм Gradle для задания версионных требований к зависимости (включая транзитивную), не превращая её в прямую зависимость. Объявляется внутри блока `constraints {}`:

```kotlin
dependencies {
    constraints {
        implementation("com.google.guava:guava:32.1.2-jre") {
            because("фиксируем минимальную версию для безопасности")
        }
    }
}
```

Если `guava` нигде не объявлена как прямая зависимость, в classpath она не попадёт. Но если она является транзитивной зависимостью другого артефакта, Gradle будет использовать не меньше указанной версии.

**Чем constraint отличается от прямой зависимости.**

| Аспект | Прямая зависимость | Constraint |
|---|---|---|
| Попадает в classpath | Всегда | Только если есть прямая или транзитивная зависимость |
| Влияет на транзитивные зависимости | Нет (только на свою версию) | Да, на все пути разрешения |
| Синтаксис | `implementation("a:b:1.0")` | `constraints { implementation("a:b:1.0") }` |
| Аналог в Maven | `<dependency>` | `<dependencyManagement>` |

**Соотношение с Maven `dependencyManagement`.**

В Maven `<dependencyManagement>` задаёт «рекомендованные» версии: они применяются к подпроектам-наследникам и к BOM-импортам, но модуль при этом не добавляется в scope. Gradle dependency constraint — прямой функциональный эквивалент для единственного модуля.

Для multi-project-сборок Gradle предлагает `java-platform` plugin — точный аналог Maven BOM. Платформа публикует только `<dependencyManagement>` без кода:

```kotlin
// platform/build.gradle.kts
plugins { `java-platform` }

dependencies {
    constraints {
        api("org.apache.commons:commons-lang3:3.12.0")
        api("com.fasterxml.jackson.core:jackson-databind:2.15.2")
    }
}
```

Подпроект подключает платформу:

```kotlin
dependencies {
    implementation(platform(project(":platform")))
    implementation("com.fasterxml.jackson.core:jackson-databind") // без версии
}
```

Gradle также умеет потреблять внешние Maven BOM:

```kotlin
dependencies {
    implementation(platform("org.springframework.boot:spring-boot-dependencies:3.2.0"))
    implementation("org.springframework.boot:spring-boot-starter-web") // без версии
}
```

**Как constraints работают при публикации через `maven-publish`.**

Здесь есть принципиальное ограничение, задокументированное Gradle:

1. **Gradle Module Metadata (`.module` файл)** — constraints публикуются полностью и корректно воспроизводятся при потреблении через Gradle.

2. **Maven POM (`.pom` файл)** — стандартные dependency constraints *не* трансформируются в `<dependencyManagement>`. Gradle публикует в POM только разрешённые версии прямых зависимостей (через `versionMapping`), но не constraints на транзитивные. Это означает потерю информации для Maven-клиентов.

3. **Исключение — `java-platform` plugin.** Платформа, опубликованная через `maven-publish`, генерирует POM типа `pom` (BOM) с секцией `<dependencyManagement>`. Это единственный способ корректно опубликовать Gradle-constraints как Maven-совместимые версионные рекомендации.

4. **Известное ограничение (GitHub issue #9866):** транзитивные platform-зависимости (когда `:client` → `:core` → `platform`) не всегда корректно отражаются в опубликованном POM `:client`. Issue закрыт как «not planned».

**Практический итог.**

- Для управления версиями внутри Gradle-сборки: используйте `constraints {}` или `java-platform`.
- Для публикации version catalog/BOM, совместимого с Maven: используйте `java-platform` plugin — он генерирует корректный Maven BOM.
- Чистые constraints (не через платформу) при публикации через `maven-publish` теряются для Maven-клиентов: в POM они не попадают. Потребители Maven получат лишь то, что Gradle разрешил на момент публикации.
- Spring предоставляет собственный `dependency-management-plugin` как альтернативу, добавляя поддержку Maven-семантики исключений и автоматическую запись в `<dependencyManagement>` при генерации POM — но это сторонний плагин, не часть Gradle.

### Вопрос 39. Что такое Gradle Version Catalog и файл `libs.versions.toml`? Опишите структуру файла (секции `[versions]`, `[libraries]`, `[bundles]`, `[plugins]`), объясните, как работают type-safe-акцессоры, и назовите преимущества перед «жёсткими» строками зависимостей.

**Категория:** Maven/Gradle · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Version Catalogs — Gradle User Manual](https://docs.gradle.org/current/userguide/version_catalogs.html)

Version Catalog — централизованный список зависимостей, упрощающий управление ими в Gradle-сборках. Вместо жёстких строк с координатами по всему проекту разработчик ссылается на заранее объявленные элементы каталога.

**Структура файла `gradle/libs.versions.toml`:**

- **`[versions]`** — объявляет переиспользуемые псевдонимы версий:
  ```toml
  [versions]
  groovy = "3.0.5"
  junit5 = "5.10.0"
  ```

- **`[libraries]`** — отображает псевдонимы на GAV-координаты (group:artifact:version):
  ```toml
  [libraries]
  groovy-core = { module = "org.codehaus.groovy:groovy", version.ref = "groovy" }
  junit-jupiter = { module = "org.junit.jupiter:junit-jupiter", version.ref = "junit5" }
  ```

- **`[bundles]`** — группирует связанные библиотеки для ссылки одним алиасом:
  ```toml
  [bundles]
  groovy = ["groovy-core", "groovy-json", "groovy-nio"]
  ```

- **`[plugins]`** — объявляет идентификаторы и версии плагинов:
  ```toml
  [plugins]
  versions = { id = "com.github.ben-manes.versions", version = "0.45.0" }
  ```

**Type-safe-акцессоры:** Gradle автоматически генерирует типобезопасные акцессоры из псевдонимов. Дефисы в именах превращаются в точечную нотацию, что обеспечивает IDE-автодополнение:

| Псевдоним | Акцессор |
|-----------|----------|
| `groovy-core` | `libs.groovy.core` |
| `junit-jupiter` | `libs.junit.jupiter` |

Использование в `build.gradle.kts`:
```kotlin
dependencies {
    implementation(libs.groovy.core)
    implementation(libs.bundles.groovy)
    testImplementation(libs.junit.jupiter)
}
plugins {
    alias(libs.plugins.versions)
}
```

**Ключевые преимущества:**
- Централизованное управление: все координаты зависимостей в одном месте, видны каждому подпроекту.
- Типобезопасность: IDE-автодополнение и обнаружение ошибок при компиляции.
- Согласованность версий: единый источник истины для общих версий.
- Меньше дублирования: версии объявляются один раз и ссылаются везде через псевдоним.

Каталоги объявляют *запрошенные* версии, но не форсируют их: при разрешении конфликтов Gradle может выбрать другую версию в соответствии со своей стратегией.

#### Ответ от Claude

**Что такое Version Catalog**

Version Catalog (стабилен с Gradle 7.4) — механизм централизованного объявления зависимостей. Каталог хранится в файле `gradle/libs.versions.toml` (конвенционное расположение) и автоматически компилируется в объект `libs` с type-safe-акцессорами, доступный во всех `build.gradle(.kts)`-файлах проекта и его подмодулей.

Без каталога каждый модуль дублирует строки с координатами и версиями. При обновлении версии приходится менять все файлы — с риском пропустить один. Каталог решает эту проблему.

**Структура `gradle/libs.versions.toml`:**

```toml
[versions]
jackson = "2.17.1"
spring-boot = "3.3.0"
junit5 = "5.10.2"

[libraries]
jackson-databind  = { module = "com.fasterxml.jackson.core:jackson-databind",  version.ref = "jackson" }
jackson-jsr310    = { module = "com.fasterxml.jackson.datatype:jackson-datatype-jsr310", version.ref = "jackson" }
spring-web        = { module = "org.springframework.boot:spring-boot-starter-web", version.ref = "spring-boot" }
junit-jupiter     = { module = "org.junit.jupiter:junit-jupiter", version.ref = "junit5" }

[bundles]
jackson = ["jackson-databind", "jackson-jsr310"]

[plugins]
spring-boot-plugin = { id = "org.springframework.boot", version.ref = "spring-boot" }
```

**Как работают type-safe-акцессоры**

Gradle генерирует Java/Kotlin-код акцессоров при синхронизации проекта. Правило: дефис в имени псевдонима → точка в акцессоре.

```kotlin
// build.gradle.kts
dependencies {
    implementation(libs.jackson.databind)           // акцессор к библиотеке
    implementation(libs.bundles.jackson)            // весь бандл за одну строку
    testImplementation(libs.junit.jupiter)
}

plugins {
    alias(libs.plugins.spring.boot.plugin)         // плагин из каталога
}
```

При опечатке (`libs.jakcson.databind`) IDE и компилятор мгновенно сигнализируют об ошибке — в отличие от строки `"com.fasterxml.jackson.core:jakcson-databind:2.17.1"`, где ошибка обнаружится только при разрешении зависимостей.

**Когда необходим каталог, а когда — избыточен**

Каталог выигрывает в многомодульных проектах: все подмодули пользуются одним объектом `libs`, версия меняется в одном месте. В однофайловых микропроектах (одноразовые скрипты, прототипы) каталог — лишний файл.

**Совместное использование каталога между несколькими сборками**

Каталог можно опубликовать как самостоятельный артефакт типа `version-catalog` через плагин `version-catalog` + `maven-publish`. Другие сборки импортируют его:

```kotlin
// settings.gradle.kts
dependencyResolutionManagement {
    versionCatalogs {
        create("libs") {
            from("com.example:my-catalog:1.0.0")
        }
    }
}
```

**Ограничения**

- Version Catalog объявляет *предпочтительные* версии, но не форсирует их: конфликт может быть разрешён в пользу другой версии. Для принудительной фиксации — `strictly` или `constraints`.
- Каталог не поддерживает динамические версии (`1.+`, `latest.release`) — только статические версии или диапазоны. Это намеренное ограничение для воспроизводимости сборок.
- Изменение `libs.versions.toml` инвалидирует Configuration Cache (так как каталог — часть конфигурации сборки).

### Вопрос 40. Что такое Maven Wrapper (`mvnw`) и Gradle Wrapper (`gradlew`)? Какие файлы они создают, как их генерировать и обновлять, и почему их рекомендуется коммитить в репозиторий?

**Категория:** Maven/Gradle · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Maven Wrapper — Apache Maven](https://maven.apache.org/wrapper/)

Maven Wrapper — инструмент, гарантирующий, что все пользователи проекта используют одну и ту же версию Maven без необходимости предустанавливать её. Вместо глобально установленной команды `mvn` пользователь запускает скрипт-обёртку:

```bash
./mvnw clean install     # Linux/macOS
mvnw.cmd clean install   # Windows
```

При первом запуске обёртка автоматически скачивает указанную версию Maven, кеширует её и передаёт ей управление.

**Файлы, создаваемые Maven Wrapper:**

```
.mvn/wrapper/
├── maven-wrapper.jar          # Bootstrap JAR — скачивает и запускает Maven
└── maven-wrapper.properties   # Конфигурация: URL дистрибутива, контрольная сумма
mvnw                           # Shell-скрипт для POSIX-систем
mvnw.cmd                       # Batch-скрипт для Windows
```

Ключевой файл `.mvn/wrapper/maven-wrapper.properties`:

```properties
distributionUrl=https://repo.maven.apache.org/maven2/org/apache/maven/apache-maven/3.9.6/apache-maven-3.9.6-bin.zip
wrapperSha256Sum=<sha256-хеш>
```

**Генерация обёртки:**
```bash
mvn wrapper:wrapper                        # версия из текущей установки Maven
mvn wrapper:wrapper -Dmaven=3.9.6          # конкретная версия
```

**Опции типа дистрибутива** (`-Dtype`):
- `only-script` (по умолчанию): скрипт скачивает Maven через curl/wget — самый лёгкий вариант.
- `bin`: включает `maven-wrapper.jar` (без внешних утилит для скачивания).
- `source`: включает `MavenWrapperDownloader.java`.

#### Ответ от Claude

**Концепция Wrapper**

Maven Wrapper и Gradle Wrapper — это скрипты-обёртки, фиксирующие конкретную версию инструмента сборки прямо в репозитории проекта. Разработчику или CI-агенту не нужно предустанавливать Maven/Gradle: скрипт сам скачает нужную версию при первом запуске. Это решает проблему «работает на моей машине» — все участники проекта и все CI-серверы используют одну и ту же версию сборщика.

---

**Maven Wrapper (`mvnw`)**

*Создаваемые файлы:*

| Файл | Назначение |
|------|-----------|
| `mvnw` | Shell-скрипт для Linux/macOS |
| `mvnw.cmd` | Batch-скрипт для Windows |
| `.mvn/wrapper/maven-wrapper.jar` | Bootstrap JAR, инициирует скачивание |
| `.mvn/wrapper/maven-wrapper.properties` | URL дистрибутива Maven и SHA-256 контрольная сумма |

*Генерация и обновление:*
```bash
# Создать обёртку для текущей версии Maven
mvn wrapper:wrapper

# Создать обёртку для конкретной версии
mvn wrapper:wrapper -Dmaven=3.9.9

# Вручную отредактировать .mvn/wrapper/maven-wrapper.properties:
distributionUrl=https://repo.maven.apache.org/maven2/org/apache/maven/apache-maven/3.9.9/apache-maven-3.9.9-bin.zip
```

*Использование:*
```bash
./mvnw clean install            # Linux/macOS
./mvnw.cmd clean install        # Windows (или: mvnw.cmd clean install)
```

---

**Gradle Wrapper (`gradlew`)**

> Источник: [The Gradle Wrapper — Gradle User Manual](https://docs.gradle.org/current/userguide/gradle_wrapper.html)

*Создаваемые файлы:*

| Файл | Назначение |
|------|-----------|
| `gradlew` | Shell-скрипт для Linux/macOS |
| `gradlew.bat` | Batch-скрипт для Windows |
| `gradle/wrapper/gradle-wrapper.jar` | Bootstrap JAR |
| `gradle/wrapper/gradle-wrapper.properties` | URL дистрибутива, контрольная сумма |

*Генерация и обновление:*
```bash
# Создать/обновить обёртку до конкретной версии
gradle wrapper --gradle-version 8.9

# Из уже существующей обёртки (рекомендуется)
./gradlew wrapper --gradle-version 8.9 --distribution-type bin
```

Файл `gradle/wrapper/gradle-wrapper.properties`:
```properties
distributionBase=GRADLE_USER_HOME
distributionPath=wrapper/dists
distributionUrl=https\://services.gradle.org/distributions/gradle-8.9-bin.zip
distributionSha256Sum=<sha256-хеш>
zipStoreBase=GRADLE_USER_HOME
zipStorePath=wrapper/dists
```

*Типы дистрибутива (`distributionType`):*
- **`bin`** (по умолчанию) — только среда выполнения; оптимален для CI (меньший размер).
- **`all`** — включает исходники и документацию Gradle; нужен для корректной работы IDE (IntelliJ IDEA подсвечивает Gradle API).

*Использование:*
```bash
./gradlew build       # Linux/macOS
gradlew.bat build     # Windows
```

---

**Почему Wrapper-файлы рекомендуется коммитить в репозиторий**

1. **Воспроизводимость сборки.** Любой разработчик после клонирования репозитория немедленно может собрать проект без ручной установки инструментов.

2. **Изоляция от глобального окружения.** Разные проекты могут использовать разные версии Maven/Gradle, не конфликтуя между собой.

3. **CI/CD упрощается.** CI-агент не должен управлять версиями инструментов — всё зафиксировано в репозитории. Типичная команда в CI: `./mvnw clean verify` или `./gradlew build`.

4. **Обновления явные и прослеживаемые.** Изменение версии сборщика — это PR с diff в одном файле свойств, проходящий через code review.

5. **Безопасность.** Контрольная сумма SHA-256 в `*.properties` защищает от атак на цепочку поставок: если дистрибутив подменён, обёртка откажется его запускать.

**Расхождение источников:** некоторые руководства рекомендуют не коммитить `gradle-wrapper.jar` / `maven-wrapper.jar` из соображений безопасности (бинарный файл в репозитории). Официальная позиция Gradle — коммитить все файлы обёртки; для проверки подлинности JAR следует использовать контрольную сумму `wrapperSha256Sum`. В Maven аналогично: тип `only-script` позволяет избежать коммита JAR-файла совсем.
