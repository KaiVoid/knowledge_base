### Вопрос 1. Что такое Apache Cassandra и для каких задач она предназначена?

**Категория:** Cassandra · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Apache Cassandra Docs — Overview](https://cassandra.apache.org/doc/latest/cassandra/architecture/overview.html)

Apache Cassandra — это «открытая распределённая NoSQL-база данных», которая «реализует партиционированную wide-column модель хранения с семантикой согласованности в конечном счёте (eventually consistent)». Проектные цели: полная multi-primary репликация, глобальная доступность при низкой задержке, масштабирование на обычном (commodity) оборудовании, линейный рост пропускной способности с добавлением узлов, онлайн-балансировка и рост кластера, запросы по ключу партиции, гибкая схема. Cassandra намеренно отказывается от возможностей, требующих межпартиционной координации: распределённых транзакций, JOIN и внешних ключей.

#### Ответ от Claude

**Apache Cassandra** — распределённая NoSQL-СУБД с **wide-column** моделью хранения, рассчитанная на огромные объёмы данных и высокую нагрузку на запись при отсутствии единой точки отказа.

**Ключевые свойства:**
- **Распределённость без мастера** (masterless, peer-to-peer): все узлы равноправны (см. [вопрос 2](#вопрос-2-какая-архитектура-у-cassandra-почему-её-называют-masterless)).
- **Линейная масштабируемость:** добавили узлы — пропорционально выросла пропускная способность.
- **Высокая доступность** и геораспределённость (репликация между дата-центрами).
- **Настраиваемая согласованность** (tunable consistency) для каждого запроса (см. [вопрос 6](#вопрос-6-что-такое-настраиваемая-согласованность-tunable-consistency-и-как-работают-уровни-согласованности)).
- **Оптимизация под запись:** запись почти никогда не блокируется (LSM-storage, см. [вопрос 8](#вопрос-8-как-устроен-путь-записи-write-path-в-cassandra)).

**Когда применять:** большие объёмы временных рядов, логи/события, ленты, IoT, профили — там, где нужны масштаб, доступность и предсказуемая запись. **Когда не стоит:** сложные ad-hoc запросы, JOIN, транзакции между сущностями, строгая немедленная согласованность (подробнее в [вопросе 14](#вопрос-14-чем-cassandra-отличается-от-реляционной-субд-и-когда-её-выбирать)).

### Вопрос 2. Какая архитектура у Cassandra? Почему её называют masterless?

**Категория:** Cassandra · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Apache Cassandra Docs — Dynamo](https://cassandra.apache.org/doc/latest/cassandra/architecture/dynamo.html)

Cassandra перенимает модель распределённой системы Amazon Dynamo, устраняя единые точки отказа. Каждый узел независимо выполняет три основные функции: координацию запросов, членство в кластере и локальное хранение. Это обеспечивает «всегда доступные чтения и записи с низкой задержкой» в глобальных развёртываниях без необходимости в master-узле, координирующем все операции. Информацию о кластере узлы поддерживают через протокол **gossip**: «каждую секунду каждый узел кластера выбирает случайный другой узел, чтобы обменяться с ним состоянием».

#### Ответ от Claude

Cassandra построена по модели **Amazon Dynamo** — это **peer-to-peer** архитектура без выделенного мастера.

**Что значит masterless:**
- Все узлы **равноправны**: нет узла-координатора, нет master/slave. Любой узел может принять любой запрос и стать **координатором** этого запроса.
- Нет единой точки отказа: падение любого узла не останавливает кластер.
- Узлы выстроены в логическое **кольцо** (ring) по диапазонам токенов (см. [вопрос 3](#вопрос-3-как-cassandra-распределяет-данные-по-узлам-токены-кольцо-и-vnodes)).

**Как узлы «знают» друг о друге — gossip:** децентрализованный протокол; раз в секунду каждый узел обменивается с одним случайным узлом информацией о состоянии (живость, токены, версия схемы). Живость определяется детектором отказов **Phi Accrual**.

**Отличие от типичных распределённых СУБД:** в системах с мастером (например, классический MongoDB replica set, HDFS NameNode) отказ мастера требует выбора нового. В Cassandra такого нет вовсе — это упрощает доступность ценой отказа от сильной согласованности «из коробки».

### Вопрос 3. Как Cassandra распределяет данные по узлам (токены, кольцо и vnodes)?

**Категория:** Cassandra · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Apache Cassandra Docs — Dynamo](https://cassandra.apache.org/doc/latest/cassandra/architecture/dynamo.html)

Вместо наивного хеширования по модулю Cassandra использует **консистентное хеширование**, где «каждый узел отображается на один или несколько токенов на непрерывном хеш-кольце». Хеш ключа партиции попадает в позицию на кольце, и Cassandra идёт по кольцу по часовой стрелке, чтобы найти узлы-реплики. Преимущество: «когда число узлов меняется, консистентное хеширование перемещает лишь малую долю ключей». **Виртуальные узлы (vnodes)** назначают несколько токенов одному физическому узлу: новые узлы принимают примерно равную долю данных от соседей, выводимые узлы равномерно отдают данные, а отказавшие — равномерно распределяют нагрузку.

#### Ответ от Claude

**Партиционирование через консистентное хеширование:**
1. У каждой строки есть **partition key**. **Партиционер** (по умолчанию `Murmur3Partitioner`) хеширует его в **токен** — точку на кольце значений.
2. Кольцо токенов поделено между узлами. Узел, отвечающий за диапазон, в который попал токен, хранит эту партицию (плюс реплики на следующих по кольцу узлах).

**Зачем консистентное хеширование:** при добавлении/удалении узла перемещается лишь малая доля ключей (соседние диапазоны), а не вся база — это и даёт линейную масштабируемость.

**Виртуальные узлы (vnodes):** один физический узел владеет не одним большим диапазоном, а **множеством мелких токенов**. Это:
- выравнивает распределение данных и нагрузки;
- ускоряет ребалансировку при добавлении/выводе узла (данные стекаются равномерно со всех соседей);
- равномерно размазывает нагрузку при отказе узла.

**Связь с моделью данных:** все строки с одним partition key всегда лежат на одних и тех же узлах-репликах. Поэтому выбор ключа партиции критичен (см. [вопрос 5](#вопрос-5-в-чём-разница-между-primary-key-partition-key-и-clustering-key)).

### Вопрос 4. Как работает репликация в Cassandra? Какие есть стратегии и что такое replication factor?

**Категория:** Cassandra · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Apache Cassandra Docs — Dynamo](https://cassandra.apache.org/doc/latest/cassandra/architecture/dynamo.html)

Cassandra реплицирует каждую партицию на несколько физических узлов согласно **replication factor (RF)**. Например, RF=3 означает, что каждую партицию хранят три разных узла. Все изменения несут временные метки; разрешение конфликтов — по принципу **last-write-wins** («последняя запись побеждает»). **SimpleStrategy** использует единый целочисленный RF, игнорируя дата-центры и стойки, — полезна только для тестов. **NetworkTopologyStrategy** (рекомендуется для продакшена) задаёт RF на каждый дата-центр и старается выбирать реплики из разных стоек: «все продакшен-развёртывания должны использовать NetworkTopologyStrategy».

#### Ответ от Claude

**Replication factor (RF)** — сколько копий каждой партиции хранит кластер. RF=3 → три узла держат каждую строку. RF задаётся на уровне **keyspace**.

**Стратегии размещения реплик:**

| Стратегия | Описание | Применение |
|---|---|---|
| **SimpleStrategy** | RF одним числом, без учёта топологии (ДЦ/стойки) | только разработка/тесты, один ДЦ |
| **NetworkTopologyStrategy** | RF задаётся отдельно на каждый дата-центр; реплики выбираются из разных стоек | **продакшен**, мульти-ДЦ |

**Как выбираются реплики:** первая реплика — узел-владелец токена; остальные — следующие по кольцу узлы (для `NetworkTopologyStrategy` — с учётом размещения в разных стойках/ДЦ, чтобы пережить отказ стойки или целого дата-центра).

**Разрешение конфликтов — last-write-wins (LWW):** при расхождении версий побеждает запись с наибольшей временной меткой. Это важное следствие: из-за рассинхронизации часов или одновременных записей возможна потеря «проигравшего» обновления — поэтому в Cassandra избегают шаблонов read-modify-write без LWT (см. [вопрос 15](#вопрос-15-что-такое-lightweight-transactions-lwt-в-cassandra-и-зачем-они-нужны)).

RF сам по себе не определяет согласованность чтения/записи — её задаёт **consistency level** (см. [вопрос 6](#вопрос-6-что-такое-настраиваемая-согласованность-tunable-consistency-и-как-работают-уровни-согласованности)).

### Вопрос 5. В чём разница между primary key, partition key и clustering key?

**Категория:** Cassandra · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Apache Cassandra Docs — Data modeling: Introduction](https://cassandra.apache.org/doc/4.0/cassandra/data_modeling/intro.html)

«Partition key генерируется из первого поля primary key… остальные поля (компоненты) — это clustering keys, которые используются для сортировки данных внутри партиции». Partition key может состоять из нескольких полей, если они сгруппированы как первый компонент primary key, что позволяет гибко распределять данные по узлам кластера.

#### Ответ от Claude

В Cassandra **primary key = partition key + clustering columns**:

```sql
PRIMARY KEY ((country, city), created_at, id)
--           └── partition key ──┘  └ clustering columns ┘
```

| Компонент | Что определяет |
|---|---|
| **Partition key** (1-й компонент PK) | **На каком узле** лежат данные: хешируется в токен. Все строки с одним partition key — в одной партиции на одних репликах. |
| **Clustering key** (остальные компоненты) | **Порядок сортировки** строк **внутри** партиции (ASC/DESC). Позволяет эффективно читать диапазоны. |
| **Composite/compound partition key** | partition key из нескольких колонок (в двойных скобках) — чтобы дробить крупные партиции. |

**Ключевые следствия:**
- **Распределение** управляется только partition key. Плохой выбор → «горячие» партиции (перекос нагрузки) или слишком широкие партиции.
- **Сортировка и диапазонные запросы** возможны только по clustering columns в порядке их объявления.
- Запросить можно эффективно только по полному partition key (плюс префиксу clustering columns). Запрос без partition key требует обхода всего кластера (`ALLOW FILTERING` — антипаттерн).

Если PK состоит из одной колонки — она же и есть partition key (clustering columns отсутствуют).

### Вопрос 6. Что такое настраиваемая согласованность (tunable consistency) и как работают уровни согласованности?

**Категория:** Cassandra · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Apache Cassandra Docs — Dynamo](https://cassandra.apache.org/doc/latest/cassandra/architecture/dynamo.html)

Cassandra реализует компромиссы согласованности **на каждую операцию**. «Операции записи всегда отправляются на все реплики, независимо от уровня согласованности. Уровень согласованности лишь определяет, сколько ответов ждёт координатор». Уровни включают ONE/TWO/THREE, QUORUM (большинство, n/2 + 1), ALL, LOCAL_QUORUM, LOCAL_ONE, ANY. Действует принцип Dynamo: «W + R > RF, где W — уровень согласованности записи, R — чтения, RF — replication factor». При RF=3 и QUORUM на чтение и запись (по 2 ответа) «хотя бы одна реплика гарантированно участвует и в записи, и в чтении… кворумы пересекаются, и запись будет видна чтению».

#### Ответ от Claude

**Tunable consistency** — в Cassandra согласованность не фиксирована, а **задаётся для каждого запроса** через **consistency level (CL)**: компромисс «согласованность ↔ доступность/задержка».

**Важно:** запись всегда уходит на **все** реплики; CL определяет лишь, сколько подтверждений ждёт координатор, прежде чем ответить клиенту.

**Основные уровни:**

| CL | Сколько реплик должны ответить |
|---|---|
| `ONE` / `TWO` / `THREE` | 1 / 2 / 3 реплики |
| `QUORUM` | большинство: `RF/2 + 1` |
| `LOCAL_QUORUM` | большинство **в локальном ДЦ** (без межДЦ-задержки) |
| `EACH_QUORUM` | кворум в **каждом** ДЦ (только запись) |
| `ALL` | все реплики (максимум согласованности, минимум доступности) |
| `ANY` | только запись: достаточно даже hinted handoff |

**Формула сильной согласованности:** если **W + R > RF**, множества реплик чтения и записи **гарантированно пересекаются**, и чтение увидит последнюю запись. Пример: RF=3, запись `QUORUM` (W=2) + чтение `QUORUM` (R=2): 2+2 > 3 → строгая согласованность с переживанием отказа 1 узла.

**Практика мульти-ДЦ:** обычно используют `LOCAL_QUORUM`, чтобы не платить задержку на межДЦ-обмен, сохраняя сильную согласованность внутри ДЦ.

### Вопрос 7. К какому типу систем по теореме CAP относится Cassandra?

**Категория:** Cassandra · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Apache Cassandra Docs — Overview](https://cassandra.apache.org/doc/latest/cassandra/architecture/overview.html)

Cassandra реализует «партиционированную wide-column модель хранения с семантикой согласованности в конечном счёте (eventually consistent)» и нацелена на «глобальную доступность при низкой задержке». Она намеренно отказывается от возможностей, требующих межпартиционной координации (распределённых транзакций, JOIN, внешних ключей), отдавая приоритет доступности и низкой задержке.

#### Ответ от Claude

По теореме CAP (при сетевом разделении система выбирает между Consistency и Availability) Cassandra — это **AP-система**: при разделении сети предпочитает **доступность** и **устойчивость к разделению**, жертвуя строгой согласованностью. Согласованность достигается **в конечном счёте** (eventual consistency).

**Уточнение — CAP не «чёрно-белый»:** благодаря tunable consistency Cassandra можно настроить ближе к согласованности. При `W + R > RF` (например, оба `QUORUM`) чтение видит последнюю запись — это «сильная согласованность» в обычных условиях. Но при сетевом разделении, когда кворум недостижим, такие запросы провалятся (выбор в пользу C), а с `CL=ONE` — продолжат работать, отдавая возможно устаревшие данные (выбор в пользу A).

**Корректнее говорить:** Cassandra по умолчанию AP с eventual consistency, но уровень согласованности настраивается на каждую операцию (PACELC: при разделении — выбор A против C, а в обычном режиме — компромисс между задержкой L и согласованностью C). Механизмы, доводящие реплики до согласованности, — read repair, hinted handoff, anti-entropy repair (см. [вопрос 12](#вопрос-12-как-cassandra-доводит-реплики-до-согласованности-read-repair-hinted-handoff-anti-entropy-repair)).

### Вопрос 8. Как устроен путь записи (write path) в Cassandra?

**Категория:** Cassandra · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Apache Cassandra Docs — Storage Engine](https://cassandra.apache.org/doc/4.1/cassandra/architecture/storage_engine.html)

«Любые данные, записываемые в Cassandra, сначала пишутся в commit log, а затем в memtable. Это обеспечивает долговечность (durability) в случае неожиданного завершения работы». «Memtable — это структуры в памяти, где Cassandra буферизует записи; как правило, на таблицу приходится одна активная memtable». Когда memtable заполняется, она сбрасывается (flush) на диск в виде **SSTable**. SSTable неизменяемы: «по мере сброса SSTable из memtable… Cassandra запускает compaction, объединяющий несколько SSTable в одну».

#### Ответ от Claude

Cassandra использует **LSM-дерево** (Log-Structured Merge-tree), поэтому запись всегда последовательная и быстрая. Путь записи:

1. **Commit log** (на диске, append-only): запись сначала добавляется сюда — это гарантия долговечности. При сбое и рестарте узел проигрывает commit log и восстанавливает несброшенные данные.
2. **Memtable** (в памяти): одновременно запись попадает в memtable — буфер в RAM, по одной активной на таблицу. Внутри отсортирована по ключу.
3. **Flush в SSTable**: когда memtable заполняется (по памяти или по лимиту commit log), она сбрасывается на диск в виде **SSTable** — неизменяемого отсортированного файла. После сброса соответствующие сегменты commit log можно переиспользовать.

**Почему это даёт высокую скорость записи:**
- Нет чтения перед записью (no read-before-write), нет обновления данных «на месте».
- Всё пишется **последовательно** (append) — оптимально для дисков.
- SSTable **неизменяемы** — нет блокировок на обновление файлов.

**Обратная сторона:** одни и те же данные оказываются размазаны по многим SSTable (новые версии, обновления, удаления) → чтение и место на диске страдают, поэтому нужен **compaction** (см. [вопрос 10](#вопрос-10-что-такое-compaction-и-какие-есть-стратегии-уплотнения)).

### Вопрос 9. Как устроен путь чтения (read path) и зачем нужен Bloom filter?

**Категория:** Cassandra · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Apache Cassandra Docs — Storage Engine](https://cassandra.apache.org/doc/4.1/cassandra/architecture/storage_engine.html)

Каждая SSTable состоит из нескольких файлов: **Filter.db** — «Bloom-фильтр ключей партиций в SSTable»; **Index.db** — отображает ключи партиций в позиции в файле данных; **Data.db** — содержит сами строки, упорядоченные по partition key и clustering key. Так как данные одной строки могут быть размазаны по нескольким SSTable, чтение может потребовать обращения к нескольким из них.

#### Ответ от Claude

Поскольку данные одной партиции могут лежать в memtable и в нескольких SSTable, чтение собирает их вместе. Упрощённый путь чтения:

1. **Memtable** — проверяется первой (самые свежие данные).
2. **Row cache / key cache** (если включены) — ускоряют поиск.
3. **Bloom filter** каждой SSTable — вероятностная структура, быстро отвечающая «**этого ключа точно нет** в SSTable» или «возможно, есть». Позволяет **пропускать** SSTable, не открывая их (ложноположительные срабатывания возможны, ложноотрицательных — нет).
4. **Partition index / summary** — находит позицию ключа в файле данных у SSTable-кандидатов.
5. **Слияние версий**: из всех источников берутся фрагменты строки, объединяются по временным меткам (last-write-wins), отброшенные tombstone-ом данные не возвращаются.

**Роль Bloom filter:** без него каждое чтение пришлось бы искать во всех SSTable. Bloom filter резко сокращает число дисковых обращений, отсекая SSTable, в которых ключа заведомо нет.

**Следствие:** чем больше SSTable у партиции (мало compaction) и чем больше tombstone, тем медленнее чтение — отсюда важность уплотнения и контроля удалений ([вопросы 10](#вопрос-10-что-такое-compaction-и-какие-есть-стратегии-уплотнения)–[11](#вопрос-11-что-такое-tombstone-и-зачем-нужен-gc_grace_seconds)).

### Вопрос 10. Что такое compaction и какие есть стратегии уплотнения?

**Категория:** Cassandra · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Apache Cassandra Docs — Compaction overview](https://cassandra.apache.org/doc/stable/cassandra/managing/operating/compaction/overview.html)

Compaction объединяет SSTable, чтобы повысить производительность и освободить место. Поскольку SSTable неизменяемы, обновления создают дублирующиеся версии в нескольких файлах; по мере накопления SSTable чтения замедляются, так как приходится обращаться к большему числу файлов. Compaction консолидирует версии, удаляет устаревшие данные и tombstone — всё это ускоряет чтение и сокращает расход диска. Стратегии: STCS (по умолчанию для не-time-series на HDD), LCS (для read-heavy и частых обновлений/удалений), TWCS (для TTL и временных рядов), UCS (рекомендуется для большинства нагрузок).

#### Ответ от Claude

**Compaction (уплотнение)** — фоновый процесс слияния нескольких SSTable в новые, при котором:
- объединяются версии одной строки (берётся свежая по timestamp);
- физически удаляются перезаписанные данные и истёкшие tombstone;
- сокращается число SSTable → ускоряются чтения и освобождается место.

Без compaction SSTable копились бы бесконечно, чтения деградировали бы, а удалённые данные никогда не освобождали бы место.

**Стратегии уплотнения:**

| Стратегия | Для чего |
|---|---|
| **STCS** (SizeTieredCompactionStrategy) | объединяет SSTable примерно одинакового размера; универсальная, хороша для write-heavy |
| **LCS** (LeveledCompactionStrategy) | уровни фиксированного размера; для **read-heavy** и частых обновлений — гарантирует, что строка лежит в малом числе SSTable |
| **TWCS** (TimeWindowCompactionStrategy) | группирует SSTable по временным окнам; для **временных рядов и TTL-данных** — позволяет целиком дропать истёкшие окна |
| **UCS** (UnifiedCompactionStrategy) | новая универсальная стратегия, рекомендуется для большинства нагрузок; настраивается под поведение остальных |

**Расхождение по «дефолту»:** исторически по умолчанию использовалась STCS; в свежей документации Apache рекомендует **UCS** как универсальный выбор для всех типов нагрузок. Выбор стратегии существенно влияет на производительность чтения/записи и расход диска.

### Вопрос 11. Что такое tombstone и зачем нужен gc_grace_seconds?

**Категория:** Cassandra · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Apache Cassandra Docs — Compaction overview](https://cassandra.apache.org/doc/stable/cassandra/managing/operating/compaction/overview.html)

Tombstone — это маркеры удаления со встроенной датой истечения: вместо физического удаления данных Cassandra вставляет помеченную временем запись-маркер. «Tombstone проходят по пути записи Cassandra и записываются в SSTable на одном или нескольких узлах». Они должны пережить `gc_grace_seconds` (по умолчанию 10 дней), чтобы не возникало «зомби-данных» — удалённых записей, которые «воскресают» после восстановления узла: «если узел остаётся недоступным дольше gc_grace_seconds, его удалённые данные будут восстановлены (repair) обратно на другие узлы и снова появятся в кластере».

#### Ответ от Claude

**Tombstone (надгробие)** — это специальный маркер удаления. В Cassandra нельзя «стереть на месте»: SSTable неизменяемы, а узлы-реплики независимы. Поэтому `DELETE` (а также истечение TTL) пишет tombstone с временной меткой, который при чтении «перекрывает» более старые данные.

**Зачем gc_grace_seconds (по умолчанию 10 дней):** tombstone нельзя удалять сразу. Представьте: реплика была недоступна в момент удаления. Если бы tombstone исчез немедленно после compaction на других узлах, то после возвращения «отставшей» реплики anti-entropy repair размножил бы со неё **старые данные** обратно — удалённая запись **воскресла бы** (зомби-данные). `gc_grace_seconds` — это окно, в течение которого tombstone обязан дожить и быть доставленным на все реплики (через repair / hinted handoff). Только после этого срока compaction вправе его удалить.

**Практические проблемы tombstone:**
- **Деградация чтения:** много tombstone в партиции → чтение перебирает массу «надгробий». Есть пороги предупреждений/ошибок (`tombstone_warn_threshold`, `tombstone_failure_threshold`).
- **Антипаттерн «очередь»:** массовые удаления/частые `DELETE` или запись `null` в широкие партиции порождают лавину tombstone.
- **Важно:** обязательно запускать **repair чаще, чем раз в gc_grace_seconds**, иначе риск воскрешения данных.

### Вопрос 12. Как Cassandra доводит реплики до согласованности (read repair, hinted handoff, anti-entropy repair)?

**Категория:** Cassandra · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Apache Cassandra Docs — Dynamo](https://cassandra.apache.org/doc/latest/cassandra/architecture/dynamo.html)

При недоступности реплики во время записи Cassandra сохраняет **hint** на узле-координаторе и позднее проигрывает его, доставляя изменение, — это повышает долговечность записи даже при временном отказе реплик. Помимо best-effort механизмов (read repair и hints), Cassandra гарантирует согласованность в конечном счёте через **anti-entropy repair** с использованием **деревьев Меркла**: реплики вычисляют иерархические хеш-деревья, сравнивают их и синхронизируют расходящиеся диапазоны. Cassandra расширяет исходную модель Dynamo возможностями sub-range и инкрементального repair.

#### Ответ от Claude

Cassandra использует три взаимодополняющих механизма «лечения» расхождений между репликами:

1. **Hinted handoff (подсказки).** Если во время записи реплика недоступна, координатор сохраняет у себя **hint** — отложенную запись. Когда реплика возвращается, hint проигрывается и доставляет ей пропущенное изменение. Защищает запись от временных отказов (есть окно хранения hints).
2. **Read repair (починка при чтении).** При чтении с CL > ONE координатор сравнивает ответы реплик; если обнаружено расхождение, он отправляет свежие данные отставшим репликам — данные «лечатся» по ходу чтения.
3. **Anti-entropy repair (плановый repair).** Запускается оператором (`nodetool repair`). Реплики строят **деревья Меркла** (иерархические хеши диапазонов), сравнивают их и синхронизируют только расходящиеся диапазоны. Это полная, гарантированная синхронизация (в отличие от двух best-effort механизмов выше).

**Связь с tombstone:** регулярный repair обязателен **чаще, чем gc_grace_seconds**, иначе удалённые данные могут воскреснуть (см. [вопрос 11](#вопрос-11-что-такое-tombstone-и-зачем-нужен-gc_grace_seconds)). Вместе эти механизмы и обеспечивают «согласованность в конечном счёте».

### Вопрос 13. Почему в Cassandra моделирование данных строится «от запросов» и поощряется денормализация?

**Категория:** Cassandra · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Apache Cassandra Docs — Data modeling: Introduction](https://cassandra.apache.org/doc/4.0/cassandra/data_modeling/intro.html)

«В Cassandra моделирование данных управляется запросами (query-driven). Паттерны доступа к данным и запросы приложения определяют структуру и организацию данных». «В отличие от реляционной модели, где запросы используют соединения таблиц (JOIN) для получения данных из нескольких таблиц, в Cassandra JOIN не поддерживаются, поэтому все нужные поля (колонки) должны быть сгруппированы в одной таблице». «Данные дублируются по нескольким таблицам — этот процесс называется денормализацией. Дублирование данных и высокая пропускная способность записи используются для достижения высокой производительности чтения».

#### Ответ от Claude

В реляционных БД сначала проектируют нормализованную схему (сущности и связи), а запросы пишут потом, соединяя таблицы через JOIN. В Cassandra **наоборот**: моделирование **query-driven** — сначала выписывают запросы приложения, а под каждый паттерн доступа создают свою таблицу.

**Почему так:**
- **JOIN не поддерживается** (он требовал бы межузловой координации и убил бы масштабируемость). Все нужные для запроса колонки должны лежать **в одной таблице и одной партиции**.
- Поэтому одни и те же данные **намеренно дублируют** в нескольких таблицах под разные запросы — это **денормализация**.
- Дешёвая и быстрая запись (LSM, см. [вопрос 8](#вопрос-8-как-устроен-путь-записи-write-path-в-cassandra)) делает дублирование приемлемой ценой за быстрые чтения.

**Практические правила:**
- «Одна таблица — один запрос» (часто несколько таблиц на одну сущность).
- Запрос всегда идёт по полному partition key — иначе нужен дорогой обход кластера (`ALLOW FILTERING` — антипаттерн).
- Согласованность дублей при обновлении — забота приложения (часто через batch-записи в рамках одной партиции).

Это фундаментальный сдвиг мышления: в Cassandra **схема следует за запросами**, а не наоборот.

### Вопрос 14. Чем Cassandra отличается от реляционной СУБД и когда её выбирать?

**Категория:** Cassandra · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Apache Cassandra Docs — Overview](https://cassandra.apache.org/doc/latest/cassandra/architecture/overview.html)

Cassandra предоставляет CQL — SQL-подобный язык для работы с данными, организованными в keyspaces, таблицы, партиции, строки и колонки. Среди возможностей — коллекции (sets, maps, lists), пользовательские типы/функции/агрегаты, storage-attached indexing (SAI), однопартиционные lightweight transactions. При этом Cassandra «намеренно избегает возможностей, требующих межпартиционной координации, таких как распределённые транзакции, JOIN и ограничения внешних ключей, отдавая приоритет доступности и низкой задержке».

#### Ответ от Claude

| Аспект | Реляционная СУБД (PostgreSQL/MySQL) | Apache Cassandra |
|---|---|---|
| Модель | Нормализованные таблицы со связями | Wide-column, денормализованные таблицы «под запрос» |
| Масштабирование | В основном вертикальное; шардирование сложно | Горизонтальное, линейное, из коробки |
| Архитектура | Обычно один primary + реплики | Masterless P2P, без единой точки отказа |
| JOIN / транзакции между сущностями | Да, ACID | Нет JOIN; транзакции только однопартиционные (LWT) |
| Согласованность | Сильная (ACID) | Настраиваемая, по умолчанию eventual (AP) |
| Сильная сторона | Сложные запросы, целостность, гибкие ad-hoc выборки | Огромный объём, высокая запись, гео-HA |

**Когда выбирать Cassandra:**
- Очень большие объёмы данных, не помещающиеся на один сервер.
- Высокая и постоянная **нагрузка на запись** (логи, метрики, IoT, события, ленты).
- Нужна **гео-распределённость** и доступность 24/7 без единой точки отказа.
- Запросы заранее известны и идут по ключу.

**Когда НЕ стоит:**
- Нужны сложные ad-hoc запросы, агрегаты, JOIN.
- Требуются транзакции между сущностями и строгая немедленная согласованность.
- Небольшой объём данных (тогда реляционная СУБД проще и мощнее).

Кратко: Cassandra — про **масштаб, запись и доступность**, реляционная СУБД — про **гибкость запросов и целостность**.

### Вопрос 15. Что такое lightweight transactions (LWT) в Cassandra и зачем они нужны?

**Категория:** Cassandra · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Apache Cassandra Docs — Overview](https://cassandra.apache.org/doc/latest/cassandra/architecture/overview.html)

Среди возможностей Cassandra документация называет «однопартиционные lightweight transactions с атомарными операциями» (single-partition lightweight transactions with atomic operations). При этом распределённые (многопартиционные) транзакции Cassandra намеренно не поддерживает, отдавая приоритет доступности и низкой задержке.

#### Ответ от Claude

**Lightweight transactions (LWT)** — это условные операции с семантикой **compare-and-set** в пределах **одной партиции**. Примеры в CQL:

```sql
INSERT INTO users (login, ...) VALUES ('mike', ...) IF NOT EXISTS;
UPDATE accounts SET balance = 100 WHERE id = 1 IF balance = 120;
```

**Зачем они нужны:** обычные записи Cassandra разрешают конфликты по принципу last-write-wins, что не годится для случаев вроде «зарегистрировать пользователя, только если логин ещё не занят» или безопасного read-modify-write. LWT дают **линеаризуемость** для таких операций.

**Как это работает — Paxos:** LWT реализованы через консенсус-протокол **Paxos**. Чтобы согласовать условие между репликами, координатор выполняет дополнительные раунды (prepare/propose, read, commit). Поэтому LWT:
- **значительно медленнее** обычных записей (несколько round-trip между репликами) — это плата за корректность;
- работают **только в пределах одной партиции** (межпартиционных распределённых транзакций нет);
- используют отдельный уровень согласованности `SERIAL` / `LOCAL_SERIAL`.

**Вывод:** LWT — инструмент «точечной» строгой согласованности (уникальность, условные обновления). Использовать их следует экономно: на горячем пути массовые LWT убьют производительность. Для координации сложнее однопартиционной Cassandra не предназначена.
