### Вопрос 1. Что такое ClickHouse и для каких задач он предназначен?

**Категория:** ClickHouse · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [ClickHouse Docs — What is ClickHouse?](https://clickhouse.com/docs/intro)

«ClickHouse® — это высокопроизводительная колоночно-ориентированная (column-oriented) SQL-СУБД для онлайн-аналитической обработки (OLAP)». В колоночных системах «таблицы хранятся как набор колонок, то есть значения каждой колонки записаны последовательно одно за другим». Это позволяет «читать с диска только колонки, нужные для запроса, избегая лишнего ввода-вывода для неиспользуемых данных» — что особенно эффективно для фильтрации и агрегации в аналитике.

#### Ответ от Claude

**ClickHouse** — это колоночная SQL-СУБД с открытым исходным кодом, созданная для **OLAP** (аналитическая обработка): агрегации, отчёты, дашборды по огромным объёмам данных (миллиарды и триллионы строк) с откликом за доли секунды.

**Для чего хорош:**
- Аналитика и агрегации (`SUM`, `COUNT`, `GROUP BY`, оконные функции) по большим таблицам.
- Логи, метрики, события, телеметрия, IoT, time-series, веб-аналитика.
- Поддерживает знакомый **SQL** (с расширениями).

**Ключевые принципы:**
- **Колоночное хранение** — читаются только нужные колонки (см. [вопрос 2](#вопрос-2-чем-колоночное-хранение-отличается-от-строкового-и-почему-это-важно-для-аналитики)).
- **Высокое сжатие** данных одного типа в колонке (см. [вопрос 11](#вопрос-11-как-в-clickhouse-устроено-сжатие-данных)).
- **Векторизованное выполнение** запросов (SIMD, обработка пачками) (см. [вопрос 15](#вопрос-15-что-такое-векторизованное-выполнение-запросов-в-clickhouse)).

**Для чего НЕ предназначен:** частые точечные обновления/удаления отдельных строк, транзакции, OLTP-нагрузка (см. [вопросы 13](#вопрос-13-каковы-слабые-стороны-clickhouse-и-когда-его-не-стоит-использовать)–[14](#вопрос-14-как-в-clickhouse-выполняются-update-и-delete)).

### Вопрос 2. Чем колоночное хранение отличается от строкового и почему это важно для аналитики?

**Категория:** ClickHouse · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [ClickHouse Docs — What is ClickHouse?](https://clickhouse.com/docs/intro)

«В строково-ориентированной базе данных последовательные строки таблицы хранятся одна за другой». Из-за этого даже когда запросу нужны лишь отдельные колонки, система вынуждена грузить с диска целые блоки, содержащие и ненужные колонки. В колоночной системе «таблицы хранятся как набор колонок, значения каждой колонки записаны последовательно», поэтому «с диска читаются только колонки, требуемые для запроса, что устраняет лишний ввод-вывод».

#### Ответ от Claude

**Строковое (row-oriented) хранение** (PostgreSQL, MySQL): значения одной строки лежат рядом. Хорошо для OLTP — взять/обновить всю строку целиком по ключу.

**Колоночное (column-oriented) хранение** (ClickHouse): значения одной **колонки** лежат рядом, последовательно.

**Почему это критично для аналитики (OLAP):**
1. **Меньше I/O.** Аналитический запрос обычно трогает несколько колонок из десятков/сотен. Колоночная БД читает только их, а не всю строку — экономия чтения часто >90% на широких таблицах.
2. **Выше сжатие.** В одной колонке лежат однотипные, похожие значения → сжимаются в разы лучше (см. [вопрос 11](#вопрос-11-как-в-clickhouse-устроено-сжатие-данных)).
3. **Векторизация.** Непрерывный массив значений одной колонки удобно обрабатывать пачками через SIMD (см. [вопрос 15](#вопрос-15-что-такое-векторизованное-выполнение-запросов-в-clickhouse)).

**Обратная сторона:** взять одну строку целиком дорого (собирать из многих колонок), а точечные обновления неэффективны — поэтому для OLTP колоночные БД не подходят.

### Вопрос 3. Почему ClickHouse такой быстрый?

**Категория:** ClickHouse · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ClickHouse Docs — Why is ClickHouse so fast?](https://clickhouse.com/docs/concepts/why-clickhouse-is-so-fast)

Скорость ClickHouse обеспечивается несколькими решениями. Колоночное хранение: «колоночные хранилища особенно хорошо подходят для сжатия, так как значения одного типа и распределения лежат вместе». Векторизованное выполнение: «операторы плана запроса передают промежуточные строки результата пачками (batches), а не по одной» — это улучшает использование кэшей CPU и позволяет применять SIMD-инструкции к нескольким значениям сразу. Запросы распараллеливаются по ядрам CPU (одна «дорожка» на ядро, каждая обрабатывает свой диапазон данных) и горизонтально по узлам кластера.

#### Ответ от Claude

ClickHouse быстр благодаря сочетанию факторов, работающих вместе:

1. **Колоночное хранение** — читаются только нужные колонки, минимум I/O (см. [вопрос 2](#вопрос-2-чем-колоночное-хранение-отличается-от-строкового-и-почему-это-важно-для-аналитики)).
2. **Сжатие** — данные одного типа в колонке жмутся в 5–20 раз (LZ4/ZSTD + специализированные кодеки), что ещё сильнее снижает чтение с диска (см. [вопрос 11](#вопрос-11-как-в-clickhouse-устроено-сжатие-данных)).
3. **Векторизованное выполнение** — обработка не по строке, а пачками по 8192 значения с SIMD-инструкциями; лучше используются кэши CPU (см. [вопрос 15](#вопрос-15-что-такое-векторизованное-выполнение-запросов-в-clickhouse)).
4. **Разреженный первичный индекс** — быстрое отсечение лишних гранул данных без полного скана (см. [вопрос 6](#вопрос-6-как-устроен-первичный-ключ-и-разреженный-sparse-индекс-в-mergetree)).
5. **Параллелизм** — запрос распараллеливается по ядрам CPU и по узлам/шардам кластера.
6. **Низкоуровневые оптимизации** — например, 30+ предкомпилированных вариантов хеш-таблиц, выбираемых под конкретный запрос.

Суть: ClickHouse минимизирует и объём читаемых данных (колонки + сжатие + индекс), и стоимость их обработки (векторизация + параллелизм).

### Вопрос 4. В чём разница между OLAP и OLTP и чем ClickHouse отличается от PostgreSQL/MySQL?

**Категория:** ClickHouse · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [ClickHouse Docs — ALTER TABLE … UPDATE](https://clickhouse.com/docs/sql-reference/statements/alter/update)

В документации прямо подчёркивается отличие от транзакционных СУБД: «Префикс `ALTER TABLE` делает этот синтаксис отличным от большинства других SQL-систем. Он призван обозначить, что, в отличие от похожих запросов в OLTP-базах данных, это тяжёлая (heavy) операция, не предназначенная для частого использования». ClickHouse оптимизирован под аналитические (OLAP) нагрузки, а не транзакционные (OLTP): массовое чтение и периодические пакетные изменения, а не постоянные построчные правки.

#### Ответ от Claude

| Аспект | OLTP (PostgreSQL, MySQL) | OLAP (ClickHouse) |
|---|---|---|
| Назначение | Транзакции: вставка/обновление/чтение отдельных строк | Аналитика: агрегации по миллионам/миллиардам строк |
| Хранение | Строковое | Колоночное |
| Типичный запрос | «дай/измени строку по id» | «`SUM`/`GROUP BY` по диапазону за период» |
| Запись | Частые точечные `INSERT/UPDATE/DELETE` | Пакетные `INSERT`, редкие массовые изменения |
| Транзакции, ACID | Полноценные | Нет полноценных транзакций |
| Обновление строки | Дёшево, на месте | Дорого — мутация переписывает part (см. [вопрос 14](#вопрос-14-как-в-clickhouse-выполняются-update-и-delete)) |

**Кратко:** PostgreSQL/MySQL — для оперативных приложений (заказы, пользователи, корзина). ClickHouse — для аналитики поверх этих данных (отчёты, дашборды, метрики). Часто их используют вместе: OLTP-СУБД как источник, ClickHouse — как аналитическое хранилище. ClickHouse сознательно отказался от частых построчных правок ради скорости аналитики.

### Вопрос 5. Что такое движок MergeTree и как он хранит данные?

**Категория:** ClickHouse · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ClickHouse Docs — MergeTree](https://clickhouse.com/docs/engines/table-engines/mergetree-family/mergetree)

MergeTree организует данные в **части (parts)**, создаваемые при вставке и сливаемые (merged) в фоне. Каждая часть независимо отсортирована по первичному ключу. «Операции вставки создают части таблицы, которые фоновым процессом сливаются с другими частями», что со временем повышает эффективность хранения и запросов. `PARTITION BY` задаёт логическое деление данных (обычно по месяцу); партиции хранятся отдельно и не сливаются между собой.

#### Ответ от Claude

**MergeTree** — основной (и базовый для целого семейства) движок таблиц ClickHouse. Его устройство:

- **Данные хранятся частями (data parts).** Каждый `INSERT` создаёт новую **неизменяемую** часть — отдельный каталог с колоночными файлами, отсортированными по ключу сортировки.
- **Фоновое слияние (merge).** Специальный процесс периодически объединяет мелкие части в более крупные (отсюда название). Это сокращает число частей и ускоряет чтение. Поэтому **вставлять данные надо пачками**, а не по одной строке (иначе лавина мелких частей).
- **Сортировка по ключу.** Внутри части строки отсортированы по `ORDER BY` — это база для разреженного индекса (см. [вопрос 6](#вопрос-6-как-устроен-первичный-ключ-и-разреженный-sparse-индекс-в-mergetree)).
- **Партиционирование** (`PARTITION BY`) — крупное логическое деление (см. [вопрос 8](#вопрос-8-что-такое-партиционирование-partition-by-и-чем-оно-отличается-от-ключа-сортировки)).

**Семейство MergeTree** включает специализированные движки (Replacing/Summing/Aggregating, см. [вопрос 9](#вопрос-9-зачем-нужны-replacingmergetree-summingmergetree-и-aggregatingmergetree)) и реплицируемые версии (`ReplicatedMergeTree`, см. [вопрос 12](#вопрос-12-как-в-clickhouse-устроены-репликация-и-шардирование)). Это архитектурный фундамент ClickHouse — почти всегда таблицы создают на MergeTree-движках.

### Вопрос 6. Как устроен первичный ключ и разреженный (sparse) индекс в MergeTree?

**Категория:** ClickHouse · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [ClickHouse Docs — MergeTree](https://clickhouse.com/docs/engines/table-engines/mergetree-family/mergetree)

«Первичный ключ таблицы определяет порядок сортировки внутри каждой части таблицы (кластерный индекс)». Индекс разреженный: «первичный ключ ссылается не на отдельные строки, а на блоки по 8192 строки, называемые гранулами (granules). Это делает первичные ключи огромных наборов данных достаточно маленькими, чтобы оставаться загруженными в оперативной памяти». Настройка `index_granularity` (по умолчанию 8192) задаёт размер гранулы — минимальной неделимой единицы чтения; первая строка каждой гранулы помечается значением первичного ключа.

#### Ответ от Claude

В отличие от B-tree-индексов реляционных СУБД, индекс ClickHouse — **разреженный (sparse)**:

- Данные внутри части отсортированы по первичному ключу.
- Строки делятся на **гранулы** по `index_granularity` строк (по умолчанию **8192**).
- В индекс попадает **одна запись на гранулу** — значение ключа в **первой** строке гранулы («mark»). Поэтому индекс крошечный и целиком помещается в память даже для огромных таблиц.

**Как ускоряет запросы:** по условию из `WHERE` (по колонкам ключа) ClickHouse бинарным поиском находит гранулы, которые **могут** содержать подходящие строки, и **пропускает** остальные. Читается не вся таблица, а лишь нужные гранулы.

**Важные особенности (отличия от реляционных индексов):**
- Индекс **не обеспечивает уникальность** — это не ограничение, а только «карта» для отсечения данных.
- Эффективен, когда запросы фильтруют по **префиксу** ключа; ключ нужно подбирать под типичные запросы (от низкой кардинальности к высокой).
- Это **первичный** индекс; есть ещё вторичные «data-skipping» индексы (minmax, set, bloom_filter) для отсечения гранул по неключевым колонкам.

### Вопрос 7. В чём разница между ORDER BY (ключом сортировки) и PRIMARY KEY в MergeTree?

**Категория:** ClickHouse · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ClickHouse Docs — MergeTree](https://clickhouse.com/docs/engines/table-engines/mergetree-family/mergetree)

«Условие `ORDER BY` определяет, как данные сортируются внутри частей». Если первичный ключ явно не задан, ключом сортировки автоматически становится `ORDER BY`. «Первичный ключ определяет порядок сортировки внутри каждой части (кластерный индекс)». PRIMARY KEY может быть префиксом ORDER BY.

#### Ответ от Claude

В MergeTree это **связанные, но разные** понятия:

- **`ORDER BY` (ключ сортировки)** — определяет физический порядок строк в части. Обязателен для MergeTree.
- **`PRIMARY KEY`** — определяет, по каким колонкам строится разреженный индекс. По умолчанию **совпадает с `ORDER BY`**.

**Когда их разделяют:** `PRIMARY KEY` обязан быть **префиксом** `ORDER BY`. Это полезно, когда нужно сортировать данные по большему числу колонок (для лучшего сжатия или для семантики движка, например дедупликации в ReplacingMergeTree), но индексировать только префикс:

```sql
ORDER BY (tenant_id, user_id, event_time)
PRIMARY KEY (tenant_id, user_id)   -- индекс только по префиксу
```

**Зачем так:** индекс по короткому префиксу — меньше и быстрее в памяти, а полный `ORDER BY` обеспечивает нужную сортировку/дедупликацию. Если `PRIMARY KEY` не указан — он равен `ORDER BY` (это самый частый случай).

**Не путать с реляционным PRIMARY KEY:** здесь это не уникальность, а определение индекса/сортировки.

### Вопрос 8. Что такое партиционирование (PARTITION BY) и чем оно отличается от ключа сортировки?

**Категория:** ClickHouse · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ClickHouse Docs — MergeTree](https://clickhouse.com/docs/engines/table-engines/mergetree-family/mergetree)

«Условие `PARTITION BY` обеспечивает логическое деление данных (обычно по месяцу)». Партиции «хранятся раздельно и не сливаются между собой», что оптимизирует запросы за счёт отсечения партиций (partition pruning). Парты разных партиций никогда не объединяются при фоновом merge.

#### Ответ от Claude

**Партиционирование** (`PARTITION BY`) — это крупное **логическое** деление таблицы на независимые группы частей (например, по месяцу: `PARTITION BY toYYYYMM(date)`).

**Отличие от ключа сортировки (`ORDER BY`):**

| | `PARTITION BY` | `ORDER BY` |
|---|---|---|
| Уровень | Крупное деление таблицы на партиции | Порядок строк **внутри** части |
| Назначение | Отсечение целых партиций; управление данными | Разреженный индекс, сжатие |
| Слияние частей | Части **разных** партиций не сливаются | Сортировка для merge внутри партиции |
| Операции | `DROP PARTITION`, `DETACH`, TTL по партициям | — |

**Зачем нужно:**
- **Partition pruning:** запрос с фильтром по дате читает только нужные партиции.
- **Управление жизненным циклом:** мгновенно удалить старые данные через `ALTER TABLE ... DROP PARTITION` (намного дешевле `DELETE`), настроить TTL.

**Важная ошибка новичков:** партиционировать слишком мелко (например, по дню или по часу) — это плодит огромное число мелких частей и убивает производительность. Рекомендуется крупное деление (обычно по месяцам), а тонкую фильтрацию оставлять разреженному индексу.

### Вопрос 9. Зачем нужны ReplacingMergeTree, SummingMergeTree и AggregatingMergeTree?

**Категория:** ClickHouse · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [ClickHouse Docs — ReplacingMergeTree](https://clickhouse.com/docs/engines/table-engines/mergetree-family/replacingmergetree)

ReplacingMergeTree «удаляет дублирующиеся записи с одинаковым значением ключа сортировки», оставляя лишь одну. Но: «дедупликация происходит только во время слияния. Слияние выполняется в фоне в неизвестный момент, поэтому на него нельзя рассчитывать», и движок «не гарантирует отсутствие дубликатов». Поэтому «запросы могут давать неверные ответы». Для корректного результата на момент запроса используют модификатор `FINAL`, включающий дедупликацию и удаление на лету.

#### Ответ от Claude

Это специализированные движки семейства MergeTree, которые **во время фонового слияния** дополнительно преобразуют строки с одинаковым ключом сортировки (`ORDER BY`):

| Движок | Что делает при merge |
|---|---|
| **ReplacingMergeTree** | оставляет одну строку на ключ (последнюю/по версии) — эмуляция «обновления» / дедупликация |
| **SummingMergeTree** | суммирует числовые колонки строк с одинаковым ключом — лёгкая пред-агрегация |
| **AggregatingMergeTree** | хранит и сливает **состояния агрегатных функций** (`uniq`, `quantile`, `sum` и др.) через `-State`/`-Merge` |

**Критически важный нюанс — «в конечном счёте»:** все эти преобразования происходят **только при слиянии**, момент которого не гарантирован. Значит, до merge в таблице могут лежать дубликаты/несвёрнутые строки, и обычный `SELECT` вернёт «сырые» данные.

**Как получить корректный результат:**
- `SELECT ... FINAL` — дедупликация/свёртка на лету при запросе (дороже).
- Либо для Summing/Aggregating — всегда писать запрос с `GROUP BY` и нужной агрегацией (для AggregatingMergeTree — с суффиксом `-Merge`), не полагаясь на физическое состояние.

Эти движки обычно используются как целевые таблицы **материализованных представлений** для пред-агрегации (см. [вопрос 10](#вопрос-10-как-работают-материализованные-представления-materialized-views-в-clickhouse)).

### Вопрос 10. Как работают материализованные представления (materialized views) в ClickHouse?

**Категория:** ClickHouse · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [ClickHouse Docs — Materialized views](https://clickhouse.com/docs/materialized-view)

Инкрементальные материализованные представления «переносят стоимость вычислений со времени запроса на время вставки, благодаря чему `SELECT`-запросы становятся быстрее». В отличие от обновляемых (refreshable) представлений, которые периодически выполняют полный запрос по всему набору данных, инкрементальные обновляются по мере поступления новых данных.

#### Ответ от Claude

В ClickHouse «материализованное представление» работает **иначе, чем в PostgreSQL**. Это не периодически обновляемый снимок результата, а **триггер на вставку**.

**Инкрементальное materialized view:**
- При `INSERT` в исходную таблицу ClickHouse прогоняет SQL-запрос представления **по только что вставленному блоку** строк и записывает результат в **целевую таблицу** (обычно на движке Summing/AggregatingMergeTree).
- Так стоимость агрегации переносится со времени **чтения** на время **вставки** — последующие `SELECT` из целевой таблицы очень быстры.
- Видит только новые данные блока; исторические данные не пересчитывает.

**Типичный паттерн пред-агрегации:**
```
сырые события (MergeTree) → MV (агрегация) → витрина (AggregatingMergeTree)
```

**Отличие от реляционных СУБД:**
- В PostgreSQL `MATERIALIZED VIEW` — это сохранённый результат запроса, обновляемый целиком по `REFRESH`.
- В ClickHouse инкрементальное MV обновляется **при каждой вставке** и только по новым данным.
- Для поведения «как в Postgres» в ClickHouse есть отдельный тип — **refreshable materialized view** (периодически полностью пересчитывает запрос).

### Вопрос 11. Как в ClickHouse устроено сжатие данных?

**Категория:** ClickHouse · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ClickHouse Docs — Why is ClickHouse so fast?](https://clickhouse.com/docs/concepts/why-clickhouse-is-so-fast)

«Колоночные хранилища особенно хорошо подходят для такого сжатия, так как значения одного типа и одинакового распределения данных расположены вместе». ClickHouse поддерживает универсальные алгоритмы (например, ZSTD) и специализированные кодеки (Gorilla для чисел с плавающей точкой, Delta для целых), уменьшая и размер хранения, и время выполнения запросов за счёт снижения ограничений по вводу-выводу.

#### Ответ от Claude

Сжатие — одна из причин скорости ClickHouse, и оно естественно для колоночного хранения: рядом лежат **однотипные, похожие значения**, которые жмутся гораздо лучше, чем разнородные строки.

**Алгоритмы сжатия (на колонку):**
- **LZ4** (по умолчанию) — очень быстрый, умеренная степень сжатия.
- **ZSTD** — выше степень сжатия ценой CPU; для «холодных»/архивных данных.

**Специализированные кодеки** (можно комбинировать перед основным алгоритмом):
- **Delta / DoubleDelta** — для монотонных последовательностей (таймстемпы, счётчики): хранятся разницы, а не значения.
- **Gorilla** — для чисел с плавающей точкой (метрики).
- **T64**, **LowCardinality** (словарное кодирование для колонок с малым числом уникальных значений).

**Почему сжатие ускоряет запросы (а не только экономит диск):** дисковый и сетевой I/O — типичное узкое место. Меньше байт читается → быстрее запрос. Степень сжатия часто 5–20×.

**Практика:** каждой колонке можно назначить свой кодек под её тип данных (например, `Delta, ZSTD` для таймстемпов), что заметно повышает и сжатие, и скорость.

### Вопрос 12. Как в ClickHouse устроены репликация и шардирование?

**Категория:** ClickHouse · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [ClickHouse Docs — Data Replication](https://clickhouse.com/docs/engines/table-engines/mergetree-family/replication) · [ClickHouse Docs — Distributed Table Engine](https://clickhouse.com/docs/engines/table-engines/special/distributed)

Репликация: «Репликация асинхронная и multi-master. Запросы `INSERT` (как и `ALTER`) можно отправлять на любой доступный сервер». Она «работает на уровне отдельной таблицы, а не всего сервера». Для координации «ClickHouse использует ClickHouse Keeper для хранения метаинформации о репликах» (можно ZooKeeper, но Keeper рекомендуется); без него «нельзя создавать реплицируемые таблицы».

Шардирование: «Таблицы с движком Distributed не хранят собственных данных, но позволяют распределённую обработку запросов на нескольких серверах». Чтение «автоматически распараллеливается»; строки распределяются по шардам по ключу шардирования.

#### Ответ от Claude

В ClickHouse **репликация** и **шардирование** — независимые механизмы, которые часто используют вместе.

**Репликация (надёжность, копии данных) — `ReplicatedMergeTree`:**
- Асинхронная и **multi-master**: писать можно в **любую** реплику, остальные догоняют.
- Работает **на уровне таблицы**, а не сервера.
- Требует **ClickHouse Keeper** (или ZooKeeper) для координации: какие части существуют, какие реплики что имеют, дедупликация вставок.
- Даёт отказоустойчивость и масштабирование чтения.

**Шардирование (масштаб, разбиение данных) — движок `Distributed`:**
- `Distributed`-таблица **сама данных не хранит** — это «прокси» поверх шардов.
- При `SELECT` рассылает запрос на все шарды, исполняет параллельно и **объединяет** результаты.
- При `INSERT` распределяет строки по шардам согласно **ключу шардирования**.

**Типичная продакшен-схема:** N шардов (горизонтальный масштаб) × M реплик на шард (надёжность). Шардирование решает «данные не помещаются/не считаются на одном узле», репликация — «переживать отказ узла».

### Вопрос 13. Каковы слабые стороны ClickHouse и когда его не стоит использовать?

**Категория:** ClickHouse · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [ClickHouse Docs — ALTER TABLE … UPDATE](https://clickhouse.com/docs/sql-reference/statements/alter/update)

В документации прямо предупреждают: операции изменения данных через `ALTER TABLE` — «тяжёлые (heavy), не предназначенные для частого использования», в отличие от похожих запросов в OLTP-базах. ClickHouse оптимизирован под массовое чтение и периодические пакетные изменения, а не под постоянные построчные правки, характерные для операционных (OLTP) баз данных.

#### Ответ от Claude

ClickHouse — специализированная OLAP-СУБД, и вне своей ниши он слаб. Основные ограничения:

- **Частые точечные UPDATE/DELETE.** Реализованы как «мутации», переписывающие части целиком — это дорого и асинхронно (см. [вопрос 14](#вопрос-14-как-в-clickhouse-выполняются-update-и-delete)). Не годится для OLTP.
- **Нет полноценных транзакций** (ACID между несколькими операциями/таблицами), нет внешних ключей и привычной целостности.
- **Точечные запросы «дай строку по id»** неэффективны по сравнению со строковыми БД — ClickHouse рассчитан на агрегаты по диапазонам.
- **JOIN-ы** возможны, но требуют осторожности: большие JOIN-ы дороги по памяти; часто данные денормализуют или используют словари (dictionaries).
- **Вставка по одной строке** порождает множество мелких частей → деградация; нужно вставлять пачками.
- **Согласованность «в конечном счёте»** у спец-движков и реплик: дедупликация/свёртка происходят при merge, а не мгновенно.

**Когда не использовать:** как основную БД операционного приложения (заказы, сессии, корзина), при потребности в транзакциях и частых построчных изменениях. **Когда идеален:** аналитика, логи, метрики, события — массовые чтения и агрегации.

### Вопрос 14. Как в ClickHouse выполняются UPDATE и DELETE?

**Категория:** ClickHouse · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ClickHouse Docs — The Lightweight DELETE Statement](https://clickhouse.com/docs/sql-reference/statements/delete)

«Lightweight `DELETE` реализован как мутация, которая помечает строки как удалённые, но не удаляет их физически немедленно». К затронутым строкам «применяется маска» через скрытую колонку `_row_exists`; «мутация физически не удаляет помеченные строки — это произойдёт только при следующем слиянии (merge)». Это «повышает производительность по сравнению с традиционным `ALTER TABLE ... DELETE`», который «пересоздаёт все затронутые части». SELECT-запросы автоматически дополняются предикатом маски, чтобы пропускать удалённые строки.

#### Ответ от Claude

Поскольку части данных в ClickHouse **неизменяемы**, изменение/удаление строк — это не точечная операция «на месте», а более тяжёлый процесс.

**Тяжёлые мутации — `ALTER TABLE ... UPDATE/DELETE`:**
- Реализованы как **мутации**, которые **переписывают целиком все затронутые части**.
- Выполняются **асинхронно** в фоне (настройка `mutations_sync`).
- Документация прямо называет их «тяжёлой операцией, не предназначенной для частого использования».

**Lightweight DELETE — `DELETE FROM ... WHERE ...`:**
- Не переписывает части немедленно, а **помечает строки как удалённые** через скрытую колонку `_row_exists` (маску).
- `SELECT` автоматически отфильтровывает помеченные строки.
- Физическое удаление откладывается до ближайшего merge. Быстрее, чем `ALTER ... DELETE`, но это тоже мутация.

**Идиоматичные альтернативы частым правкам:**
- **ReplacingMergeTree** — «обновление» через вставку новой версии строки, старая схлопывается при merge (см. [вопрос 9](#вопрос-9-зачем-нужны-replacingmergetree-summingmergetree-и-aggregatingmergetree)).
- **DROP PARTITION** — мгновенно удалить целую партицию вместо построчного `DELETE` (см. [вопрос 8](#вопрос-8-что-такое-партиционирование-partition-by-и-чем-оно-отличается-от-ключа-сортировки)).
- **TTL** — автоматическое удаление устаревших данных.

**Вывод:** в ClickHouse правки данных проектируют пакетно и редко; для «обновлений» предпочитают вставку новых версий и схлопывание, а не классический `UPDATE`.

### Вопрос 15. Что такое векторизованное выполнение запросов в ClickHouse?

**Категория:** ClickHouse · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [ClickHouse Docs — Why is ClickHouse so fast?](https://clickhouse.com/docs/concepts/why-clickhouse-is-so-fast)

«ClickHouse использует слой векторизованной обработки, где операторы плана запроса передают промежуточные строки результата пачками (batches), а не по одной». Пакетная обработка улучшает использование кэшей CPU и позволяет применять SIMD-инструкции сразу к нескольким значениям. Запросы дополнительно распараллеливаются по ядрам CPU (обычно одна «дорожка» на ядро).

#### Ответ от Claude

**Векторизованное выполнение** — это обработка данных не по одной строке, а **пачками (векторами) значений** колонки за раз.

**Как это работает:**
- Операторы запроса (фильтры, агрегаты, арифметика) получают на вход не строку, а **блок** значений колонки (порядка тысяч — связано с гранулой в 8192 строки).
- К такому блоку применяются **SIMD-инструкции** процессора (SSE/AVX), обрабатывающие несколько значений одной командой.
- Данные одной колонки лежат в памяти непрерывно → отлично укладываются в **кэши CPU**, меньше промахов кэша.

**Почему это быстро:**
- Меньше накладных расходов на каждую строку (нет интерпретации «по строке»).
- Один такт CPU обрабатывает сразу несколько значений (параллелизм на уровне данных).
- Эффективное использование кэшей и предсказателя ветвлений.

**Почему именно колоночность это позволяет:** в строковом хранении значения нужной колонки «размазаны» по строкам, и собрать из них вектор дорого. В колоночном — они уже лежат подряд, идеально для SIMD. Вместе с распараллеливанием по ядрам CPU это даёт многократный прирост на аналитических операциях (см. также [вопрос 3](#вопрос-3-почему-clickhouse-такой-быстрый)).
