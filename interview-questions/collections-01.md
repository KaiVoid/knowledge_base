### Вопрос 1. Какова иерархия Java Collections Framework? Назовите основные интерфейсы и их взаимосвязь.

**Категория:** Collections · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Collections Framework Overview — Oracle Java SE 8 Docs](https://docs.oracle.com/javase/8/docs/technotes/guides/collections/overview.html)

Java Collections Framework организован в виде двух отдельных деревьев интерфейсов.

**Дерево 1 — иерархия Collection:**
- `Collection` — корневой интерфейс, представляет группу объектов-элементов
  - `Set` — коллекция без дубликатов
    - `SortedSet` — Set с элементами в отсортированном порядке
      - `NavigableSet` — SortedSet с навигационными методами (ceiling, floor и др.)
  - `List` — упорядоченная коллекция (последовательность), допускает дубликаты и доступ по индексу
  - `Queue` — коллекция для хранения элементов перед обработкой, как правило FIFO
    - `Deque` — двусторонняя очередь, поддерживает FIFO и LIFO

**Дерево 2 — иерархия Map:**
- `Map` — отображение ключей на значения; не является подтипом `Collection`
  - `SortedMap` — Map с ключами в отсортированном порядке
    - `NavigableMap` — SortedMap с навигационными методами

Для каждого интерфейса существуют стандартные реализации: `HashSet`/`TreeSet`/`LinkedHashSet` для Set; `ArrayList`/`LinkedList` для List; `ArrayDeque` для Deque; `HashMap`/`TreeMap`/`LinkedHashMap` для Map.

Все реализации общего назначения несинхронизированы. Для многопоточного использования предусмотрены конкурентные реализации: `ConcurrentHashMap`, `CopyOnWriteArrayList`, `LinkedBlockingQueue` и другие.

#### Ответ от Claude

Java Collections Framework содержит две независимые иерархии интерфейсов.

**Иерархия Collection** (корень — `java.util.Collection`):

```
Collection
├── List          — упорядоченная последовательность, допускает дубликаты, доступ по индексу
├── Set           — множество без дубликатов
│   └── SortedSet — Set с гарантированным порядком элементов
│       └── NavigableSet — расширяет SortedSet навигационными методами
└── Queue         — очередь (обычно FIFO) для хранения элементов до обработки
    └── Deque     — двусторонняя очередь (FIFO и LIFO одновременно)
```

**Иерархия Map** (корень — `java.util.Map`; Map **не** наследует Collection):

```
Map
└── SortedMap     — Map с ключами в отсортированном порядке
    └── NavigableMap — расширяет SortedMap навигационными методами
```

**Основные реализации общего назначения:**

| Интерфейс | Hash Table | Resizable Array | Balanced Tree | Linked List | Hash Table + Linked List |
|-----------|-----------|-----------------|---------------|-------------|--------------------------|
| `Set`     | `HashSet` | —               | `TreeSet`     | —           | `LinkedHashSet`          |
| `List`    | —         | `ArrayList`     | —             | `LinkedList`| —                        |
| `Deque`   | —         | `ArrayDeque`    | —             | `LinkedList`| —                        |
| `Map`     | `HashMap` | —               | `TreeMap`     | —           | `LinkedHashMap`          |

**Ключевые особенности:**
- Все интерфейсы обобщённые (`Collection<E>`, `Map<K,V>`).
- Реализации общего назначения несинхронизированы; для многопоточности используются классы из `java.util.concurrent` (`ConcurrentHashMap`, `CopyOnWriteArrayList` и др.) или обёртки `Collections.synchronizedXxx()`.
- Итераторы коллекций имеют семантику *fail-fast*: при конкурентном изменении кидают `ConcurrentModificationException`.

### Вопрос 2. Чем ArrayList отличается от LinkedList? В каких случаях следует выбирать каждый из них?

**Категория:** Collections · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [List Implementations — The Java Tutorials (Oracle)](https://docs.oracle.com/javase/tutorial/collections/implementations/list.html)

`ArrayList` реализует список на основе динамического массива и обеспечивает **константное время** доступа по индексу (O(1)). Это «просто быстро» — цитата из документации: «ArrayList offers constant-time positional access and is just plain fast».

`LinkedList` хранит элементы в двусвязном списке узлов. Добавление в начало списка и удаление из середины (при уже известном итераторе) требует **O(1)**, тогда как позиционный доступ — **O(n)**.

Рекомендация из официальной документации: «If you think you want to use a LinkedList, measure the performance of your application with both LinkedList and ArrayList before making your choice; ArrayList is usually faster.»

`LinkedList` стоит предпочесть, если часто добавляются элементы в начало списка или при итерации удаляются элементы из середины. `ArrayList` — выбор по умолчанию для большинства сценариев.

#### Ответ от Claude

**Внутренняя структура:**
- `ArrayList` — динамический массив (`Object[]`). При переполнении создаётся новый массив примерно в 1,5 раза большего размера, все элементы копируются.
- `LinkedList` — двусвязный список; каждый узел (`Node`) хранит элемент и ссылки `prev`/`next`. Также реализует интерфейс `Deque`, что делает его одновременно списком и очередью.

**Сравнение сложности операций:**

| Операция | ArrayList | LinkedList |
|---|---|---|
| `get(index)` | O(1) | O(n) |
| `add(E)` — в конец | O(1) амортизированно | O(1) |
| `add(0, E)` — в начало | O(n) | O(1) |
| `add(i, E)` — в середину | O(n) | O(n) (поиск узла) + O(1) (вставка) |
| `remove(index)` | O(n) | O(n) (поиск узла) + O(1) (удаление) |
| Итерация | Быстро (cache-friendly) | Медленнее (разброс в памяти) |

**Память:**
- `ArrayList` хранит только ссылки на объекты; при этом обычно есть «запас» зарезервированных пустых ячеек.
- `LinkedList` на каждый элемент создаёт объект-узел с двумя дополнительными ссылками — потребление памяти выше.

**Когда использовать `ArrayList`:**
- Чтение и доступ по индексу преобладают над вставкой/удалением.
- Вставка/удаление происходит в конец списка.
- Важна эффективность кэша процессора (элементы расположены последовательно в памяти).
- Это выбор по умолчанию в 90% случаев.

**Когда использовать `LinkedList`:**
- Очень частые вставки/удаления в начало или конец списка.
- Требуется функциональность очереди/деки (`addFirst`, `removeFirst`, `addLast`, `removeLast`).
- Размер списка резко меняется и важно избежать лишнего копирования массива.

> **Практический совет:** перед выбором `LinkedList` следует замерить производительность обоих вариантов — из-за лучшей локальности памяти `ArrayList` зачастую быстрее даже там, где теоретически должен проигрывать.

### Вопрос 3. Как работает HashMap внутри: как происходит вычисление bucket-индекса, что такое корзина (bucket) и как хранятся элементы?

**Категория:** Collections · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [HashMap (Java Platform SE 8) — Oracle Javadoc](https://docs.oracle.com/javase/8/docs/api/java/util/HashMap.html)

`HashMap` — реализация интерфейса `Map` на основе хэш-таблицы. Обеспечивает **константное время** выполнения операций `get` и `put` при условии равномерного распределения элементов по корзинам.

**Capacity и load factor:** Ёмкость (`capacity`) — количество корзин в хэш-таблице; по умолчанию равна 16. Коэффициент загрузки (`load factor`) — мера заполненности таблицы, при достижении которой ёмкость автоматически удваивается. Значение по умолчанию — **0.75**, что является компромиссом между временем и памятью.

**Rehashing:** Когда количество записей превышает `capacity * loadFactor`, хэш-таблица перестраивается (`rehashed`): создаётся новый массив примерно вдвое большего размера, все записи переносятся.

Для ключей, реализующих `Comparable`, при большом числе коллизий в одной корзине `HashMap` может использовать порядок сравнения для разрешения конфликтов.

#### Ответ от Claude

**Внутренняя структура:**

`HashMap` хранит пары «ключ–значение» в массиве (`Node<K,V>[] table`). Каждая ячейка массива — это **корзина (bucket)**. Изначально создаётся 16 корзин.

**Вычисление индекса корзины:**

1. Вызывается `key.hashCode()`, получается 32-битный хэш `h`.
2. Применяется дополнительное перемешивание (perturbation): `hash = h ^ (h >>> 16)` — это снижает вероятность коллизий при малых размерах таблицы.
3. Индекс корзины вычисляется как `(n - 1) & hash`, где `n` — текущая длина массива (всегда степень двойки). Это эквивалентно `hash % n`, но выполняется быстрее через битовое И.

**Хранение элементов:**

Каждая корзина содержит цепочку элементов. До Java 8 цепочка — это **односвязный список** объектов `Entry<K,V>`. Начиная с Java 8, при превышении порога `TREEIFY_THRESHOLD = 8` элементов в одной корзине список автоматически преобразуется в **красно-чёрное дерево** (структура `TreeNode`), что снижает сложность поиска с O(n) до O(log n). При уменьшении до `UNTREEIFY_THRESHOLD = 6` дерево преобразуется обратно в список.

**Rehashing:**

При превышении порога `capacity * loadFactor` (по умолчанию 16 × 0.75 = 12 элементов) массив расширяется вдвое и все элементы перераспределяются по новым корзинам. Поэтому порядок итерации по `HashMap` не гарантирован и может меняться при rehashing.

**Специальный случай null:** Ключ `null` всегда попадает в корзину 0 (хэш считается равным 0).

### Вопрос 4. Что произойдёт, если у двух ключей одинаковый hashCode(), но они не равны по equals()? Как HashMap разрешает коллизии?

**Категория:** Collections · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [How Do HashTables Deal with Collisions? — javathinking.com](https://www.javathinking.com/blog/how-do-hashtables-deal-with-collisions/)

**Коллизия** — ситуация, когда два разных ключа дают одинаковый хэш и попадают в одну корзину. Java `HashMap` использует стратегию **separate chaining** («цепочки»): каждая корзина указывает на вторичную структуру данных (связный список или дерево), хранящую все пары с одинаковым хэшем.

При вызове `put(key, value)`:
- Если корзина пуста — элемент добавляется как первый узел цепочки.
- Если в корзине уже есть элементы — HashMap проходит по цепочке и для каждого узла проверяет `key.equals(existingKey)`:
  - если `equals` вернул `true` — значение обновляется;
  - если `equals` вернул `false` — элемент добавляется в цепочку.

При вызове `get(key)` HashMap аналогично вычисляет корзину по хэшу, затем перебирает цепочку и использует `equals()` для поиска точного совпадения.

**Java 8+:** при превышении `TREEIFY_THRESHOLD = 8` связный список в корзине заменяется красно-чёрным деревом (O(log n) вместо O(n)). При уменьшении до `UNTREEIFY_THRESHOLD = 6` структура снова превращается в список.

#### Ответ от Claude

**Что происходит при одинаковом `hashCode()`, но разном `equals()`:**

Оба ключа считаются **разными** и **оба хранятся** в `HashMap` — в одной и той же корзине, но в разных узлах цепочки.

**Алгоритм разрешения коллизий (separate chaining):**

При `put(k2, v2)`, где `k1.hashCode() == k2.hashCode()` и `!k1.equals(k2)`:
1. Вычисляется индекс корзины (одинаковый для обоих ключей).
2. HashMap находит цепочку в этой корзине.
3. Перебирает узлы: для каждого проверяет `k2.equals(existingKey)`.
4. Поскольку `equals` возвращает `false`, новый узел добавляется в цепочку.
5. Оба ключа сосуществуют в одной корзине.

При `get(k2)`:
1. Вычисляется корзина.
2. HashMap перебирает цепочку, применяя `equals()` к каждому узлу.
3. Возвращается значение первого узла, где `equals` вернул `true`.

**Контракт `hashCode`/`equals` и его нарушение:**

Если `a.equals(b) == true`, то `a.hashCode() == b.hashCode()` **обязательно** — иначе `HashMap` сломается (не найдёт ключ). Обратное не обязательно: одинаковый `hashCode` не означает равенства, это и есть коллизия.

**Java 8+ оптимизация:**
- До Java 8: цепочка — односвязный список, поиск O(n) при n элементах в корзине.
- Java 8+: при достижении `TREEIFY_THRESHOLD = 8` список преобразуется в **красно-чёрное дерево**, поиск O(log n).
- При уменьшении до `UNTREEIFY_THRESHOLD = 6` дерево возвращается в список.
- Treeification происходит только если суммарная ёмкость таблицы >= `MIN_TREEIFY_CAPACITY = 64`; иначе HashMap просто делает rehashing.

**Практическое следствие:** плохо реализованный `hashCode()` (например, всегда возвращающий константу) превращает `HashMap` в связный список с O(n) на каждую операцию.

### Вопрос 5. Почему при использовании объекта в качестве ключа HashMap важно переопределять оба метода hashCode() и equals()?

**Категория:** Collections · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Java hashCode() and equals() — jenkov.com](https://jenkov.com/tutorials/java-collections/hashcode-equals.html)

Когда объект сохраняется в HashMap, его `hashCode()` вычисляется и используется для определения места хранения внутри таблицы (бакета). При поиске HashMap снова вычисляет хэш-код переданного ключа, находит нужный бакет, а затем перебирает все ключи в этом бакете с помощью `equals()`, чтобы найти точное совпадение.

Правило контракта: если два объекта равны согласно `equals()`, они **обязаны** иметь одинаковый `hashCode()`. Если переопределить только `equals()` — объекты, которые должны быть равны, могут попасть в разные бакеты, и поиск завершится неудачей. Если переопределить только `hashCode()` — объекты окажутся в одном бакете, но `equals()` вернёт `false` (сравнение по ссылке), и ключ снова не будет найден.

#### Ответ от Claude

HashMap хранит пары ключ-значение в массиве «бакетов» (корзин). Поиск и вставка работают в два этапа:

1. **hashCode()** определяет индекс бакета: `index = hashCode() & (capacity - 1)`.
2. **equals()** идентифицирует конкретный ключ внутри бакета (т.к. у нескольких ключей может оказаться одинаковый хэш-код — коллизия).

**Контракт между hashCode() и equals()** (закреплён в спецификации `Object`):
- Если `a.equals(b) == true`, то `a.hashCode() == b.hashCode()` — **обязательно**.
- Обратное необязательно: одинаковый хэш-код не означает равенства (коллизия допустима).

**Что происходит при нарушении контракта:**

| Нарушение | Следствие |
|---|---|
| Переопределён только `equals()` | Два логически равных объекта попадают в **разные** бакеты → `get()` возвращает `null` |
| Переопределён только `hashCode()` | Объекты попадают в один бакет, но `equals()` использует сравнение по ссылке (`Object` по умолчанию) → `get()` тоже возвращает `null` |
| Ни один не переопределён | HashMap работает корректно, но только при поиске по **тому же экземпляру** объекта |

**Пример корректной реализации:**
```java
class Key {
    private final String id;

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Key)) return false;
        return id.equals(((Key) o).id);
    }

    @Override
    public int hashCode() {
        return id.hashCode();
    }
}
```

Дополнительно важно, чтобы `hashCode()` был **стабильным** — не менялся, пока объект используется как ключ в карте. Изменение состояния объекта-ключа после добавления в HashMap ведёт к тому, что ключ становится «потерянным» — его нельзя будет ни найти, ни удалить. Поэтому в качестве ключей рекомендуется использовать **неизменяемые (immutable) объекты** — `String`, `Integer` и т.п.

### Вопрос 6. Чем HashSet отличается от TreeSet и LinkedHashSet? Как HashSet использует HashMap внутри?

**Категория:** Collections · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Set Implementations — The Java Tutorials, docs.oracle.com](https://docs.oracle.com/javase/tutorial/collections/implementations/set.html)

`HashSet` значительно быстрее `TreeSet` (операции за константное время против логарифмического), но не даёт никаких гарантий порядка итерации.

`LinkedHashSet` занимает промежуточное положение между `HashSet` и `TreeSet`: реализован как хэш-таблица со связным списком, проходящим через все элементы. Он обеспечивает итерацию в порядке вставки (от наименее к наиболее недавно добавленному) и работает почти так же быстро, как `HashSet`.

`TreeSet` следует использовать, когда нужен интерфейс `SortedSet` или итерация в порядке значений элементов.

#### Ответ от Claude

**Сравнительная таблица:**

| Характеристика | HashSet | LinkedHashSet | TreeSet |
|---|---|---|---|
| Внутренняя структура | `HashMap` | `LinkedHashMap` | `TreeMap` (красно-чёрное дерево) |
| Порядок итерации | Не определён | Порядок вставки | Естественный / Comparator |
| Производительность базовых операций | O(1) | O(1) (чуть медленнее HashSet) | O(log n) |
| Допустимость `null` | Одно значение `null` | Одно значение `null` | Нет (бросает `NullPointerException`) |
| Интерфейсы | `Set` | `Set` | `SortedSet`, `NavigableSet` |

**Как HashSet использует HashMap внутри:**

Согласно официальной документации Java, `HashSet` «реализует интерфейс `Set`, поддерживаемый экземпляром `HashMap`» (backed by a `HashMap` instance). Внутри `HashSet` хранит только ключи этой карты, а в качестве значений использует единственный статический объект-заглушку `PRESENT = new Object()`. Таким образом:

```java
// Упрощённая схема внутренней реализации HashSet
private transient HashMap<E, Object> map;
private static final Object PRESENT = new Object();

public boolean add(E e) {
    return map.put(e, PRESENT) == null;
}

public boolean contains(Object o) {
    return map.containsKey(o);
}

public boolean remove(Object o) {
    return map.remove(o) == PRESENT;
}
```

Все операции `HashSet` делегируются соответствующим методам внутреннего `HashMap`. Благодаря этому `HashSet` наследует все характеристики `HashMap`: константное время операций, необходимость корректных `hashCode()`/`equals()` у элементов, не потокобезопасность.

**`LinkedHashSet`** является прямым подклассом `HashSet` и внутри использует `LinkedHashMap` вместо обычного `HashMap` — за счёт двусвязного списка, соединяющего все записи в порядке вставки.

**`TreeSet`** внутри использует `TreeMap` (красно-чёрное дерево), что обеспечивает автоматическую сортировку элементов, но ценой O(log n) для каждой операции.

### Вопрос 7. Какова разница между методами poll() и remove() у интерфейса Queue? Что происходит при вызове каждого на пустой очереди?

**Категория:** Collections · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Queue (Java SE 17 & JDK 17) — docs.oracle.com](https://docs.oracle.com/en/java/javase/17/docs/api/java.base/java/util/Queue.html)

Методы `remove()` и `poll()` различаются только поведением при пустой очереди: метод `remove()` бросает исключение, тогда как `poll()` возвращает `null`.

- `poll()` — извлекает и удаляет голову очереди; если очередь пуста, возвращает `null`.
- `remove()` — извлекает и удаляет голову очереди; если очередь пуста, бросает `NoSuchElementException`.

#### Ответ от Claude

Интерфейс `Queue` предоставляет две пары методов для каждой операции — одна пара бросает исключение при отказе, другая возвращает специальное значение:

| Операция | Бросает исключение | Возвращает специальное значение |
|---|---|---|
| Добавить | `add(e)` | `offer(e)` → `false` |
| Удалить (голову) | `remove()` | `poll()` → `null` |
| Просмотреть (без удаления) | `element()` | `peek()` → `null` |

**Детали поведения:**

- **`remove()`**: Удаляет и возвращает голову очереди. При вызове на **пустой** очереди бросает `java.util.NoSuchElementException`. Подходит для случаев, когда пустая очередь является **ошибкой** (нештатной ситуацией).

- **`poll()`**: Удаляет и возвращает голову очереди. При вызове на **пустой** очереди возвращает `null`. Подходит для случаев, когда пустая очередь является **ожидаемым** состоянием.

**Практический совет:**

Использование `poll()` предпочтительно, когда очередь может быть пустой в нормальном ходе программы:
```java
Queue<String> queue = new LinkedList<>();
String item = queue.poll();
if (item != null) {
    process(item);
}
```

Использование `remove()` оправдано, когда по логике программы очередь не должна быть пустой, и пустота — это признак бага:
```java
// Ожидаем, что очередь не пуста; если пуста — это ошибка
String item = queue.remove(); // NoSuchElementException сигнализирует о проблеме
```

**Важно:** `null` не следует использовать как обычный элемент очереди, если применяется `poll()`, поскольку возвращённый `null` будет неотличим от признака пустой очереди. Это отмечено в официальной документации.

### Вопрос 8. Что такое fail-fast итератор? Какое исключение он бросает и в каких коллекциях используется?

**Категория:** Collections · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [ConcurrentModificationException (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/util/ConcurrentModificationException.html)

Это исключение может быть брошено методами, которые обнаружили параллельное изменение объекта, когда такое изменение недопустимо. Например, один поток может изменять коллекцию, пока другой поток итерирует по ней. В целом результаты итерации были бы неопределёнными. Некоторые реализации итераторов (включая реализации всех универсальных коллекций из JRE) решают эту проблему с помощью поведения «fail-fast»: такие итераторы бросают `ConcurrentModificationException`, обнаружив подобную ситуацию. Итераторы, делающие это, называются **fail-fast итераторами**, так как они быстро и чисто дают сбой, вместо того чтобы рисковать произвольным недетерминированным поведением в неопределённый момент в будущем.

#### Ответ от Claude

**Fail-fast итератор** — это итератор, который немедленно бросает исключение при обнаружении того, что коллекция была структурно изменена после создания итератора (кроме как через собственный метод `remove()` итератора).

**Механизм работы:**

Коллекция поддерживает внутренний счётчик модификаций `modCount`, который увеличивается при каждом структурном изменении (добавление, удаление элементов). При создании итератора он запоминает текущее значение `modCount` в поле `expectedModCount`. При каждом вызове `next()` итератор сравнивает `modCount` с `expectedModCount`:

```java
// Упрощённая схема проверки в ArrayList.Itr
final void checkForComodification() {
    if (modCount != expectedModCount)
        throw new ConcurrentModificationException();
}
```

**Исключение:** `java.util.ConcurrentModificationException` (наследник `RuntimeException`).

**Коллекции с fail-fast итераторами** (все стандартные коллекции из `java.util`):
- `ArrayList`, `LinkedList`, `Vector`
- `HashSet`, `LinkedHashSet`, `TreeSet`
- `HashMap`, `LinkedHashMap`, `TreeMap`, `Hashtable`
- И все остальные коллекции `java.util`

**Важные уточнения:**
1. Поведение fail-fast является «наилучшим усилием» (best-effort), а **не гарантией**. Документация Oracle прямо указывает, что на него нельзя полагаться при написании корректных программ — его следует использовать **только для обнаружения багов**.
2. Fail-fast срабатывает даже при **однопоточном** изменении коллекции во время итерации (например, в цикле `for-each`).
3. Единственный безопасный способ удалять элементы во время итерации — использовать **`Iterator.remove()`** (он обновляет `expectedModCount`).

**Противоположность — fail-safe итераторы** используются в коллекциях из пакета `java.util.concurrent` (например, `CopyOnWriteArrayList`, `ConcurrentHashMap`): они работают с копией или снимком данных и никогда не бросают `ConcurrentModificationException`.

### Вопрос 9. Как HashMap ведёт себя при операции rehashing (перехэшировании)? Когда оно происходит и как влияет на load factor и capacity?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [HashMap (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/util/HashMap.html)

Согласно официальной документации Oracle, **capacity** (ёмкость) — это количество бакетов в хэш-таблице, а **load factor** (коэффициент загрузки) — мера того, насколько заполнена таблица перед автоматическим увеличением ёмкости.

Когда количество записей превышает произведение load factor и текущей ёмкости, хэш-таблица **перехэшируется** (то есть внутренние структуры данных перестраиваются) — таблица получает примерно вдвое больше бакетов.

Формула порогового значения (threshold): `threshold = loadFactor × capacity`

Значения по умолчанию: начальная ёмкость = 16, load factor = 0.75, следовательно, порог = 12. Документация рекомендует: «Как правило, load factor по умолчанию (0.75) обеспечивает хороший компромисс между временными и пространственными затратами. Более высокие значения снижают накладные расходы на память, но увеличивают стоимость поиска».

#### Ответ от Claude

**Когда происходит rehashing**

Rehashing запускается в момент, когда количество записей в HashMap превышает пороговое значение: `threshold = capacity × loadFactor`. При настройках по умолчанию (capacity=16, loadFactor=0.75) порог равен 12: добавление тринадцатого элемента инициирует перехэширование.

**Что происходит во время rehashing**

1. Создаётся новый массив бакетов удвоенного размера (16 → 32 → 64 и т.д.).
2. Для каждой уже хранящейся пары «ключ–значение» заново вычисляется индекс бакета на основе нового размера таблицы.
3. Все элементы перераспределяются по новым бакетам.
4. Новый порог пересчитывается: `newThreshold = newCapacity × loadFactor` (например, 32 × 0.75 = 24).

**Влияние на load factor и capacity**

- **capacity** удваивается: операция всегда производится до следующей степени двойки.
- **load factor** как параметр конфигурации не изменяется — он остаётся тем же значением (0.75 по умолчанию), которое было задано при создании карты.
- **Фактическое заполнение** (entries/capacity) после rehashing резко падает примерно до 37.5% (0.75/2), поскольку то же количество элементов распределяется по вдвое большему числу бакетов.

**Производительность**

Само перехэширование является дорогостоящей операцией — O(n) по времени и памяти, — однако амортизированная сложность операций put остаётся O(1), так как перехэширование происходит всё реже по мере роста таблицы. Если заранее известно приблизительное количество элементов, рекомендуется задавать начальную ёмкость явно: `new HashMap<>(expectedSize / 0.75 + 1)` — это позволяет избежать промежуточных перехэширований.

**Важный нюанс:** rehashing в Java 8 оптимизирован — благодаря тому что capacity всегда является степенью двойки, при удвоении размера каждый элемент либо остаётся в прежнем бакете, либо перемещается ровно на `oldCapacity` позиций вперёд. Это исключает необходимость полного пересчёта хэш-кодов и делает операцию быстрее, чем в общем случае.

### Вопрос 10. Что изменилось во внутреннем устройстве HashMap начиная с Java 8? Почему LinkedList в bucket заменяется на красно-чёрное дерево и при каком условии?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [The Secret Improvement of HashMap in Java 8 — runzhuoli.me](https://runzhuoli.me/2018/08/31/the-secret-improvement-of-hashmap-in-java8.html)

Статья описывает три ключевых константы, введённых в Java 8:

- **TREEIFY_THRESHOLD = 8**: когда длина связного списка в бакете достигает 8 и более элементов, список конвертируется в красно-чёрное дерево.
- **UNTREEIFY_THRESHOLD = 6**: когда длина уменьшается до 6 (например, при удалении элементов или при resize), дерево обратно конвертируется в связный список.
- **MIN_TREEIFY_CAPACITY = 64**: дерево формируется только если общая ёмкость таблицы превышает 64. Иначе вместо treeification вызывается resize().

Мотивация: сложность поиска в связном списке — O(n), тогда как в красно-чёрном дереве — O(log n). При атаках через коллизии или при плохой реализации hashCode() это кардинально улучшает поведение в худшем случае.

#### Ответ от Claude

**Полный список изменений в Java 8 HashMap**

1. **Гибридная структура бакетов: список + красно-чёрное дерево.** До Java 8 каждый бакет хранил только односвязный список. В Java 8 список автоматически преобразуется в красно-чёрное дерево при выполнении определённых условий.

2. **Улучшенная хэш-функция.** Вместо многоступенчатого смешивания Java 7 применяется более элегантная операция: `(h = key.hashCode()) ^ (h >>> 16)`. XOR старших 16 бит с младшими позволяет лучше распределять ключи в небольших таблицах, снижая количество коллизий.

3. **Изменение порядка вставки в список.** В Java 7 новые элементы вставлялись в начало списка (head insertion), что в многопоточной среде могло приводить к бесконечному циклу при resize. В Java 8 используется tail insertion (вставка в конец), что исключает эту проблему.

**Почему LinkedList заменяется на красно-чёрное дерево**

При большом числе коллизий связный список деградирует до O(n) для операций get/put. Красно-чёрное дерево, будучи самобалансирующимся двоичным деревом поиска, гарантирует O(log n) даже в наихудшем случае. Это особенно важно при целенаправленных атаках через подбор коллизий.

**Точные условия для treeification**

Преобразование в дерево происходит только при одновременном выполнении двух условий:
- число элементов в конкретном бакете достигло `TREEIFY_THRESHOLD` (= 8);
- общая ёмкость таблицы (длина массива бакетов) не меньше `MIN_TREEIFY_CAPACITY` (= 64).

Если второе условие не выполнено (таблица меньше 64 бакетов), вместо treeification вызывается `resize()` — удвоение таблицы, которое само по себе перераспределяет элементы и устраняет длинные цепочки.

**Обратное преобразование**

При удалении элементов или во время resize, если в дереве остаётся не более `UNTREEIFY_THRESHOLD` (= 6) элементов, оно преобразуется обратно в связный список. Гистерезис между 8 и 6 предотвращает частое переключение туда-обратно.

**Детали реализации**

Узел дерева представлен классом `TreeNode`, который расширяет `LinkedHashMap.Entry`, что позволяет легко выполнять обратное преобразование. Когда бакет является деревом, в роли объекта синхронизации (и корня) выступает специальный узел `TreeBin` (в ConcurrentHashMap) или сам `TreeNode` (в HashMap).

### Вопрос 11. В чём разница между HashMap и ConcurrentHashMap с точки зрения потокобезопасности? Какой механизм синхронизации использует ConcurrentHashMap в Java 8?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ConcurrentHashMap (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/ConcurrentHashMap.html)

Документация Oracle описывает ConcurrentHashMap как хэш-таблицу, поддерживающую полный параллелизм при извлечении и высокую ожидаемую конкурентность при обновлении. Ключевые тезисы:

- Операции получения (get) **не блокируют** и могут пересекаться с операциями обновления (put, remove).
- Операции получения отражают результаты последних **завершённых** операций обновления.
- Между записью и последующим чтением одного и того же ключа существует отношение happens-before.
- В отличие от HashMap, ConcurrentHashMap **не допускает null** в качестве ключа или значения.
- Итераторы не бросают ConcurrentModificationException.

В Java 8 механизм сегментной блокировки (Segment-based locking из Java 5–7) заменён на массив узлов `Node<K,V>[]` с применением CAS-операций и тонкой блокировки на уровне отдельного бакета.

#### Ответ от Claude

**HashMap и потокобезопасность**

HashMap не является потокобезопасным. Если несколько потоков одновременно изменяют HashMap и хотя бы один из них выполняет запись, необходима внешняя синхронизация. В Java 7 конкурентный resize мог приводить к бесконечному циклу из-за head-insertion при перестройке цепочек. В Java 8 эта конкретная проблема устранена (tail-insertion), однако HashMap по-прежнему не даёт никаких гарантий видимости изменений между потоками.

**ConcurrentHashMap: механизм синхронизации в Java 8**

До Java 8 (Java 5–7) ConcurrentHashMap использовал массив `Segment`, каждый из которых был `ReentrantLock`-ом. Это обеспечивало параллелизм на уровне сегментов (по умолчанию 16), но ограничивало масштабируемость.

В Java 8 архитектура полностью переработана:

1. **Структура данных.** Хранится `volatile Node<K,V>[] table` — массив бакетов, аналогичный HashMap, но объявленный volatile для гарантии видимости изменений.

2. **CAS (Compare-And-Swap) для пустых бакетов.** При вставке в пустой бакет используется атомарная CAS-операция без захвата блокировки. Это lock-free путь для большинства вставок.

3. **synchronized на первом узле бакета.** Если бакет уже содержит элементы, поток захватывает монитор первого узла (`synchronized(f)`). Поскольку разные бакеты имеют разные первые узлы, несколько потоков могут одновременно писать в разные бакеты.

4. **TreeBin для деревьев.** Когда бакет становится красно-чёрным деревом, роль блокировки берёт на себя специальный узел `TreeBin` (не содержащий данных), что предотвращает проблемы при смене корня дерева.

5. **Неблокирующие чтения.** Операции get не используют никаких блокировок: volatile-семантика массива и ссылок на узлы гарантирует корректную видимость.

**Ключевые отличия от HashMap**

| Аспект | HashMap | ConcurrentHashMap |
|---|---|---|
| Потокобезопасность | Отсутствует | Полная |
| Механизм | Нет | CAS + synchronized на бакет |
| Null ключи/значения | Разрешены | Запрещены |
| Итератор | Fail-fast | Fail-safe (weakly consistent) |
| Производительность в многопотоке | Небезопасно | Высокая за счёт гранулярности |

**Почему null запрещён в ConcurrentHashMap:** в конкурентном контексте невозможно отличить «ключ отсутствует» от «ключ есть, но значение null», поскольку между containsKey() и get() другой поток может удалить элемент.

### Вопрос 12. Чем ConcurrentHashMap отличается от Collections.synchronizedMap()? Какой из них предпочтительнее в высоконагруженных приложениях и почему?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Java ConcurrentHashMap vs Collections.synchronizedMap() — howtodoinjava.com](https://howtodoinjava.com/java/multi-threading/synchronizedmap-vs-concurrenthashmap/)

Статья описывает принципиальное различие в механизмах блокировки:

**Collections.synchronizedMap()** оборачивает существующую Map и обеспечивает «последовательный доступ» (serial access) — только один поток может обращаться к карте в любой момент времени, как при чтении, так и при записи. Блокируется весь объект целиком.

**ConcurrentHashMap** делит данные на 16 бакетов по умолчанию и допускает «неблокирующий доступ на чтение для всех потоков», требуя блокировки только на конкретный бакет при записи. Несколько потоков могут одновременно писать в разные бакеты.

**По итерации:** ConcurrentHashMap предоставляет fail-safe итераторы (не бросают ConcurrentModificationException), synchronizedMap — fail-fast итераторы. При этом даже для synchronizedMap итерацию нужно дополнительно синхронизировать вручную блоком `synchronized(map)`.

**По null:** ConcurrentHashMap не допускает null-ключи и null-значения, synchronizedMap — в зависимости от оборачиваемой Map (HashMap допускает один null-ключ).

#### Ответ от Claude

**Механизм синхронизации**

`Collections.synchronizedMap(map)` возвращает обёртку, в которой каждый публичный метод (`get`, `put`, `remove`, `size` и т.д.) обёрнут в `synchronized(mutex)`, где mutex — это сам объект-обёртка. Это означает **блокировку всей карты** на каждую операцию, включая чтения.

`ConcurrentHashMap` в Java 8 использует **гранулярную синхронизацию**: CAS-операции для вставки в пустые бакеты и `synchronized` на первом узле конкретного бакета для остальных записей. Чтения полностью lock-free за счёт volatile-семантики.

**Производительность в высоконагруженных приложениях**

В высоконагруженных многопоточных сценариях `ConcurrentHashMap` значительно превосходит `synchronizedMap`:
- Операции чтения в ConcurrentHashMap не блокируют никаких потоков вообще.
- Операции записи блокируют только один бакет из N (типично из 16+), а не всю карту.
- Несколько потоков могут одновременно писать в разные бакеты.
- По бенчмаркам разница в скорости при конкурентной нагрузке может составлять 2–5x и более.

**Поведение итераторов**

- `synchronizedMap`: итератор fail-fast — бросает `ConcurrentModificationException` при любом изменении во время обхода. Для безопасной итерации необходимо явно синхронизировать весь блок: `synchronized(syncMap) { for (...) }`.
- `ConcurrentHashMap`: итератор weakly consistent (слабо согласованный) — не бросает исключений, отражает состояние карты на момент создания итератора, но может увидеть (или не увидеть) изменения, внесённые после начала итерации.

**Атомарные операции**

`ConcurrentHashMap` предоставляет атомарные составные операции: `putIfAbsent`, `computeIfAbsent`, `computeIfPresent`, `merge`, `compute`. У `synchronizedMap` этих гарантий нет — конструкция `if (!map.containsKey(k)) map.put(k, v)` не является атомарной, даже если каждый вызов синхронизирован.

**Когда что выбирать**

| Критерий | ConcurrentHashMap | synchronizedMap |
|---|---|---|
| Высоконагруженное многопоточное приложение | Предпочтительнее | Не рекомендуется |
| Нужно обернуть существующий Map (напр., TreeMap) | Невозможно | Применимо |
| Требуются составные атомарные операции | Встроены | Требуется внешняя синхронизация |
| Допустимы null ключи/значения | Нет | Зависит от базовой Map |
| Простота использования без тонкостей | Выше | Ниже (итерация требует ручной синхронизации) |

**Вывод:** в новом коде для конкурентного доступа следует предпочитать `ConcurrentHashMap`. `Collections.synchronizedMap()` оправдан, когда нужно сделать потокобезопасной уже существующую реализацию `Map` (например, `TreeMap` или `LinkedHashMap`), при этом понимая ограничения по производительности.
### Вопрос 13. Что такое weakly consistent итератор в ConcurrentHashMap? Чем он отличается от fail-fast итератора HashMap?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ConcurrentHashMap (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/ConcurrentHashMap.html)

Официальная документация Oracle прямо указывает: итераторы, Spliterator-ы и Enumeration-ы ConcurrentHashMap **не бросают `ConcurrentModificationException`**. Они «возвращают элементы, отражающие состояние хэш-таблицы в некоторый момент при или после создания итератора/перечисления». Это означает, что итератор видит снимок состояния на момент своего создания и может (но не гарантированно) отражать модификации, произошедшие после создания итератора. Документация также подчёркивает, что, несмотря на потокобезопасность самой карты, каждый конкретный итератор должен использоваться только одним потоком за раз.

#### Ответ от Claude

**Fail-fast итератор (HashMap)**

HashMap хранит внутренний счётчик структурных модификаций — `modCount`. При создании итератора он запоминает текущее значение `modCount` в поле `expectedModCount`. При каждом вызове `next()` итератор сравнивает текущий `modCount` с `expectedModCount`. Если значения расходятся (т.е. коллекция была структурно изменена — добавлены/удалены элементы — не через метод итератора), незамедлительно бросается `ConcurrentModificationException`. Это «best-effort» гарантия: исключение не гарантируется в многопоточной среде, но гарантируется при однопоточном изменении коллекции во время итерации. Итератор самого HashMap также имеет метод `remove()`, который корректно обновляет `modCount` и не вызывает исключение.

Ключевое ограничение: fail-fast итератор не может безопасно использоваться при конкурентном доступе — он не предназначен для многопоточных сценариев.

**Weakly consistent итератор (ConcurrentHashMap)**

Итератор ConcurrentHashMap называется «weakly consistent» (слабо согласованным). Его свойства, закреплённые в документации Oracle:

1. **Не бросает `ConcurrentModificationException`** — структурные изменения карты во время итерации безопасны.
2. **Отражает состояние на момент создания итератора** — к моменту вызова `next()` итератор гарантированно обходит элементы, существовавшие при создании итератора.
3. **Может (но не обязан) отражать последующие изменения** — вставки или удаления, произошедшие после создания итератора, могут быть видны итератору, а могут и нет; поведение не определено детерминированно.

Важный нюанс: ConcurrentHashMap **не создаёт полную копию коллекции** при создании итератора (в отличие от CopyOnWriteArrayList, который создаёт снимок). Weakly consistent семантика достигается через особую структуру сегментов (в Java 7 — explicit сегменты с блокировками; в Java 8+ — CAS-операции и volatile-чтение из массива бакетов). Это значит, что итерация не требует копирования всех данных и работает с минимальным overhead.

**Ключевые отличия в таблице:**

| Характеристика | HashMap (fail-fast) | ConcurrentHashMap (weakly consistent) |
|---|---|---|
| `ConcurrentModificationException` | Да, при обнаружении изменений | Никогда |
| Механизм | Сравнение `modCount` | Volatile-чтение через CAS |
| Снимок данных | Нет | Нет (не копирует) |
| Видимость параллельных изменений | Исключение вместо неопределённости | Может увидеть, может не увидеть |
| Потокобезопасность итерации | Нет | Да (сам итератор — одним потоком) |
| Накладные расходы | Минимальные | Минимальные |

**Практический вывод:** fail-fast итератор защищает от случайного изменения коллекции в однопоточном сценарии; weakly consistent итератор обеспечивает безопасную итерацию в многопоточной среде за счёт ослабления гарантий видимости изменений.

### Вопрос 14. Какова сложность операций get(), put(), remove() у ArrayList, LinkedList и HashMap в лучшем и худшем случае?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Runtime Complexity of Java Collections — gist.github.com/psayre23](https://gist.github.com/psayre23/c30a821239f4818b0709)

Согласно известному справочному гисту по сложности Java-коллекций:

- **ArrayList**: `get` — O(1), `add` (append) — O(1) амортизировано, `remove` — O(n).
- **LinkedList**: `get` — O(n), `add` (append) — O(1), `remove` — O(1) при известном узле, O(n) при поиске по значению.
- **HashMap**: `get`, `put`, `remove` — O(1) в среднем случае.

Гист уточняет: для ArrayList `add` в worst-case O(n) из-за копирования массива при расширении; для HashMap операции предполагают равномерное распределение хэшей.

#### Ответ от Claude

**ArrayList**

| Операция | Лучший случай | Худший случай | Пояснение |
|---|---|---|---|
| `get(index)` | O(1) | O(1) | Прямое обращение по индексу в массиве |
| `add(e)` (append) | O(1) | O(n) | Worst-case: расширение массива (System.arraycopy) |
| `add(index, e)` | O(1) | O(n) | Сдвиг элементов вправо от позиции index |
| `remove(index)` | O(1) | O(n) | Сдвиг элементов влево; удаление последнего — O(1) |
| `remove(object)` | O(n) | O(n) | Линейный поиск + сдвиг |

Амортизированная сложность `add(e)` — O(1): расширение происходит редко (размер удваивается), поэтому при n операциях суммарная стоимость O(n).

**LinkedList (двусвязный список)**

| Операция | Лучший случай | Худший случай | Пояснение |
|---|---|---|---|
| `get(index)` | O(1) | O(n) | Лучший: head/tail; худший: середина списка |
| `add(e)` (append) | O(1) | O(1) | Всегда: обновление хвоста |
| `add(index, e)` | O(1) | O(n) | Нужно дойти до позиции index |
| `remove(index)` | O(1) | O(n) | Поиск узла по индексу O(n), само удаление O(1) |
| `remove(object)` | O(1) | O(n) | Линейный поиск по значению |
| `addFirst/addLast` | O(1) | O(1) | Прямой доступ к head/tail |

**HashMap (Java 8+)**

| Операция | Лучший случай | Худший случай | Пояснение |
|---|---|---|---|
| `get(key)` | O(1) | O(log n) | Лучший: нет коллизий; худший: бакет-дерево |
| `put(key, val)` | O(1) | O(log n) | Аналогично get |
| `remove(key)` | O(1) | O(log n) | Аналогично get |

Детали worst-case для HashMap в Java 8+:
- До Java 8: при коллизиях бакет хранился как связный список — worst-case O(n).
- Java 8+: при превышении порога `TREEIFY_THRESHOLD = 8` бакет преобразуется в красно-чёрное дерево, что улучшает worst-case до O(log n).
- Реальный O(1) достигается при хорошей хэш-функции и равномерном распределении ключей.
- Патологический worst-case O(n) сохраняется в Java 8+ только если `hashCode()` всегда возвращает одно значение и бакет не может «дотреефицироваться» (например, ключи не реализуют `Comparable`).

**Итоговое сравнение (среднее/амортизированное):**

| Операция | ArrayList | LinkedList | HashMap |
|---|---|---|---|
| Доступ по индексу/ключу | O(1) | O(n) | O(1) |
| Вставка в конец | O(1)* | O(1) | O(1) |
| Удаление по индексу/ключу | O(n) | O(n) | O(1) |

*Амортизированно.

### Вопрос 15. Чем LinkedHashMap отличается от HashMap? Как он поддерживает порядок вставки и как это используется для реализации LRU-кэша?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [LinkedHashMap (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/util/LinkedHashMap.html)

Официальная документация Oracle описывает LinkedHashMap как реализацию Map с предсказуемым порядком итерации — в отличие от HashMap. LinkedHashMap поддерживает двусвязный список, пронизывающий все записи. По умолчанию порядок — порядок вставки; повторная вставка существующего ключа не влияет на порядок. Альтернативный режим — порядок доступа (access-order), задаётся через конструктор `LinkedHashMap(int, float, boolean)`. В режиме access-order при вызове `get`, `put`, `getOrDefault` и ряда других методов соответствующий элемент перемещается в конец списка. Документация прямо указывает, что этот режим пригоден для построения LRU-кэшей с помощью переопределения метода `removeEldestEntry`.

#### Ответ от Claude

**Структурные отличия от HashMap**

LinkedHashMap наследует HashMap и добавляет к каждой записи (`Entry`) два дополнительных указателя — `before` и `after`, формирующих двусвязный список через все записи карты. Это обеспечивает:

- Предсказуемый порядок итерации (в отличие от HashMap с непредсказуемым порядком).
- Итерацию за O(size), тогда как HashMap итерирует за O(capacity) (обходит все бакеты, включая пустые).
- Незначительный дополнительный overhead по памяти (два указателя на каждую запись).

Все базовые операции (`get`, `put`, `remove`, `containsKey`) остаются O(1) — LinkedHashMap полностью наследует хэш-таблицу HashMap.

**Два режима порядка**

1. **Порядок вставки (по умолчанию, `accessOrder = false`):**
   Элементы при итерации возвращаются в том порядке, в котором были добавлены. Повторный `put` уже существующего ключа не меняет его позицию в списке.

2. **Порядок доступа (`accessOrder = true`):**
   Конструктор: `new LinkedHashMap<>(initialCapacity, loadFactor, true)`.
   При любом «касании» элемента (`get`, `put`, `getOrDefault`, `compute`, `merge`, `replace` при реальной замене) он перемещается в хвост списка. Таким образом, голова списка всегда содержит наиболее давно использованный элемент.

**Реализация LRU-кэша**

LinkedHashMap с `accessOrder = true` и переопределённым `removeEldestEntry` — классический способ реализации LRU-кэша в одной строке:

```java
int capacity = 100;
Map<K, V> lruCache = new LinkedHashMap<K, V>(capacity, 0.75f, true) {
    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > capacity;
    }
};
```

Механизм работы:
- При каждом `put` или `putAll` LinkedHashMap вызывает `removeEldestEntry`.
- Если метод возвращает `true`, автоматически удаляется самый старый (наименее недавно используемый) элемент — голова двусвязного списка.
- `get` перемещает элемент в хвост (обновляет статус «использованности»).

В результате при превышении ёмкости вытесняется тот элемент, к которому дольше всего не было обращений.

**Ключевые отличия LinkedHashMap vs HashMap:**

| Характеристика | HashMap | LinkedHashMap |
|---|---|---|
| Порядок итерации | Не определён | Insertion- или access-order |
| Итерация | O(capacity) | O(size) |
| Память на запись | Меньше | +2 ссылки (before/after) |
| LRU из коробки | Нет | Да (с accessOrder=true) |
| Производительность базовых операций | O(1) | O(1) |
| Потокобезопасность | Нет | Нет |

**Практические сценарии использования LinkedHashMap:**
- Кэши с политикой вытеснения (LRU).
- Дедупликация с сохранением порядка встречаемости.
- Реализация ordered pipeline (шаги в определённой последовательности).
- Замена `Properties`/конфигурации там, где важен порядок ключей.

### Вопрос 16. Какова разница между TreeMap и HashMap? Когда использование TreeMap оправдано несмотря на O(log n) сложность операций?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [TreeMap (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/util/TreeMap.html)

Официальная документация Oracle описывает TreeMap как реализацию NavigableMap на основе красно-чёрного дерева. Ключи хранятся в отсортированном порядке — по естественному порядку (если ключи реализуют `Comparable`) или по явно переданному `Comparator`. Документация гарантирует логарифмическую сложность O(log n) для операций `containsKey`, `get`, `put` и `remove`. TreeMap предоставляет NavigableMap-методы: `firstKey()`, `lastKey()`, `floorKey()`, `ceilingKey()`, `lowerKey()`, `higherKey()`, а также представления диапазонов: `subMap()`, `headMap()`, `tailMap()`. Документация явно указывает, что TreeMap не синхронизирован.

#### Ответ от Claude

**Структурные различия**

| Характеристика | HashMap | TreeMap |
|---|---|---|
| Внутренняя структура | Массив бакетов + (список / RB-дерево) | Красно-чёрное дерево |
| Интерфейс | `Map` | `NavigableMap`, `SortedMap`, `Map` |
| Сложность get/put/remove | O(1) среднее, O(log n) worst-case | O(log n) гарантированно |
| Порядок ключей | Не определён | Сортированный (natural/Comparator) |
| `null`-ключи | Разрешён один | Запрещён (NPE) |
| Потоки | Не синхронизирован | Не синхронизирован |
| Итераторы | Fail-fast | Fail-fast |
| Память | Меньше | Больше (указатели дерева) |

**Когда TreeMap оправдан несмотря на O(log n):**

1. **Нужна отсортированная итерация по ключам.**
   Единственный способ получить элементы карты в порядке ключей без ручной сортировки. HashMap потребует `new ArrayList<>(map.keySet())` + `Collections.sort()` — это O(n log n) на каждую итерацию, тогда как TreeMap поддерживает порядок бесплатно при каждой вставке.

2. **Диапазонные запросы (range queries).**
   Методы `subMap(from, to)`, `headMap(to)`, `tailMap(from)` возвращают живые представления (view) части карты за O(log n). Аналогичная операция на HashMap — O(n) полный обход с фильтрацией.

3. **Навигация по соседним ключам.**
   - `floorKey(k)` — наибольший ключ ≤ k (тарификация, маршрутизация).
   - `ceilingKey(k)` — наименьший ключ ≥ k.
   - `lowerKey(k)` / `higherKey(k)` — строго меньший/больший.
   Эти операции на TreeMap — O(log n); на HashMap их реализация потребует O(n).

4. **Первый/последний элемент.**
   `firstKey()`, `lastKey()`, `pollFirstEntry()`, `pollLastEntry()` — O(log n). На HashMap — O(n).

5. **Планировщики и очереди событий по времени.**
   `TreeMap<Long, List<Event>>` с `headMap(now, true)` даёт все события до текущего момента как live view — естественная реализация таймерных очередей.

6. **Реализация порядков на основе кастомного `Comparator`.**
   Например, сортировка строк без учёта регистра, сортировка по длине строки и т.д.

**Когда TreeMap НЕ нужен (лучше HashMap):**

- Нужна только быстрая проверка наличия ключа / значения — O(1) vs O(log n).
- Порядок ключей не важен.
- Нужна поддержка `null`-ключа.
- Максимальная производительность вставок и поиска при большом n.

**Практический пример (тарификация):**

```java
// Карта порогов: ключ — минимальное кол-во единиц, значение — цена за единицу
TreeMap<Integer, Double> pricing = new TreeMap<>();
pricing.put(1, 10.0);
pricing.put(100, 8.0);
pricing.put(1000, 6.0);

// Найти актуальную цену для 350 единиц: наибольший порог ≤ 350
Map.Entry<Integer, Double> tier = pricing.floorEntry(350); // key=100, value=8.0
```

На HashMap эту же задачу невозможно решить без O(n) полного перебора.

### Вопрос 17. Что происходит при добавлении элементов в ArrayList после достижения его capacity? Какова стратегия роста внутреннего массива?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [How the Size of the ArrayList Increases Dynamically? — javaguides.net](https://www.javaguides.net/2018/08/how-size-of-arraylist-increases-dynamically.html)

ArrayList использует внутренний массив объектов (`Object[] elementData`) для хранения элементов. Когда массив заполняется, ArrayList создаёт новый массив большего размера и копирует в него все элементы из старого. Формула роста:

```java
int newCapacity = oldCapacity + (oldCapacity >> 1);
```

Побитовый сдвиг `>> 1` делит старую ёмкость на 2, поэтому новая ёмкость составляет **1.5× от прежней**. Например, полный массив на 10 элементов расширяется до 15. Перенос данных выполняется через `Arrays.copyOf()`. После того как новый массив создан и заполнен, старый массив перестаёт использоваться и утилизируется сборщиком мусора. Начиная с Java 8, пустой `ArrayList` создаётся с нулевой ёмкостью (ленивая инициализация): память под `DEFAULT_CAPACITY = 10` выделяется только при первом вызове `add()`.

#### Ответ от Claude

**Внутреннее устройство.** ArrayList хранит элементы в поле `Object[] elementData`. У каждого экземпляра есть *size* (текущее количество элементов) и *capacity* (длина этого массива). Capacity всегда ≥ size.

**Последовательность при переполнении.** При вызове `add(element)` ArrayList вызывает `ensureCapacityInternal(size + 1)`, которая в свою очередь вызывает `grow()`, если текущей ёмкости недостаточно. Метод `grow()` вычисляет новую ёмкость:

```java
// OpenJDK — java.util.ArrayList
private void grow(int minCapacity) {
    int oldCapacity = elementData.length;
    int newCapacity = oldCapacity + (oldCapacity >> 1); // ~1.5×
    if (newCapacity < minCapacity)
        newCapacity = minCapacity;
    if (newCapacity > MAX_ARRAY_SIZE)
        newCapacity = hugeCapacity(minCapacity);
    elementData = Arrays.copyOf(elementData, newCapacity);
}
```

`Arrays.copyOf()` создаёт новый массив нужного размера и копирует в него все существующие элементы (реализовано через `System.arraycopy` — нативный вызов). Старый массив становится недостижимым и собирается GC.

**Коэффициент роста 1.5.** Oracle намеренно не фиксирует точный алгоритм в Javadoc (документируется только гарантия амортизированного O(1) для `add`), однако в OpenJDK реализация использует множитель ≈1.5 начиная с Java 6. Это компромисс: множитель 2 (как в Vector) даёт меньше перераспределений, но больше тратит памяти; 1.5 экономнее.

**Ленивая инициализация в Java 8.** До Java 8 конструктор `new ArrayList()` сразу выделял массив размером 10. Начиная с Java 8 внутреннее поле `elementData` при создании пустого списка указывает на статическую константу `DEFAULTCAPACITY_EMPTY_ELEMENTDATA` (пустой массив нулевой длины). Первый `add()` обнаруживает это и устанавливает ёмкость равной `DEFAULT_CAPACITY = 10`. Это сокращает накладные расходы памяти для множества пустых или недолгоживущих списков.

**Примеры прогрессии роста** (начиная с `DEFAULT_CAPACITY = 10`):  
10 → 15 → 22 → 33 → 49 → 73 → 109 → ...

**Ручное управление.** Метод `ensureCapacity(int minCapacity)` позволяет заблаговременно зарезервировать место, избежав серии промежуточных перераспределений, если количество предстоящих вставок известно заранее.

### Вопрос 18. Почему операция вставки в середину LinkedList формально O(1), а на практике часто медленнее, чем у ArrayList? С чем это связано?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ArrayList vs LinkedList — javaadvent.com](https://www.javaadvent.com/2013/12/arraylist-vs-linkedlist.html)

Элементы LinkedList разбросаны по куче, тогда как ArrayList хранит ссылки в непрерывном участке памяти. Обращение к очередному узлу LinkedList означает переход по указателю к произвольному адресу — что с высокой вероятностью вызывает промах кэша процессора (cache miss). В реальных бенчмарках LinkedList показал примерно в **10 раз больше тактов процессора** и значительно больше промахов кэша L2 по сравнению с ArrayList при итерации. ArrayList выигрывает за счёт пространственной локальности: CPU-prefetcher предсказывает последовательные адреса и подгружает нужные данные заранее, не дожидаясь обращения к RAM.

#### Ответ от Claude

**Откуда берётся «O(1)» у LinkedList.** Утверждение справедливо только в очень узком смысле: если у вас **уже есть итератор (или ссылка на узел)**, указывающий на нужное место, то операция вставки между двумя узлами действительно O(1) — требует перенаправления двух указателей (`prev.next = newNode`, `newNode.next = current` и аналогично `prev`-ссылок для двусвязного списка). Именно это и гарантирует контракт `ListIterator.add()` у LinkedList.

**Почему на практике всё не так.** При типичном сценарии «вставить по индексу» (`list.add(index, element)`) LinkedList должен сначала *найти* узел с нужным индексом — а это последовательный обход от head (или tail, если индекс ближе к концу). Обход занимает O(n/2) шагов, и именно здесь возникает реальная стоимость.

Ключевые факторы, делающие LinkedList медленнее на современном железе:

1. **Cache misses при обходе.** Каждый узел LinkedList — отдельный объект в куче, выделенный в произвольный момент. При обходе `node = node.next` процессор вынужден читать память по случайному адресу. Современные процессоры работают со строками кэша (~64 байта): при последовательном обходе ArrayList следующий элемент с высокой вероятностью уже находится в кэше L1/L2. Для LinkedList каждый переход потенциально означает промах кэша и ожидание RAM (~100 нс вместо ~1 нс для L1).

2. **Аппаратный prefetcher не помогает.** CPU умеет предзагружать данные при линейном доступе к памяти. Хаотичные адреса LinkedList лишают эту оптимизацию смысла.

3. **Накладные расходы памяти на узел.** Каждый узел `Node<E>` хранит три поля: `item`, `next`, `prev`. Для объектных ссылок (64-битная JVM без сжатия) это минимум 24–32 байта служебных данных на элемент плюс заголовок объекта. ArrayList хранит только ссылку (4–8 байт) в компактном массиве.

4. **System.arraycopy у ArrayList.** Даже при вставке в середину ArrayList тратит O(n) на сдвиг элементов, однако делает это одним нативным вызовом `System.arraycopy`, который обрабатывает данные блоками с максимальным использованием кэша и SIMD-инструкций. Это значительно быстрее, чем n дереференций указателей в LinkedList.

**Итог.** Сложность O(1) у LinkedList — теоретическая и актуальна только при наличии готовой ссылки на позицию. Для реальной вставки по индексу обе структуры показывают O(n), но ArrayList почти всегда быстрее из-за лучшей локальности доступа к памяти. LinkedList имеет смысл применять преимущественно как очередь/дек с операциями только на концах.

### Вопрос 19. Опишите внутреннее устройство HashMap в Java 8+: как хранятся элементы, что происходит при коллизии, когда связный список превращается в красно-чёрное дерево и при каких условиях обратно?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Java HashMap internals — deepakvadgama.com](https://deepakvadgama.com/blog/java-hashmap-internals/)

HashMap использует массив объектов типа `Node<K,V>[]` (в Java 8 переименованы из `Entry`). Размер массива всегда является степенью двойки. Хэш-функция сначала вычисляет `key.hashCode()`, затем применяет побитовую манипуляцию `(h = key.hashCode()) ^ (h >>> 16)` для перемешивания старших бит в младшие. Индекс бакета вычисляется как `(n - 1) & hash`, что эквивалентно взятию по модулю, но работает быстрее. При коллизии узлы образуют односвязный список внутри бакета. Начиная с Java 8 при превышении 8 элементов в бакете и общей ёмкости таблицы более 64 список преобразуется в красно-чёрное дерево (`TreeNode`). Операции поиска внутри дерева — O(log n) вместо O(n). При уменьшении элементов ниже порога дерево обратно преобразуется в список.

#### Ответ от Claude

**Структура хранения.** HashMap содержит массив бакетов `Node<K,V>[] table`. Каждый `Node` хранит: `hash` (предвычисленный хэш), `key`, `value` и ссылку `next` на следующий узел при коллизии.

**Хэш-функция.** При `put(key, value)` вычисляется «дополнительная» хэш-функция:
```java
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```
XOR со сдвинутой на 16 бит копией снижает количество коллизий для хэшей, различающихся только в старших битах. Индекс бакета: `(n - 1) & hash`, где `n` — текущая ёмкость (степень двойки).

**Обработка коллизий — связный список.** Если несколько ключей попадают в один бакет, они выстраиваются в односвязный список. При поиске перебираются все узлы с проверкой `key.equals()`. Это деградация до O(n) при множестве коллизий.

**Преобразование в красно-чёрное дерево (treeify).** Три ключевые константы:
- `TREEIFY_THRESHOLD = 8` — минимальное число узлов в бакете для преобразования.
- `MIN_TREEIFY_CAPACITY = 64` — минимальная общая ёмкость таблицы для treeify. Если таблица меньше 64, вместо treeify выполняется resize (расширение), чтобы перераспределить узлы.
- `UNTREEIFY_THRESHOLD = 6` — максимальное число узлов, при котором дерево обратно превращается в список.

Преобразование происходит только при **одновременном** выполнении обоих условий: бакет содержит ≥ 8 узлов И общий размер таблицы ≥ 64. `TreeNode` является расширением `LinkedHashMap.Entry` и поддерживает двусвязный обход, поэтому после treeify в бакете хранится дерево поиска с порядком по `hash` (и `compareTo`/`System.identityHashCode` при равных хэшах).

**Обратное преобразование (untreeify).** При удалении элементов или во время resize(), если число узлов в бакете падает до ≤ 6, `TreeNode`-список снова преобразуется в обычный `Node`-список. Порог 6 (а не 7) выбран для предотвращения гистерезиса — флуктуаций между двумя структурами при добавлении/удалении пограничного элемента.

**Сложность операций:**

| Структура в бакете | get/put (среднее) | get/put (худшее) |
|--------------------|-------------------|------------------|
| Список (до Java 8) | O(1)              | O(n)             |
| Список (Java 8+)   | O(1)              | O(n)             |
| Дерево (Java 8+)   | O(1)              | O(log n)         |

**Изменения применяются к** `HashMap`, `LinkedHashMap` и `ConcurrentHashMap` (JEP 180).

### Вопрос 20. Что такое load factor и capacity в HashMap? Как они влияют на производительность, и что происходит при rehashing?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [HashMap (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/util/HashMap.html)

**Capacity** — количество бакетов в хэш-таблице. **Load factor** — мера заполнения таблицы, при которой её ёмкость автоматически увеличивается. Когда количество записей превышает произведение load factor на текущую capacity, выполняется rehashing: внутренние структуры перестраиваются так, что число бакетов приблизительно удваивается. Значения по умолчанию: capacity = 16, load factor = 0.75. Это соотношение является хорошим компромиссом между временными и пространственными затратами: высокие значения load factor снижают накладные расходы памяти, но увеличивают стоимость поиска. Если известно примерное число записей, имеет смысл задать начальную capacity так, чтобы rehashing не требовался вовсе: `initial capacity > max entries / load factor`.

#### Ответ от Claude

**Capacity (ёмкость).** Это длина внутреннего массива бакетов `Node[] table`. Всегда является степенью двойки (даже если задать другое значение в конструкторе — HashMap округлит вверх). Начальное значение по умолчанию — **16**.

**Load factor (коэффициент загрузки).** Число от 0 до 1 (по умолчанию **0.75**), определяющее, при каком заполнении таблицы происходит её расширение. Порог расширения (*threshold*) вычисляется как: `threshold = capacity × loadFactor`. При capacity=16 и loadFactor=0.75 порог равен 12: при добавлении 13-го элемента запускается rehashing.

**Процесс rehashing (resize).** Представляет собой тяжёлую операцию O(n):
1. Создаётся новый массив бакетов удвоенного размера (`newCapacity = oldCapacity × 2`).
2. Для каждого существующего узла пересчитывается позиция в новом массиве. Благодаря тому что размер всегда является степенью двойки, индекс нового бакета — это либо старый индекс, либо старый индекс + oldCapacity (определяется одним битом хэша). Это позволяет обойтись без повторного вычисления полного хэша.
3. Все узлы перераспределяются по новым бакетам. Порядок элементов в бакетах при этом может измениться.

**Влияние на производительность.**

| Параметр | Малое значение | Большое значение |
|---|---|---|
| **Load factor** | Частый rehash, больше памяти | Редкий rehash, больше коллизий, медленнее get/put |
| **Capacity** | Больше коллизий сразу | Меньше коллизий, но больше памяти |

- Значение 0.75 эмпирически обеспечивает в среднем меньше одного элемента на бакет при хорошей хэш-функции.
- При итерации по HashMap стоимость пропорциональна `capacity + size` (перебираются все бакеты, в том числе пустые), поэтому завышенная начальная ёмкость замедляет итерацию.

**Рекомендации.**
- Если заранее известно максимальное число элементов N, используйте `new HashMap<>(N / 0.75 + 1)` или `new HashMap<>((int)(N / 0.75f) + 1)`, чтобы полностью исключить rehashing.
- Повышение load factor (например, до 1.0) допустимо, когда память важнее скорости поиска.
- Понижение load factor ускоряет поиск за счёт разрежённости таблицы, но увеличивает потребление памяти.
