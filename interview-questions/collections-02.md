### Вопрос 21. Почему при переопределении equals() обязательно нужно переопределять hashCode()? Что произойдёт с объектом в HashMap, если переопределить только equals()?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Java equals() and hashCode() | DigitalOcean](https://www.digitalocean.com/community/tutorials/java-equals-hashcode)

Методы `equals()` и `hashCode()` связаны строгим контрактом: если `o1.equals(o2)` возвращает `true`, то `o1.hashCode()` должен быть равен `o2.hashCode()`. Нарушение этого контракта не вызывает исключения, но ломает работу коллекций на основе хэша — HashMap, HashSet и подобных.

При добавлении объекта в HashMap метод `put()` сначала вызывает `hashCode()`, чтобы найти нужный бакет (корзину). Затем, уже внутри бакета, вызывается `equals()` для сравнения ключей. Если переопределить только `equals()`, два логически равных объекта будут иметь разные хэш-коды (от реализации по умолчанию из `Object`, которая берёт адрес в памяти), и HashMap будет искать их в разных бакетах. В результате `get()` вернёт `null`, даже если объект с логически равным ключом уже хранится в карте.

#### Ответ от Claude

**Контракт equals/hashCode (из Javadoc класса Object)**

Официальная документация Java (docs.oracle.com, `Object.hashCode()`) формулирует контракт явно:

1. Если `a.equals(b) == true`, то `a.hashCode() == b.hashCode()` — **обязательно**.
2. Если `a.hashCode() == b.hashCode()`, это **не** означает, что `a.equals(b) == true` (коллизии допустимы).
3. `hashCode()` должен возвращать одно и то же значение при многократных вызовах на одном объекте, пока поля, участвующие в `equals()`, не изменились.

Там же прямо написано: *«Как правило, необходимо переопределять `hashCode()` каждый раз, когда переопределяется `equals()`, чтобы соблюдать общий контракт `hashCode()`: равные объекты должны иметь равные хэш-коды»*.

**Как работает HashMap**

HashMap организует хранение через массив бакетов (начиная с Java 8 — массив `Node[]`, при большом числе коллизий бакет превращается в красно-чёрное дерево). Алгоритм поиска ключа:

1. Вычислить `key.hashCode()` и применить дополнительное «перемешивание» (`(h = key.hashCode()) ^ (h >>> 16)`).
2. По хэшу определить индекс бакета: `index = hash & (capacity - 1)`.
3. Перебрать элементы внутри бакета, сравнивая через `equals()`.

**Что произойдёт, если переопределить только equals()**

Допустим, создан класс `Person` с переопределённым `equals()` (по полю `name`), но хэш-код берётся из `Object` — то есть является адресом объекта в памяти.

```java
Person p1 = new Person("Alice");
Person p2 = new Person("Alice");

p1.equals(p2); // true — логически равны

Map<Person, String> map = new HashMap<>();
map.put(p1, "developer");
map.get(p2); // null! — хотя p1.equals(p2) == true
```

Почему: `p1.hashCode() != p2.hashCode()` (разные адреса в памяти), поэтому `get(p2)` смотрит в другой бакет и ничего не находит. Объект **«теряется»** в HashMap — он туда помещён, но достать его по логически равному ключу невозможно.

**Дополнительный эффект: дубликаты в HashMap**

Если продолжить вставку таких объектов, можно накопить несколько записей с «логически одинаковыми» ключами в разных бакетах — HashMap не сможет распознать их как дубликаты. Это приводит к утечке памяти при длительной работе.

**Вывод**

Переопределять `equals()` без `hashCode()` — нарушение контракта, задокументированного в `java.lang.Object`. Джошуа Блох в книге *Effective Java* формулирует это как одно из ключевых правил: *«Переопределяй hashCode() в каждом классе, где переопределяешь equals()»*.

### Вопрос 22. Чем отличается ArrayList от LinkedList с точки зрения внутренней реализации? Почему на практике ArrayList зачастую быстрее LinkedList даже для операций вставки в середину?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Choosing the Right Implementation Between ArrayList and LinkedList — Dev.java](https://dev.java/learn/api/collections-framework/arraylist-vs-linkedlist/)

ArrayList хранит элементы в непрерывном массиве (`Object[] elementData`). LinkedList — это двусвязный список: каждый элемент обёрнут в объект `Node`, содержащий ссылки на предыдущий и следующий узлы.

Несмотря на то что теоретически вставка в произвольное место у LinkedList — O(1) (после нахождения позиции), на практике ArrayList часто выигрывает из-за **локальности кэша** (cache locality). При обходе массива CPU загружает соседние ячейки памяти в кэш-линию одновременно. LinkedList требует разыменования указателей, разбросанных по куче, что вызывает промахи кэша (cache miss) и может быть в 3 раза медленнее. Кроме того, каждый узел LinkedList занимает примерно 24 байта накладных расходов: для 1000 элементов LinkedList потребляет ~24 КБ против ~5 КБ у ArrayList.

#### Ответ от Claude

**Внутренняя реализация**

| Характеристика | ArrayList | LinkedList |
|---|---|---|
| Структура данных | Динамический массив `Object[] elementData` | Двусвязный список узлов `Node<E>` |
| Поля узла | — | `item`, `prev`, `next` |
| Доступ по индексу | O(1) — прямое обращение к массиву | O(n) — обход от начала или конца |
| Память на элемент | ~4–8 байт (ссылка в массиве) | ~24 байта (объект Node + 2 ссылки + заголовок) |

**Почему ArrayList быстрее LinkedList даже при вставке в середину**

*Теоретические сложности* — обманчивы:

- LinkedList: найти позицию — O(n), переставить указатели — O(1). Итого O(n).
- ArrayList: найти позицию — O(1), сдвинуть хвост — O(n) через `System.arraycopy`. Итого O(n).

На бумаге — одинаково. На практике — ArrayList значительно быстрее по нескольким причинам:

**1. Локальность кэша (cache locality)**

Современные процессоры работают с кэш-линиями (~64 байта). Когда CPU обращается к элементу массива, он автоматически подгружает соседние элементы в L1/L2-кэш. При следующем обращении данные уже в кэше — cache hit. LinkedList хранит узлы в произвольных местах кучи (heap). Каждое разыменование ссылки (`node.next`) может быть промахом кэша — cache miss, что стоит сотни наносекунд (200–300 тактов процессора против 4–5 для L1-попадания).

**2. System.arraycopy — нативная операция**

Сдвиг элементов в ArrayList выполняется через `System.arraycopy()` — нативный вызов JVM, который использует оптимизированные инструкции процессора (`memcpy`/`memmove`). Это чрезвычайно быстро даже для больших массивов.

**3. Накладные расходы на создание Node**

Каждая вставка в LinkedList создаёт новый объект `Node<E>` в куче, что нагружает GC и увеличивает давление на аллокатор. ArrayList лишь сдвигает уже существующие ссылки.

**4. Размер структуры в памяти**

LinkedList для 1000 элементов занимает ~24 КБ (24 байта на узел), ArrayList — ~8 КБ (8 байт на ссылку). Меньший объём данных = быстрее прогрев кэша.

**Когда LinkedList всё же выигрывает**

Только операции с концами списка: `addFirst()`, `addLast()`, `removeFirst()`, `removeLast()` — O(1), без перебора. Это делает LinkedList хорошей реализацией очереди/деки для сценариев с высокой частотой добавления/удаления с краёв. Для всего остального `ArrayList` — практически всегда лучший выбор.

### Вопрос 23. Как устроен HashSet внутри? Почему он не допускает дубликатов, и какие методы объекта используются при вызове set.add(element)?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Internal implementation of Set/HashSet (How Set Ensures Uniqueness) — Java Hungry](https://javahungry.blogspot.com/2013/08/how-sets-are-implemented-internally-in.html)

HashSet внутренне использует HashMap. При создании объекта `HashSet` одновременно создаётся связанный объект `HashMap`. Метод `add(E e)` вызывает `map.put(e, PRESENT)`, где `PRESENT` — заглушка-константа `new Object()`.

HashMap возвращает `null`, если ключ добавлен впервые, и старое значение, если ключ уже существовал. Поэтому `add()` реализован как `return map.put(e, PRESENT) == null` — возвращает `true` при успешном добавлении и `false` при попытке вставить дубликат.

Уникальность ключей в HashMap (а значит, и элементов в HashSet) определяется через `hashCode()` (для нахождения бакета) и `equals()` (для сравнения внутри бакета).

#### Ответ от Claude

**Внутренняя структура HashSet**

HashSet — это «обёртка» над HashMap. В исходном коде JDK (`java.util.HashSet`) объявлены два ключевых поля:

```java
private transient HashMap<E, Object> map;
private static final Object PRESENT = new Object();
```

Все элементы HashSet хранятся как **ключи** этого HashMap. Значением всегда служит одна и та же константа-заглушка `PRESENT`.

**Что происходит при вызове set.add(element)**

```java
public boolean add(E e) {
    return map.put(e, PRESENT) == null;
}
```

Шаг за шагом:

1. **Вычисление хэш-кода**: вызывается `element.hashCode()`. HashMap дополнительно «перемешивает» хэш: `h ^ (h >>> 16)`.
2. **Определение бакета**: `index = hash & (capacity - 1)`.
3. **Поиск в бакете**: HashMap перебирает элементы бакета и для каждого проверяет:
   - совпадение хэшей (`hash == nodeHash`)
   - `element == existingKey || element.equals(existingKey)`
4. **Если совпадение найдено**: `put()` обновляет значение (в данном случае `PRESENT` заменяется на `PRESENT` — фактически ничего не меняется) и возвращает старое значение (`PRESENT != null`). В итоге `add()` возвращает `false` — элемент не добавлен.
5. **Если совпадения нет**: `put()` создаёт новый `Node` и возвращает `null`. `add()` возвращает `true` — элемент добавлен.

**Почему HashSet не допускает дубликатов**

Механизм дедупликации целиком унаследован от HashMap: ключи в HashMap уникальны по определению. HashSet просто эксплуатирует это свойство, используя добавляемые элементы как ключи.

**Важный вывод**: уникальность в HashSet полностью зависит от корректной реализации `hashCode()` и `equals()` в хранимых объектах. Если `hashCode()` всегда возвращает одно значение, все элементы попадут в один бакет — HashSet будет работать, но с производительностью O(n) вместо O(1). Если нарушен контракт equals/hashCode — в HashSet могут появиться «логические дубликаты».

**Методы объекта, задействованные при add()**

| Метод | Когда вызывается | Зачем |
|---|---|---|
| `hashCode()` | Первым делом | Определить бакет |
| `equals()` | При нахождении бакета | Сравнить с существующими ключами |

### Вопрос 24. Что такое fail-fast итератор? Как реализован механизм обнаружения конкурентной модификации (поле modCount), и в каких коллекциях он применяется?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Fail Fast and Fail Safe Iterators in Java — GeeksforGeeks](https://www.geeksforgeeks.org/java/fail-fast-fail-safe-iterators-java/)

Fail-fast итераторы используют внутренний счётчик `modCount`, который отслеживает структурные изменения коллекции (добавление/удаление элементов). При создании итератора он сохраняет текущее значение `modCount`. При каждом вызове `next()` итератор проверяет, не изменился ли `modCount`. Если изменился — выбрасывается `ConcurrentModificationException`.

Важно: исключение бросается при структурных изменениях коллекции — добавлении или удалении элементов. При этом использование `iterator.remove()` (а не `collection.remove()`) не вызывает исключения, так как итератор сам обновляет свой `expectedModCount`.

Примеры коллекций с fail-fast итераторами: `ArrayList`, `HashMap`, `Vector`.

#### Ответ от Claude

**Что такое fail-fast итератор**

Fail-fast («падающий быстро») — итератор, который немедленно выбрасывает `ConcurrentModificationException` при обнаружении структурного изменения коллекции во время итерации. Цель — не допустить работы с коллекцией в неконсистентном состоянии.

Важная оговорка из официальной документации (`java.util.ConcurrentModificationException`, docs.oracle.com): *«Данное исключение не гарантирует выброс только при конкурентной модификации другим потоком. Оно может быть выброшено в однопоточном сценарии, если код нарушает контракт объекта»*.

**Механизм modCount**

Коллекции из `java.util` (ArrayList, HashMap, HashSet и др.) содержат поле `protected transient int modCount`, унаследованное от `AbstractList`. Поле инкрементируется при каждом **структурном изменении**: добавлении, удалении элемента, изменении размера массива и т.п.

При создании итератора его внутренний класс сохраняет снимок счётчика:

```java
// Внутри ArrayList.Itr (упрощённо)
int expectedModCount = modCount;
```

При каждом вызове `next()` выполняется проверка:

```java
final void checkForComodification() {
    if (modCount != expectedModCount)
        throw new ConcurrentModificationException();
}
```

**Безопасное удаление через iterator.remove()**

Если используется `iterator.remove()` (а не `list.remove()`), итератор сам инкрементирует `modCount` коллекции **и** обновляет `expectedModCount`:

```java
public void remove() {
    // ... удаление элемента ...
    expectedModCount = modCount; // синхронизация счётчиков
}
```

Именно поэтому `iterator.remove()` не вызывает исключения, а `collection.remove()` во время итерации — вызывает.

**В каких коллекциях применяется fail-fast**

Все стандартные коллекции из пакета `java.util`, основанные на `AbstractList` / `AbstractMap`:

- `ArrayList`, `LinkedList`, `Vector`
- `HashMap`, `LinkedHashMap`, `TreeMap`
- `HashSet`, `LinkedHashSet`, `TreeSet`

**Коллекции с fail-safe итераторами** (работают на копии данных):

- `CopyOnWriteArrayList` — итератор работает по снимку массива
- `ConcurrentHashMap` — сегментированная блокировка, без `ConcurrentModificationException`

**Пример нарушения и правильного кода**

```java
// Неправильно — выбросит ConcurrentModificationException
List<String> list = new ArrayList<>(List.of("a", "b", "c"));
for (String s : list) {
    if (s.equals("b")) list.remove(s); // структурное изменение!
}

// Правильно — через Iterator.remove()
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    if (it.next().equals("b")) it.remove(); // безопасно
}

// Правильно — через removeIf (Java 8+)
list.removeIf(s -> s.equals("b"));
```

**Ключевые факты**

- Fail-fast — это **best-effort** механизм, не гарантия. JVM не обязана выбросить исключение в каждом случае нарушения.
- `modCount` — `int`, может переполниться и «сброситься», что теоретически скроет модификацию. На практике крайне редко.
- Начиная с Java 8, для коллекций доступны методы `removeIf()`, `replaceAll()` и потоки `stream()` — они безопасны и не требуют ручного управления итератором.

### Вопрос 25. В чём разница между Iterator и ListIterator? Какие дополнительные возможности предоставляет ListIterator?

**Категория:** Collections · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [ListIterator (Java SE 21 & JDK 21) — docs.oracle.com](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/util/ListIterator.html)

`ListIterator` — это итератор для списков, который позволяет программисту обходить список в обоих направлениях, модифицировать список во время итерации и получать текущую позицию итератора. `ListIterator` не имеет текущего элемента: его позиция курсора всегда находится между элементом, возвращённым вызовом `previous()`, и элементом, возвращённым вызовом `next()`.

Дополнительные методы по сравнению с `Iterator`:
- `hasPrevious()` — возвращает `true`, если при обходе в обратном направлении есть ещё элементы
- `previous()` — возвращает предыдущий элемент и перемещает курсор назад
- `nextIndex()` — возвращает индекс элемента, который будет возвращён следующим вызовом `next()`
- `previousIndex()` — возвращает индекс элемента, который будет возвращён следующим вызовом `previous()`
- `set(E e)` — заменяет последний элемент, возвращённый `next()` или `previous()`, на указанный элемент
- `add(E e)` — вставляет элемент в список перед элементом, который был бы возвращён следующим вызовом `next()`

#### Ответ от Claude

**Iterator** — базовый интерфейс для обхода коллекций, входящий в Java Collections Framework. Предоставляет три метода:
- `hasNext()` — проверяет наличие следующего элемента
- `next()` — возвращает следующий элемент и сдвигает курсор вперёд
- `remove()` — удаляет последний возвращённый элемент из коллекции (опциональная операция)
- `forEachRemaining(Consumer)` — добавлен в Java 8, выполняет действие для каждого оставшегося элемента

Применим ко всем реализациям `Collection`: `List`, `Set`, `Queue`, `Deque` и др.

**ListIterator** расширяет `Iterator` и предназначен исключительно для `List`-реализаций. Получается через `list.listIterator()` или `list.listIterator(int index)` — второй вариант позволяет начать итерацию с заданной позиции.

Ключевые отличия и дополнительные возможности `ListIterator`:

| Возможность | Iterator | ListIterator |
|---|---|---|
| Направление обхода | только вперёд | вперёд и назад |
| Применимость | любая `Collection` | только `List` |
| Замена элемента (`set`) | нет | да |
| Вставка элемента (`add`) | нет | да |
| Информация об индексе | нет | `nextIndex()`, `previousIndex()` |

**Семантика курсора.** Курсор `ListIterator` находится между элементами: он не указывает на конкретный элемент, а располагается между соседними. Поэтому `nextIndex()` и `previousIndex()` всегда возвращают корректные значения — индексы элементов по обе стороны от курсора.

**Ограничение `set()` и `add()`.** Вызов `set()` заменяет последний элемент, возвращённый `next()` или `previous()`. Метод `add()` вставляет элемент непосредственно перед позицией курсора, не нарушая семантику продолжения итерации: вставленный элемент не будет возвращён `previous()`, но не будет возвращён и `next()` (курсор сдвигается вправо).

### Вопрос 26. Чем ConcurrentHashMap отличается от Collections.synchronizedMap(HashMap)? Как изменилась внутренняя реализация ConcurrentHashMap между Java 7 (сегментная блокировка) и Java 8 (CAS + synchronized по бакету)?

**Категория:** Collections · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [ConcurrentHashMap (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/ConcurrentHashMap.html)

`ConcurrentHashMap` поддерживает полный параллелизм операций чтения и высококонкурентный параллелизм операций записи. Операции получения данных (в том числе `get`) как правило не блокируются и могут выполняться параллельно с операциями изменения (в том числе `put` и `remove`). Операции получения данных отражают результаты последних **завершённых** операций изменения. Итераторы и разбиватели возвращают элементы, отражающие состояние таблицы в некоторый момент с момента создания итератора или позже. Они не генерируют `ConcurrentModificationException`. Агрегатные операции статуса, включая `size`, `isEmpty` и `containsValue`, как правило, полезны только тогда, когда карта не претерпевает параллельных обновлений в других потоках.

#### Ответ от Claude

**Сравнение `ConcurrentHashMap` и `Collections.synchronizedMap(HashMap)`:**

`Collections.synchronizedMap` создаёт обёртку над обычной `HashMap`, которая синхронизирует **каждый метод целиком** через единственный мьютекс (сам объект обёртки). Это означает, что в любой момент времени только один поток может выполнять любую операцию — чтение или запись. При итерации необходимо явно захватывать блокировку снаружи: `synchronized(map) { for (...) }`.

`ConcurrentHashMap` изначально проектировалась для высокой конкурентности:
- Операции чтения (`get`, `containsKey`) **не блокируются** вообще — используют `volatile`-чтение.
- Итераторы не бросают `ConcurrentModificationException`.
- Атомарные составные операции (`putIfAbsent`, `computeIfAbsent`, `replace`) доступны из коробки.
- `null`-ключи и `null`-значения **не допускаются** (в отличие от `HashMap`).

**Java 7 — сегментная блокировка (Segment-based locking):**

Внутри хранился массив объектов `Segment<K,V>`, каждый из которых наследовал `ReentrantLock` и содержал собственную хеш-таблицу. По умолчанию сегментов было 16 (задавалось параметром `concurrencyLevel`). При записи блокировался только тот сегмент, которому принадлежал ключ — это позволяло до 16 потоков писать одновременно в разные сегменты. Чтение было lock-free (volatile-чтение полей). Размер всей структуры был жёстко привязан к степени двойки, ближайшей сверху к `concurrencyLevel`.

**Java 8 — CAS + `synchronized` по бакету (Node-based locking):**

В Java 8 реализация была полностью переписана. Отдельные сегменты исчезли; вместо них:

1. **Таблица `Node<K,V>[] table`** объявлена `volatile` — видима всем потокам без блокировок.
2. **Вставка в пустой бакет** выполняется через **CAS** (Compare-And-Swap) — атомарная операция без захвата монитора.
3. **Вставка в непустой бакет** выполняется через `synchronized` блок на **голове бакета** (`synchronized (f) { ... }`) — блокируется только один бакет, а не сегмент из нескольких бакетов.
4. **Длинные цепочки** (>= 8 элементов) преобразуются в **красно-чёрные деревья** (как в `HashMap` с Java 8), при этом для дерева используется специальный узел `TreeBin`, который сохраняет стабильную точку блокировки даже при поворотах дерева.
5. **Ресайзинг** — кооперативный: несколько потоков одновременно переносят бакеты из старой таблицы в новую, каждый поток берёт себе диапазон бакетов через CAS.

Параметр `concurrencyLevel` в Java 8 является лишь подсказкой для начального размера таблицы и больше не определяет количество сегментов.

**Итог:** Если нужна потокобезопасная карта с хорошей производительностью при многопоточном доступе — используйте `ConcurrentHashMap`. `Collections.synchronizedMap` оправдан только для редкого многопоточного доступа или совместимости с легаси-кодом.

### Вопрос 27. Итераторы ConcurrentHashMap являются fail-safe. Что это означает на практике — гарантирует ли итератор, что он увидит все изменения, внесённые после его создания?

**Категория:** Collections · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [ConcurrentHashMap (Java SE 21 & JDK 21) — docs.oracle.com](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/util/concurrent/ConcurrentHashMap.html)

Итераторы, сплитераторы и перечисления возвращают элементы, отражающие состояние хеш-таблицы в некоторый момент — в момент создания итератора или после него. Они не генерируют `ConcurrentModificationException`. Однако итераторы предназначены для использования только одним потоком одновременно. Имейте в виду, что результаты агрегатных операций статуса, включая `size`, `isEmpty` и `containsValue`, как правило, полезны только тогда, когда карта не претерпевает параллельных обновлений в других потоках.

#### Ответ от Claude

**Важное терминологическое уточнение.** Официальная документация Oracle называет итераторы `ConcurrentHashMap` **weakly consistent** («слабо согласованными»), а не «fail-safe». Термин «fail-safe» — это неформальный сленг, который широко используется в обучающих материалах, но не является частью официальной спецификации Java.

**Что означает «weakly consistent» на практике:**

1. **Нет `ConcurrentModificationException`.** Итератор не отслеживает `modCount` и не бросает исключений при изменении карты во время обхода. Это принципиальное отличие от `HashMap`, итератор которого является fail-fast.

2. **Гарантия видимости на момент создания.** Итератор **гарантированно** обойдёт все элементы, существовавшие в момент его создания (при условии, что они не были удалены до момента, когда итератор до них дойдёт).

3. **НЕ гарантирует видимость изменений после создания.** Элементы, добавленные после создания итератора, **могут быть, а могут и не быть** видны в текущем обходе — это зависит от того, в каком бакете они окажутся и дошёл ли уже итератор до этого бакета.

4. **Отражение удалений непредсказуемо.** Если ключ удалён после создания итератора, но итератор ещё не прошёл соответствующий бакет, этот ключ **не будет** возвращён. Если итератор уже прошёл этот бакет — элемент уже был возвращён.

5. **Агрегатные операции не атомарны.** `putAll` и `clear` во время обхода могут быть видны частично: итератор увидит только часть добавленных/удалённых элементов.

**Практический вывод:**

Итератор `ConcurrentHashMap` НЕ является «снимком» (snapshot) состояния карты на момент создания — в отличие, например, от итераторов `CopyOnWriteArrayList`, которые работают по реальному снимку. Итератор обходит живую структуру, и его слабая согласованность означает, что для агрегатных операций (подсчёт, поиск по всем элементам) результат будет актуален «приблизительно», но не атомарно точен. Для точных агрегатов следует использовать методы `mappingCount()` вместо `size()`, либо обеспечивать внешнюю синхронизацию на уровне бизнес-логики.

### Вопрос 28. В чём разница между HashMap, LinkedHashMap и TreeMap по порядку итерации, сложности операций и допустимости null-ключей?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [LinkedHashMap (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/util/LinkedHashMap.html)

`LinkedHashMap` — это реализация хеш-таблицы и связного списка интерфейса `Map` с предсказуемым порядком итерации. Данная реализация отличается от `HashMap` тем, что поддерживает двусвязный список, связывающий все её записи. Этот связный список определяет порядок итерации — как правило, это порядок вставки ключей в карту. Обратите внимание, что повторная вставка ключа не влияет на порядок итерации. `LinkedHashMap` предоставляет константное время выполнения для базовых операций (`add`, `contains` и `remove`) при условии, что хеш-функция равномерно распределяет элементы по бакетам. Производительность ненамного ниже, чем у `HashMap`, из-за накладных расходов на поддержание связного списка, за исключением одного исключения: итерация по `LinkedHashMap` требует времени, пропорционального размеру карты, независимо от её ёмкости. Итерация по `HashMap` скорее всего обойдётся дороже и потребует времени, пропорционального ёмкости.

#### Ответ от Claude

**Порядок итерации:**

- **`HashMap`** — порядок не определён и не гарантируется. Более того, порядок может меняться при перехешировании (расширении таблицы). Нельзя рассчитывать на какой-либо устойчивый порядок.
- **`LinkedHashMap`** — поддерживает двусвязный список поверх хеш-таблицы. По умолчанию — порядок вставки (insertion-order): ключи обходятся в том порядке, в котором впервые вставлялись. Повторная вставка существующего ключа не меняет его позицию. Альтернативный режим — порядок последнего доступа (access-order), активируется конструктором `LinkedHashMap(capacity, loadFactor, true)` — полезен для реализации LRU-кеша.
- **`TreeMap`** — реализует `SortedMap`/`NavigableMap`; ключи хранятся в отсортированном порядке: по естественному порядку (`Comparable`) или по `Comparator`, переданному в конструктор. `keySet()` возвращает ключи в возрастающем порядке; `descendingKeySet()` — в убывающем.

**Сложность операций:**

| Операция | HashMap | LinkedHashMap | TreeMap |
|---|---|---|---|
| `get` / `put` / `remove` | O(1) амортизированно | O(1) амортизированно | O(log n) |
| `containsKey` | O(1) амортизированно | O(1) амортизированно | O(log n) |
| Итерация | O(capacity + size) | O(size) | O(size) |
| `firstKey` / `lastKey` | нет | нет | O(log n) |
| `floorKey` / `ceilingKey` | нет | нет | O(log n) |

`TreeMap` основан на красно-чёрном дереве и даёт гарантированное O(log n) для всех операций навигации.

**Допустимость null-ключей:**

- **`HashMap`** — допускает ровно один `null`-ключ и любое количество `null`-значений.
- **`LinkedHashMap`** — наследует поведение `HashMap`; допускает `null`-ключ и `null`-значения.
- **`TreeMap`** — **не допускает** `null`-ключ при использовании естественного порядка, так как сравнение вызывает `compareTo()`, которое бросает `NullPointerException`. При передаче кастомного `Comparator`, который явно обрабатывает `null`, поведение определяется этим компаратором.

**Когда использовать:**
- `HashMap` — когда порядок не важен и нужна максимальная скорость доступа.
- `LinkedHashMap` — когда нужен предсказуемый порядок итерации (порядок вставки) или LRU-семантика, при этом хочется сохранить O(1) для базовых операций.
- `TreeMap` — когда нужна сортировка ключей или операции навигации (`subMap`, `headMap`, `tailMap`, `floorKey`, `ceilingKey`).
### Вопрос 29. Что произойдёт, если в качестве ключа HashMap использовать изменяемый объект (mutable key), и после добавления элемента изменить поле, участвующее в hashCode()?

**Категория:** Collections · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Memory Leak Due To Mutable Keys in Java Collections — HeapHero](https://blog.heaphero.io/memory-leak-due-to-mutable-keys-in-java-collections/)

HashMap использует внутренний массив бакетов (buckets). При вставке пары ключ-значение `hashCode()` ключа определяет, в какой бакет будет помещена запись. При последующем поиске HashMap снова вычисляет `hashCode()` и только затем использует `equals()` для точного сравнения внутри бакета.

Если мутировать ключ после вставки (изменив поле, участвующее в `hashCode()`), объект начинает возвращать другое хеш-значение. Это создаёт катастрофическое несоответствие: запись физически осталась в старом бакете, но все новые обращения ведут в другой бакет.

Попытки удалить запись обоими способами — по старому и новому значению ключа — завершатся неудачей. В первом случае объект лежит в нужном бакете, но `equals()` возвращает `false` (значение поля изменилось). Во втором случае новый хеш ведёт в совершенно другой бакет, где записи нет. Такие записи становятся «зомби-объектами» — физически существуют в HashMap, но стали недостижимы. Если подобный код работает в цикле, HashMap будет расти неограниченно, что в конечном счёте приведёт к `OutOfMemoryError: Java heap space`.

#### Ответ от Claude

**Механизм хранения и поиска в HashMap** основан на неизменности хеш-кода ключа на протяжении всего его жизненного цикла в карте. При вставке (`put`) вычисляется `hashCode()` ключа, результат преобразуется в индекс бакета (`(n-1) & hash`), и запись кладётся в этот бакет. При поиске (`get`) та же операция выполняется снова — и только если хеш совпадает, HashMap просматривает цепочку в бакете с помощью `equals()`.

**Что происходит при мутации ключа:**

1. **Запись не найдена (`get` возвращает `null`).** Новый хеш ведёт в другой бакет — там записи нет.
2. **Удаление не работает (`remove` возвращает `null`).** По той же причине HashMap ищет запись не там, где она лежит.
3. **Запись «застревает» в карте как зомби-объект.** Она продолжает занимать память, но никак не может быть получена или удалена через публичный API.
4. **Потенциальная утечка памяти.** Если такие записи накапливаются в цикле, HashMap неограниченно растёт до `OutOfMemoryError`.

**Пример сценария:**
```java
class Key {
    String value;
    Key(String v) { this.value = v; }
    public int hashCode() { return value.hashCode(); }
    public boolean equals(Object o) { return o instanceof Key k && k.value.equals(value); }
}

Map<Key, String> map = new HashMap<>();
Key k = new Key("original");
map.put(k, "data");

k.value = "mutated";          // меняем поле, участвующее в hashCode()

map.get(k);                   // null — не найдено
map.containsKey(k);           // false — не найдено
map.remove(k);                // null — удалить не удалось
map.size();                   // 1 — запись по-прежнему в карте
```

**Дополнительный нюанс — поиск со старым ключом.** Если у вас остался ещё один объект с оригинальным значением `"original"`, `map.get(originalKey)` тоже вернёт `null`, потому что `equals()` сравнит его с уже мутировавшим объектом `"mutated"` в бакете — и они не совпадут.

**Правило:** ключи HashMap обязаны быть **эффективно неизменяемыми** — либо использовать immutable-класс (`String`, `Integer` и т.д.), либо гарантировать, что поля, задействованные в `hashCode()` и `equals()`, не изменяются после помещения объекта в карту (например, объявить их `final`).

### Вопрос 30. Объясните разницу между Queue и Deque. Какие методы добавления/удаления элементов предпочтительнее использовать и почему (offer/poll vs add/remove)?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Deque (Java Platform SE 8) — Oracle Docs](https://docs.oracle.com/javase/8/docs/api/java/util/Deque.html)

**Queue** — интерфейс, представляющий коллекцию для FIFO-доступа (первым вошёл — первым вышел): элементы добавляются в хвост, извлекаются из головы.

**Deque** (Double-Ended Queue) расширяет `Queue` и поддерживает добавление и удаление элементов с **обоих концов**. Deque может работать как в режиме FIFO-очереди, так и в режиме LIFO-стека.

Для каждой операции в `Queue`/`Deque` существует два варианта метода:
- Методы, **бросающие исключение** (`add`, `remove`, `element`) — предполагают, что операция должна успешно выполниться; при сбое бросают `IllegalStateException` или `NoSuchElementException`.
- Методы, **возвращающие специальное значение** (`offer`, `poll`, `peek`) — возвращают `false` или `null` при неудаче, никогда не бросают исключений.

Документация явно рекомендует: «При использовании очереди с ограниченной ёмкостью (capacity-restricted), как правило, предпочтительнее использовать `offer()`, который корректно сигнализирует о невозможности добавить элемент, возвращая `false`, вместо `add()`, который сообщает об ошибке только через исключение».

#### Ответ от Claude

**Queue vs Deque — структурная разница:**

`Queue` — линейная очередь с доступом только с одного конца каждой операции: вставка в хвост, извлечение из головы (FIFO). Основные реализации: `LinkedList`, `ArrayDeque`, `PriorityQueue`.

`Deque` (Double-Ended Queue) расширяет `Queue` и добавляет операции на **обоих концах**: `addFirst`/`offerFirst`, `addLast`/`offerLast`, `removeFirst`/`pollFirst`, `removeLast`/`pollLast`. Это позволяет использовать `Deque` и как очередь (FIFO), и как стек (LIFO). Основные реализации: `ArrayDeque`, `LinkedList`.

**Сравнительная таблица методов Queue:**

| Операция | Бросает исключение | Возвращает значение |
|---|---|---|
| Вставка | `add(e)` — `IllegalStateException` если полна | `offer(e)` — возвращает `false` если полна |
| Извлечение | `remove()` — `NoSuchElementException` если пуста | `poll()` — возвращает `null` если пуста |
| Просмотр | `element()` — `NoSuchElementException` если пуста | `peek()` — возвращает `null` если пуста |

**Когда использовать `offer`/`poll` (предпочтительнее в большинстве случаев):**
- Для очередей с ограниченной ёмкостью (`ArrayBlockingQueue`, `LinkedBlockingQueue`) — `offer` вернёт `false` без исключения, позволяя реализовать логику отклонения или ожидания.
- В конкурентных системах с обратным давлением (backpressure) — нормальная ситуация, когда очередь может быть полна или пуста.
- В коде, где пустая очередь — ожидаемое состояние (цикл обработки).

**Когда использовать `add`/`remove`:**
- Когда пустая очередь или переполнение означают программную ошибку (fail-fast поведение).
- Когда нужна немедленная диагностика через стек-трейс.

**Практическая рекомендация:** В большинстве продакшн-кода предпочтительнее `offer`/`poll` — они делают обработку граничных случаев явной без риска неперехваченных исключений. Исключение — `PriorityQueue` и `ArrayDeque` без ограничения ёмкости: для них `add` и `offer` ведут себя одинаково, `remove` и `poll` различаются только поведением на пустой коллекции.

### Вопрос 31. Как работает PriorityQueue? Какова структура данных внутри, и какова сложность операций offer() и poll()?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PriorityQueue (Java Platform SE 8) — Oracle Docs](https://docs.oracle.com/javase/8/docs/api/java/util/PriorityQueue.html)

Реализация `PriorityQueue` основана на **бинарной куче** (priority heap). Очередь с приоритетами не ограничена по размеру, но имеет внутреннюю ёмкость (initial capacity = 11 по умолчанию), управляющую размером массива для хранения элементов. По мере добавления элементов ёмкость растёт автоматически.

Голова очереди — наименьший элемент согласно заданному упорядочиванию. Если наименьших элементов несколько (при равном приоритете), голова выбирается произвольно.

Документация явно указывает сложность операций:

> «Данная реализация обеспечивает O(log n) время для методов постановки и извлечения из очереди (`offer`, `poll`, `remove()` и `add`); линейное время для методов `remove(Object)` и `contains(Object)`; и константное время для методов извлечения без удаления (`peek`, `element` и `size`).»

#### Ответ от Claude

**Внутренняя структура данных:**

`PriorityQueue` реализована на основе **бинарной min-кучи** (binary min-heap), хранящейся в виде массива `Object[] queue`. Бинарная куча — это полное двоичное дерево, в котором каждый родитель не превышает своих потомков (свойство кучи). Хранение в массиве позволяет вычислять позиции по формулам:
- Левый потомок элемента в позиции `i`: `2*i + 1`
- Правый потомок: `2*i + 2`
- Родитель: `(i - 1) / 2`
- Корень (минимальный элемент): индекс `0`

По умолчанию используется **естественный порядок** (natural ordering — `Comparable`), что делает её min-heap. Для реализации max-heap или произвольного порядка передаётся `Comparator`.

**Сложность операций:**

| Операция | Сложность | Описание |
|---|---|---|
| `offer(e)` / `add(e)` | O(log n) | Элемент добавляется в конец массива, затем «всплывает» вверх (sift up) до нужной позиции |
| `poll()` / `remove()` | O(log n) | Корень (минимум) извлекается, последний элемент ставится на его место и «тонет» вниз (sift down) |
| `peek()` / `element()` | O(1) | Возврат элемента с индекса 0 без удаления |
| `remove(Object)` | O(n) | Требует линейного поиска по массиву |
| `contains(Object)` | O(n) | Линейный поиск |
| `size()` | O(1) | Поле `size` |

**Как работает `offer(e)` — sift up:**
1. Элемент добавляется в позицию `size` (конец массива).
2. Сравнивается с родителем `(i-1)/2`.
3. Если меньше родителя — меняется местами с ним; процесс повторяется.
4. Останавливается, когда элемент не меньше родителя или достигнут корень.
5. В худшем случае нужно пройти высоту дерева — O(log n).

**Как работает `poll()` — sift down:**
1. Корень (минимум) сохраняется для возврата.
2. Последний элемент массива перемещается в корень.
3. Сравнивается с меньшим из потомков.
4. Если больше меньшего потомка — меняется с ним; процесс повторяется.
5. Останавливается, когда элемент не больше потомков или достигнут лист.
6. В худшем случае — O(log n).

**Важные ограничения:**
- Не допускает `null`-элементов.
- Не гарантирует сортированного порядка при итерации (итератор не обходит кучу по порядку).
- Не является потокобезопасной; для многопоточности используется `PriorityBlockingQueue`.

### Вопрос 32. Что такое TREEIFY_THRESHOLD, UNTREEIFY_THRESHOLD и MIN_TREEIFY_CAPACITY в HashMap? Зачем введено ограничение MIN_TREEIFY_CAPACITY = 64?

**Категория:** Collections · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [openjdk-jdk11u/HashMap.java — AdoptOpenJDK на GitHub](https://github.com/AdoptOpenJDK/openjdk-jdk11u/blob/master/src/java.base/share/classes/java/util/HashMap.java)

Исходный код OpenJDK содержит следующие константы с комментариями:

**`TREEIFY_THRESHOLD = 8`** — «Пороговое количество элементов в бакете для перехода от связного списка к дереву. Бакеты преобразуются в деревья при добавлении элемента в бакет, содержащий не менее указанного числа узлов. Значение должно быть больше 2 и не менее 8, чтобы соответствовать предположениям, заложенным в логику удаления из дерева при обратном преобразовании в обычные бакеты».

**`UNTREEIFY_THRESHOLD = 6`** — «Пороговое количество элементов в бакете для обратного преобразования дерева в список при операции resize. Должно быть меньше `TREEIFY_THRESHOLD` и не более 6, чтобы соответствовать логике обнаружения уменьшения размера при удалении».

**`MIN_TREEIFY_CAPACITY = 64`** — «Минимальная ёмкость таблицы, при которой бакеты могут быть преобразованы в деревья. Иначе, если в бакете слишком много узлов, таблица увеличивается через resize. Должно быть не менее `4 * TREEIFY_THRESHOLD`, чтобы исключить конфликты между порогами resize и treeification».

#### Ответ от Claude

В Java 8 бакеты HashMap могут хранить элементы в виде **связного списка** или **красно-чёрного дерева** (Red-Black Tree). Три константы управляют переключением между этими структурами.

**`TREEIFY_THRESHOLD = 8`**

Когда в одном бакете накапливается 8 элементов (коллизии), связный список преобразуется в красно-чёрное дерево. Это улучшает сложность операций в данном бакете с O(n) до O(log n), что критично при плохой функции хеширования или атаке на хеш-функцию (hash flooding). Значение 8 выбрано на основании анализа вероятности: при случайном распределении вероятность попасть в 8 элементов одного бакета составляет около 0.00000006 — порог, при котором накладные расходы на дерево оправданы.

**`UNTREEIFY_THRESHOLD = 6`**

При операции `resize` (расширении таблицы) элементы перераспределяются по бакетам. Если после перераспределения в бакете-дереве осталось 6 или менее элементов, дерево обратно конвертируется в связный список. Значение **намеренно ниже 8** (гистерезис) — это предотвращает **флаппинг**: ситуацию, при которой добавление и удаление одного элемента вокруг порога вызывало бы постоянное дорогостоящее переключение между структурами. Связный список потребляет меньше памяти, чем `TreeNode` (дерево использует `TreeNode`, который вдвое тяжелее обычного `Node`).

**`MIN_TREEIFY_CAPACITY = 64`**

Это условие-предохранитель: treeification происходит **только** при ёмкости таблицы ≥ 64. Если таблица меньше — вместо преобразования в дерево вызывается `resize()`.

**Зачем именно 64?** Комментарий в исходнике гласит: «должно быть не менее `4 * TREEIFY_THRESHOLD`» — то есть не менее 32. Значение 64 выбрано с запасом. Логика следующая:

- При маленькой таблице (< 64 ячеек) коллизии в 8+ элементов — скорее всего, следствие нехватки бакетов, а не плохого хеширования. Решение: **расширить таблицу**, а не усложнять структуру бакета.
- После `resize` таблица удваивается, элементы разбрасываются по двум бакетам-потомкам, что естественным образом сокращает число коллизий.
- Если начать treeification при малой таблице, сразу после `resize` бакеты опустеют ниже `UNTREEIFY_THRESHOLD` и придётся снова делать обратное преобразование — двойные накладные расходы.
- При таблице ≥ 64 ячеек 8+ элементов в одном бакете — признак реальной проблемы хеширования, и дерево оправдано.

**Взаимодействие трёх констант в одной схеме:**

```
Размер таблицы < 64         Размер таблицы >= 64
        |                           |
  бакет >= 8 элементов       бакет >= 8 элементов
        |                           |
    resize()                  treeify() → Red-Black Tree
                                    |
                              после resize/remove:
                              бакет <= 6 элементов
                                    |
                              untreeify() → LinkedList
```

**Итог:** Три константы совместно реализуют адаптивную стратегию: сначала расширять таблицу (дёшево и эффективно), а дерево использовать только как крайнюю меру для действительно перегруженных бакетов в достаточно большой таблице.
### Вопрос 33. Можно ли положить null-ключ в HashMap, Hashtable, ConcurrentHashMap и TreeMap? Объясните поведение каждой коллекции.

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Oracle Java SE 8 Javadoc — HashMap](https://docs.oracle.com/javase/8/docs/api/java/util/HashMap.html)

Официальная документация HashMap явно указывает: «Данная реализация предоставляет все опциональные операции Map и допускает null-значения и null-ключ. (Класс HashMap грубо эквивалентен Hashtable, за исключением того, что он не синхронизирован и допускает null.)»

Для метода `get()` документация уточняет: «Возврат null не обязательно означает отсутствие маппинга для данного ключа — вполне возможно, что маппинг явно задаёт ключ на null. Для различия этих двух случаев следует использовать операцию `containsKey`.»

Hashtable (тот же источник серии Javadoc, [Hashtable](https://docs.oracle.com/javase/8/docs/api/java/util/Hashtable.html)) явно запрещает null: «Ни ключ, ни значение не могут быть null» — метод `put()` бросает `NullPointerException`, если ключ или значение равны null.

ConcurrentHashMap ([Javadoc](https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/ConcurrentHashMap.html)) аналогично запрещает null: «Подобно Hashtable, но в отличие от HashMap, данный класс не допускает использование null в качестве ключа или значения.»

TreeMap ([Javadoc](https://docs.oracle.com/javase/8/docs/api/java/util/TreeMap.html)) при использовании естественного порядка бросает `NullPointerException` при попытке добавить null-ключ.

#### Ответ от Claude

**HashMap** допускает ровно один null-ключ и произвольное количество null-значений. Null-ключ обрабатывается особым образом: вместо вызова `hashCode()` (который выбросил бы NPE) HashMap помещает запись в корзину с индексом 0 через специальный внутренний путь. Поскольку `get(null)` может вернуть null как по причине отсутствия ключа, так и по причине явного маппинга в null, для разграничения этих случаев нужно использовать `containsKey(null)`.

**Hashtable** не допускает ни null-ключей, ни null-значений. Любая попытка передать null в `put()`, `get()`, `containsKey()` и другие методы немедленно бросает `NullPointerException`. Это историческое решение унаследовано с времён Java 1.0 — Hashtable синхронизирован и позиционировался как строго типизированное хранилище.

**ConcurrentHashMap** также запрещает null и для ключей, и для значений. Причина — семантическая: в конкурентной среде `get(key)` возвращающий null не позволяет определить, был ли ключ явно смаплен в null другим потоком, или же его вообще нет в карте. В однопоточном HashMap эту неоднозначность можно устранить последующим вызовом `containsKey()`, но между `get()` и `containsKey()` в многопоточной среде состояние карты может измениться, что делает проверку бессмысленной. Запрет null устраняет неоднозначность: если `get()` вернул null — ключ отсутствует, точка.

**TreeMap** хранит ключи в отсортированном порядке (Red-Black дерево) и при каждой операции вставки/поиска вызывает `compareTo()` или компаратор. Сравнение с null невозможно — естественная сортировка (`Comparable`) не поддерживает null, что приводит к `NullPointerException`. Обходной путь: передать кастомный `Comparator`, который явно обрабатывает null, например `Comparator.nullsFirst(Comparator.naturalOrder())`. В этом случае TreeMap сможет хранить null-ключи.

**Сводная таблица:**

| Коллекция | null-ключ | null-значение | Исключение |
|---|---|---|---|
| HashMap | Да (один) | Да (много) | — |
| Hashtable | Нет | Нет | NullPointerException |
| ConcurrentHashMap | Нет | Нет | NullPointerException |
| TreeMap (natural order) | Нет | Да | NullPointerException |
| TreeMap (custom comparator) | Зависит от компаратора | Да | Зависит от компаратора |

### Вопрос 34. HashMap не является потокобезопасным. Какие конкретно проблемы могут возникнуть при конкурентном доступе — в частности, что происходило в Java 7 при одновременном rehashing из нескольких потоков (infinite loop)?

**Категория:** Collections · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [JavaByPatel — Infinite loop in HashMap](https://javabypatel.blogspot.com/2016/01/infinite-loop-in-hashmap.html)

Проблема бесконечного цикла в HashMap возникает при одновременном rehash'инге из нескольких потоков. В Java 7 метод `transfer()`, выполняющий перемещение элементов из старой таблицы в новую, вставляет каждый узел в начало цепочки (head insertion), что разворачивает порядок узлов. Если два потока одновременно выполняют `transfer()`, может сформироваться циклический связный список: Thread 1 получает ссылки на узлы A → B, Thread 2 завершает свой `transfer()` и разворачивает порядок в B → A, после чего Thread 1 возобновляет работу и в результате своих операций устанавливает A.next = B и B.next = A — образуется цикл. Любой последующий вызов `get()`, попадающий в этот bucket, войдёт в бесконечный цикл при обходе цепочки.

#### Ответ от Claude

**Проблемы конкурентного доступа к HashMap в целом:**

1. **Потеря данных при конкурентной записи (lost update).** Если два потока одновременно вызывают `put()` и оба попадают в одну корзину, один из них может перезаписать результат работы другого. Запись, сделанная одним потоком, окажется потеряна.

2. **Гонка при проверке размера и расширении.** HashMap расширяется, когда количество элементов превышает `capacity * loadFactor` (по умолчанию 0.75). Если два потока одновременно обнаружат, что порог превышен, оба запустят rehash, что приведёт к дублированию элементов или их потере.

3. **Видимость записей.** Из-за отсутствия барьеров памяти записи одного потока могут быть не видны другому (проблема кешей процессоров), поэтому `get()` может не найти только что добавленный ключ.

4. **Бесконечный цикл (только Java 7) — главная проблема:**

В Java 7 при вставке в корзину с коллизией HashMap использовал **head insertion** — каждый новый узел помещался в начало цепочки. При rehash'инге метод `transfer()` обходил старую цепочку и добавлял узлы в новую таблицу тем же способом, что разворачивало их порядок.

**Пошаговое возникновение цикла:**

Допустим, в одной корзине находятся узлы: `A → B → null`.

- **Поток T1** начинает `transfer()`. Он берёт указатель `e = A`, `e.next = B`. Планировщик переключает CPU на T2, и T1 приостанавливается.
- **Поток T2** полностью завершает rehash. Из-за head insertion порядок в новой таблице становится `B → A → null`.
- **Поток T1** возобновляется. Его локальная переменная `e` всё ещё указывает на узел A, а `next` — на узел B. T1 помещает A в начало новой цепочки: цепочка становится `A → null`. Затем T1 переходит к узлу B. Он тоже помещает B в начало: `B → A → null`. Но затем T1 хочет обработать `A.next` — а это уже узел A сам (потому что T2 уже установил `A.next = null`, а T1 снова переставил его в голову). В итоге T1 устанавливает `A.next = B`, а `B.next` уже указывает на A — получается цикл: `A ↔ B`.

После этого любой `get()` по ключу, хэш которого попадает в эту корзину, будет бесконечно обходить `A → B → A → B → ...`.

**Почему в Java 8 бесконечный цикл исчез?** В Java 8 HashMap при rehash'инге сохраняет порядок элементов (не разворачивает цепочку), используя отдельные указатели на «голову» и «хвост» (`loHead/loTail`, `hiHead/hiTail`). Это устранило возможность формирования циклического списка. Однако другие проблемы конкурентного доступа (потеря данных, некорректный счётчик размера) остались, поэтому HashMap по-прежнему непригоден для многопоточного использования без внешней синхронизации.

**Решение:** использовать `ConcurrentHashMap`, либо оборачивать HashMap в `Collections.synchronizedMap()` (с заметно меньшей производительностью из-за глобальной блокировки).

### Вопрос 35. Почему для операций get() в ConcurrentHashMap не требуется блокировка начиная с Java 8? Как достигается согласованное чтение?

**Категория:** Collections · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Oracle Java SE 8 Javadoc — ConcurrentHashMap](https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/ConcurrentHashMap.html)

Официальная документация: «Операции чтения (включая `get`) в целом не блокируются, поэтому могут перекрываться с операциями обновления (включая `put` и `remove`). Операции чтения отражают результаты последних **завершённых** операций обновления, начавшихся до начала чтения. (Более формально: операция обновления для данного ключа несёт отношение _happens-before_ с любой (не-null) операцией чтения для того же ключа, возвращающей обновлённое значение.)»

#### Ответ от Claude

**Архитектурные изменения в Java 8**

В Java 7 ConcurrentHashMap делился на сегменты (`Segment[]`), каждый из которых являлся отдельным `ReentrantLock`. Чтение (`get()`) в Java 7 также не требовало захвата блокировки — это обеспечивалось через `volatile`-поля, — но сама архитектура была сложнее.

В Java 8 сегменты были полностью убраны. Вместо них — единый массив корзин `Node<K,V>[] table` и тонкогранульная блокировка на уровне каждой корзины (синхронизируется только первый узел корзины при записи), плюс CAS-операции для вставки в пустую корзину.

**Почему get() не нуждается в блокировке**

Основа — `volatile`-семантика и отношение happens-before из Java Memory Model (JMM):

1. **`volatile`-массив таблицы.** Ссылка на таблицу корзин объявлена как `volatile Node<K,V>[] table`. Любое чтение этого поля гарантированно увидит актуальное состояние после последней записи.

2. **`volatile`-поля узла.** Внутренний класс `Node<K,V>` содержит поля:
   ```java
   volatile V val;
   volatile Node<K,V> next;
   ```
   `volatile`-запись в `val` при `put()` устанавливает happens-before с `volatile`-чтением `val` при `get()`. Это гарантирует, что читающий поток увидит значение, которое писавший поток сохранил до своей записи.

3. **`tabAt()` через Unsafe.** Для чтения элемента массива по индексу используется:
   ```java
   static final <K,V> Node<K,V> tabAt(Node<K,V>[] tab, int i) {
       return (Node<K,V>) U.getObjectVolatile(tab, ((long)i << ASHIFT) + ABASE);
   }
   ```
   `getObjectVolatile` выполняет volatile-чтение элемента массива. Массивы в Java не делают элементы volatile автоматически, поэтому Unsafe используется явно.

**Схема работы `get()`:**

```
1. Вычислить хэш ключа → spread(key.hashCode())
2. tabAt(table, hash & (n-1)) — volatile-чтение головного узла корзины
3. Проверить: hash == head.hash && key.equals(head.key)
4. Если нет — обходить цепочку через node.next (volatile)
5. Если узел — TreeBin (при деревьях), вызвать find() без блокировки
6. Вернуть node.val (volatile-чтение)
```

**Что именно гарантирует JMM:**

Когда поток-писатель завершает `put()` и публикует узел через `volatile`-запись (`setTabAt()` / `U.putObjectVolatile()`), все предшествующие операции записи (включая установку `val` и `next`) становятся видимы любому потоку, который после этого момента выполнит соответствующее `volatile`-чтение. Именно это гарантирует документация фразой «reflects results of the most recently _completed_ update operations».

**Важная оговорка:** «согласованное чтение» означает, что `get()` увидит результат любого **завершённого** `put()`. Если запись идёт параллельно прямо сейчас, `get()` может увидеть как старое, так и новое значение — оба варианта корректны. ConcurrentHashMap не даёт гарантий линеаризуемости между несколькими ключами в рамках одной «транзакции».

### Вопрос 36. Чем WeakHashMap отличается от обычного HashMap? В каких сценариях его применяют, и как сборщик мусора взаимодействует с weak-ссылками на ключи?

**Категория:** Collections · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Oracle Java SE 8 Javadoc — WeakHashMap](https://docs.oracle.com/javase/8/docs/api/java/util/WeakHashMap.html)

«Запись в WeakHashMap будет автоматически удалена, когда её ключ перестанет находиться в «обычном» использовании. Точнее: наличие маппинга для данного ключа не помешает ключу быть отброшенным сборщиком мусора, то есть стать финализируемым, финализированным и затем восстановленным. Когда ключ отброшен, его запись эффективно удаляется из карты.»

Документация также предупреждает о нарушении стандартных инвариантов Map: «Поскольку сборщик мусора может отбросить ключи в любой момент, WeakHashMap может вести себя так, будто некий неизвестный поток молча удаляет записи. В частности, даже при синхронизации возможно: метод `size()` возвращает меньшие значения со временем; `isEmpty()` возвращает false, затем true; `containsKey()` — true, затем false для одного ключа; `get()` — значение, затем null для одного ключа.»

Относительно значений: «Объекты-значения в WeakHashMap хранятся посредством обычных сильных ссылок. Поэтому следует избегать ситуации, когда значения прямо или косвенно ссылаются на свои же ключи — это предотвращало бы их сборку.»

#### Ответ от Claude

**Принцип работы: сильные vs слабые ссылки**

В обычном `HashMap` ключи хранятся через **сильные ссылки (strong references)**. Пока объект является ключом в HashMap, у него есть хотя бы одна сильная ссылка — из самой карты. GC не может его удалить, даже если нигде в прикладном коде на него больше нет ссылок. Это потенциальный источник утечек памяти.

В `WeakHashMap` каждый ключ обёрнут в `WeakReference`. Слабая ссылка не засчитывается GC как «живая» ссылка. Как только ключ не имеет **сильных** ссылок извне карты, GC вправе его удалить.

**Механизм взаимодействия с GC через ReferenceQueue:**

```
Когда GC собирает ключ-объект:
  1. Соответствующий WeakReference помещается в ReferenceQueue (expungeQueue)
  2. При следующем обращении к WeakHashMap (put, get, size, итерация и др.)
     вызывается метод expungeStaleEntries()
  3. expungeStaleEntries() опрашивает ReferenceQueue
  4. Для каждого собранного ключа соответствующая запись (ключ + значение)
     удаляется из внутренней таблицы
```

Таким образом, запись не исчезает мгновенно в момент сборки ключа GC — она удаляется **лениво** при следующей операции с картой. До этого момента `size()` может возвращать «завышенное» значение.

**Ключевые отличия от HashMap:**

| Характеристика | HashMap | WeakHashMap |
|---|---|---|
| Тип ссылки на ключ | Сильная | Слабая (WeakReference) |
| Удаление записей | Только явное (remove) | Автоматически после сборки ключа GC |
| Предсказуемость size() | Стабильна | Может уменьшаться сама по себе |
| Потокобезопасность | Нет | Нет (нужна внешняя синхронизация) |
| Допустимость null-ключа | Да (один) | Да (один, обрабатывается особо) |
| Производительность | Выше | Чуть ниже из-за WeakReference-обёрток |

**Важная оговорка о значениях:** Значения в WeakHashMap хранятся через **сильные ссылки**. Если значение содержит обратную ссылку на ключ (например, `map.put(key, new Holder(key))`), то GC не сможет собрать ключ — сильная ссылка на ключ из значения предотвращает сборку. Для обхода этого паттерна значение тоже нужно оборачивать в `WeakReference`.

**Сценарии применения:**

1. **Кэши, привязанные к жизненному циклу объектов-ключей.** Классический пример из «Effective Java» Джошуа Блоха: кэш, где записи должны жить ровно столько, сколько живёт ключ снаружи. Как только клиентский код «забывает» объект-ключ (убирает на него сильные ссылки), запись в кэше тоже исчезает.

2. **Canonicalizing mappings (канонизирующие отображения).** Хранение дополнительных атрибутов для объектов без вмешательства в их класс: `WeakHashMap<Connection, Metadata>` — когда соединение закрывается и все ссылки на него пропадают, метаданные удаляются автоматически.

3. **Слушатели и обработчики событий.** Хранение слушателей так, чтобы они не удерживались в памяти после того, как подписчик стал недостижим.

**Ограничения:** WeakHashMap не подходит как кэш общего назначения (для этого лучше `SoftReference`-based кэши — soft-ссылки держатся, пока не кончается память). WeakHashMap не является потокобезопасным — для многопоточного использования нужна внешняя синхронизация или альтернативы вроде `Collections.synchronizedMap(new WeakHashMap<>())` (но это не устраняет гонку при итерации).
### Вопрос 37. Почему ёмкость (capacity) HashMap всегда является степенью двойки? Как это связано с вычислением индекса корзины через битовое AND вместо операции деления по модулю?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [The power of the "Power of Two" — 0equalsTo1 on Medium](https://0equalsto1.medium.com/the-power-of-the-power-of-two-c5720e240411)

HashMap поддерживает ёмкость таблицы равной степени двойки, чтобы заменить дорогостоящую операцию взятия остатка от деления (`%`) быстрой битовой операцией `&`. Это возможно благодаря математическому свойству: `a % m == a & (m - 1)` — но только когда `m` является степенью двойки.

**Почему степень двойки?**

Когда ёмкость `n` — степень двойки, то `n - 1` в двоичном представлении имеет все младшие биты, установленные в 1. Например:
- ёмкость 16: `n - 1 = 15 = 0b00001111`
- ёмкость 32: `n - 1 = 31 = 0b00011111`

Операция `(n - 1) & hash` выбирает ровно столько младших бит хэша, сколько нужно для индексирования, и делает это за один такт процессора. Операция деления по модулю требует итеративного вычитания и сдвигов — десятки тактов.

**Дополнительные преимущества:**
- Битовое AND не даёт отрицательных результатов — нет проблем с отрицательными хэш-кодами.
- Удвоение ёмкости при resize — это простой сдвиг влево (`n << 1`), а не умножение.
- Равномерное распределение: маска `(n-1)` позволяет большему числу бит хэш-кода влиять на итоговый индекс.

**Пример:** С ёмкостью 16 и хэшем 3373707:
```
(16 - 1) & 3373707 = 15 & 3373707 = 11
```
Это тот же результат, что и `3373707 % 16 = 11`, но вычислен быстрее.

Константа `DEFAULT_INITIAL_CAPACITY` определена именно так, чтобы подчеркнуть побитовую природу:
```java
static final int DEFAULT_INITIAL_CAPACITY = 1 << 4; // 16
```

#### Ответ от Claude

Ёмкость HashMap **всегда является степенью двойки** по единственной причине: это позволяет заменить медленное деление по модулю быстрой битовой операцией AND при вычислении индекса корзины.

**Математическая основа**

Для любого целого `a` и степени двойки `m` выполняется равенство:
```
a % m  ==  a & (m - 1)
```
Это работает только при `m = 2^k`. При такой ёмкости `m - 1` — это бинарная маска, у которой установлены ровно `k` младших бит. Операция AND с этой маской выбирает `k` младших бит хэша — что математически эквивалентно остатку от деления на `m`.

**Реализация в HashMap**

```java
// Вычисление индекса корзины (из исходников OpenJDK):
tab[i = (n - 1) & hash]
```

где `n` — текущая ёмкость таблицы (всегда степень двойки).

Например, при ёмкости 16 (`n = 16`):
- `n - 1 = 15 = 0b00001111`
- Любой хэш, прошедший AND с этой маской, даёт число от 0 до 15

**Как HashMap обеспечивает степень двойки**

Метод `tableSizeFor(cap)` округляет любое переданное значение вверх до ближайшей степени двойки:

```java
static final int tableSizeFor(int cap) {
    int n = -1 >>> Integer.numberOfLeadingZeros(cap - 1);
    return (n < 0) ? 1 : (n >= MAXIMUM_CAPACITY) ? MAXIMUM_CAPACITY : n + 1;
}
```

Если пользователь задаёт `new HashMap<>(10)`, реальная начальная ёмкость будет 16 (следующая степень двойки).

**Производительность**

Операция деления по модулю реализуется через целочисленное деление и умножение — это многотактная операция. Битовое AND выполняется за один такт непосредственно аппаратными схемами процессора. На миллиардах операций `get`/`put` это даёт ощутимый выигрыш.

**Связь с resize**

Когда таблица увеличивается вдвое, новая ёмкость — тоже степень двойки. Это позволяет применить элегантный трюк при перераспределении элементов (см. Вопрос 2): не нужно пересчитывать полный индекс каждого элемента — достаточно одной битовой проверки.

### Вопрос 38. Как HashMap в Java 8 оптимизирует перераспределение элементов при resize? Объясните трюк с одним битом (high/low split): почему достаточно проверить `(hash & oldCapacity) == 0`, чтобы определить новую позицию элемента?

**Категория:** Collections · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Deepak Vadgama — Java HashMap Internals](https://deepakvadgama.com/blog/java-hashmap-internals/)

При удвоении ёмкости таблицы новый индекс элемента вычисляется как `hash & (newCapacity - 1)`. Поскольку `newCapacity = 2 * oldCapacity`, маска расширяется ровно на один старший бит. Java 8 использует это свойство следующим образом:

- Если у элемента этот новый бит в хэше равен 0 — элемент остаётся на **старом** индексе.
- Если этот бит равен 1 — элемент перемещается на индекс `oldIndex + oldCapacity`.

Проверку этого единственного бита делает выражение `(e.hash & oldCap) == 0`. Здесь `oldCap` сам является степенью двойки, то есть в его двоичном представлении ровно один бит установлен в 1 — именно тот, который добавляется к маске при расширении.

Код из исходников HashMap:

```java
if ((e.hash & oldCap) == 0) {
    if (loTail == null) loHead = e;
    else loTail.next = e;
    loTail = e;
} else {
    if (hiTail == null) hiHead = e;
    else hiTail.next = e;
    hiTail = e;
}
```

Элементы группируются в два списка (lo и hi) прямо в исходном порядке, а затем помещаются целым списком в новую ячейку:
```java
if (loTail != null) { loTail.next = null; newTab[j] = loHead; }
if (hiTail != null) { hiTail.next = null; newTab[j + oldCap] = hiHead; }
```

#### Ответ от Claude

**Контекст: что происходит при resize**

Когда число элементов превышает `capacity * loadFactor`, HashMap создаёт новый массив вдвое большего размера и перераспределяет все элементы. До Java 8 для каждого элемента заново вычислялся индекс: `hash & (newCapacity - 1)`. Java 8 вводит оптимизацию, которая позволяет избежать этого.

**Математическое объяснение трюка**

Пусть старая ёмкость `oldCap = 16` (двоичное: `10000`), новая ёмкость `newCap = 32`.

Старая маска: `oldCap - 1 = 15 = 0b01111` (4 бита)
Новая маска: `newCap - 1 = 31 = 0b11111` (5 бит)

Разница — ровно один бит: бит номер 4 (значение 16 = `oldCap`).

Значит, новый индекс элемента отличается от старого только этим одним битом:
- Если бит 4 в хэше равен 0: `newIndex = oldIndex` (элемент остаётся на месте)
- Если бит 4 в хэше равен 1: `newIndex = oldIndex + oldCap` (элемент сдвигается)

Проверить бит 4 хэша можно прямо так:
```java
(e.hash & oldCap) == 0   // oldCap = 0b10000, маска ровно на нужный бит
```

**Пример**

Хэш элемента: `...1_0011` (в двоичном)

При `oldCap = 16`:
- `hash & (16 - 1)` = `...0011` & `01111` = `0011` = **3** (старый индекс)

При `newCap = 32`:
- `hash & (32 - 1)` = `...10011` & `11111` = `10011` = **19** (= 3 + 16)
- Новый бит (5-й) = 1, значит `(hash & 16) != 0` → переходит в список hi

Для хэша `...0_0011`:
- Новый индекс = `0011` = **3** (тот же)
- `(hash & 16) == 0` → остаётся в списке lo

**Что это даёт**

1. **Не нужно пересчитывать полный индекс** для каждого элемента — только одна битовая операция AND.
2. **Порядок элементов в списках сохраняется** — lo-список и hi-список формируются в исходном порядке обхода. (Важное отличие от Java 7, где порядок переворачивался, что могло приводить к бесконечным циклам при конкурентном доступе.)
3. **Равномерное распределение** — при равномерном распределении хэшей примерно половина элементов остаётся на месте, половина — перемещается.

**Соответствие с деревьями (TreeNode)**

Аналогичная логика применяется и для корзин-деревьев в методе `TreeNode.split()`: дерево разбивается на lo/hi по тому же критерию `(e.hash & bit) == 0`. Если в получившейся части меньше `UNTREEIFY_THRESHOLD` (6) элементов, она разворачивается обратно в связный список.

### Вопрос 39. Как работает внутренняя хэш-функция HashMap — `(h = key.hashCode()) ^ (h >>> 16)`? Зачем выполняется XOR хэша с его правым сдвигом на 16 бит, и что бы изменилось без этой операции?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Deepak Vadgama — Java HashMap Internals](https://deepakvadgama.com/blog/java-hashmap-internals/)

Проблема, которую решает эта функция: при типичных размерах HashMap (16, 32, 64 и т.д.) для вычисления индекса используются **только младшие биты** хэш-кода. Если у ключей плохое распределение в младших битах (а верхняя часть хэша — разная), элементы всё равно попадают в одну корзину.

Решение — «перемешать» старшие биты в младшие перед вычислением индекса:

```java
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

`h >>> 16` сдвигает хэш вправо на 16 позиций (беззнаковый сдвиг), помещая старшие 16 бит на место младших. XOR с оригинальным хэшем смешивает биты обеих половин. В результате и верхние, и нижние биты исходного хэш-кода влияют на итоговое значение.

**Пример с Float-числами.** Константы 2.0f, 18.0f, 215.0f имеют одинаковые младшие биты в своём IEEE 754 представлении — без перемешивания они бы коллидировали. После применения XOR-сдвига различия из старших бит проникают в младшие — коллизий нет.

Разработчики выбрали именно этот простой (и быстрый) способ перемешивания, поскольку HashMap — сердцевинная структура данных, вызываемая миллиарды раз. Более дорогие хэш-функции дали бы лучшее распределение, но неоправданно увеличили бы накладные расходы.

#### Ответ от Claude

**Контекст: почему недостаточно просто вызвать `hashCode()`**

Индекс корзины вычисляется как `(n - 1) & hash`, где `n` — ёмкость таблицы. При типичных небольших значениях `n` (16, 32, 64) маска `n - 1` охватывает только 4–6 младших бит. Старшие 16–28 бит хэш-кода при вычислении индекса полностью игнорируются.

Если хэш-коды ключей различаются только в старших битах (что реально для `Float`, `Double`, `Integer`, а также объектов с адресом памяти в основе), все они попадут в одну корзину — производительность падает до O(n) или O(log n).

**Что делает функция**

```java
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

1. `h = key.hashCode()` — получает оригинальный 32-битный хэш.
2. `h >>> 16` — беззнаковый сдвиг вправо на 16 бит: старшие 16 бит перемещаются на место младших 16 бит, старшие 16 становятся нулями.
3. `h ^ (h >>> 16)` — XOR оригинала со сдвинутой копией. В результате:
   - Младшие 16 бит результата = `(оригинальные младшие 16) XOR (оригинальные старшие 16)`
   - Старшие 16 бит результата = оригинальные старшие 16 (без изменений)

**Визуализация (32-битный хэш)**

```
Оригинал h:       AAAA AAAA AAAA AAAA  BBBB BBBB BBBB BBBB
h >>> 16:         0000 0000 0000 0000  AAAA AAAA AAAA AAAA
h ^ (h >>> 16):   AAAA AAAA AAAA AAAA  ABAB ABAB ABAB ABAB
                  ^^^^ старшие (не изменились) ^^^^  ^^^^ младшие = A XOR B ^^^^
```

Теперь младшие биты несут в себе информацию как из нижней, так и из верхней половины исходного хэша. При любом разумном размере таблицы различия в старших битах ключей будут влиять на распределение по корзинам.

**Что было бы без этой операции**

Допустим, есть 4 ключа с хэш-кодами:
```
0x00010001  →  младшие 4 бита: 0001
0x00020001  →  младшие 4 бита: 0001
0x00030001  →  младшие 4 бита: 0001
0x00040001  →  младшие 4 бита: 0001
```
Все четыре попадут в корзину 1 при ёмкости 16 — коллизия. После применения `^ (h >>> 16)` их индексы рассредоточатся по разным корзинам.

**Почему сдвиг именно на 16 бит**

int в Java — 32-битное число. Сдвиг на половину (16 бит) — оптимальный компромисс: одной операцией «складываются» обе половины числа. Сдвиг на 8 или 24 бита перемешал бы биты хуже. Сдвиг на 16 бит гарантирует, что каждый из оригинальных 32 бит хэша оказывает влияние на индекс корзины.

**Почему XOR, а не AND или OR**

- AND: обнуляет биты (теряет информацию из одной из половин).
- OR: устанавливает биты в 1 (теряет информацию).
- XOR: сохраняет максимум информации из обеих половин, при этом является инволюцией (обратим), не даёт систематического смещения в сторону 0 или 1.

**Расхождений между источниками не обнаружено.** Все рассмотренные источники (Baeldung, Medium, deepakvadgama.com, 0equalsto1.medium.com) единодушно описывают одну и ту же логику. Различается только глубина примеров.

### Вопрос 40. Как устроен EnumMap внутри? Почему он быстрее HashMap для ключей-перечислений, и какие ограничения на ключи он накладывает?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [EnumMap (Java SE 17 & JDK 17) — docs.oracle.com](https://docs.oracle.com/en/java/javase/17/docs/api/java.base/java/util/EnumMap.html)

Enum maps представлены внутренне как массивы. Такое представление крайне компактно и эффективно. Все базовые операции выполняются за константное время. Вероятно (хотя и не гарантировано), что они выполняются быстрее своих аналогов из HashMap.

Ключи перечисления хранятся в естественном порядке (порядке объявления констант enum). Итераторы коллекционных представлений слабо-согласованы: они никогда не бросают ConcurrentModificationException и могут отражать или не отражать изменения, произошедшие после создания итератора.

Ограничения на ключи:
- Все ключи должны быть константами **одного** типа enum, который задаётся явно или неявно при создании карты.
- Null-ключи **запрещены**: вставка null-ключа вызывает NullPointerException. При этом проверка наличия null-ключа (containsKey) и его удаление (remove) работают без исключений.
- Null-значения допускаются.
- Класс не синхронизирован. Для многопоточного использования требуется внешняя синхронизация: `Collections.synchronizedMap(new EnumMap<>(...))`.

#### Ответ от Claude

**Внутреннее устройство**

EnumMap хранит данные в двух массивах одинакового размера, равного числу констант в данном enum-типе:
- массив значений (`Object[] vals`) — по одному слоту на каждую константу;
- вспомогательный массив или битовая маска для отслеживания, какие ключи присутствуют.

Индекс в массиве определяется через `Enum.ordinal()` — порядковый номер константы в объявлении enum. Таким образом, операция `get(key)` сводится к `vals[key.ordinal()]` — прямое обращение к элементу массива без вычисления хеша и разрешения коллизий.

**Почему быстрее HashMap**

HashMap при работе с ключом выполняет: вычисление `hashCode()`, определение индекса корзины, обход связного списка или дерева при коллизиях, сравнение через `equals()`. EnumMap не делает ничего из перечисленного:

1. **Нет хеширования** — индекс вычисляется через `ordinal()`, что эквивалентно чтению поля int.
2. **Нет коллизий** — каждая константа enum имеет уникальный ordinal, поэтому коллизии исключены по определению.
3. **Лучшая локальность кэша** — последовательный массив лучше помещается в кэш процессора, чем разрозненные узлы связного списка / дерева HashMap.
4. **Предсказуемый размер** — размер массива фиксирован числом констант enum и никогда не перераспределяется (rehash).

Согласно официальной документации Oracle (Java Tutorial, раздел «Map Implementations»): «Если вам нужно отобразить enum на значение, всегда предпочитайте EnumMap массиву».

**Ограничения на ключи**

| Ограничение | Поведение |
|---|---|
| Тип ключа | Строго один конкретный enum-тип, задаётся при создании |
| null-ключ при вставке | NullPointerException |
| null-ключ при поиске/удалении | Разрешено, исключений нет |
| null-значения | Разрешены |
| Потокобезопасность | Не синхронизирован |

**Пример использования**

```java
enum Day { MON, TUE, WED, THU, FRI, SAT, SUN }

EnumMap<Day, String> schedule = new EnumMap<>(Day.class);
schedule.put(Day.MON, "Совещание");
schedule.put(Day.FRI, "Ретроспектива");

// Итерация в порядке объявления констант
for (Map.Entry<Day, String> e : schedule.entrySet()) {
    System.out.println(e.getKey() + ": " + e.getValue());
}
```

Итерация всегда идёт в порядке объявления констант enum (MON, TUE, …, SUN), что является детерминированным поведением, в отличие от HashMap.
