### Вопрос 41. Чем IdentityHashMap отличается от HashMap? Какой метод используется для сравнения ключей, и в каких реальных сценариях применяется IdentityHashMap?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [IdentityHashMap (Java SE 17 & JDK 17) — docs.oracle.com](https://docs.oracle.com/en/java/javase/17/docs/api/java.base/java/util/IdentityHashMap.html)

Данный класс реализует Map-интерфейс с применением ссылочного равенства вместо объектного при сравнении ключей и значений. Иначе говоря, в IdentityHashMap два ключа k1 и k2 считаются равными тогда и только тогда, когда `(k1 == k2)`. В обычных реализациях Map (например, HashMap) два ключа k1 и k2 считаются равными при `(k1 == null ? k2 == null : k1.equals(k2))`.

**Это не универсальная реализация Map.** Хотя данный класс реализует интерфейс Map, он намеренно нарушает общий контракт Map, который требует использования метода equals при сравнении объектов. Этот класс предназначен только для тех редких случаев, когда требуется семантика ссылочного равенства.

Типичное применение — топологически-сохраняющие преобразования графов объектов, такие как сериализация или глубокое копирование. Для выполнения подобных преобразований программа должна поддерживать «таблицу узлов», отслеживающую все уже обработанные ссылки на объекты. Таблица узлов не должна отождествлять различные объекты, даже если они оказываются equal.

Другое типичное применение — ведение proxy-объектов. Например, средство отладки может захотеть поддерживать proxy-объект для каждого объекта отлаживаемой программы.

**Внутренняя реализация:** простая таблица с линейным пробированием. Массив содержит ключи и значения вперемешку, с ключами по чётным индексам и значениями по нечётным. Такое расположение обеспечивает лучшую локальность для больших таблиц по сравнению с использованием раздельных массивов. Для хеширования используется `System.identityHashCode(Object)`.

#### Ответ от Claude

**Ключевое различие: способ сравнения ключей**

| Аспект | HashMap | IdentityHashMap |
|---|---|---|
| Метод сравнения ключей | `key1.equals(key2)` | `key1 == key2` (ссылочное равенство) |
| Метод хеширования | `key.hashCode()` | `System.identityHashCode(key)` |
| Нарушает контракт Map | Нет | **Да** — намеренно |
| Внутренняя структура | Массив корзин + цепочки / деревья | Линейное пробирование (один массив) |

**Метод хеширования**

IdentityHashMap использует `System.identityHashCode(key)` — это нативный метод, возвращающий хеш-код по умолчанию (как правило, производный от адреса объекта в памяти), независимо от того, переопределил ли объект `hashCode()`. Два объекта, «равные» по `equals()`, но являющиеся разными экземплярами, получат (как правило) разные identity hash codes и будут храниться как два независимых ключа.

**Внутренняя структура**

Реализована в виде одного массива с чередующимися ключами и значениями:
- чётные индексы (0, 2, 4, …) — ключи;
- нечётные индексы (1, 3, 5, …) — значения.

Коллизии разрешаются линейным пробированием (linear probing). Согласно документации Oracle, такая схема обеспечивает лучшую локальность кэша по сравнению с раздельными массивами или цепочками HashMap. Для многих реализаций JVM и наборов операций IdentityHashMap работает быстрее HashMap, который использует chaining (цепочки).

**Реальные сценарии применения**

1. **Глубокое копирование объектного графа.** При копировании графа с циклическими или разделяемыми ссылками нужно отслеживать уже скопированные объекты по их идентичности. HashMap не подойдёт, если несколько разных объектов возвращают одинаковый `equals()`.

2. **Сериализация.** Стандартная Java-сериализация (`ObjectOutputStream`) внутри использует IdentityHashMap для отслеживания уже сериализованных объектов — именно чтобы корректно обработать разделяемые ссылки и циклы.

3. **Proxy / инструментирование.** Фреймворки отладки, профилировщики или AOP-контейнеры могут хранить метаданные для каждого конкретного экземпляра объекта. Если два объекта равны по `equals()`, их всё равно нужно различать.

4. **Маркировка при обходе графа.** Алгоритмы обхода деревьев/графов, где нужно пометить «посещённые» узлы по ссылке, а не по значению.

**Важное предупреждение**

IdentityHashMap намеренно нарушает контракт интерфейса Map (который требует использования `equals()`). Его нельзя применять как универсальную замену HashMap — только в перечисленных специфических случаях.

```java
String a = new String("hello");
String b = new String("hello");

// HashMap: a и b — один ключ (equals возвращает true)
Map<String, Integer> hashMap = new HashMap<>();
hashMap.put(a, 1);
hashMap.put(b, 2); // перезапишет значение для a
System.out.println(hashMap.size()); // 1

// IdentityHashMap: a и b — разные ключи (разные ссылки)
Map<String, Integer> identityMap = new IdentityHashMap<>();
identityMap.put(a, 1);
identityMap.put(b, 2); // добавит второй ключ
System.out.println(identityMap.size()); // 2
```

### Вопрос 42. В чём разница между интерфейсами Comparable и Comparator? Когда следует реализовывать Comparable в классе, а когда создавать отдельный Comparator?

**Категория:** Collections · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Comparable (Java SE 17 & JDK 17) — docs.oracle.com](https://docs.oracle.com/en/java/javase/17/docs/api/java.base/java/lang/Comparable.html)

Данный интерфейс устанавливает **полное упорядочение** объектов каждого класса, который его реализует. Это упорядочение называется **естественным упорядочением** класса, а метод `compareTo` — его методом естественного сравнения.

Списки (и массивы) объектов, реализующих данный интерфейс, могут автоматически сортироваться методами `Collections.sort` и `Arrays.sort`. Объекты, реализующие этот интерфейс, могут использоваться в качестве ключей в сортированной карте или элементов сортированного множества без необходимости указывать компаратор.

Настоятельно рекомендуется, чтобы естественное упорядочение было согласовано с `equals`, то есть чтобы `(e1.compareTo(e2) == 0)` имело то же булево значение, что и `e1.equals(e2)`, для любых e1 и e2 класса C. Нарушение этого условия необходимо явно указывать в документации, например: «Внимание: данный класс имеет естественное упорядочение, несогласованное с equals».

#### Ответ от Claude

**Сравнительная таблица**

| Аспект | Comparable | Comparator |
|---|---|---|
| Пакет | `java.lang` | `java.util` |
| Метод | `compareTo(T o)` — сравнивает `this` с `o` | `compare(T o1, T o2)` — сравнивает два внешних объекта |
| Где реализуется | Внутри самого класса | В отдельном классе / лямбде |
| Тип упорядочения | Единственное «естественное» | Произвольное, можно создать любое число |
| Изменение исходного класса | Требуется | Не требуется |
| Функциональный интерфейс | Нет | Да (`@FunctionalInterface` с Java 8) |

**Comparable — естественный порядок**

Comparable реализуется непосредственно в классе и определяет единственный «канонический» способ сравнения его объектов. Согласно документации Oracle, реализующий класс получает:
- автоматическую сортировку в `Collections.sort()` и `Arrays.sort()` без дополнительных аргументов;
- возможность использоваться в `TreeSet` и `TreeMap` без явного компаратора.

Метод `compareTo(T o)` должен возвращать:
- отрицательное число, если `this < o`;
- ноль, если `this == o` (по смыслу);
- положительное число, если `this > o`.

```java
public class Product implements Comparable<Product> {
    private String name;
    private double price;

    @Override
    public int compareTo(Product other) {
        return Double.compare(this.price, other.price); // естественный порядок: по цене
    }
}

List<Product> products = new ArrayList<>(...);
Collections.sort(products); // сортируется по цене автоматически
```

**Comparator — внешний/дополнительный порядок**

Comparator — отдельный объект (класс или лямбда), задающий правило сравнения извне. Начиная с Java 8 является `@FunctionalInterface`, что позволяет записывать его как лямбду или ссылку на метод. Согласно документации Oracle, Comparator передаётся явным аргументом в `Arrays.sort()`, `Collections.sort()`, конструкторы `TreeSet`/`TreeMap`.

```java
// Несколько разных порядков для одного класса:
Comparator<Product> byName  = Comparator.comparing(Product::getName);
Comparator<Product> byPrice = Comparator.comparingDouble(Product::getPrice).reversed();
Comparator<Product> byNameThenPrice = byName.thenComparing(byPrice);

products.sort(byNameThenPrice);
```

**Когда использовать Comparable**

- Есть **один очевидный «главный» порядок** для объектов класса (например, числа — по значению, строки — лексикографически, даты — хронологически).
- Класс находится под вашим контролем и его можно изменить.
- Объекты будут использоваться в `TreeSet`/`TreeMap` без явного компаратора.

**Когда использовать Comparator**

- Нужно **несколько разных стратегий сортировки** (по имени, по цене, по дате и т.д.).
- Класс **не под вашим контролем** (библиотечный, сторонний) или его нельзя изменить.
- Нужна **разовая или ситуационная** сортировка, не отражающая «природу» объекта.
- Требуется null-безопасное сравнение (`Comparator.nullsFirst()` / `Comparator.nullsLast()`).

**Согласованность с equals**

И `Comparable.compareTo()`, и `Comparator.compare()` рекомендуется делать согласованными с `equals()`: если `a.compareTo(b) == 0`, то должно выполняться `a.equals(b)`. Несоблюдение этого условия приводит к некорректному поведению `TreeSet` и `TreeMap`, которые используют сравнение вместо `equals` для определения уникальности элементов. Примечательное исключение из стандартной библиотеки: `BigDecimal` — `compareTo` считает 4.0 и 4.00 равными, тогда как `equals` — нет.

### Вопрос 43. Что произойдёт, если реализация Comparator (или compareTo у Comparable) несовместима с equals()? Какие коллекции нарушат своё поведение, и почему?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Comparator (Java Platform SE 8) — Oracle Docs](https://docs.oracle.com/javase/8/docs/api/java/util/Comparator.html)

Порядок, задаваемый компаратором `c` на множестве элементов `S`, называется **согласованным с equals**, если и только если:

```
c.compare(e1, e2) == 0  имеет то же булево значение, что и  e1.equals(e2)
```

для каждой пары `e1` и `e2` из `S`.

В целом это не является строгим требованием: «generally the case, but not strictly required that `(compare(x, y) == 0) == (x.equals(y))`». Тем не менее, любой компаратор, нарушающий это условие, должен явно документировать данный факт.

**Что происходит при несовместимости:**
Если добавить два объекта `a` и `b`, для которых выполняется `(a.equals(b) && c.compare(a, b) != 0)`, в пустой `TreeSet` с компаратором `c`, вторая операция `add` вернёт `true` и размер множества увеличится, хотя `Set.add()` обязан возвращать `false` при повторном добавлении эквивалентного элемента.

Поведение `TreeSet` при этом остаётся строго определённым, однако он перестаёт соответствовать общему контракту интерфейса `Set`.

#### Ответ от Claude

**Определение совместимости с equals**

Упорядочивание, определяемое `Comparator` (или `compareTo` у `Comparable`), считается **совместимым с equals**, если выполняется условие:

```
compare(a, b) == 0  ⟺  a.equals(b)
```

Это требование зафиксировано в Javadoc обоих интерфейсов (`Comparator`, `Comparable`) и в документации ключевых коллекций.

---

**Какие коллекции нарушат поведение**

Коллекции делятся на два лагеря по механизму сравнения:

| Коллекция | Механизм сравнения | Нарушится при несовместимости? |
|---|---|---|
| `HashSet`, `HashMap` | `equals()` + `hashCode()` | Нет — используют equals |
| `TreeSet`, `TreeMap` | `compareTo()` / `compare()` | **Да** — используют только компаратор |

`TreeSet` и `TreeMap` **полностью игнорируют `equals()`** при определении дублей и порядка. Для них два элемента эквивалентны тогда и только тогда, когда `compare(a, b) == 0`.

**Последствие 1 — ложные дубли в TreeSet:**
Если `compare(a, b) != 0`, но `a.equals(b)`, то TreeSet добавит оба объекта — нарушив контракт `Set`, который гарантирует отсутствие дублей по `equals`.

**Последствие 2 — потеря элементов в TreeSet:**
Обратная ситуация: если `compare(a, b) == 0`, но `!a.equals(b)`, TreeSet считает объекты одинаковыми и отбрасывает второй — хотя по `equals` они различны.

**Последствие 3 — нарушение контракта Map:**
В `TreeMap` ключи `a` и `b`, где `a.equals(b)`, но `compare(a, b) != 0`, будут занимать разные слоты. Операции `get(a)` и `get(b)` вернут разные значения, хотя `Map` обязан считать такие ключи эквивалентными.

---

**Реальный пример — BigDecimal**

Классический пример из стандартной библиотеки: `BigDecimal`. Его натуральный порядок **несовместим с equals**:

```java
BigDecimal a = new BigDecimal("1.0");
BigDecimal b = new BigDecimal("1.00");

a.equals(b);       // false — разные масштабы (scale)
a.compareTo(b);    // 0    — одинаковое числовое значение
```

Следствие (подтверждено Javadoc `BigDecimal`):
- В `HashSet` оба объекта хранятся отдельно (работает `equals`)
- В `TreeSet` второй объект не будет добавлен (работает `compareTo` → 0)

Oracle явно предупреждает: *«care should be exercised if BigDecimal objects are used as keys in a SortedMap or elements in a SortedSet since BigDecimal's natural ordering is inconsistent with equals»*.

---

**Почему `HashSet`/`HashMap` не затронуты**

`HashSet` и `HashMap` используют `equals()` и `hashCode()` для сравнения, а не `compareTo`. Несовместимость компаратора на них не влияет. Проблема возникает исключительно в **отсортированных** коллекциях (`SortedSet`, `SortedMap` и их реализации `TreeSet`, `TreeMap`), а также при явном использовании `Comparator` с методами `Collections.sort()`, `Arrays.sort()` или `Stream.sorted()` (там несовместимость влияет только на порядок, но не на дубликаты, поскольку List дубли разрешает).

---

**Итог**

Нарушают контракт при несовместимости comparator/compareTo с equals:
- `TreeSet` — нарушает контракт `Set` (дубли по equals / потеря элементов)
- `TreeMap` — нарушает контракт `Map` (два «равных» ключа по equals занимают разные слоты)

Не нарушают:
- `HashSet`, `LinkedHashSet`, `HashMap`, `LinkedHashMap`, `ArrayDeque`, `ArrayList` — не используют `compareTo`/`compare` для эквивалентности

### Вопрос 44. Как в Java 8 появились фабричные методы Comparator.comparing() и thenComparing()? Как с их помощью строится цепочка компараторов по нескольким полям?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Comparator (Java Platform SE 8) — Oracle Docs](https://docs.oracle.com/javase/8/docs/api/java/util/Comparator.html)

В Java 8 интерфейс `Comparator` был аннотирован как `@FunctionalInterface` и расширен рядом статических и default-методов. Среди них:

**`comparing(Function<? super T, ? extends U> keyExtractor)`** — статический метод, принимающий функцию-экстрактор ключа и возвращающий `Comparator<T>`, который сравнивает объекты по извлечённому ключу, реализующему `Comparable`:

```java
static <T,U extends Comparable<? super U>> Comparator<T>
    comparing(Function<? super T,? extends U> keyExtractor)
```

Перегруженная версия принимает дополнительный `Comparator<? super U> keyComparator` для сравнения самого ключа.

**`thenComparing(Comparator<? super T> other)`** — default-метод, создающий лексикографический компаратор: если первичное сравнение даёт 0, применяется `other`. Также существуют перегрузки, принимающие функцию-экстрактор вместо компаратора.

Существуют примитивные специализации: `comparingInt`, `comparingLong`, `comparingDouble` и `thenComparingInt`, `thenComparingLong`, `thenComparingDouble` — они избегают автобоксинга.

Пример из документации:

```java
Comparator<String> cmp = Comparator.comparingInt(String::length)
    .thenComparing(String.CASE_INSENSITIVE_ORDER);
```

#### Ответ от Claude

**Предпосылки появления в Java 8**

До Java 8 компаратор создавался исключительно через анонимный класс с явной реализацией метода `compare()`. Это было многословно и нечитаемо, особенно при сортировке по нескольким полям. Java 8 сделала `Comparator` функциональным интерфейсом (`@FunctionalInterface`) и добавила к нему статические фабричные методы и default-методы, позволяющие строить компараторы через лямбды и ссылки на методы.

---

**Как работает `Comparator.comparing()`**

Статический метод `comparing()` принимает `Function<T, U>` — функцию-экстрактор, которая извлекает ключ сравнения из объекта типа `T`. Метод возвращает готовый `Comparator<T>`, сравнивающий объекты по значению этого ключа.

Сигнатуры:

```java
// Ключ должен реализовывать Comparable
static <T, U extends Comparable<? super U>> Comparator<T>
    comparing(Function<? super T, ? extends U> keyExtractor)

// Ключ сравнивается указанным компаратором
static <T, U> Comparator<T>
    comparing(Function<? super T, ? extends U> keyExtractor,
              Comparator<? super U> keyComparator)
```

Пример:

```java
// До Java 8
Comparator<Employee> byName = new Comparator<Employee>() {
    public int compare(Employee e1, Employee e2) {
        return e1.getName().compareTo(e2.getName());
    }
};

// Java 8 — лямбда
Comparator<Employee> byName = (e1, e2) -> e1.getName().compareTo(e2.getName());

// Java 8 — ссылка на метод через Comparator.comparing()
Comparator<Employee> byName = Comparator.comparing(Employee::getName);
```

---

**Как работает `thenComparing()`**

`thenComparing()` — default-метод, реализующий лексикографическую логику: если первичный компаратор возвращает `0` (объекты равны по первому ключу), применяется вторичный компаратор. Можно строить цепочку произвольной длины.

Сигнатуры:

```java
// Принимает готовый компаратор
default Comparator<T> thenComparing(Comparator<? super T> other)

// Принимает экстрактор с Comparable-ключом
default <U extends Comparable<? super U>> Comparator<T>
    thenComparing(Function<? super T, ? extends U> keyExtractor)

// Принимает экстрактор и дополнительный компаратор ключа
default <U> Comparator<T>
    thenComparing(Function<? super T, ? extends U> keyExtractor,
                  Comparator<? super U> keyComparator)
```

---

**Пример: сортировка по нескольким полям**

```java
// Сотрудники: сначала по фамилии, затем по имени, затем по возрасту убывающе
Comparator<Employee> comparator = Comparator
    .comparing(Employee::getLastName)
    .thenComparing(Employee::getFirstName)
    .thenComparingInt(Employee::getAge).reversed();

List<Employee> employees = ...;
employees.sort(comparator);

// То же самое для Stream
employees.stream()
    .sorted(Comparator
        .comparing(Employee::getLastName)
        .thenComparing(Employee::getFirstName))
    .collect(Collectors.toList());
```

---

**Примитивные специализации**

Для примитивных ключей предусмотрены специализированные методы, избегающие автобоксинга:

```java
Comparator.comparingInt(Employee::getAge)
Comparator.comparingLong(File::length)
Comparator.comparingDouble(Product::getPrice)

// В цепочке
Comparator.comparing(Employee::getName)
    .thenComparingInt(Employee::getAge)
```

---

**Дополнительные возможности**

- `Comparator.naturalOrder()` — возвращает компаратор натурального порядка
- `Comparator.reverseOrder()` — обратный натуральный порядок
- `.reversed()` — инвертирует любой компаратор
- `Comparator.nullsFirst(comparator)` / `Comparator.nullsLast(comparator)` — обработка `null`-значений

Пример с null-безопасностью:

```java
Comparator<Employee> comparator = Comparator
    .comparing(Employee::getDepartment, Comparator.nullsLast(Comparator.naturalOrder()))
    .thenComparing(Employee::getName);
```

---

**Итог**

`Comparator.comparing()` — статическая фабрика, превращающая функцию-экстрактор в полноценный компаратор. `thenComparing()` — default-метод, добавляющий вторичный критерий сортировки. Вместе они образуют fluent API для построения многоуровневых компараторов без анонимных классов, с поддержкой лямбд и ссылок на методы.

### Вопрос 45. Почему метод size() в ConcurrentHashMap возвращает приблизительное значение? Какой механизм (CounterCell, LongAdder) используется для подсчёта элементов, и чем mappingCount() отличается от size()?

**Категория:** Collections · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [ConcurrentHashMap (Java SE 21) — Oracle Docs](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/util/concurrent/ConcurrentHashMap.html)

Из документации класса:

> «Bear in mind that the results of aggregate status methods including `size`, `isEmpty`, and `containsValue` are typically useful only when a map is not undergoing concurrent updates in other threads. Otherwise the results of these methods reflect transient states that may be adequate for monitoring or estimation purposes, but not for program control.»

Метод `mappingCount()` (добавлен в Java 8):

```java
public long mappingCount()
```

> «Returns the number of mappings. This method should be used instead of `size()` because a ConcurrentHashMap may contain more mappings than can be represented as an int. The value returned is an estimate; the actual count may differ if there are concurrent insertions or removals.»

Метод `size()` возвращает `int`, а при превышении `Integer.MAX_VALUE` — именно `Integer.MAX_VALUE`. `mappingCount()` возвращает `long`.

#### Ответ от Claude

**Почему точный подсчёт дорог в конкурентной среде**

В классическом `HashMap` счётчик элементов — одно поле `size`, обновляемое при каждом `put`/`remove`. В однопоточном коде это работает без накладных расходов. В конкурентной среде единственный счётчик становится узким местом: каждая операция изменения должна атомарно обновлять его, что при большом числе потоков приводит к серьёзной конкуренции (cache line contention).

Предыдущая реализация Java 7 решала эту проблему через сегменты (`Segment`): каждый сегмент хранил свой счётчик. В Java 8 сегменты были упразднены, а для подсчёта введён принципиально иной механизм, заимствованный из `LongAdder`.

---

**Механизм подсчёта: baseCount + CounterCell[]**

Начиная с Java 8, `ConcurrentHashMap` использует два элемента для хранения счётчика:

1. **`baseCount`** — `volatile long`-поле, обновляемое через CAS (Compare-And-Swap). Используется в отсутствие конкуренции: если поток успешно применяет CAS к `baseCount`, CounterCell не нужен.

2. **`counterCells`** — массив объектов `CounterCell`. Когда при попытке CAS-обновления `baseCount` возникает конкуренция (CAS проигрывает), поток вместо повторной попытки пишет в один из элементов массива. Каждый элемент — отдельная `volatile long`-ячейка, защищённая аннотацией `@Contended` для устранения ложного разделения кэш-строк (false sharing).

Класс `CounterCell` — внутренний адаптер, идейно повторяющий `Striped64` (базовый класс `LongAdder`). Из комментария в исходниках OpenJDK:

```
/**
 * A padded cell for distributing counts.
 * Adapted from LongAdder and Striped64.
 */
@sun.misc.Contended
static final class CounterCell {
    volatile long value;
    CounterCell(long x) { value = x; }
}
```

Поле `baseCount` в классе:
```java
// Base counter value, used mainly when there is no contention,
// but also as a fallback during table initialization races.
// Updated via CAS.
private transient volatile long baseCount;

// Table of counter cells. When non-null, size is a power of 2.
private transient volatile CounterCell[] counterCells;
```

---

**Как работает `sumCount()`**

Метод `size()` (и `mappingCount()`) вызывают `sumCount()`, который складывает `baseCount` со значениями всех ячеек массива:

```java
final long sumCount() {
    CounterCell[] cs = counterCells;
    long sum = baseCount;
    if (cs != null) {
        for (CounterCell c : cs)
            if (c != null)
                sum += c.value;
    }
    return sum;
}
```

Этот обход выполняется **без каких-либо блокировок**. В момент суммирования другие потоки могут параллельно изменять как `baseCount`, так и ячейки массива. Поэтому результат является **снимком (snapshot)** текущего состояния, а не гарантированно точным значением. Отсюда и предупреждение в документации Oracle: результат полезен для мониторинга и оценки, но не для управляющей логики программы.

---

**Чем `mappingCount()` отличается от `size()`**

Оба метода вызывают один и тот же `sumCount()`. Их отличия:

| Характеристика | `size()` | `mappingCount()` |
|---|---|---|
| Тип возврата | `int` | `long` |
| Ограничение | при n > Integer.MAX_VALUE возвращает Integer.MAX_VALUE | возвращает реальное (приблизительное) значение long |
| Версия Java | с Java 1.5 | с Java 8 |
| Точность | одинакова для обоих | одинакова для обоих |

Таким образом, `mappingCount()` — это **не более точный**, а более **ёмкий** метод: он нужен для карт, потенциально содержащих более 2 миллиардов записей (что реально для долгоживущих in-memory хранилищ). Точность у обоих методов одинакова и определяется вызовом `sumCount()` без синхронизации.

---

**Связь с LongAdder**

`java.util.concurrent.atomic.LongAdder` (также появился в Java 8) реализует ту же идею: при обновлении без конкуренции обновляется общее поле `base`, при конкуренции — выбирается ячейка из массива `cells` (тот же `Striped64`). `LongAdder.sum()` суммирует все ячейки так же, как `sumCount()`. Разработчики `ConcurrentHashMap` адаптировали этот механизм непосредственно в самом классе, не используя `LongAdder` напрямую, поскольку требовалась интеграция с логикой обнаружения конкуренции внутри операций `addCount()`.

---

**Расхождение между источниками**

Некоторые статьи (в т.ч. на Medium) утверждают, что `mappingCount()` «более точен» чем `size()`. Это неверно: оба метода вызывают один и тот же `sumCount()` без блокировок и дают одинаково приблизительный результат. Единственное реальное преимущество `mappingCount()` — тип `long`, позволяющий корректно обрабатывать карты с более чем `Integer.MAX_VALUE` записями.

---

**Итог**

`size()` возвращает приблизительное значение, потому что подсчёт реализован через `sumCount()` — беззамочное суммирование `baseCount` и массива `CounterCell[]`. Такой распределённый счётчик (аналог `LongAdder`) минимизирует конкуренцию между потоками при записи ценой точности при чтении. `mappingCount()` использует тот же механизм, но возвращает `long` вместо `int`, что делает его предпочтительным для больших карт.
### Вопрос 46. Какие атомарные составные операции предоставляет ConcurrentHashMap — compute(), computeIfAbsent(), computeIfPresent(), merge()? Чем они безопаснее конструкции «if(!map.containsKey(k)) map.put(k, v)», даже если оба метода синхронизированы?

**Категория:** Collections · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Java ConcurrentMap — Jenkov Tutorials](https://jenkov.com/tutorials/java-util-concurrent/concurrentmap.html)

Даже если методы `containsKey()` и `put()` по отдельности являются потокобезопасными, составная конструкция вида

```java
if (!map.containsKey("key1")) {
    map.put("key1", "value1");
}
```

не является потокобезопасной. Два потока могут одновременно вызвать `containsKey("key1")`, оба получат `false`, затем оба выполнят `put()` — второй вызов перезапишет результат первого. Это классическая ошибка типа «slipped condition» (условие выскользнуло).

**putIfAbsent()** решает проблему для простого случая вставки: реализация `ConcurrentMap` гарантирует, что только один поток вставит значение для одного и того же ключа.

**computeIfAbsent()** делает то же самое, но позволяет вычислять значение через лямбду:

```java
concurrentMap.computeIfAbsent("key2", (key) -> "Value for key " + key);
```

«Только одному потоку за раз разрешается выполнять `computeIfAbsent()` для одного и того же отсутствующего ключа. Если два потока вызовут `computeIfAbsent()` для одного и того же отсутствующего ключа, один из них вычислит и вставит значение, а второй не вычислит и не вставит ничего.»

**computeIfPresent()** аналогичен, но выполняется только если ключ уже присутствует в отображении.

#### Ответ от Claude

**Суть проблемы: составная проверка не атомарна**

Конструкция `if(!map.containsKey(k)) map.put(k, v)` состоит из двух раздельных операций. Между проверкой и вставкой планировщик может переключить поток — другой поток успеет вставить значение для того же ключа. Синхронизация отдельных методов (`containsKey` и `put`) не помогает: каждый метод захватывает и освобождает монитор независимо. Совместная атомарность двух вызовов не обеспечивается.

**Четыре атомарных составных операции ConcurrentHashMap**

Все четыре метода обладают ключевой гарантией, зафиксированной в Javadoc (Java 21):

> «The entire method invocation is performed atomically.» («Всё обращение к методу выполняется атомарно.»)

Это означает: на время выполнения метода соответствующий бакет (или его часть) блокируется, и никакое другое обновление этого же ключа невозможно.

| Метод | Условие выполнения | Что делает |
|---|---|---|
| `computeIfAbsent(K key, Function<K,V> f)` | ключ отсутствует или mapped на null | Вычисляет значение функцией `f`, вставляет результат |
| `computeIfPresent(K key, BiFunction<K,V,V> f)` | ключ присутствует и mapped не на null | Перевычисляет значение; если `f` вернула null — удаляет запись |
| `compute(K key, BiFunction<K,V,V> f)` | всегда | Передаёт в `f` текущее значение (или null); результат null — удаление |
| `merge(K key, V value, BiFunction<V,V,V> f)` | всегда | Если ключ отсутствует — вставляет `value`; иначе применяет `f(старое, value)` |

**Примеры применения**

```java
// Ленивая инициализация (thread-safe):
map.computeIfAbsent("stats", k -> new LongAdder()).increment();

// Атомарное обновление счётчика:
map.compute("hits", (k, v) -> v == null ? 1L : v + 1);

// merge — удобный способ агрегировать значения:
map.merge("hits", 1L, Long::sum);

// computeIfPresent — обновить только существующую запись:
map.computeIfPresent("cache-key", (k, v) -> recompute(v));
```

**Почему синхронизация отдельных методов не спасает**

Даже в `Collections.synchronizedMap` конструкция `if(!map.containsKey(k)) map.put(k,v)` опасна без внешней синхронизации на весь блок, потому что два отдельных `synchronized`-вызова — это две отдельные критические секции. `ConcurrentHashMap` же атомизирует всю логику «проверить — вычислить — записать» в рамках одного вызова метода, удерживая блокировку бакета на всё это время.

**Ограничения**

Согласно Javadoc, функция, переданная в эти методы, не должна изменять саму карту во время вычисления — это может привести к `IllegalStateException`. Вычисление должно быть коротким и простым, так как часть обновлений других потоков может блокироваться на время его выполнения.

### Вопрос 47. Что такое параллельные bulk-операции в ConcurrentHashMap (forEach, reduce, search) появившиеся в Java 8? Что означает параметр parallelismThreshold и как он управляет параллелизмом?

**Категория:** Collections · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [ConcurrentHashMap (Java Platform SE 8) — Oracle Javadoc](https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/ConcurrentHashMap.html)

В Java 8 в `ConcurrentHashMap` добавлены три семейства bulk-операций, безопасных для конкурентного использования:

- **forEach** — выполняет заданное действие для каждого элемента; вариант с трансформером сначала преобразует элемент, а потом применяет действие.
- **search** — возвращает первый ненулевой результат применения поисковой функции к элементам; при нахождении результата обработка остальных элементов прекращается, результаты остальных параллельных вызовов функции игнорируются.
- **reduce** — накапливает результат применения трансформации к каждому элементу с помощью функции-редьюсера. Функция редукции не должна зависеть от порядка (должна быть ассоциативной и коммутативной).

Каждая из операций принимает параметр `parallelismThreshold`:

> «Methods proceed sequentially if the current map size is estimated to be less than the given threshold. Using a value of `Long.MAX_VALUE` suppresses all parallelism. Using a value of `1` results in maximal parallelism by partitioning into enough subtasks to fully utilize the `ForkJoinPool.commonPool()` that is used for all parallel computations.»

Таким образом, `parallelismThreshold` — это оценочное число элементов, при котором операция переходит к параллельному выполнению. Если карта содержит меньше элементов, чем порог, операция выполняется последовательно.

#### Ответ от Claude

**Три семейства bulk-операций**

В Java 8 `ConcurrentHashMap` получил 12+ методов, организованных в три семейства. Каждый метод принимает первым аргументом `long parallelismThreshold`.

**forEach** — перебор всех элементов:

```java
// Простой перебор пар:
map.forEach(1, (k, v) -> System.out.println(k + "=" + v));

// С трансформацией: сначала преобразуем, потом действуем:
map.forEach(1000, (k, v) -> k + ":" + v, System.out::println);

// Варианты: forEachKey, forEachValue, forEachEntry
```

**search** — поиск первого подходящего элемента:

```java
// Возвращает первый ключ, значение которого > 100, или null:
String found = map.search(1000, (k, v) -> v > 100 ? k : null);
// Варианты: searchKeys, searchValues, searchEntries
```

При нахождении результата параллельные задачи прекращают работу. Важно: порядок перебора не гарантирован.

**reduce** — агрегирование:

```java
// Сумма всех значений:
long total = map.reduceToLong(1000,
    (k, v) -> v,          // трансформер
    0L,                    // basis (нейтральный элемент)
    Long::sum);            // редьюсер

// Для примитивов: reduceToDouble, reduceToLong, reduceToInt
// (не боксируют промежуточные результаты)
```

Функция редьюсера должна быть ассоциативной и коммутативной, так как порядок слияния промежуточных результатов при параллельном выполнении не определён.

**Параметр parallelismThreshold: как управляет параллелизмом**

`parallelismThreshold` — это порог по числу элементов карты (оценочный), при превышении которого операция разбивается на подзадачи и передаётся в `ForkJoinPool.commonPool()`.

| Значение | Поведение |
|---|---|
| `Long.MAX_VALUE` | Всегда последовательно (параллелизм подавлен) |
| `1` | Максимальный параллелизм: всегда используется ForkJoinPool |
| Произвольное N | Параллельно, только если карта содержит >= N элементов |

Механизм: операция рекурсивно делит таблицу на части (через внутренний `Traverser`), создавая задачи типа `ForkJoinTask`. Каждая подзадача обрабатывает подмножество бакетов. Деление продолжается, пока размер подзадачи больше порога.

**Практические рекомендации**

- Начинайте с крайних значений (`1` или `Long.MAX_VALUE`), измеряйте производительность, затем подбирайте промежуточный порог.
- Overhead от параллельного деления на задачи оправдан только при больших картах и дорогих функциях трансформации/редукции.
- Функции не должны иметь побочных эффектов на состояние карты — это может привести к непредсказуемым результатам.
- Элементы `ConcurrentHashMap` не упорядочены, и при разных параллельных запусках порядок обработки может отличаться — корректность функции не должна зависеть от порядка.

**Отличие от Stream API**

Bulk-операции `ConcurrentHashMap` не требуют получения `entrySet().stream()` и снятия снапшота данных — они работают непосредственно с текущим состоянием карты, которая может изменяться в ходе операции. Результаты отражают состояние на момент начала операции, но не гарантируют строгую согласованность в случае конкурентных изменений.

### Вопрос 48. Как ConcurrentHashMap в Java 8 реализует кооперативный resize (transfer)? Каким образом несколько потоков одновременно переносят бакеты из старой таблицы в новую?

**Категория:** Collections · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [How ConcurrentHashMap Works Internally After Java 8? — Java Concept Of The Day](https://javaconceptoftheday.com/how-concurrenthashmap-works-internally-after-java-8/)

Когда число пар ключ–значение превышает порог (load factor * capacity), один поток инициирует расширение: создаёт новую таблицу вдвое большего размера и начинает переносить бакеты из старой таблицы в новую. Другие потоки, которые в это время обращаются к старой таблице, обнаруживают, что идёт расширение, и не ждут. Вместо этого они тоже начинают переносить бакеты из старой таблицы в новую.

Для координации используются три служебных поля:

- **`sizeCtl`** — управляющее поле. Во время расширения содержит отрицательное значение: `-(1 + число активных потоков-переносчиков)`.
- **`transferIndex`** — указатель на конец старой таблицы, который декрементируется по мере переноса бакетов. Каждый поток атомарно забирает себе диапазон бакетов.
- **`MIN_TRANSFER_STRIDE`** — минимальное число бакетов, которое получает каждый поток за один раз.

Когда все элементы бакета перенесены в новую таблицу, в старом бакете устанавливается специальный узел `ForwardingNode`. Этот узел сигнализирует другим потокам: содержимое бакета перенесено, операции чтения и записи для этого бакета следует направлять в новую таблицу. `ForwardingNode` хранит ссылку на новую таблицу.

#### Ответ от Claude

**Предпосылки: зачем нужен кооперативный resize**

В Java 7 `ConcurrentHashMap` был разделён на независимые сегменты (`Segment[]`), каждый из которых имел собственный lock и собственную таблицу. Расширение каждого сегмента происходило независимо и только одним потоком. В Java 8 архитектура сегментов была упразднена — вместо неё используется единый массив нод (`Node<K,V>[] table`) и гранулярная блокировка на уровне бакета через `synchronized`. Однако единая таблица означает единый resize — и чтобы он не стал узким местом, был реализован кооперативный (cooperative) механизм.

**Ключевые поля координации**

| Поле | Тип | Назначение |
|---|---|---|
| `sizeCtl` | `volatile int` | Управляющее поле. `-1` — идёт инициализация; `-(1+N)` — идёт resize, N активных потоков-переносчиков; положительное — следующий порог resize |
| `transferIndex` | `volatile int` | Индекс следующего бакета для переноса. Изначально равен длине старой таблицы; уменьшается CAS-операциями |
| `MIN_TRANSFER_STRIDE` | `int` (константа 16) | Минимальный размер «страйда» — блока бакетов, захватываемого одним потоком за раз |
| `nextTable` | `volatile Node<K,V>[]` | Ссылка на новую таблицу, создаваемую в процессе resize |

**Пошаговый процесс кооперативного resize**

1. **Инициация.** Когда поток, выполняющий `put()`, обнаруживает, что `size >= threshold`, он вызывает `tryPresize()`, который создаёт `nextTable` вдвое большего размера и атомарно устанавливает `sizeCtl = -(1+1)` (один активный переносчик). Затем вызывается `transfer(tab, nextTable)`.

2. **Захват страйда.** Метод `transfer()` делит старую таблицу на блоки — «страйды» (stride). Каждый поток атомарно резервирует себе страйд через CAS на поле `transferIndex`:
   ```
   stride = max(MIN_TRANSFER_STRIDE, oldCapacity / 8 / NCPU)
   transferIndex -= stride  // CAS
   ```
   Поток будет переносить бакеты в диапазоне `[transferIndex, transferIndex + stride)`.

3. **Перенос бакета.** Для каждого бакета в своём страйде поток:
   - Захватывает `synchronized(бакет)`.
   - Разбивает цепочку (или дерево) на два подмножества по признаку `(hash & oldCapacity) == 0`: «низкие» остаются на той же позиции в новой таблице, «высокие» — уходят на позицию `oldIndex + oldCapacity`.
   - Устанавливает переработанные цепочки в соответствующие бакеты `nextTable`.
   - Помещает в бакет старой таблицы специальный узел `ForwardingNode`.

4. **ForwardingNode и MOVED-хэш.** `ForwardingNode` — это специальная реализация `Node`, у которой `hash == MOVED` (константа `-1`). Она хранит ссылку на `nextTable`. Когда поток, выполняющий `put()` или `get()` в старой таблице, видит в бакете узел с `hash == MOVED`, он понимает, что бакет перенесён, и:
   - При `get()` — переходит по ссылке `nextTable` и ищет там.
   - При `put()` — вызывает `helpTransfer()`, то есть сам включается в перенос оставшихся бакетов, прежде чем вставить своё значение.

5. **Завершение.** Когда последний страйд перенесён, поток, завершивший его, «передаёт эстафету»: устанавливает `table = nextTable`, `nextTable = null`, обновляет `sizeCtl` до нового порога.

**Схема взаимодействия потоков**

```
Поток A (insert)                Поток B (insert)               Поток C (insert)
    |                               |                               |
    | size >= threshold             |                               |
    | -> создаёт nextTable          |                               |
    | -> sizeCtl = -(1+1)          |                               |
    | -> transfer, страйд [48,64)   |                               |
    |                               | видит sizeCtl < 0            |
    |                               | -> helpTransfer               |
    |                               | -> страйд [32,48)             |
    |                               |                               | видит ForwardingNode
    |                               |                               | -> helpTransfer
    |                               |                               | -> страйд [16,32)
```

**Итоговые гарантии**

- Чтение (`get`) из уже перенесённых бакетов перенаправляется в `nextTable` через `ForwardingNode` без блокировки.
- Запись (`put`) в уже перенесённый бакет сначала помогает завершить resize (`helpTransfer`), что ускоряет общий процесс.
- CAS на `transferIndex` гарантирует, что два потока не возьмут один и тот же страйд.
- Количество одновременно работающих потоков-переносчиков отражается в `sizeCtl`: значение `-(1+N)` означает N активных потоков.

### Вопрос 49. Чем CopyOnWriteArrayList отличается от Collections.synchronizedList(ArrayList)? При каком соотношении операций чтения и записи каждый вариант предпочтительнее?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [CopyOnWriteArrayList (Java SE 21 & JDK 21) — Oracle Javadoc](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/util/concurrent/CopyOnWriteArrayList.html)

`CopyOnWriteArrayList` — потокобезопасный вариант `ArrayList`, в котором все изменяющие операции (`add`, `set` и т.д.) реализованы путём создания свежей копии внутреннего массива. Это является дорогостоящей операцией, но может оказаться более эффективным, чем альтернативы, когда операции обхода (traversal) существенно превышают по количеству операции изменения, и полезно тогда, когда нельзя или нежелательно синхронизировать обходы коллекции, но при этом необходимо исключить интерференцию между параллельными потоками.

Метод `iterator()` реализован в стиле «моментального снимка» (snapshot): он ссылается на состояние массива на момент создания итератора. Этот массив никогда не изменяется на протяжении всего времени жизни итератора, поэтому интерференция исключена, и итератор гарантированно не выбросит `ConcurrentModificationException`. Итератор не поддерживает операцию `remove`.

`Collections.synchronizedList()` возвращает синхронизированный (потокобезопасный) список, backed исходным списком. Для гарантии последовательного доступа критически важно, чтобы весь доступ к исходному списку осуществлялся через возвращённый список. При обходе через `Iterator` пользователь обязан вручную синхронизироваться по возвращённому списку; несоблюдение этого требования может привести к недетерминированному поведению:

```java
List<String> list = Collections.synchronizedList(new ArrayList<>());
// ...
synchronized (list) {
    Iterator<String> i = list.iterator(); // обязательно внутри synchronized-блока
    while (i.hasNext())
        foo(i.next());
}
```

#### Ответ от Claude

**Механизм синхронизации**

`Collections.synchronizedList(ArrayList)` оборачивает список в декоратор, который синхронизирует каждый отдельный метод (`get`, `add`, `remove` и т.д.) на одном мониторе (`this`). Это означает, что в любой момент времени только один поток может выполнять любую операцию — будь то чтение или запись. При обходе списка через итератор пользователь сам обязан удерживать блокировку на всё время итерации (иначе возможен `ConcurrentModificationException` или гонка данных).

`CopyOnWriteArrayList` при каждой изменяющей операции (`add`, `set`, `remove`) создаёт полную копию внутреннего массива, применяет изменение к копии, а затем атомарно заменяет ссылку на массив новой. Операции чтения (`get`, `contains`, итерация) работают с текущим снимком массива **без какой-либо блокировки**. Итераторы всегда работают с снимком на момент создания итератора и никогда не выбрасывают `ConcurrentModificationException`.

**Ключевые отличия**

| Характеристика | `synchronizedList(ArrayList)` | `CopyOnWriteArrayList` |
|---|---|---|
| Блокировка при чтении | Да (на весь `get`/итерацию) | Нет |
| Блокировка при записи | Да | Да (на время копирования + swap) |
| Стоимость записи | O(1) амортизированно | O(n) — полное копирование массива |
| Итерация без внешней синхронизации | Нет — требуется `synchronized`-блок | Да — всегда безопасна |
| `ConcurrentModificationException` | Возможен при итерации без блокировки | Никогда |
| Видимость записей в итераторе | Видит изменения в рамках блокировки | Не видит — работает со снимком |
| Память | Один массив | Дополнительная копия при каждой записи |

**При каком соотношении чтения/записи что предпочесть**

- `CopyOnWriteArrayList` предпочтителен, когда **чтения существенно преобладают над записями** (классический пример — список обработчиков событий, `Observer`-паттерн, конфигурационные данные). Запись происходит редко, зато чтение — часто и, возможно, одновременно из многих потоков без блокировок. Oracle Javadoc прямо указывает: «хорошо подходит для поддержки списков обработчиков событий, где изменения редки, а обходы часты».

- `Collections.synchronizedList(ArrayList)` предпочтителен, когда **записи и чтения примерно равновесны или записи часты**. В этом случае накладные расходы на копирование массива при каждой записи у `CopyOnWriteArrayList` перевешивают выгоду от отсутствия блокировок при чтении. `synchronizedList` обеспечивает O(1) запись (амортизированно), тогда как `CopyOnWriteArrayList` — O(n) при каждом изменении.

**Важное замечание о консистентности**

Итераторы `CopyOnWriteArrayList` не отражают изменений, сделанных после их создания. Если другой поток добавил элемент во время итерации, итератор этого не увидит. `synchronizedList` при правильно взятой блокировке даёт полную видимость всех изменений.

**Источники, использованные для верификации:**
- [CopyOnWriteArrayList (Java SE 21) — Oracle](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/util/concurrent/CopyOnWriteArrayList.html)
- [List Implementations — The Java Tutorials (Oracle)](https://docs.oracle.com/javase/tutorial/collections/implementations/list.html)
- [Collections.synchronizedList (Java SE 21) — Oracle](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/util/Collections.html#synchronizedList(java.util.List))

### Вопрос 50. Чем WeakReference отличается от SoftReference? Почему WeakHashMap использует именно WeakReference, а не SoftReference, для ключей?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [java.lang.ref (Java SE 24 & JDK 24) — Oracle Javadoc](https://docs.oracle.com/en/java/javase/24/docs/api/java.base/java/lang/ref/package-summary.html)

Предусмотрено три типа объектов-ссылок, каждый слабее предыдущего: *мягкие* (soft), *слабые* (weak) и *фантомные* (phantom). Каждый тип соответствует определённому уровню достижимости (reachability).

**SoftReference** (мягкие ссылки) предназначены для реализации кешей, чувствительных к памяти (memory-sensitive caches). Объект *мягко достижим*, если он не является строго достижимым, но может быть достигнут через мягкую ссылку. Мягкие ссылки очищаются сборщиком мусора по его усмотрению в ответ на нехватку памяти.

**WeakReference** (слабые ссылки) предназначены для реализации *канонизирующих отображений* (canonicalizing mappings), которые не препятствуют сборке мусора ключей (или значений). Слабые ссылки не препятствуют тому, чтобы объекты, на которые они ссылаются, стали финализируемыми, затем финализированными, и наконец — утилизированными. Слабые ссылки очищаются при первом обнаружении сборщиком мусора факта, что объект больше не является строго или мягко достижимым.

В классе `WeakHashMap` каждый ключ хранится косвенно в виде референта слабой ссылки. Хеш-таблица, использующая слабые ссылки для реализации слабых ключей, может опрашивать свою очередь ссылок при каждом обращении к таблице — именно так работает класс `WeakHashMap`.

#### Ответ от Claude

**Основное различие: момент очистки сборщиком мусора**

| Характеристика | `SoftReference` | `WeakReference` |
|---|---|---|
| Уровень достижимости | Softly reachable — не строго, но достижим через soft-ссылку | Weakly reachable — не строго и не мягко, достижим только через weak-ссылку |
| Когда GC очищает | Только при нехватке памяти, по усмотрению GC | При первом обнаружении, что объект больше не строго достижим, независимо от памяти |
| Гарантия очистки | До `OutOfMemoryError` JVM гарантирует очистку всех soft-ссылок | При следующем GC-цикле после потери строгой достижимости |
| Типичное применение | Кеш, чувствительный к памяти (изображения, результаты вычислений) | Канонизирующие отображения, слушатели событий без явной отписки |

Oracle Javadoc явно декларирует назначение каждого типа: «Soft references are for implementing memory-sensitive caches, weak references are for implementing canonicalizing mappings that do not prevent their keys (or values) from being reclaimed» (`java.lang.ref` package summary).

**Почему `WeakHashMap` использует именно `WeakReference`, а не `SoftReference`**

Семантическая цель `WeakHashMap` — автоматически удалять запись из карты тогда, когда ключ перестаёт использоваться в остальной части программы. Иными словами, карта не должна сама по себе удерживать ключ в памяти.

1. **`SoftReference` нарушила бы эту семантику.** Мягкая ссылка продлевает жизнь объекта до момента нехватки памяти. Ключ, который никто больше не использует, оставался бы в карте потенциально очень долго — пока JVM не окажется под давлением памяти. Это противоречит принципу «запись удаляется, когда ключ более не нужен».

2. **`WeakReference` обеспечивает правильное поведение.** Как только последняя сильная (strong) ссылка на ключ исчезает из остального кода, GC при следующем цикле очищает слабую ссылку, ключ попадает в `ReferenceQueue`, и `WeakHashMap` при следующем обращении к карте удаляет соответствующую запись.

3. **`WeakHashMap` предназначен для канонизирующих отображений и дополнительных данных.** Oracle Javadoc для `WeakReference` прямо указывает: «weak references are most often used to implement canonicalizing mappings». `WeakHashMap` как раз реализует такую семантику.

**Важная деталь: значения хранятся через сильные ссылки**

Значения в `WeakHashMap` хранятся через обычные сильные ссылки. Если значение само содержит сильную ссылку на ключ (прямо или косвенно), то ключ не будет собран мусором, даже если больше нигде не используется. В таких случаях значение следует обернуть в `WeakReference`.

**Практические примеры применения**

- `WeakReference` / `WeakHashMap`: дополнительные атрибуты объектов (метаданные DOM-узлов в браузере, слушатели событий, которые не требуют явной отписки).
- `SoftReference`: кеши изображений, результаты дорогих вычислений — данные стоит держать, пока есть память, но не принципиально.

**Источники, использованные для верификации:**
- [java.lang.ref (Java SE 24) — Oracle](https://docs.oracle.com/en/java/javase/24/docs/api/java.base/java/lang/ref/package-summary.html)
- [WeakHashMap (Java Platform SE 8) — Oracle](https://docs.oracle.com/javase/8/docs/api/java/util/WeakHashMap.html)
- [WeakReference (Java Platform SE 8) — Oracle](https://docs.oracle.com/javase/8/docs/api/java/lang/ref/WeakReference.html)

### Вопрос 51. Почему ArrayDeque предпочтительнее LinkedList при использовании в роли стека или очереди? Какова внутренняя структура ArrayDeque и как он обрабатывает null-элементы?

**Категория:** Collections · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [ArrayDeque (Java SE 17 & JDK 17) — docs.oracle.com](https://docs.oracle.com/en/java/javase/17/docs/api/java.base/java/util/ArrayDeque.html)

`ArrayDeque` — изменяемая массивная реализация интерфейса `Deque`. У очередей на основе массива нет ограничений по размеру: они растут по мере необходимости. Конструктор по умолчанию обеспечивает начальную ёмкость, достаточную для хранения 16 элементов.

Данный класс, скорее всего, будет быстрее класса `Stack` при использовании в роли стека и быстрее `LinkedList` при использовании в роли очереди. Большинство операций `ArrayDeque` выполняются за амортизированное константное время. Исключение составляют `remove`, `removeFirstOccurrence`, `removeLastOccurrence`, `contains`, `iterator.remove()` и bulk-операции — они работают за линейное время.

Null-элементы запрещены. Класс не является потокобезопасным.

#### Ответ от Claude

**Внутренняя структура**

`ArrayDeque` хранит элементы в циклическом (ring buffer) массиве объектов. Два указателя — `head` и `tail` — фиксируют начало и конец «активной» части буфера. Добавление в голову декрементирует `head`, добавление в хвост инкрементирует `tail`. При достижении границ массива индексы «оборачиваются» по модулю длины. Когда буфер заполняется, создаётся новый массив удвоенного размера (по умолчанию начальная ёмкость — 16).

Такой layout обеспечивает непрерывность элементов в памяти (с учётом кольцевой природы) и хорошую локальность кэша при последовательном обходе.

**Почему быстрее LinkedList**

| Аспект | ArrayDeque | LinkedList |
|---|---|---|
| Внутренняя структура | Циклический массив | Двусвязный список Node-объектов |
| Доступ к head/tail | O(1), прямой индекс | O(1), но через разыменование указателя |
| Аллокация при добавлении | Редкая (только при расширении) | Каждый раз — новый объект Node |
| Локальность кэша | Высокая (элементы рядом в памяти) | Низкая (Node-объекты разбросаны по куче) |
| Давление на GC | Минимальное | Каждая вставка создаёт Node-объект |

Официальный Javadoc Oracle прямо указывает: «likely to be faster than Stack when used as a stack, and faster than LinkedList when used as a queue».

**Null-элементы**

`ArrayDeque` запрещает null. Это архитектурное решение: `null` используется внутри некоторых операций (например, как sentinel-значение при поиске), и его разрешение привело бы к неоднозначности.

**Потокобезопасность**

`ArrayDeque` не является потокобезопасным. Для конкурентного использования применяется `ArrayBlockingQueue` (ограниченная ёмкость) или `LinkedBlockingDeque`.

**Когда всё же использовать LinkedList**

Если нужен `ListIterator` (двунаправленная итерация) или часто выполняется `remove()` в середине при уже известной позиции итератора — `LinkedList` корректнее. Для работы с концами коллекции `ArrayDeque` предпочтительнее почти всегда.

### Вопрос 52. В чём разница между Collections.unmodifiableList() и List.of() с точки зрения неизменяемости? Почему первый не гарантирует истинной иммутабельности?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Creating Unmodifiable Lists, Sets, and Maps (Java SE 11) — docs.oracle.com](https://docs.oracle.com/en/java/javase/11/core/creating-immutable-lists-sets-and-maps.html)

Неизменяемые коллекции (unmodifiable) не позволяют добавлять, удалять и заменять элементы. Попытка модификации выбрасывает `UnsupportedOperationException`. Они представляют собой реальные структуры данных (а не представления), реализованные через приватные классы за статическими фабричными методами.

При этом коллекция является по-настоящему неизменяемой только тогда, когда и она сама, и её элементы неизменяемы: «an unmodifiable collection is only immutable if the elements contained in the collection are immutable».

`List.of()` (Java 9+) возвращает поистине неизменяемый экземпляр. Null-значения не допускаются. Реализация обеспечивает меньший overhead: объект без накладных расходов на поддержку модификаций занимает значительно меньше памяти.

#### Ответ от Claude

**Collections.unmodifiableList() — «блокировочная обёртка»**

`Collections.unmodifiableList(list)` возвращает представление (view) исходного списка. Запросы на чтение делегируются оригинальному списку; запросы на изменение бросают `UnsupportedOperationException`. Ключевое ограничение: оригинальный список при этом остаётся изменяемым.

```java
List<String> original = new ArrayList<>(List.of("a", "b", "c"));
List<String> view = Collections.unmodifiableList(original);

// view.add("d"); // UnsupportedOperationException
original.add("d");           // Успешно!
System.out.println(view);    // [a, b, c, d] — view изменился!
```

Это означает, что если кто-то держит ссылку на оригинальный список, он может незаметно изменить содержимое «неизменяемого» представления. Потокобезопасность не гарантируется — конкурентное изменение оригинала через другую ссылку опасно.

**List.of() — истинно иммутабельная коллекция (Java 9+)**

`List.of(...)` создаёт полностью независимую, истинно неизменяемую коллекцию. Нет ссылки на внешний изменяемый список, которую кто-то мог бы задействовать.

```java
List<String> immutable = List.of("a", "b", "c");
// immutable.add("d"); // UnsupportedOperationException
// Нет внешнего оригинала — изменить нельзя никак
```

**Сводное сравнение**

| Характеристика | `Collections.unmodifiableList()` | `List.of()` |
|---|---|---|
| Доступна с | Java 1.2 | Java 9 |
| Тип | View (обёртка) | Независимый объект |
| Защита от изменения через оригинал | Нет | Да (оригинала нет) |
| Допустимость null | Зависит от базового списка | Нет (NullPointerException) |
| Потокобезопасность | Нет (зависит от исходного списка) | Да (нет изменяемых ссылок) |
| Потребление памяти | Больше (живёт поверх оригинала) | Меньше (компактная реализация) |
| Дублирование элементов | Разрешено | Разрешено (в отличие от Set.of) |
| Сериализуемость | Да | Зависит от версии JDK |

**List.copyOf() — лучшее из двух вариантов**

Если есть уже существующий список и нужна истинная неизменяемая копия: `List.copyOf(original)` (Java 10+). Если оригинал уже является неизменяемым экземпляром `List.of`, возвращается та же ссылка (без лишнего копирования).

**Практическая рекомендация**

Для нового кода, где нужна гарантированная иммутабельность, предпочитайте `List.of()` / `List.copyOf()`. `Collections.unmodifiableList()` остаётся полезным при работе с легаси-кодом или когда нужно контролируемое изменение исходного списка через другие пути.

### Вопрос 53. Что такое BlockingQueue? Какие четыре формы операций вставки/извлечения она предоставляет, и как реализуется паттерн «производитель–потребитель»?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [BlockingQueue (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/BlockingQueue.html)

`BlockingQueue` — это `Queue`, которая дополнительно поддерживает операции, ожидающие появления элемента в очереди при извлечении и ожидающие появления свободного места при вставке.

Методы `BlockingQueue` существуют в четырёх формах с различным поведением при невозможности немедленного выполнения операции:

| | Выбрасывает исключение | Возвращает специальное значение | Блокирует | Ограничена по времени |
|---|---|---|---|---|
| Вставка | `add(e)` | `offer(e)` | `put(e)` | `offer(e, time, unit)` |
| Извлечение | `remove()` | `poll()` | `take()` | `poll(time, unit)` |
| Проверка | `element()` | `peek()` | — | — |

`BlockingQueue` реализации потокобезопасны. Null-элементы запрещены: использование null в качестве «poison pill» (сигнальное значение конца потока) невозможно — для этого нужны специальные sentinel-объекты.

#### Ответ от Claude

**Что такое BlockingQueue**

`BlockingQueue<E>` (пакет `java.util.concurrent`) — интерфейс, расширяющий `Queue<E>` операциями с блокирующей семантикой. Предназначен для реализации паттерна «производитель–потребитель» без ручной синхронизации: производители кладут элементы, а потребители извлекают, и оба могут блокироваться при необходимости.

**Четыре формы операций**

| Операция | Выбрасывает исключение | Возвращает значение (не блокирует) | Блокирует бесконечно | Ограничена по времени |
|---|---|---|---|---|
| Вставка | `add(e)` — `IllegalStateException` если полна | `offer(e)` — false если полна | `put(e)` — ждёт места | `offer(e, long, TimeUnit)` |
| Извлечение | `remove()` — `NoSuchElementException` если пуста | `poll()` — null если пуста | `take()` — ждёт элемента | `poll(long, TimeUnit)` |

**Ключевые блокирующие методы**

- `put(E e)` — помещает элемент в очередь, блокируя вызывающий поток, если очередь заполнена.
- `take()` — извлекает и удаляет голову очереди, блокируя вызывающий поток, если очередь пуста.

Оба метода реагируют на `InterruptedException`: прерывание потока снимает блокировку и выбрасывает исключение.

**Паттерн «производитель–потребитель»**

```java
BlockingQueue<Task> queue = new ArrayBlockingQueue<>(100);

// Производитель
new Thread(() -> {
    try {
        while (true) {
            queue.put(generateTask()); // блокирует при полной очереди
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    }
}).start();

// Потребитель
new Thread(() -> {
    try {
        while (true) {
            Task task = queue.take(); // блокирует при пустой очереди
            process(task);
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    }
}).start();
```

Без `BlockingQueue` потребовались бы явный `synchronized`, `wait()` и `notifyAll()` — что многословно и подвержено ошибкам.

**Основные реализации**

| Реализация | Особенность |
|---|---|
| `ArrayBlockingQueue` | Ограниченная ёмкость, один lock |
| `LinkedBlockingQueue` | Опционально ограниченная, два lock (put/take) |
| `PriorityBlockingQueue` | Приоритетная, неограниченная |
| `SynchronousQueue` | Нет буфера — каждый put ждёт take |
| `DelayQueue` | Элементы доступны только по истечении задержки |

**Важные ограничения**

- Null-элементы запрещены во всех реализациях.
- Методы `size()`, `contains()`, `drainTo()` могут давать неточные результаты при конкурентном использовании.
- `BlockingQueue` сама по себе не задаёт механизм «завершения» потока — для этого используют poison pill-объекты или `InterruptedException`.

### Вопрос 54. Чем ArrayBlockingQueue отличается от LinkedBlockingQueue? Как их механизм блокировки влияет на пропускную способность при многопоточной записи и чтении одновременно?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ArrayBlockingQueue vs. LinkedBlockingQueue — Baeldung](https://www.baeldung.com/java-arrayblockingqueue-vs-linkedblockingqueue)

`ArrayBlockingQueue` — ограниченная (bounded) очередь на основе массива. Ёмкость задаётся при создании и не меняется. Использует **один** `ReentrantLock` для операций `put` и `take`.

`LinkedBlockingQueue` — опционально ограниченная очередь на основе связных узлов. Если ёмкость не задана, устанавливается `Integer.MAX_VALUE`. Использует **два** отдельных `ReentrantLock` — один для `put`, другой для `take`. Узлы создаются динамически при каждой вставке.

Связные очереди типично обеспечивают более высокую пропускную способность, чем очереди на основе массива, но менее предсказуемую производительность в большинстве конкурентных приложений.

#### Ответ от Claude

**Внутренняя структура**

`ArrayBlockingQueue` хранит элементы в массиве фиксированного размера. При создании вся память выделяется сразу — даже если реальное число элементов меньше ёмкости.

`LinkedBlockingQueue` хранит элементы в динамически создаваемых узлах `Node<E>`. Каждая операция `put` аллоцирует новый узел в куче; `take` — освобождает его, давая GC возможность его собрать.

**Механизм блокировки: одна vs две блокировки**

| Характеристика | `ArrayBlockingQueue` | `LinkedBlockingQueue` |
|---|---|---|
| Число блокировок | Один `ReentrantLock` | Два `ReentrantLock`: `putLock` и `takeLock` |
| Параллелизм put+take | Нет — оба ждут одного lock | Да — put и take не мешают друг другу |
| Проблема contention | Высокая при одновременных put/take | Ниже — разные lock-ы |

Именно два lock-а объясняют более высокую пропускную способность `LinkedBlockingQueue` в сценарии «производитель пишет, потребитель читает одновременно»: поток-производитель захватывает `putLock`, поток-потребитель — независимо захватывает `takeLock`. У `ArrayBlockingQueue` оба потока конкурируют за один общий lock.

**Параметр fairness (честность)**

`ArrayBlockingQueue` поддерживает опциональную политику честности: `new ArrayBlockingQueue<>(size, true)` гарантирует FIFO-порядок среди ожидающих потоков. Это снижает пропускную способность, но устраняет starvation. `LinkedBlockingQueue` такой опции не предоставляет.

**Память и GC**

`ArrayBlockingQueue`: фиксированный массив, давление на GC минимально, но память выделена заранее в полном объёме.

`LinkedBlockingQueue`: каждый элемент — отдельный объект Node. При высокой интенсивности операций нагрузка на GC выше. Зато память расходуется пропорционально текущему числу элементов.

**Когда что выбирать**

- `ArrayBlockingQueue`: когда важна предсказуемая ёмкость и ограниченная нагрузка на GC; когда нужна опция fairness.
- `LinkedBlockingQueue`: когда производители и потребители работают одновременно при высокой интенсивности; когда ёмкость не нужно фиксировать заранее.

### Вопрос 55. Что такое EnumSet? Как он реализован внутри (RegularEnumSet, JumboEnumSet) и почему обеспечивает производительность выше, чем HashSet?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [EnumSet (Java SE 21 & JDK 21) — docs.oracle.com](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/util/EnumSet.html)

`EnumSet` — специализированная реализация `Set` для работы с enum-типами. Все элементы должны принадлежать единственному enum-типу, указанному явно или неявно при создании множества.

Внутри enum sets представлены в виде **битовых векторов**. Это представление крайне компактно и эффективно. Все базовые операции выполняются за константное время. Они, скорее всего (хотя и не гарантированно), будут значительно быстрее, чем аналоги в `HashSet`. Даже bulk-операции выполняются за константное время, если их аргументом является тоже enum set.

Null-элементы не допускаются. Итератор обходит элементы в порядке их объявления в enum.

#### Ответ от Claude

**Что такое EnumSet**

`EnumSet<E extends Enum<E>>` — реализация `Set`, разработанная специально для хранения констант enum-типа. Создаётся только через фабричные методы: `EnumSet.of(...)`, `EnumSet.allOf(...)`, `EnumSet.noneOf(...)`, `EnumSet.range(...)`, `EnumSet.complementOf(...)`.

**Внутренняя реализация: два варианта**

JVM автоматически выбирает реализацию в зависимости от числа констант в enum:

**`RegularEnumSet`** — используется, когда enum содержит **не более 64 констант**. Всё множество хранится в одном поле типа `long` (64 бита). Каждому значению enum соответствует один бит: `i`-я константа enum занимает `i`-й бит. Операции `add`, `contains`, `remove` реализованы через битовые операции OR, AND, NOT.

**`JumboEnumSet`** — используется, если констант **больше 64**. Внутри — массив `long[]`, где первые 64 константы хранятся в `long[0]`, следующие 64 — в `long[1]` и т.д. Операции требуют дополнительного вычисления индекса в массиве.

**Почему быстрее HashSet**

| Операция | HashSet | EnumSet |
|---|---|---|
| `contains(e)` | Вычислить hashCode, найти бакет, сравнить equals | Bitwise AND одного long — одна инструкция |
| `add(e)` | Найти бакет, возможно — разрешить коллизию | Bitwise OR — одна инструкция |
| `containsAll(other EnumSet)` | O(n) — перебор элементов | O(1) — AND двух long |
| Память | Объекты Node + хэш-таблица | Один long для ≤64 констант |

Все bulk-операции (`addAll`, `removeAll`, `retainAll`, `containsAll`) между двумя `EnumSet` выполняются за O(1) как битовые операции между `long`-значениями. Это недостижимо для любой другой реализации `Set`.

**Фабричные методы**

```java
enum Day { MON, TUE, WED, THU, FRI, SAT, SUN }

EnumSet<Day> weekdays = EnumSet.range(Day.MON, Day.FRI);
EnumSet<Day> weekend  = EnumSet.complementOf(weekdays);
EnumSet<Day> all      = EnumSet.allOf(Day.class);
EnumSet<Day> none     = EnumSet.noneOf(Day.class);
EnumSet<Day> selected = EnumSet.of(Day.MON, Day.WED, Day.FRI);
```

**Ограничения**

- Null-элементы запрещены (NullPointerException).
- Не синхронизирован. Для многопоточного использования: `Collections.synchronizedSet(EnumSet.noneOf(...))`.
- Порядок итерации совпадает с порядком объявления констант в enum — детерминирован.

### Вопрос 56. Какие навигационные методы добавляет NavigableMap по сравнению с SortedMap? Чем subMap() в NavigableMap отличается от subMap() в SortedMap?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Java NavigableMap — jenkov.com](https://jenkov.com/tutorials/java-collections/navigablemap.html)

`NavigableMap` расширяет `SortedMap` навигационными методами для нахождения ближайшего ключа:

- `lowerKey(K key)` — наибольший ключ, **строго меньший** переданного.
- `floorKey(K key)` — наибольший ключ, **меньший или равный** переданному.
- `ceilingKey(K key)` — наименьший ключ, **больший или равный** переданному.
- `higherKey(K key)` — наименьший ключ, **строго больший** переданного.

Методы `subMap`, `headMap`, `tailMap` в `NavigableMap` принимают дополнительные булевы аргументы, задающие включительность/исключительность границ. Аналогичные методы в `SortedMap` всегда используют полуоткрытый интервал (нижняя граница включается, верхняя — нет). `NavigableMap` также предоставляет `descendingKeySet()` и `descendingMap()` для обхода в обратном порядке.

#### Ответ от Claude

**Что добавляет NavigableMap**

`NavigableMap<K, V>` (Java 6+, реализован `TreeMap`) расширяет `SortedMap<K, V>` четырьмя группами навигационных возможностей.

**1. Методы поиска ближайшего ключа**

| Метод | Возвращает |
|---|---|
| `lowerKey(k)` | Наибольший ключ строго меньше `k` (аналог `<`) |
| `floorKey(k)` | Наибольший ключ меньше или равен `k` (аналог `<=`) |
| `ceilingKey(k)` | Наименьший ключ больше или равен `k` (аналог `>=`) |
| `higherKey(k)` | Наименьший ключ строго больше `k` (аналог `>`) |
| `lowerEntry(k)` | Аналог `lowerKey`, но возвращает `Map.Entry` |
| `floorEntry(k)` / `ceilingEntry(k)` / `higherEntry(k)` | Аналогично, возвращают `Map.Entry` |

Все методы возвращают `null`, если подходящего ключа нет.

**2. Методы первого/последнего элемента с удалением**

- `pollFirstEntry()` — извлекает и удаляет запись с наименьшим ключом.
- `pollLastEntry()` — извлекает и удаляет запись с наибольшим ключом.

**3. Методы диапазонов с настраиваемыми границами**

`SortedMap.subMap(fromKey, toKey)` всегда возвращает полуоткрытый интервал `[fromKey, toKey)` — нижняя граница включается, верхняя — нет. Изменить это нельзя.

`NavigableMap.subMap(fromKey, fromInclusive, toKey, toInclusive)` принимает явные флаги включительности:

```java
TreeMap<Integer, String> map = new TreeMap<>();
// от 2 включительно до 4 включительно — [2, 4]
NavigableMap<Integer, String> sub = map.subMap(2, true, 4, true);

// от 2 исключительно до 4 исключительно — (2, 4)
NavigableMap<Integer, String> sub2 = map.subMap(2, false, 4, false);
```

Аналогично расширены `headMap(toKey, inclusive)` и `tailMap(fromKey, inclusive)`.

**4. Обратный порядок**

- `descendingKeySet()` — `NavigableSet` с ключами в убывающем порядке.
- `descendingMap()` — представление карты с обратным порядком ключей (все изменения синхронизированы с оригиналом).

**Практический пример**

```java
TreeMap<Integer, String> prices = new TreeMap<>();
prices.put(10, "бронза"); prices.put(50, "серебро"); prices.put(100, "золото");

// Наибольший порог, не превышающий 75:
Map.Entry<Integer, String> tier = prices.floorEntry(75); // key=50, "серебро"

// Диапазон цен от 10 до 50 включительно:
NavigableMap<Integer, String> range = prices.subMap(10, true, 50, true);
```

### Вопрос 57. Какие алгоритмы предоставляет утилитный класс Collections? Назовите ключевые методы и опишите, чем sort() отличается от binarySearch() по предусловиям.

**Категория:** Collections · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Lesson: Algorithms (The Java Tutorials — Collections) — docs.oracle.com](https://docs.oracle.com/javase/tutorial/collections/algorithms/index.html)

Класс `Collections` содержит полиморфные алгоритмы — реализации многократно используемой функциональности, поставляемые в JDK. Большинство алгоритмов работают с объектами `List`, но некоторые — с произвольными `Collection`.

**Сортировка.** `sort(List)` реорганизует `List` в возрастающем порядке. Используется алгоритм слияния (merge sort) — гарантированная сложность O(n log n). Алгоритм стабилен (равные элементы не меняют взаимного порядка).

**Поиск.** `binarySearch(List, key)` выполняет бинарный поиск в отсортированном `List`. Первая форма предполагает, что список упорядочен по естественному порядку элементов. При нахождении возвращает индекс; при отсутствии — `-(insertion point) - 1`. Если список не отсортирован — результат не определён.

**Прочие алгоритмы:** `reverse(List)`, `shuffle(List)`, `fill(List, value)`, `copy(List dest, List src)`, `swap(List, i, j)`, `addAll(Collection, T...)`, `frequency(Collection, element)`, `disjoint(Collection, Collection)`.

#### Ответ от Claude

**Обзор класса Collections**

`java.util.Collections` — утилитный класс (все методы статические), содержащий алгоритмы и фабричные методы для коллекций. Делится на три категории: алгоритмы, оборачивающие методы (synchronizedXxx, unmodifiableXxx), вспомогательные методы (singleton, emptyList, nCopies).

**Ключевые алгоритмы**

| Метод | Что делает | Сложность |
|---|---|---|
| `sort(List<T>)` | Сортировка в естественном порядке | O(n log n) |
| `sort(List<T>, Comparator<T>)` | Сортировка с компаратором | O(n log n) |
| `binarySearch(List, key)` | Бинарный поиск в **отсортированном** списке | O(log n) |
| `reverse(List)` | Разворот порядка элементов | O(n) |
| `shuffle(List)` | Случайная перестановка | O(n) |
| `fill(List, value)` | Заполнение все элементов значением | O(n) |
| `copy(List dest, List src)` | Копирование src в dest | O(n) |
| `swap(List, i, j)` | Обмен двух элементов | O(1) |
| `frequency(Collection, elem)` | Число вхождений элемента | O(n) |
| `disjoint(Collection, Collection)` | Нет общих элементов? | O(n) |
| `min(Collection)` / `max(Collection)` | Минимум / максимум | O(n) |
| `nCopies(int n, T o)` | Неизменяемый список из n копий объекта | O(1) |
| `singletonList(T o)` | Неизменяемый список из одного элемента | O(1) |
| `emptyList()` | Неизменяемый пустой список | O(1) |

**Различие sort() и binarySearch(): предусловия**

`sort()` — не требует предусловий на порядок, сам устанавливает порядок. Работает на любом `List`.

`binarySearch()` — **требует**, чтобы список был **предварительно отсортирован** в том же порядке, что и компаратор поиска. Если список не отсортирован, результат непредсказуем — метод не проверяет это условие. Официальная документация прямо указывает: «The list must be sorted into ascending order prior to making this call». Типичная ошибка:

```java
List<Integer> list = Arrays.asList(3, 1, 4, 1, 5);
Collections.binarySearch(list, 4); // непредсказуемый результат (список не отсортирован)

Collections.sort(list);
int idx = Collections.binarySearch(list, 4); // корректно: возвращает 3
```

**Возвращаемое значение binarySearch при отсутствии элемента**

При отсутствии ключа возвращается `-(insertion point) - 1`, где insertion point — позиция, куда элемент был бы вставлен, чтобы список остался отсортированным. Смещение `-1` гарантирует, что результат всегда отрицателен, даже если insertion point равен 0.

### Вопрос 58. Что такое Spliterator? Чем он отличается от Iterator, какие характеристики (SIZED, ORDERED, DISTINCT и др.) он декларирует, и как используется параллельными потоками?

**Категория:** Collections · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Spliterator (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/util/Spliterator.html)

`Spliterator` — объект для обхода и разделения элементов источника. Источником может быть массив, `Collection`, канал ввода-вывода или генерирующая функция.

`Spliterator` покрывает либо некоторую часть, либо все элементы источника. Обход достигается через `tryAdvance()` (по одному элементу) или `forEachRemaining()` (оптом). Разделение через `trySplit()` — новый `Spliterator`, покрывающий префикс, тогда как текущий покрывает суффикс.

Характеристики (`characteristics()`):
- `ORDERED` — порядок обхода определён
- `DISTINCT` — все элементы различны (как в `Set`)
- `SORTED` — элементы обходятся в отсортированном порядке
- `SIZED` — `estimateSize()` возвращает точное число элементов до исчерпания
- `SUBSIZED` — все дочерние сплитераторы тоже `SIZED`
- `NONNULL` — элементы не содержат null
- `IMMUTABLE` — источник не может быть структурно изменён
- `CONCURRENT` — источник безопасно доступен для конкурентной модификации

#### Ответ от Claude

**Что такое Spliterator и зачем он нужен**

`Spliterator<T>` (Java 8) — это интерфейс, реализующий «разделяемый итератор» (splittable iterator). Его главное назначение: поддерживать эффективный **параллельный обход** коллекции путём рекурсивного разбиения источника данных на независимые части.

**Сравнение с Iterator**

| Аспект | Iterator | Spliterator |
|---|---|---|
| Обход | `hasNext()` + `next()` | `tryAdvance()` + `forEachRemaining()` |
| Разбиение | Нет | `trySplit()` — возвращает «левую половину» |
| Параллелизм | Не предназначен | Основа parallel streams |
| Информация о размере | Нет | `estimateSize()` |
| Характеристики источника | Нет | Набор флагов characteristics |
| Накладные расходы | Выше (два вызова per элемент) | Ниже (один вызов per элемент) |

**Ключевые методы**

- `tryAdvance(Consumer<T> action)` — выполняет `action` над следующим элементом; возвращает `false`, если элементов больше нет.
- `forEachRemaining(Consumer<T> action)` — обходит все оставшиеся элементы.
- `trySplit()` — возвращает новый `Spliterator`, покрывающий часть элементов, или `null`, если разбиение невозможно.
- `estimateSize()` — оценочное число оставшихся элементов.

**Характеристики (characteristics)**

Характеристики — битовые флаги, сообщающие фреймворку потоков свойства источника:

| Флаг | Значение | Где встречается |
|---|---|---|
| `SIZED` | Точное число элементов известно | `ArrayList`, массивы |
| `SUBSIZED` | Все дочерние сплитераторы тоже `SIZED` | `ArrayList` |
| `ORDERED` | Порядок обхода определён | `List`, `SortedSet` |
| `DISTINCT` | Нет дублей | `Set` |
| `SORTED` | Обходится в заданном порядке | `TreeSet`, `TreeMap` |
| `NONNULL` | Нет null-элементов | `ArrayDeque`, `EnumSet` |
| `IMMUTABLE` | Источник не изменяется | `List.of()`, массивы через `Arrays.spliterator` |
| `CONCURRENT` | Источник изменяем конкурентно | `ConcurrentHashMap` |

**Роль в параллельных потоках**

При вызове `collection.parallelStream()` фреймворк:
1. Получает `Spliterator` через `spliterator()`.
2. Рекурсивно вызывает `trySplit()`, создавая поддерево задач `ForkJoinTask`.
3. Каждый подзадача обрабатывает свой `Spliterator` независимо.
4. Характеристики влияют на оптимизации: `SIZED` позволяет точно распределить нагрузку; `DISTINCT` + `SORTED` позволяют убрать шаг дедупликации; `ORDERED` обязывает сохранять порядок в `findFirst()`.

```java
// Пример: parallel stream использует Spliterator под капотом
long count = IntStream.range(0, 1_000_000)
    .parallel()
    .filter(n -> n % 2 == 0)
    .count();
```

**Стандартные коллекции и их Spliterator**

`ArrayList` — `SIZED`, `SUBSIZED`, `ORDERED`. `HashSet` — `SIZED`, `DISTINCT`. `TreeSet` — `SIZED`, `DISTINCT`, `SORTED`, `ORDERED`. `ConcurrentHashMap.keySet()` — `CONCURRENT`, `DISTINCT`, `NONNULL`.

### Вопрос 59. Гарантирует ли Collections.synchronizedMap() потокобезопасность при итерации? Почему при обходе через Iterator необходима внешняя синхронизация?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Collections (Java SE 17 & JDK 17) — docs.oracle.com](https://docs.oracle.com/en/java/javase/17/docs/api/java.base/java/util/Collections.html)

Согласно официальной документации метода `synchronizedMap`:

> «Для гарантии последовательного доступа крайне важно, чтобы весь доступ к исходной карте осуществлялся только через возвращённую карту. Пользователь должен вручную синхронизироваться по возвращённой карте при обходе через любое из представлений коллекции (Set или Collection):»

```java
Map m = Collections.synchronizedMap(new HashMap());
    ...
Set s = m.keySet(); // синхронизация не нужна
    ...
synchronized (m) {  // синхронизация по карте m, а не по s!
    Iterator i = s.iterator(); // обязательно внутри synchronized-блока
    while (i.hasNext())
        foo(i.next());
}
```

> «Несоблюдение этого требования может привести к недетерминированному поведению.»

#### Ответ от Claude

**Что синхронизирует synchronizedMap, а что — нет**

`Collections.synchronizedMap(map)` оборачивает каждый **отдельный** метод объекта-обёртки в `synchronized(mutex)` блок. Это означает:

- `put()`, `get()`, `remove()`, `size()`, `containsKey()` и т.д. — каждый из них атомарен в рамках одного вызова.
- Итерация через `Iterator` — **нет**. Итерация — это последовательность множества вызовов `next()`, и между этими вызовами монитор не удерживается.

**Почему итерация без внешней синхронизации опасна**

Пусть поток A итерирует по карте (вызов `next()` — синхронизован, но краткосрочно). Поток B в промежутке между двумя вызовами `next()` у потока A вставляет или удаляет элемент. Результат зависит от реализации базовой карты (`HashMap`): `modCount` изменится, и следующий вызов `next()` у потока A выбросит `ConcurrentModificationException` — даже несмотря на то что каждый отдельный `put`/`get` был синхронизован.

**Правильный паттерн**

```java
Map<String, String> syncMap = Collections.synchronizedMap(new HashMap<>());

// НЕПРАВИЛЬНО — между next() другой поток может изменить карту:
for (String key : syncMap.keySet()) {
    process(syncMap.get(key)); // ConcurrentModificationException!
}

// ПРАВИЛЬНО — удерживаем блокировку на весь период итерации:
synchronized (syncMap) {
    Iterator<String> it = syncMap.keySet().iterator();
    while (it.hasNext()) {
        process(syncMap.get(it.next())); // безопасно
    }
}
```

Блокировка берётся именно по объекту `syncMap` (обёртка использует себя как mutex).

**Ключевое отличие от ConcurrentHashMap**

`ConcurrentHashMap` предоставляет weakly-consistent итераторы, которые не бросают `ConcurrentModificationException` и не требуют внешней синхронизации. Однако они могут не увидеть изменений, внесённых после создания итератора. `synchronizedMap` при правильной внешней блокировке видит всё — но требует удержания блокировки на весь цикл, что блокирует других читателей.

**Итог**

`Collections.synchronizedMap()` синхронизирует каждый **отдельный вызов метода**, но не **составные операции** вроде итерации. Для безопасной итерации необходимо явно брать блокировку по объекту-обёртке на весь период обхода. Если итерация важна и нужна высокая конкурентность — используйте `ConcurrentHashMap`.

### Вопрос 60. Что такое List.copyOf(), Set.copyOf(), Map.copyOf() добавленные в Java 10? Чем они отличаются от List.of() и когда возвращают ту же ссылку без копирования?

**Категория:** Collections · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Creating Unmodifiable Lists, Sets, and Maps (Java SE 11) — docs.oracle.com](https://docs.oracle.com/en/java/javase/11/core/creating-immutable-lists-sets-and-maps.html)

Метод `copyOf(collection)` создаёт полностью независимую неизменяемую коллекцию:

- Если коллекция-источник изменяема — создаётся независимая копия.
- Если коллекция-источник уже является неизменяемой (создана через `List.of()`, `Set.of()` и т.д.) — возвращается **та же ссылка** без создания новой копии.

Это позволяет избежать избыточного копирования в сценарии «defensive copy» без дополнительных проверок со стороны кода.

#### Ответ от Claude

**Что такое copyOf-методы**

`List.copyOf(Collection<E>)`, `Set.copyOf(Collection<E>)`, `Map.copyOf(Map<K,V>)` — статические фабричные методы, добавленные в Java 10 (JEP 269 — фактически обновление). Возвращают неизменяемые коллекции, содержащие те же элементы, что и источник, но независимые от него.

**Чем отличаются от List.of()**

| Аспект | `List.of(a, b, c)` | `List.copyOf(existing)` |
|---|---|---|
| Источник | Явно перечисленные элементы | Уже существующая коллекция |
| Использование | Инициализация константного набора | Защитное копирование переданной коллекции |
| Null-элементы | Запрещены (NPE) | Запрещены (NPE) — копируются без null |
| Порядок | Определён переданными аргументами | Сохраняется порядок итерации источника |

**Оптимизация: возврат той же ссылки**

`List.copyOf(source)` проверяет, является ли источник уже неизменяемой коллекцией собственной реализации. Если да — возвращает ту же ссылку:

```java
List<String> original = List.of("a", "b", "c"); // неизменяемый
List<String> copy = List.copyOf(original);

System.out.println(original == copy); // true — та же ссылка!

List<String> mutable = new ArrayList<>(List.of("a", "b", "c"));
List<String> copy2 = List.copyOf(mutable);
System.out.println(mutable == copy2); // false — создана копия
```

Это оптимизация важна в «defensive copy» сценариях: если метод принимает `Collection` и хочет создать неизменяемую версию, `copyOf` автоматически избегает двойного копирования.

**Практический паттерн — защитное копирование**

```java
public class Registry {
    private final List<String> entries;

    public Registry(Collection<String> source) {
        // copyOf: если source уже List.of — нет лишней аллокации;
        // если mutable — создаётся независимая копия
        this.entries = List.copyOf(source);
    }

    public List<String> getEntries() {
        return entries; // безопасно возвращать — неизменяема
    }
}
```

**Set.copyOf() и Map.copyOf()**

`Set.copyOf(collection)` — аналогично, но дубликаты в исходной коллекции запрещены (бросает `IllegalArgumentException`). `Map.copyOf(map)` — создаёт неизменяемую копию карты; дубликаты ключей запрещены (что невозможно в корректном `Map`).

**Итог**

`copyOf`-методы — «умные» защитные копировщики: избегают лишней аллокации для уже иммутабельных коллекций и гарантируют независимость от источника для изменяемых. `List.of()` — для создания с нуля; `List.copyOf()` — для создания из существующей коллекции.
