### Вопрос 1. В чём разница между процессом и потоком? Что такое контекстное переключение?

**Категория:** Concurrency · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Process vs. Thread | Baeldung on Computer Science](https://www.baeldung.com/cs/process-vs-thread)

Процесс — это выполняющаяся программа вместе со всеми её данными, ресурсами (файлами, сетевыми соединениями) и метаинформацией, которую хранит операционная система. Поток (thread) — это «облегчённый процесс»: у него есть собственный стек и счётчик команд, но в отличие от процесса он разделяет адресное пространство памяти с другими потоками того же процесса.

Ключевые различия, которые выделяет источник:

- **Изоляция памяти.** Потоки одного процесса используют общую память, поэтому повреждение памяти одним потоком может затронуть все остальные. ОС изолирует процессы друг от друга: сбой одного не влияет на другие.
- **Коммуникация.** Взаимодействие между потоками быстрее и проще, чем между процессами (IPC).
- **Контекстное переключение.** Переключение контекста (context switch) — это системное событие, при котором ОС снимает текущую задачу с CPU и передаёт его другой, сохраняя состояние первой (регистры, указатель стека, счётчик команд и т.д.). Переключение между потоками значительно дешевле, чем между процессами, потому что потоки разделяют адресное пространство и большую часть ресурсов — сохранять нужно только «приватные» данные потока.

---

#### Ответ от Claude

**Процесс** — это самостоятельный экземпляр выполняющейся программы. ОС выделяет каждому процессу изолированное адресное пространство, собственные ресурсы (файловые дескрипторы, сетевые сокеты), а также хранит PCB (Process Control Block) с метаинформацией: идентификатор процесса, состояние, таблицу страниц памяти и т.д.

**Поток (Thread)** — единица выполнения внутри процесса. Все потоки одного процесса:
- разделяют общее адресное пространство (кучу, статические переменные, код);
- имеют собственный стек вызовов и счётчик команд (program counter);
- имеют собственный набор регистров.

Ключевые различия:

| Характеристика | Процесс | Поток |
|---|---|---|
| Память | Изолированное адресное пространство | Общее адресное пространство процесса |
| Создание/завершение | Дорогостоящее (fork/exec) | Дешевле |
| Изоляция при сбоях | Высокая — сбой одного не влияет на другие | Низкая — сбой потока может «уронить» весь процесс |
| Коммуникация | IPC (pipes, sockets, shared memory) | Разделяемая память, быстрее и проще |
| Переключение контекста | Дорогостоящее | Дешевле |

**Контекстное переключение (Context Switch)** — процедура, при которой ОС приостанавливает текущий поток/процесс и передаёт управление CPU другому:

1. Сохранение состояния текущего потока: значения регистров (включая PC и SP), флаги процессора.
2. Обновление PCB/TCB (Thread Control Block) с новым состоянием.
3. Загрузка состояния следующего потока из его TCB.
4. При переключении между процессами — дополнительно смена таблицы страниц памяти (TLB-flush), что значительно дороже.

В контексте JVM: Java-потоки (platform threads) отображаются 1:1 на потоки ОС (native threads), поэтому переключение контекста происходит на уровне ОС. Это одна из причин, по которой в Java 21 появились виртуальные потоки (Virtual Threads / Project Loom) — они управляются JVM и переключаются значительно дешевле.

**Почему это важно при разработке на Java:** создание большого числа платформенных потоков ограничено ресурсами ОС и стоимостью переключения контекста. При высоких нагрузках (тысячи конкурентных задач) предпочтительны пулы потоков (`ExecutorService`) или виртуальные потоки (Java 21+).

### Вопрос 2. Какие способы создания потока существуют в Java? В чём отличие реализации Runnable от наследования Thread?

**Категория:** Concurrency · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Defining and Starting a Thread (The Java™ Tutorials > Essential Java Classes > Concurrency)](https://docs.oracle.com/javase/tutorial/essential/concurrency/runthread.html)

Официальная документация Oracle описывает два основных способа создания потока:

**Способ 1 — реализация интерфейса `Runnable`:**
```java
public class HelloRunnable implements Runnable {
    public void run() {
        System.out.println("Hello from a thread!");
    }
    public static void main(String[] args) {
        (new Thread(new HelloRunnable())).start();
    }
}
```

**Способ 2 — наследование от класса `Thread`:**
```java
public class HelloThread extends Thread {
    public void run() {
        System.out.println("Hello from a thread!");
    }
    public static void main(String[] args) {
        (new HelloThread()).start();
    }
}
```

Оба способа требуют вызова `start()`, а не `run()`, — только `start()` запускает новый поток. Документация явно рекомендует первый вариант (Runnable), поскольку он более гибкий: позволяет задаче наследоваться от другого класса, что невозможно при наследовании `Thread` (Java не поддерживает множественное наследование). Кроме того, `Runnable`-подход лучше совместим с высокоуровневыми API управления потоками (`ExecutorService` и т.д.).

---

#### Ответ от Claude

В Java существует несколько способов создания и запуска потоков:

**1. Наследование от `Thread`**
```java
public class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("Running in: " + Thread.currentThread().getName());
    }
}
new MyThread().start();
```

**2. Реализация `Runnable` (рекомендуется)**
```java
public class MyTask implements Runnable {
    @Override
    public void run() {
        System.out.println("Running via Runnable");
    }
}
new Thread(new MyTask()).start();
```

**3. Лямбда-выражение (Runnable как функциональный интерфейс)**
```java
new Thread(() -> System.out.println("Lambda thread")).start();
```

**4. `Callable` + `FutureTask` (когда нужен результат)**
```java
Callable<Integer> task = () -> 42;
FutureTask<Integer> future = new FutureTask<>(task);
new Thread(future).start();
Integer result = future.get(); // блокирует до получения результата
```

**5. `ExecutorService` (пул потоков, production-подход)**
```java
ExecutorService executor = Executors.newFixedThreadPool(4);
executor.submit(() -> System.out.println("Thread pool task"));
executor.shutdown();
```

**6. Виртуальные потоки (Java 21+)**
```java
Thread.ofVirtual().start(() -> System.out.println("Virtual thread"));
```

---

**Ключевые отличия `Runnable` от наследования `Thread`:**

| Аспект | `extends Thread` | `implements Runnable` |
|---|---|---|
| Наследование | Класс не может расширять другой класс | Класс может наследоваться от произвольного класса |
| Разделение ответственностей | Задача и поток объединены | Задача отделена от механизма запуска |
| Совместимость с пулами потоков | Неудобно | Нативная поддержка (`ExecutorService`) |
| Принцип дизайна | Нарушает «IS-A» (объект не является потоком по смыслу) | Соответствует «composition over inheritance» |
| Переиспользование задачи | Один объект — один поток | Один `Runnable` можно передать в несколько потоков |

**Важная ошибка-ловушка:** вызов `thread.run()` вместо `thread.start()` не создаёт новый поток — метод `run()` исполняется в вызывающем потоке.

**Рекомендация:** для большинства задач использовать `Runnable` (или лямбду) в сочетании с `ExecutorService`. `Callable` — когда нужен результат или обработка исключений. Прямое создание потоков (`new Thread(...)`) оправдано только для простых одноразовых задач.

### Вопрос 3. Что такое ключевое слово volatile и какие гарантии оно даёт? В чём его отличие от synchronized?

**Категория:** Concurrency · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Java Volatile Keyword — jenkov.com](https://jenkov.com/tutorials/java-concurrency/volatile.html)

Ключевое слово `volatile` решает проблему видимости переменных между потоками. Когда переменная объявлена `volatile`:

- **Видимость:** каждое чтение volatile-переменной происходит из основной памяти (main memory), а не из кэша CPU; каждая запись сразу же сбрасывается в основную память.
- **Cascading-эффект:** при записи в volatile-переменную в main memory также записываются все переменные, видимые данному потоку до момента записи. При чтении volatile-переменной из main memory также обновляются все остальные переменные, видимые потоку.
- **Гарантия happens-before:** инструкции, выполненные до записи в volatile, не могут быть переупорядочены после неё (запрет instruction reordering).

Однако `volatile` не обеспечивает атомарность сложных операций: операция `counter++` (читай-изменяй-записывай) небезопасна даже с `volatile`, поскольку это три отдельных шага, не защищённых от параллельного вмешательства. Для таких случаев нужен `synchronized` или классы из `java.util.concurrent.atomic`.

---

#### Ответ от Claude

**`volatile`** — модификатор поля в Java, дающий две ключевые гарантии, закреплённые в Java Memory Model (JMM, JSR-133):

**1. Гарантия видимости (Visibility Guarantee)**

Без `volatile` JVM и процессор вправе кэшировать значение переменной в регистре CPU или L1/L2-кэше. Это означает, что изменение, сделанное одним потоком, может быть невидимо другому. `volatile` запрещает кэширование:
- запись в volatile-поле немедленно сбрасывается в main memory;
- чтение volatile-поля всегда происходит из main memory.

**2. Гарантия happens-before (Happens-Before Guarantee)**

JMM гарантирует: «запись в volatile-переменную happens-before любого последующего чтения этой переменной другим потоком». Это также означает:
- все операции записи, произошедшие до записи в volatile (в том числе в невялатильные переменные), тоже становятся видимы потоку, который прочитает volatile.
- запрещается переупорядочивание инструкций (instruction reordering) вокруг volatile-операций.

---

**Отличие `volatile` от `synchronized`:**

| Характеристика | `volatile` | `synchronized` |
|---|---|---|
| Видимость | Да | Да |
| Атомарность | Только для одиночных чтений/записей | Да (весь блок атомарен) |
| Взаимное исключение (mutex) | Нет | Да (только один поток в блоке) |
| Блокировка потоков | Нет | Да (поток может быть заблокирован) |
| Применимость | Поля | Методы и блоки кода |
| Производительность | Выше | Ниже (накладные расходы на монитор) |

**Когда `volatile` достаточно:**
- Один поток пишет, несколько читают (single-writer/multiple-readers).
- Новое значение не зависит от предыдущего (например, флаг `volatile boolean running = true`).

**Когда `volatile` недостаточно:**
- Операции типа «прочитать-изменить-записать»: `counter++`, `i = i + 1`.
- Составные условия: `if (a == 0) a = 1` (между проверкой и записью может вмешаться другой поток).

В таких случаях используют:
- `synchronized`-блоки/методы;
- классы `AtomicInteger`, `AtomicLong`, `AtomicReference` из `java.util.concurrent.atomic`;
- `Lock`-реализации из `java.util.concurrent.locks`.

**Пример правильного использования `volatile`:**
```java
public class Worker {
    private volatile boolean running = true;

    public void stop() {
        running = false; // запись видна всем потокам
    }

    public void run() {
        while (running) { // всегда читает актуальное значение
            doWork();
        }
    }
}
```

### Вопрос 4. Что такое deadlock? Приведите пример ситуации, в которой он возникает, и опишите способы его предотвращения.

**Категория:** Concurrency · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Deadlock (The Java™ Tutorials > Essential Java Classes > Concurrency)](https://docs.oracle.com/javase/tutorial/essential/concurrency/deadlock.html)

Документация Oracle описывает дедлок как ситуацию, в которой два или более потоков навсегда заблокированы в ожидании друг друга. Приводится пример с «поклоном»:

```java
public class Deadlock {
    static class Friend {
        private final String name;
        public Friend(String name) { this.name = name; }
        public String getName() { return this.name; }

        public synchronized void bow(Friend bower) {
            System.out.format("%s: %s has bowed to me!%n",
                this.name, bower.getName());
            bower.bowBack(this);
        }
        public synchronized void bowBack(Friend bower) {
            System.out.format("%s: %s has bowed back to me!%n",
                this.name, bower.getName());
        }
    }

    public static void main(String[] args) {
        final Friend alphonse = new Friend("Alphonse");
        final Friend gaston   = new Friend("Gaston");
        new Thread(() -> alphonse.bow(gaston)).start();
        new Thread(() -> gaston.bow(alphonse)).start();
    }
}
```

Что происходит:
1. Поток 1 вызывает `alphonse.bow(gaston)` — захватывает монитор объекта `alphonse`.
2. Поток 2 вызывает `gaston.bow(alphonse)` — захватывает монитор объекта `gaston`.
3. Поток 1 пытается вызвать `gaston.bowBack(...)` — ждёт освобождения монитора `gaston`.
4. Поток 2 пытается вызвать `alphonse.bowBack(...)` — ждёт освобождения монитора `alphonse`.

Оба потока ждут вечно — дедлок.

---

#### Ответ от Claude

**Deadlock (взаимная блокировка)** — ситуация в многопоточной программе, при которой два или более потока навсегда заблокированы: каждый держит ресурс (блокировку), нужный другому, и одновременно ждёт ресурс, удерживаемый другим.

**Четыре необходимых условия возникновения дедлока (условия Коффмана):**
1. **Взаимное исключение** — ресурс может быть захвачен только одним потоком.
2. **Удержание и ожидание** — поток, удерживая один ресурс, ожидает другой.
3. **Отсутствие вытеснения** — ресурс нельзя принудительно забрать у потока.
4. **Циклическое ожидание** — цепочка потоков, каждый из которых ждёт ресурс следующего.

**Классический пример на Java:**
```java
Object lockA = new Object();
Object lockB = new Object();

// Поток 1: захватывает A, затем пытается захватить B
Thread t1 = new Thread(() -> {
    synchronized (lockA) {
        System.out.println("T1: держит A, ждёт B");
        synchronized (lockB) {
            System.out.println("T1: держит A и B");
        }
    }
});

// Поток 2: захватывает B, затем пытается захватить A
Thread t2 = new Thread(() -> {
    synchronized (lockB) {
        System.out.println("T2: держит B, ждёт A");
        synchronized (lockA) {
            System.out.println("T2: держит B и A");
        }
    }
});

t1.start();
t2.start();
// При неудачном стечении обстоятельств оба потока зависнут навсегда
```

---

**Способы предотвращения дедлоков:**

**1. Упорядочивание захвата блокировок (Lock Ordering)**

Самый простой и надёжный способ. Если все потоки захватывают блокировки всегда в одном и том же порядке, циклическое ожидание невозможно.

```java
// Вместо произвольного порядка — всегда сначала lockA, потом lockB
Thread t1 = new Thread(() -> {
    synchronized (lockA) { synchronized (lockB) { /* работа */ } }
});
Thread t2 = new Thread(() -> {
    synchronized (lockA) { synchronized (lockB) { /* работа */ } }
});
```

**2. Захват с таймаутом (Lock Timeout)**

Использование `ReentrantLock.tryLock(timeout, unit)` вместо `synchronized`: если поток не может захватить блокировку за отведённое время, он освобождает уже захваченные блокировки и повторяет попытку позже.

```java
ReentrantLock lockA = new ReentrantLock();
ReentrantLock lockB = new ReentrantLock();

boolean acquiredA = lockA.tryLock(100, TimeUnit.MILLISECONDS);
boolean acquiredB = lockB.tryLock(100, TimeUnit.MILLISECONDS);
if (!acquiredA || !acquiredB) {
    if (acquiredA) lockA.unlock();
    if (acquiredB) lockB.unlock();
    // повторить или обработать ошибку
}
```

**3. Минимизация числа одновременно удерживаемых блокировок**

Чем меньше блокировок поток удерживает одновременно, тем ниже риск дедлока. Стоит пересмотреть дизайн, чтобы избежать вложенных `synchronized`-блоков.

**4. Использование высокоуровневых примитивов из `java.util.concurrent`**

`ConcurrentHashMap`, `BlockingQueue`, `Semaphore`, `ReadWriteLock` и др. разработаны с учётом безопасности конкурентного доступа и снижают вероятность дедлоков при правильном использовании.

**5. Обнаружение дедлоков (Deadlock Detection)**

Инструменты для обнаружения в runtime:
- **`jstack <pid>`** — дамп потоков JVM, явно показывает дедлоки с цепочкой ожидания.
- **`ThreadMXBean`** из `java.lang.management`:
```java
ThreadMXBean tmx = ManagementFactory.getThreadMXBean();
long[] deadlockedIds = tmx.findDeadlockedThreads();
if (deadlockedIds != null) {
    // обработка / логирование / алерт
}
```

**Итог:** лучшая стратегия — предотвращение на этапе проектирования: единый порядок захвата блокировок и минимизация их числа. `tryLock` с таймаутом — запасной вариант для сложных сценариев. `ThreadMXBean` полезен для мониторинга в продакшне.

### Вопрос 5. Для чего используется ключевое слово synchronized? Какова разница между синхронизацией метода и синхронизацией блока кода?

**Категория:** Concurrency · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Intrinsic Locks and Synchronization — Oracle Java Tutorials](https://docs.oracle.com/javase/tutorial/essential/concurrency/locksync.html)

Синхронизация в Java построена вокруг внутренних блокировок (intrinsic locks, или monitor locks). Каждый объект имеет связанную с ним внутреннюю блокировку. Поток обязан получить блокировку перед доступом к полям объекта и освободить её по окончании работы. Только один поток может владеть блокировкой в один момент времени; остальные потоки блокируются при попытке получить уже занятую блокировку.

**Синхронизированный метод** автоматически получает блокировку при вызове и освобождает её при выходе (в том числе при выбросе исключения). Для статического метода блокировкой является объект класса (`Class`), а не экземпляр.

**Синхронизированный блок** требует явного указания объекта-монитора. Это позволяет применять более тонкую синхронизацию — защищать только критическую секцию, а не весь метод, и использовать разные объекты-мониторы для независимых полей:

```java
public class MsLunch {
    private long c1 = 0;
    private long c2 = 0;
    private Object lock1 = new Object();
    private Object lock2 = new Object();

    public void inc1() {
        synchronized(lock1) { c1++; }
    }
    public void inc2() {
        synchronized(lock2) { c2++; }
    }
}
```

Благодаря отдельным блокировкам `inc1()` и `inc2()` могут выполняться параллельно, что снижает лишнее блокирование. Также поддерживается **повторный вход (reentrant synchronization)**: поток может повторно получить блокировку, которой уже владеет, что предотвращает взаимоблокировку самого потока.

#### Ответ от Claude

**Назначение `synchronized`** — обеспечить взаимное исключение и видимость памяти при совместном доступе нескольких потоков к общим данным, предотвращая состояния гонки и ошибки согласованности памяти.

**Механизм работы.** Каждый объект Java имеет встроенный монитор (intrinsic lock). `synchronized` заставляет поток захватить монитор перед входом в защищённый участок и освободить его при выходе — в том числе при выбросе исключения. Это гарантирует два свойства:
1. **Взаимное исключение** — в каждый момент времени защищённый участок выполняет не более одного потока.
2. **Видимость памяти** — при входе в блок переменные обновляются из главной памяти; при выходе — изменения сбрасываются в главную память. Тем самым устанавливается отношение happens-before.

**Синхронизированный метод:**
```java
public synchronized void increment() {
    count++;
}
```
- Блокировка — `this` (или `MyClass.class` для `static synchronized`).
- Защищает весь метод целиком.
- Конструкторы не могут быть `synchronized` (синтаксическая ошибка).

**Синхронизированный блок:**
```java
public void increment() {
    synchronized(this) {
        count++;
    }
    // остальной код выполняется без блокировки
}
```
- Явно задаёт объект-монитор — может быть любым объектом, не обязательно `this`.
- Защищает только указанный участок кода, а не весь метод.
- Позволяет использовать **несколько независимых блокировок** для разных полей, повышая параллелизм.

**Ключевые отличия:**

| Критерий | `synchronized`-метод | `synchronized`-блок |
|---|---|---|
| Монитор | `this` / `Class` | Любой объект |
| Охват | Весь метод | Только указанный участок |
| Гранулярность | Грубая | Тонкая |
| Параллелизм | Ниже | Выше (несколько блокировок) |

**Практическое правило:** если синхронизация нужна лишь для части метода или требуется разграничить блокировки для независимых полей — используйте блок. Если защищать нужно весь метод и единственный монитор — `synchronized` на методе читается лаконичнее.

**Дополнительно:** не рекомендуется синхронизироваться на строковых литералах и объектах-обёртках примитивов (Integer и т.п.) — из-за интернирования и кэширования несколько участков кода могут неожиданно делить один монитор. Синхронизация работает только в рамках одной JVM и не применима в распределённых системах.

### Вопрос 6. Что такое race condition (состояние гонки)? Как его можно обнаружить и устранить?

**Категория:** Concurrency · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Race Conditions and Critical Sections — jenkov.com](https://jenkov.com/tutorials/java-concurrency/race-conditions-and-critical-sections.html)

**Состояние гонки (race condition)** — ситуация, при которой результат работы программы зависит от относительного порядка или чередования выполнения нескольких потоков. **Критическая секция (critical section)** — участок кода, выполняемый несколькими потоками, где порядок их выполнения влияет на итоговый результат.

Состояния гонки проявляются в двух основных паттернах:

1. **Read-Modify-Write**: потоки читают переменную, изменяют её и записывают обратно. Если выполнение потоков чередуется, итоговое значение может быть некорректным. Например, если два потока одновременно инкрементируют счётчик, обновление одного потока может перезаписать обновление другого.
2. **Check-Then-Act**: поток проверяет условие (например, содержит ли Map определённый ключ), а затем действует на основании этого. Между проверкой и действием другой поток может изменить состояние, вызвав непредсказуемое поведение.

Для предотвращения состояний гонки необходимо сделать критическую секцию **атомарной**, чтобы только один поток мог выполнять её в каждый момент времени. Решения: синхронизированные блоки, Lock-объекты, атомарные переменные.

#### Ответ от Claude

**Состояние гонки (race condition)** — дефект многопоточной программы, при котором её корректность зависит от непредсказуемого порядка планирования потоков. Результат выполнения становится недетерминированным и может отличаться от запуска к запуску.

**Классический пример:** операция `count++` не является атомарной — она разбивается на три шага:
1. Чтение текущего значения `count` из памяти.
2. Вычисление `count + 1`.
3. Запись нового значения обратно.

Если два потока одновременно находятся между шагами 1 и 3, оба прочитают одно и то же значение и запишут одинаковый результат — одно из инкрементирований будет потеряно.

**Два основных паттерна:**

| Паттерн | Описание | Пример |
|---|---|---|
| Read-Modify-Write | Чтение → изменение → запись без атомарности | `count++`, накопление суммы |
| Check-Then-Act | Проверка условия → действие, между которыми состояние меняется | `if (!map.containsKey(k)) map.put(k, v)` |

**Как обнаружить:**
- **Статический анализ:** инструменты FindBugs/SpotBugs, Checkstyle, IntelliJ IDEA находят потенциально небезопасный доступ к общим полям без синхронизации.
- **Динамический анализ:** инструменты ThreadSanitizer (TSan), Java PathFinder, Helgrind отслеживают обращения к памяти в процессе выполнения и сигнализируют о гонках.
- **Стресс-тестирование:** многократный запуск тестов с высокой конкуренцией потоков (например, с помощью `@RepeatedTest` + `ExecutorService`) позволяет воспроизвести нестабильное поведение.
- **Код-ревью:** поиск участков, где несколько потоков обращаются к изменяемым общим данным без явной синхронизации.

**Как устранить:**

1. **`synchronized`-блоки / методы:** простейший способ; гарантируют взаимное исключение через монитор объекта.
   ```java
   public synchronized void increment() { count++; }
   ```
2. **`java.util.concurrent.locks.Lock`:** более гибкий контроль (попытка захвата с таймаутом, readWriteLock).
   ```java
   private final Lock lock = new ReentrantLock();
   public void increment() {
       lock.lock();
       try { count++; } finally { lock.unlock(); }
   }
   ```
3. **Атомарные переменные (`java.util.concurrent.atomic`):** lock-free реализации на основе CAS (compare-and-swap); предпочтительны для простых счётчиков и флагов.
   ```java
   private AtomicInteger count = new AtomicInteger(0);
   public void increment() { count.incrementAndGet(); }
   ```
4. **Потокобезопасные коллекции (`java.util.concurrent`):** `ConcurrentHashMap`, `CopyOnWriteArrayList` и др. вместо `HashMap`, `ArrayList`.
5. **Неизменяемые объекты (immutable):** если объект нельзя изменить, состояние гонки в принципе невозможно.

**Важно:** синхронизация не всегда означает `synchronized`. Выбор механизма зависит от сложности критической секции и требований к производительности. Атомарные переменные быстрее для простых операций; `Lock` удобнее, когда нужен `tryLock` или `lockInterruptibly`.

### Вопрос 7. Что такое модель памяти Java (Java Memory Model)? Что означает отношение happens-before?

**Категория:** Concurrency · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Memory Consistency Errors — Oracle Java Tutorials](https://docs.oracle.com/javase/tutorial/essential/concurrency/memconsist.html)

**Ошибки согласованности памяти (memory consistency errors)** возникают, когда разные потоки имеют несогласованные представления об одних и тех же данных.

Рассмотрим пример:
```java
int counter = 0;  // общая переменная

// Поток A
counter++;

// Поток B (вскоре после)
System.out.println(counter);
```

В однопоточном контексте ожидается вывод `1`. Однако при выполнении в разных потоках поток B может вывести `0`, поскольку нет никакой гарантии, что изменение `counter` потоком A будет видно потоку B.

**Отношение happens-before** — ключ к предотвращению ошибок согласованности памяти. Это гарантия того, что запись в память, выполненная одним конкретным оператором, видна другому конкретному оператору.

Действия, устанавливающие happens-before:
- **`Thread.start()`** — всё, что произошло до вызова `start()`, видно в новом потоке.
- **`Thread.join()`** — все изменения завершившегося потока видны после успешного `join()`.
- Синхронизация и средства `java.util.concurrent`.

#### Ответ от Claude

**Java Memory Model (JMM)** — спецификация, часть Java Language Specification (глава 17 JLS), которая определяет:
- как и когда потоки могут видеть значения, записанные другими потоками в общие переменные;
- какие оптимизации (переупорядочивание инструкций, кэширование в регистрах/CPU-кэше) JVM и процессор имеют право выполнять.

**Почему JMM необходима.** Современные CPU и JIT-компиляторы активно переупорядочивают инструкции и кэшируют данные в регистрах/L1-L3 кэшах ради производительности. Каждый поток исполняется на своём CPU с локальным кэшем. Без явных правил нет гарантии, что изменения, сделанные одним потоком, окажутся видимы другому и в правильном порядке.

**Структура памяти по JMM:**
- **Thread Stack (стек потока):** локальные переменные и кадры вызовов. Примитивные локальные переменные существуют только на стеке и не разделяются между потоками.
- **Heap (куча):** все объекты. Объекты на куче могут разделяться между потоками, если у них есть ссылки.

**Happens-before** — фундаментальное понятие JMM. Отношение happens-before (A *happens-before* B) означает: **все записи в память, сделанные до точки A, гарантированно видны при чтении в точке B**. Это не физический порядок по времени — это гарантия видимости.

**Правила, устанавливающие happens-before (ключевые):**

| Правило | Описание |
|---|---|
| **Program Order** | В одном потоке каждая операция happens-before следующей |
| **Monitor Lock** | Освобождение монитора (`unlock`) happens-before захвата того же монитора (`lock`) другим потоком |
| **Volatile Write** | Запись в `volatile`-переменную happens-before любого последующего чтения этой же переменной |
| **Thread Start** | `Thread.start()` happens-before любой операции в запущенном потоке |
| **Thread Join** | Все операции завершившегося потока happens-before возврата из `Thread.join()` |
| **Transitivity** | Если A happens-before B и B happens-before C, то A happens-before C |

**Практический смысл:** если между двумя потоками не установлено отношение happens-before, JVM **легально** может не доставить запись одного потока читателю другого (даже если физически запись произошла раньше по системным часам).

**Инструменты для установки happens-before:**
- `synchronized` (блок/метод) — захват монитора при входе, освобождение при выходе.
- `volatile` — запись happens-before чтения; также запрещает переупорядочивание смежных инструкций.
- `java.util.concurrent` — `Lock`, `CountDownLatch`, `Semaphore`, `AtomicXxx` и т.д. все гарантируют happens-before.

**Пример с `volatile`:**
```java
volatile boolean ready = false;
int data = 0;

// Поток A (писатель)
data = 42;        // (1)
ready = true;     // (2) — volatile запись

// Поток B (читатель)
if (ready) {      // (3) — volatile чтение; (2) happens-before (3)
    use(data);    // (4) — data гарантированно равна 42, т.к. (1) happens-before (2) happens-before (3) happens-before (4)
}
```

### Вопрос 8. Что такое Callable и Future? Чем Callable отличается от Runnable, и как получить результат асинхронного вычисления?

**Категория:** Concurrency · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Future (Java SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/Future.html)

Интерфейс `Future<V>` представляет результат асинхронного вычисления. Он предоставляет методы для проверки завершённости вычисления, ожидания результата и отмены задачи.

Основные методы:
- `boolean cancel(boolean mayInterruptIfRunning)` — попытка отменить выполнение задачи.
- `boolean isCancelled()` — возвращает `true`, если задача была отменена до завершения.
- `boolean isDone()` — возвращает `true`, если задача завершена (нормально, через исключение или отмену).
- `V get()` — ожидает завершения и возвращает результат; блокирует вызывающий поток.
- `V get(long timeout, TimeUnit unit)` — ожидает не дольше указанного времени.

Пример использования:
```java
Future<String> future = executor.submit(new Callable<String>() {
    public String call() {
        return searcher.search(target);
    }
});

displayOtherThings(); // выполняем другую работу

try {
    displayText(future.get()); // блокируется до готовности результата
} catch (ExecutionException ex) {
    cleanup();
}
```

Методы `get()` выбрасывают: `CancellationException` (задача отменена), `ExecutionException` (задача выбросила исключение), `InterruptedException` (поток прерван во время ожидания), `TimeoutException` (версия с таймаутом).

#### Ответ от Claude

**`Runnable`** — функциональный интерфейс из `java.lang`, существующий с Java 1.0:
```java
public interface Runnable {
    void run();
}
```
Не возвращает результат, не объявляет проверяемых исключений.

**`Callable<V>`** — функциональный интерфейс из `java.util.concurrent`, добавленный в Java 5:
```java
public interface Callable<V> {
    V call() throws Exception;
}
```
Возвращает результат обобщённого типа `V` и может выбрасывать **проверяемые исключения**.

**Сравнительная таблица:**

| Критерий | `Runnable` | `Callable<V>` |
|---|---|---|
| Пакет | `java.lang` | `java.util.concurrent` |
| Появился в | Java 1.0 | Java 5 |
| Метод | `void run()` | `V call() throws Exception` |
| Возвращает результат | Нет | Да |
| Проверяемые исключения | Нет | Да |
| Запуск через `Thread` | Да | Нет (только через `ExecutorService`) |
| Запуск через `ExecutorService` | Да | Да |

**`Future<V>`** — интерфейс из `java.util.concurrent`, представляющий дескриптор ещё не завершённого асинхронного вычисления. Он позволяет:
- узнать, завершилась ли задача (`isDone()`);
- получить результат, заблокировавшись при необходимости (`get()`);
- получить результат с таймаутом (`get(timeout, unit)`);
- отменить задачу (`cancel()`).

**Типичный сценарий получения результата:**
```java
ExecutorService executor = Executors.newFixedThreadPool(4);

// Создаём Callable и отправляем на выполнение
Callable<Integer> task = () -> {
    // долгое вычисление
    return 42;
};

Future<Integer> future = executor.submit(task);

// Продолжаем другую работу, пока задача выполняется
doSomethingElse();

// Получаем результат (блокируется, если задача ещё не завершена)
try {
    Integer result = future.get(5, TimeUnit.SECONDS); // с таймаутом
    System.out.println("Результат: " + result);
} catch (TimeoutException e) {
    future.cancel(true); // отменяем, если не уложились
} catch (ExecutionException e) {
    Throwable cause = e.getCause(); // исходное исключение из call()
}

executor.shutdown();
```

**Ключевые моменты:**
- `ExecutorService.submit(Callable)` возвращает `Future<V>`, несущий результат.
- `ExecutorService.submit(Runnable)` тоже возвращает `Future<?>`, но `get()` всегда вернёт `null` — Future используется только для отслеживания завершения.
- `future.get()` **блокирует** вызывающий поток до завершения задачи; чтобы не блокировать, предварительно проверяйте `isDone()`.
- Для цепочек асинхронных вычислений в Java 8+ рекомендуется `CompletableFuture`, который расширяет `Future` и добавляет неблокирующие коллбэки (`thenApply`, `thenCompose`, `exceptionally` и др.).
### Вопрос 9. Что такое ExecutorService? Чем он отличается от прямого создания потоков через new Thread()? Назовите основные фабричные методы Executors.

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ExecutorService (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/ExecutorService.html)

`ExecutorService` — это интерфейс, расширяющий `Executor` и предоставляющий методы для управления завершением работы, а также методы, которые могут возвращать объект `Future` для отслеживания прогресса одной или нескольких асинхронных задач. Документация описывает рекомендуемый паттерн двухфазного завершения работы через `shutdown()` + `awaitTermination()`. В качестве примера приведён класс `NetworkService`, использующий `Executors.newFixedThreadPool(poolSize)` — это типичный сценарий замены прямого создания потоков на пул потоков, который переиспользует рабочие потоки.

#### Ответ от Claude

**Что такое ExecutorService**

`ExecutorService` — интерфейс из пакета `java.util.concurrent`, предоставляющий каркас для асинхронного выполнения задач. Он расширяет интерфейс `Executor` и добавляет:
- методы управления жизненным циклом пула (`shutdown`, `shutdownNow`, `awaitTermination`);
- методы отправки задач с возвратом `Future` (`submit`, `invokeAll`, `invokeAny`).

**Отличия от `new Thread()`**

| Аспект | `new Thread()` | `ExecutorService` |
|---|---|---|
| Управление потоками | Создаётся новый поток на каждую задачу | Потоки переиспользуются из пула |
| Результат задачи | Сложно получить (нужен общий state) | Возвращает `Future<T>` |
| Отмена задачи | Нет удобного механизма | `Future.cancel()` |
| Масштабирование | Неограниченное число потоков — риск OOM | Контролируемый размер пула и очередь |
| Обработка ошибок | Исключения теряются, если не перехвачены | Видны через `Future.get()` |

**Основные фабричные методы класса `Executors`**

- `Executors.newFixedThreadPool(int n)` — пул с фиксированным числом потоков `n`; лишние задачи ставятся в неограниченную очередь `LinkedBlockingQueue`.
- `Executors.newCachedThreadPool()` — пул с динамическим числом потоков; неиспользуемые потоки завершаются через 60 секунд. Подходит для кратких асинхронных задач.
- `Executors.newSingleThreadExecutor()` — один поток, гарантирует последовательное выполнение задач в порядке FIFO.
- `Executors.newScheduledThreadPool(int n)` — пул для отложенного или периодического выполнения задач; возвращает `ScheduledExecutorService`.
- `Executors.newVirtualThreadPerTaskExecutor()` *(Java 21+)* — создаёт виртуальный поток на каждую задачу (Project Loom).

> Под капотом большинство методов создают `ThreadPoolExecutor` с нужными параметрами. Для продакшн-кода часто рекомендуют создавать `ThreadPoolExecutor` напрямую, чтобы явно управлять очередью и политикой отклонения задач.

### Вопрос 10. В чём разница между методами shutdown() и shutdownNow() у ExecutorService?

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ExecutorService (Java Platform SE 7) — docs.oracle.com](https://docs.oracle.com/javase/7/docs/api/java/util/concurrent/ExecutorService.html)

**`shutdown()`** — инициирует упорядоченное завершение работы, при котором ранее отправленные задачи выполняются, но новые задачи не принимаются. Повторный вызов не имеет дополнительного эффекта.

**`shutdownNow()`** — пытается остановить все активно выполняющиеся задачи, прекращает обработку ожидающих задач и возвращает список задач, ожидавших выполнения. Нет гарантий, что активно выполняющиеся задачи будут остановлены. Типичная реализация отменяет задачи через `Thread.interrupt()`, поэтому задачи, не реагирующие на прерывание, могут не завершиться.

В документации приведён рекомендуемый паттерн двухфазного завершения работы, сочетающий оба метода с `awaitTermination()`.

#### Ответ от Claude

| Характеристика | `shutdown()` | `shutdownNow()` |
|---|---|---|
| Новые задачи | Отклоняются | Отклоняются |
| Задачи в очереди | Будут выполнены | Отменяются, возвращаются в `List<Runnable>` |
| Работающие задачи | Продолжают выполняться | Получают сигнал прерывания `Thread.interrupt()` |
| Возвращаемое значение | `void` | `List<Runnable>` — задачи, не начавшие выполнение |
| Характер завершения | «Мягкое» (graceful) | «Жёсткое» (best-effort) |

**Важное замечание о `shutdownNow()`:** метод опирается на кооперативное прерывание. Если задача не проверяет флаг `Thread.interrupted()` и не перехватывает `InterruptedException`, она продолжит работу несмотря на вызов `shutdownNow()`. Гарантий немедленной остановки нет.

**Рекомендуемый паттерн** из официальной документации:

```java
void shutdownAndAwaitTermination(ExecutorService pool) {
    pool.shutdown(); // запрет новых задач, текущие продолжают работу
    try {
        if (!pool.awaitTermination(60, TimeUnit.SECONDS)) {
            pool.shutdownNow(); // прерываем текущие задачи
            if (!pool.awaitTermination(60, TimeUnit.SECONDS))
                System.err.println("Pool did not terminate");
        }
    } catch (InterruptedException ie) {
        pool.shutdownNow();
        Thread.currentThread().interrupt();
    }
}
```

Сначала вызывается `shutdown()` с ожиданием `awaitTermination()`, и только если пул не завершился за отведённое время — `shutdownNow()`.

### Вопрос 11. Что такое ReentrantLock? Какие преимущества он имеет перед synchronized (например, tryLock, lockInterruptibly, честность)?

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ReentrantLock (Java SE 17 & JDK 17) — docs.oracle.com](https://docs.oracle.com/en/java/javase/17/docs/api/java.base/java/util/concurrent/locks/ReentrantLock.html)

`ReentrantLock` — это реентерабельный взаимоисключающий замок с той же базовой семантикой, что и неявный монитор-замок, получаемый методами `synchronized`, но с расширенными возможностями. Класс поддерживает как честную (fair), так и нечестную (unfair, по умолчанию) политику захвата. Если замок сконфигурирован как честный, то при конкуренции он отдаёт предпочтение потоку, ожидающему дольше всего. Нечестный режим не гарантирует порядок, но обеспечивает более высокую пропускную способность. Метод `tryLock()` не соблюдает настройки честности — он всегда «вклинивается» в очередь. Документация рекомендует освобождать замок в блоке `finally`.

#### Ответ от Claude

**Что такое ReentrantLock**

`ReentrantLock` — конкретная реализация интерфейса `Lock` из `java.util.concurrent.locks`. «Reentrant» означает, что один поток может захватить замок несколько раз подряд (счётчик захватов увеличивается), и должен столько же раз вызвать `unlock()`, чтобы полностью освободить замок.

**Преимущества перед `synchronized`**

| Возможность | `synchronized` | `ReentrantLock` |
|---|---|---|
| Реентерабельность | Да | Да |
| Честность (fairness) | Нет | Да (`new ReentrantLock(true)`) |
| Прерываемое ожидание | Нет | `lockInterruptibly()` |
| Попытка без блокировки | Нет | `tryLock()` |
| Попытка с тайм-аутом | Нет | `tryLock(long, TimeUnit)` |
| Несколько условий | Нет (один `wait/notify`) | `newCondition()` — любое число |
| Мониторинг | Нет | `getHoldCount()`, `getQueueLength()` и др. |

**Ключевые методы**

- **`tryLock()`** — немедленно возвращает `true`, если замок свободен, иначе `false`. Не блокирует поток. Важно: не соблюдает настройки честности даже на fair-замке.
- **`tryLock(long timeout, TimeUnit unit)`** — ожидает захвата не дольше указанного времени; возвращает `false` по истечению таймаута. Реагирует на прерывание.
- **`lockInterruptibly()`** — ожидает захвата, но если поток прерван, бросает `InterruptedException`. В `synchronized`-блоке поток не может выйти из ожидания через прерывание.
- **Честность (fair lock):** `new ReentrantLock(true)` гарантирует, что потоки получают замок в порядке обращения (FIFO). Предотвращает «голодание» (starvation), но снижает пропускную способность из-за дополнительного планирования.

**Обязательный паттерн использования**

```java
private final ReentrantLock lock = new ReentrantLock();

public void doWork() {
    lock.lock();
    try {
        // критическая секция
    } finally {
        lock.unlock(); // ВСЕГДА в finally, иначе deadlock при исключении
    }
}
```

В отличие от `synchronized`, `ReentrantLock` не освобождает замок автоматически — забытый `unlock()` приводит к дедлоку.

**Когда использовать `synchronized`, а когда `ReentrantLock`**

- `synchronized` — простые случаи взаимного исключения без специальных требований; код проще и не требует блока `finally`.
- `ReentrantLock` — нужны `tryLock`, тайм-аут, прерываемое ожидание, честность или несколько `Condition`-объектов.

### Вопрос 12. Что такое AtomicInteger и операция CAS (Compare-And-Swap)? Как AtomicInteger обеспечивает атомарность без synchronized?

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [AtomicInteger (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/atomic/AtomicInteger.html)

`AtomicInteger` — это значение типа `int`, которое может обновляться атомарно. Применяется в задачах вроде атомарных инкрементируемых счётчиков и не является заменой `Integer`. Ключевой метод `compareAndSet(int expect, int update)` атомарно устанавливает значение равным `update`, если текущее значение равно `expect`; возвращает `true` при успехе и `false`, если фактическое значение не совпало с ожидаемым. Это и есть примитивная операция CAS.

#### Ответ от Claude

**Что такое AtomicInteger**

`AtomicInteger` — класс из пакета `java.util.concurrent.atomic` (Java 5+), предоставляющий значение типа `int` с поддержкой атомарных операций без использования блокировок (`lock-free`). Доступны такие операции, как атомарный инкремент, декремент, сложение и условное обновление.

**Что такое CAS (Compare-And-Swap)**

CAS — атомарная операция с тремя аргументами:
1. **V** — адрес ячейки памяти (переменная).
2. **A** — ожидаемое (старое) значение.
3. **B** — новое значение.

Процессор атомарно выполняет: «если значение по адресу V равно A — записать B и вернуть true; иначе ничего не делать и вернуть false». Вся операция — одна неделимая инструкция на уровне процессора (например, `LOCK CMPXCHG` на x86). Никакой другой поток не может вмешаться между проверкой и записью.

**Как AtomicInteger использует CAS**

Вместо `synchronized`-блока `AtomicInteger` применяет оптимистичный цикл повторных попыток (spin loop):

```java
// Упрощённая реализация incrementAndGet() до Java 9
public final int incrementAndGet() {
    for (;;) {
        int current = get();          // читаем текущее значение
        int next = current + 1;       // вычисляем новое
        if (compareAndSet(current, next)) // пробуем атомарно обновить
            return next;              // успех — возвращаем результат
        // неудача — другой поток успел изменить значение, повторяем
    }
}
```

В Java 9+ операция упрощена до одного JVM-intrinsic `VarHandle.getAndAdd()`, который транслируется в `LOCK XADD` на x86, что ещё быстрее.

**Почему это быстрее `synchronized`**

| Аспект | `synchronized` | `AtomicInteger` (CAS) |
|---|---|---|
| Механизм | Блокировка (mutex в ОС) | Аппаратная инструкция ЦП |
| Блокировка потока | Да — поток засыпает в очереди | Нет — поток крутится в цикле |
| Переключение контекста | Возможно (дорого) | Отсутствует |
| Производительность при низкой конкуренции | Хорошая | Отличная |
| Производительность при высокой конкуренции | Деградирует из-за блокировок | Spin loop увеличивает нагрузку на CPU |

**ABA-проблема**

У CAS есть один известный изъян: если значение изменилось с A → B → A, CAS не обнаружит изменения и посчитает, что оно не менялось. Для ссылочных типов это критично. Решение — `AtomicStampedReference` или `AtomicMarkableReference`, хранящие вместе со значением версионный штамп.

**Основные методы AtomicInteger**

- `get()` / `set(int)` — чтение/запись с семантикой volatile.
- `getAndIncrement()` / `incrementAndGet()` — атомарный постфиксный/префиксный инкремент.
- `compareAndSet(int expect, int update)` — базовая операция CAS.
- `addAndGet(int delta)` — атомарное сложение.
### Вопрос 13. В чём разница между ConcurrentHashMap и Collections.synchronizedMap()? Как ConcurrentHashMap достигает масштабируемости?

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ConcurrentHashMap (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/ConcurrentHashMap.html)

`ConcurrentHashMap` — реализация хэш-таблицы с поддержкой **полного параллелизма операций чтения** и **высокой ожидаемой параллельности операций записи**. Она обеспечивает потокобезопасность без блокировки всей таблицы: операции чтения (в том числе `get`) в общем случае не блокируются и могут перекрываться с операциями обновления (`put`, `remove`). В отличие от `Hashtable`, заблокировать таблицу целиком для предотвращения любого доступа невозможно. Итераторы являются weakly consistent: отражают состояние таблицы на момент создания или позже, никогда не бросают `ConcurrentModificationException`. `null`-ключи и `null`-значения **не допускаются**. Агрегатные методы `size()`, `isEmpty()`, `containsValue()` возвращают приблизительные значения при параллельных обновлениях.

#### Ответ от Claude

**Механизм синхронизации**

`Collections.synchronizedMap(map)` оборачивает произвольную `Map` в декоратор, который для каждого метода захватывает единственный мьютекс (intrinsic lock на объекте-обёртке). Это означает: в любой момент только один поток может выполнять любую операцию — чтение или запись.

`ConcurrentHashMap` использует принципиально другой подход:
- **Чтение (get)** — полностью неблокирующее, выполняется без захвата каких-либо блокировок.
- **Запись (put, remove)** — в **Java 7** использовала сегментную блокировку (16 сегментов по умолчанию, каждый — отдельный `ReentrantLock`). В **Java 8** сегменты убраны: блокировка берётся на уровне отдельного бакета (первого узла бина) через `synchronized`, а для вставки в пустой бин используется атомарная операция **CAS** (Compare-And-Swap) без блокировки вообще.

**Сравнительная таблица**

| Характеристика | `synchronizedMap` | `ConcurrentHashMap` |
|---|---|---|
| Стратегия блокировки | Один глобальный мьютекс | Бакет-уровень (CAS + synchronized на узел) |
| Параллельные чтения | Блокируют друг друга | Полностью параллельны |
| Параллельные записи | Нет (глобальная блокировка) | До N бакетов одновременно |
| `null`-ключи/значения | Разрешены (если базовая Map поддерживает) | Запрещены, бросает `NullPointerException` |
| Итераторы | Fail-fast (`ConcurrentModificationException`) | Weakly consistent, не бросает исключений |
| `size()`, `isEmpty()` | Точно (под блокировкой) | Приблизительно при конкурентных обновлениях |
| Составные операции | Не атомарны без внешней синхронизации | Атомарные `putIfAbsent`, `compute`, `merge` |

**Как ConcurrentHashMap достигает масштабируемости**

1. **Неблокирующее чтение.** `get` использует `volatile`-семантику для чтения узлов — видит последние записанные значения без блокировки.
2. **Мелкозернистая блокировка при записи.** Блокируется только нужный бин, а не вся таблица, что позволяет N потокам одновременно писать в разные бины.
3. **CAS-вставка в пустой бин.** Если бин пуст, новый узел вставляется атомарно через `Unsafe.compareAndSwapObject` — без `synchronized` вообще.
4. **Параллельный resize.** При расширении таблицы несколько потоков могут одновременно переносить бины в новый массив.
5. **`LongAdder`-подобный счётчик.** Метод `size()` использует распределённый счётчик (аналог `LongAdder`) для избежания contention при подсчёте.

**Когда использовать `synchronizedMap`**

- Нужна полная линеаризуемость составных операций без ручной синхронизации.
- Оборачивается нестандартная `Map`-реализация.
- Нагрузка низкая и простота важнее производительности.

### Вопрос 14. Что такое ThreadLocal? Для каких задач его применяют и какие риски утечки памяти с ним связаны?

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Thread-Local Variables — docs.oracle.com (Java 25 Core Libraries)](https://docs.oracle.com/en/java/javase/25/core/thread-local-variables.html)

`ThreadLocal` — класс, предоставляющий переменные, изолированные по потокам: каждый поток, обращающийся к такой переменной, получает собственную независимо инициализированную копию. Переменная обычно объявляется как `static final` поле, что позволяет широко использовать её в коде без передачи через параметры методов. Значения сохраняются на протяжении всей жизни потока или до явного вызова `remove()`. В пулах потоков это создаёт риск утечки: значение, установленное одной задачей, может «просочиться» в следующую задачу того же потока. Документация отмечает, что **Scoped Values** (Project Loom) могут устранить эти недостатки, предоставляя более безопасный механизм передачи контекста.

#### Ответ от Claude

**Что такое ThreadLocal**

`ThreadLocal<T>` (пакет `java.lang`) — контейнер, у которого каждый поток имеет собственную изолированную копию значения. Метод `get()` возвращает копию текущего потока; `set(T)` — устанавливает её; `remove()` — удаляет. Реализован через внутренний `Thread.threadLocals` — карту `ThreadLocalMap`, которая хранится непосредственно в объекте `Thread`.

```java
private static final ThreadLocal<SimpleDateFormat> DATE_FORMAT =
    ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));
```

**Типичные области применения**

| Задача | Пример |
|---|---|
| Потокобезопасность не-thread-safe объектов | `SimpleDateFormat`, `Random` (до `ThreadLocalRandom`) |
| Передача контекста по стеку вызовов | User ID, tenant ID в веб-запросе без передачи через параметры |
| Управление транзакциями/соединениями | Spring-овый `TransactionSynchronizationManager` хранит `Connection` в `ThreadLocal` |
| Трассировка и логирование | MDC в SLF4J/Logback использует `ThreadLocal` для correlation ID |

**Риски утечки памяти**

1. **Пул потоков + отсутствие `remove()`.** В серверном приложении потоки из пула живут долго и никогда не завершаются. Если задача установила значение в `ThreadLocal`, но не вызвала `remove()`, значение остаётся в `Thread.threadLocals` навсегда (пока поток жив). При большом объекте-значении это прямая утечка памяти.

2. **Слабые ссылки на ключи — мнимая защита.** Ключи в `ThreadLocalMap` хранятся как `WeakReference<ThreadLocal>`. Когда сам объект `ThreadLocal` теряет сильную ссылку (экземпляр `ThreadLocal` собирается GC), запись в карте становится «осиротевшей» — ключ `null`, значение живёт. Такие «stale entries» не освобождаются сразу: они удаляются только при следующих вызовах `set`/`get`/`remove` на других ключах, но этот момент может никогда не наступить.

3. **Утечка ClassLoader в Application Server.** Если значение `ThreadLocal` содержит объект, загруженный `ClassLoader`'ом веб-приложения, а поток принадлежит серверному пулу, то при перезагрузке приложения `ClassLoader` не может быть выгружен GC — остаётся сильная цепочка: `Thread → ThreadLocalMap → значение → ClassLoader`. Это классическая утечка в Tomcat/JBoss.

**Рекомендации по безопасному использованию**

```java
// Всегда оборачивать в try/finally при использовании в пулах потоков
ThreadLocal<Context> ctx = new ThreadLocal<>();
try {
    ctx.set(new Context(...));
    doWork();
} finally {
    ctx.remove(); // критически важно!
}
```

- Использовать `remove()` в блоке `finally` или через `Filter`/`Interceptor`.
- В Java 20+ рассмотреть **ScopedValue** (Preview) — неизменяемый и автоматически ограниченный по области видимости аналог `ThreadLocal`.

### Вопрос 15. Что такое CompletableFuture? Как с его помощью выстроить цепочку асинхронных операций (thenApply, thenCompose, thenCombine)? Как в нём обрабатываются исключения?

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [CompletableFuture (Java SE 21 & JDK 21) — docs.oracle.com](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/util/concurrent/CompletableFuture.html)

`CompletableFuture<T>` реализует интерфейс `Future<T>` и `CompletionStage<T>`. Метод `thenApply` возвращает новый `CompletionStage`, который выполняется с результатом текущего стейджа как аргументом переданной функции (аналог `Stream.map`). Метод `thenCompose` возвращает новый `CompletionStage`, завершённый с тем же значением, что и `CompletionStage`, возвращаемый переданной функцией (аналог `Stream.flatMap`). Метод `thenCombine` выполняется, когда **оба** стейджа — текущий и переданный в аргументе — завершаются нормально, применяя `BiFunction` к обоим результатам. Исключения оборачиваются в `CompletionException`; `get()` перевыбрасывает их как `ExecutionException`, `join()` — как непроверяемый `CompletionException` напрямую.

#### Ответ от Claude

**Что такое CompletableFuture**

`CompletableFuture<T>` (Java 8+, пакет `java.util.concurrent`) — реализация `Future` с поддержкой функционального построения цепочек асинхронных вычислений. В отличие от обычного `Future`, результат можно установить вручную (`complete(T)`, `completeExceptionally(Throwable)`), а цепочки колбэков регистрируются декларативно.

По умолчанию асинхронные стейджи выполняются в `ForkJoinPool.commonPool()`. Можно передать кастомный `Executor` в методы с суффиксом `Async`.

**Методы построения цепочки**

`thenApply(Function<T, U>)` — трансформирует результат текущего стейджа. Функция получает значение типа `T` и возвращает `U`. Цепочка продолжается со значением `U`. Аналог `map` в Stream API.

```java
CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> 42)
    .thenApply(n -> "Result: " + n); // "Result: 42"
```

`thenCompose(Function<T, CompletionStage<U>>)` — плоская цепочка: функция возвращает новый `CompletableFuture<U>`, и итоговый стейдж завершается значением этого вложенного `Future`. Используется, когда следующий шаг сам асинхронный. Аналог `flatMap`.

```java
CompletableFuture<User> future = CompletableFuture.supplyAsync(() -> userId)
    .thenCompose(id -> fetchUserAsync(id)); // fetchUserAsync возвращает CF<User>
```

`thenCombine(CompletionStage<U>, BiFunction<T, U, V>)` — объединяет результаты **двух независимых** `CompletableFuture`, когда **оба** завершатся. Подходит для параллельного запуска двух независимых вычислений с последующим объединением результатов.

```java
CompletableFuture<String> priceF = CompletableFuture.supplyAsync(() -> getPrice());
CompletableFuture<String> nameF  = CompletableFuture.supplyAsync(() -> getName());
CompletableFuture<String> result = priceF.thenCombine(nameF, (p, n) -> n + ": " + p);
```

**Разница между `thenApply` и `thenCompose`**

| Метод | Функция возвращает | Результирующий тип |
|---|---|---|
| `thenApply(f)` | `U` (обычное значение) | `CompletableFuture<U>` |
| `thenCompose(f)` | `CompletableFuture<U>` | `CompletableFuture<U>` (не `CompletableFuture<CompletableFuture<U>>`) |

**Обработка исключений**

Если любой стейдж цепочки завершился с исключением, оно оборачивается в `CompletionException` и **пробрасывается по всей цепочке** — последующие `thenApply`/`thenCompose` пропускаются.

Способы перехватить исключение:

`exceptionally(Function<Throwable, T>)` — срабатывает только при исключении, возвращает значение-заглушку для продолжения цепочки.

```java
future.exceptionally(ex -> "default value");
```

`handle(BiFunction<T, Throwable, U>)` — срабатывает **всегда** (и при успехе, и при ошибке); один из параметров будет `null`. Позволяет и трансформировать результат, и обработать ошибку в одном месте.

```java
future.handle((result, ex) -> ex != null ? "error: " + ex.getMessage() : result);
```

`whenComplete(BiConsumer<T, Throwable>)` — аналог `handle`, но не меняет тип результата; удобен для побочных эффектов (логирование).

**Получение результата: `get()` vs `join()`**

- `get()` — checked exception: бросает `ExecutionException` (проверяемое).
- `join()` — бросает `CompletionException` (непроверяемое), удобнее в лямбдах.

### Вопрос 16. Что такое ForkJoinPool и алгоритм work-stealing? В каких сценариях ForkJoinPool предпочтительнее обычного ThreadPoolExecutor?

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Fork/Join — The Java™ Tutorials (Essential Java Classes > Concurrency), docs.oracle.com](https://docs.oracle.com/javase/tutorial/essential/concurrency/forkjoin.html)

Fork/Join Framework проектировался для задач, которые естественно разбиваются на рекурсивные подзадачи. В центре фреймворка — `ForkJoinPool`, расширение `AbstractExecutorService`. Ключевое отличие от обычных пулов — **алгоритм work-stealing**: рабочие потоки, исчерпавшие свою очередь задач, могут **красть задачи** из очередей других занятых потоков. Это обеспечивает эффективную загрузку всех процессоров. Паттерн использования: если порция работы достаточно мала — выполнить напрямую; иначе — разбить на две части, вызвать их рекурсивно через `fork()`/`join()` и дождаться результатов. Точка входа — `ForkJoinTask` (подклассы `RecursiveAction` без результата и `RecursiveTask<V>` с результатом).

#### Ответ от Claude

**Что такое ForkJoinPool**

`ForkJoinPool` (Java 7+, пакет `java.util.concurrent`) — специализированный `ExecutorService`, оптимизированный для рекурсивного разделения задач (divide-and-conquer). Может выполнять `ForkJoinTask`, а также обычные `Runnable`/`Callable`. Статический `ForkJoinPool.commonPool()` используется по умолчанию во многих местах JDK (включая `CompletableFuture`, параллельные стримы).

По умолчанию параллелизм равен `Runtime.getRuntime().availableProcessors()`. Все рабочие потоки — **daemon-потоки**.

**Алгоритм work-stealing**

Каждый рабочий поток имеет собственную **двустороннюю очередь (deque)** задач. Поток берёт задачи **с головы** своей очереди (LIFO — горячие задачи в кэше). Когда очередь пуста, поток **крадёт** задачу **с хвоста** очереди другого потока (FIFO — самые старые, наиболее крупные задачи). Это минимизирует синхронизацию (конкуренция за хвост очереди, а не за голову) и обеспечивает локальность кэша для потока-владельца.

```
Поток A (своя очередь): [task1, task2, task3] <- берёт task1 с головы
Поток B (пустая очередь): крадёт task3 с хвоста очереди Поток A
```

Преимущество: задачи-потомки (fork) выполняются в том же потоке (кэш тёплый), а крупные задачи у других потоков становятся кандидатами для кражи.

**RecursiveTask / RecursiveAction**

```java
class SumTask extends RecursiveTask<Long> {
    private static final int THRESHOLD = 1000;
    private final long[] array;
    private final int lo, hi;

    @Override
    protected Long compute() {
        if (hi - lo <= THRESHOLD) {
            // Базовый случай: считаем напрямую
            long sum = 0;
            for (int i = lo; i < hi; i++) sum += array[i];
            return sum;
        }
        int mid = (lo + hi) / 2;
        SumTask left  = new SumTask(array, lo, mid);
        SumTask right = new SumTask(array, mid, hi);
        left.fork();                     // запускаем в пуле
        return right.compute() + left.join(); // join ждёт результата
    }
}
```

**ForkJoinPool vs ThreadPoolExecutor: когда что выбирать**

| Критерий | `ForkJoinPool` | `ThreadPoolExecutor` |
|---|---|---|
| Структура задач | Рекурсивные, порождающие подзадачи | Независимые, не порождающие подзадачи |
| Work-stealing | Да — эффективно при разном объёме задач | Нет — единая очередь, возможно contention |
| Блокирующие задачи | Плохо: занимают worker-потоки, снижают параллелизм | Лучше: можно настроить размер пула |
| Настройка очереди | Не настраивается | `LinkedBlockingQueue`, `SynchronousQueue`, приоритетные очереди |
| Периодические/scheduled задачи | Нет | Через `ScheduledThreadPoolExecutor` |
| Использование по умолчанию в JDK | `CompletableFuture`, параллельные стримы | `Executors.newFixedThreadPool`, сервлет-контейнеры |

**Когда ForkJoinPool предпочтительнее**

1. **Рекурсивные алгоритмы** — сортировка, обход деревьев, рекурсивный поиск.
2. **Параллельные вычисления над массивами/коллекциями** — обработка больших массивов с разбиением на части.
3. **Задачи с непредсказуемой трудоёмкостью** — work-stealing балансирует нагрузку автоматически.
4. **Параллельные стримы** — `stream().parallel()` под капотом использует `commonPool()`.

**Когда НЕ использовать ForkJoinPool**

- Задачи содержат блокирующие I/O-операции (sleep, JDBC, HTTP) — потоки блокируются и параллелизм деградирует. В таких случаях лучше `ThreadPoolExecutor` с достаточным числом потоков или виртуальные потоки (Java 21+).
- Нужна точная настройка очереди задач и политики отклонения.

### Вопрос 17. Что такое виртуальные потоки (Virtual Threads, Project Loom, Java 21)? Чем они отличаются от платформенных потоков и в каких случаях их использование целесообразно?

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Java Project Loom: Unlocking the Power of Java Virtual Threads — blog.ycrash.io](https://blog.ycrash.io/java-project-loom-virtual-threads/)

**Архитектура и планировщик.** Платформенные потоки (platform threads) работают по схеме «один Java-поток — один OS-поток». Их количество ограничено ресурсами ОС, а стек каждого потока фиксирован и задаётся при старте. Виртуальные потоки управляются исключительно JVM-планировщиком и не связаны напрямую с OS-потоками. JVM монтирует виртуальный поток на «поток-носитель» (carrier thread). Когда виртуальный поток входит в состояние ожидания (например, ждёт I/O), JVM размонтирует его, сохраняет стек на куче и освобождает carrier thread для другой работы.

**Память.** Виртуальный стек хранится в heap и может динамически расти, тогда как стек платформенного потока занимает предвыделённую память (~1 МБ по умолчанию). Это позволяет запускать миллионы виртуальных потоков одновременно.

**Когда использовать виртуальные потоки:** I/O-зависимые нагрузки (REST-вызовы, запросы к БД, работа с брокерами сообщений), микросервисная архитектура с высокой конкурентностью.

**Когда не использовать:** CPU-intensive задачи (тяжёлые вычисления, сериализация) — здесь виртуальные потоки не дают преимущества.

**Проблема «приколачивания» (pinning).** Если виртуальный поток входит в блок `synchronized`, он не может быть размонтирован с carrier thread, что нивелирует его преимущества. В Java 21 это могло вызывать деградацию производительности; проблема была исправлена в Java 24. Рекомендуется использовать `ReentrantLock` вместо `synchronized` в коде, работающем с виртуальными потоками.

#### Ответ от Claude

**Что такое виртуальные потоки.** Virtual Threads — легковесные потоки, введённые в предварительном режиме в Java 19 и стабилизированные в Java 21 в рамках Project Loom. Ключевое отличие от платформенных потоков в том, что они управляются JVM, а не операционной системой.

**Сравнение с платформенными потоками:**

| Характеристика | Платформенный поток | Виртуальный поток |
|---|---|---|
| Соответствие OS-потоку | 1:1 | M:N (многие на немногих) |
| Размер стека | ~1 МБ (фиксирован) | Несколько КБ, динамический (в heap) |
| Создание | Дорогостоящее | Дешёвое (можно создавать миллионы) |
| Планировщик | ОС | JVM (ForkJoinPool) |
| Блокировка I/O | Блокирует OS-поток | JVM отсоединяет от carrier thread |

**Модель работы.** JVM поддерживает небольшой пул «потоков-носителей» (carrier threads). Когда виртуальный поток начинает блокирующую операцию (I/O, `sleep`, `lock.await()`), JVM размонтирует его стек в heap и позволяет carrier thread взять другой виртуальный поток. По завершении ожидания виртуальный поток снова монтируется на свободный carrier thread.

**Создание:**
```java
// Через фабричный метод
Thread vt = Thread.ofVirtual().start(() -> System.out.println("virtual!"));

// Через ExecutorService
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    executor.submit(() -> doWork());
}
```

**Когда целесообразно применять:**
- I/O-bound задачи: HTTP-запросы, JDBC, чтение файлов, очереди сообщений.
- Высококонкурентные сервисы, где тысячи соединений держатся открытыми в ожидании ответа.
- Замена пула потоков с ограниченным размером в сценариях «thread-per-request».

**Когда виртуальные потоки не помогают:**
- CPU-bound вычисления — нет блокировок, нет выгрузки, нет выигрыша.
- Код с тяжёлым использованием `synchronized` (до Java 24) — проблема pinning: виртуальный поток «прикалывается» к carrier thread и не может быть выгружен. Решение — заменить `synchronized` на `ReentrantLock`.
- `ThreadLocal`-злоупотребления: виртуальных потоков может быть миллион, а ThreadLocal создаётся для каждого — возможна утечка памяти.

**Итог.** Виртуальные потоки не ускоряют отдельный поток. Они радикально увеличивают пропускную способность при I/O-нагрузке, позволяя писать простой блокирующий код с эффективностью реактивного подхода, не усложняя архитектуру.

### Вопрос 18. Что такое CountDownLatch и CyclicBarrier? В чём принципиальное отличие между ними? Приведите пример использования каждого.

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [CountDownLatch and CyclicBarrier in Java — medium.com/hprog99](https://medium.com/hprog99/countdownlatch-and-cyclicbarrier-in-java-9fec58575f51)

**CountDownLatch** — синхронизационный механизм, позволяющий одному или нескольким потокам ожидать завершения набора операций. Инициализируется счётчиком; каждый рабочий поток вызывает `countDown()`, уменьшая счётчик. Когда счётчик достигает нуля, ожидающий поток продолжает выполнение.

```java
CountDownLatch latch = new CountDownLatch(3);
Thread t1 = new Thread(() -> { /* работа */ latch.countDown(); });
// ... аналогично t2, t3
latch.await(); // главный поток ждёт
```

**CyclicBarrier** — позволяет нескольким потокам ждать друг друга в общей точке перед продолжением. После срабатывания барьера может использоваться повторно.

```java
CyclicBarrier barrier = new CyclicBarrier(3);
Thread t1 = new Thread(() -> { /* работа */ barrier.await(); /* продолжение */ });
```

**Ключевые отличия:**

| Аспект | CountDownLatch | CyclicBarrier |
|---|---|---|
| Переиспользование | Одноразовый | Переиспользуемый |
| Обработка исключений | InterruptedException только у текущего потока | BrokenBarrierException у всех ожидающих |
| Применение | Ждать завершения операций | Координировать параллельное выполнение |

#### Ответ от Claude

**CountDownLatch** — барьер с убывающим счётчиком. Один или несколько потоков вызывают `await()` и блокируются, пока другие потоки не уменьшат счётчик до нуля через `countDown()`. После обнуления счётчика **нельзя сбросить** — лач одноразовый.

```java
// Сценарий: запустить сервис только после инициализации 3 компонентов
CountDownLatch readyLatch = new CountDownLatch(3);

for (int i = 0; i < 3; i++) {
    final int id = i;
    new Thread(() -> {
        initComponent(id);       // инициализация компонента
        readyLatch.countDown();  // сигнал готовности
    }).start();
}

readyLatch.await(); // главный поток ждёт все компоненты
startService();
```

**CyclicBarrier** — барьер, в котором все участвующие потоки ждут друг друга. Когда все N потоков вызвали `await()`, барьер «срабатывает», и все продолжают выполнение. После этого **счётчик автоматически сбрасывается**, что позволяет использовать барьер повторно (отсюда «cyclic»). Опционально можно передать `Runnable`-действие, которое выполнится ровно один раз при каждом срабатывании барьера.

```java
// Сценарий: параллельная обработка данных по фазам
int parties = 3;
CyclicBarrier barrier = new CyclicBarrier(parties, () -> {
    System.out.println("Фаза завершена, объединяем результаты");
});

for (int i = 0; i < parties; i++) {
    final int id = i;
    new Thread(() -> {
        while (hasMorePhases()) {
            processPhase(id);   // работа в текущей фазе
            barrier.await();    // ждём всех участников
            // все продолжают следующую фазу
        }
    }).start();
}
```

**Принципиальные отличия:**

| Критерий | CountDownLatch | CyclicBarrier |
|---|---|---|
| Направление ожидания | Один (или несколько) ждут остальных | Все ждут всех |
| Переиспользование | Нет (одноразовый) | Да (автосброс) |
| Что считается | Задачи/события (можно уменьшать несколько раз из одного потока) | Потоки (каждый поток уменьшает счётчик один раз) |
| Действие при срабатывании | Нет | Опциональный `Runnable` |
| Поломка при исключении | Не влияет на другие потоки | `BrokenBarrierException` у всех ожидающих |
| Типичное применение | Инициализация перед стартом, ожидание N завершённых задач | Параллельные алгоритмы с фазами, игры с ожиданием всех игроков |

**Правило выбора:** если вам нужно «дождаться, пока всё будет готово, и продолжить один раз» — CountDownLatch. Если нужно «все вместе дошли до точки и вместе идут дальше, и так повторяется» — CyclicBarrier.

### Вопрос 19. Чем отличается volatile от synchronized? В каких случаях volatile недостаточно и почему?

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Difference Between Volatile and Synchronized Keywords in Java — dzone.com](https://dzone.com/articles/difference-between-volatile-and-synchronized-keywo)

**volatile** гарантирует, что значение переменной всегда синхронизировано между локальной памятью потока и основной памятью. Любой поток, обращающийся к такой переменной, видит актуальное значение. Однако volatile не обеспечивает взаимного исключения: несколько потоков могут выполнять код одновременно.

**synchronized** обеспечивает и синхронизацию памяти, и взаимное исключение: только один поток единовременно может выполнять защищённый блок. При входе в блок поток обновляет все переменные из главной памяти, при выходе — сбрасывает изменения обратно.

Таким образом: volatile — только видимость, synchronized — видимость + атомарность + взаимное исключение.

Производительность: volatile дешевле synchronized, так как не требует захвата/освобождения монитора. Но synchronized всё равно необходим там, где нужна атомарность составных операций.

#### Ответ от Claude

**volatile — гарантия видимости.** Запись в `volatile`-переменную немедленно становится видна всем потокам. JVM запрещает кешировать такую переменную в регистре CPU или в локальном кеше потока. Помимо этого, запись в `volatile` создаёт happens-before барьер: все предыдущие записи данного потока также становятся видны тому потоку, который читает это `volatile`-значение.

**synchronized — гарантия видимости + атомарности + взаимного исключения.** При входе в блок поток захватывает монитор объекта (lock), при выходе — освобождает. В один момент времени внутри блока может находиться только один поток.

**Таблица сравнения:**

| Свойство | volatile | synchronized |
|---|---|---|
| Видимость изменений | Да | Да |
| Атомарность составных операций | Нет | Да |
| Взаимное исключение (mutex) | Нет | Да |
| Применимость | Только к полям | К методам и блокам |
| Производительность | Выше | Ниже (overhead блокировки) |
| Блокировка потока | Нет | Да (при конкуренции) |

**Когда volatile недостаточно — конкретные случаи:**

1. **Составные операции (read-modify-write).** `count++` — не атомарная операция, это три шага: прочитать, увеличить, записать. Между чтением и записью другой поток может сделать то же самое:
   ```java
   private volatile int count = 0;
   // Небезопасно! При конкурентном вызове теряются инкременты:
   public void increment() { count++; }
   ```
   Решение: `synchronized`, `AtomicInteger` или `LongAdder`.

2. **Условные обновления (check-then-act).** Проверка и изменение — две раздельные операции, которые могут быть разделены вмешательством другого потока:
   ```java
   if (instance == null) {        // поток A проверил: null
       instance = new Singleton(); // поток B тоже проверил: null, создал объект
   }                               // поток A создаёт второй объект
   ```

3. **Несколько связанных переменных.** Если инвариант программы зависит от согласованности нескольких полей, volatile не поможет — между обновлением первого и второго поля другой поток может увидеть промежуточное состояние:
   ```java
   volatile int x, y; // инвариант: x == y
   // поток A: x = 1; y = 1; — между записями поток B может увидеть x=1, y=0
   ```

**Когда volatile достаточно:**
- Простой флаг (boolean), который один поток пишет, другие только читают: `volatile boolean running = true;`
- Публикация ссылки на неизменяемый объект (immutable object).
- Двойная проверка блокировки (double-checked locking) в Java 5+ — поле `instance` должно быть `volatile`:
  ```java
  private volatile Singleton instance;
  ```

### Вопрос 20. Что такое happens-before? Перечислите основные правила happens-before в Java Memory Model и приведите пример нарушения видимости без него.

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [JSR-133 Java Memory Model FAQ — cs.umd.edu](https://www.cs.umd.edu/~pugh/java/memoryModel/jsr-133-faq.html)

Java Memory Model создаёт частичное упорядочивание операций с памятью (чтение/запись поля, lock, unlock) и других операций потока (start, join). Когда одно действие happens-before другого, первое гарантированно упорядочено перед вторым и видимо ему.

Основные правила:

1. **Program Order** — каждое действие в потоке happens-before каждого последующего действия в том же потоке согласно порядку программы.
2. **Monitor Lock** — разблокировка монитора happens-before каждой последующей блокировки того же монитора.
3. **Volatile** — запись в volatile-поле happens-before каждого последующего чтения того же volatile-поля.
4. **Thread Start** — вызов `start()` на потоке happens-before любого действия в запущенном потоке.
5. **Thread Join** — все действия в потоке happens-before успешного возврата из `join()` на этом потоке.

Нарушение видимости: если два потока синхронизируются на разных объектах-мониторах, happens-before между ними не устанавливается. Точно так же чтение одного volatile-поля не гарантирует видимость записей другого volatile-поля.

#### Ответ от Claude

**Что такое happens-before.** Happens-before — это отношение частичного порядка между операциями в Java Memory Model (JMM), введённое в JSR-133. Если операция A happens-before операции B, то:
1. Все изменения, сделанные операцией A, **гарантированно видны** операции B.
2. Операция A **упорядочена перед** B с точки зрения семантики программы (хотя реальное железо может переупорядочивать инструкции — JMM лишь требует соблюдения наблюдаемого эффекта).

Отношение транзитивно: если A hb B и B hb C, то A hb C.

**Основные правила happens-before в JMM:**

1. **Program Order Rule (порядок программы).** В пределах одного потока каждая операция happens-before любой последующей операции. Это не означает отсутствие переупорядочивания на уровне CPU — это означает, что внутри потока результат всегда наблюдается как последовательный.

2. **Monitor Lock Rule (правило монитора).** Разблокировка (`unlock`) монитора happens-before любой последующей блокировки (`lock`) того же монитора. Это основа корректности `synchronized`.

3. **Volatile Variable Rule (правило volatile).** Запись в `volatile`-переменную happens-before любого последующего чтения той же переменной. «Последующее» означает в реальном времени — после того, как запись была видна читающему потоку.

4. **Thread Start Rule (правило старта потока).** Всё, что делает поток до вызова `thread.start()`, happens-before любого действия в запущенном потоке.

5. **Thread Join Rule (правило join).** Все действия потока T happens-before успешного возврата из `T.join()` в другом потоке.

6. **Transitivity (транзитивность).** Если A hb B и B hb C, то A hb C.

7. **Default Initialization Rule.** Запись значений по умолчанию (0, false, null) в поля объекта happens-before любого действия потока с этим объектом.

**Пример нарушения видимости без happens-before:**

```java
// Поля без volatile и без синхронизации
boolean ready = false;
int value = 0;

// Поток-писатель
new Thread(() -> {
    value = 42;     // (1)
    ready = true;   // (2)
}).start();

// Поток-читатель
new Thread(() -> {
    while (!ready) { /* spin */ }  // (3)
    System.out.println(value);     // (4) — может напечатать 0!
}).start();
```

Проблем здесь две:
- **Видимость:** поток-читатель может никогда не увидеть `ready = true`, потому что JVM может кешировать значение в регистре CPU. Бесконечный цикл — вполне реальный исход.
- **Переупорядочивание:** даже если `ready = true` стало видимым, CPU или компилятор могут переставить операции (1) и (2) местами. Тогда читатель увидит `ready = true`, но `value` ещё не записан — напечатается 0.

Happens-before здесь не установлен: нет `volatile`, нет `synchronized` на одном мониторе, нет `start()`/`join()` между этими операциями.

**Исправление:**

```java
volatile boolean ready = false; // volatile устанавливает hb между (2) и (3)
int value = 0;                  // видимо благодаря транзитивности: (1) hb (2) hb (3) hb (4)
```

Запись в `volatile`-поле `ready` создаёт happens-before для чтения `ready` в другом потоке. По правилу транзитивности: запись `value = 42` (1) happens-before записи `ready = true` (2) по program order, а (2) happens-before чтения `ready` (3) по volatile rule — значит, (1) happens-before (4), и `value == 42` гарантированно.
