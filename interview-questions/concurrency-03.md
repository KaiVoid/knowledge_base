### Вопрос 41. Что такое CompletableFuture.allOf() и anyOf()? Чем они отличаются по семантике? Какие подводные камни возникают при использовании ForkJoinPool.commonPool() по умолчанию в CompletableFuture?

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [CompletableFuture (Java SE 21 & JDK 21) — Oracle Docs](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/util/concurrent/CompletableFuture.html)

**`allOf(CompletableFuture<?>... cfs)`**

```java
static CompletableFuture<Void> allOf(CompletableFuture<?>... cfs)
```

Возвращает новый `CompletableFuture<Void>`, который завершается, когда **все** переданные CompletableFuture завершены. Если какой-либо из них завершается исключительно, возвращаемый future тоже завершается исключением. Возвращаемый тип — `Void`, так как метод не агрегирует результаты — он лишь сигнализирует о завершении всех задач.

**`anyOf(CompletableFuture<?>... cfs)`**

```java
static CompletableFuture<Object> anyOf(CompletableFuture<?>... cfs)
```

Возвращает новый `CompletableFuture<Object>`, который завершается, когда завершился **любой один** из переданных CompletableFuture, с тем же результатом. Если первый завершившийся future завершился исключением, результирующий future тоже завершается этим исключением.

**Использование commonPool по умолчанию:**

Все async-методы без явно переданного `Executor` выполняются в `ForkJoinPool.commonPool()` (если он поддерживает уровень параллелизма не менее 2; иначе для каждой задачи создаётся отдельный поток).

#### Ответ от Claude

**Семантическое сравнение allOf() и anyOf()**

| Аспект | `allOf()` | `anyOf()` |
|---|---|---|
| Условие завершения | **Все** futures завершены | **Любой один** future завершён |
| Тип результата | `CompletableFuture<Void>` | `CompletableFuture<Object>` |
| Результат | нет (Void) | результат первого завершившегося |
| Исключение | если хотя бы один завершился с ошибкой | если первый завершился с ошибкой |
| Остальные futures | ждём все | продолжают выполняться, но результат игнорируется |

**Подводный камень allOf(): нет агрегации результатов**

`allOf()` возвращает `CompletableFuture<Void>`, из которого нельзя получить результаты дочерних futures. Их нужно извлекать отдельно после завершения `allOf`:

```java
CompletableFuture<String> f1 = CompletableFuture.supplyAsync(() -> "a");
CompletableFuture<String> f2 = CompletableFuture.supplyAsync(() -> "b");

CompletableFuture.allOf(f1, f2).join();

String r1 = f1.join(); // теперь безопасно
String r2 = f2.join();
```

**Подводный камень anyOf(): тип Object и незавершённые futures**

`anyOf()` возвращает `CompletableFuture<Object>`, что требует небезопасного приведения типов. Оставшиеся futures продолжают работать в фоне, потребляя ресурсы, но их результаты никто не ждёт.

**Подводные камни ForkJoinPool.commonPool()**

**1. Ограниченный размер пула**

Размер commonPool по умолчанию = `Runtime.getRuntime().availableProcessors() - 1`. На машине с 4 ядрами это всего 3 потока. Пул проектировался для CPU-интенсивных задач (fork/join-рекурсия), а не для I/O-операций.

**2. Блокирующие задачи вызывают голодание потоков (thread starvation)**

CompletableFuture и `Stream.parallelStream()` по умолчанию используют один и тот же commonPool. Если одна часть приложения отправляет в commonPool долгоживущие блокирующие задачи (например, HTTP-запросы, запросы к БД), поток занят и не может обработать другие задачи. Это приводит к каскадной деградации: несвязанные async-операции начинают ждать десятки секунд.

**3. Непредсказуемость в контейнерной среде**

JVM до Java 10 не умела корректно читать CPU-лимиты Docker/cgroups. С Java 10+ ситуация улучшена флагом `XX:+UseContainerSupport` (включён по умолчанию), но конфигурации через `--cpu-shares` могут по-прежнему давать неверный результат.

**Как избегать проблем**

Передавать явный `Executor` для всех async-методов CompletableFuture:

```java
ExecutorService ioPool = Executors.newFixedThreadPool(50);

CompletableFuture.supplyAsync(() -> callRemoteService(), ioPool)
    .thenApplyAsync(result -> transform(result), ioPool);
```

Либо переопределить пул по умолчанию, расширив `CompletableFuture` и переопределив метод `defaultExecutor()` (доступен с Java 9).

**Правило:** commonPool подходит только для коротких CPU-bound задач. Для I/O, блокирующих вызовов и длительных операций всегда используйте выделенный пул с размером, соответствующим уровню параллелизма ваших задач.
### Вопрос 42. Что такое «безопасная публикация» объекта (safe publication) в Java Memory Model? Перечислите корректные идиомы публикации и объясните, почему финальные поля дают особые гарантии видимости.

**Категория:** Concurrency · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Chapter 17. Threads and Locks — Java Language Specification SE 8](https://docs.oracle.com/javase/specs/jls/se8/html/jls-17.html)

Объект считается **полностью инициализированным**, когда завершается его конструктор. Поток, который может увидеть ссылку на объект лишь после того, как объект полностью инициализирован, гарантированно увидит корректно инициализированные значения всех `final`-полей этого объекта (раздел 17.5 JLS).

Модель использования для финальных полей:
- установите все `final`-поля объекта внутри его конструктора;
- не записывайте ссылку на строящийся объект туда, где другой поток может её увидеть до завершения конструктора.

Если это правило соблюдено, любой поток, получивший ссылку на объект, всегда увидит корректно сконструированную версию его `final`-полей.

Пример из JLS 17.5:

```java
class FinalFieldExample {
    final int x;
    int y;
    static FinalFieldExample f;

    public FinalFieldExample() {
        x = 3;  // финальное поле
        y = 4;  // обычное поле
    }

    static void writer() {
        f = new FinalFieldExample();
    }

    static void reader() {
        if (f != null) {
            int i = f.x;  // гарантированно 3
            int j = f.y;  // может быть 0 — нет гарантии
        }
    }
}
```

JMM определяет следующие happens-before гарантии (раздел 17.4.5):
- разблокировка монитора happens-before последующей блокировки того же монитора;
- запись в `volatile`-переменную happens-before последующего чтения той же переменной;
- `Thread.start()` happens-before любого действия в запущенном потоке;
- все действия в потоке happens-before `Thread.join()` возвращает результат.

Для `final`-полей JMM вводит особое действие — *freeze* (заморозка): при выходе из конструктора каждое `final`-поле «замораживается», и это freeze happens-before любого чтения данного поля через любую ссылку на объект. Это делает финальные поля видимыми без дополнительной синхронизации.

#### Ответ от Claude

**Безопасная публикация (safe publication)** — это гарантия того, что объект, полностью сконструированный в одном потоке, виден другим потокам именно в полностью инициализированном состоянии. Без специальных механизмов JMM позволяет компилятору и процессору переупорядочивать операции, из-за чего другой поток может увидеть ссылку на объект раньше, чем его конструктор завершится (проблема «частично сконструированного объекта»).

**Корректные идиомы публикации:**

1. **`volatile`-поле.** Запись `volatile`-ссылки happens-before её чтения любым другим потоком. При записи в `volatile`-переменную все переменные, видимые пишущему потоку, также сбрасываются в основную память (расширенная гарантия видимости, описанная Jenkov).

```java
private volatile MyObject ref;

// поток-писатель
ref = new MyObject();

// поток-читатель
MyObject obj = ref; // видит полностью инициализированный объект
```

2. **Синхронизированный блок.** При входе в `synchronized`-блок поток обновляет значения всех видимых переменных из основной памяти; при выходе — сбрасывает их обратно. Разблокировка happens-before последующей блокировки того же монитора.

3. **Статический инициализатор.** Объект, инициализированный в статическом блоке или статическом поле, публикуется безопасно благодаря гарантиям загрузки класса JVM (аналог синхронизации). Это основа идиомы *initialization-on-demand holder*.

```java
class Singleton {
    private static class Holder {
        static final Singleton INSTANCE = new Singleton();
    }
    public static Singleton getInstance() { return Holder.INSTANCE; }
}
```

4. **`Thread.start()`.** Все действия потока до вызова `start()` happens-before любого действия в новом потоке.

5. **`Thread.join()`.** Все действия завершившегося потока happens-before возврата из `join()` в ожидающем потоке.

**Почему `final`-поля дают особые гарантии:**

JMM вводит специальное действие *freeze* для финальных полей. При завершении конструктора каждое `final`-поле «замораживается». Правила JMM (раздел 17.5) гарантируют, что:
- freeze happens-before любого чтения этого `final`-поля;
- гарантия распространяется через цепочку ссылок: если поток прочитал ссылку на объект, он увидит корректные значения всех `final`-полей этого объекта, **даже без какой-либо дополнительной синхронизации**.

Обычные (не `final`) поля такой гарантии не имеют: без happens-before другой поток может увидеть их значение по умолчанию (0, null, false), даже если конструктор уже записал туда другое значение.

Практическое следствие: неизменяемые объекты (все поля `final`) можно безопасно публиковать через любой механизм, включая незащищённую запись в обычное поле — при условии, что запись происходит однократно и до того, как другие потоки получат ссылку. Однако на практике рекомендуется использовать `volatile` или `synchronized` для самой ссылки, чтобы исключить гонку при её публикации.

**Расхождение между источниками:** JLS чётко формулирует гарантию freeze для `final`-полей; Jenkov делает акцент на расширенной гарантии видимости `volatile` (пигги-бэкинг). Оба источника согласны в том, что неизменяемые объекты безопаснее изменяемых, а `volatile`-ссылка достаточна для безопасной публикации полностью инициализированного объекта.

### Вопрос 43. Что такое ложное пробуждение (spurious wakeup) при использовании Object.wait() и Condition.await()? Почему ожидание на условии всегда нужно оборачивать в цикл while, а не в if?

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Condition (Java SE 24 & JDK 24) — Oracle Docs](https://docs.oracle.com/en/java/javase/24/docs/api/java.base/java/util/concurrent/locks/Condition.html)

При ожидании на `Condition` допускается «ложное пробуждение» (spurious wakeup) — пробуждение потока без вызова `signal()`, `signalAll()`, прерывания или истечения таймаута. Это является уступкой семантике нижележащей платформы.

Документация явно предписывает:

> «Condition должен всегда ожидаться в цикле, проверяющем предикат состояния, на котором производится ожидание.»

Реализация вправе исключить возможность spurious wakeup, однако рекомендуется всегда считать, что они могут произойти, и всегда ждать в цикле.

Пример из документации `awaitNanos()`:

```java
boolean aMethod(long timeout, TimeUnit unit) throws InterruptedException {
    long nanosRemaining = unit.toNanos(timeout);
    lock.lock();
    try {
        while (!conditionBeingWaitedFor()) {
            if (nanosRemaining <= 0L)
                return false;
            nanosRemaining = theCondition.awaitNanos(nanosRemaining);
        }
        // ...
        return true;
    } finally {
        lock.unlock();
    }
}
```

Поток может быть пробуждён по одной из пяти причин: явный `signal()`, `signalAll()`, прерывание потока, истечение таймаута или spurious wakeup.

#### Ответ от Claude

**Ложное пробуждение (spurious wakeup)** — это ситуация, когда поток, вызвавший `Object.wait()` или `Condition.await()`, выходит из ожидания без того, чтобы другой поток явно вызвал `notify()`/`notifyAll()` или `signal()`/`signalAll()`, без прерывания и без истечения таймаута.

**Причины существования spurious wakeup:**

Спецификация POSIX для `pthread_cond_wait` (на которой основаны реализации JVM на Linux/Unix) явно допускает spurious wakeup как уступку аппаратным и операционным системам. Некоторые архитектуры процессоров и планировщики ОС не гарантируют, что пробуждение потока всегда связано с явным сигналом. Следовательно, JMM переносит это требование наверх: реализация JVM может пробудить поток в любой момент.

**Почему `while`, а не `if`:**

```java
// НЕПРАВИЛЬНО — уязвимо к spurious wakeup
synchronized (lock) {
    if (!conditionMet) {
        lock.wait();
    }
    // поток продолжает работу, но условие может быть ложным
    doWork();
}

// ПРАВИЛЬНО — защита от spurious wakeup
synchronized (lock) {
    while (!conditionMet) {
        lock.wait();
    }
    // условие гарантированно выполнено
    doWork();
}
```

При использовании `if` поток после пробуждения (в том числе ложного) немедленно переходит к `doWork()`, не проверяя, выполнено ли условие на самом деле. Цикл `while` заставляет поток повторно проверить предикат и вернуться в ожидание, если условие не выполнено.

**Дополнительные причины использовать `while` (помимо spurious wakeup):**

1. **Гонка после пробуждения.** Между пробуждением потока и повторным захватом монитора другой поток может изменить состояние, сделав условие снова ложным. `while` повторно проверяет предикат уже после захвата монитора.

2. **`notifyAll()` пробуждает всех.** Если используется `notifyAll()`, просыпаются все ожидающие потоки, но условие может быть выполнено только для одного из них. Остальные должны вернуться в ожидание.

**Аналогия для `Condition.await()`:**

```java
lock.lock();
try {
    while (!conditionMet) {      // всегда while, не if
        condition.await();
    }
    doWork();
} finally {
    lock.unlock();
}
```

Правило одно для обоих API: **предикат всегда проверяется в цикле `while`**. Это консенсусная рекомендация как в официальной документации Oracle (Condition Javadoc), так и в обучающих материалах Jenkov. Пренебрежение этим правилом является классической ошибкой параллельного программирования.

### Вопрос 44. Как ReentrantReadWriteLock решает проблему «читатели и писатели»? Когда возникает голодание писателей (writer starvation) и как его предотвратить? Чем отличается честный и нечестный режим замка?

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ReentrantReadWriteLock (Java SE 21 & JDK 21) — Oracle Docs](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/util/concurrent/locks/ReentrantReadWriteLock.html)

`ReentrantReadWriteLock` реализует интерфейс `ReadWriteLock` и поддерживает следующую семантику:

- Несколько потоков могут одновременно удерживать **блокировку чтения**, если ни один поток не держит блокировку записи.
- Только один поток может удерживать **блокировку записи**, и при этом ни один поток не должен держать блокировку чтения.

**Нечестный режим (по умолчанию):**
> «Порядок входа в блокировки чтения и записи не определён, с учётом ограничений повторного входа. Нечестная блокировка при постоянном конкурировании может бесконечно откладывать один или несколько потоков читателей или писателей, но обычно имеет более высокую пропускную способность, чем честная блокировка.»

**Честный режим:**
> «Потоки конкурируют за вход, используя политику приблизительного порядка поступления. При освобождении удерживаемой блокировки либо одному самому долго ожидающему потоку-писателю будет выделена блокировка записи, либо группе потоков-читателей, ожидающих дольше всех ожидающих писателей, будет выделена блокировка чтения.»

Поток, пытающийся получить честную блокировку чтения (не повторно), заблокируется, если блокировка записи удерживается **или** существует ожидающий поток-писатель.

Поддерживается **понижение блокировки** (write → read): поток может захватить блокировку записи, затем блокировку чтения, и затем освободить блокировку записи. Повышение (read → write) не поддерживается.

Пример из документации (кешированные данные):

```java
class CachedData {
    Object data;
    boolean cacheValid;
    final ReentrantReadWriteLock rwl = new ReentrantReadWriteLock();

    void processCachedData() {
        rwl.readLock().lock();
        if (!cacheValid) {
            rwl.readLock().unlock();
            rwl.writeLock().lock();
            try {
                if (!cacheValid) {
                    data = retrieveData();
                    cacheValid = true;
                }
                // понижение: захватываем чтение до освобождения записи
                rwl.readLock().lock();
            } finally {
                rwl.writeLock().unlock();
            }
        }
        try {
            use(data);
        } finally {
            rwl.readLock().unlock();
        }
    }
}
```

#### Ответ от Claude

**Проблема «читатели и писатели»** — классическая задача параллельного программирования: необходимо обеспечить параллельное чтение данных несколькими потоками при исключительном доступе на запись.

**Как ReentrantReadWriteLock решает проблему:**

`ReentrantReadWriteLock` реализует `ReadWriteLock`, предоставляя две взаимосвязанных блокировки:

- **Блокировка чтения (`readLock()`):** может удерживаться несколькими потоками одновременно, если ни один не держит блокировку записи. Подходит для операций, не изменяющих состояние.
- **Блокировка записи (`writeLock()`):** эксклюзивная — требует отсутствия любых других читателей и писателей. Подходит для мутирующих операций.

Это даёт выигрыш в производительности по сравнению с `synchronized` или `ReentrantLock` в сценариях с преобладанием чтения: читатели не блокируют друг друга.

```java
ReentrantReadWriteLock rwl = new ReentrantReadWriteLock();
Lock r = rwl.readLock();
Lock w = rwl.writeLock();

// Чтение — параллельно для нескольких потоков
r.lock();
try { return data; } finally { r.unlock(); }

// Запись — эксклюзивно
w.lock();
try { data = newValue; } finally { w.unlock(); }
```

**Голодание писателей (writer starvation):**

В нечестном режиме (по умолчанию) писатель может бесконечно ждать, если поток читателей постоянно пополняется: новые читатели захватывают блокировку чтения сразу, пока хоть один читатель её удерживает, и писатель никогда не дожидается момента, когда блокировка свободна. Это называется **writer starvation**.

**Способы предотвращения:**

1. **Честный режим (`new ReentrantReadWriteLock(true)`):** реализует политику приблизительного порядка поступления. В честном режиме новый читатель блокируется, если есть хотя бы один ожидающий писатель — это предотвращает бесконечное вытеснение писателей. Недостаток: заметно меньшая пропускная способность из-за накладных расходов на упорядочивание.

2. **`StampedLock` (Java 8+):** предлагает оптимистичное чтение (`tryOptimisticRead()`), при котором не захватывается никакая блокировка. Если за время чтения не произошло записи (проверяется через `validate(stamp)`), результат действителен. Это снижает конкуренцию и уменьшает риск starvation за счёт того, что читатели реже блокируют писателей. `StampedLock` не является реентерабельным.

**Честный vs нечестный режим — подробное сравнение:**

| Аспект | Нечестный (по умолчанию) | Честный (`true`) |
|---|---|---|
| Порядок предоставления | Не определён | Приближённый FIFO |
| Пропускная способность | Выше | Ниже |
| Риск starvation | Есть (для писателей и читателей) | Значительно снижен |
| Новый читатель при ожидающем писателе | Может пройти сразу | Блокируется |
| `tryLock()` | Игнорирует очередь | Тоже игнорирует (bypasses fairness) |

**Понижение блокировки (lock downgrading):**

`ReentrantReadWriteLock` поддерживает понижение write → read: поток, удерживающий блокировку записи, может захватить блокировку чтения, а затем освободить блокировку записи. Обратное (повышение read → write) не поддерживается и приведёт к взаимной блокировке.

**Ограничения:** максимум 65 535 рекурсивных блокировок записи и 65 535 блокировок чтения; блокировка чтения не поддерживает `Condition`.
### Вопрос 45. Что такое ScopedValue (Project Loom, Java 21+) и чем он принципиально отличается от ThreadLocal? В каких сценариях ScopedValue предпочтительнее и почему он лучше подходит для виртуальных потоков?

**Категория:** Concurrency · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [JEP 446: Scoped Values (Preview)](https://openjdk.org/jeps/446)

`ScopedValue` — механизм для безопасного и эффективного разделения **неизменяемых** данных между методами внутри потока и с дочерними потоками, не прибегая к явной передаче аргументов. Задаётся как `final static`-поле типа `ScopedValue<T>`.

Проблемы `ThreadLocal`, которые решает `ScopedValue`:

1. **Неограниченная мутируемость** — любой код, имеющий доступ к `ThreadLocal`, может вызвать `set()` в любой момент; `ScopedValue` делает значение доступным только на чтение.
2. **Дорогостоящее наследование** — при создании дочернего потока через `InheritableThreadLocal` дочерний поток вынужден скопировать **все** переменные родителя, что создаёт значительный расход памяти. У `ScopedValue` дочерние потоки в `StructuredTaskScope` разделяют одну и ту же привязку без копирования.
3. **Неограниченное время жизни** — значение в `ThreadLocal` живёт до тех пор, пока не вызван `remove()`, что приводит к утечкам памяти, особенно в пулах потоков. `ScopedValue` автоматически освобождает привязку при выходе из скопа (нормальном или по исключению).

API:

```java
private static final ScopedValue<String> USER_ROLE = ScopedValue.newInstance();

// Привязка значения и выполнение кода в скопе
ScopedValue.runWhere(USER_ROLE, "admin", () -> handleRequest());

// С возвратом результата
String result = ScopedValue.callWhere(USER_ROLE, "guest", this::buildResponse);

// Несколько привязок сразу
ScopedValue.where(KEY1, val1).where(KEY2, val2).run(() -> doWork());

// Чтение значения
String role = USER_ROLE.get();              // NoSuchElementException если не привязан
String roleOrDefault = USER_ROLE.orElse("anonymous");
boolean bound = USER_ROLE.isBound();
```

Вложенные скопы поддерживают **перепривязку** (rebinding): внутренний скоп может переопределить значение, а при его завершении автоматически восстанавливается значение внешнего скопа.

#### Ответ от Claude

**Что такое ScopedValue**

`ScopedValue<T>` — финализированный в JDK 25 (JEP 506) механизм передачи неизменяемых данных через стек вызовов без явной передачи параметров. В JDK 21–24 — preview API (флаг `--enable-preview`). Концептуально это «неявный параметр метода»: данные доступны всем вызываемым методам в рамках скопа, но не за его пределами.

**Принципиальные отличия от ThreadLocal**

| Аспект | ThreadLocal | ScopedValue |
|---|---|---|
| Мутируемость | Любой код может вызвать `set()` в любой момент | Значение неизменяемо в рамках скопа; перепривязка создаёт новый вложенный скоп |
| Время жизни | До явного `remove()` или смерти потока | Автоматически ограничено блоком `runWhere`/`callWhere` |
| Наследование в дочерних потоках | `InheritableThreadLocal` копирует весь `ThreadLocalMap` — O(n) по памяти | Дочерние потоки в `StructuredTaskScope` разделяют привязку родителя без копирования — O(1) |
| Хранение | Отдельный `ThreadLocalMap` в каждом объекте `Thread` | Контекст скопа — отдельная структура, не привязанная к объекту потока |
| Утечки памяти | Высокий риск при использовании пулов потоков | Исключены: привязка уничтожается при выходе из скопа |
| Работа с виртуальными потоками | При миллионах виртуальных потоков и `InheritableThreadLocal` — экспоненциальный рост копий в куче | Все виртуальные потоки разделяют одну копию данных без дублирования |

**Почему ScopedValue лучше для виртуальных потоков**

Виртуальные потоки дёшевы и создаются в огромных количествах. С `InheritableThreadLocal` каждый дочерний виртуальный поток копирует весь `ThreadLocalMap` родителя — при тысячах и миллионах потоков это исчерпывает кучу. `ScopedValue` устраняет эту проблему: неизменяемое значение разделяется по ссылке, копирования нет. Кроме того, `StructuredTaskScope.fork()` автоматически передаёт привязки `ScopedValue` в дочерние потоки, тогда как с `ThreadLocal` этого не происходит — дочерние задачи теряют контекст.

**Когда предпочитать ScopedValue**

- Передача контекста запроса (principal, locale, trace-id) через глубокий стек вызовов без изменения сигнатур.
- Любой код, использующий `StructuredTaskScope` и виртуальные потоки.
- Когда важна явная ограниченность жизни данных и отсутствие утечек.

**Когда ThreadLocal по-прежнему уместен**

- Мутируемое состояние, специфичное для потока (например, кэш в рамках запроса, требующий обновления).
- Легаси-код, который нет смысла переписывать.

**Примечание по версиям:** JEP 446 (Java 21, preview) → JEP 464 (Java 22) → JEP 481 (Java 23) → JEP 487 (Java 24) → JEP 506 (Java 25, finalized). В Java 21–24 требуется `--enable-preview`.

### Вопрос 46. Что такое структурированный параллелизм (Structured Concurrency) и StructuredTaskScope в Java? Какую проблему он решает по сравнению с произвольным порождением задач через ExecutorService?

**Категория:** Concurrency · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [JEP 505: Structured Concurrency (Fifth Preview)](https://openjdk.org/jeps/505)

Структурированный параллелизм — подход, при котором если задача должна решаться конкурентно, то **все порождённые потоки стартуют и завершаются в одном и том же блоке кода**. Жизненный цикл дочерних потоков ограничен скопом родительской задачи — так же, как вызов метода не может продолжаться дольше вызывающего метода.

`StructuredTaskScope` — основной строительный блок. Используется через try-with-resources:

```java
try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
    Supplier<Invoice>        invoice = scope.fork(() -> fetchInvoice(id));
    Supplier<PaymentStatus>  payment = scope.fork(() -> fetchPayment(id));

    scope.join();               // ожидает завершения обеих задач
    scope.throwIfFailed();      // пробрасывает исключение, если одна из задач упала

    return new Result(invoice.get(), payment.get());
}
// при выходе из блока все незавершённые подзадачи гарантированно отменены
```

Встроенные политики:

- **`ShutdownOnFailure`** — при первом исключении немедленно отменяет остальные подзадачи; после `join()` вызывается `throwIfFailed()` для проброса ошибки наверх.
- **`ShutdownOnSuccess`** — возвращает результат первой успешно завершившейся задачи и отменяет остальные (сценарий «invoke-any»).

Каждая подзадача выполняется в **новом виртуальном потоке** (не из пула), что исключает исчерпание ресурсов.

#### Ответ от Claude

**Проблемы произвольного порождения задач через ExecutorService**

При классическом подходе с `ExecutorService` возникают три системные проблемы:

1. **Утечки потоков (thread leaks).** Если одна подзадача завершается с ошибкой, другие продолжают работу в фоне без какого-либо контроля. Разработчик обязан реализовать отмену вручную — через `future.cancel()` или флаги прерывания.

2. **Неравномерное распространение ошибок.** При одновременном ожидании нескольких `Future` ошибка в одном из них не прерывает ожидание остальных. Пример: если `invoiceFuture` готов, но `paymentFuture` ещё выполняется и `customer.get()` блокирует поток — исключение обнаруживается только после завершения всех задач.

3. **Проблемы наблюдаемости и отладки.** Потоки не имеют явного отношения «родитель — потомок» на уровне API, что усложняет диагностику и трассировку.

**Что даёт StructuredTaskScope**

`StructuredTaskScope` устанавливает **явную иерархию задач**: подзадачи не могут пережить родительскую задачу, что гарантирует инвариант — при выходе из блока `try` все дочерние потоки завершены. Это устраняет все три проблемы:

- Отмена незавершённых подзадач — **автоматическая**, встроена в политику закрытия скопа.
- Ошибка немедленно останавливает ненужную работу (при `ShutdownOnFailure`).
- Иерархия потоков становится явной и видимой отладчику и профилировщику.

**Ключевые детали API**

```java
// fork() — запускает подзадачу в новом виртуальном потоке,
//           возвращает Subtask<T> (не Future<T>!)
Subtask<String> task = scope.fork(() -> callExternalApi());

// join() — ждёт завершения всех подзадач (или срабатывания политики shutdown)
scope.join();

// joinUntil(Instant deadline) — ожидание с дедлайном
scope.joinUntil(Instant.now().plusSeconds(5));

// throwIfFailed() — ShutdownOnFailure: пробрасывает сохранённое исключение
scope.throwIfFailed();

// result() — ShutdownOnSuccess: возвращает результат первой успешной задачи
Object winner = scope.result();
```

После `join()` состояние подзадачи: `SUCCESS`, `FAILED` или `UNAVAILABLE` (отменена). Результат получается через `subtask.get()`, исключение — через `subtask.exception()`.

**Политики (Java 21–24) и джойнеры (Java 25+)**

В Java 21–24 — две встроенные политики: `ShutdownOnFailure` и `ShutdownOnSuccess`. Начиная с JDK 25 (JEP 505) политики заменены на **джойнеры** (`Joiner`):
- `Joiner.awaitAllSuccessfulOrThrow()` — ждёт все; при первой ошибке бросает исключение.
- `Joiner.anySuccessfulResultOrThrow()` — возвращает первый успешный результат.
- `Joiner.allSuccessfulOrThrow()` — возвращает `Stream` всех результатов.

**Вложенные скопы и отмена**

`StructuredTaskScope` поддерживает вложенность. Прерывание родительской задачи автоматически распространяется на все дочерние скопы. `ScopedValue`-привязки родительского потока наследуются дочерними виртуальными потоками без дополнительных усилий.

**Статус API:** preview с Java 21 (JEP 428 — incubator с Java 19). Для Java 21–24 требуется `--enable-preview`.

### Вопрос 47. Что такое ForkJoinPool.ManagedBlocker? Зачем он нужен и как позволяет выполнять блокирующие операции внутри ForkJoinPool без деградации параллелизма?

**Категория:** Concurrency · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [ForkJoinPool.ManagedBlocker (Java SE 21 & JDK 21)](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/util/concurrent/ForkJoinPool.ManagedBlocker.html)

`ForkJoinPool.ManagedBlocker` — интерфейс для расширения управляемого параллелизма для задач, выполняющихся в `ForkJoinPool`. Предоставляет два метода:

- **`boolean isReleasable()`** — возвращает `true`, если блокировка уже не нужна (например, блокировка уже получена). Должен допускать повторный вызов.
- **`boolean block() throws InterruptedException`** — выполняет фактическую блокировку; возвращает `true`, если дальнейшая блокировка не требуется.

Метод `ForkJoinPool.managedBlock(ManagedBlocker)` выполняет возможно блокирующую задачу и при работе внутри `ForkJoinPool` **может активировать запасной поток-компенсатор**, чтобы обеспечить достаточный параллелизм, пока текущий поток заблокирован.

Внутренний цикл выглядит примерно так:

```
while (!blocker.isReleasable() && !blocker.block()) {}
```

Пример реализации для `BlockingQueue`:

```java
class QueueTaker<E> implements ForkJoinPool.ManagedBlocker {
    final BlockingQueue<E> queue;
    volatile E item = null;

    QueueTaker(BlockingQueue<E> q) { this.queue = q; }

    public boolean block() throws InterruptedException {
        if (item == null) item = queue.take();
        return true;
    }

    public boolean isReleasable() {
        return item != null || (item = queue.poll()) != null;
    }

    public E getItem() { return item; }
}
```

Пример реализации для `ReentrantLock`:

```java
class ManagedLocker implements ForkJoinPool.ManagedBlocker {
    final ReentrantLock lock;
    boolean hasLock = false;

    ManagedLocker(ReentrantLock lock) { this.lock = lock; }

    public boolean block() {
        if (!hasLock) lock.lock();
        return true;
    }

    public boolean isReleasable() {
        return hasLock || (hasLock = lock.tryLock());
    }
}
```

#### Ответ от Claude

**Проблема: деградация параллелизма при блокировке в ForkJoinPool**

`ForkJoinPool` (используется в параллельных потоках через `commonPool()`, а также напрямую) создаётся с заданным **уровнем параллелизма** (по умолчанию — число доступных процессоров). Когда рабочий поток блокируется на обычной блокирующей операции (например, `BlockingQueue.take()`, `ReentrantLock.lock()`, `Thread.sleep()`), он продолжает занимать слот в пуле, но не выполняет никакой работы. Если заблокированы несколько потоков, другие задачи в очереди не могут выполняться — возникает деградация параллелизма, а в крайнем случае — **взаимоблокировка** (deadlock): все рабочие потоки заблокированы, а задачи, от которых они зависят, стоят в очереди.

**Решение: ManagedBlocker и компенсационные потоки**

`ForkJoinPool.ManagedBlocker` позволяет сигнализировать пулу: «я собираюсь заблокироваться». В ответ `ForkJoinPool.managedBlock()` **активирует поток-компенсатор** (spare thread), поддерживая фактический параллелизм на целевом уровне. Когда заблокированный поток разблокируется, компенсатор становится ненужным и может быть завершён.

Механизм работы `managedBlock`:

1. Вызывается `blocker.isReleasable()` — если возвращает `true`, блокировка не нужна, выход.
2. Если `false` — пул **создаёт поток-компенсатор** для поддержания параллелизма.
3. Вызывается `blocker.block()` — выполняется фактическая блокирующая операция.
4. Цикл `while (!blocker.isReleasable() && !blocker.block())` повторяется до освобождения.

**Использование**

```java
QueueTaker<String> taker = new QueueTaker<>(blockingQueue);
ForkJoinPool.managedBlock(taker);
String item = taker.getItem();
```

**Рекомендации по реализации**

- `isReleasable()` должен выполнять **неблокирующую** проверку (например, `tryLock()`, `queue.poll()`). Если условие уже выполнено — возвращать `true` немедленно.
- `block()` должен выполнять реальную блокирующую операцию и возвращать `true` только после освобождения.
- Оба метода должны допускать повторный вызов и быть потокобезопасными.

**Когда применять**

- Блокирующие операции ввода-вывода или ожидания внутри задач параллельного потока (`parallelStream()`).
- Ожидание на `BlockingQueue`, `Semaphore`, `ReentrantLock` внутри `ForkJoinPool`.
- Адаптация легаси-кода с блокирующими вызовами для использования в `ForkJoinPool`.

**Когда не применять**

- Если архитектурно возможно избежать блокировок внутри `ForkJoinPool` — предпочтительнее переработать код (использовать неблокирующие структуры данных или вынести блокирующую операцию за пределы пула).
- Для виртуальных потоков (Project Loom) необходимость в `ManagedBlocker` значительно снижается: JVM самостоятельно демонтирует виртуальный поток с платформенного при блокировке, не удерживая рабочий поток пула.

**Расхождение между источниками:** Официальная документация Oracle описывает компенсатор как «запасной поток, который **может быть** активирован» (may arrange), подчёркивая, что это не гарантировано, а является эвристикой. JavaSpecialists (Issue 223) подтверждает практическое поведение: компенсатор действительно создаётся, однако предупреждает, что число компенсаторов ограничено (`commonPool` имеет предел на максимальное число потоков) — при злоупотреблении `ManagedBlocker` можно исчерпать и этот предел.

### Вопрос 48. Как понизить блокировку записи до блокировки чтения (lock downgrade) в ReentrantReadWriteLock? Почему обратная операция — повышение (lock upgrade) — не поддерживается?

**Категория:** Concurrency · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [ReentrantReadWriteLock (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/locks/ReentrantReadWriteLock.html)

Повторный вход (reentrancy) позволяет понижать уровень блокировки — с блокировки записи до блокировки чтения. Для этого нужно: сначала захватить блокировку записи, затем захватить блокировку чтения, а после — освободить блокировку записи. Однако повышение — с блокировки чтения до блокировки записи — **невозможно**.

Правило reentrancy: **писатель может захватить блокировку чтения, но читатель — нет**. Если читатель попытается захватить блокировку записи, он никогда не добьётся успеха.

Ниже приведён пример из официальной документации — обновление кэша с последующим понижением блокировки:

```java
class CachedData {
    Object data;
    volatile boolean cacheValid;
    final ReentrantReadWriteLock rwl = new ReentrantReadWriteLock();

    void processCachedData() {
        rwl.readLock().lock();
        if (!cacheValid) {
            // Перед захватом блокировки записи необходимо освободить блокировку чтения
            rwl.readLock().unlock();
            rwl.writeLock().lock();
            try {
                // Повторная проверка: другой поток мог успеть изменить данные
                if (!cacheValid) {
                    data = ...
                    cacheValid = true;
                }
                // Понижение: захватываем блокировку чтения, ещё удерживая блокировку записи
                rwl.readLock().lock();
            } finally {
                rwl.writeLock().unlock(); // Отпускаем запись, блокировку чтения ещё держим
            }
        }

        try {
            use(data);
        } finally {
            rwl.readLock().unlock();
        }
    }
}
```

Класс поддерживает максимум 65 535 рекурсивных блокировок записи и 65 535 блокировок чтения.

#### Ответ от Claude

**Механизм понижения блокировки (lock downgrade)**

Понижение блокировки — это паттерн, при котором поток переходит от эксклюзивного доступа (write lock) к совместному (read lock), не отпуская данные между этими двумя захватами. Алгоритм:

1. Захватить `writeLock`.
2. Выполнить запись/обновление данных.
3. **Не отпуская** `writeLock`, захватить `readLock`.
4. Освободить `writeLock` (теперь поток держит только `readLock`).
5. Использовать данные под `readLock`.
6. Освободить `readLock`.

Ключевой момент — шаги 3 и 4 выполняются именно в таком порядке. Если сначала освободить `writeLock`, а затем взять `readLock`, другой поток может успеть захватить `writeLock` и изменить данные — гарантия видимости обновлённого значения будет нарушена.

**Почему lock upgrade не поддерживается**

Повышение блокировки — переход от `readLock` к `writeLock` — не поддерживается из-за неустранимого риска взаимоблокировки (deadlock). Источники (Oracle Javadoc, статьи на medium.com) сходятся в одном объяснении:

- Блокировка записи ждёт, пока все читатели завершат работу.
- Если читающий поток A пытается захватить `writeLock`, он должен сначала дождаться, пока **все** активные `readLock` будут освобождены.
- Но сам поток A удерживает один из `readLock` и не может его освободить, пока не получит `writeLock`.
- Если ещё и поток B в этот момент попытается сделать то же самое — оба потока заблокируют друг друга навсегда.

Даже в случае одного потока без других читателей Java-реализация не делает исключения: правило «читатель не может получить writeLock» применяется безусловно, чтобы избежать скрытых дедлоков в многопоточных сценариях.

**Сравнение downgrade vs upgrade**

| Операция | Поддерживается | Причина |
|---|---|---|
| Write → Read (downgrade) | Да | Поток уже имеет эксклюзивный доступ; добавить разделяемый — безопасно |
| Read → Write (upgrade) | Нет | Круговое ожидание: writeLock ждёт освобождения readLock, которого поток сам и не отпустит |

**Альтернатива при необходимости upgrade** — явно освободить `readLock`, захватить `writeLock`, и перепроверить состояние (double-check, как в примере выше с `cacheValid`), поскольку за время между двумя захватами данные могут измениться.

### Вопрос 49. Как работает orTimeout() и completeOnTimeout() в CompletableFuture (Java 9+)? Чем они принципиально отличаются друг от друга и от future.get(timeout, unit)?

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [CompletableFuture (Java SE 11 & JDK 11) — docs.oracle.com](https://docs.oracle.com/en/java/javase/11/docs/api/java.base/java/util/concurrent/CompletableFuture.html)

**orTimeout(long timeout, TimeUnit unit)**

Сигнатура:
```java
public CompletableFuture<T> orTimeout(long timeout, TimeUnit unit)
```

Исключительно завершает данный `CompletableFuture` с `TimeoutException`, если он не был завершён иным образом до истечения указанного тайм-аута. Возвращает тот же экземпляр `CompletableFuture`. Добавлен в Java 9.

**completeOnTimeout(T value, long timeout, TimeUnit unit)**

Сигнатура:
```java
public CompletableFuture<T> completeOnTimeout(T value, long timeout, TimeUnit unit)
```

Нормально завершает данный `CompletableFuture` с заданным значением, если он не был завершён иным образом до истечения указанного тайм-аута. Возвращает тот же экземпляр `CompletableFuture`. Добавлен в Java 9.

**get(long timeout, TimeUnit unit)** (из интерфейса `Future`)

Блокирует вызывающий поток не дольше указанного времени, ожидая завершения future. При истечении тайм-аута бросает `TimeoutException` из самого `get()`. Наследован от `Future<T>`.

#### Ответ от Claude

**Три подхода к тайм-ауту CompletableFuture**

**1. `get(long timeout, TimeUnit unit)` — блокирующий подход**

Вызов `get()` **блокирует вызывающий поток** до тех пор, пока future не завершится или не истечёт тайм-аут. Это синхронный вызов: поток стоит и ждёт. При тайм-ауте бросает `TimeoutException` непосредственно из `get()`, и сам `CompletableFuture` при этом **не изменяет своё состояние** — он остаётся в незавершённом состоянии (pending). Фоновая задача при этом продолжает работать.

```java
try {
    String result = future.get(1, TimeUnit.SECONDS); // поток заблокирован
} catch (TimeoutException e) {
    // future всё ещё не завершена, фоновая задача работает
}
```

**2. `orTimeout(long timeout, TimeUnit unit)` — неблокирующее исключение**

Метод возвращается **немедленно**, не блокируя поток. Он планирует: если future не завершится за указанное время, она будет завершена **исключительно** — с `TimeoutException` внутри `ExecutionException`. Вся цепочка `.exceptionally()` / `.handle()` / `.whenComplete()` будет вызвана с этим исключением.

```java
CompletableFuture<Integer> future = CompletableFuture.supplyAsync(this::longComputation)
    .orTimeout(1, TimeUnit.SECONDS);

future.get(); // бросит ExecutionException(TimeoutException) через ~1 секунду
```

Важно: согласно источнику 4comprehension.com, **фоновая задача при этом продолжает выполняться** — тайм-аут завершает future, но не отменяет и не прерывает задачу в пуле потоков.

**3. `completeOnTimeout(T value, long timeout, TimeUnit unit)` — неблокирующий фолбэк**

Аналогично `orTimeout()` возвращается немедленно. Если future не завершится за указанное время, она завершится **нормально** — с переданным значением-фолбэком. Никакого исключения нет; последующие `.thenApply()` / `.thenAccept()` получат это значение и продолжат работу в штатном режиме.

```java
CompletableFuture<Integer> future = CompletableFuture.supplyAsync(this::longComputation)
    .completeOnTimeout(42, 1, TimeUnit.SECONDS);

Integer result = future.get(); // вернёт 42 при тайм-ауте
```

Ограничение, отмеченное в источнике 4comprehension.com: параметр `value` — это уже вычисленный объект, а не `Supplier<T>`. Значение по умолчанию нельзя вычислить лениво.

**Сравнительная таблица**

| Критерий | `get(timeout, unit)` | `orTimeout()` | `completeOnTimeout(value, ...)` |
|---|---|---|---|
| Блокирует поток | Да | Нет | Нет |
| Состояние future при тайм-ауте | Не изменяется | Завершается исключительно (`TimeoutException`) | Завершается нормально (переданным значением) |
| Тип завершения при тайм-ауте | — | Исключительное | Нормальное |
| Цепочка `.thenApply()` | Не вызывается | Не вызывается | Вызывается с фолбэком |
| Цепочка `.exceptionally()` | Не вызывается | Вызывается | Не вызывается |
| Фоновая задача при тайм-ауте | Продолжает работать | Продолжает работать | Продолжает работать |
| Появилось в Java | 1 (Future) | 9 | 9 |

**Практические рекомендации**

- Используйте `orTimeout()`, когда тайм-аут означает ошибку и нужно передать её по всей цепочке.
- Используйте `completeOnTimeout()`, когда есть разумное значение по умолчанию и тайм-аут — допустимый сценарий (например, кэшированное значение вместо свежего).
- Избегайте `get(timeout)` в неблокирующих/реактивных пайплайнах — он разрушает асинхронность.

### Вопрос 50. Как правильно использовать AtomicReference для атомарного обновления объекта-ссылки? Приведите пример применения compareAndSet() для реализации lock-free обновления составного состояния.

**Категория:** Concurrency · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [AtomicReference (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/atomic/AtomicReference.html)

`AtomicReference<V>` — объектная ссылка, которая может обновляться атомарно. Находится в пакете `java.util.concurrent.atomic`. Доступна с Java 1.5.

Метод `compareAndSet`:

```java
public final boolean compareAndSet(V expect, V update)
```

Атомарно устанавливает значение в `update`, если текущее значение `==` ожидаемому (`expect`). Возвращает `true` при успехе; `false` означает, что фактическое значение не совпало с ожидаемым.

Из описания пакета `java.util.concurrent.atomic`:

> Это набор инструментальных классов для lock-free потокобезопасного программирования с **одной переменной**. Метод `compareAndSet` не является универсальной заменой блокировкам и применим только тогда, когда критические обновления объекта ограничены **одной переменной**.

Паттерн атомарного обновления через CAS-цикл:

```java
long getAndTransform(AtomicLong var) {
    long prev, next;
    do {
        prev = var.get();
        next = transform(prev);
    } while (!var.compareAndSet(prev, next));
    return prev;
}
```

#### Ответ от Claude

**Как работает `AtomicReference` и `compareAndSet`**

`AtomicReference<V>` хранит ссылку на объект типа `V` и предоставляет атомарные операции над этой ссылкой. Под капотом `compareAndSet()` транслируется JVM в единую машинную инструкцию CAS (Compare-And-Swap), которая на уровне процессора выполняется неделимо.

Важное уточнение из Javadoc (jenkov.com): сравнение в `compareAndSet()` происходит по **ссылочному равенству** (`==`), а не по `equals()`. Это принципиально: метод сравнивает, является ли сохранённая ссылка тем же объектом в памяти, а не «равным» объектом.

**Паттерн lock-free обновления составного состояния**

Для атомарного обновления нескольких полей одновременно используется следующая идиома: составное состояние упаковывается в **неизменяемый (immutable) объект**, и `AtomicReference` хранит ссылку на актуальный снимок состояния. При необходимости изменения создаётся новый объект, и CAS атомарно заменяет ссылку.

```java
// Неизменяемый класс, описывающий составное состояние
final class Config {
    final String host;
    final int port;
    final int maxConnections;

    Config(String host, int port, int maxConnections) {
        this.host = host;
        this.port = port;
        this.maxConnections = maxConnections;
    }
}

// Lock-free обновление
class ConfigHolder {
    private final AtomicReference<Config> configRef =
        new AtomicReference<>(new Config("localhost", 5432, 10));

    public void updateMaxConnections(int newMax) {
        Config current, next;
        do {
            current = configRef.get();                           // читаем текущий снимок
            next = new Config(current.host, current.port, newMax); // создаём новый
        } while (!configRef.compareAndSet(current, next));      // атомарно заменяем
        // Если другой поток успел изменить configRef между get() и CAS —
        // CAS вернёт false, и цикл повторится с новым current
    }

    public Config get() {
        return configRef.get();
    }
}
```

**Почему объект должен быть неизменяемым**

Если `AtomicReference` хранит ссылку на **изменяемый** объект, атомарность теряется: CAS защищает только операцию замены ссылки, но не защищает поля самого объекта. Другой поток, получив ссылку через `get()`, может увидеть объект в промежуточном состоянии, если его поля изменяются без синхронизации. Неизменяемость объекта гарантирует, что любой поток, получивший ссылку, увидит полностью сформированный и согласованный снимок состояния.

**Проблема ABA**

CAS сравнивает ссылки. Если поток A прочитал ссылку на объект X, другой поток B заменил X на Y, а затем снова на X (тот же адрес), поток A не заметит промежуточного изменения — CAS пройдёт успешно. Это классическая проблема ABA. Для её решения используется `AtomicStampedReference<V>`, который дополнительно хранит числовую метку (stamp/version), обновляемую при каждом изменении.

**Ограничения AtomicReference**

Согласно Javadoc пакета `java.util.concurrent.atomic`, `compareAndSet` применим только тогда, когда критические обновления объекта ограничены **одной** переменной. Для сложных многошаговых транзакций (несколько независимых `AtomicReference`) lock-free алгоритмы резко усложняются; в таких случаях лучше рассмотреть `StampedLock`, `synchronized` или транзакционную память (`VarHandle` с Java 9+).

**Сравнение с `compareAndExchange` (Java 9+)**

В Java 9 добавлен `compareAndExchange(V expect, V update)`, который в отличие от `compareAndSet` возвращает **фактическое** значение, а не `boolean`. Это позволяет избежать повторного `get()` в CAS-цикле при неудаче, что немного эффективнее в высококонкурентных сценариях.
### Вопрос 51. Какие гарантии happens-before даёт завершение статического инициализатора класса? Почему инициализация через статический вложенный класс-держатель (Initialization-on-demand holder) потокобезопасна без synchronized?

**Категория:** Concurrency · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Java Language Specification, §12.4.2 — Detailed Initialization Procedure](https://docs.oracle.com/javase/specs/jls/se21/html/jls-12.html#jls-12.4)

Для каждого класса C существует уникальный блокировщик инициализации LC (class initialization lock). Процедура инициализации выглядит следующим образом:

1. Поток захватывает LC.
2. Если другой поток уже инициализирует C — текущий поток освобождает LC, блокируется и ждёт.
3. Если текущий поток уже инициализирует C (рекурсивный случай) — немедленный возврат.
4. Если C уже полностью инициализирован — немедленный возврат.
5. Если C находится в ошибочном состоянии — бросается `NoClassDefFoundError`.
6. Поток отмечает C как «инициализируется текущим потоком», освобождает LC.
7. Выполняются статические инициализаторы и инициализаторы статических полей в порядке объявления.
8. После успешного завершения: поток снова захватывает LC, отмечает C как «полностью инициализирован», уведомляет все ожидающие потоки (`notifyAll`), освобождает LC.

JLS гарантирует: **завершение статических инициализаторов класса C happens-before любого последующего обращения к статическим полям C из любого потока**. Механизм — именно lock/unlock на LC: release на LC по окончании инициализации happens-before acquire того же LC в любом другом потоке, убеждающемся в завершённости инициализации. Таким образом, все записи, сделанные в статическом инициализаторе, гарантированно видны читателям.

#### Ответ от Claude

**Гарантия happens-before для статического инициализатора**

Java Memory Model (JMM, JLS §17.4.5) строится на понятии *синхронизирующих действий* (synchronization actions). Захват и освобождение монитора (блокировки) являются такими действиями, и между ними JMM устанавливает happens-before ребро.

Механизм класс-инициализации (JLS §12.4.2) использует внутренний объект — class initialization lock LC, уникальный для каждого класса. Когда поток-инициализатор завершает выполнение всех статических инициализаторов и инициализаторов статических полей, он:
1. захватывает LC,
2. переводит класс в состояние «fully initialized»,
3. вызывает `notifyAll` на LC,
4. освобождает LC.

Любой другой поток, обращающийся к статическому полю класса C впервые, должен убедиться, что C инициализирован — для этого он синхронизируется на LC (в оптимизированных реализациях JVM это может быть сведено к проверке флага, но JMM-ребро всё равно гарантируется). Из правила happens-before для мониторов: *release(LC) happens-before acquire(LC)*, а значит, всё, что происходило в статическом инициализаторе, **happens-before** чтения любого статического поля другим потоком.

Итого: записи в статическом инициализаторе полностью видны любому потоку, который читает статическое поле после завершения инициализации — без дополнительной синхронизации со стороны прикладного кода.

---

**Initialization-on-demand holder (IODH): почему потокобезопасно без synchronized**

Идиома выглядит так:

```java
public class Singleton {
    private Singleton() {}

    private static final class Holder {
        static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;  // единственная точка обращения к Holder
    }
}
```

Ключевые свойства:

1. **Ленивость.** Внутренний класс `Holder` — отдельный класс. JLS §12.4.1 гласит: класс инициализируется непосредственно перед первым активным использованием. Поскольку `Holder` нигде не упоминается до вызова `getInstance()`, он не загружается при загрузке `Singleton`.

2. **Однократность.** JLS §12.4.2 гарантирует, что class initialization lock обеспечивает взаимное исключение: инициализатор `Holder` выполнится ровно один раз, даже при конкурентном вызове `getInstance()` несколькими потоками.

3. **Видимость.** После завершения инициализации `Holder` (то есть после того, как `Holder.INSTANCE` присвоено значение) существует happens-before ребро между статическим инициализатором `Holder` и любым последующим чтением `Holder.INSTANCE`. Это означает: все потоки увидят полностью сконструированный объект `Singleton`.

4. **Отсутствие volatile и synchronized.** Все три свойства обеспечиваются JVM-механизмом загрузки классов — дополнительная синхронизация в методе `getInstance()` не нужна. В «горячем» пути (класс уже инициализирован) JVM не обязана захватывать LC, однако JMM-ребро уже установлено ранее — производительность на уровне обычного чтения статического поля.

**Расхождение между источниками.** Shipilev («Safe Public Construction», 2014) уточняет, что JLS разрешает JVM *пропускать* re-acquire LC при каждом чтении статического поля, если JVM может доказать, что видимость уже обеспечена другими средствами. Это даёт возможность оптимизировать горячий путь без нарушения гарантий JMM. Wikipedia и happycoders.eu описывают только верхнеуровневое правило («JVM ensures sequential initialization»), не упоминая эту оптимизацию — на практике результат одинаков, но понимание механизма важно для Senior-уровня.

**Единственное ограничение идиомы:** если конструктор `Singleton` выбросит исключение, класс `Holder` перейдёт в ошибочное состояние и последующие обращения к `getInstance()` будут бросать `NoClassDefFoundError`, а не повторять попытку инициализации.

### Вопрос 52. Почему использование ThreadLocal с виртуальными потоками (Java 21+) может привести к взрыву потребления памяти, даже без классической утечки? Как InheritableThreadLocal ведёт себя при наследовании в виртуальных потоках?

**Категория:** Concurrency · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [JEP 444: Virtual Threads (openjdk.org)](https://openjdk.org/jeps/444)

Виртуальные потоки поддерживают переменные `ThreadLocal` и `InheritableThreadLocal`, как и платформенные потоки, — это необходимо для обратной совместимости. Однако JEP 444 явно предупреждает:

> «Because virtual threads can be very numerous, use thread locals only after careful consideration. In particular, do not use thread locals to pool costly resources among multiple tasks sharing the same thread in a thread pool.»

Проблема не в классической утечке (отсутствии вызова `remove()`), а в **масштабе**: если ранее приложение держало 200 платформенных потоков с `ThreadLocal`-кэшем, то с виртуальными потоками то же приложение может держать одновременно миллион виртуальных потоков, каждый со своим `ThreadLocalMap` — и суммарное потребление памяти вырастает линейно.

JEP также отмечает: из JDK-библиотек были убраны многие собственные `ThreadLocal`-переменные, чтобы уменьшить footprint при запуске миллионов потоков. Например, уменьшены начальные размеры буферов `BufferedOutputStream` и `BufferedWriter`. В качестве долгосрочного решения JEP предлагает `ScopedValue` (JEP 446, preview в Java 21; финализировано в JEP 506, Java 23).

#### Ответ от Claude

**Механизм «взрыва» памяти без классической утечки**

Классическая утечка `ThreadLocal` в пуле потоков — это когда поток не вызывает `remove()`, и значение живёт до конца жизни потока, который потом переиспользуется с «грязным» состоянием. Этот сценарий хорошо изучен.

С виртуальными потоками возникает **качественно иная проблема**:

1. **Масштаб одновременных потоков.** Платформенные потоки потребляют ~1 МБ стека каждый; приложение держит их сотни или единицы тысяч. Виртуальные потоки дёшевы (~kilобайты стека, выделяемые динамически), и типичная модель — один виртуальный поток на запрос. При 10 000 RPS и времени ответа 500 мс в полёте одновременно около 5 000 потоков; при высокой нагрузке — сотни тысяч или миллионы.

2. **ThreadLocalMap живёт вместе с потоком.** Каждый виртуальный поток (`Thread.currentThread()` возвращает именно его) имеет собственный `ThreadLocalMap`. Если код кладёт в `ThreadLocal` объект весом 10 КБ (например, кэш, форматтер, буфер), то миллион одновременных виртуальных потоков создаёт ~10 ГБ накладных расходов — без единой «утечки» в классическом смысле: каждый `ThreadLocalMap` будет корректно собран GC после завершения потока.

3. **Паттерн «кэш в ThreadLocal» ломается.** В модели пула потоков `ThreadLocal`-кэш был разумен: дорогой объект создаётся один раз на поток и переиспользуется многими задачами. В модели «поток на запрос» кэш создаётся заново для каждого запроса и уничтожается вместе с потоком — кэширования не происходит, только накладные расходы на создание и GC.

**InheritableThreadLocal при наследовании в виртуальных потоках**

`InheritableThreadLocal` работает следующим образом: при создании дочернего потока (`new Thread(...)` или через builder) JVM копирует `ThreadLocalMap` родителя в дочерний поток, применяя `childValue()` к каждому значению. Это справедливо и для виртуальных потоков.

Специфика поведения с виртуальными потоками:

1. **Наследование происходит в момент создания, однократно.** Если виртуальный поток создан через `Thread.ofVirtual().start(runnable)` или через `ExecutorService` (например, `Executors.newVirtualThreadPerTaskExecutor()`), он наследует значения `InheritableThreadLocal` от родительского потока (того, кто создаёт виртуальный поток). Последующие изменения в родителе **не видны** уже созданному дочернему виртуальному потоку.

2. **Многократное копирование — дополнительный накладной расход.** Если одна задача (виртуальный поток) порождает много дочерних виртуальных потоков, каждый дочерний получает свою копию всего `ThreadLocalMap` родителя. При глубоких деревьях задач (structured concurrency через `StructuredTaskScope.fork()`) копирование накапливается. JEP 444 прямо указывает на этот источник потребления памяти: «child threads have to allocate storage for every thread-local variable previously written in the parent thread».

3. **StructuredTaskScope.fork() vs. new Thread().** При использовании `StructuredTaskScope.fork()` (API структурированного параллелизма) дочерние задачи являются виртуальными потоками, и `InheritableThreadLocal` наследуется от форкающего потока. Это может приводить к неожиданному распространению контекста (MDC, security context, tenant ID) в дочерние задачи — или, наоборот, к его отсутствию, если порождение идёт через executor, родителем которого является не прикладной поток.

4. **Opt-out через builder.** Начиная с Java 21 можно отключить наследование при создании виртуального потока:
   ```java
   Thread.ofVirtual()
       .inheritInheritableThreadLocals(false)
       .start(runnable);
   ```

**Рекомендуемая альтернатива: ScopedValue**

`ScopedValue` (JEP 429 preview → JEP 446 preview Java 21 → JEP 506 финал Java 23) решает обе проблемы:
- Значения **иммутабельны** — дочерние потоки читают родительское значение без копирования, что устраняет накладные расходы `InheritableThreadLocal`.
- Значения **ограничены по времени жизни** динамической областью видимости (`ScopedValue.where(...).run(...)`), что исключает нежелательное сохранение состояния между задачами.

**Диагностика.** JEP 444 предоставляет системное свойство `jdk.traceVirtualThreadLocals`: при его установке JVM печатает стектрейс каждый раз, когда виртуальный поток записывает значение в `ThreadLocal` — это помогает найти проблемные места при миграции.

**Расхождение между источниками.** JEP 444 и Oracle-документация описывают `InheritableThreadLocal` как поддерживаемый, но нежелательный механизм. Ряд статей (javathinking.com) уточняет, что если родителем виртуального потока является другой виртуальный поток (а не carrier/platform thread), наследование идёт от родительского виртуального потока — поведение консистентно с моделью `Thread`, но может удивить разработчиков, ожидающих наследование от «настоящего» потока.
### Вопрос 53. Опишите механизм монтирования/демонтирования (mount/unmount) виртуального потока на несущий поток (carrier thread). Где хранятся стековые фреймы виртуального потока, когда он демонтирован? Что такое continuation и какую роль оно играет в этом процессе?

**Категория:** Concurrency · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Virtual Threads — Oracle Java SE 21 Core Libraries Guide](https://docs.oracle.com/en/java/javase/21/core/virtual-threads.html)

Когда среда выполнения Java планирует виртуальный поток, она монтирует его на платформенный поток — тот называется несущим (carrier). Затем операционная система планирует несущий поток в обычном режиме. После выполнения некоторого кода виртуальный поток может демонтироваться со своего несущего — как правило, это происходит при выполнении блокирующей операции ввода-вывода. Стековые трассировки несущего и виртуального потока разделены: исключение, выброшенное внутри виртуального потока, не включает кадры несущего, и наоборот.

Виртуальные потоки реализованы по аналогии с виртуальной памятью: подобно тому как операционная система отображает большое виртуальное адресное пространство на ограниченный объём RAM, среда выполнения Java отображает большое количество виртуальных потоков на малое количество потоков ОС.

#### Ответ от Claude

**Механизм монтирования и демонтирования**

Монтирование (mount) — это процесс назначения виртуального потока на конкретный несущий поток (platform thread) для исполнения. С точки зрения Java-кода это прозрачно: виртуальный поток не знает, на каком именно несущем он выполняется, и несущий может меняться между двумя точками возобновления.

При обнаружении блокирующей операции (I/O, `LockSupport.park()`, `Thread.sleep()` и другие виртуально-потоко-осведомлённые методы JDK) выполняется демонтирование (unmount): стековые кадры виртуального потока переносятся обратно в кучу, несущий поток освобождается и может принять другой виртуальный поток. Когда операция завершается, виртуальный поток снова становится готовым к выполнению и будет смонтирован на любой доступный несущий.

**Где хранятся стековые кадры при демонтировании**

Стековые кадры виртуального потока хранятся в куче Java в виде объектов-фрагментов стека (stack chunk objects). Это динамические, растущие/уменьшающиеся структуры, управляемые сборщиком мусора. При монтировании JVM копирует кадры из кучи на стек несущего потока (лениво, по мере необходимости — не все кадры сразу); при демонтировании — копирует их обратно в кучу. Благодаря этому виртуальные потоки изначально занимают лишь сотни байт, а не мегабайты, как платформенные потоки с фиксированным стеком в нативной памяти.

**Что такое continuation и его роль**

Continuation (продолжение) — это абстракция нижнего уровня, инкапсулирующая текущее состояние исполнения виртуального потока: счётчик программы, локальные переменные, цепочку стековых кадров. По сути, это «замороженный» снимок потока выполнения, который можно приостановить (`yield`) и возобновить позже.

Жизненный цикл continuation внутри виртуального потока:
1. При создании виртуального потока для него создаётся объект `Continuation`, оборачивающий пользовательскую задачу (Runnable).
2. При монтировании планировщик запускает (или возобновляет) continuation на несущем потоке.
3. При встрече блокирующей операции continuation вызывает `yield` — стек сериализуется в кучу, несущий освобождается.
4. После завершения блокирующей операции continuation снова ставится в очередь планировщика и при следующем монтировании возобновляется с точки `yield`.

Таким образом, continuation — это механизм, позволяющий JVM реализовать кооперативное (не вытесняющее) планирование виртуальных потоков без изменений в байткоде пользователя.

**Источники:** JEP 444 (openjdk.org/jeps/444); Oracle Java SE 21 Core Libraries Guide (docs.oracle.com/en/java/javase/21/core/virtual-threads.html); «Virtual Threads Under the Hood» (medium.com/@gauravrmsc).

### Вопрос 54. Каков размер пула несущих потоков (carrier threads) для виртуальных потоков по умолчанию и как он соотносится с количеством доступных процессоров? Почему планировщик виртуальных потоков использует ForkJoinPool в режиме FIFO, а не LIFO?

**Категория:** Concurrency · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [JEP 444: Virtual Threads — OpenJDK](https://openjdk.org/jeps/444)

Планировщик виртуальных потоков — это реализация `java.util.concurrent.ForkJoinPool`, работающая в режиме work-stealing с очередью FIFO. Данный пул отличается от общего пула (`ForkJoinPool.commonPool()`), используемого, например, параллельными стримами, который работает в режиме LIFO. Степень параллелизма планировщика — количество платформенных потоков, доступных для планирования виртуальных потоков — по умолчанию равна числу доступных процессоров (`Runtime.availableProcessors()`). Это значение можно изменить через системное свойство `jdk.virtualThreadScheduler.parallelism`. Количество платформенных потоков в пуле планировщика может временно превышать число доступных процессоров; максимальное количество регулируется свойством `jdk.virtualThreadScheduler.maxPoolSize`.

#### Ответ от Claude

**Размер пула несущих потоков**

По умолчанию количество несущих потоков (parallelism) равно `Runtime.availableProcessors()` — то есть числу логических процессоров машины. Это не жёсткий потолок числа платформенных потоков в пуле: при наступлении блокирующих операций, которые не позволяют демонтировать виртуальный поток (pinning), JVM временно расширяет пул за счёт дополнительных потоков-компенсаторов, чтобы общий параллелизм оставался на целевом уровне. Максимальный размер пула ограничен значением `jdk.virtualThreadScheduler.maxPoolSize` (по умолчанию `max(parallelism, 256)`).

Ключевые системные свойства:
- `jdk.virtualThreadScheduler.parallelism` — целевая степень параллелизма (по умолчанию = число ядер ЦП).
- `jdk.virtualThreadScheduler.maxPoolSize` — максимальное количество платформенных потоков в пуле (по умолчанию `max(parallelism, 256)`).
- `jdk.virtualThreadScheduler.minRunnable` — минимальное число потоков в состоянии runnable (по умолчанию `max(parallelism / 2, 1)`).

**Почему FIFO, а не LIFO**

Стандартный `ForkJoinPool` (в том числе common pool параллельных стримов) работает в режиме LIFO (Last-In-First-Out) — это оптимально для рекурсивных задач типа «разделяй и властвуй» (divide-and-conquer), где дочерние задачи зависят от результатов родительских и выгодно завершать их скорее, пока данные горячие в кэше процессора.

Виртуальные потоки — принципиально иная рабочая нагрузка: они независимы друг от друга, не образуют иерархий fork-join, а их основная характеристика — огромное количество (миллионы) самостоятельных I/O-задач. Для такого профиля FIFO предпочтительнее по ряду причин:

1. **Справедливость и отсутствие голодания.** В режиме LIFO постоянно поступающие новые задачи могут вытеснять давно ожидающие, что приводит к indefinite starvation. FIFO гарантирует, что задачи обрабатываются в порядке поступления.
2. **Предсказуемая латентность.** Для серверных приложений равномерное время ответа важнее, чем максимизация локальности кэша.
3. **Work-stealing с FIFO.** При похищении задач из чужих очередей (work stealing) поток берёт самые старые задачи, что дополнительно страхует от голодания.

Расхождение между источниками: некоторые статьи описывают FIFO как «less cache-friendly but fairer» без ссылки на официальную документацию с явным обоснованием. JEP 444 фиксирует факт использования FIFO, но не детализирует мотивацию публично; приведённое обоснование выводится из общей теории планирования и архитектурных соображений, изложенных в мейлинг-листах проекта Loom.

**Источники:** JEP 444 (openjdk.org/jeps/444); Alibaba Cloud — «Exploration of Java Virtual Threads» (alibabacloud.com/blog/601860); Rock the JVM — «The Ultimate Guide to Java Virtual Threads» (rockthejvm.com/articles/the-ultimate-guide-to-java-virtual-threads).

### Вопрос 55. Что такое pinning (закрепление) виртуального потока на несущем потоке? В каких двух принципиально разных ситуациях оно возникает? Чем поведение synchronized отличается от ReentrantLock при блокировке внутри виртуального потока?

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Virtual Threads — Oracle Java SE 21 Core Libraries Guide](https://docs.oracle.com/en/java/javase/21/core/virtual-threads.html)

Виртуальный поток не может демонтироваться во время блокирующей операции, если он закреплён (pinned) на несущем. Закрепление возникает в двух ситуациях:

1. Виртуальный поток выполняет код внутри блока или метода `synchronized`.
2. Виртуальный поток выполняет нативный метод или внешнюю функцию (Foreign Function).

Закрепление не делает приложение некорректным, однако может снизить его масштабируемость. Если обнаружено, что закрепление является длительным и частым, следует заменить `synchronized` на `ReentrantLock` в этих конкретных местах.

Для диагностики используются:
- JDK Flight Recorder: событие `jdk.VirtualThreadPinned` (по умолчанию включено, порог 20 мс).
- Системное свойство `-Djdk.tracePinnedThreads=full` — полная трассировка при закреплении.
- `-Djdk.tracePinnedThreads=short` — только проблемные кадры.

#### Ответ от Claude

**Что такое pinning**

Закрепление (pinning) — это состояние, при котором виртуальный поток не может демонтироваться со своего несущего во время блокирующей операции. Несущий платформенный поток вынужден оставаться заблокированным вместе с виртуальным, что лишает другие виртуальные потоки возможности использовать этот несущий. При достаточном числе одновременно закреплённых виртуальных потоков пул несущих может быть полностью исчерпан и планировщик остановится.

**Два принципиально разных случая закрепления**

1. **Синхронизированный блок/метод (`synchronized`)** (актуально для Java 21–23).
   Причина: в Java 21–23 JVM отслеживает владение монитором объекта по несущему платформенному потоку, а не по виртуальному. Если бы JVM демонтировала виртуальный поток внутри `synchronized`, другой виртуальный поток, смонтированный на тот же несущий, формально «унаследовал» бы владение монитором, нарушив семантику взаимного исключения. Поэтому демонтирование запрещено. Начиная с Java 24 (JEP 491) проблема устранена: JVM теперь отслеживает владение монитором по виртуальному потоку, и `synchronized` больше не вызывает закрепления.

2. **Нативные методы и иностранные функции (JNI / Foreign Function & Memory API)**.
   Причина: JVM не может безопасно управлять состоянием потока внутри нативного кода (C/C++ и т.п.). Нативный код может хранить собственные указатели на поток ОС, делать прямые системные вызовы и т.д. — демонтирование при этом физически невозможно. Этот вид закрепления остался после JEP 491 и является ожидаемым поведением.

**Сравнение `synchronized` и `ReentrantLock` (Java 21–23)**

| Аспект | `synchronized` | `ReentrantLock` |
|---|---|---|
| Механизм блокировки | Монитор объекта, привязан к несущему потоку (Java 21–23) | `LockSupport.park()` — виртуально-потоко-осведомлённый |
| Поведение при блокировке | Несущий поток блокируется вместе с виртуальным (pinning) | Виртуальный поток демонтируется, несущий свободен |
| Масштабируемость | Ухудшается при длительных I/O внутри блока | Сохраняется: несущий обслуживает другие виртуальные потоки |
| Статус в Java 24+ | Pinning устранён (JEP 491) | Поведение не изменилось |

`ReentrantLock` при ожидании захвата локa вызывает `LockSupport.park()`, который является точкой демонтирования, понятной JVM-планировщику. Виртуальный поток «засыпает» в куче, несущий освобождается. Когда лок становится доступным, `LockSupport.unpark()` возвращает виртуальный поток в очередь готовых.

**Практические рекомендации (Java 21–23)**

- Заменять `synchronized` на `ReentrantLock` следует только в блоках, которые: (а) выполняются часто И (б) содержат потенциально длительные операции (I/O, сетевые вызовы, `sleep`).
- Короткие in-memory операции внутри `synchronized` закреплению практически не вредят: несущий освобождается очень быстро.
- Использовать JFR-событие `jdk.VirtualThreadPinned` для профилирования перед рефакторингом.

**Расхождение между источниками:** ряд материалов (написанных до 2024 года) описывает `synchronized` как постоянную проблему, не упоминая JEP 491. Начиная с JDK 24 эта проблема устранена; актуальным остаётся только второй вид закрепления — нативные методы.

**Источники:** Oracle Java SE 21 Core Libraries Guide (docs.oracle.com/en/java/javase/21/core/virtual-threads.html); JEP 491 — Synchronize Virtual Threads without Pinning (openjdk.org/jeps/491); «Java Virtual Threads: The Pinning Problem» Shubham Raizada (shbhmrzd.github.io/java/concurrency/virtual-threads/2026/04/25/java-virtual-threads-pinning-and-the-deadlock-problem.html).

### Вопрос 56. Как обнаружить случаи pinning виртуальных потоков в production? Назовите два механизма JDK для диагностики pinning с указанием конкретных имён событий/свойств и порогового значения по умолчанию.

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Virtual Threads — Java SE 21 Core Libraries Developer Guide (docs.oracle.com)](https://docs.oracle.com/en/java/javase/21/core/virtual-threads.html)

JDK предоставляет два механизма для обнаружения pinning.

**1. JDK Flight Recorder (JFR).** Событие `jdk.VirtualThreadPinned` фиксируется всякий раз, когда виртуальный поток блокируется в состоянии pinning. Событие включено по умолчанию с порогом **20 мс**: если поток был прикреплён к carrier-потоку дольше этого значения, событие записывается в JFR-лог. Для анализа собранной записи используется команда:

```
jfr print --events jdk.VirtualThreadPinned recording.jfr
```

**2. Системное свойство `jdk.tracePinnedThreads`.** Запуск приложения с флагом `-Djdk.tracePinnedThreads=full` выводит полный стек-трейс при каждой блокировке прикреплённого потока, выделяя нативные фреймы и фреймы, удерживающие монитор. Значение `short` ограничивает вывод только проблемными фреймами.

> Примечание: начиная с Java 24 флаг `jdk.tracePinnedThreads` устарел и не оказывает эффекта — в JDK 24+ проблема pinning внутри `synchronized` решена на уровне JVM (JEP 491).

#### Ответ от Claude

Два официальных механизма диагностики pinning:

**Механизм 1 — JFR-событие `jdk.VirtualThreadPinned`** (рекомендован для production)

- Имя события: `jdk.VirtualThreadPinned`
- Включено по умолчанию: **да**
- Пороговое значение по умолчанию: **20 мс** (подтверждено в файле конфигурации JFR `default.jfc`: `<setting name="threshold">20 ms</setting>`)
- Событие фиксируется, когда виртуальный поток заблокирован в pinning-состоянии дольше порога; вместе с событием записывается стек-трейс
- Анализ: `jfr print --events jdk.VirtualThreadPinned recording.jfr`
- Можно также подписаться программно через `RecordingStream` API для онлайн-мониторинга

**Механизм 2 — системное свойство `jdk.tracePinnedThreads`** (для локальной отладки / staging)

- Флаг: `-Djdk.tracePinnedThreads=full` — полный стек-трейс с выделением нативных фреймов и фреймов, удерживающих монитор
- Флаг: `-Djdk.tracePinnedThreads=short` — только проблемные фреймы
- Подходит для быстрого обнаружения в dev/staging; в production создаёт накладные расходы

**Расхождение между источниками.** JDK 21 и JDK 23: оба механизма актуальны. JDK 24+ (JEP 491): `synchronized`-блоки больше не вызывают pinning, поэтому `jdk.tracePinnedThreads` потерял большую часть практической ценности и был удалён. JFR-событие `jdk.VirtualThreadPinned` остаётся актуальным — pinning через нативные вызовы (JNI/FFM → обратный вызов в Java) по-прежнему возможен даже в JDK 24+.

**Практика.** В production-сценариях JFR является предпочтительным: он работает с минимальными накладными расходами, пороговое значение можно настроить в JFR-конфигурации, а само событие активно по умолчанию без дополнительных флагов запуска.

### Вопрос 57. В чём разница между Thread.ofVirtual().start(Runnable), Thread.startVirtualThread(Runnable) и Executors.newVirtualThreadPerTaskExecutor()? Когда предпочтителен каждый из трёх вариантов?

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [JEP 444: Virtual Threads (openjdk.org)](https://openjdk.org/jeps/444)

JEP 444 описывает три способа создания виртуальных потоков.

**`Thread.ofVirtual().start(Runnable)`** — создаёт и запускает виртуальный поток через API построителя (`Thread.Builder`). Построитель позволяет задавать имя потока, `ThreadFactory` с общими свойствами для группы потоков и другие параметры. Созданные таким образом потоки по умолчанию включаются в мониторинг и видны в thread dump.

**`Thread.startVirtualThread(Runnable)`** — удобный метод, эквивалентный `Thread.ofVirtual().start(task)`. Создаёт виртуальный поток и немедленно планирует его выполнение. Добавлен в Java 21 как сокращение для наиболее простых сценариев.

**`Executors.newVirtualThreadPerTaskExecutor()`** — возвращает `ExecutorService`, создающий новый виртуальный поток для каждой отправленной задачи. `ExecutorService` не использует пул; это лёгкий объект, который можно создавать по необходимости. Реализует `AutoCloseable`, что позволяет использовать конструкцию try-with-resources: `close()` дожидается завершения всех задач. Подходит для сценариев fan-out с результирующими значениями (`Future`/`Callable`).

#### Ответ от Claude

**Сравнительная таблица**

| Аспект | `Thread.ofVirtual().start()` | `Thread.startVirtualThread()` | `Executors.newVirtualThreadPerTaskExecutor()` |
|---|---|---|---|
| API-уровень | `Thread.Builder` | статический метод `Thread` | `ExecutorService` |
| Конфигурация | имя, `ThreadFactory`, наследование | нет дополнительных опций | нет дополнительных опций на уровне потока |
| Возвращаемое значение | `Thread` | `Thread` | `ExecutorService` → `Future<T>` |
| `Callable` / результат | нет | нет | да |
| `try-with-resources` | нет | нет | да (AutoCloseable) |
| Типичный сценарий | одиночный поток с именем | одиночный безымянный поток | множество задач, ожидание завершения |

**Когда предпочтителен каждый вариант**

- `Thread.ofVirtual().start()` — нужно задать имя потока (для логирования, JFR, thread dump) или создать `ThreadFactory` для переиспользования конфигурации; управление единственным потоком напрямую.
- `Thread.startVirtualThread()` — максимально лаконичный код; fire-and-forget одиночная задача, где имя и настройки не важны. По сути синтаксический сахар над `Thread.ofVirtual().start()`.
- `Executors.newVirtualThreadPerTaskExecutor()` — нужен `ExecutorService`-контракт: работа с `Future<T>`/`Callable<T>`, `invokeAll`, `invokeAny`; интеграция с фреймворками, принимающими `ExecutorService` (Spring, Jakarta Concurrency); управляемый жизненный цикл через try-with-resources, особенно в fan-out сценариях (параллельный вызов нескольких сервисов и ожидание всех результатов).

**Расхождение между источниками.** `dev.java` отмечает, что `Thread.startVirtualThread()` предназначен «для быстрого демо», а `Executors.newVirtualThreadPerTaskExecutor()` — единственный из трёх, поддерживающий `Callable`. Официальная документация Oracle (`docs.oracle.com`) описывает все три как полноценные production-инструменты; в production-коде выбор определяется требованиями к конфигурации и управлению задачами.

### Вопрос 58. Почему создавать пул виртуальных потоков — антипаттерн? Что именно происходит, если заменить Executors.newFixedThreadPool(N) на пул с виртуальными потоками внутри?

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [JEP 444: Virtual Threads (openjdk.org)](https://openjdk.org/jeps/444)

JEP 444 прямо указывает: «Пулинг виртуальных потоков с целью ограничения конкурентности является антипаттерном, которого следует избегать, несмотря на распространённость такой практики с платформенными потоками». Пулы создавались для совместного использования дорогостоящих ресурсов, а виртуальные потоки не являются дорогостоящим ресурсом и никогда не должны объединяться в пул.

Если разработчики хотят ограничить конкурентность (например, сервис не выдерживает более 20 одновременных запросов), они должны использовать конструкцию, предназначенную именно для этого — класс `Semaphore`:

```java
Semaphore sem = new Semaphore(20);

Result callService() throws InterruptedException {
    sem.acquire();
    try {
        return service.call();
    } finally {
        sem.release();
    }
}
```

Отправка задач в пул потоков ставит их в очередь ожидания свободного потока; семафор внутри создаёт очередь потоков, заблокированных на нём, — структура эквивалентна. Поскольку виртуальные потоки сами по себе являются задачами, итоговая конструкция функционально идентична пулу, но без накладных расходов на пулинг.

#### Ответ от Claude

**Почему пул виртуальных потоков — антипаттерн**

Пул потоков решает две задачи: (1) повторное использование дорогостоящих ресурсов (платформенных потоков = нити ОС), (2) побочный эффект — ограничение конкурентности. Виртуальные потоки уничтожают предпосылку (1): их создание стоит порядка микросекунд, а стек занимает несколько КБ против ~1 МБ у платформенного потока. Следовательно, пул виртуальных потоков решает несуществующую проблему.

**Что конкретно происходит при замене `newFixedThreadPool(N)` пулом виртуальных потоков**

- Если реализовать `ExecutorService` с фиксированным числом виртуальных потоков N, то задачи, превышающие N одновременных слотов, будут ставиться во внутреннюю очередь — абсолютно так же, как и с платформенными потоками. Никакого выигрыша: виртуальные потоки не освобождают carrier при ожидании места в очереди (очередь управляется Java-кодом, а не блокирующим I/O). Пул при этом удерживает N виртуальных потоков занятыми или заблокированными на задачах из очереди.
- Возникает риск утечки данных: `ThreadLocal`-переменные виртуального потока из задачи A могут остаться в потоке, когда он переходит к задаче B (если реализация пула не очищает их между задачами).
- Теряется главное преимущество виртуальных потоков — возможность иметь столько одновременных задач, сколько нужно приложению, без ограничений ресурсами ОС.

**Правильное решение для ограничения конкурентности — `Semaphore`**

```java
// Ограничиваем не потоки, а количество одновременных вызовов внешнего сервиса
Semaphore sem = new Semaphore(20);

Result callExternalService() throws InterruptedException {
    sem.acquire();       // виртуальный поток блокируется (дёшево), carrier освобождается
    try {
        return externalService.call();
    } finally {
        sem.release();
    }
}
```

Семафор позволяет запускать неограниченное число виртуальных потоков (по одному на задачу), но одновременно к защищённому ресурсу обращаются не более N из них. Виртуальный поток, ждущий `sem.acquire()`, размонтируется с carrier-потока — именно для этого и нужны виртуальные потоки.

**Исключение.** Пулинг объектов соединений (JDBC connection pool) — не антипаттерн: там пулируются не потоки, а соединения с БД, которые действительно являются дорогостоящим внешним ресурсом. Пул соединений функционально ведёт себя как семафор, ограничивая конкурентность к БД.

**Расхождений между источниками не обнаружено**: JEP 444, Oracle Core Libraries Guide и inside.java единодушны в оценке пулинга виртуальных потоков как антипаттерна и в рекомендации `Semaphore`.
### Вопрос 59. Как ограничить конкурентность при использовании виртуальных потоков, если нельзя использовать пул с фиксированным размером? Приведите пример ограничения одновременных обращений к внешнему сервису через Semaphore.

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Virtual Threads — Oracle Java 21 Core Libraries Guide](https://docs.oracle.com/en/java/javase/21/core/virtual-threads.html)

Виртуальные потоки не являются дефицитным ресурсом, поэтому их не следует объединять в пул. Пулы потоков были придуманы для управления дорогими платформенными потоками; для виртуальных потоков этот паттерн контрпродуктивен.

Для ограничения конкурентного доступа к внешнему сервису следует использовать `java.util.concurrent.Semaphore`. Семафор инициализируется числом допустимых одновременных обращений; перед вызовом сервиса поток вызывает `acquire()` (блокируется, если разрешений нет), а после — `release()` в блоке `finally`.

```java
Semaphore sem = new Semaphore(10);

Result callService() throws InterruptedException {
    sem.acquire();
    try {
        return limitedExternalService.call();
    } finally {
        sem.release();
    }
}
```

При таком подходе не более 10 виртуальных потоков одновременно обращаются к сервису. Заблокированные потоки не потребляют платформенный поток — виртуальный поток размонтируется с носителя, освобождая его для других задач. Блокировка с виртуальными потоками «дёшева».

Пулы соединений с базой данных сами по себе работают как семафор: пул из 10 соединений заблокирует одиннадцатый поток, пытающийся получить соединение.

#### Ответ от Claude

Виртуальные потоки масштабируются за счёт дешёвой блокировки, а не за счёт ограниченности ресурса. Поэтому правильный инструмент ограничения конкурентности — `Semaphore`, а не `Executors.newFixedThreadPool`.

**Почему не пул?** Пул потоков ограничивает конкурентность косвенно — через нехватку потоков. Для виртуальных потоков создание сотен тысяч потоков бесплатно, и ограничить очередь пулом не получится так же прозрачно. Кроме того, пул скрывает намерение: семафор явно выражает семантику «не более N одновременных операций».

**Паттерн с Semaphore:**

```java
private static final Semaphore LIMITER = new Semaphore(10);

public Result fetchFromExternalService() throws InterruptedException {
    LIMITER.acquire();
    try {
        return externalService.call(); // HTTP-запрос, JDBC и т.п.
    } finally {
        LIMITER.release();
    }
}
```

**Что происходит при вызове `acquire()` без свободных разрешений:**
1. Виртуальный поток блокируется внутри `LockSupport.park`.
2. Среда исполнения размонтирует виртуальный поток с носителя (carrier thread).
3. Носитель немедленно становится доступен для других виртуальных потоков.
4. Когда разрешение освобождается, виртуальный поток ставится в очередь планировщика и монтируется обратно.

Таким образом блокировка на семафоре не удерживает ни одного платформенного потока — именно это отличает виртуальные потоки от платформенных, где заблокированный поток занимает OS-поток целиком.

**Практические рекомендации:**
- Число разрешений должно соответствовать реальной пропускной способности внешнего сервиса или пула соединений, а не числу ядер.
- Если используется JDBC-пул (HikariCP и т.п.), дополнительный семафор не нужен — пул сам является семафором.
- Для продвинутого контроля можно использовать `tryAcquire(timeout, unit)`, чтобы не блокироваться вечно.

Источники: [Oracle Java 21 Core Libraries — Virtual Threads](https://docs.oracle.com/en/java/javase/21/core/virtual-threads.html), [Inside.java Sip #94 — Managing Throughput with Virtual Threads](https://inside.java/2024/02/04/sip094/)

### Вопрос 60. Почему виртуальные потоки не дают выигрыша для CPU-bound задач? Опишите конкретный сценарий, где замена платформенных потоков виртуальными снижает производительность.

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Virtual Threads — Oracle Java 21 Core Libraries Guide](https://docs.oracle.com/en/java/javase/21/core/virtual-threads.html)

Виртуальные потоки предназначены для повышения пропускной способности (throughput), а не скорости (latency). Они не выполняют код быстрее платформенных потоков. Вся их польза возникает из-за дешёвой блокировки: когда виртуальный поток встречает блокирующую операцию ввода-вывода, он размонтируется с носителя, освобождая его для другого виртуального потока.

Для CPU-интенсивных задач это свойство не применимо: поток не блокируется и не размонтируется — он непрерывно занимает носитель. Иметь больше потоков, чем ядер процессора, не способно увеличить пропускную способность в случае CPU-нагруженных задач.

Для распределения вычислительной работы по ядрам следует использовать параллельные потоки (`parallelStream`) или рекурсивные задачи `ForkJoinPool`.

#### Ответ от Claude

**Механизм, объясняющий отсутствие выигрыша**

Виртуальный поток выполняется на платформенном потоке-носителе (carrier thread). По умолчанию число носителей равно числу CPU-ядер (`Runtime.getRuntime().availableProcessors()`). Планировщик виртуальных потоков размонтирует поток с носителя только тогда, когда тот блокируется (I/O, `sleep`, `wait`, `park` и т.п.). Если код непрерывно вычисляет, размонтирования не происходит — носитель занят столько же, сколько занял бы платформенный поток.

Итог: для CPU-bound нагрузки виртуальные потоки === платформенные потоки по потреблению ядер, но добавляют накладные расходы планировщика (монтирование/размонтирование, управление стеком в heap).

**Конкретный сценарий снижения производительности**

Допустим, сервис обрабатывает 1 000 запросов, каждый из которых выполняет ресоёмкое шифрование AES (чистое вычисление, без I/O):

```java
// Платформенный подход: пул из N потоков (N == кол-во ядер)
ExecutorService pool = Executors.newFixedThreadPool(
        Runtime.getRuntime().availableProcessors());

// Виртуальные потоки: создаём 1 000 виртуальных потоков
ExecutorService vt = Executors.newVirtualThreadPerTaskExecutor();
```

При использовании виртуальных потоков:
1. Создаётся 1 000 виртуальных потоков — каждый монтируется на один из N носителей.
2. Ни один из них не блокируется — все удерживают носителей до завершения вычисления.
3. Фактически единовременно работают только N потоков (носителей), остальные ждут в очереди планировщика.
4. Добавляется overhead: аллокация heap-стека на каждый виртуальный поток, дополнительные переключения контекста при ротации в очереди планировщика.

В результате — та же или меньшая пропускная способность, чем у пула платформенных потоков размером N, при большем потреблении памяти.

**Сравнение:**

| Характеристика | Платформенный пул (N потоков) | Виртуальные потоки (1000 шт.) |
|---|---|---|
| Активных носителей | N | N |
| Памяти на потоки | ~1 МБ * N (OS stack) | heap-фрагменты * 1000 |
| Overhead планировщика | минимальный | выше (JVM scheduler) |
| Итоговая пропускная способность | базовая | равна или ниже |

**Правило:** виртуальные потоки дают выигрыш только тогда, когда задача большую часть времени проводит в состоянии ожидания (blocked). Для CPU-bound задач применяйте `ForkJoinPool` или `parallelStream`.
