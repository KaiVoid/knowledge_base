### Вопрос 61. Как получить дамп потоков для виртуальных потоков? Почему виртуальные потоки не отображаются в стандартном thread dump через jstack, и какую команду jcmd нужно использовать вместо него?

**Категория:** Concurrency · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Virtual Threads — Oracle Java 21 Core Libraries Guide](https://docs.oracle.com/en/java/javase/21/core/virtual-threads.html)

Для наблюдения за виртуальными потоками следует создавать дамп потоков командой `jcmd`:

```bash
jcmd <PID> Thread.dump_to_file -format=text <файл>
jcmd <PID> Thread.dump_to_file -format=json <файл>
```

Дамп, создаваемый `jcmd`, перечисляет все потоки, включая как платформенные, так и виртуальные. Однако он не содержит адреса объектов, информацию о блокировках, статистику JNI, статистику кучи и другие сведения, которые присутствуют в традиционных дампах потоков. Поскольку виртуальных потоков может быть очень много, создание нового дампа не приостанавливает приложение.

#### Ответ от Claude

**Почему `jstack` не показывает виртуальные потоки**

`jstack` — исторически инструмент для платформенных потоков (OS-потоков, зарегистрированных в JVM). Виртуальные потоки существуют в heap JVM как объекты (`VirtualThread`), а не как OS-потоки; в момент выполнения каждый виртуальный поток смонтирован на носителе (carrier), но вне этого момента ОС «не знает» о его существовании. По этой причине `jstack`, опирающийся на механизм OS-level thread enumeration, не видит виртуальные потоки — он отображает только активные платформенные потоки.

Кроме того, виртуальных потоков может быть миллионы, и полная остановка приложения для их перечисления была бы неприемлема.

**Команда `jcmd` для дампа виртуальных потоков**

```bash
# Текстовый формат (человекочитаемый)
jcmd <PID> Thread.dump_to_file -format=text /tmp/vt-dump.txt

# JSON-формат (машиночитаемый, рекомендуется для анализа)
jcmd <PID> Thread.dump_to_file -format=json /tmp/vt-dump.json
```

Дамп через `jcmd` включает:
- все платформенные потоки;
- виртуальные потоки, заблокированные на сетевом вводе-выводе;
- виртуальные потоки, созданные через `ExecutorService` (при использовании соответствующих настроек трассировки).

Дамп не включает: адреса объектов, информацию о блокировках (lock ownership), JNI-статистику, статистику heap.

**Важное отличие от традиционного дампа:** генерация дампа через `jcmd` не приостанавливает приложение (no stop-the-world), что критично при большом числе виртуальных потоков.

**Дополнительные инструменты диагностики виртуальных потоков:**

| Инструмент | Назначение |
|---|---|
| `jcmd <PID> Thread.dump_to_file` | Полный дамп с виртуальными потоками |
| JDK Flight Recorder (JFR) | События виртуальных потоков: mount/unmount, pinning |
| JDK Mission Control | GUI-визуализация JFR-данных |
| `-Djdk.tracePinnedThreads=full` | Обнаружение pinning — случаев, когда виртуальный поток удерживает носитель |

**Расхождение источников:** документация Oracle явно называет `jcmd Thread.dump_to_file` рекомендуемым инструментом и не упоминает `jstack` для виртуальных потоков. Исследование Netflix Engineering подтверждает, что `jstack` не показывает стеки виртуальных потоков в Java 21, а `jcmd` — показывает, но без информации о блокировках (это ограничение Java 21, улучшенное в последующих версиях).

Источники: [Oracle Java 21 Core Libraries — Virtual Threads](https://docs.oracle.com/en/java/javase/21/core/virtual-threads.html), [Inside.java Sip #94 — Managing Throughput with Virtual Threads](https://inside.java/2024/02/04/sip094/)
### Вопрос 62. Что изменилось в JEP 491 (JDK 24) относительно pinning на synchronized? Каким образом JVM была доработана так, что виртуальные потоки перестали закрепляться на несущем потоке при входе в synchronized-блок?

**Категория:** Concurrency · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Java Virtual Threads: The Pinning Problem, the Deadlock, and the Fix in Java 24 — shbhmrzd.github.io](https://shbhmrzd.github.io/java/concurrency/virtual-threads/2026/04/25/java-virtual-threads-pinning-and-the-deadlock-problem.html)

До Java 24 JVM отслеживала владельца монитора по идентификатору OS-потока (carrier thread). Когда виртуальный поток входил в `synchronized`-блок, именно несущий поток регистрировался как держатель монитора. Из-за этого виртуальный поток не мог быть размонтирован (unmount): если бы carrier освободился, получилось бы, что монитором владеет поток, который его явно не захватывал.

JEP 491 фундаментально изменил эту архитектуру. Теперь монитор ассоциирован с самим виртуальным потоком, а не с carrier. Это означает: когда виртуальный поток блокируется внутри `synchronized`-блока (например, вызов `socket.read()` или `Thread.sleep()`), он может быть размонтирован. Виртуальный поток «паркуется» в куче, освобождая carrier для других виртуальных потоков. Когда блокирующая операция завершается, виртуальный поток монтируется обратно на любой свободный carrier, сохраняя при этом владение монитором.

#### Ответ от Claude

**Корень проблемы (до JDK 24)**

Реализация `synchronized` в HotSpot хранила владельца объектного монитора как ссылку на OS-поток (platform thread). При входе виртуального потока в `synchronized`-блок его carrier регистрировался как владелец монитора. Размонтирование было запрещено: если бы carrier ушёл под другой виртуальный поток, JVM считала бы, что новый виртуальный поток уже держит монитор — нарушение семантики взаимного исключения.

**Что изменил JEP 491**

JEP 491 («Synchronize Virtual Threads without Pinning», интегрирован в JDK 24) переработал реализацию `synchronized` так, что виртуальные потоки могут захватывать, удерживать и освобождать мониторы независимо от своих carrier'ов. Ключевые изменения:

1. **Смена субъекта владения монитором.** Монитор теперь привязан к виртуальному потоку (`VirtualThread`), а не к OS-потоку. При монтировании и размонтировании JVM выполняет необходимые операции bookkeeping, перенося контекст монитора вместе с виртуальным потоком.

2. **Блокировка при захвате монитора.** Если виртуальный поток не может сразу захватить монитор (он занят), поток размонтируется и carrier освобождается — точно так же, как при блокирующем I/O. Когда монитор освобождается, JVM выбирает ожидающий виртуальный поток и передаёт его в планировщик для монтирования.

3. **`Object.wait()` внутри synchronized.** Вызов `wait()` теперь тоже размонтирует виртуальный поток, не удерживая carrier. При `notify()`/`notifyAll()` виртуальный поток ставится обратно в очередь планировщика и монтируется на свободный carrier.

**Результат**

JEP 491 устраняет практически все случаи pinning, связанные с `synchronized`. Код, который раньше требовал замены `synchronized` на `ReentrantLock` ради масштабируемости виртуальных потоков, теперь работает корректно без изменений. Единственный оставшийся случай pinning — вызовы нативного кода (JNI/FFM), который лежит вне контроля JVM.

### Вопрос 63. Что происходит при вызове нативного метода (JNI/FFM) из виртуального потока? Почему pinning на нативных вызовах не был устранён в JEP 491 и сохраняется до сих пор?

**Категория:** Concurrency · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Java Virtual Threads: The Pinning Problem, the Deadlock, and the Fix in Java 24 — shbhmrzd.github.io](https://shbhmrzd.github.io/java/concurrency/virtual-threads/2026/04/25/java-virtual-threads-pinning-and-the-deadlock-problem.html)

Когда виртуальный поток вызывает нативный метод через JNI или Foreign Function & Memory API, он обязан выполняться на OS-потоке: JVM не может размонтировать виртуальный поток посреди исполнения нативного кода, потому что нативный код может манипулировать thread-local storage или вызывать блокирующие системные вызовы OS.

Нативный код находится вне контроля JVM. Размонтирование в середине его исполнения сломало бы предположения нативных библиотек об идентичности потока и его thread-local состоянии. Это делает данную границу (Java → native) жёстким техническим барьером. JEP 491 явно признаёт это ограничение неустранимым на уровне JVM.

#### Ответ от Claude

**Что происходит при вызове JNI/FFM**

Когда виртуальный поток входит в нативный фрейм (вызов `native`-метода через JNI или Foreign Function из FFM API), JVM принудительно закрепляет (pins) его на текущем carrier. Пока нативный фрейм присутствует в стеке, виртуальный поток не может быть размонтирован. Если внутри нативного кода происходит блокирующая операция — carrier блокируется вместе с ним.

**Фундаментальная причина, по которой JEP 491 не затронул нативный pinning**

JEP 491 смог исправить `synchronized`-pinning, потому что вся соответствующая логика (мониторы объектов) находится под полным контролем JVM. С нативным кодом ситуация принципиально иная:

1. **Thread-local storage (TLS) OS-потока.** Нативный код (C/C++) нередко использует `pthread_getspecific()` / `__thread`-переменные — данные, привязанные к конкретному OS-потоку по его идентификатору. Если JVM размонтирует виртуальный поток с carrier A и смонтирует его на carrier B, нативный код неожиданно окажется в TLS carrier B — это неопределённое поведение.

2. **Непрерывность стека вызовов.** Нативные функции (JNI `CallXxxMethod`, `GetXxxField` и т.д.) хранят JNI environment (`JNIEnv*`), привязанный к конкретному OS-потоку. Переключение carrier'а в середине нативного вызова нарушило бы инварианты `JNIEnv`.

3. **Обратный вызов из нативного кода в Java.** Нативный код может вызывать Java-методы через `CallXxxMethod`. Если в этот момент произошла бы смена carrier, JVM-фреймы на стеке OS-потока оказались бы некорректными.

4. **Блокирующие syscall'ы.** Нативный код может напрямую вызывать `read()`, `write()`, `mutex_lock()` и другие блокирующие вызовы ядра. JVM об этих блокировках ничего не знает и не может перехватить момент для размонтирования.

**Практические последствия**

- При вызове нативного метода из виртуального потока carrier блокируется на время нативного вызова. Планировщик JVM может создать дополнительный carrier (компенсационный поток), чтобы не остановить всю работу, но это увеличивает число OS-потоков.
- Если нативный код вызывает назад Java-код (`JNI upcall`), и этот код блокируется на мониторе или I/O — виртуальный поток остаётся заколотым, JVM генерирует событие `jdk.VirtualThreadPinned` в JFR.
- Длинные нативные вызовы в критическом пути запроса сводят на нет масштабируемость виртуальных потоков. Рекомендация: выносить длительные нативные операции в отдельный пул OS-потоков и взаимодействовать с ними асинхронно.

**Расхождение источников.** Официальная документация Oracle по JDK 21 (до JEP 491) называла два случая pinning: `synchronized` и нативные фреймы. После JEP 491 первый случай устранён; нативный pinning остаётся единственным задокументированным случаем. Некоторые вторичные источники (InfoQ) упоминают также pinning при инициализации классов — это узкий и редкий случай, не выделяемый в отдельную категорию в официальной документации JDK 24.

### Вопрос 64. Чем модель «один виртуальный поток на запрос» (thread-per-request) отличается от реактивного подхода (Project Reactor, RxJava) с точки зрения модели программирования, отладки и производительности? В каких случаях реактивный подход всё ещё предпочтительнее?

**Категория:** Concurrency · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Virtual Threads — Oracle Java SE 21 Core Libraries](https://docs.oracle.com/en/java/javase/21/core/virtual-threads.html)

В модели thread-per-request сервер выделяет поток на каждый входящий запрос на всё время его обработки. Это позволяет писать простой, последовательный, блокирующий код: поток читает запрос, обрабатывает его (возможно, выполняя блокирующие операции) и пишет ответ — всё в одном потоке. Такой код легко отлаживать в дебаггере, профилировать в профилировщике и наблюдать через thread dump.

Асинхронный подход не даёт значительных преимуществ виртуальным потокам. Пример: код, написанный в неблокирующем стиле с `CompletableFuture`, не получит выигрыша от виртуальных потоков. Oracle прямо рекомендует: «Избегайте смешения синхронного блокирующего кода с асинхронными фреймворками». Блокирование виртуального потока является дешёвым и поощряется, потому что виртуальных потоков в системе может быть очень много.

#### Ответ от Claude

**Модель программирования**

| Аспект | Thread-per-request (виртуальные потоки) | Реактивный подход (Reactor, RxJava) |
|---|---|---|
| Стиль кода | Синхронный, императивный, последовательный | Асинхронный, декларативный, цепочки операторов |
| Семантика блокировки | Блокирование дёшево — поток паркуется в куче | Блокирование запрещено — нарушает event loop |
| Абстракция | `java.lang.Thread`, привычная Java-модель | `Mono<T>`, `Flux<T>`, специфичный API |
| Управление ошибками | try/catch/finally | `onError`, `doOnError`, операторы восстановления |
| Управление потоком данных | Нет встроенного backpressure | Backpressure через Reactive Streams |

**Отладка**

*Виртуальные потоки.* Стек вызовов — это обычный синхронный стек Java. Дебаггер показывает полный стек, thread dump содержит осмысленный trace. Исключения выбрасываются там, где ожидается, а не в колбэках. Это огромное преимущество по сравнению с реактивным кодом.

*Project Reactor.* Стек вызовов по умолчанию содержит десятки операторных фреймов Reactor, скрывающих реальный код приложения. Reactor предоставляет `Hooks.onOperatorDebug()` и Java Agent (`ReactorDebugAgent`) для добавления checkpoint-информации, но это усложняет конфигурацию и снижает производительность. Трассировка в распределённой системе (Micrometer) требует явного проброса контекста через `Context` API.

**Производительность**

*Виртуальные потоки* выигрывают при I/O-bound нагрузке с высоким количеством одновременных запросов: каждый заблокированный поток не удерживает OS-поток, позволяя запускать десятки тысяч параллельных операций на небольшом пуле carrier'ов. CPU-bound задачи не выигрывают — количество реально выполняющихся потоков ограничено числом CPU.

*Реактивный подход* имеет более низкие накладные расходы на переключение контекста, поскольку event loop никогда не блокируется. Бенчмарки показывают, что Spring WebFlux при высоких нагрузках (800+ concurrent users) стабильно выигрывает у виртуальных потоков по пропускной способности (+12–62 rps) и задержке. Реактивный подход также позволяет точно управлять планировщиками (`Schedulers.boundedElastic()`, `Schedulers.parallel()`).

**Когда реактивный подход по-прежнему предпочтительнее**

1. **Backpressure и потоковая обработка.** Если источник данных может генерировать данные быстрее, чем потребитель их обрабатывает, Reactive Streams предоставляют стандартный механизм backpressure (`Publisher/Subscriber`). Виртуальные потоки не имеют встроенного аналога.

2. **Streaming-ответы и SSE/WebSocket.** `Flux<T>` естественно моделирует бесконечные или длинные потоки данных, WebFlux поддерживает Server-Sent Events и реактивные WebSocket из коробки.

3. **Существующий реактивный стек.** Если команда уже владеет реактивным подходом, а приложение построено на WebFlux + R2DBC + реактивных Redis/MongoDB-клиентах, переход на виртуальные потоки не даст преимуществ и потребует рефакторинга.

4. **Максимальная пропускная способность при предсказуемой нагрузке.** Если профилирование показывает, что реактивный event loop даёт лучший throughput и latency при конкретном профиле нагрузки — это веское основание его сохранить.

5. **Комбинирование потоков.** Операторы `flatMap`, `zip`, `merge`, `switchMap` — мощный инструментарий для оркестрации нескольких асинхронных источников без явного управления потоками.

**Расхождение источников.** Oracle (docs.oracle.com) рекомендует виртуальные потоки как основную модель и советует избегать смешения с асинхронными фреймворками. Baeldung и vincenzoracca.com приводят бенчмарки, где WebFlux превосходит виртуальные потоки по throughput при высоких нагрузках. Это расхождение объясняется тем, что Oracle описывает модель программирования и упрощение кода, тогда как бенчмарки сосредоточены на предельной пропускной способности. На практике выбор определяется профилем задачи: для большинства I/O-bound сервисов виртуальные потоки достаточны; для максимального throughput и работы с потоками данных реактивный подход сохраняет преимущества.

### Вопрос 65. Что произойдёт, если виртуальный поток выполняет блокирующую операцию, а все carrier threads в ForkJoinPool-планировщике уже заняты (например, из-за pinning)? Как планировщик компенсирует нехватку потоков-носителей?

**Категория:** Concurrency · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [JEP 444: Virtual Threads — openjdk.org](https://openjdk.org/jeps/444)

Планировщик виртуальных потоков — это `ForkJoinPool`, работающий в режиме FIFO, а его параллелизм по умолчанию равен числу доступных процессоров.

**Обычные блокирующие операции (не pinning):** Если виртуальный поток блокируется в операции, которую JDK умеет обрабатывать (например, `Object.wait()`, файловый I/O), но при этом не может быть демонтирован из-за ограничений уровня JDK или ОС, реализация такой операции _компенсирует_ захват OS-потока, временно расширяя параллелизм планировщика. В результате число платформенных потоков в `ForkJoinPool` планировщика может временно превысить число доступных процессоров. Максимальное количество платформенных потоков, доступных планировщику, регулируется системным свойством `jdk.virtualThreadScheduler.maxPoolSize` (по умолчанию — 256).

**Pinning (synchronized / native):** Когда виртуальный поток исполняет код внутри блока или метода `synchronized` (или вызывает нативный метод), он _прикрепляется_ к своему carrier thread и не может быть с него демонтирован. **Для pinning планировщик НЕ компенсирует нехватку, расширяя параллелизм.** Увеличение параллелизма само по себе не решает проблему: если достаточно много виртуальных потоков заблокировано внутри `synchronized`, никакое значение параллелизма не поможет.

**Сценарий исчерпания (scheduler starvation / deadlock):** Если N виртуальных потоков (где N равно числу carrier threads) прикреплены к своим носителям и ждут чужой виртуальный поток, который сам не может быть запущен — система зависает. Это не классический deadlock (ни один поток не держит замок, которого ждёт другой), а именно scheduler starvation: все носители захвачены, прогресс невозможен.

**Рекомендация:** Заменять `synchronized`-блоки, защищающие потенциально долгие I/O-операции, на `java.util.concurrent.locks.ReentrantLock`. Начиная с Java 24 (JEP 491) большинство случаев pinning на мониторах объектов устранено на уровне JVM.

#### Ответ от Claude

**Механизм компенсации — два разных случая:**

Планировщик виртуальных потоков (`ForkJoinPool` в FIFO-режиме) применяет разные стратегии в зависимости от природы блокировки.

1. **Блокирующие операции без pinning** (например, `LockSupport.park()`, `Object.wait()`, ряд I/O-операций уровня JDK). Когда виртуальный поток не может быть демонтирован из-за OS- или JDK-ограничений, JDK-реализация операции запрашивает у `ForkJoinPool` временное расширение параллелизма: в пул добавляется дополнительный carrier thread. Число carrier-потоков может превысить `Runtime.availableProcessors()`. Потолок задаётся свойством `jdk.virtualThreadScheduler.maxPoolSize` (по умолчанию 256).

2. **Pinning** (код внутри `synchronized` или нативный вызов). Виртуальный поток намертво прикреплён к своему carrier: он не может демонтироваться, и **планировщик не запускает компенсирующий поток**. Обоснование из JEP 444: компенсационное увеличение числа потоков не масштабируется — если миллион виртуальных потоков заходит в `synchronized`, никакое число carrier threads не спасёт.

**Сценарий полного исчерпания (scheduler starvation):**

Пусть CPU = 4, значит pool содержит 4 carrier thread. Виртуальные потоки VT-1..VT-4 входят в `synchronized`-блок, делают медленный I/O и прикрепляются к своим носителям CT-1..CT-4. Когда поступают VT-5, VT-6 и т. д., ни один carrier thread не доступен для их монтирования. Система не падает с исключением — она зависает: `ForkJoinPool` не создаст новый поток, потому что pinning-компенсации нет.

**Ключевые отличия от обычного thread pool exhaustion:**

- В классическом пуле поток просто не вернётся в пул — задача встанет в очередь и будет выполнена после освобождения потока.
- При pinning-starvation carrier-потоки физически заняты навсегда (пока pinned VT не завершит блокирующую операцию), и ни один из ожидающих виртуальных потоков не получит carrier, даже если среди них есть поток, выход которого освободил бы заблокированный synchronized-блок. Это может привести к полному зависанию приложения.

**Решения:**

- Заменить `synchronized` на `ReentrantLock` для секций, содержащих долгие I/O-операции.
- Мигрировать на Java 24+, где JEP 491 устраняет pinning для мониторов объектов.
- Мониторинг: JFR-событие `jdk.VirtualThreadPinned` и флаг JVM `-Djdk.tracePinnedThreads=full` помогают обнаружить problematic pinning в продакшне.

### Вопрос 66. Виртуальный поток демонтирован с carrier thread и ждёт завершения I/O. Может ли он при возобновлении быть смонтирован на другой carrier thread (не тот, с которого был снят)? Какие из этого следуют ограничения на использование ThreadLocal vs ScopedValue?

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Virtual Threads — Oracle Java SE 21 Core Libraries](https://docs.oracle.com/en/java/javase/21/core/virtual-threads.html)

После демонтажа виртуального потока с carrier thread носитель освобождается: «After a virtual thread unmounts from its carrier, the carrier is free, which means that the Java runtime scheduler can mount a different virtual thread on it.» Код, исполняемый в виртуальном потоке, не осознаёт, на каком именно carrier thread он работает.

Когда блокирующая операция завершается, планировщик повторно монтирует виртуальный поток на _какой-либо_ доступный carrier thread — это может быть как тот же самый, так и любой другой. Закрепление за конкретным OS-потоком не гарантируется.

Относительно `ThreadLocal`: виртуальные потоки полностью поддерживают `ThreadLocal` и `InheritableThreadLocal`, так же как платформенные потоки. Данные `ThreadLocal` привязаны к виртуальному потоку (а не к carrier), поэтому они корректно переносятся при смене носителя. Однако «consider using the safer and more efficient scoped values» — `ScopedValue` рекомендован как более безопасная и эффективная альтернатива для случаев «однонаправленной передачи» данных без использования параметров методов.

#### Ответ от Claude

**Может ли виртуальный поток возобновиться на другом carrier thread?**

Да, это штатное поведение. Виртуальный поток не привязан к конкретному OS-потоку. После того как блокирующая операция (например, ожидание сокета) завершается, JDK помещает виртуальный поток обратно в очередь планировщика (`ForkJoinPool`), и тот монтирует его на _любой_ свободный carrier thread. В большинстве случаев это будет другой физический поток.

**Что из этого следует для `ThreadLocal`:**

`ThreadLocal`-переменные хранятся в `Thread.threadLocals` — поле самого объекта `Thread`. Поскольку виртуальный поток — это полноценный `Thread`-объект (а не carrier), его `ThreadLocal`-карта путешествует вместе с ним. При смене carrier данные не теряются и не смешиваются с данными другого виртуального потока. В этом смысле `ThreadLocal` работает корректно.

Однако есть тонкости:

- **Нет identity-гарантии по carrier.** Паттерны, завязанные на `Thread.currentThread()` возвращающем _конкретный_ платформенный поток (например, привязка к CPU-ядру, off-heap буферам, специфичным ресурсам carrier-потока), с виртуальными потоками ненадёжны.
- **`InheritableThreadLocal` и structured concurrency.** При использовании `StructuredTaskScope.fork()` дочерние виртуальные потоки наследуют `InheritableThreadLocal`-значения через копирование. Это создаёт накладные расходы и потенциальные сюрпризы при мутации значений из дочернего потока.
- **Масштаб.** С миллионами виртуальных потоков каждый несёт свою копию `ThreadLocal`-карты. При хранении тяжёлых объектов это ведёт к росту памяти (подробнее — в вопросе 15).

**Почему `ScopedValue` предпочтительнее:**

`ScopedValue` (финализирован в Java 23, JEP 446/506) решает конкретные проблемы виртуальных потоков:

| Свойство | `ThreadLocal` | `ScopedValue` |
|---|---|---|
| Область видимости | На протяжении всей жизни потока | Ограничена явно заданным scope (`ScopedValue.where(...).run(...)`) |
| Изменяемость | Мутируемый (set/get) | Иммутабельный в пределах scope |
| Наследование дочерними потоками | Копирование всей карты | Дочерние потоки разделяют биндинг родителя без копирования |
| Очистка | Требует явного `remove()`, иначе утечка | Автоматически по выходу из scope |
| Производительность при miллионах потоков | Каждый поток — своя копия карты | Общая структура без копирования |

Таким образом, когда нужно передать контекст (userId, traceId, tenant) в методы внутри одного виртуального потока или его дочерних потоков — `ScopedValue` является рекомендованным решением. `ThreadLocal` остаётся допустимым для накопления состояния, специфичного для конкретного потока (например, буферов), — при условии, что объекты лёгкие и их количество экземпляров предсказуемо.

### Вопрос 67. Почему использование тяжёлых объектов в ThreadLocal при миллионах виртуальных потоков может привести к нехватке памяти, даже если утечки в классическом смысле нет? Чем это отличается от аналогичной ситуации с пулом из 200 платформенных потоков?

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Virtual Threads — Oracle Java SE 21 Core Libraries](https://docs.oracle.com/en/java/javase/21/core/virtual-threads.html)

Официальная документация Oracle описывает паттерн-антипаттерн:

Кэширование в `ThreadLocal` полезно только тогда, когда поток — и, следовательно, дорогостоящий объект, хранящийся в thread-local, — совместно используется и повторно применяется несколькими задачами, как это происходит при пуле платформенных потоков. Многие задачи могут вызывать `foo` в пуле потоков, но поскольку пул содержит лишь несколько потоков, объект будет создан всего несколько раз — по одному на поток пула — и затем кэширован и повторно использован.

Однако **виртуальные потоки никогда не помещаются в пул и никогда не переиспользуются для несвязанных задач**. Поскольку у каждой задачи есть свой виртуальный поток, каждый вызов `foo` из разной задачи будет инициировать создание нового `SimpleDateFormat`. Более того, поскольку одновременно может выполняться очень много виртуальных потоков, дорогостоящий объект может потреблять значительный объём памяти. Эти результаты — прямая противоположность тому, чего стремится достичь кэширование в thread-local.

Также отмечается: «The JDK removed many uses of thread locals from the java.base module in preparation for virtual threads in order to reduce memory footprint when running with millions of threads.»

#### Ответ от Claude

**Суть проблемы — разница в жизненном цикле и масштабе:**

**Пул из 200 платформенных потоков:**

Платформенные потоки создаются один раз, живут всё время работы приложения и обслуживают задачи последовательно. Если в `ThreadLocal` хранится тяжёлый объект (например, `SimpleDateFormat`, `Calendar`, `Connection` к внешнему сервису, большой буфер), он создаётся ровно 200 раз — по одному на поток — и затем переиспользуется всеми задачами, которые попадают на этот поток. Суммарное потребление памяти: `200 × size(object)`. Утечки нет, поскольку `ThreadLocalMap` очищается вместе с потоком при завершении приложения или явно через `remove()`.

**Миллион виртуальных потоков:**

Каждый виртуальный поток — это отдельная задача со своим собственным объектом `Thread`. При первом вызове `cachedFormatter.get()` создаётся новый экземпляр дорогостоящего объекта и сохраняется в `ThreadLocalMap` данного виртуального потока. Поскольку виртуальные потоки не переиспользуются (каждый создаётся под одну задачу), кэш работает в лучшем случае один раз за всю жизнь потока.

Если в системе одновременно существует 1 000 000 виртуальных потоков, каждый несёт свой экземпляр объекта: суммарное потребление `1 000 000 × size(object)`. При объекте размером всего 1 КБ — это уже 1 ГБ heap только на эти объекты. При 10 КБ — 10 ГБ.

**Почему это не «классическая утечка»:**

Классическая утечка `ThreadLocal` — это когда `ThreadLocal`-значение никогда не вызывает `remove()`, и поскольку поток не завершается (пул), объект живёт в heap вечно. Здесь иначе: каждый виртуальный поток завершается после задачи и будет GC-собран вместе с `ThreadLocalMap`. Утечки нет в том смысле, что памяти не накапливается _навсегда_. Проблема — в _одновременном_ присутствии миллионов экземпляров в heap в момент пиковой нагрузки. GC не успевает — или не может — собрать их раньше, чем придут новые задачи.

**Итог сравнения:**

| Параметр | 200 платформенных потоков | 1 000 000 виртуальных потоков |
|---|---|---|
| Число экземпляров объекта | 200 (константа) | До 1 000 000 (по числу concurrent VT) |
| Переиспользование | Да, многократное | Нет (1 поток = 1 задача) |
| Риск OOM | Минимален | Высокий при тяжёлых объектах |
| Характер проблемы | Нет проблемы | Пиковое потребление heap |

**Рекомендации:**

1. Не хранить тяжёлые или мутируемые кэшируемые объекты в `ThreadLocal` при использовании виртуальных потоков.
2. Заменить мутируемые объекты иммутабельными разделяемыми аналогами, где возможно (например, `SimpleDateFormat` → `DateTimeFormatter`).
3. Для передачи контекста использовать `ScopedValue` — он не создаёт копии при наследовании и автоматически освобождается по выходу из scope.
4. Помнить, что асинхронные фреймворки (Netty, Reactor) также могут кэшировать объекты в `ThreadLocal`, рассчитывая на маленький пул потоков: смешение таких фреймворков с виртуальными потоками воспроизводит ту же проблему.

### Вопрос 68. Какие события JDK Flight Recorder (JFR) были добавлены специально для виртуальных потоков? Что фиксирует событие jdk.VirtualThreadPinned и при каком условии оно эмитируется по умолчанию?

**Категория:** Concurrency · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Virtual Threads — Oracle Java 21 Core Libraries](https://docs.oracle.com/en/java/javase/21/core/virtual-threads.html)

В JDK 21 (JEP 444) добавлены четыре новых JFR-события для виртуальных потоков:

- **jdk.VirtualThreadStart** — фиксирует запуск виртуального потока. По умолчанию отключено.
- **jdk.VirtualThreadEnd** — фиксирует завершение виртуального потока. По умолчанию отключено.
- **jdk.VirtualThreadPinned** — фиксирует ситуацию, когда виртуальный поток заблокировался, оставаясь «прикреплённым» (pinned) к несущему потоку, то есть не освободил его. По умолчанию **включено** с порогом 20 мс: событие эмитируется только если прикрепление длилось дольше этого порога.
- **jdk.VirtualThreadSubmitFailed** — фиксирует ошибку при попытке запустить или «распарковать» виртуальный поток (обычно из-за нехватки ресурсов). По умолчанию **включено**.

Просмотр событий:
```bash
jfr print --events jdk.VirtualThreadPinned,jdk.VirtualThreadSubmitFailed recording.jfr
```

Дополнительно для трассировки прикреплений используется системное свойство:
```
-Djdk.tracePinnedThreads=full   # полный стек-трейс с нативными фреймами
-Djdk.tracePinnedThreads=short  # только проблемные фреймы
```

#### Ответ от Claude

В JDK 21 (JEP 444) для виртуальных потоков добавлены четыре JFR-события:

| Событие | Что фиксирует | По умолчанию |
|---|---|---|
| `jdk.VirtualThreadStart` | Запуск виртуального потока | Отключено |
| `jdk.VirtualThreadEnd` | Завершение виртуального потока | Отключено |
| `jdk.VirtualThreadPinned` | Виртуальный поток заблокировался, не освободив несущий поток | **Включено**, порог 20 мс |
| `jdk.VirtualThreadSubmitFailed` | Не удалось запустить или распарковать виртуальный поток | **Включено** |

**jdk.VirtualThreadPinned — подробнее.**

Виртуальный поток называется «прикреплённым» (pinned), когда он не может демонтироваться с несущего потока (carrier thread) при блокировке. Это происходит в двух ситуациях:
1. Виртуальный поток выполняет код внутри блока `synchronized` или `synchronized`-метода.
2. Виртуальный поток вызывает нативный метод или функцию Foreign Function & Memory API.

В таком состоянии несущий OS-поток остаётся занятым на всё время блокировки, что снижает пропускную способность.

Событие эмитируется по умолчанию, но только если прикрепление длилось **более 20 мс** — это позволяет избежать накладных расходов на короткие прикрепления. Порог настраивается через конфигурацию JFR-записи.

**Расхождение между источниками:** Документ Oracle Java 21 и inside.java/sip094 по-разному описывают, какие события включены по умолчанию. Согласно docs.oracle.com/javase/21/core/virtual-threads.html, `VirtualThreadStart` и `VirtualThreadEnd` отключены, а `VirtualThreadPinned` и `VirtualThreadSubmitFailed` включены — это согласуется с JEP 444.

**Примечание о JEP 491 (JDK 24+):** Начиная с JDK 24 (JEP 491) виртуальные потоки перестали прикрепляться при входе в `synchronized`-блоки — теперь они корректно демонтируются и там. Таким образом, `jdk.VirtualThreadPinned` в JDK 24+ будет срабатывать значительно реже: преимущественно только при вызове нативных методов.

### Вопрос 69. Чем виртуальный поток (virtual thread) отличается от платформенного (platform thread) на уровне JVM? Как соотносятся виртуальные потоки, несущие потоки (carrier threads) и пул ForkJoinPool?

**Категория:** Concurrency · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Virtual Threads — Oracle Java 21 Core Libraries](https://docs.oracle.com/en/java/javase/21/core/virtual-threads.html)

**Платформенный поток** — тонкая обёртка над OS-потоком. Он работает на своём OS-потоке на протяжении всего жизненного цикла: OS-поток захвачен платформенным потоком с момента создания до завершения. Планирование выполняется операционной системой. Число платформенных потоков ограничено числом доступных OS-потоков.

**Виртуальный поток** реализован непосредственно JDK, а не операционной системой. JDK имеет собственный планировщик для виртуальных потоков. Стеки виртуальных потоков хранятся в куче Java (GC-managed heap) в виде объектов stack chunk, которые растут и уменьшаются по мере выполнения. Виртуальный поток не привязан к конкретному OS-потоку на всё время жизни.

**Несущий поток (carrier thread)** — платформенный поток, на который виртуальный поток монтируется для выполнения. Планировщик JDK монтирует виртуальный поток на несущий поток, после чего OS планирует уже этот несущий поток в обычном режиме. Когда виртуальный поток выполняет блокирующую операцию, он демонтируется с несущего потока, освобождая его для другого виртуального потока.

**ForkJoinPool как планировщик.** Планировщик виртуальных потоков по умолчанию — это специальный `ForkJoinPool`, работающий в режиме FIFO. Степень параллелизма по умолчанию равна числу доступных процессоров. Этот пул отличается от «общего» (`commonPool`), используемого в параллельных стримах. Настраивается через:
- `jdk.virtualThreadScheduler.parallelism` — целевое число несущих потоков
- `jdk.virtualThreadScheduler.maxPoolSize` — максимальное число несущих потоков

#### Ответ от Claude

**Платформенный поток vs виртуальный поток: ключевые отличия на уровне JVM**

| Аспект | Платформенный поток | Виртуальный поток |
|---|---|---|
| Реализация | Тонкая обёртка над OS-потоком | Реализован JDK, не OS |
| Планировщик | Операционная система | JVM (ForkJoinPool) |
| Хранение стека | В памяти, управляемой OS | В Java-куче (объекты stack chunk) |
| Привязка к OS-потоку | На весь жизненный цикл | Только во время выполнения (монтирование) |
| Масштаб | Тысячи (ограничены OS) | Миллионы на один JVM |
| Накладные расходы | Высокие (~1 МБ стек) | Минимальные (стек растёт/сжимается) |

**Внутреннее устройство виртуального потока.**

В основе виртуального потока лежит объект **Continuation** (Project Loom). Continuation инкапсулирует выполнение метода `run()` вместе с полным стеком вызовов. При демонтировании стек сериализуется в кучу; при монтировании — восстанавливается в стек несущего потока. Это позволяет одному OS-потоку последовательно обслуживать миллионы виртуальных потоков.

**Соотношение: виртуальные потоки, несущие потоки, ForkJoinPool**

```
Виртуальные потоки (N, могут быть миллионы)
    |   монтируются на
    v
Несущие потоки = ForkJoinPool-worker-1, ..., ForkJoinPool-worker-K
    (K = число CPU-ядер по умолчанию)
    |   каждый несущий поток — это
    v
Платформенный поток → OS-поток → CPU-ядро
```

Планировщик — это `ForkJoinPool` в режиме **FIFO** (асинхронный режим). В отличие от стандартного `commonPool`, используемого в параллельных стримах, этот пул создаётся отдельно и предназначен исключительно для виртуальных потоков. Work-stealing позволяет эффективно перераспределять задачи между несущими потоками.

Параллелизм пула по умолчанию равен числу доступных процессоров (`Runtime.getRuntime().availableProcessors()`). При прикреплении (pinning) планировщик может временно расширять пул сверх этого числа, добавляя компенсирующие потоки, чтобы не допустить голодания.

### Вопрос 70. Что такое CopyOnWriteArrayList? Как работает механизм copy-on-write, почему итераторы этой коллекции никогда не бросают ConcurrentModificationException, и в каких сценариях она предпочтительна?

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [CopyOnWriteArrayList (Java SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/CopyOnWriteArrayList.html)

Потокобезопасный вариант `ArrayList`, в котором все мутирующие операции (`add`, `set` и т.д.) реализованы путём создания свежей копии лежащего в основе массива.

Обычно это слишком затратно, но может быть эффективнее альтернатив, когда операции обхода (traversal) значительно превышают по количеству операции мутации, и полезно тогда, когда нельзя или не хочется синхронизировать обходы, но при этом необходимо исключить взаимные помехи между параллельными потоками. Метод-итератор «snapshot» использует ссылку на состояние массива на момент создания итератора. Этот массив никогда не изменяется за всё время жизни итератора, поэтому помех быть не может, и итератор гарантированно не бросает `ConcurrentModificationException`.

#### Ответ от Claude

**Механизм copy-on-write:**

При любой мутирующей операции (`add`, `set`, `remove`) `CopyOnWriteArrayList`:
1. Захватывает внутренний замок (`ReentrantLock`).
2. Создаёт новый массив, который является копией текущего (с применением изменения).
3. Атомарно заменяет ссылку на внутренний массив на новую копию.
4. Освобождает замок.

Читатели всегда работают с той ссылкой на массив, которую они получили в начале операции — снимок (snapshot). Пока писатель создаёт новую копию, читатели продолжают видеть старый массив. После атомарной замены новые читатели увидят новый массив. Старые итераторы продолжат работу с исходным снимком и никогда не увидят изменений.

**Почему итераторы не бросают `ConcurrentModificationException`:**

В `ArrayList` итератор хранит `modCount` и при каждой операции проверяет его. В `CopyOnWriteArrayList` итератор держит ссылку на конкретный массив-снимок, сделанный при создании итератора. Этот массив иммутабелен — никакие операции записи на него не влияют. Проверять `modCount` нет смысла: массив изменить нельзя физически.

**Сравнение с synchronized ArrayList:**

| Аспект | `CopyOnWriteArrayList` | `Collections.synchronizedList(ArrayList)` |
|---|---|---|
| Чтение | Без блокировки | Блокирующее (на всём объекте) |
| Запись | Создаёт копию массива | Блокирующее |
| Итерация | Snapshot, без блокировки | Требует ручной синхронизации итератора |
| `ConcurrentModificationException` | Никогда | Возможен без синхронизации итератора |
| Потребление памяти | Высокое при частых записях | Умеренное |

**Когда предпочтительна:**

- Списки событийных слушателей (listeners/observers): регистрация/удаление редки, уведомления частые.
- Кэши и конфигурации: запись редкая, чтение очень частое.
- Ситуации, когда `ConcurrentModificationException` критичен, а запись допустимо замедлить.

**Когда не подходит:** при частых операциях записи (каждый `add` — новый массив), при больших коллекциях (копирование 1 000 000 элементов на каждое добавление), при необходимости видеть свежие изменения в текущем итераторе.

### Вопрос 71. Какие встроенные политики отклонения задач (RejectedExecutionHandler) предоставляет ThreadPoolExecutor? Опишите поведение каждой политики и поясните, как реализовать собственный обработчик отклонения.

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ThreadPoolExecutor (Java SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/ThreadPoolExecutor.html)

Предусмотрены четыре предопределённые политики отклонения задач:

- **`AbortPolicy`** — обработчик по умолчанию, выбрасывает `RejectedExecutionException` при отклонении.
- **`CallerRunsPolicy`** — выполняет отклонённую задачу в потоке вызывающего метода `execute()`, если исполнитель не завершён. Это обеспечивает простой механизм обратного давления (back pressure), замедляющий скорость подачи новых задач.
- **`DiscardPolicy`** — задача, которую не удалось выполнить, просто отбрасывается.
- **`DiscardOldestPolicy`** — если исполнитель не завершён, отбрасывается задача в начале очереди (самая старая ожидающая), затем повторяется попытка выполнения текущей задачи.

#### Ответ от Claude

**Четыре встроенных политики:**

| Политика | Поведение | Когда применять |
|---|---|---|
| `AbortPolicy` (по умолчанию) | Выбрасывает `RejectedExecutionException` | Когда отклонение недопустимо и должно быть явно обработано |
| `CallerRunsPolicy` | Задача выполняется в потоке вызывающего | Back pressure: замедлить отправителя, ничего не терять |
| `DiscardPolicy` | Молча отбрасывает задачу | Несрочные задачи, потеря которых допустима (метрики, логи) |
| `DiscardOldestPolicy` | Удаляет самую старую задачу из очереди и повторяет попытку | Когда свежие данные важнее устаревших (например, обновление UI) |

**Когда задача отклоняется:**

`RejectedExecutionHandler.rejectedExecution()` вызывается, когда:
1. Пул достиг `maximumPoolSize` **и** очередь заполнена.
2. Вызван метод `shutdown()` или `shutdownNow()`, а задачи продолжают поступать.

**Пример настройки:**

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    2,                              // corePoolSize
    4,                              // maximumPoolSize
    60, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(10),   // ограниченная очередь
    new ThreadPoolExecutor.CallerRunsPolicy() // back pressure
);
```

**Собственный `RejectedExecutionHandler`:**

```java
executor.setRejectedExecutionHandler((task, pool) -> {
    if (!pool.isShutdown()) {
        log.warn("Задача отклонена, ждём места в очереди: {}", task);
        try {
            // Блокируемся до появления места (осторожно: может заблокировать вызывающий поток)
            pool.getQueue().put(task);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new RejectedExecutionException("Прерван при ожидании очереди", e);
        }
    }
});
```

**Расхождение источников:** Baeldung (`baeldung.com/java-rejectedexecutionhandler`) подробно разбирает каждую политику и подчёркивает, что `CallerRunsPolicy` обеспечивает backpressure без потерь, но за счёт блокировки вызывающего потока. Oracle Javadoc фиксирует контракт: `rejectedExecution` вызывается только при невозможности принять задачу — после исчерпания и очереди, и пула.

### Вопрос 72. Какие состояния (Thread.State) может иметь поток в Java? Опишите каждое из шести состояний, переходы между ними и чем BLOCKED отличается от WAITING и TIMED_WAITING.

**Категория:** Concurrency · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Thread.State (Java SE 17 & JDK 17) — docs.oracle.com](https://docs.oracle.com/en/java/javase/17/docs/api/java.base/java/lang/Thread.State.html)

Поток может находиться только в одном из следующих состояний:

- **NEW** — поток создан, но `start()` ещё не вызван.
- **RUNNABLE** — поток выполняется в JVM; при этом он может ожидать ресурс операционной системы (например, процессорное время).
- **BLOCKED** — поток заблокирован в ожидании монитора для входа в синхронизированный блок или метод, либо повторного входа в него после вызова `Object.wait`.
- **WAITING** — поток ждёт неопределённо долго, пока другой поток выполнит определённое действие. Переход в это состояние происходит при вызове `Object.wait()` без тайм-аута, `Thread.join()` без тайм-аута или `LockSupport.park()`.
- **TIMED_WAITING** — поток ждёт в течение заданного времени. Переход происходит при вызове `Thread.sleep`, `Object.wait(timeout)`, `Thread.join(timeout)`, `LockSupport.parkNanos`, `LockSupport.parkUntil`.
- **TERMINATED** — поток завершил выполнение.

#### Ответ от Claude

**Шесть состояний потока Java:**

| Состояние | Когда | Методы-переходы |
|---|---|---|
| `NEW` | После `new Thread()`, до `start()` | — |
| `RUNNABLE` | После `start()`, выполняется или ждёт ЦП | `start()` |
| `BLOCKED` | Ждёт монитор (`synchronized`) | Вход в `synchronized`-блок |
| `WAITING` | Ждёт бесконечно уведомления | `wait()`, `join()`, `LockSupport.park()` |
| `TIMED_WAITING` | Ждёт с тайм-аутом | `sleep(ms)`, `wait(ms)`, `join(ms)`, `parkNanos` |
| `TERMINATED` | Завершил выполнение `run()` | Естественное или исключение |

**Ключевое различие BLOCKED vs WAITING vs TIMED_WAITING:**

- **BLOCKED** — поток хочет войти в `synchronized`-блок, но монитор занят другим потоком. Переход обратно в RUNNABLE происходит **автоматически**, как только монитор освободится — планировщик JVM выбирает один из заблокированных потоков.

- **WAITING** — поток сам добровольно «уснул», ожидая явного сигнала: `notify()`/`notifyAll()` из `wait()`, завершения целевого потока из `join()`, или `unpark()` из `park()`. Без явного сигнала поток не пробудится никогда.

- **TIMED_WAITING** — то же, что WAITING, но с дедлайном: если сигнал не придёт за указанное время, поток сам вернётся в RUNNABLE (или BLOCKED, если ждёт монитор).

**Диаграмма переходов (упрощённо):**

```
NEW → start() → RUNNABLE ⇄ BLOCKED (synchronized monitor)
RUNNABLE → wait()/join()/park() → WAITING → notify()/unpark()/join() → RUNNABLE
RUNNABLE → sleep(ms)/wait(ms)/join(ms) → TIMED_WAITING → истёк таймер → RUNNABLE
RUNNABLE → run() завершился → TERMINATED
```

**Важные замечания:**

- `RUNNABLE` в Java включает и «готов к выполнению» (waiting for CPU), и «выполняется на CPU». ОС разделяет эти понятия, JVM — нет.
- Поток в `WAITING` из-за `Object.wait()` может пробудиться **spuriously** (ложное пробуждение) без `notify()`. Поэтому проверку условия всегда нужно оборачивать в `while`, не в `if`.
- `Thread.getState()` возвращает снимок состояния; к моменту использования значения поток уже может сменить состояние.

**Расхождение источников:** Baeldung (baeldung.com/java-thread-lifecycle) и Oracle Javadoc согласуются в описании шести состояний. Расхождение только терминологическое: ОС-уровневое состояние «running» vs «ready» объединены в JVM под одним `RUNNABLE`.

### Вопрос 73. Что такое Exchanger? Как работает метод exchange() и какие гарантии happens-before он предоставляет? Приведите классический пример применения паттерна swap-буферов.

**Категория:** Concurrency · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Exchanger (Java SE 21 & JDK 21) — docs.oracle.com](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/util/concurrent/Exchanger.html)

`Exchanger<V>` — точка синхронизации, в которой два потока могут обменяться объектами. Каждый поток передаёт свой объект при входе в `exchange()`, ждёт партнёра и получает его объект при возврате. `Exchanger` можно рассматривать как двунаправленную форму `SynchronousQueue`.

Гарантия happens-before: для каждой пары потоков, успешно обменявшихся через `Exchanger`, действия до вызова `exchange()` в каждом потоке happens-before действий, следующих за возвратом из соответствующего `exchange()` в другом потоке.

Пример (swap-буферы): один поток заполняет буфер, другой опустошает. Когда заполненный буфер готов — потоки обмениваются: заполняющий получает пустой буфер, опустошающий получает полный.

#### Ответ от Claude

**Как работает `exchange()`:**

`Exchanger<V>` позволяет ровно двум потокам встретиться в точке обмена и атомарно передать объекты друг другу. Первый поток, вызвавший `exchange(obj)`, блокируется и ждёт второго. Когда второй вызывает `exchange(obj2)`, оба потока немедленно разблокируются: первый получает `obj2`, второй получает `obj`. Если второй поток не приходит — первый остаётся заблокированным (или бросает `TimeoutException` при использовании `exchange(obj, timeout, unit)`).

**Гарантия happens-before:**

Всё, что поток A сделал до `exchange()`, гарантированно видно потоку B после его `exchange()` — и наоборот. Это аналогично барьеру памяти: JVM не позволяет переупорядочить операции записи через точку обмена.

**Классический пример swap-буферов:**

```java
class SwapBuffers {
    static final Exchanger<DataBuffer> exchanger = new Exchanger<>();

    // Поток-заполнитель
    static class Filler implements Runnable {
        public void run() {
            DataBuffer buf = new DataBuffer(); // пустой буфер
            try {
                while (true) {
                    fill(buf);                       // заполняем данными
                    buf = exchanger.exchange(buf);   // меняем на пустой
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }
    }

    // Поток-потребитель
    static class Drainer implements Runnable {
        public void run() {
            DataBuffer buf = new DataBuffer(); // изначально пустой
            try {
                while (true) {
                    buf = exchanger.exchange(buf); // получаем полный
                    drain(buf);                    // опустошаем данные
                    buf.clear();                   // готовим для обмена
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }
    }
}
```

**Ключевые ограничения:**

- `Exchanger` работает только с парой потоков (ровно два). Для большего числа потоков нужны другие примитивы.
- Если один из потоков аварийно завершится, партнёр зависнет в `exchange()` навсегда — используйте `exchange(obj, timeout, unit)` с разумным тайм-аутом в production-коде.

**Применения:** конвейерная обработка данных; двусторонняя передача данных между продюсером и консьюмером; алгоритмы, требующие обмена состоянием между двумя параллельными вычислениями.

### Вопрос 74. Что такое DelayQueue? Как работает интерфейс Delayed, в каком порядке извлекаются элементы и как реализовать задачу с отложенным выполнением с помощью этой очереди?

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [DelayQueue (Java SE 21 & JDK 21) — docs.oracle.com](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/util/concurrent/DelayQueue.html)

Неограниченная блокирующая очередь (`BlockingQueue`) элементов, реализующих интерфейс `Delayed`, из которой элемент можно извлечь только тогда, когда истёк его задержанный срок. Головным элементом очереди является тот элемент `Delayed`, чей срок задержки истёк раньше всех в прошлом. Если ни у одного элемента задержка не истекла, `poll()` возвращает `null`. Вставка никогда не блокируется: очередь неограничена.

Истечение срока происходит, когда метод `getDelay(TimeUnit.NANOSECONDS)` элемента возвращает значение, меньшее или равное нулю.

#### Ответ от Claude

**Интерфейс `Delayed`:**

```java
public interface Delayed extends Comparable<Delayed> {
    long getDelay(TimeUnit unit);
}
```

Метод `getDelay()` возвращает оставшееся время задержки в запрошенных единицах. Когда значение становится ≤ 0, элемент считается «созревшим» и доступен для извлечения. `Comparable` используется для упорядочивания: элемент с наименьшим оставшимся временем стоит первым.

**Порядок извлечения:**

Элементы извлекаются в порядке истечения задержки (наименьшая задержка — первым). При равной задержке порядок не определён. Элемент с истёкшей задержкой является «головой» очереди и доступен через `take()` и `poll()`. Элементы с неистёкшей задержкой по-прежнему учитываются в `size()`, но недоступны для извлечения.

**Реализация задачи с отложенным выполнением:**

```java
class DelayedTask implements Delayed {
    private final String name;
    private final long startTime; // нс, когда задача «созреет»

    public DelayedTask(String name, long delayMs) {
        this.name = name;
        this.startTime = System.nanoTime() + TimeUnit.MILLISECONDS.toNanos(delayMs);
    }

    @Override
    public long getDelay(TimeUnit unit) {
        long remaining = startTime - System.nanoTime();
        return unit.convert(remaining, TimeUnit.NANOSECONDS);
    }

    @Override
    public int compareTo(Delayed other) {
        return Long.compare(
            getDelay(TimeUnit.NANOSECONDS),
            other.getDelay(TimeUnit.NANOSECONDS)
        );
    }

    @Override public String toString() { return name; }
}

// Использование
DelayQueue<DelayedTask> queue = new DelayQueue<>();
queue.add(new DelayedTask("Задача через 3с", 3000));
queue.add(new DelayedTask("Задача через 1с", 1000));
queue.add(new DelayedTask("Задача через 5с", 5000));

// Рабочий поток
while (true) {
    DelayedTask task = queue.take(); // блокируется до созревания следующей задачи
    System.out.println("Выполняю: " + task);
}
// Порядок: 1с → 3с → 5с
```

**Типичные сценарии:**

- Кэши с временем жизни (TTL): элемент добавляется с задержкой, равной TTL; отдельный поток извлекает «созревшие» и удаляет из основного хранилища.
- Повторные попытки с отсрочкой (retry with backoff).
- Планировщик задач в реализации ScheduledThreadPoolExecutor (внутренне использует аналогичный механизм).

**Ограничения:** очередь неограничена — `put()` и `offer()` никогда не блокируются. `null`-элементы запрещены. `DrainTo()` извлекает только элементы с истёкшей задержкой.

### Вопрос 75. Что такое BlockingQueue и какие четыре группы операций она предоставляет? Чем ArrayBlockingQueue отличается от LinkedBlockingQueue по механизму блокировки, памяти и поддержке fairness?

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Guide to java.util.concurrent.BlockingQueue — Baeldung](https://www.baeldung.com/java-blocking-queue)

`BlockingQueue` — потокобезопасная очередь из `java.util.concurrent`, блокирующая поток-производитель при полной очереди и поток-потребитель при пустой. Решает задачу «производитель — потребитель» без явной синхронизации.

Четыре группы операций:

| Группа | Вставка | Удаление |
|---|---|---|
| Бросает исключение | `add(e)` | `remove()` |
| Возвращает спецзначение | `offer(e)` → `true`/`false` | `poll()` → элемент или `null` |
| Блокирует | `put(e)` | `take()` |
| Блокирует с тайм-аутом | `offer(e, timeout, unit)` | `poll(timeout, unit)` |

#### Ответ от Claude

**Назначение `BlockingQueue`:**

`BlockingQueue` из `java.util.concurrent` решает классическую задачу «производитель — потребитель» без явной синхронизации: потоки-производители кладут данные в очередь, потоки-потребители берут. Если очередь полна — `put()` блокирует производителя; если пуста — `take()` блокирует потребителя. Вся синхронизация скрыта внутри реализации.

**Различия ArrayBlockingQueue и LinkedBlockingQueue:**

| Аспект | `ArrayBlockingQueue` | `LinkedBlockingQueue` |
|---|---|---|
| Структура данных | Массив фиксированного размера | Связный список из узлов |
| Ограниченность | Всегда ограничена (capacity обязателен) | По умолчанию `Integer.MAX_VALUE` |
| Выделение памяти | Всё выделяется при создании | Узлы создаются по мере добавления |
| Механизм блокировки | Один `ReentrantLock` для put и take | Два отдельных замка: `putLock` и `takeLock` |
| Параллелизм put/take | Не параллельны (один замок) | Параллельны (разные замки) |
| Поддержка fairness | Да: `new ArrayBlockingQueue(n, true)` | Нет |
| Пропускная способность | Ниже при высоком параллелизме | Выше (два замка снижают конкуренцию) |

**Ключевое преимущество двух замков в `LinkedBlockingQueue`:** производитель и потребитель могут работать одновременно без блокировки друг друга (put и take захватывают разные замки). В `ArrayBlockingQueue` единый замок означает, что `put()` и `take()` всегда исключают друг друга.

**Рекомендации по выбору:**

- `ArrayBlockingQueue(n)` — нужен точный контроль ёмкости очереди и/или политика честного доступа. Предсказуемое потребление памяти.
- `LinkedBlockingQueue` — нужна максимальная пропускная способность при высоком параллелизме (много одновременных производителей и потребителей). Если ёмкость не указана — следите за ростом очереди в production.

### Вопрос 76. Как работает ScheduledExecutorService? Чем scheduleAtFixedRate() отличается от scheduleWithFixedDelay()? Что происходит, если выполнение задачи превышает период, и что случится при непойманном исключении?

**Категория:** Concurrency · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ScheduledExecutorService (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/ScheduledExecutorService.html)

**`scheduleAtFixedRate(command, initialDelay, period, unit)`** — создаёт и выполняет периодическое действие, первый запуск — через `initialDelay`, затем через `initialDelay + period`, `initialDelay + 2 * period` и т.д. Если какое-либо выполнение задачи занимает больше периода, то последующие выполнения могут начаться с опозданием, **но не будут выполняться параллельно**.

**`scheduleWithFixedDelay(command, initialDelay, delay, unit)`** — создаёт и выполняет периодическое действие с заданной задержкой **между завершением** одного выполнения **и началом** следующего.

#### Ответ от Claude

**Концептуальное различие:**

`scheduleAtFixedRate` измеряет интервал от **начала** одного выполнения до **начала** следующего. `scheduleWithFixedDelay` измеряет задержку от **конца** одного выполнения до **начала** следующего.

**Временная диаграмма (period/delay = 10 с, выполнение = 3 с):**

```
scheduleAtFixedRate:
  0с:  [===] запуск 1 (3с)
  10с: [===] запуск 2 (3с)   <- старт через 10с от предыдущего старта
  20с: [===] запуск 3 (3с)

scheduleWithFixedDelay:
  0с:  [===] запуск 1 (3с)
  13с: [===] запуск 2 (3с)   <- старт через 10с после КОНЦА предыдущего
  26с: [===] запуск 3 (3с)
```

**Что происходит, если выполнение превышает период (только для `scheduleAtFixedRate`):**

```
scheduleAtFixedRate (period = 10с, выполнение = 15с):
  0с:  [===============] запуск 1 (15с)
  15с: [===============] запуск 2  <- старт сразу после конца, но НЕ параллельно
```

Очередной старт откладывается до завершения предыдущего, но никогда не запускается параллельно. В `scheduleWithFixedDelay` такой проблемы нет: следующий старт всегда после полного завершения предыдущего.

**Критическая ловушка — непойманное исключение:**

```java
ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(1);

// ОПАСНО: если задача бросит исключение — она молча прекратит выполняться
scheduler.scheduleAtFixedRate(() -> {
    riskyOperation(); // бросает RuntimeException
}, 0, 10, TimeUnit.SECONDS);

// ПРАВИЛЬНО: всегда оборачивать в try-catch
scheduler.scheduleAtFixedRate(() -> {
    try {
        riskyOperation();
    } catch (Exception e) {
        log.error("Ошибка в периодической задаче", e);
    }
}, 0, 10, TimeUnit.SECONDS);
```

Если задача бросает непойманное исключение, `ScheduledExecutorService` **молча прекращает её выполнение** без какого-либо предупреждения. Исключение можно получить только через `Future.get()` от возвращённого `ScheduledFuture`. Это одна из самых распространённых ловушек в production-коде: задача «исчезает» после первой ошибки.

**Когда что использовать:**

- `scheduleAtFixedRate` — мониторинг, синхронизация с внешней системой по расписанию, «каждые N секунд независимо от времени выполнения».
- `scheduleWithFixedDelay` — задачи с переменным временем выполнения, где важна пауза между итерациями, а не фиксированный ритм.

### Вопрос 77. Что такое ForkJoinTask, RecursiveTask и RecursiveAction в Java? Как реализовать алгоритм «разделяй и властвуй» с помощью ForkJoinPool, и чем fork/join отличается от обычного ExecutorService?

**Категория:** Concurrency · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Guide to the Fork/Join Framework in Java — Baeldung](https://www.baeldung.com/java-fork-join)

`ForkJoinTask` — базовый тип задач, выполняемых внутри `ForkJoinPool`. На практике следует расширять один из двух подклассов: `RecursiveAction` — для задач без возвращаемого значения, и `RecursiveTask<V>` — для задач, возвращающих результат. Оба имеют абстрактный метод `compute()`, в котором определяется логика задачи.

#### Ответ от Claude

**Иерархия ForkJoin:**

```
ForkJoinTask<V>
├── RecursiveAction   (extends ForkJoinTask<Void>) — не возвращает результат
└── RecursiveTask<V>  (extends ForkJoinTask<V>)    — возвращает V
```

**RecursiveAction vs RecursiveTask:**

| Аспект | `RecursiveAction` | `RecursiveTask<V>` |
|---|---|---|
| Возвращаемое значение | Нет (аналог `Runnable`) | Есть (аналог `Callable<V>`) |
| Метод `compute()` | `protected void compute()` | `protected V compute()` |
| Применение | Параллельная сортировка, запись, side-effects | Параллельные вычисления с результатом |

**Пример RecursiveTask — параллельная сумма массива:**

```java
class SumTask extends RecursiveTask<Long> {
    private static final int THRESHOLD = 1000;
    private final int[] array;
    private final int from, to;

    SumTask(int[] array, int from, int to) {
        this.array = array; this.from = from; this.to = to;
    }

    @Override
    protected Long compute() {
        if (to - from <= THRESHOLD) {
            // Базовый случай — считаем напрямую
            long sum = 0;
            for (int i = from; i < to; i++) sum += array[i];
            return sum;
        }
        // Разделяем задачу
        int mid = (from + to) / 2;
        SumTask left  = new SumTask(array, from, mid);
        SumTask right = new SumTask(array, mid, to);
        left.fork();               // асинхронно запускаем левую часть
        long rightResult = right.compute(); // правую — в текущем потоке
        long leftResult  = left.join();     // ждём левую
        return leftResult + rightResult;
    }
}

ForkJoinPool pool = new ForkJoinPool();
int[] array = IntStream.rangeClosed(1, 1_000_000).toArray();
long sum = pool.invoke(new SumTask(array, 0, array.length));
```

**Чем ForkJoinPool отличается от ExecutorService:**

| Аспект | `ExecutorService` (ThreadPool) | `ForkJoinPool` |
|---|---|---|
| Алгоритм планирования | FIFO-очередь | Work-stealing (каждый поток — своя дека) |
| Применение | Независимые задачи | Рекурсивные, делимые задачи |
| Блокировка при `join()` | Блокирует поток | Поток «крадёт» другие задачи пока ждёт |
| Параллелизм | Число потоков в пуле | Число ядер CPU (по умолчанию) |

**Work-stealing** — ключевое преимущество: когда поток ждёт `join()` подзадачи, он не простаивает — берёт задачи из хвоста деки другого потока, обеспечивая максимальную загрузку CPU при рекурсивных алгоритмах.

**Практическое правило:** `ForkJoinPool.commonPool()` (доступен с Java 8) используется автоматически параллельными стримами (`stream().parallel()`). Для собственных `RecursiveTask` создавайте отдельный пул, чтобы не конкурировать с параллельными стримами.

### Вопрос 78. Что такое TransferQueue и LinkedTransferQueue? Чем метод transfer() принципиально отличается от put() в обычной BlockingQueue?

**Категория:** Concurrency · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Guide to the Java TransferQueue — Baeldung](https://www.baeldung.com/java-transfer-queue)

`TransferQueue` — расширение `BlockingQueue`, предоставляющее дополнительные методы для передачи элементов непосредственно ожидающему потребителю. Ключевой метод — `transfer(E e)`, который блокирует производителя **до тех пор, пока потребитель не заберёт элемент**. В отличие от `put()`, который помещает элемент в очередь и немедленно возвращается, `transfer()` гарантирует фактическую передачу.

`LinkedTransferQueue` — единственная реализация `TransferQueue` в стандартной библиотеке. Реализована на основе CAS-операций без блокировок, сочетает свойства `ConcurrentLinkedQueue`, `SynchronousQueue` и неограниченной `LinkedBlockingQueue`.

#### Ответ от Claude

**Семантическое различие `put()` vs `transfer()`:**

| Метод | Блокируется до... | Гарантия |
|---|---|---|
| `put(e)` | Появления места в очереди | Элемент помещён в очередь (но не обязательно прочитан) |
| `transfer(e)` | Фактического получения элемента потребителем | Потребитель уже держит элемент |
| `tryTransfer(e)` | Не блокируется | `true` — если потребитель был готов немедленно |
| `tryTransfer(e, timeout, unit)` | До тайм-аута | `true` — если передача произошла в срок |

**Пример разницы в поведении:**

```java
LinkedTransferQueue<String> queue = new LinkedTransferQueue<>();

// Поток-потребитель — появится через 2 секунды
new Thread(() -> {
    try {
        Thread.sleep(2000);
        String item = queue.take();
        System.out.println("Получено: " + item);
    } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
}).start();

// С put(): немедленно возвращается, элемент лежит в очереди
queue.put("данные");
System.out.println("put() вернулся сразу"); // выводится немедленно

// С transfer(): блокируется ~2 секунды, пока потребитель не возьмёт
queue.transfer("данные");
System.out.println("transfer() вернулся после получения потребителем"); // через ~2с
```

**Когда полезен `TransferQueue`:**

- Backpressure: производитель не должен опережать потребителя более чем на одну «транзакцию».
- Сценарии с подтверждением: производитель должен знать, что его данные приняты.
- Симуляция синхронного вызова через очередь: `transfer()` — семантика «отправить и дождаться получения».

**`LinkedTransferQueue` vs `SynchronousQueue`:** `SynchronousQueue` не имеет буфера — элемент передаётся только при наличии ожидающего потребителя. `LinkedTransferQueue` имеет буфер: `put()` кладёт в очередь без ожидания, а `transfer()` ждёт потребителя. Это даёт возможность выбирать семантику на уровне каждого вызова.

### Вопрос 79. Что такое VarHandle (Java 9, JEP 193)? Чем он превосходит sun.misc.Unsafe и AtomicFieldUpdater? Опишите четыре режима доступа к памяти: plain, opaque, acquire/release, volatile.

**Категория:** Concurrency · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [VarHandle (Java SE 21 & JDK 21) — docs.oracle.com](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/lang/invoke/VarHandle.html)

`VarHandle` — типизированная ссылка на переменную, поддерживающая различные режимы доступа. Режимы делятся на четыре группы:

- **Plain** (`get`/`set`) — без гарантий видимости между потоками.
- **Opaque** (`getOpaque`/`setOpaque`) — только битовая атомарность, без межпоточных гарантий порядка.
- **Acquire/Release** (`getAcquire`/`setRelease`) — односторонний барьер: всё до `setRelease` видно после `getAcquire` в другом потоке.
- **Volatile** (`getVolatile`/`setVolatile`) — полная семантика `volatile`, двусторонний барьер.

Дополнительно: атомарные операции `compareAndSet`, `compareAndExchange`, `getAndSet`, `getAndAdd` и барьерные методы `fullFence`, `acquireFence`, `releaseFence`.

#### Ответ от Claude

**Почему VarHandle появился:**

До Java 9 низкоуровневые атомарные операции были доступны только через `sun.misc.Unsafe` — внутренний, недокументированный API, требующий обхода security manager. Попытки инкапсулировать его через `AtomicIntegerFieldUpdater` / `AtomicReferenceFieldUpdater` были громоздкими и медленными. `VarHandle` (JEP 193) предоставляет те же возможности через публичный, безопасный и JIT-оптимизируемый API.

**Преимущества VarHandle:**

| Критерий | `sun.misc.Unsafe` | `AtomicXxxFieldUpdater` | `VarHandle` |
|---|---|---|---|
| Публичный API | Нет | Да | Да |
| Типобезопасность | Нет (raw offsets) | Частичная | Да |
| JIT-оптимизации | Базовые | Ограниченные | Полные (intrinsified) |
| Богатство режимов доступа | Базовые | Нет | Полный спектр |

**Четыре уровня гарантий памяти (от слабых к сильным):**

```
Plain < Opaque < Acquire/Release < Volatile
(меньше гарантий = выше производительность)
```

- **Plain** — читает/пишет поле без каких-либо барьеров. Для однопоточного кода.
- **Opaque** — операция атомарна на уровне бит (не будет «рваного» чтения `long`). Никаких happens-before с другими переменными.
- **Acquire/Release** — `setRelease` делает все предыдущие записи видимыми потоку, который прочитает ту же переменную через `getAcquire`. Одностороннее ограждение: дешевле `volatile` при паттерне «писатель → читатель».
- **Volatile** — полная двусторонняя гарантия, как у поля `volatile`.

**Пример создания и использования:**

```java
class Counter {
    private long count = 0;

    private static final VarHandle COUNT;
    static {
        try {
            COUNT = MethodHandles.lookup()
                .findVarHandle(Counter.class, "count", long.class);
        } catch (ReflectiveOperationException e) {
            throw new Error(e);
        }
    }

    public void increment() {
        COUNT.getAndAdd(this, 1L); // атомарный инкремент без synchronized
    }

    public long get() {
        return (long) COUNT.getVolatile(this); // volatile-чтение
    }
}
```

**Практическое правило выбора режима:** для lock-free структур данных используйте `acquire/release` там, где достаточно однонаправленной синхронизации; `volatile` — только при необходимости полной двусторонней гарантии. Plain/Opaque — для внутренних структур с ручным управлением видимостью (редкий случай в прикладном коде).

### Вопрос 80. Что такое ConcurrentSkipListMap? Как работает структура данных skip list, почему операции в ConcurrentSkipListMap в среднем занимают O(log n), и чем она отличается от ConcurrentHashMap?

**Категория:** Concurrency · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [ConcurrentSkipListMap (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/ConcurrentSkipListMap.html)

Масштабируемая конкурентная `ConcurrentNavigableMap` реализация. Карта сортируется согласно естественному порядку ключей или `Comparator`'у, указанному при создании. Ожидаемые средние затраты по времени на операции `containsKey`, `get`, `put` и `remove` составляют `log(n)`. Операции вставки, удаления, обновления и доступа безопасно выполняются конкурентно несколькими потоками. Итераторы **слабо согласованные** (`weakly consistent`): они не бросают `ConcurrentModificationException` и могут отражать изменения, произошедшие после создания итератора.

#### Ответ от Claude

**Структура данных skip list:**

Skip list — вероятностная структура данных, представляющая собой многоуровневый связный список. Нижний уровень — обычный упорядоченный список всех элементов. Каждый вышестоящий уровень содержит подмножество элементов уровня ниже (вероятность включения — обычно 0,5). Верхние уровни работают как «скоростные шоссе» для быстрого поиска.

```
Уровень 3:  1 ─────────────────────────────────────── 64
Уровень 2:  1 ──────────── 16 ─────────── 32 ──────── 64
Уровень 1:  1 ── 4 ──────── 16 ─── 21 ─── 32 ── 48 ── 64
Уровень 0:  1 ─ 3 ─ 4 ─ 7 ─ 16 ─ 21 ─ 25 ─ 32 ─ 48 ─ 64  (все элементы)
```

Поиск начинается с верхнего уровня и «спускается» вниз при необходимости. Математически ожидаемая сложность поиска — O(log n).

**Почему `ConcurrentSkipListMap` lock-free:**

В отличие от `TreeMap` (который требует полной блокировки для балансировки), skip list позволяет вставку и удаление с использованием **CAS-операций** (Compare-And-Swap) на отдельных узлах. Разные потоки могут одновременно модифицировать разные части списка без глобальной блокировки. Алгоритм Doug Lea (`java.util.concurrent`) реализует не только CAS, но и специальный механизм «маркированных» узлов для корректного удаления.

**ConcurrentSkipListMap vs ConcurrentHashMap:**

| Аспект | `ConcurrentHashMap` | `ConcurrentSkipListMap` |
|---|---|---|
| Сортировка ключей | Нет | Да (естественный порядок или Comparator) |
| Сложность get/put | O(1) в среднем | O(log n) в среднем |
| Range-операции | Нет | Да: `subMap`, `headMap`, `tailMap`, `floorKey`, `ceilingKey` |
| Потребление памяти | Меньше | Больше (указатели нескольких уровней) |
| Concurrent аналог | `HashMap` | `TreeMap` |
| Итераторы | Weakly consistent | Weakly consistent |

**Когда выбирать `ConcurrentSkipListMap`:**

- Нужны конкурентные операции с диапазонами ключей: «все записи с timestamp от A до B».
- Нужна конкурентная `NavigableMap` / `SortedMap` — например, очередь приоритетов с произвольным доступом по ключу.
- Нужен конкурентный аналог `TreeMap`.

**Не подходит**, если порядок ключей не важен: `ConcurrentHashMap` быстрее (O(1) vs O(log n)) и потребляет меньше памяти.

**Расхождение источников:** Baeldung (baeldung.com/java-concurrent-skip-list-map) описывает `ConcurrentSkipListMap` как «mostly lock-free» с применением CAS. Oracle Javadoc формально гарантирует только log(n) и безопасность параллельного доступа, не раскрывая детали реализации. Исходный код JDK подтверждает использование CAS и алгоритм маркировки узлов при удалении.
