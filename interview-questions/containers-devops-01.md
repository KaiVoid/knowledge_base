### Вопрос 1. В чём разница между Docker-образом и Docker-контейнером? Как из одного образа запустить несколько контейнеров?

**Категория:** Контейнеры/DevOps · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [The Difference Between Docker Images and Containers — AWS](https://aws.amazon.com/compare/the-difference-between-docker-images-and-containers/)

Docker-образ — это неизменяемый шаблон, содержащий инструкции и зависимости, необходимые для создания контейнера. Docker-контейнер — это самостоятельный, готовый к запуску экземпляр образа. Образ неизменяем и состоит из слоёв только для чтения; контейнер добавляет поверх них изменяемый слой для записи. Из одного образа можно горизонтально масштабировать приложение, запуская несколько экземпляров командой `docker run` с разными именами контейнеров — каждый будет независимым процессом на одной машине или в кластере.

#### Ответ от Claude

**Docker-образ (image)** — это неизменяемый (read-only) артефакт, хранящийся на диске. Он содержит всё необходимое для запуска приложения: исходный код, среду выполнения, библиотеки, переменные окружения и конфигурационные файлы. Образ представляет собой стек слоёв — каждый слой фиксирует набор изменений файловой системы (добавление, удаление, модификацию файлов) и неизменяем после создания.

**Docker-контейнер (container)** — это живой запущенный экземпляр образа. При старте Docker использует union filesystem (OverlayFS или аналог): все слои образа объединяются в единое представление файловой системы, а поверх них создаётся тонкий изменяемый слой (container layer), куда записываются все изменения, сделанные внутри контейнера. Образ при этом остаётся нетронутым.

**Ключевые различия:**

| Характеристика | Образ | Контейнер |
|---|---|---|
| Состояние | Статичный артефакт на диске | Работающий процесс |
| Изменяемость | Неизменяемый (read-only) | Имеет изменяемый слой поверх образа |
| Жизненный цикл | Существует до явного удаления | Запускается, останавливается, удаляется |
| Аналогия | Класс (class) в ООП | Экземпляр (instance) класса |

**Запуск нескольких контейнеров из одного образа:**

Поскольку каждый контейнер получает свой изолированный слой для записи, а слои образа остаются общими, из одного образа можно запустить произвольное количество контейнеров одновременно:

```bash
docker run -d --name app1 -p 8081:8080 my-app:latest
docker run -d --name app2 -p 8082:8080 my-app:latest
docker run -d --name app3 -p 8083:8080 my-app:latest
```

Каждый контейнер (`app1`, `app2`, `app3`) изолирован: имеет свой сетевой интерфейс, файловую систему изменений, PID-пространство и конфигурацию. Слои самого образа при этом не дублируются на диске — они разделяются между всеми контейнерами, что экономит память и место на диске.

### Вопрос 2. Что такое слои (layers) в Docker-образе? Почему порядок инструкций в Dockerfile влияет на время повторной сборки?

**Категория:** Контейнеры/DevOps · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Using the build cache — Docker Docs](https://docs.docker.com/get-started/docker-concepts/building-images/using-the-build-cache/)

Каждая инструкция в Dockerfile создаёт отдельный слой образа. При повторной сборке Docker проверяет, можно ли переиспользовать слой из предыдущей сборки (кэш). Кэш инвалидируется каскадно: если слой изменился, все последующие слои также считаются устаревшими и пересобираются заново. Поэтому инструкции, которые меняются редко (установка зависимостей), следует размещать в начале Dockerfile, а часто меняющиеся (копирование кода приложения) — в конец. Эта оптимизация позволяет снизить время пересборки с 16+ секунд до ~1 секунды при изменении только кода приложения.

#### Ответ от Claude

**Слой (layer)** — это зафиксированный набор изменений файловой системы (добавление, удаление или модификация файлов), создаваемый одной инструкцией Dockerfile. Слои неизменяемы и идентифицируются хешем содержимого. Итоговый образ — это стек таких слоёв, интерпретируемый union filesystem снизу вверх.

Пример: Dockerfile с инструкциями `FROM`, `RUN apt-get install`, `COPY . .` создаёт три слоя. Каждый хранится отдельно и может переиспользоваться другими образами, если начало их истории совпадает.

**Механизм кэша и порядок инструкций:**

Docker BuildKit при каждой сборке проходит по инструкциям и для каждой проверяет кэш. Кэш считается валидным, если:
- команда инструкции не изменилась,
- для `COPY`/`ADD` — не изменились копируемые файлы,
- все предшествующие слои также попали в кэш.

**Каскадная инвалидация** — главный принцип: как только хотя бы один слой признаётся изменённым, кэш для него и всех последующих слоёв сбрасывается, и они пересобираются заново.

**Практическое следствие — порядок инструкций от «стабильных» к «изменчивым»:**

Плохой порядок (исходный код копируется до установки зависимостей):
```dockerfile
FROM eclipse-temurin:21-jre
COPY . /app          # меняется при каждом коммите — инвалидирует всё ниже
RUN mvn package      # пересобирается при каждом изменении кода
```

Хороший порядок (зависимости отделены от кода):
```dockerfile
FROM eclipse-temurin:21-jre
COPY pom.xml /app/           # меняется редко — кэш стабилен
RUN mvn dependency:go-offline # кэшируется до изменения pom.xml
COPY src/ /app/src/           # меняется часто, но только пересобирает финальный слой
RUN mvn package
```

Аналогичная стратегия для Node.js-приложений: сначала `COPY package.json`, `RUN npm install`, затем `COPY . .` — изменение исходного кода не пересобирает `node_modules`.

**Дополнительные рекомендации:**
- Объединяйте связанные `RUN`-команды в одну через `&&`, чтобы не плодить лишние слои и не оставлять кэш пакетных менеджеров в промежуточных слоях.
- Используйте многоэтапную сборку (multi-stage build) для разделения этапа компиляции и финального образа.

### Вопрос 3. Какие инструкции Dockerfile вы знаете? В чём разница между CMD и ENTRYPOINT, а также между COPY и ADD?

**Категория:** Контейнеры/DevOps · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Dockerfile reference — Docker Docs](https://docs.docker.com/reference/dockerfile/)

Основные инструкции Dockerfile: `FROM` (базовый образ), `RUN` (команда во время сборки), `COPY` (копирование файлов из контекста сборки), `ADD` (копирование с поддержкой tar-архивов и remote URL), `WORKDIR` (рабочая директория), `ENV` (переменные окружения), `ARG` (переменные сборки), `EXPOSE` (декларация порта), `CMD` (команда по умолчанию), `ENTRYPOINT` (исполняемый файл контейнера), `USER` (пользователь), `VOLUME` (точки монтирования).

`CMD` задаёт команду и/или аргументы по умолчанию, которые можно полностью переопределить через аргументы `docker run`. `ENTRYPOINT` определяет основной исполняемый файл контейнера — аргументы `docker run` дополняют `ENTRYPOINT`, а не заменяют его. При совместном использовании `ENTRYPOINT` задаёт команду, а `CMD` — аргументы по умолчанию для неё. `COPY` выполняет только базовое копирование файлов из контекста сборки. `ADD` расширяет эту функциональность: автоматически распаковывает локальные tar-архивы и поддерживает загрузку по HTTPS/Git URL. Рекомендация: использовать `COPY` везде, где не нужны специальные возможности `ADD`.

#### Ответ от Claude

**Основные инструкции Dockerfile:**

| Инструкция | Назначение |
|---|---|
| `FROM` | Задаёт базовый образ (обязательна, первая инструкция) |
| `RUN` | Выполняет команду во время сборки образа, создаёт новый слой |
| `COPY` | Копирует файлы/директории из контекста сборки или предыдущего этапа |
| `ADD` | Как COPY, плюс распаковка tar-архивов и загрузка по URL |
| `WORKDIR` | Устанавливает рабочую директорию для последующих инструкций |
| `ENV` | Задаёт переменные окружения (доступны и во время сборки, и во время выполнения) |
| `ARG` | Переменная, передаваемая при сборке (`--build-arg`); недоступна в runtime |
| `EXPOSE` | Документирует порт, на котором слушает приложение (не открывает его реально) |
| `CMD` | Команда и/или аргументы по умолчанию при старте контейнера |
| `ENTRYPOINT` | Основной исполняемый файл контейнера |
| `USER` | Устанавливает пользователя для последующих инструкций и при запуске |
| `VOLUME` | Объявляет точки монтирования для постоянных данных |
| `LABEL` | Добавляет метаданные к образу (ключ=значение) |
| `HEALTHCHECK` | Определяет команду для проверки состояния контейнера |

---

**CMD vs ENTRYPOINT:**

`CMD` определяет команду (и/или аргументы) по умолчанию, которая выполняется при старте контейнера. Её можно полностью заменить, передав аргументы в `docker run`:
```bash
docker run my-image echo "hello"  # полностью заменяет CMD
```

`ENTRYPOINT` задаёт основной исполняемый файл контейнера. Аргументы `docker run` не заменяют `ENTRYPOINT`, а дописываются к нему. Изменить `ENTRYPOINT` можно только флагом `--entrypoint`:
```bash
docker run --entrypoint /bin/sh my-image
```

**Типичный паттерн совместного использования:**
```dockerfile
ENTRYPOINT ["java", "-jar", "/app/app.jar"]
CMD ["--spring.profiles.active=prod"]
```
Здесь `ENTRYPOINT` — неизменяемая часть команды, `CMD` — аргументы по умолчанию, которые можно переопределить: `docker run my-image --spring.profiles.active=dev`.

**Важно:** обе инструкции поддерживают две формы записи:
- **exec-форма** `["команда", "аргумент"]` — предпочтительна: процесс запускается напрямую, получает сигналы ОС (SIGTERM и др.).
- **shell-форма** `команда аргумент` — оборачивается в `/bin/sh -c`, что означает PID 1 занимает shell, а не ваше приложение; сигналы могут не доходить.

---

**COPY vs ADD:**

`COPY` — простое и предсказуемое копирование файлов или директорий из контекста сборки (или из предыдущего этапа multi-stage build) в файловую систему образа:
```dockerfile
COPY src/ /app/src/
COPY --from=builder /app/target/app.jar /app/app.jar
```

`ADD` делает всё то же самое плюс:
1. **Автораспаковка tar-архивов**: `ADD archive.tar.gz /opt/` — содержимое архива будет распаковано в `/opt/`.
2. **Загрузка по URL**: `ADD https://example.com/file.txt /tmp/file.txt` — файл будет скачан и помещён в образ (не рекомендуется: лучше использовать `RUN curl` для большего контроля и кэширования).

**Рекомендация (официальная):** всегда используйте `COPY`, если вам не нужны специфические возможности `ADD`. `COPY` явнее выражает намерение и не создаёт неожиданных побочных эффектов (например, непреднамеренной распаковки архива).
### Вопрос 4. Что такое многоэтапная сборка (multi-stage build) в Docker и зачем она нужна для Java-приложений? Приведите пример Dockerfile с двумя стадиями: Maven-сборка и финальный образ на JRE.

**Категория:** Контейнеры/DevOps · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Multi-stage builds | Docker Docs](https://docs.docker.com/get-started/docker-concepts/building-images/multi-stage-builds/)

Многоэтапная сборка позволяет определить несколько стадий в одном Dockerfile, каждая из которых выполняет отдельную задачу. «Разделяя среду сборки от финальной среды выполнения, вы можете значительно уменьшить размер образа и поверхность атаки.»

Для Java-приложений это особенно выгодно: на первой стадии используется полный JDK вместе с Maven для компиляции и упаковки артефакта, а на второй стадии — только JRE, куда копируется уже готовый JAR. Официальная документация Docker приводит следующий пример с использованием Eclipse Temurin:

```dockerfile
FROM eclipse-temurin:21.0.8_9-jdk-jammy AS builder
WORKDIR /opt/app
COPY .mvn/ .mvn
COPY mvnw pom.xml ./
RUN ./mvnw dependency:go-offline
COPY ./src ./src
RUN ./mvnw clean install

FROM eclipse-temurin:21.0.8_9-jre-jammy AS final
WORKDIR /opt/app
EXPOSE 8080
COPY --from=builder /opt/app/target/*.jar /opt/app/*.jar
ENTRYPOINT ["java", "-jar", "/opt/app/*.jar"]
```

Результат: итоговый образ уменьшается с ~880 МБ до ~428 МБ, так как в нём отсутствуют Maven, исходный код и JDK-инструменты.

#### Ответ от Claude

**Что такое multi-stage build**

Многоэтапная сборка — это возможность Docker (появилась в версии 17.05), позволяющая описать несколько независимых стадий в одном `Dockerfile` с помощью нескольких инструкций `FROM`. Каждая стадия может использовать свой базовый образ, а артефакты между стадиями передаются директивой `COPY --from=<stage>`. В финальный образ попадает только то, что явно скопировано в последнюю стадию; все промежуточные слои отбрасываются.

**Зачем это нужно для Java-приложений**

Java-проекты требуют для сборки полный JDK и Maven (или Gradle) — сотни мегабайт инструментов, которые не нужны в production. Без multi-stage сборки разработчики вынуждены либо предварительно собирать артефакт вне Docker и копировать готовый JAR, либо получать «жирный» образ с JDK и Maven внутри. Multi-stage build решает эту проблему:

- **Уменьшение размера образа** — финальный образ содержит только JRE и JAR (разница может составлять 400–600 МБ).
- **Сокращение поверхности атаки** — компилятор, исходный код и build-инструменты не попадают в production-образ.
- **Воспроизводимость сборки** — весь pipeline (от исходников до образа) описан в одном файле; разработчику не нужно отдельно устанавливать Maven.

**Пример Dockerfile: Maven → JRE**

```dockerfile
# --- Стадия 1: сборка ---
FROM maven:3.9-eclipse-temurin-21 AS builder
WORKDIR /app
# Сначала копируем pom.xml и скачиваем зависимости (кеш-слой)
COPY pom.xml .
RUN mvn dependency:go-offline -B
# Затем копируем исходники и собираем JAR
COPY src ./src
RUN mvn clean package -DskipTests -B

# --- Стадия 2: финальный образ на JRE ---
FROM eclipse-temurin:21-jre-jammy
WORKDIR /app
EXPOSE 8080
# Копируем только готовый JAR из стадии builder
COPY --from=builder /app/target/*.jar app.jar
ENTRYPOINT ["java", "-jar", "app.jar"]
```

**Ключевые детали примера:**

| Элемент | Назначение |
|---|---|
| `AS builder` | Именование стадии для ссылки через `--from` |
| `dependency:go-offline` | Скачивает зависимости в отдельный слой — при изменении только кода Maven-зависимости берутся из кеша |
| `-DskipTests` | Пропуск тестов на этапе сборки образа (тесты запускают в CI отдельно) |
| `eclipse-temurin:21-jre-jammy` | Только JRE (без компилятора javac и инструментов JDK) |
| `COPY --from=builder` | Берёт файл из стадии `builder`, не из хост-системы |

**Итог:** в финальный образ попадают только JRE и один JAR-файл. Maven, JDK, исходный код и весь кеш `.m2` отбрасываются автоматически.

### Вопрос 5. Что такое Docker Registry? В чём разница между Docker Hub и приватным реестром (например, Harbor или AWS ECR)?

**Категория:** Контейнеры/DevOps · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [What is a registry? | Docker Docs](https://docs.docker.com/get-started/docker-concepts/the-basics/what-is-a-registry/)

Docker Registry — это «централизованное место для хранения и распространения образов контейнеров». Реестр служит хранилищем, откуда разработчики берут (pull) и куда публикуют (push) образы. Внутри реестра образы организованы по репозиториям — каждый репозиторий является набором связанных образов, различающихся тегами (например, `project-a:v1.0`, `project-a:v2.0`).

**Docker Hub** — стандартный публичный реестр. Бесплатный план включает неограниченные публичные репозитории и один приватный. **Приватные реестры** (Harbor, JFrog Artifactory, GitLab Container Registry, Amazon ECR, Azure ACR, Google Artifact Registry) предназначены для организаций, которым необходимы контроль доступа, безопасность и соответствие требованиям внутренней политики.

#### Ответ от Claude

**Что такое Docker Registry**

Docker Registry (реестр образов) — это сервис, предназначенный для хранения, версионирования и распространения образов контейнеров. Он работает по протоколу OCI Distribution Specification (бывший Docker Registry HTTP API V2). Взаимодействие с реестром происходит через стандартные команды Docker CLI: `docker pull` (скачать образ), `docker push` (опубликовать образ), `docker login` (аутентификация).

Структура иерархии: **Реестр → Репозиторий → Тег**. Например: `registry.example.com/myapp/backend:1.2.0`.

**Docker Hub**

Docker Hub — официальный публичный реестр Docker, Inc., используемый по умолчанию при отсутствии явного адреса реестра в имени образа (`nginx` = `docker.io/library/nginx:latest`).

Характеристики:
- Крупнейшая публичная библиотека образов: официальные образы ОС, баз данных, языков.
- Бесплатный план: неограниченные публичные репозитории, 1 приватный, ограниченный rate limit для анонимных pull (100 запросов/6 часов) и для авторизованных (200 запросов/6 часов на бесплатном тарифе).
- Платные планы снимают ограничения pull rate и увеличивают число приватных репозиториев.
- Поддержка webhooks, автосборки из GitHub/Bitbucket.

**Harbor (приватный self-hosted реестр)**

Harbor — open-source реестр, выпускаемый под эгидой CNCF (статус graduated project). Разворачивается на собственной инфраструктуре или в Kubernetes.

Ключевые возможности:
- **RBAC** — гибкое разграничение прав доступа по проектам и ролям.
- **Сканирование уязвимостей** — интеграция с Trivy, Clair для проверки образов на CVE.
- **Репликация** — синхронизация образов между несколькими реестрами.
- **Immutable tags** — защита тегов от перезаписи.
- **Подпись образов** (Cosign/Notary) — верификация целостности.
- **SBOM** — автоматическая генерация Software Bill of Materials при публикации.
- Полная совместимость с OCI v2.

**AWS ECR (Elastic Container Registry)**

ECR — управляемый реестр Amazon Web Services, глубоко интегрированный в AWS-экосистему.

Ключевые возможности:
- **Интеграция с IAM** — управление доступом через AWS Identity and Access Management; отдельного логина не требуется при работе в ECS/EKS с правильной IAM-ролью.
- **Lifecycle policies** — автоматическое удаление старых образов по правилам.
- **Автосканирование** — проверка на уязвимости (Enhanced scanning через Amazon Inspector).
- **Cross-region replication** — репликация образов между регионами AWS.
- **Immutable tags** — запрет перезаписи существующих тегов.
- Публичные репозитории через ECR Public Gallery.
- Оплата по факту использования (хранение + передача данных).

**Сравнительная таблица**

| Критерий | Docker Hub | Harbor | AWS ECR |
|---|---|---|---|
| Тип размещения | Облако (SaaS) | Self-hosted / on-prem | Облако AWS (SaaS) |
| Доступность | Публичный + приватный | Только приватный | Приватный + публичный галерея |
| Управление доступом | Пользователи/организации Docker | RBAC по проектам | AWS IAM |
| Сканирование образов | Базовое (платно) | Trivy/Clair (бесплатно) | Amazon Inspector (платно) |
| Стоимость | Freemium | Бесплатно (нужны серверы) | По использованию |
| Rate limiting | Есть на бесплатном плане | Нет | Нет |
| Лучший выбор | Open-source, публичные образы | Enterprise, on-prem, Kubernetes | AWS-инфраструктура (ECS/EKS) |

**Когда что использовать:**
- **Docker Hub** — публикация open-source проектов, получение официальных базовых образов.
- **Harbor** — enterprise-окружения с жёсткими требованиями к безопасности, on-prem, мультикластерные среды Kubernetes.
- **AWS ECR** — если приложение развёртывается в AWS (ECS, EKS, Lambda Container Image) и нужна бесшовная интеграция с IAM/CodeBuild/CodePipeline.

### Вопрос 6. Чем Docker-контейнер отличается от виртуальной машины? Какие ресурсы они изолируют по-разному?

**Категория:** Контейнеры/DevOps · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Difference Between Docker and VM | AWS](https://aws.amazon.com/compare/the-difference-between-docker-vm/)

«Виртуальная машина запускает собственное ядро и гостевую операционную систему вместе с приложениями и их зависимостями.» Гипервизор управляет распределением физических ресурсов: каждой VM выделяется фиксированный объём вычислительных ресурсов заранее, и они резервируются на всё время работы VM.

«Docker-контейнеры совместно используют ресурсы ядра хостовой операционной системы напрямую» и упаковывают только зависимости, специфичные для приложения — без целой ОС.

С точки зрения безопасности: «Поскольку VM запускает целую операционную систему, существует дополнительный уровень изоляции при работе приложений», однако контейнеры «подвергаются риску при наличии уязвимостей в ядре».

#### Ответ от Claude

**Фундаментальное архитектурное отличие**

Виртуальная машина и Docker-контейнер решают схожую задачу — изолировать среду выполнения приложения — но делают это на разных уровнях абстракции.

```
Виртуальная машина:          Docker-контейнер:
┌─────────────────┐          ┌─────────────────┐
│   Приложение    │          │   Приложение    │
├─────────────────┤          ├─────────────────┤
│  Гостевая ОС   │          │  Зависимости    │
│ (ядро + libs)  │          │  (libs, bins)   │
├─────────────────┤          ├─────────────────┤
│   Гипервизор   │          │  Docker Engine  │
├─────────────────┤          ├─────────────────┤
│   Хостовая ОС  │          │  Хостовая ОС   │
├─────────────────┤          │  (общее ядро)  │
│   Железо       │          ├─────────────────┤
└─────────────────┘          │   Железо       │
                             └─────────────────┘
```

**Виртуальная машина** работает поверх гипервизора (Type 1: KVM, Hyper-V, ESXi — непосредственно на железе; Type 2: VirtualBox, VMware Workstation — поверх хостовой ОС). Каждая VM имеет **собственное ядро ОС**, полный набор системных библиотек и, как правило, отдельный процесс init (systemd). Гипервизор эмулирует виртуальное железо: процессор, память, диск, сетевую карту.

**Docker-контейнер** использует **ядро хостовой ОС** совместно со всеми другими контейнерами. Контейнер содержит только пользовательское пространство (user space): библиотеки, исполняемые файлы и само приложение. Docker Engine не эмулирует железо — он использует два встроенных механизма ядра Linux.

**Механизмы изоляции в Linux-контейнерах**

| Механизм ядра | Что изолирует |
|---|---|
| **namespaces** | Видимость ресурсов: процессы (PID), сеть (NET), файловая система (MNT), пользователи (USER), имя хоста (UTS), межпроцессное взаимодействие (IPC) |
| **cgroups** (control groups) | Потребление ресурсов: CPU, RAM, disk I/O, сеть — лимиты и квоты |
| **seccomp** | Фильтрация системных вызовов |
| **AppArmor / SELinux** | Мандатный контроль доступа |

Благодаря namespaces контейнер «думает», что он единственный на машине: у него своя таблица процессов, свой сетевой стек, свои точки монтирования. Cgroups гарантируют, что контейнер не «съест» ресурсы соседей.

**Сравнение по ключевым параметрам**

| Параметр | Docker-контейнер | Виртуальная машина |
|---|---|---|
| Ядро ОС | Общее с хостом | Собственное (гостевое) |
| Размер образа | Десятки МБ | Гигабайты |
| Время запуска | Миллисекунды — секунды | Секунды — минуты |
| Потребление RAM | Только приложение + libs | ОС + приложение + libs |
| Потребление CPU | Прямое (нет накладных расходов на эмуляцию) | Накладные расходы гипервизора |
| Уровень изоляции | Процессный (namespace + cgroup) | Полная ОС-изоляция |
| Безопасность | Ниже: уязвимость ядра затрагивает все контейнеры | Выше: гостевое ядро изолировано от хоста |
| Портируемость | Высокая (образ = архив файловой системы) | Ниже (зависит от гипервизора) |
| Плотность | Сотни контейнеров на одном хосте | Единицы — десятки VM |

**Что они изолируют по-разному**

- **Файловая система**: VM — отдельный виртуальный диск; контейнер — union filesystem (overlay2) поверх хостовой ФС, с видимостью только своих слоёв.
- **Сеть**: VM — виртуальный NIC с выделенным MAC/IP через гипервизор; контейнер — виртуальный сетевой интерфейс (veth pair) в отдельном network namespace, управляемый Docker (bridge, host, overlay и т.д.).
- **Процессы**: VM видит все процессы внутри своей ОС изолированно; контейнер изолирует через PID namespace — снаружи хост видит процессы контейнера, но контейнер видит только свои.
- **Ядро**: VM — полностью изолировано; контейнер — общее с хостом. Это главный security trade-off: kernel exploit может скомпрометировать все контейнеры на хосте.

**Практический вывод для Java-разработчика**

- Используйте Docker-контейнеры для быстрых dev/test-окружений, микросервисов, CI/CD.
- Используйте VM там, где нужна полная изоляция ОС: разные ядра (Linux + Windows на одном хосте), жёсткие требования безопасности (PCI DSS, HIPAA), legacy-приложения, требующие специфической ОС.
- В production нередко комбинируют: VM обеспечивает ОС-изоляцию и мультитенантность, Docker-контейнеры запускаются внутри VM (подход, используемый в Kubernetes: узел кластера — это VM, поды — контейнеры).

### Вопрос 7. Что такое Pod в Kubernetes? Почему в один Pod иногда помещают несколько контейнеров, и какой паттерн это описывает?

**Категория:** Контейнеры/DevOps · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Pods | Kubernetes](https://kubernetes.io/docs/concepts/workloads/pods/)

Pod — наименьшая развёртываемая единица вычислений в Kubernetes. Pod представляет собой группу из одного или нескольких контейнеров с общими ресурсами хранения и сети, а также спецификацией того, как запускать эти контейнеры. Содержимое Pod всегда размещается совместно, планируется вместе и работает в общем контексте (общие Linux namespaces, cgroups).

Документация выделяет два основных сценария: Pod с одним контейнером (наиболее распространённый) и Pod с несколькими тесно связанными контейнерами, которым необходимо совместно использовать ресурсы. Второй сценарий называется «относительно продвинутым» и рекомендуется только тогда, когда контейнеры действительно плотно связаны. Контейнеры внутри Pod разделяют сетевое пространство имён (один IP-адрес) и могут монтировать общие тома.

#### Ответ от Claude

**Pod** — это атомарная единица планирования в Kubernetes: группа одного или нескольких контейнеров, которые всегда запускаются на одном узле, разделяют один IP-адрес, пространство портов и файловые тома. Pod является эфемерным объектом — его не восстанавливают после сбоя, для этого служат контроллеры (Deployment, StatefulSet и др.).

**Зачем помещать несколько контейнеров в один Pod?**

Несколько контейнеров объединяют в один Pod, когда они тесно связаны функционально и должны совместно использовать ресурсы — общую файловую систему или localhost-сеть. Реплицировать приложение через несколько контейнеров в одном Pod не нужно: для горизонтального масштабирования создают несколько реплик Pod.

**Паттерны многоконтейнерных Pod** (описаны в официальном блоге kubernetes.io «The Distributed System ToolKit: Patterns for Composite Containers»):

1. **Sidecar (Помощник)** — основной контейнер, рядом с которым работает вспомогательный, расширяющий его возможности, не изменяя код основного. Классические примеры: контейнер-синхронизатор файлов, агент для сбора логов, агент мониторинга (Prometheus exporter). Начиная с Kubernetes 1.29, нативные sidecar-контейнеры объявляются в поле `.spec.initContainers` с `restartPolicy: Always`, что даёт им независимый жизненный цикл.

2. **Ambassador (Посол)** — контейнер-прокси, который абстрагирует взаимодействие основного контейнера с внешними сервисами. Например, проксирует соединение к нескольким экземплярам базы данных, предоставляя основному приложению единый localhost-адрес.

3. **Adapter (Адаптер)** — контейнер, который стандартизирует вывод или интерфейс основного контейнера под единый формат. Например, преобразует метрики приложения в формат, ожидаемый системой мониторинга.

Все три паттерна опираются на одно ключевое свойство Pod: контейнеры в нём разделяют localhost-сеть и могут монтировать общие тома, что делает их взаимодействие максимально дешёвым и быстрым.

### Вопрос 8. Чем Deployment отличается от простого Pod в Kubernetes? Что произойдёт с подами при обновлении образа в Deployment?

**Категория:** Контейнеры/DevOps · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Deployments | Kubernetes](https://kubernetes.io/docs/concepts/workloads/controllers/deployment/)

Deployment управляет набором Pod для выполнения рабочей нагрузки приложения, обычно не сохраняющего состояние. Deployment обеспечивает декларативные обновления для Pod и ReplicaSet: вы описываете желаемое состояние, а контроллер Deployment изменяет фактическое состояние до желаемого с контролируемой скоростью.

При изменении образа контейнера создаётся новый ReplicaSet, который постепенно масштабируется вверх, тогда как старый ReplicaSet масштабируется вниз. Новая ревизия создаётся только при изменении `.spec.template` (например, при обновлении образа или меток контейнера). По умолчанию максимальное число недоступных Pod и максимальное число новых Pod, создаваемых одновременно, равно 1. Эти значения настраиваются через `maxUnavailable` и `maxSurge`.

#### Ответ от Claude

**Простой Pod** — это эфемерная запись в кластере: если узел упадёт или Pod завершится с ошибкой, никто его не пересоздаст. Pod не обеспечивает репликацию, не умеет обновлять себя и не контролирует желаемое количество экземпляров.

**Deployment** — это контроллер верхнего уровня, который:
- Декларирует **желаемое состояние** (нужное количество реплик, образ, настройки ресурсов).
- Автоматически создаёт и поддерживает **ReplicaSet**, который в свою очередь следит за нужным числом Pod.
- Пересоздаёт упавшие Pod.
- Выполняет **Rolling Update** при изменении шаблона Pod.
- Хранит историю ревизий и поддерживает **rollback**.

**Что происходит при обновлении образа:**

1. Вы меняете образ (например, `kubectl set image deployment/nginx-deployment nginx=nginx:1.25`), что изменяет поле `.spec.template`.
2. Deployment создаёт **новый ReplicaSet** с обновлённым шаблоном.
3. Новый ReplicaSet начинает создавать новые Pod с новым образом.
4. Kubernetes ждёт, пока новый Pod перейдёт в состояние Ready.
5. Только после этого из старого ReplicaSet удаляется один Pod.
6. Процесс повторяется итерационно, пока все Pod не будут заменены.

По умолчанию `maxUnavailable: 25%` и `maxSurge: 25%` (начиная с актуальных версий Kubernetes). Это означает, что одновременно можно иметь не более 25% Pod сверх желаемого числа и не более 25% недоступных Pod. Параметры задаются числом или процентом в `.spec.strategy.rollingUpdate`.

Если обновление проходит неудачно (Pod не переходит в Ready), Deployment приостанавливается. Можно выполнить откат: `kubectl rollout undo deployment/nginx-deployment`.

**Ключевое отличие архитектуры:** Pod — это конкретный экземпляр; Deployment — декларация намерения, которую кластер непрерывно реализует.

### Вопрос 9. Что такое Service в Kubernetes? Объясните разницу между типами ClusterIP, NodePort и LoadBalancer.

**Категория:** Контейнеры/DevOps · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Service | Kubernetes](https://kubernetes.io/docs/concepts/services-networking/service/)

Service в Kubernetes — это метод предоставления сетевого доступа к приложению, работающему в виде одного или нескольких Pod. Service API является абстракцией, которая помогает предоставлять группы Pod через сеть. Каждый объект Service определяет логический набор конечных точек (endpoint) — обычно это Pod — вместе с политикой доступа к ним.

Service решает проблему динамической природы Pod: поскольку Pod создаются и удаляются динамически, Service предоставляет стабильный адрес. Документация определяет следующие типы:

- **ClusterIP** (по умолчанию) — предоставляет внутренний IP кластера, доступный только внутри кластера.
- **NodePort** — открывает Service на статическом порту каждого узла (диапазон 30000–32767); доступен извне по адресу `<NodeIP>:<NodePort>`.
- **LoadBalancer** — интегрируется с облачным провайдером для создания внешнего балансировщика нагрузки с выделенным внешним IP.

Типы спроектированы как вложенная функциональность: каждый следующий уровень добавляет возможности предыдущего.

#### Ответ от Claude

**Service** — это постоянная сетевая абстракция над динамически создаваемыми Pod. Поскольку у Pod нет постоянного IP (при пересоздании он меняется), Service предоставляет стабильный виртуальный IP (ClusterIP) и DNS-имя, за которым стоит группа Pod, выбираемых по `selector` (labels). Трафик распределяется между Pod через механизм kube-proxy (iptables или IPVS).

**Типы Service:**

**ClusterIP** (тип по умолчанию)
- Создаёт виртуальный IP, доступный только внутри кластера.
- Используется для взаимодействия между сервисами внутри кластера (например, бэкенд обращается к базе данных).
- Недоступен снаружи кластера напрямую.
- Пример: микросервис A вызывает микросервис B по имени `http://service-b:8080`.

**NodePort**
- Надстройка над ClusterIP: дополнительно открывает один порт на каждом узле кластера.
- Порт выбирается из диапазона 30000–32767 (настраивается флагом `--service-node-port-range`).
- Доступен снаружи по адресу `<IP-любого-узла>:<NodePort>`.
- Применяется в средах без облачного провайдера (bare metal, локальная разработка) или для отладки. В продакшене используется редко из-за ограниченного диапазона портов и зависимости от IP узлов.

**LoadBalancer**
- Надстройка над NodePort: дополнительно запрашивает у облачного провайдера (AWS, GCP, Azure и др.) внешний балансировщик нагрузки.
- Балансировщик получает стабильный внешний IP и перенаправляет трафик на NodePort всех узлов.
- Создание балансировщика происходит асинхронно; внешний IP появляется в поле `.status.loadBalancer.ingress`.
- Основной способ открыть сервис наружу в облачных кластерах. На bare metal требует дополнительного инструмента (например, MetalLB).

**Иерархия вложенности:** LoadBalancer включает NodePort, который включает ClusterIP. Это означает, что LoadBalancer-сервис также доступен по ClusterIP внутри кластера и по NodePort на каждом узле.

**Дополнительный тип ExternalName** (для справки): перенаправляет трафик на внешнее DNS-имя через CNAME-запись, без использования селекторов и Pod.
### Вопрос 10. В чём разница между ConfigMap и Secret в Kubernetes? Как их можно передать в контейнер — через переменные окружения или монтирование файла?

**Категория:** Контейнеры/DevOps · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [ConfigMaps | Kubernetes](https://kubernetes.io/docs/concepts/configuration/configmap/)

ConfigMap — объект Kubernetes для хранения **неконфиденциальных** данных конфигурации в виде пар ключ-значение. ConfigMap не обеспечивает секретности или шифрования. Если данные, которые нужно хранить, являются конфиденциальными, следует использовать Secret, а не ConfigMap, или применять дополнительные сторонние инструменты для обеспечения конфиденциальности.

Данные ConfigMap не могут превышать 1 МиБ. Оба объекта — ConfigMap и Secret — можно передать в контейнер двумя способами: через переменные окружения и через монтирование тома (volume mount). При использовании переменных окружения обновления ConfigMap **не применяются автоматически** — требуется перезапуск Pod-а. При монтировании тома kubelet периодически проверяет актуальность значений и обновляет их без перезапуска.

#### Ответ от Claude

**Назначение и ключевые отличия**

| Характеристика | ConfigMap | Secret |
|---|---|---|
| Тип данных | Неконфиденциальные | Конфиденциальные (пароли, токены, ключи) |
| Шифрование | Нет (plain text в etcd) | base64-кодирование по умолчанию; шифрование at rest — опционально |
| Лимит размера | 1 МиБ | Нет жёсткого лимита, но рекомендуется хранить небольшие данные |
| Типы | Нет специальных типов | Opaque, tls, dockerconfigjson, basic-auth, ssh-auth и др. |
| Хранилище | etcd, plain text | etcd; по умолчанию unencrypted — важно включать Encryption at Rest |

**Важно:** base64 — это кодирование, а не шифрование. Значение легко декодировать командой `echo "..." | base64 -d`. Для настоящей защиты необходимо включить Encryption at Rest или использовать внешние хранилища (HashiCorp Vault, AWS Secrets Manager).

---

**Способ 1: Переменные окружения (env)**

Для ConfigMap:
```yaml
env:
  - name: APP_ENV
    valueFrom:
      configMapKeyRef:
        name: app-config
        key: environment
```

Для Secret:
```yaml
env:
  - name: DB_PASSWORD
    valueFrom:
      secretKeyRef:
        name: db-credentials
        key: password
```

Также можно подтянуть **все** ключи из ConfigMap/Secret одной директивой `envFrom`:
```yaml
envFrom:
  - configMapRef:
      name: app-config
  - secretRef:
      name: db-credentials
```

Ограничение: переменные окружения **не обновляются** при изменении ConfigMap/Secret — нужен перезапуск Pod-а.

---

**Способ 2: Монтирование тома (volumeMount)**

Для ConfigMap:
```yaml
spec:
  volumes:
    - name: config-volume
      configMap:
        name: app-config
  containers:
    - name: app
      volumeMounts:
        - name: config-volume
          mountPath: /etc/config
          readOnly: true
```

Каждый ключ ConfigMap становится отдельным файлом в `/etc/config/`. Kubelet периодически синхронизирует изменения — Pod **не нужно перезапускать**.

Для Secret:
```yaml
spec:
  volumes:
    - name: secret-volume
      secret:
        secretName: db-credentials
  containers:
    - name: app
      volumeMounts:
        - name: secret-volume
          mountPath: /etc/secrets
          readOnly: true
```

Secret-тома хранятся в tmpfs (RAM-backed filesystem) — данные **никогда не записываются на диск**.

---

**Когда что использовать**

- ConfigMap: URL базы данных, имена топиков Kafka, параметры логирования, feature-флаги
- Secret: пароли, токены API, приватные ключи TLS, credentials для Docker Registry

### Вопрос 11. Что такое liveness probe и readiness probe в Kubernetes? Что произойдёт с Pod-ом, если liveness probe перестанет отвечать? Как это влияет на трафик через Service?

**Категория:** Контейнеры/DevOps · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Liveness, Readiness, and Startup Probes | Kubernetes](https://kubernetes.io/docs/concepts/configuration/liveness-readiness-startup-probes/)

**Liveness probe** определяет, когда нужно перезапустить контейнер. Если контейнер провалит liveness probe больше раз, чем разрешено настройкой `failureThreshold`, kubelet **убивает и перезапускает** контейнер.

**Readiness probe** определяет, когда контейнер готов принимать трафик. Когда readiness probe переходит в состояние Failure, контроллер EndpointSlice **удаляет IP-адрес Pod-а из EndpointSlice всех Service**, которые соответствуют этому Pod-у. Pod остаётся запущенным, но перестаёт получать трафик.

Оба зонда можно использовать одновременно: readiness гарантирует, что трафик не достигнет неготового контейнера, а liveness обеспечивает перезапуск при зависании.

#### Ответ от Claude

**Liveness Probe**

Цель: обнаружить, что приложение «застряло» (deadlock, утечка памяти без краша, бесконечный цикл) — работает, но прогресса нет.

Что происходит при провале:
1. Kubelet считает количество последовательных неудач.
2. После превышения `failureThreshold` (по умолчанию 3) kubelet **убивает контейнер** и перезапускает его согласно `restartPolicy` (All/OnFailure).
3. Если Pod продолжает падать — применяется экспоненциальная выдержка перезапуска (до 5 минут), счётчик `RESTARTS` в `kubectl get pods` растёт.
4. На трафик через Service это **прямого влияния не оказывает** в момент отказа — Pod исключается из эндпоинтов лишь если одновременно не проходит и readiness probe.

Пример (HTTP GET):
```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
  initialDelaySeconds: 10
  periodSeconds: 5
  timeoutSeconds: 3
  failureThreshold: 3
```

---

**Readiness Probe**

Цель: сигнализировать о временной неготовности — прогрев кэша, ожидание соединения с БД, высокая нагрузка.

Что происходит при провале:
1. Контроллер EndpointSlice **удаляет IP Pod-а** из эндпоинтов всех Service, которые его выбирают.
2. Service перестаёт направлять трафик на этот Pod (kube-proxy обновляет правила iptables/IPVS).
3. Контейнер **не перезапускается** — он продолжает работу.
4. Как только readiness probe снова успешна — Pod возвращается в ротацию.

Пример (TCP):
```yaml
readinessProbe:
  tcpSocket:
    port: 8080
  initialDelaySeconds: 15
  periodSeconds: 10
```

---

**Сравнительная таблица**

| Характеристика | Liveness Probe | Readiness Probe |
|---|---|---|
| Действие при провале | Перезапуск контейнера | Исключение из эндпоинтов Service |
| Pod продолжает работать? | Нет (убивается) | Да |
| Влияние на трафик | Косвенное (через restartPolicy) | Прямое (IP удаляется из EndpointSlice) |
| Когда применять | Deadlock, зависание | Прогрев, временная недоступность |

---

**Типы зондов** (применимы к обоим):
- `httpGet` — HTTP GET-запрос; успех: статус 200–399
- `tcpSocket` — попытка открыть TCP-соединение
- `exec` — выполнение команды внутри контейнера; успех: exit code 0
- `grpc` — gRPC Health Checking Protocol (с Kubernetes 1.24+)

---

**Важные предупреждения:**
- Liveness probe должен быть **простым и быстрым**. Если probe сам зависает под нагрузкой — возникает каскадный перезапуск всех Pod-ов.
- Не делайте liveness probe зависимым от внешних сервисов (БД, Kafka): их недоступность не означает, что приложение нужно перезапускать.
- Для медленно стартующих приложений используйте **startup probe** — он блокирует запуск liveness/readiness до успешной инициализации.
- `initialDelaySeconds` должен покрывать время старта приложения, иначе Pod войдёт в цикл перезапусков ещё до готовности.

### Вопрос 12. Как правильно задать .dockerignore и почему это важно? Какие файлы Java-проекта точно не должны попасть в образ?

**Категория:** Контейнеры/DevOps · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Building best practices | Docker Docs](https://docs.docker.com/build/building/best-practices/)

Файл `.dockerignore` позволяет исключить файлы и директории, которые не нужны в контексте сборки Docker. Это работает аналогично `.gitignore`. Исключение ненужных файлов уменьшает размер build context, что напрямую влияет на скорость передачи контекста сборщику (особенно при использовании удалённого builder-а). Кроме того, это предотвращает случайное попадание секретов и чувствительных данных в образ. В официальном руководстве по контейнеризации Java-приложений Docker рекомендует исключать: `**/.git`, `**/target`, `**/.classpath`, `**/.project`, `**/.settings`, `**/docker-compose*`, `**/Dockerfile*`, `**/node_modules`, `**/.env`, `**/.vscode`, `**/*.log`.

#### Ответ от Claude

**Что такое .dockerignore и как он работает**

Файл `.dockerignore` размещается в корне build context (рядом с `Dockerfile`) и перечисляет паттерны файлов/директорий, которые **не включаются** в build context при выполнении `docker build`. Это аналог `.gitignore` с такой же синтаксической системой паттернов (glob, `**`, исключения через `!`).

Когда Docker-клиент выполняет `docker build`, он сначала собирает build context — архив всех файлов из текущей директории — и отправляет его daemon-у или удалённому builder-у. Чем меньше context, тем быстрее сборка и меньше риск утечки данных.

---

**Почему это важно**

1. **Производительность**: директория `target/` Maven-проекта может весить сотни мегабайт (скомпилированные классы, JAR-ы, Surefire-отчёты). Директория `.git/` — гигабайты истории. Исключение этих папок может сократить контекст на 95–99% и ускорить передачу в десятки раз.

2. **Стабильность кэша слоёв**: если в образ попадают файлы, которые часто меняются (логи, артефакты сборки), Docker инвалидирует кэш слоёв при каждой сборке, даже если исходный код не изменился.

3. **Безопасность**: случайное включение `.env`, `application-local.properties`, `secrets.yaml`, приватных ключей или `.m2/repository` с credentials может привести к утечке секретов в образ и Docker Registry.

4. **Предотвращение конфликтов**: если `target/` попадает в образ через `COPY . .`, может нарушиться воспроизводимость сборки — в образ попадут артефакты локальной компиляции вместо чистых.

---

**Типичный .dockerignore для Java/Maven проекта**

```
# Система контроля версий
.git
.gitignore
.gitattributes

# Артефакты сборки Maven/Gradle
target/
build/
out/
*.class
*.jar
*.war
*.ear

# IDE-файлы
.idea/
*.iml
.classpath
.project
.settings/
.vscode/
*.swp

# Конфигурации окружения (секреты!)
.env
.env.*
application-local.properties
application-local.yml
secrets.yaml

# Локальный репозиторий Maven (если присутствует)
.m2/

# Тесты (если не нужны в образе)
src/test/

# Docker-файлы (необязательно исключать, но рекомендуется)
Dockerfile
docker-compose*.yml
.dockerignore

# Логи и временные файлы
*.log
logs/
tmp/

# Документация
README.md
CHANGELOG.md
LICENSE
docs/
```

---

**Файлы Java-проекта, которые точно не должны попасть в образ**

| Файл/директория | Причина |
|---|---|
| `target/` | Артефакты Maven; могут весить сотни МБ; в образе нужен только финальный JAR |
| `build/` | Аналог `target/` для Gradle |
| `.git/` | История репозитория; гигабайты ненужных данных |
| `.m2/` | Локальный кэш Maven-зависимостей; может содержать credentials |
| `.env` / `*.properties` с секретами | Прямая утечка паролей и ключей в образ |
| `src/test/` | Тестовый код не нужен в production-образе |
| `.idea/`, `*.iml`, `.classpath` | IDE-метаданные; не влияют на работу приложения |
| `docker-compose*.yml` | Конфигурация оркестрации; не нужна внутри образа |
| `*.log`, `logs/` | Логи разработки; не должны попадать в образ |

---

**Синтаксис паттернов**

- `target/` — исключить директорию `target` в корне
- `**/target/` — исключить `target` на любой глубине (важно для mono-repo)
- `*.log` — все файлы `.log` в корне
- `**/*.log` — все файлы `.log` рекурсивно
- `!src/main/resources/application.properties` — исключение из исключений (включить файл обратно)

---

**Стратегия для multi-stage builds**

При использовании multi-stage Dockerfile `.dockerignore` особенно важен для первой стадии (builder). Если используется паттерн:
```dockerfile
FROM maven:3.9-eclipse-temurin-21 AS build
COPY . .
RUN mvn package -DskipTests

FROM eclipse-temurin:21-jre
COPY --from=build /app/target/app.jar /app/app.jar
```
...то `.git/`, `*.md`, IDE-файлы и `target/` (прошлой сборки) должны быть исключены, чтобы инструкция `COPY . .` не засоряла кэш и не копировала лишнее.

### Вопрос 13. Как настроить JVM в контейнере, чтобы она корректно определяла лимиты памяти (cgroup)? Какие флаги JVM используются начиная с Java 10?

**Категория:** Контейнеры/DevOps · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [JVM in Linux containers, surviving the isolation | BellSoft Java](https://bell-sw.com/announcements/2020/10/28/JVM-in-Linux-containers-surviving-the-isolation/)

Начиная с OpenJDK 10 (и в бэкпорте для 8u191+) JVM умеет читать cgroup-файлы для определения лимита памяти контейнера вместо общей памяти хоста. Это управляется флагом `-XX:+UseContainerSupport` (включён по умолчанию). На его основе вычисляется размер кучи через флаг `-XX:MaxRAMPercentage` (по умолчанию 25%), который задаёт максимальный хип как процент от доступной памяти контейнера. Также существуют `-XX:InitialRAMPercentage` (по умолчанию 1.5625%) и `-XX:MinRAMPercentage` (по умолчанию 1.875%). Важно понимать, что эти флаги управляют только размером кучи; дополнительную память потребляют стеки потоков, Metaspace, direct memory и code cache.

#### Ответ от Claude

**Проблема до Java 10**

До Java 8u131 / Java 9 JVM не знала о cgroup-ограничениях. Она «видела» всю оперативную память хоста и могла выделить кучу, превышающую лимит контейнера — это приводило к убийству процесса ядром (OOM Killer).

Первая попытка — экспериментальный флаг `-XX:+UseCGroupMemoryLimitForHeap` (Java 8u131, Java 9). Он требовал явной активации и работал нестабильно.

**Начиная с Java 10 (и бэкпорт в Java 8u191)**

Флаг `-XX:+UseContainerSupport` стал основным решением и **включён по умолчанию** в OpenJDK 10+, а также в OpenJDK 8u191+ (в 8u191–8u242 нужна явная активация; с 8u251 — включён по умолчанию). Флаг `UseCGroupMemoryLimitForHeap` помечен устаревшим.

Механизм работы: JVM читает файл `/sys/fs/cgroup/memory/memory.limit_in_bytes` (cgroup v1) или `/sys/fs/cgroup/memory.max` (cgroup v2) и использует полученное значение вместо объёма RAM хоста.

**Основные флаги управления памятью**

| Флаг | Назначение | Значение по умолчанию |
|------|-----------|----------------------|
| `-XX:+UseContainerSupport` | Включить чтение cgroup-лимитов | `true` (Java 10+, 8u191+) |
| `-XX:MaxRAMPercentage=N` | Максимальный хип как % от лимита контейнера | `25.0` |
| `-XX:InitialRAMPercentage=N` | Начальный хип как % от лимита контейнера | `1.5625` |
| `-XX:MinRAMPercentage=N` | Применяется когда общая память < 200 МБ | `50.0` (для малых контейнеров) |

**Типичная практика для production-контейнеров:**

```bash
java -XX:+UseContainerSupport \
     -XX:MaxRAMPercentage=75.0 \
     -XX:InitialRAMPercentage=50.0 \
     -jar app.jar
```

`MaxRAMPercentage=75` — разумное значение для контейнеров, целиком отданных под одно Java-приложение: 25% остаётся на нехиповые области (Metaspace, стеки потоков, direct memory, JIT code cache).

**Проверка**

```bash
# Посмотреть эффективные флаги и определённые лимиты
docker run --memory=512m openjdk:17 java \
  -XX:+PrintFlagsFinal -version 2>&1 | grep -E 'MaxHeapSize|UseContainerSupport'
```

**Итог:** начиная с Java 10 основной инструмент — `-XX:+UseContainerSupport` (включён по умолчанию) в паре с `-XX:MaxRAMPercentage` (рекомендуется выставлять явно, т.к. умолчание 25% слишком мало). Старый флаг `UseCGroupMemoryLimitForHeap` устарел и не должен использоваться в новых проектах.

### Вопрос 14. Что такое ReplicaSet в Kubernetes и как Deployment управляет им при rolling update? Как откатить деплой к предыдущей версии?

**Категория:** Контейнеры/DevOps · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Deployments | Kubernetes](https://kubernetes.io/docs/concepts/workloads/controllers/deployment/)

ReplicaSet гарантирует, что заданное количество реплик Pod'а работает в любой момент времени. При обновлении Deployment создаёт новый ReplicaSet и постепенно масштабирует его вверх, одновременно уменьшая старый — это и есть rolling update. Откат выполняется командами:

```bash
# Просмотр истории
kubectl rollout history deployment/nginx-deployment

# Откат к предыдущей ревизии
kubectl rollout undo deployment/nginx-deployment

# Откат к конкретной ревизии
kubectl rollout undo deployment/nginx-deployment --to-revision=2
```

Поле `revisionHistoryLimit` (по умолчанию 10) определяет, сколько старых ReplicaSet хранится для возможности отката.

#### Ответ от Claude

**ReplicaSet**

ReplicaSet — контроллер Kubernetes, задача которого: поддерживать заданное количество идентичных Pod'ов запущенными. Он непрерывно сравнивает текущее состояние с желаемым (`.spec.replicas`) и создаёт или удаляет Pod'ы. ReplicaSet является нижележащим примитивом; напрямую с ним работать не рекомендуется — для этого предназначен Deployment.

Имя ReplicaSet строится по шаблону `<deployment-name>-<pod-template-hash>`.

**Как Deployment управляет ReplicaSet при rolling update**

При изменении `.spec.template` (например, новый образ) Deployment:

1. Создаёт **новый ReplicaSet** с хешем нового pod template.
2. Постепенно увеличивает количество реплик в новом ReplicaSet.
3. Одновременно уменьшает количество реплик в старом ReplicaSet.
4. Фиксирует новую **ревизию** (`REVISION`).

Скорость процесса контролируется стратегией:

```yaml
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # максимум дополнительных Pod'ов сверх desired
      maxUnavailable: 0  # максимум недоступных Pod'ов в процессе обновления
```

- `maxSurge` и `maxUnavailable` могут быть числами или процентами (например, `25%`).
- Оба не могут быть равны 0 одновременно.
- По умолчанию оба равны `25%`.

Дополнительный параметр `minReadySeconds` задаёт время (в секундах), которое Pod должен быть в статусе Ready перед тем, как Deployment продолжит обновление.

**Откат деплоя**

Старые ReplicaSet не удаляются сразу — они сохраняются в количестве, определённом `.spec.revisionHistoryLimit` (по умолчанию 10). Это позволяет быстро откатиться.

```bash
# Проверить статус текущего обновления
kubectl rollout status deployment/my-app

# Посмотреть историю ревизий
kubectl rollout history deployment/my-app

# Посмотреть детали конкретной ревизии
kubectl rollout history deployment/my-app --revision=3

# Откатить к предыдущей ревизии
kubectl rollout undo deployment/my-app

# Откатить к конкретной ревизии
kubectl rollout undo deployment/my-app --to-revision=2

# Приостановить обновление
kubectl rollout pause deployment/my-app

# Возобновить обновление
kubectl rollout resume deployment/my-app
```

После отката Deployment меняет местами текущий и целевой ReplicaSet: старый становится «новым» и масштабируется вверх по той же схеме rolling update.

**Важный нюанс:** при каждом `rollout undo` номер ревизии **инкрементируется**, а не уменьшается. Откат к ревизии 2 создаёт ревизию 4 (если была 3) с тем же pod template, что был в ревизии 2.

### Вопрос 15. Назовите несколько принципов методологии 12-факторного приложения и объясните, как они отражаются в конфигурации Kubernetes (например, хранение конфигурации в среде окружения, stateless-процессы).

**Категория:** Контейнеры/DevOps · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [How do you build 12-factor apps using Kubernetes? | Mirantis](https://www.mirantis.com/blog/how-do-you-build-12-factor-apps-using-kubernetes/)

Методология 12-factor app была сформулирована инженерами Heroku для построения масштабируемых SaaS-приложений. Kubernetes напрямую реализует многие её принципы:

- **Config (фактор III):** конфигурация хранится в переменных окружения, а не в коде. Kubernetes реализует это через `ConfigMap` (для несекретных данных) и `Secret` (для чувствительных данных), которые монтируются в Pod как env-переменные.
- **Processes (фактор VI):** приложение выполняется как набор stateless процессов, не хранящих состояние между запросами. Kubernetes Deployment управляет такими взаимозаменяемыми Pod'ами.
- **Concurrency (фактор VIII):** масштабирование достигается запуском большего числа процессов. В Kubernetes это реализуется через `kubectl scale` или Horizontal Pod Autoscaler (HPA).
- **Logs (фактор XI):** приложение пишет логи в stdout/stderr, не занимаясь их маршрутизацией. Kubernetes агрегирует логи контейнеров через `kubectl logs` и передаёт в системы сбора логов (Fluentd, Loki и др.).

#### Ответ от Claude

Методология 12-factor app — набор принципов для построения современных cloud-native приложений, сформулированных командой Heroku. Kubernetes реализует большинство из них на платформенном уровне.

**Все 12 факторов:**

1. Codebase — одна кодовая база в VCS, много деплоев
2. Dependencies — явное объявление зависимостей
3. Config — конфигурация в переменных окружения
4. Backing Services — внешние сервисы как подключаемые ресурсы
5. Build, Release, Run — строгое разделение стадий сборки, релиза и запуска
6. Processes — stateless процессы без разделяемого состояния
7. Port Binding — экспорт сервиса через биндинг порта
8. Concurrency — масштабирование через запуск дополнительных процессов
9. Disposability — быстрый старт и корректное завершение
10. Dev/Prod Parity — минимальные различия между окружениями
11. Logs — логи как потоки событий
12. Admin Processes — административные задачи как разовые процессы

**Ключевые принципы и их реализация в Kubernetes**

**Фактор III — Config (конфигурация в окружении)**

12-factor требует: не хранить конфигурацию в коде; всё, что меняется между деплоями (dev/staging/prod), должно передаваться через переменные окружения.

Kubernetes-реализация:
- `ConfigMap` — хранение несекретной конфигурации (URLs, флаги, настройки)
- `Secret` — хранение чувствительных данных (пароли, токены, ключи)
- Оба ресурса монтируются в Pod как env-переменные или файлы:

```yaml
env:
  - name: DATABASE_URL
    valueFrom:
      configMapKeyRef:
        name: app-config
        key: database_url
  - name: DB_PASSWORD
    valueFrom:
      secretKeyRef:
        name: app-secret
        key: password
```

**Фактор VI — Processes (stateless-процессы)**

12-factor требует: никакого sticky state в процессе. Сессионные данные, кэш — во внешние backing services (Redis, БД).

Kubernetes-реализация: Deployment управляет взаимозаменяемыми Pod'ами. Любой Pod может быть убит и пересоздан на другом узле без потери данных. `PersistentVolume` используется только для реально персистентных данных (базы данных), а не для состояния приложения.

**Фактор VIII — Concurrency (конкурентность через процессы)**

12-factor требует: горизонтальное масштабирование через добавление процессов, а не вертикальное.

Kubernetes-реализация:
- `kubectl scale deployment/my-app --replicas=10`
- `HorizontalPodAutoscaler` — автоматическое масштабирование по метрикам CPU/памяти/custom metrics

**Фактор IX — Disposability (утилизируемость)**

12-factor требует: быстрый старт (секунды) и корректное завершение (graceful shutdown) по SIGTERM.

Kubernetes-реализация:
- `readinessProbe` — Pod не получает трафик, пока не готов
- `livenessProbe` — Pod перезапускается при зависании
- `preStop` hook + `terminationGracePeriodSeconds` — время на корректное завершение

**Фактор XI — Logs (логи как потоки)**

12-factor требует: приложение пишет в stdout/stderr и не управляет файлами логов.

Kubernetes-реализация: контейнер пишет в stdout → kubelet захватывает → `kubectl logs` читает → внешние агрегаторы (Fluentd, Fluent Bit, Loki) собирают и хранят.

**Фактор X — Dev/Prod Parity (паритет окружений)**

Kubernetes-реализация: один и тот же манифест (Helm chart или Kustomize overlay) деплоится в dev/staging/prod, различаясь только значениями ConfigMap и количеством реплик. Docker-образ один и тот же для всех окружений — меняется только конфигурация.
### Вопрос 16. Как выглядит типичный CI/CD-конвейер для Java-приложения с Docker и Kubernetes? Опишите шаги от git push до обновления Pod-ов в кластере.

**Категория:** Контейнеры/DevOps · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ArgoCD + GitHub Actions: A Complete GitOps CI/CD Workflow for Kubernetes Applications (Medium)](https://medium.com/@mehmetkanus17/argocd-github-actions-a-complete-gitops-ci-cd-workflow-for-kubernetes-applications-ed2f91d37641)

Статья описывает полный цикл GitOps-пайплайна для Kubernetes:

1. **Триггер (Code Push)** — разработчик пушит код в ветку `main`. GitHub Actions автоматически запускает workflow.
2. **Сборка Docker-образа** — workflow логинится в GitHub Container Registry (GHCR) и собирает образ через Docker Buildx.
3. **Публикация в реестр** — образ пушится в GHCR с двумя тегами: `latest` и SHA коммита для трассировки версий.
4. **Обновление GitOps-репозитория** — workflow клонирует отдельный GitOps-репозиторий и обновляет в нём файл `deployment.yaml`, заменяя тег образа на новый.
5. **Коммит манифеста** — изменённый манифест коммитится и пушится обратно в GitOps-репозиторий.
6. **Обнаружение изменений ArgoCD** — ArgoCD непрерывно отслеживает GitOps-репозиторий и обнаруживает изменения в течение интервала reconciliation (по умолчанию 180 секунд, при оптимизации — 15 секунд).
7. **Синхронизация с кластером** — ArgoCD сверяет желаемое состояние (Git) с текущим состоянием кластера и применяет обновлённый Deployment.
8. **Rolling update Pod-ов** — Kubernetes запускает постепенную замену старых Pod-ов новыми с обновлённым образом, обеспечивая zero-downtime.

#### Ответ от Claude

Типичный CI/CD-конвейер для Java-приложения с Docker и Kubernetes состоит из двух логических частей: **CI (Continuous Integration)** — от коммита до публикации образа, и **CD (Continuous Delivery/Deployment)** — от образа до обновления Pod-ов в кластере.

**Шаг 1. Git push и триггер**

Разработчик выполняет `git push` в основную ветку (или создаёт Pull Request). CI-система (Jenkins, GitHub Actions, GitLab CI, TeamCity) получает уведомление через webhook и запускает пайплайн.

**Шаг 2. Сборка и тестирование**

```
mvn clean verify
# или
./gradlew build test
```

Включает компиляцию исходников, запуск unit-тестов, интеграционных тестов, статический анализ кода (SonarQube, Checkstyle), проверку покрытия. Если тесты не прошли — пайплайн прерывается.

**Шаг 3. Сборка Docker-образа**

```dockerfile
docker build -t my-registry/my-app:${GIT_COMMIT_SHA} .
```

Образ тегируется SHA коммита (а не `latest`) для однозначной трассировки версий. Опционально — сканирование образа на уязвимости (Trivy, Snyk, Clair).

**Шаг 4. Публикация образа в Container Registry**

```bash
docker push my-registry/my-app:${GIT_COMMIT_SHA}
```

Реестр: Docker Hub, GitHub Container Registry, Amazon ECR, Google Artifact Registry, Harbor (self-hosted). Credentials хранятся в секретах CI-системы.

**Шаг 5. Обновление манифестов (классический подход)**

CI напрямую применяет манифест с новым тегом:

```bash
kubectl set image deployment/my-app \
  my-app=my-registry/my-app:${GIT_COMMIT_SHA}
# или
kubectl apply -f k8s/deployment.yaml
```

**Шаг 5a. Обновление манифестов (GitOps-подход)**

В GitOps (рекомендуемый современный подход) CI не обращается к кластеру напрямую. Вместо этого:
- CI обновляет файл `deployment.yaml` в отдельном GitOps-репозитории, заменяя тег образа.
- Изменение коммитится и пушится в GitOps-репозиторий.
- ArgoCD или Flux CD, непрерывно отслеживающие этот репозиторий, обнаруживают расхождение между желаемым состоянием (Git) и текущим состоянием кластера.
- GitOps-оператор применяет манифест в кластер. Кластер не требует входящих credentials из CI — ArgoCD работает изнутри кластера.

**Шаг 6. Rolling Update в Kubernetes**

Kubernetes выполняет rolling update согласно стратегии Deployment:

```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1        # Сколько новых Pod-ов можно добавить сверх желаемого числа
    maxUnavailable: 0  # Сколько Pod-ов может быть недоступно в процессе обновления
```

Kubernetes поочерёдно создаёт новые Pod-ы с обновлённым образом, ждёт их готовности (readiness probe), затем завершает старые Pod-ы. При ошибке — автоматический rollback.

**Шаг 7. Проверка здоровья**

После обновления Kubernetes проверяет `readinessProbe` и `livenessProbe`. При неуспешных пробах новый Pod не включается в ротацию Service, старые Pod-ы остаются живыми.

**Итоговая схема конвейера:**

```
git push
    │
    ▼
[Webhook] CI-система
    │
    ▼
mvn/gradle build + тесты
    │
    ▼
docker build + scan (Trivy)
    │
    ▼
docker push → Container Registry
    │
    ├─ (Классический CD) ──► kubectl apply → Kubernetes API
    │
    └─ (GitOps CD) ──► update deployment.yaml → Git commit/push
                              │
                              ▼
                        ArgoCD/Flux CD обнаруживает diff
                              │
                              ▼
                        kubectl apply (изнутри кластера)
                              │
                              ▼
                        Rolling Update Pod-ов
```

**Ключевые инструменты:**
- CI: Jenkins, GitHub Actions, GitLab CI, CircleCI
- Сборка: Maven, Gradle
- Реестр: Docker Hub, ECR, GCR, Harbor
- CD классический: kubectl, Helm
- CD GitOps: ArgoCD, Flux CD, Kustomize
- Безопасность образов: Trivy, Snyk Container

```dockerfile
FROM bellsoft/liberica-runtime-container:jdk-21-stream-musl AS builder
WORKDIR /home/app
ADD spring-petclinic-main /home/app/spring-petclinic-main

```

**3. JRE вместо JDK** — JRE занимает ~50 МБ, полный JDK Lite — 85 МБ, полный JDK — 180 МБ. В production JDK не нужен.

**4. Custom JRE через jlink** — минимальный runtime только с нужными Java-модулями, итоговый размер ~40 МБ:

```dockerfile
RUN jlink --add-modules java.base --compress=2 --strip-debug --output /opt/customjdk
```

**5. Thin JAR / разделение слоёв** — зависимости в отдельном слое Docker (меняются редко), код приложения в отдельном (меняется часто).

#### Ответ от Claude

Существует несколько категорий приёмов уменьшения размера Docker-образа для Spring Boot приложений. Ниже — наиболее эффективные, проверенные по нескольким источникам (docs.spring.io, bell-sw.com, baeldung.com).

---

**Приём 1. Multi-stage builds (многоэтапная сборка)**

Ключевая техника: сборочный контейнер содержит JDK, Maven/Gradle и все инструменты сборки. В финальный образ копируется только артефакт. Без multi-stage итоговый образ включает весь build-toolchain (~450–600 МБ); с multi-stage — только JRE + JAR (~100–300 МБ в зависимости от базового образа).

```dockerfile
# Стадия 1: сборка
FROM eclipse-temurin:21-jdk AS builder
WORKDIR /app
COPY pom.xml .
COPY src ./src
RUN mvn clean package -DskipTests

# Стадия 2: runtime (только JRE)
FROM eclipse-temurin:21-jre-alpine
WORKDIR /app
COPY --from=builder /app/target/*.jar app.jar
ENTRYPOINT ["java", "-jar", "app.jar"]
```

---

**Приём 2. Лёгкий базовый образ: Alpine или Distroless**

*Alpine* — минималистичный Linux (около 5 МБ), содержит пакетный менеджер `apk` и базовые утилиты. `eclipse-temurin:21-jre-alpine` даёт образ порядка 180–220 МБ.

*Distroless* (Google) — образ содержит только приложение и необходимые runtime-библиотеки. Отсутствуют shell, пакетный менеджер, утилиты. `gcr.io/distroless/java21-debian12` — ~192 МБ базового образа. Итоговый образ с приложением ~200–210 МБ.

Сравнение:
- `openjdk:21` (full JDK, Debian): ~450–600 МБ
- `eclipse-temurin:21-jre-alpine`: ~180–220 МБ
- `gcr.io/distroless/java21-debian12`: ~200–210 МБ
- Alpaquita Linux + Liberica JRE (musl): ~52 МБ

Distroless предпочтителен с точки зрения безопасности (нет shell — нет вектора атаки через `docker exec`/`kubectl exec`). Alpine предпочтителен по размеру, но требует тестирования совместимости (musl libc vs glibc).

Пример с distroless:

```dockerfile
FROM eclipse-temurin:21-jdk AS builder
WORKDIR /app
COPY . .
RUN mvn clean package -DskipTests

FROM gcr.io/distroless/java21-debian12
WORKDIR /app
COPY --from=builder /app/target/*.jar app.jar
ENTRYPOINT ["java", "-jar", "app.jar"]
```

---

**Приём 3. Spring Boot Layered JAR (расслоённый JAR)**

Начиная с Spring Boot 2.3, плагин Spring Boot поддерживает расслоённые JAR (`layered JAR`). Содержимое JAR разбивается на слои по частоте изменений:

1. `dependencies` — сторонние библиотеки стабильных версий (меняются редко)
2. `spring-boot-loader` — загрузчик Spring Boot
3. `snapshot-dependencies` — SNAPSHOT-зависимости
4. `application` — классы приложения (меняются при каждом деплое)

Каждый слой — отдельный Docker layer. При изменении только кода приложения Docker пересобирает только слой `application` (несколько сотен КБ вместо нескольких десятков МБ). Это значительно ускоряет сборку и уменьшает объём передаваемых данных.

Dockerfile с использованием `jarmode=tools` (Spring Boot 3.x):

```dockerfile
# Стадия 1: извлечение слоёв
FROM eclipse-temurin:21-jre AS builder
WORKDIR /builder
COPY target/*.jar application.jar
RUN java -Djarmode=tools -jar application.jar extract --layers --destination extracted

# Стадия 2: финальный образ
FROM eclipse-temurin:21-jre-alpine
WORKDIR /application
# Слои копируются в порядке от редко к часто изменяемых
COPY --from=builder /builder/extracted/dependencies/ ./
COPY --from=builder /builder/extracted/spring-boot-loader/ ./
COPY --from=builder /builder/extracted/snapshot-dependencies/ ./
COPY --from=builder /builder/extracted/application/ ./
ENTRYPOINT ["java", "-jar", "application.jar"]
```

В Spring Boot 2.x вместо `jarmode=tools` использовался `jarmode=layertools`.

---

**Приём 4. Custom JRE через jlink**

`jlink` (доступен с Java 9) создаёт минимальный JRE, включающий только те Java-модули, которые реально нужны приложению. Типичный Spring Boot нуждается в 10–15 модулях из 70+. Результат: JRE ~30–60 МБ вместо полного JRE ~150–200 МБ. Итоговый образ — менее 100 МБ.

```dockerfile
FROM eclipse-temurin:21-jdk AS jre-build
RUN jlink \
    --add-modules java.base,java.logging,java.xml,java.naming,java.desktop,java.management,java.security.jgss,java.instrument \
    --strip-debug \
    --no-man-pages \
    --no-header-files \
    --compress=2 \
    --output /custom-jre

FROM debian:bookworm-slim
COPY --from=jre-build /custom-jre /opt/java
ENV PATH="/opt/java/bin:$PATH"
COPY target/*.jar app.jar
ENTRYPOINT ["java", "-jar", "app.jar"]
```

---

**Приём 5. GraalVM Native Image (продвинутый уровень)**

Компиляция Spring Boot приложения в нативный бинарник (AOT). Образ: ~50–80 МБ, старт: <100 мс, потребление памяти в разы меньше. Требует `spring-native` или `spring-boot-starter-...` совместимых зависимостей и значительно более долгую сборку.

---

**Дополнительные советы:**
- `.dockerignore` — исключить `target/`, `*.iml`, `.git`, `node_modules` и другие лишние файлы из build context
- Использовать `--no-install-recommends` / `apk add --no-cache` для минимизации устанавливаемых системных пакетов
- Один процесс на контейнер, не устанавливать ничего лишнего в runtime-образ

**Итоговое сравнение размеров (ориентировочно):**

| Подход | Размер образа |
|--------|--------------|
| `openjdk:21` + fat JAR | 450–600 МБ |
| Multi-stage + JRE slim | 200–300 МБ |
| Multi-stage + Alpine JRE | 150–200 МБ |
| Distroless Java 21 | 200–210 МБ |
| Alpine + jlink custom JRE | 60–100 МБ |
| Alpaquita + Liberica JRE | ~52 МБ |
| GraalVM Native Image | 50–80 МБ |

### Вопрос 18. Как хранить чувствительные данные (пароли, токены) в Kubernetes так, чтобы они не попали в git-репозиторий или логи? Какие ограничения у Secret «из коробки» и как их усилить?

**Категория:** Контейнеры/DevOps · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Good practices for Kubernetes Secrets (kubernetes.io)](https://kubernetes.io/docs/concepts/security/secrets-good-practices/)

Официальная документация Kubernetes описывает следующее:

**Ограничения Kubernetes Secrets «из коробки»:**
- Secret-объекты хранятся в etcd **в незашифрованном виде** по умолчанию. Любой, кто имеет доступ к API-серверу или к etcd, может получить или изменить Secret.
- Base64-кодирование — это не шифрование, а лишь представление данных.
- Пользователи, имеющие право создавать Pod-ы, могут косвенно получить доступ к любому Secret в пространстве имён.

**Рекомендации для администраторов кластера:**
- Настроить шифрование Secret-данных в etcd (`EncryptionConfiguration`).
- Ограничить доступ к etcd только для администраторов кластера.
- Применять наименее привилегированный RBAC: для компонентов, которым нужно только чтение конкретного Secret, выдавать только `get` по конкретному ресурсу, не `list`/`watch`.
- Использовать внешние системы управления секретами через Secrets Store CSI Driver (AWS Secrets Manager, Azure Key Vault, GCP Secret Manager, HashiCorp Vault).

**Рекомендации для разработчиков:**
- Не логировать значения Secret-ов.
- Для высокочувствительных данных предпочитать монтирование через volume, а не через переменные окружения (переменные среды могут попасть в process listings, crash dumps, логи).
- Не передавать Secret-манифесты с реальными данными в git (никогда не коммитить YAML с base64-значениями).
- Ограничивать доступ к Secret-ам только тем контейнерам, которым они нужны.

#### Ответ от Claude

**Почему Kubernetes Secret небезопасен «из коробки»**

1. **Base64 — не шифрование.** Значения в `.data` поля хранятся в base64. Декодирование тривиально: `echo "dXNlcm5hbWU=" | base64 -d` → `username`. Это лишь удобный формат представления, не защита.

2. **Etcd хранит данные без шифрования по умолчанию.** Если злоумышленник получит доступ к etcd (snapshot, volume диска), все Secret-ы читаются в открытом виде.

3. **RBAC не настроен жёстко по умолчанию.** По умолчанию широкие права могут позволить нежелательным субъектам читать Secret-ы.

4. **Env var — небезопасный способ передачи.** Переменные окружения видны через `/proc/<pid>/environ`, crash dump-ы, некоторые логирующие фреймворки.

5. **Нет встроенного аудита доступа.** Kubernetes не отслеживает, кто и когда обращался к конкретному Secret.

6. **Secret-манифест в git = компрометация.** Если разработчик случайно закоммитит `secret.yaml` с base64-данными, секрет скомпрометирован навсегда (история git не удаляется легко).

**Как хранить Secret в Kubernetes безопасно**

**1. Никогда не коммитить Secret-манифесты с реальными значениями в git**

Плохая практика:
```yaml
# НЕ ДЕЛАТЬ ТАК — этот файл нельзя коммитить в git
apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
data:
  password: bXlwYXNzd29yZA==  # base64 от "mypassword"
```

Вместо этого — один из подходов ниже.

**2. Включить шифрование Secret-ов в etcd (Encryption at Rest)**

Настраивается через `EncryptionConfiguration` на API-сервере:

```yaml
# /etc/kubernetes/encryption-config.yaml
apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
- resources:
  - secrets
  providers:
  - aescbc:
      keys:
      - name: key1
        secret: <base64-encoded-32-byte-key>
  - identity: {}  # fallback для чтения незашифрованных (при миграции)
```

После включения все новые и пересохранённые Secret-ы шифруются AES-CBC (или AES-GCM, KMS) перед записью в etcd.

**3. Настроить минимально привилегированный RBAC**

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: app-secret-reader
  namespace: production
rules:
- apiGroups: [""]
  resources: ["secrets"]
  verbs: ["get"]               # только get, не list/watch
  resourceNames: ["db-secret"] # только конкретный Secret
```

Важно: `list` на secrets позволяет прочитать содержимое всех Secret-ов в namespace.

**4. Передавать Secret через volume, а не через env**

Через env (менее безопасно):
```yaml
env:
- name: DB_PASSWORD
  valueFrom:
    secretKeyRef:
      name: db-secret
      key: password
```

Через volume (рекомендуется для высокочувствительных данных):
```yaml
volumes:
- name: db-secret-vol
  secret:
    secretName: db-secret
    defaultMode: 0400  # только чтение, только владелец
containers:
- name: app
  volumeMounts:
  - name: db-secret-vol
    mountPath: /etc/secrets
    readOnly: true
```

Файл монтируется в tmpfs (in-memory), не записывается на диск узла.

**5. Усиление через внешние системы управления секретами**

| Инструмент | Принцип | Хранение в git | Подходит для |
|-----------|---------|----------------|-------------|
| **Sealed Secrets** (Bitnami) | Шифрование Secret публичным ключом контроллера; зашифрованный `SealedSecret` безопасно хранится в git | Зашифрованный SealedSecret | GitOps-команды, статичные секреты |
| **External Secrets Operator** | Синхронизирует секреты из внешних хранилищ (AWS SM, Vault, GCP SM) в Kubernetes Secret | Только ссылка (имя в хранилище) | Команды с уже существующими облачными секретами |
| **Secrets Store CSI Driver** | Монтирует секреты из внешних хранилищ напрямую в Pod как volume (без создания Secret-объекта в etcd) | Только ссылка | Высокочувствительные данные, нет хранения в etcd |
| **HashiCorp Vault** | Полноценный секретный менеджер: динамические секреты, ротация, аудит, fine-grained ACL | Нет (секреты только в Vault) | Enterprise, динамические БД credentials |

*Sealed Secrets* — практичный выбор для GitOps: `SealedSecret` YAML безопасно коммитится в git, расшифровать его может только контроллер внутри кластера.

*External Secrets Operator (ESO)* — `ExternalSecret` объект указывает путь к секрету во внешнем хранилище; ESO периодически синхронизирует его в Kubernetes Secret.

```yaml
# ExternalSecret — безопасно коммитить в git
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: db-credentials
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: aws-secrets-manager
    kind: SecretStore
  target:
    name: db-secret  # имя создаваемого Kubernetes Secret
  data:
  - secretKey: password
    remoteRef:
      key: prod/myapp/db     # путь в AWS Secrets Manager
      property: password
```

**6. Дополнительные меры**

- Включить **Audit Logging** Kubernetes API-сервера для отслеживания обращений к Secret-ам.
- Не логировать значения переменных окружения или содержимое Secret-файлов в коде приложения.
- Использовать **short-lived credentials** и авторотацию там, где это поддерживается (Vault dynamic secrets, AWS IAM Roles for Service Accounts).
- Для образов в private registry — хранить `imagePullSecrets` аналогично другим секретам.
- Сканировать git-репозиторий на наличие случайно закоммиченных секретов: **truffleHog**, **gitleaks**, **git-secrets**.

**Итоговый чеклист безопасности Secret:**

- [ ] Включено шифрование etcd at rest (`EncryptionConfiguration`)
- [ ] RBAC ограничивает доступ до конкретных Secret-ов и конкретных глаголов (`get`, не `list`)
- [ ] Secret-манифесты с реальными данными отсутствуют в git
- [ ] Для GitOps используется Sealed Secrets или ESO
- [ ] Высокочувствительные данные монтируются через volume, а не env
- [ ] Включён аудит доступа к API Secrets

### Вопрос 19. В чём принципиальная разница между Docker-образом и Docker-контейнером? Что происходит на уровне файловой системы, когда контейнер запускается поверх образа?

**Категория:** Контейнеры/DevOps · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Understanding the image layers | Docker Docs](https://docs.docker.com/get-started/docker-concepts/building-images/understanding-image-layers/)

Docker-образ — это неизменяемый (read-only) набор слоёв файловой системы, каждый из которых соответствует одной инструкции в Dockerfile. Слои складываются в стек с помощью union-файловой системы, которая представляет их как единое дерево директорий. Благодаря этому образы, использующие одинаковые базовые слои (например, один и тот же `ubuntu:22.04`), хранят эти слои на диске только один раз и совместно используют их между несколькими образами.

Когда вы выполняете `docker run`, Docker берёт все read-only слои образа и добавляет поверх них тонкий writable-слой (container layer). Именно в нём фиксируются все изменения, которые контейнер вносит в файловую систему во время работы: создаваемые файлы, логи, временные данные. Все файлы, которые контейнер не трогает, остаются в нижних слоях образа и не копируются. Когда контейнер удаляется, его writable-слой уничтожается вместе с ним — данные теряются, если не были сохранены через том (volume) или bind mount.

#### Ответ от Claude

**Docker-образ** — это статический, иммутабельный артефакт: упорядоченный стек read-only слоёв файловой системы, каждый из которых создаётся одной инструкцией Dockerfile (`FROM`, `RUN`, `COPY`, `ADD` и т.д.). Образ описывает состояние файловой системы приложения, но сам по себе не выполняется.

**Docker-контейнер** — это живой, исполняемый экземпляр образа. Контейнер имеет собственный PID-неймспейс, сетевой интерфейс, изолированную файловую систему и может менять своё состояние в процессе работы.

**Что происходит на уровне файловой системы при `docker run`:**

Docker использует драйвер хранилища overlay2 (стандарт на современных Linux-системах). Он организует mount с четырьмя компонентами:

- **lowerdir** — один или несколько read-only слоёв образа (снизу вверх: базовый образ, затем слои Dockerfile).
- **upperdir** — единственный read-write слой контейнера, изначально пустой.
- **workdir** — внутренняя рабочая директория OverlayFS (используется при операциях copy-up).
- **merged** — виртуальная точка монтирования, которую видит процесс внутри контейнера: единое дерево файловой системы, образованное наложением всех слоёв.

Механизм copy-on-write (CoW) работает так:
1. **Чтение файла**, существующего только в lowerdir: файл отдаётся напрямую из нижнего слоя без какого-либо копирования.
2. **Первая запись в существующий файл**: OverlayFS выполняет операцию `copy_up` — копирует файл целиком из lowerdir в upperdir, после чего модификация применяется уже к копии в upperdir. Исходный файл в lowerdir остаётся нетронутым.
3. **Создание нового файла**: файл сразу создаётся в upperdir.
4. **Удаление файла из lowerdir**: в upperdir создаётся специальный «whiteout»-файл, который маскирует файл из нижнего слоя — в merged он становится невидимым.

**Ключевые следствия:**
- Несколько контейнеров, запущенных из одного образа, разделяют неизменяемые слои lowerdir, тратя лишь по одному upperdir каждый — экономия диска и памяти значительная.
- При удалении контейнера upperdir удаляется; read-only слои образа не затрагиваются.
- `copy_up` копирует файл целиком (не блок), поэтому первая запись в большой файл может быть заметно медленнее последующих.
- Для персистентных данных нужны Docker volumes или bind mounts — они монтируются в обход overlay и не теряются при удалении контейнера.

### Вопрос 20. Dockerfile создаёт слои при каждой инструкции. Как работает механизм copy-on-write в overlay-файловой системе? Почему порядок инструкций в Dockerfile критичен для эффективности кеширования слоёв?

**Категория:** Контейнеры/DevOps · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Optimize cache usage in builds | Docker Docs](https://docs.docker.com/build/cache/optimize/)

Каждая инструкция в Dockerfile транслируется в слой образа. Docker проверяет, можно ли переиспользовать слой из предыдущей сборки. Если слой не изменился — используется кеш. Как только кеш инвалидируется на каком-либо слое, все последующие слои тоже пересобираются с нуля, даже если их содержимое не менялось. Поэтому инструкции следует располагать от наиболее стабильных к наиболее часто меняющимся. Практический пример: сначала копировать только файлы зависимостей (например, `pom.xml`), установить зависимости, а уже потом копировать исходный код. Тогда дорогостоящий шаг установки зависимостей не будет повторяться при каждом изменении кода.

#### Ответ от Claude

**Механизм copy-on-write (CoW) в OverlayFS**

OverlayFS — это union-файловая система ядра Linux, которую Docker использует через драйвер overlay2. Она объединяет несколько директорий в единое представление:

- **lowerdir** (один или несколько, read-only) — слои образа.
- **upperdir** (один, read-write) — слой контейнера.
- **merged** — то, что видит процесс в контейнере.
- **workdir** — внутреннее scratch-пространство OverlayFS для атомарных операций.

Принцип CoW:

| Операция | Что происходит |
|---|---|
| Чтение файла только из lowerdir | Файл отдаётся напрямую, без копирования |
| Первая запись в файл из lowerdir | copy_up: весь файл копируется в upperdir, затем изменяется там |
| Повторная запись в тот же файл | Запись идёт прямо в upperdir, copy_up не повторяется |
| Создание нового файла | Файл сразу создаётся в upperdir |
| Удаление файла из lowerdir | В upperdir создаётся whiteout-файл; файл скрывается в merged |

Важная особенность: OverlayFS работает на уровне файлов, а не блоков. `copy_up` копирует файл целиком, даже если изменился только один байт. Это может создать задержку при первой записи в большой файл.

**Почему порядок инструкций в Dockerfile критичен**

Docker BuildKit проверяет кеш слоёв последовательно. Логика инвалидации:

1. Если инструкция изменилась (другая команда в `RUN`, другой хеш файлов в `COPY`) — кеш этого слоя инвалидируется.
2. **Все последующие слои автоматически инвалидируются**, даже если они не изменились.

Из этого вытекает золотое правило: **располагай инструкции от стабильных к нестабильным**.

**Для Java-приложения (Maven):**

Плохой порядок (пересборка зависимостей при каждом изменении кода):
```dockerfile
COPY . /app          # любое изменение кода -> инвалидация
RUN mvn dependency:resolve
RUN mvn package
```

Правильный порядок (зависимости кешируются отдельно от кода):
```dockerfile
COPY pom.xml /app/         # меняется редко -> кеш стабилен
RUN mvn dependency:resolve # выполняется только при изменении pom.xml
COPY src /app/src           # меняется часто, но только этот слой и далее
RUN mvn package -DskipTests
```

Дополнительные правила:
- `FROM` — один раз, базовый образ меняется редко, ставить первым.
- Установку системных зависимостей (`RUN apt-get install`) — после `FROM`, до копирования кода.
- `COPY`/`ADD` — только нужные файлы; использовать `.dockerignore` чтобы случайные изменения не инвалидировали кеш.
- Объединять связанные `RUN`-команды через `&&` — это уменьшает число слоёв и не оставляет промежуточных артефактов (например, кеша apt).
- `ENV`, `LABEL`, `EXPOSE` — не создают тяжёлых слоёв, порядок не критичен, но по соглашению ставятся ближе к началу.
