### Вопрос 21. Объясните многоэтапную сборку (multi-stage build) для Java-приложения: зачем использовать отдельный этап компиляции (например, Maven/Gradle) и отдельный runtime-этап (JRE/distroless)? Какой реальный выигрыш в размере образа?

**Категория:** Контейнеры/DevOps · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Multi-stage builds | Docker Docs](https://docs.docker.com/build/building/multi-stage/)

Multi-stage builds позволяют использовать несколько инструкций `FROM` в одном Dockerfile. Каждая `FROM` начинает новый этап сборки и может использовать свой базовый образ. Результат из одного этапа можно выборочно скопировать в другой с помощью `COPY --from=<stage>`. Это позволяет включить в финальный образ только то, что необходимо для работы приложения, не включая инструменты сборки (Maven, Gradle, JDK, исходный код). Итоговый образ существенно меньше, а поверхность атаки — меньше, поскольку в нём отсутствует компилятор, менеджер пакетов и другие инструменты, которые могут быть использованы злоумышленником.

#### Ответ от Claude

**Проблема без multi-stage build**

Наивный подход — один Dockerfile на базе `maven:3.9-eclipse-temurin-21`:
```dockerfile
FROM maven:3.9-eclipse-temurin-21
WORKDIR /app
COPY . .
RUN mvn package -DskipTests
CMD ["java", "-jar", "target/app.jar"]
```
Такой образ включает: Maven (~350 MB), весь JDK (~400 MB), исходный код, кеш локального репозитория Maven (~100–500 MB и более). Итоговый образ — легко 700 MB–1 GB+.

**Multi-stage build: структура**

```dockerfile
# --- Этап 1: сборка ---
FROM maven:3.9-eclipse-temurin-21 AS builder
WORKDIR /app
# Сначала зависимости (кеш)
COPY pom.xml .
RUN mvn dependency:go-offline -q
# Затем исходники
COPY src ./src
RUN mvn package -DskipTests -q

# --- Этап 2: runtime ---
FROM eclipse-temurin:21-jre-alpine
WORKDIR /app
COPY --from=builder /app/target/app.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
```

В финальный образ попадает только JRE + JAR-файл приложения.

**Реальный выигрыш в размере образа**

| Вариант | Примерный размер образа |
|---|---|
| `maven:3.9-eclipse-temurin-21` (без multi-stage) | ~700 MB – 1 GB+ |
| `eclipse-temurin:21-jre` (multi-stage, Debian) | ~250–300 MB |
| `eclipse-temurin:21-jre-alpine` (multi-stage) | ~100–150 MB |
| `gcr.io/distroless/java21` (distroless) | ~120–160 MB |
| Custom JRE через `jlink` + distroless/scratch | ~50–100 MB |

Типичное сокращение при переходе от «без multi-stage» к `jre-alpine` — **70–85%** от исходного размера.

**Зачем разделять этапы**

1. **Исключение инструментов сборки из production-образа.** Maven, Gradle, JDK-компилятор не нужны в runtime. Их отсутствие уменьшает поверхность атаки (нет компилятора, нет shell в distroless-образах).
2. **Уменьшение размера образа** ускоряет: pull в CI/CD, деплой на узлы кластера, запуск контейнеров.
3. **Изоляция кеша зависимостей.** `dependency:go-offline` на отдельном слое кешируется эффективно; при изменении только кода этот слой не пересобирается.
4. **Отсутствие исходного кода в финальном образе.** Это важно при наличии чувствительной бизнес-логики.

**Distroless vs Alpine**

- **Alpine** (`eclipse-temurin:21-jre-alpine`): есть shell `/bin/sh`, пакетный менеджер `apk`, удобно для отладки. Размер JRE-образа ~100–150 MB.
- **Distroless** (`gcr.io/distroless/java21`): нет shell, нет пакетного менеджера, нет `/bin`, только JRE и glibc-зависимости. Меньший размер, минимальная поверхность атаки, но невозможно зайти `docker exec` с интерактивным shell. Для отладки добавляют отдельный `:debug`-вариант distroless-образа.

**Дополнительная оптимизация: jlink**

Начиная с Java 9, инструмент `jlink` позволяет создать минимальный custom JRE, включающий только те Java-модули, которые реально использует приложение. В сочетании с distroless/scratch это даёт образы размером 50–80 MB.

```dockerfile
FROM eclipse-temurin:21 AS jlink-stage
RUN $JAVA_HOME/bin/jlink \
    --add-modules java.base,java.logging,java.sql \
    --strip-debug \
    --no-man-pages \
    --compress=2 \
    --output /custom-jre

FROM debian:bookworm-slim
COPY --from=jlink-stage /custom-jre /opt/jre
COPY --from=builder /app/target/app.jar /app.jar
ENTRYPOINT ["/opt/jre/bin/java", "-jar", "/app.jar"]
```
### Вопрос 22. Чем отличается COPY от ADD в Dockerfile? В каких случаях использование ADD может быть опасным с точки зрения безопасности и воспроизводимости сборки?

**Категория:** Контейнеры/DevOps · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Docker Best Practices: Understanding the Differences Between ADD and COPY Instructions in Dockerfiles](https://www.docker.com/blog/docker-best-practices-understanding-the-differences-between-add-and-copy-instructions-in-dockerfiles/)

Инструкция `COPY` просто копирует файлы и директории из контекста сборки в образ — без дополнительной логики. Инструкция `ADD` делает всё то же самое, но имеет два дополнительных поведения: скачивание файлов по URL и автоматическое извлечение tar-архивов (tar.gz и др.) в указанную директорию.

Риски `ADD`:
- **Удалённые URL**: при скачивании внешнего ресурса нет проверки целостности (контрольная сумма не верифицируется). Содержимое файла на удалённом сервере может измениться в любой момент — сборка перестанет быть воспроизводимой, а в образ может попасть скомпрометированный контент.
- **Неявная распаковка архивов**: если структура архива не ожидается разработчиком, в образ могут быть извлечены лишние файлы.
- **Трудность аудита**: расширенная функциональность `ADD` делает Dockerfile менее прозрачным.

Рекомендация Docker: по умолчанию использовать `COPY`; для скачивания файлов — `RUN curl` или `RUN wget` с явной проверкой контрольной суммы (что делает шаг аудируемым и воспроизводимым); `ADD` применять только когда авторасспаковка архива действительно необходима и осознана.

#### Ответ от Claude

**COPY** и **ADD** — две инструкции Dockerfile для помещения файлов в образ. Они похожи синтаксически, но принципиально различаются по поведению.

**COPY** — простая, предсказуемая инструкция: копирует файлы или директории из контекста сборки (локальной файловой системы) в указанный путь внутри образа. Ничего больше.

**ADD** — расширенный вариант с двумя дополнительными возможностями:
1. **Скачивание по URL**: `ADD https://example.com/file.tar.gz /app/` загружает файл из интернета во время сборки.
2. **Автоматическая распаковка архивов**: `ADD archive.tar.gz /app/` — архив будет автоматически распакован в `/app/`.

**Опасности ADD с точки зрения безопасности:**

- **Отсутствие проверки целостности при загрузке по URL.** Docker не проверяет контрольную сумму скачиваемого файла. Если содержимое на удалённом сервере изменится (взлом CDN, подмена файла), образ будет собран с вредоносным содержимым без каких-либо предупреждений. Это классический вектор атаки на цепочку поставок (supply chain attack).
- **Непредсказуемость распаковки.** Архив может содержать абсолютные пути или symlink'и, ведущие за пределы целевой директории (path traversal). В результате в образ могут попасть файлы в неожиданных местах.
- **Сложность аудита.** Неявное поведение затрудняет ревью Dockerfile — поведение `ADD` неочевидно без знания исходного файла.

**Опасности ADD с точки зрения воспроизводимости:**

- Удалённый URL — внешняя зависимость. Файл может исчезнуть, URL может стать недоступным, а содержимое — измениться. Это нарушает детерминированность сборки: одна и та же команда через неделю может дать другой результат.
- Docker-кэш для `ADD <URL>` работает на основе временной метки (`Last-Modified`), а не содержимого — кэш может не инвалидироваться при фактическом изменении файла.

**Лучшие практики:**

```dockerfile
# Плохо — непредсказуемо, небезопасно
ADD https://example.com/tool.tar.gz /opt/

# Хорошо — явная загрузка с верификацией SHA256
RUN curl -fsSL https://example.com/tool.tar.gz -o /tmp/tool.tar.gz \
    && echo "abc123...  /tmp/tool.tar.gz" | sha256sum -c \
    && tar -xzf /tmp/tool.tar.gz -C /opt/ \
    && rm /tmp/tool.tar.gz

# Хорошо — ADD оправдан только для локального архива с авторасспаковкой
ADD local-archive.tar.gz /app/

# Лучше для простого копирования
COPY ./config /app/config
```

**Итог:** `COPY` следует использовать по умолчанию. `ADD` оправдан только для авторасспаковки **локального** tar-архива. Загрузку внешних файлов всегда нужно делать через `RUN curl`/`RUN wget` с явной проверкой контрольной суммы.

### Вопрос 23. Что такое Docker Registry и в чём разница между публичным (Docker Hub) и приватным реестром? Как работает механизм push/pull с точки зрения слоёв — передаются ли уже существующие слои повторно?

**Категория:** Контейнеры/DevOps · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [docker image push | Docker Docs](https://docs.docker.com/reference/cli/docker/image/push/)

Docker Registry — централизованное хранилище для Docker-образов. Docker Hub — публичный реестр по умолчанию, доступный всем. При выполнении `docker push` по умолчанию демон отправляет до пяти слоёв одновременно. Слои образа загружаются и отправляются по отдельности. Вывод команды `push` при работе с реестром версии 2.0 включает дайджест образа. При отправке реестр проверяет наличие каждого слоя по его дайджесту (SHA256): для уже существующих слоёв выводится сообщение «Layer already exists» — они повторно не передаются.

#### Ответ от Claude

**Docker Registry** — это сервер (или сервис), реализующий стандартизированный API для хранения и раздачи Docker-образов. Образ хранится не монолитом, а в виде набора слоёв (layers) плюс манифест (manifest), описывающий их структуру.

**Публичный реестр (Docker Hub):**
- Открыт для всех: любой может скачивать публичные образы без аутентификации.
- Бесплатный план включает неограниченное количество публичных репозиториев и 1 приватный.
- Содержит официальные образы (ubuntu, nginx, postgres и т.д.) и пользовательские.
- Ограничения: rate limiting на pull для анонимных пользователей (100 запросов за 6 часов), зависимость от доступности внешнего сервиса.

**Приватный реестр:**
- Развёртывается внутри инфраструктуры организации (self-hosted: Docker Distribution/Registry, Harbor, Nexus, JFrog Artifactory) или как managed-сервис (AWS ECR, Azure ACR, GCP Artifact Registry, GitLab Registry).
- Доступ контролируется через аутентификацию/авторизацию — образы не видны внешним пользователям.
- Нет rate limiting, данные остаются внутри периметра безопасности, возможна интеграция с корпоративным SSO.
- Позволяет хранить внутренние образы с чувствительными конфигурациями, не выкладывая их наружу.

**Как работает механизм push/pull со слоями:**

Docker использует **content-addressable storage**: каждый слой идентифицируется SHA256-дайджестом своего содержимого. Это ключевой принцип дедупликации.

**Push (отправка образа):**

1. Docker-клиент читает манифест образа и получает список дайджестов всех слоёв.
2. Для каждого слоя клиент отправляет в реестр **HEAD-запрос** вида `HEAD /v2/{name}/blobs/{digest}`.
3. Если реестр возвращает **200 OK** — слой уже есть, клиент пропускает его загрузку (выводится `Layer already exists`).
4. Если реестр возвращает **404 Not Found** — слой отсутствует, клиент начинает загрузку через POST/PUT-запросы.
5. В конце загружается манифест, связывающий все слои с тегом образа.

**Pull (скачивание образа):**

1. Клиент запрашивает манифест образа по имени и тегу (или дайджесту).
2. Из манифеста извлекается список дайджестов слоёв.
3. Для каждого слоя проверяется его наличие в **локальном кэше** клиента (по тому же SHA256-дайджесту).
4. Загружаются только отсутствующие локально слои.

**Практические следствия:**
- Если два образа (например, `myapp:v1` и `myapp:v2`) имеют общие базовые слои (один и тот же `FROM ubuntu:22.04`), при push второго образа общие слои уже будут в реестре — повторная передача не происходит.
- Аналогично при pull: если на машине уже есть образ с ubuntu-слоями, новый образ на той же базе скачает только дельту.
- Эта оптимизация существенно экономит полосу пропускания и ускоряет CI/CD-пайплайны.

**Важно:** дедупликация работает в рамках **одного реестра** на стороне сервера. При push в другой реестр (например, из Docker Hub в ECR) слои нужно загрузить туда заново, даже если они идентичны.

### Вопрос 24. В Kubernetes чем отличается liveness probe от readiness probe? Что произойдёт, если liveness probe упадёт из-за временной перегрузки самого приложения (например, long GC pause в JVM)?

**Категория:** Контейнеры/DevOps · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Liveness, Readiness, and Startup Probes | Kubernetes](https://kubernetes.io/docs/concepts/workloads/pods/probes/)

**Liveness probe** определяет, нужно ли перезапустить контейнер. Она отвечает на вопрос: «находится ли приложение в неисправимом состоянии (дедлок, зависание)?» При провале liveness probe kubelet **убивает и перезапускает** контейнер согласно `restartPolicy` пода.

**Readiness probe** определяет, готов ли контейнер принимать трафик. При провале readiness probe под исключается из `EndpointSlices` соответствующих Service — трафик на него не направляется, но **контейнер не перезапускается** и продолжает работать.

Документация особо предупреждает: некорректно настроенные liveness probe могут вызвать каскадные отказы — перезапуски под нагрузкой увеличивают число ошибок клиентов и дополнительно нагружают оставшиеся поды.

#### Ответ от Claude

**Liveness probe** («проба живучести»):
- Отвечает на вопрос: «жив ли процесс? способен ли он вообще работать?»
- Реагирует на **невосстановимые** состояния: дедлок, утечка памяти, застрявший поток.
- При провале (после `failureThreshold` последовательных неудач): kubelet **убивает контейнер** и запускает новый.

**Readiness probe** («проба готовности»):
- Отвечает на вопрос: «может ли под принимать запросы прямо сейчас?»
- Реагирует на **временные** состояния: прогрев кэша, ожидание подключения к БД, превышение допустимой нагрузки.
- При провале: под убирается из ротации (удаляется из `EndpointSlice`), но **не перезапускается**. Когда проба снова начнёт проходить — под автоматически вернётся в ротацию.

| Параметр | Liveness | Readiness |
|---|---|---|
| Реакция на провал | Перезапуск контейнера | Исключение из трафика |
| Типичный сценарий | Дедлок, зависание процесса | Прогрев, временная перегрузка |
| Контейнер продолжает работать после провала | Нет | Да |

**Что происходит при падении liveness probe из-за long GC pause в JVM:**

Stop-the-World GC (особенно Full GC или G1 Mixed GC) может приостановить все потоки JVM на сотни миллисекунд или даже секунды. В этот момент приложение не может ответить на HTTP health-check или выполнить exec-команду.

Если `timeoutSeconds` у liveness probe меньше длительности паузы GC — probe завершится с ошибкой. После `failureThreshold` таких провалов kubelet **перезапустит под**. Это опасный сценарий:

1. Нагрузка возрастает → GC давление усиливается → паузы длиннее.
2. liveness probe падает → поды перезапускаются.
3. Нагрузка перераспределяется на оставшиеся поды → они тоже начинают испытывать GC pressure.
4. Каскадный рестарт всего деплоймента при живом и способном к восстановлению приложении.

**Правильные практики для JVM-приложений:**

1. **Использовать startup probe** для медленно стартующих JVM-сервисов, чтобы не смешивать логику старта и логику работающего приложения:
   ```yaml
   startupProbe:
     httpGet:
       path: /actuator/health/liveness
       port: 8080
     failureThreshold: 30
     periodSeconds: 10   # 5 минут на старт
   ```

2. **Увеличить `timeoutSeconds` и `failureThreshold`** у liveness probe с запасом на GC:
   ```yaml
   livenessProbe:
     httpGet:
       path: /actuator/health/liveness
       port: 8080
     periodSeconds: 10
     timeoutSeconds: 5    # учитываем возможную GC-паузу
     failureThreshold: 3  # 3 × 10с = 30с до перезапуска
   ```

3. **Не использовать exec-пробы с JVM** — `exec` форкает новый процесс, что само по себе создаёт нагрузку на JVM во время GC.

4. **Разделять endpoints**: liveness (`/health/liveness`) должен проверять только факт работоспособности процесса (не дедлок), readiness (`/health/readiness`) — готовность к трафику (зависимости, очереди, прогрев).

5. **Принцип**: liveness probe должна реагировать только на **невосстановимые** состояния. GC — восстановимое состояние; правильная реакция на него — readiness probe, которая временно выведет под из ротации, пока нагрузка не спадёт.
### Вопрос 25. Что такое startup probe и зачем она нужна отдельно от liveness probe? Как она помогает Java-приложениям с долгим стартом (Spring Boot, Quarkus) избежать преждевременных перезапусков?

**Категория:** Контейнеры/DevOps · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Configure Liveness, Readiness and Startup Probes | Kubernetes](https://kubernetes.io/docs/tasks/configure-pod-container/configure-liveness-readiness-startup-probes/)

Startup probe проверяет, завершило ли приложение в контейнере инициализацию. Если такая проверка настроена, она отключает проверки liveness и readiness до тех пор, пока не завершится успехом. Это гарантирует, что liveness и readiness пробы не помешают запуску приложения.

Для защиты медленно стартующих контейнеров можно настроить startup probe с тем же endpoint, что и у liveness probe, но с более высоким значением `failureThreshold * periodSeconds`. Например, для приложения, которому может потребоваться до 5 минут на запуск, задают `failureThreshold: 30` и `periodSeconds: 10` — таким образом разрешается до 300 секунд на инициализацию. После того как startup probe впервые завершается успехом, она больше никогда не запускается. Далее работу принимают liveness и readiness пробы.

Пример конфигурации для Spring Boot с Actuator:
```yaml
startupProbe:
  httpGet:
    path: /actuator/health
    port: 8080
  failureThreshold: 30
  periodSeconds: 10
```

#### Ответ от Claude

**Что такое startup probe и зачем она нужна**

Startup probe — это отдельный тип проверки работоспособности контейнера в Kubernetes, цель которой — убедиться, что приложение завершило процесс инициализации. Пока startup probe не сообщит об успехе (`Success`), kubelet не запускает liveness и readiness пробы. Это принципиальное отличие от liveness probe, которая работает на протяжении всего жизненного цикла контейнера и предназначена для обнаружения зависаний и деградации уже запущенного приложения.

Без startup probe разработчик сталкивается с дилеммой:
- Установить короткий `initialDelaySeconds` у liveness probe — и тогда Kubernetes будет убивать контейнер до завершения его инициализации.
- Установить большой `initialDelaySeconds` — и тогда настоящие зависания работающего приложения будут обнаруживаться с большой задержкой.

Startup probe решает эту проблему элегантно: она предоставляет продолжительный временной бюджет на старт, после чего liveness probe работает с агрессивными интервалами без какого-либо риска.

**Как это работает пошагово**

1. Контейнер запускается.
2. Kubelet начинает выполнять startup probe согласно `periodSeconds`.
3. Если startup probe возвращает `Success` — она прекращается навсегда для данного контейнера, управление передаётся liveness и readiness пробам.
4. Если startup probe возвращает `Failure` суммарно `failureThreshold` раз — контейнер перезапускается согласно `restartPolicy`.

**Расчёт максимального времени запуска**

Максимальное время, отводимое на инициализацию: `failureThreshold × periodSeconds`.

Например: `failureThreshold: 30`, `periodSeconds: 10` → 300 секунд (5 минут) на старт.

**Применение для Java-приложений (Spring Boot, Quarkus)**

JVM-приложения известны долгим временем инициализации: загрузка классов, инициализация Spring контекста, Flyway-миграции, прогрев соединений с БД могут занимать 30–120 секунд и более. Quarkus значительно быстрее в native-режиме, но в JVM-режиме картина аналогична Spring Boot.

Spring Boot Actuator (начиная с версии 2.3) предоставляет встроенные эндпоинты `/actuator/health/liveness` и `/actuator/health/readiness`, которые нативно поддерживают Kubernetes probes. Для startup probe можно использовать `/actuator/health` или создать специализированный эндпоинт.

Рекомендуемая конфигурация для типичного Spring Boot приложения:

```yaml
startupProbe:
  httpGet:
    path: /actuator/health
    port: 8080
  failureThreshold: 30   # 30 попыток × 10 сек = 300 сек на старт
  periodSeconds: 10

livenessProbe:
  httpGet:
    path: /actuator/health/liveness
    port: 8080
  periodSeconds: 5
  failureThreshold: 3

readinessProbe:
  httpGet:
    path: /actuator/health/readiness
    port: 8080
  periodSeconds: 5
```

**Сравнительная таблица трёх проб**

| Характеристика | Startup Probe | Liveness Probe | Readiness Probe |
|---|---|---|---|
| Когда работает | Только при старте | Постоянно | Постоянно |
| При неудаче | Перезапуск контейнера | Перезапуск контейнера | Исключение из Service endpoints |
| Цель | Инициализация завершена? | Приложение живо? | Приложение готово к трафику? |
| Для медленного старта | Да | Нет | Нет |

### Вопрос 26. В чём разница между ConfigMap и Secret в Kubernetes? Secret хранит данные в base64 — это шифрование? Как правильно защитить секреты в кластере?

**Категория:** Контейнеры/DevOps · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Secrets | Kubernetes](https://kubernetes.io/docs/concepts/configuration/secret/)

Secret — объект Kubernetes для хранения небольших объёмов чувствительных данных: паролей, токенов, ключей. Использование Secret позволяет не включать конфиденциальные данные непосредственно в код приложения или в спецификацию Pod.

По умолчанию Secret хранится в etcd в незашифрованном виде. Любой, у кого есть доступ к API или к etcd, может прочитать Secret или изменить его. Для безопасного использования Secrets необходимо как минимум:
1. Включить шифрование данных в etcd (encryption at rest).
2. Включить или настроить правила RBAC с минимально необходимыми привилегиями доступа к Secrets.
3. Ограничить доступ к Secret конкретными контейнерами.
4. Рассмотреть использование внешних провайдеров хранения секретов.

Важно: `list` доступ к Secrets неявно позволяет читать их содержимое. Пользователь, который может создавать Pod с использованием Secret, может видеть значение этого Secret.

#### Ответ от Claude

**Разница между ConfigMap и Secret**

| Характеристика | ConfigMap | Secret |
|---|---|---|
| Назначение | Нечувствительные конфигурационные данные | Конфиденциальные данные (пароли, токены, ключи, сертификаты) |
| Кодирование | Открытый текст (UTF-8) | base64-кодирование |
| Хранение в etcd | Незашифрованно | Незашифрованно по умолчанию (требует дополнительной настройки) |
| Размер | До 1 МБ | До 1 МБ |
| Отображение в `kubectl get` | Значения видны сразу | Значения скрыты (но доступны через `-o yaml`) |
| Тип ресурса | `ConfigMap` | `Secret` (типы: `Opaque`, `kubernetes.io/tls`, `kubernetes.io/dockerconfigjson` и др.) |

**Base64 — это шифрование? Нет.**

Это распространённое заблуждение. Base64 — это схема кодирования двоичных данных в ASCII-строку. Она полностью обратима без какого-либо ключа. Пример:

```bash
echo -n "mypassword" | base64
# bXlwYXNzd29yZA==

echo "bXlwYXNzd29yZA==" | base64 --decode
# mypassword
```

Base64 в Secrets нужна лишь для того, чтобы безопасно хранить произвольные бинарные данные (например, TLS-сертификаты) в JSON/YAML-формате API Kubernetes. Никакой защиты она не обеспечивает.

По умолчанию Secrets хранятся в etcd в незашифрованном виде. Любой, у кого есть прямой доступ к etcd или достаточные права RBAC (`kubectl get secret -o yaml`), может прочитать все значения.

**Как правильно защитить секреты в кластере**

*1. Encryption at Rest (шифрование в хранилище)*

Необходимо явно включить шифрование Secret-объектов в etcd. Настраивается через `EncryptionConfiguration` на уровне kube-apiserver:

```yaml
apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
  - resources:
      - secrets
    providers:
      - aescbc:
          keys:
            - name: key1
              secret: <base64-encoded-32-byte-key>
      - identity: {}
```

После применения этой конфигурации все новые Secrets будут зашифрованы перед записью в etcd. Существующие нужно перезаписать командой `kubectl get secrets --all-namespaces -o json | kubectl replace -f -`.

*2. RBAC с минимальными привилегиями*

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: secret-reader
rules:
- apiGroups: [""]
  resources: ["secrets"]
  verbs: ["get"]
  resourceNames: ["specific-secret-name"]  # только конкретные секреты
```

Важные нюансы RBAC:
- Права `list` и `watch` на Secrets дают возможность читать все их значения — нужно выдавать с осторожностью.
- Право `create` на Pod косвенно даёт доступ к Secret, который Pod монтирует.

*3. Ограничение доступа на уровне Pod*

Монтировать Secret только в те контейнеры, которым он действительно нужен:

```yaml
spec:
  containers:
  - name: app
    volumeMounts:
    - name: db-secret
      mountPath: /etc/secrets
      readOnly: true
  - name: sidecar
    # Secret здесь не монтируется
  volumes:
  - name: db-secret
    secret:
      secretName: db-credentials
```

Предпочитать монтирование через Volume (а не через env vars): если процесс упадёт с дампом памяти или в логах окажутся env vars, секрет из переменной окружения будет раскрыт.

*4. Внешние хранилища секретов*

Для production-окружений рекомендуется использовать внешние системы управления секретами:
- **HashiCorp Vault** — наиболее распространённое решение, поддерживает ротацию, аудит, динамические секреты.
- **AWS Secrets Manager / Azure Key Vault / GCP Secret Manager** — облачные решения для соответствующих платформ.
- **External Secrets Operator** — Kubernetes-оператор, синхронизирующий секреты из внешних хранилищ в Secret-объекты.
- **Secrets Store CSI Driver** — монтирует секреты из внешних хранилищ непосредственно как volume, минуя etcd.

*5. Не хранить Secret-манифесты в Git*

Файл YAML с base64-закодированным паролем в репозитории — это открытый пароль. Альтернативы:
- **Sealed Secrets** (Bitnami) — шифруют Secret асимметричным ключом, зашифрованный манифест можно безопасно хранить в Git.
- **SOPS** (Mozilla) — шифрование файлов с секретами с использованием KMS, PGP, age.

### Вопрос 27. Объясните разницу между Deployment и StatefulSet. В каких сценариях для Java-микросервиса нужен StatefulSet, а не Deployment?

**Категория:** Контейнеры/DevOps · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [StatefulSets | Kubernetes](https://kubernetes.io/docs/concepts/workloads/controllers/statefulset/)

StatefulSet — это объект API рабочей нагрузки, предназначенный для управления stateful-приложениями. Как и Deployment, StatefulSet управляет Pod'ами на основе идентичной спецификации контейнера. Но в отличие от Deployment, StatefulSet поддерживает устойчивую идентичность для каждого Pod'а. Эти Pod'ы создаются из одной спецификации, но не взаимозаменяемы: каждый имеет постоянный идентификатор, который сохраняется при перепланировании.

StatefulSet незаменим, когда приложение требует одного или нескольких из следующего:
- Стабильных, уникальных сетевых идентификаторов.
- Стабильного, постоянного хранилища данных.
- Упорядоченного, постепенного развёртывания и масштабирования.
- Упорядоченных, автоматических rolling update.

Каждый Pod в StatefulSet получает порядковый индекс (от 0 до N-1) и стабильное сетевое имя в формате `$(podname).$(servicename).$(namespace).svc.cluster.local`. При масштабировании Pod'ы создаются в порядке от 0 до N-1 и удаляются в обратном порядке. Каждый Pod должен быть Running и Ready перед запуском следующего.

#### Ответ от Claude

**Ключевые различия между Deployment и StatefulSet**

| Характеристика | Deployment | StatefulSet |
|---|---|---|
| Идентичность Pod'ов | Случайные имена (напр., `app-7d4f8b-xk2p9`) | Стабильные порядковые имена (`app-0`, `app-1`, `app-2`) |
| Взаимозаменяемость Pod'ов | Да — любой Pod идентичен другому | Нет — каждый Pod уникален и имеет постоянный идентификатор |
| Сетевой идентификатор | Единый ClusterIP через Service | Уникальный DNS для каждого Pod через Headless Service |
| Постоянное хранилище | Общий PVC или ephemeral | Отдельный PVC для каждого Pod (volumeClaimTemplates) |
| Порядок развёртывания | Параллельное создание | Строго последовательное (0, 1, 2...) |
| Порядок удаления | Произвольный | Строго обратный (N-1, ..., 1, 0) |
| Rolling Update | Все Pod'ы обновляются параллельно | Строго последовательно в обратном порядке |
| Headless Service | Не требуется | Обязателен (`clusterIP: None`) |
| Применение | Stateless микросервисы, REST API | Базы данных, кластеризованные приложения |

**Как работает StatefulSet на практике**

StatefulSet при создании 3 реплик последовательно запускает `app-0` → `app-1` → `app-2`. Каждый Pod получает:
- Стабильное имя хоста: `app-0`, `app-1`, `app-2`
- DNS-имя: `app-0.app-svc.namespace.svc.cluster.local`
- Свой PersistentVolumeClaim: `data-app-0`, `data-app-1`, `data-app-2`

Если Pod `app-1` перезапускается или переносится на другой узел, он сохраняет то же имя, тот же DNS и подключается к тому же PVC. Для Deployment этого нет — перезапущенный Pod получит случайное новое имя.

**Когда Java-микросервису нужен StatefulSet**

*Сценарий 1: Embedded кластерное состояние*

Когда Java-приложение само является частью кластера и использует leader election, основанный на стабильных идентификаторах (например, Apache Kafka-брокер, Zookeeper, Hazelcast IMDG, Apache Ignite). Каждый узел кластера должен иметь стабильный DNS-адрес, чтобы другие узлы могли его найти при перезапуске.

```yaml
# Пример headless service для Kafka-брокера
apiVersion: v1
kind: Service
metadata:
  name: kafka-headless
spec:
  clusterIP: None
  selector:
    app: kafka
  ports:
  - port: 9092
```

*Сценарий 2: Приложение с локальным состоянием, привязанным к конкретному Pod'у*

Если Java-сервис хранит данные локально (файловый кэш, локальная очередь, шард данных) и нужно гарантировать, что один и тот же Pod всегда работает с одними и теми же данными на одном диске.

*Сценарий 3: Приложение, встраивающее базу данных*

Quarkus или Spring Boot с встроенным H2/Derby/RocksDB для локального хранения, если нужна персистентность при перезапусках Pod'а.

*Сценарий 4: Сессионная репликация с жёсткой привязкой*

Некоторые legacy Java EE приложения (WildFly, JBoss) используют сессионную репликацию на основе multicast или точечного соединения между конкретными узлами кластера. StatefulSet обеспечивает предсказуемые DNS-имена, необходимые для такой конфигурации.

**Когда Java-микросервису достаточно Deployment**

Большинство современных Java-микросервисов (REST API, gRPC-сервисы, event-обработчики) — stateless: они не хранят состояние между запросами, а всё состояние вынесено во внешние системы (PostgreSQL, Redis, Kafka). Для таких приложений Deployment предпочтительнее: он проще в управлении, позволяет параллельное масштабирование и не требует Headless Service.

**Важные ограничения StatefulSet**

- Удаление StatefulSet не удаляет автоматически связанные PVC (защита данных). PVC нужно удалять вручную.
- Требует явного создания Headless Service.
- Rolling update менее гибок и требует особого внимания к порядку: Pod N-1 должен быть `Running+Ready` перед обновлением Pod N-2.

### Вопрос 28. Как Kubernetes выполняет rolling update при обновлении образа в Deployment? Что такое maxSurge и maxUnavailable? Как откатиться к предыдущей версии?

**Категория:** Контейнеры/DevOps · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Update a Deployment Without Downtime | Kubernetes](https://kubernetes.io/docs/tasks/run-application/update-deployment-rolling/)

Rolling update — постепенная замена старых Pod'ов новыми при сохранении доступности приложения. Обновление запускается любым изменением поля `.spec.template` Deployment'а (например, через `kubectl set image` или `kubectl apply`).

Стратегия `RollingUpdate` управляется двумя параметрами:
- **maxUnavailable** — максимальное количество Pod'ов, которые могут быть недоступны в процессе обновления (по умолчанию 25%, округляется вниз).
- **maxSurge** — максимальное количество Pod'ов сверх желаемого числа реплик, которые могут существовать одновременно (по умолчанию 25%, округляется вверх).

Мониторинг и откат:
```bash
kubectl rollout status deployment/nginx-deployment
kubectl rollout history deployment/nginx-deployment
kubectl rollout undo deployment/nginx-deployment
kubectl rollout undo deployment/nginx-deployment --to-revision=1
```

#### Ответ от Claude

**Механизм rolling update**

Rolling Update — стратегия обновления по умолчанию для Deployment в Kubernetes. При изменении образа (или любого поля `.spec.template`) Kubernetes создаёт новый ReplicaSet и начинает постепенно масштабировать его вверх, одновременно масштабируя старый ReplicaSet вниз.

Процесс (пример: 3 реплики, maxSurge=1, maxUnavailable=1):
1. Исходное состояние: 3 Pod'а старой версии (назовём old-rs).
2. Создаётся новый ReplicaSet (new-rs). Kubernetes создаёт 1 новый Pod (итого 4 = 3 желаемых + 1 surge).
3. Когда новый Pod становится Ready, удаляется 1 старый Pod (3 Pod'а: 2 старых + 1 новый).
4. Цикл повторяется до полной замены. Старый ReplicaSet масштабируется до 0, но остаётся в истории.

**maxSurge и maxUnavailable**

```yaml
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # или "25%" — разрешает временно иметь replicas+1 Pod'ов
      maxUnavailable: 0  # или "25%" — 0 означает нулевой downtime (always available)
```

- `maxSurge: 0, maxUnavailable: 1` — экономия ресурсов, но кратковременное уменьшение мощности.
- `maxSurge: 1, maxUnavailable: 0` — нулевой downtime, требует дополнительных ресурсов.
- Оба параметра принимают абсолютное число или процент от `replicas`.
- Kubernetes не считает Terminating Pod'ы как доступные при вычислении `availableReplicas`.

**Триггер обновления**

Rollout запускается только при изменении `.spec.template` (образ, метки Pod'а, env-переменные). Изменение `.spec.replicas` (масштабирование) rollout не запускает.

**Откат к предыдущей версии**

По умолчанию Kubernetes хранит историю Deployment (`.spec.revisionHistoryLimit`, по умолчанию 10 ревизий):

```bash
# Просмотр истории
kubectl rollout history deployment/my-app

# Подробности конкретной ревизии
kubectl rollout history deployment/my-app --revision=2

# Откат к предыдущей версии
kubectl rollout undo deployment/my-app

# Откат к конкретной ревизии
kubectl rollout undo deployment/my-app --to-revision=2

# Проверка статуса отката
kubectl rollout status deployment/my-app
```

Откат сам по себе является новой ревизией — Kubernetes берёт Pod template из указанной ревизии и создаёт новый rollout.

**Дополнительные инструменты**

```bash
# Пауза/возобновление (для применения нескольких изменений атомарно)
kubectl rollout pause deployment/my-app
kubectl set image deployment/my-app app=myapp:v2
kubectl set env deployment/my-app ENV=production
kubectl rollout resume deployment/my-app

# Обнаружение зависшего rollout (progressDeadlineSeconds, по умолчанию 600 сек)
kubectl describe deployment my-app
```

### Вопрос 29. Docker использовал containerd и runc под капотом. Начиная с Kubernetes 1.20, dockershim был признан устаревшим. Почему Kubernetes отказался от Docker как container runtime? Что изменилось с точки зрения CRI (Container Runtime Interface)?

**Категория:** Контейнеры/DevOps · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Updated: Dockershim Removal FAQ | Kubernetes](https://kubernetes.io/blog/2022/02/17/dockershim-faq/)

Dockershim был временным адаптерным слоем внутри kubelet, который позволял Kubernetes работать с Docker Engine, не реализующим интерфейс CRI. Само название «shim» (прокладка) отражало временный характер решения.

Причины удаления:
1. **Изначально временное решение** — Kubernetes Enhancement Proposal явно предусматривал его удаление.
2. **Бремя поддержки** — поддержка dockershim становилась всё более затратной для команды Kubernetes.
3. **Несовместимость со стандартами** — Docker Engine не реализует CRI, требуя специальный код-адаптер.
4. **Блокировка новых возможностей** — cgroups v2, user namespaces и другие функции плохо работали через dockershim.

Таймлайн: v1.20 (декабрь 2020) — объявление устаревания; v1.24 (2022) — полное удаление.

Все образы, собранные через `docker build`, продолжают работать с любым CRI-совместимым рантаймом — они соответствуют OCI-стандарту.

#### Ответ от Claude

**Исторический контекст: почему появился dockershim**

Когда Kubernetes создавался (2014), Docker был единственным доминирующим инструментом контейнеризации. Kubernetes изначально был жёстко привязан к Docker Engine как единственному container runtime. По мере роста экосистемы появились альтернативы (rkt, hypernetes), и в 2016 году был введён **Container Runtime Interface (CRI)** — стандартизированный gRPC-интерфейс между kubelet и container runtime.

Docker Engine не реализовывал CRI (и не планировал). Чтобы не сломать существующих пользователей, внутри kubelet добавили **dockershim** — небольшую прослойку, транслирующую CRI-вызовы kubelet в Docker API.

**Архитектура до и после**

До (с dockershim):
```
kubelet → dockershim (в kubelet) → Docker Engine → containerd → runc → контейнер
```

После (containerd напрямую):
```
kubelet → CRI plugin (в containerd) → runc → контейнер
```

Или с CRI-O:
```
kubelet → CRI-O → runc/crun → контейнер
```

Весь слой Docker Engine (с его daemon-процессом, docker CLI, swarm, volumes API и пр.) был лишним — kubelet использовал лишь малую часть функциональности Docker.

**Что такое CRI**

CRI — это protobuf/gRPC интерфейс, определяющий два сервиса:
- **RuntimeService** — управление Pod'ами и контейнерами (RunPodSandbox, CreateContainer, StartContainer, StopContainer и др.).
- **ImageService** — управление образами (PullImage, RemoveImage и др.).

Любой container runtime, реализующий этот интерфейс, может быть использован Kubernetes без специального кода в kubelet.

**Причины отказа от dockershim**

1. **Двойная трансляция** — запрос проходил через dockershim → Docker API → containerd. Лишний слой снижал производительность и усложнял отладку.
2. **Техдолг** — поддержка vendor-специфичного кода в ядре Kubernetes противоречила принципам open source и открытых стандартов.
3. **Заблокированные функции** — cgroups v2, user namespaces, RuntimeClass — всё это было труднее реализовать через dockershim.
4. **Несоответствие OCI/CRI** — Docker Engine имеет собственный API поверх containerd, тогда как containerd имеет нативный CRI-плагин (cri-plugin).

**Что изменилось для пользователей**

- **Образы не затронуты** — `docker build` создаёт OCI-образы, работающие с любым CRI-рантаймом.
- **Команды внутри Pod'а** — не изменились (exec, logs работают через CRI).
- **Изменилось** — `docker ps` на ноде больше не покажет контейнеры Kubernetes (нужно `crictl ps`).
- **Изменилось** — если кластер использовал Docker socket (`/var/run/docker.sock`) для DinD (Docker in Docker) — требуется миграция.

**Альтернативные рантаймы**

| Runtime | Описание |
|---------|----------|
| **containerd** | Де-факто стандарт, нативный CRI-плагин, лёгкий, высокая производительность |
| **CRI-O** | Создан специально для Kubernetes, минималистичный |
| **cri-dockerd** | Внешний shim от Mirantis/Docker для тех, кто хочет сохранить Docker Engine |

**Версионная граница**: Kubernetes 1.20 — deprecation warning; Kubernetes 1.24 (май 2022) — dockershim полностью удалён из kubelet.

### Вопрос 30. Readiness probe у сервиса ходит в стороннюю БД или очередь для проверки. Какие проблемы это создаёт при каскадных сбоях? Как правильно проектировать readiness probe для Java-приложения, чтобы не получить эффект thundering herd?

**Категория:** Контейнеры/DevOps · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Kubernetes Readiness and Liveness Probes in Spring Boot Apps | Alexander Obregon, Medium](https://medium.com/@AlexanderObregon/kubernetes-readiness-and-liveness-probes-in-spring-boot-apps-71b6e639f869)

В Spring Boot readiness probe включается через:
```
management.endpoint.health.probes.enabled=true
```
Это открывает эндпоинт `/actuator/health/readiness`.

Можно добавить зависимости в readiness-группу:
```
management.endpoint.health.group.readiness.include=readinessState,db,redis
```

Критическая проблема: если база данных или внешняя зависимость включена в readiness-проверку, сбой БД отмечает Pod как not-ready, и трафик к нему прекращается. Это создаёт риск каскадного сбоя: при падении БД весь сервис становится недоступным вместо graceful degradation.

Liveness probe, в отличие от readiness, должна фокусироваться на том, «жив» ли сам процесс, а не на внешних зависимостях — это предотвращает сценарий, когда временный сбой БД перезапускает все контейнеры.

#### Ответ от Claude

**Что такое thundering herd в контексте readiness probe**

Thundering herd (эффект толпы) применительно к readiness probe — это ситуация, когда:
1. Внешняя зависимость (БД, очередь) испытывает нагрузку или кратковременный сбой.
2. Все Pod'ы одновременно возвращают `Not Ready` → Kubernetes убирает их из EndpointSlice.
3. Все входящие запросы начинают возвращать ошибки (503).
4. БД восстанавливается → все Pod'ы одновременно начинают проверку readiness.
5. Сотни Pod'ов одновременно устанавливают соединения с БД → пиковая нагрузка → БД снова перегружена.
6. Цикл повторяется.

**Проблемы при каскадных сбоях**

1. **Самоусиливающийся сбой** — readiness probe, проверяющий БД, превращает кратковременный тайм-аут БД в полный outage сервиса.

2. **Эффект лавины при восстановлении** — все Pod'ы, становясь Ready одновременно, получают весь накопленный трафик (retry-шторм) и снова перегружают только что восстановившуюся БД.

3. **Ложные срабатывания** — сетевое моргание (network blip) длиной в несколько секунд может убрать Pod из ротации на `failureThreshold * periodSeconds` секунд.

4. **Проблема с liveness probe** — если ту же проверку БД поставить в liveness probe (худший сценарий), все Pod'ы начнут перезапускаться, что создаёт дополнительный storm соединений при старте каждого JVM.

**Принципы правильного проектирования readiness probe**

**Принцип 1: Разделить ответственность**

| Probe | Цель | Что проверять |
|-------|------|----------------|
| **Liveness** | «Жив ли процесс?» | Только внутреннее состояние JVM, deadlock-свободность |
| **Readiness** | «Готов ли принимать трафик?» | Прогрев кэша, инициализация пула соединений |
| **Startup** | «Завершился ли старт?» | Факт загрузки приложения (для медленных JVM) |

**Принцип 2: Readiness probe не должен проверять внешние зависимости в рантайме**

Правило из сообщества (srcco.de, learnk8s.io): проверка внешних зависимостей уместна только при **старте** (через startupProbe или initialDelaySeconds) — убедиться, что соединение установлено до начала приема трафика. В рантайме этого делать не следует.

**Принцип 3: Внутреннее состояние вместо внешних вызовов**

Вместо «иду в БД и проверяю» — держать внутренний флаг/индикатор здоровья:
```java
// Readiness endpoint
@GetMapping("/health/ready")
public ResponseEntity<String> readiness() {
    if (!connectionPool.isHealthy()) {
        // Pool сам мониторит соединения в фоне
        return ResponseEntity.status(503).body("DB pool unhealthy");
    }
    return ResponseEntity.ok("ready");
}
```

Пул соединений (HikariCP, c3p0) мониторит соединения самостоятельно — readiness probe проверяет статус пула, а не делает запрос в БД.

**Принцип 4: Circuit breaker вместо прямых проверок**

Если проверка внешней зависимости действительно нужна, её следует обернуть в circuit breaker (Resilience4j):
- Пока circuit открыт (зависимость недоступна), probe возвращает cached-статус последнего успешного состояния или degraded-режим.
- Это предотвращает эффект thundering herd: Pod может оставаться Ready, возвращая деградированные ответы, вместо того чтобы полностью выходить из ротации.

**Принцип 5: Настройка параметров для смягчения thundering herd**

```yaml
readinessProbe:
  httpGet:
    path: /actuator/health/readiness
    port: 8080
  initialDelaySeconds: 15       # Дать JVM время прогреться
  periodSeconds: 10             # Не проверять слишком часто
  timeoutSeconds: 3             # Разумный тайм-аут
  failureThreshold: 3           # 3 провала подряд = not-ready (30 сек буфер)
  successThreshold: 1           # 1 успех возвращает в ротацию

startupProbe:
  httpGet:
    path: /actuator/health/readiness
    port: 8080
  failureThreshold: 30          # 30 * 10 сек = 5 минут на старт JVM
  periodSeconds: 10
```

**Практика для Spring Boot**

```properties
# Включить Kubernetes probes
management.endpoint.health.probes.enabled=true

# Readiness группа - ТОЛЬКО внутреннее состояние
management.endpoint.health.group.readiness.include=readinessState

# Liveness группа - только жизнеспособность процесса
management.endpoint.health.group.liveness.include=livenessState

# НЕ включать db, redis, diskSpace в readiness для runtime-проверок
# management.endpoint.health.group.readiness.include=readinessState,db  <- ОПАСНО
```

Spring Boot Actuator автоматически управляет `readinessState` через `ApplicationEventPublisher` — при недостатке ресурсов или перегрузке приложение само себя помечает как not-ready без внешних проверок.

**Итог: антипаттерн vs паттерн**

Антипаттерн:
```
readinessProbe → HTTP GET /health → приложение → SELECT 1 FROM dual → БД
```

Паттерн:
```
readinessProbe → HTTP GET /actuator/health/readiness → приложение → HikariCP.isRunning()
                                                                    ↑
                                                 HikariCP сам следит за пулом в фоне
```

### Вопрос 31. Опишите полный жизненный цикл запроса к Kubernetes Service типа ClusterIP: как работает kube-proxy, iptables/ipvs правила, DNS-резолюция через CoreDNS — и где могут быть узкие места при высокой нагрузке?

**Категория:** Контейнеры/DevOps · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Virtual IPs and Service Proxies | Kubernetes](https://kubernetes.io/docs/reference/networking/virtual-ips/)

Каждый экземпляр kube-proxy следит за control plane Kubernetes на предмет добавления и удаления объектов Service и EndpointSlice. Для каждого Service kube-proxy вызывает соответствующие API (в зависимости от режима kube-proxy) для настройки узла: захватить трафик к ClusterIP:port сервиса и перенаправить его к одному из endpoint'ов.

В режиме **iptables** (дефолт для Linux): для каждого endpoint kube-proxy устанавливает цепочку iptables-правил, которые выбирают backend-под случайным образом (или по session affinity). Трафик идёт: ClusterIP:port → цепочка правил сервиса → цепочка правил endpoint → Destination NAT (DNAT) → IP backend-пода:port.

В режиме **IPVS**: используется hash-таблица и алгоритмы балансировки уровня ядра (round-robin, least-connection и др.). Сложность обработки соединения O(1) — не зависит от количества сервисов в кластере.

В режиме **nftables** (новый, Linux): современная замена iptables с улучшенной производительностью.

При масштабировании с iptables: при 10 000 подов и сервисов накапливаются десятки тысяч правил; синхронизация при изменении сервиса занимает значительное время. Ключевые параметры tuning: `minSyncPeriod` (агрегация изменений) и `syncPeriod` (полная периодическая синхронизация). С Kubernetes 1.28+ поддерживается incremental update — обновляются только изменившиеся правила.

#### Ответ от Claude

**Полный жизненный цикл запроса к ClusterIP-сервису:**

**1. DNS-резолюция через CoreDNS**

CoreDNS разворачивается как Pod (обычно 2 реплики) и регистрируется как сервис `kube-dns` в namespace `kube-system`. Каждый Pod получает в `/etc/resolv.conf`:
```
nameserver 10.96.0.10   # ClusterIP сервиса kube-dns
search default.svc.cluster.local svc.cluster.local cluster.local
options ndots:5
```

Когда приложение обращается к `my-service`, резолвер ядра видит, что в имени меньше 5 точек (настройка `ndots:5`), и последовательно пробует:
1. `my-service.default.svc.cluster.local` → CoreDNS возвращает ClusterIP
2. Если не найдено — `my-service.svc.cluster.local`, `my-service.cluster.local` и т. д.

CoreDNS хранит зону `cluster.local` в памяти, отвечает практически мгновенно. Внешние имена форвардятся в upstream DNS.

**2. Пакет уходит с клиентского пода на ClusterIP**

ClusterIP — виртуальный IP, который не назначен ни на один сетевой интерфейс. Его задача — быть точкой перехвата. Пакет с `dst=ClusterIP:port` уходит с пода и попадает в цепочку netfilter на узле.

**3. kube-proxy наблюдает за API server**

kube-proxy запускается как DaemonSet на каждом узле. Он watch'ит объекты Service и EndpointSlice через API server. При изменении (новый Pod, изменение порта, удаление) kube-proxy обновляет правила маршрутизации в ядре.

**4. Режим iptables (default)**

kube-proxy создаёт цепочки:
- `KUBE-SERVICES` — точка входа, перебирает все сервисы
- `KUBE-SVC-<hash>` — цепочка конкретного сервиса, содержит правила с вероятностным выбором (statistic mode nth): первый endpoint выбирается с вероятностью 1/N, второй — 1/(N-1) и т. д.
- `KUBE-SEP-<hash>` — цепочка endpoint'а, выполняет DNAT: меняет dst с ClusterIP на реальный PodIP

Обратный трафик: ядро хранит conntrack-запись, SNAT при ответе происходит автоматически, клиент получает ответ как будто от ClusterIP.

**5. Режим IPVS**

kube-proxy создаёт dummy-интерфейс `kube-ipvs0` и назначает на него все ClusterIP адреса. Ядро перехватывает трафик через IPVS (IP Virtual Server) — специализированный L4 балансировщик в ядре Linux. Используется hash-таблица: O(1) lookup вместо O(N) обхода цепочек iptables. Поддерживает алгоритмы: rr, lc, dh, sh, sed, nq.

**Где узкие места при высокой нагрузке:**

| Узкое место | Причина | Решение |
|---|---|---|
| iptables O(N) | С ростом числа сервисов/подов время обработки пакетов растёт линейно | Перейти на IPVS или nftables |
| iptables синхронизация | Добавление/удаление endpoint'а требует перезаписи всей цепочки | IPVS: атомарное обновление одной записи |
| CoreDNS ndots:5 | При обращении к external API каждый запрос порождает 5 DNS-запросов (4 NXDOMAIN + 1 успешный) | Понизить `ndots` до 2-3, использовать FQDN с точкой на конце |
| CoreDNS throughput | Малое число реплик CoreDNS под высокой нагрузкой | Масштабировать CoreDNS, включить `cache` плагин, настроить `autopath` |
| conntrack таблица | При большом числе соединений таблица conntrack переполняется | Увеличить `nf_conntrack_max`, мониторить метрики |
| kube-proxy CPU | Частые изменения endpoint'ов → постоянная перезапись правил | Оптимизировать `minSyncPeriod`, использовать EndpointSlices |

**Схема потока запроса:**
```
[Client Pod]
     |
  resolv.conf → CoreDNS → ClusterIP
     |
  TCP connect to ClusterIP:port
     |
  netfilter (PREROUTING chain)
     |
  KUBE-SERVICES → KUBE-SVC-xxx → KUBE-SEP-yyy
     |
  DNAT: dst = PodIP:containerPort
     |
  [Backend Pod]
     |
  ответ → conntrack SNAT → [Client Pod]
```

**Важно для production:** начиная с Kubernetes 1.19, рекомендуется использовать EndpointSlices вместо Endpoints (меньше нагрузка на API server при большом числе подов), а с 1.28+ — режим IPVS или nftables для кластеров с более чем ~500 сервисами.

### Вопрос 32. Что такое QoS классы (Guaranteed, Burstable, BestEffort) в Kubernetes? Как requests и limits на CPU/memory влияют на то, какой под будет evicted первым при нехватке ресурсов на ноде? Как это учитывать для Java-сервисов, где heap size задаётся отдельно?

**Категория:** Контейнеры/DevOps · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Pod Quality of Service Classes | Kubernetes](https://kubernetes.io/docs/concepts/workloads/pods/pod-qos/)

Kubernetes присваивает каждому поду один из трёх классов QoS на основе ограничений ресурсов, заданных для контейнеров. Эта классификация используется для принятия решений о том, какие поды будут evicted при нехватке ресурсов на узле.

**Guaranteed** — строжайшие гарантии. Критерии: каждый контейнер обязан иметь limit и request по памяти и CPU, при этом request должен быть равен limit. Такие поды убиваются последними — только если они превышают свои limits или нет подов с более низким приоритетом.

**Burstable** — средний приоритет. Критерии: не соответствует Guaranteed, но хотя бы один контейнер имеет request или limit по памяти или CPU. Evict'ятся только после того, как все BestEffort-поды уже удалены.

**BestEffort** — наименьшие гарантии. Критерии: ни один контейнер не имеет ни requests, ни limits. Kubelet предпочитает evict'ить эти поды первыми при resource pressure.

При eviction из-за давления ресурсов кандидатами на удаление являются только поды, **превышающие свои resource requests**. QoS-класс является иммутабельным: присваивается при создании пода и не меняется в течение его жизни.

#### Ответ от Claude

**QoS-классы: точные критерии**

| Класс | CPU request | CPU limit | Memory request | Memory limit | Примечание |
|---|---|---|---|---|---|
| **Guaranteed** | обязателен | обязателен, = request | обязателен | обязателен, = request | Все контейнеры в поде |
| **Burstable** | хотя бы один задан | опционально | хотя бы один задан | опционально | limit != request или частично не задан |
| **BestEffort** | не задан | не задан | не задан | не задан | Полное отсутствие resources |

*Хинт:* если задан только limit без request — Kubernetes автоматически устанавливает request = limit. Такой под будет **Guaranteed**, а не Burstable.

**Механизм eviction при нехватке ресурсов**

Kubelet отслеживает два вида давления на узле:
- **Memory pressure** — сигнал из cgroups или `/proc/meminfo`
- **DiskPressure** — заполнение файловой системы

Порядок eviction:
1. **BestEffort** — evict'ятся первыми, вне зависимости от фактического потребления
2. **Burstable, превышающие request** — evict'ятся в порядке убывания (кто больше превысил request в процентном отношении, тот первый)
3. **Guaranteed** — evict'ятся только если BestEffort и Burstable исчерпаны, или если поды превышают свои limits (в этом случае происходит OOMKill от ядра, а не plановый eviction)

**Разница OOMKill и Eviction:**
- **OOMKill** (exit code 137) — ядро Linux убивает контейнер, превысивший cgroup memory limit. Немедленно, без предупреждения. Контролируется limits.
- **Eviction** — kubelet проактивно завершает под при Node pressure. Контролируется requests и QoS-классом.

**Java-специфика: почему стандартный подход не работает**

JVM потребляет память не только через heap. Полная модель памяти процесса JVM:

```
Общее потребление ≈ Heap (-Xmx)
                  + Metaspace (классы, дефолт ~256 MB, может расти)
                  + Code Cache (JIT-скомпилированный код, ~256 MB)
                  + (threads × stack_size) (дефолт 512 KB на поток)
                  + Direct Buffers (NIO, Netty, часто несколько сотен MB)
                  + JVM overhead (~100 MB)
```

**Типичная ошибка:** установить `-Xmx` = memory limit контейнера.
```yaml
# ПЛОХО: OOMKill неизбежен
resources:
  limits:
    memory: "1Gi"
  requests:
    memory: "1Gi"
# И в Dockerfile или env:
# JAVA_OPTS=-Xmx1g
```
JVM займёт 1 ГБ heap + ~500 MB non-heap → итого ~1.5 ГБ → ядро убьёт процесс (OOMKill).

**Правильный подход: -XX:MaxRAMPercentage**

С JDK 8u191+ и JDK 11+ JVM умеет читать cgroup memory limits:
```yaml
resources:
  requests:
    memory: "1536Mi"
  limits:
    memory: "1536Mi"
```
```
JAVA_OPTS=-XX:MaxRAMPercentage=65.0
         -XX:InitialRAMPercentage=50.0
         -XX:+UseContainerSupport  # включён по дефолту с JDK 11
```

При `limit=1536Mi` и `MaxRAMPercentage=65` → heap ≈ 1 ГБ, остаток ≈ 512 МБ для non-heap. Это safe-зона.

**Как устанавливать QoS для Java-сервисов:**

```yaml
# РЕКОМЕНДУЕТСЯ для критичных Java-сервисов (Guaranteed):
resources:
  requests:
    memory: "1536Mi"   # request = limit → Guaranteed
    cpu: "1000m"
  limits:
    memory: "1536Mi"
    cpu: "1000m"
```

Если нужен Burstable (CPU-burst, но стабильная память):
```yaml
resources:
  requests:
    memory: "1536Mi"   # memory request = limit → no OOMKill-eviction surprise
    cpu: "500m"        # CPU request < limit → Burstable, но CPU может бурстить
  limits:
    memory: "1536Mi"
    cpu: "2000m"
```

**Рекомендации для Java в Kubernetes:**

1. **Никогда не задавайте** `-Xmx` >= memory limit — всегда оставляйте 25–35% для non-heap
2. **Используйте `-XX:MaxRAMPercentage`** вместо хардкода `-Xmx`, чтобы JVM сам адаптировался к cgroup limit
3. **Memory request = memory limit** для production-сервисов — это гарантирует Guaranteed QoS и предотвращает неожиданный eviction
4. **Мониторьте Metaspace**: утечки classloader (особенно в приложениях с динамической загрузкой классов — Spring, OSGi) — частая причина OOMKill в Java, которую не видно в heap-метриках
5. **Не смешивайте QoS**: если один init-container BestEffort, а основной Guaranteed — весь под получит Burstable

*Нарушение в Java-проектах:*
- Сборка приложения прямо при запуске контейнера (`CMD mvn spring-boot:run`)
- Один Docker-образ содержит JDK, Maven и runtime — нет разделения фаз, образ раздут
- `kubectl exec` + редактирование конфигов внутри запущенного контейнера — нарушение неизменности релиза

**Фактор VI — Processes (Процессы)**

- Хранение сессий в памяти (HttpSession в памяти Tomcat) — при перезапуске или масштабировании сессии теряются. Решение: Spring Session + Redis
- Использование `Singleton`-кэша в JVM-памяти как разделяемого состояния между запросами (Guava Cache, Caffeine без репликации). При горизонтальном масштабировании разные реплики видят разное состояние
- EJB Stateful Session Beans — архитектурно требуют sticky sessions

---

**Фактор IX — Disposability (Ликвидируемость)**

*Влияние на K8s-манифесты:*
```yaml
# ПРАВИЛЬНО: graceful shutdown и liveness/readiness пробы
spec:
  containers:
  - name: app
    lifecycle:
      preStop:
        exec:
          command: ["sh", "-c", "sleep 5"]  # дать time load balancer убрать pod
    livenessProbe:
      httpGet:
        path: /actuator/health/liveness
        port: 8080
      initialDelaySeconds: 60   # время на прогрев JVM
    readinessProbe:
      httpGet:
        path: /actuator/health/readiness
        port: 8080
  terminationGracePeriodSeconds: 30
```

*Нарушение в Java-проектах:*
- Отсутствие обработки `SIGTERM` — Spring Boot по умолчанию делает graceful shutdown (с 2.3+), но нужно включить `server.shutdown=graceful`
- Не задан `initialDelaySeconds` для readinessProbe при тяжёлых Spring Boot приложениях — K8s начинает слать трафик до того, как ApplicationContext поднят, что приводит к 503
- Долгий старт JVM + прогрев JIT (30–60 секунд для большого приложения) без соответствующих настроек проб

---

**Фактор X — Dev/Prod Parity (Паритет сред)**

*Нарушение в Java-проектах:*
- Разные версии зависимостей: H2 в dev, PostgreSQL в prod — SQL-запросы работают по-разному
- `docker-compose.yml` с одним образом, а в prod — другой базовый образ (другая версия JDK, другой дистрибутив)
- Использование `spring.profiles.active=dev` в Dockerfile без переопределения в K8s — образ несёт в себе dev-конфиг

---

**Фактор XI — Logs (Логи)**

*Влияние на Dockerfile и K8s:*
```yaml
# Нет специальной настройки в манифесте — достаточно писать в stdout
# Kubernetes автоматически собирает stdout/stderr контейнеров
```

*Нарушение в Java-проектах:*
- Logback/Log4j2 настроен на запись в файл внутри контейнера (`/var/log/app/app.log`) — логи теряются при пересоздании пода, недоступны через `kubectl logs`
- Асинхронный аппендер с буферизацией — при OOMKill буфер не сбрасывается, последние события теряются
- Запись в файл и ротация средствами log4j2 — дублирование с механизмами K8s log rotation

### Вопрос 34. В CI/CD конвейере образ собирается один раз и продвигается через среды (dev → staging → prod) без пересборки. Как это соотносится с принципом Build/Release/Run из 12-factor app? Как управлять конфигурацией, специфичной для среды, если образ неизменяем?

**Категория:** Контейнеры/DevOps · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [V. Build, release, run — The Twelve-Factor App](https://12factor.net/build-release-run)

Двенадцатифакторное приложение использует строгое разделение между стадиями сборки, выпуска и запуска.

**Стадия сборки (Build)** преобразует репозиторий кода в исполняемый пакет, называемый «сборкой». На этой стадии получают зависимости, компилируют бинарники и активы.

**Стадия выпуска (Release)** берёт результат сборки и объединяет его с конфигурацией текущей среды, производя уникально идентифицируемый развёртываемый юнит — «релиз», — готовый к немедленному исполнению.

**Стадия запуска (Run)** запускает приложение в исполняемой среде, используя выбранный релиз.

Релизы неизменяемы и образуют список только для добавления — релиз нельзя изменить после создания. Любое изменение должно порождать новый релиз. Это принципиально: невозможно изменить код в рантайме, потому что нет способа передать изменение обратно на стадию сборки.

Конфигурацию двенадцатифакторное приложение хранит в переменных среды (Factor III — Config). Переменные среды: легко меняются между деплоями без изменения кода; маловероятно будут случайно закоммичены в репозиторий; являются языково- и ОС-независимым стандартом. Конфигурация — это всё, что отличается между деплоями (staging, production, developer environments): Database URL, credentials, per-deploy значения.

#### Ответ от Claude

**Соответствие принципу Build/Release/Run**

Практика «собрать образ один раз и продвигать через среды без пересборки» является точной реализацией принципа Build/Release/Run из 12-factor app. Разберём соответствие:

- **Build** (сборка) — Docker-образ собирается в CI один раз из конкретного git-коммита. Это неизменяемый артефакт, привязанный к ревизии кода.
- **Release** (релиз) — образ + конфигурация среды = конкретный релиз. Каждое окружение (dev, staging, prod) формирует свой релиз из одного образа, но с разными переменными среды или ConfigMap/Secret.
- **Run** (запуск) — Kubernetes или другой оркестратор запускает релиз в нужной среде.

Пересборка образа для каждой среды нарушает принцип: разные сборки дают разные бинарники, что делает трассировку ненадёжной («что именно прошло тестирование?») и усложняет откат.

**Управление конфигурацией при неизменяемом образе**

Поскольку образ не должен содержать конфигурацию конкретной среды, конфигурация передаётся в контейнер извне:

1. **Переменные среды** (12-factor: Factor III) — передаются через `env` в Pod spec или через `envFrom` из ConfigMap/Secret. Это основной механизм, рекомендуемый 12-factor.

2. **Kubernetes ConfigMap** — хранит нечувствительную конфигурацию (URLs, флаги фич, таймауты). Монтируется либо как файл (volume), либо как env-переменные. ConfigMap можно пометить как `immutable: true` для защиты от случайных изменений.

3. **Kubernetes Secret** — для чувствительных данных (пароли, токены, TLS-сертификаты). Механизм тот же, но значения base64-кодированы и контролируются RBAC. В production рекомендуется интеграция с Vault или external-secrets-operator.

4. **Init-контейнеры** — если конфигурация требует сложной инициализации, init-контейнер может вычитать её из внешнего хранилища и записать в shared volume перед стартом основного контейнера.

**Пример: одинаковый образ в разных средах**

```yaml
# dev namespace
env:
  - name: DATABASE_URL
    valueFrom:
      configMapKeyRef:
        name: app-config-dev
        key: database_url
  - name: DB_PASSWORD
    valueFrom:
      secretKeyRef:
        name: app-secrets-dev
        key: password

# prod namespace — тот же образ, другой ConfigMap/Secret
env:
  - name: DATABASE_URL
    valueFrom:
      configMapKeyRef:
        name: app-config-prod
        key: database_url
```

**Важные ограничения и практические нюансы**

- **Что не должно быть в образе**: credentials, URLs баз данных, feature flags — всё, что меняется между средами.
- **Что допустимо в образе**: версия приложения, статические ресурсы, бинарники, дефолтные значения конфигурации, которые можно переопределить снаружи.
- **Версионирование**: каждый образ должен иметь уникальный тег (commit SHA или семантическая версия). Тег `latest` недопустим в production — нарушает трассируемость.
- **Frontend-особенность**: для SPA/React-приложений передача конфигурации через env-переменные во время сборки (вшитые в JS-бандл) нарушает принцип «одна сборка для всех сред». Решение — runtime-конфигурация через `/config.json` endpoint или entrypoint-скрипт, который генерирует конфигурационный файл из env перед запуском nginx.

**Итог**: «Build once, promote everywhere» — это прямая реализация 12-factor Factor V. Конфигурация среды живёт вне образа — в Kubernetes ConfigMap, Secret, env-переменных — и инжектируется в момент создания Pod (стадия Release). Образ остаётся неизменяемым, а релиз уникален для каждой среды.

### Вопрос 35. Как правильно передавать JVM-параметры (-Xmx, -Xms, GC настройки) в контейнеризованное Java-приложение так, чтобы они соотносились с limits в Kubernetes, а не с памятью хост-машины? Что такое UseContainerSupport и с какой версии JDK он включён по умолчанию?

**Категория:** Контейнеры/DevOps · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [What Does the UseContainerSupport VM Parameter Do in Docker? — javathinking.com](https://www.javathinking.com/blog/what-does-usecontainersupport-vm-parameter-do/)

`UseContainerSupport` появился в Java 10 (март 2018) как стабильная фича и был бэкпортирован в Java 8 Update 191 (июль 2018) как экспериментальная функциональность.

Параметр позволяет JVM «определять и соблюдать ограничения ресурсов, установленные Docker-контейнерами». Без него Java считает, что у неё есть доступ ко всем ресурсам хоста, что потенциально приводит к чрезмерному выделению памяти или неэффективному планированию CPU.

Когда UseContainerSupport включён, JVM читает Linux cgroup-файлы для определения ограничений контейнера, а не характеристик хоста. Для памяти она «использует лимит памяти контейнера для вычисления heap и других регионов памяти (metaspace, стек и т.д.)», а не предполагает доступность всей памяти хоста.

Поведение по умолчанию:
- Java 8u191–8u242: отключён по умолчанию, требует явного флага `-XX:+UseContainerSupport`
- Java 8u251+: включён по умолчанию
- Java 10+: включён по умолчанию

Java 11+ по умолчанию выставляет `MaxRAMPercentage=25`, выделяя лишь 25% памяти контейнера под heap. Рекомендуется устанавливать 70–80% для большинства приложений, оставляя 20–30% для non-heap памяти.

#### Ответ от Claude

**Проблема без UseContainerSupport**

До введения container awareness JVM определяла доступную память через системные вызовы, возвращающие параметры хостовой машины — например, узла Kubernetes с 128 ГБ RAM. JVM применяла эвристику «heap = 25% от RAM», получала 32 ГБ, хотя Pod был ограничен 2 ГБ. Это приводило к немедленному OOMKill со стороны ядра Linux (cgroup limit) ещё до полной инициализации JVM.

**Что такое UseContainerSupport**

`-XX:+UseContainerSupport` — флаг JVM, включающий чтение ограничений ресурсов из Linux cgroups (v1 и v2) вместо чтения параметров хоста. При включённом флаге JVM читает:
- `/sys/fs/cgroup/memory/memory.limit_in_bytes` (cgroups v1)
- `/sys/fs/cgroup/memory.max` (cgroups v2)

На основании этих значений JVM вычисляет дефолтный размер heap, выбирает GC-алгоритм, определяет дефолтное количество потоков в ForkJoinPool и настраивает thread pools.

**История версий**

| JDK версия | Статус UseContainerSupport |
|---|---|
| JDK 8u131–8u190 | Экспериментальный флаг `-XX:+UnlockExperimentalVMOptions -XX:+UseCGroupMemoryLimitForHeap` |
| JDK 8u191–8u242 | Бэкпортирован как `-XX:+UseContainerSupport`, но **отключён по умолчанию** |
| JDK 8u251+ | **Включён по умолчанию** |
| JDK 10+ | **Включён по умолчанию** |
| JDK 17+ | Включён по умолчанию, поддерживает cgroups v1 и v2 |

Важно: старый экспериментальный флаг `UseCGroupMemoryLimitForHeap` устарел и не должен использоваться совместно с `UseContainerSupport`.

**Управление памятью heap**

Вместо абсолютных значений `-Xmx`/`-Xms` рекомендуется использовать процентные флаги:

```bash
# Выделить 75% памяти контейнера под heap
-XX:MaxRAMPercentage=75.0
-XX:InitialRAMPercentage=50.0

# Если нужна точная цифра — можно использовать Xmx,
# но тогда нужно вручную коррелировать с limits в Kubernetes
-Xmx1536m  # для контейнера с limits.memory: 2Gi
```

Дефолтное значение `MaxRAMPercentage=25.0` — типично слишком низкое. Для production рекомендуется 70–80%, оставляя 20–30% для non-heap.

**Структура памяти JVM в контейнере**

UseContainerSupport решает только проблему heap, но общее потребление памяти JVM шире:

```
Heap                (MaxRAMPercentage — управляется JVM)
Metaspace           (по умолчанию unbounded — нужен -XX:MaxMetaspaceSize)
Thread stacks       (~512KB–1MB на поток × число потоков)
Direct Memory       (NIO, Netty — управляется -XX:MaxDirectMemorySize)
JVM internal code   (JIT-компилятор, CodeCache — -XX:ReservedCodeCacheSize)
```

Типичная формула для Kubernetes limits:
```
memory limit = MaxHeap + Metaspace + DirectMemory + threads*stackSize + overhead
```
Упрощённо: `limits.memory = Xmx * 1.3` — но это грубая оценка, требует профилирования.

**Пример конфигурации Pod в Kubernetes**

```yaml
resources:
  requests:
    memory: "1Gi"
    cpu: "500m"
  limits:
    memory: "2Gi"
    cpu: "2"
env:
  - name: JAVA_OPTS
    value: >-
      -XX:+UseContainerSupport
      -XX:MaxRAMPercentage=75.0
      -XX:InitialRAMPercentage=50.0
      -XX:MaxMetaspaceSize=256m
      -XX:+UseG1GC
      -XX:MaxGCPauseMillis=200
```

**GC-настройки относительно лимитов**

- `UseG1GC` — рекомендуется для большинства production-сценариев (heap > 4 ГБ)
- `UseZGC` или `UseShenandoahGC` — для требований к малым задержкам (heap > 8 ГБ)
- `UseSerialGC` — для контейнеров с 1 CPU (JVM определяет это автоматически при UseContainerSupport)
- `ParallelGCThreads` / `ConcGCThreads` JVM автоматически ограничивает по количеству CPU из cgroups

**Практические рекомендации**

1. Всегда используйте JDK 11+ (желательно 17+) для полной поддержки cgroups v1/v2.
2. Устанавливайте `MaxRAMPercentage` вместо абсолютного `-Xmx` для гибкости.
3. Явно ограничивайте Metaspace (`-XX:MaxMetaspaceSize`): без лимита Metaspace может вырасти и вытеснить heap.
4. Устанавливайте `resources.requests` и `resources.limits` — JVM читает именно `limits`.
5. Если `limits` не установлены — JVM при UseContainerSupport может читать «бесконечность» и использовать параметры хоста.

### Вопрос 36. Опишите стратегию canary deployment в Kubernetes без использования service mesh (только нативные ресурсы). Какие ограничения у этого подхода и где он ломается при stateful-трафике?

**Категория:** Контейнеры/DevOps · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Canary Deployment in Kubernetes: The Basics and a Quick Tutorial — Octopus Deploy / Codefresh](https://octopus.com/devops/kubernetes-deployments/canary-deployments/)

Canary deployment в Kubernetes без service mesh реализуется через два отдельных Deployment-ресурса (stable и canary) и один общий Service с label selector, который охватывает поды обоих деплойментов.

Сервис указывает только на общий лейбл (например, `app: nginx`), который есть у подов обоих деплойментов. Kubernetes Service равномерно балансирует трафик по всем подходящим подам, поэтому распределение трафика определяется соотношением числа реплик: 9 stable + 1 canary = 10% трафика на canary.

Управление трафиком — ручное: чтобы изменить долю трафика, нужно менять `replicas` в деплойментах через `kubectl scale`. Нет встроенной логики автоматического продвижения или отката. Точное процентное разделение (например, ровно 1% на canary) невозможно без большого числа подов — при 9+1 репликах это 10%, а не 1%.

Подход работает для простых сценариев с приблизительным разделением трафика; для точных весов, маршрутизации по заголовкам или автоматизации нужны service mesh (Istio, Linkerd) или специализированные операторы (Argo Rollouts, Flagger).

#### Ответ от Claude

**Реализация canary deployment нативными ресурсами Kubernetes**

Стратегия основана на трёх компонентах: два Deployment (stable + canary), один Service и общий label.

```yaml
# Deployment: stable версия
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-stable
spec:
  replicas: 9
  selector:
    matchLabels:
      app: myapp
      track: stable
  template:
    metadata:
      labels:
        app: myapp
        track: stable
    spec:
      containers:
      - name: myapp
        image: myapp:1.0.0
---
# Deployment: canary версия
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-canary
spec:
  replicas: 1
  selector:
    matchLabels:
      app: myapp
      track: canary
  template:
    metadata:
      labels:
        app: myapp
        track: canary
    spec:
      containers:
      - name: myapp
        image: myapp:1.1.0
---
# Service: матчит оба деплоймента по общему лейблу
apiVersion: v1
kind: Service
metadata:
  name: myapp-svc
spec:
  selector:
    app: myapp  # НЕ включает track: — матчит и stable, и canary
  ports:
  - port: 80
    targetPort: 8080
```

**Механика распределения трафика**

kube-proxy (или eBPF в случае Cilium) реализует балансировку через iptables/IPVS rules, распределяя запросы равномерно по всем endpoint'ам, соответствующим selector. Распределение пропорционально количеству готовых (Ready) подов:

```
9 stable podов + 1 canary pod = ~10% трафика на canary
```

Управление долей трафика — через kubectl scale:
```bash
# Увеличить canary до 20%
kubectl scale deployment myapp-stable --replicas=8
kubectl scale deployment myapp-canary --replicas=2

# Промоутировать: canary стал stable
kubectl scale deployment myapp-canary --replicas=10
kubectl scale deployment myapp-stable --replicas=0
```

**Ограничения нативного подхода**

1. **Нет точного весового контроля**: минимальная гранулярность — 1/N, где N — общее число подов. Для 1% трафика нужно 100 подов суммарно. Неприемлемо в большинстве сценариев.

2. **Нет маршрутизации по содержимому запроса**: невозможно направить только beta-пользователей или трафик с определённым заголовком (`X-Canary: true`) на canary. kube-proxy не анализирует L7-заголовки.

3. **Нет автоматического отката**: требуется внешняя система мониторинга и ручное или скриптовое вмешательство. Отсутствует встроенная проверка success rate/latency.

4. **Ручное управление**: масштабирование реплик error-prone, требует операционной дисциплины.

5. **Нет поддержки gRPC/WebSocket weighted routing**: Service балансирует на уровне TCP-соединений, что некорректно для протоколов с multiplexing.

**Где ломается при stateful-трафике**

Основная проблема: kube-proxy балансирует каждый новый TCP-соединение независимо. Для HTTP/1.1 (одно соединение = один запрос) это приемлемо, но для stateful-трафика возникают серьёзные проблемы:

1. **Session affinity через clientIP (частичное решение)**:
   Kubernetes Service поддерживает `sessionAffinity: ClientIP` (таймаут по умолчанию — 10800 секунд). Трафик от одного IP будет идти на один под. Но:
   - Один IP может скрывать тысячи пользователей (NAT, load balancer)
   - При рестарте пода sticky-сессия сбрасывается, следующий запрос попадёт на под другой версии
   - Не работает с canary: нельзя задать, чтобы один пользователь «прилип» именно к canary, а другой — к stable

2. **In-memory сессии**: если stable и canary хранят сессионное состояние в памяти (без внешнего хранилища), пользователь при переключении между версиями теряет сессию. Это типичная проблема для Java-приложений с HttpSession без репликации.

3. **Stateful protocolы (WebSocket, gRPC streaming)**: долгоживущие соединения «приклеиваются» к конкретному поду на всё время соединения. При canary это означает неравномерное распределение — первый под получит непропорционально много трафика. Масштабирование canary-деплоймента не изменит распределение существующих соединений.

4. **Database schema migrations**: если canary требует изменений схемы БД, обратно несовместимых со stable, одновременная работа двух версий невозможна. Нужны expand/contract migrations.

5. **Distributed cache/состояние**: если версии используют разные форматы сериализации (например, изменилась структура объекта в Redis), пользователь, переключившийся между версиями, получит ошибку десериализации.

**Решения при stateful-трафике**

- **Ingress-контроллер с поддержкой canary**: nginx-ingress поддерживает аннотации `nginx.ingress.kubernetes.io/canary: "true"` и `nginx.ingress.kubernetes.io/canary-weight: "10"` — это даёт точный весовой контроль на L7, но всё ещё без session-aware routing.
- **Service mesh (Istio, Linkerd)**: поддерживает weighted routing по VirtualService, cookie-based routing, header-based routing — решает большинство stateful-проблем.
- **Argo Rollouts / Flagger**: операторы прогрессивной доставки с автоматическим анализом метрик и откатом.
- **Внешнее хранилище сессий**: Redis/Memcached для HttpSession — снимает проблему in-memory state при переключении между версиями.

**Вывод**: нативный canary deployment в Kubernetes (два Deployment + один Service) работает для stateless HTTP-приложений с приблизительными требованиями к разделению трафика. Для stateful-трафика, точных весов или автоматизации нужны либо ingress-контроллер с canary-аннотациями, либо service mesh, либо специализированные операторы.
### Вопрос 37. Что такое HPA (HorizontalPodAutoscaler) в Kubernetes? По каким метрикам он умеет масштабировать поды, и как выглядит его конфигурация для Java-сервиса, который нужно масштабировать по CPU?

**Категория:** Контейнеры/DevOps · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Horizontal Pod Autoscaling | Kubernetes](https://kubernetes.io/docs/tasks/run-application/horizontal-pod-autoscale/)

HorizontalPodAutoscaler (HPA) — это ресурс Kubernetes API, который автоматически масштабирует рабочие нагрузки (Deployment, StatefulSet и другие масштабируемые объекты), изменяя количество реплик подов на основании наблюдаемых метрик. Это горизонтальное масштабирование в отличие от вертикального, которое добавляет ресурсы уже существующим подам.

Контроллер HPA работает как управляющий цикл с интервалом по умолчанию 15 секунд. На каждой итерации он запрашивает метрики, вычисляет желаемое количество реплик по формуле:

```
desiredReplicas = ceil(currentReplicas * (currentMetricValue / desiredMetricValue))
```

Если соотношение `currentMetricValue / desiredMetricValue` находится в пределах допуска (по умолчанию ±10%), масштабирование не выполняется.

**Поддерживаемые типы метрик (autoscaling/v2):**

1. **Resource** — CPU и память на уровне пода (в процентах от resource requests или в абсолютных значениях). Требует, чтобы у контейнеров были заданы `requests`. Данные поставляет Metrics Server через API `metrics.k8s.io`.
2. **ContainerResource** — аналогично Resource, но на уровне отдельного контейнера внутри пода (бета начиная с Kubernetes 1.27).
3. **Pods** — произвольные метрики на под из кастомного API `custom.metrics.k8s.io` (например, количество обрабатываемых запросов).
4. **Object** — единственная метрика, описывающая конкретный объект кластера (например, количество RPS на Ingress).
5. **External** — метрики из внешних систем (очередь сообщений, облачные сервисы) через API `external.metrics.k8s.io`.

**Пример конфигурации для Java-сервиса с масштабированием по CPU:**

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: java-service-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: java-service
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
      - type: Percent
        value: 50
        periodSeconds: 60
    scaleUp:
      stabilizationWindowSeconds: 0
      policies:
      - type: Percent
        value: 100
        periodSeconds: 15
```

`averageUtilization: 70` означает, что HPA будет стремиться поддерживать среднюю загрузку CPU по всем подам на уровне 70% от значения `requests.cpu`. `stabilizationWindowSeconds: 300` для scale-down предотвращает резкое уменьшение числа подов сразу после кратковременного снижения нагрузки.

#### Ответ от Claude

**Что такое HPA**

HorizontalPodAutoscaler — объект Kubernetes (API-группа `autoscaling`), который следит за метриками целевого ресурса (Deployment, StatefulSet, ReplicationController) и автоматически изменяет значение `.spec.replicas` в пределах заданного диапазона `minReplicas`..`maxReplicas`. Актуальная стабильная версия API — `autoscaling/v2`.

**Алгоритм масштабирования**

```
desiredReplicas = ceil(currentReplicas * (currentMetricValue / desiredMetricValue))
```

Пример: текущая утилизация CPU — 200m, целевая — 100m, текущих реплик — 3. Желаемых реплик: `ceil(3 * 200/100)` = 6.

Если отклонение от целевого значения не превышает порог допуска (по умолчанию 10%), масштабирование не выполняется. С Kubernetes 1.33 этот порог можно настраивать через feature gate `HPAConfigurableTolerance`.

**Типы метрик**

| Тип | Источник | Пример использования |
|---|---|---|
| Resource | Metrics Server (`metrics.k8s.io`) | CPU/память в % от requests |
| ContainerResource | Metrics Server | CPU конкретного контейнера в поде |
| Pods | Custom Metrics API | HTTP-запросов в секунду на под |
| Object | Custom Metrics API | RPS на Ingress-объект |
| External | External Metrics API | Длина очереди в Kafka/RabbitMQ |

**Поведение при масштабировании (behavior)**

`autoscaling/v2` позволяет настраивать политики scale-up и scale-down независимо:
- `stabilizationWindowSeconds` — окно стабилизации: HPA смотрит на все рекомендации за этот период и выбирает наиболее консервативную. По умолчанию 300 сек для scale-down, 0 для scale-up.
- `policies` — максимальная скорость изменения: можно ограничить изменение в подах (`Pods`) или в процентах (`Percent`) за период `periodSeconds`.

**Полная конфигурация для Java-сервиса**

Критически важно: контейнер должен иметь заданные `resources.requests.cpu`, иначе HPA не сможет вычислить процент утилизации.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: java-service
spec:
  replicas: 2
  selector:
    matchLabels:
      app: java-service
  template:
    metadata:
      labels:
        app: java-service
    spec:
      containers:
      - name: app
        image: myregistry/java-service:latest
        resources:
          requests:
            cpu: "500m"
            memory: "512Mi"
          limits:
            cpu: "1000m"
            memory: "1Gi"
        readinessProbe:
          httpGet:
            path: /actuator/health/readiness
            port: 8080
          initialDelaySeconds: 30
          periodSeconds: 10
---
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: java-service-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: java-service
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
      - type: Percent
        value: 50
        periodSeconds: 60
    scaleUp:
      stabilizationWindowSeconds: 0
      policies:
      - type: Percent
        value: 100
        periodSeconds: 15
      - type: Pods
        value: 4
        periodSeconds: 15
      selectPolicy: Max
```

**Особенности для Java-сервисов:**
- `readinessProbe` обязателен: HPA не включает поды без готовности в расчёт метрик, что предотвращает преждевременное масштабирование при длительном прогреве JVM.
- При использовании Spring Boot рекомендуется `/actuator/health/readiness` (доступен с Spring Boot 2.3+).
- Pods со статусом `Terminating` исключаются из расчёта.
- Поды, у которых нет метрик (например, только что запустились), учитываются консервативно: 0% при scale-up, 100% при scale-down.

**Предварительные требования:** в кластере должен быть установлен Metrics Server; для кастомных и внешних метрик — соответствующие адаптеры (например, Prometheus Adapter).

### Вопрос 38. Как JVM определяет количество доступных процессоров внутри контейнера? Почему CPU limit в Kubernetes не всегда правильно транслируется в ActiveProcessorCount, и как это влияет на размер ForkJoinPool, GC-потоки и производительность?

**Категория:** Контейнеры/DevOps · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Kubernetes CPU limits — When JVM sees more than it should | Mike My Bytes](https://mikemybytes.com/2026/03/12/kubernetes-cpu-limits-when-jvm-sees-more-than-it-should/)

JVM определяет количество доступных процессоров, читая настройки Linux control groups (cgroups). В системах с cgroups v2, если задан CPU limit, JVM вычисляет количество ядер из файла `cpu.max` по формуле `ceil(quota / period)`. Однако, если лимит не задан, cgroups v2 явно сообщает «неограниченное CPU», и JVM видит все физические ядра хост-узла, вне зависимости от того, какие `requests` заданы в спецификации пода.

Ключевая проблема: `cpu.requests` в Kubernetes вообще не видны JVM ни в cgroups v1, ни в cgroups v2. Это создаёт несоответствие: под, запрашивающий 3 CPU на узле с 32 ядрами, запустит JVM, которая увидит все 32 ядра и создаст пулы потоков под них.

Это затрагивает несколько подсистем:
- **JIT-потоки**: количество масштабируется логарифмически от числа ядер (от 2 потоков при 1 ядре до 15 при 32 ядрах).
- **GC-потоки (G1GC)**: `ParallelGCThreads` равно числу ядер до 8, затем `floor(5/8 * availableCPUs)`.
- **ForkJoinPool**: параллелизм по умолчанию `max(availableCPUs - 1, 1)`.
- **Scheduler виртуальных потоков (Java 21+)**: также использует число ядер.

Решение: явно задавать `-XX:ActiveProcessorCount=N`, где N соответствует реальному CPU-лимиту контейнера (округлённому вверх). Этот флаг заставляет JVM использовать указанное значение вместо автоопределения.

#### Ответ от Claude

**Как JVM определяет процессоры в контейнере**

Начиная с JDK 8u191 (а с JDK 11 — по умолчанию), JVM имеет встроенную поддержку контейнеров через флаг `-XX:+UseContainerSupport` (включён по умолчанию). При запуске JVM читает параметры cgroup:

- **cgroups v1**: файлы `cpu.cfs_quota_us` и `cpu.cfs_period_us`. Число процессоров: `ceil(quota / period)`.
- **cgroups v2** (поддержка добавлена в Java 15): файл `cpu.max` с форматом `quota period`. Поддержка cgroups v2 в OpenJDK появилась в Java 15 и была бэкпортирована в Java 11.0.16+.

Результат доступен через `Runtime.getRuntime().availableProcessors()`.

**Почему CPU limit не всегда правильно транслируется**

Есть несколько сценариев, когда JVM видит неверное количество ядер:

1. **Нет CPU limit** — если в спецификации пода не задан `limits.cpu`, cgroup не устанавливает квоту, и JVM видит все ядра узла. `requests.cpu` при этом никак не влияет на определение числа процессоров — это лишь планировщику Kubernetes, но не JVM.

2. **CPU limit меньше 1 ядра** — при лимите 500m JVM вычислит `ceil(0.5)` = 1. Это корректно с точки зрения формулы, но означает, что физически поду выделяется лишь 50% одного ядра с активным CPU-throttling при нагрузке.

3. **Несоответствие между requests и limits** — под с `requests: 1` CPU и `limits: 4` CPU заставит JVM думать, что у неё 4 ядра, тогда как планировщик гарантирует только 1.

4. **Старые версии JDK без cgroups v2** — OpenJDK до 11.0.16 не поддерживает cgroups v2 и может некорректно читать лимиты на современных Linux-дистрибутивах.

**Влияние на JVM-подсистемы**

| Подсистема | Как зависит от числа ядер |
|---|---|
| ForkJoinPool (common pool) | `parallelism = max(availableProcessors - 1, 1)` |
| G1GC ParallelGCThreads | N (при N <= 8), `floor(5/8 * N)` при N > 8 |
| GC ConcGCThreads (G1, ZGC) | `max(ParallelGCThreads / 4, 1)` |
| JIT-компилятор (C2-потоки) | логарифмически от N |
| Virtual Thread scheduler (Java 21+) | `availableProcessors()` |

Если JVM видит 32 ядра вместо 2, ForkJoinPool создаст 31 поток параллелизма вместо 1, G1GC запустит до 20 параллельных потоков вместо 2. Все эти потоки конкурируют за 2 реальных ядра, что вызывает интенсивное переключение контекста и деградацию производительности.

**Расхождение между источниками**: статья mikemybytes.com рекомендует устанавливать `ActiveProcessorCount` равным CPU limit. Статья pretius.com предлагает значение, равное `2x yourCpuLimit`, аргументируя тем, что это снижает CPU-throttling при пиковой нагрузке. Официальная документация OpenJDK и Red Hat Developer рекомендуют `ceil(cpuLimit)` как наиболее точное значение.

**Как исправить**

1. **`-XX:ActiveProcessorCount=N`** — явное указание числа процессоров. JVM будет использовать N при вычислении размеров пулов. Не ограничивает реальное использование CPU, только влияет на определение числа ядер.

2. **`-XX:+UseContainerSupport`** (включён по умолчанию с JDK 11) — базовая поддержка контейнеров. Помогает только при наличии CPU limit.

3. **Задать `limits.cpu` в манифесте** — без лимита JVM всегда видит все ядра узла.

**Рекомендуемая конфигурация:**

```yaml
# В манифесте Deployment — обязательно задать limits
resources:
  requests:
    cpu: "1"
  limits:
    cpu: "2"

# В параметрах JVM
env:
- name: JAVA_OPTS
  value: >-
    -XX:+UseContainerSupport
    -XX:ActiveProcessorCount=2
    -XX:MaxRAMPercentage=75.0
```

**Выбор GC**: при `ActiveProcessorCount=1` JVM по умолчанию выбирает Serial GC. При 2+ ядрах и достаточном объёме heap — G1GC. Для предсказуемости рекомендуется задавать GC явно (`-XX:+UseG1GC` или `-XX:+UseZGC`).

### Вопрос 39. Что такое Ingress в Kubernetes? Чем Ingress отличается от Service типа LoadBalancer, и как Ingress-контроллер (например, nginx) реализует path-based и host-based маршрутизацию, а также TLS-терминацию?

**Категория:** Контейнеры/DevOps · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Ingress | Kubernetes](https://kubernetes.io/docs/concepts/services-networking/ingress/)

Ingress — объект Kubernetes API, управляющий внешним доступом к сервисам кластера, как правило по протоколу HTTP/HTTPS. Ingress обеспечивает балансировку нагрузки, SSL-терминацию и виртуальный хостинг на основе имён.

Ingress не работает сам по себе: для его работы в кластере должен быть запущен Ingress-контроллер. Контроллер — это отдельный компонент (например, nginx, HAProxy, Traefik), который читает объекты Ingress из API-сервера и соответственно конфигурирует прокси-сервер.

**Path-based маршрутизация** (fan-out): один IP-адрес, маршрутизация по пути URL:

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: fanout-ingress
spec:
  ingressClassName: nginx
  rules:
  - host: example.com
    http:
      paths:
      - path: /foo
        pathType: Prefix
        backend:
          service:
            name: service1
            port:
              number: 4200
      - path: /bar
        pathType: Prefix
        backend:
          service:
            name: service2
            port:
              number: 8080
```

**Host-based маршрутизация** (name-based virtual hosting): разные хосты на одном IP:

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: name-virtual-host-ingress
spec:
  ingressClassName: nginx
  rules:
  - host: foo.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: service1
            port:
              number: 80
  - host: bar.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: service2
            port:
              number: 80
```

**TLS-терминация**:

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: tls-example-ingress
spec:
  ingressClassName: nginx
  tls:
  - hosts:
    - https-example.foo.com
    secretName: testsecret-tls
  rules:
  - host: https-example.foo.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: service1
            port:
              number: 80
```

Секрет `testsecret-tls` должен содержать ключи `tls.crt` (сертификат) и `tls.key` (приватный ключ) в формате PEM.

#### Ответ от Claude

**Что такое Ingress**

Ingress — объект Kubernetes (API-группа `networking.k8s.io/v1`), декларативно описывающий правила маршрутизации внешнего HTTP/HTTPS-трафика к сервисам кластера. Сам по себе объект Ingress — только конфигурация; реальную обработку трафика выполняет **Ingress-контроллер** — отдельный деплой в кластере.

Примечание: начиная с Kubernetes 1.22 API Ingress заморожен и больше не развивается. Kubernetes-проект рекомендует переходить на Gateway API (`gateway.networking.k8s.io`) для новых кластеров, однако Ingress по-прежнему широко используется на практике.

**Ingress vs Service типа LoadBalancer**

| Характеристика | Ingress | Service LoadBalancer |
|---|---|---|
| Протоколы | HTTP/HTTPS (L7) | Любые TCP/UDP (L4) |
| Маршрутизация | По хосту и пути URL | Только по порту |
| TLS-терминация | Встроенная поддержка | Зависит от облачного провайдера |
| Стоимость | Один внешний LB на весь кластер | Отдельный LB на каждый сервис |
| Правила routing | Гибкие, декларативные | Нет (только проброс порта) |
| Требования | Ingress-контроллер в кластере | Поддержка облаком или MetalLB |

Главное экономическое преимущество Ingress: один облачный Load Balancer обслуживает весь HTTP/HTTPS-трафик кластера через единственный Ingress-контроллер, тогда как каждый Service типа LoadBalancer создаёт отдельный облачный балансировщик.

**Как работает nginx Ingress-контроллер**

1. **Watch API-сервера** — контроллер подписывается на изменения объектов `Ingress` и `IngressClass`.
2. **Генерация конфига** — при изменении Ingress-объекта контроллер генерирует/обновляет конфигурацию nginx (`nginx.conf`).
3. **Reload** — nginx перезагружает конфигурацию (с поддержкой graceful reload для минимизации прерываний соединений).
4. **Проксирование** — nginx принимает входящий трафик и проксирует его на ClusterIP соответствующего Service, который в свою очередь балансирует между подами через kube-proxy/eBPF.

**Path-based маршрутизация**

Запросы направляются к разным сервисам в зависимости от пути URL. Поддерживаются три типа совпадения (`pathType`):
- `Prefix` — совпадение по префиксу пути (рекомендуемый вариант).
- `Exact` — точное совпадение пути.
- `ImplementationSpecific` — поведение определяется контроллером.

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: java-app-path-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:
  - host: api.example.com
    http:
      paths:
      - path: /users
        pathType: Prefix
        backend:
          service:
            name: user-service
            port:
              number: 8080
      - path: /orders
        pathType: Prefix
        backend:
          service:
            name: order-service
            port:
              number: 8080
```

**Host-based маршрутизация**

Контроллер читает заголовок `Host` HTTP-запроса и направляет трафик к соответствующему сервису. Реализует концепцию виртуальных хостов (аналог `server_name` в nginx).

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: java-app-host-ingress
spec:
  ingressClassName: nginx
  rules:
  - host: api.company.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 8080
  - host: admin.company.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: admin-service
            port:
              number: 8080
```

**TLS-терминация**

Ingress-контроллер расшифровывает HTTPS-трафик на стороне балансировщика, а к сервисам обращается по HTTP внутри кластера (или по HTTPS, если настроен backend SSL).

Секрет с сертификатом создаётся командой:

```bash
kubectl create secret tls my-tls-secret \
  --cert=path/to/tls.crt \
  --key=path/to/tls.key
```

Ingress с TLS:

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: java-app-tls-ingress
  annotations:
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
spec:
  ingressClassName: nginx
  tls:
  - hosts:
    - api.company.com
    - admin.company.com
    secretName: company-tls-secret
  rules:
  - host: api.company.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 8080
  - host: admin.company.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: admin-service
            port:
              number: 8080
```

Секция `tls` привязывает хосты к секрету с сертификатом. Можно использовать wildcard-сертификат или отдельные сертификаты на каждый хост через несколько элементов в массиве `tls`. Автоматическое получение сертификатов Let's Encrypt реализуется через **cert-manager**.

**Практические аспекты для Java-сервисов:**
- Аннотация `nginx.ingress.kubernetes.io/proxy-read-timeout` увеличивает таймаут чтения — важно для Java-сервисов с длинными запросами.
- `nginx.ingress.kubernetes.io/proxy-body-size` задаёт максимальный размер тела запроса.
- При использовании gRPC (например, из Spring gRPC) требуется `nginx.ingress.kubernetes.io/backend-protocol: "GRPC"`.

### Вопрос 40. Что такое Helm и зачем он нужен в Kubernetes? Опишите структуру Helm-чарта (Chart.yaml, values.yaml, templates/), как работает Go-шаблонизация и как деплоить один чарт в разные среды (dev/staging/prod) с разными values.

**Категория:** Контейнеры/DevOps · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Charts | Helm](https://helm.sh/docs/topics/charts/)

Helm — менеджер пакетов для Kubernetes. Пакет называется чартом (chart). Чарт — набор файлов, описывающих взаимосвязанные Kubernetes-ресурсы. Один чарт может описывать как что-то простое (один Pod с memcached), так и сложное приложение (полный стек из HTTP-серверов, баз данных, кэшей).

**Структура чарта:**

```
mychart/
  Chart.yaml          # Метаданные чарта (обязательно)
  values.yaml         # Значения по умолчанию
  charts/             # Зависимые чарты (subcharts)
  templates/          # Go-шаблоны, генерирующие Kubernetes-манифесты
  templates/NOTES.txt # Необязательные инструкции для пользователя
  templates/_helpers.tpl # Переиспользуемые хелперы (не рендерятся в манифест)
  LICENSE
  README.md
```

**Chart.yaml** — обязательный файл с метаданными. Основные поля:

```yaml
apiVersion: v2        # v2 для Helm 3+
name: my-java-service
description: A Java microservice chart
type: application     # или library
version: 1.0.0        # Версия самого чарта (SemVer)
appVersion: "2.3.1"   # Версия приложения внутри чарта
```

**values.yaml** — значения по умолчанию, доступные в шаблонах через объект `.Values`:

```yaml
replicaCount: 1
image:
  repository: myregistry/my-java-service
  tag: "latest"
resources:
  limits:
    cpu: 500m
    memory: 512Mi
```

**Go-шаблонизация** в директории `templates/` использует синтаксис `{{ }}`. Шаблоны имеют доступ к нескольким встроенным объектам:

- `.Values` — значения из `values.yaml` и переданных `-f`-файлов
- `.Release.Name`, `.Release.Namespace` — метаданные текущего релиза
- `.Chart.Name`, `.Chart.Version` — метаданные из `Chart.yaml`

Пример шаблона Deployment:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ .Release.Name }}-{{ .Chart.Name }}
spec:
  replicas: {{ .Values.replicaCount }}
  template:
    spec:
      containers:
      - name: {{ .Chart.Name }}
        image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
        resources:
          {{- toYaml .Values.resources | nindent 10 }}
```

**Деплой в разные среды** осуществляется через отдельные файлы значений, переопределяющих нужные параметры:

```
mychart/
  values.yaml          # Базовые значения по умолчанию
  values-dev.yaml      # Переопределения для dev
  values-staging.yaml  # Переопределения для staging
  values-prod.yaml     # Переопределения для prod
```

Команды установки/обновления:

```bash
# Первичный деплой
helm install my-service ./mychart -f values-prod.yaml

# Обновление с новыми значениями
helm upgrade my-service ./mychart -f values-prod.yaml

# Предпросмотр без деплоя
helm install --dry-run --debug my-service ./mychart -f values-dev.yaml
```

Базовый `values.yaml` остаётся неизменным; файлы окружения содержат только перекрывающие поля (replicas, resource limits, image tag, ingress hostname и т.п.).

#### Ответ от Claude

**Что такое Helm**

Helm — де-факто стандартный менеджер пакетов для Kubernetes. Он решает проблему управления набором YAML-манифестов: вместо десятков отдельных файлов, которые нужно поддерживать в нескольких копиях для разных окружений, Helm предоставляет единый параметризованный пакет — чарт. **Структура Helm-чарта**

```

  charts/              # Зависимости (subcharts)
```

```yaml
apiVersion: v2          # Обязательно для Helm 3

version: 1.2.0          # Версия чарта (SemVer, обновляется при каждом изменении чарта)
appVersion: "3.0.1"     # Версия приложения (справочно, не влияет на деплой)
description: Java microservice Helm chart
type: application       # application | library
```

**values.yaml** — единая точка конфигурации. Задаёт значения по умолчанию, которые можно переопределить при деплое:

```yaml

  tag: "1.0.0"
  pullPolicy: IfNotPresent

  port: 8080
```

**Go-шаблонизация**

Helm использует стандартный Go `text/template`, расширенный библиотекой Sprig (~70 дополнительных функций: `toYaml`, `nindent`, `quote`, `default` и т.д.).

Основные конструкции:

```yaml
# Подстановка значения
replicas: {{ .Values.replicaCount }}

# Значение по умолчанию (функция Sprig)
tag: {{ .Values.image.tag | default "latest" | quote }}

# Условная логика
{{- if .Values.ingress.enabled }}
apiVersion: networking.k8s.io/v1

...
{{- end }}

# Цикл по списку
{{- range .Values.env }}
- name: {{ .name }}
  value: {{ .value | quote }}
{{- end }}

# YAML-блок с правильными отступами
resources:
  {{- toYaml .Values.resources | nindent 2 }}
```

Тире `-` внутри `{{-` и `-}}` удаляет пробельные символы слева/справа, что критично для корректных отступов в итоговом YAML.

**Named templates** объявляются в `_helpers.tpl` и переиспользуются:

```
{{- define "mychart.labels" -}}
app.kubernetes.io/name: {{ .Chart.Name }}
app.kubernetes.io/instance: {{ .Release.Name }}
{{- end }}
```

Использование: `{{- include "mychart.labels" . | nindent 4 }}`.

Стратегия: базовый `values.yaml` содержит prod-safe значения по умолчанию; файлы окружений перекрывают только то, что меняется.

```yaml
# values-dev.yaml
replicaCount: 1
image:
  tag: "dev-latest"
resources:
  requests:
    cpu: 100m
    memory: 128Mi
```

```yaml
# values-prod.yaml
replicaCount: 5
```

Команды:

```bash
# dev — используется значение из values-dev.yaml, остальное из values.yaml
helm upgrade --install java-service ./mychart \
  -f values.yaml \
  -f values-dev.yaml \
  --namespace dev

# prod
helm upgrade --install java-service ./mychart \
  -f values.yaml \
  -f values-prod.yaml \
  --namespace prod

# Переопределение одного параметра без файла
helm upgrade java-service ./mychart -f values-prod.yaml \
  --set image.tag="3.0.2"
```

Флаг `-f` можно передавать несколько раз; файлы применяются слева направо, каждый следующий перекрывает предыдущий. Флаг `--set` имеет наивысший приоритет и перекрывает всё.

**Важные нюансы:**

- `helm upgrade --install` — идемпотентная команда: создаёт релиз, если его нет, или обновляет существующий.
- `helm diff` (плагин) позволяет увидеть разницу до применения.
- В CI/CD принято хранить values-файлы отдельно от чарта (например, в GitOps-репозитории), передавая их через `-f`.
- Secrets не следует хранить в `values.yaml` в открытом виде — используются Helm Secrets (плагин с SOPS) или внешние решения (Vault, External Secrets Operator).
