### Вопрос 1. Сколько примитивных типов в Java? Перечислите их с размерами в битах и диапазонами значений.

**Категория:** Core Java — основы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Primitive Data Types — The Java™ Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/java/nutsandbolts/datatypes.html)

В Java существует **8 примитивных типов данных**:

| Тип      | Размер    | Минимальное значение | Максимальное значение |
|----------|-----------|----------------------|-----------------------|
| `byte`   | 8 бит     | −128                 | 127                   |
| `short`  | 16 бит    | −32 768              | 32 767                |
| `int`    | 32 бит    | −2 147 483 648       | 2 147 483 647         |
| `long`   | 64 бит    | −2⁶³                 | 2⁶³−1                 |
| `float`  | 32 бит    | ~1.4 × 10⁻⁴⁵         | ~3.4 × 10³⁸           |
| `double` | 64 бит    | ~4.9 × 10⁻³²⁴        | ~1.8 × 10³⁰⁸          |
| `char`   | 16 бит    | `''` (0)       | `'\xEF\xBF\xBF'` (65 535)   |
| `boolean`| не определён точно | `false`     | `true`                |

Целочисленные типы (`byte`, `short`, `int`, `long`) используют представление в дополнительном коде (two's complement). `float` и `double` соответствуют стандарту IEEE 754. Размер `boolean` точно не определён спецификацией — он представляет одно логическое значение.

#### Ответ от Claude

В Java ровно **8 примитивных типов**. Их можно разбить на четыре группы:

**Целочисленные (знаковые, two's complement):**

| Тип     | Бит | Минимум                  | Максимум                |
|---------|-----|--------------------------|-------------------------|
| `byte`  | 8   | −128                     | 127                     |
| `short` | 16  | −32 768                  | 32 767                  |
| `int`   | 32  | −2 147 483 648 (−2³¹)    | 2 147 483 647 (2³¹−1)   |
| `long`  | 64  | −9 223 372 036 854 775 808 (−2⁶³) | 9 223 372 036 854 775 807 (2⁶³−1) |

**Числа с плавающей точкой (IEEE 754):**

| Тип      | Бит | Мин. положительное значение | Макс. положительное значение | Точность          |
|----------|-----|-----------------------------|-------------------------------|-------------------|
| `float`  | 32  | ~1.4 × 10⁻⁴⁵                | ~3.4 × 10³⁸                   | ~6–7 знач. цифр   |
| `double` | 64  | ~4.9 × 10⁻³²⁴               | ~1.8 × 10³⁰⁸                  | ~15–16 знач. цифр |

**Символьный:**

| Тип    | Бит | Минимум         | Максимум          |
|--------|-----|-----------------|-------------------|
| `char` | 16  | `''` (0)  | `'\xEF\xBF\xBF'` (65 535) |

`char` — беззнаковый 16-битный тип, представляет одну кодовую единицу UTF-16.

**Логический:**

| Тип       | Бит                  | Значения          |
|-----------|----------------------|-------------------|
| `boolean` | не определён точно * | `true` / `false`  |

\* Согласно JLS (Java Language Specification) и JVM Spec, размер `boolean` на уровне исходного кода не зафиксирован. На уровне JVM отдельный `boolean` представляется как `int` (4 байта), а в массивах (`boolean[]`) — как `byte` (1 байт).

**Важные нюансы:**
- `float` и `double` не подходят для точных денежных вычислений — для этого используют `java.math.BigDecimal`.
- С Java SE 8 классы `Integer` и `Long` поддерживают беззнаковые операции через статические методы (`Integer.toUnsignedString`, `Long.compareUnsigned` и др.), хотя сам тип `int` по-прежнему знаковый.
- Значения по умолчанию для полей класса: `0` (целочисленные), `0.0f`/`0.0d` (вещественные), `''` (`char`), `false` (`boolean`).

### Вопрос 2. В чём разница между примитивными типами и ссылочными? Где в памяти хранятся примитивы, а где — объекты?

**Категория:** Core Java — основы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Java Data Types — jenkov.com](https://jenkov.com/tutorials/java/data-types.html)

Переменная примитивного типа **хранит значение напрямую** в выделенной для неё памяти. Переменная ссылочного типа (объектного) — хранит **ссылку на объект**, но не сам объект. Это означает, что несколько переменных могут ссылаться на один и тот же объект, тогда как с примитивами это невозможно. Начиная с Java 5, язык автоматически конвертирует примитивы в объектные обёртки (autoboxing), однако попытка распаковать `null`-ссылку приведёт к `NullPointerException`.

#### Ответ от Claude

**Принципиальные различия:**

| Характеристика         | Примитивный тип              | Ссылочный тип (объект)              |
|------------------------|------------------------------|-------------------------------------|
| Что хранит переменная  | Само значение                | Ссылку (адрес) на объект в куче     |
| Значение по умолчанию  | Числовой 0 / false / ''| `null`                              |
| Передача в метод       | По значению (копия)          | По значению, но значение — ссылка   |
| Сравнение `==`         | Сравнивает значения          | Сравнивает адреса объектов          |
| Может быть `null`?     | Нет                          | Да                                  |
| Типы                   | byte, short, int, long, float, double, char, boolean | Всё остальное: String, массивы, классы, интерфейсы |

**Где хранятся примитивы:**

Согласно спецификации JVM (JVMS §2.5), каждый поток имеет свой **стек (Java Virtual Machine Stack)**. Стек состоит из фреймов — по одному на каждый вызов метода. Каждый фрейм содержит массив локальных переменных. **Примитивные локальные переменные** хранятся непосредственно в этом массиве на стеке.

**Где хранятся объекты:**

Все объекты (экземпляры классов и массивы) размещаются в **куче (Heap)**, которая является общей для всех потоков. Сборщик мусора (GC) управляет освобождением памяти в куче.

**Важный нюанс — поля объектов:**

Если примитив является **полем объекта** (а не локальной переменной), то он хранится **внутри самого объекта на куче**. В стеке хранятся только локальные переменные метода. Пример:

```java
public class Person {
    int age;       // примитив — хранится в куче, внутри объекта Person
    String name;   // ссылка — хранится в куче, внутри объекта Person;
                   // сам String-объект — тоже в куче
}

void method() {
    int x = 42;           // локальный примитив — на стеке
    Person p = new Person(); // ссылка p — на стеке; объект Person — в куче
}
```

**Производительность:**

Примитивы значительно быстрее и занимают меньше памяти, чем их объектные обёртки (wrapper classes). Например, `int` занимает 4 байта, тогда как объект `Integer` — минимум 16 байт (с учётом заголовка объекта в HotSpot JVM) плюс накладные расходы на разыменование ссылки.

### Вопрос 3. Что такое автоупаковка (autoboxing) и автораспаковка (unboxing)? Приведите пример, когда автораспаковка может выбросить NullPointerException.

**Категория:** Core Java — основы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Autoboxing — Java SE 8 Language Features (docs.oracle.com)](https://docs.oracle.com/javase/8/docs/technotes/guides/language/autoboxing.html)

**Autoboxing** и **unboxing** автоматизируют конвертацию между примитивами и их обёрточными классами, избавляя от ручного написания кода вроде `intValue()`. Документация указывает на три важных предостережения:

1. **NullPointerException**: Выражение типа `Integer` может иметь значение `null`. При попытке автоматически распаковать его будет выброшен `NullPointerException`.
2. **Разница `==` для ссылок и примитивов**: оператор `==` на `Integer` сравнивает ссылки, а на `int` — значения.
3. **Накладные расходы**: boxing/unboxing несёт затраты по производительности.

Oracle прямо указывает: _«An Integer is not a substitute for an int; autoboxing and unboxing blur the distinction between primitive types and reference types, but they do not eliminate it.»_ (Integer не является заменой int; autoboxing/unboxing размывают границу, но не устраняют её.)

#### Ответ от Claude

**Автоупаковка (Autoboxing)** — автоматическое преобразование примитивного значения в объект соответствующего обёрточного класса, выполняемое компилятором Java. Компилятор вставляет вызов `ClassName.valueOf(primitive)`.

**Автораспаковка (Unboxing)** — обратное преобразование: из объекта обёрточного класса в примитив. Компилятор вставляет вызов `objectInstance.primitiveValue()` (например, `intValue()`).

Таблица соответствий:

| Примитив  | Обёрточный класс |
|-----------|-----------------|
| `byte`    | `Byte`          |
| `short`   | `Short`         |
| `int`     | `Integer`       |
| `long`    | `Long`          |
| `float`   | `Float`         |
| `double`  | `Double`        |
| `char`    | `Character`     |
| `boolean` | `Boolean`       |

**Когда происходит autoboxing:**
- При присвоении примитива переменной обёрточного типа: `Integer i = 42;`
- При передаче примитива в метод, ожидающий объект: `list.add(5);`

**Когда происходит unboxing:**
- При присвоении объекта-обёртки примитивной переменной: `int x = integerObject;`
- В арифметических операциях: `int result = integerA + integerB;`
- В условиях: `if (booleanWrapper) { ... }`

**Пример NullPointerException при автораспаковке:**

```java
// Пример 1: прямая распаковка null
Integer nullableValue = null;  // ссылка null, не присвоено значение
int primitive = nullableValue;  // компилятор генерирует nullableValue.intValue()
                                 // → NullPointerException!

// Пример 2: null из Map.get()
Map<String, Integer> scores = new HashMap<>();
int score = scores.get("Alice");  // get() вернёт null → NullPointerException!

// Пример 3: тернарный оператор с null
Integer a = null;
int result = (a != null) ? a : 0;  // безопасно — явная проверка
int bad    = true ? a : 0;         // NullPointerException! — unboxing 'a'

// Пример 4: арифметика с null
Integer x = null;
int y = x + 1;  // NullPointerException при попытке распаковать x
```

**Практические рекомендации:**
- Всегда проверяйте обёрточные типы на `null` перед использованием в арифметике или присвоении примитиву.
- Используйте `Objects.requireNonNullElse(value, defaultValue)` или явные проверки.
- Oracle рекомендует применять autoboxing только при необходимости (например, при работе с коллекциями), избегая его в вычислительно интенсивном коде.

### Вопрос 4. Что выведет следующий код и почему? `Integer a = 127; Integer b = 127; Integer x = 128; Integer y = 128; System.out.println(a == b); System.out.println(x == y);`

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Chapter 5. Conversions and Contexts — Java Language Specification SE 17 (docs.oracle.com)](https://docs.oracle.com/javase/specs/jls/se17/html/jls-5.html)

Согласно разделу 5.1.7 «Boxing Conversion» JLS, спецификация гарантирует:

> _«If the value p being boxed is the result of evaluating a constant expression of type boolean, byte, char, short, int, or long, and the result is true, false, a character in the range '' to '' inclusive, or an **integer in the range −128 to 127 inclusive**, then let a and b be the results of any two boxing conversions of p. It is always the case that a == b.»_

Это «прагматический компромисс» (pragmatic compromise): кешировать все значения нецелесообразно, поэтому гарантируется только диапазон −128..127. За его пределами boxing-конверсия **может** (но не обязана) возвращать разные объекты.

#### Ответ от Claude

**Вывод программы:**
```
true
false
```

**Подробное объяснение:**

Строка `Integer a = 127;` — это autoboxing. Компилятор преобразует её в:
```java
Integer a = Integer.valueOf(127);
```

Согласно документации `Integer.valueOf(int i)`:
> _«This method will always cache values in the range **-128 to 127**, inclusive, and may cache other values outside of this range.»_

В реализации HotSpot JVM класс `Integer` содержит внутренний класс `IntegerCache`, который при загрузке классов создаёт и кеширует объекты `Integer` для всех значений от −128 до 127.

**Для значения 127:**
```java
Integer a = Integer.valueOf(127);  // возвращает объект из кеша
Integer b = Integer.valueOf(127);  // возвращает тот же объект из кеша
a == b  // true — оба указывают на один и тот же объект в памяти
```

**Для значения 128:**
```java
Integer x = Integer.valueOf(128);  // кеш не покрывает 128 → new Integer(128)
Integer y = Integer.valueOf(128);  // снова новый объект → new Integer(128)
x == y  // false — два разных объекта в куче
```

**Почему именно −128..127?**

Это требование JLS §5.1.7. Диапазон выбран как прагматический компромисс между памятью и производительностью: небольшие целые числа используются наиболее часто. Нижняя граница (-128) соответствует минимуму типа `byte`; верхняя (127) — его максимуму.

**Нюансы:**
- Верхнюю границу кеша можно расширить через JVM-параметр `-XX:AutoBoxCacheMax=<N>` (только в HotSpot), но JLS гарантирует только до 127 включительно.
- Для корректного сравнения значений объектов `Integer` всегда используйте `.equals()`:
  ```java
  System.out.println(x.equals(y));  // true — сравнение значений
  ```
- Тот же кеш применяется к `Byte`, `Short`, `Long` (диапазон −128..127), `Character` (диапазон 0..127), `Boolean` (оба значения кешируются всегда).
- Оператор `==` между `Integer` и `int` приведёт к unboxing и сравнению значений: `Integer x = 128; x == 128` вернёт `true`.

### Вопрос 5. Что такое пул строк (String Pool)? В какой области памяти он находится в современных версиях JVM?

**Категория:** Core Java — основы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Java String Pool—How Unbounded intern() Causes PermGen OOM | TheCodeForge](https://thecodeforge.io/java/string-pool-java/)

Java String Pool — это выделенная область памяти, которая кэширует строковые литералы и явно интернированные строки, чтобы сократить дублирование объектов в памяти. Пул работает как хеш-таблица: когда несколько частей кода ссылаются на одинаковое строковое значение, они разделяют один и тот же объект, а не создают дубликаты.

**История расположения в памяти:**
- До Java 7: пул находился в PermGen (Permanent Generation) — области фиксированного размера, не подлежащей сборке мусора. Это создавало критическую проблему: неограниченные вызовы `intern()` могли исчерпать PermGen и вызвать `OutOfMemoryError`.
- Java 7+: пул переехал в основную кучу (Heap), что сделало интернированные строки доступными для сборщика мусора при отсутствии ссылок на них.
- Java 8: PermGen полностью упразднён и заменён Metaspace (нативная память). Metaspace хранит только метаданные классов; строковый пул остаётся в Heap.

#### Ответ от Claude

**String Pool (пул строк)** — специальная область внутри кучи (Heap) JVM, реализованная в виде хеш-таблицы, которая хранит уникальные экземпляры строковых литералов. Благодаря ей несколько переменных могут ссылаться на один и тот же объект `String`, вместо того чтобы создавать идентичные копии в памяти.

**Эволюция расположения в памяти:**

| Версия JVM | Где находится String Pool |
|---|---|
| Java 6 и ниже | PermGen (Permanent Generation) — фиксированный размер, не GC-управляемая |
| Java 7+ | Heap (основная куча) — управляется GC, может расти и уменьшаться |
| Java 8+ | Heap (Metaspace заменил PermGen, но строковый пул остался в Heap) |

**Ключевые факты:**
- Перевод пула в Heap (Java 7) решил проблему `OutOfMemoryError: PermGen space` при чрезмерном использовании `intern()`.
- Строки в пуле подлежат сборке мусора — если на них нет ссылок, GC их удалит.
- Размер хеш-таблицы пула по умолчанию в Java 7u40+ составляет 60 013 записей (в Java 6 было всего 1009). Размер можно задать JVM-флагом `-XX:StringTableSize`.
- Metaspace (Java 8+) хранит **метаданные классов**, а не строковый пул; строковый пул по-прежнему расположен в Heap.

### Вопрос 6. Чем отличается создание строки через литерал (String s = "hello") от создания через new String("hello")? Сколько объектов создаётся в каждом случае?

**Категория:** Core Java — основы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [How the String Class Works in Java | DZone](https://dzone.com/articles/string-creation-looking-under-the-hood-1)

При выполнении `String langName = new String("Java");` процесс проходит в два этапа:

- **При загрузке класса:** строковый литерал `"Java"` автоматически добавляется в String Constant Pool (SCP) — специальную область внутри кучи.
- **При вызове конструктора:** вызов `new String()` создаёт дополнительный объект в обычной (не пуловой) области кучи.

Таким образом, вопрос «сколько объектов создаётся» зависит от контекста:
- Объектов, созданных непосредственно этим оператором: один (не-пуловый экземпляр).
- Итого объектов типа `String` с данным содержимым в памяти: два (один в SCP, один в обычной куче) — при условии, что до этого такого литерала в пуле не было.

Строковый литерал загружается из пула через инструкцию байт-кода `LDC` (Load Constant) ещё на этапе загрузки класса.

#### Ответ от Claude

**Создание через литерал: `String s = "hello"`**

1. При загрузке класса JVM проверяет String Pool на наличие строки `"hello"`.
2. Если такой строки нет — создаёт один объект в пуле.
3. Если уже есть — возвращает ссылку на существующий объект; новый объект **не создаётся**.
4. Переменная `s` ссылается непосредственно на объект в пуле.

**Количество новых объектов:** 0 или 1 (0 — если `"hello"` уже есть в пуле, 1 — если нет).

**Создание через конструктор: `String s = new String("hello")`**

1. При загрузке класса JVM обрабатывает литерал `"hello"` и помещает его в пул (если там его ещё нет).
2. При выполнении `new String(...)` **всегда** создаётся новый объект в обычной куче (вне пула).

**Количество новых объектов:** 1 или 2:
- **1 объект** — если `"hello"` уже есть в пуле (создаётся только heap-объект вне пула).
- **2 объекта** — если `"hello"` ещё нет в пуле (создаётся и пуловый, и heap-объект вне пула).

**Практический пример:**
```java
String a = "hello";           // 1 объект в пуле (первое упоминание)
String b = "hello";           // 0 новых объектов, b == a → true
String c = new String("hello"); // 1 новый объект в heap (вне пула)
String d = new String("hello"); // ещё 1 новый объект в heap

System.out.println(a == b);   // true  — обе ссылки на один пуловый объект
System.out.println(a == c);   // false — c это отдельный heap-объект
System.out.println(c == d);   // false — разные heap-объекты
System.out.println(c.equals(d)); // true — содержимое одинаковое
```

**Вывод:** Использование литерала предпочтительно — компилятор может переиспользовать пуловый объект. `new String()` следует применять только тогда, когда явно нужен независимый объект (например, для синхронизации — но это считается плохой практикой).

### Вопрос 7. Что делает метод String.intern()? В каких случаях его имеет смысл использовать?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [String (Java SE 17 & JDK 17) — Oracle Documentation](https://docs.oracle.com/en/java/javase/17/docs/api/java.base/java/lang/String.html)

```java
public String intern()
```

Возвращает каноническое представление строкового объекта.

Метод возвращает строку, гарантированно равную данной строке (по методу `equals()`). Если данный строковый объект уже интернирован — возвращает его самого. Иначе — возвращает интернированную строку с той же последовательностью символов.

Следствие: для любых двух строк `s` и `t` выражение `s.intern() == t.intern()` истинно тогда и только тогда, когда `s.equals(t)` истинно.

#### Ответ от Claude

**Что делает `String.intern()`:**

Метод проверяет, есть ли в String Pool строка с таким же содержимым (по `equals()`):
- Если **есть** — возвращает ссылку на пуловый объект.
- Если **нет** — добавляет данную строку в пул и возвращает ссылку на неё.

```java
String s1 = new String("hello"); // heap-объект вне пула
String s2 = s1.intern();         // возвращает пуловый "hello"
String s3 = "hello";             // тот же пуловый объект

System.out.println(s2 == s3);    // true — оба указывают на пуловый объект
System.out.println(s1 == s3);    // false — s1 остался вне пула
```

**Когда имеет смысл использовать:**

1. **Много дублирующихся строк в памяти.** Если приложение хранит тысячи одинаковых строк (например, названия столбцов из БД, XML-теги, ключи из конфигурации) — `intern()` позволяет каждому уникальному значению существовать в единственном экземпляре. В тестах память сокращалась в 28 раз (с 1,08 ГБ до 38 МБ).

2. **Сравнение строк по ссылке (`==`) вместо `equals()`.** После интернирования строки гарантированно сравнимы через `==`, что быстрее. Используется во внутренних структурах компиляторов, парсеров, ORM.

3. **Динамически создаваемые строки с ограниченным словарём.** Если строки создаются в runtime (например, через `substring()`, конкатенацию), но их уникальных значений мало — `intern()` устранит дубликаты.

**Когда НЕ стоит использовать:**

- Когда строки уникальны или почти уникальны — `intern()` лишь добавит overhead на поиск в пуле без экономии памяти.
- Когда критично время отклика — метод требует обращения к хеш-таблице пула и синхронизации.
- Для короткоживущих строк — сборщик мусора их и так быстро удалит.

**Современная альтернатива — G1GC String Deduplication:**
Флаг `-XX:+UseStringDeduplication` (G1 GC, а с Java 18 также ZGC, Serial GC, Parallel GC) автоматически дедуплицирует строки без явных вызовов `intern()`. Работает только для долгоживущих объектов; `intern()` же применим к любым строкам с любым GC.

### Вопрос 8. Почему класс String является immutable? Перечислите конкретные преимущества, которые это даёт (безопасность, кэш hashCode, многопоточность, String Pool).

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [String: Why it Is Immutable? | DZone](https://dzone.com/articles/string-why-it-is-immutable)

Статья выделяет следующие причины и преимущества неизменяемости `String` в Java:

1. **Безопасность ClassLoader.** Строка используется как аргумент при загрузке классов. Если бы `String` был изменяемым, злоумышленник мог бы подменить имя класса в процессе загрузки (например, изменить `com.generic.PasswordChecker` на `com.hacked.PasswordHacker`).

2. **Потокобезопасность.** Неизменяемость делает `String` потокобезопасным неявно: несколько потоков могут безопасно разделять один экземпляр без механизмов синхронизации.

3. **Производительность HashMap.** Поскольку строки неизменяемы, их хеш-код вычисляется один раз и кэшируется. Повторного вычисления не нужно, что делает строки эффективными ключами словарей.

4. **Экономия памяти (String Pool).** Пул строк позволяет нескольким переменным с одинаковыми значениями ссылаться на один объект, сокращая потребление памяти в куче.

5. **Безопасность данных.** В приложениях, обрабатывающих чувствительные данные (пароли, реквизиты счетов), неизменяемость предотвращает несанкционированное изменение значений.

**Важное уточнение:** переприсвоение переменной (`a = "new value"`) не изменяет исходный объект — оно лишь перенаправляет ссылку на новый объект `String`, оставляя старый нетронутым.

#### Ответ от Claude

**Почему `String` неизменяем — техническая реализация:**

Класс `String` объявлен как `public final class String`, а его внутреннее хранилище — `private final byte[] value` (до Java 9 — `private final char[] value`). Слово `final` на уровне класса запрещает наследование; `private final` на уровне поля запрещает замену самого массива. При этом массив байт не передаётся наружу напрямую — все методы возвращают копии или новые `String`. Совокупность этих мер делает `String` надёжно неизменяемым.

**Конкретные преимущества:**

**1. String Pool и экономия памяти**
Пул строк возможен именно потому, что строки не меняются: если бы один поток мог изменить строку `"hello"` в пуле, все остальные ссылки на неё сразу бы получили другое значение. Неизменяемость гарантирует, что разделение одного экземпляра безопасно.

**2. Кэширование hashCode**
В классе `String` поле `private int hash` хранит вычисленный хеш-код и заполняется лениво при первом вызове `hashCode()`. Поскольку содержимое строки никогда не меняется, повторно вычислять хеш не нужно. Это делает `String` идеальным ключом для `HashMap`/`HashSet` — операции `get`/`put` работают быстрее.

**3. Потокобезопасность**
Любой неизменяемый объект потокобезопасен по определению: нечего синхронизировать, если состояние никогда не изменяется. Строку можно передавать между потоками без блокировок, `volatile`-переменных или `synchronized`-блоков.

**4. Безопасность**
- **ClassLoader:** имя загружаемого класса передаётся как `String`. Если бы строка была изменяемой, вредоносный код мог бы подменить имя класса после проверки, но до фактической загрузки (TOCTOU-атака).
- **Сетевые соединения и файловые пути:** хост, порт, путь к файлу — всё это `String`. Изменяемость открывала бы возможность подмены параметров после проверки безопасности.
- **Пароли в памяти:** хотя для паролей рекомендуется использовать `char[]` (чтобы явно обнулить после использования), неизменяемость `String` как минимум гарантирует, что никто снаружи не сможет изменить значение.

**5. Безопасная передача по ссылке**
Строку можно передавать в методы и между потоками без защитного копирования — вызываемый код точно не изменит оригинал. Это упрощает программирование и устраняет целый класс ошибок.

**Итоговая таблица:**

| Преимущество | Механизм |
|---|---|
| String Pool | Разделение объектов безопасно, т.к. значение не изменится |
| Кэш hashCode | Вычисляется один раз, сохраняется в поле `hash` |
| Многопоточность | Неизменяемый объект потокобезопасен без синхронизации |
| Безопасность (ClassLoader) | Имя класса нельзя подменить после проверки |
| Безопасность (данные) | Чувствительные данные нельзя изменить извне |
| Передача по ссылке | Не нужно защитное копирование |

### Вопрос 9. Каков контракт между методами equals() и hashCode()? Что произойдёт при нарушении этого контракта при использовании объекта как ключа в HashMap?

**Категория:** Core Java — основы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Java hashCode() and equals() — jenkov.com](https://jenkov.com/tutorials/java-collections/hashcode-equals.html)

Контракт между `hashCode()` и `equals()` состоит из двух фундаментальных правил:

1. Если `object1` и `object2` равны согласно методу `equals()`, они **обязаны** иметь одинаковый хеш-код.
2. Если `object1` и `object2` имеют одинаковый хеш-код, это **не означает**, что они равны.

Метод `hashCode()` используется в хеш-коллекциях (`HashMap`, `HashSet`, `HashTable`): хеш-код определяет, в какой «бакет» поместить объект. Затем внутри бакета вызывается `equals()`, чтобы найти точный объект. Если равные объекты возвращают разные хеш-коды, коллекция ищет объект в неправильном бакете и не находит его.

#### Ответ от Claude

**Контракт** (согласно Javadoc `java.lang.Object`, Oracle JDK 21):

- **Правило 1 (обязательное):** Если `a.equals(b) == true`, то `a.hashCode() == b.hashCode()` должно быть истинным.
- **Правило 2 (не обязательное):** Обратное не требуется — объекты с одинаковым хеш-кодом не обязаны быть равными (это нормальная коллизия).
- **Правило 3 (согласованность):** `hashCode()` должен возвращать одно и то же значение на протяжении всего времени жизни объекта, пока поля, участвующие в `equals()`, не меняются.

**Как HashMap использует оба метода:**

При вызове `map.put(key, value)` HashMap вычисляет `key.hashCode()`, определяет бакет и помещает запись туда. При вызове `map.get(key)` HashMap снова вычисляет хеш, находит бакет, затем перебирает все ключи в бакете, сравнивая их через `equals()`.

**Что происходит при нарушении контракта:**

```java
class Point {
    int x, y;
    Point(int x, int y) { this.x = x; this.y = y; }

    @Override
    public boolean equals(Object o) {
        if (!(o instanceof Point p)) return false;
        return x == p.x && y == p.y;
    }
    // hashCode() НЕ переопределён — возвращает адрес объекта
}

Map<Point, String> map = new HashMap<>();
Point p1 = new Point(1, 2);
Point p2 = new Point(1, 2);

map.put(p1, "origin");
System.out.println(map.get(p2)); // null !

// p1.equals(p2) == true, но p1.hashCode() != p2.hashCode()
// get() ищет в другом бакете и не находит значение
```

Таким образом, нарушение контракта делает хеш-коллекции ненадёжными: значение есть на карте, но его нельзя получить по логически равному ключу.

**Правило для разработчика:** Если переопределяете `equals()` — всегда переопределяйте `hashCode()`, используя те же поля. IDE (IntelliJ IDEA, Eclipse) и библиотека Lombok генерируют оба метода автоматически.

### Вопрос 10. Какие требования предъявляет контракт equals() (рефлексивность, симметричность, транзитивность, согласованность, сравнение с null)?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Object (Java SE 21 & JDK 21) — docs.oracle.com](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/lang/Object.html)

Метод `equals()` реализует отношение эквивалентности на ненулевых ссылках. Согласно Javadoc `java.lang.Object`, контракт включает пять свойств:

1. **Рефлексивность (Reflexive):** Для любой ненулевой ссылки `x`, `x.equals(x)` должно возвращать `true`.
2. **Симметричность (Symmetric):** Для любых ненулевых ссылок `x` и `y`, `x.equals(y)` возвращает `true` тогда и только тогда, когда `y.equals(x)` возвращает `true`.
3. **Транзитивность (Transitive):** Для любых ненулевых ссылок `x`, `y`, `z`: если `x.equals(y) == true` и `y.equals(z) == true`, то `x.equals(z)` должно быть `true`.
4. **Согласованность (Consistent):** При многократных вызовах `x.equals(y)` результат не должен меняться, если состояние объектов, задействованных в сравнении, не изменилось.
5. **Сравнение с null:** Для любой ненулевой ссылки `x`, `x.equals(null)` должно возвращать `false`.

#### Ответ от Claude

Контракт `equals()` определён в Javadoc `java.lang.Object` и представляет собой пять обязательных свойств:

**1. Рефлексивность**
```java
x.equals(x) == true  // всегда, для любого x != null
```
Объект всегда равен самому себе. Нарушение: `set.contains(x)` может вернуть `false` сразу после `set.add(x)`.

**2. Симметричность**
```java
x.equals(y) == y.equals(x)  // всегда
```
Типичное нарушение — при наследовании, когда дочерний класс переопределяет `equals()` несимметрично относительно родительского. Например, `MyString.equals(String)` возвращает `true`, а `String.equals(MyString)` — `false`.

**3. Транзитивность**
```java
if (x.equals(y) && y.equals(z)) then x.equals(z) == true
```
Труднее всего соблюсти при добавлении полей через наследование. Классическая проблема: класс `ColorPoint extends Point` — добавление поля `color` ломает транзитивность, если `equals()` в `ColorPoint` учитывает цвет, а в `Point` — нет. Джошуа Блох в «Effective Java» рекомендует использовать композицию вместо наследования в таких случаях.

**4. Согласованность (Idempotentность)**
```java
x.equals(y) // возвращает одно и то же значение при неизменном состоянии x и y
```
`equals()` не должен зависеть от изменяемых внешних ресурсов (время, сеть, случайные числа). Для URL Oracle не рекомендует зависимость от IP-адреса хоста.

**5. Сравнение с null**
```java
x.equals(null) == false  // всегда, без выброса NullPointerException
```
Правильная реализация: `if (obj == null) return false;`. Метод не должен бросать `NullPointerException`.

**Шаблон правильной реализации:**
```java
@Override
public boolean equals(Object obj) {
    if (this == obj) return true;               // рефлексивность + оптимизация
    if (obj == null) return false;              // сравнение с null
    if (getClass() != obj.getClass()) return false;  // тип
    MyClass other = (MyClass) obj;
    return Objects.equals(field1, other.field1)
        && field2 == other.field2;
}
```

### Вопрос 11. Что будет, если переопределить equals(), но не переопределить hashCode()? Продемонстрируйте конкретный сценарий поломки с HashSet или HashMap.

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Why to Override equals() and hashCode() method? — GeeksforGeeks](https://www.geeksforgeeks.org/java/override-equalsobject-hashcode-method/)

Если переопределить только `equals()`, хеш-коллекции работают неправильно. Объекты получают разные хеш-коды от реализации `Object` (на основе адреса памяти) и помещаются в разные бакеты, несмотря на логическое равенство.

Пример из статьи:
```java
class Geek {
    String name;
    int id;
    Geek(String name, int id) { this.name = name; this.id = id; }

    @Override
    public boolean equals(Object obj) {
        if (this == obj) return true;
        if (obj == null || getClass() != obj.getClass()) return false;
        Geek geek = (Geek) obj;
        return name.equals(geek.name) && id == geek.id;
    }
    // hashCode() НЕ переопределён
}
```

При использовании двух логически равных объектов `g1` и `g2` как ключей в `HashMap`, оба значения (`"CSE"` и `"IT"`) оказываются в карте одновременно — HashMap не распознаёт их как один ключ. Это нарушает контракт Map, допускающий только уникальные ключи.

#### Ответ от Claude

**Механизм поломки:**

Когда `hashCode()` не переопределён, используется реализация `Object.hashCode()`, которая, как правило, основана на адресе объекта в памяти. Два разных объекта, даже если они логически равны по `equals()`, получат разные хеш-коды и попадут в разные бакеты коллекции. Метод `equals()` в этой ситуации вообще не будет вызван — до него просто не доходит.

**Демонстрация поломки с HashSet:**

```java
class Person {
    String name;
    int age;

    Person(String name, int age) {
        this.name = name;
        this.age = age;
    }

    @Override
    public boolean equals(Object obj) {
        if (!(obj instanceof Person p)) return false;
        return age == p.age && Objects.equals(name, p.name);
    }
    // hashCode() НЕ переопределён!
}

HashSet<Person> set = new HashSet<>();
Person p1 = new Person("Alice", 30);
Person p2 = new Person("Alice", 30);

set.add(p1);
set.add(p2); // добавит второй объект! HashSet не видит дубликат

System.out.println(p1.equals(p2));  // true  — equals работает
System.out.println(set.size());     // 2     — но set содержит ДВА объекта!
System.out.println(set.contains(p2)); // false — не нашёл в нужном бакете!
```

**Демонстрация поломки с HashMap:**

```java
Map<Person, String> map = new HashMap<>();
Person key1 = new Person("Bob", 25);
map.put(key1, "developer");

Person key2 = new Person("Bob", 25); // логически равен key1
System.out.println(map.get(key2)); // null — значение не найдено!
```

**Почему так происходит (пошагово):**

1. `set.add(p1)` → вычисляется `p1.hashCode()` (допустим, `12345`) → объект помещается в бакет `12345`.
2. `set.add(p2)` → вычисляется `p2.hashCode()` (допустим, `67890`, другой адрес) → HashSet ищет в бакете `67890`, там пусто → добавляет `p2` как новый элемент.
3. `set.contains(p2)` → вычисляется `p2.hashCode()` → `67890` → бакет `67890` содержит только `p2`, но при следующем вызове с другим объектом будет `false`.

**Правильное исправление:**

```java
@Override
public int hashCode() {
    return Objects.hash(name, age); // те же поля, что и в equals()
}
```

**Ключевое правило:** `hashCode()` и `equals()` должны использовать **одинаковый набор полей**. Нарушение этого правила — один из самых сложных для отладки багов: программа компилируется и работает, но логика коллекций ведёт себя непредсказуемо.

### Вопрос 12. Чем отличается == от equals() при сравнении строк? Почему для строк всегда нужно использовать equals()?

**Категория:** Core Java — основы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Guide to Java String Pool — baeldung.com](https://www.baeldung.com/java-string-pool)

Оператор `==` проверяет ссылочное равенство — ссылаются ли две переменные на один и тот же объект в памяти. JVM хранит пул строковых литералов (String Pool): когда создаётся строковый литерал, JVM ищет в пуле строку с таким же содержимым и, если находит, возвращает ссылку на уже существующий объект.

Поэтому два строковых литерала с одинаковым содержимым указывают на один объект в пуле, и `==` между ними возвращает `true`. Однако строка, созданная через `new String(...)`, всегда создаёт новый объект в куче (Heap), минуя пул. В этом случае `==` вернёт `false` даже при одинаковом содержимом.

Метод `equals()` у `String` сравнивает содержимое посимвольно и не зависит от того, как объект был создан.

#### Ответ от Claude

**Разница между `==` и `equals()`:**

- `==` — оператор сравнения **ссылок** (адресов в памяти). Возвращает `true` только если обе переменные указывают на один и тот же объект.
- `String.equals()` — переопределённый метод, который сравнивает **содержимое** строк посимвольно.

**Как работает String Pool:**

JVM поддерживает специальный кэш строковых констант — String Pool (с Java 7+ находится в Heap). Все строковые литералы автоматически интернируются: при создании литерала JVM проверяет пул, и если строка с таким значением уже есть — возвращает существующую ссылку. Строки, созданные через `new String(...)`, создают новый объект в Heap, не попадая в пул.

**Демонстрация:**

```java
String a = "hello";          // литерал → попадает в String Pool
String b = "hello";          // литерал → JVM возвращает ту же ссылку из пула
String c = new String("hello"); // явное создание → новый объект в Heap
String d = c.intern();       // ручное интернирование → ссылка из пула

System.out.println(a == b);           // true  — один объект в пуле
System.out.println(a == c);           // false — разные объекты (пул vs Heap)
System.out.println(a == d);           // true  — intern() вернул ссылку из пула

System.out.println(a.equals(b));      // true
System.out.println(a.equals(c));      // true — содержимое одинаковое
System.out.println(a.equals(d));      // true
```

**Почему `==` для строк ненадёжен:**

1. **Зависит от способа создания объекта.** Строки из разных источников (из файла, из сети, через `StringBuilder`, через `new String(...)`) не попадают в пул автоматически и дают разные ссылки.

2. **Поведение может меняться в зависимости от JVM и оптимизаций.** Компилятор может интернировать константные выражения, но это деталь реализации, на которую нельзя полагаться.

3. **Типичный баг:**
```java
// Получили строку из ввода пользователя или из БД
String input = scanner.nextLine(); // "admin"
String expected = "admin";

if (input == expected) { // НЕПРАВИЛЬНО — почти всегда false
    grantAccess();
}

if (input.equals(expected)) { // ПРАВИЛЬНО — сравнивает содержимое
    grantAccess();
}
```

4. **Защита от NullPointerException:** При сравнении с известным литералом размещайте его слева:
```java
"expected".equals(userInput); // не бросит NPE, если userInput == null
```

**Итог:** Оператор `==` для строк работает «случайно правильно» только для интернированных литералов из одного и того же кода. В реальных приложениях строки приходят из разных источников, поэтому **всегда используйте `equals()`** для сравнения строк по значению.
### Вопрос 13. Как правильно реализовать immutable-класс? Перечислите все необходимые условия (final класс, final поля, нет сеттеров, защитное копирование изменяемых полей).

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [A Strategy for Defining Immutable Objects — The Java™ Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/essential/concurrency/imstrat.html)

Официальная документация Oracle описывает четыре правила для создания неизменяемого класса:

1. **Не предоставляйте методы-«сеттеры»** — методы, изменяющие поля или объекты, на которые ссылаются поля.
2. **Все поля должны быть `final` и `private`.**
3. **Не позволяйте подклассам переопределять методы.** Простейший способ — объявить класс как `final`. Более продвинутый вариант — сделать конструктор `private` и создавать экземпляры через фабричные методы.
4. **Если поля ссылаются на изменяемые объекты — не позволяйте этим объектам меняться:**
   - Не предоставляйте методы, модифицирующие эти изменяемые объекты.
   - Не раскрывайте ссылки на изменяемые объекты: никогда не сохраняйте ссылку на внешний изменяемый объект, переданный в конструкторе — если необходимо, создайте копию и храните ссылку на неё.
   - Аналогично, создавайте копии внутренних изменяемых объектов при возврате их из методов, чтобы не отдавать оригинал.

В качестве полного примера в документации приводится класс `ImmutableRGB`: все поля — `final private`, класс объявлен `final`, сеттеров нет, методы вроде `invert()` возвращают новый экземпляр вместо изменения существующего.

#### Ответ от Claude

Immutable-объект — это объект, состояние которого не может измениться после завершения конструктора. Для правильной реализации immutable-класса необходимо выполнить следующие условия:

**1. Объявить класс как `final`**

Это запрещает создание подклассов, которые могли бы нарушить контракт неизменяемости, переопределив методы.

```java
public final class Money {
    // ...
}
```

Альтернатива — сделать конструктор `private` и предоставлять экземпляры только через статические фабричные методы, что тоже исключает возможность расширения.

**2. Все поля — `private final`**

`private` скрывает прямой доступ снаружи. `final` гарантирует, что ссылка на значение/объект будет присвоена ровно один раз — в конструкторе.

```java
private final int amount;
private final String currency;
private final List<String> tags;
```

**3. Не предоставлять сеттеры**

Никаких методов, изменяющих состояние объекта. Все геттеры должны только читать, но не мутировать.

**4. Защитное копирование (defensive copying) для изменяемых полей**

Это самое часто пропускаемое условие. Если поле ссылается на изменяемый объект (массив, коллекция, `Date`, пользовательский класс с сеттерами), нужно:

- **В конструкторе**: копировать входящий объект, не сохранять оригинал.
- **В геттерах**: возвращать копию, а не оригинальную ссылку.

```java
public final class ImmutablePerson {
    private final String name;
    private final List<String> hobbies;

    public ImmutablePerson(String name, List<String> hobbies) {
        this.name = name;
        // Защитная копия при получении — иначе внешний код может изменить переданный список
        this.hobbies = List.copyOf(hobbies); // Java 10+, либо new ArrayList<>(hobbies)
    }

    public String getName() {
        return name;
    }

    public List<String> getHobbies() {
        // Возвращаем немодифицируемый вид, а не ссылку на внутреннее поле
        return Collections.unmodifiableList(hobbies);
    }
}
```

**Почему это важно.** Если сохранить оригинальную ссылку на `List`, переданный в конструктор, вызывающий код может позже сделать `list.add(...)` и изменить состояние якобы immutable-объекта. Аналогично, если геттер возвращает прямую ссылку на внутренний `List`, получатель может его изменить.

**Итоговый чеклист:**

| Условие | Зачем |
|---|---|
| `final class` | Запрет наследования и переопределения методов |
| `private final` поля | Запрет прямого доступа и повторного присвоения |
| Нет сеттеров | Нет публичного API для мутации |
| Защитное копирование в конструкторе | Изоляция от изменений снаружи |
| Защитное копирование в геттерах | Изоляция от изменений через возвращённые ссылки |

**Важное замечание:** `final` поле само по себе не делает объект immutable — оно лишь фиксирует ссылку. Если поле ссылается на изменяемый объект (`List`, `Date`, массив), его содержимое можно изменить без нарушения `final`. Именно поэтому защитное копирование — обязательное, а не опциональное условие.

### Вопрос 14. В чём разница между final-переменной, final-методом и final-классом? Можно ли изменить состояние объекта, на который ссылается final-переменная?

**Категория:** Core Java — основы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Writing Final Classes and Methods — The Java™ Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/java/IandI/final.html)

Документация Oracle описывает `final` применительно к методам и классам:

**Final-методы.** Ключевое слово `final` в объявлении метода означает, что метод не может быть переопределён подклассами. Метод стоит делать `final`, если:
- он имеет реализацию, которую не следует изменять,
- он критически важен для поддержания согласованного состояния объекта,
- он вызывается из конструктора (переопределение конструктором подкласса может дать неожиданные результаты).

Пример из документации:
```java
class ChessAlgorithm {
    enum ChessPlayer { WHITE, BLACK }
    final ChessPlayer getFirstPlayer() {
        return ChessPlayer.WHITE;
    }
}
```

**Final-классы.** Класс, объявленный как `final`, не может быть расширен (унаследован). Это полезно при создании immutable-классов — именно так устроен, например, `String`. Важно: `final` у класса не означает, что объекты этого класса неизменяемы — это лишь запрет на наследование.

О `final`-переменных туториал в этом разделе не говорит, однако из стандартной семантики Java следует: `final`-переменная фиксирует ссылку, но не состояние объекта, на который она указывает.

#### Ответ от Claude

Ключевое слово `final` имеет три различных значения в зависимости от контекста применения.

**Final-переменная**

Переменная, объявленная `final`, может быть присвоена только один раз. После первого присвоения попытка переприсвоить её вызовет ошибку компиляции.

```java
final int x = 10;
x = 20; // Ошибка компиляции

final List<String> list = new ArrayList<>();
list = new ArrayList<>(); // Ошибка компиляции — нельзя переприсвоить ссылку
list.add("hello");        // OK — состояние объекта меняется свободно
```

**Ключевой вопрос: можно ли изменить состояние объекта, на который ссылается `final`-переменная?**

**Да, можно.** `final` фиксирует только ссылку (адрес объекта в памяти), но не содержимое объекта. Если объект изменяемый (имеет сеттеры, `add`, `put` и т.п.), его состояние можно менять свободно — `final` этому не препятствует.

```java
final Date date = new Date(0);
date.setTime(System.currentTimeMillis()); // Допустимо — state объекта изменён

final int[] arr = {1, 2, 3};
arr[0] = 99; // Допустимо — элемент массива изменён
arr = new int[]{};  // Ошибка компиляции — ссылку менять нельзя
```

Чтобы объект был действительно неизменяемым, нужно проектировать его как immutable-класс (см. вопрос №13), а не просто хранить в `final`-переменной.

---

**Final-метод**

Метод, объявленный `final`, не может быть переопределён (`@Override`) в подклассах. Виртуальная диспетчеризация для него не происходит — вызывается всегда именно та реализация, которая объявлена в классе.

```java
class Base {
    public final void doSomething() {
        System.out.println("Base version — always");
    }
}

class Child extends Base {
    @Override
    public void doSomething() { // Ошибка компиляции
    }
}
```

Используется, когда реализация критична для корректности класса и её переопределение могло бы нарушить инварианты.

---

**Final-класс**

Класс, объявленный `final`, не может быть унаследован. Любая попытка написать `class Child extends FinalClass` вызовет ошибку компиляции.

```java
public final class String { ... }  // Нельзя унаследоваться от String
```

`final`-класс подразумевает, что все его методы тоже `final` де-факто (наследоваться нельзя, значит и переопределять некому). Это самый строгий из трёх вариантов.

---

**Сравнительная таблица:**

| Применение | Что запрещает | На что не влияет |
|---|---|---|
| `final` переменная | Повторное присвоение ссылки/значения | Изменение состояния объекта |
| `final` метод | Переопределение в подклассах | Вызов метода, поведение метода |
| `final` класс | Наследование | Изменение полей экземпляров класса |

### Вопрос 15. Что означает ключевое слово static применительно к полю, методу и вложенному классу? В каком порядке инициализируются static-поля и static-блоки?

**Категория:** Core Java — основы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Understanding Class Members — The Java™ Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/java/javaOO/classvars.html)

Документация Oracle объясняет:

**Static-поля (class variables).** Поля с модификатором `static` принадлежат классу, а не отдельным экземплярам. Существует ровно одна копия такого поля, общая для всех объектов класса, в одном фиксированном месте в памяти. Обращаться к ним можно через имя класса: `ClassName.field`.

```java
private static int numberOfBicycles = 0;
```

**Static-методы (class methods).** Методы с модификатором `static` принадлежат классу. Их следует вызывать через имя класса, без создания экземпляра. Типичное применение — доступ к статическим полям.

```java
public static int getNumberOfBicycles() {
    return numberOfBicycles;
}
```

Static-методы **не могут** обращаться к полям и методам экземпляра напрямую: они должны использовать явную ссылку на объект. Также в них нельзя использовать `this`, так как экземпляра не существует.

#### Ответ от Claude

**Static-поле (переменная класса)**

Static-поле принадлежит классу, а не конкретному объекту. Оно существует в единственном экземпляре — независимо от того, сколько объектов класса создано (даже если ноль). Все экземпляры разделяют одно и то же значение.

```java
public class Counter {
    public static int count = 0; // одно на весь класс
    public int id;               // своё у каждого объекта

    public Counter() {
        count++;
        this.id = count;
    }
}

Counter a = new Counter(); // count = 1, a.id = 1
Counter b = new Counter(); // count = 2, b.id = 2
// Counter.count == 2, a.count == 2, b.count == 2 — все видят одно поле
```

Согласно Java Language Specification (§8.3.1.1): «Если поле объявлено `static`, существует ровно одно воплощение этого поля, сколько бы экземпляров класса ни было создано».

---

**Static-метод (метод класса)**

Static-метод принадлежит классу и вызывается без создания объекта. Он не имеет доступа к `this` и, следовательно, не может напрямую обращаться к полям и методам экземпляра (подробнее — в вопросе №16).

```java
public class MathUtils {
    public static int square(int n) {
        return n * n;
    }
}

MathUtils.square(5); // вызов без экземпляра
```

---

**Static nested class (статический вложенный класс)**

Вложенный класс с модификатором `static` принадлежит объемлющему классу, а не его экземплярам. Он ведёт себя как обычный класс верхнего уровня, просто «упакованный» для удобства внутрь другого класса.

- Для его создания **не нужен** экземпляр внешнего класса.
- Он имеет прямой доступ только к **static**-членам внешнего класса.
- К нестатическим членам внешнего класса может обращаться только через явную ссылку на объект.

```java
class Outer {
    private static int staticField = 10;
    private int instanceField = 20;

    static class StaticNested {
        void show() {
            System.out.println(staticField);         // OK
            // System.out.println(instanceField);   // Ошибка компиляции
        }
    }
}

Outer.StaticNested nested = new Outer.StaticNested(); // без экземпляра Outer
```

Отличие от нестатического внутреннего класса (`inner class`): inner class всегда привязан к конкретному экземпляру внешнего класса и создаётся только через него (`outer.new Inner()`).

---

**Порядок инициализации static-полей и static-блоков**

Статическая инициализация происходит **при загрузке класса** (classloading) — один раз за время работы JVM. Правила (из документации Oracle и JLS):

1. **Статические поля-константы** (`static final` с compile-time значением) инициализируются первыми, до остальных статических полей.
2. **Все остальные static-поля и static-блоки инициализируются в порядке их текстуального появления в исходном коде** — сверху вниз.
3. Статическая инициализация выполняется **до** любой инициализации экземпляров (конструкторов, инициализаторов экземпляров).
4. Если static-поле инициализируется вызовом метода — метод выполняется в момент инициализации.

```java
class Example {
    static int a = 10;           // 1-й: a = 10

    static {
        System.out.println("Static block 1, a=" + a); // 2-й
        b = 20;
    }

    static int b = 5;            // 3-й: b = 5 (перезапишет 20)

    static {
        System.out.println("Static block 2, b=" + b); // 4-й: b=5
    }
}
// Вывод:
// Static block 1, a=10
// Static block 2, b=5
```

Вложенные static-классы инициализируются **лениво** — только при первом обращении к ним, а не при загрузке внешнего класса.

### Вопрос 16. Может ли static-метод обращаться к нестатическим полям класса напрямую? Почему? Может ли нестатический метод обращаться к static-полям?

**Категория:** Core Java — основы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Understanding Class Members — The Java™ Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/java/javaOO/classvars.html)

Документация Oracle чётко формулирует правила:

**Static-методы** не могут обращаться к переменным экземпляра и методам экземпляра напрямую — они обязаны использовать ссылку на объект. Также в static-методах нельзя использовать ключевое слово `this`, поскольку нет экземпляра, к которому оно могло бы относиться.

**Нестатические методы** (методы экземпляра) могут напрямую обращаться к переменным экземпляра и методам экземпляра, а также напрямую обращаться к статическим переменным и статическим методам.

#### Ответ от Claude

**Может ли static-метод обращаться к нестатическим полям напрямую?**

**Нет.** Static-метод не привязан ни к какому экземпляру класса. Нестатическое поле — это поле конкретного объекта: у каждого экземпляра оно своё. Если бы static-метод мог обращаться к нестатическому полю без указания объекта, JVM не знала бы, поле *какого именно* объекта использовать.

Попытка скомпилирует ошибку: `"Non-static field 'X' cannot be referenced from a static context"`.

```java
public class Example {
    private int instanceField = 42;
    private static int staticField = 100;

    public static void staticMethod() {
        // System.out.println(instanceField); // ОШИБКА КОМПИЛЯЦИИ
        System.out.println(staticField);      // OK — static поле принадлежит классу
    }
}
```

Единственный способ получить доступ к нестатическому полю из static-метода — через явную ссылку на объект:

```java
public static void staticMethod(Example obj) {
    System.out.println(obj.instanceField); // OK — явная ссылка на конкретный объект
}
```

**Почему так?**

Static-метод существует на уровне класса и может быть вызван даже тогда, когда не создан ни один экземпляр. Нестатические поля существуют только в контексте конкретного объекта — они создаются при `new` и уничтожаются при сборке мусора. Без ссылки на объект у static-метода нет «точки привязки» для нестатических членов. Именно поэтому в static-методе также нельзя использовать `this` — `this` — это ссылка на текущий экземпляр, которого в статическом контексте не существует.

---

**Может ли нестатический метод обращаться к static-полям?**

**Да, без ограничений.** Нестатический метод вызывается в контексте конкретного объекта, у которого есть ссылка на класс. Static-поле принадлежит классу и доступно всегда — независимо от того, есть ли экземпляры или нет. Поэтому нестатический метод может обращаться как к своим полям экземпляра, так и к static-полям класса.

```java
public class Example {
    private int instanceField = 5;
    private static int staticField = 100;

    public void instanceMethod() {
        System.out.println(instanceField);  // OK — поле этого экземпляра
        System.out.println(staticField);    // OK — статическое поле класса
        staticMethod();                     // OK — вызов static-метода из нестатического
    }

    public static void staticMethod() {
        System.out.println(staticField);    // OK
        // System.out.println(instanceField); // ОШИБКА
    }
}
```

**Сводная таблица доступа:**

| Метод | К static-полям/методам | К нестатическим полям/методам |
|---|---|---|
| static-метод | Да (напрямую) | Нет (только через ссылку на объект) |
| Нестатический метод | Да (напрямую) | Да (напрямую, через `this`) |

Асимметрия объясняется тем, что нестатический метод всегда имеет скрытый параметр `this` — ссылку на конкретный экземпляр — и через него может получить доступ к чему угодно. Static-метод такого параметра не имеет.
### Вопрос 17. Почему метод toString() рекомендуется переопределять? Что возвращает реализация по умолчанию из Object?

**Категория:** Core Java — основы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Object (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/lang/Object.html)

Метод `toString()` возвращает строковое представление объекта. В общем случае метод `toString` возвращает строку, которая «текстуально представляет» данный объект. Результат должен быть лаконичным, но информативным представлением, понятным человеку.

Реализация по умолчанию в классе `Object` возвращает строку, состоящую из:
- имени класса, экземпляром которого является объект;
- символа `@`;
- беззнакового шестнадцатеричного представления хэш-кода объекта.

Формула: `getClass().getName() + '@' + Integer.toHexString(hashCode())`

Пример вывода: `java.lang.Object@4aa298b7`

Документация явно указывает: **«Рекомендуется, чтобы все подклассы переопределяли этот метод»**, поскольку реализация по умолчанию предоставляет минимальную информацию о состоянии объекта.

#### Ответ от Claude

**Что возвращает реализация по умолчанию?**

Метод `toString()` класса `Object` возвращает строку в формате:
```
getClass().getName() + '@' + Integer.toHexString(hashCode())
```
Например: `com.example.Person@1b6d3586`. Это имя класса, символ `@` и хэш-код объекта в шестнадцатеричном виде.

**Почему рекомендуется переопределять?**

1. **Отладка.** При выводе объекта через `System.out.println(obj)` или в логах строка `Person@1b6d3586` не несёт никакой информации о содержимом объекта. Переопределённый `toString()` позволяет увидеть состояние полей сразу.

2. **Неявные вызовы.** `toString()` вызывается автоматически при конкатенации строк (`"Объект: " + obj`), при передаче в `String.format`, при выводе в `println` и в других местах. Без переопределения везде будет бессмысленная строка вида `ClassName@hashCode`.

3. **Читаемость кода и логов.** Осмысленное текстовое представление упрощает поддержку: в стектрейсах и журналах сразу видно, с каким объектом произошла ошибка.

4. **Соглашение платформы.** Java API (коллекции, Optional, исключения) активно использует `toString()` для вывода содержимого. Следуя этому соглашению в собственных классах, вы обеспечиваете единообразное поведение.

**Пример:**
```java
// До переопределения
Person p = new Person("Alice", 30);
System.out.println(p); // com.example.Person@4aa298b7

// После переопределения
@Override
public String toString() {
    return "Person{name='" + name + "', age=" + age + "}";
}
System.out.println(p); // Person{name='Alice', age=30}
```

**Что важно учесть при реализации:** результат `toString()` не должен содержать чувствительных данных (пароли, токены); формат не является контрактом и не должен использоваться для парсинга — для этого есть геттеры.

### Вопрос 18. Почему использование оператора + для конкатенации строк в цикле неэффективно? Какие альтернативы следует использовать и почему?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Efficient code vs string concatenation — BellSoft Java](https://bell-sw.com/announcements/2021/06/25/efficient-code-vs-string-concatenation-solve-this-java-snippet/)

Конкатенация строк через `+` в цикле имеет **квадратичную сложность O(n²)**, а не линейную. Поскольку строки в Java неизменяемы, каждая операция конкатенации копирует все предыдущее содержимое. Для n итераций это даёт сумму арифметической прогрессии: `(1) + (1+1) + (1+1+1) + ...`, что равно приблизительно `(n²-n)/2` скопированных символов.

На уровне байт-кода в Java 9+ компилятор генерирует вызов `invokedynamic`, который функционально эквивалентен:
```java
s = new StringBuilder().append(s).append(1).toString();
```
Это означает, что на каждой итерации цикла создаётся **новый `StringBuilder`**, заполняется и конвертируется в строку — дорогая операция, повторяемая n раз.

**Решение:** создать единственный `StringBuilder` перед циклом и переиспользовать его:
```java
StringBuilder sb = new StringBuilder(s);
for (int i = 0; i < n; i++)
    sb.append(1);
return sb.toString();
```
Такой подход даёт **ускорение в 5-11 раз** по сравнению с наивной конкатенацией, а сложность снижается до O(n).

#### Ответ от Claude

**Почему `+` в цикле неэффективен?**

Класс `String` в Java является неизменяемым (`immutable`): объект строки не может быть изменён после создания. Любая операция конкатенации создаёт **новый объект `String`**, копируя в него содержимое обоих операндов.

В цикле это приводит к следующей картине:
- Итерация 1: создаётся строка длиной 1, копируется 1 символ
- Итерация 2: создаётся строка длиной 2, копируется 2 символа
- Итерация n: создаётся строка длиной n, копируется n символов

Суммарное количество копирований: `1 + 2 + ... + n = n*(n+1)/2`, то есть **O(n²)**. При 10 000 итерациях это ~50 миллионов копирований символов вместо 10 000.

**Важная тонкость о компиляторе:** для *одиночных* выражений вида `a + b + c` компилятор сам заменяет `+` на `StringBuilder`. Но когда строка *накапливается* в переменной вне цикла, оптимизация не применяется — на каждой итерации создаётся новый `StringBuilder`, конвертируется в `String` и тут же выбрасывается.

```java
// ПЛОХО: O(n²) — на каждой итерации новый StringBuilder + новый String
String result = "";
for (int i = 0; i < n; i++) {
    result += data[i]; // эквивалентно: result = new StringBuilder(result).append(data[i]).toString()
}
```

**Альтернативы:**

1. **`StringBuilder` (однопоточный контекст, предпочтительный выбор):**
```java
StringBuilder sb = new StringBuilder();
for (int i = 0; i < n; i++) {
    sb.append(data[i]);
}
String result = sb.toString(); // O(n) — амортизированное O(1) на append
```
`StringBuilder` хранит внутренний массив символов. Метод `append()` добавляет символы в существующий массив, удваивая его ёмкость при необходимости. Итоговая сложность — **O(n)**.

2. **`StringBuffer` (многопоточный контекст):** аналогичен `StringBuilder`, но все методы синхронизированы (`synchronized`). Используется только когда строка строится из нескольких потоков — в остальных случаях его накладные расходы на синхронизацию не оправданы.

3. **`String.join()` и `Collectors.joining()`** — для объединения коллекций/массивов:
```java
String result = String.join(", ", list);
// или
String result = list.stream().collect(Collectors.joining(", "));
```

**Итоговое правило:** если конкатенация строк происходит в теле цикла или является многошаговой операцией — всегда используйте `StringBuilder`.

### Вопрос 19. Какой тип данных следует использовать для хранения денежных значений в Java и почему нельзя использовать float или double?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [BigDecimal (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/math/BigDecimal.html)

`BigDecimal` — неизменяемое число с произвольной точностью и знаком в десятичном представлении. Значение хранится как пара: беззнаковое целое произвольной точности (unscaled value) и 32-битный масштаб (scale). Формула: `unscaledValue × 10^(−scale)`.

Документация явно предупреждает о проблемах с `double`:

> «Результаты этого конструктора могут быть непредсказуемыми. Можно предположить, что `new BigDecimal(0.1)` создаёт `BigDecimal`, в точности равный 0.1, но на самом деле он равен `0.1000000000000000055511151231257827021181583404541015625`. Это происходит потому, что 0.1 не может быть точно представлен как `double`.»

Рекомендуемый подход — использовать строковый конструктор:
```java
BigDecimal correct = new BigDecimal("0.1"); // точно 0.1
// или
BigDecimal value = BigDecimal.valueOf(0.1);  // рекомендуемая альтернатива
```

`BigDecimal` предоставляет восемь режимов округления и полный контроль над точностью — критично для финансовых расчётов.

#### Ответ от Claude

**Почему нельзя использовать `float` и `double` для денег?**

Типы `float` и `double` используют **двоичное представление с плавающей точкой** (стандарт IEEE 754). Большинство десятичных дробей — в том числе такие обыденные, как `0.1` или `0.2` — **не могут быть точно представлены в двоичной системе счисления**, так же как `1/3` не может быть точно представлена в десятичной. Это приводит к накапливающимся ошибкам округления:

```java
double a = 0.1 + 0.2;
System.out.println(a); // 0.30000000000000004 — не 0.3!

double price = 1.03 - 0.42;
System.out.println(price); // 0.6100000000000001
```

Для разовых операций погрешность ничтожна, но при тысячах финансовых транзакций ошибки **накапливаются** и приводят к реальным расхождениям в расчётах (пенни теряются или появляются «из воздуха»).

**Что использовать вместо этого?**

**`java.math.BigDecimal`** — стандартный выбор для денежных значений:

- Хранит числа в **десятичном** виде (не двоичном), поэтому `0.1` представлено точно.
- Поддерживает произвольную точность.
- Предоставляет явное управление округлением через `RoundingMode` (`HALF_UP`, `HALF_EVEN` и др.) — критично для финансовых правил.
- Является неизменяемым (`immutable`), что безопасно при многопоточном использовании.

```java
BigDecimal a = new BigDecimal("0.1");
BigDecimal b = new BigDecimal("0.2");
System.out.println(a.add(b)); // 0.3 — точно!

BigDecimal price = new BigDecimal("10.00");
BigDecimal tax   = new BigDecimal("0.07");
BigDecimal total = price.multiply(BigDecimal.ONE.add(tax))
                        .setScale(2, RoundingMode.HALF_UP);
System.out.println(total); // 10.70
```

**Критическое правило:** никогда не создавайте `BigDecimal` из значения `double` через конструктор `new BigDecimal(0.1)` — он унаследует неточность `double`. Всегда используйте **строковый конструктор** `new BigDecimal("0.1")` или фабричный метод `BigDecimal.valueOf(0.1)`.

**Альтернативы `BigDecimal`:**
- Хранить деньги в **целых числах** (`long`) как наименьшие единицы валюты (копейки, центы). Быстро и без накладных расходов `BigDecimal`, но требует аккуратной работы с масштабом.
- Библиотека **Joda-Money / JavaMoney (JSR-354)** — предоставляет высокоуровневые типы `Money` и `CurrencyUnit` поверх `BigDecimal`.

**Итог:** `float`/`double` — для научных вычислений, где допустима относительная погрешность. `BigDecimal` — для финансов, налогов, бухгалтерии, где требуется **абсолютная точность**.

### Вопрос 20. Что такое «утечка ссылки» (reference leak) в контексте создания immutable-класса? Как защитное копирование (defensive copy) предотвращает нарушение неизменяемости?

**Категория:** Core Java — основы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [A Strategy for Defining Immutable Objects — docs.oracle.com](https://docs.oracle.com/javase/tutorial/essential/concurrency/imstrat.html)

Oracle Java Tutorials формулируют стратегию создания immutable-классов. В части обработки изменяемых полей документация указывает:

Если поля объекта содержат ссылки на изменяемые объекты, нельзя позволять этим объектам быть изменёнными:
- не предоставляйте методы, которые модифицируют изменяемые объекты;
- **никогда не сохраняйте ссылки на внешние изменяемые объекты, переданные в конструктор** — если необходимо, создавайте копии и сохраняйте ссылки на копии;
- **создавайте копии внутренних изменяемых объектов** при необходимости, чтобы не возвращать оригинальные ссылки из методов.

Пример из документации — класс `ImmutableRGB`, где операция `invert()` не модифицирует существующий объект, а возвращает новый:
```java
public ImmutableRGB invert() {
    return new ImmutableRGB(255 - red, 255 - green, 255 - blue,
                            "Inverse of " + name);
}
```

#### Ответ от Claude

**Что такое «утечка ссылки» (reference leak)?**

Утечка ссылки — ситуация, когда внешний код получает прямой доступ к изменяемому объекту, хранящемуся внутри предположительно immutable-класса, и может его модифицировать в обход гарантий неизменяемости.

Утечка возможна в двух точках:

**1. В конструкторе — внешний код сохраняет ссылку на объект, переданный конструктору:**

```java
// УЯЗВИМЫЙ вариант
public final class ImmutablePeriod {
    private final List<String> items;

    public ImmutablePeriod(List<String> items) {
        this.items = items; // сохраняем ту же ссылку!
    }

    public List<String> getItems() { return items; }
}

// Атака через конструктор:
List<String> mutableList = new ArrayList<>(Arrays.asList("a", "b"));
ImmutablePeriod period = new ImmutablePeriod(mutableList);
mutableList.add("c"); // мутируем оригинал — объект period тоже изменился!
System.out.println(period.getItems()); // [a, b, c] — неизменяемость нарушена
```

**2. В геттере — внешний код получает ссылку на внутренний изменяемый объект:**

```java
// УЯЗВИМЫЙ геттер
public List<String> getItems() {
    return items; // возвращаем оригинал!
}

// Атака через геттер:
period.getItems().add("d"); // мутируем внутренний список напрямую
```

**Как защитное копирование (defensive copy) решает проблему?**

Защитное копирование — создание **независимой копии** изменяемого объекта, чтобы внешний код работал только с копией, не имея доступа к внутреннему состоянию.

```java
public final class ImmutablePeriod {
    private final List<String> items;

    // Защитная копия в конструкторе
    public ImmutablePeriod(List<String> items) {
        this.items = List.copyOf(items); // Java 10+: создаём неизменяемую копию
        // или: this.items = new ArrayList<>(items); // до Java 10
    }

    // Защитная копия в геттере
    public List<String> getItems() {
        return Collections.unmodifiableList(items);
        // или: return new ArrayList<>(items);
        // или (Java 10+): return List.copyOf(items);
    }
}

// Теперь атаки не работают:
List<String> mutableList = new ArrayList<>(Arrays.asList("a", "b"));
ImmutablePeriod period = new ImmutablePeriod(mutableList);
mutableList.add("c");                  // оригинал изменён — на period не влияет
period.getItems().add("d");            // UnsupportedOperationException — защита сработала
```

**Ключевые принципы защитного копирования:**

1. **Порядок важен:** копирование в конструкторе должно происходить **до** проверок валидации, иначе между копированием и проверкой другой поток может успеть изменить переданный объект (TOCTOU-уязвимость).

2. **Глубокое vs поверхностное копирование:** `new ArrayList<>(original)` создаёт поверхностную (shallow) копию списка — элементы не копируются. Если элементы тоже изменяемы, нужно **глубокое копирование** (deep copy).

3. **Для массивов:** `array.clone()` или `Arrays.copyOf()` создаёт поверхностную копию. Для примитивных массивов этого достаточно.

4. **Java 10+:** `List.copyOf()`, `Set.copyOf()`, `Map.copyOf()` создают неизменяемые (unmodifiable) копии — отличный выбор для защиты как на входе, так и на выходе.

5. **Дата/время:** классы `java.util.Date` и `Calendar` изменяемы — необходимо копировать при хранении в immutable-классе. В современном коде предпочтительны неизменяемые `java.time.LocalDate`, `Instant` и др., не требующие копирования.

**Итог:** утечка ссылки — это сохранение или возврат прямой ссылки на изменяемый объект, что позволяет внешнему коду нарушить инвариант «неизменяемости». Защитное копирование разрывает эту связь, гарантируя, что внутреннее состояние immutable-класса контролируется исключительно им самим.
