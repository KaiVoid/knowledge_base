### Вопрос 21. Чем отличается хранение примитива int от Integer в памяти? Где физически хранится значение примитива, объявленного как локальная переменная, а где — как поле объекта?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [How Java's Primitive Data Types Are Stored in Memory](https://medium.com/@AlexanderObregon/how-javas-primitive-data-types-are-stored-in-memory-b2b38a239263)

Локальные примитивные переменные, объявленные внутри методов, хранятся в памяти стека. При вызове метода JVM создаёт стековый фрейм (stack frame), который содержит место для этих переменных. Поскольку размеры примитивов фиксированы (int = 4 байта), Java может выделить ровно столько места, сколько нужно, без каких-либо накладных расходов.

Когда примитивы объявлены как поля класса, они хранятся в памяти кучи (heap) вместе с родительским объектом. В отличие от локальных переменных, они не создают отдельных объектов-обёрток — примитивные данные живут непосредственно внутри структуры объекта в куче.

Разница между `int` и `Integer` существенна:
- Примитив `int` занимает 4 байта.
- Объект `Integer` требует ссылку в стеке (4–8 байт) плюс выделение памяти в куче: хранит само значение, метаданные класса, поле hashCode и выравнивающие байты — итого около 16 байт на типичных JVM.

#### Ответ от Claude

**Хранение `int` (примитив)**

| Место объявления | Физическое расположение |
|---|---|
| Локальная переменная метода | Стековый фрейм (stack frame) текущего потока |
| Поле объекта (instance field) | Куча (heap), внутри тела объекта |
| Поле класса (static field) | Область метаданных класса (Metaspace в Java 8+) |

Примитив `int` всегда занимает ровно **4 байта** и хранится по значению — никаких ссылок, никаких накладных расходов.

**Хранение `Integer` (объект-обёртка)**

`Integer` — это полноценный объект в куче. Его структура:
- **Object header (заголовок объекта)**: 12 байт при включённых сжатых указателях (Compressed OOPs, включено по умолчанию в HotSpot) или 16 байт без сжатия. Заголовок содержит mark word (8 байт: флаги GC, хэш-код, lock bits) и class pointer (4 или 8 байт).
- **Поле `value` (int)**: 4 байта.
- **Выравнивание (padding)**: до кратного 8 байтам.

Итого: типичный `Integer` занимает **16 байт** в куче + 4–8 байт под ссылку в стеке или другом объекте. То есть минимум в 4 раза больше, чем примитив.

**Ключевые следствия:**
1. Локальный `int x = 5;` — значение `5` находится прямо в стековом фрейме, доступ без разыменования.
2. Поле `class Foo { int x = 5; }` — значение `5` находится в теле объекта `Foo` в куче, но по-прежнему без отдельного выделения памяти под само число.
3. `Integer i = 42;` — в стеке (или теле объекта) хранится **ссылка** на объект `Integer` в куче; значение `42` лежит внутри того объекта.
4. Autoboxing (`Integer i = someInt`) порождает объект в куче (или берёт его из кэша IntegerCache), что создаёт давление на GC.

### Вопрос 22. Что будет напечатано? Integer a = 127; Integer b = 127; Integer c = 128; Integer d = 128; System.out.println(a == b); System.out.println(c == d); Объясните, почему и каков размер кэша Integer по умолчанию.

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Java Integer Cache: Why Integer.valueOf(127) == Integer.valueOf(127) Is True](https://dzone.com/articles/java-integer-cache-why-integervalueof127-integerva)

JVM кэширует объекты `Integer` в диапазоне от **-128 до 127** включительно. Когда вы пишете `Integer x = 127;`, компилятор Java преобразует это в `Integer.valueOf(127)`. Метод `valueOf()` проверяет, попадает ли значение в кэшированный диапазон, и если да — возвращает кэшированный объект, иначе создаёт новый.

Для значения 127: обе переменные ссылаются на **один и тот же кэшированный объект**, поэтому `a == b` возвращает `true`.

Для значения 128: значение выходит за границы кэша, создаются **разные объекты** в куче, поэтому `c == d` возвращает `false`.

Диапазон кэша по умолчанию: от -128 до 127. Максимальная граница может быть расширена с помощью JVM-флага `-XX:AutoBoxCacheMax=<N>`.

#### Ответ от Claude

**Результат выполнения кода:**
```
true
false
```

**Подробное объяснение**

При autoboxing компилятор транслирует присваивание `Integer a = 127` в `Integer a = Integer.valueOf(127)`. Внутри `Integer` находится приватный статический вложенный класс `IntegerCache`, который при загрузке класса инициализирует массив заранее созданных объектов `Integer` для диапазона **[-128, 127]**.

Упрощённый исходный код `Integer.valueOf()` из JDK:
```java
public static Integer valueOf(int i) {
    if (i >= IntegerCache.low && i <= IntegerCache.high)
        return IntegerCache.cache[i + (-IntegerCache.low)];
    return new Integer(i);
}
```

- `Integer a = 127` и `Integer b = 127` → оба вызывают `valueOf(127)` → оба получают **одну и ту же ссылку** из кэша → `a == b` есть `true`.
- `Integer c = 128` и `Integer d = 128` → оба вызывают `valueOf(128)` → 128 выходит за `IntegerCache.high` → создаются **два разных объекта** `new Integer(128)` → `c == d` есть `false`.

**Параметры кэша по умолчанию:**
- Нижняя граница: фиксировано **-128** (изменить нельзя).
- Верхняя граница: **127** по умолчанию, но может быть увеличена JVM-флагом `-XX:AutoBoxCacheMax=<N>` (где N ≥ 127). Новая верхняя граница будет `max(127, N)`.

**Аналогичное кэширование у других типов:**
- `Byte`, `Short`, `Long`: фиксированный диапазон [-128, 127], не настраивается.
- `Character`: фиксированный диапазон [0, 127].
- `Boolean`: кэшируются оба значения (`Boolean.TRUE` и `Boolean.FALSE`).
- `Float`, `Double`: кэша нет.

**Практический вывод:** для сравнения `Integer` по значению всегда используйте `.equals()`, а не `==`. Оператор `==` для объектов сравнивает ссылки, и его поведение зависит от кэша — это частый источник тонких ошибок.

### Вопрос 23. Чем String s = "hello" отличается от String s = new String("hello") с точки зрения пула строк и расположения объектов в памяти?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Difference between String literal and New String object in Java](https://www.java67.com/2014/08/difference-between-string-literal-and-new-String-object-Java.html)

**String literal** (`String s = "Java"`):
- Автоматически помещается в пул строк (String Pool).
- Если строка с таким значением уже существует в пуле, JVM возвращает ссылку на существующий объект — новый объект не создаётся.
- Повторное использование одного и того же объекта делает литералы более эффективными по памяти.

**new String()** (`String s = new String("Java")`):
- Всегда создаёт новый объект в куче (heap).
- Объект **не** помещается в пул автоматически, если только явно не вызвать `intern()`.

При `String s = new String("Java")` может создаться **два объекта**: один в пуле строк (для литерала `"Java"`, если его там ещё нет) и один в куче (для `new String(...)`). Переменная `s` ссылается только на объект в куче.

Исторически пул строк находился в пространстве PermGen (Java 1.6 и ранее), начиная с **Java 7** он перемещён в основную кучу (heap) и участвует в сборке мусора.

#### Ответ от Claude

**Расположение объектов в памяти**

```
String s1 = "hello";              // литерал
String s2 = "hello";              // литерал
String s3 = new String("hello");  // явный конструктор
```

| Переменная | Создаётся ли новый объект в куче | Ссылается на пул |
|---|---|---|
| `s1` | Нет (объект в пуле создан при загрузке класса) | Да |
| `s2` | Нет | Да (тот же объект, что и s1) |
| `s3` | Да (новый объект в куче) | Нет |

Следствие:
```java
s1 == s2       // true  — одна и та же ссылка из пула
s1 == s3       // false — разные объекты
s1.equals(s3)  // true  — одинаковое содержимое
```

**Механизм пула строк**

Пул строк (String Pool, он же String Intern Pool или String Constant Pool) — это специальная область в куче, где JVM хранит единственный экземпляр каждой уникальной строки. При использовании литерала компилятор записывает строку в константный пул класса (class file constant pool), а JVM при загрузке класса интернирует её — то есть помещает в единый глобальный пул, если её там ещё нет, либо возвращает уже существующую.

**Важное замечание о `new String("hello")`**

Конструктор `new String("hello")` принимает литерал `"hello"` как аргумент. Это значит, что литерал `"hello"` **всё равно** попадает в пул строк при загрузке класса (он входит в constant pool класса). Дополнительно конструктор создаёт ещё один объект `String` в обычной куче. Итого при первом выполнении строки `new String("hello")` в памяти существуют два объекта: один в пуле, один в куче.

**Эволюция расположения пула строк:**
- **Java 6 и ранее**: пул хранится в PermGen (постоянное поколение) — фиксированный размер, риск `OutOfMemoryError: PermGen space`.
- **Java 7+**: пул перемещён в основную кучу — GC может собирать неиспользуемые интернированные строки, снижая риск переполнения.
- **Java 8+**: PermGen полностью удалён, заменён Metaspace; пул по-прежнему в куче.

### Вопрос 24. Что делает метод String.intern()? Когда его имеет смысл использовать, а когда — нет?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [When to use intern() method of String in Java?](https://javarevisited.blogspot.com/2015/12/when-to-use-intern-method-of-string-in-java.html)

Метод `intern()` возвращает каноническое представление строки. При его вызове JVM ищет строку с таким же содержимым в пуле. Если найдена — возвращается ссылка на строку из пула. Если нет — строка добавляется в пул и возвращается ссылка на неё.

Метод полезен для снижения потребления памяти при наличии большого числа дублирующихся строк, созданных программно (например, через конкатенацию или `new String(...)`), которые не попадают в пул автоматически.

Когда **не** следует использовать `intern()`:
- В Java 6 и ранее неконтролируемое использование `intern()` может вызвать `OutOfMemoryError: PermGen space`, поскольку пул был расположен в ограниченной области PermGen.
- В современных версиях Java злоупотребление `intern()` «засоряет внутренние таблицы JDK» и может привести к проблемам с производительностью, а не к её улучшению.
- В Java 8 с GC G1 существует автоматическая дедупликация строк (`-XX:+UseStringDeduplication`), которая не требует ручного вмешательства.

#### Ответ от Claude

**Что делает `String.intern()`**

Метод определён в `java.lang.String` и является `native`. Его контракт (из Javadoc):

> Возвращает каноническое представление строки. Если пул уже содержит строку, равную данной по `equals()`, — возвращается строка из пула. Иначе данная строка добавляется в пул и возвращается ссылка на неё.

```java
String s1 = new String("hello"); // объект в куче, не в пуле
String s2 = s1.intern();         // s2 указывает на объект в пуле
String s3 = "hello";             // литерал — тоже объект в пуле

System.out.println(s2 == s3); // true — одна и та же ссылка
System.out.println(s1 == s3); // false — s1 в куче, s3 в пуле
```

**Когда имеет смысл использовать:**

1. **Большое число одинаковых строк из внешних источников** — например, при парсинге XML/JSON, чтении CSV, обработке сетевых протоколов, где одни и те же теги/ключи встречаются тысячи раз. Интернирование позволяет держать в памяти один экземпляр каждого уникального значения.
2. **Сравнение строк по `==`** — после `intern()` допустимо использовать `==` вместо `equals()`, что быстрее. Применяется в высоконагруженных парсерах и компиляторах.
3. **Снижение нагрузки на GC** — меньше живых объектов в куче.

**Когда не следует использовать:**

1. **Java 6 и ранее**: пул находился в PermGen фиксированного размера. Интернирование большого числа уникальных строк легко приводит к `OutOfMemoryError: PermGen space`.
2. **Уникальные строки**: если строки редко повторяются, `intern()` только добавляет накладные расходы на поиск в пуле без выгоды по памяти.
3. **Данные от пользователя / из недоверенных источников**: злоумышленник может спровоцировать интернирование миллионов уникальных строк, исчерпав память (DoS).
4. **Когда достаточно `equals()`**: для корректного сравнения строк `equals()` всегда надёжнее. Использование `==` после `intern()` — это оптимизация, требующая строгого контроля.

**Альтернативы в современных JVM:**

- **`-XX:+UseStringDeduplication`** (Java 8+, только G1 GC): JVM автоматически дедуплицирует строки в куче, обнаруживая экземпляры с одинаковым содержимым и заставляя их разделять один символьный массив. Не требует изменений в коде.
- **Явное кэширование через `Map<String, String>`** или `String.intern()` с контролем объёма — управляемая альтернатива для специфических сценариев.

**Изменения поведения по версиям:**
- **Java ≤ 6**: пул в PermGen, `intern()` опасен при большом объёме.
- **Java 7+**: пул перенесён в кучу, неиспользуемые интернированные строки подлежат GC — `intern()` стал безопаснее.
- **Java 8+**: появилась автоматическая дедупликация строк (G1), снизившая потребность в ручном `intern()`.
### Вопрос 25. Перечислите требования (контракт) к правильной реализации equals(): рефлексивность, симметричность, транзитивность, согласованность, сравнение с null. Что произойдёт, если нарушить симметричность при использовании объекта в HashSet?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Object (Java SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/lang/Object.html)

Метод `equals` реализует отношение эквивалентности на ненулевых ссылках на объекты. Согласно официальной документации Oracle, реализация должна удовлетворять следующим свойствам:

1. **Рефлексивность** — для любого ненулевого значения ссылки `x`, `x.equals(x)` должен возвращать `true`.
2. **Симметричность** — для любых ненулевых значений ссылок `x` и `y`, `x.equals(y)` возвращает `true` тогда и только тогда, когда `y.equals(x)` возвращает `true`.
3. **Транзитивность** — для любых ненулевых значений ссылок `x`, `y` и `z`, если `x.equals(y)` возвращает `true` и `y.equals(z)` возвращает `true`, то `x.equals(z)` должен возвращать `true`.
4. **Согласованность** — для любых ненулевых значений ссылок `x` и `y` многократные вызовы `x.equals(y)` последовательно возвращают `true` или последовательно возвращают `false`, при условии что информация, используемая в сравнениях equals, не изменяется.
5. **Сравнение с null** — для любого ненулевого значения ссылки `x`, `x.equals(null)` должен возвращать `false`.

По умолчанию метод `equals` класса `Object` реализует наиболее строгое отношение эквивалентности: для любых ненулевых ссылок `x` и `y` метод возвращает `true` только тогда, когда `x` и `y` указывают на один и тот же объект (`x == y`).

#### Ответ от Claude

**Контракт метода equals()**

Согласно спецификации `java.lang.Object`, корректная реализация `equals()` обязана соответствовать пяти требованиям:

| Свойство | Формальное определение |
|---|---|
| **Рефлексивность** | `x.equals(x) == true` для любого ненулевого `x` |
| **Симметричность** | Если `x.equals(y) == true`, то и `y.equals(x) == true` |
| **Транзитивность** | Если `x.equals(y)` и `y.equals(z)`, то `x.equals(z)` |
| **Согласованность** | Повторные вызовы `x.equals(y)` дают одинаковый результат, пока объекты не изменились |
| **Сравнение с null** | `x.equals(null) == false` для любого ненулевого `x` |

**Последствия нарушения симметричности в HashSet**

`HashSet` (и `HashMap`) при поиске объекта действует в два шага:
1. Вычисляет `hashCode()` и определяет бакет (корзину).
2. Внутри бакета перебирает элементы, вызывая `equals()`.

Если симметричность нарушена — например, `a.equals(b) == true`, но `b.equals(a) == false` — поведение коллекции становится непредсказуемым:

- **`set.contains(a)`** может вернуть `true`, а **`set.contains(b)`** — `false` (или наоборот), хотя `a.equals(b)` утверждает, что они равны.
- При итерации по множеству оба объекта могут присутствовать одновременно, что нарушает инвариант `Set` (отсутствие дубликатов).
- Результат зависит от того, какой объект был добавлен первым и в какой бакет он попал.

**Классический пример нарушения** — расширение класса с переопределённым `equals()` и добавление нового поля в подкласс. Тогда `child.equals(parent)` учитывает дополнительное поле, а `parent.equals(child)` — нет. Именно поэтому нарушение симметричности при наследовании является одной из самых распространённых ошибок.

**Как реализовать корректно:**
```java
@Override
public boolean equals(Object o) {
    if (this == o) return true;               // рефлексивность
    if (o == null || getClass() != o.getClass()) return false; // null + тип
    MyClass that = (MyClass) o;
    return Objects.equals(field1, that.field1)
        && Objects.equals(field2, that.field2);
}
```

### Вопрос 26. Почему при переопределении equals() обязательно нужно переопределить hashCode()? Что произойдёт с объектом в HashMap, если equals() переопределён, а hashCode() — нет?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Why to Override equals(Object) and hashCode() method? — GeeksforGeeks](https://www.geeksforgeeks.org/java/override-equalsobject-hashcode-method/)

Согласно Джошуа Блоху («Effective Java»): «Вы обязаны переопределить `hashCode()` в каждом классе, который переопределяет `equals()`».

Фундаментальная причина — в том, как работают хеш-коллекции: они используют `hashCode()` для нахождения бакета, а затем `equals()` для сравнения объектов внутри бакета. Оба метода должны работать согласованно.

Контракт `Object` устанавливает: «Если два объекта равны согласно `equals(Object)`, то вызов `hashCode()` на каждом из них должен возвращать одинаковое целое число».

**Что происходит без переопределения `hashCode()` (только `equals()`):** дефолтный `hashCode()` генерирует разные значения для логически равных объектов. Это приводит к тому, что равные объекты попадают в разные бакеты, и `HashMap` считает их отдельными ключами. В итоге оба значения сохраняются — создаются «дублирующие ключи» вопреки логическому равенству, контракт нарушается, поведение коллекции становится некорректным.

#### Ответ от Claude

**Контракт hashCode() из спецификации java.lang.Object**

Официальный контракт требует трёх свойств:
1. **Согласованность в рамках одного запуска** — повторные вызовы `hashCode()` на неизменённом объекте всегда возвращают одно и то же число.
2. **Равные объекты имеют одинаковый хеш-код** — если `x.equals(y) == true`, то `x.hashCode() == y.hashCode()` (обязательное требование).
3. **Неравные объекты могут иметь разные хеш-коды** — это не обязательно, но желательно для производительности.

**Механизм работы HashMap**

`HashMap` использует двухшаговый алгоритм поиска:
1. `hashCode()` → номер бакета (корзины в массиве).
2. `equals()` → идентификация нужного элемента внутри бакета.

Если шаг 1 не находит нужный бакет — шаг 2 никогда не выполнится.

**Что происходит, если переопределить только equals(), но не hashCode()**

```java
public class Person {
    private String name;
    
    @Override
    public boolean equals(Object o) {
        if (!(o instanceof Person)) return false;
        return name.equals(((Person) o).name);
    }
    // hashCode() НЕ переопределён — используется Object.hashCode()
    // который основан на адресе объекта в памяти
}

Person p1 = new Person("Alice");
Person p2 = new Person("Alice");

p1.equals(p2);          // true  — equals говорит «равны»
p1.hashCode() == p2.hashCode(); // false — разные адреса в памяти

Map<Person, String> map = new HashMap<>();
map.put(p1, "developer");
map.get(p2);            // null! — p2 попадает в другой бакет
```

**Итог нарушения контракта:**
- `map.get(p2)` возвращает `null`, хотя логически `p2` — это тот же ключ.
- `map.containsKey(p2)` возвращает `false`.
- `HashMap` создаёт два отдельных слота для `p1` и `p2`, нарушая семантику Map.
- Аналогичная проблема возникает в `HashSet`: оба «равных» объекта сохраняются в множестве.

**Правило:** переопределяйте оба метода всегда вместе. В современном Java удобно использовать `Objects.hash()` или IDE-генерацию, а в Java 16+ — `record`, который генерирует оба метода автоматически.

### Вопрос 27. Как правильно создать неизменяемый (immutable) класс? Перечислите все шаги, включая обращение с изменяемыми полями-объектами (например, полем типа List или Date).

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [How to Create an Immutable Class in Java — DigitalOcean](https://www.digitalocean.com/community/tutorials/how-to-create-immutable-class-in-java)

Для создания неизменяемого класса необходимо соблюдать следующие принципы:

1. Пометить класс как `final`, чтобы предотвратить создание подклассов.
2. Объявить все поля `private final`.
3. Полностью исключить сеттеры.
4. Выполнять **защитное копирование** (defensive copy) изменяемых аргументов конструктора.
5. Возвращать **защитные копии** из геттеров для изменяемых объектов.

Для изменяемых коллекций (Map, List) небезопасно возвращать прямую ссылку на внутреннее поле:
```java
public HashMap<String, String> getConfig() {
    return config; // Небезопасно — открывает внутреннее состояние
}
```
Правильный подход — возвращать копию:
```java
public Map<String, String> getConfig() {
    return new HashMap<>(config);
}
```
Или неизменяемый вид:
```java
return Collections.unmodifiableMap(new HashMap<>(config));
```

Для полей типа `Date` — копировать при сохранении и при возврате:
```java
// В конструкторе:
this.createdAt = new Date(date.getTime());
// В геттере:
public Date getCreatedAt() {
    return new Date(createdAt.getTime());
}
```

#### Ответ от Claude

**Полный чеклист создания immutable-класса**

**Шаг 1. Объявить класс `final`**
```java
public final class ImmutablePerson { ... }
```
Без `final` подкласс может добавить изменяемые поля или переопределить методы, нарушив неизменяемость.

**Шаг 2. Все поля — `private final`**
```java
private final String name;
private final int age;
private final Date birthDate;
private final List<String> roles;
```
`private` — запрет прямого доступа снаружи; `final` — запрет повторного присвоения.

**Шаг 3. Никаких сеттеров**
Любой метод `setXxx()` нарушает неизменяемость — его быть не должно.

**Шаг 4. Инициализация только через конструктор с защитным копированием входных данных**

```java
public ImmutablePerson(String name, int age, Date birthDate, List<String> roles) {
    this.name = name;
    this.age = age;
    // Защитная копия Date — иначе вызывающий код может изменить дату
    this.birthDate = new Date(birthDate.getTime());
    // Защитная копия List — иначе вызывающий код может изменить список
    this.roles = new ArrayList<>(roles);
}
```

Без защитной копии в конструкторе вызывающий код может сделать:
```java
List<String> roles = new ArrayList<>(List.of("admin"));
ImmutablePerson p = new ImmutablePerson("Alice", 30, now, roles);
roles.add("superuser"); // Мутирует внутреннее состояние p!
```

**Шаг 5. Геттеры изменяемых полей возвращают защитные копии**

```java
public Date getBirthDate() {
    return new Date(birthDate.getTime()); // Копия, не оригинал
}

public List<String> getRoles() {
    return Collections.unmodifiableList(roles); // или new ArrayList<>(roles)
}
```

**Полный пример:**
```java
public final class ImmutablePerson {
    private final String name;
    private final Date birthDate;
    private final List<String> roles;

    public ImmutablePerson(String name, Date birthDate, List<String> roles) {
        this.name = name;
        this.birthDate = new Date(birthDate.getTime());
        this.roles = new ArrayList<>(roles);
    }

    public String getName() { return name; }

    public Date getBirthDate() { return new Date(birthDate.getTime()); }

    public List<String> getRoles() { return Collections.unmodifiableList(roles); }
}
```

**Важные нюансы:**

- **Глубокое vs поверхностное копирование** — если `List` содержит изменяемые объекты (не `String`, а POJO), нужно копировать и их тоже.
- **`Date` устарел** — в современном Java лучше использовать `java.time.LocalDate` / `java.time.Instant`, которые уже неизменяемы.
- **Java 16+ Records** — `record` автоматически делает поля `final` и не предоставляет сеттеры, но защитное копирование в конструкторе по-прежнему нужно прописывать вручную.
- **`Collections.unmodifiableList`** оборачивает оригинальный список — если кто-то держит ссылку на оригинал, он всё равно может его изменить. Поэтому обязательно сначала скопировать, а потом обернуть: `Collections.unmodifiableList(new ArrayList<>(roles))`.

### Вопрос 28. Что означает final для переменной, для метода и для класса? Можно ли изменить состояние объекта, на который указывает final-ссылка?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [final Keyword in Java — GeeksforGeeks](https://www.geeksforgeeks.org/java/final-keyword-in-java/)

Ключевое слово `final` в Java применяется к переменным, методам и классам:

- **Final-переменная** — после присвоения значение изменить нельзя. Однако когда `final`-переменная хранит ссылку на объект, внутреннее состояние объекта может изменяться. Ограничение касается только самой ссылки: нельзя перенаправить её на другой объект, но можно изменить поля или вызвать мутирующие методы объекта. Статья демонстрирует это на примере `StringBuilder`: ссылку переопределить нельзя, но метод `append()` модифицирует содержимое.
- **Final-метод** — не может быть переопределён в подклассе. Метод при этом наследуется и может вызываться, его лишь нельзя заменить другой реализацией.
- **Final-класс** — не может быть унаследован. Ни один другой класс не может расширить (`extend`) такой класс. Класс `String` — показательный пример `final`-класса в стандартной библиотеке Java.

#### Ответ от Claude

**`final` для переменной**

Запрещает повторное присвоение переменной после её инициализации.

```java
final int x = 10;
x = 20; // Ошибка компиляции

final List<String> list = new ArrayList<>();
list = new LinkedList<>(); // Ошибка компиляции — нельзя переназначить ссылку
list.add("hello");         // OK — состояние объекта изменяется, ссылка та же
```

Виды `final`-переменных:
- **Локальная переменная** — может быть объявлена без инициализации, но должна быть инициализирована до первого использования (effectively final).
- **Поле экземпляра** — должно быть инициализировано либо в объявлении, либо в конструкторе.
- **Статическое поле** — должно быть инициализировано либо в объявлении, либо в статическом инициализаторе.

**`final` для метода**

Запрещает переопределение метода (`@Override`) в подклассах. Метод наследуется и может вызываться — лишь нельзя предоставить другую реализацию.

```java
public class Base {
    public final void doSomething() { ... }
}

public class Child extends Base {
    @Override
    public void doSomething() { ... } // Ошибка компиляции
}
```

Используется, когда нужно гарантировать неизменность поведения: например, в шаблонном методе (`Template Method` pattern), где шаги алгоритма фиксированы.

**`final` для класса**

Запрещает наследование от класса. Ни один класс не может указать его в `extends`.

```java
public final class ImmutablePoint {
    private final int x, y;
    ...
}

public class MyPoint extends ImmutablePoint { } // Ошибка компиляции
```

Примеры `final`-классов в JDK: `String`, `Integer`, `Long`, `Math`, `System`. Как правило, они финализированы ради безопасности или гарантии неизменяемости.

**Можно ли изменить состояние объекта, на который указывает final-ссылка?**

Да. `final` фиксирует только саму ссылку (адрес в памяти), но не состояние объекта, на который она указывает.

```java
final StringBuilder sb = new StringBuilder("Hello");
sb.append(", World!"); // OK — sb всё ещё указывает на тот же объект
System.out.println(sb); // "Hello, World!"

sb = new StringBuilder(); // Ошибка компиляции — нельзя изменить саму ссылку
```

**Итог:** чтобы объект был неизменяемым, одного `final` недостаточно — нужна полноценная реализация immutable-класса (см. вопрос №27). `final`-ссылка лишь гарантирует, что вы не начнёте ссылаться на другой объект, но не защищает от мутации текущего.

### Вопрос 29. В каком порядке выполняются static-блоки инициализации, блоки инициализации экземпляра и конструкторы при создании объекта класса-наследника? Что произойдёт, если в static-блоке будет выброшено непроверяемое исключение?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [In what order do static blocks and initialization blocks execute when using inheritance? — CodeLessGenie](https://www.codelessgenie.com/blog/in-what-order-do-static-blocks-and-initialization-blocks-execute-when-using-inheritance/)

При использовании наследования в Java порядок выполнения следующий:

1. **Статические блоки родительского класса** — выполняются первыми при первой загрузке класса в память.
2. **Статические блоки дочернего класса** — выполняются после завершения инициализации родителя.
3. **Блоки инициализации экземпляра родительского класса** — выполняются при создании экземпляра.
4. **Конструктор родительского класса** — запускается после блоков инициализации экземпляра родителя.
5. **Блоки инициализации экземпляра дочернего класса** — выполняются следующими.
6. **Конструктор дочернего класса** — запускается последним.

Статические блоки выполняются один раз при загрузке класса в память, независимо от количества создаваемых экземпляров. Блоки инициализации экземпляра выполняются каждый раз при создании нового объекта, непосредственно перед конструктором. Внутри каждого класса блоки выполняются в том порядке, в котором они объявлены в исходном коде (сверху вниз).

#### Ответ от Claude

**Полный порядок выполнения при создании объекта класса-наследника:**

```
1. static-блоки и static-поля суперкласса (один раз, при первой загрузке)
2. static-блоки и static-поля подкласса (один раз, при первой загрузке)
3. Блоки инициализации экземпляра суперкласса (каждый раз)
4. Конструктор суперкласса (каждый раз)
5. Блоки инициализации экземпляра подкласса (каждый раз)
6. Конструктор подкласса (каждый раз)
```

Несколько ключевых уточнений:

- **Static-блоки**: выполняются ровно один раз — когда JVM впервые загружает класс. Если классов в иерархии несколько, сначала загружается и инициализируется корневой суперкласс (кроме `Object`), затем каждый следующий потомок по цепочке.
- **Блоки инициализации экземпляра**: компилятор буквально вставляет их содержимое в начало каждого конструктора (после вызова `super()`). Поэтому они выполняются при каждом создании объекта, до тела конструктора.
- **Порядок внутри одного класса**: несколько static-блоков (или несколько блоков инициализации экземпляра) выполняются в том порядке, в котором они объявлены в файле — сверху вниз.

**Пример:**
```java
class Parent {
    static { System.out.println("1. Parent static block"); }
    { System.out.println("3. Parent instance block"); }
    Parent() { System.out.println("4. Parent constructor"); }
}

class Child extends Parent {
    static { System.out.println("2. Child static block"); }
    { System.out.println("5. Child instance block"); }
    Child() { System.out.println("6. Child constructor"); }
}

// new Child() выведет:
// 1. Parent static block
// 2. Child static block
// 3. Parent instance block
// 4. Parent constructor
// 5. Child instance block
// 6. Child constructor
```

**Что происходит при непроверяемом исключении в static-блоке:**

Если в static-блоке (или при инициализации статического поля) выбрасывается непроверяемое исключение (unchecked exception или `Error`), JVM оборачивает его в `java.lang.ExceptionInInitializerError` и выбрасывает его при первой попытке использовать класс. Класс при этом **навсегда помечается как «сломанный»** в рамках текущего сеанса JVM.

При любой последующей попытке обратиться к этому классу (создать объект, вызвать статический метод и т.д.) JVM будет выбрасывать `java.lang.NoClassDefFoundError: Could not initialize class <ClassName>` — уже без оборачивания, сразу. Это значит, что класс становится недоступным до перезапуска JVM.

```java
class Broken {
    static {
        if (true) throw new RuntimeException("Oops");
    }
}

// Первое обращение:
// ExceptionInInitializerError caused by: RuntimeException("Oops")

// Второе и последующие обращения:
// NoClassDefFoundError: Could not initialize class Broken
```

Проверяемые исключения (`checked exceptions`) в static-блоке вызывают ошибку компиляции — их нельзя объявить в сигнатуре блока. Поэтому если нужно обработать проверяемое исключение, его следует перехватить внутри блока и при необходимости перебросить как `RuntimeException` или `Error`.

### Вопрос 30. Что такое «blank final» поле? В каких точках оно должно быть инициализировано и может ли blank final поле быть static?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Static and Non Static Blank Final Variables in Java — GeeksforGeeks](https://www.geeksforgeeks.org/java/static-non-static-blank-final-variables-java/)

**Blank final** (пустая финальная переменная) — это `final`-переменная, которая объявляется без немедленного присвоения начального значения; она должна быть присвоена ровно один раз до того, как будет использована.

**Нестатические blank final поля** должны быть инициализированы в конструкторе класса. Каждый объект получает свою копию финальной переменной, которую нельзя изменить после присвоения. Инициализировать их в обычных методах нельзя.

**Статические blank final поля** — да, blank final может быть статическим. Такие поля существуют в единственном экземпляре для всего класса. Они должны быть инициализированы в статическом блоке инициализации. Инициализировать их в конструкторе или обычных методах нельзя.

#### Ответ от Claude

**Blank final** — это `final`-поле (или локальная переменная), объявленное без инициализатора. Термин введён в Java Language Specification (JLS). Например:

```java
class Example {
    final int x;           // blank final поле экземпляра
    static final int Y;    // blank static final поле
}
```

**Правила инициализации blank final полей экземпляра:**

Согласно JLS §16.9, blank final поле экземпляра должно быть **определённо присвоено (definitely assigned)** перед завершением каждого конструктора класса. Допустимые точки инициализации:

1. **Блок инициализации экземпляра** (`{ ... }`) — выполняется перед каждым конструктором.
2. **Инициализатор объявления** — при объявлении поля, но это уже не «blank», а обычный final.
3. **Конструктор** — в теле каждого конструктора класса (кроме тех, что делегируют через `this(...)`; там поле считается присвоенным после `this(...)`).

Присваивание в обычных методах запрещено — компилятор выдаст ошибку.

```java
class Person {
    final String name;  // blank final

    Person(String name) {
        this.name = name;  // OK — в конструкторе
    }
    
    void setName(String name) {
        this.name = name;  // ОШИБКА КОМПИЛЯЦИИ
    }
}
```

**Правила инициализации blank static final полей:**

Согласно JLS §16.8, blank static final поле должно быть **определённо присвоено** в статическом блоке инициализации или статическом инициализаторе поля. Конструктор и методы для этого не подходят.

```java
class Config {
    static final String DB_URL;  // blank static final

    static {
        DB_URL = System.getenv("DB_URL");  // OK — в static-блоке
    }
}
```

**Может ли blank final быть static?** Да, безусловно. JLS явно описывает оба варианта: `blank final member field` (нестатическое) и `blank final class variable` (статическое, т.е. `static final`).

**Гарантия «определённого присвоения» (definite assignment):**

Компилятор Java использует анализ потока управления, чтобы убедиться, что:
- blank final присваивается ровно один раз во всех возможных путях выполнения,
- blank final не читается до присвоения.

Если компилятор не может доказать, что переменная будет присвоена во всех ветках (например, только в одной ветке `if` без `else`), он выдаёт ошибку компиляции.

```java
class Bad {
    final int value;
    Bad(boolean flag) {
        if (flag) {
            value = 1;
        }
        // ОШИБКА: value может остаться неинициализированным
    }
}

class Good {
    final int value;
    Good(boolean flag) {
        if (flag) {
            value = 1;
        } else {
            value = 0;
        }
        // OK: value присваивается во всех ветках
    }
}
```

**Практическое применение blank final:**
- Инициализация из конструктора (dependency injection, immutable value objects).
- Инициализация static final констант из окружения (`System.getenv`, `System.getProperty`).
- Условная инициализация, которую нельзя выразить в одну строку.

### Вопрос 31. Почему String в Java является immutable? Назовите как минимум три причины (безопасность, кэширование hashCode, String Pool, потокобезопасность) и объясните каждую.

**Категория:** Core Java — основы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Why String Is Immutable in Java – Key Reasons Explained — JavaKnowledgeBase](https://javaknowledgebase.com/why-string-is-immutable-in-java/)

String в Java является иммутабельным — после создания его содержимое нельзя изменить. Вот ключевые причины:

**String Pool (эффективность памяти):** String Pool — это специальная область памяти в куче Java, где хранятся уникальные строковые литералы. Когда создаётся строка с помощью двойных кавычек, JVM проверяет, существует ли она в пуле, и если да — возвращает ссылку на существующий объект, снижая потребление памяти. Если бы строки были изменяемыми, изменение одного объекта в пуле затронуло бы все ссылки на него.

**Безопасность:** Строки используются для хранения паролей, URL баз данных, имён пользователей. Иммутабельность предотвращает изменение их значений вредоносным кодом. Изменяемые строки могли бы быть модифицированы посторонним кодом в том же приложении.

**Постоянство хэш-кода:** «Метод `hashCode()` для строки вычисляется один раз и кэшируется для дальнейшего использования.» Это делает строки надёжными ключами в `HashMap` и `HashSet` на протяжении всего времени их жизни.

**Потокобезопасность:** Несколько потоков могут безопасно использовать один и тот же объект `String` без механизмов синхронизации. Иммутабельность исключает гонки данных в конкурентных окружениях.

#### Ответ от Claude

`String` в Java объявлен иммутабельным (неизменяемым) намеренно — это фундаментальное дизайнерское решение языка, обоснованное четырьмя взаимосвязанными причинами.

**1. Безопасность (Security)**

Строки широко используются для передачи чувствительных данных: путей к файлам, паролей, URL-адресов баз данных, имён пользователей. Если бы `String` был изменяемым, злоумышленный код мог бы изменить значение строки после прохождения проверки безопасности, но до её фактического использования:

```java
// Гипотетическая уязвимость при изменяемых строках:
String path = "/safe/path";
checkAccess(path);          // проверка пройдена
path.setValue("/etc/passwd"); // злоумышленник изменяет строку
openFile(path);             // открывается недопустимый файл
```

Иммутабельность гарантирует, что строка не может быть изменена между моментом проверки и моментом использования (TOCTOU-атаки невозможны).

**2. Кэширование hashCode**

Поскольку содержимое строки никогда не меняется, её хэш-код вычисляется лишь однажды — при первом вызове `hashCode()` — и затем сохраняется в поле `private int hash`. При последующих вызовах возвращается кэшированное значение:

```java
// Упрощённая реализация в OpenJDK:
private int hash; // по умолчанию 0

public int hashCode() {
    int h = hash;
    if (h == 0 && !hashIsZero) {
        h = computeHash();
        if (h == 0) {
            hashIsZero = true;
        } else {
            hash = h;
        }
    }
    return h;
}
```

Это делает строки идеальными ключами для `HashMap`, `HashSet` и других хэш-структур: хэш-код строки гарантированно стабилен на протяжении всего времени жизни объекта.

**3. String Pool (пул строк)**

JVM поддерживает пул интернированных строк (String Pool / String Constant Pool), где хранятся уникальные экземпляры строковых литералов. Когда в коде встречается строковый литерал, JVM проверяет пул и возвращает ссылку на уже существующий объект, если он там есть:

```java
String a = "hello";
String b = "hello";
System.out.println(a == b); // true — один объект в пуле
```

Эта оптимизация работает только при иммутабельности: если бы строки были изменяемыми, изменение `a` повлекло бы изменение `b`, что полностью сломало бы концепцию пула.

**4. Потокобезопасность (Thread Safety)**

Иммутабельные объекты по природе своей потокобезопасны — их состояние не может измениться после создания. Несколько потоков могут читать один и тот же объект `String` одновременно без синхронизации, блокировок и опасности гонок данных:

```java
// Безопасно — никакой синхронизации не требуется:
String shared = "shared value";
Runnable task = () -> System.out.println(shared);
new Thread(task).start();
new Thread(task).start();
```

Если бы `String` был изменяемым, совместное использование строк между потоками требовало бы явной синхронизации или копирования.

**Техническая реализация иммутабельности:**

- Класс `String` объявлен как `final` — от него нельзя унаследоваться и переопределить поведение.
- Внутренний массив символов (`char[] value` до Java 9, `byte[] value` с Java 9) объявлен `private final` — прямой доступ к нему извне невозможен.
- Все методы `String`, которые «изменяют» строку (`substring`, `replace`, `concat` и т.д.), возвращают **новый** объект `String`, не трогая исходный.

### Вопрос 32. Как изменилось расположение String Pool в памяти JVM между Java 6 и Java 7+? Почему это важно с точки зрения OutOfMemoryError?

**Категория:** Core Java — основы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Java String Pool — How Unbounded intern() Causes PermGen OOM — TheCodeForge](https://thecodeforge.io/java/string-pool-java/)

**Java 6 и ранее:** String Pool располагался в **PermGen (Permanent Generation)** — области памяти фиксированного размера, отдельной от основной кучи. PermGen хранил метаданные классов, интернированные строки и внутренние структуры JVM, но **не подвергался сборке мусора**. Критическое ограничение: PermGen имел жёсткий потолок размера (задавался через `-XX:MaxPermSize`). После заполнения он не мог сжиматься, поэтому бесконтрольные вызовы `intern()` были катастрофическими. Как указывается в статье: «один цикл с вызовом `intern()` на уникальных строках мог исчерпать PermGen» и вызвать падение с `OutOfMemoryError: PermGen space`.

**Java 7+:** String Pool был перемещён на **основную кучу (main heap)**. Это архитектурное изменение кардинально всё изменило:
- Строки в пуле стали **доступны сборщику мусора**, когда на них нет ссылок.
- Пул теперь может расти и уменьшаться в ходе обычных циклов GC.
- Java 8 полностью упразднила PermGen, заменив его Metaspace (нативная память).

Как говорится в источнике: «Пул переехал из PermGen в кучу в Java 7, что сделало строки в пуле доступными для GC» и «строки в пуле, на которые нет ссылок, теперь наконец могут быть собраны». В современных JVM чрезмерное использование `intern()` всё ещё может привести к исчерпанию кучи, но сборка мусора теперь предоставляет «предохранительный клапан», и специфическая ошибка `PermGen OOM` — это проблема исключительно Java 6 и старше.

#### Ответ от Claude

**Расположение String Pool в Java 6 и ранее:**

До Java 7 String Pool (пул строк, также называемый String Constant Pool или String Intern Pool) располагался в **PermGen (Permanent Generation)** — специальной фиксированной области памяти JVM, отдельной от основной кучи (Heap). PermGen предназначался для хранения:
- загруженных метаданных классов и методов,
- интернированных строк (String Pool),
- статических переменных (до Java 7),
- пула констант.

Размер PermGen задавался параметром `-XX:MaxPermSize` и **не мог динамически расти** сверх установленного лимита. Типичные значения — от 32 МБ до 96 МБ.

**Расположение String Pool в Java 7+:**

В Java 7 String Pool был перенесён в **основную кучу (Heap)** — туда же, где находятся обычные Java-объекты. Это стало возможным без потери семантики: строки в пуле теперь подчиняются тем же правилам сборки мусора, что и остальные объекты на куче.

В Java 8 PermGen был **полностью упразднён** и заменён на **Metaspace** — область в нативной памяти процесса (вне кучи JVM). Metaspace хранит метаданные классов, но не String Pool.

**Сводная таблица:**

| Параметр | Java 6 | Java 7 | Java 8+ |
|---|---|---|---|
| Расположение String Pool | PermGen | Heap | Heap |
| Хранилище метаданных классов | PermGen | PermGen | Metaspace |
| Подвержен GC | Нет | Да (Heap) | Да (Heap) |
| Размер пула по умолчанию | ~1 009 бакетов | ~60 013 бакетов | ~65 536 бакетов |

**Почему это важно с точки зрения OutOfMemoryError:**

**Проблема в Java 6:** Метод `String.intern()` позволяет явно помещать строку в пул. В Java 6 каждая уникальная интернированная строка навсегда оставалась в PermGen и никогда не собиралась GC. При массовом интернировании строк (например, в цикле обработки пользовательского ввода, XML-тегов или данных из БД) PermGen быстро заполнялся, и JVM бросала:
```
java.lang.OutOfMemoryError: PermGen space
```

Это было особенно коварно: PermGen имел отдельный лимит от `-Xmx`, поэтому даже при большом значении `-Xmx` PermGen мог переполниться.

**Решение в Java 7+:** Поскольку String Pool теперь живёт в куче, строки из пула, на которые больше нет внешних ссылок, **могут быть собраны сборщиком мусора**. Это устраняет главную проблему — пул больше не является «чёрной дырой» памяти. При нехватке памяти теперь возникает стандартная:
```
java.lang.OutOfMemoryError: Java heap space
```
которую можно решить увеличением `-Xmx`, тогда как `PermGen OOM` в Java 6 требовал отдельного увеличения `-XX:MaxPermSize`.

**Практические выводы:**
- В Java 6 следует избегать массового использования `intern()` с динамическими строками.
- В Java 7+ `intern()` безопаснее, но всё равно требует осторожности — чрезмерное интернирование может увеличить давление на GC.
- Переход с Java 6 на Java 7+ устраняет целый класс ошибок `PermGen OOM`, связанных со String Pool.
- Конфигурационный параметр `-XX:StringTableSize` (задаёт число бакетов в хэш-таблице пула) доступен в обеих версиях, но актуален и в современных JVM для тюнинга высоконагруженных приложений.
### Вопрос 33. Что напечатает следующий код и почему? double x = Double.NaN; System.out.println(x == x); System.out.println(x != x); Как корректно проверять на NaN?

**Категория:** Core Java — основы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Nan (Not a Number) in Java — GeeksforGeeks](https://www.geeksforgeeks.org/java/nan-not-number-java/)

Сравнение `Double.NaN == Double.NaN` возвращает `false`. Это контринтуитивное поведение объясняется тем, что «любое сравнение с NaN вернёт false». Аналогично, `x != x` для NaN-значения возвращает `true`, потому что «оператор неравенства `!=` возвращает true, если хотя бы один операнд является NaN». Java следует стандарту IEEE 754 для чисел с плавающей точкой, который трактует NaN как неупорядоченное значение. Стандарт существует потому, что некоторые математические операции порождают неопределённые результаты — например, `Math.sqrt(-1)` или `0.0 / 0.0` — которые нельзя корректно представить. Чтобы правильно проверить значение на NaN, нужно использовать метод `Double.isNaN()`:

```java
Double y = new Double(0.0 / 0.0);
System.out.println(y.isNaN()); // вернёт true
```

#### Ответ от Claude

**Что напечатает код:**

```
false
true
```

**Почему так происходит — стандарт IEEE 754:**

`Double.NaN` — это специальное значение типа `double`, определённое стандартом IEEE 754. Оно представляет результат неопределённых арифметических операций (`0.0/0.0`, `Math.sqrt(-1)`, `Double.POSITIVE_INFINITY - Double.POSITIVE_INFINITY` и т.п.).

По спецификации IEEE 754 и Java Language Specification (§15.21.1), NaN считается неупорядоченным: **ни одно сравнение, в котором участвует NaN, не возвращает `true`, за исключением `!=`**. Это означает:

- `NaN == NaN` → `false` (NaN не равен самому себе)
- `NaN != NaN` → `true` (единственный оператор, возвращающий `true` с NaN)
- `NaN < x`, `NaN > x`, `NaN <= x`, `NaN >= x` → все возвращают `false`

**Как правильно проверять на NaN:**

1. **Рекомендуемый способ — `Double.isNaN(x)` или `Float.isNaN(x)`:**

```java
double x = Double.NaN;
if (Double.isNaN(x)) {
    System.out.println("x is NaN"); // будет напечатано
}
```

2. **Через `Double` (boxed) — метод экземпляра:**

```java
Double boxed = Double.NaN;
System.out.println(boxed.isNaN()); // true
```

3. **Трюк `x != x`** (работает, но не рекомендован для читаемости):

```java
boolean isNaN = (x != x); // true только для NaN
```

**Важный нюанс — поведение boxing:**

`Double.equals()` для двух boxed-объектов, оба содержащих NaN, вернёт `true`, хотя `Double.NaN == Double.NaN` возвращает `false`. Это объясняется тем, что `equals()` реализован через `Double.doubleToLongBits()`, а не через оператор `==`. Поэтому поведение при сравнении примитива и объекта различается.

```java
Double a = Double.NaN;
Double b = Double.NaN;
System.out.println(a == b);       // false (boxed == primitve semantics не применяется здесь, но: unboxing)
System.out.println(a.equals(b));  // true  (использует doubleToLongBits)
```

**Итог:** всегда используйте `Double.isNaN()` — это явно, безопасно и соответствует намерению кода.

### Вопрос 34. Компилятор Java оптимизирует конкатенацию строк через StringBuilder. Когда эта оптимизация НЕ работает (например, конкатенация в цикле) и что именно генерируется в байткоде в каждом случае?

**Категория:** Core Java — основы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [String concatenation in Java 9: Untangling invokeDynamic — Guardsquare](https://www.guardsquare.com/blog/string-concatenation-java-9-untangling-invokedynamic)

В Java 8 конкатенация строк компилировалась непосредственно в явные вызовы `StringBuilder`:

```
new StringBuilder()
append(String)
append(String)
toString()
```

Это порождало многословный байткод с несколькими вызовами методов на каждую операцию конкатенации. В Java 9 (JEP 280) была введена «indified String concatenation»: вместо явных вызовов `StringBuilder` компилятор генерирует одну инструкцию `invokedynamic`, которая делегирует работу bootstrap-методу во время выполнения. Bootstrap-методом служит `StringConcatFactory`, получающий «рецепт» — строку со специальными маркерами (``, ``), указывающими, брать ли значения из стека или из констант. Главное преимущество — возможность менять стратегию конкатенации без перекомпиляции кода.

#### Ответ от Claude

**Три ключевых случая:**

**1. Конкатенация вне цикла (компилятор оптимизирует)**

До Java 9 компилятор преобразовывал:
```java
String result = "Hello " + name + "!";
```
В эквивалент:
```java
String result = new StringBuilder().append("Hello ").append(name).append("!").toString();
```

Байткод (Java 8) содержал: `new StringBuilder`, затем цепочку вызовов `invokevirtual StringBuilder.append`, затем `invokevirtual StringBuilder.toString`.

Начиная с Java 9, тот же исходный код компилируется в один вызов `invokedynamic` с bootstrap-методом `StringConcatFactory.makeConcatWithConstants`. JVM самостоятельно выбирает оптимальную стратегию в runtime, без изменений в байткоде при обновлении JVM.

**2. Конкатенация в цикле — оптимизация НЕ работает**

```java
String result = "";
for (int i = 0; i < n; i++) {
    result += "data";  // ПРОБЛЕМА
}
```

Для **Java 8** компилятор не может оптимизировать это в единый `StringBuilder` снаружи цикла. В каждой итерации генерируется:
- создание нового `StringBuilder`
- `append` накопленной строки `result` (которая растёт с каждой итерацией)
- `append` добавляемой строки
- `toString()` с записью обратно в `result`

Итог: создаётся N объектов `StringBuilder`, N промежуточных объектов `String`, сложность — O(n²) по памяти и времени. При 1 000 000 итераций — миллион объектов, серьёзная нагрузка на GC.

Для **Java 9+** в цикле по-прежнему генерируется `invokedynamic` на каждой итерации, но каждый вызов создаёт новый объект-конкатенатор. Принципиальная проблема — создание промежуточных строк в каждой итерации — остаётся.

**3. Правильное решение для цикла**

```java
StringBuilder sb = new StringBuilder();
for (int i = 0; i < n; i++) {
    sb.append("data");
}
String result = sb.toString();
```

Один объект `StringBuilder` на весь цикл, O(n) сложность.

**Почему компилятор не может оптимизировать цикл автоматически:**

Компилятор не анализирует поток управления внутри цикла достаточно глубоко, чтобы вынести `StringBuilder` за пределы итерации. Эта задача требует сложного анализа зависимостей и эффектов, выходящего за рамки javac. В теории JIT-компилятор (HotSpot) мог бы это сделать, но на практике эта оптимизация не реализована: цикличная конкатенация строк через `+` остаётся ответственностью разработчика.

**Как проверить байткод:** `javap -c -verbose MyClass.class`

**Итог:** используйте `+` для простых объединений вне циклов (компилятор справится), и `StringBuilder` — при конкатенации внутри циклов или при построении строк по частям.

### Вопрос 35. Можно ли переопределить hashCode() так, чтобы он всегда возвращал одну константу? Формально ли это нарушает контракт? Каковы реальные последствия для производительности HashMap?

**Категория:** Core Java — основы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [The 3 things you should know about hashCode() — EclipseSource](https://eclipsesource.com/blogs/2012/09/04/the-3-things-you-should-know-about-hashcode/)

Фундаментальное правило контракта: «объекты, которые равны, должны иметь одинаковый хеш-код в рамках одного запущенного процесса». Однако обратное не требуется — неравные объекты могут иметь одинаковый хеш-код, не нарушая контракта. Если `hashCode()` всегда возвращает одно значение, все объекты попадают в один бакет. Хотя это формально удовлетворяет контракту, это вызывает серьёзную деградацию производительности. Вместо «рассортировки объектов по ячейкам, как в наборщиках кассах», HashMap вынужден линейно перебирать все записи в единственном бакете. Для больших наборов данных это превращает операции HashMap из почти константных в линейный перебор, фактически делая структуру данных бесполезной.

#### Ответ от Claude

**Можно ли вернуть константу:**

Да, технически это разрешено Java. Пример:

```java
@Override
public int hashCode() {
    return 42; // абсолютно легальный код
}
```

**Формально ли это нарушает контракт hashCode:**

Нет, контракт формально не нарушается. Контракт `hashCode()` (JavaDoc `Object.hashCode()`) требует:

1. **Стабильность:** несколько вызовов `hashCode()` на одном объекте должны возвращать одно значение в рамках одного запуска программы (при условии, что объект не изменился).
2. **Согласованность с equals:** если `a.equals(b) == true`, то `a.hashCode() == b.hashCode()`.
3. **Обратное не требуется:** если `a.hashCode() == b.hashCode()`, `a.equals(b)` может быть `false` (коллизия — допустима).

Константный `hashCode` удовлетворяет всем трём условиям:
- Значение всегда одинаково → стабильность соблюдена.
- Равные объекты имеют одинаковый хеш → условие 2 соблюдено.

JavaDoc прямо говорит: «не требуется, чтобы неравные объекты имели различные хеш-коды», — и добавляет, что реализация, возвращающая константу, «теоретически корректна, но практически катастрофична».

**Реальные последствия для HashMap:**

HashMap хранит записи в массиве бакетов. Индекс бакета для ключа вычисляется как `(n - 1) & hash`, где `hash` — производная от `hashCode()`. При константном `hashCode` все объекты попадают в **один и тот же бакет**.

До Java 8: бакет реализован как связный список. Сложность `get()` и `put()` деградирует до **O(n)** — линейный поиск по всем элементам.

Начиная с Java 8 (JEP по улучшению HashMap): когда число элементов в одном бакете превышает порог `TREEIFY_THRESHOLD = 8`, связный список автоматически преобразуется в сбалансированное красно-чёрное дерево. Это улучшает худший случай с **O(n)** до **O(log n)**. Однако для работы treeification требуется, чтобы ключи реализовывали `Comparable`; иначе дерево не строится и деградация остаётся O(n).

**Итоговая таблица сложности:**

| Ситуация | get/put сложность |
|---|---|
| Хороший hashCode (равномерное распределение) | O(1) амортизировано |
| Константный hashCode, Java < 8 | O(n) |
| Константный hashCode, Java 8+, Comparable ключи | O(log n) |
| Константный hashCode, Java 8+, не Comparable | O(n) |

**Практический вывод:** константный `hashCode` превращает `HashMap` в структуру с линейным или логарифмическим поиском. Для 1 000 000 элементов вместо ~1 операции поиска придётся перебирать от 1 000 000 до 20 элементов. Это делает использование `HashMap` бессмысленным.

### Вопрос 36. Как static-поле ведёт себя при наследовании? Что происходит, если дочерний класс «переопределяет» (скрывает) static-поле родителя, и как это связано с понятием hiding vs overriding?

**Категория:** Core Java — основы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Overriding and Hiding Methods — Oracle Java Tutorials](https://docs.oracle.com/javase/tutorial/java/IandI/override.html)

Официальная документация Oracle разграничивает два понятия. Для методов экземпляра применяется термин **overriding (переопределение)**: версия метода, которая вызывается, определяется типом объекта в runtime. Для статических методов применяется термин **hiding (скрытие)**: версия скрытого статического метода, которая вызывается, зависит от того, вызывается ли он из суперкласса или подкласса. Пример из документации:

```java
public class Animal {
    public static void testClassMethod() {
        System.out.println("The static method in Animal");
    }
    public void testInstanceMethod() {
        System.out.println("The instance method in Animal");
    }
}

public class Cat extends Animal {
    public static void testClassMethod() {
        System.out.println("The static method in Cat");
    }
    public void testInstanceMethod() {
        System.out.println("The instance method in Cat");
    }

    public static void main(String[] args) {
        Cat myCat = new Cat();
        Animal myAnimal = myCat;
        Animal.testClassMethod();      // "The static method in Animal"
        myAnimal.testInstanceMethod(); // "The instance method in Cat"
    }
}
```

Принципиальное отличие: статические методы разрешаются на этапе компиляции по типу ссылки, а не по типу объекта в runtime.

#### Ответ от Claude

**Как static-поле ведёт себя при наследовании:**

Статические поля принадлежат классу, а не объекту. При наследовании дочерний класс **видит** статическое поле родителя, и оно физически одно — не создаётся отдельная копия для каждого класса в иерархии.

```java
class Parent {
    static int count = 0;
}
class Child extends Parent {}

Child.count++;          // изменяет Parent.count
System.out.println(Parent.count); // 1 — то же самое поле
```

**Что такое hiding (скрытие) статического поля:**

Если дочерний класс объявляет статическое поле с тем же именем, что и у родителя, дочернее поле **скрывает** (hides) родительское — не переопределяет его. Оба поля существуют независимо:

```java
class Parent {
    static String role = "Parent";
}
class Child extends Parent {
    static String role = "Child"; // скрывает Parent.role, не переопределяет
}

System.out.println(Parent.role); // "Parent"
System.out.println(Child.role);  // "Child"

Parent ref = new Child();
System.out.println(ref.role);    // "Parent" — тип ссылки, не тип объекта!
```

**Ключевое отличие hiding от overriding:**

| Аспект | Overriding (методы экземпляра) | Hiding (статические методы и поля) |
|---|---|---|
| Что разрешает | Тип объекта в runtime | Тип ссылки в compile-time |
| Полиморфизм | Да | Нет |
| Применимость | Только к instance-методам | К static-методам и полям |
| JVM механизм | `invokevirtual` (динамическая диспетчеризация) | `invokestatic` / прямая ссылка (статически) |

**Демонстрация разницы для методов:**

```java
class Animal {
    public static void staticMethod() { System.out.println("Animal static"); }
    public void instanceMethod()      { System.out.println("Animal instance"); }
}

class Dog extends Animal {
    public static void staticMethod() { System.out.println("Dog static"); }
    @Override
    public void instanceMethod()      { System.out.println("Dog instance"); }
}

Animal a = new Dog();
a.staticMethod();   // "Animal static"  — hiding: тип ссылки (Animal)
a.instanceMethod(); // "Dog instance"   — overriding: тип объекта (Dog)
```

**Почему static нельзя «по-настоящему» переопределить:**

Полиморфизм (overriding) работает через виртуальную таблицу методов (vtable) — структуру, построенную по типу объекта в heap. Статические методы и поля не входят в vtable: они привязаны к классу как таковому и разрешаются компилятором на этапе компиляции по типу ссылки. Поэтому передача объекта подкласса через ссылку суперкласса не влияет на то, какой static-метод или static-поле будет вызвано/прочитано.

**Практические ловушки:**

1. Компилятор разрешает вызов статического метода через ссылку на объект (`obj.staticMethod()`), но это синтаксический сахар — фактически вызывается метод по типу ссылки, а не по типу объекта. Это плохой стиль; IDEs предупреждают об этом.
2. Нельзя объявить поле `static` в дочернем классе и `instance` с тем же именем в родительском (и наоборот) — компилятор запрещает смешение.
3. При сериализации, рефлексии и прочих метапрограммных техниках важно явно указывать класс для получения нужного static-поля.
### Вопрос 37. Почему класс-обёртка Integer является final и immutable? Что плохого могло бы произойти, если бы Integer можно было наследовать и изменять его значение?

**Категория:** Core Java — основы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Primitive Wrapper Classes are Immutable in Java — TutorialsPoint](https://www.tutorialspoint.com/primitive-wrapper-classes-are-immutable-in-java)

Классы-обёртки (Integer, Byte, Long, Float, Double, Character, Boolean, Short) спроектированы как неизменяемые по нескольким причинам.

**Потокобезопасность.** Поскольку состояние неизменяемых объектов не может быть изменено после создания, они автоматически синхронизированы. Явная блокировка не требуется — несколько потоков могут безопасно делить одни и те же экземпляры.

**Стабильность состояния.** Неизменяемые объекты-обёртки не могут войти в несогласованное состояние после создания, что исключает ошибки, связанные с неожиданным изменением значения в ходе выполнения программы.

**Безопасное разделение и кэширование.** Ссылки на неизменяемые объекты можно свободно передавать без копирования. Если в десятке мест приложения нужен Integer со значением 42, можно использовать один экземпляр. На этом основан целочисленный кэш (-128..127).

**Надёжность как ключей HashMap.** Неизменяемые классы-обёртки служат надёжными ключами карт, поскольку их hashCode никогда не меняется, что исключает потери при поиске.

Демонстрация: если бы обёртки были изменяемыми, возникли бы непреднамеренные модификации:
```java
Integer i = new Integer(20);
operate(i);  // внутри: i = i + 1
System.out.println(i);  // Если бы Integer был изменяемым — вывело бы 21!
```

#### Ответ от Claude

**Почему `Integer` является `final`**

Объявление класса `final` запрещает создание подклассов. Это ключевое решение, поскольку именно наследуемость в сочетании с переопределением методов нарушила бы гарантии, на которых строится вся платформа.

**Почему `Integer` является immutable**

`Integer` хранит единственное поле `private final int value`, которое инициализируется в конструкторе и никогда не изменяется. Нет ни одного setter-метода.

---

**Что плохого произошло бы, если бы `Integer` можно было наследовать и изменять**

1. **Нарушение кэша Integer.valueOf().** JVM кэширует значения от -128 до 127. Если бы можно было создать подкласс `MutableInteger extends Integer` и изменить внутреннее значение, все потребители кэшированного экземпляра увидели бы изменение:
   ```java
   Integer a = Integer.valueOf(100); // кэшированный экземпляр
   // если бы Integer был изменяемым:
   a.setValue(999); // сломало бы все код, получившие тот же кэшированный объект
   ```

2. **Поломка контракта hashCode/equals при использовании в коллекциях.** `HashMap` и `HashSet` вычисляют позицию элемента по `hashCode` в момент вставки. Если значение изменится, `hashCode` изменится тоже, и объект станет «потерянным» в коллекции: `containsKey` вернёт `false` для ранее помещённого ключа.
   ```java
   Map<Integer, String> map = new HashMap<>();
   Integer key = new Integer(5);
   map.put(key, "five");
   key.setValue(999); // если бы это было возможно
   map.get(key);      // вернёт null — ключ потерян
   ```

3. **Нарушение потокобезопасности.** Неизменяемые объекты безопасны в многопоточной среде без дополнительной синхронизации. Изменяемый `Integer` потребовал бы явных барьеров памяти везде, где объект передаётся между потоками.

4. **Уязвимость класса через подкласс.** Без `final` злоумышленник мог бы создать подкласс, переопределить `equals()` и `hashCode()` так, чтобы экземпляры выглядели равными разным значениям, нарушая контракт метода `equals` (симметричность, транзитивность).

5. **Проблемы автоупаковки (autoboxing).** Компилятор неявно конвертирует `int` в `Integer`. Если бы значение могло меняться, семантика примитивов (передача по значению) и объектов (передача по ссылке) конфликтовала бы непредсказуемо:
   ```java
   int x = 5;
   Integer y = x; // autoboxing
   // если y изменился бы — x остался бы прежним, что сбивало бы с толку
   ```

**Итог.** `final` + `immutable` — это не просто удобство, это архитектурное требование для корректной работы кэширования, коллекций, многопоточности и автоупаковки. Эти свойства взаимозависимы: убери одно — рухнет вся система гарантий.

### Вопрос 38. Каков результат: String s = "a" + "b" + "c"; — будет ли создан один объект в пуле или три? Как компилятор обрабатывает конкатенацию строковых литералов на этапе компиляции?

**Категория:** Core Java — основы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Understanding String Interning and Constant Folding in Java — Medium (Danylo Liakhovetskyi)](https://medium.com/@danylo.liakh/understanding-string-interning-and-constant-folding-in-java-bb4516d0f870)

Когда вы пишете `String s = "a" + "b" + "c"`, компилятор Java выполняет **constant folding** (свёртку констант) на этапе компиляции, объединяя все строковые литералы в единственный литерал `"abc"` ещё до запуска программы.

Создаётся **один объект** — единственная интернированная строка `"abc"` в пуле строк. Промежуточных объектов от конкатенации не возникает, поскольку эта оптимизация происходит при компиляции, а не в рантайме.

Constant folding работает только при конкатенации исключительно литеральных значений. Если в конкатенации участвует переменная — даже строковая — компилятор не может выполнить свёртку, и объект создаётся в рантайме.

Именно поэтому:
```java
String a = "Hello";
String b = "Hel" + "lo"; // compile-time constant expression
System.out.println(a == b); // true — один и тот же интернированный объект
```

#### Ответ от Claude

**Результат: создаётся ОДИН объект в пуле строк**

Компилятор Java обрабатывает `String s = "a" + "b" + "c"` следующим образом:

**Механизм: Constant Folding (свёртка констант)**

Это оптимизация, при которой компилятор вычисляет значение выражения целиком на этапе компиляции. Нормативное описание содержится в Java Language Specification (JLS §3.10.5 и §15.29):

> *«Строки, являющиеся значениями константных выражений, интернируются, чтобы разделять уникальные экземпляры, как если бы выполнялся метод `String.intern()`»*

И (JLS §15.29):
> *«Строки, конкатенированные из константных выражений, вычисляются на этапе компиляции и трактуются затем как если бы они были литералами»*

Таким образом, `"a" + "b" + "c"` компилятор превращает в один строковый литерал `"abc"`, который помещается в constant pool класса в байт-коде. При загрузке класса JVM интернирует этот литерал — и в пуле строк (String Pool / String Intern Pool) существует ровно **один объект**.

**Практическое подтверждение**

```java
String s1 = "abc";
String s2 = "a" + "b" + "c";   // compile-time constant expression
String s3 = "a" + "b" + "c";   // тот же литерал

System.out.println(s1 == s2);  // true — один объект в пуле
System.out.println(s1 == s3);  // true
System.out.println(s2 == s3);  // true
```

**Ключевое отличие: переменные разрушают constant folding**

```java
String a = "a";
String s4 = a + "b" + "c";    // a — переменная, не константа

System.out.println(s1 == s4); // false — новый объект в heap
```

Здесь `a` — не `final`, поэтому компилятор не знает её значение на этапе компиляции. Конкатенация происходит в рантайме с использованием `StringBuilder` (до Java 9) или `StringConcatFactory` (с Java 9+), и новый объект создаётся в куче, а не в пуле.

**Если переменная объявлена как `final`, constant folding применяется снова:**
```java
final String a = "a";
String s5 = a + "b" + "c";    // final — компилятор подставляет константу
System.out.println(s1 == s5); // true — снова один объект из пула
```

**Сводка**

| Выражение | Место создания | Количество объектов |
|---|---|---|
| `"a" + "b" + "c"` | String Pool | 1 (только `"abc"`) |
| `a + "b" + "c"` (a — обычная переменная) | Heap | 1 новый в heap |
| `a + "b" + "c"` (a — final) | String Pool | 1 (из пула) |

**Важно:** промежуточные строки `"a"`, `"b"`, `"c"`, `"ab"` при конкатенации литералов **не создаются** вообще. Компилятор производит сразу результирующую строку.

### Вопрос 39. Чем отличается реализация equals() и hashCode() для записей (Java Records, Java 16+) от ручной реализации в обычном классе? Какие поля участвуют в сравнении по умолчанию?

**Категория:** Core Java — основы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [6 Record Classes — Oracle Java 17 Documentation](https://docs.oracle.com/en/java/javase/17/language/records.html)

В Java Records реализации `equals()` и `hashCode()` генерируются автоматически. Согласно документации:

> *«Реализации методов `equals` и `hashCode` определяют, что два объекта класса-записи равны, если они одного типа и содержат равные значения компонент»*

```java
// Эквивалентный обычный класс (требует ручного написания equals и hashCode)
public final class Rectangle {
    private final double length;

    public Rectangle(double length, double width) { ... }
    double length() { return this.length; }
    double width()  { return this.width; }
    public boolean equals(Object obj) { /* ручная реализация */ }
    public int hashCode() { /* ручная реализация */ }
}
```

#### Ответ от Claude

**Автоматическая генерация в Records**

При объявлении записи:
```java
record Point(int x, int y) { }
```

компилятор автоматически синтезирует:
- `equals(Object o)` — сравнивает тип и все компоненты
- `hashCode()` — хеш по всем компонентам
- `toString()` — строковое представление всех компонент
- канонический конструктор
- accessor-методы `x()` и `y()`

**Какие поля участвуют в сравнении**

Участвуют **все и только компоненты**, перечисленные в заголовке записи (record header). Дополнительные поля, объявленные в теле записи (instance fields), в Java Records запрещены — запись не может иметь нестатических полей, помимо компонент. Поэтому «по умолчанию» означает «все поля записи без исключения».

Сгенерированный `equals` эквивалентен примерно такому коду:
```java
@Override
public boolean equals(Object obj) {
    if (this == obj) return true;
    if (!(obj instanceof Point other)) return false;
    return this.x == other.x && this.y == other.y;
}
```

Для ссылочных типов используется `Objects.equals(a, b)` (null-безопасно), для примитивов — `==`.

**Сравнение с ручной реализацией в обычном классе**

| Аспект | Java Record | Обычный класс |
|---|---|---|
| Источник реализации | Генерируется компилятором | Пишется вручную (или IDE/Lombok) |
| Поля в сравнении | Все компоненты из заголовка | Разработчик выбирает сам |
| Риск ошибки | Минимален | Высок (забытое поле, неверный тип) |
| Проверка типа | Встроена (instanceof + getClass check) | Нужно реализовать вручную |
| Контракт equals-hashCode | Гарантирован автоматически | Ответственность разработчика |
| Возможность кастомизации | Можно переопределить | Полный контроль |

**Нюанс: записи с мутабельными компонентами**

Если компонент записи является мутабельным объектом (например, `List` или `Date`), автоматически сгенерированный `hashCode()` может изменить своё значение при модификации этого объекта. Это нарушит инвариант при использовании записи как ключа в `HashMap`/`HashSet`. В таком случае рекомендуется либо использовать неизменяемые компоненты (`List.copyOf()`), либо явно переопределить `hashCode()`:

```java
record PersonRecord(String name, List<String> tags) {
    // Небезопасно: hashCode зависит от мутабельного List
    // Решение: сделать копию в compact constructor
    PersonRecord {
        tags = List.copyOf(tags); // неизменяемая копия
    }
}
```

**Переопределение в записи**

Разрешено переопределять `equals()` и `hashCode()` в теле записи, но согласно официальной документации Oracle, переопределённые реализации должны сохранять ту же семантику: два экземпляра записи равны тогда и только тогда, когда они одного типа и все их компоненты равны.

### Вопрос 40. Что такое «defensive copy» в контексте immutability? Покажите пример, где его отсутствие нарушает неизменяемость класса, и объясните, где именно (в конструкторе или геттере) нужно делать копию.

**Категория:** Core Java — основы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Ensuring Immutability in Java: Defensive Copying of Lists — Medium (Vinotech)](https://medium.com/@vino7tech/ensuring-immutability-in-java-defensive-copying-of-lists-8dc61cfb5a07)

Defensive copy (защитная копия) — техника, при которой создаются независимые копии мутабельных объектов при передаче в конструктор или возврате из геттера, чтобы предотвратить модификацию внутреннего состояния иммутабельного класса извне.

Без защитной копии неизменяемость нарушается:
```java
public final class BrokenImmutable {
    private final List<String> items;

    public BrokenImmutable(List<String> items) {
        this.items = items; // Прямая ссылка — нет копии!
    }

    public List<String> getItems() {
        return items; // Возвращает прямую ссылку!
    }
}

// Это ломает неизменяемость:
List<String> original = new ArrayList<>();
original.add("secret");
BrokenImmutable obj = new BrokenImmutable(original);
original.add("exposed"); // Изменяет внутреннее состояние obj!
```

Решение — защитные копии в обоих местах:
```java
// В конструкторе:
public ImmutableClass(List<String> items) {
    this.items = new ArrayList<>(items);
}

// В геттере:
public List<String> getItems() {
    return new ArrayList<>(items);
}
```

Альтернатива: `Collections.unmodifiableList()` — предотвращает модификации без накладных расходов на копирование.

#### Ответ от Claude

**Что такое defensive copy**

Defensive copy (защитная копия) — это практика создания копии мутабельного объекта при его получении извне (в конструкторе) или при его выдаче наружу (в геттере). Цель — разорвать связь между внутренним состоянием класса и внешними ссылками на мутабельные объекты.

**Пример нарушения immutability без defensive copy**

Рассмотрим класс с полем типа `Date` (мутабельный класс):

```java
public final class ImmutablePeriod {
    private final Date start;
    private final Date end;

    public ImmutablePeriod(Date start, Date end) {
        this.start = start; // ОШИБКА: прямая ссылка без копии
        this.end   = end;
    }

    public Date getStart() { return start; } // ОШИБКА: возвращает прямую ссылку
    public Date getEnd()   { return end; }
}

// Атака через конструктор:
Date start = new Date();
Date end   = new Date();
ImmutablePeriod period = new ImmutablePeriod(start, end);
end.setTime(0); // Изменяем end — и внутреннее состояние period тоже изменилось!

// Атака через геттер:
period.getEnd().setTime(0); // Получили ссылку и изменили внутренний объект!
```

**Где именно нужно делать копию**

Копию нужно делать **в обоих местах** по разным причинам:

**1. В конструкторе** — защита от атаки через внешнюю ссылку после создания объекта:
```java
public ImmutablePeriod(Date start, Date end) {
    // Делаем копию ДО валидации, чтобы предотвратить TOCTOU-атаку:
    this.start = new Date(start.getTime()); // defensive copy
    this.end   = new Date(end.getTime());   // defensive copy

    if (this.start.after(this.end)) {
        throw new IllegalArgumentException("start after end");
    }
}
```

Важно: копию нужно делать **до** проверки валидности, иначе в многопоточной среде возможна атака типа TOCTOU (time-of-check/time-of-use): значение может измениться между проверкой и сохранением.

**2. В геттере** — защита от атаки через возвращённую ссылку:
```java
public Date getStart() {
    return new Date(start.getTime()); // defensive copy
}

public Date getEnd() {
    return new Date(end.getTime());   // defensive copy
}
```

**Аналогичная схема для коллекций**

```java
public final class Department {
    private final List<String> employees;

    public Department(List<String> employees) {
        this.employees = new ArrayList<>(employees); // defensive copy в конструкторе
    }

    public List<String> getEmployees() {
        return new ArrayList<>(employees); // defensive copy в геттере
        // Альтернатива: Collections.unmodifiableList(employees)
    }
}
```

**Когда достаточно только конструктора, а когда нужен и геттер**

- **Только конструктор** защищает от изменения через ссылку, переданную при создании.
- **Только геттер** защищает от изменения через ссылку, полученную при чтении.
- Для полной immutability нужны **оба**.

**Альтернативы и нюансы**

| Подход | Плюсы | Минусы |
|---|---|---|
| `new ArrayList<>(list)` | Копия независима | Стоимость O(n) памяти |
| `Collections.unmodifiableList(list)` | Нет копирования | Не защищает от изменения через оригинальную ссылку |
| `List.copyOf(list)` (Java 10+) | Неизменяемая копия | Не допускает null-элементов |

Для глубоко вложенных мутабельных объектов (например, `List<Person>`, где `Person` тоже мутабелен) поверхностной копии недостаточно — требуется глубокое (deep) копирование каждого элемента.

**Правило:** неизменяемость класса ровно настолько сильна, насколько неизменяемы его поля. Если поле ссылается на мутабельный объект — без defensive copy класс лишь притворяется иммутабельным.
