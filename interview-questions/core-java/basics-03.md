### Вопрос 41. Что такое расширяющее (widening) и сужающее (narrowing) преобразование примитивных типов? В каких случаях компилятор выполняет его автоматически, а в каких требует явного каста? Что происходит со значением при сужении int до byte?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [The Java Language Specification SE 21, §5.1.2–5.1.3](https://docs.oracle.com/javase/specs/jls/se21/html/jls-5.html#jls-5.1.2)

**Расширяющее преобразование (Widening Primitive Conversion, §5.1.2)**

Спецификация определяет 19 расширяющих преобразований:

- `byte` → `short`, `int`, `long`, `float`, `double`
- `short` → `int`, `long`, `float`, `double`
- `char` → `int`, `long`, `float`, `double`
- `int` → `long`, `float`, `double`
- `long` → `float`, `double`
- `float` → `double`

Расширяющее преобразование не теряет информацию о порядке величины в следующих случаях (значение сохраняется точно):
- из целочисленного типа в другой целочисленный тип;
- из `byte`, `short` или `char` во floating-point тип;
- из `int` в `double`.

Потеря **точности** (но не порядка) возможна при: `int → float`, `long → float`, `long → double`. В этих случаях применяется политика IEEE 754 «round to nearest» (округление до ближайшего). Расширяющее преобразование не вызывает исключений во время выполнения.

**Сужающее преобразование (Narrowing Primitive Conversion, §5.1.3)**

Спецификация определяет 22 сужающих преобразования:

- `short` → `byte`, `char`
- `char` → `byte`, `short`
- `int` → `byte`, `short`, `char`
- `long` → `byte`, `short`, `char`, `int`
- `float` → `byte`, `short`, `char`, `int`, `long`
- `double` → `byte`, `short`, `char`, `int`, `long`, `float`

Сужающее преобразование может терять информацию о величине, знаке и точности.

**Что происходит при сужении `int` до `byte`:**
Сужающее преобразование знакового целого числа в тип T просто отбрасывает все биты, кроме n младших, где n — количество бит в типе T. Для `byte` n = 8. Это может изменить знак результата по сравнению с исходным значением.

Пример из спецификации: `(byte)255 == -1`, поскольку 255 = `0xFF`, и при сохранении только 8 младших бит в дополнительном коде получается -1.

Компилятор применяет расширяющее преобразование **автоматически** в контекстах присваивания, вызова метода и числовых операций. Сужающее преобразование **требует явного каста**, за исключением: когда выражение является константным выражением типа `byte`, `short`, `char` или `int`, и его значение укладывается в целевой тип (например, `byte b = 42;` компилируется без каста, а `byte b = 300;` — нет).

#### Ответ от Claude

**Расширяющее преобразование (widening)**

Происходит, когда тип-приёмник «шире» типа-источника (может вместить большее диапазон значений). JLS §5.1.2 перечисляет 19 таких преобразований — например, `byte → int`, `int → long`, `float → double`. Компилятор применяет расширяющее преобразование **неявно** в трёх контекстах:

1. **Контекст присваивания** (`int i = 42; long l = i;`).
2. **Контекст вызова метода** — когда тип фактического аргумента уже, чем тип параметра.
3. **Числовой контекст** — при вычислении арифметических выражений.

Гарантии: расширяющее преобразование между целочисленными типами сохраняет значение точно (знаковые типы расширяются sign-extension, `char` — zero-extension). Преобразования `int → float`, `long → float`, `long → double` могут потерять несколько младших бит (потеря точности, но не порядка), поскольку формат IEEE 754 имеет ограниченную мантиссу. Пример: `int big = 1234567890; float approx = big;` — разница `big - (int)approx` равна `-46`.

**Сужающее преобразование (narrowing)**

Тип-приёмник «у́же» источника. JLS §5.1.3 перечисляет 22 таких преобразования. Требует **явного каста** (`(byte) value`), потому что компилятор не может гарантировать отсутствие потери данных.

Исключение: если выражение является **константным выражением времени компиляции** (compile-time constant expression, JLS §15.29) типа `byte`, `short`, `char` или `int`, и его числовое значение укладывается в диапазон целевого типа, то явный каст не нужен. Именно поэтому `byte b = 42;` — допустимо, а `byte b = 128;` — ошибка компиляции.

**Что происходит при `int → byte`:**

Алгоритм: отбрасываются все биты, кроме 8 младших (n = 8 для `byte`). Результат интерпретируется в дополнительном коде:

- `(byte) 127` → `0x7F` → `127` (без изменений, укладывается в byte)
- `(byte) 128` → `0x80` → `-128` (смена знака)
- `(byte) 255` → `0xFF` → `-1`
- `(byte) 256` → `0x100`, остаются 8 бит → `0x00` → `0`

Формула: результат = `(int_value & 0xFF)`, интерпретированный как знаковый byte. Исключения при сужении **не бросаются** — это гарантируется JLS §5.1.3.

### Вопрос 42. Что такое числовое продвижение (numeric promotion) в Java? Почему выражение `byte a = 1; byte b = 2; byte c = a + b;` не компилируется, хотя сумма укладывается в диапазон byte?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [The Java Language Specification SE 21, §5.6 (Numeric Contexts)](https://docs.oracle.com/javase/specs/jls/se21/html/jls-5.html#jls-5.6)

JLS §5.6 описывает числовое продвижение (numeric promotion), применяемое к операндам арифметических операторов. Существует два вида:

**Унарное числовое продвижение (§5.6.1):** если операнд имеет тип `byte`, `short` или `char`, он расширяется до `int`; если `int` — остаётся `int`; `long`, `float`, `double` — остаются без изменений.

**Двоичное числовое продвижение (Binary Numeric Promotion, §5.6.2):** применяется к паре операндов. Правила по приоритету:
1. Если хотя бы один операнд типа `double` — оба приводятся к `double`.
2. Иначе, если хотя бы один типа `float` — оба к `float`.
3. Иначе, если хотя бы один типа `long` — оба к `long`.
4. **Иначе — оба приводятся к `int`**, даже если оба операнда были `byte`, `short` или `char`.

Из спецификации (§5.6): «In a numeric arithmetic context or a numeric array context, the promoted type is `int`, and any expressions that are not of type `int` undergo widening primitive conversion to `int`.»

Пример из спецификации:
```java
byte b = 0x1f;
char c = 'G';
int control = c & b; // char и byte повышаются до int
```

Поэтому в выражении `a + b`, где `a` и `b` типа `byte`, оба операнда повышаются до `int`, и результат имеет тип `int`. Присвоить `int` переменной типа `byte` без явного каста нельзя (сужающее преобразование).

#### Ответ от Claude

**Числовое продвижение (numeric promotion)** — механизм, при котором Java автоматически приводит операнды арифметических, битовых и сдвиговых операций к более широкому типу перед вычислением. Это часть спецификации JLS §5.6.

**Почему Java так устроена?**

JVM оперирует в основном 32-битными (int) и 64-битными (long) значениями на уровне байткода. Для типов `byte`, `short`, `char` не существует отдельных инструкций сложения, умножения и т.д. — JVM использует `iadd`, `imul` и т.п. (операции над int). Поэтому на уровне спецификации Java закрепила правило: любая арифметика с типами «у́же» `int` выполняется как арифметика `int`.

**Разбор примера `byte a = 1; byte b = 2; byte c = a + b;`:**

1. `byte a = 1;` и `byte b = 2;` — допустимо: литералы являются константными выражениями, 1 и 2 укладываются в `byte`, компилятор выполняет неявное сужение.
2. `a + b` — компилятор применяет binary numeric promotion: оба операнда `a` и `b` повышаются до `int`, операция `+` выполняется как `int + int`, результат имеет тип **`int`**.
3. `byte c = a + b;` — попытка присвоить `int` переменной `byte` — **сужающее преобразование**, которое требует явного каста. Компилятор выдаёт ошибку: `incompatible types: possible lossy conversion from int to byte`.

**Ключевое различие:** `byte c = 1 + 2;` **скомпилируется** — потому что `1 + 2` является константным выражением; компилятор вычисляет значение (3) на этапе компиляции, проверяет, что 3 укладывается в `byte`, и выполняет неявное сужение. Для переменных `a` и `b` такой оптимизации нет, поскольку их значения не известны на этапе компиляции как compile-time constants (они — effectively final, но не compile-time constant expressions по определению JLS §15.29).

**Исправление:**
```java
byte c = (byte)(a + b); // явный каст — правильный способ
```

**Расхождений между источниками нет:** JLS SE 8, SE 10, SE 21 единообразно описывают правило binary numeric promotion в §5.6.

### Вопрос 43. Почему переменные, захватываемые в лямбда-выражении или анонимном классе, должны быть final или effectively final? Что произошло бы, если бы это ограничение отсутствовало?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Lambda Expressions — The Java Tutorials (Oracle)](https://docs.oracle.com/javase/tutorial/java/javaOO/lambdaexpressions.html)

Официальный туториал Oracle сообщает:

«Like local and anonymous classes, a lambda expression can only access local variables and parameters of the enclosing block that are final or effectively final.»

Переменная является **effectively final**, если её значение не изменяется после первоначального присваивания, даже если перед ней не стоит ключевое слово `final`.

Пример из туториала:
```java
void methodInFirstLevel(int x) {
    int z = 2;  // effectively final — не переприсваивается

    Consumer<Integer> myConsumer = (y) -> {
        System.out.println("x = " + x); // OK
        System.out.println("z = " + z); // OK
    };
}
```

Если попытаться изменить значение `z` внутри лямбды или после неё, компилятор выдаёт ошибку: «Local variable z defined in an enclosing scope must be final or effectively final».

Причина — локальные переменные хранятся в стеке и имеют ограниченное время жизни. Лямбда может выполниться после того, как метод, объявивший переменную, уже завершился. Поэтому среда выполнения **копирует** значение локальной переменной в лямбду, а не передаёт ссылку на ячейку стека.

#### Ответ от Claude

**Механизм захвата переменных**

Когда лямбда-выражение (или анонимный класс) обращается к локальной переменной охватывающего метода, JVM не может передать ссылку на ячейку стека напрямую — стек кадра метода прекращает существование, как только метод завершается, тогда как лямбда может жить значительно дольше (например, будучи переданной в другой поток или возвращённой из метода). Поэтому компилятор **копирует значение** локальной переменной в поле синтезированного класса, реализующего функциональный интерфейс.

JLS §15.27.2 («Lambda Body») фиксирует ограничение явно: любая локальная переменная, чьё значение используется в теле лямбды, должна быть `final` или effectively final. Rationale из спецификации:

> «The restriction to effectively final variables alleviates problems that come from the need to capture local variable values (as opposed to capturing variables themselves, which would require heap-allocated mutable cells). In particular, it avoids a form of capture confusion wherein the captured value of a local variable in one lambda expression could differ from the captured value of the same local variable in another lambda expression, due to an assignment in between the two lambda expressions.»

**Что произошло бы без ограничения?**

1. **Несогласованность между лямбдами.** Если бы переменную разрешили изменять, два экземпляра лямбды, захватившие одну и ту же переменную в разные моменты времени, «видели» бы разные значения. Это трудно отлаживается и нарушает принцип наименьшего удивления.

2. **Гонка данных (data race) в многопоточной среде.** Лямбда часто передаётся в пул потоков (`ExecutorService`, `CompletableFuture` и т.д.). Если бы лямбда держала mutable-ссылку на стековую переменную, а основной поток мог бы её изменять, возникли бы гонки без какой-либо синхронизации — и без каких-либо предупреждений компилятора.

3. **Необходимость heap-allocated mutable cells.** Для разрешения захвата изменяемых переменных компилятору пришлось бы оборачивать каждую такую переменную в объект на куче (аналогично `int[]` длиной 1 или `AtomicInteger`). Это добавило бы неявный overhead и сделало бы семантику захвата неочевидной.

**Поле vs. локальная переменная**

Ограничение распространяется **только на локальные переменные** (включая параметры методов). Поля объекта (`this.field`) и статические поля (`ClassName.staticField`) лямбда может читать и изменять без ограничений — они живут на куче и не «умирают» вместе со стековым кадром метода. Именно поэтому паттерн с `int[] counter = {0};` или `AtomicInteger counter = new AtomicInteger()` позволяет обойти ограничение при осознанной необходимости.

**Анонимные классы**

До Java 8 анонимные классы требовали явного ключевого слова `final` перед захватываемой переменной. Java 8 ввела понятие effectively final, распространив его одновременно на анонимные классы и лямбды: теперь `final` можно не писать явно, если переменная фактически не изменяется.

**Расхождений между источниками нет:** JLS SE 8 и SE 21 единообразно описывают ограничение в §15.27.2, а официальный туториал Oracle воспроизводит его с аналогичным обоснованием.

### Вопрос 44. Что такое effectively final? Чем оно отличается от явного final? Приведите пример переменной, которая является effectively final, и пример, которая — нет.

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [What is Effectively Final variable of Java 8? — Javarevisited](https://javarevisited.blogspot.com/2015/03/what-is-effectively-final-variable-of.html)

Effectively final (фактически финальная) переменная — это нефинальная локальная переменная или параметр метода, значение которой никогда не меняется после инициализации. В отличие от переменной с явным ключевым словом `final`, такая переменная не требует явного объявления — компилятор сам определяет её неизменность по использованию.

**Явный final:**
```java
final String message = "Hello";
// компилятор обеспечивает неизменность через ключевое слово
```

**Effectively final:**
```java
String message = "Hello";
// переменная не переприсваивается, поэтому компилятор считает её финальной
```

**Пример effectively final переменной (используется в лямбде):**
```java
String effectiveFinal = "I am non final local variable";
Runnable r = () -> {
    System.out.println(effectiveFinal); // OK в Java 8
};
```

**Пример переменной, которая НЕ является effectively final:**
```java
String notEffectivelyFinal = "initial";
notEffectivelyFinal = "changed"; // переприсвоение нарушает effectively final
Runnable r = () -> {
    System.out.println(notEffectivelyFinal); // Ошибка компиляции
};
```

Понятие effectively final введено в Java 8. Лямбда-выражения и анонимные классы могут захватывать только те локальные переменные, которые объявлены `final` или являются effectively final. Это ограничение предотвращает проблемы параллелизма, связанные с захватом изменяемых локальных переменных.

#### Ответ от Claude

**Определение (JLS 4.12.4)**

Переменная является *effectively final* (фактически финальной), если она не объявлена с ключевым словом `final`, но её значение не изменяется ни разу после инициализации. Другими словами, если добавить `final` в объявление такой переменной, программа останется корректной с точки зрения компилятора.

Понятие введено в Java 8 (JLS §4.12.4). До Java 8 анонимные классы требовали явного `final`; Java 8 облегчила это требование, распространив его на effectively final переменные.

**Отличия от явного final**

| Критерий | Явный `final` | Effectively final |
|---|---|---|
| Синтаксис | Ключевое слово `final` обязательно | Ключевое слово отсутствует |
| Обеспечение компилятором | Компилятор запрещает переприсвоение на уровне синтаксиса | Компилятор проверяет отсутствие переприсвоений по факту использования |
| Компиляция констант | Строковые/примитивные `final`-поля со значением времени компиляции становятся compile-time constants и могут инлайниться | Effectively final переменные инлайнингу не подвергаются |
| Применимость | Поля классов, локальные переменные, параметры | Только локальные переменные и параметры методов |

**Примеры**

```java
// 1. Effectively final — переменная не переприсваивается
int x = 42;
Runnable r1 = () -> System.out.println(x); // OK

// 2. НЕ effectively final — переприсвоение после инициализации
int y = 10;
y = 20; // это переприсвоение лишает переменную статуса effectively final
Runnable r2 = () -> System.out.println(y); // Ошибка компиляции

// 3. НЕ effectively final — инкремент является скрытым переприсвоением
int count = 0;
count++; // эквивалентно count = count + 1
Runnable r3 = () -> System.out.println(count); // Ошибка компиляции

// 4. Effectively final — изменение состояния объекта не нарушает статус
// (меняется содержимое объекта, а не ссылка)
List<String> list = new ArrayList<>();
list.add("item"); // ссылка list не переприсваивается — effectively final
Runnable r4 = () -> System.out.println(list.size()); // OK
```

**Важные нюансы**

- Ограничение на effectively final распространяется только на **локальные переменные**. Поля экземпляра и статические поля класса захватываются в лямбдах без ограничений (через неявную ссылку `this` или через класс).
- Причина ограничения — семантика захвата: JVM копирует значение локальной переменной в экземпляр лямбды/анонимного класса. Если переменная могла бы изменяться, копия стала бы несинхронизированной с оригиналом, что привело бы к гонкам данных.
- Компилятор не выполняет дополнительных оптимизаций для effectively final переменных, в отличие от явных `final`-констант (которые могут инлайниться в байткоде).

### Вопрос 45. С какой версии Java в switch-операторе можно использовать строки? Как компилятор реализует такой switch под капотом — какие вызовы методов генерируются в байткоде?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Strings in switch Statements — Oracle Java 8 Documentation](https://docs.oracle.com/javase/8/docs/technotes/guides/language/strings-switch.html)

Строки в `switch`-операторе поддерживаются начиная с **JDK 7**.

`switch`-оператор сравнивает строковый объект в своём выражении с выражениями, связанными с каждой меткой `case`, **как если бы использовался метод `String.equals()`**; соответственно, сравнение строк в `switch` является регистрозависимым (case-sensitive).

Java-компилятор генерирует из `switch`-операторов со строками **более эффективный байткод**, чем из цепочек `if-then-else`.

Пример:
```java
public String getTypeOfDay(String dayOfWeekArg) {
    switch (dayOfWeekArg) {
        case "Monday":
            return "Start of work week";
        case "Friday":
            return "End of work week";
        case "Saturday":
        case "Sunday":
            return "Weekend";
        default:
            throw new IllegalArgumentException("Invalid day: " + dayOfWeekArg);
    }
}
```

#### Ответ от Claude

**С какой версии**

Поддержка строк в `switch`-операторе появилась в **Java 7 (JDK 7, 2011)**. До этой версии в `switch` можно было использовать только `int`, `char`, `byte`, `short`, их обёртки и `enum`.

**Как компилятор реализует switch-на-строке под капотом**

JVM-инструкции `tableswitch` и `lookupswitch` работают только с целочисленными значениями. Поэтому компилятор преобразует строковый `switch` в **двухпроходную схему**:

**Проход 1 — диспетчеризация по хеш-коду**

Компилятор вычисляет `String.hashCode()` для каждой строки-метки ещё на этапе компиляции. В байткоде генерируется вызов `hashCode()` на входной строке, после чего применяется инструкция `lookupswitch` (или `tableswitch` — в зависимости от плотности значений хешей). Каждому хешу соответствует переход к блоку проверки равенства.

**Проход 2 — точное сравнение через equals**

Внутри каждого хеш-блока генерируется вызов `String.equals()` для подтверждения точного совпадения (так как хеш-коллизии возможны — например, строки `"FB"` и `"Ea"` имеют одинаковый хеш-код). При совпадении переменной-индексу присваивается целочисленное значение (0, 1, 2, ...).

**Финальный switch по индексу**

Вторая инструкция `tableswitch` переключается по этому целочисленному индексу и переходит к нужной ветви кода.

**Схематично:**
```
// Исходный код
switch (s) {
    case "foo": ...; break;
    case "bar": ...; break;
}

// Псевдокод, генерируемый компилятором
int idx = -1;
switch (s.hashCode()) {           // lookupswitch по хешам
    case 101574:                  // hashCode("foo")
        if (s.equals("foo")) idx = 0;
        break;
    case 97299:                   // hashCode("bar")
        if (s.equals("bar")) idx = 1;
        break;
}
switch (idx) {                    // tableswitch по индексу
    case 0: /* код для "foo" */ break;
    case 1: /* код для "bar" */ break;
}
```

**Обработка хеш-коллизий**

Если несколько строк-меток имеют одинаковый хеш-код, компилятор последовательно генерирует несколько вызовов `equals()` в одном хеш-блоке, присваивая каждой совпавшей строке свой уникальный индекс.

**Выбор между tableswitch и lookupswitch**

- `tableswitch` используется, когда значения хешей образуют достаточно плотный диапазон (Jump Table — O(1) по времени).
- `lookupswitch` используется при разреженных значениях хешей (бинарный поиск по отсортированной таблице — O(log n)).

Компилятор выбирает инструкцию, оптимальную по соотношению скорость/размер байткода.

**Итоговые вызовы методов в байткоде**

При компиляции строкового `switch` компилятор всегда генерирует:
1. `invokevirtual java/lang/String.hashCode:()I` — для хеш-диспетчеризации
2. `invokevirtual java/lang/String.equals:(Ljava/lang/Object;)Z` — для точного сравнения (один или несколько раз при коллизиях)

### Вопрос 46. Что произойдёт при целочисленном переполнении в Java? Выбрасывается ли исключение? Что напечатает `System.out.println(Integer.MAX_VALUE + 1);`? Как обнаружить переполнение до его возникновения?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Java Integer Overflow and Underflow — javathinking.com](https://www.javathinking.com/blog/how-does-java-handle-integer-underflows-and-overflows-and-how-would-you-check-for-it/)

Java использует **двоичное дополнение (two's complement)** для представления целых чисел, что приводит к молчаливому обороту значений при выходе за границу диапазона типа.

**Java НЕ выбрасывает исключение при переполнении.** Значение просто оборачивается.

```java
int maxInt = Integer.MAX_VALUE; // 2 147 483 647
int overflowResult = maxInt + 1;
System.out.println(overflowResult); // Вывод: -2147483648
```

Результат равен `Integer.MIN_VALUE`, потому что в двоичном представлении с дополнением до двух значение циклически переходит из максимального в минимальное.

**Способы обнаружения переполнения:**

1. **`Math.addExact()` (Java 8+)** — рекомендуемый подход, выбрасывает `ArithmeticException`:
```java
try {
    int safeSum = Math.addExact(Integer.MAX_VALUE, 1);
} catch (ArithmeticException e) {
    System.out.println("Переполнение обнаружено");
}
```

2. **Предварительная проверка значений:**
```java
if (a > 0 && b > Integer.MAX_VALUE - a) {
    throw new ArithmeticException("Переполнение при сложении");
}
return a + b;
```

3. **Использование `BigInteger`** для полного исключения переполнения с последующей проверкой результата.

#### Ответ от Claude

**Что происходит при переполнении**

Java использует **32-битное знаковое представление по методу двоичного дополнения (two's complement)** для типа `int`. Диапазон значений: от `-2^31` (`-2 147 483 648`) до `2^31 - 1` (`2 147 483 647`).

При выходе за границу диапазона **исключение НЕ выбрасывается**. JVM просто обрезает результат до 32 бит, что приводит к «обороту» значения. Это поведение гарантировано спецификацией языка (JLS §15.18.2) и полностью предсказуемо.

**Что напечатает код**

```java
System.out.println(Integer.MAX_VALUE + 1);
// Вывод: -2147483648
```

Объяснение: `Integer.MAX_VALUE` в двоичном виде — это `0111 1111 1111 1111 1111 1111 1111 1111`. Добавление 1 переводит старший бит в 1, что в two's complement означает отрицательное число: `1000 0000 0000 0000 0000 0000 0000 0000` = `Integer.MIN_VALUE` = `-2 147 483 648`.

**Почему нет исключения**

Это сознательное архитектурное решение: проверка переполнения на каждую целочисленную операцию существенно замедлила бы выполнение программ (несовместимость с аппаратными инструкциями CPU), нарушила бы совместимость с C/C++, а большинству программ такая проверка не нужна.

**Как обнаружить переполнение**

**Способ 1: `Math.addExact()` и аналоги (Java 8+, рекомендуется)**

Начиная с Java 8 в классе `Math` появились «точные» арифметические методы, выбрасывающие `ArithmeticException` при переполнении:

```java
// Сложение
Math.addExact(int x, int y)
Math.addExact(long x, long y)

// Вычитание
Math.subtractExact(int x, int y)

// Умножение
Math.multiplyExact(int x, int y)

// Инкремент/декремент
Math.incrementExact(int a)
Math.decrementExact(int a)

// Отрицание
Math.negateExact(int a)

// Конвертация long -> int с проверкой
Math.toIntExact(long value)
```

Пример:
```java
try {
    int result = Math.addExact(Integer.MAX_VALUE, 1);
} catch (ArithmeticException e) {
    // переполнение обнаружено до возникновения
}
```

**Способ 2: Предварительная проверка условий**

```java
// Проверка до сложения двух положительных int
if (b > 0 && a > Integer.MAX_VALUE - b) {
    throw new ArithmeticException("Overflow");
}
int result = a + b;
```

**Способ 3: Использование `long` как промежуточного типа**

```java
long result = (long) a + (long) b;
if (result > Integer.MAX_VALUE || result < Integer.MIN_VALUE) {
    throw new ArithmeticException("Overflow");
}
return (int) result;
```

**Способ 4: `BigInteger` (наиболее безопасный, но медленный)**

```java
BigInteger big = BigInteger.valueOf(a).add(BigInteger.valueOf(b));
if (big.bitLength() > 31) {
    throw new ArithmeticException("Overflow");
}
return big.intValue();
```

**Расхождение источников:** Некоторые источники (Medium, Baeldung) упоминают метод `Math.exactAdd()` — это ошибка. Правильное название: `Math.addExact()`. Метод `Math.toIntExact(long)` также появился в Java 8 и часто упускается в обзорных статьях, хотя является удобным инструментом при конвертации `long` → `int`.

**Вывод для практики:** В большинстве бизнес-приложений достаточно `Math.addExact()`. Для финансовых и криптографических систем предпочтителен `BigInteger` или `long`-арифметика с явными проверками.

### Вопрос 47. Чем `>>` (signed right shift) отличается от `>>>` (unsigned right shift)? Приведите пример, когда результат этих операторов будет различаться для отрицательного числа.

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Bitwise and Bit Shift Operators — The Java Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/java/nutsandbolts/op3.html)

Оператор `>>` — знаковый сдвиг вправо. После сдвига левая (старшая) позиция заполняется в зависимости от знакового расширения: если число отрицательное, сдвинутые позиции заполняются единицами, если положительное — нулями. Таким образом, знак числа сохраняется.

Оператор `>>>` — беззнаковый сдвиг вправо. Всегда помещает 0 в крайнюю левую позицию, независимо от знака исходного числа. В результате отрицательное число становится положительным.

Пример:

```java
int negative = -1; // все 32 бита равны 1: 11111111111111111111111111111111

int a = negative >> 1;   // результат: -1
                         // знаковое расширение: 11111111111111111111111111111111

int b = negative >>> 1;  // результат: 2147483647 (Integer.MAX_VALUE)
                         // заполнение нулём слева: 01111111111111111111111111111111
```

Ещё один наглядный пример с числом -12:

```java
int x = -12; // двоичное (дополнение до двух): 11111111111111111111111111110100

int c = x >> 1;   // результат: -6
                  // 11111111111111111111111111111010

int d = x >>> 1;  // результат: 2147483642
                  // 01111111111111111111111111111010
```

#### Ответ от Claude

**Механизм работы операторов**

Оба оператора сдвигают все биты числа на заданное количество позиций вправо, при этом младшие биты, вышедшие за пределы, отбрасываются. Разница — в том, чем заполняется освободившееся место с левой (старшей) стороны:

| Оператор | Заполнение старшего бита | Знак числа после сдвига |
|----------|--------------------------|------------------------|
| `>>` | Копией знакового бита (sign extension) | Сохраняется |
| `>>>` | Всегда нулём | Всегда неотрицательный |

Для положительных чисел оба оператора дают одинаковый результат, поскольку знаковый бит равен 0.

**Подробный пример с -1**

```java
int n = -1;
// Двоичное представление (32 бита, дополнение до двух):
// 11111111 11111111 11111111 11111111

System.out.println(n >> 1);   // -1
// 11111111 11111111 11111111 11111111  (MSB скопирован)

System.out.println(n >>> 1);  // 2147483647
// 01111111 11111111 11111111 11111111  (MSB = 0)
```

**Подробный пример с -12**

```java
int n = -12;
// Двоичное (32 бита): 11111111 11111111 11111111 11110100

System.out.println(n >> 2);   // -3
// 11111111 11111111 11111111 11111101

System.out.println(n >>> 2);  // 1073741821
// 00111111 11111111 11111111 11111101
```

**Когда важно это различие**

- `>>` используется в арифметических вычислениях, когда нужно сохранить знак (например, деление отрицательного числа на степень двойки).
- `>>>` используется при работе с битовыми масками, хеш-функциями и протоколами, где число трактуется как беззнаковая последовательность бит. Классический пример — реализация `HashMap.hash()` в JDK: `(h = key.hashCode()) ^ (h >>> 16)`.

Источники согласуются: [документация Oracle](https://docs.oracle.com/javase/tutorial/java/nutsandbolts/op3.html) и [JavaRevisited](https://javarevisited.blogspot.com/2015/02/difference-between-right-shift-and.html) дают идентичное описание поведения.

### Вопрос 48. Каковы правила модификаторов доступа в Java (private, package-private, protected, public)? Что означает package-private по умолчанию и чем protected отличается от package-private при наследовании?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Controlling Access to Members of a Class — The Java Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/java/javaOO/accesscontrol.html)

В Java существуют четыре уровня доступа, управляемые тремя модификаторами:

| Модификатор | Класс | Пакет | Подкласс в другом пакете | Весь мир |
|-------------|-------|-------|--------------------------|----------|
| `public` | да | да | да | да |
| `protected` | да | да | да | нет |
| *(нет модификатора)* | да | да | нет | нет |
| `private` | да | нет | нет | нет |

**private** — самый ограничивающий уровень. Доступ открыт только внутри того же класса, в котором объявлен член. Подклассы не могут напрямую обращаться к приватным членам суперкласса.

**Package-private (без модификатора)** — уровень по умолчанию, когда модификатор явно не указан. Доступ открыт для любого кода внутри того же пакета. Подкласс, расположенный в другом пакете, не имеет доступа к таким членам.

**protected** — открывает доступ внутри пакета (как package-private), плюс дополнительно разрешает доступ подклассам из других пакетов.

**public** — доступен везде без ограничений.

Ключевое отличие protected от package-private: если подкласс находится в другом пакете, он может обращаться к `protected`-членам суперкласса, но не может обращаться к package-private-членам.

#### Ответ от Claude

**Четыре уровня доступа**

Java предоставляет четыре уровня видимости. Ниже — от наиболее ограниченного к наименее ограниченному:

**1. `private`**
Самый строгий уровень. Член доступен только внутри объявляющего класса. Даже подкласс не может обратиться к приватному полю родителя напрямую — только через унаследованные публичные/protected-методы (геттеры и т.п.).

**2. Package-private (без модификатора)**
Уровень по умолчанию — модификатор не пишется вообще. Член видим любому коду в том же пакете. Подкласс из другого пакета теряет доступ, как будто это `private` для него.

```java
// пакет com.example.a
class Parent {
    int packageField = 42; // package-private
}

// пакет com.example.b
class Child extends Parent {
    void test() {
        System.out.println(packageField); // ОШИБКА компиляции
    }
}
```

**3. `protected`**
Расширяет package-private: добавляет доступ для подклассов из любых пакетов. Но важный нюанс: доступ через наследование ограничен — подкласс может обращаться к `protected`-члену только через ссылку на свой собственный тип (или подтип), но не через ссылку на суперкласс.

```java
// пакет com.example.a
public class Parent {
    protected int protectedField = 42;
}

// пакет com.example.b
public class Child extends Parent {
    void test() {
        System.out.println(protectedField);     // OK — через this
        Child c = new Child();
        System.out.println(c.protectedField);   // OK — через ссылку на Child
        Parent p = new Parent();
        System.out.println(p.protectedField);   // ОШИБКА — ссылка на Parent из другого пакета
    }
}
```

**4. `public`**
Открыт для любого кода в любом пакете. Именно такой уровень нужен для формирования публичного API класса.

**Сводная таблица**

| Модификатор | Тот же класс | Тот же пакет | Подкласс в другом пакете | Все остальные |
|-------------|:---:|:---:|:---:|:---:|
| `private` | + | — | — | — |
| *(нет)* | + | + | — | — |
| `protected` | + | + | + | — |
| `public` | + | + | + | + |

**Главное отличие protected от package-private**

Package-private предоставляет «горизонтальный» доступ: в пределах одного пакета. Protected добавляет «вертикальный» доступ: по иерархии наследования, пересекая границы пакетов. Это позволяет проектировать расширяемые классы с защищёнными хуками для подклассов, не раскрывая детали реализации всем пользователям библиотеки.

Оба источника — [Oracle Tutorials](https://docs.oracle.com/javase/tutorial/java/javaOO/accesscontrol.html) и [Jenkov.com](https://jenkov.com/tutorials/java/access-modifiers.html) — дают идентичное описание.

### Вопрос 49. Что такое hiding (скрытие) метода и чем оно отличается от overriding (переопределения)? Можно ли «переопределить» статический метод в подклассе? Что выведет код: `Animal a = new Dog(); a.staticMethod();` если в обоих классах есть static-метод с одинаковой сигнатурой?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Overriding and Hiding Methods — The Java Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/java/IandI/override.html)

Официальная документация Oracle чётко разграничивает два случая:

Если подкласс определяет **instance-метод** с той же сигнатурой, что и в суперклассе, говорят, что подкласс **переопределяет** (overrides) этот метод.

Если подкласс определяет **static-метод** с той же сигнатурой, что и в суперклассе, говорят, что подкласс **скрывает** (hides) этот метод.

> "The version of the overridden instance method that gets invoked is the one in the subclass. The version of the hidden static method that gets invoked depends on whether it is invoked from the superclass or the subclass."

Пример из документации:

```java
public class Animal {
    public static void testClassMethod() {
        System.out.println("The static method in Animal");
    }
    public void testInstanceMethod() {
        System.out.println("The instance method in Animal");
    }
}

public class Cat extends Animal {
    public static void testClassMethod() {
        System.out.println("The static method in Cat");
    }
    public void testInstanceMethod() {
        System.out.println("The instance method in Cat");
    }

    public static void main(String[] args) {
        Cat myCat = new Cat();
        Animal myAnimal = myCat;
        Animal.testClassMethod();       // The static method in Animal
        myAnimal.testInstanceMethod();  // The instance method in Cat
    }
}
```

Вывод:
```
The static method in Animal
The instance method in Cat
```

Статический метод вызван через ссылку типа `Animal`, поэтому выполняется метод `Animal`. Instance-метод вызван на объекте `Cat`, поэтому выполняется метод `Cat` (полиморфизм).

#### Ответ от Claude

**Overriding (переопределение) — instance-методы**

Когда подкласс объявляет нестатический метод с той же сигнатурой и совместимым возвращаемым типом, что и в суперклассе, происходит переопределение. Ключевые свойства:

- Разрешение вызова происходит **в рантайме** на основе фактического типа объекта (dynamic dispatch, late binding).
- Реализуется через таблицу виртуальных методов (vtable).
- Аннотация `@Override` позволяет компилятору проверить корректность переопределения.
- Метод подкласса не может быть менее доступным, чем метод суперкласса.

**Hiding (скрытие) — static-методы**

Когда подкласс объявляет статический метод с той же сигнатурой, что и в суперклассе, происходит скрытие. Ключевые свойства:

- Разрешение вызова происходит **на этапе компиляции** на основе объявленного (статического) типа ссылки (early binding).
- Полиморфизм не работает — вызывается метод того класса, чей тип у ссылки.
- Технически «переопределить» статический метод невозможно: JVM не использует vtable для статических методов.

**Что выведет `Animal a = new Dog(); a.staticMethod();`?**

```java
class Animal {
    public static void staticMethod() {
        System.out.println("Animal.staticMethod");
    }
}

class Dog extends Animal {
    public static void staticMethod() {
        System.out.println("Dog.staticMethod");
    }
}

public class Main {
    public static void main(String[] args) {
        Animal a = new Dog();
        a.staticMethod(); // выведет: Animal.staticMethod
    }
}
```

Вывод: `Animal.staticMethod`

Несмотря на то что в рантайме объект имеет тип `Dog`, компилятор разрешает вызов статического метода по **объявленному типу ссылки** — `Animal`. Метод `Dog.staticMethod()` скрывает метод суперкласса, но не заменяет его полиморфно.

**Сводная таблица различий**

| Характеристика | Overriding | Hiding |
|----------------|------------|--------|
| Применяется к | Instance-методам | Static-методам |
| Разрешение вызова | Runtime (тип объекта) | Compile time (тип ссылки) |
| Полиморфизм | Да | Нет |
| `@Override` | Применима | Компилятор предупредит, если метод static |
| Можно ли вызвать скрытый метод | Через `super.method()` | Через `SuperClass.method()` |

**Практический вывод**

Вызов статических методов через ссылку на объект (`a.staticMethod()`) — плохая практика именно из-за неочевидного поведения. Статические методы следует вызывать через имя класса (`Animal.staticMethod()`, `Dog.staticMethod()`), чтобы намерение было явным.

Оба источника — [Oracle Tutorials](https://docs.oracle.com/javase/tutorial/java/IandI/override.html) и материалы по [скрытию методов](https://www.geeksforgeeks.org/java/method-hiding-in-java/) — согласуются в описании поведения.

### Вопрос 50. Что такое Compact Strings, введённые в Java 9 (JEP 254)? Как изменилось внутреннее хранилище класса String — с `char[]` на `byte[]` — и что это даёт по памяти для строк, содержащих только Latin-1 символы?

**Категория:** Core Java — основы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Java 9 - Compact Strings Improvement [JEP 254] — HowToDoInJava](https://howtodoinjava.com/java9/compact-strings/)

До Java 9 класс `String` хранил символы в массиве `char[]`, где каждый символ занимал ровно 2 байта (UTF-16). Oracle проанализировала heap-дампы реальных приложений и выяснила, что большинство строк содержат только символы Latin-1 (коды 0–255), которым достаточно одного байта.

JEP 254 изменил внутреннее представление класса `String`:

**До Java 9:**
```java
private final char[] value; // 2 байта на символ
```

**Начиная с Java 9:**
```java
private final byte[] value; // 1 байт для Latin-1, 2 байта для UTF-16
private final byte coder;   // флаг кодировки: LATIN1 (0) или UTF16 (1)
```

Поле `coder` хранит информацию о кодировке, применённой при создании строки. JVM автоматически выбирает оптимальную кодировку: если все символы строки укладываются в Latin-1 (Unicode-значение меньше `Ā`), каждый символ хранится в одном байте, экономя 50% памяти. Если хотя бы один символ выходит за пределы Latin-1, вся строка хранится в UTF-16 (2 байта на символ).

Аналогичные изменения внесены в `AbstractStringBuilder`, `StringBuilder` и `StringBuffer`.

Флаг `-XX:-CompactStrings` отключает оптимизацию и принудительно включает UTF-16 для всех строк — это может быть полезно при работе с приложениями, где преобладают многобайтовые символы.

#### Ответ от Claude

JEP 254 (реализован в Java 9) — чисто внутренняя оптимизация класса `String`: публичный API не изменился, обратная совместимость полностью сохранена.

**Суть изменения.** До Java 9 `String` всегда использовал `char[] value`, где один символ занимал 2 байта (кодировка UTF-16). Начиная с Java 9 поле сменилось на `byte[] value` плюс однобайтовый флаг `byte coder`:

```java
// Внутренние константы класса String (Java 9+)
static final byte LATIN1 = 0;
static final byte UTF16  = 1;
```

При создании строки JVM проверяет все символы:
- Если каждый символ вписывается в Latin-1 (код –ÿ), строка кодируется как Latin-1 — 1 байт на символ, `coder = LATIN1`.
- Если хотя бы один символ выходит за пределы Latin-1 (кириллица, иероглифы, эмодзи и т.д.), строка кодируется как UTF-16 — 2 байта на символ, `coder = UTF16`.

**Выигрыш по памяти.** Для типичных англоязычных строк (ASCII-подмножество Latin-1) экономия составляет ровно 50%: строка из 10 символов занимала 20 байт в `char[]`, теперь занимает 10 байт в `byte[]`. Анализ Google и Oracle показал, что в большинстве production-приложений на JVM от 40 до 70% строк состоят исключительно из Latin-1 символов. Меньшее потребление heap означает сниженную нагрузку на GC.

**Область действия.** Изменены `String`, `AbstractStringBuilder`, `StringBuilder`, `StringBuffer` и внутренние интринсики HotSpot.

**Управление.** Флаг `-XX:-CompactStrings` (по умолчанию `true`) позволяет отключить оптимизацию. При отключении все строки хранятся в UTF-16 — поведение идентично Java 8.

**Расхождений между источниками не обнаружено:** JEP 254 и документация HowToDoInJava описывают механизм одинаково.

### Вопрос 51. Что такое String Deduplication в G1 GC (`-XX:+UseStringDeduplication`)? Чем этот механизм отличается от `String.intern()`? Какой из них применяется к живым объектам в куче, а какой — к пулу строк?

**Категория:** Core Java — основы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [String Deduplication of G1 Garbage Collector in Java 8 — Medium (Saurav Omar)](https://sauravomar01.medium.com/string-deduplication-of-g1-garbage-collector-in-java-8-cb8914a139f4)

String Deduplication — это механизм G1 GC, который автоматически обнаруживает дублирующиеся строки в куче и заставляет их совместно использовать один и тот же внутренний массив символов. Включается флагом `-XX:+UseStringDeduplication` (доступно с Java 8 Update 20).

GC посещает живые `String`-объекты, сохраняет хеш-значение рядом со слабой ссылкой на внутренний массив. Когда обнаруживаются два объекта с одинаковым хешем, система сравнивает их содержимое символ за символом. При совпадении один из объектов перенаправляется на уже существующий массив, и оригинальный массив освобождается сборщиком мусора.

**Ключевые отличия от `String.intern()`:**
- `String.intern()` — нативный метод с высокими накладными расходами; он использует хеш-таблицу фиксированного размера (по умолчанию 1009 бакетов), что при большом количестве интернированных строк деградирует до O(n).
- String Deduplication работает автоматически в фоне во время сборки мусора, без изменения кода приложения.
- Дедупликация обрабатывает живые объекты в куче; `intern()` помещает строку в пул строк (String Pool / StringTable).

#### Ответ от Claude

**JEP 192 (String Deduplication in G1)** — механизм, направленный на снижение потребления heap за счёт автоматического совместного использования внутреннего байтового (до Java 9 — символьного) массива для строк с одинаковым содержимым.

**Как работает дедупликация:**

1. Во время GC-паузы (young collection в G1) JVM выявляет строки-кандидаты на дедупликацию и помещает слабые ссылки на них в очередь.
2. Фоновый поток дедупликации (VM internal thread) обрабатывает очередь: вычисляет хеш строки, ищет совпадение в хеш-таблице дедупликации.
3. При совпадении поле `value` одного `String`-объекта перенаправляется на `byte[]` (или `char[]` в Java 8) другого объекта. Оригинальный массив становится недостижимым и впоследствии собирается GC.
4. Сами `String`-объекты остаются разными — идентичность объектов (`==`) не меняется. Дедуплицируется только внутренний массив.

**String Deduplication vs `String.intern()`:**

| Характеристика | `UseStringDeduplication` | `String.intern()` |
|---|---|---|
| Объект воздействия | Живые `String`-объекты в куче | String Pool (StringTable) |
| Инициатор | JVM автоматически, фоново | Явный вызов из кода |
| Изменяет идентичность объекта? | Нет | Да (возвращает каноническую ссылку из пула) |
| Память | Освобождает дублирующиеся `byte[]` | Помещает объект в пул, предотвращает создание дубликатов |
| Накладные расходы | Низкие (фоновый поток) | Высокие (нативный метод, хеш-таблица) |
| Применимость | Только G1 (с Java 8u20), ZGC и другие с Java 18+ | Любой GC и версия Java |

**Взаимодействие механизмов.** Согласно JEP 192: когда строка интернируется и добавляется в `StringTable`, она сразу дедуплицируется, и её `byte[]` добавляется в таблицу дедупликации. После этого строка не дедуплицируется повторно, чтобы не мешать оптимизациям компилятора для строковых литералов.

**Краткий ответ:** `UseStringDeduplication` работает с живыми объектами в куче, дедуплицируя их внутренний массив. `String.intern()` работает со String Pool — возвращает каноническую ссылку из пула, предотвращая появление дубликатов на уровне объектов.

### Вопрос 52. Почему нарушение транзитивности в `equals()` особенно сложно поймать при наследовании? Опишите классическую проблему «ColorPoint extends Point» и объясните, почему Джошуа Блох рекомендует использовать композицию вместо наследования при добавлении полей, участвующих в equals.

**Категория:** Core Java — основы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Overriding .equals method for inheriting classes in Java — Medium (Hussein Maziad)](https://hussein-maziad.medium.com/overriding-equals-method-for-inheriting-classes-in-java-fd89e35810fd)

Проблема возникает, когда подкласс добавляет новое поле, влияющее на логику равенства. Классический пример — `ColorPoint extends Point`.

**Шаг 1: Попытка — нарушение симметрии.**

Если `ColorPoint.equals()` проверяет тип через `instanceof ColorPoint`, то сравнение `point.equals(colorPoint)` даст `true` (Point игнорирует цвет), а `colorPoint.equals(point)` — `false` (ColorPoint требует тип ColorPoint). Нарушена симметрия.

**Шаг 2: Попытка — нарушение транзитивности.**

Исправление через «color-blind» подход: если аргумент — `Point`, а не `ColorPoint`, делегируем `o.equals(this)`:

```java
@Override
public boolean equals(Object o) {
    if (!(o instanceof Point)) return false;
    if (!(o instanceof ColorPoint)) return o.equals(this);
    return super.equals(o) && ((ColorPoint) o).color.equals(color);
}
```

Теперь:
```
Point p = new Point(1, 1);
ColorPoint red  = new ColorPoint(1, 1, "RED");
ColorPoint blue = new ColorPoint(1, 1, "BLUE");

p.equals(red)   // true
p.equals(blue)  // true
red.equals(blue) // false — нарушение транзитивности!
```

**Решение — композиция:**

```java
public class ColorPoint {
    private final String color;
    private final Point point;

    public ColorPoint(int x, int y, String color) {
        this.point = new Point(x, y);
        this.color = color;
    }

    public Point asPoint() { return point; }

    @Override
    public boolean equals(Object o) {
        if (!(o instanceof ColorPoint)) return false;
        ColorPoint cp = (ColorPoint) o;
        return cp.point.equals(point) && cp.color.equals(color);
    }
}
```

`ColorPoint` больше не является подтипом `Point`, поэтому смешанное сравнение `Point` и `ColorPoint` невозможно. Контракт `equals()` соблюдается в полной мере.

#### Ответ от Claude

**Почему это сложно обнаружить.** Нарушение транзитивности при наследовании коварно по нескольким причинам:

1. Каждый отдельный тест (`a.equals(b)`, `b.equals(c)`) возвращает ожидаемый результат. Проблема проявляется только при трёхстороннем сравнении, которое редко встречается в юнит-тестах.
2. Нарушение не бросает исключений — оно тихо приводит к некорректной логике в коллекциях: `HashSet`, `TreeSet`, `HashMap` могут дать неверные результаты при поиске и вставке.
3. Нарушение симметрии и транзитивности зависит от порядка вызова (`a.equals(b)` vs `b.equals(a)`), что делает баг недетерминированным в зависимости от порядка элементов в коллекции.

**Два сломанных подхода при наследовании:**

*Вариант A — проверка через `instanceof ColorPoint`:*
```java
// В ColorPoint
@Override
public boolean equals(Object o) {
    if (!(o instanceof ColorPoint)) return false;
    return super.equals(o) && ((ColorPoint) o).color.equals(color);
}
// point.equals(colorPoint) == true  (Point принимает любой Point-совместимый)
// colorPoint.equals(point) == false (ColorPoint отвергает не-ColorPoint)
// --> нарушена СИММЕТРИЯ
```

*Вариант B — «color-blind» (делегирование к Point):*
```java
@Override
public boolean equals(Object o) {
    if (!(o instanceof Point)) return false;
    if (!(o instanceof ColorPoint)) return o.equals(this);
    return super.equals(o) && ((ColorPoint) o).color.equals(color);
}
// p.equals(red) == true, p.equals(blue) == true, но red.equals(blue) == false
// --> нарушена ТРАНЗИТИВНОСТЬ
```

**Фундаментальный предел.** Джошуа Блох в «Effective Java» (Item 10) формулирует теорему: *невозможно расширить инстанциируемый класс, добавив новое значимое поле, при этом сохранив контракт `equals()`* — если только нарушением симметрии или транзитивности не пренебречь. Это не вопрос качества реализации — это математическое ограничение.

**Почему композиция решает проблему.** При композиции `ColorPoint` не является подтипом `Point`:

```java
public class ColorPoint {
    private final Point point; // «has-a» вместо «is-a»
    private final Color color;

    public Point asPoint() { return point; } // view-метод

    @Override
    public boolean equals(Object o) {
        if (!(o instanceof ColorPoint)) return false;
        ColorPoint cp = (ColorPoint) o;
        return point.equals(cp.point) && color.equals(cp.color);
    }
}
```

Поскольку `ColorPoint` не является `Point`, смешанное сравнение `point.equals(colorPoint)` просто невозможно — разнотипные объекты не сравниваются. Все три свойства (`рефлексивность`, `симметрия`, `транзитивность`) выполняются тривиально. Метод `asPoint()` даёт возможность получить геометрическую проекцию при необходимости.

**Расхождений между источниками не обнаружено:** описание проблемы у Hussein Maziad, Elizaveta Bagaeva и в классической книге Джошуа Блох «Effective Java» (3-е изд.) совпадает. Единственное расхождение — некоторые авторы предлагают альтернативу через `canEqual()` (паттерн Scala-стиля), которую Блох не рассматривает; однако все сходятся, что композиция — наиболее чистое решение для Java.
### Вопрос 53. Что такое «определённое присвоение» (definite assignment) в Java? Как компилятор гарантирует, что локальная переменная будет инициализирована до использования? Распространяется ли это правило на поля класса?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Chapter 16. Definite Assignment — Java Language Specification SE 21](https://docs.oracle.com/javase/specs/jls/se21/html/jls-16.html)

Каждая локальная переменная и каждое «пустое» (blank) `final`-поле должны иметь определённо присвоенное значение в момент любого обращения к этому значению. Если доступ к локальной переменной или пустому `final`-полю `x` происходит до того, как `x` определённо присвоена, возникает ошибка компиляции.

Идея определённого присвоения состоит в том, что присвоение значения локальной переменной или пустому `final`-полю должно произойти на каждом возможном пути выполнения до точки обращения к переменной.

Анализ учитывает структуру операторов и выражений; операторы `!`, `&&`, `||` и `? :`, а также булевы константные выражения обрабатываются специальным образом. За исключением этих случаев значения выражений в потоковом анализе **не учитываются**.

Пример из спецификации: даже если условие `n > 2` при `n = 5` заведомо истинно, компилятор не отслеживает конкретные значения переменных во время статического анализа:

```java
{
    int k;
    int n = 5;
    if (n > 2)
        k = 3;
    System.out.println(k); // Ошибка компиляции: k может быть не инициализирована
}
```

Обращение к значению переменной — это использование простого имени переменной (или для поля — простого имени поля, квалифицированного `this`) в любом месте выражения, кроме левой части оператора простого присваивания `=`.

Для пустых `final`-переменных действует дополнительное правило «определённого неприсвоения»: каждая пустая `final`-переменная должна быть присвоена не более одного раза; в момент присваивания она должна быть определённо **не** присвоена.

#### Ответ от Claude

**Определённое присвоение (definite assignment)** — это статический анализ потока управления, который компилятор Java выполняет на этапе компиляции. Цель анализа: гарантировать, что локальная переменная получит значение на **каждом возможном пути выполнения** до того, как это значение будет прочитано.

**На что распространяется правило (JLS §16):**

| Тип переменной | Definite assignment |
|---|---|
| Локальная переменная | Да — обязательно. Без инициализации → ошибка компиляции |
| Пустое `final`-поле (blank final field) | Да — обязательно. Должно быть присвоено ровно один раз |
| Обычное поле экземпляра (instance field) | Нет — автоматически получает значение по умолчанию |
| Статическое поле (static field) | Нет — автоматически получает значение по умолчанию |

**Почему поля класса не требуют definite assignment:**
Согласно JLS §4.12.5, каждая переменная класса (static), переменная экземпляра и элемент массива инициализируются значением по умолчанию в момент создания (`0`, `0.0`, `false`, `''`, `null` в зависимости от типа). Таким образом, для полей класса «необъявленное значение» физически невозможно: значение по умолчанию всегда уже существует. Для локальных переменных JVM такой автоматической инициализации **не** производит — это сделано намеренно, чтобы не скрывать программные ошибки.

**Как компилятор выполняет анализ:**

Анализ является структурным (flow-based), а не value-based: компилятор не вычисляет конкретные значения выражений — он отслеживает только граф потока управления. Исключения: операторы `&&`, `||`, `? :` и `!`, а также булевы константные выражения, которые получают особую трактовку (компилятор понимает, что правый операнд `&&` выполняется только при истинности левого и т.п.).

```java
// Ошибка: компилятор не «знает», что n > 2 всегда истинно при n = 5
int k;
int n = 5;
if (n > 2) k = 3;
System.out.println(k); // compile error

// OK: компилятор видит, что оба ветки присваивают значение
int m;
if (flag) m = 1; else m = 2;
System.out.println(m); // OK

// OK: специальная обработка &&
int x;
if (flag && (x = compute()) > 0)
    System.out.println(x); // OK: x определённо присвоена, если условие истинно
```

**Пустые final-поля (blank final fields):**
Для них работают оба правила одновременно:
- **Definite assignment**: должны быть присвоены на каждом пути (обычно в конструкторе).
- **Definite unassignment**: должны быть присвоены **не более одного раза**.

```java
class Example {
    final int value;

    Example(boolean flag) {
        if (flag) value = 1;
        else      value = 2;
        // OK: оба пути присваивают ровно один раз
    }
}
```

**Переменные, объявленные в образцах (pattern variables):** с Java 16+ переменные, объявляемые в паттернах (`instanceof Pattern p`), **не подпадают** под правила definite assignment — они всегда инициализируются механизмом сопоставления с образцом.

**Источники:**
- JLS SE 21, Chapter 16 «Definite Assignment»: https://docs.oracle.com/javase/specs/jls/se21/html/jls-16.html
- JLS SE 21, §4.12.5 «Initial Values of Variables»: https://docs.oracle.com/javase/specs/jls/se21/html/jls-4.html

### Вопрос 54. В каком порядке инициализируются поля и блоки инициализации при создании нескольких объектов одного класса? Что выведет код, в котором у класса есть static-блок, блок инициализации экземпляра и конструктор, если создать два объекта подряд?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Initializing Fields — The Java Tutorials (Oracle)](https://docs.oracle.com/javase/tutorial/java/javaOO/initial.html)

Java позволяет инициализировать поля класса непосредственно в объявлении, а также использовать для этого статические блоки инициализации и блоки инициализации экземпляра.

**Статический блок инициализации** выполняется один раз при загрузке класса:
```java
static {
    // код инициализации статических полей
}
```
Если в классе несколько статических блоков, они вызываются в порядке объявления в исходном тексте.

**Блок инициализации экземпляра** (без ключевого слова `static`) копируется компилятором в каждый конструктор. Это позволяет использовать один блок кода для нескольких конструкторов:
```java
{
    // код инициализации полей экземпляра
}
```

Таким образом, при каждом создании нового объекта сначала выполняются блоки инициализации экземпляра, затем — тело конструктора. Статический блок при повторном создании объектов повторно **не** выполняется.

#### Ответ от Claude

**Полный порядок инициализации** определяется спецификацией Java (JLS §12.4 и §12.5) и разбивается на два независимых этапа.

**Этап 1 — Инициализация класса (выполняется один раз при первом обращении к классу, JLS §12.4):**

1. Инициализируется суперкласс (рекурсивно).
2. Присваиваются значения константным статическим полям (`static final` с compile-time значением).
3. Выполняются статические инициализаторы полей и `static`-блоки **в порядке объявления в исходном тексте** (как единый блок).

**Этап 2 — Создание каждого экземпляра (`new`, JLS §12.5):**

1. Выделяется память; все поля экземпляра получают значения по умолчанию (`0`, `null`, `false`…).
2. Рекурсивно вызывается конструктор суперкласса (явный `super(...)` или неявный).
3. Выполняются инициализаторы полей экземпляра и блоки инициализации экземпляра **в порядке объявления в исходном тексте**.
4. Выполняется тело конструктора (код после вызова `super(...)`).

**Ключевые свойства:**

| Блок | Когда выполняется | Сколько раз |
|---|---|---|
| `static`-блок | При первой загрузке/инициализации класса | Один раз за время жизни класса в JVM |
| Блок инициализации экземпляра `{ }` | При каждом вызове `new` | По одному разу на каждый объект |
| Конструктор | При каждом вызове `new` | По одному разу на каждый объект |

**Пример — что выведет код при создании двух объектов:**

```java
class Demo {
    static int staticCounter = 0;

    static {
        System.out.println("static-блок, staticCounter=" + staticCounter);
    }

    {
        System.out.println("блок экземпляра #" + ++staticCounter);
    }

    Demo() {
        System.out.println("конструктор #" + staticCounter);
    }
}

class Main {
    public static void main(String[] args) {
        Demo d1 = new Demo();
        Demo d2 = new Demo();
    }
}
```

**Вывод:**
```
static-блок, staticCounter=0
блок экземпляра #1
конструктор #1
блок экземпляра #2
конструктор #2
```

**Объяснение:**
- При `new Demo()` первый раз класс `Demo` ещё не загружен → JVM инициализирует класс: сначала присваивается `staticCounter = 0`, затем выполняется `static`-блок.
- Затем (в рамках той же операции `new`) выполняется блок инициализации экземпляра, после чего — тело конструктора.
- При `new Demo()` второй раз класс уже инициализирован → `static`-блок **не** выполняется. Снова выполняются блок экземпляра и конструктор.

**Важное уточнение из JLS §12.5 (шаги 4 и 5):**
Компилятор копирует содержимое блока инициализации экземпляра в каждый конструктор **сразу после** вызова `super(...)`. Поэтому блок экземпляра всегда выполняется **раньше** тела конструктора, но **позже** конструктора суперкласса — вне зависимости от физического расположения блока в тексте класса.

**Пример с суперклассом (подтверждение порядка):**

```java
class Parent {
    static { System.out.println("Parent: static"); }
    { System.out.println("Parent: экземпляр"); }
    Parent() { System.out.println("Parent: конструктор"); }
}

class Child extends Parent {
    static { System.out.println("Child: static"); }
    { System.out.println("Child: экземпляр"); }
    Child() { System.out.println("Child: конструктор"); }
}

// new Child() выведет:
// Parent: static
// Child: static
// Parent: экземпляр
// Parent: конструктор
// Child: экземпляр
// Child: конструктор
```

**Источники:**
- JLS SE 21, §12.4 «Initialization of a Class or Interface»: https://docs.oracle.com/javase/specs/jls/se21/html/jls-12.html
- JLS SE 21, §12.5 «Creation of New Class Instances»: https://docs.oracle.com/javase/specs/jls/se21/html/jls-12.html
- Oracle Java Tutorials, «Initializing Fields»: https://docs.oracle.com/javase/tutorial/java/javaOO/initial.html

### Вопрос 55. Кэшируются ли объекты `Long` при автоупаковке так же, как `Integer`? Что выведет код `Long a = 127L, b = 127L, c = 128L, d = 128L; System.out.println(a == b); System.out.println(c == d);` и почему? Чем кэш `Long` отличается от кэша `Integer`?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [HowToDoInJava — Internal Caching in Wrapper Classes](https://howtodoinjava.com/java-examples/internal-cache-wrapper-classes/)

Классы-обёртки `Byte`, `Short`, `Integer` и `Long` кэшируют объекты для значений в диапазоне −128..127: их фабричные методы `valueOf()` возвращают объекты из внутреннего кэша, а не создают новые. Поэтому при автоупаковке двух одинаковых значений из этого диапазона `==` вернёт `true` (одна и та же ссылка), а для значений вне диапазона — `false` (разные объекты). У `Long` для этого есть внутренний класс `LongCache`.

#### Ответ от Claude

**Вывод программы:**
```
true
false
```

`Long a = 127L` — это автоупаковка, компилятор вставляет `Long.valueOf(127L)`. Метод `Long.valueOf` кэширует значения в диапазоне **−128..127** (внутренний класс `Long.LongCache`), возвращая для них один и тот же объект. Поэтому:

- `127L` входит в кэш → `a` и `b` ссылаются на один объект → `a == b` это `true`.
- `128L` вне кэша → создаются два разных объекта → `c == d` это `false`.

**Главное отличие от `Integer`:** верхнюю границу кэша `Integer` можно расширить флагом JVM `-XX:AutoBoxCacheMax=<N>`. Для `Long` это **невозможно** — диапазон жёстко зашит в коде (`if (l >= -128 && l <= 127)` в `Long.valueOf`), и `-XX:AutoBoxCacheMax` на `Long` не влияет.

**Практический вывод:** сравнивать значения объектов `Long` через `==` нельзя — это сравнение ссылок, работающее «случайно» лишь в диапазоне кэша. Используйте `equals()` или сравнивайте примитивы (`a.longValue() == b.longValue()`). Если в выражении `==` участвуют `Long` и `long`, происходит распаковка и сравниваются значения (а распаковка `null` бросит NPE). Тот же кэш −128..127 действует и для `Byte`, `Short`, `Character` (0..127), а `Boolean` кэширует оба значения.

Источники: [HowToDoInJava — Internal Caching in Wrapper Classes](https://howtodoinjava.com/java-examples/internal-cache-wrapper-classes/), [Understanding Java's Long and Integer Cache](https://blog.pikodat.com/posts/understanding-javas-long-and-integer-cache/).

### Вопрос 56. Чем абстрактный класс отличается от интерфейса? В каких случаях следует выбирать абстрактный класс, а в каких — интерфейс?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Abstract Methods and Classes — The Java Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/java/IandI/abstract.html)

Официальный туториал Oracle описывает различия так:

**Абстрактный класс:**
- Может содержать нестатические и не-`final` поля.
- Методы могут быть `public`, `protected`, `private` и иметь реализацию.
- Подкласс может наследовать только **один** абстрактный класс.

**Интерфейс:**
- Все поля автоматически `public static final`.
- Все методы `public`; до Java 8 — только абстрактные; начиная с Java 8 — также `default` и `static`.
- Класс может реализовывать **несколько** интерфейсов.

**Когда использовать абстрактный класс:**
- Несколько тесно связанных классов должны совместно использовать код.
- Ожидается наличие защищённых (`protected`) или закрытых (`private`) полей и методов.
- Нужны нестатические и не-`final` поля (состояние, которое может изменяться у каждого экземпляра).

**Когда использовать интерфейс:**
- Несвязанные классы должны реализовывать общий контракт (например, `Comparable`, `Cloneable`).
- Важно задать поведение, не связывая себя с конкретной реализацией.
- Требуется множественное наследование типов.

Примеры из JDK: `HashMap` расширяет абстрактный класс `AbstractMap` и одновременно реализует несколько интерфейсов (`Serializable`, `Cloneable`, `Map`).

#### Ответ от Claude

**Ключевые структурные различия (Java 21)**

| Характеристика | Абстрактный класс | Интерфейс |
|---|---|---|
| Поля экземпляра | Любые типы и модификаторы | Только `public static final` |
| Методы | `abstract`, конкретные любого доступа | `abstract`, `default`, `static`, `private` (Java 9+) |
| Конструкторы | Есть (не вызываются напрямую через `new`) | Отсутствуют |
| Наследование | Только одного класса | Можно реализовать несколько |
| Инициализация | Через конструктор | Через инициализаторы полей |

**Почему разграничение важно**

С Java 8+ интерфейсы получили `default`-методы и статические методы, что сблизило их с абстрактными классами. Тем не менее ключевое преимущество абстрактного класса сохраняется: **состояние экземпляра**. Если базовый тип должен хранить изменяемые поля (например, `AbstractList` хранит поле `modCount`), необходим абстрактный класс.

**Практические критерии выбора**

Выбирайте **абстрактный класс**, если:
1. Подклассы принадлежат одной концептуальной иерархии («является» — is-a) и разделяют внутреннее состояние.
2. Нужна защищённая реализация (`protected`-методы), которую нельзя объявить в интерфейсе.
3. Вы хотите, чтобы изменение базового поведения в одном месте автоматически применялось ко всем подклассам.

Выбирайте **интерфейс**, если:
1. Контракт предназначен для классов из разных иерархий.
2. Нужно множественное наследование поведения (через `default`-методы).
3. Важна гибкость: реализующий класс может быть унаследован ещё от одного класса.

**Расхождений между источниками нет:** Oracle Tutorials и JLS SE 21 (§8.1.1.1 и §9) описывают ограничения согласованно.

### Вопрос 57. Что такое enum в Java? Какие возможности предоставляет enum-тип: поля, конструкторы, методы? Перечислите методы, которые компилятор добавляет автоматически (`values()`, `valueOf()`, `ordinal()`, `name()`).

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Enum Types — The Java Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/java/javaOO/enum.html)

`enum` (перечисление) — специальный тип данных, позволяющий переменной принимать одно из заранее определённых именованных констант.

**Базовый пример:**
```java
public enum Day {
    SUNDAY, MONDAY, TUESDAY, WEDNESDAY,
    THURSDAY, FRIDAY, SATURDAY
}
```

**Возможности enum:**

- **Поля и конструктор:** enum может иметь поля и конструкторы. Конструктор должен быть `private` или package-private (по умолчанию `private`). Создать экземпляры нельзя вручную — они создаются компилятором:

```java
public enum Planet {
    MERCURY(3.303e+23, 2.4397e6),
    EARTH  (5.976e+24, 6.37814e6);

    private final double mass;
    private final double radius;

    Planet(double mass, double radius) {
        this.mass = mass;
        this.radius = radius;
    }

    double surfaceGravity() {
        return 6.67300E-11 * mass / (radius * radius);
    }
}
```

**Автоматически добавляемые методы:**

- `values()` — возвращает массив всех констант в порядке объявления.
- `valueOf(String name)` — возвращает константу по имени.

Методы, унаследованные от `java.lang.Enum`:
- `ordinal()` — порядковый номер (начиная с 0).
- `name()` — имя константы в точности как объявлено.
- `compareTo(E o)` — сравнение по порядку объявления.

#### Ответ от Claude

**Что такое enum**

`enum` — это особый вид класса (JLS §8.9), неявно наследующий `java.lang.Enum<E>`. Все константы перечисления являются `public static final` полями своего типа, создаваемыми при первом обращении к классу. Поскольку `java.lang.Enum` уже является суперклассом, наследовать другой класс `enum` не может. Зато enum **может реализовывать интерфейсы**.

**Полный список возможностей enum**

| Возможность | Допустимо? | Примечание |
|---|---|---|
| Поля | Да | Обычно `private final` |
| Конструктор | Да | Всегда `private` (неявно) |
| Экземплярные методы | Да | Любые |
| Абстрактные методы | Да | Тогда каждая константа должна предоставить тело |
| Статические методы | Да | Обычная статика |
| Реализация интерфейсов | Да | |
| Наследование другого класса | Нет | Уже наследует `Enum<E>` |
| `new EnumType()` | Нет | Создание запрещено |

**Автоматически добавляемые компилятором методы**

```java
// Генерируется компилятором автоматически:
public static MyEnum[] values()           // массив всех констант по порядку
public static MyEnum valueOf(String name) // поиск по имени (NPE / IllegalArgumentException)
```

**Методы от `java.lang.Enum`**

```java
public final String name()       // имя точно как в объявлении ("MONDAY")
public final int    ordinal()    // позиция с нуля ("MONDAY" → 1 для Day)
public String       toString()   // по умолчанию == name(), можно переопределить
public final boolean equals(Object) // final — == на уровне ссылок, сравнение безопасно через ==
public final int    hashCode()   // final
public final int    compareTo(E) // по ordinal()
```

**Важные нюансы**

- Сравнение констант через `==` **безопасно и предпочтительно** — каждая константа является синглтоном.
- `ordinal()` не следует использовать в бизнес-логике: при перестановке констант значения изменятся, что может сломать сериализацию или базы данных. Вместо этого заводите отдельное поле.
- `EnumSet` и `EnumMap` — специализированные коллекции, работающие через битовые маски и `ordinal()`, обеспечивают максимальную производительность при работе с перечислениями.

**Расхождений между источниками нет:** Oracle Tutorials, JLS SE 21 §8.9 и Javadoc `java.lang.Enum` описывают поведение согласованно.

### Вопрос 58. Что такое ковариантный тип возврата (covariant return type) при переопределении метода? Каковы правила переопределения методов в Java в части типа возврата, проверяемых исключений и модификаторов доступа?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Overriding and Hiding Methods — The Java Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/java/IandI/override.html)

Метод подкласса переопределяет метод суперкласса, если он имеет ту же сигнатуру (имя и типы параметров) и совместимый тип возврата.

**Ковариантный тип возврата** — возможность указать в переопределяющем методе более конкретный (дочерний) тип, нежели в оригинале. Доступна начиная с Java 5:

```java
class Animal {
    Animal create() { return new Animal(); }
}

class Dog extends Animal {
    @Override
    Dog create() { return new Dog(); } // Dog — подтип Animal, OK
}
```

**Правила переопределения:**

1. **Тип возврата:** должен быть *return-type-substitutable* — совпадать с типом суперкласса или быть его подтипом (ковариантность). Для примитивов тип должен совпадать точно.

2. **Проверяемые исключения:** переопределяющий метод может объявлять только **подмножество** проверяемых исключений оригинала. Нельзя добавлять новые проверяемые исключения, которых нет в суперклассе.

3. **Модификатор доступа:** доступ можно только **расширять**, но не сужать. `protected` → `public` — допустимо; `public` → `protected` — ошибка компиляции.

#### Ответ от Claude

**Ковариантный тип возврата (Covariant Return Type)**

Введён в **Java 5** (JLS §8.4.5). Позволяет переопределяющему методу возвращать тип, который является подтипом типа, возвращаемого оригинальным методом. Это делает код более типобезопасным без дополнительного каста:

```java
// Без ковариантности (Java 1.4 и ранее):
Animal a = dog.create(); // нужен каст до Dog вручную

// С ковариантностью (Java 5+):
Dog d = dog.create(); // каст не нужен — компилятор уже знает тип
```

Ковариантность работает только для **ссылочных типов**. Для примитивных типов возврата тип должен совпадать точно.

**Три правила переопределения по JLS §8.4.8.1**

**Правило 1 — Тип возврата (return-type-substitutability):**

| Оригинал | Переопределение | Допустимо? |
|---|---|---|
| `Animal` | `Dog` (подкласс) | Да (ковариантность) |
| `int` | `int` | Да (совпадение) |
| `int` | `long` | Нет |
| `void` | `void` | Да |
| `Object` | `String` | Да |

**Правило 2 — Проверяемые исключения (checked exceptions):**

Переопределяющий метод может:
- Объявлять **те же** исключения.
- Объявлять **подтипы** исключений из суперкласса.
- Вообще **не объявлять** исключения.
- Добавлять любые **непроверяемые** (`RuntimeException`, `Error`) исключения.

Нельзя: добавлять новые **проверяемые** исключения, не объявленные в суперклассе.

```java
class Parent {
    void method() throws IOException {}
}
class Child extends Parent {
    void method() throws FileNotFoundException {} // OK: FileNotFoundException extends IOException
    // void method() throws Exception {}          // ОШИБКА: Exception шире IOException
}
```

**Правило 3 — Модификатор доступа:**

Доступ можно только расширять:

```
private -> (ничего) -> protected -> public
```

- `protected` → `public` — допустимо.
- `public` → `protected` — ошибка компиляции (сужение).

**Дополнительные ограничения**

- `final`-метод не может быть переопределён.
- `static`-метод не переопределяется, а скрывается (hiding).
- `@Override` помогает компилятору поймать ошибки — если метод не является настоящим переопределением, выдаётся ошибка компиляции.

**Расхождений между источниками нет:** JLS SE 21 §8.4.8.1 и Oracle Tutorials описывают правила согласованно.

### Вопрос 59. Что такое стирание типов (type erasure) в обобщениях Java? Почему `List<String>` и `List<Integer>` являются одним и тем же типом в рантайме? Какие ограничения накладывает стирание типов?

**Категория:** Core Java — основы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Type Erasure — The Java Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/java/generics/erasure.html)

**Стирание типов (type erasure)** — механизм, при котором Java-компилятор удаляет всю информацию о параметрах типа из байткода после проверки корректности. Это сделано ради **обратной совместимости** с кодом, написанным до введения дженериков (Java 5).

Компилятор выполняет три действия при стирании типов:

1. **Заменяет параметры типа:** если параметр неограничен (`<T>`), он заменяется на `Object`; если ограничен (`<T extends Comparable>`), — на верхнюю границу (`Comparable`).

2. **Вставляет касты:** компилятор автоматически добавляет необходимые касты в байткод для обеспечения типовой безопасности.

3. **Генерирует мостовые методы (bridge methods):** для сохранения полиморфизма в обобщённых иерархиях наследования.

Результат: в байткоде не создаётся отдельных классов для `List<String>` и `List<Integer>` — оба представлены как `List`. Информация о типовых аргументах отсутствует во время выполнения.

#### Ответ от Claude

**Что такое стирание типов**

Дженерики в Java реализованы через **стирание типов** (type erasure): во время компиляции компилятор проверяет типовую корректность и затем **удаляет** все параметры типа из байткода. В рантайме JVM видит только «сырые» (raw) типы.

```java
// Исходный код:
List<String> strings = new ArrayList<>();
List<Integer> ints   = new ArrayList<>();

// После компиляции в байткоде оба превращаются в:
List strings = new ArrayList();
List ints    = new ArrayList();
```

Поэтому в рантайме:
```java
strings.getClass() == ints.getClass(); // true — оба ArrayList
```

**Как компилятор обеспечивает безопасность**

Вместо информации о типе в байткоде появляются явные касты:

```java
// Исходный:
String s = strings.get(0);

// После стирания:
String s = (String) strings.get(0); // компилятор вставил каст автоматически
```

**Ограничения, вызванные стиранием типов**

| Ограничение | Причина | Обходной путь |
|---|---|---|
| `new T()` — запрещено | Тип `T` неизвестен в рантайме | Передавать `Class<T>` как параметр |
| `new T[10]` — запрещено | Аналогично; массивы reifiable | Использовать `List<T>` или `(T[]) new Object[10]` |
| `instanceof List<String>` — запрещено | Параметр стёрт | `instanceof List<?>` (wildcard — допустим) |
| `List<String>.class` — не существует | Нет объекта `Class` для параметризованного типа | `List.class` (raw) |
| Перегрузка `void f(List<String>)` и `void f(List<Integer>)` — запрещена | После стирания сигнатуры одинаковы | Переименовать методы |

**Мостовые методы (bridge methods)**

Для поддержки полиморфизма компилятор генерирует bridge-методы. Пример:

```java
class Box<T> {
    void set(T value) {}
}
class StringBox extends Box<String> {
    @Override
    void set(String value) {}  // После стирания: set(Object) нет!
}
```

Компилятор генерирует синтетический метод `void set(Object value)`, который делегирует к `set(String value)`. Это видно через рефлексию (`method.isBridge()`).

**Heapollution (загрязнение кучи)**

Из-за стирания типов возможен `ClassCastException` в неожиданном месте:

```java
List rawList = new ArrayList();
rawList.add("hello");
List<Integer> intList = rawList; // предупреждение компилятора, не ошибка
Integer i = intList.get(0);      // ClassCastException в рантайме здесь, не при добавлении
```

**Расхождений между источниками нет:** Oracle Tutorials и JLS SE 21 §4.6 («Type Erasure») описывают механизм идентично.

### Вопрос 60. Что такое цепочка конструкторов (constructor chaining) с помощью `this()` и `super()`? Каковы ограничения на их использование? Что произойдёт, если не вызвать `super()` явно?

**Категория:** Core Java — основы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Using the this Keyword — The Java Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/java/javaOO/thiskey.html)

Ключевое слово `this` внутри конструктора позволяет вызвать другой конструктор того же класса. Это называется **явным вызовом конструктора** и используется для повторного использования кода и задания значений по умолчанию:

```java
public class Rectangle {
    private int x, y, width, height;

    public Rectangle() {
        this(0, 0, 1, 1); // вызывает 4-аргументный конструктор
    }

    public Rectangle(int width, int height) {
        this(0, 0, width, height); // делегирует полный конструктор
    }

    public Rectangle(int x, int y, int width, int height) {
        this.x = x;
        this.y = y;
        this.width = width;
        this.height = height;
    }
}
```

**Обязательное правило:** вызов `this(...)` должен быть **первым оператором** в теле конструктора.

Аналогично, `super(...)` вызывает конструктор суперкласса и тоже должен быть первым оператором. Если конструктор не содержит явного `super(...)`, компилятор автоматически вставляет `super()` (вызов конструктора суперкласса без аргументов). Если такого конструктора у суперкласса нет, возникает ошибка компиляции.

#### Ответ от Claude

**Цепочка конструкторов**

Java позволяет из одного конструктора вызвать другой конструктор того же или родительского класса. Для этого существуют два специальных вызова:

- `this(...)` — делегирование другому конструктору **того же класса**.
- `super(...)` — делегирование конструктору **суперкласса**.

**Правила использования `this()` и `super()`**

1. **Первый оператор:** любой из этих вызовов должен быть **первым оператором** в теле конструктора. Иметь и `this(...)`, и `super(...)` в одном конструкторе невозможно — они взаимно исключают друг друга.

2. **Нет рекурсии:** `this(...)` не может прямо или косвенно вызывать сам себя — компилятор обнаруживает циклическую цепочку как ошибку.

3. **Автоматическая вставка `super()`:** если конструктор не содержит ни `super(...)`, ни `this(...)`, компилятор **неявно вставляет** `super()` (вызов конструктора без аргументов суперкласса). Если у суперкласса нет публичного или защищённого конструктора без аргументов, возникает ошибка компиляции.

**Схема потока управления**

```
new Child(a, b)
    └─> Child(int a, int b)         // конструктор подкласса
         └─> super() или super(...)  // конструктор суперкласса (явный или неявный)
              └─> Object()           // корень иерархии
         // затем: блок инициализации экземпляра Child
         // затем: тело конструктора Child (после super)
```

**Пример: цепочка через `this()`**

```java
class Config {
    private final String host;
    private final int port;
    private final boolean ssl;

    // Основной конструктор
    public Config(String host, int port, boolean ssl) {
        this.host = host;
        this.port = port;
        this.ssl  = ssl;
    }

    // Делегирует основному с дефолтными значениями
    public Config(String host, int port) {
        this(host, port, false); // ПЕРВЫЙ оператор
    }

    // Ещё один удобный вариант
    public Config(String host) {
        this(host, 8080); // ПЕРВЫЙ оператор
    }
}
```

**Что произойдёт, если не вызвать `super()` явно**

Компилятор вставляет неявный `super()`. Если суперкласс имеет только конструктор с аргументами (без конструктора без аргументов), возникает **ошибка компиляции**:

```java
class Animal {
    Animal(String name) { } // нет конструктора без аргументов
}

class Dog extends Animal {
    Dog() {
        // компилятор пытается вставить super(), но его нет — ошибка компиляции
    }

    Dog(String name) {
        super(name); // явный вызов — OK
    }
}
```

**Практический совет**

Паттерн «телескопических конструкторов» (telescoping constructors) через `this(...)` работает, но при большом числе параметров становится неудобным. Альтернатива — паттерн Builder (Effective Java, Item 2), который устраняет необходимость в многочисленных конструкторах-перегрузках.

**Расхождений между источниками нет:** Oracle Tutorials, JLS SE 21 §8.8.7 («Explicit Constructor Invocations») описывают правила согласованно.
