### Вопрос 21. В чём разница между `List<Object>`, `List<?>` и сырым типом `List`? Когда каждый из них уместен?

**Категория:** Generics · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Unbounded Wildcards — The Java™ Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/java/generics/unboundedWildcards.html)

В официальном туториале Oracle объясняется, что `List<?>` («список неизвестного типа») и `List<Object>` — это **не одно и то же**. Метод `printList(List<Object> list)` нельзя вызвать с аргументом `List<Integer>` или `List<String>`, потому что `List<Integer>` **не является** подтипом `List<Object>`. Однако `printList(List<?> list)` принимает список любого типа, поскольку `List<A>` является подтипом `List<?>` для любого конкретного типа A. Из списка `List<?>` можно только читать элементы (как `Object`), но ничего нельзя добавить — кроме `null`. Сырой тип `List` (без параметра) существует исключительно для обратной совместимости с кодом до JDK 5.0 и не рекомендуется в новом коде.

#### Ответ от Claude

**`List<Object>`** — параметризованный тип, где параметр явно задан как `Object`. Позволяет добавлять в список объекты любого типа, однако является более ограничительным в отношении вызывающего кода: метод `void foo(List<Object> list)` **не принимает** `List<String>`, `List<Integer>` и т. д. Это следствие инвариантности дженериков в Java — `List<String>` не является подтипом `List<Object>`, несмотря на то что `String extends Object`.

```java
void printAll(List<Object> list) {
    for (Object o : list) System.out.println(o);
}
// Вызов с List<String> → ошибка компиляции!
```

**`List<?>`** («список неизвестного типа», unbounded wildcard) — более широкий тип: `List<A>` является подтипом `List<?>` для любого `A`. Позволяет передавать в метод список любого параметра. Ограничение: из такого списка можно только читать (получаемый тип — `Object`), а записать можно лишь `null`. Это даёт компилятору гарантию типобезопасности.

```java
void printAll(List<?> list) {
    for (Object o : list) System.out.println(o); // OK
    list.add("hello"); // Ошибка компиляции!
}
// Вызов с List<String>, List<Integer> и т. д. — OK
```

**Сырой тип `List`** (raw type) — имя дженерик-класса без параметров. Отключает проверки типов компилятора: можно добавлять и читать объекты любого типа, но с предупреждениями `unchecked`. Существует исключительно для совместимости со старым кодом (до JDK 5) и **не должен использоваться** в новом коде.

```java
List raw = new ArrayList();
raw.add("text");
raw.add(42); // Компилятор не возразит — но это небезопасно
```

**Когда использовать каждый:**

| Тип | Когда применять |
|-----|----------------|
| `List<Object>` | Когда метод должен **записывать** в список разнородные объекты и тип элементов принципиально `Object` |
| `List<?>` | Когда метод только **читает** из списка и должен принимать списки любого типа (максимальная гибкость API) |
| Сырой `List` | **Избегать**; допустимо только при работе с legacy-кодом или вызове рефлекторного API |

Дополнительно: `List<?>` и `Map<?,?>` являются **reifiable**-типами (информация о типе сохраняется в runtime), тогда как `List<? extends Object>` — нет. Поэтому только `List<?>` можно использовать с `instanceof` и как элементы массива.

### Вопрос 22. Объясните принцип PECS (Producer Extends, Consumer Super). Приведите конкретный пример метода, где нарушение PECS делает API менее гибким.

**Категория:** Generics · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [More Fun with Wildcards — The Java™ Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/extra/generics/morefun.html)

В официальном туториале Oracle объясняется принцип через метод `writeAll`. Если API возвращает (производит) значения типа `T` — используйте `? extends T`; если API принимает (потребляет) значения типа `T` — используйте `? super T`. Пример: `TreeSet(Comparator<? super E> c)` — компаратор является *потребителем*, поэтому применяется `super`. Аналогично, `Collection<? extends T>` как параметр чтения (продюсер) позволяет передавать коллекции подтипов. Комбинируя оба подхода, можно написать: `public static <T> T writeAll(Collection<? extends T> coll, Sink<? super T> snk)` — здесь коллекция является продюсером (`extends`), а `Sink` — консьюмером (`super`).

#### Ответ от Claude

**PECS** (Producer Extends, Consumer Super) — мнемоническое правило Джошуа Блоха (из книги «Effective Java») для выбора bounded wildcard при проектировании API.

**Суть правила:**
- Если коллекция/параметр **поставляет (produce)** элементы типа `T` в ваш код (т. е. вы из неё читаете) → используйте `<? extends T>`
- Если коллекция/параметр **потребляет (consume)** элементы типа `T` (т. е. вы в неё пишете) → используйте `<? super T>`

---

**Пример: метод копирования элементов**

Правильная сигнатура с PECS:
```java
public static <T> void copy(List<? extends T> src, List<? super T> dst) {
    for (T item : src) {
        dst.add(item);
    }
}
```

Вызов:
```java
List<Integer> ints = List.of(1, 2, 3);
List<Number> nums = new ArrayList<>();
copy(ints, nums); // OK: Integer extends Number
```

`src` — продюсер (читаем из него → `extends`).  
`dst` — консьюмер (пишем в него → `super`).

---

**Нарушение PECS и потеря гибкости API:**

```java
// Неправильно — без wildcards:
public static <T> void copyBad(List<T> src, List<T> dst) {
    for (T item : src) {
        dst.add(item);
    }
}
```

При такой сигнатуре компилятор требует, чтобы оба списка имели **одинаковый** параметр типа. Следующий корректный с точки зрения типобезопасности вызов не скомпилируется:

```java
List<Integer> ints = List.of(1, 2, 3);
List<Number> nums = new ArrayList<>();
copyBad(ints, nums); // Ошибка компиляции: Integer != Number
```

---

**Ещё один типичный пример — метод `addAll`:**

```java
// Правильно (как в java.util.Collection):
boolean addAll(Collection<? extends E> c);
// Позволяет: list.addAll(List<Integer>) для List<Number>

// Неправильно:
boolean addAllBad(Collection<E> c);
// Запрещает: list.addAll(List<Integer>) для List<Number> → ошибка компиляции
```

---

**Граничные случаи:**
- Если коллекция **и читается, и пишется** — не используйте wildcard вообще, применяйте конкретный тип `T`.
- `List<?>` (unbounded) — когда тип совершенно не важен (только методы `Object`).
- PECS применяется к параметрам метода; в **возвращаемых типах** wildcards лучше не использовать — это создаёт неудобства вызывающему коду.

### Вопрос 23. Почему невозможно создать массив обобщённого типа (`new T[10]`)? Как обойти это ограничение без потери типобезопасности?

**Категория:** Generics · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [The Fine Print — The Java™ Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/extra/generics/fineprint.html)

В официальном туториале Oracle объясняется, что создание массивов параметризованных типов запрещено из-за того, что такое сочетание нарушает типобезопасность. Пример из документации:

```java
List<String>[] lsa = new List<String>[10]; // Запрещено!
Object[] oa = (Object[]) lsa;
List<Integer> li = new ArrayList<Integer>();
li.add(3);
oa[1] = li; // Не выдаст ArrayStoreException (всё стирается до List)!
String s = lsa[1].get(0); // ClassCastException в runtime
```

Массивы в Java ковариантны и хранят runtime-информацию о компонентном типе (`ArrayStoreException`). Дженерики инвариантны и не сохраняют информацию о параметре типа после стирания. Их сочетание ломает гарантии типобезопасности. Как обходной путь допустимы массивы с unbounded wildcard: `List<?>[] lsa = new List<?>[10];` — они reifiable-типы и не нарушают типобезопасность.

#### Ответ от Claude

**Причина запрета**

Проблема — это конфликт между двумя механизмами типизации Java:

1. **Массивы** — **ковариантны** (`String[]` является подтипом `Object[]`) и хранят runtime-информацию о типе элементов. JVM при записи в массив проверяет совместимость типов и бросает `ArrayStoreException` при нарушении.

2. **Дженерики** — **инвариантны** (`List<String>` не является подтипом `List<Object>`) и работают через **type erasure**: вся информация о параметрах типа стирается компилятором — в bytecode остаётся только сырой тип.

Если бы `new T[10]` было разрешено, оба механизма вступили бы в противоречие: массив не мог бы проверить тип элемента в runtime (тип `T` уже стёрт), что привело бы к `ClassCastException` там, где её не ожидают.

**Конкретная демонстрация проблемы:**

```java
// Если бы это было разрешено:
List<String>[] lsa = new List<String>[10]; // Запрещено компилятором
Object[] oa = lsa; // Разрешено: ковариантность массивов
oa[1] = new ArrayList<Integer>(); // Не даст ArrayStoreException!
                                   // Runtime видит только List[], не List<String>[]
String s = lsa[1].get(0); // ClassCastException — нарушена типобезопасность
```

---

**Способы обойти ограничение:**

**1. Использовать `ArrayList<T>` вместо массива (рекомендуется)**

```java
List<T> list = new ArrayList<>();
```

Это самый идиоматичный способ в Java — `ArrayList` внутри сам управляет массивом `Object[]`.

**2. Создать массив через `Array.newInstance()` с передачей `Class<T>` токена**

```java
@SuppressWarnings("unchecked")
public <T> T[] createArray(Class<T> clazz, int size) {
    return (T[]) Array.newInstance(clazz, size);
}
// Использование:
String[] arr = createArray(String.class, 10);
```

Требует передачи `Class<T>` как runtime type token — это стандартный шаблон из «Effective Java». Приведение `(T[])` небезопасно само по себе (unchecked cast), поэтому используется `@SuppressWarnings("unchecked")` только после осознанной проверки.

**3. Создать массив `Object[]` с последующим кастом (внутри класса)**

```java
public class Stack<E> {
    private Object[] elements;

    @SuppressWarnings("unchecked")
    public Stack(int capacity) {
        elements = new Object[capacity]; // OK
    }

    @SuppressWarnings("unchecked")
    public E pop() {
        return (E) elements[--size];
    }
}
```

Этот подход используется в `java.util.ArrayList`: внутри хранится `Object[]`, а cast происходит при чтении. Unchecked cast локализован и безопасен, если класс контролирует все записи в массив.

**4. Массив с unbounded wildcard (reifiable тип)**

```java
List<?>[] lsa = new List<?>[10]; // Разрешено — List<?> reifiable
```

Работает, но требует кастов при чтении, менее удобен.

---

**Итог:** Предпочтительный выбор — `ArrayList<T>`. Если массив необходим (производительность, API), используйте `Array.newInstance(clazz, size)` с явной передачей `Class<T>` и минимально точным `@SuppressWarnings("unchecked")`.

### Вопрос 24. Что произойдёт, если попытаться перегрузить два метода с сигнатурами `void process(List<String> list)` и `void process(List<Integer> list)`? Почему?

**Категория:** Generics · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Restrictions on Generics — The Java™ Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/java/generics/restrictions.html)

Официальный туториал Oracle прямо указывает: класс не может иметь два перегруженных метода, которые получат **одинаковую сигнатуру после стирания типов** (type erasure). Приведённый пример аналогичен исходному вопросу:

```java
public class Example {
    public void print(Set<String> strSet) { }
    public void print(Set<Integer> intSet) { }
}
```

После стирания типов оба метода получают идентичные сигнатуры `print(Set)` и разделят одно представление в classfile. Это **ошибка времени компиляции**. Документация явно формулирует: «The overloads would all share the same classfile representation and will generate a compile-time error».

#### Ответ от Claude

**Что произойдёт:** **Ошибка компиляции.** Компилятор откажет с сообщением вида:
```
error: name clash: process(List<Integer>) and process(List<String>)
have the same erasure
```

**Почему: механизм type erasure**

Java реализует дженерики через **стирание типов** (type erasure): компилятор проверяет корректность обобщённого кода на этапе компиляции, но затем **удаляет** все параметры типов из bytecode. В результате:

- `List<String>` → становится `List` (сырой тип)
- `List<Integer>` → становится `List` (сырой тип)

После стирания обе перегруженные версии метода имеют одну и ту же сигнатуру:

```java
void process(List list)
```

JVM видит одну и ту же сигнатуру дважды — это конфликт в classfile, поэтому компилятор отказывает ещё на этапе javac.

---

**Пример с конкретной ошибкой:**

```java
public class Processor {
    // Ошибка компиляции: оба метода после erasure → process(List)
    public void process(List<String> list) {
        System.out.println("Strings");
    }

    public void process(List<Integer> list) {
        System.out.println("Integers");
    }
}
```

---

**Почему Java не может это разрешить:**

1. **Обратная совместимость**: стирание типов было выбрано, чтобы дженерики работали с legacy-кодом без дженериков. Это означает, что в bytecode параметры типов не существуют.

2. **JVM не знает о дженериках**: диспетчеризация методов (virtual dispatch) происходит в runtime на уровне JVM, которая видит только стёртые типы. Различить `List<String>` и `List<Integer>` невозможно.

3. **Требование JLS**: Java Language Specification (раздел 8.4.9 «Overloading») запрещает двум методам в одном классе иметь одинаковое стирание сигнатуры.

---

**Как обойти ограничение:**

```java
// Вариант 1: переименовать методы
public void processStrings(List<String> list) { ... }
public void processIntegers(List<Integer> list) { ... }

// Вариант 2: использовать один метод с параметром типа
public <T> void process(List<T> list) {
    for (T item : list) { ... }
}

// Вариант 3: использовать паттерн Visitor или instanceof (не рекомендуется)
public void process(List<?> list) {
    if (!list.isEmpty() && list.get(0) instanceof String) { ... }
    else if (!list.isEmpty() && list.get(0) instanceof Integer) { ... }
}
```

**Важный нюанс:** если методы отличаются ещё и возвращаемым типом или количеством параметров — ситуация не меняется: одинакового стирания типов параметров достаточно для ошибки компиляции.

### Вопрос 25. Можно ли использовать примитивные типы в качестве параметров обобщений? Как это влияет на производительность и что происходит под капотом?

**Категория:** Generics · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Restrictions on Generics — The Java™ Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/java/generics/restrictions.html)

Использовать примитивные типы в качестве параметров обобщений нельзя. Следующий код вызовет ошибку компиляции:

```java
Pair<int, char> p = new Pair<>(8, 'a');  // ошибка компиляции
```

Вместо этого необходимо использовать классы-обёртки:

```java
Pair<Integer, Character> p = new Pair<>(8, 'a');  // корректно
```

Ограничение обусловлено **стиранием типов (type erasure)**: Java реализует обобщения через стирание типов для сохранения обратной совместимости. Во время выполнения вся информация о параметрах типов удаляется, и остаётся только сырой тип (raw type). Поскольку параметры типов заменяются на `Object` (или на границу ограничения), а примитивы не являются подтипами `Object`, они не могут быть использованы как аргументы типа. Компилятор автоматически применяет автоупаковку при работе с обобщёнными коллекциями: `new Pair<>(8, 'a')` преобразуется в `new Pair<>(Integer.valueOf(8), Character.valueOf('a'))`.

#### Ответ от Claude

**Прямой ответ:** нет, примитивные типы (`int`, `double`, `char`, `boolean` и т.д.) не могут быть параметрами обобщений. Это фундаментальное ограничение Java, вытекающее из реализации дженериков через стирание типов.

**Почему так устроено — механизм под капотом:**

Java-дженерики реализованы через **стирание типов** (type erasure): во время компиляции все параметры типа заменяются своими ограничениями (либо `Object`, если ограничений нет). Поэтому `List<T>` в байткоде превращается в `List`, где все операции чтения/записи работают с типом `Object`. Поскольку примитивные типы (`int`, `long`, `double` и т.п.) не являются подклассами `Object`, они принципиально несовместимы с этим механизмом.

При использовании `List<Integer>` вместо `List<int>` в игру вступает **автоупаковка (autoboxing)**:

```java
List<Integer> list = new ArrayList<>();
list.add(42);          // компилятор вставляет: Integer.valueOf(42)
int x = list.get(0);   // компилятор вставляет: list.get(0).intValue()
```

**Влияние на производительность:**

1. **Память.** Объект `Integer` занимает ~16 байт на куче (заголовок объекта + 4 байта на значение + выравнивание), тогда как примитив `int` занимает ровно 4 байта. Разница в 4 раза. Для `Boolean` разрыв ещё больше: объект `Boolean` занимает ~16 байт против 1 байта примитива — разница в 16 раз.

2. **Обращение к данным.** Примитивы хранятся на стеке или непосредственно в массиве (locality of reference). Объекты-обёртки хранятся в куче, а переменная содержит только ссылку. Доступ к значению требует дополнительного разыменования указателя — это промах в кэш процессора.

3. **Стоимость boxing/unboxing.** Каждое `Integer.valueOf(int)` может создавать новый объект (исключение: значения от -128 до 127 кэшируются JVM). Каждый `intValue()` — дополнительный вызов метода.

4. **Garbage Collector.** Большое количество объектов-обёрток создаёт давление на GC, что ведёт к частым паузам в GC-чувствительных приложениях.

Официальная документация Oracle прямо предупреждает: «It is not appropriate to use autoboxing and unboxing for scientific computing, or other performance-sensitive numerical code».

**Практические рекомендации:**

- Для производительно-критичных коллекций примитивов использовать специализированные библиотеки: **Eclipse Collections**, **Trove**, **FastUtil** — они хранят данные в массивах примитивов (`int[]`) без боксинга.
- Избегать автоупаковки в горячих циклах.
- Остерегаться NPE при анбоксинге: `Integer x = null; int y = x;` бросит `NullPointerException`.
- Не сравнивать обёртки через `==`: это сравнение ссылок, а не значений (кроме кэшированного диапазона −128..127).

**Будущее:** Project Valhalla (планируется для будущих версий JDK) должен решить эту проблему через **специализацию дженериков** (generic specialization), позволяя `List<int>` хранить примитивы напрямую, без боксинга, с потенциальным ускорением в 2–4 раза и снижением потребления памяти на 50–75%.

### Вопрос 26. Что такое wildcard capture и для чего компилятор Java использует helper-методы при работе с `<?>`?

**Категория:** Generics · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Wildcard Capture and Helper Methods — The Java™ Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/java/generics/capture.html)

В некоторых ситуациях компилятор выводит конкретный тип из wildcard-выражения. Например, список может быть объявлен как `List<?>`, но при вычислении выражения компилятор выводит конкретный тип из контекста. Такой сценарий называется **wildcard capture**.

Пример кода, который **не компилируется**:

```java
void foo(List<?> i) {
    i.set(0, i.get(0));  // ошибка компиляции!
}
```

Компилятор видит: `i.get(0)` возвращает тип `CAP#1` (захваченный wildcard), а `set` требует аргумент того же типа `CAP#1`. Однако компилятор теряет связь между ними и рапортует, что нашёл `Object` вместо `CAP#1`.

Решение — **приватный helper-метод**, который «захватывает» wildcard через вывод типа:

```java
void foo(List<?> i) {
    fooHelper(i);
}

private <T> void fooHelper(List<T> l) {
    l.set(0, l.get(0));  // компилируется!
}
```

Компилятор выводит, что `T` — это `CAP#1` (переменная захваченного типа) при вызове `fooHelper(i)`, и код успешно компилируется. По соглашению helper-методы именуются `оригинальноеИмяHelper`.

#### Ответ от Claude

**Что такое wildcard capture:**

Wildcard capture (захват wildcard) — это механизм, при котором компилятор Java присваивает анонимному типу `?` конкретную внутреннюю переменную типа с именем вида `CAP#N`. Это происходит, когда компилятор встречает выражение с типом `List<?>` и должен отслеживать, что все обращения к этому конкретному экземпляру работают с одним и тем же (хотя и неизвестным) типом.

**Проблема — и почему компилятор не может решить её напрямую:**

```java
void foo(List<?> i) {
    i.set(0, i.get(0));  // ошибка: Object несовместим с CAP#1
}
```

Компилятор знает, что `i.get(0)` возвращает «некий тип `CAP#1`». `i.set(0, value)` требует аргумент того же типа `CAP#1`. Однако при анализе сигнатуры `set(int, ? super CAP#1)` компилятор теряет точность: возвращаемый тип `get()` становится `Object`, а это не совместимо с `CAP#1`. Иначе говоря, компилятор не может доказать, что объект, извлечённый из `get`, безопасно вставить обратно через `set`, несмотря на то что операция семантически безопасна.

**Решение — helper-метод:**

```java
void foo(List<?> i) {
    fooHelper(i);
}

private <T> void fooHelper(List<T> l) {
    l.set(0, l.get(0));  // T известен, set(int, T) принимает T, get() возвращает T
}
```

Когда компилятор обрабатывает вызов `fooHelper(i)`, он выводит тип `T = CAP#1` (конкретная переменная захваченного типа). Теперь внутри `fooHelper` `T` — это одна и та же переменная типа, поэтому `l.get(0)` возвращает `T`, а `l.set(0, T)` принимает `T`. Типы сходятся, и код компилируется.

**Когда helper не помогает:**

Helper-методы решают только те случаи, когда операция семантически безопасна, но компилятор не может это доказать из-за ограничений системы типов. Если сама операция небезопасна, helper не поможет:

```java
// Это небезопасно — helper не исправит ситуацию
void swapFirst(List<? extends Number> l1, List<? extends Number> l2) {
    l1.set(0, l2.get(0));  // CAP#1 и CAP#2 — разные типы, присвоение небезопасно
}
```

Здесь `l1` может быть `List<Integer>`, а `l2` — `List<Double>`. Присваивание `Double` в `List<Integer>` нарушило бы типобезопасность, и компилятор правильно отклоняет код.

**Видимость wildcard capture:**

В обычной разработке wildcard capture прозрачен: разработчик не «видит» `CAP#1` явно. Он становится виден только в сообщениях об ошибках компилятора, например: `error: method set in interface List<E> cannot be applied to given types; required: CAP#1`. Именно такое сообщение является сигналом к применению паттерна с helper-методом.

### Вопрос 27. Что такое heap pollution? При каких условиях возникает и почему varargs с обобщёнными типами особенно опасны? Как `@SafeVarargs` влияет на ситуацию?

**Категория:** Generics · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Improved Compiler Warnings When Using Non-Reifiable Formal Parameters with Varargs Methods (docs.oracle.com)](https://docs.oracle.com/javase/8/docs/technotes/guides/language/non-reifiable-varargs.html)

**Heap pollution** возникает, когда переменная параметризованного типа ссылается на объект, не соответствующий этому параметризованному типу. Это возможно только при операциях, порождающих предупреждения компилятора об unchecked-операциях.

Пример возникновения heap pollution:

```java
List l = new ArrayList<Number>();
List<String> ls = l;              // unchecked warning
l.add(0, new Integer(42));        // unchecked warning
String s = ls.get(0);             // ClassCastException в рантайме
```

Особую опасность представляют **varargs-методы с непереифицируемыми параметрами** (non-reifiable types — типы, информация о которых недоступна в рантайме из-за стирания). Компилятор преобразует `T... elements` в `T[] elements`, а из-за стирания — в `Object[] elements`. Это открывает «дыру»:

```java
public static void faultyMethod(List<String>... l) {
    Object[] objectArray = l;                        // допустимо: массив — подтип Object[]
    objectArray[0] = Arrays.asList(42);              // heap pollution!
    String s = l[0].get(0);                          // ClassCastException в рантайме
}
```

Java SE 7+ генерирует предупреждение при объявлении такого метода:

```
warning: [varargs] Possible heap pollution from parameterized vararg type
```

Аннотация `@SafeVarargs` — это **утверждение программиста** о том, что тело метода не выполняет небезопасных операций с varargs-параметром. Она подавляет предупреждения как в месте объявления метода, так и в местах его вызова. Альтернатива — `@SuppressWarnings({"unchecked", "varargs"})` — подавляет предупреждение только в месте объявления, но не в местах вызова.

#### Ответ от Claude

**Heap pollution — определение:**

Heap pollution («загрязнение кучи») — это состояние, при котором переменная параметризованного типа `List<String>` указывает в куче на объект, который в действительности является `List<Integer>` (или иным несовместимым типом). Это нарушает инварианты системы типов и может привести к `ClassCastException` в неожиданных местах — например, при простом чтении из коллекции без явного приведения типов в пользовательском коде.

**Условия возникновения:**

Heap pollution возникает, когда компилятор не может проверить корректность операции с параметризованными типами. Триггеры:

1. **Присвоение raw-типа параметризованной переменной:**
   ```java
   List l = new ArrayList<Number>();
   List<String> ls = l;  // unchecked warning → heap pollution
   ```

2. **Небезопасное приведение типов:**
   ```java
   Object obj = new ArrayList<Integer>();
   List<String> ls = (List<String>) obj;  // unchecked cast → heap pollution
   ```

3. **Varargs с обобщёнными типами** — наиболее неочевидный случай.

**Почему varargs с обобщёнными типами особенно опасны:**

Varargs — синтаксический сахар: `T... args` компилируется в `T[] args`. Однако в Java создание массивов обобщённых типов запрещено (`new List<String>[10]` — ошибка компиляции), тогда как для varargs компилятор создаёт такой массив *неявно*, без видимого предупреждения в месте вызова (до Java 7).

Цепочка небезопасности:

```java
public static void faultyMethod(List<String>... l) {
    // Компилятор создал List<String>[] — информация о String стёрта до Object
    Object[] objectArray = l;            // List[] — подтип Object[], присвоение легально
    objectArray[0] = Arrays.asList(42);  // записываем List<Integer> в Object[0]
    // l[0] теперь List<Integer>, но l имеет тип List<String>[] — heap pollution!
    String s = l[0].get(0);             // ClassCastException здесь
}
```

Опасность усиливается тем, что **ClassCastException возникает не там, где произошло загрязнение**, а в произвольном месте при чтении данных — иногда совсем в другом методе.

**Как `@SafeVarargs` влияет на ситуацию:**

`@SafeVarargs` (добавлена в Java 7) — это аннотация-утверждение программиста, которая:

1. **Подавляет предупреждения** об unchecked-операциях как в месте *объявления* метода, так и в каждом месте его *вызова* (в отличие от `@SuppressWarnings`, которая работает только в месте объявления).

2. **Не даёт никаких гарантий рантайма** — это чисто compile-time инструмент. Если программист ошибся и метод на самом деле небезопасен, `@SafeVarargs` ничего не защитит: `ClassCastException` всё равно возникнет.

**Ограничения применения `@SafeVarargs`** (compile-time error при нарушении):
- Нельзя применять к методам с фиксированной арностью (не-varargs).
- Нельзя применять к instance-методам, которые не являются `static`, `final` или `private`. Это связано с тем, что такие методы могут быть переопределены, и переопределение может нарушить гарантию безопасности.

**Безопасный пример применения:**

```java
@SafeVarargs
public static <T> List<T> listOf(T... elements) {
    // Только читаем из массива, не присваиваем в него несовместимые типы
    return new ArrayList<>(Arrays.asList(elements));
}
```

**Правило принятия решения:** `@SafeVarargs` уместна, если метод только *читает* из varargs-массива и не *сохраняет ссылку* на него в переменной другого параметризованного типа. Если метод записывает в массив из внешних источников — это небезопасно, и аннотация здесь неуместна.

### Вопрос 28. Что такое bridge method (мостовой метод)? Когда компилятор генерирует его автоматически и какую роль он играет при наследовании обобщённых классов?

**Категория:** Generics · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Effects of Type Erasure and Bridge Methods — The Java™ Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/java/generics/bridgeMethods.html)

Bridge-методы — это **синтетические методы**, генерируемые компилятором Java при компиляции класса, расширяющего параметризованный класс или реализующего параметризованный интерфейс, когда в результате стирания типов сигнатуры методов перестают совпадать.

**Проблема:** дано:

```java
public class Node<T> {
    public void setData(T data) { /* ... */ }
}

public class MyNode extends Node<Integer> {
    public void setData(Integer data) { /* ... */ }
}
```

После стирания типов `Node<T>` превращается в `Node` с методом `setData(Object data)`, а `MyNode.setData(Integer data)` уже **не переопределяет** `setData(Object)` — у них разные сигнатуры. Полиморфизм нарушается.

**Решение компилятора:** автоматически генерируется bridge-метод:

```java
// Сгенерирован компилятором
public void setData(Object data) {
    setData((Integer) data);  // делегирует к реальному методу
}
```

Теперь при вызове `node.setData("Hello")` через ссылку типа `Node` срабатывает bridge-метод, который пытается привести `"Hello"` к `Integer` и бросает `ClassCastException`. Полиморфизм восстановлен, а типобезопасность сохранена.

#### Ответ от Claude

**Что такое bridge method:**

Bridge method (мостовой метод) — это синтетический метод, автоматически добавляемый компилятором Java в байткод класса. Он невидим в исходном коде, помечается флагами `ACC_SYNTHETIC` и `ACC_BRIDGE` в `.class`-файле, и обнаруживается через рефлексию как `method.isBridge() == true` и `method.isSynthetic() == true`.

**Когда компилятор генерирует bridge-методы:**

**Случай 1: Наследование обобщённого класса с конкретным типом**

```java
public class Node<T> {
    public void setData(T data) { ... }
}

public class MyNode extends Node<Integer> {
    @Override
    public void setData(Integer data) { ... }
}
```

После стирания `Node<T>` → `Node`, метод становится `setData(Object data)`. Метод `MyNode.setData(Integer)` имеет другую сигнатуру и не переопределяет родительский. Чтобы восстановить полиморфизм, компилятор добавляет:

```java
// Bridge-метод в MyNode (генерируется автоматически):
public void setData(Object data) {
    setData((Integer) data);  // делегирует к MyNode.setData(Integer)
}
```

**Случай 2: Ковариантный возвращаемый тип**

Это второй важный случай, не связанный с дженериками напрямую. JVM требует точного совпадения сигнатур методов для переопределения, включая возвращаемый тип. Если подкласс переопределяет метод с более конкретным возвращаемым типом:

```java
class Animal {
    public Animal create() { return new Animal(); }
}

class Dog extends Animal {
    @Override
    public Dog create() { return new Dog(); }  // ковариантный тип
}
```

Компилятор генерирует bridge-метод `Animal create()` в `Dog`, который вызывает `Dog create()` и возвращает его результат как `Animal`. Это позволяет вызывать метод через ссылку типа `Animal` и получать правильное поведение.

**Роль bridge-методов:**

1. **Восстановление полиморфизма после стирания типов.** Без bridge-метода виртуальная диспетчеризация через ссылку на базовый класс не работала бы корректно.

2. **Сохранение типобезопасности.** Bridge-метод содержит явное приведение типа (`(Integer) data`), которое бросает `ClassCastException`, если в метод передан неверный тип. Это именно то поведение, которое должна обеспечивать типобезопасная система.

3. **Поддержка ковариантных возвращаемых типов.** На уровне JVM нет понятия ковариантного переопределения; bridge-метод обеспечивает обратную совместимость с bytecode.

**Наблюдение через рефлексию:**

```java
for (Method m : MyNode.class.getDeclaredMethods()) {
    System.out.println(m.getName() + " bridge=" + m.isBridge() + " synthetic=" + m.isSynthetic());
}
// setData bridge=false synthetic=false  (пользовательский метод)
// setData bridge=true  synthetic=true   (bridge-метод)
```

**Важный нюанс для разработчиков:**

Bridge-методы могут появляться в стек-трейсах и результатах рефлексии. При переборе методов через `getDeclaredMethods()` их нужно фильтровать (`m.isBridge()`), чтобы не обрабатывать дважды. Аннотации на пользовательском методе **не переносятся** на bridge-метод автоматически, что имеет значение для фреймворков (Spring AOP, Jackson), работающих через рефлексию.
### Вопрос 29. Чем reifiable-тип отличается от non-reifiable? Почему `List<String>` не является reifiable и какие проблемы это создаёт с `instanceof` и рефлексией?

**Категория:** Generics · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Non-Reifiable Types (The Java™ Tutorials — Generics)](https://docs.oracle.com/javase/tutorial/java/generics/nonReifiableVarargsType.html)

**Reifiable-тип** — это тип, полная информация о котором доступна JVM во время выполнения. К таким типам относятся: примитивы, негенерические типы, сырые типы (raw types) и вызовы параметризованных типов с неограниченным wildcard (`List<?>`).

**Non-reifiable-тип** — тип, параметры которого были удалены компилятором в ходе стирания типов (type erasure). Это все вызовы дженериков с конкретными аргументами: `List<String>`, `Map<String, Integer>` и т. д.

Так как `List<String>` является non-reifiable, JVM во время выполнения не способна отличить его от `List<Number>` — оба типа превращаются в обычный `List`. По этой причине:

- выражение `x instanceof List<String>` является **ошибкой компиляции** — допустимо лишь `x instanceof List`;
- нельзя создавать массивы параметризованных типов: `new List<String>[10]` — ошибка компиляции;
- non-reifiable varargs-параметры вызывают **heap pollution**: переменная параметризованного типа может ссылаться на объект не того типа, что приводит к `ClassCastException` в непредсказуемых местах.

#### Ответ от Claude

**Reifiable-тип (реифицируемый)** — тип, чья информация *полностью* доступна JVM в рантайме. К реифицируемым типам относятся:

- примитивы (`int`, `long`, …);
- негенерические классы и интерфейсы (`String`, `Object`);
- сырые типы (`List`, `Map`);
- параметризованные типы с неограниченным wildcard (`List<?>`, `Map<?,?>`);
- массивы реифицируемых типов (`String[]`, `List<?>[]`).

**Non-reifiable-тип** — тип, параметры которого уничтожены стиранием. Сюда попадают все конкретные параметризации: `List<String>`, `Map<String, Integer>`, а также типы с ограниченными wildcard (`List<? extends Number>`).

**Почему `List<String>` не реифицируемый.** Компилятор Java реализует дженерики через *стирание типов*: при компиляции все параметры типов заменяются их верхними границами (или `Object` для неограниченных), а в байт-код вставляются синтетические касты. В результате `List<String>` и `List<Integer>` в рантайме — это один и тот же класс `java.util.List`. JVM никогда не видела параметр `String` — он существовал только во время компиляции.

**Проблемы, которые это создаёт:**

1. **`instanceof` с параметризованным типом — ошибка компиляции.** Нельзя написать:
   ```java
   if (obj instanceof List<String>) { ... } // compile error
   ```
   Допустима только проверка по сырому типу:
   ```java
   if (obj instanceof List) { ... } // OK
   ```
   Начиная с Java 16 (pattern matching) синтаксис `obj instanceof List<?> list` разрешён, потому что `List<?>` — реифицируемый тип.

2. **Нельзя создать массив параметризованного типа.** `new List<String>[10]` — ошибка компиляции: JVM не смогла бы проверять тип элементов при записи в массив (ArrayStoreException работает только с реифицируемыми типами), что открывало бы дыру в системе типов.

3. **Heap pollution при varargs.** Если метод объявлен как `void foo(List<String>... lists)`, компилятор создаёт массив `List[]` (сырой тип). Через него можно записать `List<Integer>` туда, где ожидается `List<String>`, — и `ClassCastException` возникнет не в точке записи, а где-то при чтении. Именно поэтому такие методы требуют аннотации `@SafeVarargs`.

4. **Рефлексия над экземпляром не возвращает параметры типа.** `obj.getClass()` для экземпляра `ArrayList<String>` вернёт `class java.util.ArrayList` без какого-либо упоминания `String`. Узнать параметры можно только по *статической структуре* (см. вопрос №32).

### Вопрос 30. Что такое super type tokens (паттерн Neal Gafter)? Зачем они нужны, как работают и в каких фреймворках реально применяются?

**Категория:** Generics · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Super Type Tokens — Neal Gafter's Blog (2006)](https://gafter.blogspot.com/2006/12/super-type-tokens.html)

Когда в JDK 5 добавили дженерики, класс `Class` стал обобщённым: тип `String.class` теперь `Class<String>`. Гилад Браха ввёл термин «type token» для паттерна, при котором объект `Class<T>` передаётся методу, чтобы тот мог работать с типом `T` безопасно. Джошуа Блох назвал это паттерном «Typesafe Heterogeneous Container» (THC).

Проблема: нельзя написать `List<String>.class` — это ошибка компиляции. Стирание типов не позволяет создать type token для параметризованного типа.

Нил Гафтер предложил обходной путь — **super type tokens**: абстрактный класс `TypeRef<T>`, экземпляры которого создаются как анонимные подклассы:

```java
public abstract class TypeRef<T> {
    private final Type type;
    protected TypeRef() {
        Type superclass = getClass().getGenericSuperclass();
        type = ((ParameterizedType) superclass).getActualTypeArguments()[0];
    }
    public Type getType() { return type; }
}
```

Клиент пишет:
```java
TypeRef<List<String>> ref = new TypeRef<List<String>>() {};
```

Это работает, потому что компилятор сохраняет информацию о параметрах generic-суперкласса в байт-коде анонимного подкласса. Через `getGenericSuperclass()` + `getActualTypeArguments()` можно получить `List<String>` как объект `java.lang.reflect.Type`.

#### Ответ от Claude

**Зачем нужны super type tokens.**

Стандартные type tokens (объекты `Class<T>`) не работают с параметризованными типами: `List<String>.class` — ошибка компиляции, потому что в рантайме нет отдельного класса для `List<String>`. Это делает невозможным написать типобезопасный API вроде «десериализуй JSON в `List<String>`» — нельзя передать тип как параметр.

**Как работает паттерн.**

Ключевое наблюдение: Java *сохраняет* в байт-коде информацию о generic-параметрах **суперкласса** конкретного подкласса. Если написать:

```java
abstract class TypeRef<T> {
    private final Type type;
    protected TypeRef() {
        // getGenericSuperclass() возвращает ParameterizedType для TypeRef<List<String>>
        ParameterizedType pt = (ParameterizedType) getClass().getGenericSuperclass();
        this.type = pt.getActualTypeArguments()[0]; // → List<String>
    }
    public Type getType() { return type; }
}
```

…то создание анонимного подкласса:

```java
TypeRef<List<String>> ref = new TypeRef<List<String>>() {};
```

приводит к тому, что компилятор генерирует анонимный класс, суперкласс которого записан в байт-коде как `TypeRef<java.util.List<java.lang.String>>`. Рефлексия позволяет прочитать эту запись.

**Ограничения паттерна (указаны самим Гафтером).**

Паттерн не работает правильно при вложенных нереифицированных типах через переменные типа: если `T` — это `List<U>`, где `U` — ещё одна переменная, то рефлексия вернёт `List<U>` (переменную типа), а не конкретный класс. Поэтому паттерн надёжен только при явном указании конкретных типов.

**Реальные применения в фреймворках.**

| Фреймворк | Класс | Пример использования |
|-----------|-------|---------------------|
| **Jackson** | `com.fasterxml.jackson.core.type.TypeReference<T>` | `mapper.readValue(json, new TypeReference<List<String>>() {})` |
| **Gson** | `com.google.gson.reflect.TypeToken<T>` | `gson.fromJson(json, new TypeToken<List<String>>(){}.getType())` |
| **Spring Framework** | `org.springframework.core.ParameterizedTypeReference<T>` | `restTemplate.exchange(url, GET, null, new ParameterizedTypeReference<List<Foo>>() {})` |
| **Guava** | `com.google.common.reflect.TypeToken<T>` | Расширенный вариант с поддержкой сложных иерархий наследования |

Spring прямо указывает в документации `ParameterizedTypeReference`, что класс реализует паттерн super type tokens Нила Гафтера.

### Вопрос 31. Что означает объявление `class Enum<E extends Enum<E>>`? Зачем именно такой рекурсивный параметр типа?

**Категория:** Generics · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [AngelikaLanger.com — Java Generics FAQ: Type Parameters](https://angelikalanger.com/GenericsFAQ/FAQSections/TypeParameters.html)

`Enum<E extends Enum<E>>` — это *рекурсивный параметр типа* (recursive type parameter). Конструкция означает: «`Enum` — обобщённый тип, для которого в качестве аргумента допустимы только подтипы самого `Enum`, причём каждый такой подтип при параметризации передаёт себя в качестве аргумента типа своему родителю».

Когда вы пишете `enum Color { RED, GREEN, BLUE }`, компилятор неявно разворачивает это в `class Color extends Enum<Color>`. Тем самым класс `Color` наследует метод `compareTo(Color)` вместо `compareTo(Enum)` — то есть метод принимает строго тот же тип, что и его получатель, что обеспечивает **типобезопасное сравнение**: нельзя сравнить константу `Color` с константой `Season` на уровне компилятора.

#### Ответ от Claude

**Разбор объявления `public abstract class Enum<E extends Enum<E>>`.**

Здесь `E` — параметр типа класса `Enum` с рекурсивным ограничением: `E extends Enum<E>`. Это означает три вещи одновременно:

1. `E` должен быть подтипом `Enum` (очевидное ограничение).
2. При параметризации `Enum` конкретным типом `X` выполняется `X extends Enum<X>` — то есть `X` сам является аргументом типа для своего суперкласса `Enum`.
3. Только сами перечисления (enum-классы), сгенерированные компилятором, удовлетворяют этому ограничению — попытка написать `Enum<String>` не скомпилируется, потому что `String` не расширяет `Enum<String>`.

**Зачем нужна именно такая конструкция.**

Ключевая задача — сделать методы `compareTo` и `ordinal` **типобезопасными** на уровне конкретного перечисления, а не на уровне `Enum` вообще.

Метод `compareTo` объявлен в `Comparable<E>`, который реализует `Enum<E extends Enum<E>>`:

```java
public abstract class Enum<E extends Enum<E>> implements Comparable<E> {
    public final int compareTo(E other) { ... }
}
```

Когда компилятор раскрывает `enum Planet { MERCURY, VENUS }` в `class Planet extends Enum<Planet>`, тип `E` подставляется как `Planet`. Поэтому метод `compareTo` приобретает подпись `compareTo(Planet)`, а не `compareTo(Enum)`. Вызов `Planet.MERCURY.compareTo(DayOfWeek.MONDAY)` даст **ошибку компиляции**, а не ClassCastException в рантайме.

**Дополнительные применения рекурсивных параметров типа.**

Та же идиома используется в паттерне «fluent builder» для сохранения конкретного типа при цепочке вызовов:

```java
abstract class Builder<T, B extends Builder<T, B>> {
    public B withName(String name) { ...; return (B) this; }
    public abstract T build();
}
class PersonBuilder extends Builder<Person, PersonBuilder> { ... }
```

Без рекурсивного параметра `withName(...)` возвращал бы `Builder`, а не `PersonBuilder`, и цепочка вызовов ломалась бы при наследовании.

**Является ли ограничение полностью строгим?**

Нет: `E extends Enum<E>` — это *ограничение на параметр типа*, но компилятор не запрещает технически написать `class Foo extends Enum<Bar>` в обход. Однако в случае настоящих `enum`-объявлений компилятор генерирует `class Color extends Enum<Color>` автоматически, и на практике ограничение выполняется всегда.

### Вопрос 32. Какая информация о дженериках НЕ стирается и остаётся доступной через рефлексию в байт-коде? Как её получить программно?

**Категория:** Generics · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Java Reflection: Retrieving Generic Type Arguments at Runtime Despite Type Erasure](https://sqlpey.com/java/java-reflection-retrieve-generic-type-runtime/)

Несмотря на стирание типов, в байт-коде Java сохраняется информация о generic-сигнатурах в трёх контекстах:

**1. Суперкласс.** Через `getGenericSuperclass()`:
```java
Class<T> persistentClass = (Class<T>)
    ((ParameterizedType) getClass().getGenericSuperclass())
        .getActualTypeArguments()[0];
```

**2. Сигнатуры методов.** Через `getGenericParameterTypes()`:
```java
Type[] genericParameterTypes = method.getGenericParameterTypes();
if (genericParameterTypes[i] instanceof ParameterizedType) {
    Type[] parameters = ((ParameterizedType) genericParameterTypes[i])
        .getActualTypeArguments();
}
```

**3. Анонимные подклассы.** Наиболее надёжный способ:
```java
chill(new ArrayList<SpiderMan>() {});
```
Создание анонимного подкласса заставляет JVM сохранить конкретные аргументы типа в дескрипторе класса.

Важное ограничение: «для стандартного объявления переменной обобщённого типа (например, `List<T> listVariable`) стирание типов не позволяет определить `T` в рантайме».

#### Ответ от Claude

**Что и почему сохраняется в байт-коде.**

Компилятор Java обязан (согласно спецификации JVM, атрибут `Signature`) записывать generic-сигнатуру в байт-код для любого класса, интерфейса, конструктора или метода, чья сигнатура в исходном коде содержит ссылки на параметры типа или параметризованные типы. Это нужно для поддержки рефлексии, отладчиков и компиляторов-потребителей (например, при раздельной компиляции зависимостей).

**Что конкретно не стирается:**

| Что сохраняется | Метод рефлексии | Тип результата |
|-----------------|-----------------|----------------|
| Generic-суперкласс класса | `Class.getGenericSuperclass()` | `ParameterizedType` |
| Generic-интерфейсы класса | `Class.getGenericInterfaces()` | `Type[]` |
| Generic-тип поля | `Field.getGenericType()` | `Type` |
| Generic-возвращаемый тип метода | `Method.getGenericReturnType()` | `Type` |
| Generic-типы параметров метода | `Method.getGenericParameterTypes()` | `Type[]` |
| Generic-исключения метода | `Method.getGenericExceptionTypes()` | `Type[]` |
| Параметры типа самого класса | `Class.getTypeParameters()` | `TypeVariable[]` |

**Важно:** это информация о *статической структуре* (сигнатурах), не об *экземплярах*. Вызов `obj.getClass()` никогда не вернёт `ArrayList<String>` — только `ArrayList`.

**Как получить информацию программно — примеры:**

```java
// 1. Generic-тип поля
class Container {
    List<String> names;
}
Field f = Container.class.getDeclaredField("names");
Type type = f.getGenericType();                    // java.util.List<java.lang.String>
if (type instanceof ParameterizedType pt) {
    Type arg = pt.getActualTypeArguments()[0];     // class java.lang.String
}

// 2. Generic-суперкласс (основа super type tokens)
class StringList extends ArrayList<String> {}
Type superType = StringList.class.getGenericSuperclass();   // ArrayList<String>
ParameterizedType pt = (ParameterizedType) superType;
Type arg = pt.getActualTypeArguments()[0];                  // class java.lang.String

// 3. Параметры метода
class Parser {
    public void parse(Map<String, List<Integer>> data) {}
}
Method m = Parser.class.getMethod("parse", Map.class);
Type paramType = m.getGenericParameterTypes()[0];           // Map<String, List<Integer>>

// 4. TypeVariable (не конкретный тип, а переменная)
class Box<T extends Comparable<T>> {}
TypeVariable<?>[] tvs = Box.class.getTypeParameters();
System.out.println(tvs[0].getName());                      // "T"
System.out.println(tvs[0].getBounds()[0]);                 // java.lang.Comparable<T>
```

**Иерархия типов `java.lang.reflect.Type`:**

- `Class<T>` — обычный конкретный тип (`String`, `int[]`)
- `ParameterizedType` — параметризованный тип (`List<String>`, `Map<K,V>`)
- `TypeVariable<D>` — переменная типа (`T`, `E`)
- `WildcardType` — wildcard (`? extends Number`, `? super Integer`)
- `GenericArrayType` — массив обобщённого типа (`T[]`, `List<String>[]`)

**Ключевое ограничение.** Если переменная объявлена как `List<String> list = new ArrayList<>()`, то через рефлексию на *экземпляре* (`list.getClass()`) параметр `String` не восстановим. Восстановить его можно только из статической сигнатуры поля/метода/суперкласса. Именно на этом основан паттерн super type tokens (вопрос №30): создание анонимного подкласса — единственный способ «закрепить» конкретный тип в сигнатуре суперкласса и прочитать его рефлексией.
### Вопрос 33. Почему невозможно создать обобщённый класс-исключение (`class MyException<T> extends Exception`)? На каком уровне это ограничение и почему?

**Категория:** Generics · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Java Exceptions and generic types — Michael Scharhag](https://www.mscharhag.com/java/java-exceptions-and-generic-types)

Java запрещает создание обобщённых классов-исключений. Код вида `public class MyException<T> extends RuntimeException` не компилируется.

Фундаментальная причина — стирание типов (type erasure). Во время выполнения все обобщённые параметры стираются, поэтому `MyException<String>` и `MyException<Integer>` становятся неотличимыми — оба превращаются в `MyException`. Это делает невозможным использование разных блоков `catch` для разных параметризаций одного и того же типа исключения: JVM не может определить, какой именно обработчик должен выполниться.

Дополнительная проблема — инвариантность обобщений Java: `MyException<Integer>` нельзя поймать обработчиком `catch (MyException<Number> e)`, даже если `Integer` является подтипом `Number`. Это делает гипотетические обобщённые исключения ещё менее полезными.

#### Ответ от Claude

Ограничение зафиксировано на двух уровнях одновременно.

**1. Уровень спецификации языка (JLS).** Раздел JLS 8.1.2 прямо запрещает обобщённым классам расширять `java.lang.Throwable`. Это compile-time error:

```java
class MathException<T> extends Exception { }    // ошибка компиляции
class QueueFullException<T> extends Throwable { } // ошибка компиляции
```

**2. Фундаментальная причина — стирание типов (type erasure).** Java реализует обобщения через стирание: в байт-коде параметры типа заменяются их границами (или `Object`). На этапе выполнения `MyException<String>` и `MyException<Integer>` неразличимы — оба являются просто `MyException`. Механизм `catch` работает на уровне JVM, которая видит лишь raw-тип, поэтому написать два разных обработчика для `catch (MyException<String> e)` и `catch (MyException<Integer> e)` невозможно — компилятор и JVM не смогут их разграничить.

**Что разрешено, несмотря на ограничение.** Хотя нельзя объявить параметр типа на самом классе исключения, параметры типа можно использовать в предложении `throws` обобщённого метода или класса:

```java
// Разрешено: параметр типа в throws
abstract class Task<E extends IOException> {
    abstract void execute() throws E;
}

// Разрешено: параметр типа в throws обобщённого метода
<T extends Exception> void doWork() throws T { ... }

// Запрещено: поймать экземпляр параметра типа
public static <T extends Exception> void handle() {
    try { ... }
    catch (T e) { }  // ошибка компиляции
}
```

**Итог.** Ограничение действует на уровне JLS и обусловлено тем, что стирание типов лишает обобщённые исключения всякого смысла: catch-блоки обрабатываются во время выполнения, где параметры типа уже стёрты.

### Вопрос 34. Можно ли объявить `static`-поле обобщённого типа (`static T instance`)? Почему это запрещено и к чему привело бы разрешение?

**Категория:** Generics · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Restrictions on Generics — The Java Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/java/generics/restrictions.html)

Нельзя создавать статические поля, тип которых — параметр типа класса. Рассмотрим пример:

```java
public class MobileDevice<T> {
    private static T os;  // ошибка компиляции
}
```

Если бы это было разрешено, возникла бы следующая ситуация:

```java
MobileDevice<Smartphone> phone = new MobileDevice<>();
MobileDevice<Pager>       pager = new MobileDevice<>();
MobileDevice<TabletPC>    tablet = new MobileDevice<>();
```

Статическое поле `os` является общим для всех экземпляров класса. Оно не может одновременно быть `Smartphone`, `Pager` и `TabletPC` — отсюда неразрешимая неоднозначность. Поэтому статические поля параметра типа запрещены.

#### Ответ от Claude

**Запрет зафиксирован в JLS.** Спецификация прямо указывает: «Это ошибка времени компиляции — обращаться к параметру типа обобщённого класса C в объявлении статического члена C».

**Причина 1 — общий экземпляр поля.** Статическое поле принадлежит самому классу (raw type), а не его конкретной параметризации. `MobileDevice.os` — одно поле на всю JVM. Если одновременно существуют `MobileDevice<Smartphone>` и `MobileDevice<Pager>`, то какой тип у этого единственного поля? Ответа нет — два несовместимых типа претендуют на одно место в памяти.

**Причина 2 — стирание типов.** Обобщения Java реализованы через стирание: во время выполнения `T` превращается в `Object` (или в нижнюю границу). Статическое поле `static T os` при стирании стало бы `static Object os` — что ввело бы в заблуждение: компилятор проверяет типы, а JVM видит `Object`.

**К чему привело бы разрешение.** Разные «копии» класса для разных параметров типа разделяли бы одно поле с неопределённым типом. Это нарушило бы типобезопасность: `MobileDevice<String>` мог бы записать `String`, а `MobileDevice<Integer>` прочитать из того же поля, ожидая `Integer`.

**Как обойти ограничение.**

```java
// Вариант 1: статический словарь с type-token
public class Registry<T> {
    private static final Map<Class<?>, Object> store = new HashMap<>();

    public static <V> void set(Class<V> type, V value) {
        store.put(type, value);
    }

    @SuppressWarnings("unchecked")
    public static <V> V get(Class<V> type) {
        return (V) store.get(type);
    }
}

// Вариант 2: статическое поле в нестатическом вложенном классе с собственным параметром типа
```

**Итог.** Объявить `static T instance` нельзя — это compile-time error. Разрешение привело бы к тому, что одно статическое поле должно было бы одновременно принадлежать нескольким несовместимым типам, что невозможно ни семантически, ни технически при стирании.

### Вопрос 35. Как работает вывод типов (type inference) при вызове обобщённых методов? Приведите пример, где компилятор не может вывести тип и требует явного указания `<Type>`.

**Категория:** Generics · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Type Inference — The Java Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/java/generics/genTypeInference.html)

Вывод типов — это способность компилятора Java определять аргументы типа для вызова обобщённого метода, глядя на объявление метода и переданные аргументы. Алгоритм вывода определяет типы аргументов и, если доступно, тип, в который присваивается или возвращается результат, и затем находит наиболее специфичный тип, подходящий для всех аргументов.

Пример вызова, где тип не может быть выведен в Java SE 7:

```java
// Метод:
void processStringList(List<String> stringList) { ... }

// В Java SE 7 — ошибка компиляции: вывод даёт List<Object>
processStringList(Collections.emptyList());

// Решение — явное указание типа (type witness):
processStringList(Collections.<String>emptyList());
```

В Java SE 8 это было исправлено: компилятор научился учитывать ожидаемый тип параметра метода (target type), поэтому явный тип-свидетель стал необязательным.

#### Ответ от Claude

**Как работает вывод типов.** Компилятор Java анализирует вызов обобщённого метода и пытается установить параметры типа, используя три источника информации:
1. Типы фактических аргументов метода.
2. Ожидаемый тип результата (target type) — тип переменной, в которую присваивается возвращаемое значение, или тип параметра вызывающего метода.
3. Очевидный ожидаемый тип возврата.

Алгоритм ищет наиболее специфичный тип, удовлетворяющий всем ограничениям.

**Пример успешного вывода:**

```java
static <T> T pick(T a1, T a2) { return a2; }

Serializable s = pick("d", new ArrayList<String>());
// Компилятор выводит T = Serializable
// (общий супертип String и ArrayList<String>)
```

**Пример, где компилятор не может вывести тип (Java SE 7).**

`Collections.emptyList()` объявлен как `public static <T> List<T> emptyList()`. Компилятор обязан определить `T` только по контексту вызова, но если контекста недостаточно — он выбирает `Object`:

```java
void processStringList(List<String> stringList) { ... }

// Java 7: ОШИБКА — вывод типа даёт List<Object>, несовместим с List<String>
processStringList(Collections.emptyList());

// Решение: явный тип-свидетель (type witness)
processStringList(Collections.<String>emptyList());
```

Синтаксис `Collections.<String>emptyList()` — это тип-свидетель (type witness): угловые скобки перед именем метода явно задают аргумент типа.

**Java SE 8 расширила вывод типов.** Начиная с Java 8, компилятор учитывает тип параметра вызывающего метода как target type, поэтому пример выше компилируется без явного свидетеля. Однако есть случаи, где явное указание всё ещё требуется:

```java
// Пример, где вывод не срабатывает даже в Java 8:
// если аргумент используется в «цепочке» вызовов без явного присваивания
Object result = Collections.emptyList(); // T = Object — компилятор выводит Object,
                                          // а не нужный нам тип
```

Ещё один практический пример:

```java
// Метод принимает два List<T>
static <T> List<T> merge(List<T> a, List<T> b) { ... }

// Если один из аргументов — пустой список, компилятор может не вывести T:
List<String> result = merge(list, Collections.emptyList()); // OK в Java 8
// Но в более сложных цепочках может потребоваться:
List<String> result = merge(list, Collections.<String>emptyList());
```

**Итог.** Вывод типов анализирует аргументы и целевой тип. Явный тип-свидетель `<Type>` нужен тогда, когда компилятор не имеет достаточного контекста: чаще всего это вызовы вспомогательных методов (таких как `Collections.emptyList()`) в позициях, где target type недоступен или неоднозначен.

### Вопрос 36. Дан метод `<T extends Comparable<? super T>> T max(List<? extends T> list)` — зачем здесь `? super T` в `Comparable` вместо просто `Comparable<T>`? Что это даёт при наследовании?

**Категория:** Generics · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [More Fun with Wildcards — The Java Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/extra/generics/morefun.html)

Первоначальная попытка написать метод `max` даёт сигнатуру:

```java
public static <T extends Comparable<T>> T max(Collection<T> coll)
```

Но этот код оказывается слишком ограничительным. Рассмотрим тип, сравнимый с произвольными объектами:

```java
class Foo implements Comparable<Object> { ... }
Collection<Foo> cf = ...;
Collections.max(cf); // отклоняется компилятором!
```

Вызов отклоняется, потому что выведенный тип должен быть `Foo`, а `Foo` не реализует `Comparable<Foo>` — только `Comparable<Object>`.

Не требуется, чтобы `T` было сравнимо именно с собой. Достаточно, чтобы `T` было сравнимо с одним из своих супертипов. Это даёт:

```java
public static <T extends Comparable<? super T>> T max(Collection<T> coll)
```

Данное рассуждение применимо практически к любому использованию `Comparable`, предназначенному для произвольных типов: всегда следует использовать `Comparable<? super T>`.

#### Ответ от Claude

**Проблема с `Comparable<T>`.** Ограничение `T extends Comparable<T>` требует, чтобы тип `T` был сравним именно с экземплярами самого себя. Это слишком строго при наследовании.

**Классический пример с иерархией:**

```java
class Animal implements Comparable<Animal> {
    @Override
    public int compareTo(Animal other) { ... }
}

class Dog extends Animal { }
```

`Dog` не реализует `Comparable<Dog>` напрямую — он наследует реализацию `Comparable<Animal>`. При ограничении `T extends Comparable<T>` метод `max(List<Dog>)` не скомпилируется: `Dog` удовлетворяет `Comparable<Animal>`, но не `Comparable<Dog>`. С `Comparable<? super T>` это работает, потому что `Animal` является супертипом `Dog`.

**Что означает `? super T`.** Нижнеограниченный wildcard `? super T` означает «любой тип, являющийся `T` или супертипом `T`». Ограничение `T extends Comparable<? super T>` читается: «тип `T` реализует `Comparable` по отношению к `T` или к какому-либо предку `T`». Этого достаточно для сравнения любых двух экземпляров `T` между собой.

**Формальный разбор всей сигнатуры:**

```java
<T extends Comparable<? super T>> T max(List<? extends T> list)
```

- `T extends Comparable<? super T>` — `T` умеет сравнивать себя с `T` (или с более общим типом).
- `List<? extends T>` — список элементов типа `T` или любого его подтипа (producer: PECS).

**Почему это важно при наследовании — итоговая схема:**

```
Animal implements Comparable<Animal>
   └── Dog extends Animal

// Без ? super T:
<T extends Comparable<T>> T max(List<T> list)
// max(List<Dog>) — ОШИБКА: Dog не реализует Comparable<Dog>

// С ? super T:
<T extends Comparable<? super T>> T max(List<T> list)
// max(List<Dog>) — OK: Dog расширяет Animal, Animal реализует Comparable<Animal>,
// Animal является супертипом Dog → условие выполнено
```

**Практическое правило.** Именно по этой причине в стандартной библиотеке Java (`Collections.max`, `Collections.sort`, `TreeMap` и др.) везде применяется `Comparable<? super T>`, а не `Comparable<T>`. Это делает методы применимыми к полным иерархиям классов, а не только к конкретным типам, явно реализующим `Comparable` для самих себя.
### Вопрос 37. Что происходит при смешивании raw-типов и параметризованных типов? Как компилятор трактует unchecked-предупреждения и чем грозит их игнорирование?

**Категория:** Generics · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [OBJ03-J. Prevent heap pollution — SEI CERT Oracle Coding Standard for Java](https://cmu-sei.github.io/secure-coding-standards/sei-cert-oracle-coding-standard-for-java/rules/object-orientation-obj/obj03-j)

Heap pollution возникает, когда переменная параметризованного типа ссылается на объект, который не является объектом этого параметризованного типа. Эта проблема появляется, когда разработчики совмещают обобщённый код с raw-объявлениями типов, создавая нарушения типобезопасности, которые могут не проявиться вплоть до момента выполнения программы.

Основная причина кроется в требованиях обратной совместимости Java. Обобщённые типы появились в Java 5, но легаси-код опирался на raw-типы. Когда эти системы смешиваются без соответствующих защитных мер, компилятор выдаёт unchecked-предупреждения — и игнорирование этих предупреждений способно замаскировать серьёзные проблемы.

Когда raw-типы взаимодействуют с параметризованными коллекциями, компилятор не может применить ограничения типов. Например:

```java
List list = new ArrayList();
List<String> ls = list; // Unchecked-предупреждение
```

Информация о параметризованном типе отбрасывается во время выполнения, что позволяет несовместимым объектам проникнуть в коллекции, считающиеся типобезопасными.

В типичном сценарии целое число добавляется в `List<String>` без немедленного исключения. Исключение `ClassCastException` возникает лишь тогда, когда код пытается извлечь и привести значение, — и это место может быть очень далеко от источника проблемы, что существенно затрудняет отладку.

#### Ответ от Claude

**Что происходит при смешивании raw-типов и параметризованных типов**

Raw-тип (сырой тип) — это использование обобщённого класса или интерфейса без указания аргументов типа, например `List` вместо `List<String>`. Java допускает смешивание raw-типов с параметризованными ради обратной совместимости с кодом до Java 5, однако это порождает ряд проблем.

Присвоение параметризованного типа raw-переменной компилятор принимает без предупреждений:

```java
List<String> stringList = new ArrayList<>();
List rawList = stringList;  // OK, предупреждений нет
```

Обратное — присвоение raw-типа параметризованной переменной — генерирует `unchecked conversion` предупреждение:

```java
List rawList = new ArrayList();
List<Integer> intList = rawList;  // warning: unchecked conversion
```

Вызов параметризованного метода через raw-ссылку также даёт предупреждение:

```java
List<String> stringList = new ArrayList<>();
List rawList = stringList;
rawList.add(42);  // warning: unchecked call to add(E)
```

**Как компилятор трактует unchecked-предупреждения**

Слово «unchecked» означает, что компилятор не может гарантировать корректность операции из-за недостаточной информации о типах. Это следствие **type erasure**: во время компиляции параметры типов стираются — `List<String>` и `List<Integer>` в байткоде становятся просто `List`. Компилятор вставляет приведения типов в точках использования, но при смешивании с raw-типами эти приведения могут быть пропущены.

По умолчанию компилятор выводит предупреждения в упрощённом виде. Для получения детальной информации используется флаг:

```
javac -Xlint:unchecked MyClass.java
```

**Чем грозит игнорирование unchecked-предупреждений**

Главная угроза — **heap pollution** (загрязнение кучи): ситуация, когда переменная параметризованного типа в рантайме ссылается на объект, не соответствующий этому параметризованному типу. Это приводит к:

1. **`ClassCastException` в непредсказуемом месте.** Исключение возникает не там, где было некорректное присваивание, а там, где код пытается использовать значение. Разрыв между местом ошибки и местом падения может быть огромным, что делает отладку крайне сложной:

```java
List rawList = new ArrayList();
rawList.add(42);                      // добавляем Integer
List<String> strings = rawList;       // unchecked warning — игнорируем
String s = strings.get(0);           // ClassCastException здесь, а не выше!
```

2. **Потеря гарантий типобезопасности.** Весь смысл generics — перенести проверки из рантайма в compile-time. Игнорируя предупреждения, разработчик вручную отказывается от этой защиты.

3. **Сложность поддержки.** Код с необоснованными `@SuppressWarnings("unchecked")` скрывает потенциальные баги от следующего разработчика.

**Правила работы с unchecked-предупреждениями**

- Устранять предупреждения, используя параметризованные типы вместо raw.
- Если подавление через `@SuppressWarnings("unchecked")` всё же необходимо (например, при работе с легаси API), применять аннотацию к максимально узкой области (одна строка, не метод и не класс) и обязательно оставлять комментарий с объяснением.
- Для легаси-кода использовать `Collections.checkedList()` / `Collections.checkedMap()` — обёртки, добавляющие проверку типов в рантайме.

### Вопрос 38. Объясните разницу между upper-bounded wildcard (`? extends T`) и bounded type parameter (`<T extends Bound>`). Когда использовать один, а когда другой?

**Категория:** Generics · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Item 31: Use bounded wildcards to increase API flexibility — InformIT (Effective Java)](https://www.informit.com/articles/article.aspx?p=2861454&seqNum=6)

Ключевое различие между `? extends T` и `<T extends X>` заключается в следующем:

**Upper Bounded Wildcard (`? extends T`):** используется в параметрах методов для принятия подтипов T. Обеспечивает гибкость для вызывающего кода. Не может использоваться как возвращаемый тип (это вынуждает клиентов самостоятельно работать с wildcard). Пример: `List<? extends Number>` принимает `List<Integer>`, `List<Double>`.

**Bounded Type Parameter (`<T extends X>`):** объявляет переменную типа с ограничением. Используется, когда нужно ссылаться на конкретный тип на протяжении всего метода. Пример: `<T extends Comparable<T>>` гарантирует, что T сравним сам с собой.

**Принцип PECS (Producer Extends, Consumer Super):**

- **Producer** (поставляет данные): используй `? extends T`
  - Пример: `pushAll(Iterable<? extends E> src)` — производит экземпляры E
- **Consumer** (принимает данные): используй `? super T`
  - Пример: `popAll(Collection<? super E> dst)` — потребляет экземпляры E

Использовать wildcards для более простого публичного API, когда параметр типа встречается только один раз. Использовать bounded type parameters, когда нужно ссылаться на ограниченный тип несколько раз внутри метода или в возвращаемом типе.

#### Ответ от Claude

**Синтаксис и область применения**

| | `? extends T` (wildcard) | `<T extends Bound>` (type parameter) |
|---|---|---|
| Где используется | В сигнатурах методов как тип аргумента | В объявлениях классов и методов |
| Что означает | «Какой-то неизвестный тип, являющийся подтипом T» | «Конкретный тип T, ограниченный сверху Bound» |
| Доступ к типу внутри | Недоступен — тип анонимен | Доступен — можно использовать T по имени |

**Bounded type parameter — `<T extends Bound>`**

Объявляет именованную переменную типа. Используется, когда:

1. **Нужно ссылаться на тип в нескольких местах** — в параметрах, возвращаемом значении, теле метода:

```java
// T используется и в параметре, и в возвращаемом типе
public <T extends Comparable<T>> T max(List<T> list) {
    T result = list.get(0);
    for (T item : list) {
        if (item.compareTo(result) > 0) result = item;
    }
    return result;  // возвращаем T, а не Comparable — точный тип сохранён
}
```

2. **Нужны множественные ограничения** — wildcard не поддерживает `&`:

```java
<T extends Number & Comparable<T> & Serializable>  // OK
List<? extends Number & Comparable<?>>              // Ошибка компиляции
```

3. **Нужно создавать экземпляры или взаимодействовать с типом как с конкретным** — например, через рефлексию или `Class<T>`.

4. **Нужно связать типы нескольких параметров** между собой:

```java
public <K, V extends Comparable<V>> K keyWithMaxValue(Map<K, V> map) { ... }
```

**Upper-bounded wildcard — `? extends T`**

Обозначает анонимный тип. Используется, когда:

1. **Нужна гибкость на стороне вызывающего кода** — метод принимает коллекции разных, но совместимых типов, не зная точного типа:

```java
public double sumOfList(List<? extends Number> list) {
    double sum = 0;
    for (Number n : list) sum += n.doubleValue();
    return sum;
}

sumOfList(new ArrayList<Integer>());  // OK
sumOfList(new ArrayList<Double>());   // OK
sumOfList(new ArrayList<Float>());    // OK
```

2. **Тип используется только один раз** в сигнатуре метода (нет нужды его именовать).

3. **Только чтение из коллекции** — `List<? extends T>` является «read-only» в смысле добавления: единственное значение, которое можно добавить — `null`. Это следствие типобезопасности: компилятор не знает точный тип, поэтому запрещает запись.

**Принцип PECS (из Effective Java, Joshua Bloch)**

> **P**roducer **E**xtends, **C**onsumer **S**uper

- Если параметр **производит** значения типа T для вас (вы читаете из него) → `? extends T`
- Если параметр **потребляет** значения типа T от вас (вы пишете в него) → `? super T`
- Если и производит, и потребляет → конкретный тип (ни wildcard не подойдёт)

```java
// src — Producer: мы читаем из него элементы типа E
// dst — Consumer: мы пишем в него элементы типа E
public static <E> void copy(List<? extends E> src, List<? super E> dst) {
    for (E e : src) dst.add(e);
}
```

**Правило выбора**

Используй **`? extends T`** (wildcard), если:
- Тип встречается в сигнатуре только один раз.
- Нужна гибкость для вызывающего кода.
- Метод только читает из коллекции.
- Не нужно возвращать этот тип (wildcard в возвращаемом типе — antipattern, он «заражает» код вызывающей стороны).

Используй **`<T extends Bound>`** (type parameter), если:
- Тип нужен в нескольких местах (параметры + возвращаемое значение + тело).
- Нужны множественные ограничения через `&`.
- Нужно связать несколько аргументов одним типом.
- Нужно возвращать значение точного типа T.

### Вопрос 39. Что такое `Collections.checkedList()` / `Collections.checkedMap()`? Чем они отличаются от обычных параметризованных коллекций и когда их применение оправдано?

**Категория:** Generics · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Collections (Java SE 21 & JDK 21) — Oracle Javadoc](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/util/Collections.html)

`Collections.checkedList(List<E> list, Class<E> type)` возвращает динамически типобезопасное представление (view) переданного списка. Любая попытка вставить элемент неправильного типа немедленно вызывает `ClassCastException`. Аналогично работает `Collections.checkedMap(Map<K,V> m, Class<K> keyType, Class<V> valueType)` — он также перехватывает нарушения типа при модификации значений через `Map.Entry` из набора записей.

Гарантия сформулирована так: если до создания представления коллекция не содержала элементов неправильного типа, а все последующие операции выполняются через это представление, то коллекция гарантированно никогда не будет содержать элемент неправильного типа.

Методы существуют с Java 1.5. Вставка `null` допускается, если её допускает базовая коллекция. Представление сериализуемо, если сериализуема базовая коллекция.

Обычная параметризованная коллекция (`List<String>`) обеспечивает проверку типов **только на этапе компиляции**: из-за стирания типов (`type erasure`) в байт-коде информация о параметре типа отсутствует. Если устаревший («legacy») код через непараметризованную ссылку (`raw type`) вставит элемент неправильного типа, компилятор не предупредит, а `ClassCastException` возникнет позже — в произвольном месте при чтении элемента. `checkedList` добавляет проверку **в точке вставки**, что немедленно показывает трассировку стека с источником нарушения.

#### Ответ от Claude

**Суть проблемы.** Обобщённые типы в Java реализованы через стирание (`type erasure`): параметр `<E>` существует только во время компиляции и не сохраняется в байт-коде. Поэтому параметризованная коллекция сама по себе не может выполнить проверку типа при добавлении элемента в рантайме. Это создаёт уязвимость при взаимодействии с кодом, использующим сырые типы (`raw types`) или небезопасные приведения.

**Что делают `checkedList` / `checkedMap`.** Оба метода возвращают обёртку (decorator) вокруг исходной коллекции, которая при каждой операции добавления вызывает `Class.cast()` с заранее сохранённым объектом `Class<E>`. Если тип не совпадает — немедленно бросается `ClassCastException` с понятной трассировкой, указывающей на проблемное место. Согласно официальному Javadoc Oracle, `checkedMap` дополнительно перехватывает нарушения типа при изменении значений через `Map.Entry` из `entrySet()`.

**Отличия от обычной параметризованной коллекции:**

| Свойство | `List<String>` | `Collections.checkedList(list, String.class)` |
|---|---|---|
| Проверка типа | Только компилятор (compile-time) | Компилятор + рантайм (runtime) |
| Реакция на нарушение типа | `ClassCastException` при чтении, в произвольном месте | `ClassCastException` немедленно при вставке |
| Защита от raw-type кода | Нет | Да |
| Накладные расходы | Нет | Небольшие (`instanceof` при каждой вставке) |

**Когда применение оправдано:**

1. Отладка «heap pollution» (загрязнения кучи): когда нужно найти, где именно в большой кодовой базе в коллекцию вставляется элемент неправильного типа.
2. Передача коллекции в стороннюю библиотеку или унаследованный код, который работает с raw types и может нарушить типовую корректность коллекции.
3. В многоуровневых системах, где коллекция пересекает границы модулей и необходима дополнительная гарантия целостности на runtime-уровне.

В современном коде, где весь код написан с использованием дженериков без raw types, необходимость в `checkedXXX` минимальна. Эти методы — инструмент защиты и диагностики при интеграции с устаревшим кодом.

### Вопрос 40. Что такое «type witness» (тип-свидетель) в вызове обобщённого метода, например `Collections.<String>emptyList()`? Когда он обязателен, а когда компилятор справляется без него?

**Категория:** Generics · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Type Inference — The Java Tutorials (Oracle)](https://docs.oracle.com/javase/tutorial/java/generics/genTypeInference.html)

Type witness (тип-свидетель) — это явное указание типового аргумента в угловых скобках при вызове обобщённого метода:

```java
BoxDemo.<Integer>addBox(Integer.valueOf(10), listOfIntegerBoxes);
```

В большинстве случаев компилятор способен вывести тип самостоятельно по аргументам метода, и указание type witness избыточно:

```java
BoxDemo.addBox(Integer.valueOf(20), listOfIntegerBoxes); // компилятор сам выводит Integer
```

В Java SE 7 type witness был обязателен, когда компилятор не мог вывести тип из аргументов и вынужден был использовать `Object` по умолчанию. Например:

```java
// Java SE 7 — не компилируется:
processStringList(Collections.emptyList());
// Ошибка: List<Object> несовместим с List<String>

// Java SE 7 — компилируется с type witness:
processStringList(Collections.<String>emptyList());
```

Начиная с Java SE 8, механизм вывода типов был расширен: теперь компилятор учитывает **тип целевого контекста**, в том числе тип параметра вызываемого метода. Поэтому тот же вызов в Java 8+ работает без type witness — компилятор выводит `T = String` из ожидаемого типа параметра метода `processStringList`.

Алгоритм вывода типов использует только аргументы вызова, целевые типы и очевидный ожидаемый тип возвращаемого значения.

#### Ответ от Claude

**Определение.** Type witness — синтаксическая конструкция вида `ClassName.<TypeArg>methodName(args)`, позволяющая программисту явно указать компилятору, какой тип подставить на место параметра типа обобщённого метода, вместо того чтобы полагаться на автоматический вывод.

**Синтаксис.** При вызове через экземпляр: `obj.<Type>method(args)`. При статическом вызове: `ClassName.<Type>method(args)`. Угловые скобки ставятся **после точки, перед именем метода**.

**Как работает вывод типов без type witness.** Компилятор применяет алгоритм вывода (`type inference`), анализируя:
- типы фактических аргументов метода;
- целевой тип выражения (тип переменной, в которую присваивается результат, или ожидаемый тип параметра вызывающего метода — начиная с Java 8).

**Когда type witness обязателен (или был обязателен):**

1. **Java 7 и ниже — при передаче в метод без присваивания.** Если возвращаемое значение обобщённого метода без аргументов сразу передаётся как аргумент другому методу, компилятор Java 7 не учитывал тип параметра принимающего метода как целевой контекст и выводил `Object`:
   ```java
   // Java 7: ошибка компиляции
   processStringList(Collections.emptyList());
   // Java 7: работает
   processStringList(Collections.<String>emptyList());
   ```

2. **Неоднозначность при нескольких применимых перегрузках**, когда вывод типа приводит к неоднозначному или нежелательному результату.

3. **Вложенные обобщённые вызовы**, где цепочка вывода типов не может распространиться на внутренний вызов.

**Когда type witness не нужен (Java 8+).** Расширенный механизм poly expressions позволяет компилятору учитывать целевой тип из контекста (target typing), включая тип параметра вызывающего метода. Поэтому `Collections.emptyList()` корректно выводится как `List<String>` в большинстве практических сценариев.

**Практический совет.** В современном коде (Java 8+) type witness нужен редко. Его применение уместно как документационный приём (явно подчеркнуть намеренный тип) или при реальной неоднозначности вывода. Если type witness требуется в Java 8+, это часто признак сложной иерархии типов или ошибки проектирования.
