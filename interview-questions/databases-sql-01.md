### Вопрос 1. Какие виды JOIN существуют в SQL? Чем INNER JOIN отличается от LEFT JOIN и FULL OUTER JOIN?

**Категория:** БД и SQL · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Oracle Database 19c SQL Language Reference — Joins](https://docs.oracle.com/en/database/oracle/oracle-database/19/sqlrf/Joins.html)

Документация Oracle описывает следующие виды соединений:

- **Inner Join (внутреннее соединение)** — возвращает только те строки, для которых выполняется условие соединения (JOIN condition evaluates to TRUE). Строки без пары в другой таблице исключаются из результата.
- **Left Outer Join (левое внешнее)** — возвращает все строки из левой таблицы плюс совпадающие строки из правой. Если для строки левой таблицы нет пары в правой, колонки правой таблицы заполняются NULL.
- **Right Outer Join (правое внешнее)** — зеркальное отражение Left Outer Join: возвращает все строки из правой таблицы и совпадающие из левой.
- **Full Outer Join (полное внешнее)** — возвращает все строки из обеих таблиц. Там, где пара не найдена, подставляются NULL. Фактически это объединение результатов Left и Right Outer Join.
- **Cross Join (декартово произведение)** — возвращает все возможные комбинации строк обеих таблиц без условия соединения. Oracle предупреждает: использовать только при намеренной необходимости в декартовом произведении.

Oracle рекомендует использовать синтаксис FROM ... JOIN ... ON вместо устаревшего оператора (+).

#### Ответ от Claude

**Виды JOIN в SQL**

| Тип JOIN | Что возвращает |
|----------|---------------|
| INNER JOIN | Только строки, у которых есть совпадение в обеих таблицах |
| LEFT (OUTER) JOIN | Все строки левой таблицы + совпадающие из правой (при отсутствии пары — NULL) |
| RIGHT (OUTER) JOIN | Все строки правой таблицы + совпадающие из левой (при отсутствии пары — NULL) |
| FULL (OUTER) JOIN | Все строки из обеих таблиц, при отсутствии пары — NULL с соответствующей стороны |
| CROSS JOIN | Декартово произведение: каждая строка левой таблицы с каждой строкой правой |
| SELF JOIN | Таблица соединяется сама с собой (технически это не отдельный тип, а применение любого JOIN к одной таблице) |

**Ключевые различия:**

- **INNER JOIN vs LEFT JOIN:** INNER JOIN включает только строки с совпадением; LEFT JOIN дополнительно включает строки из левой таблицы, для которых нет пары — вместо данных правой таблицы будут NULL. Используйте LEFT JOIN, когда важно не потерять записи из «главной» таблицы.

- **INNER JOIN vs FULL OUTER JOIN:** INNER JOIN — это пересечение двух множеств. FULL OUTER JOIN — объединение: он возвращает все строки из обеих таблиц, заполняя NULL там, где совпадения нет. Это эквивалент LEFT JOIN UNION RIGHT JOIN.

**Практический пример:**

Таблицы `employees` (id, name, dept_id) и `departments` (id, name):

```sql
-- INNER JOIN: только сотрудники с отделом
SELECT e.name, d.name
FROM employees e
INNER JOIN departments d ON e.dept_id = d.id;

-- LEFT JOIN: все сотрудники, у сотрудников без отдела dept.name = NULL
SELECT e.name, d.name
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.id;

-- FULL OUTER JOIN: все сотрудники И все отделы, NULL там где нет пары
SELECT e.name, d.name
FROM employees e
FULL OUTER JOIN departments d ON e.dept_id = d.id;
```

**Важные уточнения:**
- Слова `OUTER` и `INNER` в стандарте SQL являются необязательными: `LEFT JOIN` = `LEFT OUTER JOIN`, `JOIN` = `INNER JOIN`.
- В MySQL нет нативного синтаксиса FULL OUTER JOIN, его эмулируют через `LEFT JOIN UNION RIGHT JOIN`.
- CROSS JOIN без WHERE-условия на больших таблицах создаёт огромные результирующие наборы.

### Вопрос 2. Что такое первичный ключ и внешний ключ? В чём разница между ними?

**Категория:** БД и SQL · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Oracle Bookshelf — Primary Key-Foreign Key Relationships](https://docs.oracle.com/cd/E12100_01/books/admintool/admintool_DataModeling4.html)

Документация Oracle Administration Tool описывает отношения первичного и внешнего ключей следующим образом:

**Первичный ключ (Primary Key)** — столбец или набор столбцов, в которых каждое значение уникально и однозначно идентифицирует одну строку таблицы.

**Внешний ключ (Foreign Key)** — столбец или набор столбцов в одной таблице, ссылающихся на столбцы первичного ключа другой таблицы.

Отношение первичный ключ — внешний ключ определяет связь «один ко многим» между таблицами реляционной базы данных. Например, таблица фактов `Sales` содержит строку для каждой транзакции, а таблица измерений `Date` — строку для каждой даты. Внешний ключ в `Sales` ссылается на первичный ключ в `Date`, что позволяет объединять данные через JOIN.

**Основные ограничения:**

| Аспект | Первичный ключ | Внешний ключ |
|--------|---------------|--------------|
| Уникальность | Обязателен | Не требуется |
| NULL | Не допускается | Допускается (необязательная связь) |
| Количество на таблицу | Один | Несколько |
| Ссылочная целостность | Не ссылается | Значение должно существовать как PK в другой таблице или быть NULL |

#### Ответ от Claude

**Первичный ключ (Primary Key, PK)**

Первичный ключ — это столбец или комбинация столбцов, которые **однозначно идентифицируют каждую строку** таблицы. Ограничения:
- Значения должны быть уникальными (UNIQUE).
- Значения не могут быть NULL (NOT NULL).
- Таблица может иметь только **один** первичный ключ.
- СУБД автоматически создаёт уникальный индекс по первичному ключу.

```sql
CREATE TABLE employees (
    id       BIGINT PRIMARY KEY,
    name     VARCHAR(100) NOT NULL
);
```

Первичный ключ может быть:
- **Естественным** (natural) — значимый атрибут: паспортный номер, ИНН.
- **Суррогатным** (surrogate) — искусственный идентификатор: автоинкрементный `id`, UUID. На практике предпочтительнее — не зависит от бизнес-логики.
- **Составным** (composite) — из нескольких столбцов, например (order_id, product_id).

**Внешний ключ (Foreign Key, FK)**

Внешний ключ — это столбец или комбинация столбцов, **ссылающихся на первичный ключ (или уникальный ключ) другой таблицы**. Он обеспечивает ссылочную целостность (referential integrity): нельзя вставить значение, которого нет в родительской таблице.

```sql
CREATE TABLE orders (
    id          BIGINT PRIMARY KEY,
    employee_id BIGINT,
    FOREIGN KEY (employee_id) REFERENCES employees(id)
);
```

Ограничения:
- Значение FK должно присутствовать в PK родительской таблицы **или** быть NULL.
- Одна таблица может иметь **несколько** внешних ключей.
- NULL означает отсутствие связи (необязательное отношение).

**Поведение при удалении/обновлении родительской записи (ON DELETE / ON UPDATE):**

| Действие | Описание |
|----------|----------|
| CASCADE | Удалить/обновить дочерние записи автоматически |
| SET NULL | Поставить NULL в FK дочерней таблицы |
| RESTRICT | Запретить удаление/обновление, если есть дочерние записи |
| NO ACTION | Аналогично RESTRICT, проверяется в конце транзакции |
| SET DEFAULT | Поставить значение по умолчанию (поддерживается не всеми СУБД) |

**Итоговое сравнение:**

| Характеристика | Первичный ключ | Внешний ключ |
|----------------|---------------|--------------|
| Цель | Идентификация строки внутри таблицы | Связь с другой таблицей |
| NULL | Запрещён | Разрешён |
| Уникальность | Обязательна | Не требуется |
| Количество | Один на таблицу | Несколько |
| Индекс | Создаётся автоматически | Рекомендуется создавать вручную |
| Ссылочная целостность | Не применимо | Обеспечивает |

### Вопрос 3. Что такое нормализация базы данных? Опишите 1НФ, 2НФ и 3НФ.

**Категория:** БД и SQL · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Tutorialspoint — DBMS Normalization](https://www.tutorialspoint.com/dbms/database_normalization.htm)

Нормализация — метод устранения аномалий базы данных и поддержания согласованности. Она решает три проблемы:
- **Аномалии обновления** — данные дублируются в нескольких местах, и при обновлении одного экземпляра остальные остаются несовместимыми.
- **Аномалии удаления** — при удалении записи теряются связанные данные.
- **Аномалии вставки** — невозможно добавить данные без вставки несуществующих записей.

**1НФ:** Все атрибуты должны иметь атомарные домены — каждый атрибут содержит единственное неделимое значение. Запрещены повторяющиеся группы и многозначные атрибуты.

**2НФ:** Каждый неключевой атрибут должен быть **полностью функционально зависим** от всего первичного ключа, а не от его части. Нарушением является частичная зависимость: в таблице `Student_Project` с составным ключом (Stu_ID, Proj_ID) поле Stu_Name зависит только от Stu_ID — это частичная зависимость.

**3НФ:** Требует соблюдения 2НФ и отсутствия **транзитивных зависимостей** среди неключевых атрибутов. Если Stu_ID -> Zip -> City, то City транзитивно зависит от Stu_ID через Zip — нарушение 3НФ.

#### Ответ от Claude

**Что такое нормализация?**

Нормализация базы данных — процесс приведения схемы к нормальным формам (НФ) путём разбиения таблиц на более мелкие и установления связей между ними. Цель — устранить избыточность данных и предотвратить аномалии при операциях INSERT, UPDATE, DELETE.

Нормальные формы образуют иерархию: каждая следующая включает требования предыдущей. На практике для большинства OLTP-систем достаточно 3НФ.

---

**Первая нормальная форма (1НФ)**

Правила:
1. Все значения в столбцах атомарны (неделимы) — нельзя хранить список или массив в одной ячейке.
2. Нет повторяющихся групп столбцов (например, phone1, phone2, phone3).
3. Каждая строка уникально идентифицируется (есть первичный ключ).

Нарушение 1НФ (многозначный атрибут):
```
| order_id | products           |
|----------|--------------------|
| 1        | "Книга, Ручка"     |
```

После приведения к 1НФ:
```
| order_id | product |
|----------|---------|
| 1        | Книга   |
| 1        | Ручка   |
```

---

**Вторая нормальная форма (2НФ)**

Требования: таблица в 1НФ + **отсутствие частичных зависимостей**.

Частичная зависимость возникает только при **составном первичном ключе**: когда неключевой атрибут зависит не от всего ключа, а лишь от его части.

Нарушение 2НФ:
```
| student_id | course_id | student_name | grade |
```
Ключ: (student_id, course_id). Поле `student_name` зависит только от `student_id` — частичная зависимость.

После приведения к 2НФ — разбить на две таблицы:
```
students(student_id PK, student_name)
enrollments(student_id FK, course_id FK, grade)
```

**Важно:** если первичный ключ состоит из одного столбца, таблица в 1НФ автоматически находится в 2НФ.

---

**Третья нормальная форма (3НФ)**

Требования: таблица в 2НФ + **отсутствие транзитивных зависимостей**.

Транзитивная зависимость: неключевой атрибут A зависит от неключевого атрибута B, а B зависит от первичного ключа. То есть: PK -> B -> A.

Нарушение 3НФ:
```
| employee_id | dept_id | dept_name |
```
Цепочка: employee_id -> dept_id -> dept_name. Поле `dept_name` транзитивно зависит от `employee_id` через `dept_id`.

После приведения к 3НФ:
```
employees(employee_id PK, dept_id FK)
departments(dept_id PK, dept_name)
```

---

**Итоговое сравнение нормальных форм:**

| НФ | Ключевое требование | Что устраняет |
|----|---------------------|---------------|
| 1НФ | Атомарность значений, нет повторяющихся групп | Многозначные атрибуты |
| 2НФ | 1НФ + полная зависимость от всего ключа | Частичные зависимости |
| 3НФ | 2НФ + нет транзитивных зависимостей | Транзитивные зависимости |

**Денормализация** — намеренный отказ от полной нормализации ради производительности. Применяется в аналитических системах (OLAP, DWH), где важна скорость чтения, а не минимизация аномалий записи.

### Вопрос 4. Что такое транзакция? Для чего используются команды COMMIT и ROLLBACK?

**Категория:** БД и SQL · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Oracle Database Concepts — Transactions](https://docs.oracle.com/en/database/oracle/oracle-database/26/cncpt/transactions.html)

Транзакция — это логическая, атомарная единица работы, которая содержит один или несколько SQL-операторов. Она группирует SQL-операторы таким образом, чтобы они либо все были зафиксированы (применены к базе данных), либо все отменены.

**COMMIT** завершает текущую транзакцию и делает все изменения, выполненные в её рамках, постоянными. После вызова COMMIT: изменения становятся видны другим сессиям, снимаются все блокировки строк и таблиц, удаляются точки сохранения (savepoints).

**ROLLBACK** отменяет всю работу, выполненную в рамках текущей транзакции: все изменения данных, сделанные с момента последнего COMMIT или ROLLBACK, откатываются назад с использованием сегментов отмены (undo segments), снимаются блокировки.

Пример из документации:
```sql
UPDATE accounts SET balance = balance - 500 WHERE account_id = 3209;
UPDATE accounts SET balance = balance + 500 WHERE account_id = 3208;
INSERT INTO journal VALUES (3209, 3208, 500, SYSDATE);
COMMIT; -- все три операции становятся постоянными атомарно
```

#### Ответ от Claude

**Транзакция** — это логически неделимая последовательность операций над базой данных, которая выполняется как единое целое. Транзакция либо выполняется полностью, либо не выполняется вовсе. Начинается она с первого DML-оператора (INSERT / UPDATE / DELETE) и завершается командой COMMIT или ROLLBACK.

**COMMIT** — фиксирует транзакцию:
- Все изменения, накопленные с начала транзакции, записываются в базу данных как постоянные.
- Снимаются все блокировки (locks), удаляются savepoints, изменения становятся видны другим сессиям.
- В Oracle при COMMIT в redo log записывается SCN (System Change Number).

**ROLLBACK** — откатывает транзакцию:
- Все изменения с момента последнего COMMIT (или начала транзакции) отменяются.
- База данных возвращается в то состояние, в котором она была до начала транзакции.
- Реализуется через undo-сегменты (журнал отмены), в которых хранятся «старые» значения строк.

**ROLLBACK TO SAVEPOINT** — частичный откат: отменяются изменения только до именованной точки сохранения, сама транзакция при этом не завершается.

Важные нюансы:
- DDL-операторы (CREATE, ALTER, DROP) в большинстве СУБД вызывают неявный COMMIT и не могут быть отменены через ROLLBACK.
- В MySQL по умолчанию включён режим `autocommit=1`, при котором каждый оператор является отдельной транзакцией. Явная транзакция начинается с `START TRANSACTION` или `BEGIN`.
- В PostgreSQL транзакция начинается с `BEGIN`, в Oracle — неявно с первым DML.

Типичный пример — банковский перевод:
```sql
START TRANSACTION;
UPDATE accounts SET balance = balance - 1000 WHERE id = 1; -- списание
UPDATE accounts SET balance = balance + 1000 WHERE id = 2; -- зачисление
COMMIT; -- оба изменения фиксируются вместе
-- или ROLLBACK; -- если что-то пошло не так — оба изменения отменяются
```

### Вопрос 5. Что означает аббревиатура ACID? Объясните каждое свойство.

**Категория:** БД и SQL · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Oracle Tuxedo Mainframe Adapter — ACID Properties](https://docs.oracle.com/en/database/oracle/tuxedo-mainframe-adapter/22c/tmaag/acid-properties.html)

ACID — стандартный термин баз данных, описывающий характеристики, достигаемые с помощью транзакционной природы БД.

- **Atomicity (Атомарность):** Все изменения, которые транзакция вносит в базу данных, либо фиксируются все, либо ни одно. Транзакция либо завершается полностью, либо не выполняется вовсе — промежуточных состояний нет.
- **Consistency (Согласованность):** Успешная транзакция переводит базу данных из одного корректного состояния в другое корректное состояние. База остаётся в допустимом состоянии до и после транзакции.
- **Isolation (Изоляция):** Изменения, которые транзакция вносит в базу данных, не видны другим операциям до тех пор, пока транзакция не завершит свою работу.
- **Durability (Долговечность):** Изменения, внесённые транзакцией в базу данных, переживают последующие системные или медиасбои. После COMMIT данные сохраняются даже при сбое системы.

#### Ответ от Claude

**ACID** — акроним, описывающий четыре фундаментальных свойства транзакций, гарантирующих надёжность и целостность данных.

**A — Atomicity (Атомарность)**
Транзакция — неделимая единица: либо выполняются все её операции, либо не выполняется ни одна. При сбое в середине транзакции база возвращается в исходное состояние. Реализуется через undo-журнал (журнал отмены).

Пример: если банковский перевод включает два UPDATE (списание и зачисление), то при ошибке на втором шаге первый UPDATE тоже откатывается. Не может быть ситуации «деньги списаны, но не зачислены».

**C — Consistency (Согласованность)**
Транзакция переводит базу данных из одного консистентного состояния в другое. Все ограничения целостности (constraints), триггеры и правила бизнес-логики должны выполняться до и после транзакции. Если транзакция нарушает ограничение (например, FOREIGN KEY), она откатывается.

Пример: сумма остатков на всех счетах до и после перевода должна оставаться неизменной.

**I — Isolation (Изоляция)**
Параллельно выполняемые транзакции не должны влиять друг на друга. Изменения, незафиксированные в одной транзакции, не видны другим транзакциям (в зависимости от уровня изоляции). Реализуется через блокировки (locks) или MVCC (Multi-Version Concurrency Control).

Уровни изоляции (от слабого к строгому, стандарт SQL):
1. **Read Uncommitted** — грязное чтение допустимо.
2. **Read Committed** — читаются только зафиксированные данные (защита от dirty read).
3. **Repeatable Read** — повторное чтение одной строки возвращает одно значение (защита от non-repeatable read).
4. **Serializable** — транзакции выполняются строго последовательно (защита от phantom read).

**D — Durability (Долговечность)**
После успешного COMMIT изменения сохраняются навсегда и переживают любые сбои: отключение питания, сбой ОС, падение процесса СУБД. Реализуется через redo-журнал (WAL — Write-Ahead Log): изменения записываются в журнал на диске до подтверждения транзакции клиенту.

Взаимосвязь свойств: атомарность и долговечность обеспечиваются журналами (undo и redo), согласованность — ограничениями и правилами БД, изоляция — механизмами управления параллельным доступом (блокировки, MVCC).

### Вопрос 6. Что такое индекс в базе данных и как он ускоряет выборку?

**Категория:** БД и SQL · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Use The Index, Luke — Anatomy of an SQL Index](https://use-the-index-luke.com/sql/anatomy)

Индекс — это отдельная структура данных, создаваемая командой `CREATE INDEX`. Она хранит копию индексируемых данных таблицы и занимает собственное дисковое пространство — это «чистая избыточность», аналог предметного указателя в книге.

База данных комбинирует две структуры данных для поддержания производительности при постоянных изменениях:
1. **Двусвязный список (doubly linked list)** — упорядочивает записи в строго определённой последовательности, обеспечивая быстрый поиск в отсортированных данных.
2. **B-дерево (B-Tree)** — сбалансированная древовидная структура, позволяющая поддерживать порядок при вставках, удалениях и обновлениях без перемещения большого объёма данных.

Концепция аналогична поиску в телефонном справочнике: «Поиск данных в упорядоченном наборе быстр и прост, потому что порядок сортировки определяет положение каждой записи».

Компромисс: индексы значительно ускоряют SELECT-запросы, но замедляют INSERT, UPDATE, DELETE, поскольку при каждом изменении данных нужно обновлять и индексные структуры.

#### Ответ от Claude

**Индекс** — это вспомогательная структура данных, связанная с таблицей, которая позволяет СУБД быстро находить строки по значениям одного или нескольких столбцов без полного перебора (full table scan) всей таблицы.

**Аналогия:** оглавление книги — вместо того чтобы листать все страницы, читатель сразу переходит на нужную.

**Внутреннее устройство (B-Tree индекс)**

Наиболее распространённый тип индекса — B-Tree (сбалансированное дерево):
- **Узлы ветвей (branch blocks)** содержат диапазоны ключей и указатели на дочерние узлы.
- **Листовые узлы (leaf blocks)** хранят значения ключей и соответствующие ROWID — физические адреса строк в таблице.
- Все листовые узлы связаны в двусвязный список, что позволяет эффективно выполнять диапазонные запросы (BETWEEN, >, <).
- Дерево сбалансировано: глубина от корня до любого листа одинакова, поэтому поиск любого значения занимает O(log n) операций вместо O(n) при полном сканировании.

**Как ускоряется выборка:**

1. **Index Unique Scan** — для запросов вида `WHERE id = 5` по уникальному ключу: СУБД проходит от корня дерева до листа за O(log n) и возвращает ровно одну строку.
2. **Index Range Scan** — для `WHERE last_name LIKE 'A%'` или `WHERE age BETWEEN 20 AND 30`: СУБД находит начало диапазона и затем проходит по двусвязному списку листовых блоков.
3. **Covering Index (Index-Only Scan)** — если индекс содержит все столбцы из SELECT, СУБД вообще не обращается к таблице.

**Другие типы индексов:**
- **Bitmap-индекс** — для столбцов с малым числом уникальных значений (пол, статус). Эффективен для аналитики, плохо подходит для OLTP из-за блокировок.
- **Hash-индекс** (PostgreSQL, Memory engine MySQL) — O(1) для точного поиска, не работает для диапазонных запросов.
- **Функциональный индекс** — индексирует результат выражения, например `UPPER(email)`.

**Цена индексов (trade-offs):**
- Каждый индекс занимает дополнительное место на диске.
- При INSERT, UPDATE, DELETE СУБД обязана обновить все индексы таблицы, что замедляет запись. Чем больше индексов — тем медленнее DML.
- Оптимизатор запросов может выбрать индекс неоптимально или вовсе проигнорировать его (например, при низкой селективности: `WHERE gender = 'M'` при 50% строк с таким значением).

**Когда создавать индекс:**
- Столбцы в условиях WHERE, JOIN ON, ORDER BY, GROUP BY с высокой селективностью.
- Внешние ключи (FOREIGN KEY) — для избежания полного сканирования при каскадных операциях.
- **Не стоит** индексировать небольшие таблицы (full scan быстрее), часто обновляемые столбцы с низкой селективностью.
### Вопрос 7. В чём разница между Statement и PreparedStatement в JDBC? Почему PreparedStatement предпочтительнее?

**Категория:** БД и SQL · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Using Prepared Statements (The Java™ Tutorials > JDBC Database Access > JDBC Basics)](https://docs.oracle.com/javase/tutorial/jdbc/basics/prepared.html)

`PreparedStatement` — это специальный тип объекта SQL-запроса в JDBC, производный от общего класса `Statement`. В отличие от обычных объектов `Statement`, он получает SQL-запрос в момент создания и предварительно компилируется СУБД.

Ключевые отличия:

| Аспект | Statement | PreparedStatement |
|--------|-----------|-------------------|
| Компиляция SQL | Компилируется при каждом выполнении | Предварительно компилируется при создании |
| Параметры | Только статический SQL | Параметризованный SQL с заменителями `?` |
| Повторное использование | Менее эффективно при многократном выполнении | Оптимизирован для повторного использования |

Преимущества `PreparedStatement`:
1. **Производительность** — СУБД компилирует запрос один раз при создании, повторные выполнения пропускают этап компиляции.
2. **Безопасность** — защита от SQL-инъекций: "Prepared statements always treat client-supplied data as content of a parameter and never as a part of an SQL statement."
3. **Предварительная компиляция** — SQL-запрос отправляется в СУБД и компилируется сразу при создании объекта.

Пример повторного использования с разными параметрами:

```java
String updateString = "update COFFEES set SALES = ? where COF_NAME = ?";
PreparedStatement updateSales = con.prepareStatement(updateString);

updateSales.setInt(1, 100);
updateSales.setString(2, "French_Roast");
updateSales.executeUpdate();

// Повторное использование с новыми значениями
updateSales.setInt(1, 50);
updateSales.setString(2, "Espresso");
updateSales.executeUpdate();
```

#### Ответ от Claude

**Statement** и **PreparedStatement** — два интерфейса JDBC для выполнения SQL-запросов к базе данных. `PreparedStatement` расширяет `Statement`.

**Ключевые различия:**

1. **Компиляция запроса**
   - `Statement`: каждый вызов `execute()` передаёт строку SQL в СУБД, которая заново её парсит, компилирует и строит план выполнения.
   - `PreparedStatement`: SQL компилируется один раз при вызове `connection.prepareStatement(sql)`. При последующих вызовах СУБД переиспользует уже готовый план выполнения.

2. **Параметры**
   - `Statement`: работает только со статическими SQL-строками; параметры приходится вставлять конкатенацией строк, что опасно.
   - `PreparedStatement`: поддерживает заменители `?` (IN-параметры), значения которых задаются типобезопасными методами: `setInt()`, `setString()`, `setDate()` и т. д.

3. **Производительность**
   При многократном выполнении одного и того же запроса с разными параметрами `PreparedStatement` существенно быстрее — СУБД не тратит время на повторный парсинг и построение плана. Особенно заметно при batch-операциях.

4. **Безопасность (SQL-инъекции)**
   `Statement` с конкатенацией строк уязвим для SQL-инъекций. `PreparedStatement` полностью отделяет структуру запроса от данных: переданные значения никогда не интерпретируются как SQL-код.

5. **Типобезопасность**
   Методы `set*()` выполняют корректное экранирование и преобразование типов, что снижает вероятность ошибок при работе с датами, числами и строками.

**Когда использовать `Statement`:**
Подходит для DDL-операций (CREATE, ALTER, DROP), которые выполняются один раз и не принимают пользовательский ввод.

**Когда использовать `PreparedStatement`:**
Во всех остальных случаях — DML-операции (SELECT, INSERT, UPDATE, DELETE), особенно с пользовательским вводом или при многократном выполнении.

**Итог:** `PreparedStatement` предпочтительнее из-за трёх факторов одновременно — производительность, безопасность и читаемость кода.

### Вопрос 8. Что такое SQL-инъекция и как PreparedStatement защищает от неё?

**Категория:** БД и SQL · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [SQL Injection Prevention Cheat Sheet — OWASP Cheat Sheet Series](https://cheatsheetseries.owasp.org/cheatsheets/SQL_Injection_Prevention_Cheat_Sheet.html)

SQL-инъекция возникает, когда злоумышленники внедряют вредоносный SQL-код в запросы приложения, использующие конкатенацию строк с пользовательским вводом. По определению OWASP: "SQL Injection allows evil attacker code to change the structure of a web application's SQL statement in a way that can steal data, modify data, or potentially facilitate command injection to the underlying OS."

**Уязвимый код:**

```java
String query = "SELECT account_balance FROM user_data WHERE user_name = "
    + request.getParameter("customerName");
Statement statement = connection.createStatement();
ResultSet results = statement.executeQuery(query);
```

Если пользователь введёт `tom' or '1'='1`, запрос превратится в:
```sql
SELECT account_balance FROM user_data WHERE user_name = 'tom' or '1'='1'
```
Условие `'1'='1'` всегда истинно, и запрос вернёт все записи.

**Защита с помощью PreparedStatement:**

```java
String custname = request.getParameter("customerName");
String query = "SELECT account_balance FROM user_data WHERE user_name = ?";
PreparedStatement pstmt = connection.prepareStatement(query);
pstmt.setString(1, custname);
ResultSet results = pstmt.executeQuery();
```

Заменитель `?` сообщает базе данных, что следующее значение является данными, а не SQL-кодом. Даже если пользователь введёт `tom' or '1'='1`, база данных будет искать пользователя с буквальным именем `tom' or '1'='1` — то есть такой пользователь просто не будет найден, инъекция не сработает.

Ключевой принцип по OWASP: "Parameterized queries force the developer to define all SQL code first and pass in each parameter to the query later", обеспечивая чёткое разделение кода и данных.

#### Ответ от Claude

**SQL-инъекция (SQL Injection)** — вид атаки на веб-приложения, при котором злоумышленник вставляет (инъецирует) произвольный SQL-код в параметры запроса. Если приложение строит SQL-запросы конкатенацией строк, атакующий может изменить логику запроса: получить доступ к чужим данным, обойти аутентификацию, удалить или изменить данные, а в некоторых случаях — выполнить команды на уровне ОС.

**Пример атаки:**

```java
// Уязвимый код — конкатенация строк
String sql = "SELECT * FROM users WHERE login='" + login + "' AND password='" + password + "'";
```

Если злоумышленник введёт в поле логина строку `admin' --`, запрос примет вид:
```sql
SELECT * FROM users WHERE login='admin' --' AND password='anything'
```
Символы `--` начинают комментарий в SQL: всё после них игнорируется, и проверка пароля обходится. Атакующий входит под учётной записью `admin` без знания пароля.

**Как PreparedStatement защищает:**

Механизм защиты основан на **разделении структуры запроса и данных**:

1. При вызове `connection.prepareStatement(sql)` SQL-шаблон с заменителями `?` компилируется СУБД в план выполнения.
2. Методы `setString()`, `setInt()` и т. д. передают значения параметров отдельным протокольным сообщением — после того, как структура запроса уже зафиксирована.
3. СУБД гарантирует, что переданное значение будет интерпретировано **только как данные**, а не как часть SQL-синтаксиса. Одинарные кавычки, комментарии, ключевые слова — всё это в значении параметра безопасно.

```java
// Безопасный код — PreparedStatement
String sql = "SELECT * FROM users WHERE login = ? AND password = ?";
PreparedStatement pstmt = connection.prepareStatement(sql);
pstmt.setString(1, login);
pstmt.setString(2, password);
ResultSet rs = pstmt.executeQuery();
```

Даже при вводе `admin' --` значение будет передано как строка `admin' --` и просто не совпадёт ни с одной записью — инъекция невозможна.

**Дополнительные меры защиты** (по OWASP, помимо PreparedStatement):
- Хранимые процедуры (если параметризованы правильно)
- Валидация входных данных (allowlist)
- Принцип минимальных привилегий для учётной записи БД
- Экранирование спецсимволов (крайняя мера, менее надёжная)

PreparedStatement — это **первая линия обороны** и стандарт де-факто в Java-разработке.

### Вопрос 9. Как соотношения one-to-one, one-to-many и many-to-many представляются в виде таблиц реляционной БД?

**Категория:** БД и SQL · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Relationships in SQL — One-to-One, One-to-Many, Many-to-Many](https://www.geeksforgeeks.org/sql/relationships-in-sql-one-to-one-one-to-many-many-to-many/)

**One-to-One (один-к-одному)**

Каждая запись в таблице A связана ровно с одной записью в таблице B, и наоборот. Реализуется через внешний ключ с ограничением `UNIQUE`:

```sql
CREATE TABLE users (
    user_id INT PRIMARY KEY,
    username VARCHAR(50)
);
CREATE TABLE user_profiles (
    profile_id INT PRIMARY KEY,
    user_id INT UNIQUE,
    profile_data VARCHAR(255),
    FOREIGN KEY (user_id) REFERENCES users(user_id)
);
```

Ограничение `UNIQUE` на столбце `user_id` гарантирует, что один профиль соответствует ровно одному пользователю.

**One-to-Many (один-ко-многим)**

Одна запись в таблице A может быть связана со многими записями в таблице B, но каждая запись в таблице B связана только с одной записью в таблице A. Внешний ключ размещается на стороне «многих»:

```sql
CREATE TABLE departments (
    department_id INT PRIMARY KEY,
    department_name VARCHAR(50)
);
CREATE TABLE employees (
    employee_id INT PRIMARY KEY,
    employee_name VARCHAR(50),
    department_id INT,
    FOREIGN KEY (department_id) REFERENCES departments(department_id)
);
```

**Many-to-Many (многие-ко-многим)**

Каждая запись в таблице A может быть связана со многими записями в таблице B, и наоборот. Требует промежуточной таблицы-связи (junction table):

```sql
CREATE TABLE students (
    student_id INT PRIMARY KEY,
    student_name VARCHAR(50)
);
CREATE TABLE courses (
    course_id INT PRIMARY KEY,
    course_name VARCHAR(50)
);
CREATE TABLE student_courses (
    student_id INT,
    course_id INT,
    PRIMARY KEY (student_id, course_id),
    FOREIGN KEY (student_id) REFERENCES students(student_id),
    FOREIGN KEY (course_id) REFERENCES courses(course_id)
);
```

Таблица `student_courses` содержит внешние ключи на обе основные таблицы, а составной первичный ключ гарантирует уникальность пар.

#### Ответ от Claude

В реляционных базах данных связи между сущностями моделируются с помощью внешних ключей (FOREIGN KEY). Тип связи определяет, в какую таблицу и с какими ограничениями помещается внешний ключ.

---

**One-to-One (один-к-одному)**

Каждая строка в таблице A соответствует не более чем одной строке в таблице B.

*Реализация:* В одну из таблиц добавляется внешний ключ, ссылающийся на первичный ключ другой таблицы, с ограничением `UNIQUE` на этом столбце — это и обеспечивает уникальность связи.

*Пример:* Таблицы `passport` и `person` — у каждого человека ровно один паспорт:

```sql
CREATE TABLE person (
    person_id INT PRIMARY KEY,
    full_name  VARCHAR(100)
);
CREATE TABLE passport (
    passport_id INT PRIMARY KEY,
    person_id   INT UNIQUE,               -- UNIQUE гарантирует 1-to-1
    series      VARCHAR(10),
    FOREIGN KEY (person_id) REFERENCES person(person_id)
);
```

*Типичное применение:* Разбиение широкой таблицы на основную и дополнительную (например, редко читаемые поля — в отдельную таблицу), либо хранение разных аспектов одной сущности (пользователь и его настройки).

---

**One-to-Many (один-ко-многим)**

Одна строка в таблице A может быть связана с несколькими строками в таблице B, но каждая строка в B ссылается ровно на одну строку в A.

*Реализация:* Внешний ключ помещается в таблицу на стороне «многих» (B), без ограничения `UNIQUE`.

*Пример:* Один отдел содержит много сотрудников:

```sql
CREATE TABLE department (
    department_id   INT PRIMARY KEY,
    department_name VARCHAR(50)
);
CREATE TABLE employee (
    employee_id   INT PRIMARY KEY,
    employee_name VARCHAR(100),
    department_id INT,                    -- FK на стороне "многих"
    FOREIGN KEY (department_id) REFERENCES department(department_id)
);
```

*Типичное применение:* Самая распространённая связь. Примеры: автор — книги, заказ — позиции заказа, категория — товары.

---

**Many-to-Many (многие-ко-многим)**

Каждая строка в таблице A может быть связана со многими строками в таблице B, и наоборот.

*Реализация:* Прямое представление невозможно двумя таблицами. Необходима третья — **таблица-связь** (junction table, associative table, cross-reference table). Она содержит два внешних ключа — по одному на каждую из основных таблиц. Составной первичный ключ из этих двух столбцов гарантирует уникальность пар.

*Пример:* Студент может записаться на несколько курсов; курс могут посещать несколько студентов:

```sql
CREATE TABLE student (
    student_id   INT PRIMARY KEY,
    student_name VARCHAR(100)
);
CREATE TABLE course (
    course_id   INT PRIMARY KEY,
    course_name VARCHAR(100)
);
CREATE TABLE student_course (   -- таблица-связь
    student_id INT,
    course_id  INT,
    enrolled_at DATE,           -- можно добавлять атрибуты самой связи
    PRIMARY KEY (student_id, course_id),
    FOREIGN KEY (student_id) REFERENCES student(student_id),
    FOREIGN KEY (course_id)  REFERENCES course(course_id)
);
```

*Типичное применение:* Теги и статьи, пользователи и роли, актёры и фильмы. В таблицу-связь можно добавлять собственные атрибуты (дата записи, статус и т. д.).

---

**Сводная таблица:**

| Тип связи | Где размещается FK | Ограничение | Нужна ли доп. таблица |
|-----------|--------------------|-------------|----------------------|
| One-to-One | В одной из двух таблиц | FK + UNIQUE | Нет |
| One-to-Many | В таблице «многих» | FK (без UNIQUE) | Нет |
| Many-to-Many | В таблице-связи | Составной PK | Да |

### Вопрос 10. Чем отличаются WHERE и HAVING? В каком случае нельзя использовать WHERE вместо HAVING?

**Категория:** БД и SQL · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [The Difference Between WHERE and HAVING in SQL — DataCamp](https://www.datacamp.com/tutorial/difference-between-where-and-having-clause-in-sql)

Фундаментальное отличие — это момент применения фильтрации. WHERE фильтрует отдельные строки до группировки и агрегации, а HAVING фильтрует уже сгруппированные данные после агрегации. WHERE выполняется сразу после FROM (до GROUP BY), GROUP BY затем обрабатывает данные, и только потом HAVING оценивает результаты.

WHERE не может содержать агрегатные функции — такие как COUNT(), SUM(), AVG(), MIN(), MAX(). HAVING же предназначен именно для фильтрации по результатам агрегатных функций. WHERE работает с SELECT, UPDATE и DELETE, тогда как HAVING применяется только в SELECT.

Пример, где WHERE заменить нельзя:
```sql
SELECT city, AVG(rental_price)
FROM rentals
GROUP BY city
HAVING AVG(rental_price) < 2700;
```

WHERE эффективнее с точки зрения производительности, поскольку сокращает набор строк до группировки.

#### Ответ от Claude

**Ключевое отличие: момент применения фильтра**

Порядок выполнения SQL-запроса:
```
FROM → WHERE → GROUP BY → (агрегатные функции) → HAVING → SELECT → ORDER BY
```

- **WHERE** — фильтрует строки *до* группировки, работает с атрибутами отдельных строк.
- **HAVING** — фильтрует группы *после* группировки и агрегации, работает с результатами агрегатных функций.

**Когда WHERE нельзя заменить HAVING**

WHERE нельзя использовать вместо HAVING во всех случаях, когда условие фильтрации содержит агрегатную функцию (COUNT, SUM, AVG, MIN, MAX). На момент выполнения WHERE агрегация ещё не произошла — движок БД просто не знает значений COUNT(*) или SUM(amount).

Пример: найти отделы, в которых более 5 сотрудников:
```sql
-- ПРАВИЛЬНО
SELECT department_id, COUNT(*) AS emp_count
FROM employees
GROUP BY department_id
HAVING COUNT(*) > 5;

-- ОШИБКА — COUNT(*) нельзя использовать в WHERE
SELECT department_id, COUNT(*) AS emp_count
FROM employees
WHERE COUNT(*) > 5   -- синтаксическая ошибка
GROUP BY department_id;
```

**Совместное использование WHERE и HAVING**

Оба условия могут присутствовать в одном запросе. WHERE снижает число строк до группировки (выгоднее для производительности), HAVING доотфильтровывает уже готовые группы:
```sql
SELECT department_id, AVG(salary) AS avg_sal
FROM employees
WHERE hire_date > '2020-01-01'   -- сначала убираем старых сотрудников
GROUP BY department_id
HAVING AVG(salary) > 60000;      -- затем фильтруем группы по средней зп
```

**Дополнительные отличия**

| Критерий | WHERE | HAVING |
|---|---|---|
| Агрегатные функции | Нельзя | Можно |
| Момент применения | До GROUP BY | После GROUP BY |
| Применимость | SELECT, UPDATE, DELETE | Только SELECT |
| Использование индексов | Да | Ограниченно |
| Производительность | Выше (меньше строк для GROUP BY) | Ниже (обрабатывает все группы) |

**Итог:** WHERE нельзя заменить HAVING везде, где условие фильтрации основано на агрегатной функции — это единственный случай, когда HAVING обязателен.

### Вопрос 11. Какие уровни изоляции транзакций существуют? Перечислите их от самого слабого к самому строгому.

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [MySQL 8.4 Reference Manual — 17.7.2.1 Transaction Isolation Levels](https://dev.mysql.com/doc/refman/8.4/en/innodb-transaction-isolation-levels.html)

InnoDB поддерживает все четыре уровня изоляции стандарта SQL:1992. По умолчанию используется REPEATABLE READ.

- **READ UNCOMMITTED** — SELECT-операторы выполняются без блокировок, возможно чтение более ранней версии строки. Допускает «грязное чтение».
- **READ COMMITTED** — каждый consistent read устанавливает и читает собственный свежий снимок данных. InnoDB блокирует только записи индекса, не промежутки. Снижает вероятность дедлоков.
- **REPEATABLE READ** (по умолчанию) — consistent read внутри одной транзакции читает снимок, установленный первым чтением. Для блокирующих чтений (SELECT...FOR UPDATE) при поиске по уникальному индексу блокируется только конкретная запись; иначе используются gap-блокировки или next-key-блокировки.
- **SERIALIZABLE** — аналогичен REPEATABLE READ, но неявно преобразует обычные SELECT в SELECT...FOR SHARE при отключённом autocommit. Применяется для XA-транзакций и отладки проблем конкурентного доступа.

#### Ответ от Claude

Стандарт ANSI/ISO SQL определяет четыре уровня изоляции транзакций. От самого слабого к самому строгому:

**1. READ UNCOMMITTED (Чтение незафиксированных данных)**
Самый слабый уровень. Транзакция может читать изменения, ещё не зафиксированные другими транзакциями. Блокировки чтения не применяются. На практике используется крайне редко.

**2. READ COMMITTED (Чтение зафиксированных данных)**
Транзакция видит только зафиксированные изменения. Каждый оператор SELECT работает со своим актуальным снимком данных. Является уровнем по умолчанию в PostgreSQL, Oracle, SQL Server и многих других СУБД.

**3. REPEATABLE READ (Повторяемое чтение)**
Транзакция видит данные в том состоянии, в котором они находились на момент начала транзакции (или первого чтения). Повторное чтение той же строки всегда возвращает одинаковый результат. Уровень по умолчанию в MySQL InnoDB.

**4. SERIALIZABLE (Сериализуемость)**
Самый строгий уровень. Транзакции выполняются так, как если бы они шли строго последовательно (одна за другой), а не параллельно. Полностью исключает все аномалии параллельного доступа. Достигается через диапазонные блокировки или механизм SSI (Serializable Snapshot Isolation).

**Сводная таблица аномалий (стандарт SQL)**

| Уровень изоляции | Грязное чтение | Неповторяемое чтение | Фантомное чтение |
|---|---|---|---|
| READ UNCOMMITTED | Возможно | Возможно | Возможно |
| READ COMMITTED | Невозможно | Возможно | Возможно |
| REPEATABLE READ | Невозможно | Невозможно | Возможно* |
| SERIALIZABLE | Невозможно | Невозможно | Невозможно |

*По стандарту SQL фантомное чтение возможно при REPEATABLE READ, однако PostgreSQL и MySQL InnoDB реализуют этот уровень строже стандарта: PostgreSQL за счёт MVCC предотвращает фантомы, MySQL InnoDB использует gap-блокировки.

**Принцип выбора:** чем выше уровень изоляции, тем выше согласованность данных, но ниже конкурентность (пропускная способность). Рекомендуется выбирать наименьший уровень, достаточный для конкретного приложения.

### Вопрос 12. Что такое «грязное чтение» (dirty read), «неповторяющееся чтение» (non-repeatable read) и «фантомное чтение» (phantom read)? При каком уровне изоляции каждая из этих аномалий возможна?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 13.2. Transaction Isolation](https://www.postgresql.org/docs/current/transaction-iso.html)

Документация PostgreSQL определяет три аномалии параллельного доступа:

- **Dirty Read (Грязное чтение)** — транзакция читает данные, записанные параллельной незафиксированной транзакцией.
- **Nonrepeatable Read (Неповторяемое чтение)** — транзакция повторно читает данные, которые уже читала ранее, и обнаруживает, что они были изменены другой транзакцией (зафиксированной после первоначального чтения).
- **Phantom Read (Фантомное чтение)** — транзакция повторно выполняет запрос, возвращающий набор строк по некоему условию, и обнаруживает, что этот набор изменился из-за другой недавно зафиксированной транзакции.

Таблица аномалий по стандарту SQL (с учётом особенностей PostgreSQL):

| Уровень изоляции | Грязное чтение | Неповторяемое чтение | Фантомное чтение |
|---|---|---|---|
| Read Uncommitted | Допустимо (но не в PG) | Возможно | Возможно |
| Read Committed | Невозможно | Возможно | Возможно |
| Repeatable Read | Невозможно | Невозможно | Допустимо (но не в PG) |
| Serializable | Невозможно | Невозможно | Невозможно |

PostgreSQL в режиме Read Uncommitted ведёт себя как Read Committed; в режиме Repeatable Read дополнительно предотвращает фантомные чтения — это поведение строже, чем требует стандарт.

#### Ответ от Claude

**Грязное чтение (Dirty Read)**

Транзакция T1 читает строку, изменённую транзакцией T2, которая ещё не была зафиксирована. Если T2 откатится, T1 уже работала с данными, которые «никогда не существовали».

```
T1: BEGIN;
T2: BEGIN;
T2: UPDATE accounts SET balance = 1000 WHERE id = 1;  -- ещё не COMMIT
T1: SELECT balance FROM accounts WHERE id = 1;        -- видит 1000 (грязное чтение!)
T2: ROLLBACK;                                          -- изменение отменено
-- T1 работала с несуществующими данными
```

*Возможно при:* READ UNCOMMITTED.
*Исключается при:* READ COMMITTED, REPEATABLE READ, SERIALIZABLE.

---

**Неповторяемое чтение (Non-Repeatable Read)**

Транзакция T1 дважды читает одну и ту же строку и получает разные значения: между двумя чтениями T2 изменила эту строку и зафиксировала изменение.

```
T1: BEGIN;
T1: SELECT balance FROM accounts WHERE id = 1;   -- возвращает 500
T2: BEGIN;
T2: UPDATE accounts SET balance = 800 WHERE id = 1;
T2: COMMIT;
T1: SELECT balance FROM accounts WHERE id = 1;   -- возвращает 800 (другой результат!)
T1: COMMIT;
```

*Возможно при:* READ UNCOMMITTED, READ COMMITTED.
*Исключается при:* REPEATABLE READ, SERIALIZABLE.

---

**Фантомное чтение (Phantom Read)**

Транзакция T1 дважды выполняет запрос с одинаковым условием и получает разный *набор строк*: между двумя запросами T2 добавила или удалила строки, удовлетворяющие условию, и зафиксировала это.

```
T1: BEGIN;
T1: SELECT * FROM orders WHERE amount > 1000;    -- возвращает 3 строки
T2: BEGIN;
T2: INSERT INTO orders (amount) VALUES (1500);
T2: COMMIT;
T1: SELECT * FROM orders WHERE amount > 1000;    -- возвращает 4 строки (фантом!)
T1: COMMIT;
```

Отличие от неповторяемого чтения: меняется не значение в существующей строке, а состав набора строк (появляются «призраки» — новые строки).

*Возможно при:* READ UNCOMMITTED, READ COMMITTED, REPEATABLE READ (по стандарту SQL).
*Исключается при:* SERIALIZABLE.

---

**Итоговая таблица**

| Уровень изоляции | Грязное чтение | Неповторяемое чтение | Фантомное чтение |
|---|---|---|---|
| READ UNCOMMITTED | Возможно | Возможно | Возможно |
| READ COMMITTED | Невозможно | Возможно | Возможно |
| REPEATABLE READ | Невозможно | Невозможно | Возможно* |
| SERIALIZABLE | Невозможно | Невозможно | Невозможно |

*По стандарту SQL фантомное чтение допустимо при REPEATABLE READ, однако ряд СУБД предотвращает его на этом уровне: PostgreSQL и MySQL InnoDB используют MVCC/gap-блокировки и на практике исключают фантомы уже при REPEATABLE READ.

**Мнемоника:** каждый следующий уровень изоляции закрывает ещё одну аномалию:
- READ COMMITTED закрывает грязное чтение,
- REPEATABLE READ закрывает неповторяемое чтение,
- SERIALIZABLE закрывает фантомное чтение.
### Вопрос 13. Какова цена индекса? Какие накладные расходы он создаёт при операциях INSERT, UPDATE, DELETE?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Index Basics: Hidden Costs Associated With Indexes — SQL Nuggets](https://sqlnuggets.com/index-basics-hidden-costs-associated-indexes/)

Создание индекса означает, что СУБД сохраняет на диске постоянную копию данных столбцов, по которым построен индекс. Эта копия не временная — она хранится и поддерживается в актуальном состоянии всё время жизни индекса.

При операциях записи происходит умножение работы: если таблица имеет 5 некластерных индексов, то один INSERT превращается в 6 фактических вставок — по одной в таблицу и в каждый индекс. То же верно для DELETE. При UPDATE обновляются только те индексы, в которые входят изменяемые столбцы.

Со временем из-за модификаций данных индексы фрагментируются, что снижает их эффективность при чтении. Устранение фрагментации (REBUILD / REORGANIZE) — ресурсоёмкая операция, требующая окон обслуживания.

Неиспользуемые индексы не приносят пользы запросам, но продолжают потреблять место на диске, увеличивать размер резервных копий и замедлять операции записи.

#### Ответ от Claude

**Виды накладных расходов от индекса**

**1. Стоимость хранения**
Каждый индекс — это отдельная структура данных (как правило, B-дерево), хранимая на диске. Она содержит копию ключевых столбцов плюс указатели на строки таблицы (для некластерных индексов) или непосредственно данные (для кластерного). На таблицу с 10 индексами может приходиться в несколько раз больше места на диске, чем на саму таблицу.

**2. Overhead при INSERT**
Каждая новая строка должна быть вставлена не только в таблицу, но и во все индексы таблицы. При N некластерных индексах один `INSERT` порождает N+1 фактических операций записи. Кроме того, вставка может вызвать разбиение страниц B-дерева (page split), что дополнительно нагружает I/O.

**3. Overhead при DELETE**
Аналогично INSERT — запись удаляется из таблицы и из всех индексов. Физическое удаление из B-дерева также может требовать балансировки узлов.

**4. Overhead при UPDATE**
Если обновляемые столбцы входят в индекс, СУБД обязана удалить старую запись в индексе и вставить новую с обновлённым значением. Фактически UPDATE по индексированному столбцу — это DELETE + INSERT в индексе. Индексы, не содержащие изменяемый столбец, UPDATE не затрагивает.

**5. Фрагментация**
Частые INSERT/UPDATE/DELETE приводят к логической фрагментации индекса: страницы B-дерева заполняются неравномерно, логический порядок перестаёт совпадать с физическим. Фрагментированный индекс читается медленнее (больше I/O). Устранение требует регулярного обслуживания (REBUILD или REORGANIZE).

**6. Влияние на транзакции и блокировки**
При обновлении индексов удерживаются блокировки на страницах индексного дерева, что увеличивает вероятность конкуренции в высоконагруженных системах.

**Измеримые цифры (данные Percona для PostgreSQL)**
Бенчмарк Percona показал: увеличение числа индексов с 7 до 39 снизило пропускную способность до ~42% от исходной (с ~1400 до ~600 TPS), а среднюю задержку транзакции увеличило с 11 до 26 мс (+136%). Это в идеальных условиях; в реальных системах деградация сильнее.

**Вывод**
Индекс — это компромисс: ускорение чтения в обмен на замедление записи, дополнительное место и накладные расходы на обслуживание. Принцип: индекс должен быть оправдан реальными запросами; неиспользуемые индексы следует удалять.

### Вопрос 14. Чем кластерный (clustered) индекс отличается от некластерного (non-clustered)? Может ли таблица иметь несколько кластерных индексов?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Clustered vs Non-Clustered Index: Complete SQL Guide — DBvis](https://www.dbvis.com/thetable/clustered-vs-non-clustered-index-complete-sql-guide/)

**Кластерный индекс** определяет физический порядок хранения строк таблицы на диске: данные сортируются и хранятся в соответствии со значением ключевого столбца. В листовых узлах B-дерева находятся сами строки данных. Поскольку данные могут быть упорядочены только одним способом, таблица может иметь только один кластерный индекс.

**Некластерный индекс** — это отдельная структура, хранящая отсортированные значения ключа и указатели на строки таблицы. Он не изменяет физический порядок данных. Из-за дополнительного перехода по указателю некластерный индекс немного медленнее при поиске конкретной строки, но занимает меньше памяти в индексной структуре. Таблица может иметь множество некластерных индексов.

Таблица не может иметь несколько кластерных индексов, поскольку строки могут быть физически упорядочены лишь одним способом.

#### Ответ от Claude

**Структурное различие**

| Характеристика | Кластерный индекс | Некластерный индекс |
|---|---|---|
| Физический порядок данных | Определяет порядок строк на диске | Не влияет на порядок строк |
| Содержимое листовых узлов | Сами строки таблицы | Ключ + указатель (RID или кластерный ключ) |
| Количество на таблицу | Максимум 1 | Много (практический лимит: сотни) |
| Дополнительное хранение | Нет (данные уже в индексе) | Требует отдельного пространства |
| Поиск строки | Прямой — данные в листе | Двухэтапный — индекс → таблица (key lookup) |

**Кластерный индекс (Clustered Index)**
Данные таблицы физически упорядочены по значению ключа кластерного индекса. Листовой уровень B-дерева — это и есть страницы данных таблицы. Чтение по диапазону значений ключа происходит максимально эффективно (sequential I/O). В SQL Server PRIMARY KEY по умолчанию создаёт кластерный индекс.

**Некластерный индекс (Non-Clustered Index)**
Хранится отдельно от таблицы. Листовые узлы содержат ключ индекса + локатор строки. В SQL Server локатором служит либо RID (Row ID — для heap-таблиц), либо значение кластерного ключа (если таблица имеет кластерный индекс). Для получения полной строки требуется второй шаг — Key Lookup или RID Lookup, что добавляет I/O. Покрывающий индекс (INCLUDE-столбцы) позволяет избежать этого второго шага.

**Может ли таблица иметь несколько кластерных индексов?**
Нет. Данные физически могут быть упорядочены только одним способом, поэтому кластерный индекс может быть только один. Попытка создать второй кластерный индекс в SQL Server вызывает ошибку. Если таблица не имеет кластерного индекса, она называется heap (куча) — строки хранятся без упорядочивания.

**Особенность PostgreSQL**
PostgreSQL не имеет кластерных индексов в смысле SQL Server. Все таблицы хранятся как неупорядоченные heap-структуры. Команда `CLUSTER table USING index` физически переупорядочивает таблицу один раз, но после этого порядок не поддерживается автоматически при новых вставках. То есть в PostgreSQL «кластеризация» — одноразовая операция, а не постоянное свойство.

**Когда что использовать**
- Кластерный: запросы по диапазону (`BETWEEN`, `>=`, `<=`), часто используемый для сортировки столбец (дата, ID).
- Некластерный: точечные поиски по не-ключевым столбцам, покрывающие индексы для конкретных запросов, составные индексы для фильтрации по нескольким столбцам.

### Вопрос 15. В каких ситуациях денормализация базы данных может быть оправдана?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Denormalization: When, Why, and How — Red Gate Blog](https://www.red-gate.com/blog/denormalization-when-why-and-how/)

Денормализация — это намеренное добавление избыточных данных в нормализованную схему ради ускорения чтения и упрощения запросов. Она применяется тогда, когда реальные запросы, реальные пользователи и реальные SLA показывают, что JOIN-операции и вычисления на лету обходятся слишком дорого.

Статья выделяет четыре ситуации, когда денормализация оправдана:
1. **Хранение истории**: данные меняются со временем, и нужно сохранять значения, действовавшие в момент создания записи.
2. **Ускорение запросов**: когда запросы требуют JOIN нескольких таблиц для доступа к часто нужным данным — добавление избыточных атрибутов устраняет дорогостоящие операции.
3. **Ускорение отчётов**: предварительное вычисление статистики избавляет от повторного «перекапывания» всей базы.
4. **Предвычисление значений**: хранение уже рассчитанных производных значений вместо их вычисления при каждом запросе.

Каждое решение о денормализации необходимо документировать; изменения должны согласованно распространяться на все копии данных.

#### Ответ от Claude

**Что такое денормализация**
Денормализация — сознательное введение избыточности в схему (дублирование столбцов, предвычисленные значения, объединение таблиц) с целью ускорить чтение и упростить запросы. Это инструмент оптимизации, применяемый после того, как нормализованная схема не справляется с требованиями по производительности.

**Когда денормализация оправдана**

**1. OLAP, аналитика и отчётность (Data Warehouse)**
Нормализация оптимальна для OLTP (транзакционные системы с частыми мелкими записями). OLAP-нагрузка — редкие, но тяжёлые запросы с агрегацией по миллионам строк — выигрывает от денормализованных схем (звезда, снежинка). Хранилища данных (DWH) проектируются денормализованными изначально. Примеры: BI-дашборды, Hadoop/Spark пайплайны, OLAP-кубы.

**2. Высоконагруженные read-heavy системы**
Когда профиль нагрузки — многократно больше чтений, чем записей (каталог товаров, лента новостей, профили пользователей), JOIN нескольких таблиц при каждом запросе становится узким местом. Дублирование часто читаемых атрибутов (например, `category_name` рядом с `category_id`) устраняет этот JOIN.

**3. Хранение исторических данных**
Нормализованная схема хранит актуальное состояние: если цена товара изменится, все старые заказы будут ссылаться на новую цену. Денормализация — запись снапшота значения в момент события — решает эту проблему. Пример: поле `price_at_order_time` в таблице `order_items`.

**4. Предвычисленные агрегаты и счётчики**
Пересчёт `COUNT(likes)`, `SUM(revenue)`, `AVG(rating)` при каждом запросе дорог. Хранение этих значений в денормализованном поле (обновляемом триггером или фоновым процессом) резко снижает нагрузку. Используется в социальных сетях (`followers_count`), e-commerce (`total_reviews`).

**5. Сокращение числа JOIN при глубоких иерархиях**
Если схема имеет 4+ уровней master-detail, запрос «дай мне данные низшего уровня вместе с атрибутами высшего» требует многочисленных JOIN. Денормализация через «короткие ключи» (short-circuit keys) позволяет связать запись низшего уровня напрямую с высшим, минуя промежуточные таблицы.

**6. Кэширование и материализованные представления**
Материализованные представления (MATERIALIZED VIEW в PostgreSQL, Indexed View в SQL Server) — управляемая форма денормализации, поддерживаемая самой СУБД с периодическим обновлением.

**Условия и ограничения**
- Денормализовать следует только после того, как исчерпаны другие средства: правильные индексы, оптимизация запросов, устранение N+1.
- Необходимо измерить фактический bottleneck (профилировщик, EXPLAIN ANALYZE) — денормализация без данных о производительности приводит к усложнению схемы без выгоды.
- Каждое денормализованное поле — источник потенциальной рассинхронизации (data drift). Нужен механизм синхронизации: триггеры, транзакционные обновления, фоновые джобы или event-driven подход.
- Денормализация увеличивает сложность записи и повышает риск аномалий данных — это осознанный компромисс, а не бесплатный выигрыш.

**Итог**
Денормализация оправдана в OLAP/DWH по определению; в OLTP — только при доказанном bottleneck в read-нагрузке, при наличии чёткого плана поддержания согласованности данных.
### Вопрос 16. Что такое deadlock в контексте транзакций? Как его можно обнаружить и предотвратить?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation — Explicit Locking: Deadlocks](https://www.postgresql.org/docs/current/explicit-locking.html#LOCKING-DEADLOCKS)

Взаимоблокировка (deadlock) возникает, когда две (или более) транзакции удерживают блокировки, которые нужны друг другу, образуя цикл ожидания, из которого ни одна из транзакций не может выйти самостоятельно.

Пример из документации PostgreSQL: транзакция 1 обновляет строку с acctnum=11111, затем пытается обновить acctnum=22222; транзакция 2 в это же время обновляет acctnum=22222 и пытается обновить acctnum=11111. Обе транзакции ждут друг друга — deadlock.

**Обнаружение.** PostgreSQL автоматически обнаруживает взаимоблокировки и разрешает их, прерывая (ROLLBACK) одну из участвующих транзакций. Предсказать, какая транзакция станет «жертвой», нельзя и не следует на это полагаться. Текущие блокировки можно наблюдать через системное представление `pg_locks`.

**Предотвращение:**
1. Всегда захватывать блокировки на несколько объектов в одном и том же порядке во всех транзакциях.
2. При первой блокировке объекта сразу запрашивать наиболее строгий режим, который потребуется.
3. Если гарантировать порядок захвата заранее невозможно — реализовать на уровне приложения повторное выполнение (retry) транзакций, завершившихся из-за deadlock.
4. Избегать долго открытых транзакций (например, пока приложение ждёт ввода пользователя).

#### Ответ от Claude

**Что такое deadlock.**
Deadlock (взаимоблокировка, мёртвая блокировка) — ситуация, при которой две или более транзакции бесконечно ждут освобождения ресурсов, заблокированных друг другом. Классическая схема: транзакция A удерживает блокировку на ресурсе X и ожидает ресурс Y; транзакция B удерживает блокировку на ресурсе Y и ожидает ресурс X. Ни одна из них не может продолжить работу.

```
T1: LOCK X → waiting for Y
T2: LOCK Y → waiting for X
      ↑________________↓
          deadlock
```

**Необходимые условия для возникновения deadlock (условия Коффмана):**
- Взаимное исключение — ресурс используется только одной транзакцией одновременно.
- Удерживание и ожидание — транзакция удерживает уже захваченные ресурсы, ожидая новых.
- Отсутствие принудительного изъятия — ресурс не может быть принудительно отобран.
- Циклическое ожидание — существует цикл транзакций, каждая из которых ждёт следующую.

**Обнаружение.**
Большинство СУБД (PostgreSQL, MySQL InnoDB, Oracle, MS SQL Server) реализуют автоматическое обнаружение deadlock через алгоритм поиска циклов в графе ожидания транзакций (wait-for graph). При обнаружении цикла СУБД выбирает «жертву» (обычно транзакцию с наименьшим объёмом выполненной работы или наиболее дешёвую для отката) и принудительно откатывает её. Остальные транзакции в цикле продолжают выполнение. В PostgreSQL граф обслуживается процессом lock manager; интервал проверки настраивается параметром `deadlock_timeout` (по умолчанию 1 секунда).

Инструменты мониторинга:
- PostgreSQL: `SELECT * FROM pg_locks;`, `pg_stat_activity`, журнал с `log_lock_waits = on`.
- MySQL: `SHOW ENGINE INNODB STATUS;` — секция `LATEST DETECTED DEADLOCK`.
- Приложение должно перехватывать соответствующие исключения (`SQLTransactionRollbackException`, код `40P01` в PostgreSQL).

**Предотвращение.**
1. **Упорядочение ресурсов** — самый эффективный способ. Все транзакции должны захватывать блокировки на несколько строк/таблиц в одном и том же заранее оговорённом порядке (например, всегда по возрастанию primary key).
2. **Наиболее строгая блокировка с первой операции** — если транзакция заранее знает, что ей понадобится эксклюзивный доступ, следует запрашивать его сразу, а не эскалировать позже.
3. **Короткие транзакции** — чем быстрее транзакция освобождает блокировки, тем ниже вероятность пересечения с другой.
4. **SELECT FOR UPDATE / SELECT FOR SHARE** — явная блокировка строк ещё на этапе чтения позволяет предсказуемо удерживать ресурсы с самого начала транзакции.
5. **Повторное выполнение (retry)** — на уровне приложения отлавливать ошибки deadlock и повторно запускать транзакцию (с экспоненциальной задержкой или джиттером).
6. **Оптимистичная блокировка** — версионирование строк (поле `version` или `updated_at`) как альтернатива пессимистичным блокировкам там, где конфликты редки.

**Типичный Java-антипаттерн:**
```java
// Транзакция 1 (поток A)           // Транзакция 2 (поток B)
UPDATE accounts SET ... WHERE id=1; UPDATE accounts SET ... WHERE id=2;
UPDATE accounts SET ... WHERE id=2; UPDATE accounts SET ... WHERE id=1;
// ^^^ Разный порядок → возможен deadlock
```

**Исправление:** закрепить порядок — всегда обновлять строки по возрастанию id.

### Вопрос 17. Что такое SELF JOIN? Приведите пример задачи, где он необходим.

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [W3Schools — SQL Self Join](https://www.w3schools.com/sql/sql_join_self.asp)

SELF JOIN — это обычная операция JOIN, применяемая к одной и той же таблице. Таблица при этом соединяется сама с собой.

Синтаксис:
```sql
SELECT column_name(s)
FROM table1 T1, table1 T2
WHERE condition;
```

Здесь T1 и T2 — разные псевдонимы (aliases) одной и той же таблицы.

Пример: найти пары клиентов из одного города:
```sql
SELECT A.CustomerName AS CustomerName1,
       B.CustomerName AS CustomerName2,
       A.City
FROM Customers A, Customers B
WHERE A.CustomerID <> B.CustomerID
  AND A.City = B.City
ORDER BY A.City;
```

Условие `A.CustomerID <> B.CustomerID` исключает совпадение строки с самой собой.

#### Ответ от Claude

**Что такое SELF JOIN.**
SELF JOIN — это соединение таблицы с самой собой. Это не отдельный тип JOIN в SQL, а применение любого стандартного JOIN (INNER, LEFT, CROSS и т.д.) к одной таблице дважды. Чтобы различить два «экземпляра» таблицы, используются псевдонимы (table aliases). Физически таблица не копируется — СУБД просто дважды обращается к ней в рамках одного запроса.

**Синтаксис (современный ANSI SQL):**
```sql
SELECT a.column1, b.column2
FROM my_table AS a
JOIN my_table AS b ON a.some_column = b.other_column;
```

**Когда SELF JOIN необходим — классические задачи:**

1. **Иерархические структуры (сотрудник — менеджер).** Таблица `employees` содержит колонку `manager_id`, ссылающуюся на `employee_id` в той же таблице. Чтобы вывести рядом имя сотрудника и имя его руководителя:

```sql
SELECT e.employee_name AS employee,
       m.employee_name AS manager
FROM employees AS e
LEFT JOIN employees AS m ON e.manager_id = m.employee_id;
```

LEFT JOIN здесь важен: сотрудник без руководителя (CEO) тоже попадёт в результат со значением NULL в колонке manager.

2. **Поиск пар строк, удовлетворяющих условию.** Найти пары клиентов из одного города (пример W3Schools выше).

3. **Сравнение значений в соседних строках.** Найти все заказы, у которых сумма выше, чем у предыдущего заказа того же клиента (при наличии колонки `order_date`):

```sql
SELECT a.order_id, a.amount AS current_amount,
       b.amount AS previous_amount
FROM orders AS a
JOIN orders AS b
  ON a.customer_id = b.customer_id
 AND a.order_date > b.order_date;
```

4. **Категориальные иерархии (категория — подкатегория).** Таблица `categories` с колонкой `parent_id`.

5. **Граф/сеть:** Таблица `connections(user_id, friend_id)` — найти общих друзей двух пользователей.

**Важно при написании SELF JOIN:**
- Всегда задавать оба псевдонима — без них запрос не скомпилируется (неоднозначные ссылки на колонки).
- При поиске пар без дублей использовать условие `a.id < b.id` вместо `a.id <> b.id` — иначе пара (A, B) и (B, A) войдут оба раза.
- При работе с иерархиями глубже одного уровня рассматривать рекурсивные CTE (`WITH RECURSIVE`) как более гибкую альтернативу.

### Вопрос 18. В чём разница между Statement, PreparedStatement и CallableStatement в JDBC? Когда используется каждый из них?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Oracle Java Tutorials — Using Prepared Statements](https://docs.oracle.com/javase/tutorial/jdbc/basics/prepared.html)

`PreparedStatement` — особый вид SQL-объекта, который принимает предварительно скомпилированный SQL-запрос в момент создания, в отличие от обычного `Statement`. Главное преимущество — защита от SQL-инъекций: подготовленный запрос всегда рассматривает данные клиента как значение параметра, никогда как часть SQL-кода.

Создание и использование:
```java
String sql = "UPDATE coffees SET sales = ? WHERE cof_name = ?";
PreparedStatement pstmt = con.prepareStatement(sql);
pstmt.setInt(1, 100);
pstmt.setString(2, "French_Roast");
pstmt.executeUpdate();
```

Параметры заменяют `?`-заполнители через методы `setXXX()`. Индексация параметров начинается с 1. После первого выполнения запрос уже скомпилирован и при повторных вызовах выполняется быстрее. Незаменён там, где одно и то же выражение выполняется многократно с разными значениями.

#### Ответ от Claude

**Иерархия интерфейсов JDBC:**
```
java.sql.Statement
    └── java.sql.PreparedStatement
            └── java.sql.CallableStatement
```

Каждый следующий интерфейс расширяет предыдущий, добавляя возможности.

---

**Statement**

Используется для выполнения статических SQL-запросов без параметров.

```java
Statement stmt = conn.createStatement();
ResultSet rs = stmt.executeQuery("SELECT * FROM products");
```

Характеристики:
- SQL компилируется СУБД при каждом вызове `execute*()` — низкая производительность при повторных запросах.
- Нет поддержки параметров — значения конкатенируются в строку → риск SQL-инъекций.
- Подходит для DDL-запросов (`CREATE TABLE`, `DROP INDEX`) и разовых динамических запросов, строящихся целиком в коде без пользовательского ввода.

Когда использовать: DDL-операции при инициализации схемы; разовые административные запросы; случаи, когда SQL известен полностью на этапе компиляции и не содержит внешних данных.

---

**PreparedStatement**

Используется для параметризованных SQL-запросов, выполняемых многократно.

```java
String sql = "INSERT INTO orders (customer_id, amount) VALUES (?, ?)";
PreparedStatement pstmt = conn.prepareStatement(sql);

pstmt.setInt(1, customerId);
pstmt.setBigDecimal(2, amount);
pstmt.executeUpdate();

// Повторное использование с другими значениями:
pstmt.setInt(1, anotherCustomerId);
pstmt.setBigDecimal(2, anotherAmount);
pstmt.executeUpdate();
```

Характеристики:
- SQL компилируется (разбирается и оптимизируется) СУБД один раз при вызове `prepareStatement()` — последующие выполнения быстрее.
- Параметры передаются отдельно от SQL-текста → полная защита от SQL-инъекций.
- Поддерживает все три группы методов выполнения: `executeQuery()`, `executeUpdate()`, `execute()`.
- Позволяет добавлять запросы в batch: `addBatch()` / `executeBatch()`.

Когда использовать: любые DML-запросы (INSERT, UPDATE, DELETE, SELECT) с пользовательскими или внешними данными; запросы, выполняемые в цикле; массовые вставки через batch.

---

**CallableStatement**

Используется для вызова хранимых процедур и функций базы данных.

```java
// Синтаксис вызова процедуры
CallableStatement cstmt = conn.prepareCall("{call get_order_total(?, ?)}");

// IN-параметр
cstmt.setInt(1, orderId);

// OUT-параметр — обязательно регистрировать тип
cstmt.registerOutParameter(2, Types.DECIMAL);

cstmt.execute();

// Получить значение OUT-параметра
BigDecimal total = cstmt.getBigDecimal(2);
```

Типы параметров хранимых процедур:

| Тип | Описание | JDBC |
|-----|----------|------|
| IN | Входное значение | `setXXX(index, value)` |
| OUT | Выходное значение, возвращаемое процедурой | `registerOutParameter(index, type)` + `getXXX(index)` |
| INOUT | Принимает значение и возвращает (возможно изменённое) | `setXXX()` + `registerOutParameter()` |

Когда использовать: вызов хранимых процедур/функций СУБД; операции с OUT- и INOUT-параметрами; вызов курсоров, возвращаемых хранимыми процедурами Oracle/PostgreSQL.

**Сравнительная таблица:**

| Критерий | Statement | PreparedStatement | CallableStatement |
|---|---|---|---|
| Компиляция SQL | При каждом вызове | Один раз при создании | Один раз при создании |
| Параметры | Нет | `?`-заполнители | `?`-заполнители + IN/OUT/INOUT |
| Защита от SQL-инъекций | Нет | Да | Да |
| Производительность повторных вызовов | Низкая | Высокая | Высокая |
| Batch-операции | Да (addBatch) | Да (addBatch) | Ограниченно |
| Основное назначение | Статический SQL, DDL | Параметризованные DML/DQL | Хранимые процедуры |
| Наследует | — | Statement | PreparedStatement |

**Важный нюанс:** PreparedStatement даёт прирост производительности только если объект переиспользуется (или если используется пул PreparedStatement на уровне connection pool). Создание нового `PreparedStatement` на каждый вызов сводит преимущество к нулю — само по себе `prepareStatement()` стоит дороже `createStatement()`.
### Вопрос 19. Что такое пул соединений (connection pooling) в JDBC и зачем он нужен?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Chapter 8 Connection Pooling with Connector/J — docs.oracle.com](https://docs.oracle.com/cd/E17952_01/connector-j-en/connector-j-usagenotes-j2ee-concepts-connection-pooling.html)

Connection pooling — это техника создания и управления пулом соединений, готовых к использованию любым потоком, которому они потребуются. Большинство приложений нуждаются в соединении с базой данных только во время активной обработки транзакции, что зачастую занимает лишь миллисекунды. В остальное время соединение простаивает. Пул соединений позволяет использовать простаивающее соединение другим потоком для полезной работы.

Когда потоку нужно выполнить работу с БД, он запрашивает соединение из пула. Соединение выдаётся этому потоку эксклюзивно. По окончании работы поток возвращает соединение в пул для повторного использования. С точки зрения программиста это выглядит как обычный вызов `DriverManager.getConnection()`.

Основные преимущества:
1. **Сокращение времени установки соединения** — повторное использование уже установленных соединений устраняет накладные расходы на сетевое взаимодействие и инициализацию JDBC-драйвера.
2. **Упрощение модели программирования** — каждый поток работает со стандартным JDBC-интерфейсом, не задумываясь о деталях пула.
3. **Контроль потребления ресурсов** — предотвращает расточительное использование ресурсов и непредсказуемое поведение под высокой нагрузкой.

Критически важно всегда закрывать `Connection`, `Statement` и `ResultSet` в блоке `finally` (или через try-with-resources): закрытые объекты возвращаются в пул, незакрытые — приводят к утечке ресурсов (буферы, блокировки, сокеты) как на стороне клиента, так и на сервере БД.

#### Ответ от Claude

**Что такое пул соединений**

Установка нового соединения с базой данных — дорогая операция: TCP-handshake, аутентификация, инициализация сессии на стороне СУБД. При высокой нагрузке создание нового соединения на каждый запрос существенно замедляет приложение и перегружает БД. Пул соединений решает эту проблему, заблаговременно создавая набор соединений и переиспользуя их.

**Жизненный цикл соединения в пуле:**
1. При старте приложения пул создаёт N «тёплых» соединений (warm connections).
2. При необходимости приложение вызывает `dataSource.getConnection()` — пул выдаёт свободное соединение.
3. После завершения работы `connection.close()` не разрывает TCP-соединение, а возвращает его в пул.
4. Если все соединения заняты, новый запрос ждёт до `connectionTimeout`, после чего выбрасывается `SQLException`.

**Основные библиотеки пулов в Java:**

| Библиотека | Особенности |
|---|---|
| **HikariCP** | Самый быстрый, минималистичный (~165 KB), используется по умолчанию в Spring Boot |
| **Apache DBCP2** | Зрелый, входит в экосистему Apache Commons |
| **c3p0** | Один из первых пулов, сейчас редко выбирается для новых проектов |
| **Tomcat JDBC** | Встроен в Tomcat, хорошая интеграция с сервером приложений |

**Ключевые параметры конфигурации HikariCP:**

```java
HikariConfig config = new HikariConfig();
config.setJdbcUrl("jdbc:postgresql://localhost:5432/mydb");
config.setUsername("user");
config.setPassword("password");

// Максимальное число соединений в пуле (по умолчанию 10)
config.setMaximumPoolSize(10);

// Минимальное число простаивающих соединений
// (если не задан — пул фиксированного размера = maximumPoolSize)
config.setMinimumIdle(5);

// Максимальное время ожидания соединения из пула (мс), default 30 000
config.setConnectionTimeout(30_000);

// Максимальное время простоя соединения перед удалением из пула (мс), default 600 000
config.setIdleTimeout(600_000);

// Максимальное время жизни соединения (мс), default 1 800 000
// Рекомендуется устанавливать чуть меньше лимита на стороне БД
config.setMaxLifetime(1_800_000);

HikariDataSource dataSource = new HikariDataSource(config);
```

**Типичный паттерн использования:**

```java
// Connection.close() возвращает соединение в пул, не закрывает его физически
try (Connection conn = dataSource.getConnection();
     PreparedStatement ps = conn.prepareStatement("SELECT * FROM orders WHERE id = ?")) {
    ps.setLong(1, orderId);
    try (ResultSet rs = ps.executeQuery()) {
        // обработка результата
    }
}
```

**Рекомендация по размеру пула** (от авторов HikariCP): формула `connections = (core_count * 2) + effective_spindle_count` часто даёт лучшую пропускную способность, чем интуитивно большое число. Избыточное количество соединений увеличивает накладные расходы на переключение контекста в СУБД.

### Вопрос 20. Как управлять транзакциями вручную через JDBC (setAutoCommit, commit, rollback, Savepoint)?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Using Transactions — The Java™ Tutorials, docs.oracle.com](https://docs.oracle.com/javase/tutorial/jdbc/basics/transactions.html)

По умолчанию JDBC-соединение работает в режиме auto-commit: каждый SQL-оператор выполняется в отдельной транзакции и немедленно фиксируется. Чтобы объединить несколько операторов в одну транзакцию, нужно отключить этот режим:

```java
con.setAutoCommit(false);
```

После отключения изменения не фиксируются до явного вызова `commit()`. При ошибке вызывается `rollback()`, отменяющий все изменения текущей транзакции:

```java
public void updateCoffeeSales(HashMap<String, Integer> salesForWeek) throws SQLException {
    String updateString = "UPDATE COFFEES SET SALES = ? WHERE COF_NAME = ?";
    String updateStatement = "UPDATE COFFEES SET TOTAL = TOTAL + ? WHERE COF_NAME = ?";

    try (PreparedStatement updateSales = con.prepareStatement(updateString);
         PreparedStatement updateTotal = con.prepareStatement(updateStatement)) {

        con.setAutoCommit(false);
        for (Map.Entry<String, Integer> e : salesForWeek.entrySet()) {
            updateSales.setInt(1, e.getValue());
            updateSales.setString(2, e.getKey());
            updateSales.executeUpdate();

            updateTotal.setInt(1, e.getValue());
            updateTotal.setString(2, e.getKey());
            updateTotal.executeUpdate();
            con.commit();
        }
    } catch (SQLException e) {
        if (con != null) {
            con.rollback();
        }
    }
}
```

Для частичного отката внутри транзакции используются точки сохранения (Savepoint):

```java
con.setAutoCommit(false);
Savepoint save1 = con.setSavepoint();

updatePrice.executeUpdate();

if (newPrice > maximumPrice) {
    con.rollback(save1); // откат только до точки сохранения
}
con.commit();
```

Точки сохранения автоматически освобождаются при `commit()` или полном `rollback()`. Можно освободить их явно: `con.releaseSavepoint(save1)`.

#### Ответ от Claude

**Режим auto-commit и его отключение**

По умолчанию каждое JDBC-соединение работает в режиме `autoCommit = true`: каждый SQL-оператор автоматически оборачивается в транзакцию и сразу фиксируется. Чтобы управлять транзакцией вручную, нужно отключить этот режим **до** выполнения первого оператора транзакции.

```java
Connection conn = dataSource.getConnection();
conn.setAutoCommit(false); // переходим в ручное управление
```

**commit() — фиксация изменений**

`commit()` делает все изменения с момента последнего `commit()`/`rollback()` постоянными и освобождает удерживаемые БД-локи. Вызывается только при отключённом auto-commit.

**rollback() — полная отмена**

`rollback()` отменяет все изменения текущей транзакции и освобождает локи. Всегда вызывается в `catch`-блоке для обеспечения целостности данных.

**Полный шаблон управления транзакцией:**

```java
Connection conn = null;
try {
    conn = dataSource.getConnection();
    conn.setAutoCommit(false);

    // бизнес-логика: несколько SQL-операций
    try (PreparedStatement ps1 = conn.prepareStatement("UPDATE accounts SET balance = balance - ? WHERE id = ?");
         PreparedStatement ps2 = conn.prepareStatement("UPDATE accounts SET balance = balance + ? WHERE id = ?")) {

        ps1.setBigDecimal(1, amount);
        ps1.setLong(2, fromId);
        ps1.executeUpdate();

        ps2.setBigDecimal(1, amount);
        ps2.setLong(2, toId);
        ps2.executeUpdate();
    }

    conn.commit(); // всё прошло успешно — фиксируем

} catch (SQLException e) {
    if (conn != null) {
        try {
            conn.rollback(); // ошибка — откатываем
        } catch (SQLException ex) {
            // логируем исключение отката
        }
    }
    throw new RuntimeException("Транзакция отменена", e);
} finally {
    if (conn != null) {
        try {
            conn.setAutoCommit(true); // восстанавливаем состояние перед возвратом в пул
            conn.close();
        } catch (SQLException e) {
            // логируем
        }
    }
}
```

**Savepoint — частичный откат**

`Savepoint` позволяет откатить только часть транзакции, не отменяя всё. Это удобно при сложной бизнес-логике с необязательными шагами.

```java
conn.setAutoCommit(false);

// первый обязательный шаг
insertOrder(conn, order);

// создаём точку сохранения перед необязательным шагом
Savepoint spBeforePromo = conn.setSavepoint("beforePromo");

try {
    applyPromoDiscount(conn, order); // может упасть
} catch (SQLException e) {
    // откатываем только скидку, заказ остаётся
    conn.rollback(spBeforePromo);
}

// второй обязательный шаг
updateInventory(conn, order);

conn.commit(); // фиксируем всё (или всё без скидки, если откатили)
```

Важно: `conn.rollback(savepoint)` откатывает изменения только **до указанной точки**, но не завершает транзакцию — нужен последующий `commit()` или `rollback()`.

**Уровни изоляции транзакций** (устанавливаются через `conn.setTransactionIsolation()`):

| Константа | Грязное чтение | Неповторяемое чтение | Фантомное чтение |
|---|---|---|---|
| `TRANSACTION_READ_UNCOMMITTED` | возможно | возможно | возможно |
| `TRANSACTION_READ_COMMITTED` | защита | возможно | возможно |
| `TRANSACTION_REPEATABLE_READ` | защита | защита | возможно |
| `TRANSACTION_SERIALIZABLE` | защита | защита | защита |

Уровень изоляции должен устанавливаться **до** начала транзакции (до первого SQL-оператора после `setAutoCommit(false)`).
