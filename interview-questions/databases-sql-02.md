### Вопрос 21. Чем INNER JOIN отличается от LEFT JOIN, и что вернёт LEFT JOIN, если в правой таблице нет совпадений? Что произойдёт, если в правой таблице несколько строк подходят под условие JOIN?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Table Expressions — PostgreSQL Documentation](https://www.postgresql.org/docs/current/queries-table-expressions.html)

Согласно документации PostgreSQL:

**INNER JOIN:** для каждой строки R1 из таблицы T1 результирующий набор содержит строку для каждой строки T2, удовлетворяющей условию соединения с R1. Строки из T1, не имеющие совпадений в T2, **исключаются** из результата.

**LEFT OUTER JOIN:** сначала выполняется inner join. Затем для каждой строки T1, не удовлетворяющей условию соединения ни с одной строкой T2, в результат добавляется строка с NULL-значениями в столбцах T2. Таким образом, LEFT JOIN гарантирует **не менее одной строки** для каждой строки из T1.

**Умножение строк (row multiplication):** оба типа соединения умножают строки при наличии нескольких совпадений в правой таблице. Например, если T1 содержит 1 строку, а T2 — 3 совпадающие строки, и INNER JOIN, и LEFT JOIN вернут **3 строки**. Разница проявляется только тогда, когда строка из T1 не имеет совпадений в T2: INNER JOIN исключит её, LEFT JOIN включит с NULL-значениями.

#### Ответ от Claude

**INNER JOIN — только совпадения**

INNER JOIN возвращает строки, для которых найдено совпадение **в обеих таблицах** по условию ON. Строки без пары отбрасываются с обеих сторон.

```sql
-- Только заказы, у которых есть клиент, и только клиенты, у которых есть заказ
SELECT c.name, o.amount
FROM customers c
INNER JOIN orders o ON c.id = o.customer_id;
```

**LEFT JOIN (LEFT OUTER JOIN) — все строки левой таблицы**

LEFT JOIN возвращает **все** строки левой таблицы. Для строк, у которых нет совпадения в правой таблице, столбцы правой таблицы заполняются значениями `NULL`.

```sql
-- Все клиенты; у клиентов без заказов в столбцах orders будет NULL
SELECT c.name, o.amount
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id;
```

**Что вернёт LEFT JOIN при отсутствии совпадений в правой таблице:**

Строка из левой таблицы всё равно попадёт в результат, но все столбцы из правой таблицы будут равны `NULL`.

```
-- customers: {id=1, name='Alice'}, {id=2, name='Bob'}
-- orders: {id=10, customer_id=1, amount=500}

-- Результат LEFT JOIN:
-- name    | amount
-- Alice   | 500
-- Bob     | NULL   <-- Bob есть, но заказов у него нет
```

**Что произойдёт, если в правой таблице несколько строк подходят под условие JOIN:**

Происходит умножение строк (row multiplication). Каждая строка левой таблицы комбинируется с **каждой** подходящей строкой правой таблицы. Итоговое число строк для данного ключа: `N(левая) × N(правая)`.

```sql
-- customers: {id=1, name='Alice'}
-- orders: {id=10, customer_id=1, amount=100},
--         {id=11, customer_id=1, amount=200},
--         {id=12, customer_id=1, amount=300}

-- Результат LEFT JOIN:
-- name  | amount
-- Alice | 100
-- Alice | 200
-- Alice | 300   <-- три строки для одного клиента
```

Это поведение одинаково для INNER JOIN и LEFT JOIN. Принципиальная разница только в обработке строк без совпадений.

**Практические последствия умножения строк:**

1. **Агрегаты могут давать неверный результат.** Если затем делать `SUM(o.amount)`, получим сумму всех заказов — это ожидаемо. Но если в запросе есть ещё один JOIN с другой таблицей, строки могут умножиться повторно, и `SUM` посчитает одни и те же значения несколько раз.
2. **Дубликаты в SELECT.** При выборке столбцов только из левой таблицы одна и та же строка будет повторяться столько раз, сколько совпадений нашлось в правой.
3. **Решение через DISTINCT или агрегацию.** Если нужно знать только факт наличия совпадения — используйте `EXISTS`/`IN` вместо JOIN; если нужна агрегация — делайте её на подзапросе до JOIN.

**Сводная таблица различий:**

| Сценарий | INNER JOIN | LEFT JOIN |
|---|---|---|
| Совпадение найдено (1:1) | строка включается | строка включается |
| Совпадение не найдено (левая без пары) | строка исключается | строка включается, правые столбцы = NULL |
| Несколько совпадений в правой таблице | строки умножаются | строки умножаются |
| Строка только в правой таблице без пары | строка исключается | строка исключается |

### Вопрос 22. В чём разница между WHERE и HAVING? Можно ли использовать HAVING без GROUP BY? Что произойдёт, если написать WHERE на агрегатной функции?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 2.7. Aggregate Functions](https://www.postgresql.org/docs/current/tutorial-agg.html)

`WHERE` отбирает строки **до** вычисления групп и агрегатов (то есть управляет тем, какие строки попадут в агрегатную функцию), тогда как `HAVING` отбирает строки групп **после** вычисления групп и агрегатов.

В клаузе `WHERE` нельзя использовать агрегатные функции — не имеет смысла применять агрегат для определения того, какие строки станут входными данными для агрегата. Пример некорректного запроса:

```sql
SELECT city FROM weather
WHERE temp_lo = max(temp_lo);  -- НЕВЕРНО: нельзя использовать max() в WHERE
```

Правильный вариант с подзапросом:

```sql
SELECT city FROM weather
WHERE temp_lo = (SELECT max(temp_lo) FROM weather);
```

Использование `HAVING` без `GROUP BY` технически допустимо, но редко полезно. Та же самая условие может быть применено эффективнее на этапе `WHERE`. Использование фильтра в `WHERE` вместо `HAVING` предпочтительнее, когда агрегат не нужен, — это позволяет избежать лишних группировок и вычислений.

#### Ответ от Claude

**Разница между WHERE и HAVING**

| Аспект | WHERE | HAVING |
|--------|-------|--------|
| Когда применяется | До группировки и агрегации | После группировки и агрегации |
| На что фильтрует | Отдельные строки таблицы | Группы (результаты GROUP BY) |
| Агрегатные функции | Запрещены | Разрешены и являются основным назначением |
| Производительность | Эффективнее (меньше строк идёт в агрегацию) | Работает с уже сгруппированными данными |

**Порядок обработки SQL-запроса:**
```
FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY
```

Пример, иллюстрирующий разницу:

```sql
-- WHERE фильтрует строки ДО агрегации
SELECT department, COUNT(*) AS cnt
FROM employees
WHERE salary > 50000          -- сначала отсеиваем сотрудников с маленькой зарплатой
GROUP BY department
HAVING COUNT(*) > 5;          -- затем оставляем только крупные отделы
```

**Можно ли HAVING без GROUP BY?**

Да, технически можно. В этом случае вся таблица рассматривается как одна группа:

```sql
SELECT COUNT(*) FROM orders HAVING COUNT(*) > 100;
```

Это корректный запрос — он вернёт количество строк, если оно больше 100, или пустой результат. Однако это редкий и нецелесообразный паттерн: то же самое проще достигается подзапросом или вообще другой логикой. На практике `HAVING` без `GROUP BY` является признаком путаницы в коде.

**Что произойдёт, если использовать агрегат в WHERE?**

СУБД выдаст ошибку времени компиляции (до выполнения запроса). Примеры сообщений:

- PostgreSQL: `ERROR: aggregate functions are not allowed in WHERE`
- MySQL: `ERROR 1111 (HY000): Invalid use of group function`
- Oracle: `ORA-00934: group function is not allowed here`

Причина: `WHERE` обрабатывается до группировки, когда агрегаты ещё не вычислены. Это принципиальное ограничение языка SQL, а не особенность конкретной СУБД.

**Правильные альтернативы:**

```sql
-- Хочу найти товары с ценой выше средней:
-- НЕВЕРНО:
SELECT name FROM products WHERE price > AVG(price);

-- ВЕРНО вариант 1 — подзапрос:
SELECT name FROM products WHERE price > (SELECT AVG(price) FROM products);

-- ВЕРНО вариант 2 — если нужна фильтрация по агрегату группы:
SELECT category, AVG(price)
FROM products
GROUP BY category
HAVING AVG(price) > 1000;
```

### Вопрос 23. Что такое нормальные формы 1NF, 2NF, 3NF? Приведи пример таблицы, которая находится в 2NF, но нарушает 3NF — и как её исправить.

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Normal Forms in DBMS — GeeksforGeeks](https://www.geeksforgeeks.org/dbms/normal-forms-in-dbms/)

Нормальные формы — это последовательные правила для устранения избыточности и сохранения целостности базы данных.

**1NF** — наиболее базовый уровень нормализации: каждая ячейка таблицы должна содержать только атомарное (неделимое) значение, каждая строка уникальна, каждый столбец имеет уникальное имя. Пример нарушения: столбец с несколькими номерами телефонов в одной ячейке. Исправление: разбить на отдельные строки.

**2NF** — выполняются условия 1NF плюс отсутствие частичных зависимостей: каждый неключевой атрибут должен зависеть от **всего** первичного ключа, а не от его части. Пример нарушения: в таблице с составным ключом (StudentID, CourseID) поле StudentName зависит только от StudentID. Исправление: вынести StudentName в отдельную таблицу Students.

**3NF** — выполняются условия 2NF плюс отсутствие транзитивных зависимостей: неключевые атрибуты не должны зависеть от других неключевых атрибутов. Пример нарушения: в таблице записи (StudentID, CourseID, Instructor, InstructorDepartment) поле InstructorDepartment зависит от Instructor, а не напрямую от ключа. Исправление: вынести (Instructor, InstructorDepartment) в отдельную таблицу Instructors.

#### Ответ от Claude

**1NF (Первая нормальная форма)**

Требования:
1. Все значения атомарны (нет множественных значений в одной ячейке, нет вложенных таблиц).
2. Все строки уникальны (есть первичный ключ).
3. Порядок строк и столбцов не несёт смысловой нагрузки.

Пример нарушения 1NF:

| OrderID | CustomerName | Products |
|---------|-------------|---------|
| 1 | Иванов | Хлеб, Молоко, Яйца |

После приведения к 1NF:

| OrderID | CustomerName | Product |
|---------|-------------|---------|
| 1 | Иванов | Хлеб |
| 1 | Иванов | Молоко |
| 1 | Иванов | Яйца |

**2NF (Вторая нормальная форма)**

Требования: таблица в 1NF плюс отсутствие частичных зависимостей — каждый неключевой атрибут функционально зависит от **всего** составного первичного ключа, а не от его части.

Актуально только при составном первичном ключе. Если ключ простой (один столбец), таблица в 1NF автоматически в 2NF.

Пример нарушения 2NF — таблица `OrderItem`:

| OrderID (PK) | ProductID (PK) | Quantity | ProductName | ProductPrice |
|---|---|---|---|---|
| 1 | 101 | 3 | Хлеб | 50 |
| 1 | 102 | 1 | Молоко | 80 |

`ProductName` и `ProductPrice` зависят только от `ProductID`, а не от всего ключа `(OrderID, ProductID)` — частичная зависимость.

Исправление — декомпозиция:

**OrderItem:** (OrderID, ProductID, Quantity)

**Product:** (ProductID, ProductName, ProductPrice)

**3NF (Третья нормальная форма)**

Требования: таблица в 2NF плюс отсутствие транзитивных зависимостей — неключевой атрибут не должен зависеть от другого неключевого атрибута.

**Пример таблицы в 2NF, нарушающей 3NF:**

Таблица `Employee`:

| EmployeeID (PK) | Name | DepartmentID | DepartmentName | DepartmentBudget |
|---|---|---|---|---|
| 1 | Иванов | 10 | Разработка | 1000000 |
| 2 | Петров | 10 | Разработка | 1000000 |
| 3 | Сидоров | 20 | Маркетинг | 500000 |

Функциональные зависимости:
- `EmployeeID → DepartmentID` (прямая зависимость от ключа)
- `DepartmentID → DepartmentName` (зависимость неключевого от неключевого)
- `DepartmentID → DepartmentBudget` (то же самое)

Транзитивная зависимость: `EmployeeID → DepartmentID → DepartmentName, DepartmentBudget`

Проблемы:
- **Аномалия обновления**: если изменить бюджет отдела, придётся обновить несколько строк.
- **Аномалия удаления**: при увольнении всех сотрудников отдела потеряем данные об отделе.
- **Аномалия вставки**: нельзя добавить отдел без сотрудников.

**Исправление — декомпозиция до 3NF:**

**Employee:** (EmployeeID, Name, DepartmentID)

| EmployeeID | Name | DepartmentID |
|---|---|---|
| 1 | Иванов | 10 |
| 2 | Петров | 10 |
| 3 | Сидоров | 20 |

**Department:** (DepartmentID, DepartmentName, DepartmentBudget)

| DepartmentID | DepartmentName | DepartmentBudget |
|---|---|---|
| 10 | Разработка | 1000000 |
| 20 | Маркетинг | 500000 |

Теперь каждый неключевой атрибут зависит только от первичного ключа своей таблицы — транзитивная зависимость устранена.

### Вопрос 24. Чем BCNF отличается от 3NF? Приведи пример таблицы, которая в 3NF, но не в BCNF.

**Категория:** БД и SQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Boyce-Codd Normal Form (BCNF) — GeeksforGeeks](https://www.geeksforgeeks.org/dbms/boyce-codd-normal-form-bcnf/)

BCNF — это более строгая версия 3NF. Для каждой нетривиальной функциональной зависимости X → Y левая часть X должна быть суперключом.

Приводится пример таблицы `(Student, Teacher, Subject)` со следующими функциональными зависимостями:
- (Student, Teacher) → Subject
- (Student, Subject) → Teacher
- Teacher → Subject

Таблица находится в 3NF, так как нет транзитивных зависимостей. Однако она не в BCNF: в зависимости `Teacher → Subject` Teacher является детерминантом, но не является кандидатным ключом. Удаление записи о студенте может привести к потере информации о том, какой преподаватель ведёт какой предмет — аномалия удаления, которую BCNF предотвращает.

#### Ответ от Claude

**Формальное определение**

**3NF**: Для каждой нетривиальной функциональной зависимости X → Y выполняется хотя бы одно из условий:
1. X является суперключом, ИЛИ
2. Y является частью некоторого кандидатного ключа (праймовый атрибут).

**BCNF**: Для каждой нетривиальной функциональной зависимости X → Y выполняется условие:
1. X является суперключом (кандидатным ключом или его надмножеством).

BCNF не допускает «послабления» 3NF для праймовых атрибутов. Это делает BCNF строже.

**Ключевое различие одной фразой:** BCNF = 3NF, но без исключений для праймовых атрибутов.

**Пример: таблица в 3NF, но не в BCNF**

Классический пример — запись о курсах в университете.

Таблица `CourseEnrollment (Student, Course, Teacher)`:

| Student | Course | Teacher |
|---------|--------|---------|
| Иванов | Математика | Смирнов |
| Иванов | Физика | Орлов |
| Петров | Математика | Смирнов |
| Сидоров | Математика | Козлов |

Ограничения предметной области:
- Каждый студент по каждому курсу занимается у одного преподавателя.
- Каждый преподаватель ведёт только один курс.
- По одному курсу может быть несколько преподавателей.

Функциональные зависимости:
- `(Student, Course) → Teacher` — знаем студента и курс, знаем преподавателя.
- `(Student, Teacher) → Course` — знаем студента и преподавателя, знаем курс.
- `Teacher → Course` — преподаватель ведёт только один курс.

Кандидатные ключи: `(Student, Course)` и `(Student, Teacher)`.

Все атрибуты праймовые (каждый входит в какой-либо кандидатный ключ).

**Почему в 3NF?**

Для каждой ФЗ нарушение 3NF отсутствует — 3NF нарушается только если: левая часть не суперключ И правая часть непраймовая. Здесь все атрибуты праймовые, значит второе условие невыполнимо. Таблица в 3NF.

**Почему не в BCNF?**

В зависимости `Teacher → Course`: Teacher не является суперключом (он не однозначно определяет Student). BCNF требует, чтобы левая часть каждой ФЗ была суперключом — это условие нарушено.

**Аномалии, которые возникают:**

Если Козлов перестаёт вести Математику и начинает вести Информатику — нужно обновить несколько строк. Если Козлов не ведёт ни одного студента — мы теряем информацию о том, что он ведёт Математику.

**Декомпозиция до BCNF:**

Таблица `TeacherCourse (Teacher, Course)`:

| Teacher | Course |
|---------|--------|
| Смирнов | Математика |
| Орлов | Физика |
| Козлов | Математика |

Таблица `StudentTeacher (Student, Teacher)`:

| Student | Teacher |
|---------|---------|
| Иванов | Смирнов |
| Иванов | Орлов |
| Петров | Смирнов |
| Сидоров | Козлов |

**Важный практический нюанс:** декомпозиция до BCNF не всегда сохраняет все функциональные зависимости. В данном примере ФЗ `(Student, Course) → Teacher` при декомпозиции теряется — её нельзя проверить без JOIN. Это известный компромисс: 3NF гарантирует сохранение всех ФЗ (dependency-preserving decomposition), BCNF этого не гарантирует. В реальных системах часто останавливаются на 3NF именно по этой причине.

**Сравнительная таблица:**

| Аспект | 3NF | BCNF |
|--------|-----|------|
| Строгость | Менее строгая | Более строгая |
| Праймовые атрибуты | Могут зависеть от не-суперключа | Не допускается |
| Сохранение ФЗ | Гарантировано | Не гарантировано |
| Остаточная избыточность | Возможна | Минимальна |
| Применимость | Рекомендуется на практике | Теоретически идеальна, сложнее реализовать |
### Вопрос 25. Как работает B-tree индекс внутри? Почему поиск по индексу O(log n), а полный скан таблицы O(n)? Почему индекс не помогает при запросе с LIKE '%value%'?

**Категория:** БД и SQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [B-trees and database indexes — PlanetScale](https://planetscale.com/blog/btrees-and-database-indexes)

B-tree организует данные иерархически через узлы (nodes) и дочерние указатели (child pointers). Архитектура включает три типа узлов: корневой узел (root node) — верхняя точка входа; внутренние узлы (internal nodes) — промежуточные организационные узлы; листовые узлы (leaf nodes) — нижний уровень, содержащий фактические значения.

Внутри каждого узла элементы хранятся в отсортированном порядке. Любой дочерний узел слева от ключа содержит только ключи, меньшие этого ключа; дочерние узлы справа содержат ключи, которые больше него.

Алгоритм поиска использует эту упорядоченность: начинается с корневого узла, проверяет наличие искомого ключа в текущем узле, определяет, куда логически попадает ключ, следует по соответствующему дочернему указателю вниз, и повторяет до нахождения или исчерпания дерева. Критически важно: «вам нужно посетить только один узел на каждом уровне дерева». Поскольку каждый узел может содержать сотни или тысячи ключей, глубина дерева растёт логарифмически с размером данных, что обеспечивает производительность O(log n).

При полном сканировании, напротив, требуется просмотреть каждый листовой узел с данными. Нельзя пропустить узлы без риска пропустить строки, поэтому необходимо последовательно пройти все n записей.

LIKE '%value%' не может использовать индекс из-за принципа упорядочивания: поиск по паттерну с ведущим символом подстановки требует проверки значений, которые не вписываются в порядок слева направо, используемый индексами. Без известного префикса не существует начальной точки — база данных не может сузить поиск без чтения каждой записи.

#### Ответ от Claude

**Внутренняя структура B-tree индекса**

B-tree (в СУБД чаще B+ tree) — это сбалансированное дерево поиска. Структура состоит из трёх типов узлов:

- **Корневой узел (root)** — единственный узел верхнего уровня, точка входа для любого поиска.
- **Внутренние узлы (internal nodes)** — содержат ключи и указатели на дочерние узлы. Ключи играют роль разделителей: левое поддерево содержит ключи меньше разделителя, правое — больше.
- **Листовые узлы (leaf nodes)** — содержат фактические значения ключей и указатели на строки таблицы (или сами строки в случае кластерного индекса). В B+ tree все листовые узлы связаны в двусвязный список, что обеспечивает эффективное сканирование диапазонов.

Все листовые узлы находятся на одной глубине — это свойство «сбалансированности» (balanced). Дерево автоматически перебалансируется при вставках и удалениях.

**Почему поиск O(log n)**

При поиске конкретного значения алгоритм начинает с корня и на каждом уровне делает единственное сравнение, чтобы выбрать нужный дочерний узел. Поскольку каждый узел может содержать от десятков до тысяч ключей (высокий коэффициент ветвления, обычно 100–1000 в реальных СУБД), дерево остаётся очень «широким» и «низким». Для таблицы с миллиардом строк глубина B+ tree составит лишь 3–4 уровня. Количество шагов = высота дерева = O(log_b(n)), где b — коэффициент ветвления. Это и есть O(log n).

**Почему полный скан таблицы O(n)**

При отсутствии индекса (или когда оптимизатор выбирает полный скан) СУБД обязана прочитать каждую строку таблицы по очереди — пропустить ни одну нельзя, иначе можно не найти подходящие строки. Количество операций прямо пропорционально n — числу строк.

**Почему LIKE '%value%' не использует индекс**

B-tree индекс использует отсортированный порядок для быстрого нахождения начальной точки диапазона. Запрос LIKE 'value%' работает, потому что у него есть известный префикс: индекс переходит к первой записи, начинающейся с 'value', и сканирует вперёд.

Запрос LIKE '%value%' начинается с символа подстановки, то есть искомая строка может встретиться в любом месте значения. Индекс упорядочен по первому символу, затем по второму и т.д. Нет способа вычислить начало диапазона — каждое значение в индексе потенциально может совпасть. Поэтому СУБД вынуждена выполнить полный скан индекса или таблицы.

Аналогично LIKE '%value' (только суффикс) тоже не использует обычный B-tree. Для поиска подстрок применяют специализированные структуры: полнотекстовые индексы (FULLTEXT в MySQL, tsvector в PostgreSQL) или триграммные индексы (pg_trgm в PostgreSQL).

### Вопрос 26. Что такое кластерный (clustered) и некластерный (non-clustered) индекс? Почему кластерный индекс может быть только один? Что происходит с некластерным индексом при использовании кластерного?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Clustered and Nonclustered Indexes — Microsoft Learn](https://learn.microsoft.com/en-us/sql/relational-databases/indexes/clustered-and-nonclustered-indexes-described?view=sql-server-ver17)

Кластерный индекс сортирует и хранит строки данных в таблице или представлении на основе значений его ключей. Это те самые столбцы, включённые в определение индекса. На одну таблицу может быть только один кластерный индекс, потому что строки данных могут храниться только в одном порядке. Единственный случай, когда строки данных в таблице хранятся в отсортированном порядке — это когда таблица содержит кластерный индекс. Если у таблицы нет кластерного индекса, её строки хранятся в неупорядоченной структуре, называемой кучей (heap).

Некластерный индекс имеет структуру, отдельную от строк данных. Некластерный индекс содержит значения ключей некластерного индекса, и каждая запись ключевого значения имеет указатель на строку данных, содержащую это значение. Указатель из строки индекса в некластерном индексе на строку данных называется указателем строки (row locator). Структура указателя строки зависит от того, хранятся ли страницы данных в куче или в кластерной таблице. Для кучи указатель строки — это указатель на строку. Для кластерной таблицы указателем строки является ключ кластерного индекса.

SQL Server автоматически создаёт кластерный индекс при определении ограничения PRIMARY KEY, если только кластерный индекс ещё не существует.

#### Ответ от Claude

**Кластерный индекс (Clustered Index)**

Кластерный индекс определяет физический порядок хранения строк таблицы на диске. Листовые узлы B+ tree кластерного индекса содержат не указатели на строки, а сами строки данных целиком. Иными словами, кластерный индекс — это сама таблица, организованная в виде B+ tree.

Пример: PRIMARY KEY на столбце `id` создаёт кластерный индекс. Строки физически хранятся на диске в порядке возрастания `id`.

**Некластерный индекс (Non-Clustered Index)**

Некластерный индекс — отдельная структура B+ tree, существующая параллельно с таблицей. Листовые узлы содержат только значения индексируемых столбцов и указатель (row locator) на фактическую строку. На одну таблицу можно создать множество некластерных индексов (в SQL Server — до 999).

**Почему кластерный индекс только один**

Кластерный индекс диктует физический порядок хранения строк на диске. Физически строки можно отсортировать только по одному критерию одновременно — данные не могут одновременно быть упорядочены по `id` и по `name`. Поэтому создание второго кластерного индекса физически невозможно.

**Что происходит с некластерным индексом при наличии кластерного**

Это ключевая деталь, которую часто упускают. Поведение row locator в некластерном индексе меняется в зависимости от наличия кластерного:

- **Таблица-куча (без кластерного индекса):** row locator в некластерном индексе — это RID (Row Identifier), физический адрес строки: номер файла + номер страницы + слот на странице.
- **Кластерная таблица (с кластерным индексом):** row locator в некластерном индексе — это значение ключа кластерного индекса. Для поиска строки СУБД сначала находит ключ в некластерном индексе, затем выполняет второй поиск по кластерному индексу (key lookup / bookmark lookup).

Это означает: поиск через некластерный индекс на кластерной таблице требует двух обходов B+ tree. Поэтому оптимизатор иногда предпочитает полный скан таблицы, если некластерный индекс слишком «широкий» по охвату строк.

Также важно: при добавлении кластерного индекса все некластерные индексы автоматически перестраиваются, потому что их row locator меняется с RID на ключ кластерного индекса.

**Особенности в разных СУБД**

- **SQL Server / MySQL InnoDB:** PRIMARY KEY = кластерный индекс по умолчанию.
- **PostgreSQL:** не использует термин «кластерный индекс» в этом смысле. Команда `CLUSTER` физически переупорядочивает таблицу один раз, но порядок не поддерживается автоматически. Все индексы PostgreSQL по сути некластерные (heap-based), а row locator — это всегда TID (tuple ID).

### Вопрос 27. Что такое покрывающий индекс (covering index)? Как он позволяет избежать обращения к таблице, и в каких случаях он неприменим?

**Категория:** БД и SQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Covering index: an index containing all queried columns — Use The Index, Luke](https://use-the-index-luke.com/sql/clustering/index-only-scan-covering-index)

Покрывающий индекс — это индекс, содержащий все столбцы, на которые ссылается SQL-запрос, из секций WHERE, SELECT и других. Это позволяет базе данных удовлетворить весь запрос, используя только индекс, полностью минуя обращение к базовой таблице.

Когда индекс является «покрывающим», база данных выполняет index-only scan. Вместо того чтобы получать записи индекса и затем обращаться за строками в таблицу, система читает всё необходимое непосредственно из листовых узлов индекса. Это исключает дорогостоящее чтение блоков таблицы.

Например, при индексе на `(subsidiary_id, eur_value)` запрос `SELECT SUM(eur_value) WHERE subsidiary_id = ?` не требует обращения к таблице, так как оба столбца — и фильтруемый, и агрегируемый — присутствуют в индексе.

Польза зависит от числа обращаемых строк и коэффициента кластеризации индекса. Агрегирующие запросы по многим строкам получают наибольший выигрыш. Добавление предикатов на столбцы, отсутствующие в индексе, вынуждает обращаться к таблице. Функциональные индексы (например, `UPPER(last_name)`) также препятствуют index-only scan при выборке оригинального столбца.

#### Ответ от Claude

**Что такое покрывающий индекс**

Покрывающий индекс — это обычный индекс, который содержит все столбцы, необходимые конкретному запросу: столбцы из SELECT, WHERE, JOIN и ORDER BY. «Покрывающим» — это не тип индекса, а роль, которую конкретный индекс играет для конкретного запроса. Тот же индекс может быть покрывающим для одного запроса и непокрывающим для другого.

**Как он позволяет избежать обращения к таблице**

В обычном сценарии поиск по некластерному индексу выполняется в два этапа:
1. Обход B+ tree индекса — нахождение подходящих ключей.
2. Для каждого найденного ключа — переход по row locator к таблице за недостающими столбцами (table lookup / heap fetch).

Второй шаг — дорогостоящий: это случайный ввод-вывод. При покрывающем индексе листовые узлы уже содержат все нужные данные, второй шаг полностью устраняется. СУБД выполняет «index-only scan».

В MySQL (InnoDB) это ещё более выгодно: вторичный некластерный индекс хранит значения индексных столбцов + ключ PRIMARY KEY. Если запрос запрашивает только эти поля, дополнительного обращения к кластерному индексу нет вообще.

Идентификатор в EXPLAIN (MySQL): `Using index` в столбце Extra — признак того, что СУБД использует покрывающий индекс.

**Техника INCLUDE (included columns)**

В SQL Server и PostgreSQL (с версии 11) можно добавлять в индекс «включённые столбцы» (included columns / INCLUDE), которые не входят в ключ индекса, но хранятся в листовых узлах. Это позволяет создавать покрывающие индексы, не раздувая ключ:

```sql
-- SQL Server / PostgreSQL
CREATE INDEX idx_covering ON orders (customer_id) INCLUDE (order_date, total_amount);
```

Запрос `SELECT order_date, total_amount FROM orders WHERE customer_id = 5` выполнится как index-only scan.

**Когда покрывающий индекс неприменим или малоэффективен**

1. **Запрос обращается к столбцам вне индекса.** Если SELECT включает хотя бы один столбец, которого нет в индексе (и не добавлен через INCLUDE), СУБД вынуждена обратиться к таблице.

2. **Слишком широкий индекс.** Включение большого числа столбцов увеличивает размер индекса, замедляет вставки/обновления и может привести к тому, что индекс перестанет помещаться в память.

3. **Запрос возвращает большую долю строк.** Если запрос возвращает 20–30% и более строк таблицы, оптимизатор часто предпочтёт полный скан таблицы — последовательное чтение эффективнее множества обращений к индексу.

4. **Функции над индексируемыми столбцами.** `WHERE UPPER(name) = 'IVAN'` не использует индекс на `name`. Нужен функциональный индекс на `UPPER(name)`, но тогда выборка `name` (оригинала) уже не покрывается.

5. **Частые обновления индексируемых столбцов.** Покрывающий индекс на часто изменяемых столбцах создаёт высокую нагрузку на запись.

6. **PostgreSQL: видимость строк (visibility map).** В PostgreSQL index-only scan требует, чтобы все страницы таблицы были отмечены как «all-visible» в карте видимости. Активные транзакции или таблицы с интенсивным UPDATE/DELETE могут принудить СУБД к обращению к heap для проверки видимости строки (т.н. heap fetch для проверки MVCC), сводя преимущество к нулю.
### Вопрос 28. Какова цена индекса? Почему большое количество индексов на часто обновляемой таблице замедляет INSERT/UPDATE/DELETE?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Indexes Can Hurt Performance: Exploring the Negative Effects and the Costs Involved (Percona Blog)](https://www.percona.com/blog/postgresql-indexes-can-hurt-you-negative-effects-and-the-costs-involved/)

Каждая DML-операция (INSERT, UPDATE, DELETE) обязана обновить все индексы таблицы. Если у таблицы пять индексов, один INSERT фактически превращается в шесть операций записи. Автор статьи перечисляет несколько измерений цены индекса:

- **Операционные накладные расходы на запись.** Каждое изменение строки требует соответствующего изменения в каждом B-дереве. Число операций записи растёт линейно с числом индексов.
- **Хранилище.** Индексы потребляют дисковое пространство, нередко сопоставимое с размером самой таблицы, а иногда превышающее его. Это увеличивает время создания резервных копий и реплик.
- **Раздувание индексов (bloat).** UPDATE и DELETE оставляют «мёртвые» указатели в страницах B-дерева. Autovacuum вынужден обходить все индексы в ходе очистки — при большом числе индексов он может работать часами.
- **Давление на память.** Страницы индексов должны находиться в shared_buffers/page cache. Чем больше индексов, тем меньше памяти остаётся для страниц таблицы, что снижает эффективность кэша.
- **Усиление WAL.** Для каждого изменённого индексного блока PostgreSQL пишет запись в журнал опережающей записи (WAL), что умножает объём I/O при записи на диск.

Вывод автора: перед созданием индекса необходим анализ соотношения выгоды от чтения и стоимости записи для конкретной рабочей нагрузки.

#### Ответ от Claude

**Из чего складывается цена индекса**

Индекс — это вспомогательная структура данных (чаще всего B-дерево), которую СУБД поддерживает в актуальном состоянии синхронно с таблицей. Цена складывается из нескольких составляющих:

1. **Накладные расходы на запись (write amplification).** При каждом INSERT в таблицу БД должна вставить новую запись в каждый индекс. При UPDATE — если изменяется индексируемый столбец, старый ключ удаляется из B-дерева и вставляется новый. При DELETE — ключ помечается как удалённый во всех индексах. Итого: таблица с N индексами превращает одну логическую операцию записи в N+1 физических операций с B-деревьями.

2. **Случайный I/O.** Изменения в B-дереве — это случайные записи в различные страницы дерева, тогда как вставка в heap-таблицу нередко является последовательной (append). Случайный I/O на HDD значительно дороже последовательного; на SSD разрыв меньше, но всё равно существует.

3. **WAL-амплификация.** PostgreSQL фиксирует изменения индексных страниц в журнале опережающей записи (WAL). Чем больше индексов, тем объёмнее WAL — это замедляет fsync и репликацию.

4. **Раздувание (bloat) и затраты на VACUUM.** UPDATE в PostgreSQL создаёт новую версию строки (MVCC), старая версия становится «мёртвой». Все индексы содержат указатели на мёртвые версии, которые autovacuum должен обойти и зачистить. С ростом числа индексов фаза очистки индексов в VACUUM занимает всё больше времени и ресурсов.

5. **Давление на кэш.** Страницы индексов занимают место в shared_buffers (PostgreSQL) или buffer pool (MySQL). Это вытесняет «горячие» страницы таблицы из кэша, увеличивая число физических чтений при SELECT.

6. **Хранилище.** Индексы занимают дополнительное дисковое пространство — иногда больше, чем сама таблица. Это увеличивает время резервного копирования и восстановления.

**Почему это критично для часто обновляемых таблиц**

На read-heavy таблице эти расходы амортизируются многократно ускоренным чтением. На write-heavy таблице (очереди сообщений, журналы событий, таблицы логов) записей гораздо больше, чем чтений по индексируемым столбцам, и накладные расходы превышают выгоду. Практическое правило: каждый лишний индекс на горячей таблице — это скрытый налог на каждую операцию записи.

**Рекомендации**

- Регулярно выявлять и удалять неиспользуемые индексы (в PostgreSQL — через представление `pg_stat_user_indexes`, поле `idx_scan`).
- Частичные индексы (`WHERE condition`) покрывают только нужное подмножество строк и меньше раздуваются.
- Покрывающие индексы (`INCLUDE`) позволяют обойтись меньшим числом отдельных индексов.

### Вопрос 29. Расшифруй ACID. Что означает «изолированность» на практике — почему она реализована не как «полная изоляция», а через уровни?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ACID Transactions Explained: Atomicity, Consistency, Isolation & Durability (MotherDuck)](https://motherduck.com/learn/acid-transactions-sql/)

ACID — акроним из четырёх свойств транзакций, гарантирующих надёжность обработки данных:

- **Atomicity (Атомарность):** все операции транзакции либо выполняются полностью, либо не выполняются вовсе. Частичных изменений не остаётся.
- **Consistency (Согласованность):** транзакция переводит базу данных из одного корректного состояния в другое, соблюдая все ограничения и правила целостности.
- **Isolation (Изолированность):** параллельные транзакции не интерферируют друг с другом.
- **Durability (Долговечность):** зафиксированные (committed) изменения сохраняются даже после сбоя системы.

По вопросу изолированности: базы данных балансируют между согласованностью и производительностью. Полная изоляция везде создаёт узкие места при одновременном доступе нескольких транзакций к данным. Поэтому SQL-стандарт предоставляет уровни изоляции — READ UNCOMMITTED, READ COMMITTED, REPEATABLE READ и SERIALIZABLE — позволяя организациям выбирать нужное соотношение производительности и корректности.

#### Ответ от Claude

**Расшифровка ACID**

| Свойство | Суть |
|---|---|
| **A — Atomicity** (атомарность) | Транзакция неделима: либо все её операции зафиксированы, либо все откатываются. Реализуется через undo-журнал (rollback) или MVCC. |
| **C — Consistency** (согласованность) | Транзакция оставляет БД в валидном состоянии: не нарушаются ограничения (FOREIGN KEY, UNIQUE, CHECK), инварианты бизнес-логики. Ответственность разделена между СУБД (структурные ограничения) и приложением (бизнес-правила). |
| **I — Isolation** (изолированность) | Параллельно выполняющиеся транзакции не должны видеть промежуточные состояния друг друга. |
| **D — Durability** (долговечность) | Зафиксированная транзакция сохранена на диске и переживёт сбой питания или краш процесса. Реализуется через WAL/redo-журнал и fsync. |

**Что означает «изолированность» идеально**

В теории «полная изоляция» означает, что каждая транзакция выполняется так, как если бы она была единственной в системе — Serializable. Математически это означает, что результат любого набора параллельных транзакций эквивалентен некоторому последовательному (serial) порядку их выполнения.

**Почему полная изоляция дорого стоит**

Реализация полной сериализуемости требует одного из двух механизмов:

1. **Строгие блокировки (Two-Phase Locking, 2PL):** транзакция удерживает все захваченные блокировки до момента фиксации. Это практически означает, что читатели блокируют писателей и наоборот — пропускная способность системы падает, растёт вероятность дедлоков.

2. **Serializable Snapshot Isolation (SSI):** используется в PostgreSQL. Менее ограничительный, но всё равно требует отслеживания предикатных зависимостей и принудительного отката «виновных» транзакций при обнаружении аномалий (serialization anomaly). Это влечёт за собой дополнительный CPU-overhead и логику повтора в приложении.

На практике большинство OLTP-приложений не нуждаются в полной сериализуемости: например, чтение списка товаров в интернет-магазине спокойно переносит появление новой строки от параллельной транзакции. Платить ценой пропускной способности за защиту от аномалий, которые в конкретном сценарии невозможны или допустимы, нецелесообразно.

**Почему изолированность реализована через уровни**

SQL-стандарт (ISO/IEC 9075) формализовал компромисс, введя четыре уровня изоляции. Каждый уровень точно описывает, какие аномалии он допускает:

- **READ UNCOMMITTED** — минимальная защита, высокая конкурентность.
- **READ COMMITTED** — устраняет грязное чтение; достаточно для большинства OLTP-сценариев.
- **REPEATABLE READ** — устраняет дополнительно неповторяемое чтение.
- **SERIALIZABLE** — устраняет все аномалии; максимальная корректность, минимальная конкурентность.

Приложение выбирает уровень осознанно: транзакция формирования финансового отчёта требует SERIALIZABLE, а простая выборка для отображения пользователю — READ COMMITTED.

**Реализации изолированности в современных СУБД**

Большинство СУБД (PostgreSQL, MySQL InnoDB, Oracle) реализуют изоляцию через **MVCC (Multi-Version Concurrency Control)**: вместо блокировки строк при чтении хранятся версии строк с временными метками. Читатели видят «снимок» данных на момент начала транзакции (или команды), не мешая писателям. Это позволяет достичь уровня READ COMMITTED или REPEATABLE READ без блокировок при чтении — и именно поэтому READ COMMITTED является разумным и производительным значением по умолчанию.

### Вопрос 30. Перечисли уровни изоляции транзакций и соответствующие аномалии. При каком уровне возможны dirty read, non-repeatable read и phantom read? Какой уровень используется по умолчанию в PostgreSQL и MySQL InnoDB?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 13.2. Transaction Isolation](https://www.postgresql.org/docs/current/transaction-iso.html)

Официальная документация PostgreSQL описывает четыре уровня изоляции и аномалии следующим образом (таблица из документации):

| Уровень изоляции | Dirty Read | Non-repeatable Read | Phantom Read | Serialization Anomaly |
|---|---|---|---|---|
| Read Uncommitted | Допускается (в PG не происходит) | Возможно | Возможно | Возможно |
| Read Committed | Невозможно | Возможно | Возможно | Возможно |
| Repeatable Read | Невозможно | Невозможно | Допускается (в PG не происходит) | Возможно |
| Serializable | Невозможно | Невозможно | Невозможно | Невозможно |

Уровень по умолчанию в PostgreSQL — **Read Committed**. PostgreSQL реализует Read Uncommitted как идентичный Read Committed (из-за архитектуры MVCC грязное чтение физически невозможно). Repeatable Read в PostgreSQL также не допускает фантомного чтения, хотя стандарт SQL это допускает.

#### Ответ от Claude

**Три ключевые аномалии**

**Dirty Read (грязное чтение)** — транзакция T2 читает строку, изменённую транзакцией T1, которая ещё не зафиксирована. Если T1 откатится, T2 работала с несуществующими данными.

**Non-repeatable Read (неповторяемое чтение)** — транзакция T2 дважды читает одну и ту же строку и получает разные значения, потому что между двумя чтениями транзакция T1 зафиксировала изменение этой строки.

**Phantom Read (фантомное чтение)** — транзакция T2 дважды выполняет один и тот же запрос с условием (например, `WHERE balance > 1000`) и получает разные наборы строк, потому что T1 между двумя выполнениями вставила или удалила строки, удовлетворяющие условию.

**Сводная таблица уровней изоляции и аномалий**

| Уровень изоляции | Dirty Read | Non-repeatable Read | Phantom Read |
|---|---|---|---|
| **READ UNCOMMITTED** | Возможен | Возможно | Возможно |
| **READ COMMITTED** | Невозможен | Возможно | Возможно |
| **REPEATABLE READ** | Невозможен | Невозможно | Возможно* |
| **SERIALIZABLE** | Невозможен | Невозможно | Невозможно |

\* По стандарту SQL допускается. На практике PostgreSQL и MySQL InnoDB предотвращают фантомное чтение на уровне REPEATABLE READ — каждый своими средствами.

**При каком уровне возможны аномалии**

- **Dirty Read** — возможен только при READ UNCOMMITTED.
- **Non-repeatable Read** — возможен при READ UNCOMMITTED и READ COMMITTED.
- **Phantom Read** — возможен при READ UNCOMMITTED, READ COMMITTED и (по стандарту) REPEATABLE READ.

**Уровни по умолчанию**

- **PostgreSQL:** `READ COMMITTED`. Каждая команда внутри транзакции видит снимок данных на момент её начала (не начала транзакции). Это означает, что два последовательных SELECT в одной транзакции могут вернуть разные результаты (non-repeatable read допустим). Грязное чтение невозможно в принципе — MVCC не позволяет видеть незафиксированные версии строк даже при READ UNCOMMITTED.

- **MySQL InnoDB:** `REPEATABLE READ`. Первый SELECT в транзакции устанавливает «снимок» (consistent read snapshot), и все последующие SELECT в этой транзакции видят данные на тот же момент времени. Фантомное чтение дополнительно блокируется механизмом gap locks и next-key locks при использовании блокирующих операций (`SELECT FOR UPDATE`, `FOR SHARE`).

**Особенности реализаций**

- **PostgreSQL + REPEATABLE READ:** реализован через Snapshot Isolation. Фантомных чтений нет, потому что транзакция видит строго одну и ту же версию снимка. При обнаружении конфликта сериализации транзакция завершается ошибкой `ERROR: could not serialize access due to concurrent update` — приложение должно повторить её.

- **PostgreSQL + SERIALIZABLE:** использует Serializable Snapshot Isolation (SSI) с предикатными блокировками. Предотвращает все аномалии, включая write-skew и serialization anomaly. Требует логики повтора транзакции в приложении.

- **MySQL InnoDB + REPEATABLE READ:** при обычных (non-locking) SELECT фантомное чтение практически невозможно за счёт consistent read snapshot. При `SELECT FOR UPDATE` / DML-операциях применяются gap locks и next-key locks, исключающие вставку строк в «зазоры» диапазона.

- **READ UNCOMMITTED** — практически не используется в продакшне, так как грязное чтение делает данные ненадёжными. В PostgreSQL этот уровень ведёт себя идентично READ COMMITTED.
### Вопрос 31. Почему в MySQL InnoDB на уровне REPEATABLE READ фантомные чтения почти невозможны, хотя по стандарту SQL они допустимы на этом уровне? Что такое gap lock и next-key lock?

**Категория:** БД и SQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [MySQL 8.4 Reference Manual — 17.7.4 Phantom Rows](https://dev.mysql.com/doc/refman/8.4/en/innodb-next-key-locking.html)

Проблема фантомного чтения возникает, когда один и тот же запрос в рамках транзакции возвращает разные наборы строк при разных выполнениях. Например, если `SELECT` выполняется дважды, но второй раз возвращает строку, которой не было при первом вызове — это «фантомная строка».

Чтобы предотвратить фантомы, InnoDB использует алгоритм **next-key locking** — комбинацию блокировки индексной записи и блокировки промежутка (gap lock) перед ней. При сканировании индекса InnoDB устанавливает next-key lock на каждую встреченную запись, тем самым запрещая другим сессиям вставку новых строк в отсканированные диапазоны. Отключение gap lock (например, при `READ COMMITTED`) приводит к возможности фантомных чтений.

#### Ответ от Claude

**Почему SQL-стандарт допускает фантомы на REPEATABLE READ, а InnoDB — нет**

Стандарт SQL (ISO/IEC 9075) определяет REPEATABLE READ как уровень, при котором повторное чтение одной строки гарантированно возвращает те же данные, однако не запрещает появление новых строк, вставленных другими транзакциями (фантомы). Стандарт позволяет реализациям быть строже — именно этим воспользовалась InnoDB.

InnoDB устраняет фантомы двумя механизмами:

1. **Consistent read (снимок)** — обычные `SELECT` без `FOR UPDATE`/`FOR SHARE` читают снимок базы данных, зафиксированный в момент первой операции чтения в транзакции. Вновь вставленные строки в этот снимок не попадают, поэтому повторный `SELECT` вернёт идентичный результат.

2. **Next-key lock для блокирующих чтений** — `SELECT ... FOR UPDATE`, `UPDATE`, `DELETE` блокируют не только существующие строки, но и промежутки (gaps) между ними. Это не позволяет другим транзакциям вставить строки, которые попали бы в результат запроса.

---

**Gap lock (блокировка промежутка)**

Gap lock блокирует интервал между двумя соседними индексными значениями (или до первого/после последнего значения в индексе), не затрагивая сами записи. Цель — исключительно запретить `INSERT` в этот промежуток.

Ключевые свойства gap lock:
- «Чисто запретительный» (purely inhibitive) — не блокирует чтение и обновление существующих строк
- **Не конфликтует сам с собой**: два gap lock на один и тот же промежуток могут сосуществовать одновременно у разных транзакций
- Нет разницы между shared и exclusive gap lock — они выполняют одну и ту же функцию
- Отключается при уровне `READ COMMITTED`

Пример: запрос `SELECT c1 FROM t WHERE c1 BETWEEN 10 AND 20 FOR UPDATE` устанавливает gap lock на весь диапазон, не позволяя вставить значение 15, даже если такой строки ещё нет.

---

**Next-key lock**

Next-key lock — это комбинация record lock (блокировка конкретной индексной записи) + gap lock на промежуток, предшествующий этой записи. Именно next-key lock является стандартным режимом InnoDB при REPEATABLE READ.

Для индекса со значениями 10, 11, 13, 20 next-key locks покрывают следующие интервалы:
```
(-inf, 10]
(10, 11]
(11, 13]
(13, 20]
(20, +inf)
```

Квадратная скобка означает, что граница включена (record lock на саму запись); круглая — исключена (только gap).

**Исключение для уникальных индексов**: при поиске по уникальному индексу с единственным условием равенства InnoDB устанавливает только record lock, без gap lock — промежуток не нужно защищать, так как вставка дубликата всё равно будет отклонена ограничением уникальности.

---

**Итог**

| Механизм | Назначение | Когда применяется |
|---|---|---|
| Consistent read | Фантомы при обычном SELECT | REPEATABLE READ, неблокирующий SELECT |
| Gap lock | Запрет INSERT в промежуток | Блокирующие запросы, диапазонный поиск |
| Next-key lock | Gap lock + record lock | Стандартный режим InnoDB на RR |

### Вопрос 32. Что такое дедлок (deadlock)? Как СУБД его обнаруживает и разрешает? Какие паттерны написания транзакций помогают его предотвратить?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [MySQL 8.4 Reference Manual — 17.7.5.2 Deadlock Detection (+ InnoDB Data Locking Part 3)](https://dev.mysql.com/doc/refman/8.4/en/innodb-deadlock-detection.html)

InnoDB автоматически обнаруживает дедлоки транзакций и откатывает одну или несколько транзакций, чтобы разорвать цикл. Жертвой выбирается транзакция с наименьшим весом (определяется числом вставленных, обновлённых и удалённых строк). Обнаружение основано на поиске цикла в **wait-for graph** (граф ожидания): если транзакция A ждёт блокировку у B, а B — у A, граф содержит цикл, и зафиксирован дедлок. Начиная с MySQL 8.0.18 поиск цикла вынесен в отдельный фоновый поток и не блокирует рабочие потоки транзакций.

#### Ответ от Claude

**Что такое дедлок**

Дедлок (deadlock, взаимная блокировка) — ситуация, при которой две или более транзакции бесконечно ждут друг друга, потому что каждая удерживает ресурс, необходимый другой.

Классический пример:
```
Транзакция A: удерживает блокировку строки 1, ждёт блокировку строки 2
Транзакция B: удерживает блокировку строки 2, ждёт блокировку строки 1
```
Ни одна из них не может продолжить — дедлок.

---

**Обнаружение дедлока**

СУБД используют **граф ожидания (wait-for graph)**:
- **Узлы** — транзакции
- **Рёбра** — «транзакция A ждёт ресурс, заблокированный транзакцией B»

Дедлок = цикл в этом графе. СУБД периодически (или при каждом новом ожидании) проверяет граф на наличие цикла с помощью обхода в глубину (DFS).

**InnoDB (MySQL)**: начиная с версии 8.0.18 поиск цикла выполняется в отдельном фоновом потоке на разреженном (sparse) графе — только транзакции как узлы, по одному исходящему ребру на транзакцию. Это делает обнаружение быстрым и неблокирующим. Ограничения: не более 200 транзакций в wait-for list и не более 1 000 000 проверяемых блокировок — при превышении транзакция-инициатор проверки автоматически откатывается.

**PostgreSQL**: использует аналогичный механизм wait-for graph с периодической проверкой (по умолчанию каждые `deadlock_timeout` миллисекунд, по умолчанию 1 секунда).

---

**Разрешение дедлока: выбор жертвы**

Как только СУБД обнаруживает цикл, она должна «разорвать» его, откатив одну из транзакций. Критерии выбора жертвы:

- **InnoDB**: откатывается транзакция с наименьшим «весом» — минимальным числом изменённых строк (чтобы откат был как можно дешевле). Транзакции с высоким приоритетом (например, используемые Group Replication) защищены от выбора жертвой.
- **PostgreSQL**: откатывается транзакция, обнаружившая дедлок (та, которая последней встала в ожидание).

Откатившаяся транзакция получает ошибку (`ERROR 1213: Deadlock found` в MySQL, `ERROR 40P01` в PostgreSQL). Приложение обязано поймать эту ошибку и повторить транзакцию.

---

**Паттерны предотвращения дедлоков**

**1. Фиксированный порядок захвата блокировок (Lock Ordering)**

Самый эффективный метод. Все транзакции должны захватывать блокировки ресурсов в одном и том же заранее определённом порядке. Если A и B всегда блокируют таблицу X перед таблицей Y — цикл невозможен.

```sql
-- Неправильно: транзакция 1 блокирует A потом B, транзакция 2 — B потом A
-- Правильно: обе транзакции блокируют A потом B
BEGIN;
SELECT * FROM tableA WHERE id = 1 FOR UPDATE;  -- всегда первой
SELECT * FROM tableB WHERE id = 2 FOR UPDATE;  -- всегда второй
COMMIT;
```

**2. Минимизация времени удержания блокировок**

Чем короче транзакция, тем меньше окно для конфликта:
- Не делать тяжёлых вычислений или внешних вызовов внутри транзакции
- Читать данные до начала транзакции, если это возможно
- Откладывать начало транзакции как можно позже

**3. Явное упреждающее получение блокировки**

Использование `SELECT ... FOR UPDATE` в начале транзакции позволяет сразу заявить о намерении изменить строку, избегая перехода от shared lock к exclusive lock (lock upgrade — частая причина дедлоков).

```sql
BEGIN;
SELECT * FROM orders WHERE id = 42 FOR UPDATE;  -- сразу exclusive lock
UPDATE orders SET status = 'processing' WHERE id = 42;
COMMIT;
```

**4. Повторные попытки при дедлоке**

В приложении необходимо реализовать retry-логику с экспоненциальной задержкой:

```java
int maxRetries = 3;
for (int attempt = 0; attempt < maxRetries; attempt++) {
    try {
        executeTransaction();
        break;
    } catch (DeadlockLoserDataAccessException e) {
        if (attempt == maxRetries - 1) throw e;
        Thread.sleep(100 * (1 << attempt)); // exponential backoff
    }
}
```

**5. Снижение уровня изоляции (с осторожностью)**

Переход с `SERIALIZABLE` или `REPEATABLE READ` на `READ COMMITTED` уменьшает количество gap lock и next-key lock, что снижает вероятность дедлоков. Однако это может допустить аномалии чтения — решение применимо только если бизнес-логика его допускает.

**6. Индексирование WHERE-условий**

Запросы без индекса выполняют full scan и блокируют значительно больше строк (или всю таблицу). Корректные индексы сужают диапазон блокировок до минимально необходимого.

**Диагностика дедлоков в MySQL:**
```sql
SHOW ENGINE INNODB STATUS;  -- секция LATEST DETECTED DEADLOCK
```

### Вопрос 33. Чем первичный ключ (PRIMARY KEY) отличается от уникального индекса (UNIQUE INDEX)? Может ли PRIMARY KEY содержать NULL? Что такое составной первичный ключ и когда его лучше избегать?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 5.5. Constraints](https://www.postgresql.org/docs/current/ddl-constraints.html)

Первичный ключ означает, что столбец или группа столбцов может использоваться как уникальный идентификатор строк в таблице. Для этого значения должны быть одновременно уникальными и ненулевыми. Таким образом, следующие два определения таблицы принимают одни и те же данные:

```sql
CREATE TABLE products (
    product_no integer UNIQUE NOT NULL,
    name text
);
CREATE TABLE products (
    product_no integer PRIMARY KEY,
    name text
);
```

Первичные ключи могут включать несколько столбцов. Таблица может иметь не более одного первичного ключа, но может иметь любое количество уникальных ограничений. Уникальное ограничение допускает NULL-значения: по умолчанию два NULL не считаются равными друг другу, поэтому в столбце с UNIQUE может присутствовать несколько строк с NULL. Это поведение можно изменить через `NULLS NOT DISTINCT`.

#### Ответ от Claude

**Ключевые отличия PRIMARY KEY от UNIQUE INDEX:**

| Характеристика | PRIMARY KEY | UNIQUE INDEX / UNIQUE |
|---|---|---|
| NULL в столбце | запрещён (неявно NOT NULL) | разрешён (несколько NULL допускаются, т.к. NULL != NULL) |
| Количество на таблицу | ровно один | неограниченно |
| Цель | уникальный идентификатор строки | гарантия отсутствия дублей в бизнес-атрибуте |
| Использование в FK | цель по умолчанию для REFERENCES | не является целью по умолчанию |
| Создаваемый индекс | уникальный B-tree (автоматически) | уникальный B-tree (автоматически) |

**Может ли PRIMARY KEY содержать NULL?**

Нет. Стандарт SQL и все основные СУБД (PostgreSQL, MySQL, Oracle, SQL Server) запрещают NULL в столбцах первичного ключа. При объявлении `PRIMARY KEY` СУБД автоматически добавляет ограничение `NOT NULL`. Это принципиально отличает его от `UNIQUE`, где NULL разрешён: поскольку NULL != NULL (неизвестное не равно неизвестному), несколько NULL в уникальном столбце не считаются дублями.

**Составной первичный ключ:**

Составной (composite) первичный ключ — первичный ключ из двух и более столбцов:

```sql
CREATE TABLE order_items (
    order_id   integer,
    product_no integer,
    quantity   integer,
    PRIMARY KEY (order_id, product_no)
);
```

Составной PK гарантирует уникальность только комбинации всех указанных столбцов.

**Когда составного первичного ключа лучше избегать:**

1. **Изменчивость бизнес-данных.** Если один из столбцов ключа может измениться (например, код товара, email), каскадные UPDATE по всем FK-ссылкам становятся дорогостоящими и рискованными.
2. **Использование в качестве FK.** Дочерние таблицы вынуждены тащить все столбцы составного ключа, что увеличивает размер таблиц и усложняет JOIN.
3. **ORM и фреймворки.** Hibernate и большинство ORM плохо работают с составными ключами: требуют отдельный `@IdClass` или `@EmbeddedId`, что усложняет код.
4. **Высоконагруженные вставки.** Широкий ключ замедляет вставку и обновление индекса.
5. **Репликация и шардирование.** Составные ключи могут создавать проблемы при partition pruning и репликации.

**Рекомендация:** в большинстве случаев предпочтительнее суррогатный ключ (автоинкрементный `id` или UUID) с отдельным `UNIQUE` ограничением на бизнес-поля. Составные PK оправданы в таблицах-связках (many-to-many), где нет иных подходящих FK-ссылок.

### Вопрос 34. Что такое внешний ключ (FOREIGN KEY)? Что произойдёт при удалении родительской записи с опциями ON DELETE CASCADE, SET NULL и RESTRICT? В каких ситуациях внешние ключи намеренно не ставят в продуктивных системах?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 5.5. Constraints](https://www.postgresql.org/docs/current/ddl-constraints.html)

Внешний ключ указывает, что значения в столбце (или группе столбцов) должны соответствовать значениям, присутствующим в некоторой строке другой таблицы. Это называется поддержанием ссылочной целостности между двумя связанными таблицами.

Поведение при удалении родительской строки определяется опцией ON DELETE:

- **RESTRICT** — предотвращает удаление ссылочной строки. Это более строгий вариант по сравнению с `NO ACTION`: он не позволяет откладывать проверку ограничения до конца транзакции.
- **CASCADE** — при удалении ссылочной строки зависимые строки удаляются автоматически.
- **SET NULL** — при удалении ссылочной строки в столбце внешнего ключа устанавливается NULL. Важно: столбец не должен быть объявлен как `NOT NULL`.

Выбор опции зависит от семантики: если дочерняя таблица представляет компонент родительской и не может существовать без неё — подходит `CASCADE`. Если объекты независимы — `RESTRICT` или `NO ACTION`.

#### Ответ от Claude

**Что такое FOREIGN KEY:**

FOREIGN KEY — ограничение ссылочной целостности, гарантирующее, что значение в столбце дочерней таблицы обязательно присутствует в ссылочном столбце родительской таблицы (или равно NULL при отсутствии NOT NULL). Ссылочный столбец родительской таблицы должен быть PRIMARY KEY или иметь UNIQUE ограничение.

```sql
CREATE TABLE orders (
    order_id   integer PRIMARY KEY,
    customer_id integer REFERENCES customers(id)
);
```

**Поведение при удалении родительской записи:**

| Опция | Что происходит | Когда применять |
|---|---|---|
| `ON DELETE RESTRICT` | Ошибка немедленно; удаление родителя блокируется | Объекты независимы; нельзя "потерять" дочерние записи |
| `ON DELETE CASCADE` | Дочерние строки удаляются автоматически вместе с родителем | Дочерние данные — компонент родителя (заказ и его позиции) |
| `ON DELETE SET NULL` | В столбце FK дочерней строки проставляется NULL | Связь необязательна; дочерняя запись может существовать без родителя |

**Дополнительные нюансы:**

- `NO ACTION` (поведение по умолчанию) — аналогично RESTRICT, но позволяет отложить проверку до конца транзакции (при `DEFERRABLE INITIALLY DEFERRED`).
- `SET DEFAULT` — проставляет дефолтное значение; после этого строка должна удовлетворять FK (т.е. дефолтное значение должно быть в родительской таблице).
- `RESTRICT` vs `NO ACTION`: принципиальное отличие — RESTRICT нельзя отложить даже в deferrable-транзакции, он срабатывает сразу.

**Когда внешние ключи намеренно не ставят в продуктивных системах:**

1. **Микросервисная архитектура.** Каждый сервис владеет своей БД; FK между разными базами технически невозможен. Ссылочная целостность обеспечивается на уровне приложения или через eventual consistency.

2. **Высокая нагрузка на запись.** При каждой INSERT/UPDATE/DELETE с FK СУБД выполняет дополнительные проверки (lookup в родительской таблице), что замедляет операции. На таблицах с миллионами вставок в секунду это становится узким местом.

3. **Массовая загрузка данных (bulk load / ETL).** При загрузке больших объёмов FK-проверки критически замедляют процесс. Стандартная практика: отключить FK на время загрузки (`DISABLE TRIGGER ALL` / `SET foreign_key_checks=0`), загрузить данные, включить и верифицировать.

4. **Шардирование и горизонтальное масштабирование.** Родительская и дочерняя строки могут находиться на разных шардах; СУБД не умеет проверять FK через границы шардов.

5. **Аналитические/хранилища данных (DWH).** В колоночных СУБД (ClickHouse, BigQuery) FK либо не поддерживаются, либо декларируются без принуждения (`NOT ENFORCED`) — только как метаданные для оптимизатора.

6. **Команды с legacy-кодом.** Некоторые команды сознательно жертвуют декларативными ограничениями ради простоты деплоя и миграций, перекладывая контроль на уровень приложения.

**Риски отказа от FK:** аномалии данных (осиротевшие строки), труднее отлаживать баги, нет автоматических каскадов. Решение — компенсирующие проверки на уровне приложения и регулярные reconciliation-скрипты.

### Вопрос 35. Чем Statement отличается от PreparedStatement в JDBC? Почему PreparedStatement защищает от SQL-инъекций на уровне протокола? Как именно параметры передаются на сервер СУБД?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Using Prepared Statements — The Java Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/jdbc/basics/prepared.html)

Главная особенность объекта PreparedStatement — в отличие от Statement, SQL-запрос передаётся ему в момент создания (а не выполнения). Преимущество в том, что в большинстве случаев этот SQL-запрос сразу отправляется в СУБД и предкомпилируется там. В результате при выполнении PreparedStatement СУБД может просто запустить уже скомпилированный план без повторного разбора SQL.

Параметры передаются через setter-методы (`setInt`, `setString` и т.д.) по номеру плейсхолдера `?`:

```java
String sql = "UPDATE coffees SET sales = ? WHERE cof_name = ?";
PreparedStatement pstmt = con.prepareStatement(sql);
pstmt.setInt(1, 100);
pstmt.setString(2, "Espresso");
pstmt.executeUpdate();
```

Ключевое свойство PreparedStatement: пользовательские данные всегда трактуются как содержимое параметра, а не как часть SQL-выражения — что предотвращает SQL-инъекции.

#### Ответ от Claude

**Отличия Statement от PreparedStatement:**

| Характеристика | Statement | PreparedStatement |
|---|---|---|
| Создание | `con.createStatement()` | `con.prepareStatement(sql)` — SQL передаётся сразу |
| Компиляция | При каждом `execute(sql)` | Один раз при `prepareStatement()` |
| Параметры | Конкатенация строк | Плейсхолдеры `?` + setter-методы |
| SQL-инъекции | Уязвим | Защищён на уровне протокола |
| Производительность (повторные вызовы) | Хуже (разбор каждый раз) | Лучше (план кэшируется) |
| Читаемость кода | Проще для одноразовых запросов | Требует предварительного создания объекта |

**Уязвимость Statement:**

```java
// ОПАСНО: злоумышленник может передать: ' OR '1'='1
String query = "SELECT * FROM users WHERE name = '" + userInput + "'";
statement.execute(query);
// Итоговый SQL: SELECT * FROM users WHERE name = '' OR '1'='1'
// -- вернёт все строки таблицы
```

**Почему PreparedStatement защищает на уровне протокола:**

Защита обеспечивается не экранированием спецсимволов, а **архитектурным разделением** SQL-структуры и данных:

1. **Фаза подготовки.** При вызове `prepareStatement(sql)` СУБД получает шаблон запроса с плейсхолдерами `?`, разбирает его, строит план выполнения и возвращает дескриптор (statement handle). На этом этапе структура запроса зафиксирована.

2. **Фаза выполнения.** При вызове `execute()` по протоколу (для PostgreSQL — Extended Query Protocol; для MySQL — Binary Protocol) драйвер JDBC отправляет параметры **отдельным сообщением** от SQL-шаблона — как типизированные бинарные значения, а не как текст. СУБД подставляет их в готовый план без повторного разбора.

3. **Невозможность инъекции.** Поскольку SQL-парсер уже отработал на шаге 1 и план зафиксирован, значение параметра, каким бы оно ни было (например, `'; DROP TABLE users; --`), воспринимается СУБД исключительно как строковое значение. Изменить семантику запроса через данные уже невозможно.

```java
// БЕЗОПАСНО: даже злоумышленный ввод не меняет структуру запроса
String sql = "SELECT * FROM users WHERE name = ?";
PreparedStatement pstmt = con.prepareStatement(sql);
pstmt.setString(1, userInput); // "' OR '1'='1" — просто строка-параметр
ResultSet rs = pstmt.executeQuery();
```

**Как параметры передаются на сервер (детали протокола):**

- **PostgreSQL (Extended Query Protocol):** клиент отправляет сообщение `Parse` (SQL с `?`), затем `Bind` (значения параметров), затем `Execute`. Параметры идут в сообщении `Bind` как бинарные или текстовые значения — отдельно от SQL.
- **MySQL (Binary Protocol):** `COM_STMT_PREPARE` отправляет SQL, `COM_STMT_EXECUTE` отправляет параметры в бинарном формате с типами.
- **Oracle:** использует `OCI_STMT_PREPARE` + `OCIBindByPos`, аналогичный принцип.

**Важные оговорки:**

- Защита работает только при использовании `?`-плейсхолдеров. Если динамически подставлять имена таблиц или столбцов через конкатенацию — PreparedStatement не защитит.
- CallableStatement (для хранимых процедур) расширяет PreparedStatement и наследует те же защитные свойства.
- OWASP рекомендует parameterized queries (PreparedStatement) как Defense Option #1 против SQL-инъекций.
### Вопрос 36. Что такое оконная функция (window function) в SQL? Чем она отличается от агрегатной функции с GROUP BY?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 3.5. Window Functions](https://www.postgresql.org/docs/current/tutorial-window.html)

Оконная функция выполняет вычисление над набором строк таблицы, связанных с текущей строкой. Это похоже на то, что умеет агрегатная функция, но, в отличие от обычных агрегатных функций, применение оконных функций **не группирует строки в одну выходную строку** — каждая строка сохраняет свою отдельную идентичность.

Для определения окна применяется предложение `OVER`. Например, запрос ниже показывает зарплату каждого сотрудника вместе со средней зарплатой по его отделу:

```sql
SELECT depname, empno, salary,
       avg(salary) OVER (PARTITION BY depname)
FROM empsalary;
```

Результат: каждая строка содержит данные конкретного сотрудника, а рядом — среднее по отделу (значение повторяется для каждого сотрудника отдела).

Ключевые части синтаксиса:
- `PARTITION BY` — делит строки на разделы (аналог GROUP BY, но без схлопывания строк).
- `ORDER BY` внутри `OVER` — задаёт порядок обработки строк внутри окна.

Оконные функции разрешены только в списке `SELECT` и в предложении `ORDER BY`; использование в `WHERE`, `GROUP BY` или `HAVING` запрещено, поскольку они выполняются уже после этих фаз.

#### Ответ от Claude

Оконная функция (window function) — это функция SQL, которая вычисляет результат для каждой строки на основе «окна» (набора связанных строк), не схлопывая строки в одну, в отличие от агрегатных функций с `GROUP BY`.

**Главное отличие от GROUP BY:**

| Критерий | Оконная функция | Агрегат с GROUP BY |
|---|---|---|
| Количество выходных строк | Равно числу входных строк | Одна строка на группу |
| Сохранность отдельных строк | Да, каждая строка остаётся | Нет, строки схлопываются |
| Доступ к исходным полям | Да, все колонки доступны | Нет после группировки |
| Синтаксис | Требует `OVER (...)` | Требует `GROUP BY` |

**Пример: средняя зарплата по отделу**

С `GROUP BY` — только агрегированный результат (одна строка на отдел):
```sql
SELECT department, AVG(salary)
FROM employee
GROUP BY department;
```

С оконной функцией — каждый сотрудник + среднее по его отделу в одной строке:
```sql
SELECT employee_name, salary,
       AVG(salary) OVER (PARTITION BY department) AS dept_avg
FROM employee;
```

**Когда использовать:**
- `GROUP BY` — когда нужна только агрегированная сводка по группам.
- Оконные функции — когда нужно одновременно видеть детали каждой строки и агрегат по группе, или когда нужны позиционные функции (`LAG`, `LEAD`, `ROW_NUMBER` и др.), которые недоступны в `GROUP BY`-запросах.

Источники: [PostgreSQL docs — tutorial-window](https://www.postgresql.org/docs/current/tutorial-window.html), [LearnSQL — Window Functions vs GROUP BY](https://learnsql.com/blog/sql-window-functions-vs-group-by/).

### Вопрос 37. В чём разница между ROW_NUMBER(), RANK() и DENSE_RANK()? Что вернёт каждая из функций, если несколько строк имеют одинаковое значение ORDER BY?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [LearnSQL.com — Overview of Ranking Functions in SQL](https://learnsql.com/blog/sql-rank-function-vs-row-number-vs-dense-rank/)

Все три функции — ранжирующие оконные функции, но по-разному обрабатывают ситуацию, когда несколько строк имеют одинаковое значение в `ORDER BY` (так называемые «связки», ties).

- **ROW_NUMBER()** — присваивает каждой строке уникальный последовательный номер, даже если значения одинаковы. Порядок строк с одинаковым значением не детерминирован (если не задан дополнительный критерий сортировки).
- **RANK()** — одинаковым значениям присваивает одинаковый ранг, после чего **пропускает** следующие номера. Если две строки получили ранг 2, следующая строка получит ранг 4 (ранг 3 пропущен).
- **DENSE_RANK()** — одинаковым значениям присваивает одинаковый ранг, но **не пропускает** следующие номера. Если две строки получили ранг 2, следующая строка получит ранг 3.

Пример с таблицей продаж, где два продавца имеют одинаковый результат ($11 000):

| Продавец | Продажи | ROW_NUMBER() | RANK() | DENSE_RANK() |
|---|---|---|---|---|
| Алиса | 15 000 | 1 | 1 | 1 |
| Борис | 11 000 | 2 | 2 | 2 |
| Виктор | 11 000 | 3 | 2 | 2 |
| Галина | 8 000 | 4 | 4 | 3 |

`RANK()` пропускает позицию 3 (Галина сразу на 4-м месте), `DENSE_RANK()` — нет (Галина на 3-м месте).

#### Ответ от Claude

Все три функции реализованы как оконные функции ранжирования и требуют `ORDER BY` внутри `OVER (...)`. Официальные описания из документации PostgreSQL 18:

- `row_number() → bigint` — возвращает номер текущей строки внутри раздела, начиная с 1. Всегда уникален.
- `rank() → bigint` — ранг текущей строки **с пропусками**; равен `row_number` первой строки в группе равных значений.
- `dense_rank() → bigint` — ранг текущей строки **без пропусков**; считает группы равных значений.

**Поведение при совпадающих значениях ORDER BY:**

Предположим, таблица с оценками:

```sql
SELECT student, score,
       ROW_NUMBER() OVER (ORDER BY score DESC) AS rn,
       RANK()       OVER (ORDER BY score DESC) AS rnk,
       DENSE_RANK() OVER (ORDER BY score DESC) AS dr
FROM scores;
```

| student | score | ROW_NUMBER | RANK | DENSE_RANK |
|---|---|---|---|---|
| Анна | 90 | 1 | 1 | 1 |
| Борис | 85 | 2 | 2 | 2 |
| Вера | 85 | 3 | 2 | 2 |
| Дмитрий | 70 | 4 | 4 | 3 |

**Итог:**
- `ROW_NUMBER`: всегда 1, 2, 3, 4 — без учёта совпадений.
- `RANK`: 1, 2, 2, **4** — ранг 3 пропущен, потому что два ряда заняли позицию 2.
- `DENSE_RANK`: 1, 2, 2, **3** — пропусков нет, следующий ранг всегда +1.

**Когда применять:**
- `ROW_NUMBER` — пагинация, уникальная идентификация строк, дедупликация (выбор одной из дублирующихся записей).
- `RANK` — спортивные таблицы, конкурсы, где пропуск позиции считается корректным (два серебра — нет бронзы).
- `DENSE_RANK` — рейтинги без «дыр», когда важна непрерывность позиций (категории товаров, уровни скидок).

Источники: [PostgreSQL docs — functions-window](https://www.postgresql.org/docs/current/functions-window.html), [LearnSQL — Ranking Functions Overview](https://learnsql.com/blog/sql-rank-function-vs-row-number-vs-dense-rank/).

### Вопрос 38. Для чего используются функции LAG() и LEAD()? Приведите пример запроса, где они заменяют SELF JOIN.

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [LearnSQL.com — The LAG Function and the LEAD Function in SQL](https://learnsql.com/blog/lead-and-lag-functions-in-sql/)

`LAG()` и `LEAD()` — позиционные оконные функции, которые позволяют обращаться к значению другой строки без использования команды `SELF JOIN`.

- `LAG()` возвращает значение из строки, стоящей **выше** текущей (из предыдущих строк по заданному порядку).
- `LEAD()` возвращает значение из строки, стоящей **ниже** текущей (из последующих строк).

Синтаксис:
```sql
LAG(expression [, offset [, default_value]]) OVER (ORDER BY columns)
LEAD(expression [, offset [, default_value]]) OVER (ORDER BY columns)
```

Аргумент `offset` (по умолчанию 1) задаёт, на сколько строк назад/вперёд смотреть. Если предыдущей/следующей строки нет, возвращается `default_value` (по умолчанию `NULL`).

Пример — сравнение продаж текущего года с предыдущим:
```sql
SELECT
   year,
   total_sale AS current_total_sale,
   LAG(total_sale) OVER (ORDER BY year) AS previous_total_sale,
   total_sale - LAG(total_sale) OVER (ORDER BY year) AS difference
FROM annual_sale;
```

Тот же результат через SELF JOIN был бы значительно сложнее: требовал бы явного соединения таблицы с самой собой по условию `a.year = b.year - 1`.

#### Ответ от Claude

**Назначение LAG() и LEAD():**

Эти функции относятся к классу позиционных (offset) оконных функций. Они нужны для сравнения текущей строки с соседними строками в том же наборе результатов — без дополнительных соединений таблиц.

Официальные сигнатуры (PostgreSQL 18):
- `lag(value, offset, default) → anycompatible` — значение из строки на `offset` строк **перед** текущей.
- `lead(value, offset, default) → anycompatible` — значение из строки на `offset` строк **после** текущей.

**Типичные сценарии использования:**
- Сравнение «месяц к месяцу» / «год к году» (динамика продаж, прибыль, трафик).
- Вычисление разницы между соседними записями (дельта, темп роста).
- Поиск «следующего события» в хронологическом ряду (следующий заказ клиента, следующий логин).

**Пример: замена SELF JOIN функцией LAG()**

Задача: для каждого месяца вычислить разницу продаж с предыдущим месяцем.

Вариант с SELF JOIN (громоздко, требует явного алиасинга и условия соединения):
```sql
SELECT
    a.month,
    a.sales AS current_sales,
    b.sales AS prev_sales,
    a.sales - b.sales AS difference
FROM monthly_sales a
LEFT JOIN monthly_sales b
    ON a.month = b.month + 1;
```

Вариант с LAG() (компактно, нет дублирования таблицы):
```sql
SELECT
    month,
    sales AS current_sales,
    LAG(sales) OVER (ORDER BY month) AS prev_sales,
    sales - LAG(sales) OVER (ORDER BY month) AS difference
FROM monthly_sales;
```

Оба запроса дают одинаковый результат. Для первой строки `prev_sales` будет `NULL` (нет предыдущей строки).

**Пример с LEAD() — предсказание следующего события:**
```sql
SELECT
    order_id,
    customer_id,
    order_date,
    LEAD(order_date) OVER (PARTITION BY customer_id ORDER BY order_date) AS next_order_date
FROM orders;
```

Здесь `PARTITION BY customer_id` обеспечивает, что для каждого клиента функция смотрит только на его собственные заказы. `LEAD` возвращает дату следующего заказа этого же клиента; для последнего заказа — `NULL`.

**Расхождений между источниками не обнаружено.** Поведение `LAG`/`LEAD` идентично описано в документации PostgreSQL и на LearnSQL.

Источники: [PostgreSQL docs — functions-window](https://www.postgresql.org/docs/current/functions-window.html), [LearnSQL — LAG and LEAD Functions](https://learnsql.com/blog/lead-and-lag-functions-in-sql/).

### Вопрос 39. Что такое фрейм окна (window frame) в оконных функциях? Объясните разницу между ROWS BETWEEN и RANGE BETWEEN, и как неявный фрейм по умолчанию зависит от наличия ORDER BY.

**Категория:** БД и SQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [14.20.3 Window Function Frame Specification — MySQL 8.0 Reference Manual](https://docs.oracle.com/cd/E17952_01/mysql-8.0-en/window-functions-frames.html)

Фрейм окна — это подмножество текущего раздела (partition), определяемое относительно текущей строки. Синтаксис фрейма:

```
frame_clause: frame_units frame_extent

frame_units: {ROWS | RANGE}

frame_extent: {frame_start | BETWEEN frame_start AND frame_end}

frame_start/frame_end:
    CURRENT ROW
  | UNBOUNDED PRECEDING
  | UNBOUNDED FOLLOWING
  | expr PRECEDING
  | expr FOLLOWING
```

**ROWS**: фрейм определяется физическими позициями строк. Смещение `expr PRECEDING` означает «expr строк перед текущей». При `ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING` в окно входят ровно три физические строки: предыдущая, текущая и следующая.

**RANGE**: фрейм определяется диапазоном значений по ключу сортировки. При `RANGE BETWEEN 10 PRECEDING AND 10 FOLLOWING` в окно входят все строки, чьё значение ORDER BY-выражения лежит в диапазоне `[current_value - 10, current_value + 10]`. Граница `CURRENT ROW` при RANGE означает «все строки-peers» — строки, имеющие то же значение ORDER BY, что и текущая.

**Фрейм по умолчанию** (когда предложение FRAME явно не задано):

- При наличии `ORDER BY`: `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` — окно от первой строки раздела до текущей строки включая всех peers.
- При отсутствии `ORDER BY`: `RANGE BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING` — всё окно целиком (все строки раздела являются peers).

Важно: добавление `ORDER BY` меняет фрейм по умолчанию, что может изменить результаты агрегатов (например, значения `SUM()` станут нарастающим итогом вместо общей суммы раздела).

Функции, учитывающие фрейм: `SUM()`, `AVG()`, `COUNT()`, `FIRST_VALUE()`, `LAST_VALUE()`, `NTH_VALUE()`. Функции, игнорирующие фрейм и всегда работающие с целым разделом: `RANK()`, `DENSE_RANK()`, `ROW_NUMBER()`, `NTILE()`, `PERCENT_RANK()`, `CUME_DIST()`, `LAG()`, `LEAD()`.

#### Ответ от Claude

**Фрейм окна** (window frame) — это динамическое подмножество строк внутри раздела (partition), вычисляемое для каждой строки отдельно. Именно фрейм определяет, какие строки участвуют в агрегирующей функции на каждом шаге.

**ROWS BETWEEN** работает с физическими позициями строк:

```sql
SUM(amount) OVER (
    ORDER BY order_date
    ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
)
```

Окно всегда включает ровно текущую строку и не более двух строк перед ней по порядку сортировки — независимо от значений. Если таблица содержит дубликаты дат, каждая строка обрабатывается отдельно.

**RANGE BETWEEN** работает с диапазоном значений ORDER BY-выражения:

```sql
SUM(amount) OVER (
    ORDER BY order_date
    RANGE BETWEEN INTERVAL 2 DAY PRECEDING AND CURRENT ROW
)
```

Здесь в окно войдут все строки, чья дата попадает в диапазон `[current_date - 2 days, current_date]`. Граница `CURRENT ROW` включает всех «peers» — строки с тем же значением ключа сортировки, что у текущей строки.

**Ключевое отличие при дубликатах.** Если в таблице есть несколько строк с одинаковым `order_date`, то:

- `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` для каждой строки добавляет ровно одну следующую строку — нарастающий итог растёт пошагово.
- `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` (это же фрейм по умолчанию при наличии ORDER BY) — для всех строк с одинаковой датой включит одинаковое множество peers, поэтому все они получат одно и то же значение агрегата.

**Фрейм по умолчанию:**

| Наличие ORDER BY | Неявный фрейм |
|---|---|
| `ORDER BY` есть | `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` |
| `ORDER BY` нет | `RANGE BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING` |

Практический вывод: для нарастающего итога без сюрпризов с дубликатами рекомендуется явно указывать `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`. Для скользящих окон по реальным диапазонам значений (например, «все события за последние 7 дней») используется `RANGE BETWEEN INTERVAL 7 DAY PRECEDING AND CURRENT ROW`.

**Замечание по производительности** (на основе SQL Server): RANGE всегда использует on-disk window spool, тогда как ROWS может обрабатываться в памяти — это важно учитывать на больших объёмах данных.

Источники для перекрёстной проверки: [MySQL 8.0 Docs](https://docs.oracle.com/cd/E17952_01/mysql-8.0-en/window-functions-frames.html), [Simple Talk / Red Gate — Window Functions Part 2](https://www.red-gate.com/simple-talk/databases/sql-server/learn/window-functions-in-sql-server-part-2-the-frame/), [jOOQ Docs — Window Frame](https://www.jooq.org/doc/latest/manual/sql-building/column-expressions/window-functions/window-frame/).

### Вопрос 40. Как с помощью оконных функций вычислить скользящее среднее за последние N строк и нарастающий итог (running total)?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [What a Moving Average Is and How to Compute it in SQL — LearnSQL.com](https://learnsql.com/blog/moving-average-in-sql/)

Скользящее среднее (moving average) — это среднее значение текущего элемента и нескольких предшествующих, вычисляемое в «скользящем» окне. Оно сглаживает данные и помогает выявлять тренды, убирая случайный шум.

Базовый синтаксис для скользящего среднего за последние N строк:

```sql
SELECT *,
  AVG(Price) OVER(
      ORDER BY Date
      ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
  ) AS moving_average
FROM stock_price;
```

Здесь `ROWS BETWEEN 2 PRECEDING AND CURRENT ROW` задаёт окно из трёх строк: текущей и двух предыдущих. Для первых строк, где предшественников меньше N, окно автоматически уменьшается до доступного количества строк.

Пример с несколькими периодами и партиционированием:

```sql
SELECT *,
  AVG(Price) OVER(ORDER BY Date
      ROWS BETWEEN 1 PRECEDING AND CURRENT ROW)  AS moving_avg_2day,
  AVG(Price) OVER(ORDER BY Date
      ROWS BETWEEN 29 PRECEDING AND CURRENT ROW) AS moving_avg_30day
FROM stock_price;
```

При анализе нескольких категорий добавляется `PARTITION BY`:

```sql
SELECT *,
  AVG(confirmed_day) OVER(
      PARTITION BY country
      ORDER BY date
      ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
  ) AS moving_avg_7day
FROM confirmed_covid;
```

#### Ответ от Claude

**Нарастающий итог (running total)** вычисляется функцией `SUM()` с фреймом от начала раздела до текущей строки:

```sql
SELECT
    order_id,
    order_date,
    amount,
    SUM(amount) OVER (
        ORDER BY order_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS running_total
FROM orders;
```

Важно указывать `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` явно. Без явного фрейма при наличии ORDER BY используется неявный `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` — при дублирующихся датах все строки с одинаковой датой получат одно и то же значение (сумму всей группы peers), а не пошаговый нарастающий итог.

С `PARTITION BY` — отдельный итог для каждого клиента:

```sql
SELECT
    customer_id,
    order_date,
    amount,
    SUM(amount) OVER (
        PARTITION BY customer_id
        ORDER BY order_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS customer_running_total
FROM orders;
```

**Скользящее среднее за последние N строк** — функция `AVG()` с ограниченным фреймом:

```sql
-- 3-дневное скользящее среднее (текущий + 2 предыдущих)
SELECT
    sale_date,
    revenue,
    AVG(revenue) OVER (
        ORDER BY sale_date
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    ) AS moving_avg_3day
FROM daily_sales;
```

Параметр `N PRECEDING` задаёт именно количество строк перед текущей, поэтому `ROWS BETWEEN N-1 PRECEDING AND CURRENT ROW` даёт окно шириной N строк.

Сравнение двух подходов:

| Задача | Фрейм | Пример |
|---|---|---|
| Нарастающий итог | `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` | `SUM(amount) OVER (ORDER BY date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)` |
| Скользящее среднее за N строк | `ROWS BETWEEN N-1 PRECEDING AND CURRENT ROW` | `AVG(val) OVER (ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW)` |

**Почему ROWS, а не RANGE для этих задач?** RANGE группирует строки-peers по значению ключа сортировки. Если несколько строк имеют одинаковую дату — RANGE включит их все в текущую границу, что нарушает логику «ровно N предшествующих записей». ROWS работает с физическими позициями и даёт предсказуемый результат.

Источники для перекрёстной проверки: [LearnSQL.com — Moving Average](https://learnsql.com/blog/moving-average-in-sql/), [MySQL 8.0 Docs — Window Frames](https://docs.oracle.com/cd/E17952_01/mysql-8.0-en/window-functions-frames.html), [EssentialSQL — Moving Average](https://www.essentialsql.com/calculate-a-moving-average-in-sql-using-a-windows-function/).
