### Вопрос 41. Что такое NTILE(), PERCENT_RANK() и CUME_DIST()? Приведите практический сценарий применения каждой из них.

**Категория:** БД и SQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [MySQL 8.4 Reference Manual — 14.20.1 Window Function Descriptions](https://dev.mysql.com/doc/refman/8.4/en/window-function-descriptions.html)

**NTILE(N)** разбивает строки раздела на N примерно равных групп (бакетов) и присваивает каждой строке номер её группы. N должно быть положительным целым числом (от 1 до 2^63). Требует `ORDER BY` для осмысленного распределения.

```sql
SELECT val,
  NTILE(2) OVER w AS ntile2,
  NTILE(4) OVER w AS ntile4
FROM numbers
WINDOW w AS (ORDER BY val);
```

При количестве строк, не делящемся на N ровно, первые бакеты будут содержать на одну строку больше.

**PERCENT_RANK()** возвращает относительный ранг строки в виде значения от 0 до 1. Формула: `(rank - 1) / (rows - 1)`. Первая строка всегда получает 0, последняя — 1. Строки с одинаковыми значениями ORDER BY делят одинаковый rank.

**CUME_DIST()** возвращает кумулятивное распределение: долю строк раздела, чьё значение меньше или равно текущей строке. Формула: `(количество строк <= current) / (total rows)`. Значения в диапазоне от 0 (исключительно) до 1 включительно.

Сравнение PERCENT_RANK и CUME_DIST на одних данных (val: 1,1,2,3,3,3,4,4,5):

```
| val | percent_rank | cume_dist |
|  1  |   0.0000     |  0.2222   |
|  1  |   0.0000     |  0.2222   |
|  2  |   0.2500     |  0.3333   |
|  3  |   0.3750     |  0.6667   |
|  3  |   0.3750     |  0.6667   |
|  3  |   0.3750     |  0.6667   |
|  4  |   0.7500     |  0.8889   |
|  4  |   0.7500     |  0.8889   |
|  5  |   1.0000     |  1.0000   |
```

Все три функции игнорируют настройки фрейма и всегда работают с целым разделом.

#### Ответ от Claude

Все три функции относятся к классу ранговых оконных функций и помогают определить относительное положение строки внутри упорядоченного набора данных. Фрейм окна на них не влияет — они всегда обрабатывают весь раздел.

**NTILE(N)**

Делит строки раздела на N равных (или максимально равных) бакетов и возвращает номер бакета (от 1 до N). Если число строк не делится на N без остатка, первые `(число строк mod N)` бакетов получают на одну строку больше.

Формула назначения: бакет = `CEIL(row_number * N / total_rows)`.

Практический сценарий — разбиение клиентов на квартили по объёму покупок для сегментации (A/B/C/D):

```sql
SELECT
    customer_id,
    total_spend,
    NTILE(4) OVER (ORDER BY total_spend DESC) AS spend_quartile
FROM customers;
-- quartile=1 — топ 25%, quartile=4 — нижние 25%
```

**PERCENT_RANK()**

Возвращает процентильный ранг строки: долю строк раздела, строго меньших текущей. Формула: `(RANK() - 1) / (COUNT(*) - 1)`. Диапазон: 0..1. Первая строка всегда равна 0.

Расхождение с CUME_DIST: PERCENT_RANK показывает долю строк, у которых значение **строго меньше** текущего (на основе RANK), а CUME_DIST — долю строк, у которых значение **меньше или равно** (включая peers).

Практический сценарий — определить, в каком перцентиле находится зарплата сотрудника относительно коллег в том же отделе:

```sql
SELECT
    employee_id,
    department,
    salary,
    PERCENT_RANK() OVER (
        PARTITION BY department
        ORDER BY salary
    ) AS salary_percentile
FROM employees;
-- 0.0 = минимальная зарплата в отделе, 1.0 = максимальная
```

**CUME_DIST()**

Возвращает кумулятивное распределение: долю строк раздела, значение которых меньше или равно текущему. Формула: `(количество строк с value <= current_value) / total_rows`. Диапазон: (0, 1]. Минимальное значение больше 0 (не исключительно), максимальное — ровно 1.

Практический сценарий — найти сотрудников с зарплатой в верхних 30% по отделу (CUME_DIST >= 0.7):

```sql
SELECT
    employee_id,
    department,
    salary,
    CUME_DIST() OVER (
        PARTITION BY department
        ORDER BY salary
    ) AS cum_dist
FROM employees
WHERE CUME_DIST() OVER (
    PARTITION BY department ORDER BY salary
) >= 0.70;
-- Выбирает верхние 30% по зарплате в каждом отделе
```

**Итоговое сравнение:**

| Функция | Формула | Диапазон | Практический вопрос |
|---|---|---|---|
| `NTILE(N)` | Деление на N бакетов | 1..N | «К какой группе/квартилю относится строка?» |
| `PERCENT_RANK()` | (rank-1)/(rows-1) | [0, 1] | «Сколько строк строго ниже текущей?» |
| `CUME_DIST()` | rows_le_current/total | (0, 1] | «Какова доля строк не выше текущей?» |

Расхождение между источниками: SQLServerCentral описывает CUME_DIST через «rank/count», что соответствует формуле с учётом ties; MySQL Reference Manual уточняет, что peers получают одинаковое значение CUME_DIST равное позиции последнего peer, делённой на total — оба описания математически эквивалентны.

Источники для перекрёстной проверки: [MySQL 8.4 Reference Manual](https://dev.mysql.com/doc/refman/8.4/en/window-function-descriptions.html), [SQLServerCentral — PERCENT_RANK vs CUME_DIST](https://www.sqlservercentral.com/articles/whats-the-difference-between-percent_rank-and-cume_dist), [PracticeWindowFunctions — Percentile & Distribution](https://www.practicewindowfunctions.com/learn/percentile_distribution).

### Вопрос 42. Что означает вывод команды EXPLAIN (или EXPLAIN ANALYZE в PostgreSQL)? Как читать план запроса: что такое Seq Scan, Index Scan и Bitmap Index Scan, и при каких условиях оптимизатор выбирает каждый из них?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 14.1. Using EXPLAIN](https://www.postgresql.org/docs/current/using-explain.html)

Команда EXPLAIN отображает план выполнения, который планировщик PostgreSQL строит для заданного запроса. Вывод представляет собой дерево узлов плана. Каждый узел описывается одной строкой формата:

```
Seq Scan on tenk1  (cost=0.00..445.00 rows=10000 width=244)
```

Числа в скобках означают (слева направо):

- **Startup cost** — оценочная стоимость до начала вывода первой строки (например, время сортировки в Sort-узле).
- **Total cost** — оценочная стоимость при условии, что узел выполняется до конца и возвращает все строки.
- **rows** — оценочное количество строк, которые вернёт узел.
- **width** — оценочная средняя ширина одной строки в байтах.

Стоимость измеряется в условных единицах, традиционно приравненных к числу операций чтения страниц с диска (параметр `seq_page_cost = 1.0`). Стоимость узла верхнего уровня включает стоимость всех дочерних узлов.

**EXPLAIN ANALYZE** отличается тем, что запрос реально выполняется, и к каждому узлу добавляется фактическая статистика:

```
Index Scan using fi on foo  (cost=0.00..5.98 rows=1 width=4)
                             (actual time=0.009..0.025 rows=1 loops=1)
```

Здесь `actual time` — реальное время в миллисекундах (startup..total), `rows` — реальное число строк, `loops` — сколько раз выполнялся узел. Поскольку ANALYZE реально изменяет данные при DML-запросах, такие запросы следует оборачивать в транзакцию с последующим ROLLBACK.

**Типы сканирования:**

**Seq Scan (Sequential Scan)** — полный перебор всех строк таблицы по порядку физического хранения. Оптимизатор выбирает этот метод, когда: таблица мала и накладные расходы на обращение к индексу превысят выигрыш; запрос не содержит условия WHERE; условие WHERE неселективно и возвращает значительную долю строк (приблизительно более 10%); подходящего индекса нет.

**Index Scan** — двухэтапный процесс: PostgreSQL обращается к индексу (B-дереву), получает ссылку на строку таблицы (heap pointer), затем читает саму строку из кучи (heap). Строки возвращаются в порядке обхода индекса, что может не совпадать с физическим порядком на диске, поэтому каждое обращение к куче — потенциально случайный ввод-вывод. Оптимизатор предпочитает Index Scan, когда запрос возвращает очень небольшое число строк (высокая селективность условия), например при поиске по первичному ключу или уникальному столбцу.

**Bitmap Index Scan + Bitmap Heap Scan** — гибридный подход, состоящий из двух узлов. Сначала Bitmap Index Scan обходит индекс и строит в памяти битовую карту страниц таблицы, в которых есть подходящие строки. Затем Bitmap Heap Scan использует битовую карту для последовательного (а не случайного) чтения этих страниц. Это устраняет главный недостаток Index Scan — хаотичный случайный доступ к диску. Оптимизатор выбирает Bitmap Scan, когда запрос возвращает умеренно большое число строк — слишком много для Index Scan, но недостаточно для оправдания полного Seq Scan. Дополнительное преимущество: несколько битовых карт от разных индексов можно объединить операциями AND/OR, что позволяет одновременно использовать несколько индексов для условий, соединённых через AND или OR.

#### Ответ от Claude

**Что такое EXPLAIN**

EXPLAIN показывает план выполнения запроса, выбранный планировщиком PostgreSQL, не выполняя сам запрос. Вывод — дерево узлов; нижние узлы читают данные из таблиц, верхние — агрегируют, сортируют, соединяют.

Формат каждого узла:

```
<Тип узла> on <таблица>  (cost=<startup>..<total> rows=<N> width=<B>)
```

| Поле | Значение |
|------|---------|
| startup cost | Стоимость до выдачи первой строки (0 для Seq Scan, больше — для Sort) |
| total cost | Суммарная стоимость; верхний узел включает всё поддерево |
| rows | Оценка числа выходных строк (не обязательно совпадает с числом прочитанных) |
| width | Средний размер строки в байтах |

Единица стоимости условна: `seq_page_cost = 1.0`, `random_page_cost = 4.0` (по умолчанию), `cpu_tuple_cost = 0.01`.

**EXPLAIN ANALYZE**

Реально выполняет запрос и добавляет фактические метрики: `actual time=<ms>..<ms>`, `rows=<N>`, `loops=<N>`. Большое расхождение между `rows` (оценка) и реальными `rows` означает устаревшую или некорректную статистику — решается `ANALYZE` на таблице.

**Три типа сканирования**

**Seq Scan.** Читает все страницы таблицы последовательно. Выбирается, если: таблица мала; условия неселективны (ожидается > ~10 % строк); подходящего индекса нет. Стоимость: `disk_pages * seq_page_cost + rows * cpu_tuple_cost`. Последовательный I/O дёшев, поэтому Seq Scan часто побеждает на небольших и средних таблицах с низкоселективными фильтрами.

**Index Scan.** Обходит B-дерево индекса и для каждого найденного указателя выполняет отдельную случайную операцию чтения из кучи. Выбирается при высокой селективности: точечный поиск по ключу, поиск по уникальному значению, LIMIT с небольшим значением. Стоимость растёт при увеличении числа строк из-за накладных расходов случайного I/O. Особый случай: **Index Only Scan** — когда все нужные колонки уже есть в индексе и обращение к куче не требуется.

**Bitmap Index Scan / Bitmap Heap Scan.** Двухфазный алгоритм:
1. `Bitmap Index Scan` — обходит один или несколько индексов и строит в памяти битовую карту страниц кучи, содержащих подходящие строки.
2. `Bitmap Heap Scan` — читает отмеченные страницы в порядке их физического расположения (как при Seq Scan), затем перепроверяет условие (`Recheck Cond`) при необходимости.

Выбирается при умеренной селективности — больше строк, чем оправдывает Index Scan, но меньше, чем требует Seq Scan. Главное преимущество: несколько индексов можно объединить через `BitmapAnd` / `BitmapOr`, что невозможно при обычном Index Scan.

**Сводная таблица выбора типа сканирования**

| Сценарий | Тип сканирования |
|----------|----------------|
| Таблица мала или условие неселективно | Seq Scan |
| Точечный поиск, поиск по PK/уникальному ключу | Index Scan |
| Несколько строк нужно получить по диапазону | Bitmap Index Scan |
| Несколько индексов для разных условий AND/OR | Bitmap Index Scan с BitmapAnd/BitmapOr |
| Все нужные данные в индексе | Index Only Scan |

Расхождения между источниками: официальная документация PostgreSQL не устанавливает жёсткий порог (10 %) для перехода между типами сканирования — это лишь эмпирическое правило; реальное решение оптимизатор принимает на основе оцениваемой стоимости, которая зависит от `random_page_cost`, объёма данных и статистики распределения.

### Вопрос 43. Какие алгоритмы соединения таблиц (join algorithms) может показать EXPLAIN? Опишите Nested Loop Join, Hash Join и Merge Join: когда оптимизатор предпочтёт каждый из них?

**Категория:** БД и SQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 51.5. Planner/Optimizer](https://www.postgresql.org/docs/current/planner-optimizer.html)

PostgreSQL реализует три алгоритма соединения:

**Nested Loop Join.** Правая (inner) таблица сканируется по одному разу для каждой строки левой (outer) таблицы. Этот метод прост в реализации, но при большом размере обеих таблиц может потребовать значительного времени. Особенно эффективен, когда для правой таблицы доступен индексный скан: значения из текущей строки левой таблицы используются как ключи для поиска по индексу.

**Merge Join.** Перед соединением каждая из таблиц сортируется по атрибутам соединения. Затем обе таблицы сканируются параллельно, совпадающие строки объединяются в результирующие строки. Преимущество: каждая таблица сканируется только один раз. Необходимая сортировка может быть получена либо явным шагом Sort, либо за счёт сканирования по индексу, упорядоченному по ключу соединения.

**Hash Join.** Правая таблица сканируется и загружается в хэш-таблицу, в которой атрибуты соединения используются как ключи. Затем левая таблица сканируется, и для каждой её строки значения атрибутов соединения применяются для поиска совпадений в хэш-таблице. Эффективен для таблиц, не имеющих пригодного упорядочения.

Оптимизатор генерирует планы для всех возможных вариантов соединения каждой пары таблиц и выбирает тот, оценочная стоимость которого наименьшая.

#### Ответ от Claude

**Три алгоритма JOIN в PostgreSQL**

В выводе EXPLAIN они отображаются как узлы `Nested Loop`, `Hash Join`, `Merge Join`.

**Nested Loop Join**

Принцип: для каждой строки внешней (outer) таблицы выполняется полный или индексный скан внутренней (inner) таблицы.

```
Nested Loop
  -> Seq Scan on orders          (outer, N строк)
  -> Index Scan on order_items   (inner, по каждой строке outer)
```

Сложность: O(N * M) в худшем случае (без индекса), O(N * log M) с индексом на inner-таблице.

Оптимизатор предпочитает Nested Loop, когда:
- Внешняя таблица очень мала (единицы или десятки строк).
- На внутренней таблице есть индекс по ключу соединения — тогда стоимость каждой итерации снижается до O(log M).
- Соединение по неравенству (`<`, `>`), при котором Hash Join и Merge Join неприменимы.
- Требуется вернуть первые строки как можно быстрее (LIMIT с небольшим значением), так как Nested Loop не требует предварительных вычислений.

**Hash Join**

Принцип: в первой фазе (build) правая (меньшая) таблица сканируется и загружается в хэш-таблицу по ключу соединения. Во второй фазе (probe) левая таблица сканируется, для каждой строки вычисляется хэш и ищется совпадение в таблице. Если хэш-таблица не помещается в `work_mem`, PostgreSQL использует батч-хэширование с записью части данных на диск (batch mode).

Сложность: O(N + M) при нахождении хэш-таблицы в памяти.

Оптимизатор предпочитает Hash Join, когда:
- Обе таблицы велики и нет подходящих индексов.
- Данные не отсортированы по ключу соединения.
- `work_mem` достаточен для хранения хэш-таблицы меньшей из сторон.
- Соединение по равенству (`=`); Hash Join неприменим для неравенств.

**Merge Join**

Принцип: обе стороны должны быть отсортированы по ключу соединения. После сортировки два отсортированных потока сканируются одновременно (как слияние в сортировке слиянием), совпадающие строки объединяются.

Сложность: O(N + M) при уже отсортированных данных; O(N log N + M log M) с учётом сортировки.

Оптимизатор предпочитает Merge Join, когда:
- Обе таблицы уже отсортированы по ключу соединения — например, обе читаются через Index Scan по тому же столбцу.
- Необходимая сортировка достигается бесплатно (индексный скан уже возвращает строки в нужном порядке).
- Размеры обеих таблиц велики и нет возможности построить хэш-таблицу в памяти.
- Соединение по равенству с поддержкой операторов сортировки (mergejoinable operators).

**Как оптимизатор принимает решение**

Оптимизатор PostgreSQL использует модель стоимости: для каждой пары таблиц он генерирует возможные варианты для всех трёх алгоритмов и выбирает план с наименьшей оценочной стоимостью. Стоимость определяется следующими факторами:

| Фактор | Влияние |
|--------|---------|
| Оценочное число строк каждой из сторон | Чем больше inner-таблица, тем дороже Nested Loop без индекса |
| Наличие индекса на inner-таблице | Удешевляет Nested Loop |
| Уже отсортированные данные | Удешевляет Merge Join |
| Доступный `work_mem` | Определяет, поместится ли хэш-таблица в память для Hash Join |
| Тип оператора JOIN | Неравенства допускают только Nested Loop |
| `random_page_cost` vs `seq_page_cost` | Влияет на сравнительную стоимость Index Scan внутри Nested Loop |

Главная причина неоптимального выбора — неверная оценка числа строк из-за устаревшей статистики. Если планировщик ожидает 15 строк, а реально возвращается 150 000, он выберет Nested Loop (оптимальный для 15 строк), хотя нужен Hash Join.

Принудительно отключить алгоритмы можно параметрами `enable_nestloop`, `enable_hashjoin`, `enable_mergejoin` (только для диагностики, не для продакшена).

### Вопрос 44. Что такое составной (composite) индекс? Объясните правило «левого префикса» (leftmost prefix rule) и как порядок колонок в составном индексе влияет на то, будет ли он использоваться запросом.

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [MySQL 8.4 Reference Manual: 10.3.6 Multiple-Column Indexes](https://dev.mysql.com/doc/refman/8.4/en/multiple-column-indexes.html)

Составной (многоколоночный) индекс строится по нескольким столбцам таблицы. MySQL может использовать многоколоночный индекс для запросов, проверяющих все колонки индекса, или только первую колонку, или первые две колонки и т.д. Условие: колонки должны образовывать **левый префикс** определения индекса.

Пример:

```sql
CREATE TABLE test (
    id         INT NOT NULL,
    last_name  CHAR(30) NOT NULL,
    first_name CHAR(30) NOT NULL,
    PRIMARY KEY (id),
    INDEX name (last_name, first_name)
);
```

Для индекса `(last_name, first_name)` индексный поиск возможен для:
- `(last_name)` — только первая колонка.
- `(last_name, first_name)` — обе колонки.

Запросы, которые используют индекс:

```sql
SELECT * FROM test WHERE last_name = 'Jones';
SELECT * FROM test WHERE last_name = 'Jones' AND first_name = 'John';
SELECT * FROM test WHERE last_name = 'Jones' AND first_name >= 'M' AND first_name < 'N';
```

Запросы, которые **не** используют индекс:

```sql
SELECT * FROM test WHERE first_name = 'John';
SELECT * FROM test WHERE last_name = 'Jones' OR first_name = 'John';
```

Для трёхколоночного индекса `(col1, col2, col3)` индексный поиск работает для `col1`; `col1, col2`; `col1, col2, col3`. Запросы только по `col2` или только по `col2, col3` индекс не используют.

#### Ответ от Claude

**Что такое составной индекс**

Составной (composite, multi-column, concatenated) индекс — это индекс, построенный по двум или более столбцам таблицы. Физически он реализуется как B-дерево, в котором ключи сортируются лексикографически: сначала по первому столбцу, при равенстве первого — по второму, при равенстве второго — по третьему и т.д. Аналогия: телефонная книга сортируется сначала по фамилии, затем по имени.

Пример создания:

```sql
CREATE INDEX idx_name ON employees (department_id, salary, hire_date);
```

**Правило левого префикса**

База данных может использовать составной индекс `(A, B, C)` только если условия запроса охватывают **непрерывную последовательность столбцов, начиная с самого левого**:

| Условие в WHERE | Использует индекс? | Почему |
|-----------------|-------------------|--------|
| `A = ?` | Да | Левый префикс (A) |
| `A = ? AND B = ?` | Да | Левый префикс (A, B) |
| `A = ? AND B = ? AND C = ?` | Да | Полный ключ |
| `B = ?` | Нет | Пропущен A — первая колонка |
| `B = ? AND C = ?` | Нет | Пропущен A |
| `A = ? AND C = ?` | Частично (только по A) | B пропущен; C не может использоваться как индекс |

**Почему это правило абсолютно**

B-дерево хранит ключи, отсортированные по конкатенации значений столбцов. Если пропустить первый столбец и искать только по второму, значения второго столбца разбросаны по всему дереву — не существует непрерывного диапазона узлов, который можно было бы обойти. Поэтому СУБД вынуждена выполнить полное сканирование таблицы или индекса.

**Влияние диапазонных условий**

Применение индекса «останавливается» на первом столбце с диапазонным условием (`<`, `>`, `BETWEEN`, `LIKE 'prefix%'`). Все последующие столбцы индекса становятся недоступны для индексного поиска (хотя могут участвовать в index-only scan как хранимые значения).

```sql
-- Индекс (A, B, C):
WHERE A = 1 AND B > 5 AND C = 3
-- A используется полностью, B — как диапазон, C — НЕ используется для поиска
```

Именно поэтому рекомендуется ставить столбцы с условиями равенства (`=`) перед столбцами с диапазонными условиями.

**Порядок столбцов: практические правила**

1. **Столбцы с условием равенства — первыми.** Они максимально сужают диапазон поиска в B-дереве.
2. **Столбцы с диапазонными условиями или ORDER BY — последними.** После них остальные столбцы индекса уже не применяются для фильтрации.
3. **Высокая кардинальность — ближе к началу.** Столбцы с большим числом различных значений быстрее сужают пространство поиска.
4. **Учитывайте реальные запросы.** Индекс проектируется под фактические паттерны доступа приложения.

Пример: если чаще используется фильтр по `department_id` (равенство), а потом сортировка по `salary`, правильный порядок — `(department_id, salary)`, а не `(salary, department_id)`.

**Проверка использования индекса**

В PostgreSQL: `EXPLAIN SELECT ...` — ищите `Index Scan` или `Bitmap Index Scan` с упоминанием нужного индекса.
В MySQL: `EXPLAIN SELECT ...` — столбец `key` показывает, какой индекс выбран; `key_len` — сколько байт ключа использовано (по `key_len` можно определить, сколько колонок задействовано).

Расхождения между источниками: MySQL-документация описывает правило левого префикса применительно к B-Tree индексам; в PostgreSQL поведение аналогично, но оптимизатор PostgreSQL может использовать Bitmap Index Scan для неполных префиксов в некоторых случаях — это не нарушение правила, а использование индекса в режиме частичного сканирования с последующей фильтрацией.

### Вопрос 45. Как выбрать оптимальный порядок колонок в составном индексе: что важнее — поставить вперёд более селективный столбец или тот, по которому выполняется условие равенства (=)?

**Категория:** БД и SQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Use The Index, Luke — The right column order in multi-column indexes](https://use-the-index-luke.com/sql/where-clause/the-equals-operator/concatenated-keys)

Составной (конкатенированный) индекс работает как телефонный справочник: отсортирован сначала по первому столбцу, затем — по второму. Если поиск идёт только по второму столбцу — индекс не используется, так как значения второго столбца не упорядочены глобально.

Главное правило выбора порядка — **убедиться, что индекс используется как можно большим числом запросов**. Это важнее селективности. Столбец с условием равенства (`=`) должен стоять перед столбцом с условием диапазона (`<`, `>`, `BETWEEN`): как только встречается диапазонное условие, все последующие столбцы индекса перестают ограничивать диапазон сканирования.

Пример: для запроса `WHERE subsidiary_id = 20 AND date_of_birth > '1971-01-01'` оптимален индекс `(subsidiary_id, date_of_birth)`. Индекс `(date_of_birth, subsidiary_id)` вынуждает сканировать весь диапазон дат и применять фильтр по `subsidiary_id` уже после сканирования — это значительно медленнее.

Дополнительно, на [отдельной странице о мифах](https://use-the-index-luke.com/sql/myth-directory/most-selective-first) автор явно опровергает расхожее заблуждение: «наиболее селективный столбец — всегда первым». Это неверно. Существуют случаи, когда менее селективный столбец выгоднее ставить на первое место (например, Oracle может применить INDEX SKIP SCAN для слабоселективного ведущего столбца).

#### Ответ от Claude

Вопрос содержит скрытое противопоставление, которого на практике **нет**: условие равенства (`=`) и высокая селективность — не конкуренты, а разные критерии, применяемые на разных этапах анализа.

**Правило 1 (приоритетное): сначала равенство, потом диапазон.**

B-tree-индекс — это отсортированная структура. После того как встречается условие диапазона (`<`, `>`, `BETWEEN`, `LIKE 'prefix%'`), индекс уже не может использоваться для сужения диапазона по следующим столбцам — они становятся «фильтрами после сканирования» (filter predicates), а не «предикатами доступа» (access predicates).

Пример. Запрос:
```sql
SELECT * FROM orders
WHERE status = 'NEW' AND created_at > '2024-01-01';
```
Оптимальный индекс: `(status, created_at)`.
Индекс `(created_at, status)` заставит просматривать всё, что позже 2024-01-01, отсеивая `status` уже вне индексного дерева.

**Правило 2 (вторичное): среди столбцов одного типа (все равенства или все диапазоны) — более селективный вперёд.**

Если оба столбца участвуют в условиях равенства, то выгоднее поставить более селективный первым — он отсеет больше строк на раннем этапе, и второй предикат будет обрабатываться уже по меньшей выборке. Однако разница здесь менее критична, чем соблюдение правила 1.

**Правило 3: порядок с учётом «покрытия запросов».**

Если индекс используется несколькими разными запросами, приоритет отдаётся тому порядку, который позволяет большему числу запросов использовать индекс через leading-префикс. Иногда менее селективный столбец ставят первым именно потому, что большинство запросов фильтруют по нему.

**Итог (сводная таблица):**

| Ситуация | Рекомендация |
|---|---|
| Один столбец `=`, другой диапазон | Сначала `=`, потом диапазон |
| Оба столбца `=` | Более селективный вперёд (вторично — частота появления в запросах) |
| Оба диапазона | Более селективный вперёд, но выигрыш от второго столбца минимален |
| Разные запросы, разные паттерны | Выбрать порядок, покрывающий большинство запросов через leading-префикс |

**Расхождения между источниками:** Use The Index, Luke явно называет «селективность первой» мифом и считает её вторичным критерием. Ряд других ресурсов (systemoverflow.com) сочетают оба правила, описывая их как взаимодополняющие — это более корректная и практически применимая трактовка. Разночтений в правиле «равенство перед диапазоном» нет: все источники согласны.

### Вопрос 46. Чем hash-индекс отличается от B-tree индекса? В каких сценариях hash-индекс выигрывает, а в каких — проигрывает? Какие СУБД поддерживают hash-индексы и с какими ограничениями?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [B+ Tree vs Hash Index (and when to use them) — SQLpipe](https://www.sqlpipe.com/blog/b-tree-vs-hash-index-and-when-to-use-them)

**Структура:**

- B+-tree — самобалансирующееся дерево. Ключи хранятся во внутренних узлах, значения — в листовых. Листовые узлы связаны в двусвязный список, что обеспечивает эффективный range-scan.
- Hash-индекс применяет хеш-функцию к ключу и помещает запись в соответствующий «бакет». Никакого порядка между записями нет.

**Производительность:**

- B+-tree: поиск за O(log N) по числу страниц.
- Hash-индекс: теоретически O(1) для точного поиска.

**Поддерживаемые операции:**

Hash-индекс работает **только с оператором `=`**. Как только нужны `<`, `>`, `BETWEEN`, `ORDER BY` или `LIKE` — хеш-функция уничтожает информацию о порядке, и индекс бесполезен.

**Когда hash выигрывает:**

Исключительно при high-volume точных поисках по равенству на больших таблицах, где B-tree начинает проигрывать из-за накладных расходов на обход нескольких уровней дерева.

**Рекомендация автора:** «Используйте B+-tree, если только вам не нужно сэкономить 3 обращения к диску». Большинство СУБД по умолчанию используют B-tree именно из-за его универсальности.

#### Ответ от Claude

**Принципиальные различия:**

| Характеристика | B-tree | Hash-индекс |
|---|---|---|
| Структура | Сбалансированное дерево | Хеш-таблица с бакетами |
| Сложность поиска | O(log N) | O(1) теоретически |
| Поддерживаемые операторы | `=`, `<`, `>`, `<=`, `>=`, `BETWEEN`, `LIKE prefix%`, `IS NULL` | Только `=` |
| Сортировка / ORDER BY | Да (данные физически упорядочены) | Нет |
| Range-запросы | Да | Нет |
| Составной индекс | Поддерживает частичное использование (leading prefix) | Требует equality по всем колонкам ключа |

**Когда hash-индекс выигрывает:**

- Исключительно точные поиски по равенству (`=`) на больших таблицах.
- Данные уникальны или почти уникальны (низкое число коллизий).
- Рабочая нагрузка read-heavy с одним типом запроса.
- В PostgreSQL прирост составляет 10–22% по сравнению с B-tree при таких условиях (данные EnterpriseDB).

**Когда hash-индекс проигрывает:**

- Любые диапазонные запросы (`>`, `<`, `BETWEEN`).
- Сортировка и `ORDER BY`.
- Паттерны `LIKE 'prefix%'`.
- Запросы по подмножеству колонок составного ключа.
- Высокое число дубликатов (коллизии снижают производительность ниже B-tree).
- После рестарта сервера в системах, где hash-индекс хранится только в памяти.

**Поддержка в СУБД:**

| СУБД | Статус hash-индексов | Ограничения |
|---|---|---|
| **PostgreSQL** | Поддерживаются явно (`USING HASH`) | Только оператор `=`; нет поддержки multi-column; до PostgreSQL 10 не журналировались (небезопасны при crash) |
| **MySQL InnoDB** | Нет явных hash-индексов | InnoDB автоматически строит Adaptive Hash Index (AHI) в памяти для часто запрашиваемых страниц B-tree; управляется параметром `innodb_adaptive_hash_index`; при рестарте пересоздаётся |
| **MySQL MEMORY** | Поддерживаются явно | Только для таблиц типа MEMORY (in-memory); данные теряются при рестарте |
| **SQL Server** | Только для memory-optimized таблиц (In-Memory OLTP) | Требует указания `BUCKET_COUNT`; только `=` по всем колонкам ключа; производительность деградирует при большом числе дубликатов |
| **Oracle** | Hash clusters (не hash indexes) | Другая концепция: данные физически размещаются по хешу; нельзя создать кластерный индекс на hash cluster |
| **SQLite** | Не поддерживаются | Только B-tree |

**Расхождения между источниками:** PostgreSQL-документация и EnterpriseDB утверждают, что преимущество hash-индекса в PostgreSQL реально, но ограничено 10–22% и только для read-heavy equality workload. SQLpipe и другие обзорные ресурсы считают выигрыш несущественным и рекомендуют придерживаться B-tree в общем случае. Расхождений в части ограничений (только `=`, нет range) нет — все источники согласны.

### Вопрос 47. Что такое частичный (partial) индекс? Приведите пример, когда он даёт существенный выигрыш перед обычным индексом по тому же столбцу.

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 11.8. Partial Indexes](https://www.postgresql.org/docs/current/indexes-partial.html)

Частичный индекс строится над подмножеством строк таблицы — теми, что удовлетворяют предикату, заданному в `WHERE`-условии определения индекса.

**Синтаксис:**
```sql
CREATE INDEX index_name ON table_name (column)
    WHERE predicate;
```

**Пример 1 — неоплаченные заказы:**

Если таблица содержит миллионы заказов, но приложение работает почти исключительно с неоплаченными:
```sql
CREATE INDEX orders_unbilled_index ON orders (order_nr)
    WHERE billed IS NOT TRUE;
```
Обычный индекс по `order_nr` включал бы все строки. Частичный — только актуальные (например, 1% от общего объёма). Запрос `SELECT * FROM orders WHERE billed IS NOT TRUE AND order_nr < 10000` использует именно этот индекс.

**Пример 2 — уникальность по подмножеству:**
```sql
CREATE UNIQUE INDEX tests_success_constraint ON tests (subject, target)
    WHERE success;
```
Допускает множество записей с `success = false`, но гарантирует уникальность успешных результатов по паре `(subject, target)`.

**Пример 3 — исключение частых значений:**

Если большинство строк имеют `client_ip` из внутренней подсети, их выгодно исключить из индекса:
```sql
CREATE INDEX access_log_client_ip_ix ON access_log (client_ip)
WHERE NOT (client_ip > inet '192.168.100.0'
           AND client_ip < inet '192.168.100.255');
```

Планировщик PostgreSQL использует частичный индекс, только если может доказать, что `WHERE`-условие запроса **математически влечёт** предикат индекса.

#### Ответ от Claude

**Что такое частичный индекс:**

Частичный индекс — это индекс, содержащий записи только для строк, удовлетворяющих заданному предикату (`WHERE`-условие в определении индекса). Он поддерживается в PostgreSQL, частично в SQLite. MySQL не поддерживает partial indexes напрямую (можно имитировать через generated columns или функциональные индексы).

**Механизм выигрыша:**

Обычный индекс по столбцу `status` в таблице `orders` с 10 млн строк содержит 10 млн записей. Если 99% заказов имеют `status = 'COMPLETED'`, а приложение запрашивает почти исключительно `status = 'PENDING'` (100 тыс. строк), то:

- Обычный индекс: 10 млн записей, большой размер, медленная загрузка в память.
- Частичный индекс `WHERE status = 'PENDING'`: 100 тыс. записей — в 100 раз меньше.

```sql
-- Таблица с 10 млн заказов, из них 100 тыс. со статусом PENDING
CREATE INDEX idx_orders_pending ON orders (created_at)
    WHERE status = 'PENDING';

-- Запрос использует этот маленький индекс:
SELECT * FROM orders
WHERE status = 'PENDING'
  AND created_at > NOW() - INTERVAL '7 days';
```

**Конкретные сценарии существенного выигрыша:**

1. **Скошенное распределение статусов.** Большинство строк — архивные/завершённые, активно запрашивается малое подмножество. Классика: `WHERE processed = FALSE`, `WHERE deleted_at IS NULL`.

2. **Уникальность по условию.** Стандартный `UNIQUE` индекс нельзя ограничить условием. Частичный unique-индекс — можно:
```sql
CREATE UNIQUE INDEX idx_active_email ON users (email)
    WHERE active = TRUE;
-- Позволяет нескольким деактивированным пользователям иметь один email
```

3. **Исключение NULL или частых служебных значений.** Если 80% строк имеют `priority IS NULL` и запросы всегда фильтруют по конкретному priority — индекс `WHERE priority IS NOT NULL` на порядок меньше.

**Ограничения и подводные камни:**

- Планировщик PostgreSQL должен статически доказать, что предикат запроса влечёт предикат индекса. Параметризованные запросы (prepared statements с переменными) могут не использовать partial index, если планировщик не может вывести значение параметра на этапе планирования.
- Нельзя использовать вместо партиционирования: создание десятков непересекающихся partial-индексов менее эффективно, чем один составной индекс или партиционированная таблица.
- Не переносимы между СУБД: синтаксис и поддержка существенно различаются.

**Расхождений между источниками нет:** PostgreSQL-документация, atlasgo.io и datacamp.com описывают механизм и ограничения одинаково.
### Вопрос 48. Что такое функциональный (expression) индекс? Почему запрос с WHERE LOWER(email) = ? не использует обычный индекс на столбце email, и как это исправить?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 11.7. Indexes on Expressions](https://www.postgresql.org/docs/current/indexes-expressional.html)

Индекс можно строить не только по столбцу таблицы, но и по результату функции или скалярного выражения, вычисленного из одного или нескольких столбцов. Такая возможность называется индексами по выражениям (indexes on expressions).

Пример: чтобы организовать регистронезависимый поиск по имени, можно создать следующий индекс:

```sql
CREATE INDEX test1_lower_col1_idx ON test1 (lower(col1));
```

Если запрос выглядит как `WHERE lower(col1) = 'value'`, этот индекс будет использован оптимизатором.

Обычный индекс на столбце `email` хранит оригинальные значения в их фактическом регистре. Когда запрос содержит `WHERE LOWER(email) = ?`, СУБД обязана применить функцию `LOWER()` к каждой строке перед сравнением — оригинальные значения в индексе для этого не подходят, поэтому планировщик не может задействовать обычный индекс и вынужден выполнять полный последовательный просмотр таблицы (Seq Scan).

Индексы по выражениям относительно затратны в обслуживании: выражение должно вычисляться при каждой вставке строки и при каждом обновлении, не являющемся HOT-обновлением. Поэтому их следует применять в первую очередь тогда, когда скорость выборки важнее скорости вставки и обновления.

#### Ответ от Claude

**Что такое функциональный (expression) индекс.**
Функциональный индекс (в PostgreSQL называется «index on expression») — это индекс, который хранит не значение столбца, а результат произвольного детерминированного выражения над одним или несколькими столбцами. Оптимизатор использует такой индекс тогда и только тогда, когда выражение в предикате запроса точно совпадает с выражением, заданным при создании индекса.

**Почему `WHERE LOWER(email) = ?` не использует обычный индекс.**
Обычный B-tree индекс `CREATE INDEX idx_email ON users(email)` хранит исходные строки в том регистре, в котором они записаны в таблице (например, `User@Example.COM`). Условие `WHERE LOWER(email) = 'user@example.com'` требует сначала применить `LOWER()` к каждому значению и лишь потом сравнить результат с константой. Индекс по исходному столбцу не позволяет обойти эту трансформацию: планировщик не знает, какой строке соответствует результат `LOWER()`, не вычислив его явно. Следствие — планировщик выбирает Seq Scan и вычисляет `LOWER(email)` для каждой строки таблицы.

**Как исправить — создать функциональный индекс.**
```sql
CREATE INDEX idx_email_lower ON users (LOWER(email));
```
Теперь индекс хранит уже вычисленные значения `LOWER(email)`. Запрос

```sql
SELECT * FROM users WHERE LOWER(email) = 'user@example.com';
```

точно совпадает с выражением индекса, и планировщик выполняет Index Scan вместо Seq Scan.

**Альтернатива для PostgreSQL — тип `citext` или оператор `ILIKE`.**
PostgreSQL предоставляет расширение `citext` (case-insensitive text), которое делает все сравнения регистронезависимыми автоматически, а обычный индекс по `citext`-столбцу работает без дополнительных усилий. Оператор `ILIKE` также выполняет регистронезависимый поиск, но для него нужен отдельный индекс с классом операторов `text_pattern_ops` или расширение `pg_trgm`.

**Важные условия корректной работы.**
1. Выражение должно быть детерминированным (одинаковый результат для одинаковых входных данных).
2. Выражение в запросе должно точно совпадать с выражением в индексе — `lower(email)` и `LOWER(email)` идентичны, но `lower(trim(email))` уже другое выражение.
3. Функциональный индекс требует пересчёта выражения при каждой вставке и не-HOT-обновлении, что незначительно увеличивает стоимость записи.

**Источники:**
- PostgreSQL Documentation 18, §11.7 Indexes on Expressions: https://www.postgresql.org/docs/current/indexes-expressional.html
- Baeldung SQL, Case-Insensitive Search in SQL: https://www.baeldung.com/sql/case-insensitive-search

### Вопрос 49. Что такое оптимистичная блокировка (optimistic locking) и как она реализуется в реляционных БД? Чем она принципиально отличается от пессимистичной блокировки (SELECT FOR UPDATE)?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Vlad Mihalcea — A beginner's guide to database locking and the lost update phenomena](https://vladmihalcea.com/a-beginners-guide-to-database-locking-and-the-lost-update-phenomena/)

Проблема потерянного обновления (lost update) возникает, когда пользователь перезаписывает текущее состояние базы данных, не зная, что другой пользователь изменил данные в промежутке между их загрузкой и сохранением.

**Пессимистичная блокировка (SELECT FOR UPDATE).**
Транзакция немедленно захватывает блокировку записи в момент чтения строки:

```sql
SELECT * FROM PRODUCT WHERE ID = 1 FOR UPDATE;
```

Пока Алиса удерживает блокировку, запрос Боба на ту же строку блокируется до момента коммита Алисы. После разблокировки Боб автоматически видит актуальные данные и может безопасно произвести своё обновление. Условие корректной работы: обе транзакции должны использовать `SELECT FOR UPDATE`; если одна из них читает данные без блокировки, проблема потерянного обновления сохраняется.

**Оптимистичная блокировка (version column).**
Вместо блокировки используется версионный столбец. При обновлении в предикате `WHERE` указывается ожидаемая версия:

```sql
UPDATE PRODUCT
SET (LIKES, QUANTITY, VERSION) = (6, 7, 3)
WHERE (ID, VERSION) = (1, 2);
```

Если другая транзакция уже изменила строку и инкрементировала версию, условие `WHERE` не совпадёт ни с одной строкой (`UPDATE 0 rows`). Hibernate при обнаружении нулевого числа изменённых строк выбрасывает `OptimisticLockException` и откатывает транзакцию.

#### Ответ от Claude

**Суть проблемы.**
Обе стратегии блокировки решают проблему потерянного обновления (lost update): ситуацию, при которой параллельно выполняющиеся транзакции читают одни и те же данные, независимо изменяют их и при сохранении одна из транзакций молча перезаписывает изменения другой.

**Пессимистичная блокировка.**
Транзакция захватывает явную блокировку строки при чтении, не позволяя другим транзакциям читать или изменять её до фиксации.

```sql
-- Транзакция 1
BEGIN;
SELECT * FROM orders WHERE id = 42 FOR UPDATE;
-- строка заблокирована; транзакция 2 будет ждать
UPDATE orders SET status = 'processing' WHERE id = 42;
COMMIT;
```

Механизм: `SELECT ... FOR UPDATE` запрашивает эксклюзивную (write) блокировку строки. Другие транзакции, пытающиеся выполнить `SELECT ... FOR UPDATE` или любой `UPDATE`/`DELETE` на той же строке, блокируются до тех пор, пока первая транзакция не зафиксирует или не откатит изменения. Конфликт предотвращается заранее, а не обнаруживается постфактум.

**Оптимистичная блокировка.**
Никакая блокировка строки не захватывается. Вместо этого в таблице поддерживается версионный столбец (`version`, `updated_at` и т.п.), который инкрементируется при каждом успешном обновлении. Логика проверки конфликта:

```sql
-- Читаем строку (version = 5)
SELECT id, quantity, version FROM inventory WHERE id = 10;

-- Пытаемся обновить, проверяя версию
UPDATE inventory
   SET quantity = 99,
       version  = 6
 WHERE id = 7
   AND version = 5;
-- Если другая транзакция успела обновить строку раньше,
-- version уже не 5, и UPDATE вернёт 0 затронутых строк —
-- конфликт обнаружен, приложение должно повторить операцию.
```

В JPA/Hibernate версионный столбец объявляется аннотацией `@Version`; фреймворк автоматически выбрасывает `OptimisticLockException` при нулевом числе обновлённых строк.

**Принципиальные различия.**

| Аспект | Пессимистичная | Оптимистичная |
|---|---|---|
| Момент обнаружения конфликта | До изменения (предотвращение) | При фиксации (обнаружение) |
| Блокировка строки в БД | Да, явная | Нет |
| Поведение при конфликте | Ожидание (блокировка) | Исключение + ручной retry |
| Область применения | В рамках одной транзакции БД | Работает и между транзакциями (HTTP-запросы) |
| Риск дедлоков | Есть | Нет (нет блокировок) |
| Применимость при высокой конкуренции | Эффективнее | Много retry, деградация производительности |
| Применимость при низкой конкуренции | Избыточные блокировки | Эффективнее |

**Когда что выбирать.**
- Пессимистичная блокировка оправдана при высокой конкуренции за одни и те же строки или когда повторная попытка операции слишком дорога (например, долгий расчёт).
- Оптимистичная блокировка предпочтительна в веб-приложениях, где чтение и запись разделены несколькими HTTP-запросами (и, соответственно, несколькими транзакциями БД), а конфликты редки.

**Расхождение между источниками.**
Vladmihalcea.com и документация Hibernate согласны в механизме (версионный столбец + `UPDATE ... WHERE id = ? AND version = ?`). Документация Hibernate дополнительно описывает режимы `LockMode.PESSIMISTIC_READ` и `LockMode.PESSIMISTIC_WRITE`, которые транслируются в `SELECT ... FOR UPDATE` на уровне SQL. Расхождений по существу нет.

**Источники:**
- Vlad Mihalcea — A beginner's guide to database locking and the lost update phenomena: https://vladmihalcea.com/a-beginners-guide-to-database-locking-and-the-lost-update-phenomena/
- Vlad Mihalcea — Optimistic vs. Pessimistic Locking: https://vladmihalcea.com/optimistic-vs-pessimistic-locking/
- Hibernate ORM 5.1 User Guide — Locking: https://docs.hibernate.org/orm/5.1/userguide/html_single/chapters/locking/Locking.html

### Вопрос 50. На один и тот же заказ одновременно приходит ~500 параллельных запросов «оплатить». Какую стратегию блокировки выбрать, чтобы избежать lost update, и почему пессимистичная здесь предпочтительнее оптимистичной?

**Категория:** БД и SQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Habr — Java-собеседование 2025: разбор вопросов](https://habr.com/ru/articles/1043906/)

В статье: предпочтительна пессимистичная блокировка через `SELECT ... FOR UPDATE`. Оптимистичная блокировка (`@Version`) при такой высокой конкуренции на одну строку даст массу конфликтов и повторных попыток.

#### Ответ от Claude

Проблема — **lost update**: параллельные транзакции читают одно состояние заказа и затирают изменения друг друга.

- **Оптимистичная блокировка** (`@Version`): транзакция при коммите проверяет, не изменилась ли версия строки; если изменилась — `OptimisticLockException` и retry. При **высокой конкуренции на одну строку** (500 запросов к одному заказу) почти все транзакции будут конфликтовать и бесконечно ретраить — это «живёт», но жжёт ресурсы.
- **Пессимистичная блокировка** (`SELECT ... FOR UPDATE`): первая транзакция берёт блокировку строки, остальные ждут в очереди и видят уже актуальное состояние. Это сериализует доступ к конкретному заказу — то, что нужно при высокой контенции на одну запись.

Вывод: при интенсивной борьбе за **одну** строку выбирают пессимистичную (`FOR UPDATE`); оптимистичная лучше при редких конфликтах. Дополнительно стоит обеспечить **идемпотентность** оплаты (ключ идемпотентности), чтобы повторные запросы не списывали дважды. См. [PostgreSQL — Explicit Locking](https://www.postgresql.org/docs/current/explicit-locking.html).

### Вопрос 51. Чем ON отличается от USING в синтаксисе JOIN? В каких случаях USING неприменим?

**Категория:** БД и SQL · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 7.2. Table Expressions](https://www.postgresql.org/docs/current/queries-table-expressions.html)

Предложение `ON` является наиболее общим видом условия соединения: оно принимает булево выражение, как и предложение `WHERE`. Пара строк из T1 и T2 совпадает, если выражение `ON` возвращает `true`. Результат `JOIN ON` содержит все столбцы из T1, за которыми следуют все столбцы из T2, — включая дублирующиеся столбцы из обеих сторон условия.

Предложение `USING` — сокращённая форма, применимая в частном случае, когда обе стороны соединения используют одно и то же имя для соединяемого столбца (или столбцов). Оно принимает список имён столбцов через запятую и формирует условие соединения, включающее сравнение на равенство для каждого из них. Так, `USING (a, b)` порождает условие `ON T1.a = T2.a AND T1.b = T2.b`.

Кроме того, `JOIN USING` подавляет избыточные столбцы в выводе: нет нужды выводить оба столбца из совпадающей пары, поскольку они заведомо равны. Тогда как `JOIN ON` выводит все столбцы из T1, а затем все столбцы из T2, `JOIN USING` выводит один столбец для каждой совпадающей пары (в указанном порядке), затем оставшиеся столбцы T1, затем оставшиеся столбцы T2.

> `USING` достаточно безопасен при изменениях в схеме таблиц, поскольку объединяются только перечисленные столбцы.

#### Ответ от Claude

**ON** — универсальное условие соединения; задаётся произвольным булевым выражением. Позволяет соединять по неодинаково называющимся столбцам, использовать неравенства, диапазоны, функции, составные выражения. В результате оба столбца-ключа присутствуют раздельно.

```sql
SELECT o.id, c.name
FROM orders o
JOIN customers c ON o.customer_id = c.id;
-- столбцы customer_id и id присутствуют оба
```

**USING** — синтаксический сахар для частного случая: столбцы с обеих сторон имеют **одинаковое имя** и объединяются по равенству. Главное отличие от `ON` — дублирующийся столбец появляется в результате **один раз**, что удобно при `SELECT *`.

```sql
SELECT *
FROM orders
JOIN customers USING (customer_id);
-- customer_id появится только один раз
```

**Когда USING неприменим:**

1. Столбцы называются по-разному (`orders.customer_id` vs `customers.id`) — нужен `ON`.
2. Условие соединения содержит неравенство, диапазон, функцию или составное выражение — нужен `ON`.
3. Необходимо явно сослаться на оба столбца по отдельности в `SELECT` или `WHERE` — `USING` объединяет их в один, что может мешать.
4. Субд не поддерживает синтаксис `USING` (например, ранние версии SQL Server до 2012 не поддерживали; MySQL — поддерживает).

Расхождений между источниками по данному вопросу не выявлено: PostgreSQL-документация и Baeldung описывают поведение идентично.

### Вопрос 52. Что такое NATURAL JOIN? Почему его принято избегать в продуктивном коде?

**Категория:** БД и SQL · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 7.2. Table Expressions](https://www.postgresql.org/docs/current/queries-table-expressions.html)

`NATURAL` — сокращённая форма `USING`: он автоматически формирует список `USING` из всех имён столбцов, присутствующих в обеих входных таблицах. Как и при `USING`, совпадающие столбцы появляются в выводе только один раз. Если у таблиц нет общих имён столбцов, `NATURAL JOIN` ведёт себя как `CROSS JOIN`.

> `NATURAL` значительно рискованнее `USING`, поскольку любые изменения схемы одной из таблиц, порождающие новое совпадающее имя столбца, автоматически включат этот столбец в условие соединения.

#### Ответ от Claude

**Что такое NATURAL JOIN.** Оператор, который автоматически определяет условие соединения, сравнивая значения всех столбцов с одинаковыми именами в обеих таблицах. Разработчику не нужно явно перечислять столбцы.

```sql
-- Предположим: таблица employees(id, dept_id, name)
--              таблица departments(dept_id, name, location)
-- Общие столбцы: dept_id И name

SELECT * FROM employees NATURAL JOIN departments;
-- Фактически: ON employees.dept_id = departments.dept_id
--              AND employees.name = departments.name  <-- НЕОЖИДАННО!
```

**Почему его избегают в продуктивном коде:**

1. **Неявность условия.** Глядя на запрос, невозможно сказать, по каким столбцам происходит соединение, — это скрыто в структуре схемы. Код трудно читать и ревьюить.

2. **Хрупкость при эволюции схемы.** Добавление в любую из таблиц столбца с именем, совпадающим с именем столбца другой таблицы, **молча меняет семантику запроса**. Пример: добавили столбец `updated_at` в обе таблицы — `NATURAL JOIN` начнёт фильтровать строки по этому столбцу без каких-либо ошибок компилятора или предупреждений.

3. **Скрытый CROSS JOIN при отсутствии общих столбцов.** Если у таблиц не оказалось ни одного общего имени столбца, `NATURAL JOIN` деградирует до декартова произведения. Это катастрофа на больших таблицах, о которой легко не догадаться.

4. **Проблемы с технической колонкой `id`.** Многие схемы используют имя `id` в каждой таблице; при `NATURAL JOIN` такие строки не совпадут, что приводит к пустому результату без очевидной причины.

**Безопасная альтернатива:** использовать `JOIN ... USING (column_list)` — это даёт те же преимущества (вывод одного столбца), но соединяемые столбцы перечислены явно и не меняются при изменении схемы.

```sql
-- Безопасно и явно:
SELECT * FROM employees JOIN departments USING (dept_id);
```

### Вопрос 53. Для чего используется CROSS JOIN? Приведите практический пример, где он нужен, и объясните, почему его применение на больших таблицах опасно.

**Категория:** БД и SQL · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 7.2. Table Expressions](https://www.postgresql.org/docs/current/queries-table-expressions.html)

Для каждой возможной комбинации строк из T1 и T2 (то есть декартова произведения) результирующая таблица содержит строку, состоящую из всех столбцов T1, за которыми следуют все столбцы T2. Если таблицы содержат N и M строк соответственно, результирующая таблица будет содержать N × M строк.

`FROM T1 CROSS JOIN T2` эквивалентно `FROM T1 INNER JOIN T2 ON TRUE`, а также `FROM T1, T2`.

Важное замечание о привязке операторов: эта последняя эквивалентность не выполняется точно, когда появляется более двух таблиц, потому что `JOIN` связывает теснее, чем запятая. Например, `FROM T1 CROSS JOIN T2 INNER JOIN T3 ON condition` — это не то же самое, что `FROM T1, T2 INNER JOIN T3 ON condition`, поскольку в первом случае `condition` может ссылаться на T1, а во втором — нет.

#### Ответ от Claude

**Что делает CROSS JOIN.** Возвращает декартово произведение двух таблиц: каждая строка первой таблицы соединяется с каждой строкой второй. Условие соединения (`ON` / `WHERE`) отсутствует или не используется для отбора строк на уровне JOIN.

**Практические случаи применения:**

**1. Генерация всех комбинаций (матрица расписания, тарифная сетка).** Классический пример — построить матрицу «все сотрудники × все проекты» для планирования или для последующего LEFT JOIN с реальными данными:

```sql
-- Сгенерировать все возможные назначения сотрудник-проект
SELECT e.name AS employee, p.title AS project
FROM employees e
CROSS JOIN projects p
ORDER BY e.name, p.title;
```

**2. Генерация непрерывного ряда дат.** `CROSS JOIN` с таблицей-генератором чисел позволяет построить полный диапазон дат, в том числе дни без событий:

```sql
-- Полный календарь января 2025
SELECT (DATE '2025-01-01' + n.i)::date AS day
FROM generate_series(0, 30) AS n(i);

-- Или через CROSS JOIN с двумя рядами для декартова расширения:
SELECT months.m, days.d
FROM (VALUES (1),(2),(3)) AS months(m)
CROSS JOIN (VALUES (1),(2),(3),(4),(5)) AS days(d);
```

**3. Комбинаторное заполнение тестовых данных.** При подготовке нагрузочных тестов удобно скрестить небольшое число справочных строк, получив сразу тысячи комбинаций.

**Почему CROSS JOIN опасен на больших таблицах:**

Результирующий набор растёт **мультипликативно**: N строк × M строк = N·M строк. Таблицы по 1000 строк дадут 1 000 000 строк; по 10 000 — уже 100 000 000 строк. Это:
- Переполняет буферы сортировки и хэш-таблицы, вынуждая СУБД сбрасывать данные на диск.
- Потребляет оперативную память и дисковое I/O пропорционально произведению размеров таблиц.
- Блокирует ресурсы сервера на долгое время, что влияет на конкурирующие запросы.

Особенно опасна **случайная** декартова операция: когда разработчик перечисляет несколько таблиц через запятую и забывает условие `WHERE` / `ON`, — СУБД молча выполняет `CROSS JOIN` без предупреждения. Именно поэтому современные гайды рекомендуют всегда использовать явный синтаксис `JOIN ... ON`, чтобы случайное декартово произведение стало синтаксической ошибкой, а не тихой катастрофой производительности.

```sql
-- ОПАСНО: случайный CROSS JOIN
SELECT * FROM orders, customers;        -- N × M строк!

-- БЕЗОПАСНО: явное условие соединения
SELECT * FROM orders JOIN customers ON orders.customer_id = customers.id;
```

### Вопрос 54. Почему фильтр на столбец правой таблицы в секции WHERE превращает LEFT JOIN фактически в INNER JOIN? Как это исправить?

**Категория:** БД и SQL · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [MySQL 8.4 Reference Manual — 10.2.1.10 Outer Join Simplification](https://dev.mysql.com/doc/refman/8.4/en/outer-join-simplification.html)

MySQL (и другие СУБД) выполняют оптимизацию, называемую «упрощением внешнего соединения» (outer join simplification): если условие в секции WHERE является **null-rejected** для внешнего соединения, оптимизатор автоматически конвертирует LEFT/RIGHT JOIN во INNER JOIN.

Условие называется null-rejected, если оно гарантированно возвращает FALSE или UNKNOWN для любой строки с NULL-дополнением, порождаемой внешним соединением. Примеры null-rejected условий: `T2.B > 3`, `T2.C IS NOT NULL`, `T2.C <= T1.C` — любое из них обращается к столбцу правой таблицы и не может быть истинным, когда все её столбцы равны NULL.

Пример из документации:

```sql
-- Исходный запрос
SELECT * FROM T1
  LEFT JOIN T2 ON T2.A = T1.A
  LEFT JOIN T3 ON T3.B = T1.B
WHERE T3.C > 0;
```

После упрощения MySQL выполняет этот запрос как:

```sql
SELECT * FROM T1
  LEFT JOIN T2 ON T2.A = T1.A
  INNER JOIN T3 ON T3.B = T1.B
WHERE T3.C > 0;
```

Условие `T3.C > 0` ссылается на столбец правой таблицы T3. Для строк, у которых в T3 нет совпадения, значение `T3.C` равно NULL, и `NULL > 0` равно UNKNOWN — то есть строка отфильтровывается. Тем самым все «неподходящие» строки, которые LEFT JOIN должен был сохранить с NULL-значениями, исчезают из результата.

#### Ответ от Claude

**Механизм проблемы.** LEFT JOIN возвращает все строки левой таблицы, подставляя NULL в столбцы правой таблицы там, где совпадений нет. Условие фильтрации в секции WHERE применяется *после* того, как соединение уже выполнено. Если условие ссылается на столбец правой таблицы и не учитывает NULL (например, `right_table.status = 'active'`), то для всех «неподходящих» строк вычисление даёт NULL = 'active' → UNKNOWN → строка удаляется. В итоге в результате остаются только строки, у которых совпадение нашлось, — то есть поведение эквивалентно INNER JOIN.

```sql
-- ПРОБЛЕМА: LEFT JOIN ведёт себя как INNER JOIN
SELECT e.name, d.department_name
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.id
WHERE d.department_name = 'Engineering';
-- Сотрудники без отдела исчезают, потому что NULL = 'Engineering' → FALSE
```

**Исправление: перенести фильтр в секцию ON.**

```sql
-- ПРАВИЛЬНО: фильтр переносится в ON, LEFT JOIN сохраняет строки без совпадений
SELECT e.name, d.department_name
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.id AND d.department_name = 'Engineering';
-- Сотрудники без отдела остаются в результате с NULL в department_name
```

Условие в ON определяет, *какие строки правой таблицы участвуют* в соединении; строки левой таблицы без совпадения при этом сохраняются с NULL. Условие в WHERE фильтрует *уже собранный* результат, удаляя строки.

**Исключение.** Если намеренно нужно найти строки, у которых *нет* совпадения в правой таблице, фильтр `WHERE right_table.id IS NULL` не нарушает семантику LEFT JOIN — он является null-accepted, то есть истинен именно для NULL-дополненных строк.

### Вопрос 55. Как эмулировать FULL OUTER JOIN в MySQL, где он исторически не поддерживался? Напишите запрос через UNION.

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [MySQL Worklog WL#1604 — Support FULL [OUTER] JOIN by rewriting with UNION](https://dev.mysql.com/worklog/task/?id=1604)

Собственного синтаксиса `FULL OUTER JOIN` в MySQL нет. Worklog WL#1604 фиксирует, что FULL OUTER JOIN можно эмулировать через объединение LEFT JOIN и RIGHT JOIN с помощью UNION:

```sql
(SELECT ... FROM a LEFT JOIN b ON a.id = b.id)
UNION
(SELECT ... FROM a RIGHT JOIN b ON a.id = b.id)
```

`UNION` (без `ALL`) устраняет дублирование строк, у которых совпадение в обеих таблицах найдено: такие строки появляются и в LEFT JOIN, и в RIGHT JOIN, но UNION оставит их лишь один раз.

#### Ответ от Claude

**Что такое FULL OUTER JOIN.** Он возвращает все строки из обеих таблиц: там, где совпадение есть — объединённую строку; там, где нет — строку из одной таблицы с NULL в столбцах другой. MySQL не поддерживает этот синтаксис нативно ни в одной из текущих версий (проверено по официальному руководству MySQL 8.4).

**Эмуляция через UNION.**

Наиболее распространённый вариант — использовать UNION без ALL:

```sql
SELECT a.id, a.name, b.order_total
FROM customers a
LEFT JOIN orders b ON a.id = b.customer_id

UNION

SELECT a.id, a.name, b.order_total
FROM customers a
RIGHT JOIN orders b ON a.id = b.customer_id;
```

- Первый SELECT (LEFT JOIN) даёт все строки из `customers` плюс совпадающие строки из `orders`.
- Второй SELECT (RIGHT JOIN) даёт все строки из `orders` плюс совпадающие строки из `customers`.
- UNION объединяет результаты и удаляет полные дубликаты (строки, у которых совпадение нашлось с обеих сторон, появятся оба раза с одинаковыми значениями и будут схлопнуты в одну).

**Оптимизированный вариант с явным исключением дублей** (экономит память при больших таблицах, так как UNION ALL не строит временную таблицу для дедупликации):

```sql
SELECT a.id, a.name, b.order_total
FROM customers a
LEFT JOIN orders b ON a.id = b.customer_id

UNION ALL

SELECT a.id, a.name, b.order_total
FROM customers a
RIGHT JOIN orders b ON a.id = b.customer_id
WHERE a.id IS NULL;
```

Здесь второй SELECT с `WHERE a.id IS NULL` возвращает только те строки из `orders`, у которых не нашлось совпадения в `customers` — именно они отсутствовали в результате LEFT JOIN. UNION ALL просто склеивает наборы без сортировки/дедупликации, что быстрее.

**Замечание об эквивалентности.** Внутри MySQL RIGHT JOIN преобразуется оптимизатором в LEFT JOIN с переставленными таблицами, поэтому оба варианта синтаксически корректны. СУБД, поддерживающие FULL OUTER JOIN нативно (PostgreSQL, Oracle, SQL Server, MariaDB 10.0+), позволяют записать это одним оператором:

```sql
SELECT a.id, a.name, b.order_total
FROM customers a
FULL OUTER JOIN orders b ON a.id = b.customer_id;
```

### Вопрос 56. Что такое составное условие соединения (composite join condition)? Когда оно необходимо и как его записать?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL 18 Documentation — 7.2. Table Expressions](https://www.postgresql.org/docs/current/queries-table-expressions.html)

Секция ON является наиболее общей формой условия соединения: она принимает булево выражение того же вида, что и условие WHERE. Пара строк из двух таблиц считается совпавшей, если выражение ON вычисляется в TRUE.

Предложение USING является сокращением для ситуации, когда обе стороны соединения используют одно и то же имя для соединяемых столбцов. Оно принимает список имён столбцов через запятую и формирует условие соединения, включающее проверку равенства для каждой пары. Например, соединение T1 и T2 с `USING (a, b)` эквивалентно условию `ON T1.a = T2.a AND T1.b = T2.b`.

```sql
-- через USING (оба столбца должны иметь одинаковые имена в обеих таблицах)
SELECT * FROM t1 JOIN t2 USING (a, b);

-- через ON (явная запись составного условия)
SELECT * FROM t1 JOIN t2 ON t1.a = t2.a AND t1.b = t2.b;
```

#### Ответ от Claude

**Определение.** Составное условие соединения — это условие JOIN, которое включает сравнение нескольких пар столбцов одновременно. Строки двух таблиц считаются совпавшими только тогда, когда все указанные пары столбцов равны между собой.

**Когда необходимо.**

1. **Составной первичный / внешний ключ.** Если первичный ключ таблицы состоит из нескольких столбцов (например, `(order_id, line_number)` в таблице позиций заказа), внешний ключ и соединение также должны охватывать все эти столбцы.

2. **Партиционирование или версионирование данных.** Таблицы могут совместно использовать идентификатор записи и идентификатор периода/версии, и соединение правильно только тогда, когда оба совпадают.

3. **Устранение неоднозначности.** Когда один столбец не даёт однозначного совпадения (например, один и тот же `product_id` существует в нескольких складах или магазинах), нужен второй ключевой столбец.

**Синтаксис через ON (универсальный — столбцы могут иметь разные имена):**

```sql
SELECT ol.order_id, ol.line_number, p.product_name
FROM order_lines ol
INNER JOIN price_list p
  ON ol.product_id = p.product_id
 AND ol.region_code = p.region_code;
```

**Синтаксис через USING (сокращённый — требует одинаковых имён столбцов в обеих таблицах):**

```sql
SELECT * FROM order_lines
JOIN price_list USING (product_id, region_code);
-- эквивалентно: ON order_lines.product_id = price_list.product_id
--           AND order_lines.region_code = price_list.region_code
```

Преимущество USING: совпавшие столбцы появляются в результате один раз, а не дважды. Недостаток: требует одинаковых имён столбцов.

**Важный нюанс.** В ON можно комбинировать проверки равенства с другими условиями (неравенства, диапазоны, функции). Это уже не «составное условие» в строгом смысле, а «нееквивалентное соединение» (non-equi join), которое не оптимизируется хэш-соединением так же эффективно, как обычный equi-join. Составное условие в стандартном смысле — это всегда несколько проверок вида `col_a = col_b`, объединённых через AND.

### Вопрос 57. Что такое semi-join и anti-join? Как они выражаются через EXISTS / NOT EXISTS, и почему это часто предпочтительнее JOIN?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 9.24. Subquery Expressions](https://www.postgresql.org/docs/current/functions-subquery.html)

**Semi-join** — логическая операция, возвращающая строки из левой таблицы, для которых существует хотя бы одна совпадающая строка в правой таблице. В отличие от обычного JOIN, повторяющиеся совпадения не порождают дополнительных строк в результате.

**Anti-join** — обратная операция: возвращает строки из левой таблицы, для которых в правой таблице нет ни одной совпадающей строки.

В стандартном SQL semi-join и anti-join не имеют собственного синтаксиса. Они выражаются через подзапросы с EXISTS / NOT EXISTS:

```sql
-- Semi-join: строки из tab1, у которых есть совпадение в tab2
SELECT col1
FROM tab1
WHERE EXISTS (SELECT 1 FROM tab2 WHERE col2 = tab1.col2);

-- Anti-join: строки из tab1, у которых НЕТ совпадения в tab2
SELECT col1
FROM tab1
WHERE NOT EXISTS (SELECT 1 FROM tab2 WHERE col2 = tab1.col2);
```

Документация PostgreSQL уточняет: подзапрос в EXISTS «вычисляется ровно настолько, чтобы определить, возвращает ли он хотя бы одну строку, а не до полного завершения». Это позволяет планировщику использовать оптимизированные алгоритмы выполнения — semi-join и anti-join узлы — которые прекращают сканирование правой таблицы сразу после первого совпадения.

#### Ответ от Claude

**Semi-join** возвращает строки из левой таблицы при наличии хотя бы одного совпадения в правой, без размножения строк. **Anti-join** — при отсутствии любого совпадения.

**SQL-выражение:**

```sql
-- Semi-join через EXISTS
SELECT e.id, e.name
FROM employees e
WHERE EXISTS (
    SELECT 1 FROM orders o WHERE o.employee_id = e.id
);

-- Anti-join через NOT EXISTS
SELECT e.id, e.name
FROM employees e
WHERE NOT EXISTS (
    SELECT 1 FROM orders o WHERE o.employee_id = e.id
);
```

**Почему EXISTS / NOT EXISTS предпочтительнее обычного JOIN:**

1. **Нет размножения строк (fan-out).** `INNER JOIN` на отношении «один ко многим» дублирует строки левой таблицы для каждого совпадения. С EXISTS результат содержит ровно по одной строке на каждую строку левой таблицы — независимо от числа совпадений справа.

2. **Корректная семантика.** Цель запроса — проверить факт существования, а не извлечь данные правой таблицы. EXISTS точно выражает это намерение.

3. **Раннее прекращение вычисления.** Планировщик может остановить поиск в правой таблице сразу после первого совпадения, тогда как JOIN вынужден находить все совпадения.

4. **Безопасность с NULL.** NOT EXISTS корректно обрабатывает NULL в правой таблице, тогда как NOT IN с NULL-значениями даёт ловушку (см. вопрос 8).

**Планировщик PostgreSQL** автоматически распознаёт EXISTS-подзапросы как semi-join / anti-join и применяет соответствующие узлы плана. Запись `SELECT 1` внутри EXISTS является общепринятым соглашением — содержимое списка SELECT для EXISTS несущественно.

**Альтернативный способ anti-join через LEFT JOIN:**
```sql
SELECT e.id, e.name
FROM employees e
LEFT JOIN orders o ON o.employee_id = e.id
WHERE o.employee_id IS NULL;
```
Этот вариант семантически эквивалентен NOT EXISTS, однако менее читаем и требует аккуратности: если у одного сотрудника несколько заказов, нужен DISTINCT или GROUP BY, иначе LEFT JOIN / IS NULL выдаст правильный результат только случайно — в зависимости от того, берётся ли строка с NULL из строк без совпадений.

### Вопрос 58. В чём ловушка NOT IN с подзапросом, который может вернуть NULL? Чем NOT EXISTS лучше в таком случае?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 9.24. Subquery Expressions](https://www.postgresql.org/docs/current/functions-subquery.html)

Документация PostgreSQL явно предупреждает:

> «Обратите внимание: если левое выражение возвращает NULL, или если среди значений правой части нет равных, но хотя бы одна строка правой части возвращает NULL, то результатом конструкции NOT IN будет NULL, а не TRUE. Это соответствует стандартным правилам SQL для булевых комбинаций со значениями NULL.»

Иными словами, при наличии хотя бы одного NULL в результате подзапроса выражение `col NOT IN (подзапрос)` не возвращает ни одной строки — даже если ни одно из непустых значений не совпадает с col. Это происходит потому, что сравнение с NULL по правилам трёхзначной логики SQL всегда даёт UNKNOWN, а NOT IN требует, чтобы все сравнения были TRUE.

NOT EXISTS не имеет этой проблемы: он возвращает TRUE, если подзапрос не вернул ни одной строки, независимо от того, содержат ли строки NULL.

#### Ответ от Claude

**Суть ловушки**

В трёхзначной логике SQL сравнение с NULL даёт UNKNOWN (не FALSE и не TRUE). Конструкция `value NOT IN (a, b, NULL)` разворачивается в:

```
value <> a  AND  value <> b  AND  value <> NULL
```

Последнее условие `value <> NULL` всегда даёт UNKNOWN, поэтому всё выражение — UNKNOWN. WHERE отфильтровывает строки, где условие не TRUE, поэтому такие строки не попадают в результат.

**Пример:**

```sql
-- Таблица departments содержит строки (1), (2), (NULL)
SELECT id FROM employees
WHERE department_id NOT IN (SELECT id FROM departments);
-- Результат: 0 строк — даже если в employees есть сотрудники
-- с department_id, которого нет среди непустых значений departments!
```

**Почему NOT EXISTS безопаснее:**

```sql
SELECT id FROM employees e
WHERE NOT EXISTS (
    SELECT 1 FROM departments d WHERE d.id = e.department_id
);
```

Здесь подзапрос коррелирован: для каждой строки employees он проверяет, есть ли совпадающая строка в departments. NULL в столбце `departments.id` просто не совпадёт с `e.department_id` (поскольку NULL = anything даёт UNKNOWN, а не TRUE), поэтому такая строка не считается «совпадением» — и NOT EXISTS вернёт TRUE для сотрудников, у которых нет совпадения ни с одной строкой.

**Обходные пути при необходимости использовать NOT IN:**

```sql
-- Вариант 1: явно исключить NULL из подзапроса
SELECT id FROM employees
WHERE department_id NOT IN (
    SELECT id FROM departments WHERE id IS NOT NULL
);

-- Вариант 2 (PostgreSQL): использовать ALL
SELECT id FROM employees
WHERE department_id <> ALL (
    SELECT id FROM departments WHERE id IS NOT NULL
);
```

**Итог:** NOT EXISTS является предпочтительным идиомом для anti-join, поскольку:
- корректно работает при наличии NULL в данных;
- явно выражает намерение «не существует совпадения»;
- планировщик PostgreSQL преобразует его в эффективный узел anti-join.

### Вопрос 59. Как размножение строк (fan-out) при JOIN один-ко-многим искажает результаты агрегатных функций SUM и COUNT? Приведите пример и способы устранения.

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [The problem of SQL fanouts — Google Cloud Community / Technical Tips & Tricks](https://discuss.google.dev/gc/Technical-Tips-Tricks/The-problem-of-SQL-fanouts/ta-p/587483)

Fan-out возникает, когда JOIN-операция увеличивает число строк в результате по сравнению с основной таблицей. Если таблица customers (3 строки) соединяется с таблицей orders (4 строки) по отношению «один ко многим», в результате JOIN появляется 4 строки — строка клиента с несколькими заказами дублируется.

**Последствия для агрегатов:**

- `COUNT(*)` возвращает 4 вместо 3 клиентов — один клиент посчитан дважды.
- `SUM(visits)` возвращает 10 вместо 8 — визиты продублированного клиента суммируются дважды.

**Типология по риску fan-out:**

| Тип JOIN | Риск fan-out | Надёжность агрегатов |
|----------|--------------|----------------------|
| Один-к-одному | Нет | Безопасно |
| Многие-к-одному | Нет | Безопасно для основной таблицы |
| Один-ко-многим | Высокий | Нельзя доверять |

**Рекомендуемые способы устранения:**

1. **Изменить порядок JOIN** — начинать с самой детальной таблицы (orders, а не customers).

2. **Предварительная агрегация в подзапросе:**
   ```sql
   SELECT c.id, c.name, o.total
   FROM customers c
   LEFT JOIN (
       SELECT customer_id, SUM(amount) AS total
       FROM orders
       GROUP BY customer_id
   ) AS o ON o.customer_id = c.id;
   ```

#### Ответ от Claude

**Что такое fan-out**

При JOIN таблицы A (один) с таблицей B (многие) каждая строка A размножается на столько копий, сколько совпадающих строк есть в B. Это правильное поведение для задач соединения данных, но губительно при агрегировании по столбцам таблицы A.

**Пример:**

```sql
-- customers: id, name, monthly_visits
-- (1, 'Алиса', 5), (2, 'Боб', 3)
-- orders: id, customer_id, amount
-- (101, 1, 100), (102, 1, 200), (103, 2, 150)

SELECT c.name, COUNT(*) AS cnt, SUM(c.monthly_visits) AS visits, SUM(o.amount) AS revenue
FROM customers c
JOIN orders o ON o.customer_id = c.id
GROUP BY c.name;
```

Результат:

| name  | cnt | visits | revenue |
|-------|-----|--------|---------|
| Алиса | 2   | 10     | 300     |
| Боб   | 1   | 3      | 150     |

`cnt` для Алисы равен 2 (а не 1), `visits` равен 10 (а не 5) — строка клиента продублировалась для каждого заказа. `revenue` для Алисы правилен (300), поскольку суммируется столбец из таблицы orders.

**Способы устранения:**

**1. Предварительная агрегация в подзапросе / CTE**

```sql
WITH order_totals AS (
    SELECT customer_id, SUM(amount) AS revenue
    FROM orders
    GROUP BY customer_id
)
SELECT c.id, c.name, c.monthly_visits, ot.revenue
FROM customers c
LEFT JOIN order_totals ot ON ot.customer_id = c.id;
```

Подзапрос сворачивает orders до одной строки на клиента до JOIN — fan-out устраняется.

**2. COUNT DISTINCT / SUM DISTINCT (когда есть уникальный ключ)**

```sql
SELECT COUNT(DISTINCT c.id) AS customer_count,
       SUM(DISTINCT c.monthly_visits) AS total_visits,  -- ОСТОРОЖНО: только если visits уникальны!
       SUM(o.amount) AS revenue
FROM customers c
JOIN orders o ON o.customer_id = c.id;
```

DISTINCT внутри COUNT работает надёжно. `SUM(DISTINCT ...)` опасен: он отбрасывает строки с одинаковым значением, что корректно только если дубликаты — единственная причина повторения. Использовать с осторожностью.

**3. Оконные функции (без GROUP BY)**

```sql
SELECT DISTINCT
    c.id,
    c.name,
    SUM(o.amount) OVER (PARTITION BY c.id) AS revenue,
    c.monthly_visits   -- значение из основной таблицы, не агрегируется
FROM customers c
JOIN orders o ON o.customer_id = c.id;
```

**Диагностика fan-out:** сравните `COUNT(*)` после JOIN с `COUNT(DISTINCT primary_key)` основной таблицы. Если они различаются — fan-out есть.

**Итог:** наиболее надёжный и читаемый способ — предварительная агрегация в подзапросе или CTE, которая превращает отношение «один ко многим» в «один к одному» до выполнения JOIN.

### Вопрос 60. Что такое equi-join и non-equi-join? Приведите пример non-equi-join и объясните, когда он нужен.

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [An Illustrated Guide to the SQL Non Equi Join — LearnSQL.com](https://learnsql.com/blog/illustrated-guide-sql-non-equi-join/)

**Equi-join** — это обычное соединение таблиц с оператором равенства (`=`) в условии ON. Записи сопоставляются по совпадающим значениям столбцов:

```sql
JOIN apartment ON apartment.id = person.apartment_id
```

**Non-equi-join** — это соединение с любым оператором сравнения, кроме `=`: `<`, `>`, `<=`, `>=`, `!=`, `<>`, а также `BETWEEN ... AND`. Оно необходимо, когда нужно выразить диапазонное или неравенственное отношение между таблицами.

Пример: подобрать квартиры покупателям по ценовому диапазону, исключив уже выбранные:

```sql
SELECT first_name, last_name, price, city
FROM person
JOIN apartment
  ON apartment.id != person.apartment_id
 AND price BETWEEN min_price AND max_price
ORDER BY last_name;
```

Non-equi-join применяется когда:
- нужно сопоставить значение одной таблицы с диапазоном значений другой;
- требуется исключить определённые совпадения (оператор `!=`);
- нужно найти все возможные пары без дублирования (например, `r1.id < r2.id` при self-join);
- вычисляются нарастающие итоги (условие `d1.date >= d2.date`).

#### Ответ от Claude

**Equi-join** — соединение, условие которого содержит исключительно оператор `=`. Это наиболее распространённая форма JOIN; большинство INNER JOIN, LEFT JOIN и прочих пишутся именно так.

**Non-equi-join** — соединение, в условии которого используется любой оператор сравнения, кроме `=` (`<`, `>`, `<=`, `>=`, `<>`, `BETWEEN`). Оба термина — не отдельные виды синтаксиса, а классификация по типу предиката в ON-условии.

Когда нужен non-equi-join:

1. **Диапазонное сопоставление** — попасть в ценовой коридор, налоговую шкалу, временной интервал:

```sql
-- Определить налоговую ставку по размеру зарплаты
SELECT e.name, e.salary, t.rate
FROM employees e
JOIN tax_brackets t
  ON e.salary BETWEEN t.min_salary AND t.max_salary;
```

2. **Генерация уникальных пар без дублей** (self-join):

```sql
SELECT a.name AS player1, b.name AS player2
FROM players a
JOIN players b ON a.id < b.id;
```

3. **Поиск перекрывающихся интервалов**:

```sql
SELECT a.id, b.id
FROM bookings a
JOIN bookings b
  ON a.id <> b.id
 AND a.start_date < b.end_date
 AND a.end_date   > b.start_date;
```

4. **Нарастающий итог** — суммировать все предыдущие строки для каждой текущей.

Важные аспекты производительности: non-equi-join не может использовать обычный B-tree-индекс по равенству так же эффективно, как equi-join. СУБД вынуждена применять Nested Loop или Hash Join с расширенным фильтром. При большом объёме данных нужно следить за наличием подходящих индексов и при необходимости использовать range-индексы или партиционирование.
