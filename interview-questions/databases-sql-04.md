### Вопрос 61. Когда для фильтрации по связанной таблице лучше использовать EXISTS вместо JOIN? В чём семантическое и потенциальное производительностное различие?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [SQL JOIN or EXISTS? Chances Are, You're Doing it Wrong — jOOQ Blog](https://blog.jooq.org/sql-join-or-exists-chances-are-youre-doing-it-wrong/)

INNER JOIN — это «отфильтрованное декартово произведение» двух таблиц: он объединяет строки и возвращает все совпадения. EXISTS реализует **полусоединение (semi-join)**: проверяет лишь факт наличия совпадающей строки, не добавляя столбцы из второй таблицы в результат.

Проблема использования JOIN вместо EXISTS:

> «Если вы выбираете INNER JOIN, чтобы найти клиентов, у которых есть совпадающие сотрудники, вы создаёте декартово произведение между customer и staff, а затем применяете фильтр.»

Это приводит к дублированию строк из главной таблицы, если во второй таблице несколько совпадений. Чтобы убрать дубли, добавляют DISTINCT, но это меняет семантику запроса и добавляет дорогостоящую операцию.

Правильный паттерн с EXISTS:

```sql
SELECT *
FROM customer AS c
WHERE EXISTS (
  SELECT *
  FROM staff AS s
  WHERE c.last_name = s.last_name
)
```

EXISTS останавливает поиск в подзапросе сразу после первого совпадения (short-circuit), что даёт выигрыш в производительности.

Принцип: «Если столбцы из таблицы не нужны в SELECT — переместите её из FROM в WHERE».

#### Ответ от Claude

**Семантическое различие**

- `JOIN` реализует **обычное соединение**: строки из обеих таблиц объединяются и попадают в результирующий набор. Если во второй таблице несколько совпадений на одну строку главной — главная строка будет продублирована.
- `EXISTS` реализует **полусоединение (semi-join)**: проверяет лишь факт существования хотя бы одной совпадающей строки в подзапросе. Столбцы из подзапроса не проецируются в результат, дублей не возникает.

Пример, где JOIN даёт неверный результат:

```sql
-- orders содержит несколько строк на одного customer
-- JOIN продублирует customers
SELECT c.name
FROM customers c
JOIN orders o ON c.id = o.customer_id;  -- дубли!

-- EXISTS возвращает каждого customer ровно один раз
SELECT c.name
FROM customers c
WHERE EXISTS (
    SELECT 1 FROM orders o WHERE o.customer_id = c.id
);
```

**Когда использовать EXISTS**

1. Нужна фильтрация по связанной таблице, но её столбцы не нужны в SELECT.
2. Связь «один ко многим» и дубли в результате недопустимы.
3. Нужен `NOT EXISTS` — аналог антисоединения (anti-join). `NOT IN` с nullable-столбцом небезопасен из-за трёхзначной логики SQL (NULL-сравнение), `NOT EXISTS` лишён этой проблемы.

**Производительность**

- EXISTS обладает **коротким замыканием**: как только найдено первое совпадение, подзапрос прерывается. Это особенно важно при большом числе совпадений во второй таблице.
- JOIN без DISTINCT может обрабатывать избыточные строки, а DISTINCT требует дополнительной сортировки/хеширования.
- Современные оптимизаторы (PostgreSQL, MySQL 8+, SQL Server) часто **автоматически переписывают** коррелированный EXISTS в semi-join на уровне плана выполнения, поэтому разрыв в производительности между хорошо написанным JOIN и EXISTS может быть небольшим. Тем не менее явное использование EXISTS делает намерение очевидным и исключает семантические ошибки.
- Расхождение источников: ряд бенчмарков (SQLShack) показывает, что на non-nullable столбцах EXISTS и JOIN дают сравнимую производительность; преимущество EXISTS становится явным при nullable-столбцах в NOT EXISTS/NOT IN и при большом коэффициенте совпадений.

### Вопрос 62. Что такое LATERAL JOIN (или CROSS APPLY в SQL Server)? Чем он отличается от обычного подзапроса в FROM и в каких задачах незаменим?

**Категория:** БД и SQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 7.2. Table Expressions — §7.2.1.5 LATERAL Subqueries](https://www.postgresql.org/docs/current/queries-table-expressions.html#QUERIES-LATERAL)

Ключевое слово `LATERAL` перед подзапросом или табличной функцией в секции `FROM` позволяет этому подзапросу ссылаться на столбцы элементов `FROM`, стоящих перед ним. Без `LATERAL` каждый подзапрос вычисляется независимо и не может использовать значения других таблиц из того же `FROM`.

Процесс выполнения: для каждой строки (или набора строк) элемента `FROM`, предоставляющего перекрёстно-ссылаемые столбцы, LATERAL-элемент вычисляется с использованием значений этой строки; полученные строки соединяются с исходной строкой как обычно.

Пример с табличной функцией (основной практический сценарий):

```sql
SELECT p1.id, p2.id, v1, v2
FROM polygons p1, polygons p2,
     LATERAL vertices(p1.poly) v1,
     LATERAL vertices(p2.poly) v2
WHERE (v1 <-> v2) < 10 AND p1.id != p2.id;
```

Здесь функция `vertices(polygon)` возвращает множество вершин полигона. Без `LATERAL` передать `p1.poly` в функцию было бы невозможно.

Пример с `LEFT JOIN LATERAL`:

```sql
SELECT m.name
FROM manufacturers m
LEFT JOIN LATERAL get_product_names(m.id) pname ON true
WHERE pname IS NULL;
```

Находит производителей, у которых нет продуктов. `LEFT JOIN` гарантирует, что строки без результата из функции всё равно попадут в выборку (с NULL).

#### Ответ от Claude

**Что такое LATERAL JOIN**

`LATERAL` (PostgreSQL, стандарт SQL:1999) и его эквиваленты `CROSS APPLY` / `OUTER APPLY` (SQL Server, Oracle) — это механизм «коррелированного FROM»: подзапрос или табличная функция в секции FROM получает право ссылаться на столбцы таблиц, перечисленных ранее в том же FROM.

Соответствие между СУБД:
- `CROSS JOIN LATERAL` = `CROSS APPLY`
- `LEFT JOIN LATERAL ... ON true` = `OUTER APPLY`

**Отличие от обычного подзапроса в FROM**

| Свойство | Обычный подзапрос в FROM | LATERAL подзапрос |
|---|---|---|
| Ссылки на внешние столбцы | Запрещены | Разрешены |
| Порядок выполнения | Один раз, независимо | Для каждой строки внешней таблицы |
| Возвращаемых строк | Фиксированный набор | Зависит от текущей строки |
| Аналог | Производная таблица | Коррелированный подзапрос, но в FROM |

```sql
-- Обычный подзапрос — НЕ может ссылаться на u.id
SELECT u.name, s.total
FROM users u,
     (SELECT SUM(amount) AS total FROM orders WHERE user_id = u.id) s;
-- Ошибка: u.id недоступен

-- LATERAL — может
SELECT u.name, s.total
FROM users u,
     LATERAL (SELECT SUM(amount) AS total FROM orders WHERE user_id = u.id) s;
```

**Задачи, где LATERAL незаменим**

1. **Top-N строк на каждую группу** — классический «N последних заказов для каждого клиента»:

```sql
SELECT c.name, o.order_date, o.amount
FROM customers c
CROSS JOIN LATERAL (
    SELECT order_date, amount
    FROM orders
    WHERE customer_id = c.id
    ORDER BY order_date DESC
    LIMIT 3
) o;
```

Без LATERAL это решается громоздкими оконными функциями с дополнительным CTE или коррелированным подзапросом в SELECT (который не может вернуть несколько строк).

2. **Передача аргументов в set-returning функцию** — единственный способ передать значение строки в функцию, возвращающую таблицу:

```sql
SELECT p.id, v.vertex
FROM polygons p
CROSS JOIN LATERAL unnest(p.vertices_array) AS v(vertex);
```

3. **Вычисление производного набора строк, зависящего от предыдущей таблицы**, когда результат не может быть выражен простым JOIN-предикатом.

4. **LEFT JOIN LATERAL для «необязательных» вычислений** — когда подзапрос может вернуть ноль строк, а строка главной таблицы должна сохраниться:

```sql
SELECT e.name, last_review.score
FROM employees e
LEFT JOIN LATERAL (
    SELECT score
    FROM performance_reviews
    WHERE employee_id = e.id
    ORDER BY review_date DESC
    LIMIT 1
) last_review ON true;
```

**Производительность**

LATERAL выполняет подзапрос для каждой строки внешней таблицы (аналог Nested Loop). Это эффективно, если внутренний запрос использует индекс по коррелированному столбцу. При отсутствии индекса — полный перебор для каждой строки, что может быть медленнее альтернативных решений с оконными функциями.

### Вопрос 63. Как наличие индекса на столбце соединения влияет на выбор алгоритма JOIN оптимизатором? В каких сценариях индекс на join-колонке не помогает?

**Категория:** БД и SQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Use The Index, Luke — Improving Join Performance](https://use-the-index-luke.com/sql/join)

Правильная индексация JOIN зависит от того, какой из трёх алгоритмов использует база данных.

**Nested Loop Join (соединение вложенными циклами).** База данных обращается ко внутренней (inner) таблице ровно по одному разу на каждую строку внешней (outer) таблицы. Это эквивалентно двум последовательным запросам: «для каждой строки из driving-запроса выполни lookup во второй таблице». Индекс на join-колонке внутренней таблицы здесь критичен: без него для каждой строки внешней таблицы придётся делать полный проход по внутренней. С индексом каждый lookup — это быстрый поиск по B-tree. Алгоритм эффективен, когда driving-запрос возвращает небольшое количество строк.

**Hash Join (соединение с хешированием).** Одна таблица целиком загружается в хеш-таблицу в памяти (в качестве ключей используются join-атрибуты). Затем вторая таблица сканируется, и для каждой строки выполняется поиск по хеш-таблице. Ключевое свойство: **индексирование join-предикатов не улучшает производительность Hash Join**. Хеш-таблица сама обеспечивает быстрый поиск, делая B-tree-индексы на join-колонках избыточными. Для ускорения Hash Join нужно индексировать независимые условия WHERE, которые фильтруют одну из таблиц до построения хеша, — это уменьшает размер хеш-таблицы.

**Sort-Merge Join (соединение сортировкой и слиянием).** Обе стороны сортируются по join-атрибутам, затем проходятся параллельно. Индексы на join-колонках сами по себе не нужны алгоритму, однако **если индекс обеспечивает нужный порядок строк**, оптимизатор может использовать Index Scan вместо явной сортировки и тем самым исключить дорогую операцию Sort. Индексировать join-предикаты для ускорения самого слияния нет смысла.

**Вывод:** индекс на join-колонке помогает только при Nested Loop Join (для lookup на внутренней таблице) и косвенно при Merge Join (для устранения сортировки). При Hash Join индекс на join-колонке игнорируется.

#### Ответ от Claude

Оптимизатор PostgreSQL (и большинства других СУБД) выбирает алгоритм JOIN на основе оценки стоимости (cost-based optimization). Ниже разобраны три алгоритма, роль индекса в каждом и сценарии, когда индекс не помогает.

---

**1. Nested Loop Join**

Принцип: для каждой строки внешней таблицы выполняется один проход по внутренней.

```sql
-- Оптимизатор выберет Nested Loop при маленькой выборке + индекс на join-колонке:
EXPLAIN SELECT *
FROM tenk1 t1
JOIN tenk2 t2 ON t1.unique2 = t2.unique2
WHERE t1.unique1 < 10;

-- Пример плана:
-- Nested Loop
--   -> Bitmap Heap Scan on tenk1 t1  (rows=10)
--   -> Index Scan using tenk2_unique2 on tenk2 t2
--        Index Cond: (unique2 = t1.unique2)   <-- lookup по индексу
```

Индекс на `t2.unique2` используется напрямую: значение из каждой строки t1 передаётся как параметр в Index Scan по t2. Алгоритм эффективен, когда внешняя выборка мала (оптимизатор PostgreSQL переключается на Hash Join уже при ~100 строках в примере из документации).

---

**2. Hash Join**

Принцип: одна таблица хешируется в память; другая сканируется и проверяется по хеш-таблице.

```sql
-- При большей выборке оптимизатор переходит на Hash Join:
EXPLAIN SELECT *
FROM tenk1 t1
JOIN tenk2 t2 ON t1.unique2 = t2.unique2
WHERE t1.unique1 < 100;

-- Пример плана:
-- Hash Join
--   Hash Cond: (t2.unique2 = t1.unique2)
--   -> Seq Scan on tenk2 t2              <-- полный скан, индекс не используется
--   -> Hash
--        -> Bitmap Heap Scan on tenk1 t1
```

Индекс на `t2.unique2` **не используется**: обе таблицы сканируются последовательно. Хеш-таблица сама даёт O(1)-lookup. Индексировать join-колонку ради Hash Join бессмысленно.

Чтобы ускорить Hash Join, нужен индекс на **фильтрующем** условии WHERE, а не на join-предикате:

```sql
-- Индекс на sale_date сокращает число строк ДО построения хеша:
CREATE INDEX idx_sales_date ON sales(sale_date);

SELECT e.name, s.amount
FROM employees e
JOIN sales s ON e.id = s.employee_id
WHERE s.sale_date >= '2024-01-01';
-- Индекс на sale_date используется при Seq Scan/Bitmap Scan на sales,
-- уменьшая хеш-таблицу. Индекс на employee_id Hash Join не поможет.
```

---

**3. Merge Join**

Принцип: обе стороны сортируются по join-ключу, затем проходятся параллельно.

```sql
EXPLAIN SELECT *
FROM tenk1 t1
JOIN onek t2 ON t1.unique2 = t2.unique2
WHERE t1.unique1 < 100;

-- Пример плана:
-- Merge Join
--   Merge Cond: (t1.unique2 = t2.unique2)
--   -> Index Scan using tenk1_unique2 on tenk1 t1   <-- уже отсортировано
--   -> Index Scan using onek_unique2 on onek t2      <-- уже отсортировано
```

Индекс на join-колонке **косвенно помогает**: если Index Scan возвращает строки в нужном порядке, явная операция Sort исключается. Но сам алгоритм слияния индекс не использует для lookup — он просто читает обе отсортированные последовательности.

Если данные уже не отсортированы и индекса нет, Merge Join добавляет дорогую Sort-ноду, и оптимизатор часто предпочитает Hash Join.

---

**Сценарии, когда индекс на join-колонке не помогает**

| Сценарий | Почему индекс бесполезен |
|---|---|
| Hash Join (любой размер таблиц) | Обе стороны сканируются целиком; хеш-таблица заменяет lookup |
| Большая внешняя выборка при Nested Loop | Стоимость N index-lookup > стоимость одного Hash Join; оптимизатор переключается на Hash Join |
| Merge Join без сортировки по индексу | Сам merge-алгоритм не делает lookup; индекс нужен только для устранения Sort |
| Низкая селективность join-колонки | Индекс возвращает большую долю таблицы; Sequential Scan дешевле Index Scan |
| Параллельный Hash Join | Параллельные воркеры делят хеш-таблицу; индекс на join-колонке всё равно не читается |
| JOIN по выражению / функции | Индекс по колонке не покрывает `LOWER(email) = LOWER(t2.email)`; нужен функциональный индекс или индекс не поможет совсем |
| Маленькие таблицы | Sequential Scan дешевле любого index lookup; оптимизатор не использует индекс вообще |

---

**Практические выводы**

1. Индекс на join-колонке **нужен** в первую очередь для Nested Loop Join на внутренней (probe) таблице — это классический use case.
2. Для Hash Join индексируй **фильтрующие** условия (WHERE-предикаты на отдельных таблицах), а не join-предикаты.
3. Для Merge Join индекс на join-колонке устраняет Sort, но не ускоряет само слияние.
4. Оптимизатор всегда выбирает план с наименьшей оценочной стоимостью. Проверяй реальный план через `EXPLAIN (ANALYZE, BUFFERS)` — он покажет, используется ли индекс и на каком шаге.
5. Параметры `enable_nestloop`, `enable_hashjoin`, `enable_mergejoin` позволяют принудительно отключить алгоритм для диагностики.

### Вопрос 64. Что такое CTE (Common Table Expression)? Чем WITH отличается от подзапроса в FROM и когда нужен WITH RECURSIVE?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 7.8. WITH Queries (Common Table Expressions)](https://www.postgresql.org/docs/current/queries-with.html)

Конструкция WITH предоставляет способ записи вспомогательных операторов для использования в рамках более крупного запроса. Эти вспомогательные операторы, часто называемые Common Table Expressions (CTE), можно рассматривать как определения временных таблиц, существующих только в рамках одного запроса.

Модификатор RECURSIVE меняет WITH с простого синтаксического удобства в функциональность, которая иначе недостижима в стандартном SQL. Используя RECURSIVE, запрос WITH может ссылаться на собственный вывод. Структура рекурсивного запроса:

```sql
WITH RECURSIVE t(n) AS (
    VALUES (1)           -- нерекурсивная часть (anchor)
    UNION ALL
    SELECT n+1 FROM t WHERE n < 100  -- рекурсивная часть
)
SELECT sum(n) FROM t;
```

Нерекурсивный терм вычисляется один раз и образует начальное отношение. Рекурсивный терм вычисляется многократно, на каждой итерации действуя на результат предыдущей итерации. Итерации останавливаются, когда рекурсивный терм возвращает пустое отношение.

#### Ответ от Claude

**Что такое CTE**

CTE (Common Table Expression) — именованный подзапрос, определяемый в начале SQL-оператора через ключевое слово `WITH`. CTE позволяет разбить сложный запрос на читаемые именованные части, которые действуют как временные виртуальные таблицы в рамках одного оператора.

```sql
WITH regional_sales AS (
    SELECT region, SUM(amount) AS total_sales
    FROM orders
    GROUP BY region
),
top_regions AS (
    SELECT region FROM regional_sales WHERE total_sales > 10000
)
SELECT r.region, o.product, SUM(o.amount) AS revenue
FROM orders o
JOIN top_regions r ON o.region = r.region
GROUP BY r.region, o.product;
```

**Чем CTE отличается от подзапроса в FROM**

| Аспект | Подзапрос в FROM | CTE (WITH) |
|--------|-----------------|------------|
| Именование | Только псевдоним | Именованное, переиспользуемое |
| Повторное использование | Нельзя ссылаться дважды | Можно ссылаться из нескольких мест |
| Читаемость | Ухудшается при вложенности | Хорошая, линейная структура |
| Материализация | Зависит от оптимизатора | В PostgreSQL по умолчанию материализуется (начиная с 12 — оптимизатор может инлайнить) |
| Рекурсия | Невозможна | Возможна через RECURSIVE |

Важный нюанс: до PostgreSQL 12 CTE всегда материализовались (вычислялись один раз в отдельный временный результат). С PostgreSQL 12 оптимизатор может инлайнить CTE в родительский запрос, если это выгоднее. Явное управление: `WITH ... AS MATERIALIZED (...)` или `WITH ... AS NOT MATERIALIZED (...)`.

**Когда нужен WITH RECURSIVE**

Рекурсивный CTE необходим для запросов по иерархическим и графовым структурам, где глубина дерева заранее неизвестна:

```sql
-- Вывод всей иерархии сотрудников от CEO
WITH RECURSIVE employee_hierarchy AS (
    -- Нерекурсивная часть: корень иерархии
    SELECT id, name, manager_id, 1 AS level
    FROM employees
    WHERE manager_id IS NULL

    UNION ALL

    -- Рекурсивная часть: добавляем подчинённых
    SELECT e.id, e.name, e.manager_id, h.level + 1
    FROM employees e
    JOIN employee_hierarchy h ON e.manager_id = h.id
)
SELECT level, name FROM employee_hierarchy ORDER BY level, name;
```

Типичные задачи для WITH RECURSIVE:
1. Обход дерева категорий (корень → листья)
2. Поиск пути в графе
3. Вычисление транзитивного замыкания
4. Генерация числовых серий (замена generate_series на тех СУБД, где её нет)

**Защита от бесконечной рекурсии:** PostgreSQL прекращает рекурсию при возврате пустого набора строк. Для явного ограничения глубины применяют `CYCLE`-клаузу (PostgreSQL 14+) или счётчик уровней в условии `WHERE level < N`.

### Вопрос 65. Чем обычное представление (VIEW) отличается от материализованного (MATERIALIZED VIEW)? Когда материализованное представление даёт значительный выигрыш и как его обновлять?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 39.3. Materialized Views](https://www.postgresql.org/docs/current/rules-materializedviews.html)

Материализованные представления в PostgreSQL используют систему правил аналогично обычным представлениям, но сохраняют результаты в табличной форме. Ключевое отличие: материализованные представления хранят фактические данные, а не выполняют запрос при каждом обращении.

Исходный запрос, использованный для инициализации представления, хранится — точно так же, как запрос обычного представления, — чтобы впоследствии можно было обновить данные:

```sql
CREATE MATERIALIZED VIEW mymatview AS SELECT * FROM mytab;
REFRESH MATERIALIZED VIEW mymatview;
```

В отличие от обычных представлений, материализованные представления не обновляются автоматически при изменении базовых таблиц. Требуется явный вызов `REFRESH MATERIALIZED VIEW`. Команда `REFRESH MATERIALIZED VIEW CONCURRENTLY` позволяет обновлять представление без эксклюзивной блокировки (необходим уникальный индекс).

Документация показывает пример: после создания индекса по материализованному представлению сводки продаж время запроса сократилось с 188 мс до 0.117 мс.

#### Ответ от Claude

**Сравнение VIEW и MATERIALIZED VIEW**

| Аспект | VIEW (обычное) | MATERIALIZED VIEW |
|--------|---------------|-------------------|
| Хранение данных | Нет (виртуальная таблица) | Да (физически на диске) |
| Выполнение запроса | При каждом обращении | При создании и при REFRESH |
| Актуальность данных | Всегда актуальна | До следующего REFRESH |
| Индексы | Нельзя создать индекс на view | Можно создавать индексы |
| Обновление при изменении базовых данных | Автоматически | Только вручную (REFRESH) |
| Производительность чтения | Зависит от сложности базового запроса | Высокая (как обычная таблица) |

**Синтаксис**

```sql
-- Обычное представление
CREATE VIEW sales_by_region AS
SELECT region, SUM(amount) AS total
FROM orders
GROUP BY region;

-- Материализованное представление
CREATE MATERIALIZED VIEW sales_by_region_mv AS
SELECT region, SUM(amount) AS total
FROM orders
GROUP BY region;

-- Индекс на материализованном представлении
CREATE INDEX idx_mv_region ON sales_by_region_mv (region);

-- Обновление данных (с блокировкой)
REFRESH MATERIALIZED VIEW sales_by_region_mv;

-- Обновление без блокировки (нужен уникальный индекс)
REFRESH MATERIALIZED VIEW CONCURRENTLY sales_by_region_mv;
```

**Когда материализованное представление даёт значительный выигрыш**

1. **Тяжёлые аналитические запросы** — многоуровневые агрегации, JOIN нескольких больших таблиц, оконные функции по миллионам строк. Если такой запрос выполняется секунды, а результат читается тысячи раз — материализация оправдана.

2. **Отчёты и дашборды** — данные для BI-инструментов можно обновлять по расписанию (ночью), а в течение дня читать из быстрой структуры.

3. **Агрегированные снапшоты** — суммарные показатели по историческим данным, которые не меняются часто.

4. **Запросы к внешним источникам (Foreign Data Wrappers)** — материализация результатов избавляет от сетевых запросов к внешней БД при каждом чтении.

**Ограничения**

- Данные устаревают до следующего REFRESH. В сценариях, где актуальность критична, материализованные представления не подходят.
- `REFRESH MATERIALIZED VIEW` (без CONCURRENTLY) блокирует представление на время обновления — читатели ждут.
- `REFRESH CONCURRENTLY` требует уникального индекса на представлении.

**Расхождения между источниками:** PostgreSQL-документация и Baeldung единодушны в основных характеристиках. Существуют мнения, что `NOT MATERIALIZED` опцию в обычных CTE стоит использовать для оптимизации вместо материализованных представлений в ряде случаев — но это разные инструменты с разными областями применения.

### Вопрос 66. Чем UNION отличается от UNION ALL? Когда использование UNION вместо UNION ALL является ошибкой?

**Категория:** БД и SQL · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 7.4. Combining Queries (UNION, INTERSECT, EXCEPT)](https://www.postgresql.org/docs/current/queries-union.html)

Результаты двух запросов можно объединять с помощью операций над множествами: объединение (union), пересечение (intersection) и разность (difference).

`UNION` объединяет строки двух запросов. Результат запроса UNION не содержит дубликатов строк — аналогично `SELECT DISTINCT`. Операция UNION ALL не устраняет дублирующиеся строки.

Требования к совместимости: оба запроса должны возвращать одинаковое число столбцов, а соответствующие столбцы должны иметь совместимые типы данных.

Стандарт SQL требует, чтобы `DISTINCT` было поведением по умолчанию при слиянии запросов — в отличие от `SELECT`, где по умолчанию используется `ALL`. Это кажется несогласованным, но вряд ли изменится.

#### Ответ от Claude

**Принципиальное различие**

- `UNION` — объединяет строки из двух запросов и **удаляет все дубликаты** в результирующем наборе. Для удаления дубликатов СУБД выполняет дополнительную операцию (сортировка или хеширование) — это затрачивает CPU и память.
- `UNION ALL` — объединяет строки из двух запросов **без удаления дубликатов**. Просто склеивает наборы строк.

```sql
-- Таблица a: {1, 2, 3}
-- Таблица b: {2, 3, 4}

SELECT val FROM a UNION SELECT val FROM b;
-- Результат: {1, 2, 3, 4}  <-- дубликаты 2 и 3 удалены

SELECT val FROM a UNION ALL SELECT val FROM b;
-- Результат: {1, 2, 3, 2, 3, 4}  <-- все строки обоих запросов
```

**Производительность**

`UNION` добавляет шаг дедупликации: СУБД обязана отсортировать или захешировать весь результат для нахождения дубликатов. На больших наборах данных это:
- Занимает дополнительную память.
- Потребляет CPU-время на сортировку.
- При нехватке `work_mem` сбрасывает данные на диск.

`UNION ALL` пропускает этот шаг полностью и всегда быстрее.

**Когда UNION вместо UNION ALL является ошибкой**

1. **Эмуляция FULL OUTER JOIN через LEFT JOIN + RIGHT JOIN**: ровно совпадающие строки попадают в оба запроса — UNION здесь необходим. Замена на UNION ALL приведёт к дублям. Это один из редких случаев, когда UNION семантически обязателен.

2. **Когда данные из двух запросов по определению не пересекаются**: объединение продаж за разные годы, объединение записей из разных партиций по непересекающимся диапазонам. Использование UNION здесь несёт накладные расходы дедупликации без практической пользы — правильнее UNION ALL.

3. **Сбор данных для агрегации**: если UNION ALL + GROUP BY заменяется на UNION, агрегация произойдёт **после** удаления дублей, что изменит результаты COUNT, SUM.

**Практическое правило:** UNION ALL — выбор по умолчанию. UNION используется только тогда, когда семантически требуется удаление дублей (данные из обоих запросов могут содержать одинаковые строки, и дубли нежелательны).

### Вопрос 67. Что такое DISTINCT ON в PostgreSQL? Чем он отличается от стандартного DISTINCT и как используется на практике?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: SELECT — DISTINCT clause](https://www.postgresql.org/docs/current/sql-select.html#SQL-DISTINCT)

Если указывается `SELECT DISTINCT ON (expression [, ...])`, из результирующего набора исключаются все строки, в которых заданные выражения совпадают. Из каждого набора дублирующихся строк сохраняется только первая строка.

Первая строка каждого набора непредсказуема, если не задано `ORDER BY` для обеспечения нужного порядка строк. Выражение(я) `DISTINCT ON` должны совпадать с крайними левыми выражениями `ORDER BY`.

```sql
SELECT DISTINCT ON (location) location, time, report
    FROM weather_reports
    ORDER BY location, time DESC;
```

Запрос выбирает самый последний отчёт о погоде для каждого места: строки группируются по `location`, внутри каждой группы сортируются по убыванию `time`, и возвращается первая строка группы.

#### Ответ от Claude

**Стандартный DISTINCT**

`SELECT DISTINCT` удаляет дубли строк по **всем** выбранным столбцам. Уникальность определяется по всей строке целиком.

```sql
SELECT DISTINCT department, city
FROM employees;
-- Возвращает уникальные пары (department, city)
```

**DISTINCT ON — PostgreSQL-специфичное расширение**

`SELECT DISTINCT ON (column_list)` возвращает **одну строку из каждой группы**, где группировка определяется по `column_list`, но в результат включаются любые другие столбцы таблицы. Какая именно строка сохраняется из группы — определяется `ORDER BY`.

```sql
-- Для каждого клиента — только последний заказ
SELECT DISTINCT ON (customer_id)
    customer_id,
    order_date,
    amount
FROM orders
ORDER BY customer_id, order_date DESC;
```

Здесь `DISTINCT ON (customer_id)` означает «для каждого уникального `customer_id` взять только одну строку». `ORDER BY customer_id, order_date DESC` — из строк с одинаковым `customer_id` взять ту, у которой наибольшая `order_date`.

**Ключевое требование:** выражения в `DISTINCT ON (...)` должны быть самыми левыми в `ORDER BY`. Это необходимо, чтобы СУБД могла определить «первую» строку в каждой группе.

**Сравнение с альтернативами**

Та же задача «по одной строке из каждой группы» без DISTINCT ON решается громоздко:

```sql
-- Через ROW_NUMBER (ANSI SQL, работает везде)
SELECT customer_id, order_date, amount
FROM (
    SELECT *,
           ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY order_date DESC) AS rn
    FROM orders
) t
WHERE rn = 1;
```

`DISTINCT ON` в PostgreSQL — более лаконичный синтаксис для того же результата.

**Когда применять DISTINCT ON**

1. Нужна «последняя» / «первая» строка по каждой группе.
2. Нужен «представитель» каждой группы с произвольными другими столбцами.
3. Нужна альтернатива коррелированному подзапросу или оконным функциям для простых случаев.

**Ограничение:** DISTINCT ON является расширением PostgreSQL и не переносится на другие СУБД (MySQL, SQL Server, Oracle).

### Вопрос 68. Что такое COALESCE и NULLIF? Как они используются для обработки NULL-значений в SQL?

**Категория:** БД и SQL · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 9.18. Conditional Expressions — COALESCE, NULLIF](https://www.postgresql.org/docs/current/functions-conditional.html)

`COALESCE(value [, ...])` возвращает первый из своих аргументов, который не является NULL. NULL возвращается только в том случае, если все аргументы равны NULL. Функция часто используется для подстановки значения по умолчанию при отображении данных:

```sql
SELECT COALESCE(description, short_description, '(none)') ...
```

Как и при работе с оператором CASE, аргументы вычисляются только при необходимости — то есть аргументы правее первого ненулевого значения не вычисляются.

`NULLIF(value1, value2)` возвращает NULL, если `value1` равен `value2`; в противном случае возвращает `value1`. Это обратная операция к COALESCE: используется для преобразования специальных значений (например, пустой строки или нуля) в NULL.

```sql
SELECT NULLIF(value, '(none)') ...
```

#### Ответ от Claude

**COALESCE — первое ненулевое значение**

`COALESCE(expr1, expr2, ..., exprN)` просматривает аргументы слева направо и возвращает первый, не равный NULL. Это стандартная функция SQL (есть во всех СУБД).

```sql
-- Подставить значение по умолчанию вместо NULL
SELECT name, COALESCE(phone, email, 'Контакт не указан') AS contact
FROM customers;

-- Заменить NULL на 0 в числовых вычислениях
SELECT product, COALESCE(discount, 0) AS discount
FROM products;

-- COALESCE в агрегации: игнорировать NULL при конкатенации
SELECT COALESCE(first_name, '') || ' ' || COALESCE(last_name, '') AS full_name
FROM users;
```

Важно: аргументы вычисляются **лениво** — как только найдено ненулевое значение, остальные не вычисляются (short-circuit evaluation).

**NULLIF — превратить значение в NULL**

`NULLIF(expr1, expr2)` возвращает NULL, если `expr1 = expr2`, иначе возвращает `expr1`. По сути: «если значение равно заданному sentinel-значению — заменить его на NULL».

```sql
-- Избежать деления на ноль (делаем знаменатель NULL вместо 0)
SELECT total / NULLIF(quantity, 0) AS unit_price
FROM order_items;

-- Превратить пустую строку в NULL
SELECT NULLIF(description, '') AS description
FROM products;

-- Убрать «нулевые» метки-заменители
SELECT NULLIF(status, 'N/A') AS status
FROM applications;
```

**COALESCE + NULLIF — типичная комбинация**

```sql
-- Среднее только по ненулевым и непустым значениям:
-- NULLIF превращает '' в NULL, COALESCE используется для fallback
SELECT AVG(NULLIF(score::numeric, 0)) AS avg_nonzero_score
FROM survey_responses;
```

**Сравнение**

| Функция | Что делает | Когда использовать |
|---------|-----------|-------------------|
| `COALESCE(a, b, c)` | Возвращает первый не-NULL | Замена NULL на дефолтное значение |
| `NULLIF(a, sentinel)` | Возвращает NULL если a = sentinel | Преобразование «ложных» значений в NULL |
| `IS NULL / IS NOT NULL` | Проверка на NULL | Условная фильтрация |

**Различия между СУБД:** `COALESCE` и `NULLIF` — стандарт SQL, работают везде. `ISNULL` (SQL Server / MySQL) принимает только два аргумента и эквивалентна `COALESCE(a, b)`. `NVL` (Oracle) — тоже двухаргументный аналог COALESCE.

### Вопрос 69. Что такое TRUNCATE? Чем он отличается от DELETE и DROP с точки зрения DDL/DML, транзакций, производительности и структуры таблицы?

**Категория:** БД и SQL · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Comparing TRUNCATE TABLE, DELETE, and DROP TABLE in SQL — LearnSQL.com](https://learnsql.com/blog/difference-between-truncate-delete-and-drop-table-in-sql/)

Три команды SQL для удаления данных принципиально отличаются по поведению:

- **DELETE** — DML-команда. Удаляет одну, несколько или все строки таблицы. Поддерживает `WHERE`-условие для выборочного удаления. Всегда обратима через ROLLBACK. Работает медленнее других команд — удаляет строки по одной с ведением журнала каждой операции.
- **TRUNCATE TABLE** — DDL-команда. Удаляет все строки таблицы, сохраняя её структуру. Не поддерживает `WHERE`. Обратимость зависит от СУБД: в PostgreSQL и SQL Server — обратима (транзакционная), в MySQL и Oracle — не обратима. Выполняется значительно быстрее DELETE.
- **DROP TABLE** — DDL-команда. Удаляет всю таблицу целиком: данные, структуру, индексы, ограничения, триггеры и права доступа. Уничтожает таблицу полностью.

Поведение автоинкремента:
- DELETE сохраняет текущее значение автоинкрементного счётчика.
- TRUNCATE сбрасывает автоинкрементный счётчик на начальное значение.

#### Ответ от Claude

**Сводная таблица**

| Характеристика | DELETE | TRUNCATE | DROP TABLE |
|---|---|---|---|
| Тип команды | DML | DDL | DDL |
| Что удаляет | Выбранные строки | Все строки | Всю таблицу (структура + данные) |
| WHERE-условие | Да | Нет | Нет |
| Сохранение структуры | Да | Да | Нет |
| Индексы, ограничения | Сохраняются | Сохраняются | Удаляются |
| Транзакционность | Всегда | Зависит от СУБД | Зависит от СУБД |
| Производительность | Медленнее | Быстрее | Быстро |
| AUTO_INCREMENT | Сохраняется | Сбрасывается | Не применимо |
| Триггеры BEFORE/AFTER DELETE | Срабатывают | Не срабатывают | Не применимо |

**DELETE**

```sql
DELETE FROM orders WHERE status = 'cancelled';  -- выборочное удаление
DELETE FROM orders;                              -- удалить все строки (медленно)
```

DELETE обрабатывает строки по одной, записывая каждое удаление в журнал транзакций (WAL/redo log). Это позволяет ROLLBACK, но делает операцию медленной на больших таблицах. В PostgreSQL (MVCC) удалённые строки физически не исчезают сразу — они помечаются как мёртвые и убираются VACUUM позже.

**TRUNCATE**

```sql
TRUNCATE TABLE orders;
TRUNCATE TABLE orders, order_items;  -- несколько таблиц сразу (PostgreSQL)
TRUNCATE TABLE orders RESTART IDENTITY;  -- сбросить счётчик
```

TRUNCATE удаляет все страницы данных с минимальным журналированием. Это на порядок быстрее DELETE при большом числе строк. В PostgreSQL TRUNCATE транзакционен — можно выполнить ROLLBACK. При наличии FOREIGN KEY ссылок нужен CASCADE: `TRUNCATE orders CASCADE`.

**DROP TABLE**

```sql
DROP TABLE orders;               -- ошибка, если таблица не существует
DROP TABLE IF EXISTS orders;     -- безопаснее
DROP TABLE orders CASCADE;       -- удалить и зависимые объекты
```

DROP полностью уничтожает таблицу. Восстановление возможно только из бэкапа. В MySQL операция необратима; в PostgreSQL и SQL Server — обратима в рамках транзакции.

**Когда что использовать**

- DELETE: удаление конкретных строк по условию; когда нужны триггеры или журнал изменений.
- TRUNCATE: быстрая очистка таблицы целиком (например, перед повторной загрузкой данных в ETL).
- DROP: полное уничтожение таблицы (демо-среда, удаление устаревшей схемы).

### Вопрос 70. Что такое хранимая процедура и функция в SQL? Чем они принципиально отличаются?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Function vs. Stored Procedure in PostgreSQL — Baeldung on SQL](https://www.baeldung.com/sql/postgresql-function-vs-stored-procedure)

Функции и хранимые процедуры в PostgreSQL — оба механизма для инкапсуляции переиспользуемой бизнес-логики на стороне СУБД. Однако между ними есть принципиальные различия:

**Возвращаемые значения:** функции должны возвращать значение (скалярное или табличное) и имеют тип возврата. Хранимые процедуры могут не возвращать ничего или возвращать несколько наборов результатов через параметры INOUT.

**Управление транзакциями:** функции не могут использоваться для управления транзакциями в SQL. Хранимые процедуры поддерживают транзакционное управление — `COMMIT` и `ROLLBACK` внутри тела процедуры.

**Использование в SQL-выражениях:** функции можно использовать в SELECT, WHERE, HAVING. Хранимые процедуры вызываются через `CALL` и не могут быть встроены в SQL-выражения.

**Изменение состояния БД:** функции не должны изменять состояние базы данных — они используются для вычислений, трансформаций и проверок. Хранимые процедуры могут выполнять INSERT, UPDATE, DELETE.

#### Ответ от Claude

**Функция (FUNCTION)**

Функция — именованный блок кода, принимающий входные параметры и обязательно возвращающий значение (скалярное, табличное, composite-тип).

```sql
-- PostgreSQL: функция вычисления НДС
CREATE OR REPLACE FUNCTION calculate_vat(price NUMERIC, rate NUMERIC DEFAULT 0.20)
RETURNS NUMERIC AS $$
BEGIN
    RETURN price * rate;
END;
$$ LANGUAGE plpgsql IMMUTABLE;

-- Использование в SELECT:
SELECT product_name, price, calculate_vat(price) AS vat
FROM products;
```

Ключевые свойства:
- Может использоваться везде, где допустимо выражение: SELECT, WHERE, JOIN ON, DEFAULT.
- Не может содержать COMMIT/ROLLBACK (в большинстве СУБД).
- Делится на: IMMUTABLE (не читает и не меняет БД), STABLE (может читать, не меняет), VOLATILE (может всё).

**Хранимая процедура (STORED PROCEDURE)**

Процедура — именованный блок кода для выполнения операций (бизнес-логика, ETL, пакетная обработка). Не обязана возвращать значение; может содержать транзакционную логику.

```sql
-- PostgreSQL: процедура перевода средств
CREATE OR REPLACE PROCEDURE transfer_funds(
    from_account INT,
    to_account   INT,
    amount       NUMERIC
)
LANGUAGE plpgsql AS $$
BEGIN
    UPDATE accounts SET balance = balance - amount WHERE id = from_account;
    UPDATE accounts SET balance = balance + amount WHERE id = to_account;
    COMMIT;  -- явный COMMIT внутри процедуры (PostgreSQL 11+)
END;
$$;

-- Вызов:
CALL transfer_funds(101, 202, 500.00);
```

**Сравнительная таблица**

| Критерий | FUNCTION | STORED PROCEDURE |
|---|---|---|
| Возврат значения | Обязателен | Необязателен |
| Использование в SELECT / WHERE | Да | Нет |
| COMMIT / ROLLBACK внутри | Нет (PostgreSQL < 14) | Да (PostgreSQL 11+) |
| Параметры OUT / INOUT | Ограниченно | Полная поддержка |
| Основное назначение | Вычисления, трансформации | Бизнес-логика, пакетная обработка |
| Вызов | Как выражение или SELECT func() | CALL procedure() |

**Различия между СУБД:** В MySQL до версии 8.0 хранимые процедуры появились с версии 5.0, а функции — ещё раньше. В Oracle разграничение аналогичное, но процедуры дополнительно поддерживают IN/OUT-параметры на уровне пакетов. В SQL Server — схожая модель, но функции жёстко ограничены (нельзя даже временные таблицы в большинстве типов функций).

### Вопрос 71. Что такое секционирование (partitioning) таблицы в PostgreSQL? Какие стратегии секционирования существуют и что такое partition pruning?

**Категория:** БД и SQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 5.12. Table Partitioning](https://www.postgresql.org/docs/current/ddl-partitioning.html)

Секционирование — разбиение логически большой таблицы на более мелкие физические части. PostgreSQL поддерживает три вида декларативного секционирования:

**Секционирование по диапазону (RANGE):** таблица секционируется по диапазонам значений ключевого столбца. Каждая секция охватывает непересекающийся диапазон. Используется для временных рядов: дата или временная метка.

**Секционирование по списку (LIST):** явно задаётся список ключевых значений, принадлежащих каждой секции.

**Секционирование по хешу (HASH):** значение ключевого столбца хешируется, каждая секция принимает строки, у которых остаток от деления хеша на заданный модуль равен заданному значению.

Преимущества секционирования:
- Существенное ускорение запросов, обращающихся к небольшой части данных.
- Более эффективные последовательные сканирования отдельных секций.
- Быстрое удаление большого объёма данных через `DROP TABLE` или `DETACH PARTITION`.
- Перенос редко используемых данных на более медленные носители.

#### Ответ от Claude

**Что такое секционирование таблицы**

Секционирование — это разбиение одной логической «родительской» таблицы на несколько физических «дочерних» таблиц-секций. С точки зрения SQL запросы выполняются к родительской таблице, а СУБД сама определяет, в каких секциях искать данные.

**Три стратегии секционирования**

**1. RANGE (по диапазону)**

```sql
CREATE TABLE orders (
    id          BIGINT,
    order_date  DATE NOT NULL,
    amount      NUMERIC
) PARTITION BY RANGE (order_date);

CREATE TABLE orders_2023 PARTITION OF orders
    FOR VALUES FROM ('2023-01-01') TO ('2024-01-01');

CREATE TABLE orders_2024 PARTITION OF orders
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');
```

Когда использовать: временные ряды (дата, месяц, год). Наиболее распространённая стратегия для архивации и удаления старых данных.

**2. LIST (по списку)**

```sql
CREATE TABLE customers (
    id      BIGINT,
    country VARCHAR(2)
) PARTITION BY LIST (country);

CREATE TABLE customers_eu PARTITION OF customers
    FOR VALUES IN ('DE', 'FR', 'IT', 'ES');

CREATE TABLE customers_us PARTITION OF customers
    FOR VALUES IN ('US', 'CA');
```

Когда использовать: категориальные данные с малым числом дискретных значений (страна, регион, тип).

**3. HASH (по хешу)**

```sql
CREATE TABLE events (
    id   BIGINT,
    data JSONB
) PARTITION BY HASH (id);

CREATE TABLE events_0 PARTITION OF events FOR VALUES WITH (MODULUS 4, REMAINDER 0);
CREATE TABLE events_1 PARTITION OF events FOR VALUES WITH (MODULUS 4, REMAINDER 1);
CREATE TABLE events_2 PARTITION OF events FOR VALUES WITH (MODULUS 4, REMAINDER 2);
CREATE TABLE events_3 PARTITION OF events FOR VALUES WITH (MODULUS 4, REMAINDER 3);
```

Когда использовать: равномерное распределение нагрузки, когда нет явного критерия диапазона или списка (горизонтальное шардирование по первичному ключу).

**Partition Pruning (устранение ненужных секций)**

Partition pruning — оптимизация, при которой планировщик исключает из плана запроса секции, которые заведомо не могут содержать искомые строки. Это происходит на этапе планирования (статический pruning) или на этапе выполнения (dynamic pruning при параметризованных запросах).

```sql
-- Запрос к секционированной таблице:
SELECT * FROM orders WHERE order_date >= '2024-06-01' AND order_date < '2024-07-01';
-- Оптимизатор видит, что нужна только секция orders_2024 — остальные не сканируются
```

Параметр `enable_partition_pruning = on` (включён по умолчанию). В `EXPLAIN` можно увидеть, сколько секций реально участвует в плане.

**Когда секционирование оправдано**

- Таблица содержит больше данных, чем помещается в RAM сервера.
- Часто выполняется архивация/удаление старых данных (DROP/DETACH секции быстрее DELETE + VACUUM).
- Запросы фильтруют по ключу секционирования и получают прирост от pruning.
- Нужно раздельное хранение горячих и архивных данных.

**Ограничения:** уникальные индексы и первичные ключи должны включать все столбцы ключа секционирования. Глобальные индексы (across partitions) в PostgreSQL не поддерживаются — каждая секция имеет свой локальный индекс.

### Вопрос 72. Что такое типы данных JSON и JSONB в PostgreSQL? Чем они отличаются, когда предпочтительнее JSONB и как индексировать JSONB?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 8.14. JSON Types](https://www.postgresql.org/docs/current/datatype-json.html)

PostgreSQL предоставляет два типа данных для хранения JSON: `json` и `jsonb`. Оба принимают на вход практически одинаковые наборы значений. Принципиальное практическое различие заключается в эффективности: тип `json` хранит точную копию входного текста, которую при обработке необходимо заново разбирать. Тип `jsonb` хранит данные в разобранном бинарном формате, что делает его обработку несколько медленнее при записи, зато значительно ускоряет обработку при чтении, поскольку разбор не требуется. `jsonb` также поддерживает индексирование.

| Свойство | json | jsonb |
|---|---|---|
| Хранение | Точная копия входного текста | Разобранный бинарный формат |
| Производительность записи | Быстрее | Медленнее (парсинг при вставке) |
| Производительность чтения | Медленнее (перепарсинг при каждом доступе) | Быстрее |
| Поддержка GIN-индексов | Нет | Да |
| Порядок ключей | Сохраняется | Не сохраняется |
| Дубликаты ключей | Все сохраняются | Последнее значение |

Общая рекомендация: использовать `jsonb`, если нет специфических требований, требующих `json`.

#### Ответ от Claude

**JSON vs JSONB: в чём разница**

`json` — хранит JSON «как есть» в виде текстовой строки. При каждом чтении и обработке PostgreSQL заново разбирает текст. Поскольку исходный текст сохраняется точно, тип хранит также незначимые пробелы, исходный порядок ключей и дубликаты ключей.

`jsonb` — хранит JSON в разобранном бинарном формате. При вставке производится парсинг (чуть медленнее записи), зато при чтении разбор не нужен. Ключи сортируются, дубликаты ключей схлопываются до последнего значения, незначимые пробелы отбрасываются.

```sql
-- json: сохраняет порядок и пробелы
SELECT '{"b": 2, "a": 1}'::json;
-- Результат: {"b": 2, "a": 1}

-- jsonb: сортирует ключи
SELECT '{"b": 2, "a": 1}'::jsonb;
-- Результат: {"a": 1, "b": 2}
```

**Операторы и доступ к данным**

```sql
-- Получить значение по ключу (text)
SELECT data->>'name' FROM users;

-- Получить значение по ключу (jsonb объект)
SELECT data->'address' FROM users;

-- Вложенный доступ
SELECT data->'address'->>'city' FROM users;

-- Проверка наличия ключа (только jsonb)
SELECT * FROM users WHERE data ? 'email';

-- Containment — содержит ли jsonb документ другой (только jsonb)
SELECT * FROM users WHERE data @> '{"role": "admin"}';
```

**Индексирование JSONB**

GIN-индекс (Generalized Inverted Index) — основной тип для JSONB. Поддерживает операторы `?`, `?|`, `?&`, `@>`:

```sql
-- GIN-индекс с оператором jsonb_ops (по умолчанию)
CREATE INDEX idx_users_data ON users USING GIN (data);

-- GIN с jsonb_path_ops — меньший размер, только @> и @?
CREATE INDEX idx_users_data_path ON users USING GIN (data jsonb_path_ops);

-- Индекс по конкретному полю (как обычный B-tree)
CREATE INDEX idx_users_role ON users ((data->>'role'));
```

**Практическое правило выбора типа**

- `jsonb` — выбор по умолчанию для новых проектов: быстрее чтение, поддерживает индексы, удобные операторы.
- `json` — когда принципиально важно сохранить точное текстовое представление (например, для аудита входящих JSON-запросов «как пришли»).

**Расхождения между источниками:** официальная документация PostgreSQL и ресурс Baeldung согласны в рекомендации использовать `jsonb` по умолчанию. Некоторые источники указывают на исключения при работе с legacy-системами, требующими конкретного порядка ключей.

### Вопрос 73. Что такое EXPLAIN ANALYZE BUFFERS в PostgreSQL? Что показывают поля shared hit, shared read, и как интерпретировать вывод для диагностики производительности?

**Категория:** БД и SQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 14.1. Using EXPLAIN](https://www.postgresql.org/docs/current/using-explain.html)

Опция `BUFFERS` выводит статистику использования буферов. Для каждого узла плана показываются:

- **shared hit** — количество блоков, найденных в shared buffer cache (быстрое чтение из памяти).
- **shared read** — количество блоков, прочитанных с диска (медленнее; блок помещается в кэш после чтения).
- **shared dirtied** — блоки, изменённые данным запросом.
- **shared written** — блоки, записанные на диск из shared buffers.

Аналогичная статистика предоставляется для локальных буферов (local hit, read и т.д.), используемых временными таблицами. Временные файлы при sort или hash join: `temp read` и `temp written`.

Начиная с PostgreSQL 18, `ANALYZE` автоматически включает `BUFFERS`, что означает обязательный вывод буферной статистики при реальном выполнении запроса.

#### Ответ от Claude

**EXPLAIN ANALYZE vs EXPLAIN ANALYZE BUFFERS**

`EXPLAIN ANALYZE` выполняет запрос и добавляет реальное время и число строк к каждому узлу плана. `BUFFERS` добавляет ещё и информацию о работе с буферным кешем.

```sql
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT u.name, COUNT(o.id) AS order_count
FROM users u
JOIN orders o ON o.user_id = u.id
GROUP BY u.name;
```

**Ключевые поля в выводе**

```
Seq Scan on orders  (cost=0.00..1234.56 rows=50000 width=8)
                    (actual time=0.05..8.32 rows=50000 loops=1)
  Buffers: shared hit=412 read=88
```

| Поле | Значение |
|------|---------|
| `shared hit=N` | N блоков прочитано из shared buffer cache (оперативная память) |
| `shared read=N` | N блоков прочитано с диска (блоки не были в кэше) |
| `shared dirtied=N` | N блоков помечены «грязными» (изменены, нужно записать на диск) |
| `shared written=N` | N блоков записано на диск из кэша в ходе этого запроса |
| `temp read/written=N` | Блоки временных файлов (sort spill, hash join spill) |

**Как интерпретировать**

**Хороший сценарий:** `shared hit` >> `shared read`. Большинство данных читается из кэша — минимальный дисковый I/O.

**Плохой сценарий:** `shared read` >> `shared hit`. Данные не помещаются в `shared_buffers` или кэш сбрасывается другими запросами. Решение: увеличить `shared_buffers`, оптимизировать запрос (уменьшить число читаемых блоков через индексы).

**Сигнал проблемы:** высокий `temp read/written`. Sort или Hash Join сбрасывают данные на диск из-за недостатка `work_mem`. Решение: увеличить `work_mem` для конкретного запроса: `SET work_mem = '256MB'; EXPLAIN ANALYZE ...`.

**Полный синтаксис с форматированием**

```sql
-- Наиболее информативный вариант для диагностики
EXPLAIN (ANALYZE, BUFFERS, VERBOSE, FORMAT TEXT)
SELECT ...;

-- JSON-формат для инструментов
EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)
SELECT ...;
```

**Практический алгоритм диагностики**

1. Найти узел с наибольшим `actual time`.
2. Проверить `shared read` этого узла — велик ли дисковый I/O?
3. Сравнить `rows` (оценка) с `actual rows` — расхождение > 10× сигнализирует об устаревшей статистике (`ANALYZE table`).
4. Если есть `temp read/written` — увеличить `work_mem`.
5. Если `shared hit` мал относительно `shared read` на часто выполняемом запросе — рассмотреть индекс или увеличение `shared_buffers`.

**Замечание:** `EXPLAIN ANALYZE` реально выполняет запрос. Для DML-запросов (UPDATE, DELETE) оборачивать в транзакцию с ROLLBACK: `BEGIN; EXPLAIN ANALYZE UPDATE ...; ROLLBACK;`.

### Вопрос 74. Что такое WAL (Write-Ahead Log) в PostgreSQL? Как он обеспечивает долговечность (Durability) транзакций?

**Категория:** БД и SQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: Chapter 30. Reliability and the Write-Ahead Log](https://www.postgresql.org/docs/current/wal-intro.html)

Центральная концепция WAL — изменения в файлах данных (где располагаются таблицы и индексы) должны записываться только после того, как эти изменения зафиксированы в журнале, то есть только после того, как записи журнала перемещены в устойчивое хранилище.

Если следовать этой процедуре, нет необходимости сбрасывать страницы данных на диск при каждом COMMIT транзакции, поскольку известно, что в случае краша можно восстановить базу данных, используя журнал: любые изменения, не применённые к страницам данных, могут быть воспроизведены из журнальных записей. Это называется crash recovery (восстановление после аварии).

WAL существенно сокращает число дисковых записей, поскольку для обеспечения сохранности транзакции необходимо записать на диск лишь файл журнала, а не каждую изменённую страницу данных. Журнальный файл записывается последовательно, что значительно дешевле случайной записи страниц данных.

#### Ответ от Claude

**Что такое WAL и зачем он нужен**

WAL (Write-Ahead Log) — журнал с упреждающей записью: все изменения сначала записываются в журнал на диск, и лишь затем применяются к страницам данных. Это фундамент для реализации свойства **Durability** (долговечности) из ACID.

Проблема без WAL: если СУБД изменила страницу данных в памяти, но не успела сбросить на диск до краша системы, изменения потеряны. Откидывать каждую изменённую страницу на диск при COMMIT — дорого: это случайный I/O по всему файлу данных.

**Как WAL обеспечивает долговечность**

```
Транзакция BEGIN
   |
   ↓ Изменения производятся в shared buffers (память)
   |
   ↓ Каждое изменение записывается как WAL-запись в WAL-буфер (память)
   |
   ↓ При COMMIT: WAL-буфер сбрасывается на диск (fsync WAL-файла)
   |              → это единственная обязательная дисковая операция при COMMIT
   |
   ↓ Страницы данных записываются на диск позже (checkpoint или background writer)
```

При краше после COMMIT: WAL-запись на диске есть, страница данных может быть не обновлена. PostgreSQL при старте воспроизводит WAL-записи (replay) и применяет все зафиксированные изменения — база восстанавливается в консистентное состояние.

**Почему WAL эффективен**

1. WAL пишется **последовательно** в конец файла — sequential I/O значительно дешевле случайного.
2. При COMMIT обязательна fsync только WAL-файла, а не всех изменённых страниц данных.
3. Страницы данных сбрасываются на диск группами во время checkpoint — это аморизирует стоимость.

**Checkpoint**

Checkpoint — момент, когда PostgreSQL гарантирует, что все грязные страницы данных (с изменениями, более старыми, чем checkpoint LSN) сброшены на диск. После checkpoint при крашe нужно воспроизвести WAL только с момента последнего checkpoint, что ограничивает время восстановления.

```sql
CHECKPOINT;  -- принудительный checkpoint (для DBA)
```

Настройки checkpoint: `checkpoint_timeout`, `max_wal_size`.

**Дополнительные применения WAL**

- **Репликация** (Streaming Replication): WAL-записи передаются по сети на реплику.
- **Logical Decoding**: WAL используется для CDC (Change Data Capture) через `pg_logical`.
- **Point-in-Time Recovery (PITR)**: непрерывная архивация WAL позволяет восстановить БД на любой момент времени.

**Параметр `synchronous_commit`**

- `on` (по умолчанию): COMMIT ждёт fsync WAL — максимальная долговечность.
- `off`: COMMIT не ждёт fsync — ускорение вставок, но риск потери последних ~`wal_writer_delay` мс при крашe (данные не теряются безвозвратно, но последние транзакции могут откатиться).

### Вопрос 75. Что такое GROUPING SETS, ROLLUP и CUBE? Чем они отличаются от обычного GROUP BY и в каких аналитических задачах применяются?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 7.2.4. GROUPING SETS, CUBE, and ROLLUP](https://www.postgresql.org/docs/current/queries-table-expressions.html#QUERIES-GROUPING-SETS)

Данные, выбранные секциями FROM и WHERE, группируются отдельно по каждому указанному набору группировки, агрегаты вычисляются для каждой группы так же, как при простом GROUP BY, после чего результаты объединяются и возвращаются.

**ROLLUP** — сокращённая запись для набора группировок, включающего заданный список выражений и все его префиксы вплоть до пустого списка. Например, `ROLLUP (e1, e2, e3)` эквивалентно `GROUPING SETS ((e1, e2, e3), (e1, e2), (e1), ())`. Типичное применение: анализ иерархических данных, например суммарная зарплата по отделу, подразделению и компании в целом.

**CUBE** — сокращённая запись для набора группировок, включающего заданный список выражений и все возможные его подмножества (булеан). Например, `CUBE (a, b, c)` эквивалентно `GROUPING SETS ((a,b,c),(a,b),(a,c),(a),(b,c),(b),(c),())`.

При использовании нескольких элементов группировки в GROUP BY они образуют декартово произведение наборов группировки. В строках результата столбцы, не вошедшие в текущий набор группировки, заменяются значением NULL. Для различения того, какой именно набор группировки породил конкретную строку, можно использовать функцию `GROUPING()`.

#### Ответ от Claude

**Зачем нужны расширения GROUP BY**

Стандартный `GROUP BY` группирует строки только по одному указанному набору колонок. Аналитические отчёты часто требуют одновременно нескольких уровней агрегации: итоги по каждому городу, по каждой стране и общий итог. Без расширений это решается через `UNION ALL` нескольких запросов, что громоздко и медленно.

`GROUPING SETS`, `ROLLUP` и `CUBE` — часть стандарта SQL:1999. Поддерживаются в PostgreSQL, Oracle, SQL Server и MySQL 8+.

---

**GROUPING SETS — явное указание нескольких наборов группировки**

```sql
SELECT region, city, SUM(sales) AS total
FROM orders
GROUP BY GROUPING SETS (
    (region, city),  -- итог по паре (регион, город)
    (region),        -- итог по региону
    ()               -- общий итог
);
```

Результат — объединение трёх GROUP BY в одном запросе. В строках, где конкретный столбец не участвует в группировке, он принимает значение NULL.

```
region   | city      | total
---------|-----------|------
Север    | Москва    | 1200
Север    | Тверь     |  400
Север    | NULL      | 1600   <- итог по Северу
Юг       | Краснодар |  900
Юг       | NULL      |  900   <- итог по Югу
NULL     | NULL      | 2500   <- общий итог
```

---

**ROLLUP — иерархические итоги**

`ROLLUP (e1, e2, e3)` генерирует агрегаты по: `(e1, e2, e3)`, `(e1, e2)`, `(e1)`, `()`. Число генерируемых наборов = число колонок + 1. Идеально подходит для отчётов с подитогами по иерархии.

```sql
SELECT year, quarter, month, SUM(revenue) AS rev
FROM sales
GROUP BY ROLLUP (year, quarter, month);
-- итоги: по месяцам, по кварталам, по годам, общий итог
```

---

**CUBE — все комбинации измерений**

`CUBE (a, b, c)` генерирует агрегаты по всем 2^N подмножествам колонок. Для трёх колонок — 8 наборов группировки. Применяется в OLAP-отчётах, когда нужны срезы по всем возможным сочетаниям измерений.

```sql
SELECT product, region, year, SUM(sales)
FROM fact_sales
GROUP BY CUBE (product, region, year);
-- 8 комбинаций: все три, каждые две, каждая по одной, общий итог
```

---

**Различение NULL от отсутствия группировки: функция GROUPING()**

NULL в результате ROLLUP/CUBE может означать как «нет данных», так и «эта колонка не участвует в данном наборе группировки». Функция `GROUPING(col)` возвращает 1, если столбец не является частью текущего набора группировки, и 0 в противном случае.

```sql
SELECT
    CASE WHEN GROUPING(region) = 1 THEN 'Все регионы' ELSE region END AS region,
    CASE WHEN GROUPING(city)   = 1 THEN 'Все города'  ELSE city   END AS city,
    SUM(sales) AS total
FROM orders
GROUP BY ROLLUP (region, city);
```

---

**Сравнение трёх расширений**

| Расширение | Число наборов группировки | Применение |
|---|---|---|
| `GROUPING SETS (...)` | Задаётся явно | Произвольные комбинации |
| `ROLLUP (e1, e2, e3)` | N + 1 (= 4 для трёх колонок) | Иерархические итоги |
| `CUBE (e1, e2, e3)` | 2^N (= 8 для трёх колонок) | Многомерный OLAP-анализ |

**Источники:**
- PostgreSQL Documentation 18: 7.2.4. GROUPING SETS, CUBE, and ROLLUP: https://www.postgresql.org/docs/current/queries-table-expressions.html#QUERIES-GROUPING-SETS
- PostgreSQL Documentation 18: 9.21. Aggregate Functions — функция GROUPING(): https://www.postgresql.org/docs/current/functions-aggregate.html

### Вопрос 76. Что такое SAVEPOINT и когда он используется? Как реализовать частичный откат транзакции?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: SAVEPOINT](https://www.postgresql.org/docs/current/sql-savepoint.html)

`SAVEPOINT` устанавливает новую точку сохранения в рамках текущей транзакции. Точка сохранения — это специальная отметка внутри транзакции, которая позволяет откатить все команды, выполненные после неё, сохранив команды, выполненные до неё.

Синтаксис:
```sql
SAVEPOINT savepoint_name;
ROLLBACK TO SAVEPOINT savepoint_name;
RELEASE SAVEPOINT savepoint_name;
```

Пример использования:
```sql
BEGIN;
INSERT INTO table1 VALUES (1);
SAVEPOINT my_savepoint;
INSERT INTO table1 VALUES (2);
ROLLBACK TO SAVEPOINT my_savepoint;
INSERT INTO table1 VALUES (3);
COMMIT;
-- В таблице окажутся значения 1 и 3
```

`RELEASE SAVEPOINT` уничтожает точку сохранения без отката изменений; операции до точки остаются в силе. Если `ROLLBACK TO SAVEPOINT` выполняется к несуществующей точке — ошибка; если после отката выполняется `ROLLBACK TO` той же точки снова — снова допустимо, поскольку ROLLBACK не уничтожает точку.

#### Ответ от Claude

**Что такое SAVEPOINT**

SAVEPOINT — именованная контрольная точка внутри транзакции. Позволяет откатить только часть изменений (до точки), не отменяя всю транзакцию. Это механизм для реализации вложенных (nested) или условных операций внутри одной транзакции.

**Жизненный цикл SAVEPOINT**

```sql
BEGIN;

-- Шаг 1: обязательное действие
INSERT INTO orders (customer_id, status) VALUES (42, 'NEW') RETURNING id INTO order_id;

SAVEPOINT before_payment;           -- установить точку

-- Шаг 2: рискованное действие
UPDATE payment_methods SET locked = TRUE WHERE customer_id = 42;
INSERT INTO payments (order_id, amount) VALUES (order_id, 199.99);

-- Если платёжный шаг провалился:
ROLLBACK TO SAVEPOINT before_payment;  -- откатываем только шаг 2, заказ остаётся

-- Альтернативный путь без оплаты
INSERT INTO notifications (customer_id, message) VALUES (42, 'Ожидаем оплату');

RELEASE SAVEPOINT before_payment;   -- точка больше не нужна

COMMIT;  -- фиксируем: заказ + уведомление
```

**Вложенные SAVEPOINT**

```sql
BEGIN;
SAVEPOINT sp1;
    INSERT INTO a VALUES (1);
    SAVEPOINT sp2;
        INSERT INTO b VALUES (2);
    ROLLBACK TO SAVEPOINT sp2;   -- откат b, a остаётся
    INSERT INTO b VALUES (3);    -- новый b = 3
RELEASE SAVEPOINT sp1;
COMMIT;
-- Результат: a=1, b=3
```

**SAVEPOINT в Java (JDBC)**

```java
conn.setAutoCommit(false);

// Обязательный шаг
insertOrder(conn, order);

Savepoint sp = conn.setSavepoint("beforeDiscount");
try {
    applyDiscount(conn, order);  // может выбросить бизнес-исключение
} catch (Exception e) {
    conn.rollback(sp);           // только скидка откатывается
} finally {
    conn.releaseSavepoint(sp);   // освобождаем точку
}

conn.commit();  // фиксируем заказ (с или без скидки)
```

**Когда SAVEPOINT необходим**

1. Многошаговые бизнес-операции, где некоторые шаги необязательны.
2. Итерационная обработка в цикле (обработка каждого элемента с возможностью пропустить «плохой» без отмены остальных).
3. Автоматическая обработка конфликтов при вставке: попробовать INSERT, при нарушении уникальности — откатить к savepoint и выполнить UPDATE.

**Ограничения**

- SAVEPOINT существуют только в рамках одной транзакции. После COMMIT или полного ROLLBACK все точки исчезают.
- `RELEASE SAVEPOINT` не откатывает изменения — он только удаляет точку.
- Нельзя использовать SAVEPOINT при `autocommit = true` (транзакции нет).

### Вопрос 77. Что такое курсор (CURSOR) в SQL? Когда он используется вместо обычного SELECT?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: DECLARE (cursor)](https://www.postgresql.org/docs/current/sql-declare.html)

`DECLARE` позволяет создать курсор — механизм инкапсулирования запроса для получения из него строк по несколько за раз. Это полезно для запросов, возвращающих большое количество строк.

Курсоры объявляются и используются внутри транзакции. Без `WITH HOLD` курсор закрывается по завершении транзакции.

```sql
BEGIN;
DECLARE curs1 CURSOR FOR SELECT * FROM foo;
DECLARE curs2 CURSOR FOR SELECT * FROM foo WHERE key = 23;
DECLARE curs3 CURSOR FOR SELECT * FROM foo LIMIT 10;
FETCH curs1;       -- получить следующую строку
FETCH 5 FROM curs2; -- получить 5 строк
CLOSE curs1;
COMMIT;
```

Опция `SCROLL` разрешает движение назад по курсору (`FETCH PRIOR`, `FETCH LAST`), но требует дополнительных ресурсов. Без `SCROLL` курсор только вперёд.

`WITH HOLD` позволяет курсору пережить конец транзакции (holdable cursor): данные материализуются в временную таблицу при COMMIT.

#### Ответ от Claude

**Что такое курсор**

Курсор — именованный «указатель» на результирующий набор SELECT-запроса, позволяющий читать строки по частям (построчно или небольшими пакетами), а не загружать весь результат в память сразу. Курсор держит открытым соединение с СУБД и читает данные инкрементально.

**Синтаксис PostgreSQL**

```sql
BEGIN;

-- Объявить курсор
DECLARE my_cursor CURSOR FOR
    SELECT id, name, email FROM customers ORDER BY id;

-- Получить первые 100 строк
FETCH 100 FROM my_cursor;

-- Ещё 100 строк
FETCH 100 FROM my_cursor;

-- Конкретные направления (только при SCROLL):
-- FETCH PRIOR FROM my_cursor;
-- FETCH FIRST FROM my_cursor;
-- FETCH LAST FROM my_cursor;

CLOSE my_cursor;
COMMIT;
```

**Когда курсор нужен вместо обычного SELECT**

1. **Очень большой результирующий набор**: SELECT * из таблицы с 10 млн строк загрузит всё в память клиента. Курсор позволяет читать по 1000 строк и обрабатывать пакетами.

2. **Пакетная обработка (batch processing)**: ETL, миграции, масштабная обработка с прогрессом. Читаем пачками, обрабатываем, пишем результаты, читаем следующую пачку.

3. **Хранимые процедуры с итерационной логикой**: в PL/pgSQL курсор используется для обхода результата в цикле `FOR ... IN`.

```sql
-- PL/pgSQL: обход с курсором
DO $$
DECLARE
    cur CURSOR FOR SELECT id, salary FROM employees;
    rec RECORD;
BEGIN
    OPEN cur;
    LOOP
        FETCH cur INTO rec;
        EXIT WHEN NOT FOUND;
        UPDATE employees SET salary = rec.salary * 1.1 WHERE id = rec.id;
    END LOOP;
    CLOSE cur;
END;
$$;
```

**Курсоры в JDBC (серверные курсоры)**

PostgreSQL JDBC-драйвер при `setFetchSize(N)` автоматически использует серверный курсор для передачи данных порциями:

```java
PreparedStatement ps = conn.prepareStatement("SELECT * FROM large_table");
ps.setFetchSize(500);  // PostgreSQL создаст курсор и будет отдавать по 500 строк
ResultSet rs = ps.executeQuery();
while (rs.next()) { /* обработка */ }
```

**Ограничения курсоров**

- По умолчанию курсор закрывается при завершении транзакции. `WITH HOLD` — исключение.
- Держат открытыми ресурсы на сервере (memory, snapshot) — важно явно закрывать: `CLOSE cursor_name`.
- Передвижение назад требует `SCROLL` и буферизации — дополнительный расход памяти.
- В большинстве случаев лучше использовать `LIMIT / OFFSET` или `keyset pagination` — они проще и не держат открытый курсор.

### Вопрос 78. Что такое последовательность (SEQUENCE) в PostgreSQL? Как она работает и почему значения могут быть непоследовательными?

**Категория:** БД и SQL · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: CREATE SEQUENCE](https://www.postgresql.org/docs/current/sql-createsequence.html)

Последовательность (SEQUENCE) — специальный однострочный объект базы данных, используемый для генерации уникальных числовых идентификаторов. Значения, возвращаемые функцией `nextval`, не могут быть возвращены повторно, даже при откате транзакции.

```sql
CREATE SEQUENCE myserial
    START WITH 1
    INCREMENT BY 1
    NO MINVALUE
    NO MAXVALUE
    CACHE 1;

SELECT nextval('myserial');
```

Значения последовательностей являются «нетранзакционными»: если транзакция вызывает `nextval`, а затем откатывается, значение считается использованным и не возвращается в пул. Это намеренное проектное решение для предотвращения блокировок.

Параметр `CACHE N` позволяет предварительно выделять N значений в память, что повышает производительность. Однако при сбое сервера не использованные кешированные значения теряются, что увеличивает непоследовательность идентификаторов.

#### Ответ от Claude

**Что такое SEQUENCE**

Последовательность (SEQUENCE) — объект базы данных, генерирующий монотонно возрастающие (или убывающие) числа. Гарантирует уникальность значений при параллельных обращениях. Используется для автоматической генерации первичных ключей.

**Создание и использование**

```sql
-- Явное создание последовательности
CREATE SEQUENCE order_id_seq
    START WITH 1000
    INCREMENT BY 1
    MINVALUE 1
    MAXVALUE 9223372036854775807
    CACHE 10;   -- предварительно кешировать 10 значений

-- Получить следующее значение
SELECT nextval('order_id_seq');  -- 1000
SELECT nextval('order_id_seq');  -- 1001

-- Посмотреть текущее значение (не увеличивает счётчик)
SELECT currval('order_id_seq');  -- 1001

-- Сбросить на заданное значение
SELECT setval('order_id_seq', 5000);
```

**Тип данных SERIAL и GENERATED**

```sql
-- SERIAL — синтаксический сахар, создаёт SEQUENCE автоматически
CREATE TABLE orders (
    id SERIAL PRIMARY KEY,  -- = integer + CREATE SEQUENCE + DEFAULT nextval(...)
    ...
);

-- Современный синтаксис (SQL-стандарт, PostgreSQL 10+):
CREATE TABLE orders (
    id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    ...
);
```

**Почему значения могут быть непоследовательными**

1. **Откат транзакции (ROLLBACK)**. Если транзакция вызвала `nextval` и откатилась — значение «сгорело». Следующая транзакция получит следующее значение, а предыдущее никогда не будет использовано.

```sql
BEGIN;
INSERT INTO orders (...) VALUES (...);  -- nextval() вернул 100
ROLLBACK;  -- транзакция отменена

-- Следующая вставка получит 101, значение 100 потеряно
INSERT INTO orders (...) VALUES (...);  -- nextval() = 101
```

2. **CACHE > 1**. Каждая сессия кеширует N значений из последовательности. При крашe или рестарте неиспользованные кешированные значения теряются. При `CACHE 10` потенциально теряется до 9 значений.

3. **Параллельные вставки**. Несколько сессий получают значения из общего счётчика одновременно — порядок записей в таблице может не совпадать с порядком ID.

**Это намеренное поведение:** SEQUENCE нетранзакционна для производительности. Если бы откат возвращал значение в пул, другие сессии должны были бы ждать — блокировка разрушила бы параллелизм.

**Практические выводы**

- ID из SEQUENCE уникален, но не гарантированно непрерывен.
- Не следует полагаться на непрерывность ID в бизнес-логике.
- Для вставки без пропусков нужен другой механизм (COUNT + 1 в транзакции — но это блокировки).
- `GENERATED ALWAYS AS IDENTITY` предпочтительнее `SERIAL` в современном PostgreSQL — запрещает вставку явного значения.

### Вопрос 79. Что такое представление (VIEW)? Можно ли через VIEW выполнять INSERT/UPDATE/DELETE, и каковы условия для этого?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: CREATE VIEW](https://www.postgresql.org/docs/current/sql-createview.html)

`CREATE VIEW` определяет представление запроса. Представление не материализуется физически. Вместо этого запрос выполняется каждый раз, когда к представлению обращается запрос.

Обновляемые представления: простые представления автоматически являются обновляемыми — система позволяет выполнять `INSERT`, `UPDATE` и `DELETE` для них так же, как для обычных таблиц. Представление является автоматически обновляемым, если выполняются все следующие условия:
- Ровно одна таблица или обновляемое представление в секции FROM.
- Нет секций WITH, DISTINCT, GROUP BY, HAVING, LIMIT, OFFSET на верхнем уровне.
- Нет операторов UNION, INTERSECT, EXCEPT на верхнем уровне.
- Нет агрегатных, оконных функций и функций возврата множества в списке SELECT.
- Все столбцы в списке SELECT являются ссылками на столбцы базовой таблицы; нет дубликатов. Столбцы из базовой таблицы, отсутствующие в SELECT, могут принимать значения по умолчанию.

#### Ответ от Claude

**Что такое VIEW**

Представление (VIEW) — именованный SELECT-запрос, сохранённый в базе данных. При обращении к представлению СУБД подставляет его определение в запрос. VIEW не хранит данные — это виртуальная таблица.

```sql
CREATE VIEW active_employees AS
SELECT id, name, department, salary
FROM employees
WHERE terminated_at IS NULL;

-- Использование как обычной таблицы:
SELECT * FROM active_employees WHERE department = 'Engineering';
```

**Обновляемые VIEW (updatable views)**

PostgreSQL автоматически делает простые VIEW обновляемыми. DML-операции на VIEW транслируются СУБД в операции на базовой таблице.

Условия автоматической обновляемости:
- Ровно одна базовая таблица.
- Нет DISTINCT, GROUP BY, HAVING, LIMIT, OFFSET.
- Нет UNION, INTERSECT, EXCEPT.
- Нет агрегатных или оконных функций.
- Все колонки в SELECT — прямые ссылки на колонки таблицы.

```sql
-- active_employees обновляем: одна таблица, нет агрегатов
UPDATE active_employees SET salary = 90000 WHERE id = 5;
-- Транслируется в: UPDATE employees SET salary = 90000 WHERE id = 5

INSERT INTO active_employees (name, department, salary)
VALUES ('Иванов', 'Engineering', 80000);
-- Транслируется в: INSERT INTO employees (name, department, salary, ...) VALUES (...)
```

**WITH CHECK OPTION**

Предотвращает вставку/обновление строк, которые не будут видны через VIEW:

```sql
CREATE VIEW active_employees AS
SELECT * FROM employees
WHERE terminated_at IS NULL
WITH CHECK OPTION;

-- Попытка вставить уволенного сотрудника через view:
INSERT INTO active_employees (name, terminated_at) VALUES ('X', '2023-01-01');
-- ОШИБКА: new row violates check option for view
```

**Неавтоматически обновляемые VIEW**

Если VIEW нарушает условия обновляемости (GROUP BY, JOIN, UNION...) — DML невозможен напрямую. Решения:

1. **INSTEAD OF триггеры** — перехватывают INSERT/UPDATE/DELETE на VIEW и вручную транслируют в нужные операции:

```sql
CREATE TRIGGER trg_insert_employee_view
    INSTEAD OF INSERT ON complex_employee_view
    FOR EACH ROW EXECUTE FUNCTION fn_insert_employee_view();
```

2. **Правила (RULE)** — устаревший механизм перезаписи запросов. Не рекомендуется для новых проектов.

**Практические выводы**

- VIEW — инструмент абстракции: скрывает сложность, упрощает доступ, обеспечивает безопасность (выдавать права на VIEW, не на таблицу).
- Для безопасного DML через VIEW на фильтрованных данных используют `WITH CHECK OPTION`.
- Для сложных VIEW — `INSTEAD OF`-триггеры.
- Если нужна производительность — использовать MATERIALIZED VIEW.

### Вопрос 80. Что такое триггер (TRIGGER) в SQL? В каких случаях он применяется и какие у него подводные камни?

**Категория:** БД и SQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: CREATE TRIGGER](https://www.postgresql.org/docs/current/sql-createtrigger.html)

`CREATE TRIGGER` определяет новый триггер. Триггер связывается с указанной таблицей, представлением или внешней таблицей и выполняет функцию при наступлении определённых событий.

Триггер может быть указан как срабатывающий до (BEFORE) или после (AFTER) попытки операции или вместо неё (INSTEAD OF). Срабатывание — для каждой строки, изменённой операцией (FOR EACH ROW), или один раз для всего оператора (FOR EACH STATEMENT). Триггеры BEFORE FOR EACH ROW могут вернуть NULL для пропуска операции над текущей строкой.

Внутри функции триггера доступны специальные переменные: `NEW` (новые значения строки при INSERT/UPDATE) и `OLD` (старые значения при UPDATE/DELETE).

```sql
CREATE TRIGGER check_update
    BEFORE UPDATE ON accounts
    FOR EACH ROW
    EXECUTE FUNCTION check_account_update();
```

#### Ответ от Claude

**Что такое триггер**

Триггер — объект базы данных, автоматически выполняющий SQL-функцию при наступлении определённых событий (INSERT, UPDATE, DELETE, TRUNCATE) на таблице или представлении. Триггер не вызывается явно — СУБД запускает его автоматически.

**Виды триггеров**

| Параметр | Варианты |
|---|---|
| Момент | BEFORE (до операции), AFTER (после), INSTEAD OF (вместо, только для VIEW) |
| Гранулярность | FOR EACH ROW (для каждой строки), FOR EACH STATEMENT (один раз на оператор) |
| Событие | INSERT, UPDATE, UPDATE OF col, DELETE, TRUNCATE |

**Типичные применения**

```sql
-- 1. Автоматическое обновление поля updated_at
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_updated_at
    BEFORE UPDATE ON orders
    FOR EACH ROW EXECUTE FUNCTION set_updated_at();

-- 2. Аудит изменений (таблица истории)
CREATE OR REPLACE FUNCTION audit_salary_change()
RETURNS TRIGGER AS $$
BEGIN
    IF OLD.salary <> NEW.salary THEN
        INSERT INTO salary_audit (employee_id, old_salary, new_salary, changed_at)
        VALUES (OLD.id, OLD.salary, NEW.salary, NOW());
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_salary_audit
    AFTER UPDATE OF salary ON employees
    FOR EACH ROW EXECUTE FUNCTION audit_salary_change();

-- 3. Проверка бизнес-правил (вместо CHECK constraint для сложных условий)
CREATE OR REPLACE FUNCTION check_balance()
RETURNS TRIGGER AS $$
BEGIN
    IF NEW.balance < 0 THEN
        RAISE EXCEPTION 'Баланс не может быть отрицательным: %', NEW.balance;
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

**Переменные NEW и OLD**

| Операция | NEW | OLD |
|---|---|---|
| INSERT | Новая строка | NULL |
| UPDATE | Новые значения | Старые значения |
| DELETE | NULL | Удаляемая строка |

В BEFORE-триггере: если функция возвращает NULL — строка пропускается (операция не выполняется для неё). Если возвращает NEW — операция выполняется (можно изменить NEW до записи).

**Подводные камни**

1. **Скрытая логика**. Триггеры выполняются невидимо для вызывающей стороны. Новый разработчик не видит в коде приложения, что происходит ещё что-то. Сложность отладки резко возрастает.

2. **Каскадные триггеры**. Триггер может вызвать DML-операцию, которая вызовет другой триггер и т.д. Глубокая рекурсия ведёт к непредсказуемому поведению и может достигать лимита глубины вызовов.

3. **Производительность**. Каждый `FOR EACH ROW`-триггер добавляет вызов PL/pgSQL-функции на каждую строку. Для массовых UPDATE/INSERT тысяч строк это существенные накладные расходы.

4. **TRUNCATE не вызывает ROW-триггеры**. Если логика очистки таблицы зависит от триггеров, TRUNCATE обойдёт её.

5. **Порядок выполнения нескольких триггеров**. Несколько триггеров одного типа на одной таблице выполняются в **алфавитном порядке имён**. Это нельзя изменить без переименования триггеров.

6. **Сложность транзакционного поведения**. В BEFORE-триггере невозможно узнать, завершится ли транзакция COMMIT или ROLLBACK. Побочные эффекты (отправка email, внешние вызовы) в триггерах опасны.

**Альтернативы триггерам**

- Аудит: `pg_audit` расширение, change data capture через WAL.
- Обновление `updated_at`: выполнять в приложении или через ORM.
- Бизнес-правила: CHECK constraints для простых условий; логика приложения для сложных.
