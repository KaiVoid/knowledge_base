### Вопрос 1. Что такое паттерн Singleton? Напишите потокобезопасную реализацию с double-checked locking и объясните, зачем нужен volatile.

**Категория:** Паттерны · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Double-Checked Locking with Singleton | Baeldung](https://www.baeldung.com/java-singleton-double-checked-locking)

Singleton — паттерн проектирования, гарантирующий существование единственного экземпляра класса и предоставляющий глобальную точку доступа к нему. Проблема простой «ленивой» инициализации в многопоточной среде состоит в том, что несколько потоков могут одновременно пройти проверку `instance == null` и создать несколько экземпляров. Классическое решение — синхронизировать метод целиком, но это вводит ненужную блокировку при каждом обращении к уже созданному объекту.

Double-checked locking решает задачу: сначала делается проверка без блокировки, затем, если объект ещё не создан, берётся монитор и делается повторная проверка уже внутри синхронизированного блока. Поле должно быть объявлено `volatile`, чтобы предотвратить проблемы видимости кэша между потоками (cache incoherence). Без `volatile` один поток может увидеть не до конца инициализированный объект из-за переупорядочивания инструкций процессором или компилятором.

```java
public class Singleton {
    private static volatile Singleton instance;

    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

Альтернатива — использовать `enum`, что считается наиболее лаконичным и безопасным способом реализации Singleton.

#### Ответ от Claude

**Что такое Singleton**

Singleton — порождающий паттерн, гарантирующий наличие ровно одного экземпляра класса в JVM и предоставляющий статическую точку доступа к нему. Типичные применения: пул соединений, конфигурация, логгер, кэш.

**Почему нужен double-checked locking**

Наивная «ленивая» инициализация (`if (instance == null) instance = new Singleton()`) некорректна в многопоточной среде: два потока могут одновременно пройти проверку и создать два разных объекта. Синхронизация всего метода безопасна, но дорого обходится после первого создания объекта — блокировка берётся при каждом вызове. Double-checked locking сочетает корректность с производительностью: блокировка берётся лишь один раз — при первом создании.

**Зачем нужен volatile**

Оператор `instance = new Singleton()` компилируется в три шага на уровне байт-кода:
1. Выделить память для объекта.
2. Инициализировать поля объекта (вызов конструктора).
3. Присвоить ссылку переменной `instance`.

JIT-компилятор и процессор вправе переставлять независимые инструкции (instruction reordering). Без `volatile` шаги 2 и 3 могут поменяться местами: другой поток увидит ненулевую `instance`, но объект ещё не инициализирован, и обращение к его полям приведёт к непредсказуемому поведению.

Модификатор `volatile` (согласно Java Memory Model, JSR-133) создаёт отношение happens-before: запись в `volatile`-поле видна всем последующим чтениям этого поля в любом потоке, и запрещает переупорядочивание относительно этой записи. Таким образом, другой поток либо увидит `null`, либо полностью инициализированный объект — промежуточных состояний нет.

**Потокобезопасная реализация с double-checked locking**

```java
public class Singleton {
    // volatile обязателен: без него возможно чтение частично инициализированного объекта
    private static volatile Singleton instance;

    private Singleton() {}   // запрещаем внешнее создание

    public static Singleton getInstance() {
        if (instance == null) {                    // первая проверка — без блокировки
            synchronized (Singleton.class) {
                if (instance == null) {            // вторая проверка — внутри монитора
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

**Альтернативы**

1. **Initialization-on-demand holder (Bill Pugh Singleton)** — ленивая инициализация без `volatile` через вложенный статический класс; загрузчик классов гарантирует атомарность:

```java
public class Singleton {
    private Singleton() {}

    private static class Holder {
        static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}
```

2. **Enum-Singleton** — рекомендация Джошуа Блоха («Effective Java»); защищает от сериализации и рефлексии:

```java
public enum Singleton {
    INSTANCE;
    // методы...
}
```

**Важно:** double-checked locking работает корректно только начиная с Java 5 (JSR-133), которая зафиксировала семантику `volatile`. В Java 1.4 и ниже эта реализация содержит гонку данных даже с `volatile`.

### Вопрос 2. Чем отличается Factory Method от Abstract Factory? Когда вы выберете один паттерн вместо другого?

**Категория:** Паттерны · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Factory Comparison | Refactoring.Guru](https://refactoring.guru/design-patterns/factory-comparison)

**Factory Method** — порождающий паттерн, предоставляющий интерфейс для создания объектов в суперклассе, но позволяющий подклассам изменять тип создаваемых объектов. Основан на наследовании: абстрактный «создатель» объявляет фабричный метод, подклассы его переопределяют и возвращают конкретные продукты. Фокус — на создании одного типа объекта через механизм наследования.

**Abstract Factory** — позволяет создавать семейства связанных объектов, не привязываясь к их конкретным классам. Основан на композиции: клиент работает с абстрактной фабрикой, конкретные реализации которой порождают совместимые наборы объектов (например, Chair + Sofa + Table в едином стиле — Modern или Victorian).

Ключевое различие в масштабе: Factory Method работает с одним продуктом через наследование; Abstract Factory — с семейством взаимосвязанных продуктов через делегирование. Многие проекты начинают с Factory Method и эволюционируют до Abstract Factory по мере усложнения.

#### Ответ от Claude

**Factory Method**

Порождающий паттерн, определяющий интерфейс для создания объекта, но откладывающий решение о том, объект какого класса создавать, на подклассы.

Структура:
- `Creator` (абстрактный класс или интерфейс) объявляет `factoryMethod()`, возвращающий `Product`.
- `ConcreteCreator` переопределяет `factoryMethod()` и возвращает `ConcreteProduct`.
- Бизнес-логика в `Creator` использует `factoryMethod()` без знания о конкретном типе.

```java
// Продукт
interface Button { void render(); }

class WindowsButton implements Button {
    public void render() { System.out.println("Windows button"); }
}
class MacButton implements Button {
    public void render() { System.out.println("Mac button"); }
}

// Создатель
abstract class Dialog {
    public void renderWindow() {
        Button btn = createButton();   // вызов фабричного метода
        btn.render();
    }
    protected abstract Button createButton();  // фабричный метод
}

class WindowsDialog extends Dialog {
    protected Button createButton() { return new WindowsButton(); }
}
class MacDialog extends Dialog {
    protected Button createButton() { return new MacButton(); }
}
```

**Abstract Factory**

Порождающий паттерн, предоставляющий интерфейс для создания семейств взаимосвязанных объектов без указания их конкретных классов.

Структура:
- `AbstractFactory` объявляет методы создания всех продуктов семейства (`createButton()`, `createCheckbox()`).
- `ConcreteFactory` реализует создание всего семейства для одного варианта (Windows, Mac).
- Клиент работает только с абстрактными интерфейсами.

```java
interface GUIFactory {
    Button createButton();
    Checkbox createCheckbox();
}

class WindowsFactory implements GUIFactory {
    public Button createButton()     { return new WindowsButton(); }
    public Checkbox createCheckbox() { return new WindowsCheckbox(); }
}
class MacFactory implements GUIFactory {
    public Button createButton()     { return new MacButton(); }
    public Checkbox createCheckbox() { return new MacCheckbox(); }
}

// Клиент не знает о конкретных классах
class Application {
    private Button button;
    private Checkbox checkbox;
    Application(GUIFactory factory) {
        button   = factory.createButton();
        checkbox = factory.createCheckbox();
    }
}
```

**Ключевые отличия**

| Критерий | Factory Method | Abstract Factory |
|---|---|---|
| Механизм | Наследование | Композиция / делегирование |
| Масштаб | Один продукт | Семейство взаимосвязанных продуктов |
| Расширение | Добавить подкласс Creator | Добавить ConcreteFactory |
| Сложность | Проще | Сложнее |
| Продукты совместимы | Не гарантировано | Гарантировано по условию |

**Когда выбрать Factory Method**

- Нужно создавать один тип объекта, а конкретный тип определяется в рантайме или подклассами.
- Хотите дать пользователям фреймворка возможность расширять его компоненты через наследование.
- Есть одна «ось изменчивости» (тип создаваемого объекта).

**Когда выбрать Abstract Factory**

- Система должна работать с несколькими семействами продуктов (например, виджеты для Windows и Mac).
- Нужно гарантировать совместимость объектов внутри семейства (нельзя смешать Mac-кнопку с Windows-чекбоксом).
- Есть несколько «осей изменчивости», и все они должны меняться согласованно.
- Вы уже используете несколько Factory Method в одном классе — это сигнал перейти к Abstract Factory.

**Практическое правило:** начинайте с Factory Method; если обнаруживаете, что вам нужно создавать согласованные наборы объектов — переходите к Abstract Factory.

### Вопрос 3. Зачем нужен паттерн Builder? Чем он лучше конструктора с большим числом параметров?

**Категория:** Паттерны · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Builder | Refactoring.Guru](https://refactoring.guru/design-patterns/builder)

Паттерн Builder решает проблему «телескопического конструктора» (telescoping constructor): когда объект имеет много параметров (часть из которых опциональна), приходится создавать множество перегруженных конструкторов — или передавать `null` для неиспользуемых параметров.

Пример с пиццей:
```java
class Pizza {
    Pizza(int size) { ... }
    Pizza(int size, boolean cheese) { ... }
    Pizza(int size, boolean cheese, boolean pepperoni) { ... }
}
```

Количество конструкторов растёт экспоненциально. Builder позволяет строить объект пошагово, вызывая только те шаги, которые реально нужны для данной конфигурации. После реализации паттерна не нужно «запихивать» десятки параметров в конструкторы.

Преимущества:
1. Чистый код: каждый шаг имеет понятное имя метода вместо безымянного параметра в длинном списке.
2. Гибкость: создавать объекты разных конфигураций, не плодя конструкторы.
3. Переиспользование: один Builder может строить разные представления объекта.
4. Разделение ответственностей: логика конструирования отделена от бизнес-логики.

#### Ответ от Claude

**Проблема: telescoping constructor**

Когда объект имеет много полей (часть обязательна, часть опциональна), разработчики прибегают к одному из двух подходов, оба с недостатками:

1. **Перегрузка конструкторов** (telescoping constructor):

```java
class HttpRequest {
    HttpRequest(String url) { ... }
    HttpRequest(String url, String method) { ... }
    HttpRequest(String url, String method, int timeout) { ... }
    HttpRequest(String url, String method, int timeout, Map<String,String> headers) { ... }
    // и т.д. — число конструкторов растёт как 2^N
}
```

Недостатки: много однотипного кода; при вызове непонятно, что означает каждый аргумент (`new HttpRequest("http://...", "GET", 30, null)` — что такое `30` и `null`?).

2. **JavaBeans (сеттеры)**:

```java
HttpRequest req = new HttpRequest();
req.setUrl("http://...");
req.setMethod("GET");
```

Недостатки: объект находится в неконсистентном состоянии между вызовами сеттеров; невозможно сделать класс иммутабельным.

**Решение: паттерн Builder**

Builder отделяет конструирование объекта от его представления и позволяет строить объект пошагово через fluent-интерфейс.

```java
public class HttpRequest {
    private final String url;        // обязательное
    private final String method;     // обязательное
    private final int timeout;       // опциональное
    private final Map<String,String> headers;  // опциональное
    private final String body;       // опциональное

    private HttpRequest(Builder builder) {
        this.url     = builder.url;
        this.method  = builder.method;
        this.timeout = builder.timeout;
        this.headers = builder.headers;
        this.body    = builder.body;
    }

    public static class Builder {
        // обязательные поля — в конструктор Builder
        private final String url;
        private final String method;
        // опциональные — значения по умолчанию
        private int timeout = 30;
        private Map<String,String> headers = new HashMap<>();
        private String body = "";

        public Builder(String url, String method) {
            this.url    = url;
            this.method = method;
        }

        public Builder timeout(int timeout) {
            this.timeout = timeout;
            return this;
        }
        public Builder header(String name, String value) {
            this.headers.put(name, value);
            return this;
        }
        public Builder body(String body) {
            this.body = body;
            return this;
        }

        public HttpRequest build() {
            // здесь можно добавить валидацию
            return new HttpRequest(this);
        }
    }
}

// Использование — читается как предложение на английском
HttpRequest req = new HttpRequest.Builder("https://api.example.com", "POST")
        .timeout(60)
        .header("Content-Type", "application/json")
        .body("{\"key\":\"value\"}")
        .build();
```

**Сравнение подходов**

| Критерий | Telescoping constructor | JavaBeans (сеттеры) | Builder |
|---|---|---|---|
| Читаемость вызова | Низкая | Средняя | Высокая |
| Иммутабельность | Возможна | Невозможна | Возможна |
| Консистентность | Гарантирована | Не гарантирована | Гарантирована (в `build()`) |
| Опциональные поля | Много конструкторов | Легко | Легко |
| Валидация | Разрозненная | Разрозненная | Централизована в `build()` |

**Когда использовать Builder**

- Класс имеет 4+ параметров, особенно если многие опциональны.
- Нужна иммутабельность объекта.
- Нужна централизованная валидация при создании.
- Нужна читаемость вызывающего кода (особенно в тестах).

**В реальных проектах** Builder часто генерируется автоматически:
- Аннотация `@Builder` из библиотеки Lombok.
- Встроенный `record` (Java 16+) с `with`-методами через библиотеки.
- `Builder` в Protobuf, Guava (`ImmutableList.builder()`), `StringBuilder`.

**Важный нюанс:** Builder добавляет дополнительный класс и бойлерплейт, поэтому для простых объектов с 2–3 параметрами обычный конструктор предпочтительнее.
### Вопрос 4. Что такое паттерн Prototype? Чем поверхностное (shallow) копирование отличается от глубокого (deep)?

**Категория:** Паттерны · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Cloneable (Java SE 17 & JDK 17) — docs.oracle.com](https://docs.oracle.com/en/java/javase/17/docs/api/java.base/java/lang/Cloneable.html)

Класс реализует интерфейс `Cloneable`, чтобы сообщить методу `Object.clone()`, что для него законно выполнять побайтовое копирование (field-for-field copy) экземпляров этого класса. Вызов `Object.clone()` на объекте, не реализующем `Cloneable`, приводит к выбросу `CloneNotSupportedException`. По соглашению, классы, реализующие этот интерфейс, должны переопределять `Object.clone()` (который является `protected`) публичным методом. `Cloneable` является маркерным интерфейсом и не содержит метода `clone()`.

Метод `Object.clone()` создаёт новый экземпляр класса и инициализирует все его поля точным содержимым соответствующих полей исходного объекта — то есть выполняет **поверхностное копирование (shallow copy)**, а не глубокое. Для реализации глубокого копирования необходимо самостоятельно скопировать все изменяемые объекты, составляющие внутреннюю «глубокую структуру» клонируемого объекта, и заменить ссылки на них ссылками на копии.

#### Ответ от Claude

**Паттерн Prototype** — это порождающий паттерн проектирования, который позволяет создавать новые объекты путём копирования (клонирования) существующего объекта-прототипа, не завися от конкретного класса этого объекта.

**Зачем нужен.** Напрямую скопировать объект извне невозможно, если у него есть приватные поля. Кроме того, прямое создание через `new ConcreteClass()` привязывает код к конкретной реализации. Прототип делегирует клонирование самому объекту через общий интерфейс с методом `clone()`.

**Реализация в Java.** Стандартный способ — реализовать маркерный интерфейс `java.lang.Cloneable` и переопределить `Object.clone()` как `public`:

```java
public class Address implements Cloneable {
    private String city;
    private String street;

    @Override
    public Address clone() {
        try {
            return (Address) super.clone();
        } catch (CloneNotSupportedException e) {
            throw new AssertionError(); // не случится, Cloneable реализован
        }
    }
}

public class Person implements Cloneable {
    private String name;
    private Address address; // изменяемое поле-ссылка

    @Override
    public Person clone() {
        try {
            Person copy = (Person) super.clone();
            // для глубокого копирования нужно явно клонировать вложенные объекты:
            copy.address = this.address.clone();
            return copy;
        } catch (CloneNotSupportedException e) {
            throw new AssertionError();
        }
    }
}
```

**Поверхностное (shallow) копирование** — `Object.clone()` по умолчанию создаёт новый объект и копирует значения всех полей «как есть»: примитивы копируются по значению, а ссылочные поля — только ссылки. В результате оригинал и копия **разделяют** одни и те же вложенные объекты. Изменение изменяемого вложенного объекта через копию отразится и на оригинале.

**Глубокое (deep) копирование** — все вложенные изменяемые объекты также рекурсивно копируются. Копия полностью независима от оригинала на всех уровнях вложенности.

| Аспект | Shallow copy | Deep copy |
|--------|-------------|-----------|
| Примитивы | Копируются по значению | Копируются по значению |
| Ссылки на объекты | Копируется ссылка (общий объект) | Создаётся новый объект |
| Независимость | Частичная | Полная |
| Производительность | Быстрее | Медленнее |
| Когда применять | Все вложенные объекты неизменяемы | Есть изменяемые вложенные объекты |

**Альтернативы `Cloneable`.** Механизм `Cloneable` в Java считается дефектным (метод `clone()` объявлен в `Object`, а не в `Cloneable`). На практике чаще используют:
- конструктор копирования: `public Person(Person other) { ... }`;
- статическую фабрику: `Person.copyOf(original)`;
- сериализацию/десериализацию (дорого по производительности);
- библиотеки глубокого копирования (например, Apache Commons Lang `SerializationUtils.clone()`).

**Когда использовать паттерн Prototype:** когда создание объекта с нуля дороже его копирования; когда классы, которые нужно клонировать, известны только во время выполнения; когда нужно создавать объекты, аналогичные уже существующим, без привязки к конкретным классам.

### Вопрос 5. В чём разница между паттернами Adapter и Decorator? Приведите пример каждого из стандартной библиотеки Java.

**Категория:** Паттерны · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Wrapper Implementations — The Java™ Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/collections/implementations/wrapper.html)

Обёрточные реализации (Wrapper Implementations) делегируют всю реальную работу указанной коллекции, добавляя при этом дополнительную функциональность поверх того, что предлагает данная коллекция. Для любителей паттернов проектирования — это пример паттерна **Decorator**. Все реализации находятся в классе `Collections`, который состоит исключительно из статических методов. Существует три вида обёрточных реализаций: синхронизирующие (добавляют потокобезопасность), немодифицируемые (убирают возможность изменения) и проверяющие тип (динамическая типовая безопасность). Один недостаток: невозможно вызвать методы, не входящие в интерфейс обёрнутой реализации.

#### Ответ от Claude

**Adapter** и **Decorator** — оба паттерна являются структурными и оба «оборачивают» существующий объект, но решают принципиально разные задачи.

**Adapter (Адаптер)**
- **Цель:** преобразовать **несовместимый интерфейс** в тот, который ожидает клиент. Клиент получает другой интерфейс — не тот, что у адаптируемого объекта.
- Новая функциональность не добавляется — только «перевод» между интерфейсами.
- Применяется, когда нужно использовать существующий класс, но его интерфейс не соответствует требуемому.

**Пример из стандартной библиотеки Java — `InputStreamReader`:**

```java
// InputStream — байтовый поток, Reader — символьный поток.
// InputStreamReader адаптирует InputStream к интерфейсу Reader.
InputStream byteStream = new FileInputStream("file.txt");
Reader charReader = new InputStreamReader(byteStream, StandardCharsets.UTF_8);
BufferedReader br = new BufferedReader(charReader);
```

Официальная документация Java SE 17 (`java.io.InputStreamReader`) гласит: «An InputStreamReader is a bridge from byte streams to character streams: It reads bytes and decodes them into characters using a specified charset.» `InputStreamReader` расширяет `Reader`, принимая `InputStream` — это классический объектный адаптер: клиент работает с `Reader`-интерфейсом, хотя «под капотом» находится байтовый `InputStream`. Аналогично устроен `OutputStreamWriter`.

Ещё один пример: `Arrays.asList(array)` адаптирует массив к интерфейсу `List`.

---

**Decorator (Декоратор)**
- **Цель:** динамически добавить **новое поведение** объекту, **сохраняя тот же интерфейс**. Клиент работает через тот же интерфейс — расширяется только поведение.
- Декораторы можно рекурсивно вкладывать друг в друга произвольное число раз.
- Применяется как альтернатива наследованию для избежания «взрыва подклассов».

**Пример из стандартной библиотеки Java — `BufferedInputStream`:**

```java
// FileInputStream — базовый поток чтения файла.
// BufferedInputStream добавляет буферизацию — новое поведение,
// при этом интерфейс остаётся тем же: InputStream.
InputStream raw = new FileInputStream("file.txt");
InputStream buffered = new BufferedInputStream(raw);
InputStream checked = new DataInputStream(buffered); // ещё один декоратор
```

`BufferedInputStream` расширяет `FilterInputStream extends InputStream` — то есть и декоратор, и декорируемый объект имеют общий тип `InputStream`. Документация Java SE 17 (`java.io.BufferedInputStream`): «A BufferedInputStream adds functionality to another input stream — namely, the ability to buffer the input and to support the mark and reset methods.»

Также из `java.util.Collections`:
```java
List<String> list = new ArrayList<>();
List<String> syncList = Collections.synchronizedList(list);   // Decorator: добавляет синхронизацию
List<String> readOnly  = Collections.unmodifiableList(list);  // Decorator: убирает мутабельность
```

---

**Сравнение паттернов:**

| Аспект | Adapter | Decorator |
|--------|---------|-----------|
| Цель | Изменить интерфейс | Расширить поведение |
| Интерфейс на выходе | Другой (новый) | Тот же или расширенный |
| Добавляет функциональность | Нет | Да |
| Рекурсивная вложенность | Не типично | Ключевая черта |
| Когда реализуется | После написания кода | Проектируется заранее |
| Пример в JDK | `InputStreamReader`, `Arrays.asList` | `BufferedInputStream`, `Collections.synchronizedList` |

### Вопрос 6. Что такое паттерн Facade и когда его стоит применять? Чем он отличается от простой utility-обёртки?

**Категория:** Паттерны · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Facade — refactoring.guru](https://refactoring.guru/design-patterns/facade)

Facade — структурный паттерн проектирования, который предоставляет упрощённый интерфейс к библиотеке, фреймворку или любому другому сложному набору классов. Фасад предоставляет простой интерфейс к сложной подсистеме, скрывая её сложность. В отличие от фреймворков и библиотек, фасад предоставляет только те функции, которые действительно нужны клиентам. Паттерн особенно полезен при работе с комплексными библиотеками и API. Фасад делает ваш код независимым от сложности подсистемы. Паттерн применяется, когда нужно предоставить простой интерфейс для сложной подсистемы или когда нужно структурировать подсистему на слои, создавая фасады в качестве входных точек на каждом уровне.

#### Ответ от Claude

**Facade (Фасад)** — структурный паттерн проектирования (GoF), который предоставляет упрощённый единый интерфейс к сложной подсистеме, состоящей из множества взаимосвязанных классов.

**Суть.** Фасад «знает», каким объектам подсистемы нужно переадресовать запрос, и оркестрирует их совместную работу. При этом сами классы подсистемы не подозревают о существовании фасада и остаются без изменений.

**Канонический пример (из refactoring.guru):**

```java
// Подсистема — несколько сложных классов
class VideoFile { ... }
class CodecFactory { static Codec extract(VideoFile f) { ... } }
class BitrateReader {
    static VideoFile read(VideoFile f, Codec c) { ... }
    static VideoFile convert(VideoFile f, Codec c) { ... }
}
class AudioMixer { VideoFile fix(VideoFile f) { ... } }

// Фасад — скрывает всю оркестрацию за одним методом
public class VideoConversionFacade {
    public File convertVideo(String fileName, String format) {
        VideoFile file = new VideoFile(fileName);
        Codec sourceCodec = CodecFactory.extract(file);
        Codec destCodec = createDestinationCodec(format);
        VideoFile buffer = BitrateReader.read(file, sourceCodec);
        VideoFile intermediate = BitrateReader.convert(buffer, destCodec);
        return new AudioMixer().fix(intermediate);
    }
}

// Клиент знает только о фасаде:
File mp4 = new VideoConversionFacade().convertVideo("video.avi", "mp4");
```

**Когда применять:**
1. Есть сложная подсистема из нескольких взаимозависимых классов, и клиенту нужно лишь узкое подмножество её возможностей.
2. Необходимо разбить систему на слои — фасады служат точками входа в каждый слой, снижая связность между слоями.
3. Нужно изолировать клиентский код от изменений в подсистеме (замена реализации не затронет клиентов).
4. Много «шаблонного» кода инициализации и конфигурации, который дублируется у клиентов.

**Примеры из Java-экосистемы:**
- `javax.faces.context.FacesContext` в Jakarta EE — фасад над несколькими контекстами жизненного цикла JSF.
- `SLF4J` — фасад над различными фреймворками логирования (Log4j, Logback, java.util.logging).
- `java.net.URL` — фасад над протоколами, соединениями, потоками.

---

**Чем Facade отличается от utility-обёртки (utility class):**

| Аспект | Facade | Utility-обёртка |
|--------|--------|----------------|
| Оборачивает | Несколько взаимосвязанных классов (подсистему) | Один класс или одну операцию |
| Состояние | Может хранить состояние и управлять жизненным циклом объектов подсистемы | Как правило, `static`-методы, без состояния |
| Оркестрация | Координирует взаимодействие нескольких объектов | Просто вызывает один объект или выполняет разрозненные вспомогательные операции |
| Связность | Снижает связность клиента с подсистемой | Не меняет архитектурную связность |
| Цель | Архитектурное упрощение | Удобство повторного использования кода |
| Пример | `VideoConversionFacade`, `SLF4J` | `java.util.Objects`, `java.util.Arrays`, `Collections` |

**Ключевое отличие:** фасад *оркестрирует несколько объектов*, управляет порядком их вызовов и, возможно, их жизненным циклом. Utility-класс — это набор независимых вспомогательных статических методов без скрытой сложности подсистемы.

**Потенциальная проблема.** Если фасад берёт на себя слишком много обязанностей, он превращается в «God Object» — антипаттерн. Признак: фасад зависит от большинства классов приложения. Решение — выделить несколько специализированных фасадов.

### Вопрос 7. Объясните паттерн Observer. Как он реализован в Swing/AWT? Какие у него недостатки при неаккуратном использовании?

**Категория:** Паттерны · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Observer — Refactoring Guru](https://refactoring.guru/design-patterns/observer)

Observer (Наблюдатель) — поведенческий паттерн проектирования, который позволяет одним объектам следить за изменениями в других. Паттерн определяет механизм подписки, позволяющий нескольким объектам получать уведомления обо всех событиях, происходящих с наблюдаемым объектом.

Структура паттерна включает пять элементов:
- **Publisher (Издатель)** — отправляет события и управляет подписками; хранит список ссылок на подписчиков и предоставляет методы для добавления и удаления из списка.
- **Subscriber Interface (Интерфейс подписчика)** — объявляет метод уведомления (обычно `update`).
- **Concrete Subscribers (Конкретные подписчики)** — реализуют интерфейс и обрабатывают уведомления.
- **Events/Context Data** — данные, передаваемые при уведомлении.
- **Client** — создаёт издателей и регистрирует подписчиков.

Из недостатков источник явно называет один: подписчики получают уведомления в произвольном порядке, что может стать проблемой, если порядок оповещения важен.

#### Ответ от Claude

**Суть паттерна.** Observer определяет отношение «один ко многим» между объектами: когда состояние одного объекта (Subject / Publisher) изменяется, все зависящие от него объекты (Observers / Subscribers) автоматически оповещаются. Паттерн реализует принцип слабой связанности: издатель знает о подписчиках только через абстрактный интерфейс.

**Ключевые участники:**
- `Subject` / `Publisher` — хранит список наблюдателей, предоставляет методы `addObserver`, `removeObserver`, `notifyObservers`.
- `Observer` / `Listener` — интерфейс с методом `update(event)`.
- `ConcreteObserver` — реализует реакцию на событие.

**Реализация в Swing/AWT.** Swing и AWT реализуют Observer через систему событий и слушателей (listeners):
- Компонент (например, `JButton`) является Subject — он хранит список зарегистрированных слушателей через `EventListenerList`.
- Слушатели (например, `ActionListener`, `MouseListener`, `ChangeListener`) являются Observer-ами.
- Регистрация: `button.addActionListener(listener)`, дерегистрация: `button.removeActionListener(listener)`.
- При возникновении события компонент создаёт объект-событие (например, `ActionEvent`) и последовательно вызывает метод (`actionPerformed`) у каждого зарегистрированного слушателя.
- Весь вызов происходит в потоке Event Dispatch Thread (EDT); пакет `javax.swing.event` явно предупреждает, что большинство Swing API не потокобезопасно.

**Недостатки при неаккуратном использовании:**

1. **Утечки памяти (Lapsed Listener Problem)** — наиболее известная проблема. Subject хранит сильную ссылку на Observer. Если Observer не отписывается (не вызывает `removeListener`), он не может быть собран сборщиком мусора, пока жив Subject. Это одна из самых распространённых утечек памяти в Java-приложениях с GUI. Решения: явный вызов `removeListener` в методах жизненного цикла компонента; хранение слабых ссылок (`WeakReference`) на подписчиков.

2. **Блокировка Event Dispatch Thread** — в Swing/AWT все слушатели вызываются в EDT. Если слушатель выполняет долгую операцию (запрос к БД, сетевой вызов), интерфейс зависает. Решение: переносить тяжёлые операции в `SwingWorker` или отдельный поток.

3. **Неопределённый порядок оповещения** — подписчики уведомляются в порядке регистрации (или произвольно), что может привести к труднодиагностируемым багам при наличии зависимостей между слушателями.

4. **Каскадные обновления** — один Observer при обновлении может изменять Subject, что инициирует новый цикл уведомлений. При циклических зависимостях это приводит к бесконечной рекурсии или переполнению стека.

5. **Неожиданные уведомления** — Observer не знает, какие другие наблюдатели подписаны на тот же Subject; это затрудняет отладку и рассуждение о порядке побочных эффектов.

### Вопрос 8. Что такое паттерн Strategy? Чем он отличается от паттерна State?

**Категория:** Паттерны · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Strategy vs State Pattern — Baeldung on Computer Science](https://www.baeldung.com/cs/design-state-pattern-vs-strategy-pattern)

Оба паттерна структурно схожи — оба основаны на композиции и изменяют поведение объекта-контекста, делегируя работу вспомогательным объектам. Ключевые различия:

- **Цель Strategy** — создать иерархию взаимозаменяемых алгоритмов. Клиент сам выбирает нужный алгоритм и явно передаёт его контексту.
- **Цель State** — управлять поведением объекта в зависимости от его внутреннего состояния. Смена поведения инициируется самим контекстом или конкретными состояниями.
- В Strategy клиент должен знать о существующих стратегиях и менять их явно.
- В State конкретные состояния могут знать друг о друге и самостоятельно инициировать переходы между состояниями, тогда как стратегии почти никогда не знают друг о друге.

#### Ответ от Claude

**Паттерн Strategy.**

Strategy определяет семейство алгоритмов, помещает каждый в отдельный класс и делает их объекты взаимозаменяемыми. Контекст хранит ссылку на объект-стратегию и делегирует ей выполнение работы.

Участники:
- `Context` — хранит ссылку на `Strategy`, вызывает её метод.
- `Strategy` (интерфейс) — объявляет метод выполнения алгоритма.
- `ConcreteStrategy` — реализует конкретный алгоритм.

Типичный пример: `Comparator<T>` в Java — это и есть стратегия сортировки, которую клиент явно передаёт в `Collections.sort()`.

**Паттерн State.**

State позволяет объекту изменять своё поведение при изменении внутреннего состояния — так, что объект выглядит изменившим свой класс. Контекст хранит ссылку на текущий объект-состояние и делегирует ему обработку запросов.

Пример: медиаплеер в состоянии `PlayingState` обрабатывает нажатие кнопки иначе, чем в `PausedState`. При нажатии кнопки `PlayingState` сам переключает контекст в `PausedState`.

**Ключевые отличия:**

| Аспект | Strategy | State |
|---|---|---|
| Цель | Выбор алгоритма | Управление жизненным циклом объекта |
| Кто меняет поведение | Клиент (явно, снаружи) | Контекст или само состояние (изнутри) |
| Осведомлённость объектов | Стратегии не знают друг о друге | Состояния могут знать друг о друге и инициировать переходы |
| Изменяемость | Алгоритм можно поменять в любой момент | Переходы определяются логикой состояний |
| Аналогия | Переключение режима оплаты (карта, PayPal, криптовалюта) | Светофор: красный → зелёный → жёлтый → красный |

**Итог:** State можно рассматривать как расширение Strategy. Если объекты-поведения независимы, взаимозаменяемы и выбираются клиентом — это Strategy. Если поведения зависят от внутреннего состояния контекста, могут переключать друг друга и образуют конечный автомат — это State.

### Вопрос 9. Что такое паттерн Template Method? В чём его связь с принципом Голливуда («не звони нам — мы позвоним»)?

**Категория:** Паттерны · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Template Method Pattern — springframework.guru](https://springframework.guru/gang-of-four-design-patterns/template-method-pattern/)

Template Method — поведенческий паттерн проектирования, который, согласно определению Gang of Four, «определяет скелет алгоритма в операции, откладывая некоторые шаги до подклассов».

Абстрактный класс содержит шаблонный метод — единственный метод, который оркестрирует алгоритм, вызывая другие методы в заданной последовательности. Часть методов реализована конкретно (общие шаги), другие объявлены абстрактными (требуют реализации в подклассах). Также возможны hook-методы (необязательные, не финальные), которые подкласс может переопределить, не меняя структуры алгоритма.

Паттерн воплощает принцип Голливуда («не звони нам — мы позвоним»): вместо того чтобы подклассы вызывали методы суперкласса, абстрактный базовый класс управляет потоком алгоритма и сам вызывает методы подклассов по мере необходимости. Такая инверсия управления предотвращает проблему «гнилой зависимости» (dependency rot), когда низкоуровневые компоненты зависят от высокоуровневых.

#### Ответ от Claude

**Суть паттерна.**

Template Method определяет скелет алгоритма в методе базового класса, откладывая реализацию отдельных шагов в подклассы. Подклассы могут переопределять шаги алгоритма, не меняя его общей структуры.

**Структура:**
- **Абстрактный базовый класс** содержит:
  - `templateMethod()` — как правило, объявляется `final`, чтобы подклассы не могли изменить порядок шагов.
  - Абстрактные методы-шаги (primitive operations) — обязательны к реализации в подклассах.
  - Конкретные методы с дефолтной реализацией — опционально переопределяются.
  - Hook-методы — пустые или с минимальной реализацией; подкласс переопределяет их по желанию.
- **Конкретный подкласс** реализует абстрактные шаги, предоставляя специфичное поведение.

**Пример в JDK.** `java.util.AbstractList` и `java.util.AbstractSet` — это шаблонные методы в JDK. Метод `AbstractList.indexOf()` определяет общий алгоритм поиска, вызывая абстрактный метод `get(int index)`, который реализуют `ArrayList`, `LinkedList` и другие конкретные подклассы.

**Связь с принципом Голливуда.**

Принцип Голливуда гласит: «Не звони нам — мы позвоним тебе» ("Don't call us, we'll call you"). В контексте проектирования это означает инверсию управления: высокоуровневый компонент (абстрактный класс / фреймворк) сам управляет потоком выполнения и вызывает методы низкоуровневых компонентов (подклассов / приложения) в нужный момент.

В Template Method это реализуется так:
- Подкласс не вызывает `super.templateMethod()` самостоятельно — он только реализует шаги.
- Абстрактный суперкласс (фреймворк) вызывает реализации подкласса через полиморфизм.
- Подкласс «регистрирует» своё поведение посредством переопределения методов и ждёт, пока суперкласс его «позовёт».

Это фундаментальный принцип большинства фреймворков: Spring вызывает `afterPropertiesSet()` / `@PostConstruct` у бинов; JUnit вызывает `setUp()`, тестовые методы и `tearDown()` у тест-классов — разработчик не управляет порядком, фреймворк делает это сам.

**Сравнение с Strategy.** Оба паттерна позволяют варьировать часть алгоритма. Разница в механизме: Template Method использует наследование (подкласс переопределяет шаги), Strategy — композицию (контекст делегирует работу объекту-стратегии). Паттерн Template Method считается менее гибким, поскольку привязывает алгоритм к иерархии классов; Strategy позволяет менять алгоритм в рантайме.
### Вопрос 10. Что такое паттерн Command? Как его использовать для реализации механизма undo/redo?

**Категория:** Паттерны · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Command in Java / Design Patterns — Refactoring.Guru](https://refactoring.guru/design-patterns/command/java/example)

Пример на сайте демонстрирует текстовый редактор с поддержкой отмены операций. Command — поведенческий паттерн, который «превращает запросы в самостоятельные объекты, содержащие всю информацию о запросе». Это позволяет передавать запросы в виде аргументов метода, откладывать или ставить их в очередь, а также поддерживать отменяемые операции.

Ключевые классы примера:
- `Command` (абстрактный базовый класс) — хранит резервную копию состояния редактора и объявляет методы `execute()` и `undo()`.
- `CopyCommand`, `PasteCommand`, `CutCommand` — конкретные команды; каждая перед изменением состояния вызывает `backup()`.
- `CommandHistory` — стек (`Stack<Command>`) выполненных команд.
- `Editor` — инвокер, создающий объекты команд по нажатию кнопок UI.

Механизм undo: при нажатии кнопки создаётся объект команды, вызывается `executeCommand()` — который запускает команду и помещает её в историю. При нажатии «Отменить» последняя команда извлекается из стека (`pop`) и вызывается её `undo()`, восстанавливающий ранее сохранённое состояние.

#### Ответ от Claude

**Паттерн Command** (Команда) — поведенческий паттерн проектирования, инкапсулирующий запрос в виде объекта. Это позволяет параметризировать клиентов с разными запросами, ставить запросы в очередь или записывать их в журнал, а также поддерживать отмену операций.

**Участники паттерна:**
- **Command** — интерфейс (или абстрактный класс) с методом `execute()` и, при поддержке undo, методом `undo()`.
- **ConcreteCommand** — реализует конкретную операцию; хранит ссылку на Receiver и все параметры вызова.
- **Receiver** — объект, содержащий бизнес-логику (реальную работу делает он).
- **Invoker** (отправитель) — хранит ссылку на команду и вызывает её; не знает о конкретных типах команд.
- **Client** — создаёт объекты ConcreteCommand, передаёт им Receiver и связывает с Invoker.

**Реализация undo/redo через стек истории:**

```java
// Интерфейс команды
public interface Command {
    boolean execute();
    void undo();
}

// Конкретная команда — вставка текста
public class PasteCommand implements Command {
    private final Editor editor;
    private String backup;

    public PasteCommand(Editor editor) {
        this.editor = editor;
    }

    @Override
    public boolean execute() {
        backup = editor.getText();           // сохраняем состояние ДО
        editor.setText(editor.getText() + editor.getClipboard());
        return !editor.getClipboard().isEmpty();
    }

    @Override
    public void undo() {
        editor.setText(backup);              // восстанавливаем состояние
    }
}

// Инвокер с историей команд
public class CommandHistory {
    private final Deque<Command> history = new ArrayDeque<>();

    public void push(Command c) { history.push(c); }

    public Command pop() {
        return history.isEmpty() ? null : history.pop();
    }
}

// Использование
public class EditorApp {
    private final CommandHistory history = new CommandHistory();

    public void executeCommand(Command command) {
        if (command.execute()) {
            history.push(command);
        }
    }

    public void undo() {
        Command command = history.pop();
        if (command != null) {
            command.undo();
        }
    }
}
```

**Два подхода к реализации undo:**
1. **Сохранение состояния** — команда перед выполнением сохраняет снимок объекта (`backup`). При undo восстанавливает его. Прост в реализации, но требует памяти под каждый снимок.
2. **Обратная операция** — команда реализует операцию, противоположную выполненной (например, для `+5` обратная — `-5`). Экономит память, но не всегда возможна (например, для необратимых операций типа удаления файла).

**Для redo** достаточно второго стека: при undo команда перемещается из стека истории в стек повтора; при redo — обратно с повторным вызовом `execute()`.

**Применение в Java/Spring:**
- `java.lang.Runnable` — упрощённый аналог Command без undo.
- `javax.swing.Action` — расширение Command для Swing UI.
- Spring Batch — шаги (`Step`) как команды с поддержкой retry/rollback.
- Транзакции в Spring можно рассматривать как механизм отмены на уровне БД.

**Связь с Memento:** При сложных объектах снимок состояния выносят в отдельный объект Memento; команда хранит не само состояние, а Memento.

### Вопрос 11. Почему реализация Singleton через enum считается лучшей? Какие проблемы она решает по сравнению с double-checked locking?

**Категория:** Паттерны · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Java Singleton Design Pattern Best Practices with Examples — DigitalOcean](https://www.digitalocean.com/community/tutorials/java-singleton-design-pattern-best-practices-examples)

Статья описывает семь реализаций Singleton. Для Double-Checked Locking (DCL) объясняется ключевая проблема: без `volatile` JVM может переупорядочить шаги создания объекта: (1) выделить память, (2) выполнить конструктор, (3) присвоить ссылку переменной `instance`. Без `volatile` присваивание может произойти до завершения конструктора, и другой поток увидит частично инициализированный объект. Ключевое слово `volatile` устанавливает отношение happens-before между записью и последующими чтениями.

Для Enum Singleton источник цитирует Joshua Bloch: реализация «обеспечивает встроенную защиту от атак через рефлексию и сериализацию». Конкретно:
- **Рефлексия:** JVM запрещает рефлективное создание экземпляров enum; `Constructor.newInstance()` на enum бросает `IllegalArgumentException`.
- **Сериализация:** JVM при десериализации enum возвращает уже существующую константу, не вызывая конструктор и не требуя `readResolve()`.

Итоговая таблица: Enum — единственная реализация, которая одновременно thread-safe, reflection-safe и serialization-safe.

#### Ответ от Claude

**Singleton через enum** — рекомендация из книги Joshua Bloch «Effective Java» (3-е издание, Item 3). Реализация выглядит так:

```java
public enum DatabaseConnection {
    INSTANCE;

    private final String url = "jdbc:postgresql://localhost:5432/app";

    public Connection getConnection() {
        // ... получение соединения
        return null;
    }
}

// Использование
DatabaseConnection.INSTANCE.getConnection();
```

**Проблемы Double-Checked Locking (DCL) и как enum их решает:**

**1. Проблема частично инициализированного объекта (без volatile — до Java 5)**

```java
// ОПАСНАЯ реализация (без volatile)
public class Singleton {
    private static Singleton instance; // НЕТ volatile

    public static Singleton getInstance() {
        if (instance == null) {                     // проверка 1 (без lock)
            synchronized (Singleton.class) {
                if (instance == null) {             // проверка 2 (с lock)
                    instance = new Singleton();     // ОПАСНО
                }
            }
        }
        return instance;
    }
}
```

Инструкция `instance = new Singleton()` не атомарна. JVM может переупорядочить её в:
- шаг 1: выделить память
- шаг 3 (переупорядочено): записать ссылку в `instance`
- шаг 2: вызвать конструктор

Поток B, выполняя первую проверку `instance == null`, увидит ненулевую ссылку на ещё не инициализированный объект и вернёт его — это причина трудновоспроизводимых NullPointerException или некорректного поведения.

**Исправление через `volatile`:**

```java
private static volatile Singleton instance; // volatile обязателен
```

`volatile` запрещает переупорядочивание инструкций и гарантирует happens-before между записью и последующим чтением. Это работает корректно начиная с Java 5 (JSR-133, пересмотренная модель памяти).

**Оставшиеся проблемы DCL даже с volatile:**

**2. Уязвимость к атакам через рефлексию:**
```java
// Можно сломать DCL-синглтон через рефлексию:
Constructor<Singleton> c = Singleton.class.getDeclaredConstructor();
c.setAccessible(true);
Singleton instance2 = c.newInstance(); // создаёт НОВЫЙ экземпляр!
```

Для защиты нужно вручную добавлять проверку в конструктор:
```java
private Singleton() {
    if (instance != null) throw new RuntimeException("Use getInstance()");
}
```

**3. Уязвимость к сериализации:**
При использовании `Serializable` десериализация создаёт новый экземпляр. Для защиты нужно вручную реализовать `readResolve()`:
```java
protected Object readResolve() { return instance; }
```

**Почему enum решает все три проблемы:**

| Проблема | DCL | Enum |
|---|---|---|
| Thread safety | Требует `volatile` | Гарантируется JVM при загрузке класса |
| Рефлексия | Требует защиты в конструкторе | JVM бросает `IllegalArgumentException` при любой попытке |
| Сериализация | Требует `readResolve()` | JVM возвращает существующую константу автоматически |
| Краткость | ~15 строк | 3 строки |

**Ограничение enum Singleton:** не поддерживает ленивую инициализацию (lazy initialization). Если создание экземпляра дорого и он может не понадобиться, предпочтительна реализация через статический внутренний класс (паттерн Билла Пью):

```java
public class Singleton {
    private Singleton() {}

    private static class Holder {
        private static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}
```

Она thread-safe (загрузка класса `Holder` гарантирует happens-before), ленива (класс `Holder` загружается только при первом вызове `getInstance()`), но при этом также уязвима к рефлексии и сериализации.

**Итог:** enum — лучший выбор, когда lazy initialization не нужна; паттерн Билла Пью — когда нужна ленивая инициализация, но можно пренебречь защитой от рефлексии (в контролируемой среде).

### Вопрос 12. Чем паттерн Proxy отличается от Decorator? Какие виды проксирования используются в Spring (JDK dynamic proxy vs. CGLIB) и когда Spring выбирает каждый из них?

**Категория:** Паттерны · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Proxying Mechanisms :: Spring Framework — docs.spring.io](https://docs.spring.io/spring-framework/reference/core/aop/proxying.html)

Официальная документация Spring описывает правила выбора механизма проксирования:

- **JDK Dynamic Proxy** используется по умолчанию, если целевой объект реализует хотя бы один интерфейс. Все интерфейсы, реализуемые целевым типом, будут проксированы.
- **CGLIB** используется, если целевой объект не реализует ни одного интерфейса — создаётся сгенерированный в рантайме подкласс целевого типа.

Для принудительного использования CGLIB: `@EnableAspectJAutoProxy(proxyTargetClass = true)` или `proxy-target-class="true"` в XML.

Ограничения CGLIB: нельзя проксировать `final`-классы (не расширяются) и advise `final`-методы (не переопределяются).

Важная проблема: **self-invocation** — вызов метода через `this` обходит прокси, и перехватчики (advice) не применяются. Решение — рефакторинг или инъекция самой зависимости через прокси.

#### Ответ от Claude

**Proxy vs Decorator: ключевые отличия**

Оба паттерна — структурные, оба оборачивают объект и делегируют ему вызовы. Однако их **намерение** принципиально различается:

| Аспект | Proxy | Decorator |
|---|---|---|
| **Цель** | Управление *доступом* к объекту | Добавление *поведения* к объекту |
| **Интерфейс** | Остаётся тем же | Остаётся тем же или расширяется |
| **Знание об объекте** | Proxy может сам управлять жизненным циклом объекта | Декоратор получает готовый объект снаружи |
| **Вложенность** | Как правило, один прокси | Декораторы можно вкладывать рекурсивно |
| **Время связывания** | Обычно известен конкретный тип на этапе компиляции | Объект внедряется в рантайме |

**Proxy:** клиент не знает, что работает не с реальным объектом. Прокси контролирует: когда создать объект (virtual proxy), кому разрешить доступ (protection proxy), как кешировать результаты (caching proxy), как логировать вызовы (logging proxy), как работать с удалённым объектом (remote proxy).

**Decorator:** клиент явно оборачивает объект, добавляя поведение. Классический пример: `BufferedReader` над `FileReader` в Java IO.

```java
// Decorator — клиент сам управляет оборачиванием
InputStream is = new BufferedInputStream(new GZIPInputStream(new FileInputStream("file")));

// Proxy — клиент получает объект, не зная о прокси
@Transactional  // Spring создаёт прокси прозрачно для клиента
public void transferMoney() { ... }
```

---

**JDK Dynamic Proxy vs CGLIB в Spring**

**JDK Dynamic Proxy:**
- Генерирует прокси-класс, реализующий те же интерфейсы, что и целевой бин.
- Реализован в `java.lang.reflect.Proxy` + `InvocationHandler`.
- Все вызовы методов интерфейса перехватываются через `InvocationHandler.invoke()`.
- Работает только для методов, объявленных в интерфейсе — методы класса, не входящие в интерфейс, не перехватываются.

```java
// Целевой класс реализует интерфейс — Spring использует JDK proxy
public interface UserService {
    void createUser(String name);
}

@Service
public class UserServiceImpl implements UserService {
    public void createUser(String name) { ... }
}
```

**CGLIB (Code Generation Library):**
- Генерирует подкласс целевого класса в байткоде.
- Переопределяет все не-final публичные методы, вставляя перехватчики (interceptors).
- Не требует наличия интерфейса.
- Включён в `spring-core` как `org.springframework.cglib`.

```java
// Целевой класс без интерфейса — Spring использует CGLIB
@Service
public class OrderService {
    public void placeOrder(Order order) { ... }
}
```

**Правила выбора (Spring Framework, ядро):**
1. Если бин реализует хотя бы один интерфейс → **JDK Dynamic Proxy**.
2. Если бин не реализует ни одного интерфейса → **CGLIB**.
3. Если явно задано `proxy-target-class=true` или `proxyTargetClass=true` → **CGLIB** в любом случае.

**Spring Boot (с версии 2.0) меняет умолчание:**
Начиная с Spring Boot 2.0, свойство `spring.aop.proxy-target-class` по умолчанию равно `true`. Это означает, что **Spring Boot по умолчанию использует CGLIB** для всех бинов, даже если те реализуют интерфейсы. Это сделано для устранения ряда неочевидных проблем (например, когда бин внедряется по типу класса, а не по типу интерфейса).

Переопределить можно через `application.properties`:
```properties
spring.aop.proxy-target-class=false
```

**Ограничения CGLIB:**
- Нельзя проксировать `final`-классы.
- Нельзя advise `final`-методы (не переопределяются подклассом).
- Нельзя advise `private`-методы.
- Конструктор не должен иметь побочных эффектов (хотя Spring использует Objenesis для создания объекта без вызова конструктора в CGLIB 3.2+).

**Ограничения JDK Dynamic Proxy:**
- Работает только с методами интерфейса.
- Если код инжектирует бин по типу класса (а не интерфейса), возникнет `ClassCastException`.

**Проблема self-invocation (общая для обоих видов прокси):**

```java
@Service
public class PaymentService {
    @Transactional
    public void processPayment() {
        this.sendNotification(); // НЕ будет транзакционным — вызов через this, не через прокси
    }

    @Transactional
    public void sendNotification() { ... }
}
```

Решение: вынести `sendNotification()` в отдельный бин, либо инжектировать `PaymentService` сам в себя (`@Autowired private PaymentService self`), либо использовать AspectJ-weaving.

**Практическое правило выбора:**
- Используйте интерфейсы и JDK proxy, если хотите следовать принципу «программируй под интерфейс».
- Используйте CGLIB (или оставьте Spring Boot умолчание), если удобнее работать с конкретными классами или нужно перехватывать методы, не объявленные в интерфейсе.

### Вопрос 13. Как паттерн Abstract Factory применяется в Spring? Приведите конкретный пример из Spring-контекста (например, BeanFactory vs. ApplicationContext).

**Категория:** Паттерны · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Chapter 3. Beans, BeanFactory and the ApplicationContext — Spring Framework 1.2.x Reference](https://docs.spring.io/spring-framework/docs/1.2.x/reference/beans.html)

Spring строит IoC-контейнер вокруг двух интерфейсов: `BeanFactory` и `ApplicationContext`. `BeanFactory` — корневой интерфейс, объявляющий фабричные методы (`getBean(String)`, `getBean(String, Class)`, `containsBean`, `isSingleton`), которые возвращают готовые бины из контейнера. Клиентский код никогда не создаёт бины напрямую — он лишь запрашивает продукт у фабрики, не зная о его конкретном классе.

`ApplicationContext` расширяет `BeanFactory` и добавляет enterprise-возможности: публикацию событий, i18n, загрузку ресурсов. Конкретные реализации (`ClassPathXmlApplicationContext`, `FileSystemXmlApplicationContext`, `XmlWebApplicationContext`) выступают **конкретными фабриками**, каждая из которых читает конфигурацию из своего источника и создаёт одно и то же семейство бинов.

Spring поддерживает три способа «производства» объектов: через конструктор (`<bean class="..."/>`), через статический фабричный метод (`factory-method="createInstance"`) и через метод экземпляра (`factory-bean="..." factory-method="..."`). Кроме того, интерфейс `FactoryBean` позволяет одному бину самому быть фабрикой: метод `getObject()` возвращает продукт, а обращение к `&beanName` возвращает саму фабрику.

#### Ответ от Claude

**Суть паттерна и его проявление в Spring**

Abstract Factory — порождающий паттерн, предоставляющий интерфейс для создания **семейств** связанных объектов без привязки клиентского кода к конкретным классам. Spring реализует этот паттерн сразу на нескольких уровнях.

**Уровень 1: BeanFactory как абстрактная фабрика**

`org.springframework.beans.factory.BeanFactory` — это и есть абстрактная фабрика. Её фабричные методы (`getBean(Class<T>)`, `getBean(String, Class<T>)`) — это «создающие операции» в терминологии GoF. Конкретные реализации — `DefaultListableBeanFactory`, `XmlBeanFactory` (устарел) — являются конкретными фабриками, создающими конкретные продукты-бины.

**Уровень 2: ApplicationContext расширяет абстракцию**

`ApplicationContext` расширяет `BeanFactory` и представляет «улучшенную» абстрактную фабрику. Иерархия конкретных фабрик:

| Конкретная фабрика | Источник конфигурации |
|---|---|
| `ClassPathXmlApplicationContext` | XML из classpath |
| `FileSystemXmlApplicationContext` | XML из файловой системы |
| `AnnotationConfigApplicationContext` | Java-конфигурация (`@Configuration`) |
| `GenericWebApplicationContext` | Web-среда |

Клиентский код работает только с интерфейсом `ApplicationContext`, не зная, какая именно реализация используется. Это и есть суть Abstract Factory.

**Уровень 3: FactoryBean — фабрика внутри фабрики**

Интерфейс `FactoryBean<T>` позволяет любому бину становиться дополнительной фабрикой:

```java
public interface FactoryBean<T> {
    T getObject() throws Exception;       // продукт
    Class<?> getObjectType();             // тип продукта
    default boolean isSingleton() { return true; }
}
```

Пример из Spring Data: `LocalContainerEntityManagerFactoryBean` создаёт `EntityManagerFactory` — целое семейство JPA-объектов. Клиент запрашивает `getBean("entityManagerFactory")` и получает готовый `EntityManagerFactory`, не зная о деталях его инициализации.

**Пример кода**

```java
// Абстрактная фабрика — интерфейс
ApplicationContext ctx = new AnnotationConfigApplicationContext(AppConfig.class);

// Клиент запрашивает продукт по типу — не знает о конкретном классе
DataSource ds = ctx.getBean(DataSource.class);
UserRepository repo = ctx.getBean(UserRepository.class);
```

При тестировании `ApplicationContext` можно заменить на `GenericApplicationContext` с тестовыми заглушками — клиентский код меняться не будет. Это и демонстрирует полиморфизм Abstract Factory.

**Дополнение: отличие от Factory Method**

В Spring также широко используется паттерн Factory Method (например, в `AbstractApplicationContext` — метод `createBeanFactory()`, который подклассы переопределяют). Отличие: Factory Method создаёт один продукт; Abstract Factory — семейства взаимосвязанных продуктов (бинов с их зависимостями).

### Вопрос 14. Как паттерн Decorator используется в Java I/O (InputStream/OutputStream)? Какие плюсы и минусы такого подхода?

**Категория:** Паттерны · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [FilterInputStream — Java Platform SE 8 API Documentation](https://docs.oracle.com/javase/8/docs/api/java/io/FilterInputStream.html)

`FilterInputStream` содержит некоторый другой `InputStream`, используя его как основной источник данных, возможно трансформируя данные «на лету» или предоставляя дополнительную функциональность. Класс `FilterInputStream` сам по себе просто переопределяет все методы `InputStream` версиями, которые передают все вызовы содержащемуся потоку. Подклассы `FilterInputStream` могут дополнительно переопределять часть методов и обеспечивать дополнительные поля и методы.

Известные подклассы: `BufferedInputStream`, `DataInputStream`, `PushbackInputStream`, `CheckedInputStream` (из `java.util.zip`), `DigestInputStream`, `CipherInputStream`, `InflaterInputStream`, `DeflaterInputStream`, `LineNumberInputStream`, `ProgressMonitorInputStream`.

#### Ответ от Claude

**Структура паттерна в Java I/O**

Паттерн Decorator (Обёртка) реализован в `java.io` через следующую иерархию:

```
InputStream (абстрактный компонент)
├── FileInputStream      (конкретный компонент — источник данных)
├── ByteArrayInputStream (конкретный компонент — источник данных)
├── PipedInputStream     (конкретный компонент — источник данных)
└── FilterInputStream    (абстрактный декоратор — держит ссылку на InputStream)
    ├── BufferedInputStream    (конкретный декоратор — буферизация)
    ├── DataInputStream        (конкретный декоратор — чтение примитивов)
    ├── PushbackInputStream    (конкретный декоратор — откат прочитанных байт)
    ├── CheckedInputStream     (конкретный декоратор — контрольная сумма)
    ├── DigestInputStream      (конкретный декоратор — дайджест/хэш)
    ├── CipherInputStream      (конкретный декоратор — шифрование)
    └── InflaterInputStream    (конкретный декоратор — распаковка)
```

Аналогичная иерархия реализована для `OutputStream` через `FilterOutputStream`.

**Как это работает**

`FilterInputStream` хранит ссылку на оборачиваемый поток в защищённом поле `in` и делегирует ему все операции:

```java
public class FilterInputStream extends InputStream {
    protected volatile InputStream in;

    protected FilterInputStream(InputStream in) {
        this.in = in;
    }

    public int read() throws IOException {
        return in.read(); // чистое делегирование
    }
}
```

Подкласс переопределяет только нужные методы, добавляя поведение:

```java
// BufferedInputStream.read() читает из буфера, заполняя его из in.read()
```

**Пример композиции декораторов**

```java
// Читаем сжатый зашифрованный файл с буферизацией
InputStream raw = new FileInputStream("data.gz.enc");
InputStream decrypted = new CipherInputStream(raw, cipher);
InputStream decompressed = new InflaterInputStream(decrypted);
InputStream buffered = new BufferedInputStream(decompressed, 8192);
DataInputStream data = new DataInputStream(buffered);

int value = data.readInt(); // прозрачно проходит через всю цепочку
```

**Плюсы подхода**

1. **Гибкая композиция.** Функциональность добавляется и комбинируется произвольно без изменения базовых классов. Порядок обёрток меняет поведение: `BufferedInputStream(CipherInputStream(...))` — сначала расшифровка, потом буферизация.
2. **Принцип открытости/закрытости.** Базовые классы (`FileInputStream`, `SocketInputStream`) не изменяются, новое поведение добавляется снаружи.
3. **Предотвращение взрыва классов.** Без паттерна Decorator пришлось бы создавать классы для каждой комбинации: `BufferedFileInputStream`, `BufferedEncryptedFileInputStream`, `DataBufferedFileInputStream` и т.д. — экспоненциальный рост.
4. **Единый интерфейс.** Клиентский код работает с `InputStream` и не знает, сколько декораторов стоит в цепочке.
5. **Принцип единственной ответственности.** Каждый декоратор решает одну задачу.

**Минусы подхода**

1. **Сложность инициализации.** Вложенные конструкторы трудно читать, особенно в глубоких цепочках: `new DataInputStream(new BufferedInputStream(new GZIPInputStream(new FileInputStream(path))))`.
2. **Трудная отладка.** Исключение возникает глубоко в цепочке — стектрейс содержит несколько уровней делегирования, сложно понять, в каком декораторе произошла ошибка.
3. **Много мелких объектов.** Каждый декоратор — отдельный объект на куче; длинные цепочки создают давление на GC.
4. **Потеря специфических типов.** После оборачивания нельзя просто привести `InputStream` обратно к `FileInputStream` для вызова специфических методов без `instanceof` и каскада приведений.
5. **Непоследовательность иерархии.** Дизайнеры `java.io` смешали в иерархии одного дерева и источники данных, и декораторы (все унаследованы от `InputStream`), что нарушает чистоту паттерна. Jenkov.com прямо указывает: `FilterInputStream` практически не несёт ценности сам по себе — можно было напрямую наследоваться от `InputStream`.
6. **Устаревший дизайн API.** В современном Java предпочтительнее использовать `java.nio.channels` и `java.io.InputStream.transferTo()`, которые снижают необходимость в ручных цепочках декораторов.

### Вопрос 15. Как паттерн Template Method реализован в Spring (например, JdbcTemplate, RestTemplate)? Какой метод является «шаблонным», а какие — «крючками» для переопределения?

**Категория:** Паттерны · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [JdbcTemplate — Spring Framework 7.0.7 API Javadoc](https://docs.spring.io/spring-framework/docs/current/javadoc-api/org/springframework/jdbc/core/JdbcTemplate.html)

`JdbcTemplate` упрощает работу с JDBC и помогает избежать типичных ошибок. Класс выполняет основной JDBC-процесс, оставляя прикладному коду задачу предоставить SQL и извлечь результаты. Код, использующий этот класс, должен реализовывать только callback-интерфейсы с чётко определённым контрактом. Основные callback-интерфейсы: `PreparedStatementCreator` (создаёт `PreparedStatement` по заданному `Connection`), `ResultSetExtractor` (извлекает значения из `ResultSet`), `RowMapper` (отображает одну строку результата в объект Java), `RowCallbackHandler` (обрабатывает строки без возврата значений), `PreparedStatementSetter` (устанавливает параметры PreparedStatement). Защищённые методы-расширения: `applyStatementSettings(Statement)`, `handleWarnings(Statement)`, `translateException(String, String, SQLException)`.

#### Ответ от Claude

**Суть паттерна Template Method**

Template Method (GoF) определяет скелет алгоритма в базовом классе, делегируя часть шагов подклассам или callback-объектам. В Spring паттерн реализован в вариации **с callback-интерфейсами** вместо наследования, что делает его более гибким — не нужно создавать подклассы.

**JdbcTemplate**

Шаблонный алгоритм (скелет, всегда выполняется одинаково):

```
1. Получить Connection из DataSource
2. Создать Statement / PreparedStatement
3. Применить настройки (fetchSize, maxRows, queryTimeout)
4. Выполнить SQL
5. Передать ResultSet в callback для обработки
6. Перевести SQLExceptions в Spring DataAccessException
7. Закрыть ресурсы (Connection, Statement, ResultSet)
```

Публичные «шаблонные» методы (`query`, `update`, `execute`) реализуют этот алгоритм полностью. Шаги, делегируемые пользователю через callback-интерфейсы:

| Callback-интерфейс | Метод-крючок | Назначение |
|---|---|---|
| `RowMapper<T>` | `T mapRow(ResultSet rs, int rowNum)` | Маппинг одной строки в объект |
| `ResultSetExtractor<T>` | `T extractData(ResultSet rs)` | Полный контроль над обходом ResultSet |
| `RowCallbackHandler` | `void processRow(ResultSet rs)` | Обработка строки без возврата |
| `PreparedStatementCreator` | `PreparedStatement createPreparedStatement(Connection con)` | Создание statement с параметрами |
| `PreparedStatementSetter` | `void setValues(PreparedStatement ps)` | Установка параметров |
| `PreparedStatementCallback<T>` | `T doInPreparedStatement(PreparedStatement ps)` | Произвольные действия над statement |
| `ConnectionCallback<T>` | `T doInConnection(Connection con)` | Низкоуровневый доступ к Connection |

Защищённые методы — **традиционные крючки** для наследников:

```java
protected void applyStatementSettings(Statement stmt)   // настройки перед выполнением
protected void handleWarnings(Statement stmt)            // обработка SQLWarning
protected DataAccessException translateException(...)    // перевод исключений
```

Пример использования:

```java
// RowMapper — крючок для шага «обработать строку»
List<User> users = jdbcTemplate.query(
    "SELECT id, name FROM users WHERE age > ?",
    (rs, rowNum) -> new User(rs.getLong("id"), rs.getString("name")),
    18
);

// ResultSetExtractor — крючок для шага «обработать весь ResultSet»
Map<Long, List<String>> grouped = jdbcTemplate.query(
    "SELECT dept_id, name FROM employees",
    rs -> {
        Map<Long, List<String>> result = new LinkedHashMap<>();
        while (rs.next()) {
            result.computeIfAbsent(rs.getLong("dept_id"), k -> new ArrayList<>())
                  .add(rs.getString("name"));
        }
        return result;
    }
);
```

**RestTemplate**

`RestTemplate` реализует Template Method аналогичным образом. Шаблонный метод `execute(URI, HttpMethod, RequestCallback, ResponseExtractor<T>)` — это скелет HTTP-запроса:

```
1. Создать ClientHttpRequest
2. Выполнить RequestCallback (добавить заголовки, тело)
3. Выполнить запрос
4. Передать ответ в ResponseExtractor
5. Обработать ошибки через ResponseErrorHandler
6. Вернуть результат
```

Крючки:

| Callback | Метод-крючок | Назначение |
|---|---|---|
| `RequestCallback` | `void doWithRequest(ClientHttpRequest request)` | Настройка запроса (заголовки, тело) |
| `ResponseExtractor<T>` | `T extractData(ClientHttpResponse response)` | Извлечение данных из ответа |
| `ResponseErrorHandler` | `void handleError(ClientHttpResponse response)` | Обработка HTTP-ошибок |

Высокоуровневые методы (`getForObject`, `postForEntity`, `exchange`) — это удобные обёртки над `execute`, которые сами создают нужные callback-объекты.

**AbstractController в Spring MVC**

Ещё один классический пример: `AbstractController` реализует Template Method через наследование (GoF-стиль):

```java
// Шаблонный метод — не переопределяется
public final ModelAndView handleRequest(HttpServletRequest request,
                                        HttpServletResponse response) {
    // проверка метода, кэш-заголовки, синхронизация сессии...
    return handleRequestInternal(request, response); // вызов крючка
}

// Крючок — обязателен к переопределению
protected abstract ModelAndView handleRequestInternal(
    HttpServletRequest request, HttpServletResponse response);
```

**Ключевое отличие вариантов реализации**

| Вариант | Spring-пример | Расширение |
|---|---|---|
| Template Method через callback | `JdbcTemplate`, `RestTemplate` | Передача lambda / анонимного класса |
| Template Method через наследование | `AbstractController`, `AbstractApplicationContext` | Создание подкласса |

Spring предпочитает callback-вариант, потому что он не требует создания подкласса ради одного шага алгоритма и хорошо сочетается с лямбдами Java 8+.
### Вопрос 16. Паттерн Observer в Java: чем java.util.Observable/Observer отличается от EventListener-подхода? Почему Observable признан устаревшим (deprecated) в Java 9?

**Категория:** Паттерны · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Observable (Java SE 9 & JDK 9) — docs.oracle.com](https://docs.oracle.com/javase/9/docs/api/java/util/Observable.html)

Официальный javadoc класса `Observable` содержит следующую аннотацию устаревания:

> "This class and the Observer interface have been deprecated. The event model supported by Observer and Observable is quite limited, the order of notifications delivered by Observable is unspecified, and state changes are not in one-for-one correspondence with notifications. For a richer event model, consider using the java.beans package. For reliable and ordered messaging among threads, consider using one of the concurrent data structures in the java.util.concurrent package. For reactive streams style programming, see the Flow API."

Класс помечен `@Deprecated(since="9")`. Oracle рекомендует три альтернативы в зависимости от задачи: пакет `java.beans` для расширенной модели событий, структуры `java.util.concurrent` для надёжного обмена сообщениями между потоками, и `Flow` API для реактивного программирования.

#### Ответ от Claude

**Архитектурные различия между Observable/Observer и EventListener**

`java.util.Observable` (класс) и `java.util.Observer` (интерфейс) образуют классическую реализацию паттерна Observer из JDK 1.0. Подход на основе `java.util.EventListener` (введён в JDK 1.1 вместе с Beans/AWT event model) принципиально отличается:

| Аспект | Observable/Observer | EventListener-подход |
|---|---|---|
| Тип Observable | Конкретный класс (нужно наследоваться) | Интерфейс — можно реализовать в любом классе |
| Типизация событий | Один метод `update(Observable o, Object arg)` — нет различия по типу события | Отдельный интерфейс-слушатель на каждый тип события (`ActionListener`, `MouseListener` и т.д.) |
| Информация о событии | Только объект-аргумент без типа | `EventObject` с источником и полной информацией |
| Сериализация | Observable не реализует `Serializable`, все поля приватные — подклассы не могут сериализовать состояние | Зависит от конкретной реализации |

**Причины признания Observable устаревшим в Java 9**

Официальная документация Oracle называет три технические причины:

1. **Ограниченная модель событий** — `Observable`/`Observer` не позволяют передать структурированную информацию о том, что именно изменилось. Метод `update(Observable o, Object arg)` принимает `Object`, что требует небезопасного приведения типов.

2. **Неопределённый порядок уведомлений** — порядок, в котором `Observable` уведомляет наблюдателей, не специфицирован и зависит от реализации.

3. **Несоответствие изменений и уведомлений** — изменения состояния не находятся в отношении "один к одному" с уведомлениями: можно накопить несколько изменений и разослать одно уведомление (через `setChanged()`), но семантика этого поведения не гарантирована.

**Дополнительные структурные проблемы дизайна:**

- `Observable` — это класс, а не интерфейс. Из-за отсутствия множественного наследования в Java класс, уже расширяющий другой класс, не может стать наблюдаемым без обёртки.
- Метод `setChanged()` объявлен `protected`, что исключает использование через композицию: нельзя создать экземпляр `Observable` и делегировать ему работу — только наследование.
- `Observable` не реализует `Serializable`. Все внутренние поля приватные, поэтому подклассы не могут корректно сериализовать Observer-состояние.
- Классы не развивались с JDK 1.0. Ещё в 1999 году Джош Блох (Josh Bloch) отметил, что они "largely unused in the JDK and has been superseded by the 1.1 Beans/AWT event model".

**Рекомендуемые современные альтернативы:**

- `java.beans.PropertyChangeListener` / `PropertyChangeSupport` — для event-driven уведомлений об изменениях свойств.
- `java.util.concurrent` (BlockingQueue, Flow.Publisher/Subscriber) — для потокобезопасного обмена сообщениями.
- `java.util.concurrent.Flow` (реактивные стримы, Reactive Streams API, Java 9+) — для асинхронного реактивного программирования.
- Собственная реализация паттерна Observer через обобщённые интерфейсы без наследования от `Observable`.

Важно: депрекация сделана **без** флага `forRemoval = true`, поэтому классы остаются в JDK, но их использование в новом коде не рекомендуется.

### Вопрос 17. Как паттерн Strategy применяется в Spring Security (например, AuthenticationProvider, PasswordEncoder)? Какие преимущества это даёт при расширении?

**Категория:** Паттерны · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Servlet Authentication Architecture — docs.spring.io](https://docs.spring.io/spring-security/reference/servlet/authentication/architecture.html)

Официальная документация Spring Security описывает `ProviderManager` как наиболее распространённую реализацию `AuthenticationManager`. Вместо самостоятельной обработки запроса на аутентификацию `ProviderManager` делегирует работу списку `AuthenticationProvider` — каждый из которых опрашивается по очереди. Каждый `AuthenticationProvider` отвечает за один конкретный тип аутентификации: например, `DaoAuthenticationProvider` — за аутентификацию по паре username/password, `JwtAuthenticationProvider` — за JWT-токены. Если ни один `AuthenticationProvider` не может обработать запрос, `ProviderManager` обращается к родительскому `AuthenticationManager`. По умолчанию после успешной аутентификации `ProviderManager` очищает чувствительные данные (пароли) из объекта `Authentication`.

#### Ответ от Claude

**Паттерн Strategy в Spring Security**

Паттерн Strategy предполагает определение семейства алгоритмов, инкапсуляцию каждого из них и обеспечение их взаимозаменяемости без изменения кода клиента. Spring Security применяет этот паттерн в нескольких ключевых местах.

**AuthenticationProvider — стратегия аутентификации**

Интерфейс `AuthenticationProvider` определяет контракт:

```java
public interface AuthenticationProvider {
    Authentication authenticate(Authentication authentication)
        throws AuthenticationException;
    boolean supports(Class<?> authentication);
}
```

`ProviderManager` выступает контекстом паттерна Strategy: он хранит список `AuthenticationProvider` и перебирает их, вызывая `supports()` для выбора подходящей стратегии. Встроенные реализации: `DaoAuthenticationProvider`, `LdapAuthenticationProvider`, `JaasAuthenticationProvider`, `RememberMeAuthenticationProvider`, `AnonymousAuthenticationProvider`, `PreAuthenticatedAuthenticationProvider`.

Для добавления нового механизма аутентификации (например, аутентификация по одноразовому коду, по сертификату, через внешний OAuth-провайдер) достаточно реализовать интерфейс `AuthenticationProvider` и зарегистрировать его в `ProviderManager` — существующий код не меняется.

**PasswordEncoder — стратегия кодирования паролей**

Интерфейс `PasswordEncoder` определяет:

```java
public interface PasswordEncoder {
    String encode(CharSequence rawPassword);
    boolean matches(CharSequence rawPassword, String encodedPassword);
    default boolean upgradeEncoding(String encodedPassword) { return false; }
}
```

Встроенные реализации: `BCryptPasswordEncoder`, `Argon2PasswordEncoder`, `Pbkdf2PasswordEncoder`, `SCryptPasswordEncoder`, `NoOpPasswordEncoder` (только для тестов).

Особого внимания заслуживает `DelegatingPasswordEncoder` — это составная стратегия, которая делегирует работу конкретному `PasswordEncoder` на основе префикса в закодированном пароле (`{bcrypt}...`, `{pbkdf2}...`). Это позволяет одновременно поддерживать старые пароли в legacy-формате и кодировать новые по современным стандартам — без пересохранения всей базы паролей.

**Преимущества паттерна Strategy в Spring Security**

1. **Принцип открытости/закрытости (OCP)**: фреймворк закрыт для модификации, но открыт для расширения. Новый провайдер аутентификации не требует изменений в `ProviderManager`.

2. **Независимая замена алгоритмов**: алгоритм хеширования паролей можно сменить (например, с `SHA-256` на `bcrypt`) только заменой бина `PasswordEncoder` — без изменений в слое аутентификации.

3. **Комбинирование стратегий**: `ProviderManager` может содержать несколько `AuthenticationProvider` одновременно, поддерживая разные механизмы входа (форма + SSO + API-ключ) в одном приложении.

4. **Тестируемость**: стратегии легко подменяются моками в тестах. Например, `NoOpPasswordEncoder` используется в тестовом окружении.

5. **Постепенная миграция**: `DelegatingPasswordEncoder` позволяет мигрировать алгоритм хеширования паролей без единовременной перезаписи всей базы — каждый пароль обновляется при следующем успешном входе.

### Вопрос 18. Можно ли реализовать паттерн Builder через Lombok @Builder? Какие ограничения у этой реализации по сравнению с ручной?

**Категория:** Паттерны · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [@Builder — projectlombok.org](https://projectlombok.org/features/Builder)

Официальная документация Lombok описывает `@Builder` как аннотацию, которая генерирует реализацию паттерна Builder для класса. Среди задокументированных ограничений:

- Поля с `@Singular` нельзя частично переопределять вручную: "You cannot manually provide some or all parts of a @Singular node; the code lombok generates is too complex for this."
- Отсортированные коллекции с `@Singular` требуют, чтобы тип элемента реализовывал `Comparable` — передать кастомный `Comparator` невозможно.
- Инициализаторы `@Builder.Default` не могут ссылаться на `this`, `super` или нестатические члены класса.
- Явно написанные конструкторы не будут устанавливать значения по умолчанию из `@Builder.Default` — это может привести к трудноуловимым багам.
- Все методы внутри сгенерированного builder-класса генерируются как `public`, даже если семантика требовала бы `protected`.

#### Ответ от Claude

**Да, паттерн Builder реализуется через `@Builder`**

Аннотация `@Builder` из Lombok генерирует полноценный builder: статический метод `builder()`, вложенный класс `XxxBuilder` с методами-сеттерами для каждого поля, и метод `build()`. Для коллекций дополнительная аннотация `@Singular` генерирует методы добавления одного элемента и очистки коллекции. Это покрывает большинство типовых сценариев с минимальным количеством кода.

**Ограничения `@Builder` по сравнению с ручной реализацией**

1. **Валидация невозможна без обходных путей**

   Ручной Builder позволяет вставить произвольную валидацию в метод `build()`:
   ```java
   public Person build() {
       if (age < 0 || age > 150) throw new IllegalArgumentException("Invalid age");
       if (name == null || name.isBlank()) throw new IllegalArgumentException("Name required");
       return new Person(this);
   }
   ```
   В `@Builder` единственный встроенный инструмент — `@NonNull` на поле (генерирует проверку на null). Для более сложной валидации необходимо переопределять метод `build()` вручную или использовать `@Builder.ObtainVia` — что нивелирует часть удобства аннотации.

2. **Обязательные поля без значения по умолчанию**

   В ручном Builder можно сделать некоторые поля обязательными на уровне API (например, через отдельный метод или проверку в `build()`). В `@Builder` все поля опциональны по умолчанию. Имитация обязательности через `@NonNull` добавляет только null-проверку, а не проверку факта вызова сеттера.

3. **Вычисляемые поля**

   Если одно поле вычисляется из другого (например, `fullName = firstName + " " + lastName`), ручной Builder решает это тривиально в `build()`. С `@Builder` требуется приватный конструктор с `@Builder(builderClassName = ..., buildMethodName = ...)` и дополнительная логика — код становится трудночитаемым.

4. **Проблемы с наследованием**

   `@Builder` плохо работает с иерархиями классов. Для builder'а подкласса необходимо явно задавать `@Builder` на каждом уровне иерархии и использовать `@SuperBuilder` (доступен начиная с Lombok 1.18.2). Ручная реализация позволяет использовать generics и self-referential типы (`Builder<T extends Builder<T>>`) для поддержки наследования builder'а.

5. **`@Builder.Default` и явные конструкторы**

   Если в классе существует явно написанный конструктор, значения из `@Builder.Default` не будут применены через этот конструктор — только через lombok-генерируемый. Это приводит к тому, что в некоторых сценариях поля получают `null`/`0` вместо ожидаемого дефолта.

6. **`@Singular` — отсутствие кастомного Comparator**

   Для `SortedSet` и `SortedMap` с `@Singular` элементы должны реализовывать `Comparable`. Передать пользовательский `Comparator` через Lombok невозможно — нужна ручная реализация.

7. **Прозрачность кода и отладка**

   Сгенерированный код невидим в исходниках. IDE-плагины показывают его, но stack trace при ошибке и code review становятся сложнее. Ручной Builder полностью прозрачен.

**Итог**

`@Builder` отлично подходит для простых value-объектов (DTO, конфигурации) без сложной валидации и наследования. Для доменных объектов с инвариантами, обязательными полями и вычисляемыми свойствами ручная реализация Builder предпочтительнее, так как даёт полный контроль над процессом конструирования.

### Вопрос 19. Как паттерн Command используется в Spring Batch (Step, Tasklet)? Что даёт оборачивание логики в объект-команду по сравнению с прямым вызовом метода?

**Категория:** Паттерны · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [TaskletStep :: Spring Batch Reference](https://docs.spring.io/spring-batch/reference/step/tasklet.html)

Интерфейс `Tasklet` содержит единственный метод `execute(StepContribution, ChunkContext)`, который вызывается `TaskletStep` снова и снова до тех пор, пока не будет возвращён `RepeatStatus.FINISHED` или не будет выброшено исключение. Каждый вызов оборачивается в транзакцию. Для случаев, когда нужно адаптировать уже существующий класс к интерфейсу `Tasklet` без его модификации, Spring Batch предоставляет `MethodInvokingTaskletAdapter` — адаптер, позволяющий передать целевой объект и имя вызываемого метода.

Пример конфигурации:

```java
@Bean
public MethodInvokingTaskletAdapter myTasklet() {
    MethodInvokingTaskletAdapter adapter = new MethodInvokingTaskletAdapter();
    adapter.setTargetObject(fooDao());
    adapter.setTargetMethod("updateFoo");
    return adapter;
}
```

Таким образом, `TaskletStep` выступает в роли **Invoker**: он знает только об интерфейсе `Tasklet`, не зависит от конкретной реализации бизнес-логики.

#### Ответ от Claude

**Паттерн Command в Spring Batch**

Паттерн Command предполагает инкапсуляцию запроса (операции) в виде отдельного объекта с единым методом вызова. В Spring Batch эта идея реализована через три компонента:

| Роль в паттерне | Реализация в Spring Batch |
|---|---|
| Command (команда) | Интерфейс `Tasklet` с методом `execute()` |
| ConcreteCommand | Конкретная реализация `Tasklet` (например, `SystemCommandTasklet`, пользовательский бин) |
| Invoker | `TaskletStep` — вызывает `execute()` в цикле внутри транзакции |
| Client | `Job`-конфигурация, собирающая шаги |

**Как это работает**

`TaskletStep` вызывает `tasklet.execute(contribution, chunkContext)` в транзакционном цикле:

```java
@Bean
public Step cleanupStep(JobRepository jobRepository,
                        PlatformTransactionManager transactionManager) {
    return new StepBuilder("cleanupStep", jobRepository)
        .tasklet(cleanupTasklet(), transactionManager)
        .build();
}

@Bean
public Tasklet cleanupTasklet() {
    return (contribution, chunkContext) -> {
        // любая произвольная логика
        return RepeatStatus.FINISHED;
    };
}
```

**Что даёт оборачивание в объект-команду**

1. **Декаплинг вызывающего кода от реализации.** `TaskletStep` не знает, что именно делает `Tasklet` — вызывает хранимую процедуру, архивирует файлы или выполняет системную команду. Это позволяет менять реализацию без изменения инфраструктурного кода.

2. **Автоматическое транзакционное управление.** Каждый вызов `execute()` оборачивается в транзакцию фреймворком — при прямом вызове метода транзакцию нужно было бы организовывать вручную.

3. **Единая точка управления жизненным циклом.** Фреймворк контролирует повторные вызовы, обработку исключений, обновление `StepExecution`, запись метрик — вызывающий код получает всё это «бесплатно».

4. **Поддержка повторных попыток и пропуска (retry/skip).** Интеграция с инфраструктурой Spring Batch (`StepContribution`) позволяет накапливать статистику по шагам.

5. **Адаптируемость через `MethodInvokingTaskletAdapter`.** Существующий сервисный метод можно «обернуть» в `Tasklet` без модификации его класса — классический приём паттерна Adapter поверх Command.

6. **Очередность и параллелизм.** Объекты-команды можно ставить в очередь, выполнять параллельно в разных `Step`, переиспользовать в разных `Job`.

**Сравнение: прямой вызов vs объект-команда**

```java
// Прямой вызов — жёсткая связь, нет управления транзакцией, нет метрик
public void runStep() {
    cleanupService.deleteOldRecords();
}

// Объект-команда — фреймворк управляет транзакцией, метриками, повторами
@Bean
public Tasklet deleteOldRecordsTasklet() {
    return (contribution, chunkContext) -> {
        cleanupService.deleteOldRecords();
        return RepeatStatus.FINISHED;
    };
}
```

При прямом вызове вся ответственность за транзакцию, обработку ошибок, ведение журнала и сигнализацию о завершении ложится на вызывающий код. Объект-команда делегирует эти обязанности фреймворку.

### Вопрос 20. Как в Spring реализован паттерн Proxy для AOP (@Transactional, @Cacheable)? Почему self-invocation (вызов метода внутри того же бина) не перехватывается Spring AOP-прокси?

**Категория:** Паттерны · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Proxying Mechanisms :: Spring Framework Reference](https://docs.spring.io/spring-framework/reference/core/aop/proxying.html)

Spring AOP использует два механизма создания прокси:

- **JDK Dynamic Proxy** — применяется, если целевой объект реализует хотя бы один интерфейс. Прокси реализует те же интерфейсы и перехватывает вызовы через `InvocationHandler`.
- **CGLIB Proxy** — применяется, если целевой объект не реализует интерфейсов. Создаёт подкласс целевого класса во время выполнения.

Проблема self-invocation описана в документации так: когда проксируемый объект вызывает метод на себе через `this`, вызов идёт напрямую к реальному объекту, минуя прокси, и никакой advice не применяется.

```java
public class SimplePojo implements Pojo {
    public void foo() {
        this.bar(); // вызов идёт напрямую, минуя прокси — advice не сработает
    }
    public void bar() { /* ... */ }
}
```

Визуально: `Клиент → Прокси → foo() → this.bar()` — прокси не перехватывает `bar()`.

Рекомендованное решение из документации — рефакторинг кода для устранения self-invocation или использование AspectJ, который применяет advice на уровне байт-кода, а не через прокси.

#### Ответ от Claude

**Как Spring реализует паттерн Proxy для AOP**

Паттерн Proxy в Spring AOP работает следующим образом: при старте контекста Spring создаёт прокси-объект вместо оригинального бина. Клиентский код получает ссылку на прокси, а не на реальный объект. Когда клиент вызывает метод, прокси перехватывает вызов, выполняет необходимую дополнительную логику (открытие транзакции, проверку кэша) и делегирует вызов реальному объекту.

**Два типа прокси**

| Тип прокси | Когда используется | Механизм |
|---|---|---|
| JDK Dynamic Proxy | Целевой класс реализует интерфейс | `java.lang.reflect.Proxy` + `InvocationHandler` |
| CGLIB Proxy | Целевой класс не реализует интерфейс (или `proxyTargetClass=true`) | Генерация подкласса во время выполнения |

Начиная со Spring Boot 2.x, CGLIB является прокси-механизмом по умолчанию даже для классов с интерфейсами (если явно не задано иное).

**Пример для @Transactional**

```java
// Что разработчик пишет:
@Service
public class OrderService {
    @Transactional
    public void createOrder(Order order) { /* ... */ }
}

// Что Spring создаёт в контексте (упрощённо):
// OrderService$$SpringCGLIB$$0 extends OrderService {
//     public void createOrder(Order order) {
//         transactionManager.begin();
//         try {
//             super.createOrder(order); // вызов реального метода
//             transactionManager.commit();
//         } catch (Exception e) {
//             transactionManager.rollback();
//             throw e;
//         }
//     }
// }
```

**Почему self-invocation не перехватывается**

Ключевая причина: ссылка `this` внутри класса указывает на **реальный объект**, а не на прокси. Прокси существует снаружи и перехватывает только внешние вызовы.

```
Внешний клиент → [Прокси] → реальный объект
                     ↑
              advice применяется здесь

реальный объект → this.method() → реальный объект
                                      ↑
                          прокси обходится стороной,
                          advice НЕ применяется
```

```java
@Service
public class PaymentService {

    @Transactional
    public void processPayment() {
        // ...
        validatePayment(); // ПРОБЛЕМА: вызов через this, транзакция @Transactional на
                           // validatePayment не откроется отдельно
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void validatePayment() {
        // Эта транзакция НЕ откроется — прокси не перехватил вызов
    }
}
```

**Способы решения**

1. **Self-injection** — внедрить сам бин через `@Autowired` и вызывать методы через него:
   ```java
   @Autowired
   private PaymentService self;

   public void processPayment() {
       self.validatePayment(); // идёт через прокси
   }
   ```

2. **Вынести метод в другой бин** — наиболее чистое решение с точки зрения архитектуры.

3. **`AopContext.currentProxy()`** — получить текущий прокси явно (не рекомендуется, сильная связь с фреймворком):
   ```java
   ((PaymentService) AopContext.currentProxy()).validatePayment();
   ```
   Требует `@EnableAspectJAutoProxy(exposeProxy = true)`.

4. **AspectJ weaving** — compile-time или load-time weaving встраивает advice прямо в байт-код, поэтому self-invocation перехватывается. Прокси не нужен.

**Ограничения CGLIB-прокси**

- `final`-классы проксировать нельзя (нельзя создать подкласс).
- `final`-методы не перехватываются (нельзя переопределить в подклассе).
- `private`-методы не перехватываются.
