### Вопрос 21. Почему в реализации Singleton с двойной проверкой блокировки (double-checked locking) поле instance обязательно должно быть объявлено как volatile? Что произойдёт без volatile в Java до версии 5 и после?

**Категория:** Паттерны · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [The "Double-Checked Locking is Broken" Declaration](https://www.cs.umd.edu/~pugh/java/memoryModel/DoubleCheckedLocking.html)

Авторы (в том числе Билл Пью) объясняют: запись, инициализирующая объект `Helper`, и запись в поле `helper` могут быть переупорядочены компилятором или процессором. В результате поток, вызывающий `getHelper()`, может увидеть ненулевую ссылку на объект `Helper`, но при этом прочитать значения полей этого объекта по умолчанию (до того, как конструктор их проинициализировал).

Начиная с JDK 5, семантика `volatile` была расширена в соответствии с JSR-133: запись в `volatile`-поле теперь не может быть переупорядочена относительно любых предшествующих операций чтения и записи, а чтение из `volatile`-поля не может быть переупорядочено относительно любых последующих операций. Объявление поля `volatile` создаёт необходимые happens-before отношения и делает double-checked locking безопасным — но только в Java 5 и выше.

Авторы, тем не менее, рекомендуют вместо этого паттерна использовать идиому «Initialization On Demand Holder» (статический вложенный класс), которая опирается на гарантии загрузчика классов.

#### Ответ от Claude

**Double-Checked Locking: полный разбор**

**Что такое DCL и зачем он нужен**

Double-Checked Locking (DCL) — попытка получить ленивую инициализацию Singleton с минимальными накладными расходами синхронизации. Идея: первая проверка `null` без блокировки для быстрого пути (уже создан), вторая проверка внутри `synchronized`-блока для безопасного создания.

```java
public class Singleton {
    private static volatile Singleton instance; // volatile ОБЯЗАТЕЛЕН

    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {                    // 1-я проверка (без блокировки)
            synchronized (Singleton.class) {
                if (instance == null) {            // 2-я проверка (с блокировкой)
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

**Почему нужен volatile: проблема переупорядочивания инструкций**

Выражение `instance = new Singleton()` компилируется не в одну атомарную операцию, а приблизительно в три:

```
1. memory = allocate(Singleton.class)  // выделить память
2. invokeConstructor(memory)            // проинициализировать объект
3. instance = memory                   // присвоить ссылку полю
```

JVM (и процессор) имеют право переупорядочивать операции 2 и 3 в отсутствие барьеров памяти, поскольку в рамках одного потока наблюдаемый результат не меняется:

```
1. memory = allocate(Singleton.class)
3. instance = memory                   // сначала присваивание (объект ещё не готов!)
2. invokeConstructor(memory)            // потом инициализация
```

**Гонка данных без volatile (Java 4 и ранее)**

```
Поток A                          Поток B
---------                        ---------
if (instance == null) {
  synchronized(...) {
    if (instance == null) {
      memory = allocate(...)
      instance = memory           // instance != null, но объект НЕ готов
                                           if (instance == null)  // false — выходит
                                           return instance        // возвращает СЛОМАННЫЙ объект!
      invokeConstructor(memory)
    }
  }
}
```

Поток B видит `instance != null`, выходит из первого `if` и возвращает частично инициализированный объект. Работа с таким объектом — неопределённое поведение (NPE, неверные значения полей).

**Ситуация до Java 5**

До Java 5 модель памяти Java (JMM) вообще не давала никаких гарантий против такого переупорядочивания. Более того, даже объявление поля `volatile` не помогало, так как старая спецификация `volatile` запрещала переупорядочивание только между двумя `volatile`-операциями, но не между `volatile` и обычными операциями. Именно поэтому DCL был официально признан «сломанным» до Java 5.

**Что изменилось в Java 5 (JSR-133)**

JSR-133 ввёл новую Java Memory Model с усиленной семантикой `volatile`:

- **Запись в `volatile`-поле** имеет тот же эффект на память, что и **освобождение монитора (monitor release)**.
- **Чтение `volatile`-поля** имеет тот же эффект, что и **захват монитора (monitor acquire)**.
- Запись в `volatile` не может быть переупорядочена относительно ЛЮБЫХ предшествующих операций чтения/записи в том же потоке.

Это означает: если поток A записывает `instance` (volatile), гарантируется, что всё, что поток A делал до этой записи (включая вызов конструктора), будет видно потоку B до того, как поток B прочитает `instance`.

**Ситуация после Java 5 без volatile**

Формально: хотя современные JVM на x86-архитектуре редко реализуют переупорядочивание store-store (запись перед конструктором), спецификация JMM это не запрещает. На архитектурах с более слабой моделью памяти (ARM, POWER) проблема воспроизводится реально. Без `volatile` поведение остаётся **неопределённым по спецификации** вне зависимости от версии JVM.

**Альтернативы DCL**

```java
// 1. Initialization On Demand Holder (рекомендуется)
public class Singleton {
    private static class Holder {
        static final Singleton INSTANCE = new Singleton();
    }
    public static Singleton getInstance() {
        return Holder.INSTANCE; // безопасно благодаря гарантиям загрузчика классов
    }
}

// 2. Enum Singleton (наиболее надёжный способ)
public enum Singleton {
    INSTANCE;
}
```

**Итог**

| Ситуация | Поведение |
|---|---|
| Без volatile, Java <= 4 | Сломан: переупорядочивание не запрещено ни JMM, ни volatile |
| Без volatile, Java >= 5 | Сломан по спецификации: UB, на ARM/POWER воспроизводится |
| С volatile, Java >= 5 | Безопасен: JSR-133 гарантирует happens-before через volatile-запись |

### Вопрос 22. Чем Singleton в Spring-контексте отличается от GoF-Singleton? Может ли в одном JVM-процессе существовать несколько экземпляров одного и того же @Component с областью видимости singleton?

**Категория:** Паттерны · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Bean Scopes :: Spring Framework — docs.spring.io](https://docs.spring.io/spring-framework/reference/core/beans/factory-scopes.html)

Официальная документация Spring явно разграничивает эти понятия:

> «Концепция singleton-бина в Spring отличается от паттерна Singleton, определённого в книге Gang of Four (GoF). GoF Singleton жёстко фиксирует область видимости объекта: создаётся ровно один экземпляр данного класса на ClassLoader. Область видимости Spring-синглтона лучше всего описывается как per-container и per-bean.»

Это означает: если вы объявляете один бин для конкретного класса в одном Spring-контейнере, контейнер создаёт ровно один экземпляр этого класса согласно данному определению бина.

При этом Spring управляет scope через конфигурацию (аннотацию или XML), а не через жёсткую кодировку в самом классе, — scope задаётся снаружи и при желании легко меняется.

#### Ответ от Claude

**GoF Singleton vs Spring Singleton — принципиальные отличия:**

| Критерий | GoF Singleton | Spring Singleton |
|---|---|---|
| Область гарантии | Один экземпляр на ClassLoader | Один экземпляр на bean-определение на контейнер |
| Где фиксируется | В коде самого класса (private constructor + static field) | В конфигурации контейнера снаружи класса |
| Кто контролирует | Сам класс | IoC-контейнер |
| Изменяемость scope | Невозможна без изменения кода | Меняется через аннотацию или XML |

**Может ли в одном JVM существовать несколько экземпляров @Component(scope=singleton)?**

Да, и это принципиально важно. Возможны сразу несколько сценариев:

1. **Несколько ApplicationContext в одном JVM.** Например, в веб-приложении может быть родительский контекст (DispatcherServlet-контекст) и дочерний (root WebApplicationContext). Каждый контекст ведёт собственный кеш singleton-бинов — один и тот же класс получает по одному экземпляру в каждом контексте. В тестах `@SpringBootTest` зачастую создаётся отдельный тестовый контекст — ещё один экземпляр.

2. **Несколько bean-определений одного и того же класса в одном контексте.** Если объявить два бина с разными id/именами, основанных на одном классе, Spring создаст два отдельных экземпляра. Singleton-ность гарантируется только в пределах одного bean-id:

```java
@Bean("serviceA")
public MyService serviceA() { return new MyService(); }

@Bean("serviceB")
public MyService serviceB() { return new MyService(); }
// В контексте — два разных экземпляра MyService
```

3. **Другой ClassLoader.** Если один класс загружен разными ClassLoader-ами (например, в OSGi-окружении или при горячей замене), даже GoF Singleton не защищает от нескольких экземпляров.

**Итог:** GoF Singleton — это самозащита класса (один экземпляр на ClassLoader, жёсткая гарантия). Spring Singleton — это декларативный контракт контейнера (один экземпляр на bean-id на ApplicationContext). Сам класс, помеченный `@Component`, не является GoF Singleton — ничто не мешает создать его через `new MyComponent()` за пределами контейнера.

### Вопрос 23. Почему реализация Singleton через enum (Effective Java, Item 3) считается наиболее надёжной? Какие проблемы классических реализаций она решает — в том числе связанные с сериализацией и рефлексией?

**Категория:** Паттерны · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Java Singletons Using Enum — dzone.com](https://dzone.com/articles/java-singletons-using-enum)

Enum-синглтон решает две критические уязвимости классических реализаций:

**Сериализация.** При десериализации обычного класса стандартный механизм Java создаёт новый объект в обход конструктора. Добавление `readResolve()` помогает, но требует ручного кода и является источником ошибок. Enum-значения сериализуются автоматически и гарантированно, при десериализации JVM возвращает существующий экземпляр `INSTANCE` — новый объект не создаётся.

**Рефлексия.** С помощью `Constructor.setAccessible(true)` злоумышленник может вызвать приватный конструктор любого класса и получить второй экземпляр. Конструктор enum не доступен через рефлексию — попытка вызвать его выбрасывает `IllegalArgumentException`. Это железная гарантия на уровне JVM.

Итоговый вывод статьи: «Гарантируется 100%, что только один экземпляр singleton присутствует в JVM. Этот метод рекомендован как наилучший способ создания синглтонов в Java».

#### Ответ от Claude

**Классические реализации и их проблемы:**

*1. Public final field / статический инициализатор:*
```java
public class Elvis {
    public static final Elvis INSTANCE = new Elvis();
    private Elvis() {}
}
```
- Уязвим к рефлексии: `Constructor<Elvis> c = Elvis.class.getDeclaredConstructor(); c.setAccessible(true); Elvis second = c.newInstance();` — получаем второй экземпляр.
- Уязвим к сериализации: реализация `Serializable` требует `readResolve()`, иначе каждый `ObjectInputStream.readObject()` создаёт новый экземпляр.

*2. Lazy initialization с double-checked locking:*
```java
private static volatile Elvis instance;
public static Elvis getInstance() {
    if (instance == null) { synchronized(Elvis.class) { if (instance == null) instance = new Elvis(); } }
    return instance;
}
```
- Те же проблемы с рефлексией и сериализацией.
- Дополнительная сложность: без `volatile` возможна публикация частично сконструированного объекта.

**Enum-реализация (Effective Java, Item 3):**
```java
public enum Elvis {
    INSTANCE;
    public void leaveTheBuilding() { ... }
}
```

**Почему это надёжнее:**

1. **Защита от рефлексии.** Спецификация JVM запрещает создание экземпляра enum через рефлексию. `Method.invoke()` и `Constructor.newInstance()` для enum выбрасывают `IllegalArgumentException: Cannot reflectively create enum objects`. Это гарантия на уровне JVM, а не только соглашение.

2. **Защита от сериализации.** Enum по спецификации Java сериализуется по имени (`name()`) и десериализуется через `Enum.valueOf()`, который возвращает уже существующий экземпляр. Метод `readObject()` для enum не вызывается вовсе — JVM перехватывает этот путь. Никакой `readResolve()` не нужен.

3. **Защита от клонирования.** `Enum` расширяет `java.lang.Enum`, который объявляет `clone()` как `final`, выбрасывающий `CloneNotSupportedException`. Случайное клонирование невозможно.

4. **Потокобезопасность.** Инициализация enum-констант JVM гарантирует корректную публикацию — это часть загрузки класса, которая потокобезопасна по определению.

5. **Краткость.** Весь код — одна строка объявления константы.

**Ограничение, которое надо знать:** enum не может расширять другой класс (он уже неявно расширяет `java.lang.Enum`). Если singleton должен наследовать бизнес-класс — enum не подходит. Реализация интерфейсов разрешена.

**Позиция Джошуа Блоха (Effective Java):** «Одноэлементный enum-тип — наилучший способ реализовать singleton». Эта рекомендация остаётся актуальной для Java 8+ и более поздних версий.

### Вопрос 24. В чём принципиальная разница между Factory Method и Abstract Factory? Когда добавление нового продукта требует изменения только одного класса, а когда — целой иерархии?

**Категория:** Паттерны · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Abstract Factory — refactoring.guru](https://refactoring.guru/design-patterns/abstract-factory)

Паттерны различаются масштабом задачи:

**Factory Method** создаёт один тип продукта. Клиентский код работает с фабричным методом через переопределение в подклассе — выбор продукта задаётся наследованием.

**Abstract Factory** управляет семействами взаимосвязанных продуктов. Каждая конкретная фабрика реализует полный набор методов для создания всех продуктов семейства (например, `createChair()`, `createSofa()`, `createCoffeeTable()`), гарантируя их совместимость внутри семейства.

Расширяемость у них асимметрична:
- Добавить **новое семейство** (вариант) в Abstract Factory легко: создаётся новая конкретная фабрика без изменений существующего кода — это расширение по вертикали.
- Добавить **новый тип продукта** (горизонтально) в Abstract Factory тяжело: нужно изменить интерфейс абстрактной фабрики и все её конкретные реализации. Это нарушает принцип открытости/закрытости.

Дизайны обычно начинают с Factory Method и эволюционируют в сторону Abstract Factory по мере усложнения требований.

#### Ответ от Claude

**Factory Method**

*Структура:* Базовый класс (Creator) объявляет абстрактный (или виртуальный) фабричный метод, возвращающий продукт через интерфейс. Конкретные подклассы переопределяют этот метод и решают, какой именно продукт создать.

```java
abstract class Dialog {
    // Фабричный метод
    abstract Button createButton();

    void render() {
        Button btn = createButton(); // использует продукт
        btn.render();
    }
}

class WindowsDialog extends Dialog {
    @Override Button createButton() { return new WindowsButton(); }
}

class WebDialog extends Dialog {
    @Override Button createButton() { return new HtmlButton(); }
}
```

*Ключевые характеристики:*
- Один фабричный метод — один тип продукта.
- Расширение через наследование (новый подкласс Creator).
- Клиент работает с Creator-ом, не зная о конкретном продукте.

**Abstract Factory**

*Структура:* Интерфейс фабрики объявляет методы создания нескольких взаимосвязанных продуктов. Конкретные фабрики реализуют весь набор — это гарантирует совместимость продуктов внутри семейства.

```java
interface GUIFactory {
    Button createButton();
    Checkbox createCheckbox();
}

class MacFactory implements GUIFactory {
    public Button createButton()   { return new MacButton(); }
    public Checkbox createCheckbox() { return new MacCheckbox(); }
}

class WinFactory implements GUIFactory {
    public Button createButton()   { return new WinButton(); }
    public Checkbox createCheckbox() { return new WinCheckbox(); }
}
```

**Когда изменение затрагивает один класс vs целую иерархию:**

| Операция | Factory Method | Abstract Factory |
|---|---|---|
| Добавить новый **вариант** существующего продукта | Новый подкласс Creator | Новый класс ConcreteFactory (только один новый класс) |
| Добавить новый **тип** продукта | Изменить один Creator + добавить новый продукт | Изменить интерфейс AbstractFactory + ВСЕ ConcreteFactory (нарушение OCP) |

**Практический ответ на вопрос:**

- **Только один класс** нужно менять, когда в Abstract Factory добавляется новое семейство (новая платформа, новый стиль): достаточно создать одну новую ConcreteFactory. Остальной код не трогается.

- **Целую иерархию** нужно менять, когда в Abstract Factory добавляется новый тип продукта (например, к Chair и Sofa добавляется Table): необходимо добавить метод `createTable()` в интерфейс AbstractFactory и реализовать его во всех существующих ConcreteFactory. Это неизбежная цена за гарантию совместимости внутри семейства.

В Factory Method добавление нового продукта другого типа вообще не вписывается в паттерн — потребуется добавить новый отдельный фабричный метод или перейти к Abstract Factory.

**Когда выбирать что:**
- Factory Method — когда нужно делегировать создание одного объекта подклассам, а семейства продуктов не предполагаются.
- Abstract Factory — когда система должна работать с семействами взаимосвязанных объектов и важно гарантировать их совместимость (UI-компоненты одной платформы, драйверы одного вендора, DAO-слой одной СУБД).
### Вопрос 25. Чем FactoryBean<T> в Spring отличается от BeanFactory? Как Spring определяет, когда возвращать сам FactoryBean, а когда — объект, который он создаёт (символ & в getBean)?

**Категория:** Паттерны · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [FactoryBean (Spring Framework 7.0.7 API)](https://docs.spring.io/spring-framework/docs/current/javadoc-api/org/springframework/beans/factory/FactoryBean.html)

Интерфейс `FactoryBean<T>` — это точка расширения для логики создания объектов в IoC-контейнере Spring. Бин, реализующий `FactoryBean`, используется как фабрика: сам контейнер управляет жизненным циклом `FactoryBean`, но не управляет жизненным циклом объектов, которые он производит. Интерфейс содержит три метода:
- `T getObject()` — возвращает создаваемый объект (может быть `null`);
- `Class<?> getObjectType()` — возвращает тип создаваемого объекта (используется для autowiring без создания экземпляра);
- `boolean isSingleton()` — по умолчанию `true`; возвращает `false` для прототипов.

Когда вы вызываете `getBean("myBean")`, контейнер возвращает результат `getObject()`, а не сам `FactoryBean`. Чтобы получить сам экземпляр фабрики, нужно использовать префикс `&`: `getBean("&myBean")`. Эта логика завязана на константу `BeanFactory.FACTORY_BEAN_PREFIX = "&"`, введённую в Spring 6.2.6 также как символьный вариант `FACTORY_BEAN_PREFIX_CHAR`. В Spring насчитывается более 80 реализаций `FactoryBean` (например, `ProxyFactoryBean`, `JndiObjectFactoryBean`, `LocalEntityManagerFactoryBean`).

#### Ответ от Claude

**BeanFactory** — это корневой интерфейс IoC-контейнера Spring. Он отвечает за хранение, разрешение зависимостей и управление жизненным циклом всех бинов. `ApplicationContext` является расширением `BeanFactory`.

**FactoryBean\<T\>** — это специальный интерфейс, который реализует сам бин, когда ему нужна сложная логика инициализации. Такой бин регистрируется в контейнере как обычный бин, но вместо своего экземпляра отдаёт продукт метода `getObject()`.

Различие в одной фразе: `BeanFactory` — это контейнер; `FactoryBean` — это бин-фабрика внутри контейнера.

**Как работает символ `&` в `getBean`:**

Spring обрабатывает имя при вызове `getBean` следующим образом:
- Если имя начинается с `&` (значение константы `BeanFactory.FACTORY_BEAN_PREFIX`), Spring отрезает префикс и возвращает сам экземпляр `FactoryBean` — не вызывая `getObject()`.
- Если имя не содержит `&`, Spring находит бин. Если бин реализует `FactoryBean`, вызывается `getObject()` и возвращается его результат.

```java
// myBean зарегистрирован как FactoryBean<DataSource>

// Вернёт DataSource (результат getObject())
DataSource ds = ctx.getBean("myBean", DataSource.class);

// Вернёт сам FactoryBean
FactoryBean<?> factory = ctx.getBean("&myBean", FactoryBean.class);
```

**Ключевые нюансы для Senior-уровня:**

1. Метод `getObjectType()` вызывается контейнером ещё до создания объекта — это позволяет autowiring по типу без преждевременной инициализации. Возврат `null` из `getObjectType()` исключает `FactoryBean` из autowiring.

2. `isSingleton()` возвращает `true` по умолчанию — результат `getObject()` кэшируется контейнером. При `false` каждый вызов `getBean` создаёт новый объект.

3. Контейнер управляет жизненным циклом самого `FactoryBean`, но **не** объектов, которые он создаёт. Если нужна очистка порождённых объектов, `FactoryBean` должен реализовывать `DisposableBean`.

4. Начиная с Spring 5, вместо `FactoryBean` в большинстве случаев предпочтительнее использовать `@Bean`-методы — они прозрачнее и не требуют знания соглашения о `&`.

### Вопрос 26. Builder-паттерн и Lombok @Builder: какие ограничения Lombok накладывает на наследование и валидацию? Как реализовать Builder с обязательными полями без Lombok?

**Категория:** Паттерны · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [@Builder — Project Lombok](https://projectlombok.org/features/Builder)

Аннотация `@Builder` генерирует статический внутренний класс `FooBuilder` с fluent-методами для каждого поля. Аннотация не поддерживает механизм обязательных полей: если поле не установлено в процессе сборки, оно получит значение `0 / null / false`. Никакого встроенного механизма принудительной проверки обязательных параметров нет — разработчик должен самостоятельно добавить валидацию в метод `build()`. Интеграция с фреймворками валидации отсутствует. `@Builder` неявно генерирует пакетный конструктор (как если бы на классе стоял `@AllArgsConstructor(access = AccessLevel.PACKAGE)`), что создаёт проблемы при наследовании.

Для решения проблем с наследованием в Lombok 1.18 введена экспериментальная аннотация `@SuperBuilder`. Она генерирует два абстрактных внутренних класса-строителя на класс и умеет включать поля суперклассов. Обязательное условие: все суперклассы в иерархии должны быть также аннотированы `@SuperBuilder`. Смешивать `@Builder` и `@SuperBuilder` в одной иерархии нельзя.

#### Ответ от Claude

**Ограничения `@Builder` при наследовании:**

`@Builder` генерирует конструктор только для полей текущего класса. Поля суперкласса игнорируются. При попытке скомпилировать подкласс с `@Builder`, унаследованный от класса, не помеченного `@Builder`, возникает ошибка компиляции, потому что Lombok не может вызвать `super(...)` с корректным набором аргументов.

Решение — аннотация `@SuperBuilder` (Lombok >= 1.18):
- требует, чтобы весь стек суперклассов использовал `@SuperBuilder`;
- несовместима с `@Builder` в одной иерархии;
- при `toBuilder = true` все суперклассы тоже обязаны иметь `toBuilder = true`.

**Ограничения по валидации:**

Lombok не интегрирован ни с Bean Validation (JSR 380), ни с другими фреймворками. Единственный доступный инструмент — `@NonNull` на поле, что добавляет null-проверку при вызове соответствующего setter-метода билдера, но не при вызове `build()`. Кастомная валидация должна быть написана вручную в методе `build()`, что требует явного переопределения его через `@Builder(buildMethodName = "innerBuild")` + обёртку.

**Реализация Builder с обязательными полями без Lombok:**

Классический подход — передача обязательных полей через конструктор самого `Builder`:

```java
public final class Order {
    private final String orderId;   // обязательное
    private final String product;   // обязательное
    private final int quantity;     // необязательное, по умолчанию 1

    private Order(Builder builder) {
        this.orderId = builder.orderId;
        this.product = builder.product;
        this.quantity = builder.quantity;
    }

    public static Builder builder(String orderId, String product) {
        return new Builder(orderId, product);
    }

    public static final class Builder {
        private final String orderId;
        private final String product;
        private int quantity = 1;

        private Builder(String orderId, String product) {
            if (orderId == null || orderId.isBlank())
                throw new IllegalArgumentException("orderId is required");
            if (product == null || product.isBlank())
                throw new IllegalArgumentException("product is required");
            this.orderId = orderId;
            this.product = product;
        }

        public Builder quantity(int quantity) {
            this.quantity = quantity;
            return this;
        }

        public Order build() {
            return new Order(this);
        }
    }
}
```

Обязательные поля передаются через фабричный метод `builder(orderId, product)` — их невозможно забыть на этапе компиляции.

**Альтернатива — Step Builder (staged builder):**

Обеспечивает компиляционный контроль порядка задания обязательных полей через цепочку интерфейсов. Каждый шаг возвращает следующий интерфейс, а `build()` доступен только после прохождения всех обязательных шагов. Это наиболее строгая форма проверки, но требует значительного количества бойлерплейта.

### Вопрос 27. Prototype-паттерн: в чём разница между поверхностным (shallow) и глубоким (deep) копированием? Почему интерфейс Cloneable считается сломанным в Java, и какие альтернативы предпочтительнее?

**Категория:** Паттерны · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Object (Java Platform SE 8) — Oracle Documentation](https://docs.oracle.com/javase/8/docs/api/java/lang/Object.html)

Метод `Object.clone()` создаёт и возвращает копию объекта. Официальная документация описывает его как "shallow copy" — поверхностное копирование: создаётся новый экземпляр того же класса, и все поля копируются присваиванием. Для примитивных типов это означает полную независимость копии. Для ссылочных типов копия и оригинал указывают на одни и те же вложенные объекты, что создаёт общее изменяемое состояние. Для достижения глубокого копирования документация рекомендует вручную скопировать изменяемые вложенные объекты после вызова `super.clone()`. Интерфейс `Cloneable` является маркерным — он не объявляет метода `clone()`. Класс, не реализующий `Cloneable`, получит `CloneNotSupportedException` при вызове `clone()`. Все массивы считаются реализующими `Cloneable` автоматически.

#### Ответ от Claude

**Shallow copy (поверхностное копирование):**

Создаётся новый объект; все поля копируются по значению. Для примитивов и иммутабельных типов (например, `String`, `Integer`) это даёт полную независимость. Для ссылочных полей на изменяемые объекты копия и оригинал разделяют одни и те же экземпляры — изменение вложенного объекта через копию затронет оригинал.

```java
// shallow: addresses в обоих объектах указывают на один и тот же List
Person clone = (Person) original.clone();
clone.getAddresses().add(new Address("Moscow")); // изменит и original!
```

**Deep copy (глубокое копирование):**

Рекурсивно копируются все ссылочные поля. Копия полностью независима от оригинала.

**Почему `Cloneable` считается сломанным:**

Джошуа Блох в книге «Effective Java» и в интервью называет дизайн `Cloneable` глубоко порочным. Основные причины:

1. **Маркерный интерфейс без метода.** `Cloneable` не объявляет `clone()`. Нельзя выполнить полиморфное клонирование через `Cloneable`-ссылку: приведение к `Cloneable` и вызов `.clone()` не скомпилируется.

2. **Метод `clone()` объявлен `protected` в `Object`.** Для публичного доступа каждый класс вынужден явно переопределять его с `public`.

3. **Объект создаётся без вызова конструктора.** `Object.clone()` обходит конструктор через JVM-механизм, что может нарушить инварианты класса, установленные в конструкторе.

4. **Несовместимость с `final`-полями.** Если ссылочное поле объявлено `final`, после `super.clone()` его нельзя перезаписать новым значением для глубокого копирования.

5. **Хрупкий контракт.** Корректная реализация требует соблюдения незадокументированного соглашения вызова `super.clone()` по всей цепочке наследования.

6. **Проверяемое исключение.** `clone()` выбрасывает `CloneNotSupportedException`, что делает код громоздким.

**Предпочтительные альтернативы:**

| Подход | Когда использовать | Достоинства | Недостатки |
|---|---|---|---|
| Copy constructor | Большинство случаев | Явный, типобезопасный, управляемый | Нужно писать вручную |
| Copy factory method | Полиморфное копирование | Не нужна реализация в подклассах | Нужно писать вручную |
| Serialization / deserialization | Готовый граф объектов с `Serializable` | Работает "из коробки" (`SerializationUtils.clone`) | Медленно, требует `Serializable` |
| Сторонние библиотеки | Быстрое прототипирование | Минимум кода | Зависимость, reflection overhead |

Пример copy constructor:

```java
public class Address {
    private final String city;
    private final List<String> tags;

    // copy constructor — глубокая копия
    public Address(Address other) {
        this.city = other.city;
        this.tags = new ArrayList<>(other.tags); // копируем изменяемый список
    }
}
```

Блох и Дуг Ли не рекомендуют использовать `Cloneable` в новом коде. Исключение — копирование массивов: `array.clone()` остаётся наиболее эффективным способом.
### Вопрос 28. Чем Adapter отличается от Facade? Можно ли сказать, что Adapter адаптирует интерфейс, а Facade упрощает набор интерфейсов? Приведите пример из стандартной библиотеки Java для каждого.

**Категория:** Паттерны · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Facade vs. Adapter Pattern: Understanding Key Differences and Applications — Software Patterns Lexicon](https://softwarepatternslexicon.com/java/structural-patterns/facade-pattern/facade-vs-adapter-pattern/)

Ключевое различие — в намерении (intent). **Facade** предоставляет упрощённый интерфейс к сложной подсистеме: он создаёт новый унифицированный интерфейс, агрегирующий несколько компонентов. **Adapter** позволяет двум несовместимым интерфейсам работать вместе, преобразуя один интерфейс в другой, которого ожидает клиент. Adapter работает с существующим интерфейсом, Facade — определяет новый. Использовать Adapter следует при интеграции нового компонента с существующей системой, имеющей другой интерфейс; Facade — когда нужно упростить взаимодействие со сложной подсистемой.

#### Ответ от Claude

**Формально верно:** Adapter адаптирует (преобразует) один интерфейс в другой, Facade упрощает набор интерфейсов (группу взаимодействующих объектов), предоставляя единую точку входа.

**Разница по намерению (GoF intent):**

| Критерий | Adapter | Facade |
|---|---|---|
| Цель | Сделать несовместимые интерфейсы совместимыми | Упростить сложную подсистему |
| Количество оборачиваемых объектов | Как правило, один | Может быть несколько (целая подсистема) |
| Интерфейс | Преобразует существующий | Определяет новый, более удобный |
| Клиент знает о подсистеме? | Нет — он видит только целевой интерфейс | Может знать, Facade — дополнительный слой удобства |
| Изменение интерфейса | Обязательно | Не обязательно (может просто делегировать) |

**Примеры из стандартной библиотеки Java:**

*Adapter:*
- `java.io.InputStreamReader(InputStream)` — адаптирует байтовый поток (`InputStream`) к символьному (`Reader`). Клиент работает через интерфейс `Reader`, а внутри делегирует `InputStream`.
- `java.io.OutputStreamWriter(OutputStream)` — аналогично для записи.
- `java.util.Arrays#asList(T...)` — адаптирует массив к интерфейсу `List`.
- `java.util.Collections#enumeration(Collection)` — адаптирует `Collection` к устаревшему интерфейсу `Enumeration`.

*Facade:*
- `javax.faces.context.FacesContext` — скрывает за единым интерфейсом такие компоненты, как `LifeCycle`, `ViewHandler`, `NavigationHandler`, `HttpServletRequest`/`HttpServletResponse`. Большинство клиентов не подозревают об их существовании.
- `java.net.URL` — метод `openConnection()` скрывает создание нужного `URLConnection` (HTTP, HTTPS, FTP и т.д.) и всю логику установки соединения.
- `javax.xml.parsers.DocumentBuilderFactory` — скрывает сложность создания парсера XML (выбор реализации, конфигурацию, namespace-aware и т.д.).
- Вызов `dataSource.getConnection()` в JDBC — за одним методом скрыты загрузка драйвера, создание или выдача соединения из пула, обновление статистики.

**Вывод:** Утверждение вопроса корректно, но с уточнением. Adapter работает с *одним* существующим интерфейсом и преобразует его в другой. Facade работает с *набором* интерфейсов (подсистемой) и создаёт поверх них новый, более простой интерфейс. При этом Facade не устраняет несовместимость — он добавляет слой удобства; Adapter же именно устраняет несовместимость.

### Вопрос 29. Decorator и Proxy имеют структурно одинаковую диаграмму классов — оба оборачивают объект с тем же интерфейсом. В чём разница по намерению (intent) и по тому, кто управляет жизненным циклом обёртываемого объекта?

**Категория:** Паттерны · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Decorator Pattern vs. Proxy Pattern — doeken.org](https://doeken.org/blog/decorator-vs-proxy-pattern)

Структурное сходство паттернов обманчиво: оба оборачивают экземпляр существующего интерфейса классом, реализующим тот же интерфейс и делегирующим вызовы. Однако намерения принципиально различаются. **Decorator** — аддитивен: он расширяет функциональность, всегда вызывая обёртываемый объект и дополняя результат (логирование, диспетчеризация событий). **Proxy** — рестриктивен: он может изменить поведение или вовсе запретить вызов (бросить исключение, вернуть кешированный ответ). Ключевое различие в жизненном цикле: Decorator **требует** передачи готового экземпляра через конструктор, не зная и не заботясь о том, как тот создан. Proxy **не требует** готового экземпляра — он может получить его извне, а может создать сам. «Decorator requires an instance of the interface it is wrapping, while a Proxy does not require such an instance.»

#### Ответ от Claude

**Намерение (intent по GoF):**

- **Decorator** — «Attach additional responsibilities to an object dynamically» (динамически добавить объекту новые обязанности). Цель: расширение поведения через композицию вместо наследования. Decorator всегда вызывает обёртываемый объект и добавляет что-то до/после. Несколько декораторов можно накладывать в любом порядке.
- **Proxy** — «Provide a surrogate or placeholder for another object to control access to it» (управлять доступом к объекту). Цель: контроль доступа, а не расширение функциональности. Proxy может не вызвать обёртываемый объект вовсе (lazy initialization, access control, кеш, remote proxy).

**Разница по жизненному циклу обёртываемого объекта:**

| Аспект | Decorator | Proxy |
|---|---|---|
| Создание обёртываемого объекта | Клиент создаёт и передаёт в конструктор | Proxy создаёт сам (или получает — оба варианта допустимы) |
| Владение | Отношение агрегации — объект может существовать независимо | Отношение композиции — жизненный цикл управляется Proxy |
| Знание о конкретном типе | Decorator работает через интерфейс, не знает конкретики | Proxy может знать конкретный тип (например, для lazy init) |

Источник refactoring.guru явно указывает: «Proxies manage the full lifecycle of their service objects», тогда как у Decorator «composition is always controlled by the client».

**Практические примеры в Java:**

- *Decorator:* `java.io.BufferedInputStream` оборачивает любой `InputStream`, клиент сам создаёт и передаёт исходный поток. Можно вложить `BufferedInputStream(new GZIPInputStream(new FileInputStream(...)))`.
- *Proxy:* `java.lang.reflect.Proxy` — JDK dynamic proxy; Spring AOP proxy — клиент получает прокси, не зная деталей создания объекта за ним. `java.rmi.stub` — remote proxy, скрывает сетевой вызов.

**Итог:** Оба паттерна структурно неотличимы (оба реализуют интерфейс цели и держат ссылку на объект того же интерфейса). Разница — только в намерении и семантике: Decorator добавляет поведение и всегда делегирует; Proxy контролирует доступ и управляет жизненным циклом. На практике выбор названия паттерна сигнализирует архитектурное намерение.

### Вопрос 30. Spring AOP использует Proxy-паттерн. Когда Spring выбирает JDK dynamic proxy, а когда CGLIB? Почему вызов метода внутри того же бина (self-invocation) не перехватывается аспектом, и как это обойти?

**Категория:** Паттерны · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Proxying Mechanisms — Spring Framework Reference Documentation](https://docs.spring.io/spring-framework/reference/core/aop/proxying.html)

Spring AOP автоматически выбирает тип прокси по цели: если целевой объект реализует хотя бы один интерфейс — используется JDK dynamic proxy (проксируются все реализуемые интерфейсы); если интерфейсов нет — создаётся CGLIB-прокси (подкласс целевого типа, генерируемый в рантайме). Принудительно переключить на CGLIB можно через `@EnableAspectJAutoProxy(proxyTargetClass = true)`.

**Ограничения CGLIB:** `final`-классы и `final`-методы не могут быть проксированы (нельзя создать подкласс/переопределить). `private`-методы и пакетные методы родительского класса также не советуются.

**Проблема self-invocation:** когда проксированный объект вызывает собственный метод через `this`, вызов идёт напрямую к реальному объекту, минуя прокси — advice не срабатывает. Документация предлагает три подхода:
1. Рефакторинг — убрать self-invocation (рекомендуется).
2. Self-injection — внедрить прокси-версию бина в себя и вызывать через неё.
3. `AopContext.currentProxy()` — получить текущий прокси (не рекомендуется, сильная связность).

#### Ответ от Claude

**Когда Spring выбирает JDK dynamic proxy, а когда CGLIB:**

| Условие | Прокси |
|---|---|
| Целевой бин реализует хотя бы один интерфейс | JDK dynamic proxy |
| Целевой бин не реализует ни одного интерфейса | CGLIB |
| `proxyTargetClass = true` (принудительно) | CGLIB в любом случае |
| Spring Boot 2.x+ (без конфигурации) | CGLIB по умолчанию (`spring.aop.proxy-target-class=true` по умолчанию) |

Начиная со Spring Boot 2.0, дефолт изменился: CGLIB используется по умолчанию вне зависимости от наличия интерфейсов. Это решило ряд проблем с `@Configuration` и транзакциями, но требует наличия no-arg конструктора. Откатить поведение: `spring.aop.proxy-target-class=false`.

**JDK dynamic proxy** работает через `java.lang.reflect.Proxy`: создаёт анонимный класс, реализующий целевые интерфейсы, и перехватывает вызовы через `InvocationHandler`. Требует наличия интерфейса.

**CGLIB** генерирует подкласс целевого класса, переопределяя методы и добавляя перехватчики. Не требует интерфейсов. Ограничения: не работает с `final`-классами и `final`-методами; `private`-методы не перехватываются.

**Почему self-invocation не перехватывается:**

Spring AOP работает на уровне прокси-объектов. Когда клиент вызывает метод бина — он вызывает его на прокси, и прокси применяет advice. Но когда метод внутри бина вызывает `this.anotherMethod()` — `this` ссылается на реальный объект, а не на прокси. Прокси в цепочку вызова не попадает, поэтому advice игнорируется.

```java
@Service
public class OrderService {
    @Transactional
    public void createOrder() {
        // this.validate() вызывается напрямую на реальном объекте,
        // @Transactional на validate() НЕ сработает
        this.validate();
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void validate() { ... }
}
```

**Способы обойти self-invocation:**

1. **Рефакторинг (рекомендуется):** вынести вызываемый метод в отдельный бин. Клиент внедряет этот бин и вызывает через него — прокси задействован.

2. **Self-injection:** внедрить сам бин в себя (через `@Autowired` или `ApplicationContext`), вызывать проксированный метод через эту ссылку.
   ```java
   @Service
   public class OrderService {
       @Autowired
       private OrderService self; // это прокси

       public void createOrder() {
           self.validate(); // идёт через прокси — advice сработает
       }

       @Transactional(propagation = Propagation.REQUIRES_NEW)
       public void validate() { ... }
   }
   ```

3. **`AopContext.currentProxy()` (крайний случай):** получить текущий прокси из контекста AOP. Требует `@EnableAspectJAutoProxy(exposeProxy = true)`. Жёстко связывает код со Spring AOP — не рекомендуется.
   ```java
   ((OrderService) AopContext.currentProxy()).validate();
   ```

4. **AspectJ (load-time или compile-time weaving):** AspectJ применяет advice прямо в байткоде, поэтому self-invocation перехватывается. Требует более сложной настройки, зато полностью устраняет проблему.

**Важно:** проблема self-invocation одинаково касается всех AOP-аннотаций: `@Transactional`, `@Cacheable`, `@Secured`, `@Async` и любых кастомных аспектов.
### Вопрос 31. Strategy и Template Method решают схожую задачу — изменение части алгоритма. В чём ключевое архитектурное отличие (наследование vs композиция)? Когда предпочесть один подход другому?

**Категория:** Паттерны · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Template Method — Refactoring.Guru](https://refactoring.guru/design-patterns/template-method)

Template Method — поведенческий паттерн, который определяет скелет алгоритма в суперклассе, позволяя подклассам переопределять отдельные шаги без изменения общей структуры. Template Method работает на уровне класса, то есть является статическим — изменение поведения происходит путём расширения подкласса.

Strategy, напротив, работает на уровне объекта и позволяет переключать поведение во время выполнения программы, предоставляя объекту различные стратегии через интерфейс.

Ключевое различие по словам источника: «Template Method использует наследование и позволяет изменять части алгоритма, расширяя их в подклассах. Strategy использует композицию и позволяет изменять части поведения объекта, предоставляя ему различные стратегии».

Когда использовать Template Method:
- когда нужно разрешить клиентам расширять только конкретные шаги алгоритма, не меняя всю структуру;
- когда несколько классов содержат почти идентичные алгоритмы с незначительными отличиями — общие шаги выносятся в суперкласс.

Когда использовать Strategy:
- когда нужна гибкость переключения поведения во время выполнения;
- когда существует несколько независимых алгоритмов для одной задачи;
- когда предпочтительна композиция вместо наследования.

#### Ответ от Claude

**Template Method** и **Strategy** оба позволяют варьировать часть алгоритма, но делают это принципиально разными механизмами.

**Ключевое архитектурное отличие:**

| Характеристика | Template Method | Strategy |
|---|---|---|
| Механизм | Наследование | Композиция |
| Уровень | Класс (статически) | Объект (динамически) |
| Изменение поведения | Через подкласс (compile-time) | Через инъекцию объекта (runtime) |
| Связанность | Жёсткая (подкласс знает о родителе) | Слабая (контекст знает только интерфейс) |
| Количество алгоритмов | Один скелет, вариации в шагах | Семейство независимых алгоритмов |

**Template Method** определяет неизменяемый скелет алгоритма в суперклассе (`final`-метод) и объявляет «крючки» — абстрактные или переопределяемые методы, которые подкласс заполняет конкретной реализацией. Изменить поведение можно только создав новый подкласс.

```java
// Template Method: скелет зафиксирован в суперклассе
public abstract class DataProcessor {
    // Шаблонный метод — неизменяем
    public final void process() {
        readData();
        processData();  // абстрактный — подкласс переопределяет
        writeData();
    }
    protected abstract void processData();
    protected void readData() { /* реализация по умолчанию */ }
    protected void writeData() { /* реализация по умолчанию */ }
}

public class CsvDataProcessor extends DataProcessor {
    @Override
    protected void processData() { /* CSV-логика */ }
}
```

**Strategy** выносит изменяемую часть алгоритма в отдельную иерархию объектов-стратегий и инжектирует нужную реализацию в контекст. Стратегию можно менять в runtime без пересоздания контекста.

```java
// Strategy: поведение инжектируется как объект
public interface SortStrategy {
    void sort(int[] data);
}

public class Sorter {
    private SortStrategy strategy;

    public Sorter(SortStrategy strategy) {
        this.strategy = strategy;
    }

    public void setStrategy(SortStrategy strategy) {
        this.strategy = strategy;  // смена на лету
    }

    public void sort(int[] data) {
        strategy.sort(data);
    }
}
```

**Когда предпочесть Template Method:**
- Алгоритм имеет фиксированную структуру, меняются только отдельные шаги.
- Логика вариаций тесно связана с базовой реализацией (много общего состояния суперкласса).
- Создание нового поведения означает создание нового типа (подкласса), а не нового объекта.
- Пример из JDK: `java.io.InputStream` — методы `read(byte[], int, int)` реализованы через абстрактный `read()`.

**Когда предпочесть Strategy:**
- Нужна замена алгоритма во время выполнения (разные пользователи, конфигурация, A/B-тест).
- Алгоритмы полностью независимы и не делят общее состояние.
- Нужно комбинировать несколько алгоритмов (Context может содержать несколько Strategy).
- Следование принципу «предпочитай композицию наследованию» (GoF) и OCP: добавить новую стратегию можно, не трогая контекст.
- Пример из JDK: `java.util.Comparator` — передаётся как Strategy в `Collections.sort()`.

**Практическое правило:** если поведение фиксируется на этапе компиляции и логически принадлежит типу — Template Method; если поведение должно меняться независимо от типа объекта, конфигурироваться или переключаться в runtime — Strategy.

### Вопрос 32. Как Template Method реализован в Spring: приведите конкретные примеры (JdbcTemplate, RestTemplate, AbstractBeanFactory). Почему в Spring 5+ предпочтительнее передавать callback-лямбду, а не создавать подкласс?

**Категория:** Паттерны · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Using the JDBC Core Classes — Spring Framework Reference Documentation](https://docs.spring.io/spring-framework/reference/data-access/jdbc/core.html)

JdbcTemplate реализует паттерн Template Method через механизм callback-интерфейсов. Он берёт на себя весь шаблонный код работы с JDBC: создание и освобождение ресурсов, обработку исключений, трансляцию SQL-исключений в `DataAccessException`. Прикладной код предоставляет только SQL и логику маппинга результатов.

До Java 8 callback реализовывался через анонимный класс:
```java
Actor actor = jdbcTemplate.queryForObject(
    "select first_name, last_name from t_actor where id = ?",
    new RowMapper<Actor>() {
        public Actor mapRow(ResultSet rs, int rowNum) throws SQLException {
            Actor newActor = new Actor();
            newActor.setFirstName(rs.getString("first_name"));
            newActor.setLastName(rs.getString("last_name"));
            return newActor;
        }
    },
    1212L);
```

В современном Spring (Java 8+) тот же callback записывается лямбдой:
```java
Actor actor = jdbcTemplate.queryForObject(
    "select first_name, last_name from t_actor where id = ?",
    (resultSet, rowNum) -> {
        Actor newActor = new Actor();
        newActor.setFirstName(resultSet.getString("first_name"));
        newActor.setLastName(resultSet.getString("last_name"));
        return newActor;
    },
    1212L);
```

Начиная со Spring 6.1 рекомендуется использовать `JdbcClient` — fluent API, объединяющий возможности `JdbcTemplate` и `NamedParameterJdbcTemplate`.

#### Ответ от Claude

Spring широко использует паттерн Template Method, реализуя его в двух формах: классическое наследование (AbstractBeanFactory) и callback-based Template (JdbcTemplate, RestTemplate).

**1. JdbcTemplate**

`JdbcTemplate` — классический пример callback-based Template Method. Неизменяемый алгоритм (шаблонный метод) включает: получение соединения → создание Statement → выполнение SQL → обработка ResultSet → трансляция исключений → закрытие ресурсов. Изменяемые шаги делегируются callback-интерфейсам:

- `RowMapper<T>` — маппинг строки ResultSet в объект;
- `PreparedStatementCreator` — создание PreparedStatement с параметрами;
- `ResultSetExtractor<T>` — извлечение данных из всего ResultSet;
- `RowCallbackHandler` — обработка строк без возврата значения.

```java
// Callback как лямбда (Java 8+)
List<Actor> actors = jdbcTemplate.query(
    "select first_name, last_name from t_actor",
    (rs, rowNum) -> new Actor(rs.getString("first_name"), rs.getString("last_name"))
);
```

**2. RestTemplate**

`RestTemplate` применяет тот же подход для HTTP. Защищённый метод `doExecute(URI, HttpMethod, RequestCallback, ResponseExtractor)` содержит неизменяемый алгоритм: создание запроса → вызов `requestCallback.doWithRequest(request)` → отправка → вызов `responseExtractor.extractData(response)` → закрытие ресурсов.

Публичные методы (`getForObject`, `postForEntity`, `exchange` и пр.) — это удобные обёртки, которые формируют нужные callback и делегируют в `doExecute`. При необходимости можно вызвать `execute()` напрямую со своим `RequestCallback` и `ResponseExtractor`.

```java
// Пример прямого вызова execute со своим ResponseExtractor
String result = restTemplate.execute(
    "https://example.com/api",
    HttpMethod.GET,
    request -> request.getHeaders().set("X-Custom", "value"),  // RequestCallback
    response -> new String(response.getBody().readAllBytes())   // ResponseExtractor
);
```

**3. AbstractBeanFactory**

`AbstractBeanFactory` — пример классического наследования через Template Method. Публичный `getBean()` делегирует в защищённый `doGetBean()`, который содержит полный алгоритм получения бина: разрешение алиасов → поиск в singleton-кеше → проверка у родительского BeanFactory → мерж BeanDefinition → вызов абстрактных методов.

Абстрактные методы, которые обязаны реализовать подклассы:
- `getBeanDefinition(String beanName)` — получить определение бина (реализует `DefaultListableBeanFactory`);
- `createBean(String beanName, RootBeanDefinition mbd, Object[] args)` — создать экземпляр (реализует `AbstractAutowireCapableBeanFactory`).

```
AbstractBeanFactory.getBean(String)
    -> doGetBean(...)
        -> containsBeanDefinition(name)   // abstract
        -> getBeanDefinition(name)        // abstract
        -> createBean(name, mbd, args)    // abstract
```

**Почему в Spring 5+ предпочтительнее лямбды, а не подклассы**

Для JdbcTemplate и RestTemplate создавать подкласс было бы избыточно — они не предназначены для наследования. Callback-интерфейсы (`RowMapper`, `RequestCallback`, `ResponseExtractor`) являются функциональными (один метод), поэтому с Java 8 они идеально заменяются лямбда-выражениями. Это даёт ряд преимуществ:

1. **Нет лишних классов.** Лямбда — это не новый тип, компилятор генерирует `invokedynamic`-инструкцию, не создавая отдельный `.class`-файл на каждый анонимный класс.
2. **Меньше церемоний.** Лямбда убирает шаблонный код `new RowMapper<>() { public T mapRow(...) { ... } }`.
3. **Инлайн-определение.** Логика callback расположена рядом с вызовом, код легче читать.
4. **Совместимость с method reference.** Если маппинг вынесен в метод класса, можно использовать `MyMapper::mapRow` без создания объекта-обёртки.
5. **Spring 6.1+: JdbcClient.** Fluent API полностью построен на лямбда-friendly дизайне и заменяет типичное использование `JdbcTemplate` для простых сценариев.

Наследование от шаблонных классов Spring (например, `AbstractBeanFactory`) оправдано только при расширении самого фреймворка, но не в прикладном коде.

### Вопрос 33. В чём разница между паттернами Command и Strategy? Оба инкапсулируют поведение в объект — что принципиально отличает их по намерению? Как Command поддерживает undo/redo и очередь отложенного выполнения?

**Категория:** Паттерны · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Command — Refactoring.Guru](https://refactoring.guru/design-patterns/command)

Command — поведенческий паттерн, который преобразует запросы в самостоятельные объекты, содержащие всю информацию о запросе. Это позволяет передавать запросы как аргументы методов, откладывать их выполнение, ставить в очередь, а также поддерживать отмену операций.

Несмотря на то что Command и Strategy оба могут использоваться для параметризации объекта некоторым действием, их намерения принципиально различны. **Command** преобразует операцию в объект для отложенного выполнения, постановки в очередь и поддержки истории команд. **Strategy** описывает разные способы выполнения одного и того же действия, позволяя переключать алгоритмы в рамках одного контекста.

Для реализации undo/redo паттерн Command предлагает следующий подход: команды, изменяющие состояние, сохраняют резервную копию состояния до выполнения. История команд — это стек объектов-команд. Операция undo извлекает последнюю команду из стека и восстанавливает сохранённое состояние. Команды только для чтения (например, copy) не сохраняются в истории.

Команды также поддерживают сериализацию: их можно преобразовать в строку, записать в файл или базу данных и восстановить как объекты для отложенного выполнения.

#### Ответ от Claude

**Разница по намерению**

Оба паттерна инкапсулируют поведение в объект, но с разными целями:

| Аспект | Command | Strategy |
|---|---|---|
| Намерение | Инкапсулировать **запрос** как объект | Инкапсулировать **алгоритм** как объект |
| Вопрос | «Что и когда выполнить?» | «Как выполнить?» |
| Получатель | Разные объекты-получатели (Receiver) | Один контекст |
| Время выполнения | Может быть отложено, поставлено в очередь | Выполняется немедленно в момент вызова |
| История | Команды хранят историю для undo/redo | Алгоритм не хранит историю |
| Состояние | Команда может хранить состояние (параметры, backup) | Стратегия обычно без состояния |
| Отношение к действию | Одна команда = одно конкретное действие | Одна стратегия = один вариант алгоритма |

**Структурное сходство и различие**

Визуально оба паттерна выглядят похоже: интерфейс с одним методом (`execute()` / `sort()`), набор реализаций. Разница не в структуре, а в семантике и контексте применения.

```java
// Strategy: разные алгоритмы для одной задачи (сортировка)
interface SortStrategy {
    void sort(int[] data);
}
class QuickSort implements SortStrategy { ... }
class MergeSort implements SortStrategy { ... }

// Command: запрос как объект с конкретным получателем
interface Command {
    void execute();
    void undo();
}
class PasteCommand implements Command {
    private final Editor editor;
    private final String backup;

    PasteCommand(Editor editor) {
        this.editor = editor;
        this.backup = editor.getText();  // сохраняем состояние ДО
    }

    @Override
    public void execute() {
        editor.insertText(Clipboard.get());
    }

    @Override
    public void undo() {
        editor.setText(backup);  // восстанавливаем состояние
    }
}
```

**Реализация undo/redo**

Паттерн Command поддерживает undo/redo через стек истории. Алгоритм:

1. Перед выполнением команда сохраняет резервную копию состояния Receiver.
2. После успешного `execute()` команда помещается в стек `history`.
3. `undo()` извлекает последнюю команду из стека и вызывает восстановление состояния.
4. `redo()` хранится в отдельном стеке (или итератор в истории) и повторно вызывает `execute()`.

```java
public class CommandHistory {
    private final Deque<Command> history = new ArrayDeque<>();
    private final Deque<Command> redoStack = new ArrayDeque<>();

    public void execute(Command command) {
        command.execute();
        history.push(command);
        redoStack.clear();  // новое действие сбрасывает redo-стек
    }

    public void undo() {
        if (!history.isEmpty()) {
            Command command = history.pop();
            command.undo();
            redoStack.push(command);
        }
    }

    public void redo() {
        if (!redoStack.isEmpty()) {
            Command command = redoStack.pop();
            command.execute();
            history.push(command);
        }
    }
}
```

**Очередь отложенного выполнения**

Command-объект самодостаточен: содержит всё необходимое для выполнения (ссылку на получателя, параметры). Это позволяет:

1. **Поставить команды в очередь** — `BlockingQueue<Command>` для пула потоков или задач;
2. **Сериализовать** — преобразовать в JSON/бинарный формат, сохранить в БД, передать по сети;
3. **Восстановить и выполнить позже** — deserialize + execute;
4. **Batch-исполнение** — макрос как список команд.

```java
// Очередь команд для асинхронного выполнения
class CommandQueue {
    private final BlockingQueue<Command> queue = new LinkedBlockingQueue<>();
    private final ExecutorService executor = Executors.newSingleThreadExecutor();

    public CommandQueue() {
        executor.submit(() -> {
            while (!Thread.currentThread().isInterrupted()) {
                Command cmd = queue.take();  // блокируется до прихода команды
                cmd.execute();
            }
        });
    }

    public void enqueue(Command command) {
        queue.put(command);  // ставим в очередь без немедленного выполнения
    }
}
```

**Примеры в JDK и экосистеме:**
- `java.lang.Runnable` / `java.util.concurrent.Callable` — Command без undo;
- `javax.swing.Action` — Command с метаданными (icon, tooltip);
- `java.util.concurrent.Future` — результат отложенного Command;
- Spring Batch `Step`, `Tasklet` — команды с поддержкой повтора и отката транзакций.

**Итог:** Strategy отвечает на вопрос «каким способом?» и переключает алгоритм в одном контексте. Command отвечает на вопрос «что сделать и когда?» и превращает действие в гражданский объект первого класса, пригодный для хранения, передачи, отмены и отложенного выполнения.
### Вопрос 34. Observer-паттерн и проблема «lapsed listener»: почему наблюдатель может стать причиной утечки памяти? Как WeakReference помогает её избежать, и какие издержки это несёт?

**Категория:** Паттерны · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Lapsed listener problem — Wikipedia](https://en.wikipedia.org/wiki/Lapsed_listener_problem)

Проблема «устаревшего слушателя» (lapsed listener) возникает в паттерне Observer, когда слушатель не отписывается от субъекта по завершении работы. Субъект удерживает сильную (strong) ссылку на наблюдателя, что препятствует его сборке мусором. Утечка каскадируется: вместе с наблюдателем в памяти остаётся весь граф объектов, на которые он ссылается. Кроме потребления памяти, «мёртвый» наблюдатель продолжает получать и обрабатывать уведомления, расходуя CPU впустую. Решение — хранить слушателей через WeakReference: если единственные ссылки на объект слабые, GC вправе его утилизировать. Субъект должен при обходе списка удалять «протухшие» (cleared) слабые ссылки. Статья фиксирует, что проблема особенно характерна для языков с GC и что подробные издержки подхода в ней не рассмотрены.

#### Ответ от Claude

**Механизм утечки (lapsed listener)**

В классическом Observer субъект (`Subject`) хранит список наблюдателей как коллекцию сильных ссылок (`List<Observer>`). Пока объект субъекта жив (часто — всё время жизни приложения), GC не может собрать ни один зарегистрированный наблюдатель, даже если с ним больше не работают из остального кода. Вместе с наблюдателем удерживается весь объектный граф: UI-компоненты, кэши, соединения с БД и т. д. Помимо утечки памяти, «мёртвый» слушатель продолжает вызываться при каждой нотификации — CPU расходуется впустую.

Типичный сценарий: фрагмент или диалог в Android-приложении регистрирует слушатель в глобальном синглтоне-источнике событий. При уничтожении UI `removeObserver()` не вызывается; синглтон удерживает фрагмент (и всё Activity) в памяти неограниченно долго.

**Решение через WeakReference**

Субъект может хранить слабые ссылки на наблюдателей:

```java
private final List<WeakReference<Observer>> listeners = new CopyOnWriteArrayList<>();

public void addObserver(Observer o) {
    listeners.add(new WeakReference<>(o));
}

public void notifyObservers(Event e) {
    Iterator<WeakReference<Observer>> it = listeners.iterator();
    while (it.hasNext()) {
        Observer o = it.next().get();
        if (o == null) {
            it.remove();   // "протухшая" ссылка — удаляем
        } else {
            o.onEvent(e);
        }
    }
}
```

Как только сильных ссылок на наблюдателя не остаётся, GC его собирает, а `WeakReference.get()` начинает возвращать `null`. Альтернатива — `WeakHashMap<Observer, Void>`, где ключи автоматически вычищаются GC.

JavaFX предоставляет готовое решение: `WeakChangeListener<T>` — обёртка вокруг обычного `ChangeListener`, которую `ObservableValue` хранит слабо.

**Издержки WeakReference**

1. **Неопределённость времени сборки.** GC собирает объект в недетерминированный момент; наблюдатель может прожить несколько циклов GC после того, как логически уже «умер». Нельзя полагаться на точное время отписки.

2. **Аннулирование при отсутствии удерживающей ссылки.** Если код регистрирует наблюдателя анонимным lambda или локальной переменной, единственная ссылка исчезает сразу после возврата из метода. GC немедленно собирает такой объект, и слушатель вообще не будет работать:

   ```java
   // НЕВЕРНО: лямбда не удерживается никем, будет немедленно собрана GC
   eventBus.addObserver(event -> doSomething(event));
   ```

   Чтобы WeakReference работал корректно, где-то в коде должна оставаться сильная ссылка с таким же жизненным циклом, как у наблюдателя (поле объекта-владельца, переменная класса и т. д.).

3. **Усложнение кода.** Каждый вызов `get()` требует проверки на `null`, итератор должен удалять протухшие записи. При конкурентном доступе нужна синхронизация или thread-safe-коллекция.

4. **Накладные расходы GC.** `WeakReference`-объекты попадают в очередь ссылок (`ReferenceQueue`), что создаёт дополнительную нагрузку на GC при большом числе слушателей.

5. **Не устраняет корневую причину.** Правильный дизайн требует явной отписки (`removeObserver()`). WeakReference — лишь страховочная сетка, а не замена дисциплине управления жизненным циклом.

**Альтернативы**

- Явный вызов `removeObserver()` в `destroy()` / `onStop()` / `close()` — самый надёжный подход.
- Использование `AutoCloseable`-подписки с try-with-resources.
- Реактивные библиотеки (RxJava, Project Reactor) с `Disposable`/`Subscription` для управления жизненным циклом.

### Вопрос 35. Как Observer реализован в Spring через ApplicationEvent / ApplicationListener и @EventListener? Чем синхронная публикация событий отличается от асинхронной (@Async), и какой паттерн лежит в основе механизма?

**Категория:** Паттерны · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Spring Framework Reference — Application Events and Listeners](https://docs.spring.io/spring-framework/reference/core/beans/context-introduction.html)

Spring реализует обработку событий поверх паттерна Observer. `ApplicationContext` содержит встроенный `ApplicationEventMulticaster`, который рассылает события всем зарегистрированным слушателям. Событие — объект, унаследованный от `ApplicationEvent` (или произвольный POJO начиная с Spring 4.2). Слушатель регистрируется либо реализацией интерфейса `ApplicationListener<E>`, либо аннотацией `@EventListener` на методе любого Spring-бина. Публикация производится через `ApplicationEventPublisher.publishEvent()`. По умолчанию — синхронная: `publishEvent()` блокируется до завершения всех слушателей, что позволяет слушателям работать в транзакционном контексте издателя. Для асинхронного режима слушатель помечается `@Async` (требует `@EnableAsync`); исключения из такого слушателя не распространяются к издателю. Слушатели можно упорядочить аннотацией `@Order`. Для событий, привязанных к транзакции, используется `@TransactionalEventListener` с фазами `AFTER_COMMIT`, `BEFORE_COMMIT`, `AFTER_ROLLBACK`, `AFTER_COMPLETION`.

#### Ответ от Claude

**Паттерн в основе**

Механизм событий Spring — это реализация паттерна **Observer** (также известного как Publish-Subscribe). Роли:

- **Субъект (Subject / Publisher):** `ApplicationEventPublisher` — интерфейс, реализованный `ApplicationContext`. Бины-издатели внедряют его через `@Autowired` или реализацию `ApplicationEventPublisherAware`.
- **Наблюдатели (Observers / Listeners):** бины, реализующие `ApplicationListener<E>` или имеющие методы с `@EventListener`.
- **Посредник (Mediator):** `ApplicationEventMulticaster` (по умолчанию `SimpleApplicationEventMulticaster`) — обходит всех зарегистрированных слушателей и вызывает соответствующие методы.

**Три способа объявить слушателя**

1. **Интерфейс `ApplicationListener<E>`** (традиционный подход, со Spring 1.0):

   ```java
   @Component
   public class OrderListener implements ApplicationListener<OrderCreatedEvent> {
       @Override
       public void onApplicationEvent(OrderCreatedEvent event) {
           sendEmail(event.getOrderId());
       }
   }
   ```

2. **Аннотация `@EventListener`** (со Spring 4.2, предпочтительный подход):

   ```java
   @Component
   public class NotificationService {
       @EventListener
       public void handleOrder(OrderCreatedEvent event) {
           sendEmail(event.getOrderId());
       }
   }
   ```

   Поддерживает фильтрацию через SpEL: `@EventListener(condition = "#event.amount > 1000")`.

3. **Несколько типов событий на один метод:**

   ```java
   @EventListener({ContextStartedEvent.class, ContextRefreshedEvent.class})
   public void handleContextEvents() { ... }
   ```

**Публикация события**

```java
@Service
public class OrderService {
    @Autowired
    private ApplicationEventPublisher publisher;

    public void createOrder(Order order) {
        // бизнес-логика...
        publisher.publishEvent(new OrderCreatedEvent(this, order.getId()));
    }
}
```

Событие может быть произвольным POJO (не обязательно наследоваться от `ApplicationEvent`) начиная с Spring 4.2.

**Синхронная публикация (по умолчанию)**

`publishEvent()` вызывает всех слушателей **в том же потоке и в той же транзакции**, что и издатель. Метод блокируется до тех пор, пока все слушатели не завершатся. Исключение в слушателе поднимается к издателю.

Преимущества: простота, участие в транзакции издателя.
Недостатки: медленный слушатель тормозит издателя; исключение в слушателе может откатить транзакцию.

**Асинхронная публикация через `@Async`**

```java
@Configuration
@EnableAsync
public class AsyncConfig { }

@Component
public class AsyncListener {
    @EventListener
    @Async
    public void handleOrder(OrderCreatedEvent event) {
        // выполняется в отдельном потоке (TaskExecutor)
    }
}
```

Ключевые отличия от синхронного режима:

| Характеристика | Синхронный | Асинхронный (`@Async`) |
|---|---|---|
| Поток выполнения | Поток издателя | Отдельный поток из пула |
| Блокировка издателя | Да | Нет |
| Транзакционный контекст | Унаследован от издателя | Отсутствует (новый поток) |
| Распространение исключений | Да (к издателю) | Нет (`AsyncUncaughtExceptionHandler`) |
| Возврат события для цепочки | Поддерживается | Не поддерживается |
| Propagation ThreadLocal/MDC | Автоматически | Требует настройки `TaskDecorator` |

**Транзакционные события**

`@TransactionalEventListener` — расширение `@EventListener`, позволяющее привязать вызов слушателя к фазе транзакции:

```java
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void afterCommit(OrderCreatedEvent event) {
    // вызывается только после успешного коммита транзакции
}
```

**Полностью асинхронный мультикастер**

Вместо `@Async` на каждом слушателе можно сделать весь `ApplicationEventMulticaster` асинхронным:

```java
@Bean
ApplicationEventMulticaster applicationEventMulticaster(TaskExecutor executor) {
    SimpleApplicationEventMulticaster multicaster = new SimpleApplicationEventMulticaster();
    multicaster.setTaskExecutor(executor);
    return multicaster;
}
```

Это делает все события асинхронными глобально.

### Вопрос 36. Вы применяете цепочку Decorator'ов поверх сервиса. Как обеспечить корректный порядок оборачивания при конфигурации через Spring? Какие проблемы возникают при использовании @Primary и @Qualifier?

**Категория:** Паттерны · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Ways to decorate Spring beans — QAware Engineering Blog](https://www.qaware.de/en/blog/spring-decorate-bean/)

Статья описывает несколько подходов к декорированию Spring-бинов. При использовании `@Primary` декоратор помечается как основная реализация интерфейса, а через `@Qualifier` в него инжектируется оригинальный бин. Это работает для одного декоратора, но усложняется при цепочке. При невозможности контролировать создание оригинального бина рекомендуется `BeanPostProcessor`: он вызывается для каждого бина в контексте, позволяет перехватить нужный тип и вернуть обёрнутую версию. Альтернативы: Java-конфигурация (`@Bean`-методы с явным порядком вызовов), кастомные квалификаторы, собственный `AutowireCandidateResolver`. Недостатки `@Qualifier` — высокая связность (все точки инжекции должны знать конкретное имя бина).

#### Ответ от Claude

**Задача**

Допустим, есть интерфейс `NotificationService` и три реализации: `BasicNotificationService` (ядро), `LoggingDecorator` (добавляет логирование) и `ValidationDecorator` (добавляет валидацию). Нужно собрать цепочку: `ValidationDecorator` → `LoggingDecorator` → `BasicNotificationService`.

**Подход 1: явная Java-конфигурация через `@Bean` (наиболее надёжный)**

```java
@Configuration
public class NotificationConfig {

    @Bean
    public NotificationService basicNotificationService() {
        return new BasicNotificationService();
    }

    @Bean
    public NotificationService loggingNotificationService() {
        return new LoggingDecorator(basicNotificationService());
    }

    @Bean
    @Primary
    public NotificationService validationNotificationService() {
        return new ValidationDecorator(loggingNotificationService());
    }
}
```

Порядок оборачивания полностью контролируется явно. `@Primary` указывает, какой бин инжектировать по умолчанию во весь остальной код. Нет неоднозначности — каждый метод возвращает свою версию с явно заданным делегатом.

**Подход 2: `@Component` + `@Qualifier` (декларативный, но хрупкий)**

```java
@Component("basicNotificationService")
public class BasicNotificationService implements NotificationService { ... }

@Component("loggingNotificationService")
public class LoggingDecorator implements NotificationService {
    public LoggingDecorator(
            @Qualifier("basicNotificationService") NotificationService delegate) { ... }
}

@Component("validationNotificationService")
@Primary
public class ValidationDecorator implements NotificationService {
    public ValidationDecorator(
            @Qualifier("loggingNotificationService") NotificationService delegate) { ... }
}
```

**Подход 3: `BeanPostProcessor` (для бинов, которые нельзя трогать)**

```java
@Component
public class NotificationDecoratorPostProcessor implements BeanPostProcessor {

    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) {
        if (bean instanceof NotificationService ns
                && !(bean instanceof LoggingDecorator)
                && !(bean instanceof ValidationDecorator)) {
            return new ValidationDecorator(new LoggingDecorator(ns));
        }
        return bean;
    }
}
```

Полезен при декорировании автоконфигурированных или сторонних бинов.

**Проблемы `@Primary`**

1. **Единственность на тип.** Пометить `@Primary` можно только один бин данного типа. При двух и более декораторах с `@Primary` Spring бросает `NoUniqueBeanDefinitionException`. Решение — только один декоратор помечается `@Primary`, остальные различаются по имени.

2. **Не решает проблему цепочки.** `@Primary` говорит Spring «используй этот по умолчанию», но не описывает, в каком порядке оборачивать. Если два декоратора оба претендуют на `@Primary`, нет механизма разрешения порядка.

3. **Скрытая зависимость.** Потребитель получает украшенный бин незаметно для себя, что затрудняет понимание того, какой код реально выполняется.

**Проблемы `@Qualifier`**

1. **Риск циклической зависимости.** Если декоратор объявлен как компонент и через `@Qualifier` запрашивает тот же тип интерфейса (например, `NotificationService`) без явного указания имени — Spring может попытаться инжектировать сам бин в себя или создать цикл, особенно при использовании `@Autowired` без строгого имени.

   ```java
   // ОПАСНО: если имя не совпадает точно с именем базового бина
   @Autowired
   @Qualifier("notificationService")  // может подхватить сам декоратор!
   private NotificationService delegate;
   ```

2. **Жёсткая связность.** Все классы-декораторы должны знать точные имена бинов своих делегатов. При переименовании бина или добавлении нового слоя нужно вручную обновлять квалификаторы.

3. **Нет гарантии порядка при сканировании.** При `@ComponentScan` порядок инициализации бинов не гарантирован. Если `LoggingDecorator` пытается получить `BasicNotificationService`, который ещё не создан, Spring обычно справляется, но при ленивых бинах или сложных конфигурациях возможны сюрпризы.

4. **`NoUniqueBeanDefinitionException` без квалификатора.** Если в контексте есть несколько реализаций одного интерфейса (все три: базовый + два декоратора) и потребитель делает `@Autowired NotificationService`, без `@Primary` или `@Qualifier` Spring бросит исключение.

**Рекомендованный подход для цепочки декораторов**

Предпочтительна **явная Java-конфигурация** (`@Configuration` + `@Bean`): порядок детерминирован, нет двусмысленности, нет риска циклических зависимостей, проще читать и тестировать. `@Qualifier` применяют только там, где Java-конфигурация неудобна. `BeanPostProcessor` — для интеграции с внешними библиотеками. При необходимости упорядочить несколько независимых декораторов-аспектов лучше использовать Spring AOP с `@Order`, который явно задаёт приоритет советов.

### Вопрос 37. Что такое паттерн Visitor и что такое «двойная диспетчеризация» (double dispatch)? Почему Java не поддерживает double dispatch из коробки и как Visitor решает эту проблему?

**Категория:** Паттерны · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Visitor and Double Dispatch — refactoring.guru](https://refactoring.guru/design-patterns/visitor-double-dispatch)

Проблема возникает, когда нужно применить внешнюю операцию к иерархии классов. Если написать перегруженный метод `export(Shape shape)`, компилятор привяжет вызов к версии с базовым типом ещё на этапе компиляции — это называется ранним (статическим) связыванием. Даже если в рантайме объект является `Circle`, вызов попадёт в `export(Shape)`, а не в `export(Circle)`.

**Почему Java не поддерживает double dispatch из коробки.** Перегруженные методы разрешаются статически (по объявленному типу параметра на момент компиляции), а переопределённые — динамически (по типу получателя в рантайме). Java реализует одиночную диспетчеризацию: вызов метода зависит от типа одного получателя (`this`). Двойная диспетчеризация потребовала бы учёта типа ещё и аргумента в рантайме, что в Java не реализовано.

**Как Visitor решает проблему.** Применяется двухшаговый приём:

1. **Первая диспетчеризация** — вызов переопределённого метода `accept(visitor)` на элементе. JVM выбирает реализацию по рантайм-типу элемента.
2. **Вторая диспетчеризация** — внутри `accept()` элемент вызывает `visitor.visit(this)`. Здесь `this` имеет конкретный тип, известный компилятору, поэтому разрешается нужная перегрузка `visit()`.

```java
interface Shape {
    void accept(Visitor v);
}

class Circle implements Shape {
    @Override
    public void accept(Visitor v) {
        v.visit(this); // this — Circle, перегрузка разрешается статически
    }
}

interface Visitor {
    void visit(Circle c);
    void visit(Square s);
}
```

Таким образом, итоговый вызов определяется типами двух объектов: элемента (через `accept`) и визитора (через `visit`), что и является эмуляцией double dispatch.

#### Ответ от Claude

**Одиночная диспетчеризация в Java.** В Java вызов виртуального метода разрешается по типу объекта-получателя (`this`) в рантайме. Тип аргумента при выборе перегрузки учитывается только на этапе компиляции — по объявленному (статическому) типу. Именно поэтому Java говорят, что поддерживает лишь single dispatch: только один тип (получатель) участвует в динамическом разрешении.

**Double dispatch** — механизм, при котором выбор конкретной реализации зависит от рантайм-типов двух объектов одновременно: и получателя, и аргумента. Ни Java, ни большинство mainstream-языков (C++, C#) не поддерживают это напрямую (исключение — языки с мультиметодами, например Clojure или CLOS).

**Паттерн Visitor** эмулирует double dispatch за счёт двух последовательных вызовов с динамической диспетчеризацией:

1. `element.accept(visitor)` — первый вызов переопределённого метода; JVM переходит в реализацию конкретного элемента (`Circle.accept`, `Square.accept` и т.д.).
2. `visitor.visit(this)` — внутри `accept` компилятор знает точный тип `this`, поэтому статически выбирает правильную перегрузку `visit`. Если одновременно существует несколько конкретных визиторов, полиморфизм по получателю (`visitor`) даёт ещё одно динамическое разрешение.

**Итоговая картина:**

| Шаг | Механизм | Что определяется динамически |
|-----|----------|------------------------------|
| `element.accept(visitor)` | override (позднее связывание) | тип элемента |
| `visitor.visit(this)` | overload (раннее связывание) + override | перегрузка по типу `this`; реализация по типу `visitor` |

**Зачем нужен Visitor.** Паттерн позволяет добавлять новые операции к стабильной иерархии классов без изменения самих классов. Вместо засорения каждого класса методами (`draw()`, `export()`, `validate()`...) все операции выносятся в отдельные классы-визиторы.

### Вопрос 38. Где паттерн Visitor применяется в JDK? Объясните роль javax.lang.model.element.ElementVisitor и как он используется при обработке аннотаций во время компиляции.

**Категория:** Паттерны · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [ElementVisitor (Java SE 17 & JDK 17) — docs.oracle.com](https://docs.oracle.com/en/java/javase/17/docs/api/java.compiler/javax/lang/model/element/ElementVisitor.html)

`ElementVisitor<R,P>` — интерфейс-визитор для программных элементов (классов, методов, полей, пакетов и т.д.), реализованный в стиле паттерна Visitor. Используется, когда конкретный вид элемента неизвестен на этапе компиляции кода процессора.

Интерфейс параметризован двумя типами: `R` — тип возвращаемого значения методов визита (используйте `Void`, если возврат не нужен), `P` — тип дополнительного параметра (используйте `Void`, если параметр не нужен).

Базовые методы интерфейса:

```
R visitType(TypeElement e, P p)            // класс, интерфейс, enum, record
R visitExecutable(ExecutableElement e, P p) // метод, конструктор
R visitVariable(VariableElement e, P p)     // поле, параметр, локальная переменная
R visitPackage(PackageElement e, P p)       // пакет
R visitTypeParameter(TypeParameterElement e, P p) // параметр типа
R visitModule(ModuleElement e, P p)         // модуль (Java 9+)
R visitRecordComponent(RecordComponentElement e, P p) // компонент record (Java 16+)
R visitUnknown(Element e, P p)              // неизвестный вид (для будущих версий)
```

**Важное предупреждение из Javadoc.** Не реализуйте интерфейс напрямую: в будущих версиях Java к нему могут быть добавлены методы, что сломает бинарную совместимость. Вместо этого следует расширять одну из абстрактных вспомогательных реализаций из пакета `javax.lang.model.util`: `SimpleElementVisitorN`, `ElementKindVisitorN` или `ElementScannerN`.

#### Ответ от Claude

**Место ElementVisitor в архитектуре annotation processing.**

Annotation Processing API (пакеты `javax.annotation.processing` и `javax.lang.model`) позволяет анализировать и генерировать код на этапе компиляции (`javac`). Центральный объект — `Element`, представляющий любой элемент программы (класс, метод, поле и т.д.). Поскольку процессор получает `Element` без информации о конкретном подтипе, для безопасной типизированной обработки используется `ElementVisitor`.

**Механизм двойной диспетчеризации.** Интерфейс `Element` объявляет метод:

```java
<R, P> R accept(ElementVisitor<R, P> v, P p);
```

Конкретные реализации (`TypeElement`, `ExecutableElement` и др.) переопределяют `accept` и вызывают соответствующий `visitXxx(this, p)`:

```java
// Упрощённая реализация TypeElement
public <R, P> R accept(ElementVisitor<R, P> v, P p) {
    return v.visitType(this, p); // первая диспетчеризация — тип элемента
}
```

Таким образом реализуется Visitor: первый вызов (`accept`) разрешает тип элемента, второй (`visitXxx`) — логику конкретного визитора.

**Применение в процессоре аннотаций.**

```java
@SupportedAnnotationTypes("com.example.MyAnnotation")
public class MyProcessor extends AbstractProcessor {

    @Override
    public boolean process(Set<? extends TypeElement> annotations,
                           RoundEnvironment roundEnv) {
        ElementVisitor<Void, Void> visitor = new SimpleElementVisitor14<>() {
            @Override
            public Void visitType(TypeElement e, Void p) {
                // обрабатываем класс/интерфейс с аннотацией
                processorEnv.getMessager().printMessage(
                    Diagnostic.Kind.NOTE, "Найден тип: " + e.getQualifiedName());
                return null;
            }

            @Override
            public Void visitExecutable(ExecutableElement e, Void p) {
                // обрабатываем метод/конструктор
                return null;
            }
        };

        for (Element elem : roundEnv.getElementsAnnotatedWith(MyAnnotation.class)) {
            elem.accept(visitor, null); // диспетчеризация по типу элемента
        }
        return true;
    }
}
```

**Семейства реализаций из `javax.lang.model.util`.**

| Семейство | Назначение |
|-----------|-----------|
| `AbstractElementVisitorN` | Скелетная реализация; бросает исключение для всех методов |
| `SimpleElementVisitorN` | Возвращает значение по умолчанию; удобен для частичной обработки |
| `ElementKindVisitorN` | Разбивает visitType на подвиды (CLASS, INTERFACE, ENUM, RECORD) |
| `ElementScannerN` | Рекурсивно обходит элемент и все вложенные элементы |

Суффикс N в именах классов соответствует версии Java (6, 7, 8, 9, 14...). Рекомендуется брать актуальную версию для максимального покрытия конструкций языка.

**Другие применения Visitor в JDK.**

- `javax.lang.model.type.TypeVisitor` — аналог для типов (`TypeMirror`).
- `javax.lang.model.element.AnnotationValueVisitor` — для значений аннотаций.
- `com.sun.source.tree.TreeVisitor` (внутренний API) — обход AST компилятора javac.
- `java.nio.file.FileVisitor` — обход файловой системы (семантически схож, хотя иерархия не классическая).

### Вопрос 39. Каковы недостатки паттерна Visitor? Почему добавление нового типа элемента в иерархию ломает открытость/закрытость (OCP), и как это сравнивается с добавлением новой операции?

**Категория:** Паттерны · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Visitor Design Pattern — refactoring.guru](https://refactoring.guru/design-patterns/visitor)

**Добавление новой операции** — сильная сторона паттерна. Достаточно создать новый класс, реализующий `Visitor`; существующие классы элементов остаются нетронутыми. OCP соблюдается со стороны элементов.

**Добавление нового типа элемента** — слабая сторона. Нужно обновить все существующие визиторы: добавить метод `visitNewType()` в интерфейс Visitor и реализовать его в каждом конкретном визиторе. Документация прямо указывает: «You need to update all visitors each time a class gets added to or removed from the element hierarchy.»

**Дополнительные недостатки по refactoring.guru:**

- Все визиторы нужно обновлять при изменении иерархии элементов.
- Визиторы могут не иметь доступа к приватным полям и методам элементов.
- Паттерн вносит дополнительную сложность по сравнению с более простыми решениями.

Вывод: паттерн Visitor идеален, когда операции меняются часто, а иерархия элементов стабильна.

#### Ответ от Claude

**Фундаментальная асимметрия Visitor.**

Паттерн Visitor представляет собой компромисс: он переносит «ось расширения» с типов элементов на операции. Это делает две задачи принципиально разными по стоимости:

| Задача | Стоимость | OCP |
|--------|-----------|-----|
| Добавить новую операцию | Создать один новый класс-визитор | Соблюдается: элементы не трогаем |
| Добавить новый тип элемента | Изменить интерфейс Visitor + все N конкретных визиторов | Нарушается: модифицируем существующие классы |

**Почему добавление нового элемента нарушает OCP.**

Принцип открытости/закрытости (Open/Closed Principle, Bertrand Meyer / Robert Martin) гласит: программные сущности должны быть открыты для расширения и закрыты для модификации. При добавлении, например, `Triangle` в иерархию фигур:

1. В интерфейс `ShapeVisitor` добавляется метод `void visit(Triangle t)` — это модификация существующего интерфейса.
2. Все конкретные визиторы (`AreaCalculator`, `PerimeterCalculator`, `DrawingVisitor` и т.д.) вынуждены реализовать новый метод — это модификация существующих классов.
3. При N визиторах изменяется N+1 файл (интерфейс + все реализации).

Если новый метод сделать `default` в интерфейсе (Java 8+), компиляция не сломается, но поведение по умолчанию часто будет некорректным и потребует переопределения вручную.

**Сравнение с подходом «метод в каждом классе» (альтернатива Visitor).**

В противоположном подходе каждый тип элемента содержит методы для всех операций:

| Задача | Visitor | Метод в классе |
|--------|---------|---------------|
| Новая операция | +1 класс | Изменить все N классов |
| Новый тип элемента | Изменить N+1 сущность | +1 класс |

Это известная дилемма «Expression Problem» (Wadler, 1998): в типичных ООП-языках нельзя одновременно легко расширять и типы, и операции без изменения существующего кода.

**Полный список недостатков Visitor.**

1. **Нарушение OCP при добавлении типов** — главный и системный недостаток, описан выше.

2. **Нарушение инкапсуляции.** Визитор вынужден работать с состоянием элемента. Либо элемент предоставляет публичные геттеры (нарушает инкапсуляцию), либо визитор получает доступ к пакетным/protected-членам (нарушает закрытость). Закон Деметры также нарушается: визитор «знает» внутреннее устройство чужих объектов.

3. **Сложность поддержки при нестабильной иерархии.** Sourcemaking прямо пишет: «If the subclasses in the aggregate node hierarchy are not stable, keeping the Visitor subclasses in sync requires a prohibitive amount of effort.»

4. **Нарушение SRP у конкретных визиторов.** Один класс-визитор вынужден содержать реализации для всех типов элементов. При большом числе типов класс становится монолитным.

5. **Сложность введения.** Паттерн требует изменить существующую иерархию элементов (добавить `accept()`), что невозможно, если иерархия принадлежит чужому коду (библиотека без исходников).

6. **Overhead двойного вызова.** Два виртуальных вызова вместо одного; на горячих путях это может быть заметно.

**Когда несмотря на недостатки применяют Visitor.**

- Иерархия типов стабильна (AST компилятора, DOM-дерево, алгебраические типы данных).
- Операций много и они меняются независимо (форматирование, валидация, сериализация, оптимизация).
- Операции должны накапливать состояние между посещениями нескольких узлов.

**Альтернативы, ослабляющие проблему.**

- **Sealed classes + pattern matching** (Java 21+): `switch (shape) { case Circle c -> ...; case Square s -> ...; }` — компилятор проверяет полноту, добавление нового `sealed`-подтипа вызывает compile-time ошибку в коде, использующем switch без `default`, что явно сигнализирует о необходимости обновления.
- **Мультиметоды** (библиотеки типа `method-dispatch`) — эмуляция истинного multiple dispatch.

### Вопрос 40. В чём разница между паттернами Mediator и Observer? Когда имеет смысл выбрать централизованный медиатор вместо децентрализованной публикации событий?

**Категория:** Паттерны · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Mediator — refactoring.guru](https://refactoring.guru/design-patterns/mediator)

Главная цель Mediator — устранить взаимные зависимости между компонентами системы. Вместо прямого общения компоненты взаимодействуют только через объект-медиатор.

Observer, напротив, устанавливает динамические однонаправленные связи: одни объекты следят за состоянием других, динамически подписываясь и отписываясь от уведомлений.

Разница между паттернами зачастую едва уловима. В большинстве случаев можно реализовать любой из них, а иногда оба одновременно. Существует популярная реализация Mediator, опирающаяся на Observer: медиатор играет роль издателя, а компоненты выступают подписчиками, подписывающимися на события медиатора и отписывающимися от них.

Если все компоненты превратить в издателей, позволив им устанавливать динамические связи между собой, получится децентрализованная сеть наблюдателей — без единого управляющего объекта. Mediator же обеспечивает централизованное управление: один объект координирует все взаимодействия и является единственной точкой зависимости для остальных компонентов.

#### Ответ от Claude

**Ключевые различия**

| Аспект | Mediator | Observer |
|---|---|---|
| Топология | Звезда: все компоненты — через медиатор | Граф: издатели и подписчики могут связываться напрямую |
| Управление | Централизованное — медиатор знает о всех участниках | Децентрализованное — каждый издатель управляет своим списком подписчиков |
| Осведомлённость компонентов | Компонент знает только о медиаторе | Подписчик знает об издателе (или о брокере) |
| Связь событие-реакция | Медиатор сам определяет, кто и как реагирует | Все подписчики получают событие и реагируют независимо |

**Когда выбирать Mediator**

- Компоненты сильно связаны: изменение одного требует изменения многих других.
- Логика взаимодействия сложная и должна быть собрана в одном месте (например, форма с несколькими UI-элементами, где нажатие кнопки влияет сразу на несколько других компонентов).
- Нужно централизованное логирование или аудит всех взаимодействий.
- Компоненты не должны знать друг о друге вообще — только о медиаторе.

**Когда выбирать Observer**

- Связи между объектами динамические и меняются во время выполнения.
- Несколько независимых подписчиков должны реагировать на одно событие без взаимной координации.
- Издатель не должен знать, сколько подписчиков у него есть и кто они (слабая связанность).
- Система строится на событийной модели (event bus, реактивные стримы).

**Важный нюанс (расхождение источников):** refactoring.guru подчёркивает, что граница между паттернами размытая — Mediator нередко реализуется поверх Observer (медиатор является издателем, а компоненты — его подписчиками). Принципиальное отличие не в механизме передачи события, а в том, кто несёт ответственность за координацию реакций: при Mediator эту ответственность берёт на себя медиатор, при Observer каждый подписчик реагирует самостоятельно.
