### Вопрос 21. Что такое функциональный интерфейс? Может ли он содержать default- и static-методы? Что произойдёт, если добавить в него второй абстрактный метод?

**Категория:** Функц. и Stream API · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [FunctionalInterface (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/lang/FunctionalInterface.html)

Функциональный интерфейс — это интерфейс, который содержит **ровно один абстрактный метод**. Поскольку default-методы имеют реализацию, они не являются абстрактными. Если интерфейс объявляет абстрактный метод, переопределяющий один из публичных методов `java.lang.Object`, это **не засчитывается** в счётчик абстрактных методов, так как любая реализация интерфейса унаследует реализацию из `Object`.

Аннотация `@FunctionalInterface` носит информационный характер. Если она присутствует, компилятор обязан выдать **ошибку**, если аннотируемый тип не удовлетворяет требованиям функционального интерфейса. Компилятор также считает любой интерфейс, соответствующий этому определению, функциональным **вне зависимости от наличия аннотации**.

#### Ответ от Claude

**Функциональный интерфейс** — интерфейс с ровно одним абстрактным методом (SAM — Single Abstract Method). Именно этот метод является целевым типом (target type) для лямбда-выражений и ссылок на методы.

**Могут ли присутствовать default- и static-методы?** Да, без ограничений. Default- и static-методы имеют реализацию, поэтому не являются абстрактными и не учитываются при подсчёте абстрактных методов. Пример стандартной библиотеки — `Function<T,R>`, которая помимо `apply(T t)` содержит default-методы `andThen` и `compose`.

```java
@FunctionalInterface
interface MyTransformer<T, R> {
    R transform(T input);                          // единственный абстрактный метод

    default MyTransformer<T, R> log() {            // разрешено: default-метод
        return input -> {
            R result = this.transform(input);
            System.out.println(input + " -> " + result);
            return result;
        };
    }

    static <T> MyTransformer<T, T> identity() {   // разрешено: static-метод
        return t -> t;
    }
}
```

**Что произойдёт при добавлении второго абстрактного метода?**
- Если интерфейс помечен `@FunctionalInterface` — **ошибка компиляции**: компилятор обязан её сгенерировать.
- Если аннотация отсутствует — интерфейс **молча перестаёт быть функциональным**: лямбда-выражения и ссылки на методы к нему уже не применимы, но сам интерфейс остаётся валидным обычным интерфейсом с двумя абстрактными методами.

Исключение: абстрактный метод, переопределяющий публичный метод `Object` (например, `toString()` или `equals()`), не нарушает правило SAM и не считается дополнительным абстрактным методом.

### Вопрос 22. Какие четыре основных встроенных функциональных интерфейса есть в java.util.function? Приведите сигнатуры их единственных абстрактных методов.

**Категория:** Функц. и Stream API · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [java.util.function (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/util/function/package-summary.html)

Пакет `java.util.function` предоставляет функциональные интерфейсы — целевые типы для лямбда-выражений и ссылок на методы. Базовые «формы» интерфейсов:

- **Function\<T, R\>** — принимает один аргумент типа T и возвращает результат типа R. Абстрактный метод: `R apply(T t)`.
- **Consumer\<T\>** — принимает один аргумент типа T, ничего не возвращает (side-effect). Абстрактный метод: `void accept(T t)`.
- **Predicate\<T\>** — предикат (булева функция) от одного аргумента типа T. Абстрактный метод: `boolean test(T t)`.
- **Supplier\<T\>** — поставщик результата типа T, не принимает аргументов. Абстрактный метод: `T get()`.

#### Ответ от Claude

Четыре основных функциональных интерфейса из `java.util.function` охватывают все базовые «формы» функций:

| Интерфейс | Абстрактный метод | Смысл |
|---|---|---|
| `Function<T, R>` | `R apply(T t)` | Преобразование T → R |
| `Consumer<T>` | `void accept(T t)` | Потребление T без возврата |
| `Predicate<T>` | `boolean test(T t)` | Проверка условия на T |
| `Supplier<T>` | `T get()` | Поставка T без входных данных |

**Краткие примеры:**
```java
Function<String, Integer>  len      = String::length;          // "hello" -> 5
Consumer<String>           printer  = System.out::println;     // "hi"    -> void
Predicate<String>          isEmpty  = String::isEmpty;         // ""      -> true
Supplier<List<String>>     newList  = ArrayList::new;          // ()      -> []
```

**Расширения, о которых важно знать:**
- `BiFunction<T, U, R>` — два входных аргумента: `R apply(T t, U u)`.
- `UnaryOperator<T>` — специализация `Function<T,T>` для одного типа.
- `BinaryOperator<T>` — специализация `BiFunction<T,T,T>`.
- Примитивные специализации: `IntFunction<R>`, `ToIntFunction<T>`, `IntConsumer`, `IntSupplier`, `IntPredicate` — позволяют избежать boxing/unboxing.

### Вопрос 23. Чем лямбда отличается от анонимного внутреннего класса с точки зрения захвата переменных (variable capture) и ключевого слова this?

**Категория:** Функц. и Stream API · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Lambda Expressions (The Java™ Tutorials) — docs.oracle.com](https://docs.oracle.com/javase/tutorial/java/javaOO/lambdaexpressions.html)

Лямбда-выражения отличаются от анонимных классов в части области видимости. Анонимный класс создаёт **новую область видимости**: он может скрывать (shadow) переменные внешнего scope, объявляя одноимённые локальные переменные. Лямбда-выражение, напротив, **не вводит нового уровня видимости** — она лексически вложена в окружающий контекст и не может переобъявить переменную, уже объявленную во внешнем scope.

В лямбде ключевое слово `this` ссылается на **экземпляр объемлющего класса**, тогда как в анонимном классе `this` ссылается на **экземпляр самого анонимного класса**.

Пример из документации:
```java
// в анонимном классе параметр с тем же именем разрешён (shadowing):
Consumer<Integer> anonConsumer = new Consumer<Integer>() {
    public void accept(Integer x) { ... }  // x скрывает внешний x
};

// в лямбде переобъявление запрещено:
Consumer<Integer> lambda = (x) -> { ... };  // ошибка, если x уже есть во внешнем scope
```

#### Ответ от Claude

Различия между лямбдой и анонимным классом по двум ключевым аспектам:

**1. Ключевое слово `this`**

В **анонимном классе** `this` указывает на экземпляр самого анонимного класса. Анонимный класс — это полноценный класс с собственными полями и идентичностью.

В **лямбде** `this` указывает на экземпляр **объемлющего класса**, потому что лямбда не является объектом с состоянием — у неё нет собственных полей, нет отдельного `this`.

```java
public class Example {
    private String name = "outer";

    void demo() {
        Runnable anon = new Runnable() {
            private String name = "anon";
            public void run() {
                System.out.println(this.name); // "anon" — this == экземпляр анон. класса
            }
        };

        Runnable lambda = () -> {
            System.out.println(this.name);     // "outer" — this == экземпляр Example
        };
    }
}
```

**2. Захват переменных и область видимости (variable capture / scoping)**

| Аспект | Лямбда | Анонимный класс |
|---|---|---|
| Вводит новую область видимости | Нет (лексически вложена) | Да (новый scope) |
| Может скрыть (shadow) переменную внешнего scope | Нет | Да |
| Может переобъявить параметр с именем из внешнего scope | Нет — ошибка компиляции | Да |
| Захват локальных переменных | Только effectively final | Только effectively final |
| Захват полей объекта | Свободный доступ через `this` | Свободный доступ через `this` (своё) |

Оба механизма одинаково ограничивают захват локальных переменных (они должны быть `final` или effectively final), однако **только анонимный класс** может объявлять собственные поля и скрывать переменные из внешнего контекста.

### Вопрос 24. Почему переменная из внешнего scope, захватываемая лямбдой, должна быть final или effectively final? Какова техническая причина этого ограничения (с точки зрения модели памяти JVM)?

**Категория:** Функц. и Stream API · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Lambda and final variables — javax0.github.io](https://javax0.github.io/2021/12/08/lambda-and-final-variables.html)

Техническая причина заключается в разнице между стеком и кучей. Локальные переменные хранятся в **стеке** и уничтожаются при возврате из метода. Лямбда-объект, напротив, хранится в **куче** и может пережить вызывающий метод (например, если передана в другой поток).

Чтобы лямбда могла «видеть» локальную переменную после завершения метода, Java **копирует значение** переменной в замыкание (closure). Однако Java принципиально не делает настоящих замыканий (как Groovy): лямбды работают не с переменными, а с **константами — скопированными значениями**. Требование `effectively final` гарантирует, что скопированное значение всегда совпадает с текущим значением переменной, и не создаёт иллюзии того, что изменение переменной внутри лямбды влияет на оригинал.

```java
public static Function<Integer, Integer> incrementer(final int step) {
    return (Integer i) -> i + step;  // step — скопированное значение, не переменная
}
```

#### Ответ от Claude

**Техническая причина — расположение в памяти JVM:**

Локальные переменные метода хранятся в **стеке потока** (thread stack). Стек является приватным для каждого потока и уничтожается при завершении метода (stack frame pop). Лямбда-объект — как и любой объект — хранится в **куче** (heap) и живёт независимо от того, завершился ли породивший её метод.

**Механизм захвата — копирование значения:**

Когда лямбда захватывает локальную переменную, JVM не хранит ссылку на ячейку стека (её после выхода из метода не существует). Вместо этого **значение переменной копируется** в поля lambda-объекта на куче (на уровне байткода это выглядит как передача в конструктор синтетического класса, созданного компилятором).

Если бы переменная могла изменяться после создания лямбды, возникло бы два независимых значения: оригинал в стеке и копия в куче. Они были бы рассинхронизированы, что порождает трудно уловимые баги — изменение переменной в методе не отразилось бы на лямбде, и наоборот.

**Требование JLS (Java Language Specification §15.27.2):**
> "The restriction to effectively final variables prohibits access to dynamically-changing local variables, whose capture would likely introduce concurrency problems."

**Отличие от полей объекта и статических переменных:**

Поля объекта и статические переменные хранятся в куче — там, где и лямбда-объект. Доступ к ним происходит **динамически по ссылке** (через `this`), а не через копирование. Поэтому для полей ограничение effectively final не применяется — лямбда всегда читает актуальное значение с кучи, а не устаревшую копию.

```java
class Counter {
    private int count = 0;                          // поле — в куче, доступ динамический

    Runnable makeIncrementer() {
        int step = 5;                               // локальная — в стеке
        // int step = 5; step++; — ошибка: не effectively final
        return () -> count += step;                 // step скопировано в лямбду (5)
                                                    // count доступно через this (динамически)
    }
}
```

**Итог:** Ограничение effectively final — это не произвольное языковое решение, а прямое следствие того, что JVM копирует локальные переменные при захвате, а копии и оригиналы не синхронизируются автоматически. Immutability захваченной переменной делает семантику предсказуемой и предотвращает гонки данных при передаче лямбды между потоками.
### Вопрос 25. Перечислите четыре вида method references (ссылок на методы). Для каждого приведите пример и объясните, к какому функциональному интерфейсу он приводится.

**Категория:** Функц. и Stream API · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Method References — The Java™ Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/java/javaOO/methodreferences.html)

Официальный туториал Oracle перечисляет четыре вида ссылок на методы в виде таблицы:

| Вид | Синтаксис | Примеры |
|-----|-----------|---------|
| Ссылка на статический метод | `КлассСодержащий::имяСтатическогоМетода` | `Person::compareByAge` |
| Ссылка на метод конкретного экземпляра | `конкретныйОбъект::имяМетода` | `myComparisonProvider::compareByName` |
| Ссылка на метод произвольного экземпляра заданного типа | `Тип::имяМетода` | `String::compareToIgnoreCase` |
| Ссылка на конструктор | `Класс::new` | `HashSet::new` |

Примеры эквивалентных лямбда-выражений из документации:
- `Person::compareByAge` ≡ `(a, b) -> Person.compareByAge(a, b)`
- `String::compareToIgnoreCase` ≡ `(String a, String b) -> a.compareToIgnoreCase(b)`
- `HashSet::new` ≡ `() -> { return new HashSet<>(); }` — соответствует `Supplier<HashSet>`

#### Ответ от Claude

Ссылки на методы (method references) — это синтаксический сахар над лямбда-выражениями, используемый везде, где ожидается функциональный интерфейс. Существует ровно четыре вида.

**1. Ссылка на статический метод**

```java
// Синтаксис: КлассСодержащий::staticMethodName
Arrays.sort(array, Integer::compare);
// Эквивалент: (a, b) -> Integer.compare(a, b)
```

Первый параметр функционального интерфейса становится первым аргументом статического метода. В примере выше `Integer::compare` приводится к `Comparator<Integer>` (или `BiFunction<Integer, Integer, Integer>`).

**2. Ссылка на метод конкретного экземпляра**

```java
// Синтаксис: конкретныйОбъект::instanceMethodName
String prefix = "Hello, ";
Function<String, String> greeter = prefix::concat;
// Эквивалент: name -> prefix.concat(name)
```

Метод вызывается на заранее известном, захваченном объекте. Здесь `prefix::concat` приводится к `Function<String, String>`.

**3. Ссылка на метод произвольного экземпляра заданного типа**

```java
// Синтаксис: ТипЭлемента::instanceMethodName
Arrays.sort(strings, String::compareToIgnoreCase);
// Эквивалент: (a, b) -> a.compareToIgnoreCase(b)
```

Первый параметр функционального интерфейса становится получателем (receiver) вызова метода, остальные — его аргументами. `String::compareToIgnoreCase` приводится к `Comparator<String>` (сигнатура: `(String, String) -> int`).

**4. Ссылка на конструктор**

```java
// Синтаксис: Класс::new
Supplier<ArrayList<String>> listFactory = ArrayList::new;
// Эквивалент: () -> new ArrayList<>()

Function<Integer, ArrayList<String>> sized = ArrayList::new;
// Эквивалент: n -> new ArrayList<>(n)
```

Компилятор выводит нужный конструктор по сигнатуре целевого функционального интерфейса. `ArrayList::new` может быть и `Supplier<ArrayList>`, и `Function<Integer, ArrayList>` — зависит от контекста.

**Итоговая таблица:**

| Вид | Синтаксис | Целевой ФИ (пример) |
|-----|-----------|----------------------|
| Статический метод | `Class::staticMethod` | `BiFunction`, `Comparator`, `Predicate` |
| Метод конкретного экземпляра | `obj::method` | `Supplier`, `Function`, `Runnable` |
| Метод произвольного экземпляра | `Type::method` | `Function`, `Comparator`, `BiFunction` |
| Конструктор | `Class::new` | `Supplier`, `Function`, `BiFunction` |

Ключевое правило: компилятор сопоставляет сигнатуру метода/конструктора с сигнатурой абстрактного метода целевого функционального интерфейса — количество и типы параметров должны совпадать (с учётом того, что для вида №3 получатель становится первым параметром).

### Вопрос 26. В чём разница между промежуточными (intermediate) и терминальными (terminal) операциями стрима? Почему промежуточные операции ленивы (lazy) и что это даёт на практике?

**Категория:** Функц. и Stream API · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [java.util.stream — Java Platform SE 8 (docs.oracle.com)](https://docs.oracle.com/javase/8/docs/api/java/util/stream/package-summary.html)

Официальная документация Java 8 даёт следующие определения (перевод):

**Промежуточные операции** возвращают новый стрим. Они всегда ленивы: выполнение промежуточной операции, например `filter()`, не производит никакой фильтрации — вместо этого создаётся новый стрим, при обходе которого содержатся элементы исходного стрима, соответствующие предикату.

**Терминальные операции**, такие как `Stream.forEach` или `IntStream.sum`, могут обходить стрим, чтобы получить результат или произвести побочный эффект. После выполнения терминальной операции конвейер стрима считается потреблённым и не может быть использован повторно.

Ленивость даёт два важных преимущества:
1. **Слияние операций (fusion):** в конвейере типа filter–map–sum операции могут быть объединены в единственный проход по данным с минимальным промежуточным состоянием.
2. **Избегание лишних вычислений:** для операций типа «найти первую строку длиннее 1000 символов» достаточно проверить ровно столько строк, сколько нужно, не перебирая весь источник. Это становится ещё важнее, когда источник — бесконечный стрим.

#### Ответ от Claude

**Промежуточные операции** — это операции, которые преобразуют один стрим в другой. Они не запускают обработку данных немедленно, а только описывают конвейер. Примеры: `filter()`, `map()`, `flatMap()`, `distinct()`, `sorted()`, `peek()`, `limit()`, `skip()`.

**Терминальные операции** — это операции, которые запускают конвейер и производят результат или побочный эффект. После их выполнения стрим закрывается и не может быть использован повторно. Примеры: `collect()`, `forEach()`, `reduce()`, `count()`, `findFirst()`, `anyMatch()`, `min()`, `max()`.

**Почему промежуточные операции ленивы?**

При вызове `filter()`, `map()` и т.п. JVM не начинает перебирать источник данных. Вместо этого каждая операция «оборачивает» предыдущий стрим в новый объект-декоратор, сохраняя ссылку на предикат или функцию. Реальный обход начинается только в момент вызова терминальной операции.

Это позволяет JVM строить оптимальный план выполнения. Например:

```java
list.stream()
    .filter(s -> s.startsWith("A"))   // ленивый
    .map(String::toUpperCase)          // ленивый
    .findFirst();                      // терминальный — запускает конвейер
```

Здесь данные проходят через `filter` и `map` **в одном проходе**, а не в двух отдельных итерациях. Как только `filter` пропустил первый подходящий элемент и `map` его преобразовал — `findFirst()` сразу возвращает результат, не обрабатывая оставшиеся элементы.

**Практические преимущества лени:**

1. **Short-circuit (ранний выход):** операции `findFirst()`, `anyMatch()`, `limit()` прекращают обработку, как только получен нужный результат. Без лени пришлось бы перебирать весь источник.

2. **Fusion (слияние операций):** несколько промежуточных операций могут выполняться в одном проходе, без создания промежуточных коллекций.

3. **Бесконечные стримы:** `Stream.iterate()` и `Stream.generate()` порождают элементы потенциально бесконечно. Без лени любая промежуточная операция зависла бы навсегда.

```java
// Работает именно благодаря лени:
Stream.iterate(0, n -> n + 1)
      .filter(n -> n % 2 == 0)
      .limit(10)
      .collect(Collectors.toList());
```

**Дополнительно:** среди промежуточных операций выделяют *stateless* (не хранят состояния между элементами: `filter`, `map`) и *stateful* (требуют просмотра всего стрима или его части: `sorted`, `distinct`). Последние ограничивают эффективность ленивого выполнения.

### Вопрос 27. Что произойдёт, если создать стрим из коллекции и вызвать на нём два терминальных оператора подряд? Каким исключением это завершится и почему?

**Категория:** Функц. и Stream API · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Stream (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/util/stream/Stream.html)

Официальная документация Java 8 говорит (перевод):

Стрим должен быть использован (вызов промежуточной или терминальной операции) **только один раз**. Это исключает, например, «разветвлённые» стримы, в которых один источник питает два или более конвейера, или множественные обходы одного стрима. Реализация стрима **может выбросить `IllegalStateException`**, если обнаружит повторное использование стрима.

Однако поскольку некоторые операции могут возвращать сам исходный объект стрима вместо нового, обнаружить повторное использование во всех случаях не гарантируется.

#### Ответ от Claude

**Что происходит:**

При вызове второй терминальной операции на уже использованном стриме будет выброшено исключение `java.lang.IllegalStateException` с сообщением:

```
stream has already been operated upon or closed
```

**Пример:**

```java
Stream<Integer> numbers = Stream.of(1, 2, 3, 4, 5);

long count = numbers.count();           // Первая терминальная операция — OK
Optional<Integer> first = numbers.findFirst(); // Вторая — выброс исключения
```

**Почему именно `IllegalStateException`?**

Стрим — это не коллекция данных, а описание конвейера обработки данных «в движении». После того как терминальная операция запустила конвейер, он отработал и закрылся: все внутренние итераторы источника были потреблены, промежуточные объекты-декораторы отработали. Объект стрима переходит в «закрытое» состояние, что и обозначает `IllegalStateException` (нарушение состояния объекта).

Важный нюанс из официальной документации: реализация **может** (но не обязана) бросить это исключение. Для некоторых стримов или операций обнаружение повторного использования не гарантировано — поведение может оказаться непредсказуемым вместо явного исключения.

**Как правильно исправить:**

Способ 1 — создавать новый стрим каждый раз:
```java
List<Integer> list = List.of(1, 2, 3, 4, 5);
long count = list.stream().count();
Optional<Integer> first = list.stream().findFirst(); // Новый стрим
```

Способ 2 — использовать `Supplier<Stream<T>>` для многократного получения свежего стрима:
```java
Supplier<Stream<Integer>> streamSupplier = () -> Stream.of(1, 2, 3, 4, 5);

long count = streamSupplier.get().count();
Optional<Integer> first = streamSupplier.get().findFirst();
```

**Дизайн-решение:** запрет повторного использования стрима — намеренное архитектурное решение. Оно позволяет реализациям применять оптимизации (fusion, short-circuit, параллелизм), не беспокоясь о повторных обходах. Стрим следует модели «одного использования», аналогичной `Iterator`.

### Вопрос 28. Объясните разницу между map() и flatMap(). Когда flatMap() необходим, а не просто удобен?

**Категория:** Функц. и Stream API · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Stream (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/util/stream/Stream.html)

Официальная документация Java 8 описывает методы следующим образом (перевод):

**`map()`** — возвращает стрим, состоящий из результатов применения данной функции к элементам данного стрима. Сигнатура: `<R> Stream<R> map(Function<? super T, ? extends R> mapper)`. Это операция «один к одному»: каждый элемент преобразуется ровно в один элемент.

**`flatMap()`** — возвращает стрим, состоящий из результатов замены каждого элемента содержимым соответствующего стрима, порождённого применением данной функции-отображения. Сигнатура: `<R> Stream<R> flatMap(Function<? super T, ? extends Stream<? extends R>> mapper)`. Каждый преобразованный стрим закрывается после того, как его содержимое помещено в результирующий стрим.

Пример из документации:
```java
// Преобразовать каждый заказ в стрим его позиций, затем «расплющить»:
orders.flatMap(order -> order.getLineItems().stream())

// Разбить строки на слова:
Stream<String> lines = Files.lines(path, StandardCharsets.UTF_8);
Stream<String> words = lines.flatMap(line -> Stream.of(line.split(" +")));
```

#### Ответ от Claude

**Ключевое различие:**

- `map()`: функция `T → R`, результат — `Stream<R>`. Преобразование **один к одному**.
- `flatMap()`: функция `T → Stream<R>`, результат — `Stream<R>` (сплющенный). Преобразование **один ко многим** с последующим «расплющиванием» (flattening).

**Наглядно:**

```
map(f):     [a, b, c]  →  [f(a), f(b), f(c)]
                                ↕ каждый элемент → один элемент

flatMap(f): [a, b, c]  →  [f(a)...] + [f(b)...] + [f(c)...]
                                ↕ каждый элемент → стрим → всё сливается в один стрим
```

**Код: разница типов:**

```java
List<String> words = List.of("Hello World", "Java Streams");

// map — получаем Stream<String[]>, нельзя работать как со словами:
Stream<String[]> wrong = words.stream()
    .map(s -> s.split(" "));

// flatMap — получаем Stream<String>, каждое слово отдельным элементом:
Stream<String> right = words.stream()
    .flatMap(s -> Arrays.stream(s.split(" ")));
```

**Когда `flatMap()` необходим (а не просто удобен):**

1. **Вложенные коллекции** — когда нужно работать с элементами коллекций внутри коллекции:

```java
List<List<Integer>> nested = List.of(List.of(1, 2), List.of(3, 4), List.of(5));

// map даст Stream<List<Integer>> — неудобно, нельзя посчитать сумму напрямую
// flatMap даёт Stream<Integer>:
int sum = nested.stream()
    .flatMap(Collection::stream)
    .mapToInt(Integer::intValue)
    .sum(); // 15
```

2. **Объекты с коллекциями-полями** — обязательный случай, нельзя обойтись `map`:

```java
// Получить все теги из всех статей в плоский список:
List<String> allTags = articles.stream()
    .flatMap(article -> article.getTags().stream())
    .distinct()
    .collect(Collectors.toList());
```

3. **Работа с `Optional` как со стримом** (Java 9+):

```java
// Отфильтровать пустые Optional и достать значения:
List<Optional<String>> optionals = List.of(Optional.of("a"), Optional.empty(), Optional.of("b"));
List<String> values = optionals.stream()
    .flatMap(Optional::stream) // Java 9: Optional.stream() возвращает стрим из 0 или 1 элемента
    .collect(Collectors.toList()); // ["a", "b"]
```

**Когда `map` достаточен:**

Если функция преобразования возвращает одиночное значение (не коллекцию и не стрим), `map` — правильный выбор. Использование `flatMap` там, где нужен `map`, усложняет код и снижает читаемость.

**Итог:**

| | `map()` | `flatMap()` |
|--|---------|-------------|
| Функция-mapper | `T → R` | `T → Stream<R>` |
| Кратность | 1 к 1 | 1 к N (включая 0) |
| Когда использовать | Простое преобразование | Разворачивание вложенных структур |
| Тип результата | `Stream<R>` | `Stream<R>` (сплющенный) |

`flatMap()` необходим каждый раз, когда применение `map` дало бы `Stream<Stream<R>>` или `Stream<Collection<R>>` — то есть вложенный тип, требующий дополнительного разворачивания.
### Вопрос 29. Чем отличаются orElse() и orElseGet() у Optional? В каком случае orElse() приводит к незапланированному вызову дорогостоящего метода, даже когда значение присутствует?

**Категория:** Функц. и Stream API · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Optional (Java SE 17 & JDK 17) — Oracle Docs](https://docs.oracle.com/en/java/javase/17/docs/api/java.base/java/util/Optional.html)

`orElse(T other)` — если значение присутствует, возвращает его, иначе возвращает `other`. Параметр `other` может быть `null`. Никаких исключений не бросает.

`orElseGet(Supplier<? extends T> supplier)` — если значение присутствует, возвращает его, иначе возвращает результат, произведённый supplying-функцией. Бросает `NullPointerException`, если значение отсутствует и переданный supplier равен `null`.

Ключевое различие: `orElse()` принимает уже вычисленное значение (eager evaluation — значение вычисляется всегда при вызове), а `orElseGet()` принимает `Supplier`, который вызывается лениво — только когда значение в `Optional` отсутствует.

#### Ответ от Claude

**Сигнатуры методов:**

```java
public T orElse(T other)
public T orElseGet(Supplier<? extends T> supplier)
```

**Принципиальное различие — момент вычисления аргумента:**

| Метод | Тип вычисления | Аргумент вычисляется |
|---|---|---|
| `orElse(T other)` | Eager (жадное) | Всегда, до вызова метода |
| `orElseGet(Supplier)` | Lazy (ленивое) | Только когда `Optional` пуст |

**Проблема незапланированного вызова дорогостоящего метода:**

В Java аргументы метода вычисляются до его вызова (стратегия вычисления — call by value). Поэтому когда вы пишете:

```java
Optional<String> opt = Optional.of("hello");
String result = opt.orElse(expensiveDbCall()); // expensiveDbCall() БУДЕТ вызван!
```

Метод `expensiveDbCall()` выполнится в любом случае — JVM вычислит аргумент перед передачей в `orElse()`. Если `Optional` содержит значение, результат вызова просто отбрасывается, но сам вызов уже произошёл. Это может вызвать:
- Лишние обращения к БД или внешним сервисам
- Нежелательные побочные эффекты
- Деградацию производительности

**Правильное решение — использовать `orElseGet()`:**

```java
Optional<String> opt = Optional.of("hello");
String result = opt.orElseGet(() -> expensiveDbCall()); // вызов НЕ произойдёт
```

Supplier-лямбда создаётся как объект, но её тело (`expensiveDbCall()`) выполнится только если `Optional` пуст.

**Практические правила выбора:**

- `orElse("default")` — используйте для простых константных значений, уже вычисленных переменных. Стоимость создания `Supplier`-объекта здесь избыточна.
- `orElseGet(() -> ...)` — используйте всегда, когда вычисление дефолтного значения имеет стоимость: обращение к БД, сетевые вызовы, сложные вычисления, операции с файловой системой, или когда есть побочные эффекты.

**Дополнительно:** `orElseGet()` бросает `NullPointerException` если supplier равен `null` и Optional пуст, тогда как `orElse(null)` является легальным вызовом.

### Вопрос 30. Когда следует использовать reduce(), а когда collect()? В чём принципиальная разница между иммутабельной свёрткой (reduce) и мутабельной редукцией (collect) с точки зрения работы в параллельных стримах?

**Категория:** Функц. и Stream API · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Reduction — The Java™ Tutorials (Oracle)](https://docs.oracle.com/javase/tutorial/collections/streams/reduction.html)

`reduce` — операция свёртки, при которой каждый шаг создаёт новое значение. Если логика `reduce` предполагает добавление элементов в коллекцию, то каждый раз при обработке элемента аккумулятором создаётся новая коллекция с этим элементом — что неэффективно.

`collect` — мутабельная редукция: изменяет или мутирует существующее значение вместо создания нового. Принимает три параметра: `supplier` (фабрика контейнера), `accumulator` (добавляет элемент в контейнер, не создавая новый), `combiner` (объединяет два контейнера — используется в параллельных стримах). При запуске `collect` с параллельным стримом JDK создаёт новый поток всякий раз, когда функция-комбинатор создаёт новый объект, поэтому о синхронизации беспокоиться не нужно.

#### Ответ от Claude

**Иммутабельная свёртка — `reduce()`:**

```java
// identity + accumulator
int sum = numbers.stream().reduce(0, Integer::sum);

// identity + accumulator + combiner (для параллельных стримов с разными типами)
<U> U reduce(U identity, BiFunction<U, ? super T, U> accumulator, BinaryOperator<U> combiner);
```

Каждый шаг аккумулятора возвращает новое значение — предыдущее не изменяется. Результат — единственный иммутабельный объект.

**Мутабельная редукция — `collect()`:**

```java
// supplier + accumulator + combiner
List<String> list = stream.collect(
    ArrayList::new,          // supplier: создаёт новый контейнер
    ArrayList::add,          // accumulator: добавляет элемент в контейнер (мутирует)
    ArrayList::addAll        // combiner: сливает два контейнера
);
```

Аккумулятор изменяет (мутирует) контейнер на месте, а не создаёт новый. Это принципиально эффективнее для построения коллекций.

**Сравнение в контексте параллельных стримов:**

| Аспект | `reduce()` | `collect()` |
|---|---|---|
| Тип результата | Иммутабельное значение | Мутабельный контейнер |
| Параллелизм | Требует ассоциативности и отсутствия состояния | Использует отдельные контейнеры per-thread + combiner |
| Безопасность потоков | Не нужна (нет общего состояния) | JDK управляет слиянием, синхронизация не нужна |
| Concurrent-вариант | Нет | `groupingByConcurrent()`, `toConcurrentMap()` — один shared контейнер |

**Для `reduce()` в параллельных стримах требуется:**
1. **Ассоциативность** аккумулятора: `(a op b) op c == a op (b op c)` — иначе результат непредсказуем при разбивке на подзадачи.
2. **Совместимость identity**: `identity op x == x` для любого `x`.
3. **Наличие combiner** при несовпадении типов — он объединяет частичные результаты параллельных веток.

**Для `collect()` в параллельных стримах:**
Каждый поток работает со своим независимым контейнером (supplier вызывается per-thread). В конце combiner сливает контейнеры. При использовании `CONCURRENT`-коллектора (например, `groupingByConcurrent`) все потоки пишут в один общий контейнер — combiner не нужен, но теряется порядок.

**Практическое правило:**
- `reduce()` — для агрегирования примитивов и простых объектов (сумма, максимум, конкатенация строк с небольшим N).
- `collect()` — для построения коллекций, `Map`, `String` через `joining()`, любых контейнерных структур.
- Строковую конкатенацию через `reduce(String::concat)` использовать не следует: сложность O(n²) из-за создания нового объекта `String` на каждом шаге; корректная альтернатива — `collect(Collectors.joining())`.

### Вопрос 31. Collectors.toMap() бросает IllegalStateException при дублирующихся ключах. Как это исправить? Какие ещё параметры принимает toMap() в перегруженных версиях?

**Категория:** Функц. и Stream API · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Collectors (Java Platform SE 8) — Oracle Docs](https://docs.oracle.com/javase/8/docs/api/java/util/stream/Collectors.html)

`toMap(Function keyMapper, Function valueMapper)` — возвращает `Collector`, аккумулирующий элементы в `Map`. Если среди отображённых ключей есть дубликаты (по `Object.equals(Object)`), при выполнении операции сбора выбрасывается `IllegalStateException`.

Для разрешения коллизий существует перегрузка с merge-функцией: `toMap(Function keyMapper, Function valueMapper, BinaryOperator mergeFunction)`. Merge-функция вызывается при дублировании ключей и объединяет два значения в одно. Пример: `toMap(Person::getName, Person::getAddress, (s, a) -> s + ", " + a)`.

Четырёхпараметрическая версия `toMap(keyMapper, valueMapper, mergeFunction, Supplier mapSupplier)` позволяет также задать фабрику `Map`-реализации.

#### Ответ от Claude

**Причина `IllegalStateException`:**

`toMap()` без merge-функции реализован примерно так:

```java
(map, element) -> {
    K key = keyMapper.apply(element);
    if (map.containsKey(key))
        throw new IllegalStateException("Duplicate key " + key);
    map.put(key, valueMapper.apply(element));
}
```

Любые два элемента стрима, давшие одинаковый ключ, вызовут исключение.

**Исправление — передать merge-функцию (`BinaryOperator<V>`):**

```java
// Оставить первое вхождение
Map<String, Person> map = persons.stream()
    .collect(Collectors.toMap(
        Person::getName,
        Function.identity(),
        (existing, replacement) -> existing   // keep first
    ));

// Оставить последнее вхождение
.collect(Collectors.toMap(
    Person::getName,
    Function.identity(),
    (existing, replacement) -> replacement    // keep last
));

// Объединить значения
Map<String, String> phoneBook = persons.stream()
    .collect(Collectors.toMap(
        Person::getName,
        Person::getPhone,
        (p1, p2) -> p1 + ", " + p2           // concat
    ));
```

**Три перегрузки `Collectors.toMap()`:**

| Перегрузка | Сигнатура | Применение |
|---|---|---|
| 2 параметра | `toMap(keyMapper, valueMapper)` | Уникальные ключи гарантированы |
| 3 параметра | `toMap(keyMapper, valueMapper, mergeFunction)` | Дублирующиеся ключи нужно разрешить |
| 4 параметра | `toMap(keyMapper, valueMapper, mergeFunction, mapFactory)` | Нужна конкретная реализация `Map` |

**Четырёхпараметрическая версия** позволяет передать `Supplier<M>` для выбора реализации `Map`:

```java
// TreeMap — отсортированный порядок ключей
Map<String, Person> sorted = persons.stream()
    .collect(Collectors.toMap(
        Person::getName,
        Function.identity(),
        (a, b) -> a,
        TreeMap::new
    ));

// ConcurrentHashMap — потокобезопасная карта
Map<String, Person> concurrent = persons.stream()
    .collect(Collectors.toMap(
        Person::getName,
        Function.identity(),
        (a, b) -> a,
        ConcurrentHashMap::new
    ));
```

**Важно:** двухпараметрическая `toMap()` возвращает `HashMap`, порядок элементов в которой не гарантирован. Если нужен `LinkedHashMap` (сохранение порядка встречаемости), используйте четырёхпараметрическую версию с `LinkedHashMap::new`.

### Вопрос 32. Объясните разницу между Collectors.groupingBy() и Collectors.partitioningBy(). Какой downstream-коллектор позволяет получить Map<K, Long> с количеством элементов в каждой группе?

**Категория:** Функц. и Stream API · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Collectors (Java SE 17 & JDK 17) — Oracle Docs](https://docs.oracle.com/en/java/javase/17/docs/api/java.base/java/util/stream/Collectors.html)

`groupingBy(classifier)` — возвращает `Collector`, реализующий каскадную операцию «группировки» над входными элементами типа `T`, группируя их в соответствии с функцией классификации и возвращая результат в виде `Map<K, List<T>>`. Эквивалентен `groupingBy(classifier, toList())`. Не является конкурентным; для параллельных стримов следует рассмотреть `groupingByConcurrent()`.

`partitioningBy(predicate)` — возвращает `Collector`, разбивающий входные элементы согласно `Predicate` и организующий их в `Map<Boolean, List<T>>`. Возвращаемый `Map` всегда содержит записи для обоих ключей — `false` и `true`.

`counting()` — возвращает `Collector` типа `Collector<T, ?, Long>`, подсчитывающий количество входных элементов. Реализован как `reducing(0L, e -> 1L, Long::sum)`.

#### Ответ от Claude

**`groupingBy()` — группировка по произвольному классификатору:**

```java
// groupingBy(classifier) → Map<K, List<T>>
Map<City, List<Person>> byCity = persons.stream()
    .collect(Collectors.groupingBy(Person::getCity));

// groupingBy(classifier, downstream) → Map<K, D>
Map<City, Long> countByCity = persons.stream()
    .collect(Collectors.groupingBy(Person::getCity, Collectors.counting()));

// groupingBy(classifier, mapFactory, downstream) → M extends Map<K, D>
Map<City, Long> sortedCountByCity = persons.stream()
    .collect(Collectors.groupingBy(
        Person::getCity, TreeMap::new, Collectors.counting()));
```

Тип ключа `K` — произвольный, определяется возвращаемым типом функции-классификатора. Количество групп заранее неизвестно.

**`partitioningBy()` — разбиение по предикату:**

```java
// partitioningBy(predicate) → Map<Boolean, List<T>>
Map<Boolean, List<Student>> passFail = students.stream()
    .collect(Collectors.partitioningBy(s -> s.getGrade() >= 60));

// partitioningBy(predicate, downstream) → Map<Boolean, D>
Map<Boolean, Long> countPassFail = students.stream()
    .collect(Collectors.partitioningBy(
        s -> s.getGrade() >= 60, Collectors.counting()));
```

Тип ключа всегда `Boolean`. Результирующий `Map` **гарантированно содержит оба ключа** (`true` и `false`) — даже если одна из групп пуста.

**Сравнительная таблица:**

| Характеристика | `groupingBy` | `partitioningBy` |
|---|---|---|
| Тип ключа | `K` — любой | `Boolean` (только `true`/`false`) |
| Количество групп | Произвольное | Ровно 2 |
| Классификатор | `Function<T, K>` | `Predicate<T>` |
| Тип результата по умолчанию | `Map<K, List<T>>` | `Map<Boolean, List<T>>` |
| Гарантия обоих ключей | Нет — пустые группы отсутствуют | Да — оба ключа всегда присутствуют |
| Параллельный вариант | `groupingByConcurrent()` | Нет аналога |

**Downstream-коллектор для получения `Map<K, Long>`:**

`Collectors.counting()` — возвращает `Collector<T, ?, Long>`, подсчитывающий количество элементов в группе. Если группа пуста — результат `0`.

```java
// Map<Department, Long> — количество сотрудников по отделам
Map<Department, Long> headcount = employees.stream()
    .collect(Collectors.groupingBy(
        Employee::getDepartment,
        Collectors.counting()
    ));

// Map<Boolean, Long> — количество прошедших/не прошедших порог
Map<Boolean, Long> passFailCount = students.stream()
    .collect(Collectors.partitioningBy(
        s -> s.getGrade() >= 60,
        Collectors.counting()
    ));
```

**Когда использовать `partitioningBy` вместо `groupingBy`:** когда логика деления строго бинарная и вам важно явное присутствие обоих ключей в результирующем `Map` (например, отчёт «прошли / не прошли»). `groupingBy` с `Boolean`-классификатором дал бы аналогичный результат, но не гарантирует присутствие ключа для пустой группы.

### Вопрос 33. Что такое stateful и stateless промежуточные операции? Приведите примеры из Stream API. Почему stateful-операции (sorted, distinct) опасны в параллельных стримах?

**Категория:** Функц. и Stream API · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [java.util.stream (Java Platform SE 8) — Oracle Docs](https://docs.oracle.com/javase/8/docs/api/java/util/stream/package-summary.html)

Промежуточные операции делятся на **stateless** и **stateful**.

**Stateless-операции** (например, `filter` и `map`) не сохраняют состояние от ранее обработанных элементов при обработке нового элемента — каждый элемент может быть обработан независимо от других. Конвейеры, содержащие только stateless-операции, могут быть обработаны за один проход (как последовательно, так и параллельно) с минимальной буферизацией данных.

**Stateful-операции** (например, `distinct` и `sorted`) могут включать состояние от ранее обработанных элементов при обработке новых. Такие операции могут потребовать обработки всего входного потока перед выдачей результата — например, нельзя выдать отсортированный результат, не увидев все элементы. В результате при параллельном вычислении некоторые конвейеры со stateful-операциями могут потребовать нескольких проходов по данным или значительной буферизации.

#### Ответ от Claude

**Stateless-операции** обрабатывают каждый элемент независимо, не сохраняя никакого контекста о предыдущих элементах.

Примеры: `filter()`, `map()`, `flatMap()`, `peek()`, `mapToInt()`, `mapToObj()`.

```java
// Stateless: каждый элемент обрабатывается независимо
stream.filter(x -> x > 0).map(x -> x * 2)
```

**Stateful-операции** должны «помнить» информацию о ранее встреченных элементах, чтобы корректно обработать текущий.

Примеры: `sorted()`, `distinct()`, `limit()`, `skip()`.

```java
// Stateful: distinct должен помнить все уже виденные элементы
// sorted должен накопить все элементы перед выдачей результата
stream.distinct().sorted()
```

**Почему stateful-операции опасны в параллельных стримах:**

1. **Необходимость глобального состояния.** `distinct()` вынужден отслеживать уже встреченные элементы из всех потоков одновременно. Это требует синхронизации или использования потокобезопасных структур (например, `ConcurrentHashMap`), что создаёт контention и снижает пропускную способность.

2. **Блокирующая семантика `sorted()`.** Операция не может выдать ни одного элемента, пока не получит весь входной поток. При параллельном разбиении это означает: каждый поток накапливает свой подмассив, затем все результаты объединяются и сортируются. Выигрыш от параллелизма минимален или вовсе отсутствует.

3. **Нарушение порядка при `limit()` и `skip()`.** В параллельном стриме элементы поступают вне исходного порядка, поэтому `limit(N)` и `skip(N)` вынуждены восстанавливать порядок встречаемости, снижая эффективность.

4. **Недетерминированность.** Если stateful-операция зависит от внешнего изменяемого состояния (например, лямбда с полем класса), результат может отличаться от запуска к запуску.

**Практическое правило:** в параллельных стримах следует избегать или минимизировать stateful-операции; если `sorted()` или `distinct()` необходимы — помещайте их как можно ближе к финальной терминальной операции и убедитесь, что источник данных поддерживает эффективное разбиение.

### Вопрос 34. Как параллельный стрим разбивает данные на части? Что такое Spliterator, какие характеристики (SIZED, ORDERED, DISTINCT и др.) он несёт и как они влияют на выбор стратегии параллелизации?

**Категория:** Функц. и Stream API · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Spliterator (Java Platform SE 8) — Oracle Docs](https://docs.oracle.com/javase/8/docs/api/java/util/Spliterator.html)

`Spliterator` — объект для обхода и разбиения элементов источника. Он сообщает набор характеристик через битовые флаги метода `characteristics()`:

- **SIZED** — оценка `estimateSize()` соответствует точному числу элементов, которые будут встречены при полном обходе.
- **ORDERED** — элементы имеют определённый порядок встречаемости; `trySplit()` обязан возвращать строгий префикс.
- **DISTINCT** — гарантия уникальности всех элементов.
- **SORTED** — элементы следуют определённому порядку сортировки; также должен быть установлен ORDERED.
- **SUBSIZED** — все сплитераторы, полученные через `trySplit()`, будут одновременно SIZED и SUBSIZED.
- **IMMUTABLE** — источник не может быть структурно изменён во время обхода.
- **CONCURRENT** — источник может безопасно изменяться несколькими потоками одновременно.
- **NONNULL** — все элементы гарантированно ненулевые.

Метод `trySplit()` разделяет сплитератор на два: один покрывает часть элементов, второй — оставшуюся часть. Фреймворк рекурсивно вызывает `trySplit()` до тех пор, пока оценка размера не опустится ниже целевого порога (примерно `estimateSize / (parallelism * 8)`), после чего оставшиеся элементы обрабатываются последовательно.

#### Ответ от Claude

**Что такое Spliterator**

`Spliterator<T>` (от *splittable iterator*) — интерфейс, появившийся в Java 8, который описывает, как данные следует обходить и разбивать. В отличие от `Iterator`, он нативно поддерживает параллельную обработку.

Четыре ключевых метода:
- `tryAdvance(Consumer)` — обработать следующий элемент последовательно.
- `forEachRemaining(Consumer)` — обойти все оставшиеся элементы.
- `trySplit()` — разбить на два сплитератора; возвращает `null`, если разбиение невозможно или нецелесообразно.
- `estimateSize()` — вернуть оценку числа оставшихся элементов.

**Процесс разбиения параллельного стрима**

```
Source Spliterator
       ↓ trySplit()
  [Left]   [Right]
    ↓         ↓
 trySplit() trySplit()
[LL][LR] [RL][RR]   ... пока размер > порога
                        каждый кусок → forEachRemaining()
                        результаты объединяются снизу вверх
```

Фреймворк (`ForkJoinPool` + `CountedCompleter`) рекурсивно делит задачу. Целевой размер порции ≈ `estimateSize() / (commonPoolParallelism * 8)`.

**Как характеристики влияют на стратегию**

| Характеристика | Что означает | Влияние на параллелизацию |
|---|---|---|
| **SIZED** | Точный размер известен до начала обхода | Позволяет равномерно распределить работу; `ArrayList` делится ровно пополам |
| **SUBSIZED** | Все дочерние сплитераторы тоже SIZED | Гарантирует точный баланс нагрузки на всех уровнях рекурсии |
| **ORDERED** | Фиксированный порядок встречаемости | `trySplit()` обязан возвращать строгий префикс; операции типа `findFirst()` обязаны соблюдать порядок — это добавляет накладные расходы |
| **DISTINCT** | Все элементы уникальны | Позволяет оптимизировать `distinct()` — пропустить дополнительную дедупликацию |
| **SORTED** | Элементы отсортированы | Позволяет оптимизировать `sorted()` — возможно слияние вместо полной сортировки |
| **IMMUTABLE** | Источник не изменяется | Не нужна защита от `ConcurrentModificationException` |
| **CONCURRENT** | Источник потокобезопасен для изменений | Можно читать и изменять одновременно; обычно несовместим с SIZED |
| **NONNULL** | Нет null-элементов | Можно пропустить null-проверки |

**Практические примеры**

- `ArrayList` → SIZED + SUBSIZED + ORDERED: делится точно пополам, отличная параллелизация.
- `HashSet` → SIZED + DISTINCT: делится по бакетам, без гарантии ORDERED.
- `LinkedList` → ORDERED, но не SIZED эффективно: плохая параллелизация (нужно пройти половину списка, чтобы найти середину).
- `Stream.generate()` → ни SIZED, ни SUBSIZED: невозможно предсказать размер; `trySplit()` возвращает фиксированный батч.

**Вывод:** SIZED + SUBSIZED — наиболее важные характеристики для эффективного параллелизма. ORDERED без SIZED создаёт дополнительные ограничения и может снизить производительность.

### Вопрос 35. В каком пуле потоков выполняется параллельный стрим по умолчанию? Как изменить параллелизм или задать собственный пул? Какой риск возникает при использовании общего ForkJoinPool.commonPool() в серверном приложении?

**Категория:** Функц. и Stream API · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Be Aware of ForkJoinPool#commonPool() — DZone](https://dzone.com/articles/be-aware-of-forkjoinpoolcommonpool)

По умолчанию параллельный стрим использует `ForkJoinPool.commonPool()`, который автоматически определяет размер, исходя из числа доступных ядер процессора. При количестве ядер более 3 создаётся `(количество ядер - 1)` потоков.

**Ключевые риски в серверных приложениях:**

- **Thread starvation при блокирующих задачах.** Когда блокирующие операции (I/O, обращения к БД, `Thread.sleep()`) занимают потоки commonPool, «вы можете очень легко заблокировать выполнение любой другой части вашей системы, которая также использует commonPool в это время».
- **Непредсказуемое поведение при малом числе ядер.** Когда `parallelism == 1`, JDK создаёт новый поток для каждой отправленной задачи вместо переиспользования пула, что может неожиданно деградировать производительность.

**Рекомендации:**
- Использовать commonPool только для вычислительных (неблокирующих) задач.
- Для блокирующих операций применять отдельный `ThreadPoolExecutor` или выделенный `ForkJoinPool`.

#### Ответ от Claude

**Пул по умолчанию**

Параллельный стрим выполняется в `ForkJoinPool.commonPool()` — пуле, разделяемом между всеми параллельными стримами и `CompletableFuture` (если не указан явный executor) в рамках всего JVM-процесса.

Размер пула: `Runtime.getRuntime().availableProcessors() - 1` потоков (т.е. на 8-ядерной машине — 7 потоков).

**Как изменить уровень параллелизма или задать собственный пул**

*Способ 1: системное свойство (меняет размер commonPool глобально)*
```bash
-Djava.util.concurrent.ForkJoinPool.common.parallelism=4
```

*Способ 2: выделенный ForkJoinPool (рекомендуемый способ)*
```java
ForkJoinPool customPool = new ForkJoinPool(4);
try {
    List<Result> result = customPool.submit(
        () -> dataList.parallelStream()
                      .map(this::process)
                      .collect(Collectors.toList())
    ).get();
} finally {
    customPool.shutdown();
}
```
Когда стрим запускается внутри задачи, уже выполняющейся в `ForkJoinPool`, он использует этот пул вместо commonPool.

*Важно:* этот приём работает благодаря тому, что реализация Stream API проверяет, находится ли текущий поток уже в `ForkJoinPool`, и использует его. Формально это недокументированное поведение, однако оно стабильно работает начиная с Java 8 и по-прежнему применяется на практике.

**Риски commonPool в серверных приложениях**

1. **Thread starvation (голодание потоков).** Пул разделяется между всеми компонентами приложения. Если один тяжёлый запрос запускает параллельный стрим с блокирующими операциями (SQL-запросы, HTTP-вызовы), он может занять все потоки commonPool. Все остальные параллельные операции в приложении встанут в очередь.

2. **Deadlock.** Если задача внутри commonPool сама ожидает выполнения другой задачи в том же commonPool (и пул полностью занят), возникает deadlock или многосекундное ожидание.

3. **Непредсказуемые задержки.** В условиях высокой нагрузки на сервере (тысячи RPS) несколько запросов могут одновременно использовать commonPool, что приводит к очередям и деградации latency всего приложения.

4. **Неудачный масштаб для I/O.** Число потоков commonPool определяется числом CPU-ядер, что оптимально только для CPU-bound задач. Для I/O-bound задач нужно значительно больше потоков.

**Итоговое правило:** используйте `parallelStream()` с commonPool только для CPU-bound вычислений без блокировок и без I/O. Для всего остального — выделенный пул с явным управлением жизненным циклом (`shutdown()` в `@PreDestroy`).

### Вопрос 36. Метод collect() принимает три параметра: supplier, accumulator, combiner. Для чего нужен combiner и при каком условии он вызывается? Что произойдёт, если combiner некорректно реализован?

**Категория:** Функц. и Stream API · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Collector (Java SE 24 & JDK 24) — Oracle Docs](https://docs.oracle.com/en/java/javase/24/docs/api/java.base/java/util/stream/Collector.html)

Метод `collect()` принимает три функции:

- **supplier** (`Supplier<A>`) — создаёт новый мутабельный контейнер результатов. При параллельном выполнении может вызываться несколько раз — каждый раз должен возвращать новый экземпляр.
- **accumulator** (`BiConsumer<A, T>`) — добавляет (fold) один элемент входного потока в контейнер результатов.
- **combiner** (`BinaryOperator<A>`) — принимает два частичных результата и объединяет их в один.

Согласно документации: *«Функция, принимающая два частичных результата и объединяющая их. Функция combiner может сложить состояние одного аргумента в другой и вернуть его, либо вернуть новый контейнер результатов»*.

Пример из документации:
```java
(left, right) -> { left.addAll(right); return left; }
```

**Ключевое ограничение — ассоциативность:** «Ограничение ассоциативности гласит, что разбиение вычислений должно давать эквивалентный результат». Если combiner реализован некорректно, параллельное и последовательное выполнения дадут разные результаты.

#### Ответ от Claude

**Три параметра `collect()`**

```java
<R> R collect(Supplier<R> supplier,
              BiConsumer<R, ? super T> accumulator,
              BiConsumer<R, R> combiner)
```

- **supplier** — фабрика контейнера. Создаёт пустой контейнер для накопления результата. Например: `ArrayList::new`, `StringBuilder::new`.
- **accumulator** — добавляет один элемент стрима в контейнер. Например: `List::add`, `(sb, s) -> sb.append(s)`.
- **combiner** — объединяет два частично заполненных контейнера в один. Например: `(left, right) -> { left.addAll(right); return left; }`.

**Когда вызывается combiner**

В **последовательном стриме** combiner не вызывается вовсе: существует единственный контейнер, в который накапливаются все элементы через accumulator.

В **параллельном стриме** процесс выглядит так:
```
Источник данных разбивается на N частей
         ↓
[supplier() → accumulator * k1]  [supplier() → accumulator * k2]  ...
         ↓                                ↓
    partialResult1               partialResult2
         ↓_________________________________↓
              combiner(partialResult1, partialResult2)
                       ↓
                  finalResult
```

Combiner вызывается при каждом слиянии двух подрезультатов. При N частях будет `N-1` вызовов combiner.

**Что произойдёт при некорректной реализации combiner**

1. **Потеря данных.** Если combiner не переносит все элементы из правого контейнера в левый (или не создаёт новый объединённый контейнер), часть элементов будет утеряна. Параллельный стрим вернёт неполный результат, а последовательный — корректный.

2. **Нарушение ассоциативности.** Combiner должен давать одинаковый результат независимо от порядка слияния:
   ```
   combine(combine(A, B), C) == combine(A, combine(B, C))
   ```
   Нарушение этого условия приводит к недетерминированным результатам в зависимости от того, как ForkJoinPool разбил задачу.

3. **Непотокобезопасность.** Если combiner изменяет контейнер, одновременно используемый другим потоком, возможны `ConcurrentModificationException` или тихое повреждение данных.

4. **Бесшумная ошибка.** Наиболее опасный случай: при последовательном запуске тест проходит, при параллельном — результат неверен. Ошибка трудно воспроизводима.

**Пример корректного и некорректного combiner**

```java
// Корректно: переносим все элементы из right в left
List<String> result = stream.collect(
    ArrayList::new,
    List::add,
    List::addAll   // combiner
);

// Некорректно: combiner ничего не делает — потеря данных при параллельном выполнении
List<String> broken = stream.collect(
    ArrayList::new,
    List::add,
    (left, right) -> {}  // правая часть игнорируется!
);
```

**Требования к корректному combiner:**
- Ассоциативность: порядок слияния не влияет на результат.
- Совместимость с accumulator: `combine(a, b)` должно быть эквивалентно добавлению всех элементов `b` в `a` через accumulator.
- Потокобезопасность: два потока не должны одновременно изменять один и тот же контейнер.

### Вопрос 37. Что такое Collectors.teeing() (Java 12)? Как он позволяет применить два независимых коллектора к одному стриму одновременно и объединить их результаты?

**Категория:** Функц. и Stream API · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Collectors (Java SE 12 & JDK 12) — Oracle Javadoc](https://docs.oracle.com/en/java/javase/12/docs/api/java.base/java/util/stream/Collectors.html)

`Collectors.teeing()` — метод, добавленный в Java 12, который возвращает составной (`Collector`), являющийся комбинацией двух нижестоящих коллекторов.

Сигнатура:
```java
public static <T, R1, R2, R> Collector<T, ?, R> teeing(
    Collector<? super T, ?, R1> downstream1,
    Collector<? super T, ?, R2> downstream2,
    BiFunction<? super R1, ? super R2, R> merger)
```

Каждый элемент стрима, переданный в результирующий коллектор, обрабатывается обоими нижестоящими коллекторами, а затем их результаты объединяются с помощью указанной функции слияния (`merger`) в итоговый результат.

Типичный пример использования — вычисление среднего значения за один проход: первый коллектор суммирует элементы, второй считает их количество, а `BiFunction` вычисляет среднее:
```java
double average = Stream.of(1, 2, 3, 4, 5)
    .collect(Collectors.teeing(
        Collectors.summingInt(Integer::intValue),
        Collectors.counting(),
        (sum, count) -> (double) sum / count
    ));
// average = 3.0
```

#### Ответ от Claude

`Collectors.teeing()` появился в Java 12 (JEP не был выделен отдельно — вошёл в набор улучшений стандартной библиотеки). Название метода отсылает к понятию «T-разветвитель» (T-junction) из сантехники и электроники: поток данных разветвляется на два независимых пути, а затем результаты сходятся в одной точке.

**Сигнатура:**
```java
public static <T, R1, R2, R> Collector<T, ?, R> teeing(
    Collector<? super T, ?, R1> downstream1,
    Collector<? super T, ?, R2> downstream2,
    BiFunction<? super R1, ? super R2, R> merger)
```

**Как работает:**
1. Каждый элемент стрима поступает **одновременно** в `downstream1` и `downstream2`.
2. После завершения стрима оба коллектора дают результаты `R1` и `R2`.
3. `BiFunction merger` принимает оба результата и возвращает итоговое значение `R`.

Стрим проходится ровно **один раз** — это принципиальное отличие от альтернативного подхода, при котором пришлось бы либо дважды пройти по стриму, либо собирать данные во временные переменные.

**Примеры использования:**

Вычисление среднего за один проход:
```java
double avg = IntStream.rangeClosed(1, 10).boxed()
    .collect(Collectors.teeing(
        Collectors.summingInt(Integer::intValue), // sum = 55
        Collectors.counting(),                    // count = 10
        (sum, count) -> (double) sum / count      // 5.5
    ));
```

Нахождение минимального и максимального значений одновременно:
```java
record MinMax(Optional<Integer> min, Optional<Integer> max) {}

MinMax result = Stream.of(3, 1, 4, 1, 5, 9, 2, 6)
    .collect(Collectors.teeing(
        Collectors.minBy(Comparator.naturalOrder()),
        Collectors.maxBy(Comparator.naturalOrder()),
        MinMax::new
    ));
```

**До Java 12** такой же результат требовал двух проходов или сохранения промежуточного состояния:
```java
// Старый подход — два прохода или промежуточная коллекция
List<Integer> list = stream.collect(Collectors.toList());
int sum = list.stream().mapToInt(i -> i).sum();
long count = list.size();
double avg = (double) sum / count;
```

**Ограничения:** `teeing()` объединяет только **два** коллектора. Для большего числа параллельных редукций нужно вкладывать `teeing()` друг в друга или использовать `summarizingInt` / `Statistics`-классы.

### Вопрос 38. Что такое Collectors.flatMapping() (Java 9)? Чем он отличается от Collectors.mapping(), и в каких сценариях он незаменим при многоуровневой группировке (groupingBy + flatMapping)?

**Категория:** Функц. и Stream API · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Collectors (Java SE 21 & JDK 21) — Oracle Javadoc](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/util/stream/Collectors.html)

`Collectors.flatMapping()` — коллектор, добавленный в Java 9, который адаптирует нижестоящий коллектор, принимающий элементы типа `U`, к коллектору, принимающему элементы типа `T`, применяя к каждому входному элементу функцию плоского отображения перед накоплением. Функция плоского отображения преобразует входной элемент в стрим, покрывающий ноль или более выходных элементов, которые затем накапливаются нижестоящим коллектором. Каждый отображённый стрим закрывается после того, как его содержимое помещено нижестоящему коллектору. Если отображённый стрим равен `null`, вместо него используется пустой стрим.

Сигнатура:
```java
public static <T, U, A, R> Collector<T, ?, R> flatMapping(
    Function<? super T, ? extends Stream<? extends U>> mapper,
    Collector<? super U, A, R> downstream)
```

Коллекторы `flatMapping()` наиболее полезны при использовании в многоуровневой редукции, например, в качестве нижестоящего коллектора для `groupingBy` или `partitioningBy`. Например, для стрима заказов (`Order`) накопление набора позиций (`LineItem`) для каждого клиента:

```java
Map<String, Set<LineItem>> itemsByCustomerName
    = orders.stream().collect(
        groupingBy(Order::getCustomerName,
                   flatMapping(order -> order.getLineItems().stream(),
                               toSet())));
```

#### Ответ от Claude

**Разница между `Collectors.mapping()` и `Collectors.flatMapping()`:**

| Характеристика | `Collectors.mapping()` | `Collectors.flatMapping()` |
|---|---|---|
| Добавлен | Java 8 | Java 9 |
| Функция-маппер возвращает | `U` (одиночное значение) | `Stream<U>` (поток значений) |
| Аналог в Stream API | `Stream.map()` | `Stream.flatMap()` |
| Результат для вложенных коллекций | `Collection<Collection<U>>` (вложенность) | `Collection<U>` (плоская структура) |

**Сигнатуры для сравнения:**
```java
// mapping: маппер возвращает одно значение
public static <T, U, A, R> Collector<T, ?, R> mapping(
    Function<? super T, ? extends U> mapper,
    Collector<? super U, A, R> downstream)

// flatMapping: маппер возвращает Stream
public static <T, U, A, R> Collector<T, ?, R> flatMapping(
    Function<? super T, ? extends Stream<? extends U>> mapper,
    Collector<? super U, A, R> downstream)
```

**Сценарий, где flatMapping незаменим:**

Пусть каждый сотрудник принадлежит одному отделу и имеет список курсов:
```java
record Staff(String department, List<String> courses) {}

List<Staff> staffs = List.of(
    new Staff("CS", List.of("Algorithms", "DS")),
    new Staff("CS", List.of("Algorithms", "Computer Architecture")),
    new Staff("Math", List.of("Probability", "Discrete Maths"))
);
```

Задача: получить `Map<String, Set<String>>` — уникальные курсы по каждому отделу.

Попытка с `Collectors.mapping()`:
```java
// НЕВЕРНО: возвращает Set<List<String>>, а не Set<String>
Map<String, Set<List<String>>> wrong = staffs.stream()
    .collect(Collectors.groupingBy(
        Staff::department,
        Collectors.mapping(Staff::courses, Collectors.toSet())
    ));
// {CS=[["Algorithms","DS"], ["Algorithms","Computer Architecture"]], ...}
// — вложенные списки, дубликаты "Algorithms" не устранены
```

Правильное решение с `flatMapping()`:
```java
Map<String, Set<String>> departmentToCourses = staffs.stream()
    .collect(Collectors.groupingBy(
        Staff::department,
        Collectors.flatMapping(staff -> staff.courses().stream(),
                               Collectors.toSet())
    ));
// {CS=[Algorithms, DS, Computer Architecture], Math=[Probability, Discrete Maths]}
```

`flatMapping` разворачивает каждый `List<String>` в `Stream<String>` и передаёт отдельные строки в нижестоящий `toSet()`, устраняя и вложенность, и дубликаты.

**Почему не использовать `flatMap()` до `groupingBy()`?**

```java
// Этот подход теряет связь элемента со своим сотрудником/объектом:
staffs.stream()
    .flatMap(s -> s.courses().stream()) // теперь у нас стрим String без dept
    .collect(Collectors.groupingBy(???)) // невозможно сгруппировать по dept
```

Именно поэтому `flatMapping()` как downstream-коллектор незаменим: он работает **внутри** группы, уже разбитой по ключу, тогда как предварительный `flatMap()` разрушил бы структуру группировки.

### Вопрос 39. Для чего предназначен Collectors.filtering() (Java 9)? Чем результат groupingBy с downstream-фильтрацией через Collectors.filtering() отличается от применения filter() до groupingBy?

**Категория:** Функц. и Stream API · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Collectors (Java SE 21 & JDK 21) — Oracle Javadoc](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/util/stream/Collectors.html)

`Collectors.filtering()` адаптирует коллектор к принятию элементов того же типа `T`, применяя предикат к каждому входному элементу и накапливая только те, для которых предикат возвращает `true`.

Сигнатура:
```java
public static <T, A, R> Collector<T, ?, R> filtering(
    Predicate<? super T> predicate,
    Collector<? super T, A, R> downstream)
```

Коллекторы `filtering()` наиболее полезны при использовании в многоуровневой редукции, например, в качестве нижестоящего коллектора для `groupingBy` или `partitioningBy`. Например, для стрима сотрудников — накопить сотрудников по отделам с зарплатой выше порогового значения:

```java
Map<Department, Set<Employee>> wellPaidEmployeesByDepartment
    = employees.stream().collect(
        groupingBy(Employee::getDepartment,
                   filtering(e -> e.getSalary() > 2000,
                             toSet())));
```

Коллектор `filtering` отличается от операции `filter()` в стриме. Предположим, что в некотором отделе нет сотрудников с зарплатой выше порога. Использование `filtering`-коллектора, как показано выше, даст отображение из этого отдела на пустое `Set`. Если же вместо него использовать `filter()` в стриме, отображения для этого отдела **вообще не будет**.

#### Ответ от Claude

**Назначение `Collectors.filtering()`:**

Метод добавлен в Java 9 как downstream-коллектор для многоуровневых редукций. Он применяет фильтр **внутри** каждой группы, а не до разбиения на группы.

**Ключевое различие — поведение при пустых группах:**

```java
List<Employee> employees = List.of(
    new Employee("IT",  3000),
    new Employee("IT",  1500),
    new Employee("HR",  1200),
    new Employee("HR",  1800)
    // В отделе "HR" никто не зарабатывает > 2000
);
```

Подход 1 — `filter()` **до** `groupingBy()`:
```java
Map<String, List<Employee>> result1 = employees.stream()
    .filter(e -> e.getSalary() > 2000)          // HR-сотрудники отфильтрованы здесь
    .collect(Collectors.groupingBy(Employee::getDepartment));
// Результат: {IT=[Employee(IT, 3000)]}
// Ключ "HR" ОТСУТСТВУЕТ в Map — группа исчезла
```

Подход 2 — `Collectors.filtering()` **внутри** `groupingBy()`:
```java
Map<String, List<Employee>> result2 = employees.stream()
    .collect(Collectors.groupingBy(
        Employee::getDepartment,
        Collectors.filtering(e -> e.getSalary() > 2000, Collectors.toList())
    ));
// Результат: {IT=[Employee(IT, 3000)], HR=[]}
// Ключ "HR" ПРИСУТСТВУЕТ с пустым списком
```

**Сводная таблица различий:**

| Аспект | `filter()` до `groupingBy()` | `Collectors.filtering()` (downstream) |
|---|---|---|
| Момент фильтрации | До группировки | После группировки, внутри каждой группы |
| Группы без элементов | Отсутствуют в результирующей Map | Присутствуют с пустой коллекцией |
| Применимость | Общая фильтрация стрима | Избирательная фильтрация внутри групп |
| Семантика | «Не включать в группировку» | «Собрать группу, но взять только подходящие» |

**Когда важно сохранять пустые группы:**

1. Отчётность: нужно показать все отделы, даже если в них нет сотрудников с нужными характеристиками.
2. Дальнейшая обработка: код, потребляющий `Map`, ожидает ключи для всех категорий и не должен обрабатывать `null`/`getOrDefault`.
3. Логика «нулевого результата» должна быть видна явно, а не скрыта отсутствием ключа.

**Комбинирование с другими downstream-коллекторами:**

`Collectors.filtering()` можно вкладывать вместе с `flatMapping()` или `mapping()`:
```java
Map<String, List<String>> result = employees.stream()
    .collect(Collectors.groupingBy(
        Employee::getDepartment,
        Collectors.filtering(
            e -> e.getSalary() > 2000,
            Collectors.mapping(Employee::getName, Collectors.toList())
        )
    ));
// Для каждого отдела — список имён высокооплачиваемых сотрудников (пустой, если таких нет)
```
### Вопрос 40. Что делает Collectors.collectingAndThen()? Приведите пример: как с его помощью получить неизменяемый список или применить дополнительное преобразование к результату сборки?

**Категория:** Функц. и Stream API · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Collectors (Java SE 21 & JDK 21) — Oracle Javadoc](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/util/stream/Collectors.html)

`collectingAndThen` адаптирует существующий `Collector`, добавляя к нему дополнительный шаг финальной трансформации (finishing transformation).

**Сигнатура:**
```java
public static <T, A, R, RR> Collector<T, A, RR> collectingAndThen(
    Collector<T, A, R> downstream,
    Function<R, RR> finisher)
```

**Параметры:**
- `downstream` — базовый коллектор, который выполняет сбор элементов;
- `finisher` — функция, применяемая к результату базового коллектора.

**Возвращает:** коллектор, который выполняет действие базового коллектора, а затем применяет дополнительный шаг финализации.

**Пример из документации — неизменяемый список:**
```java
List<String> list = people.stream()
    .collect(collectingAndThen(toList(), Collections::unmodifiableList));
```

Здесь сначала `toList()` собирает элементы в обычный `ArrayList`, затем `Collections::unmodifiableList` оборачивает его в представление, запрещающее модификацию.

#### Ответ от Claude

`Collectors.collectingAndThen(downstream, finisher)` — это адаптер-коллектор, который позволяет применить произвольную функцию-трансформацию к итоговому результату другого коллектора. Шаблон использования: «собери так-то, а потом преобразуй результат вот так».

**Как работает внутри:**

Метод берёт характеристики `downstream`-коллектора и заменяет его finisher-функцию на композицию `downstream.finisher().andThen(finisher)`. Таким образом весь процесс сбора остаётся прежним, меняется только последний шаг.

**Пример 1 — неизменяемый список (Java 8+):**
```java
List<String> names = stream.collect(
    collectingAndThen(toList(), Collections::unmodifiableList));
```

Начиная с Java 10 есть более краткий способ для этого конкретного случая:
```java
List<String> names = stream.collect(Collectors.toUnmodifiableList());
```

**Пример 2 — получить количество элементов в группах и сразу сделать результат неизменяемым:**
```java
Map<String, Long> counts = stream.collect(
    collectingAndThen(
        groupingBy(Person::getDepartment, counting()),
        Collections::unmodifiableMap));
```

**Пример 3 — дополнительное преобразование типа:**
```java
// Собрать в List, затем преобразовать в массив
String[] arr = stream.collect(
    collectingAndThen(toList(), list -> list.toArray(new String[0])));
```

**Пример 4 — взять только первый элемент после сортировки:**
```java
Optional<String> first = stream.collect(
    collectingAndThen(
        toList(),
        list -> list.stream().sorted().findFirst()));
```

**Ключевые свойства:**
- Финальная трансформация выполняется ровно один раз после завершения всего сбора.
- Характеристика `IDENTITY_FINISH` у результирующего коллектора всегда отсутствует, поскольку finisher больше не является функцией-идентичностью.
- Характеристики `CONCURRENT` и `UNORDERED` наследуются от `downstream`-коллектора (за исключением `IDENTITY_FINISH`).
