### Вопрос 1. Вы заметили, что команда `docker build` для создания образов занимает значительно больше времени, чем ожидалось. После анализа Dockerfile выяснилось, что некоторые инструкции выполняются повторно, даже если они не изменялись. Выберите верный вариант решения проблемы.

**Категория:** Docker (HH) · **Уровень:** Advanced

**Варианты ответа:**
- Использование кеширования слоев Docker — все неизмененные инструкции будут пропускаться, что значительно сократит время сборки
- Уменьшение размера базового образа сократит время сборки, исключив повторное выполнение неизмененных слоев
- Сборка на сервере с более мощным оборудованием сократит время выполнения инструкций и устраняет повторное выполнение неизмененных слоев
- Удаление ненужных инструкций включит кеширование и исключит повторное выполнение неизмененных слоев
- Установка флага `--no-cache` оптимизирует сборку за счет пропуска слоев с ненужными данными

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/advanced.md)

**Правильный ответ:** Использование кеширования слоев Docker — все неизмененные инструкции будут пропускаться, что значительно сократит время сборки

#### Ответ от Claude

Docker строит образ послойно: каждая инструкция в Dockerfile создаёт отдельный слой. Механизм кеша работает следующим образом: если содержимое инструкции и всех предыдущих слоев не изменилось, Docker повторно использует уже существующий слой из кеша вместо его пересборки. Таким образом, переупорядочив инструкции (редко меняющиеся — выше, часто меняющиеся — ниже), можно добиться максимального попадания в кеш.

Остальные варианты неверны:
- Уменьшение базового образа влияет на размер итогового образа, но не устраняет повторное выполнение неизмененных инструкций.
- Более мощное железо ускорит выполнение инструкций, но не предотвратит их повторный запуск.
- Удаление «ненужных» инструкций не включает кеширование само по себе.
- Флаг `--no-cache` делает прямо противоположное — полностью отключает кеш и форсирует пересборку всех слоев.

Источник: [Docker Build cache — официальная документация](https://docs.docker.com/build/cache/).

### Вопрос 2. В вашем Dockerfile используется инструкция `ADD` для копирования файлов и извлечения архива. Вы заметили, что сборка образа занимает больше времени, чем ожидалось, а размер образа увеличивается из-за ненужных файлов. После изменений: (1) сборка образа стала занимать меньше времени; (2) размер образа уменьшился, так как больше нет ненужных файлов из архива. Какие изменения были внесены?

**Категория:** Docker (HH) · **Уровень:** Advanced

```dockerfile
FROM nginx:latest
ADD /app.tar.gz /usr/share/nginx/html
ADD /config/nginx.conf /etc/nginx/nginx.conf
CMD ["nginx", "-g", "daemon off;"]
```

**Варианты ответа:**
- Установка минимальной версии Nginx для уменьшения размера базового образа
- Уменьшение размера образа, удаление архива после его извлечения в Dockerfile
- Замена `ADD` на `COPY` для копирования файлов и распаковка архива заранее
- Использование `ADD` для всех операций, так как это универсальная инструкция
- Перемещение `CMD` перед инструкцией `ADD` для ускорения сборки

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/advanced.md)

**Правильный ответ:** Замена `ADD` на `COPY` для копирования файлов и распаковка архива заранее

#### Ответ от Claude

Инструкция `ADD` умеет не только копировать файлы, но и автоматически распаковывать локальные tar-архивы прямо в образ. Это означает, что `ADD /app.tar.gz ...` распакует архив внутрь слоя, но все промежуточные файлы из архива могут остаться в образе, раздувая его размер.

Правильное решение: распаковать архив заранее (до сборки образа, на хост-машине) и затем использовать `COPY` для копирования уже готовых файлов. `COPY` делает ровно то, что написано — копирует файлы без дополнительной логики. Это даёт:
- Предсказуемое поведение (нет «магии» автоматической распаковки).
- Меньший размер образа: в слой попадают только нужные файлы.
- Ускорение сборки: не тратится время на распаковку внутри сборщика.

Источники: [Dockerfile reference — ADD vs COPY](https://docs.docker.com/reference/dockerfile/#add); [Best practices — ADD or COPY](https://docs.docker.com/build/building/best-practices/#add-or-copy).

### Вопрос 3. В вашем Docker Compose файле для одного из сервисов включена настройка `healthcheck`, чтобы отслеживать его состояние. Однако вы замечаете, что зависимые сервисы запускаются сразу, игнорируя состояние `healthcheck`. Это приводит к тому, что они не могут подключиться к сервису, пока тот полностью не инициализирован. Вы проверяете файл и видите, что секция `depends_on` указана, но дополнительные параметры отсутствуют. Какой недостаток в конфигурации может это вызывать?

**Категория:** Docker (HH) · **Уровень:** Advanced

**Варианты ответа:**
- Наличие дополнительного сервиса в секции `volumes`
- Неправильная конфигурация секции `healthcheck`
- Отсутствие лимитов ресурсов в сервисах
- Отсутствие параметра `condition` в зависимости `depends_on`
- Использование устаревшей версии Docker Compose файла

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/advanced.md)

**Правильный ответ:** Отсутствие параметра `condition` в зависимости `depends_on`

#### Ответ от Claude

По умолчанию `depends_on` гарантирует лишь порядок запуска контейнеров (зависимый сервис стартует после указанного), но не ожидает его готовности. Чтобы зависимый сервис ждал успешного прохождения `healthcheck`, необходимо явно указать условие:

```yaml
depends_on:
  db:
    condition: service_healthy
```

Доступные значения `condition`:
- `service_started` — поведение по умолчанию (просто запуск).
- `service_healthy` — ждёт, пока `healthcheck` сервиса вернёт «healthy».
- `service_completed_successfully` — ждёт нормального завершения (для одноразовых задач).

Источник: [Compose file reference — depends_on](https://docs.docker.com/compose/how-tos/startup-order/); [Compose Specification — depends_on](https://docs.docker.com/reference/compose-file/services/#depends_on).

### Вопрос 4. Ваше приложение использует два контейнера: API и Redis. Они должны быть изолированы от других контейнеров в системе, но при этом поддерживать прямое взаимодействие между собой. Вы настроили отдельную сеть, но контейнеры не могут подключиться друг к другу. Почему возникла проблема?

**Категория:** Docker (HH) · **Уровень:** Advanced

**Варианты ответа:**
- Для контейнеров не настроены алиасы в сети, что мешает их взаимодействию через DNS
- Поскольку контейнеры не подключены к одной и той же сети, они не могут взаимодействовать друг с другом
- Контейнеры подключены к разным `bridge`-сетям, что мешает их взаимодействию
- Один из контейнеров использует параметр `--network=host`, нарушая изоляцию сети и взаимодействие с другими контейнерами
- В секции `networks` отсутствует явное указание имени сети, что мешает контейнерам подключаться к одной сети

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/advanced.md)

**Правильный ответ:** Поскольку контейнеры не подключены к одной и той же сети, они не могут взаимодействовать друг с другом

#### Ответ от Claude

В Docker контейнеры могут взаимодействовать друг с другом только в том случае, если они находятся в одной сети. Даже если сеть создана командой `docker network create`, контейнеры сами по себе к ней не присоединяются — нужно явно подключить каждый контейнер к этой сети через `--network <имя_сети>` при запуске или через `docker network connect`. При использовании Docker Compose оба сервиса должны быть перечислены под одной пользовательской сетью.

Внутри одной пользовательской bridge-сети Docker обеспечивает встроенный DNS: контейнеры могут обращаться друг к другу по имени сервиса (или имени контейнера). Алиасы — опциональная функция и для базового взаимодействия не требуются.

Источник: [Docker networking overview](https://docs.docker.com/engine/network/); [Use bridge networks](https://docs.docker.com/engine/network/drivers/bridge/).

Когда войти в контейнер нельзя, а проблема носит сетевой характер, правильная стратегия — использовать команды инспекции без изменения состояния контейнера:

- `docker network inspect <network_name>` — показывает все подключённые контейнеры, их IP-адреса, драйвер сети, настройки IPAM. Позволяет убедиться, что контейнер подключён к правильным сетям.

- `docker logs` и `docker cp /etc/hosts` не дадут информацию о сетевых задержках.

### Вопрос 6. Контейнер запущен в пользовательском пространстве имён (`userns-remap`), но при этом всё ещё может изменять состояние ядра Linux (например, менять параметры ядра через `sysctl`). Как это возможно?

**Категория:** Docker (HH) · **Уровень:** Advanced

**Варианты ответа:**
- Применена политика `--security-opt apparmor=unconfined`
- Параметр `--privileged` был применён частично, что оставило некоторые возможности
- Контейнер запущен с параметром `--cap-add=CAP_SYS_ADMIN`
- Контейнер использует хостовый `cgroup` (`--cgroupns=host`)
- Использован `--network host`, который дает доступ к управлению сетью и ядром

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/advanced.md)

**Правильный ответ:** Контейнер запущен с параметром `--cap-add=CAP_SYS_ADMIN`

#### Ответ от Claude

`userns-remap` перемапирует UID/GID процессов внутри контейнера, так что «root» внутри контейнера соответствует непривилегированному пользователю на хосте. Это значительно снижает риски, но **не отменяет** Linux capabilities, явно добавленные при запуске.

`CAP_SYS_ADMIN` — одна из наиболее мощных capabilities: она предоставляет доступ к широкому набору системных операций, в том числе к изменению параметров ядра через `sysctl`. Даже при включённом `userns-remap`, если контейнер запущен с `--cap-add=CAP_SYS_ADMIN`, процесс внутри получает эту возможность.

Остальные варианты:
- `apparmor=unconfined` снимает ограничения AppArmor, но само по себе не даёт `sysctl`-доступа без соответствующей capability.
- `--privileged` «частично» не существует — флаг либо применён полностью, либо нет.
- `--cgroupns=host` касается пространства имён cgroups, а не capabilities ядра.
- `--network host` даёт доступ к сетевому стеку хоста, но не к `sysctl` общесистемных параметров.

Источники: [Docker security — Linux capabilities](https://docs.docker.com/engine/security/#linux-kernel-capabilities); [docker run --cap-add](https://docs.docker.com/reference/cli/docker/container/run/#cap-add).

### Вопрос 7. В вашей системе используются общие тома Docker для хранения данных базы данных и логов. Недавно вы столкнулись с проблемами: (1) несколько томов были повреждены после неожиданного завершения работы контейнеров; (2) данные базы данных оказались частично утерянными, а логи не обновлялись; (3) резервное копирование настроено, но занимает слишком много времени. Что вызвало эти проблемы?

**Категория:** Docker (HH) · **Уровень:** Advanced

**Варианты ответа:**
- Отсутствие регулярного тестирования восстановления из резервных копий
- Неоптимальная конфигурация политики резервного копирования
- Отсутствие настроек автоматического создания снапшотов томов
- Использование томов без проверки доступности при запуске контейнеров
- Использование локальных томов без отказоустойчивого хранилища

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/advanced.md)

**Правильный ответ:** Использование локальных томов без отказоустойчивого хранилища

#### Ответ от Claude

Локальные тома Docker (`driver: local`) хранят данные на файловой системе хоста. При внезапном завершении контейнера (сбой питания, OOM-kill, `docker kill`) буферизованные данные, не успевшие записаться на диск, могут быть потеряны или повреждены — особенно если сама СУБД не успела сделать checkpoint.

Кроме того, локальное хранилище:
- Не реплицируется: данные существуют только на одном хосте.
- Не предоставляет встроенной защиты от отказов диска.
- Резервное копирование всего тома может занимать долгое время из-за большого объёма данных без инкрементальных снапшотов.

Решение — использовать отказоустойчивые распределённые системы хранения (NFS, Ceph, Amazon EFS, GlusterFS) с поддержкой репликации и снапшотов, либо volume-плагины с соответствующими возможностями.

Источник: [Docker volumes](https://docs.docker.com/engine/storage/volumes/); [Use a volume driver](https://docs.docker.com/engine/storage/volumes/#use-a-volume-driver).

### Вопрос 8. Вы настраивали систему мониторинга для контейнерных приложений, но столкнулись с проблемой, что уведомления о повышенной нагрузке (например, превышение предела CPU или памяти) не всегда приходят своевременно. Почему это происходит?

**Категория:** Docker (HH) · **Уровень:** Advanced

**Варианты ответа:**
- Grafana используется для постоянного мониторинга, но не настроена система для отправки уведомлений, а используется только визуализация данных
- Используется Docker CLI для логирования и анализа, но логи ежедневно просматриваются вручную
- Используется cAdvisor, но не настроены автоматические уведомления на основе собранных метрик, что приводит к отсутствию мгновенных оповещений
- Настроен Prometheus Alertmanager, но не учтено, что правила уведомлений могут быть некорректно настроены для контейнерных ресурсов
- Используется команда `docker logs` для мониторинга контейнеров, но при этом уведомления не настроены, так как система мониторинга настроена только для ручного анализа логов

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/advanced.md)

**Правильный ответ:** Настроен Prometheus Alertmanager, но не учтено, что правила уведомлений могут быть некорректно настроены для контейнерных ресурсов

#### Ответ от Claude

Prometheus Alertmanager — правильный инструмент для отправки уведомлений, однако сами правила алертинга (`alerting rules`) должны быть корректно адаптированы под метрики контейнеров. Типичные проблемы:

- Неверные имена метрик: для контейнеров cAdvisor предоставляет метрики вида `container_cpu_usage_seconds_total` и `container_memory_usage_bytes`, а не общесистемные `node_cpu_*`. Использование неправильных метрик в PromQL-выражениях даст пустой результат, и алерт никогда не сработает.
- Некорректные пороговые значения или временные окна (`for:` в правиле).
- Ошибка в лейблах фильтрации (например, пропущен фильтр `container!="POD"`).

Остальные варианты описывают другие конфигурации (без Alertmanager), не совпадающие с описанной ситуацией.

Источники: [Prometheus Alerting rules](https://prometheus.io/docs/prometheus/latest/configuration/alerting_rules/); [cAdvisor metrics](https://github.com/google/cadvisor/blob/master/docs/storage/prometheus.md).

### Вопрос 10. Что безопаснее: использование инструкции `USER` в Dockerfile или параметра `-u <user>` при запуске контейнера?

**Категория:** Docker (HH) · **Уровень:** Advanced

**Варианты ответа:**
- `USER` в Dockerfile, так как это позволяет избежать необходимости указывать пользователя при каждом запуске
- `-u <user>` при запуске, так как это позволяет гибко менять пользователя без изменения Dockerfile
- `USER` в Dockerfile, так как это обеспечивает безопасность на этапе сборки и предотвращает запуск контейнера от `root`
- `-u <user>` при запуске, так как это повышает производительность контейнера
- `USER` в Dockerfile, так как это упрощает управление контейнерами и снижает вероятность ошибок

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/advanced.md)

**Правильный ответ:** `USER` в Dockerfile, так как это обеспечивает безопасность на этапе сборки и предотвращает запуск контейнера от `root`

#### Ответ от Claude

Инструкция `USER` в Dockerfile задаёт пользователя, от имени которого будут выполняться последующие инструкции `RUN`, `CMD` и `ENTRYPOINT`. Это означает:

- Безопасность встроена в образ: куда бы образ ни был развёрнут, контейнер запустится от непривилегированного пользователя по умолчанию.
- Нет риска забыть указать `-u` при `docker run` — человеческий фактор исключён.
- Принцип наименьших привилегий соблюдается на уровне артефакта (образа).

Параметр `-u` при запуске переопределяет пользователя только для конкретного запуска контейнера. Это удобно для отладки, но не является надёжной мерой безопасности: если `-u` не указан, контейнер запустится от пользователя, заданного в образе (или от root, если `USER` отсутствует).

Флаг `--privileged` при `docker run` даёт контейнеру почти все Linux capabilities, доступ к устройствам хоста и снимает ограничения seccomp/AppArmor. Фактически контейнер получает уровень доступа, близкий к root на хосте. Это объясняет все три наблюдаемые симптома:

### Вопрос 12. Почему `overlay`-сеть в Docker лучше подходит для связи контейнеров на разных хостах?

**Категория:** Docker (HH) · **Уровень:** Advanced

**Варианты ответа:**
- Оптимизирована для связи контейнеров между разными хостами, снижая задержки
- Позволяет контейнерам использовать IP-адреса хоста, сильно ускоряя к ним доступ
- Автоматически масштабирует все сетевые ресурсы по мере возрастания нагрузки
- Использует частичное сжатие трафика, за счет этого уменьшает нагрузку на сеть
- Обеспечивает полную изоляцию контейнеров, блокируя любой внешний трафик

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/advanced.md)

**Правильный ответ:** Оптимизирована для связи контейнеров между разными хостами, снижая задержки

#### Ответ от Claude

Overlay-сеть создаёт виртуальную распределённую сеть поверх физической инфраструктуры, используя технологию VXLAN (Virtual Extensible LAN) для инкапсуляции трафика. Ключевые особенности:

- Контейнеры на разных хостах получают IP-адреса из одного подсети и «видят» друг друга как будто они в одной локальной сети.
- Docker использует встроенный распределённый key-value store (при Swarm mode — встроенный Raft) для хранения топологии сети.
- Трафик шифруется (опционально, через `--opt encrypted`) при передаче между хостами.

Это принципиальное отличие от bridge-сети, которая работает только в пределах одного хоста. Для multi-host коммуникации overlay — стандартный выбор в Docker Swarm.

Замечание: формулировка «снижая задержки» в варианте ответа несколько условна — overlay добавляет небольшой overhead на инкапсуляцию VXLAN по сравнению с прямым host-networking. Правильный смысл: overlay специально спроектирована для межхостовой связи контейнеров.

Источник: [Use overlay networks](https://docs.docker.com/engine/network/drivers/overlay/).

### Вопрос 13. Вы развернули приложение на Docker, которое выполняет обработку большого количества сетевых запросов. Однако вы замечаете увеличенные задержки в сетевых операциях. Логи показывают, что стандартная сеть Docker имеет высокую задержку при обработке запросов. Это снижает производительность приложения, особенно при высокой нагрузке. Какой способ поможет поднять производительность и снизить задержки?

**Категория:** Docker (HH) · **Уровень:** Advanced

**Варианты ответа:**
- Добавить лимиты на количество подключений для каждого контейнера
- Ограничить доступ контейнеров к внешним API
- Настроить репликацию контейнеров для обработки сетевых запросов
- Использовать пользовательскую сеть `overlay` для оптимизации сетевого взаимодействия
- Произвести замену текущей сети на bridge-сеть Docker

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/advanced.md)

**Правильный ответ:** Использовать пользовательскую сеть `overlay` для оптимизации сетевого взаимодействия

#### Ответ от Claude

⚠️ HH отмечает `overlay`-сеть как ответ, однако с технической точки зрения этот ответ требует уточнения.

Стандартная bridge-сеть Docker (которая «имеет высокую задержку» по условию задачи) использует NAT и userspace-маршрутизацию, что добавляет overhead. Для снижения задержек на **одном хосте** наиболее эффективным является `--network host`, который позволяет контейнеру использовать сетевой стек хоста напрямую, минуя Docker NAT.

Тем не менее, в контексте вопроса (если речь о Swarm-окружении с контейнерами на разных узлах), пользовательская overlay-сеть является правильным выбором по сравнению со стандартной bridge-сетью. Для single-host сценария с минимальными задержками следует рассматривать `host`-сеть.

### Вопрос 14. Какое преимущество использования секции `secrets` перед передачей данных через общие тома (volumes)?

**Категория:** Docker (HH) · **Уровень:** Advanced

**Варианты ответа:**
- Тома требуют настройки сетевых политик для каждого контейнера
- Секреты автоматически кэшируются для ускорения запуска контейнеров
- Секреты можно изменять динамически без перезапуска сервисов
- Тома не поддерживают права доступа только для чтения
- Docker управляет жизненным циклом секретов и ограничивает их доступность

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/advanced.md)

**Правильный ответ:** Docker управляет жизненным циклом секретов и ограничивает их доступность

#### Ответ от Claude

Docker Secrets предоставляет специализированный механизм для работы с конфиденциальными данными (паролями, токенами, ключами):

- **Управление жизненным циклом**: секреты создаются, обновляются и удаляются через API Docker/Swarm; при удалении секрета из сервиса он автоматически убирается из контейнеров.
- **Ограниченная доступность**: секрет доступен только тем сервисам, которым он явно назначен, и монтируется как файл в `/run/secrets/<secret_name>` внутри контейнера с правами 0400 (только для чтения владельцем).
- **Шифрование в transit**: в Docker Swarm секреты передаются по зашифрованному каналу и хранятся в зашифрованном Raft-хранилище.

В отличие от томов:
- Том доступен любому контейнеру, к которому он примонтирован, без дополнительного контроля доступа.
- Данные в томе не шифруются по умолчанию.
- Жизненный цикл тома и содержимого не связан с жизненным циклом сервиса.

Источник: [Manage sensitive data with Docker secrets](https://docs.docker.com/engine/swarm/secrets/).

### Вопрос 15. Ваше приложение в Docker Swarm использует Docker Secrets для передачи конфиденциальных данных. При добавлении новых секретов вы заметили, что старый секрет перестает быть доступным сервису, который его использует. Что нужно сделать, чтобы исправить это?

**Категория:** Docker (HH) · **Уровень:** Advanced

**Варианты ответа:**
- Перезапустить сервис, чтобы он получил доступ к новым и старым секретам
- Добавить все секреты в общий том, доступный для сервиса
- Удалить новые секреты и проверить конфигурацию Docker Secrets
- Обновить настройки ротации секретов для всех сервисов
- Сохранить старые секреты как переменную окружения для временного доступа

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/advanced.md)

**Правильный ответ:** Перезапустить сервис, чтобы он получил доступ к новым и старым секретам

#### Ответ от Claude

В Docker Swarm секреты монтируются в контейнер при его запуске. Когда вы обновляете конфигурацию сервиса (добавляете новый секрет через `docker service update --secret-add ...`), Swarm выполняет rolling update — перезапускает задачи (задачи-контейнеры) сервиса по очереди. Только что запущенные контейнеры получают доступ ко всем секретам, назначенным сервису на момент запуска.

Если старый секрет перестал быть доступен, вероятная причина: при обновлении сервиса старый секрет был случайно убран из списка секретов сервиса, либо обновление конфигурации не было применено корректно. Перезапуск сервиса с правильной конфигурацией (с явным перечислением всех нужных секретов) решает проблему.

Обратите внимание: секреты в Docker Swarm **нельзя изменить** после создания — только создать новый секрет и обновить сервис для использования нового.

Источник: [Manage sensitive data with Docker secrets — Rotate a secret](https://docs.docker.com/engine/swarm/secrets/#example-rotate-a-secret).
