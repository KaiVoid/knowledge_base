### Вопрос 1. В процессе разработки вы часто тестируете разные версии приложения в контейнерах. Ваша задача — создать и сразу запустить новый контейнер из указанного образа, чтобы он выполнил команду `/start.sh`. Какую команду нужно использовать для выполнения этой задачи?

**Категория:** Docker (HH) · **Уровень:** Basic

**Варианты ответа:**
- `docker pull image-name /start.sh`
- `docker run image-name /start.sh`
- `docker start image-name /start.sh`
- `docker build image-name /start.sh`
- `docker create image-name /start.sh`

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/basic.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/basic.md)

**Правильный ответ:** `docker run image-name /start.sh`

#### Ответ от Claude

`docker run` — единственная команда, которая **создаёт** новый контейнер из образа и **немедленно его запускает**. Аргумент после имени образа (`/start.sh`) переопределяет команду по умолчанию, заданную в Dockerfile (`CMD` или `ENTRYPOINT`).

Остальные варианты:
- `docker pull` — загружает образ из реестра, не создаёт контейнер.
- `docker start` — запускает уже существующий (остановленный) контейнер по его имени или ID, не по имени образа.
- `docker build` — собирает образ из Dockerfile, не запускает контейнер.
- `docker create` — создаёт контейнер, но не запускает его (нужен последующий `docker start`).

Источник: [docker run](https://docs.docker.com/reference/cli/docker/container/run/).

### Вопрос 2. Каково ключевое различие между командами `docker exec` и `docker attach` при взаимодействии с работающим контейнером?

**Категория:** Docker (HH) · **Уровень:** Basic

**Варианты ответа:**
- `docker exec` удаляет выполняемый образ, а `docker attach` подключает существующий и работающий контейнер к сети
- `docker exec` завершает выполнение контейнера, а `docker attach` продолжает работу существующего и работающего контейнера
- `docker exec` перезапускает основной процесс контейнера, а `docker attach` лишь выводит логи его выполнения
- `docker exec` создает новый контейнер, а `docker attach` использует уже существующий работающий контейнер
- `docker exec` запускает новую команду внутри контейнера, а `docker attach` подключается к текущему процессу контейнера

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/basic.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/basic.md)

**Правильный ответ:** `docker exec` запускает новую команду внутри контейнера, а `docker attach` подключается к текущему процессу контейнера

#### Ответ от Claude

Ключевое различие:

- **`docker exec <container> <command>`** — запускает **новый процесс** внутри работающего контейнера. Наиболее частый пример: `docker exec -it mycontainer bash` — открывает новую интерактивную оболочку. Завершение этого процесса не останавливает контейнер.

- **`docker attach <container>`** — подключает стандартные потоки (stdin/stdout/stderr) терминала к **уже запущенному основному процессу** контейнера (PID 1). Если нажать `Ctrl+C`, сигнал SIGINT будет отправлен главному процессу, что может привести к остановке контейнера. Для отключения без остановки нужна последовательность `Ctrl+P, Ctrl+Q`.

Практический вывод: для интерактивной отладки обычно используют `docker exec -it <container> bash`, а не `docker attach`, так как это безопаснее.

Источники: [docker exec](https://docs.docker.com/reference/cli/docker/container/exec/); [docker attach](https://docs.docker.com/reference/cli/docker/container/attach/).

### Вопрос 3. Вы хотите развернуть систему мониторинга, включающую три сервиса: (1) агент сбора метрик; (2) сервис обработки метрик; (3) интерфейс визуализации. #### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/basic.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/basic.md)

**Правильный ответ:** Разделить каждый сервис на отдельную сеть и настроить межсетевые маршруты так, чтобы они ограничили взаимодействие между агентом и интерфейсом визуализации

#### Ответ от Claude

Для реализации требований — сервис обработки доступен агенту и интерфейсу визуализации, но агент не имеет прямого доступа к интерфейсу — используется разделение на сети:

```yaml
services:
  agent:
    image: metrics-agent:latest
    networks:
      - collection-network

  processor:
    image: metrics-processor:latest
    networks:
      - collection-network
      - visualization-network

  visualizer:
    image: metrics-ui:latest
    ports:
      - "4000:4000"
    networks:
      - visualization-network

networks:
  collection-network:
  visualization-network:
```

В этой конфигурации:
- `agent` и `processor` в одной сети (`collection-network`) — данные передаются.
- `processor` и `visualizer` в другой сети (`visualization-network`) — визуализация получает данные.
- `agent` и `visualizer` в **разных сетях** — прямого взаимодействия нет.

Источник: [Compose networking](https://docs.docker.com/compose/how-tos/networking/).

### Вопрос 4. Какую функцию выполняет инструкция `ADD` в Dockerfile?

**Категория:** Docker (HH) · **Уровень:** Basic

**Варианты ответа:**
- Копирует файлы из хоста в контейнер
- Указывает команду, которая добавляет метаданные в образ
- Выполняет команду во время сборки образа
- Устанавливает переменные окружения для процесса сборки
- Указывает команду, которая будет выполнена при запуске контейнера

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/basic.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/basic.md)

**Правильный ответ:** Копирует файлы из хоста в контейнер

#### Ответ от Claude

Инструкция `ADD` копирует файлы и директории из контекста сборки (хоста) в файловую систему образа. Это её базовая функция, общая с `COPY`.

Дополнительные возможности `ADD` сверх `COPY`:
- Автоматическая распаковка локальных tar-архивов (`.tar`, `.tar.gz`, `.tar.bz2` и др.) в целевую директорию внутри образа.
- Загрузка файлов по URL (не рекомендуется: плохо взаимодействует с кешем, безопаснее использовать `RUN curl`).

Остальные варианты соответствуют другим инструкциям Dockerfile:
- Метаданные в образ — `LABEL`.
- Выполнение команды во время сборки — `RUN`.
- Переменные окружения при сборке — `ENV` или `ARG`.
- Команда при запуске контейнера — `CMD` или `ENTRYPOINT`.

Источник: [Dockerfile reference — ADD](https://docs.docker.com/reference/dockerfile/#add).

### Вопрос 5. Какая главная причина использования многослойных образов при создании Dockerfile?

**Категория:** Docker (HH) · **Уровень:** Basic

**Варианты ответа:**
- Снижают нагрузку на сам процессор при сборке образа
- Позволяют повторно использовать кэш для ускорения сборки
- Обеспечивают совместимость контейнеров между разными ОС
- Поддерживают большее количество инструкций в Dockerfile
- Автоматически обновляют все зависимости при изменении кода

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/basic.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/basic.md)

**Правильный ответ:** Позволяют повторно использовать кэш для ускорения сборки

#### Ответ от Claude

Каждая инструкция в Dockerfile (`FROM`, `COPY`, `RUN`, `ENV` и т.д.) создаёт отдельный слой (layer) в образе. Docker кеширует каждый слой по отдельности. При повторной сборке:

- Если инструкция и все предыдущие слои не изменились, Docker берёт слой из кеша, не пересчитывая его.
- Только слои начиная с первого изменённого пересобираются заново.

Это означает, что правильно упорядоченный Dockerfile (сначала редко меняющиеся инструкции — установка зависимостей, потом часто меняющиеся — копирование кода) собирается значительно быстрее при итеративной разработке.

Остальные варианты неверны:
- Слои не снижают нагрузку на CPU при первичной сборке.
- Совместимость между ОС обеспечивает базовый образ (`FROM`), а не слоистая структура.
- Количество инструкций в Dockerfile ничем не ограничено.
- Автоматическое обновление зависимостей Docker не выполняет.

Источник: [Docker Build cache](https://docs.docker.com/build/cache/); [Dockerfile layers](https://docs.docker.com/build/guide/layers/).

### Вопрос 6. Вы разрабатываете микросервис, который должен быть упакован в минимальный Docker-образ. Требуется использовать официальный базовый образ и обеспечить, чтобы приложение автоматически компилировалось и запускалось в контейнере. Приложение не использует внешние зависимости. Выберите ответ, в котором перечислены все инструкции, которые обязательно нужно включить.

**Категория:** Docker (HH) · **Уровень:** Basic

**Варианты ответа:**
- `FROM`, `RUN` и `CMD`
- `WORKDIR` и `ENV`
- `RUN` и `CMD`
- `FROM`, `COPY`, `RUN` и `CMD`
- `FROM` и `WORKDIR`

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/basic.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/basic.md)

**Правильный ответ:** `FROM`, `COPY`, `RUN` и `CMD`

#### Ответ от Claude

Для микросервиса, который нужно скомпилировать и запустить, минимальный набор инструкций:

- **`FROM`** — обязательна; задаёт базовый образ (например, `FROM golang:1.22-alpine`). Без неё Dockerfile недействителен.
- **`COPY`** — копирует исходный код приложения в образ для последующей компиляции.
- **`RUN`** — выполняет команду компиляции во время сборки образа (например, `RUN go build -o /app ./...`).
- **`CMD`** — задаёт команду по умолчанию, которая запускает скомпилированное приложение при старте контейнера.

Почему остальные варианты неверны:
- `FROM` + `RUN` + `CMD` без `COPY` — невозможно скомпилировать код, которого нет в образе.
- `WORKDIR` + `ENV` — недостаточно; без `FROM` Dockerfile вообще не работает.
- `RUN` + `CMD` без `FROM` — синтаксическая ошибка; `FROM` обязательна.
- `FROM` + `WORKDIR` — нет ни копирования кода, ни компиляции, ни команды запуска.

Источник: [Dockerfile reference](https://docs.docker.com/reference/dockerfile/); [Writing a Dockerfile](https://docs.docker.com/get-started/docker-concepts/building-images/writing-a-dockerfile/).

### Вопрос 7. Как секция `services` в Docker Compose файле влияет на конфигурацию и взаимодействие контейнеров в многоконтейнерной среде?

**Категория:** Docker (HH) · **Уровень:** Basic

**Варианты ответа:**
- Задает настройки сети между контейнерами, обеспечивая их взаимодействие
- Определяет переменные окружения для хоста, влияя на его поведение
- Указывает список контейнеров, их конфигурацию и взаимосвязи, что позволяет обеспечить совместную работу
- Перечисляет доступные образы, упрощая выбор для запуска контейнеров
- Создает шаблоны для автоматического запуска новых контейнеров, оптимизируя их масштабирование

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/basic.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/basic.md)

**Правильный ответ:** Указывает список контейнеров, их конфигурацию и взаимосвязи, что позволяет обеспечить совместную работу

#### Ответ от Claude

Секция `services` — центральная часть Docker Compose файла. Каждый элемент в ней описывает один сервис (контейнер) и его конфигурацию:

```yaml
services:
  web:
    image: nginx:latest
    ports:
      - "80:80"
    depends_on:
      - db
  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: secret
```

В секции `services` задаются:
- Образ (`image`) или инструкция сборки (`build`).
- Порты (`ports`), тома (`volumes`), переменные окружения (`environment`).
- Зависимости между сервисами (`depends_on`).
- Сети (`networks`), лимиты ресурсов (`deploy.resources`), политика перезапуска (`restart`).

Именно совокупность этих параметров и определяет «конфигурацию и взаимосвязи» контейнеров. Настройки сетей и образы описываются в отдельных верхнеуровневых секциях (`networks`, `volumes`), а `services` ссылается на них.

Источник: [Compose file reference — services](https://docs.docker.com/reference/compose-file/services/).

### Вопрос 8. Вы настраиваете среду разработки для команды. Приложение включает три контейнера: API-сервер, кэш и база данных. При этом: (1) все три контейнера должны запускаться и останавливаться одновременно; (2) кэш должен использовать постоянное хранилище, чтобы данные сохранялись между перезапусками; (3) API-сервер должен автоматически перезапускаться при сбое. Как настроить Docker Compose для реализации этих требований?

**Категория:** Docker (HH) · **Уровень:** Basic

**Варианты ответа:**
- Создать Docker Compose файл с секцией `networks` и подключить все три контейнера к одной сети
- Использовать команду `docker-compose up` для запуска контейнеров и команду `docker-compose down` для остановки контейнеров
- Использовать параметр `build` в Docker Compose файле для всех контейнеров, чтобы обеспечить их одновременный синхронный запуск
- Добавить параметр `depends_on` для кэша и базы данных, чтобы запускать эти три контейнера перед API-сервером
- Создать файл с секциями `services` для всех контейнеров, добавить `volumes` для кэша и установить параметр `restart: always` для API-сервера

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/basic.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/basic.md)

**Правильный ответ:** Создать файл с секциями `services` для всех контейнеров, добавить `volumes` для кэша и установить параметр `restart: always` для API-сервера

#### Ответ от Claude

Полная конфигурация, закрывающая все три требования:

```yaml
services:
  api:
    image: myapi:latest
    restart: always        # требование 3: автоперезапуск при сбое

  cache:
    image: redis:latest
    volumes:
      - cache-data:/data   # требование 2: постоянное хранилище

  db:
    image: postgres:16
    volumes:
      - db-data:/var/lib/postgresql/data

volumes:
  cache-data:
  db-data:
```

Требование 1 (одновременный запуск/остановка) реализуется самим Docker Compose: `docker-compose up` запускает все сервисы из файла, `docker-compose down` останавливает их.

`restart: always` перезапускает контейнер при любом завершении (включая сбои и ручной `docker stop`). Если нужно перезапускать только при ошибках — используйте `restart: on-failure`.

Источник: [Compose file reference — restart](https://docs.docker.com/reference/compose-file/services/#restart); [Volumes in Compose](https://docs.docker.com/reference/compose-file/volumes/).

### Вопрос 9. Вы разрабатываете приложение из 3 микросервисов, каждый запускается в отдельном контейнере. Необходимо, чтобы контейнеры могли свободно обмениваться данными между собой, но при этом оставались изолированными от сети хоста по соображениям безопасности. В будущем нужно добавить новые контейнеры в эту же сеть без изменения существующей конфигурации. Какую команду и тип сети вы выберете?

**Категория:** Docker (HH) · **Уровень:** Basic

**Варианты ответа:**
- Тип сети `none` и команду `docker run --net=none` для полной изоляции контейнеров
- Тип сети `custom` и команду `docker network connect` для подключения контейнеров к сети
- Тип сети `macvlan` и команду `docker network create` для создания сети с прямым доступом к физическому интерфейсу хоста
- Тип сети `host` и команду `docker run --net=host` для запуска контейнеров
- Тип сети `bridge` и команду `docker network create` для создания новой сети

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/basic.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/basic.md)

**Правильный ответ:** Тип сети `bridge` и команду `docker network create` для создания новой сети

#### Ответ от Claude

Пользовательская bridge-сеть — оптимальный выбор для данного сценария:

```bash
# Создать изолированную пользовательскую bridge-сеть
docker network create --driver bridge app-network

# Запустить контейнеры микросервисов в этой сети
docker run -d --name service1 --network app-network service1:latest
docker run -d --name service2 --network app-network service2:latest
docker run -d --name service3 --network app-network service3:latest

# Добавить новый контейнер без изменения конфигурации существующих
docker run -d --name service4 --network app-network service4:latest
```

Преимущества пользовательской bridge-сети:
- Контейнеры изолированы от сети хоста и от других сетей Docker.
- Встроенный DNS: контейнеры обращаются друг к другу по имени.
- Новые контейнеры подключаются к сети при запуске без изменения уже работающих.

Почему остальные неверны:
- `none` — полная изоляция, контейнеры не смогут общаться друг с другом.
- `custom` — не существующий тип сети Docker.
- `macvlan` — даёт контейнерам прямой доступ к физической сети (противоположность изоляции от хоста).
- `host` — контейнер разделяет сеть с хостом, нет изоляции.

Источник: [Use bridge networks](https://docs.docker.com/engine/network/drivers/bridge/); [docker network create](https://docs.docker.com/reference/cli/docker/network/create/).

### Вопрос 10. Какие основные функции выполняет Docker Registry для управления Docker-образами?

#### Ответ от Claude

Docker Registry — это сервер для **хранения и распространения Docker-образов**. Его роль в экосистеме Docker:

- **Хранение**: образы хранятся в виде слоёв (layers), что позволяет экономить место при наличии общих базовых слоёв.
- **Распространение**: предоставляет API для `docker push` (загрузка образа в реестр) и `docker pull` (скачивание образа из реестра).
- **Версионирование**: поддерживает теги (`:latest`, `:1.0`, `:production`), позволяя хранить несколько версий образа.

Публичный реестр Docker — Docker Hub (`hub.docker.com`). Также существуют приватные реестры: GitHub Container Registry, AWS ECR, Google Artifact Registry, собственный `registry:2`.

Остальные варианты — функции других инструментов:
- Создание образов — Dockerfile + `docker build`.
- Мониторинг контейнеров — Prometheus, cAdvisor, `docker stats`.
- Безопасность контейнеров — Docker Security (capabilities, seccomp, AppArmor).
- Управление конфигурациями — Docker Compose, Kubernetes ConfigMaps, Vault.

Источник: [CNCF Distribution Registry — Overview](https://distribution.github.io/distribution/); [Docker Hub](https://docs.docker.com/docker-hub/).
