### Вопрос 1. Вы запускаете команду `docker pull my_registry/my_image:latest`, но образ не загружается, и выводится ошибка. В результате другие контейнеры, которые зависят от этого образа, не могут быть запущены, что вызывает сбои в работе приложения. Какая причина могла привести к этой ошибке?

**Категория:** Docker (HH) · **Уровень:** Intermediate

**Варианты ответа:**
- Неправильный тег образа в команде, который не соответствует имеющимся в реестре версиям
- Ограниченные сетевые настройки Docker, запрещающие взаимодействие с внешними реестрами
- Неправильные данные авторизации при доступе к приватному реестру, что заблокировало загрузку образа
- Отсутствие соединения с интернетом на сервере, из-за чего команда `pull` не выполняется
- Использование неверного флага `--auth` для авторизации в реестре

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/intermediate.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/intermediate.md)

**Правильный ответ:** Неправильные данные авторизации при доступе к приватному реестру, что заблокировало загрузку образа

#### Ответ от Claude

Реестр `my_registry/...` является приватным (нестандартный домен вместо `docker.io`). Обращение к приватному реестру требует предварительной аутентификации через `docker login <registry>`. Если учётные данные не введены, отсутствуют или устарели, Docker вернёт ошибку вида `unauthorized: authentication required` или `denied: access forbidden`.

Остальные варианты менее вероятны в данном сценарии:
- Неправильный тег дал бы ошибку `manifest unknown`, а не проблему с авторизацией.
- Отсутствие интернета дало бы `connection refused` или `timeout`, не специфичную для авторизации.
- Флага `--auth` у команды `docker pull` не существует — это несуществующий вариант.

Для исправления: выполнить `docker login my_registry` с корректными именем пользователя и паролем/токеном.

Источник: [docker login](https://docs.docker.com/reference/cli/docker/login/); [docker pull](https://docs.docker.com/reference/cli/docker/image/pull/).

### Вопрос 2. Вы работаете над Dockerfile для создания образа микросервисного приложения. Во время тестирования вы обнаружили следующие проблемы: размер образа слишком велик и его загрузка занимает много времени; во время работы контейнера возникают ошибки, связанные с отсутствием необходимых пакетов; сборка занимает значительно больше времени из-за повторной установки зависимостей. Что поможет исправить все эти ошибки?

**Категория:** Docker (HH) · **Уровень:** Intermediate

**Варианты ответа:**
- Установка зависимостей во время запуска контейнера, а не во время его сборки
- Добавление команды для удаления временных файлов и кеша после установки зависимостей
- Использование промежуточного образа multi-stage build для установки и сборки всех компонентов и анализ списка устанавливаемых пакетов
- Объединение всех инструкций `RUN` в один слой для минимизации числа слоев
- Установление всех зависимостей в одном слое Dockerfile для уменьшения количества слоев и анализ списка устанавливаемых пакетов

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/intermediate.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/intermediate.md)

**Правильный ответ:** Использование промежуточного образа multi-stage build для установки и сборки всех компонентов и анализ списка устанавливаемых пакетов

#### Ответ от Claude

Multi-stage build позволяет решить все три проблемы одновременно:

1. **Уменьшение размера образа.** В финальный образ попадают только артефакты (скомпилированное приложение), а не инструменты сборки, компиляторы и build-зависимости. Например, Go-приложение весом 10 МБ собирается в образе с компилятором (~600 МБ), а финальный образ базируется на `scratch` или `alpine` и содержит только бинарник.

2. **Устранение отсутствующих пакетов.** Анализ списка устанавливаемых пакетов помогает явно определить все runtime-зависимости и включить их в финальный образ.

3. **Ускорение повторных сборок.** Слои с зависимостями, которые меняются редко, кешируются Docker и не пересобираются при изменении кода приложения.

Пример структуры multi-stage build:
```dockerfile
FROM maven:3.9 AS builder
COPY . /app
RUN mvn -f /app/pom.xml clean package -DskipTests

FROM eclipse-temurin:21-jre-alpine
COPY --from=builder /app/target/app.jar /app.jar
CMD ["java", "-jar", "/app.jar"]
```

Источник: [Multi-stage builds](https://docs.docker.com/build/building/multi-stage/).

### Вопрос 3. Вы используете следующий Docker Compose файл. При запуске контейнеров сервис `api` не может подключиться к `redis`, а в логах сервиса `api` фиксируются ошибки соединения с сетью. Что поможет решить эту проблему?

**Категория:** Docker (HH) · **Уровень:** Intermediate

```yaml
version: "3.8"
services:
  api:
    image: myapi:latest
    environment:
      API_KEY: key123
  redis:
    image: redis:latest
```

**Варианты ответа:**
- Удаление настройки `environment` из сервиса `api` и подключение сервисов к одной сети
- Смена версии Compose файла на 2.1 и подключение обоих сервисов к одной сети
- Добавление секции `networks` в Compose файл и подключение сервисов к одной сети
- Перезапуск Compose файла с флагом `--force-recreate` и подключение двух сервисов к одной сети
- Добавить переменные окружения для сервиса `redis` и подключение обоих сервисов к одной сети

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/intermediate.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/intermediate.md)

**Правильный ответ:** Добавление секции `networks` в Compose файл и подключение сервисов к одной сети

#### Ответ от Claude

В данном Compose-файле у сервисов не задана явная сеть. Начиная с Docker Compose версии 2, если сети не указаны явно, Compose автоматически создаёт сеть `default` и подключает к ней все сервисы в файле. Это должно обеспечивать взаимодействие между `api` и `redis` по имени сервиса.

Однако если проблема воспроизводится, рекомендуется явно задать сеть:

```yaml
version: "3.8"
services:
  api:
    image: myapi:latest
    environment:
      API_KEY: key123
    networks:
      - app-network
  redis:
    image: redis:latest
    networks:
      - app-network

networks:
  app-network:
    driver: bridge
```

Явное определение сети делает конфигурацию понятнее, исключает неочевидное поведение и позволяет задавать дополнительные параметры (алиасы, driver options).

Источник: [Compose networking](https://docs.docker.com/compose/how-tos/networking/); [Compose file reference — networks](https://docs.docker.com/reference/compose-file/networks/).

### Вопрос 4. Вы настраиваете тестовую среду для приложения, которое использует контейнеры для веб-сервера и Redis. Требуется создать такую конфигурацию, чтобы веб-сервер и Redis могли взаимодействовать друг с другом, но при этом Redis был полностью изолирован от всех остальных контейнеров. Как это сделать?

**Категория:** Docker (HH) · **Уровень:** Intermediate

**Варианты ответа:**
- Использовать тип сети `none` для Redis и тип сети `bridge` для веб-сервера
- Использовать тип сети `host` и запустить Redis с помощью команды `docker run --net=host`
- Использовать тип сети `bridge` и добавить правило `firewall` для ограничения доступа к Redis
- Создать сеть `overlay` и подключить только веб-сервер к этой сети
- Использовать тип сети `bridge` и подключить оба контейнера к одной сети с помощью `docker network connect`

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/intermediate.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/intermediate.md)

**Правильный ответ:** Использовать тип сети `bridge` и подключить оба контейнера к одной сети с помощью `docker network connect`

#### Ответ от Claude

Пользовательская bridge-сеть в Docker обеспечивает сетевую изоляцию: контейнеры в ней видят только друг друга и не доступны из других пользовательских сетей (если явно не подключены к ним). Алгоритм:

```bash
# Создать отдельную изолированную сеть
docker network create app-network

# Запустить Redis и веб-сервер, подключив к этой сети
docker run -d --name redis --network app-network redis:latest
docker run -d --name webserver --network app-network mywebserver:latest

# Либо подключить к сети уже запущенный контейнер
docker network connect app-network redis
```

Внутри `app-network` контейнеры обращаются друг к другу по имени контейнера (`redis`, `webserver`). Контейнеры в других сетях не имеют прямого доступа к Redis.

Остальные варианты неверны:
- `none` полностью изолирует Redis от всех сетей, в том числе от веб-сервера.
- `host` устраняет сетевую изоляцию полностью.
- Правила firewall добавляют сложность и ненадёжны при частой смене контейнеров.
- `overlay` предназначена для multi-host окружений.

Источник: [Use bridge networks](https://docs.docker.com/engine/network/drivers/bridge/); [docker network connect](https://docs.docker.com/reference/cli/docker/network/connect/).

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/intermediate.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/intermediate.md)

**Правильный ответ:** Использовать `docker stats` для получения информации об использовании ресурсов, `docker inspect` для анализа лимитов и `docker cp` для копирования конфигурационного файла

#### Ответ от Claude

Набор из трёх команд покрывает все три подзадачи:

1. **`docker stats <container>`** — выводит в реальном времени метрики потребления CPU, памяти (используется / лимит), сетевого I/O и блочного I/O. Для однократного снимка: `docker stats --no-stream <container>`.

Все три команды неразрушающие и не прерывают работу контейнера.

Источники: [docker stats](https://docs.docker.com/reference/cli/docker/container/stats/); [docker inspect](https://docs.docker.com/reference/cli/docker/container/inspect/); [docker cp](https://docs.docker.com/reference/cli/docker/container/cp/).

### Вопрос 6. Вы запустили локальный Docker Registry, но обнаружили, что соединения осуществляются только через HTTP. Это вызывает обеспокоенность по поводу безопасности, особенно в случае работы с конфиденциальными данными. Вы решили настроить HTTPS, чтобы защитить передаваемую информацию. - Отключить доступ к Registry из внешней сети, чтобы исключить риски
- Настроить сертификаты SSL/TLS и подключить их к Docker Registry
- Настроить Firewall для защиты соединений через HTTP
- Использовать параметр `--insecure-registry` для доступа через HTTP
- Добавить параметр `--net=host`, чтобы исключить использование HTTP

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/intermediate.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/intermediate.md)

**Правильный ответ:** Настроить сертификаты SSL/TLS и подключить их к Docker Registry

#### Ответ от Claude

Для перехода с HTTP на HTTPS в локальном Docker Registry необходимо:

1. Получить или сгенерировать SSL/TLS сертификат (для production — от доверенного CA, для локальной разработки — самоподписанный через `openssl`).
2. Передать сертификат и ключ в контейнер Registry через переменные окружения:

```bash
docker run -d \

  -e REGISTRY_HTTP_TLS_KEY=/certs/domain.key \
  -v /path/to/certs:/certs \
  registry:2
```

Остальные варианты неверны:
- Отключение внешнего доступа не решает проблему незашифрованной передачи данных внутри сети.
- Firewall не шифрует трафик.
- `--insecure-registry` — прямо противоположное решение, оно разрешает небезопасный HTTP на стороне клиента Docker.
- `--net=host` меняет режим сети контейнера, но не добавляет TLS.

### Вопрос 7. Контейнер `cache-server` работает нестабильно, но его нужно безопасно завершить с сохранением состояния. Задачи: (1) завершить контейнер корректно, дав приложениям время закрыться; (2) сохранить его файловую систему в виде нового образа; (3) перезапустить новый контейнер на основе этого состояния. Какими командами это можно сделать?

**Категория:** Docker (HH) · **Уровень:** Intermediate

**Варианты ответа:**
- Использовать `docker kill` для немедленного завершения, затем `docker save` для сохранения контейнера и `docker analyze` для его анализа
- Завершить работу контейнера через `docker stop`, затем сохранить состояние с помощью `docker commit` и запустить новый контейнер через `docker run`
- Использовать `docker pause`, затем `docker commit` и потом `docker unpause` для восстановления состояния контейнера
- Настроить `restart=always` и перезапустить контейнер через `docker start`
- Войти через `docker exec` в контейнер, остановить процессы вручную и создать дамп командой `docker export`

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/intermediate.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/intermediate.md)

**Правильный ответ:** Завершить работу контейнера через `docker stop`, затем сохранить состояние с помощью `docker commit` и запустить новый контейнер через `docker run`

#### Ответ от Claude

Последовательность команд для трёх задач:

```bash
# 1. Корректное завершение: отправляет SIGTERM, ждёт (по умолчанию 10 сек), затем SIGKILL
docker stop cache-server

# 2. Сохранение состояния файловой системы в новый образ
docker commit cache-server cache-server-snapshot:v1

# 3. Запуск нового контейнера на основе сохранённого состояния
docker run -d --name cache-server-new cache-server-snapshot:v1
```

Почему именно этот набор:
- `docker stop` — в отличие от `docker kill` (посылает SIGKILL немедленно), посылает SIGTERM и ждёт корректного завершения.
- `docker commit` — создаёт новый образ из текущего состояния файловой системы контейнера (включая все изменения поверх базового образа).
- `docker run` — запускает новый контейнер из сохранённого образа.

Важное замечание: `docker commit` фиксирует файловую систему, но не данные в volumes. Если критические данные хранятся в томах — их нужно резервировать отдельно.

Источники: [docker stop](https://docs.docker.com/reference/cli/docker/container/stop/); [docker commit](https://docs.docker.com/reference/cli/docker/container/commit/).

### Вопрос 8. Какой способ подключения связанного тома для сохранения данных контейнера НЕ поможет для успешной работы с ним?

**Категория:** Docker (HH) · **Уровень:** Intermediate

**Варианты ответа:**
- Использование тома, созданного через `docker volume create`, с последующей его привязкой к контейнеру с флагом `-v`
- Использование временной файловой системы `--tmpfs`, где данные хранятся в оперативной памяти
- Добавление параметра `VOLUME` в Dockerfile для автоматического создания тома без явной привязки
- Применение bind-монтирования с `--mount type=bind`, привязывая директорию хоста к контейнеру
- Использование сети для хранения данных через NFS или CIFS, монтируя их в контейнер

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/intermediate.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/intermediate.md)

**Правильный ответ:** Использование временной файловой системы `--tmpfs`, где данные хранятся в оперативной памяти

#### Ответ от Claude

`--tmpfs` монтирует **временную файловую систему в оперативной памяти** контейнера. Данные в `tmpfs` существуют только пока контейнер запущен и полностью теряются при его остановке или перезапуске. Это диаметрально противоположно задаче «сохранения данных».

Все остальные варианты обеспечивают персистентность данных:
- **`docker volume create` + `-v`**: именованный том хранится на хосте в управляемом Docker каталоге и переживает перезапуск/удаление контейнера.
- **`VOLUME` в Dockerfile**: автоматически создаёт анонимный том при запуске контейнера (данные сохраняются на хосте).
- **`--mount type=bind`**: привязывает директорию хоста — данные напрямую живут на файловой системе хоста.
- **NFS/CIFS**: сетевое хранилище, данные хранятся на удалённом сервере.

`--tmpfs` имеет своё применение: временные файлы, чувствительные данные, которые не должны попасть на диск (например, секреты в памяти).

Источник: [Volumes](https://docs.docker.com/engine/storage/volumes/); [tmpfs mounts](https://docs.docker.com/engine/storage/tmpfs/).

### Вопрос 9. Вы администрируете систему, где контейнер Nginx используется для хранения логов. Вам нужно регулярно сохранять их копии в отдельную директорию на хосте, чтобы избежать потери данных при сбое контейнера. Также важно, чтобы процесс резервного копирования не прерывал работу Nginx и оставался полностью автоматизированным. Выберите подходящий метод.

**Категория:** Docker (HH) · **Уровень:** Intermediate

**Варианты ответа:**
- Сохранять логи внутри файловой системы контейнера Nginx без подключения тома
- Настроить задачу `cron` для автоматического копирования логов из контейнера на хост
- Подключить том Docker к контейнеру Nginx и настроить синхронизацию с хостовой системой
- Настроить контейнер Nginx в режиме `read-only` для полной защиты данных пользователей
- Использовать параметр `--restart=no` для контейнера Nginx, чтобы периодически его перезапускать

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/intermediate.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/intermediate.md)

**Правильный ответ:** Подключить том Docker к контейнеру Nginx и настроить синхронизацию с хостовой системой

#### Ответ от Claude

Правильный подход — использовать Docker Volume (или bind mount) для директории с логами Nginx:

```bash
docker run -d \
  --name nginx \
  -v /host/logs/nginx:/var/log/nginx \
  nginx:latest
```

После этого логи пишутся напрямую в директорию `/host/logs/nginx` на хосте. Для автоматизированного резервного копирования можно настроить `rsync`, `logrotate` или любой другой инструмент на хосте — они будут работать с обычной директорией файловой системы, не затрагивая контейнер.

Остальные варианты неверны:
- Хранение логов внутри контейнера: при сбое или удалении контейнера все логи теряются.
- `cron` + `docker cp`: копирование через API Docker менее эффективно и зависит от работающего контейнера.
- `read-only` режим запрещает запись и в директорию логов тоже — Nginx не сможет писать логи.
- `--restart=no` + периодический перезапуск: прерывает работу Nginx и не решает задачу сохранения логов.

Источник: [Use volumes](https://docs.docker.com/engine/storage/volumes/); [Bind mounts](https://docs.docker.com/engine/storage/bind-mounts/).

### Вопрос 10. Какой open-source инструмент подходит для автоматического сбора метрик и имеет удобный и полный интерфейс для анализа?

**Категория:** Docker (HH) · **Уровень:** Intermediate

**Варианты ответа:**
- Prometheus
- Telegraf
- Logstash
- Elasticsearch
- Grafana

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/intermediate.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/intermediate.md)

**Правильный ответ:** Prometheus

#### Ответ от Claude

⚠️ HH отмечает Prometheus, однако этот вопрос неоднозначен: Prometheus специализируется на **сборе метрик** (scraping, хранение time-series), но его встроенный веб-интерфейс (PromUI) весьма минималистичен и не считается «удобным и полным интерфейсом для анализа». Для визуализации Prometheus обычно используют совместно с **Grafana**.

Тем не менее, среди предложенных вариантов Prometheus является наиболее близким к условию:
- **Telegraf** — агент сбора метрик (часть стека InfluxDB), не имеет UI для анализа.
- **Logstash** — инструмент обработки логов (не метрик), часть ELK-стека.
- **Elasticsearch** — поисковый движок для хранения и поиска данных, не специализируется на метриках.
- **Grafana** — инструмент **визуализации** метрик, но сам по себе не собирает метрики (требует источника данных).

Prometheus = сбор метрик + хранение + базовый UI + язык запросов PromQL. В связке с Grafana образует полноценный стек мониторинга.

Источник: [Prometheus overview](https://prometheus.io/docs/introduction/overview/); [Collect Docker metrics with Prometheus](https://docs.docker.com/engine/daemon/prometheus/).

### Вопрос 11. Вы анализируете производительность микросервисного приложения в Docker-контейнерах с использованием Grafana и Elastic Stack. Один из сервисов начал работать медленнее, чем ожидалось, а логи показывают регулярные таймауты. Мониторинг фиксирует высокие задержки при взаимодействии с базой данных: трафик идет через виртуальные интерфейсы bridge-сети Docker и NAT, что дает дополнительную прослойку. Что произойдет, если заменить стандартную сеть Docker `bridge` на `host`?

**Категория:** Docker (HH) · **Уровень:** Intermediate

**Варианты ответа:**
- Проблема лимита подключений к базе данных останется, так как она не связана с типом сети Docker
- Таймауты при взаимодействии с базой данных могут сохраняться, если проблема связана с ограничением ресурсов хоста
- Задержки при взаимодействии с базой данных уменьшатся, так как сеть `host` упрощает маршрутизацию
- Скорость взаимодействия между контейнерами повысится, так как сеть `host` устраняет часть сетевых накладных расходов
- Пропадет возможность изолировать сеть контейнеров, так как сеть `host` напрямую использует сетевые интерфейсы хоста

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/intermediate.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/intermediate.md)

**Правильный ответ:** Пропадет возможность изолировать сеть контейнеров, так как сеть `host` напрямую использует сетевые интерфейсы хоста

#### Ответ от Claude

При использовании `--network host` контейнер разделяет сетевой стек с хостом: он использует те же сетевые интерфейсы, IP-адреса и порты. Это означает:

- **Нет сетевой изоляции**: контейнер виден в сети хоста напрямую, все его порты доступны без явного маппинга `-p`.
- **Нет NAT**: трафик идёт без преобразования адресов, что действительно снижает latency.
- **Конфликты портов**: два контейнера в `host`-сети не могут слушать один и тот же порт.

Ответ HH правилен в том, что главное последствие — **потеря сетевой изоляции**. Это верно и важно с точки зрения безопасности.

Дополнительный контекст: переход на `host`-сеть действительно может снизить задержки (убирает NAT и bridge-overhead), но ценой изоляции. Для production-систем это компромисс, который нужно оценивать явно.

Источник: [Use host networking](https://docs.docker.com/engine/network/drivers/host/); [Network drivers overview](https://docs.docker.com/engine/network/drivers/).

### Вопрос 12. Вы создаете пайплайн CI/CD с использованием Docker и GitLab CI. На этапе развертывания требуется развернуть контейнеры в тестовой среде. Как Docker упрощает этот процесс?

**Категория:** Docker (HH) · **Уровень:** Intermediate

**Варианты ответа:**
- Создает контейнеры из одного образа, минимизируя расхождения в окружениях
- Исключает необходимость тестирования приложений
- Исключает настройку сетей между контейнерами
- Позволяет использовать общие образы для всех этапов пайплайна
- Автоматически развертывает приложения в продакшн

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — docker/intermediate.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/docker/intermediate.md)

**Правильный ответ:** Создает контейнеры из одного образа, минимизируя расхождения в окружениях

#### Ответ от Claude

Ключевое преимущество Docker в CI/CD — **воспроизводимость среды**: один и тот же Docker-образ используется на всех этапах пайплайна (сборка, юнит-тесты, интеграционные тесты, деплой в тестовую среду). Это гарантирует, что тестирование происходит в точно такой же среде, в которой приложение будет работать в production.

Расхождения в окружениях («у меня работает, в тестах — нет») — один из наиболее частых источников проблем в CI/CD, и Docker решает эту проблему фундаментально.

Остальные варианты ошибочны:
- Docker не исключает необходимость тестирования — он улучшает его качество.
- Сети между контейнерами по-прежнему нужно настраивать (через Compose или оркестратор).
- Автоматический деплой в production — задача оркестратора (Kubernetes, Swarm), а не самого Docker.

Источник: [Use Docker with GitLab CI/CD](https://docs.gitlab.com/ee/ci/docker/using_docker_images.html); [Docker CI/CD](https://docs.docker.com/build/ci/).
