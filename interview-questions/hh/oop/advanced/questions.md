### Вопрос 1. Какое определение наиболее точно отражает понятие инициализации класса в ООП?

**Категория:** OOP (HH) · **Уровень:** Advanced

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — oop/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/oop/advanced.md)

**Правильный ответ:** Процесс присвоения начальных значений полям данных класса

#### Ответ от Claude

Инициализация класса — это процесс, при котором полям объекта (или самого класса) присваиваются начальные значения. В Java инициализация делится на два этапа: инициализацию класса (выполнение статических инициализаторов и присвоение начальных значений статическим полям) и инициализацию экземпляра (выполнение конструктора и инициализаторов экземпляра). Таким образом, наиболее точное определение — «присвоение начальных значений полям данных класса», что отличает инициализацию от простого объявления поля или выделения памяти.

Источники: [Oracle Java Tutorial — Object Lifecycle](https://docs.oracle.com/javase/tutorial/java/javaOO/objectcreation.html); [JLS §12.4 Initialization of Classes and Interfaces](https://docs.oracle.com/javase/specs/jls/se21/html/jls-12.html#jls-12.4).

### Вопрос 2. Какое из перечисленных действий нарушает принцип инкапсуляции в ООП?

**Категория:** OOP (HH) · **Уровень:** Advanced

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — oop/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/oop/advanced.md)

**Правильный ответ:** Непосредственное изменение приватных полей класса извне без использования методов класса

#### Ответ от Claude

Инкапсуляция — один из четырёх фундаментальных принципов ООП — предполагает сокрытие внутреннего состояния объекта и предоставление доступа к нему только через контролируемый публичный интерфейс (методы). Прямое изменение приватных полей извне (например, через рефлексию или нарушение модификаторов доступа) нарушает этот принцип, так как внешний код получает неограниченный доступ к реализации, что делает класс хрупким при изменениях. Правильный подход — использование getter/setter-методов или других методов публичного API.

Источники: [Oracle Java Tutorial — Encapsulation](https://docs.oracle.com/javase/tutorial/java/concepts/); [Refactoring Guru — Инкапсуляция поля](https://refactoring.guru/ru/encapsulate-field).

### Вопрос 3. Какой шаблон проектирования НЕ основан на реализации принципа инкапсуляции ООП?

**Категория:** OOP (HH) · **Уровень:** Advanced

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — oop/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/oop/advanced.md)

**Правильный ответ:** Информационный эксперт

#### Ответ от Claude

«Информационный эксперт» (Information Expert) — это принцип из GRASP (General Responsibility Assignment Software Patterns), а не шаблон проектирования GoF. Он определяет, какому объекту следует назначить ответственность: тому, кто обладает наибольшим количеством необходимой информации. Этот принцип не строится на механизме инкапсуляции (сокрытии данных), а скорее описывает распределение обязанностей на основе знания о данных. Классические паттерны GoF, основанные на инкапсуляции, — это, например, Стратегия (Strategy), Состояние (State) и Заместитель (Proxy).

Источники: [Refactoring Guru — Паттерны проектирования](https://refactoring.guru/ru/design-patterns); Craig Larman, «Applying UML and Patterns», §17 — GRASP: Information Expert.

### Вопрос 4. В каком из следующих случаев соблюдён принцип открытости/закрытости?

**Категория:** OOP (HH) · **Уровень:** Advanced

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — oop/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/oop/advanced.md)

**Правильный ответ:** Класс Shape определяет абстрактный метод calculateArea(), и новые геометрические фигуры могут быть созданы путём наследования от Shape и реализации этого метода

#### Ответ от Claude

Принцип открытости/закрытости (Open/Closed Principle, OCP) из SOLID гласит: программные сущности должны быть **открыты для расширения** и **закрыты для изменения**. Описанная ситуация полностью соответствует OCP: базовый класс `Shape` с абстрактным методом `calculateArea()` не изменяется при добавлении новых фигур — каждая новая фигура реализует метод через наследование, расширяя систему без модификации существующего кода. Это классическое применение OCP через наследование и полиморфизм.

Источники: [Refactoring Guru — Open/Closed Principle](https://refactoring.guru/ru/solid/open-closed-principle); Robert C. Martin, «Clean Architecture», глава 8 — OCP.

### Вопрос 5. Какой модификатор доступа следует использовать, чтобы обеспечить доступность полей и методов класса из самого класса, из производного класса и из других классов в той же сборке, но не из других сборок?

**Категория:** OOP (HH) · **Уровень:** Advanced

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — oop/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/oop/advanced.md)

**Правильный ответ:** Protected Internal

#### Ответ от Claude

Модификатор `protected internal` существует в **C#** и обеспечивает доступ: из самого класса, из производных классов, а также из любого кода в той же сборке (assembly). Этого нет в Java — в Java понятие «сборка» отсутствует, и ближайший аналог — `protected`, который открывает доступ классу, подклассам и классам того же пакета.

Вопрос сформулирован в контексте общего ООП и подразумевает C#. В Java подобная комбинация в виде единого модификатора не реализована.

Источники: [Microsoft Docs — protected internal (C#)](https://learn.microsoft.com/en-us/dotnet/csharp/language-reference/keywords/protected-internal); [Oracle Java Tutorial — Controlling Access to Members](https://docs.oracle.com/javase/tutorial/java/javaOO/accesscontrol.html).

### Вопрос 6. Если несколько интерфейсов, реализованных абстрактным классом, содержат одинаковые методы, то…

**Категория:** OOP (HH) · **Уровень:** Advanced

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — oop/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/oop/advanced.md)

**Правильный ответ:** … это не вызывает конфликта: класс предоставляет единственную реализацию этого метода, и она будет удовлетворять обоим интерфейсам

#### Ответ от Claude

В Java, если абстрактный класс реализует два интерфейса, содержащих метод с одинаковой сигнатурой, конфликта не возникает: класс предоставляет одну реализацию метода, которая удовлетворяет требованиям обоих интерфейсов одновременно. Это принципиальное отличие от «проблемы ромба» (diamond problem) при множественном наследовании реализации, которой в Java нет. В Java 8+ конфликт с `default`-методами интерфейсов разрешается явным переопределением в классе, но для абстрактного класса единственная реализация метода закрывает требования всех реализуемых интерфейсов.

Источники: [Oracle Java Tutorial — Interfaces](https://docs.oracle.com/javase/tutorial/java/IandI/index.html); [JLS §9.4.1 Inheritance and Overriding](https://docs.oracle.com/javase/specs/jls/se21/html/jls-9.html#jls-9.4.1).

### Вопрос 7. В какой из ситуаций ниже корректно применить статические методы класса?

**Категория:** OOP (HH) · **Уровень:** Advanced

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — oop/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/oop/advanced.md)

**Правильный ответ:** Когда нужно обеспечить доступ к методам без создания объекта

#### Ответ от Claude

Статические методы принадлежат классу, а не его экземплярам, и могут вызываться без создания объекта (`ClassName.method()`). Это делает их подходящими для утилитарных операций, которые не зависят от состояния экземпляра (например, `Math.sqrt()`, `Collections.sort()`). Статические методы также используются в паттерне «Фабричный метод» (static factory methods) для управления созданием экземпляров. Использование статических методов для логики, требующей обращения к полям экземпляра, является ошибкой проектирования.

Источники: [Oracle Java Tutorial — Understanding Instance and Class Members](https://docs.oracle.com/javase/tutorial/java/javaOO/classvars.html); Joshua Bloch, «Effective Java», 3rd ed., Item 1 — Consider static factory methods instead of constructors.

### Вопрос 8. Какое утверждение верно?

**Категория:** OOP (HH) · **Уровень:** Advanced

**Варианты ответа:**
- В Java дочерний класс может переопределить конструктор без параметров родительского класса
- В C++ наследование позволяет дочернему классу переопределить методы родительского класса

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — oop/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/oop/advanced.md)

**Правильный ответ:** В Java дочерний класс может переопределить конструктор без параметров родительского класса

#### Ответ от Claude

⚠️ **Ответ HH фактически неверен.** В Java конструкторы **не наследуются** и, следовательно, **не переопределяются**. Дочерний класс не наследует конструкторы родителя, а создаёт свои собственные. Он может вызвать конструктор родителя через `super(...)`, но это не переопределение. Согласно JLS §8.8: «Constructors are not members and are not inherited.»

Второй вариант — «В C++ наследование позволяет дочернему классу переопределить методы родительского класса» — является истинным утверждением: в C++ виртуальные методы переопределяются в дочерних классах через механизм `virtual` / `override`.

Таким образом, оба варианта сформулированы с инверсией истинности. По всей видимости, HH-квиз содержит ошибку в этом вопросе.

Источники: [JLS §8.8 Constructor Declarations](https://docs.oracle.com/javase/specs/jls/se21/html/jls-8.html#jls-8.8); [Oracle Java Tutorial — Constructors](https://docs.oracle.com/javase/tutorial/java/javaOO/constructors.html).

### Вопрос 9. Динамический полиморфизм достигается в основном с помощью…

**Категория:** OOP (HH) · **Уровень:** Advanced

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — oop/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/oop/advanced.md)

**Правильный ответ:** Виртуальных методов

#### Ответ от Claude

Динамический полиморфизм (полиморфизм времени выполнения, runtime polymorphism) — это способность вызывать переопределённый метод на основе фактического типа объекта, а не типа переменной. Он реализуется через виртуальные методы и механизм диспетчеризации (virtual dispatch). В C++ методы явно объявляются `virtual`; в Java все нестатические и непривативные методы неявно виртуальны — JVM использует таблицу виртуальных методов (vtable) для разрешения вызовов во время выполнения. Это отличает динамический полиморфизм от статического (перегрузка методов), который разрешается на этапе компиляции.

Источники: [Oracle Java Tutorial — Polymorphism](https://docs.oracle.com/javase/tutorial/java/IandI/polymorphism.html); Bjarne Stroustrup, «The C++ Programming Language», §20.3 — Virtual Functions.

### Вопрос 10. Выберите НЕПОДХОДЯЩИЙ вариант продолжения тезиса: «Снижения сильной связанности классов в иерархии наследования можно достигнуть с помощью…»

**Категория:** OOP (HH) · **Уровень:** Advanced

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — oop/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/oop/advanced.md)

**Правильный ответ:** Использования модификаторов доступа

#### Ответ от Claude

Снижение связанности (low coupling) в иерархии наследования достигается через: программирование на интерфейсах, использование паттернов «Стратегия» и «Компоновщик» вместо глубокого наследования, внедрение зависимостей (Dependency Injection). Модификаторы доступа (`private`, `protected`, `public`) управляют видимостью членов и являются инструментом инкапсуляции, а не снижения связанности между классами в иерархии. Они не уменьшают зависимость дочернего класса от родительского — дочерний класс всё равно остаётся зависим от реализации родителя.

Источники: [Refactoring Guru — Опутыватели связями (Couplers)](https://refactoring.guru/ru/smells/couplers); Robert C. Martin, «Clean Architecture», глава 15 — What is Architecture?

### Вопрос 11. Выберите верное суждение о сериализации с наследованием.

**Категория:** OOP (HH) · **Уровень:** Advanced

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — oop/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/oop/advanced.md)

**Правильный ответ:** В процессе сериализации с наследованием может быть использован механизм виртуальных функций

#### Ответ от Claude

При сериализации объектов в языках с поддержкой виртуальных функций (C++, C#) механизм виртуальной диспетчеризации может использоваться для полиморфной сериализации — когда метод сериализации определён в базовом классе как виртуальный и переопределён в подклассах для учёта их специфических полей. В Java стандартный механизм сериализации (`java.io.Serializable`) также задействует иерархию наследования: если родительский класс не реализует `Serializable`, его поля не сериализуются, и необходимо реализовать методы `writeObject`/`readObject` или использовать Externalizable для контроля над процессом.

Источники: [Oracle Java Object Serialization](https://docs.oracle.com/javase/8/docs/platform/serialization/spec/serial-arch.html); [Oracle Tutorial — Serializable Objects](https://docs.oracle.com/javase/tutorial/jndi/objects/serial.html).

### Вопрос 12. Какое из утверждений об особенностях объектно-ориентированных языков программирования верно?

**Категория:** OOP (HH) · **Уровень:** Advanced

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — oop/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/oop/advanced.md)

**Правильный ответ:** Множественное наследование может быть реализовано в Python и С++

#### Ответ от Claude

Python и C++ поддерживают множественное наследование классов: дочерний класс может напрямую наследоваться от нескольких родительских классов. В Python используется алгоритм MRO (Method Resolution Order, C3-линеаризация) для разрешения конфликтов. В C++ порядок обхода базовых классов определяется при объявлении. Java не поддерживает множественное наследование классов (только классов — интерфейсы можно реализовывать множественно), что является одним из ключевых архитектурных решений языка для избежания «проблемы ромба».

Источники: [Python docs — Multiple Inheritance](https://docs.python.org/3/tutorial/classes.html#multiple-inheritance); [Oracle Java Tutorial — Why Java doesn't support multiple inheritance](https://docs.oracle.com/javase/tutorial/java/IandI/multipleinheritance.html).

**Правильный ответ:** Класс — эльф, объект — маг, атрибуты — запас здоровья (HP), запас магии (мана), методы — заклинания, которые использует маг

### Вопрос 14. Выберите верное суждение о компонентно-ориентированном программировании.

**Категория:** OOP (HH) · **Уровень:** Advanced

#### Оригинальный ответ из интернета
> Источник: [HH Skill Verification — oop/advanced.md](https://github.com/Londeren/hh-skill-verifications-quizzes/blob/main/oop/advanced.md)

**Правильный ответ:** Основная идея концепции — создание программных компонентов, которые могут быть использованы в разных приложениях

#### Ответ от Claude

Компонентно-ориентированное программирование (Component-Based Software Engineering, CBSE) — это парадигма, при которой система строится из самодостаточных, повторно используемых программных компонентов с чётко определёнными интерфейсами. Ключевая идея — возможность использования одного и того же компонента в разных приложениях без изменения его кода. Примеры: JavaBeans в Java EE, компоненты в Spring Framework, Angular-компоненты в frontend. Компоненты отличаются от классов ООП более высоким уровнем абстракции и явным определением предоставляемых и требуемых интерфейсов.

Источники: [Robert C. Martin, «Clean Architecture», глава 12 — Components](https://www.oreilly.com/library/view/clean-architecture-a/9780134494272/); [Oracle Java Tutorial — JavaBeans](https://docs.oracle.com/javase/tutorial/javabeans/).

**Правильный ответ:** Для работы с объектно-ориентированными базами данных можно использовать Java, Python и Ruby

#### Ответ от Claude

Объектно-ориентированные базы данных (ООСУБД) хранят данные в виде объектов, аналогично ООП. Языки Java, Python и Ruby имеют библиотеки и ORM-инструменты для работы с такими базами (например, db4o для Java, ZODB для Python). Остальные варианты ложны: реляционные БД имеют иную модель данных, а не «более гибкую и масштабируемую»; объект в ООСУБД — не строка таблицы (это описание реляционной модели); ООСУБД поддерживают как одинарное, так и множественное наследование; типы связей в ООСУБД не ограничены «один к одному» и «один ко многим».

Источники: [Wikipedia — Object Database](https://en.wikipedia.org/wiki/Object_database); [Python ZODB documentation](https://zodb.org/en/latest/).
