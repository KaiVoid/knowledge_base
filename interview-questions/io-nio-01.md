### Вопрос 1. Чем отличается байтовый поток (InputStream/OutputStream) от символьного (Reader/Writer)? Когда следует использовать каждый из них?

**Категория:** IO/NIO · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Character Streams (The Java Tutorials — Essential Java Classes > Basic I/O)](https://docs.oracle.com/javase/tutorial/essential/io/charstreams.html)

Платформа Java хранит символьные значения по конвенции Unicode. Character-stream I/O автоматически переводит этот внутренний формат в и из локального набора символов. На западных локалях локальный набор символов обычно является 8-битным расширением ASCII.

Символьные потоки содержат 16-битные Unicode-символы, а не 8-битные байты. Все классы символьных потоков реализованы через `Reader` и `Writer`. Как и байтовые потоки, существуют символьные потоки, специализированные для файлового ввода-вывода: `FileReader` и `FileWriter`. Пример, аналогичный байтовому `CopyBytes`, но работающий с символами: `CopyCharacters` — принципиальное отличие лишь в том, что `FileReader` и `FileWriter` используются вместо `FileInputStream` и `FileOutputStream`.

Символьные потоки часто являются «обёртками» над байтовыми: `FileReader` использует `FileInputStream` внутри, `FileWriter` — `FileOutputStream`. Символьный поток выполняет трансляцию кодировки, байтовый — физический ввод-вывод.

Следует использовать байтовые потоки только для самого примитивного ввода-вывода. Для работы с текстом предпочтительны символьные потоки.

#### Ответ от Claude

**Фундаментальное различие**

| Критерий | Байтовые потоки (`InputStream`/`OutputStream`) | Символьные потоки (`Reader`/`Writer`) |
|---|---|---|
| Единица данных | 8-битный байт (`int` 0–255) | 16-битный символ Unicode (`char`) |
| Кодировка | Не выполняется — передаются сырые байты | Автоматическая трансляция Unicode ↔ charset |
| Назначение | Бинарные данные | Текстовые данные |
| Базовые классы | `InputStream`, `OutputStream` | `Reader`, `Writer` |

**Байтовые потоки** читают и пишут данные байт за байтом через метод `read()` / `write(int b)`. Они не имеют представления о кодировке и не пытаются интерпретировать содержимое. Это нижний уровень: именно байтовые потоки взаимодействуют с операционной системой напрямую.

**Символьные потоки** надстроены над байтовыми: `FileReader` внутри использует `FileInputStream`, но добавляет слой декодирования байт → символ по заданной или платформенной кодировке. Метод `read()` возвращает `int`, соответствующий символу Unicode (0–65535).

**Когда использовать каждый**

- Байтовые потоки (`InputStream`/`OutputStream`) — при работе с **бинарными данными**: изображениями, аудио, видео, сериализованными объектами, архивами, сетевыми протоколами, шифрованием. Единственное правило: если данные не являются текстом — используйте байтовые потоки.
- Символьные потоки (`Reader`/`Writer`) — при работе с **текстом**: чтение/запись `.txt`, `.xml`, `.json`, `.csv`, консольный ввод-вывод, обработка строк. Они корректно обрабатывают многобайтовые кодировки (UTF-8, UTF-16) и автоматически адаптируются под локаль.

**Важная деталь:** Используя `FileReader` без явного указания кодировки, вы получаете зависимость от платформенного charset (`Charset.defaultCharset()`). Начиная с Java 11 предпочтительнее явно указывать кодировку:

```java
// Безопасно — явная кодировка
new FileReader("file.txt", StandardCharsets.UTF_8)          // Java 11+
new InputStreamReader(new FileInputStream("file.txt"), StandardCharsets.UTF_8)  // любая версия
```

### Вопрос 2. Какие классы позволяют преобразовать байтовый поток в символьный и обратно? Как они устроены?

**Категория:** IO/NIO · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [InputStreamReader (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/io/InputStreamReader.html)

`InputStreamReader` — это мост от байтовых потоков к символьным: он читает байты и декодирует их в символы, используя заданный charset. Charset может быть указан по имени, задан явно, либо используется платформенный charset по умолчанию. Каждый вызов одного из методов `read()` у `InputStreamReader` может вызвать чтение одного или нескольких байт из нижележащего байтового потока. Для достижения максимальной эффективности рекомендуется оборачивать `InputStreamReader` в `BufferedReader`:

```java
BufferedReader in = new BufferedReader(new InputStreamReader(System.in));
```

Иерархия класса:
```
java.lang.Object
    java.io.Reader
        java.io.InputStreamReader
            java.io.FileReader
```

Прямой известный подкласс: `FileReader`.

#### Ответ от Claude

**Классы-мосты**

Java предоставляет два класса-моста (bridge) для двустороннего преобразования:

| Направление | Класс | Наследует от |
|---|---|---|
| байт → символ | `InputStreamReader` | `Reader` |
| символ → байт | `OutputStreamWriter` | `Writer` |

**`InputStreamReader` — байт → символ**

Обёртывает `InputStream` и декодирует байты в символы (`char`) через заданный `Charset`. Внутри содержит `StreamDecoder` (Sun-внутренний класс), который буферирует байты и применяет алгоритм декодирования согласно charset. Один символ может требовать 1–4 байта (зависит от кодировки): именно поэтому каждый вызов `read()` может читать несколько байт из нижележащего потока.

```java
// байтовый поток → символьный с явной кодировкой
InputStream byteStream = new FileInputStream("file.txt");
Reader charStream = new InputStreamReader(byteStream, StandardCharsets.UTF_8);
```

Конструкторы класса:
- `InputStreamReader(InputStream in)` — использует платформенный charset (не рекомендуется для переносимого кода)
- `InputStreamReader(InputStream in, String charsetName)` — кидает `UnsupportedEncodingException`
- `InputStreamReader(InputStream in, Charset cs)` — предпочтительный вариант
- `InputStreamReader(InputStream in, CharsetDecoder dec)` — для тонкой настройки декодирования

**`OutputStreamWriter` — символ → байт**

Симметричный класс в обратном направлении: принимает символы (`char`) и кодирует их в байты через `CharsetEncoder`, записывая результат в нижележащий `OutputStream`.

```java
// символьный поток → байтовый с явной кодировкой
OutputStream byteStream = new FileOutputStream("output.txt");
Writer charStream = new OutputStreamWriter(byteStream, StandardCharsets.UTF_8);
```

**`FileReader` и `FileWriter`** — это просто удобные подклассы `InputStreamReader` и `OutputStreamWriter` соответственно, которые внутри открывают файл через `FileInputStream`/`FileOutputStream`. До Java 11 они не принимали параметр кодировки, что являлось известной ловушкой. Начиная с Java 11 добавлены конструкторы с `Charset`.

**Рекомендация:** всегда оборачивайте мост-классы в `BufferedReader`/`BufferedWriter` для амортизации затрат на декодирование:

```java
BufferedReader reader = new BufferedReader(
    new InputStreamReader(socket.getInputStream(), StandardCharsets.UTF_8)
);
```

### Вопрос 3. Для чего нужна буферизация при работе с потоками? Назовите классы-обёртки для буферизации байтовых и символьных потоков.

**Категория:** IO/NIO · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Buffered Streams (The Java Tutorials — Essential Java Classes > Basic I/O)](https://docs.oracle.com/javase/tutorial/essential/io/buffers.html)

Примеры ввода-вывода, которые мы видели до сих пор, используют небуферизованный ввод-вывод. Это означает, что каждый запрос на чтение или запись обрабатывается напрямую операционной системой. Это может сделать программу значительно менее эффективной, поскольку каждый такой запрос зачастую вызывает обращение к диску, сетевую активность или другую относительно дорогостоящую операцию.

Для снижения таких накладных расходов платформа Java реализует буферизованные потоки ввода-вывода. Буферизованные входные потоки считывают данные из области памяти, называемой буфером; нативный API ввода вызывается только тогда, когда буфер пуст. Аналогично буферизованные выходные потоки записывают данные в буфер, и нативный API вывода вызывается только тогда, когда буфер заполнен.

Программа может преобразовать небуферизованный поток в буферизованный, используя идиому обёртки:

```java
inputStream  = new BufferedReader(new FileReader("xanadu.txt"));
outputStream = new BufferedWriter(new FileWriter("characteroutput.txt"));
```

Для буферизации существуют четыре буферизованных класса-потока: `BufferedInputStream` и `BufferedOutputStream` создают буферизованные байтовые потоки, тогда как `BufferedReader` и `BufferedWriter` создают буферизованные символьные потоки.

#### Ответ от Claude

**Зачем нужна буферизация**

Каждый системный вызов `read()` или `write()` без буферизации транслируется в вызов операционной системы, что сопряжено с переключением контекста (user-space → kernel-space) и реальным вводом-выводом с диска или сети. Эти операции на несколько порядков медленнее, чем работа с оперативной памятью. Буферизация решает проблему: данные накапливаются в памяти, и обращение к ОС происходит блоками, а не побайтово.

**Принцип работы:**
- **Буферизованное чтение**: при первом `read()` из буфера данных нет — поток считывает из источника сразу целый блок (по умолчанию 8192 байт/символов) в внутренний массив. Последующие вызовы `read()` отдают данные из этого массива без обращения к ОС до его исчерпания.
- **Буферизованная запись**: данные складываются в внутренний массив; реальная запись происходит либо при заполнении буфера, либо при явном вызове `flush()`, либо при закрытии потока.

**Четыре класса-обёртки для буферизации**

| Тип потока | Чтение | Запись |
|---|---|---|
| Байтовый | `BufferedInputStream` | `BufferedOutputStream` |
| Символьный | `BufferedReader` | `BufferedWriter` |

Все четыре класса являются обёртками (Decorator pattern) — они не умеют читать из источника напрямую, а принимают в конструкторе уже существующий поток:

```java
// Байтовые потоки с буферизацией
BufferedInputStream  bin  = new BufferedInputStream(new FileInputStream("data.bin"));
BufferedOutputStream bout = new BufferedOutputStream(new FileOutputStream("out.bin"));

// Символьные потоки с буферизацией
BufferedReader br = new BufferedReader(new FileReader("text.txt"));
BufferedWriter bw = new BufferedWriter(new FileWriter("out.txt"));

// Можно задать размер буфера явно (второй аргумент — размер в байтах/символах)
BufferedReader br2 = new BufferedReader(new FileReader("text.txt"), 16384);
```

**Дополнительные возможности буферизованных классов**

- `BufferedInputStream` поддерживает `mark(int readlimit)` и `reset()` — возврат к отмеченной позиции в потоке.
- `BufferedReader` добавляет метод `readLine()` — чтение целой строки до символа перевода строки (`\n`, `\r`, `\r\n`). Этот метод недоступен в небуферизованном `FileReader`.
- `BufferedWriter` добавляет `newLine()` — запись платформо-зависимого разделителя строк.
- `flush()` вызывает реальную запись буфера не дожидаясь его заполнения — необходимо при сетевой передаче или интерактивном вводе-выводе.

**Важно:** при цепочке оборачиваний закрывать нужно только внешний поток — он закроет все внутренние:

```java
try (BufferedReader br = new BufferedReader(new InputStreamReader(
        new FileInputStream("file.txt"), StandardCharsets.UTF_8))) {
    String line;
    while ((line = br.readLine()) != null) {
        System.out.println(line);
    }
} // автоматически закрывает BufferedReader -> InputStreamReader -> FileInputStream
```
### Вопрос 4. В чём разница между FileInputStream и FileReader? Когда нельзя использовать FileReader?

**Категория:** IO/NIO · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [FileReader (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/io/FileReader.html)

`FileReader` — удобный класс для чтения символьных файлов. Его конструкторы предполагают, что подходят кодировка символов по умолчанию и размер буфера байт по умолчанию. Если нужно указать эти значения самостоятельно, следует создать `InputStreamReader` поверх `FileInputStream`. `FileReader` предназначен для чтения потоков символов. Для чтения потоков сырых байт следует использовать `FileInputStream`.

#### Ответ от Claude

**FileInputStream** (наследует `InputStream`) читает **сырые байты** из файла. Возвращает байты в диапазоне 0–255. Используется для бинарных данных: изображений, аудио, исполняемых файлов — любых данных, где нет понятия «символ» и «кодировка».

**FileReader** (наследует `InputStreamReader` → `Reader`) читает **символы** (char) из текстового файла. Внутри использует `FileInputStream` и декодирует байты в символы с помощью кодировки.

Ключевые различия:

| Аспект | FileInputStream | FileReader |
|---|---|---|
| Единица чтения | байт (0–255) | символ (char) |
| Кодировка | не применяется | системная по умолчанию |
| Иерархия | InputStream | Reader |
| Назначение | бинарные данные | текстовые данные |

**Когда нельзя использовать FileReader:**

1. **При чтении бинарных файлов** (изображения, архивы, исполняемые файлы) — FileReader интерпретирует байты как символы, что исказит данные.
2. **Когда нужна конкретная кодировка** (UTF-8, Windows-1251 и т.д.) — FileReader всегда использует системную кодировку по умолчанию (`Charset.defaultCharset()`), и это нельзя изменить через его конструктор. В этом случае нужно использовать `InputStreamReader` поверх `FileInputStream`:

```java
// Правильно — явно указываем кодировку
Reader reader = new InputStreamReader(
    new FileInputStream("file.txt"),
    StandardCharsets.UTF_8
);

// Ещё лучше в современном Java (7+)
Reader reader = Files.newBufferedReader(path, StandardCharsets.UTF_8);
```

3. **Когда нужен контроль над размером буфера байт** — FileReader не позволяет это настроить, тогда как `InputStreamReader` + `FileInputStream` дают полный контроль.

Начиная с Java 11, в конструкторе `FileReader` появилась поддержка явного указания `Charset`, однако до этого единственным правильным способом было использование `InputStreamReader`.

### Вопрос 5. Что такое сериализация и десериализация? Что нужно сделать, чтобы объект класса можно было сериализовать?

**Категория:** IO/NIO · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Serializable (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/io/Serializable.html)

Классы, чьи экземпляры должны быть сериализуемы, должны реализовать интерфейс `java.io.Serializable`. Serializability класса включается реализацией интерфейса `java.io.Serializable`. Классы, которые не реализуют этот интерфейс, не смогут сериализовать или десериализовать своё состояние. Все подтипы сериализуемого класса сами являются сериализуемыми. Интерфейс сериализации не имеет методов и полей и служит лишь для обозначения семантики сериализуемости. Настоятельно рекомендуется объявлять поле `serialVersionUID` явно, поскольку вычисление по умолчанию очень чувствительно к деталям класса, которые могут варьироваться в зависимости от реализации компилятора.

#### Ответ от Claude

**Сериализация** — процесс преобразования состояния объекта Java в последовательность байт (байтовый поток), пригодную для сохранения (в файл, БД) или передачи по сети.

**Десериализация** — обратный процесс: восстановление объекта Java из байтового потока.

**Что нужно сделать, чтобы объект можно было сериализовать:**

1. **Реализовать маркерный интерфейс `java.io.Serializable`**. Это интерфейс без методов и полей — он лишь сообщает JVM, что объекты данного класса можно сериализовать:

```java
public class Person implements Serializable {
    private String name;
    private int age;
}
```

2. **Объявить `serialVersionUID`** (настоятельно рекомендуется):

```java
public class Person implements Serializable {
    private static final long serialVersionUID = 1L;
    private String name;
    private int age;
}
```

`serialVersionUID` — это уникальный идентификатор версии класса. При десериализации JVM сверяет `serialVersionUID` из потока байт с текущим классом. Если они не совпадают, выбрасывается `InvalidClassException`. Без явного объявления JVM вычисляет его автоматически на основе структуры класса, что может привести к несовместимости при малейшем изменении класса.

3. **Все нестатические поля класса тоже должны быть сериализуемы** (реализовывать `Serializable`) или помечены как `transient`. Статические поля не сериализуются.

**Как выполнить сериализацию и десериализацию:**

```java
// Сериализация
try (ObjectOutputStream oos = new ObjectOutputStream(
        new FileOutputStream("person.dat"))) {
    oos.writeObject(new Person("Alice", 30));
}

// Десериализация
try (ObjectInputStream ois = new ObjectInputStream(
        new FileInputStream("person.dat"))) {
    Person p = (Person) ois.readObject();
}
```

**Дополнительные нюансы:**
- Все подклассы сериализуемого класса автоматически являются сериализуемыми.
- Если класс наследует несериализуемый суперкласс, у суперкласса должен быть доступный конструктор без аргументов — он вызывается при десериализации для инициализации полей суперкласса.
- Для кастомной логики сериализации можно реализовать методы `private void writeObject(ObjectOutputStream)` и `private void readObject(ObjectInputStream)`.

### Вопрос 6. Для чего служит ключевое слово transient? Какое значение получит transient-поле типа int после десериализации?

**Категория:** IO/NIO · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Java Object Serialization Specification: 1 — System Architecture — docs.oracle.com](https://docs.oracle.com/javase/7/docs/platform/serialization/spec/serial-arch.html)

Поля, помеченные как `transient`, не являются постоянными и не будут сохранены никаким механизмом персистентности. Пометка поля предотвращает появление его состояния в потоке и его восстановление при десериализации. Согласно спецификации, значения каждого поля объекта — независимо от того, является ли оно transient, static или нет — устанавливаются в значение по умолчанию для типа поля.

#### Ответ от Claude

**Ключевое слово `transient`** указывает JVM исключить данное поле из процесса сериализации. Поле с этим модификатором не записывается в байтовый поток при сериализации и не восстанавливается из него при десериализации.

**Основные случаи применения:**

1. **Конфиденциальные данные** — пароли, ключи шифрования, токены, которые не должны покидать JVM в сериализованном виде:
```java
public class User implements Serializable {
    private static final long serialVersionUID = 1L;
    private String username;
    private transient String password; // не будет сериализован
}
```

2. **Поля, которые нельзя сериализовать** — `Thread`, `Socket`, `InputStream`, соединения с БД (`Connection`) и другие объекты, привязанные к ресурсам JVM.

3. **Вычислимые поля** — значения, которые легко пересчитать из других полей при десериализации, что позволяет сэкономить место в потоке.

4. **Поля, нарушающие граф сериализации** — например, циклические ссылки или поля типов без `Serializable`.

**Значение transient-поля типа `int` после десериализации:**

После десериализации `transient`-поле типа `int` получит значение **`0`** — стандартное значение по умолчанию для примитива `int` в Java.

Полная таблица значений по умолчанию для `transient`-полей после десериализации:

| Тип | Значение после десериализации |
|---|---|
| `int`, `short`, `byte`, `long` | `0` |
| `double`, `float` | `0.0` |
| `char` | `''` (нулевой символ) |
| `boolean` | `false` |
| Любой объектный тип | `null` |

Пример:

```java
public class Counter implements Serializable {
    private static final long serialVersionUID = 1L;
    private int count = 10;
    private transient int cachedHash = 42;
    private transient String cachedString = "hello";
}

// После сериализации и десериализации:
// count         == 10    (восстановлено из потока)
// cachedHash    == 0     (transient int -> 0 по умолчанию)
// cachedString  == null  (transient String -> null по умолчанию)
```

Если после десериализации `transient`-поля необходимо восстановить в ненулевые значения, это делают через метод `private void readObject(ObjectInputStream in)`:

```java
private void readObject(ObjectInputStream in)
        throws IOException, ClassNotFoundException {
    in.defaultReadObject();
    this.cachedHash = computeHash(); // восстанавливаем вручную
}
```
### Вопрос 7. Что такое serialVersionUID? Что произойдёт при десериализации, если serialVersionUID в классе не совпадает с тем, что записан в потоке?

**Категория:** IO/NIO · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Serializable (Java SE 21 & JDK 21) — docs.oracle.com](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/io/Serializable.html)

Среда выполнения сериализации ассоциирует с каждым сериализуемым классом номер версии — `serialVersionUID`, — который используется при десериализации для проверки того, что отправитель и получатель сериализованного объекта загрузили классы, совместимые друг с другом по отношению к сериализации. Если получатель загрузил класс объекта с `serialVersionUID`, отличающимся от `serialVersionUID` класса на стороне отправителя, десериализация завершится исключением `InvalidClassException`. Сериализуемый класс может явно объявить `serialVersionUID` как поле `static final long`. Если класс явно не объявляет `serialVersionUID`, то среда выполнения сериализации вычислит его значение по умолчанию на основе различных аспектов класса. Однако настоятельно рекомендуется, чтобы все сериализуемые классы явно объявляли `serialVersionUID`, поскольку вычисление значения по умолчанию крайне чувствительно к деталям класса, которые могут варьироваться в зависимости от реализации компилятора, что способно вызывать непредвиденные `InvalidClassException` при десериализации.

#### Ответ от Claude

`serialVersionUID` — это статическое финальное поле типа `long`, которое служит идентификатором версии сериализуемого класса. Объявляется следующим образом:

```java
private static final long serialVersionUID = 42L;
```

**Назначение.** При сериализации объекта JVM записывает в поток значение `serialVersionUID` класса. При десериализации JVM считывает это значение из потока и сравнивает с `serialVersionUID` текущей версии класса в classpath. Если значения совпадают — десериализация продолжается; если нет — бросается `java.io.InvalidClassException` с сообщением о несовпадении версий.

**Автоматическое вычисление.** Если `serialVersionUID` не объявлен явно, JVM вычисляет его как 64-битный хэш (по алгоритму SHA-1) на основе: имени класса, его модификаторов, имён реализуемых интерфейсов, непривилегированных полей, конструкторов и методов. Это делает автоматически вычисленный UID нестабильным: добавление/удаление любого поля или метода изменит его, что сломает десериализацию ранее сохранённых объектов.

**При несовпадении UIDs:**
- Бросается `InvalidClassException: <имя класса>; local class incompatible: stream classdesc serialVersionUID = X, local class serialVersionUID = Y`.
- Десериализация прерывается; объект не восстанавливается.
- Исключение — проверяемое (checked), поэтому его необходимо обрабатывать.

**Рекомендации:**
- Всегда объявляйте `serialVersionUID` явно — это гарантирует предсказуемое поведение.
- Используйте модификатор `private` (поле относится только к текущему классу, наследники объявляют своё).
- Меняйте значение намеренно, когда вносите несовместимые изменения в структуру класса и хотите запретить десериализацию старых объектов.
- Оставляйте значение неизменным при совместимых изменениях (добавление нового поля, которое получит значение по умолчанию при десериализации старого объекта).
- Для генерации значения можно использовать утилиту `serialver` из JDK или встроенные средства IDE.

### Вопрос 8. Сериализуются ли статические поля класса? Почему?

**Категория:** IO/NIO · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Java Object Serialization Specification: 1 - System Architecture — docs.oracle.com](https://docs.oracle.com/en/java/javase/11/docs/specs/serialization/serial-arch.html)

Стандартные сериализуемые поля класса определяются как нетранзиентные и нестатические поля (non-transient and non-static fields). Таким образом, статические поля явно исключены из сериализации. Причина концептуальна: статические поля принадлежат классу, а не экземпляру объекта. Поскольку сериализация предназначена для сохранения состояния отдельных экземпляров объектов, статические поля, представляющие состояние уровня класса, совместно используемое всеми экземплярами, логически исключаются из этого процесса. Сериализация нескольких экземпляров одного класса с сохранением общих статических данных создала бы непредсказуемые и сложные проблемы: избыточные записи одних и тех же данных, неопределённость при восстановлении общего состояния из разных потоков.

#### Ответ от Claude

**Нет, статические поля не сериализуются.**

**Причина: статические поля принадлежат классу, а не объекту.**

Сериализация в Java — это механизм сохранения и восстановления *состояния объекта* (экземпляра). Статические поля — это состояние *класса*, разделяемое между всеми экземплярами. Поэтому они не являются частью состояния конкретного объекта и не включаются в поток сериализации.

Спецификация Java Serialization прямо гласит:
> "Default serializable fields of a class are defined to be the non-transient and non-static fields."

**Что происходит при десериализации:**
- Статические поля не читаются из потока.
- После десериализации объекта статические поля имеют то значение, которое они имеют в JVM *в момент десериализации* (либо значение по умолчанию при инициализации класса, либо значение, установленное в ходе выполнения программы).

**Пример:**
```java
public class Counter implements Serializable {
    private static final long serialVersionUID = 1L;
    private static int instanceCount = 0; // не сериализуется
    private int id;                        // сериализуется

    public Counter() {
        this.id = ++instanceCount;
    }
}
```
При сериализации `Counter` в поток записывается только `id`. При десериализации `instanceCount` не восстанавливается из потока — он сохраняет текущее значение в JVM.

**Аналогичное поведение у `transient`-полей** — они также исключаются из сериализации, но по другой причине: разработчик явно помечает их как не требующие сохранения (например, кэши, временные вычисления, чувствительные данные). После десериализации `transient`-поля получают значение по умолчанию (`null`, `0`, `false`).

**Итог:** `static` и `transient` поля объединяет одно — они не попадают в поток сериализации. Разница в семантике: `static` исключены по природе (принадлежат классу), `transient` — по явному решению разработчика.

### Вопрос 9. Чем Path из java.nio.file отличается от старого класса File? Какие преимущества даёт новый API?

**Категория:** IO/NIO · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Legacy File I/O Code (The Java Tutorials: Basic I/O) — docs.oracle.com](https://docs.oracle.com/javase/tutorial/essential/io/legacy.html)

Класс `java.io.File` имел ряд существенных недостатков, которые привели к созданию `java.nio.file.Path` в Java SE 7. Во-первых, многие методы класса `File` не бросали исключений при ошибках — например, при неудачном удалении файла программа получала только сигнал «удаление не выполнено», но не могла узнать причину: файл не существовал, недостаточно прав или иная проблема. Во-вторых, метод `rename` работал непоследовательно на разных платформах. В-третьих, отсутствовала реальная поддержка символических ссылок. В-четвёртых, не хватало поддержки метаданных: прав доступа, владельца файла и атрибутов безопасности. В-пятых, методы плохо масштабировались на директории с большим числом файлов — листинг больших директорий мог приводить к зависанию серверов и проблемам с памятью. Новый API предоставляет богатую функциональность и улучшенную обработку ошибок. Для миграции существующего кода класс `File` предоставляет метод `toPath()`, а `Path` — метод `toFile()`.

#### Ответ от Claude

`java.io.File` существует с Java 1.0; `java.nio.file.Path` появился в Java 7 (NIO.2, JSR-203) как полноценная замена.

**Ключевые отличия:**

| Аспект | `java.io.File` | `java.nio.file.Path` |
|---|---|---|
| Тип | Класс (конкретный) | Интерфейс |
| Создание экземпляра | `new File("path")` | `Paths.get("path")` / `Path.of("path")` (Java 11+) |
| Обработка ошибок | Методы возвращают `boolean`; при ошибке причина неизвестна | Методы бросают содержательные исключения (`IOException` и подклассы) |
| Символические ссылки | Не поддерживаются | Полная поддержка (`Files.isSymbolicLink()`, `Files.readSymbolicLink()`, управление циклическими ссылками при обходе дерева) |
| Метаданные | Ограниченны (`length()`, `lastModified()`, базовые права) | Полные атрибуты через `Files.readAttributes()`, поддержка POSIX, ACL, расширенных атрибутов |
| Операции с файлами | На самом объекте `File` (`file.delete()`) | В утилитном классе `Files` (`Files.delete(path)`) |
| Листинг директорий | `File.list()` / `File.listFiles()` — возвращает весь массив сразу | `Files.newDirectoryStream()` — ленивый итератор, не загружает всё в память |
| Обход файлового дерева | Нет | `Files.walkFileTree()`, `Files.walk()` (Java 8+) |
| Работа с путями | Конкатенация строк | `path.resolve()`, `path.relativize()`, `path.normalize()`, `path.getParent()`, `path.getFileName()` |
| Файловые системы | Только локальная ОС | Подключаемые (`FileSystem`, `FileSystems`): zip-архивы, in-memory FS, облачные хранилища |
| Атомарные операции | Нет | `Files.move()` может быть атомарным (`StandardCopyOption.ATOMIC_MOVE`) |
| Watch Service | Нет | `WatchService` — мониторинг изменений в директории |

**Преимущества `Path` / NIO.2:**

1. **Надёжная обработка ошибок.** `Files.delete(path)` бросает `NoSuchFileException` или `AccessDeniedException` вместо молчаливого возврата `false`.
2. **Богатая работа с путями.** `path.normalize()` убирает `.` и `..`; `path.relativize(other)` вычисляет относительный путь; `path.resolve("subdir")` безопасно конкатенирует части.
3. **Масштабируемость.** `DirectoryStream` обходит директорию лениво, не загружая всё содержимое в память.
4. **Расширяемость.** `FileSystem` API позволяет работать с любыми файловыми системами единообразно.
5. **Функциональный стиль.** `Files.walk()` возвращает `Stream<Path>`, что интегрируется с Stream API (Java 8+).

**Совместимость.** Классы взаимно конвертируемы:
```java
Path path = file.toPath();   // File → Path
File file = path.toFile();   // Path → File (только для локальных путей)
```

`java.io.File` не deprecated и не будет удалён в обозримом будущем, однако для нового кода рекомендуется использовать `Path` и `Files`.
### Вопрос 10. Как с помощью класса Files скопировать, переместить или удалить файл? Какие методы для этого используются?

**Категория:** IO/NIO · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Java NIO Files — jenkov.com](https://jenkov.com/tutorials/java-nio/files.html)

Класс `Files` (пакет `java.nio.file`) предоставляет статические методы для операций с файлами.

**Копирование — `Files.copy()`**
Копирует файл из одного пути в другой. Если целевой файл уже существует, по умолчанию выбрасывается `FileAlreadyExistsException`. Чтобы разрешить перезапись, передаётся флаг `StandardCopyOption.REPLACE_EXISTING`:

```java
Path src  = Paths.get("data/logging.properties");
Path dest = Paths.get("data/logging-copy.properties");
Files.copy(src, dest, StandardCopyOption.REPLACE_EXISTING);
```

**Перемещение — `Files.move()`**
Перемещает (или переименовывает) файл. Принимает те же опции `CopyOption`:

```java
Path src  = Paths.get("data/logging-copy.properties");
Path dest = Paths.get("data/subdir/logging-moved.properties");
Files.move(src, dest, StandardCopyOption.REPLACE_EXISTING);
```

**Удаление — `Files.delete()`**
Удаляет файл; при ошибке выбрасывает `IOException`:

```java
Files.delete(Paths.get("data/subdir/logging-moved.properties"));
```

#### Ответ от Claude

Класс `java.nio.file.Files` (Java 7+) содержит три группы методов для базовых файловых операций.

**Копирование**

| Сигнатура | Назначение |
|-----------|-----------|
| `Files.copy(Path src, Path dest, CopyOption... opts)` | Копирует файл или директорию |
| `Files.copy(InputStream in, Path dest, CopyOption... opts)` | Копирует поток в файл |
| `Files.copy(Path src, OutputStream out)` | Копирует файл в поток |

Флаги `CopyOption` (из `StandardCopyOption`):
- `REPLACE_EXISTING` — перезаписать существующий целевой файл.
- `COPY_ATTRIBUTES` — перенести атрибуты файла (в т.ч. `last-modified-time`).
- `NOFOLLOW_LINKS` — копировать саму символическую ссылку, а не её цель.

При копировании директории её содержимое **не копируется** — создаётся пустая директория. Для рекурсивного копирования используют `Files.copy()` в связке с `Files.walkFileTree()`.

```java
import static java.nio.file.StandardCopyOption.*;

Files.copy(source, target, REPLACE_EXISTING, COPY_ATTRIBUTES);
```

**Перемещение / переименование**

`Files.move(Path src, Path dest, CopyOption... opts)` — атомарно перемещает или переименовывает файл. Без `REPLACE_EXISTING` при наличии целевого файла бросает `FileAlreadyExistsException`. Флаг `StandardCopyOption.ATOMIC_MOVE` гарантирует атомарность на уровне ОС (если файловая система поддерживает).

```java
Files.move(source, target, REPLACE_EXISTING);
```

**Удаление**

| Метод | Поведение |
|-------|-----------|
| `Files.delete(Path)` | Удаляет файл; бросает исключение при ошибке |
| `Files.deleteIfExists(Path)` | Удаляет файл; возвращает `false`, если файла не было, не бросая исключения |

Типы исключений при `Files.delete()`:
- `NoSuchFileException` — файл не существует.
- `DirectoryNotEmptyException` — директория не пуста.
- `IOException` — прочие ошибки (права доступа и т.д.).

Символическая ссылка удаляется сама, а не её цель. Директория должна быть пустой, иначе удаление не выполнится.

```java
try {
    Files.delete(path);
} catch (NoSuchFileException e) {
    System.err.println("Файл не найден: " + path);
} catch (DirectoryNotEmptyException e) {
    System.err.println("Директория не пуста: " + path);
} catch (IOException e) {
    System.err.println("Ошибка удаления: " + e);
}
```

### Вопрос 11. Чем Java IO отличается от Java NIO в плане модели работы с данными (потоко-ориентированность против буферо-ориентированности, блокирующий против неблокирующего ввода-вывода)?

**Категория:** IO/NIO · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Java NIO vs. IO — jenkov.com](https://jenkov.com/tutorials/java-nio/nio-vs-io.html)

**Потоко-ориентированность (Stream Oriented) против буферо-ориентированности (Buffer Oriented)**

Java IO работает со потоками: данные читаются по одному или нескольким байтам за раз непосредственно из потока. Прочитанные данные нигде не кешируются и к ним нельзя вернуться без дополнительных усилий.

Java NIO читает данные в буфер, который затем обрабатывается по мере необходимости. Можно перемещаться вперёд-назад по буферу, что даёт большую гибкость при обработке.

**Блокирующий (Blocking) против неблокирующего (Non-blocking) ввода-вывода**

Потоки Java IO блокирующие: когда поток вызывает `read()` или `write()`, он заблокирован до завершения операции и ничего другого в это время делать не может.

В неблокирующем режиме NIO поток может запросить чтение из канала и получить только то, что доступно прямо сейчас (или ничего). Поток не остаётся заблокированным и может параллельно обрабатывать другие каналы.

**Селекторы (Selectors)**

NIO предоставляет механизм `Selector`, позволяющий одному потоку следить за несколькими каналами ввода-вывода. Это делает NIO предпочтительным для серверов с тысячами одновременных соединений с небольшими объёмами данных (например, чат-серверы).

#### Ответ от Claude

Java IO и Java NIO различаются по трём ключевым аспектам.

**1. Потоко-ориентированность vs. буферо-ориентированность**

*Java IO (java.io)* работает через потоки (`InputStream`/`OutputStream`, `Reader`/`Writer`). Данные читаются/пишутся байт за байтом или небольшими порциями строго последовательно. Возврат к уже прочитанным данным невозможен без обёртки вроде `PushbackInputStream`.

*Java NIO (java.nio)* строится вокруг `Buffer` и `Channel`. Данные сначала читаются целиком в буфер, затем приложение обрабатывает их в произвольном порядке — можно двигаться вперёд и назад внутри буфера.

```
IO:   Thread → Stream → byte-by-byte → обработка
NIO:  Thread → Channel → Buffer → обработка (произвольный доступ)
```

**2. Блокирующий vs. неблокирующий ввод-вывод**

*Java IO* — всегда блокирующий. Вызов `read()` или `write()` приостанавливает поток до завершения операции. Для обслуживания N соединений нужно N потоков — плохо масштабируется.

*Java NIO* — поддерживает неблокирующий режим. Поток вызывает `channel.read(buffer)` и сразу получает управление обратно с тем, что успело поступить (или 0). Благодаря этому один поток может одновременно обслуживать множество каналов.

**3. Механизм Selector**

NIO вводит `Selector` (мультиплексор): один поток регистрирует несколько `SelectableChannel` (сетевые сокеты, серверные сокеты) и вызывает `selector.select()`, получая набор каналов, готовых к чтению/записи. Это позволяет обрабатывать тысячи одновременных соединений в одном потоке.

**Сравнительная таблица**

| Характеристика | Java IO | Java NIO |
|----------------|---------|----------|
| Ориентация | Поток (Stream) | Буфер (Buffer) |
| Блокировка | Всегда блокирующий | Блокирующий и неблокирующий |
| Доступ к данным | Последовательный | Произвольный (в буфере) |
| Мультиплексирование | Нет | Selector |
| Лучше подходит для | Простые файловые операции, мало соединений | Высоконагруженные серверы, тысячи соединений |

**Когда что использовать:**
- Java IO — когда число соединений невелико, но объём данных на каждое соединение велик (например, передача файлов).
- Java NIO — когда соединений много, но каждое передаёт небольшие порции данных (например, HTTP-серверы, чаты, игровые серверы).

### Вопрос 12. Что такое Buffer в NIO? Объясните назначение атрибутов capacity, limit, position и mark. Как работают методы flip(), clear() и rewind()?

**Категория:** IO/NIO · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Buffer (Java Platform SE 8) — docs.oracle.com](https://docs.oracle.com/javase/8/docs/api/java/nio/Buffer.html)

Буфер — контейнер конечного количества элементов примитивного типа. Ключевые атрибуты:

- **capacity** — количество элементов, которое может содержать буфер. Никогда не бывает отрицательным и не меняется.
- **limit** — индекс первого элемента, который не должен быть прочитан или записан. Никогда не бывает отрицательным и не превышает capacity.
- **position** — индекс следующего элемента для чтения или записи. Никогда не бывает отрицательным и не превышает limit.
- **mark** — индекс, к которому будет сброшен position при вызове `reset()`. Не определён до вызова `mark()`.

Инвариант: `0 <= mark <= position <= limit <= capacity`.

**Методы:**
- `clear()` — устанавливает `limit = capacity`, `position = 0`, сбрасывает mark. Готовит буфер к заполнению.
- `flip()` — устанавливает `limit = position`, `position = 0`, сбрасывает mark. Переключает буфер из режима записи в режим чтения.
- `rewind()` — устанавливает `position = 0`, limit не изменяет, сбрасывает mark. Позволяет перечитать данные заново.

#### Ответ от Claude

**Что такое Buffer**

`java.nio.Buffer` — абстрактный базовый класс для буферов примитивных типов. Реализации: `ByteBuffer`, `CharBuffer`, `ShortBuffer`, `IntBuffer`, `LongBuffer`, `FloatBuffer`, `DoubleBuffer`. В большинстве случаев используется `ByteBuffer`, так как каналы (`Channel`) работают именно с ним.

Создание буфера:
```java
ByteBuffer buf = ByteBuffer.allocate(1024);   // в heap-памяти JVM
ByteBuffer dir = ByteBuffer.allocateDirect(1024); // в native-памяти (быстрее для I/O)
```

**Четыре атрибута и инвариант**

```
0  <=  mark  <=  position  <=  limit  <=  capacity
```

| Атрибут | Описание |
|---------|---------|
| `capacity` | Полный размер буфера в элементах. Задаётся при создании, не меняется никогда. |
| `limit` | Граница допустимых данных: элементы с индексом >= limit недоступны. В режиме записи `limit == capacity`; в режиме чтения — число реально записанных байт. |
| `position` | Указатель на следующий элемент для чтения/записи. Автоматически сдвигается при каждой операции `get()`/`put()`. |
| `mark` | «Закладка»: `buf.mark()` запоминает текущий `position`; `buf.reset()` восстанавливает его. Не определён (= -1) до первого вызова `mark()`. При `flip()`, `clear()` и `rewind()` сбрасывается в -1. |

**Методы управления состоянием**

**`flip()` — переход от записи к чтению**

Устанавливает `limit = position`, затем `position = 0`. Mark сбрасывается. После вызова буфер готов к последовательному чтению ровно тех данных, что были записаны.

```java
ByteBuffer buf = ByteBuffer.allocate(10);
buf.put((byte) 1);
buf.put((byte) 2);
// position=2, limit=10
buf.flip();
// position=0, limit=2  <-- теперь читаем ровно 2 байта
while (buf.hasRemaining()) {
    System.out.println(buf.get()); // 1, 2
}
```

**`clear()` — сброс для повторной записи**

Устанавливает `position = 0`, `limit = capacity`. Mark сбрасывается. Данные в буфере физически **не стираются** — они остаются до перезаписи, поэтому метод называется «clear», а не «erase».

```java
buf.clear();            // готовим буфер к новой записи
channel.read(buf);      // заполняем данными
```

**`rewind()` — повторное чтение тех же данных**

Устанавливает `position = 0`, `limit` **не меняет**. Позволяет перечитать уже прочитанные данные или записать их в несколько каналов.

```java
buf.flip();
channel1.write(buf);    // записываем данные в первый канал
buf.rewind();           // сбрасываем position, limit не трогаем
channel2.write(buf);    // записываем те же данные во второй канал
```

**Сводная таблица**

| Метод | position | limit | mark |
|-------|----------|-------|------|
| `flip()` | 0 | бывший position | сброшен |
| `clear()` | 0 | capacity | сброшен |
| `rewind()` | 0 | без изменений | сброшен |

**Типичный цикл работы с буфером и каналом:**

```java
ByteBuffer buf = ByteBuffer.allocate(48);

// 1. Чтение из канала в буфер
int bytesRead = channel.read(buf);

while (bytesRead != -1) {
    buf.flip();                     // переключаем в режим чтения
    while (buf.hasRemaining()) {
        System.out.print((char) buf.get());
    }
    buf.clear();                    // переключаем в режим записи
    bytesRead = channel.read(buf);
}
```
### Вопрос 13. Что такое Channel в NIO? Чем FileChannel отличается от SocketChannel? Как выполняется чтение и запись данных через канал?

**Категория:** IO/NIO · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Java NIO Channel — jenkov.com](https://jenkov.com/tutorials/java-nio/channels.html)

Channel в Java NIO — это сущность, через которую выполняются операции ввода-вывода. В отличие от потоков (Stream) канал позволяет и читать, и писать данные, может работать асинхронно и всегда работает в связке с буфером (Buffer): данные читаются из канала в буфер, а пишутся из буфера в канал.

**FileChannel** читает и пишет данные в файлы на диске. **SocketChannel** работает с сетевыми TCP-сокетами и передаёт данные по сети.

Ключевое различие в режиме блокировки: FileChannel всегда работает в блокирующем режиме и не может быть переведён в неблокирующий. SocketChannel поддерживает неблокирующий режим через `configureBlocking(false)`, что позволяет вызывать `connect()`, `read()` и `write()` асинхронно.

Чтение данных через FileChannel:
```java
RandomAccessFile aFile = new RandomAccessFile("data/nio-data.txt", "rw");
FileChannel inChannel = aFile.getChannel();
ByteBuffer buf = ByteBuffer.allocate(48);

int bytesRead = inChannel.read(buf);
while (bytesRead != -1) {
    buf.flip();
    while (buf.hasRemaining()) {
        System.out.print((char) buf.get());
    }
    buf.clear();
    bytesRead = inChannel.read(buf);
}
aFile.close();
```

После чтения необходимо вызвать `buf.flip()`, чтобы переключить буфер из режима записи в режим чтения. Запись выполняется в цикле, так как нет гарантии, что все байты будут записаны за один вызов `write()`.

#### Ответ от Claude

**Channel** — центральная абстракция Java NIO, представляющая открытое соединение с сущностью (файлом, сокетом, устройством), через которую можно выполнять операции чтения и/или записи. Интерфейс `java.nio.channels.Channel` является базовым; все конкретные каналы реализуют его подинтерфейсы.

Принципиальные отличия каналов от потоков (InputStream/OutputStream):
- Канал **двунаправлен** — один объект может и читать, и писать (в зависимости от типа канала).
- Работа идёт **только через Buffer** — прямая передача байтов невозможна.
- Возможна **асинхронная/неблокирующая** работа (для сетевых каналов).

**FileChannel** (`java.nio.channels.FileChannel`):
- Предназначен для работы с файлами на диске.
- **Всегда блокирующий** — метод `configureBlocking(false)` не поддерживается и бросает `IllegalBlockingModeException`.
- Дополнительные возможности: управление позицией (`position()`), получение размера (`size()`), усечение файла (`truncate()`), сброс данных на диск (`force()`), эффективное копирование между каналами (`transferTo()` / `transferFrom()`), отображение файла в память (`map()`), блокировки файла (`lock()`).
- Открывается не напрямую, а через `FileInputStream`, `FileOutputStream` или `RandomAccessFile`.

**SocketChannel** (`java.nio.channels.SocketChannel`):
- Представляет TCP-соединение с удалённым хостом.
- **Поддерживает неблокирующий режим** (`configureBlocking(false)`).
- В неблокирующем режиме `connect()` может вернуться немедленно; завершение соединения проверяется через `finishConnect()`.
- Предназначен для работы с `Selector` — именно сочетание SocketChannel + Selector позволяет одному потоку обслуживать тысячи соединений.
- Открывается через `SocketChannel.open()`.

**Паттерн чтения через канал** (общий для обоих типов):
```java
ByteBuffer buf = ByteBuffer.allocate(1024);
int bytesRead = channel.read(buf); // данные пишутся В буфер
while (bytesRead != -1) {
    buf.flip();                    // переключить в режим чтения
    while (buf.hasRemaining()) {
        byte b = buf.get();        // читаем из буфера
    }
    buf.clear();                   // сброс для следующей порции
    bytesRead = channel.read(buf);
}
```

**Паттерн записи через канал**:
```java
ByteBuffer buf = ByteBuffer.wrap("Hello NIO".getBytes());
buf.flip();
while (buf.hasRemaining()) {
    channel.write(buf); // нет гарантии, что запишутся все байты за раз
}
```

Итог различий FileChannel vs SocketChannel:

| Характеристика | FileChannel | SocketChannel |
|---|---|---|
| Назначение | Файловый I/O | Сетевой TCP I/O |
| Неблокирующий режим | Нет | Да |
| Работа с Selector | Нет | Да |
| Открытие | Через stream/RandomAccessFile | `SocketChannel.open()` |
| Дополнительно | map, transferTo, lock, force | finishConnect, connect |

### Вопрос 14. Что такое Selector в NIO и как он позволяет одному потоку обслуживать несколько каналов? Как зарегистрировать канал и обработать SelectionKey?

**Категория:** IO/NIO · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Java NIO Selector — jenkov.com](https://jenkov.com/tutorials/java-nio/selectors.html)

Selector — компонент Java NIO, который позволяет одному потоку следить за несколькими каналами и определять, какие из них готовы к операциям ввода-вывода (чтению, записи и т.д.). Это снижает накладные расходы на переключение контекста потоков и экономит память.

Чтобы использовать канал с Selector, канал должен быть переведён в неблокирующий режим и зарегистрирован:
```java
channel.configureBlocking(false);
SelectionKey key = channel.register(selector, SelectionKey.OP_READ);
```

FileChannel не поддерживает неблокирующий режим и не может быть зарегистрирован в Selector.

После регистрации метод `register()` возвращает объект `SelectionKey`, который содержит:
- **Interest Set** — набор операций, за которыми следим (OP_CONNECT, OP_ACCEPT, OP_READ, OP_WRITE);
- **Ready Set** — набор операций, к которым канал готов прямо сейчас;
- ссылку на Channel;
- ссылку на Selector;
- опциональный прикреплённый объект (attachment).

Цикл обработки готовых каналов:
```java
Set<SelectionKey> selectedKeys = selector.selectedKeys();
Iterator<SelectionKey> keyIterator = selectedKeys.iterator();
while (keyIterator.hasNext()) {
    SelectionKey key = keyIterator.next();
    if (key.isReadable()) {
        // обработать чтение
    }
    keyIterator.remove(); // обязательно удалить вручную
}
```

#### Ответ от Claude

**Selector** (`java.nio.channels.Selector`) — мультиплексор каналов, реализующий паттерн «reactor». Он опрашивает зарегистрированные каналы и сообщает, какие из них готовы к выполнению I/O-операций, не блокируя поток для каждого отдельного канала.

**Принцип работы (один поток — много каналов):**
В традиционной модели на каждое соединение выделяется поток. При тысячах соединений это приводит к огромным затратам памяти и накладным расходам на переключение контекста. Selector позволяет одному потоку вызвать `select()`, который заблокируется до тех пор, пока хотя бы один из зарегистрированных каналов не станет готов, после чего поток обработает только готовые каналы.

**Шаги работы с Selector:**

1. Создание Selector:
```java
Selector selector = Selector.open();
```

2. Перевод канала в неблокирующий режим и регистрация:
```java
ServerSocketChannel serverChannel = ServerSocketChannel.open();
serverChannel.configureBlocking(false);
// Регистрируем интерес к входящим соединениям
SelectionKey key = serverChannel.register(selector, SelectionKey.OP_ACCEPT);
```

Допустимые операции-интересы (битовые флаги):
- `SelectionKey.OP_CONNECT` (8) — завершение соединения клиентом
- `SelectionKey.OP_ACCEPT` (16) — принятие входящего соединения сервером
- `SelectionKey.OP_READ` (1) — канал готов к чтению
- `SelectionKey.OP_WRITE` (4) — канал готов к записи

Несколько операций объединяются через побитовое ИЛИ: `SelectionKey.OP_READ | SelectionKey.OP_WRITE`.

3. Ожидание готовности (`select()`):
```java
int readyCount = selector.select();      // блокирующий вызов
// selector.select(1000)                 // с таймаутом в мс
// selector.selectNow()                  // неблокирующий
```
Метод `select()` возвращает количество каналов, ставших готовыми с момента последнего вызова.

4. Обработка готовых каналов:
```java
Set<SelectionKey> selectedKeys = selector.selectedKeys();
Iterator<SelectionKey> iter = selectedKeys.iterator();

while (iter.hasNext()) {
    SelectionKey key = iter.next();

    if (key.isAcceptable()) {
        // ServerSocketChannel готов принять соединение
        ServerSocketChannel server = (ServerSocketChannel) key.channel();
        SocketChannel client = server.accept();
        client.configureBlocking(false);
        client.register(selector, SelectionKey.OP_READ);
    } else if (key.isReadable()) {
        // SocketChannel готов к чтению
        SocketChannel client = (SocketChannel) key.channel();
        ByteBuffer buf = ByteBuffer.allocate(256);
        client.read(buf);
    }

    iter.remove(); // ОБЯЗАТЕЛЬНО: Selector не удаляет ключи автоматически
}
```

**Важные детали:**
- `selector.selectedKeys()` возвращает ключи каналов, готовых **с последнего вызова `select()`**. Selector **не удаляет** ключи из этого набора автоматически — если не вызвать `iter.remove()`, на следующей итерации ключ снова появится в наборе, что приведёт к повторной обработке.
- `selector.keys()` возвращает **все** зарегистрированные ключи (не только готовые).
- Для пробуждения заблокированного `select()` из другого потока используется `selector.wakeup()`.
- FileChannel **не может** быть зарегистрирован в Selector, так как не реализует `SelectableChannel`.
- Selector закрывается через `selector.close()`.

**Типичное применение:** серверы с высокой конкурентностью (HTTP-серверы, чат-серверы), где число соединений значительно превышает разумное число потоков. Именно этот механизм лежит в основе Netty, Undertow и других высокопроизводительных Java-серверов.

### Вопрос 15. Если суперкласс реализует Serializable, наследует ли подкласс это поведение автоматически? Как запретить сериализацию подкласса, если суперкласс Serializable?

**Категория:** IO/NIO · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Serializable (Java SE 21) — docs.oracle.com](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/io/Serializable.html)

Официальная документация прямо указывает:

> «All subtypes of a serializable class are themselves serializable.»
> (Все подтипы сериализуемого класса сами являются сериализуемыми.)

Таким образом, подкласс автоматически наследует `Serializable` от суперкласса без явного объявления.

Для случая, когда суперкласс **не** сериализуем, а подкласс реализует `Serializable`, документация требует, чтобы у несериализуемого суперкласса был доступный конструктор без аргументов: при десериализации именно он будет вызван для инициализации полей суперкласса, тогда как поля суперкласса не записываются в поток при сериализации.

Документация не предоставляет встроенного языкового механизма для запрета сериализации подкласса, если суперкласс сериализуем. На практике применяется переопределение методов `writeObject()` и `readObject()` с выбрасыванием `NotSerializableException`.

#### Ответ от Claude

**Наследование Serializable:**

Да — если суперкласс реализует `java.io.Serializable`, все его подклассы **автоматически** становятся сериализуемыми, поскольку `Serializable` — маркерный интерфейс, а наследование интерфейсов в Java транзитивно. Подкласс не обязан явно объявлять `implements Serializable`.

Это подтверждается спецификацией Java Object Serialization и Javadoc: «All subtypes of a serializable class are themselves serializable.»

**Как запретить сериализацию подкласса:**

Прямого языкового механизма нет — нельзя «снять» реализованный интерфейс. Применяются следующие подходы:

**Способ 1 (рекомендуемый): переопределить `writeObject` и `readObject`, бросающие исключение:**
```java
public class Parent implements Serializable {
    private String data;
}

public class Child extends Parent {
    // Запрещаем сериализацию
    private void writeObject(ObjectOutputStream out) throws IOException {
        throw new NotSerializableException("Child cannot be serialized");
    }

    private void readObject(ObjectInputStream in) throws IOException {
        throw new NotSerializableException("Child cannot be deserialized");
    }
}
```
При попытке сериализовать `Child` будет выброшено `NotSerializableException`, которое `ObjectOutputStream` перехватит и пробросит как `IOException`.

**Способ 2: переопределить `writeReplace` / `readResolve`, бросающие `ObjectStreamException`:**
```java
private Object writeReplace() throws ObjectStreamException {
    throw new NotSerializableException("Child cannot be serialized");
}
```

**Способ 3: использовать `serialPersistentFields` равным пустому массиву** — это не запрещает сериализацию, но исключает поля из потока; полноценной защитой не является.

**Дополнительные нюансы:**

- `serialVersionUID` подкласс должен объявлять самостоятельно, даже если суперкласс уже объявил свой — иначе возможны `InvalidClassException` при десериализации после изменений класса.
- Поля подкласса, помеченные `transient`, не сериализуются, но это не влияет на сериализацию суперклассовых полей.
- Если нужно запретить сериализацию на уровне дизайна, лучше сделать родительский класс не `Serializable`, а интерфейс/адаптер, реализующий сериализацию, вынести отдельно.
- Механизм `writeObject`/`readObject` работает потому, что `ObjectOutputStream` проверяет их наличие через рефлексию и вызывает вместо стандартной логики; выброс исключения внутри этих методов прерывает процесс.
### Вопрос 16. Чем интерфейс Externalizable отличается от Serializable? Когда его стоит использовать?

**Категория:** IO/NIO · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Java Externalizable Example - More Efficient Serialization (howtodoinjava.com)](https://howtodoinjava.com/java/serialization/java-externalizable-example/)

При использовании `Serializable` JVM полностью берёт на себя ответственность за выполнение сериализации и десериализации. `Externalizable` перекладывает эту задачу на разработчика: он обязан реализовать методы `writeExternal()` и `readExternal()`.

Классы, реализующие `Externalizable`, должны иметь публичный конструктор без аргументов, поскольку JVM вызывает его при десериализации — до вызова `readExternal()`.

По производительности `Externalizable` превосходит `Serializable`, так как разработчик полностью контролирует процесс, исключая издержки рефлексии в рантайме. `Serializable` относительно медленнее именно из-за этой рефлексии.

В `readExternal()` поля обязательно читаются в той же последовательности и теми же типами, в которых они были записаны в `writeExternal()`.

Рекомендации по выбору:
- `Serializable` — если нужно сериализовать объект целиком без особых требований.
- `Externalizable` — если требуется частичная сериализация, кастомная логика или экономия места/скорости.

#### Ответ от Claude

`Externalizable` расширяет `Serializable` (технически наследует его), но меняет модель ответственности: вся логика чтения и записи состояния объекта ложится на программиста.

**Ключевые различия:**

| Аспект | `Serializable` | `Externalizable` |
|---|---|---|
| Кто управляет процессом | JVM (автоматически) | Разработчик (вручную) |
| Какие поля сохраняются | Все нестатические и нетранзиентные | Только те, что явно записаны |
| Конструктор при десериализации | Конструктор без аргументов первого несериализуемого суперкласса | Обязателен `public` no-arg конструктор самого класса |
| Производительность | Медленнее (рефлексия) | Быстрее (полный контроль, без рефлексии) |
| Формат потока | Определяется JVM | Полностью определяется разработчиком |
| Применимость к внутренним классам | Работает | Не работает — нет no-arg конструктора у нестатических внутренних классов |

**Интерфейс `Externalizable`:**

```java
public interface Externalizable extends Serializable {
    void writeExternal(ObjectOutput out) throws IOException;
    void readExternal(ObjectInput in) throws IOException, ClassNotFoundException;
}
```

Важно: методы `writeExternal`/`readExternal` являются `public`, что означает, что данные объекта становятся общедоступными через эти методы. Для чувствительных данных это нежелательно.

**Пример реализации:**

```java
public class User implements Externalizable {
    private String name;
    private transient String password; // не сериализуем по умолчанию

    public User() {} // обязателен!

    @Override
    public void writeExternal(ObjectOutput out) throws IOException {
        out.writeUTF(name);
        // password намеренно не пишем
    }

    @Override
    public void readExternal(ObjectInput in) throws IOException {
        name = in.readUTF();
        // поля читаем строго в том же порядке, в каком писали
    }
}
```

**Когда использовать `Externalizable`:**
- Нужно сохранить только часть полей (аналог `transient`, но с более тонким контролем).
- Требуется нестандартный формат потока (например, компрессия или шифрование).
- Критична производительность при сериализации очень большого количества объектов.
- Необходимо полное управление обратной совместимостью при эволюции класса.

**Когда оставаться на `Serializable`:**
- Нужна простая сериализация всего объекта.
- Класс является нестатическим внутренним классом.
- Хочется использовать `writeObject`/`readObject` для точечной кастомизации без полного переписывания логики.

Источники: [Oracle Serialization Spec](https://docs.oracle.com/en/java/javase/11/docs/specs/serialization/serial-arch.html), [Oracle Externalizable API](https://docs.oracle.com/en/java/javase/11/docs/api/java.base/java/io/Externalizable.html)

### Вопрос 17. Как кастомизировать процесс сериализации? Что позволяют сделать методы writeObject() и readObject()?

**Категория:** IO/NIO · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Serializable (Java Platform SE 8) — Oracle Javadoc](https://docs.oracle.com/javase/8/docs/api/java/io/Serializable.html)

Классы, требующие специальной обработки в процессе сериализации и десериализации, должны реализовать закрытые методы с точно такими сигнатурами:

```java
private void writeObject(java.io.ObjectOutputStream out) throws IOException
private void readObject(java.io.ObjectInputStream in) throws IOException, ClassNotFoundException
```

Метод `writeObject` отвечает за запись состояния объекта в поток так, чтобы соответствующий `readObject` мог его восстановить. Дефолтный механизм сохранения полей объекта можно вызвать через `out.defaultWriteObject()`.

Метод `readObject` отвечает за чтение данных из потока и восстановление полей. Можно вызвать `in.defaultReadObject()` для восстановления нестатических и нетранзиентных полей — этот вызов также корректно обрабатывает добавленные в новых версиях поля (эволюция класса).

Существует также метод `readObjectNoData()`, который инициализирует состояние объекта когда поток сериализации не содержит этот класс в качестве суперкласса — полезен при несоответствии версий или с изменёнными потоками.

#### Ответ от Claude

Java позволяет классу взять под контроль часть процесса сериализации, не прибегая к полному переписыванию через `Externalizable`. Для этого в классе, реализующем `Serializable`, объявляются `private` методы с конкретными сигнатурами.

**Сигнатуры методов (строго такие, иначе механизм не сработает):**

```java
private void writeObject(ObjectOutputStream out) throws IOException
private void readObject(ObjectInputStream in) throws IOException, ClassNotFoundException
private void readObjectNoData() throws ObjectStreamException
```

Методы обнаруживаются через рефлексию; модификатор `private` обязателен — это гарантирует, что каждый класс в иерархии отвечает только за свои поля.

**Что позволяет `writeObject()`:**

1. Вызвать `out.defaultWriteObject()` для записи всех штатных полей (нестатических, нетранзиентных), а затем дописать дополнительные данные вручную.
2. Полностью заменить дефолтный механизм и писать поля в произвольном порядке/формате через `putFields()` + `writeFields()`.
3. Сериализовать поля, помеченные как `transient`, предварительно преобразовав их в сериализуемый вид.
4. Добавить метаданные версионирования или контрольные суммы.

**Что позволяет `readObject()`:**

1. Вызвать `in.defaultReadObject()` для восстановления штатных полей, а затем прочитать дополнительные данные вручную (в том же порядке, в каком они были записаны).
2. Восстановить транзиентные поля из кастомных данных потока.
3. Провести валидацию восстановленного объекта.
4. Обеспечить миграцию данных при эволюции класса (старый класс не имел поля — присвоить значение по умолчанию).

**Важное правило:** `defaultWriteObject()` или `writeFields()` должны быть вызваны ровно один раз до записи любых дополнительных данных. Аналогично для `defaultReadObject()` или `readFields()`.

**Пример — сериализация транзиентного поля:**

```java
public class Connection implements Serializable {
    private String host;
    private int port;
    private transient Socket socket; // не сериализуем напрямую

    private void writeObject(ObjectOutputStream out) throws IOException {
        out.defaultWriteObject();         // host и port записываются автоматически
        out.writeBoolean(socket != null); // сохраняем признак наличия соединения
    }

    private void readObject(ObjectInputStream in)
            throws IOException, ClassNotFoundException {
        in.defaultReadObject();           // восстанавливаем host и port
        boolean wasConnected = in.readBoolean();
        if (wasConnected) {
            // переустанавливаем соединение
            this.socket = new Socket(host, port);
        }
    }
}
```

**Дополнительные механизмы кастомизации:**

| Метод | Назначение |
|---|---|
| `writeReplace()` | Заменить объект другим перед записью в поток |
| `readResolve()` | Заменить десериализованный объект другим (паттерн Singleton, enum-подобные классы) |
| `readObjectNoData()` | Инициализация, если класс появился в иерархии позже версии потока |
| `serialVersionUID` | Управление совместимостью версий |

**Отличие от `Externalizable`:** `writeObject`/`readObject` позволяют точечно расширить или скорректировать стандартный процесс, тогда как `Externalizable` требует полностью переписать его с нуля и навязывает `public` no-arg конструктор.

Источники: [Oracle Serializable Javadoc](https://docs.oracle.com/javase/8/docs/api/java/io/Serializable.html), [Oracle Serialization Spec — Output](https://docs.oracle.com/en/java/javase/11/docs/specs/serialization/output.html), [Oracle Serialization Spec — Input](https://docs.oracle.com/en/java/javase/11/docs/specs/serialization/input.html)

### Вопрос 18. Что такое memory-mapped file (отображаемый в память файл) и как его создать через FileChannel.map()? В каких сценариях это даёт выигрыш в производительности?

**Категория:** IO/NIO · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Java MappedByteBuffer - Read Memory-mapped Files (howtodoinjava.com)](https://howtodoinjava.com/java/nio/memory-mapped-files-mappedbytebuffer/)

Memory-mapped файл позволяет напрямую отобразить виртуальную память из пространства пользователя на страницы файловой системы, исключая копирование буферов. С memory-mapped файлом можно делать вид, что весь файл находится в памяти, и обращаться к нему как к очень большому массиву.

Для создания отображения нужно открыть `RandomAccessFile`, получить его канал и вызвать `FileChannel.map()`:

```java
RandomAccessFile file = new RandomAccessFile("data.dat", "rw");
MappedByteBuffer buffer = file.getChannel()
    .map(FileChannel.MapMode.READ_WRITE, 0, length);
```

Существует три режима отображения: `READ_ONLY`, `READ_WRITE` и `PRIVATE` (копирование при записи).

Преимущества по производительности:
- Процесс видит данные файла напрямую через память — нет системных вызовов `read()`/`write()`.
- Данные подгружаются по мере обращения через механизм page fault.
- ОС выполняет интеллектуальное кеширование.
- Нет накладных расходов на копирование буферов.
- Очень большие файлы (до 2 ГБ) загружаются без пропорционального потребления памяти.

Лучше всего подходит для: обработки больших файлов (128 МБ+), случайного доступа по разным регионам файла, высокопроизводительных сценариев ввода-вывода.

#### Ответ от Claude

**Что такое memory-mapped file**

Memory-mapped file — это механизм ОС, при котором регион файла проецируется непосредственно в виртуальное адресное пространство процесса. После отображения работа с файлом выглядит как работа с обычным массивом байт в памяти: никаких явных вызовов `read()`/`write()`, никаких промежуточных копирований. Запись в буфер автоматически откатывается в файл (в режиме `READ_WRITE`).

В Java memory-mapped файл представлен классом `java.nio.MappedByteBuffer`, создаваемым через `FileChannel.map()`.

**Сигнатура `FileChannel.map()`:**

```java
public abstract MappedByteBuffer map(FileChannel.MapMode mode,
                                     long position,
                                     long size) throws IOException
```

| Параметр | Описание |
|---|---|
| `mode` | Режим отображения (см. ниже) |
| `position` | Смещение от начала файла (>= 0) |
| `size` | Размер отображаемого региона (>= 0, <= `Integer.MAX_VALUE`) |

**Режимы отображения:**

| Режим | Поведение |
|---|---|
| `MapMode.READ_ONLY` | Только чтение; попытка записи бросает `ReadOnlyBufferException` |
| `MapMode.READ_WRITE` | Чтение и запись; изменения распространяются в файл |
| `MapMode.PRIVATE` | Copy-on-write: изменения видны только текущему процессу и не попадают в файл |

**Полный пример создания и использования:**

```java
try (FileChannel channel = FileChannel.open(
        Path.of("largefile.bin"),
        StandardOpenOption.READ, StandardOpenOption.WRITE)) {

    long fileSize = channel.size();
    MappedByteBuffer buffer = channel.map(
            FileChannel.MapMode.READ_WRITE, 0, fileSize);

    // работаем как с массивом байт
    byte firstByte = buffer.get(0);
    buffer.put(100, (byte) 42); // запись в позицию 100

    // принудительный сброс изменений на диск (актуально для READ_WRITE)
    buffer.force();
}
// канал можно закрыть — маппинг остаётся валидным до GC буфера
```

**Дополнительные методы `MappedByteBuffer`:**

- `load()` — принудительно подгружает содержимое в физическую память (устраняет будущие page fault).
- `isLoaded()` — проверяет, находится ли содержимое в физической памяти (подсказка, не гарантия).
- `force()` / `force(index, length)` — сбрасывает изменения на устройство хранения (только для `READ_WRITE`).

**Когда memory-mapped файл даёт выигрыш в производительности:**

1. **Большие файлы (десятки МБ и выше).** Для файлов размером в несколько десятков килобайт накладные расходы на создание маппинга перевешивают выгоду — Oracle прямо указывает на это в Javadoc `FileChannel.map()`.

2. **Случайный доступ.** Если нужно многократно читать/записывать разные регионы файла, традиционные потоки требуют seek + read на каждое обращение. С маппингом доступ к любой позиции — это просто обращение к индексу буфера.

3. **Межпроцессная коммуникация (IPC).** Несколько процессов могут отображать один и тот же файл в режиме `READ_WRITE` и обмениваться данными через общую память.

4. **Высокочастотное чтение одного файла.** После подгрузки страниц в кэш ОС повторные чтения выполняются без дисковых операций.

5. **Базы данных, индексы, журналы.** Chronicle Map, MapDB, RocksDB и аналогичные решения используют memory-mapped файлы именно потому, что это позволяет работать с данными большего объёма, чем доступная RAM, без явного управления кешем.

**Ограничения и нюансы:**

- Максимальный размер одного маппинга ограничен `Integer.MAX_VALUE` (~2 ГБ); для больших файлов создают несколько перекрывающихся маппингов.
- `MappedByteBuffer` не освобождается через `close()` — требуется GC или недокументированный `Cleaner` (до Java 9) / `ByteBuffer.cleaner().clean()` через `sun.misc.Unsafe` (нежелательно в продакшне).
- Изменения видимости между процессами зависят от ОС и не гарантированы спецификацией.
- Маппинг остаётся валидным после закрытия `FileChannel`.

Источники: [Oracle FileChannel Javadoc](https://docs.oracle.com/javase/8/docs/api/java/nio/channels/FileChannel.html), [Oracle MappedByteBuffer SE 21](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/nio/MappedByteBuffer.html)
### Вопрос 19. Как работает WatchService из java.nio.file? Как с его помощью отслеживать создание, изменение и удаление файлов в каталоге?

**Категория:** IO/NIO · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Watching a Directory for Changes — The Java Tutorials (docs.oracle.com)](https://docs.oracle.com/javase/tutorial/essential/io/notification.html)

WatchService — API уведомлений об изменениях файловой системы из пакета `java.nio.file`, появившийся в Java 7. Принцип работы:

1. Создать экземпляр через `FileSystems.getDefault().newWatchService()`.
2. Зарегистрировать объект `Path` (каталог) с указанием интересующих типов событий: `ENTRY_CREATE`, `ENTRY_DELETE`, `ENTRY_MODIFY`, `OVERFLOW`.
3. Запустить цикл обработки событий: вызвать `watcher.take()` (блокирующий вызов) или `watcher.poll()` (неблокирующий), получить `WatchKey`, обойти его события через `key.pollEvents()`, обработать каждое, а затем обязательно вызвать `key.reset()`, чтобы ключ снова мог принимать события.

Пример регистрации и обработки:
```java
WatchService watcher = FileSystems.getDefault().newWatchService();
Path dir = Paths.get("/tmp/watched");
dir.register(watcher, ENTRY_CREATE, ENTRY_DELETE, ENTRY_MODIFY);

for (;;) {
    WatchKey key = watcher.take(); // блокируется до события
    for (WatchEvent<?> event : key.pollEvents()) {
        WatchEvent.Kind<?> kind = event.kind();
        if (kind == OVERFLOW) continue;
        Path filename = ((WatchEvent<Path>) event).context();
        System.out.println(kind + ": " + filename);
    }
    if (!key.reset()) break; // ключ стал невалидным
}
```

Четыре типа событий из `StandardWatchEventKinds`:
- `ENTRY_CREATE` — создана запись в каталоге;
- `ENTRY_DELETE` — запись удалена;
- `ENTRY_MODIFY` — запись изменена;
- `OVERFLOW` — часть событий могла быть потеряна (принимается всегда, независимо от регистрации).

#### Ответ от Claude

**Архитектура WatchService**

`WatchService` реализует паттерн «наблюдатель» на уровне ОС: где возможно, делегирует к нативным механизмам (`inotify` на Linux, `FSEvents`/`kqueue` на macOS, `ReadDirectoryChangesW` на Windows), а там, где нативной поддержки нет — использует поллинг.

**Жизненный цикл WatchKey**

Каждая регистрация каталога возвращает `WatchKey`. Ключ проходит через состояния:

- **Ready** (начальное) — принимает события;
- **Signalled** — есть накопленные события; ключ попадает в очередь сервиса;
- **Invalid** — ключ отменён (`cancel()`), отслеживаемый каталог удалён, или сервис закрыт.

Вызов `key.reset()` после обработки всех событий переводит ключ обратно в состояние Ready. Если `reset()` возвращает `false`, ключ стал невалидным и мониторинг этого каталога прекратился.

**Методы получения ключей**

| Метод | Поведение |
|---|---|
| `take()` | Блокирует поток до появления события |
| `poll()` | Немедленно возвращает ключ или `null` |
| `poll(long, TimeUnit)` | Ждёт не дольше указанного времени |

**Практический шаблон использования**

```java
try (WatchService watchService = FileSystems.getDefault().newWatchService()) {
    Path dir = Paths.get("/var/data");
    dir.register(watchService,
        StandardWatchEventKinds.ENTRY_CREATE,
        StandardWatchEventKinds.ENTRY_MODIFY,
        StandardWatchEventKinds.ENTRY_DELETE);

    while (true) {
        WatchKey key = watchService.take();
        for (WatchEvent<?> event : key.pollEvents()) {
            WatchEvent.Kind<?> kind = event.kind();
            if (kind == StandardWatchEventKinds.OVERFLOW) {
                // переполнение: часть событий потеряна, пересканируем каталог
                continue;
            }
            Path changed = (Path) event.context(); // только имя файла, не полный путь
            Path fullPath = dir.resolve(changed);
            System.out.printf("Событие %s: %s%n", kind.name(), fullPath);
        }
        if (!key.reset()) {
            break; // каталог удалён или сервис закрыт
        }
    }
} catch (IOException | InterruptedException e) {
    Thread.currentThread().interrupt();
}
```

**Важные оговорки**

1. `event.context()` возвращает только имя файла (не полный путь) — нужно разрешать через `dir.resolve(filename)`.
2. Одно физическое изменение файла может порождать несколько событий `ENTRY_MODIFY` — это платформенно-зависимо.
3. `OVERFLOW` не гарантирует перечисление потерянных событий — при его получении рекомендуется пересканировать каталог вручную.
4. Рекурсивное слежение (поддиректории) из коробки не поддерживается: каждый подкаталог нужно регистрировать отдельно.
5. Изменения на сетевых дисках могут не детектироваться.
6. Сервис является `Closeable` — обязательно закрывать в `try-with-resources`.
7. `WatchService` потокобезопасен, но вызывать `reset()` на одном ключе должен только один поток.

### Вопрос 20. Что такое Scatter/Gather в NIO (ScatteringByteChannel / GatheringByteChannel)? В каких задачах это полезно?

**Категория:** IO/NIO · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Java NIO Scatter / Gather — jenkov.com](https://jenkov.com/tutorials/java-nio/scatter-gather.html)

Scatter/Gather — механизм Java NIO для чтения/записи данных между каналом и несколькими буферами за один системный вызов.

**Scattering Read** (рассеивающее чтение): операция читает данные из канала сразу в массив буферов. Данные последовательно заполняют буфер за буфером — переходит к следующему только после того, как текущий заполнен до предела (`limit`).

```java
ByteBuffer header = ByteBuffer.allocate(128);
ByteBuffer body   = ByteBuffer.allocate(1024);
ByteBuffer[] buffers = { header, body };
channel.read(buffers);
```

**Gathering Write** (собирающая запись): операция записывает данные из нескольких буферов в канал за один вызов. Записывает только данные между `position` и `limit` каждого буфера, поэтому хорошо подходит для разделов переменной длины.

```java
ByteBuffer header = ByteBuffer.allocate(128);
ByteBuffer body   = ByteBuffer.allocate(1024);
// заполнить header и body...
ByteBuffer[] buffers = { header, body };
channel.write(buffers);
```

**Scattering read** хорошо подходит для сообщений с фиксированным заголовком — каждый буфер получает свою часть. **Gathering write** лучше справляется с разделами переменной длины, поскольку записывает ровно столько данных, сколько реально содержит буфер.

#### Ответ от Claude

**Определение**

Scatter/Gather — концепция векторного ввода-вывода (vectored I/O): одна операция канала работает сразу с массивом буферов, что позволяет обойтись без промежуточного копирования.

- `ScatteringByteChannel` расширяет `ReadableByteChannel` и добавляет методы `read(ByteBuffer[])` и `read(ByteBuffer[], int offset, int length)`.
- `GatheringByteChannel` расширяет `WritableByteChannel` и добавляет методы `write(ByteBuffer[])` и `write(ByteBuffer[], int offset, int length)`.

Оба интерфейса реализуют `FileChannel`, `SocketChannel`, `DatagramChannel` и `Pipe.SourceChannel` / `Pipe.SinkChannel`.

**Scattering Read — рассеивающее чтение**

Читает байты из канала последовательно в каждый буфер массива: сначала заполняет `buffers[0]` до `limit`, затем переходит к `buffers[1]` и т.д.

```java
ByteBuffer fixedHeader = ByteBuffer.allocate(8);   // 8 байт: маркер + длина тела
ByteBuffer body        = ByteBuffer.allocate(4096);

long bytesRead = channel.read(new ByteBuffer[]{ fixedHeader, body });

fixedHeader.flip();
body.flip();
```

Ограничение: буфер должен заполниться полностью, прежде чем начнётся заполнение следующего. Поэтому scattering read идеально подходит для заголовков фиксированного размера, но не для полей переменной длины.

**Gathering Write — собирающая запись**

Записывает данные из нескольких буферов в канал последовательно. Для каждого буфера записываются байты между `position` и `limit`, что позволяет гибко управлять тем, сколько данных реально попадёт в канал.

```java
ByteBuffer header = ByteBuffer.allocate(8);
ByteBuffer body   = ByteBuffer.allocate(512);
// ... заполнить данными, вызвать flip() ...
long written = channel.write(new ByteBuffer[]{ header, body });
```

**Когда это полезно**

| Сценарий | Польза |
|---|---|
| Сетевые протоколы (TCP/UDP) с фиксированным заголовком и переменным телом | Один `read()`/`write()` вместо нескольких системных вызовов |
| Бинарные форматы файлов (заголовок + данные) | Точное разбиение по буферам без ручного `System.arraycopy` |
| Высоконагруженные серверы (Netty, NIO-фреймворки) | Снижение числа системных вызовов, меньше копирований памяти |
| Протоколы с несколькими секциями разной длины (gathering write) | Каждая секция в своём буфере, запись одним вызовом |

**Преимущество перед обычным I/O**

Без scatter/gather программист вынужден либо копировать данные в один большой буфер перед записью, либо делать несколько отдельных вызовов `write()`. Scatter/Gather перекладывает эту работу на ОС, что эффективнее — особенно там, где JVM может делегировать к нативным `readv()`/`writev()` системным вызовам.
