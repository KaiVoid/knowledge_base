### Вопрос 1. Что такое @Entity в JPA? Какие обязательные условия должны выполняться для класса, помеченного этой аннотацией?

**Категория:** JPA/Hibernate · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Requirements for Entity Classes — The Java EE 6 Tutorial](https://docs.oracle.com/cd/E19798-01/821-1841/bnbqb/index.html)

Аннотация `@Entity` помечает класс как постоянный объект предметной области. Как правило, сущность представляет таблицу в реляционной базе данных, а каждый экземпляр сущности соответствует строке в этой таблице.

Обязательные требования к классу сущности:

1. Класс должен быть помечен аннотацией `javax.persistence.Entity`.
2. Класс должен иметь `public` или `protected` конструктор без аргументов (могут быть и другие конструкторы).
3. Класс не должен быть объявлен как `final`; методы и поля, участвующие в персистентности, также не должны быть `final`.
4. Если экземпляр сущности передаётся по значению как detached-объект (например, через удалённый интерфейс session bean), класс должен реализовывать `Serializable`.
5. Поля-переменные персистентного состояния должны быть объявлены `private`, `protected` или package-private; доступ к ним клиентов должен осуществляться через методы-аксессоры или бизнес-методы.
6. Сущность может расширять как классы-сущности, так и обычные классы; обычные классы могут расширять классы-сущности. Разрешены абстрактные классы-сущности.

#### Ответ от Claude

`@Entity` — аннотация из пакета `jakarta.persistence` (ранее `javax.persistence`), которая сообщает JPA-провайдеру (например, Hibernate), что данный Java-класс является управляемой сущностью и должен отображаться на таблицу базы данных.

**Обязательные условия для класса-сущности:**

1. **Аннотация `@Entity`** — класс должен быть помечен этой аннотацией.
2. **Беспараметрический конструктор** — должен быть `public` или `protected`. Он необходим провайдеру для создания экземпляров через рефлексию. Дополнительные конструкторы с параметрами допускаются.
3. **Класс не должен быть `final`** — провайдер (Hibernate) создаёт прокси-подклассы для реализации ленивой загрузки. Если класс или его персистентные методы/поля объявлены `final`, это невозможно.
4. **Наличие `@Id`** — у сущности должен быть объявлен первичный ключ (поле или свойство, помеченное `@Id`). Без него провайдер не сможет идентифицировать строку.
5. **Класс должен быть top-level** — enum и interface не могут быть сущностями.

**Дополнительные рекомендации (не строго обязательные по спецификации, но практически значимые):**

- Реализация `Serializable` требуется, только если экземпляры будут передаваться по значению как detached-объекты (например, в распределённых системах или через HTTP-сессию).
- Персистентные поля рекомендуется объявлять `private`/`protected` с геттерами/сеттерами.

**Пример минимально корректной сущности:**

```java
@Entity
public class Product {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;

    public Product() {} // обязателен

    // геттеры и сеттеры
}
```

### Вопрос 2. Какие стратегии генерации первичного ключа существуют в JPA (@GeneratedValue)? Чем отличаются IDENTITY, SEQUENCE, TABLE и AUTO?

**Категория:** JPA/Hibernate · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [GenerationType — Java EE 7 Specification APIs](https://docs.oracle.com/javaee/7/api/javax/persistence/GenerationType.html)

JPA определяет четыре стратегии генерации первичных ключей через перечисление `GenerationType`:

- **`AUTO`** — провайдер персистентности автоматически выбирает подходящую стратегию для конкретной базы данных. Является значением по умолчанию для атрибута `strategy`.
- **`IDENTITY`** — провайдер использует колонку автоинкремента базы данных для генерации первичных ключей.
- **`SEQUENCE`** — провайдер использует объект последовательности базы данных. Может использоваться совместно с аннотацией `@SequenceGenerator`.
- **`TABLE`** — провайдер использует специальную таблицу базы данных для хранения значений первичных ключей. Может использоваться совместно с аннотацией `@TableGenerator`.

#### Ответ от Claude

Аннотация `@GeneratedValue` используется вместе с `@Id` для указания того, как JPA-провайдер должен автоматически генерировать значения первичного ключа.

```java
@Id
@GeneratedValue(strategy = GenerationType.SEQUENCE)
private Long id;
```

**Стратегии и их различия:**

| Стратегия  | Механизм                              | Поддержка БД                          | Batch INSERT | Производительность |
|------------|---------------------------------------|---------------------------------------|--------------|--------------------|
| `IDENTITY` | Автоинкремент колонки в БД            | MySQL, PostgreSQL, SQL Server, H2     | Нет          | Высокая            |
| `SEQUENCE` | Объект SEQUENCE в БД                  | Oracle, PostgreSQL, DB2, H2           | Да           | Высокая            |
| `TABLE`    | Отдельная таблица-генератор           | Любая БД                              | Да           | Низкая             |
| `AUTO`     | Провайдер выбирает стратегию сам      | Любая БД                              | Зависит      | Зависит            |

**`IDENTITY`**
БД генерирует ID после вставки строки (`AUTO_INCREMENT` в MySQL, `SERIAL` в PostgreSQL). Hibernate узнаёт ID только после выполнения `INSERT`. Из-за этого **пакетная вставка (batch insert) невозможна** — каждый `INSERT` выполняется немедленно.

**`SEQUENCE`**
Hibernate заблаговременно запрашивает следующее значение у объекта последовательности (`SELECT NEXTVAL('seq_name')`). Это позволяет назначить ID до вставки и **поддерживает batch insert**. Рекомендуемая стратегия при использовании Oracle или PostgreSQL. По умолчанию Hibernate использует одну общую последовательность `hibernate_sequence` (Hibernate 5) или генерирует последовательность на каждую таблицу (Hibernate 6).

**`TABLE`**
Hibernate хранит счётчики в отдельной таблице (`hibernate_sequences`). Работает с любой СУБД, но требует дополнительных `SELECT ... FOR UPDATE` / `UPDATE` запросов при каждой генерации ID, что **снижает производительность**. Применяется редко — только когда БД не поддерживает ни автоинкремент, ни последовательности.

**`AUTO`**
Провайдер выбирает стратегию сам на основе типа данных поля и возможностей диалекта БД. В Hibernate 5 для числовых типов обычно выбирается `SEQUENCE`, в Hibernate 4 и ниже — может быть `TABLE`. Не рекомендуется для продакшена из-за непредсказуемости поведения при смене версии Hibernate или СУБД.

**Практическая рекомендация:** используйте `SEQUENCE` с явно объявленным `@SequenceGenerator` — это даёт максимальный контроль и производительность.

### Вопрос 3. Что такое контекст персистентности (Persistence Context)? Чем он отличается от SessionFactory/EntityManagerFactory?

**Категория:** JPA/Hibernate · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Persistence Contexts — Hibernate ORM 5.1 User Guide](https://docs.hibernate.org/orm/5.1/userguide/html_single/chapters/pc/PersistenceContext.html)

Контекст персистентности — это «контекст для работы с постоянными данными». Как `org.hibernate.Session`, так и `javax.persistence.EntityManager` представляют именно такой контекст.

В рамках контекста персистентности каждая сущность проходит через четыре состояния:

1. **Transient (Переходное)** — только что созданный объект, не связанный ни с каким контекстом персистентности. Как правило, не имеет идентификатора и не имеет представления в БД.
2. **Managed/Persistent (Управляемое)** — объект с присвоенным идентификатором, связанный с активным контекстом персистентности. Может ещё не существовать физически в БД.
3. **Detached (Отсоединённое)** — объект с идентификатором, но уже не связанный с контекстом (после закрытия контекста или явного вызова `evict`/`detach`).
4. **Removed (Удалённое)** — объект, запланированный к удалению из БД, ещё связанный с контекстом.

Область действия контекста персистентности ограничена жизненным циклом `Session`/`EntityManager`. После закрытия контекста все связанные сущности переходят в состояние Detached.

#### Ответ от Claude

**Контекст персистентности (Persistence Context)** — это набор управляемых экземпляров сущностей, ассоциированных с конкретной единицей работы (Unit of Work). Он реализует паттерн Identity Map: для каждого уникального `(тип, id)` в рамках одного контекста существует ровно один Java-объект. Контекст отслеживает изменения сущностей (dirty checking) и синхронизирует их с БД при сбросе (flush).

В JPA контекст персистентности доступен через `EntityManager`, в Hibernate — через `Session`.

**Сравнение с SessionFactory / EntityManagerFactory:**

| Характеристика            | Persistence Context (`Session` / `EntityManager`) | `SessionFactory` / `EntityManagerFactory`      |
|---------------------------|----------------------------------------------------|------------------------------------------------|
| Уровень                   | Единица работы (транзакция / запрос)               | Уровень приложения                             |
| Время жизни               | Короткое (per-request или per-transaction)         | Всё время жизни приложения                     |
| Потокобезопасность        | Не потокобезопасен (один поток)                    | Потокобезопасен, создаётся один раз            |
| Назначение                | Управляет сущностями, выполняет запросы            | Создаёт Session/EntityManager, хранит метаданные и кэш 2-го уровня |
| Кэширование               | Кэш 1-го уровня (встроен, всегда активен)          | Кэш 2-го уровня (опционален, разделяется между сессиями) |

**Жизненный цикл сущности в контексте персистентности:**

```
new Object()         persist()          flush/commit
   Transient  ───────────► Managed ──────────────► БД
                              │
                 close() /    │  remove()
                 detach()     ▼
                           Removed
                Detached ◄──────── Managed
```

**Типы контекста персистентности (по области действия):**

- **Transaction-scoped** (по умолчанию в JEE/Spring) — контекст живёт в рамках одной транзакции. После коммита все сущности становятся detached.
- **Extended** — контекст переживает несколько транзакций (используется, например, в stateful session beans или с `EXTENDED` в Spring).

**SessionFactory vs EntityManagerFactory:**

`EntityManagerFactory` — стандартный JPA-интерфейс; `SessionFactory` — Hibernate-специфичный. В современных версиях Hibernate `SessionFactory` расширяет `EntityManagerFactory`. Используйте `EntityManagerFactory` для переносимости между провайдерами; `SessionFactory` — если нужны Hibernate-специфичные возможности (например, `StatelessSession`, явное управление кэшем 2-го уровня через `SessionFactory.getCache()`).

### Вопрос 4. Назовите состояния жизненного цикла JPA-сущности. Как сущность переходит из одного состояния в другое?

**Категория:** JPA/Hibernate · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [8.2. Entity Lifecycle Management — Oracle Fusion Middleware](https://docs.oracle.com/cd/E21764_01/apirefs.1111/e13946/ejb3_overview_em_lifecycle.html)

JPA определяет четыре состояния сущности:

1. **New (Новая/Transient)** — сущность существует в памяти, но ещё не сохранена и не имеет идентификатора в базе данных.
2. **Managed (Управляемая/Persistent)** — сущность ассоциирована с `EntityManager` и синхронизируется с базой данных.
3. **Detached (Отсоединённая)** — сущность ранее была управляемой, но больше не ассоциирована ни с каким `EntityManager`.
4. **Removed (Удалённая)** — сущность управляемая, но помечена для удаления из хранилища.

Переходы между состояниями через методы `EntityManager`:

- `persist(entity)`: New → Managed (при вызове на Detached бросает `IllegalArgumentException`)
- `remove(entity)`: Managed → Removed (при вызове на Detached бросает `IllegalArgumentException`)
- `merge(entity)`: Detached → Managed (возвращает новый управляемый экземпляр; оригинал остаётся detached); New → Managed; при вызове на Removed бросает `IllegalArgumentException`
- `refresh(entity)`: Managed → Managed (перечитывает состояние из БД; при вызове на Detached бросает `IllegalArgumentException`)
- Закрытие контекста (`close()`) или `detach(entity)`: Managed → Detached

#### Ответ от Claude

JPA-спецификация (JSR-338) определяет четыре состояния жизненного цикла сущности:

**1. Transient (Транзитная / New)**
Объект только что создан оператором `new`, не имеет идентификатора базы данных и не связан ни с каким контекстом персистентности (`PersistenceContext`). Hibernate «не знает» о нём.

**2. Persistent / Managed (Управляемая)**
Сущность имеет идентификатор и ассоциирована с активным `PersistenceContext`. Все изменения автоматически отслеживаются (dirty checking) и будут записаны в БД при сбросе (flush) контекста. Запись в БД при этом может ещё не существовать (вставка произойдёт при flush/commit).

**3. Detached (Отсоединённая)**
Сущность имеет идентификатор, но контекст персистентности закрыт или сущность была явно отсоединена. Изменения больше не отслеживаются автоматически.

**4. Removed (Удалённая)**
Сущность управляемая, помечена для удаления. Будет удалена из БД при ближайшем flush/commit.

**Переходы между состояниями:**

| Метод | Откуда | Куда | Примечание |
|---|---|---|---|
| `em.persist(e)` | Transient | Managed | На Detached бросает `IllegalArgumentException` |
| `em.remove(e)` | Managed | Removed | На Detached бросает `IllegalArgumentException` |
| `em.merge(e)` | Detached | Managed | Возвращает новый managed-экземпляр; на Removed бросает исключение |
| `em.merge(e)` | Transient | Managed | Создаёт новый managed-экземпляр |
| `em.detach(e)` / `em.clear()` / `em.close()` | Managed | Detached | |
| `em.refresh(e)` | Managed | Managed | Перечитывает состояние из БД |
| `em.find()` / JPQL-запрос | — | Managed | Загружает из БД сразу как managed |

**Hibernate-специфичные методы** (`Session`-API):
- `session.save(e)` / `session.persist(e)`: Transient → Persistent
- `session.update(e)` / `session.saveOrUpdate(e)` / `session.lock(e, ...)`: Detached → Persistent (реаттач)
- `session.delete(e)`: Managed/Detached → Removed (в отличие от JPA `remove()`, принимает и Detached)
- `session.evict(e)`: Managed → Detached

**Важные нюансы:**
- `persist()` vs `merge()`: `persist()` работает только с Transient-сущностями и сразу делает их managed; `merge()` принимает Detached и возвращает *новый* managed-экземпляр (оригинальный объект остаётся detached).
- Каскадирование: переходы можно автоматически распространять на связанные сущности через `CascadeType` (PERSIST, MERGE, REMOVE, REFRESH, DETACH, ALL).
- `persist()`, `remove()`, `merge()` требуют активной транзакции.

### Вопрос 5. В чём разница между FetchType.LAZY и FetchType.EAGER? Какой тип загрузки используется по умолчанию для @OneToMany и @ManyToOne?

**Категория:** JPA/Hibernate · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [FetchType (Java EE 7 Specification APIs) — docs.oracle.com](https://docs.oracle.com/javaee/7/api/javax/persistence/FetchType.html)

Согласно спецификации Java EE 7:

- **EAGER** — *требование* к провайдеру персистентности: данные должны быть загружены немедленно (eagerly fetched).
- **LAZY** — *подсказка* провайдеру персистентности: данные могут быть загружены лениво (lazily fetched) при первом обращении. Провайдер вправе проигнорировать эту подсказку и загрузить данные сразу.

Для аннотации `@OneToMany` значение по умолчанию: `javax.persistence.FetchType.LAZY`.
Для аннотации `@ManyToOne` значение по умолчанию: `javax.persistence.FetchType.EAGER`.

#### Ответ от Claude

**FetchType.EAGER (жадная загрузка)**
Связанные данные загружаются немедленно вместе с родительской сущностью — в том же SQL-запросе (или в дополнительном, но до возврата результата). Это *требование* к провайдеру: данные гарантированно будут доступны при возврате объекта.

**FetchType.LAZY (ленивая загрузка)**
Связанные данные не загружаются сразу. Hibernate создаёт прокси-объект (для `@ManyToOne`/`@OneToOne`) или пустую прокси-коллекцию (для `@OneToMany`/`@ManyToMany`). Реальный SQL-запрос к БД выполняется только при первом обращении к данным. Это *подсказка* провайдеру: по спецификации он вправе загрузить данные и сразу, хотя Hibernate на практике уважает LAZY.

**Значения по умолчанию согласно JPA-спецификации (JSR-338):**

| Аннотация | FetchType по умолчанию |
|---|---|
| `@OneToMany` | **LAZY** |
| `@ManyToMany` | **LAZY** |
| `@ManyToOne` | **EAGER** |
| `@OneToOne` | **EAGER** |
| `@Basic` | **EAGER** |
| `@ElementCollection` | **LAZY** |

**Ключевые различия и последствия:**

| Аспект | EAGER | LAZY |
|---|---|---|
| Когда загружается | При загрузке родителя | При первом обращении |
| SQL-запросы | Немедленно (JOIN или дополнительный SELECT) | Откладываются |
| Доступность вне сессии | Данные уже загружены | `LazyInitializationException` если сессия закрыта |
| Риск | Избыточная загрузка данных | N+1 запросов, `LazyInitializationException` |

**Рекомендации:**
- Hibernate и JPA-эксперты (включая Влада Михалча) рекомендуют устанавливать все ассоциации как LAZY и явно указывать жадную загрузку там, где нужно — через `JOIN FETCH` в JPQL или `@EntityGraph`.
- Значение по умолчанию EAGER для `@ManyToOne` считается практически неудачным решением в спецификации: оно провоцирует N+1 проблему при коллекциях.
- `LazyInitializationException` возникает, когда к LAZY-коллекции или прокси обращаются вне открытой сессии/транзакции (типичная ошибка в слое представления — т.н. «Open Session in View» антипаттерн).

### Вопрос 6. Что такое проблема N+1 запросов? Приведите пример, когда она возникает при использовании @OneToMany с LAZY-загрузкой.

**Категория:** JPA/Hibernate · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [The N+1 Query Problem — vladmihalcea.com](https://vladmihalcea.com/n-plus-1-query-problem/)

Проблема N+1 возникает, когда фреймворк доступа к данным выполняет N дополнительных SQL-запросов для получения тех же данных, которые можно было бы получить в одном основном запросе.

Пример: сущности `PostComment` с ленивой связью `@ManyToOne` на `Post`:

```java
List<PostComment> comments = entityManager
    .createQuery("select pc from PostComment pc", PostComment.class)
    .getResultList();

for (PostComment comment : comments) {
    // Каждое обращение к comment.getPost().getTitle()
    // триггерит отдельный SELECT к таблице post
    LOGGER.info("Post: '{}', Review: '{}'",
        comment.getPost().getTitle(),
        comment.getReview());
}
```

Если в БД 4 комментария — выполняется 1 запрос за комментариями + 4 запроса за постами = 5 запросов вместо 1.

Решение — использовать `JOIN FETCH`:

```java
List<PostComment> comments = entityManager.createQuery(
    "select pc from PostComment pc join fetch pc.post p",
    PostComment.class)
    .getResultList();
```

Это генерирует единственный SQL с JOIN, устраняя N+1.

#### Ответ от Claude

**Суть проблемы N+1**

Проблема N+1 — это ситуация, при которой для загрузки N связанных объектов выполняется 1 «основной» SQL-запрос плюс N отдельных SQL-запросов (по одному на каждый объект), тогда как логически всё можно получить одним запросом с JOIN.

**Пример с @OneToMany и LAZY-загрузкой**

```java
@Entity
public class Author {
    @Id
    Long id;
    String name;

    @OneToMany(mappedBy = "author", fetch = FetchType.LAZY)
    List<Book> books;
}

@Entity
public class Book {
    @Id
    Long id;
    String title;

    @ManyToOne
    Author author;
}
```

```java
// 1 запрос: SELECT * FROM author
List<Author> authors = em.createQuery("select a from Author a", Author.class)
    .getResultList();

for (Author author : authors) {
    // Для каждого автора — отдельный запрос:
    // SELECT * FROM book WHERE author_id = ?
    System.out.println(author.getBooks().size());
}
```

Если авторов 100, в базу уйдёт 101 SQL-запрос: 1 + 100.

**Почему возникает**
Hibernate при LAZY-загрузке помещает в поле `books` прокси-коллекцию. Как только код обращается к содержимому коллекции (`.size()`, итерация и т.д.), Hibernate выполняет SELECT по идентификатору родителя. Это происходит внутри цикла — по одному запросу на каждую итерацию.

**Решения:**

1. **JOIN FETCH в JPQL** — наиболее распространённый и рекомендуемый способ:
   ```java
   List<Author> authors = em.createQuery(
       "select distinct a from Author a join fetch a.books", Author.class)
       .getResultList();
   // Генерирует: SELECT DISTINCT a.*, b.* FROM author a JOIN book b ON b.author_id = a.id
   ```

2. **@EntityGraph** (JPA 2.1+) — декларативный способ без изменения JPQL:
   ```java
   @EntityGraph(attributePaths = {"books"})
   List<Author> findAll(); // в Spring Data JPA
   ```

3. **@BatchSize** (Hibernate-специфично) — загружает коллекции пачками:
   ```java
   @OneToMany(mappedBy = "author", fetch = FetchType.LAZY)
   @BatchSize(size = 25)
   List<Book> books;
   // Вместо N запросов — ceil(N/25) запросов
   ```

4. **@Fetch(FetchMode.SUBSELECT)** (Hibernate-специфично) — загружает все коллекции одним подзапросом:
   ```java
   @OneToMany(mappedBy = "author")
   @Fetch(FetchMode.SUBSELECT)
   List<Book> books;
   // SELECT * FROM book WHERE author_id IN (SELECT id FROM author WHERE ...)
   ```

**Важные нюансы:**
- N+1 может возникать и при EAGER-загрузке: Hibernate иногда выполняет отдельные SELECT вместо JOIN для коллекций (зависит от конфигурации).
- При использовании `JOIN FETCH` для коллекций может возникнуть проблема дублирования строк — решается через `DISTINCT` в JPQL или `Set` вместо `List`.
- Для обнаружения N+1 в продакшене используют инструменты: Hibernate statistics, p6spy, datasource-proxy, или просто включают логирование SQL (`show_sql=true`, `format_sql=true`).
- Spring Data JPA не решает N+1 автоматически — разработчик должен явно использовать `@EntityGraph` или `@Query` с `JOIN FETCH`.
### Вопрос 7. Что такое кэш первого уровня в Hibernate? Когда он очищается и можно ли его отключить?

**Категория:** JPA/Hibernate · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Hibernate Caching - First Level Cache (DigitalOcean)](https://www.digitalocean.com/community/tutorials/hibernate-caching-first-level-cache)

Кэш первого уровня в Hibernate — это хранилище объектов-сущностей, привязанное к объекту `Session`. Он автоматически кэширует сущности, полученные в ходе операций с базой данных, сокращая количество повторных обращений к БД. Объекты, закэшированные в одной сессии, не видны другим сессиям — это обеспечивает изоляцию данных. Кэш очищается тремя способами: `session.clear()` удаляет все объекты из кэша, `session.evict(object)` удаляет конкретный объект, `session.close()` закрывает сессию и автоматически очищает весь кэш. Кэш первого уровня включён по умолчанию, и отключить его нельзя — нет никакой конфигурационной опции для этого.

#### Ответ от Claude

**Что такое кэш первого уровня**

Кэш первого уровня (First-Level Cache, L1) — это встроенный кэш Hibernate, который неразрывно связан с объектом `Session` (или `EntityManager` в терминах JPA). Он также называется *persistence context* — контекст персистентности. Каждый раз, когда вы загружаете сущность через `get()`, `load()`, `list()`, `find()` и т.д., объект помещается в этот кэш. При повторном обращении к той же сущности в рамках одной сессии Hibernate возвращает её из кэша, не выполняя лишний SQL-запрос.

Кэш привязан к конкретной сессии: объекты из одной `Session` не видны другим `Session`. Когда сессия закрывается, все закэшированные объекты теряются.

**Когда кэш очищается**

Существуют три способа очистки:

1. **`session.close()` / закрытие контекста** — кэш очищается автоматически при завершении сессии. Все managed-сущности переходят в состояние detached или теряются.
2. **`session.clear()`** — явная полная очистка: все объекты удаляются из кэша, все не сброшенные (unflushed) изменения отменяются.
3. **`session.evict(entity)`** — точечное удаление одного конкретного объекта и его коллекций из кэша. Объект переходит в состояние detached.

Важно отличать *flush* от *clear*: `session.flush()` синхронизирует состояние кэша с базой данных (выполняет pending SQL), но не очищает кэш. `clear()` именно удаляет объекты из кэша.

При пакетной обработке больших объёмов данных рекомендуется периодически вызывать `flush()` + `clear()`, чтобы не допустить переполнения памяти.

**Можно ли отключить кэш первого уровня**

Нет. Кэш первого уровня является фундаментальной частью работы Hibernate и отключить его нельзя — нет никакой конфигурационной опции для этого. Это принципиальное отличие от кэша второго уровня (L2), который опционален и настраивается.

Единственная альтернатива — использовать `StatelessSession`. Stateless-сессия не имеет первого уровня кэша, не взаимодействует с кэшем второго уровня и кэшем запросов. Это полезно для пакетных операций, но при этом теряются все преимущества управления жизненным циклом сущностей.

### Вопрос 8. Чем отличаются методы EntityManager: persist(), merge(), remove() и detach()?

**Категория:** JPA/Hibernate · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [EntityManager — Oracle Java EE 7 Specification APIs](https://docs.oracle.com/javaee/7/api/javax/persistence/EntityManager.html)

Согласно спецификации Java EE 7:

- **`persist(Object entity)`** — делает экземпляр управляемым и персистентным. Сущность будет вставлена в базу данных при следующем flush или коммите транзакции. Выбрасывает `EntityExistsException`, если сущность уже существует, и `TransactionRequiredException`, если нет активной транзакции.
- **`merge(T entity)`** — переносит состояние переданной сущности в текущий контекст персистентности. Возвращает управляемый экземпляр (может быть другим объектом). Выбрасывает `IllegalArgumentException`, если передан removed-объект.
- **`remove(Object entity)`** — помечает управляемую сущность для удаления. Сущность будет удалена из базы данных при flush/коммите. Выбрасывает `IllegalArgumentException` для detached-сущностей.
- **`detach(Object entity)`** — переводит управляемую сущность в состояние detached, убирая её из контекста персистентности. Не сброшенные изменения не будут синхронизированы с базой. Доступен начиная с JPA 2.0.

#### Ответ от Claude

**Состояния сущности в JPA**

Прежде чем разбирать методы, важно понять четыре состояния сущности:

- **Transient (новая)** — объект создан через `new`, не связан с контекстом персистентности, нет записи в БД.
- **Managed (управляемая)** — объект находится под управлением `EntityManager`, изменения отслеживаются автоматически.
- **Detached (отсоединённая)** — объект был managed, но контекст закрылся или вызван `detach()`. Объект существует в памяти, но изменения не отслеживаются.
- **Removed (удалённая)** — объект помечен для удаления, будет удалён при flush/коммите.

**`persist(Object entity)`**

Переводит сущность из состояния *transient* → *managed*. При следующем flush выполняется `INSERT` в базу данных.

Поведение по состояниям:
- Transient → Managed (штатный сценарий)
- Managed → без изменений (игнорируется, но каскадируется)
- Removed → Managed (отменяет удаление)
- Detached → выбрасывает `IllegalArgumentException` (или `EntityExistsException`)

Ключевое ограничение: нельзя передавать detached-сущности — это частая ошибка у новичков ("detached entity passed to persist").

**`merge(T entity)`**

Переводит сущность из состояния *detached* → *managed*. Возвращает новый управляемый экземпляр (исходный объект остаётся detached!). Состояние переданного объекта копируется в управляемый экземпляр.

Поведение по состояниям:
- Detached → создаётся/обновляется managed-копия, возвращается managed-экземпляр
- Transient → создаётся новая managed-сущность с теми же данными
- Managed → игнорируется (каскадируется)
- Removed → выбрасывает `IllegalArgumentException`

Важно: `merge()` возвращает управляемый объект, а переданный аргумент остаётся detached. Работать нужно с возвращённым значением.

**`remove(Object entity)`**

Переводит сущность из состояния *managed* → *removed*. При flush выполняется `DELETE` из базы данных.

Поведение по состояниям:
- Managed → Removed (штатный сценарий)
- New/Transient → игнорируется (каскадируется)
- Removed → игнорируется
- Detached → выбрасывает `IllegalArgumentException`

Если нужно удалить detached-сущность, сначала нужно её "вернуть" через `merge()`, а затем вызвать `remove()` на возвращённом managed-экземпляре.

**`detach(Object entity)`**

Переводит сущность из состояния *managed* → *detached*. Убирает её из контекста персистентности. Не сброшенные изменения не будут синхронизированы с БД. Появился в JPA 2.0.

| Метод | Вход | Выход | Действие в БД |
|-------|------|-------|---------------|
| `persist()` | Transient | Managed | INSERT при flush |
| `merge()` | Detached/Transient | Managed (возвращает!) | UPDATE или INSERT |
| `remove()` | Managed | Removed | DELETE при flush |
| `detach()` | Managed | Detached | нет |

### Вопрос 9. Что означает атрибут mappedBy в аннотациях @OneToMany и @ManyToMany? Какая сторона является «владеющей» и почему это важно?

**Категория:** JPA/Hibernate · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [OneToMany — Oracle Java EE 7 Specification APIs](https://docs.oracle.com/javaee/7/api/javax/persistence/OneToMany.html)

Атрибут `mappedBy` указывает поле или свойство сущности, которое является владельцем (owning side) отношения. Требуется, если только отношение не является однонаправленным. Значение `mappedBy` — это имя поля на owning side, которое "держит" внешний ключ или ссылку на таблицу связи. Пример для `@OneToMany`:

```java
// Customer (inverse side)
@OneToMany(cascade=ALL, mappedBy="customer")
public Set<Order> getOrders() { return orders; }

// Order (owning side — имеет внешний ключ)
@ManyToOne
@JoinColumn(name="CUST_ID", nullable=false)
public Customer getCustomer() { return customer; }
```

Сторона `@ManyToOne` (Order) является владеющей и содержит внешний ключ `CUST_ID` в таблице. Сторона `@OneToMany` (Customer) является обратной (inverse) и использует `mappedBy`, чтобы указать провайдеру, что физически отношение управляется полем `customer` в сущности `Order`.

#### Ответ от Claude

**Что такое mappedBy**

`mappedBy` — это атрибут, используемый на *обратной (inverse) стороне* двунаправленного отношения. Его значение — имя поля (или property) на *владеющей стороне*, которое управляет этим отношением на уровне базы данных.

Проще говоря: `mappedBy="customer"` означает "я не управляю этим отношением, им управляет поле `customer` в соответствующей сущности".

**Владеющая сторона (Owning Side)**

Владеющая сторона — это сторона, которая физически отображается на базу данных:
- В отношении `@OneToMany` / `@ManyToOne` — владеющей всегда является сторона `@ManyToOne` ("многие"), потому что именно там хранится внешний ключ в таблице.
- В отношении `@ManyToMany` — владеющей может быть любая сторона; именно она определяет join-таблицу через `@JoinTable`.
- В отношении `@OneToOne` — владеющей является сторона, содержащая внешний ключ (`@JoinColumn`).

**Обратная сторона (Inverse Side)**

Обратная сторона использует `mappedBy` и не создаёт никаких DDL-объектов (ни внешних ключей, ни join-таблиц). Она предоставляет лишь навигационный путь для работы с объектами в Java.

**Почему это важно — практические последствия**

1. **Генерация схемы БД.** Hibernate создаёт внешние ключи и join-таблицы только по описанию на владеющей стороне. Если вы поставите `@JoinColumn` на обратную сторону (без `mappedBy`), Hibernate может создать лишние столбцы или таблицы.

2. **Синхронизация изменений.** Hibernate отслеживает изменения отношения только через владеющую сторону. Если вы добавите элемент только в коллекцию на обратной стороне (annotated `mappedBy`), изменение не будет сохранено в БД.

```java
// НЕПРАВИЛЬНО: изменение только обратной стороны — не сохранится
customer.getOrders().add(order);

// ПРАВИЛЬНО: нужно установить владеющую сторону
order.setCustomer(customer);
// И опционально (для согласованности объектного графа в памяти):
customer.getOrders().add(order);
```

3. **Дублирование записей.** Если в `@ManyToMany` обе стороны не имеют `mappedBy`, Hibernate создаст две join-таблицы или продублирует записи.

**Итоговые правила:**
- Сторона с `mappedBy` — обратная (inverse), не управляет БД.
- Сторона без `mappedBy` — владеющая, управляет DDL и DML.
- В `@OneToMany` / `@ManyToOne` владеет всегда сторона `@ManyToOne`.
- В `@ManyToMany` выбор владеющей стороны произвольный, но должен быть явным.
- Изменения в отношении всегда устанавливаются через владеющую сторону.
### Вопрос 10. Что такое LazyInitializationException? При каких условиях она возникает и как её избежать?

**Категория:** JPA/Hibernate · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [LazyInitializationException - What it is and the best way to fix it](https://thorben-janssen.com/lazyinitializationexception/)

Hibernate бросает `LazyInitializationException`, когда ему нужно инициализировать лениво загружаемую ассоциацию с другой сущностью, но активного контекста сессии нет. Типичный сценарий: загружается сущность с ленивыми ассоциациями, `EntityManager`/`Session` закрывается, а затем в слое приложения происходит попытка обратиться к неинициализированной ассоциации (например, `author.getBooks()`).

Антипаттерны, которых следует избегать:
- **FetchType.EAGER** — ухудшает производительность и порождает проблему N+1, загружая все ассоциации независимо от необходимости.
- **Open Session in View** — держит сессию открытой во время рендеринга вьюхи, создавая несколько независимых транзакций с риском получить несогласованные данные.
- **hibernate.enable_lazy_load_no_trans** — открывает временные сессии для ленивой загрузки, бесполезно умножая соединения с БД.

Правильные решения:
1. `LEFT JOIN FETCH` в JPQL:
   ```java
   TypedQuery<Author> q = em.createQuery(
       "SELECT a FROM Author a LEFT JOIN FETCH a.books", Author.class);
   ```
2. `@NamedEntityGraph` / `EntityGraph` API — декларативное или программное определение графа загрузки.
3. **DTO-проекции** — выборка только нужных колонок через constructor expression.

Все корректные подходы инициализируют ассоциации внутри сервисного слоя до возврата данных клиенту.

#### Ответ от Claude

**Что такое LazyInitializationException**

`org.hibernate.LazyInitializationException` — исключение Hibernate, наследующее `HibernateException`. Официальное описание (Hibernate Javadoc 6.6): *«Indicates an attempt to access unfetched data outside the context of an open stateful Session. For example, this exception occurs when an uninitialized proxy or collection is accessed after the session was closed.»*

Проще говоря: Hibernate при ленивой загрузке создаёт прокси-объект вместо реальных данных. Чтобы «разбудить» прокси и выполнить SELECT, нужна открытая `Session`. Если сессия уже закрыта — Hibernate не может выполнить запрос и бросает `LazyInitializationException`.

**Условия возникновения**

1. Ассоциация помечена `FetchType.LAZY` (по умолчанию для `@OneToMany` и `@ManyToMany`; также часто выставляется явно для `@ManyToOne`/`@OneToOne`).
2. Сущность была загружена внутри транзакции/сессии.
3. Транзакция завершена, `EntityManager` / `Session` закрыт.
4. После закрытия сессии происходит обращение к неинициализированной ленивой ассоциации — например, вызов геттера коллекции или навигация по полю-прокси в слое представления или тесте.

**Как избежать: правильные подходы**

| Подход | Когда использовать |
|---|---|
| `JOIN FETCH` в JPQL/HQL | Конкретный use case требует ассоциацию — самый эффективный вариант |
| `@EntityGraph` / `@NamedEntityGraph` | Нужно переиспользуемое определение графа загрузки независимо от запроса |
| DTO-проекция (constructor expression) | UI нужны только поля, а не управляемые сущности — максимальная производительность |
| `@BatchSize(size = N)` | Много записей с одинаковыми ассоциациями — снижает число запросов с N до N/batchSize |

Пример `JOIN FETCH`:
```java
// JPQL
"SELECT a FROM Author a LEFT JOIN FETCH a.books WHERE a.id = :id"

// Spring Data JPA
@Query("SELECT a FROM Author a LEFT JOIN FETCH a.books")
List<Author> findAllWithBooks();
```

**Антипаттерны (не использовать)**

- **`FetchType.EAGER` на ассоциации** — не решает проблему системно: при JPQL-запросах Hibernate всё равно генерирует N+1 SELECT (Hibernate игнорирует маппинг-уровень EAGER для JPQL и выполняет отдельные запросы). Кроме того, загружаются данные даже там, где они не нужны.
- **Open Session in View** — держит `Session` открытой на всё время HTTP-запроса включая рендеринг. Это антипаттерн: транзакция завершена, но Hibernate выполняет дополнительные SELECT в режиме auto-commit при каждом обращении к ленивой ассоциации из шаблона, увеличивая нагрузку на БД и риск получения несогласованных данных.
- **`hibernate.enable_lazy_load_no_trans = true`** — при каждом обращении к ленивому прокси открывается новая временная сессия и транзакция. Лечит симптом, а не причину; существенно увеличивает количество соединений с БД.

**Рекомендация Hibernate:** все ассоциации статически помечать `LAZY` и использовать динамические стратегии загрузки (`JOIN FETCH`, `EntityGraph`) там, где данные действительно нужны.

### Вопрос 11. Как устроен кэш второго уровня в Hibernate? Какие провайдеры кэша существуют и какие стратегии конкурентного доступа (READ_ONLY, READ_WRITE и др.) они поддерживают?

**Категория:** JPA/Hibernate · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [19.2. The Second Level Cache — Hibernate Core Reference](https://docs.hibernate.org/stable/core.old/reference/en/html/performance-cache.html)

Hibernate предоставляет четыре стратегии конкурентного доступа для кэша второго уровня:

- **read-only** — самая простая и производительная стратегия. Используется только для данных, которые никогда не изменяются. Безопасна при кластерном развёртывании.
- **read-write** — подходит для случаев, когда данные обновляются и сериализуемая изоляция транзакций не требуется. В JTA-среде требует настройки `hibernate.transaction.manager_lookup_class`. Не рекомендуется для кластера, если реализация кэша не поддерживает блокировки.
- **nonstrict-read-write** — допускает редкие устаревшие чтения при одновременных обращениях. Подходит, если данные обновляются редко и одновременные изменения одного объекта крайне маловероятны.
- **transactional** — полностью транзакционный кэш (ACID). Требует JTA и совместимого провайдера (исторически — JBoss TreeCache/Infinispan).

Совместимость провайдеров со стратегиями (по документации):

| Провайдер | read-only | nonstrict-read-write | read-write | transactional |
|---|---|---|---|---|
| EHCache | + | + | + | |
| OSCache | + | + | + | |
| SwarmCache | + | + | | |
| JBoss Cache 1.x | + | | | + |
| JBoss Cache 2 | + | | | + |

#### Ответ от Claude

**Архитектура кэширования в Hibernate**

Hibernate поддерживает два уровня кэша:

- **Кэш первого уровня (L1)** — привязан к `Session` / `EntityManager`, работает всегда, хранит управляемые сущности в рамках одной транзакции. Не требует настройки.
- **Кэш второго уровня (L2)** — привязан к `SessionFactory`, разделяется между всеми сессиями, опционален, требует явной настройки.

Когда Hibernate ищет сущность по ID, он сначала проверяет L1, затем L2, и только потом идёт в БД. Результат из БД сохраняется в оба уровня.

**Как устроен L2-кэш**

L2-кэш хранит данные в виде дезагрегированного состояния (по сути — Map полей), а не управляемых объектов. При извлечении из кэша Hibernate собирает объект заново. Кэш работает на уровне регионов (regions) — логических именованных разделов, соответствующих классу сущности или коллекции.

Для включения L2-кэша:
```xml
<!-- persistence.xml -->
<property name="hibernate.cache.use_second_level_cache" value="true"/>
<property name="hibernate.cache.region.factory_class"
          value="org.hibernate.cache.jcache.internal.JCacheRegionFactory"/>
```

Сущность должна быть явно помечена как кэшируемая:
```java
@Entity
@Cacheable
@Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
public class Product { ... }
```

**Провайдеры кэша (актуальные — Hibernate 5/6)**

Hibernate интегрируется с провайдерами через интерфейс `RegionFactory`:

| Провайдер | Описание |
|---|---|
| **EHCache** (через JCache) | Самый распространённый, поддерживает все стратегии кроме TRANSACTIONAL (при использовании через JCache-адаптер) |
| **Infinispan** | Поддерживает все четыре стратегии, включая TRANSACTIONAL; подходит для кластерного развёртывания; является предпочтительным провайдером для WildFly/Quarkus |
| **JCache (JSR-107)** | Стандартный Java Caching API — абстракция поверх любой совместимой реализации (EHCache 3, Caffeine, Hazelcast и др.) |
| **Caffeine** (через JCache) | Высокопроизводительный in-process кэш, только для одиночных узлов |
| **Hazelcast** (через JCache) | Распределённый кэш, подходит для кластеров |

**Стратегии конкурентного доступа (`CacheConcurrencyStrategy`)**

**READ_ONLY**
- Для сущностей, которые никогда не изменяются после создания (справочники, перечисления).
- Попытка обновить такую сущность приводит к исключению.
- Самая производительная стратегия; безопасна в кластере.
- Поддерживается всеми провайдерами.

**READ_WRITE**
- Для сущностей, которые обновляются, при условии что сериализуемая изоляция транзакций не нужна.
- Использует механизм «мягких блокировок» (soft locks): при обновлении запись временно помечается как заблокированная, другие читатели в это время идут в БД.
- Обеспечивает согласованность для одной записи, но не гарантирует сериализуемость транзакции целиком.
- Не рекомендуется для кластерного кэша, если реализация не поддерживает блокировки.

**NONSTRICT_READ_WRITE**
- Допускает кратковременное чтение устаревших данных при конкурентных обновлениях.
- Кэш обновляется после коммита транзакции (не синхронно с ней).
- Подходит, если одновременные изменения одного объекта маловероятны и небольшая задержка обновления кэша приемлема.
- Проще в настройке, чем READ_WRITE, и работает в кластере лучше.

**TRANSACTIONAL**
- Полная транзакционная семантика (XA): изменения кэша и БД откатываются вместе.
- Требует JTA-среды и провайдера с поддержкой транзакций (Infinispan в транзакционном режиме).
- Максимальная согласованность, но наибольшие накладные расходы.

**Кэш запросов (Query Cache)**
Дополнительно к кэшу сущностей Hibernate поддерживает кэш запросов (`hibernate.cache.use_query_cache=true`). Хранит ключи результатов (не сами данные), работает только совместно с L2-кэшем сущностей. Эффективен для стабильных параметрических запросов к редко изменяемым таблицам.

### Вопрос 12. Как решить проблему N+1 с помощью JOIN FETCH в JPQL? Почему простого FetchType.EAGER недостаточно и как он сам может провоцировать N+1?

**Категория:** JPA/Hibernate · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [N+1 query problem with JPA and Hibernate — Vlad Mihalcea](https://vladmihalcea.com/n-plus-1-query-problem/)

Проблема N+1 запросов возникает, когда фреймворк доступа к данным выполняет N дополнительных SQL-запросов для получения данных, которые могли бы быть выбраны в основном запросе. Чем больше N, тем больше запросов и тем сильнее удар по производительности.

При использовании ассоциаций `@ManyToOne` или `@OneToOne` с жадной загрузкой (Eager — поведение по умолчанию) Hibernate автоматически подгружает связанные сущности. Однако при JPQL-запросах Hibernate не может автоматически внедрить JOIN — он сначала выбирает родительские записи, а затем для каждой выполняет отдельный запрос за связанными данными, провоцируя N+1.

Рекомендуемое исправление — добавить `JOIN FETCH` в JPQL:
```java
select pc from PostComment pc join fetch pc.post p
```
Это объединяет все данные в единый SQL-запрос с JOIN.

Дополнительные решения:
- Переключиться на `FetchType.LAZY` там, где ассоциации не всегда нужны.
- Использовать кэш второго уровня, который хранит как результаты запросов, так и сами сущности.

Основной принцип: получать все нужные данные за один запрос, а не загружать связанные записи по частям.

#### Ответ от Claude

**Что такое проблема N+1**

Проблема N+1 — ситуация, когда для загрузки N сущностей выполняется 1 запрос за «родительскими» объектами плюс N отдельных запросов за их ассоциациями. Пример: список из 100 заказов и загрузка клиента для каждого — итого 101 запрос вместо одного.

```
-- 1 запрос за заказами
SELECT * FROM orders;

-- N запросов за каждым клиентом
SELECT * FROM customers WHERE id = 1;
SELECT * FROM customers WHERE id = 2;
-- ... ещё 98 запросов
```

**Почему FetchType.EAGER не решает, а провоцирует N+1**

`FetchType.EAGER` определяет поведение загрузки на уровне маппинга сущности. Однако при выполнении JPQL/HQL-запроса Hibernate рассматривает запрос как явный план загрузки и не модифицирует его автоматически.

Механизм появления N+1 при EAGER:
1. JPQL `SELECT o FROM Order o` превращается в `SELECT * FROM orders` — Hibernate не добавляет JOIN автоматически.
2. После получения результата Hibernate видит, что ассоциация помечена EAGER.
3. Для каждой записи Hibernate немедленно (eager!) выполняет отдельный `SELECT` за ассоциированной сущностью.

Итог: FetchType.EAGER с JPQL гарантирует N+1. С `Session.get()` / `find()` по ID Hibernate может оптимизировать запрос через JOIN, но с `@Query`/`createQuery` — нет.

Кроме того, EAGER загружает ассоциацию всегда, даже когда она не нужна, что расточительно.

**Решение: JOIN FETCH в JPQL**

`JOIN FETCH` — директива JPQL/HQL, которая инструктирует Hibernate загрузить ассоциацию в том же SQL-запросе через JOIN:

```java
// Загружаем все посты вместе с комментариями — один SQL-запрос
String jpql = "SELECT p FROM Post p LEFT JOIN FETCH p.comments";
List<Post> posts = em.createQuery(jpql, Post.class).getResultList();

// Аналог в Spring Data JPA
@Query("SELECT p FROM Post p LEFT JOIN FETCH p.comments WHERE p.author = :author")
List<Post> findByAuthorWithComments(@Param("author") String author);
```

Генерируемый SQL:
```sql
SELECT p.*, c.*
FROM posts p
LEFT JOIN comments c ON c.post_id = p.id
WHERE p.author = ?
```

Один запрос вместо N+1.

**Ограничения JOIN FETCH**

- **Дублирование строк при `@OneToMany`**: SQL JOIN раскрывает декартово произведение — если у поста 3 комментария, он появится в результате 3 раза. Решение: `SELECT DISTINCT p FROM Post p LEFT JOIN FETCH p.comments` или обработка дедупликации на стороне Java.
- **Только одна коллекция**: JOIN FETCH одновременно можно применить только к одной коллекции `@OneToMany` / `@ManyToMany`. Для двух коллекций Hibernate бросает `MultipleBagFetchException`. Решение: использовать `Set` вместо `List`, либо делать несколько запросов.
- **Пагинация несовместима**: при использовании `JOIN FETCH` с `@OneToMany` и одновременной пагинацией (`setMaxResults`) Hibernate вынужден загружать весь результат в память и выполнять пагинацию там. Это приводит к предупреждению `HHH90003004` и потенциальному OutOfMemory.

**Альтернативные решения**

**`@BatchSize`** — снижает количество запросов с N до N/batch_size:
```java
@OneToMany
@BatchSize(size = 25)
private List<Comment> comments;
```
Hibernate загружает ассоциации пакетами через `WHERE id IN (...)`.

**`@EntityGraph`** — декларативное указание графа загрузки, не привязанное к тексту запроса:
```java
@EntityGraph(attributePaths = {"comments", "tags"})
List<Post> findAll();
```

**`@Fetch(FetchMode.SUBSELECT)`** — загружает все коллекции через один подзапрос:
```java
@OneToMany
@Fetch(FetchMode.SUBSELECT)
private List<Comment> comments;
// SQL: SELECT * FROM comments WHERE post_id IN (SELECT id FROM posts WHERE ...)
```

**Сравнение подходов**

| Подход | Запросов | Пагинация | Применимость |
|---|---|---|---|
| Без оптимизации (LAZY) | N+1 | + | — |
| `FetchType.EAGER` + JPQL | N+1 | + | Антипаттерн |
| `JOIN FETCH` | 1 | Ограничено | Одна коллекция |
| `@BatchSize` | N/batch+1 | + | Универсально |
| `@EntityGraph` | 1–2 | Ограничено | Гибко |
| DTO-проекция | 1 | + | Только чтение |

**Рекомендация Hibernate:** по умолчанию все ассоциации — `LAZY`; использовать `JOIN FETCH` или `@EntityGraph` там, где данные действительно нужны в конкретном use case.
### Вопрос 13. Чем отличается JPQL от Criteria API? В каких сценариях предпочтительнее использовать Criteria API?

**Категория:** JPA/Hibernate · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [JPQL Versus The Criteria API — Bekwam Courses](https://courses.bekwam.net/public_tutorials/bkcourse_wildfly_jpql_vs_criteriaapi.html)

JPQL — строковый язык запросов, похожий на SQL. Его недостаток в том, что при сложных условиях JPQL-строка фрагментируется: разработчику нужно мысленно склеивать части запроса, а параметры задаются отдельно через `setParameter()`. Ошибки (например, опечатка в имени поля) обнаруживаются только в рантайме.

Criteria API, напротив, работает исключительно с Java-кодом и использует Metamodel — автогенерируемые классы, отражающие структуру сущностей. Если указать несуществующее поле сущности, компилятор выдаст предупреждение. Criteria API лучше подходит для динамического построения запросов: условия формируются в виде отдельных объектов `Predicate`, которые затем комбинируются по необходимости. Практическая рекомендация автора: «когда параметров много или логика сложная — я выбираю Criteria API».

#### Ответ от Claude

**JPQL (Java Persistence Query Language)** — строковый язык запросов, синтаксически похожий на SQL, но оперирующий именами сущностей и их полей, а не именами таблиц и колонок. Запросы записываются как обычные Java-строки.

**Criteria API** — программный, типобезопасный способ построения запросов через цепочку вызовов объектов `CriteriaBuilder`, `CriteriaQuery`, `Root`, `Predicate` и т.д.

**Ключевые различия:**

| Аспект | JPQL | Criteria API |
|---|---|---|
| Форма записи | Строка | Java-объекты |
| Типобезопасность | Нет (ошибки — в рантайме) | Да (ошибки — на этапе компиляции) |
| Читаемость | Высокая (при простых запросах) | Ниже (более многословный код) |
| Динамические запросы | Неудобно (конкатенация строк) | Удобно (условное добавление `Predicate`) |
| Metamodel | Не поддерживается | Поддерживается (`Post_.title`) |
| Сложные JOIN | Легко читаются | Громоздко |

**Когда предпочтительнее Criteria API:**
1. **Динамические запросы** — когда набор условий фильтрации заранее неизвестен (фильтры по чекбоксам, выпадающим спискам на UI). Строковую конкатенацию в JPQL поддерживать сложно и опасно.
2. **Требование типобезопасности** — при использовании JPA Metamodel ошибки в именах полей ловятся компилятором, а не в продакшене.
3. **Рефакторинг** — при переименовании полей сущности IDE автоматически найдёт все места использования через Metamodel, тогда как строковые JPQL-запросы не будут видны статическому анализу.

**Когда предпочтительнее JPQL:**
1. Простые статические запросы — JPQL читается значительно проще.
2. Запросы с именованием (`@NamedQuery`) — они проверяются при старте контекста.
3. Сложные JOIN с несколькими таблицами — JPQL-синтаксис лаконичнее.

**Важно:** начиная с Hibernate 5.2, собственный (проприетарный) Criteria API Hibernate помечен как устаревший. Следует использовать стандартный JPA Criteria API (`jakarta.persistence.criteria`).

**Пример динамического запроса на Criteria API:**
```java
CriteriaBuilder cb = em.getCriteriaBuilder();
CriteriaQuery<Post> cq = cb.createQuery(Post.class);
Root<Post> root = cq.from(Post.class);

List<Predicate> predicates = new ArrayList<>();
if (title != null) {
    predicates.add(cb.like(root.get(Post_.title), "%" + title + "%"));
}
if (authorId != null) {
    predicates.add(cb.equal(root.get(Post_.author), authorId));
}
cq.where(predicates.toArray(new Predicate[0]));

return em.createQuery(cq).getResultList();
```

Тот же запрос на JPQL потребовал бы ручной конкатенации строк с потенциальными ошибками.

### Вопрос 14. Как правильно реализовать двустороннее отношение @ManyToMany в JPA? Почему рекомендуется использовать промежуточную сущность вместо прямого @ManyToMany?

**Категория:** JPA/Hibernate · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Best way to map the JPA and Hibernate ManyToMany relationship — Vlad Mihalcea](https://vladmihalcea.com/the-best-way-to-use-the-manytomany-annotation-with-jpa-and-hibernate/)

При двустороннем `@ManyToMany` одна сторона является владеющей (owning side) и содержит `@JoinTable`, другая — обратной (inverse side) и использует `mappedBy`. Необходимо использовать `Set` вместо `List`: при удалении элемента из `List`-ассоциации Hibernate удаляет все строки в join-таблице и заново вставляет оставшиеся, тогда как при `Set` удаляется только одна конкретная строка. Следует добавлять вспомогательные методы (`addTag()`, `removeTag()`) для синхронизации обеих сторон ассоциации. Нельзя использовать `CascadeType.ALL` или `CascadeType.REMOVE` на `@ManyToMany` — это может вызвать цепочку удалений и уничтожить обе стороны ассоциации. Для `equals`/`hashCode` на стороне `Tag` рекомендуется использовать бизнес-ключ (`@NaturalId`).

#### Ответ от Claude

**Базовая реализация двустороннего @ManyToMany**

Одна сущность является **владеющей стороной** (содержит `@JoinTable`), вторая — **обратной стороной** (использует `mappedBy`):

```java
@Entity
public class Post {
    @ManyToMany
    @JoinTable(
        name = "post_tag",
        joinColumns = @JoinColumn(name = "post_id"),
        inverseJoinColumns = @JoinColumn(name = "tag_id")
    )
    private Set<Tag> tags = new HashSet<>();

    // Вспомогательные методы синхронизации
    public void addTag(Tag tag) {
        tags.add(tag);
        tag.getPosts().add(this);
    }

    public void removeTag(Tag tag) {
        tags.remove(tag);
        tag.getPosts().remove(this);
    }
}

@Entity
public class Tag {
    @ManyToMany(mappedBy = "tags")
    private Set<Post> posts = new HashSet<>();
}
```

**Обязательные правила для прямого @ManyToMany:**
1. **Set вместо List** — при удалении элемента из `List` Hibernate удаляет все записи join-таблицы для данного родителя и вставляет оставшиеся заново; `Set` удаляет только одну строку.
2. **Вспомогательные методы** — без `addTag()`/`removeTag()` легко получить рассинхронизацию в памяти: одна сторона знает о связи, другая — нет.
3. **Только CascadeType.PERSIST и CascadeType.MERGE** — `CascadeType.REMOVE` на `@ManyToMany` удалит саму связанную сущность (напр., `Tag`), а не только строку в join-таблице, что может затронуть другие ассоциации.
4. **Корректный equals/hashCode** — использовать бизнес-ключ (`@NaturalId`) или неизменяемые поля, иначе `Set`-операции работают некорректно.

**Почему рекомендуется промежуточная сущность вместо прямого @ManyToMany:**

Прямой `@ManyToMany` не позволяет хранить дополнительные данные в join-таблице. Как только появляется нужда в поле (например, `created_at`, `role`, `quantity`) — нужна промежуточная сущность.

```java
@Entity
public class PostTag {
    @EmbeddedId
    private PostTagId id;

    @ManyToOne
    @MapsId("postId")
    private Post post;

    @ManyToOne
    @MapsId("tagId")
    private Tag tag;

    private LocalDateTime createdOn; // дополнительное поле
}

@Embeddable
public class PostTagId implements Serializable {
    private Long postId;
    private Long tagId;
    // equals + hashCode
}
```

**Преимущества промежуточной сущности:**
- Возможность хранить любые дополнительные атрибуты связи.
- Полный контроль над жизненным циклом: можно использовать `orphanRemoval`.
- Чище управляется через `@OneToMany` + `@ManyToOne` — стандартные инструменты.
- Удаление строки join-таблицы — явная операция, а не побочный эффект.

**Рекомендация:** при проектировании новых систем сразу использовать промежуточную сущность — это избавит от рефакторинга при добавлении новых требований к данным связи.

### Вопрос 15. В чём разница между CascadeType.REMOVE и orphanRemoval = true? Когда нужно использовать каждый из вариантов?

**Категория:** JPA/Hibernate · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Understanding CascadeType.REMOVE vs. orphanRemoval in JPA — Learn IT University](https://learn-it-university.com/understanding-the-difference-between-cascadetype-remove-and-orphanremoval-in-jpa/)

`CascadeType.REMOVE` активируется исключительно при удалении родительской сущности: если родитель удалён — дочерние сущности удаляются вместе с ним. `orphanRemoval = true` работает шире: оно срабатывает всякий раз, когда связь разрывается — будь то установка поля в `null` или удаление объекта из коллекции родителя. При `orphanRemoval` ребёнок автоматически удаляется, если он больше не связан ни с каким родителем. `CascadeType.REMOVE` следует использовать, когда дочерние сущности потенциально могут существовать самостоятельно; `orphanRemoval` — для жёстко связанных отношений, когда дочерняя сущность не имеет смысла без родителя (например, изображение профиля пользователя).

#### Ответ от Claude

**CascadeType.REMOVE** и **orphanRemoval = true** оба приводят к удалению дочерних сущностей, но триггер у них разный.

**CascadeType.REMOVE**

Срабатывает только когда вызывается `em.remove(parent)` — то есть при явном удалении родителя. Hibernate каскадирует операцию удаления на все связанные дочерние сущности.

```java
@Entity
public class Order {
    @OneToMany(mappedBy = "order", cascade = CascadeType.REMOVE)
    private List<LineItem> items;
}

// При удалении order — все LineItem тоже удаляются
em.remove(order);
```

Если же просто очистить коллекцию (`order.getItems().clear()`) или убрать связь (`item.setOrder(null)`), то при `CascadeType.REMOVE` ничего не произойдёт — дочерние сущности останутся в базе как «осиротевшие» записи.

**orphanRemoval = true**

Срабатывает при разрыве связи: удаление дочернего объекта из коллекции родителя или установка ссылки в `null` автоматически ставит этот объект в очередь на удаление при следующем flush.

```java
@Entity
public class UserProfile {
    @OneToOne(orphanRemoval = true)
    private ProfileImage image;
}

// Установка в null => ProfileImage будет удалён из БД
profile.setImage(null);
em.flush(); // DELETE FROM profile_image WHERE id = ?
```

```java
@Entity
public class Order {
    @OneToMany(mappedBy = "order", orphanRemoval = true)
    private List<LineItem> items;
}

// Удаление из коллекции => LineItem будет удалён из БД
order.getItems().remove(item); // DELETE при flush
```

**Сравнительная таблица:**

| Аспект | CascadeType.REMOVE | orphanRemoval = true |
|---|---|---|
| Когда срабатывает | Только при `em.remove(parent)` | При `em.remove(parent)` И при разрыве связи |
| Очистка коллекции | Не удаляет | Удаляет |
| Установка поля в null | Не удаляет | Удаляет |
| Уровень работы | Каскадирование JPA-операций | Управление жизненным циклом объектов |
| Применимость | `@OneToOne`, `@OneToMany`, `@ManyToOne` | `@OneToOne`, `@OneToMany` |

**Совместное использование:** `orphanRemoval = true` включает в себя поведение `CascadeType.REMOVE` — при удалении родителя дочерние объекты тоже удаляются. Поэтому явно добавлять оба одновременно на одну связь не нужно.

**Когда использовать каждый вариант:**

- `CascadeType.REMOVE` — когда дочерние сущности могут теоретически существовать без родителя или быть повторно привязаны к другому родителю. Пример: `ShipmentInfo` у заказа — нужно удалять только при удалении самого заказа, но не при замене информации о доставке.
- `orphanRemoval = true` — когда дочерняя сущность не имеет смысла вне контекста родителя и должна быть удалена сразу при отвязке. Пример: `ProfileImage` у `User`, `Address` у `Employee`, строки `LineItem` у `Order`.

**Важный нюанс:** `orphanRemoval` не применяется к `@ManyToMany` и `@ManyToOne` — только к `@OneToOne` и `@OneToMany`, так как в `@ManyToMany` дочерняя сущность потенциально связана с несколькими родителями и «осиротевшей» быть не может.
### Вопрос 16. Что происходит с сущностью в состоянии detached при вызове merge()? Чем это отличается от поведения persist()?

**Категория:** JPA/Hibernate · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [37.3 Managing Entities — Java EE 7 Tutorial (Oracle)](https://docs.oracle.com/javaee/7/tutorial/persistence-intro003.htm)

При вызове `merge()` на отсоединённой (detached) сущности JPA копирует её состояние в существующий управляемый (managed) экземпляр с тем же идентификатором, либо создаёт новый managed-экземпляр, если такового нет в текущем persistence context. Метод возвращает ссылку на этот managed-экземпляр — оригинальный detached-объект при этом остаётся отсоединённым. Операция каскадируется на связанные сущности, у которых указан `CascadeType.MERGE`.

`persist()` при вызове на detached-сущности выбрасывает `IllegalArgumentException` либо вызывает откат транзакции при коммите — это нарушение жизненного цикла JPA. `persist()` предназначен исключительно для новых (transient) сущностей: он переводит их из состояния new/transient в managed.

#### Ответ от Claude

**Жизненный цикл сущности (4 состояния):**
- **Transient (new)** — объект создан через `new`, не связан с persistence context и не имеет записи в БД.
- **Managed (persistent)** — объект ассоциирован с активным persistence context; изменения отслеживаются и синхронизируются с БД при flush.
- **Detached** — объект имеет идентификатор и соответствующую запись в БД, но persistence context, в котором он был managed, уже закрыт (или сущность была явно detach-нута).
- **Removed** — объект помечен для удаления из БД.

**Поведение `merge()` на detached-сущности:**

1. Hibernate проверяет, есть ли в текущем persistence context managed-экземпляр с тем же `@Id`.
2. Если есть — копирует поля detached-объекта в найденный managed-экземпляр.
3. Если нет — загружает запись из БД по `@Id` и создаёт новый managed-экземпляр, копируя в него поля detached-объекта.
4. **Возвращает ссылку на managed-экземпляр** — это принципиально: оригинальный detached-объект не становится managed.
5. Если запись по идентификатору не найдена в БД — поведение зависит от реализации (по JPA-спецификации может быть выброшено исключение или создан новый объект; Hibernate создаёт INSERT).

```java
User detached = ...; // ранее отсоединён
User managed = entityManager.merge(detached); // managed != detached
// detached по-прежнему detached!
// managed.getId() == detached.getId()
```

**Поведение `persist()` на detached-сущности:**

`persist()` предназначен только для transient-объектов. При вызове на detached-сущности:
- JPA-спецификация предписывает выброс `IllegalArgumentException` (или `EntityExistsException`) либо неуспешный коммит транзакции.
- Hibernate реализует это как выброс `PersistenceException: detached entity passed to persist`.

Причина: `persist()` не умеет различать «уже существующую в БД» и «новую» сущность — если объект имеет ненулевой `@Id`, Hibernate трактует это как конфликт состояний.

**Сводная таблица:**

| Метод | Transient | Managed | Detached | Removed |
|---|---|---|---|---|
| `persist()` | managed (INSERT) | без изменений (игнор) | **Exception** | managed (отменяет удаление) |
| `merge()` | managed (INSERT) | managed (копирует поля) | managed-копия (UPDATE/INSERT) | Exception |

**Дополнительные нюансы:**
- После `merge()` следует работать только с возвращённым объектом; изменения в оригинальном detached-объекте не будут отслежены.
- `merge()` умнее: он сам определяет, выполнять INSERT или UPDATE, анализируя наличие записи в БД.
- В Hibernate Session API есть метод `saveOrUpdate()`, который ведёт себя похоже на `merge()`, но реассоциирует оригинальный объект — это нативный Hibernate API, не часть JPA-спецификации.

### Вопрос 17. Почему стратегия GenerationType.IDENTITY несовместима с батчингом (batch insert) в Hibernate? Какую стратегию использовать вместо неё для повышения производительности вставки?

**Категория:** JPA/Hibernate · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Hibernate ORM 5.2 User Guide — Batching](https://docs.hibernate.org/orm/5.2/userguide/html_single/chapters/batch/Batching.html)

В разделе «Session batching» документация прямо указывает:

> «Hibernate disables insert batching at the JDBC level transparently if you use an identity identifier generator.»

То есть Hibernate молча отключает JDBC-батчинг для сущностей с IDENTITY-генератором. Причина — техническое ограничение: при использовании IDENTITY БД генерирует первичный ключ только после выполнения INSERT, поэтому Hibernate не может накапливать несколько INSERT-операторов перед отправкой в БД.

#### Ответ от Claude

**Техническая причина несовместимости:**

Hibernate использует механизм write-behind (отложенная запись): изменения накапливаются в persistence context и отправляются в БД при flush батчем через `PreparedStatement.executeBatch()`. Для этого Hibernate должен знать идентификатор каждой сущности до момента отправки запроса — он нужен для управления identity map (first-level cache).

С `GenerationType.IDENTITY`:
- БД генерирует `id` через `AUTO_INCREMENT` / `SERIAL` / `GENERATED ALWAYS AS IDENTITY` — но значение становится известно **только после** выполнения INSERT.
- Чтобы узнать `id`, Hibernate вынужден выполнять каждый INSERT немедленно и по одному.
- Накопить пакет INSERT и отправить их разом — невозможно.
- Hibernate **молча** отключает батчинг (нет ни ошибки, ни предупреждения в дефолтной конфигурации).

**С `GenerationType.SEQUENCE`:**
- Hibernate заранее запрашивает диапазон идентификаторов у sequence-объекта БД.
- Раздаёт `id` сущностям до INSERT.
- Накапливает INSERT-запросы и отправляет их единым батчем через JDBC.
- Число обращений к sequence сокращается благодаря оптимизатору `pooled` (по умолчанию при `allocationSize > 1`).

**Настройка для батч-вставок с SEQUENCE:**

```java
@Entity
public class Order {

    @Id
    @GeneratedValue(
        strategy = GenerationType.SEQUENCE,
        generator = "order_seq"
    )
    @SequenceGenerator(
        name = "order_seq",
        sequenceName = "order_id_seq",
        allocationSize = 50   // pooled-оптимизатор: один вызов sequence на 50 вставок
    )
    private Long id;
}
```

```properties
# application.properties
spring.jpa.properties.hibernate.jdbc.batch_size=50
spring.jpa.properties.hibernate.order_inserts=true
spring.jpa.properties.hibernate.order_updates=true
```

**Оптимизаторы sequence в Hibernate:**
- `none` — каждый `id` запрашивается у БД отдельно (эквивалентно IDENTITY по количеству round-trip'ов).
- `pooled` (дефолт при `allocationSize > 1`) — значение sequence используется как **верхняя граница** диапазона; совместим с другими клиентами, пишущими в ту же таблицу.
- `pooled-lo` — значение sequence используется как **нижняя граница** диапазона; требует специфической синхронизации при совместном доступе.

**Что делать для MySQL/MariaDB (нет нативных SEQUENCE):**

MySQL до версии 8.0 не поддерживает объекты SEQUENCE. Варианты:
1. MariaDB 10.3+ поддерживает SEQUENCE — можно использовать `GenerationType.SEQUENCE`.
2. `GenerationType.TABLE` — эмулирует sequence через отдельную таблицу; поддерживает батчинг, но медленнее из-за блокировок (`SELECT FOR UPDATE`).
3. Hibernate `pooled` batch sequence generator (библиотека `hibernate-types` от Vlad Mihalcea) — оптимизирует batch с IDENTITY для MySQL через нестандартный генератор.
4. Использовать jOOQ или `JdbcTemplate` для массовых вставок в обход Hibernate.

**Итого:** для максимальной производительности batch insert следует использовать `GenerationType.SEQUENCE` с `allocationSize`, равным размеру батча, и включённым `hibernate.jdbc.batch_size`.

### Вопрос 18. Что такое Open Session in View (OSIV) паттерн в контексте Spring + Hibernate? Какие проблемы он решает и какие создаёт?

**Категория:** JPA/Hibernate · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [OpenSessionInViewInterceptor — Spring Framework 4.3 API (docs.spring.io)](https://docs.spring.io/spring-framework/docs/4.3.29.RELEASE_to_4.3.30.RELEASE/Spring%20Framework%204.3.30.RELEASE/org/springframework/orm/hibernate4/support/OpenSessionInViewInterceptor.html)

`OpenSessionInViewInterceptor` — Spring-перехватчик HTTP-запросов, реализующий паттерн «Open Session in View». Он **открывает Hibernate Session в начале запроса и привязывает её к потоку исполнения на всё время обработки** запроса, включая рендеринг view.

Цель — обеспечить возможность lazy-загрузки ассоциаций в слое представления (view) после завершения транзакции. Без него при обращении к незагруженной коллекции после закрытия сессии возникает `LazyInitializationException`.

Документация предупреждает: применение перехватчика к уже существующей логике может вызвать проблемы — в частности, при повторной ассоциации (reassociation) persistent-объектов с Hibernate Session нужно делать это в самом начале обработки запроса, чтобы избежать конфликтов с уже загруженными экземплярами тех же объектов.

#### Ответ от Claude

**Что такое OSIV и зачем он появился:**

В типичном Spring MVC / Spring Boot приложении без OSIV жизненный цикл Session/EntityManager ограничен транзакцией: сессия открывается при старте `@Transactional`-метода и закрывается при его завершении. Когда управление передаётся в контроллер или шаблон (Thymeleaf, JSP и т.д.) для рендеринга, сессия уже закрыта. Любое обращение к незагруженной lazy-ассоциации (`@OneToMany`, `@ManyToMany` с `FetchType.LAZY`) вызывает `LazyInitializationException`.

OSIV решает эту проблему: Session открывается при получении HTTP-запроса (в фильтре или перехватчике) и закрывается только после отправки ответа клиенту.

**Реализация в Spring Boot:**

Spring Boot автоматически регистрирует `OpenEntityManagerInViewInterceptor` (JPA-аналог `OpenSessionInViewInterceptor`). По умолчанию `spring.jpa.open-in-view=true`. При запуске приложения Spring Boot выводит предупреждение в лог:

```
WARN o.s.b.autoconfigure.orm.jpa.JpaBaseConfiguration$JpaWebConfiguration:
spring.jpa.open-in-view is enabled by default.
Therefore, database queries may be performed during view rendering.
Explicitly configure spring.jpa.open-in-view to disable this warning
```

**Что OSIV решает:**
- Устраняет `LazyInitializationException` при доступе к lazy-ассоциациям в view-слое или после завершения транзакции.
- Упрощает разработку в простых приложениях — не нужно думать о явной загрузке всех ассоциаций в сервисном слое.

**Проблемы, которые OSIV создаёт:**

1. **Удержание соединения из пула на всё время запроса.** Session держит JDBC-соединение открытым от начала до конца HTTP-запроса, включая время рендеринга и любые задержки в view. При высокой нагрузке это исчерпывает пул соединений и снижает throughput.

2. **Скрытые N+1 запросы.** Lazy-загрузка в view не видна в сервисном слое и не покрывается тестами сервисов. Разработчик не замечает проблемы до нагрузочного тестирования.

3. **Запросы вне транзакции.** После завершения `@Transactional`-метода lazy-загрузка в view выполняется в режиме auto-commit. Для PostgreSQL это означает, что каждый lazy-запрос — отдельная транзакция с flush transaction log на диск, что создаёт I/O-нагрузку.

4. **Нарушение разделения ответственности.** Слой базы данных «протекает» в view-слой: изменения состояния managed-сущностей в view могут непредсказуемо попасть в БД при flush.

5. **Сложность тестирования.** Интеграционные тесты сервисного слоя не воспроизводят поведение в production, так как OSIV активен только при наличии web-контекста.

**Рекомендации:**

- В production-приложениях с высокой нагрузкой рекомендуется отключить OSIV: `spring.jpa.open-in-view=false`.
- Вместо этого явно загружать нужные ассоциации в сервисном слое через `JOIN FETCH` или `EntityGraph`.
- Использовать DTO-проекции для read-only операций — они не требуют открытой сессии и не подвержены `LazyInitializationException`.
- Если OSIV уже включён в legacy-проекте, отключать его надо осторожно: сначала нужно найти все `LazyInitializationException`, которые он скрывал.

```properties
# Отключить OSIV (рекомендуется для высоконагруженных приложений)
spring.jpa.open-in-view=false
```

### Вопрос 19. Чем отличается merge() от persist() в EntityManager? Что вернёт merge() и изменится ли состояние переданного объекта?

**Категория:** JPA/Hibernate · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [8.2. Entity Lifecycle Management — Oracle Fusion Middleware Persistence API Reference](https://docs.oracle.com/cd/E15523_01/apirefs.1111/e13946/ejb3_overview_em_lifecycle.html)

**persist()** переводит новый (transient) объект в управляемое (managed) состояние. Если передать управляемую сущность — вызов игнорируется (каскады всё равно применяются). Если передать detached-объект — будет выброшен `IllegalArgumentException`. Для removed-сущности вызов persist() возвращает её обратно в managed-состояние.

**merge()** предназначен для работы с detached-объектами. Метод возвращает **управляемую копию** переданной сущности — новый (или уже существующий в контексте) managed-объект с состоянием, скопированным из переданного экземпляра. Исходный detached-объект при этом **не изменяется и не становится managed**. Если передать transient-объект — будет создана новая управляемая сущность с тем же состоянием. Если передать removed-объект — будет выброшен `IllegalArgumentException`. Каскады применяются через `CascadeType.MERGE`.

Важный паттерн использования: после вызова `merge()` нужно работать именно с **возвращённым экземпляром**, а не с тем, который был передан в метод.

#### Ответ от Claude

**persist() — для новых объектов:**
- Принимает transient (новую) сущность и регистрирует её в Persistence Context; при следующем flush/commit будет выполнен INSERT.
- Если сущность уже managed — вызов игнорируется (каскады всё равно срабатывают).
- Если сущность removed — переводит обратно в managed.
- Если передан **detached**-объект — выбрасывает `IllegalArgumentException` (или транзакция откатывается при commit, в зависимости от реализации). Это ключевое ограничение.

**merge() — для detached-объектов:**
- Ищет в Persistence Context сущность с тем же идентификатором.
  - Если находит — копирует состояние переданного объекта на уже существующий managed-экземпляр и возвращает его.
  - Если не находит — загружает из БД (или создаёт новую managed-сущность, если ID не задан/запись отсутствует) и копирует состояние.
- **Возвращает managed-экземпляр** — всегда отличный объект от того, что был передан (другая ссылка в памяти).
- **Переданный объект остаётся в detached-состоянии** — его состояние не меняется, он не становится managed.
- Умеет автоматически определять, нужен INSERT или UPDATE.

**Итоговая сравнительная таблица:**

| Аспект | persist() | merge() |
|---|---|---|
| Назначение | Добавление нового объекта | Повторное присоединение detached-объекта |
| Detached-объект | `IllegalArgumentException` | Копирует состояние, возвращает managed-копию |
| Возвращаемое значение | void (тот же объект становится managed) | Новый managed-объект (другая ссылка) |
| Изменяется ли переданный объект | Да — сам объект становится managed | Нет — переданный объект остаётся detached |
| Cascade | `CascadeType.PERSIST` | `CascadeType.MERGE` |

**Практическое правило:** после `merge()` всегда используй возвращённый экземпляр, а не исходный. Это особенно важно при работе с auto-generated идентификаторами и version-полями.

### Вопрос 20. Опишите все состояния жизненного цикла сущности в JPA (transient, managed, detached, removed). Что происходит при вызове persist() на detached-объекте?

**Категория:** JPA/Hibernate · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Chapter 3. Persistence Contexts — Hibernate Developer Guide](https://docs.hibernate.org/devguide/en-US/html/ch03.html)

Hibernate определяет четыре состояния жизненного цикла сущности:

**Transient (новый):** объект только что создан через `new`, не связан ни с каким Persistence Context, не имеет записи в БД и идентификатора (или идентификатор задан вручную, но нет соответствующей строки в БД).

**Managed/Persistent (управляемый):** объект имеет идентификатор и привязан к активному Persistence Context. Все изменения автоматически отслеживаются и будут синхронизированы с БД при flush.

**Detached (отсоединённый):** объект имеет идентификатор и соответствующую строку в БД, но больше не связан с Persistence Context (например, после закрытия Session или вызова `evict()`/`clear()`). Изменения не отслеживаются.

**Removed (удалённый):** объект имеет идентификатор, ещё связан с Persistence Context, но помечен к удалению — при следующем flush будет выполнен DELETE.

Для работы с detached-объектами документация рекомендует использовать операцию **merge**: «принять входящий detached-экземпляр и скопировать его данные на новый managed-экземпляр».

#### Ответ от Claude

**Четыре состояния жизненного цикла JPA-сущности:**

**1. Transient (новый/переходный)**
- Создан через `new`, нет связи с Persistence Context.
- Нет записи в БД (или есть ID, но нет соответствующей строки).
- JPA/Hibernate полностью игнорирует такой объект: изменения не отслеживаются, при закрытии контекста — просто удаляется GC.
- Переход в Managed: `entityManager.persist(entity)`.

**2. Managed (managed/persistent — управляемый)**
- Связан с активным Persistence Context; имеет идентификатор.
- Все изменения полей автоматически фиксируются через dirty checking и синхронизируются с БД при flush (до commit или явного `flush()`).
- Переходы: в Detached — при закрытии контекста или вызове `evict()`/`detach()`/`clear()`; в Removed — при вызове `remove()`.

**3. Detached (отсоединённый)**
- Имеет идентификатор и соответствующую строку в БД, но Persistence Context уже не следит за объектом.
- Причины перехода: закрытие `EntityManager`/`Session`, конец транзакции (в случае TRANSACTION-scoped контекста), явный вызов `detach(entity)`, `evict()`, `clear()`, а также сериализация/десериализация.
- Изменения не отслеживаются автоматически.
- Переход обратно в Managed: `merge()`.

**4. Removed (удалённый)**
- Managed-объект, помеченный к удалению через `entityManager.remove(entity)`.
- При flush выполняется DELETE в БД.
- Если вызвать `persist()` на removed-сущности — она возвращается в Managed.

**Схема переходов:**
```
Transient  --persist()-->  Managed  --remove()-->  Removed
                              ^                        |
                              |--persist()-------------|
                              |
                           detach()/close()
                              |
                              v
                           Detached  --merge()-->  (новый Managed)
```

**Что происходит при persist() на detached-объекте:**

Согласно спецификации JPA (JSR-338, §3.2.4) и Oracle-документации: если передать detached-объект в `EntityManager.persist()`, провайдер **выбросит `IllegalArgumentException`**. Некоторые реализации могут вместо этого откатить транзакцию при commit с `PersistenceException`. Hibernate 5/6 следует спецификации: генерируется `IllegalArgumentException` с сообщением вида `"detached entity passed to persist"`.

Правильный подход: для повторного присоединения detached-объекта нужно использовать `merge()`, а не `persist()`.
