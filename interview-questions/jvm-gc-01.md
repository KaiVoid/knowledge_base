### Вопрос 1. Какие области памяти выделяет JVM? Опишите назначение каждой: Heap, Stack, Metaspace, PC Register, Native Method Stack.

**Категория:** JVM и GC · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Chapter 2. The Structure of the Java Virtual Machine — docs.oracle.com](https://docs.oracle.com/javase/specs/jvms/se8/html/jvms-2.html)

Спецификация JVM SE 8 определяет следующие области памяти времени выполнения (run-time data areas):

**Общие для всех потоков (JVM-wide):**

- **Heap (Куча)** — создаётся при старте JVM, существует до её остановки. Это область, из которой выделяется память для всех экземпляров классов и массивов. Освобождение памяти производится автоматически сборщиком мусора — объекты никогда не уничтожаются явно. Если вычислению требуется больше памяти, чем доступно, выбрасывается `OutOfMemoryError`.

- **Method Area (Область методов)** — также создаётся при старте JVM и разделяется между всеми потоками. Хранит структуры каждого класса: пул констант времени выполнения, данные полей и методов, код методов и конструкторов. Аналогична сегменту `.text` в исполняемых файлах ОС. При нехватке памяти — `OutOfMemoryError`.

- **Run-Time Constant Pool** — часть Method Area, создаётся для каждого класса или интерфейса при их загрузке. Содержит числовые литералы и символические ссылки на методы и поля, разрешаемые во время выполнения.

**Приватные для каждого потока (per-thread):**

- **PC (Program Counter) Register** — каждый поток имеет собственный счётчик команд. Для Java-метода содержит адрес текущей исполняемой инструкции JVM. Для нативного метода значение **не определено (undefined)**.

- **JVM Stack** — создаётся при создании потока. Хранит **фреймы** (frames) — по одному на каждый вызов метода. Каждый фрейм содержит массив локальных переменных, стек операндов и ссылку на пул констант класса. При превышении максимального размера стека — `StackOverflowError`; при невозможности выделить память — `OutOfMemoryError`.

- **Native Method Stacks** — поддерживают выполнение нативных методов (написанных на C/C++). Функционируют аналогично JVM Stack, но для кода вне JVM-байткода. Те же ошибки: `StackOverflowError` / `OutOfMemoryError`.

#### Ответ от Claude

JVM делит память на **разделяемые между потоками** и **приватные для каждого потока** области. В спецификации JVM SE формально описаны пять основных областей; Metaspace — это конкретная реализация Method Area в HotSpot начиная с Java 8.

**Общие (shared) области:**

| Область | Назначение |
|---|---|
| **Heap** | Хранит все экземпляры объектов и массивы. Управляется GC. При нехватке — `OutOfMemoryError: Java heap space`. |
| **Method Area / Metaspace** | Хранит метаданные классов: структуры классов, байткод методов, пул констант. В Java 8+ реализована как **Metaspace** — область нативной памяти (вне Java-кучи), растущая динамически. |

**Приватные (per-thread) области:**

| Область | Назначение |
|---|---|
| **JVM Stack** | Стек вызовов метода. Каждый вызов — новый фрейм с локальными переменными, стеком операндов и ссылкой на пул констант. Примитивные локальные переменные хранятся здесь. При переполнении — `StackOverflowError`. |
| **PC Register** | Счётчик команд: адрес текущей инструкции JVM-байткода. Для нативных методов — значение undefined. Каждый поток имеет свой. |
| **Native Method Stack** | Аналог JVM Stack для нативных методов (C/C++). Используется JNI. При переполнении — `StackOverflowError`. |

**Ключевые уточнения:**
- Metaspace — это **реализация** Method Area в HotSpot JVM (не отдельная спецификационная область). В Java 8 она была перенесена из Java-кучи в **нативную память**, что позволяет ей расти динамически (до `MaxMetaspaceSize` или системного лимита).
- Heap и Method Area/Metaspace **видны всем потокам**; JVM Stack, PC Register и Native Method Stack — **приватны** для каждого потока.
- String Pool (с Java 7+) и статические поля объектов хранятся в **Heap**, а не в Metaspace.

### Вопрос 2. Что такое Heap и Stack в JVM? Чем они отличаются и что хранится в каждой из этих областей?

**Категория:** JVM и GC · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Java Memory Model — jenkov.com](https://jenkov.com/tutorials/java-concurrency/java-memory-model.html)

Модель памяти JVM делит память между **стеками потоков** (thread stacks) и **кучей** (heap).

**Thread Stack (стек потока):**
Каждый поток, выполняющийся в JVM, имеет собственный стек, содержащий информацию о том, какие методы были вызваны в данном потоке до текущей точки выполнения. Помимо этого, стек потока содержит все локальные переменные для каждого выполняемого метода. Поток может получить доступ только к своему собственному стеку. Локальные переменные примитивных типов (`boolean`, `byte`, `short`, `char`, `int`, `long`, `float`, `double`) полностью хранятся в стеке потока и, следовательно, не видны другим потокам.

**Heap (куча):**
Куча содержит все объекты, созданные в Java-приложении, независимо от того, какой поток создал объект. Когда локальная переменная является ссылкой на объект, сама ссылка хранится в стеке потока, но объект хранится в куче. Переменные-члены объекта хранятся в куче вместе с самим объектом — это справедливо как для примитивных типов, так и для ссылок на другие объекты. Статические переменные класса также хранятся в куче вместе с определением класса.

#### Ответ от Claude

**Heap (куча)** и **Stack (стек)** — две принципиально разные области памяти JVM с разными целями, временем жизни и правилами доступа.

**Heap:**
- Разделяется между **всеми потоками** приложения.
- Хранит: все экземпляры объектов (`new Object()`), массивы, переменные-члены объектов (в том числе примитивные поля), String Pool (с Java 7+), статические переменные (с Java 8 хранятся в объектах-зеркалах классов на куче).
- Время жизни объекта — с момента создания (`new`) до тех пор, пока на него есть хотя бы одна достижимая ссылка. После — GC освобождает память.
- Управляется **сборщиком мусора**.
- При нехватке памяти — `OutOfMemoryError: Java heap space`.

**Stack (JVM Stack):**
- **Приватен** для каждого потока — другие потоки не имеют к нему доступа.
- Хранит: фреймы вызовов методов. Каждый фрейм содержит:
  - **Локальные переменные** примитивных типов (хранятся целиком).
  - **Ссылки** на объекты (сами объекты — на куче).
  - Стек операндов и ссылку на пул констант.
- Работает по принципу **LIFO**: при вызове метода фрейм добавляется, при возврате — удаляется.
- Управляется **JVM автоматически** (не GC).
- При переполнении — `StackOverflowError` (типично при бесконечной рекурсии).

**Ключевые отличия:**

| Характеристика | Heap | Stack |
|---|---|---|
| Видимость | Все потоки | Только владелец-поток |
| Что хранится | Объекты, массивы, поля объектов | Локальные переменные, ссылки, фреймы |
| Управление памятью | Сборщик мусора (GC) | JVM (автоматически, LIFO) |
| Время жизни данных | До сбора GC | До выхода из метода |
| Ошибка переполнения | `OutOfMemoryError` | `StackOverflowError` |
| Потокобезопасность | Требует синхронизации | Изолирован (потокобезопасен) |

**Важный нюанс:** примитив, объявленный как локальная переменная метода, хранится **в стеке**. Тот же примитив, объявленный как поле объекта, хранится **в куче** вместе с объектом. Ссылка (reference) всегда хранится там, где объявлена: если это локальная переменная — в стеке; если поле объекта — в куче.

### Вопрос 3. Что такое PermGen и почему он был заменён на Metaspace в Java 8? Чем Metaspace лучше?

**Категория:** JVM и GC · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Dealing with "java.lang.OutOfMemoryError: PermGen space" Error — docs.oracle.com (troubleshooting guide)](https://docs.oracle.com/javase/8/docs/technotes/guides/troubleshoot/memleaks002.html)

Начиная с Java 8, метаданные классов Java хранятся в нативной памяти и называются **Metaspace**. По умолчанию использование нативной памяти для метаданных классов не ограничено. Можно установить верхнюю границу с помощью параметра `MaxMetaSpaceSize`. Ошибка `java.lang.OutOfMemoryError: PermGen space` в Java 8 больше не выбрасывается, вместо неё используется `OutOfMemoryError: Metaspace`. Чтобы снизить вероятность подобных ошибок, можно уменьшить размер Java Heap — это освободит больше пространства для Metaspace.

#### Ответ от Claude

**PermGen (Permanent Generation)** — область памяти, существовавшая в HotSpot JVM до Java 8 как часть **Java-кучи** (heap). В ней хранились:
- Метаданные загруженных классов (описание методов, полей, байткод).
- Интернированные строки (`String.intern()`).
- Статические переменные.
- Пул констант классов.

**Главная проблема PermGen:** фиксированный размер. По умолчанию он был небольшим (типично 64–256 МБ), и при загрузке большого количества классов (например, в серверах приложений с множеством деплоев) возникала печально известная ошибка:
```
java.lang.OutOfMemoryError: PermGen space
```
Разработчикам приходилось вручную подбирать `-XX:MaxPermSize`, что было неудобно и требовало перезапуска.

**Metaspace (Java 8+)** заменил PermGen со следующими ключевыми изменениями:

| Характеристика | PermGen (до Java 8) | Metaspace (Java 8+) |
|---|---|---|
| Расположение | Часть Java Heap | **Нативная память** (вне Java Heap) |
| Размер | Фиксированный (`-XX:MaxPermSize`) | Динамически растёт (по умолчанию не ограничен) |
| Ошибка при переполнении | `OutOfMemoryError: PermGen space` | `OutOfMemoryError: Metaspace` |
| Хранение статических полей | Да | **Нет** — перенесены в Heap (в объекты-зеркала классов) |
| String Pool | Да (до Java 7; в Java 7 — уже Heap) | В Heap (начиная с Java 7) |
| Флаг настройки | `-XX:MaxPermSize` | `-XX:MaxMetaspaceSize` |

**Преимущества Metaspace:**
1. **Нет жёсткого лимита по умолчанию** — Metaspace растёт по мере необходимости, используя нативную память ОС.
2. **Автоматическая очистка** — GC может выгружать классы, когда метаданные достигают порогового значения, что снижает вероятность OOM.
3. **Нет необходимости подбирать размер вручную** — устраняет распространённую ошибку конфигурации.
4. **Уменьшение размера Heap даёт Metaspace больше нативной памяти** — они используют разные пространства адресов.

**Важный нюанс:** несмотря на то что Metaspace по умолчанию не ограничен, рекомендуется устанавливать `-XX:MaxMetaspaceSize` в production, чтобы избежать неконтролируемого потребления нативной памяти при утечках (например, при бесконечной загрузке классов через рефлексию или ByteBuddy/cglib без выгрузки).

### Вопрос 4. Что такое сборка мусора (Garbage Collection) в Java? Какие объекты считаются «мусором» и по какому принципу GC их находит?

**Категория:** JVM и GC · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Understanding Memory Management (JRockit JVM) — docs.oracle.com](https://docs.oracle.com/cd/E13150_01/jrockit_jvm/jrockit/geninfo/diagnos/garbage_collect.html)

Мусором называются объекты, которые больше не используются приложением. В процессе сборки мусора неиспользуемые объекты удаляются из памяти, освобождая место для новых объектов.

**Фаза маркировки (Mark Phase):** Все объекты, достижимые из Java-потоков, нативных дескрипторов и других корневых источников, **помечаются как живые**, а также все объекты, достижимые из этих объектов, и так далее. Этот процесс идентифицирует и помечает все объекты, которые ещё используются, а остальные считаются мусором.

**Фаза очистки (Sweep Phase):** Куча обходится в поисках разрывов между живыми объектами. Эти разрывы записываются в **список свободных блоков (free list)** и становятся доступны для новых выделений памяти.

JVM использует **генерационный сборщик мусора** (generational collection): для новых объектов выделяется специальная область — **nursery (молодое поколение)**. Когда она заполняется, выполняется «молодой сбор» (young collection), долгоживущие объекты перемещаются в **старшее поколение** (old space). Это обосновано тем, что большинство объектов живут очень недолго.

#### Ответ от Claude

**Garbage Collection (GC)** — механизм **автоматического управления памятью** в Java. Разработчик не освобождает память вручную; JVM автоматически обнаруживает объекты, которые более не нужны программе, и освобождает занятую ими память.

**Какие объекты считаются «мусором»?**

Объект считается мусором, если он **недостижим (unreachable)** — то есть на него нет ни одной ссылки из «живой» части программы. Точнее: объект является мусором, если нет пути по цепочке ссылок от любого **GC Root** до этого объекта.

**Что такое GC Roots?**

GC Roots — это «точки входа» для алгоритма трассировки. К ним относятся:
- **Локальные переменные и параметры методов**, находящихся в стеке выполняющихся потоков.
- **Статические поля классов** (ссылки, хранящиеся в объектах-зеркалах классов на куче).
- **JNI-ссылки** (объекты, на которые ссылается нативный код через JNI).
- **Активные потоки** (Thread objects).
- Объекты, на которые ссылаются классы, загруженные системным ClassLoader'ом.

**Как GC находит мусор: алгоритм Mark-and-Sweep**

Все реализации GC в HotSpot JVM (CMS, G1, ZGC, Shenandoah) являются **трассирующими коллекторами (tracing collectors)**:

1. **Фаза маркировки (Mark):** GC начинает с GC Roots и рекурсивно обходит граф объектов по ссылкам, помечая каждый достигнутый объект как «живой». Объекты, до которых добраться не удалось, остаются непомеченными — они и есть мусор.

2. **Фаза очистки (Sweep):** Непомеченные объекты удаляются, их память возвращается в пул свободных блоков.

3. **Фаза уплотнения (Compact, опционально):** Живые объекты перемещаются в начало кучи, устраняя фрагментацию (применяется, например, в G1 GC).

**Важно:** Java GC использует трассировку достижимости, а **не подсчёт ссылок (reference counting)**. Это означает, что **циклические ссылки** (объект A ссылается на B, B ссылается на A, но оба недостижимы из GC Roots) корректно обрабатываются — оба объекта будут собраны.

**Генерационный принцип:**

Основная гипотеза: **большинство объектов умирают молодыми** (weak generational hypothesis). Поэтому куча делится на поколения:
- **Young Generation (Eden + 2 Survivor):** Новые объекты создаются в Eden. Minor GC выполняется часто и быстро.
- **Old Generation (Tenured):** Долгоживущие объекты, пережившие несколько Minor GC. Major GC (или Full GC) выполняется реже и дороже.

Это позволяет GC большую часть времени работать только с небольшой молодой генерацией, не обходя весь heap.
### Вопрос 5. Объясните структуру Young Generation: что такое Eden и два Survivor (S0/S1)? Как объект перемещается между этими пространствами?

**Категория:** JVM и GC · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [3 Generations — Oracle Java SE 8 GC Tuning Guide](https://docs.oracle.com/javase/8/docs/technotes/guides/vm/gctuning/generations.html)

Young Generation (молодое поколение) состоит из пространства Eden и двух пространств Survivor. Большинство объектов изначально размещаются в Eden. Одно из пространств Survivor в любой момент времени пустое и служит местом назначения для живых объектов из Eden; другое пространство Survivor является местом назначения при следующей сборке мусора. Объекты копируются между пространствами Survivor до тех пор, пока не «состарятся» достаточно для перемещения в Tenured (старое) поколение. Затраты на Minor GC пропорциональны числу живых объектов, а не мёртвых — поэтому молодое поколение, заполненное мёртвыми объектами, собирается очень быстро.

#### Ответ от Claude

**Структура Young Generation**

Young Generation (молодое поколение) делится на три части:

- **Eden** — пространство, куда JVM помещает все новые объекты при их создании. По размеру значительно больше, чем каждый из Survivor.
- **Survivor 0 (S0)** и **Survivor 1 (S1)** — два пространства-«выживальщика» одинакового размера. В любой момент времени одно из них активно (содержит «выжившие» объекты), а другое — полностью пустое.

**Как объект перемещается между пространствами**

1. **Создание объекта** — объект размещается в Eden.
2. **Первый Minor GC** — когда Eden заполняется, JVM запускает Minor GC. Живые (достижимые) объекты из Eden копируются в пустой Survivor (например, S0); их возраст (age) устанавливается в 1. Мёртвые объекты просто удаляются.
3. **Последующие Minor GC** — на каждом следующем Minor GC живые объекты из Eden и из текущего заполненного Survivor (S0) копируются в другой, пустой Survivor (S1); их возраст увеличивается на 1. Роли S0 и S1 меняются («ping-pong»): теперь S1 активен, а S0 очищен.
4. **Повышение в Old Generation (Tenuring)** — когда возраст объекта превышает порог `MaxTenuringThreshold` (по умолчанию 15), при следующем Minor GC он копируется уже в Old Generation. Также в Old Generation попадают объекты, которые не помещаются в Survivor из-за нехватки места, и крупные объекты, которые изначально размещаются напрямую в Old Generation.

**Ключевые свойства:**
- Одно из Survivor всегда пусто — это гарантия, что всегда есть свободное место для копирования.
- JVM динамически выбирает `TenuringThreshold`, стремясь заполнять Survivor примерно наполовину.
- Параметр `-XX:MaxTenuringThreshold=15` задаёт максимальный порог возраста.

### Вопрос 6. Что такое Minor GC и Major GC (Full GC)? В чём разница и когда каждый из них срабатывает?

**Категория:** JVM и GC · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Managing Memory and Garbage Collection — Oracle Sun Java System Application Server Performance Tuning Guide](https://docs.oracle.com/cd/E19900-01/819-4742/6n6sfgmkr/index.html)

**Minor GC** срабатывает, когда заполняется молодое поколение (Eden + Survivor). Использует быстрый алгоритм копирования и занимает несколько миллисекунд. Например: `[GC 50650K->21808K(76868K), 0.0478645 secs]`.

**Full GC** собирает весь heap — и молодое, и старое поколение — и происходит значительно реже, но занимает значительно больше времени: от десятков миллисекунд до нескольких секунд в зависимости от размера heap. Пример: `[Full GC 52970K->1690K(76868K), 0.54789968 secs]`. Во время любого GC приложение не работает (Stop-the-World). Паузы Full GC более 4 секунд могут вызывать timeout-ошибки во внешних системах.

#### Ответ от Claude

**Minor GC**

- **Что собирает:** только Young Generation (Eden + S0 + S1).
- **Когда срабатывает:** когда Eden переполняется и новый объект не помещается в него.
- **Длительность паузы:** миллисекунды — сотни миллисекунд; пауза коротка, так как обрабатывается небольшая область heap.
- **Частота:** часто, так как молодые объекты создаются и умирают быстро.
- **Алгоритм:** копирование живых объектов в Survivor или Old Generation.

**Major GC**

- **Что собирает:** только Old Generation (Tenured).
- **Когда срабатывает:** когда Old Generation заполняется — обычно вследствие накопления объектов, переживших несколько Minor GC.
- **Длительность паузы:** значительно дольше Minor GC — от сотен миллисекунд до секунд.
- **Частота:** намного реже Minor GC.

**Full GC**

- **Что собирает:** весь heap — Young Generation + Old Generation + Metaspace (если нужно).
- **Когда срабатывает:**
  - Явный вызов `System.gc()` (рекомендация, а не гарантия).
  - Нехватка места в Old Generation при повышении объектов из Young.
  - Заполнение Metaspace (до Java 8 — PermGen).
  - Неудача при выделении памяти в Old Generation (promotion failure).
  - G1 GC: при ошибке эвакуации (evacuation failure).
- **Длительность паузы:** самые длинные паузы; весь heap обрабатывается с Stop-the-World.
- **Особенность:** в современной документации Oracle термины «Major GC» и «Full GC» нередко используются как синонимы, однако точнее: Major GC = сборка Old Generation, Full GC = сборка всего heap включая компактизацию.

**Итоговое сравнение:**

| Характеристика | Minor GC | Major/Full GC |
|---|---|---|
| Область | Young Generation | Old Generation / весь heap |
| Частота | Высокая | Низкая |
| Пауза | Короткая (мс) | Длинная (сотни мс — секунды) |
| Влияние на приложение | Минимальное | Значительное |

### Вопрос 7. Что такое Stop-the-World (STW) пауза? Почему она возникает при сборке мусора и чем опасна для приложения?

**Категория:** JVM и GC · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Introduction to Garbage Collection Tuning — Oracle Java SE 10](https://docs.oracle.com/javase/10/gctuning/introduction-garbage-collection-tuning.htm)

Stop-the-World (STW) — это период, когда сборщик мусора останавливает все потоки приложения для выполнения операций управления памятью. Фундаментальная опасность STW иллюстрируется законом Амдала: если приложение тратит даже 1% времени на GC на однопроцессорной системе, это превращается в более чем 20% потери производительности на 32-процессорной системе. При 10% времени на GC потери производительности превышают 75% на 32 ядрах. Два ключевых показателя GC — **throughput** (доля времени вне GC) и **паузы** (время, когда приложение кажется неотзывчивым).

#### Ответ от Claude

**Что такое STW-пауза**

Stop-the-World (STW) — это механизм, при котором JVM останавливает все пользовательские потоки приложения, чтобы сборщик мусора мог безопасно выполнить свою работу. Пока пауза активна, приложение полностью «замораживается»: не обрабатываются запросы, не выполняются вычисления, не продвигаются таймеры.

**Почему STW необходима**

Корень проблемы — согласованность состояния heap:

1. **Граф достижимости должен быть стабилен.** Чтобы определить, какие объекты «живые», GC обходит граф ссылок от GC-корней. Если потоки продолжают работать, они могут создавать новые ссылки, обнулять существующие или перемещать объекты — граф будет меняться прямо во время обхода, и GC не сможет получить корректную картину.
2. **Перемещение объектов требует атомарности.** Copying-коллекторы (в том числе Minor GC в HotSpot) перемещают живые объекты в новые регионы и обновляют все ссылки на них. Делать это при работающих потоках небезопасно: поток может прочитать старый адрес объекта до обновления ссылки.
3. **Фаза разметки (marking) в конкурентных коллекторах.** Даже CMS и G1, которые делают большую часть работы конкурентно, всё равно требуют коротких STW-пауз для initial mark и remark, чтобы «зафиксировать» начальный и финальный снимок множества живых объектов.

**Чем STW опасна для приложения**

- **Задержки (latency).** Для веб-серверов, финансовых систем и real-time приложений даже пауза в 200–500 мс может привести к истечению таймаутов на клиентской стороне и ошибкам.
- **Деградация производительности на многоядерных системах.** Согласно закону Амдала, 1% времени в GC на 32-ядерной машине даёт потерю 20% суммарной производительности; 10% времени в GC — потерю более 75%.
- **Cascade failures.** Длинные паузы Full GC (секунды) могут вызвать timeout в смежных системах: балансировщики нагрузки, СУБД, внешние API — все они ждут ответа, которого не получают.
- **Непредсказуемость.** STW-паузы не детерминированы по времени начала, что делает SLA-гарантии трудновыполнимыми.

**Как современные GC снижают STW**

- **G1 GC** — разбивает heap на регионы, собирает инкрементально, целевая пауза задаётся через `-XX:MaxGCPauseMillis` (по умолчанию 200 мс).
- **ZGC и Shenandoah** — конкурентное перемещение объектов; STW-паузы менее 1–10 мс независимо от размера heap.
- **CMS** — конкурентная фаза marking, короткие STW только для initial/remark.

### Вопрос 8. Какие виды OutOfMemoryError существуют в Java? Назовите минимум три и объясните причину каждого (например: java.lang.OutOfMemoryError: Java heap space, Metaspace, GC overhead limit exceeded).

**Категория:** JVM и GC · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [3 Troubleshoot Memory Leaks — Oracle Java SE 25 Troubleshooting Guide](https://docs.oracle.com/en/java/javase/25/troubleshoot/troubleshooting-memory-leaks.html)

Oracle перечисляет следующие виды `OutOfMemoryError`:

1. **Java heap space** — объект не может быть размещён в heap из-за нехватки места или невозможности расширить heap. Причины: утечка памяти (приложение держит ссылки на ненужные объекты), недостаточный размер heap, чрезмерное использование финализаторов.

2. **GC Overhead limit exceeded** — GC тратит более 98% времени на сборку, восстанавливая менее 2% heap в течение 5 последовательных сборок. Означает, что живые данные едва умещаются в heap. Можно отключить флагом `-XX:-UseGCOverheadLimit`.

3. **Metaspace** — нативная память для метаданных Java-классов исчерпана. Объём ограничен параметром `MaxMetaSpaceSize`. Решение: увеличить `MaxMetaSpaceSize` или уменьшить размер Java heap.

4. **Requested array size exceeds VM limit** — попытка создать массив, размер которого превышает лимит реализации JVM (независимо от наличия свободной памяти в heap).

5. **Out of swap space** — не удалось выполнить аллокацию в нативной куче; нативная память исчерпана.

6. **Compressed class space** — на 64-битных платформах с включённым `UseCompressedClassPointers` исчерпано фиксированное пространство для метаданных классов (`CompressedClassSpaceSize`).

#### Ответ от Claude

В Java `java.lang.OutOfMemoryError` — это `Error` (не `Exception`), сигнализирующий, что JVM не может выделить память для нового объекта или для внутренних нужд. Ниже — наиболее важные виды:

**1. Java heap space**

```
java.lang.OutOfMemoryError: Java heap space
```

*Причина:* heap заполнен — JVM не может выделить память для нового объекта, и GC не способен освободить достаточно места.

*Типичные сценарии:*
- Утечка памяти: статические коллекции, кеши без ограничения, слушатели событий, удерживающие объекты дольше нужного.
- Недостаточный размер heap для нагрузки (`-Xmx` слишком мал).
- Очередь финализаторов растёт быстрее, чем finalizer-поток её обрабатывает.

*Диагностика:* heap dump + анализ в Eclipse MAT или JDK Mission Control.

---

**2. GC overhead limit exceeded**

```
java.lang.OutOfMemoryError: GC overhead limit exceeded
```

*Причина:* JVM тратит более 98% процессорного времени на GC, а освобождает при этом менее 2% heap — и так в течение 5 подряд идущих сборок. Это защитный механизм: приложение де-факто уже не работает, лишь гоняет GC вхолостую.

*Отличие от "Java heap space":* heap может быть технически не полон, но настолько заполнен живыми объектами, что любая аллокация немедленно вызывает GC без результата.

*Как отключить (не рекомендуется):* `-XX:-UseGCOverheadLimit`

---

**3. Metaspace**

```
java.lang.OutOfMemoryError: Metaspace
```

*Причина:* исчерпана нативная память, отведённая под метаданные классов JVM (имена классов, байткод, константный пул и т.д.). Metaspace появился в Java 8 вместо PermGen. По умолчанию не имеет жёсткого ограничения (растёт до системного лимита), но при установленном `-XX:MaxMetaspaceSize` бросает этот error.

*Типичные сценарии:*
- Динамическая генерация классов (фреймворки с проксированием: Spring, Hibernate, CGLib).
- Утечка ClassLoader — ClassLoader не может быть GC-собран, пока хоть один его класс достижим.
- Деплой/передеплой без перезапуска JVM в контейнере сервлетов.

---

**4. Requested array size exceeds VM limit**

```
java.lang.OutOfMemoryError: Requested array size exceeds VM limit
```

*Причина:* попытка создать массив, размер которого превышает максимально допустимый в данной реализации JVM (на HotSpot это чуть меньше `Integer.MAX_VALUE` — около 2 млрд элементов). Память в heap может быть свободна.

---

**5. Out of swap space**

```
java.lang.OutOfMemoryError: Out of swap space?
```

*Причина:* не удалось выделить нативную (не-heap) память операционной системой. Возникает при исчерпании RAM + swap на уровне ОС, либо при нативных утечках в JNI-коде.

---

**Сводная таблица трёх основных:**

| Тип | Причина | Решение |
|---|---|---|
| Java heap space | Утечка или мало `-Xmx` | Увеличить heap / найти утечку |
| GC overhead limit exceeded | Heap почти полон, GC работает впустую | Увеличить heap / оптимизировать аллокации |
| Metaspace | Слишком много загруженных классов | Увеличить `MaxMetaspaceSize` / найти утечку ClassLoader |
### Вопрос 9. Что такое ClassLoader в Java? Назовите три стандартных загрузчика классов (Bootstrap, Platform/Extension, Application) и за что каждый отвечает.

**Категория:** JVM и GC · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [ClassLoader (Java SE 17 & JDK 17) — docs.oracle.com](https://docs.oracle.com/en/java/javase/17/docs/api/java.base/java/lang/ClassLoader.html)

ClassLoader — это абстрактный класс, ответственный за динамическую загрузку классов во время выполнения. Он преобразует двоичное имя класса в объект `Class`, который JVM может использовать. Помимо классов, ClassLoader умеет находить ресурсы, идентифицируемые путями, разделёнными символом `/`.

**Три встроенных загрузчика (Java 9+):**

1. **Bootstrap ClassLoader** — встроенный загрузчик JVM, не имеет родителя. В коде представлен как `null`. Загружает базовые классы JDK runtime (ядро Java SE платформы: `java.lang`, `java.util` и т.д.).

2. **Platform ClassLoader** — загружает классы платформы: API Java SE, их реализации, а также специфичные для JDK классы среды выполнения, определённые на уровне платформы или её предков. Доступен через `ClassLoader.getPlatformClassLoader()`.

3. **System (Application) ClassLoader** — также известен как загрузчик приложения. Загружает классы из classpath приложения, module path и специфичные для JDK инструменты. Доступен через `ClassLoader.getSystemClassLoader()`. Является родителем по умолчанию для новых экземпляров ClassLoader.

#### Ответ от Claude

**ClassLoader** — это механизм JVM, отвечающий за динамическую загрузку байт-кода классов из различных источников (файловая система, сеть, архивы JAR) и преобразование его в объекты `Class` во время выполнения программы. ClassLoader также отвечает за поиск ресурсов, связанных с приложением.

**Три стандартных загрузчика:**

| Загрузчик | Что загружает | Как получить |
|-----------|--------------|--------------|
| **Bootstrap** | Ядро JDK: `java.lang`, `java.util`, `java.io` и другие базовые классы Java SE | Представлен как `null` в API |
| **Platform** (ранее Extension) | Классы платформы Java SE, реализации API, JDK-специфичные классы runtime | `ClassLoader.getPlatformClassLoader()` |
| **Application** (System) | Классы из `-classpath`/`-cp`, module path приложения | `ClassLoader.getSystemClassLoader()` |

**Важное замечание об изменении в Java 9:** до Java 9 второй загрузчик назывался **Extension ClassLoader** и загружал расширения из директории `$JAVA_HOME/lib/ext`. После перехода на систему модулей (JPMS) в Java 9 он был переименован в **Platform ClassLoader** с расширенной ответственностью за загрузку модулей платформы. Иерархия: Bootstrap → Platform → Application.

Дочерние загрузчики видят классы, загруженные родительскими, но не наоборот.

### Вопрос 10. Что такое принцип делегирования (parent delegation model) в ClassLoader? Зачем он нужен и что произойдёт, если его нарушить?

**Категория:** JVM и GC · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Class Loading — docs.oracle.com/javase/jndi/tutorial](https://docs.oracle.com/javase/jndi/tutorial/beyond/misc/classloader.html)

Когда загрузчику классов требуется загрузить класс или ресурс, он сначала консультируется со своим родительским загрузчиком, прежде чем пытаться найти элемент самостоятельно. Родитель в свою очередь консультируется со своим родителем, и так далее. Только после того как все загрузчики-предки не могут найти элемент, текущий загрузчик занимается поиском сам.

Конкретная реализация `loadClass()` выполняет следующие действия по порядку:
1. Если класс уже был загружен — возвращает его.
2. Делегирует поиск нового класса родительскому загрузчику.
3. Если родительский загрузчик не находит класс, вызывает `findClass()` для поиска и загрузки класса самостоятельно.

Эта модель обеспечивает: видимость классов родителей дочерними загрузчиками, предотвращение дублирования загрузки, безопасность за счёт невозможности переопределить базовые классы Java.

#### Ответ от Claude

**Принцип делегирования (Parent Delegation Model)** — это алгоритм загрузки классов, при котором загрузчик перед самостоятельной попыткой загрузить класс всегда сначала передаёт запрос своему родительскому загрузчику.

**Алгоритм `loadClass()` в псевдокоде:**

```java
protected Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
    // 1. Проверить кэш — вдруг класс уже загружен
    Class<?> c = findLoadedClass(name);
    if (c == null) {
        // 2. Делегировать родителю
        if (parent != null) {
            c = parent.loadClass(name, false);
        } else {
            // Родитель — Bootstrap (представлен как null)
            c = findBootstrapClass(name);
        }
        if (c == null) {
            // 3. Только если родитель не нашёл — искать самому
            c = findClass(name);
        }
    }
    return c;
}
```

**Зачем нужна делегирование:**

1. **Безопасность** — злоумышленник не сможет подменить стандартный класс `java.lang.String` собственной реализацией: Bootstrap ClassLoader всегда загрузит его раньше.
2. **Уникальность классов** — один и тот же класс не загружается дважды: при повторном запросе возвращается уже загруженный экземпляр.
3. **Предсказуемость** — приложение всегда получает «официальные» реализации базовых классов JDK.
4. **Видимость** — дочерние загрузчики видят классы родителей, что позволяет использовать стандартные библиотеки из кода приложения.

**Что происходит при нарушении делегирования:**

- **ClassCastException / LinkageError** — если один и тот же класс загружен двумя разными загрузчиками (например, `com.example.Foo` загружен и ApplicationClassLoader, и кастомным загрузчиком), JVM считает их **разными типами**, даже если байт-код идентичен. Попытка привести объект одного к другому вызовет `ClassCastException`.
- **Нарушение безопасности** — кастомный загрузчик может загрузить поддельный `java.lang.SecurityManager` до обращения к Bootstrap, обходя защиту платформы (именно поэтому JVM с Java 9 дополнительно защищает пространство имён `java.*`).
- **Проблемы с фреймворками** — OSGi, некоторые серверы приложений (например, Tomcat) намеренно нарушают делегирование (child-first loading) для изоляции модулей. Это требует аккуратного управления загрузчиками во избежание конфликтов версий.

Реальный пример легитимного нарушения: **Thread Context ClassLoader** (TCCL) в Java EE/Jakarta EE позволяет коду системных библиотек (загруженных Bootstrap или Platform) обращаться к классам приложения через явную передачу загрузчика потока.

### Вопрос 11. Что такое генерационная гипотеза (Generational Hypothesis) и как она обосновывает разделение Heap на Young и Old Generation?

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Generations — Java SE 8 GC Tuning Guide, docs.oracle.com](https://docs.oracle.com/javase/8/docs/technotes/guides/vm/gctuning/generations.html)

Слабая генерационная гипотеза (Weak Generational Hypothesis) — это фундаментальный принцип стратегии сборки мусора в Java:

> «Большинство объектов живут очень короткое время.»

Это эмпирически наблюдаемое свойство означает, что подавляющее большинство выделяемых объектов становятся мусором вскоре после создания (например, объекты-итераторы, живущие лишь в течение одного цикла цикла), тогда как небольшая доля живёт долго.

JVM разделяет кучу на два поколения:

**Young Generation (молодое поколение):**
- Пространство Eden — место начального выделения памяти для большинства новых объектов
- Два пространства Survivor — одно всегда пустое, служит назначением для живых объектов из Eden; живые объекты копируются между ними до «взросления»

**Tenured Generation (старое поколение):**
- Содержит объекты, пережившие несколько циклов GC.

Минорные сборки выполняются часто и быстро, поскольку молодое поколение обычно заполнено мёртвыми объектами. Стоимость сборки пропорциональна числу **живых** объектов, а не общему их числу: «Молодое поколение, заполненное мёртвыми объектами, собирается очень быстро».

#### Ответ от Claude

**Генерационная гипотеза** (в литературе чаще — **Слабая генерационная гипотеза**, Weak Generational Hypothesis) — эмпирически установленное наблюдение: **большинство объектов «умирают молодыми»**, то есть становятся недостижимыми вскоре после создания.

Это подтверждается практикой: временные объекты (`StringBuilder`, итераторы, DTO в обработчиках запросов, объекты внутри методов) создаются и тут же отбрасываются. Долгоживущие объекты (кэши, сессии, статические структуры данных) составляют меньшинство.

**Как гипотеза обосновывает разделение Heap:**

| | Young Generation | Old Generation (Tenured) |
|-|-----------------|--------------------------|
| **Что хранит** | Свежие объекты (Eden + 2×Survivor) | Объекты, пережившие N минорных GC |
| **Размер** | Меньше (~1/3 кучи по умолчанию) | Больше (~2/3 кучи) |
| **Частота GC** | Высокая (minor GC) | Низкая (major/full GC) |
| **Скорость GC** | Очень высокая (миллисекунды) | Медленнее (сотни мс — секунды) |
| **Алгоритм** | Copying (копирование живых объектов) | Mark-Sweep-Compact |

**Жизненный цикл объекта:**

1. Объект создаётся в **Eden**.
2. При минорном GC живые объекты из Eden копируются в **Survivor S0**.
3. При следующем минорном GC — из Eden и S0 в **Survivor S1**; возраст объектов увеличивается.
4. Объекты, достигшие порога возраста (`-XX:MaxTenuringThreshold`, по умолчанию 15), **продвигаются (promote)** в Old Generation.
5. Когда Old Generation заполняется — выполняется **major (full) GC**.

**Ключевое преимущество:** поскольку большинство объектов умирают в Eden ещё до первого Survivor-копирования, минорный GC оперирует преимущественно **мёртвыми** объектами, а его стоимость пропорциональна числу **живых** — то есть остаётся малой. Это даёт высокую пропускную способность при коротких паузах.

### Вопрос 12. Как работает алгоритм G1 GC (Garbage-First)? Что такое регионы (regions), почему G1 называют «сборщиком с предсказуемыми паузами» и с какой версии Java он является сборщиком по умолчанию?

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Garbage-First (G1) Garbage Collector — Java SE 8 GC Tuning Guide, docs.oracle.com](https://docs.oracle.com/javase/8/docs/technotes/guides/vm/gctuning/g1_gc.html)

G1 — это серверный сборщик мусора для многопроцессорных машин с большими кучами (от 6 ГБ). Ключевые механизмы:

1. **Concurrent Global Marking** — разметка живых объектов по всей куче параллельно с работой приложения, что предотвращает паузы, пропорциональные размеру кучи.
2. **Selective Collection** — определяет и собирает регионы с наибольшим количеством мусора в первую очередь (отсюда название «Garbage-First»).
3. **Parallel Evacuation** — копирует живые объекты между регионами параллельно, уплотняя память.
4. **Pause Prediction Model** — использует исторические данные для оценки, сколько регионов можно собрать в рамках целевого времени паузы.

**Регионы:** G1 делит кучу на **равные по размеру фиксированные регионы** (смежные диапазоны виртуальной памяти). Каждый регион является либо частью логического молодого поколения, либо старого. «Humongous»-объекты (>50% размера региона) обрабатываются отдельно.

G1 обеспечивает предсказуемые паузы через параметр `-XX:MaxGCPauseMillis=<N>`. Документ оговаривает, что G1 «планируется как долгосрочная замена CMS».

#### Ответ от Claude

**G1 GC (Garbage-First)** — генерационный конкурентный сборщик мусора, представленный в Java 7u4, ставший сборщиком **по умолчанию начиная с Java 9** (JEP 248).

**Что такое регионы:**

Вместо монолитных непрерывных областей Young/Old G1 делит всю кучу на **равные регионы фиксированного размера** (от 1 до 32 МБ, всегда степень двойки, обычно ~2000 регионов). Каждый регион динамически выполняет одну из ролей:

| Роль региона | Описание |
|-------------|----------|
| **Eden** | Выделение памяти для новых объектов |
| **Survivor** | Живые объекты после минорного GC |
| **Old** | Долгоживущие объекты |
| **Humongous** | Объекты ≥ 50% размера региона |
| **Free** | Свободный, готов принять любую роль |

Регионы **не обязаны быть физически смежными** — это даёт гибкость: можно динамически перераспределять размер Young/Old без остановки приложения.

**Как работает алгоритм (два основных цикла):**

**1. Young-Only фаза:**
- Выполняются обычные минорные GC (STW: Stop-The-World), эвакуирующие Eden и Survivor регионы.
- Когда заполненность Old Generation достигает порога IHOP (`-XX:InitiatingHeapOccupancyPercent`, по умолчанию 45%), стартует **Concurrent Marking** — параллельная разметка живых объектов.
- Завершается паузами Remark и Cleanup.

**2. Space-Reclamation фаза:**
- **Mixed Collections** — GC собирает и молодые, и выбранные старые регионы.
- G1 продолжает, пока эвакуация регионов даёт достаточный «выигрыш» свободного пространства.

**Почему G1 называют «сборщиком с предсказуемыми паузами»:**

G1 использует **модель предсказания пауз** на основе статистики предыдущих сборок:
- Для каждого региона оценивается стоимость его эвакуации (сколько живых объектов, сколько времени занимает копирование).
- Перед каждой GC G1 выбирает такое количество регионов, чтобы суммарное время паузы не превысило цель: `-XX:MaxGCPauseTimeMillis=200` (200 мс по умолчанию).
- Имя «Garbage-First» отражает стратегию: сначала собираются регионы с **наибольшим процентом мусора** — максимальная польза при минимальных паузах.

Важно: цель паузы — **вероятностная**, а не жёсткая гарантия реального времени.

**С какой версии Java G1 является сборщиком по умолчанию:**

- **Java 7u4** — G1 впервые поддерживается как production-ready.
- **Java 9** — G1 становится сборщиком по умолчанию для серверных конфигураций (JEP 248), заменив Parallel GC. Серверная конфигурация: ≥2 аппаратных потока и ≥1792 МБ ОЗУ.
- На однопроцессорных машинах или с малым объёмом памяти по-прежнему использовался Serial GC.

**Явное включение:** `-XX:+UseG1GC` (не требуется с Java 9+, но допустимо для явности).
### Вопрос 13. Чем ZGC принципиально отличается от G1? Как ZGC достигает пауз менее 10 мс на heap объёмом в сотни гигабайт — что такое «цветные указатели» (colored pointers) и барьеры загрузки (load barriers)?

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Deep-dive of ZGC's Architecture — dev.java](https://dev.java/learn/jvm/tool/garbage-collection/zgc-deepdive/)

ZGC достигает пауз менее 1 мс за счёт того, что почти все операции GC — маркировка, перемещение объектов, обработка ссылок, выгрузка классов, сканирование стеков потоков — выполняются конкурентно (параллельно с работой приложения). STW-фазы сводятся к коротким точкам синхронизации, время которых не зависит от размера живого множества.

**Цветные указатели (Colored Pointers):** ZGC хранит метаданные прямо в 64-битном указателе на объект — 20 бит зарезервировано под метаданные. Из них 4 бита используются активно:
- **Finalizable** — объект достижим только через финализаторы;
- **Remapped** — указатель уже обновлён после перемещения объекта;
- **Marked0 / Marked1** — флаги маркировки, чередующиеся между GC-циклами.

**Барьеры загрузки (Load Barriers):** При каждом чтении ссылки на объект из кучи JVM (через C2) вставляет небольшой код-барьер. Барьер проверяет «цвет» указателя и, если объект был перемещён, немедленно обновляет адрес («самоисцеление» указателя), не блокируя поток. Именно это позволяет ZGC перемещать объекты конкурентно.

В отличие от G1, который выполняет эвакуацию (перемещение объектов из регионов) в STW-фазе, ZGC делает это полностью конкурентно.

#### Ответ от Claude

**Ключевые архитектурные отличия ZGC от G1:**

| Аспект | G1 GC | ZGC |
|---|---|---|
| Паузы STW | Настраиваемые, типично 50–200 мс, растут с heap | Менее 1 мс, не зависят от размера heap |
| Эвакуация объектов | Выполняется в STW-фазе | Выполняется конкурентно с приложением |
| Барьеры | Write barriers (отслеживание записей) | Load barriers (отслеживание чтений) |
| Метаданные | Отдельные структуры (card table, remembered sets) | Закодированы в самом указателе (colored pointers) |
| Регионы | Фиксированного размера | Динамические: Small (2 МБ), Medium (4–32 МБ), Large |
| Overhead CPU | Умеренный | Выше из-за конкурентной работы |

**Как ZGC достигает пауз менее 1 мс на heap от ГБ до ТБ:**

1. **Всё конкурентно.** Маркировка, перемещение объектов, обработка слабых ссылок, выгрузка классов — всё это идёт параллельно с приложением. STW нужен только для начального и финального root-сканирования (минимальное множество корней).

2. **Цветные указатели (Colored Pointers).** В 64-битном указателе ZGC использует 4 метабита:
   - `Marked0` / `Marked1` — состояние маркировки (два флага чередуются между циклами, чтобы не сбрасывать все биты);
   - `Remapped` — объект уже «переехал» и указатель обновлён;
   - `Finalizable` — объект реферируется только через finalizer.
   
   Благодаря этому GC знает состояние каждого объекта без дополнительных таблиц, что позволяет атомарно отслеживать и обновлять состояние объектов в конкурентном режиме.

3. **Барьеры загрузки (Load Barriers).** Компилятор C2 вставляет небольшой инлайн-код при каждом чтении ссылки из heap (`obj.field`). Этот код:
   - Проверяет «цвет» (биты метаданных) указателя;
   - Если объект был перемещён, немедленно обновляет указатель до нового адреса;
   - Возвращает корректный (актуальный) указатель вызывающему коду.
   
   Такой «self-healing» механизм позволяет перемещать объекты конкурентно: старые указатели автоматически исправляются при следующем же обращении, без STW.

**Когда использовать ZGC vs G1:**
- **ZGC** — когда критична стабильно низкая задержка (финтех, игры, real-time сервисы), особенно на heap > 4 ГБ.
- **G1** — стандартный выбор для большинства серверных приложений, где баланс пропускной способности и задержки важнее максимально низких пауз.

**Примечание о точности:** Официальная цель ZGC — паузы не более 10 мс (JEP 333). На практике на современных JVM (Java 15+, где ZGC стал production-ready) паузы составляют менее 1 мс. Встречающееся в документации «менее 10 мс» — это первоначальная цель проекта, фактические паузы значительно ниже.

### Вопрос 14. Что такое JIT-компилятор в JVM? Чем он отличается от AOT-компиляции? Объясните понятия C1 и C2 (tiered compilation) и что такое «горячий» метод (hot method).

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [4 Compilation Optimization — Oracle JRockit/HotSpot Migration Guide](https://docs.oracle.com/javacomponents/jrockit-hotspot/migration-guide/comp-opt.htm)

HotSpot JVM включает интерпретатор байткода и два JIT-компилятора:
- **C1 (Client)** — быстрая компиляция с небольшим набором оптимизаций, подходит для быстрого старта;
- **C2 (Server)** — медленная компиляция с агрессивными оптимизациями, обеспечивает максимальную пиковую производительность.

По умолчанию JVM интерпретирует байткод и JIT-компилирует только «горячие» методы, превысившие порог вызовов. Пороги:
- Server JVM: **10 000** вызовов;
- Client JVM: **1 500** вызовов.

Tiered Compilation (введена в Java 7, по умолчанию в Java 8+) объединяет оба компилятора. Метод проходит 5 уровней: 0 — интерпретатор, 1–3 — C1 с разной степенью профилирования, 4 — C2 с полными оптимизациями на основе накопленного профиля.

#### Ответ от Claude

**JIT-компилятор (Just-In-Time):**

JIT — это компилятор, встроенный в JVM, который во время выполнения программы переводит байткод (`.class`-файлы) в нативный машинный код для конкретной платформы. JIT работает по принципу «compile on demand»: сначала код интерпретируется, и JVM собирает профиль выполнения (как часто вызываются методы, какие типы используются), а затем компилирует «горячие» участки с учётом этой информации.

**JIT vs AOT:**

| Характеристика | JIT (Just-In-Time) | AOT (Ahead-Of-Time) |
|---|---|---|
| Когда компилирует | Во время выполнения | До запуска (при сборке) |
| Оптимизации | На основе реального профиля выполнения | Статические, без профиля |
| Время старта | Медленнее (нужно «прогреться») | Быстрее (нативный код готов) |
| Пиковая производительность | Выше (profile-guided оптимизации) | Ниже (нет данных о реальных путях) |
| Портируемость | Максимальная (байткод один, нативный код разный) | Платформо-зависимый артефакт |
| Пример | HotSpot JVM по умолчанию | GraalVM Native Image, Java 9 AOT (JEP 295) |

**Tiered Compilation — 5 уровней:**

| Уровень | Компилятор | Описание |
|---|---|---|
| 0 | Интерпретатор | Байткод выполняется построчно, собирается профиль |
| 1 | C1 | Быстрая компиляция без профилирования (тривиальные методы) |
| 2 | C1 | Компиляция с ограниченным профилированием |
| 3 | C1 | Компиляция с полным профилированием (счётчики вызовов, типы) |
| 4 | C2 | Агрессивная оптимизация на основе полного профиля уровня 3 |

Типичный путь «горячего» метода: 0 → 3 → 4. Простые методы могут остановиться на уровне 1 или 2.

**«Горячий» метод (Hot Method):**

Метод считается «горячим», когда превышает порог выполнения. HotSpot отслеживает два счётчика:
- **Invocation Counter** — число вызовов метода (порог по умолчанию: 10 000 для Server JVM);
- **Back-Edge Counter** — число итераций циклов внутри метода (для компиляции прямо во время выполнения цикла — OSR, On-Stack Replacement).

При превышении порога JVM ставит метод в очередь на JIT-компиляцию, не прерывая его выполнение.

**Оптимизации C2 (примеры):**
- Inline-подстановка вызовов методов (inlining);
- Развёртка циклов (loop unrolling);
- Устранение мёртвого кода (dead code elimination);
- Escape analysis — если объект не выходит за пределы метода, он может быть выделен на стеке или скалярно замещён (без выделения в heap);
- Девиртуализация вызовов (если тип известен из профиля).

**Деоптимизация:** Если предположения C2 оказываются неверными (например, приходит новый подтип), JVM выполняет деоптимизацию — откатывает нативный код обратно к интерпретируемому или C1, затем перекомпилирует с новыми данными.

### Вопрос 15. Что такое Java Memory Model (JMM)? Объясните отношение happens-before: какие операции его устанавливают и зачем оно нужно для корректной работы многопоточных программ?

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Java Memory Model (JMM) and Happens-Before Relationships Explained — prgrmmng.com](https://prgrmmng.com/java-memory-model-happens-before)

JMM — это спецификация, определяющая, «как и когда изменения, сделанные одним потоком, становятся видимыми другим». Без неё многопоточные программы вели бы себя непредсказуемо из-за:
- **CPU-кешей** (в 10–100 раз быстрее основной памяти);
- **Store-буферов** (асинхронная запись в память);
- **Переупорядочивания инструкций** компилятором (JIT);
- **Переупорядочивания инструкций** процессором (pipeline).

**Happens-before** — базовое понятие JMM. Если операция A happens-before операции B, то все эффекты A гарантированно видны B, и A логически упорядочена перед B.

Операции, устанавливающие happens-before (8 правил):
1. **Program Order** — в одном потоке каждая операция HB следующей;
2. **Monitor Lock Rule** — снятие блокировки HB перед захватом той же блокировки;
3. **Volatile Variable Rule** — запись в volatile HB перед всеми последующими чтениями;
4. **Thread Start Rule** — `Thread.start()` HB перед первой операцией нового потока;
5. **Thread Join Rule** — все операции потока HB перед возвратом из `Thread.join()`;
6. **Interruption Rule** — `thread.interrupt()` HB перед обнаружением прерывания;
7. **Finalizer Rule** — завершение конструктора HB перед запуском финализатора;
8. **Transitivity** — если A HB B и B HB C, то A HB C.

#### Ответ от Claude

**Java Memory Model (JMM) — что это и зачем нужна:**

JMM — это формальная спецификация (часть JLS, Java Language Specification), описывающая:
1. **Видимость (Visibility):** когда запись одного потока становится видна другому потоку;
2. **Атомарность (Atomicity):** какие операции неделимы;
3. **Упорядоченность (Ordering):** в каком порядке потоки наблюдают операции друг друга.

**Проблема, которую решает JMM:**

Без JMM поток B может не увидеть запись потока A даже после её завершения по следующим причинам:
- Значение осталось в регистре/L1-кеше процессора A и не сброшено в RAM;
- Компилятор переупорядочил инструкции для оптимизации;
- Процессор выполнил инструкции out-of-order.

JMM не запрещает эти оптимизации — она лишь определяет **при каких условиях** поток гарантированно видит записи другого потока. Именно для этого и существует happens-before.

**Happens-Before (HB) — подробно:**

HB — это отношение частичного порядка на операциях чтения/записи. Если A happens-before B:
- Всё, что записал поток до A (включительно), видно потоку, выполняющему B и последующие операции;
- A логически упорядочена перед B (хотя физически процессор может выполнять иначе, JVM обязана соблюсти наблюдаемый результат).

**Правила установления HB:**

| Правило | Описание |
|---|---|
| **Program Order** | Внутри одного потока: операция N HB операции N+1 |
| **Monitor Lock** | `unlock(m)` HB `lock(m)` для того же монитора `m` |
| **Volatile Write** | Запись в volatile-переменную HB любому последующему чтению той же переменной |
| **Thread Start** | `thread.start()` HB любой операции внутри стартованного потока |
| **Thread Join** | Любая операция завершённого потока HB возврату из `thread.join()` |
| **Thread Interrupt** | `thread.interrupt()` HB обнаружению прерывания потоком |
| **Finalizer** | Завершение конструктора объекта HB запуску его `finalize()` |
| **Transitivity** | Если A HB B и B HB C, то A HB C |
| **Synchronizers** | Операции с `CountDownLatch`, `Semaphore`, `CyclicBarrier` и т.д. устанавливают HB согласно их контракту |

**Пример с volatile:**
```java
// Поток 1:
sharedData = 42;       // (1) обычная запись
flag = true;           // (2) запись в volatile

// Поток 2:
if (flag) {            // (3) чтение volatile — HB установлен с (2)
    use(sharedData);   // (4) гарантированно видит 42 благодаря transitivity: (1) HB (2) HB (3) HB (4)
}
```

**Пример без HB (data race):**
```java
// Без volatile/synchronized нет HB между потоками:
// Поток 2 может увидеть flag == true, но sharedData == 0
```

**Зачем HB нужен на практике:**
- Позволяет писать корректные многопоточные программы без избыточной синхронизации;
- Даёт формальную основу для рассуждений о видимости данных между потоками;
- Определяет минимально необходимые барьеры памяти (memory barriers), которые JVM должна вставить в нативный код.

### Вопрос 16. Что делает ключевое слово volatile в Java? Гарантирует ли оно атомарность операций? Приведите пример, где volatile недостаточно и нужен synchronized или AtomicInteger.

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Difference between Atomic, Volatile and Synchronized in Java — javarevisited.blogspot.com](https://javarevisited.blogspot.com/2020/04/difference-between-atomic-volatile-and-synchronized-in-java-multi-threading.html)

`volatile` гарантирует **видимость**, но **не атомарность**. Запись в volatile-переменную happens-before всем последующим чтениям этой переменной. Кроме того, volatile запрещает компилятору кешировать переменную и переупорядочивать операции вокруг неё.

Однако `count++` — трёхшаговая операция (читай → изменяй → записывай), и даже при `volatile` два потока могут прочитать одинаковое значение и оба записать одинаковый результат, потеряв одно приращение.

**Пример сломанного кода с volatile:**
```java
private volatile int count = 1;
public int getCount() {
    return count++;  // НЕ атомарно — потери обновлений
}
// Ожидали 5001, получили 4174–4293 (варьируется от запуска к запуску)
```

**Правильный подход:**
- `synchronized` — обеспечивает взаимное исключение и видимость;
- `AtomicInteger` — атомарные операции через CAS (compare-and-swap) без явных блокировок, производительнее `synchronized` для одной переменной.

#### Ответ от Claude

**Что делает volatile:**

`volatile` обеспечивает два гарантии в JMM:

1. **Видимость (Visibility):** Запись volatile-переменной сразу сбрасывается в основную память; чтение всегда происходит из основной памяти, а не из кеша CPU. Таким образом, все потоки всегда видят актуальное значение.

2. **Запрет переупорядочивания (Ordering):** JVM не может переставить запись volatile с предшествующими ей операциями (и чтение — с последующими), что обеспечивает правило Volatile Variable Rule (happens-before).

**Атомарность — что volatile гарантирует, а что нет:**

- **Гарантирует атомарность** для чтения и записи примитивов размером до 64 бит: `int`, `long`, `double`, ссылки — volatile-чтение/запись атомарны (без volatile `long`/`double` могут читаться/записываться по 32 бита двумя операциями).
- **Не гарантирует атомарность** составных операций: `count++`, `count += n`, `if (x == 0) x = 1` и т.д. — это несколько операций, которые могут прерваться другим потоком между шагами.

**Пример: где volatile недостаточно**

```java
// НЕПРАВИЛЬНО — volatile не помогает для count++
public class BrokenCounter {
    private volatile int count = 0;

    public void increment() {
        count++; // READ → INCREMENT → WRITE (три шага!)
        // Два потока могут прочитать одинаковый count = 5,
        // оба вычислят 6, оба запишут 6 → одно приращение потеряно
    }
}

// ПРАВИЛЬНО — вариант 1: synchronized
public class SynchronizedCounter {
    private int count = 0;

    public synchronized void increment() {
        count++; // только один поток одновременно
    }
    public synchronized int get() { return count; }
}

// ПРАВИЛЬНО — вариант 2: AtomicInteger (предпочтительно для одной переменной)
public class AtomicCounter {
    private final AtomicInteger count = new AtomicInteger(0);

    public void increment() {
        count.incrementAndGet(); // атомарный CAS: одна hardware-инструкция
    }
    public int get() { return count.get(); }
}
```

**Когда volatile достаточно:**

```java
// Флаг остановки потока — volatile достаточно:
// один поток пишет, другой читает — нет составных операций
private volatile boolean running = true;

public void stop() { running = false; }

public void run() {
    while (running) { /* ... */ }
}
```

**Сравнительная таблица:**

| Механизм | Видимость | Атомарность составных операций | Взаимное исключение | Производительность |
|---|---|---|---|---|
| `volatile` | Да | Нет | Нет | Высокая |
| `synchronized` | Да | Да | Да | Средняя (блокировка) |
| `AtomicInteger` | Да | Да (для числовых операций) | Нет (lock-free) | Высокая (CAS) |

**Когда что использовать:**
- **volatile** — один поток пишет, один или несколько читают; простой флаг или статус; ссылка на иммутабельный объект, заменяемый атомарно.
- **synchronized** — составные операции над несколькими переменными; нужен инвариант между переменными; редкий доступ.
- **AtomicInteger/AtomicReference** — независимые числовые счётчики и CAS-операции; высокая конкуренция на одной переменной.

### Вопрос 17. Что такое утечка памяти (memory leak) в Java? Может ли она произойти, если GC работает? Приведите реальный пример (например, статические коллекции, слушатели событий, ClassLoader leak).

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [How to Spot and Fix Memory Leaks in Java? — Last9](https://last9.io/blog/how-to-spot-and-fix-memory-leaks-in-java/)

Утечка памяти в Java возникает, когда приложение «непреднамеренно удерживает память, которая больше не используется, приводя к избыточному потреблению памяти». Несмотря на то что сборщик мусора автоматически управляет памятью, он не может освободить объекты, на которые до сих пор есть активные ссылки. Если приложение поддерживает ненужные ссылки на объекты, GC не может вмешаться — ответственность за предотвращение утечек лежит на разработчике.

**Статические коллекции.** Статические поля — «одна из наиболее частых причин утечек памяти». Объекты, хранящиеся в статических кешах или коллекциях, никогда не собираются GC, так как статические поля принадлежат классу, а не экземпляру. Разработчик обязан явно удалять объекты, которые больше не нужны.

**Слушатели событий.** Незарегистрированные слушатели вызывают утечки в событийно-ориентированных системах. Объекты, подписанные на события через слушателей, остаются в памяти, если эти слушатели не отменены регистрацию после окончания использования.

**ThreadLocal-переменные.** Данные ThreadLocal привязаны к жизненному циклу потока. Если значения ThreadLocal не удаляются явно после использования, они остаются в памяти — особенно опасно в средах с пулами потоков.

#### Ответ от Claude

**Определение.** Утечка памяти в Java — ситуация, когда объекты больше не нужны приложению, но продолжают удерживаться в памяти через цепочку ссылок GC-корней. GC не может собрать объект до тех пор, пока на него существует хотя бы одна живая ссылка. Это принципиальное отличие от C/C++: в Java утечка — всегда проблема **достижимости**, а не забытого `free()`.

**Может ли утечка произойти при работающем GC?** Да, и это главное заблуждение. GC работает корректно — он собирает все *недостижимые* объекты. Однако если объект *достижим* из живого GC-корня (статическое поле, поток, стек), но логически «мёртв» с точки зрения бизнес-логики, GC не имеет права его трогать. Итог: `OutOfMemoryError` при технически исправном GC.

**Пример 1 — статическая коллекция:**
```java
public class Cache {
    // Статическое поле живёт столько же, сколько ClassLoader класса Cache
    private static final List<byte[]> CACHE = new ArrayList<>();

    public void process(byte[] data) {
        CACHE.add(data); // объекты накапливаются, никогда не удаляются
    }
}
```
Объекты в `CACHE` недостижимы извне бизнес-логики, но достижимы из статического поля класса — GC не удалит их никогда. Решение: использовать `WeakHashMap`, явно вызывать `remove()`, или применять кеши с ограниченным временем жизни (Caffeine, Guava Cache).

**Пример 2 — слушатели событий:**
```java
// Регистрируем слушателя
eventBus.addListener(myListener);

// Забываем отменить регистрацию после того, как listener больше не нужен
// eventBus.removeListener(myListener); // <- эта строка никогда не вызывается
```
`eventBus` (как правило, статический синглтон) хранит ссылку на `myListener`, который в свою очередь держит ссылку на внешний объект. Пока `eventBus` жив, весь граф объектов неудаляем. Это классическая утечка в GUI-фреймворках (Swing, SWT, JavaFX), а также в event bus-системах (Guava EventBus, Spring ApplicationEventPublisher).

**Пример 3 — ClassLoader leak (hot-deploy):**
При каждом передеплое веб-приложения сервер приложений создаёт новый `WebAppClassLoader`. Старый должен быть собран GC. Но если, например, JDBC-драйвер зарегистрировался в `DriverManager` (который принадлежит системному ClassLoader) или ThreadLocal в пуле потоков сервера держит ссылку на объект, загруженный старым ClassLoader, — старый ClassLoader не может быть собран. После нескольких передеплоев накапливается `OutOfMemoryError: Metaspace` (Java 8+) или `PermGen space` (Java 7-).

**Другие типичные паттерны:** незакрытые `InputStream`/`Connection`, объекты в `ThreadLocal` без вызова `remove()`, подписки на JMX MBeans, объекты в `HttpSession` без явного удаления.

**Инструменты диагностики:** heap dump + Eclipse MAT, VisualVM, JProfiler, `jmap -dump:live,...`, флаг `-XX:+HeapDumpOnOutOfMemoryError`.

### Вопрос 18. Как ClassLoader связан с OutOfMemoryError: Metaspace? Что такое «утечка ClassLoader» и в каких случаях она возникает (например, при hot-deploy в серверах приложений)?

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ClassLoader Leaks — Mattias Jiderhamn Blog](https://java.jiderhamn.se/category/classloader-leaks/)

Утечки ClassLoader возникают, когда неиспользуемые ClassLoader не могут быть собраны GC, потому что внешние ссылки препятствуют их удалению. Это вызывает ошибки `java.lang.OutOfMemoryError: PermGen space` (Java 7) или `Metaspace` (Java 8+), так как память накапливается с каждым удалённым экземпляром приложения.

**Причины в сценариях hot-deploy:** серверы приложений используют пулы потоков, которые живут дольше отдельных приложений. При передеплое старый ClassLoader должен быть собран GC, но если ссылки с уровня сервера остаются, ClassLoader бесконечно удерживается в памяти. Многократный передеплой без правильной очистки в итоге исчерпывает доступную память.

**Три категории утечек:**
1. **Ссылки извне веб-приложения** — системные классы или код сервера приложений держат ссылки на классы или сам ClassLoader.
2. **Незавершённые потоки** внутри ClassLoader — потоки переживают экземпляры приложений через пулы потоков сервера.
3. **Неочищенные ThreadLocal** — хранят значения, загруженные ClassLoader; серверы переиспользуют потоки, поэтому значения сохраняются до явного удаления.

**Конкретные виновники:** Apache Axis (незакрытые ThreadLocal в XML-утилитах), Apache Batik (статические потоки с бесконечными циклами), Java Advanced Imaging (хуки завершения на уровне JVM), `java.sql.DriverManager` (удерживает ссылки на JDBC-драйверы), `java.beans.Introspector` (кеши строгих ссылок).

#### Ответ от Claude

**Как Metaspace связан с ClassLoader.** В JVM (Java 8+) область Metaspace хранит метаданные классов: байткод методов, поля, константный пул, дескрипторы типов. Каждый загруженный класс занимает место в Metaspace. Ключевое правило: **класс живёт ровно столько, сколько живёт его ClassLoader**. Если ClassLoader невозможно собрать GC (потому что на него или на любой его объект существует живая ссылка снаружи), все классы этого ClassLoader остаются в Metaspace вечно. При многократном передеплое Metaspace заполняется «мёртвыми» копиями классов — и в итоге выбрасывается `OutOfMemoryError: Metaspace`.

**Механизм утечки ClassLoader пошагово:**
1. Сервер приложений (Tomcat, JBoss, WebSphere) создаёт `WebAppClassLoader` при деплое.
2. Веб-приложение загружает классы, регистрирует потоки, создаёт ThreadLocal, регистрирует JDBC-драйверы.
3. При передеплое/анддеплое сервер создаёт новый `WebAppClassLoader` и должен собрать старый.
4. Но: если хотя бы один объект из шага 2 остаётся достижимым через GC-корень за пределами старого ClassLoader — весь старый ClassLoader, и все его классы, остаются в памяти.

**Типичные GC-корни, удерживающие ClassLoader:**
- `java.sql.DriverManager` — статический реестр в системном ClassLoader, хранит ссылки на `Driver`-экземпляры, загруженные через `Class.forName()` в WebAppClassLoader.
- **ThreadLocal в пуле потоков сервера** — поток принадлежит серверу (системный ClassLoader), но значение `ThreadLocal` является экземпляром класса из WebAppClassLoader.
- **Статические поля в родительском ClassLoader** — логгеры, MBean-ы и другие синглтоны, зарегистрированные в системном ClassLoader, держат ссылку на объекты приложения.
- **Shutdown hooks** — `Runtime.addShutdownHook()` регистрирует Thread на уровне JVM; этот поток держит ссылку на Runnable из WebAppClassLoader.
- **Сторонние библиотеки**: log4j, Commons Logging, Hibernate/CGLIB, Apache Axis — все они в разное время были известны созданием таких утечек.

**Симптомы:** после каждого передеплоя память Metaspace растёт и не возвращается к исходному уровню. После N передеплоев — `OOM: Metaspace`.

**Диагностика:** `jmap -dump:live,format=b,file=heap.hprof <pid>` + Eclipse Memory Analyzer. MAT покажет «Class Loader Explorer» — сколько ClassLoader живы и что их удерживает. Инструмент [classloader-leak-prevention](https://github.com/mjiderhamn/classloader-leak-prevention) Маттиаса Йидерхамна автоматически закрывает большинство известных утечек через `ServletContextListener`.

**Настройки JVM:** `-XX:MetaspaceSize=256m -XX:MaxMetaspaceSize=512m` задают начальный и максимальный размер Metaspace. Увеличение `MaxMetaspaceSize` — временный костыль, а не решение; настоящее решение — устранить ссылки.

**Профилактика:** регистрировать `Driver` через `DriverManager.deregisterDriver()` в `contextDestroyed()`, вызывать `ThreadLocal.remove()` в `finally`-блоках, останавливать собственные потоки в `contextDestroyed()`, не регистрировать shutdown hooks из кода приложения.

### Вопрос 19. Что такое «продвижение» (promotion) объекта из Young в Old Generation? Что такое Tenuring Threshold и как настроить его через JVM-флаги?

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Understanding Generational GC: Young, Old & Promotion — GCeasy Blog](https://blog.gceasy.io/understanding-generational-gc-young-old-promotion/)

«Продвижение» объекта (promotion) — процесс перемещения объекта из Young Generation в Old Generation после того, как он пережил определённое количество циклов GC. Порог продвижения (tenuring threshold) определяет, сколько раз объект должен пережить сборку мусора в Young Generation, прежде чем он будет перемещён в Old Generation.

По умолчанию порог равен 15 циклам; управляется флагом `-XX:MaxTenuringThreshold`. «Если указать `-XX:MaxTenuringThreshold=15`, то любой объект, переживший 15 циклов GC в Young Gen, будет продвинут в Old Gen».

**Пошаговый процесс:**
1. Новые объекты создаются в Eden Space.
2. Когда Eden заполняется, запускается Minor GC; недостижимые объекты удаляются.
3. Выжившие объекты перемещаются в Survivor Space S0 с возрастом 1.
4. При следующем Minor GC выжившие объекты «пинг-понгятся» между S0 и S1, возраст каждый раз увеличивается на 1.
5. Как только возраст объекта достигает порога продвижения (по умолчанию 15), объект перемещается в Old Generation.

**Досрочное продвижение** возникает при нехватке места в Survivor Spaces, при давлении памяти в Young Gen или при JVM-эргономике, корректирующей ограничения пространства.

**Ключевые JVM-флаги:** `-XX:MaxTenuringThreshold=N` — устанавливает порог продвижения.

#### Ответ от Claude

**Что такое продвижение (promotion).** Продвижение — это перемещение живого объекта из Young Generation (молодого поколения) в Old Generation (старое поколение, также называемое Tenured). Generational GC основан на **гипотезе о поколениях**: большинство объектов умирают молодыми. Объекты, которые «выживают» дольше ожидаемого, считаются долгоживущими и продвигаются в Old Gen, где собираются реже (Major GC / Full GC).

**Структура Young Generation:**
- **Eden Space** — все новые объекты аллоцируются здесь (кроме огромных объектов, которые сразу попадают в Old Gen — «humongous allocation» в G1).
- **Survivor Space S0 и S1 (From/To)** — два пространства, между которыми выжившие объекты «пинг-понгятся»; в каждый момент одно из них пустое.

**Механизм возраста объекта (age).** Каждый раз, когда объект переживает Minor GC, его возраст (age) увеличивается на 1. Возраст хранится в заголовке объекта (object header) в 4 битах — поэтому максимальный возраст равен 15.

**Tenuring Threshold** — динамически вычисляемый порог (от 1 до `MaxTenuringThreshold`), при достижении которого объект продвигается. JVM сама понижает текущий порог, если Survivor Space переполнено сверх целевой заполненности (`TargetSurvivorRatio`).

**Когда происходит продвижение:**
1. Возраст объекта достиг `TenuringThreshold` (штатный случай).
2. В Survivor Space нет места для выжившего объекта (forced promotion / «продвижение из-за давления»).
3. Объект слишком большой для Eden/Survivor и аллоцируется сразу в Old Gen.
4. JVM динамически снизила порог ниже возраста объекта.

**Ключевые JVM-флаги:**

| Флаг | Описание | Значение по умолчанию |
|---|---|---|
| `-XX:MaxTenuringThreshold=N` | Максимальный возраст до продвижения (0–15) | 15 |
| `-XX:InitialTenuringThreshold=N` | Начальное значение динамического порога | 7 |
| `-XX:TargetSurvivorRatio=N` | Целевая заполненность Survivor Space (%) при расчёте порога | 50 |
| `-XX:SurvivorRatio=N` | Соотношение Eden к одному Survivor (Eden = N * Survivor) | 8 |
| `-XX:+PrintTenuringDistribution` | Выводить распределение возрастов объектов после каждого Minor GC | false |

**Пример вывода `-XX:+PrintTenuringDistribution`:**
```
Desired survivor size 8388608 bytes, new threshold 6 (max 15)
- age   1:    2000000 bytes,    2000000 total
- age   2:    1500000 bytes,    3500000 total
- age   3:     500000 bytes,    4000000 bytes total
```
Строка `new threshold 6` означает: JVM снизила текущий порог до 6, потому что при пороге 6 Survivor будет заполнен примерно до `TargetSurvivorRatio`%.

**Настройка на практике:** если `-XX:+PrintTenuringDistribution` показывает, что большинство объектов доживают до `MaxTenuringThreshold` и лишь потом продвигаются — они на самом деле долгоживущие, и можно уменьшить `MaxTenuringThreshold` чтобы быстрее продвигать их (уменьшение нагрузки на Minor GC). Если объекты продвигаются досрочно (threshold опускается до 1–2), нужно увеличить Survivor Space (`-XX:SurvivorRatio` или `-XX:NewRatio`) или уменьшить `TargetSurvivorRatio`.

**Примечание по G1 GC.** В G1 те же флаги работают аналогично. ZGC и Shenandoah — это в основном конкурентные сборщики, и классическая схема Eden/Survivor/Tenuring применяется к ним ограниченно (в Generational ZGC, Java 21+, поколения появились, но механизм может отличаться).

### Вопрос 20. Как выбрать подходящий GC-алгоритм (Serial, Parallel, G1, ZGC, Shenandoah) для конкретной задачи? Какие параметры JVM используют для явного выбора сборщика мусора?

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Available Collectors — Java SE 21 GC Tuning Guide (Oracle)](https://docs.oracle.com/en/java/javase/21/gctuning/available-collectors.html)

Oracle описывает четыре основных сборщика для Java 21 и даёт рекомендации по выбору:

**Serial GC** (`-XX:+UseSerialGC`): использует один поток для всей работы GC. Без накладных расходов на межпоточное взаимодействие. Лучше всего подходит для однопроцессорных машин или небольших приложений на мультипроцессорных системах. Когда использовать: небольшие наборы данных (~100 МБ), один процессор, нет требований к паузам.

**Parallel GC** (`-XX:+UseParallelGC`): использует несколько потоков для ускорения GC. Generational-сборщик, аналогичный Serial. Когда использовать: средние и большие наборы данных на многопроцессорном/многопоточном оборудовании; пиковая производительность в приоритете.

**G1 GC** (`-XX:+UseG1GC`): преимущественно конкурентный сборщик. Выполняет дорогостоящую работу параллельно с приложением. Масштабируется от малых до больших машин с большой памятью. Добивается целей по времени паузы с высокой вероятностью. По умолчанию на большинстве систем. Когда использовать: время отклика важнее пропускной способности; паузы GC должны быть короткими.

**ZGC** (`-XX:+UseZGC -XX:+ZGenerational`): полностью конкурентный сборщик. Максимальные паузы менее 1 миллисекунды, независимо от размера кучи. Работает с кучами от сотен МБ до 16 ТБ. Компромисс: меньшая пропускная способность ради ультранизкой задержки. Когда использовать: высокоприоритетные требования ко времени отклика; latency-sensitive приложения.

**Пошаговый подход к выбору от Oracle:**
1. Запустить приложение с выбором GC по умолчанию (VM сам выберет подходящий).
2. Настроить размер кучи для улучшения производительности.
3. Если необходимо — выбрать сборщик по приведённым рекомендациям.
4. Детально настроить размеры кучи и поколений.
5. В крайнем случае — попробовать другой сборщик.

#### Ответ от Claude

**Краткий ориентир по выбору GC.**

| Сборщик | JVM-флаг | Паузы | Пропускная способность | Типичный use case |
|---|---|---|---|---|
| Serial | `-XX:+UseSerialGC` | Высокие (STW) | Низкая | Малые контейнеры, кучи < 100 МБ, 1 ядро |
| Parallel | `-XX:+UseParallelGC` | Средние (STW параллельный) | Максимальная | Batch-обработка, аналитика, ETL |
| G1 | `-XX:+UseG1GC` | Предсказуемые (≤ 200 мс цель) | Хорошая | Большинство веб-сервисов и микросервисов |
| ZGC (Gen) | `-XX:+UseZGC -XX:+ZGenerational` | Ультранизкие (< 1 мс) | Ниже G1 на ~10–15% | Latency-critical сервисы, большие кучи (> 32 ГБ) |
| Shenandoah | `-XX:+UseShenandoahGC` | Низкие (~ 1–5 мс) | Ниже G1 | Альтернатива ZGC, не-Oracle JDK (OpenJDK, RedHat) |

**Serial GC.** Один поток выполняет всю работу GC, приостанавливая все прикладные потоки (Stop-The-World, STW). Смысл использовать только когда накладные расходы на синхронизацию нескольких потоков перевешивают выигрыш — то есть в окружениях с одним CPU, или в крошечных микроконтейнерах (< 200 МБ RAM), или для тестовых/батч-задач с жёстким ограничением ресурсов.

**Parallel GC (Throughput Collector).** Несколько GC-потоков выполняют Minor и Major GC параллельно, но по-прежнему с полной STW-паузой. Цель — максимальная пропускная способность: GC занимает меньше реального времени, хотя каждая пауза всё равно «стоп-мир». Отличный выбор для batch-задач (Spark, Hadoop, ETL), где задержка отдельного запроса не критична, важно сколько данных обработано за час.

**G1 GC.** Куча делится на регионы одинакового размера (1–32 МБ); GC собирает сначала регионы с наибольшим количеством мусора (отсюда Garbage-First). Большая часть работы выполняется конкурентно с приложением; STW-паузы ограничены целевым временем (`-XX:MaxGCPauseMillis=200` по умолчанию). Является сборщиком по умолчанию начиная с Java 9. Хорошо работает для куч от 4 ГБ до ~32 ГБ. Выше ~32 ГБ G1 начинает проигрывать ZGC/Shenandoah из-за роста времени STW-фаз.

**ZGC.** Полностью конкурентный сборщик, включая фазу уплотнения (compaction) кучи. Практически нет STW-пауз: все паузы < 1 мс независимо от размера кучи (16 ТБ максимум). С Java 21 рекомендуется Generational ZGC (`-XX:+UseZGC -XX:+ZGenerational`), который добавил молодое поколение и улучшил пропускную способность. Накладные расходы на CPU: ~8–20% дополнительно. Требует запаса памяти (heap headroom) для работы конкурентных фаз — не стоит устанавливать `-Xmx` вплотную к доступной RAM.

**Shenandoah.** Схож с ZGC по подходу: конкурентное сжатие кучи, паузы в диапазоне единиц миллисекунд. Разработан Red Hat, включён в OpenJDK (доступен в AdoptOpenJDK/Temurin, но исторически отсутствовал в Oracle JDK). Хорошая альтернатива ZGC на JDK < 21 или в не-Oracle дистрибутивах.

**Практические рекомендации по выбору:**

- **Новый веб-сервис, куча 2–16 ГБ** → оставьте G1 по умолчанию, настройте `-XX:MaxGCPauseMillis`.
- **Latency SLA < 10 мс, куча > 8 ГБ, Java 21+** → Generational ZGC.
- **Batch/ETL, максимальный throughput** → Parallel GC.
- **Малый контейнер с 256–512 МБ RAM** → Serial GC или G1 (G1 имеет меньший overhead памяти в плотных средах).
- **Большая куча (> 32 ГБ), нет Java 21** → Shenandoah или ZGC (non-generational).
- **Производительность в тестах** → Epsilon GC (`-XX:+UseEpsilonGC`) — no-op сборщик для устранения влияния GC из benchmark-сценария.

**Ключевые дополнительные флаги:**
- `-XX:MaxGCPauseMillis=N` — целевое максимальное время паузы (G1, ZGC учитывают как hint).
- `-XX:GCTimeRatio=N` — соотношение времени приложения к времени GC (Parallel GC).
- `-Xms` / `-Xmx` — минимальный/максимальный размер кучи.
- `-XX:+PrintGCDetails -XX:+PrintGCDateStamps` (или `-Xlog:gc*` в Java 9+) — подробные GC-логи для диагностики.
- `-XX:+UseStringDeduplication` — дедупликация строк (только G1).

**Важно:** всегда тестируйте выбранный GC на реальной нагрузке с вашими данными — синтетические бенчмарки могут значительно отличаться от prod-поведения.
