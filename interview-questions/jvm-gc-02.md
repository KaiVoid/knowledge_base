### Вопрос 21. Какие области памяти существуют в JVM (heap, stack, Metaspace, code cache и т.д.) и что хранится в каждой из них?

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [The Structure of the Java Virtual Machine — Java SE 8 Edition (docs.oracle.com)](https://docs.oracle.com/javase/specs/jvms/se8/html/jvms-2.html)

Спецификация JVM определяет следующие области памяти («Runtime Data Areas»):

- **PC Register (счётчик команд)** — каждый поток имеет свой счётчик. Содержит адрес выполняемой инструкции JVM; если метод нативный — значение не определено.
- **Java Virtual Machine Stack (стек JVM)** — создаётся для каждого потока в момент его создания. Хранит фреймы (frames): локальные переменные, стек операндов, ссылку на runtime constant pool текущего метода. Аналогичен стеку вызовов в C. При исчерпании — `StackOverflowError`; при невозможности расширить — `OutOfMemoryError`.
- **Heap (куча)** — разделяется между всеми потоками, создаётся при старте JVM. Единственное место, где размещаются экземпляры классов и массивы. Память освобождается сборщиком мусора; не требует непрерывного расположения в физической памяти.
- **Method Area (область методов)** — также разделяется между потоками, создаётся при старте. Хранит структуры каждого загруженного класса: runtime constant pool, данные полей и методов, байткод методов и конструкторов. Логически является частью heap, но реализации не обязаны применять к ней GC.
- **Runtime Constant Pool** — табличное представление `constant_pool` из `.class`-файла; выделяется из области методов. Содержит константы времени компиляции (числовые литералы) и символические ссылки на поля/методы, разрешаемые во время выполнения.
- **Native Method Stack** — аналог JVM-стека, но для нативных методов (написанных не на Java). Обычно выделяется на каждый поток в момент его создания; может иметь фиксированный или динамический размер.

#### Ответ от Claude

Память JVM делится на две категории: **разделяемая между потоками** (heap, method area) и **локальная для каждого потока** (PC register, JVM stack, native method stack). Помимо областей из спецификации, в реальных реализациях HotSpot присутствуют дополнительные регионы.

**Области, определённые спецификацией JVM:**

| Область | Область видимости | Что хранится |
|---|---|---|
| **Heap** | Общая | Все объекты (`new`) и массивы; управляется GC |
| **Method Area** | Общая | Метаданные классов, байткод методов/конструкторов, runtime constant pool, данные полей |
| **Runtime Constant Pool** | Общая (часть Method Area) | Числовые константы, строковые литералы, символические ссылки на поля и методы |
| **JVM Stack** | На поток | Фреймы: локальные переменные (примитивы + ссылки), стек операндов, ссылка на constant pool |
| **PC Register** | На поток | Адрес текущей инструкции JVM (undefined для нативных методов) |
| **Native Method Stack** | На поток | Данные нативных методов (C/C++), вызванных через JNI |

**Дополнительные области в HotSpot JVM:**

- **Young Generation** (подраздел Heap) — новые объекты. Делится на **Eden** (основное место аллокации) и два **Survivor Space** (S0/S1). Сборка — Minor GC, выполняется часто.
- **Old Generation (Tenured)** (подраздел Heap) — объекты, «пережившие» несколько Minor GC. Сборка — Major/Full GC, выполняется редко.
- **Metaspace** — реализация Method Area в HotSpot начиная с Java 8. Размещается в **native memory** (за пределами Java heap). Автоматически расширяется; ограничивается флагами `-XX:MetaspaceSize` и `-XX:MaxMetaspaceSize`. Хранит: описания классов, дескрипторы методов/полей, аннотации, статические переменные ссылочного типа.
- **Code Cache** — кэш JIT-скомпилированного машинного кода. JIT-компилятор (C1/C2) помещает сюда скомпилированные методы («горячие методы»). Управляется флагами `-XX:ReservedCodeCacheSize`. При переполнении JVM выводит предупреждение и может деоптимизировать методы.

**Что где лежит — практические примеры:**
- `new MyObject()` → тело объекта в Heap (Eden)
- `int x = 5` внутри метода → в JVM Stack (фрейм текущего метода)
- `static final String FOO = "bar"` → ссылка в Metaspace, строка в Heap (String Pool)
- байткод метода `doWork()` → в Metaspace (Method Area)
- нативный метод `System.arraycopy()` → использует Native Method Stack

### Вопрос 22. Чем Metaspace отличается от PermGen? Почему в Java 8 было принято решение избавиться от PermGen и что это изменило для приложений?

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Permgen vs Metaspace in Java — Baeldung](https://www.baeldung.com/java-permgen-metaspace)

**Расположение памяти:**
PermGen размещался внутри Java heap. Metaspace использует native memory (память, предоставляемую операционной системой напрямую).

**Управление размером:**
PermGen имел фиксированный максимальный размер: по умолчанию 64 МБ на 32-битной JVM и 82 МБ на 64-битной. Это требовало ручной настройки через флаг `-XX:MaxPermSize`. Metaspace динамически расширяется по умолчанию, занимая столько native memory, сколько доступно в ОС.

**Сборка мусора:**
Метаданные в PermGen освобождались в рамках стандартных GC-пауз. Metaspace позволяет выполнять конкурентное освобождение (concurrent deallocation) метаданных классов без паузы GC.

**Ошибки переполнения:**
PermGen генерировал `java.lang.OutOfMemoryError: PermGen space`. Metaspace генерирует `java.lang.OutOfMemoryError: Metaspace`.

**Что изменилось для приложений:**
Флаги `-XX:PermSize` и `-XX:MaxPermSize` стали устаревшими (выводят предупреждение при использовании в Java 8+). Вместо них следует использовать `-XX:MetaspaceSize` и `-XX:MaxMetaspaceSize`. Мониторинг расширился: теперь необходимо отслеживать native memory, а не только heap.

#### Ответ от Claude

**Краткая история:** PermGen (Permanent Generation) существовал с первых версий HotSpot и являлся фиксированным разделом Java heap для хранения метаданных классов. В Java 8 (2014) он был полностью удалён и заменён Metaspace.

**Сравнительная таблица:**

| Характеристика | PermGen (до Java 8) | Metaspace (Java 8+) |
|---|---|---|
| **Физическое расположение** | Внутри Java heap (contiguous heap memory) | Native memory (off-heap, выделяется ОС) |
| **Максимальный размер по умолчанию** | 64 МБ (32-bit JVM) / 82 МБ (64-bit) | Не ограничен (ограничен только объёмом native RAM) |
| **Управление размером** | `-XX:PermSize`, `-XX:MaxPermSize` (фиксированный предел) | `-XX:MetaspaceSize`, `-XX:MaxMetaspaceSize` (опционально) |
| **Автоматический рост** | Нет | Да, динамически расширяется |
| **Освобождение метаданных** | В рамках Full GC (с остановкой мира) | Конкурентно, без GC-паузы |
| **Ошибка переполнения** | `OutOfMemoryError: PermGen space` | `OutOfMemoryError: Metaspace` |
| **Конкуренция за память** | Конкурировал с объектами за heap | Не конкурирует с heap |

**Почему PermGen был удалён:**

1. **Частый `OutOfMemoryError: PermGen space`** — особенно в приложениях с динамической генерацией классов (JSP-компиляция, CGLib, рефлексия) и при горячем передеплое в серверах приложений. Фиксированный лимит не поддавался автоматической настройке.
2. **Сложность тюнинга** — разработчики вынуждены были угадывать правильный `MaxPermSize` под каждое приложение; слишком мало — OOM, слишком много — впустую занятая heap.
3. **Конкуренция с объектами** — метаданные классов и пользовательские объекты конкурировали за единое heap-пространство.
4. **Влияние на GC** — PermGen обрабатывался в Full GC, увеличивая паузы.
5. **Унификация с JRockit** — Oracle объединял HotSpot и JRockit (в котором PermGen не было), и унификация требовала устранения этого различия.

**Что изменилось для приложений:**

- **Исчезновение классического `OutOfMemoryError: PermGen space`** — теперь приложения могут загружать сколько угодно классов (в пределах доступной native RAM).
- **Новая угроза: утечки native memory** — если приложение постоянно создаёт новые ClassLoader'ы без их выгрузки (типичная ситуация при горячем передеплое в старых версиях Tomcat), native memory будет расти бесконтрольно.
- **Устаревшие JVM-флаги** — `-XX:MaxPermSize` следует убрать из скриптов запуска; в Java 8 они только выводят предупреждение, в более новых версиях могут вызвать ошибку запуска.
- **Новые флаги мониторинга** — для контроля Metaspace используют `jcmd <pid> VM.native_memory` или JVM-метрики в системах мониторинга.
- **Статические переменные** — начиная с Java 8 статические переменные примитивных типов и ссылки хранятся в Metaspace (сами объекты по-прежнему в heap).

### Вопрос 23. Объясните модель делегирования ClassLoader: Bootstrap → Extension (Platform) → Application. Почему она устроена именно так и что произойдёт, если нарушить иерархию?

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ClassLoader in Java — GeeksforGeeks](https://www.geeksforgeeks.org/java/classloader-in-java/)

**Три встроенных ClassLoader:**

- **Bootstrap ClassLoader** — верхушка иерархии, часть самой JVM (реализован в нативном коде). Загружает ядро Java API из `rt.jar` / JRT (Java Runtime Image в Java 9+), включая `java.lang.Object`. Не имеет родителя; при запросе `getClassLoader()` для классов, загруженных им, возвращается `null`.
- **Extension ClassLoader (→ Platform ClassLoader в Java 9+)** — загружает расширения платформы: до Java 9 из `<JAVA_HOME>/jre/lib/ext`, с Java 9 — из модульной системы. Является дочерним по отношению к Bootstrap.
- **Application (System) ClassLoader** — загружает классы из classpath приложения (CLASSPATH, `-cp`, `-classpath`). Является дочерним по отношению к Extension/Platform.

**Механизм делегирования (parent-first):**

1. Application ClassLoader получает запрос на загрузку класса.
2. Делегирует запрос Extension ClassLoader.
3. Extension делегирует Bootstrap ClassLoader.
4. Bootstrap пытается найти класс в своих источниках; если нашёл — загружает.
5. Если не нашёл — управление возвращается Extension, который ищет в своих источниках.
6. Если не нашёл — Application ClassLoader ищет в classpath.
7. Если не нашёл никто — выбрасывается `ClassNotFoundException`.

**Принципы:**
- **Visibility (видимость)** — дочерний ClassLoader видит классы, загруженные родительским, но не наоборот.
- **Uniqueness (уникальность)** — один и тот же класс не загружается дважды одним ClassLoader.

#### Ответ от Claude

**Модель делегирования (Parent Delegation Model)** — архитектурный паттерн, встроенный в `java.lang.ClassLoader` начиная с JDK 1.2. Суть: прежде чем попытаться загрузить класс самостоятельно, ClassLoader всегда сначала делегирует запрос своему родителю (рекурсивно вверх по иерархии).

**Три встроенных уровня:**

```
Bootstrap ClassLoader        ← нативный код, rt.jar / java.* (Java 9+: java.base модуль)
       |
Extension / Platform CL      ← <JAVA_HOME>/jre/lib/ext (Java 8) / platform модули (Java 9+)
       |
Application / System CL      ← CLASSPATH приложения
       |
(пользовательские ClassLoader'ы)
```

**Алгоритм loadClass (реализован в `ClassLoader.loadClass()`):**

```java
protected Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
    // 1. Проверить кэш: не загружен ли класс уже?
    Class<?> c = findLoadedClass(name);
    if (c == null) {
        // 2. Делегировать родителю (или Bootstrap, если parent == null)
        try {
            c = parent != null ? parent.loadClass(name, false) : findBootstrapClass(name);
        } catch (ClassNotFoundException e) {
            // 3. Родитель не нашёл — ищем сами
            c = findClass(name);
        }
    }
    return c;
}
```

**Почему иерархия устроена именно так:**

1. **Безопасность** — нельзя подменить `java.lang.String` или `java.lang.Object` собственной реализацией: Bootstrap всегда загружает эти классы первым. Если злонамеренный jar содержит класс `java.lang.String`, он будет проигнорирован — Bootstrap уже загрузил настоящий.
2. **Уникальность классов** — один класс, загруженный дважды разными ClassLoader'ами, — это **два разных типа** в JVM. Они несовместимы: `instanceof` возвращает `false`, a cast бросает `ClassCastException`. Делегирование вверх гарантирует, что класс загружается ровно один раз.
3. **Стабильность платформы** — серверные и расширительные классы изолированы от пользовательского кода; пользователь не может случайно «затенить» (shadow) системный класс, разместив одноимённый файл в classpath.

**Что произойдёт при нарушении иерархии:**

| Сценарий | Следствие |
|---|---|
| Класс с именем `java.lang.String` помещён в classpath | Bootstrap загрузит оригинальный String; пользовательский файл будет проигнорирован. В Java 9+ SecurityManager дополнительно запрещает классы в пакетах `java.*` из пользовательских загрузчиков |
| Два ClassLoader'а загружают один и тот же бизнес-класс | `ClassCastException` при попытке привести объект, полученный из одного CL, к типу из другого |
| Дочерний CL загружает класс раньше родителя (child-first) | «Затенение» родительских классов; возможны `NoSuchMethodError` или `IncompatibleClassChangeError` при несовпадении версий |
| Циклическая зависимость между ClassLoader'ами | `ClassCircularityError` |

**Java 9+ изменения:** Extension ClassLoader переименован в **Platform ClassLoader**. Модули (`module-info.java`) добавляют ещё один слой контроля: классы из одного модуля видят только то, что явно экспортируется другим модулем, независимо от ClassLoader.

### Вопрос 24. Как реализована загрузка классов в контейнерах (Tomcat, OSGi)? Почему они нарушают стандартную модель делегирования и к каким проблемам это может привести?

**Категория:** JVM и GC · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Apache Tomcat 9 (9.0.x) — Class Loader HOW-TO (tomcat.apache.org)](https://tomcat.apache.org/tomcat-9.0-doc/class-loader-howto.html)

Tomcat создаёт собственную иерархию ClassLoader'ов:

```
Bootstrap (JVM)
    |
  System        ← bootstrap.jar, tomcat-juli.jar
    |
  Common        ← $CATALINA_HOME/lib (Catalina, Jasper, Servlet API)
  /    \
WebApp1  WebApp2  ← /WEB-INF/classes и /WEB-INF/lib/*.jar каждого приложения
```

**Инверсия делегирования в WebappClassLoader:** при загрузке класса из веб-приложения `WebappX ClassLoader` смотрит в **локальные репозитории первым** (WEB-INF/classes, WEB-INF/lib), и только затем делегирует родительским загрузчикам. Это противоположно стандартной модели Java (parent-first).

**Исключения из инверсии:** классы JavaEE API (Servlet, JSP, EL, WebSocket) всегда делегируются родителю первыми — это предотвращает появление нескольких несовместимых версий этих спецификационных классов.

**Включить стандартную модель** можно через атрибут `<Loader delegate="true"/>` в `context.xml` приложения.

#### Ответ от Claude

**Tomcat: модель изоляции веб-приложений**

Tomcat реализует иерархию из нескольких уровней (в базовой конфигурации):

```
Bootstrap (JVM core)
     |
   System (CLASSPATH Tomcat: bootstrap.jar, tomcat-juli.jar)
     |
   Common ($CATALINA_HOME/lib: Servlet API, Catalina, Jasper)
     |
 WebappX (WEB-INF/classes + WEB-INF/lib/*.jar — один на приложение)
```

Ключевое отклонение от стандарта — **child-first (инверсия делегирования)** в `WebappClassLoader`:

- Стандарт Java: сначала parent, потом сам.
- Tomcat WebappClassLoader: сначала сам (WEB-INF), потом parent.

**Зачем нужна инверсия в Tomcat:**
1. **Изоляция приложений** — два задеплоенных приложения могут использовать разные версии одной библиотеки (например, одно — Guava 30, другое — Guava 32), не конфликтуя друг с другом.
2. **Независимость от версий контейнера** — приложение не привязано к версии библиотеки, поставляемой вместе с Tomcat.
3. **Соответствие спецификации Servlet** (раздел 9.7.2) — спецификация требует, чтобы классы приложения имели приоритет над классами контейнера.

**OSGi: гранулярная изоляция на уровне бандлов**

OSGi (Open Service Gateway Initiative) реализует принципиально иную модель — **peer-based (одноранговую)**:

- Каждый бандл (bundle = JAR с метаданными в `MANIFEST.MF`) получает **собственный ClassLoader**.
- Вместо иерархии типа «родитель — ребёнок» используется **таблица маршрутизации пакетов**: `имя_пакета → ClassLoader_бандла, который его экспортирует`.
- При необходимости загрузить класс из пакета `com.example.api` бандл смотрит в свою таблицу и обращается напрямую к ClassLoader'у экспортирующего бандла — **минуя иерархию**.
- Делегирование к родителю происходит **только для `java.*` пакетов** (ядро JDK).

**Зачем OSGi нарушает иерархию:**
1. **Версионирование** — несколько версий одного пакета могут сосуществовать в одном JVM одновременно; бандл указывает точную версию в `Import-Package`.
2. **Динамический жизненный цикл** — бандлы можно загружать, обновлять, останавливать и выгружать без перезапуска JVM.
3. **Инкапсуляция** — не экспортированные пакеты полностью скрыты от других бандлов.

**Проблемы, возникающие из-за нарушения стандартной модели:**

| Проблема | Описание | Причина |
|---|---|---|
| **ClassCastException** | Объект класса `Foo` нельзя привести к типу `Foo` | Один и тот же класс загружен двумя разными ClassLoader'ами: `CL1.Foo != CL2.Foo` даже если байткод идентичен |
| **ClassNotFoundException в OSGi** | Класс не найден, хотя jar присутствует в classpath | Пакет не указан в `Import-Package` в `MANIFEST.MF` бандла |
| **Утечки памяти в Tomcat** | native / heap память не освобождается после анде плоя | `WebappClassLoader` не выгружен: на него держат ссылки ThreadLocal'ы, статические поля, драйверы JDBC, зарегистрированные через `DriverManager` |
| **Конфликты версий** | `NoSuchMethodError`, `IncompatibleClassChangeError` | Одна библиотека загружается разными ClassLoader'ами в разных версиях; при обмене объектами типы оказываются несовместимы |
| **`Class.forName()` нарушения в OSGi** | Динамическая загрузка не работает | `Class.forName()` использует ClassLoader вызывающего класса, который может не видеть нужный пакет |
| **Thread Context ClassLoader** | SPI-механизмы (JDBC, JAXP) не находят реализации | В стандартном Java SPI используется TCCL (Thread Context ClassLoader), который может не совпадать с ClassLoader'ом бандла/webapp |

**Практические рекомендации:**
- В Tomcat: избегать хранения ссылок на `WebappClassLoader` в долгоживущих объектах; явно отменять регистрацию JDBC-драйверов в `ServletContextListener.contextDestroyed()`.
- В OSGi: правильно объявлять `Import-Package` / `Export-Package`; использовать директиву `uses` для транзитивных зависимостей пакетов; избегать `Class.forName()` без явного указания ClassLoader.

### Вопрос 25. В чём разница между ClassNotFoundException и NoClassDefFoundError? Приведите реальный сценарий, при котором возникает каждая из них.

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ClassNotFoundException vs NoClassDefFoundError | Baeldung](https://www.baeldung.com/java-classnotfoundexception-and-noclassdeffounderror)

`ClassNotFoundException` — это проверяемое исключение (`checked exception`), которое возникает, когда приложение пытается загрузить класс по его строковому имени, но определение класса не найдено в classpath. Это происходит при вызовах `Class.forName()`, `ClassLoader.loadClass()` или `ClassLoader.findSystemClass()`. Реальный сценарий: попытка загрузить JDBC-драйвер без добавления нужной зависимости в classpath — `Class.forName("oracle.jdbc.driver.OracleDriver")` выбросит `ClassNotFoundException`.

`NoClassDefFoundError` — это ошибка (`Error`, не `Exception`), которая возникает, когда класс существовал во время компиляции, но JVM не может найти его в classpath во время выполнения. Нередко причиной служит исключение в статическом блоке инициализации: класс скомпилирован успешно, но при попытке загрузки статический блок упал с исключением, и JVM помечает класс как недоступный (`ExceptionInInitializerError`), а последующие попытки использовать его дают `NoClassDefFoundError`. Также возникает, если скомпилировать классы A и B (где B зависит от A), а потом удалить `A.class` — при запуске B появится `NoClassDefFoundError`.

#### Ответ от Claude

**ClassNotFoundException** и **NoClassDefFoundError** — два разных механизма сигнализации о проблемах с загрузкой классов в JVM.

**ClassNotFoundException**

- Тип: `checked exception` (наследует `ReflectiveOperationException → Exception`)
- Иерархия: `Object → Throwable → Exception → ReflectiveOperationException → ClassNotFoundException`
- Возникает, когда **имя класса известно только в runtime** и класс запрашивается динамически через:
  - `Class.forName(String)`
  - `ClassLoader.loadClass(String, boolean)`
  - `ClassLoader.findSystemClass(String)`
- Класс не был найден в classpath **в момент обращения**

Реальный сценарий: динамическая загрузка JDBC-драйвера без нужного JAR в classpath:
```java
// Выбросит ClassNotFoundException, если ojdbc.jar не в classpath
Class.forName("oracle.jdbc.driver.OracleDriver");
```

**NoClassDefFoundError**

- Тип: `Error` (наследует `LinkageError → Error`)
- Иерархия: `Object → Throwable → Error → LinkageError → NoClassDefFoundError`
- Возникает, когда класс **существовал на этапе компиляции**, но при запуске JVM не может найти его определение (`*.class`-файл)
- Также возникает, когда класс был загружен, но его **статический инициализатор выбросил исключение** (`ExceptionInInitializerError`) — последующие попытки обратиться к классу дают `NoClassDefFoundError`

Реальные сценарии:
1. Удалён `SomeClass.class` после компиляции, но код другого класса ссылается на него при выполнении
2. Несовпадение версий JAR-зависимостей: код скомпилирован против версии X, в runtime присутствует версия Y
3. Статический инициализатор упал:
```java
class Broken {
    static int val = Integer.parseInt("not-a-number"); // NumberFormatException
}
// При использовании Broken: NoClassDefFoundError (caused by ExceptionInInitializerError)
```

**Ключевые отличия**

| Характеристика | ClassNotFoundException | NoClassDefFoundError |
|---|---|---|
| Тип | `checked Exception` | `Error` (unchecked) |
| Момент | Runtime, динамическая загрузка | Runtime, но класс был известен компилятору |
| Причина | Класс не найден в classpath при динамическом запросе | Класс был найден при компиляции, но исчез или не инициализировался при запуске |
| Обязательная обработка | Да (`try-catch` или `throws`) | Нет |
| Типичная ситуация | `Class.forName(...)`, плагины, OSGI | Удалённый JAR, сломанный статический блок, рассинхрон версий |

### Вопрос 26. Опишите жизненный цикл объекта в heap: Eden → Survivor S0/S1 → Old Generation. Как работает механизм tenuring и что означает параметр -XX:MaxTenuringThreshold?

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [3 Generations — Java SE 8 GC Tuning Guide (Oracle)](https://docs.oracle.com/javase/8/docs/technotes/guides/vm/gctuning/generations.html)

Молодое поколение (Young Generation) состоит из трёх областей: Eden и двух Survivor-пространств. Большинство объектов изначально размещается в Eden. Одно Survivor-пространство всегда пусто и служит целевой областью для живых объектов из Eden при Minor GC; второе — источником при следующей сборке. Объекты копируются между Survivor-пространствами до тех пор, пока не станут «достаточно старыми» для переноса (tenured) в поколение Tenured/Old. Порог переноса (tenuring threshold) JVM выбирает динамически на каждой сборке — так, чтобы Survivor-пространство оставалось примерно наполовину заполненным. Слабая гипотеза о поколениях гласит: большинство объектов живут недолго, поэтому Minor GC работает быстро, а Major GC — дорогостоящая операция на всём heap.

#### Ответ от Claude

**Жизненный цикл объекта в heap**

1. **Eden** — новый объект создаётся здесь (`new`). Большинство объектов умирают именно в Eden, не переходя дальше.
2. **Minor GC (Young Collection)** — срабатывает при заполнении Eden. Живые объекты из Eden и из активного Survivor-пространства копируются в **пустое** Survivor-пространство. Мёртвые объекты просто игнорируются (их память освобождается сразу). Survivor-пространства меняются ролями: бывшее «to» становится «from», бывшее «from» становится «to» (пустым).
3. **Survivor S0/S1** — два равных по размеру пространства. В каждый момент одно из них активно («from»), другое — пусто («to»). При каждом Minor GC объекты копируются из «from» в «to», одновременно увеличивается их **возраст (age)** — счётчик в заголовке объекта.
4. **Tenuring** — когда возраст объекта достигает tenuring threshold, объект переносится в **Old Generation** (Tenured). Old Generation собирается значительно реже — только при Major GC или Full GC.

**Механизм tenuring**

Каждый объект имеет в заголовке (object header) поле возраста — 4 бита (начиная с Java 5u6, когда biased locking уменьшил это поле с 5 до 4 бит). Поэтому максимальный возраст — **15**.

JVM вычисляет **динамический** tenuring threshold при каждой Minor GC: он выбирается таким образом, чтобы Survivor-пространство оставалось примерно наполовину заполненным. Если объект накопил возраст, превысивший этот порог — он переводится в Old Generation.

**Параметр `-XX:MaxTenuringThreshold`**

Задаёт **максимальный возраст**, до которого объект может дожить в Survivor-пространствах. Это верхняя граница для динамического порога:
- Диапазон допустимых значений: **0–15**
- Значение по умолчанию: **15** для Parallel GC, **6** для CMS
- При `MaxTenuringThreshold=0` все выжившие объекты сразу переносятся в Old Generation, минуя Survivor-пространства

Значение >15 (до Java 1.5.0_05 можно было ставить до 31) начиная с Java 1.5.0_06 интерпретируется JVM как «никогда не переносить» — объекты остаются в Survivor навсегда, что приводит к его переполнению и деградации.

**Практические настройки**

```bash
# Типичные флаги для диагностики
-XX:+PrintTenuringDistribution   # Вывод распределения возрастов при каждой GC
-XX:MaxTenuringThreshold=10      # Перенос в Old раньше (если объекты явно долгоживущие)
-XX:SurvivorRatio=8              # Размер Survivor = Eden / 8 (по умолчанию)
```

**Если Survivor переполняется** до достижения порога, объекты переносятся в Old немедленно — это называется *premature promotion* и ведёт к более частым Major GC.

### Вопрос 27. Что такое Card Table и Write Barrier? Для чего они нужны в генерационных сборщиках и как влияют на производительность приложения?

**Категория:** JVM и GC · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [JVM Anatomy Quark #13: Intergenerational Barriers (Aleksey Shipilёv)](https://shipilev.net/jvm/anatomy-quarks/13-intergenerational-barriers/)

В генерационных сборщиках существует проблема: при сборке только молодого поколения необходимо знать, какие объекты старого поколения ссылаются на объекты молодого — иначе «живые» объекты могут быть ошибочно собраны. Полное сканирование Old Generation при каждой Minor GC слишком дорого. Решение — **Card Table**: большой массив байт, где каждый байт («карта») соответствует диапазону адресов в heap (типично 512 байт). При записи ссылки из объекта старого поколения в поле объекта (молодого или любого) выполняется **Write Barrier** — небольшой кусок кода, который «загрязняет» (dirty) соответствующий байт в card table. При Minor GC сборщик сканирует только «грязные» карты для поиска inter-generational ссылок. Барьер выполняется безусловно при каждой записи ссылки. Бенчмарк автора (обход HashMap из 1М элементов): Epsilon GC — 19 683 мкс, Parallel GC — 25 224 мкс. Hardware counters показали ~3 дополнительных store-операции за итерацию, ~22 лишних цикла ЦП и ~12 дополнительных инструкций, плюс повышенное число L1 cache miss из-за «загрязнения» card table. Вывод: write barrier влияет на производительность даже когда GC в данный момент не работает.

#### Ответ от Claude

**Проблема, которую решают Card Table и Write Barrier**

В генерационных сборщиках heap разбит на поколения (как минимум Young и Old). При Minor GC собирается только Young Generation, что позволяет работать быстро. Однако возникает задача: объект в Old Generation может хранить ссылку на объект в Young Generation. Если при Minor GC не учесть такие «inter-generational» ссылки, Young-объект будет ошибочно признан мёртвым и уничтожен — катастрофа для корректности программы.

Наивное решение — сканировать весь Old Generation при каждой Minor GC — слишком медленно. Вместо этого используется структура **Remembered Set** (в HotSpot реализованная через **Card Table**).

**Card Table**

Card Table — большой массив байт. Каждый байт («карта») покрывает фиксированный диапазон адресов в heap. В HotSpot это **512 байт heap на 1 байт card table**. Карта помечается как «грязная» (dirty), когда в соответствующей области heap появляется потенциальная inter-generational ссылка. При Minor GC сборщик сканирует только «грязные» карты вместо всего Old Generation.

**Write Barrier**

Write Barrier — это небольшой кусок кода, который JIT-компилятор вставляет **после каждой записи ссылки** в поле объекта. В простейшем виде (пост-барьер для card marking):

```java
// Логика Write Barrier при выполнении: obj.field = ref
CARD_TABLE[address_of(obj) >> 9] = 0;   // dirty card (0 = dirty in HotSpot)
```

Барьер выполняется **безусловно** при каждой записи ссылки — без проверки, является ли ссылка inter-generational. Это упрощает реализацию и позволяет избежать дорогого условного перехода в горячем коде, хотя приводит к тому, что в «грязных» картах оказываются и intra-generational записи.

**Влияние на производительность**

По измерениям Aleksey Shipilёv (обход 1М-элементного HashMap):
- Epsilon GC (без барьеров): ~19 684 мкс
- Parallel GC (с write barrier): ~25 224 мкс — замедление на ~28%

Источники накладных расходов:
1. **Дополнительные store-инструкции** (~3 на итерацию цикла)
2. **Лишние циклы CPU** (~22 на итерацию)
3. **L1 cache miss** — card table «загрязняет» кэш-линии, даже если GC в данный момент не работает

**Важный вывод**: накладные расходы write barrier присутствуют постоянно, независимо от того, происходит ли сборка мусора в данный момент. Именно поэтому существуют «barrier-free» коллекторы (например, Epsilon GC) для специализированных workload.

**Write Barrier в разных коллекторах**

| Коллектор | Тип барьера | Назначение |
|---|---|---|
| Parallel GC, G1 (post-write) | Card-marking write barrier | Поддержка card table для inter-generational ссылок |
| G1 (pre-write, SATB) | Snapshot-At-The-Beginning барьер | Корректность concurrent marking |
| Shenandoah | Read + Write barrier | Поддержка to-space invariant при concurrent evacuation |
| ZGC | Load barrier | Обеспечение корректности при concurrent relocation |

### Вопрос 28. Объясните, как работает G1 GC: что такое регионы, Remembered Set, Concurrent Marking Cycle и evacuation pause. В чём отличие G1 от Parallel GC?

**Категория:** JVM и GC · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [9 Garbage-First Garbage Collector — Java SE 8 GC Tuning Guide (Oracle)](https://docs.oracle.com/javase/8/docs/technotes/guides/vm/gctuning/g1_gc.html)

G1 разбивает heap на равные по размеру **регионы** — непрерывные диапазоны виртуальной памяти. Регион является единицей и выделения памяти, и рекламации. Регионы бывают Eden, Survivor, Old и Humongous (для объектов размером > половины региона). G1 выполняет **конкурентную глобальную маркировку** (concurrent global marking) для определения живых объектов по всему heap. После маркировки G1 знает, какие регионы содержат больше всего мусора, и собирает их первыми — отсюда название «Garbage-First». Идентифицированные регионы собираются посредством **эвакуации** (evacuation): живые объекты копируются из одного или нескольких регионов в один новый регион, что одновременно выполняет и уплотнение (compaction), и освобождение памяти. Эвакуация выполняется параллельно на многопроцессорных системах. G1 использует **модель предсказания пауз**: пользователь задаёт целевое время паузы (`-XX:MaxGCPauseMillis`, по умолчанию 200 мс), и G1 на основе этой цели выбирает количество регионов для коллекции. В отличие от Parallel Compaction GC, выполняющего уплотнение только для всего heap сразу (что даёт значительные паузы), G1 выполняет рекламацию пространства инкрементально, шагами и параллельно.

#### Ответ от Claude

**Регионы (Regions)**

G1 разбивает heap на **равные по размеру регионы** (степени двойки от 1 до 32 МБ, целевое количество ~2048 регионов). Каждый регион — непрерывный диапазон виртуальной памяти. Регион может быть:
- **Eden** — для новых аллокаций
- **Survivor** — для выживших при Young GC
- **Old** — для долгоживущих объектов
- **Humongous** — для объектов размером > 50% региона (занимают один или несколько смежных регионов)

Логически heap по-прежнему делится на молодое и старое поколения, но физически регионы молодого и старого поколения могут быть разбросаны по всему heap — в отличие от традиционных коллекторов с фиксированными непрерывными областями.

**Remembered Set (RSet)**

Поскольку G1 может собирать один регион независимо от других, ему нужно знать, какие объекты из **других регионов** ссылаются на объекты в данном регионе. Для этого каждый регион имеет собственный **Remembered Set** — структуру данных, отслеживающую «входящие» cross-region ссылки.

Механизм работы:
1. При записи cross-region ссылки приложением выполняется **post-write barrier**, который регистрирует запись в буфере
2. **Concurrent Refinement threads** обрабатывают буфер в фоне и обновляют RSet соответствующих регионов
3. При эвакуации региона G1 сканирует его RSet, чтобы найти все внешние ссылки и обновить их после перемещения объектов

Overhead: RSet требует памяти (~20% heap в худшем случае при небольшом размере регионов). Увеличение размера региона (`-XX:G1HeapRegionSize`) уменьшает количество cross-region ссылок и снижает размер RSet.

**Concurrent Marking Cycle**

Запускается автоматически при достижении порога заполнения heap (по умолчанию 45%, параметр `-XX:InitiatingHeapOccupancyPercent`). Цикл маркировки состоит из нескольких фаз:

1. **Initial Mark** (STW) — начало маркировки root-объектов; совмещается с обычной Young GC паузой
2. **Root Region Scan** (concurrent) — сканирование Survivor-регионов из Initial Mark для поиска ссылок в Old Generation
3. **Concurrent Mark** (concurrent) — обход всего heap и маркировка живых объектов; использует алгоритм **SATB (Snapshot-At-The-Beginning)**. Все объекты, живые на момент начала маркировки, считаются живыми; объекты, созданные в процессе, также помечаются
4. **Remark** (STW) — финализация маркировки; обработка SATB-буферов, reference processing, выгрузка классов
5. **Cleanup** (частично STW, частично concurrent) — освобождение полностью пустых регионов, подготовка кандидатов для Space-Reclamation фазы

**Evacuation Pause (STW)**

Это основная «stop-the-world» фаза G1. Делится на два типа:

- **Young-only collection pause** — собирает только молодые регионы (Eden + Survivor)
- **Mixed collection pause** — собирает молодые + некоторые Old-регионы с наибольшим количеством мусора

Процесс эвакуации:
1. G1 определяет **Collection Set (CSet)** — набор регионов для сборки
2. Живые объекты копируются из CSet-регионов в новые регионы (уплотнение «по ходу»)
3. После эвакуации исходные регионы становятся полностью свободными

G1 использует **модель предсказания пауз** (основана на декающих средних по историческим данным) для подбора CSet такого размера, чтобы уложиться в `-XX:MaxGCPauseMillis` (по умолчанию 200 мс).

**Отличия G1 от Parallel GC**

| Характеристика | Parallel GC | G1 GC |
|---|---|---|
| Структура heap | Непрерывные области (Young/Old) | Равные регионы произвольного назначения |
| Уплотнение | Только полное, для всего heap | Инкрементальное, регион за регионом |
| Паузы Old Gen | Длинные Full GC с полным compaction | Короткие Mixed GC (incremental) |
| Управление паузами | Не управляется | Целевое время паузы (`-XX:MaxGCPauseMillis`) |
| Конкурентная работа | Минимальная | Concurrent Marking, Concurrent Refinement |
| Накладные расходы | Ниже (меньше concurrent работы) | Выше (RSet, barьеры, фоновые потоки) |
| Применимость | Batch-задачи, максимальный throughput | Latency-чувствительные приложения, heap ≥ 6 ГБ |

**Когда G1 хуже Parallel GC**: при малом heap (<4 ГБ) и batch-нагрузке накладные расходы на RSet и concurrent-потоки снижают throughput. Parallel GC в таких случаях предпочтителен.

**Параметры G1**

```bash
-XX:+UseG1GC                          # Включить G1 (по умолчанию с Java 9)
-XX:MaxGCPauseMillis=200              # Целевое время паузы (мс)
-XX:InitiatingHeapOccupancyPercent=45 # Порог запуска Concurrent Marking
-XX:G1HeapRegionSize=<n>              # Размер региона (1–32 МБ, степень 2)
-XX:ConcGCThreads=<n>                 # Число потоков concurrent marking
```
### Вопрос 29. Как ZGC достигает пауз менее 1 мс независимо от размера heap? Объясните роль цветных указателей (colored pointers) и барьеров нагрузки (load barriers).

**Категория:** JVM и GC · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Key Techniques of ZGC Garbage Collector — allpcb.com/allelectrohub](https://www.allpcb.com/allelectrohub/key-techniques-of-zgc-garbage-collector)

ZGC кодирует метаданные прямо внутри 64-битного указателя. Структура указателя делится следующим образом:

- **Младшие 42 бита** — адрес объекта (поддерживает heap до 4 ТБ).
- **4 бита метаданных** — флаги: Finalizable (F), Remapped (R), Marked1 (M1), Marked0 (M0).
- **18 старших бит** — зарезервированы.

Указатель считается «хорошим» (good), когда ровно один метабит выставлен, а остальные сброшены (например, 0100, 0010, 0001). Это позволяет потокам проверять состояние объекта без дополнительного обращения к памяти.

Вместо write barriers ZGC использует **load barriers (read barriers)**: при каждом чтении ссылки на объект из heap компилятор вставляет лёгкую проверку. Если цвет указателя «хороший» — fast path, доступ идёт мгновенно. Если цвет «плохой» (объект перемещён) — slow path: барьер консультируется с таблицей пересылки (forwarding table), обновляет указатель на месте («self-healing») и возвращает корректный адрес.

Для работы с перемещёнными объектами ZGC применяет **multi-view virtual address mapping**: одна и та же физическая память отображается в три виртуальных диапазона — по одному для каждого состояния метабита (Marked0 view: 4–8 ТБ, Marked1 view: 8–12 ТБ, Remapped view: 16–20 ТБ). Смена активного представления позволяет ZGC производить маркировку и перемещение конкурентно с потоками приложения.

Весь цикл GC содержит лишь три коротких STW-паузы: Initial Mark, Remark и Initial Relocate — каждая устанавливает лишь булев флаг состояния. Все тяжёлые фазы (маркировка, перемещение, перепривязка ссылок, выгрузка классов) выполняются конкурентно. В итоге паузы не зависят от размера heap и составляют менее 1 мс.

#### Ответ от Claude

**Цель ZGC** — обеспечить паузы менее 1 мс вне зависимости от размера heap (поддерживается от 8 МБ до 16 ТБ). Это достигается за счёт трёх взаимосвязанных механизмов: цветных указателей, барьеров нагрузки и виртуального многозонального отображения памяти.

---

**1. Цветные указатели (Colored Pointers)**

ZGC использует 64-битные указатели, в которых часть битов отведена под метаданные GC:

| Биты | Назначение |
|------|-----------|
| 63–46 | Зарезервированы (18 бит) |
| 45 | Finalizable — объект доступен только через финализатор |
| 44 | Remapped — ссылка не ведёт в relocation set, т.е. актуальна |
| 43 | Marked1 — объект помечен живым в нечётном цикле |
| 42 | Marked0 — объект помечен живым в чётном цикле |
| 41–0 | Адрес объекта (42 бита → до 4 ТБ heap) |

Два отдельных бита для маркировки (Marked0/Marked1) нужны потому, что при следующем цикле маркировки ZGC попутно перепривязывает все указатели предыдущего состояния к Remapped — и может освободить forwarding tables без дополнительного STW-прохода.

Указатель с ровно одним активным метабитом называется «хорошим» (good pointer). Это позволяет проверять состояние объекта без отдельной таблицы метаданных и без дополнительного обращения к памяти.

---

**2. Барьеры нагрузки (Load Barriers)**

Барьер нагрузки — это код, который JIT-компилятор автоматически вставляет при каждом чтении ссылки на объект из heap (но не примитивов и не стекового значения). Логика:

- **Fast path**: цвет указателя соответствует ожидаемому («good») — барьер завершается почти мгновенно.
- **Slow path**: цвет «плохой» — объект был перемещён в текущем цикле релокации. Барьер:
  1. Обращается к **forwarding table** для получения нового адреса.
  2. При необходимости самостоятельно перемещает объект (если ещё не перемещён конкурентным потоком GC).
  3. Обновляет указатель «на месте» (**self-healing**), чтобы следующий доступ шёл по fast path.

Это принципиальное отличие от G1 и CMS, которые использовали write barriers: ZGC не требует STW для перепривязки ссылок — она происходит лениво при каждом чтении.

---

**3. Виртуальное многозональное отображение (Multi-Mapping)**

Поскольку x86 не поддерживает аппаратное маскирование указателей (в отличие от некоторых ARM), ZGC отображает одну и ту же физическую память в три разных диапазона виртуальных адресов:

- `[0, 4 ТБ)` — Java heap (фактические адреса объектов)
- `[4 ТБ, 8 ТБ)` — Marked0 view
- `[8 ТБ, 12 ТБ)` — Marked1 view
- `[16 ТБ, 20 ТБ)` — Remapped view

Переключение активного представления позволяет потокам GC и потокам приложения работать с объектами конкурентно: они «смотрят» на одну и ту же память через разные виртуальные линзы.

---

**4. Фазы цикла ZGC и STW-паузы**

| Фаза | Тип | Описание |
|------|-----|----------|
| Pause Mark Start | STW < 1 мс | Установка флагов, обход корней |
| Concurrent Mark | Конкурентно | Трассировка живых объектов |
| Pause Mark End | STW < 1 мс | Завершение маркировки |
| Concurrent Relocation Prepare | Конкурентно | Выбор relocation set |
| Pause Relocate Start | STW < 1 мс | Перемещение корневых ссылок |
| Concurrent Relocate | Конкурентно | Копирование объектов, обновление ссылок через барьеры |

Три STW-паузы выполняют лишь минимальные действия (установка флагов, сканирование корней). Вся трудоёмкая работа — конкурентна, поэтому время паузы практически не зависит от размера heap и стабильно укладывается в 1 мс.

**Сноска об ограничениях**: цветные указатели требуют 64-битной платформы — ZGC недоступен на 32-битных JVM. До Java 21 поддерживались heap до 4 ТБ (42 бита адреса); в Generational ZGC (JEP 439, Java 21+) архитектура расширена для поддержки молодого и старого поколений, сохраняя тот же принцип работы.

### Вопрос 30. Какие виды OutOfMemoryError существуют в JVM (heap space, Metaspace, GC overhead limit exceeded, Direct buffer memory, unable to create new native thread)? Назовите причину и способ диагностики для каждого.

**Категория:** JVM и GC · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Understand the OutOfMemoryError Exception — docs.oracle.com](https://docs.oracle.com/javase/8/docs/technotes/guides/troubleshoot/memleaks002.html)

Официальная документация Oracle описывает следующие разновидности `OutOfMemoryError`:

**1. Java heap space** — heap исчерпан, GC не может освободить достаточно памяти. Причины: неправильно настроенный размер heap, непреднамеренные долгоживущие ссылки (утечки памяти), чрезмерное использование финализаторов. Диагностика: анализ heap dump, мониторинг очереди финализируемых объектов.

**2. GC Overhead Limit Exceeded** — JVM тратит более 98% времени на сборку мусора, восстанавливая менее 2% heap в течение 5+ последовательных сборок. Диагностика: увеличить размер heap, флаг `-XX:-UseGCOverheadLimit` отключает проверку.

**3. Metaspace** — исчерпана нативная память, отведённая под метаданные классов. Диагностика: увеличить `-XX:MaxMetaspaceSize`, уменьшить Java heap для освобождения нативной памяти.

**4. Requested Array Size Exceeds VM Limit** — попытка выделить массив размером больше, чем позволяет реализация VM.

**5. Out of swap space** — сбой выделения нативной памяти при нехватке swap.

**6. Compressed class space** — исчерпано пространство для сжатых указателей классов (`UseCompressedClassPointers`). Диагностика: увеличить `-XX:CompressedClassSpaceSize`.

**7. Native method allocation failure** — ошибка выделения памяти в JNI-методе.

#### Ответ от Claude

`OutOfMemoryError` — это `Error` (не `Exception`), возникающий когда JVM не может выделить память в той или иной области. Различные виды OOME требуют разных подходов к диагностике.

---

**1. `Java heap space`**

**Что происходит:** JVM не может выделить новый объект в heap — ни в Eden, ни в Old Generation, даже после полного GC.

**Причины:**
- Утечка памяти: долгоживущие объекты (коллекции, кэши без вытеснения, статические поля) удерживают ссылки.
- Недостаточный `-Xmx`.
- Разовый пик нагрузки (например, загрузка огромного файла целиком в память).
- Чрезмерное использование `finalize()`: объекты попадают в очередь финализатора и не могут быть собраны сразу.

**Диагностика:**
- `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/dump.hprof` — автоматический dump при OOM.
- Анализ dump в Eclipse MAT или VisualVM: ищите доминирующих потребителей (Dominator Tree), аномально растущие коллекции.
- Мониторинг: JMX, Micrometer/Prometheus — график `jvm.memory.used` для Old Generation.
- `jcmd <pid> GC.heap_info` — оперативный снимок состояния heap.

---

**2. `Metaspace`**

**Что происходит:** Исчерпана нативная память, используемая для хранения метаданных классов (bytecode, описания методов, constant pool). С Java 8 заменила PermGen.

**Причины:**
- Утечка ClassLoader (особенно в application server при hot-reload): каждый перезапуск webapp создаёт новый ClassLoader с теми же классами, старый не GC-ится.
- Динамическая генерация классов (CGLIB, ASM, Byte Buddy, Groovy, Kotlin lambdas через `invokedynamic`).
- Не установлен `-XX:MaxMetaspaceSize` → metaspace растёт неограниченно до исчерпания нативной памяти процесса.

**Диагностика:**
- `-XX:+PrintClassHistogramBeforeFullGC` или `jcmd <pid> GC.class_histogram` — список классов по количеству экземпляров.
- Heap dump + MAT: Class Loader Explorer → ищите несколько экземпляров одного ClassLoader.
- `-Xlog:class+load=info` (JVM unified logging) — трассировка загрузки классов.
- Native Memory Tracking: `-XX:NativeMemoryTracking=detail`, затем `jcmd <pid> VM.native_memory`.

---

**3. `GC overhead limit exceeded`**

**Что происходит:** JVM обнаруживает, что тратит более 98% процессорного времени на GC, но при этом освобождает менее 2% heap — более 5 последовательных сборок. Это защитный механизм, предотвращающий «тихое зависание».

**Причины:**
- Объём живых данных вплотную приближается к `-Xmx`.
- Масштабные утечки памяти.
- Загрузка в heap слишком большого датасета.

**Диагностика:**
- Анализ GC-логов: `-Xlog:gc*` — соотношение времени GC к времени работы приложения.
- GCEasy, GCViewer — визуальный анализ GC-пауз и throughput.
- Heap dump для выявления живых объектов, удерживающих память.
- Временное отключение проверки: `-XX:-UseGCOverheadLimit` (не рекомендуется в production, только для диагностики).

---

**4. `Direct buffer memory`**

**Что происходит:** Исчерпана off-heap память, выделяемая через `ByteBuffer.allocateDirect()` или `FileChannel.map()`. Лимит контролируется параметром `-XX:MaxDirectMemorySize` (по умолчанию равен `-Xmx`).

**Причины:**
- NIO-интенсивные приложения (Netty, Kafka, gRPC) не освобождают буферы вовремя.
- `DirectByteBuffer` освобождается не явно, а через `sun.misc.Cleaner` при GC — если GC редкий, буферы накапливаются.
- Отсутствие явных вызовов `((DirectBuffer) buf).cleaner().clean()` или `sun.nio.ch.DirectBuffer.cleaner()`.

**Диагностика:**
- `jcmd <pid> VM.native_memory` с `-XX:NativeMemoryTracking=detail`.
- JMX: `java.nio:type=BufferPool,name=direct` — атрибуты `MemoryUsed`, `Count`.
- `-Djdk.nio.maxCachedBufferSize=0` — отключает кэширование прямых буферов в JDK (помогает в диагностике).
- Native Memory Tracking или async-profiler с режимом `-e malloc`.

---

**5. `unable to create new native thread`**

**Что происходит:** JVM не может создать новый поток ОС. Каждый Java-поток требует нативного потока ОС со стеком (`-Xss`, по умолчанию 512 КБ–1 МБ).

**Причины:**
- Достигнут лимит потоков процесса: `ulimit -u` (Linux) или `kernel.threads-max`.
- Достигнут лимит файловых дескрипторов: каждый поток требует FD.
- Недостаточно нативной памяти для стека потока: `(общая память) - (-Xmx) - (-XX:MaxMetaspaceSize) - (прочее) < Xss × число потоков`.
- Утечка потоков: потоки создаются, но не завершаются (нет `ExecutorService.shutdown()`).

**Диагностика:**
- `cat /proc/<pid>/status | grep Threads` — текущее число потоков процесса.
- `ulimit -a` — проверка лимитов.
- Thread dump: `jcmd <pid> Thread.print` или `kill -3 <pid>` — анализ стека всех потоков.
- JMX: `java.lang:type=Threading` — `ThreadCount`, `PeakThreadCount`, `TotalStartedThreadCount`.
- Инструменты: VisualVM, JProfiler — граф создания потоков.

---

**Сводная таблица**

| Тип OOME | Область памяти | Ключевой признак | Первый шаг диагностики |
|----------|---------------|-----------------|----------------------|
| Java heap space | Java heap (-Xmx) | Рост Old Gen до предела | Heap dump + MAT |
| Metaspace | Native (class metadata) | Рост числа ClassLoader-ов | `GC.class_histogram` |
| GC overhead limit | Java heap | GC > 98% CPU | GC-лог + GCEasy |
| Direct buffer memory | Off-heap (NIO) | Рост `direct` BufferPool | JMX `java.nio:type=BufferPool` |
| Unable to create native thread | Native (thread stacks) | ThreadCount → лимит ОС | Thread dump + `ulimit` |

### Вопрос 31. Что такое утечка ClassLoader? Как воспроизвести такую утечку и как обнаружить её с помощью heap dump?

**Категория:** JVM и GC · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Classloader leaks I – How to find classloader leaks with Eclipse Memory Analyser (MAT) — java.jiderhamn.se](https://java.jiderhamn.se/2011/12/11/classloader-leaks-i-how-to-find-classloader-leaks-with-eclipse-memory-analyser-mat/)

**Что такое утечка ClassLoader.** Статья объясняет: «Старые, неиспользуемые ClassLoader-ы не собираются GC, потому что на них есть ссылка снаружи самого ClassLoader-а». Это означает, что хотя веб-приложение или плагин уже «выгружен», его ClassLoader (и все загруженные им классы) продолжают занимать PermGen / Metaspace.

**Автоматический heap dump.** Автор рекомендует добавить параметр `-XX:+HeapDumpOnOutOfMemoryError`, чтобы JVM автоматически создавала dump при OOM. Это устраняет неопределённость: dump снимается в момент реальной проблемы.

**Обнаружение через Eclipse MAT.**
1. Загрузить файл `.hprof` в Eclipse Memory Analyzer.
2. В меню выбрать **Open Query Browser → Java Basics → Class Loader Explorer**.
3. Список ClassLoader-ов в dump покажет все экземпляры — в том числе те, которые должны были быть собраны GC.
4. Правой кнопкой на подозрительном ClassLoader-е → **Path To GC Roots** (исключая weak/soft/phantom-ссылки) — цепочка сильных ссылок укажет точное место утечки.

Типичные виновники: ThreadLocal-переменные в пуле потоков контейнера, глобальные кэши, JMX MBean-ы, статические поля классов уровня приложения, сохраняющие ссылки на объекты плагина.

#### Ответ от Claude

**Суть утечки ClassLoader**

ClassLoader в Java — это объект, который загружает байт-код и является «владельцем» всех загруженных им классов. Между ClassLoader и его классами существует двусторонняя зависимость: класс хранит ссылку на свой ClassLoader, а ClassLoader хранит ссылки на все загруженные классы.

Ключевое следствие: **ссылка на любой объект любого класса удерживает в памяти весь ClassLoader и весь набор загруженных им классов**.

Утечка ClassLoader происходит, когда:
1. Создан пользовательский (custom) ClassLoader — например, при деплое webapp в Tomcat каждый веб-приложение получает `WebAppClassLoader`.
2. Приложение выгружается (undeploy/hot-reload), но ClassLoader не может быть собран GC — на него или на объекты его классов сохранились **сильные ссылки** из «внешнего» кода (из ClassLoader-а контейнера или JVM).
3. Metaspace постепенно заполняется метаданными классов, пока не возникает `OutOfMemoryError: Metaspace`.

---

**Как воспроизвести утечку (минимальный пример)**

```java
// Утечка через static field в классе контейнерного уровня
public class LeakyRegistry {
    // Статическое поле в классе, загруженном системным ClassLoader-ом
    static final List<Object> registry = new ArrayList<>();

    public static void registerService(Object service) {
        registry.add(service); // service загружен webapp ClassLoader-ом
    }
}

// В коде webapp:
// LeakyRegistry.registerService(new MyService()); 
// При undeploy: MyService → держит WebAppClassLoader → не GC-ится
```

Другой классический паттерн — **ThreadLocal в пуле потоков контейнера**:

```java
// В webapp (загружен WebAppClassLoader):
private static final ThreadLocal<MyObject> tl = new ThreadLocal<>();

// В сервлете:
tl.set(new MyObject());
// Если не вызван tl.remove() до завершения запроса:
// Thread (из пула Tomcat) → ThreadLocalMap → Entry.value = MyObject
//   → MyObject.getClass().getClassLoader() = WebAppClassLoader → УТЕЧКА
```

При каждом hot-reload создаётся новый `WebAppClassLoader`, но предыдущий не освобождается. Metaspace растёт «ступенчато» после каждого редеплоя.

**Другие распространённые причины:**
- JDBC-драйвер, зарегистрированный через `DriverManager` из webapp ClassLoader-а, не отменённый при `contextDestroyed`.
- JMX MBean-ы, зарегистрированные из webapp без отмены регистрации.
- Логбэки (Logback, Log4j) с аппендерами, содержащими ссылки на классы webapp.
- CGLIB/Byte Buddy прокси, сохранённые в статических кэшах уровня контейнера.

---

### Вопрос 32. Что такое safepoint в JVM? Почему GC-паузы на практике могут быть длиннее, чем кажется по логам сборщика?

**Категория:** JVM и GC · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [JVM Pauses — It's more than GC — blanco.io](https://blanco.io/blog/jvm-safepoint-pauses/)

**Что такое safepoint.** Статья определяет: «Safepoint — это точка во время выполнения программы, в которой известны все GC-корни и содержимое всех heap-объектов согласованно». В точке safepoint все потоки должны приостановиться, чтобы JVM могла выполнить критические операции: GC не является единственным поводом для safepoint.

**Почему GC-логи не отражают реальную длину паузы.** Фактическое время остановки состоит из двух фаз:
1. **TTSP (Time to Reach Safepoint)** — время ожидания, пока все потоки достигнут точки safepoint.
2. **Время операции** — выполнение самой VM-операции (GC, отзыв biased lock и т.д.).

GC-логи обычно фиксируют только время фазы 2. В реальном примере из статьи GC-логи показывали паузы ~24 секунды, тогда как данные safepoint статистики обнаружили паузы более 5 минут — критически важное расхождение для latency-sensitive приложений.

**Причины длинного TTSP.** Потоки медленно достигают safepoint при: инициализации больших объектов (zeroing массивов), операциях копирования массивов, выделении JNI handle-ов, критических JNI-регионах, операциях с memory-mapped файлами через NIO.

#### Ответ от Claude

**Что такое safepoint**

Safepoint — это точка в выполнении потока, где состояние потока полностью известно JVM: все стековые ссылки находятся в картированных позициях, регистры либо содержат известные значения, либо сохранены. В точке safepoint JVM может безопасно исследовать или изменять heap и стеки потоков.

**Важно**: safepoint — это не просто «пауза для GC». Многие VM-операции требуют глобального safepoint.

---

**Операции, требующие safepoint (STW)**

| Операция | Описание |
|----------|----------|
| Stop-the-world GC фазы | Initial Mark, Remark, Full GC и т.д. |
| Снятие biased lock (RevokeBias) | При конкуренции за biased lock |
| Деоптимизация JIT-кода (Deoptimize) | После инвалидации assumptions |
| Перекомпиляция кода | При изменении профиля выполнения |
| Class redefinition | Инструментирование через JVMTI/javaagent |
| Heap dump | `jmap`, `jcmd GC.heap_dump` |
| Thread dump | `jcmd Thread.print` |
| Code cache flush | Очистка кэша JIT |
| Monitor inflation | При конкуренции за `synchronized`-блок |

---

**Механизм достижения safepoint**

JVM не останавливает потоки принудительно. Вместо этого:

1. JVM устанавливает глобальный флаг «запрошен safepoint».
2. В сгенерированном JIT-кодe в ключевых точках (back-edge цикла, вызов метода) вставлены **safepoint polls** — проверки этого флага.
3. Поток, обнаружив флаг, сам паркуется на safepoint-барьере.
4. JVM ждёт, пока **все** потоки достигнут safepoint. Это время называется **TTSP (Time To Safe Point)**.

---

**Почему реальная пауза длиннее, чем в GC-логах**

GC-логи сообщают только время выполнения самой GC-фазы (фазу **vmop**). Полная пауза приложения = **TTSP + vmop + resume_time**.

Пример из safepoint-лога:
```
         vmop                    [threads: total initial_wait final_wait] [time]
RevokeBias                        [    48     0      3500  ]  [5023 ms]
```
Здесь `final_wait=3500 мс` (spin/block время ожидания safepoint) + `5023 мс` (сама операция) = **более 8 секунд** полной паузы. GC-лог бы показал 0 — ведь GC не запускался.

---

**Конкретные причины длинного TTSP**

**1. Counted loops (счётные циклы)**

Наиболее частая причина. JIT-компилятор считает цикл с `int`-счётчиком «counted loop» и **не вставляет safepoint poll в тело цикла** — для оптимизации. Если такой цикл выполняется долго (миллисекунды), все остальные потоки ждут его завершения.

```java
// Этот цикл НЕ содержит safepoint poll — JIT оптимизирует как counted loop
for (int i = 0; i < Integer.MAX_VALUE; i++) {
    data[i] = compute(i); // может работать секунды без safepoint
}
```

**Решение:** `-XX:+UseCountedLoopSafepoints -XX:LoopStripMiningIter=1000` — включает «strip mining» (разбивку цикла на фрагменты с safepoint-ами). В G1, ZGC и Shenandoah включено по умолчанию.

**2. Перегрузка CPU (CPU over-subscription)**

Если активных потоков больше, чем ядер CPU, потоки, которые должны добежать до safepoint, могут быть вытеснены планировщиком ОС. JVM ждёт их «пробуждения».

**3. Page faults при memory-mapped I/O**

Поток, работающий с `MappedByteBuffer`, может встретить отсутствующую страницу памяти. ОС приостанавливает поток на время подкачки страницы (major page fault) — safepoint недостижим, пока страница не загружена.

**4. JNI critical regions**

Код в `GetPrimitiveArrayCritical()` / `GetStringCritical()` не может быть прерван до вызова `Release*`. JVM ждёт выхода из критической секции.

---

**Как диагностировать**

**Включить полное логирование пауз (Java 11+):**
```bash
-Xlog:safepoint*=info:file=/tmp/safepoint.log:time,uptime,level,tags
-Xlog:gc*=info:file=/tmp/gc.log:time,uptime,level,tags
```

**Для старых JVM (Java 8):**
```bash
-XX:+PrintGCApplicationStoppedTime
-XX:+PrintSafepointStatistics
-XX:PrintSafepointStatisticsCount=1
```

Лог `PrintGCApplicationStoppedTime` покажет **полное** время остановки:
```
Total time for which application threads were stopped: 46.7196326 seconds,
Stopping threads took: 46.6686622 seconds
```
Здесь 46,6 секунды — это чистый TTSP, а vmop занял лишь ~50 мс.

**Найти проблемный поток:**
```bash
-XX:+SafepointTimeout -XX:SafepointTimeoutDelay=500
```
При превышении 500 мс JVM напечатает стек потока, не достигшего safepoint.

**async-profiler** с режимом `wall-clock` (`-e wall`) показывает, где потоки «застревают» вне safepoint-опросов.

---

**Ключевые выводы**

1. GC-лог показывает время **только GC-операции**, не включая TTSP.
2. Реальная пауза приложения = TTSP + время операции. TTSP может доминировать.
3. Не-GC операции (RevokeBias, Deoptimize, heap dump) также вызывают STW-паузы, невидимые в GC-логах.
4. Counted loops без safepoint polls — наиболее частая причина аномально долгого TTSP.
5. Для полной картины необходимо включать safepoint-логирование отдельно от GC-логирования.
### Вопрос 33. Как работает JIT-компилятор: что такое tiered compilation, C1 и C2? Когда метод компилируется, а когда происходит деоптимизация (deoptimization) и почему?

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [How Tiered Compilation works in OpenJDK — Microsoft for Java Developers](https://devblogs.microsoft.com/java/how-tiered-compilation-works-in-openjdk/)

OpenJDK HotSpot использует два JIT-компилятора совместно с интерпретатором, реализуя пятиуровневую систему (тиры 0–4):

- **Уровень 0 — интерпретатор**: начальное выполнение, самый дешёвый по накладным расходам, но медленный.
- **Уровень 1 — C1 без профилирования**: быстрая компиляция для «тёплых» методов без сбора данных профиля.
- **Уровень 2 — C1 с базовым профилированием**: C1 собирает счётчики вызовов и итераций циклов.
- **Уровень 3 — C1 с полным профилированием**: C1 собирает исчерпывающие метрики для передачи в C2.
- **Уровень 4 — C2**: максимальная долгосрочная производительность с использованием накопленных профильных данных; профилирование прекращается.

**C1 (Client Compiler)** компилирует быстро, но генерирует менее оптимальный код — приоритет отдаётся быстрому старту. **C2 (Server Compiler)** компилирует дольше, но производительность итогового кода на 30%+ выше.

**Когда метод компилируется.** Интерпретатор ведёт для каждого метода счётчик вызовов и счётчик итераций циклов (back-edge counter). При превышении порогового значения метод ставится в очередь на компиляцию. Сама компиляция выполняется в отдельном потоке, не прерывая выполнение метода. Если C1-компилятор занят, JVM может направить метод сразу на C2.

**Деоптимизация.** C2 делает агрессивные оптимистичные предположения о поведении кода (например, «ветка X никогда не будет взята» или «тип всегда Y»). Для проверки этих предположений в скомпилированный код вставляются предикаты (traps). Если предикат срабатывает (предположение нарушено), поток останавливает выполнение скомпилированного метода в текущей точке и возобновляет его в интерпретаторе с того же места. Затем JVM перепрофилирует метод и при необходимости перекомпилирует его с учётом новых данных.

#### Ответ от Claude

**Архитектура JIT в HotSpot**

HotSpot VM содержит два JIT-компилятора:

- **C1 (Client Compiler)** — быстрый компилятор первого уровня оптимизации. Выполняет простые оптимизации (инлайнинг небольших методов, устранение мёртвого кода) и основную цель — минимизировать время компиляции, обеспечив приемлемое ускорение относительно интерпретатора. Генерирует код с инструментацией для сбора профиля.
- **C2 (Server Compiler / Opto)** — оптимизирующий компилятор, работающий медленнее, но генерирующий высококачественный машинный код. Использует профиль, собранный C1, для спекулятивных оптимизаций: агрессивный инлайнинг, loop unrolling, vectorization, устранение проверок на null, девиртуализация вызовов и т.д.

**Пять уровней Tiered Compilation (включена по умолчанию с Java 8)**

| Уровень | Компонент | Описание |
|---------|-----------|----------|
| 0 | Интерпретатор | Старт выполнения |
| 1 | C1, без профиля | Быстрая компиляция, нет overhead профилирования |
| 2 | C1, базовый профиль | Счётчики вызовов и back-edge |
| 3 | C1, полный профиль | Добавляются branch profiles, type profiles |
| 4 | C2 | Максимальная оптимизация, профилирование выключено |

Типичный путь метода: 0 → 3 → 4 (тёплые) или 0 → 2 → 3 → 4 (с задержкой C2). Простые/редко вызываемые методы могут остаться на уровне 1 или 2.

**Когда компилируется метод**

Интерпретатор ведёт два счётчика: invocation counter (сколько раз метод вызывался) и backedge counter (сколько раз выполнялась обратная ветка цикла). При превышении порогов (например, ~2000 вызовов для C1 и ~10000–15000 для C2, точные значения зависят от версии JVM и могут быть изменены флагами `-XX:CompileThreshold`, `-XX:Tier3InvocationThreshold` и т.д.) метод ставится в очередь компиляции. Компилятор работает в фоновом потоке, не блокируя выполнение.

**Деоптимизация (Deoptimization)**

Деоптимизация — это механизм отката скомпилированного метода обратно в интерпретатор. Происходит, когда оптимистичные предположения C2 нарушаются в рантайме:

1. **Нарушение type profile** — C2 девиртуализировал вызов, предположив, что тип всегда `A`. Появляется объект типа `B` — предположение нарушено.
2. **Редкая ветка** — C2 не скомпилировал ветку, считавшуюся мёртвой (`uncommon_trap`). При её первом исполнении — деоптимизация.
3. **Загрузка нового класса** — нарушает инлайнинг или CHA-предположения (Class Hierarchy Analysis).
4. **Null pointer / ArrayIndexOutOfBounds** в специфических позициях, где C2 убрал проверку.

После деоптимизации метод может быть перекомпилирован с учётом новых данных профиля (recompilation).

**Ключевые флаги для диагностики:**
- `-XX:+PrintCompilation` — логировать компиляции
- `-XX:+TraceDeoptimization` — логировать деоптимизации
- `-XX:-TieredCompilation` — отключить tiered compilation (использовать только C2)

### Вопрос 34. Что такое escape analysis в JIT-компиляторе? Как он позволяет размещать объекты на стеке вместо heap и в каких случаях эта оптимизация не срабатывает?

**Категория:** JVM и GC · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Escape Analysis — DZone](https://dzone.com/articles/escape-analysis)

Escape Analysis (EA) — это техника JIT-компилятора, определяющая, может ли вновь созданный объект «вытечь» за пределы метода или потока, в котором он был создан. Если объект не покидает свой контекст, открывается ряд оптимизаций.

**Три состояния escape-объекта:**
- **NoEscape** — объект невидим за пределами текущего метода/потока; максимум оптимизаций.
- **ArgEscape** — объект передаётся как аргумент, но не виден за пределами вызова.
- **GlobalEscape** — объект возвращается из метода или присваивается статическому полю; оптимизации невозможны.

**Scalar Replacement (скалярная замена):** вопреки распространённому заблуждению, EA в Java (до Project Valhalla) не выполняет «настоящее» размещение на стеке. Вместо этого компилятор «распаковывает» поля объекта в отдельные переменные на стеке или регистры CPU. Аллокация объекта на heap полностью устраняется.

**Lock Elision (устранение блокировок):** если объект синхронизации классифицирован как NoEscape или ArgEscape, другие потоки не могут его захватить — блокировки полностью удаляются из скомпилированного кода. Например, `synchronized(new Object())` или операции с `StringBuffer`, созданным локально.

EA включён по умолчанию в C2; отключить можно флагом `-XX:-DoEscapeAnalysis`.

#### Ответ от Claude

**Что такое Escape Analysis**

Escape Analysis (EA) — это анализ указателей, выполняемый C2-компилятором (и частично C1 в режиме профилирования) для каждого объекта, созданного внутри компилируемого метода. Цель — определить, может ли ссылка на объект стать видимой вне метода или вне текущего потока.

**Классификация объектов:**

| Статус | Описание | Оптимизации |
|--------|----------|-------------|
| NoEscape | Ссылка не покидает метод и не видна другим потокам | Scalar replacement, lock elision, stack allocation |
| ArgEscape | Ссылка передаётся как аргумент, но вызываемый метод её не «роняет» | Lock elision |
| GlobalEscape | Ссылка возвращается, сохраняется в поле/static — видна снаружи | Нет оптимизаций |

**Оптимизации, которые открывает EA:**

1. **Scalar Replacement** — основная оптимизация в HotSpot. Объект «разбирается» на составляющие поля, которые помещаются в стековые слоты или регистры. Heap-аллокация полностью устраняется, GC не видит такой объект вовсе. Пример: локальный `Point(x, y)` превращается в две int-переменные.

2. **Stack Allocation** — в теории возможна полная аллокация объекта на стеке потока (thread-local), но в HotSpot реализована через scalar replacement, а не через прямую stack-аллокацию (Project Valhalla/Value Types планирует расширить это).

3. **Lock Elision** — если объект-монитор имеет статус NoEscape, JVM доказывает, что другие потоки никогда не войдут в этот `synchronized`-блок, и удаляет инструкции захвата/освобождения монитора.

**Когда EA не срабатывает:**

- **Метод не инлайнится** — EA работает в контексте скомпилированного блока. Если вызов уходит в не-инлайненный метод, компилятор не может отследить судьбу аргумента → объект считается ArgEscape или GlobalEscape.
- **Объект слишком большой** — EA имеет бюджет анализа; очень большие объектные графы анализируются консервативно.
- **Присвоение в поле другого объекта** — `someField.ref = localObj` — объект сразу GlobalEscape.
- **Возврат из метода** — `return localObj` — GlobalEscape.
- **Хранение в статическом поле** — `STATIC_FIELD = localObj` — GlobalEscape.
- **Элементы массива** — присвоение `arr[i] = obj` часто мешает EA, так как анализ алиасинга массивов консервативен.
- **Autoboxing в generic-контексте** — боксинг примитивов, переданных в generic-метод, нередко создаёт escaping-объекты.
- **Метод слишком большой** — C2 не инлайнит методы крупнее определённого порога, что ограничивает область EA.
- **Виртуальный вызов** — если тип получателя неизвестен, инлайнинг невозможен, EA теряет видимость.

**Диагностика:**
```
-XX:+PrintEscapeAnalysis          # вывод результатов EA
-XX:+PrintEliminateAllocations    # вывод устранённых аллокаций
-XX:-DoEscapeAnalysis             # отключить EA
```

**Важный нюанс:** EA — это JIT-оптимизация, работающая только на горячем коде (уровень 4 / C2). Код, не достигший порога компиляции, не получает этих оптимизаций.

### Вопрос 35. Что гарантирует ключевое слово volatile с точки зрения Java Memory Model? Почему volatile недостаточно для атомарности составных операций типа i++?

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Java Volatile and Atomicity — Stack a Byte](https://stackabyte.com/tutorials/Java/java-volatile-keyword-and-atomicity)

`volatile` даёт две гарантии:

**Видимость (Visibility):** любое чтение volatile-переменной вернёт последнее записанное значение. Запись немедленно становится видимой всем потокам.

**Запрет переупорядочивания (Reordering Prevention):** `volatile` действует как memory barrier — компилятор и CPU не могут переставить инструкции через точку обращения к volatile-переменной.

**Почему volatile не обеспечивает атомарность `i++`:**

Операция `i++` состоит из трёх отдельных шагов:
1. **Read** — прочитать текущее значение `i`
2. **Modify** — вычислить `i + 1`
3. **Write** — записать новое значение

Пример гонки: поток 1 читает `counter = 5`, поток 2 читает `counter = 5` (видит последнее значение благодаря volatile), оба увеличивают до 6, оба записывают 6. Итог: 6 вместо ожидаемого 7.

`volatile` гарантирует видимость каждой отдельной операции чтения/записи, но не атомарность их последовательности. Для безопасного инкремента следует использовать `AtomicInteger.incrementAndGet()`.

#### Ответ от Claude

**Гарантии `volatile` в Java Memory Model (JSR-133)**

Начиная с Java 5 (JSR-133), `volatile` даёт три взаимосвязанных гарантии:

**1. Видимость (Visibility)**
Запись в volatile-переменную happens-before (видна) каждому последующему чтению этой же переменной в любом потоке. Компилятор и JVM не имеют права кешировать значение в регистре — каждое чтение обращается к «свежей» памяти, каждая запись немедленно публикуется.

Из JLS 17.4.5: «A write to a volatile field happens-before every subsequent read of that field.»

**2. Предотвращение переупорядочивания (Ordering / Memory Barrier)**
Volatile-операция — это полный memory fence:
- **Volatile write** — все записи, выполненные до него (в порядке программы), будут видны после него. Эффект аналогичен `monitor release`.
- **Volatile read** — все чтения, выполненные после него, увидят актуальные значения. Эффект аналогичен `monitor acquire`.

Это значит, что если поток A записал несколько обычных полей, а затем volatile-флаг, то поток B, прочитавший этот флаг, гарантированно увидит все предшествующие записи потока A.

**3. Атомарность одиночных чтений/записей**
Чтение и запись volatile-переменных являются атомарными даже для `long` и `double` (по умолчанию операции с 64-битными типами могут быть неатомарными на 32-битных платформах).

**Почему volatile недостаточно для `i++`**

`i++` — составная операция, в байткоде раскрывающаяся как:
```
GETFIELD i        // 1. Прочитать значение
ICONST_1          // 2. Подготовить константу 1
IADD              // 3. Сложить
PUTFIELD i        // 4. Записать результат
```

Шаги 1–4 не являются единой атомарной операцией. Между шагами 1 и 4 другой поток может завершить свой цикл 1–4, записав промежуточное значение. volatile гарантирует, что каждый отдельный GETFIELD и PUTFIELD увидит актуальное значение, но **не исключает гонку** между ними.

**Решения:**
- `AtomicInteger.incrementAndGet()` — использует CAS (Compare-And-Swap), атомарная операция уровня CPU.
- `synchronized` блок — взаимное исключение гарантирует, что только один поток выполняет инкремент одновременно.
- `LongAdder` — высокопроизводительный счётчик для сценариев с высокой конкуренцией.

**Отличие от `synchronized`**

| Свойство | `volatile` | `synchronized` |
|----------|-----------|----------------|
| Видимость | Да | Да |
| Запрет переупорядочивания | Да | Да |
| Взаимное исключение (mutex) | Нет | Да |
| Атомарность составных операций | Нет | Да |
| Overhead | Минимальный | Выше (монитор) |

`volatile` — это инструмент для однопотоковой публикации состояния при одном производителе и множестве потребителей. Когда несколько потоков пишут — нужен `synchronized` или `Atomic`-классы.

### Вопрос 36. Объясните отношение happens-before в Java Memory Model. Какие действия устанавливают это отношение помимо volatile и synchronized?

**Категория:** JVM и GC · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [JSR 133 (Java Memory Model) FAQ — University of Maryland](https://www.cs.umd.edu/~pugh/java/memoryModel/jsr-133-faq.html)

Документ JSR-133 FAQ определяет happens-before следующим образом: если действие A happens-before действия B, то A **видимо и упорядочено** перед B. Это не означает, что A физически произошло раньше — это гарантия видимости и порядка с точки зрения памяти.

**Правила, устанавливающие happens-before (из JSR-133 FAQ):**

1. **Порядок программы (Program Order):** каждое действие в потоке happens-before каждого последующего действия в том же потоке.
2. **Мониторные блокировки:** unlock монитора happens-before каждого последующего lock того же монитора.
3. **Volatile-поля:** запись в volatile-поле happens-before каждого последующего чтения того же поля.
4. **Thread.start():** вызов `start()` на потоке happens-before любого действия в запущенном потоке.
5. **Thread.join():** все действия в потоке T happens-before возврата из `T.join()` в другом потоке.

Дополнительно упоминаются **final-поля:** корректная инициализация final-поля в конструкторе создаёт гарантии видимости — другие потоки видят правильно инициализированные final-поля без дополнительной синхронизации, если объект не «вытек» из конструктора.

#### Ответ от Claude

**Что такое happens-before**

Happens-before (hb) — это частичный порядок на множестве действий в Java-программе, определённый в JSR-133 (Java Memory Model, Java 5+). Формально: если `hb(A, B)`, то **все эффекты действия A видны и упорядочены перед B**. Ключевое: это не утверждение о физическом времени выполнения, а гарантия видимости памяти.

Если между записью W и чтением R отсутствует happens-before отношение — это **data race**, и программа может наблюдать непредсказуемые результаты.

**Полный список правил happens-before (JLS 17.4.5)**

**1. Program Order Rule**
Если действия X и Y находятся в одном потоке и X предшествует Y в порядке программы, то `hb(X, Y)`. Внутри одного потока порядок всегда сохраняется (хотя CPU/компилятор могут переупорядочивать инструкции, эффект для текущего потока неотличим от последовательного).

**2. Monitor Lock Rule**
Unlock монитора M `synchronizes-with` каждым последующим lock того же M. Это означает: все записи, сделанные в synchronized-блоке, видны потоку, который войдёт в synchronized-блок на том же мониторе позже.

**3. Volatile Variable Rule**
Запись в volatile-переменную V `synchronizes-with` каждым последующим чтением V. Дополнительно: все записи, предшествующие volatile write (по Program Order), становятся видимыми после volatile read — это «пигги-бэкинг» на volatile.

**4. Thread Start Rule**
Вызов `thread.start()` `synchronizes-with` первым действием в запущенном потоке. Всё, что поток-родитель делал до `start()`, видно дочернему потоку при его старте.

**5. Thread Join Rule**
Все действия потока T `synchronizes-with` успешным возвратом из `T.join()` в вызывающем потоке. Всё, что сделал поток T до завершения, видно потоку, вернувшемуся из `join()`.

**6. Thread Interruption Rule**
Если поток T1 прерывает поток T2 (`T1.interrupt(T2)`), это `synchronizes-with` моментом, когда T2 (или другой поток) обнаруживает прерывание через `Thread.interrupted()`, `Thread.isInterrupted()` или `InterruptedException`.

**7. Default Value Initialization Rule**
Запись значений по умолчанию (0, false, null) в поля объекта при его создании `synchronizes-with` первым действием каждого потока. Гарантирует, что ни один поток не увидит «мусор» вместо дефолтных значений.

**8. Constructor-to-Finalizer Rule**
Завершение конструктора объекта happens-before запуска финализатора (`finalize()`) этого же объекта.

**9. Транзитивность**
Если `hb(X, Y)` и `hb(Y, Z)`, то `hb(X, Z)`. Это позволяет строить цепочки синхронизации.

**Дополнительно: final-поля (JLS 17.5)**
Корректно опубликованные final-поля дают специальные гарантии: если объект не «вытек» из конструктора (this-reference escape), то все потоки видят правильно инициализированные final-поля без какой-либо дополнительной синхронизации. Это исключение — final-поля защищены механизмом, отличным от стандартного happens-before.

**Практические следствия**

```java
// Пример piggy-backing на volatile:
int a = 0;
volatile int b = 0;

// Поток 1:
a = 42;        // обычная запись
b = 1;         // volatile write

// Поток 2:
if (b == 1) {  // volatile read
    // Здесь гарантированно виден a == 42
    // потому что: hb(a=42, b=1) по Program Order
    //             hb(b=1, b==1) по Volatile Rule
    //             hb(a=42, чтение a) по транзитивности
}
```

**Что НЕ создаёт happens-before:**
- `Thread.sleep()` — явно указано в JLS, что sleep не имеет семантики синхронизации.
- `Thread.yield()` — аналогично.
- Чтение/запись не-volatile полей без синхронизации.
- Вызов `System.out.println()` — несмотря на то что внутри `synchronized`, это деталь реализации, не гарантированная спецификацией.
### Вопрос 37. Может ли инструкция записи в память быть переупорядочена компилятором или процессором? Как JMM описывает допустимые переупорядочения и как final-поля защищают от них при публикации объекта?

**Категория:** JVM и GC · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [JSR 133 (Java Memory Model) FAQ — cs.umd.edu](https://www.cs.umd.edu/~pugh/java/memoryModel/jsr-133-faq.html)

Да, и компилятор, и процессор могут переупорядочивать инструкции. Компилятор может решить, что перемещение операции записи на более поздний момент более эффективно. Процессоры также выполняют инструкции не в порядке программы, а данные перемещаются между регистрами, кэшами и основной памятью непредсказуемым образом.

Новая модель памяти (JMM, Java 5+, JSR-133) вводит частичный порядок *happens-before*, который ограничивает допустимые переупорядочения:
- каждое действие в потоке предшествует последующим действиям этого же потока в порядке программы;
- снятие блокировки монитора предшествует последующему захвату того же монитора;
- запись в volatile-поле предшествует последующему чтению этого же поля;
- вызов `thread.start()` предшествует действиям в запущенном потоке;
- все действия потока предшествуют успешному возврату из `thread.join()`.

Относительно `final`-полей: старая модель никак не выделяла `final`-поля. JSR-133 исправил это — теперь после завершения конструктора все `final`-поля «заморожены», и любой другой поток, получивший ссылку на объект, гарантированно увидит корректно инициализированные значения этих полей. Ключевое условие — ссылка на объект не должна «утекать» во время конструктора. При этом для публикации самой ссылки на объект синхронизация всё равно необходима.

#### Ответ от Claude

**Да, переупорядочения реальны и происходят на двух уровнях:**

1. **Компилятор (JIT):** оптимизирует порядок инструкций в рамках «as-if-serial» семантики — то есть результат в *одном* потоке не меняется, но другие потоки могут увидеть промежуточное состояние.
2. **Процессор:** современные CPU выполняют инструкции out-of-order, а многоуровневая иерархия кэшей (L1/L2/L3) и store-буферы создают ситуацию, когда запись, выполненная одним ядром, не сразу видна другому.

**Что JMM говорит о допустимых переупорядочениях**

JMM (JSR-133, Java 5+) не запрещает переупорядочения — он описывает, при каких условиях результаты действий одного потока *гарантированно* видны другому. Инструмент для этого — отношение **happens-before**:

| Правило | Суть |
|---|---|
| Program Order | Действия в одном потоке упорядочены между собой |
| Monitor Lock | `unlock(m)` happens-before последующего `lock(m)` |
| Volatile Variable | Запись в `volatile` happens-before последующего чтения этого же поля |
| Thread Start | `thread.start()` happens-before любого действия в запущенном потоке |
| Thread Join | Все действия потока happens-before возврата из `join()` |
| Transitivity | Если A hb B и B hb C, то A hb C |

Всё, что не подпадает под эти правила, может быть переупорядочено произвольным образом — компилятором или процессором. Например, два независимых store-а в разные поля могут поменяться местами.

На уровне реализации JMM задаётся через **барьеры памяти** (memory barriers / fences). JSR-133 Cookbook определяет четыре типа:

- `LoadLoad` — запрет на переупорядочение двух нагрузок;
- `StoreStore` — запрет переупорядочения двух записей;
- `LoadStore` — запрет перестановки загрузки и последующей записи;
- `StoreLoad` — самый «дорогой» барьер, запрет переупорядочения записи и последующей загрузки; на x86 реализуется инструкцией `mfence` или `lock add`.

**Как final-поля защищают при публикации объекта**

До JSR-133 `final` не давало никаких гарантий видимости. JSR-133 добавил два правила для `final`-полей:

1. **Freeze action (заморозка):** запись в `final`-поле внутри конструктора не может быть переупорядочена с последующей записью, которая публикует ссылку на объект (`StoreStore`-барьер перед выходом из конструктора). Это означает, что любой поток, получивший ссылку на объект *после* завершения его конструктора, гарантированно увидит корректные значения всех `final`-полей.

2. **Initial load ordering:** первая загрузка `final`-поля не может быть переупорядочена с первой загрузкой ссылки на содержащий его объект.

Пример проблемы, которую решают `final`-поля:

```java
class Config {
    final int port;
    final String host;
    Config(int port, String host) {
        this.port = port;  // запись в final
        this.host = host;  // запись в final
    }
    // StoreStore-барьер вставляется здесь JIT-компилятором
}

// Публикация через volatile — гарантирует видимость ссылки
volatile Config config = new Config(8080, "localhost");
```

Без `final` другой поток мог бы увидеть `config != null`, но `port == 0` или `host == null` из-за переупорядочения в конструкторе.

**Важное ограничение:** `final` гарантирует видимость *значений полей*, но не гарантирует видимость *самой ссылки* на объект для другого потока. Для безопасной публикации ссылки всё равно требуется `volatile`, синхронизированный блок или другие механизмы happens-before.

**Escape во время конструктора** нарушает гарантии: если `this` утечёт до завершения конструктора (например, через регистрацию в глобальном реестре), другой поток может увидеть частично инициализированный объект даже с `final`-полями.

### Вопрос 38. Как выбрать между G1, ZGC и Shenandoah для конкретного приложения? Какие метрики и характеристики нагрузки (throughput, latency, heap size) определяют этот выбор?

**Категория:** JVM и GC · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Java Garbage Collectors: G1 vs ZGC vs Shenandoah — toolshelf.tech](https://toolshelf.tech/blog/java-garbage-collectors-g1-vs-zgc-vs-shenandoah/)

Выбор сборщика мусора определяется тремя ключевыми характеристиками: **throughput** (процент времени, когда JVM выполняет прикладной код, а не GC), **latency** (p99/p999 задержки, особенно важные в микросервисной архитектуре) и **heap size**.

Рекомендации по размеру кучи:
- G1: оптимален для 4–16 ГБ, плохо масштабируется после 32 ГБ;
- ZGC: от 10 ГБ до 10 ТБ — масштабируется исключительно хорошо;
- Shenandoah: более эффективен на кучах малого и среднего размера.

Типичные паузы:
- G1: 200–500 мс;
- ZGC: менее 1 мс, независимо от размера кучи;
- Shenandoah: менее 1 мс.

Рекомендации по типу нагрузки:
- **Batch-обработка**: G1 или Parallel GC (приоритет throughput);
- **REST API / микросервисы с SLA < 50 мс**: ZGC (JDK 21+) или Shenandoah;
- **Кучи < 2 ГБ**: G1 достаточен; накладные расходы конкурентных сборщиков перевешивают выгоду.

Начинать следует с G1 по умолчанию, переходить на ZGC или Shenandoah только если паузы GC — реальное узкое место по данным мониторинга.

#### Ответ от Claude

**Исходная точка: G1 — умолчание по умолчанию**

G1 является сборщиком по умолчанию начиная с Java 9. Он подходит для большинства приложений и требует минимальной настройки. Переходить на ZGC или Shenandoah стоит только тогда, когда измерения (JDK Flight Recorder, GC logs, APM) показывают, что именно GC-паузы нарушают SLA.

**Сравнительная таблица характеристик**

| Характеристика | G1 | ZGC (Generational, JDK 21+) | Shenandoah |
|---|---|---|---|
| Паузы STW | 200–500 мс (типично) | < 1 мс, не зависят от heap | < 10 мс, обычно < 5 мс |
| Throughput | ~95–98% | ~85–93% | ~90–95% |
| Оптимальный размер кучи | 4–32 ГБ | 8 ГБ — 16 ТБ | 2 ГБ — 10 ТБ |
| Накладные расходы памяти | умеренные | +15–25% к heap | умеренные |
| Накладные расходы CPU | умеренные | +5–10% | умеренные |
| Поддержка compressed oops | да | нет (на больших heaps) | да |
| Доступность | OpenJDK / Oracle JDK | OpenJDK / Oracle JDK | OpenJDK (Red Hat origin) |
| Минимальная версия Java | 9 | 15 (продакшн), gen. mode — 21 | 12 (experimental), 15 (продакшн) |

**Ключевые метрики для выбора**

1. **Pause time (p99/p999)** — самая важная метрика. Снимается через GC-логи (`-Xlog:gc*`) или JFR. Если p99 > целевого SLA — рассматривайте ZGC или Shenandoah.

2. **Throughput** — процент времени, потраченного на прикладной код vs GC. Для batch-задач, ETL, аналитики throughput важнее latency — здесь G1 или Parallel GC выигрывают у конкурентных сборщиков.

3. **Heap size и footprint** — ZGC требует больше памяти (~25% overhead); в контейнерах с жёсткими лимитами это может привести к OOMKill. Shenandoah лучше работает на кучах до 64 ГБ.

4. **Allocation rate** — высокая скорость аллокации (> 1 ГБ/с) хуже переносится G1 (частые young GC); ZGC и Shenandoah справляются лучше за счёт конкурентного уплотнения.

5. **Object lifetime distribution** — если большинство объектов долгоживущие, G1 страдает от fragmentation в Old Gen; ZGC со своим конкурентным compaction решает эту проблему.

**Алгоритм выбора**

```
Heap < 2 ГБ  → G1 (concurrent GC overhead не окупается)
Heap 2–32 ГБ, SLA на паузы > 100 мс → G1
Heap 2–32 ГБ, SLA на паузы < 50 мс  → Shenandoah или ZGC
Heap > 32 ГБ, паузы G1 неприемлемы  → ZGC (generational, JDK 21+)
Batch/ETL, throughput > latency      → G1 или Parallel GC
Нужна максимальная стабильность пауз независимо от heap → ZGC
Red Hat / non-Oracle OpenJDK         → Shenandoah хорошо поддержан
```

**Детали по каждому сборщику**

*G1 (Garbage-First):*
- Разбивает heap на регионы (~2048 регионов), собирает регионы с наибольшим количеством мусора первыми.
- Ориентируется на цель по паузам (`-XX:MaxGCPauseMillis=200`), но это цель, не гарантия.
- Mixed GC (сбор Young + часть Old) — ключевой механизм, но при большом heap размер Old Generation всё равно сказывается на длительности Full GC.
- Флаг: `-XX:+UseG1GC` (умолчание с Java 9).

*ZGC (Z Garbage Collector):*
- Полностью конкурентный: все фазы, кроме трёх коротких STW (инициализация, корневой снимок, финальная отметка), выполняются параллельно с приложением.
- Использует colored pointers (окрашенные указатели) и load barriers для конкурентного relocate объектов.
- С JDK 21 — Generational ZGC (`-XX:+ZGenerational`) значительно улучшил throughput, сохранив sub-ms паузы.
- Требует больше RAM (~25%); в контейнерах проверяйте heap + off-heap.
- Флаг: `-XX:+UseZGC -XX:+ZGenerational` (Java 21+).

*Shenandoah:*
- Аналогично ZGC — конкурентный компактинг, но использует Brooks forwarding pointers вместо colored pointers.
- Поддерживает compressed oops на больших heap, что снижает memory footprint.
- Хуже, чем ZGC, масштабируется при очень больших кучах (> 100 ГБ), но лучше на средних.
- Доступен в Red Hat builds, OpenJDK, GraalVM; в Oracle JDK — не включён.
- Флаг: `-XX:+UseShenandoahGC`.

**Практические рекомендации**

- Всегда начинайте с измерений: включите GC-логи (`-Xlog:gc*:file=gc.log:time,uptime,level,tags`) и/или JFR перед сменой сборщика.
- Тестируйте под реальной нагрузкой: производительность GC сильно зависит от паттернов аллокации вашего приложения.
- В Kubernetes/Docker учитывайте `-XX:+UseContainerSupport` (включён по умолчанию с Java 10) — JVM корректно определяет limits контейнера.
- ZGC generational (JDK 21+) сейчас — лучший выбор для новых latency-sensitive сервисов при наличии ресурсов.
- Oracle JDK не включает Shenandoah; если вы используете Oracle JDK и нужна ultra-low latency — ZGC.
### Вопрос 39. Чем отличаются SoftReference, WeakReference и PhantomReference? Когда GC собирает объекты, достижимые только через каждый из этих типов ссылок, и для чего используется ReferenceQueue?

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [java.lang.ref (Java Platform SE 8) — Oracle Docs](https://docs.oracle.com/javase/8/docs/api/java/lang/ref/package-summary.html)

В Java существуют три типа ссылочных объектов, каждый из которых соответствует своему уровню достижимости (от сильного к слабому).

**Уровни достижимости (от сильного к слабому):**
1. **Strongly reachable** — объект достижим хотя бы одним потоком без прохождения через ссылочные объекты.
2. **Softly reachable** — объект не является strongly-reachable, но достижим через SoftReference.
3. **Weakly reachable** — объект не является ни strongly-, ни softly-reachable, но достижим через WeakReference.
4. **Phantom reachable** — объект не является ни strongly-, ни softly-, ни weakly-reachable, был финализирован, и на него указывает хотя бы одна PhantomReference.
5. **Unreachable** — объект не является достижимым ни одним из перечисленных способов.

**SoftReference.** GC очищает мягкие ссылки по собственному усмотрению в ответ на нехватку памяти. Ключевая гарантия: все мягкие ссылки на softly-reachable-объекты гарантированно будут очищены до того, как JVM выбросит `OutOfMemoryError`. При этом GC поощряется к тому, чтобы отдавать предпочтение очистке старых и редко используемых ссылок (LRU-политика). Основное применение — кэши, чувствительные к нехватке памяти.

**WeakReference.** Когда GC определяет, что объект является weakly-reachable, он атомарно очищает все слабые ссылки на него и объявляет объект финализируемым. Слабые ссылки не предотвращают сборку мусора: с точки зрения GC объект ведёт себя так, словно его не существует. Основное применение — канонизирующие отображения (canonicalizing mappings), например `WeakHashMap`.

**PhantomReference.** Фантомные ссылки ставятся в очередь после того, как коллектор определяет, что их референты могут быть переработаны. Метод `get()` у PhantomReference всегда возвращает `null` — это сделано намеренно, чтобы исключить воскрешение объекта в процессе cleanup. Требует обязательной передачи `ReferenceQueue` в конструктор. Основное применение — планирование действий по очистке ресурсов после финализации.

**ReferenceQueue.** Программа может зарегистрировать ссылочный объект в очереди `ReferenceQueue` при его создании. После того как GC определяет изменение уровня достижимости референта, он очищает ссылку и добавляет её в связанную очередь. Извлечь ссылки из очереди можно двумя способами: `poll()` (неблокирующий) и `remove()` (блокирующий). Для SoftReference и WeakReference регистрация в очереди необязательна — они полезны сами по себе. Для PhantomReference — обязательна, так как `get()` всегда возвращает `null` и без очереди PhantomReference совершенно бесполезна.

#### Ответ от Claude

**Иерархия достижимости и поведение GC**

Пакет `java.lang.ref` определяет пять уровней достижимости (от наиболее сильного к наиболее слабому): strongly reachable → softly reachable → weakly reachable → phantom reachable → unreachable. Уровень определяет, когда именно GC вправе собрать объект.

| Тип ссылки | Когда GC собирает объект | `get()` после очистки | Основное применение |
|---|---|---|---|
| `SoftReference` | При нехватке памяти (последний рубеж перед `OutOfMemoryError`) | `null` | Memory-sensitive кэши |
| `WeakReference` | При первом же GC-цикле, когда нет сильных/мягких ссылок | `null` | `WeakHashMap`, канонизирующие отображения |
| `PhantomReference` | После финализации, когда объект может быть переработан | Всегда `null` (даже до очистки) | Пост-финализационный cleanup ресурсов |

**SoftReference — подробности.** GC очищает SoftReference только под давлением памяти. HotSpot использует флаг `-XX:SoftRefLRUPolicyMSPerMB` (по умолчанию 1000 мс на МБ свободной кучи): чем больше свободной памяти, тем дольше мягкие ссылки выживают. Гарантируется, что все мягкие ссылки будут очищены *до* броска `OutOfMemoryError`. Это делает их пригодными для кэшей, которые автоматически освобождают память при дефиците, не требуя ручного управления.

**WeakReference — подробности.** Слабые ссылки очищаются атомарно, как только объект перестаёт быть strongly- или softly-reachable. GC обрабатывает это в один шаг: очищает ссылку и объявляет объект финализируемым. Не существует гарантий относительно точного момента очистки, но это происходит при ближайшем GC-цикле. `WeakHashMap` при каждом обращении к таблице опрашивает свою `ReferenceQueue`, чтобы удалить записи с уже собранными ключами.

**PhantomReference — подробности.** PhantomReference ставится в очередь *после* того, как объект был финализирован. В отличие от SoftReference и WeakReference, GC не очищает PhantomReference автоматически при постановке в очередь: объект остаётся phantom-reachable до тех пор, пока ссылка не будет явно очищена или сама не станет недостижимой. Именно поэтому `get()` всегда возвращает `null` — это исключает случайное воскрешение объекта. Начиная с Java 9 рекомендуется использовать `java.lang.ref.Cleaner` вместо PhantomReference вручную.

**ReferenceQueue — механизм работы.**
```
ReferenceQueue<MyClass> queue = new ReferenceQueue<>();
WeakReference<MyClass> ref = new WeakReference<>(obj, queue);
// После сборки объекта GC добавляет ref в queue
Reference<?> enqueued = queue.poll(); // неблокирующий опрос
Reference<?> enqueued = queue.remove(); // блокирующее ожидание
```
При обнаружении изменения достижимости GC: (1) очищает ссылку (для Soft и Weak), (2) помещает объект-ссылку в связанную очередь. Программа читает из очереди и выполняет необходимые действия (например, закрытие ресурсов). PhantomReference *требует* наличия очереди — без неё она полностью бесполезна.

**Расхождения между источниками.** Oracle Docs (Java 8) гласит, что PhantomReference не очищается коллектором автоматически при постановке в очередь — объект остаётся phantom-reachable. Ряд вторичных источников ошибочно утверждает, что GC очищает её, как Weak/Soft. Верно: GC *не* очищает PhantomReference автоматически; это ответственность программы. Кроме того, некоторые источники говорят, что WeakReference собирается «при каждом» GC-цикле — это верно лишь в смысле «при ближайшем минорном или мажорном GC», но не означает немедленную очистку.

### Вопрос 40. Что такое TLAB (Thread-Local Allocation Buffer)? Зачем он нужен и что происходит, когда выделяемый объект не помещается в TLAB текущего потока?

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [JVM Anatomy Quark #4: TLAB allocation — Aleksey Shipilev](https://shipilev.net/jvm/anatomy-quarks/4-tlab-allocation/)

TLAB — это блок памяти, заранее выделенный отдельному потоку из Java-кучи (как правило, из пространства Eden). Вместо того чтобы каждый раз обращаться к разделяемому аллокатору GC, поток управляет выделением памяти внутри своего буфера самостоятельно.

**Зачем нужен TLAB.** Без TLAB каждое создание объекта требовало бы вызова в нативный код VM с синхронизацией между потоками — а создание объектов в Java происходит очень часто. TLAB устраняет эту конкуренцию: каждый поток аллоцирует из собственного буфера без блокировок.

**Быстрый путь (fast path).** Внутри TLAB выделение сводится к операции «bump-the-pointer»: текущий указатель смещается на размер объекта, проверяется, что он не вышел за предел буфера, и объект инициализируется. По данным из статьи, это выполняется буквально несколькими машинными инструкциями. Отключение TLAB (`-XX:-UseTLAB`) снижает скорость аллокации в ~5 раз у одного потока, а при нескольких потоках — в ~20 раз из-за конкуренции.

**Медленный путь (slow path).** Когда объект не помещается в оставшееся пространство текущего TLAB, JVM переходит на slow path — обращается к рантайму GC. Принимается одно из двух решений:
- Если оставшееся свободное место в TLAB меньше порогового значения (waste limit), TLAB «уходит на пенсию» (retire), поток получает новый TLAB, и объект аллоцируется в нём.
- Если оставшееся свободное место превышает waste limit, объект аллоцируется напрямую в общем пространстве Eden (вне TLAB), а текущий TLAB продолжает использоваться — чтобы не терять оставшееся место.

**Размер TLAB** адаптируется динамически: JVM балансирует между накладными расходами на частое получение новых TLAB (маленький размер) и потерями на «waste» (большой размер). Флаг `-XX:MinTLABSize` задаёт нижнюю границу. По результатам бенчмарков из статьи, TLAB размером от ~16 КБ обеспечивает почти максимальную пропускную способность.

#### Ответ от Claude

**Что такое TLAB**

TLAB (Thread-Local Allocation Buffer) — небольшая область памяти в пространстве Eden (Young Generation), закреплённая за конкретным потоком. Каждый поток имеет свой TLAB и аллоцирует объекты из него без какой-либо синхронизации с другими потоками.

**Зачем нужен TLAB**

В JVM выделение памяти под объекты — одна из самых частых операций. Если бы все потоки конкурировали за один общий указатель аллокации в Eden, это порождало бы постоянную синхронизацию. TLAB решает эту проблему: буфер выделяется потоку под синхронизацией (редко), а все объекты внутри буфера аллоцируются без синхронизации (часто). Согласно измерениям Aleksey Shipilev, отключение TLAB снижает скорость аллокации в 5–20 раз в зависимости от числа потоков.

**Как работает быстрый путь (fast path)**

Внутри TLAB работает «bump-the-pointer»-аллокация — JIT компилирует её inline в пользовательский код:
1. Загрузить текущий указатель TLAB (`top`).
2. Прибавить размер объекта.
3. Сравнить с пределом (`end`).
4. Если не превышен — обновить `top`, обнулить слот заголовка объекта.

Это буквально несколько машинных инструкций.

**Что происходит, когда объект не помещается в TLAB (slow path)**

JVM применяет механизм waste limit (управляется флагом `-XX:TLABWasteTargetPercent`, по умолчанию 1% от размера TLAB):

1. **Объект мал, но TLAB кончился (остаток < waste limit):**
   Текущий TLAB «уходит на пенсию» (retire) — оставшееся место помечается как потраченное (waste). Поток запрашивает у GC новый TLAB и аллоцирует объект в нём.

2. **Объект не помещается, но в TLAB ещё много места (остаток > waste limit):**
   Объект аллоцируется напрямую в разделяемом пространстве Eden (вне TLAB, с синхронизацией). Текущий TLAB продолжает использоваться потоком, чтобы сохранить его оставшееся место. Порог waste limit при каждой такой медленной аллокации увеличивается на `-XX:TLABWasteIncrement` (по умолчанию 4%).

3. **Объект слишком большой для любого TLAB (например, большой массив):**
   Объект аллоцируется напрямую в разделяемом Eden (или в Humongous-регионе для G1GC, если объект превышает половину размера G1-региона), с необходимой синхронизацией. Такая аллокация значительно медленнее fast path.

**Настройка TLAB**

| Флаг | Описание | По умолчанию |
|---|---|---|
| `-XX:+UseTLAB` | Включить TLAB | включено |
| `-XX:MinTLABSize` | Минимальный размер TLAB | 2 КБ |
| `-XX:TLABWasteTargetPercent` | Начальный порог waste limit (% от TLAB) | 1% |
| `-XX:TLABWasteIncrement` | Шаг увеличения waste limit при slow-аллокации | 4 |

Размер TLAB определяется JVM адаптивно исходя из частоты аллокаций каждого потока и числа потоков. Все основные GC в OpenJDK (G1GC, ZGC, Shenandoah, ParallelGC) поддерживают TLAB.
