### Вопрос 41. Что такое Compressed OOPs (-XX:+UseCompressedOops) и почему JVM автоматически отключает эту оптимизацию при размере heap более 32 ГБ? Как это влияет на потребление памяти и производительность?

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Java HotSpot Virtual Machine Performance Enhancements — Oracle Docs (Java 23)](https://docs.oracle.com/en/java/javase/23/vm/java-hotspot-virtual-machine-performance-enhancements.html)

Compressed OOPs (Ordinary Object Pointers) — оптимизация в HotSpot JVM, при которой ссылки на объекты в 64-битной JVM представляются не как 64-битные адреса, а как 32-битные смещения относительно базового адреса кучи.

**Как работает кодирование.** Поскольку объекты в куче выровнены по границе 8 байт, три младших бита любого адреса объекта всегда равны нулю. Это позволяет хранить не полный адрес, а смещение в единицах объектов (не байт):

- **Декодирование:** `native_address = base + (compressed_oop << 3)`
- **Кодирование:** `compressed_oop = (native_address - base) >> 3`

32-битный указатель способен адресовать 2^32 = ~4 миллиарда объектов. При выравнивании по 8 байт это даёт: 4 млрд × 8 байт = **~32 ГБ**.

**Почему отключается при heap > 32 ГБ.** При куче более 32 ГБ 32-битное смещение не покрывает весь диапазон адресов даже с масштабированием на 8 — адресное пространство исчерпывается. JVM вынуждена переключиться на полные 64-битные указатели.

**Влияние на память.** На 64-битной системе без Compressed OOPs каждая ссылка занимает 8 байт. С Compressed OOPs — 4 байта. По данным Oracle, потребление кучи приложением на LP64-системе без сжатых ссылок может быть примерно в 1.5 раза больше, чем в 32-битном режиме ILP32.

**Влияние на производительность.** Сжатые ссылки сокращают объём данных, помещаемых в кэш процессора, что улучшает cache locality и снижает давление на шину памяти. Стоимость кодирования/декодирования (сдвиг и сложение) незначительна по сравнению с выигрышем от меньшего footprint. Для кучи менее 4 ГБ JVM может использовать байтовые смещения вместо объектных — и тогда сдвиг не нужен вовсе (zero-based compressed OOPs).

**Включение по умолчанию.** Compressed OOPs включены по умолчанию начиная с Java SE 6u23 и в Java SE 7+ для 64-битных процессов при `-Xmx` < 32 ГБ.

#### Ответ от Claude

**Что такое Compressed OOPs**

На 64-битной платформе каждая ссылка на объект занимает 8 байт. Compressed OOPs (Ordinary Object Pointers) — встроенная в HotSpot оптимизация, которая хранит эти ссылки как 32-битные целые числа-смещения, а не как полные 64-битные адреса. JIT-компилятор и интерпретатор прозрачно кодируют/декодируют ссылки при обращениях.

**Как работает кодирование: математика предела 32 ГБ**

Все объекты в JVM выровнены по 8 байт (3 младших бита адреса всегда = 0). Благодаря этому JVM хранит не байтовый адрес, а индекс объекта (смещение в единицах по 8 байт):

```
native_address = heap_base + (compressed_oop × 8)
compressed_oop = (native_address - heap_base) / 8
```

Максимальное значение 32-битного беззнакового числа: 2^32 - 1 ≈ 4.29 млрд.
Максимальный адресуемый байт: 4.29 млрд × 8 байт ≈ **34.35 ГБ** (на практике округляется до ~32 ГБ с учётом выравнивания самой кучи и накладных расходов).

При heap > 32 ГБ диапазон адресов перестаёт укладываться в 32 бита даже с масштабированием — JVM автоматически отключает Compressed OOPs и переходит на полные 8-байтные ссылки.

**Режимы работы Compressed OOPs**

В зависимости от размера кучи HotSpot использует разные режимы оптимизации:

| Размер heap | Режим | Смещение | Масштабирование |
|---|---|---|---|
| < 4 ГБ | Zero-based, byte offset | 0 | нет |
| 4–32 ГБ | Zero-based, scaled | 0 | × 8 |
| > 32 ГБ | Compressed OOPs отключены | — | — |

При куче < 4 ГБ JVM использует байтовые смещения (без умножения), что ещё быстрее. При куче 4–32 ГБ с нулевым базовым адресом декодирование — просто сдвиг влево на 3 бита.

**Влияние на потребление памяти**

Каждая ссылка на объект: 8 байт без Compressed OOPs vs 4 байта с ними. Экономия особенно значительна для:
- Заголовков объектов (object header содержит ссылки на класс).
- Массивов объектов.
- Коллекций (ArrayList, HashMap и т.п. — ссылки на элементы).
- Полей-ссылок в POJO.

По оценке Oracle, в типичном приложении потребление кучи на LP64 без сжатых ссылок на ~50% выше, чем в ILP32-режиме.

**Влияние на производительность**

*Плюсы:* меньший footprint → больше объектов в CPU-кэше → меньше cache-miss → меньше нагрузка на шину памяти. Это зачастую важнее стоимости самих инструкций encode/decode.

*Минусы:* несколько дополнительных инструкций (shift + add) при каждом разыменовании ссылки. На практике этот оверхед пренебрежимо мал.

**Расширение предела с помощью ObjectAlignmentInBytes**

Флаг `-XX:ObjectAlignmentInBytes=16` (вместо 8 по умолчанию) увеличивает предел до ~64 ГБ при сохранении Compressed OOPs. Ограничения: значение должно быть степенью двойки в диапазоне 8–256; увеличение выравнивания увеличивает внутреннюю фрагментацию объектов (padding).

**Настройка**

```
-XX:+UseCompressedOops   # явное включение (по умолчанию при heap < 32 ГБ)
-XX:-UseCompressedOops   # явное отключение
-XX:ObjectAlignmentInBytes=16  # расширение предела до ~64 ГБ
```

Дополнительно, начиная с Java 15, существует `-XX:+UseCompressedClassPointers` — отдельный флаг для сжатия ссылок на метаданные классов в Metaspace (ранее управлялся совместно с `UseCompressedOops`).

**Расхождения между источниками.** Ряд источников называет порог «32 ГБ» без уточнений. Точнее: теоретический предел составляет 2^32 × 8 = 32 ГБ, но конкретное пороговое значение может незначительно варьироваться в зависимости от реализации JVM и выравнивания кучи. Oracle официально документирует порог как «менее 32 гигабайт».

### Вопрос 42. Что хранится в заголовке объекта (object header) в HotSpot JVM? Объясните назначение mark word и klass pointer, и как biased locking влияет на структуру mark word.

**Категория:** JVM и GC · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Where Object Headers Are Stored and What They Contain in Java Memory — Alexander Obregon, Medium](https://medium.com/@AlexanderObregon/where-object-headers-are-stored-and-what-they-contain-in-java-memory-5002b9fb6ee4)

Каждый Java-объект в HotSpot JVM предваряется заголовком (object header), который располагается в heap-памяти непосредственно перед полями объекта. Заголовок состоит из двух компонентов: mark word и class pointer (klass pointer).

**Mark word** — первые 8 байт (на 64-битной JVM) или 4 байта (на 32-битной JVM). Это многоцелевое поле: оно переиспользует одни и те же биты под разные данные в зависимости от текущего состояния объекта. В нём хранятся: identity hash code, счётчик возраста для generational GC, информация о синхронизации.

**Klass pointer** — расположен сразу за mark word; ссылается на метаданные класса в Metaspace. Связывает объект со структурой, описывающей его тип, поля, методы и C++-vtable. После создания объекта этот указатель не меняется. На 64-битной JVM с включёнными Compressed OOPs (-XX:+UseCompressedOops, включены по умолчанию) klass pointer занимает 4 байта; без сжатия — 8 байт.

**Размер заголовка:**
- 32-битная JVM: 4 байта (mark word) + 4 байта (klass pointer) = 8 байт
- 64-битная JVM + Compressed OOPs: 8 байт (mark word) + 4 байта (klass pointer) = 12 байт
- 64-битная JVM без сжатия: 8 байт + 8 байт = 16 байт

**Состояния mark word** (нижние 2 бита — lock bits):
- Unlocked (01): хранит identity hash code (31 бит), возраст GC (4 бита)
- Lightweight locked (00): указатель на lock record в стеке текущего потока
- Heavyweight/inflated (10): указатель на раздутый монитор (ObjectMonitor)
- Marked for GC (11): используется сборщиком мусора при маркировке

#### Ответ от Claude

**Структура заголовка объекта в HotSpot JVM**

Заголовок объекта (object header) хранится в heap вместе с объектом и содержит два поля:

**1. Mark word**

Mark word — многоцелевое поле переменного содержимого. Его размер: 8 байт на 64-битной JVM, 4 байта на 32-битной. Нижние 2 бита кодируют состояние блокировки. Вот разметка mark word для 64-битной JVM согласно исходному коду OpenJDK (`markWord.hpp`):

```
Unlocked (01):   [unused:25 | hash:31 | unused:1 | age:4 | unused:1 | lock:2]
Lightweight (00):[ptr к lock record в стеке                           | lock:2]
Heavyweight (10):[ptr к ObjectMonitor                                  | lock:2]
Marked GC (11):  [ptr к forwarding address                             | lock:2]
Biased (101):    [JavaThread* | epoch:2 | age:4 | biased_lock:1 | lock:2]
```

Поля в нормальном (unlocked) состоянии:
- **hash (31 бит):** identity hash code; вычисляется лениво при первом вызове `System.identityHashCode()` и кешируется здесь
- **age (4 бита):** счётчик выживания в young generation, максимум 15 (параметр `-XX:MaxTenuringThreshold`)
- **lock (2 бита):** кодируют текущий режим синхронизации

**2. Klass pointer**

Указывает на структуру `Klass` в Metaspace: описание типа объекта, его полей, методов, C++-vtable для виртуальной диспетчеризации. С включёнными Compressed OOPs занимает 4 байта, без них — 8 байт.

**Biased locking (смещённая блокировка)**

Biased locking — оптимизация для случая, когда один поток многократно захватывает один и тот же монитор. В этом режиме mark word принимает вид:

```
[JavaThread* (54 бит) | epoch (2 бита) | age (4 бита) | biased_lock=1 | lock=01]
```

- **JavaThread\*:** указатель на поток-владелец (pointer выровнен по 256 байт, что освобождает младшие биты под служебные поля)
- **epoch:** метка актуальности bias, хранится и в заголовке объекта, и в метаданных класса; при массовой отзыве смещения (bulk revocation) эпоха в классе инкрементируется, что автоматически инвалидирует все существующие смещения объектов этого класса
- **biased_lock=1, lock=01:** отличают biased от unlocked (там biased_lock=0)

Важные следствия biased locking:
- Если у объекта был вычислен identity hash code (hash занял соответствующие биты), biased locking для этого объекта невозможен — hash и JavaThread\* занимают одни и те же биты, bias отзывается
- При конкуренции потоков bias отзывается (revocation), объект переходит в lightweight locked или heavyweight locked; revocation — stop-the-world операция для выявляющего потока
- Начиная с JDK 15 biased locking отключён по умолчанию (JEP 374), в JDK 21 флаг -XX:+UseBiasedLocking удалён окончательно

**Источники:**
- [OpenJDK markWord.hpp — исходный код разметки mark word](https://github.com/openjdk/jdk21u/blob/master/src/hotspot/share/oops/markWord.hpp)
- [Java Objects Inside Out — Aleksey Shipilev (детальный анализ с JOL)](https://shipilev.net/jvm/objects-inside-out/)
- [Biased-locking Synchronization — OpenJDK Wiki HotSpot](https://wiki.openjdk.org/spaces/HotSpot/pages/11829266/Biased-locking+Synchronization)

### Вопрос 43. Почему метод finalize() считается устаревшим и опасным? Какие проблемы он создаёт (задержка GC, нагрузка на finalizer-поток, воскрешение объектов)? Чем его заменяет Cleaner API (Java 9+) и в чём его преимущества?

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Clean your Memory: From Finalize to Cleaner — Nicolas Frankel, blog.frankel.ch](https://blog.frankel.ch/java-cleaner/)

Метод `finalize()` — переопределяемый метод класса `Object`, который JVM вызывает перед сборкой объекта. Изначально предназначался для освобождения нативных ресурсов. Однако у него есть серьёзные проблемы:

- **Непредсказуемость:** JVM решает когда (и решает ли вообще) вызывать `finalize()`. Полагаться на него для освобождения ресурсов нельзя.
- **Дополнительный GC-цикл:** Объекты с `finalize()` не удаляются при первой недостижимости — они помещаются в очередь финализации (finalizer queue), вызывается `finalize()`, и только в следующем цикле GC объект может быть собран. Это удваивает задержку.
- **Нагрузка на finalizer-поток:** Финализация выполняется в специальном daemon-потоке (Finalizer thread). Если приложение создаёт объекты быстрее, чем финализатор успевает их обработать, очередь растёт, вызывая `OutOfMemoryError`.
- **Проблемы с потоками:** `finalize()` может выполняться в произвольном потоке, что вносит трудноотлаживаемые гонки данных.

**Cleaner API (Java 9+)** использует `PhantomReference` и фоновый daemon-поток под капотом. Объект регистрируется с действием очистки (cleaning action). Когда объект становится phantom-достижимым, действие выполняется. При этом `Cleaner` реализует `AutoCloseable`, что позволяет вызывать очистку детерминировано через `try-with-resources`.

Ключевое правило: cleaning action не должен содержать ссылку на регистрируемый объект (иначе объект не станет phantom-достижимым). Рекомендуется оформлять действие отдельным статическим вложенным классом.

#### Ответ от Claude

**Проблемы finalize()**

`finalize()` объявлен устаревшим в Java 9 (`@Deprecated`), помечен для удаления в Java 18 (JEP 421) и фактически удалён из практики из-за комплекса проблем:

**1. Два цикла GC вместо одного**

Когда GC обнаруживает недостижимый объект с переопределённым `finalize()`, он не удаляет его немедленно. Вместо этого:
1. Объект помещается в специальную очередь (finalizer queue / `java.lang.ref.FinalReference`)
2. Финализатор-поток (`java.lang.ref.Finalizer`) вызывает `finalize()` на объекте
3. Только после следующего прохода GC объект может быть фактически собран

Итог: объекты с `finalize()` гарантированно переживают минимум одну лишнюю GC-итерацию и с высокой вероятностью продвигаются из Young в Old generation (promoted), нагружая major GC.

**2. Перегрузка finalizer-потока и OutOfMemoryError**

Finalizer-поток — единственный (по умолчанию), и у него нет гарантий приоритета. При высокой скорости создания объектов очередь финализации растёт быстрее, чем финализатор её опустошает. Поскольку объекты в очереди удерживаются от сборки, heap заполняется, что приводит к `OutOfMemoryError`, даже если бизнес-логика давно «отпустила» объекты.

**3. Воскрешение объектов (object resurrection)**

Внутри `finalize()` можно присвоить `this` статическому полю или иной GC root:

```java
static MyResource zombie;

@Override
protected void finalize() throws Throwable {
    zombie = this; // объект снова достижим — "воскресение"
}
```

Объект становится снова достижимым и не будет собран. JVM вызывает `finalize()` у каждого объекта не более одного раза, поэтому при следующей недостижимости `finalize()` уже не вызывается — и объект тихо утечёт или будет собран без очистки. Паттерн крайне сложно отлаживать.

**4. Прочие риски**
- Исключение в `finalize()` игнорируется JVM, очистка не завершается
- Невозможно детерминировано освободить ресурс в нужный момент
- Финализируемые объекты хуже оптимизируются JIT

**Cleaner API (java.lang.ref.Cleaner, Java 9+)**

`Cleaner` строится на `PhantomReference`: объект регистрируется вместе с действием очистки (`Runnable`). Когда объект становится phantom-достижимым (GC уже готов его собрать), действие ставится в очередь и выполняется в daemon-потоке Cleaner'а.

```java
public class NativeResource implements AutoCloseable {
    private static final Cleaner CLEANER = Cleaner.create();

    // Внутренний класс НЕ хранит ссылку на внешний NativeResource
    private static class CleanAction implements Runnable {
        private final long nativeHandle;
        CleanAction(long handle) { this.nativeHandle = handle; }

        @Override
        public void run() {
            freeNative(nativeHandle); // освобождение нативного ресурса
        }
    }

    private final Cleaner.Cleanable cleanable;

    public NativeResource(long handle) {
        cleanable = CLEANER.register(this, new CleanAction(handle));
    }

    @Override
    public void close() {
        cleanable.clean(); // детерминированная очистка через try-with-resources
    }
}
```

**Преимущества Cleaner перед finalize():**
- Не задерживает сборку: объект собирается немедленно, действие выполняется уже после
- Детерминированный вызов через `close()` / `try-with-resources` без ожидания GC
- Нет риска воскрешения: PhantomReference не позволяет получить ссылку на объект из cleaning action
- Исключения в cleaning action изолированы и не влияют на GC
- Возможность использовать несколько экземпляров `Cleaner` с разными потоками для изоляции

**Важно:** `Cleaner` — страховочный механизм, не основной. Для детерминированного освобождения ресурсов предпочтительнее `try-with-resources` с `AutoCloseable`. `Cleaner` применяется, когда нужно гарантировать очистку нативных ресурсов на случай, если `close()` не был вызван.

### Вопрос 44. Как анализировать GC-логи в production? Объясните разницу между старыми флагами (-XX:+PrintGCDetails, -XX:+PrintGCDateStamps) и новым унифицированным логированием (-Xlog:gc* в Java 9+). Какие метрики нужно отслеживать в GC-логах для диагностики проблем с производительностью?

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Java GC Logs: How to Read and Debug Fast — Last9 Blog](https://last9.io/blog/java-gc-logs/)

**Старые флаги (Java 8 и ниже)**

В Java 8 GC-логирование настраивается набором отдельных флагов:

```
-XX:+PrintGCDetails -XX:+PrintGCDateStamps -XX:+PrintGCCause
-XX:+PrintTenuringDistribution -Xloggc:/path/to/gc.log
```

Ротация логов управляется тремя отдельными флагами:

**Ключевые метрики для диагностики производительности**

**1. Время пауз (Pause Times)**
Stop-the-world паузы — прямое влияние на latency. Следует отслеживать:
- Среднее время паузы
- Максимальное время паузы (P99, P99.9)
- Паузы > 200 мс видны пользователю; паузы > 1 с — критичны

Диагностика: если паузы растут — возможно, Old generation не успевает очищаться, heap слишком мал, или выбран неподходящий GC.

**2. GC Throughput**
Доля времени, которую JVM тратит на полезную работу (не GC):

```
throughput = (total_time - gc_time) / total_time * 100%
```

Норма: > 95%. Если ниже — GC потребляет слишком много ресурсов.

**3. Occupancy перед и после GC (Heap Utilization)**
Строка вида `33280K->5096K(125952K)` показывает: объём heap до GC → после GC (всего heap). Высокое значение «после» при многократных GC — признак утечки или недостаточного heap.

**4. Frequency коллекций**
- Частые minor GC (каждые секунды) → высокий allocation rate, слишком малый Young generation
- Частые Full GC → утечка памяти, фрагментация heap, или неверно подобран GC

**5. Promotion rate (темп продвижения)**
Объекты, переходящие из Young в Old generation. Высокий promotion rate при нормальном allocation rate означает, что объекты живут дольше, чем ожидается — возможна ошибка архитектуры или утечка.

**6. Тренд Old Generation**
Если Old generation растёт от Full GC к Full GC и не уменьшается до baseline — это утечка памяти. В логах: сравниваем heap after GC по серии Full GC событий.

### Вопрос 45. Что такое «humongous-объекты» в G1 GC? Почему их частое выделение деградирует производительность, и как это диагностировать и устранить через JVM-флаги и изменение кода?

**Категория:** JVM и GC · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Garbage-First (G1) Garbage Collector — Oracle Java 17 GC Tuning Guide](https://docs.oracle.com/en/java/javase/17/gctuning/garbage-first-g1-garbage-collector1.html)

Humongous-объекты — это объекты, размер которых равен или превышает половину размера региона G1. Такие объекты выделяются непосредственно в старом поколении в виде последовательности смежных регионов (помеченных `StartsHumongous` и `ContinuesHumongous`), минуя молодое поколение.

**Проблемы производительности:**
- Пространство в последнем регионе последовательности, оставшееся после размещения объекта, теряется до тех пор, пока весь объект не будет освобождён. Это приводит к фрагментации кучи.
- Humongous-объекты никогда не перемещаются, даже во время Full GC. Из-за этого может возникнуть преждевременный Full GC: если в куче ещё есть свободная память, но нет достаточного количества смежных регионов.
- Каждое выделение humongous-объекта проверяет порог IHOP и может немедленно инициировать цикл конкурентной маркировки.
- До JDK 8u40 такие объекты жили до полного GC. Сейчас G1 может досрочно освобождать примитивные массивы (`bool`, целые, числа с плавающей точкой), если на них мало ссылок — это называется eager reclamation и управляется флагом `-XX:+G1EagerReclaimHumongousObjects` (включён по умолчанию).

**Диагностика:**
Включить логирование уровня `info` для подсистемы `gc+heap`:
```
-Xlog:gc+heap=info
```
В логах искать строки вида:
```
Humongous regions: X->Y
```
Если `Y` велико относительно числа регионов старого поколения — в приложении активно используются humongous-объекты.

**Устранение через JVM-флаги:**
- `-XX:G1HeapRegionSize=<N>` — увеличить размер региона (степень двойки, от 1 МБ до 32 МБ, а в Java 20+ до 512 МБ). Если регион стал больше, то прежде humongous-объекты перестают быть таковыми и размещаются по обычному пути.
- `-Xmx<size>` — увеличить размер кучи, что снижает давление от фрагментации.
- `-XX:+G1EagerReclaimHumongousObjects` — убедиться, что eager reclamation включён (по умолчанию включён с Java 8u60 / Java 9).

**Устранение через изменение кода:**
- Найти места создания крупных `byte[]`, `char[]`, `String`, больших коллекций или кешированных объектов.
- Заменить буферизованную загрузку данных (например, десериализация JSON целиком) на потоковую обработку (streaming parsers — Jackson `JsonParser`, Gson `JsonReader`).
- Использовать пулы буферов (например, `ByteBuffer` из пула, Netty `ByteBufAllocator`) вместо регулярного создания новых массивов.
- Пересмотреть кешируемые структуры данных: разбить крупный кеш-массив на несколько сегментов меньшего размера.

#### Ответ от Claude

**Что такое humongous-объект**

В G1 GC humongous-объект — это объект, размер которого >= половины размера региона G1. Размер региона задаётся флагом `-XX:G1HeapRegionSize` и по умолчанию выбирается автоматически (эргономически) как такое значение, при котором вся куча делится примерно на 2048 регионов; допустимые значения — степени двойки от 1 МБ до 32 МБ (в Java 20+ до 512 МБ).

Такие объекты размещаются не в молодом поколении, а сразу в старом — в виде цепочки смежных регионов. Первый регион маркируется `StartsHumongous`, каждый последующий — `ContinuesHumongous`. Остаток пространства в последнем регионе пустует до момента освобождения всего объекта.

**Почему это деградирует производительность**

1. **Фрагментация и преждевременный Full GC.** Humongous-объекты занимают смежные регионы. Если таких последовательностей много — куча фрагментируется. G1 может не найти достаточного числа смежных свободных регионов для нового humongous-объекта даже при наличии суммарно достаточного свободного пространства. Это вызывает Full GC до исчерпания памяти.

2. **Объекты не перемещаются.** В отличие от обычных объектов, humongous-объекты не копируются ни во время evacuation-пауз, ни во время Full GC. Это означает, что компактизация кучи невозможна — фрагментация только накапливается.

3. **Досрочный запуск маркировки.** Каждое выделение humongous-объекта проверяет порог IHOP. Если порог превышен, G1 немедленно инициирует Initial Mark — даже если по другим критериям это было бы преждевременно. Частые humongous-аллокации приводят к более частым циклам маркировки и лишним паузам.

4. **Неэффективное использование памяти.** Объект размером, например, 1 МБ + 1 байт при размере региона 2 МБ займёт два полных региона (4 МБ в общей сложности), потратив почти 50% зря.

**Диагностика**

Включить логирование:
```
-Xlog:gc+heap=info
```
Искать в логах:
```
Humongous regions: X->Y
```
Значение `Y` — текущее число регионов, занятых humongous-объектами. Если оно велико относительно числа регионов старого поколения — проблема подтверждена.

Для детальной диагностики:
```
-Xlog:gc*=debug
```
Строки `G1 Humongous Allocation` укажут на конкретные аллокации.

**Устранение через JVM-флаги**

| Флаг | Действие |
|---|---|
| `-XX:G1HeapRegionSize=<N>m` | Увеличить размер региона. Объекты, бывшие humongous, перейдут в разряд обычных. |
| `-Xmx<size>` | Увеличить кучу — больше регионов, меньше вероятность фрагментации. |
| `-XX:+G1EagerReclaimHumongousObjects` | Включён по умолчанию. Позволяет освобождать humongous примитивные массивы досрочно, не дожидаясь цикла маркировки. |

**Устранение через изменение кода**

- Переходить на потоковую обработку данных (streaming JSON/XML parsers) вместо загрузки всего документа в память.
- Использовать пулы буферов (`ByteBuffer` pool, Netty `PooledByteBufAllocator`) — переиспользование вместо создания новых крупных массивов.
- Разбивать крупные кеши и структуры данных на сегменты меньшего размера.
- Профилировать аллокации инструментами: async-profiler с режимом `alloc`, JFR-событие `jdk.ObjectAllocationInNewTLAB`/`jdk.ObjectAllocationOutsideTLAB`.

**Расхождений между источниками не выявлено.** Oracle Java 17, Java 20 GC Tuning Guides и HeapHero blog согласуются в определении, причинах деградации и рекомендуемых флагах.

### Вопрос 46. Что такое IHOP (InitiatingHeapOccupancyPercent) в G1 GC и что такое Adaptive IHOP? Как неправильно выставленный IHOP приводит к Full GC, и как его правильно настраивать на основе GC-логов?

**Категория:** JVM и GC · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Garbage-First (G1) Garbage Collector — Oracle Java 17 GC Tuning Guide](https://docs.oracle.com/en/java/javase/17/gctuning/garbage-first-g1-garbage-collector1.html)

**IHOP (InitiatingHeapOccupancyPercent)** — это порог заполненности старого поколения, при превышении которого G1 инициирует начальную фазу маркировки (Concurrent Start / Initial Mark). Выражается в процентах от размера старого поколения. По умолчанию — 45%.

**Adaptive IHOP** (доступен с Java 9, включён по умолчанию через `-XX:+G1UseAdaptiveIHOP`): G1 автоматически определяет оптимальное значение IHOP, наблюдая за тем, сколько времени занимает маркировка и сколько памяти выделяется в старом поколении в течение цикла маркировки. Внутренне Adaptive IHOP стремится запустить маркировку так, чтобы к моменту начала первой mixed-коллекции старое поколение оказалось заполнено примерно до уровня `максимальный размер старого поколения - G1HeapReservePercent`.

Флаг `-XX:InitiatingHeapOccupancyPercent=45` при включённом Adaptive IHOP используется лишь как стартовое значение — пока накоплено недостаточно наблюдений (менее `G1AdaptiveIHOPNumInitialSamples`, по умолчанию 3 цикла).

**Как неправильный IHOP приводит к Full GC:**
- Если IHOP слишком высок — маркировка начинается слишком поздно, старое поколение успевает заполниться до завершения маркировки. G1 не успевает освободить место, возникает evacuation failure, а затем Full GC.
- Full GC из-за высокого IHOP можно обнаружить в логах по строке: `Pause Full (Allocation Failure)`.

**Настройка на основе GC-логов:**
1. Включить детальное логирование: `-Xlog:gc*=debug`.
2. Найти `Pause Full (Allocation Failure)` — это подтверждение Full GC из-за нехватки места.
3. Убедиться, что перед Full GC нет успевшей завершиться concurrent marking — это означает, что маркировка начинается слишком поздно.
4. Варианты коррекции:
   - **Оставить Adaptive IHOP и увеличить буфер**: `-XX:G1ReservePercent=15` (дефолт 10) — тогда Adaptive IHOP будет выбирать более ранний старт маркировки.
   - **Отключить Adaptive IHOP и выставить вручную**: `-XX:-G1UseAdaptiveIHOP -XX:InitiatingHeapOccupancyPercent=35` (меньше дефолта 45).

#### Ответ от Claude

**Что такое IHOP**

`InitiatingHeapOccupancyPercent` (IHOP) — порог заполненности старого поколения (в процентах от его текущего максимального размера), при достижении которого G1 запускает фазу Concurrent Start (Initial Mark). Это запускает цикл конкурентной маркировки, цель которого — определить, какие объекты в старом поколении живы, чтобы затем освободить мёртвые в ходе mixed GC (фазы Space Reclamation).

Значение по умолчанию: **45%** от размера старого поколения.

**Что такое Adaptive IHOP**

Начиная с Java 9, G1 поддерживает автоматическую адаптацию IHOP — **Adaptive IHOP** (`-XX:+G1UseAdaptiveIHOP`, включён по умолчанию).

Механизм работы:
- G1 наблюдает за двумя параметрами: скоростью выделения памяти в старом поколении и продолжительностью цикла маркировки.
- На основе этих наблюдений G1 вычисляет, при каком уровне заполненности старого поколения нужно начать маркировку, чтобы она успела завершиться до того, как старое поколение переполнится.
- Внутренний целевой уровень для начала первой mixed-коллекции: `максимальный размер старого поколения - G1HeapReservePercent`.
- Первые `G1AdaptiveIHOPNumInitialSamples` (по умолчанию 3) цикла маркировки используется фиксированное значение `-XX:InitiatingHeapOccupancyPercent` как начальное приближение.

**Как неправильный IHOP приводит к Full GC**

G1 использует конкурентную маркировку, чтобы подготовить информацию для освобождения памяти в старом поколении. Если маркировка не успевает завершиться до исчерпания старого поколения — возникает evacuation failure, что может привести к Full GC.

Сценарии:

1. **IHOP слишком высокий (маркировка начинается слишком поздно).** Старое поколение заполняется быстрее, чем успевает завершиться маркировка. G1 не успевает запустить mixed GC. Следствие: `Pause Full (Allocation Failure)`.

2. **Скачкообразный рост аллокаций.** Если приложение начало аллоцировать значительно быстрее, чем в предыдущих наблюдаемых циклах, Adaptive IHOP не успевает перестроиться — поведение аналогично случаю 1.

3. **IHOP слишком низкий (маркировка начинается слишком рано).** Избыточные циклы маркировки, лишние паузы, рост накладных расходов, но Full GC не вызывает напрямую.

**Диагностика по GC-логам**

Включить логирование:
```
-Xlog:gc*=debug
```

Ключевые признаки в логах:

| Строка в логе | Значение |
|---|---|
| `Pause Full (Allocation Failure)` | Full GC из-за исчерпания памяти — основной индикатор проблемы |
| `Concurrent Mark Abort` | Маркировка прервана — старое поколение переполнилось раньше завершения |
| `Pause Young (Concurrent Start)` | Начало цикла маркировки — смотреть, при каком % заполненности запускается |
| `GC Heap before` / `GC Heap after` | Динамика заполненности старого поколения |

Если `Pause Full` встречается регулярно, а перед ним нет успешно завершённой маркировки — IHOP слишком высокий или скорость аллокации возросла.

**Настройка IHOP**

Вариант 1 (рекомендуемый при включённом Adaptive IHOP): увеличить буфер резерва, тем самым заставив Adaptive IHOP выбирать более ранний старт маркировки:
```
-XX:G1ReservePercent=15    # дефолт 10
```

Вариант 2 (при предсказуемой нагрузке): отключить адаптацию и выставить значение вручную:
```
-XX:-G1UseAdaptiveIHOP
-XX:InitiatingHeapOccupancyPercent=35    # ниже дефолта 45
```

Вариант 3 (если маркировка медленная): увеличить число потоков конкурентной маркировки:
```
-XX:ConcGCThreads=<N>    # дефолт: ParallelGCThreads / 4
```

**Общая рекомендация:** сначала попробовать увеличение `-XX:G1ReservePercent` с оставленным Adaptive IHOP — это безопаснее, чем ручная настройка. Ручное отключение Adaptive IHOP оправдано только если нагрузка на приложение стабильна и хорошо предсказуема.

**Расхождений между источниками не выявлено.** Oracle Java 17 GC Tuning Guide, Oracle Java 8 GC Tuning Guide и HeapHero G1GC Performance Tuning Guide согласуются в описании механизма и способах устранения.

### Вопрос 47. Что такое evacuation failure в G1 GC? При каких условиях она возникает, почему она приводит к Full GC, и какие флаги помогают снизить вероятность её возникновения?

**Категория:** JVM и GC · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Garbage-First Garbage Collector Tuning — Oracle Java 8 GC Tuning Guide](https://docs.oracle.com/javase/8/docs/technotes/guides/vm/gctuning/g1_gc_tuning.html)

Evacuation failure (или to-space overflow / to-space exhausted) возникает, когда G1 GC не может найти достаточно свободных регионов для размещения объектов, выживших при сборке мусора (survivor-объектов и объектов, продвигаемых в старое поколение). Симптомы появляются в GC-логах как:

```
[GC pause (G1 Evacuation Pause) (mixed) (to-space exhausted), 0.1957310 secs]
[GC pause (G1 Evacuation Pause) (mixed) (to-space overflow), 0.1957310 secs]
```

**Причины:**
- Куча уже заполнена до максимума, нет свободных регионов для эвакуации объектов.
- В частности, пики выделения humongous-объектов могут мгновенно занять все свободные регионы, не оставив to-space для эвакуации.

**Связь с Full GC:**
Если G1 не может завершить эвакуацию из-за нехватки пространства — возникает полный (stop-the-world) Full GC, который выполняет полную компактизацию кучи.

**Флаги для снижения вероятности:**
1. Увеличить резервное пространство: `-XX:G1ReservePercent=10` (увеличить значение и соответственно увеличить кучу).
2. Запускать маркировку раньше: `-XX:InitiatingHeapOccupancyPercent=35` (снизить относительно дефолта 45).
3. Увеличить число потоков маркировки: `-XX:ConcGCThreads=n`.

#### Ответ от Claude

**Что такое evacuation failure**

Evacuation failure — сбой эвакуации объектов в ходе GC-паузы G1. В процессе обычной (young или mixed) коллекции G1 копирует живые объекты из регионов-источников (которые нужно освободить) в регионы-приёмники (survivor-регионы или регионы старого поколения). Если в момент копирования свободных регионов-приёмников не хватает — эвакуация проваливается.

В логах это обозначается тегом `to-space exhausted` или `to-space overflow`:
```
[GC pause (G1 Evacuation Pause) (young) (to-space exhausted), ...]
```

**Поведение G1 при evacuation failure**

G1 не прерывает паузу немедленно. Вместо этого:
1. Объекты, уже перемещённые до сбоя, остаются в новых местах.
2. Объекты, которые ещё не были скопированы, остаются на месте — ссылки на них корректируются «на месте».
3. GC-пауза завершается, приложение возобновляет работу.

Сам по себе evacuation failure добавляет минимальные накладные расходы и не приводит к немедленному Full GC. Однако если G1 предполагает, что большинство объектов уже эвакуировано и оставшегося пространства достаточно для продолжения работы — а это предположение не выполняется — G1 в конечном счёте запускает Full GC.

**Почему evacuation failure приводит к Full GC**

Full GC запускается, когда после evacuation failure G1 приходит к выводу, что не может удовлетворить запросы аллокации обычными средствами. Full GC выполняет полную остановку мира (stop-the-world) и последовательную (или параллельную начиная с Java 10) компактизацию всей кучи. Это на 1–2 порядка медленнее, чем обычная пауза G1.

**Условия возникновения**

1. **Недостаточный размер кучи.** Если живые объекты занимают бо́льшую часть кучи, то к моменту эвакуации свободных регионов просто нет.

2. **Пики humongous-аллокаций.** Серия крупных humongous-объектов может мгновенно занять все свободные регионы, не оставив to-space для эвакуации — это основная задокументированная причина evacuation failure при работающем G1.

3. **Слишком позднее начало конкурентной маркировки (высокий IHOP).** Маркировка не успевает завершиться и запустить mixed GC для освобождения старого поколения — старое поколение переполняется.

4. **Слишком медленная конкурентная маркировка.** Мало потоков маркировки не успевают обработать живые объекты до заполнения кучи.

5. **Завышенная молодёжная генерация.** Большой young region => больше объектов нужно эвакуировать за одну паузу => выше вероятность исчерпать to-space.

**Диагностика**

Включить логирование:
```
-Xlog:gc*=debug
```
Ключевые строки в логах:

| Строка | Значение |
|---|---|
| `to-space exhausted` | Evacuation failure произошёл |
| `Pause Full (Allocation Failure)` | Full GC как следствие |
| `Humongous regions: X->Y` | Рост humongous-занятости (при `gc+heap=info`) |

Full GC всегда предшествует один или несколько `to-space exhausted` — это характерный паттерн.

**Флаги для снижения вероятности**

| Флаг | Значение по умолчанию | Действие |
|---|---|---|
| `-XX:G1ReservePercent=<N>` | 10 | Резервирует N% кучи специально для to-space. Увеличение до 15–20 снижает вероятность evacuation failure, но уменьшает полезную память. |
| `-XX:InitiatingHeapOccupancyPercent=<N>` | 45 | Снижение значения запускает маркировку раньше — старое поколение успевает освободиться до того, как to-space иссякнет. |
| `-XX:ConcGCThreads=<N>` | ParallelGCThreads / 4 | Увеличение числа потоков конкурентной маркировки ускоряет её завершение. |
| `-XX:G1HeapRegionSize=<N>m` | эргономически | Увеличение размера региона снижает число humongous-объектов, один из ключевых триггеров evacuation failure. |
| `-Xmx<size>` / `-Xms<size>` | — | Увеличение максимального размера кучи (и фиксация Xms=Xmx) даёт G1 больше свободных регионов для to-space. |

**Антипаттерны, которые усугубляют проблему**

- Использование `-Xmn` или `-XX:NewRatio` с G1 — эти флаги лишают G1 возможности самостоятельно управлять размером молодого поколения. Это нарушает механизм адаптации и повышает риск evacuation failure.

**Расхождений между источниками не выявлено.** Oracle Java 8 GC Tuning Guide, Oracle Java 17 и Java 20 GC Tuning Guides, а также HeapHero G1GC Performance Tuning Guide согласуются в определении, причинах и рекомендуемых флагах. Единственное уточнение: в Java 20 документации упоминается экспериментальный флаг `-XX:+UsePreventiveGC`, который позволяет G1 превентивно запускать дополнительные young-коллекции для досрочного освобождения humongous-регионов и снижения вероятности evacuation failure — но он доступен только с `-XX:+UnlockDiagnosticVMOptions` и не рекомендуется для продакшена без тщательного тестирования.
### Вопрос 48. Что такое Code Cache в JVM? Объясните трёхсегментную структуру CodeHeap (non-method, profiled, non-profiled) в Java 9+. Что происходит при переполнении Code Cache и как это предотвратить через флаг -XX:ReservedCodeCacheSize?

**Категория:** JVM и GC · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Poonam Parhar — Why do I get message 'CodeCache is full. Compiler has been disabled'?](https://poonamparhar.github.io/codecache_is_full/)

Code Cache — это область памяти JVM, в которой хранится нативный код, сгенерированный JIT-компилятором. По умолчанию максимальный размер Code Cache составлял 48 МБ на большинстве платформ. Когда приложение компилирует большое количество методов, Code Cache может быть исчерпан.

Поведение при переполнении Code Cache менялось от версии к версии:

- **JDK 6**: флаг `UseCodeCacheFlushing` отключён по умолчанию. При заполнении Code Cache его сброс не выполняется — компиляция просто полностью и навсегда прекращается.
- **JDK 7u4+**: `UseCodeCacheFlushing` включён по умолчанию. JVM инициирует экстренный сброс (emergency flushing), удаляя «старшую половину» скомпилированных методов (nmethods), чтобы освободить место. Однако возникали проблемы: компилятор мог не перезапускаться даже после освобождения значительного объёма кэша, а сам сброс мог вызывать чрезмерную нагрузку на CPU.
- **JDK 8+**: все проблемы с экстренным сбросом Code Cache были устранены через пять целевых исправлений.

Когда Code Cache заполняется, JVM выводит в лог предупреждение: `"CodeCache is full. Compiler has been disabled."`. Компилятор отключается до тех пор, пока свободное место не превысит значение `CodeCacheMinimumFreeSpace` (по умолчанию 500 КБ).

Для предотвращения переполнения рекомендуется увеличить `ReservedCodeCacheSize` до значения, превышающего реальный объём скомпилированного кода:
```
-XX:ReservedCodeCacheSize=256m
```

#### Ответ от Claude

**Что такое Code Cache**

Code Cache — специальная область памяти JVM, предназначенная для хранения нативного кода, сгенерированного JIT-компилятором (C1 и C2). Каждый раз, когда метод становится «горячим» (достигает порогового числа вызовов), JIT компилирует его байткод в машинный код и помещает результат в Code Cache. Оттуда нативный код исполняется напрямую, без интерпретации, что даёт значительный прирост производительности.

**Трёхсегментная структура CodeHeap (Java 9+, JEP 197)**

В Java 9 согласно JEP 197 единый Code Cache был разбит на три отдельных сегмента (CodeHeap):

| Сегмент | Содержимое | Время жизни | Флаг конфигурации |
|---|---|---|---|
| **Non-method** | Внутренние структуры JVM: буферы компилятора, интерпретатор байткода, адаптеры, стабы | Постоянное (никогда не вытесняется) | `-XX:NonMethodCodeHeapSize` |
| **Profiled** | Методы, скомпилированные C1 (Tier 1–3) — с инструментацией для сбора профилировочных данных, лёгкая оптимизация | Ограниченное (до реоптимизации компилятором C2) | `-XX:ProfiledCodeHeapSize` |
| **Non-profiled** | Методы, скомпилированные C2 — полностью оптимизированный нативный код без профилирующих вставок | Потенциально постоянное | `-XX:NonProfiledCodeHeapSize` |

Сегмент **Non-method** имеет фиксированный размер около 3 МБ и остаётся неизменным. Оставшееся пространство (после вычета Non-method) распределяется поровну между сегментами Profiled и Non-profiled.

Разделение дало ряд практических преимуществ:
- Сокращение времени сканирования при GC-развёртке (sweeper пропускает Non-method сегмент).
- Снижение фрагментации высокооптимизированного кода.
- Возможность независимой настройки размеров каждого сегмента.

**Что происходит при переполнении Code Cache**

Ключевая особенность сегментированной модели: если один сегмент заполняется, а в другом ещё есть место, компилятор всё равно отключается для этого типа кода — динамическое перераспределение между сегментами не предусмотрено. JVM выводит предупреждение:

```
VM warning: CodeCache is full. The compiler has been disabled.
```

После этого все методы продолжают выполняться только в режиме интерпретации (или из уже скомпилированного кода), что приводит к значительной деградации производительности. Новые «горячие» методы перестают компилироваться.

Начиная с JDK 8, механизм экстренного сброса (`UseCodeCacheFlushing`, включён по умолчанию) позволяет JVM автоматически освобождать место, удаляя устаревшие или редко используемые nmethods, после чего компилятор может быть перезапущен.

**Диагностика и предотвращение**

Инструменты диагностики:
- `jcmd <pid> Compiler.CodeHeap_Analytics` — анализ состояния сегментов (JDK 16+).
- `-XX:+PrintCodeCache` — вывод статистики Code Cache при завершении JVM.
- `-XX:+UnlockDiagnosticVMOptions -XX:+PrintCodeCacheOnCompilation` — подробная трассировка.

Настройка размера:
```
# Увеличить общий размер Code Cache (рекомендуется >= 256 МБ для крупных приложений)
-XX:ReservedCodeCacheSize=256m

# Настройка отдельных сегментов (при наличии специфических потребностей)
-XX:NonMethodCodeHeapSize=5m
-XX:ProfiledCodeHeapSize=122m
-XX:NonProfiledCodeHeapSize=122m
```

Типовые рекомендации: установить `ReservedCodeCacheSize` равным пиковому потреблению плюс 20%, но не менее 256 МБ. Для проектов на Kotlin часто используется значение 320 МБ. При тайлированной компиляции (включена по умолчанию с Java 8) потребность в Code Cache существенно возрастает — JVM автоматически умножает базовый размер на 5.

### Вопрос 49. Что такое On-Stack Replacement (OSR) в HotSpot JVM? Как JIT-компилятор заменяет интерпретируемый код на нативный прямо во время выполнения долгого цикла, не дожидаясь выхода из метода?

**Категория:** JVM и GC · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Understanding the On Stack Replacement (OSR) optimisation in the HotSpot C1 compiler — JITWatch Wiki (AdoptOpenJDK)](https://github.com/AdoptOpenJDK/jitwatch/wiki/Understanding-the-On-Stack-Replacement-(OSR)-optimisation-in-the-HotSpot-C1-compiler)

On-Stack Replacement (OSR) — техника, позволяющая JVM заменить интерпретируемый код скомпилированным нативным кодом прямо во время выполнения метода, без ожидания его завершения и повторного вызова. Это особенно важно для долгих циклов, которые не успевают стать «горячими» по числу вызовов метода.

OSR-компиляция требует трёх ключевых компонентов:
1. **Точки входа (BCI)** — индекс байткода, с которого начинается компиляция.
2. **Захвата состояния JVM** — текущих значений локальных переменных, стека операндов и мониторов.
3. **Соглашения о вызове** — стандартного механизма передачи управления от интерпретатора к скомпилированному коду.

Процесс передачи состояния:
- Интерпретатор выделяет «OSR-буфер» (`malloc`) и копирует в него текущее состояние JVM.
- Передаёт указатель на этот буфер как единственный аргумент скомпилированному OSR-коду.
- Скомпилированный код «мигрирует» состояние из буфера в регистры и стек, затем освобождает буфер.

C1 использует подход «компиляция всего метода»: компилируется весь метод, но с добавлением специального OSR entry block в нужной точке. Это упрощает реализацию и ускоряет компиляцию. C2 использует более сложный подход с `ciTypeFlow`: поддерживает несколько точек OSR-входа и агрессивно устраняет недостижимый код.

#### Ответ от Claude

**Зачем нужен OSR**

Обычно JIT-компилятор отслеживает «горячие» методы по числу вызовов (`invocation counter`). Однако программа может содержать метод, вызываемый лишь раз, но выполняющий миллион итераций внутреннего цикла. В этом случае традиционный счётчик вызовов не сработает — метод никогда не достигнет порога компиляции, и весь цикл будет исполняться интерпретатором.

OSR решает эту проблему: он позволяет JVM «перепрыгнуть» в скомпилированный код прямо посередине работающего цикла, не дожидаясь завершения метода.

**Механизм backedge-счётчика**

HotSpot JVM ведёт два счётчика для каждого метода:
- `invocation counter` — число вызовов метода.
- `backedge counter` — число выполненных обратных переходов в цикле (loop backedge).

При каждой итерации цикла интерпретатор инкрементирует backedge counter. Как только счётчик превышает порог (`CompileThreshold` × `OnStackReplacePercentage / 100`, или `BackEdgeThreshold`), JVM запрашивает OSR-компиляцию этого метода начиная с точки входа в цикл.

**Процесс замены кода**

1. Интерпретатор продолжает выполнять итерации цикла, пока JIT-компилятор в фоне компилирует метод.
2. Компилятор создаёт специальный **OSR stub** — точку входа, адаптированную под состояние, которое может прийти от интерпретатора.
3. Интерпретатор выделяет **OSR-буфер** и копирует туда всё текущее состояние JVM: локальные переменные, стек операндов, захваченные мониторы.
4. При следующем обратном переходе цикла (backedge) управление передаётся нативному коду через OSR stub.
5. OSR stub «мигрирует» состояние из буфера в регистры и стековые слоты, предусмотренные нативным кодом.
6. Выполнение продолжается уже в нативном коде — без перезапуска метода, без потери контекста.

В логах компиляции OSR-методы отмечаются символом `%`:
```
4603 85 % b Test::test @ 2 (23 bytes)
```
Цифра после `@` — BCI (bytecode index), с которого начинается OSR-вход.

**Различия C1 и C2**

| Аспект | C1 (Client compiler) | C2 (Server compiler) |
|---|---|---|
| Подход к компиляции | Компилирует весь метод, добавляет OSR entry block | Использует `ciTypeFlow` для анализа потока управления |
| Точек OSR-входа | Одна | Может быть несколько |
| Работа с недостижимым кодом | Сохраняет (код патчится) | Агрессивно удаляет (несколько OSR-компиляций по мере прохождения по методу) |

После OSR-компиляции JVM также ставит в очередь **обычную** компиляцию всего метода. Это гарантирует, что при следующем вызове метод выполнится уже в полностью оптимизированном варианте, а не через OSR entry.

**Диагностика**

```bash
# Включить подробный лог компиляций
```

### Вопрос 50. Как работает инлайнинг (inlining) методов в JIT-компиляторе C2? Какие ограничения влияют на применимость инлайнинга (-XX:MaxInlineSize, -XX:FreqInlineSize), и почему чрезмерная глубина иерархии классов или использование интерфейсов мешает инлайнингу?

**Категория:** JVM и GC · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [JVM JIT optimization techniques — Advanced Web Machinery](https://advancedweb.hu/jvm-jit-optimization-techniques/)

Инлайнинг — JVM-оптимизация, при которой вызов метода заменяется телом вызываемого метода прямо на месте вызова. Это устраняет накладные расходы на создание стекового фрейма и открывает возможности для вторичных оптимизаций, таких как устранение общих подвыражений (Subexpression Elimination) и устранение heap-аллокаций (Heap Allocation Elimination — часть Escape Analysis), которые невозможны через границы отдельных методов.

JVM классифицирует вызовы по степени полиморфизма:
1. **Monomorphic** — только одна реализация. Легко инлайнится.
2. **Bimorphic** — две возможные реализации. Можно инлайнить с двумя ветками.
3. **Megamorphic** — более двух реализаций. Инлайнинг невозможен.

Экспериментальные результаты производительности: монофорфные вызовы примерно в два раза быстрее, чем бимофрорные и мегаморфные.

Пороговые размеры методов для инлайнинга:
- Методы меньше **35 байт** инлайнятся по умолчанию (`-XX:MaxInlineSize=35`).
- «Горячие» методы (часто вызываемые) инлайнятся при размере до **325 байт** (`-XX:FreqInlineSize=325`, платформозависимое значение).

#### Ответ от Claude

**Как работает инлайнинг в C2**

Инлайнинг заменяет инструкцию вызова метода (`invokevirtual`, `invokeinterface`, `invokestatic` и др.) непосредственно телом вызываемого метода. После подстановки тела компилятор работает с единым блоком кода, что открывает широкие возможности для дальнейших оптимизаций:

- **Constant folding** — константные выражения вычисляются при компиляции.
- **Dead code elimination** — недостижимые ветки удаляются.
- **Escape Analysis** — если объект не «убегает» за пределы инлайненного кода, он может быть выделен на стеке или вовсе устранён.
- **Loop optimizations** — цикл из вызывающего и вызываемого методов оптимизируется совместно.

В C2 инлайнинг происходит на этапе парсинга байткода и дополнительно — на этапе «инкрементального инлайнинга» (incremental/late inlining), который выполняется после парсинга и позволяет инлайнить методы, ставшие кандидатами уже в ходе оптимизации графа.

**Ограничения по размеру**

| Флаг | Значение по умолчанию | Значение |
|---|---|---|
| `-XX:MaxInlineSize` | 35 байт | Максимальный размер метода (в байткодах) для безусловного инлайнинга |
| `-XX:FreqInlineSize` | 325 байт (платформозависимо) | Максимальный размер «горячего» метода для инлайнинга |
| `-XX:MaxInlineLevel` | 9 | Максимальная глубина вложенности инлайнинга |
| `-XX:MaxRecursiveInlineLevel` | 1 | Максимальная глубина рекурсивного инлайнинга |

Метод размером более 35 байт инлайнится только если он «горячий» (часто вызывается в данном контексте) и не превышает 325 байт. Метод размером более 325 байт не инлайнится никогда, независимо от частоты вызовов. Размер учитывается в байтах байткода, а не нативного кода — это важно, так как ассерты (`assert`) считаются в размер метода даже при отключённом режиме.

**Почему интерфейсы и глубокие иерархии мешают инлайнингу**

Корень проблемы — **виртуальная диспетчеризация**. Компилятор не может инлайнить вызов, если не знает заранее, какую именно реализацию метода нужно подставить.

**Полиморфные встроенные кэши (Polymorphic Inline Cache, PIC)**

HotSpot использует PIC для отслеживания типов, реально встречавшихся в данном call site во время выполнения. На основе этой информации вызовы классифицируются:

- **Monomorphic call site**: встречался ровно один тип получателя. C2 генерирует type guard (проверку типа) и инлайнит единственную реализацию. Если guard не срабатывает — fallback через vtable.
- **Bimorphic call site**: встречались два типа. C2 генерирует два guard + два инлайненных тела.
- **Megamorphic call site**: три и более типов. PIC переходит в режим VMT dispatch (через vtable stub), инлайнинг не применяется.

**Интерфейсные вызовы (`invokeinterface`)**

Вызовы через интерфейс (Java Interfaces, `invokeinterface`) исторически сложнее для оптимизации, чем виртуальные вызовы через vtable (`invokevirtual`). Интерфейс не задаёт конкретное место в vtable — компилятор должен искать реализацию через itable (interface table). Однако JIT применяет те же эвристики PIC: если в конкретном call site всегда встречался один тип — вызов мегаморфным не является и может быть инлайнен.

**Глубина иерархии классов**

Глубокая иерархия наследования увеличивает вероятность того, что в конкретном call site окажется много разных реализаций (разные подклассы). Чем больше подклассов реально используется в runtime — тем выше шанс перехода call site в состояние megamorphic, что делает инлайнинг невозможным.

Дополнительный фактор: при наличии нескольких подклассов CHA (Class Hierarchy Analysis) не может статически гарантировать единственность реализации, поэтому C2 не вправе применять агрессивный инлайнинг без runtime-проверок.

**Практические рекомендации**

- Избегайте глубоких иерархий и большого числа реализаций одного интерфейса в критических по производительности местах.
- Используйте `final` классы и методы там, где это оправдано: компилятор точно знает, что других реализаций нет.
- Применяйте `-XX:+PrintInlining` (с `-XX:+UnlockDiagnosticVMOptions`) для диагностики: в выводе будут сообщения `"too big"`, `"callee is too large"`, `"megamorphic"`.
- Помните, что искусственное дробление методов ниже 35 байт ради инлайнинга контрпродуктивно — ухудшается читаемость кода, а компилятор сам справляется с инлайнингом часто вызываемых методов чуть большего размера.

### Вопрос 51. Что такое Compressed Class Space и как он соотносится с Metaspace? Чем управляет флаг -XX:CompressedClassSpaceSize и что происходит при его переполнении?

**Категория:** JVM и GC · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Understand the OutOfMemoryError Exception — Oracle Java 8 Troubleshooting Guide](https://docs.oracle.com/javase/8/docs/technotes/guides/troubleshoot/memleaks002.html)

На 64-битных платформах, когда включён флаг `-XX:+UseCompressedClassPointers`, указатели на метаданные классов хранятся не как полные 64-битные адреса, а как 32-битные смещения. Для хранения этих смещений JVM выделяет специальную область нативной памяти — **Compressed Class Space**.

Существует два типа метаданных классов:
1. **klass-метаданные** — структуры типа `Klass` (внутреннее представление класса в HotSpot). Они хранятся в пространстве, ограниченном параметром `CompressedClassSpaceSize`.
2. **Прочие метаданные** — байткод методов, пулы констант, аннотации и другие данные. Они хранятся в **Metaspace**.

Размер Compressed Class Space задаётся флагом `-XX:CompressedClassSpaceSize` и по умолчанию равен 1 ГБ. Пространство резервируется через `mmap` при старте JVM и заполняется по мере загрузки классов.

При попытке задать недопустимое значение (например, `-XX:CompressedClassSpaceSize=4g`) JVM откажется запускаться:

```
CompressedClassSpaceSize of 4294967296 is invalid;
must be between 1048576 and 3221225472.
```

Если в процессе работы Compressed Class Space исчерпывается (загружается слишком много классов или происходит утечка через ClassLoader), JVM выбрасывает:

```
java.lang.OutOfMemoryError: Compressed class space
```

Способы устранения:
- Увеличить размер: `-XX:CompressedClassSpaceSize=<новый_размер>`
- Отключить сжатые указатели: `-XX:-UseCompressedClassPointers` (при этом освобождения Compressed Class Space как такового не происходит — указатели становятся 64-битными)

#### Ответ от Claude

**Архитектура памяти классов в HotSpot**

В 64-битной JVM метаданные загружённых классов хранятся в нативной памяти (вне Java-кучи). Начиная с Java 8 (после удаления PermGen), эта область называется **Metaspace**. Однако при включённых compressed class pointers Metaspace логически делится на две независимые зоны:

| Зона | Что хранит | Управляющий флаг |
|------|-----------|-----------------|
| **Non-class Metaspace** | Байткод методов, пулы констант, аннотации, дескрипторы полей | `-XX:MaxMetaspaceSize` |
| **Compressed Class Space** | Структуры `Klass` (C++-объект, описывающий тип в HotSpot) | `-XX:CompressedClassSpaceSize` |

Разделение обусловлено оптимизацией: чтобы 32-битный compressed class pointer (`_klass` в объектном заголовке) указывал на `Klass`-структуру, все `Klass`-объекты должны располагаться в едином непрерывном регионе, начало которого известно JVM заранее. Этот регион и есть Compressed Class Space — он резервируется одним блоком через `mmap` в начале работы JVM.

**Флаг `-XX:CompressedClassSpaceSize`**

- Определяет размер зарезервированного региона для `Klass`-структур.
- Значение по умолчанию: **1 ГБ** (резервируется виртуальная память, физически выделяется по мере необходимости).
- Допустимый диапазон: от 1 МБ (1 048 576 байт) до 3 ГБ (3 221 225 472 байта) — ограничен размером 32-битного адресного пространства смещений.
- Активен только при `-XX:+UseCompressedClassPointers` (включён по умолчанию на 64-битных JVM с Java heap < 32 ГБ).

**Соотношение с `-XX:MaxMetaspaceSize`**

`-XX:MaxMetaspaceSize` ограничивает **суммарный** объём committed-памяти для обеих зон: committed Compressed Class Space + committed non-class Metaspace. Это важно: можно исчерпать MaxMetaspaceSize даже если в каждой из зон по отдельности ещё есть место.

**Что происходит при переполнении**

Если загружается больше классов, чем помещается в Compressed Class Space, JVM выбрасывает:

```
java.lang.OutOfMemoryError: Compressed class space
```

Типичные причины:
- Утечка `ClassLoader` — динамически порождаемые загрузчики (например, в OSGi, JRebel, Groovy-скриптах) не выгружаются и продолжают накапливать `Klass`-структуры.
- Генерация кода в рантайме (через ASM, ByteBuddy, cglib) без явного управления жизненным циклом загрузчика.

**Диагностика и решение**

1. Убедиться, что речь именно о Compressed class space (не о non-class Metaspace): сообщение OOM должно содержать текст `Compressed class space`.
2. Включить NMT (`-XX:NativeMemoryTracking=detail`) и проверить категорию `Class`.
3. Увеличить лимит: `-XX:CompressedClassSpaceSize=2g` (но не более 3 ГБ).
4. Отключить UseCompressedClassPointers (`-XX:-UseCompressedClassPointers`) — тогда `Klass`-указатели становятся 64-битными, Compressed Class Space не нужен, но потребление памяти объектами немного возрастёт.
5. Найти и устранить утечку ClassLoader с помощью heap dump + анализатора (Eclipse MAT, JProfiler).

**Расхождения между источниками:** Oracle Troubleshooting Guide (Java 8) описывает флаг как управляющий «пространством для 32-битных смещений класс-указателей», тогда как более подробное описание архитектуры (разделение на non-class Metaspace и Class Space) дано в GC Tuning Guide (Java 11+). Оба документа согласны в том, что `MaxMetaspaceSize` охватывает обе зоны суммарно.

### Вопрос 52. Как использовать Native Memory Tracking (NMT) для диагностики утечек нативной памяти в JVM? Объясните порядок работы: включение, снятие baseline, сравнение снимков через jcmd VM.native_memory. Какие категории памяти отслеживает NMT?

**Категория:** JVM и GC · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Native Memory Tracking — Oracle Java 11 VM Guide](https://docs.oracle.com/en/java/javase/11/vm/native-memory-tracking.html)

Native Memory Tracking (NMT) — функция HotSpot JVM для отслеживания внутреннего потребления нативной памяти. Включается параметром командной строки:

```
-XX:NativeMemoryTracking=[off | summary | detail]
```

- `off` — отключено (по умолчанию).
- `summary` — агрегированные данные по подсистемам.
- `detail` — данные по отдельным точкам аллокации (call sites).

Включение NMT снижает производительность JVM на **5–10%**, а каждый malloc-заголовок получает дополнительные 2 слова памяти для учёта.

**Доступ к данным через jcmd:**

```bash
jcmd <pid> VM.native_memory [команда] [scale=KB|MB|GB]
```

Доступные команды:

| Команда | Описание |
|---------|----------|
| `summary` | Сводный отчёт по категориям |
| `detail` | Подробный отчёт + карта виртуальной памяти + отчёт по call sites |
| `baseline` | Создать снимок для последующего сравнения |
| `summary.diff` | Сравнение текущего сводного отчёта с baseline |
| `detail.diff` | Сравнение текущего детального отчёта с baseline |
| `shutdown` | Отключить NMT (повторно включить без перезапуска JVM нельзя) |

**Порядок диагностики утечки:**

1. Запустить JVM с NMT: `java -XX:NativeMemoryTracking=detail MyApp`
2. Дать приложению прогреться.
3. Зафиксировать baseline: `jcmd <pid> VM.native_memory baseline`
4. Дать приложению поработать под нагрузкой.
5. Снять diff: `jcmd <pid> VM.native_memory summary.diff scale=MB`
6. При необходимости детализировать: `jcmd <pid> VM.native_memory detail.diff`
7. Повторять шаги 4–6 периодически, наблюдая за приростом в конкретных категориях.

Для захвата статистики при завершении JVM: `-XX:+UnlockDiagnosticVMOptions -XX:+PrintNMTStatistics`.

**Ограничения:** NMT не отслеживает аллокации из сторонних нативных библиотек и из JDK class libraries. Для таких случаев используются OS-инструменты (например, `pmap`, `valgrind`, `jemalloc`).

#### Ответ от Claude

**Что такое NMT и зачем он нужен**

JVM потребляет нативную (не-heap) память для множества собственных нужд: хранения метаданных классов, стеков потоков, JIT-кеша, данных GC и прочего. Штатные JVM-инструменты (jstat, heap dump) не показывают эту область. NMT — встроенный механизм, который инструментирует внутренние `malloc`/`mmap` вызовы HotSpot и раскладывает аллокации по категориям.

**Включение**

NMT включается только при старте JVM — во время работы изменить уровень отслеживания нельзя:

```bash
# Только суммарные данные (меньше overhead)
java -XX:NativeMemoryTracking=summary -jar myapp.jar

# Детальные данные с call sites (больше overhead, нужен для точного поиска источника утечки)
java -XX:NativeMemoryTracking=detail -jar myapp.jar
```

**Порядок диагностики нативной утечки**

```bash
# 1. Прогреть приложение (дать JIT и ClassLoader завершить основную работу)

# 2. Снять baseline — точка отсчёта
jcmd <pid> VM.native_memory baseline

# 3. Выдержать интервал (минуты или часы — в зависимости от скорости утечки)

# 4. Посмотреть сводные изменения
jcmd <pid> VM.native_memory summary.diff scale=MB

# 5. При обнаружении подозрительного роста — детализировать
jcmd <pid> VM.native_memory detail.diff scale=MB

# 6. Повторять шаги 3–5 для подтверждения тренда
```

В выводе `summary.diff` каждое число со знаком `+`/`-` показывает изменение относительно baseline:

```
-                     Class (reserved=1118333KB +47579KB, committed=117949KB +89963KB)
                                (classes #68532 +58527)
                                (malloc=8317KB +2523KB #5461 +3371)
                                (mmap: reserved=1110016KB +45056KB, committed=109632KB +87440KB)
```

Если `Class` или `Internal` стабильно растёт — это индикатор возможной утечки через ClassLoader или внутренние буферы.

**Категории памяти NMT**

Согласно документации Oracle (Java 21):

| Категория | Что включает |
|-----------|-------------|
| **Java Heap** | Java-куча — объекты приложения |
| **Class** | Метаданные загруженных классов (Metaspace + Compressed Class Space) |
| **Thread** | Стеки потоков, данные структур потоков (resource area, handle area) |
| **Code** | JIT-скомпилированный нативный код (Code Cache) |
| **GC** | Внутренние структуры GC: card table, remembered sets (кроме G1 card set) |
| **GCCardSet** | Card set данные G1 GC (только G1, Java 17+) |
| **Compiler** | Рабочие буферы JIT-компилятора |
| **Internal** | Данные, не входящие в другие категории: JVMTI, системные свойства, парсер аргументов |
| **Other** | Прочие аллокации |
| **Symbol** | Таблица символов (имена классов, методов, полей) |
| **Native Memory Tracking** | Накладные расходы самого NMT |
| **Arena Chunk** | Блоки в пуле арен |
| **Logging** | Внутреннее логирование JVM |
| **Arguments** | Аргументы командной строки |
| **Module** | Данные системы модулей (JPMS) |

**Разница между reserved и committed**

- `reserved` — виртуальная память, зарезервированная через `mmap` (не занимает RAM).
- `committed` — физически выделенная память, которую JVM реально использует.

При диагностике утечки нужно следить за ростом именно `committed`.

**Ограничения NMT**

- NMT **не видит** аллокации из нативных библиотек, подключённых через JNI (например, JNI-код, Netty epoll, SQLite JDBC). Для таких случаев используют `pmap`, `jemalloc` с профилировщиком или `Async-Profiler` в режиме malloc.
- NMT **не видит** аллокации из JDK class libraries (например, DirectByteBuffer через `java.nio`), кроме случаев, когда HotSpot сам управляет этой памятью через `os::malloc`.
- Уровень отслеживания нельзя изменить на ходу; `jcmd VM.native_memory shutdown` лишь останавливает сбор данных, но не позволяет перезапустить NMT без рестарта JVM.
- Включение NMT даёт 5–10% overhead по производительности и ~2 слова дополнительной памяти на каждый `malloc`.

**Расхождения между источниками:** Документация Java 8 (docs.oracle.com/javase/8) перечисляет меньший набор категорий (без `GCCardSet`, `Logging`, `Module`, `Arguments`). Эти категории добавлялись в более поздних версиях. Документация Java 21 (docs.oracle.com/en/java/javase/21) содержит полный актуальный список — на неё следует ориентироваться в современных проектах.

### Вопрос 53. Как работает Shenandoah GC? Что такое Brooks pointer (forwarding pointer) и как он позволяет перемещать объекты конкурентно? Чем Shenandoah отличается от G1 и ZGC?

**Категория:** JVM и GC · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [JEP 189: Shenandoah: A Low-Pause-Time Garbage Collector (Experimental) — openjdk.org](https://openjdk.org/jeps/189)

Shenandoah — регион-ориентированный сборщик мусора, выполняющий **конкурентное уплотнение** (concurrent compaction) в дополнение к конкурентной маркировке. Ключевой механизм — **Brooks pointer**: дополнительное слово-указатель, добавляемое к каждому объекту. В обычном состоянии Brooks pointer указывает на сам объект; при перемещении объекта указатель обновляется на новый адрес — так все обращения через старый указатель автоматически перенаправляются.

Для корректного одновременного доступа потоков приложения и GC-потоков при эвакуации Shenandoah использует **read и write barriers**, обеспечивающих инвариант «to-space»: поток приложения всегда видит актуальный адрес объекта. Барьер проверяет Brooks pointer при каждом чтении ссылки и при необходимости перенаправляет доступ.

Цикл Shenandoah включает короткие STW-паузы: Init Mark (инициализация маркировки и сканирование корней) и Final Mark (завершение маркировки), а также Init Update References и Final Update References. Все ресурсоёмкие фазы — конкурентные: Concurrent Mark, Concurrent Cleanup, Concurrent Evacuation, Concurrent Update References.

Shenandoah не использует remembered sets в отличие от G1. Вместо этого каждый регион отслеживается независимо. Это упрощает управление регионами, но исключает генерационную сборку (до появления Generational Shenandoah). Сборщик стал experimental в JDK 12 (JEP 189) и production-ready в JDK 15 (JEP 379).

#### Ответ от Claude

**Ключевая идея Shenandoah**

Shenandoah — регион-ориентированный сборщик с целью пауз менее 10 мс независимо от размера heap. В отличие от G1, который выполняет эвакуацию (перемещение объектов) в STW-паузе, Shenandoah делает это **конкурентно** с работающим приложением.

**Brooks pointer (forwarding pointer)**

Для конкурентного перемещения каждый объект в heap получает дополнительное слово — **Brooks pointer** (по имени автора техники, Rodney Brooks):

- В обычном состоянии: `brooks_ptr → сам объект` (самоссылка).
- При конкурентной эвакуации: объект копируется в новое место, затем атомарно (через CAS) обновляется `brooks_ptr → новое местоположение`.
- Потоки приложения, обращающиеся к старому адресу, прозрачно перенаправляются через Brooks pointer на новую копию.
- Финальное обновление всех ссылок происходит в фазе Concurrent Update References.

Overhead Brooks pointer: +1 слово (8 байт на 64-битной JVM) на каждый объект.

**Фазы GC-цикла Shenandoah**

| Фаза | Тип | Описание |
|------|-----|----------|
| Init Mark | STW (короткая) | Инициализация маркировки, сканирование GC-корней |
| Concurrent Mark | Конкурентная | Трассировка живых объектов по всему heap |
| Final Mark | STW (короткая) | Завершение маркировки, формирование Collection Set |
| Concurrent Cleanup | Конкурентная | Освобождение полностью пустых регионов |
| Concurrent Evacuation | Конкурентная | Копирование объектов из Collection Set в новые регионы |
| Init Update References | STW (короткая) | Инициализация фазы обновления ссылок |
| Concurrent Update References | Конкурентная | Обновление всех старых ссылок на новые адреса объектов |
| Final Update References | STW (короткая) | Завершение обновления корневых ссылок |

**Сравнение Shenandoah с G1 и ZGC**

| Аспект | G1 | Shenandoah | ZGC |
|--------|-----|------------|-----|
| Concurrent compaction | Нет (STW) | Да | Да |
| Механизм relocate | Evacuation Pause (STW) | Brooks pointer + read/write barriers | Colored pointer + load barrier |
| Remembered Sets | Да (per-region RSet) | Нет | Нет |
| Генерационность | Да | Нет (Generational Shenandoah — экспериментально) | Generational ZGC (Java 21+) |
| Overhead на объект | Нет | +1 слово (Brooks pointer) | Метабиты в указателе |
| Паузы | 50–200 мс | < 10 мс | < 1 мс |
| Production-ready | Java 9 (default) | Java 15 (JEP 379) | Java 15 (JEP 377) |

**Включение Shenandoah:**
```
-XX:+UseShenandoahGC
```

**Расхождения между источниками:** Red Hat Developer Blog отмечает, что Generational Shenandoah остаётся экспериментальным на части дистрибутивов, тогда как Generational ZGC уже production-ready в Java 21+. JEP 189 и OpenJDK Wiki согласуются в описании фаз и механизма Brooks pointer.

- [JEP 379: Shenandoah (Production) — openjdk.org](https://openjdk.org/jeps/379)

### Вопрос 54. Как работает Serial GC и Parallel GC? Чем они отличаются, когда каждый из них предпочтителен и каковы их ключевые JVM-флаги?

**Категория:** JVM и GC · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [JVM Garbage Collectors — Baeldung](https://www.baeldung.com/jvm-garbage-collectors)

**Serial GC** — простейшая реализация: сборка мусора выполняется в **одном потоке**. Все потоки приложения останавливаются на время сборки (stop-the-world). Включается флагом `-XX:+UseSerialGC`. Подходит для однопроцессорных машин и приложений с небольшим heap.

**Parallel GC** (также называемый Throughput Collector) отличается тем, что использует **несколько потоков** для сборки мусора, что уменьшает время пауз на многоядерных системах. Оба поколения (Young и Old) собираются параллельно. Приложение всё равно останавливается на время GC-паузы, но несколько потоков GC работают одновременно. Parallel GC был **сборщиком по умолчанию с Java 5 до Java 8**. Включается флагом `-XX:+UseParallelGC`.

Настраиваемые параметры Parallel GC:
- `-XX:ParallelGCThreads=<N>` — число потоков GC (по умолчанию: число процессоров)
- `-XX:MaxGCPauseMillis=<N>` — целевое максимальное время паузы
- `-XX:GCTimeRatio=<N>` — целевое соотношение времени приложения к времени GC

#### Ответ от Claude

**Serial GC**

Serial GC — самый простой сборщик мусора в JVM. Все фазы (Minor GC в Young Generation и Major/Full GC в Old Generation) выполняются **последовательно в одном потоке** с полной остановкой всех потоков приложения.

Алгоритм работы:
- **Young Generation**: Copying collector — живые объекты из Eden и Survivor копируются в другой Survivor или Old Generation.
- **Old Generation**: Mark-Sweep-Compact — маркировка живых объектов, удаление мёртвых, уплотнение heap для устранения фрагментации.

Когда применять:
- Однопроцессорные машины (только один ЦП не создаёт накладных расходов на многопоточность).
- Приложения с heap < 100 МБ.
- Batch-задачи, не чувствительные к задержкам.
- Контейнеры с ограниченными CPU-ресурсами (особенно если выделен 1 vCPU).

Флаг: `-XX:+UseSerialGC`

**Parallel GC (Throughput Collector)**

Parallel GC использует **несколько параллельных потоков** для выполнения GC, что ускоряет сборку на многоядерных системах. Главная цель — максимизировать пропускную способность приложения (throughput), жертвуя предсказуемостью пауз.

Особенности:
- Young Generation собирается параллельно (как и в Serial GC — copying, но несколькими потоками).
- Old Generation собирается параллельно: Mark-Sweep-Compact выполняется в нескольких потоках одновременно. До Java 7u4 Old Generation собирался только в 1 поток; начиная с Java 7u4 `-XX:+UseParallelGC` автоматически включает `-XX:+UseParallelOldGC`.
- Всё по-прежнему stop-the-world — приложение стоит во время GC.

**Сравнительная таблица**

| Характеристика | Serial GC | Parallel GC |
|---|---|---|
| Число потоков GC | 1 | N (по умолчанию = CPU count) |
| Young Gen алгоритм | Copying, 1 поток | Copying, N потоков |
| Old Gen алгоритм | Mark-Sweep-Compact, 1 поток | Mark-Sweep-Compact, N потоков |
| Оптимизация под | Минимальный footprint | Максимальный throughput |
| Применение | Одноядерные, малые heap | Batch-задачи, научные расчёты |
| Default в | Всегда доступен | Java 5 – Java 8 |
| Флаг | `-XX:+UseSerialGC` | `-XX:+UseParallelGC` |

**GC Ergonomics в Parallel GC**

Parallel GC поддерживает поведенческую настройку (behavior-based tuning):
- `-XX:MaxGCPauseMillis=<N>` — подсказка GC стремиться к паузам не длиннее N мс (более мелкий Young Gen).
- `-XX:GCTimeRatio=<N>` — целевая доля времени в GC: `1 / (1 + N)`. При `GCTimeRatio=19` GC может занимать не более 5% времени. По умолчанию: 99 (не более ~1% на GC).
- `-XX:+UseAdaptiveSizePolicy` (включено по умолчанию) — автоматически подбирает размеры Young Gen, Old Gen и tenuring threshold для достижения целей.

**Источники:**
- [JVM Garbage Collectors — Baeldung](https://www.baeldung.com/jvm-garbage-collectors)
- [5 Available Collectors — Oracle Java 9 GC Tuning Guide](https://docs.oracle.com/javase/9/gctuning/available-collectors.htm)
- [2 Ergonomics — Oracle Java 8 GC Tuning Guide](https://docs.oracle.com/javase/8/docs/technotes/guides/vm/gctuning/ergonomics.html)

### Вопрос 55. Что такое GC ergonomics в JVM? Как JVM автоматически выбирает сборщик мусора и настраивает размер heap? Объясните поведенческие цели: -XX:GCTimeRatio, -XX:MaxGCPauseMillis и флаг -XX:+UseAdaptiveSizePolicy.

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [2 Ergonomics — Oracle Java 8 GC Tuning Guide (docs.oracle.com)](https://docs.oracle.com/javase/8/docs/technotes/guides/vm/gctuning/ergonomics.html)

**Ergonomics** — процесс, при котором JVM и GC автоматически оптимизируют производительность приложения без необходимости задавать параметры вручную. JVM устанавливает платформо-зависимые значения по умолчанию для: сборщика мусора, размера heap и компилятора.

**Автоматический выбор конфигурации:** Если машина является server-class (2+ физических процессора, 2+ ГБ ОЗУ), JVM по умолчанию выбирает: Throughput GC (Parallel), начальный heap = 1/64 физической памяти, максимальный heap = 1/4 физической памяти, серверный компилятор (C2). Исключение: 32-битная Windows — там используется Client VM.

**Поведенческие цели (behavior-based tuning):**

- **Throughput goal** (`-XX:GCTimeRatio=<N>`): желаемое соотношение времени GC к времени приложения = `1 / (1 + N)`. По умолчанию N=99, то есть GC занимает менее 1% времени.
- **Pause time goal** (`-XX:MaxGCPauseMillis=<N>`): максимальная длительность GC-паузы (в мс). GC будет уменьшать размеры поколений, чтобы укладываться в цель. Целевое время паузы имеет приоритет над пропускной способностью.

**-XX:+UseAdaptiveSizePolicy** (включён по умолчанию с Parallel GC): JVM автоматически подбирает размеры Young и Old поколений, чтобы выполнить заданные цели. Порядок приоритетов: сначала цель по паузам → затем цель по пропускной способности → затем минимизация footprint.

#### Ответ от Claude

**Что такое GC Ergonomics**

GC Ergonomics — механизм автоматической настройки JVM, позволяющий добиваться хорошей производительности без ручного задания всех параметров. Ergonomics охватывает:
1. Автоматический выбор GC-алгоритма.
2. Автоматический выбор начального и максимального размера heap.
3. Динамическую адаптацию размеров поколений в процессе работы (`UseAdaptiveSizePolicy`).

**Автоматический выбор GC**

JVM определяет тип машины при запуске:

| Условие | Тип машины | Умолчания |
|---------|------------|-----------|
| 2+ CPU и 2+ ГБ RAM | Server-class | Parallel GC (Java 5–8), G1 GC (Java 9+), начальный heap 1/64 RAM, макс. heap 1/4 RAM |
| Иначе | Client-class | Serial GC, меньший heap |

**Поведенческие цели**

**1. Throughput Goal (`-XX:GCTimeRatio=<N>`)**

Задаёт желаемую долю времени GC от общего времени работы JVM:

```
доля_GC = 1 / (1 + N)
```

Примеры:
- `GCTimeRatio=99` (по умолчанию) → GC занимает не более ~1% времени.
- `GCTimeRatio=19` → не более ~5% времени на GC.

Если цель не достигается — JVM увеличивает размеры поколений (больше heap → реже GC → больше времени на приложение).

**2. Pause Time Goal (`-XX:MaxGCPauseMillis=<N>`)**

Подсказка GC: паузы не должны превышать N мс. JVM уменьшает размеры поколений для частых, но коротких сборок. Более мелкий Young Gen — более частые Minor GC с короткими паузами.

**Приоритет целей** (в порядке убывания):
1. Pause time goal (приоритет 1)
2. Throughput goal (приоритет 2)
3. Footprint (минимальный heap, если обе цели выполнены)

**`-XX:+UseAdaptiveSizePolicy`**

Включён по умолчанию совместно с Parallel GC. Алгоритм на каждой GC-итерации:

1. Если пауза > `MaxGCPauseMillis` → уменьшить размер поколений.
2. Если пауза в норме, но throughput < цели → увеличить размеры поколений.
3. Если обе цели выполнены → уменьшить heap для экономии памяти.

Автоматически корректируются: размер Eden, размер Survivor, `TenuringThreshold`.

**Автоматический sizing Heap**

Heap растёт и сжимается в рамках `[-Xms, -Xmx]` согласно параметрам:
- `-XX:MinHeapFreeRatio=40` (по умолчанию): если свободного heap < 40% → heap расширяется.
- `-XX:MaxHeapFreeRatio=70` (по умолчанию): если свободного heap > 70% → heap сужается.

**Практические рекомендации:**
- Зафиксируйте `-Xms = -Xmx` в production для предотвращения дорогих операций расширения/сужения heap.
- Не задавайте `MaxGCPauseMillis` для Parallel GC без понимания компромисса: уменьшение пауз → рост частоты GC → потенциальное снижение throughput.
- G1 и ZGC имеют собственные механизмы адаптации (Adaptive IHOP у G1), похожие на ergonomics по принципу.

**Источники:**
- [2 Ergonomics — Oracle Java 8 GC Tuning Guide](https://docs.oracle.com/javase/8/docs/technotes/guides/vm/gctuning/ergonomics.html)
- [4 Sizing the Generations — Oracle Java 8 GC Tuning Guide](https://docs.oracle.com/javase/8/docs/technotes/guides/vm/gctuning/sizing.html)
- [Garbage Collection Ergonomics — Oracle Java 7](https://docs.oracle.com/javase/7/docs/technotes/guides/vm/gc-ergonomics.html)

### Вопрос 56. Как создать heap dump в Java? Опишите все основные способы: jmap, jcmd, -XX:+HeapDumpOnOutOfMemoryError, программный вызов через HotSpotDiagnosticMXBean.

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Different Ways to Capture Java Heap Dumps — Baeldung](https://www.baeldung.com/java-heap-dump-capture)

Существует несколько способов создания heap dump в Java:

**1. jmap** — инструмент командной строки для получения снимков памяти запущенной JVM:
```
jmap -dump:live,format=b,file=<путь_к_файлу> <pid>
```
Параметр `live` позволяет выгрузить только живые объекты (перед дампом выполняется GC). `format=b` — бинарный формат HPROF. Примечание: jmap помечен как экспериментальный и не поддерживается официально.

**2. jcmd** — предпочтительный современный инструмент:
```
jcmd <pid> GC.heap_dump <путь_к_файлу>
```

**3. -XX:+HeapDumpOnOutOfMemoryError** — автоматический дамп при `OutOfMemoryError`:
```
java -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/dump.hprof MyApp
```
Не добавляет overhead при нормальной работе. Рекомендуется всегда включать в production.

**4. Программный вызов через `HotSpotDiagnosticMXBean`:**
```java
MBeanServer server = ManagementFactory.getPlatformMBeanServer();
HotSpotDiagnosticMXBean mxBean = ManagementFactory.newPlatformMXBeanProxy(
    server, "com.sun.management:type=HotSpotDiagnostic",
    HotSpotDiagnosticMXBean.class);
mxBean.dumpHeap(filePath, live);
```

Все методы создают файл в **HPROF-формате** (бинарный), который можно открыть в Eclipse MAT, VisualVM, JProfiler и других инструментах. Файл `.hprof` нельзя перезаписать — при повторном создании нужно указать другое имя.

#### Ответ от Claude

**Что такое heap dump и зачем он нужен**

Heap dump — бинарный снимок состояния Java heap в конкретный момент времени. Содержит все живые объекты, их поля, ссылки между ними и метаданные классов. Используется для диагностики утечек памяти, анализа `OutOfMemoryError` и оптимизации потребления памяти.

Формат файла — **HPROF** (бинарный). Анализируется инструментами: Eclipse MAT, VisualVM, JProfiler, JDK Mission Control.

**Способы создания heap dump**

**1. jmap (инструмент командной строки)**

```bash
# Дамп только живых объектов (перед дампом выполняется Full GC)
jmap -dump:live,format=b,file=/tmp/heap.hprof <pid>

# Дамп всех объектов (включая недостижимые)
jmap -dump:format=b,file=/tmp/heap.hprof <pid>
```

Предупреждение: `jmap` помечен как экспериментальный и не поддерживается официально. На некоторых платформах требует прав суперпользователя. В production предпочтительнее использовать `jcmd`.

**2. jcmd (предпочтительный способ)**

```bash
jcmd <pid> GC.heap_dump /tmp/heap.hprof
```

`jcmd` — официально поддерживаемый инструмент JDK, заменяющий `jmap` в большинстве сценариев. Требует, чтобы JVM и jcmd запускались от одного пользователя (или root).

**3. -XX:+HeapDumpOnOutOfMemoryError (рекомендуется для production)**

```bash
java -XX:+HeapDumpOnOutOfMemoryError \
     -XX:HeapDumpPath=/var/log/app/heap-%p.hprof \
     -jar myapp.jar
```

- Дамп создаётся автоматически при `OutOfMemoryError` (в момент возникновения).
- Нет overhead при нормальной работе — рекомендуется всегда включать в production.
- `%p` в пути подставляет PID процесса для уникальности имени файла.
- Если путь — директория (`-XX:HeapDumpPath=/var/log/app/`), JVM сама назначит имя файла.

**4. Программный вызов через HotSpotDiagnosticMXBean**

```java
import com.sun.management.HotSpotDiagnosticMXBean;
import java.lang.management.ManagementFactory;
import javax.management.MBeanServer;

public static void dumpHeap(String filePath, boolean live) throws Exception {
    MBeanServer server = ManagementFactory.getPlatformMBeanServer();
    HotSpotDiagnosticMXBean mxBean = ManagementFactory.newPlatformMXBeanProxy(
        server,
        "com.sun.management:type=HotSpotDiagnostic",
        HotSpotDiagnosticMXBean.class
    );
    mxBean.dumpHeap(filePath, live); // live=true: только достижимые объекты
}
```

Используется, когда нужно создать дамп из кода приложения (например, при обнаружении подозрительного роста памяти через JMX-мониторинг). `live=true` выполняет GC перед дампом.

**5. VisualVM / JConsole (графические инструменты)**

В VisualVM: правый клик на процессе → «Heap Dump». Дамп открывается прямо в интерфейсе инструмента для интерактивного анализа. Удобно для разработки, не подходит для production-серверов без GUI.

**Сравнение способов**

| Способ | Когда использовать | Overhead |
|--------|-------------------|---------|
| `HeapDumpOnOutOfMemoryError` | Production, автоматический дамп при OOM | Нет (до OOM) |

### Вопрос 57. Как анализировать heap dump с помощью Eclipse Memory Analyzer (MAT)? Объясните понятия: Dominator Tree, Leak Suspects Report, Retained Heap vs Shallow Heap, OQL.

**Категория:** JVM и GC · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Tool to Analyze Offline Java Heap Dumps — Baeldung](https://www.baeldung.com/java-analyze-offline-heap-dumps)

**Eclipse Memory Analyzer (MAT)** — индустриальный стандарт для анализа Java heap dumps. Работает с файлами `.hprof` любого размера.

**Dominator Tree (дерево доминаторов)** — ключевое представление, показывающее, какие объекты «удерживают» память. Объект A доминирует объект B, если любой путь от GC-корней до B проходит через A. Если удалить A из heap, все доминируемые им объекты тоже станут недостижимы. Dominator Tree позволяет найти объекты с максимальным «retained heap» — корень утечки.

**Leak Suspects Report** — автоматический анализ, который находит объекты, удерживающие значительный процент heap (по умолчанию > 10%). Для каждого подозреваемого строится цепочка ссылок до GC-корней («Path to GC Roots»).

**Shallow Heap** — непосредственный размер объекта (только его поля, без учёта объектов, на которые он ссылается). **Retained Heap** — размер всей памяти, которая была бы освобождена при удалении данного объекта (включая все доминируемые объекты).

**OQL (Object Query Language)** — SQL-подобный язык запросов к heap dump. Пример: `SELECT * FROM java.util.ArrayList WHERE size > 1000` — найти все ArrayList с более чем 1000 элементов.

#### Ответ от Claude

**Что такое Eclipse MAT**

Eclipse Memory Analyzer Tool (MAT) — инструмент для анализа heap dump (.hprof). Поддерживает дампы объёмом в десятки гигабайт. Доступен как standalone-приложение и как плагин для Eclipse IDE.

**Ключевые концепции**

**Shallow Heap vs Retained Heap**

- **Shallow Heap** — размер самого объекта: только его поля и заголовок. Для `Object[]` размером 100 элементов shallow heap — это только размер массива ссылок, без учёта самих объектов.
- **Retained Heap** — суммарная память, которая была бы освобождена GC, если бы данный объект стал недостижимым. Это shallow heap объекта + shallow heap всех объектов, достижимых только через него (его «доминируемое поддерево»).

Пример: `HashMap` (shallow: 48 байт) может иметь retained heap в сотни МБ, если удерживает тысячи объектов.

**Dominator Tree**

Дерево доминаторов показывает иерархию объектов по retained heap. Объект A доминирует объект B, если единственный путь от GC-корней до B проходит через A. Дерево позволяет ответить на вопрос: «Кто держит эту память?»

В MAT: меню **Window → Heap Dump Overview → Dominator Tree** или **Actions → Dominator Tree**.

**Leak Suspects Report**

Автоматический отчёт, запускаемый при открытии дампа (или вручную: **Reports → Leak Suspects**). MAT анализирует объекты с наибольшим retained heap (> 10% общего heap по умолчанию) и строит для каждого:
- Описание подозреваемого объекта.
- Кратчайший путь от GC-корней («Path to GC Roots», исключая слабые/фантомные ссылки).
- Список объектов в retained set.

**OQL (Object Query Language)**

SQL-подобный язык запросов к heap dump:

```sql
-- Найти все ArrayList с более чем 1000 элементов
SELECT * FROM java.util.ArrayList WHERE size > 1000

-- Найти все строки длиннее 10000 символов
SELECT * FROM java.lang.String s WHERE s.value.length > 10000

-- Найти все экземпляры определённого класса с конкретным полем
SELECT * FROM com.example.MyClass c WHERE c.userId = null
```

Доступ в MAT: **File → Open Query Browser → OQL**.

**Типичный воркфлоу анализа**

1. Открыть `.hprof`: **File → Open Heap Dump**.
2. Запустить **Leak Suspects Report** — получить автоматические подсказки.
3. Открыть **Dominator Tree** — найти объекты с наибольшим retained heap.
4. Для подозрительного объекта: правый клик → **Path to GC Roots** (exclude weak/soft/phantom references) — это укажет, кто удерживает объект от сборки GC.
5. Изучить **Histogram** — распределение числа экземпляров по классам (много экземпляров одного класса — признак утечки).
6. Использовать **OQL** для специфических запросов.

**Дополнительные возможности MAT**
- **Thread Overview**: состояние всех потоков в момент дампа, объекты в стеке каждого потока.
- **Top Consumers**: топ объектов и классов по потреблению памяти.
- Экспорт отчётов в HTML/ZIP для совместного анализа.

### Вопрос 58. Что такое Java Flight Recorder (JFR) и JDK Mission Control (JMC)? Как их использовать для анализа GC-поведения приложения в production?

**Категория:** JVM и GC · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Monitoring Java Applications with Flight Recorder — Baeldung](https://www.baeldung.com/java-flight-recorder-monitoring)

Java Flight Recorder (JFR) — инструмент мониторинга и профилирования, встроенный в JDK, который собирает информацию о событиях в JVM с **минимальным влиянием на производительность** — он спроектирован как «always-on» инструмент для production-окружений.

JFR собирает три типа событий: **instant** (записывается немедленно при возникновении), **duration** (записывается, если длительность превышает порог), **sample** (семплирование активности системы).

**Запуск записи:**
- Через командную строку: `java -XX:StartFlightRecording=duration=200s,filename=flight.jfr MyApp`
- Через jcmd для работающего процесса: `jcmd <pid> JFR.start duration=100s filename=flight.jfr`

**Анализ:** JFR сохраняет данные в файл `.jfr`. Основной инструмент анализа — **JDK Mission Control (JMC)**: открывает `.jfr`-файл и предоставляет графический интерфейс с вкладками для CPU, памяти, GC, потоков и I/O. Включает Event Browser для детального просмотра конкретных событий.

#### Ответ от Claude

**Что такое JFR**

Java Flight Recorder — встроенный в HotSpot JVM механизм сбора событий с крайне низким overhead (менее 1–2% на большинстве приложений). Спроектирован для постоянного включения в production-системах. До Java 11 JFR был коммерческой функцией Oracle JDK; начиная с Java 11 (JEP 328) открыт и включён в OpenJDK.

**GC-события, записываемые JFR**

| Событие JFR | Что показывает |
|-------------|---------------|
| `jdk.GarbageCollection` | Каждая GC-пауза: тип (Minor/Major), длительность, причина |
| `jdk.GCHeapSummary` | Размер heap до и после GC: used, committed |
| `jdk.YoungGarbageCollection` | Детали Minor GC: время, паузы, размеры |
| `jdk.OldGarbageCollection` | Детали Major/Full GC |
| `jdk.G1HeapSummary` | Данные по регионам G1 |
| `jdk.PSHeapSummary` | Статистика Parallel GC по поколениям |

```bash
jcmd <pid> JFR.check

# Остановка и сохранение записи до истечения duration
jcmd <pid> JFR.stop name=diag filename=/tmp/diag.jfr

# Просмотр JFR файла в CLI (Java 14+)
jfr print --events jdk.GarbageCollection /tmp/app.jfr
```

JMC — графический инструмент для анализа `.jfr`-файлов. Ключевые вкладки для анализа GC:

- **Memory** — график использования heap по времени: занятость heap, committed, размеры Young/Old Generation.

### Вопрос 59. Что такое -XX:NewRatio и -XX:SurvivorRatio? Как эти параметры влияют на распределение памяти между поколениями и между Eden и Survivor пространствами?

**Категория:** JVM и GC · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [4 Sizing the Generations — Oracle Java 8 GC Tuning Guide](https://docs.oracle.com/javase/8/docs/technotes/guides/vm/gctuning/sizing.html)

**NewRatio** управляет соотношением между молодым и старым поколениями. Значение `-XX:NewRatio=3` означает, что соотношение Young : Old = 1 : 3, то есть Young Generation займёт 1/4 от общего размера heap, а Old Generation — 3/4. По умолчанию для 64-битной Server JVM значение NewRatio = 2 (Young = 1/3 heap, Old = 2/3 heap).

**SurvivorRatio** управляет соотношением внутри Young Generation между Eden и одним из Survivor пространств. Значение `-XX:SurvivorRatio=6` означает, что каждый Survivor занимает 1/6 от Eden. Поскольку Survivor пространств два, каждый занимает 1/8 от общего Young Generation (два Survivor = 2/6 Eden = 1/3 Young, Eden = 2/3 Young при SurvivorRatio=2; при SurvivorRatio=6 каждый Survivor = 1/8 Young).

По умолчанию SurvivorRatio = 8 (каждый Survivor = 1/10 Young Generation).

JVM динамически выбирает tenuring threshold так, чтобы Survivor пространство оставалось примерно наполовину заполненным.

#### Ответ от Claude

**NewRatio: соотношение Young и Old Generation**

`-XX:NewRatio=<N>` задаёт соотношение старого поколения к молодому:

```
Old Generation : Young Generation = N : 1
Размер Young = heap_max / (N + 1)
Размер Old   = heap_max × N / (N + 1)
```

| NewRatio | Young (% heap) | Old (% heap) |
|----------|---------------|-------------|
| 1 | 50% | 50% |
| **2** (дефолт 64-bit server) | **33%** | **67%** |
| 3 | 25% | 75% |
| 4 | 20% | 80% |

**Практическое влияние:**
- **Маленький NewRatio (большой Young Gen):** реже Minor GC, но каждая Minor GC собирает больше объектов, и паузы могут быть дольше. Больше шансов, что объект умрёт в Young Gen.
- **Большой NewRatio (маленький Young Gen):** частые Minor GC с короткими паузами, но больше объектов преждевременно продвигается в Old Gen (premature promotion), что нагружает Major GC.

**SurvivorRatio: соотношение Eden и Survivor**

`-XX:SurvivorRatio=<N>` задаёт соотношение Eden к одному Survivor:

```
Eden : Survivor = N : 1
Размер Eden    = Young / (N + 2)  — два Survivor занимают 2 части
Размер Survivor = Young / (N + 2)
```

| SurvivorRatio | Eden (% Young) | Один Survivor (% Young) |
|---------------|---------------|------------------------|
| **8** (дефолт) | **80%** | **10%** |
| 6 | 75% | 12.5% |
| 4 | 66.7% | 16.7% |
| 2 | 50% | 25% |

**Практическое влияние SurvivorRatio:**
- **Маленький SurvivorRatio (большие Survivor):** больше объектов выживает в Survivor, медленнее попадают в Old Gen. Полезно для объектов средней «продолжительности жизни».
- **Большой SurvivorRatio (маленькие Survivor):** при переполнении Survivor объекты сразу уходят в Old Gen (premature promotion), даже не достигнув MaxTenuringThreshold.

**UseAdaptiveSizePolicy и взаимодействие с этими флагами**

При включённом `-XX:+UseAdaptiveSizePolicy` (по умолчанию с Parallel GC) JVM **динамически переопределяет** размеры Eden и Survivor, игнорируя `SurvivorRatio`. Если нужно задать фиксированные пропорции — надо отключить: `-XX:-UseAdaptiveSizePolicy`.

С G1 GC флаги `NewRatio` и `SurvivorRatio` **не применяются**: G1 управляет размерами регионов и поколений самостоятельно. Для G1 используют `-XX:G1NewSizePercent` и `-XX:G1MaxNewSizePercent`.

**Связанные флаги**

| Флаг | Действие |
|------|---------|
| `-XX:NewSize=<N>` | Задать начальный размер Young Gen в байтах |
| `-XX:MaxNewSize=<N>` | Задать максимальный размер Young Gen |
| `-XX:NewRatio=<N>` | Соотношение Old:Young |
| `-XX:SurvivorRatio=<N>` | Соотношение Eden:Survivor |
| `-XX:MinSurvivorRatio=3` | Минимальный SurvivorRatio при UseAdaptiveSizePolicy |

- [4 Sizing the Generations — Oracle Java 8 GC Tuning Guide](https://docs.oracle.com/javase/8/docs/technotes/guides/vm/gctuning/sizing.html)
- [Garbage Collection Ergonomics — Oracle Java 8 VM Guide](https://docs.oracle.com/javase/8/docs/technotes/guides/vm/gc-ergonomics.html)

### Вопрос 60. Что такое Remembered Set (RSet) в G1 GC? Как он поддерживается, сколько памяти потребляет и какие флаги позволяют управлять его overhead?

**Категория:** JVM и GC · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Garbage-First (G1) Garbage Collector — Oracle Java 17 GC Tuning Guide](https://docs.oracle.com/en/java/javase/17/gctuning/garbage-first-g1-garbage-collector1.html)

Remembered Set (RSet) — структура данных в G1 GC, которая помогает управлять ссылками между разными регионами. Каждый регион имеет собственный RSet, в котором хранятся сведения об объектах из других регионов, ссылающихся на объекты данного региона. Это позволяет G1 собирать отдельный регион независимо, не сканируя весь heap: при эвакуации региона GC использует его RSet, чтобы найти все «внешние» ссылки и обновить их после перемещения объектов.

**Поддержание RSet:** Когда поток приложения записывает ссылку, JIT-компилятор вставляет **post-write barrier**, который регистрирует запись в thread-local буфере (Dirty Card Queue). Специальные фоновые потоки — **Concurrent Refinement threads** — обрабатывают этот буфер в фоновом режиме и обновляют RSet соответствующих регионов.

**Overhead RSet:** RSets могут занимать значительный объём памяти. Поскольку каждый регион хранит информацию о входящих ссылках из других регионов, в худшем случае (множество мелких регионов с перекрёстными ссылками) накладные расходы могут достигать 20% от heap.

**Способы снижения overhead:**
- Увеличить размер региона: `-XX:G1HeapRegionSize=<N>m` — меньше регионов → меньше cross-region ссылок → меньше RSet-записей.
- Управлять числом потоков рефайнинга: `-XX:G1ConcRefinementThreads=<N>`.

#### Ответ от Claude

**Зачем G1 нужен Remembered Set**

G1 GC разбивает heap на регионы и может собирать отдельные регионы независимо (без сканирования всего heap). Однако объект в одном регионе может хранить ссылку на объект в другом регионе. При сборке региона B нужно знать: какие объекты из других регионов ссылаются на объекты в B? Иначе GC не сможет найти все ссылки и корректно обновить их после перемещения.

Наивное решение — сканировать весь heap — слишком медленно. Решение G1 — **Remembered Set**: компактная структура данных, хранящая «входящие» ссылки.

**Структура Remembered Set**

Каждый регион имеет свой RSet. Начиная с Java 17 RSet реализован через **G1 Card Set** (JEP 376) — иерархическая структура с несколькими уровнями детализации (arrays, hash sets, bitmap), автоматически выбирающая наиболее компактное представление в зависимости от числа входящих ссылок.

RSet хранит не сами ссылки, а **грязные карты (dirty cards)**: ссылки на карты card table (блоки по 512 байт) в регионах-источниках, из которых могут приходить inter-region ссылки. При эвакуации GC сканирует карты из RSet и проверяет, есть ли в них ссылки на объекты в эвакуируемом регионе.

**Механизм поддержания RSet**

```
Поток приложения: obj.field = ref   ← запись ссылки
       ↓
Post-write barrier (вставляется JIT-компилятором)
       ↓
Dirty Card Queue (per-thread буфер)
       ↓
Concurrent Refinement Threads (фоновые, G1-специфичные)
       ↓
Обновление RSet затронутых регионов
```

При высокой частоте записей Concurrent Refinement Threads могут не успевать опустошать буферы. Тогда подключаются потоки-мутаторы (сами потоки приложения) — это называется **mutation refinement** и приводит к кратковременному замедлению приложения.

**Потребление памяти**

Накладные расходы RSets зависят от числа cross-region ссылок:
- При малом числе входящих ссылок: компактный массив.
- При большом числе: переходит к bitmap-представлению.
- В худшем случае (очень много cross-region ссылок, мелкие регионы): до 10–20% от heap.

До Java 17 (старый RSet через HashTable): структуры RSets не вытеснялись и могли расти бесконтрольно. С Java 17 (`G1CardSet`) введены лимиты: `-XX:G1RemSetArrayOfCardsEntries`, `G1RemSetHashtableNumBuckets` — ограничивают память RSet per-region.

**Флаги управления overhead RSet**

| Флаг | Значение по умолчанию | Действие |
|------|--------------------|---------|
| `-XX:G1HeapRegionSize=<N>m` | Автоматически (~heap/2048) | Увеличение → меньше регионов → меньше cross-region ссылок → меньше RSet |
| `-XX:G1ConcRefinementThreads=<N>` | `ParallelGCThreads` | Число Concurrent Refinement потоков |
| `-XX:G1RSetScanThreads=<N>` | 1 | Число потоков, сканирующих RSet при эвакуации |
| `-XX:G1RemSetFreeMemoryThreshold=10` | 10 | Порог памяти (%), при котором сжимаются Card Sets |

**Практические рекомендации**
- Если NMT (`jcmd <pid> VM.native_memory`) показывает большую категорию `GCCardSet` → много cross-region ссылок. Решение: увеличить `G1HeapRegionSize`.
- При высоком CPU на Concurrent Refinement потоках — увеличить `G1ConcRefinementThreads`.
- Избегать `-XX:NewRatio` и `-XX:NewSize` с G1: они лишают G1 возможности самостоятельно управлять размером Young Gen и могут нарушить работу RSet-механизма.
