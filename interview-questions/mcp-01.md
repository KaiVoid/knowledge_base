### Вопрос 1. Что такое Model Context Protocol (MCP) и какую ключевую проблему интеграции он решает?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Anthropic — Introducing the Model Context Protocol](https://www.anthropic.com/news/model-context-protocol)

Model Context Protocol — это открытый стандарт, позволяющий разработчикам строить безопасные двусторонние соединения между источниками данных и AI-инструментами. Протокол выступает универсальным стандартом для подключения AI-систем к различным репозиториям данных и бизнес-приложениям.

Ключевая проблема: даже самые мощные модели страдают от изоляции от реальных данных — они «заперты за информационными силосами и устаревшими системами». Фундаментальная сложность состоит в том, что каждый новый источник данных требовал отдельной реализации, что делало построение действительно интегрированных систем крайне трудноосуществимым.

MCP решает эту проблему, предоставляя «более простой и надёжный способ дать AI-системам доступ к нужным данным» через единый стандартизированный протокол. Вместо разрозненных интеграций MCP предлагает единый протокол, позволяющий организациям масштабировать свои AI-возможности, одновременно сохраняя согласованный контекст между разными инструментами и наборами данных.

#### Ответ от Claude

MCP (Model Context Protocol) — открытый стандарт от Anthropic, обеспечивающий безопасные двусторонние соединения между AI-приложениями и источниками данных или инструментами.

**Проблема, которую решает MCP:**

До появления MCP каждое новое подключение AI-модели к внешнему сервису или источнику данных требовало написания отдельного кастомного коннектора. Это приводило к:
- фрагментированным интеграциям, которые сложно поддерживать;
- изоляции AI-моделей от корпоративных данных («information silos»);
- невозможности масштабировать систему без экспоненциального роста количества адаптеров.

**Как MCP это решает:**

MCP заменяет разрозненные коннекторы единым протоколом. Разработчик один раз реализует MCP-сервер для своего инструмента — и он сразу становится доступен любому AI-приложению, поддерживающему MCP. Это переводит задачу интеграции с квадратичной сложности M×N к линейной M+N.

### Вопрос 2. В чём суть проблемы N×M интеграции в AI и как MCP её решает?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Klavis AI — Solving the N×M Integration Problem](https://www.klavis.ai/blog/mcp-solving-n-x-m-integration-problem)

Проблема N×M возникает, когда разработка AI требует написания кастомного кода для соединения каждой модели с каждым внешним инструментом. При наличии N моделей и M приложений разработчики сталкиваются с N × M уникальными интеграциями.

**Математический пример:** «4 модели × 10 инструментов = 40 интеграций». Эта сложность «растёт квадратично, а не линейно», создавая «парализующую сложность». Дополнительные осложнения: разные схемы аутентификации в API, уникальные схемы данных для каждой интеграции, различные требования к ограничению частоты запросов, уязвимость к изменениям сторонних API.

**Как MCP трансформирует уравнение:**

Model Context Protocol сводит задачу к архитектуре N + M. Используя тот же пример: «4 модели + 10 инструментов = 14 интеграций».

Структурный сдвиг:
- **MCP Clients**, встроенные в AI-приложения, изучают один стандартизированный язык.
- **MCP Servers** оборачивают внешние инструменты — для каждого сервиса требуется лишь одна реализация.

Этот «стандартизированный промежуточный слой» разделяет модели и инструменты. Как только инструмент получает MCP-сервер, «любое AI-приложение, поддерживающее MCP, может мгновенно его использовать», исключая избыточные кастомные коннекторы.

#### Ответ от Claude

**Суть проблемы N×M:**

При разработке AI-систем, которым нужно работать с внешними инструментами, без стандарта каждый из N AI-клиентов (Claude, GPT, Gemini и т.д.) должен реализовывать отдельный коннектор к каждому из M инструментов (GitHub, Jira, БД и т.д.). Итого: N × M коннекторов.

| Без MCP | С MCP |
|---------|-------|
| 4 модели × 10 инструментов = **40 интеграций** | 4 + 10 = **14 реализаций** |

**Как MCP решает проблему:**

MCP вводит стандартизированный промежуточный слой:
- Каждый **инструмент** реализует **один MCP-сервер** (1 реализация вместо N).
- Каждое **AI-приложение** реализует **одного MCP-клиента** (1 реализация вместо M).

Любой MCP-клиент сразу «понимает» любой MCP-сервер — без дополнительного кода. Сложность роста снижается с квадратичной O(N×M) до линейной O(N+M).

### Вопрос 3. Когда был объявлен MCP, кто его создал и какова история его стандартизации?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Wikipedia — Model Context Protocol](https://en.wikipedia.org/wiki/Model_Context_Protocol)

MCP был представлен компанией Anthropic в ноябре 2024 года. Протокол «создан в Anthropic инженерами Дэвидом Сориа Паррой и Джастином Спар-Саммерсом». Он был разработан для решения проблем фрагментированных систем данных и проблемы «N×M» интеграции, с которой сталкивались разработчики при создании кастомных коннекторов.

**Ключевые этапы принятия:**
- **Март 2025**: OpenAI официально принял MCP в своих продуктах, включая приложение ChatGPT Desktop.
- **Апрель 2025**: Google DeepMind принял стандарт.
- **Сентябрь 2025**: OpenAI расширил поддержку MCP в приложениях ChatGPT, открыв сторонний доступ.
- **Декабрь 2025**: Anthropic передала MCP в Agentic AI Foundation (AAIF) — направленный фонд под управлением Linux Foundation, сооснованный Anthropic, Block и OpenAI.
- **Апрель 2026**: AAIF провёл первый MCP Dev Summit North America в Нью-Йорке — около 1200 участников.

#### Ответ от Claude

**Создание:**
MCP был публично представлен Anthropic **25 ноября 2024 года**. Авторы протокола — инженеры Anthropic Дэвид Сориа Парра (David Soria Parra) и Джастин Спар-Саммерс (Justin Spahr-Summers).

**История стандартизации:**

| Дата | Событие |
|------|---------|
| Ноябрь 2024 | Анонс MCP от Anthropic как открытого стандарта |
| Март 2025 | OpenAI принял MCP: ChatGPT Desktop, Agents SDK, Responses API |
| Апрель 2025 | Google DeepMind поддержал MCP в Gemini SDK; управляемые серверы для Maps, BigQuery, Kubernetes Engine |
| Декабрь 2025 | Передача MCP в Linux Foundation (AAIF), сооснованную Anthropic, Block и OpenAI |

Передача в Linux Foundation закрепила статус MCP как вендор-нейтрального отраслевого стандарта — по аналогии с Kubernetes и PyTorch.

### Вопрос 4. Что такое Agentic AI Foundation (AAIF) и почему Anthropic передала MCP в Linux Foundation?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Anthropic — Donating the Model Context Protocol](https://www.anthropic.com/news/donating-the-model-context-protocol-and-establishing-of-the-agentic-ai-foundation)

**Что такое AAIF:**

Agentic AI Foundation — это направленный фонд под управлением Linux Foundation, созданный для продвижения развития агентного AI. Его миссия — обеспечить, чтобы «агентный AI развивался прозрачно, совместно и в интересах общества посредством стратегических инвестиций, построения сообщества и совместной разработки открытых стандартов».

**Почему Anthropic передала MCP:**

Anthropic передала протокол, чтобы укрепить свою приверженность сохранению MCP как нейтрального, вендор-агностичного стандарта. Компания подчеркнула, что «открытое программное обеспечение необходимо для построения безопасной и инновационной экосистемы для агентного AI». Передача гарантирует, что протокол останется управляемым сообществом под независимым попечительством.

**Структура руководства и поддержка:**
- **Сооснователи:** Anthropic, Block и OpenAI.
- **Поддерживающие организации:** Google, Microsoft, AWS, Cloudflare, Bloomberg.
- Linux Foundation обеспечивает институциональное управление.
- MCP стал основополагающим проектом наряду с goose (Block) и AGENTS.md (OpenAI).

#### Ответ от Claude

**AAIF (Agentic AI Foundation)** — направленный фонд (directed fund) в составе Linux Foundation, сооснованный Anthropic, Block и OpenAI в декабре 2025 года.

**Цель фонда:** обеспечить прозрачное, коллаборативное и направленное на общественный интерес развитие агентного AI через инвестиции, построение сообщества и совместную разработку открытых стандартов.

**Почему Anthropic передала MCP:**

1. **Вендор-нейтральность** — ни одна компания не должна контролировать критический инфраструктурный протокол.
2. **Долгосрочная независимость** — управление через Linux Foundation гарантирует устойчивость вне зависимости от позиции Anthropic.
3. **Открытость** — открытое ПО признано Anthropic необходимым условием безопасной AI-экосистемы.

**Аналогия:** Linux Foundation аналогичным образом управляет Kubernetes (Google → LF) и PyTorch (Meta → LF). Технические решения по MCP остаются у существующих мейнтейнеров — AAIF обеспечивает только нейтральное управление.

**Поддержка фонда:** Google, Microsoft, AWS, Cloudflare, Bloomberg.

### Вопрос 5. Как MCP соотносится с Language Server Protocol (LSP) и какие архитектурные идеи он оттуда заимствовал?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [MCP Specification — Overview](https://modelcontextprotocol.io/specification/latest)

Из спецификации MCP: «MCP черпает некоторое вдохновение из Language Server Protocol, который стандартизировал добавление поддержки языков программирования в целую экосистему инструментов разработки. Аналогичным образом MCP стандартизирует интеграцию дополнительного контекста и инструментов в экосистему AI-приложений».

Оба протокола используют JSON-RPC в качестве базового формата сообщений. MCP строит stateful-сессии с поддержкой согласования возможностей (capability negotiation) — паттерн, непосредственно унаследованный от LSP.

#### Ответ от Claude

**Language Server Protocol (LSP)** — стандарт от Microsoft (2016), решавший аналогичную проблему в мире IDE: до LSP каждый редактор должен был реализовывать поддержку каждого языка отдельно (та же проблема N×M). LSP стандартизировал интерфейс «редактор — языковой сервер», что позволило любому редактору с LSP-клиентом работать с любым LSP-совместимым языковым сервером.

**MCP применяет ту же идею к AI:**

| Аспект | LSP | MCP |
|--------|-----|-----|
| Проблема | N редакторов × M языков | N AI-клиентов × M инструментов |
| Решение | Стандартный протокол IDE↔языковой сервер | Стандартный протокол AI↔контекстный сервер |
| Транспорт | JSON-RPC | JSON-RPC 2.0 |
| Инициализация | Capability negotiation | Capability negotiation |

**Что заимствовал MCP из LSP:**
- Использование JSON-RPC как базового протокола сообщений.
- Stateful-сессии с явным управлением жизненным циклом.
- Согласование возможностей (capabilities) при инициализации — обе стороны объявляют, что они поддерживают.
- Идею взаимозаменяемости реализаций без привязки к конкретному поставщику.

### Вопрос 6. Какова роль каждого из трёх участников архитектуры MCP: Host, Client и Server?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [MCP — Architecture overview (docs)](https://modelcontextprotocol.io/docs/learn/architecture)

Три ключевых участника архитектуры MCP:

- **MCP Host** — AI-приложение, которое координирует и управляет одним или несколькими MCP-клиентами. Примеры: Claude Code, Claude Desktop, Visual Studio Code.
- **MCP Client** — компонент, который поддерживает соединение с MCP-сервером и получает контекст от него для использования хостом.
- **MCP Server** — программа, которая предоставляет контекст MCP-клиентам.

**Конкретный пример:** Visual Studio Code выступает MCP-хостом. Когда VS Code устанавливает соединение с MCP-сервером Sentry, среда выполнения VS Code создаёт объект MCP-клиента для этого соединения. Когда VS Code затем подключается к серверу локальной файловой системы — создаётся ещё один MCP-клиент для этого соединения.

#### Ответ от Claude

Архитектура MCP включает три участника:

**MCP Host (хост)**
- AI-приложение (Claude Desktop, VS Code, Claude Code и др.).
- Создаёт и управляет множеством MCP-клиентов.
- Контролирует разрешения клиентов и жизненный цикл соединений.
- Реализует политики безопасности и обрабатывает авторизации пользователя.
- Агрегирует контекст от всех подключённых серверов.

**MCP Client (клиент)**
- Компонент внутри хоста, создаваемый хостом для каждого MCP-сервера.
- Поддерживает одно выделенное stateful-соединение с конкретным сервером (отношение 1:1).
- Маршрутизирует JSON-RPC сообщения между хостом и сервером.
- Управляет подписками и нотификациями от сервера.

**MCP Server (сервер)**
- Программа, предоставляющая контекст и функциональность клиентам.
- Предоставляет примитивы: Tools (инструменты), Resources (ресурсы), Prompts (шаблоны).
- Может работать локально (STDIO) или удалённо (Streamable HTTP).
- Не видит весь разговор и не взаимодействует с другими серверами напрямую.

### Вопрос 7. Почему каждый MCP Client поддерживает соединение ровно с одним MCP Server (отношение 1:1)?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [MCP Specification — Architecture](https://modelcontextprotocol.io/specification/2025-11-25/architecture)

Из спецификации архитектуры MCP:

«Каждый клиент создаётся хостом и поддерживает изолированное серверное соединение»:
- Устанавливает один stateful-сеанс на сервер.
- Поддерживает границы безопасности между серверами.

Ключевой принцип дизайна: «Серверы не должны иметь возможности читать весь разговор и не должны "видеть" другие серверы». Серверы получают только необходимую контекстную информацию. Полная история разговора остаётся у хоста. Каждое серверное соединение поддерживает изоляцию. Взаимодействия между серверами контролируются хостом, а процесс хоста обеспечивает границы безопасности.

#### Ответ от Claude

Изоляция 1:1 (один клиент — один сервер) является преднамеренным архитектурным решением, продиктованным соображениями безопасности.

**Причины:**

1. **Изоляция контекста.** Каждый MCP-сервер получает только ту информацию, которая ему необходима. Сервер не может прочитать весь разговор пользователя с AI и не «видит» другие серверы. Это предотвращает несанкционированный сбор данных.

2. **Чёткие границы безопасности.** Каждое клиентское соединение изолировано — компрометация одного сервера не даёт доступа к данным других серверов.

3. **Управляемая оркестрация.** Host является единственной точкой агрегации контекста от всех серверов. Host контролирует, что и кому передаётся, реализуя политики авторизации.

4. **Простота реализации серверов.** Сервер отвечает только за свою узкую функциональность, не нуждаясь в знании об остальной системе.

**Итог:** Отношение 1:1 гарантирует, что серверы никогда не видят весь разговор и не взаимодействуют друг с другом напрямую — за всю межсерверную координацию отвечает Host.

### Вопрос 8. Какие два транспортных механизма определяет MCP и в каких сценариях применяется каждый?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [MCP — Architecture overview (docs)](https://modelcontextprotocol.io/docs/learn/architecture)

MCP поддерживает два транспортных механизма:

- **Stdio-транспорт** — использует стандартные потоки ввода/вывода для прямой коммуникации между процессами на одной машине, обеспечивая оптимальную производительность без сетевых накладных расходов. Локальные MCP-серверы, использующие STDIO-транспорт, как правило, обслуживают одного MCP-клиента.

- **Streamable HTTP-транспорт** — использует HTTP POST для сообщений от клиента к серверу с опциональными Server-Sent Events для возможностей стриминга. Этот транспорт обеспечивает коммуникацию с удалёнными серверами и поддерживает стандартные методы HTTP-аутентификации, включая bearer-токены, API-ключи и кастомные заголовки. MCP рекомендует использовать OAuth для получения токенов аутентификации. Удалённые MCP-серверы, использующие Streamable HTTP транспорт, как правило, обслуживают многих MCP-клиентов.

Транспортный уровень абстрагирует детали коммуникации от уровня протокола, обеспечивая одинаковый формат сообщений JSON-RPC 2.0 для всех транспортных механизмов.

#### Ответ от Claude

MCP определяет два транспортных механизма:

**1. Stdio-транспорт**
- Использует стандартные потоки ввода/вывода (stdin/stdout).
- Применяется для **локальных серверов** на той же машине, что и клиент.
- Не требует сетевого стека — минимальные накладные расходы.
- Типично обслуживает одного клиента.
- Пример: Claude Desktop запускает локальный сервер файловой системы через Stdio.

**2. Streamable HTTP-транспорт**
- HTTP POST для запросов клиента к серверу.
- Опциональные Server-Sent Events (SSE) для стриминга ответов сервера.
- Применяется для **удалённых серверов** (работающих в облаке или на другом хосте).
- Поддерживает стандартные методы аутентификации: bearer-токены, API-ключи, OAuth.
- Один сервер может обслуживать множество клиентов одновременно.
- Пример: официальный Sentry MCP-сервер, работающий на платформе Sentry.

**Ключевое отличие:** оба транспорта используют одинаковый формат сообщений JSON-RPC 2.0 — транспортный уровень абстрагирован от уровня протокола.

### Вопрос 9. Почему MCP использует JSON-RPC 2.0 в качестве основного формата сообщений?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [MCP Specification — Base Protocol](https://modelcontextprotocol.io/specification/2025-03-26/basic)

Из спецификации базового протокола MCP:

Все сообщения между MCP-клиентами и серверами должны следовать спецификации JSON-RPC 2.0. Протокол определяет три типа сообщений:

- **Requests (Запросы)** — отправляются от клиента к серверу или наоборот, для инициирования операции. Запросы обязаны включать строковый или целочисленный ID. ID запроса не должен быть null и не должен повторяться в рамках одной сессии.
- **Responses (Ответы)** — отправляются в ответ на запросы, содержат результат или ошибку операции. Ответ обязан включать тот же ID, что и соответствующий запрос. Ответ содержит либо result, либо error — не оба одновременно.
- **Notifications (Уведомления)** — отправляются от клиента к серверу или наоборот как одностороннее сообщение. Получатель не должен отправлять ответ. Уведомления не должны включать ID.

#### Ответ от Claude

**Почему JSON-RPC 2.0:**

JSON-RPC 2.0 — лёгкий, читаемый человеком протокол удалённых вызовов процедур, основанный на JSON. Он идеально подошёл для MCP по нескольким причинам:

1. **Независимость от языка** — JSON поддерживается во всех языках программирования, что упрощает реализацию MCP-серверов и клиентов на любом стеке.
2. **Человекочитаемость** — сообщения легко отлаживать и логировать.
3. **Стандартизированность** — устоявшийся стандарт с чёткой семантикой.
4. **Соответствие паттернам LSP** — LSP, вдохновивший MCP, также использует JSON-RPC.

**Три типа сообщений JSON-RPC в MCP:**

| Тип | ID | Описание |
|-----|----|----------|
| **Request** | обязателен (строка или число, не null) | Инициирует операцию; ожидает ответ |
| **Response** | совпадает с ID запроса | Содержит `result` или `error` (не оба) |
| **Notification** | отсутствует | Одностороннее сообщение; ответ не ожидается |

**Важно:** Также поддерживается батчинг — отправка массива запросов/уведомлений. MCP-реализации обязаны уметь принимать батчи (но не обязаны их отправлять).

### Вопрос 10. Что происходит в фазе Lifecycle Management при установке MCP-соединения?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [MCP — Architecture overview (docs)](https://modelcontextprotocol.io/docs/learn/architecture)

MCP — stateful-протокол, требующий управления жизненным циклом. Цель lifecycle management — согласовать возможности (capabilities), которые поддерживают и клиент, и сервер.

Процесс инициализации включает три критически важных шага:
1. **Согласование версии протокола** — поле `protocolVersion` (например, «2025-06-18») гарантирует, что клиент и сервер используют совместимые версии. Если совместимую версию согласовать не удаётся, соединение следует разорвать.
2. **Обнаружение возможностей** — объект `capabilities` позволяет каждой стороне объявить поддерживаемые функции, включая примитивы (tools, resources, prompts) и поддержку нотификаций.
3. **Обмен идентификаторами** — объекты `clientInfo` и `serverInfo` предоставляют идентификационную информацию и версии для отладки.

После успешной инициализации клиент отправляет уведомление `notifications/initialized`, сигнализируя о готовности.

#### Ответ от Claude

Управление жизненным циклом (Lifecycle Management) — это обязательная фаза установки MCP-соединения, обеспечивающая, что клиент и сервер «договорились» о возможностях до начала реальной работы.

**Последовательность инициализации:**

1. **Клиент отправляет запрос `initialize`** с:
   - `protocolVersion` — версия протокола, которую поддерживает клиент.
   - `capabilities` — объект с объявленными возможностями клиента (например, `elicitation`, `sampling`).
   - `clientInfo` — имя и версия клиента.

2. **Сервер отвечает** с:
   - `protocolVersion` — согласованная версия (или несовместимая — тогда соединение разрывается).
   - `capabilities` — возможности сервера (например, `tools: {listChanged: true}`, `resources: {}`).
   - `serverInfo` — имя и версия сервера.

3. **Клиент отправляет нотификацию `notifications/initialized`** — подтверждение готовности к работе.

**Что обеспечивает согласование возможностей:**
- Клиент знает, какие примитивы сервер поддерживает (tools/resources/prompts).
- Сервер знает, что клиент умеет (sampling, elicitation, обработка нотификаций об изменениях).
- Обе стороны не тратят время на запросы неподдерживаемых операций.

### Вопрос 11. Какие три серверных примитива определяет MCP (Tools, Resources, Prompts) и чем они отличаются?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [MCP Specification — Overview](https://modelcontextprotocol.io/specification/latest)

Из спецификации MCP, раздел Features:

Серверы могут предоставлять клиентам любые из следующих возможностей:
- **Resources (Ресурсы)** — контекст и данные, для использования пользователем или AI-моделью.
- **Prompts (Промпты)** — шаблонные сообщения и рабочие процессы для пользователей.
- **Tools (Инструменты)** — функции для выполнения AI-моделью.

Согласно обзору архитектуры: «MCP определяет три основных примитива, которые серверы могут предоставлять: Tools — исполняемые функции, которые AI-приложения могут вызывать для выполнения действий (например, файловые операции, API-вызовы, запросы к базам данных); Resources — источники данных, предоставляющие контекстную информацию AI-приложениям (например, содержимое файлов, записи БД, ответы API); Prompts — переиспользуемые шаблоны, помогающие структурировать взаимодействие с языковыми моделями (например, системные промпты, примеры few-shot)».

#### Ответ от Claude

MCP определяет три серверных примитива, различающихся по тому, **кто инициирует их использование**:

| Примитив | Управляет | Назначение | Пример |
|----------|-----------|------------|--------|
| **Tools** | AI-модель | Исполняемые функции — модель сама решает, когда вызвать | Запрос к БД, вызов API, операция с файлом |
| **Resources** | Приложение (хост) | Источники данных только для чтения — хост решает, что включить в контекст | Схема БД, содержимое файла, документация |
| **Prompts** | Пользователь | Переиспользуемые шаблоны — пользователь явно выбирает шаблон | Системные промпты, few-shot примеры |

**Ключевое различие:** вопрос «кто решает, когда это использовать»:
- Tools — **модель** автономно вызывает в ходе генерации ответа.
- Resources — **приложение** включает в контекст на основе эвристик или UI.
- Prompts — **пользователь** явно выбирает из предложенных шаблонов.

Каждый тип примитива имеет ассоциированные методы обнаружения (`*/list`) и выполнения (`tools/call`, `resources/read` и т.д.).

### Вопрос 12. Что такое примитив Sampling в MCP и зачем он нужен серверу?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [MCP — Sampling (modelcontextprotocol.info)](https://modelcontextprotocol.info/docs/concepts/sampling/)

Sampling — это мощная возможность MCP, позволяющая серверам запрашивать LLM-комплишены через клиента. Как указано в документации: «Sampling — это мощная возможность MCP, которая позволяет серверам запрашивать LLM-комплишены через клиент, обеспечивая сложные агентные поведения при сохранении безопасности и конфиденциальности».

Серверы могут использовать Sampling для выполнения сложных задач, таких как анализ ресурсов, принятие интеллектуальных решений на основе контекста и управление многошаговыми рабочими процессами — без необходимости реализовывать собственную LLM-инфраструктуру.

**Как работает human-in-the-loop:**
1. **Проверка запроса** — клиенты изучают и могут изменить промпт перед отправкой в LLM.
2. **Проверка ответа** — пользователи видят сгенерированный вывод и могут изменить или отклонить его.
3. **Контроль модели** — пользователи решают, какая модель используется.

Документация подчёркивает: «Этот дизайн с участием человека в процессе гарантирует, что пользователи сохраняют контроль над тем, что LLM видит и генерирует».

#### Ответ от Claude

**Sampling** — клиентский примитив MCP (в отличие от Tools/Resources/Prompts, которые серверные), позволяющий MCP-серверу **запрашивать LLM-комплишен через хост-клиент** с помощью метода `sampling/createMessage`.

**Зачем это серверу:**

Сервер может нуждаться в LLM для сложных задач — анализа ресурсов, принятия решений на основе контекста, многошаговых рабочих процессов. Без Sampling серверу пришлось бы встраивать собственный SDK к языковой модели и управлять API-ключами. Sampling позволяет серверу оставаться **model-agnostic**: он делегирует вызов к LLM хосту, который уже имеет настроенный доступ к модели.

**Поток Sampling:**
1. Сервер отправляет клиенту запрос `sampling/createMessage` с промптом.
2. Клиент/хост показывает запрос пользователю (human-in-the-loop).
3. Пользователь может изменить промпт, выбрать модель или отклонить запрос.
4. Хост выполняет LLM-запрос и возвращает ответ серверу.
5. Сервер получает результат, но **не видит**, какая конкретно модель была использована и что ещё передавалось модели.

**Безопасность:** пользователь контролирует три аспекта: происходит ли sampling вообще; точный промпт, отправляемый в LLM; что сервер увидит в ответе.

### Вопрос 13. Каковы принципы безопасности и доверия (Security & Trust), закреплённые в спецификации MCP?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [MCP Specification — Overview](https://modelcontextprotocol.io/specification/latest)

Из раздела Security and Trust & Safety спецификации MCP:

«Model Context Protocol открывает мощные возможности через пути произвольного доступа к данным и выполнения кода. С этой мощью связаны важные соображения безопасности и доверия, которые все разработчики должны тщательно учитывать».

Четыре ключевых принципа:
1. **Согласие и контроль пользователя** — пользователи должны явно согласиться со всеми операциями доступа к данным. Пользователи должны сохранять контроль над тем, какие данные передаются и какие действия предпринимаются.
2. **Конфиденциальность данных** — хосты должны получить явное согласие пользователя перед передачей пользовательских данных серверам. Хосты не должны передавать данные ресурсов в другие места без согласия пользователя.
3. **Безопасность инструментов** — инструменты представляют произвольное выполнение кода и должны использоваться с надлежащей осторожностью. Описания инструментов (аннотации) следует считать ненадёжными, если только они не получены от доверенного сервера. Хосты обязаны получить явное согласие пользователя перед вызовом любого инструмента.
4. **Контроль LLM Sampling** — пользователи должны явно одобрять любые запросы на LLM Sampling. Пользователи должны контролировать: произойдёт ли сэмплинг вообще; фактический промпт, который будет отправлен; что сервер сможет увидеть в результате.

#### Ответ от Claude

Спецификация MCP закрепляет четыре принципа безопасности и доверия. Важно отметить: MCP **не может принудительно** обеспечить эти принципы на уровне протокола — ответственность лежит на разработчиках реализаций.

**Принципы:**

1. **User Consent and Control (Согласие и контроль пользователя)**
   - Пользователь явно соглашается на каждый доступ к данным и операцию.
   - Пользователь сохраняет контроль над тем, что передаётся и какие действия выполняются.

2. **Data Privacy (Конфиденциальность данных)**
   - Хост не передаёт пользовательские данные серверам без явного согласия.
   - Хост не пересылает данные ресурсов третьим сторонам без разрешения.

3. **Tool Safety (Безопасность инструментов)**
   - Инструменты — это произвольное выполнение кода; к ним нужно относиться с осторожностью.
   - Описания и аннотации инструментов считаются **ненадёжными**, если сервер не является доверенным.
   - Хост обязан получить явное согласие пользователя перед вызовом любого инструмента.

4. **LLM Sampling Controls (Контроль Sampling)**
   - Пользователь явно одобряет каждый запрос на LLM-сэмплинг.
   - Пользователь контролирует: факт сэмплинга, содержание промпта, что сервер увидит в ответе.

### Вопрос 14. Какова роль MCP Host как оркестратора безопасности по сравнению с ролью MCP Client?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [MCP Specification — Architecture](https://modelcontextprotocol.io/specification/2025-11-25/architecture)

Из спецификации архитектуры MCP:

**Host** действует как контейнер и координатор:
- Создаёт множество клиентских экземпляров и управляет ими.
- Контролирует разрешения и жизненный цикл клиентских соединений.
- Обеспечивает соблюдение политик безопасности и требований согласия.
- Обрабатывает решения об авторизации пользователя.
- Координирует интеграцию AI/LLM и сэмплинг.
- Управляет агрегацией контекста от всех клиентов.

**Clients** (каждый клиент создаётся хостом и поддерживает изолированное серверное соединение):
- Устанавливает один stateful-сеанс на сервер.
- Управляет согласованием протокола и обменом возможностями.
- Маршрутизирует протокольные сообщения в обоих направлениях.
- Управляет подписками и нотификациями.
- Поддерживает границы безопасности между серверами.

Хост-приложение создаёт множество клиентов и управляет ими, каждый из которых имеет отношение 1:1 с конкретным сервером.

#### Ответ от Claude

Host и Client — два разных уровня иерархии в MCP, выполняющих принципиально разные функции.

**MCP Host — оркестратор безопасности:**

| Ответственность | Описание |
|-----------------|----------|
| Управление клиентами | Создаёт, запускает и завершает жизненный цикл всех MCP-клиентов |
| Политики безопасности | Реализует правила авторизации и согласия пользователя |
| Агрегация контекста | Собирает контекст от всех серверов и передаёт его LLM |
| LLM-интеграция | Координирует обращения к языковой модели (включая Sampling) |
| Изоляция серверов | Гарантирует, что серверы не видят друг друга |

**MCP Client — управление одним сеансом:**

| Ответственность | Описание |
|-----------------|----------|
| Один сеанс | Поддерживает stateful-соединение ровно с одним сервером |
| Обмен возможностями | Выполняет capability negotiation при инициализации |
| Маршрутизация сообщений | Двунаправленная передача JSON-RPC сообщений |
| Подписки | Управляет подписками на нотификации сервера |

**Ключевой вывод:** Host — единственная точка оркестрации и безопасности. Client намеренно «тупой» — он только управляет одним соединением, не зная о других серверах и не имея доступа к полному контексту разговора.

### Вопрос 15. Как механизм Capability Negotiation в MCP обеспечивает расширяемость и обратную совместимость?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [MCP Specification — Architecture](https://modelcontextprotocol.io/specification/2025-11-25/architecture)

Из спецификации архитектуры MCP:

«Model Context Protocol использует систему согласования возможностей, в которой клиенты и серверы явно объявляют поддерживаемые функции при инициализации. Capabilities определяют, какие функции протокола и примитивы доступны в течение сеанса».

- Серверы объявляют возможности, такие как подписки на ресурсы, поддержка инструментов и шаблоны промптов.
- Клиенты объявляют возможности, такие как поддержка Sampling и обработка нотификаций.
- Обе стороны **обязаны** соблюдать объявленные возможности в течение сеанса.
- Дополнительные возможности могут быть согласованы через расширения протокола.

«Принципы дизайна»: «Функции могут добавляться в серверы и клиенты поэтапно. Базовый протокол предоставляет минимальную необходимую функциональность. Дополнительные возможности могут быть согласованы по мере необходимости. Серверы и клиенты развиваются независимо. Протокол спроектирован для будущей расширяемости. Обратная совместимость поддерживается».

#### Ответ от Claude

**Capability Negotiation (согласование возможностей)** — механизм MCP, при котором клиент и сервер при инициализации взаимно объявляют поддерживаемые функции.

**Как это работает:**

При инициализации (`initialize` запрос) каждая сторона передаёт объект `capabilities`:

**Серверные возможности:**
- `tools` / `tools.listChanged` — поддержка инструментов и нотификаций об изменениях.
- `resources` / `resources.subscribe` / `resources.listChanged` — поддержка ресурсов.
- `prompts` / `prompts.listChanged` — поддержка шаблонов.

**Клиентские возможности:**
- `sampling` — клиент может обрабатывать запросы Sampling от сервера.
- `elicitation` — клиент может запрашивать дополнительные данные у пользователя.

**Как это обеспечивает расширяемость:**

1. **Поэтапное добавление функций** — новые примитивы и возможности добавляются в протокол без нарушения работы существующих реализаций: старые клиенты просто не объявляют новую возможность, и сервер не использует её.
2. **Независимая эволюция** — серверы и клиенты развиваются независимо друг от друга.
3. **Обратная совместимость** — клиент, не поддерживающий новую возможность, не получит от сервера вызовы, требующие этой возможности.

> Источник: [Wikipedia — Model Context Protocol](https://en.wikipedia.org/wiki/Model_Context_Protocol)

Согласно статье Wikipedia о MCP: MCP сравнивают с function calling — механизмом, представленным OpenAI в 2023 году. Оба подхода позволяют языковым моделям взаимодействовать с внешними инструментами, однако MCP предлагает стандартизированный, вендор-нейтральный протокол, тогда как традиционный function calling требует коннекторов, специфичных для поставщика.

Статья отмечает, что с ростом популярности MCP он стал де-факто отраслевым стандартом для интеграции инструментов с AI-системами, поддерживаемым крупнейшими игроками — Anthropic, OpenAI, Google DeepMind и Microsoft.

#### Ответ от Claude

**Function calling** (например, OpenAI Function Calling 2023) и **MCP** решают похожую задачу — подключение LLM к внешним инструментам — но на разных уровнях абстракции.

**Сравнение:**

| Аспект | Function Calling | MCP |
|--------|-----------------|-----|
| Привязка к поставщику | Да — специфично для OpenAI, Anthropic и т.д. | Нет — вендор-нейтральный открытый стандарт |
| Переносимость | Код инструмента привязан к конкретному провайдеру LLM | Один MCP-сервер работает с любым MCP-клиентом |
| Stateful-сессии | Нет | Да — поддерживает состояние и жизненный цикл |
| Дополнительные примитивы | Только вызов функций | Tools + Resources + Prompts + Sampling |
| Сложность реализации | Проще для одного провайдера | Сложнее, но масштабируемее |

**Когда выбирать Function Calling:**
- Простое приложение с одним LLM-провайдером.
- Минимальная задержка критична.
- Не планируется смена или добавление LLM-провайдеров.

**Когда выбирать MCP:**
- Агентные системы с несколькими инструментами и провайдерами.
- Требуется масштабируемость и переиспользование серверов.
- Нужны дополнительные примитивы (Resources, Prompts, Sampling).
- Важна независимость от конкретного LLM-вендора.

### Вопрос 18. Какие три основных примитива определяет MCP на стороне сервера и кто управляет каждым из них?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [MCP Architecture overview — modelcontextprotocol.io](https://modelcontextprotocol.io/docs/concepts/architecture)

Из документации по архитектуре MCP:

«MCP определяет три основных примитива, которые серверы могут предоставлять:
- **Tools** — исполняемые функции, которые AI-приложения могут вызывать для выполнения действий (например, файловые операции, API-вызовы, запросы к базам данных).
- **Resources** — источники данных, предоставляющие контекстную информацию AI-приложениям (например, содержимое файлов, записи БД, ответы API).
- **Prompts** — переиспользуемые шаблоны, помогающие структурировать взаимодействие с языковыми моделями (например, системные промпты, few-shot примеры)».

Из той же документации: Sampling «позволяет серверам запрашивать завершения языковой модели от AI-приложения клиента. Это полезно, когда авторы серверов хотят получить доступ к языковой модели, но хотят оставаться независимыми от модели».

#### Ответ от Claude

Три серверных примитива MCP и их управление:

**1. Tools — управляются моделью**

LLM самостоятельно решает, когда и какой инструмент вызвать, на основе понимания контекста и запроса пользователя. Хост должен получить согласие пользователя перед исполнением.

**2. Resources — управляются приложением**

Хост-приложение решает, какие ресурсы включить в контекст модели. Приложение может делать это автоматически (эвристики), по явному выбору пользователя (UI) или по запросу самой модели.

**3. Prompts — управляются пользователем**

Пользователь явно выбирает шаблон из предложенных. Это переиспользуемые рецепты для типовых взаимодействий с LLM.

**Сводная таблица:**

| Примитив | Кто управляет | Характер | Методы |
|----------|---------------|----------|--------|
| Tools | Модель | Активные действия | `tools/list`, `tools/call` |
| Resources | Приложение | Данные только для чтения | `resources/list`, `resources/read` |
| Prompts | Пользователь | Шаблоны взаимодействия | `prompts/list`, `prompts/get` |

### Вопрос 19. Что такое Tool в MCP? Из каких обязательных полей состоит его определение и как клиент вызывает инструмент?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Tools — MCP Specification (modelcontextprotocol.io)](https://modelcontextprotocol.io/docs/concepts/tools)

Из документации MCP по Tools:

«Model Context Protocol (MCP) позволяет серверам предоставлять инструменты, которые могут быть вызваны языковыми моделями. Инструменты позволяют моделям взаимодействовать с внешними системами, такими как запросы к базам данных, вызов API или выполнение вычислений. Каждый инструмент уникально идентифицируется именем и включает метаданные, описывающие его схему».

Инструменты в MCP разработаны с расчётом на управление моделью: языковая модель может автоматически обнаруживать и вызывать инструменты на основе своего контекстуального понимания.

Определение инструмента включает:
- `name`: уникальный идентификатор инструмента.
- `title`: необязательное человекочитаемое имя для отображения.
- `description`: человекочитаемое описание функциональности.
- `inputSchema`: JSON Schema, определяющий ожидаемые параметры.
- `outputSchema` (опционально): JSON Schema для ожидаемой структуры вывода.
- `annotations` (опционально): свойства, описывающие поведение инструмента.

Поток обнаружения и вызова: клиент сначала отправляет `tools/list` для получения списка инструментов, затем вызывает `tools/call` с именем и аргументами.

#### Ответ от Claude

**Tool (инструмент) в MCP** — исполняемая функция, которую языковая модель может вызвать для взаимодействия с внешними системами. Tools являются model-controlled примитивом: LLM самостоятельно решает, когда и какой инструмент вызвать.

**Обязательные поля определения инструмента:**

| Поле | Тип | Описание |
|------|-----|----------|
| `name` | строка | Уникальный идентификатор инструмента в рамках сервера |
| `description` | строка | Человекочитаемое описание функциональности |
| `inputSchema` | объект (JSON Schema) | Описание ожидаемых входных параметров |

Дополнительные поля: `title` (отображаемое имя), `outputSchema` (схема структурированного ответа), `annotations` (метаданные поведения — считаются ненадёжными, если сервер не является доверенным).

**Поток обнаружения и вызова:**

1. Клиент отправляет `tools/list` — получает массив доступных инструментов.
2. LLM анализирует список и решает, какой инструмент вызвать.
3. Клиент отправляет `tools/call` с полями `name` (точное имя инструмента) и `arguments` (параметры согласно `inputSchema`).
4. Сервер возвращает результат в поле `content` (массив объектов разных типов: text, image, resource_link и т.д.) и опциональном `structuredContent`.

**Обработка ошибок:** инструменты могут возвращать ошибки двумя способами — протокольные ошибки JSON-RPC (неизвестный инструмент, неверные аргументы) и ошибки выполнения инструмента (поле `isError: true` в результате).

### Вопрос 20. Что такое Resource в MCP? Как ресурс идентифицируется, какие URI-схемы поддерживаются и в каком виде возвращается содержимое?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Resources — MCP Specification (modelcontextprotocol.io)](https://modelcontextprotocol.io/docs/concepts/resources)

Из документации MCP по Resources:

«Model Context Protocol (MCP) предоставляет стандартизированный способ для серверов предоставлять ресурсы клиентам. Ресурсы позволяют серверам делиться данными, обеспечивающими контекст для языковых моделей, такими как файлы, схемы баз данных или специфичная для приложения информация. Каждый ресурс уникально идентифицируется URI».

Ресурсы в MCP разработаны с расчётом на **управление приложением**: хост-приложения определяют, как включать контекст, исходя из своих потребностей.

**Поддерживаемые URI-схемы:**
- `https://` — ресурс, доступный в вебе.
- `file://` — ресурсы, работающие как файловая система.
- `git://` — интеграция с системой контроля версий Git.
- Кастомные URI-схемы (должны соответствовать RFC 3986).

**Содержимое ресурса** — текстовое или бинарное:
- Текстовое: поле `text` со строковым содержимым.
- Бинарное: поле `blob` с данными в кодировке base64.

Для чтения ресурса клиент отправляет `resources/read` с URI ресурса.

#### Ответ от Claude

**Resource (ресурс) в MCP** — источник данных только для чтения, предоставляющий контекст языковой модели. В отличие от Tools (model-controlled), Resources являются **application-controlled**: хост-приложение решает, какие ресурсы включать в контекст.

**Идентификация ресурса:**

Каждый ресурс идентифицируется **URI согласно RFC 3986**. Примеры:
- `file:///project/src/main.rs`
- `https://example.com/api/docs`
- `git://repo/commit/abc123`

**Поддерживаемые URI-схемы:**

| Схема | Назначение |
|-------|-----------|
| `https://` | Ресурсы в вебе — клиент может получить напрямую |
| `file://` | Файлоподобные ресурсы (не обязательно реальная ФС) |
| `git://` | Интеграция с Git |
| Кастомные | По RFC 3986 |

**Структура содержимого ресурса:**

Ресурс возвращает либо текстовые данные, либо бинарные:

```json
// Текстовое содержимое
{ "uri": "...", "mimeType": "text/plain", "text": "..." }

// Бинарное содержимое (base64)
{ "uri": "...", "mimeType": "image/png", "blob": "<base64>" }
```

**Поток работы с ресурсами:**
1. `resources/list` — обнаружение доступных ресурсов.
2. `resources/read` с URI — получение содержимого.
3. `resources/subscribe` (опционально) — подписка на изменения ресурса.

Ресурсы также поддерживают **аннотации**: `audience` (для кого — user/assistant), `priority` (0.0–1.0), `lastModified` (ISO 8601 временная метка).
