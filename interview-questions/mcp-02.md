### Вопрос 21. Что такое Prompt в MCP и чем он отличается от Tool и Resource с точки зрения модели управления?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Prompts — MCP Specification (modelcontextprotocol.io)](https://modelcontextprotocol.io/docs/concepts/prompts)

Prompts (промпты) в MCP — это стандартизированный способ, с помощью которого серверы предоставляют клиентам шаблоны сообщений для взаимодействия с языковыми моделями. Промпты спроектированы как **управляемые пользователем (user-controlled)**: они экспонируются от сервера к клиенту с расчётом на то, что пользователь явно их выбирает.

Обычно промпты активируются через инициированные пользователем команды в интерфейсе — например, в виде slash-команд. При этом реализациям разрешено экспонировать промпты через любой подходящий интерфейсный паттерн; протокол не предписывает конкретную модель взаимодействия.

Три примитива MCP различаются по модели управления:
- **Tools** — управляются **моделью (model-controlled)**: языковая модель самостоятельно выбирает, когда вызвать инструмент.
- **Resources** — управляются **приложением (application-driven)**: хост-приложение решает, какой контекст включить.
- **Prompts** — управляются **пользователем (user-controlled)**: пользователь явно выбирает нужный шаблон.

Для работы с промптами клиент отправляет запрос `prompts/list`, получая список доступных промптов с их аргументами, и `prompts/get` с передачей значений аргументов, получая готовый набор сообщений для диалога с моделью. Серверы, поддерживающие промпты, обязаны объявить capability `prompts` при инициализации; если список промптов может меняться, указывается `listChanged: true`.

#### Ответ от Claude

**Prompt** в MCP — это переиспользуемый шаблон взаимодействия с языковой моделью. Сервер заранее готовит структурированные сообщения, которые клиент получает и подставляет в диалог с LLM.

Ключевое различие трёх серверных примитивов — в том, **кто инициирует использование**:

| Примитив | Кем управляется | Пример инициатора |
|---|---|---|
| **Tool** | Моделью | LLM сам решает вызвать `search_db` |
| **Resource** | Приложением | Хост автоматически добавляет схему БД в контекст |
| **Prompt** | Пользователем | Пользователь выбирает slash-команду `/code_review` |

Жизненный цикл промптов:
1. `prompts/list` — клиент получает список доступных промптов (поддерживает пагинацию).
2. `prompts/get` с аргументами — клиент получает готовые сообщения (`messages[]`) с ролями `user`/`assistant` для передачи в LLM.
3. `notifications/prompts/list_changed` — уведомление об изменении списка (если сервер объявил `listChanged: true`).

Промпт может включать аргументы (`required: true/false`), которые автодополняются через Completion API, и содержать контент разных типов: текст, изображение, аудио, встроенный ресурс.

### Вопрос 22. Как работает транспорт stdio в MCP: кто запускает сервер, как передаются сообщения и что разрешено писать в stdout?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Transports — MCP Specification 2025-03-26 (modelcontextprotocol.io)](https://modelcontextprotocol.io/specification/2025-03-26/basic/transports)

В **stdio**-транспорте:
- Клиент запускает MCP-сервер как дочерний процесс (subprocess).
- Сервер читает JSON-RPC сообщения из стандартного ввода (`stdin`) и отправляет сообщения в стандартный вывод (`stdout`).
- Сообщения разделяются символами новой строки и **не должны** содержать встроенных переводов строк.
- Сервер **МОЖЕТ** писать UTF-8 строки в стандартный поток ошибок (`stderr`) для нужд логирования. Клиент может перехватывать, пересылать или игнорировать эти логи.
- Сервер **НЕ ДОЛЖЕН** писать в `stdout` ничего, кроме валидных MCP-сообщений.
- Клиент **НЕ ДОЛЖЕН** писать в `stdin` сервера ничего, кроме валидных MCP-сообщений.

#### Ответ от Claude

Транспорт **stdio** — это самый простой и производительный вариант для локального развёртывания MCP.

**Архитектура:**
- Клиент запускает сервер как **дочерний процесс** (subprocess), сервер не является отдельным сервисом.
- Связь осуществляется исключительно через стандартные потоки процесса.

**Правила передачи сообщений:**

| Поток | Разрешено | Запрещено |
|---|---|---|
| `stdin` (вход сервера) | Только валидные MCP/JSON-RPC сообщения | Любой другой контент |
| `stdout` (выход сервера) | Только валидные MCP/JSON-RPC сообщения | Логи, отладочный вывод |
| `stderr` (ошибки сервера) | Строки логов в UTF-8 | — |

**Формат сообщений:** каждое JSON-RPC сообщение — это одна строка (newline-delimited). Встроенные символы новой строки (`\n`) внутри сообщения запрещены.

**Завершение работы:** клиент завершает соединение, закрывая `stdin` сервера; при необходимости отправляет `SIGTERM`, затем `SIGKILL`.

Stdio-транспорт обслуживает одного клиента на один экземпляр сервера и не требует сетевых настроек — изоляция обеспечивается средствами операционной системы.

### Вопрос 23. Как работает фаза инициализации MCP-соединения? Какие шаги выполняются и какую информацию передают клиент и сервер?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Lifecycle — MCP Specification 2025-03-26 (modelcontextprotocol.io)](https://modelcontextprotocol.io/specification/2025-03-26/basic/lifecycle)

Фаза инициализации **обязана** быть первым взаимодействием между клиентом и сервером. В ходе неё стороны устанавливают совместимость версий протокола, обмениваются и согласовывают capabilities, а также передают информацию об имплементации.

Клиент обязан инициировать эту фазу, отправив запрос `initialize`, содержащий:
- Поддерживаемую версию протокола (`protocolVersion`).
- Capabilities клиента.
- Информацию об имплементации клиента (`clientInfo`).

Сервер должен ответить своими capabilities и `serverInfo`.

После успешной инициализации клиент обязан отправить уведомление `notifications/initialized`, сигнализирующее о готовности к нормальной работе.

Ограничения в ходе инициализации:
- Клиент не должен отправлять запросы, кроме `ping`, до получения ответа сервера на `initialize`.
- Сервер не должен отправлять запросы, кроме `ping` и `logging`, до получения уведомления `initialized`.

#### Ответ от Claude

Инициализация MCP-соединения состоит из трёх обязательных шагов:

**Шаг 1 — Запрос клиента (`initialize`):**
```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "protocolVersion": "2025-03-26",
    "capabilities": { "roots": { "listChanged": true }, "sampling": {} },
    "clientInfo": { "name": "ExampleClient", "version": "1.0.0" }
  }
}
```

**Шаг 2 — Ответ сервера:**
```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "protocolVersion": "2025-03-26",
    "capabilities": {
      "prompts": { "listChanged": true },
      "resources": { "subscribe": true, "listChanged": true },
      "tools": { "listChanged": true }
    },
    "serverInfo": { "name": "ExampleServer", "version": "1.0.0" }
  }
}
```

**Шаг 3 — Уведомление клиента о готовности:**
```json
{ "jsonrpc": "2.0", "method": "notifications/initialized" }
```

**Что нельзя делать до завершения инициализации:**
- Клиент не отправляет обычные запросы (только `ping`) до получения ответа сервера.
- Сервер не отправляет запросы (только `ping` и `logging`) до получения `notifications/initialized`.

Запрос `initialize` не может быть частью JSON-RPC batch.

### Вопрос 24. Как сервер MCP объявляет поддержку инструментов, ресурсов и промптов? Что означает sub-capability listChanged?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Lifecycle — MCP Specification 2025-03-26 (modelcontextprotocol.io)](https://modelcontextprotocol.io/specification/2025-03-26/basic/lifecycle)

Клиент и сервер обмениваются capabilities в ходе инициализации. Ключевые серверные capabilities включают:

| Capability | Описание |
|---|---|
| `prompts` | Предлагает шаблоны промптов |
| `resources` | Предоставляет читаемые ресурсы |
| `tools` | Экспонирует вызываемые инструменты |
| `logging` | Отправляет структурированные лог-сообщения |
| `completions` | Поддерживает автодополнение аргументов |

Объекты capabilities могут описывать sub-capabilities:
- `listChanged`: поддержка уведомлений об изменении списка (для prompts, resources и tools).
- `subscribe`: поддержка подписок на изменения отдельных элементов (только для resources).

#### Ответ от Claude

Сервер объявляет свои capabilities в теле ответа на запрос `initialize` — в поле `result.capabilities`. Каждый поддерживаемый примитив указывается отдельным ключом.

**Пример объявления всех серверных capabilities:**
```json
{
  "capabilities": {
    "tools":     { "listChanged": true },
    "resources": { "subscribe": true, "listChanged": true },
    "prompts":   { "listChanged": true },
    "logging":   {}
  }
}
```

**Sub-capability `listChanged`** означает, что сервер будет активно уведомлять клиента при изменении списка:
- `notifications/tools/list_changed` — изменился список инструментов.
- `notifications/resources/list_changed` — изменился список ресурсов.
- `notifications/prompts/list_changed` — изменился список промптов.

Без `listChanged: true` клиент может периодически опрашивать список, но уведомлений не получит.

**Sub-capability `subscribe`** (только для resources) означает, что клиент может подписаться на изменения **конкретного** ресурса через `resources/subscribe`.

Если ключ capability отсутствует в ответе сервера — соответствующий примитив сервером не поддерживается.

### Вопрос 25. В чём разница между Streamable HTTP и устаревшим HTTP+SSE транспортом в MCP?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Transports — MCP Specification 2025-03-26 (modelcontextprotocol.io)](https://modelcontextprotocol.io/specification/2025-03-26/basic/transports)

**Streamable HTTP** (введён в спецификации версии 2025-03-26) заменяет устаревший **HTTP+SSE** транспорт (версия 2024-11-05).

В новом транспорте сервер предоставляет **единственный MCP-endpoint**, поддерживающий оба HTTP-метода:
- `POST` — для отправки сообщений от клиента к серверу.
- `GET` — для открытия SSE-потока и получения уведомлений от сервера.

В старом HTTP+SSE транспорте было **два разных endpoint-а**: отдельный SSE-endpoint для получения событий и отдельный POST-endpoint для отправки сообщений.

Для обратной совместимости клиент, желающий поддерживать оба транспорта, должен сначала попытаться отправить `POST` с `InitializeRequest`. Если запрос вернул успешный ответ — перед ним новый Streamable HTTP сервер. Если вернулся HTTP 4xx (например, 405 или 404) — необходимо отправить `GET` на тот же URL и ожидать SSE-поток с событием `endpoint`, характерным для старого транспорта.

#### Ответ от Claude

Streamable HTTP — это эволюция HTTP-транспорта MCP, упрощающая архитектуру за счёт унификации endpoint-ов.

**Сравнение транспортов:**

| Аспект | HTTP+SSE (2024-11-05, устарел) | Streamable HTTP (2025-03-26) |
|---|---|---|
| Endpoint-ы | Два отдельных: SSE-endpoint + POST-endpoint | Один MCP-endpoint (POST и GET) |
| Отправка сообщений | POST на отдельный URL | POST на единый MCP-endpoint |
| Получение событий | GET на отдельный SSE-URL | GET на тот же MCP-endpoint |
| Управление сессиями | Не стандартизировано | `Mcp-Session-Id` заголовок |
| Возобновление | Не предусмотрено | Через `Last-Event-ID` |

**Алгоритм определения версии транспорта на клиенте:**
1. Отправить `POST` с `InitializeRequest` на URL сервера.
2. Если ответ успешный — это Streamable HTTP сервер.
3. Если вернулся HTTP 4xx (405/404) — отправить `GET` на тот же URL, ожидать SSE-поток с событием `endpoint`. Это старый HTTP+SSE сервер.

Серверы, желающие поддерживать оба вида клиентов, должны держать оба набора endpoint-ов одновременно.

### Вопрос 26. Как Streamable HTTP транспорт MCP управляет сессиями? Что такое Mcp-Session-Id и как сервер его использует?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Transports — MCP Specification 2025-03-26 (modelcontextprotocol.io)](https://modelcontextprotocol.io/specification/2025-03-26/basic/transports)

Сессия MCP — это логически связанное взаимодействие между клиентом и сервером, начинающееся с фазы инициализации.

При Streamable HTTP сервер **может** назначить ID сессии, включив его в заголовок `Mcp-Session-Id` HTTP-ответа на `InitializeResult`.

Требования к session ID:
- Должен быть глобально уникальным и криптографически безопасным (например, UUID, JWT или криптографический хэш).
- Должен содержать только видимые ASCII-символы (диапазон 0x21–0x7E).

Если `Mcp-Session-Id` возвращён сервером при инициализации, клиент **обязан** включать его во все последующие HTTP-запросы.

Серверы, требующие session ID:
- Должны отвечать `400 Bad Request` на запросы без `Mcp-Session-Id` (кроме инициализации).
- Должны отвечать `404 Not Found` на запросы с неизвестным или истёкшим session ID.

При получении `404` клиент **обязан** начать новую сессию, отправив новый `InitializeRequest` без session ID. Клиент, завершающий работу, должен отправить `HTTP DELETE` на MCP-endpoint с `Mcp-Session-Id` для явного завершения сессии.

#### Ответ от Claude

Управление сессиями — опциональная функция Streamable HTTP транспорта, позволяющая серверу поддерживать состояние между запросами.

**Жизненный цикл сессии:**

1. **Создание:** сервер включает `Mcp-Session-Id` в ответ на `InitializeResult`.
2. **Использование:** клиент добавляет `Mcp-Session-Id` в заголовок каждого последующего запроса.
3. **Завершение клиентом:** `HTTP DELETE` на MCP-endpoint с `Mcp-Session-Id`.
4. **Завершение сервером:** сервер перестаёт принимать запросы с этим ID, отвечая `404`.

**Реакция на ошибки:**
- `400 Bad Request` — запрос пришёл без `Mcp-Session-Id`, а сервер его требует.
- `404 Not Found` — session ID неизвестен или истёк; клиент начинает новую инициализацию.

**Требования к session ID:**
- Глобально уникальный (UUID v4, JWT, криптографический хэш).
- Только видимые ASCII-символы (0x21–0x7E; пробелы и управляющие символы запрещены).

Без механизма сессий сервер остаётся **stateless**: каждый запрос обрабатывается независимо. Сессии нужны только тогда, когда сервер хранит состояние между запросами.

### Вопрос 27. Что такое Resource Templates в MCP? Зачем они нужны и как задаётся шаблон URI?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Resources — MCP Specification (modelcontextprotocol.io)](https://modelcontextprotocol.io/docs/concepts/resources)

Resource templates позволяют серверам предоставлять параметризованные ресурсы с использованием URI-шаблонов по стандарту [RFC 6570](https://datatracker.ietf.org/doc/html/rfc6570). Аргументы шаблонов могут автодополняться через Completion API. Эта операция поддерживает пагинацию.

Клиент получает список шаблонов через метод `resources/templates/list`. Пример ответа:

```json
{
  "resourceTemplates": [
    {
      "uriTemplate": "file:///{path}",
      "name": "Project Files",
      "description": "Access files in the project directory",
      "mimeType": "application/octet-stream"
    }
  ]
}
```

#### Ответ от Claude

**Resource Templates** — механизм динамической адресации ресурсов в MCP. Вместо того чтобы перечислять все ресурсы заранее, сервер объявляет шаблоны URI, по которым клиент может запросить любой подходящий ресурс.

**Зачем нужны:**
- Без шаблонов сервер должен заранее знать и перечислить все доступные ресурсы через `resources/list`.
- С шаблонами можно адресовать произвольные объекты: любой файл в файловой системе, любую запись в БД, любую страницу документации.

**Как задаётся шаблон URI:**
Формат — стандарт **RFC 6570** (URI Templates). Параметры указываются в фигурных скобках:

| Шаблон | Пример URI | Описание |
|---|---|---|
| `file:///{path}` | `file:///src/main.rs` | Любой файл по пути |
| `db:///{table}/{id}` | `db:///users/42` | Запись в таблице по ID |
| `gh://{owner}/{repo}` | `gh://torvalds/linux` | GitHub-репозиторий |

**Получение шаблонов:** `resources/templates/list` (с поддержкой пагинации).

**Автодополнение аргументов:** значения параметров шаблона могут подсказываться через Completion API — это позволяет клиенту предоставлять удобный UI для выбора ресурса.

Для чтения параметризованного ресурса клиент подставляет значения в шаблон и вызывает `resources/read` с получившимся URI.

### Вопрос 28. Как в MCP работают подписки на изменения ресурсов (resource subscriptions)? Как сервер объявляет поддержку этой функции?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Resources — MCP Specification (modelcontextprotocol.io)](https://modelcontextprotocol.io/docs/concepts/resources)

Протокол поддерживает опциональные подписки на изменения ресурсов. Клиент может подписаться на конкретные ресурсы и получать уведомления при их изменении.

Сервер объявляет поддержку подписок в capabilities при инициализации:
```json
{ "capabilities": { "resources": { "subscribe": true } } }
```

Клиент подписывается через запрос `resources/subscribe`:
```json
{ "jsonrpc": "2.0", "id": 4, "method": "resources/subscribe", "params": { "uri": "file:///project/src/main.rs" } }
```

При изменении ресурса сервер отправляет уведомление `notifications/resources/updated`:
```json
{ "jsonrpc": "2.0", "method": "notifications/resources/updated", "params": { "uri": "file:///project/src/main.rs" } }
```

Последовательность работы:
1. Клиент подписывается — `resources/subscribe`.
2. Ресурс изменяется.
3. Сервер уведомляет — `notifications/resources/updated`.
4. Клиент перечитывает ресурс — `resources/read`.

#### Ответ от Claude

Подписки на ресурсы — это **push-механизм** MCP, позволяющий клиенту получать актуальные данные без постоянного опроса сервера.

**Условие использования:** сервер должен объявить `capabilities.resources.subscribe: true` при инициализации. Если эта capability не объявлена, метод `resources/subscribe` недоступен.

**Полный жизненный цикл подписки:**

```
Клиент                           Сервер
  |                                |
  |-- resources/subscribe -------->|  (подписка на URI)
  |<-- подтверждение (ack) --------|
  |                                |  ... ресурс изменился ...
  |<-- notifications/resources/updated -- (URI изменённого ресурса)
  |-- resources/read ------------->|  (клиент перечитывает)
  |<-- актуальное содержимое ------|
```

**Важные детали:**
- Уведомление содержит только URI изменившегося ресурса, но **не само содержимое**. Клиент должен сделать отдельный `resources/read`.
- `subscribe` и `listChanged` — независимые sub-capabilities: `subscribe` отслеживает изменения **содержимого** конкретного ресурса, `listChanged` — изменения **состава** списка ресурсов.
- Поддержка обеих, одной или ни одной из sub-capabilities допустима.

### Вопрос 29. Что такое tool annotations в MCP? Перечислите четыре hint-поля и объясните их значения по умолчанию.

**Категория:** Model Context Protocol (MCP) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Tool Annotations as Risk Vocabulary — MCP Blog (blog.modelcontextprotocol.io)](https://blog.modelcontextprotocol.io/posts/2026-03-16-tool-annotations/)

Tool annotations — механизм, позволяющий серверам описывать поведение инструментов через метаданные. Согласно спецификации, это «подсказки» (hints), а не гарантии. Четыре основных поля и их значения по умолчанию:

| Поле | По умолчанию | Назначение |
|---|---|---|
| `readOnlyHint` | `false` | Изменяет ли инструмент окружение? |
| `destructiveHint` | `true` | Деструктивны ли изменения? |
| `idempotentHint` | `false` | Безопасен ли повторный вызов? |
| `openWorldHint` | `true` | Взаимодействует ли с внешними сущностями? |

Дефолты намеренно осторожны: инструмент без аннотаций по умолчанию считается способным изменять среду (`readOnlyHint: false`), деструктивным (`destructiveHint: true`), небезопасным при повторном вызове (`idempotentHint: false`) и обращающимся к внешним сервисам (`openWorldHint: true`).

#### Ответ от Claude

**Tool annotations** — это необязательные метаданные инструмента (поле `annotations` в объекте Tool), дающие клиенту и пользователю подсказки о поведении инструмента без его вызова.

**Четыре hint-поля:**

| Поле | Тип | По умолчанию | Семантика |
|---|---|---|---|
| `readOnlyHint` | boolean | `false` | `true` — инструмент только читает данные, не изменяет среду |
| `destructiveHint` | boolean | `true` | `true` — изменения могут быть необратимы (удаление, перезапись) |
| `idempotentHint` | boolean | `false` | `true` — повторный вызов с теми же аргументами безопасен |
| `openWorldHint` | boolean | `true` | `true` — инструмент обращается к внешним сервисам/интернету |

**Логика умолчаний — принцип «худшего случая»:**
- Инструмент без аннотаций считается наиболее опасным вариантом.
- `destructiveHint: true` по умолчанию означает: если сервер не сказал, что инструмент безопасен — считай его деструктивным.
- `readOnlyHint: false` по умолчанию — если не заявлено иное, инструмент может изменять данные.

Это позволяет клиентам применять консервативную политику для инструментов без аннотаций и автоматизировать подтверждения (например, не требовать confirm для `readOnlyHint: true, idempotentHint: true`).

### Вопрос 30. Почему tool annotations в MCP считаются hints, а не гарантиями безопасности? Как это влияет на архитектуру безопасности?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Tool Annotations as Risk Vocabulary — MCP Blog (blog.modelcontextprotocol.io)](https://blog.modelcontextprotocol.io/posts/2026-03-16-tool-annotations/)

Спецификация явно определяет: **«аннотации не гарантируют достоверного описания поведения инструмента»** (annotations are not guaranteed to faithfully describe tool behavior).

Ключевые причины:
1. **Ненадёжный сервер может лгать** — сервер может заявить `readOnlyHint: true`, но в действительности удалять файлы.
2. **Аннотации не защищают от prompt injection** — они не влияют на поведение LLM, а лишь информируют клиентскую сторону.
3. **Опасность зависит от контекста** — уровень риска инструмента определяется комбинацией всех инструментов в сессии и политиками клиента.

Аннотации работают как входные данные для политик клиента, а не как принудительные механизмы безопасности.

Согласно спецификации MCP, клиенты **MUST** считать аннотации инструментов ненадёжными, если сервер не является доверенным:
> *«For trust & safety and security, clients MUST consider tool annotations to be untrusted unless they come from trusted servers.»*

#### Ответ от Claude

Аннотации — это **декларации сервера о своём намерении**, но не технически верифицированные ограничения. Любой сервер может объявить произвольные значения аннотаций.

**Почему это не механизм безопасности:**

1. **Нет верификации на уровне протокола.** MCP не проверяет, что инструмент с `readOnlyHint: true` действительно не пишет данные.
2. **Ненадёжный сервер может злоупотребить.** Вредоносный сервер объявит все инструменты как `readOnly` и `idempotent`, чтобы обойти пользовательские подтверждения.
3. **Prompt injection не блокируется аннотациями.** Атака может прийти через контент ресурсов, а не через metadata инструментов.

**Правильная архитектура безопасности MCP-клиента:**

| Уровень | Механизм |
|---|---|
| Сетевой | Изоляция, firewall, принцип наименьших привилегий |
| Аутентификация | OAuth, API-ключи, проверка identity сервера |
| Авторизация | Политики допустимых операций для конкретного сервера |
| UX-слой | Аннотации используются здесь: для информирования пользователя и автоматизации confirm-диалогов |
| Аудит | Логирование всех вызовов инструментов |

Аннотации допустимо использовать для **улучшения UX** (не требовать подтверждения для явно read-only операций) при условии, что сервер — доверенный.

### Вопрос 31. Каковы два механизма сообщений об ошибках при вызове Tool в MCP и чем они различаются?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Tools — MCP Specification (modelcontextprotocol.io)](https://modelcontextprotocol.io/docs/concepts/tools)

Tools используют два механизма сообщения об ошибках:

1. **Protocol Errors** — стандартные JSON-RPC ошибки для таких случаев, как:
   - Неизвестный инструмент.
   - Неверные аргументы.
   - Ошибки сервера.

2. **Tool Execution Errors** — ошибки выполнения, возвращаемые в результате вызова с `isError: true`:
   - Сбои API.
   - Некорректные входные данные.
   - Ошибки бизнес-логики.

Пример Protocol Error:
```json
{ "jsonrpc": "2.0", "id": 3, "error": { "code": -32602, "message": "Unknown tool: invalid_tool_name" } }
```

Пример Tool Execution Error:
```json
{ "jsonrpc": "2.0", "id": 4, "result": { "content": [{ "type": "text", "text": "Failed to fetch weather data: API rate limit exceeded" }], "isError": true } }
```

#### Ответ от Claude

MCP разграничивает два вида ошибок при работе с инструментами, потому что они имеют разную природу и разных адресатов.

**Сравнение двух механизмов:**

| Аспект | Protocol Error | Tool Execution Error |
|---|---|---|
| Структура ответа | Поле `error` в JSON-RPC ответе | Поле `result` с `isError: true` |
| Когда возникает | До или вместо выполнения инструмента | В процессе выполнения инструмента |
| Примеры | Неизвестный инструмент (`-32602`), невалидные аргументы, сервер недоступен | Ошибка внешнего API, превышен лимит, файл не найден |
| Адресат | Транспортный/инфраструктурный уровень клиента | Языковая модель и пользователь |

**Ключевое различие:** Tool Execution Error — это **успешный с точки зрения протокола ответ**, содержащий информацию об ошибке предметной области. Это позволяет модели получить контекст ошибки (например, «API rate limit exceeded») и принять решение — повторить попытку позже или сообщить пользователю, не разрывая RPC-взаимодействие.

Protocol Error же означает системную проблему: запрос некорректен или сервер не может его обработать вообще.

### Вопрос 32. Как MCP обеспечивает обратную совместимость Streamable HTTP с устаревшим HTTP+SSE транспортом? Какой алгоритм определения версии транспорта на клиенте?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Transports — MCP Specification 2025-03-26 (modelcontextprotocol.io)](https://modelcontextprotocol.io/specification/2025-03-26/basic/transports)

Клиенты и серверы могут поддерживать обратную совместимость с устаревшим HTTP+SSE транспортом (версия протокола 2024-11-05) следующим образом:

**Серверы**, желающие поддерживать старых клиентов, должны:
- Продолжать хостить как SSE-endpoint, так и POST-endpoint старого транспорта, одновременно с новым MCP-endpoint.

**Клиенты**, желающие поддерживать старые серверы, должны:
1. Принять URL MCP-сервера (может указывать на старый или новый транспорт).
2. Попытаться отправить `POST` с `InitializeRequest` на URL сервера с заголовком `Accept`:
   - Если запрос успешен — перед клиентом новый Streamable HTTP сервер.
   - Если вернулся HTTP 4xx (например, 405 или 404):
     - Отправить `GET` на тот же URL, ожидая SSE-поток.
     - Когда придёт событие `endpoint` — перед клиентом старый HTTP+SSE сервер; использовать этот транспорт для всей последующей коммуникации.

#### Ответ от Claude

Задача совместимости возникла потому, что Streamable HTTP (2025-03-26) полностью заменил HTTP+SSE (2024-11-05), но существующие серверы обновляются постепенно.

**Алгоритм автоопределения транспорта на клиенте:**

```
1. POST InitializeRequest → URL сервера
        |
        ├── HTTP 2xx → Streamable HTTP сервер (новый транспорт)
        │              Продолжать работу по новому протоколу
        │
        └── HTTP 4xx (405/404) → GET → URL сервера
                |
                └── SSE-поток с событием "endpoint"
                    → HTTP+SSE сервер (старый транспорт)
                       Использовать старый протокол
```

**Требования к серверу для поддержки обоих видов клиентов:**
- Держать **оба** набора endpoint-ов одновременно:
  - Новый единый MCP-endpoint (POST + GET).
  - Старые раздельные SSE-endpoint и POST-endpoint.
- Возможно объединить старый POST-endpoint и новый MCP-endpoint, но это может усложнить реализацию.

Такой подход обеспечивает плавную миграцию экосистемы: новые клиенты используют Streamable HTTP, старые — автоматически переключаются на HTTP+SSE.

### Вопрос 33. Как Streamable HTTP транспорт MCP поддерживает возобновление разорванного соединения? Как используется механизм Last-Event-ID?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Transports — MCP Specification 2025-03-26 (modelcontextprotocol.io)](https://modelcontextprotocol.io/specification/2025-03-26/basic/transports)

Для поддержки возобновления разорванных соединений и повторной доставки потерянных сообщений:

1. Серверы **могут** присваивать поле `id` своим SSE-событиям (согласно стандарту SSE). Если ID присутствует, он должен быть **глобально уникальным** в рамках всех потоков сессии.

2. Если клиент хочет возобновить соединение после разрыва, он должен отправить HTTP `GET` на MCP-endpoint с заголовком `Last-Event-ID`, указывающим ID последнего полученного события.
   - Сервер **может** воспроизвести сообщения, которые были бы отправлены после этого ID на том же потоке.
   - Сервер **не должен** воспроизводить сообщения, которые были бы доставлены на другом потоке.

3. IDs событий присваиваются серверами на основе **потока** (per-stream), выступая курсором внутри конкретного потока.

Разрыв соединения **не должен** интерпретироваться как отмена запроса. Для явной отмены клиент отправляет `CancelledNotification`.

#### Ответ от Claude

Механизм возобновления основан на стандарте SSE (Server-Sent Events) и решает проблему потери сообщений при нестабильных сетевых соединениях.

**Как работает Last-Event-ID:**

```
Нормальная работа:
  Сервер → SSE event id="evt-42" data=...
  Сервер → SSE event id="evt-43" data=...
  [соединение разрывается]

Возобновление клиентом:
  GET /mcp
  Last-Event-ID: evt-42

Сервер повторно доставляет:
  SSE event id="evt-43" data=... (было потеряно)
  [и продолжает с текущего места]
```

**Ключевые правила:**
- Event ID должен быть **глобально уникальным** в рамках сессии (или клиента, если сессий нет).
- Сервер воспроизводит только события **того же потока** (stream), что был разорван. Если событие было на другом потоке — повторная доставка запрещена.
- Присвоение ID — опционально (`MAY`). Сервер может не поддерживать возобновление.
- Разрыв TCP-соединения ≠ отмена запроса. Для отмены нужен явный `CancelledNotification`.

Это принципиально важно для долгоживущих агентских задач: если соединение упало во время длительного вызова инструмента, клиент может подключиться заново и получить результат, а не запускать задачу заново.

### Вопрос 34. Как MCP решает проблему согласования версий протокола при несовместимости клиента и сервера?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Lifecycle — MCP Specification 2025-03-26 (modelcontextprotocol.io)](https://modelcontextprotocol.io/specification/2025-03-26/basic/lifecycle)

В запросе `initialize` клиент **обязан** отправить версию протокола, которую он поддерживает — это должна быть **последняя** поддерживаемая версия.

Если сервер поддерживает запрошенную версию — он **обязан** ответить той же версией. Если нет — сервер **обязан** ответить другой поддерживаемой версией (предпочтительно своей последней).

Если клиент **не поддерживает** версию, возвращённую сервером, он **должен** разорвать соединение.

Пример ответа сервера при несовместимой версии:
```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32602,
    "message": "Unsupported protocol version",
    "data": { "supported": ["2024-11-05"], "requested": "1.0.0" }
  }
}
```

#### Ответ от Claude

Согласование версий протокола происходит в ходе фазы инициализации по принципу «клиент предлагает — сервер принимает или контрпредлагает».

**Алгоритм согласования:**

```
Клиент                                   Сервер
  |                                         |
  |-- initialize (protocolVersion: "2025-03-26") -->|
  |                                         |
  |     [сервер поддерживает 2025-03-26?]   |
  |                                         |
  | ДА: |<-- result (protocolVersion: "2025-03-26") --|
  |     Соединение установлено              |
  |                                         |
  | НЕТ:|<-- result (protocolVersion: "2024-11-05") --|
  |     [клиент поддерживает 2024-11-05?]  |
  | ДА: Продолжить работу                   |
  | НЕТ: РАЗОРВАТЬ СОЕДИНЕНИЕ               |
```

**Правила:**
1. Клиент всегда предлагает свою **последнюю** поддерживаемую версию.
2. Если сервер её поддерживает — возвращает ту же версию.
3. Если нет — возвращает **свою лучшую** поддерживаемую версию.
4. Если клиент не поддерживает версию из ответа сервера — он **обязан** разорвать соединение.

**При критической несовместимости** сервер возвращает JSON-RPC ошибку `-32602` с полем `data.supported` — массивом поддерживаемых версий. Это позволяет клиенту выдать понятное сообщение пользователю о необходимости обновления.

### Вопрос 35. Сравните архитектурные сценарии применения stdio и Streamable HTTP транспортов MCP с точки зрения масштабирования, безопасности и размещения сервера.

**Категория:** Model Context Protocol (MCP) · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [MCP Architecture overview — modelcontextprotocol.io](https://modelcontextprotocol.io/docs/concepts/architecture)

MCP поддерживает два транспортных механизма:

- **Stdio-транспорт:** использует стандартные потоки ввода/вывода для прямого взаимодействия процессов на одной машине. Оптимальная производительность без сетевых накладных расходов.
- **Streamable HTTP-транспорт:** использует HTTP POST для сообщений от клиента к серверу с опциональными Server-Sent Events для стриминга. Обеспечивает удалённое взаимодействие и поддерживает стандартные HTTP-методы аутентификации — bearer-токены, API-ключи, произвольные заголовки. MCP рекомендует использовать OAuth для получения токенов аутентификации.

Локальные MCP-серверы с stdio-транспортом, как правило, обслуживают одного MCP-клиента; удалённые серверы с Streamable HTTP-транспортом обычно обслуживают множество MCP-клиентов.

Пример: Visual Studio Code — MCP-хост. При подключении к удалённому серверу Sentry использует Streamable HTTP; при подключении к локальному серверу файловой системы — stdio.

#### Ответ от Claude

**Сравнительная таблица транспортов:**

| Аспект | stdio | Streamable HTTP |
|---|---|---|
| Где работает сервер | Локально, как subprocess клиента | Независимый процесс, локально или удалённо |
| Клиентов на сервер | Один (1:1) | Множество (1:N) |
| Сетевые расходы | Нет | Есть (HTTP overhead) |
| Аутентификация | ОС-уровень (процессная изоляция) | Bearer-токены, OAuth, API-ключи |
| Авторизация | Права пользователя ОС | Явная (на уровне HTTP) |
| DNS rebinding атаки | Невозможны (нет сети) | Требуется проверка Origin-заголовка |
| Масштабирование | Не масштабируется (один клиент) | Горизонтальное масштабирование |
| Развёртывание | Локальный бинарный файл / npm-пакет | Docker, облако, микросервис |
| Типичный use case | Файловая система, локальные CLI-инструменты | Внешние API, корпоративные сервисы, SaaS |

**Рекомендации по выбору:**
- **stdio** — для инструментов, работающих с ресурсами текущего пользователя (файлы, локальная БД, shell-команды). Безопаснее для чувствительных данных.
- **Streamable HTTP** — для централизованных сервисов, команд разработки (один сервер для всех), облачных интеграций, когда требуется стандартная HTTP-аутентификация.

При Streamable HTTP серверы **обязаны** валидировать Origin-заголовок для защиты от DNS rebinding. При локальном запуске следует биндиться только на 127.0.0.1.

### Вопрос 36. Что такое Model Context Protocol (MCP) и чем он отличается от обычного function calling?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [MCP vs. Function Calling: How They Differ and Which to Use (Descope)](https://www.descope.com/blog/post/mcp-vs-function-calling)

**Model Context Protocol (MCP)** — открытый стандарт, разработанный Anthropic в ноябре 2024 года. Это протокол на основе клиент-серверной архитектуры, обеспечивающий универсальную базу для интеграции инструментов ИИ.

Основные отличия:

| Аспект | Function Calling | MCP |
|---|---|---|
| Архитектура | Инструменты встроены в запросы LLM | Отдельные серверы с протоколом |
| Связанность | Плотная связь с приложением | Модульная, развязанная архитектура |
| Переносимость | Зависит от провайдера LLM | Работает с любым провайдером |
| Безопасность учётных данных | На уровне приложения | Изолированные на сервере |
| Сложность | Простая реализация | Требует больше начальной работы |

Преимущества MCP:
1. **Переиспользуемость**: MCP-сервер создаётся один раз и используется любым совместимым клиентом.
2. **Независимость от провайдера**: один сервер работает с OpenAI и Claude без переписывания кода.
3. **Безопасность**: принцип наименьших привилегий — приложение получает доступ только к явно предоставленному.
4. **Масштабируемость**: каждый инструмент работает независимо и масштабируется отдельно.

#### Ответ от Claude

**Function calling** — это механизм, встроенный в API языковых моделей: разработчик включает описания функций прямо в запрос к LLM, модель возвращает структурированный JSON с именем функции и аргументами, приложение вызывает функцию и передаёт результат обратно.

**MCP** — это стандартизированный протокол (JSON-RPC 2.0) с клиент-серверной архитектурой, где инструменты работают в отдельных процессах-серверах.

**Ключевые различия:**

| Аспект | Function Calling | MCP |
|---|---|---|
| Где живут инструменты | В коде приложения / в запросе к LLM | В отдельном MCP-сервере |
| Учётные данные | Приложение держит все секреты | Каждый сервер хранит свои секреты |
| Переиспользование | Нет (переписывать под каждый LLM) | Да (один сервер для Claude, GPT и др.) |
| Обнаружение инструментов | Статическое (в коде) | Динамическое (`tools/list` + уведомления) |
| Транспорт | Нет (часть HTTP-запроса к LLM) | stdio или Streamable HTTP |

**Аналогия:** function calling — как прямые звонки по номерам телефона, MCP — как стандартизированный телефонный протокол, позволяющий любому устройству дозвониться до любого сервиса.

### Вопрос 37. Какие три типа примитивов определяет MCP на стороне сервера и как они различаются по модели взаимодействия?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Model Context Protocol — официальная документация: Architecture](https://modelcontextprotocol.io/docs/concepts/architecture)

MCP определяет три основных серверных примитива:

- **Tools** — исполняемые функции, которые ИИ-приложения могут вызывать для выполнения действий (например, операции с файлами, вызовы API, запросы к БД).
- **Resources** — источники данных, предоставляющие контекстную информацию ИИ-приложениям (например, содержимое файлов, записи БД, ответы API).
- **Prompts** — переиспользуемые шаблоны, помогающие структурировать взаимодействие с языковыми моделями (например, системные промпты, few-shot примеры).

У каждого типа примитива есть ассоциированные методы для обнаружения (`*/list`), получения (`*/get`) и в ряде случаев выполнения (`tools/call`). MCP-клиенты используют методы `*/list` для динамического обнаружения доступных примитивов.

Для примера: MCP-сервер, предоставляющий контекст о базе данных, может экспонировать инструменты для запросов к БД, ресурс со схемой БД и промпт с few-shot примерами для работы с инструментами.

#### Ответ от Claude

Три серверных примитива MCP различаются прежде всего тем, **кто управляет их использованием** и **какую роль они играют в диалоге**:

| Примитив | Управляется | Роль | Методы |
|---|---|---|---|
| **Tools** | Моделью (model-controlled) | Действия: запросы, вызовы API, вычисления | `tools/list`, `tools/call` |
| **Resources** | Приложением (application-driven) | Контекст: данные, файлы, схемы | `resources/list`, `resources/read`, `resources/templates/list` |
| **Prompts** | Пользователем (user-controlled) | Шаблоны диалога, slash-команды | `prompts/list`, `prompts/get` |

**Детальное сравнение:**

- **Tools** — модель самостоятельно решает вызвать инструмент на основе контекста разговора. Результат возвращается в `result.content`. Примеры: `search_web`, `run_sql`, `send_email`.

- **Resources** — хост-приложение определяет, какие данные добавить в контекст. Ресурс идентифицируется по URI (например, `file:///project/schema.sql`). Поддерживают подписки на изменения.

- **Prompts** — пользователь явно выбирает шаблон (через UI, slash-команду). Сервер возвращает готовый набор сообщений с ролями `user`/`assistant` для подачи в LLM.

Все три примитива поддерживают уведомление `list_changed` при изменении своего состава, что позволяет клиенту динамически обновлять информацию без перезапуска.

### Вопрос 38. Какие транспортные механизмы поддерживает MCP и в чём их ключевое различие?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Model Context Protocol — официальная документация: Transports](https://modelcontextprotocol.io/docs/concepts/transports)

MCP использует JSON-RPC для кодирования сообщений. JSON-RPC сообщения должны быть в кодировке UTF-8.

Протокол определяет два стандартных транспортных механизма:

1. **stdio** — коммуникация через стандартный ввод/вывод:
   - Клиент запускает MCP-сервер как дочерний процесс.
   - Сервер читает JSON-RPC сообщения из `stdin` и пишет в `stdout`.
   - Сообщения разделяются символами новой строки.
   - Сервер не должен писать в `stdout` ничего, кроме валидных MCP-сообщений.
   - Логи допускается писать в `stderr`.

2. **Streamable HTTP** — заменяет устаревший HTTP+SSE транспорт:
   - Сервер — независимый процесс, может обслуживать множество клиентов.
   - Использует HTTP POST (сообщения от клиента к серверу) и опционально SSE для стриминга.
   - Сервер предоставляет единственный MCP-endpoint, поддерживающий и POST, и GET.

Клиенты должны поддерживать stdio везде, где это возможно. Допускается реализация кастомных транспортов.

#### Ответ от Claude

MCP определяет два стандартных транспорта — они охватывают два принципиально разных сценария развёртывания:

**stdio — для локальных процессов:**
- Клиент запускает сервер как subprocess своего же процесса.
- Обмен через `stdin`/`stdout`; логи в `stderr`.
- Один клиент — один сервер.
- Нет сетевых накладных расходов и сетевой поверхности атаки.
- Безопасность — изоляция процессов ОС.

**Streamable HTTP — для удалённых сервисов:**
- Сервер — отдельный независимый HTTP-сервис.
- Один сервер обслуживает множество клиентов одновременно.
- Поддерживает OAuth, bearer-токены, API-ключи.
- Единый MCP-endpoint: `POST` для запросов, `GET` для SSE-потока уведомлений.
- Опциональные сессии через `Mcp-Session-Id`.
- Требует защиты от DNS rebinding (проверка Origin).

**Главное различие:** stdio — это **локальное IPC** (inter-process communication), Streamable HTTP — это **распределённый сетевой протокол**. Выбор определяется тем, где живёт сервер: рядом с клиентом или в сети.

Кроме стандартных транспортов, спецификация допускает создание кастомных транспортов при условии сохранения формата JSON-RPC и lifecycle-требований MCP.

### Вопрос 39. Когда предпочтительнее использовать function calling, а когда MCP? Назовите конкретные критерии выбора.

**Категория:** Model Context Protocol (MCP) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [MCP vs Function Calling — How They Actually Work Together (Portkey)](https://portkey.ai/blog/mcp-vs-function-calling/)

Фундаментальное различие в роли: **function calling — это фаза 1, генерация намерения**; **MCP — это фаза 2, стандартизированная инфраструктура выполнения**.

| Фактор | Function calling | MCP |
|---|---|---|
| Количество инструментов | Менее 5 | 10 и более |
| Провайдеры LLM | Один | Несколько |
| Команды-потребители | Одна | Несколько |
| Сложность настройки | ~20 строк кода | Отдельный сервер |

На практике зрелые production-системы часто комбинируют оба подхода: function calling обрабатывает запрос инструмента модели внутри цикла разговора, тогда как MCP управляет маршрутизацией и выполнением на инфраструктурном уровне.

Ключевой вывод: MCP не заменяет function calling, а дополняет его, решая проблемы масштабирования, фрагментации форматов между провайдерами и управления множеством инструментов.

#### Ответ от Claude

Function calling и MCP решают задачи на разных уровнях, поэтому выбор зависит от контекста.

**Критерии выбора function calling:**
- Прототип или небольшой проект (1–5 инструментов).
- Один LLM-провайдер, и смена не планируется.
- Инструменты тесно связаны с конкретным приложением.
- Скорость реализации важнее переиспользуемости (~20 строк кода против отдельного сервера).
- Все учётные данные уже есть в основном приложении.

**Критерии выбора MCP:**
- 10+ инструментов с перспективой роста.
- Несколько команд или продуктов используют одни и те же инструменты.
- Требуется мультипровайдерная архитектура (Claude + GPT + другие).
- Инструменты работают с чувствительными данными — нужна изоляция учётных данных.
- Инструменты должны обновляться без перевыката основного приложения.
- Требуется централизованное логирование и управление инструментами.

**Как они работают вместе в production:**
```
LLM (function calling) → генерирует tool_call JSON
       ↓
MCP-клиент → маршрутизирует вызов на нужный MCP-сервер
       ↓
MCP-сервер → выполняет инструмент, возвращает result
       ↓
MCP-клиент → передаёт результат обратно в LLM
```

Function calling выражает **намерение** модели; MCP обеспечивает **выполнение** этого намерения стандартизированным способом.

### Вопрос 40. Как в MCP реализовано динамическое обнаружение инструментов и почему это принципиально важно для агентских систем?

**Категория:** Model Context Protocol (MCP) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Model Context Protocol — официальная документация: Tools](https://modelcontextprotocol.io/docs/concepts/tools)

Для обнаружения доступных инструментов клиенты отправляют запрос `tools/list`. Эта операция поддерживает пагинацию.

Sub-capability `listChanged: true` указывает, что сервер будет отправлять уведомления при изменении списка доступных инструментов:
```json
{ "jsonrpc": "2.0", "method": "notifications/tools/list_changed" }
```

Получив это уведомление, клиент повторно запрашивает `tools/list` для получения актуального списка. Такая схема создаёт цикл обновления, поддерживающий актуальность представления клиента о доступных инструментах.

Значение для ИИ-приложений: при получении уведомления об изменении инструментов приложение немедленно обновляет реестр инструментов и доступные возможности LLM, что позволяет продолжающимся разговорам всегда иметь доступ к актуальному набору инструментов.

#### Ответ от Claude

**Динамическое обнаружение инструментов** в MCP реализовано через двухступенчатый механизм: начальный pull и последующие push-уведомления.

**Механизм работы:**

```
1. Инициализация:
   Клиент → tools/list → Сервер
   Клиент ← список инструментов ← Сервер
   (клиент сохраняет список, регистрирует в LLM)

2. Обновление при изменении:
   Сервер → notifications/tools/list_changed → Клиент
   Клиент → tools/list → Сервер            (повторный запрос)
   Клиент ← обновлённый список ← Сервер
   (клиент обновляет реестр, уведомляет LLM)
```

**Почему это критично для агентских систем:**

| Свойство | Function Calling | MCP (динамическое) |
|---|---|---|
| Список инструментов | Фиксирован в запросе к LLM | Обновляется во время работы |
| Добавление нового инструмента | Требует изменения кода | Сервер уведомляет, клиент обновляется |
| Длительные агентские сессии | Список устаревает | Всегда актуален |
| A/B замена инструмента | Перевыкат приложения | Прозрачная замена |

Для **агентских систем** с длительными задачами (автономные агенты, workflow-автоматизация) динамичность принципиальна: агент, работающий несколько часов, может получить доступ к новым инструментам, добавленным администратором, без перезапуска. Это позволяет строить адаптивные агентские среды, где набор возможностей системы меняется в ответ на события реального мира.
