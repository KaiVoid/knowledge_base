### Вопрос 21. Что такое ISR (In-Sync Replicas) в Kafka? Что происходит, если ISR-список сужается до одной реплики, и как это влияет на гарантии durability при acks=all?

**Категория:** Брокеры сообщений · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Kafka Replication — Confluent Documentation](https://docs.confluent.io/kafka/design/replication.html)

ISR (In-Sync Replicas) — это динамически поддерживаемый набор реплик партиции, которые синхронизированы с лидером. Лидер отслеживает, какие реплики не отстают. Критерий «синхронности» — реплика не должна отставать от лидера дольше, чем `replica.lag.time.max.ms` (по умолчанию 30 секунд). Если реплика-последователь зависает или перестаёт подтягивать сообщения, лидер исключает её из ISR. Когда реплика наверстала отставание, лидер добавляет её обратно.

При настройке `acks=all` продюсер ожидает подтверждения записи от всех реплик в текущем ISR. Параметр `min.insync.replicas` задаёт минимально допустимый размер ISR: если ISR оказывается меньше этого значения, брокер отклоняет записи с исключением `NotEnoughReplicasException`. Если `min.insync.replicas` не настроен и ISR сужается до одной реплики (лидера), `acks=all` фактически эквивалентно `acks=1` — данные подтверждаются без реальной репликации.

#### Ответ от Claude

**Что такое ISR**

ISR (In-Sync Replicas) — множество реплик конкретной партиции, которые «в синхронизации» с лидером. Для каждой партиции существует один лидер и нуль или более последователей (followers). Каждый последователь постоянно тянет (pull) сообщения от лидера по одному сокет-каналу.

Реплика считается «в синхронизации» (in-sync), если:
1. Она поддерживает активную сессию с контроллером кластера (через heartbeat в ZooKeeper / KRaft).
2. Она не отстаёт от лидера дольше `replica.lag.time.max.ms` (по умолчанию 30 секунд).

Если реплика нарушает хотя бы одно из этих условий, лидер исключает её из ISR. Когда реплика полностью наверстала отставание, лидер возвращает её в ISR.

**Жизненный цикл ISR:**

```
Нормальная работа:  ISR = {leader, follower-1, follower-2}
Follower-2 завис:   ISR = {leader, follower-1}       ← через replica.lag.time.max.ms
Follower-2 догнал:  ISR = {leader, follower-1, follower-2}
```

**Как acks=all взаимодействует с ISR**

При `acks=all` (синоним `acks=-1`) продюсер получает подтверждение только после того, как сообщение записано лидером и всеми текущими членами ISR. Это гарантирует: если любая из in-sync реплик выживет, данные не будут потеряны.

**Сужение ISR до одной реплики**

Когда в ISR остаётся только лидер, `acks=all` фактически деградирует до `acks=1`: лидер подтверждает запись немедленно, не дожидаясь ни одной другой реплики. Если лидер упадёт в этот момент до репликации данных, сообщение будет потеряно — несмотря на `acks=all`.

```
ISR = {leader}  +  acks=all  →  эффективно acks=1  →  риск потери данных
```

**Защита: min.insync.replicas**

Параметр `min.insync.replicas` (на уровне топика или брокера) задаёт минимально допустимый размер ISR для принятия записи. Если `|ISR| < min.insync.replicas`, брокер отклоняет запись с исключением `NotEnoughReplicasException`.

Рекомендуемая конфигурация для strong durability:

```properties
# Топик или брокер
replication.factor=3
min.insync.replicas=2

# Продюсер
acks=all
enable.idempotence=true
```

При такой конфигурации:
- Допускается выход из строя одного брокера из трёх без потери данных.
- Запись не будет принята, если ISR сократится до 1 (менее `min.insync.replicas=2`), что лучше, чем тихая потеря данных.

**Компромисс availability vs durability:**

| Конфигурация | Durability | Availability |
|---|---|---|
| `acks=1` | Слабая (только лидер) | Высокая |
| `acks=all`, нет `min.insync.replicas` (=1) | Слабая при сужении ISR | Высокая |
| `acks=all` + `min.insync.replicas=2` | Сильная | Снижается при отказе реплик |
| `acks=all` + `min.insync.replicas=replication.factor` | Максимальная | Минимальная |

**unclean.leader.election.enable**

Связанный параметр: если все ISR-реплики недоступны, Kafka по умолчанию (`unclean.leader.election.enable=false`) будет ждать возврата хотя бы одной из них, а не назначать лидером отставшую реплику. Включение `true` повышает доступность ценой возможной потери подтверждённых данных.

**Итог:** `acks=all` без `min.insync.replicas >= 2` не даёт реальных гарантий durability при сужении ISR. Для production-систем необходима связка `acks=all` + `min.insync.replicas=2` + `replication.factor=3`.
### Вопрос 22. Объясни механизм leader election в Kafka при падении брокера-лидера. Кто координирует выборы, откуда берётся список кандидатов и что такое unclean leader election?

**Категория:** Брокеры сообщений · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Understanding Kafka's Controller and Broker Election Process | Reintech media](https://reintech.io/blog/understanding-kafka-controller-broker-election)

Контроллер (Controller) — это обычный брокер Kafka, взявший на себя дополнительные обязанности по управлению кластером. В каждый момент времени ровно один брокер является контроллером; именно он обрабатывает выборы лидеров разделов, распространяет метаданные и выполняет административные операции.

**Выборы контроллера.** В традиционном режиме (ZooKeeper) брокеры соревнуются за создание временного (ephemeral) узла `/controller`. Первый, кому это удалось, становится контроллером. При его падении ZooKeeper автоматически удаляет узел, запуская новые выборы. В режиме KRaft (Kafka 3.0+) используется алгоритм Raft: большинство контроллер-eligible узлов должны согласовать нового лидера; это ускоряет восстановление и исключает зависимость от ZooKeeper.

**In-Sync Replicas (ISR).** ISR — это набор реплик, полностью синхронизированных с лидером. Реплика остаётся в ISR, пока запрашивает сообщения в пределах допустимой задержки (`replica.lag.time.max.ms`). При падении лидера контроллер выбирает нового лидера только из ISR, что гарантирует сохранность данных.

**Алгоритм выбора лидера раздела:**
1. Если ISR не пуст — выбирается первая доступная реплика из ISR (как правило, первая по списку).
2. Если ISR пуст и `unclean.leader.election.enable=true` — может быть выбрана любая доступная реплика, даже отставшая.
3. Если ISR пуст и `unclean.leader.election.enable=false` (по умолчанию) — раздел остаётся недоступным до восстановления ISR-реплики.

**Unclean leader election** (грязные выборы) — ситуация, когда лидером назначается реплика, не входящая в ISR. Такая реплика могла не получить часть сообщений, подтверждённых лидером, поэтому эти сообщения будут безвозвратно потеряны. Настройка `unclean.leader.election.enable=true` повышает доступность ценой возможной потери данных; по умолчанию отключена (приоритет консистентности).

#### Ответ от Claude

**Роль контроллера.** Контроллер — это один из обычных брокеров кластера, которому дополнительно поручены задачи: следить за состоянием брокеров, управлять списками ISR и инициировать выборы лидеров разделов. В кластере всегда ровно один активный контроллер.

**Как контроллер избирается.**
- *ZooKeeper-режим* — брокеры при старте пытаются создать ephemeral-узел `/controller` в ZooKeeper. Кто первый создал — тот контроллер. При падении узла ZooKeeper автоматически удаляет эфемерный узел, и оставшиеся брокеры начинают новую гонку.
- *KRaft-режим* (Kafka 2.8+, production с 3.3+) — ZooKeeper полностью исключён. Специальные брокеры (controller-eligible) реализуют алгоритм Raft: квorum из большинства выбирает активного контроллера. Это даёт более быстрый failover и упрощает операционирование.

**ISR (In-Sync Replicas) — источник кандидатов.** Для каждого раздела Kafka ведёт список ISR — реплик, не отстающих от лидера более чем на `replica.lag.time.max.ms` (по умолчанию 30 секунд). При записи сообщения с `acks=all` лидер ждёт подтверждения от всех реплик в ISR. Список ISR хранится в ZooKeeper (или в KRaft-журнале) и поддерживается в актуальном состоянии контроллером.

**Что происходит при падении лидера.**
1. Контроллер обнаруживает отсутствие брокера (через сессионный таймаут ZooKeeper или heartbeat в KRaft).
2. Контроллер обходит список ISR для каждого раздела, где упавший брокер был лидером.
3. Из доступных членов ISR выбирается первый в списке — он становится новым лидером.
4. Контроллер рассылает `LeaderAndIsr` запросы всем репликам раздела и обновляет метаданные кластера.
5. Продюсеры и потребители получают новые метаданные и переключаются на нового лидера.

**Unclean leader election** — назначение лидером реплики вне ISR. Это случается, когда все ISR-реплики недоступны одновременно. Поскольку такая реплика отставала от лидера, часть подтверждённых сообщений в ней отсутствует. После её назначения лидером эти сообщения теряются необратимо (нарушается гарантия "committed = durably written").

- `unclean.leader.election.enable=false` (дефолт) — раздел уходит в offline до восстановления хотя бы одной ISR-реплики. Данные не теряются, но доступность снижается.
- `unclean.leader.election.enable=true` — раздел остаётся доступным, но возможна потеря данных. Оправдано только там, где доступность важнее консистентности (например, логи метрик).

**Замечание по KRaft.** В KRaft также существует задокументированная проблема (KAFKA-19148): при одновременном reassignment раздела и смене лидера возможна «grязная» выборы даже при `unclean.leader.election.enable=false`, что является известным багом, а не штатным поведением.

### Вопрос 23. Как Kafka достигает exactly-once семантики? Что такое idempotent producer (PID, sequence number), как работают транзакции (transactional.id) и что означает isolation.level=read_committed на стороне потребителя?

**Категория:** Брокеры сообщений · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Exactly-once semantics with Kafka transactions | Strimzi Blog](https://strimzi.io/blog/2023/05/03/kafka-transactions/)

Kafka достигает exactly-once семантики через три взаимосвязанных механизма.

**Idempotent producer и PID/Epoch.** Когда продюсер вызывает `initTransactions()` (или при `enable.idempotence=true`), брокер выделяет ему уникальный Producer ID (PID). При повторном подключении с тем же `transactional.id` продюсер получает тот же PID, но с увеличенным epoch. Комбинация PID + epoch идентифицирует конкретную логическую сессию продюсера.

Каждое сообщение несёт sequence number (отдельный для каждой пары PID + TopicPartition). Брокер отвергает сообщение, если его sequence number не равен ровно `last_committed + 1`. Таким образом дубли при ретраях отфильтровываются, а пропуски обнаруживаются.

**Transactions и Transaction Coordinator.** Один из брокеров является Transaction Coordinator для данного продюсера (определяется хешом от `transactional.id` по партициям топика `__transaction_state`). Координатор управляет жизненным циклом транзакции через двухфазный коммит:
1. Ongoing — данные пишутся в целевые партиции.
2. PrepareCommit/PrepareAbort — решение фиксируется в `__transaction_state`.
3. CompleteCommit/CompleteAbort — control records (маркеры) записываются в пользовательские партиции.

При рестарте продюсер с тем же `transactional.id` получает увеличенный epoch, старый экземпляр (zombie producer) получает `ProducerFencedException` при следующей попытке записи.

**isolation.level=read_committed.** Брокер вводит понятие Last Stable Offset (LSO) — наибольший offset, до которого нет незакрытых транзакций. Потребитель с `read_committed` не может продвинуться за LSO: открытые транзакции блокируют чтение. Записи из отменённых транзакций фильтруются и никогда не достигают клиентского приложения.

#### Ответ от Claude

Exactly-once semantics (EOS) в Kafka — гарантия того, что каждое сообщение будет обработано ровно один раз, без потерь и без дублей, даже при сбоях сети или брокера. EOS строится на трёх уровнях.

**1. Idempotent Producer (защита от дублей внутри одной сессии).**

Включается через `enable.idempotence=true` (дефолт в Kafka 3.0+, требует `acks=all`, `retries > 0`, `max.in.flight.requests.per.connection <= 5`).

При инициализации брокер выдаёт продюсеру **Producer ID (PID)** — уникальный числовой идентификатор. Для каждой пары (PID, TopicPartition) брокер ведёт счётчик **sequence number**, начиная с 0. Продюсер инкрементирует счётчик с каждым новым batch. Брокер принимает batch, только если `incoming_seq = last_committed_seq + 1`:
- `incoming_seq <= last_committed_seq` — дубль, брокер отвечает успехом без повторной записи.
- `incoming_seq > last_committed_seq + 1` — пропуск, брокер возвращает ошибку `OutOfOrderSequenceException`.

Sequence numbers реплицируются в ISR, поэтому новый лидер после failover сохраняет возможность дедупликации.

**Ограничение:** idempotency гарантируется только в рамках одной PID-сессии и одного раздела. При рестарте продюсера PID меняется, и защита от дублей между сессиями не действует без `transactional.id`.

**2. Транзакции (атомарная запись через несколько разделов).**

Транзакции решают задачу: либо все сообщения batch записаны атомарно, либо ни одно. Применяются в паттерне consume-transform-produce.

Настройка: уникальный `transactional.id` на каждый логический поток (producer instance). `transactional.id` должен быть стабильным между рестартами.

Жизненный цикл транзакции:
```
producer.initTransactions();          // регистрация у Transaction Coordinator
producer.beginTransaction();
producer.send(...);                   // запись в целевые топики
producer.sendOffsetsToTransaction(offsets, groupMetadata); // атомарный коммит offset
producer.commitTransaction();         // или producer.abortTransaction()
```

**Transaction Coordinator** — компонент одного из брокеров. Выбирается детерминированно: `hash(transactional.id) % partitions_of___transaction_state`. Хранит состояние транзакций в топике `__transaction_state`.

**Двухфазный коммит:**
1. *Ongoing* — данные пишутся в целевые партиции (ещё не видны `read_committed`-потребителям).
2. *PrepareCommit/PrepareAbort* — координатор записывает решение в `__transaction_state` (с этого момента исход транзакции определён необратимо).
3. *CompleteCommit/CompleteAbort* — координатор пишет control records (маркеры COMMIT/ABORT) в каждую затронутую партицию.

**Fencing zombie producers.** При повторном `initTransactions()` с тем же `transactional.id` epoch увеличивается. Любой старый экземпляр, пытающийся писать со старым epoch, получает `ProducerFencedException`.

**3. isolation.level=read_committed на стороне потребителя.**

Без этой настройки транзакции не дают EOS на стороне потребителя.

**Last Stable Offset (LSO)** — максимальный offset, за которым нет открытых транзакций. Брокер не позволяет `read_committed`-потребителю продвинуться за LSO: это блокирует чтение «в-полёте» транзакций.

- `isolation.level=read_committed` — потребитель видит только сообщения из завершённых (COMMIT) транзакций и нетранзакционные сообщения; данные из отменённых (ABORT) транзакций фильтруются брокером и никогда не поступают в приложение.
- `isolation.level=read_uncommitted` (дефолт) — потребитель видит все записи в порядке offset, включая незакрытые и отменённые транзакции.

**Важно:** EOS гарантирует exactly-once только внутри Kafka (producer → broker → consumer offset commit). Если потребитель записывает результат во внешнюю систему без внешней транзакционной координации, EOS на этом уровне не обеспечивается.

### Вопрос 24. Чем отличается eager rebalancing от cooperative (incremental) rebalancing в Kafka? Почему классический протокол вызывает «stop-the-world» и как CooperativeStickyAssignor решает эту проблему?

**Категория:** Брокеры сообщений · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Incremental Cooperative Rebalancing in Apache Kafka | Confluent Blog](https://www.confluent.io/blog/incremental-cooperative-rebalancing-in-kafka/)

**Eager Rebalancing (классический протокол)** работает по принципу «остановить всё»: при любом изменении в группе (новый потребитель, выход потребителя, истечение heartbeat) все потребители отзывают все свои партиции и входят в состояние ожидания. Группа-лидер получает полный список подписок и строит новое распределение с нуля. Только после его утверждения потребители получают новые назначения и возобновляют обработку. В это время пропускная способность группы равна нулю. В тестах для кластера с 900 задачами стабилизация после перебалансировки занимала около 14 минут.

Проблемы eager-протокола:
- Партиции, которые не нужно перемещать, всё равно на время отзываются.
- При добавлении одного потребителя прерываются все остальные.
- В многоарендных окружениях ребалансировка одного коннектора останавливает несвязанные.

**Cooperative Incremental Rebalancing** (KIP-429, Kafka 2.4+) исходит из принципа: «ресурсы, которые не нужно перемещать, не должны прекращать использоваться». Вместо одного массового перераспределения используются последовательные раунды:

*Первый раунд* — лидер группы определяет, какие партиции должны сменить владельца. Партиции, остающиеся у текущих владельцев, назначаются сразу. Партиции, меняющие владельца, не назначаются никому в этом раунде — их текущие владельцы их отзывают.

*Второй раунд* — освободившиеся партиции назначаются новым владельцам.

Потребители, чьи партиции не затронуты, продолжают обработку на протяжении всего процесса. В тестах производительность возросла на 83–833% в разных сценариях.

**CooperativeStickyAssignor** реализует оба принципа одновременно: sticky (стремится сохранить текущие назначения) + cooperative (инкрементальные изменения). Каждый потребитель передаёт в subscription поле `ownedPartitions` — список партиций, которыми он владеет сейчас. Assignor строит новое распределение, по возможности сохраняя существующее, и только для партиций, меняющих владельца, запускает двухраундовый процесс перераспределения.

#### Ответ от Claude

**Почему ребалансировка вообще нужна.** Когда потребитель присоединяется к группе, выходит из неё (или истекает его `session.timeout.ms`), подписка меняется — Kafka должна перераспределить партиции топиков между активными потребителями. Этим управляет Group Coordinator (один из брокеров).

**Eager Rebalancing: механизм и проблема.**

При любом триггере ребалансировки протокол требует от каждого потребителя:
1. Отозвать (revoke) абсолютно все назначенные партиции и вызвать `onPartitionsRevoked()`.
2. Отправить `JoinGroup` запрос координатору.
3. Ожидать, пока координатор не получит `JoinGroup` от всех членов группы.
4. Лидер группы строит новое полное распределение.
5. Все потребители получают `SyncGroup` с новыми назначениями и вызывают `onPartitionsAssigned()`.

Этапы 1–5 — это «stop-the-world»: ни один потребитель не обрабатывает сообщения. Даже партиции, которые останутся у того же потребителя, временно отзываются. При большом числе партиций или длительных `onPartitionsRevoked`-обработчиках пауза может достигать десятков секунд.

Используемые assignors: `RangeAssignor` (дефолт до Kafka 2.4), `RoundRobinAssignor` — оба работают только с eager-протоколом.

**Cooperative (Incremental) Rebalancing: механизм.**

Введён в KIP-429 (Kafka 2.4), стал дефолтом в Kafka 3.1 с `ConsumerConfig.PARTITION_ASSIGNMENT_STRATEGY_CONFIG = CooperativeStickyAssignor`.

Ключевая идея: потребители сообщают, какими партициями они уже владеют (`ownedPartitions` в `JoinGroup`). Assignor в первом раунде назначает только «стабильные» партиции (не меняющие владельца). Партиции, которые нужно переместить, в первом раунде не назначаются никому.

**Двухраундовый процесс:**

*Раунд 1:*
- Потребитель A и B подали `JoinGroup`. Assignor видит, что 2 из 10 партиций нужно переместить с A на B.
- A продолжает держать 8 «стабильных» партиций и продолжает обработку.
- A отзывает 2 перемещаемые партиции, вызывается `onPartitionsRevoked()` только для них.
- Завершение раунда 1 вызывает второй `JoinGroup`.

*Раунд 2:*
- Теперь 2 партиции свободны. Assignor назначает их B.
- B вызывает `onPartitionsAssigned()` для 2 новых партиций.
- Обработка на A не прерывалась (только 2 партиции были на паузе в промежутке между раундами).

**Новый callback `onPartitionsLost()`** вызывается при вынужденном отзыве (потребитель был исключён из группы без штатного завершения), чтобы отличить нормальный revoke от аварийного.

**CooperativeStickyAssignor.**

Наследует `StickyAssignor` (максимальная «липкость» — партиции как можно меньше перемещаются) и добавляет cooperative-поведение. Алгоритм:
1. Получает `ownedPartitions` от всех потребителей.
2. Строит оптимальное распределение, минимизируя перемещения.
3. Для перемещаемых партиций запускает двухраундовый процесс.
4. Остальные партиции остаются у своих владельцев без прерывания.

**Итоговое сравнение:**

| Характеристика | Eager | Cooperative (Incremental) |
|---|---|---|
| Протокол KIP | — | KIP-429 |
| Партиции отзываются | Все, у всех | Только перемещаемые |
| Stop-the-world | Да, полный | Нет (или минимальный во 2-м раунде) |
| Кол-во раундов | 1 | 2 (при перемещениях) |
| Assignors | Range, RoundRobin | CooperativeStickyAssignor |
| Default с версии | — | Kafka 3.1+ |
| Callback при потере | onPartitionsRevoked | onPartitionsLost (новый) |

**Миграция.** При переходе с eager на cooperative требуется rolling restart: в промежуточном состоянии часть потребителей использует старый протокол, часть — новый. Kafka корректно обрабатывает это смешение в течение одного цикла ребалансировки.
### Вопрос 25. Consumer в Kafka вызывает poll() слишком редко и группа постоянно ребалансируется. Какие именно параметры (max.poll.interval.ms, session.timeout.ms, heartbeat.interval.ms) за что отвечают и как их правильно соотнести между собой?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Consumer Configs | Apache Kafka](https://kafka.apache.org/41/configuration/consumer-configs/)

Официальная документация Kafka 4.1 даёт следующие определения:

- **session.timeout.ms** (по умолчанию 45 000 мс) — таймаут, используемый для обнаружения сбоев клиента при работе с групповым управлением. Клиент периодически отправляет брокеру heartbeat-сигналы. Если брокер не получит heartbeat до истечения этого таймаута, он удаляет консьюмера из группы и инициирует ребалансировку. Значение должно находиться в пределах, заданных на брокере: `group.min.session.timeout.ms` и `group.max.session.timeout.ms`. Не применяется, если `group.protocol` = "consumer" — в этом случае управление переходит к конфигурации брокера `group.consumer.session.timeout.ms`.

- **heartbeat.interval.ms** (по умолчанию 3 000 мс) — ожидаемый интервал между heartbeat-сигналами к координатору группы. Heartbeat-сигналы поддерживают активность сессии и позволяют брокеру оповещать консьюмер о начале ребалансировки. Применимо только при `group.protocol` = "classic". Рекомендуется устанавливать не более одной трети от `session.timeout.ms`.

- **max.poll.interval.ms** (по умолчанию 300 000 мс, т.е. 5 минут) — максимально допустимая задержка между последовательными вызовами `poll()` при использовании группового управления. Если интервал превышен, консьюмер считается упавшим и партиции перераспределяются. Для статических консьюмеров (с `group.instance.id`) партиции не переназначаются немедленно — вместо этого консьюмер прекращает отправку heartbeat-сигналов, и переназначение произойдёт после истечения `session.timeout.ms`.

#### Ответ от Claude

Три параметра решают разные проблемы, образуя иерархию таймаутов.

**Что за что отвечает:**

| Параметр | Дефолт | Кто контролирует | Что произойдёт при нарушении |
|---|---|---|---|
| `heartbeat.interval.ms` | 3 000 мс | Клиент → Брокер | Слишком большой интервал → брокер быстрее посчитает консьюмер мёртвым |
| `session.timeout.ms` | 45 000 мс | Брокер | Отсутствие heartbeat дольше таймаута → брокер исключает консьюмер из группы и запускает ребалансировку |
| `max.poll.interval.ms` | 300 000 мс | Клиент (self-report) | Слишком долгая обработка между вызовами `poll()` → консьюмер сам покидает группу (livelock protection) |

**Ключевое различие между `session.timeout.ms` и `max.poll.interval.ms`:**

- `session.timeout.ms` защищает от **сбоя процесса**: если приложение упало или сеть прервалась, брокер не получит heartbeat и исключит консьюмер.
- `max.poll.interval.ms` защищает от **livelock**: консьюмер жив, heartbeat отправляет, но `poll()` не вызывает — т.е. завис на обработке одного батча. В этой ситуации консьюмер сам проактивно выходит из группы, не дожидаясь `session.timeout.ms`.

Начиная с Kafka 0.10.1 heartbeat-поток вынесен в фоновый (отдельный от основного потока обработки) поток. Именно поэтому появился `max.poll.interval.ms`: приложение может продолжать слать heartbeat, но при этом никогда не вызывать `poll()` — без этого параметра такой консьюмер не был бы обнаружен.

**Правильные соотношения:**

```
heartbeat.interval.ms  ≈  session.timeout.ms / 3
session.timeout.ms     <  max.poll.interval.ms
```

Пример для приложения с долгой обработкой (до 2 минут на батч):

```properties
max.poll.interval.ms=180000      # 3 минуты — с запасом
session.timeout.ms=45000         # 45 секунд (дефолт)
heartbeat.interval.ms=15000      # 15 секунд = 45000 / 3
max.poll.records=50              # уменьшить батч, чтобы укладываться в таймаут
```

**Причина постоянных ребалансировок при редком `poll()`:**

Если `poll()` вызывается реже, чем раз в `max.poll.interval.ms`, консьюмер самостоятельно покидает группу с кодом `REBALANCE_IN_PROGRESS`. Брокер запускает ребалансировку, партиция перераспределяется другому консьюмеру, затем «медленный» консьюмер снова присоединяется — и цикл повторяется.

**Решение:** либо уменьшить `max.poll.records` (меньше работы за один батч), либо увеличить `max.poll.interval.ms`, либо вынести тяжёлую обработку в асинхронный поток (но тогда нужно управлять смещениями вручную).

### Вопрос 26. Сколько consumer'ов в одной consumer group реально могут читать топик с 6 партициями? Что произойдёт, если запустить 7 consumer'ов? А если создать вторую consumer group на тот же топик?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Kafka Consumer Design: Consumers, Consumer Groups, and Offsets | Confluent Documentation](https://docs.confluent.io/kafka/design/consumer-design.html)

Каждая партиция в каждый момент времени потребляется ровно одним консьюмером внутри группы. Это означает: при фиксированном количестве партиций одновременно активно читать могут не более стольких консьюмеров, сколько партиций. Consumer group — это набор консьюмеров одного приложения, работающих совместно для потребления и обработки сообщений. Координатор группы равномерно распределяет партиции подписанных топиков между участниками группы.

#### Ответ от Claude

**Сколько консьюмеров реально читают топик с 6 партициями:**

Максимальное количество **активных** консьюмеров в одной группе равно количеству партиций. Для топика с 6 партициями — **не более 6 консьюмеров** будут реально читать данные.

Фундаментальное правило Kafka: одна партиция в рамках одной consumer group назначается строго одному консьюмеру. Один консьюмер при этом может получить несколько партиций.

**Что произойдёт при 7 консьюмерах:**

7-й консьюмер окажется **полностью простаивающим (idle)**. Ему не будет назначена ни одна партиция — все 6 уже заняты другими участниками группы. Он останется членом группы, будет отправлять heartbeat, участвовать в ребалансировках, но не получит никаких сообщений.

Это — допустимое состояние, используемое для горячего резерва: если один из 6 активных консьюмеров упадёт, 7-й немедленно получит его партиции после ребалансировки (обычно занимает секунды).

**Что произойдёт при второй consumer group:**

Каждая consumer group — независимый подписчик. Она имеет **собственный указатель смещения (offset)** для каждой партиции, хранящийся в топике `__consumer_offsets`. Вторая группа начнёт читать топик с самого начала (если `auto.offset.reset=earliest`) или с текущего конца (если `latest`), совершенно независимо от первой.

Таким образом, второй consumer group с 6 и более консьюмерами будут **полностью и независимо** читать все те же 6 партиций, не мешая первой группе. Это основа паттерна Pub/Sub в Kafka: один топик → множество независимых потребителей.

**Итоговая схема:**

```
Топик: 6 партиций (P0..P5)

Consumer Group A (6 консьюмеров):
  C1 → P0, C2 → P1, C3 → P2, C4 → P3, C5 → P4, C6 → P5

Consumer Group A (7 консьюмеров):
  C1 → P0, C2 → P1, C3 → P2, C4 → P3, C5 → P4, C6 → P5
  C7 → (idle, партиций нет)

Consumer Group B (независимая):
  CB1 → P0, CB2 → P1, CB3 → P2, CB4 → P3, CB5 → P4, CB6 → P5
  (собственные offsets, не связанные с Group A)
```

### Вопрос 27. Kafka гарантирует порядок сообщений внутри партиции, но не между партициями. Как при этом обеспечить глобальный порядок для связанных событий (например, всех событий одного пользователя)?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Apache Kafka Partition Key: A Comprehensive Guide | Confluent](https://www.confluent.io/learn/kafka-partition-key/)

Kafka использует хеш-функцию от ключа партиции для маппинга ключа на конкретную партицию. Поскольку порядок гарантирован внутри партиции, использование постоянного ключа гарантирует последовательную обработку связанных событий. Для пользовательских сценариев рекомендуется использовать идентификатор пользователя (customer ID / user ID) в качестве ключа: все взаимодействия конкретного пользователя направляются в одну партицию, что критично в сценариях отслеживания заказов или финансовых транзакций. Глобальный порядок (через все события системы) требует единственной партиции, но это жертвует параллелизмом и ограничивает пропускную способность.

#### Ответ от Claude

**Суть проблемы:**

Kafka гарантирует строгий порядок только **внутри одной партиции**. При нескольких партициях сообщения из разных партиций могут приходить к консьюмеру в произвольном порядке относительно друг друга.

**Стратегии обеспечения порядка:**

**1. Ключ сообщения как маршрутизатор (основной способ)**

Установить в качестве ключа (`ProducerRecord.key`) идентификатор сущности, события которой должны быть упорядочены — например, `userId`. Kafka применяет детерминированный хеш к ключу:

```
partition = hash(key) % numPartitions
```

Все события пользователя `userId=42` всегда попадут в одну и ту же партицию. Консьюмер, закреплённый за этой партицией, обработает их строго в порядке записи.

Это **per-entity ordering** — порядок гарантирован для каждого пользователя независимо, что является правильным решением для большинства бизнес-задач (обработка заказов, финансовые операции, история действий пользователя).

**2. Единственная партиция (глобальный порядок)**

Создать топик с одной партицией обеспечивает абсолютный глобальный порядок всех событий. Это применимо только при очень низком объёме — одна партиция означает одного записывающего продюсера без параллелизма и одного читающего консьюмера на группу. На практике это антипаттерн для высоконагруженных систем.

**3. Идемпотентный продюсер + acks=all**

Для предотвращения дублирования и нарушения порядка при ретраях продюсера используется комбинация:

```properties
enable.idempotence=true
acks=all
max.in.flight.requests.per.connection=5  # до 5 при идемпотентности
```

Идемпотентность гарантирует, что повторная отправка не создаст дубликат и не нарушит порядок внутри партиции (каждое сообщение получает sequence number).

**Важный нюанс при смене числа партиций:**

Хеш-маппинг `hash(key) % numPartitions` зависит от числа партиций. При увеличении числа партиций уже записанные события пользователя остаются в старой партиции, а новые попадут в другую — порядок между старыми и новыми событиями в этот момент нарушится. Увеличение числа партиций требует либо ребалансировки данных, либо работы с временным окном несогласованности.

**Итоговые рекомендации:**

| Потребность | Решение |
|---|---|
| Порядок событий одного пользователя | Ключ = userId |
| Порядок событий одного заказа | Ключ = orderId |
| Абсолютный глобальный порядок | 1 партиция (только для low-throughput) |
| Защита от дублирования при ретраях | enable.idempotence=true + acks=all |
### Вопрос 28. Что такое consumer lag? Как его измерить? Назови хотя бы два инструмента или метода мониторинга и объясни, при каких условиях lag будет расти даже при живом consumer'е.

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Kafka Consumer Lag: How to Monitor, Check & Reduce — Sematext](https://sematext.com/blog/kafka-consumer-lag-offsets-monitoring/)

Consumer lag — это разница между последним оффсетом, сохранённым брокером (Log End Offset), и последним зафиксированным оффсетом consumer'а для данной партиции. Формула: **Consumer Lag = Log End Offset − Consumer Offset**.

Для измерения используется встроенный CLI-инструмент Kafka:

```bash
bin/kafka-consumer-groups.sh --bootstrap-server localhost:9092 --describe --group [consumer-group-name]
```

Вывод содержит колонку LAG — дельту оффсетов для каждой партиции.

Два инструмента мониторинга:
1. **kafka-consumer-groups.sh** — встроенный скрипт, показывает CURRENT-OFFSET, LOG-END-OFFSET и LAG для каждой партиции в группе.
2. **LinkedIn Burrow** — специализированный инструмент мониторинга consumer lag; послужил основой для многих коммерческих решений.

Причины роста lag при живом consumer'е: всплески трафика, превышающие скорость обработки; неэффективность кода (медленные запросы к БД, сложная бизнес-логика); узкие места в downstream-системах; неравномерная нагрузка из-за skew по ключам партиций; программные баги, снижающие скорость.

#### Ответ от Claude

**Определение.** Consumer lag — количество сообщений, на которые consumer отстаёт от producer'а в рамках одной партиции. Формально: `lag = logEndOffset − consumerCommittedOffset`. Это главный индикатор здоровья pipeline: lag ~ 0 означает, что consumer успевает; растущий lag — сигнал проблемы.

**Как измерить.**

1. **kafka-consumer-groups.sh** (встроенный CLI). Команда:
   ```bash
   bin/kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
     --describe --group my-group
   ```
   Выводит таблицу с колонками TOPIC, PARTITION, CURRENT-OFFSET, LOG-END-OFFSET, LAG, CONSUMER-ID, HOST. LAG = LOG-END-OFFSET − CURRENT-OFFSET для каждой партиции.

2. **JMX-метрики** (программный / мониторинговый подход). Kafka-consumer экспортирует через JMX:
   - `records-lag-max` — максимальный lag среди всех партиций клиента (MBean: `kafka.consumer:type=consumer-fetch-manager-metrics,client-id={id}`)
   - `records-lag` и `records-lag-avg` — lag конкретной партиции (MBean: `kafka.consumer:type=consumer-fetch-manager-metrics,partition={p},topic={t},client-id={id}`)
   Доступны через jconsole, JMX-экспортер Prometheus, Micrometer/Spring Boot Actuator.

**Два инструмента/метода мониторинга:**

1. **kafka-consumer-groups.sh** — минимальный инструмент без зависимостей; достаточен для ручной диагностики. Работает с любым брокером Kafka, не требует ничего кроме доступа к bootstrap-серверу.
2. **JMX + Prometheus JMX Exporter** (или Micrometer) — метрика `kafka.consumer.fetch.manager.records.lag` поступает в Prometheus/Grafana/Datadog и позволяет строить алерты по порогу lag и по скорости его роста. LinkedIn Burrow — ещё один популярный специализированный инструмент, который эвристически оценивает, движется ли consumer вперёд или стоит.

**Когда lag растёт при живом consumer'е.**

Consumer работает (heartbeat отправляет, оффсеты двигаются), но lag растёт, если producer записывает быстрее, чем consumer обрабатывает:

| Причина | Механизм |
|---|---|
| **Медленная обработка** | Например, consumer тратит 50 мс на сообщение, а приходит 1 000 сообщений/с в партицию — одного consumer'а физически не хватит |
| **Внешние bottleneck** | Запрос к медленной БД / HTTP-вызов внутри poll-цикла увеличивает время на сообщение |
| **Rebalance** | Во время rebalance consumer не обрабатывает сообщения; lag накапливается; при частых rebalance'ах (GC-паузы → пропуск heartbeat, добавление/удаление consumer'ов) lag систематически растёт |
| **Hot partition (key skew)** | Одна партиция получает непропорционально много сообщений; один consumer за ней не успевает, остальные простаивают |
| **GC-паузы** | Длинный Stop-The-World > `max.poll.interval.ms` триггерит rebalance; даже без rebalance — паузы = нулевая обработка |
| **Сетевые проблемы** | Высокая latency fetch-запросов к брокеру снижает пропускную способность даже при быстрой бизнес-логике |

Ключевой вывод: lag растёт всегда, когда **скорость продюсера > скорость обработки consumer'а**, независимо от того, жив consumer или нет. Решения: горизонтальное масштабирование (больше партиций + consumer'ов), оптимизация бизнес-логики, async-обработка, батчинг.

### Вопрос 29. Производитель отправил сообщение в Kafka с acks=1, брокер подтвердил, но сразу после этого лидер упал до репликации на фолловеры. Сообщение потеряно? Как настроить продюсера, чтобы этого не произошло, и какова цена этой надёжности?

**Категория:** Брокеры сообщений · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Producer Configs — Apache Kafka 4.2 Documentation](https://kafka.apache.org/42/configuration/producer-configs/)

При `acks=1` лидер записывает сообщение в свой локальный лог и отвечает producer'у, не дожидаясь подтверждения от фолловеров. Если лидер упадёт сразу после ответа, но до того как фолловеры успеют реплицировать запись, сообщение будет потеряно.

Для предотвращения потерь используется `acks=all` (эквивалентно `acks=-1`): лидер ждёт подтверждения от всего набора in-sync replicas (ISR) перед ответом producer'у. Это гарантирует, что запись не пропадёт, пока жива хотя бы одна реплика из ISR.

Типовая надёжная конфигурация: replication factor = 3, `min.insync.replicas = 2`, `acks=all`. При этом допустимо падение одной реплики без потери данных.

Включение идемпотентности (`enable.idempotence=true`) требует `acks=all`, `retries > 0`, `max.in.flight.requests.per.connection <= 5` и предотвращает дублирование при ретраях.

#### Ответ от Claude

**Да, сообщение потеряно** — это штатное поведение `acks=1`.

**Механизм потери:**
1. Producer отправляет запись.
2. Лидер пишет в свой локальный лог.
3. Лидер отвечает producer'у `ACK`.
4. **Лидер падает** — фолловеры ещё не получили запись.
5. Kafka выбирает нового лидера из ISR — но в ISR этой записи нет.
6. Запись физически исчезает; consumer никогда её не увидит.

Producer считает отправку успешной — это silent data loss.

**Как защититься: конфигурация producer'а.**

```properties
# Продюсер ждёт подтверждения от всех ISR
acks=all

# Повторы при временных ошибках (NOT_ENOUGH_REPLICAS и т.п.)
retries=2147483647          # "бесконечно" — Kafka сама ограничит по delivery.timeout.ms
delivery.timeout.ms=120000  # общий таймаут доставки

# Идемпотентность: дедупликация дублей при ретраях
enable.idempotence=true     # с Kafka 3.0 включена по умолчанию

# Обязательно при enable.idempotence=true
max.in.flight.requests.per.connection=5
```

**Конфигурация на стороне брокера/топика:**

```properties
# Топик/брокер: минимум 2 реплики должны подтвердить запись
min.insync.replicas=2

# Достаточный replication factor
replication.factor=3
```

При `acks=all` + `min.insync.replicas=2` + `replication.factor=3`:
- Запись считается успешной только если 2 из 3 реплик её зафиксировали.
- Если живых реплик в ISR < `min.insync.replicas`, брокер вернёт ошибку `NotEnoughReplicasException` — producer узнает о проблеме, а не потеряет данные молча.

**Цена надёжности:**

| Аспект | Последствие |
|---|---|
| **Latency** | Продюсер ждёт подтверждения от всех ISR; задержка растёт на время репликации (обычно единицы мс в LAN, но зависит от сети и нагрузки фолловеров) |
| **Throughput** | Снижается: producer не может отправлять следующий батч до получения ACK от ISR |
| **Доступность** | Если ISR сокращается до < `min.insync.replicas` (например, 1 из 3 реплик), партиция становится **недоступна для записи** до восстановления реплик |
| **Задержка при сбоях** | При падении фолловера лидер ждёт таймаута ISR shrink, прежде чем исключить его из ISR и ответить |

Согласно бенчмаркам, переход с `acks=1` на `acks=all` увеличивает latency на 30–50% для чувствительных к задержке систем. Для финансовых транзакций, аудит-логов и событий, критичных для бизнеса, эта цена оправдана.

**Итог:** `acks=all` + `min.insync.replicas=2` + `replication.factor=3` + `enable.idempotence=true` — production-стандарт для систем с требованием zero data loss.

### Вопрос 30. В чём разница между at-most-once, at-least-once и exactly-once гарантиями доставки? Приведи конкретный пример конфигурации Kafka (producer + consumer) для каждой из них.

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Design — Apache Kafka 3.0 Documentation](https://kafka.apache.org/30/design/design/)

Kafka поддерживает три гарантии доставки сообщений:

- **At-most-once** («не более одного раза»): сообщения могут быть потеряны, но никогда не обрабатываются повторно.
- **At-least-once** («не менее одного раза»): сообщения никогда не теряются, но могут быть обработаны повторно.
- **Exactly-once** («ровно один раз»): каждое сообщение обрабатывается ровно один раз.

Consumer сам управляет своей позицией в логе. Гарантия зависит от того, когда он фиксирует оффсет:
- **At-most-once**: оффсет фиксируется до обработки сообщения — при сбое сообщение не будет перечитано, возможна потеря.
- **At-least-once**: оффсет фиксируется после обработки — при сбое сообщение будет перечитано, возможно дублирование.
- **Exactly-once**: требует идемпотентного вывода или атомарной координации фиксации оффсета с записью результата.

Для Kafka-to-Kafka процессинга транзакционная поддержка (с версии 0.11.0.0) позволяет атомарно записывать обработанные данные и consumer-оффсеты вместе, обеспечивая exactly-once.

#### Ответ от Claude

**Концептуальное отличие трёх семантик:**

| Гарантия | Потери возможны? | Дубли возможны? | Типичный use-case |
|---|---|---|---|
| At-most-once | Да | Нет | Метрики, логи, где потеря некритична |
| At-least-once | Нет | Да | Большинство бизнес-событий с идемпотентной обработкой |
| Exactly-once | Нет | Нет | Финансовые транзакции, дедупликация критична |

---

**At-most-once**

Сообщение обрабатывается не более одного раза — но может не обработаться вовсе.

Producer:
```properties
acks=0                    # не ждём подтверждения от брокера
retries=0                 # не повторяем при ошибке
```

Consumer:
```properties
enable.auto.commit=true   # оффсет фиксируется автоматически до/вместе с получением
auto.commit.interval.ms=1000
```

Механизм: consumer получает сообщения, auto-commit фиксирует оффсет сразу. Если consumer падает в процессе обработки — оффсет уже зафиксирован, сообщение теряется.

---

**At-least-once**

Сообщение гарантированно не потеряется, но при сбое может быть обработано несколько раз.

Producer:
```properties
acks=all                  # ждём подтверждения от всех ISR
retries=2147483647        # повторяем при ошибке
enable.idempotence=false  # дедупликация на уровне producer'а отключена (простой вариант)
```

Consumer:
```properties
enable.auto.commit=false  # ручное управление оффсетом
```

```java
// Consumer-код: фиксируем оффсет только после успешной обработки
ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
for (ConsumerRecord<String, String> record : records) {
    process(record);          // сначала обрабатываем
}
consumer.commitSync();        // потом фиксируем оффсет
```

При сбое между `process()` и `commitSync()` сообщения будут перечитаны. Важно: бизнес-логика должна быть идемпотентной (повторная обработка того же сообщения не даёт побочных эффектов).

---

**Exactly-once**

Каждое сообщение обрабатывается ровно один раз — без потерь и без дублей.

Producer:
```properties
enable.idempotence=true           # обязательно; требует acks=all, retries>0, max.in.flight<=5
transactional.id=my-producer-1   # уникальный ID; включает транзакционный режим
acks=all
max.in.flight.requests.per.connection=5
```

Consumer:
```properties
isolation.level=read_committed    # читаем только транзакционно зафиксированные сообщения
enable.auto.commit=false
```

```java
// Транзакционный producer-код
producer.initTransactions();

try {
    producer.beginTransaction();
    producer.send(new ProducerRecord<>("output-topic", key, value));
    // Атомарно фиксируем оффсеты входного топика вместе с записью результата
    producer.sendOffsetsToTransaction(currentOffsets, consumerGroupMetadata);
    producer.commitTransaction();
} catch (ProducerFencedException e) {
    producer.close();
} catch (KafkaException e) {
    producer.abortTransaction();
}
```

**Как это работает:** идемпотентный producer получает от брокера `ProducerID` и порядковые номера — брокер дедуплицирует ретраи. Транзакции добавляют атомарность: либо все записи транзакции видны consumer'ам, либо ни одна (`read_committed` скрывает незакоммиченные и абортированные транзакции). Вместе они дают exactly-once для Kafka-to-Kafka процессинга.

**Ограничение:** exactly-once в Kafka гарантирует семантику внутри самой Kafka (и Kafka Streams). Для внешних систем (БД, HTTP) требуется дополнительная идемпотентность на стороне этих систем — Kafka не может управлять транзакциями во внешних хранилищах.

**Требования к брокеру:** версия Kafka >= 0.11.0.0 для идемпотентности; для транзакций — `replication.factor >= 3`, `min.insync.replicas = 2` рекомендованы для топиков, участвующих в транзакциях.

### Вопрос 31. Что такое идемпотентный потребитель и зачем он нужен, если уже есть exactly-once в Kafka? В каких случаях даже при exactly-once на уровне брокера нужна идемпотентность на уровне бизнес-логики?

**Категория:** Брокеры сообщений · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Idempotent Reader — Confluent Developer Patterns](https://developer.confluent.io/patterns/event-processing/idempotent-reader/)

Паттерн идемпотентного читателя (Idempotent Reader) решает проблему дублирующихся событий в потоках данных. Согласно документации, идемпотентный читатель должен учитывать две причины появления дублей: операционные сбои и ошибки в логике приложения. Даже при наличии exactly-once на уровне брокера дубли могут возникать из-за багов в приложении-источнике — оно может ошибочно опубликовать одно и то же событие несколько раз. С точки зрения платформы Kafka это будут разные события, и брокер не сможет определить их как дубли. Для защиты от операционных сбоев используется `isolation.level="read_committed"`, чтобы читать только зафиксированные транзакции. Для защиты от логических дублей применяют два подхода: отслеживание уникальных идентификаторов событий через внешнее хранилище или проектирование бизнес-логики таким образом, чтобы повторная обработка одного и того же события давала тот же результат — например, через upsert в базе данных по ключу event ID.

#### Ответ от Claude

**Что такое идемпотентный потребитель**

Идемпотентный потребитель (Idempotent Consumer) — это потребитель сообщений, обработка которого дает одинаковый результат вне зависимости от того, сколько раз одно и то же сообщение было доставлено и обработано. Иными словами, повторная обработка не порождает дублирующихся побочных эффектов: повторная запись в БД, повторная отправка email, повторное списание средств и т.д.

**Что гарантирует exactly-once в Kafka и где заканчивается его действие**

Exactly-once в Kafka реализован через два механизма:

1. **Идемпотентный продюсер** — брокер присваивает продюсеру уникальный Producer ID (PID) и отслеживает порядковые номера сообщений. При повторной отправке брокер обнаруживает дубль и не записывает сообщение повторно в лог.
2. **Транзакции** — продюсер атомарно публикует batch сообщений в несколько партиций и топиков. Потребитель с `isolation.level=read_committed` видит только полностью зафиксированные транзакции.

Важное ограничение, которое явно указывает документация Confluent: **exactly-once гарантируется только в рамках самого Kafka**. Стоит приложению выйти за пределы Kafka — записать в базу данных, вызвать внешний API, отправить HTTP-запрос — и гарантия перестаёт действовать. Kafka не может контролировать транзакции внешних систем.

**Конкретные сценарии, где нужна идемпотентность на уровне бизнес-логики**

1. **Сбой после обработки, но до коммита offset.** Consumer обработал сообщение и записал результат в БД, но упал до коммита offset. При перезапуске (или при ребалансировке) он получит то же сообщение снова и обработает его второй раз. Это классический сценарий дублирования, не связанный с гарантиями брокера.

2. **Внешние системы вне транзакционного контекста Kafka.** Kafka Streams гарантирует exactly-once только для операций read-process-write внутри самого Kafka. Если обработчик вызывает REST API платёжного шлюза, отправляет email или пишет в PostgreSQL вне транзакции Kafka, эти операции происходят вне зоны гарантий.

3. **Баги в upstream-приложении.** Приложение-источник может опубликовать одно и то же событие дважды (например, из-за ошибки retry-логики или двойного сохранения). С позиции Kafka это будут два разных сообщения с разными offset'ами — брокер не считает их дублями, и exactly-once здесь не поможет.

4. **Многоэтапные пайплайны с несколькими потребителями.** В сложных топологиях, где сообщение проходит через несколько сервисов, гарантии exactly-once распространяются только на участки, управляемые Kafka Streams. Каждый сервис с внешними side-эффектами обязан быть идемпотентным самостоятельно.

5. **Consumer group rebalance в момент обработки.** При ребалансировке обработанное, но не закоммиченное сообщение будет назначено другому потребителю.

**Практические реализации**

- **Upsert по event ID** — запись в БД с ON CONFLICT DO UPDATE или аналогом; повторная запись не создаёт дубль.
- **Таблица обработанных событий** — перед обработкой проверяется, не было ли уже обработано данное event ID; после — ID сохраняется. Проверка и сохранение выполняются в одной транзакции с самой бизнес-операцией.
- **Идемпотентные операции по природе** — SET вместо INCREMENT, UPDATE с абсолютным значением вместо относительного.
- **Redis/внешний key-value store** — хранение set обработанных ID с TTL для быстрой проверки дублей.

**Итог**

Exactly-once в Kafka — это гарантия инфраструктурного уровня: сообщение не потеряется и не будет записано в лог дважды. Идемпотентный потребитель — это гарантия прикладного уровня: бизнес-операция выполнится ровно один раз, даже если сообщение доставлено несколько раз. Оба механизма дополняют друг друга и решают разные классы проблем.

### Вопрос 32. Объясни разницу между типами exchange в RabbitMQ: direct, fanout, topic, headers. В каком сценарии ты выбрал бы headers exchange вместо topic exchange, несмотря на его меньшую производительность?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Part 4: RabbitMQ Exchanges, routing keys and bindings — CloudAMQP](https://www.cloudamqp.com/blog/part4-rabbitmq-for-beginners-exchanges-routing-keys-bindings.html)

**Direct Exchange** — маршрутизирует сообщения в очереди по точному совпадению routing key. Сообщение идёт в очередь (или очереди), у которой binding key в точности совпадает с routing key сообщения.

**Fanout Exchange** — полностью игнорирует routing key и отправляет копию каждого сообщения во все привязанные очереди. Идеален для широковещательной рассылки одного сообщения множеству потребителей.

**Topic Exchange** — маршрутизирует по шаблонному совпадению с разделителем-точкой (`.`). Символ `*` заменяет ровно одно слово, `#` — ноль и более слов. Позволяет подписчикам выражать интерес к категориям сообщений через паттерны.

**Headers Exchange** — маршрутизирует по атрибутам заголовков сообщения, а не по routing key. Включает параметр `x-match`: значение `all` требует совпадения всех указанных заголовков, `any` — хотя бы одного. Подходит для сообщений с несколькими неупорядоченными критериями маршрутизации.

#### Ответ от Claude

**Direct Exchange**

Маршрутизирует сообщения в очередь по точному совпадению routing key. Одна и та же очередь может иметь несколько binding key (реализация multicast через direct), а несколько очередей — один и тот же binding key. Используется для простых сценариев: задачи воркерам, команды конкретному сервису.

Пример: exchange `orders`, очередь `orders.new` привязана с ключом `"new"`. Сообщение с routing key `"new"` попадёт именно в эту очередь.

**Fanout Exchange**

Routing key игнорируется полностью. Каждое сообщение рассылается во все привязанные к exchange очереди. Аналог broadcast. Используется для уведомлений, аудит-лога, кэш-инвалидации — когда одно событие должно достичь всех подписчиков.

**Topic Exchange**

Наиболее гибкий из «routing key» типов. Ключ разбивается на слова через точку. Паттерны привязки:
- `*` — ровно одно слово (`logs.*.error` совпадёт с `logs.app.error`, но не с `logs.app.db.error`)
- `#` — ноль или более слов (`logs.#` совпадёт с `logs`, `logs.app`, `logs.app.db.error`)

Используется в event-driven архитектурах, где потребители подписываются на категории событий (например, `payments.#` для всех событий платежей).

**Headers Exchange**

Маршрутизация основана на заголовках (`headers`) сообщения — произвольных key-value атрибутах — а не на routing key. При привязке очереди указывается набор заголовков и параметр `x-match`:
- `x-match: all` — сообщение должно содержать **все** указанные заголовки с нужными значениями.
- `x-match: any` — достаточно совпадения **хотя бы одного** заголовка.

Routing key при этом игнорируется.

**Когда выбрать headers exchange вместо topic exchange**

Несмотря на меньшую производительность (сравнение заголовков требует разбора AMQP-таблицы для каждого сообщения, тогда как topic использует дерево паттернов), headers exchange предпочтителен в нескольких сценариях:

1. **Маршрутизация по нескольким независимым атрибутам нестрокового типа.** Topic exchange работает исключительно со строковыми routing key. Если критерии маршрутизации включают числа, булевы значения или составные условия — например, `priority=high AND region=eu AND format=pdf` — headers exchange выражает это нативно. В topic exchange пришлось бы кодировать все атрибуты в строку-ключ, что хрупко и трудно масштабируется.

2. **Комбинирование условий через `x-match: any`.** Topic exchange не имеет эквивалента логическому OR между независимыми атрибутами. Headers exchange с `x-match: any` позволяет очереди получать сообщения, если совпадает хотя бы один из нескольких атрибутов — например, уведомление нужно доставить, если `priority=critical` ИЛИ `escalate=true`.

3. **Маршрутизация без смысловой иерархии в ключе.** Topic exchange предполагает иерархическую структуру ключа (`a.b.c`). Если атрибуты сообщения не образуют иерархию и не имеют фиксированного порядка, моделирование через topic key искусственно и ведёт к комбинаторному взрыву паттернов.

4. **Совместимость с legacy или внешними системами**, которые уже кладут метаданные в заголовки AMQP-сообщений и не управляют routing key.

**Практический итог**

В подавляющем большинстве случаев topic exchange более производителен, читаем и достаточен. Headers exchange выбирают осознанно, когда критерии маршрутизации многомерны, неупорядочены, содержат нестроковые типы или требуют логики OR/AND по независимым атрибутам.

### Вопрос 33. Как работает Dead Letter Exchange (DLX) в RabbitMQ? При каких условиях сообщение попадает в DLX? Почему использование DLX в кластере без quorum queues не гарантирует безопасность, и как quorum queues это исправляют?

**Категория:** Брокеры сообщений · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Dead Letter Exchanges — официальная документация RabbitMQ](https://www.rabbitmq.com/docs/dlx)

Dead Letter Exchange (DLX) — это обычный RabbitMQ exchange любого типа, в который перенаправляются сообщения из очередей при наступлении определённых событий. Настраивается через политику или аргумент `x-dead-letter-exchange` при объявлении очереди. Сообщение становится "мёртвым" в четырёх случаях: потребитель отклонил его через `basic.reject` или `basic.nack` с `requeue=false`; истёк TTL сообщения; очередь превысила лимит длины; на quorum queue исчерпан лимит повторных доставок (`delivery-limit`). Если истекает сама очередь целиком — её сообщения НЕ dead-letter'ятся. Для маршрутизации используется либо специальный ключ `x-dead-letter-routing-key`, либо оригинальный routing key сообщения. Ключевое ограничение стандартного поведения: мёртвые сообщения перепубликуются **без подтверждений (publisher confirms)**, что означает возможность потери сообщений при недоступности целевой очереди.

#### Ответ от Claude

**Что такое Dead Letter Exchange**

DLX — это стандартный RabbitMQ exchange (может быть direct, fanout, topic или headers), который назначается источниковой очереди в качестве получателя "мёртвых" сообщений. Когда сообщение становится неживым, брокер перепубликует его в DLX, откуда оно маршрутизируется в одну или несколько dead letter queue (DLQ) — обычных очередей для дальнейшего анализа или повторной обработки.

Настройка производится двумя способами:
- Через политику (policy): `rabbitmqctl set_policy DLX ".*" '{"dead-letter-exchange":"my-dlx"}' --apply-to queues`
- Через аргумент при объявлении очереди: `x-dead-letter-exchange: "my-dlx"` (и опционально `x-dead-letter-routing-key`).

**Условия попадания сообщения в DLX**

Сообщение dead-letter'ится при наступлении одного из следующих событий:

1. **Отклонение потребителем** — вызов `basic.reject` или `basic.nack` с параметром `requeue=false`. Если `requeue=true`, сообщение возвращается в очередь и dead-letter'ом не считается.
2. **Истечение TTL сообщения** — либо TTL установлен на конкретное сообщение (`expiration` в свойствах), либо на всю очередь через `x-message-ttl`.
3. **Превышение длины очереди** — очередь достигла ограничения `x-max-length` или `x-max-length-bytes`, и поведение при переполнении (`overflow`) настроено на отбрасывание.
4. **Превышение лимита доставок на quorum queue** — специфично для quorum queues: если количество попыток доставки сообщения превысило `delivery-limit`.

Важный нюанс: если истекает сама очередь (`x-expires`), её сообщения НЕ переходят в DLX — они просто удаляются.

**Маршрутизация dead-lettered сообщений**

- Если у очереди задан `x-dead-letter-routing-key`, мёртвое сообщение публикуется в DLX с этим ключом.
- Если не задан — используются оригинальные routing key сообщения.
- К сообщению добавляется заголовок `x-death` с информацией о причине и источнике dead-lettering (для трассировки).

**Почему DLX в кластере без quorum queues не безопасен**

Стандартный (classic queue) механизм dead-lettering работает по принципу "at-most-once": сообщение перепубликуется в DLX **без включённых publisher confirms**. Это означает:

1. **Нет подтверждения доставки** — брокер отправляет сообщение в DLX и немедленно удаляет его из источниковой очереди, не дожидаясь подтверждения от целевой очереди.
2. **Потеря при недоступности узла** — classic queue живёт на одном узле кластера. Если этот узел недоступен в момент dead-lettering'а, сообщение теряется безвозвратно.
3. **Отсутствие репликации** — classic queue без mirroring не реплицируется; при падении узла-хоста очередь и её содержимое недоступны.
4. **Нет механизма retry** — если доставка в DLX не удалась (DLX или целевая очередь недоступны), попытка не повторяется.

**Как quorum queues решают эту проблему**

Quorum queues (введены в RabbitMQ 3.8, основаны на алгоритме Raft) решают обе проблемы:

1. **Репликация через Raft-консенсус** — данные очереди реплицируются на кворум узлов. При отказе одного узла очередь остаётся доступной и данные не теряются.

2. **At-least-once dead-lettering** (доступно с RabbitMQ 3.10, opt-in):
   - Включается политикой `dead-letter-strategy: at-least-once` совместно с `overflow: reject-publish`.
   - Внутри quorum queue создаётся выделенный **internal dead-letter consumer process**, расположенный на лидер-узле очереди.
   - Этот процесс работает как обычный потребитель с ручным подтверждением: читает мёртвые сообщения из внутреннего хранилища источниковой очереди и публикует их в DLX **с включёнными publisher confirms**.
   - Исходное сообщение удаляется из источниковой очереди только после получения подтверждения от всех целевых очередей.
   - Если целевая очередь временно недоступна, internal consumer периодически повторяет попытки.

**Компромиссы at-least-once dead-lettering**

- Сообщения могут быть доставлены в DLQ **более одного раза** (at-least-once, не exactly-once) — потребители DLQ должны быть идемпотентными.
- Мёртвые сообщения удерживаются в памяти internal consumer'а (prefetch 32) до получения confirms, что влияет на потребление памяти.
- Требуется включённый feature flag `stream_queue`.

**Итоговое сравнение**

| Характеристика | Classic Queue + DLX | Quorum Queue + DLX (at-least-once) |
|---|---|---|
| Репликация | Нет (по умолчанию) | Да (Raft) |
| Publisher confirms для DLX | Нет | Да |
| Гарантия доставки в DLX | At-most-once | At-least-once |
| Потеря при отказе узла | Возможна | Нет |
| Дубли в DLQ | Нет | Возможны (нужна идемпотентность) |

### Вопрос 34. Что такое prefetch count (QoS) в RabbitMQ и зачем его выставлять? Что произойдёт, если оставить prefetch=0 (неограниченный) при медленном consumer'е и большом потоке сообщений?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Consumer Prefetch | RabbitMQ](https://www.rabbitmq.com/docs/consumer-prefetch)

Prefetch count управляется через метод `basic.qos` протокола AMQP 0-9-1 и определяет максимальное количество неподтверждённых (unacknowledged) сообщений, которые брокер может отправить consumer'у одновременно. Как только лимит достигнут, RabbitMQ перестаёт отправлять новые сообщения на канал до тех пор, пока хотя бы одно из outstanding-сообщений не будет подтверждено (ack). Значение `0` трактуется как бесконечное — ограничений на доставку нет. Настройка применяется отдельно к каждому новому consumer'у на канале, а не делится между всеми consumer'ами канала (в отличие от спецификации AMQP). Значения в диапазоне 100–300 обычно дают оптимальную пропускную способность; финальное значение подбирается экспериментально для конкретной нагрузки.

#### Ответ от Claude

**Что такое prefetch count (QoS)**

Prefetch count — это механизм управления потоком (flow control) в RabbitMQ, реализуемый через AMQP-метод `basic.qos`. Он задаёт максимальное количество сообщений, доставленных consumer'у, но ещё не подтверждённых (unacknowledged). Пока это окно заполнено, брокер не отправляет новые сообщения на канал.

**Зачем выставлять prefetch**

1. **Защита consumer'а от перегрузки.** Без ограничения RabbitMQ мгновенно отправит все накопленные сообщения из очереди в TCP-буфер клиента; если consumer медленный, они будут накапливаться в памяти JVM/процесса.
2. **Равномерное распределение нагрузки.** При нескольких consumer'ах с round-robin dispatch конечный consumer не «жадно» захватит все сообщения раньше, чем успеет их обработать.
3. **Контроль back-pressure.** Брокер сохраняет неотправленные сообщения в очереди (на диске или в RAM), что позволяет другим consumer'ам их подхватить.

**Что происходит при prefetch=0 (неограниченный) + медленный consumer + большой поток**

- Брокер сбрасывает все сообщения из очереди в сетевой буфер и далее в память процесса consumer'а сразу.
- Consumer накапливает гигантское число неподтверждённых сообщений (in-flight messages) в своей памяти.
- Возможно **OOM (OutOfMemoryError)** на стороне consumer'а — JVM либо упадёт, либо начнёт интенсивный GC.
- При падении consumer'а **все накопленные им unacked-сообщения** будут возвращены в очередь брокером — это может вызвать резкий всплеск нагрузки на другие consumer'ы.
- Другие consumer'ы не получат сообщений вообще, пока первый «жадный» consumer жив, — нарушается балансировка.
- На стороне брокера очередь опустеет (сообщения переданы), но ни одно из них не будет фактически обработано до завершения медленного consumer'а.

**Рекомендации**

| Сценарий | Рекомендуемый prefetch |
|---|---|
| CPU-intensive обработка | 1–10 |
| I/O-bound (БД, HTTP) | 10–50 |
| Быстрая обработка (in-memory) | 100–300 |
| Автоматический ack (at most once) | 0 допустим, но осознанно |

Официальная документация рекомендует диапазон **100–300** как оптимальный для большинства сценариев, но конкретное значение всегда определяется нагрузочным тестированием.

### Вопрос 35. RabbitMQ гарантирует FIFO-порядок сообщений в одной очереди. При каких условиях этот порядок может нарушиться? Назови минимум два сценария (hints: priority queue, nack с requeue).

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Broker Semantics | RabbitMQ](https://www.rabbitmq.com/docs/semantics)

RabbitMQ гарантирует: сообщения, опубликованные через один канал, прошедшие через один exchange и одну очередь, будут получены в том же порядке, в котором они были отправлены. Начиная с версии 2.7.0 гарантия усилена: сообщения остаются в порядке публикации даже после requeue или при закрытии канала с unacked-сообщениями. Однако при наличии нескольких subscriber'ов на очереди возможны ситуации, когда отдельные consumer'ы наблюдают сообщения не в порядке публикации — если другой subscriber выполнил requeue, часть consumer'ов может получить сообщения в изменённом порядке.

#### Ответ от Claude

**Базовая гарантия RabbitMQ**

RabbitMQ гарантирует FIFO в рамках одного пути: один publisher-канал → один exchange → одна очередь → один consumer-канал. Начиная с версии 2.7.0 это распространяется и на redelivery после закрытия канала.

**Сценарии нарушения порядка**

**Сценарий 1: Priority Queue (очередь с приоритетами)**

Если очередь объявлена с аргументом `x-max-priority`, RabbitMQ переупорядочивает сообщения по приоритету перед доставкой. Сообщение, опубликованное позже, но с более высоким приоритетом, будет доставлено раньше сообщения с низким приоритетом, пришедшего первым. Пример: три сообщения A (priority=1), B (priority=3), C (priority=2) будут доставлены в порядке B → C → A, хотя в очередь попали в порядке A → B → C.

**Сценарий 2: basic.nack / basic.reject с requeue=true при множественных consumer'ах**

Когда consumer возвращает сообщение с `requeue=true`, RabbitMQ делает попытку вернуть его на исходную позицию в очереди, но документация явно оговаривает: «если это невозможно (из-за конкурентных доставок и подтверждений от других consumer'ов), сообщение будет поставлено в позицию ближе к голове очереди». В многопоточном сценарии это означает, что requeued-сообщение окажется перед другими сообщениями, которые были поставлены в очередь позже него, что нарушает исходный порядок.

**Дополнительные сценарии**

- **Несколько consumer'ов на одной очереди:** каждый consumer получает разные сообщения и обрабатывает их с разной скоростью; с точки зрения глобального бизнес-порядка обработка нарушается.
- **Consumer priorities** (`x-consumer-priority`): consumer с более высоким приоритетом получает сообщения раньше остальных, что влияет на порядок обработки.
- **Automatic requeue при обрыве соединения:** если consumer упал с unacked-сообщениями, они возвращаются в очередь, но за это время другие consumer'ы могли продвинуть хвост очереди вперёд.

**Как гарантировать строгий FIFO**

Использовать **Single Active Consumer** (`x-single-active-consumer=true`) в сочетании с `prefetch=1`. В этом случае только один consumer обрабатывает сообщения, и requeue гарантированно восстанавливает оригинальную позицию.

### Вопрос 36. Паттерн Transactional Outbox: зачем он нужен при работе с Kafka или RabbitMQ? Какую проблему он решает и чем отличается от прямой публикации события внутри транзакции БД?

**Категория:** Брокеры сообщений · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Pattern: Transactional outbox | microservices.io](https://microservices.io/patterns/data/transactional-outbox.html)

Проблема: сервис должен атомарно обновить базу данных И отправить сообщение брокеру. Прямое использование распределённых транзакций (2PC) зачастую невозможно — база данных или брокер могут не поддерживать 2PC, а даже если и поддерживают, это вносит сильную связанность. Решение: сервис сначала сохраняет сообщение в специальную таблицу outbox в той же транзакции, что обновляет бизнес-сущности. Отдельный процесс (Message Relay) читает таблицу outbox и публикует сообщения брокеру. Это гарантирует, что сообщение отправится тогда и только тогда, когда транзакция БД зафиксирована. Если сервис упадёт до отправки, сообщение останется в outbox и будет отправлено после восстановления. Реализуется через два подхода: Polling Publisher (периодический опрос таблицы outbox) и Transaction Log Tailing / CDC (отслеживание лога транзакций БД, например через Debezium).

#### Ответ от Claude

**Проблема: Dual Write**

Когда микросервис должен и сохранить данные в БД, и опубликовать событие в брокер, возникает «двойная запись» (dual write). Эти два действия не атомарны между собой:

1. **БД зафиксировалась, брокер упал** → данные обновлены, событие потеряно → downstream-сервисы не знают об изменении.
2. **Событие отправлено, БД откатилась** → подписчики получили событие о несуществующем факте → инконсистентность данных.

Распределённые транзакции (2PC / XA) теоретически решают это, но на практике: Kafka не поддерживает XA; большинство облачных managed-сервисов тоже; 2PC вносит высокую latency и точку отказа (координатор).

**Решение: Transactional Outbox**

Идея: включить запись события в ту же локальную транзакцию БД, что и изменение бизнес-данных. Для этого в схеме БД создаётся таблица `outbox` (или `events`, `domain_events`). В одной транзакции сервис:
1. Обновляет бизнес-таблицы.
2. Вставляет запись о событии в таблицу `outbox`.

Отдельный компонент **Message Relay** читает `outbox` и публикует события в брокер, после чего помечает их как отправленные (или удаляет).

**Чем отличается от прямой публикации внутри транзакции БД**

| Аспект | Прямая публикация | Transactional Outbox |
|---|---|---|
| Атомарность | Нет — два независимых ресурса | Да — запись в outbox атомарна с бизнес-данными |
| Потеря события при сбое | Возможна (сбой после commit, до send) | Невозможна — событие в БД до отправки |
| Rollback | Событие уже отправлено, не отозвать | Событие не отправлено, откатится вместе с транзакцией |
| Зависимость от доступности брокера | Транзакция блокируется если брокер недоступен | Транзакция успешна, relay повторит попытку позже |
| Сложность | Низкая | Выше (нужен relay, таблица outbox) |

**Прямая публикация внутри транзакции** не работает даже если вызвать `broker.send()` внутри блока `@Transactional`. Spring-транзакция управляет только JDBC-соединением; вызов брокера происходит сразу, а не при commit. При последующем rollback транзакции БД событие уже ушло подписчикам.

**Реализации Message Relay**

- **Polling Publisher:** фоновый поток/scheduler периодически делает `SELECT ... WHERE sent = false`, публикует и помечает отправленными. Просто, но добавляет latency и нагрузку на БД.
- **Transaction Log Tailing / CDC:** инструмент типа Debezium читает WAL (PostgreSQL) или binlog (MySQL) и публикует изменения таблицы outbox в Kafka/RabbitMQ в реальном времени. Минимальная latency, нет polling-нагрузки, но сложнее в эксплуатации.

**At-least-once семантика**

Relay может опубликовать одно и то же событие дважды (при сбое после publish, но до пометки «отправлено»). Поэтому consumer'ы событий должны быть **идемпотентны** — либо по бизнес-логике, либо через дедупликацию по уникальному `event_id`.
### Вопрос 37. Как работает log compaction в Kafka? Чем cleanup.policy=compact отличается от cleanup.policy=delete? Что такое tombstone-запись и когда она окончательно удаляется из лога?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Kafka Log Compaction — Confluent Documentation](https://docs.confluent.io/kafka/design/log_compaction.html)

Лог Kafka состоит из двух частей: **head** (голова) и **tail** (хвост). Head — это «живая» часть лога с плотной последовательной нумерацией смещений, куда поступают все новые сообщения. Tail — уже скомпактированная часть, где для каждого ключа сохранено только последнее значение.

Compaction выполняется фоновым потоком-cleaner, который периодически сканирует партиции, подходящие для уплотнения. Cleaner не блокирует чтение и может быть ограничен по пропускной способности I/O, чтобы не мешать продюсерам и консюмерам. Cleaner строит **offset map** — отображение ключа на последнее смещение с этим ключом — затем переписывает сегменты лога, выбрасывая устаревшие записи.

Ключевое свойство: смещения остаются постоянными и валидными даже если запись с этим смещением была удалена в ходе compaction.

**cleanup.policy=delete** — политика по умолчанию. Старые сегменты удаляются целиком при достижении порога по времени (`retention.ms`) или размеру (`retention.bytes`). Удаление происходит на уровне сегментов, а не отдельных сообщений.

**cleanup.policy=compact** — Kafka сохраняет только последнее значение для каждого ключа. Старые значения того же ключа удаляются при compaction. Это позволяет восстановить актуальное состояние, читая лог с самого начала. Оба значения можно задать через запятую: `cleanup.policy=delete,compact` — в этом случае compaction применяется к хвосту, а старые сегменты дополнительно удаляются по времени/размеру.

**Tombstone-запись** — это сообщение с ключом и null-значением (payload = null). Такая запись сигнализирует об удалении ключа из лога. Tombstone удаляет все предшествующие записи с тем же ключом в ходе compaction. Сам tombstone удаляется из лога не сразу: он сохраняется в течение времени, заданного параметром `delete.retention.ms` (по умолчанию 86 400 000 мс = 24 часа). Это даёт консюмерам гарантированный временной интервал для наблюдения tombstone-а при чтении с offset=0. Если консюмер не успевает прочитать лог в течение `delete.retention.ms`, он рискует пропустить tombstone и не узнать об удалении ключа.

Ключевые параметры compaction:
- `min.compaction.lag.ms` (по умолчанию 0) — минимальное время, в течение которого запись не будет включена в compaction после записи в лог.
- `max.compaction.lag.ms` — максимальное время, по истечении которого запись обязательно попадёт в compaction.
- `min.cleanable.dirty.ratio` (по умолчанию 0.5) — порог «грязных» байт (ещё не скомпактированных) к общему объёму лога, при превышении которого запускается очередной цикл compaction.

#### Ответ от Claude

**Как работает log compaction**

Kafka делит лог партиции на две логические части:

- **Head (голова)** — последние, ещё не скомпактированные сегменты. Содержит все записи, включая дубликаты по ключу.
- **Tail (хвост)** — уже скомпактированные сегменты. Содержит только последнее значение каждого ключа.

Фоновый **log cleaner** (пул потоков) периодически выбирает партицию для compaction по следующему критерию: отношение объёма «грязных» (dirty) байт к общему объёму лога превышает `min.cleanable.dirty.ratio`. Cleaner строит хэш-карту (offset map): ключ → последнее смещение с этим ключом. Затем переписывает сегменты, выбрасывая записи, чьё смещение меньше максимального для данного ключа. Смещения оставшихся записей не изменяются — это важно для корректности консюмеров.

**Отличие cleanup.policy=compact от cleanup.policy=delete**

| Аспект | cleanup.policy=delete | cleanup.policy=compact |
|---|---|---|
| Единица удаления | Целый сегмент лога | Отдельные записи (по ключу) |
| Критерий удаления | Возраст/размер сегмента | Наличие более новой записи с тем же ключом |
| Гарантия | Сообщения старше N часов/байт будут удалены | Для каждого ключа будет сохранено последнее значение |
| Возможность воссоздать текущее состояние | Нет (если данные истекли) | Да (лог — полный снимок состояния) |
| Поддержка null-значений | Нет особой семантики | Tombstone: null-значение = удаление ключа |

Комбинация `cleanup.policy=delete,compact` позволяет применять оба механизма: compaction удаляет устаревшие значения ключей, а delete убирает совсем старые сегменты по времени/размеру.

**Tombstone-запись**

Tombstone — это запись с ненулевым ключом и `null` в качестве значения. Она выполняет роль маркера удаления:
1. При compaction все предшествующие записи с тем же ключом удаляются.
2. Сам tombstone остаётся в логе на время `delete.retention.ms` (по умолчанию 24 часа).
3. По истечении `delete.retention.ms` tombstone окончательно удаляется из лога при следующем проходе cleaner.

Задержка удаления tombstone критична: если tombstone исчезнет раньше, чем отстающий консюмер его прочтёт, консюмер не узнает об удалении ключа и получит неверный снимок состояния. Поэтому `delete.retention.ms` задаёт верхнюю границу допустимого отставания консюмера при работе с компактированными топиками.

### Вопрос 38. Что такое High Watermark (HW) и Log End Offset (LEO) в Kafka? Почему consumer не может читать сообщения выше High Watermark, и как эти метрики влияют на поведение ISR-реплик?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Kafka: Log End Offset (LEO) vs High Watermark (HW) in Leader Replica](https://www.w3tutorials.net/blog/kafka-difference-between-log-end-offset-leo-vs-high-watermark-hw/)

**Log End Offset (LEO)** — смещение следующего сообщения, которое будет записано в реплику. LEO ведётся независимо каждой репликой (лидером и всеми фолловерами).

**High Watermark (HW)** — наибольшее смещение, успешно скопированное во все реплики из набора ISR (In-Sync Replicas). HW представляет собой минимум из LEO всех ISR-реплик данной партиции.

Разница между понятиями:

| Характеристика | LEO | HW |
|---|---|---|
| Область видимости | На каждую реплику (у каждой своё) | На партицию (единое значение) |
| Назначение | Отслеживает позицию следующей записи | Маркирует зафиксированные, надёжно реплицированные данные |
| Обновляется | Немедленно при записи сообщения | После того как все ISR-фолловеры подтвердили получение |
| Видимость консюмеру | Не доступно | Консюмеры читают до HW-1 включительно |

Консюмеры не могут читать сообщения выше HW, поскольку эти сообщения ещё не гарантированно реплицированы во все ISR-реплики. Если лидер упадёт до репликации, новый лидер не будет иметь этих сообщений, и консюмер получил бы данные, которые впоследствии исчезли — это нарушило бы консистентность.

Лидер пересчитывает HW как минимум из LEO всех ISR-реплик. Когда все ISR-реплики догоняют лидера, HW совпадает с LEO лидера и новые сообщения становятся доступны консюмерам.

#### Ответ от Claude

**Log End Offset (LEO)**

LEO — это смещение, которое будет присвоено следующему сообщению при записи в данную реплику. Каждая реплика (и лидер, и каждый фолловер) ведёт своё LEO независимо. Как только сообщение записано в локальный лог реплики, её LEO увеличивается на 1.

**High Watermark (HW)**

HW — это наибольшее смещение, для которого все реплики из набора ISR подтвердили запись. Формально: `HW = min(LEO_leader, LEO_follower1, LEO_follower2, ...)` по всем ISR-репликам. HW растёт монотонно и является «рубежом надёжности»: все смещения строго меньше HW гарантированно присутствуют во всех ISR-репликах.

**Почему консюмеры не могут читать выше HW**

Сообщения в диапазоне `[HW, LEO_leader)` называются **uncommitted** (незафиксированными). Лидер уже записал их локально, но не все ISR-фолловеры ещё их скопировали. Если лидер упадёт в этот момент, новый лидер (один из фолловеров) может не иметь этих сообщений — и консюмер, прочитавший их заранее, увидит «исчезнувшие» данные. Официальная документация Kafka формулирует это так: «Only committed messages are ever given out to the consumer» (только зафиксированные сообщения отдаются консюмеру).

**Как HW влияет на поведение ISR-реплик**

Фолловеры работают как специализированные консюмеры: они периодически отправляют лидеру **FetchRequest**, начиная с текущего LEO фолловера. Лидер возвращает данные и текущее значение HW. Процесс продвижения HW:

1. Продюсер записывает сообщение на лидера — LEO лидера увеличивается.
2. Фолловер отправляет FetchRequest, получает новые данные, записывает их — LEO фолловера увеличивается.
3. Лидер узнаёт о новом LEO фолловера из следующего FetchRequest (в заголовке запроса указывается fetch offset).
4. Лидер пересчитывает HW как `min(LEO всех ISR)` и продвигает его.
5. Фолловер узнаёт обновлённый HW из ответа на следующий FetchRequest.

Если фолловер не присылает FetchRequest в течение `replica.lag.time.max.ms` (по умолчанию 30 секунд) или не успевает догнать LEO лидера, лидер исключает его из ISR. Это немедленно влияет на HW: теперь HW вычисляется без учёта отставшей реплики, что может позволить HW продвинуться быстрее, но снижает уровень надёжности (меньше реплик гарантируют сохранность).

**Связь с параметром acks у продюсера**

- `acks=0` — продюсер не ждёт никакого подтверждения.
- `acks=1` — продюсер ждёт подтверждения только от лидера (сообщение выше HW до репликации во фолловеры).
- `acks=all` (или `-1`) — продюсер ждёт, пока все ISR-реплики подтвердят запись, то есть сообщение попадает под HW до того как продюсер получает подтверждение.

### Вопрос 39. Как параметры batch.size, linger.ms и compression.type влияют на производительность Kafka-producer? Какой компромисс между latency и throughput возникает при увеличении linger.ms? Какие алгоритмы компрессии сравнительно лучше для CPU vs степени сжатия?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Apache Kafka Message Compression — Confluent Blog](https://www.confluent.io/blog/apache-kafka-message-compression/)

Продюсер поддерживает буферы ненадёжных записей для каждой партиции, с размером, заданным параметром `batch.size`. Накопление батча позволяет объединить несколько сообщений в один сетевой запрос, снижая overhead. Маленький батч делает батчинг редким явлением и может снижать пропускную способность (размер 0 полностью отключает батчинг). Слишком большой батч избыточно расходует память.

`linger.ms` задаёт время ожидания дополнительных записей перед отправкой батча. По умолчанию значение изменено с 0 до 5 мс в Kafka 4.0, так как «эффективность больших батчей, как правило, обеспечивает схожую или меньшую latency продюсера, несмотря на увеличение паузы». Увеличение `linger.ms` ведёт к меньшему количеству, но более крупных запросов при ненагруженной системе — за счёт небольшой дополнительной задержки.

Алгоритмы сжатия (значения `compression.type`): `none`, `gzip`, `snappy`, `lz4`, `zstd`.

- **gzip** — высокая степень сжатия, но максимальная нагрузка на CPU; не рекомендуется в высоконагруженных системах.
- **snappy** — умеренное сжатие при минимальных затратах CPU; подходит для приоритета скорости над степенью сжатия.
- **lz4** — самый быстрый алгоритм по скорости сжатия/разжатия; рекомендован по умолчанию для приоритета производительности.
- **zstd** — современный алгоритм (появился в Kafka 2.1.0): степень сжатия сопоставима с gzip, скорость ближе к lz4; рекомендован, если нужна высокая степень сжатия при умеренной нагрузке на CPU.

Компрессия применяется к целому батчу, поэтому эффективность батчинга напрямую влияет на степень сжатия: чем крупнее батч, тем лучше сжимается.

#### Ответ от Claude

**batch.size**

`batch.size` задаёт максимальный размер батча в байтах для одной партиции. Продюсер накапливает сообщения в буфере и отправляет их одним запросом, как только буфер достигает `batch.size` или истекает `linger.ms`. Увеличение `batch.size` повышает throughput за счёт снижения количества сетевых запросов и улучшения степени сжатия (компрессия применяется к целому батчу). Недостаток — повышенный расход памяти (по одному буферу на каждую активную партицию).

**linger.ms**

`linger.ms` — максимальное время, в течение которого продюсер ждёт дополнительных сообщений, прежде чем отправить батч. Если батч достигает `batch.size` раньше истечения `linger.ms`, он отправляется немедленно.

Компромисс latency vs throughput:
- `linger.ms=0` — сообщения отправляются немедленно, минимальная задержка, но каждое сообщение может уйти отдельным запросом (низкий throughput при маленькой нагрузке).
- Увеличение `linger.ms` позволяет накопить больше сообщений в батч — меньше запросов, выше throughput, лучше сжатие. Но добавляется искусственная задержка, что критично для latency-sensitive приложений.
- В Kafka 4.0 значение по умолчанию изменено с 0 до 5 мс, так как на практике прирост throughput от небольшой задержки перевешивает её стоимость.

Типичная рекомендация: для максимального throughput — `linger.ms` от 20 до 100 мс, для минимальной latency — `linger.ms=0` или `linger.ms=1`.

**compression.type**

Компрессия применяется к целому батчу (не к отдельному сообщению), что повышает эффективность: чем больше батч, тем выше коэффициент сжатия. Компрессия снижает сетевой трафик и нагрузку на брокер при записи, но добавляет CPU-нагрузку на продюсере (компрессия) и консюмере (декомпрессия).

Важно: если `compression.type` топика отличается от `compression.type` продюсера, брокер будет перекомпрессировать батч при записи — это создаёт избыточную нагрузку. Рекомендуется согласовывать настройки.

**Сравнение алгоритмов**

| Алгоритм | Скорость (CPU) | Степень сжатия | Когда выбирать |
|---|---|---|---|
| **lz4** | Максимальная | Низкая / средняя | Приоритет скорости и низкой latency; общая рекомендация для производительности |
| **snappy** | Высокая | Средняя | Баланс скорости и сжатия; подходит для потоков с умеренным CPU |
| **zstd** | Средняя | Высокая (близко к gzip) | Лучший выбор, когда нужна высокая степень сжатия без overhead gzip; Kafka 2.1+ |
| **gzip** | Низкая (высокий CPU) | Максимальная | Не рекомендуется для высоких нагрузок; пригоден при критичном ограничении по трафику |
| **none** | — | — | Отладка или случаи, где сжатие не нужно |

Итоговый выбор:
- **Для CPU-эффективности и скорости**: lz4.
- **Для максимального сжатия при разумном CPU**: zstd (рекомендован для большинства production-систем на Kafka 2.1+).
- **gzip** — избегать в high-throughput сценариях из-за высокой CPU-нагрузки.

Эффективность компрессии напрямую зависит от размера батча: параметры `batch.size` и `linger.ms` должны настраиваться в связке с `compression.type`.

### Вопрос 40. Что такое статическое членство в consumer group (static membership, group.instance.id) в Kafka? Какую проблему оно решает и как изменяется поведение при рестарте consumer по сравнению с динамическим членством?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [KIP-345: Introduce static membership protocol to reduce consumer rebalances](https://cwiki.apache.org/confluence/display/KAFKA/KIP-345:+Introduce+static+membership+protocol+to+reduce+consumer+rebalances)

Статическое членство (static membership) введено в Apache Kafka 2.3 через KIP-345. Основная проблема, которую оно решает, — избыточные rebalance при перезапуске consumer-экземпляров, особенно критичные для stateful-приложений.

При обычном (динамическом) членстве каждый перезапуск consumer генерирует новый member ID, что запускает полный rebalance группы. В тяжёлых stateful-приложениях (например, Kafka Streams) rebalance означает перераспределение партиций и огромный объём переноса состояния между экземплярами. При rolling deployment каждый рестарт экземпляра запускал отдельный rebalance, что могло заблокировать всю группу на длительное время.

Статическое членство вводит параметр `group.instance.id` — уникальный строковый идентификатор, который пользователь назначает каждому consumer-экземпляру. Брокер хранит маппинг `group.instance.id` → `member.id` и кешированное назначение партиций.

Поведение при рестарте:
- **Динамическое членство**: consumer при отключении отправляет `LeaveGroup` запрос, координатор запускает rebalance немедленно. При повторном подключении consumer получает новый `member.id` и снова участвует в rebalance.
- **Статическое членство**: consumer при отключении NOT отправляет `LeaveGroup`. Координатор ждёт истечения `session.timeout.ms`. Если consumer возвращается до истечения таймаута с тем же `group.instance.id`, координатор возвращает ему кешированное назначение партиций без rebalance.

Рекомендуется сочетать статическое членство с увеличенным `session.timeout.ms` (до 30 минут и более), чтобы дать время на перезапуск приложения без запуска лишнего rebalance.

#### Ответ от Claude

**Проблема динамического членства.** В стандартной модели Kafka каждый consumer идентифицируется ephemeral `member.id`, генерируемым при каждом `JoinGroup`. Любой рестарт — даже кратковременный — приводит к немедленному rebalance всей группы, так как consumer отправляет `LeaveGroup` при отключении. Для stateful-приложений (Kafka Streams, приложения с локальным кэшем) это катастрофа: rebalance перераспределяет партиции, и новый владелец вынужден заново восстанавливать состояние — дорогостоящая операция при больших объёмах данных.

**Механизм static membership.** KIP-345 (доступен с Kafka 2.3) вводит параметр `group.instance.id` (тип `string`, по умолчанию `null`). Когда параметр задан, consumer считается статическим членом группы. Брокер-координатор хранит маппинг `group.instance.id` → `member.id` и кеш назначения партиций для каждого статического члена.

**Поведение при рестарте — сравнение:**

| Аспект | Динамическое членство | Статическое членство |
|---|---|---|
| Идентификатор | Ephemeral `member.id`, генерируется при каждом `JoinGroup` | Постоянный `group.instance.id`, задаётся пользователем |
| `LeaveGroup` при отключении | Отправляется — rebalance немедленно | Не отправляется — координатор ждёт `session.timeout.ms` |
| Повторное подключение до истечения таймаута | Новый `member.id`, полный rebalance | Получает кешированное назначение, rebalance не происходит |
| Повторное подключение после истечения таймаута | Rebalance | Rebalance (таймаут истёк — coordinator убрал члена) |

**Взаимодействие с `session.timeout.ms`.** Статическое членство целесообразно сочетать с увеличенным `session.timeout.ms`. Если перезапуск занимает больше `session.timeout.ms`, координатор всё равно исключит consumer из группы и запустит rebalance. KIP-345 поднял рекомендуемый максимум `session.timeout.ms` до 30 минут для сценариев со статическими членами.

**Типичные сценарии применения:** rolling deployment приложений с Kafka Streams, consumer-приложения с тяжёлым in-memory состоянием, ситуации с частыми кратковременными сбоями сети.

**Ограничение:** в каждый момент времени в группе может быть только один член с данным `group.instance.id`. Попытка второго consumer с тем же ID войти в группу вызовет ошибку.
