### Вопрос 41. Как работает follower fetching (KIP-392) в Kafka? Что такое rack awareness применительно к consumer, для чего используется параметр client.rack, и какой компромисс возникает между latency и сетевыми расходами?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [KIP-392: Allow consumers to fetch from closest replica](https://cwiki.apache.org/confluence/display/KAFKA/KIP-392:+Allow+consumers+to+fetch+from+closest+replica)

До KIP-392 consumer мог читать данные только у лидера партиции, независимо от того, в какой зоне доступности (AZ) находился сам consumer. В мультидатацентровых и мультизонных облачных развёртываниях (AWS, GCP, Azure) это порождало дорогостоящий кросс-зональный трафик.

KIP-392 (доступен с Kafka 2.4) позволяет consumer читать из follower-реплики, находящейся в той же зоне. Механизм работает следующим образом:

1. Consumer указывает свою зону через параметр `client.rack` (строка, по умолчанию пустая).
2. В каждом Fetch-запросе consumer передаёт значение `client.rack` брокеру-лидеру.
3. Брокер использует реализацию `replica.selector.class` (по умолчанию `LeaderSelector`; для rack-aware логики — `RackAwareReplicaSelector`) для выбора предпочтительной реплики.
4. `RackAwareReplicaSelector` ищет online-реплику с `broker.rack`, совпадающим с `client.rack`. Если такие реплики есть — выбирает наиболее актуальную (наименее отстающую). Если совпадений нет — fallback на лидера.
5. Брокер возвращает consumer поле `PreferredReadReplica` в ответе на Fetch-запрос, и дальнейшие запросы consumer направляет напрямую к этой реплике.

Компромисс между latency и стоимостью:
- **Стоимость**: чтение из follower той же AZ исключает межзональный трафик, что в AWS/GCP означает нулевую стоимость за передачу данных внутри AZ.
- **Latency**: follower-реплики могут отставать от лидера. Uncommitted записи недоступны у follower до момента распространения high watermark. Это увеличивает end-to-end latency между продюсером и consumer.

#### Ответ от Claude

**Проблема до KIP-392.** Kafka традиционно позволяла читать только из лидера партиции. В облачных развёртываниях с несколькими зонами доступности (AZ) это означало, что consumer в AZ-B читал данные у лидера в AZ-A, порождая межзональный трафик. В AWS, GCP, Azure такой трафик тарифицируется как egress и может составлять значительную статью расходов.

**Механизм follower fetching (KIP-392, Kafka 2.4+).**

Протокол изменён следующим образом:
1. На брокерах задаётся `broker.rack` — идентификатор AZ (например, `us-east-1a`).
2. На consumer задаётся `client.rack` — идентификатор AZ consumer-приложения. Значение должно совпадать с `broker.rack` желаемого follower-брокера.
3. На брокерах задаётся `replica.selector.class=org.apache.kafka.common.replica.RackAwareReplicaSelector` для включения rack-aware логики выбора реплик.
4. Consumer включает `client.rack` в каждый Fetch-запрос.
5. Брокер-лидер, получив запрос, определяет `PreferredReadReplica`: ищет online-реплику с `broker.rack == client.rack`, выбирает наиболее актуальную. При отсутствии совпадения — возвращает лидера.
6. Consumer получает `PreferredReadReplica` в ответе и начинает отправлять запросы напрямую этой реплике.

**Rack awareness применительно к consumer** означает, что consumer «знает» о своём физическом/сетевом расположении (AZ, стойка, датацентр) и предпочитает читать из реплики, максимально близкой по сети.

**Параметр `client.rack`** (тип `string`, по умолчанию пустая строка) — идентификатор расположения consumer. В AWS рекомендуется использовать Availability Zone ID (например, `use2-az1`), а не Zone Name (`us-east-2a`), так как имена зон могут отличаться между аккаунтами AWS при одинаковом физическом расположении.

**Компромисс latency vs. сетевые расходы:**

| Аспект | Чтение у лидера | Чтение у follower (rack-aware) |
|---|---|---|
| Межзональный трафик | Есть при разных AZ | Отсутствует при совпадении rack |
| Стоимость сети | Высокая (egress between AZ) | Минимальная |
| End-to-end latency | Минимальная (лидер всегда актуален) | Выше на величину lag репликации |
| Риск | Нет | OFFSET_NOT_AVAILABLE при отставании follower |

Follower-реплика отстаёт от лидера на время распространения high watermark. Uncommitted записи consumer у follower не увидит до момента подтверждения репликации. Это увеличивает consumer lag относительно момента записи продюсера.

**Расширение KIP-881** (Kafka 3.x) дополняет KIP-392: делает rack-aware не только fetching, но и само назначение партиций при rebalance — consumer в AZ-A получает в assignment преимущественно партиции, чьи leaders или followers находятся в AZ-A.

### Вопрос 42. Для чего в Kafka используются параметры fetch.min.bytes и fetch.max.wait.ms на стороне consumer? Как они взаимодействуют между собой, и какое влияние оказывают на throughput и latency?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Consumer Configs | Apache Kafka](https://kafka.apache.org/41/configuration/consumer-configs/)

**fetch.min.bytes** (тип: `int`, значение по умолчанию: `1`)
"The minimum amount of data the server should return for a fetch request. If insufficient data is available the request will wait for that much data to accumulate before answering the request. Setting this to a larger value will cause the server to wait for larger amounts of data to accumulate which can improve server throughput a bit at the cost of some additional latency."

**fetch.max.wait.ms** (тип: `int`, значение по умолчанию: `500`)
"The maximum amount of time the server will block before answering the fetch request there isn't sufficient data to immediately satisfy the requirement given by fetch.min.bytes."

Параметры работают как пара условий: брокер отвечает на fetch-запрос, когда выполняется хотя бы одно из двух:
- накоплено не менее `fetch.min.bytes` данных, или
- прошло `fetch.max.wait.ms` миллисекунд с момента получения запроса.

Значение `fetch.min.bytes=1` означает, что ответ будет отправлен, как только появится хотя бы один байт данных, что минимизирует latency. Увеличение `fetch.min.bytes` заставляет брокер ждать накопления большего объёма данных — это повышает throughput за счёт увеличения latency.

#### Ответ от Claude

**Назначение параметров.** `fetch.min.bytes` и `fetch.max.wait.ms` управляют поведением брокера при обработке Fetch-запросов от consumer. Их цель — позволить оператору гибко настраивать баланс между latency (задержкой доставки) и throughput (пропускной способностью).

**Механизм взаимодействия.**

Когда consumer отправляет Fetch-запрос, брокер проверяет, сколько данных доступно:
- Если данных >= `fetch.min.bytes` — ответ отправляется немедленно.
- Если данных < `fetch.min.bytes` — запрос помещается в «purgatory» (внутренняя структура брокера для отложенных запросов) и брокер ждёт.

Из purgatory запрос выходит при наступлении первого из двух событий:
1. Накопилось не менее `fetch.min.bytes` данных.
2. Истекло `fetch.max.wait.ms` миллисекунд.

Таким образом, `fetch.max.wait.ms` является верхней границей ожидания и гарантирует, что consumer получит ответ не позже чем через заданный интервал, даже если порог `fetch.min.bytes` не достигнут.

**Влияние на throughput и latency:**

`fetch.min.bytes` (по умолчанию: 1 байт):
- Значение `1` оптимизировано для минимальной latency: ответ отправляется при появлении любых данных.
- Увеличение значения заставляет брокера формировать более крупные батчи. Это снижает количество Fetch-запросов в единицу времени, уменьшает накладные расходы на сеть и повышает throughput. Цена — увеличение latency, так как consumer дольше ждёт первого байта.

`fetch.max.wait.ms` (по умолчанию: 500 мс):
- Ограничивает максимальную latency: при любом значении `fetch.min.bytes` consumer получит ответ не позже 500 мс.
- Уменьшение значения снижает максимальную latency, но может привести к частым ответам с малым объёмом данных (если партиция не очень активна).
- Увеличение значения допускает более длинные периоды ожидания, что полезно при высоком `fetch.min.bytes` для формирования крупных батчей.

**Взаимозависимость:**

Параметры образуют логическое «OR»-условие: ответ при `fetch.min.bytes` ИЛИ при `fetch.max.wait.ms`. Если выставить большой `fetch.min.bytes` и маленький `fetch.max.wait.ms`, таймаут будет срабатывать раньше порога объёма — и выигрыша в throughput не будет. Для эффективного batching нужно согласованно увеличивать оба значения.

**Важное ограничение:** значение `fetch.max.wait.ms` должно быть меньше `request.timeout.ms` (по умолчанию 30 000 мс). В противном случае consumer может получить timeout-ошибку раньше, чем брокер успеет ответить.

**Типичные сценарии настройки:**

| Сценарий | fetch.min.bytes | fetch.max.wait.ms |
|---|---|---|
| Минимальная latency (real-time) | 1 (по умолчанию) | 100-500 мс |
| Высокий throughput (batch-обработка) | 65536 (64 KB) и выше | 500-1000 мс |
| Сбалансированный режим | 1024-8192 | 500 мс (по умолчанию) |

### Вопрос 43. Чем отличается Publisher Confirms в RabbitMQ от Consumer Acknowledgements? Как брокер подтверждает получение сообщения от publisher, что означает nack со стороны брокера, и что происходит при публикации сообщения с флагом mandatory в exchange без привязанных очередей?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Consumer Acknowledgements and Publisher Confirms | RabbitMQ](https://www.rabbitmq.com/docs/confirms)

Publisher Confirms и Consumer Acknowledgements — это два полностью ортогональных механизма, которые решают схожие задачи, но в разных направлениях взаимодействия.

**Consumer Acknowledgements** — подтверждения идут от потребителя к брокеру. Потребитель сообщает, что сообщение успешно получено и обработано, и брокер может пометить его для удаления. Методы: `basic.ack` (успешная обработка), `basic.reject` (неуспешная обработка одного сообщения), `basic.nack` (отклонение одного или нескольких сообщений).

**Publisher Confirms** — подтверждения идут от брокера к издателю. Механизм охватывает взаимодействие издателя с узлом, к которому он подключён, а также с лидером реплики очереди или стрима. Чтобы включить Publisher Confirms, канал переводится в режим подтверждений (`channel.confirmSelect()`). После этого брокер отвечает `basic.ack` (сообщение принято и обработано) или `basic.nack` (внутренняя ошибка, брокер отказывается от ответственности за сообщение).

`basic.nack` со стороны брокера означает, что произошла внутренняя ошибка в Erlang-процессе, ответственном за очередь. Это редкий случай — он сигнализирует, что брокер не может гарантировать обработку сообщения, и клиент вправе повторить публикацию.

При использовании флага `mandatory`: если сообщение опубликовано с `mandatory=true` и не может быть маршрутизировано ни в одну очередь, брокер сначала отправляет издателю `basic.return` (уведомление о невозможности маршрутизации), а затем — `basic.ack`. При `mandatory=false` (значение по умолчанию) не маршрутизированное сообщение просто отбрасывается или перенаправляется в alternate exchange, если он настроен.

#### Ответ от Claude

**Publisher Confirms vs Consumer Acknowledgements**

Оба механизма решают проблему надёжной доставки, но ортогональны и не зависят друг от друга:

| Характеристика | Consumer Acknowledgements | Publisher Confirms |
|---|---|---|
| Направление | Потребитель -> Брокер | Брокер -> Издатель |
| Цель | Подтвердить успешную обработку, разрешить удаление | Подтвердить, что брокер принял сообщение |
| Методы | `basic.ack`, `basic.nack`, `basic.reject` | `basic.ack`, `basic.nack` |
| Активация | Включена по умолчанию (в автоматическом или ручном режиме) | Канал переводится в confirm-режим: `channel.confirmSelect()` |

**Что означает broker nack для издателя**

`basic.nack` от брокера сигнализирует о внутренней ошибке в Erlang-процессе очереди — брокер отказывается от ответственности за сообщение. Это не штатная ситуация (в отличие от `basic.nack` от потребителя); при получении `basic.nack` от брокера рекомендуется повторная публикация.

**Флаг mandatory и exchange без привязанных очередей**

- `mandatory=false` (умолчание): не маршрутизированное сообщение молча отбрасывается, либо перенаправляется в alternate exchange.
- `mandatory=true`: брокер возвращает сообщение издателю через `basic.return` до отправки `basic.ack`. Издатель обязан зарегистрировать `ReturnListener` для обработки возвращённых сообщений (логирование, повторная отправка на другой exchange и т.д.).

Таким образом, даже при `mandatory=true` брокер всё равно отправляет `basic.ack` — сообщение было технически получено и обработано брокером (решение о не-маршрутизации принято), поэтому confirm-ack и return — не взаимоисключающие события.

### Вопрос 44. В чём разница между Connection и Channel в RabbitMQ? Почему Channel не является потокобезопасным, и какова рекомендованная модель использования соединений и каналов в многопоточном Java-приложении?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Java Client API Guide | RabbitMQ](https://www.rabbitmq.com/client-libraries/java-api-guide)

**Connection** — это единственное TCP-соединение с брокером RabbitMQ. TCP-соединения ресурсоёмки: они требуют выделения сокетов, потребляют память на клиенте и сервере, учитываются в правилах брандмауэра.

**Channel** — это логическое «лёгкое соединение» внутри одного TCP-соединения. Несколько каналов мультиплексируются поверх одного соединения, каждый с уникальным идентификатором. Канал существует только в контексте соединения и не может существовать самостоятельно.

**Почему Channel не потокобезопасен:** разделение одного экземпляра `Channel` между потоками следует избегать. Одновременная публикация на общем канале из нескольких потоков приводит к некорректному перемешиванию фреймов на уровне протокола, что вызывает исключение протокола на уровне соединения и немедленное закрытие соединения брокером. При использовании ручных подтверждений вызов `basicAck` с флагом `multiple=true` из потока, отличного от получающего, небезопасен и приводит к двойному подтверждению.

**Рекомендованная модель:** один канал на поток (`Channel` per thread). Для высоконагруженных приложений — переход к пулу каналов (channel pool) при появлении проблем с памятью. Рекомендуется использовать долгоживущие каналы, а не создавать и закрывать их при каждой операции.

#### Ответ от Claude

**Connection vs Channel**

| Характеристика | Connection | Channel |
|---|---|---|
| Природа | Реальное TCP-соединение | Логический виртуальный канал внутри TCP |
| Ресурсы | Тяжёлый (сокет, память, firewall rules) | Лёгкий (небольшая память на клиенте и сервере) |
| Число | Минимально (1 на приложение или пул) | Много на одно соединение |
| Потокобезопасность | Безопасен для совместного использования | Не потокобезопасен |

**Почему Channel не потокобезопасен**

AMQP 0-9-1 передаёт данные в виде бинарных фреймов. Когда два потока одновременно пишут в один канал, их фреймы перемежаются на уровне TCP-стрима. Брокер не может корректно разобрать такой поток и закрывает соединение с ошибкой протокола. Дополнительные проблемы:

1. **Двойные подтверждения:** вызов `basicAck(deliveryTag, multiple=true)` из другого потока подтверждает диапазон тегов, включая теги чужого потока.
2. **Publisher Confirms:** состояние sequence number корраптится при конкурентном доступе.

**Рекомендованная модель для многопоточного Java-приложения**

1. **Одно (или несколько) долгоживущих Connection** на приложение — создаётся при старте, не пересоздаётся при каждой операции.
2. **Один Channel на поток** — самая простая и безопасная стратегия. Канал создаётся при старте потока, используется только в нём.
3. **Channel pool** (например, через Spring AMQP `CachingConnectionFactory`) — при высокой конкурентности и большом числе потоков. Пул управляет набором каналов с синхронизированным доступом; конкретный канал в каждый момент используется только одним потоком.
4. **Исключение:** один поток-потребитель и один поток-издатель могут безопасно использовать один канал, если публикация и подтверждения не смешиваются — но это хрупкая конструкция, лучше избегать.

Практическое правило: никогда не вызывать `Channel#basicPublish` одновременно из нескольких потоков без явной синхронизации; если синхронизация нужна — лучше завести отдельный канал.

### Вопрос 45. Как в RabbitMQ работает TTL на уровне сообщения (свойство expiration) и на уровне очереди (x-message-ttl)? Что происходит, когда заданы оба значения? Чем поведение при истечении TTL у classic queue отличается от quorum queue с точки зрения dead-lettering?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Time-To-Live and Expiration | RabbitMQ](https://www.rabbitmq.com/docs/ttl)

**Per-message TTL (свойство expiration):** издатель устанавливает свойство `expiration` при публикации сообщения. Значение — строковое представление числа миллисекунд. Брокер использует это значение для вычисления времени жизни конкретного сообщения.

**Per-queue TTL (x-message-ttl):** TTL задаётся для всей очереди через аргумент `x-message-ttl` при объявлении очереди или через политику (`message-ttl`). Применяется ко всем сообщениям в очереди равномерно.

**Когда заданы оба значения:** применяется меньшее из двух значений — то есть сообщение истекает раньше.

**Поведение при истечении TTL:**
- **Quorum queues:** мёртвые письма (dead-lettering) для истёкших сообщений происходят только в момент, когда сообщение достигает головы очереди (head of the queue).
- **Classic queues:** мёртвые письма применяются в нескольких случаях — в том числе когда сообщение достигает головы очереди, а также при уведомлении очереди об изменении политики.

#### Ответ от Claude

**TTL на уровне сообщения (свойство expiration)**

Устанавливается при публикации через AMQP-свойство `expiration` (строка с числом миллисекунд). Позволяет задавать индивидуальное время жизни для каждого сообщения. Брокер вычитает прошедшее время с момента попадания сообщения в очередь и удаляет его при достижении порога.

**TTL на уровне очереди (x-message-ttl)**

Задаётся аргументом `x-message-ttl` (тип long, миллисекунды) при объявлении очереди или политикой `message-ttl`. Применяется ко всем сообщениям очереди единообразно. Может быть изменён через политику без пересоздания очереди.

**Когда заданы оба TTL**

Применяется меньшее значение из двух. Это гарантирует, что более строгое ограничение всегда побеждает.

**Различия в dead-lettering между classic queue и quorum queue**

| Аспект | Classic Queue | Quorum Queue |
|---|---|---|
| Момент dead-lettering при TTL | При достижении головы очереди; также при изменении политики, затрагивающей очередь | Только при достижении головы очереди |
| Гарантия доставки в DLX | At-most-once: сообщение может быть потеряно, если целевая очередь недоступна | At-least-once (опционально): внутренний потребитель подтверждает dead-letter только после получения publisher confirm от целевой очереди |
| Активация at-least-once | Не поддерживается | Требует: `dead-letter-strategy=at-least-once`, `overflow=reject-publish`, включённый feature flag `stream_queue` |
| Поведение при expiration до попадания в голову | Сообщение может быть удалено раньше (при изменении политики) | Сообщение остаётся в очереди до достижения головы, только там проверяется TTL |

**Ключевое практическое отличие:** в classic queue сообщение с истёкшим TTL, находящееся не у головы очереди, всё равно занимает место и не будет удалено до достижения головы (если только не меняется политика). В quorum queue поведение аналогично — сообщение удаляется/dead-letterится строго в момент извлечения из головы. Таким образом, при большом числе сообщений в очереди фактический момент dead-lettering может значительно отставать от истечения TTL в обоих типах очередей, однако quorum queue исключает промежуточное срабатывание по изменению политики.

Расхождение между источниками: официальная документация RabbitMQ по TTL упоминает, что classic queue dead-letterит при изменении политики как дополнительный триггер; страница по quorum queues этот сценарий не описывает, указывая только на head-of-queue момент.

### Вопрос 46. Чем quorum queue отличается от classic queue и stream queue в RabbitMQ? В каких сценариях нужно использовать каждый тип, и почему classic queue deprecated для новых production-развёртываний начиная с RabbitMQ 3.12?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Quorum Queues | RabbitMQ Official Documentation](https://www.rabbitmq.com/docs/quorum-queues)

Quorum queue — это современный тип очереди в RabbitMQ, основанный на алгоритме консенсуса Raft. Это durable, replicated FIFO-очередь, которая является стандартным выбором для систем, требующих высокой доступности и безопасности данных.

Алгоритм Raft обеспечивает согласованность между большинством членов очереди: (N/2)+1, где N — общее число участников. Quorum queue требует минимум 3 узла кластера для обеспечения fault tolerance (кластер переживает потерю 1 из 3 узлов).

Ключевые отличия quorum от classic: quorum всегда реплицируется, всегда durable, поддерживает at-least-once dead-lettering, poison message handling (delivery-limit), consumer timeouts и delayed retry. Classic queue не имеет ни одной из этих возможностей. Начиная с RabbitMQ 4.0 classic queue является явно non-replicated — поддержка зеркалированных classic queues (mirrored classic queues) полностью удалена.

Stream — принципиально иной тип: append-only лог с non-destructive read semantics. Сообщения не удаляются при чтении, потребители могут перечитывать данные с любого offset. Stream подходит для fan-out (тысячи потребителей читают одни данные), replay-сценариев, высокого throughput. Quorum queue разрушает сообщение после ack (как classic), тогда как stream его сохраняет.

Рекомендации по выбору:
- Classic: временные очереди с низкими требованиями к надёжности, сценарии где потеря данных допустима, максимальная производительность без репликации.
- Quorum: критичные бизнес-очереди (заказы, платежи, события), где требуется at-least-once и высокая доступность при сбоях узлов.
- Stream: высокий throughput, fan-out к множеству независимых потребителей, replay событий, IoT-данные, аналитические пайплайны.

#### Ответ от Claude

**Три типа очередей RabbitMQ и их архитектурные отличия**

RabbitMQ с версии 3.x поддерживает три принципиально разных типа очередей.

**Classic Queue** — исходный тип с момента создания RabbitMQ. Хранит сообщения в индексированном хранилище на диске (или в памяти), работает как FIFO-буфер. Не реплицирует данные между узлами — при падении узла очередь и все сообщения в ней теряются. До версии 4.0 существовал механизм «зеркалирования» (mirrored classic queues), но он имел критичный недостаток: синхронизация при добавлении нового зеркала блокирует всю очередь. Начиная с RabbitMQ 3.12 команда официально объявила classic queue deprecated для production как основного хранилища критичных данных, а в 4.0 механизм зеркалирования полностью удалён. Classic queue в 4.0 — явно non-replicated тип.

**Quorum Queue** — построен на алгоритме консенсуса Raft. Каждый узел кластера, входящий в quorum данной очереди, хранит Write-Ahead Log (WAL) с репликами сообщений. Сообщение считается успешно принятым только после подтверждения кворумом (N/2+1) участников. Это обеспечивает строгие гарантии durability: если лидер падает, новый лидер выбирается из реплик с полными данными. Quorum queue всегда durable (non-durable quorum queue создать нельзя). Дополнительно поддерживает: at-least-once dead-lettering (DLX-события сами подтверждаются кворумом), delivery-limit для poison message handling, consumer timeout, приоритеты до 32 уровней.

**Stream** — append-only сегментированный лог, аналог топика в Kafka. Принципиальное отличие от очередей: сообщения не удаляются при получении — потребитель указывает offset, с которого читать. Это позволяет тысячам независимых потребителей читать одни и те же сообщения (fan-out без копирования). Stream также реплицируется, но семантика потребления кардинально иная. Ограничение: нет семантики «обработал и удали».

**Сценарии использования:**

| Тип | Когда использовать |
|---|---|
| Classic | Временные очереди, dev/test окружения, high-speed low-criticality workloads, где потеря нескольких сообщений при отказе узла допустима |
| Quorum | Всё, что требует at-least-once delivery с гарантией durability: финансовые транзакции, заказы, уведомления, интеграционные события между сервисами |
| Stream | Fan-out к множеству потребителей, replay событий, IoT-ingestion, аналитические пайплайны, audit log |

**Почему classic deprecated с 3.12:**

1. Зеркалирование classic queues (единственный механизм HA) имело фундаментальную проблему: синхронизация нового зеркала полностью блокирует очередь. Quorum queues решают эту проблему через Raft.
2. По результатам нагрузочного тестирования команды RabbitMQ, quorum queues показывают более высокий и стабильный throughput по сравнению с mirrored classic queues при схожих гарантиях надёжности.
3. Поддержка двух конкурирующих механизмов HA усложняла кодовую базу. Deprecation classic mirroring — шаг к упрощению архитектуры.
4. Кластер из 3+ узлов (обязательное условие для quorum queues) стал де-факто стандартом для любого production RabbitMQ. Это убрало барьер для миграции.

В RabbitMQ 4.0 при старте узла автоматически выполняется миграция V1 classic queue storage в V2 (более эффективный формат хранения). Mirrored classic queues в 4.0 создать невозможно.

### Вопрос 47. Чем KRaft mode (KIP-500) отличается от традиционного режима Kafka с ZooKeeper? Как хранятся метаданные кластера в KRaft, что такое controller quorum, и почему начиная с Kafka 4.0 ZooKeeper-режим полностью удалён?

**Категория:** Брокеры сообщений · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [KRaft — Apache Kafka Without ZooKeeper | Confluent Developer](https://developer.confluent.io/learn/kraft/)

KRaft — реализация алгоритма консенсуса Raft, встроенного непосредственно в Apache Kafka. Технология введена через KIP-500 и объявлена готовой к production в версии 3.3.1.

В ZooKeeper-режиме метаданные кластера (конфигурации топиков, назначения партиций, ACL, список брокеров) хранились во внешней системе — Apache ZooKeeper. Kafka-контроллер (один из брокеров, выбираемый через ZooKeeper) читал метаданные из ZooKeeper и рассылал их другим брокерам. Это создавало двойную зависимость: ZooKeeper определял лидера контроллера, а контроллер транслировал метаданные.

В KRaft-режиме вся система консолидируется в самой Kafka. Метаданные хранятся во внутреннем топике `__cluster_metadata` в виде событийного журнала (event log). Каждое изменение — создание топика, переназначение партиций, обновление конфигурации — записывается как событие в этот журнал. Брокеры подписываются на `__cluster_metadata` и формируют актуальное состояние кластера локально.

Controller quorum — группа специализированных процессов (или брокеров в combined mode), реализующих Raft-консенсус для управления журналом метаданных. Один из контроллеров выбирается active leader и обрабатывает все операции записи. При отказе лидера кворум выбирает нового в течение миллисекунд, причём новый лидер уже имеет все закоммиченные записи метаданных в памяти — никакой синхронизации не требуется.

Начиная с Kafka 4.0 (релиз 2024) ZooKeeper-режим полностью удалён из кодовой базы. Все новые кластеры обязаны использовать KRaft.

#### Ответ от Claude

**Архитектурное сравнение: ZooKeeper-режим vs KRaft**

**ZooKeeper-режим (до Kafka 3.x)**

В классической архитектуре Kafka использовала Apache ZooKeeper как внешнюю систему координации. ZooKeeper хранил:
- список активных брокеров (узлы `/brokers/ids/`)
- конфигурации топиков и партиций
- ACL и квоты
- текущего controller-лидера

Цепочка распространения метаданных: изменение в ZooKeeper → Kafka Controller получает watch-уведомление → Controller генерирует LeaderAndIsr/UpdateMetadata RPC → все брокеры. Это создавало задержки и точки отказа. При смене controller-лидера новый контроллер читал всё состояние из ZooKeeper — при большом числе партиций это занимало десятки секунд. Практический предел — около 200 000 партиций на кластер из-за узкого горлышка ZooKeeper.

**KRaft-режим (KIP-500)**

KRaft встраивает алгоритм консенсуса Raft непосредственно в Kafka. Ключевые элементы архитектуры:

*Хранение метаданных.* Метаданные записываются в специальный внутренний топик `__cluster_metadata` (партиция 0) в формате событийного журнала. Каждое изменение — дельта-событие (RecordBatch). Журнал периодически компактируется снимками (snapshots), чтобы не расти бесконечно. Брокеры читают `__cluster_metadata` как обычные потребители и строят локальную копию состояния кластера в памяти.

*Controller Quorum.* Отдельная группа процессов (рекомендуется 3 или 5 controller-нод) реализует Raft-кворум. В isolated mode контроллеры — отдельные JVM-процессы без партиций данных. В combined mode один и тот же процесс является и брокером, и контроллером (удобно для dev/test). Raft-лидер (active controller) принимает все изменения метаданных, реплицирует их на follower-контроллеры и подтверждает клиенту после получения ответа от кворума. Если лидер падает, Raft-выборы завершаются за миллисекунды — новый лидер уже имеет актуальный журнал в памяти.

**Ключевые преимущества KRaft:**

| Аспект | ZooKeeper | KRaft |
|---|---|---|
| Время failover контроллера | Секунды — десятки секунд | Миллисекунды |
| Макс. число партиций | ~200 000 | Миллионы (заявлено) |
| Число процессов для запуска кластера | Kafka + ZooKeeper (отдельный ансамбль) | Только Kafka |
| Модель безопасности | Раздельная: SASL для ZK и для Kafka | Единая |
| Распространение метаданных | Push через controller RPC | Pull через топик __cluster_metadata |

**Почему ZooKeeper удалён в Kafka 4.0:**

1. ZooKeeper как внешняя зависимость требовал отдельной операционной экспертизы, мониторинга, отдельного ансамбля нод (3–5 отдельных процессов), отдельных backups.
2. Двуслойное хранение метаданных (ZooKeeper + кеш в контроллере) создавало возможность рассинхронизации.
3. ZooKeeper-сессии могли истечь при пиковых нагрузках, вызывая каскадные проблемы.
4. Масштабирование до миллионов партиций было практически невозможным в ZK-режиме из-за ограничений хранилища ZooKeeper.

KRaft объявлен production-ready в Kafka 3.3.1 (2022). В Kafka 3.5 ZooKeeper помечен как deprecated. В Kafka 4.0 (2024) код ZooKeeper-режима полностью удалён из основного репозитория. Перед обновлением до 4.0 требуется запустить online-миграцию с ZK на KRaft (KIP-866).

### Вопрос 48. Объясни политику хранения данных (retention) в Kafka: какие два типа retention существуют (time-based и size-based), как организованы log segments и когда именно Kafka удаляет данные — на уровне записей или сегментов? Что такое active segment и почему он не удаляется даже по истечению retention-периода?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Deep dive into Apache Kafka storage internals: segments, rolling and retention | Strimzi Blog](https://strimzi.io/blog/2021/12/17/kafka-segment-retention/)

Каждая партиция Kafka разделена на сегменты — физические файлы на диске. Партиция имеет ровно один активный сегмент, открытый для записи. Неактивные сегменты — read-only и используются потребителями, читающими старые записи. Когда активный сегмент заполняется, он «откатывается» (rolled): закрывается и открывается новый.

Kafka поддерживает два типа retention:

**Time-based**: Параметры `log.retention.ms`, `log.retention.minutes` или `log.retention.hours` (по умолчанию 7 дней). Сегмент помечается на удаление, если его последняя запись старше retention-периода.

**Size-based**: Параметр `log.retention.bytes` — максимальный объём партиции в байтах. Если суммарный размер всех сегментов превышает лимит, самые старые сегменты удаляются.

Удаление происходит на уровне целых сегментов, а не отдельных записей. Активный сегмент никогда не удаляется — только закрытые. Фоновый поток брокера проверяет закрытые сегменты каждые `log.retention.check.interval.ms` (по умолчанию 5 минут). Сегмент, подлежащий удалению, переименовывается с расширением `.deleted`, затем физически удаляется с задержкой `log.segment.delete.delay.ms` (по умолчанию 1 минута).

Практическое следствие: фактическое время хранения первой записи в сегменте может быть значительно больше, чем `retention.ms`. Например, при retention 10 минут: запись попадает в новый сегмент, сегмент заполняется ещё 8 минут, затем ещё 5 минут до следующей проверки, ещё 1 минута на физическое удаление — итого 24 минуты. `retention.ms` — это нижняя граница, а не точное время хранения.

#### Ответ от Claude

**Архитектура хранения данных в Kafka: сегменты и политики retention**

**Структура partition log**

Каждая топик-партиция в Kafka хранится как последовательность сегментов — пар файлов на диске:

- `{base_offset}.log` — бинарный файл с записями (RecordBatch)
- `{base_offset}.index` — offset-индекс (смещение → позиция в .log)
- `{base_offset}.timeindex` — временной индекс (timestamp → offset)

`base_offset` в имени файла — это смещение первой записи в сегменте. Например, `00000000000000000000.log` — первый сегмент, `00000000000000001234.log` — сегмент, начинающийся с offset 1234.

**Active segment**

В каждый момент времени ровно один сегмент открыт для записи — active segment. Все новые сообщения пишутся в него. Остальные сегменты — read-only (frozen/closed).

Активный сегмент «откатывается» (segment roll) при выполнении любого из условий:
- размер достиг `log.segment.bytes` (по умолчанию 1 ГБ)
- возраст старейшей записи в сегменте превысил `log.roll.ms` / `log.roll.hours` (по умолчанию 7 дней)
- offset-индекс заполнен до `log.index.size.max.bytes` (по умолчанию 10 МБ)

После roll активный сегмент закрывается, становится read-only, и открывается новый пустой сегмент.

**Два типа retention**

*Time-based retention* — управляется параметрами `log.retention.ms` (приоритетный), `log.retention.minutes`, `log.retention.hours`. По умолчанию: 7 дней (`log.retention.hours=168`). Брокер сравнивает timestamp последней записи в закрытом сегменте с текущим временем. Если разница превышает retention — сегмент удаляется.

*Size-based retention* — управляется `log.retention.bytes` (по умолчанию -1, то есть отключён). Лимит задаётся на партицию, не на топик. Если суммарный размер всех сегментов партиции превышает лимит, самые старые закрытые сегменты удаляются по одному до тех пор, пока размер не войдёт в норму.

Можно использовать оба типа одновременно. Срабатывает то условие, которое выполняется первым.

**Почему удаляются сегменты, а не отдельные записи**

Kafka спроектирована как append-only лог. Удаление отдельных записей из середины .log-файла потребовало бы переписи файла — дорогостоящей операции случайной записи. Вместо этого сегменты — атомарные единицы удаления: файл просто unlink с диска. Это даёт O(1) операцию удаления вне зависимости от числа записей в сегменте, минимальную нагрузку на диск при cleanup, и возможность использовать sendfile/zero-copy при чтении неизменяемых сегментов.

**Почему active segment не удаляется**

Активный сегмент открыт для записи, в него идут новые сообщения. Удаление открытого файла нарушило бы целостность данных и поток записи. Это принципиальное архитектурное ограничение: cleanup-поток брокера работает только с closed segments. Следствие: если retention-период истёк, но все данные ещё в активном сегменте (например, медленный топик с редкими сообщениями), данные не будут удалены, пока сегмент не закроется.

**Процесс удаления (детально):**

1. Фоновый поток `kafka-log-retention` проверяет закрытые сегменты каждые `log.retention.check.interval.ms` (по умолчанию 5 минут).
2. Сегмент, нарушающий политику, переименовывается: `*.log` → `*.log.deleted` (аналогично `.index.deleted`, `.timeindex.deleted`).
3. Файлы физически удаляются через `log.segment.delete.delay.ms` (по умолчанию 1 минута) — задержка нужна, чтобы потребители, уже открывшие файл, успели завершить чтение.

**Важный нюанс:** `retention.ms` — это нижняя граница (гарантия неудаления «свежих» данных), но не точная верхняя граница времени хранения. Реальное время хранения старейшей записи в сегменте равно: `retention.ms + время заполнения сегмента + интервал проверки + задержка удаления`. При медленных топиках и крупных сегментах это может быть значительно больше настроенного retention.

### Вопрос 49. Как выполнить reassignment партиций между брокерами в Kafka (например, при добавлении нового брокера в кластер)? Как работает инструмент kafka-reassign-partitions.sh, что происходит во время переноса партиции на уровне репликации, и как throttling защищает production-трафик в этот период?

**Категория:** Брокеры сообщений · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Basic Kafka Operations | Apache Kafka](https://kafka.apache.org/41/operations/basic-kafka-operations/)

Инструмент `kafka-reassign-partitions.sh` позволяет переместить партиции с текущего набора брокеров на новые. Он равномерно распределяет все партиции указанных топиков по новому набору брокеров. Процесс инициируется вручную, но выполняется полностью автоматически: Kafka добавляет новую реплику на целевой машине как follower, позволяет ей реплицироваться, отмечает её как in-sync после полной синхронизации, а затем удаляет одну из существующих реплик на оригинальном сервере.

Инструмент работает в трёх режимах:

**--generate**: принимает список топиков и список брокеров, генерирует кандидат-план переназначения в виде JSON без каких-либо изменений в кластере.

**--execute**: запускает фактическое перемещение партиций на основе предоставленного плана (JSON-файл).

**--verify**: проверяет статус переназначения всех партиций, перечисленных в последней операции --execute. Если переназначение завершено, опция --verify также снимает throttle.

Kafka позволяет применять throttle к трафику репликации, устанавливая верхнюю границу пропускной способности при перемещении реплик между машинами. Это полезно при ребалансировке кластера, начальной загрузке нового брокера или добавлении/удалении брокеров, так как ограничивает влияние этих data-intensive операций на пользователей.

Throttle задаётся флагом `--throttle` со значением в байтах/секунду:
```bash
bin/kafka-reassign-partitions.sh \
  --bootstrap-server localhost:9092 \
  --execute \
  --reassignment-json-file bigger-cluster.json \
  --throttle 700000000
```

Механизм throttle работает, измеряя скорость получения и передачи данных для партиций, перечисленных в списках `replication.throttled.replicas` на каждом брокере, и сравнивая их с конфигурацией `replication.throttled.rate`. После завершения ребалансировки throttle снимается через опцию `--verify`. Важно своевременно снять throttle, так как его сохранение может привести к ограничению обычного replication-трафика.

#### Ответ от Claude

**Общий сценарий использования**

Типичный случай: в кластер добавлен новый брокер, но он не получает данных автоматически — Kafka не перемещает существующие партиции. Для ребалансировки нужно вручную запустить `kafka-reassign-partitions.sh`.

**Трёхэтапный рабочий процесс**

1. **Generate** — генерация плана без изменений в кластере:
```bash
bin/kafka-reassign-partitions.sh \
  --bootstrap-server localhost:9092 \
  --generate \
  --topics-to-move-json-file topics.json \
  --broker-list "1,2,3,4"
```
Инструмент выводит два JSON: текущее расположение партиций (для отката) и предлагаемый план. Оба нужно сохранить.

2. **Execute** — применение плана с ограничением пропускной способности:
```bash
bin/kafka-reassign-partitions.sh \
  --bootstrap-server localhost:9092 \
  --execute \
  --reassignment-json-file plan.json \
  --throttle 50000000
```

3. **Verify** — проверка статуса и автоматическое снятие throttle после завершения:
```bash
bin/kafka-reassign-partitions.sh \
  --bootstrap-server localhost:9092 \
  --verify \
  --reassignment-json-file plan.json
```

**Что происходит на уровне репликации**

Процесс переноса партиции на новый брокер состоит из строго определённых шагов:

1. **Добавление follower-реплики**: новый брокер назначается follower для переносимой партиции. Он начинает реплицировать все сегменты лога от текущего leader.

2. **Фаза catch-up**: новая реплика выгружает исторические данные с leader-брокера. На этом этапе leader продолжает обслуживать чтения и записи от клиентов без прерываний.

3. **Вхождение в ISR**: когда новая реплика догоняет leader (её отставание по записям не превышает `replica.lag.time.max.ms`), она добавляется в In-Sync Replica set (ISR). Только члены ISR участвуют в подтверждении записей (при `acks=all`).

4. **Удаление старой реплики**: после того как новая реплика в ISR, одна из старых реплик удаляется с исходного брокера. Если удаляемая реплика была leader, происходит автоматический leader election среди оставшихся членов ISR.

Весь процесс выполняется без останова — запись и чтение данных продолжаются.

**Механизм throttling**

Throttling реализован через два уровня конфигурации:

**На уровне брокеров** (ограничение пропускной способности):
- `leader.replication.throttled.rate` — максимальное количество байт/с, которое leader-реплика отдаёт для репликации
- `follower.replication.throttled.rate` — максимальное количество байт/с, которое follower-реплика принимает

**На уровне топиков** (какие реплики throttle применяется):
- `leader.replication.throttled.replicas` — список пар `[partitionId]:[brokerId]` на стороне leader
- `follower.replication.throttled.replicas` — список пар `[partitionId]:[brokerId]` на стороне follower

По умолчанию при запуске с флагом `--throttle` инструмент автоматически применяет:
- leader-throttle — ко всем репликам, существовавшим до начала ребалансировки (любая из них может быть leader в процессе)
- follower-throttle — только к целевым брокерам-получателям

**Почему throttling важен для production**: перенос партиций — disk- и network-intensive операция. Без ограничения она может «забить» сетевой канал брокера, что вызовет отставание обычных follower-реплик, их выпадение из ISR, и в итоге повысит задержку записи клиентов (acks=-1/all) вплоть до `NotEnoughReplicasException`.

**Важные предостережения**:
- Слишком низкое значение throttle может замедлить репликацию настолько, что новые реплики никогда не догонят leader (особенно при высоком темпе записи). Мониторить метрику `kafka.server:type=FetcherLagMetrics`.
- После завершения необходимо снять throttle через `--verify` или вручную, иначе обычная репликация будет постоянно ограничена.

### Вопрос 50. Объясни роль Transaction Coordinator в Kafka. Как определяется, какой брокер является координатором для конкретного transactional.id? Что такое zombie fencing, как epoch защищает от одновременной работы двух экземпляров transactional producer, и какова роль топика __transaction_state?

**Категория:** Брокеры сообщений · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Transactions in Apache Kafka | Confluent](https://www.confluent.io/blog/transactions-apache-kafka/)

Transaction Coordinator — это модуль, работающий внутри каждого брокера Kafka. Каждый `transactional.id` через простую хеш-функцию отображается на конкретную партицию transaction log, что означает: ровно один координатор «владеет» каждым `transactional.id`. Это устраняет необходимость в распределённом консенсусе по состоянию транзакций.

Когда producer вызывает `initTransactions()`, координатор:
- закрывает все незавершённые транзакции с данным `transactional.id`
- увеличивает связанное значение epoch

Любые producers с тем же `transactional.id`, но с устаревшим epoch становятся «зомби» и «отгораживаются» — их будущие транзакционные записи отклоняются.

Внутренний топик `__transaction_state` хранит метаданные транзакций, но не сами сообщения. Координатор держит состояние транзакций в памяти, одновременно записывая обновления в этот реплицированный лог для обеспечения durability. Состояния транзакций включают: `Ongoing`, `PrepareCommit`, `PrepareAbort`, `CompleteCommit`, `CompleteAbort`.

**Жизненный цикл транзакции:**

1. **Инициализация**: `producer.initTransactions()` регистрирует `transactional.id`, фенсирует зомби.
2. **Начало транзакции**: `producer.beginTransaction()`.
3. **Регистрация партиций**: перед отправкой в новую партицию producer регистрирует её у координатора.
4. **Коммит**: `commitTransaction()` запускает двухфазный коммит:
   - Фаза 1: координатор записывает состояние `prepare_commit` в transaction log.
   - Фаза 2: координатор записывает transaction commit markers во все участвующие партиции.
5. **Потребители** читают сообщения только после появления маркеров (при `isolation.level=read_committed`).

#### Ответ от Claude

**Роль Transaction Coordinator**

Transaction Coordinator — это компонент, выполняющийся на каждом брокере Kafka. Он не является отдельным процессом, а встроен в брокер. Основные функции:
- выдача Producer ID (PID) и epoch новым transactional producer-ам
- ведение журнала состояний транзакций в `__transaction_state`
- координация двухфазного коммита (2PC) между партициями
- обработка таймаутов транзакций и восстановление после сбоев

**Как определяется координатор для конкретного transactional.id**

Алгоритм детерминированный и основан на хешировании:

```
partition_id = abs(transactional.id.hashCode()) % num_partitions(__transaction_state)
```

Брокер, являющийся leader для найденной партиции `__transaction_state`, и становится Transaction Coordinator для данного `transactional.id`. По умолчанию `__transaction_state` имеет 50 партиций (`transaction.state.log.num.partitions`).

Это означает:
- один `transactional.id` всегда маршрутизируется к одному и тому же координатору (до тех пор, пока не произойдёт leader election по этой партиции)
- каждый брокер является координатором для некоторого подмножества всех `transactional.id` в кластере
- distributed consensus не требуется: нет необходимости спрашивать несколько брокеров

Когда producer стартует, он отправляет запрос `FindCoordinator` любому брокеру, указывая свой `transactional.id`. Брокер вычисляет нужную партицию, возвращает адрес её leader — он и будет координатором.

**Топик __transaction_state**

`__transaction_state` — внутренний compacted топик Kafka. В нём хранятся не сами сообщения транзакции, а только метаданные:
- `transactional.id` и связанный с ним PID + epoch
- список партиций, участвующих в текущей транзакции
- текущее состояние транзакции

Состояния (конечный автомат):
```
Empty → Ongoing → PrepareCommit → CompleteCommit
                → PrepareAbort  → CompleteAbort
```

Запись в лог происходит перед каждым изменением состояния (write-ahead log), что обеспечивает восстановление транзакции при падении координатора: новый leader для партиции `__transaction_state` прочитает лог и продолжит незавершённую транзакцию (commit или abort, в зависимости от состояния).

**Zombie fencing и механизм epoch**

Проблема: producer A обработал сообщение, завис (network partition, GC pause, медленное выполнение), был перезапущен как producer B с тем же `transactional.id`. Producer A «ожил» и пытается завершить свою транзакцию — это «зомби».

Решение через epoch:

1. При `initTransactions()` producer B отправляет запрос `InitProducerId` координатору.
2. Координатор находит в `__transaction_state` запись для этого `transactional.id`.
3. Координатор **инкрементирует epoch**: например, был epoch=3, стал epoch=4.
4. Producer B получает тот же PID, но epoch=4.
5. Все брокеры-лидеры партиций теперь ожидают записи только с epoch >= 4.
6. Producer A пытается записать с epoch=3 — получает `ProducerFencedException`. Транзакция A отклонена.

Ключевой инвариант: в любой момент времени только один экземпляр producer с данным `transactional.id` имеет актуальный epoch и может совершать транзакции. Kafka гарантирует это без каких-либо распределённых блокировок.

**Примечание о Kafka 4.0**: в новом протоколе (KIP-890, клиенты Kafka 4.0) epoch инкрементируется при каждой транзакции (не только при инициализации producer), что дополнительно защищает от дублирования сообщений в редком краевом случае на границе двух транзакций подряд.

**Полный жизненный цикл транзакции (краткая схема)**

```
Producer                      Coordinator               Брокеры данных
   |                               |                          |
   |-- InitProducerId ------------>|                          |
   |<-- PID + epoch ---------------|                          |
   |                               |                          |
   |-- BeginTransaction (local) -->|                          |
   |                               |                          |
   |-- AddPartitionsToTxn -------->|                          |
   |   (регистрация партиций)      |-- write Ongoing -------->|
   |                               |                          |
   |-- ProduceRequest (data) ---------------------->|         |
   |                               |                          |
   |-- EndTransaction(commit) ---->|                          |
   |                          write PrepareCommit             |
   |                               |-- WriteTxnMarkers ------>|
   |                               |   (COMMIT marker)        |
   |                          write CompleteCommit            |
   |<-- success ------------------|                          |
```

Consumer с `isolation.level=read_committed` видит данные только после получения COMMIT-маркера.

### Вопрос 51. В топике Kafka 3 партиции, а в одной consumer group — 4 консюмера. Что произойдёт и почему? Как это связано с гарантией порядка?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Habr — Java-собеседование 2025: разбор вопросов](https://habr.com/ru/articles/1043906/)

В статье: один консюмер будет простаивать. Партиция в пределах группы обрабатывается только одним консюмером, поэтому при числе консюмеров больше числа партиций «лишние» консюмеры не получают партиций.

#### Ответ от Claude

Каждая партиция в рамках одной consumer group назначается **ровно одному** консюмеру, при этом один консюмер может читать несколько партиций. Партиций 3, консюмеров 4 → три консюмера получают по партиции, **четвёртый простаивает** (idle). Он не бесполезен: при падении одного из активных консюмеров произойдёт rebalance и резервный подхватит освободившуюся партицию (горячий резерв для отказоустойчивости).

Связь с порядком: Kafka гарантирует порядок сообщений **только внутри партиции**. Поскольку партицию читает один консюмер, порядок по партиции сохраняется. Чтобы реально масштабировать параллелизм потребления, нужно увеличивать число партиций, а не консюмеров. Подробнее — [Apache Kafka — Consumers](https://kafka.apache.org/documentation/#intro_consumers).

### Вопрос 52. Чем KStream отличается от KTable в Kafka Streams? Как работает дуализм потока и таблицы (stream-table duality) и что такое changelog-топик?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Core Concepts | Apache Kafka Streams](https://kafka.apache.org/42/streams/core-concepts/)

KStream и KTable — две фундаментальные абстракции Kafka Streams, описывающие данные с разных точек зрения.

**KStream** — абстракция потока записей, где каждая запись интерпретируется как независимое событие (INSERT). Запись не заменяет предыдущую запись с тем же ключом — они сосуществуют в потоке.

**KTable** — абстракция changelog-потока от таблицы с первичным ключом. Каждая новая запись с уже существующим ключом интерпретируется как UPDATE: последнее значение по ключу — актуальное. Запись с null-значением означает DELETE.

**Дуализм потока и таблицы (stream-table duality):** «поток можно рассматривать как таблицу, а таблицу — как поток». Поток есть changelog таблицы: восстановив его с начала, получим текущее состояние таблицы. Таблица есть «срез» потока на текущий момент.

Механизм восстановления: каждый стейт-стор KTable подкреплён changelog-топиком — внутренним топиком Kafka, куда дублируются все изменения состояния. При рестарте или перераспределении задачи Kafka Streams реплицирует стейт-стор заново, реплея changelog-топик.

#### Ответ от Claude

**KStream: поток независимых событий**

KStream описывает непрерывную последовательность событий. Каждая запись — самостоятельный факт («добавление в append-only лог»): повторный ключ не означает обновления — оба события присутствуют в потоке. Типичный сценарий: события кликов, транзакций, логов.

**KTable: материализованное представление**

KTable хранит актуальное состояние: только последнее значение по каждому ключу. Запись с ключом K и значением V означает «текущее значение K равно V». Запись с null-значением означает удаление ключа. Типичный сценарий: текущий баланс счёта, последняя позиция пользователя, агрегированные метрики.

**Практическое отличие:**

| Характеристика | KStream | KTable |
|---|---|---|
| Интерпретация записи | INSERT (событие) | UPSERT (обновление состояния) |
| null-значение | Допустимо как значение | DELETE (удаление ключа) |
| Хранение состояния | Нет (stateless) | Да (state store) |
| Типичные операции | map, filter, flatMap | groupBy, aggregate, join |

**Changelog-топик и state store**

Каждый KTable с персистентным state store сопровождается внутренним changelog-топиком (имя вида `{application-id}-{store-name}-changelog`). Все изменения state store записываются в этот топик. При падении или переназначении задачи Kafka Streams восстанавливает state store путём реплея changelog-топика — это обеспечивает fault tolerance без внешнего хранилища.

**GlobalKTable** — особый вариант: реплицирует весь топик на каждый экземпляр приложения (не разбивается на задачи), что позволяет выполнять join любой партиции KStream с полным содержимым GlobalKTable без перераспределения данных.

### Вопрос 53. Какие типы временных окон поддерживает Kafka Streams? Чем tumbling window отличается от hopping window и session window? Что такое grace period и зачем он нужен?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Windowing in Kafka Streams — Confluent Blog](https://www.confluent.io/blog/windowing-in-kafka-streams/)

Kafka Streams поддерживает четыре типа временных окон для агрегации данных.

**Tumbling window** — фиксированные не перекрывающиеся окна. Каждое событие попадает ровно в одно окно. Это частный случай hopping window, где размер шага равен размеру окна. Пример: агрегация транзакций за каждые 55 секунд.

**Hopping window** — фиксированные перекрывающиеся окна. Задаются двумя параметрами: размер окна и шаг продвижения. Когда шаг меньше размера — окна перекрываются, одно событие может попасть в несколько окон. Пример: скользящее среднее кликов за 2-минутные окна с шагом 10 секунд.

**Sliding window** — окно активируется при наступлении события и охватывает события в заданном временном диапазоне относительно друг друга. Используется для joins: объединить связанные события, произошедшие в течение заданного периода.

**Session window** — управляемое событиями окно без фиксированного размера. Определяется «gap бездействия» (inactivity gap): если за этот период не поступило новых событий с тем же ключом — сессия закрывается. Если события продолжают поступать — окно расширяется. Пример: сессии просмотра видео пользователем.

**Grace period** — период после формального окончания окна, в течение которого события с опоздавшими timestamps ещё принимаются в это окно. После истечения grace period опоздавшие события игнорируются. После KIP-633 в Kafka Streams нет дефолтного grace period (в отличие от ksqlDB, где дефолт 24 часа).

#### Ответ от Claude

**Четыре типа окон в Kafka Streams**

Все типы окон применяются к операциям агрегации (`windowedBy()`). Временная привязка событий зависит от настройки `withTimestampExtractor` (по умолчанию — время записи из заголовка Kafka).

**Tumbling window** (`TimeWindows.ofSizeWithNoGrace(Duration.ofSeconds(55))`):
- Фиксированный размер, нет перекрытия.
- Событие принадлежит ровно одному окну.
- Окна не имеют разрывов (gap-less) — полностью покрывают временную шкалу.

**Hopping window** (`TimeWindows.of(Duration.ofMinutes(2)).advanceBy(Duration.ofSeconds(10))`):
- Фиксированный размер, настраиваемый шаг.
- При шаге < размера: одно событие присутствует в нескольких соседних окнах.
- Подходит для скользящих метрик.

**Sliding window** (`SlidingWindows.withTimeDifferenceAndNoGrace(Duration.ofHours(3))`):
- Окно определяется парой событий: включает все события, разница timestamp которых не превышает заданного значения.
- Границы окна подстраиваются под данные, а не фиксированы по времени.
- Используется для stream-stream joins.

**Session window** (`SessionWindows.ofInactivityGapWithNoGrace(Duration.ofMinutes(30))`):
- Нет фиксированного размера — определяется активностью.
- Пока события одного ключа поступают с промежутком <= gap — сессия продолжается.
- Когда два соседних события выходят за пределы gap — создаётся новая сессия.

**Grace period** — временной буфер для опоздавших событий:

```java
TimeWindows.ofSizeAndGrace(
    Duration.ofSeconds(55),   // размер окна
    Duration.ofSeconds(10)    // grace period: принимать опоздавшие события ещё 10 сек
)
```

Без grace period событие, опоздавшее хотя бы на 1 миллисекунду после закрытия окна, молча отбрасывается. Grace period позволяет принять события с небольшим опозданием (например, при сетевых задержках или clock skew между узлами). Session window не имеет grace period — он управляется исключительно inactivity gap.

**Сравнительная таблица:**

| Тип окна | Размер | Перекрытие | Управление | Типичный use case |
|---|---|---|---|---|
| Tumbling | Фиксированный | Нет | По времени | Периодические отчёты |
| Hopping | Фиксированный | Да | По времени | Скользящие метрики |
| Sliding | Динамический | Да | По парам событий | Stream-stream join |
| Session | Динамический | Нет | По активности | Пользовательские сессии |

### Вопрос 54. Что такое Kafka Connect? Чем connector отличается от task? Чем standalone mode отличается от distributed mode, и как distributed mode обеспечивает fault tolerance?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [User Guide | Apache Kafka Connect](https://kafka.apache.org/41/kafka-connect/user-guide/)

Kafka Connect — фреймворк для построения масштабируемых и надёжных пайплайнов данных между Kafka и внешними системами. Он обеспечивает «message-at-a-time» обработку через коннекторы.

**Workers** — runtime-процессы, выполняющие коннекторы. Два режима:
- **Standalone**: один процесс, простая настройка, нет fault tolerance.
- **Distributed**: несколько процессов с автоматической балансировкой нагрузки, fault tolerance и динамическим масштабированием.

**Connectors** — приложения-мосты, интегрирующие Kafka с внешними системами:
- **Source connectors**: извлекают данные из внешней системы и пишут в Kafka-топики.
- **Sink connectors**: читают из Kafka-топиков и пишут во внешние системы.

**Tasks** — единицы исполнения внутри коннектора. Параметр `tasks.max` задаёт максимальное число параллельных задач, хотя коннектор может создать меньше задач, чем запрошено.

В distributed mode все worker-процессы используют один `group.id`. При добавлении, отключении или аварийном завершении воркера остальные автоматически перераспределяют между собой все коннекторы и задачи.

#### Ответ от Claude

**Kafka Connect: назначение и архитектура**

Kafka Connect — компонент экосистемы Apache Kafka для интеграции Kafka с внешними системами (СУБД, хранилищами, поисковыми индексами, стриминговыми платформами) без написания custom producer/consumer кода. Он стандартизирует разработку, конфигурацию и мониторинг data pipeline.

**Connector vs Task**

| Понятие | Назначение | Аналогия |
|---|---|---|
| Connector | Высокоуровневая конфигурация интеграции: что, откуда, куда | Blueprint |
| Task | Единица параллельной работы: читает/пишет конкретный фрагмент данных | Worker thread |

Коннектор определяет, какие задачи создать и как разделить между ними работу (например, разные таблицы БД или партиции источника). Каждая задача получает свой набор данных для обработки. Параметр `tasks.max` ограничивает параллелизм, но реальное число задач определяет коннектор.

**Standalone mode**

Один JVM-процесс. Подходит для разработки, тестирования, однократных миграций. Конфигурация хранится в файлах на диске. При падении процесса всё останавливается — нет автоматического восстановления. Подходит для сценариев, где простота важнее отказоустойчивости.

**Distributed mode**

Несколько worker-процессов с одинаковым `group.id` образуют кластер Connect. Конфигурации коннекторов, смещения (offsets) и статусы хранятся в трёх внутренних Kafka-топиках:
- `config.storage.topic` — конфигурации коннекторов и задач
- `offset.storage.topic` — прогресс (смещения) source-коннекторов
- `status.storage.topic` — статусы коннекторов и задач

**Fault tolerance в distributed mode:**

1. Воркеры используют протокол consumer group Kafka для координации.
2. При добавлении нового воркера — ребаланс, задачи перераспределяются между всеми участниками.
3. При падении воркера — ребаланс, его задачи подхватывают оставшиеся.
4. Прогресс source-коннекторов (офсеты во внешней системе) хранится в Kafka — после рестарта задача продолжает с того же места.

**REST API** distributed mode предоставляет REST API (порт 8083 по умолчанию) для управления коннекторами: создание (`POST /connectors`), мониторинг (`GET /connectors/{name}/status`), удаление (`DELETE /connectors/{name}`).

### Вопрос 55. Зачем нужен Confluent Schema Registry и как он решает проблему эволюции схем? Какие типы совместимости (BACKWARD, FORWARD, FULL) существуют, почему BACKWARD является дефолтным и как schema ID встраивается в сообщение?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Schema Evolution & Compatibility | Confluent Documentation](https://docs.confluent.io/current/schema-registry/avro.html)

Schema Registry предоставляет централизованный репозиторий для управления и валидации схем, используемых при сериализации данных. Он обеспечивает проверку совместимости при эволюции схем.

Поддерживаются четыре режима совместимости:

**BACKWARD** (дефолтный): новые схемы способны читать данные, записанные старыми схемами. Допускается добавление опциональных полей и удаление полей. При BACKWARD консюмеры обновляются первыми.

**FORWARD**: старые схемы способны читать данные, записанные новыми схемами. При FORWARD продюсеры обновляются первыми.

**FULL**: схема совместима как BACKWARD, так и FORWARD. Допускается только добавление/удаление опциональных полей с дефолтами.

Для Avro/Protobuf существуют также транзитивные варианты: `BACKWARD_TRANSITIVE`, `FORWARD_TRANSITIVE`, `FULL_TRANSITIVE` — проверка совместимости проводится не только с последней версией схемы, но со всеми предыдущими.

**Почему BACKWARD — дефолт:** «Основная причина, по которой BACKWARD является дефолтным и предпочтительным режимом для Kafka — возможность перемотать консюмера к началу топика». Это позволяет переобработать исторические данные с обновлённой схемой.

#### Ответ от Claude

**Проблема без Schema Registry**

При использовании Kafka без контроля схем производители и потребители рискуют deserialization-ошибками при любом изменении структуры данных: добавлении поля, переименовании типа, удалении поля. При большом числе сервисов координировать обновления вручную крайне сложно.

**Как работает Schema Registry**

Schema Registry — отдельный сервис с REST API. Схемы (Avro, JSON Schema, Protobuf) регистрируются в Registry по «subject» — обычно `{topic-name}-value` и `{topic-name}-key`. Каждая зарегистрированная схема получает уникальный числовой ID.

**Формат сообщения («magic byte» протокол Confluent):**

```
[0x00] [schema_id: 4 байта big-endian] [Avro-encoded payload]
```

Первый байт `0x00` — «magic byte», сигнализирующий, что сообщение использует Confluent Wire Format. Следующие 4 байта — ID схемы. Консюмер читает schema ID, запрашивает схему у Registry (с кешированием) и десериализует payload.

**Четыре режима совместимости:**

| Режим | Кто может читать что | Кто обновляется первым |
|---|---|---|
| BACKWARD | Новая схема читает старые данные | Консюмеры |
| FORWARD | Старая схема читает новые данные | Продюсеры |
| FULL | Оба направления | Независимо |
| TRANSITIVE | Как выше, но проверка со всеми версиями | — |

**Почему BACKWARD дефолт:** Kafka — система хранения (retention). Консюмер может перемотать топик к любому историческому смещению. Если новая схема совместима BACKWARD, консюмер после обновления способен десериализовать и старые, и новые сообщения из топика — это фундаментальное требование для replay сценариев.

**Допустимые изменения схемы Avro по режиму:**
- **BACKWARD**: добавить опциональное поле с default, удалить поле.
- **FORWARD**: удалить опциональное поле с default, добавить поле.
- **FULL**: добавить/удалить только опциональные поля с default.
- Переименование поля и смена типа нарушают совместимость во всех режимах.

**Кеширование:** KafkaAvroSerializer/Deserializer кешируют схемы локально, чтобы не обращаться к Registry при каждом сообщении. При промахе кеша — HTTP GET-запрос к Registry.

### Вопрос 56. Как реализовать задержанную (delayed) доставку сообщений в RabbitMQ? Что такое плагин rabbitmq-delayed-message-exchange, как объявить x-delayed-message exchange, где хранятся сообщения в период ожидания и каковы ключевые ограничения плагина?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [rabbitmq-delayed-message-exchange | GitHub (rabbitmq)](https://github.com/rabbitmq/rabbitmq-delayed-message-exchange/blob/main/README.md)

Плагин `rabbitmq-delayed-message-exchange` добавляет поддержку отложенной/запланированной доставки сообщений в RabbitMQ. После активации плагина пользователь объявляет exchange с типом `x-delayed-message` и аргументом `x-delayed-type`, указывающим базовый тип маршрутизации (direct, topic, fanout и т.д.). Сообщения публикуются с заголовком `x-delay` — числом миллисекунд задержки. После истечения задержки сообщение маршрутизируется в очереди согласно базовому типу exchange.

**Хранение задержанных сообщений:** они хранятся в таблице Mnesia с одной дисковой репликой на текущем узле. Сообщения переживают перезапуск узла (durability на диске), однако таймеры не персистентны — при рестарте они инициализируются заново из таблицы. Поскольку хранение происходит на единственном узле, потеря этого узла означает потерю всех отложенных сообщений.

**Ограничения:** плагин не подходит для высоких объёмов; не поддерживает флаг `mandatory` (нет гарантии маршрутизации через заданное время); существуют ограничения максимальной задержки, определяемые размером erlang-таймера.

#### Ответ от Claude

**Сценарий использования**

Delayed messaging нужен при реализации: retry с экспоненциальной задержкой, напоминаний («отправить email через 24 часа»), отложенного запуска задач (job scheduling), временной блокировки повторных попыток.

**Как работает плагин**

Плагин перехватывает сообщения, опубликованные в `x-delayed-message` exchange. Если заголовок `x-delay` присутствует — сообщение помещается во внутреннюю таблицу Mnesia на `x-delay` миллисекунд. По истечении этого времени сообщение маршрутизируется через базовый exchange-тип (задан `x-delayed-type`), как обычное сообщение.

**Объявление и использование (Java AMQP Client):**

```java
// Объявление exchange
Map<String, Object> args = new HashMap<>();
args.put("x-delayed-type", "direct");
channel.exchangeDeclare("delayed-exchange", "x-delayed-message", true, false, args);

// Публикация с задержкой 5 секунд
AMQP.BasicProperties props = new AMQP.BasicProperties.Builder()
    .headers(Map.of("x-delay", 5000))
    .build();
channel.basicPublish("delayed-exchange", "routing-key", props, body);
```

**Где хранятся задержанные сообщения:**

Mnesia — встроенная СУБД Erlang. Данные:
- Хранятся на диске одного узла (не реплицируются в кластере).
- Переживают плановый рестарт.
- Таймеры восстанавливаются при старте плагина из таблицы.
- Потеря узла = потеря всех задержанных сообщений этого узла.

**Ключевые ограничения:**

| Ограничение | Детали |
|---|---|
| Нет репликации | Задержанные сообщения не реплицируются между узлами кластера |
| Низкая масштабируемость | Не подходит для высокого объёма задержанных сообщений |
| Нет mandatory | Нельзя гарантировать наличие очереди в момент доставки |
| Потеря при падении узла | Одна дисковая реплика на узел |
| Максимальная задержка | Ограничена размером Erlang-таймера |

**Альтернативы:** для высоконагруженных систем рекомендуется использовать внешний планировщик (Quartz, Spring Scheduler) + хранение задач в БД (Transactional Outbox) вместо плагина.

### Вопрос 57. Как работает механизм Priority Queue в RabbitMQ? Что такое аргумент x-max-priority, в каком диапазоне задаются приоритеты, как prefetch влияет на доставку по приоритету и поддерживают ли quorum queues приоритеты?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Priority Support in Queues | RabbitMQ Official Documentation](https://www.rabbitmq.com/docs/priority)

RabbitMQ поддерживает приоритетные очереди. Чтобы объявить очередь с поддержкой приоритетов, используется необязательный аргумент `x-max-priority` — положительное целое число от 1 до 255. Документация настоятельно рекомендует использовать от 2 до 4 уровней приоритета, чтобы не расходовать лишние ресурсы.

Издатель задаёт приоритет сообщения через поле `priority` в `basic.properties`: большее число означает более высокий приоритет. Сообщения без явного приоритета в classic queue обрабатываются как приоритет 0.

Quorum queues всегда поддерживают 32 уровня приоритета (0–31) без дополнительной настройки; для них `x-max-priority` не требуется и игнорируется. Сообщения без поля priority в quorum queue получают уровень 4 по умолчанию.

**Влияние prefetch на приоритет:** настройка `basic.qos` ограничивает число сообщений, одновременно доставляемых потребителю. Если лимит prefetch достигнут, высокоприоритетные сообщения вынуждены ждать в очереди за уже отправленными низкоприоритетными. Поэтому значение prefetch должно быть тщательно подобрано для корректной работы приоритетов.

#### Ответ от Claude

**Как реализованы приоритеты в RabbitMQ**

Приоритетная очередь реализована внутри как набор подочередей — по одной на каждый уровень приоритета. Когда потребитель запрашивает сообщение, брокер сначала проверяет подочередь с наивысшим занятым приоритетом и доставляет из неё. Таким образом, при наличии сообщений разных приоритетов высокоприоритетные всегда идут раньше.

**Объявление (Java AMQP Client):**

```java
Map<String, Object> args = new HashMap<>();
args.put("x-max-priority", 5);  // поддерживаем приоритеты 0..5
channel.queueDeclare("priority-queue", true, false, false, args);

// Публикация с приоритетом 3
AMQP.BasicProperties props = new AMQP.BasicProperties.Builder()
    .priority(3)
    .build();
channel.basicPublish("", "priority-queue", props, body);
```

**Диапазоны и рекомендации:**

| Тип очереди | Поддерживаемый диапазон | Рекомендация |
|---|---|---|
| Classic queue | 0–255 (задаётся x-max-priority) | 2–4 уровня — оптимально |
| Quorum queue | 0–31 (всегда, без настройки) | default priority = 4 при отсутствии поля |

Каждый уровень приоритета потребляет память и CPU. Большое число уровней без реальной необходимости — лишняя нагрузка.

**Как prefetch нарушает приоритет**

`basic.qos(prefetchCount=N)` разрешает брокеру отправить не более N сообщений потребителю без получения ack. Если N сообщений низкого приоритета уже отправлено и не подтверждено, высокоприоритетное сообщение останется в очереди — prefetch-лимит уже исчерпан. Чтобы приоритеты работали корректно, prefetch должен быть минимальным (в идеале 1), что позволяет брокеру после каждого ack выбрать самое приоритетное сообщение.

**Голодание (starvation):** при непрерывном потоке высокоприоритетных сообщений низкоприоритетные могут ждать бесконечно — RabbitMQ не реализует aging (автоповышение приоритета с течением времени). Это необходимо учитывать в проектировании.

### Вопрос 58. Как работает партиционер (Partitioner) в Kafka-producer? Что делает дефолтный партиционер при наличии и отсутствии ключа, что такое Sticky Partitioner и почему он улучшает latency при низком throughput?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Apache Kafka Producer Improvements: Sticky Partitioner — Confluent Blog](https://www.confluent.io/blog/apache-kafka-producer-improvements-sticky-partitioner/)

До появления Sticky Partitioner Kafka при отсутствии ключа сообщения циклически перебирала партиции топика (round-robin), создавая множество мелких батчей с высокими накладными расходами. Sticky Partitioner (Kafka 2.4+) выбирает одну партицию для всех записей без ключа и «прилипает» к ней до тех пор, пока батч не заполнится или не истечёт `linger.ms`. После отправки батча случайно выбирается следующая партиция.

Результаты тестирования: при 3 продюсерах, отправляющих 1000 сообщений/с в топик с 16 партициями, Sticky Partitioner достигал примерно вдвое меньшей latency на 99-м персентиле. При `linger.ms=1000` разрыв увеличивался до 5x.

#### Ответ от Claude

**Как партиционер выбирает целевую партицию**

Партиционер вызывается при каждом `ProducerRecord` и определяет, в какую партицию топика направить запись. Логика зависит от наличия ключа и настроек.

**DefaultPartitioner (Kafka < 2.4, логика)**

1. Если задана явная партиция (`ProducerRecord.partition()`) — использует её.
2. Если задан ключ — выбирает партицию по `murmur2(key) % numPartitions`. Все записи с одинаковым ключом попадают в одну партицию (гарантия порядка по ключу).
3. Если ключа нет — round-robin по партициям.

Проблема round-robin: каждая запись без ключа попадает в следующую партицию по кругу. При малом throughput это создаёт много партиционированных батчей с единственной записью — маленький батч = высокие накладные расходы = высокая latency.

**Sticky Partitioner (Kafka 2.4+, поведение по умолчанию)**

Для записей без ключа: выбрать одну партицию и направлять в неё все записи до тех пор, пока:
- батч не достигнет `batch.size` (16 384 байт по умолчанию), или
- не истечёт `linger.ms`.

После отправки — случайный выбор новой партиции.

Эффект: батчи заполняются быстрее (по размеру, а не по таймауту), что снижает latency. Распределение по партициям остаётся равномерным в долгосрочной перспективе.

**Статус в современных версиях Kafka**

В Kafka 3.3+ поведение по умолчанию инкорпорирует sticky-логику напрямую в DefaultPartitioner. Отдельный класс `UniformStickyPartitioner` объявлен deprecated. Вместо явного задания `partitioner.class=UniformStickyPartitioner` рекомендуется:
```properties
# не задавать partitioner.class
partitioner.ignore.keys=true  # игнорировать ключ при партиционировании
```

**Кастомный партиционер**

Реализуется через интерфейс `org.apache.kafka.clients.producer.Partitioner`:

```java
public class CustomPartitioner implements Partitioner {
    @Override
    public int partition(String topic, Object key, byte[] keyBytes,
                         Object value, byte[] valueBytes, Cluster cluster) {
        // бизнес-логика: VIP-записи → партиция 0
        if ("VIP".equals(key)) return 0;
        return murmur2(keyBytes) % cluster.partitionCountForTopic(topic);
    }
}
```

### Вопрос 59. В чём разница между Federation и Shovel плагинами в RabbitMQ? Как каждый из них соединяет брокеры, когда использовать Federation, а когда Shovel?

**Категория:** Брокеры сообщений · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Distributed RabbitMQ | RabbitMQ Official Documentation](https://www.rabbitmq.com/docs/distributed)

Federation и Shovel — два плагина для соединения брокеров RabbitMQ через сеть.

**Federation** предназначен для построения федерации exchanges или очередей между брокерами с помощью протокола AMQP (опционально с TLS). Федерированные exchanges соединены однонаправленными point-to-point ссылками. Это высокоуровневый механизм с «мнением» о топологии: цель — распределить exchanges или очереди между брокерами, не требуя полного mesh-соединения всех узлов.

**Shovel** работает на более низком уровне: простой потребитель на одной стороне и publisher на другой. Shovel потребляет сообщения из очереди брокера-источника и публикует их в exchange брокера-назначения. Подходит для явного перемещения данных между любыми двумя брокерами, в том числе в разных административных доменах.

Federation больше подходит для pub/sub паттернов и балансировки очередей на уровне WAN; Shovel — для точечных ETL-пайплайнов, одноразовых миграций, явного перемещения данных.

#### Ответ от Claude

**Federation**

Federation позволяет «подписаться» одному брокеру (downstream) на exchange или очередь другого брокера (upstream). Механизм: downstream-брокер создаёт внутреннего consumer на upstream, реплицирует сообщения локально и маршрутизирует их по своим правилам. Federation прозрачна для приложений: они публикуют в локальный exchange, не зная, что часть сообщений приходит из другого датацентра.

Типичные сценарии:
- Географически распределённая pub/sub система (публикация в одном датацентре, потребление в другом).
- Балансировка нагрузки между кластерами RabbitMQ.
- High-availability без полного кластера (кластеризация требует надёжной LAN, federation — нет).

**Shovel**

Shovel — простейший конвейер: consumer + publisher. Shovel читает из одной очереди (источник) и публикует в exchange другого брокера (назначение). После подтверждения доставки сообщение удаляется из источника. Shovel WAN-friendly: при разрыве соединения он автоматически переподключается и продолжает с места остановки.

Типичные сценарии:
- Одноразовая миграция очередей между кластерами.
- ETL-пайплайн: из legacy-брокера в новый.
- Явная маршрутизация данных между административно изолированными системами.

**Сравнительная таблица:**

| Характеристика | Federation | Shovel |
|---|---|---|
| Уровень абстракции | Высокий (exchange/queue) | Низкий (queue → exchange) |
| Прозрачность | Прозрачна для приложений | Явная конфигурация пайплайна |
| Гибкость | Стандартизированная топология | Полный контроль |
| Основные паттерны | Pub/sub, балансировка | ETL, миграция, точечная маршрутизация |
| Разбивка на подписки | Адаптируется к новым bindingам | 1 shovel = 1 пайплайн |
| WAN-дружественность | Да | Да (auto-reconnect) |
| Сообщение удаляется из источника | Нет (копирование) | Да (перемещение) |

**Ключевое отличие:** Federation копирует сообщения (pub/sub семантика), Shovel перемещает их (queue-forwarding семантика).

### Вопрос 60. Как реализовать retry-механизм при обработке сообщений Kafka в Spring Kafka? Чем @RetryableTopic (non-blocking retry) отличается от DefaultErrorHandler с blocking retry? Что такое Dead Letter Topic и как настроить маршрутизацию неудачных сообщений?

**Категория:** Брокеры сообщений · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Implementing Retry in Kafka Consumer | Baeldung](https://www.baeldung.com/spring-retry-kafka-consumer)

Spring Kafka предоставляет два основных подхода к реализации повторных попыток.

**Blocking retry (DefaultErrorHandler):** синхронные повторные попытки в рамках той же listener-нити. `DefaultErrorHandler` принимает `FixedBackOff` для задания интервала и числа попыток. Поддерживает `addRetryableExceptions()` и `addNotRetryableExceptions()` для гранулярного контроля. После исчерпания попыток можно использовать `DeadLetterPublishingRecoverer` для отправки сообщения в Dead Letter Topic (DLT).

**Non-blocking retry (@RetryableTopic):** при ошибке сообщение перенаправляется в специальный retry-топик с настраиваемой задержкой. Поддерживает экспоненциальный backoff через аннотацию `@Backoff`. Основной поток обработки не блокируется. После исчерпания попыток — в DLT. Метод-обработчик DLT помечается `@DltHandler`.

Важное ограничение: `AckMode.BATCH` не совместим с retry-механизмами, поскольку подтверждает сразу группу сообщений, не позволяя повторить конкретное.

#### Ответ от Claude

**Проблема без retry**

При ошибке в `@KafkaListener` (временный сбой БД, сетевая ошибка) сообщение по умолчанию либо теряется (auto-commit), либо создаёт бесконечный цикл повтора. Нужен управляемый механизм повторных попыток с ограничением числа и задержками.

**Подход 1: DefaultErrorHandler (blocking retry)**

Повторные попытки выполняются в той же нити, не продвигая offset до исчерпания попыток.

```java
@Bean
public DefaultErrorHandler errorHandler(KafkaTemplate<?, ?> template) {
    var recoverer = new DeadLetterPublishingRecoverer(template);
    var backOff = new FixedBackOff(1000L, 3);  // 3 попытки с паузой 1 сек
    var handler = new DefaultErrorHandler(recoverer, backOff);
    handler.addNotRetryableExceptions(IllegalArgumentException.class);
    return handler;
}
```

При исчерпании попыток `DeadLetterPublishingRecoverer` публикует сообщение в топик `{original-topic}.DLT` с заголовками-метаданными об ошибке.

**Подход 2: @RetryableTopic (non-blocking retry)**

Сообщение при ошибке перенаправляется в отдельный retry-топик; основной consumer продолжает работу.

```java
@RetryableTopic(
    attempts = "4",
    backoff = @Backoff(delay = 1000, multiplier = 2.0),
    include = {RetryableException.class},
    autoCreateTopics = "true"
)
@KafkaListener(topics = "my-topic", groupId = "my-group")
public void listen(String message) {
    // обработка
}

@DltHandler
public void handleDlt(String message, @Header KafkaHeaders.RECEIVED_TOPIC String topic) {
    // логирование / алертинг неудачных сообщений
}
```

Spring автоматически создаёт топики: `my-topic-retry-1000`, `my-topic-retry-2000`, `my-topic-retry-4000`, `my-topic-dlt`.

**Сравнение подходов:**

| Характеристика | DefaultErrorHandler | @RetryableTopic |
|---|---|---|
| Характер retry | Блокирующий (нить ждёт) | Неблокирующий (message в retry-топик) |
| Влияние на throughput | Снижает (нить занята) | Минимальное |
| Порядок сообщений | Сохраняется | Нарушается (retry идут позже основного потока) |
| Настройка | Программная | Аннотационная |
| Backoff стратегия | FixedBackOff / ExponentialBackOff | @Backoff с multiplier |
| DLT | DeadLetterPublishingRecoverer | Автоматически через @DltHandler |

**Когда что использовать:**
- `DefaultErrorHandler` — когда порядок сообщений критичен, задержки между попытками малы, retry-топики нежелательны.
- `@RetryableTopic` — когда задержки между попытками измеряются минутами/часами, порядок некритичен, нужен минимальный impact на latency основного потока.
