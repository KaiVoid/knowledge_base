### Вопрос 1. Чем монолитная архитектура отличается от микросервисной? Назовите ключевые преимущества и недостатки каждого подхода.

**Категория:** Микросервисы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Microservices Pattern: Monolithic Architecture](https://microservices.io/patterns/monolithic.html)

Монолитная архитектура — это паттерн, при котором приложение реализуется как **единый развёртываемый/исполняемый компонент**, использующий одну базу данных. Все бизнес-возможности (субдомены) приложения объединены в этом едином компоненте, поэтому все операции являются локальными, а не распределёнными.

**Преимущества монолита:**
- Взаимодействие между компонентами проще: всё локально, легче отлаживать и понимать.
- Коммуникация эффективнее за счёт отсутствия сетевых вызовов.
- Транзакции можно реализовать как ACID, без необходимости в eventual consistency.
- Отсутствует runtime- и design-time-связность между отдельными компонентами (нет накладных расходов распределённой системы).

**Недостатки монолита:**
- По мере роста единый крупный компонент становится трудно понимать и поддерживать.
- Команды вынуждены работать в одном репозитории, что требует постоянной координации и снижает автономию.
- Медленный пайплайн деплоя: при каждом изменении нужно собирать и тестировать всё приложение целиком.
- Привязанность к единому технологическому стеку для всех субдоменов.
- Невозможно выделять субдомены с особыми требованиями по ресурсам, доступности или безопасности.

Микросервисная архитектура появилась как альтернативный паттерн, призванный решить именно эти проблемы — прежде всего при масштабировании приложения и росте команды.

#### Ответ от Claude

**Монолитная архитектура** — это подход, при котором всё приложение собрано и развёртывается как единый артефакт (JAR, WAR, EXE). Все модули (UI, бизнес-логика, слой доступа к данным) работают в одном процессе и обращаются к одной базе данных.

**Микросервисная архитектура** (согласно Мартину Фаулеру) — это подход, при котором приложение строится как набор небольших сервисов, каждый из которых выполняется в собственном процессе и взаимодействует с другими через облегчённые механизмы (как правило, HTTP/REST или асинхронные сообщения). Каждый сервис независимо развёртывается и сосредоточен вокруг одной бизнес-возможности.

**Ключевое отличие:** монолит требует пересборки и переразвёртывания всего приложения при любом изменении; микросервисы позволяют переразвёртывать каждый сервис независимо.

---

**Преимущества монолита:**
| # | Преимущество | Пояснение |
|---|---|---|
| 1 | Простота разработки на старте | Нет распределённых вызовов, нет брокеров, единая кодовая база |
| 2 | Простота транзакций | ACID-транзакции внутри одного процесса, не нужна saga/2PC |
| 3 | Проще отлаживать | Единый стектрейс, нет сетевых сбоев между компонентами |
| 4 | Низкие операционные издержки | Один деплой, один сервис для мониторинга |

**Недостатки монолита:**
| # | Недостаток | Пояснение |
|---|---|---|
| 1 | Сложность масштабирования кода | Большая кодовая база трудно понимаема и сопровождаема |
| 2 | Низкая автономия команд | Конфликты при слиянии, необходима координация |
| 3 | Медленный деплой | Пересборка и тестирование всего при любом изменении |
| 4 | Единый стек технологий | Нельзя использовать разные языки/базы для разных частей |
| 5 | Масштабирование «всё или ничего» | Нельзя масштабировать только нагруженную часть |

---

**Преимущества микросервисов:**
| # | Преимущество | Пояснение |
|---|---|---|
| 1 | Независимый деплой | Изменение одного сервиса не требует пересборки остальных |
| 2 | Гибкость стека | Каждый сервис может использовать свой язык, базу данных, библиотеки |
| 3 | Изолированные сбои | Падение одного сервиса не обваливает систему целиком |
| 4 | Масштабирование отдельных частей | Можно горизонтально масштабировать только нагруженные сервисы |
| 5 | Автономия команд | Небольшая команда владеет сервисом end-to-end |

**Недостатки микросервисов:**
| # | Недостаток | Пояснение |
|---|---|---|
| 1 | Сложность распределённой системы | Сетевые сбои, latency, частичные отказы, отладка распределённых трейсов |
| 2 | Сложность транзакций | Нет глобальных ACID-транзакций; нужны паттерны Saga, Outbox |
| 3 | Операционные накладные расходы | Нужны service discovery, distributed tracing, CI/CD на каждый сервис |
| 4 | Избыточность для малых приложений | Overhead не оправдан, если команда и функциональность невелики |

**Вывод:** монолит — хороший выбор на старте или для небольших команд; микросервисы оправданы при росте команды, необходимости независимого деплоя отдельных частей и высоких требованиях к масштабируемости.

### Вопрос 2. Что такое связность (coupling) и связанность (cohesion) применительно к микросервисам? Почему важно стремиться к низкой связности и высокой связанности?

**Категория:** Микросервисы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Microservices Pattern: A pattern language for microservices — microservices.io](https://microservices.io/post/architecture/2025/03/10/premium-microservices-rules-7-loosely-design-time-coupled-services-part-1.html)

Ключевая цель проектирования микросервисов — минимизация design-time coupling (связности на этапе разработки) и максимизация cohesion (связанности). Сервис с высокой связанностью сосредоточен вокруг одной бизнес-возможности, а его элементы тесно относятся друг к другу. Слабая связность означает, что изменение одного сервиса не требует изменения другого. Именно это сочетание — «высокая cohesion + низкий coupling» — делает разработку, деплой и эволюцию микросервисов гибкими и масштабируемыми.

#### Ответ от Claude

**Coupling (связность/зависимость)** — степень взаимозависимости между модулями (сервисами). Высокий coupling означает: изменение одного сервиса вынуждает изменять другой. Низкий coupling означает: сервисы можно развивать, деплоить и масштабировать независимо друг от друга.

**Cohesion (связанность/сплочённость)** — степень внутренней «тематической цельности» модуля (сервиса). Высокая cohesion означает: все элементы сервиса относятся к одной ответственности, изменяются вместе и по одной причине. Часто формулируется принципом *«код, который меняется вместе, живёт вместе»*.

---

**Виды coupling в микросервисах:**

| Вид | Описание | Опасность |
|---|---|---|
| Runtime coupling | Сервис A недоступен, если сервис B недоступен (синхронный вызов) | Каскадные сбои |
| Design-time coupling | Изменение API сервиса B требует изменения кода сервиса A | Замедляет независимый деплой |
| Data coupling | Сервисы используют одну таблицу/базу данных | Изменение схемы ломает несколько команд |
| Temporal coupling | Сервисы должны быть запущены одновременно | Снижает доступность системы |

---

**Почему важна низкая coupling:**
1. **Независимый деплой** — команда сервиса A может выпускать релизы без координации с командой сервиса B.
2. **Изоляция сбоев** — падение сервиса B не блокирует сервис A (при асинхронной коммуникации).
3. **Технологическая независимость** — каждый сервис может менять внутренние технологии, не затрагивая соседей.
4. **Параллельная разработка** — команды работают автономно без постоянной синхронизации.

**Почему важна высокая cohesion:**
1. **Понятность** — сервис, сосредоточенный вокруг одной бизнес-функции, легче понять и поддерживать.
2. **Локализованные изменения** — новая требования к «Заказам» затрагивают только сервис Orders, а не половину системы.
3. **Правильные границы сервисов** — высокая cohesion сигнализирует, что сервис правильно выделен по бизнес-домену (DDD bounded context).
4. **Меньше нежелательных зависимостей** — когда сервис делает «одно дело», у него меньше поводов вызывать другие сервисы.

**Связь между двумя принципами:** Мартин Фаулер предупреждает — если границы сервисов определены неверно, сложность лишь переносится из кода в связи между сервисами. Высокая cohesion помогает провести правильные границы, а низкий coupling — удержать их во времени. Вместе они обеспечивают главное преимущество микросервисов: возможность независимо развивать и доставлять каждый сервис.

### Вопрос 3. Как микросервисы могут взаимодействовать друг с другом? Перечислите основные способы синхронного и асинхронного взаимодействия.

**Категория:** Микросервисы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Microservices Pattern: Messaging](https://microservices.io/patterns/communication-style/messaging.html)

Паттерн Messaging описывает асинхронный способ взаимодействия микросервисов: сервисы обмениваются **сообщениями через каналы сообщений** (message channels). Поддерживается пять стилей общения:
- **Request/response** — отправитель ждёт быстрого ответа.
- **Notifications** — отправитель посылает сообщение, не ожидая ответа.
- **Request/async response** — отправитель ожидает ответа, но не блокируется.
- **Publish/subscribe** — сервис публикует сообщение для нескольких подписчиков.
- **Publish/async response** — публикация с опциональным асинхронным ответом.

**Преимущества:** слабый runtime coupling (брокер буферизует сообщения, консьюмер не обязан быть доступен в момент отправки), повышенная доступность.

**Недостатки:** необходим высокодоступный брокер (Kafka, RabbitMQ); request/reply сложнее реализовывать асинхронно. Популярные реализации: Apache Kafka, RabbitMQ.

#### Ответ от Claude

Микросервисы взаимодействуют двумя основными способами: **синхронным** (клиент ждёт ответа) и **асинхронным** (клиент не ждёт ответа немедленно).

---

### Синхронное взаимодействие (RPI — Remote Procedure Invocation)

Клиент отправляет запрос и блокируется до получения ответа. Оба сервиса должны быть доступны одновременно.

| Технология | Особенности |
|---|---|
| **REST / HTTP** | Самый распространённый вариант. Ресурсная модель, JSON/XML, легко отлаживать и документировать (OpenAPI). |
| **gRPC** | Бинарный протокол поверх HTTP/2, Protocol Buffers. Эффективнее REST по скорости и размеру сообщений; типизированный контракт; поддерживает двунаправленный стриминг. |
| **Apache Thrift** | Кросс-языковой RPC-фреймворк; реже встречается в новых проектах. |
| **GraphQL** | Позволяет клиенту запрашивать только нужные поля; удобен для агрегации данных из нескольких сервисов через API Gateway. |

**Преимущества синхронного взаимодействия:**
- Простота реализации и отладки.
- Немедленный ответ; удобен для запросов, где нужен результат «здесь и сейчас».

**Недостатки синхронного взаимодействия:**
- **Runtime coupling** — если вызываемый сервис недоступен, вызывающий тоже деградирует.
- Цепочка синхронных вызовов (A → B → C → D) мультиплицирует риск сбоя и latency.
- Требует паттернов отказоустойчивости: Circuit Breaker, Retry, Timeout.

---

### Асинхронное взаимодействие (Messaging)

Сервис отправляет сообщение в брокер и не ждёт ответа. Брокер гарантирует доставку консьюмеру позже.

| Стиль | Описание | Пример применения |
|---|---|---|
| **Notification (Fire-and-Forget)** | Отправитель шлёт событие, ответ не нужен. | Уведомление «заказ создан» → Email-сервис |
| **Request/Async Response** | Запрос через очередь, ответ придёт в другую очередь. | Долгие расчёты, batch-операции |
| **Publish/Subscribe (Pub/Sub)** | Продюсер публикует событие; несколько подписчиков обрабатывают независимо. | Domain events в Event-Driven Architecture |
| **Event Streaming** | Непрерывный поток событий с возможностью replay (журнал). | Apache Kafka, аналитика, Event Sourcing |

**Популярные брокеры:** Apache Kafka (высокая пропускная способность, retention, replay), RabbitMQ (гибкая маршрутизация, AMQP), Amazon SQS/SNS.

**Преимущества асинхронного взаимодействия:**
- **Слабый runtime coupling** — продюсер и консьюмер независимы; сервис-консьюмер может быть временно недоступен.
- **Повышенная доступность** — брокер буферизует сообщения.
- **Масштабируемость** — легко добавить новых подписчиков без изменения продюсера.

**Недостатки асинхронного взаимодействия:**
- Сложнее отлаживать и трассировать (нужен distributed tracing: Jaeger, Zipkin).
- Необходим надёжный, высокодоступный брокер.
- Паттерн request/reply через очереди сложнее реализовать, чем синхронный вызов.
- Eventual consistency: данные между сервисами согласуются со временем, а не мгновенно.

---

### Итоговое сравнение

| Критерий | Синхронное (REST/gRPC) | Асинхронное (Messaging) |
|---|---|---|
| Coupling | Высокий runtime coupling | Низкий runtime coupling |
| Доступность | Зависит от доступности вызываемого | Брокер буферизует, не зависит |
| Сложность | Простая реализация | Требует инфраструктуры брокера |
| Применение | Запросы с немедленным ответом, CRUD | События, интеграция, долгие операции |
| Консистентность | Сильная (synchronous) | Eventual consistency |

На практике в микросервисных системах используют **оба подхода**: синхронный — для запросов к пользователю (read-операции, API Gateway), асинхронный — для интеграции между сервисами (domain events, saga-оркестрация).

### Вопрос 4. Что такое API Gateway и зачем он нужен в микросервисной архитектуре? Какие задачи он решает?

**Категория:** Микросервисы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [API gateways — Azure Architecture Center | Microsoft Learn](https://learn.microsoft.com/en-us/azure/architecture/microservices/design/gateway)

API Gateway — это централизованная точка входа для управления взаимодействием между клиентами и сервисами приложения. Он действует как обратный прокси, перенаправляя запросы клиентов к нужным сервисам, и может выполнять различные сквозные задачи: аутентификацию, SSL-терминацию, mutual TLS и ограничение частоты запросов.

Без API Gateway клиенты вынуждены напрямую обращаться к отдельным сервисам, что порождает ряд проблем:
- **Сложность клиентского кода**: клиент должен отслеживать множество эндпоинтов и самостоятельно обрабатывать сбои.
- **Тесная связанность**: клиент должен знать о внутренней декомпозиции сервисов, что усложняет рефакторинг.
- **Повышенная задержка**: одна операция может требовать вызовов к нескольким сервисам — множество сетевых round-trip'ов.
- **Дублирование сквозной логики**: каждый публичный сервис сам реализует аутентификацию, SSL, rate limiting.
- **Расширенная поверхность атаки**: публичные эндпоинты каждого сервиса увеличивают риски безопасности.

Паттерны использования API Gateway:
- **Gateway Routing** — маршрутизация запросов клиентов к нужным сервисам через единую точку входа (L7 routing).
- **Gateway Aggregation** — агрегация нескольких вызовов к разным сервисам в один ответ клиенту, снижая «болтливость».
- **Gateway Offloading** — вынесение сквозной функциональности (SSL-терминация, аутентификация, rate limiting, логирование, кэширование ответов, WAF, GZIP-сжатие) на уровень gateway, чтобы отдельные сервисы не реализовывали её самостоятельно.

#### Ответ от Claude

**API Gateway** — компонент микросервисной архитектуры, выступающий единственной точкой входа (single entry point) для всех клиентских запросов. По существу это обратный прокси-сервер с расширенными возможностями управления трафиком.

**Зачем нужен API Gateway**

В микросервисной архитектуре каждый бизнес-домен живёт в отдельном сервисе. Клиент (браузер, мобильное приложение, сторонний API-потребитель) без Gateway вынужден знать адреса всех сервисов, самостоятельно агрегировать ответы и дублировать сквозную логику. Gateway устраняет эти проблемы.

**Задачи, которые решает API Gateway**

1. **Маршрутизация (Gateway Routing)** — принимает запрос и на основе пути, заголовков или метода HTTP перенаправляет его в нужный микросервис. Клиент знает только один адрес.

2. **Агрегация ответов (Gateway Aggregation)** — вместо нескольких последовательных запросов от клиента gateway сам параллельно вызывает несколько сервисов, объединяет результаты и возвращает один ответ. Снижает latency и нагрузку на сеть.

3. **Вынесение сквозной логики (Gateway Offloading)** — централизованное выполнение задач, которые иначе пришлось бы дублировать в каждом сервисе:
   - Аутентификация и авторизация (JWT, OAuth2)
   - SSL/TLS-терминация и mutual TLS
   - Rate limiting и throttling
   - Логирование и трассировка (correlation-id)
   - Кэширование ответов
   - CORS
   - Сжатие (GZIP)
   - Web Application Firewall (WAF)

4. **Сокрытие внутренней топологии** — клиенты не знают о количестве и расположении микросервисов. Сервисы можно рефакторить, переносить и масштабировать без изменений на стороне клиента.

5. **Паттерн Backends for Frontends (BFF)** — отдельный gateway для каждого типа клиента (web, mobile, partners) с оптимизированным API под каждый канал.

**Популярные реализации**: Spring Cloud Gateway, Netflix Zuul, Kong, NGINX, AWS API Gateway, Azure API Management.

**Важно**: API Gateway сам может стать единой точкой отказа и узким местом по производительности, поэтому его необходимо делать высокодоступным (несколько инстансов за балансировщиком) и избегать размещения в нём бизнес-логики.

### Вопрос 5. Что такое Service Discovery? Объясните разницу между client-side и server-side discovery. Как работает Netflix Eureka в Spring Cloud?

**Категория:** Микросервисы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Service Discovery: Eureka Clients — Spring Cloud Netflix](https://cloud.spring.io/spring-cloud-netflix/multi/multi__service_discovery_eureka_clients.html)

Наличие `spring-cloud-starter-netflix-eureka-client` в classpath делает приложение одновременно Eureka-«инстансом» (то есть оно регистрирует себя) и Eureka-«клиентом» (оно может запрашивать реестр для обнаружения других сервисов).

Сервис недоступен для обнаружения клиентами до тех пор, пока инстанс, сервер и клиент не получат одинаковые метаданные в своём локальном кэше. У Eureka-сервера нет постоянного хранилища данных — все инстансы сервисов обязаны отправлять сердцебиения (heartbeats), чтобы поддерживать регистрацию актуальной. Heartbeat по умолчанию отправляется каждые 30 секунд. Если сердцебиения прекращаются, инстанс удаляется из реестра.

#### Ответ от Claude

**Service Discovery** — механизм автоматического обнаружения сетевых адресов (хост + порт) работающих экземпляров сервисов в динамической среде. В микросервисной архитектуре сервисы запускаются и останавливаются, меняют IP-адреса (особенно в Kubernetes/Docker), масштабируются горизонтально — жёстко прописанные адреса неприменимы.

**Два подхода к Service Discovery**

**Client-Side Discovery (обнаружение на стороне клиента)**

Клиент сам отвечает за получение адреса нужного сервиса:
1. Клиент обращается к Service Registry (реестру сервисов) и получает список доступных инстансов.
2. Клиент сам выбирает инстанс (применяя алгоритм балансировки: round-robin, least-connections, zone-aware и т.д.).
3. Клиент напрямую вызывает выбранный инстанс.

Плюсы: гибкость в выборе алгоритма балансировки, меньше компонентов инфраструктуры.
Минусы: логика обнаружения дублируется в каждом клиентском сервисе; зависимость от клиентской библиотеки реестра.

Пример: **Netflix Eureka + Spring Cloud LoadBalancer**.

**Server-Side Discovery (обнаружение на стороне сервера)**

Клиент обращается к промежуточному компоненту (load balancer / API Gateway), который сам запрашивает реестр и направляет запрос к нужному инстансу:
1. Клиент отправляет запрос на балансировщик.
2. Балансировщик запрашивает Service Registry.
3. Балансировщик выбирает инстанс и проксирует запрос.

Плюсы: клиентский код не зависит от библиотеки реестра; единое место управления балансировкой.
Минусы: балансировщик — дополнительный компонент инфраструктуры, потенциальная точка отказа.

Пример: **AWS ELB + AWS ECS**, **Kubernetes Service** (kube-proxy + DNS).

**Как работает Netflix Eureka в Spring Cloud**

Eureka состоит из двух частей: **Eureka Server** (реестр) и **Eureka Client** (библиотека в каждом сервисе).

*Eureka Server* — отдельное Spring Boot-приложение:
```java
@SpringBootApplication
@EnableEurekaServer
public class EurekaServerApp { ... }
```
Настраивается в `application.yml`: порт, имя зоны, параметры репликации (при кластеризации серверов).

*Регистрация сервиса*: при старте сервис с зависимостью `spring-cloud-starter-netflix-eureka-client` автоматически регистрируется в Eureka Server, передавая метаданные: `spring.application.name`, хост, порт, URL health check.

*Heartbeat*: каждые 30 секунд (по умолчанию) клиент отправляет heartbeat. Если сервер не получает heartbeat в течение 90 секунд — инстанс исключается из реестра.

*Обнаружение*: клиент кэширует реестр локально и обновляет его каждые 30 секунд. Вместо `http://localhost:8080/api` сервис вызывает `http://order-service/api`, где `order-service` — это `spring.application.name` целевого сервиса. Spring Cloud LoadBalancer разрешает имя в реальный адрес через кэшированный реестр.

*Self-Preservation Mode*: при потере значительной доли heartbeats Eureka переходит в режим защиты и перестаёт удалять инстансы, предполагая сетевые проблемы, а не реальное падение сервисов.

**Замечание**: в современных проектах на Kubernetes чаще используют встроенный Service Discovery через kube-dns и Kubernetes Services, а Eureka применяется преимущественно в non-Kubernetes средах.

### Вопрос 6. В чём разница между REST и gRPC для межсервисного взаимодействия? В каких случаях предпочтительнее использовать gRPC?

**Категория:** Микросервисы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [gRPC vs REST — Difference Between Application Designs | AWS](https://aws.amazon.com/compare/the-difference-between-grpc-and-rest/)

REST следует «entity-oriented design»: клиенты запрашивают ресурсы через HTTP-глаголы (GET, POST, PUT, DELETE). gRPC использует «service-oriented design»: клиенты вызывают функции на сервере, как локальные процедурные вызовы.

Ключевые различия:
- **Протокол**: gRPC всегда использует HTTP/2; REST, как правило, HTTP/1.1.
- **Формат данных**: REST использует JSON/XML (текстовые форматы, удобные для чтения и отладки, но менее эффективные). gRPC использует Protocol Buffers — бинарный формат, который сериализуется и десериализуется быстрее.
- **Потоковая передача**: gRPC поддерживает bidirectional streaming — и клиент, и сервер могут отправлять несколько сообщений одновременно в одном соединении. REST такой возможности не предоставляет.
- **Связанность**: REST слабо связан — изменения на сервере не обязательно требуют обновления клиента. gRPC тесно связан: клиент и сервер должны использовать один и тот же `.proto`-файл.

gRPC предпочтительнее при: высоконагруженных внутренних системах, требующих потоковой передачи данных в реальном времени; polyglot-окружениях (несколько языков программирования); системах, где критична задержка.

#### Ответ от Claude

**REST** (Representational State Transfer) и **gRPC** (Google Remote Procedure Call) — два подхода к межсервисному взаимодействию с принципиально разными философиями и техническими характеристиками.

**Сравнительная таблица**

| Характеристика | REST | gRPC |
|---|---|---|
| Транспортный протокол | HTTP/1.1 (реже HTTP/2) | HTTP/2 (обязательно) |
| Формат данных | JSON / XML (текст) | Protocol Buffers (бинарный) |
| Контракт API | OpenAPI/Swagger (опционально) | `.proto`-файл (обязателен) |
| Кодогенерация | Не обязательна | Обязательна (protoc) |
| Потоковая передача | Нет (только SSE/WebSocket в обход) | 4 режима: unary, server-stream, client-stream, bidirectional |
| Поддержка браузерами | Полная | Ограниченная (требует gRPC-Web) |
| Связанность клиент/сервер | Слабая (loose coupling) | Тесная (tight coupling через .proto) |
| Производительность | Умеренная | Высокая (бинарный протокол, мультиплексирование HTTP/2) |
| Читаемость сообщений | Высокая (JSON) | Низкая (бинарный формат) |

**Технические детали**

*REST* работает по модели ресурсов: URL идентифицирует ресурс, HTTP-метод — действие над ним. Тело запроса/ответа — JSON. Простота, повсеместная поддержка инструментов, лёгкая отладка через curl/Postman.

*gRPC* работает по модели процедурных вызовов: клиент вызывает метод на удалённом сервере как будто локально. Контракт описывается в `.proto`-файле, из которого генерируется код клиента и сервера на любом поддерживаемом языке (Java, Go, Python, C++, и др.). HTTP/2 обеспечивает мультиплексирование (несколько запросов в одном TCP-соединении), header compression (HPACK), бинарную передачу фреймов.

**Protocol Buffers** дают до 70–90% сокращения размера полезной нагрузки по сравнению с JSON и значительно быстрее сериализуются/десериализуются.

**Четыре модели взаимодействия в gRPC**:
1. **Unary** — классический запрос/ответ, как в REST.
2. **Server Streaming** — клиент отправляет один запрос, сервер возвращает поток сообщений.
3. **Client Streaming** — клиент отправляет поток сообщений, сервер возвращает один ответ.
4. **Bidirectional Streaming** — обе стороны обмениваются потоками сообщений независимо.

**Когда предпочтительнее gRPC**

1. **Внутренняя межсервисная коммуникация** (east-west трафик) — когда важна производительность и оба конца под нашим контролем.
2. **Высоконагруженные системы с низкой latency** — финансовые платформы, real-time биддинг, игровые серверы.
3. **Потоковая передача данных** — телеметрия, лог-агрегация, события в реальном времени.
4. **Polyglot-архитектуры** — gRPC генерирует типобезопасные клиенты для 10+ языков из одного `.proto`-файла, гарантируя единообразный контракт.
5. **Большие объёмы передаваемых данных** — бинарный формат существенно экономит трафик.

**Когда предпочтительнее REST**

1. **Публичные API** — широкая поддержка браузерами, инструментами, понятность для внешних разработчиков.
2. **Простые CRUD-операции** — избыточность gRPC не оправдана.
3. **Необходимость человекочитаемого формата** — отладка, интеграция без кодогенерации.
4. **Команды с разным уровнем подготовки** — порог входа в REST ниже.

**Практическая рекомендация**: многие современные архитектуры применяют гибридный подход — REST для внешних (north-south) API и gRPC для внутреннего (east-west) межсервисного взаимодействия.
### Вопрос 7. Что такое паттерн Circuit Breaker? Опишите три состояния автоматического выключателя (Closed, Open, Half-Open) и как они работают в Resilience4j.

**Категория:** Микросервисы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [CircuitBreaker — resilience4j](https://resilience4j.readme.io/docs/circuitbreaker)

Circuit Breaker в Resilience4j реализован в виде конечного автомата (finite state machine) с тремя основными состояниями: CLOSED, OPEN и HALF_OPEN, а также тремя специальными: METRICS_ONLY, DISABLED и FORCED_OPEN.

**CLOSED (Закрыт)** — нормальный режим работы. Все вызовы проходят к защищаемому сервису. Circuit Breaker отслеживает результаты вызовов через скользящее окно (sliding window). Когда доля неуспешных вызовов достигает или превышает настраиваемый порог (по умолчанию 50%), автоматический выключатель переходит в состояние OPEN.

**OPEN (Открыт)** — сервис считается недоступным. Все входящие вызовы немедленно отклоняются с исключением `CallNotPermittedException` без реального обращения к защищаемому сервису. Автоматический выключатель остаётся в этом состоянии в течение заданного интервала (`waitDurationInOpenState`, по умолчанию 60 000 мс), после чего переходит в состояние HALF_OPEN.

**HALF_OPEN (Полуоткрыт)** — тестовое состояние. Пропускается ограниченное число вызовов (`permittedNumberOfCallsInHalfOpenState`, по умолчанию 10), чтобы проверить, восстановился ли сервис. Если метрики этих вызовов остаются в допустимых пределах — переход обратно в CLOSED; если нет — снова в OPEN.

Скользящее окно может быть двух типов:
- **COUNT_BASED** — учитывает последние N вызовов (кольцевой буфер, O(1));
- **TIME_BASED** — учитывает вызовы за последние N секунд (частичные агрегации, O(1)).

Помимо доли ошибок, Circuit Breaker умеет переходить в OPEN и по доле «медленных» вызовов (`slowCallRateThreshold`), превышающих заданный порог по времени выполнения.

#### Ответ от Claude

**Паттерн Circuit Breaker** («автоматический выключатель») защищает систему от каскадных отказов: когда зависимый сервис деградирует или недоступен, Circuit Breaker прекращает передавать ему вызовы, немедленно возвращая ошибку. Это освобождает ресурсы вызывающей стороны и даёт проблемному сервису время на восстановление.

**Три основных состояния в Resilience4j:**

| Состояние | Поведение | Переход |
|-----------|-----------|---------|
| **CLOSED** | Вызовы проходят к сервису; ошибки и медленные ответы фиксируются в скользящем окне | → OPEN, когда `failureRateThreshold` или `slowCallRateThreshold` достигнут при наличии не менее `minimumNumberOfCalls` вызовов |
| **OPEN** | Все вызовы немедленно отклоняются с `CallNotPermittedException` | → HALF_OPEN автоматически по истечении `waitDurationInOpenState` |
| **HALF_OPEN** | Пропускается ровно `permittedNumberOfCallsInHalfOpenState` вызовов для зондирования | → CLOSED при успехе; → OPEN при повторном превышении порога |

**Ключевые параметры конфигурации (значения по умолчанию):**
- `failureRateThreshold` = 50% — порог доли ошибок
- `slidingWindowType` = COUNT_BASED — тип окна
- `slidingWindowSize` = 100 — размер окна (вызовы или секунды)
- `minimumNumberOfCalls` = 100 — минимум вызовов для расчёта метрик
- `waitDurationInOpenState` = 60 000 мс — время ожидания в OPEN
- `permittedNumberOfCallsInHalfOpenState` = 10 — число тестовых вызовов

**Специальные состояния:** DISABLED (Circuit Breaker выключен, всё пропускается), FORCED_OPEN (принудительно OPEN), METRICS_ONLY (метрики собираются, но пороги не срабатывают).

**Практический пример (Spring Boot + Resilience4j):**
```java
@CircuitBreaker(name = "paymentService", fallbackMethod = "fallback")
public String callPaymentService() { ... }

public String fallback(CallNotPermittedException ex) {
    return "Service temporarily unavailable";
}
```

В `application.yml`:
```yaml
resilience4j:
  circuitbreaker:
    instances:
      paymentService:
        failureRateThreshold: 50
        waitDurationInOpenState: 30s
        permittedNumberOfCallsInHalfOpenState: 5
        slidingWindowSize: 10
```

Паттерн Circuit Breaker рекомендуется применять совместно с Retry (повторные попытки) и Bulkhead (изоляция пулов потоков), образуя эшелонированную защиту.

### Вопрос 8. Что такое Bounded Context в Domain-Driven Design (DDD) и как это понятие помогает определять границы микросервисов?

**Категория:** Микросервисы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [bliki: Bounded Context — martinfowler.com](https://martinfowler.com/bliki/BoundedContext.html)

Bounded Context («ограниченный контекст») — центральный паттерн стратегического DDD, работающий с крупными моделями и большими командами. Основная идея: полная унификация доменной модели для большой системы нецелесообразна и экономически невыгодна. Вместо одной глобальной модели DDD разделяет сложный домен на несколько Bounded Context и описывает связи между ними.

Главный фактор, определяющий границы контекстов, — человеческая культура и язык: модели служат единым языком (Ubiquitous Language) между разработчиками и экспертами предметной области, и при смене этого языка возникает новый контекст. Например, термин «customer» («клиент») имеет разный смысл в отделе продаж и в отделе поддержки — это признак двух разных контекстов.

Технические различия в представлении данных (in-memory-объект против реляционной таблицы) также могут быть основанием для выделения отдельного контекста. Для связи между контекстами используются механизмы отображения (context map), описывающие, как понятия одного контекста переводятся в понятия другого.

#### Ответ от Claude

**Bounded Context** — это явная граница, внутри которой конкретная доменная модель строго согласована и имеет однозначный смысл. За пределами этой границы та же сущность (например, «Drone» или «Order») может выглядеть совершенно иначе и содержать другой набор атрибутов.

**Как Bounded Context помогает определять границы микросервисов:**

1. **Принцип соответствия:** Каждый Bounded Context является кандидатом на отдельный микросервис. Microsoft рекомендует: «проектируй микросервис не меньше агрегата и не больше Bounded Context».

2. **Ubiquitous Language (единый язык):** Внутри контекста разработчики и эксперты предметной области используют один и тот же язык в коде, документации и разговорах. Разные значения одного слова в разных отделах — сигнал к выделению отдельного контекста.

3. **Изоляция моделей:** Drone в микросервисе управления дронами содержит техническое состояние, историю обслуживания и характеристики. Drone в микросервисе планирования доставки содержит только доступность и ожидаемое время прибытия. Один и тот же реальный объект имеет две разные модели в двух контекстах.

4. **Context Map:** Явная карта контекстов описывает интеграционные точки и отношения между ними. Основные паттерны отношений:
   - **Customer-Supplier** — один контекст (upstream) поставляет данные другому (downstream)
   - **Anti-Corruption Layer** — защитный слой трансляции, ограждающий модель от изменений во внешнем контексте
   - **Open Host Service / Published Language** — открытый API с хорошо определённым форматом для downstream-потребителей
   - **Separate Ways** — контексты не интегрируются, развиваются независимо

5. **Классификация поддоменов** (помогает расставить приоритеты):
   - **Core subdomain** — ключевое конкурентное преимущество, требует глубокой проработки DDD
   - **Supporting subdomain** — необходим для работы бизнеса, но не даёт конкурентного преимущества
   - **Generic subdomain** — стандартная задача, можно использовать готовое решение (например, IAM-системы)

**Практические признаки правильно выделенного Bounded Context:**
- Микросервис может разворачиваться независимо
- Изменение бизнес-логики внутри контекста не требует изменений в других контекстах
- Команда, владеющая контекстом, понимает его полностью без координации с другими командами
- Схема данных и база данных принадлежат только этому контексту

**Анти-паттерн:** нарушение границ возникает, когда несколько микросервисов разделяют одну базу данных или когда изменение в одном сервисе неизбежно требует изменений в другом — это признак неверно проведённых границ.

### Вопрос 9. Что такое eventual consistency (итоговая согласованность)? Чем она отличается от строгой согласованности и когда её применение оправдано?

**Категория:** Микросервисы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Microservice Trade-Offs — martinfowler.com](https://martinfowler.com/articles/microservice-trade-offs.html)

В монолитных системах с традиционными транзакциями обновления множества объектов происходят атомарно — всё вместе или ничего. Микросервисы отказываются от этого подхода, поскольку «распределённые транзакции нежелательны (и тому есть веские причины)». Вместо этого разработчики должны самостоятельно обнаруживать рассогласованное состояние данных до того, как код совершит действие, о котором потом «пожалеет».

Автор иллюстрирует итоговую согласованность бытовым примером: «Вы делаете обновление, обновляете экран — изменения нет. Ждёте минуту-две, снова нажимаете обновить — вот оно». Бизнес часто ставит доступность выше согласованности, поэтому итоговая согласованность соответствует реальным бизнес-приоритетам, хотя и создаёт временные расхождения данных, которыми разработчик обязан активно управлять.

#### Ответ от Claude

**Eventual consistency (итоговая согласованность)** — модель согласованности данных в распределённых системах, гарантирующая, что при отсутствии новых обновлений все реплики данных в конечном итоге сойдутся к одному значению. Временное расхождение допустимо; промежуток сходимости зависит от архитектуры системы.

**Сравнение с сильной (строгой) согласованностью:**

| Критерий | Сильная согласованность | Итоговая согласованность |
|----------|------------------------|--------------------------|
| Гарантия | Каждое чтение получает последнее записанное значение | Данные в конце концов синхронизируются, но не мгновенно |
| Задержка | Выше (нужна координация между узлами) | Ниже (запись не ждёт подтверждения всех реплик) |
| Доступность | Ниже (CAP-теорема: C vs A при сбое сети) | Выше (система продолжает работать при сетевых разделениях) |
| Сложность | Ниже для разработчика | Выше (надо обрабатывать конфликты и временные расхождения) |
| Примеры | Банковские транзакции, РСУБД + 2PC | Amazon DynamoDB, Apache Cassandra, очереди сообщений |

**Теорема CAP и её связь с согласованностью:**
Согласно теореме CAP (Brewer, 2000), распределённая система не может одновременно обеспечить: Consistency (согласованность), Availability (доступность) и Partition tolerance (устойчивость к разделению сети). Поскольку сетевые разделения неизбежны, системы выбирают между CP (согласованность + отказоустойчивость) и AP (доступность + отказоустойчивость). AP-системы принимают итоговую согласованность.

**BASE vs ACID:**
- **ACID** (Atomicity, Consistency, Isolation, Durability) — модель реляционных БД, гарантирующая строгую согласованность
- **BASE** (Basically Available, Soft state, Eventually consistent) — модель распределённых NoSQL-систем, принимающая итоговую согласованность ради доступности

**Почему микросервисы вынуждены принимать итоговую согласованность:**
1. Каждый микросервис владеет собственной базой данных (Database-per-Service)
2. Распределённые транзакции (2PC) нежелательны из-за высокой связанности и риска блокировок
3. Обновление данных в нескольких сервисах происходит через события/сообщения (асинхронно)

**Паттерны для управления итоговой согласованностью:**
- **Saga Pattern** — цепочка локальных транзакций; при сбое выполняются компенсирующие транзакции (rollback через обратные действия)
- **Outbox Pattern** — атомарная запись в БД + таблица исходящих событий, гарантирующая «хотя бы однократную» доставку сообщений
- **CQRS** — разделение модели записи и модели чтения; реплика для чтения обновляется асинхронно

**Когда итоговая согласованность оправдана:**
- Высокая доступность важнее немедленной точности данных (интернет-магазины, социальные сети, системы аналитики)
- Система рассчитана на глобальное распределение с репликацией в нескольких регионах
- Нагрузка слишком велика для синхронной координации

**Когда итоговая согласованность НЕ подходит:**
- Финансовые операции, требующие строгой атомарности (перевод денег между счетами)
- Медицинские системы, где устаревшие данные могут причинить вред
- Операции с инвентарём, где двойная продажа недопустима

**Практическое замечание:** разработчик, работающий с итоговой согласованностью, обязан явно обрабатывать сценарии временного расхождения: идемпотентность обработчиков событий, механизмы обнаружения конфликтов и компенсирующие операции.

### Вопрос 10. Что такое Trace ID и Span ID? Зачем они нужны в распределённой трассировке и какие инструменты (Zipkin, Jaeger, Micrometer Tracing) используются в экосистеме Spring?

**Категория:** Микросервисы · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Tracing :: Spring Boot](https://docs.spring.io/spring-boot/reference/actuator/tracing.html)

Spring Boot предоставляет управление зависимостями и автоконфигурацию для Micrometer Tracing — фасада над популярными библиотеками трассировки. Micrometer Tracing играет роль, аналогичную SLF4J для логирования: предоставляет единый API, позволяя при этом выбирать конкретную реализацию.

**Trace ID** — уникальный идентификатор всего распределённого потока запроса через несколько сервисов. **Span ID** — уникальный идентификатор единичной операции внутри трейса. Оба значения автоматически добавляются в логи через MDC (Mapped Diagnostic Context). Формат по умолчанию в логах: `[{traceId}-{spanId}]`, например `[803B448A0489F84084905D3093480352-3425F23BB2432450]`.

Spring Boot поставляет автоконфигурацию для двух реализаций:
- **OpenTelemetry с OTLP** (зависимость `spring-boot-starter-opentelemetry`)
- **OpenZipkin Brave с Zipkin** (зависимость `spring-boot-starter-zipkin`)

Для трассировки 100% запросов необходимо выставить `management.tracing.sampling.probability: 1.0`.

Автоматическое распространение контекста трассировки работает только если HTTP-клиенты (`RestTemplate`, `WebClient`) созданы через автоконфигурированные билдеры Spring Boot. При создании вручную (`new RestTemplate()`) контекст не распространяется.

#### Ответ от Claude

**Базовые понятия**

Распределённая трассировка (distributed tracing) — механизм наблюдаемости, позволяющий отследить полный путь запроса через несколько микросервисов.

- **Trace** — полный путь запроса от входной точки до финального ответа, охватывающий все сервисы.
- **Span** — единица работы внутри трейса: один HTTP-вызов, запрос к БД, вызов внешнего API. Спаны могут быть вложенными (parent span / child span).
- **Trace ID** — уникальный идентификатор, общий для всех спанов одного трейса. Передаётся между сервисами в заголовках HTTP (W3C `traceparent` или B3-заголовки Zipkin).
- **Span ID** — уникальный идентификатор конкретного спана. У каждого спана есть свой Span ID и (кроме корневого) Parent Span ID.

Пример: пользователь отправляет запрос в API Gateway — формируется Trace ID. Когда Gateway вызывает Order Service, он передаёт тот же Trace ID и свой Span ID как Parent Span ID. Order Service создаёт дочерний спан. Таким образом, вся цепочка вызовов связана одним Trace ID.

**Зачем это нужно**

1. Диагностика задержек: в UI Zipkin/Jaeger можно увидеть waterfall-диаграмму и точно определить, какой сервис или вызов к БД тормозит.
2. Анализ ошибок: по Trace ID можно собрать все логи из всех сервисов, относящиеся к одному запросу.
3. Построение карты зависимостей сервисов (Service Dependency Graph).

**Инструменты в экосистеме Spring**

| Инструмент | Роль |
|---|---|
| **Micrometer Tracing** | Фасад (абстракция) над реализациями трассировки — аналог SLF4J. Встроен в Spring Boot 3+. |
| **OpenZipkin Brave** | Реализация трассировки; экспортирует спаны в **Zipkin** (UI для просмотра трейсов). |
| **OpenTelemetry (OTel)** | Современный стандарт; экспортирует через OTLP в **Jaeger**, Grafana Tempo, и др. |
| **Zipkin** | Система сбора и визуализации трейсов. Проект OpenZipkin. |
| **Jaeger** | Система трассировки от CNCF, поддерживает OTel. Предпочтительна для Kubernetes-окружений. |

**Как работает в Spring Boot 3+**

Spring Boot 3+ включает Micrometer Tracing из коробки. Достаточно добавить зависимость моста (bridge) к нужному бэкенду:

```xml
<!-- Для Zipkin/Brave -->
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-tracing-bridge-brave</artifactId>
</dependency>
<dependency>
    <groupId>io.zipkin.reporter2</groupId>
    <artifactId>zipkin-reporter-brave</artifactId>
</dependency>

<!-- Для OpenTelemetry / Jaeger -->
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-tracing-bridge-otel</artifactId>
</dependency>
<dependency>
    <groupId>io.opentelemetry</groupId>
    <artifactId>opentelemetry-exporter-otlp</artifactId>
</dependency>
```

Spring Boot автоматически:
- Присваивает каждому входящему запросу Trace ID и Span ID.
- Добавляет их в MDC — они отображаются в каждой строке логов.
- Распространяет контекст в исходящих HTTP-вызовах (через автоконфигурированные `RestTemplate`/`WebClient`) и в Kafka-сообщениях.

**Важные нюансы**

- `management.tracing.sampling.probability` управляет процентом трассируемых запросов (по умолчанию `0.1` — 10%; для разработки ставят `1.0`).
- Baggage — механизм передачи произвольных пользовательских данных (например, tenant ID) вместе с контекстом трассировки между сервисами.
- До Spring Boot 3 для этих целей использовался **Spring Cloud Sleuth**, который теперь устарел и заменён Micrometer Tracing.

### Вопрос 11. Что такое паттерн Saga и для чего он применяется? Опишите разницу между хореографией (choreography) и оркестровкой (orchestration) в контексте Saga.

**Категория:** Микросервисы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Saga choreography pattern — AWS Prescriptive Guidance](https://docs.aws.amazon.com/prescriptive-guidance/latest/cloud-design-patterns/saga-choreography.html)

Паттерн Saga помогает сохранить целостность данных в распределённых транзакциях, охватывающих несколько сервисов, используя подписку на события. В распределённой системе, где сервисы хранят данные в разных хранилищах, поддерживать согласованность данных между ними крайне сложно.

В реляционных базах данных для атомарности используется 2PC (двухфазный коммит): фаза подготовки и фаза фиксации. В распределённых системах с паттерном «база данных на сервис» 2PC неприменим: каждая транзакция распределена по разным базам данных и нет единого координатора. Решением является паттерн Saga.

**Хореография (Choreography):** сервисы реагируют на события других сервисов. Каждый участник публикует событие после завершения локальной транзакции, а следующий сервис подписывается на это событие и начинает свою транзакцию. При сбое запускаются компенсирующие транзакции.

Проблемы хореографии: по мере роста числа микросервисов хореография усложняется, возникают циклические зависимости, сложнее реализовать глобальные паттерны отказоустойчивости (таймауты, повторы). End-to-end мониторинг хуже, чем при оркестровке.

#### Ответ от Claude

**Что такое Saga и зачем она нужна**

В архитектуре микросервисов каждый сервис владеет собственной базой данных (паттерн Database per Service). Это исключает возможность использования классических ACID-транзакций через несколько сервисов. Традиционный способ обеспечения атомарности — двухфазный коммит (2PC) — неприменим в такой среде (подробнее в вопросе №12).

**Saga** — паттерн управления распределёнными транзакциями, при котором длинная бизнес-транзакция разбивается на последовательность локальных транзакций. Каждая локальная транзакция обновляет базу данных своего сервиса и инициирует следующий шаг (через событие или команду). Если один шаг завершается неудачей, запускаются **компенсирующие транзакции** (compensating transactions) для всех предыдущих успешных шагов — это логический откат, а не физический ROLLBACK.

**Пример:** оформление заказа в интернет-магазине:
1. Order Service: создать заказ (статус PENDING)
2. Inventory Service: зарезервировать товар
3. Payment Service: списать деньги
4. Order Service: подтвердить заказ (статус CONFIRMED)

Если шаг 3 (оплата) завершается неудачей — Order Service отменяет заказ, Inventory Service освобождает резервацию.

**Хореография (Choreography)**

Децентрализованный подход: нет координатора, каждый сервис реагирует на события других сервисов.

- Order Service публикует событие `OrderCreated`
- Inventory Service подписан на `OrderCreated`, резервирует товар, публикует `InventoryReserved`
- Payment Service подписан на `InventoryReserved`, списывает деньги, публикует `PaymentProcessed`
- Order Service подписан на `PaymentProcessed`, подтверждает заказ

Транспорт: обычно брокер сообщений (Kafka, RabbitMQ).

Плюсы:
- Слабая связанность сервисов
- Нет единой точки отказа
- Хорошо масштабируется
- Проще добавить нового участника без изменения существующих

Минусы:
- Трудно отслеживать общий flow транзакции
- Сложно реализовать сложную бизнес-логику и условные переходы
- Риск циклических зависимостей
- Тяжело отлаживать и тестировать
- Бизнес-логика размазана по нескольким сервисам

**Оркестровка (Orchestration)**

Централизованный подход: существует выделенный **оркестратор** (Saga Orchestrator), который координирует весь процесс, посылая командные запросы участникам и ожидая ответов.

- Orchestrator отправляет команду `ReserveInventory` в Inventory Service
- Получает ответ `InventoryReserved`
- Отправляет команду `ProcessPayment` в Payment Service
- Получает ответ `PaymentProcessed` или `PaymentFailed`
- При `PaymentFailed` отправляет команду `ReleaseInventory`

Технологии: Apache Camel, Axon Framework, Temporal, AWS Step Functions, самописный оркестратор через Kafka + State Machine.

Плюсы:
- Единое место для бизнес-логики транзакции — легко понять flow
- Проще реализовывать сложные условия и обработку ошибок
- Лучшая наблюдаемость (observability)
- Упрощённый код в отдельных сервисах

Минусы:
- Оркестратор — потенциальная единая точка отказа (решается через отказоустойчивость оркестратора)
- Оркестратор может стать «God service» и начать знать слишком много о других сервисах
- Более сложная первоначальная реализация

**Сравнительная таблица**

| Критерий | Хореография | Оркестровка |
|---|---|---|
| Управление | Децентрализованное | Централизованное |
| Связанность | Низкая | Средняя (через оркестратор) |
| Сложность flow | Неявный, распределён | Явный, в одном месте |
| Отладка | Сложная | Проще |
| Масштабируемость | Высокая | Зависит от оркестратора |
| Единая точка отказа | Нет | Оркестратор (решаемо) |
| Подходит для | Простых, независимых потоков | Сложных бизнес-процессов |

**Ключевые требования к реализации**

- **Идемпотентность**: каждый участник должен корректно обрабатывать повторные вызовы.
- **Компенсирующие транзакции**: должны быть определены для каждого шага.
- **Transactional Outbox Pattern**: решает проблему dual write — атомарная запись в БД и публикация события.
- **Отсутствие изоляции**: Saga не обеспечивает ACID-изоляцию; необходимо учитывать аномалии (dirty reads, lost updates).

### Вопрос 12. Как обеспечить согласованность данных в системе, где каждый микросервис владеет своей базой данных? Почему нежелательно использовать распределённые транзакции (2PC)?

**Категория:** Микросервисы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Data Consistency in Microservices Architecture — DZone](https://dzone.com/articles/data-consistency-in-microservices-architecture)

В распределённых системах атомарная операция нередко охватывает несколько микросервисов. Когда данные хранятся в разных независимых хранилищах, сбои могут привести систему в несогласованное состояние (например, деньги списаны, но заказ не создан).

2PC пытается обеспечить атомарное обновление нескольких баз данных, однако плохо работает в современных облачных средах. В индустрии нет широко признанного решения для атомарного обновления данных в нескольких разнородных источниках данных.

Вместо этого инженеры принимают модель BASE (Basically Available, Soft state, Eventually consistent) и берут на себя обеспечение согласованности вручную. Наиболее устоявшийся подход — паттерн Saga. Также рекомендуются:

- **Change Data Capture (CDC)**: инструменты вроде Debezium перехватывают изменения в БД и публикуют события.
- **Event-first архитектуры**: события становятся единственным источником истины.
- **Сверка (Reconciliation)**: периодическое сравнение записей между сервисами для обнаружения несоответствий.

Рекомендация: минимизируйте необходимость в распределённой согласованности через правильное проектирование границ сервисов.

#### Ответ от Claude

**Проблема: Database per Service**

Паттерн «база данных на сервис» (Database per Service) — один из ключевых принципов микросервисной архитектуры. Он обеспечивает независимость, слабую связанность и возможность независимого деплоя. Однако он порождает главную сложность: как сохранить согласованность данных, когда одна бизнес-операция затрагивает несколько сервисов?

**Почему нежелательно использовать 2PC (Two-Phase Commit)**

2PC — протокол атомарного коммита транзакций в распределённых системах:
1. **Фаза подготовки (Prepare):** координатор спрашивает всех участников «готов ли ты закоммитить?»
2. **Фаза фиксации (Commit):** если все ответили «да», координатор отправляет команду Commit; если хоть один ответил «нет» — всем Rollback.

**Проблемы 2PC в микросервисной архитектуре:**

1. **Блокирующий протокол (Blocking Protocol).** Во время выполнения 2PC все участники удерживают блокировки на ресурсах вплоть до получения решения от координатора. Это резко снижает пропускную способность и увеличивает задержки.

2. **Единая точка отказа (Single Point of Failure).** Если координатор упадёт между фазой подготовки и фазой коммита, участники зависнут с удерживаемыми блокировками — неопределённо долго.

3. **Низкая доступность (Availability).** Согласно теореме CAP, в условиях сетевого разделения (partition) система вынуждена выбирать между согласованностью (consistency) и доступностью (availability). 2PC выбирает согласованность ценой доступности.

4. **Технологическая привязка.** Классический XA/JTA 2PC требует, чтобы все участники поддерживали XA-протокол и находились под управлением единого транзакционного менеджера. Разнородные хранилища (PostgreSQL + MongoDB + Kafka) не могут участвовать в одной XA-транзакции.

5. **Производительность.** 2PC существенно медленнее локальных транзакций из-за сетевых round-trip'ов между координатором и участниками.

**Стратегии обеспечения согласованности**

**1. Паттерн Saga (основной подход)**

Разбивает длинную транзакцию на последовательность локальных транзакций с компенсирующими операциями (см. вопрос №11). Принимает **eventual consistency** — временную несогласованность, которая со временем устраняется.

**2. Transactional Outbox Pattern**

Решает проблему «dual write» — невозможности атомарно обновить БД и опубликовать сообщение в брокер:
- Сервис пишет запись в таблицу `outbox` той же БД в одной локальной транзакции с основными данными.
- Отдельный процесс (relay) читает записи из outbox и публикует их в брокер (Kafka/RabbitMQ).
- Инструменты CDC (Debezium) автоматизируют этот процесс через чтение WAL (Write-Ahead Log).

**3. Event Sourcing**

Вместо хранения текущего состояния хранятся все события (факты), которые к нему привели. Event Store является единственным источником истины. Любой сервис может воспроизвести (replay) события для восстановления своего состояния.

**4. Change Data Capture (CDC)**

Инструменты типа Debezium отслеживают изменения в WAL базы данных и публикуют их как события в Kafka. Позволяет сервисам реагировать на изменения в других сервисах без явного API-взаимодействия.

**5. Idempotent APIs + Retry**

Все операции записи должны быть идемпотентными (повторный вызов с тем же Idempotency Key не создаёт дубликатов). Это позволяет безопасно повторять запросы при временных сбоях.

**6. Тщательное проектирование границ сервисов**

Наилучшая стратегия — минимизировать необходимость в межсервисной согласованности. Если операции чаще всего принадлежат одному ограниченному контексту (Bounded Context), их следует разместить в одном сервисе с одной базой данных.

**Модель BASE как альтернатива ACID**

| Свойство | ACID | BASE |
|---|---|---|
| Согласованность | Строгая, немедленная | Eventual (со временем) |
| Доступность | Может снижаться ради консистентности | Приоритет доступности |
| Изоляция | Полная | Отсутствует (управляется явно) |
| Применимость | Монолит, одна БД | Распределённые системы |

**Итог:** в микросервисной архитектуре распределённые транзакции (2PC) заменяются на комбинацию паттернов: Saga + Transactional Outbox + Idempotency. Это сложнее, чем ACID, но единственный практичный путь к масштабируемой и доступной системе.
### Вопрос 13. Что такое паттерн Outbox и как он помогает решить проблему надёжной публикации событий совместно с обновлением базы данных?

**Категория:** Микросервисы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Microservices Pattern: Transactional Outbox](https://microservices.io/patterns/data/transactional-outbox.html)

Сервис, публикующий сообщения, сначала сохраняет сообщение в базе данных в рамках той же транзакции, которая обновляет бизнес-сущности. Отдельный процесс затем читает эти сообщения и отправляет их в брокер.

**Проблема:** Сервисам часто нужно атомарно обновить БД и опубликовать сообщение (например, при участии в sagах или при публикации доменных событий). Использование распределённых транзакций (2PC) ненадёжно и создаёт нежелательную связанность. Без 2PC существует риск, что транзакция зафиксируется без отправки сообщения, или приложение упадёт до его доставки.

**Архитектура паттерна** включает четыре компонента:
- **Sender** — сервис, обновляющий бизнес-сущности
- **Database** — хранит как сущности, так и outbox
- **Message outbox** — таблица (реляционная БД) или свойство документа (NoSQL) с ожидающими отправки сообщениями
- **Message relay** — считывает и публикует сообщения из outbox в брокер

**Два способа реализации Message Relay:**
1. **Polling Publisher** — периодически опрашивает таблицу outbox на наличие новых сообщений. Работает с любой SQL-базой, но плохо масштабируется и не поддерживается всеми NoSQL.
2. **Transaction Log Tailing** — подписывается на журнал транзакций СУБД (например, MySQL binlog, PostgreSQL WAL, MongoDB oplog) и публикует каждое сообщение, вставленное в outbox.

**Недостатки:** разработчик может забыть опубликовать сообщение после обновления; Message Relay может опубликовать сообщение более одного раза при сбое, поэтому потребители должны быть идемпотентными.

#### Ответ от Claude

**Суть проблемы**

В микросервисной архитектуре часто возникает задача: нужно атомарно и в одном сервисе *сохранить изменения в БД* и *отправить событие в брокер сообщений* (Kafka, RabbitMQ и др.). Наивный подход — сначала commit в БД, потом publish в брокер — ненадёжен: между двумя операциями сервис может упасть, и событие не будет отправлено. Обратный порядок (сначала publish, потом commit) также ненадёжен: БД может откатиться после успешной публикации. Распределённые транзакции (2PC) технически решают задачу, но создают сильную связанность и снижают доступность.

**Паттерн Transactional Outbox**

Решение: включить публикацию сообщения в ту же локальную транзакцию БД, что и обновление данных.

1. В одной транзакции сервис:
   - обновляет бизнес-сущность (например, `Orders`),
   - вставляет запись в специальную таблицу `OUTBOX` (содержит: id, aggregate\_type, aggregate\_id, event\_type, payload).
2. Отдельный процесс — **Message Relay** — периодически читает `OUTBOX` и публикует сообщения в брокер.
3. После успешной публикации запись в `OUTBOX` помечается как отправленная (или удаляется).

**Гарантия:** сообщение будет опубликовано тогда и только тогда, когда транзакция БД зафиксирована. Если сервис упадёт после commit, но до отправки — Message Relay подберёт запись при следующем запуске.

**Два варианта Message Relay:**

| Подход | Как работает | Плюсы | Минусы |
|---|---|---|---|
| **Polling Publisher** | Периодически делает `SELECT` по таблице OUTBOX | Прост в реализации, совместим с любой SQL-СУБД | Возможная задержка, нагрузка на БД, сложно поддерживать порядок |
| **Transaction Log Tailing (CDC)** | Читает WAL/binlog/oplog СУБД через инструменты типа Debezium | Минимальная задержка, не нагружает приложение, надёжный порядок | Сложнее в настройке, зависит от СУБД |

**Важно:** Message Relay может доставить одно сообщение более одного раза (at-least-once delivery), поэтому потребители **обязаны** реализовывать идемпотентность (см. паттерн Idempotent Consumer).

**Готовые реализации:** Debezium (CDC для PostgreSQL/MySQL/MongoDB), Eventuate Tram, Axon Framework, Transactional Outbox в Spring с `@Transactional`.

**Связанные паттерны:** Saga, Domain Events, Idempotent Consumer, Transaction Log Tailing.

### Вопрос 14. Как вы определяете правильный размер и границы микросервиса? Какие принципы DDD (Aggregate, Entity, Value Object) вы используете при проектировании?

**Категория:** Микросервисы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [bliki: DDD_Aggregate — Martin Fowler](https://martinfowler.com/bliki/DDD_Aggregate.html)

DDD Aggregate — это кластер доменных объектов, которые можно рассматривать как единое целое. Пример: заказ (Order) со строками заказа (Line Items) — это отдельные объекты, объединённые в один агрегат. Каждый агрегат имеет корень (Aggregate Root) и границу. Корень — единственный участник агрегата, на который могут держать ссылки внешние объекты. Все внешние запросы проходят только через корень, что позволяет обеспечить целостность всего агрегата.

Ключевое правило: **транзакции не должны пересекать границы агрегата**. Это правило напрямую определяет границы микросервисов: каждый агрегат — это логическая единица согласованности. Агрегаты — базовый элемент хранения и передачи данных.

#### Ответ от Claude

**Стратегический уровень: Bounded Context как граница сервиса**

Первый шаг при проектировании — разбить предметную область на **Bounded Context'ы** (Ограниченные контексты). Bounded Context — это явная граница, внутри которой модель предметной области единообразна и имеет собственный «Ubiquitous Language» (единый язык). Классические сигналы необходимости разграничения: одно и то же слово («Продукт», «Клиент») означает разные вещи в разных частях бизнеса.

Каждый Bounded Context — потенциальный кандидат на отдельный микросервис. Разбиение по **бизнес-возможностям** (decompose by business capability) и **поддоменам DDD** (core / supporting / generic subdomain) — два взаимодополняющих способа найти эти границы.

**Тактический уровень: DDD-строительные блоки**

| Концепция | Определение | Роль в проектировании сервиса |
|---|---|---|
| **Entity** | Объект с уникальной идентичностью (ID), изменяемый со временем | `Order`, `Customer` — сущности имеют жизненный цикл |
| **Value Object** | Объект без идентичности, определяется своими атрибутами, неизменяем | `Money`, `Address`, `DateRange` — копируются, не ссылаются |
| **Aggregate** | Кластер Entity и Value Object с единым корнем (Aggregate Root) | Единица транзакционной согласованности |
| **Aggregate Root** | Единственная точка входа в агрегат извне | Контролирует инварианты всего агрегата |
| **Domain Service** | Операция, не принадлежащая ни одному агрегату | Вычисления, затрагивающие несколько агрегатов |
| **Repository** | Абстракция над хранением агрегатов | Один репозиторий — один агрегат |

**Правила определения границ агрегата:**

1. **Транзакционная согласованность:** всё, что должно быть согласовано в рамках одной транзакции, — в одном агрегате. Если нужна транзакция между двумя объектами, они, вероятно, один агрегат.
2. **Маленькие агрегаты:** чем меньше агрегат, тем лучше масштабируемость. Предпочитайте ссылки по ID вместо включения вложенных агрегатов.
3. **Одна транзакция — один агрегат:** если бизнес-операция затрагивает несколько агрегатов, используйте eventual consistency (Saga, Domain Events).
4. **Каждый агрегат — потенциальный микросервис:** в идеале сервис владеет одним или несколькими связанными агрегатами, но не разделяет агрегат между сервисами.

**Практические принципы выбора размера сервиса:**

- **Single Responsibility:** сервис отвечает за одну бизнес-возможность.
- **Высокая связность внутри, слабая между сервисами:** всё, что часто меняется вместе — в одном сервисе.
- **Автономия:** сервис может быть развёрнут, масштабирован и обновлён независимо.
- **Избегайте chatty interactions:** если два «сервиса» постоянно синхронно вызывают друг друга — это признак неправильного разбиения.
- **Команда:** принцип «пицца-команды» (2-pizza team) — хорошая эвристика размера.
- **Декомпозиция по поддоменам:** Core Domain (то, что отличает бизнес) — заслуживает отдельного, тщательно спроектированного сервиса; Generic subdomain (авторизация, уведомления) — можно использовать готовое решение.

**Пример:** для интернет-магазина: `Order` (корень) + `OrderLineItem` (Entity внутри агрегата) + `Money` (Value Object) образуют агрегат `Order`. Агрегат `Customer` — отдельный. Сервис `OrderService` владеет агрегатом `Order`, `CustomerService` — агрегатом `Customer`. Взаимодействие между ними — через события (OrderPlaced → CustomerService обновляет статистику).

### Вопрос 15. Что такое идемпотентность в контексте микросервисов? Почему она критически важна при асинхронном взаимодействии и как её реализовать?

**Категория:** Микросервисы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Microservices Pattern: Idempotent Consumer](https://microservices.io/patterns/communication-style/idempotent-consumer.html)

Идемпотентность означает, что обработка одного и того же сообщения несколько раз даёт тот же результат, что и однократная обработка. Брокеры сообщений (Kafka, RabbitMQ) гарантируют доставку **at-least-once** для обеспечения надёжности. Это создаёт проблему: потребители могут получить и обработать дублирующие сообщения при сбоях или повторных попытках.

Без идемпотентности возникают серьёзные ошибки. Например: обработчик, вычитающий сумму дебета из баланса счёта, выдаст неверный результат, если его вызвать несколько раз для одного сообщения.

**Реализация через отдельную таблицу:** используется таблица `PROCESSED_MESSAGES` с составным первичным ключом `(subscriberId, messageID)`. Обработчик в рамках транзакции вставляет messageID; при дубликате INSERT завершится ошибкой, что позволяет откатить транзакцию и пропустить повторную обработку.

**Реализация через бизнес-сущности:** messageID сохраняется внутри самой бизнес-сущности. Это полезно для NoSQL-баз без поддержки многотабличных транзакций. Сервис `Order History` из приложения FTGO хранит максимальные event ID по каждому экземпляру агрегата, используя монотонный рост event ID.

#### Ответ от Claude

**Определение**

Операция называется **идемпотентной**, если её выполнение один или несколько раз даёт одинаковый результат. В контексте микросервисов: обработчик сообщения идемпотентен, если повторная обработка того же сообщения не изменяет конечное состояние системы.

**Почему это критически важно при асинхронном взаимодействии**

Брокеры сообщений (Kafka, RabbitMQ, AWS SQS) в большинстве случаев обеспечивают гарантию **at-least-once delivery** (доставить хотя бы раз). Это означает:
- Сообщение может быть доставлено несколько раз при сетевых сбоях.
- Consumer может упасть после обработки, но до подтверждения (ack) — брокер повторит доставку.
- При горизонтальном масштабировании и перебалансировке партиций Kafka сообщение может быть обработано дважды.

Если обработчик **не идемпотентен**, это приводит к: задвоению заказов, двойным списаниям с баланса, дублированию уведомлений, нарушению инвариантов бизнес-логики.

**Классификация операций по идемпотентности:**

| Тип операции | Идемпотентна? | Пример |
|---|---|---|
| `GET`, `SELECT` | Да | Чтение данных |
| `PUT` (установить значение) | Да | `balance = 100` |
| `DELETE by ID` | Да | Удалить запись с ID=5 |
| `POST` (создать) | Нет | Создать новый заказ |
| `PATCH` (приращение) | Нет | `balance -= 50` |

**Способы реализации идемпотентного потребителя:**

**1. Таблица обработанных сообщений (PROCESSED\_MESSAGES)**

```sql
CREATE TABLE processed_messages (
    subscriber_id VARCHAR(255) NOT NULL,
    message_id    VARCHAR(255) NOT NULL,
    processed_at  TIMESTAMP NOT NULL,
    PRIMARY KEY (subscriber_id, message_id)
);
```

Алгоритм обработчика (в одной транзакции с бизнес-логикой):
1. Начать транзакцию.
2. Попытаться вставить `(subscriberId, messageId)` в `PROCESSED_MESSAGES`.
3. Если INSERT завершился с ошибкой уникальности — сообщение уже обработано, откатить и подтвердить (ack).
4. Выполнить бизнес-логику.
5. Commit. Отправить ack.

**2. Хранение message ID в бизнес-сущности**

Для NoSQL или случаев, когда лишняя таблица нежелательна, можно хранить набор обработанных ID прямо в документе агрегата. Особенно эффективно, когда event ID монотонно возрастают (например, Kafka offset): достаточно хранить максимальный обработанный offset на партицию.

**3. Натуральная идемпотентность через дизайн**

Перепроектировать команду так, чтобы она была идемпотентна по природе:
- Вместо `"списать 50 руб."` (приращение) → `"установить баланс 950 руб."` (установка).
- Использовать `INSERT ... ON CONFLICT DO NOTHING` / `UPSERT`.
- Включать в сообщение ожидаемое текущее состояние (`expectedVersion` для оптимистичной блокировки).

**4. Idempotency Key (ключ идемпотентности)**

Клиент генерирует уникальный UUID для каждой бизнес-операции и передаёт его в заголовке сообщения (`messageId`, `idempotencyKey`). Сервер хранит этот ключ и при повторном запросе возвращает сохранённый результат вместо повторного выполнения операции. Этот подход применяется в REST API (Stripe, Twilio) и одинаково хорошо работает для HTTP и сообщений.

**Связанные паттерны:** Transactional Outbox (гарантирует at-least-once отправку, что делает идемпотентность потребителя обязательной), Saga (каждый шаг саги должен быть идемпотентным), Optimistic Locking (защищает от параллельных обновлений).

Паттерн Strangler Fig позволяет постепенно мигрировать устаревшую систему, последовательно заменяя отдельные функции новыми приложениями и сервисами. По мере замены функций новая система в конечном счёте включает все возможности старой, вытесняя её и позволяя вывести из эксплуатации.

**Суть решения.** Определив новые границы сервисов, используйте итеративный процесс замены конкретных функциональных блоков. Клиенты продолжают работать с тем же интерфейсом, не подозревая о миграции.

**Четыре фазы паттерна:**

1. Внедряется фасад (прокси) между клиентским приложением, легаси-системой и новой системой. Фасад действует как посредник. Изначально он направляет большинство запросов к легаси.
2. По мере продвижения миграции фасад постепенно переключает запросы от легаси к новой системе. С каждой итерацией в новой системе реализуется всё больше функциональности.
3. После переноса всей функциональности легаси выводится из эксплуатации. Фасад направляет все запросы исключительно к новой системе.
4. Фасад удаляется, клиент взаимодействует напрямую с новой системой.

**Когда применять:**
- Постепенная миграция крупной или сложной системы, при которой полная замена несёт высокий риск.
- Легаси-система может существовать продолжительное время в ходе миграции.

**Когда не применять:**
- Невозможно перехватить запросы к легаси-системе.
- Нет доступа к исходному коду легаси.
- Система мала и полная замена проста.
- Требуется быстрое выведение из эксплуатации.

**Ключевой риск — база данных.** Монолит обычно использует единую общую схему. При разделении базы применяется процесс «Extract → ETL → CDC (Change Data Capture) → cutover» для каждого домена по очереди.

#### Ответ от Claude

**Происхождение паттерна.** Термин ввёл Мартин Фаулер в 2004 году по аналогии с деревьями-душителями (strangler figs) тропических лесов: семя прорастает на ветке дерева-хозяина, пускает корни вниз и постепенно обвивает хозяина, в конечном счёте полностью заменяя его.

**Ключевая идея.** Никакого «большого взрыва» — система мигрирует итеративно, шаг за шагом, без остановки сервиса.

**Пошаговая реализация:**

1. **Фасад/прокси перед монолитом.** Внедрить API Gateway или реверс-прокси (например, Spring Cloud Gateway, Nginx), который транспарентно перенаправляет 100% трафика на монолит. Клиенты видят один неизменный URL.

2. **Выделение первого сервиса.** Выбрать наименее связанный bounded context (например, управление пользователями). Реализовать его как отдельный микросервис. Перенастроить роутинг в фасаде: запросы к `/users/**` теперь идут в новый сервис.

3. **Anti-Corruption Layer (ACL).** Если новые сервисы или остатки монолита вызывают друг друга — использовать ACL-адаптер для трансляции между доменными моделями. Это защищает новую систему от легаси-семантики.

4. **Миграция базы данных.** Самый сложный этап. Алгоритм: новый сервис поначалу читает/пишет в монолитную БД; затем создаётся изолированная БД домена; через ETL переносятся исторические данные; CDC (например, Debezium) синхронизирует дельту; после валидации — cutover, удаление домена из монолитной БД.

5. **Повторение для каждого домена.** Итерационно, пока монолит не опустеет.

6. **Демонтаж фасада (опционально).** После полной миграции фасад либо удаляется, либо остаётся в роли стабильного API Gateway.

**Риски и меры:**
- Фасад не должен стать единой точкой отказа — нужна кластеризация и health checks.
- Временные затраты на двойную поддержку двух систем — принять как стоимость снижения риска.
- Согласованность данных при двойной записи — использовать Saga или eventual consistency.
- Откат возможен только до момента удаления таблиц из монолитной БД.

**Паттерн плохо применим, если:** монолит не допускает перехвата запросов; нет доступа к исходному коду; система настолько мала, что проще переписать сразу.

### Вопрос 17. Какие компоненты Spring Cloud вы знаете? Опишите роль Spring Cloud Gateway, Spring Cloud Config и Spring Cloud LoadBalancer в типичной микросервисной системе.

**Категория:** Микросервисы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Spring Cloud Gateway — официальная документация](https://docs.spring.io/spring-cloud-gateway/docs/current/reference/html/)

**Spring Cloud Gateway** — API-шлюз, построенный на реактивном стеке (Spring WebFlux, Project Reactor). Основные концепции:

- **Route (маршрут)** — базовый строительный блок: ID + целевой URI + предикаты + фильтры. Маршрут срабатывает, если совокупный предикат возвращает `true`.
- **Predicate** — условие сопоставления запроса с маршрутом: по пути, методу, заголовку, cookie, IP и т.д.
- **Filter** — модификации запроса (PRE-фаза) и ответа (POST-фаза): добавление заголовков, rate limiting, circuit breaker, переписывание пути и др.

Пример конфигурации:
```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: lb://user-service
          predicates:
            - Path=/users/**
          filters:
            - StripPrefix=1
            - AddRequestHeader=X-Gateway, true
```

Схема обработки запроса: Client → Gateway Handler Mapping → Gateway Web Handler → Filter Chain (PRE) → Downstream Service → Filter Chain (POST) → Client.

**Spring Cloud LoadBalancer** заменил Netflix Ribbon (переведён в maintenance mode). Предоставляет клиентскую балансировку нагрузки. Алгоритмы: Round-Robin (по умолчанию) и Random. Интегрируется с `WebClient` (`@LoadBalanced`), `RestTemplate` и `RestClient`. Поддерживает зональную балансировку, sticky sessions, health checks и метрики через Micrometer.

**Spring Cloud Config** (из официального руководства Spring): сервер конфигурации хранит настройки в Git-репозитории и раздаёт их клиентам. Сервер активируется аннотацией `@EnableConfigServer`. Клиент подключается через `spring.config.import=optional:configserver:http://localhost:8888/`. Поддерживается динамическое обновление конфигурации без перезапуска: `@RefreshScope` + POST `/actuator/refresh`.

**Spring Cloud Gateway**

*Роль:* единая точка входа для всех клиентских запросов (API Gateway pattern). Устаревший Netflix Zuul заменён на Gateway с реактивным стеком.

*Ключевые функции:*
- Интеллектуальная маршрутизация по предикатам (Path, Method, Header, Cookie, Weight и др.).

- Интеграция с Spring Cloud LoadBalancer через схему `lb://service-name`.

*Важно:* Gateway требует Netty runtime и не совместим с традиционными Servlet-контейнерами.

**Spring Cloud Config**

*Роль:* централизованное управление конфигурацией всех микросервисов.

*Архитектура:*
```
Git-репозиторий ← Config Server → Config Client 1
                                → Config Client 2
                                → Config Client N
```

*Config Server:*
```java
@SpringBootApplication

public class ConfigServerApplication { ... }
```


```properties

```

Сервер отдаёт файл `{application-name}-{profile}.yml` из Git. Поддерживаются профили (`dev`, `prod`) и метки (branches/tags).

*Динамическое обновление:* `@RefreshScope` на бинах + POST `/actuator/refresh` на клиенте (или Spring Cloud Bus для массовой рассылки через Kafka/RabbitMQ).

*Преимущества:* версионирование конфигурации через Git-историю; конфигурация не «зашита» в Docker-образ; смена настроек без передеплоя.

**Spring Cloud LoadBalancer**

*Роль:* клиентская балансировка нагрузки — выбор конкретного экземпляра сервиса на стороне вызывающего.

*Отличие от серверной балансировки:* клиент сам получает список экземпляров из Service Discovery (Eureka/Consul) и выбирает один по алгоритму.

*Алгоритмы:*
- `RoundRobinLoadBalancer` — по умолчанию, последовательный перебор.
- `RandomLoadBalancer` — случайный выбор.
- Кастомная реализация через `ReactorLoadBalancer<ServiceInstance>`.

*Интеграция:*
```java
// С WebClient (реактивный)
@Bean @LoadBalanced
public WebClient.Builder webClientBuilder() { return WebClient.builder(); }

// С RestTemplate (блокирующий)
@Bean @LoadBalanced
public RestTemplate restTemplate() { return new RestTemplate(); }
```

После добавления `@LoadBalanced` обращение `http://user-service/api/users` автоматически резолвится в реальный адрес экземпляра.

*Дополнительные возможности:* зональная балансировка, health-check фильтрация экземпляров, sticky sessions, взвешенная балансировка, метрики Micrometer.

### Вопрос 18. Как настроить Circuit Breaker с помощью Resilience4j в Spring Boot приложении? Какие дополнительные паттерны устойчивости (Retry, Bulkhead, RateLimiter) предоставляет эта библиотека?

**Категория:** Микросервисы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Resilience4j — Getting Started (официальная документация)](https://resilience4j.readme.io/docs/getting-started-3)

Resilience4j — легковесная библиотека устойчивости для Java 17+, вдохновлённая Netflix Hystrix. Ключевые паттерны: CircuitBreaker, Retry, RateLimiter, Bulkhead, TimeLimiter.

**CircuitBreaker** реализует конечный автомат с тремя основными состояниями:
- **CLOSED** — нормальная работа, вызовы проходят.
- **OPEN** — вызовы отклоняются с `CallNotPermittedException` для защиты от каскадных сбоев.
- **HALF_OPEN** — после истечения `waitDurationInOpenState` пропускается ограниченное число тестовых вызовов для проверки восстановления.

Дополнительные состояния: DISABLED (пропускает всё), FORCED_OPEN (блокирует всё), METRICS_ONLY.

Типы скользящего окна:
- **COUNT_BASED** — отслеживает результаты последних N вызовов (кольцевой массив).
- **TIME_BASED** — отслеживает вызовы за последние N секунд.

**Конфигурация в application.yml:**
```yaml
resilience4j:
  circuitbreaker:
    instances:
      myService:
        slidingWindowSize: 10
        failureRateThreshold: 50
        waitDurationInOpenState: 30s
        permittedNumberOfCallsInHalfOpenState: 5
  retry:
    instances:
      myService:
        maxAttempts: 3
        waitDuration: 500ms
        enableExponentialBackoff: true
        exponentialBackoffMultiplier: 2
  ratelimiter:
    instances:
      myService:
        limitForPeriod: 50
        limitRefreshPeriod: 1s
        timeoutDuration: 0ms
  bulkhead:
    instances:
      myService:
        maxConcurrentCalls: 20
        maxWaitDuration: 0ms
```

**Порядок применения аспектов** (снаружи внутрь по умолчанию): Retry → CircuitBreaker → RateLimiter → TimeLimiter → Bulkhead.

Аннотации поддерживают синхронные типы, `CompletableFuture` и реактивные (`Flux`, `Mono`).

**CircuitBreaker**

*Состояния и переходы:*
- `CLOSED` → `OPEN`: когда `failureRateThreshold` (%) или `slowCallRateThreshold` (%) превышен при минимальном числе вызовов `minimumNumberOfCalls`.
- `OPEN` → `HALF_OPEN`: автоматически после `waitDurationInOpenState` (либо вручную).
- `HALF_OPEN` → `CLOSED` / `OPEN`: зависит от результатов `permittedNumberOfCallsInHalfOpenState` тестовых вызовов.

*Ключевые параметры с дефолтами:*

| Параметр | Дефолт | Описание |
|---|---|---|
| `slidingWindowType` | COUNT_BASED | Тип окна |
| `slidingWindowSize` | 100 | Кол-во вызовов или секунд |
| `minimumNumberOfCalls` | 100 | Минимум вызовов для расчёта |
| `failureRateThreshold` | 50 % | Порог ошибок для OPEN |
| `slowCallRateThreshold` | 100 % | Порог медленных вызовов |
| `slowCallDurationThreshold` | 60 000 мс | Что считать медленным |
| `waitDurationInOpenState` | 60 000 мс | Время в OPEN до HALF_OPEN |
| `permittedNumberOfCallsInHalfOpenState` | 10 | Тестовых вызовов в HALF_OPEN |

*Пример использования:*
```java
@Service
public class OrderService {

    @CircuitBreaker(name = "orderService", fallbackMethod = "fallbackOrder")
    public Order getOrder(Long id) {
        return externalClient.fetchOrder(id);
    }

    public Order fallbackOrder(Long id, Exception ex) {
        return Order.defaultOrder(id); // деградированный ответ
    }
}
```

```yaml
resilience4j:
  circuitbreaker:
    instances:
      orderService:
        slidingWindowType: COUNT_BASED
        slidingWindowSize: 10
        minimumNumberOfCalls: 5
        failureRateThreshold: 50
        waitDurationInOpenState: 10s
        permittedNumberOfCallsInHalfOpenState: 3
        recordExceptions:
          - java.io.IOException
          - java.util.concurrent.TimeoutException
        ignoreExceptions:
          - com.example.BusinessException
```

---

**Retry**

Автоматически повторяет неудачный вызов. Поддерживает экспоненциальный backoff с джиттером для предотвращения thundering herd.

```yaml
resilience4j:
  retry:
    instances:
      paymentService:
        maxAttempts: 3
        waitDuration: 500ms
        enableExponentialBackoff: true
        exponentialBackoffMultiplier: 2
        exponentialMaxWaitDuration: 10s
        retryExceptions:
          - java.net.ConnectException
        ignoreExceptions:
          - com.example.ValidationException
```

```java
@Retry(name = "paymentService", fallbackMethod = "fallbackPayment")
public PaymentResult processPayment(PaymentRequest request) { ... }
```

---

**Bulkhead**

Ограничивает число параллельных вызовов к сервису, предотвращая исчерпание ресурсов (thread starvation).

Два варианта:

*Semaphore Bulkhead* (по умолчанию) — лёгковесный, работает в том же потоке:
```yaml
resilience4j:
  bulkhead:
    instances:
      inventoryService:
        maxConcurrentCalls: 10
        maxWaitDuration: 100ms
```

*Thread Pool Bulkhead* — изолирует вызовы в отдельном пуле потоков, для асинхронных операций:
```yaml
resilience4j:
  thread-pool-bulkhead:
    instances:
      inventoryService:
        maxThreadPoolSize: 10
        coreThreadPoolSize: 5
        queueCapacity: 20
        keepAliveDuration: 20ms
```

```java
@Bulkhead(name = "inventoryService", type = Bulkhead.Type.SEMAPHORE)
public InventoryInfo getInventory(String sku) { ... }
```

---

**RateLimiter**

Ограничивает частоту вызовов методом скользящего окна с квотой.

```yaml
resilience4j:
  ratelimiter:
    instances:
      externalApi:
        limitForPeriod: 100         # вызовов за период
        limitRefreshPeriod: 1s      # период обновления квоты
        timeoutDuration: 500ms      # ожидание разрешения
```

```java
@RateLimiter(name = "externalApi", fallbackMethod = "rateLimitFallback")
public SearchResult search(String query) { ... }
```

При превышении лимита бросается `RequestNotPermitted`.

---

**TimeLimiter**

Ограничивает время выполнения асинхронной операции. Используется совместно с `@Async` / `CompletableFuture`:

```yaml
resilience4j:
  timelimiter:
    instances:
      slowService:
        timeoutDuration: 2s
        cancelRunningFuture: true
```

---

**Порядок аспектов и комбинирование**

По умолчанию порядок оборачивания (снаружи → внутрь):
`Retry(CircuitBreaker(RateLimiter(TimeLimiter(Bulkhead(function)))))`

Это значит: Retry видит исключение, которое CircuitBreaker уже зафиксировал. При необходимости порядок меняется через `*AspectOrder` свойства.

Пример комбинирования:
```java
@RateLimiter(name = "myService")
@Bulkhead(name = "myService")
@CircuitBreaker(name = "myService", fallbackMethod = "fallback")
@Retry(name = "myService")
public Result callExternalService() { ... }
```

**Мониторинг:** метрики автоматически публикуются через Spring Actuator (`/actuator/health`, `/actuator/metrics`). CircuitBreaker-состояние видно в health-индикаторах.

### Вопрос 19. Чем отличается оркестрация сервисов (orchestration) от хореографии (choreography) при построении распределённых бизнес-процессов? Назовите плюсы и минусы каждого подхода.

**Категория:** Микросервисы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Orchestration vs. choreography in microservices architecture — TechTarget](https://www.techtarget.com/searchapparchitecture/tip/Orchestration-vs-choreography-in-microservices-architecture)

**Оркестрация** использует централизованный контроллер, который вызывает каждый из сервисов, участвующих в транзакции, отслеживает результаты и либо продолжает выполнение, либо выполняет откат. Этот менеджер оркестрирует серию вызовов сервисов, известных как сага.

**Хореография** применяет децентрализованный подход, при котором сервисы могут общаться друг с другом без какого-либо контроллера. Сервисы самостоятельно потребляют релевантные данные из шины событий для выполнения своих задач асинхронно.

Плюсы оркестрации: жёсткий контроль над шагами процесса, чёткое управление транзакциями, проще понимать и отслеживать. Минусы оркестрации: единая точка отказа (если оркестратор падает — весь процесс останавливается), создаёт зависимости между сервисами, каскадные отказы при поломке одного сервиса.

Плюсы хореографии: высокая развязка обеспечивает независимость сервисов, устойчивость к отказам — они не обязательно каскадируют, гибкость и масштабируемость. Минусы хореографии: сложнее понимать и тестировать, трудно отслеживать взаимодействия через несколько брокеров, сложная асинхронная модель программирования.

Оркестрация лучше подходит для транзакционных процессов с жёсткими зависимостями, требующих немедленного ответа (например, оформление заказа в интернет-магазине). Хореография подходит для асинхронных независимых операций, допускающих итоговую согласованность (eventual consistency).

#### Ответ от Claude

**Оркестрация (Orchestration)** — централизованный подход, при котором один выделенный компонент (оркестратор) управляет всем бизнес-процессом: явно вызывает каждый сервис в нужной последовательности, получает ответы, принимает решения о переходах и обрабатывает ошибки. Классический пример реализации — паттерн Saga с оркестратором (Saga Orchestrator).

**Хореография (Choreography)** — децентрализованный подход, при котором каждый сервис реагирует на события, не зная об общей логике процесса. Сервисы публикуют события на шину (Kafka, RabbitMQ и т.д.) и подписываются на события других сервисов. Никто не «командует» — каждый участник знает только своё дело и свои триггеры. Классический пример — паттерн Saga с хореографией (Choreography-based Saga).

**Сравнение по ключевым параметрам:**

| Параметр | Оркестрация | Хореография |
|---|---|---|
| Управление | Централизованное | Децентрализованное |
| Коммуникация | Синхронная (обычно) | Асинхронная (события) |
| Связанность | Высокая (сервисы знают об оркестраторе) | Низкая (сервисы знают только о событиях) |
| Наблюдаемость | Высокая — вся логика в одном месте | Низкая — нужна распределённая трассировка |
| Масштабируемость | Ограничена оркестратором | Хорошо масштабируется |

**Плюсы оркестрации:**
- Вся бизнес-логика процесса сосредоточена в одном месте, легко читать и изменять
- Простой мониторинг и логирование: состояние процесса всегда известно
- Удобное управление компенсирующими транзакциями при откате
- Легко реализовать сложные условные переходы и ветвления

**Минусы оркестрации:**
- Оркестратор — единая точка отказа (SPOF)
- При масштабировании оркестратор становится узким местом
- Сервисы косвенно связаны через оркестратор, что затрудняет их независимую эволюцию
- Orchestration-тяжёлые системы легко превращаются в «распределённый монолит»

**Плюсы хореографии:**
- Высокая автономность сервисов: каждый сервис деплоится и масштабируется независимо
- Отказ одного сервиса не блокирует весь процесс (при правильной реализации)
- Естественная асинхронность хорошо сочетается с событийно-ориентированной архитектурой
- Команды работают независимо над своими сервисами без координации

**Минусы хореографии:**
- Тяжело отследить сквозной ход бизнес-процесса — нужны distributed tracing инструменты (Zipkin, Jaeger)
- «Event storms» — неконтролируемые цепочки событий при росте числа сервисов
- Компенсирующие транзакции сложнее реализовать и отлаживать
- Неявная бизнес-логика «размазана» по сервисам, что усложняет понимание системы

**На практике** крупные системы используют гибридный подход: оркестрация применяется для транзакционных процессов с жёсткими требованиями к консистентности (например, финансовые операции), а хореография — для слабосвязанных асинхронных потоков (уведомления, аналитика, фоновая обработка).

### Вопрос 20. Как реализовать централизованную конфигурацию для микросервисов с помощью Spring Cloud Config? Как обеспечить обновление конфигурации без перезапуска сервисов?

**Категория:** Микросервисы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Getting Started: Centralized Configuration — spring.io](https://spring.io/guides/gs/centralized-configuration/)

Spring Cloud Config Server обеспечивает централизованное управление конфигурацией для микросервисов. Настройка состоит из двух компонентов.

**Config Server** создаётся как Spring Boot приложение с аннотацией `@EnableConfigServer`. Конфигурируется указанием пути к Git-репозиторию с файлами свойств:

```properties
server.port=8888
spring.cloud.config.server.git.uri=${HOME}/Desktop/config
```

Сервер раздаёт конфигурацию на основе имени приложения; файл `application.properties` применяется ко всем клиентам глобально.

**Config Client** добавляет зависимость на Config Client и конфигурируется:

```properties
spring.application.name=a-bootiful-client
spring.config.import=optional:configserver:http://localhost:8888/
management.endpoints.web.exposure.include=*
```

**Обновление без перезапуска:** контроллер аннотируется `@RefreshScope`, что делает его refresh-aware. Обновление конфигурации инициируется POST-запросом к эндпоинту:

```
curl -X POST localhost:8080/actuator/refresh -d {} -H "Content-Type: application/json"
```

Клиент получает значения через `@Value("${property}")` или `@ConfigurationProperties`, и эндпоинт refresh заставляет его получить обновлённые значения с Config Server без перезапуска приложения.

#### Ответ от Claude

**Архитектура Spring Cloud Config**

Spring Cloud Config реализует паттерн «Externalized Configuration» из 12-факторного приложения. Система состоит из двух компонентов: Config Server (централизованное хранилище) и Config Client (каждый микросервис).

**Настройка Config Server:**

1. Добавить зависимость `spring-cloud-config-server`
2. Аннотировать главный класс `@EnableConfigServer`
3. Настроить бэкенд (Git — наиболее распространённый вариант):

```yaml
server:
  port: 8888
spring:
  cloud:
    config:
      server:
        git:
          uri: https://github.com/myorg/config-repo
          default-label: main
```

Config Server поддерживает несколько бэкендов: Git (с версионированием и ветками для окружений), файловая система (для разработки), HashiCorp Vault (для секретов), Composite (комбинация нескольких источников одновременно).

Соглашение об именовании файлов конфигурации: `{application}-{profile}.yml` или `{application}-{profile}.properties`. При запросе клиента с именем `order-service` и профилем `prod` сервер вернёт `order-service-prod.yml`, `order-service.yml` и `application.yml` (в порядке приоритета).

**Настройка Config Client:**

```yaml
spring:
  application:
    name: order-service
  config:
    import: "optional:configserver:http://localhost:8888/"
  profiles:
    active: prod
management:
  endpoints:
    web:
      exposure:
        include: refresh, busrefresh, health, info
```

**Обновление конфигурации без перезапуска — три уровня:**

**Уровень 1: `@RefreshScope` + `/actuator/refresh` (один экземпляр)**

Аннотация `@RefreshScope` помечает бин как пересоздаваемый при обновлении. При вызове POST `/actuator/refresh` Spring пересоздаёт все `@RefreshScope`-бины, загружая актуальные значения с Config Server.

```java
@RestController
@RefreshScope
public class FeatureController {
    @Value("${feature.flag.enabled:false}")
    private boolean featureEnabled;
}
```

Ограничение: работает только для конкретного экземпляра, которому отправлен запрос. Для N экземпляров нужно вызывать N раз.

**Уровень 2: Spring Cloud Bus (`/actuator/busrefresh`, несколько экземпляров)**

Spring Cloud Bus использует брокер сообщений (RabbitMQ или Kafka) для рассылки события обновления всем экземплярам сразу. Один POST-запрос к любому экземпляру (или к Config Server) распространяет RefreshRemoteApplicationEvent по всей системе.

Зависимость: `spring-cloud-starter-bus-amqp` (или `bus-kafka`). Эндпоинт: POST `/actuator/busrefresh`.

**Уровень 3: Webhook от Git-репозитория (автоматическое обновление)**

Config Server настраивается для приёма webhook-уведомлений от GitHub/GitLab при пуше в репозиторий. При получении webhook Config Server автоматически публикует событие в Spring Cloud Bus, и все клиенты обновляются без ручного вмешательства.

**Важные нюансы:**
- `@ConfigurationProperties`-бины требуют явной аннотации `@RefreshScope` или использования `@Bean` с `@RefreshScope` на фабричном методе; иначе они не пересоздаются
- Конфигурация, участвующая в инициализации контекста (DataSource, пул потоков), НЕ обновляется через `@RefreshScope` — для неё требуется перезапуск
- `@Value`-поля в `@Component`-бинах работают с `@RefreshScope` корректно
- В production обычно сочетают Git + Vault: Git хранит обычные настройки, Vault — секреты (пароли, ключи API)
