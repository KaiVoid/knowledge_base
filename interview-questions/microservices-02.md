### Вопрос 21. Чем принципиально отличается «распределённый монолит» от настоящей микросервисной архитектуры, и по каким признакам его можно опознать в production-системе?

**Категория:** Микросервисы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Is your microservice a distributed monolith? — Gremlin](https://www.gremlin.com/blog/is-your-microservice-a-distributed-monolith)

Распределённый монолит — это приложение, «задеплоенное как микросервис, но построенное как монолит». Оно использует инфраструктуру микросервисов (контейнеры, Kubernetes), сохраняя при этом ограничения монолитной архитектуры.

**Ключевые признаки:**

**1. Жёстко связанные сервисы.** Сервисы сильно зависят друг от друга функционально. Изменение одного сервиса вынуждает менять несколько других. Это проявляется на примере интернет-магазина: падение каталога товаров приводит к отказу или таймаутам фронтенда.

**2. Плохая масштабируемость.** Технически масштабировать можно, но неэффективно. Горизонтальное масштабирование требует деплоя целых экземпляров приложения со всеми зависимостями вместо отдельных компонентов. Соединения с базой данных часто становятся узким местом.

**3. Чрезмерное межсервисное взаимодействие.** Сервисы «слишком болтливы» и требуют многочисленных сетевых вызовов для выполнения базовых операций. Даже небольшое увеличение задержки (20 мс) может катастрофически ухудшить производительность — в примере из статьи минимальная сетевая задержка дала 5,5-кратное увеличение времени ответа.

Для выявления этих проблем рекомендуются техники Chaos Engineering: blackhole-атаки для проверки взаимозависимостей сервисов, CPU-атаки для оценки возможностей авто-масштабирования, latency-атаки для измерения накладных расходов на коммуникацию.

#### Ответ от Claude

**Принципиальное отличие**

Настоящие микросервисы строятся вокруг принципа независимости: каждый сервис можно разработать, протестировать, задеплоить и масштабировать независимо от других. Распределённый монолит нарушает этот принцип — сервисы физически разделены, но логически остаются единым целым.

Образно: истинные микросервисы — это независимые магазины в торговом центре, каждый со своим входом, складом и кассой. Распределённый монолит — это один большой магазин, разрезанный на секции, но с общим складом, единой кассой и персоналом, который не может работать без коллег из других секций.

**Признаки распределённого монолита в production-системе:**

**1. Совместно используемая база данных (Shared Database)**
Несколько сервисов пишут и читают из одной БД или её схемы. Это самый частый и опасный признак. Изменение схемы одной таблицы ломает несколько сервисов. Нет изоляции данных — сервисы не могут выбрать подходящую СУБД для своих нужд.

**2. Синхронные цепочки вызовов (Synchronous Call Chains)**
Сервис A вызывает B, B вызывает C, C вызывает D — всё синхронно. Латентность суммируется, а не усредняется. Падение любого звена цепочки блокирует весь запрос. В production это видно как каскадные таймауты.

**3. Совместный деплой (Lockstep Deployment)**
Изменение в одном сервисе требует одновременного деплоя других. В GitLab/CI-пайплайне это проявляется как единый пайплайн для нескольких «сервисов» или жёсткая версионная зависимость между ними.

**4. Разделяемый код не через API (Shared Libraries with Business Logic)**
Бизнес-логика вынесена в общую библиотеку (shared-lib), которую используют несколько сервисов. Изменение библиотеки требует одновременного обновления всех потребителей.

**5. Отсутствие устойчивости к частичным отказам**
В истинных микросервисах деградация одного сервиса не должна делать недоступной всю систему. Если падение сервиса A делает недоступным сервис B (который формально его не вызывает напрямую) — это признак скрытых зависимостей.

**6. Единая точка конфигурации с жёсткой связанностью**
Все сервисы знают внутренние адреса и детали реализации друг друга (не через Service Discovery, а через хардкоженные имена хостов или общий конфиг с топологией системы).

**7. Невозможность независимого масштабирования**
При увеличении нагрузки на один сервис приходится масштабировать группу сервисов или всю систему — иначе перегруженный сервис становится узким местом для остальных.

**Как проверить в production:**

- Построить граф зависимостей вызовов (например, через Jaeger/Zipkin). Длинные синхронные цепочки — красный флаг.
- Провести эксперимент: отключить один некритичный сервис. Сколько других сервисов деградирует? В истинных микросервисах — только прямые потребители, и только частично.
- Проанализировать историю деплоев: насколько часто деплои нескольких сервисов коррелируют по времени?
- Проверить схемы БД: есть ли таблицы, к которым обращаются более одного сервиса?

**Итог:** распределённый монолит получает худшее из обоих миров — операционную сложность микросервисов (сеть, распределённые трассировки, контейнеры) без их главного преимущества (независимость). Главный критерий настоящих микросервисов — возможность независимого деплоя каждого сервиса в любой момент без координации с командами других сервисов.
### Вопрос 22. Как Bounded Context из DDD помогает определять границы микросервиса? Что происходит, когда одна сущность (например, «Заказ») нужна трём разным сервисам — должна ли она дублироваться или выноситься в отдельный сервис?

**Категория:** Микросервисы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Use Domain Analysis to Model Microservices — Azure Architecture Center (Microsoft Learn)](https://learn.microsoft.com/en-us/azure/architecture/microservices/model/domain-analysis)

Статья объясняет, что одна из главных сложностей при проектировании микросервисов — определение границ каждого сервиса. Общее правило: сервис должен делать что-то одно, но применение этого правила требует тщательного обдумывания бизнес-домена.

**Bounded Context** — концепция из DDD, обозначающая границу внутри домена, в которой применяется конкретная доменная модель. Сервисы группируются по тому, разделяют ли они одну доменную модель. Создавать единую модель для разных контекстов не нужно — это усложняет систему и затрудняет её развитие, так как изменения должны удовлетворять нескольким командам.

По вопросу общих сущностей: **идентификаторы** сущностей могут разделяться между контекстами, но сами сущности не обязаны быть идентичными во всех контекстах. Каждый контекст вправе иметь свою версию сущности. Например, в контексте управления дронами важны история обслуживания, пробег, возраст, номер модели. В контексте планирования доставки нужно лишь знать, доступен ли дрон и каков ETA. Правильнее проектировать отдельные модели для одной и той же реальной сущности в двух разных контекстах — каждая включает только те атрибуты, которые актуальны внутри своего контекста.

#### Ответ от Claude

**Что такое Bounded Context и зачем он нужен**

Bounded Context (ограниченный контекст) — стратегический паттерн DDD, который задаёт явную границу, внутри которой действует конкретная доменная модель с единым языком (Ubiquitous Language). Внутри контекста каждый термин имеет одно и только одно значение — например, слово «Аккаунт» в биллинговом контексте означает физическое лицо, а в CRM-контексте — организацию.

Шаги применения DDD при проектировании микросервисов:
1. Анализ бизнес-домена (event storming, интервью с экспертами).
2. Выделение Bounded Context'ов, объединяющих связанные функции с единой моделью.
3. Применение тактических паттернов (Entity, Aggregate, Domain Service) внутри каждого контекста.
4. Идентификация микросервисов — один BC может содержать один или несколько микросервисов.

Важно: Bounded Context — это **не** то же самое, что микросервис. Как показывает Влад Хонов (vladikk.com), строгое следование принципам BC ведёт к получению «хороших монолитов», а не микросервисов. BC — граница наибольшего возможного сервиса без конфликтов моделей; микросервис возникает через дальнейшую декомпозицию BC.

**Что делать, когда «Заказ» нужен трём сервисам**

Правильный ответ — **не выносить в отдельный общий сервис** и **не делать единую разделяемую модель**. Вместо этого каждый контекст держит свою проекцию сущности:

| Контекст | Что из «Заказа» важно |
|---|---|
| Платёжный сервис | ID заказа, сумма, валюта, статус оплаты |
| Сервис доставки | ID заказа, адрес доставки, вес, габариты |
| Сервис уведомлений | ID заказа, email клиента, статус |

Каждый сервис хранит у себя только те поля, которые нужны для его собственного поведения. Идентификатор (`orderId`) разделяется — он служит ключом корреляции. Синхронизация данных между контекстами происходит через API или доменные события (например, `OrderCreated`, `OrderPaid`), но не через общую базу данных или общую модель.

**Когда выносить в отдельный сервис**

Вынесение сущности в отдельный сервис оправдано, если само управление этой сущностью является самостоятельной бизнес-возможностью (например, сервис каталога товаров). В таком случае другие сервисы получают данные через его API, а не через общую модель.

**Паттерны взаимодействия между контекстами (Context Map)**

- **Open Host Service / Published Language** — апстрим-контекст предоставляет стабильный API; предпочтительно для межсервисного взаимодействия.
- **Customer-Supplier** — команды договариваются о контракте.
- **Anti-Corruption Layer** — даунстрим-контекст строит слой-транслятор, защищающий свою модель от чужих изменений.

**Итог:** сущность «Заказ» в трёх разных сервисах — это три разные контекстно-специфичные модели, объединённые только идентификатором. Это намеренная «дублированность» данных, которая обеспечивает автономию сервисов и слабую связанность.

### Вопрос 23. Объясни три состояния Circuit Breaker (Closed → Open → Half-Open) в Resilience4j. Какой параметр отвечает за переход из Open в Half-Open, и что случится, если в Half-Open состоянии снова придёт ошибка?

**Категория:** Микросервисы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [CircuitBreaker — Resilience4j официальная документация](https://resilience4j.readme.io/docs/circuitbreaker)

Circuit Breaker реализован в виде конечного автомата с тремя основными состояниями: CLOSED, OPEN и HALF_OPEN, а также тремя специальными: METRICS_ONLY, DISABLED и FORCED_OPEN.

**CLOSED (закрыт)** — нормальная работа: все вызовы пропускаются. CircuitBreaker агрегирует результаты вызовов через скользящее окно (count-based или time-based). Если частота сбоев превышает настроенный порог — переход в OPEN.

**OPEN (открыт)** — сбойный режим: CircuitBreaker немедленно отклоняет все вызовы исключением `CallNotPermittedException`, не обращаясь к нижележащему сервису. Это предотвращает каскадные сбои.

**HALF_OPEN (полуоткрыт)** — диагностический режим: после истечения периода ожидания допускается ограниченное число пробных вызовов для проверки, восстановился ли сервис.

**waitDurationInOpenState** — параметр, определяющий, как долго circuit breaker остаётся в состоянии OPEN перед переходом в HALF_OPEN. Значение по умолчанию — 60 секунд.

Если в состоянии HALF_OPEN частота сбоев снова достигает или превышает настроенный порог, состояние возвращается обратно в OPEN. Если частота ниже порога — переход в CLOSED.

#### Ответ от Claude

**Три основных состояния**

**CLOSED (нормальная работа)**

Все запросы к защищаемому сервису пропускаются. CircuitBreaker собирает статистику через скользящее окно:
- `COUNT_BASED` — последние N вызовов (настраивается `slidingWindowSize`, по умолчанию 100).
- `TIME_BASED` — вызовы за последние N секунд.

Если `failureRateThreshold` (по умолчанию 50%) или `slowCallRateThreshold` (по умолчанию 100%) превышены — переход в OPEN.

**OPEN (circuit разомкнут — сервис считается недоступным)**

CircuitBreaker мгновенно отклоняет все входящие вызовы исключением `CallNotPermittedException`, не дожидаясь таймаутов. Это ключевое отличие от простого retry: вместо ожидания ответа от «упавшего» сервиса происходит немедленный fail-fast. После истечения `waitDurationInOpenState` — автоматический переход в HALF_OPEN.

**HALF_OPEN (зондирование)**

Пропускается ровно `permittedNumberOfCallsInHalfOpenState` (по умолчанию 10) пробных запросов. По их результатам:
- Если частота сбоев `>= failureRateThreshold` → возврат в OPEN (повторное ожидание `waitDurationInOpenState`).
- Если частота сбоев `< failureRateThreshold` → переход в CLOSED, нормальная работа возобновляется.

**Ключевой параметр перехода Open → Half-Open**

```java
CircuitBreakerConfig config = CircuitBreakerConfig.custom()
    .waitDurationInOpenState(Duration.ofSeconds(30))  // ключевой параметр
    .permittedNumberOfCallsInHalfOpenState(5)
    .failureRateThreshold(50)
    .slidingWindowSize(10)
    .build();
```

`waitDurationInOpenState` — это время (Duration), которое circuit breaker находится в состоянии OPEN до автоматического перехода в HALF_OPEN. По умолчанию 60 секунд. Также существует `automaticTransitionFromOpenToHalfOpenEnabled` (по умолчанию `false`) — если `false`, переход произойдёт только при следующем входящем вызове после истечения `waitDurationInOpenState`; если `true` — переход происходит автоматически без ожидания вызова.

**Что происходит при ошибке в Half-Open**

Ошибка в HALF_OPEN не переводит circuit breaker в OPEN немедленно после первого сбоя. Необходимо, чтобы все `permittedNumberOfCallsInHalfOpenState` вызовов были выполнены и оценены. Если итоговая частота сбоев по этим вызовам достигает `failureRateThreshold`:

1. Состояние переходит обратно в OPEN.
2. Начинается новый цикл ожидания `waitDurationInOpenState`.
3. Снова выбрасывается `CallNotPermittedException` для всех входящих запросов.

Таким образом, один сбой из пяти разрешённых пробных вызовов при пороге 50% не вернёт circuit breaker в OPEN — нужно, чтобы доля сбоев среди всех пробных запросов превысила порог.

**Специальные состояния (для полноты картины)**

- `DISABLED` — circuit breaker выключен, статистика не собирается.
- `FORCED_OPEN` — принудительно открыт, все вызовы отклоняются.
- `METRICS_ONLY` — статистика собирается, но переходов между состояниями нет.

### Вопрос 24. В чём разница между Saga с хореографией (choreography) и Saga с оркестрацией (orchestration)? Когда предпочтёшь одно другому, и каковы слабые стороны каждого подхода?

**Категория:** Микросервисы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [To Choreograph or Orchestrate your Saga — Temporal Blog](https://temporal.io/blog/to-choreograph-or-orchestrate-your-saga-that-is-the-question)

**Хореография:** микросервисы общаются через передачу сообщений, каждый обладает только локальными знаниями — поведение системы формируется как «emergent» (возникающее). Легче реализовать на начальном этапе, хорошо подходит для конвертации монолитов в микросервисы и для истинно независимых сервисов.

Недостатки: неочевидный control flow, сложно отлаживать, поскольку «порядок операций распределён по всему коду». Трудно поддерживать, если между задачами есть последовательные зависимости типа «must happen before».

**Оркестрация:** центральный «супер-микросервис» выступает брокером сообщений, направляя остальные сервисы через определённый рабочий процесс. Control flow централизован и понятен, код отдельных сервисов проще, легче отлаживать. Предпочтительна для greenfield-приложений и когда в одном шаге должны взаимодействовать несколько сервисов.

Ахиллесова пята оркестрации — брокер сообщений является единой точкой отказа.

#### Ответ от Claude

**Суть паттерна Saga**

Saga — способ управления распределёнными транзакциями в микросервисах без использования 2PC (двухфазного коммита). Транзакция разбивается на последовательность локальных транзакций; если какая-то из них завершается неудачей, выполняются компенсирующие транзакции для отката уже выполненных шагов.

**Хореография (Choreography)**

Каждый сервис публикует доменное событие после завершения своего шага. Следующий сервис подписан на это событие и реагирует на него. Центрального координатора нет.

```
Order Service → publishes OrderCreated
  → Payment Service subscribes, charges card → publishes PaymentProcessed
    → Inventory Service subscribes, reserves items → publishes ItemsReserved
      → Shipping Service subscribes, arranges delivery
```

Плюсы:
- Слабая связанность: сервисы не знают друг о друге напрямую.
- Высокая устойчивость: нет единой точки отказа.
- Хорошо масштабируется горизонтально.
- Легче начать при конвертации монолита.

Минусы:
- Трудно отследить весь flow транзакции — он «размазан» по нескольким сервисам.
- Сложная отладка: для трассировки нужен distributed tracing (Jaeger, Zipkin).
- Риск «god event» — события несут слишком много данных.
- Сложно реализовать условную логику (if/else между шагами).
- Циклические зависимости через события — трудно обнаружить.

**Оркестрация (Orchestration)**

Центральный оркестратор (saga orchestrator) явно знает весь workflow и вызывает сервисы в нужном порядке. Сервисы-участники не знают о существовании других участников.

```
Order Saga Orchestrator:
  1. → вызов Payment Service → ожидание ответа
  2. → вызов Inventory Service → ожидание ответа
  3. → вызов Shipping Service → ожидание ответа
  При сбое шага N: → вызов компенсирующих транзакций для шагов N-1...1
```

Плюсы:
- Весь workflow виден в одном месте — легко читать и изменять.
- Явное управление компенсирующими транзакциями.
- Проще мониторинг: состояние саги хранится у оркестратора.
- Естественная поддержка условной логики.

Минусы:
- Оркестратор — потенциальная единая точка отказа (решается через надёжные фреймворки: Axon, Temporal, Conductor).
- Риск «fat orchestrator» — бизнес-логика стягивается к оркестратору.
- Сервисы-участники становятся зависимы от оркестратора: tight coupling по интерфейсу.
- Сложнее первоначальная реализация.

**Сравнительная таблица**

| Критерий | Хореография | Оркестрация |
|---|---|---|
| Управление flow | Неявное, distributed | Явное, централизованное |
| Связанность | Слабая (через события) | Умеренная (через оркестратор) |
| Отладка | Сложная | Проще |
| Единая точка отказа | Нет | Да (оркестратор) |
| Условная логика | Сложно | Естественно |
| Подходит для | Простых, независимых flow | Сложных, многошаговых workflow |

**Когда выбирать хореографию**

- Небольшое число шагов (2–3).
- Сервисы действительно независимы, нет жёстких «must happen before».
- Команды сервисов автономны и не хотят внешней координации.
- Высокие требования к масштабируемости и отказоустойчивости.

**Когда выбирать оркестрацию**

- Сложный многошаговый workflow с условными ветками.
- Необходима чёткая видимость состояния бизнес-процесса.
- Требуется надёжный rollback с компенсирующими транзакциями.
- Greenfield-разработка, где архитектура ещё формируется.

**Компенсирующие транзакции (общее для обоих подходов)**

При сбое в шаге N необходимо выполнить компенсации для шагов N-1, N-2, ..., 1 в обратном порядке. Важно: компенсирующая транзакция — это не откат в смысле ACID; данные уже были видны другим сервисам, это «семантическая отмена» (например, вместо удаления заказа — его отмена с соответствующим статусом).

### Вопрос 25. Что такое паттерн Outbox и зачем он нужен при публикации событий в Kafka/RabbitMQ из микросервиса? Какую проблему он решает по сравнению с «просто вызовом producer.send() после сохранения в БД»?

**Категория:** Микросервисы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Microservices 101: Transactional Outbox and Inbox — SoftwareMill](https://softwaremill.com/microservices-101/)

Наивный подход — сохранить запись в БД, а затем вызвать `producer.send()` — создаёт критическую уязвимость. Если сервис падает после коммита транзакции, но до того, как сообщение успело уйти, уведомление теряется безвозвратно. Как формулирует источник: «если сообщение хранится только в памяти, то при краше сервиса A оно будет утеряно безвозвратно».

Паттерн Outbox решает это, добавляя выделенную таблицу `outbox` в той же базе данных. Процесс выглядит так:

1. **Сохранить намерение**: в рамках той же транзакции, что и бизнес-операция, вставить запись о сообщении в таблицу `outbox`.
2. **Фоновый опрос**: планировщик периодически читает таблицу `outbox` и публикует непрочитанные записи в брокер.
3. **Маркировка**: после успешной отправки запись помечается как доставленная или удаляется.

Доставка гарантируется **at-least-once**: даже при падении Publisher-а сообщения остаются в таблице и будут повторно обработаны. Но дубли возможны — потребители должны обеспечивать идемпотентность. Паттерн поддерживает два варианта реализации: **Polling** (SELECT ... FOR UPDATE SKIP LOCKED) и **CDC/Debezium** (чтение write-ahead log БД для снижения задержки).

#### Ответ от Claude

**Проблема dual-write**

Когда сервис должен одновременно обновить состояние в базе данных и отправить событие в Kafka/RabbitMQ, он сталкивается с задачей «двойной записи» (dual-write). Нет распределённой транзакции, охватывающей одновременно реляционную БД и брокер сообщений. Любая из следующих ситуаций приводит к рассогласованию:

- Сервис упал после `db.commit()`, но до `producer.send()` — событие потеряно, другие сервисы не узнают об изменении.
- Сервис упал после `producer.send()`, но до `db.commit()` — событие отправлено, но в БД данных нет; потребители обработают несуществующее состояние.
- Брокер временно недоступен — `producer.send()` бросает исключение; можно повторить попытку, но только если бизнес-операция ещё не закоммичена или есть надёжный механизм retry.

**Как работает Outbox**

Идея: использовать атомарность одной транзакции в одной БД. Бизнес-данные и «намерение отправить событие» записываются в одной транзакции:

```sql
BEGIN;
  INSERT INTO orders (id, ...) VALUES (...);
  INSERT INTO outbox (id, aggregate_type, payload, created_at, status)
    VALUES (gen_random_uuid(), 'Order', '{"event":"OrderCreated",...}', now(), 'PENDING');
COMMIT;
```

Либо оба insert попадают в БД, либо оба откатываются — рассогласование исключено.

Отдельный компонент (Message Relay / Publisher) читает таблицу `outbox` и публикует записи в брокер:

```
outbox table  -->  Message Relay  -->  Kafka / RabbitMQ
                   (polling or CDC)
```

**Два подхода к доставке из Outbox**

1. **Polling-based**: планировщик (или отдельный поток) регулярно выполняет `SELECT ... WHERE status = 'PENDING' FOR UPDATE SKIP LOCKED`, отправляет сообщения, затем обновляет `status = 'SENT'`. Просто в реализации, но добавляет нагрузку на БД и задержку пропорциональную интервалу опроса.

2. **CDC (Change Data Capture) через Debezium**: агент читает write-ahead log (WAL) PostgreSQL / binlog MySQL и стримит изменения в таблице `outbox` прямо в Kafka-топик. Задержка — миллисекунды, нет дополнительных SELECT-ов, но требует настройки Debezium и Kafka Connect.

**Гарантии доставки**

Паттерн обеспечивает **at-least-once delivery**: если Publisher упал после отправки в брокер, но до фиксации статуса `SENT` в outbox, запись будет обработана повторно и сообщение уйдёт дважды. Это нормально — потребители должны быть идемпотентны (проверять уже обработанные event-id). Exactly-once на уровне всей цепочки достигается именно комбинацией Outbox (at-least-once) + идемпотентный консьюмер.

**Отличие от «просто producer.send() после сохранения в БД»**

| Критерий | Naive producer.send() | Outbox pattern |
|---|---|---|
| Атомарность | Нет (две независимые операции) | Да (одна транзакция в БД) |
| Потеря сообщений | Возможна при краше | Исключена |
| Доступность брокера | Требуется в момент вызова | Не требуется; отправка асинхронна |
| Сложность реализации | Минимальная | Средняя (нужна outbox-таблица + relay) |
| Гарантия доставки | At-most-once (или none) | At-least-once |

**Готовые реализации**: Spring Modulith Outbox, Debezium Outbox Event Router, Axon Framework, Eventuate Tram.

### Вопрос 26. Чем gRPC отличается от REST на уровне транспорта и сериализации? В каких сценариях gRPC даёт реальный выигрыш, а в каких REST всё равно предпочтительнее?

**Категория:** Микросервисы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [gRPC vs. REST — Postman Blog](https://blog.postman.com/grpc-vs-rest/)

**Транспорт**: REST использует HTTP/1.1 (последовательная модель запрос-ответ), gRPC работает поверх HTTP/2, что даёт мультиплексирование множества одновременных потоков на одном соединении и сжатие заголовков.

**Сериализация**: REST передаёт данные в виде JSON или XML — форматов, удобных для чтения человеком, но «менее эффективных для передачи и разбора». gRPC использует Protocol Buffers (Protobuf) — бинарный формат, который «компактнее и быстрее при сериализации/десериализации, чем текстовые форматы, но нечитаем без декодера».

**Когда выбирать gRPC**: микросервисная архитектура с полиглотными средами, высокие требования к пропускной способности и низкой задержке, real-time стриминг данных (двунаправленный, серверный, клиентский).

**Когда выбирать REST**: публичные API с широкой доступностью для сторонних разработчиков, браузерные клиенты (нативной поддержки gRPC в браузере нет без прокси), сценарии с отладкой через curl и простой читаемостью JSON.

#### Ответ от Claude

**Транспортный уровень**

REST не привязан к конкретной версии HTTP, но традиционно строится поверх HTTP/1.1. gRPC требует HTTP/2. Разница принципиальная:

| Характеристика | HTTP/1.1 (REST) | HTTP/2 (gRPC) |
|---|---|---|
| Мультиплексирование | Нет (head-of-line blocking) | Да (множество стримов на одном TCP-соединении) |
| Сжатие заголовков | Нет | HPACK-компрессия |
| Серверный push | Нет | Есть |
| Установка соединения | Новое на каждый запрос (без keep-alive) | Одно долгоживущее соединение |
| Формат фрейминга | Текст | Бинарные фреймы |

HTTP/1.1 страдает от «блокировки головы очереди» (head-of-line blocking): новый запрос ждёт ответа на предыдущий (без HTTP pipelining). HTTP/2 устраняет это за счёт независимых стримов.

**Уровень сериализации**

REST обычно использует JSON (иногда XML). gRPC использует **Protocol Buffers (Protobuf)** — схемо-ориентированный бинарный формат. Сравнение:

| Критерий | JSON | Protobuf |
|---|---|---|
| Читаемость | Человекочитаем | Бинарный, нечитаем без схемы |
| Размер payload | Больше (текст + имена полей) | Меньше (числовые теги полей) |
| Скорость серилизации | Медленнее (text parsing) | Быстрее (до 5-7x в типичных тестах) |
| Строгость типизации | Нет встроенной схемы | IDL (.proto файлы), строгая типизация |
| Генерация кода | Не обязательна | Автоматическая для 10+ языков |

Protobuf поля идентифицируются числовыми тегами (не строками), что и даёт компактность. `.proto`-файл служит контрактом API и источником для генерации клиентского/серверного кода на Java, Go, Python, C++ и других языках.

**Четыре режима взаимодействия в gRPC**

В отличие от REST (только унарный запрос-ответ), gRPC поддерживает:

1. **Unary** — один запрос, один ответ (аналог REST).
2. **Server-streaming** — клиент отправляет запрос, сервер возвращает поток ответов.
3. **Client-streaming** — клиент стримит запросы, сервер даёт один ответ.
4. **Bidirectional streaming** — оба конца стримят одновременно по одному HTTP/2-соединению.

**Когда gRPC даёт реальный выигрыш**

- **Inter-service communication** (внутренние вызовы между микросервисами): меньшая задержка, меньший трафик, строгий контракт.
- **Высоконагруженные сервисы**: мультиплексирование HTTP/2 и компактный Protobuf снижают нагрузку на сеть и CPU.
- **Полиглотные среды**: автогенерируемые клиенты обеспечивают типобезопасность без ручного написания HTTP-клиентов.
- **Стриминг**: real-time телеметрия, live-обновления, двунаправленный чат.

**Когда REST предпочтительнее**

- **Публичные / открытые API**: REST понятен любому разработчику, поддерживается Swagger/OpenAPI, отлаживается через браузер и curl без специальных инструментов.
- **Браузерные клиенты**: браузеры не поддерживают gRPC-over-HTTP/2 напрямую; нужен gRPC-Web + прокси (Envoy). REST работает из коробки.
- **Простые CRUD-операции**: когда производительность не критична, а читаемость и простота важнее.
- **Кеширование HTTP**: gRPC не поддерживает стандартное HTTP-кеширование (ETag, Cache-Control) так просто, как REST.
- **Интеграция со сторонними системами**: REST + JSON — универсальный стандарт де-факто, совместимый со всем.

**Итог**: gRPC — выбор для внутренней шины микросервисов, где контролируются оба конца. REST — выбор для внешних API и браузерных клиентов.

### Вопрос 27. Как работает client-side service discovery с Eureka в Spring Cloud? Что произойдёт, если Eureka Server недоступен — смогут ли клиенты продолжать работу, и почему?

**Категория:** Микросервисы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Service Discovery: Eureka Clients — Spring Cloud Netflix (официальная документация)](https://cloud.spring.io/spring-cloud-netflix/multi/multi__service_discovery_eureka_clients.html)

При наличии `spring-cloud-starter-netflix-eureka-client` в classpath приложение автоматически становится и **экземпляром** (регистрирует себя), и **клиентом** (может опрашивать реестр). При регистрации сервис передаёт метаданные: хост, порт, URL индикатора здоровья, домашняя страница и другие данные. Периодически клиент отправляет heartbeat-сигналы для обновления регистрации.

Ключевое свойство — **двухуровневый кеш**: реестр кешируется на стороне сервера и клиента. Документация явно указывает: «Клиенты также имеют кеш регистраций Eureka в памяти, чтобы не обращаться к реестру при каждом запросе к сервису». Благодаря этому комбинация двух кешей и heartbeat-механизма делает Eureka достаточно устойчивой к сбоям. Сервис недоступен для обнаружения до тех пор, пока все три стороны (экземпляр, сервер, клиент) не синхронизируют метаданные в своих локальных кешах — это занимает около трёх heartbeat-циклов.

#### Ответ от Claude

**Как работает client-side service discovery с Eureka**

В модели **client-side discovery** (в отличие от server-side) клиент сам отвечает за получение списка экземпляров и выбор нужного. Это отличает её от подхода, где запросы маршрутизирует промежуточный балансировщик (как в server-side discovery).

**Компоненты**

- **Eureka Server** — реестр сервисов. Хранит таблицу `{serviceId -> [список экземпляров с host:port и метаданными]}`.
- **Eureka Client** (встроен в каждый сервис) — отвечает за регистрацию, отправку heartbeat и локальный кеш реестра.
- **Load Balancer** (Spring Cloud LoadBalancer или Ribbon) — выбирает конкретный экземпляр из кешированного списка по алгоритму (по умолчанию round-robin).

**Жизненный цикл сервиса**

1. **Регистрация**: при старте Eureka Client отправляет POST-запрос на Eureka Server с метаданными (host, port, app name, health check URL). Имя сервиса берётся из `spring.application.name`.

2. **Heartbeat**: каждые 30 секунд (по умолчанию, `eureka.instance.leaseRenewalIntervalInSeconds=30`) клиент отправляет heartbeat на сервер. Если сервер не получает heartbeat в течение `leaseExpirationDurationInSeconds` (по умолчанию 90 сек), запись помечается как просроченная и удаляется из реестра.

3. **Получение реестра**: клиент загружает полный реестр при старте, затем каждые 30 секунд получает дельту изменений (`eureka.client.registryFetchIntervalSeconds=30`). Реестр хранится локально в памяти.

4. **Обнаружение**: при вызове другого сервиса (например, через `@LoadBalanced RestTemplate` или Feign-клиент) Spring Cloud LoadBalancer обращается к локальному кешу Eureka-клиента, получает список экземпляров по `serviceId` и выбирает один — без сетевого вызова к Eureka Server.

**Что происходит, если Eureka Server недоступен**

Короткий ответ: **клиенты продолжают работу**, опираясь на локальный кеш реестра.

Детально:

- Eureka Client хранит актуальную копию реестра в памяти. При недоступности сервера обновления кеша прекращаются, но имеющиеся данные остаются. Клиент продолжает маршрутизировать трафик к экземплярам, которые были известны на момент последнего успешного обновления.
- Попытки обновить кеш повторяются каждые 30 секунд; как только сервер восстанавливается, кеш актуализируется.
- Если Eureka Server настроен в кластере (несколько узлов), клиент хранит список всех узлов и при недоступности одного переключается на следующий.

**Риски при недоступности сервера**

- **Устаревшие данные (stale registry)**: если пока Eureka Server недоступен, какие-то экземпляры упали, клиент не узнает об этом и может направлять запросы на несуществующие хосты. Это должен парировать circuit breaker (Resilience4j/Hystrix).
- **Новые экземпляры невидимы**: запущенные в период недоступности сервера новые инстансы не попадут в кеш клиента до восстановления сервера.

**Self-Preservation Mode на сервере**

Eureka Server имеет механизм самозащиты: если сервер перестаёт получать ожидаемое количество heartbeat-ов (например, из-за сетевых проблем), он переходит в режим **self-preservation** и прекращает удалять записи из реестра. Логика: лучше держать устаревшие данные, чем ошибочно удалить живые экземпляры при временном разрыве сети. Это настраивается через `eureka.server.enableSelfPreservation` (по умолчанию `true`).

**Итог по отказоустойчивости**

| Сценарий | Поведение клиентов |
|---|---|
| Eureka Server кратковременно недоступен | Работают на локальном кеше, запросы к другим сервисам продолжаются |
| Eureka Server недоступен длительно | Кеш устаревает; возможны запросы к упавшим экземплярам — нужен circuit breaker |
| Eureka Server восстановился | Кеш обновляется при следующем цикле (до 30 сек) |
| Новый экземпляр зарегистрировался | Виден клиентам через ~3 heartbeat-цикла (≈90 сек) |

### Вопрос 28. Что такое distributed tracing, какую роль играют traceId и spanId, и как Spring Cloud Sleuth / Micrometer Tracing автоматически пробрасывают их через HTTP и очереди сообщений?

**Категория:** Микросервисы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Distributed Tracing System: Spring Cloud Sleuth — DZone](https://dzone.com/articles/distributed-tracing-system-spring-cloud-sleuth-ope)

**TraceID** генерируется в момент входа запроса в систему и остаётся неизменным на всём его пути. **SpanID** меняется по мере того, как запрос переходит от одного сервиса к другому.

Трассы и спаны передаются между системами через HTTP-заголовки. Один из распространённых стандартов — **B3 Propagation**, описывающий формат заголовков для систем распределённой трассировки. Контекст передаётся через заголовки вида `X-B3-TraceId`, `X-B3-SpanId`, `X-B3-ParentSpanId`.

Spring Cloud Sleuth автоматически добавляет необходимые заголовки ко всем исходящим запросам, выполненным через `RestTemplate` или Feign, используя перехватчики (interceptors). Для этого HTTP-клиенты должны быть объявлены как Spring beans.

Пример: один запрос регистрации пользователя сохраняет единый TraceID, тогда как SpanID меняется при переходе между тремя микросервисами:
- User Service: `[user-service, 15f8d0cc9c8c9e4a, 15f8d0cc9c8c9e4a]`
- Bonus Service: `[bonus-service, 15f8d0cc9c8c9e4a, d8d79fe594bc4522]`
- Notification Service: `[notification-service, 15f8d0cc9c8c9e4a, 3d31b3eb8062cbe0]`

#### Ответ от Claude

**Distributed tracing** — механизм сквозного отслеживания запроса через несколько микросервисов, позволяющий восстановить полную цепочку вызовов, измерить задержку каждого участка и диагностировать ошибки в распределённой системе.

**Ключевые понятия:**

- **Trace** — полный путь одного запроса через систему, от точки входа до выхода. Идентифицируется уникальным **traceId** (128-битное число в стандарте OpenTelemetry).
- **Span** — единица работы внутри трейса: один сетевой вызов, запрос к БД, обработка сообщения. Каждый span имеет уникальный **spanId** и ссылку на parentSpanId (кроме корневого спана). Из иерархии спанов строится дерево вызовов (trace tree).
- **Context propagation** — механизм передачи traceId/spanId через границы сервисов.

**Форматы заголовков:**

Исторически Spring Cloud Sleuth использовал формат **B3 Propagation** (стандарт Zipkin):
```
X-B3-TraceId: 15f8d0cc9c8c9e4a
X-B3-SpanId: d8d79fe594bc4522
X-B3-ParentSpanId: 15f8d0cc9c8c9e4a
X-B3-Sampled: 1
```

Начиная со Spring Boot 3 и Micrometer Tracing (замена Sleuth), по умолчанию используется **W3C Trace Context** (стандарт IETF/OpenTelemetry):
```
traceparent: 00-15f8d0cc9c8c9e4a3b2a1c0d4e5f6789-d8d79fe594bc4522-01
tracestate: vendor-specific-data
```

**Spring Cloud Sleuth (Spring Boot 2) — как работает автопробрасывание:**

1. Sleuth добавляет `BeanPostProcessor`-инструментацию для `RestTemplate`, `WebClient`, Feign, `@Async`, JDBC и т.д.
2. При исходящем HTTP-запросе перехватчик записывает в заголовки текущий traceId/spanId.
3. На входящей стороне `TracingFilter` извлекает заголовки и восстанавливает контекст в MDC (Mapped Diagnostic Context) — переменные `traceId`/`spanId` становятся доступны в логах через `%X{traceId}`.

**Micrometer Tracing (Spring Boot 3) — актуальный подход:**

Spring Cloud Sleuth достиг конца жизненного цикла. Его функциональность перенесена в **Micrometer Tracing**, автоконфигурация встроена в Spring Boot 3. Tracer по умолчанию — OpenTelemetry, формат по умолчанию — W3C. Для сохранения совместимости с B3 нужна явная настройка:
```properties
management.tracing.propagation.type=b3
```

**Пробрасывание через очереди сообщений (Kafka, RabbitMQ):**

Асинхронные системы требуют ручной инструментации или использования поддерживаемых биндингов:

- **Spring Cloud Sleuth** поддерживал автоматическую инструментацию Spring Kafka и Spring AMQP: traceId/spanId добавлялись в заголовки сообщения (`MessageHeaders`).
- **Micrometer Tracing + OpenTelemetry** требует явного inject/extract через `Propagator` API для Kafka-продюсеров/консьюмеров, если нет готового интеграционного биндинга.

Паттерн для Kafka:
```java
// Producer: inject context into headers
propagator.inject(Context.current(), headers, (h, key, value) -> h.add(key, value.getBytes()));

// Consumer: extract context from headers
Context ctx = propagator.extract(Context.current(), headers, (h, key) -> new String(h.lastHeader(key).value()));
```

**Итог:** traceId объединяет все спаны одного запроса в единый граф; spanId идентифицирует конкретную операцию внутри этого графа. Sleuth/Micrometer Tracing берут на себя автоматическое добавление/чтение заголовков при HTTP-вызовах и предоставляют API для ручной инструментации в асинхронных сценариях.

### Вопрос 29. Почему eventual consistency — это не баг, а осознанный архитектурный выбор? Приведи конкретный сценарий, где кратковременная несогласованность данных допустима, и сценарий, где она недопустима.

**Категория:** Микросервисы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [CAP Theorem in Microservice Architecture — Codefinity](https://codefinity.com/blog/CAP-Theorem-in-Microservice-Architecture)

Eventual consistency — стратегия, при которой узлы временно могут хранить разные версии данных, но со временем все узлы сойдутся к согласованному состоянию.

**Допустимые сценарии (eventual consistency приемлема):**
- Ленты социальных сетей: лайки, комментарии могут кратковременно отображаться по-разному на разных серверах, не нарушая функциональность.
- Рекомендации в e-commerce: персонализированные подсказки на основе слегка устаревших данных о действиях пользователя остаются полезными.
- Распределение контента: статьи и медиафайлы могут постепенно реплицироваться по регионам.
- Чат-приложения: немедленная доставка сообщений важнее синхронизации статусов прочтения.

**Недопустимые сценарии (требуется строгая согласованность):**
- Финансовые транзакции: переводы денег требуют немедленной гарантированной согласованности для предотвращения овердрафта или двойных списаний.
- Управление складскими запасами: остатки товара должны мгновенно отражать актуальные данные во избежание пересортицы.
- Медицинские системы: записи о медикаментах требуют синхронной точности во избежание опасного взаимодействия препаратов.
- Системы аутентификации: учётные данные должны быть немедленно согласованы для предотвращения несанкционированного доступа.

#### Ответ от Claude

**Контекст: теорема CAP и BASE vs ACID**

Теорема CAP (Brewer, 2000) утверждает: распределённая система не может одновременно гарантировать все три свойства — **C**onsistency (согласованность), **A**vailability (доступность), **P**artition tolerance (устойчивость к разделению сети). В микросервисной архитектуре разделение сети (P) является данностью, поэтому выбор стоит между C и A.

Системы, выбирающие **AP** (доступность + устойчивость к разделению), принимают eventual consistency и работают по модели **BASE**:
- **B**asically Available — система отвечает на запросы всегда, даже с устаревшими данными
- **S**oft state — состояние может меняться со временем без входящих операций (в процессе конвергенции)
- **E**ventual consistency — данные согласуются в конечном счёте

Системы, выбирающие **CP** (согласованность + устойчивость к разделению), работают по модели **ACID** и могут отказывать в обслуживании при сетевых разделениях.

**Почему eventual consistency — это осознанный выбор, а не баг:**

1. **Физическое ограничение**: скорость света делает мгновенную глобальную согласованность невозможной. При репликации данных между дата-центрами в разных регионах задержка 50–200 мс неустранима.
2. **Компромисс производительности**: ожидание подтверждения записи от всех реплик (синхронная репликация) кратно увеличивает задержку операций записи.
3. **Доступность под нагрузкой**: при отказе части узлов AP-система продолжает работу; CP-система может стать недоступной.
4. **Бизнес-приемлемость**: многие бизнес-сценарии не требуют мгновенной согласованности — критична скорость отклика, а не абсолютная точность данных в данный момент.

**Сценарий 1: eventual consistency допустима**

*Отображение счётчика просмотров видео на YouTube/VK Video.*

Пользователь публикует видео. Счётчик просмотров увеличивается на каждом географически распределённом узле независимо. Разные пользователи в течение нескольких секунд могут видеть значения «1 234 567» и «1 234 891» — расхождение в несколько сотен просмотров. Через несколько секунд все узлы сходятся к единому значению.

**Почему допустимо:** бизнес не несёт убытков от кратковременного расхождения в счётчике; пользователи не принимают финансовых решений на основе точного числа просмотров; высокая доступность важнее точности.

**Сценарий 2: eventual consistency недопустима**

*Списание средств с банковского счёта.*

Пользователь переводит 50 000 рублей. Если транзакция сначала зафиксируется в одном узле, а затем будет реплицирована асинхронно, возможны:
- **Двойное списание** (duplicate payment): запрос был получен дважды, обе копии прошли проверку баланса до завершения первой транзакции.
- **Отрицательный баланс**: два параллельных перевода были проверены на основе устаревшего баланса.

**Почему недопустимо:** финансовые потери реальны и невосстановимы; регуляторные требования (PCI DSS, ЦБ РФ) обязывают к ACID-транзакциям; пользователь имеет законные ожидания мгновенной согласованности.

**Практические паттерны для eventual consistency:**

- **Saga pattern** — цепочка локальных транзакций с компенсирующими операциями при откате
- **Outbox pattern** — атомарная запись события в локальную БД и последующая публикация в брокер
- **CQRS** — разделение моделей чтения (может быть устаревшей) и записи (строгая согласованность)
- **Event sourcing** — хранение событий как источника истины; проекции строятся асинхронно

**Вывод:** eventual consistency — инструмент, выбор которого определяется бизнес-требованиями. Корректная архитектура явно документирует для каждого сервиса, какой уровень согласованности он гарантирует и почему.

### Вопрос 30. Какие задачи решает API Gateway помимо маршрутизации? Где проходит граница ответственности между API Gateway и самим микросервисом (аутентификация, авторизация, rate limiting, трансформация запросов)?

**Категория:** Микросервисы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [API gateways — Azure Architecture Center, Microsoft Learn](https://learn.microsoft.com/en-us/azure/architecture/microservices/design/gateway)

API Gateway — централизованная точка входа для управления взаимодействием между клиентами и микросервисами. Действует как обратный прокси и выполняет следующие cross-cutting задачи, которые иначе пришлось бы реализовывать в каждом сервисе отдельно:

**Три основных паттерна использования:**

1. **Gateway Routing** — обратный прокси для маршрутизации запросов к нужным сервисам по layer-7 правилам.
2. **Gateway Aggregation** — объединение нескольких запросов к разным сервисам в один ответ клиенту, снижая «болтливость» между клиентом и бэкендом.
3. **Gateway Offloading** — перенос cross-cutting функциональности на уровень шлюза, чтобы не дублировать её в каждом сервисе.

**Функции, которые можно вынести на Gateway (offloading):**
- SSL termination, mutual TLS
- Аутентификация
- IP allowlist / blocklist
- Rate limiting (throttling)
- Логирование и мониторинг
- Кэширование ответов
- Web Application Firewall (WAF)
- GZIP-сжатие
- Раздача статического контента

#### Ответ от Claude

**Задачи API Gateway за пределами маршрутизации:**

API Gateway — единая точка входа (single entry point) для всего внешнего трафика. Он централизует инфраструктурные задачи, которые иначе дублировались бы в каждом микросервисе.

**1. Аутентификация (Authentication)**

*Что делает Gateway:* проверяет подлинность клиента — валидирует JWT-токены (подпись, срок действия, issuer), API-ключи, OAuth 2.0 access tokens, сертификаты mTLS. При успешной проверке может обогатить запрос заголовками с информацией о пользователе (`X-User-Id`, `X-Roles`).

*Что делает микросервис:* получает уже верифицированные claims и доверяет им (trust but verify — проверка подписи токена опционально остаётся на стороне сервиса для defence-in-depth). Не реализует логику работы с identity provider.

*Граница:* Gateway отвечает за верификацию credentials; сервис — за использование identity в бизнес-логике.

**2. Авторизация (Authorization)**

Здесь граница менее однозначна и зависит от гранулярности:

*На уровне Gateway (coarse-grained authorization):* проверка «имеет ли этот роль/scope право вообще вызывать этот endpoint». Пример: `scope=read:orders` обязателен для `GET /orders`. Реализуется через OPA (Open Policy Agent), правила API Management и т.п.

*На уровне микросервиса (fine-grained authorization):* проверка «имеет ли пользователь X право читать конкретный ресурс Y». Пример: пользователь может видеть только свои заказы, а не чужие. Эта логика зависит от доменных данных и не может быть реализована на уровне шлюза.

*Принцип:* Gateway — первый рубеж, сервис — второй. Нельзя полагаться только на Gateway (defence-in-depth).

**3. Rate Limiting / Throttling**

*На уровне Gateway:* глобальные лимиты по IP, по API-ключу, по пользователю (например, 1000 req/min для базового тарифа, 10 000 req/min для премиум). Защита от DDoS и злоупотреблений до того, как запрос достиг сервиса.

*На уровне микросервиса:* защита от перегрузки со стороны других внутренних сервисов (east-west трафик не проходит через Gateway); бизнес-специфичные лимиты (например, не более 5 попыток входа в час).

**4. Трансформация запросов и ответов**

*Gateway:* изменение заголовков, преобразование протоколов (REST → gRPC), агрегация ответов нескольких сервисов в один, добавление корреляционных идентификаторов (`X-Request-Id`, `X-Trace-Id`), версионирование API (маршрутизация `/v1/` и `/v2/` к разным инстансам).

*Граница:* Gateway выполняет структурные/протокольные трансформации; сервис выполняет доменные трансформации (бизнес-правила маппинга данных).

**5. Прочие cross-cutting задачи Gateway:**
- SSL/TLS termination — расшифровка трафика на шлюзе, сервисы работают по HTTP внутри кластера
- Observability — единое место для сбора метрик, логов доступа, трейсов входящих запросов
- Кэширование ответов — снижение нагрузки на сервисы для идемпотентных запросов
- Circuit breaker — прекращение пробрасывания запросов к деградирующему сервису
- Web Application Firewall — защита от OWASP Top 10 атак

**Антипаттерн: «God Gateway»**

Вынесение бизнес-логики на уровень шлюза создаёт «умный шлюз — тупые сервисы», что нарушает принцип самодостаточности микросервисов, усложняет шлюз и превращает его в монолит. Gateway должен оставаться инфраструктурным компонентом.

**Сводная таблица ответственности:**

| Функция | Gateway | Микросервис |
|---|---|---|
| Верификация JWT-подписи | Да (основная) | Опционально (defence-in-depth) |
| Coarse-grained авторизация (роли/scopes) | Да | Нет |
| Fine-grained авторизация (доступ к ресурсу) | Нет | Да |
| Rate limiting по внешним клиентам | Да | Нет |
| Rate limiting по внутренним сервисам | Нет | Да |
| SSL termination | Да | Нет |
| Доменная бизнес-логика | Нет | Да |
| Валидация входных данных по схеме | Частично | Да |

### Вопрос 31. Как компенсирующие транзакции в Saga обеспечивают семантику «отмены», если промежуточные шаги уже отправили письма или списали деньги? Почему это не настоящий rollback?

**Категория:** Микросервисы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Saga Design Pattern — Azure Architecture Center](https://learn.microsoft.com/en-us/azure/architecture/patterns/saga)

Паттерн Saga управляет транзакциями, разбивая их на последовательность локальных транзакций. Если локальная транзакция завершается неудачно, сага выполняет серию компенсирующих транзакций, чтобы отменить изменения, сделанные предыдущими шагами.

Ключевые концепции:

- **Компенсируемые транзакции** (compensable transactions) могут быть отменены другой транзакцией с противоположным эффектом.
- **Точка невозврата** (pivot transaction) — шаг, после которого компенсируемые транзакции больше неактуальны. Все последующие действия должны быть завершены для достижения консистентного состояния.
- **Неотменяемые транзакции** (irreversible / noncompensable transactions) — не могут быть ни отменены, ни повторены.

В разделе «Problems and considerations» документ прямо указывает: **«Data can't be rolled back because saga participants commit changes to their respective databases»** — данные не могут быть откачены назад, так как участники саги уже зафиксировали изменения в своих базах данных. Также явно отмечается, что **«compensating transactions might not always succeed»**, что может оставить систему в несогласованном состоянии.

#### Ответ от Claude

**Почему компенсирующая транзакция — это не rollback**

В классических ACID-транзакциях rollback — это атомарная операция уровня СУБД: все изменения, сделанные в рамках транзакции, отбрасываются так, как будто их никогда не было. Этот механизм работает внутри единственного процесса и единственной базы данных — undo-лог позволяет восстановить предыдущее состояние страниц памяти точно и полностью.

В паттерне Saga каждый шаг — это отдельная, уже зафиксированная локальная транзакция в собственной базе данных своего сервиса. К моменту, когда начинается «откат», данные уже записаны и видны другим. Компенсирующая транзакция — это **новая прямая бизнес-операция**, которая логически инвертирует предыдущую: не удаление записи, а вставка новой противоположной записи; не возврат денег транспортным уровнем, а отдельная операция возврата в платёжной системе.

**Последствия для необратимых побочных эффектов**

Некоторые действия принципиально не могут быть «отменены» в техническом смысле:

- **Email уже отправлен** — письмо попало в почтовый ящик пользователя. «Компенсация» здесь — это отправка второго письма с текстом «Извините, ваш заказ отменён». Первое письмо при этом остаётся в истории переписки.
- **Деньги уже списаны** — платёж прошёл через банк. «Компенсация» — это отдельная операция возврата (refund), которая отображается в банковской выписке как отдельная строка. Между списанием и возвратом может пройти несколько дней.
- **Внешний API вызван** — например, курьерская служба уже приняла посылку в обработку. «Компенсация» — запрос на отзыв посылки, но курьер может уже быть в пути.

**Ключевые отличия компенсации от rollback**

| Характеристика | ACID Rollback | Saga Compensation |
|---|---|---|
| Механизм | Undo-лог в СУБД, атомарно | Новая бизнес-операция, может упасть |
| Видимость | Изменения скрыты с самого начала | Промежуточное состояние наблюдаемо другими |
| Следы | Никаких следов в базе | Запись в audit log, отдельные строки в таблицах |
| Побочные эффекты | Не возникают | Могут создавать новые побочные эффекты |
| Гарантия выполнения | Гарантирована СУБД | Должна обеспечиваться разработчиком (retry, идемпотентность) |
| Изоляция | Полная (другие транзакции не видят незафиксированные данные) | Отсутствует (dirty read между шагами саги возможен) |

**Практические следствия для разработчика**

1. **Компенсирующие транзакции должны быть идемпотентными**: при сбое сети они будут повторены, и повторный возврат денег или повторная отправка «компенсирующего» email недопустимы.
2. **Точка невозврата** (pivot transaction) должна быть явно определена: после неё компенсации невозможны, и saga должна двигаться только вперёд с retry-логикой.
3. **Необратимые операции** следует откладывать как можно дальше в цепочке шагов саги — ставить их последними, чтобы к моменту их выполнения вероятность отката была минимальной.
4. **Semantic lock** — один из практических паттернов противодействия: устанавливать флаг-семафор на записи до начала компенсируемой транзакции, чтобы другие саги не читали «грязные» данные.

Таким образом, Saga даёт **семантику eventually consistent undo**, а не атомарный rollback. Система после компенсации приходит в «логически эквивалентное исходному» состояние, но не в буквально то же самое — история операций сохраняется, а некоторые побочные эффекты остаются навсегда.

### Вопрос 32. Почему двухфазный коммит (2PC) практически не применяется в микросервисах? Какие конкретные проблемы с доступностью и производительностью он создаёт согласно теореме CAP?

**Категория:** Микросервисы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Two-Phase Commit Protocol: How It Works and When It Fails](https://distributedsystemauthority.com/two-phase-commit)

Двухфазный коммит нацелен на линеаризуемую (сильную) согласованность и находится на одном из крайних полюсов компромисса между согласованностью и доступностью, формализованного теоремой CAP.

Главная уязвимость протокола — **блокирующее поведение**: если координатор падает после отправки PREPARE, но до рассылки решения COMMIT/ABORT, все участники остаются заблокированными в неопределённом состоянии — они удерживают блокировки строк и ждут бесконечно. Протокол не допускает отказа координатора: консенсус достигается только тогда, когда все участники доступны и координатор переживает обе фазы.

По производительности: системы с требованием sub-100ms latency на P99-перцентиле, как правило, не способны поглотить синхронное удержание блокировок, которое навязывает 2PC.

Большинство реальных внедрений микросервисов поначалу пробуют 2PC через границы сервисов, но затем сталкиваются с его задержками и потерями доступности — и переходят к Saga или eventual consistency.

#### Ответ от Claude

**Что такое 2PC и как он работает**

Двухфазный коммит — это протокол распределённого согласия, в котором специальный узел-координатор управляет атомарным коммитом транзакции на нескольких участниках (когортах):

- **Фаза 1 (Prepare/Voting)**: координатор отправляет всем участникам сообщение PREPARE. Каждый участник записывает изменения в undo/redo лог, блокирует ресурсы и отвечает YES или NO.
- **Фаза 2 (Commit/Abort)**: если все ответили YES — координатор рассылает COMMIT; если хоть один ответил NO — рассылает ABORT.

**Проблемы с доступностью (CAP-анализ)**

Теорема CAP утверждает, что распределённая система может одновременно гарантировать не более двух из трёх свойств: Consistency (согласованность), Availability (доступность), Partition tolerance (устойчивость к разделению). 2PC — система класса **CP**: при сетевом разделении она жертвует доступностью ради согласованности.

Конкретные сценарии потери доступности:

1. **Падение координатора в фазе 2**: участники, ответившие YES, удерживают блокировки и не могут ни закоммититься, ни откатиться самостоятельно. Они вынуждены ждать восстановления координатора — потенциально бесконечно долго. Это называется **проблемой блокировки** (blocking problem).

2. **Сетевое разделение**: если координатор и часть когорты оказываются в разных сегментах сети, транзакция зависает. Согласно CAP, при партиции система должна выбрать между согласованностью и доступностью — 2PC всегда выбирает согласованность, делая систему недоступной.

3. **Каскадная недоступность**: в микросервисной архитектуре один недоступный сервис-участник блокирует всю транзакцию и удерживает ресурсы в других сервисах. Это противоречит принципу независимой отказоустойчивости сервисов.

**Проблемы с производительностью**

- **Количество сетевых round-trip**: минимум 2 round-trip между координатором и каждым участником (PREPARE + COMMIT). С учётом подтверждений — 4+ сетевых обмена на транзакцию.
- **Удержание блокировок через сеть**: блокировки на строках/записях удерживаются всё время, пока другие участники отвечают на PREPARE. Для геораспределённых сервисов это могут быть сотни миллисекунд.
- **Throughput**: из-за блокировок конкурентные транзакции не могут работать параллельно с перекрывающимися данными, что сильно снижает пропускную способность.
- **Latency**: накладные расходы 2PC недопустимы для user-facing API с требованием P99 < 100ms.

**Гетерогенность технологий**

Современные микросервисы используют разные базы данных: PostgreSQL, MongoDB, Redis, Kafka, сторонние API. Стандарт XA (распределённые транзакции) поддерживается только классическими реляционными СУБД. MongoDB, Redis и большинство NoSQL-систем не поддерживают XA, что делает 2PC технически неприменимым во многих микросервисных стеках.

**Нарушение принципов микросервисной архитектуры**

- **Тесная связанность**: все участники транзакции должны быть одновременно доступны, что разрушает независимую развёртываемость сервисов.
- **Единая точка отказа**: координатор является SPOF — его падение блокирует все активные транзакции.
- **Синхронное ожидание**: микросервисная архитектура стремится к асинхронному взаимодействию; 2PC требует жёсткой синхронизации.

**Альтернативы, принятые в индустрии**

На практике вместо 2PC применяют: паттерн Saga (для бизнес-транзакций), паттерн Transactional Outbox (для надёжной публикации событий), идемпотентных потребителей (для обработки дубликатов) и проектирование с eventual consistency как основной моделью согласованности.

### Вопрос 33. Как правильно обеспечить идемпотентность обработчика сообщений в микросервисе? Что конкретно нужно хранить, в каком хранилище, и как избежать состояния гонки при параллельной доставке одного и того же сообщения?

**Категория:** Микросервисы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Handling duplicate messages using the Idempotent consumer pattern](https://microservices.io/post/microservices/patterns/2020/10/16/idempotent-consumer.html)

Брокеры сообщений могут доставлять одно и то же сообщение повторно, поэтому обработчики сообщений должны использовать паттерн Idempotent Consumer для защиты от дубликатов.

Обработчик должен записывать **идентификаторы успешно обработанных сообщений**. Рекомендуемый способ для реляционных баз данных — отдельная таблица `PROCESSED_MESSAGE` с составным первичным ключом `(subscriberId, messageID)`. Попытка повторной вставки при дублирующей доставке вызывает нарушение ограничения уникальности, по которому обработчик понимает, что сообщение уже обработано, и безопасно подтверждает его без повторной обработки бизнес-логики.

Для NoSQL-баз данных идентификаторы сообщений можно встраивать непосредственно в обновляемые бизнес-сущности — этот подход не требует транзакций общего назначения.

Оптимизация для событий с монотонно возрастающими ID (например, из агрегатов Kafka): вместо хранения всех ID достаточно хранить наибольший обработанный ID от каждого экземпляра агрегата.

#### Ответ от Claude

**Почему идемпотентность необходима**

Все промышленные брокеры сообщений (Kafka, RabbitMQ, AWS SQS, ActiveMQ) гарантируют доставку **at-least-once**: сообщение будет доставлено минимум один раз, но может быть доставлено дважды и более — при перезапуске consumer, при сетевом сбое, при таймауте подтверждения. Обработчик, не учитывающий это, будет дублировать бизнес-операции: создавать два заказа, списывать деньги дважды, отправлять два письма.

**Что конкретно хранить**

Минимально необходимый набор данных для дедупликации:

| Поле | Назначение |
|---|---|
| `message_id` | Уникальный идентификатор сообщения из заголовка брокера |
| `consumer_id` | Идентификатор обработчика/очереди (один сервис может обрабатывать несколько топиков) |
| `status` | `IN_PROGRESS` / `COMPLETED` — для двухфазной резервации |
| `processed_at` | Временная метка — для TTL-очистки старых записей |
| `response_payload` | (опционально) Сохранённый результат для возврата при повторном запросе |

**В каком хранилище хранить**

Выбор хранилища определяется требованиями к атомарности операции:

**Реляционная СУБД (PostgreSQL, MySQL)** — предпочтительный вариант при наличии:
- Таблица `processed_messages(consumer_id, message_id, status, processed_at)` с `UNIQUE(consumer_id, message_id)`.
- Вставка записи выполняется **в той же транзакции**, что и бизнес-логика: `INSERT INTO processed_messages ... ON CONFLICT DO NOTHING`.
- Если `INSERT` вернул 0 затронутых строк — сообщение уже обработано, пропускаем.
- Ключевое свойство: атомарность базы гарантирует, что запись в `processed_messages` и изменение бизнес-данных либо оба произойдут, либо оба откатятся.

**Redis** — подходит для stateless-сервисов с коротким TTL дедупликации:
- `SET message:{id} "processed" NX EX 86400` — атомарная операция «установить, если не существует» с истечением через сутки.
- Проблема: Redis с настройкой `appendfsync everysec` может потерять последнюю секунду данных при падении. Для финансовых операций недостаточно надёжен без `appendfsync always`.
- Для составных операций (проверка + запись + бизнес-логика) необходимо использовать **Lua-скрипты** для атомарности, иначе возникает race condition.

**NoSQL (MongoDB, DynamoDB)** — через условные обновления:
- MongoDB: `findOneAndUpdate` с `$setOnInsert` и `upsert: true` — атомарная операция.
- DynamoDB: conditional write с `attribute_not_exists(message_id)`.

**Как избежать race condition при параллельной доставке**

Наивный подход «проверить → обработать → записать» содержит классическую **TOCTOU (Time-of-Check-to-Time-of-Use) уязвимость**:

```
Thread 1: SELECT — не найдено → продолжаем
Thread 2: SELECT — не найдено → продолжаем  (оба прошли!)
Thread 1: выполняем бизнес-логику → INSERT
Thread 2: выполняем бизнес-логику → INSERT  (дублирование!)
```

**Правильный подход — двухфазная атомарная резервация:**

**Фаза 1 — атомарная резервация** (до выполнения бизнес-логики):
```sql
INSERT INTO processed_messages (consumer_id, message_id, status, processed_at)
VALUES (?, ?, 'IN_PROGRESS', now())
ON CONFLICT (consumer_id, message_id) DO NOTHING;
```
- Если вставка успешна (1 строка) — продолжаем обработку.
- Если вставка не удалась (0 строк) — читаем существующую запись:
  - Статус `COMPLETED` → возвращаем сохранённый результат без обработки.
  - Статус `IN_PROGRESS` → конкурентный дубликат в обработке → ответить 409 Conflict с заголовком `Retry-After`.

**Фаза 2 — фиксация результата** (после успешной бизнес-логики, в той же транзакции):
```sql
UPDATE processed_messages SET status = 'COMPLETED', response_payload = ?
WHERE consumer_id = ? AND message_id = ?;
-- + commit бизнес-изменений в той же транзакции
```

**Дополнительные соображения**

- **TTL и очистка**: хранить все `message_id` вечно нецелесообразно. Устанавливать TTL равным максимальному окну повторной доставки брокера: для Kafka это обычно `retention.ms` топика, для SQS — `visibilityTimeout * maxReceiveCount`. Типичное значение — от 24 часов до 7 дней.

- **Интеграция с Transactional Outbox**: для полной гарантии exactly-once semantics на уровне системы идемпотентный consumer следует комбинировать с паттерном Transactional Outbox: входящее сообщение обрабатывается и исходящие события публикуются в единой транзакции с записью в `processed_messages`.

- **Kafka Idempotent Producer + Consumer**: на стороне Kafka можно включить `enable.idempotence=true` у producer (гарантирует exactly-once запись в один партишн) и использовать `isolation.level=read_committed` у consumer (не читает незакоммиченные транзакционные записи). Это сокращает, но не устраняет необходимость дедупликации на уровне приложения при использовании нескольких топиков или внешних систем.

- **Мониторинг**: счётчик задедуплицированных сообщений — важная операционная метрика; резкий рост указывает на проблемы с брокером или consumer group.
### Вопрос 34. Что такое паттерн Bulkhead в Resilience4j, чем SemaphoreBulkhead отличается от ThreadPoolBulkhead и в каком случае изоляция через отдельный пул потоков реально предотвращает каскадный сбой?

**Категория:** Микросервисы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Bulkhead — Resilience4j официальная документация](https://resilience4j.readme.io/docs/bulkhead)

Resilience4j предоставляет две реализации паттерна Bulkhead: `SemaphoreBulkhead`, использующую семафоры, и `FixedThreadPoolBulkhead`, использующую ограниченную очередь и фиксированный пул потоков.

`SemaphoreBulkhead` работает с разными моделями потоков и I/O. Она основана на семафоре и, в отличие от Hystrix, не предоставляет опции «теневого» пула потоков. Ответственность за корректный размер пула возлагается на клиента.

Ключевые параметры конфигурации:

- `SemaphoreBulkhead`: `maxConcurrentCalls` (по умолчанию 25) — максимальное число параллельных вызовов; `maxWaitDuration` (по умолчанию 0) — как долго поток ждёт разрешения.
- `ThreadPoolBulkhead`: `maxThreadPoolSize`, `coreThreadPoolSize`, `queueCapacity` (по умолчанию 100), `keepAliveDuration` (по умолчанию 20 мс).

Ограничивая параллельные вызовы на границах сервисов, bulkhead изолирует сбои: когда один сервис достигает лимита вызовов, запросы ставятся в очередь или отклоняются, не распространяясь далее по цепочке зависимостей.

#### Ответ от Claude

**Что такое Bulkhead**

Паттерн Bulkhead (переборка) заимствован из корабельного дела: корпус корабля разделён водонепроницаемыми переборками, и пробоина в одном отсеке не топит судно целиком. В микросервисах то же самое: ресурсы (потоки, семафорные разрешения) разбиваются на изолированные пулы для разных зависимостей, чтобы исчерпание ресурсов в одном пуле не затрагивало остальные.

**SemaphoreBulkhead**

Реализован на базе `java.util.concurrent.Semaphore`. Вызов выполняется **в том же потоке**, что и вызывающий код — семафор лишь ограничивает число одновременно вошедших. Когда все разрешения заняты, поток ждёт `maxWaitDuration` и, если не дождался, получает `BulkheadFullException`.

Характеристики:
- Минимальные накладные расходы (нет переключения контекста потоков).
- Подходит для синхронных вызовов и реактивных моделей (Reactor, RxJava), где каждый «виртуальный поток» не блокирует реальный OS-поток.
- Не обеспечивает изоляцию по потокам: если вызов блокируется надолго (например, медленный HTTP), он удерживает один из потоков общего пула приложения.

Конфигурация:
```java
BulkheadConfig config = BulkheadConfig.custom()
    .maxConcurrentCalls(10)
    .maxWaitDuration(Duration.ofMillis(100))
    .build();
```

**ThreadPoolBulkhead**

Реализован на базе `ThreadPoolExecutor` + `ArrayBlockingQueue`. Вызов **переносится в отдельный поток** из выделенного пула — вызывающий поток освобождается немедленно. Возвращает `CompletableFuture`.

Характеристики:
- Полная изоляция потоков: блокирующий вызов удерживает только поток из выделенного пула, не затрагивая пул приложения.
- Дополнительные расходы: переключение контекста, расход памяти (~50–200 МБ на пул), задержка 10–20 мс.
- Работает только с неблокирующим кодом со стороны вызывающего (возврат `CompletableFuture`).

Конфигурация:
```java
ThreadPoolBulkheadConfig config = ThreadPoolBulkheadConfig.custom()
    .maxThreadPoolSize(10)
    .coreThreadPoolSize(5)
    .queueCapacity(20)
    .keepAliveDuration(Duration.ofMillis(20))
    .build();
```

**Когда ThreadPoolBulkhead реально предотвращает каскадный сбой**

Классический сценарий каскадного сбоя через SemaphoreBulkhead:
- Приложение использует общий пул из 200 потоков (например, Tomcat thread pool).
- Сервис A вызывает медленный внешний Redis: каждый вызов блокирует поток на 30 секунд вместо обычных 5 мс.
- При 25 одновременных вызовах в Redis задействованы 25 потоков общего пула. Semaphore ограничивает вход, но 25 потоков уже заблокированы.
- Остальные 175 потоков продолжают обслуживать другие запросы — изоляция работает.
- Однако если `maxConcurrentCalls` выставлен слишком высоко (например, 150), медленный Redis способен занять большую часть пула и парализовать сервис.

ThreadPoolBulkhead устраняет этот риск принципиально иначе:
- Для Redis выделяется собственный `ThreadPoolExecutor` с `maxThreadPoolSize=10`.
- Все 10 потоков могут зависнуть в Redis — **потоки Tomcat при этом не заняты**: они вернули `CompletableFuture` и свободны для новых запросов.
- Когда очередь на 20 задач тоже заполняется, выбрасывается `BulkheadFullException` — вызывающий сервис получает быстрый отказ, а не блокировку.
- Все остальные функции (корзина, оплата, каталог) продолжают работу без деградации.

Таким образом, ThreadPoolBulkhead критически важен именно в **non-reactive (блокирующем) коде**, где потоки реально удерживаются во время ожидания ответа от зависимости. В reactive-стеке (WebFlux, Reactor), где потоки не блокируются, достаточно SemaphoreBulkhead.

**Итоговое сравнение**

| Характеристика | SemaphoreBulkhead | ThreadPoolBulkhead |
|---|---|---|
| Модель выполнения | Текущий поток | Выделенный пул потоков |
| Изоляция потоков | Нет | Да |
| Накладные расходы | Минимальные | Средние |
| Возврат | Синхронный | `CompletableFuture` |
| Подходит для | Reactive, лёгкие вызовы | Блокирующий I/O |
| Защита от thread starvation | Частичная | Полная |

### Вопрос 35. Опиши проблему «temporal coupling» между микросервисами. Как асинхронное взаимодействие через брокер решает её, и какие новые проблемы при этом появляются (порядок сообщений, back-pressure, dead-letter queue)?

**Категория:** Микросервисы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Asynchronous Messaging — Poornima Nayar](https://poornimanayar.co.uk/blog/asynchronous-messaging/)

**Temporal coupling** возникает, когда два взаимодействующих участника должны быть доступны одновременно. При синхронной коммуникации отправитель вынужден ждать ответа от получателя, прежде чем продолжить работу. Это похоже на телефонный звонок: оба участника должны присутствовать одновременно.

Асинхронный обмен сообщениями устраняет временну́ю зависимость: отправитель помещает сообщение в очередь и продолжает работу, не дожидаясь ответа. Получатель обрабатывает сообщения, когда готов.

**Проблема порядка сообщений:** очереди гарантируют доставку, но не порядок доставки. Повторные попытки доставки (retry) усугубляют ситуацию: системы нужно проектировать так, чтобы порядок сообщений не имел значения, либо применять FIFO-очереди.

**Буферизация нагрузки (аналог back-pressure):** в периоды пиковой нагрузки очередь работает как буфер, позволяя получателям вычитывать сообщения в собственном темпе, а не требуя масштабирования под пиковую мощность.

**Dead-letter queue:** сообщения, которые не удалось обработать, перемещаются в error queue или dead-letter queue. После исправления причины сбоя они могут быть повторно обработаны.

#### Ответ от Claude

**Temporal coupling: определение и проблема**

Temporal coupling (временна́я связанность) — это зависимость, при которой сервис A может выполнить свою операцию только в тот момент, когда сервис B доступен и способен ответить. При синхронном REST/gRPC-взаимодействии:

- Сервис A отправляет HTTP-запрос и блокируется в ожидании ответа.
- Если сервис B недоступен (перезапуск, сбой, высокая нагрузка), запрос A завершается ошибкой.
- Медленный ответ B прямо влияет на латентность A.
- При каскадных вызовах (A → B → C → D) недоступность D парализует всю цепочку.

Temporal coupling — это не просто «один сервис зависит от другого». Это зависимость по времени: операция невозможна, если в эту же секунду недоступен другой участник. Именно это делает синхронные микросервисные цепочки хрупкими.

**Как брокер сообщений решает temporal coupling**

При использовании брокера (Kafka, RabbitMQ, AWS SQS) сервис A публикует сообщение в топик/очередь и немедленно возвращает управление. Сервис B читает сообщение, когда готов — через секунду или через час. Брокер выступает буфером, который разрывает временну́ю зависимость:

- Сервис B может быть перезапущен — сообщения накапливаются в брокере.
- Деплой B не требует координации с A.
- Пиковая нагрузка на A не обязана совпадать с пропускной способностью B.

**Новые проблемы при асинхронном взаимодействии**

**1. Порядок сообщений (Message Ordering)**

Брокеры в общем случае не гарантируют строгий глобальный порядок. Причины: несколько партиций/шардов обрабатываются параллельно; retry-механизм может доставить более позднее сообщение раньше, чем повторно отправленное раннее; несколько consumer-инстансов читают одну очередь.

Последствия для бизнеса: если событие `OrderCancelled` обработается раньше `OrderCreated`, система может оказаться в некорректном состоянии.

Решения:
- Kafka: использовать partition key (например, `orderId`) — в пределах одной партиции порядок гарантирован.
- AWS SQS FIFO: гарантирует порядок внутри `MessageGroupId`, но ограничивает throughput.
- Идемпотентность и порядковые номера (sequence numbers) в payload — consumer сам обнаруживает и откладывает out-of-order сообщения.

**2. Back-pressure**

Back-pressure возникает, когда producer публикует сообщения быстрее, чем consumer успевает их обрабатывать. Брокер накапливает lag, потребляя память/диск. При неконтролируемом росте лага:
- Увеличивается задержка обработки (processing latency).
- Broker storage переполняется (особенно критично для in-memory брокеров, например RabbitMQ без persistence).
- Старые сообщения вытесняются, если настроена retention policy (Kafka).

Механизмы управления back-pressure:
- Горизонтальное масштабирование consumer-группы.
- Rate limiting на стороне producer (квоты Kafka, `x-max-length` в RabbitMQ).
- Реактивные стримы (Project Reactor, RxJava) с операторами `onBackpressureBuffer`/`onBackpressureDrop`.
- Мониторинг consumer lag (Kafka consumer group lag, CloudWatch metrics для SQS).

**3. Dead Letter Queue (DLQ)**

Не все сообщения можно обработать успешно: некорректный формат, ошибка бизнес-логики, недоступная база данных. Если сообщение бесконечно возвращается в очередь, оно становится «poison message» и блокирует остальные (особенно при FIFO). DLQ — это отдельная очередь, куда сообщение перемещается после исчерпания лимита попыток (`maxReceiveCount` в SQS, `x-death` в RabbitMQ).

Что нужно учесть:
- Настроить алертинг на рост DLQ — это сигнал о системной проблеме.
- Обеспечить инструмент для replay: после исправления дефекта сообщения из DLQ должны быть повторно обработаны.
- Сохранять оригинальный контекст (заголовки, timestamp, трассировочный ID) для диагностики.
- Использовать экспоненциальный backoff перед перемещением в DLQ, чтобы не перегружать починенный сервис одновременным replay всех неудачных сообщений.

**Сводная таблица компромиссов**

| Проблема | При синхронном взаимодействии | При асинхронном через брокер |
|---|---|---|
| Temporal coupling | Есть (оба должны быть онлайн) | Устранена |
| Сложность | Простой request/response | Усложнение операций: мониторинг lag, DLQ, replay |
| Видимость статуса | Синхронный ответ | Требует отдельных механизмов (correlation ID, saga) |
| Порядок | Гарантирован (HTTP) | Не гарантирован в общем случае |
| Back-pressure | Timeout/circuit breaker | DLQ, lag monitoring, scaling |
| Идемпотентность | Желательна | Обязательна (at-least-once delivery) |

Асинхронность не убирает сложность — она перераспределяет её из области доступности в область доставки, порядка и видимости.

### Вопрос 36. Как стратегия Strangler Fig помогает безопасно мигрировать монолит на микросервисы? Какие риски возникают при синхронизации данных в переходный период, пока старый и новый код сосуществуют?

**Категория:** Микросервисы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Strangler Fig Pattern — Azure Architecture Center, Microsoft Learn](https://learn.microsoft.com/en-us/azure/architecture/patterns/strangler-fig)

Паттерн Strangler Fig позволяет инкрементально мигрировать устаревшую систему, постепенно заменяя отдельные фрагменты функциональности новыми приложениями и сервисами. По мере переноса функциональности новая система в итоге перекрывает всю старую, а монолит можно безопасно вывести из эксплуатации.

**Реализация:**
1. Между клиентом, монолитом и новой системой вводится фасад (proxy). Изначально он направляет все запросы в монолит.
2. По мере миграции фасад постепенно переключает запросы на новые сервисы.
3. После переноса всей функциональности монолит деcommission'ится.
4. Фасад удаляется, клиент взаимодействует напрямую с новой системой.

**Проблемы и риски:**
- **Anti-corruption layer (ACL):** когда монолит вызывает уже мигрированный сервис, необходимо реализовать адаптер, который транслирует вызовы в новый интерфейс. Без ACL кросс-системные зависимости могут нарушить компоненты или вынудить новую систему принять устаревшие соглашения.
- **Синхронизация данных:** монолитные системы используют централизованную базу данных. При миграции отдельного сервиса требуется синхронизировать его domain-специфичные таблицы с монолитной БД через ETL + Change Data Capture (CDC). Данные в монолитной БД находятся в состоянии eventual consistency. Rollback возможен на этапе, пока domain-таблицы ещё существуют в монолитной БД; после их удаления откат значительно усложняется.
- **Фасад как единая точка отказа:** proxy-слой может стать bottleneck'ом или точкой отказа.

#### Ответ от Claude

**Суть паттерна Strangler Fig**

Название придумал Мартин Фаулер, вдохновившись тропическим инжиром (strangler fig), который прорастает вокруг дерева-хозяина и постепенно его вытесняет. В контексте миграции: новая система строится вокруг монолита, перехватывая запросы по одному функциональному домену, пока монолит не опустеет и не будет выведен из эксплуатации.

Альтернатива — «Big Bang rewrite» — переписывание с нуля — крайне рискованна: на время переписывания монолит заморожен, новые фичи не добавляются, а риск регрессий огромен.

**Поэтапная реализация**

**Этап 1. Введение фасада (proxy/façade)**
Перед монолитом ставится API Gateway или reverse proxy. Изначально 100% трафика идёт в монолит — поведение системы не меняется. Клиенты не знают о начале миграции.

**Этап 2. Выбор первого домена для миграции**
Применяются принципы Domain-Driven Design (DDD) и Event Storming для выявления Bounded Context'ов. Хороший кандидат для первой миграции: домен с наименьшим числом зависимостей внутри монолита, высокой бизнес-ценностью или явной технологической проблемой.

**Этап 3. Параллельная работа (Strangler период)**
Новый микросервис реализует функциональность домена. Фасад начинает маршрутизировать запросы к нему. Монолит продолжает обслуживать остальные домены. Для обратных вызовов монолита к мигрированному сервису реализуется Anti-Corruption Layer (ACL) — адаптер/фасад внутри монолита.

**Этап 4. Decommission монолита**
После переноса всех доменов монолит отключается, фасад может быть удалён или оставлен как постоянный API Gateway.

**Риски синхронизации данных в переходный период**

Это самая сложная часть миграции. Пока монолит и новый сервис сосуществуют, они могут работать с перекрывающимися данными.

**1. Dual-write проблема**

Если оба сервиса пишут в разные БД (монолитная PostgreSQL и новая domain-БД), нужно обеспечить согласованность между ними. Наивный dual-write (писать в обе БД в одном запросе) ненадёжен: нет атомарности — запись может успешно пройти в одну БД и упасть в другую.

Решения:
- **CDC (Change Data Capture)** через Debezium: читает transaction log монолитной БД и асинхронно реплицирует изменения в новую БД. Eventual consistency, но надёжно.
- **Event-driven синхронизация**: монолит публикует доменные события через брокер, новый сервис потребляет их для обновления своей БД.
- **Synchronizing agent**: выделенный компонент слушает очередь событий от нового сервиса и обновляет монолитную БД (описано в AWS Prescriptive Guidance).

**2. Eventual consistency**

Пока работает репликация, данные в монолитной и микросервисной БД временно расходятся. Если монолит читает свою копию данных сразу после записи через новый сервис, он может увидеть устаревшее значение. Необходимо:
- Явно задокументировать, что данные eventually consistent.
- Проверить, допустимо ли это для бизнес-логики монолита.
- Не строить бизнес-критичные синхронные операции поверх eventual consistent данных.

**3. Утечка синхронизации («Permanent dual-write pain»)**

Если синхронизация задумывалась как тактическое решение на период перехода, но реальные сроки миграции затянулись, она превращается в постоянную операционную нагрузку. Это один из главных практических рисков — техническая задолженность, которая начинает стоить дороже самой миграции.

**4. Откат становится сложнее**

По данным Microsoft Azure Architecture Center: откат возможен на этапе, пока domain-таблицы и CDC-процесс ещё существуют в монолитной БД. После удаления domain-таблиц откат требует восстановления объектов БД и replay всех изменений — это резко увеличивает риск и стоимость. Удаление legacy-объектов должно быть намеренным финальным шагом.

**5. Прокси как единая точка отказа**

Фасад, маршрутизирующий запросы, сам становится критическим компонентом. При его падении недоступны и монолит, и новые сервисы. Смягчение: использовать managed API Gateway (AWS API Gateway — serverless, multi-AZ), настроить circuit breaker на уровне фасада, обеспечить health-check и автоматический failover.

**Ключевые инструменты переходного периода**

| Задача | Инструмент |
|---|---|
| Маршрутизация запросов | API Gateway, Nginx, Envoy |
| Трансляция интерфейсов | Anti-Corruption Layer (ACL) |
| Репликация данных | CDC (Debezium), ETL |
| Потоковая синхронизация | Kafka, AWS SQS |
| Валидация согласованности | Теневое сравнение (shadow mode) |
| Выявление bounded contexts | Domain-Driven Design, Event Storming |

**Когда паттерн не подходит**

- Запросы к монолиту нельзя перехватить (нет доступа к инфраструктуре).
- Нет доступа к исходному коду монолита (нельзя встроить ACL и отключить мигрированную функциональность).
- Система настолько мала, что проще переписать целиком.
- Бизнес требует немедленного полного перехода (нет времени на инкрементальную миграцию).
### Вопрос 37. Что такое паттерн CQRS (Command Query Responsibility Segregation)? Чем модель записи (Command side) отличается от модели чтения (Query side), и какие проблемы согласованности возникают при асинхронном обновлении read-модели?

**Категория:** Микросервисы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Microservices Pattern: CQRS](https://microservices.io/patterns/data/cqrs.html)

CQRS (Command Query Responsibility Segregation) — паттерн, при котором используется разная модель для записи информации и для её чтения.

**Command side (модель записи)** — принимает команды на изменение состояния системы. Она содержит бизнес-логику, производит валидацию и публикует доменные события. Хранилище оптимизировано под транзакционные операции (CRUD).

**Query side (модель чтения)** — содержит read-only реплику данных, спроектированную специально под конкретные запросы или группы запросов. Приложение поддерживает эту базу данных актуальной, подписываясь на доменные события, публикуемые command side. Read-сторона часто использует NoSQL (документные хранилища, key-value) — технологию, максимально удобную для выполнения конкретных запросов.

**Проблемы согласованности:** поскольку синхронизация происходит асинхронно через события, возникает «replication lag» — временной разрыв, в течение которого read-модель отражает устаревшее состояние. Это называется «eventual consistency» (итоговая согласованность). Приложение должно быть спроектировано так, чтобы клиент понимал, что данные могут быть не полностью актуальными.

#### Ответ от Claude

**CQRS (Command Query Responsibility Segregation)** — архитектурный паттерн разделения ответственности между моделью записи (Command side) и моделью чтения (Query side). Идея восходит к принципу CQS Бертрана Мейера: метод либо изменяет состояние, либо возвращает данные, но не то и другое одновременно. CQRS применяет этот принцип на уровне архитектурных компонентов.

**Command side (сторона команд):**
- Принимает команды (намерения изменить состояние): `CreateOrder`, `CancelOrder`.
- Содержит богатую доменную модель и бизнес-логику.
- Валидирует данные, применяет правила, изменяет состояние агрегата.
- Публикует доменные события (`OrderCreated`, `OrderCancelled`) в шину событий.
- Хранилище — реляционная БД или event store, оптимизированная под запись.

**Query side (сторона запросов):**
- Потребляет доменные события от command side.
- Поддерживает одну или несколько денормализованных read-моделей (materialized views).
- Хранилище выбирается исходя из потребностей конкретного запроса: Redis, Elasticsearch, MongoDB, PostgreSQL.
- Не содержит бизнес-логики — только проекции событий в view.
- Обновляется асинхронно: event handler подписывается на события и обновляет read-модель.

**Проблемы согласованности при асинхронном обновлении:**

1. **Eventual consistency (итоговая согласованность):** между моментом успешного выполнения команды и обновлением read-модели существует временной лаг. Клиент, выполнивший запись, может тут же прочитать устаревшие данные.

2. **Read-your-own-writes:** пользователь, создавший заказ, сразу запрашивает его список и не видит новый заказ — read-модель ещё не обновилась. Решения: версионирование ответа команды (клиент ждёт, пока read-модель достигнет нужной версии), или показ оптимистичного UI без обращения к read-модели.

3. **Порядок событий:** при сбое или повторной доставке события возможна обработка событий не в том порядке. Требуется идемпотентность обработчиков и контроль порядка (sequence numbers).

4. **Неполная обработка при сбое consumer:** если consumer упал после записи события в command side, но до обновления view — read-модель становится рассинхронизированной. Решение: transactional outbox pattern + at-least-once delivery + идемпотентность.

**Когда применять CQRS:**
- Сложный домен, где читающая и пишущая модели существенно различаются по структуре.
- Высокая нагрузка на чтение, требующая независимого масштабирования.
- Запросы, охватывающие данные нескольких сервисов (в связке с API Composition или Event Sourcing).

**Предостережение (Martin Fowler):** CQRS — не универсальный паттерн. Он существенно усложняет архитектуру. Для большинства систем CRUD-модели достаточно; CQRS оправдан только при реальной необходимости.

### Вопрос 38. Как работает паттерн API Composition для реализации запросов, охватывающих данные из нескольких микросервисов? В каких ситуациях он неприменим и почему в таких случаях переходят на CQRS с денормализованной read-моделью?

**Категория:** Микросервисы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Microservices Pattern: API Composition](https://microservices.io/patterns/data/api-composition.html)

Паттерн API Composition реализует запросы через **API Composer** — компонент (часто API Gateway или специальный сервис), который вызывает сервисы, владеющие нужными данными, и выполняет их объединение в памяти (in-memory join).

**Пример:** запрос «получить информацию о покупателе» может требовать данных из `OrderService`, `CustomerDetailsService`, `SupportService`. API Composer вызывает каждый из них, собирает ответы и возвращает клиенту единый результат.

**Ограничения:**
- Некоторые запросы приводят к неэффективному объединению больших наборов данных в памяти.
- Доступность всей системы снижается: чем больше сервисов задействовано, тем выше вероятность сбоя одного из них.
- Рост числа обращений к базам данных увеличивает сетевой трафик и операционные расходы.

#### Ответ от Claude

**API Composition** — паттерн реализации составных запросов в микросервисной архитектуре, при котором специальный компонент (**API Composer**) обращается к нескольким сервисам-владельцам данных и объединяет результаты в памяти.

**Как работает:**

1. Клиент отправляет запрос к API Composer (им может быть API Gateway, BFF-сервис или Lambda-функция).
2. API Composer параллельно (или последовательно, если есть зависимости) вызывает каждый из сервисов по их собственному API.
3. Каждый сервис возвращает данные из своей базы данных.
4. API Composer выполняет in-memory join — объединяет данные и формирует единый ответ клиенту.

**Пример:** запрос «детали заказа с профилем клиента и историей поддержки» требует вызовов `OrderService`, `CustomerService`, `SupportService`. API Composer собирает три ответа, склеивает их по `customerId` и возвращает единый объект.

**Когда паттерн неприменим:**

1. **Большие объёмы данных с пагинацией и сортировкой:** если нужно вернуть первые 20 заказов клиента, отсортированных по нескольким атрибутам из разных сервисов, API Composer должен загрузить все данные в память, чтобы корректно отсортировать и выбрать страницу. При миллионах записей это неприемлемо.

2. **Сложные аналитические запросы:** агрегации, фильтрации и группировки по полям из нескольких сервисов невозможно эффективно выполнить в памяти composer'а.

3. **Высокая нагрузка:** каждый входящий запрос генерирует N исходящих запросов к N сервисам. При высоком RPS это создаёт каскадную нагрузку.

4. **Снижение доступности:** система деградирует, если хоть один из N сервисов недоступен (если не применяется circuit breaker с fallback).

**Почему в таких случаях переходят на CQRS с денормализованной read-моделью:**

CQRS с materialized view решает проблему принципиально иначе: данные из нескольких сервисов заранее агрегируются в одно хранилище, оптимизированное под конкретные запросы.

- Каждый сервис публикует доменные события (`OrderCreated`, `CustomerUpdated`).
- Event handler на стороне read-модели подписывается на все нужные события и обновляет денормализованную таблицу/документ.
- Запрос клиента обращается к единому read-хранилищу — без межсервисных вызовов, без in-memory join.
- Read-хранилище можно сделать на Elasticsearch (для полнотекстового поиска), Redis (для быстрого lookup), или PostgreSQL с материализованными представлениями.

**Компромисс:** CQRS с read-моделью обеспечивает лишь eventual consistency — данные обновляются асинхронно через события, поэтому read-модель может на короткое время отставать от состояния command side. API Composition, напротив, читает актуальные данные прямо из источников (strong consistency в рамках каждого сервиса).

**Правило выбора:** API Composition — первый выбор для простых, малообъёмных запросов. CQRS с denormalized view — когда требуются сложные запросы, большие объёмы, агрегации или высокая производительность чтения.

### Вопрос 39. Что такое Event Sourcing? Как из потока событий восстанавливается текущее состояние агрегата, и что такое «снапшот» (snapshot) — зачем он нужен и как влияет на производительность чтения?

**Категория:** Микросервисы · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Microservices Pattern: Event Sourcing](https://microservices.io/patterns/data/event-sourcing.html)

Event Sourcing сохраняет состояние бизнес-сущности (например, `Order` или `Customer`) в виде последовательности событий изменения состояния. Для восстановления текущего состояния приложение воспроизводит (replays) эти события.

Например, события `Customer` могут включать: `CustomerCreated`, `CustomerRelocated`, `CustomerSuspended`, `CustomerReactivated`.

**Снапшот:** некоторые сущности могут иметь большое количество событий, что делает загрузку неэффективной. Чтобы оптимизировать загрузку, приложение может периодически сохранять снапшот текущего состояния сущности. Для восстановления текущего состояния приложение находит последний снапшот и события, произошедшие после него. В результате для воспроизведения требуется значительно меньше событий.

#### Ответ от Claude

**Event Sourcing** — паттерн персистентности, при котором состояние агрегата хранится не в виде текущего снимка (строка в таблице), а в виде неизменяемого журнала событий (event log / event store). Каждое событие описывает факт, произошедший в домене: `OrderPlaced`, `OrderItemAdded`, `OrderShipped`, `OrderCancelled`.

**Как восстанавливается состояние агрегата:**

1. Загружаются все события агрегата из event store в хронологическом порядке.
2. Агрегат инициализируется в начальном (пустом) состоянии.
3. Каждое событие последовательно применяется к агрегату через метод `apply(event)`, мутируя его состояние.
4. После прохода по всем событиям агрегат находится в актуальном текущем состоянии.

Пример для `Order`:
```
OrderPlaced       → статус: PENDING,  сумма: 1000
OrderItemAdded    → позиций: 2,       сумма: 1500
PaymentConfirmed  → статус: PAID
OrderShipped      → статус: SHIPPED
```
Итоговое состояние: `SHIPPED`, 2 позиции, сумма 1500.

**Что такое снапшот (snapshot):**

Снапшот — сохранённый «срез» состояния агрегата в конкретный момент времени (после N-го события). Он содержит полное состояние агрегата и порядковый номер последнего учтённого события (sequence number).

**Зачем нужен снапшот:**

Долгоживущие агрегаты (счёт клиента за годы, складской остаток с тысячами транзакций) могут накапливать тысячи событий. Воспроизведение всего журнала при каждом обращении становится дорогостоящим по времени и ресурсам.

**Алгоритм загрузки агрегата со снапшотом:**

1. Загрузить последний снапшот агрегата (если есть).
2. Применить снапшот как начальное состояние агрегата.
3. Загрузить только события с `sequenceNumber > snapshot.sequenceNumber`.
4. Применить оставшиеся события к агрегату.

**Влияние снапшотов на производительность:**

- **Ускорение чтения:** вместо воспроизведения 10 000 событий воспроизводится, например, 50 (события после последнего снапшота). Время загрузки снижается на порядки.
- **Прозрачность:** снапшоты — оптимизация, не влияющая на корректность. Event log остаётся полным и неизменным источником истины; снапшоты можно пересоздать в любой момент.
- **Стратегии создания снапшотов:**
  - По числу событий: создавать снапшот каждые N событий (например, каждые 100).
  - По расписанию: периодически (ночью), не прерывая работу системы (Martin Fowler: снапшоты создаются параллельно без остановки приложения).
  - По запросу: явно инициируется администратором при деградации производительности.
- **Стоимость хранения:** снапшоты занимают дополнительное место, но обычно их объём значительно меньше полного event log.

**Расхождение между источниками:** microservices.io акцентирует снапшоты как решение проблемы большого числа событий у конкретного агрегата. Martin Fowler дополнительно отмечает, что снапшоты могут создаваться асинхронно, не прерывая работу системы, и что их можно пересоздать заново, перепроиграв весь event log — это важно при смене структуры снапшота.

**Связь с CQRS:** Event Sourcing часто используется совместно с CQRS. Event store является command side; события публикуются во внешнюю шину (Kafka, RabbitMQ), query side подписывается на них и строит денормализованные read-модели (materialized views). Снапшоты относятся только к command side и не затрагивают read-модели.

### Вопрос 40. Как в Event Sourcing-системе реализовать версионирование схемы событий? Что такое «upcasting» события, и как обрабатывать устаревшие события при replay, если их структура изменилась?

**Категория:** Микросервисы · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Event Versioning — Axon Framework Reference Guide](https://docs.axoniq.io/axon-framework-reference/4.11/events/event-versioning/)

В жизненном цикле Event Sourcing-приложения события неизбежно меняют свой формат. Поскольку события хранятся бессрочно, приложение должно уметь работать с несколькими версиями одного события.

**Revision (ревизия).** Event Store сохраняет рядом с каждым событием его ревизию и полное имя типа. Ревизия генерируется `RevisionResolver`; Axon предоставляет четыре реализации:
- `AnnotationRevisionResolver` — читает аннотацию `@Revision` на классе события;
- `SerialVersionUIDRevisionResolver` — использует `serialVersionUID` Java Serialization API;
- `FixedValueRevisionResolver` — возвращает фиксированное значение (удобно при версионировании всего приложения);
- `MavenArtifactRevisionResolver` — автоматически берёт версию из Maven-артефакта.

**Upcasting.** Концепция upcasting применяется к событиям точно так же, как в Java она применялась к сериализованным объектам: апкастер преобразует событие ревизии `x` в событие ревизии `x + 1`. Апкастеры выстраиваются в цепочку (`UpcasterChain`), и приложение работает исключительно с последней версией. Базовый интерфейс принимает `Stream<IntermediateEventRepresentation>` и возвращает `Stream<IntermediateEventRepresentation>`.

**IntermediateEventRepresentation** предоставляет доступ ко всем полям EventMessage, но разрешает изменять только три вещи: payload, тип payload и метаданные. Неизменяемые поля (идентификатор агрегата, временная метка и т.п.) защищены от случайной модификации.

**Реализации апкастеров** для разных сценариев:
- `SingleEventUpcaster` — преобразование «один к одному» (наиболее распространённый случай);
- `EventMultiUpcaster` — преобразование «один ко многим» (разбиение одного события на несколько);
- `ContextAwareSingleEventUpcaster` — сохраняет состояние между событиями потока;
- `ContextAwareEventMultiUpcaster` — сочетает оба предыдущих варианта;
- `EventTypeUpcaster` — специализирован для переименования типа события.

**Порядок регистрации** важен: в Spring Boot апкастеры упорядочиваются аннотацией `@Order` либо регистрируются программно через `Configurer.registerEventUpcaster()` или бином `EventUpcasterChain`.

#### Ответ от Claude

**Проблема.** В Event Sourcing события хранятся вечно и неизменно, однако схема данных эволюционирует. Это порождает ключевое противоречие: событие, записанное три года назад, должно корректно воспроизводиться (replay) в сегодняшней версии системы.

**Стратегии версионирования**

Существует несколько подходов, которые не исключают, а дополняют друг друга:

1. **Upcasting (основная стратегия).** При чтении из Event Store старые события преобразуются «на лету» к актуальной схеме прямо перед передачей в обработчик. Сами данные в хранилище не изменяются. Апкастеры выстраиваются в цепочку: v0 → v1 → v2 → ... → vN. Обработчики событий всегда работают только с последней версией.

2. **Tolerant Reader (слабая схема).** Потребитель проектируется так, чтобы игнорировать неизвестные поля и подставлять дефолтные значения для отсутствующих. Это покрывает аддитивные, не ломающие изменения (добавление нового необязательного поля) без написания явного апкастера. Формат JSON/XML хорошо поддерживает этот подход.

3. **Copy-and-Replace / Stream Transformation (по Greg Young).** Если изменения слишком радикальны, создаётся новый поток событий: старые события читаются, трансформируются и записываются как новые. Исходный поток после этого может быть заархивирован. Это разовая «миграция данных» в терминах Event Sourcing.

4. **Lazy Transformation.** Апкастер не только конвертирует событие для обработчика, но и записывает результат обратно в хранилище. Трансформация происходит один раз при первом чтении. Минус — нарушается иммутабельность лога.

**Как именно реализуется upcasting**

Каждый апкастер объявляет, какую ревизию он принимает и какую возвращает. Event Store сохраняет рядом с каждым событием его ревизию и полное имя типа (`@Revision("2")` в Axon). При replay движок ищет подходящий апкастер для каждой пары (тип, ревизия) и прогоняет событие через цепочку до актуальной версии.

Пример (Axon Framework):
```java
@Revision("1")
public class UserRegistered { ... } // старая схема: поле String timestamp

@Revision("2")
public class UserRegistered { ... } // новая схема: поле LocalDate registrationDate

// Апкастер v1 -> v2
public class UserRegisteredUpcaster extends SingleEventUpcaster {
    @Override
    protected boolean canUpcast(IntermediateEventRepresentation ir) {
        return ir.getType().equals(MetaData.from(UserRegistered.class))
               && "1".equals(ir.getType().getRevision());
    }
    @Override
    protected IntermediateEventRepresentation doUpcast(IntermediateEventRepresentation ir) {
        return ir.upcastPayload(
            MetaData.from(UserRegistered.class, "2"),
            ObjectNode.class,
            payload -> {
                String ts = payload.get("timestamp").asText();
                payload.remove("timestamp");
                payload.put("registrationDate", ts.substring(0, 10));
                return payload;
            }
        );
    }
}
```

**Правила безопасного версионирования**

- Добавление нового необязательного поля — не конфликт версий; апкастер не нужен (достаточно Tolerant Reader).
- Переименование поля, удаление поля, изменение его смысла — требуют явного апкастера и смены ревизии.
- Если преобразование невозможно (семантика события изменилась кардинально) — это уже новый тип события, а не новая версия старого.

**Расхождение источников:** Axon Framework пропагандирует upcasting как основной механизм и предлагает для него богатый API. Greg Young (InfoQ, 2017) рассматривает Stream Transformation как полноценную альтернативу для радикальных изменений и предостерегает от чрезмерного усложнения цепочек апкастеров. Оба источника единодушны в том, что хранилище событий должно оставаться иммутабельным в штатном режиме.
