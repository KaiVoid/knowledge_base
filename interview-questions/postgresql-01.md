### Вопрос 1. Что такое MVCC (Multi-Version Concurrency Control)? Как MVCC позволяет читателям не блокировать писателей? Где хранятся старые версии строк в PostgreSQL и InnoDB?

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation — Multi-Version Concurrency Control](https://www.postgresql.org/docs/current/mvcc-intro.html)

PostgreSQL использует MVCC для поддержания согласованности данных. Каждый SQL-оператор видит снимок данных таким, каким он был в определённый момент времени, вне зависимости от текущего состояния базы данных. Это защищает операторы от просмотра несогласованных данных, которые могут возникнуть при параллельном изменении одних и тех же строк.

Главное преимущество MVCC перед традиционными блокировками: «В MVCC блокировки, получаемые для чтения, не конфликтуют с блокировками, получаемыми для записи, поэтому чтение никогда не блокирует запись, а запись никогда не блокирует чтение».

#### Ответ от Claude

**Что такое MVCC**

MVCC (Multi-Version Concurrency Control) — механизм управления конкурентным доступом, при котором СУБД хранит несколько версий одной строки одновременно. Когда транзакция изменяет строку, создаётся новая версия; старая версия сохраняется для транзакций, которым нужно видеть данные до этого изменения. Каждая транзакция работает со своим согласованным «снимком» базы данных.

**Почему читатели не блокируют писателей**

В классической модели с блокировками (`Two-Phase Locking`) чтение и запись одной строки конфликтуют: писатель ставит эксклюзивную блокировку, читатель ждёт её снятия (и наоборот).

В MVCC читатель не блокирует писателя, потому что они работают с разными версиями строки:
- Писатель создаёт новую версию строки
- Читатель продолжает видеть старую версию, которая была актуальна в момент старта его снимка

Ни одна сторона не ждёт другую. Это кардинально увеличивает пропускную способность при высокой конкурентности.

---

**Где хранятся старые версии строк**

PostgreSQL и InnoDB применяют принципиально разные подходы:

**PostgreSQL — хранение в heap (куче таблицы)**

Все версии строки хранятся прямо в табличных страницах (heap). Каждый tuple (физическая строка) содержит системные поля:
- `xmin` — ID транзакции, которая создала эту версию строки (вставка или обновление)
- `xmax` — ID транзакции, которая удалила/обновила строку (если 0 — строка «живая»)

При обновлении строки:
1. Старый tuple получает `xmax = ID текущей транзакции`
2. Рядом в той же странице (или в новой) создаётся новый tuple с `xmin = ID текущей транзакции`

Старые версии остаются в heap до тех пор, пока процесс `VACUUM` их не удалит. Без регулярного VACUUM таблицы разбухают («table bloat»).

**InnoDB — хранение в undo log**

InnoDB хранит в основной таблице только актуальную версию строки, а предыдущие версии — в отдельной области, называемой **undo log** (сегмент отката). Каждая строка содержит два скрытых системных поля:
- `DB_TRX_ID` — ID транзакции, последней изменившей строку
- `DB_ROLL_PTR` — указатель на запись в undo log (предыдущую версию)

Таким образом формируется **цепочка версий** (version chain): из актуальной строки по `DB_ROLL_PTR` можно перейти к предыдущей версии, затем к ещё более ранней и т.д. При необходимости прочитать нужную версию читатель проходит по этой цепочке до тех пор, пока не найдёт версию, видимую в его снимке.

Старые записи undo log удаляются **purge-потоком** InnoDB автоматически, без явного вмешательства администратора.

---

**Сравнительная таблица**

| Аспект | PostgreSQL | InnoDB (MySQL) |
|---|---|---|
| Хранение старых версий | В heap-страницах таблицы (рядом с актуальной) | В undo log, ссылка через DB_ROLL_PTR |
| Запись delta/full | Полная копия tuple | Только изменённые значения (дельта) |
| Очистка устаревших версий | VACUUM (явный/autovacuum) | Автоматический purge-поток |
| Риск при откладывании очистки | Table bloat — раздувание таблицы | Раздувание undo tablespace |
| Маркеры видимости | xmin, xmax в заголовке tuple | DB_TRX_ID, DB_ROLL_PTR в строке |

**Важное следствие**: долго живущие транзакции опасны в обеих СУБД — пока активна хотя бы одна транзакция, которой нужна старая версия, СУБД не может её удалить. В PostgreSQL это приводит к накоплению мёртвых tuple в heap; в InnoDB — к росту undo log.

### Вопрос 2. Как PostgreSQL реализует изоляцию транзакций через MVCC? Почему в PostgreSQL фактически нет уровня Read Uncommitted и почему на уровне Repeatable Read отсутствуют фантомные чтения?

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation — Multiversion Concurrency Control](https://www.postgresql.org/docs/current/mvcc.html)

Согласно документации PostgreSQL, MVCC обеспечивает каждому запросу/транзакции согласованный снимок данных: читающие не блокируют пишущих, а пишущие — читающих. Каждая версия строки помечается идентификаторами транзакций (xmin/xmax), и видимость версии определяется снимком транзакции. Read Uncommitted в PostgreSQL ведёт себя как Read Committed (грязное чтение невозможно), а на уровне Repeatable Read используется снимок на момент начала транзакции, что исключает и неповторяющиеся, и фантомные чтения — строже, чем требует стандарт SQL.

#### Ответ от Claude

**MVCC (Multiversion Concurrency Control):** вместо блокировки строк на чтение PostgreSQL хранит несколько версий строки. Каждая версия имеет `xmin` (транзакция, создавшая версию) и `xmax` (удалившая/обновившая). Транзакция видит ту версию, что была закоммичена до её снимка. Поэтому **читающие не блокируют пишущих** и наоборот.

- **Нет Read Uncommitted:** грязное чтение в принципе невозможно — видны только закоммиченные версии; запрос уровня Read Uncommitted трактуется как Read Committed.
- **Repeatable Read без фантомов:** уровень берёт **снимок на момент первого запроса** транзакции и далее видит только его. Поэтому новые/удалённые другими транзакциями строки не появляются и не исчезают — фантомов нет (в стандарте SQL RR от фантомов не защищает, у PostgreSQL — строже).

Важно: RR не защищает от **аномалий сериализации** (write skew) — для этого есть уровень Serializable (SSI). См. [PostgreSQL — Transaction Isolation](https://www.postgresql.org/docs/current/transaction-iso.html).

### Вопрос 3. Что такое HOT-update (Heap-Only Tuple) в PostgreSQL и при каких условиях он применяется? Как HOT влияет на накопление мёртвых версий строк и размер индексов?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 66.7. Heap-Only Tuples (HOT)](https://www.postgresql.org/docs/current/storage-hot.html)

В MVCC при каждом UPDATE PostgreSQL создаёт новую версию строки. В обычном случае это требует добавления новых записей во все индексы таблицы, что вызывает значительные накладные расходы на запись и рост индексов.

HOT (Heap-Only Tuple) — оптимизация, которая позволяет обойтись без создания новых записей в индексах. Она применяется, если выполнены **оба** условия:

1. UPDATE не изменяет ни одну колонку, которая входит в индекс (summarizing-индексы, например BRIN, из этого правила исключены).
2. На той же странице кучи (heap page), где находится старая версия строки, достаточно свободного места для размещения новой версии.

Когда оба условия выполнены, PostgreSQL:
- не создаёт новых записей в индексах;
- связывает старую и новую версии строки цепочкой внутри одной страницы (HOT-chain), используя флаг `HEAP_HOT_UPDATED` и редирект `ItemId`;
- позволяет промежуточные (мёртвые) версии цепочки удалять при обычных операциях (включая SELECT) в рамках механизма *page pruning* — не дожидаясь VACUUM.

Если хотя бы одно из условий не выполнено, HOT-цепочка обрывается: последний элемент получает обычную ссылку `t_ctid` на следующую версию, и индексы обновляются в штатном режиме.

**Влияние на накопление мёртвых версий.** При HOT промежуточные версии строки убираются «на лету» через page pruning во время последующих операций чтения/записи на той же странице. Это существенно снижает количество мёртвых кортежей по сравнению с обычным UPDATE и уменьшает частоту, с которой требуется VACUUM.

**Влияние на размер индексов.** При HOT новые записи в индексы не добавляются, поэтому индексы не растут. При обычном UPDATE каждая новая версия строки порождает дополнительную запись в каждом индексе, что приводит к их постепенному раздуванию (index bloat).

Чтобы увеличить долю HOT-обновлений, рекомендуется снизить `fillfactor` таблицы (например, до 70–80%): тогда на каждой странице остаётся больше свободного места для новых версий строк.

```sql
CREATE TABLE orders (...) WITH (fillfactor = 70);
```

Статистику HOT-обновлений можно посмотреть в `pg_stat_user_tables`:

```sql
SELECT relname, n_tup_upd, n_tup_hot_upd
FROM pg_stat_user_tables;
```

#### Ответ от Claude

**Что такое HOT-update.** В PostgreSQL каждый UPDATE в рамках MVCC создаёт новую физическую версию строки (tuple). В обычном сценарии это влечёт за собой обновление каждого индекса таблицы: в каждый индекс добавляется запись, указывающая на новый `ctid` (физический адрес строки). HOT — оптимизация, которая позволяет выполнить UPDATE без изменения индексов вообще.

**Условия применения HOT (оба должны быть выполнены одновременно):**

1. Ни одна изменяемая колонка не входит ни в один индекс таблицы (обычный B-tree, Hash, GiST и т.д.). Summarizing-индексы (BRIN) составляют исключение и могут обновляться даже при HOT.
2. На той же heap-странице, где хранится старая версия строки, достаточно свободного места для размещения новой версии.

**Как работает механизм HOT.**

- Новая версия строки записывается на ту же страницу кучи. Старая версия помечается флагом `HEAP_HOT_UPDATED`, её `ItemId` превращается в redirect на новую версию.
- Все индексы по-прежнему указывают на `ItemId` старой версии. При поиске по индексу PostgreSQL находит этот `ItemId`, видит redirect и переходит к актуальной версии — без каких-либо изменений в структуре индекса.
- Образуется HOT-chain: цепочка версий внутри одной страницы, связанных редиректами.

**Page pruning — «легковесная очистка».** При обращении к странице (в том числе во время обычного SELECT или INSERT) PostgreSQL выполняет page pruning: проходит по HOT-цепочке, находит мёртвые промежуточные версии (уже невидимые ни одной активной транзакции) и помечает их как свободные. Это происходит без VACUUM, непосредственно во время обычных операций.

**Влияние на мёртвые версии строк.** При HOT мёртвые промежуточные версии убираются через page pruning значительно быстрее, чем при обычном UPDATE, который требует ожидания VACUUM. В результате объём мёртвых кортежей в таблице сокращается, таблица меньше «пухнет», а нагрузка на autovacuum снижается.

**Влияние на размер индексов.** При HOT новые записи в индексы не создаются совсем — индексы остаются стабильного размера. При обычном UPDATE каждое изменение строки порождает новую запись в каждом индексе, что при интенсивных обновлениях ведёт к значительному index bloat. HOT полностью устраняет эту проблему для строк, удовлетворяющих условиям оптимизации.

**Практический совет.** Главный рычаг управления HOT — параметр `fillfactor` (по умолчанию 100 для heap-таблиц). Снижение до 70–80% резервирует место на странице для новых версий строк и существенно увеличивает долю HOT-обновлений. Мониторинг: `n_tup_hot_upd / n_tup_upd` в `pg_stat_user_tables` — чем ближе к 1, тем эффективнее работает HOT.

### Вопрос 4. Что такое visibility map (карта видимости) в PostgreSQL? Как она связана с механизмом index-only scan и почему для работы index-only scan важно, чтобы autovacuum регулярно обрабатывал таблицу?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 66.4. Visibility Map](https://www.postgresql.org/docs/current/storage-vm.html)

Visibility map (карта видимости, VM) — вспомогательная структура данных, хранящаяся в отдельном файле рядом с файлом основных данных таблицы (с суффиксом `_vm` к filenode). Индексы собственной visibility map не имеют.

VM хранит **2 бита на каждую heap-страницу** таблицы:

- **Бит all-visible**: установлен, если все строки на странице видимы всем активным и будущим транзакциям (то есть на странице нет строк, требующих очистки). Используется для ускорения index-only scan.
- **Бит all-frozen**: установлен, если все строки на странице заморожены (frozen). Позволяет VACUUM пропустить страницу даже при anti-wraparound обходе.

**Биты устанавливает только VACUUM** (в том числе autovacuum). Любая модифицирующая операция (INSERT, UPDATE, DELETE) на странице немедленно сбрасывает соответствующий бит.

**Связь с index-only scan.** Index-only scan позволяет вернуть результат запроса, обратившись только к индексу, без чтения heap-страниц. Проблема состоит в том, что индекс не хранит информацию о видимости строк (MVCC-метаданные). Поэтому, прежде чем вернуть строку из индекса, PostgreSQL должен убедиться, что она видима для текущей транзакции. Именно здесь используется VM:

1. По найденной в индексе записи определяется heap-страница.
2. Если бит all-visible для этой страницы установлен — строка гарантированно видима, heap не читается.
3. Если бит не установлен — PostgreSQL вынужден обратиться к heap для проверки видимости, и преимущество index-only scan теряется.

VM значительно меньше самого heap (на четыре порядка), поэтому в большинстве случаев она целиком помещается в shared buffers и читается из памяти.

**Роль autovacuum.** Биты all-visible устанавливаются исключительно операцией VACUUM. Если autovacuum не запускается регулярно, биты остаются сброшенными, index-only scan превращается в обычный index scan с обращением к heap, и его преимущества полностью теряются.

#### Ответ от Claude

**Что такое visibility map.**

Visibility map (VM) — вспомогательный файл, который PostgreSQL ведёт для каждой heap-таблицы. Хранится рядом с файлом данных, имя файла — `<filenode>_vm`. Для каждой heap-страницы VM хранит 2 бита.

| Бит | Название | Когда установлен | Что означает |
|-----|----------|-----------------|--------------|
| 1 | all-visible | После VACUUM | Все строки страницы видимы любой транзакции; на странице нет мёртвых кортежей |
| 2 | all-frozen | После VACUUM FREEZE / агрессивного VACUUM | Все строки заморожены; страницу можно пропустить при anti-wraparound VACUUM |

**Кто управляет битами.**
- **Устанавливает:** только операция VACUUM (включая autovacuum).
- **Сбрасывает:** любая DML-операция (INSERT, UPDATE, DELETE, HOT-update) на соответствующей странице — немедленно при изменении.

**Связь с index-only scan.**

Index-only scan — режим выполнения запроса, при котором все нужные данные берутся из индекса, без обращения к heap. Это возможно, если все запрошенные колонки входят в индекс (или являются INCLUDE-колонками).

Однако у индексов PostgreSQL нет информации о видимости строк (MVCC-снимки хранятся только в heap). Это порождает дилемму: вернуть строку из индекса без проверки видимости нельзя, а проверить видимость без heap — тоже нельзя.

VM разрешает эту дилемму:

```
Индекс -> найдена запись -> проверяем VM-бит all-visible для heap-страницы
    ├─ бит установлен  -> строка точно видима -> возвращаем из индекса (heap НЕ читаем)
    └─ бит не установлен -> читаем heap-страницу для проверки видимости
                           (выгода index-only scan теряется, деградирует до index scan)
```

Поскольку VM на четыре порядка меньше heap, она практически всегда остаётся в shared buffers, и её чтение обходится в разы дешевле, чем случайный I/O по heap.

**Почему autovacuum критичен для index-only scan.**

После любого UPDATE или DELETE бит all-visible сбрасывается для затронутых страниц. Пока VACUUM не пройдёт по таблице и не восстановит биты, index-only scan не может работать в «чистом» режиме и вынужден обращаться к heap. Чем реже autovacuum обрабатывает таблицу, тем меньше страниц имеют установленный бит all-visible, и тем чаще index-only scan деградирует до обычного index scan.

Практический вывод: для таблиц, по которым критична производительность index-only scan, следует убедиться, что autovacuum настроен агрессивно (или запускается регулярно вручную), иначе ожидаемый прирост производительности не будет достигнут.

**Диагностика.** Модуль `pg_visibility` позволяет проверить состояние VM:

```sql
-- Сколько страниц имеют установленный бит all-visible
SELECT count(*) FILTER (WHERE all_visible) AS visible_pages,
       count(*) AS total_pages
FROM pg_visibility('my_table');
```

Просмотр планов запросов: если в плане есть `Index Only Scan` с большим числом `Heap Fetches` — это признак того, что VM устарела и autovacuum не успевает.

### Вопрос 5. Объясни механизм transaction ID wraparound в PostgreSQL: почему XID 32-битный, что произойдёт при переполнении, как работает заморозка (freeze) строк и что делает VACUUM FREEZE? Как диагностировать приближение к wraparound через pg_database.datfrozenxid?

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 24.1. Routine Vacuuming (раздел «Preventing Transaction ID Wraparound Failures»)](https://www.postgresql.org/docs/current/routine-vacuuming.html)

**Почему XID 32-битный и в чём опасность переполнения.**

PostgreSQL использует 32-битные идентификаторы транзакций (XID). Это даёт 2^32 ≈ 4 миллиарда уникальных значений. Видимость строк в MVCC определяется сравнением XID: строка, созданная транзакцией с XID X, видима транзакции с XID Y, если X «в прошлом» по отношению к Y. PostgreSQL интерпретирует это сравнение по модулю 2^32: для любого XID ровно 2 миллиарда значений считаются «прошлым» и 2 миллиарда — «будущим».

Если счётчик XID переполнится и обнулится, строки, созданные «давно», окажутся в «будущем» с точки зрения новых транзакций — они станут невидимыми. Результат: катастрофическая потеря данных.

**Механизм заморозки строк.**

Чтобы этого не произошло, PostgreSQL резервирует специальный XID — `FrozenTransactionId` (значение 2). Этот XID не следует обычным правилам сравнения и считается «старше» любого обычного XID. Строка, чей `xmin` заменён на `FrozenTransactionId` (или у которой выставлен специальный флаг `HEAP_XMIN_FROZEN` начиная с PostgreSQL 9.4), будет видима любой транзакции вне зависимости от wraparound.

VACUUM замораживает (`freezes`) строки, чей `xmin` старше порогового значения `vacuum_freeze_min_age` (по умолчанию 50 000 000 транзакций). При обычном VACUUM замораживаются только строки на страницах, у которых не установлен бит all-visible в visibility map. `VACUUM FREEZE` (или агрессивный VACUUM) сканирует все страницы таблицы и замораживает все подходящие строки принудительно.

**Параметры управления заморозкой:**

- `vacuum_freeze_min_age` (по умолчанию 50 млн): возраст XID (в транзакциях), начиная с которого строка может быть заморожена.
- `vacuum_freeze_table_age` (по умолчанию 150 млн): если `relfrozenxid` таблицы старше этого значения, VACUUM принудительно сканирует всю таблицу (агрессивный режим).
- `autovacuum_freeze_max_age` (по умолчанию 200 млн): максимально допустимый возраст `relfrozenxid`; при достижении autovacuum запускается принудительно даже на неактивной таблице.

**Диагностика через pg_database.datfrozenxid.**

```sql
-- Возраст наиболее «старого» незамороженного XID по каждой базе данных
SELECT datname, age(datfrozenxid) AS xid_age
FROM pg_database
ORDER BY xid_age DESC;

-- Таблицы, ближайшие к wraparound
SELECT c.oid::regclass AS table_name,
       greatest(age(c.relfrozenxid), age(t.relfrozenxid)) AS age
FROM pg_class c
LEFT JOIN pg_class t ON c.reltoastrelid = t.oid
WHERE c.relkind IN ('r', 'm')
ORDER BY age DESC
LIMIT 20;
```

`age(datfrozenxid)` возвращает количество транзакций от `datfrozenxid` до текущего XID. Если это значение приближается к 2 миллиардам — ситуация критическая.

**Предупреждения PostgreSQL.** При приближении к пределу PostgreSQL сам предупреждает:
- за 40 млн транзакций до переполнения — WARNING в логах;
- за 3 млн транзакций — СУБД переходит в режим only-in-transactions и отказывает в новых транзакциях с назначением XID.

#### Ответ от Claude

**Почему XID 32-битный.**

Исторически PostgreSQL выбрал 32-битный счётчик транзакций (XID) как компромисс между размером заголовков строк и практическими нуждами. 4 байта на XID — это небольшой overhead на каждый кортеж. При этом 2^32 ≈ 4,3 миллиарда значений казалось достаточным. Современные высоконагруженные системы показывают, что это ограничение вполне достижимо за годы эксплуатации. В PostgreSQL 64-битный XID (логический, используемый в WAL и репликации — LSN) и 32-битный XID (физический, хранимый в кортежах) — разные вещи; именно 32-битный физический XID создаёт проблему wraparound.

**Что произойдёт при переполнении.**

PostgreSQL определяет «старше/новее» не линейно, а по модулю 2^32: из 4 миллиардов значений 2 миллиарда считаются «прошлым», 2 миллиарда — «будущим» для любого данного XID. После wraparound (обнуления счётчика) строки, созданные «до» обнуления, окажутся в «будущем» с точки зрения новых транзакций — MVCC посчитает их невидимыми. Фактически это означает исчезновение данных без какого-либо физического удаления. Именно поэтому проблему называют катастрофической.

**Механизм заморозки строк.**

Решение — «изъять» старые XID из обычного круга сравнения. PostgreSQL резервирует специальный XID = 2 (`FrozenTransactionId`), который по определению «старше» любого обычного XID. Строка считается замороженной, если:
- (до PostgreSQL 9.4) её `xmin` физически заменён на `FrozenTransactionId = 2`;
- (PostgreSQL 9.4+) у строки выставлен флаг `HEAP_XMIN_FROZEN` в `infomask`; оригинальный `xmin` при этом сохраняется для forensic-целей.

Замороженная строка видима всем транзакциям — нынешним и будущим, независимо от wraparound.

**Что делает VACUUM FREEZE.**

Обычный VACUUM замораживает строки только на «грязных» страницах (с не установленным битом all-visible в visibility map), пропуская чистые. Это быстро, но не гарантирует заморозку всех старых строк.

`VACUUM FREEZE` (или агрессивный VACUUM, запускаемый autovacuum при достижении `autovacuum_freeze_max_age`) сканирует все страницы таблицы подряд, замораживает все строки с XID старше `vacuum_freeze_min_age` и обновляет `relfrozenxid` в `pg_class` до актуального значения. После этого также обновляется `datfrozenxid` в `pg_database`.

```sql
-- Явный запуск агрессивной заморозки
VACUUM FREEZE my_table;
```

**Диагностика приближения к wraparound.**

```sql
-- Возраст datfrozenxid по всем базам (ближайшие к опасной зоне — вверху)
SELECT datname,
       age(datfrozenxid)                          AS xid_age,
       2000000000 - age(datfrozenxid)             AS xids_remaining
FROM pg_database
ORDER BY xid_age DESC;

-- Таблицы с наибольшим возрастом relfrozenxid (включая TOAST)
SELECT c.oid::regclass                                        AS table_name,
       greatest(age(c.relfrozenxid), age(t.relfrozenxid))    AS age
FROM pg_class c
LEFT JOIN pg_class t ON c.reltoastrelid = t.oid
WHERE c.relkind IN ('r', 'm')
ORDER BY age DESC
LIMIT 20;
```

**Интерпретация значений.** Функция `age(xid)` возвращает число транзакций от переданного XID до текущего `txid_current()`. Критические пороги:

| age() | Что происходит |
|-------|----------------|
| < 200 млн | Норма (autovacuum_freeze_max_age) |
| 200 млн | autovacuum запускается принудительно |
| ~1 960 млн | WARNING в логах PostgreSQL |
| ~2 147 млн | СУБД запрещает транзакции, требующие XID (только read-only) |

**Расхождение между источниками.** Некоторые источники указывают пороговое значение «за 40 млн до предела» для WARNING. Официальная документация PostgreSQL 18 говорит о пороге в `autovacuum_freeze_max_age * 1.5` (т.е. 300 млн при дефолтных настройках) для принудительного агрессивного autovacuum, и отдельных порогах для предупреждений (40 млн) и полной блокировки (3 млн) перед `2^31`. Следует ориентироваться на актуальную документацию своей версии PostgreSQL.

**Лучшие практики:**
- Не отключать autovacuum.
- Мониторить `age(datfrozenxid)` через Prometheus/Zabbix/pg_stat_activity.
- При значениях `xid_age > 1,5 млрд` — немедленно запускать `VACUUM FREEZE` вручную.
- Избегать долгих незавершённых транзакций: они блокируют продвижение `datfrozenxid`.

### Вопрос 6. Как работает TOAST (The Oversized-Attribute Storage Technique) в PostgreSQL? Какой порог размера значения активирует TOAST, какие стратегии хранения существуют (PLAIN, EXTENDED, EXTERNAL, MAIN) и как TOAST влияет на производительность запросов?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 66.2. TOAST](https://www.postgresql.org/docs/current/storage-toast.html)

PostgreSQL использует фиксированный размер страницы (как правило, 8 КБ) и не позволяет кортежам занимать несколько страниц. Поэтому невозможно хранить очень большие значения полей напрямую. Для преодоления этого ограничения применяется техника TOAST, которая прозрачно сжимает и/или разбивает крупные значения на несколько физических строк.

TOAST-обработка активируется, когда строка превышает примерно 2 КБ (константа `TOAST_TUPLE_THRESHOLD`). Целевой размер после сжатия/выноса — также около 2 КБ (`TOAST_TUPLE_TARGET`). Максимальный размер одного логического значения — 1 ГБ (2^30 - 1 байт).

Для TOASTable-столбцов создаётся отдельная TOAST-таблица (её OID хранится в `pg_class.reltoastrelid`). Данные в ней хранятся в виде чанков с тремя столбцами:
- `chunk_id` (OID) — идентификатор TOASTed-значения;
- `chunk_seq` — порядковый номер чанка;
- `chunk_data` — собственно данные.

По паре `(chunk_id, chunk_seq)` создаётся уникальный индекс для быстрого доступа. Максимальный размер одного чанка — около 2000 байт (`TOAST_MAX_CHUNK_SIZE`). В основной таблице вместо значения хранится 18-байтный указатель (TOAST pointer datum).

Существуют четыре стратегии хранения:

- **PLAIN** — без сжатия и без выноса во внешнюю таблицу; единственная стратегия для не-TOAST-типов.
- **EXTENDED** — сначала пытается сжать, затем выносит во внешнюю таблицу; стратегия по умолчанию для большинства TOAST-совместимых типов.
- **EXTERNAL** — только вынос во внешнюю таблицу, без сжатия; ускоряет операции над подстроками (`text`, `bytea`), поскольку PostgreSQL может извлечь лишь нужную часть, но увеличивает занимаемое место.
- **MAIN** — только сжатие (вынос применяется лишь как крайняя мера, когда строка не помещается на страницу).

Стратегию можно задать командой:
```sql
ALTER TABLE t ALTER COLUMN c SET STORAGE EXTERNAL;
ALTER TABLE t SET (toast_tuple_target = 4096);
```

Влияние TOAST на производительность:
- Основная таблица становится меньше, больше строк помещается в `shared_buffers`.
- Сортировки выполняются быстрее и чаще умещаются в памяти.
- Крупные TOASTed-атрибуты извлекаются только при необходимости.
- При `UPDATE`, если крупное поле не изменилось, TOAST-данные не перезаписываются.

#### Ответ от Claude

TOAST решает фундаментальное ограничение PostgreSQL: страница данных фиксирована (обычно 8 КБ), а кортеж не может пересекать границы страниц. Когда строка превышает примерно **2 КБ** (порог `TOAST_TUPLE_THRESHOLD`), TOAST прозрачно обрабатывает «тяжёлые» столбцы.

**Механизм работы.** Для каждой таблицы с TOAST-совместимыми столбцами PostgreSQL создаёт сопутствующую TOAST-таблицу (системный каталог `pg_class.reltoastrelid`). Крупное значение нарезается на чанки по ~2000 байт и записывается туда. В основной строке остаётся компактный **18-байтный указатель**, содержащий OID TOAST-таблицы, идентификатор значения (`chunk_id`), исходный и физический размеры, а также метод сжатия.

**Четыре стратегии хранения** задаются на уровне столбца:

| Стратегия | Сжатие | Вынос в TOAST-таблицу | Примечание |
|-----------|:------:|:---------------------:|------------|
| PLAIN     | нет    | нет                   | Только для типов без TOAST-поддержки |
| EXTENDED  | да     | да (если нужно)        | По умолчанию для `text`, `jsonb`, `bytea` и др. |
| EXTERNAL  | нет    | да                    | Быстрее substring-операции; больше места |
| MAIN      | да     | только в крайнем случае | Приоритет — компактность строки |

По умолчанию метод сжатия определяется параметром `default_toast_compression` (варианты: `pglz`, `lz4`; по умолчанию `pglz`).

**Влияние на производительность:**
- Положительное: меньшие основные страницы повышают эффективность кэша (`shared_buffers`), сортировки быстрее умещаются в `work_mem`, крупные поля не читаются при `SELECT col_small FROM ...`.
- Отрицательное: чтение TOAST-данных требует дополнительных обращений к отдельной таблице; при частом чтении больших значений это снижает производительность. Кроме того, небольшой накладной расход возникает при сжатии/декомпрессии.
- При `UPDATE`: если TOAST-столбец не изменился, его данные в TOAST-таблице сохраняются без изменений — это экономит I/O.

**Расхождений между источниками не выявлено.** Все детали (пороговые значения, стратегии, структура TOAST-таблицы) подтверждены официальной документацией PostgreSQL 18.

### Вопрос 7. Опиши устройство WAL (Write-Ahead Log) в PostgreSQL: зачем он нужен, что записывается в WAL при UPDATE строки, и как checkpoint связывает WAL с данными на диске? Какие параметры управляют частотой checkpoint и что произойдёт при crash-recovery?

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 28.5. WAL Configuration](https://www.postgresql.org/docs/current/wal-configuration.html)

Центральная идея WAL: изменения файлов данных (таблиц и индексов) должны быть записаны только после того, как соответствующие WAL-записи сброшены на постоянное хранилище. Это позволяет при сбое восстановить данные, повторив WAL-записи (REDO), не полагаясь на мгновенный сброс каждой страницы данных.

Checkpoint — точка в последовательности транзакций, в которой гарантируется, что все «грязные» страницы данных сброшены на диск. После завершения checkpoint в WAL записывается специальная checkpoint-запись, а позиция checkpoint сохраняется в файле `pg_control`. При crash-recovery сервер читает `pg_control`, находит последний checkpoint и начинает REDO-восстановление с позиции WAL, указанной в checkpoint-записи (redo record). Все изменения WAL после этой позиции повторно применяются к страницам данных.

Параметры управления частотой checkpoint:
- `checkpoint_timeout` (по умолчанию 5 минут) — checkpoint запускается каждые N секунд.
- `max_wal_size` (по умолчанию 1 ГБ) — checkpoint запускается, если накопленный WAL вот-вот превысит этот предел.
- `checkpoint_completion_target` (по умолчанию 0.9) — доля интервала между checkpoint'ами, в течение которой PostgreSQL постепенно сбрасывает «грязные» буферы, сглаживая нагрузку на I/O.

Параметр `full_page_writes` (по умолчанию включён): первая модификация страницы данных после каждого checkpoint записывает в WAL содержимое всей страницы целиком — это необходимо для корректного восстановления после частичной записи страницы на диск в момент сбоя.

Уменьшение `checkpoint_timeout` или `max_wal_size` ускоряет crash-recovery (меньше WAL нужно «проигрывать»), но увеличивает объём full-page-writes в WAL и нагрузку на I/O.

#### Ответ от Claude

**Зачем нужен WAL.** PostgreSQL использует WAL как стандартный механизм обеспечения долговечности данных. Гарантия звучит так: изменения в файлах данных могут быть записаны только после того, как описывающие их WAL-записи сброшены на постоянное хранилище. Это даёт три ключевых преимущества:

1. **Crash-recovery** — при сбое БД перечитывает WAL и повторяет (REDO) все изменения, не дошедшие до файлов данных.
2. **Производительность** — вместо сброса многих разрозненных страниц данных достаточно одного последовательного `fsync` WAL-файла для подтверждения транзакции; последовательная запись намного быстрее случайной.
3. **Репликация и PITR** — WAL-поток используется для потоковой репликации и восстановления на произвольный момент времени.

**Что записывается в WAL при UPDATE строки.** WAL-записи создаёт менеджер ресурсов (resource manager) конкретного подсистемного уровня — для heap-таблиц это `Heap`/`Heap2`. Запись при UPDATE содержит:
- заголовок (`XLogRecord`): LSN предыдущей записи, идентификатор транзакции, длину, контрольную сумму, тип ресурс-менеджера;
- идентификатор страницы: tablespace / database / relation OID, номер блока;
- старый и новый образы кортежа (или только изменённые байты — FPW/delta);
- при первой модификации страницы после checkpoint (если `full_page_writes = on`) — полное содержимое страницы данных (full page image), чтобы защититься от «разорванных» (torn) записей при сбое в середине записи страницы.

Каждой записи присваивается **LSN** (Log Sequence Number) — монотонно растущее смещение в байтах в потоке WAL. LSN используется для сравнения позиций в WAL, отслеживания прогресса репликации и управления checkpoint.

**Как checkpoint связывает WAL с данными на диске.**

1. Postgres начинает checkpoint: фиксирует текущий LSN как `redo LSN`.
2. Постепенно (в течение `checkpoint_completion_target` от интервала) сбрасывает все «грязные» страницы из `shared_buffers` на диск.
3. Записывает checkpoint-запись в WAL (содержит `redo LSN`).
4. Сохраняет позицию checkpoint в файле `pg_control`.
5. WAL-сегменты, предшествующие `redo LSN`, больше не нужны для recovery и могут быть переиспользованы или удалены.

**Параметры управления частотой checkpoint:**

| Параметр | По умолчанию | Роль |
|----------|-------------|------|
| `checkpoint_timeout` | 5 мин | Максимальное время между checkpoint'ами |
| `max_wal_size` | 1 ГБ | Мягкий лимит накопленного WAL (при превышении — внеплановый checkpoint) |
| `checkpoint_completion_target` | 0.9 | Доля интервала для размазывания I/O сброса страниц |
| `full_page_writes` | on | Записывать полные страницы после checkpoint (защита от torn pages) |
| `checkpoint_warning` | 30 с | Предупреждение в лог, если checkpoint'ы идут слишком часто |

**Crash-recovery.** После рестарта PostgreSQL читает `pg_control`, находит последний валидный checkpoint и его `redo LSN`. Затем последовательно проигрывает все WAL-записи, начиная с `redo LSN`: применяет изменения к страницам данных (REDO), восстанавливает индексы и системные каталоги. После этого выполняется откат незавершённых транзакций (UNDO через таблицу `pg_xact`). Время recovery прямо пропорционально объёму WAL, накопленного с последнего checkpoint, — именно поэтому снижение `checkpoint_timeout`/`max_wal_size` ускоряет восстановление, хотя и увеличивает write-amplification в штатном режиме.

**Расхождений между источниками не выявлено.** Детали `full_page_writes`, структуры checkpoint и параметров подтверждены официальной документацией PostgreSQL (разделы 28.5 и 28.6).

### Вопрос 8. Что такое synchronous_commit в PostgreSQL и какие значения он принимает (off, local, remote_write, remote_apply, on)? Как выбор значения влияет на задержку транзакции и риск потери данных при падении primary?

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 19.5. Write Ahead Log](https://www.postgresql.org/docs/current/runtime-config-wal.html)

Параметр `synchronous_commit` определяет, сколько WAL-обработки должно быть завершено перед тем, как сервер вернёт клиенту подтверждение успеха транзакции. Значение можно задавать на уровне сессии или отдельной транзакции (`SET LOCAL synchronous_commit TO off`).

Возможные значения (от наименее до наиболее строгого):

- **off** — сервер не ждёт сброса WAL на диск; транзакция считается завершённой немедленно. Риск потери данных: до 3 × `wal_writer_delay` (по умолчанию 200 мс × 3 = 600 мс). Нет риска нарушения целостности БД — теряются только последние N транзакций.
- **local** — ждёт сброса WAL на локальный диск, не ожидает подтверждения от standby. При наличии синхронной репликации не гарантирует сохранность данных на реплике.
- **remote_write** — ждёт, пока standby получит WAL-данные и запишет их в файловый буфер ОС (не обязательно сбросит на диск). Защищает от сбоя процесса PostgreSQL на standby, но не от краша ОС на standby.
- **on** (по умолчанию) — ждёт, пока standby запишет WAL на долговременное хранилище (fsync). Транзакция безопасна, если не происходит одновременного сбоя хранилища primary и всех синхронных standby.
- **remote_apply** — ждёт, пока standby применит (проиграл) WAL-записи и данные стали видны в репликах. Обеспечивает согласованность чтений со standby сразу после COMMIT.

Если `synchronous_standby_names` не задан, значения `local`, `remote_write` и `remote_apply` эквивалентны `on` (только локальная синхронизация имеет смысл).

Фоновый процесс «WAL writer» сбрасывает незаписанные WAL-записи на диск каждые `wal_writer_delay` миллисекунд (по умолчанию 200 мс).

Сводная таблица гарантий из документации:

| Значение | Локальный диск | Standby fsync | Standby OS | Standby читает данные |
|---|:---:|:---:|:---:|:---:|
| remote_apply | да | да | да | да |
| on | да | да | да | нет |
| remote_write | да | да | нет | нет |
| local | да | нет | нет | нет |
| off | нет | нет | нет | нет |

#### Ответ от Claude

`synchronous_commit` — это параметр PostgreSQL, управляющий точкой синхронизации WAL, после которой сервер отвечает клиенту «транзакция зафиксирована». Он напрямую определяет компромисс между задержкой (`latency`) транзакции и гарантиями долговечности данных.

**Пять значений по возрастанию строгости:**

**1. `off`**
Сервер не ждёт ничего. WAL-буферы записываются на диск фоновым WAL writer'ом с периодом `wal_writer_delay` (умолч. 200 мс). Окно риска потери данных — до **3 × wal_writer_delay** (~600 мс при дефолте). Важно: это потеря данных, а не повреждение — база остаётся консистентной, просто несколько последних транзакций могут исчезнуть после краша primary. Задержка транзакции — минимальная.

**2. `local`**
Ждёт сброса WAL на локальный диск primary. Защищает от краша самого primary. При отсутствии репликации равносилен `on`. При потоковой репликации: данные безопасны на primary, но standby мог не получить их — переключение на standby возможно с потерей недавних транзакций.

**3. `remote_write`**
Дополнительно ждёт, пока standby передаст данные WAL в буферный кэш ОС (write без fsync). Защищает от сбоя процесса PostgreSQL на standby, но **не защищает** от краша ОС standby (данные в page cache могут не успеть уйти на диск). Задержка выше, чем у `local`, из-за сетевого round-trip.

**4. `on`** (значение по умолчанию)
Ждёт, пока standby сбросит WAL на долговременное хранилище (fsync). Транзакция гарантированно сохранена и на primary, и на standby. Потеря данных возможна только при одновременном катастрофическом отказе хранилищ primary и всех синхронных standby.

**5. `remote_apply`**
Строже всего: ждёт, пока standby проиграет (применит) полученные WAL-записи, то есть данные станут видны на standby в запросах SELECT. Это позволяет строить схемы «запиши на primary, читай со standby» без окна рассинхронизации. Задержка — наибольшая, так как включает время воспроизведения WAL на реплике.

**Влияние на задержку транзакций.** Значения `off` и `local` не требуют сетевого взаимодействия, поэтому задержка определяется только локальным I/O (или вовсе не включает I/O при `off`). Значения `remote_write`, `on`, `remote_apply` добавляют минимум один сетевой round-trip к standby, поэтому задержка растёт с удалённостью standby и качеством канала.

**Риск потери данных при падении primary:**

- `off` — высокий риск: последние ~600 мс транзакций при дефолтном `wal_writer_delay`.
- `local` — нет риска потери на primary, но при failover на standby данные могут быть неполными.
- `remote_write` — минимальный риск при краше только primary; риск есть при одновременном краше primary и standby (OS crash на standby).
- `on` — данные сохранены на дисках обоих узлов, риск потери практически нулевой.
- `remote_apply` — то же, что `on`, плюс гарантия видимости на standby.

**Практические рекомендации.** `off` уместен для bulk-загрузки или временных данных, где скорость важнее надёжности. `local` подходит для standalone-серверов без репликации. `on` — стандарт для production с HA. `remote_apply` — когда требуется строгая консистентность чтений со standby.

Параметр изменяется без перезагрузки сервера, может быть переопределён на уровне сессии или транзакции:
```sql
SET LOCAL synchronous_commit = 'remote_apply';
```

**Расхождений между источниками не выявлено.** Таблица гарантий и описание каждого уровня подтверждены официальной документацией PostgreSQL 18 (раздел 19.5).
### Вопрос 9. Чем отличаются типы индексов B-tree, Hash, GIN, GiST, SP-GiST и BRIN в PostgreSQL? Для каких типов запросов и данных оптимален каждый тип?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — Chapter 11. Indexes: Index Types](https://www.postgresql.org/docs/current/indexes-types.html)

**B-tree** — тип по умолчанию. Поддерживает операторы `<`, `<=`, `=`, `>=`, `>`, а также `BETWEEN`, `IN`, `IS NULL`, `IS NOT NULL` и поиск по префиксу (`LIKE 'foo%'`). Подходит для большинства задач: равенство, диапазоны, сортировка.

**Hash** — хранит только 32-битный хэш значения. Поддерживает исключительно оператор `=`. Используется только при необходимости проверки равенства; не даёт никаких преимуществ перед B-tree при диапазонных запросах или сортировке.

**GIN (Generalized Inverted Index)** — предназначен для составных структур данных (массивы, JSONB, полнотекстовый поиск). Поддерживает операторы `<@`, `@>`, `=`, `&&` для массивов. Эффективно обрабатывает запросы проверки наличия конкретного компонента внутри составного значения.

**GiST (Generalized Search Tree)** — обобщённое дерево поиска. Поддерживает геометрические типы и операции (перекрытие, включение, соседство), а также ближайший сосед: `ORDER BY location <-> point '(101,456)'`. Применяется для пространственных данных, диапазонов, полнотекстового поиска (через `tsvector`).

**SP-GiST (Space-Partitioned GiST)** — пространственно разбитое дерево поиска, реализующее несбалансированные структуры: квадродеревья, k-d деревья, радиксные деревья. Поддерживает ближайшего соседа для операторных классов с функцией расстояния. Используется для 2D-точек и других типов, для которых реализован оператор SP-GiST.

**BRIN (Block Range INdex)** — хранит агрегаты (мин/макс) по диапазонам физических блоков таблицы, а не отдельные строки. Поддерживает `<`, `<=`, `=`, `>=`, `>`. Оптимален для очень больших таблиц, где значения коррелируют с физическим порядком хранения (временные ряды, последовательно загружаемые данные). Занимает на порядки меньше места, чем B-tree, при высокой корреляции.

#### Ответ от Claude

Сводная характеристика каждого типа по данным официальной документации и vladmihalcea.com:

**B-tree**
- Алгоритм: сбалансированное дерево (фактически B+-дерево — данные хранятся в листовых узлах).
- Операторы: `<`, `<=`, `=`, `>=`, `>`, `BETWEEN`, `IN`, `IS NULL`, `IS NOT NULL`, `LIKE 'prefix%'`, `~'^prefix'`.
- Дополнительно: возвращает данные в отсортированном порядке без отдельной сортировки; поддерживает составные ключи.
- Ограничение: неэффективен для случайных значений с высокой кардинальностью (например, UUIDs) из-за низкого коэффициента заполнения страниц.
- Когда использовать: выбор по умолчанию для подавляющего большинства задач.

**Hash**
- Хранит 32-битный хэш значения; компактнее B-tree на широких строковых столбцах (пример: 272 КБ против 472 КБ для 50-символьных строк).
- Поддерживает только `=`; без диапазонов, сортировки, поиска по префиксу.
- Когда использовать: узкий сценарий — только проверка равенства на длинных строках, где выигрыш в размере индекса критичен.

**GIN (Generalized Inverted Index)**
- Инвертированный индекс: хранит отображение «компонент → список строк».
- Лучший выбор для: массивов (`@>`, `<@`, `&&`), JSONB (существование ключа, совпадение значения), полнотекстового поиска (`@@`), `pg_trgm` (неточный поиск по подстроке).
- Минус: медленные операции записи (индекс нужно обновлять по каждому компоненту), большой размер для документов с большим числом токенов.

**GiST (Generalized Search Tree)**
- Расширяемый фреймворк; конкретное дерево определяется оператором классом.
- Используется для: геометрии (PostGIS), диапазонных типов (`tsrange`, `int4range`), полнотекстового поиска, `cube`, `earthdistance`.
- Поддерживает nearest-neighbor (`<->`), что невозможно в GIN и BRIN.
- Когда использовать: пространственные, диапазонные данные и запросы по близости.

**SP-GiST (Space-Partitioned GiST)**
- Реализует несбалансированные структуры: квадродеревья, k-d деревья, радиксные деревья (префиксные деревья для строк).
- Эффективнее GiST для данных с естественным пространственным разбиением (плотные кластеры точек).
- Поддерживает nearest-neighbor для соответствующих операторных классов.
- Когда использовать: 2D-точки, IP-адреса (тип `inet`), строки с общими префиксами.

**BRIN**
- Хранит только агрегаты (мин, макс) для диапазонов блоков; по умолчанию 128 блоков на диапазон.
- Размер индекса не зависит от размера таблицы — остаётся крошечным.
- Требование: значения в столбце должны коррелировать с физическим порядком строк в таблице.
- Когда использовать: очень большие таблицы временных рядов (`created_at`, `event_time`), таблицы логов, IoT-данные — при условии последовательной вставки.
- Расплата: при низкой корреляции почти не фильтрует блоки, и PostgreSQL всё равно читает большую часть таблицы.

|---|---|---|---|---|---|---|---|

| Hash | + | - | - | - | - | - | малый |
| GIN | - | - | - | + | - | - | большой |
| GiST | + | + | - | - | + | + | средний |

### Вопрос 10. Что такое покрывающий индекс (covering index) с INCLUDE в PostgreSQL? Чем колонки в INCLUDE отличаются от колонок в ключевой части индекса, и когда index-only scan невозможен даже при наличии нужных колонок в индексе?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 11.9. Index-Only Scans and Covering Indexes](https://www.postgresql.org/docs/current/indexes-index-only-scans.html)

Покрывающий индекс — индекс, содержащий все столбцы, необходимые запросу, что позволяет выполнить index-only scan без обращения к куче (heap).

Синтаксис с INCLUDE:
```sql
CREATE INDEX tab_x_y ON tab(x) INCLUDE (y);
```

Здесь `x` — ключевой столбец (используется для навигации по дереву), `y` — «полезная нагрузка» (payload), хранится только в листовых узлах.

**Отличия колонок INCLUDE от ключевых:**
- INCLUDE-колонки не используются для поиска и навигации по индексу.
- Нет ограничений на тип данных (не требуется операторный класс).
- Не включаются в проверку уникальности: `CREATE UNIQUE INDEX ... ON tab(x) INCLUDE (y)` гарантирует уникальность только по `x`.
- Удаляются из верхних уровней B-tree через suffix truncation, что уменьшает глубину дерева.

**Когда index-only scan невозможен:**
PostgreSQL ведёт карту видимости (visibility map): бит для каждой страницы кучи означает «все строки на этой странице видимы всем транзакциям». При index-only scan после нахождения записи в индексе планировщик проверяет бит видимости соответствующей страницы. Если бит не установлен — необходимо обратиться к куче для проверки видимости строки. Таким образом, index-only scan не даёт выигрыша на недавно изменённых или часто обновляемых страницах, где карта видимости не актуализирована (VACUUM не успел пройти).

Практический вывод: покрывающий индекс с INCLUDE имеет смысл только для таблиц, изменяющихся достаточно редко — иначе большинство страниц не будут помечены в карте видимости и heap-доступ всё равно произойдёт.

#### Ответ от Claude

**Что такое покрывающий индекс**

Покрывающий (covering) индекс содержит все столбцы, нужные для выполнения конкретного запроса: и столбцы условия WHERE, и столбцы проекции SELECT. Это позволяет PostgreSQL выполнить index-only scan — вернуть результат напрямую из индекса, не читая страницы кучи.

До PostgreSQL 11 добиться этого можно было только включением всех нужных столбцов в ключевую часть индекса, что приводило к росту размера дерева и ухудшению производительности записи. Начиная с PostgreSQL 11, синтаксис INCLUDE добавляет столбцы только в листовые узлы B-tree.

**Ключевые vs. INCLUDE колонки**

| Аспект | Ключевые колонки | INCLUDE-колонки |
|---|---|---|
| Участие в навигации | Да (строят дерево) | Нет (только листовые узлы) |
| Требование оператор-класса | Обязательно | Не требуется |
| Влияние на порядок сортировки | Да | Нет |
| Участие в UNIQUE-проверке | Да | Нет |
| Suffix truncation на внутренних узлах | Не применяется | Удаляются |
| Использование в WHERE для поиска | Да | Нет |

Пример практического применения:
```sql
-- Запрос, который может стать index-only scan:
SELECT last_name FROM employees WHERE department_id = 5;

-- Индекс:
CREATE INDEX idx_emp_dept ON employees(department_id) INCLUDE (last_name);
```

**Когда index-only scan невозможен**

Три основных сценария (по данным официальной документации и use-the-index-luke.com):

1. **Страница не помечена в visibility map.** PostgreSQL поддерживает карту видимости: каждый бит означает, что все строки на странице видимы всем транзакциям. Если бит не установлен — планировщик обязан обратиться к куче. Свежеизменённые или недавно вставленные страницы не имеют установленного бита до прохода VACUUM.

2. **Таблица часто обновляется.** Если большинство страниц кучи не имеет установленного бита видимости, прирост от index-only scan будет минимальным — PostgreSQL всё равно читает heap. Для высокочастотных OLTP-таблиц INCLUDE может не дать ощутимого эффекта.

3. **Использование выражений.** Планировщик не может автоматически вывести, что `f(x)` доступно из индекса по `f(x)` при запросе `WHERE f(x) < 1 ... SELECT f(x)` — если только `x` явно не добавлен через INCLUDE:
   ```sql
   CREATE INDEX idx ON tab (f(x)) INCLUDE (x);
   ```

**Расхождение источников:** use-the-index-luke.com делает акцент на том, что index-only scan в PostgreSQL в принципе ненадёжен из-за архитектуры MVCC (версионность строк хранится только в куче) — в отличие от Oracle, где версии хранятся отдельно. Официальная документация PostgreSQL этого не оспаривает, но формулирует мягче: «scan will be a win only if a significant fraction of the table's heap pages have their all-visible map bits set».

### Вопрос 11. Что такое частичный индекс (partial index) и функциональный индекс в PostgreSQL? Приведи примеры сценариев, где частичный индекс даёт существенный выигрыш перед обычным B-tree.

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 11.8. Partial Indexes](https://www.postgresql.org/docs/current/indexes-partial.html)

**Частичный индекс** строится только по строкам таблицы, удовлетворяющим заданному предикату:
```sql
CREATE INDEX index_name ON table_name (column)
WHERE condition;
```

Примеры из документации:

1. Исключение внутренних IP-адресов из индекса по `client_ip`:
```sql
CREATE INDEX access_log_client_ip_ix ON access_log (client_ip)
WHERE NOT (client_ip > inet '192.168.100.0'
           AND client_ip < inet '192.168.100.255');
```

2. Индекс только по неоплаченным заказам:
```sql
CREATE INDEX orders_unbilled_index ON orders (order_nr)
WHERE billed IS NOT TRUE;
```
Индекс остаётся компактным, хотя таблица растёт: оплаченные заказы в него не попадают.

3. Частичный уникальный индекс — только по успешным тестам:
```sql
CREATE UNIQUE INDEX tests_success_constraint ON tests (subject, target)
WHERE success;
```

Планировщик использует частичный индекс, если предикат запроса математически следует из предиката индекса. Возможности автоматического доказательства у планировщика ограничены: работают простые сравнения, но не сложные логические выражения.

#### Ответ от Claude

**Частичный индекс (partial index)**

Частичный индекс содержит записи только для строк, соответствующих предикату WHERE. Это даёт двойное преимущество:
- индекс меньше по размеру (меньше записей);
- если индексируемый столбец есть в предикате и его значение всегда одинаково для включённых строк — его вообще не нужно включать в ключ (горизонтальное сжатие).

**Функциональный (expression) индекс**

Функциональный индекс строится по результату произвольного выражения или функции от столбца, а не по сырому значению:
```sql
CREATE INDEX idx_lower_email ON users (lower(email));
```
Теперь запрос `WHERE lower(email) = 'ivan@example.com'` воспользуется этим индексом. Требование: функция должна быть детерминированной (`IMMUTABLE`). Нарушение детерминизма (например, функция, зависящая от текущей даты) приведёт к тому, что индекс устареет сразу после вставки.

**Сценарии, где частичный индекс даёт существенный выигрыш**

1. **Очередь задач / необработанные записи.**
   Типичный запрос: «дай мне все необработанные сообщения для получателя X».
   ```sql
   CREATE INDEX messages_unprocessed ON messages (receiver)
   WHERE processed = FALSE;
   ```
   Таблица растёт в разы, но индекс не растёт: обработанные сообщения в него не попадают. По данным use-the-index-luke.com, размер индекса остаётся постоянным, пока доля необработанных записей фиксирована.

2. **Мягкое удаление (soft delete).**
   ```sql
   CREATE INDEX idx_active_users ON users (email)
   WHERE deleted_at IS NULL;
   ```
   Если 90% строк помечены как удалённые, обычный B-tree индекс по `email` содержит 10x лишних записей. Частичный индекс содержит только актуальные, размер меньше в 10 раз, поиск быстрее.

3. **Статусные столбцы со скошенным распределением.**
   ```sql
   CREATE INDEX idx_task_pending ON tasks (created_at)
   WHERE status <> 'DONE';
   ```
   По данным vladmihalcea.com: частичный индекс на таблице задач занимал 56 КБ, тогда как полный B-tree — сотни КБ. Кроме меньшего размера, такой индекс с большей вероятностью помещается в shared_buffers целиком.

4. **Частичное уникальное ограничение.**
   ```sql
   CREATE UNIQUE INDEX one_active_per_user ON subscriptions (user_id)
   WHERE status = 'ACTIVE';
   ```
   Обычный UNIQUE не позволил бы хранить историю (несколько строк с одним `user_id`). Частичный уникальный индекс ограничивает только активные строки.

**Ограничения**

- Запрос должен содержать условие, из которого логически следует предикат индекса. Планировщик имеет ограниченный инструментарий для вывода: простые равенства и неравенства работают; сложные OR/AND-комбинации могут не распознаться.
- Не стоит создавать серию непересекающихся частичных индексов по категориям вместо одного составного — это создаёт накладные расходы на планировщик и сложность сопровождения.

**Отличие от функционального индекса**

Частичный индекс отвечает на вопрос «какие строки индексировать», функциональный — «что именно индексировать в строке». Их можно совмещать:
```sql
CREATE INDEX idx_lower_email_active ON users (lower(email))
WHERE deleted_at IS NULL;
```

### Вопрос 12. Как работает GIN-индекс для JSONB в PostgreSQL? Какие классы операторов поддерживает jsonb_ops vs jsonb_path_ops, и как выбор класса влияет на размер индекса и поддерживаемые операторы (@>, ?, ?|, ?&)?

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 8.14. JSON Types](https://www.postgresql.org/docs/current/datatype-json.html)

GIN-индексы можно создавать на столбцах типа `jsonb`. Для этого существуют два встроенных класса операторов.

**jsonb_ops** — класс по умолчанию. Создаёт отдельные записи в индексе для каждого ключа и значения в документе. Для документа `{"foo": {"bar": "baz"}}` создаются три независимые записи: `foo`, `bar` и `baz`. Поддерживаемые операторы: `@>`, `@?`, `@@`, `?`, `?|`, `?&`.

```sql
CREATE INDEX idxgin ON api USING GIN (jdoc);
```

**jsonb_path_ops** — неосновной класс. Создаёт записи только для значений; каждая запись является хешем значения и всех ключей на пути к нему. Для того же документа создаётся одна запись, включающая в хеш `foo`, `bar` и `baz`. Поддерживаемые операторы: `@>`, `@?`, `@@`. Операторы `?`, `?|`, `?&` не поддерживаются.

```sql
CREATE INDEX idxginp ON api USING GIN (jdoc jsonb_path_ops);
```

Индекс `jsonb_path_ops` обычно существенно меньше по размеру, чем `jsonb_ops` на тех же данных, а точность поиска выше — особенно если запросы содержат ключи, часто встречающиеся в документах.

Исключение: `jsonb_path_ops` не может эффективно обрабатывать запросы вида `{"a": {}}` (поиск по пустому объекту) — для них потребуется полный перебор индекса.

#### Ответ от Claude

**Структура GIN-индекса.** GIN (Generalized Inverted Index) хранит пары *(ключ, список TID)*, где ключ — это элемент из индексируемого значения, а список TID — множество идентификаторов строк, содержащих этот ключ. Внутри GIN-индекс устроен как B-дерево по ключам; листовые узлы хранят либо posting list (простой список TID), либо posting tree (отдельное B-дерево по TID, если список слишком большой). Один TID может присутствовать в нескольких posting list-ах, так как документ содержит несколько ключей.

**Два класса операторов для JSONB:**

| Свойство | jsonb_ops (по умолчанию) | jsonb_path_ops |
|----------|--------------------------|----------------|
| Поддерживаемые операторы | `@>`, `@?`, `@@`, `?`, `?|`, `?&` | `@>`, `@?`, `@@` |
| Что индексируется | Каждый ключ и каждое значение раздельно | Только значения; ключи входят в хеш вместе со значением |
| Пример: `{"foo":{"bar":"baz"}}` | 3 записи: `foo`, `bar`, `baz` | 1 запись: хеш от `(foo, bar, baz)` |
| Размер индекса | Больше | Значительно меньше |
| Специфичность поиска | Ниже | Выше (меньше ложных совпадений по хешу) |

**Операторы и их смысл:**
- `@>` — проверка содержания: `'{"a":1,"b":2}'::jsonb @> '{"a":1}'` истинно.
- `?` — существование ключа верхнего уровня: `'{"a":1}'::jsonb ? 'a'`.
- `?|` — существование хотя бы одного из перечисленных ключей.
- `?&` — существование всех перечисленных ключей.
- `@?` — наличие пути jsonpath: `jdoc @? '$.tags[*] ? (@ == "qui")'`.
- `@@` — вычисление jsonpath с булевым результатом.

**Когда выбирать jsonb_path_ops:**
- Запросы используют только `@>`, `@?`, `@@`.
- Важен меньший размер индекса (особенно при большом количестве документов с повторяющимися ключами).
- Требуется более высокая избирательность поиска.

**Когда выбирать jsonb_ops:**
- Необходимы операторы существования ключей `?`, `?|`, `?&`.
- Выполняются запросы по пустым вложенным объектам `{"a": {}}` — `jsonb_path_ops` потребует полного прохода по индексу в этом случае.
- Паттерны запросов разнообразны и заранее неизвестны.

**Важное замечание о версиях:** Начиная с PostgreSQL 12, оба класса поддерживают операторы `@?` и `@@` (jsonpath). До PostgreSQL 12 эти операторы поддерживались только `jsonb_ops`.

### Вопрос 13. Опиши узлы плана выполнения запроса в EXPLAIN ANALYZE: Seq Scan, Index Scan, Index Only Scan, Bitmap Heap Scan, Nested Loop, Hash Join, Merge Join. Что означают поля 'actual rows', 'loops', 'Buffers: shared hit/read' и когда большое расхождение между плановыми и фактическими строками опасно?

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 14.1. Using EXPLAIN](https://www.postgresql.org/docs/current/using-explain.html)

`EXPLAIN ANALYZE` выполняет запрос и выводит оценочную стоимость вместе с реальными данными выполнения по каждому узлу плана.

**Типы узлов сканирования:**

- **Seq Scan** — последовательное чтение всей таблицы с диска, строка за строкой. Применяется, когда нет подходящего индекса или когда затрагивается большая доля таблицы.
- **Index Scan** — обход B-дерева индекса, проход по листовым узлам для нахождения совпадений, затем случайное чтение соответствующих строк из кучи (heap). Каждый TID из индекса сразу преобразуется в обращение к таблице.
- **Index Only Scan** — аналогичен Index Scan, но все нужные столбцы есть в индексе, поэтому обращение к heap не требуется. Дополнительно проверяется visibility map для подтверждения видимости строк.
- **Bitmap Heap Scan** — двухфазный процесс: сначала Bitmap Index Scan собирает все TID из индекса в битовую карту памяти, затем Bitmap Heap Scan читает страницы таблицы в физическом порядке. Эффективен при множестве совпадений, так как минимизирует случайный ввод-вывод.

**Типы узлов соединения:**

- **Nested Loop** — для каждой строки из внешней таблицы выполняется поиск совпадений во внутренней. Стоимость пропорциональна произведению размеров. Эффективен при небольших наборах данных или при доступе через индекс.
- **Hash Join** — строит хеш-таблицу из меньшей стороны соединения, затем проверяет каждую строку из большей стороны. Подходит для больших наборов при отсутствии подходящей сортировки. При нехватке памяти разбивается на батчи с записью на диск.
- **Merge Join** — объединяет два предварительно отсортированных набора «как молния». Требует, чтобы оба входа были упорядочены по ключам соединения. Эффективен, если данные уже отсортированы или есть подходящий индекс.

**Поля вывода EXPLAIN ANALYZE:**

- `actual rows` — среднее количество строк, возвращённых узлом за одно исполнение. Чтобы получить общее количество, нужно умножить на `loops`.
- `loops` — сколько раз узел был выполнен. Узлы внутри Nested Loop выполняются по одному разу для каждой строки внешней стороны.
- `Buffers: shared hit=N read=M` — `hit` — страницы, найденные в буферном кеше PostgreSQL (без обращения к диску); `read` — страницы, прочитанные с диска. Высокое значение `hit` говорит об эффективном использовании кеша.

**Расхождение оценочных и фактических строк.** Точность оценок — ключевое, на что нужно смотреть. Если оценки существенно расходятся с реальностью, это не всегда проблема само по себе, но может быть признаком устаревшей статистики. Необходимо учитывать: оценка отражает полное выполнение, тогда как `LIMIT` прерывает узел раньше.

#### Ответ от Claude

**Узлы сканирования:**

**Seq Scan** читает таблицу полностью, страница за страницей. Оптимизатор выбирает его, когда стоимость случайного чтения через индекс превышает стоимость последовательного чтения, либо когда индекса нет. Характерен для: небольших таблиц (они умещаются в кеше), запросов, затрагивающих более ~10–15% строк, таблиц с низкой селективностью условия.

**Index Scan** сначала обходит индекс и по каждому найденному TID случайно обращается к страницам heap. Выигрывает на высокоселективных запросах (небольшой процент строк), проигрывает при большом числе совпадений из-за накладных расходов на случайный ввод-вывод.

**Index Only Scan** аналогичен Index Scan, но heap вообще не читается, если все запрошенные столбцы есть в индексе. Единственное обращение к heap-подобной структуре — к visibility map для проверки, заморожены ли страницы. Visibility map на четыре порядка меньше heap, поэтому ввод-вывод минимален. Требует актуальной visibility map (обновляется VACUUM).

**Bitmap Heap Scan** — двухэтапный план. На первом этапе `Bitmap Index Scan` обходит индекс и строит битовую карту: один бит на страницу heap (lossy режим при нехватке work_mem) или на TID (exact режим). На втором этапе `Bitmap Heap Scan` читает heap-страницы в физическом порядке. Это снижает количество случайных операций чтения. При наличии нескольких индексов можно комбинировать несколько Bitmap Index Scan через узлы `BitmapAnd`/`BitmapOr`. В lossy-режиме узел выводит `Recheck Cond`.

**Nested Loop** — O(outer × inner) при отсутствии индекса на внутренней стороне. При наличии индекса на внутренней стороне — эффективен для небольших outer-наборов. Поле `loops` в EXPLAIN ANALYZE для внутреннего узла равно числу строк в outer-наборе: `actual rows = среднее на одну итерацию`, суммарно строк = `rows × loops`.

**Hash Join** — строит хеш-таблицу из меньшего набора (`Hash` узел в плане показывает `Batches`, `Buckets`, `Memory Usage`). Если `Batches > 1`, хеш-таблица не вместилась в `work_mem` и сбрасывается на диск — производительность резко падает. Рекомендуется контролировать значение `Batches` при анализе планов.

**Merge Join** — оба входа должны быть отсортированы по ключам соединения. Если одна из сторон не отсортирована, в плане появляется узел `Sort`. Эффективен при наличии уже отсортированных данных (индекс B-tree) или при соединении больших наборов с подходящим индексом.

**Поля EXPLAIN ANALYZE подробно:**

- `actual rows` — **среднее** число строк за одно выполнение узла. Суммарное число строк = `actual rows × loops`.
- `loops` — сколько раз выполнился узел. Для узлов внутри Nested Loop равно числу строк во внешнем наборе. При `loops > 1` время в строке `actual time` указано суммарное по всем итерациям; среднее время одной итерации = `actual time / loops`.
- `Buffers: shared hit=N read=M` — `shared hit`: страницы, найденные в shared_buffers (нет дискового ввода-вывода); `shared read`: страницы, прочитанные с диска или из OS page cache; `shared dirtied`: страницы, изменённые данным узлом; `shared written`: страницы, сброшенные на диск (обычно ноль при нормальной работе). Показатели агрегируются по узлу и всем его дочерним узлам.

**Когда расхождение оценок опасно:**

1. **Nested Loop с недооценёнными строками внешней стороны.** Оптимизатор полагает, что outer-набор мал, и выбирает Nested Loop вместо Hash Join. В действительности строк много — стоимость резко возрастает.

2. **Hash Join с недооценёнными строками на стороне хеш-таблицы.** Планировщик выделяет мало `work_mem`. Реальный объём данных вызывает `Batches > 1` — хеш-таблица выливается на диск.

3. **Цепочка соединений.** Ошибка оценки в одном узле умножается в родительских узлах. Если первый join переоценён в 10 раз, то второй join, ориентирующийся на его вывод, будет ошибаться ещё сильнее.

4. **ORDER BY + LIMIT с ошибкой оценки.** Планировщик может выбрать план, рассчитанный на быстрое получение первых строк, тогда как реальное распределение данных делает этот план медленнее полного сканирования.

**Признаки проблемы:** расхождение в 10 раз и более — повод запустить `ANALYZE` на таблице или пересмотреть гистограммы. Расхождение в 100 раз и более, особенно в нескольких узлах — серьёзная проблема со статистикой, возможно требующей `ALTER TABLE ... ALTER COLUMN ... SET STATISTICS` для увеличения числа точек гистограммы.

### Вопрос 14. Как autovacuum решает, когда запускать очистку таблицы? Объясни формулу на основе autovacuum_vacuum_threshold и autovacuum_vacuum_scale_factor. Почему дефолтные настройки плохо подходят для очень больших таблиц и как их тюнить на уровне конкретной таблицы через ALTER TABLE ... SET (autovacuum_...)?

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 24.1. Routine Vacuuming](https://www.postgresql.org/docs/current/routine-vacuuming.html)

Autovacuum daemon запускает очистку таблицы, когда число устаревших (мёртвых) кортежей превышает пороговое значение:

```
vacuum threshold = Minimum(
    autovacuum_vacuum_max_threshold,
    autovacuum_vacuum_threshold + autovacuum_vacuum_scale_factor × pg_class.reltuples
)
```

Аналогичная формула применяется для `ANALYZE`:

```
analyze threshold = autovacuum_analyze_threshold + autovacuum_analyze_scale_factor × pg_class.reltuples
```

Значения параметров по умолчанию:
- `autovacuum_vacuum_threshold` = 50 кортежей
- `autovacuum_vacuum_scale_factor` = 0.2 (20% от размера таблицы)
- `autovacuum_vacuum_max_threshold` = 100 000 000 кортежей
- `autovacuum_analyze_threshold` = 50 кортежей
- `autovacuum_analyze_scale_factor` = 0.1 (10% от размера таблицы)

Текущее число мёртвых кортежей отслеживается в `pg_stat_user_tables.n_dead_tup`.

Для переопределения настроек на уровне таблицы используется `ALTER TABLE ... SET`:

```sql
ALTER TABLE my_large_table SET (
    autovacuum_vacuum_scale_factor = 0.01,
    autovacuum_vacuum_threshold = 1000
);
```

Если параметр задан через storage parameter таблицы, autovacuum использует его вместо глобального значения из `postgresql.conf`.

#### Ответ от Claude

**Формула срабатывания autovacuum VACUUM.**

Autovacuum launcher периодически проверяет каждую таблицу. Очистка запускается, когда выполняется условие:

```
n_dead_tup > MIN(autovacuum_vacuum_max_threshold,
                 autovacuum_vacuum_threshold + autovacuum_vacuum_scale_factor × reltuples)
```

Где:
- `n_dead_tup` — накопленное число мёртвых кортежей (обновлённых или удалённых, ещё не очищенных), из `pg_stat_user_tables`.
- `autovacuum_vacuum_threshold` (по умолчанию **50**) — абсолютный минимум мёртвых кортежей, ниже которого вакуум не запускается вне зависимости от размера таблицы.
- `autovacuum_vacuum_scale_factor` (по умолчанию **0.2**) — доля от оценочного числа живых строк (`reltuples`).
- `autovacuum_vacuum_max_threshold` (по умолчанию **100 000 000**) — жёсткий потолок: если порог по формуле превышает это значение, используется именно оно. Введён в PostgreSQL 17. Значение -1 снимает ограничение сверху.
- `reltuples` — приблизительное число кортежей из `pg_class`, обновляется при `ANALYZE` и `VACUUM`.

**Пример расчёта.** Таблица с 50 000 строк, дефолтные настройки:

```
threshold = min(100000000, 50 + 0.2 × 50000) = min(100000000, 10050) = 10050
```

Вакуум запустится только после накопления 10 050 мёртвых кортежей — 20% таблицы.

**Почему дефолты плохи для больших таблиц.**

Проблема scale_factor = 0.2 в том, что порог растёт линейно с размером таблицы:

| Размер таблицы | Порог вакуума (дефолт) |
|----------------|----------------------|
| 100 000 строк  | 20 050              |
| 1 000 000 строк | 200 050            |
| 10 000 000 строк | 2 000 050          |
| 100 000 000 строк | 20 000 050        |

Для таблицы с 10 миллионами строк autovacuum не запустится, пока не накопится 2 миллиона мёртвых кортежей — 20% таблицы. Это означает:
- **Таблица раздувается** (bloat): страницы с мёртвыми кортежами занимают место на диске, не переиспользуются до вакуума.
- **Индексы раздуваются** вместе с таблицей.
- **Запросы замедляются**: seq scan вынужден читать мёртвые страницы, индексы содержат ссылки на мёртвые версии строк.
- **Заморозка транзакционных ID (anti-wraparound vacuum)** может не поспевать, что в пределе приводит к аварийному VACUUM.

До PostgreSQL 17 параметра `autovacuum_vacuum_max_threshold` не существовало, поэтому для таблицы с 500 млн строк порог достигал 100 млн мёртвых кортежей при дефолтных настройках.

**Тюнинг на уровне отдельной таблицы.**

PostgreSQL позволяет задавать параметры autovacuum как storage parameters конкретной таблицы. Это предпочтительнее изменения глобального `postgresql.conf`, так как не затрагивает остальные таблицы.

```sql
-- Большая активно обновляемая таблица: вакуум при 1% мёртвых строк
ALTER TABLE orders SET (
    autovacuum_vacuum_scale_factor = 0.01,
    autovacuum_vacuum_threshold = 1000,
    autovacuum_analyze_scale_factor = 0.005,
    autovacuum_analyze_threshold = 500
);

-- Таблица с критически важным временем отклика: ускорить вакуум
ALTER TABLE payments SET (
    autovacuum_vacuum_cost_delay = 2,     -- мс, дефолт 2мс (с PG 13)
    autovacuum_vacuum_cost_limit = 400    -- единиц ввода-вывода за задержку
);

-- Сбросить параметры таблицы на глобальные дефолты
ALTER TABLE orders RESET (autovacuum_vacuum_scale_factor);
```

Проверить текущие storage parameters таблицы:

```sql
SELECT reloptions FROM pg_class WHERE relname = 'orders';
```

Мониторинг: смотреть `pg_stat_user_tables` — поля `n_dead_tup`, `last_autovacuum`, `autovacuum_count`:

```sql
SELECT schemaname, relname,
       n_live_tup, n_dead_tup,
       round(100.0 * n_dead_tup / NULLIF(n_live_tup + n_dead_tup, 0), 2) AS dead_pct,
       last_autovacuum
FROM pg_stat_user_tables
WHERE n_live_tup > 100000
ORDER BY dead_pct DESC NULLS LAST;
```

**Расхождение источников:** Официальная документация PostgreSQL 18 указывает дефолт `autovacuum_vacuum_scale_factor = 0.2`, тогда как ряд статей (включая выдачу поиска) иногда называют значение 0.1. Значение 0.1 относится к `autovacuum_analyze_scale_factor` (для ANALYZE), а не к VACUUM. Для VACUUM дефолт — **0.2**. Параметр `autovacuum_vacuum_max_threshold` появился в **PostgreSQL 17** и отсутствует в более ранних версиях.
### Вопрос 15. Чем потоковая репликация (streaming replication) отличается от логической репликации (logical replication) в PostgreSQL? Когда логическая репликация предпочтительнее и какие ограничения у неё есть по сравнению с физической?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 27.1. Comparison of Different Solutions](https://www.postgresql.org/docs/current/different-replication-solutions.html)

Потоковая (физическая) репликация передаёт WAL-записи на уровне байтов/блоков и реплицирует весь сервер целиком. Сервер может быть либо primary, либо standby; standby всегда является получателем данных.

Логическая репликация конструирует поток логических изменений данных из WAL и позволяет реплицировать данные на уровне отдельных таблиц. Серверы выступают в роли publisher (создаёт `CREATE PUBLICATION`) и subscriber (создаёт `CREATE SUBSCRIPTION`). Сервер может одновременно быть publisher и subscriber, что позволяет организовывать двунаправленный поток данных.

Ключевое различие: потоковая репликация передаёт байты WAL; логическая декодирует WAL и передаёт логические изменения данных.

#### Ответ от Claude

**Потоковая (физическая) репликация:**

- Работает на уровне блоков диска: primary передаёт WAL-сегменты standby-серверу в режиме реального времени через постоянное TCP-соединение.
- Реплицируется весь экземпляр PostgreSQL целиком — все базы данных, все таблицы, системные каталоги.
- Standby является побайтовой копией primary; версии PostgreSQL на обоих серверах должны совпадать (мажорная версия).
- Standby недоступен для записи и поддерживает только чтение (в режиме hot standby).
- Подходит для disaster recovery, high availability, offloading read-only запросов.

**Логическая репликация:**

- Работает через механизм logical decoding: WAL декодируется в логические операции (`INSERT`, `UPDATE`, `DELETE`), которые затем применяются на subscriber.
- Реплицируются только явно включённые в публикацию таблицы (гранулярность до таблицы).
- Subscriber может быть другой версии PostgreSQL (в том числе более новой) — это позволяет делать zero-downtime мажорные обновления.
- Subscriber может быть доступен для записи и принимать данные из нескольких publisher.
- Возможна двунаправленная и мультимастер-репликация.

**Когда логическая репликация предпочтительнее:**

1. Нужно реплицировать только часть таблиц, а не весь кластер.
2. Необходим мажорный апгрейд PostgreSQL без остановки сервиса.
3. Требуется репликация между разными платформами или сборками.
4. Нужна двунаправленная или мультимастер-репликация.
5. Subscriber должен принимать собственные записи (не read-only реплика).

**Ограничения логической репликации (по сравнению с физической):**

- **DDL не реплицируется**: изменения схемы (`ALTER TABLE`, `CREATE INDEX` и др.) не передаются автоматически; схемы нужно синхронизировать вручную.
- **Последовательности (sequences) не реплицируются**: данные serial/identity-колонок реплицируются как часть строк, но сами sequence-объекты — нет; при switchover нужно обновлять вручную.
- **Large objects не реплицируются**: данные pg_largeobject не передаются; следует хранить данные в обычных таблицах.
- **Только таблицы**: представления (views), материализованные представления и foreign tables не поддерживаются.
- **TRUNCATE** реплицируется, но с ограничениями при наличии внешних ключей на таблицы вне подписки.
- **Конфликты**: если subscriber принимает записи из нескольких источников или от приложений, возможны конфликты (нарушения уникальных ключей и пр.), которые нужно разрешать вручную.
- **REPLICA IDENTITY**: для репликации `UPDATE`/`DELETE` таблица должна иметь первичный ключ или явно заданный `REPLICA IDENTITY`; при `REPLICA IDENTITY FULL` не работают типы без B-tree/Hash операторного класса.
- Более высокие накладные расходы на primary: требуется logical decoding WAL.

### Вопрос 16. Что такое hot standby в PostgreSQL и какие запросы допустимы на реплике в режиме recovery? Какой параметр управляет допустимой задержкой данных при чтении с реплики и как настроить балансировку нагрузки между primary и standby?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 26.4. Hot Standby](https://www.postgresql.org/docs/current/hot-standby.html)

Hot standby — возможность подключаться к серверу и выполнять read-only запросы в то время, когда сервер находится в режиме archive recovery или standby. Сервер начинает принимать подключения после того, как процесс восстановления достигает согласованного состояния.

На реплике в режиме hot standby разрешены только read-only операции: `SELECT`, `COPY TO`, курсорные команды (`DECLARE`, `FETCH`, `CLOSE`), `SHOW`, `SET`, `RESET`, `BEGIN`, `SAVEPOINT`, `ROLLBACK TO SAVEPOINT`, а также `LOCK TABLE` только в режимах `ACCESS SHARE`, `ROW SHARE`, `ROW EXCLUSIVE`.

Запрещены: любые DML (`INSERT`, `UPDATE`, `DELETE`, `MERGE`, `TRUNCATE`), DDL, `SELECT ... FOR UPDATE/FOR SHARE`, `BEGIN READ WRITE`, функции изменения последовательностей (`nextval()`, `setval()`), `LISTEN`, `NOTIFY`, двухфазные транзакции.

Параметр `hot_standby_feedback` (по умолчанию `off`) позволяет standby отправлять primary информацию об активных запросах, предотвращая удаление строк через VACUUM пока они нужны реплике, что снижает число конфликтов при восстановлении.

#### Ответ от Claude

**Что такое hot standby:**

Hot standby — режим работы PostgreSQL standby-сервера, при котором он принимает read-only подключения клиентов одновременно с применением WAL-записей от primary. Активируется параметром `hot_standby = on` (значение по умолчанию начиная с PostgreSQL 10) в `postgresql.conf` standby-сервера.

**Допустимые операции на реплике:**

Разрешены:
- Запросы на чтение: `SELECT`, `COPY TO`, курсоры (`DECLARE`, `FETCH`, `CLOSE`)
- Управление транзакциями: `BEGIN` (только read-only), `SAVEPOINT`, `ROLLBACK TO SAVEPOINT`, `COMMIT`
- Настройка сессии: `SET`, `RESET`, `SHOW`
- Подготовленные выражения: `PREPARE`, `EXECUTE`, `DEALLOCATE`
- `LOCK TABLE` в режимах `ACCESS SHARE`, `ROW SHARE`, `ROW EXCLUSIVE`

Запрещены:
- Любые DML: `INSERT`, `UPDATE`, `DELETE`, `MERGE`, `TRUNCATE`, `COPY FROM`
- DDL: `CREATE`, `DROP`, `ALTER` и т.д.
- `SELECT ... FOR UPDATE` / `FOR SHARE` (блокирующие чтения)
- `BEGIN READ WRITE` или `SET TRANSACTION READ WRITE`
- Модификация последовательностей: `nextval()`, `setval()`
- `LISTEN`, `NOTIFY`, двухфазные транзакции

**Параметры, управляющие задержкой данных:**

- `max_standby_streaming_delay` (по умолчанию `30s`): максимальное время, которое standby ждёт перед отменой запросов, конфликтующих с поступающими WAL-записями при потоковой репликации. Значение `-1` означает ожидание неограниченно долго (подходит для аналитических нагрузок). Значение `0` — агрессивная отмена конфликтующих запросов.
- `max_standby_archive_delay` (по умолчанию `30s`): аналогичный параметр для репликации через WAL-архив (log shipping).
- `hot_standby_feedback` (по умолчанию `off`): если включён, standby регулярно сообщает primary о самой старой активной транзакции, не позволяя VACUUM удалять строки, которые ещё видны открытым транзакциям на реплике. Устраняет большинство конфликтов вида `confl_snapshot`, но ценой потенциального раздувания таблиц на primary.

**Балансировка нагрузки между primary и standby:**

PostgreSQL не включает встроенного балансировщика запросов. Типовые подходы:

1. **На уровне приложения**: использовать два пула соединений — один к primary (read-write), другой к standby (read-only). Функция `pg_is_in_recovery()` возвращает `true` на standby и `false` на primary.

2. **PgBouncer**: настроить несколько connection pools с разными хостами; приложение выбирает пул в зависимости от типа запроса.

3. **HAProxy / Patroni**: Patroni управляет failover и предоставляет REST API для определения роли сервера; HAProxy направляет трафик на primary (порт 5000) или standby (порт 5001) в зависимости от health-check.

4. **JDBC/psycopg connection string**: JDBC-драйвер поддерживает параметр `targetServerType=preferSecondary` для автоматического выбора standby при наличии.

Мониторинг задержки репликации и конфликтов:
```sql
-- Задержка репликации
SELECT now() - pg_last_xact_replay_timestamp() AS replication_delay;

-- Конфликты по базам данных
SELECT datname, confl_snapshot, confl_lock, confl_bufferpin
FROM pg_stat_database_conflicts;
```

### Вопрос 17. Как работает SELECT ... FOR UPDATE SKIP LOCKED и зачем он нужен при реализации очередей задач в PostgreSQL? Чем SKIP LOCKED отличается от NOWAIT?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [How to implement a database job queue using SKIP LOCKED — Vlad Mihalcea](https://vladmihalcea.com/database-job-queue-skip-locked/)

`SELECT FOR UPDATE SKIP LOCKED` позволяет нескольким параллельным воркерам безопасно обрабатывать задачи из таблицы-очереди без конкуренции за блокировки. Когда одна транзакция блокирует строки, другие транзакции автоматически пропускают их и берут следующие доступные строки.

**Различие SKIP LOCKED и NOWAIT:**
- `NOWAIT`: если строки уже заблокированы, немедленно бросает исключение и транзакция завершается с ошибкой.
- `SKIP LOCKED`: молча пропускает заблокированные строки и возвращает следующие незаблокированные.

Для очередей задач `SKIP LOCKED` является предпочтительным вариантом, поскольку позволяет множеству воркеров конкурентно обрабатывать задачи без ошибок и ожидания.

Пример с двумя параллельными воркерами и 10 задачами:
- Воркер A берёт задачи с ID 0–1 и блокирует их.
- Воркер B запускается одновременно, пропускает заблокированные задачи и берёт задачи с ID 2–3.
- Оба завершаются успешно без конфликтов.

Поддерживается в PostgreSQL начиная с версии 9.5.

#### Ответ от Claude

**Как работает SELECT ... FOR UPDATE:**

`SELECT ... FOR UPDATE` блокирует выбранные строки на уровне строк (row-level lock). Другие транзакции, пытающиеся выполнить `UPDATE`, `DELETE` или `SELECT ... FOR UPDATE` на тех же строках, будут ждать, пока первая транзакция завершится (commit или rollback).

**Модификатор SKIP LOCKED:**

При указании `SKIP LOCKED` строки, которые уже заблокированы другой транзакцией, не вызывают ожидания — они молча исключаются из результата запроса. Транзакция немедленно получает только те строки, которые удалось заблокировать.

Важно: `SKIP LOCKED` обеспечивает несогласованное (inconsistent) представление данных — часть строк не видна запросу. Это недопустимо для общего использования, но идеально подходит для паттерна очереди задач.

**Модификатор NOWAIT:**

При указании `NOWAIT` строка, которую не удаётся заблокировать немедленно, вызывает немедленную ошибку (`ERROR: could not obtain lock on row in relation "..."`) и откат транзакции.

**Ключевые различия:**

| Ситуация | SKIP LOCKED | NOWAIT |
|---|---|---|
| Строка уже заблокирована | Строка пропускается молча | Немедленная ошибка |
| Результат запроса | Только незаблокированные строки | Пустой (при ошибке) |
| Поведение воркера | Берёт следующую доступную задачу | Должен обработать ошибку и повторить |
| Применимость для очередей | Оптимально | Требует retry-логики |

**Применение в очереди задач (job queue):**

Типовой паттерн:
```sql
-- Воркер берёт одну задачу из очереди
BEGIN;
SELECT id, payload
FROM job_queue
WHERE status = 'pending'
ORDER BY created_at
LIMIT 1
FOR UPDATE SKIP LOCKED;

-- обработка задачи ...

UPDATE job_queue SET status = 'done' WHERE id = <полученный id>;
COMMIT;
```

При таком подходе:
- Несколько воркеров могут работать параллельно без конфликтов.
- Каждый воркер гарантированно получает уникальную задачу: строка блокируется на время транзакции.
- Если воркер завершается аварийно (rollback), строка автоматически разблокируется и становится доступна другим воркерам.
- `ORDER BY` гарантирует честную очередь (FIFO); без него порядок выборки не определён.

**Технические детали:**

- `SKIP LOCKED` применяется только к row-level блокировкам; table-level блокировка `ROW SHARE` всё равно берётся обычным образом.
- Если одна таблица присутствует в нескольких `FOR ... ` предложениях одного запроса, используется наиболее сильный режим блокировки; таблица помечается `NOWAIT`, если хотя бы одно предложение содержит `NOWAIT`.
- Доступно начиная с PostgreSQL 9.5.
- `SKIP LOCKED` несовместим с `WITH TIES` (невозможно совместить пропуск строк с возвратом дополнительных строк при равенстве значений ORDER BY).
### Вопрос 18. Что такое advisory locks в PostgreSQL, чем они отличаются от стандартных строковых и табличных блокировок, и как используются функции pg_advisory_lock / pg_try_advisory_lock? Приведи реальный use-case.

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 13.3. Explicit Locking](https://www.postgresql.org/docs/current/explicit-locking.html)

PostgreSQL предоставляет средства для создания блокировок с определяемой приложением семантикой — advisory locks (консультативные блокировки). Поскольку система не применяет их принудительно, использование таких блокировок целиком ответственность приложения.

Advisory locks полезны для стратегий блокировки, которые плохо ложатся на модель MVCC. Например, их часто применяют для эмуляции пессимистических блокировок, характерных для систем управления «плоскими файлами».

Хотя для той же цели можно использовать флаг в обычной таблице, advisory locks работают быстрее, не вызывают раздувания таблицы (table bloat) и автоматически очищаются сервером по окончании сессии.

Advisory locks бывают двух уровней:

- **Уровень сессии (session-level):** блокировка удерживается до явного вызова `pg_advisory_unlock()` или завершения сессии. Она не следует транзакционной семантике — блокировка переживает `ROLLBACK`.
- **Уровень транзакции (transaction-level):** блокировка автоматически освобождается по окончании транзакции и не требует явного unlock. Функции: `pg_advisory_xact_lock()`, `pg_try_advisory_xact_lock()`.

Функции получения блокировок:

- `pg_advisory_lock(key bigint)` — ожидает до получения эксклюзивной блокировки (блокирующий вызов).
- `pg_try_advisory_lock(key bigint)` — возвращает `true`, если блокировка получена немедленно, или `false`, если нет (неблокирующий вызов).

Аналогично работают shared-варианты: `pg_advisory_lock_shared()` / `pg_try_advisory_lock_shared()`, разрешающие несколько одновременных читателей и блокирующие запись.

**Важное предупреждение из документации.** Следующая форма опасна, потому что LIMIT может не применяться до выполнения функции блокировки:

```sql
-- ОПАСНО: может заблокировать больше строк, чем ожидается
SELECT pg_advisory_lock(id) FROM foo WHERE id > 12345 LIMIT 100;
```

Безопасный вариант — подзапрос:

```sql
-- БЕЗОПАСНО: LIMIT применяется до вызова блокировки
SELECT pg_advisory_lock(q.id)
FROM (SELECT id FROM foo WHERE id > 12345 LIMIT 100) q;
```

#### Ответ от Claude

**Что такое advisory locks**

Advisory locks — это блокировки с пользовательской семантикой, которые PostgreSQL хранит в разделяемой памяти. В отличие от строковых (`FOR UPDATE`, `FOR SHARE`) и табличных (`LOCK TABLE`) блокировок, которые система применяет автоматически для защиты конкретных объектов, advisory locks не привязаны ни к каким строкам или таблицам. Их смысл определяет приложение — PostgreSQL лишь хранит их и предоставляет примитивы синхронизации.

**Отличия от стандартных блокировок**

| Аспект | Строковые / табличные блокировки | Advisory locks |
|---|---|---|
| Применение | Автоматически системой | Только по явному вызову из приложения |
| Привязка к данным | К конкретной строке или таблице | К произвольному числу (int4 или int8), смысл определяет приложение |
| Транзакционность | Снимаются по окончании транзакции | Session-level переживают `ROLLBACK`; transaction-level снимаются с транзакцией |
| Производительность | Стандартные накладные расходы | Быстрее, без table bloat |
| Очистка при сбое | Автоматически при завершении транзакции | Session-level автоматически при разрыве сессии |

**Функции и их различия**

```sql
-- Блокирующий вызов: ждёт, пока блокировка не станет доступна
SELECT pg_advisory_lock(12345);

-- Неблокирующий вызов: сразу возвращает true/false
SELECT pg_try_advisory_lock(12345);
-- Используется там, где нужно проверить доступность без ожидания

-- Transaction-level (автоматически снимается с транзакцией)
SELECT pg_advisory_xact_lock(12345);
SELECT pg_try_advisory_xact_lock(12345);

-- Shared-варианты (несколько читателей одновременно)
SELECT pg_advisory_lock_shared(12345);
SELECT pg_try_advisory_lock_shared(12345);
```

Идентификатор блокировки — целое число (одно `bigint` или два `int`). Приложение само назначает числовые ID для своих ресурсов.

**Реальный use-case: распределённая очередь задач (job scheduler)**

Классическая проблема: несколько воркеров конкурируют за незаблокированные задачи в таблице `jobs`. Стандартные строковые блокировки потребуют `SKIP LOCKED` и усложняют логику; advisory locks позволяют реализовать это элегантнее.

```sql
-- Воркер пытается захватить задачу без ожидания
WITH candidate AS (
    SELECT id FROM jobs
    WHERE status = 'pending'
    ORDER BY created_at
    LIMIT 1
)
SELECT id, pg_try_advisory_lock(id) AS locked
FROM candidate;
```

Если `locked = true` — воркер получил эксклюзивное право обработки этой задачи. Другой воркер, вызвав `pg_try_advisory_lock` с тем же ID, получит `false` и возьмёт следующую задачу. По завершении обработки воркер вызывает `pg_advisory_unlock(id)`.

Аналогичный паттерн применяется в распределённых системах для distributed locks: вместо создания миллиардной lock-таблицы в PostgreSQL приложение использует advisory lock с hash-ом имени ресурса (например, `hashtext('resource-name')`).

**Ресурсные ограничения**

Advisory locks хранятся в разделяемой памяти вместе с обычными блокировками. Суммарное количество ограничено параметрами `max_locks_per_transaction * max_connections`. Исчерпание этого пула приведёт к невозможности выдачи любых блокировок. Следить за текущим состоянием можно через представление `pg_locks` (столбец `locktype = 'advisory'`).

### Вопрос 19. Объясни синтаксис INSERT ... ON CONFLICT (UPSERT) в PostgreSQL. Чем ON CONFLICT DO UPDATE отличается от ON CONFLICT DO NOTHING? Как RETURNING используется для получения результата upsert-а в приложении?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — INSERT](https://www.postgresql.org/docs/current/sql-insert.html)

Конструкция `ON CONFLICT` задаёт альтернативное действие при нарушении ограничения уникальности или ограничения исключения. Такая операция неформально называется «UPSERT» (UPDATE or INSERT).

**Синтаксис:**

```sql
INSERT INTO table_name [(column, ...)]
    VALUES (...) | query
    ON CONFLICT [ conflict_target ] conflict_action
```

Где `conflict_target` — это:
- `(column_name [, ...])` — вывод уникального индекса по столбцам;
- `ON CONSTRAINT constraint_name` — явное указание ограничения;
- `(column_name) WHERE predicate` — частичный уникальный индекс с условием.

**DO NOTHING**

Просто пропускает вставку при конфликте. Указание `conflict_target` необязательно — без него обрабатываются конфликты со всеми применимыми ограничениями:

```sql
INSERT INTO distributors (did, dname)
    VALUES (7, 'Redline GmbH')
    ON CONFLICT (did) DO NOTHING;
```

**DO UPDATE**

Обновляет существующую конфликтующую строку. Указание `conflict_target` обязательно. Специальная таблица `EXCLUDED` ссылается на значения, которые предполагалось вставить:

```sql
INSERT INTO distributors (did, dname)
    VALUES (5, 'Gizmo Transglobal'), (6, 'Associated Computing, Inc')
    ON CONFLICT (did)
    DO UPDATE SET dname = EXCLUDED.dname;
```

`ON CONFLICT DO UPDATE` гарантирует атомарный результат `INSERT` или `UPDATE` — одно из двух всегда происходит, даже в условиях высокой конкурентности.

**RETURNING**

`RETURNING` возвращает значения, основанные на каждой фактически вставленной или обновлённой строке. Начиная с PostgreSQL 17, доступны квалификаторы `OLD` и `NEW` для получения значений до и после изменения:

```sql
INSERT INTO distributors (did, dname)
    VALUES (5, 'Gizmo Transglobal')
    ON CONFLICT (did)
    DO UPDATE SET dname = EXCLUDED.dname
RETURNING
    OLD.dname AS old_name,
    NEW.dname AS new_name;
```

Для строк, вставленных без конфликта, `OLD.*` возвращает `NULL`.

#### Ответ от Claude

**Общий синтаксис UPSERT**

Конструкция `INSERT ... ON CONFLICT` появилась в PostgreSQL 9.5 как стандартизированная альтернатива хакам на основе CTE с `WITH ... UPDATE`. Она гарантирует атомарность — параллельные транзакции не могут нарушить инвариант «ровно одна строка с данным ключом».

```sql
INSERT INTO products (sku, name, price)
VALUES ('ABC-100', 'Widget', 9.99)
ON CONFLICT (sku) DO UPDATE
    SET name  = EXCLUDED.name,
        price = EXCLUDED.price,
        updated_at = now();
```

**DO NOTHING vs DO UPDATE: ключевые различия**

| Аспект | DO NOTHING | DO UPDATE |
|---|---|---|
| Действие при конфликте | Строка молча пропускается | Существующая строка обновляется |
| conflict_target | Необязателен (если опущен — любой конфликт) | Обязателен |
| RETURNING при конфликте | Возвращает пустой результат для конфликтной строки | Возвращает строку после обновления |
| Гарантия атомарности | Да (либо вставлено, либо ничего) | Да (либо вставлено, либо обновлено) |
| Используется | Идемпотентные вставки, игнорирование дублей | Синхронизация данных, кэш-таблицы |

**Таблица EXCLUDED**

В блоке `DO UPDATE` специальная таблица `EXCLUDED` содержит значения строки, которую пытались вставить. Это позволяет использовать как новые, так и старые значения:

```sql
INSERT INTO counters (name, value)
VALUES ('page_views', 1)
ON CONFLICT (name)
DO UPDATE SET value = counters.value + EXCLUDED.value;
-- counters.value — текущее значение в таблице
-- EXCLUDED.value — значение из INSERT (1 в данном случае)
```

Добавить условие на обновление через `WHERE`:

```sql
ON CONFLICT (did)
DO UPDATE SET dname = EXCLUDED.dname
WHERE EXCLUDED.updated_at > distributors.updated_at;
-- обновляет только если новая версия «свежее»
```

**RETURNING: получение результата в приложении**

`RETURNING` критически важен для определения того, что именно произошло (INSERT или UPDATE), и для получения сгенерированных сервером значений (serial/UUID, `now()`, триггерные вычисления).

```sql
-- Получить id независимо от того, была ли вставка или обновление
INSERT INTO users (email, name)
VALUES ('alice@example.com', 'Alice')
ON CONFLICT (email)
DO UPDATE SET name = EXCLUDED.name
RETURNING id, (xmax = 0) AS inserted;
-- xmax = 0 означает, что строка была вставлена, а не обновлена
-- xmax != 0 означает обновление (системный столбец, указывает на удаляющую транзакцию)
```

Трюк с `xmax` — распространённый способ различить INSERT и UPDATE в одном запросе до PostgreSQL 17.

С PostgreSQL 17 квалификаторы `OLD`/`NEW` делают это явно:

```sql
INSERT INTO users (email, name)
VALUES ('alice@example.com', 'Alice')
ON CONFLICT (email)
DO UPDATE SET name = EXCLUDED.name
RETURNING
    OLD.name AS previous_name,  -- NULL при INSERT
    NEW.name AS current_name,
    NEW.id;
```

**Поведение RETURNING при DO NOTHING**

При `DO NOTHING` конфликтная строка не возвращается в `RETURNING`. Если приложению нужен id существующей строки, необходим отдельный SELECT или переход на `DO UPDATE`:

```sql
-- Не вернёт строку при конфликте:
INSERT INTO tags (name) VALUES ('java')
ON CONFLICT (name) DO NOTHING
RETURNING id;  -- пусто при конфликте

-- Вернёт строку всегда (идемпотентное обновление):
INSERT INTO tags (name) VALUES ('java')
ON CONFLICT (name) DO UPDATE SET name = EXCLUDED.name
RETURNING id;  -- всегда возвращает id
```

**Важное ограничение:** `DO UPDATE` не может передвигать строку в другую секцию партиционированной таблицы (нельзя обновлять partition key до значения, меняющего секцию).

### Вопрос 20. Почему PostgreSQL использует модель «один процесс на соединение» (process-per-connection)? Какие проблемы это создаёт при высоком числе клиентов и как PgBouncer решает их — чем transaction pooling отличается от session pooling и почему transaction pooling несовместим с некоторыми фичами PostgreSQL?

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 1.2. Architectural Fundamentals](https://www.postgresql.org/docs/current/tutorial-arch.html)

PostgreSQL использует модель «один процесс на пользователя» (process-per-user). Сервер может обрабатывать несколько параллельных клиентских соединений. Для этого запускается («разветвляется») новый процесс для каждого соединения. С этого момента клиент и новый серверный процесс общаются напрямую, без участия исходного процесса `postgres`. Таким образом, главный серверный процесс (postmaster) всегда запущен и ожидает клиентских соединений; и клиентские, и связанные серверные процессы приходят и уходят.

Из раздела о том, как устанавливаются соединения: postmaster слушает указанный TCP/IP порт и порождает новый backend-процесс при каждом запросе на соединение. Каждый backend-процесс обрабатывает ровно одно клиентское соединение и координируется с другими backend-процессами через семафоры и разделяемую память.

#### Ответ от Claude

**Почему PostgreSQL выбрал process-per-connection**

Архитектурное решение восходит к истокам PostgreSQL (POSTGRES, Беркли, 1980-е). Причины:

1. **Изоляция и надёжность.** Падение одного backend-процесса не затрагивает остальных. Нет проблем с thread-safety для библиотек, не поддерживающих многопоточность.
2. **Безопасность.** OS-level изоляция процессов — каждый клиент видит только свою память.
3. **Copy-on-write при fork().** На Linux fork() практически не дороже pthread_create() благодаря механизму CoW — страницы памяти копируются только при изменении, что позволяет backend-процессам разделять кодовые сегменты и read-only данные.

**Проблемы при высоком числе соединений**

Каждый backend-процесс потребляет ресурсы даже в состоянии ожидания:

- **Память:** каждый процесс занимает 5–10 МБ RSS (стек, локальный кэш, буферы). При 1000 соединений — 5–10 ГБ только на idle-процессы.
- **Накладные расходы на fork/exit:** создание и уничтожение процесса при каждом подключении клиента занимает время (особенно при коротких соединениях).
- **Планировщик ОС:** тысячи процессов создают нагрузку на scheduler, даже если большинство из них спит.
- **Ограничение `max_connections`:** превышение приводит к отказу новым клиентам с ошибкой `FATAL: remaining connection slots are reserved`.

На практике рекомендуется держать `max_connections` в диапазоне 100–300 и использовать connection pooler для приложений с высоким числом клиентов.

**Как PgBouncer решает проблему**

PgBouncer — легковесный connection pooler, работающий как прокси между приложениями и PostgreSQL. Приложения подключаются к PgBouncer (на его порт), а PgBouncer поддерживает небольшой пул реальных соединений к PostgreSQL.

```
Приложение (1000 соединений)
        ↓
   PgBouncer (2 KB/conn)
        ↓
PostgreSQL (20–50 реальных backend-процессов)
```

Ключевое свойство: PgBouncer расходует всего ~2 КБ на клиентское соединение (против 5–10 МБ у backend-процесса PostgreSQL).

**Session pooling vs Transaction pooling**

| Аспект | Session pooling | Transaction pooling |
|---|---|---|
| Назначение backend-соединения | На всё время клиентской сессии | Только на время транзакции |
| Совместное использование | Один клиент — одно backend-соединение | Backend возвращается в пул после каждой транзакции |
| Эффективность | Низкая (1:1) | Высокая (N клиентов : M backend, N >> M) |
| Совместимость с фичами PG | Полная | Ограниченная |

При **session pooling** клиент удерживает backend-соединение на всё время своего подключения — поведение идентично прямому подключению к PostgreSQL. Выигрыш лишь в экономии на fork/exit.

При **transaction pooling** backend-соединение занято только пока выполняется транзакция. Между транзакциями оно возвращается в пул и может быть использовано другим клиентом. Это позволяет обслуживать тысячи «спящих» клиентов с помощью десятков реальных backend-процессов.

**Почему transaction pooling несовместим с некоторыми фичами PostgreSQL**

Причина фундаментальная: session-level состояние хранится в backend-процессе. При transaction pooling после завершения транзакции клиент «теряет» свой backend — следующая транзакция может попасть на другой backend без этого состояния.

Конкретные проблемы:

- **`SET`/`RESET`** (переменные сессии, `SET search_path`, `SET TimeZone`): состояние теряется при смене backend-а. Частичный обход — использовать `SET LOCAL` (действует только в рамках транзакции).
- **Session-level advisory locks** (`pg_advisory_lock`): блокировка привязана к backend-процессу, после смены backend-а она теряется, что нарушает всю логику взаимного исключения.
- **`LISTEN`/`NOTIFY`**: подписки на каналы уведомлений привязаны к сессии. Уведомления, отправленные в промежутке между транзакциями, теряются.
- **`WITH HOLD` курсоры** (`DECLARE ... WITH HOLD`): такой курсор живёт вне транзакции, но в рамках сессии — при смене backend-а данные курсора недоступны.
- **`PREPARE`/`DEALLOCATE`** (prepared statements на уровне протокола): именованные prepared statements привязаны к сессии. Начиная с PgBouncer 1.21.0, частично решено через `max_prepared_statements` — PgBouncer отслеживает prepared statements и воспроизводит их на лету на backend.
- **Временные таблицы** с `PRESERVE ROWS`: данные хранятся в backend-процессе.

Итог: transaction pooling совместим с приложениями, которые не используют session-level состояние между транзакциями. Правильно написанные stateless-приложения (Spring с аннотацией `@Transactional`, connection pool в режиме auto-commit) работают корректно. Приложения, использующие `SET search_path`, advisory locks или LISTEN, требуют либо session pooling, либо специальных адаптаций.
