### Вопрос 21. В чём разница между SERIAL и GENERATED AS IDENTITY? Почему IDENTITY считается предпочтительным подходом начиная с PostgreSQL 10?

**Категория:** PostgreSQL · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 5.3. Identity Columns](https://www.postgresql.org/docs/current/ddl-identity-columns.html)

Столбец-идентификатор (identity column) — это специальный столбец, значения которого автоматически генерируются из неявной последовательности. Создаётся с помощью предложения `GENERATED ... AS IDENTITY`:

```sql
CREATE TABLE people (
    id bigint GENERATED ALWAYS AS IDENTITY,
    name text
);
```

Ключевое слово `ALWAYS` означает, что явно указанное пользователем значение принимается только при наличии `OVERRIDING SYSTEM VALUE` в операторе `INSERT`. Ключевое слово `BY DEFAULT` означает, что пользовательское значение имеет приоритет над сгенерированным.

Столбец автоматически получает ограничение `NOT NULL`. Уникальность при этом не гарантируется автоматически — при необходимости нужно добавить `PRIMARY KEY` или `UNIQUE`.

`SERIAL` — более старый PostgreSQL-специфичный механизм. За кулисами PostgreSQL создаёт последовательность и устанавливает `DEFAULT nextval('sequence_name')` для столбца. `GENERATED AS IDENTITY` — стандартный SQL-механизм (ISO/IEC 9075), введённый в PostgreSQL 10.

#### Ответ от Claude

**Как работает SERIAL (до PostgreSQL 10)**

`SERIAL` — это синтаксический сахар PostgreSQL, не входящий в стандарт SQL. При создании столбца `SERIAL` PostgreSQL автоматически:
1. Создаёт последовательность `<table>_<column>_seq`.
2. Устанавливает `DEFAULT nextval(...)` для столбца.
3. Устанавливает владение последовательностью через `OWNED BY`.

```sql
-- Это:
CREATE TABLE orders (id SERIAL PRIMARY KEY);

-- Эквивалентно этому:
CREATE SEQUENCE orders_id_seq;
CREATE TABLE orders (
    id integer NOT NULL DEFAULT nextval('orders_id_seq')
);
ALTER SEQUENCE orders_id_seq OWNED BY orders.id;
```

Проблемы `SERIAL`:
- Последовательность создаётся как отдельный объект, и её права нужно выдавать отдельно.
- Нет возможности явно запретить ручную вставку значений — любое число принимается без предупреждения.
- Поведение при `ALTER TABLE` (добавление/удаление столбца) менее предсказуемо.
- Не является стандартом SQL — не переносится на другие СУБД.

**Как работает GENERATED AS IDENTITY (PostgreSQL 10+)**

`GENERATED AS IDENTITY` — реализация стандарта SQL:2003. Последовательность создаётся неявно и тесно привязана к столбцу.

```sql
-- ALWAYS: запрещает ручную вставку без явного указания
CREATE TABLE orders (
    id bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY
);

-- BY DEFAULT: ручная вставка разрешена (поведение ближе к SERIAL)
CREATE TABLE orders (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY
);
```

При попытке вставить явное значение в `ALWAYS`-столбец:
```sql
INSERT INTO orders (id) VALUES (10);
-- ERROR: cannot insert into column "id"
-- HINT: Column "id" is an identity column defined as GENERATED ALWAYS.

-- Корректный обход:
INSERT INTO orders (id) OVERRIDING SYSTEM VALUE VALUES (10);
```

**Ключевые отличия**

| Аспект | SERIAL | GENERATED AS IDENTITY |
|---|---|---|
| Стандарт SQL | Нет (PostgreSQL-специфично) | Да (SQL:2003) |
| Защита от случайной вставки | Нет | Да (ALWAYS) |
| Управление через ALTER TABLE | Ограничено | Полное (`SET GENERATED`, `RESTART`) |
| Права на последовательность | Отдельно от таблицы | Вместе с таблицей |
| Типы | SMALLSERIAL, SERIAL, BIGSERIAL | Любой тип, поддерживаемый sequence |

**Рекомендация.** Официальная документация PostgreSQL рекомендует использовать `GENERATED AS IDENTITY` вместо `SERIAL` во всех новых приложениях начиная с PostgreSQL 10. `SERIAL` сохранён для обратной совместимости и не является устаревшим формально, однако новый синтаксис предпочтительнее.

### Вопрос 22. Как работает декларативное партиционирование в PostgreSQL (RANGE, LIST, HASH)? Что такое partition pruning и как он влияет на производительность запросов?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 5.12. Table Partitioning](https://www.postgresql.org/docs/current/ddl-partitioning.html)

Декларативное партиционирование — механизм разбиения одной логической таблицы на несколько физических частей (секций). Секционированная таблица не хранит данные сама по себе — она является «виртуальной» родительской таблицей. Реальные данные хранятся в дочерних таблицах-секциях.

**RANGE:** каждая секция хранит строки, у которых ключ партиционирования попадает в заданный диапазон. Нижняя граница включается, верхняя — исключается.

```sql
CREATE TABLE measurement (
    city_id  int not null,
    logdate  date not null,
    peaktemp int
) PARTITION BY RANGE (logdate);

CREATE TABLE measurement_y2008m01 PARTITION OF measurement
    FOR VALUES FROM ('2008-01-01') TO ('2008-02-01');
```

**LIST:** явно перечисляются значения ключа для каждой секции.

```sql
CREATE TABLE sales (region text) PARTITION BY LIST (region);

CREATE TABLE sales_us PARTITION OF sales
    FOR VALUES IN ('CA', 'NY', 'TX');
```

**HASH:** данные распределяются по секциям на основе хеш-функции с заданным модулем и остатком.

```sql
CREATE TABLE orders (order_id int) PARTITION BY HASH (order_id);

CREATE TABLE orders_p0 PARTITION OF orders
    FOR VALUES WITH (MODULUS 4, REMAINDER 0);
```

**Partition pruning** — оптимизация планировщика, при которой секции, не способные содержать строки, удовлетворяющие условию `WHERE`, исключаются из плана запроса. По умолчанию включена (`enable_partition_pruning = on`).

```sql
-- Без pruning (off): сканируются все секции
-- С pruning (on):   сканируется только measurement_y2008m01
EXPLAIN SELECT count(*) FROM measurement WHERE logdate >= DATE '2008-01-01';
```

#### Ответ от Claude

**Архитектура декларативного партиционирования**

Декларативное партиционирование введено в PostgreSQL 10 как замена более сложному подходу через наследование таблиц. Секционированная таблица — это логическая сущность без собственного хранилища. Секции — это обычные таблицы, каждая из которых хранит подмножество данных родительской таблицы.

```sql
-- Создание родительской таблицы
CREATE TABLE events (
    event_id   bigint    NOT NULL,
    event_date date      NOT NULL,
    payload    jsonb
) PARTITION BY RANGE (event_date);

-- Создание секций
CREATE TABLE events_2024 PARTITION OF events
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');

CREATE TABLE events_2025 PARTITION OF events
    FOR VALUES FROM ('2025-01-01') TO ('2026-01-01');
```

**RANGE-партиционирование**

Подходит для данных с естественным диапазоном: даты, временные метки, числовые идентификаторы. Границы: нижняя включена, верхняя исключена (`[lower, upper)`). Типичные сценарии: логи, метрики, события — когда нужно быстро удалять старые данные (`DROP TABLE events_2023` вместо медленного `DELETE`).

**LIST-партиционирование**

Подходит для данных с конечным набором дискретных значений: страна, регион, статус, тип. Каждой секции явно назначается набор значений ключа. Поддерживает специальное значение `DEFAULT` для строк, не попавших ни в одну секцию.

```sql
CREATE TABLE orders (country text) PARTITION BY LIST (country);
CREATE TABLE orders_ru PARTITION OF orders FOR VALUES IN ('RU');
CREATE TABLE orders_other PARTITION OF orders DEFAULT;
```

**HASH-партиционирование**

Подходит для равномерного распределения нагрузки без явного критерия диапазона или списка. Полезно для шардирования больших таблиц, когда нет удобного ключа для RANGE/LIST. Планировщик может pruning-ать HASH-секции только при точном совпадении значения ключа.

```sql
CREATE TABLE users (user_id bigint) PARTITION BY HASH (user_id);
CREATE TABLE users_p0 PARTITION OF users FOR VALUES WITH (MODULUS 3, REMAINDER 0);
CREATE TABLE users_p1 PARTITION OF users FOR VALUES WITH (MODULUS 3, REMAINDER 1);
CREATE TABLE users_p2 PARTITION OF users FOR VALUES WITH (MODULUS 3, REMAINDER 2);
```

**Partition pruning**

Pruning — это исключение нерелевантных секций из плана выполнения запроса. Происходит на двух этапах:

1. **Во время планирования (plan-time pruning):** когда значение в `WHERE` известно статически (константа или параметр запроса). Исключённые секции не попадают в план вообще.

2. **Во время выполнения (run-time pruning):** когда значение становится известно только в процессе выполнения (например, результат подзапроса или внутренний параметр цикла вложенных итераций).

```sql
-- Только секция events_2025 будет просканирована
EXPLAIN SELECT * FROM events WHERE event_date = '2025-06-15';

-- Вывод плана:
-- Append
--   -> Seq Scan on events_2025
--        Filter: (event_date = '2025-06-15'::date)
```

**Влияние на производительность**

- Pruning резко сокращает объём сканируемых данных при фильтрации по ключу партиционирования.
- Индексы на секциях меньше по размеру и лучше умещаются в `shared_buffers`.
- Удаление устаревших данных через `ALTER TABLE DETACH PARTITION` + `DROP TABLE` происходит мгновенно и не нагружает `VACUUM` (в отличие от `DELETE`).
- Слишком большое количество секций (тысячи) увеличивает время планирования — рекомендуется не превышать несколько сотен секций на одну таблицу.

**Ограничения**

- `UNIQUE` и `PRIMARY KEY` должны включать все столбцы ключа партиционирования.
- Глобальные индексы (по всей секционированной таблице) не поддерживаются — индексы создаются отдельно на каждой секции.
- Триггеры `BEFORE ROW` не могут перенаправлять строки в другую секцию.

### Вопрос 23. Что такое LATERAL JOIN? Чем он отличается от обычного подзапроса в FROM? В каких ситуациях LATERAL необходим?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 7.2. Table Expressions](https://www.postgresql.org/docs/current/queries-table-expressions.html)

Ключевое слово `LATERAL` может предшествовать подзапросу в предложении `FROM`. Это позволяет подзапросу обращаться к столбцам из предшествующих элементов `FROM`. Без `LATERAL` каждый подзапрос вычисляется независимо и не может ссылаться на другие элементы `FROM`.

```sql
SELECT * FROM foo, LATERAL (SELECT * FROM bar WHERE bar.id = foo.bar_id) ss;
```

Для каждой строки из `foo` подзапрос вычисляется заново с использованием текущего значения `foo.bar_id`. Полученные строки объединяются с исходной строкой из `foo` обычным способом.

`LATERAL` чаще всего полезен при использовании функций, возвращающих множество строк (set-returning functions): функция может принять значение из текущей строки в качестве аргумента.

```sql
SELECT p1.id, p2.id, v1, v2
FROM polygons p1, polygons p2,
     LATERAL vertices(p1.poly) v1,
     LATERAL vertices(p2.poly) v2
WHERE (v1 <-> v2) < 10 AND p1.id != p2.id;
```

Без `LATERAL` здесь невозможно передать `p1.poly` как аргумент функции `vertices()`.

Также применяется с `LEFT JOIN ... ON true` для получения строк из функции, которые могут отсутствовать:

```sql
SELECT m.name
FROM manufacturers m
LEFT JOIN LATERAL get_product_names(m.id) pname ON true
WHERE pname IS NULL;
```

Это возвращает производителей, у которых нет ни одного продукта.

#### Ответ от Claude

**Что такое LATERAL JOIN**

`LATERAL` — ключевое слово, которое разрешает подзапросу или табличной функции в предложении `FROM` обращаться к столбцам таблиц, перечисленных левее в том же `FROM`. Без `LATERAL` подзапрос замкнут сам в себе и не видит контекст внешнего запроса.

Поддерживается с PostgreSQL 9.3.

**Семантика вычисления**

Обычный подзапрос в `FROM` (derived table) вычисляется ровно один раз для всего запроса:

```sql
-- Обычный подзапрос: bar сканируется один раз
SELECT f.*, s.*
FROM foo f
JOIN (SELECT * FROM bar WHERE status = 'active') s ON s.foo_id = f.id;
```

`LATERAL`-подзапрос вычисляется отдельно для каждой строки из предшествующей таблицы — фактически это коррелированный подзапрос, но в контексте `FROM` с возможностью возвращать несколько строк:

```sql
-- LATERAL: для каждой строки foo подзапрос вычисляется заново
SELECT f.*, s.*
FROM foo f
JOIN LATERAL (
    SELECT * FROM bar WHERE bar.foo_id = f.id ORDER BY bar.created_at DESC LIMIT 3
) s ON true;
```

Этот запрос возвращает для каждой записи `foo` три последних записи из `bar` — задача, которую без `LATERAL` нельзя решить одним простым подзапросом в `FROM`.

**Чем отличается от обычного подзапроса**

| Аспект | Обычный подзапрос в FROM | LATERAL-подзапрос |
|---|---|---|
| Доступ к внешним столбцам | Нет | Да (к столбцам, стоящим левее) |
| Количество вычислений | Один раз | По одному разу на каждую строку |
| Может возвращать 0 строк | Да | Да (при LEFT JOIN LATERAL) |
| Может вызывать функцию с внешним аргументом | Нет | Да |

**Когда LATERAL необходим**

1. **Top-N на каждую строку** — выбрать N последних/лучших записей для каждого родительского объекта:

```sql
-- Последние 3 заказа для каждого клиента
SELECT c.name, o.order_date, o.total
FROM customers c
JOIN LATERAL (
    SELECT order_date, total
    FROM orders
    WHERE customer_id = c.id
    ORDER BY order_date DESC
    LIMIT 3
) o ON true;
```

2. **Set-returning функции с аргументами из внешней строки:**

```sql
SELECT u.id, t.tag
FROM users u,
     LATERAL unnest(u.tags_array) AS t(tag);
```

3. **Переиспользование вычисленных значений** — когда нужно один раз вычислить выражение и использовать его в нескольких местах `SELECT`:

```sql
SELECT e.id, calc.age_years, calc.next_anniversary
FROM employees e,
     LATERAL (
         SELECT
             date_part('year', age(e.hire_date))::int AS age_years,
             (e.hire_date + interval '1 year' * (date_part('year', age(e.hire_date))::int + 1))::date AS next_anniversary
     ) calc;
```

4. **Функции, возвращающие таблицу, когда аргументы берутся из другой таблицы:**

```sql
SELECT m.manufacturer_name, p.product_name
FROM manufacturers m
LEFT JOIN LATERAL get_products(m.id) p ON true;
```

**Отличие от коррелированного подзапроса в SELECT**

Коррелированный подзапрос в `SELECT` всегда возвращает ровно одно значение (скаляр). `LATERAL` в `FROM` может возвращать несколько строк и несколько столбцов, что значительно расширяет его применимость.

**Производительность**

`LATERAL` выполняется как nested loop: для каждой строки внешней таблицы запускается отдельное сканирование. При наличии индекса на соединяющем столбце это эффективно. При большом числе строк во внешней таблице и отсутствии индекса производительность может быть низкой — в таких случаях стоит рассмотреть обычный `JOIN` с оконной функцией (`ROW_NUMBER()`).
### Вопрос 24. Как работает рекурсивный CTE (WITH RECURSIVE)? Опишите структуру: опорный запрос, рекурсивный шаг, условие останова. Для каких задач он применяется?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation — WITH Queries (Common Table Expressions)](https://www.postgresql.org/docs/current/queries-with.html)

Необязательный модификатор `RECURSIVE` меняет `WITH` из синтаксического удобства в нечто принципиально более мощное. При использовании `RECURSIVE` запрос `WITH` может ссылаться на собственный вывод.

**Структура рекурсивного CTE:**

```sql
WITH RECURSIVE t(n) AS (
    VALUES (1)                        -- опорный запрос (anchor term)
    UNION ALL
    SELECT n+1 FROM t WHERE n < 100  -- рекурсивный шаг
)
SELECT sum(n) FROM t;
```

- **Опорный запрос (anchor term)** — неrekursивная часть, выполняется один раз и даёт начальный набор строк, помещаемых в рабочую таблицу (working table).
- **Рекурсивный шаг** — часть после `UNION ALL` (`UNION`), ссылается на имя CTE; выполняется итеративно, на каждой итерации читая рабочую таблицу.
- **Условие останова** — явное условие `WHERE` в рекурсивной части (`WHERE n < 100`), которое приводит к пустому результату и прекращает итерации.

**Алгоритм выполнения (итеративный, несмотря на название «рекурсивный»):**
1. Вычислить опорный запрос; поместить результат в рабочую таблицу.
2. Пока рабочая таблица не пуста:
   - Применить рекурсивный шаг к рабочей таблице, получить промежуточную таблицу.
   - При `UNION` — исключить дубликаты; при `UNION ALL` — оставить все строки.
   - Заменить рабочую таблицу промежуточной; добавить строки в итоговый результат.

**Типичные задачи применения:**
- Обход иерархических данных (деревья категорий, оргструктуры, BOM-деревья).
- Обход графов с обнаружением циклов (с помощью клаузы `CYCLE` или массива `path`).
- Генерация числовых последовательностей.
- Поиск связных компонент в графе.

PostgreSQL 14+ поддерживает декларативные клаузы `SEARCH DEPTH FIRST BY` / `SEARCH BREADTH FIRST BY` для управления порядком обхода и `CYCLE ... SET ... USING` для автоматического обнаружения циклов.

#### Ответ от Claude

`WITH RECURSIVE` позволяет запросу ссылаться на собственный результат. Механизм итеративный: PostgreSQL не использует системный стек вызовов — вместо этого он поочерёдно заменяет «рабочую таблицу» новыми строками, пока рекурсивная часть не вернёт пустой результат.

**Три обязательных элемента:**

| Элемент | Роль | Пример |
|---|---|---|
| Опорный запрос | Начальные строки; выполняется один раз | `SELECT id, parent_id FROM categories WHERE parent_id IS NULL` |
| Рекурсивный шаг | Читает текущую рабочую таблицу, добавляет новые строки | `SELECT c.id, c.parent_id FROM categories c JOIN tree t ON c.parent_id = t.id` |
| Условие останова | `WHERE`-фильтр или само отсутствие новых строк | `WHERE depth < 10` |

**Пример: обход дерева категорий**

```sql
WITH RECURSIVE tree AS (
    -- опорный запрос
    SELECT id, name, parent_id, 0 AS depth
    FROM categories
    WHERE parent_id IS NULL
    UNION ALL
    -- рекурсивный шаг
    SELECT c.id, c.name, c.parent_id, t.depth + 1
    FROM categories c
    JOIN tree t ON c.parent_id = t.id
    WHERE t.depth < 20  -- защита от бесконечного цикла
)
SELECT * FROM tree ORDER BY depth, name;
```

**Важные нюансы:**
- `UNION ALL` быстрее `UNION`, но не удаляет дубликаты; при обходе графов (не деревьев) возможен бесконечный цикл — нужна явная защита.
- Для обнаружения циклов в PG 14+ используется `CYCLE id SET is_cycle USING path`.
- `LIMIT` в основном запросе прекращает итерации досрочно — полезно при отладке потенциально бесконечных запросов.
- Подзапросы `WITH` в рамках одного оператора видят один и тот же снимок данных (snapshot).
- Нельзя использовать рекурсивные ссылки внутри `GROUP BY`, агрегатных функций, `ORDER BY` и подзапросов рекурсивной части.

**Типичные задачи:** иерархии (категории, меню, оргструктуры), BOM (bill of materials), обход графов, построение путей, генерация числовых рядов.

### Вопрос 25. Начиная с PostgreSQL 12 CTE по умолчанию инлайнятся (inline). Что это значит для производительности? Когда следует явно указывать MATERIALIZED, а когда NOT MATERIALIZED?

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation — WITH Queries: CTE Materialization](https://www.postgresql.org/docs/current/queries-with.html)

До PostgreSQL 12 `WITH`-запросы всегда были «барьерами оптимизации» (optimization fence): планировщик всегда материализовал CTE в промежуточную таблицу и никогда не проталкивал условия из внешнего запроса внутрь CTE. Это ограничивало возможности оптимизатора.

Начиная с **PostgreSQL 12** поведение изменилось: по умолчанию нерекурсивный, без побочных эффектов `SELECT`-CTE, на который есть только **одна ссылка**, автоматически **инлайнится** (встраивается) в родительский запрос — как если бы он был подзапросом. Это позволяет оптимизатору протолкнуть условия фильтрации и использовать индексы таблицы, лежащей внутри CTE.

Если CTE используется **более одного раза**, он по умолчанию **материализуется** — вычисляется один раз и сохраняется во временной структуре, что предотвращает повторное выполнение.

**Явные модификаторы:**

`MATERIALIZED` — принудительно материализовать CTE, даже если на него одна ссылка:
```sql
WITH w AS MATERIALIZED (
    SELECT * FROM big_table
)
SELECT * FROM w WHERE key = 123;
```
Применяется, когда CTE содержит дорогостоящую функцию с побочными эффектами, которую нельзя выполнять более одного раза, или когда нужно гарантировать ровно один проход по данным.

`NOT MATERIALIZED` — принудительно встроить CTE в запрос, даже если на него несколько ссылок:
```sql
WITH w AS NOT MATERIALIZED (
    SELECT * FROM big_table
)
SELECT * FROM w AS w1 JOIN w AS w2 ON w1.key = w2.ref
WHERE w2.key = 123;
```
Применяется, когда фильтры из внешнего запроса могут быть эффективно проброшены в CTE и дадут лучшее использование индексов, даже если CTE при этом будет выполнен несколько раз.

Ключевое ограничение: «оптимизатор не может протолкнуть ограничения из родительского запроса в многократно-используемый `WITH`-запрос, поскольку это могло бы повлиять на все его применения, тогда как должно влиять лишь на одно».

#### Ответ от Claude

**Что изменилось в PostgreSQL 12**

До PG 12 каждый CTE был «оптимизационным барьером»: планировщик всегда материализовал его в промежуточный буфер, а условия из внешнего запроса (`WHERE`, `JOIN ON`) никогда не проталкивались внутрь CTE. Это нередко приводило к полному сканированию большой таблицы внутри CTE, даже когда снаружи стоял узкий фильтр.

С PG 12 нерекурсивный CTE без побочных эффектов (`SELECT` без `VOLATILE`-функций и без `FOR UPDATE`), на который **ровно одна ссылка**, инлайнится по умолчанию. Планировщик встраивает его как подзапрос и может использовать индексы исходных таблиц с учётом внешних условий.

**Правила поведения по умолчанию (PG 12+):**

| Ситуация | Поведение |
|---|---|
| Одна ссылка, нет побочных эффектов, не рекурсивный | Инлайнинг (NOT MATERIALIZED) |
| Более одной ссылки | Материализация (MATERIALIZED) |
| Рекурсивный (`WITH RECURSIVE`) | Всегда материализация |
| Есть побочные эффекты (DML, VOLATILE) | Всегда материализация |

**Когда явно указывать `MATERIALIZED`:**
- CTE содержит дорогостоящее вычисление или VOLATILE-функцию, и её результат должен быть одинаковым при всех использованиях.
- CTE используется один раз, но его инлайнинг приводит к неоптимальному плану (например, планировщик выбирает Seq Scan вместо Index Scan из-за неверной оценки кардинальности после слияния).
- Нужна «опорная точка» для стабилизации плана при тестировании или отладке.

**Когда явно указывать `NOT MATERIALIZED`:**
- CTE используется несколько раз, но каждый раз с разными внешними фильтрами, и обход индекса будет эффективнее, чем один проход по всей таблице + сканирование временного буфера.
- CTE возвращает большой набор строк, но конкретный `JOIN` или `WHERE` реально нужен лишь небольшой части.

**Расхождение между источниками:** официальная документация описывает только поведение PG 12+. В PG 11 и старше явных модификаторов нет — CTE всегда материализовывался. При миграции со старых версий поведение запросов может измениться: план станет лучше или хуже в зависимости от конкретных данных и статистики. Для стабилизации поведения на PG 12+ можно использовать `MATERIALIZED`, чтобы явно воспроизвести старое поведение.

### Вопрос 26. Как читать вывод EXPLAIN (ANALYZE, BUFFERS)? Что означают поля actual time, rows, loops, Buffers: hit/read? По каким признакам в плане определяют узкое место?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation — 14.1. Using EXPLAIN](https://www.postgresql.org/docs/current/using-explain.html)

`EXPLAIN ANALYZE` выполняет запрос и показывает реальные метрики наряду с оценками планировщика. Опция `BUFFERS` добавляет статистику обращений к буферному кешу.

**Пример вывода:**

```
Nested Loop  (cost=4.65..118.50 rows=10 width=488)
             (actual time=0.017..0.051 rows=10 loops=1)
  Buffers: shared hit=36 read=6
  ->  Bitmap Heap Scan on tenk1 t1
        (cost=4.36..39.38 rows=10 width=244)
        (actual time=0.009..0.017 rows=10 loops=1)
        Buffers: shared hit=3 read=5
  ->  Index Scan using tenk2_unique2 on tenk2 t2
        (cost=0.29..7.90 rows=1 width=244)
        (actual time=0.003..0.003 rows=1 loops=10)
        Buffers: shared hit=24 read=1
```

**Поля:**
- `actual time=X..Y` — фактическое время в миллисекундах: X — до первой строки, Y — до последней. Если `loops > 1`, показывается среднее на одну итерацию.
- `rows` — реальное количество строк, выданных узлом (не просканированных, а именно возвращённых).
- `loops` — сколько раз узел выполнялся. Реальное суммарное время узла = `actual time` × `loops`.
- `Buffers: shared hit=N` — блоков найдено в буферном кеше (без обращения к диску).
- `Buffers: shared read=N` — блоков прочитано с диска (или из OS page cache).

Для выявления узких мест ключевые сигналы: большое расхождение оценочных (`rows=`) и реальных (`rows=`) строк указывает на неточную статистику; высокое `shared read` — на I/O-бутылочное горлышко; большое `loops` на дорогостоящем внутреннем узле — на неэффективный Nested Loop; строки «Rows Removed by Filter» — на то, что фильтрация происходит после сканирования, и нужен индекс.

#### Ответ от Claude

**Структура строки плана:**

```
<Узел>  (cost=<startup>..<total> rows=<оценка> width=<байт>)
        (actual time=<старт>..<конец> rows=<факт> loops=<N>)
  Buffers: shared hit=<N> read=<N> [dirtied=<N>] [written=<N>]
```

**Поля `actual time`:**
- Формат `X..Y` в **миллисекундах**.
- `X` — время до возврата **первой** строки (startup cost).
- `Y` — время до возврата **последней** строки (total cost).
- Если `loops > 1`, значение — **среднее за одну итерацию**; реальное суммарное время = `Y × loops`.

**Поле `rows`:**
- Фактическое количество строк, **выданных** узлом.
- Сравнение с оценочным `rows=` в скобках cost — главный инструмент диагностики ошибок планировщика.

**Поле `loops`:**
- Сколько раз узел был выполнен.
- Критично для внутренних узлов Nested Loop: внутренний узел выполняется столько раз, сколько строк выдал внешний. Если `actual time=0.003 loops=10 000`, реальные затраты — 30 мс, а не 0.003 мс.

**Поля `Buffers`:**

| Поле | Что означает |
|---|---|
| `shared hit=N` | N блоков найдено в буферном кеше PostgreSQL (shared_buffers) — без I/O |
| `shared read=N` | N блоков прочитано с диска (или из OS page cache) |
| `shared dirtied=N` | N чистых блоков были изменены в ходе запроса |
| `shared written=N` | N «грязных» блоков вытеснено из кеша на диск в процессе запроса |
| `temp read/written` | Блоки временных файлов (Sort, Hash при spill to disk) |

Высокий `shared read` при низком `shared hit` — признак I/O-бутылочного горлышка; помогает определить, какой узел плана «тяжёлый» с точки зрения дискового ввода-вывода.

**По каким признакам определяют узкое место:**

1. **Расхождение оценочных и фактических строк** (`rows=100` vs `rows=70 000`): оптимизатор ошибся в оценке кардинальности — нужно выполнить `ANALYZE` на таблице или настроить `default_statistics_target`.

2. **Большой `shared read`** при малом `shared hit`: узел читает много блоков с диска. Решение: добавить индекс, увеличить `shared_buffers`, пересмотреть запрос.

3. **`loops` × `actual time` даёт большое число** на внутреннем узле: Nested Loop неэффективен при большом числе итераций — рассмотреть Hash Join или Merge Join.

4. **`Rows Removed by Filter: N`**: значительная часть строк отсеивается фильтром после сканирования — нужен индекс на колонку фильтрации.

5. **Sort Method: external merge Disk**: Sort вышел за пределы `work_mem` и использует временные файлы — увеличить `work_mem` для сессии.

6. **Большой разрыв `actual time` у одного узла** по сравнению с суммой дочерних: узел сам по себе дорог (например, Hash Build с большим числом строк).

**Практический приём:** смотреть сверху вниз (от корня к листьям), найти узел с максимальным `actual time × loops`, затем проверить там же `Buffers: read` и расхождение rows — именно это место требует внимания.

**Важно:** `EXPLAIN ANALYZE` реально **выполняет** запрос. Для `INSERT`/`UPDATE`/`DELETE` оборачивайте в транзакцию с `ROLLBACK`:
```sql
BEGIN;
EXPLAIN (ANALYZE, BUFFERS) DELETE FROM orders WHERE status = 'cancelled';
ROLLBACK;
```
### Вопрос 27. Что такое Index-Only Scan и при каком условии PostgreSQL может его использовать? Какую роль играет visibility map в этой оптимизации?

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 11.9. Index-Only Scans and Covering Indexes](https://www.postgresql.org/docs/current/indexes-index-only-scans.html)

Index-Only Scan — это метод выполнения запроса, при котором PostgreSQL возвращает все необходимые данные непосредственно из индекса, не обращаясь к основному хранилищу таблицы (heap). При обычном Index Scan для каждой найденной в индексе строки PostgreSQL дополнительно читает соответствующую страницу кучи: ссылки из индекса на строки таблицы могут указывать на произвольно разбросанные по диску страницы, что порождает медленные случайные чтения.

Для применения Index-Only Scan необходимо соблюдение двух условий:

1. Тип индекса поддерживает хранение исходных значений столбцов. B-tree-индексы поддерживают всегда; GiST и SP-GiST — для некоторых классов операторов; GIN — не поддерживает.
2. Запрос обращается только к столбцам, сохранённым в индексе.

Пример запроса, который может использовать Index-Only Scan при наличии индекса на `(x, y)`:

```sql
SELECT x, y FROM tab WHERE x = 'key';
SELECT x FROM tab WHERE x = 'key' AND y < 42;
```

А вот этот запрос не сможет — столбец `z` отсутствует в индексе:

```sql
SELECT x, z FROM tab WHERE x = 'key';
```

Роль visibility map. PostgreSQL использует MVCC, и информация о видимости версии строки хранится только в heap-записях, но не в индексных записях. Чтобы избежать обращения к куче ради проверки видимости, PostgreSQL ведёт для каждой таблицы visibility map — битовую карту, в которой один бит соответствует одной странице кучи. Бит выставляется тогда, когда все строки на странице достаточно «старые», чтобы быть видимыми для всех текущих и будущих транзакций (страница помечена all-visible).

Алгоритм Index-Only Scan:

1. По кандидатной индексной записи определить номер страницы кучи.
2. Проверить бит visibility map для этой страницы.
   - Бит установлен — строка гарантированно видима, данные возвращаются прямо из индекса.
   - Бит не установлен — необходимо сходить в кучу для проверки видимости; преимущества Index-Only Scan для этой строки теряются.

Visibility map в четыре порядка меньше самой кучи и, как правило, целиком помещается в shared_buffers. Поэтому даже при частичной необходимости обращения к ней накладные расходы значительно ниже, чем при полноценном Index Scan. Index-Only Scan особенно эффективен для таблиц с редко изменяемыми данными, где большинство страниц помечены all-visible.

#### Ответ от Claude

**Index-Only Scan** — стратегия выполнения запроса, при которой планировщик PostgreSQL удовлетворяет запрос данными из индекса без чтения страниц heap. Это позволяет избежать случайных I/O-обращений к основному хранилищу таблицы.

**Необходимые условия (оба должны выполняться одновременно):**

1. Тип индекса физически хранит оригинальные значения столбцов:
   - B-tree — поддерживает всегда.
   - GiST и SP-GiST — только для определённых классов операторов.
   - GIN — не поддерживает (хранит лишь частичные данные — лексемы, элементы массива и т.п.).
   - Hash, BRIN — не поддерживают.

2. Запрос обращается исключительно к столбцам, присутствующим в индексе (как ключевым, так и включённым через `INCLUDE`).

**Роль visibility map:**

PostgreSQL реализует MVCC: видимость конкретной версии строки (tuple) определяется полями `xmin`/`xmax`, которые хранятся только в heap-странице, но не дублируются в индексе. Это создаёт проблему: индекс не может самостоятельно подтвердить, что найденная строка видима текущей транзакции.

Решение — visibility map (VM). Для каждой таблицы PostgreSQL ведёт побитовую карту: бит страницы установлен (all-visible), если все версии строк на этой странице гарантированно видимы всем транзакциям. VM обновляется процессом VACUUM.

Логика работы при Index-Only Scan:
- Бит VM для страницы **установлен** → строка точно видима, данные берутся из индекса (heap не читается).
- Бит VM **не установлен** → PostgreSQL вынужден обратиться к heap для проверки видимости — для таких строк выигрыш от Index-Only Scan пропадает, хотя сам план остаётся Index-Only Scan.

VM по объёму примерно в 10 000 раз меньше heap-таблицы и почти всегда остаётся в кэше. Поэтому даже при необходимости проверки VM для части строк суммарный I/O существенно ниже, чем при полноценном Index Scan.

**Практический вывод:** Index-Only Scan наиболее эффективен для таблиц с невысокой частотой обновления (INSERT-only или append-heavy таблицы, справочники), где VACUUM успевает поддерживать высокий процент all-visible страниц. Для горячих таблиц с активным UPDATE/DELETE значительная часть страниц будет не помечена all-visible, и реальный выигрыш снизится.

### Вопрос 28. Чем GIN-индекс отличается от GiST для полнотекстового поиска по tsvector? Когда следует предпочесть один над другим?

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 12.9. Preferred Index Types for Text Search](https://www.postgresql.org/docs/current/textsearch-indexes.html)

Для полнотекстового поиска в PostgreSQL доступны два типа индексов: GIN и GiST.

**GIN (Generalized Inverted Index)** — предпочтительный тип индекса для полнотекстового поиска. Как инвертированный индекс, GIN содержит запись для каждой лексемы (слова), включая сжатый список позиций совпадений. При многословных запросах GIN находит первое совпадение, а затем использует индекс для отсева строк, в которых не встречаются остальные слова. GIN хранит только лексемы из значений `tsvector`, без меток весов — поэтому при запросах с весами необходима перепроверка по строке таблицы. Время построения GIN-индекса зависит от `maintenance_work_mem`.

**GiST (Generalized Search Tree)** — lossy-индекс: может выдавать ложные совпадения (false matches), которые PostgreSQL автоматически отфильтровывает через проверку реальной строки таблицы. Каждый документ представлен сигнатурой фиксированной длины (по умолчанию 124 байта, максимум 2024 байта). Чем длиннее сигнатура, тем меньше ложных совпадений и точнее поиск, но больше размер индекса. Время построения GiST нечувствительно к `maintenance_work_mem`.

Ложные совпадения вызывают деградацию производительности из-за лишних обращений к строкам таблицы, а случайный доступ к таблице медленен — это ограничивает практическую полезность GiST-индексов для полнотекстового поиска.

**Вывод из документации:** GIN-индексы предпочтительны для большинства сценариев полнотекстового поиска.

#### Ответ от Claude

**Структурное различие:**

| Характеристика | GIN | GiST |
|---|---|---|
| Структура | Инвертированный индекс (лексема → список документов) | Дерево поиска с сигнатурами документов |
| Точность | Точный (non-lossy) | Lossy — возможны false positives |
| Перепроверка heap | Только при запросах с весами | Всегда (для отсева ложных совпадений) |
| Размер | Больше | Меньше (настраивается через `siglen`) |
| Скорость запросов | Выше (нет лишних heap-чтений) | Ниже (лишние чтения для проверки совпадений) |
| Скорость построения | Медленнее; зависит от `maintenance_work_mem` | Быстрее; не зависит от `maintenance_work_mem` |
| Скорость обновления | Медленнее | Быстрее |
| Поддержка `INCLUDE` | Нет | Да (с PostgreSQL 14) |
| Поддержка `tsquery` | Только `tsvector` | `tsvector` и `tsquery` |

**Как работает GIN для `tsvector`:**
GIN строит инвертированный указатель: для каждой лексемы хранится сжатый список TID (tuple ID) документов, в которых она встречается. При поиске по нескольким словам PostgreSQL находит список для первого слова, затем пересекает со списками остальных слов — без обращения к heap. Это даёт высокую скорость для больших текстовых коллекций.

**Как работает GiST для `tsvector`:**
Для каждого документа вычисляется битовая сигнатура: каждое слово хэшируется в один или несколько битов, все биты объединяются через OR. При поиске PostgreSQL проверяет, содержит ли сигнатура биты запрошенных слов. Если нет — документ точно не подходит. Если да — возможно ложное совпадение, документ проверяется по таблице. Настройка `siglen` позволяет управлять соотношением размер/точность.

**Когда выбирать GIN:**
- Данные преимущественно статичны или редко обновляются.
- Важна максимальная скорость поиска.
- Объём коллекции большой (сотни тысяч документов и более).
- Нет жёстких ограничений на размер индекса.
- Это стандартный выбор для большинства задач.

**Когда выбирать GiST:**
- Данные часто обновляются (INSERT/UPDATE/DELETE в таблице высокочастотные).
- Размер индекса критичен и нужно его минимизировать.
- Нужен покрывающий индекс с `INCLUDE` на tsvector-столбце.
- Требуется индексировать `tsquery` (GIN не поддерживает).
- Таблица небольшая и накладные расходы на false positive незначительны.

**Расхождение между источниками:** Официальная документация PostgreSQL утверждает, что GIN в ~3 раза быстрее при поиске, но в ~3 раза медленнее при построении. Это соотношение является ориентировочным — реальные цифры зависят от размера коллекции, числа уникальных лексем и аппаратной платформы.

### Вопрос 29. Как правильно настроить shared_buffers, work_mem, maintenance_work_mem и effective_cache_size? Почему нельзя просто выделить под shared_buffers 80% RAM и что произойдёт при слишком высоком work_mem при большом числе параллельных соединений?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — Resource Consumption (19.4)](https://www.postgresql.org/docs/current/runtime-config-resource.html)

**shared_buffers** — объём разделяемой памяти сервера для кэширования страниц данных. Умолчание — 128 МБ. На выделенном сервере с объёмом RAM от 1 ГБ рекомендуется устанавливать 25% RAM. Значения свыше 40% RAM дают убывающую отдачу, поскольку PostgreSQL всё равно опирается на кэш операционной системы. Требуется перезапуск сервера.

**work_mem** — максимальная память для одной операции сортировки или хеш-таблицы до сброса данных во временный файл. Умолчание — 4 МБ. Критически важно: параметр действует *на каждую операцию*, а не на весь запрос и не на соединение в целом. Сложный запрос с несколькими узлами сортировки и хеш-джойнами может занять `work_mem * N_операций`. При `hash_mem_multiplier` (умолчание 2.0) хеш-операции получают до `work_mem × 2`. Документация предупреждает: «суммарная память, используемая сервером, может во много раз превышать значение work_mem».

**maintenance_work_mem** — память для операций обслуживания (`VACUUM`, `CREATE INDEX`, `ALTER TABLE ADD FOREIGN KEY`). Умолчание — 64 МБ. Можно устанавливать значительно выше, чем `work_mem`, так как одновременно выполняется одна такая операция на сессию. Исключение — автовакуум: если запущено `autovacuum_max_workers` воркеров, каждый берёт `maintenance_work_mem` (или `autovacuum_work_mem`, если задан отдельно).

**effective_cache_size** — не аллокация памяти, а подсказка планировщику об оценочном объёме памяти (shared_buffers + кэш ОС), доступного для хранения данных. Влияет на выбор планировщиком index scan против seq scan. Типичное значение — 50–75% RAM.

#### Ответ от Claude

**Почему нельзя выставить shared_buffers = 80% RAM**

PostgreSQL использует двухуровневое кэширование: сначала данные попадают в `shared_buffers`, затем ОС также кэширует блоки в page cache. При `shared_buffers` 80% RAM остаток памяти почти целиком уходит под ОС, стек, соединения и прочие процессы. В результате:
- Page cache ОС сжимается до минимума — сбросы и «промахи» кэша учащаются.
- При контрольных точках (checkpoint) и операциях bgwriter возникает конкуренция за память.
- На Linux возможно OOM-killing процессов PostgreSQL.
- Сам PostgreSQL всё равно обращается к ОС за повторным чтением страниц, которые вытеснены из shrinking page cache.

Официальная рекомендация: **25% RAM** как отправная точка, практический потолок — **40% RAM**.

**Проблема high work_mem при большом числе соединений**

Одно соединение, выполняющее сложный запрос (4 сортировки + 2 hash join), потребляет до `work_mem * 6`. При 200 соединениях и `work_mem = 64 МБ` теоретическое пиковое потребление составит `64 * 6 * 200 = 76 800 МБ` — это катастрофа для 32 ГБ сервера. На практике не все соединения одновременно выполняют тяжёлые запросы, но риск OOM остаётся реальным.

Безопасная формула для оценки `work_mem`:
```
work_mem = (RAM - shared_buffers - OS_reserve) / (max_connections * avg_operations_per_query)
```
Для 32 ГБ RAM, 200 соединений, 4 операций на запрос:
```
(32 ГБ - 8 ГБ - 4 ГБ) / (200 * 4) ≈ 25 МБ
```

**Итоговые рекомендации (сверено по официальной документации и практике)**

| Параметр | Рекомендация | Примечание |
|---|---|---|
| `shared_buffers` | 25% RAM (до 40% на dedicated) | Требует перезапуска |
| `work_mem` | Считать по формуле, обычно 4–64 МБ | Можно поднять локально для конкретных сессий через `SET` |
| `maintenance_work_mem` | 256 МБ – 1 ГБ | Задать `autovacuum_work_mem` отдельно, чтобы ограничить autovacuum |
| `effective_cache_size` | 50–75% RAM | Только подсказка планировщику, не аллокация |

Расхождений между источниками не обнаружено: все авторитетные источники сходятся на 25–40% для `shared_buffers` и предупреждают о мультипликативной природе `work_mem`.

### Вопрос 30. Как работает буферный кэш PostgreSQL (shared_buffers) и что делают фоновые процессы bgwriter и checkpointer? Чем bgwriter отличается от checkpointer, какие параметры управляют частотой сброса «грязных» буферов и как диагностировать «dirty page storm»?

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — Resource Consumption, Background Writer (19.4.5)](https://www.postgresql.org/docs/current/runtime-config-resource.html#RUNTIME-CONFIG-RESOURCE-BACKGROUND-WRITER)

**Background Writer** — отдельный серверный процесс, который *превентивно* записывает «грязные» (изменённые) буферы shared_buffers на диск. Цель — поддерживать запас чистых буферов, чтобы серверные процессы при необходимости не тратили время запроса на вытеснение грязной страницы. Обратная сторона: bgwriter создаёт дополнительную суммарную нагрузку на I/O — повторно изменённая страница может быть записана несколько раз за интервал между контрольными точками.

Параметры bgwriter:
- `bgwriter_delay` (умолчание 200 мс) — пауза между рабочими раундами.
- `bgwriter_lru_maxpages` (умолчание 100) — максимум буферов за один раунд; 0 отключает bgwriter (не влияет на checkpoint).
- `bgwriter_lru_multiplier` (умолчание 2.0) — коэффициент: ожидаемое число нужных буферов × multiplier = буферов к записи. При 1.0 — «точно в срок», при >1.0 — с запасом.
- `bgwriter_flush_after` (умолчание 512 КБ на Linux) — заставляет ОС сбросить накопленные записи на физический носитель, ограничивая «грязный» page cache и снижая задержку fsync при checkpoint.

**Checkpointer** — отдельный процесс (выделен из bgwriter начиная с PostgreSQL 9.2), отвечающий исключительно за контрольные точки: сброс всех грязных буферов + fsync + запись checkpoint record в WAL. Параметры checkpointer не зависят от параметров bgwriter:
- `checkpoint_timeout` (умолчание 5 мин) — максимальный интервал между автоматическими checkpoint.
- `checkpoint_completion_target` (умолчание 0.9) — доля интервала, за которую должна завершиться запись буферов (равномерный I/O).
- `max_wal_size` (умолчание 1 ГБ) — мягкий порог роста WAL, при превышении инициируется внеплановый checkpoint.
- `checkpoint_flush_after` (умолчание 256 КБ) — аналог `bgwriter_flush_after`, но для checkpoint.

#### Ответ от Claude

**Как работает буферный кэш shared_buffers**

Все блоки данных, прочитанные или изменённые PostgreSQL, попадают в `shared_buffers` — разделяемую область памяти, доступную всем backend-процессам. Страница считается «грязной» после модификации (INSERT/UPDATE/DELETE). Перед тем как страница может быть вытеснена из кэша, она должна быть записана на диск.

Существуют три пути сброса грязной страницы:
1. **bgwriter** — превентивная запись в фоне, освобождает буферы до того, как они потребуются.
2. **checkpointer** — массовый сброс всех грязных буферов при контрольной точке с финальным fsync.
3. **Backend-процесс** — если буферов не хватает и bgwriter не успел, серверный процесс читателя вынужден сам вытеснить и записать грязную страницу прямо во время обработки запроса (наихудший сценарий для latency).

**Разница bgwriter vs checkpointer**

| Аспект | bgwriter | checkpointer |
|---|---|---|
| Цель | Освободить чистые буферы для новых запросов | Создать точку восстановления, сбросить все грязные буферы |
| Когда срабатывает | Непрерывно, по `bgwriter_delay` | По таймеру (`checkpoint_timeout`) или по объёму WAL (`max_wal_size`) |
| Охват | Выборочно — самые «давно не используемые» буферы (LRU) | Все грязные буферы без исключения |
| fsync | Нет (полагается на ОС / `bgwriter_flush_after`) | Да, обязательный fsync всех файлов |
| Параметры | `bgwriter_*` | `checkpoint_*`, `max_wal_size` |

**Dirty page storm (шторм грязных страниц)**

«Шторм» возникает, когда к моменту checkpoint накапливается огромное число грязных буферов и checkpointer вынужден сбросить их за короткий период, создавая пиковую нагрузку на I/O. Типичные причины:
- `shared_buffers` слишком мал — страницы не успевают «отстояться», bgwriter постоянно вытесняет их и перезаписывает.
- `checkpoint_completion_target` близок к 0 — checkpoint завершается слишком быстро, не размазывая I/O.
- Счётчики / «горячие» таблицы с высокой частотой обновлений одних и тех же страниц.

Парадокс: если большинство грязных страниц сбрасывает **checkpointer**, а не bgwriter — это нормально и эффективно. Частые сбросы bgwriter означают, что он перезаписывает одни и те же страницы несколько раз за checkpoint interval, увеличивая суммарный I/O без необходимости.

**Диагностика**

До PostgreSQL 17 все метрики объединял `pg_stat_bgwriter`. С PostgreSQL 17 они разделены:

```sql
-- PostgreSQL 17+: статистика checkpointer
SELECT * FROM pg_stat_checkpointer;
-- Ключевые поля: num_timed (по таймеру), num_requested (по объёму WAL),
-- write_time, sync_time, buffers_written

-- PostgreSQL 17+: статистика bgwriter
SELECT * FROM pg_stat_bgwriter;
-- Ключевые поля: buffers_clean (записано bgwriter),
-- maxwritten_clean (раунды, когда bgwriter достиг bgwriter_lru_maxpages)

-- До PostgreSQL 17 — всё в одном представлении:
SELECT * FROM pg_stat_bgwriter;
-- buffers_checkpoint — записано checkpointer
-- buffers_clean     — записано bgwriter
-- buffers_backend   — записано backend-процессами (тревожный сигнал!)
-- maxwritten_clean  — bgwriter исчерпал лимит за раунд
```

Тревожные сигналы:
- `buffers_backend` > 0 и растёт — backend-процессы вынуждены сами писать грязные страницы; нужно увеличить `shared_buffers` или `bgwriter_lru_maxpages`.
- `maxwritten_clean` высокое — bgwriter стабильно достигает лимита `bgwriter_lru_maxpages`; нужно поднять этот параметр или `bgwriter_lru_multiplier`.
- checkpoint с `num_requested` >> `num_timed` — checkpoint инициируется ростом WAL раньше таймаута; нужно увеличить `max_wal_size` или `checkpoint_timeout`.

**Рекомендации по настройке**

```
checkpoint_completion_target = 0.9   -- размазать I/O checkpoint на 90% интервала
checkpoint_timeout = 10min            -- увеличить интервал на нагруженных системах
max_wal_size = 4GB                    -- уменьшить частоту внеплановых checkpoint
bgwriter_lru_maxpages = 200           -- увеличить, если maxwritten_clean высокое
bgwriter_flush_after = 512kB          -- ограничить dirty page cache на Linux
```

### Вопрос 31. Как с помощью pg_stat_activity, pg_stat_statements и pg_locks диагностировать медленные запросы и блокировки на production-базе? Напиши SQL-запрос, который покажет все активные запросы дольше 5 секунд вместе с их wait_event и удерживаемыми блокировками.

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Lock Monitoring — PostgreSQL Wiki](https://wiki.postgresql.org/wiki/Lock_Monitoring)

Базовый способ диагностики блокировок — объединить `pg_locks` и `pg_stat_activity`. Запрос на wiki.postgresql.org, показывающий заблокированные и блокирующие процессы:

```sql
SELECT blocked_locks.pid       AS blocked_pid,
       blocked_activity.usename  AS blocked_user,
       blocking_locks.pid       AS blocking_pid,
       blocking_activity.usename AS blocking_user,
       blocked_activity.query    AS blocked_statement,
       blocking_activity.query   AS current_statement_in_blocking_process
FROM  pg_catalog.pg_locks         blocked_locks
JOIN  pg_catalog.pg_stat_activity blocked_activity
    ON blocked_activity.pid = blocked_locks.pid
JOIN  pg_catalog.pg_locks         blocking_locks
    ON  blocking_locks.locktype = blocked_locks.locktype
    AND blocking_locks.database IS NOT DISTINCT FROM blocked_locks.database
    AND blocking_locks.relation IS NOT DISTINCT FROM blocked_locks.relation
    AND blocking_locks.page     IS NOT DISTINCT FROM blocked_locks.page
    AND blocking_locks.tuple    IS NOT DISTINCT FROM blocked_locks.tuple
    AND blocking_locks.virtualxid IS NOT DISTINCT FROM blocked_locks.virtualxid
    AND blocking_locks.transactionid IS NOT DISTINCT FROM blocked_locks.transactionid
    AND blocking_locks.classid  IS NOT DISTINCT FROM blocked_locks.classid
    AND blocking_locks.objid    IS NOT DISTINCT FROM blocked_locks.objid
    AND blocking_locks.objsubid IS NOT DISTINCT FROM blocked_locks.objsubid
    AND blocking_locks.pid != blocked_locks.pid
JOIN  pg_catalog.pg_stat_activity blocking_activity
    ON blocking_activity.pid = blocking_locks.pid
WHERE NOT blocked_locks.granted;
```

Также рекомендуется современная альтернатива через функцию `pg_blocking_pids()`, которая правильно учитывает конфликты режимов блокировок и порядок очереди ожидания.

#### Ответ от Claude

**Три инструмента диагностики**

**pg_stat_activity** — представление реального времени: один ряд на каждый backend-процесс. Ключевые поля для диагностики:
- `pid` — идентификатор процесса
- `state` — `active` | `idle` | `idle in transaction` | `idle in transaction (aborted)`
- `query` — текст текущего или последнего запроса (усекается до `track_activity_query_size`, умолчание 1024 байт)
- `query_start` — момент старта текущего запроса
- `wait_event_type` / `wait_event` — тип и название события ожидания
- `xact_start` — момент начала текущей транзакции

**pg_stat_statements** — агрегированная статистика по нормализованным запросам (расширение, подключается через `shared_preload_libraries`). Ключевые поля:
- `query` — нормализованный текст (константы заменены на `$1`, `$2`, …)
- `calls` — число вызовов
- `total_exec_time` / `mean_exec_time` — суммарное и среднее время выполнения (мс)
- `shared_blks_hit` / `shared_blks_read` — попадания и промахи кэша

**pg_locks** — все активные тяжёлые блокировки (relation, tuple, transactionid и др.). Поле `granted = false` означает, что процесс ожидает получения блокировки.

**Подключение pg_stat_statements**

```sql
-- postgresql.conf:
-- shared_preload_libraries = 'pg_stat_statements'
-- pg_stat_statements.track = all

-- В целевой базе:
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;
```

**Запрос: активные запросы дольше 5 секунд с wait_event и блокировками**

```sql
SELECT
    a.pid,
    a.usename,
    a.datname,
    a.state,
    a.wait_event_type,
    a.wait_event,
    now() - a.query_start                      AS duration,
    a.xact_start,
    -- блокировки, которые удерживает этот процесс
    (
        SELECT string_agg(
            l.locktype || '/' || coalesce(l.relation::regclass::text, l.transactionid::text, '?')
            || '[' || l.mode || ']',
            ', '
        )
        FROM pg_locks l
        WHERE l.pid = a.pid
          AND l.granted = true
    )                                           AS held_locks,
    -- PIDs, которые блокирует этот процесс
    pg_blocking_pids(a.pid)                     AS blocked_by,
    left(a.query, 200)                          AS query_snippet
FROM pg_stat_activity a
WHERE a.state = 'active'
  AND a.query_start < now() - interval '5 seconds'
  AND a.pid <> pg_backend_pid()
ORDER BY duration DESC;
```

**Запрос: дерево блокировок (кто кого блокирует)**

```sql
SELECT
    blocked.pid                         AS blocked_pid,
    blocked.usename                     AS blocked_user,
    blocked.wait_event_type,
    blocked.wait_event,
    now() - blocked.query_start         AS waiting_duration,
    left(blocked.query, 150)            AS blocked_query,
    blocker.pid                         AS blocking_pid,
    blocker.usename                     AS blocking_user,
    blocker.state                       AS blocking_state,
    left(blocker.query, 150)            AS blocking_query
FROM pg_stat_activity blocked
JOIN pg_stat_activity blocker
    ON blocker.pid = ANY(pg_blocking_pids(blocked.pid))
WHERE cardinality(pg_blocking_pids(blocked.pid)) > 0
ORDER BY waiting_duration DESC;
```

**Запрос: топ медленных запросов через pg_stat_statements**

```sql
SELECT
    round(total_exec_time::numeric, 2)  AS total_ms,
    round(mean_exec_time::numeric, 2)   AS mean_ms,
    calls,
    round(
        100.0 * shared_blks_hit /
        nullif(shared_blks_hit + shared_blks_read, 0), 1
    )                                   AS cache_hit_pct,
    left(query, 200)                    AS query_snippet
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 20;
```

**Практические советы**

- Включите `log_lock_waits = on` в `postgresql.conf` — PostgreSQL будет логировать ожидания блокировок дольше `deadlock_timeout` (умолчание 1 с), что помогает ретроспективно.
- `idle in transaction` с большим `xact_start` — незакрытые транзакции, удерживающие блокировки; ищите утечки соединений в приложении.
- `wait_event_type = 'Lock'` при `state = 'active'` означает, что запрос выполняется, но заблокирован тяжёлой блокировкой.
- `wait_event_type = 'IO'` с `wait_event = 'DataFileRead'` — признак I/O-давления; проверьте `shared_buffers` и дисковую подсистему.
- Используйте `pg_blocking_pids()` вместо самосоединения `pg_locks` — функция корректно учитывает конфликты режимов блокировок (рекомендация официальной документации PostgreSQL 18).
### Вопрос 32. Что такое раздувание таблиц и индексов (table bloat / index bloat) в PostgreSQL? Чем VACUUM FULL отличается от обычного VACUUM, что делает CLUSTER и когда стоит использовать каждый из этих инструментов? Какова роль fillfactor в управлении bloat?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: Routine Vacuuming](https://www.postgresql.org/docs/current/routine-vacuuming.html)

PostgreSQL использует MVCC (Multiversion Concurrency Control): операции UPDATE и DELETE не удаляют старые версии строк немедленно — они остаются в таблице до тех пор, пока на них может ссылаться какая-либо активная транзакция. Накопление таких «мёртвых» (dead) кортежей и называется раздуванием таблицы (table bloat). Аналогичный процесс происходит в индексах — index bloat: индексные записи, ссылающиеся на удалённые/обновлённые версии строк, не удаляются сразу.

**Обычный VACUUM** рекламирует пространство мёртвых кортежей и делает его доступным для повторного использования внутри той же таблицы. Он не возвращает пространство операционной системе (за исключением страниц в самом конце файла). VACUUM получает только блокировку `SHARE UPDATE EXCLUSIVE` и выполняется параллельно с обычными операциями чтения и записи.

**VACUUM FULL** полностью перезаписывает таблицу в новый файл без лишнего пространства, возвращая освободившееся место операционной системе. Он требует блокировки `ACCESS EXCLUSIVE` на всё время работы, блокирует все операции с таблицей и требует дополнительного дискового пространства, примерно равного размеру таблицы.

**CLUSTER** физически переупорядочивает строки таблицы на диске по выбранному индексу. Как и VACUUM FULL, он перезаписывает таблицу целиком, захватывает блокировку `ACCESS EXCLUSIVE` и требует дополнительного места на диске. После кластеризации новые/обновлённые строки не сохраняют порядок автоматически — для поддержания порядка нужно периодически запускать CLUSTER повторно.

**fillfactor** — параметр хранения, определяющий, насколько плотно PostgreSQL заполняет страницы таблицы при INSERT. Значение по умолчанию — 100 (полное заполнение), допустимый диапазон — от 10 до 100. При fillfactor < 100 на каждой странице резервируется место для будущих UPDATE, что повышает вероятность размещения обновлённой версии строки на той же странице, что и оригинал (механизм HOT — Heap-Only Tuple). HOT-обновления не требуют обновления индексов, что снижает нагрузку и уменьшает bloat. Оборотная сторона: меньший fillfactor означает большее исходное использование дискового пространства.

#### Ответ от Claude

**Table bloat и index bloat** — следствие MVCC-архитектуры PostgreSQL. При выполнении UPDATE создаётся новая версия строки, а старая помечается как удалённая, но физически остаётся на странице. При DELETE строка также лишь помечается «мёртвой». Накопленные мёртвые кортежи занимают место на страницах и замедляют сканирование таблицы. В индексах аналогично накапливаются записи, ссылающиеся на мёртвые версии.

**Обычный VACUUM** (autovacuum выполняет его автоматически):
- Помечает страницы с мёртвыми кортежами как доступные для повторного использования внутри той же таблицы.
- Не возвращает пространство ОС.
- Работает параллельно с DML-операциями (`SHARE UPDATE EXCLUSIVE`).
- Должен выполняться регулярно; основной инструмент борьбы с bloat в продакшене.

**VACUUM FULL**:
- Полностью переписывает таблицу в новый файл, устраняя все зазоры.
- Возвращает место ОС.
- Блокирует таблицу (`ACCESS EXCLUSIVE`) на всё время работы — недопустимо для систем с высокой доступностью.
- Требует свободного дискового пространства, примерно равного размеру таблицы.
- Применять только в исключительных случаях: таблица деградировала настолько, что регулярный VACUUM уже не справляется с накопленным bloat.

**CLUSTER**:
- Переписывает таблицу, физически упорядочивая строки по указанному индексу.
- Устраняет bloat как побочный эффект, но основная цель — оптимизация range-запросов по индексированному столбцу (связанные строки оказываются рядом на диске).
- Также захватывает `ACCESS EXCLUSIVE` и требует дополнительного места на диске.
- Порядок не сохраняется автоматически после последующих вставок/обновлений.
- Когда использовать: если запросы систематически читают диапазоны по одному столбцу и страдают от большого числа random I/O.

**fillfactor**:
- Управляет плотностью заполнения страниц при INSERT.
- При fillfactor = 70 INSERT заполняет страницу на 70%, оставляя 30% для UPDATE на той же странице.
- Это включает механизм HOT (Heap-Only Tuple): обновлённая строка остаётся на той же странице, индексы не обновляются, bloat снижается.
- Типичные значения для часто обновляемых таблиц: 70–90.
- Для insert-only таблиц оставляют 100 (дополнительное пространство не нужно).
- Для индексов fillfactor тоже поддерживается (по умолчанию 90 для B-Tree); влияет на частоту page-split и, как следствие, index bloat.

**Итоговая рекомендация по выбору инструмента:**
| Ситуация | Инструмент |
|---|---|
| Регулярное обслуживание | autovacuum / VACUUM |
| Критическое разрастание таблицы, готовность к простою | VACUUM FULL |
| Нужно упорядочить данные по индексу + устранить bloat | CLUSTER |
| Профилактика bloat при интенсивных UPDATE | fillfactor < 100 |

### Вопрос 33. Как работает REINDEX CONCURRENTLY в PostgreSQL и чем он отличается от обычного REINDEX? Какие ограничения есть у REINDEX CONCURRENTLY и в каком случае он может завершиться с ошибкой, оставив невалидный индекс?

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: REINDEX](https://www.postgresql.org/docs/current/sql-reindex.html)

**Обычный REINDEX** перестраивает индекс с нуля. Во время работы он захватывает блокировку `ACCESS EXCLUSIVE` на конкретный индекс, что блокирует все операции записи в таблицу (INSERT, UPDATE, DELETE) до завершения операции. Читать таблицу при этом можно. Выполняется за один проход по таблице.

**REINDEX CONCURRENTLY** перестраивает индекс без захвата блокировок, препятствующих параллельным INSERT, UPDATE, DELETE на таблице. Вместо этого он выполняется в шесть отдельных транзакций:
1. Добавление нового временного индекса в `pg_index` с захватом `SHARE UPDATE EXCLUSIVE`.
2. Первый проход — построение индекса; флаг `indisready` переключается в `true`.
3. Второй проход — добавление записей, попавших в таблицу в процессе первого прохода.
4. Переключение имён и ссылок; флаг `indisvalid` у нового индекса становится `true`, у старого — `false`.
5. Ожидание завершения транзакций, ссылающихся на старый индекс, затем переключение `indisready` в `false` для старого.
6. Удаление старых индексов и снятие блокировок сессии.

**Ограничения REINDEX CONCURRENTLY:**
- Нельзя выполнять внутри блока транзакции.
- Не поддерживается для системных каталогов (`REINDEX SYSTEM ... CONCURRENTLY` недопустимо).
- Нельзя перестраивать индексы ограничений исключения (exclusion constraints).
- На одну таблицу допускается только одна параллельная перестройка индекса одновременно.
- Занимает значительно больше времени, чем обычный REINDEX.

**Сценарий с невалидным индексом.** Если во время REINDEX CONCURRENTLY возникает ошибка (например, нарушение уникальности в уникальном индексе), команда завершается с ошибкой, но оставляет в системных каталогах один или два «невалидных» (`INVALID`) индекса:
- Индекс с суффиксом `_ccnew` — временный индекс, созданный во время операции. Рекомендуемое действие: удалить его (`DROP INDEX`) и повторить `REINDEX CONCURRENTLY`.
- Индекс с суффиксом `_ccold` — оригинальный индекс, который не удалось удалить. Рекомендуемое действие: просто удалить его (`DROP INDEX`), поскольку перестройка уже завершилась успешно.

Невалидные индексы игнорируются планировщиком при выборке, но продолжают потреблять ресурсы при каждой операции DML.

#### Ответ от Claude

**Как работает обычный REINDEX:**
Захватывает `ACCESS EXCLUSIVE` на индекс, выполняет один проход по таблице, строит индекс с нуля. В течение этого времени операции записи в таблицу заблокированы. Допускается внутри транзакции.

**Как работает REINDEX CONCURRENTLY:**
Перестраивает индекс в шесть транзакций, используя только `SHARE UPDATE EXCLUSIVE` (не блокирует DML-операции). Процесс занимает значительно больше времени, так как требует двух полных проходов по таблице и ожидания завершения конкурентных транзакций между этапами.

**Сравнение:**

| Параметр | REINDEX | REINDEX CONCURRENTLY |
|---|---|---|
| Блокировка | ACCESS EXCLUSIVE | SHARE UPDATE EXCLUSIVE |
| Блокирует DML | Да (записи) | Нет |
| Проходов по таблице | 1 | 2 |
| Скорость | Быстрее | Медленнее |
| В транзакции | Можно | Нельзя |
| Системные каталоги | Можно | Нельзя |
| Exclusion constraint | Можно | Нельзя |

**Ограничения REINDEX CONCURRENTLY:**
- Не работает внутри явного блока транзакции (`BEGIN ... COMMIT`).
- Не применимо к системным каталогам (`REINDEX SYSTEM CONCURRENTLY` запрещён).
- Нельзя использовать для индексов ограничений исключения.
- Одновременно на таблице может выполняться только одна конкурентная перестройка.

**Сценарий с невалидным индексом.**
При сбое в ходе REINDEX CONCURRENTLY (нарушение уникальности, нехватка памяти, отмена клиентом и пр.) в `pg_index` могут остаться невалидные (`INVALID`) индексы:

- `idx_ccnew` — строился как новый, не завершён. Действие: `DROP INDEX idx_ccnew;`, затем повторить `REINDEX CONCURRENTLY`.
- `idx_ccold` — оригинальный индекс, успешно заменённый, но не удалённый. Действие: `DROP INDEX idx_ccold;`.

Невалидные индексы не используются планировщиком для запросов, но обновляются при каждой DML-операции — создавая накладные расходы без пользы. Их нужно удалять как можно скорее.

Обнаружить невалидные индексы можно запросом:
```sql
SELECT indexrelid::regclass AS index_name
FROM pg_index
WHERE NOT indisvalid;
```

### Вопрос 34. Как в PostgreSQL кэшируются планы выполнения prepared statements? Что такое generic plan и custom plan, как работает алгоритм выбора между ними (параметр plan_cache_mode), и в каком сценарии generic plan может привести к деградации производительности по сравнению с custom plan?

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: PREPARE](https://www.postgresql.org/docs/current/sql-prepare.html)

При первом выполнении подготовленного запроса (`PREPARE` + `EXECUTE`) PostgreSQL строит план выполнения. Существуют два типа планов:

**Custom plan** — план, построенный с учётом конкретных значений параметров текущего выполнения. Планировщик может использовать статистику распределения данных применительно к фактическим значениям `$1`, `$2` и т.д., что позволяет выбрать оптимальный индекс или метод соединения для данного набора значений. Custom plan строится заново при каждом выполнении.

**Generic plan** — план, построенный без привязки к конкретным значениям параметров. Он может переиспользоваться для любых последующих вызовов, экономя время планирования. Однако, если оптимальный план существенно зависит от значений параметров, generic plan может оказаться неэффективным.

**Алгоритм выбора (режим `auto`):**
- Первые 5 выполнений PostgreSQL строит custom plan.
- На 6-м выполнении вычисляется средняя оценочная стоимость пяти custom планов, строится generic plan и его стоимость сравнивается со средней.
- Если стоимость generic плана не превышает среднюю стоимость custom плана, дальнейшие выполнения используют generic план.
- Иначе каждое выполнение продолжает получать custom план.

**Параметр `plan_cache_mode`** позволяет переопределить автоматический выбор:
- `auto` — поведение по умолчанию (описано выше).
- `force_custom_plan` — всегда строить custom план, игнорируя кеш.
- `force_generic_plan` — всегда использовать generic план.

Параметр вычисляется в момент выполнения плана, а не в момент подготовки запроса.

#### Ответ от Claude

**Механизм кеширования планов prepared statements.**

Когда клиент выполняет `PREPARE stmt AS SELECT ...`, PostgreSQL разбирает и анализирует запрос, но откладывает полное планирование. При первых выполнениях (`EXECUTE stmt(val)`) строятся custom plans с учётом конкретных значений.

**Custom plan:**
- Строится заново при каждом выполнении.
- Планировщик видит фактические значения параметров и может выбрать индекс, учитывая их селективность.
- Бо́льшие накладные расходы на планирование, но потенциально более эффективное выполнение.

**Generic plan:**
- Строится один раз, переиспользуется при всех последующих вызовах.
- Содержит символы `$1`, `$2`, ... вместо конкретных значений.
- Меньше накладных расходов на планирование, но план фиксирован и может быть субоптимальным.

**Алгоритм автоматического выбора (auto):**
1. При выполнениях 1–5 строится custom plan.
2. После 5 выполнений вычисляется средняя оценочная стоимость custom планов.
3. Строится generic plan и его стоимость сравнивается с этим средним значением.
4. Если `стоимость_generic <= средняя_стоимость_custom`, PostgreSQL переключается на generic план для всех последующих выполнений.
5. Если нет — продолжает строить custom план и повторяет сравнение при каждом следующем вызове.

**Параметр plan_cache_mode:**
- `auto` — описанный выше алгоритм (значение по умолчанию).
- `force_custom_plan` — всегда custom, даже если generic был бы дешевле по оценке.
- `force_generic_plan` — всегда generic, пропуская первые 5 итераций custom.

**Сценарий деградации производительности с generic plan.**

Классический случай — сильно скошенное распределение данных. Например, в таблице заказов 95% строк имеют статус `'ACTIVE'`, а 1% — статус `'CANCELLED'`.

1. Первые 5 выполнений запускаются с параметром `'ACTIVE'` — строится custom plan с sequential scan (корректный выбор для 95% таблицы).
2. Generic plan оценивается как дешёвый (sequential scan дёшев по среднему), PostgreSQL переключается на него.
3. Следующее выполнение запускается с параметром `'CANCELLED'` — для 1% строк оптимален index scan, но generic plan фиксирован на sequential scan.
4. Результат: полное сканирование таблицы при запросе, который должен возвращать единичные строки.

Обратная ситуация тоже возможна: первые 5 выполнений с редким значением формируют generic plan с index scan, который затем применяется к мажоритарному значению и читает большую часть таблицы через индекс неэффективнее, чем sequential scan.

**Рекомендации:**
- При подозрении на деградацию выполните `EXPLAIN EXECUTE stmt(val)` — PostgreSQL покажет, какой план используется.
- Если generic plan стабильно неоптимален, установите `SET plan_cache_mode = force_custom_plan` на уровне сессии или конкретного запроса.
- `force_generic_plan` оправдан при очень высокой частоте выполнения однотипных запросов с предсказуемой стоимостью планирования.
- Расхождение между источниками: документация PostgreSQL говорит о первых 5 выполнениях как custom, тогда как vladmihalcea.com упоминает первые 4. Официальная документация (postgresql.org) является приоритетным источником — используется значение 5.
### Вопрос 35. Что такое расширенная статистика (CREATE STATISTICS) в PostgreSQL? Какие виды статистики она собирает (ndistinct, dependencies, mcv) и для каких запросов стандартной поколоночной статистики недостаточно? Как влияет default_statistics_target на качество планов?

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 69.2. Multivariate Statistics Examples](https://www.postgresql.org/docs/current/multivariate-statistics-examples.html)

Стандартная статистика PostgreSQL собирается отдельно для каждого столбца. Когда запрос фильтрует сразу по нескольким столбцам, планировщик по умолчанию умножает их селективности, предполагая, что столбцы независимы. Для коррелированных столбцов это приводит к грубым ошибкам оценки количества строк (cardinality estimation).

**Функциональные зависимости (dependencies)**. Если значение столбца `a` однозначно определяет значение столбца `b`, планировщик без этой статистики умножает их селективности и сильно занижает оценку. Пример:

```sql
CREATE STATISTICS stts (dependencies) ON a, b FROM t;
ANALYZE t;
EXPLAIN SELECT * FROM t WHERE a = 1 AND b = 1;
-- без статистики: rows=1 (факт: 100), с ней: rows=100
```

Ограничение: работает только с условиями равенства; не учитывает отдельные значения, только глобальную корреляцию.

**N-distinct (ndistinct)**. Используется для оценки количества групп в `GROUP BY` по нескольким столбцам. Без этой статистики планировщик перемножает количество уникальных значений каждого столбца и получает завышенную оценку. Пример:

```sql
CREATE STATISTICS stts (ndistinct) ON a, b FROM t;
ANALYZE t;
EXPLAIN SELECT count(*) FROM t GROUP BY a, b;
-- без статистики: rows=1000 (факт: 100), с ней: rows=100
```

**Списки наиболее частых значений (mcv)**. Самый точный вид: хранит реальные комбинации значений с их частотами. Умеет:
- обнаружить, что комбинация `(a=1, b=10)` вообще не встречается в таблице;
- корректно обрабатывать диапазонные условия (`a <= 49 AND b > 49`), с чем `dependencies` не справляется;
- давать верхнюю границу для комбинаций, отсутствующих в списке.

Недостаток: дороже в хранении и вычислении по сравнению с остальными видами.

**Влияние `default_statistics_target`**. Это параметр, контролирующий:
1. Размер выборки строк при `ANALYZE` — чем больше значение, тем репрезентативнее выборка.
2. Максимальное количество записей в массивах `most_common_vals` и `histogram_bounds` (по умолчанию 100).
Расширенная статистика использует ту же выборку строк, что и обычная. Поэтому увеличение `default_statistics_target` (или `ALTER TABLE ... ALTER COLUMN ... SET STATISTICS N`) улучшает качество расширенной статистики — особенно MCV-списков — ценой большего времени `ANALYZE` и объёма в `pg_statistic`.

#### Ответ от Claude

**Зачем нужна расширенная статистика**

Стандартная поколоночная статистика PostgreSQL (собираемая `ANALYZE`) хранит данные независимо для каждого столбца. Планировщик, встречая условие `WHERE a = X AND b = Y`, умножает отдельные селективности, исходя из предположения о независимости столбцов. Это предположение ломается в двух основных случаях:
- столбцы коррелированы (например, `city` и `zip_code`);
- нужно оценить количество уникальных групп по нескольким столбцам.

`CREATE STATISTICS` (появился в PostgreSQL 10, расширялся до версии 16+) позволяет создать объект статистики над группой столбцов или выражений.

**Виды статистики**

| Вид | Что хранит | Для каких запросов | Ограничения |
|-----|-----------|-------------------|-------------|
| `dependencies` | Коэффициент функциональной зависимости между столбцами (0..1) | `WHERE a = X AND b = Y` (равенство) | Только равенство, нет информации о конкретных значениях |
| `ndistinct` | Число уникальных комбинаций значений нескольких столбцов | `GROUP BY a, b` | Не помогает с фильтрацией |
| `mcv` | Список самых частых комбинаций значений с их частотами | `WHERE` по любым условиям, включая диапазоны | Дороже; не покрывает редкие значения |

Если вид не указан явно, PostgreSQL собирает все три:
```sql
CREATE STATISTICS s ON a, b FROM t;  -- ndistinct + dependencies + mcv
```

**Когда стандартной статистики недостаточно**

1. Запросы с `WHERE` по нескольким коррелированным столбцам — планировщик переоценивает или недооценивает селективность и выбирает неоптимальный метод соединения или индекс.
2. Запросы с `GROUP BY` по нескольким столбцам — неверная оценка числа групп ведёт к плохому выбору между `HashAggregate` и `Sort + GroupAggregate`.
3. Частичные индексы, партиционирование по нескольким столбцам — стандартная статистика не видит межстолбцовых зависимостей.

**`default_statistics_target` и качество планов**

Параметр `default_statistics_target` (значение по умолчанию: 100) управляет двумя вещами одновременно:
1. **Размером выборки** при `ANALYZE`: размер выборки составляет `300 * statistics_target` строк. При значении 100 это 30 000 строк.
2. **Размером MCV-списков**: максимальное количество записей в `most_common_vals` и `histogram_bounds`.

Расширенная статистика использует ту же выборку строк. Значит, увеличение `default_statistics_target` (или точечное `ALTER TABLE t ALTER COLUMN c SET STATISTICS 500`) одновременно:
- увеличивает точность функциональных зависимостей;
- позволяет хранить больше комбинаций в MCV-списке;
- улучшает точность ndistinct-оценок.

Цена: пропорционально растут время `ANALYZE` и объём данных в `pg_statistic`. Поэтому повышать `default_statistics_target` глобально не рекомендуется — лучше увеличивать его точечно для конкретных проблемных столбцов или таблиц.

**Просмотр результатов**
```sql
-- Посмотреть объекты расширенной статистики
SELECT stxname, stxkind FROM pg_statistic_ext;

-- Посмотреть MCV-список
SELECT m.* FROM pg_statistic_ext
JOIN pg_statistic_ext_data ON (oid = stxoid),
     pg_mcv_list_items(stxdmcv) m
WHERE stxname = 'my_stats';
```

### Вопрос 36. Как работает уровень изоляции Serializable (SSI — Serializable Snapshot Isolation) в PostgreSQL? Что такое write skew и почему Repeatable Read не защищает от него? Приведи пример аномалии write skew и объясни, как SSI её обнаруживает и разрывает.

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 13.2. Transaction Isolation](https://www.postgresql.org/docs/current/transaction-iso.html)

Уровень изоляции `SERIALIZABLE` обеспечивает самую строгую изоляцию транзакций. Согласно документации:

> «Этот уровень эмулирует последовательное выполнение транзакций для всех зафиксированных транзакций, как если бы транзакции выполнялись одна за другой, последовательно, а не одновременно.»

PostgreSQL реализует `SERIALIZABLE` через технику **Serializable Snapshot Isolation (SSI)**, которая строится на основе Snapshot Isolation с добавлением проверок на аномалии сериализации:

> «Уровень изоляции Serializable реализован с использованием техники, известной в академической литературе как Serializable Snapshot Isolation, которая строится на основе Snapshot Isolation путём добавления проверок на аномалии сериализации.»

**Пример write skew из документации.** Исходное состояние таблицы:

```
 class | value
-------+-------
     1 |    10
     1 |    20
     2 |   100
     2 |   200
```

Две параллельные транзакции:
```sql
-- Транзакция A:
SELECT SUM(value) FROM mytab WHERE class = 1;  -- результат: 30
INSERT INTO mytab VALUES (2, 30);

-- Транзакция B (одновременно):
SELECT SUM(value) FROM mytab WHERE class = 2;  -- результат: 300
INSERT INTO mytab VALUES (1, 300);
```

При уровне `REPEATABLE READ`: обе транзакции фиксируются. При уровне `SERIALIZABLE`: одна фиксируется, другая откатывается с ошибкой:
```
ERROR: could not serialize access due to read/write dependencies among transactions
```

**Почему Repeatable Read не защищает от write skew.** Документация явно указывает, что Repeatable Read предотвращает грязное чтение, неповторяемое чтение и фантомное чтение, но **не** предотвращает аномалии сериализации. Каждая транзакция видит консистентный снимок данных на момент своего начала, однако записи, сделанные параллельными транзакциями, остаются невидимыми. Это позволяет возникнуть логически невозможным конечным состояниям.

**Механизм обнаружения SSI.** PostgreSQL использует **предикатные блокировки** для отслеживания паттернов чтения-записи:

> «Для гарантии истинной сериализуемости PostgreSQL использует предикатные блокировки, которые позволяют определить, когда запись могла бы повлиять на результат предыдущего чтения из конкурентной транзакции, если бы та выполнилась первой.»

Предикатные блокировки не блокируют другие транзакции и не вызывают взаимоблокировок. Они видны в представлении `pg_locks` с режимом `SIReadLock`.

#### Ответ от Claude

**Что такое SSI**

PostgreSQL реализует `SERIALIZABLE` через Serializable Snapshot Isolation (SSI), введённый в версии 9.1 (2011). В отличие от традиционного строгого двухфазного блокирования (S2PL), SSI не блокирует читателей писателями и наоборот, что даёт значительно лучшую конкурентность.

Принцип работы: транзакции выполняются на снимках данных (как при Repeatable Read), но PostgreSQL дополнительно отслеживает **конфликты чтение-запись** между параллельными транзакциями. Если образуется опасная циклическая структура зависимостей, одна из транзакций откатывается.

**Что такое write skew**

Write skew — аномалия, при которой две параллельные транзакции читают пересекающиеся наборы данных и записывают изменения, основанные на этом чтении, в результате чего получается состояние, которое не могло бы возникнуть при любом последовательном выполнении транзакций.

Классический пример: система дежурств врачей. Правило бизнес-логики — хотя бы один врач должен быть на дежурстве.

```sql
-- Таблица: doctors (id, name, on_call BOOLEAN)
-- Состояние: Алиса on_call=true, Боб on_call=true

-- Транзакция A (Алиса уходит с дежурства):
BEGIN;
SELECT COUNT(*) FROM doctors WHERE on_call = true;  -- видит 2
-- Раз дежурных двое — можно снять Алису
UPDATE doctors SET on_call = false WHERE name = 'Alice';
COMMIT;

-- Транзакция B (Боб уходит с дежурства), одновременно:
BEGIN;
SELECT COUNT(*) FROM doctors WHERE on_call = true;  -- тоже видит 2 (снимок!)
-- Раз дежурных двое — можно снять Боба
UPDATE doctors SET on_call = false WHERE name = 'Bob';
COMMIT;

-- Итог при REPEATABLE READ: оба сняты, ни один не дежурит — нарушение правила!
```

При `SERIALIZABLE`: одна транзакция получит `ERROR: could not serialize access due to read/write dependencies among transactions` и должна быть повторена.

**Почему Repeatable Read не защищает**

Repeatable Read гарантирует, что транзакция видит снимок данных, зафиксированных до её начала, и не видит изменений параллельных транзакций. Именно это и является источником проблемы: обе транзакции честно читают «старое» состояние (2 дежурных) и делают корректные с их точки зрения выводы. Никакого нарушения правил Repeatable Read не происходит — просто конечный результат нарушает инвариант бизнес-логики.

Repeatable Read предотвращает:
- грязное чтение (dirty read);
- неповторяемое чтение (non-repeatable read);
- фантомное чтение (phantom read, в PostgreSQL).

Repeatable Read **не** предотвращает:
- аномалии сериализации (serialization anomalies), к которым относится write skew.

**Как SSI обнаруживает и разрывает аномалию**

SSI отслеживает два типа зависимостей между транзакциями:
- **rw-конфликт**: транзакция A прочитала данные, которые транзакция B потом записала (A читала «до» записи B).
- **wr-конфликт**: транзакция A записала данные, которые транзакция B читала после записи.

Опасная структура — **цикл rw-зависимостей**: если транзакции A и B образуют цикл (A->rw->B и B->rw->A), это означает, что никакого последовательного порядка выполнения, дающего тот же результат, не существует.

В примере с врачами:
1. Транзакция A читает строки `on_call = true` — PostgreSQL создаёт предикатную блокировку (`SIReadLock`) на этот предикат.
2. Транзакция B обновляет строку Боба — это создаёт rw-конфликт: A читала данные, которые B изменила.
3. Аналогично B читает те же строки, A изменяет строку Алисы — ещё один rw-конфликт.
4. PostgreSQL обнаруживает цикл: обе транзакции зависят друг от друга через чтение-запись.
5. По принципу «первый зафиксировавший выигрывает»: первая транзакция, достигшая `COMMIT`, фиксируется. Вторая получает ошибку сериализации.

Предикатные блокировки не блокируют чтение или запись, не вызывают дедлоков и видны в `pg_locks` с режимом `SIReadLock`.

**Практические рекомендации**

- Приложение обязано быть готово к повтору транзакции при получении `SQLSTATE 40001`.
- Избыточные ошибки сериализации возможны: PostgreSQL может откатить транзакцию, которая в действительности не создавала аномалии (ложноположительный результат для безопасности).
- SSI имеет ненулевые накладные расходы (память на предикатные блокировки). Параметры `max_pred_locks_per_transaction`, `max_pred_locks_per_relation`, `max_pred_locks_per_page` управляют агрегацией блокировок при нехватке памяти.

### Вопрос 37. Что такое savepoints и подтранзакции в PostgreSQL? Как они реализованы внутри (subtransaction XID) и почему большое количество вложенных savepoints в одной транзакции может приводить к деградации производительности?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 67.3. Subtransactions](https://www.postgresql.org/docs/current/subxacts.html)

Подтранзакции — это вложенные единицы транзакций внутри родительской транзакции, которые могут фиксироваться или откатываться независимо от родителя.

**Как запустить подтранзакцию:**
- явно командой `SAVEPOINT`;
- неявно через блок `EXCEPTION` в PL/pgSQL;
- через явные подтранзакции в PL/Python и PL/Tcl;
- из других подтранзакций (создавая дерево вложенности).

**Назначение XID.** Подтранзакции только для чтения не получают собственных XID. Подтранзакция, выполняющая запись, получает subxid из глобального пространства XID. При этом все родительские транзакции вплоть до верхнего уровня также получают ненулевые XID. Гарантируется: XID родителя всегда меньше XID любой его дочерней подтранзакции.

**Хранение.** Каталог `pg_subtrans` хранит XID непосредственного родителя для каждого subxid. Для транзакций верхнего уровня и подтранзакций только для чтения записи не создаются. Каталог `pg_xact` при фиксации транзакции верхнего уровня записывает все subcommitted подтранзакции как зафиксированные.

**Ключевое ограничение производительности.** Документация прямо указывает:

> «В разделяемой памяти для каждого бэкенда кэшируется до 64 открытых subxid; после этого порога накладные расходы на ввод-вывод значительно возрастают из-за дополнительных обращений к записям subxid в pg_subtrans.»

Единственное разумное решение — не создавать более 64 подтранзакций (savepoints или блоков `EXCEPTION` в PL/pgSQL) в одной транзакции.

#### Ответ от Claude

**Что такое savepoints и подтранзакции**

Savepoint (точка сохранения) — маркер внутри уже открытой транзакции, к которому можно откатиться, не прерывая всю транзакцию. Это реализация стандарта SQL через механизм **подтранзакций** (subtransactions).

```sql
BEGIN;
INSERT INTO orders VALUES (1, 'pending');
SAVEPOINT sp1;
INSERT INTO items VALUES (1, 'widget');  -- эта операция может провалиться
ROLLBACK TO SAVEPOINT sp1;              -- отменяем только второй INSERT
INSERT INTO items VALUES (1, 'gadget'); -- пробуем снова
RELEASE SAVEPOINT sp1;                  -- освобождаем savepoint
COMMIT;
```

Кроме явных `SAVEPOINT`, подтранзакции создаются неявно:
- блоком `BEGIN ... EXCEPTION WHEN ... END` в PL/pgSQL;
- параметром JDBC-драйвера `autosave=always` (savepoint перед каждым оператором);
- Django `transaction.atomic()`, Rails `transaction` при вложении, SQLAlchemy вложенные сессии.

**Внутренняя реализация: subtransaction XID**

Каждая подтранзакция с записью получает собственный XID из глобального 32-битного счётчика. Это критически важно: subxid используется в заголовках tuple для MVCC-проверок видимости. Транзакция из одного `BEGIN/COMMIT` с 10 savepoints и записями потребляет 11 XID вместо одного. На высоконагруженных системах это ускоряет XID wraparound.

Родительский XID всегда меньше дочернего subxid. Иерархия хранится в `pg_subtrans` (SLRU-кэш): для каждого subxid записывается XID его непосредственного родителя.

**Ограничение 64 subxid: PGPROC_MAX_CACHED_SUBXIDS**

В разделяемой памяти PostgreSQL хранит массив `PGPROC` для каждого бэкенда. Каждый бэкенд может кэшировать не более **64 открытых (не завершённых) subxid** — это константа `PGPROC_MAX_CACHED_SUBXIDS` в исходном коде.

Пока число одновременно открытых savepoints не превышает 64, все subxid хранятся в оперативной памяти и проверка видимости работает быстро.

При превышении 64 происходит **переполнение (suboverflow)**:
1. Снимок транзакции помечается флагом `suboverflowed`.
2. Снимок больше не содержит полного списка открытых subxid.
3. Для каждой проверки видимости tuple, созданного в подтранзакции, PostgreSQL вынужден обращаться к диску через SLRU-кэш `pg_subtrans`.
4. Доступ к SLRU требует захвата `SubtransSLRU` (до PG 13 — `SubtransControlLock`), что создаёт глобальное узкое место.

**Последствия переполнения**

На первичном сервере: деградация при большом количестве одновременных транзакций с overflow — каждая проверка видимости превращается в обращение к SLRU под блокировкой.

На репликах — более острая проблема. При наличии долгоживущей транзакции на первичном, реплика вынуждена постоянно обращаться к `pg_subtrans` для проверки видимости строк, изменённых в подтранзакциях. Это вызывает интенсивный thrashing SLRU-кэша. Реальный эффект: деградация TPS с ~210 000 до ~20 000, при этом CPU практически не загружен — сервер стоит в ожидании блокировок.

Случай Amazon Prime Day 2018 с Aurora PostgreSQL: слишком большое количество savepoints переполнило внутреннюю очередь обработки, что привело к производственному инциденту.

**Итоговые рекомендации**

- Не превышать 64 одновременно открытых savepoints в одной транзакции.
- Своевременно делать `RELEASE SAVEPOINT` — освобождённый savepoint не занимает слот в PGPROC.
- Отключать `autosave=always` в JDBC-драйвере для высоконагруженных приложений; использовать точечные savepoints только там, где нужно.
- Проверять `pg_stat_activity` на наличие долгоживущих транзакций и мониторить wait-события `SubtransSLRU`.
- В PL/pgSQL экономно использовать блоки `EXCEPTION` в циклах: каждая итерация с `EXCEPTION` создаёт новую подтранзакцию.

### Вопрос 38. Как работает Row-Level Security (RLS) в PostgreSQL? Чем политика RLS отличается от обычного WHERE-фильтра в представлении, что означает FORCE ROW LEVEL SECURITY и как суперпользователи и роли с атрибутом BYPASSRLS взаимодействуют с политиками?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation — Row Security Policies](https://www.postgresql.org/docs/current/ddl-rowsecurity.html)

Row-Level Security (RLS) позволяет таблице иметь политики, ограничивающие доступ к строкам для конкретных пользователей. Когда RLS включён, все обычные запросы к таблице видят и могут изменять только те строки, которые разрешены соответствующей политикой. Политики создаются командой `CREATE POLICY` и применяются к конкретным командам SQL (`SELECT`, `INSERT`, `UPDATE`, `DELETE`).

Включение RLS:
```sql
ALTER TABLE accounts ENABLE ROW LEVEL SECURITY;
```

Создание политики:
```sql
CREATE POLICY account_managers ON accounts TO managers
    USING (manager = current_user);
```

Выражение политики вычисляется для каждой строки до любых условий, поступающих из запроса пользователя. В отличие от WHERE-фильтра в представлении, политики RLS прозрачны для пользователя — он не видит их определений и не знает о фильтрации.

Несколько политик на одну таблицу комбинируются:
- **Разрешающие (PERMISSIVE)** — объединяются через `OR`.
- **Ограничивающие (RESTRICTIVE)** — объединяются через `AND`.

**FORCE ROW LEVEL SECURITY:** по умолчанию владелец таблицы обходит RLS. Команда `ALTER TABLE ... FORCE ROW LEVEL SECURITY` заставляет владельца также подчиняться политикам.

**Суперпользователи и BYPASSRLS:** согласно документации: «Суперпользователи и роли с атрибутом `BYPASSRLS` всегда обходят систему безопасности на уровне строк при обращении к таблице». Это правило действует даже при `FORCE ROW LEVEL SECURITY` — оно распространяется только на владельца таблицы, но не на суперпользователей.

#### Ответ от Claude

**Что такое RLS и как включить**

Row-Level Security — механизм PostgreSQL, ограничивающий видимость строк на уровне СУБД. После включения RLS (`ALTER TABLE t ENABLE ROW LEVEL SECURITY`) таблица по умолчанию становится полностью закрытой: если ни одна политика не совпала, строка не видна (принцип «запрещено всё, что не разрешено»). Политики применяются ко всем запросам автоматически, включая запросы через ORM и приложения.

**Отличие RLS от WHERE-фильтра в представлении**

| Аспект | WHERE-фильтр в представлении | Политика RLS |
|---|---|---|
| Где определён | В определении VIEW, видно в `\d+ view_name` | Скрыт от пользователя |
| Кто может обойти | Любой с правом `SELECT` на базовую таблицу | Только суперпользователь / BYPASSRLS |
| Охват команд | Только SELECT через это представление | SELECT, INSERT, UPDATE, DELETE — отдельно для каждой |
| Множественность | Один WHERE на представление | Несколько политик, комбинируются по OR/AND |
| Контекст выполнения | Права вызывающего (или SECURITY DEFINER) | Права вызывающего; можно использовать security-definer функции |

Ключевое преимущество RLS: политика применяется независимо от способа обращения к таблице. Прямой `SELECT * FROM accounts` не позволяет обойти ограничение в отличие от случая, когда нет `SECURITY INVOKER`-представления.

**FORCE ROW LEVEL SECURITY**

По умолчанию владелец таблицы полностью обходит RLS. Это нужно для администрирования, но создаёт риск случайного просмотра/изменения чужих данных. Команда:

```sql
ALTER TABLE accounts FORCE ROW LEVEL SECURITY;
```

— заставляет владельца тоже проходить через политики. Используется для многопользовательских SaaS-систем, где даже DBA не должен видеть данные других арендаторов без явного переключения роли.

**Суперпользователи и BYPASSRLS**

Документация PostgreSQL однозначно определяет иерархию:

1. **Суперпользователи** — всегда обходят RLS, `FORCE ROW LEVEL SECURITY` на них не распространяется.
2. **Роли с атрибутом `BYPASSRLS`** — аналогично суперпользователям, полностью минуют все политики.
3. **Владелец таблицы без `FORCE ROW LEVEL SECURITY`** — обходит политики.
4. **Владелец таблицы с `FORCE ROW LEVEL SECURITY`** — подчиняется политикам.
5. **Обычные роли** — обязаны удовлетворять политикам.

Создание роли с BYPASSRLS:
```sql
CREATE ROLE admin_user BYPASSRLS;
-- или изменить существующую:
ALTER ROLE admin_user BYPASSRLS;
```

**Практический паттерн (multi-tenant)**

```sql
-- Таблица арендаторов
ALTER TABLE orders ENABLE ROW LEVEL SECURITY;
ALTER TABLE orders FORCE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON orders
    USING (tenant_id = current_setting('app.current_tenant')::int);
```

Приложение устанавливает `SET LOCAL app.current_tenant = 42` в начале транзакции, и RLS автоматически изолирует данные между арендаторами.

### Вопрос 39. Как устроен механизм LISTEN/NOTIFY в PostgreSQL? Как он передаёт уведомления между сессиями, какой максимальный размер payload (начиная с какой версии), и в чём преимущества и ограничения этого механизма по сравнению с внешними очередями сообщений (например, Redis Pub/Sub)?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation — NOTIFY](https://www.postgresql.org/docs/current/sql-notify.html)

Команда `NOTIFY` отправляет уведомление всем сессиям, подписавшимся на указанный канал командой `LISTEN`. Уведомление содержит:
- имя канала,
- PID отправляющего серверного процесса,
- строку payload (с версии 9.0).

**Максимальный размер payload:** 8000 байт. Для передачи больших данных рекомендуется сохранять их в таблице и передавать в payload только ключ записи.

**Транзакционное поведение:** уведомления доставляются только после коммита транзакции. Если транзакция откатывается, уведомления отменяются. Одинаковые уведомления (канал + payload) в рамках одной транзакции дедуплицируются и доставляются однократно.

**Очередь:** система хранит уведомления во внутренней очереди. Если очередь заполнена (максимум 8 ГБ), команда `NOTIFY` завершится ошибкой при коммите. При заполнении на 50% появляются предупреждения.

```sql
-- Подписка
LISTEN order_updates;

-- Отправка уведомления (из другой сессии)
NOTIFY order_updates, '{"order_id": 42, "status": "shipped"}';

-- Или через функцию (удобнее для динамических имён каналов)
SELECT pg_notify('order_updates', '{"order_id": 42}');
```

Сессия, выполнившая `NOTIFY`, также получит собственное уведомление — клиентское приложение должно фильтровать их по PID.

#### Ответ от Claude

**Архитектура механизма**

LISTEN/NOTIFY — встроенный механизм асинхронных уведомлений PostgreSQL. Уведомления не хранятся в таблицах и не персистируются: это эфемерные сигналы, передаваемые через разделяемую память серверного процесса. Подписка активна ровно столько, сколько существует сессия.

**Как работает доставка**

1. Сессия A выполняет `LISTEN channel_name` — сервер регистрирует её как слушателя.
2. Сессия B выполняет `NOTIFY channel_name, 'payload'` внутри транзакции.
3. Уведомление помещается во внутреннюю очередь, но **не доставляется до коммита** транзакции.
4. После коммита PostgreSQL пробуждает все зарегистрированные сессии на данном канале.
5. Клиент вычитывает уведомление при следующем обращении к соединению (в libpq — через `PQnotifies()`).

Важно: клиент получает уведомление **только между транзакциями** — не в момент, когда он сам находится внутри активной транзакции.

**Payload и история версий**

До PostgreSQL 9.0 `NOTIFY` передавал только имя канала — никакого дополнительного контента. В версии 9.0 одновременно были сделаны два улучшения:
- добавлена возможность передавать строку payload,
- реализация переведена с табличного хранения на очередь в памяти (значительный прирост производительности).

Максимальный размер payload — **8000 байт** (ограничение сохраняется в актуальных версиях).

**Сравнение с Redis Pub/Sub**

| Характеристика | PostgreSQL LISTEN/NOTIFY | Redis Pub/Sub |
|---|---|---|
| Персистентность | Нет (в полёте) | Нет (стандартный Pub/Sub) |
| Гарантия доставки | Нет (offline-получатель теряет сообщения) | Нет |
| Транзакционность | Да — NOTIFY фиксируется вместе с транзакцией | Нет |
| Максимальный payload | 8 000 байт | Не ограничен практически |
| Дедупликация | Да (в рамках транзакции) | Нет |
| Масштабирование | Один кластер PostgreSQL | Кластер Redis, несколько узлов |
| Дополнительный инфраструктурный компонент | Не нужен | Нужен отдельный Redis |
| Производительность при высокой нагрузке | Ограничена (не для миллионов событий/сек) | Высокая |

**Ключевое преимущество LISTEN/NOTIFY:** транзакционная атомарность. Уведомление гарантированно отправляется тогда и только тогда, когда транзакция зафиксирована. Это устраняет классическую проблему «двойной записи» — когда приложение меняет данные в БД и отправляет событие во внешнюю очередь, но одно из двух действий может упасть.

**Ограничения**

- Сессия, не подключённая в момент уведомления, теряет его безвозвратно.
- 8 000 байт payload недостаточно для передачи полного тела объекта — нужно отправлять идентификатор и дочитывать из БД.
- Долгая транзакция в слушающей сессии блокирует очистку очереди, что может привести к её переполнению.
- Не подходит для высоконагруженных систем с тысячами событий в секунду и требованием гарантированной доставки.

**Типичный use case:** инвалидация кэша, уведомление фоновых воркеров о появлении новых задач в таблице, real-time обновления в веб-приложениях через WebSocket без дополнительной инфраструктуры.

### Вопрос 40. Как работают репликационные слоты (replication slots) в PostgreSQL? Чем физический слот отличается от логического, почему неиспользуемый слот может привести к неконтролируемому росту WAL на диске, и как параметр hot_standby_feedback взаимодействует со слотами?

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation — Replication Slots](https://www.postgresql.org/docs/current/warm-standby.html#STREAMING-REPLICATION-SLOTS)

Репликационные слоты — автоматизированный механизм, гарантирующий, что первичный сервер не удалит WAL-сегменты до тех пор, пока они не будут получены всеми подписчиками. Кроме того, слоты предотвращают удаление строк, которые могут вызвать конфликты восстановления на резервном сервере, — даже когда резервный сервер отключён.

Документация содержит явное предупреждение:

> «Будьте осторожны: репликационные слоты могут привести к тому, что сервер будет удерживать столько WAL-сегментов, что они заполнят пространство, выделенное для `pg_wal`. Параметр `max_slot_wal_keep_size` позволяет ограничить размер WAL-файлов, удерживаемых репликационными слотами.»

Относительно взаимодействия с `hot_standby_feedback`:

> «Аналогично, `hot_standby_feedback` сам по себе, без использования репликационного слота, обеспечивает защиту от удаления нужных строк через vacuum, но не обеспечивает защиту в любой период, когда резервный сервер не подключён.»

Создание и проверка физического слота:
```sql
SELECT * FROM pg_create_physical_replication_slot('node_a_slot');
SELECT slot_name, slot_type, active FROM pg_replication_slots;
```

#### Ответ от Claude

**Что такое репликационный слот**

Репликационный слот — постоянная (персистентная) запись на первичном сервере, которая отслеживает, до какой позиции WAL потребитель (резервный сервер или логический подписчик) уже прочитал данные. Слот хранится на диске в `pg_replslot/` и переживает перезапуск сервера. PostgreSQL гарантирует, что WAL-сегменты, на которые ссылается слот, не будут удалены до тех пор, пока слот не продвинется вперёд.

**Физический слот vs. логический слот**

| Аспект | Физический слот | Логический слот |
|---|---|---|
| Потребитель | Резервный сервер (streaming replication) | Логический подписчик (logical replication, CDC-инструменты) |
| Формат данных | Побайтовые WAL-записи (бинарный формат конкретной версии PostgreSQL) | Декодированные логические изменения (INSERT/UPDATE/DELETE) через output plugin |
| Версионная совместимость | Только идентичная мажорная версия PostgreSQL | Возможна репликация между разными версиями и даже разными СУБД |
| Что отслеживает | LSN — до какого байта WAL прочитан | LSN + состояние системного каталога для декодирования |
| Каталог | Не нужно удерживать системный каталог | Удерживает строки системного каталога (`pg_class`, `pg_attribute` и др.) для корректного декодирования изменений схемы |
| Типичное применение | HA-репликация, pg_basebackup | Debezium, Kafka Connect, pglogical, Postgres logical replication |

**Почему неиспользуемый слот опасен**

PostgreSQL не имеет таймаута для слотов. Слот, на который никто не подписан, продолжает «держать» WAL-сегменты, начиная с позиции последнего подтверждённого чтения. Если потребитель отключился или был удалён без явного удаления слота:

1. `pg_wal/` растёт неограниченно — сервер не может удалять старые сегменты.
2. Для логических слотов дополнительно накапливаются мёртвые строки в системном каталоге, потому что autovacuum не удаляет строки, необходимые для декодирования WAL.
3. Может возникнуть угроза **wraparound transaction ID** — одна из наиболее критических ситуаций, при которой PostgreSQL переходит в режим только для чтения или принудительно останавливается.

**Мониторинг и защита:**

```sql
-- Просмотр активных слотов и удерживаемого WAL
SELECT slot_name, slot_type, active, restart_lsn,
       pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)) AS lag
FROM pg_replication_slots;

-- Ограничение WAL, удерживаемого слотами (PostgreSQL 13+)
-- В postgresql.conf:
-- max_slot_wal_keep_size = '10GB'

-- Удаление ненужного слота
SELECT pg_drop_replication_slot('old_slot');
```

**Взаимодействие hot_standby_feedback со слотами**

`hot_standby_feedback = on` решает проблему конфликтов восстановления: резервный сервер регулярно сообщает первичному о своей самой старой активной транзакции, и первичный откладывает vacuum строк, которые ещё нужны для запросов на резервном.

Однако у этого механизма есть критическое ограничение: защита действует **только пока резервный подключён**. При разрыве соединения первичный больше не получает feedback, и vacuum начинает удалять строки. Когда резервный переподключается, он может столкнуться с конфликтами восстановления.

Использование **репликационного слота совместно с hot_standby_feedback** обеспечивает полную защиту:

- Слот предотвращает удаление WAL-сегментов и нужных строк даже при отключённом резервном сервере.
- `hot_standby_feedback` снижает задержку применения WAL и предотвращает накопление мёртвых строк на первичном во время активных запросов на резервном.

```
# На резервном сервере (postgresql.conf / recovery.conf)
primary_conninfo = 'host=primary.local port=5432 user=replicator'
primary_slot_name = 'standby1_slot'
hot_standby_feedback = on
```

Расхождение между источниками: документация PostgreSQL предупреждает, что `hot_standby_feedback` может вызывать раздувание (bloat) на первичном сервере при длительных транзакциях на резервном. Ряд экспертов (cybertec-postgresql.com) рекомендует использовать слоты осторожно в высоконагруженных средах и всегда устанавливать `max_slot_wal_keep_size`, чтобы слот не смог монопольно занять весь диск.
