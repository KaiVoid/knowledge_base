### Вопрос 41. Что такое WAL-архивация и Point-in-Time Recovery (PITR) в PostgreSQL? Объясни роль параметров archive_mode, archive_command и restore_command, и опиши последовательность действий при восстановлении базы на момент времени T после потери primary-узла.

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 25.3. Continuous Archiving and Point-in-Time Recovery (PITR)](https://www.postgresql.org/docs/current/continuous-archiving.html)

PostgreSQL постоянно записывает Write-Ahead Log (WAL) в директорию `pg_wal/`. В этом журнале содержатся описания всех изменений, внесённых в файлы данных. Изначально WAL предназначен для аварийного восстановления: если система падает, журнал проигрывается заново и база возвращается в согласованное состояние. Однако при включении **непрерывной архивации** можно сохранять завершённые сегменты WAL во внешнее хранилище, а затем использовать их совместно с базовой резервной копией для восстановления на любой момент времени.

**archive_mode**

Параметр включает механизм архивации. Допустимые значения: `on` и `always`. Значение `always` требуется на standby-серверах, чтобы `pg_backup_stop()` дожидался завершения архивации. Параметр является статическим — изменение требует перезапуска сервера. Помимо `archive_mode = on` необходимо установить `wal_level = replica` (или выше).

**archive_command**

Содержит shell-команду, копирующую завершённый сегмент WAL в архив. Поддерживаемые подстановки:
- `%p` — полный путь к файлу сегмента (относительно каталога кластера);
- `%f` — только имя файла;
- `%%` — литеральный символ `%`.

Пример (Unix):
```bash
archive_command = 'test ! -f /mnt/server/archivedir/%f && cp %p /mnt/server/archivedir/%f'
```

Команда вызывается только для **завершённых** сегментов WAL. Она обязана возвращать ненулевой код выхода при ошибке. Перезаписывать уже существующий архивный файл **запрещено** — при обнаружении существующего файла с другим содержимым команда должна возвращать ненулевой код.

**restore_command**

Задаётся в `postgresql.conf` перед стартом в режиме восстановления и указывает, как извлечь сегмент WAL из архива:

```bash
restore_command = 'cp /mnt/server/archivedir/%f %p'
```

Подстановки те же (`%f` — имя файла в архиве, `%p` — путь назначения). Команда обязана возвращать ненулевой код, если файл не найден: это сигнал PostgreSQL, что архив исчерпан. PostgreSQL также самостоятельно проверяет `pg_wal/` на наличие неархивированных сегментов.

**Последовательность действий при PITR после потери primary-узла**

1. Остановить сервер (если он ещё работает): `pg_ctl stop`.
2. Сохранить каталог кластера и содержимое `pg_wal/` во временное место (страховка).
3. Удалить все файлы из каталога кластера и каталогов табличных пространств.
4. Восстановить базовую резервную копию (`pg_basebackup` или низкоуровневый бэкап через `pg_backup_start/stop`) в каталог кластера. Убедиться в правильных правах владельца (системный пользователь PostgreSQL).
5. Очистить `pg_wal/`: удалить устаревшие WAL-файлы, при необходимости воссоздать директорию с корректными правами.
6. Вернуть сохранённые в шаге 2 неархивированные WAL-файлы в `pg_wal/`.
7. Создать файл `recovery.signal` в каталоге кластера (`touch recovery.signal`).
8. В `postgresql.conf` задать параметры восстановления:
   ```
   restore_command = 'cp /mnt/server/archivedir/%f %p'
   recovery_target_time = '2024-01-15 14:30:00'
   recovery_target_inclusive = true
   recovery_target_timeline = 'latest'
   ```
9. При необходимости ограничить подключения пользователей через `pg_hba.conf`.
10. Запустить сервер: `pg_ctl start`. PostgreSQL будет воспроизводить WAL-сегменты через `restore_command` до достижения целевого момента времени. После завершения файл `recovery.signal` удаляется автоматически, и сервер переходит в нормальный режим работы.
11. Проверить состояние базы, затем вернуть доступ пользователям.

#### Ответ от Claude

**Что такое WAL-архивация**

Write-Ahead Log (WAL) — это механизм, при котором все изменения сначала записываются в журнал, и лишь затем применяются к файлам данных. Это гарантирует атомарность и возможность восстановления после сбоя. Непрерывная архивация WAL (`continuous archiving`) расширяет эту концепцию: завершённые сегменты WAL копируются во внешнее хранилище, что позволяет восстановить кластер на **любой** момент в прошлом — не только на момент последнего бэкапа.

**Роль ключевых параметров**

| Параметр | Что делает | Важные ограничения |
|---|---|---|
| `archive_mode = on` | Активирует архивацию WAL | Статический, требует перезапуска; нужен `wal_level >= replica` |
| `archive_command` | Shell-команда, выполняемая при готовности сегмента | Вызывается только для завершённых сегментов; ненулевой код = ошибка, retry |
| `restore_command` | Shell-команда для извлечения сегмента при восстановлении | Задаётся перед стартом в режиме recovery; ненулевой код = файл не найден |

Дополнительный параметр `archive_timeout` позволяет принудительно переключать сегменты через заданный интервал (например, каждую минуту), снижая максимальную потерю данных при редких изменениях. Очень малые значения не рекомендуются: каждый принудительный сегмент занимает полный размер (по умолчанию 16 МБ).

**Параметры цели восстановления**

- `recovery_target_time` — восстановление до заданного момента времени (ISO 8601 или `YYYY-MM-DD HH:MI:SS`).
- `recovery_target_name` — восстановление до именованной точки, созданной `pg_create_restore_point('name')`.
- `recovery_target_xid` — восстановление до конкретного ID транзакции.
- `recovery_target_lsn` — восстановление до конкретной позиции в WAL.
- `recovery_target_inclusive` (по умолчанию `true`) — включать ли целевую транзакцию/время в восстановление.
- `recovery_target_timeline = 'latest'` — восстановление на самую свежую ветку (timeline) в архиве. После каждого PITR PostgreSQL создаёт новую timeline, чтобы разные ветки восстановления могли сосуществовать в архиве.

**Механизм timeline**

Каждое завершённое PITR-восстановление порождает новую «ветку истории» (timeline). Это позволяет: а) хранить несколько вариантов восстановления в одном архиве; б) при необходимости откатиться к более раннему варианту. Имя сегмента WAL начинается с 8-значного hex-идентификатора timeline, например: `0000000200001234000055CD`.

**Полная последовательность PITR после потери primary**

1. **Подготовка**: остановить аварийный сервер (если возможно); при наличии standby — убедиться, что он не принял на себя роль primary раньше времени.
2. **Страховка**: скопировать `pg_wal/` во временное место — там могут быть неархивированные сегменты.
3. **Очистка**: удалить все файлы кластера и табличных пространств.
4. **Восстановление базового бэкапа**: распаковать снимок (`pg_basebackup` или ручной бэкап) в каталог кластера с правильным владельцем.
5. **WAL**: очистить `pg_wal/` от устаревших файлов; скопировать туда сохранённые неархивированные сегменты из шага 2.
6. **recovery.signal**: `touch $PGDATA/recovery.signal` — этот файл сигнализирует PostgreSQL о старте в режиме восстановления.
7. **Конфигурация в postgresql.conf**:
   ```
   restore_command = 'cp /mnt/server/archivedir/%f %p'
   recovery_target_time = 'T'
   recovery_target_inclusive = true
   recovery_target_timeline = 'latest'
   ```
8. **Ограничить доступ** через `pg_hba.conf` на время восстановления.
9. **Запуск**: `pg_ctl start`. Сервер воспроизводит WAL-сегменты из архива до достижения момента T. `recovery.signal` удаляется автоматически.
10. **Проверка и разблокировка**: убедиться в корректности данных, восстановить `pg_hba.conf`.

**Важное замечание об ограничениях**: WAL-архивация не сохраняет изменения конфигурационных файлов (`postgresql.conf`, `pg_hba.conf`, `pg_ident.conf`) — они редактируются вручную и не проходят через SQL-операции.

### Вопрос 42. Чем COPY отличается от INSERT при массовой загрузке данных в PostgreSQL? Какие параметры и настройки (unlogged tables, synchronous_commit off, maintenance_work_mem, disable_triggers) ускоряют bulk-загрузку и какие компромиссы они несут?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 14.4. Populating a Database](https://www.postgresql.org/docs/current/populate.html)

Загрузка большого количества строк с помощью `COPY` почти всегда быстрее, чем с помощью `INSERT`, даже если `INSERT` использует `PREPARE` и пакетирует несколько вставок в одну транзакцию. `COPY` оптимизирован для загрузки большого количества строк и несёт значительно меньше накладных расходов при массовой загрузке данных.

**Отключение автофиксации (autocommit)**

При использовании нескольких `INSERT` рекомендуется оборачивать все вставки в одну транзакцию через `BEGIN`/`COMMIT`, избегая накладных расходов на фиксацию каждой строки отдельно.

**Удаление индексов и внешних ключей**

Если таблица не пуста, то перед загрузкой стоит удалить индексы и ограничения внешних ключей, а после загрузки пересоздать их. Создание индекса по уже загруженным данным значительно быстрее инкрементального обновления при каждой вставке. Ограничения внешних ключей следует удалять, а не просто отключать, чтобы избежать переполнения очереди событий триггеров при загрузке миллионов строк.

**Отключение триггеров**

Если нет другого выхода, можно использовать `SET session_replication_role = replica` для подавления триггеров. Альтернатива — `pg_restore --disable-triggers`, но это полностью устраняет проверку внешних ключей (а не откладывает её), что создаёт риск вставки некорректных данных.

**maintenance_work_mem**

Временное увеличение `maintenance_work_mem` ускоряет выполнение `CREATE INDEX` и `ALTER TABLE ADD FOREIGN KEY`, но не влияет существенно на сам `COPY`.

**Увеличение max_wal_size**

При загрузке больших объёмов данных рекомендуется временно увеличить `max_wal_size`, чтобы снизить частоту контрольных точек (checkpoints). Каждый checkpoint сбрасывает все «грязные» страницы на диск, что замедляет процесс.

**Отключение WAL-архивации и потоковой репликации**

Если `wal_level` установлен в `minimal`, а `archive_mode = off` и `max_wal_senders = 0`, то при загрузке в только что созданную или усечённую (`TRUNCATE`) таблицу некоторые команды могут полностью пропустить запись в WAL. Это ускоряет загрузку, но требует перезапуска сервера для изменения этих параметров.

**ANALYZE после загрузки**

После массовой загрузки необходимо выполнить `ANALYZE`, чтобы обновить статистику планировщика. Без актуальной статистики планировщик может выбирать неэффективные планы выполнения запросов.

#### Ответ от Claude

**COPY vs INSERT: ключевые различия**

`COPY` — это специализированная команда для потоковой загрузки строк. В отличие от `INSERT`, она:

1. Выполняет проверки (права доступа, существование таблицы, соответствие типов данных) **один раз** для всей операции, а не для каждой строки.
2. Работает как единая команда, обходя накладные расходы парсинга и планирования для каждой строки.
3. Не требует явного управления транзакциями — вся загрузка атомарна.
4. Поддерживает форматы `TEXT`, `CSV` и бинарный (`BINARY`); бинарный формат быстрее для числовых данных, но непереносим между версиями.

`INSERT` (даже батчевый или через `PREPARE`/`EXECUTE`) несёт накладные расходы на каждую строку: проверку блокировок, обновление индексов инкрементально и взаимодействие с менеджером транзакций.

**Оптимизации bulk-загрузки и их компромиссы**

**1. UNLOGGED TABLE**

```sql
CREATE UNLOGGED TABLE staging_data (...);
-- загрузка
ALTER TABLE staging_data SET LOGGED;  -- при необходимости сделать постоянной
```

- **Как работает**: таблица не записывает данные в WAL, поэтому каждая страница пишется на диск только один раз (а не дважды: сначала в WAL, затем в heap).
- **Компромисс**: при аварийном завершении PostgreSQL таблица автоматически **усекается до пустого состояния** при следующем старте — данные безвозвратно теряются. Таблица не реплицируется на standby-серверы (физическая репликация игнорирует unlogged-таблицы).
- **Применение**: промежуточные (staging) таблицы, кеши, временные агрегаты, данные которые можно пересчитать.

**2. synchronous_commit = off**

```sql
SET synchronous_commit = off;
-- или в postgresql.conf для всей сессии
```

- **Как работает**: PostgreSQL не ожидает подтверждения записи WAL на диск перед возвратом успеха клиенту. WAL-writer периодически сбрасывает данные с интервалом `wal_writer_delay` (по умолчанию 200 мс).
- **Максимальное окно потери данных**: 3 × `wal_writer_delay` (около 600 мс при настройках по умолчанию).
- **Принципиальное отличие от `fsync = off`**: `synchronous_commit = off` никогда не вызывает **порчи данных (corruption)** — только возможная потеря последних нескольких зафиксированных транзакций. `fsync = off` может привести к произвольной порче базы при сбое ОС.
- **Компромисс**: при аварии ОС (не просто падении PostgreSQL) могут быть потеряны транзакции, зафиксированные в последние ~600 мс.
- **Применение**: пакетные ETL-загрузки, аналитические пайплайны, логирование некритичных событий.

**3. maintenance_work_mem**

```sql
SET maintenance_work_mem = '1GB';
```

- **Как работает**: увеличивает объём памяти, доступной для операций обслуживания: `CREATE INDEX`, `ALTER TABLE ADD FOREIGN KEY`, `VACUUM`. Позволяет строить B-tree индексы, сортируя больше данных в памяти без обращения к диску.
- **Компромисс**: не влияет непосредственно на `COPY`. Эффективен только при создании/пересоздании индексов и внешних ключей **после** загрузки данных. Слишком высокое значение может вытеснить рабочие данные из page cache при наличии многих параллельных сессий.
- **Применение**: устанавливать перед `CREATE INDEX` на только что загруженной таблице.

**4. Отключение триггеров**

Два подхода:

```sql
-- Подход 1: сессионный уровень (отключает пользовательские и репликационные триггеры)
SET session_replication_role = replica;
-- ... загрузка ...
SET session_replication_role = DEFAULT;

-- Подход 2: ALTER TABLE (требует суперпользователя)
ALTER TABLE my_table DISABLE TRIGGER ALL;
-- ... загрузка ...
ALTER TABLE my_table ENABLE TRIGGER ALL;
```

- **Компромисс `--disable-triggers` / `session_replication_role`**: внешние ключи не проверяются в момент загрузки. Если загружены некорректные данные, ограничения ссылочной целостности будут нарушены. Рекомендуется: удалить FK-ограничения до загрузки и воссоздать после — это обеспечит пакетную валидацию при `ALTER TABLE ADD FOREIGN KEY`.

**5. Прочие важные оптимизации**

| Приём | Описание | Риск |
|---|---|---|
| Удалить индексы до загрузки | Пересоздать после: `CREATE INDEX` по существующим данным быстрее инкрементального обновления | Простой запросов, использующих индексы, на время загрузки |
| Увеличить `max_wal_size` | Снизить частоту checkpoint'ов (каждый сбрасывает все dirty pages) | Больший объём `pg_wal/` и более долгое crash-recovery |
| `wal_level = minimal` + `archive_mode = off` | Позволяет пропустить WAL совсем для новых/усечённых таблиц | Требует перезапуска; отключает репликацию и архивацию |
| `ANALYZE` после загрузки | Обновить статистику планировщика | Без этого возможны плохие планы выполнения |
| `pg_restore -j N` | Параллельная загрузка и создание индексов | Только для формата `directory` или `custom` |

**Рекомендуемая стратегия для максимальной скорости bulk-загрузки**:
1. `CREATE UNLOGGED TABLE` или `TRUNCATE` существующей таблицы.
2. Удалить все индексы и FK-ограничения.
3. `SET maintenance_work_mem = '1GB'` и `SET synchronous_commit = off`.
4. Загрузить данные через `COPY ... FROM`.
5. Воссоздать индексы и FK-ограничения.
6. `ANALYZE`.

### Вопрос 43. Как работает REFRESH MATERIALIZED VIEW CONCURRENTLY в PostgreSQL? Почему для него требуется уникальный индекс на материализованном представлении, и в чём разница с обычным REFRESH с точки зрения блокировок и накладных расходов?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — REFRESH MATERIALIZED VIEW](https://www.postgresql.org/docs/current/sql-refreshmaterializedview.html)

`REFRESH MATERIALIZED VIEW` полностью заменяет содержимое материализованного представления. Для использования этой команды необходимо быть владельцем представления.

Опция `CONCURRENTLY` позволяет выполнить обновление без блокировки параллельных `SELECT` на материализованном представлении. Без этой опции обновление, затрагивающее большое количество строк, потребует меньше ресурсов и завершится быстрее, но заблокирует другие соединения, пытающиеся читать из представления. Опция `CONCURRENTLY` может быть быстрее в случаях, когда затронуто небольшое количество строк.

Опция `CONCURRENTLY` допускается только при наличии хотя бы одного уникального индекса (`UNIQUE`) на материализованном представлении, который использует только имена столбцов и охватывает все строки: он не должен быть индексом по выражению или содержать условие `WHERE`.

Даже при использовании `CONCURRENTLY` одновременно может выполняться только один `REFRESH` для конкретного материализованного представления.

`CONCURRENTLY` нельзя использовать совместно с опцией `WITH NO DATA`.

**Блокировки**

- Без `CONCURRENTLY`: на материализованное представление устанавливается блокировка `ACCESS EXCLUSIVE`. Эта блокировка конфликтует со всеми другими режимами блокировок, включая `ACCESS SHARE` (используемый при `SELECT`). Таким образом, любые запросы к представлению блокируются до завершения обновления.
- С `CONCURRENTLY`: устанавливается блокировка `SHARE UPDATE EXCLUSIVE`. Этот режим совместим с `ACCESS SHARE`, то есть параллельные `SELECT` на представление продолжают выполняться без блокировки.

#### Ответ от Claude

**Внутренний алгоритм CONCURRENTLY**

Обычный `REFRESH MATERIALIZED VIEW` работает просто: старые данные удаляются, новые загружаются атомарно под блокировкой `ACCESS EXCLUSIVE`. Весь процесс исключает любой доступ к представлению извне.

`REFRESH MATERIALIZED VIEW CONCURRENTLY` реализует более сложный алгоритм:

1. **Вычисление нового содержимого**: запрос из определения представления выполняется во **временную таблицу** — читатели в это время продолжают работать со старыми данными.
2. **Diff через FULL OUTER JOIN**: PostgreSQL выполняет полное внешнее соединение между временной таблицей (новые данные) и существующим материализованным представлением (старые данные). Результат — множество строк, которые нужно вставить (есть в новых, нет в старых), и строк, которые нужно удалить (есть в старых, нет в новых).
3. **Применение изменений**: PostgreSQL вставляет новые строки и удаляет устаревшие через `INSERT`/`DELETE`, не затрагивая неизменившиеся строки.
4. **Финальный swap**: в самом конце берётся кратковременная более строгая блокировка для атомарного завершения транзакции — читатели переключаются на актуальные данные.

**Почему требуется уникальный индекс**

Уникальный индекс необходим для корректного выполнения FULL OUTER JOIN в шаге 2. Без гарантии уникальности строк PostgreSQL не может однозначно сопоставить строки старой и новой версий представления:

- При наличии дублирующихся строк JOIN создаёт декартово произведение совпадений, что делает diff некорректным.
- Уникальный индекс гарантирует отсутствие дубликатов, что позволяет точно определить добавленные и удалённые строки.

Требования к индексу (из официальной документации): должен использовать **только имена столбцов** (не выражения), **не** содержать условия `WHERE`, **охватывать все строки** представления. Нарушение любого из этих условий приводит к ошибке выполнения команды.

Известная проблема с NULL: если уникальный индекс содержит nullable-столбец, производительность может резко деградировать. `NULL = NULL` вычисляется в `NULL` (а не `TRUE`), поэтому в FULL OUTER JOIN все строки с NULL в ключевом столбце выглядят «разными», что приводит к ложному полному пересозданию данных.

**Сравнение блокировок**

| Режим | Тип блокировки | Конфликт с SELECT? | Конфликт с другими DML? |
|---|---|---|---|
| Без CONCURRENTLY | `ACCESS EXCLUSIVE` | Да, блокирует | Да, блокирует всё |
| С CONCURRENTLY | `SHARE UPDATE EXCLUSIVE` | Нет, SELECT продолжается | Конфликтует с `SHARE UPDATE EXCLUSIVE` и выше |

`SHARE UPDATE EXCLUSIVE` совместим с `ACCESS SHARE` (`SELECT`), поэтому читатели видят старые данные без задержки до момента завершения обновления — затем атомарно переключаются на новые. Ограничение: одновременно может выполняться только **один** `REFRESH CONCURRENTLY` для конкретного представления.

**Накладные расходы CONCURRENTLY**

По сравнению с обычным `REFRESH`:

1. **Дополнительная временная таблица**: полная копия нового содержимого хранится в памяти/на диске до завершения.
2. **FULL OUTER JOIN**: вычислительно дорогая операция, особенно при большом количестве строк.
3. **Два прохода по данным**: сначала вычисление, затем применение diff — вместо одного прохода (удалить всё, вставить новое).
4. **Более длинная транзакция**: держит snapshot дольше, увеличивая риск конфликтов при высокой конкурентности.

Практическое правило: `CONCURRENTLY` **выгоден**, когда между обновлениями меняется небольшой процент строк. При полном пересчёте (изменяется большинство строк) обычный `REFRESH` быстрее и эффективнее по ресурсам, несмотря на блокировку.
### Вопрос 44. Что такое deadlock detection в PostgreSQL? Как работает алгоритм обнаружения дедлока (граф ожидания), какой параметр управляет задержкой перед проверкой, и как правильно интерпретировать сообщение об ошибке дедлока в логах — что включить в журнал и как найти виновную транзакцию?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 19.12. Lock Management](https://www.postgresql.org/docs/current/runtime-config-locks.html)

**Параметр deadlock_timeout.** Определяет, сколько времени процесс ожидает получения блокировки, прежде чем PostgreSQL запустит проверку на дедлок. По умолчанию — 1 секунда (1s). Если значение задано без единиц, оно интерпретируется как миллисекунды.

**Принцип «оптимистичного ожидания».** PostgreSQL исходит из того, что дедлоки в реальных приложениях редки. Когда процесс не может немедленно получить блокировку, он засыпает и устанавливает таймер на deadlock_timeout миллисекунд. По истечении этого времени, если блокировка так и не была предоставлена, запускается алгоритм обнаружения дедлока. Такой подход снижает накладные расходы: дорогостоящая проверка не выполняется при каждом конфликте блокировок.

**Взаимодействие с log_lock_waits.** Когда параметр log_lock_waits включён, deadlock_timeout также определяет пороговое время, после которого в журнал выводится сообщение о том, что процесс ждёт блокировку дольше заданного интервала. Для диагностики задержек, связанных с блокировками, рекомендуется временно уменьшить deadlock_timeout ниже обычного рабочего значения.

**Рекомендации по настройке.** На сильно нагруженных серверах значение следует увеличить, чтобы избежать избыточных проверок. В идеале deadlock_timeout должен превышать типичное время выполнения транзакции: тогда большинство конфликтов разрешится само по себе до начала проверки.

#### Ответ от Claude

**Что такое дедлок в PostgreSQL.** Дедлок (взаимная блокировка) возникает, когда два или более процесса ждут блокировок, которые удерживают друг у друга, образуя циклическую зависимость. Ни один из них не может продолжить работу без освобождения ресурса другим.

**Алгоритм обнаружения — граф ожидания (wait-for graph, WFG).** PostgreSQL моделирует ожидание блокировок в виде ориентированного графа: вершина — процесс (бэкенд), ребро A → B означает, что процесс A ожидает блокировку, которую удерживает процесс B. Дедлок существует тогда и только тогда, когда в этом графе есть цикл. Ключевая функция в исходном коде — `FindLockCycle()` в файле `deadlock.c`: она рекурсивно обходит рёбра графа ожидания от стартового процесса наружу. Если обнаруживается цикл, функция возвращает `true` и попутно собирает список «мягких» рёбер (soft edges) — конфликтов, которые можно устранить переупорядочиванием очереди ожидания без отмены транзакции. Если такое переупорядочивание устраняет цикл, не порождая новых, дедлок разрешается без прерывания; иначе одна из транзакций выбирается жертвой и откатывается. Выбор жертвы непредсказуем и не задокументирован как детерминированный — на него не следует рассчитывать в логике приложения.

**Параметр deadlock_timeout.** Алгоритм запускается не сразу при возникновении конфликта, а после паузы, равной значению параметра `deadlock_timeout` (по умолчанию 1 с). Оптимистичный подход: большинство блокировок снимается быстро, и дорогостоящая проверка не нужна. Изменить значение может суперпользователь или пользователь с привилегией `SET`.

**Как выглядит сообщение в логах.** При обнаружении дедлока PostgreSQL генерирует запись уровня ERROR с кодом SQLSTATE `40P01`:

```
2024-05-15 12:34:52 UTC [3098] ERROR:  deadlock detected
2024-05-15 12:34:52 UTC [3098] DETAIL: Process 3098 waits for ShareLock on transaction 891717;
    blocked by process 3099.
    Process 3099 waits for ShareLock on transaction 891718; blocked by process 3098.
2024-05-15 12:34:52 UTC [3098] HINT:  See server log for query details.
2024-05-15 12:34:52 UTC [3098] CONTEXT: while updating tuple (0,1) in relation "accounts"
```

Секция DETAIL описывает цикл: каждый из участвующих процессов, тип запрашиваемой им блокировки и кто его блокирует. Секция CONTEXT указывает конкретную операцию и отношение. После HINT в лог попадают тексты запросов, выполнявшихся каждым процессом.

**Что включить для диагностики.**

```sql
-- в postgresql.conf или через ALTER SYSTEM:
log_lock_waits = on          -- логировать блокировки дольше deadlock_timeout
log_min_messages = 'WARNING' -- по умолчанию достаточно для ERROR-сообщений
log_line_prefix = '%t [%p] %q%u@%d '  -- время, PID, пользователь, база
deadlock_timeout = '500ms'   -- уменьшить для более быстрого обнаружения в dev
```

Дополнительно `log_min_error_statement = 'ERROR'` и `log_statements = 'all'` (только в dev) помогут собрать полный контекст запросов. В продакшене достаточно `log_lock_waits = on` с умеренным deadlock_timeout.

**Как найти виновную транзакцию.** В логах нет понятия «виновный» — оба (или все) процесса в цикле равно ответственны. Транзакция, запись которой содержит `ERROR: deadlock detected`, является жертвой (её PostgreSQL откатил). Чтобы найти первопричину, нужно по PID из секции DETAIL найти соответствующие записи в том же журнале до момента дедлока и восстановить последовательность запросов каждого процесса. Инструменты: `pg_locks` (во время инцидента), расширение `pg_stat_activity`, а также `auto_explain` с `log_min_duration = 0` для захвата планов. Устранение: фиксировать порядок доступа к ресурсам во всех транзакциях (всегда обновлять строки в одном и том же порядке) и по возможности снизить длительность транзакций.

### Вопрос 45. Как PostgreSQL хранит и сжимает TOAST-данные методами pglz и lz4? Начиная с какой версии доступен lz4, как выбрать метод сжатия на уровне столбца и на уровне кластера, и в каких сценариях lz4 предпочтительнее pglz?

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: 66.2. TOAST](https://www.postgresql.org/docs/current/storage-toast.html)

**Что такое TOAST.** TOAST (The Oversized-Attribute Storage Technique) — механизм прозрачного хранения значений столбцов, превышающих размер страницы (обычно 8 КБ). Каждая таблица со столбцами переменной длины (`varlena`) получает связанную TOAST-таблицу. Максимальный логический размер TOAST-значения — 1 ГБ (2^30 - 1 байт).

**Триггер TOAST.** Когда строка превышает `TOAST_TUPLE_THRESHOLD` (обычно 2 КБ), PostgreSQL применяет стратегии компрессии/выноса до тех пор, пока строка не уменьшится ниже `TOAST_TUPLE_TARGET` (тоже по умолчанию 2 КБ, но настраивается через `ALTER TABLE ... SET (toast_tuple_target = N)`).

**Стратегии хранения столбца.**

| Стратегия | Сжатие | Вынос (out-of-line) | Описание |
|-----------|--------|---------------------|----------|
| PLAIN     | Нет    | Нет                 | Только для нетостуемых типов |
| EXTENDED  | Да     | Да                  | Значение по умолчанию для большинства тостуемых типов |
| EXTERNAL  | Нет    | Да                  | Быстрые substring-операции за счёт большего объёма |
| MAIN      | Да     | Только в крайнем случае | Предпочтительно сжатие, вынос — последнее средство |

**Метод сжатия.** Метод, которым сжато конкретное значение (inline или out-of-line), сохраняется в метаданных указателя (varlena datum): там же хранятся логический и физический размеры. Выбрать метод можно через опцию `COMPRESSION` при `CREATE TABLE` или `ALTER TABLE`. Для столбцов без явного указания PostgreSQL обращается к параметру `default_toast_compression` в момент вставки данных.

#### Ответ от Claude

**Механизм TOAST.** PostgreSQL не хранит значения крупнее страницы непосредственно в строке. Вместо этого большие `varlena`-значения (text, bytea, jsonb, массивы и т.д.) либо сжимаются inline (остаются в основной куче), либо выносятся в отдельную TOAST-таблицу (`pg_toast_<OID>`), либо и то и другое. Это происходит прозрачно для запросов.

**Метод pglz.** Встроенный алгоритм PostgreSQL, реализованный в `pg_lzcompress.c`. Является алгоритмом LZ-типа (Lempel-Ziv): ищет повторяющиеся последовательности байт и заменяет их обратными ссылками. Применяется с первых версий PostgreSQL, не требует внешних библиотек и доступен всегда. Обеспечивает несколько лучшую степень сжатия (средний коэффициент ~2.23), но значительно медленнее — в первую очередь по времени компрессии.

**Метод lz4.** Высокоскоростной алгоритм сжатия, ориентированный на пропускную способность, а не на максимальную степень сжатия. Введён в PostgreSQL 14 (2021 г.). Требует, чтобы сборка PostgreSQL была выполнена с флагом `--with-lz4` и чтобы в ОС присутствовала библиотека `liblz4`. Средний коэффициент сжатия ~2.07 — примерно на 7% хуже, чем у pglz по использованию диска, зато скорость компрессии в ≈5 раз выше, а декомпрессия практически не уступает работе с несжатыми данными.

**Сравнение методов по данным тестирования (PostgreSQL 14).**

| Показатель          | pglz         | lz4                    |
|---------------------|-------------|------------------------|
| Средний коэффициент сжатия | ~2.23       | ~2.07                  |
| Скорость INSERT    | Базовая (~4.5 с на 1000 строк по 1 МБ) | ~300 мс (в ~15 раз быстрее) |
| Скорость SELECT (декомпрессия) | Базовая | ~на 20% быстрее pglz |
| Нагрузка на CPU    | Высокая     | Низкая                 |

**Как выбрать метод сжатия на уровне столбца.**

```sql
-- При создании таблицы:
CREATE TABLE documents (
    id    bigint,
    body  text  COMPRESSION lz4,
    meta  jsonb COMPRESSION pglz
);

-- Для существующего столбца:
ALTER TABLE documents ALTER COLUMN body SET COMPRESSION lz4;
-- Внимание: смена метода применяется только к новым вставкам/обновлениям.
-- Уже сохранённые данные остаются в исходном формате.
```

Для просмотра текущего метода столбца: `SELECT attcompression FROM pg_attribute WHERE attrelid = 'documents'::regclass AND attname = 'body';`

**Как выбрать метод на уровне кластера.**

```sql
-- В postgresql.conf или через ALTER SYSTEM:
default_toast_compression = 'lz4'

-- Или для текущего сеанса:
SET default_toast_compression = 'lz4';
```

Значение по умолчанию — `pglz`. Параметр доступен начиная с PostgreSQL 14.

**Когда lz4 предпочтительнее pglz.**

1. **Высокая интенсивность записи (INSERT/UPDATE большими текстами/jsonb).** Компрессия lz4 в разы быстрее, что снижает задержки операций записи и нагрузку на CPU.
2. **Чтение-ориентированные нагрузки с частым доступом к TOAST-значениям.** Более быстрая декомпрессия снижает латентность SELECT.
3. **CPU является узким местом.** pglz заметно нагружает процессор; lz4 освобождает его для других задач.
4. **Данные с высокой степенью повторяемости (логи, JSON с шаблонными структурами).** Оба метода дают хорошее сжатие, но lz4 делает это быстрее.

**Когда pglz предпочтительнее.** Если дисковое пространство критично (экономия ~7%) и CPU-нагрузка компрессии не является проблемой — например, медленные аналитические вставки при жёстких ограничениях на объём хранилища.

**Расхождение между источниками.** Статья dbi-services показывает, что при одном и том же наборе данных lz4 даёт меньший итоговый размер TOAST-таблицы (4 МБ против 12 МБ у pglz), что противоречит официальному утверждению о лучшем коэффициенте сжатия у pglz. Это расхождение, вероятно, объясняется характером тестовых данных: на высокоповторяемых строках lz4 может сжимать лучше. На разнообразных реальных данных официальные цифры (pglz ~2.23, lz4 ~2.07) более репрезентативны.

### Вопрос 46. Чем логический бэкап (pg_dump / pg_restore) отличается от физического (pg_basebackup + WAL)? В каких ситуациях следует использовать каждый подход, каковы ограничения pg_dump по консистентности при наличии внешних зависимостей и как параллельный режим (-j N) влияет на скорость восстановления?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18: Chapter 25. Backup and Restore](https://www.postgresql.org/docs/current/backup.html)

PostgreSQL предлагает три основных подхода к резервному копированию.

**SQL-дамп (логический бэкап).** Утилита `pg_dump` создаёт файл со SQL-командами, которые при выполнении воспроизведут базу данных в том состоянии, в каком она была на момент начала дампа. Дамп внутренне консистентен: он представляет снимок базы на момент запуска `pg_dump`, который не блокирует других пользователей. Важное преимущество — переносимость: вывод `pg_dump` можно загрузить в более новую версию PostgreSQL, а также на машину с другой архитектурой (например, с 32-битной на 64-битную). Это единственный способ, работающий при переносе базы на другую аппаратную платформу.

**Ограничение при использовании pg_dumpall.** Хотя каждая отдельная база данных будет внутренне консистентна, снимки разных баз в рамках одного `pg_dumpall`-вызова не синхронизированы между собой.

**Физический бэкап (уровень файловой системы).** `pg_basebackup` делает точную побайтовую копию файлов кластера. Вместе с архивированием WAL это позволяет выполнять восстановление на произвольный момент времени (PITR). Физический бэкап строго привязан к версии и платформе PostgreSQL: восстановление невозможно на другую мажорную версию или другую архитектуру.

**Непрерывное архивирование и PITR.** Комбинация базового физического бэкапа с архивированием WAL-сегментов позволяет восстановить кластер в любую точку между бэкапами. Это наиболее мощный метод для крупных продакшен-систем.

#### Ответ от Claude

**Ключевые отличия по основным характеристикам.**

| Характеристика         | Логический (pg_dump / pg_restore) | Физический (pg_basebackup + WAL) |
|------------------------|-----------------------------------|----------------------------------|
| Гранулярность          | База, схема, таблица, объект      | Весь кластер целиком             |
| Формат                 | SQL-текст, custom, directory, tar | Бинарные файлы данных            |
| Переносимость          | Между версиями и платформами      | Только та же мажорная версия и архитектура |
| Скорость создания бэкапа | Медленнее (SQL-генерация)        | Быстрее (побайтовое копирование) |
| Скорость восстановления | Медленнее (выполнение SQL)       | Быстрее (прямое копирование файлов) |
| PITR                   | Нет                               | Да (с WAL-архивированием)        |
| Глобальные объекты (роли, tablespace) | Только через pg_dumpall | Включены автоматически           |
| Онлайн-бэкап           | Да                                | Да                               |
| Репликация/standby     | Нет                               | Да (pg_basebackup + -R)          |

**Консистентность pg_dump.** pg_dump использует MVCC-снимок базы в момент старта и поддерживает его на протяжении всего дампа. Это даёт внутреннюю консистентность единственной базы. Ограничения:

1. **Внешние зависимости (роли, tablespace).** pg_dump не включает роли и пространства таблиц — они входят только в `pg_dumpall`. Восстановление дампа на новый сервер может упасть, если необходимые роли не существуют: `pg_dump` не проверяет их наличие заранее.
2. **Выборочный дамп (-n схема или -t таблица).** При ограниченном дампе PostgreSQL не включает зависимые объекты автоматически — нет гарантии, что такой дамп можно восстановить в чистую базу самостоятельно (без объектов, от которых зависит схема/таблица).
3. **Foreign tables (--include-foreign-data).** При включении данных из внешних таблиц pg_dump не проверяет, что внешняя таблица доступна для записи, и не гарантирует успешность восстановления.
4. **Многобазовая консистентность.** `pg_dumpall` вызывает `pg_dump` для каждой базы последовательно; снимки разных баз не синхронизированы, то есть кросс-базовые зависимости могут оказаться в несогласованном состоянии.
5. **Параллельный дамп и эксклюзивные блокировки.** При `pg_dump -j N` (-Fd формат) открываются N+1 соединений и используются синхронизированные снимки (с PostgreSQL 9.2 для primary, с 10 для standby). Если в процессе дампа другой клиент запрашивает эксклюзивную блокировку на таблицу, воркер pg_dump попытается получить разделяемую блокировку с опцией `NOWAIT` — и при неудаче весь дамп завершится с ошибкой.

**Параллельный режим pg_dump -j N.** Опция `-j` / `--jobs` включает параллельный дамп — только с форматом directory (`-Fd`):

```bash
pg_dump -j 4 -Fd mydb -f /backup/mydb_dir
```

Параллельный режим ускоряет создание дампа за счёт одновременной выгрузки нескольких таблиц, однако увеличивает нагрузку на сервер (N+1 соединений).

**Параллельное восстановление pg_restore -j N.** Параллельное восстановление работает с форматами custom (`-Fc`) и directory (`-Fd`) независимо от того, создавался ли дамп параллельно:

```bash
pg_restore -j 4 -Fd /backup/mydb_dir -d mydb
```

Восстановление таблиц ведётся параллельно, что существенно ускоряет процесс на многоядерных машинах — особенно для баз с большим числом крупных таблиц. При восстановлении также открываются N+1 соединений.

**Когда использовать логический бэкап.**

- Перенос базы на новую мажорную версию PostgreSQL.
- Перенос на другую OS или аппаратную платформу.
- Избирательное резервирование конкретных схем или таблиц.
- Создание человекочитаемого или переносимого архива.
- Небольшие и средние базы, где время восстановления не критично.

**Когда использовать физический бэкап.**

- Крупные продакшен-базы, где требуется минимальный RTO.
- Необходимость PITR (восстановление на произвольный момент времени).
- Настройка потоковой репликации и standby-серверов.
- Disaster recovery с минимальным временем восстановления.
- Инкрементальные бэкапы (pg_basebackup --incremental, доступно с PostgreSQL 17).

### Вопрос 47. Что такое генерируемые столбцы (generated columns) в PostgreSQL (GENERATED ALWAYS AS ... STORED)? Чем STORED отличается от VIRTUAL (и почему PostgreSQL пока поддерживает только STORED)? Какие ограничения накладываются на выражение генерации?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 17 — Generated Columns](https://www.postgresql.org/docs/17/ddl-generated-columns.html)

Генерируемый столбец — это специальный столбец таблицы, значение которого всегда вычисляется из других столбцов той же строки. Он похож на представление (view), но для отдельного столбца.

**Синтаксис:**

```sql
CREATE TABLE people (
    height_cm numeric,
    height_in numeric GENERATED ALWAYS AS (height_cm / 2.54) STORED
);
```

**STORED vs VIRTUAL:**

- **STORED**: значение вычисляется при записи (INSERT/UPDATE) и физически хранится на диске, как обычный столбец.
- **VIRTUAL**: значение вычисляется при чтении (SELECT), не занимает места на диске.

PostgreSQL 17 реализует **только STORED** генерируемые столбцы. Поддержка VIRTUAL была запланирована, но на момент выхода PostgreSQL 12 (когда появился механизм) реализация VIRTUAL была признана незрелой и отложена. VIRTUAL добавлен в PostgreSQL 18.

**Ограничения на выражение генерации:**

1. Только **иммутабельные** функции — нельзя использовать `random()`, `now()`, `CURRENT_TIMESTAMP` и другие волатильные функции.
2. Нельзя ссылаться на другие генерируемые столбцы.
3. Нельзя использовать подзапросы или ссылаться на другие таблицы.
4. Нельзя ссылаться на системные столбцы (кроме `tableoid`).
5. Нельзя задавать DEFAULT или IDENTITY для генерируемого столбца.
6. Нельзя включать в ключ секционирования (partition key).
7. Значение нельзя явно указать при INSERT/UPDATE — только ключевое слово `DEFAULT`.

#### Ответ от Claude

**Что такое генерируемые столбцы**

Генерируемые столбцы (generated columns) появились в PostgreSQL 12. Это столбцы, значения которых автоматически вычисляются из других столбцов строки и **не могут быть установлены вручную**. По смыслу аналогичны вычисляемым столбцам в других СУБД (Oracle `VIRTUAL`, SQL Server `COMPUTED`).

```sql
CREATE TABLE products (
    price_usd numeric(10,2),
    vat_rate  numeric(5,4) DEFAULT 0.20,
    price_with_vat numeric(10,2)
        GENERATED ALWAYS AS (price_usd * (1 + vat_rate)) STORED
);
```

**STORED vs VIRTUAL**

| Параметр | STORED | VIRTUAL |
|----------|--------|---------|
| Вычисление | При INSERT/UPDATE (запись) | При SELECT (чтение) |
| Хранение на диске | Да, как обычный столбец | Нет |
| Аналогия | Материализованное представление | Обычное представление |
| Поддержка в PG | С версии 12 | С версии 18 |

**Почему PostgreSQL долго поддерживал только STORED**

При разработке PostgreSQL 12 (2019) оба варианта были рассмотрены. Реализация VIRTUAL потребовала значительно большего объёма изменений в планировщике и исполнителе запросов — в частности, корректной обработки виртуальных столбцов при обновлениях, триггерах, репликации и ряде других ситуаций. Было принято решение выпустить только STORED (более простой и предсказуемый в поведении вариант), отложив VIRTUAL до более поздней версии. VIRTUAL был реализован и добавлен в PostgreSQL 18 (2025).

**Ограничения на выражение генерации**

Для обоих типов (STORED и VIRTUAL):

1. **Только иммутабельные функции** — `random()`, `now()`, `CURRENT_TIMESTAMP`, `clock_timestamp()` запрещены, так как значение должно быть воспроизводимым.
2. **Нет ссылок на другие генерируемые столбцы** — цепочки зависимостей не допускаются.
3. **Нет подзапросов** — нельзя обращаться к другим таблицам.
4. **Нет ссылок на системные столбцы** кроме `tableoid`.
5. **Нет DEFAULT и IDENTITY** — генерируемый столбец не может иметь значение по умолчанию.
6. **Нет в ключе секционирования** — нельзя использовать в PARTITION BY.

Дополнительные ограничения только для VIRTUAL (PostgreSQL 18+):

7. Нельзя использовать пользовательские функции и типы (только встроенные в PostgreSQL).

**Особенности использования:**

- При INSERT/UPDATE генерируемый столбец пропускается или указывается как `DEFAULT`:
  ```sql
  INSERT INTO products (price_usd, vat_rate) VALUES (100.00, 0.20);
  -- price_with_vat = 120.00 вычислится автоматически
  ```
- STORED-столбцы можно индексировать, использовать в ограничениях CHECK и внешних ключах.
- STORED-столбцы участвуют в логической репликации начиная с PostgreSQL 18.

### Вопрос 48. Что такое range-типы в PostgreSQL (int4range, tsrange, daterange и др.)? Зачем нужны exclusion constraints с EXCLUDE USING gist и как они решают задачу запрета пересекающихся бронирований? Какое расширение требуется для смешения скалярных и range-операторов в одном EXCLUDE?

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — Range Types](https://www.postgresql.org/docs/current/rangetypes.html)

Range-типы представляют диапазон значений некоторого базового типа (subtype). Они полезны для хранения промежутков — временных интервалов, числовых диапазонов, диапазонов дат.

**Встроенные range-типы:**

| Тип | Базовый тип |
|-----|-------------|
| `int4range` | integer |
| `int8range` | bigint |
| `numrange` | numeric |
| `tsrange` | timestamp without time zone |
| `tstzrange` | timestamp with time zone |
| `daterange` | date |

Границы диапазона задаются скобками: `[` и `]` — включительно, `(` и `)` — исключительно:

```sql
SELECT '[2024-01-01, 2024-01-10)'::daterange;
-- включает 2024-01-01, не включает 2024-01-10
```

**Exclusion constraints для запрета пересечений:**

Ограничение исключения с GIST-индексом позволяет декларативно запретить пересекающиеся диапазоны. Оператор `&&` проверяет пересечение:

```sql
CREATE TABLE reservation (
    during tsrange,
    EXCLUDE USING GIST (during WITH &&)
);
```

Любая попытка вставить строку с пересекающимся диапазоном завершится ошибкой.

**Расширение btree_gist для смешения операторов:**

Стандартный GIST-индекс поддерживает только range-операторы. Для объединения в одном EXCLUDE скалярного равенства (`=` по `room`) и range-пересечения (`&&` по `during`) требуется расширение `btree_gist`:

```sql
CREATE EXTENSION btree_gist;

CREATE TABLE room_reservation (
    room    text,
    during  tsrange,
    EXCLUDE USING GIST (room WITH =, during WITH &&)
);
```

Это позволяет запретить пересечения только для одной и той же комнаты, разрешив одновременные бронирования разных комнат.

#### Ответ от Claude

**Что такое range-типы**

Range-типы (диапазонные типы) — встроенный механизм PostgreSQL для хранения непрерывных диапазонов значений. Появились в PostgreSQL 9.2. Каждый range-тип строится на базовом типе (subtype) и хранит пару границ с метаданными об их включённости.

Встроенные range-типы:

| Тип | Базовый тип | Мультидиапазон (PG 14+) |
|-----|-------------|------------------------|
| `int4range` | `integer` | `int4multirange` |
| `int8range` | `bigint` | `int8multirange` |
| `numrange` | `numeric` | `nummultirange` |
| `tsrange` | `timestamp` (без TZ) | `tsmultirange` |
| `tstzrange` | `timestamp with TZ` | `tstzmultirange` |
| `daterange` | `date` | `datemultirange` |

**Синтаксис и операторы**

```sql
-- Включительная нижняя граница, исключительная верхняя (типичный вариант)
SELECT '[2024-01-01,2024-01-10)'::daterange;

-- Проверка пересечения (&&), вхождения (@>, <@), смежности (-|-)
SELECT '[1,5)'::int4range && '[3,8)'::int4range;  -- true
SELECT '[1,5)'::int4range @> 3;                   -- true (содержит элемент)
```

**Exclusion constraints и запрет пересекающихся бронирований**

Ограничение исключения (`EXCLUSION CONSTRAINT`) — обобщение уникального ограничения. Уникальное ограничение запрещает **равные** значения (`=`); ограничение исключения запрещает любую пару строк, для которых заданный оператор возвращает `true`.

Для range-типов оператор пересечения `&&` позволяет декларативно запретить перекрывающиеся диапазоны:

```sql
CREATE TABLE reservation (
    room_id integer,
    during  tsrange,
    EXCLUDE USING GIST (during WITH &&)
);

-- Успешная вставка:
INSERT INTO reservation VALUES (1, '[2024-06-01 10:00, 2024-06-01 12:00)');
-- Ошибка (пересечение с первой строкой):
INSERT INTO reservation VALUES (2, '[2024-06-01 11:00, 2024-06-01 13:00)');
-- ERROR: conflicting key value violates exclusion constraint
```

PostgreSQL реализует exclusion constraints через **GIST-индекс**, который поддерживает произвольные операторы с геометрической семантикой, — в отличие от B-tree (только равенство и порядок).

**Расширение btree_gist**

Стандартный GIST знает только операторы, специфичные для range- и геометрических типов. Скалярные типы (`integer`, `text`, `uuid` и т.д.) в GIST поддерживаются только через расширение `btree_gist`, которое добавляет для них GiST-классы операторов на основе B-tree-семантики.

Без `btree_gist` нельзя включить в один EXCLUDE и скалярный столбец (`=`), и range-столбец (`&&`).

```sql
CREATE EXTENSION btree_gist;

CREATE TABLE room_reservation (
    room   text,
    during tsrange,
    EXCLUDE USING GIST (room WITH =, during WITH &&)
);

-- Для одной комнаты:
INSERT INTO room_reservation VALUES ('101A', '[2024-06-01 10:00, 2024-06-01 12:00)');
-- Та же комната, пересечение -> ошибка:
INSERT INTO room_reservation VALUES ('101A', '[2024-06-01 11:00, 2024-06-01 13:00)');
-- ERROR: conflicting key value violates exclusion constraint

-- Другая комната, то же время -> успех:
INSERT INTO room_reservation VALUES ('102B', '[2024-06-01 11:00, 2024-06-01 13:00)');
-- INSERT 0 1
```

**Почему GIST, а не B-tree или UNIQUE**

- `UNIQUE` работает только с оператором `=`. Для диапазонов нужен оператор `&&` (пересечение), который не имеет смысла в B-tree.
- GIST поддерживает произвольные операторы и многомерные условия, что делает его подходящим для range-, геометрических и полнотекстовых типов.
- `btree_gist` — «мост»: оборачивает B-tree-операторы скалярных типов в GiST-интерфейс, позволяя смешивать их с range-операторами в одном составном GIST-индексе.

**Итог: паттерн бронирования без пересечений**

```sql
CREATE EXTENSION btree_gist;

CREATE TABLE bookings (
    resource_id integer NOT NULL,
    period      tstzrange NOT NULL,
    EXCLUDE USING GIST (resource_id WITH =, period WITH &&)
);
```

Этот единственный DDL-оператор гарантирует на уровне базы данных, что для одного ресурса не может быть двух пересекающихся интервалов бронирования.
### Вопрос 49. Как работают массивы в PostgreSQL и функция unnest()? Приведи примеры использования unnest() в SELECT и в FROM. Чем unnest() с несколькими аргументами (PostgreSQL 9.4+) отличается от раздельных вызовов?

**Категория:** PostgreSQL · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation — Array Functions and Operators](https://www.postgresql.org/docs/current/functions-array.html)

Функция `unnest()` разворачивает массив в набор строк. Элементы возвращаются в порядке хранения.

**Форма с одним аргументом:**
```sql
unnest(anyarray) → setof anyelement
```
```sql
SELECT unnest(ARRAY[1, 2, 3]);
-- 1
-- 2
-- 3
```

Многомерный массив разворачивается построчно:
```sql
SELECT unnest(ARRAY[['foo','bar'],['baz','quux']]);
-- foo
-- bar
-- baz
-- quux
```

**Форма с несколькими аргументами (PostgreSQL 9.4+):**
```sql
unnest(anyarray, anyarray [, ...]) → setof anyelement, anyelement [, ...]
```

Разворачивает несколько массивов параллельно, возвращая несколько столбцов. Если массивы разной длины, более короткие дополняются значениями `NULL`. Эта форма **допустима только в предложении FROM**.

```sql
SELECT * FROM unnest(ARRAY[1,2], ARRAY['foo','bar','baz']) AS x(a, b);
--  a |  b
-- ---+-----
--  1 | foo
--  2 | bar
--    | baz
```

#### Ответ от Claude

**Массивы в PostgreSQL** — это тип данных, позволяющий хранить упорядоченную коллекцию значений одного типа в одном столбце. Объявляются с квадратными скобками: `INTEGER[]`, `TEXT[]`.

**Функция `unnest()`** разворачивает массив в реляционный набор строк (тип `setof`).

---

**Использование в SELECT (одиночный аргумент):**

Работает как SRF (set-returning function) — возвращает по одной строке на каждый элемент:

```sql
SELECT unnest(ARRAY['a', 'b', 'c']) AS elem;
-- elem
-- ----
-- a
-- b
-- c
```

Вызов SRF прямо в `SELECT` (а не в `FROM`) является особенностью PostgreSQL. Начиная с PostgreSQL 10 такой вызов неявно трактуется как lateral-подзапрос и технически эквивалентен `FROM unnest(...)`.

---

**Использование в FROM:**

Каноническая форма — вызов в `FROM` с псевдонимом колонки:

```sql
SELECT elem, ordinality
FROM unnest(ARRAY['x','y','z']) WITH ORDINALITY AS t(elem, ordinality);
--  elem | ordinality
-- ------+------------
--  x    |          1
--  y    |          2
--  z    |          3
```

`WITH ORDINALITY` добавляет дополнительный столбец `bigint` с порядковым номером строки (начиная с 1).

---

**Отличие `unnest()` с несколькими аргументами от раздельных вызовов:**

| Поведение | `unnest(a, b)` | `unnest(a)` + `unnest(b)` раздельно |
|---|---|---|
| Выравнивание строк | Параллельно (по позиции) | Независимо |
| Разная длина массивов | Короткие дополняются NULL | Декартово произведение |
| Место вызова | Только в `FROM` | Можно в `SELECT` |

**Раздельные вызовы в SELECT дают декартово произведение** (N * M строк):
```sql
-- НЕПРАВИЛЬНО для параллельного разворачивания:
SELECT unnest(ARRAY[1,2,3]), unnest(ARRAY['a','b']);
-- 1, a
-- 1, b
-- 2, a
-- 2, b
-- 3, a
-- 3, b  -- 6 строк вместо 3!
```

**`unnest()` с несколькими аргументами выравнивает по позиции** (аналог `zip`):
```sql
-- ПРАВИЛЬНО:
SELECT * FROM unnest(ARRAY[1,2,3], ARRAY['a','b']) AS t(num, letter);
--  num | letter
-- -----+--------
--    1 | a
--    2 | b
--    3 |        <- NULL, т.к. второй массив короче
```

Внутренне PostgreSQL 9.4+ преобразует `unnest(a, b)` в конструкцию `ROWS FROM(unnest(a), unnest(b))`, которая явно комбинирует результаты параллельно с дополнением NULL для более коротких массивов (документировано в разделе [7.2.1.4 Table Functions](https://www.postgresql.org/docs/current/queries-table-expressions.html)).

### Вопрос 50. Что делает SELECT DISTINCT ON в PostgreSQL и чем он отличается от обычного SELECT DISTINCT? Почему ORDER BY обязателен при использовании DISTINCT ON и как правильно формулировать запрос «последняя запись на каждый ключ»?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation — SELECT](https://www.postgresql.org/docs/current/sql-select.html)

`SELECT DISTINCT ON (expression [, ...])` сохраняет только первую строку из каждой группы строк, где указанные выражения принимают одинаковые значения. Выражения `DISTINCT ON` интерпретируются по тем же правилам, что и в `ORDER BY`. Обратите внимание, что «первая строка» каждого набора непредсказуема, если не используется `ORDER BY` для обеспечения нужного порядка. Например:

```sql
SELECT DISTINCT ON (location) location, time, report
    FROM weather_reports
    ORDER BY location, time DESC;
```

Запрос получает самый последний отчёт о погоде для каждой локации. Если бы `ORDER BY` не использовался, мы бы получали случайный отчёт для каждой локации.

Выражения `DISTINCT ON` должны совпадать с крайними левыми выражениями `ORDER BY`. Предложение `ORDER BY` обычно содержит дополнительные выражения, определяющие желаемый приоритет строк внутри каждой группы `DISTINCT ON`.

#### Ответ от Claude

**`SELECT DISTINCT`** удаляет полностью одинаковые строки из результата — то есть строки, где совпадают значения **всех** выбранных столбцов. Это SQL-стандарт.

**`SELECT DISTINCT ON (col1, col2, ...)`** — расширение PostgreSQL. Он убирает дубликаты только по **указанным выражениям** и возвращает по одной строке на каждое уникальное значение этого выражения, оставляя **первую** строку из каждой группы (согласно `ORDER BY`).

| Сравнение | `SELECT DISTINCT` | `SELECT DISTINCT ON (...)` |
|---|---|---|
| Стандарт SQL | Да | Нет (PostgreSQL-расширение) |
| Базис уникальности | Все выбранные столбцы | Только указанные выражения |
| Поведение | Убирает полностью одинаковые строки | Выбирает по одной строке на группу |

---

**Почему `ORDER BY` обязателен:**

`DISTINCT ON` оставляет «первую» строку из каждой группы. Без `ORDER BY` понятие «первая» не определено — PostgreSQL может вернуть любую строку из группы в произвольном порядке (зависит от плана запроса, физического расположения данных, версии). Результат будет **нестабильным** и непредсказуемым.

Технически PostgreSQL требует, чтобы **выражения в `DISTINCT ON` совпадали с крайними левыми выражениями `ORDER BY`**. Это позволяет движку сначала отсортировать все строки, а затем взять первую строку из каждой уже упорядоченной группы.

Если написать `ORDER BY` без столбца из `DISTINCT ON` первым — запрос завершится ошибкой:
```sql
-- Ошибка: выражения DISTINCT ON должны совпадать с начальными выражениями ORDER BY
SELECT DISTINCT ON (location) location, time
FROM weather_reports
ORDER BY time DESC;  -- location не первым!
```

---

**Канонический запрос «последняя запись на каждый ключ»:**

```sql
SELECT DISTINCT ON (location)
    location, time, report
FROM weather_reports
ORDER BY location, time DESC;
```

Логика:
1. `DISTINCT ON (location)` — группировать по `location`.
2. `ORDER BY location, time DESC` — внутри каждой группы сортировать по убыванию времени.
3. `DISTINCT ON` берёт **первую** строку из каждой группы — то есть запись с максимальным `time`.

Результат — ровно одна (самая свежая) запись для каждого значения `location`.

---

**Особенности и подводные камни:**

- Столбцы, перечисленные в `DISTINCT ON`, **не обязаны** присутствовать в `SELECT`-списке (хотя обычно включаются).
- Дополнительные выражения в `ORDER BY` после столбцов из `DISTINCT ON` определяют, какая строка будет «первой» внутри каждой группы, и не влияют на группировку.
- По производительности `DISTINCT ON` часто эффективнее, чем `ROW_NUMBER() OVER (PARTITION BY ... ORDER BY ...)`, потому что не требует вычисления номеров для всех строк — но сравнение зависит от индексов и объёма данных.

### Вопрос 51. Чем материализованное представление (MATERIALIZED VIEW) отличается от обычного представления (VIEW)? Как работает REFRESH MATERIALIZED VIEW CONCURRENTLY, какой предварительный шаг необходим для его использования и почему обычный REFRESH блокирует чтение?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — REFRESH MATERIALIZED VIEW](https://www.postgresql.org/docs/current/sql-refreshmaterializedview.html)

Обычное представление (VIEW) является виртуальной таблицей: при каждом обращении PostgreSQL заново выполняет запрос, лежащий в его основе. Данные не хранятся физически. Материализованное представление (MATERIALIZED VIEW), напротив, сохраняет результат запроса в физическом хранилище (аналог таблицы). При обращении к нему данные читаются напрямую из сохранённого снимка, не выполняя базовый запрос повторно.

Команда `REFRESH MATERIALIZED VIEW` полностью заменяет содержимое материализованного представления: старые данные отбрасываются и заменяются свежими — результатом нового выполнения базового запроса.

**Обычный REFRESH** захватывает эксклюзивную блокировку на представление на всё время обновления. Любые параллельные `SELECT`-запросы к этому представлению блокируются до завершения операции.

**REFRESH MATERIALIZED VIEW CONCURRENTLY** обновляет представление без блокировки параллельных чтений: во время обновления `SELECT` продолжают работать, возвращая старые данные. Технически PostgreSQL создаёт новый снимок, вычисляет diff со старыми данными и применяет изменения строка за строкой, не удерживая блокировку чтения.

**Обязательный предварительный шаг** — наличие хотя бы одного `UNIQUE`-индекса на материализованном представлении. Индекс должен:
- использовать только имена столбцов (не выражения);
- не содержать предикат `WHERE` (то есть покрывать все строки).

Кроме того, представление должно быть уже заполнено (не находиться в состоянии `WITH NO DATA`). `CONCURRENTLY` нельзя комбинировать с `WITH NO DATA`.

Пример:
```sql
CREATE UNIQUE INDEX sales_summary_seller
  ON sales_summary (seller_no, invoice_date);

REFRESH MATERIALIZED VIEW CONCURRENTLY sales_summary;
```

#### Ответ от Claude

**VIEW vs MATERIALIZED VIEW — ключевые отличия**

| Аспект | VIEW | MATERIALIZED VIEW |
|---|---|---|
| Хранение данных | Нет (только запрос) | Физическое (как таблица) |
| Актуальность данных | Всегда свежие | Снимок на момент последнего REFRESH |
| Производительность чтения | Зависит от сложности запроса | Быстро (чтение из таблицы), поддерживает индексы |
| Обновление | Автоматически при каждом запросе | Вручную через REFRESH |
| Прямые UPDATE/DELETE | Не поддерживается (без правил) | Не поддерживается |

**Как работает обычный REFRESH**

`REFRESH MATERIALIZED VIEW name` захватывает `ExclusiveLock` на представление. Все параллельные `SELECT` ждут освобождения блокировки. Преимущество — операция быстрее при массовом обновлении, использует меньше ресурсов.

**Как работает REFRESH MATERIALIZED VIEW CONCURRENTLY**

Алгоритм:
1. PostgreSQL выполняет базовый запрос и строит новый снимок данных во временной структуре.
2. Сравнивает новый снимок со старым содержимым, используя уникальный индекс как ключ сравнения.
3. Применяет дельту: вставляет новые строки, обновляет изменившиеся, удаляет исчезнувшие.
4. На всё это время удерживает только `ShareUpdateExclusiveLock` — он не блокирует параллельные `SELECT`.

Следствие: пока идёт обновление, читатели получают старые данные. Это допустимо в большинстве аналитических сценариев.

**Предварительный шаг: UNIQUE-индекс**

Без уникального индекса PostgreSQL не может безопасно идентифицировать строки при вычислении diff, поэтому `CONCURRENTLY` требует наличия хотя бы одного индекса по столбцам (не выражению, без `WHERE`). Если индекса нет, команда завершится ошибкой.

**Почему обычный REFRESH блокирует чтение**

Обычный refresh перезаписывает данные представления целиком (truncate + insert). PostgreSQL удерживает `ExclusiveLock`, чтобы читатели не увидели промежуточное (пустое) состояние. В отличие от этого, `CONCURRENTLY` применяет изменения инкрементально, сохраняя согласованность для параллельных читателей за счёт дополнительного overhead на вычисление diff.

**Дополнительные ограничения CONCURRENTLY:**
- Одновременно допустим только один `REFRESH CONCURRENTLY` на одно представление.
- Нельзя использовать совместно с `WITH NO DATA`.
- Медленнее обычного REFRESH при большом объёме изменений.

### Вопрос 52. Что такое Row Level Security (RLS) в PostgreSQL? Как включить защиту на уровне строк, что делают предикаты USING и WITH CHECK, чем PERMISSIVE-политика отличается от RESTRICTIVE и как суперпользователь обходит RLS?

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 5.9. Row Security Policies](https://www.postgresql.org/docs/current/ddl-rowsecurity.html)

Таблицы могут иметь политики безопасности строк (row security policies), которые на уровне отдельных пользователей ограничивают доступные для запросов, вставки, обновления и удаления строки. Эта функция также известна как Row-Level Security (RLS).

Когда для таблицы включена защита на уровне строк (`ALTER TABLE ... ENABLE ROW LEVEL SECURITY`), любой обычный доступ к таблице для выборки или изменения строк должен быть явно разрешён политикой. Если для таблицы нет ни одной политики, применяется политика «запретить всё» по умолчанию: ни одна строка не видна и не может быть изменена.

Политики создаются командой `CREATE POLICY`, изменяются командой `ALTER POLICY` и удаляются командой `DROP POLICY`. Чтобы созданные политики применялись, RLS должна быть включена на таблице.

**Предикат USING** управляет тем, какие строки видны запросам. **WITH CHECK** управляет тем, какие строки могут быть вставлены или обновлены командами DML. Если указан только `USING`, он применяется и к чтению, и к записи.

**PERMISSIVE (по умолчанию)**: несколько политик объединяются логикой **OR** — доступ разрешается, если хотя бы одна политика удовлетворена. **RESTRICTIVE**: политики объединяются логикой **AND** — строка должна пройти через все ограничивающие политики.

Суперпользователи и роли с атрибутом `BYPASSRLS` всегда обходят систему RLS. Владельцы таблиц тоже обычно обходят RLS, но могут быть подчинены ей через `ALTER TABLE ... FORCE ROW LEVEL SECURITY`.

#### Ответ от Claude

**Что такое Row Level Security**

RLS (Row Level Security) — механизм PostgreSQL, позволяющий ограничивать видимость строк на уровне отдельных пользователей или ролей. Дополняет стандартную систему привилегий SQL: даже если у пользователя есть права на таблицу, RLS может ограничить, какие именно строки он видит или изменяет.

**Включение и создание политик**

```sql
-- Шаг 1: включить RLS на таблице
ALTER TABLE accounts ENABLE ROW LEVEL SECURITY;

-- Шаг 2: создать политику
CREATE POLICY account_managers ON accounts
    TO managers
    USING (manager = current_user);
```

После включения RLS без единой политики ни одна строка не доступна (default-deny).

**Предикаты USING и WITH CHECK**

| Предикат | Когда применяется | Что проверяет |
|----------|------------------|---------------|
| `USING` | SELECT, UPDATE (какие строки видны), DELETE | Видимость строки |
| `WITH CHECK` | INSERT, UPDATE (какие значения допустимы) | Корректность новых данных |

Если задан только `USING`, он работает и для чтения, и для DML-операций:

```sql
-- Пользователь видит и может изменять только свои строки
CREATE POLICY user_policy ON documents
    USING (owner = current_user)
    WITH CHECK (owner = current_user);
```

**PERMISSIVE vs RESTRICTIVE**

```sql
-- PERMISSIVE (по умолчанию): OR-логика
CREATE POLICY all_view ON data FOR SELECT USING (true);
CREATE POLICY owner_view ON data FOR SELECT USING (owner = current_user);
-- Строка доступна, если ЛЮБАЯ политика разрешает

-- RESTRICTIVE: AND-логика
CREATE POLICY local_only ON data AS RESTRICTIVE
    USING (pg_catalog.inet_client_addr() IS NULL);
-- Строка доступна только если И разрешительные, И ограничивающие политики выполнены
```

Типичный паттерн: разрешительные политики задают базовый доступ; ограничивающие дополнительно сужают его (например, запрет удалённого доступа для администраторов).

**Обход RLS**

- Суперпользователи (`pg_roles.rolsuper = true`) всегда обходят RLS.
- Роли с атрибутом `BYPASSRLS`: `CREATE ROLE etl_user BYPASSRLS;`.
- Владельцы таблиц обходят RLS по умолчанию. Принудительно подчинить их: `ALTER TABLE accounts FORCE ROW LEVEL SECURITY;`.
- Параметр `SET row_security = off` — если политики ограничили бы результат, запрос завершается ошибкой (полезно для резервного копирования).

**Практический пример: мультитенантная таблица**

```sql
CREATE TABLE tenants_data (
    tenant_id integer NOT NULL,
    payload   text
);
ALTER TABLE tenants_data ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON tenants_data
    USING (tenant_id = current_setting('app.current_tenant')::integer)
    WITH CHECK (tenant_id = current_setting('app.current_tenant')::integer);
```

Приложение устанавливает `SET app.current_tenant = 42;` при подключении, и каждый тенант видит только свои строки.

### Вопрос 53. Что такое расширение pg_stat_statements? Как оно работает, какие метрики отслеживает и как используется для диагностики медленных запросов?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — F.32. pg_stat_statements](https://www.postgresql.org/docs/current/pgstatstatements.html)

Модуль `pg_stat_statements` предоставляет средства для отслеживания статистики планирования и выполнения всех SQL-операторов, выполняемых сервером. Модуль должен быть загружен путём добавления `pg_stat_statements` в `shared_preload_libraries` в `postgresql.conf`, поскольку для его работы требуется дополнительная разделяемая память; это означает, что для добавления или удаления модуля необходим перезапуск сервера.

Для активности модуля необходимо включить вычисление идентификатора запроса — это происходит автоматически при `compute_query_id = auto` или `on`.

Расширение отслеживает следующие ключевые метрики:

- `queryid` — хеш нормализованного текста запроса для идентификации аналогичных запросов.
- `calls` — число вызовов запроса.
- `total_exec_time`, `mean_exec_time`, `stddev_exec_time` — суммарное, среднее и стандартное отклонение времени выполнения.
- `rows` — суммарное число строк, возвращённых или затронутых.
- `shared_blks_hit`, `shared_blks_read` — обращения к кешу и к диску.
- При включённом `track_io_timing` — время ввода-вывода.

Для сброса накопленной статистики используется функция `pg_stat_statements_reset()`.

#### Ответ от Claude

**Что такое pg_stat_statements и как включить**

`pg_stat_statements` — стандартное расширение PostgreSQL, которое накапливает агрегированную статистику выполнения SQL-запросов. Константы в запросах нормализуются (заменяются на `$1`, `$2` и т.д.), поэтому запросы с разными значениями параметров считаются одним запросом.

```ini
# postgresql.conf
shared_preload_libraries = 'pg_stat_statements'
compute_query_id = auto
pg_stat_statements.max = 10000        # максимум хранимых запросов
pg_stat_statements.track = all        # top — только клиентские, all — включая вложенные
pg_stat_statements.track_planning = on # отслеживать время планирования
```

После перезапуска:
```sql
CREATE EXTENSION pg_stat_statements;
```

**Ключевые метрики**

| Поле | Что означает | Применение |
|------|--------------|------------|
| `queryid` | Хеш нормализованного запроса | Идентификация одинаковых запросов |
| `calls` | Число вызовов | Частота запроса |
| `total_exec_time` | Суммарное время выполнения (мс) | Общая нагрузка на систему |
| `mean_exec_time` | Среднее время выполнения | Средняя задержка |
| `stddev_exec_time` | Стандартное отклонение | Стабильность производительности |
| `rows` | Строки, возвращённые/затронутые | Объём данных |
| `shared_blks_hit` | Обращения к буферному кешу | Cache hit |
| `shared_blks_read` | Чтения с диска | Реальный I/O |
| `total_plan_time` | Суммарное время планирования | Дорогостоящее планирование |

**Типовые запросы для диагностики**

```sql
-- Топ-10 запросов по суммарному времени
SELECT query, calls, total_exec_time, mean_exec_time, rows
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;

-- Запросы с нестабильным временем выполнения (высокий stddev)
SELECT query, mean_exec_time, stddev_exec_time,
       stddev_exec_time / mean_exec_time AS cv
FROM pg_stat_statements
WHERE calls > 100
ORDER BY cv DESC LIMIT 10;

-- Cache hit ratio по запросам
SELECT query,
       shared_blks_hit::numeric /
       NULLIF(shared_blks_hit + shared_blks_read, 0) AS hit_ratio
FROM pg_stat_statements
ORDER BY hit_ratio ASC NULLS LAST LIMIT 10;

-- Сброс статистики
SELECT pg_stat_statements_reset();
```

**Ограничения**

- Текст запроса видят только суперпользователи и члены роли `pg_read_all_stats`; остальные видят текст только своих запросов.
- `queryid` нестабилен между мажорными версиями PostgreSQL.
- При переполнении (свыше `pg_stat_statements.max` записей) наименее используемые запросы вытесняются.

### Вопрос 54. Как работает autovacuum в PostgreSQL? Какие параметры определяют порог запуска VACUUM и ANALYZE, что такое cost-based delay и как переопределить настройки autovacuum на уровне отдельной таблицы?

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 24.1. Routine Vacuuming](https://www.postgresql.org/docs/current/routine-vacuuming.html)

Autovacuum — фоновый процесс PostgreSQL, автоматически запускающий `VACUUM` и `ANALYZE`. Он динамически реагирует на активность таблиц, предотвращая раздувание и проблему переполнения XID.

**Пороги запуска VACUUM:**

```
vacuum threshold = autovacuum_vacuum_threshold + autovacuum_vacuum_scale_factor × reltuples
```

Значения по умолчанию: `autovacuum_vacuum_threshold = 50`, `autovacuum_vacuum_scale_factor = 0.2`. Для таблицы из 10 000 строк порог составит `50 + 0.2 × 10 000 = 2 050` устаревших кортежей.

**Пороги запуска ANALYZE:**

```
analyze threshold = autovacuum_analyze_threshold + autovacuum_analyze_scale_factor × reltuples
```

По умолчанию: `autovacuum_analyze_threshold = 50`, `autovacuum_analyze_scale_factor = 0.1` (10% от размера таблицы).

**Cost-based delay** — механизм ограничения нагрузки autovacuum на I/O. Процесс отслеживает внутренний счётчик стоимости: при чтении страницы из кеша — `vacuum_cost_page_hit` (1), при чтении с диска — `vacuum_cost_page_miss` (10), при «загрязнении» страницы — `vacuum_cost_page_dirty` (20). Когда накопленная стоимость достигает `autovacuum_vacuum_cost_limit` (200), процесс засыпает на `autovacuum_vacuum_cost_delay` (20 мс), а счётчик сбрасывается.

Настройки autovacuum можно задать глобально в `postgresql.conf` и переопределить на уровне конкретной таблицы через параметры хранения (`storage parameters`).

#### Ответ от Claude

**Архитектура autovacuum**

Autovacuum состоит из двух процессов:
- **Launcher** — запускается раз в `autovacuum_naptime` (умолч. 1 мин), проверяет все базы данных и запускает воркеры там, где таблицы превысили пороги.
- **Workers** — выполняют непосредственно VACUUM и ANALYZE. Максимальное число воркеров: `autovacuum_max_workers` (умолч. 3).

Autovacuum всегда запускается принудительно (даже при `autovacuum = off`), если возраст `relfrozenxid` превышает `autovacuum_freeze_max_age` (умолч. 200 млн транзакций) — защита от wraparound.

**Формулы пороговых значений**

| Операция | Формула | Дефолтные значения |
|----------|---------|-------------------|
| VACUUM | `50 + 0.20 × N строк` | threshold=50, scale=0.20 |
| ANALYZE | `50 + 0.10 × N строк` | threshold=50, scale=0.10 |
| VACUUM insert-only | `1000 + 0.20 × N строк` | insert_threshold=1000, insert_scale=0.20 |

Для таблицы с 100 000 строк autovacuum VACUUM запустится при 20 050 мёртвых кортежах.

**Cost-based delay**

Механизм предотвращает монополизацию I/O autovacuum-ом:

```
Стоимость операций:
  vacuum_cost_page_hit   = 1   (страница в shared_buffers)
  vacuum_cost_page_miss  = 10  (чтение с диска)
  vacuum_cost_page_dirty = 20  (страница была «загрязнена»)

При достижении autovacuum_vacuum_cost_limit (умолч. 200):
  → sleep autovacuum_vacuum_cost_delay (умолч. 20 мс) → сброс счётчика
```

При нескольких параллельных воркерах лимит делится между ними, ограничивая суммарную нагрузку.

**Переопределение на уровне таблицы**

```sql
-- Агрессивная настройка для горячей таблицы с частыми UPDATE
ALTER TABLE hot_table SET (
    autovacuum_vacuum_scale_factor = 0.01,   -- 1% вместо 20%
    autovacuum_vacuum_threshold = 10,
    autovacuum_vacuum_cost_delay = 5,        -- 5 мс вместо 20 мс
    autovacuum_vacuum_cost_limit = 800       -- выше лимит = быстрее
);

-- «Заморозить» статичную таблицу на длительный срок
ALTER TABLE static_lookup SET (
    autovacuum_freeze_min_age = 0,
    autovacuum_vacuum_scale_factor = 0.5
);
```

**Диагностика**

```sql
-- Состояние таблиц относительно порогов autovacuum
SELECT relname,
       n_dead_tup,
       n_live_tup,
       last_autovacuum,
       last_autoanalyze
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC LIMIT 20;

-- Текущие воркеры autovacuum
SELECT pid, query, state, wait_event_type, wait_event
FROM pg_stat_activity
WHERE query LIKE 'autovacuum:%';
```

### Вопрос 55. Как работает логическая репликация (logical replication) в PostgreSQL? Чем она отличается от физической (streaming), что такое publication и subscription, какие объекты не реплицируются и какой уровень wal_level необходим?

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — Chapter 29. Logical Replication](https://www.postgresql.org/docs/current/logical-replication.html)

Логическая репликация — метод репликации объектов данных и их изменений на основе **идентификатора репликации** (обычно первичного ключа). Она отличается от физической репликации, при которой используется побайтовое копирование блоков.

Логическая репликация использует модель «публикация-подписка»:
- **Публикация (publication)** — логическая коллекция данных для репликации на стороне издателя.
- **Подписка (subscription)** — определяет подключение к другой базе данных и набор публикаций, на которые она подписана.

При старте репликации таблицы PostgreSQL делает снимок данных на издателе и копирует его подписчику. После завершения начальной синхронизации изменения передаются непрерывно. Подписчик применяет изменения в том же порядке, что и издатель, гарантируя транзакционную консистентность в рамках одной подписки.

Подписчик может сам выступать издателем для других баз данных, что позволяет строить каскадные схемы репликации.

#### Ответ от Claude

**Сравнение физической и логической репликации**

| Аспект | Физическая (streaming) | Логическая |
|--------|----------------------|------------|
| Единица передачи | Блоки данных (WAL) | Логические изменения (INSERT/UPDATE/DELETE) |
| Гранулярность | Весь кластер | Отдельные таблицы, схемы |
| Версия PostgreSQL | Только та же мажорная | Разные мажорные версии |
| Схема должна совпадать | Да (копируется) | Нет (только данные) |
| Репликация DDL | Не требуется (блоки) | Нет |
| Последовательности (sequences) | Да | Нет (только значения таблиц) |
| Только для чтения на реплике | Да (hot standby) | Нет (полноценное RW) |
| wal_level | `replica` | `logical` |

**Настройка публикации и подписки**

```sql
-- На издателе (publisher):
-- postgresql.conf: wal_level = logical
-- pg_hba.conf: разрешить подключение роли с REPLICATION

CREATE PUBLICATION my_pub FOR TABLE orders, customers;
-- или для всех таблиц:
CREATE PUBLICATION all_pub FOR ALL TABLES;

-- На подписчике (subscriber):
CREATE SUBSCRIPTION my_sub
    CONNECTION 'host=publisher_host dbname=mydb user=replication_user password=secret'
    PUBLICATION my_pub;
```

**Что реплицируется, а что нет**

Реплицируется:
- Операции `INSERT`, `UPDATE`, `DELETE` (при наличии replica identity)
- `TRUNCATE` (с PostgreSQL 11)

Не реплицируется:
- DDL-команды (`CREATE TABLE`, `ALTER TABLE` и т.д.) — схему на подписчике нужно создать вручную
- Последовательности (`SEQUENCE`) — значения не синхронизируются
- Большие объекты (large objects)
- Представления, материализованные представления
- Foreign tables

**Replica identity**

Для `UPDATE` и `DELETE` PostgreSQL должен уметь идентифицировать обновляемую строку. По умолчанию используется PRIMARY KEY. Если его нет:

```sql
-- Использовать конкретный уникальный индекс
ALTER TABLE my_table REPLICA IDENTITY USING INDEX my_unique_idx;

-- Использовать полную копию строки (дорого, но работает без PK)
ALTER TABLE my_table REPLICA IDENTITY FULL;
```

**Типичные сценарии применения**

- Перенос данных между разными мажорными версиями PostgreSQL (zero-downtime upgrade).
- Выборочная репликация отдельных таблиц на аналитический сервер.
- Мультимастер-топологии с двунаправленной репликацией (bidirectional replication).
- Консолидация данных из нескольких источников.

### Вопрос 56. Как настроить потоковую репликацию (streaming replication) в PostgreSQL? Объясни роль параметров wal_level, max_wal_senders, primary_conninfo, hot_standby и как обрабатываются конфликты запросов на standby.

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 19.6. Replication](https://www.postgresql.org/docs/current/runtime-config-replication.html)

**wal_level** — для потоковой репликации должен быть установлен в `replica` (или `logical`). Параметр статический — требует перезапуска сервера.

**max_wal_senders** — максимальное число одновременных подключений от standby-серверов и клиентов резервного копирования (по умолчанию 10). Для отключения репликации устанавливается в 0. Рекомендуется задавать немного выше реального числа standby, чтобы обеспечить возможность повторного подключения.

**wal_keep_size** — минимальный объём WAL-файлов (в МБ), сохраняемых в `pg_wal`. По умолчанию 0 — объём определяется только позицией checkpoint. Увеличение помогает избежать разрыва репликации, если standby отстаёт.

**primary_conninfo** — строка подключения на standby для установки соединения с primary. Включает host, port, user, password; требует рестарта WAL receiver при изменении.

**hot_standby** — при значении `on` (по умолчанию) разрешает выполнение запросов на standby в режиме восстановления. Статический параметр.

Если выполняемый на standby запрос конфликтует с применением WAL (например, VACUUM на primary удаляет строки, которые читает запрос на standby), standby ждёт до `max_standby_streaming_delay` (по умолчанию 30 с) перед принудительной отменой запроса.

#### Ответ от Claude

**Схема настройки потоковой репликации**

```
Primary                          Standby
─────────────────────────────    ───────────────────────────────
wal_level = replica              primary_conninfo = '...'
max_wal_senders = 5              hot_standby = on
wal_keep_size = 1024             (опционально: primary_slot_name)
```

**Роль ключевых параметров**

| Параметр | Где задаётся | Роль | Рестарт? |
|----------|-------------|------|----------|
| `wal_level = replica` | Primary | Включает WAL-данные для репликации | Да |
| `max_wal_senders` | Primary | Лимит параллельных WAL sender процессов | Да |
| `wal_keep_size` | Primary | Минимум WAL для отставших реплик | Нет |
| `synchronous_standby_names` | Primary | Имена синхронных standbys | Нет |
| `primary_conninfo` | Standby | Строка подключения к primary | Нет* |
| `hot_standby` | Standby | Разрешить SELECT на standby | Да |
| `hot_standby_feedback` | Standby | Сообщать primary о запросах standby | Нет |

*Сигнализирует WAL receiver о рестарте соединения

**Инициализация standby**

```bash
# Создать базовую копию primary на standby
pg_basebackup -h primary_host -U replication_user \
    -D /var/lib/postgresql/data \
    -P --wal-method=stream -R
    # -R: автоматически создаёт standby.signal и primary_conninfo
```

**Мониторинг репликации**

```sql
-- На primary: состояние WAL sender-ов
SELECT client_addr, state, sent_lsn, write_lsn, flush_lsn, replay_lsn,
       sent_lsn - replay_lsn AS replication_lag
FROM pg_stat_replication;

-- На standby: состояние WAL receiver
SELECT status, received_lsn, last_msg_send_time
FROM pg_stat_wal_receiver;
```

**Конфликты запросов на standby**

На standby происходят запросы (hot standby) одновременно с применением WAL с primary. Конфликт возникает, когда:
- Primary выполняет VACUUM и удаляет строки, нужные запросу на standby.
- Primary выполняет DDL на таблице, которую читает standby.
- Lock на standby несовместим с WAL-операцией.

Механизм разрешения:
1. Standby выждет до `max_standby_streaming_delay` (умолч. 30 с).
2. Если WAL не применён за это время, конфликтующий запрос отменяется с ошибкой `ERROR: canceling statement due to conflict with recovery`.
3. При `hot_standby_feedback = on` standby сообщает primary о выполняемых запросах — primary откладывает VACUUM, избегая конфликта, но ценой возможного раздувания таблицы на primary.

**Синхронная репликация**

```ini
# postgresql.conf на primary
synchronous_standby_names = 'FIRST 1 (standby1, standby2)'
# или кворумная:
synchronous_standby_names = 'ANY 2 (s1, s2, s3)'
```

Это гарантирует, что транзакция не считается подтверждённой до записи WAL как минимум на указанных standbys.

### Вопрос 57. Чем тип jsonb отличается от json в PostgreSQL? Какие операторы поддерживает jsonb, как работают GIN-индексы с классами операторов jsonb_ops и jsonb_path_ops и в каких случаях jsonb_path_ops предпочтительнее?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 8.14. JSON Types](https://www.postgresql.org/docs/current/datatype-json.html)

PostgreSQL поддерживает два типа для хранения JSON: `json` и `jsonb`.

`json` хранит точную копию входного текста, включая незначащие пробелы и порядок ключей. При обработке требует повторного разбора.

`jsonb` хранит данные в разобранном бинарном формате. Работает медленнее на ввод (из-за конвертации), но значительно быстрее на вывод. Поддерживает GIN-индексы. Дублирующиеся ключи не сохраняются — побеждает последнее значение; порядок ключей не гарантирован. Для большинства приложений рекомендуется `jsonb`.

Для GIN-индексирования `jsonb` предусмотрены два класса операторов:

**jsonb_ops** (по умолчанию): создаёт отдельные элементы индекса для каждого ключа и значения. Поддерживает `?`, `?|`, `?&`, `@>`, `@?`, `@@`.

**jsonb_path_ops**: создаёт единственный элемент индекса на значение (хеш пути и значения). Поддерживает только `@>`, `@?`, `@@`, но индекс значительно меньше по размеру, а поиск — быстрее и специфичнее, особенно на больших наборах данных.

#### Ответ от Claude

**json vs jsonb: ключевые различия**

| Аспект | json | jsonb |
|--------|------|-------|
| Формат хранения | Текст (точная копия) | Разобранный бинарный |
| Скорость записи | Быстрее | Медленнее (парсинг при вставке) |
| Скорость чтения/обработки | Медленнее (повторный парсинг) | Значительно быстрее |
| Пробелы и порядок ключей | Сохраняются | Не сохраняются |
| Дублирующиеся ключи | Все сохраняются | Последнее значение |
| GIN-индексирование | Не поддерживается | Поддерживается |
| Рекомендация | Только если нужно точное сохранение текста | Для большинства задач |

**Операторы jsonb**

```sql
-- Извлечение значений
SELECT jdoc -> 'name'        -- jsonb (объект/массив)
SELECT jdoc ->> 'name'       -- text
SELECT jdoc #> '{a,b}'       -- по пути, результат jsonb
SELECT jdoc #>> '{a,b}'      -- по пути, результат text

-- Проверки
SELECT jdoc @> '{"status": "active"}'::jsonb  -- содержит (containment)
SELECT jdoc ? 'name'                           -- ключ существует
SELECT jdoc ?| ARRAY['a','b']                 -- хотя бы один ключ
SELECT jdoc ?& ARRAY['a','b']                 -- все ключи

-- jsonpath (PostgreSQL 12+)
SELECT jdoc @? '$.tags[*] ? (@ == "sale")'   -- jsonpath match
SELECT jdoc @@ '$.price > 100'               -- jsonpath boolean
```

**GIN: jsonb_ops vs jsonb_path_ops**

```sql
-- jsonb_ops (по умолчанию): один элемент индекса на каждый ключ и значение
CREATE INDEX idx_gin ON api USING GIN (jdoc);

-- jsonb_path_ops: один элемент индекса на значение (хеш пути)
CREATE INDEX idx_gin_path ON api USING GIN (jdoc jsonb_path_ops);
```

| Параметр | jsonb_ops | jsonb_path_ops |
|----------|-----------|---------------|
| Поддерживаемые операторы | `?`, `?|`, `?&`, `@>`, `@?`, `@@` | `@>`, `@?`, `@@` (без key-existence) |
| Размер индекса | Больше | Значительно меньше |
| Специфичность поиска | Ниже | Выше (хеш пути+значения) |
| Скорость containment-запросов | Ниже | Выше |

**Когда использовать jsonb_path_ops:**
- Запросы исключительно на containment (`@>`) и jsonpath (`@?`, `@@`).
- Большие коллекции документов, где размер индекса критичен.
- Часто встречающиеся ключи: в `jsonb_ops` ключ `"id"` порождает огромный постинг-лист, снижая специфичность; в `jsonb_path_ops` хешируется пара путь+значение.

**Когда использовать jsonb_ops:**
- Нужны проверки существования ключей (`?`, `?|`, `?&`).
- Поиск по любым ключам без заранее известных путей.
- Небольшие коллекции, где разница в размере несущественна.

**Индекс на вложенное поле**

```sql
-- Эффективнее для запросов по конкретному полю
CREATE INDEX idx_tags ON api USING GIN ((jdoc -> 'tags'));

-- Использование:
SELECT * FROM api WHERE jdoc -> 'tags' ? 'sale';
```

### Вопрос 58. Что такое PgBouncer и зачем он нужен? Объясни три режима пулинга (session, transaction, statement), какие возможности PostgreSQL недоступны в transaction-режиме и почему transaction-режим самый распространённый в продакшене.

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PgBouncer features — pgbouncer.org](https://www.pgbouncer.org/features.html)

PgBouncer — лёгкий пулер подключений для PostgreSQL. Предоставляет три стратегии ротации подключений, отличающихся моментом возврата серверного соединения в пул.

**Session pooling** — наиболее мягкий режим. Серверное соединение назначается клиенту на всё время сессии и возвращается в пул после отключения клиента. Поддерживаются все возможности PostgreSQL.

**Transaction pooling** — серверное соединение назначается клиенту только на время транзакции. Когда PgBouncer обнаруживает завершение транзакции, соединение возвращается в пул. Этот режим нарушает совместимость с рядом функций, зависящих от сессии.

**Statement pooling** — наиболее агрессивный метод. Подобен transaction pooling, но многооператорные транзакции запрещены, что принудительно переводит клиента в режим autocommit. Предназначен в первую очередь для PL/Proxy.

Функции, несовместимые с transaction pooling: `SET`/`RESET`, `LISTEN`, `WITH HOLD CURSOR`, `PREPARE / DEALLOCATE`, временные таблицы с `PRESERVE/DELETE ROWS`, `LOAD`, сессионные advisory locks.

#### Ответ от Claude

**Зачем нужен PgBouncer**

PostgreSQL создаёт отдельный процесс на каждое подключение. При большом числе соединений (сотни, тысячи) накладные расходы становятся критическими: каждый процесс потребляет ~5–10 МБ RAM, возникает конкуренция за CPU при переключении контекста. PgBouncer стоит перед PostgreSQL как прокси: приложения подключаются к PgBouncer, а PgBouncer держит небольшой пул реальных соединений к PostgreSQL.

**Три режима пулинга**

```
Режим         | Когда соединение возвращается в пул
─────────────────────────────────────────────────────────
session       | После отключения клиента
transaction   | После COMMIT / ROLLBACK (конец транзакции)
statement     | После каждого оператора (autocommit)
```

**Подробно о каждом режиме**

**Session pooling:**
- Семантически прозрачен: поддерживает все возможности PostgreSQL.
- Не даёт реального выигрыша при большом числе одновременных активных клиентов.
- Применяется для совместимости, когда приложение использует сессионные функции (SET, PREPARE и т.д.).

**Transaction pooling:**
- Наиболее распространён в продакшене: одно серверное соединение обслуживает несколько клиентских транзакций поочерёдно.
- Реальный выигрыш: 100 серверных соединений могут обслуживать тысячи клиентов при условии, что клиент держит транзакцию открытой не постоянно.
- Ограничения: после возврата в пул серверное соединение сбрасывается через `server_reset_query` (обычно `DISCARD ALL`), поэтому сессионные настройки теряются.

**Недоступные функции в transaction mode:**

| Функция | Причина |
|---------|---------|
| `SET`/`RESET` (без SCOPE LOCAL) | Сессионные настройки теряются при возврате в пул |
| `PREPARE / DEALLOCATE` | Prepared statements привязаны к сессии |
| `LISTEN / NOTIFY` | Нотификации работают на уровне сессии |
| `WITH HOLD CURSOR` | Курсор переживает транзакцию — нельзя |
| Сессионные `pg_advisory_lock` | Lock привязан к сессии, не к транзакции |
| Временные таблицы | Созданы в сессии, пул переключает соединение |

**Почему transaction-режим самый распространённый**

Типичное OLTP-приложение (веб-сервер) открывает транзакцию на время одного HTTP-запроса (~10–100 мс) и большую часть времени простаивает. В session-режиме каждое соединение занято, даже когда клиент ждёт. В transaction-режиме серверное соединение освобождается сразу после `COMMIT` и доступно следующему клиенту. Результат: 20 серверных соединений эффективно обслуживают 500 клиентских.

**Конфигурация (pgbouncer.ini)**

```ini
[pgbouncer]
pool_mode = transaction
max_client_conn = 1000
default_pool_size = 20
server_reset_query = DISCARD ALL
```

### Вопрос 59. Что такое advisory locks в PostgreSQL? Чем pg_advisory_lock отличается от pg_advisory_xact_lock? Когда advisory locks предпочтительнее обычных табличных блокировок и какие ограничения нужно учитывать?

**Категория:** PostgreSQL · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 13.3. Explicit Locking (Advisory Locks)](https://www.postgresql.org/docs/current/explicit-locking.html)

Advisory locks — это определяемые приложением блокировки, выполнение которых система не контролирует: правильное их использование — задача приложения. Advisory locks полезны для стратегий блокировки, плохо вписывающихся в модель MVCC, например для эмуляции пессимистичных блокировок. Их преимущества перед флагами в таблицах: более высокая производительность, отсутствие раздувания таблиц, автоматическая очистка сервером при завершении сессии.

**Сессионные advisory locks** требуют явного освобождения и не подчиняются транзакционной семантике: блокировка, захваченная и затем «откатившаяся» вместе с транзакцией, по-прежнему удерживается. Одна и та же блокировка может быть захвачена несколько раз; для каждого успешного запроса на захват требуется соответствующий запрос на освобождение.

**Транзакционные advisory locks** автоматически освобождаются в конце транзакции. Транзакционные и сессионные запросы на одну и ту же блокировку блокируют друг друга ожидаемым образом.

#### Ответ от Claude

**Что такое advisory locks**

Advisory locks — мягкие пользовательские блокировки в разделяемой памяти PostgreSQL. Идентифицируются одним 64-битным или двумя 32-битными целыми числами, выбранными приложением. PostgreSQL не навязывает семантику — блокировки действуют только в том смысле, в котором приложение договорилось их использовать.

**Сессионный vs транзакционный уровень**

| Аспект | Сессионный (`pg_advisory_lock`) | Транзакционный (`pg_advisory_xact_lock`) |
|--------|--------------------------------|----------------------------------------|
| Время жизни | До явного освобождения или конца сессии | До конца транзакции (COMMIT/ROLLBACK) |
| ROLLBACK транзакции | Блокировка остаётся! | Освобождается автоматически |
| Явный unlock | Требуется (`pg_advisory_unlock`) | Не нужен |
| Вложенность | Можно захватить N раз, нужно освободить N раз | Нет понятия вложенности |

```sql
-- Сессионный уровень
SELECT pg_advisory_lock(42);          -- захват (ждёт, если занято)
SELECT pg_try_advisory_lock(42);      -- не ждёт, возвращает bool
SELECT pg_advisory_unlock(42);        -- явное освобождение

-- Транзакционный уровень
BEGIN;
SELECT pg_advisory_xact_lock(42);    -- освобождается автоматически при COMMIT/ROLLBACK
SELECT pg_try_advisory_xact_lock(42);
COMMIT;

-- Shared (разделяемые) vs Exclusive
SELECT pg_advisory_lock_shared(42);   -- shared: несколько читателей
SELECT pg_advisory_lock(42);          -- exclusive: один захватчик
```

**Когда advisory locks предпочтительнее табличных**

1. **Задача, не связанная с конкретными строками**: например, блокировка «только один процесс занимается обработкой задания» — нет таблицы для блокировки.
2. **Нет раздувания таблицы**: обычный паттерн `UPDATE jobs SET locked = true WHERE id = ? AND NOT locked` порождает мёртвые кортежи.
3. **Высокая скорость**: advisory locks хранятся в shared memory, не требуют I/O.
4. **Автоматическая очистка**: при краше/дисконнекте сессионные блокировки освобождаются автоматически.

**Типичный паттерн: распределённая очередь задач**

```sql
-- Выбрать и заблокировать одну задачу для обработки
SELECT id FROM jobs
WHERE pg_try_advisory_xact_lock(id)
  AND status = 'pending'
ORDER BY id
LIMIT 1
FOR UPDATE SKIP LOCKED;
-- Транзакционный lock гарантирует, что при ROLLBACK задача снова доступна
```

**Ограничения**

- Видны в `pg_locks` (колонка `locktype = 'advisory'`).
- Занимают место в разделяемой памяти; лимит определяется `max_locks_per_transaction × max_connections`.
- Сессионные locks **не снимаются** при `ROLLBACK` — частая ошибка при exception handling.
- Нет автоматической проверки совместимости с другими объектами базы.

### Вопрос 60. Как PostgreSQL хранит статистику для планировщика запросов? Что такое default_statistics_target, как его изменить на уровне столбца и почему устаревшая или недостаточная статистика приводит к плохим планам выполнения?

**Категория:** PostgreSQL · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PostgreSQL Documentation 18 — 14.2. Statistics Used by the Planner](https://www.postgresql.org/docs/current/planner-stats.html)

Планировщик запросов PostgreSQL использует статистические данные о содержимом таблиц, чтобы оценить количество строк, возвращаемых каждой частью запроса. Затем он сравнивает стоимости различных вариантов плана и выбирает наименее дорогостоящий. Все собранные статистики хранятся в системном каталоге `pg_statistic` (или доступны через представление `pg_stats`).

Объём информации, хранимой в `pg_statistic`, задаётся параметром `default_statistics_target` (по умолчанию 100). Это значение определяет максимальное количество записей в массивах `most_common_vals` и `histogram_bounds` для каждого столбца. Повышение предела может позволить планировщику делать более точные оценки — особенно для столбцов с неравномерным распределением данных — за счёт увеличения объёма `pg_statistic` и несколько большего времени выполнения `ANALYZE`.

Объём статистики по отдельному столбцу можно задать командой `ALTER TABLE ... ALTER COLUMN ... SET STATISTICS N`. Значение `-1` означает использование `default_statistics_target`. Максимально допустимое значение — 10 000.

#### Ответ от Claude

**Что хранит PostgreSQL для планировщика**

Команда `ANALYZE` собирает статистику по каждому столбцу каждой таблицы и записывает её в `pg_statistic`. Через представление `pg_stats` она читается удобнее:

```sql
SELECT attname, n_distinct, correlation, most_common_vals, histogram_bounds
FROM pg_stats
WHERE tablename = 'orders' AND attname = 'status';
```

Ключевые собираемые данные:

| Поле | Что означает |
|------|-------------|
| `n_distinct` | Число уникальных значений (>0: абс.; <0: доля от числа строк) |
| `null_frac` | Доля NULL-значений |
| `avg_width` | Средняя ширина значения в байтах |
| `most_common_vals` | Список наиболее частых значений |
| `most_common_freqs` | Частоты для `most_common_vals` |
| `histogram_bounds` | Границы бакетов гистограммы распределения |
| `correlation` | Корреляция физического порядка строк с логическим (от -1 до 1) |

**default_statistics_target**

Определяет размер массивов `most_common_vals` и `histogram_bounds` (по умолчанию 100 записей). Больше записей = более точная статистика, но дольше ANALYZE и больше объём `pg_statistic`.

```sql
-- Глобально (в postgresql.conf или через ALTER SYSTEM)
SET default_statistics_target = 200;

-- На уровне конкретного столбца
ALTER TABLE orders ALTER COLUMN customer_id SET STATISTICS 500;
-- Вернуть к глобальному значению
ALTER TABLE orders ALTER COLUMN customer_id SET STATISTICS -1;
```

**Когда увеличивать statistics target**

1. **Столбцы с сильно неравномерным распределением** (например, `status` с 90% строк в одном значении, или `user_id` с hotspot-пользователями): увеличить до 200–500.
2. **Ключи JOIN**: точная оценка кардинальности JOIN влияет на выбор алгоритма (Nested Loop vs Hash Join) и порядок соединений.
3. **Столбцы в предикатах range-запросов**: более детальная гистограмма даёт точнее оценки диапазонов.
4. **Partitioned tables**: планировщику труднее оценивать кардинальность секций без детальной статистики.

**Почему плохая статистика ведёт к плохим планам**

Планировщик принимает все решения на основе **оценок**, а не реальных данных. При ошибке в оценке кардинальности:

| Ошибка оценки | Последствие |
|--------------|-------------|
| Оценка: 100 строк, факт: 100 000 | Nested Loop вместо Hash Join → катастрофа при JOIN |
| Оценка: 50 000 строк, факт: 10 | Sequential Scan вместо Index Scan |
| Устаревшая статистика после bulk-load | seq scan для таблицы, которую нужно сканировать через индекс |

**Инструменты диагностики**

```sql
-- Найти расхождение оценок с реальностью (нужен EXPLAIN ANALYZE)
EXPLAIN (ANALYZE, BUFFERS) SELECT ...;
-- Смотреть: rows=<оценка> vs actual rows=<факт>

-- Принудительно обновить статистику по таблице
ANALYZE orders;

-- Проверить, когда последний раз собиралась статистика
SELECT relname, last_analyze, last_autoanalyze
FROM pg_stat_user_tables
ORDER BY last_autoanalyze NULLS FIRST LIMIT 10;
```

**Расширенная статистика (PostgreSQL 10+)**

Для корреляции между несколькими столбцами одна таблица:

```sql
-- Создать многостолбцовую статистику
CREATE STATISTICS my_stat ON col1, col2 FROM my_table;
ANALYZE my_table;

-- Планировщик учтёт зависимость между col1 и col2
```
