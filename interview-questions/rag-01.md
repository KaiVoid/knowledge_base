### Вопрос 1. Что такое RAG (Retrieval-Augmented Generation) и какую проблему LLM он решает?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Prompt Engineering Guide — RAG](https://www.promptingguide.ai/techniques/rag)

RAG — это метод, который соединяет компонент поиска информации с моделью генерации текста. Как описывается в руководстве: «RAG принимает запрос и извлекает набор релевантных/поддерживающих документов из источника (например, Википедии). Документы объединяются с исходным входным промптом и передаются генератору текста, который производит итоговый ответ».

RAG решает две критические проблемы языковых моделей:

1. **Галлюцинации**: подход «помогает снизить проблему галлюцинаций», заземляя ответы в фактическом исходном материале, а не опираясь исключительно на паттерны обучающих данных.
2. **Устаревшие знания**: традиционные LLM имеют статичные параметрические знания, которые устаревают. RAG позволяет «обходить необходимость переобучения, обеспечивая доступ к актуальной информации через генерацию на основе поиска».

Это делает RAG «адаптируемым для ситуаций, где факты могут меняться со временем».

#### Ответ от Claude

RAG (Retrieval-Augmented Generation) — это архитектурный паттерн, который расширяет возможности языковых моделей, подключая к ним внешний компонент поиска. Вместо того чтобы полагаться только на знания, «запечённые» в параметрах модели при обучении, RAG динамически извлекает релевантные документы из внешней базы знаний и использует их как контекст при генерации ответа.

**Проблемы LLM, которые решает RAG:**

- **Галлюцинации** — LLM может уверенно генерировать фактически неверные утверждения. RAG заземляет ответ в реальных документах, снижая этот риск.
- **Knowledge cutoff (устаревание знаний)** — модели обучаются до определённой даты и не знают о более поздних событиях. RAG позволяет обновлять знания системы, не переобучая модель: достаточно добавить новые документы в индекс.
- **Непрозрачность** — при RAG можно указать источник конкретного утверждения, что повышает доверие и проверяемость ответов.

**Как работает RAG:**
1. Пользователь отправляет запрос.
2. Система ищет релевантные фрагменты в векторной базе данных.
3. Найденные фрагменты объединяются с запросом в промпт.
4. LLM генерирует ответ на основе предоставленного контекста.

### Вопрос 2. Какие три фазы составляют базовый пайплайн RAG: индексация, поиск и генерация?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [LlamaIndex — Understanding RAG](https://developers.llamaindex.ai/python/framework/understanding/rag)

LlamaIndex описывает RAG-пайплайн как **пять ключевых стадий**:

1. **Loading (Загрузка)**: получение данных из любых источников — текстовых файлов, PDF, веб-сайтов, баз данных, API — с помощью коннекторов из LlamaHub.
2. **Indexing (Индексация)**: создание структур данных для последующего поиска, прежде всего через векторные эмбеддинги и стратегии метаданных для эффективного поиска контекстно релевантной информации.
3. **Storing (Хранение)**: сохранение индекса и метаданных, чтобы избежать повторной обработки при следующих запусках.
4. **Querying (Запрос)**: использование различных стратегий извлечения — от простых до многошаговых — для получения релевантного контекста из индекса с помощью LLM.
5. **Evaluation (Оценка)**: измерение эффективности через объективные показатели точности, верности (faithfulness) и скорости ответа.

Ключевые компоненты внутри стадий: узлы/документы, ретриверы, роутеры, постпроцессоры узлов и синтезаторы ответов.

#### Ответ от Claude

Базовый RAG-пайплайн строится на трёх фундаментальных фазах:

**1. Индексация (Indexing)**
Документы разбиваются на фрагменты (chunks), каждый фрагмент преобразуется в векторный эмбеддинг с помощью модели эмбеддингов, и все векторы сохраняются в векторной базе данных (Pinecone, Weaviate, FAISS, Chroma и др.). Это однократная (или периодически обновляемая) операция.

**2. Поиск (Retrieval)**
При поступлении запроса пользователя он также преобразуется в эмбеддинг. Система вычисляет косинусное сходство между вектором запроса и векторами всех фрагментов, возвращая top-k наиболее похожих фрагментов.

**3. Генерация (Generation)**
Найденные фрагменты включаются в промпт вместе с исходным запросом, и LLM генерирует ответ на основе предоставленного контекста.

LlamaIndex расширяет эту схему до пяти стадий, добавляя: **Loading** (загрузку данных из разных источников) и **Storing** (персистентное хранение индекса), а также выделяя **Evaluation** как отдельную стадию измерения качества. Это деление более практично для реальных систем.

### Вопрос 3. Что такое индексация (indexing) в RAG-пайплайне и зачем нужны векторные эмбеддинги?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Hugging Face Docs — Agentic RAG](https://huggingface.co/docs/smolagents/en/examples/rag)

RAG — это метод «использования LLM для ответа на запрос пользователя, но при этом ответ основывается на информации, извлечённой из базы знаний».

Индексация обеспечивает несколько ключевых преимуществ:
- **Фактическое заземление**: снижение галлюцинаций за счёт привязки ответов к извлечённым фактам.
- **Доменная специализация**: предоставление предметно-специфических знаний без переобучения модели.
- **Актуальность знаний**: доступ к информации, выходящей за рамки обучающего среза модели.
- **Прозрачность**: возможность цитировать источники для генерируемого контента.

Технически индексация включает нарезку документов на фрагменты с помощью, например, `RecursiveCharacterTextSplitter` (размер 500 символов, перекрытие 50), их преобразование в векторы и сохранение для последующего поиска.

#### Ответ от Claude

**Индексация** — это подготовительная фаза RAG-пайплайна, в ходе которой документы преобразуются в форму, пригодную для семантического поиска.

**Этапы индексации:**
1. **Загрузка документов** — чтение из PDF, баз данных, веб-страниц, API.
2. **Нарезка на фрагменты (chunking)** — разбиение длинных документов на меньшие части (обычно 256–1024 токена).
3. **Создание эмбеддингов** — каждый фрагмент преобразуется моделью эмбеддингов в числовой вектор фиксированной размерности (например, 768 или 1536 чисел).
4. **Сохранение в векторную БД** — векторы индексируются для быстрого поиска ближайших соседей.

**Зачем нужны векторные эмбеддинги:**

Эмбеддинги кодируют **семантический смысл** текста в числовое пространство. Благодаря этому поиск работает **по смыслу, а не по ключевым словам**: запрос «как сохранить файл» найдёт документ с фразой «запись данных на диск», даже если ни одного общего слова нет. Тексты, близкие по смыслу, оказываются близко в векторном пространстве (измеряется косинусным сходством), а тематически далёкие — далеко. Это принципиально отличает векторный поиск от классического полнотекстового (BM25).

### Вопрос 4. Какие стратегии нарезки документов (chunking) существуют в RAG и как они влияют на качество поиска?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Lettria — 5 RAG Chunking Strategies](https://www.lettria.com/blogpost/5-rag-chunking-strategies-for-better-retrieval-augmented-generation)

Выделяют пять основных подходов к нарезке:

1. **Фиксированный размер (Fixed-Length Chunking)** — делит контент по числу слов или токенов (обычно 512–1024). Прост в реализации, но «разрывает предложения или идеи на полуслове», что фрагментирует контекст.

2. **Сентенциальный (Sentence-Based Chunking)** — сохраняет целые предложения, поддерживая логический поток. «Сохраняет мысли нетронутыми, что идеально для разговорного ИИ» и снижает искажения смысла из-за неполных фраз.

3. **Семантический (Semantic Chunking)** — группирует связанные идеи с помощью тематического моделирования и NLP. «Сохраняет контекстно связанные предложения вместе, улучшая точность поиска», хотя требует вычислительных ресурсов.

4. **Скользящее окно с перекрытием (Sliding Window Chunking)** — каждый фрагмент содержит части предыдущего. «Предотвращает потерю ключевой информации», обеспечивая плавные переходы и сохраняя контекст на границах.

5. **Обогащённый метаданными (Metadata-Augmented Chunking)** — добавляет теги с заголовками, временными метками, типами документов. «Повышает точность поиска, особенно в корпоративных средах», и позволяет фильтровать по категориям.

Статья подчёркивает, что правильный выбор стратегии определяет, будет ли ИИ генерировать «точные, релевантные и осмысленные ответы». Рекомендуется комбинировать стратегии в зависимости от типа документа.

#### Ответ от Claude

Стратегия нарезки — одно из ключевых решений при построении RAG-системы, напрямую влияющее на качество поиска.

| Стратегия | Суть | Плюсы | Минусы |
|---|---|---|---|
| **Фиксированный размер** | Деление по числу токенов (512–1024) | Прост, быстр | Разрывает смысловые границы |
| **По предложениям** | Целые предложения как единица | Сохраняет мысль | Неравномерный размер чанков |
| **Семантический** | Деление по смысловым сдвигам (эмбеддинги) | Лучший recall (+9%) | Дорог вычислительно |
| **Скользящее окно** | Фрагменты с перекрытием 10–20% | Сохраняет граничный контекст | Дублирование данных |
| **С метаданными** | Добавление тегов, заголовков, типов | Точная фильтрация | Требует обогащения данных |

**Как выбор стратегии влияет на поиск:**
- Слишком **большой** чанк: снижает точность (precision) — в него попадает много нерелевантного текста.
- Слишком **маленький** чанк: теряется контекст, снижается recall.
- Разрыв смысловых границ: один вопрос и его ответ могут оказаться в разных чанках.

На практике рекомендуется начинать с recursive-чанкинга (400–512 токенов, 10–20% перекрытие) и экспериментировать на реальных данных.

### Вопрос 5. В чём разница между параметрической и непараметрической памятью в контексте RAG?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [arXiv — RAG for Knowledge-Intensive NLP Tasks (Lewis et al., 2020)](https://arxiv.org/abs/2005.11401)

Оригинальная статья Lewis et al. (2020) вводит различие двух типов памяти:

**Параметрическая память** — знания, закодированные в весах нейронной сети. Авторы отмечают, что «крупные предобученные языковые модели показали способность хранить фактические знания в своих параметрах», однако «их возможности доступа к знаниям и их точного использования по-прежнему ограничены».

**Непараметрическая память** — явные, внешние знания, хранящиеся отдельно от модели. В оригинальной RAG-системе «непараметрическая память — это плотный векторный индекс Википедии, доступный через предобученный нейронный ретривер».

RAG объединяет оба типа: предобученная seq2seq-модель (BART) служит параметрической памятью, а плотный векторный индекс — непараметрической. Гибридный подход позволяет «легко обновлять знания, изменяя индекс документов без переобучения генератора».

#### Ответ от Claude

Разграничение параметрической и непараметрической памяти — концептуальная основа, введённая в оригинальной статье Lewis et al. (2020), заложившей фундамент RAG.

**Параметрическая память**
- Знания «запечены» в весах нейронной сети в процессе предобучения.
- Обновление требует дорогостоящего переобучения модели.
- Трудно понять, откуда взялось конкретное утверждение (непрозрачность).
- Пример: всё, что GPT-4 «знает» из своих весов.

**Непараметрическая память**
- Знания хранятся во внешнем корпусе документов (в оригинальной статье — плотный векторный индекс Википедии).
- Обновляется без переобучения — достаточно изменить корпус и переиндексировать.
- Прозрачна — можно указать источник каждого факта.
- Доступна через нейронный ретривер (в оригинальной статье — DPR).

**Почему гибрид лучше каждого подхода в отдельности:**
Параметрическая память хороша для языковых паттернов, рассуждений и генерации связного текста, но плоха для точных фактов и актуальной информации. Непараметрическая — наоборот. RAG комбинирует сильные стороны обоих: LLM отвечает за качество языка и рассуждения, а индекс — за фактическую точность.

### Вопрос 6. Что такое гибридный поиск (BM25 + векторный поиск) в RAG и почему он эффективнее каждого из подходов по отдельности?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Denser AI — Hybrid Search for RAG](https://denser.ai/blog/hybrid-search-for-rag/)

Гибридный поиск сочетает два взаимодополняющих метода:

- **BM25 (разреженный поиск)**: хорошо совпадает с точными ключевыми словами — кодами продуктов, идентификаторами, редкими терминами.
- **Плотный векторный поиск**: улавливает семантический смысл — перефразирования, концептуальные запросы.

Каждый метод имеет слепые зоны, которые закрывает другой:
- Плотный поиск плохо справляется с точными совпадениями вида «SKU AZ-4471», потому что эмбеддинги кодируют смысл, а не буквальные строки.
- BM25 не справляется с семантическими запросами «как вернуть сломанный товар?», если в базе знаний используется другая формулировка — «политика возврата».

Результирующие списки из двух методов объединяются через **Reciprocal Rank Fusion (RRF)**. По бенчмаркам: «настроенный гибридный подход достигает NDCG 0.7497 — прирост 7.4% по сравнению с BM25 (0.6983) или чистым векторным поиском (0.6953) по отдельности».

**RRF** — стандартный алгоритм слияния: `score(d) = Σ 1/(k + rank(d))`, где k обычно равно 60. Работает на рангах, а не на сырых оценках, что позволяет избежать несовместимости шкал BM25 и косинусного сходства. Не требует нормализации и настройки.

#### Ответ от Claude

**Гибридный поиск** — это архитектурный паттерн, при котором одновременно используются два индекса: разреженный (BM25) и плотный (векторный), а результаты объединяются в единый ранжированный список.

**Почему каждый метод в отдельности уступает:**

| | BM25 | Векторный поиск |
|---|---|---|
| **Сила** | Точные совпадения, редкие термины, коды, артикулы | Семантическое сходство, перефразирования, синонимы |
| **Слабость** | Не понимает смысл, только ключевые слова | Пропускает точные совпадения, буквенно-цифровые идентификаторы |

**Reciprocal Rank Fusion (RRF)** — ключевой алгоритм слияния:
- Формула: `score(d) = Σ 1/(60 + rank(d))` по всем спискам.
- Работает с рангами, а не с несовместимыми оценками релевантности.
- Поощряет документы, присутствующие в обоих списках.
- Не требует нормализации и дополнительной настройки.

**Практические результаты:** согласно бенчмаркам, гибридный поиск даёт 7–39% прирост recall по сравнению с каждым методом отдельно. Это делает его стандартным выбором для production RAG-систем (используется, например, в Azure AI Search).

### Вопрос 7. Что такое реранкинг (reranking) в RAG-пайплайне и зачем он нужен после первичного поиска?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [arXiv — Systematic Review of RAG Systems](https://arxiv.org/html/2507.18910v1)

Реранкинг — это этап уточнения, следующий за первичным поиском в RAG-системах. После того как быстрый плотный ретривер возвращает top-k кандидатов, модель-реранкер — как правило, кросс-энкодер-трансформер — переоценивает каждый фрагмент, **совместно кодируя запрос и документ**, и присваивает более точную оценку релевантности.

Первичный поиск использует сходство эмбеддингов, которое является «грубой заменой релевантности». Результаты top-k могут содержать нерелевантные или косвенно связанные элементы. Реранкинг устраняет это, применяя более вычислительно затратную, но точную оценку, чтобы «наиболее релевантные фрагменты оказались на верхних позициях».

Двухэтапный подход — быстрый ретривер + точный реранкер — значительно улучшает производительность на задачах, требующих точных знаний.

#### Ответ от Claude

**Реранкинг** — второй этап поиска в RAG, решающий компромисс между скоростью и точностью.

**Проблема, которую решает реранкинг:**
Первичный поиск (через bi-encoder: два отдельных энкодера для запроса и документа) работает быстро, но менее точно — он кодирует запрос и документы независимо, а сходство вычисляется как скалярное произведение или косинус. Это хорошо для быстрого отбора кандидатов, но плохо для тонкого различения релевантности.

**Как работает реранкер (кросс-энкодер):**
- Получает на вход пару (запрос, фрагмент) **одновременно**.
- Трансформер видит оба текста и их взаимодействие (cross-attention).
- Выдаёт точную оценку релевантности именно для этой пары.
- Значительно точнее bi-encoder, но медленнее — поэтому применяется только к небольшому числу кандидатов.

**Типичный пайплайн:**
1. Первичный поиск → top-50–200 кандидатов (быстро).
2. Реранкер → переоценка всех кандидатов (точно, но медленнее).
3. Выбор top-5–10 для включения в промпт LLM.

Реранкинг особенно критичен для высокоточных приложений: он отфильтровывает фрагменты, которые похожи тематически, но не содержат ответа на конкретный вопрос.

### Вопрос 8. Чем RAG отличается от файн-тюнинга (fine-tuning)? Когда выбирать один подход, а когда другой?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Contextual AI — RAG vs Fine-Tuning for Enterprise AI](https://contextual.ai/blog/rag-vs-fine-tuning-which-approach-is-right-for-enterprise-ai)

**Ключевое различие**: «RAG предоставляет контекст во время инференса; файн-тюнинг изменяет саму модель».

**Когда выбирать RAG:**
- Приложению нужна точность фактов из корпоративных документов.
- Информация меняется часто (ежемесячно и чаще).
- Нужна атрибуция источников и цитаты.
- Есть большие коллекции документов.
- Существуют требования соответствия и аудита.

RAG снижает риск галлюцинаций, поскольку ответы «заземлены в извлечённом контенте, а не опираются исключительно на информацию, усвоенную при обучении».

**Когда выбирать файн-тюнинг:**
- Нужен стабильный стиль, тональность или голос бренда.
- Требуется доменная лексика и терминология.
- Важны структурированные форматы вывода.
- Нужны специфические паттерны рассуждений.

**Вывод**: для большинства корпоративных приложений RAG предпочтителен для задач с фактической точностью, поскольку «RAG обычно превосходит файн-тюнинг» по точности фактов; гибридный подход объединяет точность RAG со стилистическими преимуществами файн-тюнинга.

#### Ответ от Claude

RAG и файн-тюнинг решают разные проблемы и не являются взаимоисключающими.

| Параметр | RAG | Файн-тюнинг |
|---|---|---|
| **Как работает** | Извлекает знания из внешней БД в момент запроса | «Запекает» знания в параметры модели |
| **Актуальность данных** | Всегда актуально (обновляй индекс) | Устаревает (нужно переобучение) |
| **Атрибуция** | Легко указать источник | Сложно отследить источник |
| **Риск галлюцинаций** | Ниже (ответ заземлён в документах) | Выше |
| **Стоимость обновления** | Низкая (переиндексация) | Высокая (переобучение) |
| **Адаптация стиля/формата** | Слабая | Сильная |
| **Специфичная лексика** | Средняя | Отличная |

**Когда выбирать RAG:**
- Данные часто меняются (документация, нормативы, цены).
- Нужна трассируемость и цитаты источников.
- Большой корпус документов, который невозможно «запечь» в модель.

**Когда выбирать файн-тюнинг:**
- Нужен специфический стиль или формат ответа.
- Требуется усвоить доменную лексику, которой нет в предобученной модели.
- Задача стабильна и данные не устаревают.

**На практике** эти подходы часто комбинируют: файн-тюнинг адаптирует поведение и стиль, RAG обеспечивает актуальные факты.

### Вопрос 9. Чем RAG отличается от использования длинного контекста (long context window) и когда какой подход лучше?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Meilisearch — RAG vs Long-Context LLMs](https://www.meilisearch.com/blog/rag-vs-long-context-llms)

**Ключевое различие**: RAG извлекает релевантную информацию из внешних источников перед генерацией, тогда как LLM с длинным контекстом обрабатывает весь ввод в рамках одного расширенного контекстного окна — иногда миллионы токенов — без поиска.

**Когда RAG эффективнее:**
- Динамические, часто обновляемые данные (корпоративные базы знаний, реальное время).
- Экономически чувствительные приложения: RAG «держит количество токенов небольшим, снижает затраты и ускоряет каждый запрос».
- Прозрачность: «каждый ответ может указывать на извлечённые фрагменты, что обеспечивает трассируемость».
- Вызов LLM с 200k+ токенов «может стоить до $20 за вызов».

**Когда длинный контекст эффективнее:**
- Суммаризация целых отчётов или исследовательских статей.
- Сложные задачи рассуждения по длинным документам.
- Проверка юридических документов без фрагментации контекста.

**Гибридный подход** — комбинирование RAG для поиска и моделей с длинным контекстом для рассуждения — «обеспечивает идеальный баланс между recall и рассуждением».

#### Ответ от Claude

С ростом контекстных окон LLM (128k, 1M токенов) возникает закономерный вопрос: зачем нужен RAG, если можно просто загрузить все документы в контекст?

| Параметр | RAG | Длинный контекст |
|---|---|---|
| **Стоимость** | Низкая (только релевантные фрагменты) | Высокая (оплата всех токенов в контексте) |
| **Скорость** | Быстрее (меньше токенов для обработки) | Медленнее (100k токенов в ~3 раза дольше) |
| **Актуальность** | Отлично (динамический индекс) | Зависит от того, что загружено в контекст |
| **Глобальные запросы** | Хуже (может пропустить связи) | Лучше (видит весь документ сразу) |
| **Точечный поиск** | Лучше (целевое извлечение) | Хуже (может «потеряться» в длинном тексте) |
| **Ограничение размера корпуса** | Нет (индекс любого размера) | Да (ограничен размером окна) |

**RAG лучше**, когда:
- Корпус больше размера контекстного окна.
- Данные обновляются часто.
- Стоимость токенов критична.
- Нужна точечная работа с конкретными фактами.

**Длинный контекст лучше**, когда:
- Нужно понять весь документ целиком.
- Важны сложные рассуждения с опорой на разные части документа.
- Данные статичны и помещаются в окно.

**Гибридный подход** (RAG для поиска + длинный контекст для рассуждения) даёт лучшие результаты для сложных задач.

### Вопрос 10. Какие три эволюционных парадигмы RAG описывает современная литература: Naive RAG, Advanced RAG, Modular RAG?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Prompt Engineering Guide — RAG Research](https://www.promptingguide.ai/research/rag)

**Naive RAG** — базовый подход с прямолинейным процессом: индексация документов, поиск релевантных по запросу, генерация ответа. Имеет существенные ограничения: «Naive RAG страдает от низкой точности (несоответствие извлечённых фрагментов) и низкого recall (неполнота извлечения всех релевантных фрагментов)». Плохо справляется с устаревшей информацией, может генерировать избыточные или галлюцинированные ответы.

**Advanced RAG** — эволюционный подход, устраняющий недостатки Naive RAG путём оптимизации каждой стадии. Улучшает предпоисковые процессы (стратегии индексации и нарезки), повышает качество поиска через лучшие модели эмбеддингов, совершенствует постобработку через реранкинг и сжатие. Фокусируется на «избегании ограничений контекстного окна и работе с зашумлённой или отвлекающей информацией».

**Modular RAG** — наиболее гибкий подход, рассматривающий RAG как компонуемые модули, перегруппируемые под конкретные нужды. Включает модули «поиска, памяти, слияния, маршрутизации, прогнозирования и адаптера задач». Позволяет разработчикам создавать кастомные воркфлоу с гибридным поиском, рекурсивным извлечением и адаптивными стратегиями.

#### Ответ от Claude

Три парадигмы RAG отражают эволюцию подхода от простого прототипа к зрелой архитектуре.

**1. Naive RAG (наивный/базовый)**
Простой пайплайн: индексация → поиск → генерация.
- Проблемы: низкая точность (в контекст попадают нерелевантные фрагменты), низкий recall (не все нужные фрагменты извлекаются), чувствительность к качеству чанкинга.
- Подходит для: прототипов, простых задач с небольшим корпусом.

**2. Advanced RAG (продвинутый)**
Улучшения на каждом этапе базового пайплайна:
- *До поиска*: улучшенный чанкинг, обогащение метаданными, предобработка запроса (HyDE, переформулировка).
- *При поиске*: лучшие модели эмбеддингов, гибридный поиск.
- *После поиска*: реранкинг, сжатие контекста, фильтрация по релевантности.
- Подходит для: production-систем с высокими требованиями к точности.

**3. Modular RAG (модульный)**
RAG как набор взаимозаменяемых компонентов:
- Модули: поиск, память, слияние результатов, маршрутизация запросов, адаптер задачи.
- Компоненты можно заменять, добавлять и переставлять без перестройки всей системы.
- Поддерживает рекурсивный поиск, адаптивный поиск, многошаговые рассуждения.
- Подходит для: сложных enterprise-систем с разнородными источниками и требованиями.

Каждая следующая парадигма не отменяет предыдущую, а расширяет набор доступных инструментов.

### Вопрос 11. Какие метрики используются для оценки качества RAG-системы (faithfulness, answer relevancy, context precision/recall)?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Deepchecks — RAG Evaluation Metrics](https://deepchecks.com/rag-evaluation-metrics-answer-relevancy-faithfulness-accuracy/)

Статья выделяет четыре основных измерения оценки RAG-систем:

1. **Точность поиска и контекстная релевантность (Retrieval Precision & Contextual Relevance)**: измеряет «долю извлечённых документов, которые действительно релевантны запросу». Контекстная релевантность оценивает «более глубокое семантическое соответствие между намерением запроса и извлечёнными фрагментами» с помощью сходства эмбеддингов или кросс-энкодерных ранжировщиков.

2. **Релевантность ответа (Answer Relevancy)**: оценивает, остаётся ли генерируемый ответ сфокусированным на намерении запроса. Использует «инструменты семантического сходства» и «оценки выравнивания запроса-ответа от кросс-энкодеров».

3. **Верность и заземлённость (Faithfulness & Groundedness)**: обеспечивает заземлённость ответов в извлечённых источниках. Faithfulness использует «пайплайны NLI (логический вывод на основе естественного языка)» для декомпозиции ответов на атомарные утверждения и проверки их поддержки контекстом.

4. **Точность на реальных данных (Real-World Accuracy)**: тестирует производительность на аутентичных, специфичных для домена данных и KPI задачи.

**RAGAS** упоминается как инструмент автоматической оценки, который «объединяет эти показатели в единую оценку, интегрируя precision@K... с другими показателями, такими как удовлетворённость пользователей».

#### Ответ от Claude

Оценка RAG-системы сложнее, чем оценка обычного ML-классификатора: нужно измерять качество как поиска, так и генерации, а также их взаимодействие.

**Основные метрики:**

| Метрика | Что измеряет | Как вычисляется |
|---|---|---|
| **Faithfulness (верность)** | Доля утверждений в ответе, подтверждённых извлечёнными фрагментами | NLI-модель проверяет каждое атомарное утверждение; порог >0.9 считается «заземлённым» |
| **Answer Relevancy** | Насколько ответ соответствует намерению запроса | Косинусное сходство между ответом и запросом; генерация обратных вопросов |
| **Context Precision** | Доля релевантных фрагментов среди всех извлечённых | precision@K: сколько из top-k фрагментов действительно нужны |
| **Context Recall** | Была ли извлечена вся необходимая информация | Сравнение с эталонными ответами; NLI-проверка полноты |

**Почему важны все четыре:**
- Высокая faithfulness при низком context recall → ответ верен найденному, но информация неполная.
- Высокий context recall при низкой faithfulness → хорошо нашли, но модель галлюцинирует поверх контекста.
- Низкий answer relevancy → ответ «заземлён» в документах, но не отвечает на вопрос.

**RAGAS** — популярный фреймворк, автоматизирующий расчёт этих метрик с помощью LLM-судей, что позволяет оценивать RAG-пайплайн без ручной разметки.

### Вопрос 12. Почему RAG не гарантирует отсутствия галлюцинаций и какие основные причины их возникновения в RAG-пайплайне?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Unstructured — RAG Pipeline Challenges](https://unstructured.io/insights/rag-pipeline-challenges-from-data-ingestion-to-retrieval)

RAG не устраняет галлюцинации, потому что в пайплайне есть несколько точек отказа, где слабые доказательства всё равно попадают к модели.

**Причины на этапе поиска:**
- **Слабый парсинг и чанкинг**: «Парсинг теряет границы таблиц, смешивает столбцы или нарушает порядок секций — поиск возвращает правдоподобные, но неверные фрагменты». Этот фундаментальный сбой вынуждает все последующие компоненты работать с деградировавшей информацией.
- **Несоответствие эмбеддинга домену**: когда общие модели эмбеддингов сталкиваются со специализированной терминологией, они могут сглаживать важные различия, возвращая близкие по поверхности, но фактически неверные фрагменты.
- **Путаница похожести с релевантностью**: концептуально связанный текст не обязательно содержит фактические ответы, но передаётся модели как валидное свидетельство.

**Причины на этапе генерации:**
- **Неконтролируемый синтез**: без явных правил модель может смешивать противоречивые источники или усреднять конфликтующие утверждения, не раскрывая причин. «Одни и те же извлечённые фрагменты могут давать разные результаты в зависимости от того, как сформулированы цитаты, область действия и поведение при отказе».
- **Недостаточные требования к заземлению**: когда промпты не принуждают к использованию доказательств, модели заполняют пробелы правдоподобными, но неподтверждёнными утверждениями.

Ключевой вывод: **«RAG настолько силён, насколько силён слой данных, собирающий контекст»**.

#### Ответ от Claude

RAG снижает риск галлюцинаций, но не устраняет их полностью — и причины этого лежат на разных уровнях пайплайна.

**Причины галлюцинаций на этапе поиска:**

1. **Некачественный парсинг документов** — если PDF парсится с ошибками (таблицы, формулы, колонки), в индекс попадают искажённые данные; поиск возвращает внешне релевантный, но фактически неверный фрагмент.
2. **Неудачный чанкинг** — разрыв вопроса и ответа между разными чанками, потеря контекста на границах.
3. **Несоответствие эмбеддинг-модели домену** — общая модель может неверно оценить сходство для доменной лексики (медицина, право, финансы).
4. **Похожесть ≠ релевантность** — фрагменты могут быть тематически близки к запросу, но не содержать ответ на конкретный вопрос.

**Причины галлюцинаций на этапе генерации:**

1. **Конфликтующие источники** — если два извлечённых фрагмента противоречат друг другу, модель может произвольно выбрать или «усреднить» их.
2. **Шум в контексте** — нерелевантные фрагменты в промпте сбивают модель с толку.
3. **Отсутствие явных правил заземления** — без инструкции «отвечай только на основе контекста» модель добавляет из параметрической памяти.
4. **Ограничения модели** — модель может неверно интерпретировать или перефразировать даже корректно найденный фрагмент.

**Ключевой принцип**: качество поиска задаёт *потолок* точности ответа. Улучшать генерацию без исправления поиска бессмысленно.

### Вопрос 13. Что такое Agentic RAG и чем он отличается от классического RAG-пайплайна?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Hugging Face Docs — Agentic RAG](https://huggingface.co/docs/smolagents/en/examples/rag)

Agentic RAG преодолевает ограничения традиционного RAG, реализуя агента с инструментами поиска. Это превращает RAG из жёсткого пайплайна в интерактивный, управляемый рассуждением процесс.

**Ограничения классического RAG:**
- Единственный шаг поиска: плохие начальные результаты ведут к плохой генерации.
- Несоответствие запроса документу: пользовательские вопросы могут не совпадать с документами-ответами.
- Ограниченное рассуждение: нет возможности для многошагового мышления или уточнения запроса.

**Что даёт Agentic RAG:**
1. Формулирует оптимизированные запросы (преобразует вопрос в форму, удобную для поиска).
2. Выполняет несколько итеративных поисков по мере необходимости.
3. Рассуждает над извлечённым контентом: анализирует, синтезирует, делает выводы.
4. Самокритика и уточнение: оценивает результаты поиска и корректирует стратегию.

**Ключевые техники:**
- **HyDE (Hypothetical Document Embedding)**: вместо прямого запроса агент формулирует гипотетический документ-ответ и ищет по нему.
- **Self-Query Refinement**: агент анализирует промежуточные результаты и выполняет уточняющие поиски.

#### Ответ от Claude

**Классический RAG** — жёсткий детерминированный пайплайн:
1. Запрос → один поиск → top-k фрагментов → генерация ответа.

**Agentic RAG** — динамическая система, где LLM является агентом с инструментом поиска:
1. Агент анализирует запрос и решает, как искать.
2. Выполняет первый поиск.
3. Анализирует результаты, формулирует уточняющие запросы при необходимости.
4. Итеративно повторяет поиск, пока не соберёт достаточно информации.
5. Синтезирует финальный ответ.

**Ключевые отличия:**

| Аспект | Классический RAG | Agentic RAG |
|---|---|---|
| Количество поисков | Один | Несколько (итеративно) |
| Формулировка запроса | Дословный запрос пользователя | Агент оптимизирует запрос |
| Рассуждение | Нет | Многошаговое |
| Гибкость | Фиксированный пайплайн | Адаптируется под задачу |
| Сложность реализации | Низкая | Высокая |
| Стоимость | Дешевле (меньше вызовов LLM) | Дороже (больше вызовов) |

**Техники Agentic RAG:**
- **HyDE**: агент генерирует «идеальный ответ» и ищет похожие документы по нему, а не по запросу.
- **Self-Query Refinement**: агент уточняет запрос на основе промежуточных результатов.
- **Multi-hop retrieval**: цепочка поисков для ответа на сложный многосоставной вопрос.

### Вопрос 14. Как RAG решает проблему knowledge cutoff (устаревания знаний) у LLM?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [arXiv — Systematic Review of RAG Systems](https://arxiv.org/html/2507.18910v1)

Языковые модели страдают от knowledge cutoff: их обучающие данные имеют фиксированную конечную дату, что делает их неспособными получать текущую информацию. Это ведёт к галлюцинациям и устаревшим ответам.

RAG решает эту проблему, поддерживая то, что называется «параметрической памятью (знания, закодированные в весах генератора) и непараметрической памятью (внешний текстовый корпус, доступный через поиск)». Извлечённые документы предоставляют актуальную, верифицируемую информацию, позволяя системе «легко обновлять знания путём изменения индекса документов без переобучения генератора».

#### Ответ от Claude

**Проблема knowledge cutoff:**
При обучении LLM «видит» тексты только до определённой даты. Всё, что произошло позже, модели неизвестно. Нельзя попросить GPT рассказать о событии прошлой недели — модель либо скажет «не знаю», либо будет галлюцинировать.

**Как RAG решает эту проблему:**

RAG выносит знания из параметров модели во внешний обновляемый индекс. Это обеспечивает:

1. **Обновление без переобучения** — чтобы добавить новую информацию, достаточно загрузить новые документы и переиндексировать корпус. Переобучение модели не требуется (оно стоит миллионы долларов и недели времени).
2. **Точечное обновление** — можно обновить только изменившиеся документы, не трогая остальной корпус.
3. **Гарантированная актуальность** — если индекс обновляется ежедневно, система всегда отвечает по актуальным данным.

**Практические применения:**
- Корпоративные базы знаний (политики, процедуры, изменяющиеся документы).
- Юридическая документация (новые законы, поправки).
- Медицинские руководства (обновляемые протоколы).
- Новостные и аналитические системы реального времени.

**Ограничение**: RAG не помогает, если нужной информации нет в корпусе — мусор на входе, мусор на выходе.

### Вопрос 15. Какие вызовы представляет RAG для enterprise-систем: токен-ограничения, безопасность, мультиисточниковость?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Microsoft Learn — RAG and Generative AI (Azure AI Search)](https://learn.microsoft.com/en-us/azure/search/retrieval-augmented-generation-overview)

Microsoft выделяет пять ключевых вызовов enterprise-RAG:

| Вызов | Описание |
|---|---|
| **Понимание запросов** | Современные пользователи задают сложные, разговорные или расплывчатые вопросы. Традиционный поиск по ключевым словам не справляется, когда запросы не совпадают с терминологией документов. |
| **Мультиисточниковость** | Корпоративный контент разбросан по SharePoint, базам данных, blob-хранилищам. Необходимо создать единый поисковый корпус, не нарушая управление данными. |
| **Ограничения токенов** | LLM принимают ограниченный ввод. Система поиска должна возвращать высокорелевантные, краткие результаты — не огромные дампы документов. (GPT-4 принимает около 128k токенов, но 10 000 страниц документации не поместятся.) |
| **Время ответа** | Пользователи ожидают AI-ответов за секунды, а не минуты. |
| **Безопасность и управление** | Открытие частного контента LLM требует детального контроля доступа: пользователи и агенты должны получать только авторизованный контент. |

Решение Microsoft (Azure AI Search): агентный поиск с LLM-планированием запросов для сложных случаев и классический гибридный поиск с семантическим реранкингом для более простых сценариев.

#### Ответ от Claude

Enterprise-RAG сталкивается с вызовами, которые нетипичны для прототипов, но критичны в production.

**1. Токен-ограничения**
LLM имеет ограниченное контекстное окно (~128k токенов у GPT-4). При 10 000 страницах документации нельзя загрузить всё в промпт. Нужен точный отбор топ-N наиболее релевантных фрагментов. Слишком большой контекст снижает качество: модель «теряется» среди нерелевантного шума.

**2. Безопасность и разграничение доступа**
Не все пользователи должны видеть все документы (финансовые данные, HR-информация, стратегические планы). RAG должен применять права доступа пользователя при поиске: возвращать только фрагменты из документов, на которые у пользователя есть права. Иначе чат-бот может раскрыть конфиденциальную информацию не тому сотруднику.

**3. Мультиисточниковость**
Корпоративные данные разбросаны по SharePoint, базам данных, Confluence, Jira, email, blob-хранилищам. Нужна единая система индексации и поиска по всем источникам с поддержкой разных форматов (PDF, Office, HTML, JSON).

**4. Понимание запросов**
Пользователи спрашивают неточно, используя корпоративный жаргон или предполагая контекст из предыдущих сообщений. Классический поиск по ключевым словам не справляется.

**5. Задержка ответа**
Пайплайн RAG включает: парсинг запроса → поиск → реранкинг → генерация LLM. При требовании ответа менее чем за 5 секунд каждый компонент должен быть оптимизирован.

### Вопрос 16. Что такое Dense Passage Retrieval (DPR) и как он используется в оригинальном RAG?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [arXiv — RAG for Knowledge-Intensive NLP Tasks (Lewis et al., 2020)](https://arxiv.org/abs/2005.11401)

Оригинальная статья Lewis et al. (2020) описывает систему, объединяющую параметрическую и непараметрическую память. В статье упоминается предобученный нейронный ретривер, который обеспечивает доступ к плотному векторному индексу Википедии. Ретривер и генератор (BART) совместно обучаются сквозным образом (end-to-end).

Конкретный механизм: «непараметрическая память — это плотный векторный индекс Википедии, доступный через предобученный нейронный ретривер», а «крупные предобученные языковые модели...по-прежнему ограничены в своих возможностях доступа к знаниям и их точного использования».

#### Ответ от Claude

**Dense Passage Retrieval (DPR)** — метод нейронного поиска, предложенный Karpukhin et al. (2020), который стал стандартным ретривером в оригинальной RAG-системе Lewis et al.

**Архитектура DPR:**
DPR использует **два отдельных BERT-энкодера**:
- Энкодер запроса: преобразует вопрос пользователя в плотный вектор.
- Энкодер документов: преобразует каждый фрагмент документа в плотный вектор того же пространства.

Релевантность вычисляется как **скалярное произведение** векторов запроса и документа. Это позволяет заранее вычислить и сохранить все вектора документов, а при поиске вычислять только вектор запроса и быстро находить ближайших соседей.

**Использование DPR в оригинальном RAG:**
1. Весь корпус Википедии (21 млн фрагментов по 100 слов) кодируется энкодером документов и сохраняется.
2. При поступлении запроса энкодер запроса преобразует его в вектор.
3. FAISS-индекс быстро находит top-k наиболее похожих фрагментов (Maximum Inner Product Search).
4. Найденные фрагменты передаются генератору BART.

**Ключевая особенность**: ретривер и генератор обучаются **совместно (end-to-end)** — это означает, что ретривер обучается возвращать именно те фрагменты, которые помогают генератору давать правильные ответы, даже без явной разметки релевантности.

### Вопрос 17. Как работает RAG-пайплайн в Azure AI Search: классический и агентный подходы?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Microsoft Learn — RAG and Generative AI (Azure AI Search)](https://learn.microsoft.com/en-us/azure/search/retrieval-augmented-generation-overview)

Azure AI Search предоставляет два подхода к RAG:

**Классический RAG:**
Приложение отправляет единственный запрос в Azure AI Search и отдельно оркестрирует передачу результатов LLM. Развёрнутый LLM формулирует ответ на основе плоского набора результатов запроса. Подход «проще, с меньшим числом компонентов, и быстрее, так как нет участия LLM в планировании запросов». Использует гибридный поиск (ключевые слова + векторный) и семантический реранкинг.

**Агентный поиск (Agentic Retrieval):**
Специализированный пайплайн для RAG-паттернов. «Использует LLM для интеллектуального разбиения сложных пользовательских запросов на сфокусированные подзапросы, выполняет их параллельно и возвращает структурированные ответы, оптимизированные для моделей завершения чата». Включает:
- Контекстно-зависимое планирование запросов с учётом истории разговора.
- Параллельное выполнение нескольких сфокусированных подзапросов.
- Структурированные ответы с данными заземления, цитатами и метаданными выполнения.
- Встроенный семантический реранкинг.

Microsoft рекомендует: «Для новых RAG-реализаций начинайте с агентного поиска. Для существующих решений рассмотрите миграцию для улучшения точности и понимания контекста».

#### Ответ от Claude

Azure AI Search — один из наиболее зрелых enterprise-инструментов для RAG, эволюционировавший от классического к агентному подходу.

**Классический RAG в Azure AI Search:**
1. Приложение отправляет один запрос в Azure AI Search.
2. Поиск выполняется через **гибридный метод**: одновременно BM25 (ключевые слова) + векторный поиск (семантика).
3. Результаты переранжируются с помощью **семантического реранкинга** (языковая модель переоценивает релевантность).
4. Top-N фрагментов возвращаются в приложение.
5. Приложение самостоятельно передаёт фрагменты + вопрос в LLM (оркестрация на стороне приложения).

**Агентный поиск (Agentic Retrieval):**
1. LLM анализирует сложный/многосоставной вопрос.
2. LLM разбивает его на **несколько целевых подзапросов** с учётом истории разговора.
3. Подзапросы выполняются **параллельно** по нескольким источникам знаний.
4. Результаты объединяются со встроенным семантическим реранкингом.
5. Возвращается **структурированный ответ** с цитатами, данными заземления и логом запросов.

**Когда использовать каждый подход:**

| | Классический RAG | Агентный поиск |
|---|---|---|
| Нужны только GA-функции | Да | Нет (preview) |
| Простые запросы | Да | Избыточен |
| Сложные/разговорные запросы | Ограниченно | Рекомендован |
| Новые системы | Можно | Рекомендован |
| Максимальная точность | Ниже | Выше |

### Вопрос 18. Что такое чанкинг в контексте RAG и зачем он нужен?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Weaviate — Chunking Strategies for RAG](https://weaviate.io/blog/chunking-strategies-for-rag)

Чанкинг — это «процесс разбиения крупных документов на меньшие, управляемые фрагменты». Основная причина необходимости чанкинга: LLM имеют ограниченные контекстные окна, поэтому разбиение документов на сфокусированные сегменты предотвращает потерю информации и сохраняет точность ответов.

**Почему чанкинг критичен для RAG:**
Некачественный чанкинг — частая причина сбоев RAG-систем, а не недостаточные ретриверы. Существует два основных вызова:

1. **Точность поиска**: слишком крупные чанки смешивают несколько концепций, создавая «шумные, усреднённые» эмбеддинги, которые маскируют отдельные темы. Небольшие, сфокусированные чанки дают точные эмбеддинги для лучшего семантического поиска.

2. **Сохранение контекста**: система должна балансировать полноту — чанки должны «быть осмысленными при чтении отдельно» — и длину, чрезмерное увеличение которой снижает качество LLM через «размывание внимания и эффект "lost in the middle"».

**Оптимальная точка чанкинга:** сохраняет логический поток автора, оставаясь достаточно малым для точного поиска и достаточно полным для понимания LLM. Ключевой вывод источника: «в production LLM RAG пайплайнах чанкинг — это не просто шаг предобработки, он определяет качество поиска и эффективность памяти агента».

#### Ответ от Claude

**Чанкинг** — это разбиение длинных документов на меньшие фрагменты (чанки) для эффективной индексации и поиска в RAG-системах.

**Зачем нужен чанкинг:**

1. **Ограничения векторного поиска** — целый документ в 100 страниц сложно точно описать одним эмбеддингом. Более короткий фрагмент имеет более сфокусированный вектор, что даёт лучшее совпадение с запросом.

2. **Ограничения контекстного окна LLM** — LLM принимает ограниченное число токенов. Нельзя передать весь 100-страничный документ как контекст; нужно выбрать только релевантные части.

3. **Экономия токенов** — плата за LLM-вызовы зависит от числа токенов. Меньший, но точный контекст дешевле огромного.

4. **Скорость** — меньше токенов в контексте → быстрее ответ LLM.

**Компромисс при выборе размера чанка:**
- Слишком большой (>2048 токенов): размытый эмбеддинг, снижение точности поиска; при передаче в LLM — много нерелевантного шума.
- Слишком маленький (<50 токенов): потеря контекста, фрагменты теряют смысл вне своего окружения.
- Оптимальный старт: 256–512 токенов с перекрытием 50–100 токенов.

### Вопрос 19. Какие основные стратегии чанкинга существуют для RAG? Назовите и кратко охарактеризуйте каждую.

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Weaviate — Chunking Strategies for RAG](https://weaviate.io/blog/chunking-strategies-for-rag)

Руководство Weaviate описывает девять стратегий чанкинга:

**Простые техники:**
- **Фиксированный размер (Fixed-Size)**: делит текст по заданному числу токенов/символов с перекрытием. «Распространённая отправная точка — размер чанка, совпадающий с контекстным окном модели эмбеддингов». Перекрытие обычно 10–20%.
- **Рекурсивный (Recursive)**: использует приоритизированные разделители (абзацы → предложения) для иерархического разбиения. Уважает структуру документа, поддерживая связность на границах.
- **Документально-ориентированный (Document-Based)**: использует встроенное форматирование документа — Markdown по заголовкам, HTML по тегам, код по функциям — для сохранения логической организации.

**Продвинутые техники:**
- **Семантический (Semantic)**: анализирует эмбеддинги предложений для определения тематических переходов, создавая чанки вокруг естественных смысловых границ.
- **На основе LLM (LLM-Based)**: использует языковые модели для определения оптимальных точек разбиения.
- **Агентный (Agentic)**: AI-агент динамически выбирает стратегию чанкинга на основе характеристик документа.
- **Позднее чанкирование (Late Chunking)**: сначала встраивает документы целиком, затем получает эмбеддинги чанков из представлений токенов в полном контексте, сохраняя отношения в масштабе всего документа.
- **Иерархический (Hierarchical)**: создаёт несколько уровней абстракции — от крупных суммарных чанков до детальных — обеспечивая как широкий, так и специфический поиск.
- **Адаптивный (Adaptive)**: динамически корректирует размер и перекрытие чанков на основе плотности контента и семантической сложности.

**Рекомендуемый старт**: 512 токенов с перекрытием 50–100 токенов.

#### Ответ от Claude

**Простые (базовые) стратегии:**

1. **Fixed-Size (фиксированный размер)** — деление по числу токенов (например, 512) с перекрытием. Самый быстрый и простой подход; недостаток — разрывает смысловые границы.

2. **Recursive (рекурсивный)** — иерархическое деление с приоритизированными разделителями: сначала по двойному переносу строки (`\n\n`), затем по одинарному (`\n`), затем по точке и т.д. Уважает структуру документа. **Рекомендуемый дефолт** для большинства случаев.

3. **Document-Based (документно-ориентированный)** — использует форматирование самого документа: заголовки Markdown, HTML-теги, границы функций в коде. Сохраняет логическую структуру документа.

**Продвинутые стратегии:**

4. **Semantic (семантический)** — анализирует изменения в эмбеддингах последовательных предложений; делит текст там, где тема меняется. Лучший recall (+9%), но требует вычислительных ресурсов.

5. **Sliding Window (скользящее окно)** — перекрытие между соседними чанками (10–20%). Гарантирует, что информация на границах не теряется.

6. **Hierarchical (иерархический)** — несколько уровней детализации: крупные чанки для понимания темы + мелкие для точного поиска.

7. **Late Chunking** — сначала весь документ кодируется в полном контексте, затем выделяются чанки. Сохраняет глобальный контекст документа в эмбеддингах чанков.

8. **Agentic** — AI-агент сам выбирает стратегию под тип документа.

9. **Adaptive** — динамически регулирует размер чанка под плотность контента.

### Вопрос 20. В чём разница между фиксированным чанкингом (fixed-size) и семантическим чанкингом? Каковы их компромиссы?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Firecrawl — Best Chunking Strategies for RAG (2026)](https://www.firecrawl.dev/blog/best-chunking-strategies-rag)

**Фиксированный размер** — деление по числу символов или токенов до достижения лимита. Это «самый быстрый в реализации подход (буквально 3 строки кода)» с нулевыми вычислительными накладными расходами. Однако может фрагментировать предложения на полуслове и давать более низкое качество поиска по сравнению с методами, учитывающими структуру.

**Семантический чанкинг** анализирует смысл, генерируя эмбеддинги для каждого предложения, сравнивая сходство между последовательными предложениями и создавая чанки там, где происходит тематический сдвиг. **Компромиссы:**
- Точность: «до 9% лучший recall, чем более простые методы», производительность достигает recall 0.919 в исследованиях Chroma.
- Стоимость: вычислительно затратен — требует встраивания каждого предложения, то есть API-вызовов или инференса локальной модели.
- Практическое замечание: «статья NAACL 2025 Findings пришла к выводу, что вычислительные затраты не оправданы стабильным приростом».

**Рекурсивное разбиение (Recursive)** уважает естественные языковые границы, пытаясь разбить текст на прогрессивно низкоуровневых разделителях: `\n\n`, `\n`, `. `, пробел, символ. «Обрабатывает большинство текстового контента хорошо» и представляет «место, где большинство команд должно начинать» — 85–90% recall при 400–512 токенах.

**Практический победитель**: для большинства приложений рекурсивное разбиение при 400–512 токенах с перекрытием 10–20% балансирует сохранение структуры с простотой реализации, являясь рекомендуемым дефолтом перед переходом к более специализированным подходам.

#### Ответ от Claude

**Фиксированный чанкинг (Fixed-Size):**
Делит текст на части заданного размера (обычно по числу символов или токенов).

- Реализация: буквально 2–3 строки кода.
- Скорость: мгновенная, нет накладных расходов.
- Качество: может разрезать предложения и абзацы посередине, что снижает точность поиска.
- Когда использовать: прототипы, MVP, когда скорость важнее точности.

**Семантический чанкинг (Semantic Chunking):**
Анализирует эмбеддинги последовательных предложений и делит текст там, где значимо меняется тема.

- Реализация: требует генерации эмбеддингов для каждого предложения при индексации.
- Скорость: значительно медленнее — нужен инференс эмбеддинг-модели для каждого предложения.
- Качество: recall на 9% выше, чем у фиксированного чанкинга (по исследованиям Chroma — 0.919).
- Когда использовать: высокоточные системы, где качество важнее стоимости. Исследование NAACL 2025 отмечает, что прирост не всегда стабилен.

**Сравнительная таблица:**

| Параметр | Fixed-Size | Semantic |
|---|---|---|
| Сложность реализации | Минимальная | Высокая |
| Скорость индексации | Быстрая | Медленная |
| Recall | Базовый | +9% |
| Стоимость индексации | Низкая | Высокая |
| Разрыв смысловых границ | Часто | Редко |

**Практическая рекомендация:** большинству команд следует начинать с **recursive-чанкинга** (400–512 токенов, 10–20% перекрытие) — он даёт 85–90% recall при минимальной сложности. Семантический чанкинг стоит рассматривать только когда recursive уже не справляется.
