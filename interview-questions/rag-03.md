### Вопрос 41. Какова проблема устаревания данных (knowledge staleness) в RAG и как она незаметно влияет на качество ответов?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [TianPan.co — The RAG Freshness Problem](https://tianpan.co/blog/2026-04-10-rag-freshness-problem-stale-embeddings-silent-failure)

Векторные эмбеддинги фиксируют смысл документа в момент индексации. После этого они остаются статичными, тогда как реальная информация меняется. Ключевая проблема формулируется так: «семантическое сходство не учитывает время» — устаревший документ по-прежнему возвращает высокий cosine score, даже если его содержимое стало неактуальным. Recall при retrieval может упасть с 0.92 до 0.74 без каких-либо системных алертов.

Выделяются три источника устаревания:

1. **Дрейф исходных документов** — реальная информация меняется (устаревшие функции, изменения политик, цен), а эмбеддинги остаются статичными. Модели, обученные на январских данных, теряют 15–20% точности при применении к июньской информации.
2. **Дрейф модели эмбеддингов** — при обновлении модели возникает геометрическое рассогласование: старые и новые векторы в одном хранилище дают бессмысленные результаты сходства, особенно при частичном переиндексировании.
3. **Дрейф границ чанков** — изменения параметров разбиения (размер чанка, overlap, preprocessing) приводят к тому, что эквивалентный контент получает геометрически различные векторы.

Система при этом выглядит работоспособной: задержки и пропускная способность в норме, ошибки не возникают, а деградация качества распределена по множеству запросов и не проявляется ни в одном из них явно.

#### Ответ от Claude

Проблема устаревания данных в RAG — это «тихий» режим отказа: система продолжает работать технически исправно, но возвращает устаревшие факты с высокой уверенностью.

**Почему сбой не обнаруживается:**
- Cosine-сходство остаётся высоким — устаревший документ семантически близок к запросу, даже если факты в нём изменились
- Latency и throughput не деградируют — стале векторы запрашиваются так же быстро, как свежие
- Деградация размазана по многим запросам — не возникает очевидного единичного сбоя

**Три источника устаревания:**

| Тип | Описание | Эффект |
|-----|----------|--------|
| Дрейф документов | Информация изменилась, эмбеддинг — нет | −15–20% accuracy за несколько месяцев |
| Дрейф модели эмбеддингов | Разные версии модели в одном векторном хранилище | Бессмысленные сравнения сходства |
| Дрейф границ чанков | Изменены параметры chunking-пайплайна | Геометрически несовместимые векторы |

**Метрики для обнаружения:**
- Cosine distance > 0.05 между старым и новым вектором одного документа — сигнал деградации
- Перекрытие топ-10 результатов ниже 70% — активная потеря качества
- Распределение свежести по типам документов (API-доки: 2 недели; compliance: 6 месяцев)

**Решение — Change Data Capture (CDC):** вместо дорогостоящей полной переиндексации (около $12 000/месяц за 1 ТБ при еженедельном переиндексировании) CDC через такие инструменты, как Debezium, обнаруживает только изменённые документы и переиндексирует их с задержкой менее минуты.

### Вопрос 42. Что такое «knowledge decay problem» в RAG-системах и как с ней бороться в продакшне?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [RAG About It — The Knowledge Decay Problem](https://ragaboutit.com/the-knowledge-decay-problem-how-to-build-rag-systems-that-stay-fresh-at-scale/)

По данным источника, 60% корпоративных RAG-проектов терпят неудачу не из-за плохого retrieval или галлюцинаций, а из-за неспособности поддерживать актуальность данных в масштабе. При пакетном индексировании окно устаревания растёт пропорционально размеру базы. Производственный объект с 50 000 документами по безопасности, обновляемыми ежедневно, сталкивается с дорогостоящими циклами полного переиндексирования, в течение которых устаревшие версии остаются доступными.

Ключевые решения:

**Инкрементальная индексация.** Система отслеживает версии документов и обновляет только изменённые чанки, снижая задержку обновления с часов до секунд.

**Freshness-aware retrieval.** Динамическое ранжирование комбинирует семантическое сходство с весами свежести. Рекомендуемое соотношение: 70% релевантность + 30% свежесть.

**Стриминговая интеграция.** Событийно-управляемые архитектуры захватывают изменения документов и распространяют их через пайплайн в течение секунд, поддерживая почти реального-времени синхронизацию.

**Мониторинг устаревания.** Необходимые метрики: максимальное устаревание (возраст самого старого документа), среднее устаревание, гистограмма по возрастным диапазонам. Пороги устаревания задаются по критичности: инструкции по безопасности — 0 дней, справочные материалы — до 90 дней.

#### Ответ от Claude

Knowledge decay problem — системная деградация качества RAG по мере того, как база знаний стареет. В отличие от единичного технического сбоя, это постепенный процесс, который сложно обнаружить до накопления критической массы ошибок.

**Почему проблема масштабируется нелинейно:**
- При батчевой переиндексации окно устаревания = время обработки всей базы
- Чем больше база, тем дольше цикл, тем шире окно
- Во время переиндексирования параллельно накапливаются новые изменения

**Стратегии решения в продакшне:**

| Подход | Механизм | Эффект |
|--------|----------|--------|
| Инкрементальная индексация | Переиндексируются только изменённые чанки | Задержка: часы → секунды |
| Freshness-aware retrieval | Score = 0.7 × semantic_sim + 0.3 × freshness_weight | Новые документы приоритизируются при равной релевантности |
| Стриминговая архитектура | Event-driven: Kafka/Kinesis → embedding → upsert | Синхронизация в реальном времени |
| Разные пороги по типам | Процедуры безопасности: 0 дней; справочники: 90 дней | Ресурсы расходуются пропорционально критичности |

**Метрики мониторинга:**
- `max_staleness` — возраст самого старого активного документа
- `avg_staleness` — медианный возраст документов в индексе
- Гистограмма распределения по возрастным диапазонам
- Автоматические алерты при превышении порогов критичных категорий

**Ключевой принцип:** свежесть базы знаний должна быть архитектурным требованием первого класса, а не постфактум-оптимизацией. Системы, не спроектированные под обновление с самого начала, рано или поздно требуют дорогостоящего рефакторинга в продакшне.

### Вопрос 43. Почему неэффективное разбиение на чанки является распространённой причиной сбоев RAG и как выбрать правильную стратегию?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Unstructured.io — Chunking for RAG Best Practices](https://unstructured.io/blog/chunking-for-rag-best-practices)

Чанкинг делит документы на небольшие фрагменты, которые retriever может извлечь и передать в контекст LLM. Два ключевых ограничения определяют эту необходимость: лимит контекстного окна LLM и лимит контекстного окна самих embedding-моделей (обычно не более 8К токенов).

Проблемы разных подходов:

- **Разбиение по символам** — делит текст на N-символьные куски с опциональным overlap, но «часто разрезает предложения пополам или даже посреди слова», полностью игнорируя структуру документа.
- **Рекурсивное/посентенсное разбиение** — использует иерархические разделители (абзацы, переводы строк, точки), снижая разрывы посреди предложений, но по-прежнему «не улавливает сложную структуру документа».

Рекомендуемые структурно-осознанные стратегии от Unstructured:

- **By Character** — комбинирует последовательные элементы, максимизируя размер чанка в рамках лимита
- **By Title** — «сохраняет границы секций», чтобы чанки не смешивали разные темы
- **By Page** — предотвращает объединение контента с разных страниц
- **By Similarity** — группирует тематически схожие элементы с помощью embedding-моделей, когда структура документа сама по себе не определяет границы

Рекомендации: начинать с ~250 токенов (~1000 символов), использовать структурно-осознанные подходы, подбирать overlap экспериментально, оценивать влияние на общую производительность RAG на доменных тест-сетах.

#### Ответ от Claude

Разбиение на чанки — фундаментальный шаг RAG-пайплайна, и ошибки здесь нельзя исправить downstream. Плохой чанкинг деградирует retrieval ещё до того, как LLM получит хоть один документ.

**Почему чанкинг критичен:**
- Embedding-модель применяет pooling к токенам чанка, сжимая их в один вектор
- Если чанк содержит несколько несвязанных тем — вектор «размывается», точность retrieval падает
- Слишком мелкие чанки теряют контекст, необходимый для ответа на вопрос
- Слишком крупные чанки засоряют контекстное окно нерелевантной информацией

**Сравнение стратегий чанкинга:**

| Стратегия | Как работает | Слабые стороны |
|-----------|-------------|----------------|
| По символам | Фиксированный N символов + overlap | Разрезает предложения, игнорирует структуру |
| Рекурсивная | Иерархические разделители (абзац → строка → точка) | Не улавливает сложную структуру документа |
| По заголовкам (By Title) | Границы секций = границы чанков | Требует структурированных документов |
| По странице (By Page) | Одна страница = один чанк | Размер варьируется |
| По сходству (By Similarity) | Embedding-кластеризация смежных элементов | Дороже вычислительно |

**Практические рекомендации:**
1. Стартовая точка — ~250 токенов (~1000 символов)
2. Использовать структурно-осознанный подход (By Title / By Page) для структурированных документов
3. Подбирать overlap экспериментально — сохраняет continuity между чанками без излишней избыточности
4. Оценивать на доменных тест-сетах: изменение стратегии чанкинга может как улучшить, так и ухудшить retrieval

### Вопрос 44. Почему обобщённые embedding-модели дают плохое качество retrieval в RAG для специализированных доменов?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Redis — Get Better RAG by Fine-Tuning Embedding Models](https://redis.io/blog/get-better-rag-by-fine-tuning-embedding-models/)

Обобщённые embedding-модели обучены на широких текстовых корпусах и не отражают специфику конкретных отраслей: финансовой, медицинской, юридической. Они не понимают отраслевую терминологию и концепции, что ведёт к неточному retrieval.

**Решение — дообучение (fine-tuning)** на доменных данных: парах (вопрос, релевантный документ). Это выравнивает метрику сходства с отраслевым контекстом.

**Технические детали:**
- Используемая модель: `BAAI/bge-base-en-v1.5`
- Форматы датасетов: положительные пары, триплеты (anchor, positive, negative), пары с оценкой сходства
- Рекомендуемая функция потерь для QA-датасетов: `MultipleNegativesRankingLoss`
- Генерация синтетических данных: LLM создаёт вопросо-ответные пары из чанков документов

**Результаты на финансовых документах (NVIDIA 10-K):**

| Метрика | До дообучения | После дообучения | Прирост |
|---------|--------------|-----------------|---------|
| NDCG@10 | 0.5950 | 0.8246 | **+38.4%** |

#### Ответ от Claude

Обобщённые embedding-модели оптимизированы для широкого семантического понимания, но не для точной близости в узком доменном пространстве. Это фундаментальное несоответствие.

**Почему общие модели не справляются:**
- Одинаковые слова имеют разный смысл в разных доменах (например, «pitch» в спорте и в финансах, «yield» в агрономии и в облигациях)
- Модель не знает, что в конкретном домене одни термины взаимозаменяемы, а другие — нет
- Векторное пространство общей модели не выровнено по оси «отраслевая релевантность»

**Процесс дообучения:**

1. Извлечь и разбить на чанки доменные документы
2. Сгенерировать синтетические QA-пары с помощью LLM (вопрос + релевантный чанк)
3. Загрузить предобученную модель (`BAAI/bge-base-en-v1.5` или аналог)
4. Обучить с `MultipleNegativesRankingLoss` — идеален для пар (вопрос, ответ) без явных негативных примеров
5. Оценить через `InformationRetrievalEvaluator` с метрикой NDCG@10

**Практический результат:** на финансовых документах (NVIDIA 10-K) прирост NDCG@10 составил +38.4% (с 0.595 до 0.825).

**Когда fine-tuning оправдан:**
- Датасет содержит специализированную терминологию
- Стандартные модели дают неудовлетворительный recall на domain-specific запросах
- Есть возможность сгенерировать или собрать достаточный объём доменных пар (от нескольких сотен)

### Вопрос 45. Что такое двухэтапный retrieval с cross-encoder reranker в RAG и какой прирост качества он даёт?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [BigData Boutique — RAG Reranking with Cross-Encoders](https://bigdataboutique.com/blog/rag-reranking-improving-retrieval-quality-with-cross-encoders)

Продакшн-RAG системы используют каскадную архитектуру из двух этапов:

1. **Первый этап (максимизация recall):** Bi-encoder embeddings + BM25 выполняют гибридный поиск по всему корпусу, возвращая 50–100 кандидатов через approximate nearest neighbor. Задача этапа — «найти правильный ответ где-то в топ-50–200, а не поставить его на первое место».
2. **Второй этап (максимизация precision):** Cross-encoder reranker переоценивает кандидатов и сужает результат до 3–5 лучших документов для передачи в LLM.

**Ключевое различие между bi-encoder и cross-encoder:**
- **Bi-encoder** кодирует запрос и документ независимо в фиксированные векторы, вычисляя сходство через dot product. Это быстро, но сходство «сглаживает» нюансы.
- **Cross-encoder** объединяет запрос и документ в одну последовательность, применяя полный self-attention к обоим. Это улавливает «отрицания, условия, тонкие семантические взаимодействия», которые bi-encoder пропускает, — но в 10–100 раз дороже на пару.

**Производительность:**
- Типичный прирост NDCG@10: +5–15 пунктов
- На лексически сложных датасетах (редкие сущности, ID, коды ошибок): +20 пунктов
- Добавляемая задержка: менее 200 мс (MiniLM-класс: 50–80 мс, BGE v2-m3: 80–200 мс)

#### Ответ от Claude

Двухэтапный retrieval — стандарт для продакшн-RAG, где важно одновременно не пропустить нужные документы (recall) и отдать LLM только самые релевантные (precision).

**Архитектура:**

```
Запрос
  │
  ▼
[Этап 1: Recall] Bi-encoder + BM25
→ 50–100 кандидатов (быстро, ~мс)
  │
  ▼
[Этап 2: Precision] Cross-encoder reranker
→ топ 3–5 документов (точно, +50–200 мс)
  │
  ▼
LLM генерирует ответ
```

**Почему bi-encoder недостаточен для точного ранжирования:**
- Независимое кодирование запроса и документа не улавливает взаимодействия между ними
- Отрицания («не поддерживает X»), условия («работает только если Y»), омонимы — всё это bi-encoder сглаживает
- Cross-encoder видит оба текста одновременно с полным self-attention — пропустить такие нюансы значительно сложнее

**Компромиссы:**

| Параметр | Bi-encoder | Cross-encoder |
|----------|-----------|---------------|
| Скорость индексации | Быстро (offline) | Не нужна |
| Скорость inference | ~мс на запрос | 50–400 мс на N кандидатов |
| Точность ранжирования | Средняя | Высокая |
| Применение | Первичный поиск | Вторичное ранжирование |

**Когда reranker не подходит:** при бюджете latency менее 50 мс end-to-end или при числе кандидатов свыше ~200 (прирост качества начинает убывать).

### Вопрос 46. Почему чистый векторный поиск недостаточен для RAG и в чём преимущество гибридного поиска (BM25 + dense retrieval)?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Fuzzy Labs — Improving RAG Performance: Hybrid Search](https://www.fuzzylabs.ai/blog-post/improving-rag-performance-hybrid-search)

Источник описывает три метода retrieval и обосновывает их комбинирование:

**Dense retrieval (семантический поиск):** сравнивает эмбеддинги запроса с сохранёнными векторами, улавливая смысл даже при разной терминологии. Но модели памяти-интенсивны, хранение плотных векторов требует значительного места, и они «не очень хороши для точного совпадения по ключевым словам, что является недостатком в ряде случаев».

**Sparse retrieval (на основе терминов):** фокусируется на ключевых словах и их весах, отлично работает для точного совпадения специфических терминов.

**BM25 (полнотекстовый поиск):** статистический алгоритм, улучшающий TF-IDF за счёт учёта насыщения терминов и длины документа. Обеспечивает «быстрые вычисления благодаря математической природе» без необходимости в векторных эмбеддингах.

**Гибридный поиск** объединяет все три подхода. Методы слияния результатов:
- **Reciprocal Rank Fusion:** математическая формула, ценящая согласованность между методами
- **Weighted Fusion:** разные веса для каждого подхода
- **Re-Ranker Models:** cross-encoder системы, переоценивающие топ-результаты

Гибридный поиск максимизирует recall и precision за счёт взаимодополняющих сильных сторон, хотя усложняет реализацию и немного снижает скорость из-за параллельной обработки.

#### Ответ от Claude

Векторный поиск — мощный инструмент для семантического понимания, но у него есть слепые пятна, которые критичны в реальных RAG-сценариях.

**Слабые места чистого dense retrieval:**

| Сценарий | Проблема | Пример |
|----------|----------|--------|
| Специфические термины | Эмбеддинг «размывает» уникальные идентификаторы | Поиск по коду ошибки `NullPointerException` или артикулу `SKU-4892` |
| Собственные имена | Семантически близкие, но фактически разные | «Иванов» vs «Иванова» |
| Точные цитаты | Эмбеддинг не гарантирует точное лексическое совпадение | Поиск конкретной фразы из договора |
| Out-of-distribution термины | Новые слова не в vocabulary модели | Аббревиатуры, брендовые названия |

**BM25 закрывает эти пробелы:** это статистический алгоритм, вычисляющий вес термина через TF-IDF с поправками на насыщение и длину документа. Не требует эмбеддингов, быстро вычисляется, идеально точен для ключевых слов.

**Гибридный поиск:**
```
Запрос
  ├──→ Dense retrieval (семантика) ──┐
  ├──→ BM25 (ключевые слова) ────────┼──→ Fusion (RRF / Weighted) ──→ Результат
  └──→ Sparse retrieval (термины) ───┘
```

**Методы слияния:**
- **Reciprocal Rank Fusion (RRF):** score = Σ 1/(rank_i + k) — ценит позиции в разных списках
- **Weighted Fusion:** явные веса для семантики и лексики (настраиваются под домен)
- **Re-Ranker:** cross-encoder на объединённых кандидатах для финального упорядочивания

**Практика:** гибридный поиск улучшает NDCG примерно на 7.4% по сравнению с каждым методом по отдельности на типичных бенчмарках. Доступен из коробки в Elasticsearch, OpenSearch, Weaviate, Qdrant.

### Вопрос 47. Что такое Corrective RAG (CRAG) и как он устраняет проблему галлюцинаций в стандартных RAG-системах?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Chitika — Implementing cRAG to Prevent AI Hallucinations](https://www.chitika.com/corrective-rag-hallucinations/)

Стандартный RAG пассивно передаёт извлечённые документы в LLM без проверки их качества. CRAG добавляет активный контроль через несколько механизмов:

**Retrieval Evaluator с confidence scoring.** Каждому извлечённому документу присваивается оценка доверия. Документы с низкими оценками фильтруются до того, как достигнут генеративной модели.

**Система оценки CRAG Benchmark.** Использует дифференциальное оценивание: правильные ответы получают +1, галлюцинации наказываются штрафом −1. Галлюцинации наказываются тяжелее пропущенных ответов, поскольку «уверенный неправильный ответ наносит больше вреда, чем признание неопределённости».

**Reflection-токены.** Работают как встроенный контрольный список, «обнаруживая пробелы и противоречия» в извлечённых данных и запуская корректирующие циклы при обнаружении расхождений.

**Динамическая коррекция retrieval.** Вместо статичного пайплайна CRAG «динамически корректирует стратегии retrieval в зависимости от контекста запроса»: декомпозиция запросов, контекстуальное переранжирование, интеграция веб-поиска в реальном времени, параллельная обработка и кросс-верификация по нескольким источникам.

#### Ответ от Claude

Corrective RAG (CRAG) — это расширение стандартного RAG, добавляющее слой самопроверки и адаптивного реагирования на низкокачественные извлечения.

**Сравнение стандартного RAG и CRAG:**

| Аспект | Стандартный RAG | CRAG |
|--------|----------------|------|
| Качество retrieval | Принимается как есть | Оценивается confidence score |
| Низкокачественные документы | Попадают в контекст LLM | Фильтруются |
| Пробелы в знаниях | Игнорируются | Обнаруживаются reflection-токенами |
| Стратегия retrieval | Фиксированная | Динамически корректируется |
| Реакция на неопределённость | Галлюцинация | Признание пробела или повторный поиск |

**Ключевые механизмы CRAG:**

1. **Retrieval Evaluator** — классификатор/скорер, оценивающий релевантность и достоверность каждого извлечённого документа
2. **Confidence threshold** — документы ниже порога отфильтровываются или заменяются результатами веб-поиска
3. **Reflection tokens** — специальные токены в промпте или пайплайне, запускающие цикл самопроверки при обнаружении противоречий
4. **Adaptive retrieval** — при низком confidence система может: расширить запрос, переформулировать его, обратиться к внешним источникам

**Принцип оценки:** галлюцинация (−1) хуже, чем отказ от ответа (0). Это намеренный дизайн для высокорисковых доменов (медицина, финансы, право), где ложная уверенность опаснее признания незнания.

**Практические результаты:** интеграция verifier-модулей позволила снизить частоту галлюцинаций до 30% согласно данным отдельных реализаций.

### Вопрос 48. Каковы типичные режимы отказа (failure modes) RAG-систем согласно классификации на 7 категорий?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Label Studio — Seven RAG Pitfalls and How to Fix Them](https://labelstud.io/blog/seven-ways-your-rag-system-could-be-failing-and-how-to-fix-them/)

Семь режимов отказа RAG-систем:

1. **Missing Content (Отсутствующий контент).** База знаний не содержит информацию для ответа. *Исправление:* убедиться, что база полна, актуальна и хорошо структурирована; валидировать покрытие через ручной ревью.

2. **Missing Top Ranked Documents (Нужный документ есть, но ранжируется слишком низко).** Релевантные документы существуют, но получают низкие оценки релевантности. *Исправление:* «оптимизировать ранжирование при retrieval путём дообучения ranker- и reranker-моделей»; скорректировать стратегии для поднятия критических чанков.

3. **Not in Context (Документ найден, но не попал в контекст).** Извлечённые документы не помещаются в контекстное окно модели из-за обрезки или плохого чанкинга. *Исправление:* скорректировать количество извлекаемых результатов, улучшить подход к обрезке, оптимизировать чанкинг для сохранения логической структуры.

4. **Not Extracted (Контекст есть, но LLM не извлёк ответ).** Модель не находит ответ в предоставленном контексте из-за шума или противоречий. *Исправление:* «удалить дубликаты и конфликтующую информацию из базы»; оптимизировать промпты для точности.

5. **Wrong Format (Неправильный формат ответа).** Ответы не имеют требуемой структуры (например, текст вместо JSON). *Исправление:* «использовать структурированные выводы, такие как JSON или предопределённые схемы»; задействовать принуждение к схеме.

6. **Incorrect Specificity (Слишком общий или слишком узкий ответ).** *Исправление:* предварительно обрабатывать запросы для уточнения намерения, показывать примеры вопросов, реализовать backend-переписывание запросов.

7. **Incomplete Answer (Неполный ответ на сложный многогранный вопрос).** *Исправление:* декомпозировать сложные запросы на подвопросы; использовать корректировки ранжирования для полного покрытия.

#### Ответ от Claude

Систематизация режимов отказа помогает диагностировать конкретную проблему вместо слепой оптимизации всего пайплайна сразу.

**Семь режимов отказа и их устранение:**

| # | Режим отказа | Где ломается | Метод устранения |
|---|-------------|-------------|-----------------|
| 1 | Missing Content | База знаний | Аудит покрытия, обновление базы |
| 2 | Missing Top Ranked | Retrieval (ранжирование) | Fine-tuning ranker/reranker |
| 3 | Not in Context | Chunking / context window | Оптимизация чанкинга, усечения |
| 4 | Not Extracted | LLM (промпт) | Дедупликация базы, оптимизация промпта |
| 5 | Wrong Format | LLM (генерация) | Structured outputs, JSON schema |
| 6 | Incorrect Specificity | Запрос / LLM | Query rewriting, примеры в промпте |
| 7 | Incomplete Answer | Retrieval + LLM | Query decomposition, расширенный топ-K |

**Диагностика по симптому:**
- Система говорит «не знаю» на вопросы, ответ на которые должен быть в базе → режим 1 или 2
- Ответ есть, но неточный / содержит факты из контекста лишь частично → режим 3 или 4
- Ответ семантически верен, но не в том формате → режим 5
- Ответ на часть вопроса, но не на весь → режим 7

**Практический подход:** для каждого режима отказа используется свой слой диагностики. Полезно логировать промежуточные результаты (retrieved chunks, их scores, финальный prompt) — это позволяет точно определить, на каком этапе возникает сбой.

### Вопрос 49. Как HyDE (Hypothetical Document Embedding) и multi-query retrieval улучшают качество retrieval в RAG?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [The Geo Community — Query Rewriting & Multi-Query RAG](https://thegeocommunity.com/blogs/generative-engine-optimization/query-rewriting-multiquery-rag/)

**HyDE (Hypothetical Document Embedding)** генерирует гипотетический ответ на вопрос и использует его эмбеддинг для поиска вместо эмбеддинга исходного запроса. Техника «работает, когда корпус плотный и описательный, например техническая документация или длинные руководства».

Оптимальные сценарии для HyDE:
- Короткие запросы, сопоставляемые с объёмными документами: HyDE перекидывает мост между кратким вводом и расширенным контентом
- Нарративный контент: лучше подходит, когда стиль формулировок важнее совпадения ключевых слов
- Допустимо некоторое семантическое смещение

**Критическое ограничение:** следует избегать в чувствительных к соответствию контекстах, поскольку «галлюцинированные детали могут сместить retrieval к нерелевантным документам».

**Multi-query retrieval** — это фаза выполнения: систематически запускает несколько переформулировок запроса параллельно и объединяет результаты. Делает неполноту попытки одиночного поиска «явной в пайплайне».

Оптимизированный пайплайн:
- Генерировать 3–6 перефразировок с использованием разных стратегий
- Параллельный retrieval с консервативными значениями k (5–10 на каждую переформулировку)
- Дедупликация и переранжирование для восстановления точности

#### Ответ от Claude

HyDE и multi-query решают одну и ту же задачу — устранить разрыв между кратким пользовательским запросом и документами в корпусе — но разными механизмами.

**HyDE (Hypothetical Document Embedding):**

```
Запрос → LLM генерирует гипотетический ответ
                    ↓
           Эмбеддинг гипотетического ответа
                    ↓
           Поиск в векторном хранилище
```

*Почему это работает:* эмбеддинг гипотетического ответа семантически ближе к стилю реальных документов, чем эмбеддинг короткого вопроса. Запрос «Как настроить Kafka?» имеет иной вектор, чем документ «Настройка Kafka: шаги 1–5...» — а гипотетический ответ LLM будет похож по стилю на реальный документ.

*Ограничение:* если LLM галлюцинирует в гипотетическом ответе, поиск будет смещён к нерелевантным документам.

**Multi-Query Retrieval:**

```
Запрос → LLM генерирует 3–6 переформулировок
              ↓ (параллельно)
        Retrieval для каждой переформулировки (k=5–10)
              ↓
        Объединение + дедупликация + reranking
              ↓
        Финальный контекст для LLM
```

*Почему это работает:* один запрос может не найти все релевантные документы из-за несовпадения терминологии. Несколько переформулировок покрывают разные аспекты и расширяют recall без изменений в корпусе.

**Сравнение методов:**

| Метод | Механизм | Лучший сценарий | Риск |
|-------|----------|----------------|------|
| HyDE | Гипотетический ответ как query | Короткий запрос + длинные документы | Галлюцинации смещают поиск |
| Multi-query | Параллельные переформулировки | Сложные многоаспектные запросы | Рост latency и стоимости |

**Практика:** на датасете AmbigQA комбинация MQRF-RAG превзошла HyDE на 1.47% по Exact Match и на 3.75% по F1.

### Вопрос 50. Как RAGAS применяется в CI/CD для автоматической оценки RAG-пайплайна в продакшне?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [INVRA — RAG Evaluation with RAGAS in Production](https://www.invra.co/en/rag-evaluation-with-ragas-measuring-faithfulness-context-precision-and-recall-in-production/)

RAGAS (Retrieval Augmented Generation Assessment) — open-source Python-фреймворк, оценивающий RAG-системы путём декомпозиции сбоев на проблемы retrieval и генерации. Обрабатывает свыше 5 миллионов оценок ежемесячно и принят AWS, Microsoft, Databricks. Поддержан Y Combinator.

**Четыре ключевые метрики:**

**Faithfulness** (Верность) — обнаруживает галлюцинации. Извлекает утверждения из сгенерированного ответа и проверяет, может ли каждое быть логически выведено из извлечённого контекста. Диапазон: 0–1.
*Исправление при низком значении:* ужесточить системный промпт с явными инструкциями заземления; снизить temperature до 0 для фактических задач.

**Answer Relevancy** (Релевантность ответа) — измеряет, отвечает ли ответ напрямую на вопрос пользователя. Генерирует гипотетические вопросы, на которые отвечал бы данный ответ, и вычисляет их сходство с оригинальным вопросом.
*Исправление:* явно привязать промпты к входным вопросам; если релевантность низкая, а faithfulness нормальная — проблема в качестве retrieval.

**Context Precision** (Точность контекста) — оценивает качество ранжирования retrieval через Average Precision; штрафует системы, где релевантные чанки погребены среди нерелевантных.
*Исправление:* развернуть cross-encoder reranker; оптимизировать размер чанков через систематическое тестирование (256–1024 токена).

**Context Recall** (Полнота контекста) — определяет, была ли захвачена вся необходимая информация. Декомпозирует эталонные ответы на утверждения и проверяет их атрибуцию к извлечённому контексту.
*Исправление:* увеличить глубину top-K retrieval; применить гибридный поиск.

**Интеграция в CI/CD:** RAGAS встраивается как набор unit-тестов. Перед каждым изменением промпта, сменой модели или обновлением базы знаний запускается автоматическая оценка. Интеграция с LangSmith и Langfuse позволяет трассировать низкоэффективные запросы.

**Ограничение:** RAGAS использует LLM в качестве судей, что вносит потенциальные несоответствия, чувствительность к промптам и позиционный bias. В критических приложениях необходима валидация с участием людей.

#### Ответ от Claude

RAGAS превращает субъективную оценку качества RAG в количественные метрики, которые можно встроить в CI/CD как автоматические тесты.

**Четыре метрики и их интерпретация:**

| Метрика | Что измеряет | Низкое значение означает |
|---------|-------------|------------------------|
| Faithfulness | Есть ли утверждения в ответе, не подкреплённые контекстом | LLM галлюцинирует |
| Answer Relevancy | Отвечает ли ответ на заданный вопрос | Ответ уклончив или не по теме |
| Context Precision | Насколько высоко ранжируются релевантные чанки | Retrieval возвращает много шума |
| Context Recall | Все ли нужные факты нашлись при retrieval | Пропущены релевантные документы |

**Диагностические комбинации:**
- Низкий Faithfulness + высокий Context Recall → проблема в LLM (галлюцинирует вопреки хорошему контексту)
- Низкий Context Recall + высокий Context Precision → retrieval находит мало документов, но хороших
- Низкий Answer Relevancy + нормальный Faithfulness → LLM отвечает точно, но не на тот вопрос, что нашёл retrieval

**CI/CD интеграция:**
```python
from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy, context_precision, context_recall

# Запускается перед каждым деплоем
results = evaluate(
    dataset,
    metrics=[faithfulness, answer_relevancy, context_precision, context_recall]
)

# Порог — gate для деплоя
assert results["faithfulness"] >= 0.85
assert results["context_recall"] >= 0.80
```

**Ограничение:** RAGAS использует LLM как судью, что означает чувствительность к промпту и позиционный bias. Для критических применений нужна периодическая валидация с участием людей.

### Вопрос 51. Как проблема переполнения контекстного окна влияет на качество RAG и какие стратегии управления токенами существуют?

**Категория:** RAG (Retrieval-Augmented Generation) · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Agenta — Top Techniques to Manage Context Length in LLMs](https://agenta.ai/blog/top-6-techniques-to-manage-context-length-in-llms)

Контекстное окно определяет максимальное количество токенов, которое LLM может обработать в одном запросе. Когда суммарный объём запроса и извлечённых чанков превышает этот лимит, приложение либо полностью завершается с ошибкой, либо контекст принудительно обрезается.

Проблему усугубляет **эффект «lost-in-the-middle»**: при заполнении окна модели «склонны придавать больший вес началу и концу промпта», потенциально недооценивая критически важный контент в середине.

**Шесть стратегий управления:**

1. **Усечение (Truncation):** наиболее простой подход — обрезать лишнее до нужного размера. Вычислительно эффективно и совместимо с любой системой, но «не учитывает семантику» и рискует потерять существенную информацию.

2. **Маршрутизация к моделям с большим окном:** при превышении лимита динамически переключаться на модель с расширенным контекстным окном. Библиотеки типа LiteLLM обеспечивают бесшовную смену моделей, хотя это увеличивает стоимость.

3. **Буферизация памяти:** особенно подходит для чат-приложений. Хранит и организует прошлые диалоги, «периодически резюмируя взаимодействия, сохраняя при этом критические сущности».

4. **Иерархическая суммаризация:** «сжимает длинные документы в многоуровневые резюме (как пирамиду)» для сохранения ключевой информации при минимизации токенов. Текст последовательно суммаризируется на нескольких уровнях.

5. **Контекстная компрессия:** в отличие от суммаризации, «сохраняет оригинальные формулировки, но удаляет избыточность», потенциально достигая «сокращения токенов на 40–60%». Строит графы знаний из извлечённых сущностей и отношений.

6. **RAG:** наиболее комплексный подход — «извлечение только наиболее релевантной информации в момент запроса, вместо того чтобы пытаться поместить весь документ, историю или датасет в промпт».

#### Ответ от Claude

Переполнение контекстного окна — одна из наиболее практических проблем продакшн-RAG. Даже с большими контекстными окнами современных моделей проблема не исчезает, а трансформируется.

**Почему большое окно не решает проблему:**
- Стоимость инференса растёт линейно (или квадратично для некоторых архитектур) с числом токенов
- Эффект «lost-in-the-middle»: чем больше документов в контексте, тем хуже модель обрабатывает те, что в середине
- Качество ответа может снизиться при заполнении контекста нерелевантными чанками

**Стратегии управления токенами:**

| Стратегия | Механизм | Лучший сценарий | Потеря качества |
|-----------|----------|----------------|-----------------|
| Усечение | Обрезать по лимиту | Быстро, без логики | Высокая (произвольная потеря) |
| Routing к большей модели | LiteLLM автоматически | Непредсказуемый объём | Нет (дороже) |
| Буферизация памяти | Резюме + ключевые сущности | Многосессионный чат | Низкая |
| Иерархическая суммаризация | Пирамида резюме | Длинные документы | Умеренная |
| Контекстная компрессия | Удаление избыточности | Точность важна | Минимальная (−40–60% токенов) |
| RAG | Retrieval только нужного | Большие базы знаний | Нет при хорошем retrieval |

**Практические рекомендации:**
1. Мониторинг заполненности окна — алерт при 80% использования
2. Контекстная компрессия перед RAG: сначала сжать чанки, потом передавать в LLM
3. Ограничивать top-K при retrieval: больше документов не всегда лучше
4. Для Q&A — RAG; для чата — буферизация памяти; для длинных документов — иерархическая суммаризация
