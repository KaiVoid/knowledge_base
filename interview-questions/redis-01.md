### Вопрос 1. Что такое Redis и для чего он используется?

**Категория:** Redis · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Redis Docs — Understand Redis data types](https://redis.io/docs/latest/develop/data-types/)

«Redis — это сервер структур данных. По своей сути Redis предоставляет набор нативных типов данных, которые помогают решать широкий круг задач: от кеширования до организации очередей и обработки событий». Redis хранит данные в оперативной памяти, работает по модели «ключ — значение» и может использоваться как база данных, кеш, брокер сообщений и очередь.

#### Ответ от Claude

**Redis** (REmote DIctionary Server) — это in-memory хранилище структур данных с открытым исходным кодом. Ключевые свойства:

- **In-memory.** Данные хранятся в RAM, поэтому операции выполняются за микросекунды; для надёжности есть персистентность на диск (см. [вопрос 5](#вопрос-5-чем-отличаются-механизмы-персистентности-rdb-и-aof)).
- **Модель «ключ — значение», но с богатыми типами значений.** Значение — это не просто строка, а одна из структур: строки, списки, множества, хеши, упорядоченные множества, потоки и др. (см. [вопрос 2](#вопрос-2-какие-основные-типы-данных-поддерживает-redis)).
- **Многофункциональность.** Применяется как кеш (чаще всего), хранилище сессий, счётчики/лидерборды (sorted sets), брокер сообщений (Pub/Sub, Streams), распределённые блокировки, rate limiting.

Главное отличие от реляционной СУБД: Redis не предназначен для сложных запросов с JOIN и реляционной целостности — он оптимизирован под быстрый доступ по ключу. От Memcached отличается богатыми типами данных, персистентностью, репликацией и кластеризацией.

### Вопрос 2. Какие основные типы данных поддерживает Redis?

**Категория:** Redis · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Redis Docs — Understand Redis data types](https://redis.io/docs/latest/develop/data-types/)

Базовые типы данных Redis:
- **Strings** — самый простой тип, последовательность байт.
- **Lists** — списки строк, упорядоченные по порядку вставки.
- **Sets** — неупорядоченные коллекции уникальных строк.
- **Hashes** — наборы пар «поле — значение» (как словарь).
- **Sorted Sets** — коллекции уникальных строк, упорядоченные по числовому score.
- **Streams** — структура типа append-only лога для записи событий по порядку.
- **Bitmaps**, **HyperLogLog**, **Geospatial indexes**, **Bitfields** — специализированные типы.

#### Ответ от Claude

**Базовые типы:**

| Тип | Что это | Типичное применение |
|---|---|---|
| **String** | строка/число до 512 МБ | кеш значений, счётчики (`INCR`), флаги |
| **List** | связный список строк (доступ по индексу, push/pop с обоих концов) | очереди, стеки, ленты |
| **Set** | неупорядоченное множество уникальных строк | теги, уникальные посетители, операции пересечения/объединения |
| **Hash** | словарь «поле → значение» внутри одного ключа | объекты (профиль пользователя) |
| **Sorted Set (ZSet)** | множество с числовым score, отсортированное | лидерборды, очереди с приоритетом, time-series |

**Специализированные типы:**

- **Streams** — append-only лог с consumer groups; основа для надёжных очередей сообщений (в отличие от Pub/Sub, см. [вопрос 8](#вопрос-8-как-работает-механизм-pubsub-в-redis-и-чем-он-отличается-от-streams)).
- **Bitmaps** — битовые операции над строкой (компактные флаги, например «заходил ли пользователь в день N»).
- **HyperLogLog** — вероятностная оценка кардинальности (число уникальных элементов) при минимальном расходе памяти.
- **Geospatial** — геопоиск по радиусу/координатам.

В современных версиях через модули доступны также JSON, поиск (RediSearch), time-series и др.

### Вопрос 3. Почему Redis такой быстрый и как он обрабатывает команды (однопоточная модель)?

**Категория:** Redis · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Redis Docs — Redis client handling](https://redis.io/docs/latest/develop/reference/clients/)

Redis использует неблокирующий ввод-вывод и мультиплексирование для обслуживания множества соединений. Согласно документации: «когда из клиента прочитаны новые данные, все запросы, содержащиеся в текущих буферах, обрабатываются последовательно (sequentially). Это улучшает локальность и избавляет от необходимости повторного прохода, чтобы понять, каким клиентам нужно время на обработку».

#### Ответ от Claude

Высокая скорость Redis обусловлена сочетанием нескольких факторов:

1. **Данные в оперативной памяти.** Нет дисковых обращений на «горячем» пути чтения/записи.
2. **Однопоточная обработка команд.** Команды выполняются одним потоком в цикле событий (event loop) на базе мультиплексирования I/O (epoll/kqueue). Это даёт два преимущества:
   - **Нет блокировок и переключений контекста** между потоками — отсутствует борьба за мьютексы.
   - **Атомарность.** Раз команды выполняются строго последовательно, каждая отдельная команда (и даже скрипт Lua / транзакция) атомарна по построению — никакой другой клиент не «вклинится» в середину.
3. **Эффективные структуры данных** с оптимизированными представлениями в памяти.
4. **Простой протокол RESP** и пакетная обработка (см. пайплайнинг, [вопрос 9](#вопрос-9-что-такое-пайплайнинг-pipelining-в-redis)).

**Важное уточнение (расхождение «однопоточности» в разных версиях):** строго однопоточным является именно *исполнение команд*. При этом ещё с ранних версий вспомогательные задачи (например, fork для RDB, закрытие соединений) выполняются отдельно, а начиная с Redis 6 появились **I/O-threads** — отдельные потоки для чтения/записи сетевых данных и парсинга. Само исполнение команд по-прежнему происходит в одном потоке. Поэтому утверждение «Redis полностью однопоточный» — упрощение: верно «команды исполняются в одном потоке».

**Следствие:** одна медленная команда `O(N)` (например, `KEYS *` на большой базе) блокирует весь сервер. Поэтому в продакшене вместо `KEYS` используют `SCAN`.

### Вопрос 4. Как работает истечение срока жизни ключей (TTL/expiration) в Redis?

**Категория:** Redis · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Redis Docs — EXPIRE command](https://redis.io/docs/latest/commands/expire/)

`EXPIRE` устанавливает тайм-аут для ключа; по истечении срока ключ автоматически удаляется. `TTL` возвращает оставшееся время жизни. Тайм-аут сбрасывается только командами, которые удаляют или полностью перезаписывают значение (`DEL`, `SET`, `GETSET`). Команды, изменяющие значение без замены (`INCR`, `LPUSH`, `HSET`), тайм-аут не трогают. `PERSIST` снимает срок жизни, делая ключ постоянным.

#### Ответ от Claude

**Установка и проверка срока:**
- `EXPIRE key seconds` / `PEXPIRE key ms` — задать тайм-аут; `EXPIREAT` — на момент времени (Unix timestamp).
- `TTL key` / `PTTL key` — оставшееся время; `PERSIST key` — снять срок.
- `SET key val EX 60` — установить значение со сроком за одну команду.

**Как именно удаляются истёкшие ключи.** Redis использует два механизма одновременно:

1. **Пассивное (ленивое) удаление.** Когда клиент обращается к ключу, Redis проверяет, не истёк ли он; если истёк — удаляет и возвращает «нет ключа». Проблема: ключ, к которому никто не обращается, занимал бы память бесконечно.
2. **Активное удаление.** Redis периодически (≈10 раз в секунду) берёт случайную выборку ключей с TTL, удаляет истёкшие; если истёкших оказалось много (порог), повторяет цикл. Это вероятностный алгоритм, не дающий точной гарантии момента удаления.

**Важно для собеседования:** истечение TTL — не точное событие. Ключ физически удаляется либо при первом обращении после истечения, либо при следующей активной проверке. Память может оставаться занятой какое-то время после формального истечения срока.

### Вопрос 5. Чем отличаются механизмы персистентности RDB и AOF?

**Категория:** Redis · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Redis Docs — Redis persistence](https://redis.io/docs/latest/operate/oss_and_stack/management/persistence/)

RDB — это компактное представление данных Redis в виде снимка (snapshot) на момент времени в одном файле; идеально для бэкапов и быстрого рестарта, но «вы должны быть готовы потерять последние минуты данных», если Redis аварийно остановится. AOF логирует каждую операцию записи; при использовании AOF Redis «гораздо более надёжен: можно выбрать разные политики fsync — без fsync, fsync раз в секунду, fsync на каждый запрос». Документация рекомендует использовать оба механизма; при рестарте, если включены оба, для восстановления используется AOF как наиболее полный.

#### Ответ от Claude

| Аспект | RDB (snapshot) | AOF (append-only file) |
|---|---|---|
| Что сохраняет | Бинарный снимок всего набора данных на момент времени | Лог всех команд-записей |
| Когда срабатывает | По расписанию (`save 900 1` и т.п.) или вручную (`BGSAVE`) | На каждую запись, с буферизацией и политикой fsync |
| Риск потери данных | Минуты (всё между снимками) | Зависит от fsync: `always` — почти 0, `everysec` — до ~1 сек, `no` — на усмотрение ОС |
| Размер файла | Компактный | Больше RDB для тех же данных |
| Скорость рестарта | Быстрее на больших данных | Медленнее (проигрывание лога), но AOF-rewrite сжимает лог |
| Влияние на производительность | `fork()` дочернего процесса может вызвать паузу на больших данных | Постоянная запись в лог |

**Ключевые тезисы:**
- **RDB** хорош для бэкапов и disaster recovery (один компактный файл легко скопировать), но допускает потерю последних минут.
- **AOF** надёжнее (меньше окно потери), append-only лог устойчив к повреждению, можно делать `BGREWRITEAOF` для компактизации.
- **Рекомендация документации** — использовать **оба** для надёжности, сопоставимой с традиционной СУБД. При рестарте с обоими включёнными приоритет у AOF как более полного.
- В Redis 7+ AOF по умолчанию хранится в виде нескольких файлов (multi-part AOF: базовый RDB-снимок + инкрементальные логи).

### Вопрос 6. Что произойдёт, когда Redis достигнет лимита памяти? Какие есть политики вытеснения (eviction)?

**Категория:** Redis · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Redis Docs — Key eviction](https://redis.io/docs/latest/develop/reference/eviction/)

Директива `maxmemory` задаёт верхний предел памяти. Когда лимит достигнут, Redis вытесняет ключи согласно выбранной `maxmemory-policy`. «Как правило: используйте `allkeys-lru`, когда ожидаете, что часть элементов будет запрашиваться значительно чаще остальных. Это очень распространённый случай по принципу Парето, поэтому `allkeys-lru` — хороший выбор по умолчанию, если нет причин предпочесть другой».

#### Ответ от Claude

При достижении `maxmemory` поведение определяется `maxmemory-policy`:

**Политики «noeviction»:**
- `noeviction` — новые записи отклоняются с ошибкой, чтение/удаление работают. (Значение по умолчанию в стандартной поставке — `noeviction`.)

**Вытеснение среди ключей с TTL (`volatile-*`):**
- `volatile-lru` — наименее недавно использованные (LRU) среди ключей с TTL.
- `volatile-lfu` — наименее часто используемые (LFU) среди ключей с TTL.
- `volatile-random` — случайные среди ключей с TTL.
- `volatile-ttl` — с наименьшим оставшимся TTL.

**Вытеснение среди всех ключей (`allkeys-*`):**
- `allkeys-lru` — LRU по всем ключам (рекомендуемый дефолт для чистого кеша).
- `allkeys-lfu` — LFU по всем ключам (хорош, когда важна частота, а не свежесть).
- `allkeys-random` — случайные по всем ключам.

**Нюансы для собеседования:**
- LRU и LFU в Redis — **приближённые**: алгоритм работает не по полному списку, а по случайной выборке (размер выборки — `maxmemory-samples`), чтобы не тратить память на точный учёт.
- LFU (появился в Redis 4.0) лучше LRU, когда есть ключи, к которым давно обращались, но часто — LRU их бы вытеснил, LFU сохранит.
- Если у ключей нет TTL, а политика `volatile-*`, то вытеснять нечего — Redis ведёт себя как `noeviction` (ошибки на запись).

### Вопрос 7. Как работают транзакции в Redis (MULTI/EXEC/WATCH)? Есть ли откат (rollback)?

**Категория:** Redis · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Redis Docs — Transactions](https://redis.io/docs/latest/develop/using-commands/transactions/)

«Все команды в транзакции сериализуются и выполняются последовательно. Запрос, посланный другим клиентом, никогда не будет обслужен в середине выполнения транзакции Redis. Это гарантирует, что команды выполняются как единая изолированная операция». При этом: «Redis не поддерживает откат (rollback) транзакций, поскольку поддержка откатов оказала бы значительное влияние на простоту и производительность Redis». `WATCH` обеспечивает поведение check-and-set (CAS): если хотя бы один отслеживаемый ключ изменён до `EXEC`, вся транзакция отменяется и `EXEC` возвращает Null.

#### Ответ от Claude

**Команды:** `MULTI` (начать), `EXEC` (выполнить очередь), `DISCARD` (отменить), `WATCH` (оптимистическая блокировка).

**Как это работает:**
1. После `MULTI` команды не выполняются, а **ставятся в очередь** (клиент получает `QUEUED`).
2. `EXEC` выполняет всю очередь **атомарно и изолированно** — никакой другой клиент не вклинится между командами (это естественно следует из однопоточности, [вопрос 3](#вопрос-3-почему-redis-такой-быстрый-и-как-он-обрабатывает-команды-однопоточная-модель)).

**Откат (rollback) — главное отличие от SQL-СУБД:** его **нет**. Если одна из команд в `EXEC` упадёт во время выполнения (например, `INCR` на строке), остальные команды всё равно выполнятся, а изменения не откатываются. Документация объясняет: ошибки внутри транзакции — обычно следствие ошибок программиста (которые должны выявляться на этапе разработки), а отказ от rollback сохраняет простоту и скорость.

Различают два вида ошибок:
- **Ошибка на этапе постановки в очередь** (например, синтаксис/неизвестная команда) — с Redis 2.6.5 вся транзакция отклоняется и не выполняется.
- **Ошибка во время `EXEC`** (неверный тип для команды) — эта команда падает, но транзакция продолжается.

**`WATCH` — оптимистическая блокировка (CAS):** отслеживаем ключи перед `MULTI`; если кто-то изменит отслеживаемый ключ до `EXEC`, транзакция отменяется и `EXEC` вернёт `nil` — клиент должен повторить операцию. Используется для безопасного «прочитать → изменить → записать».

### Вопрос 8. Как работает механизм Pub/Sub в Redis и чем он отличается от Streams?

**Категория:** Redis · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Redis Docs — Pub/Sub](https://redis.io/docs/latest/develop/pubsub/)

Redis Pub/Sub работает по принципу «fire-and-forget»: если клиент-подписчик отключается и переподключается позже, все события, доставленные за время отключения, теряются. Сообщения не сохраняются — если сообщение опубликовано, а подписчиков нет, оно «испаряется» и не может быть восстановлено. Семантика доставки — «не более одного раза» (at-most-once). Для сценариев, требующих сохранения и гарантированной доставки, Redis рекомендует использовать Streams.

#### Ответ от Claude

**Pub/Sub** — модель «издатель → канал → подписчики»:
- `SUBSCRIBE channel` / `PSUBSCRIBE pattern.*` — подписка (в т.ч. по шаблону).
- `PUBLISH channel message` — публикация.

**Ключевая характеристика — fire-and-forget, доставка at-most-once:**
- Сообщения **не хранятся**. Доставляются только тем, кто подписан **в момент** публикации.
- Если подписчиков нет — сообщение теряется. Если подписчик отключился — пропущенные сообщения он не получит.
- Нет подтверждений, нет истории, нет повторного воспроизведения.

**Pub/Sub vs Streams:**

| | Pub/Sub | Streams |
|---|---|---|
| Хранение | Нет (в памяти на момент доставки) | Да (append-only лог) |
| Гарантия доставки | At-most-once | At-least-once (с ACK) |
| Пропущенные сообщения | Теряются | Можно прочитать позже по ID |
| Consumer groups | Нет | Да (масштабирование обработки) |
| Применение | Realtime-уведомления, чаты, где потеря допустима | Надёжные очереди, event sourcing |

**Вывод:** Pub/Sub — для лёгкого realtime-вещания, где потеря сообщений не критична. Когда нужна надёжность и повтор — Streams (или внешний брокер вроде Kafka/RabbitMQ, см. [область «Брокеры сообщений»](../theory/01-knowledge-areas/04-distributed-and-ops/21-messaging.md)).

### Вопрос 9. Что такое пайплайнинг (pipelining) в Redis?

**Категория:** Redis · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Redis Docs — Pipelining](https://redis.io/docs/latest/develop/using-commands/pipelining/)

Пайплайнинг — это техника повышения производительности за счёт отправки нескольких команд сразу, не дожидаясь ответа на каждую отдельную команду. «Сервер по модели запрос/ответ можно реализовать так, чтобы он обрабатывал новые запросы, даже если клиент ещё не прочитал старые ответы. Так можно отправить серверу несколько команд, вообще не ожидая ответов, а затем прочитать все ответы за один приём». Это устраняет затраты на RTT (round-trip time) для каждой команды; в примере документации 10 000 команд выполняются примерно в 5 раз быстрее.

#### Ответ от Claude

**Проблема:** обычный протокол Redis — «запрос → ответ». На каждую команду клиент платит **RTT** (время сетевого round-trip). При тысячах команд сеть, а не Redis, становится узким местом.

**Решение — пайплайнинг:** клиент отправляет много команд подряд, **не дожидаясь** ответа на каждую, а затем читает все ответы пачкой. RTT платится один раз на всю пачку, а не на каждую команду. Дополнительно сокращается число системных вызовов: много команд читаются одним `read()`, ответы пишутся одним `write()`.

**Важные уточнения:**
- Пайплайнинг **не атомарен** — это просто пакетная отправка, между командами пачки могут вклиниться команды других клиентов. Для атомарности нужны транзакции (`MULTI`/`EXEC`) или Lua-скрипт.
- Пайплайнинг ≠ транзакция: первое про *сетевую эффективность*, второе про *атомарность/изоляцию*.
- Не стоит слать миллионы команд одной пачкой — растёт память на буферы ответов; разумно бить на батчи (например, по 10 000).

### Вопрос 10. Какие существуют стратегии кеширования с Redis и какие у них проблемы?

**Категория:** Redis · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Redis — Cache-Aside Pattern with Redis](https://redis.io/learn/howtos/solutions/microservices/caching)

«Redis — это in-memory хранилище, наиболее известное благодаря кешированию. Redis позволяет снизить нагрузку на основную базу данных, ускоряя при этом чтение из неё». Наиболее распространённый паттерн — cache-aside (lazy loading): приложение проверяет кеш; при попадании (cache hit) данные возвращаются прямо из Redis, при промахе (cache miss) приложение читает из основной БД, сохраняет результат в Redis с TTL и возвращает. Также применяются write-through (запись одновременно в кеш и БД) и write-behind (отложенная запись в БД).

#### Ответ от Claude

**Основные паттерны:**

| Паттерн | Как работает | Плюсы / минусы |
|---|---|---|
| **Cache-aside (lazy loading)** | Приложение само читает из кеша; при промахе — из БД и кладёт в кеш | Самый частый; кеш хранит только запрошенное; минус — первый запрос медленный, риск устаревания |
| **Read-through** | Кеш сам подгружает из БД при промахе (через провайдер) | Прозрачно для приложения; нужен слой кеша, умеющий это |
| **Write-through** | Запись идёт в кеш и БД синхронно | Кеш всегда свежий; запись медленнее |
| **Write-behind (write-back)** | Запись в кеш, в БД — асинхронно пачкой | Быстрая запись; риск потери данных при сбое до сброса |

**Сквозные проблемы кеширования:**
- **Инвалидация** — «одна из двух сложнейших задач в CS». Нужно вовремя удалять/обновлять устаревшие данные (TTL, явная инвалидация при записи).
- **Консистентность** кеша и БД при параллельных записях.
- **Три классические аварии** — penetration, breakdown, avalanche (подробно в [вопросе 15](#вопрос-15-что-такое-проникновение-пробой-и-лавина-кеша-cache-penetration-breakdown-avalanche-и-как-с-ними-бороться)).

**Redis vs Memcached** (частый смежный вопрос): Memcached — простой кеш «строка → строка», многопоточный, без персистентности и репликации. Redis — богатые типы данных, персистентность, репликация, кластер, Pub/Sub. Для простого кеша подойдут оба; когда нужны структуры/надёжность/HA — Redis.

### Вопрос 11. Как устроена репликация master-replica в Redis?

**Категория:** Redis · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Redis Docs — Replication](https://redis.io/docs/latest/operate/oss_and_stack/management/replication/)

Redis использует модель «лидер — последователь» (master-replica). «По умолчанию Redis использует асинхронную репликацию, которая, обладая низкой задержкой и высокой производительностью, является естественным режимом для подавляющего большинства сценариев». Реплики с версии 2.6 по умолчанию работают в режиме только для чтения (`replica-read-only`). У одного мастера может быть несколько реплик; репликацию используют как для масштабирования чтения (медленные операции `O(N)` можно выносить на реплики), так и для отказоустойчивости.

#### Ответ от Claude

**Модель:** один **master** принимает записи, одна или несколько **replica** держат его копию.

**Ключевые свойства:**
- **Асинхронность по умолчанию.** Мастер не ждёт подтверждения от реплик перед ответом клиенту. Это быстро, но создаёт **окно потери данных**: если мастер упадёт до репликации последних записей — они потеряются. Команда `WAIT` позволяет дождаться подтверждения от N реплик (псевдосинхронность), но это не полноценный синхронный консенсус.
- **Реплики read-only** по умолчанию — защищает от рассинхронизации.
- **Назначение:** (1) масштабирование чтения — клиенты читают с реплик; (2) избыточность/HA — при отказе мастера реплику можно повысить (см. Sentinel, [вопрос 12](#вопрос-12-чем-различаются-redis-sentinel-и-redis-cluster)).

**Механика синхронизации:**
- При подключении реплика делает **полную синхронизацию**: мастер делает `BGSAVE` (RDB-снимок) и шлёт его реплике, попутно буферизуя новые команды.
- Далее идёт **потоковая репликация** команд.
- При кратком разрыве возможна **частичная ресинхронизация** (PSYNC) — мастер досылает только пропущенное из буфера репликации (replication backlog), без полного снимка.

**Важно:** асинхронная репликация — причина, по которой нельзя строить надёжные распределённые блокировки на схеме «один мастер + failover» (см. [вопрос 14](#вопрос-14-как-реализовать-распределённую-блокировку-на-redis-что-такое-redlock-и-в-чём-его-критика)).

### Вопрос 12. Чем различаются Redis Sentinel и Redis Cluster?

**Категория:** Redis · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Redis Docs — High availability with Redis Sentinel](https://redis.io/docs/latest/operate/oss_and_stack/management/sentinel/)

«Redis Sentinel обеспечивает высокую доступность Redis, когда не используется Redis Cluster». Возможности Sentinel: **мониторинг** (постоянно проверяет, что мастер и реплики работают), **уведомления** (сообщает администратору/программам о проблемах), **автоматический failover** (если мастер не работает, реплика повышается до мастера, остальные реплики перенастраиваются, а клиенты узнают новый адрес) и **провайдер конфигурации** (клиенты спрашивают у Sentinel адрес текущего мастера). Кворум — число Sentinel, которые должны согласиться, что мастер недоступен, чтобы запустить failover.

#### Ответ от Claude

Это **разные решения для разных задач**:

| | Redis Sentinel | Redis Cluster |
|---|---|---|
| Задача | Высокая доступность (HA) при **единственном** наборе данных | Горизонтальное **масштабирование** + HA |
| Шардирование данных | Нет — все данные на одном мастере | Да — данные разбиты по 16384 хеш-слотам ([вопрос 13](#вопрос-13-как-redis-cluster-распределяет-данные-по-узлам-хеш-слоты)) |
| Состав | Отдельные процессы Sentinel мониторят master/replica | Сами узлы общаются по gossip-протоколу |
| Failover | Sentinel повышает реплику до мастера | Кластер сам повышает реплику отказавшего шарда |
| Когда выбирать | Данные помещаются на один сервер, нужна только отказоустойчивость | Данные/нагрузка не помещаются на один узел |

**Sentinel** — это система мониторинга и автоматического failover поверх обычной репликации (master + replicas). Важная деталь: **кворум** используется только для *обнаружения* отказа мастера; чтобы реально провести failover, нужен голос **большинства** процессов Sentinel (поэтому их разворачивают ≥3, в нечётном числе, на разных хостах).

**Cluster** — это шардированное хранилище: данные распределены по узлам, каждый шард обычно имеет свои реплики. Кластер сам обеспечивает и масштабирование, и отказоустойчивость, и поэтому Sentinel ему не нужен.

**Ограничение Cluster:** многоключевые операции работают, только если ключи в одном слоте (см. hash tags, [вопрос 13](#вопрос-13-как-redis-cluster-распределяет-данные-по-узлам-хеш-слоты)).

### Вопрос 13. Как Redis Cluster распределяет данные по узлам (хеш-слоты)?

**Категория:** Redis · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Redis Docs — Redis cluster specification](https://redis.io/docs/latest/operate/oss_and_stack/reference/cluster-spec/)

«Ключевое пространство кластера разбито на 16384 слота, что задаёт верхний предел размера кластера в 16384 master-узла (рекомендуемый максимум — порядка ~1000 узлов)». Базовый алгоритм отображения ключа в слот: `HASH_SLOT = CRC16(key) mod 16384`. Hash tags позволяют поместить несколько ключей в один слот: если ключ содержит шаблон `{...}`, хешируется только подстрока между первой `{` и первой `}`, поэтому `{user1000}.following` и `{user1000}.followers` попадают в один слот.

#### Ответ от Claude

**Хеш-слоты вместо консистентного хеширования.** Redis Cluster делит всё ключевое пространство на фиксированные **16384 слота**. Каждый ключ принадлежит слоту по формуле:

```
HASH_SLOT = CRC16(key) mod 16384
```

Слоты распределяются между master-узлами (например, 3 узла → примерно по 5461 слоту). Узел знает, какие слоты у кого, поэтому при обращении к «чужому» ключу возвращает редирект `MOVED`/`ASK`, и клиент идёт к нужному узлу (умные клиенты кешируют карту слотов).

**Зачем именно слоты:** при добавлении/удалении узла перемещаются **слоты целиком**, а не пересчитывается весь хеш — решардинг проходит инкрементально и без полной перетасовки ключей.

**Hash tags — для многоключевых операций.** В кластере команды с несколькими ключами (`MGET`, транзакции, Lua с несколькими ключами) требуют, чтобы все ключи были **в одном слоте**. Чтобы это гарантировать, в ключ добавляют тег в фигурных скобках — хешируется только он:

```
{user1000}.cart  и  {user1000}.profile  →  один слот
```

**Нюанс:** число слотов (16384) выбрано как компромисс — достаточно для тонкого распределения, но карта слотов компактна для обмена в gossip-сообщениях между узлами.

### Вопрос 14. Как реализовать распределённую блокировку на Redis? Что такое Redlock и в чём его критика?

**Категория:** Redis · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Redis Docs — Distributed Locks with Redis](https://redis.io/docs/latest/develop/clients/patterns/distributed-locks/)

Простая блокировка на одном инстансе: `SET resource_name my_random_value NX PX 30000` — ключ ставится, только если не существует (`NX`), со сроком 30 сек (`PX`). Случайное значение важно, «чтобы не удалить блокировку, созданную другим клиентом»; снятие делается Lua-скриптом, проверяющим значение. Для распределённого случая предлагается алгоритм **Redlock** на N независимых мастерах (обычно N=5): клиент пытается захватить блокировку на всех, и «тогда и только тогда, когда клиент смог получить блокировку на большинстве инстансов (минимум 3), и общее затраченное время меньше времени жизни блокировки, блокировка считается захваченной». Схема «один мастер + failover» небезопасна, так как «репликация Redis асинхронна».

#### Ответ от Claude

**Блокировка на одном инстансе:**
```
SET lock:resource <random_token> NX PX 30000   # захват
# освобождение — атомарно через Lua, чтобы не снять чужую блокировку:
if redis.call("get", KEYS[1]) == ARGV[1] then return redis.call("del", KEYS[1]) else return 0 end
```
- `NX` — захватит только если свободно; `PX` — авто-снятие по таймауту (защита от deadlock, если клиент умер).
- **Случайный токен обязателен:** снимать блокировку можно, только если токен совпал — иначе клиент, у которого блокировка уже истекла, снимет чужую.

**Почему недостаточно «один мастер + реплика с failover»:** репликация **асинхронна**. Если мастер выдал блокировку и упал до репликации, реплика повысится до мастера «без» этой блокировки — и второй клиент захватит её же. Нарушается взаимное исключение.

**Redlock** решает это без репликации: N независимых мастеров (обычно 5). Клиент пытается захватить блокировку на всех с одним токеном; блокировка считается взятой, если успел на **большинстве (≥ N/2+1)** за время меньше TTL. При неуспехе — снимает везде.

**Критика (Martin Kleppmann, [martin.kleppmann.com](https://martin.kleppmann.com/2016/02/08/how-to-do-distributed-locking.html)):** Redlock **небезопасен для задач, где блокировка нужна ради корректности**:
- **Нет fencing-токенов.** Redlock не выдаёт монотонно растущий номер, которым хранилище могло бы отбрасывать запоздавшие записи. Из-за GC-паузы или сетевой задержки клиент может продолжить запись уже после истечения своей блокировки → повреждение данных.
- **Зависимость от таймингов.** Безопасность опирается на предположения о синхронности часов, малых сетевых задержках и коротких паузах процессов — в реальной распределённой системе они не гарантированы.
- **Вывод Клеппманна:** «если блокировка нужна для корректности — не используйте Redlock; используйте полноценную систему консенсуса, например ZooKeeper». Redlock допустим как «эффективностная» блокировка (избежать лишней работы), где редкое нарушение взаимного исключения не критично.

**Расхождение источников помечаю явно:** документация Redis считает Redlock пригодным при соблюдении условий (включить AOF с `fsync`, учитывать дрейф часов); Клеппманн считает его непригодным для корректностно-критичных сценариев. Это известная открытая дискуссия — на собеседовании ценится умение её изложить, а не «правильная сторона».

### Вопрос 15. Что такое проникновение, пробой и лавина кеша (cache penetration, breakdown, avalanche) и как с ними бороться?

**Категория:** Redis · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Charlie Feng — Redis Cache Problems: Penetration, Breakdown and Avalanche](https://shayne007.github.io/2025/06/10/Redis-Cache-Problems-Penetration-Breakdown-and-Avalanche/)

- **Проникновение (penetration):** запросы к данным, которых нет ни в кеше, ни в БД, поэтому кеш никогда не заполняется и каждый запрос бьёт по БД. Решение — кешировать «пустой» результат с коротким TTL или поставить перед кешем фильтр Блума, отсекающий заведомо несуществующие ключи.
- **Пробой (breakdown):** один «горячий» ключ истекает, и тысячи параллельных запросов разом штурмуют БД. Решение — мьютекс (`SETNX`), чтобы только один поток перестраивал кеш, пока остальные ждут; либо «логически вечные» горячие ключи с фоновым обновлением.
- **Лавина (avalanche):** множество ключей истекает одновременно (или падает сам Redis), и БД захлёбывается. Решение — джиттер TTL (базовый TTL + случайное смещение), многоуровневый кеш, circuit breaker/rate limiter, HA-развёртывание Redis.

#### Ответ от Claude

Три разные аварийные ситуации кеша и меры против них:

**1. Cache penetration (проникновение) — запросы к несуществующим данным.**
Злоумышленник/баг запрашивает ключи, которых нет ни в кеше, ни в БД. Кеш всегда промахивается, нагрузка идёт прямо в БД.
- **Кешировать «отсутствие»** (null-значение) с коротким TTL.
- **Фильтр Блума** перед кешем — отсекает заведомо несуществующие ключи до обращения к Redis.

**2. Cache breakdown (пробой) — истечение одного горячего ключа.**
Очень популярный ключ истекает, и множество параллельных запросов одновременно идут перестраивать его из БД (cache stampede на одном ключе).
- **Мьютекс/распределённая блокировка** (`SET ... NX`): только один поток перестраивает значение, остальные ждут или отдают старое.
- **Логически «вечные» ключи**: горячие данные без TTL, обновляются фоном.

**3. Cache avalanche (лавина) — массовое истечение или падение Redis.**
Много ключей истекает в один момент (или падает сам Redis) → шквал запросов в БД.
- **Джиттер TTL**: к базовому сроку добавляют случайное смещение, чтобы ключи не истекали синхронно.
- **Многоуровневый кеш** (локальный in-process + Redis).
- **Защита БД**: circuit breaker, rate limiting, очереди.
- **HA для Redis** (Sentinel/Cluster), чтобы падение одного узла не обрушивало весь кеш.

**Связь понятий:** breakdown — частный случай stampede на одном ключе; avalanche — масштабный сбой на множестве ключей или всего кеша; penetration — про данные, которых вообще нет. На собеседовании важно не перепутать их и назвать конкретные меры (Bloom filter, mutex, TTL jitter).
