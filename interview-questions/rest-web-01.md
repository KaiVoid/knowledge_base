### Вопрос 1. Какие HTTP-методы являются идемпотентными, а какие — нет? Приведите примеры.

**Категория:** REST/Web · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [What are idempotent and/or safe methods? — The RESTful Cookbook](https://restcookbook.com/HTTP%20Methods/idempotency/)

Безопасные методы (safe methods) — это HTTP-методы, которые не изменяют ресурсы. Использование GET или HEAD никогда не должно менять представление ресурса. Безопасные методы можно кешировать и предзагружать без последствий.

Идемпотентный метод даёт одинаковый результат вне зависимости от того, сколько раз он вызывается. Аналогия из кода: присвоение `a = 4` идемпотентно (всегда равно 4), а `a++` — нет (результат меняется при каждом вызове).

Таблица методов:

| Метод   | Идемпотентный | Безопасный |
|---------|:-------------:|:----------:|
| GET     | Да            | Да         |
| HEAD    | Да            | Да         |
| OPTIONS | Да            | Да         |
| PUT     | Да            | Нет        |
| DELETE  | Да            | Нет        |
| POST    | Нет           | Нет        |
| PATCH   | Нет           | Нет        |

Безопасные методы никогда не изменяют ресурсы, идемпотентные — дают одинаковый результат при повторении. GET одновременно безопасен и идемпотентен, POST — ни то ни другое.

#### Ответ от Claude

**Определение (RFC 9110, секция 9.2.2).** Метод HTTP считается идемпотентным, если намеренный эффект на сервере от нескольких одинаковых запросов таким методом совпадает с эффектом от одного такого запроса. Идемпотентность касается **состояния сервера**, а не кода ответа — ответы на повторные запросы могут отличаться.

**Идемпотентные методы:** GET, HEAD, OPTIONS, TRACE, PUT, DELETE.

**Неидемпотентные методы:** POST, PATCH.

**Примеры:**

- **GET /orders/42** — сколько бы раз ни запросить заказ, данные на сервере не изменятся.
- **PUT /users/1** `{"name":"Ivan","age":30}` — повторная отправка того же тела всегда оставляет ресурс в одном и том же состоянии.
- **DELETE /subscriptions/100** — после первого вызова подписка удалена; второй вызов вернёт 404, но состояние сервера уже не изменится — подписки по-прежнему нет. Это подтверждает: идемпотентность == одинаковый **эффект**, а не одинаковый **ответ**.
- **POST /orders** — каждый вызов создаёт новый заказ, т.е. состояние сервера меняется при каждом запросе.
- **PATCH /users/1** `{"counter": {"$inc": 1}}` — инкремент при каждом вызове даёт разный результат, PATCH не идемпотентен (хотя реализацию PATCH можно сделать идемпотентной — это зависит от семантики операции).

**Практическое значение идемпотентности:** при сетевых сбоях клиент может безопасно повторять идемпотентные запросы без риска нежелательных побочных эффектов. Для неидемпотентных методов (POST, PATCH) при повторах необходима дополнительная защита, например ключи идемпотентности (Idempotency-Key).

**Важное разграничение:** все безопасные методы (GET, HEAD, OPTIONS, TRACE) являются подмножеством идемпотентных — они ни изменяют ресурс, ни имеют побочных эффектов. PUT и DELETE идемпотентны, но не безопасны, поскольку изменяют состояние сервера.

### Вопрос 2. В чём разница между HTTP-методами PUT и PATCH? Когда использовать каждый из них?

**Категория:** REST/Web · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [HTTP PATCH vs PUT: Key Differences for REST API Updates — Zuplo](https://zuplo.com/learning-center/http-patch-vs-put-whats-the-difference)

**PUT** заменяет ресурс целиком, **PATCH** обновляет только указанные поля. Это фундаментальное различие влияет на использование полосы пропускания, идемпотентность и сложность реализации.

**Идемпотентность.** PUT изначально идемпотентен — отправка одного и того же запроса несколько раз даёт одинаковый результат. MDN Web Docs отмечает: «PUT идемпотентен: вызов один раз ничем не отличается от нескольких последовательных вызовов». PATCH, напротив, не гарантирует идемпотентности.

**Когда использовать PUT:**
- Полная замена ресурса.
- Обновление конфигурации или массовая замена.
- Приоритет простоты над экономией трафика.

**Когда использовать PATCH:**
- Точечные, частичные обновления.
- Ограниченная полоса пропускания (мобильные приложения, сети с высокой задержкой).
- Обновление отдельных полей большого ресурса.

**Практический пример.** Изменение email в большом профиле пользователя: PATCH отправляет только `{"email": "new@example.com"}`, тогда как PUT передаёт весь объект пользователя. PATCH значительно экономит трафик.

**Соображения по реализации.** PUT требует, чтобы клиент управлял полным представлением ресурса. PATCH требует надёжной серверной валидации, разрешения конфликтов и авторизации на уровне полей. Для реализации PATCH рекомендуется использовать стандарты JSON Merge Patch (RFC 7396) или JSON Patch (RFC 6902).

#### Ответ от Claude

**Суть различия.**

| Характеристика      | PUT                                        | PATCH                                       |
|---------------------|--------------------------------------------|---------------------------------------------|
| Объём обновления    | Полная замена ресурса                      | Частичное обновление (только указанные поля)|
| Идемпотентность     | Да (RFC 9110)                              | Нет по умолчанию (зависит от реализации)    |
| Тело запроса        | Полное представление ресурса              | Только изменяемые поля / инструкции патча   |
| Пропущенные поля    | Обнуляются / удаляются                    | Остаются без изменений                      |
| Создание ресурса    | Возможно (PUT на несуществующий URI)       | Как правило, нет                            |

**PUT — полная замена.**

```
PUT /users/1
{
  "name": "Ivan",
  "email": "ivan@example.com",
  "age": 30
}
```
Если в теле опустить поле `age`, оно будет сброшено или удалено. Сервер обязан заменить ресурс предоставленным представлением целиком.

**PATCH — частичное обновление.**

```
PATCH /users/1
{
  "email": "new-ivan@example.com"
}
```
Только `email` будет изменён, `name` и `age` останутся прежними.

**Форматы PATCH.** Стандарт не предписывает формат тела PATCH-запроса. На практике используются:
- **JSON Merge Patch (RFC 7396)** — описывает изменения в виде фрагмента JSON (поле `null` означает удаление).
- **JSON Patch (RFC 6902)** — массив операций (`add`, `remove`, `replace`, `move`, `copy`, `test`), Content-Type: `application/json-patch+json`.

**Идемпотентность PATCH.** PATCH идемпотентен, только если операция не зависит от текущего состояния ресурса. Установка `{"status": "active"}` идемпотентна; инкремент `{"views": {"$inc": 1}}` — нет. По умолчанию клиент не может считать PATCH идемпотентным.

**Практические рекомендации:**
- Используйте **PUT**, когда у клиента есть полное и актуальное представление ресурса (например, форма редактирования, загрузившая все поля).
- Используйте **PATCH**, когда нужно обновить одно-два поля из большого ресурса — экономит трафик и снижает вероятность затирания чужих изменений при параллельных операциях.
- Если API публичный, поддерживайте оба метода, явно документируя семантику каждого.

### Вопрос 3. Что означают группы кодов состояния HTTP: 2xx, 3xx, 4xx, 5xx? Назовите наиболее распространённые коды из каждой группы.

**Категория:** REST/Web · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [What are HTTP status codes? Complete Guide for API Developers — Postman Blog](https://blog.postman.com/what-are-http-status-codes/)

**2xx — Успех.** Запрос был получен, понят и обработан успешно.
- **200 OK** — «запрос выполнен успешно, сервер вернул запрошенные данные».
- **201 Created** — ресурс успешно создан; заголовок `Location` указывает на новый ресурс.
- **204 No Content** — операция завершена успешно, тело ответа отсутствует (типично для DELETE).

**3xx — Перенаправление.** Клиенту необходимо выполнить дополнительное действие, как правило — перейти по другому URL.
- **301 Moved Permanently** — «запрошенный ресурс навсегда перемещён на новый URL».
- **302 Found** — временное перенаправление.
- **304 Not Modified** — кешированная версия актуальна, повторная загрузка не нужна.

**4xx — Ошибка клиента.** Проблема на стороне запроса.
- **400 Bad Request** — сервер не смог обработать запрос из-за некорректного синтаксиса или плохо сформированных данных.
- **401 Unauthorized** — «требуется аутентификация: она не была предоставлена или является недействительной».
- **403 Forbidden** — пользователь аутентифицирован, но не имеет прав на доступ к ресурсу.
- **404 Not Found** — «запрошенный ресурс не существует по указанному URL».
- **429 Too Many Requests** — клиент превысил лимит запросов.

**5xx — Ошибка сервера.** Сервер не смог выполнить корректный запрос.
- **500 Internal Server Error** — общая ошибка сервера, неожиданный сбой.
- **503 Service Unavailable** — сервер временно недоступен (обслуживание или перегрузка).
- **504 Gateway Timeout** — вышестоящий сервер не ответил в установленное время.

#### Ответ от Claude

RFC 9110 делит коды HTTP-ответов на пять классов по первой цифре. Ниже — подробный разбор четырёх основных, используемых в REST API.

**2xx — Успешные ответы**

Запрос принят и обработан корректно.

| Код | Название         | Когда используется                                                              |
|-----|------------------|---------------------------------------------------------------------------------|
| 200 | OK               | Стандартный успех: GET-ответ с телом, успешный PUT/PATCH с возвратом ресурса    |
| 201 | Created          | Ресурс создан (POST/PUT); заголовок `Location` должен указывать на новый ресурс |
| 202 | Accepted         | Запрос принят к асинхронной обработке, результат будет позднее                  |
| 204 | No Content       | Успех без тела ответа: DELETE, PUT/PATCH без возврата ресурса                   |
| 206 | Partial Content  | Ответ на Range-запрос (частичная загрузка файла)                                |

**3xx — Перенаправления**

Клиенту нужно сделать дополнительный шаг для получения ресурса.

| Код | Название           | Когда используется                                                    |
|-----|--------------------|-----------------------------------------------------------------------|
| 301 | Moved Permanently  | Ресурс навсегда переехал; браузеры и поисковики обновляют ссылки      |
| 302 | Found              | Временное перенаправление; исходный URL сохраняет актуальность        |
| 303 | See Other          | После POST перенаправить на результат (паттерн PRG — Post/Redirect/Get)|
| 304 | Not Modified       | Кеш актуален; тело ответа не передаётся                               |
| 307 | Temporary Redirect | Временный редирект с сохранением метода (в отличие от 302)            |
| 308 | Permanent Redirect | Постоянный редирект с сохранением метода (в отличие от 301)           |

**4xx — Ошибки клиента**

Проблема в запросе клиента; повтор без изменений бессмысленен.

| Код | Название              | Когда используется                                                       |
|-----|-----------------------|--------------------------------------------------------------------------|
| 400 | Bad Request           | Невалидный JSON, нарушение бизнес-правил, отсутствие обязательных полей  |
| 401 | Unauthorized          | Аутентификация не прошла или не предоставлена (несмотря на название — это про аутентификацию, а не авторизацию) |
| 403 | Forbidden             | Аутентификация прошла, но прав недостаточно (авторизация)                |
| 404 | Not Found             | Ресурс не существует; также используют намеренно вместо 403 для скрытия информации |
| 405 | Method Not Allowed    | Метод не поддерживается для данного URI                                  |
| 409 | Conflict              | Конфликт состояний: дублирующаяся запись, версионный конфликт            |
| 410 | Gone                  | Ресурс был, но удалён навсегда (постоянный вариант 404)                  |
| 422 | Unprocessable Entity  | Запрос синтаксически корректен, но не прошёл валидацию (используется в Spring, Rails) |
| 429 | Too Many Requests     | Превышен rate limit; заголовок `Retry-After` указывает, когда повторить  |

**5xx — Ошибки сервера**

Запрос корректен, но сервер не смог его обработать.

| Код | Название               | Когда используется                                                        |
|-----|------------------------|---------------------------------------------------------------------------|
| 500 | Internal Server Error  | Необработанное исключение, баг, краш — общий catch-all                    |
| 501 | Not Implemented        | Сервер не поддерживает функциональность, необходимую для выполнения запроса|
| 502 | Bad Gateway            | Прокси/балансировщик получил некорректный ответ от upstream-сервера       |
| 503 | Service Unavailable    | Сервер перегружен или на обслуживании; рекомендуется заголовок `Retry-After` |
| 504 | Gateway Timeout        | Upstream-сервер не ответил вовремя                                        |

**Частые путаницы:**
- **401 vs 403:** 401 — «кто ты?» (нужна аутентификация), 403 — «я знаю, кто ты, но нет» (недостаточно прав).
- **302 vs 307:** оба временные, но 302 позволяет браузеру изменить метод (POST→GET), а 307 обязывает сохранить оригинальный метод.
- **404 vs 410:** 410 явно говорит, что ресурс существовал и удалён навсегда; 404 — неопределённость.

### Вопрос 4. В чём разница между кодами 401 Unauthorized и 403 Forbidden?

**Категория:** REST/Web · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [HTTP 401 Error vs HTTP 403 Error – Status Code Responses Explained (freeCodeCamp)](https://www.freecodecamp.org/news/http-401-error-vs-http-403-error-status-code-responses-explained/)

Ключевое различие: код 401 означает «кто вы?», а код 403 — «вам сюда нельзя».

Согласно RFC 7235, код 401 указывает, что «запрос не был выполнен, поскольку ему не хватает действительных учётных данных для аутентификации целевого ресурса». RFC 7231, в свою очередь, определяет 403 как ситуацию, когда «сервер понял запрос, но отказывается его авторизовать».

Примеры использования:
- **401**: пользователь не вошёл в систему, указал неверный пароль или передал некорректный заголовок авторизации (например, отсутствует префикс «Bearer» перед JWT-токеном).
- **403**: вошедший пользователь пытается открыть страницу только для администраторов, ресурс закрыт по IP-адресу или по географическому признаку.

#### Ответ от Claude

**401 Unauthorized** — ошибка аутентификации. Сервер не знает, кто делает запрос: учётные данные отсутствуют, некорректны или просрочены. RFC 7235 (раздел 3.1) обязывает сервер вернуть заголовок `WWW-Authenticate`, указывающий схему аутентификации (Basic, Bearer и т.д.), чтобы клиент знал, как повторить запрос правильно. Повторный запрос с корректными данными может завершиться успехом.

**403 Forbidden** — ошибка авторизации. Сервер знает, кто делает запрос (аутентификация прошла успешно или идентификация не нужна), но отказывается выполнять запрос, потому что у пользователя нет нужных прав. Согласно RFC 7235: «Сервер, получивший действительные учётные данные, недостаточные для получения доступа, должен ответить кодом 403». Повторная аутентификация ничего не изменит — нужно другое разрешение.

| | 401 Unauthorized | 403 Forbidden |
|---|---|---|
| Суть проблемы | Аутентификация | Авторизация |
| Личность пользователя | Неизвестна | Известна |
| Заголовок `WWW-Authenticate` | Обязателен (RFC 7235) | Не требуется |
| Поможет ли повторная аутентификация? | Да | Нет |
| Спецификация | RFC 7235, §3.1 | RFC 7231, §6.5.3 |

**Практические примеры:**
- Обращение к API без заголовка `Authorization` → **401**.
- Обращение к API с просроченным JWT-токеном → **401**.
- Попытка обычного пользователя удалить чужой ресурс → **403**.
- Попытка получить доступ к `/admin` без роли ADMIN → **403**.

**Важная деталь:** несмотря на слово «Unauthorized» в названии, код 401 семантически означает «не аутентифицирован» (unauthenticated), а не «не авторизован». Это историческая неточность в спецификации HTTP.

### Вопрос 5. Какие основные принципы REST (ограничения)? Перечислите хотя бы четыре.

**Категория:** REST/Web · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [The Six Constraints — REST API Tutorial](https://www.restapitutorial.com/introduction/restconstraints)

Согласно диссертации Роя Филдинга, REST определяется шестью ограничениями (constraints), одно из которых является необязательным:

1. **Единообразие интерфейса (Uniform Interface)** — центральное ограничение REST. Включает четыре подпринципа: идентификация ресурсов через URI; манипуляция ресурсами через представления; самоописывающие сообщения; HATEOAS (гипермедиа как движок состояния приложения).
2. **Отсутствие состояния (Stateless)** — весь контекст, необходимый для обработки запроса, должен содержаться в самом запросе. Сервер не хранит сессионное состояние между запросами.
3. **Кешируемость (Cacheable)** — ответы должны явно помечать себя как кешируемые или некешируемые.
4. **Клиент-сервер (Client-Server)** — разделение ответственности позволяет клиенту и серверу развиваться независимо.
5. **Многоуровневая система (Layered System)** — клиент не может определить, взаимодействует ли он напрямую с сервером или через промежуточный узел (балансировщик, кеш, шлюз).
6. **Код по требованию (Code on Demand)** — необязательное ограничение: сервер может передавать исполняемый код клиенту (JavaScript, апплеты).

#### Ответ от Claude

Архитектурный стиль REST описан Роем Филдингом в его докторской диссертации 2000 года «Architectural Styles and the Design of Network-based Software Architectures» (UC Irvine). REST задаётся шестью ограничениями:

**1. Единообразие интерфейса (Uniform Interface)**
Центральный принцип, отличающий REST от других сетевых архитектур. Включает четыре подпринципа:
- Идентификация ресурсов через URI (ресурс и его представление — разные вещи).
- Манипуляция ресурсами через представления: клиент, получивший представление ресурса, имеет достаточно информации для его изменения или удаления.
- Самоописывающие сообщения: каждое сообщение несёт всю информацию, необходимую для его обработки (заголовок `Content-Type`, инструкции по кешированию и т.д.).
- HATEOAS (Hypermedia As The Engine Of Application State): клиент переходит по состояниям приложения, следуя ссылкам из ответов сервера, а не по заранее известным URL.

**2. Отсутствие состояния (Stateless)**
Каждый запрос от клиента к серверу должен содержать всю информацию, необходимую для его обработки. Сервер не хранит клиентский контекст между запросами. Это упрощает масштабирование (любой узел кластера может обработать любой запрос) и повышает надёжность.

**3. Кешируемость (Cacheable)**
Ответы должны явно или неявно помечать себя как кешируемые или некешируемые (заголовки `Cache-Control`, `ETag`, `Expires`). Правильно реализованное кеширование снижает нагрузку на сервер и уменьшает задержки.

**4. Клиент-сервер (Client-Server)**
Строгое разделение интерфейса пользователя (клиент) и хранения/обработки данных (сервер). Клиент не занимается хранением данных, сервер — отображением. Это позволяет независимо развивать клиентские и серверные компоненты.

**5. Многоуровневая система (Layered System)**
Клиент не может знать, взаимодействует ли он напрямую с конечным сервером или через промежуточные слои (балансировщики нагрузки, API-шлюзы, кеши, брандмауэры). Это позволяет добавлять новые слои инфраструктуры без изменения клиентского кода.

**6. Код по требованию (Code on Demand) — необязательное**
Сервер может расширять функциональность клиента, передавая исполняемый код (JavaScript в браузере, апплеты). Единственное необязательное ограничение из шести.

**Что чаще всего спрашивают на Junior-собеседованиях:** Stateless, Client-Server, Uniform Interface, Cacheable — эти четыре принципа обязательны для знания. Важно уметь объяснить практические последствия Stateless (токен авторизации должен передаваться в каждом запросе, а не храниться в сессии на сервере).

### Вопрос 6. В чём разница между @Controller и @RestController в Spring MVC?

**Категория:** REST/Web · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [@Controller vs @RestController — HowToDoInJava](https://howtodoinjava.com/spring-boot/controller-restcontroller/)

`@RestController` — это удобная аннотация, которая объединяет `@Controller` и `@ResponseBody`, избавляя от необходимости вручную добавлять `@ResponseBody` к каждому методу-обработчику.

`@Controller` помечает класс как Spring MVC контроллер и предназначен для традиционных веб-приложений, возвращающих HTML-страницы через шаблонизатор (Thymeleaf, JSP). Если нужно вернуть JSON, придётся явно добавлять `@ResponseBody` к каждому методу.

`@RestController` создан специально для RESTful API. Каждый метод автоматически сериализует возвращаемое значение в JSON/XML и записывает его напрямую в тело HTTP-ответа.

```java
// С @Controller нужен явный @ResponseBody для JSON
@Controller
@RequestMapping("users")
public class UserController {
    @GetMapping("{id}")
    public @ResponseBody User getUserById(@PathVariable Long id) {
        return user;
    }
}

// С @RestController @ResponseBody не нужен
@RestController
@RequestMapping("users")
public class UserController {
    @GetMapping("{id}")
    public User getUserById(@PathVariable Long id) {
        return user;
    }
}
```

#### Ответ от Claude

**`@Controller`** — стереотипная аннотация Spring MVC (мета-аннотирована `@Component`, поэтому бин регистрируется в контексте автоматически). По умолчанию методы контроллера возвращают **имя представления** (view name) — строку, которую `ViewResolver` преобразует в HTML-страницу (Thymeleaf, JSP и т.д.). Для возврата данных непосредственно в тело ответа (JSON, XML, plain text) каждый метод должен быть дополнительно помечен `@ResponseBody`.

**`@RestController`** — составная аннотация, введённая в Spring 4.0, эквивалентная `@Controller + @ResponseBody`. Аннотация `@ResponseBody` применяется ко всем методам класса автоматически. Возвращаемое значение сериализуется через `HttpMessageConverter` (по умолчанию Jackson для JSON) и записывается напрямую в тело HTTP-ответа. `ViewResolver` не вызывается.

**Ключевые различия:**

| | `@Controller` | `@RestController` |
|---|---|---|
| Входит в состав | `@Component` | `@Controller` + `@ResponseBody` |
| Возвращает по умолчанию | Имя представления (view name) | Данные в теле ответа |
| `@ResponseBody` | Нужно добавлять вручную к каждому методу | Применяется ко всем методам автоматически |
| `ViewResolver` | Используется | Не используется |
| Типичный сценарий | Server-side rendering (Thymeleaf, JSP) | REST API (JSON/XML) |
| Доступен с | Spring 2.5 | Spring 4.0 |

**Пример `@Controller` с возвратом view:**
```java
@Controller
public class PageController {
    @GetMapping("/dashboard")
    public String showDashboard(Model model) {
        model.addAttribute("user", currentUser);
        return "dashboard"; // Thymeleaf найдёт templates/dashboard.html
    }
}
```

**Пример `@RestController` для REST API:**
```java
@RestController
@RequestMapping("/api/users")
public class UserApiController {
    @GetMapping("/{id}")
    public User getUser(@PathVariable Long id) {
        return userService.findById(id); // Jackson сериализует в JSON
    }
}
```

**Важное уточнение:** если в классе с `@Controller` нужно вернуть и HTML-страницы, и JSON-ответы из разных методов, можно пометить отдельные методы `@ResponseBody`. `@RestController` так не позволяет — все методы всегда возвращают данные в тело ответа.
### Вопрос 7. Для чего используется аннотация @RequestMapping? Чем @GetMapping/@PostMapping отличаются от неё?

**Категория:** REST/Web · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Mapping Requests :: Spring Framework](https://docs.spring.io/spring-framework/reference/web/webmvc/mvc-controller/ann-requestmapping.html)

Аннотация `@RequestMapping` используется для сопоставления HTTP-запросов с методами-обработчиками в классах контроллеров. Она может применяться как на уровне класса (для определения базового пути), так и на уровне метода (для конкретного эндпоинта). Поддерживает атрибуты: `path`/`value` (URL-паттерн), `method` (HTTP-метод), `consumes`, `produces`, `params`, `headers`.

`@GetMapping`, `@PostMapping`, `@PutMapping`, `@DeleteMapping`, `@PatchMapping` — это «составные» (composed) аннотации, введённые в Spring 4.3. Они являются мета-аннотациями поверх `@RequestMapping` и переобъявляют его атрибуты с более узкой, конкретной целью. Например, `@GetMapping` эквивалентна `@RequestMapping(method = RequestMethod.GET)`. Документация рекомендует использовать составные аннотации на уровне методов, оставляя `@RequestMapping` для уровня класса (задание базового пути).

```java
@RestController
@RequestMapping("/persons")
public class PersonController {

    @GetMapping("/{id}")
    public Person getPerson(@PathVariable Long id) { ... }

    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public void add(@RequestBody Person person) { ... }
}
```

#### Ответ от Claude

`@RequestMapping` — универсальная аннотация Spring MVC для маппинга HTTP-запросов на методы контроллера. Поддерживает полный набор атрибутов:

| Атрибут | Назначение | Пример |
|---------|-----------|--------|
| `value` / `path` | URL-паттерн | `"/users/{id}"` |
| `method` | HTTP-метод(ы) | `RequestMethod.GET` |
| `consumes` | Ожидаемый Content-Type запроса | `"application/json"` |
| `produces` | Content-Type ответа | `"application/json"` |
| `params` | Условие на query-параметры | `"active=true"` |
| `headers` | Условие на заголовки | `"X-API-Version=2"` |

**Двухуровневое применение:**
- На **классе** — задаёт базовый путь, который наследуют все методы.
- На **методе** — уточняет/дополняет путь и ограничивает HTTP-метод.

**Отличие `@GetMapping`/`@PostMapping` от `@RequestMapping`:**

`@GetMapping`, `@PostMapping`, `@PutMapping`, `@DeleteMapping`, `@PatchMapping` — составные (composed) аннотации, появившиеся в Spring 4.3. Внутри каждой из них уже проставлен соответствующий `RequestMethod`, поэтому:

```java
// Старый способ — verbose
@RequestMapping(path = "/users/{id}", method = RequestMethod.GET)
public User getUser(@PathVariable Long id) { ... }

// Новый способ — лаконичный и явный
@GetMapping("/users/{id}")
public User getUser(@PathVariable Long id) { ... }
```

**Ключевые различия:**

1. **Явность HTTP-метода.** `@GetMapping` явно сигнализирует, что метод обрабатывает только GET-запросы; `@RequestMapping` без атрибута `method` по умолчанию принимает **любой** HTTP-метод — это потенциальная уязвимость или ошибка в REST API.
2. **Краткость.** Не нужно писать `method = RequestMethod.GET`.
3. **Область применения.** Составные аннотации предназначены **только для уровня метода**; `@RequestMapping` работает и на классе, и на методе.
4. **Полнота атрибутов.** Составные аннотации поддерживают те же атрибуты (`path`, `consumes`, `produces`, `params`, `headers`), что и `@RequestMapping`, — ничего не теряется.

**Практическая рекомендация:** использовать `@RequestMapping` на классе для базового пути, а `@GetMapping`/`@PostMapping` и т.д. — на методах.

```java
@RestController
@RequestMapping("/api/v1/orders")
public class OrderController {

    @GetMapping("/{id}")
    public Order getOrder(@PathVariable Long id) { ... }

    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public Order createOrder(@RequestBody OrderRequest request) { ... }

    @PutMapping("/{id}")
    public Order updateOrder(@PathVariable Long id, @RequestBody OrderRequest request) { ... }

    @DeleteMapping("/{id}")
    @ResponseStatus(HttpStatus.NO_CONTENT)
    public void deleteOrder(@PathVariable Long id) { ... }
}
```

### Вопрос 8. Для чего используется ResponseEntity в Spring MVC и когда его стоит возвращать из метода контроллера?

**Категория:** REST/Web · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [ResponseEntity :: Spring Framework](https://docs.spring.io/spring-framework/reference/web/webmvc/mvc-controller/ann-methods/responseentity.html)

`ResponseEntity` представляет весь HTTP-ответ целиком: статус-код, заголовки и тело. Используется в контроллерах Spring MVC как возвращаемый тип метода, когда необходима полная настройка HTTP-ответа. Spring самостоятельно конвертирует тело ответа через зарегистрированные `HttpMessageConverter`.

Пример из документации:
```java
@GetMapping("/something")
public ResponseEntity<String> handle() {
    String body = ...;
    String etag = ...;
    return ResponseEntity.ok().eTag(etag).body(body);
}
```

Spring MVC также поддерживает использование реактивных типов — `Mono<ResponseEntity<T>>` и `ResponseEntity<Mono<T>>` — для асинхронной обработки.

#### Ответ от Claude

`ResponseEntity<T>` — класс Spring (`org.springframework.http.ResponseEntity`), расширяющий `HttpEntity<T>` добавлением HTTP-статус-кода. Инкапсулирует три составляющих HTTP-ответа: **статус-код**, **заголовки** и **тело**.

**Когда стоит возвращать `ResponseEntity`:**

1. **Нестандартный статус-код** — когда нужно вернуть не просто `200 OK`, а, например, `201 Created`, `202 Accepted`, `204 No Content` или `404 Not Found` в зависимости от логики.
2. **Кастомные заголовки** — необходимо добавить `Location`, `ETag`, `Cache-Control`, `X-Custom-Header` и т.п.
3. **Условные ответы** — возврат разных статусов в зависимости от наличия ресурса (`200` vs `404`).
4. **Возврат ресурсов/файлов** — возврат `InputStreamResource` с нужными заголовками `Content-Disposition`.

**Когда `ResponseEntity` не нужен:**
- Метод всегда возвращает `200 OK` с телом — достаточно `@ResponseBody` или `@RestController` с прямым возвратом объекта.
- Статус задаётся статически — можно использовать `@ResponseStatus` на методе.

**Статические методы-строители (fluent API):**

```java
// 200 OK с телом
ResponseEntity.ok(user);

// 201 Created с заголовком Location
ResponseEntity.created(URI.create("/users/" + id)).body(newUser);

// 204 No Content
ResponseEntity.noContent().build();

// 404 Not Found без тела
ResponseEntity.notFound().build();

// Произвольный статус
ResponseEntity.status(HttpStatus.CONFLICT).body("User already exists");
```

**Практические примеры:**

```java
// Пример 1: создание ресурса с Location
@PostMapping("/users")
public ResponseEntity<User> createUser(@RequestBody UserRequest request) {
    User created = userService.create(request);
    URI location = URI.create("/users/" + created.getId());
    return ResponseEntity.created(location).body(created);
}

// Пример 2: получение ресурса, который может не существовать
@GetMapping("/users/{id}")
public ResponseEntity<User> getUser(@PathVariable Long id) {
    return userRepository.findById(id)
            .map(ResponseEntity::ok)
            .orElse(ResponseEntity.notFound().build());
}

// Пример 3: ответ с кастомным заголовком и ETag
@GetMapping("/documents/{id}")
public ResponseEntity<Document> getDocument(@PathVariable Long id) {
    Document doc = documentService.findById(id);
    String etag = "\"" + doc.getVersion() + "\"";
    return ResponseEntity.ok()
            .eTag(etag)
            .body(doc);
}
```

**Совет:** `ResponseEntity.of(Optional<T>)` и `ResponseEntity.ofNullable(T)` — удобные утилитные методы: возвращают `200 OK` если значение присутствует, и `404 Not Found` если нет.

```java
public ResponseEntity<User> getUser(Long id) {
    return ResponseEntity.of(userRepository.findById(id));
}
```

### Вопрос 9. Что делает аннотация @JsonIgnore в Jackson? Приведите практический пример применения.

**Категория:** REST/Web · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Jackson Annotations — @JsonIgnore](https://jenkov.com/tutorials/java-json/jackson-annotations.html)

Аннотация `@JsonIgnore` указывает Jackson игнорировать определённое поле (свойство) Java-объекта. Свойство игнорируется в обоих направлениях: при чтении JSON в Java-объект (десериализация) и при записи Java-объекта в JSON (сериализация).

Пример из туториала:

```java
import com.fasterxml.jackson.annotation.JsonIgnore;

public class PersonIgnore {
    @JsonIgnore
    public long personId = 0;

    public String name = null;
}
```

В данном примере поле `personId` полностью исключается из обработки Jackson: оно не будет включено в итоговый JSON и не будет читаться из входящего JSON. Поле `name` обрабатывается в штатном режиме.

#### Ответ от Claude

`@JsonIgnore` — маркерная аннотация из библиотеки Jackson (`com.fasterxml.jackson.annotation.JsonIgnore`), которая указывает, что помеченное поле, метод или параметр конструктора должно быть **полностью исключено из сериализации и десериализации**.

**Что делает:**
- При **сериализации** (Java → JSON): помеченное поле не попадает в итоговый JSON.
- При **десериализации** (JSON → Java): значение из JSON для этого поля игнорируется, поле в объекте остаётся равным умолчанию/null.

**Где можно ставить аннотацию:**
- На поле (`private String password`)
- На getter-методе (`getPassword()`)
- На setter-методе (`setPassword(...)`)
- На параметре конструктора с `@JsonCreator`

Достаточно поставить аннотацию **на один** из аксессоров — Jackson применит игнорирование ко всему свойству.

**Практический пример — скрытие пароля:**

```java
public class UserDto {
    private Long id;
    private String username;
    private String email;

    @JsonIgnore
    private String password;

    // getters и setters
}
```

Сериализация объекта `UserDto` даст:
```json
{
  "id": 1,
  "username": "john_doe",
  "email": "john@example.com"
}
```

Поле `password` отсутствует в JSON — чувствительные данные не утекут в ответ API.

**Пример с getter:**

```java
public class Employee {
    private String name;
    private String internalCode;

    @JsonIgnore
    public String getInternalCode() {
        return internalCode;
    }

    public void setInternalCode(String code) {
        this.internalCode = code;
    }
}
```

**Отмена игнорирования в подклассе** (value = false):

```java
public class AdminUserDto extends UserDto {

    @JsonIgnore(false) // переопределяем — пароль снова виден
    @Override
    public String getPassword() {
        return super.getPassword();
    }
}
```

**Альтернативы и отличия:**

| Аннотация | Область | Отличие |
|-----------|---------|---------|
| `@JsonIgnore` | Одно поле/метод | Полное исключение из сериализации И десериализации |
| `@JsonIgnoreProperties({"field1","field2"})` | Класс | Игнорирование списка полей на уровне класса |
| `@JsonProperty(access = READ_ONLY)` | Поле | Поле читается из JSON, но не пишется в ответ (или наоборот) |
| `@Transient` (JPA) | Поле | Исключение из персистентности, но НЕ из JSON (без `@JsonIgnore`) |

**Важный нюанс:** `@JsonIgnore` действует на **логическое свойство** (по имени), а не только на конкретный метод. Если добавить `@JsonIgnore` на getter, но оставить setter без аннотаций — свойство всё равно будет полностью проигнорировано. Чтобы разрешить только запись или только чтение, предпочтительнее использовать `@JsonProperty(access = JsonProperty.Access.WRITE_ONLY)` (для хранения пароля при десериализации, но без отдачи в JSON).

```java
public class UserDto {
    private String username;

    // Читается из JSON при создании пользователя, но никогда не отдаётся в ответе
    @JsonProperty(access = JsonProperty.Access.WRITE_ONLY)
    private String password;
}
```
### Вопрос 10. Как работает аннотация @Valid в Spring MVC? Какое исключение выбрасывается при провале валидации тела запроса?

**Категория:** REST/Web · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Validation :: Spring Framework — docs.spring.io](https://docs.spring.io/spring-framework/reference/web/webmvc/mvc-controller/ann-validation.html)

Bean Validation применяется к параметрам метода, аннотированным `@jakarta.validation.Valid` или Spring-аннотацией `@Validated`, — в частности к `@ModelAttribute`, `@RequestBody` и `@RequestPart`. Когда валидация отдельного параметра завершается неудачей и в сигнатуре метода сразу после него нет параметра типа `Errors` или `BindingResult`, Spring MVC выбрасывает `MethodArgumentNotValidException`. При наличии `BindingResult` исключение не выбрасывается — ошибки накапливаются в объекте `BindingResult` для ручной обработки. Начиная с Spring 6, при нарушении constraint-аннотаций (`@Min`, `@NotBlank` и др.) прямо на параметре метода может быть выброшен `HandlerMethodValidationException`; документация рекомендует обрабатывать оба исключения.

#### Ответ от Claude

**Как работает `@Valid` в Spring MVC**

`@Valid` — это аннотация из спецификации Jakarta Bean Validation (JSR-303/JSR-380). В Spring MVC она служит маркером, указывающим фреймворку запустить процесс валидации для аннотированного параметра метода контроллера.

**Механизм работы пошагово:**

1. Когда приходит HTTP-запрос, `DispatcherServlet` передаёт управление `RequestMappingHandlerAdapter`.
2. Адаптер использует `HandlerMethodArgumentResolver` (конкретно — `RequestResponseBodyMethodProcessor` для `@RequestBody`) для разбора тела запроса в Java-объект.
3. Если параметр аннотирован `@Valid`, резолвер делегирует валидацию в `LocalValidatorFactoryBean` — Spring-обёртку над `jakarta.validation.Validator` (по умолчанию реализация — Hibernate Validator).
4. `LocalValidatorFactoryBean` обходит поля объекта и вызывает `ConstraintValidator#isValid()` для каждой constraint-аннотации (`@NotNull`, `@Size`, `@Email` и т.д.).
5. Нарушения собираются в `BindingResult`.

**Какое исключение выбрасывается:**

- Если параметр метода — `@RequestBody` и за ним нет `BindingResult`, при провале валидации Spring выбрасывает `MethodArgumentNotValidException` (пакет `org.springframework.web.bind`).
- `MethodArgumentNotValidException` расширяет `BindException` и реализует интерфейс `ErrorResponse` (поддержка RFC 9457 / `ProblemDetail`).
- По умолчанию Spring Boot обрабатывает это исключение и возвращает ответ со статусом **HTTP 400 Bad Request**.

**Дополнительно — `HandlerMethodValidationException`:**

Начиная со Spring Framework 6, если constraint-аннотации (`@Min`, `@NotBlank` и т.п.) размещены **непосредственно на параметре метода** (а не на полях объекта), при нарушении выбрасывается `HandlerMethodValidationException`. Документация Spring рекомендует обрабатывать оба исключения.

**Пример:**

```java
@PostMapping("/users")
public ResponseEntity<Void> createUser(@Valid @RequestBody UserDto dto) {
    // При ошибке валидации dto — MethodArgumentNotValidException
    return ResponseEntity.created(...).build();
}
```

**Разница `@Valid` vs `@Validated`:**

- `@Valid` (Jakarta) — поддерживает валидацию вложенных объектов (cascaded validation), не поддерживает группы.
- `@Validated` (Spring) — поддерживает группы валидации; используется на уровне классов для включения метод-уровневой валидации через AOP (`MethodValidationPostProcessor`).

### Вопрос 11. Что такое модель зрелости Ричардсона? Опишите все четыре уровня (0–3) и что добавляет каждый из них.

**Категория:** REST/Web · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Richardson Maturity Model — martinfowler.com](https://martinfowler.com/articles/richardsonMaturityModel.html)

Модель зрелости Ричардсона разбивает принципы REST на последовательные ступени, каждая из которых добавляет новый уровень использования возможностей HTTP.

**Уровень 0 — «Болото POX» (The Swamp of POX):** HTTP используется исключительно как транспортный туннель для RPC-вызовов. Все запросы идут на один URI через POST, ответы — всегда 200 OK. Никакой семантики HTTP не используется. Примеры: XML-RPC, ранний SOAP.

**Уровень 1 — Ресурсы (Resources):** Вместо единственного endpoint появляются отдельные URI для каждого ресурса (`/doctors/mjones`, `/slots/1234`). Сложность снижается по принципу «разделяй и властвуй» — каждый ресурс обрабатывается отдельно.

**Уровень 2 — HTTP-глаголы (HTTP Verbs):** Сервис использует HTTP-методы семантически: GET для безопасного получения данных (с возможностью кэширования), POST для создания, PUT/PATCH для обновления, DELETE для удаления. HTTP-коды ответов используются по назначению: 201 Created, 409 Conflict и т.д.

**Уровень 3 — Гипермедиа (HATEOAS):** В ответы встраиваются ссылки на возможные следующие действия. Клиент не хардкодит URI — он обнаруживает их из ответов сервера. Это делает протокол самодокументируемым и позволяет серверу менять URI без поломки клиентов.

Рой Филдинг определил уровень 3 как **обязательное условие** для того, чтобы API считался настоящим REST.

#### Ответ от Claude

**Что такое модель зрелости Ричардсона**

Модель зрелости Ричардсона (Richardson Maturity Model, RMM) — классификационная схема, предложенная Леонардом Ричардсоном и популяризованная Мартином Фаулером. Она разбивает путь к «истинному REST» на четыре уровня, каждый из которых добавляет один ключевой элемент.

---

**Уровень 0 — «Болото POX» (The Swamp of POX)**

HTTP используется **только как транспорт** — по аналогии с CORBA или RPC. Все взаимодействия сводятся к одному endpoint (например, `POST /api`), в теле передаётся XML или JSON с описанием вызываемой операции. HTTP-методы не несут семантики (всё через POST), статус-коды не отражают результат (всегда 200, а ошибка прячется в теле).

*Что нового на этом уровне:* ничего — это отправная точка, «нулевой километр».

---

**Уровень 1 — Ресурсы (Resources)**

Вводится концепция **адресуемых ресурсов**: каждая сущность предметной области получает свой URI (`/orders/42`, `/customers/7`). Вместо одного «god-endpoint» появляются десятки специализированных. Благодаря этому сложность системы снижается — каждый ресурс обрабатывается отдельно.

*Что нового:* уникальные URI для ресурсов; принцип «разделяй и властвуй» для управления сложностью.

---

**Уровень 2 — HTTP-глаголы и коды ответов (HTTP Verbs)**

HTTP-методы применяются **семантически**:
- `GET` — безопасное и идемпотентное чтение (кэшируемо);
- `POST` — создание ресурса или не-идемпотентные операции;
- `PUT` / `PATCH` — обновление;
- `DELETE` — удаление.

HTTP-статус-коды отражают результат: `200 OK`, `201 Created`, `204 No Content`, `400 Bad Request`, `404 Not Found`, `409 Conflict` и т.д.

*Что нового:* стандартизированное использование HTTP-глаголов и кодов, устранение избыточного разнообразия в обработке однотипных ситуаций.

Большинство реальных API, называемых «RESTful», находятся именно на уровне 2.

---

**Уровень 3 — Гипермедиа / HATEOAS (Hypermedia Controls)**

В тело ответа встраиваются **ссылки** на возможные следующие действия. Клиент, получив ресурс, видит, какие операции над ним допустимы, — и переходит по ссылкам, а не хардкодит URI.

Пример фрагмента ответа:
```xml
<appointment>
  <slot id="1234" doctor="mjones" start="1400" end="1450"/>
  <patient id="jsmith"/>
  <link rel="cancel" uri="/slots/1234/appointment"/>
  <link rel="addTest" uri="/slots/1234/appointment/tests"/>
</appointment>
```

*Что нового:* самообнаружение API (discoverability), возможность изменять URI на сервере без поломки клиентов, самодокументируемый протокол.

Рой Филдинг настаивал, что только уровень 3 соответствует настоящему определению REST.

---

**Итоговая таблица**

| Уровень | Название | Ключевое добавление |
|---------|----------|---------------------|
| 0 | Болото POX | — (отправная точка: HTTP как туннель) |
| 1 | Ресурсы | Уникальные URI для каждого ресурса |
| 2 | HTTP-глаголы | Семантические методы + статус-коды |
| 3 | Гипермедиа | Встроенные ссылки (HATEOAS), самообнаружение |

### Вопрос 12. Что такое HATEOAS? Как оно связано с уровнем 3 модели Ричардсона и почему на практике редко достигается?

**Категория:** REST/Web · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [HATEOAS — Wikipedia](https://en.wikipedia.org/wiki/HATEOAS)

HATEOAS (Hypermedia as the Engine of Application State) — ограничение REST-архитектуры, при котором клиент взаимодействует с сетевым приложением, динамически получая информацию о доступных действиях через гипермедиа, встроенную в ответы сервера. Клиент не нуждается в предварительном знании URI — он обнаруживает их из ответов. По мере изменения состояния ресурса меняется набор доступных ссылок. Например, для банковского счёта с положительным балансом ответ содержит ссылки на пополнение, снятие, перевод и закрытие счёта; для счёта с отрицательным балансом — только ссылку на пополнение. Реализация HATEOAS требует строгих принципов проектирования, «прямо противоположных краткосрочной эффективности», что делает её непривычной для многих разработчиков. JSON и XML не являются нативными гипермедиа-форматами (в отличие от HTML), что усложняет реализацию без специализированных форматов (HAL, Siren и др.).

#### Ответ от Claude

**Что такое HATEOAS**

HATEOAS (Hypermedia as the Engine of Application State) — один из ключевых архитектурных ограничений REST, сформулированных Роем Филдингом в его диссертации 2000 года. Суть принципа: **API само описывает возможные следующие действия**, встраивая ссылки (hypermedia controls) в тело каждого ответа. Клиент начинает с одного известного URI (обычно корневого), а дальше навигирует по API, следуя ссылкам из ответов — аналогично тому, как браузер перемещается по HTML-страницам.

**Связь с уровнем 3 модели Ричардсона**

HATEOAS — это и есть уровень 3 модели Ричардсона. Уровни 1 и 2 решают вопросы структурирования URI и корректного использования HTTP-протокола, а уровень 3 добавляет самообнаружение (discoverability): клиент узнаёт о возможных операциях не из документации, а из самих ответов. Рой Филдинг прямо указывал, что без гипермедиа API не является REST в строгом смысле.

**Пример ответа с HATEOAS (HAL-нотация):**
```json
{
  "accountId": 12345,
  "balance": 1500.00,
  "_links": {
    "self":     { "href": "/accounts/12345" },
    "withdraw": { "href": "/accounts/12345/withdraw" },
    "deposit":  { "href": "/accounts/12345/deposit" },
    "transfer": { "href": "/accounts/12345/transfer" },
    "close":    { "href": "/accounts/12345/close" }
  }
}
```

Если баланс уйдёт в минус, сервер вернёт только ссылку `deposit` — клиент автоматически «узнает», что снятие недоступно.

**Почему HATEOAS редко достигается на практике**

Несмотря на теоретическую элегантность, уровень 3 редко встречается в реальных API. Причины:

1. **Клиенты не используют ссылки.** Разработчики клиентов читают документацию и хардкодят URI напрямую. Даже если сервер возвращает ссылки, клиент игнорирует их. Публичные API таких компаний, как Stripe, GitHub, Twilio, технически являются «HTTP API уровня 2», а не настоящим REST.

2. **Отсутствие единого стандарта.** Существуют конкурирующие форматы гипермедиа: HAL, JSON-LD, Siren, Collection+JSON, JSON API, Hydra. Каждый формат несёт свою семантику и требует отдельной реализации; консенсуса в индустрии нет.

3. **Сложность реализации и увеличение размера ответов.** Генерация ссылок требует дополнительной логики на сервере, увеличивает размер payload и усложняет тестирование.

4. **Семантический разрыв.** Ссылки описывают, *куда* перейти, но не объясняют *смысл* данных. Клиент всё равно нуждается во внешней документации для понимания семантики полей.

5. **Реальная польза неочевидна.** Теоретический выигрыш от смены URI на сервере (без поломки клиентов) редко реализуется: клиенты не следуют ссылкам, поэтому смена URI всё равно сломает интеграцию.

**Инструменты в экосистеме Spring**

Spring предоставляет библиотеку **Spring HATEOAS** для упрощения реализации: классы `EntityModel`, `CollectionModel`, `RepresentationModel`, построитель ссылок `WebMvcLinkBuilder`. Это снижает трудоёмкость, но не меняет фундаментальных причин непопулярности подхода.

### Вопрос 13. В чём разница между @ControllerAdvice и @ExceptionHandler на уровне контроллера? Как настроить глобальную обработку ошибок в REST API?

**Категория:** REST/Web · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Exceptions :: Spring Framework — docs.spring.io](https://docs.spring.io/spring-framework/reference/web/webmvc/mvc-controller/ann-exceptionhandler.html)

`@ExceptionHandler` позволяет обрабатывать исключения, выброшенные методами контроллера. Если аннотация размещена внутри класса `@Controller`, она применяется только к этому контроллеру. Если тот же метод объявлен в классе `@ControllerAdvice`, он применяется ко всем контроллерам приложения. При наличии обоих вариантов локальный (`@Controller`) обработчик имеет приоритет над глобальным (`@ControllerAdvice`). Метод, помеченный `@ExceptionHandler`, может принимать в параметрах сам объект исключения, `HandlerMethod`, `WebRequest`, `HttpServletRequest`/`HttpServletResponse`, `HttpSession`, `Principal` и другие. Возвращаемые типы — `ResponseEntity<T>`, `ProblemDetail`, строка-имя вида, `void` и другие.

#### Ответ от Claude

**`@ExceptionHandler` на уровне контроллера** — локальный обработчик. Метод с этой аннотацией, объявленный внутри `@Controller`/`@RestController`, перехватывает исключения **только** в этом контроллере:

```java
@RestController
public class UserController {

    @GetMapping("/users/{id}")
    public User getUser(@PathVariable Long id) {
        return userService.findById(id)
            .orElseThrow(() -> new UserNotFoundException(id));
    }

    // Действует только в UserController
    @ExceptionHandler(UserNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(UserNotFoundException ex) {
        return ResponseEntity.status(404)
            .body(new ErrorResponse("User not found", ex.getMessage()));
    }
}
```

**`@ControllerAdvice`** — глобальный перехватчик. Аннотация `@ControllerAdvice` является мета-аннотацией над `@Component`, поэтому класс автоматически обнаруживается при сканировании компонентов. Методы `@ExceptionHandler` внутри него применяются ко всем контроллерам приложения:

```java
@ControllerAdvice           // или @RestControllerAdvice для REST
public class GlobalExceptionHandler {

    @ExceptionHandler(UserNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(UserNotFoundException ex) {
        return ResponseEntity.status(404)
            .body(new ErrorResponse("Not Found", ex.getMessage()));
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleGeneric(Exception ex) {
        return ResponseEntity.status(500)
            .body(new ErrorResponse("Internal Error", ex.getMessage()));
    }
}
```

**Приоритет разрешения:** локальный `@ExceptionHandler` в контроллере всегда вызывается **раньше** глобального из `@ControllerAdvice`. Если локальный обработчик не найден, Spring ищет подходящий метод в `@ControllerAdvice`-бинах по порядку их приоритета (`@Order` / `@Priority`). При нескольких совпадениях предпочтение отдаётся совпадению по корневому (root) исключению перед совпадением по вложенному (cause).

**Сужение области действия `@ControllerAdvice`** через селекторы:

```java
// Только для контроллеров в пакете
@ControllerAdvice("com.example.api")
// Только для классов с @RestController
@ControllerAdvice(annotations = RestController.class)
// Только для конкретных типов контроллеров
@ControllerAdvice(assignableTypes = {UserController.class})
```

**Настройка глобальной обработки ошибок для REST API — типовой паттерн:**

1. Использовать `@RestControllerAdvice` (автоматически добавляет `@ResponseBody`, JSON-ответы без дополнительных аннотаций).
2. Возвращать `ResponseEntity<ErrorResponse>` или `ProblemDetail` (RFC 9457, поддерживается с Spring 6).
3. Упорядочить несколько `@ControllerAdvice`-классов через `@Order`.
4. Расширить `ResponseEntityExceptionHandler` для автоматической обработки стандартных исключений Spring MVC (400, 405, 415 и т.д.).

```java
@RestControllerAdvice
@Order(Ordered.HIGHEST_PRECEDENCE)
public class ApiExceptionHandler extends ResponseEntityExceptionHandler {

    @ExceptionHandler(EntityNotFoundException.class)
    public ProblemDetail handleEntityNotFound(EntityNotFoundException ex) {
        return ProblemDetail.forStatusAndDetail(HttpStatus.NOT_FOUND, ex.getMessage());
    }
}
```

**Ключевые отличия:**

| Аспект | `@ExceptionHandler` в контроллере | `@ControllerAdvice` |
|---|---|---|
| Область действия | Один контроллер | Все контроллеры (или выборка) |
| Приоритет | Выше (вызывается первым) | Ниже (fallback) |
| Переиспользование | Только через наследование | Без наследования |
| Бин Spring | Нет отдельного бина | Отдельный `@Component`-бин |

### Вопрос 14. Чем @RestControllerAdvice отличается от @ControllerAdvice?

**Категория:** REST/Web · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [RestControllerAdvice — Spring Framework 7.0.7 Javadoc](https://docs.spring.io/spring-framework/docs/current/javadoc-api/org/springframework/web/bind/annotation/RestControllerAdvice.html)

`@RestControllerAdvice` — это сокращённая аннотация, которая объединяет `@ControllerAdvice` и `@ResponseBody`. По сути, это `@ControllerAdvice`, чьи методы-обработчики исключений рендерят результат непосредственно в тело ответа. По умолчанию применяется ко всем контроллерам. Поддерживает те же селекторы (basePackages, basePackageClasses, assignableTypes, annotations), что и `@ControllerAdvice`.

#### Ответ от Claude

`@RestControllerAdvice` — это **составная (composed) аннотация**, эквивалентная `@ControllerAdvice + @ResponseBody`:

```java
// Определение @RestControllerAdvice в исходниках Spring:
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Documented
@ControllerAdvice
@ResponseBody
public @interface RestControllerAdvice {
    // те же атрибуты, что у @ControllerAdvice
}
```

**Практическое различие** — в том, как обрабатывается возвращаемое значение методов `@ExceptionHandler`:

| | `@ControllerAdvice` | `@RestControllerAdvice` |
|---|---|---|
| Возвращаемое значение | Интерпретируется как имя вида **или** сериализуется (если метод помечен `@ResponseBody`) | Всегда сериализуется в тело HTTP-ответа через `HttpMessageConverter` |
| Нужен `@ResponseBody` на каждом методе | Да (без него строка — имя вида) | Нет |
| Типичный use-case | MVC-приложение с шаблонами (Thymeleaf, JSP) | REST API (JSON/XML ответы) |

**Пример с `@ControllerAdvice`** — без `@ResponseBody` на методе, возвращаемая строка трактуется как имя вида:

```java
@ControllerAdvice
public class MvcExceptionHandler {

    // Возвращает вид "error" (шаблон error.html)
    @ExceptionHandler(Exception.class)
    public String handleException(Exception ex, Model model) {
        model.addAttribute("message", ex.getMessage());
        return "error";
    }

    // Чтобы вернуть JSON, нужен @ResponseBody
    @ExceptionHandler(ApiException.class)
    @ResponseBody
    public ResponseEntity<ErrorResponse> handleApiException(ApiException ex) {
        return ResponseEntity.badRequest().body(new ErrorResponse(ex.getMessage()));
    }
}
```

**Пример с `@RestControllerAdvice`** — `@ResponseBody` не нужен, все методы возвращают тело ответа:

```java
@RestControllerAdvice
public class RestExceptionHandler {

    @ExceptionHandler(EntityNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public ErrorResponse handleNotFound(EntityNotFoundException ex) {
        return new ErrorResponse("NOT_FOUND", ex.getMessage());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public ErrorResponse handleValidation(MethodArgumentNotValidException ex) {
        return new ErrorResponse("VALIDATION_ERROR", ex.getMessage());
    }
}
```

**Итог:** `@RestControllerAdvice` — это удобная аббревиатура для REST API, избавляющая от необходимости ставить `@ResponseBody` на каждый метод-обработчик. Функциональность идентична `@ControllerAdvice` с `@ResponseBody` на всех методах. Для приложений с шаблонизатором, где обработчики ошибок возвращают имена видов, следует использовать `@ControllerAdvice`.

### Вопрос 15. Как Jackson определяет, какие поля сериализовать? Какую роль играют аннотации @JsonProperty, @JsonInclude и настройки ObjectMapper?

**Категория:** REST/Web · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Jackson - Decide What Fields Get (De)Serialized — Baeldung](https://www.baeldung.com/jackson-field-serializable-deserializable-or-not)

По умолчанию Jackson 2 работает только с полями, которые либо объявлены `public`, либо имеют `public` геттер. Простейший способ сделать поле сериализуемым и десериализуемым — объявить его публичным. `public` геттер делает приватное поле сериализуемым (и десериализуемым, если есть соответствующий сеттер). Настроить видимость глобально можно через `ObjectMapper`:

```java
objectMapper.setVisibility(PropertyAccessor.FIELD, Visibility.ANY);
```

Это включает сериализацию приватных и package-private полей без геттеров. Аннотация `@JsonAutoDetect` позволяет переопределить правила видимости на уровне класса.

#### Ответ от Claude

**Как Jackson выбирает поля для сериализации**

По умолчанию Jackson применяет следующие правила видимости (auto-detection):

| Элемент | Минимальная видимость по умолчанию |
|---|---|
| Поля (fields) | `public` |
| Геттеры | `public` |
| Сеттеры | любой (`public` рекомендован) |
| Конструкторы | `public` |

Алгоритм выбора свойств: Jackson собирает кандидатов из публичных полей и публичных getter-методов (имена вида `getXxx()` / `isXxx()`), дедуплицирует их по логическому имени, после чего применяет аннотации и фильтры.

**Изменение видимости через `@JsonAutoDetect`** (на уровне класса):

```java
@JsonAutoDetect(fieldVisibility = JsonAutoDetect.Visibility.ANY)
public class User {
    private String name;    // теперь сериализуется
    private int age;        // теперь сериализуется
}
```

**Глобальное изменение видимости через `ObjectMapper`:**

```java
ObjectMapper mapper = new ObjectMapper();
// Сериализовать все поля независимо от модификатора доступа
mapper.setVisibility(PropertyAccessor.FIELD, Visibility.ANY);
// Отключить определение свойств через геттеры
mapper.setVisibility(PropertyAccessor.GETTER, Visibility.NONE);
```

---

**Аннотация `@JsonProperty`**

Маркирует поле, метод или параметр конструктора как JSON-свойство. Основные атрибуты:

- **`value`** — задаёт имя в JSON (по умолчанию — имя поля/метода):
  ```java
  @JsonProperty("user_name")
  private String userName;   // в JSON: "user_name"
  ```

- **`access`** — управляет направлением сериализации:
  ```java
  @JsonProperty(access = JsonProperty.Access.READ_ONLY)
  private String passwordHash;   // только в исходящий JSON, не читается при десериализации

  @JsonProperty(access = JsonProperty.Access.WRITE_ONLY)
  private String rawPassword;    // только десериализация (принимается), в JSON не включается
  ```
  Значения: `AUTO` (по умолчанию), `READ_ONLY`, `WRITE_ONLY`, `READ_WRITE`.

- **`required`** — флаг обязательности при десериализации (работает только для Creator Properties, т.е. параметров конструктора/фабричного метода).

- **`defaultValue`** — документирует значение по умолчанию (влияет на генерацию схемы, не на поведение во время выполнения).

Важно: `@JsonProperty` на приватном поле без геттера делает это поле видимым для Jackson (переопределяет правила видимости).

---

**Аннотация `@JsonInclude`**

Управляет условиями включения свойства в выходной JSON. Применяется к полю, методу или классу целиком.

Значения `JsonInclude.Include`:

| Значение | Описание |
|---|---|
| `ALWAYS` | Включать всегда (поведение по умолчанию) |
| `NON_NULL` | Исключить, если значение `null` |
| `NON_ABSENT` | Исключить `null` и `Optional.empty()` |
| `NON_EMPTY` | Исключить `null`, пустые строки, пустые коллекции/массивы |
| `NON_DEFAULT` | Исключить, если значение равно значению по умолчанию для типа |
| `CUSTOM` | Использовать пользовательский фильтр-класс (с Jackson 2.9) |

```java
// На уровне поля
public class Order {
    @JsonInclude(JsonInclude.Include.NON_NULL)
    private String comment;      // не включается, если null

    @JsonInclude(JsonInclude.Include.NON_EMPTY)
    private List<Item> items;    // не включается, если null или пустой список
}

// На уровне класса
@JsonInclude(JsonInclude.Include.NON_NULL)
public class ApiResponse {
    private String data;
    private String error;        // оба поля: не включаются, если null
}
```

**Важная оговорка из документации FasterXML:** аннотация оценивает включение на уровне Java-объекта, а не JSON-вывода. Например, `AtomicReference`, содержащий `null`, при `NON_NULL` всё равно сериализуется в JSON `null`, поскольку сам `AtomicReference` не равен `null`.

---

**Настройки `ObjectMapper`**

`ObjectMapper` — центральный объект Jackson для сериализации/десериализации. Должен быть сконфигурирован **до** первого вызова read/write (потокобезопасен после конфигурирования).

```java
ObjectMapper mapper = new ObjectMapper()
    // Глобальное исключение null-полей
    .setSerializationInclusion(JsonInclude.Include.NON_NULL)
    // Красивый вывод JSON
    .enable(SerializationFeature.INDENT_OUTPUT)
    // Не падать при неизвестных полях в JSON
    .disable(DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES)
    // Писать даты как строки, а не числа
    .disable(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS)
    // Сериализовать все поля (включая private)
    .setVisibility(PropertyAccessor.FIELD, Visibility.ANY);
```

**В Spring Boot** `ObjectMapper` настраивается через `application.properties` / `application.yml` или через бин `Jackson2ObjectMapperBuilderCustomizer`:

```yaml
spring:
  jackson:
    default-property-inclusion: non_null
    serialization:
      indent-output: false
      write-dates-as-timestamps: false
    deserialization:
      fail-on-unknown-properties: false
```

**Приоритет настроек (от меньшего к большему):**
1. Глобальные настройки `ObjectMapper` (наименьший приоритет).
2. Аннотации `@JsonInclude` на уровне класса.
3. Аннотации `@JsonInclude` / `@JsonProperty` на уровне поля/метода (наибольший приоритет).

**Исключение поля из сериализации** — аннотация `@JsonIgnore`:

```java
public class User {
    private String name;

    @JsonIgnore
    private String passwordHash;   // никогда не включается в JSON
}
```
### Вопрос 16. Как обработать ошибки валидации (@Valid) и вернуть клиенту структурированный JSON с описанием всех нарушений?

**Категория:** REST/Web · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Validation :: Spring Framework — docs.spring.io](https://docs.spring.io/spring-framework/reference/web/webmvc/mvc-controller/ann-validation.html)

Spring MVC поддерживает валидацию на двух уровнях. Первый — валидация отдельного параметра метода, аннотированного `@jakarta.validation.Valid` или `@Validated`: применяется к `@ModelAttribute`, `@RequestBody` и `@RequestPart`. Второй — валидация на уровне метода, когда аннотации-ограничения (`@Min`, `@NotBlank` и т.п.) объявлены непосредственно на параметрах метода.

При ошибке валидации Spring бросает одно из двух исключений:
- `MethodArgumentNotValidException` — при провале валидации одного объекта (`@RequestBody` с `@Valid`);
- `HandlerMethodValidationException` — при провале валидации параметров метода.

Для возврата структурированного JSON со всеми нарушениями нужно создать `@ControllerAdvice`, расширяющий `ResponseEntityExceptionHandler`, и переопределить метод `handleMethodArgumentNotValid`. Через `ex.getBindingResult().getFieldErrors()` извлекаются все нарушения полей. Современный подход (Spring 6+) использует `ProblemDetail` согласно RFC 9457. В Spring Boot достаточно добавить `spring.mvc.problemdetails.enabled=true` в `application.properties`.

#### Ответ от Claude

**Механизм возникновения ошибок**

Когда контроллер принимает объект с аннотацией `@Valid` и данные не проходят Bean Validation, Spring бросает `MethodArgumentNotValidException`. Это исключение содержит `BindingResult` со списком всех нарушений — как для конкретных полей (`FieldError`), так и для объекта в целом (`ObjectError`).

```java
@PostMapping("/users")
public ResponseEntity<User> createUser(@Valid @RequestBody User user) {
    return ResponseEntity.ok(userService.save(user));
    // При ошибке валидации автоматически бросается
    // MethodArgumentNotValidException
}
```

**Подход 1: @ControllerAdvice с ResponseEntityExceptionHandler (классический)**

```java
@ControllerAdvice
public class GlobalExceptionHandler extends ResponseEntityExceptionHandler {

    @Override
    protected ResponseEntity<Object> handleMethodArgumentNotValid(
            MethodArgumentNotValidException ex,
            HttpHeaders headers,
            HttpStatusCode status,
            WebRequest request) {

        Map<String, Object> body = new LinkedHashMap<>();
        body.put("timestamp", LocalDateTime.now());
        body.put("status", status.value());

        // Собираем все нарушения полей
        List<Map<String, String>> errors = ex.getBindingResult()
            .getFieldErrors()
            .stream()
            .map(fieldError -> Map.of(
                "field",   fieldError.getField(),
                "message", fieldError.getDefaultMessage(),
                "rejected", String.valueOf(fieldError.getRejectedValue())
            ))
            .toList();

        body.put("errors", errors);
        return new ResponseEntity<>(body, status);
    }
}
```

Пример JSON-ответа (HTTP 400):

```json
{
  "timestamp": "2024-05-10T12:00:00",
  "status": 400,
  "errors": [
    { "field": "email",  "message": "must be a valid email address", "rejected": "not-an-email" },
    { "field": "name",   "message": "must not be blank",             "rejected": "" }
  ]
}
```

**Подход 2: ProblemDetail / RFC 9457 (Spring 6+ / Spring Boot 3+)**

RFC 9457 определяет стандартный формат тела ответа об ошибке. Spring 6 добавил поддержку через `ProblemDetail`:

```java
@ControllerAdvice
public class GlobalExceptionHandler extends ResponseEntityExceptionHandler {

    @Override
    protected ResponseEntity<Object> handleMethodArgumentNotValid(
            MethodArgumentNotValidException ex,
            HttpHeaders headers,
            HttpStatusCode status,
            WebRequest request) {

        ProblemDetail pd = ProblemDetail.forStatusAndDetail(
            HttpStatus.BAD_REQUEST, "Запрос содержит ошибки валидации"
        );
        pd.setTitle("Validation Failed");

        List<Map<String, String>> errors = ex.getBindingResult()
            .getFieldErrors()
            .stream()
            .map(fe -> Map.of(
                "field",   fe.getField(),
                "message", fe.getDefaultMessage()
            ))
            .toList();

        pd.setProperty("errors", errors);

        return ResponseEntity.badRequest().body(pd);
    }
}
```

В Spring Boot 3 можно включить автоматическую обработку через `application.properties`:

```properties
spring.mvc.problemdetails.enabled=true
```

**Подход 3: Inline через BindingResult (без исключения)**

Если метод контроллера имеет `BindingResult` сразу за аннотированным параметром, исключение не бросается — разработчик управляет ответом сам:

```java
@PostMapping("/users")
public ResponseEntity<?> createUser(
        @Valid @RequestBody User user,
        BindingResult bindingResult) {

    if (bindingResult.hasErrors()) {
        Map<String, String> errors = new HashMap<>();
        bindingResult.getFieldErrors().forEach(fe ->
            errors.put(fe.getField(), fe.getDefaultMessage())
        );
        return ResponseEntity.badRequest().body(errors);
    }
    return ResponseEntity.ok(userService.save(user));
}
```

**Важные нюансы**

- `getFieldErrors()` возвращает только нарушения конкретных полей; `getAllErrors()` включает ещё и нарушения на уровне объекта.
- В Spring Framework 6.1+ при использовании `@Constraint` прямо на параметрах метода (без `@RequestBody`) бросается `HandlerMethodValidationException` — его тоже нужно обрабатывать.
- Сообщения можно интернационализировать через `MessageSource` — Bean Validation разрешает коды сообщений из `messages.properties`.
- Content-Type ответа при ProblemDetail автоматически становится `application/problem+json`.

### Вопрос 17. Как в Spring MVC передать переменную пути и параметр запроса в метод контроллера? В чём разница между @PathVariable и @RequestParam?

**Категория:** REST/Web · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [PathVariable (Spring Framework 7.0.7 API) — docs.spring.io](https://docs.spring.io/spring-framework/docs/current/javadoc-api/org/springframework/web/bind/annotation/PathVariable.html)

`@PathVariable` — аннотация Spring для привязки параметра метода к переменной URI-шаблона в обработчике, аннотированном `@RequestMapping`. Атрибуты: `name`/`value` (имя переменной в шаблоне, алиасы) и `required` (по умолчанию `true`). Если параметр объявлен как `Map<String, String>`, то в него помещаются все переменные пути сразу. Поддерживается с Spring 3.0.

`@RequestParam` привязывает параметр метода к параметру веб-запроса (query string, form data, multipart). Атрибуты: `name`/`value`, `required` (по умолчанию `true`), `defaultValue` (неявно делает `required=false`). Поддерживает `Map`, `MultiValueMap` и `List` для получения множественных значений. В Spring MVC охватывает query parameters, form data и части multipart-запроса.

#### Ответ от Claude

**@PathVariable — переменная пути**

Извлекает значение из сегмента URI, определённого через `{varName}` в шаблоне маппинга:

```java
// URL: GET /api/users/42
@GetMapping("/api/users/{id}")
public User getUser(@PathVariable Long id) {
    return userService.findById(id);
}

// Явное указание имени (если имена не совпадают)
@GetMapping("/api/orders/{orderId}")
public Order getOrder(@PathVariable(name = "orderId") Long id) { ... }

// Несколько переменных в пути
@GetMapping("/api/owners/{ownerId}/pets/{petId}")
public Pet getPet(@PathVariable Long ownerId, @PathVariable Long petId) { ... }

// Получение всех переменных сразу
@GetMapping("/api/{category}/{id}")
public String handle(@PathVariable Map<String, String> pathVars) {
    String category = pathVars.get("category");
    String id = pathVars.get("id");
    ...
}
```

Поддерживаются регулярные выражения в шаблоне:

```java
// URL вида /spring-web-3.0.5.jar
@GetMapping("/{name:[a-z-]+}-{version:\\d\\.\\d\\.\\d}{ext:\\.[a-z]+}")
public void handle(@PathVariable String name,
                   @PathVariable String version,
                   @PathVariable String ext) { ... }
```

**@RequestParam — параметр запроса**

Извлекает значение из query string (`?key=value`), form data или части multipart-запроса:

```java
// URL: GET /api/users?name=John&page=2
@GetMapping("/api/users")
public List<User> search(
        @RequestParam String name,
        @RequestParam(defaultValue = "0") int page,
        @RequestParam(required = false) String role) {
    return userService.search(name, page, role);
}

// Множественные значения одного параметра
// URL: ?color=red&color=blue
@GetMapping("/filter")
public List<Item> filter(@RequestParam List<String> color) { ... }

// Все параметры в Map
@GetMapping("/debug")
public String debug(@RequestParam Map<String, String> params) { ... }
```

**Сравнительная таблица**

| Характеристика         | `@PathVariable`                   | `@RequestParam`                          |
|------------------------|-----------------------------------|------------------------------------------|
| Источник значения      | Сегмент URL-пути                  | Query string, form data, multipart       |
| Пример URL             | `/users/42`                       | `/users?id=42`                           |
| Обязательность         | `required=true` по умолчанию      | `required=true` по умолчанию             |
| Значение по умолчанию  | Не поддерживается                 | `defaultValue` — поддерживается          |
| Кодирование URL        | Значение не URL-encoded           | Значение URL-encoded, автодекодирование  |
| Множественные значения | Только одно значение              | List, MultiValueMap                      |
| REST-семантика         | Идентификация ресурса             | Фильтрация, пагинация, опции             |

**Практические правила применения**

- `@PathVariable` используется для идентификации конкретного ресурса: `GET /articles/123`, `DELETE /users/42`.
- `@RequestParam` используется для необязательных модификаторов запроса: фильтрация (`?status=active`), сортировка (`?sort=name`), пагинация (`?page=0&size=20`).
- Оба параметра можно сделать необязательными через `required = false`; `@PathVariable(required = false)` появился в Spring 4.3.3.
- При несовпадении типов Spring бросает `TypeMismatchException`; кастомные конвертеры регистрируются через `ConversionService`.

### Вопрос 18. Как настроить Jackson для обработки полиморфных типов при сериализации/десериализации JSON (@JsonTypeInfo, @JsonSubTypes)?

**Категория:** REST/Web · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [JacksonPolymorphicDeserialization — FasterXML/jackson-docs Wiki (GitHub)](https://github.com/FasterXML/jackson-docs/wiki/JacksonPolymorphicDeserialization)

Jackson поддерживает полиморфную десериализацию через аннотацию `@JsonTypeInfo`. Она принимает два ключевых параметра.

Параметр `use` задаёт механизм идентификации типа:
- `Id.CLASS` — полное имя класса (`com.example.Dog`);
- `Id.MINIMAL_CLASS` — сокращённое имя класса (только пакет относительно базового);
- `Id.NAME` — логическое имя, заданное через `@JsonSubTypes`;
- `Id.CUSTOM` — пользовательский резолвер.

Параметр `include` задаёт способ встраивания информации о типе в JSON:
- `As.PROPERTY` — как отдельное поле JSON;
- `As.WRAPPER_OBJECT` — объект оборачивается в ещё один объект, ключ которого — идентификатор типа;
- `As.WRAPPER_ARRAY` — массив, где первый элемент — идентификатор типа.

Пример с `Id.NAME` и `@JsonSubTypes`:

```java
@JsonTypeInfo(use = Id.NAME, include = As.PROPERTY, property = "type")
@JsonSubTypes({
    @JsonSubTypes.Type(value = Dog.class, name = "dog"),
    @JsonSubTypes.Type(value = Cat.class, name = "cat")
})
abstract class Animal { public String name; }
```

JSON при сериализации `Dog`: `{"type":"dog","name":"Buddy","barkVolume":72.0}`.

#### Ответ от Claude

**Суть проблемы**

При десериализации JSON в поле с абстрактным типом (`Animal`) Jackson не знает, какой конкретный подкласс создать. Полиморфная обработка решает эту проблему, встраивая метаданные о типе в JSON.

**Аннотация @JsonTypeInfo**

Объявляется на базовом классе или интерфейсе иерархии.

```java
@JsonTypeInfo(
    use     = JsonTypeInfo.Id.NAME,       // способ идентификации типа
    include = JsonTypeInfo.As.PROPERTY,   // где хранить метаданные
    property = "type"                     // имя поля-дискриминатора
)
```

Варианты атрибута `use`:

| Значение           | Что пишется в JSON                              | Пример                         |
|--------------------|-------------------------------------------------|--------------------------------|
| `Id.CLASS`         | Полное имя класса                               | `"@class":"com.example.Dog"`   |
| `Id.MINIMAL_CLASS` | Укороченное имя (`.Dog` для пакета базового)    | `"@class":".Dog"`              |
| `Id.NAME`          | Логическое имя (задаётся через `@JsonSubTypes`) | `"type":"dog"`                 |
| `Id.CUSTOM`        | Результат пользовательского TypeIdResolver      | произвольно                    |

Варианты атрибута `include`:

| Значение          | Структура JSON                                       |
|-------------------|------------------------------------------------------|
| `As.PROPERTY`     | `{"type":"dog","name":"Rex","barkVolume":80}`         |
| `As.WRAPPER_OBJECT` | `{"dog":{"name":"Rex","barkVolume":80}}`            |
| `As.WRAPPER_ARRAY`  | `["dog",{"name":"Rex","barkVolume":80}]`            |
| `As.EXTERNAL_PROPERTY` | Поле типа находится во внешнем (родительском) объекте |

**Аннотация @JsonSubTypes**

Обязательна при `Id.NAME` — без неё Jackson не знает соответствие имени и класса:

```java
@JsonTypeInfo(use = JsonTypeInfo.Id.NAME, include = JsonTypeInfo.As.PROPERTY, property = "type")
@JsonSubTypes({
    @JsonSubTypes.Type(value = Dog.class, name = "dog"),
    @JsonSubTypes.Type(value = Cat.class, name = "cat")
})
public abstract class Animal {
    public String name;
}

public class Dog extends Animal {
    public double barkVolume;
}

public class Cat extends Animal {
    public boolean likesCream;
}
```

Сериализация:

```java
ObjectMapper mapper = new ObjectMapper();
Animal animal = new Dog();
animal.name = "Rex";
((Dog) animal).barkVolume = 85.0;

String json = mapper.writeValueAsString(animal);
// {"type":"dog","name":"Rex","barkVolume":85.0}
```

Десериализация:

```java
String json = "{\"type\":\"cat\",\"name\":\"Whiskers\",\"likesCream\":true}";
Animal animal = mapper.readValue(json, Animal.class);
// animal instanceof Cat == true
```

**Пример с Id.CLASS (без @JsonSubTypes)**

```java
@JsonTypeInfo(use = JsonTypeInfo.Id.CLASS, include = JsonTypeInfo.As.PROPERTY, property = "@class")
public abstract class Shape { }

public class Circle extends Shape { public double radius; }
```

JSON: `{"@class":"com.example.Circle","radius":5.0}`. Подход удобен, но хрупок при рефакторинге пакетов и небезопасен при десериализации недоверенных данных.

**Deduction-Based Polymorphism (Jackson 2.12+)**

Начиная с Jackson 2.12, можно не вставлять дискриминатор вовсе — Jackson сам выведет тип по набору полей:

```java
@JsonTypeInfo(use = JsonTypeInfo.Id.DEDUCTION)
@JsonSubTypes({
    @JsonSubTypes.Type(Dog.class),
    @JsonSubTypes.Type(Cat.class)
})
public abstract class Animal { }
```

JSON `{"name":"Rex","barkVolume":80}` будет десериализован в `Dog`, потому что только у `Dog` есть поле `barkVolume`.

**Программная (глобальная) настройка**

```java
PolymorphicTypeValidator ptv = BasicPolymorphicTypeValidator.builder()
    .allowIfSubType("com.example.")
    .build();

ObjectMapper mapper = new ObjectMapper();
mapper.activateDefaultTyping(ptv, ObjectMapper.DefaultTyping.NON_FINAL);
```

Внимание: `activateDefaultTyping` с недоверенными данными — потенциальная уязвимость (CVE-известные гаджеты десериализации). Используйте `PolymorphicTypeValidator` с явным allowlist.

**Интеграция со Spring**

В Spring Boot `ObjectMapper` конфигурируется через `spring.jackson.*` в `application.properties` или через `@Bean Jackson2ObjectMapperBuilderCustomizer`. Аннотации `@JsonTypeInfo` и `@JsonSubTypes` работают без дополнительной настройки — Jackson в Spring MVC регистрируется автоматически.
### Вопрос 19. Чем отличаются «безопасные» (safe) HTTP-методы от «идемпотентных» (idempotent)? Приведите примеры методов, которые идемпотентны, но не безопасны.

**Категория:** REST/Web · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [HTTP methods: Idempotency and Safety — mscharhag.com](https://www.mscharhag.com/api-design/http-idempotent-safe)

HTTP-методы считаются **безопасными**, если они не изменяют состояние сервера, то есть являются операциями только для чтения. **Идемпотентность** означает, что несколько одинаковых запросов дадут тот же результат, что и один запрос.

Все безопасные методы являются идемпотентными, но не все идемпотентные методы являются безопасными. **PUT** и **DELETE** — методы, которые идемпотентны, но не безопасны.

Пример DELETE: при отправке `DELETE /projects/123` сервер вернёт HTTP 200. Повторная отправка того же запроса может вернуть HTTP 404 (ресурс уже не существует). Несмотря на разные коды ответа, операция остаётся идемпотентной, поскольку «второй запрос не изменил состояние сервера». Идемпотентность — ценное свойство API, повышающее его отказоустойчивость.

#### Ответ от Claude

**Безопасные (safe) методы** — это методы, семантика которых предполагает только чтение: клиент не запрашивает и не ожидает изменения состояния на сервере. Определение дано в RFC 7231, §4.2.1. Безопасные методы: `GET`, `HEAD`, `OPTIONS`, `TRACE`.

**Идемпотентные (idempotent) методы** — методы, у которых результат N одинаковых последовательных запросов (N > 0) на состояние сервера эквивалентен результату одного запроса. Определение из RFC 7231, §4.2.2. Идемпотентные методы: `GET`, `HEAD`, `OPTIONS`, `TRACE`, `PUT`, `DELETE`.

**Ключевое различие:**

| Свойство | Описание | Что гарантирует |
|---|---|---|
| Safe (безопасный) | Не меняет состояние сервера | Отсутствие побочных эффектов |
| Idempotent (идемпотентный) | Повторные запросы дают тот же итоговый эффект на состояние | Безопасность повтора при сбоях сети |

Из определений следует: **все безопасные методы автоматически идемпотентны** (чтение не меняет состояние — сколько раз ни читай, результат тот же). Обратное неверно.

**Методы, идемпотентные, но НЕ безопасные:**

- **PUT** — заменяет ресурс целиком. Изменяет состояние сервера (не безопасен), но десять одинаковых `PUT /users/1` с одним и тем же телом оставят ресурс в одном и том же состоянии (идемпотентен).
- **DELETE** — удаляет ресурс. Изменяет состояние сервера (не безопасен), но после первого удаления повторные вызовы не меняют состояние — ресурс уже отсутствует (идемпотентен).

**Практическое значение:** идемпотентность позволяет клиенту безопасно повторять запрос при сетевом сбое, не опасаясь накопительных побочных эффектов. Именно поэтому PUT предпочтительнее POST для создания ресурса с известным идентификатором.

**POST** — не является ни безопасным, ни идемпотентным: каждый вызов `POST /orders` создаёт новый заказ.

### Вопрос 20. Является ли метод DELETE идемпотентным? Что вернёт сервер при повторном DELETE на уже удалённый ресурс — 200, 204 или 404 — и нарушает ли это идемпотентность?

**Категория:** REST/Web · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Is a HTTP DELETE request idempotent? — leedavis81.github.io](https://leedavis81.github.io/is-a-http-delete-requests-idempotent/)

Согласно RFC 2616, DELETE является идемпотентным. Спецификация определяет идемпотентность так: «побочные эффекты N > 0 одинаковых запросов совпадают с побочными эффектами одного запроса». После удаления ресурса повторные запросы не могут изменить его состояние, поэтому итоговый эффект идентичен.

Автор указывает, что ответ (статус-код) должен быть независим от результирующих побочных эффектов. Если побочные эффекты одинаковы, ответ всё же может отличаться. Возврат 404 для уже удалённого ресурса не нарушает идемпотентность операции — ресурс в обоих случаях отсутствует. Рекомендация: использовать 200 или 204 последовательно, так как они лучше отражают успешную операцию.

#### Ответ от Claude

**DELETE является идемпотентным** — это прямо закреплено в RFC 7231, §4.2.2. Обоснование: после удаления ресурса его состояние не меняется при повторных запросах — ресурса нет, и повторные DELETE не создают дополнительных изменений.

**Что вернёт сервер при повторном DELETE на уже удалённый ресурс:**

Спецификация (RFC 7231, §4.2.2) явно оговаривает, что идемпотентность относится к **состоянию сервера**, а не к коду ответа: *«the response might differ»* — ответ может отличаться.

Варианты ответа и их трактовка:

| Код | Смысл | Когда уместен |
|---|---|---|
| `200 OK` | Успех с телом ответа | Если сервер возвращает информацию об удалённом ресурсе |
| `204 No Content` | Успех без тела | Наиболее распространённый ответ на DELETE |
| `404 Not Found` | Ресурс не найден | При повторном DELETE уже удалённого ресурса |
| `410 Gone` | Ресурс существовал, но удалён | Семантически точнее 404 для известно-удалённых ресурсов |

**Нарушает ли 404 идемпотентность?**

**Нет, не нарушает.** Идемпотентность — это свойство воздействия на состояние сервера, а не тождественности HTTP-ответов. После первого DELETE ресурс отсутствует; после второго DELETE ресурс по-прежнему отсутствует — состояние сервера одинаково. RFC 7231 прямо допускает разные ответы при идемпотентных запросах.

**Рекомендации по практике:**

- Возврат `204` при повторном DELETE — наиболее «мягкий» подход, удобный для клиентов, реализующих retry-логику.
- Возврат `404` при повторном DELETE — технически корректно и не нарушает идемпотентность, но требует от клиента обработки этого кода как нормальной ситуации.
- Последовательность важнее выбора конкретного кода: API должно вести себя предсказуемо.
