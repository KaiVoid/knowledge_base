### Вопрос 1. Чем отличается аутентификация от авторизации? Приведи пример из реального приложения.

**Категория:** Безопасность · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Authentication Cheat Sheet — OWASP Cheat Sheet Series](https://cheatsheetseries.owasp.org/cheatsheets/Authentication_Cheat_Sheet.html)

Аутентификация (AuthN) — это процесс проверки того, что личность, объект или веб-сайт является тем, за кого себя выдаёт, путём проверки достоверности одного или нескольких аутентификаторов (паролей, отпечатков пальцев, токенов безопасности), используемых для подтверждения этого факта.

Авторизация (AuthZ) — это процесс предоставления или отклонения прав доступа к ресурсу. Авторизация всегда выполняется ПОСЛЕ аутентификации: сначала система выясняет, кто вы, и лишь затем — что вам разрешено делать.

Пользователь, прошедший аутентификацию, не обязательно авторизован для доступа ко всем ресурсам системы. Например, веб-приложение может иметь обычных пользователей и администраторов: оба прошли аутентификацию, но администратор авторизован выполнять действия, недоступные обычному пользователю. Кроме того, неаутентифицированный пользователь может быть авторизован для просмотра публичных ресурсов — например, страницы входа.

#### Ответ от Claude

**Аутентификация** — это проверка личности: «Ты действительно тот, за кого себя выдаёшь?» Система убеждается, что пользователь с логином `ivan@example.com` действительно является Иваном (например, он ввёл правильный пароль, прошёл 2FA или предъявил токен).

**Авторизация** — это проверка прав: «Тебе разрешено это делать?» Система, уже зная, кто вы, решает, можете ли вы выполнить конкретное действие или получить доступ к конкретному ресурсу.

**Ключевые различия:**

| Характеристика | Аутентификация | Авторизация |
|---|---|---|
| Вопрос | Кто ты? | Что тебе разрешено? |
| Порядок | Происходит первой | Происходит после аутентификации |
| Результат | Установленная личность | Набор прав/ролей |
| Провал | 401 Unauthorized | 403 Forbidden |

**Пример из реального приложения (интернет-магазин):**

1. Пользователь вводит логин и пароль — это **аутентификация**. Система проверяет учётные данные в БД. При успехе создаётся сессия/JWT-токен.
2. Аутентифицированный пользователь пытается открыть `/admin/orders` — это **авторизация**. Система проверяет, есть ли у него роль `ADMIN`. Обычный покупатель получит 403 Forbidden, несмотря на успешный вход.
3. Пользователь пытается открыть `/orders/42` (чужой заказ) — снова **авторизация**: у него нет прав на чужой ресурс.

**В Spring Security:**
```java
// Аутентификация: настройка механизма входа
http.formLogin(form -> form.loginPage("/login"));

// Авторизация: настройка прав доступа
http.authorizeHttpRequests(auth -> auth
    .requestMatchers("/admin/**").hasRole("ADMIN")
    .requestMatchers("/orders/**").authenticated()
    .anyRequest().permitAll()
);
```

Важно: термин HTTP `401 Unauthorized` исторически некорректен — он означает именно ошибку **аутентификации** (не авторизации). Ошибка авторизации — `403 Forbidden`.

### Вопрос 2. Что такое SQL-инъекция? Покажи на примере уязвимый запрос и объясни, как PreparedStatement устраняет эту уязвимость.

**Категория:** Безопасность · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [SQL Injection Prevention Cheat Sheet — OWASP Cheat Sheet Series](https://cheatsheetseries.owasp.org/cheatsheets/SQL_Injection_Prevention_Cheat_Sheet.html)

**Уязвимый код (конкатенация строк):**
```java
String query = "SELECT account_balance FROM user_data WHERE user_name = "
             + request.getParameter("customerName");
try {
    Statement statement = connection.createStatement();
    ResultSet results = statement.executeQuery(query);
}
```

Этот код напрямую конкатенирует пользовательский ввод, позволяя атакующему внедрить произвольные SQL-команды.

**Безопасный код (PreparedStatement):**
```java
String custname = request.getParameter("customerName");
String query = "SELECT account_balance FROM user_data WHERE user_name = ?";
PreparedStatement pstmt = connection.prepareStatement(query);
pstmt.setString(1, custname);
ResultSet results = pstmt.executeQuery();
```

Согласно OWASP: «база данных всегда будет различать код и данные, независимо от того, какой пользовательский ввод передан». При использовании параметризованных запросов «атакующий не может изменить намерение запроса, даже если SQL-команды вставлены атакующим».

Если атакующий введёт `tom' or '1'='1`, параметризованный подход воспримет это как буквальное строковое значение, а не как исполняемый SQL-код, защищая базу данных от инъекций.

#### Ответ от Claude

**SQL-инъекция** — это атака, при которой злоумышленник внедряет произвольный SQL-код в запрос, формируемый приложением из пользовательских данных без должной обработки. В результате атакующий может обойти аутентификацию, прочитать чужие данные, изменить или удалить записи, а в некоторых случаях — выполнить команды операционной системы.

**Уязвимый запрос:**
```java
// НЕБЕЗОПАСНО: конкатенация пользовательского ввода
String username = request.getParameter("username");
String sql = "SELECT * FROM users WHERE username = '" + username + "'";
Statement stmt = connection.createStatement();
ResultSet rs = stmt.executeQuery(sql);
```

Если атакующий передаст `username = ' OR '1'='1`, итоговый SQL станет:
```sql
SELECT * FROM users WHERE username = '' OR '1'='1'
```
Условие `'1'='1'` всегда истинно, поэтому запрос вернёт всех пользователей — авторизация обойдена.

Более опасный вариант: `'; DROP TABLE users; --` — уничтожит таблицу.

**Безопасный запрос через PreparedStatement:**
```java
// БЕЗОПАСНО: параметризованный запрос
String username = request.getParameter("username");
String sql = "SELECT * FROM users WHERE username = ?";
PreparedStatement pstmt = connection.prepareStatement(sql);
pstmt.setString(1, username);  // знак ? заменяется безопасно
ResultSet rs = pstmt.executeQuery();
```

**Почему PreparedStatement защищает:**

1. **Разделение кода и данных.** SQL-структура запроса компилируется СУБД заранее, до подстановки параметров. Значение параметра никогда не интерпретируется как SQL-код.
2. **Экранирование на уровне протокола.** При вызове `setString()` JDBC-драйвер передаёт значение как типизированный параметр, а не как часть строки запроса. Специальные символы (`'`, `"`, `;`, `--`) теряют своё синтаксическое значение.
3. **Неизменность структуры.** Даже если пользователь передаст `' OR '1'='1`, СУБД будет искать пользователя с буквальным именем `' OR '1'='1` — и ничего не найдёт.

**Важные нюансы:**
- PreparedStatement не защищает от инъекции в **динамические идентификаторы** (имена таблиц/столбцов) — их нельзя параметризовать. Для них необходимо использовать белый список допустимых значений.
- В JPA/Hibernate аналогичную защиту обеспечивают именованные параметры: `WHERE u.name = :name`.
- ORM (Hibernate, Spring Data JPA) по умолчанию используют параметризацию, но уязвимость возникает при ручном написании нативных запросов через конкатенацию строк.

### Вопрос 3. Почему нельзя хранить пароли пользователей в открытом виде или в виде MD5/SHA-1 хэша? Что такое bcrypt и зачем нужна соль?

**Категория:** Безопасность · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Password Storage Cheat Sheet — OWASP Cheat Sheet Series](https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html)

**Почему MD5/SHA-1 недостаточны.** Быстрые алгоритмы хэширования, такие как SHA-256 (и тем более MD5/SHA-1), не подходят для хранения паролей, потому что позволяют атакующим выполнять огромное количество попыток подбора в единицу времени. Современные GPU способны вычислять миллиарды MD5-хэшей в секунду, что делает перебор по словарю и брутфорс тривиальными задачами.

**Что такое соль.** Соль — это уникальная случайно сгенерированная строка, добавляемая к каждому паролю в процессе хэширования. Соль защищает от атак с использованием заранее вычисленных таблиц (rainbow tables) или баз данных хэшей, поскольку вынуждает атакующего вскрывать каждый пароль индивидуально.

**Что такое bcrypt.** Bcrypt — специализированная функция хэширования паролей, разработанная именно для безопасного хранения. Рекомендуется для устаревших систем, в которых недоступны Argon2 и scrypt. Параметр «work factor» (коэффициент сложности) определяет вычислительную стоимость: чем он выше, тем медленнее выполняется хэширование, что существенно затрудняет атаки перебором. Минимальное рекомендуемое значение work factor — **10**.

#### Ответ от Claude

**Почему нельзя хранить пароли в открытом виде:**

При компрометации базы данных (утечка, SQL-инъекция, инсайдер) все пароли немедленно становятся известны атакующему. Поскольку многие пользователи используют одни и те же пароли на разных сервисах, последствия распространяются далеко за пределы одного приложения.

**Почему MD5 и SHA-1 непригодны для паролей:**

MD5 и SHA-1 (как и SHA-256) — это **быстрые** криптографические хэш-функции. Они проектировались для проверки целостности данных, а не для защиты паролей. Их скорость — их главный недостаток при хранении паролей:

- Современная видеокарта (например, RTX 4090) способна вычислять порядка **60–100 миллиардов MD5-хэшей в секунду**.
- Весь словарь из 10 миллионов популярных паролей перебирается за доли секунды.
- Существуют готовые **rainbow tables** — огромные базы данных соответствия `хэш → пароль` для MD5/SHA-1.

Дополнительно: MD5 и SHA-1 имеют известные **коллизионные уязвимости** (два разных входа дают одинаковый хэш), хотя для атак на пароли это менее критично, чем их скорость.

**Что такое соль (salt):**

Соль — это случайное уникальное значение (обычно 16–32 байта), генерируемое отдельно для каждого пользователя и добавляемое к паролю перед хэшированием.

```
hash = bcrypt(password + salt)
```

Соль хранится в открытом виде рядом с хэшем — её секретность не нужна. Её задача — сделать одинаковые пароли у разных пользователей непохожими по хэшу:

```
Пользователь A: пароль "qwerty" + соль "a8f3..." → уникальный хэш A
Пользователь B: пароль "qwerty" + соль "7b2c..." → уникальный хэш B
```

Это уничтожает эффективность rainbow tables (таблицы строятся без учёта соли) и не позволяет атакующему одновременно взломать несколько аккаунтов с одинаковым паролем.

**Что такое bcrypt:**

Bcrypt — специализированный алгоритм хэширования паролей, разработанный в 1999 году Нильсом Провосом и Дэвидом Мазьером. Ключевые свойства:

1. **Намеренно медленный.** Алгоритм содержит настраиваемый параметр **work factor** (cost factor), экспоненциально увеличивающий время вычисления. При work factor = 10 одно хэширование занимает ~100 мс; при увеличении на 1 время удваивается.
2. **Встроенная соль.** Bcrypt автоматически генерирует и встраивает случайную соль в результирующую строку — не нужно управлять солью вручную.
3. **Формат результата.** Строка bcrypt имеет стандартный формат: `$2a$10$<22 символа соли><31 символ хэша>` — всего 60 символов.

**Пример в Java (Spring Security):**
```java
// Создание энкодера с work factor = 12
PasswordEncoder encoder = new BCryptPasswordEncoder(12);

// Хэширование пароля при регистрации
String rawPassword = "mySecretPassword";
String encodedPassword = encoder.encode(rawPassword);
// Результат: $2a$12$... (каждый вызов даёт разный хэш из-за случайной соли)

// Проверка пароля при входе
boolean matches = encoder.matches(rawPassword, encodedPassword); // true
```

**Текущие рекомендации OWASP (2025):**

Предпочтительный алгоритм — **Argon2id** (победитель Password Hashing Competition 2015). Bcrypt рекомендован для устаревших систем, где Argon2 недоступен, с work factor не ниже **10**. Никогда не использовать MD5, SHA-1, SHA-256 без адаптивного замедления.

```java
// Spring Security 5+ автоматически использует BCrypt по умолчанию
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder(12); // work factor 12 — разумный баланс
}
```

### Вопрос 4. Что такое XSS (Cross-Site Scripting)? Какие бывают виды XSS и как от него защититься на стороне Java-бэкенда?

**Категория:** Безопасность · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Types of XSS | OWASP Foundation](https://owasp.org/www-community/Types_of_Cross-Site_Scripting)

XSS (межсайтовый скриптинг) — атака, при которой злоумышленник внедряет вредоносный клиентский код (как правило, JavaScript) в веб-страницу, которую просматривают другие пользователи.

**Три основных вида XSS:**

1. **Reflected XSS (Отражённый)** — пользовательский ввод немедленно возвращается веб-приложением в ответе (например, в сообщении об ошибке или результате поиска) без надлежащей обработки и без постоянного хранения на сервере.

2. **Stored XSS (Хранимый)** — пользовательский ввод сохраняется на целевом сервере (в базе данных, форуме, поле комментария и т.д.) и впоследствии отображается другим пользователям. Этот вид особенно опасен, поскольку атака может затронуть множество пользователей.

3. **DOM-Based XSS** — атака, при которой полезная нагрузка выполняется в результате модификации DOM-среды в браузере жертвы клиентским кодом страницы. HTTP-ответ при этом не изменяется — клиентский JavaScript выполняется «неожиданным образом» из-за изменений в DOM.

OWASP также предлагает уточнённую матрицу: Server XSS (инъекция на стороне сервера) и Client XSS (инъекция на стороне клиента), которые в свою очередь могут быть Stored или Reflected.

#### Ответ от Claude

**Что такое XSS**

Cross-Site Scripting (XSS) — класс атак на веб-приложения, при которых злоумышленник внедряет вредоносный JavaScript (или иной клиентский код) в страницы, отображаемые другим пользователям. Браузер жертвы исполняет этот код в контексте доверенного сайта, что позволяет атакующему похитить сессионные куки, перехватить ввод, совершать действия от имени пользователя или перенаправлять его на фишинговые ресурсы.

**Виды XSS (классификация OWASP)**

| Вид | Где хранится payload | Как доставляется |
|-----|---------------------|-----------------|
| **Reflected XSS** | Нигде (только в URL/запросе) | Жертва переходит по специально сформированной ссылке; сервер «отражает» ввод в ответе без экранирования |
| **Stored XSS** | На сервере (БД, файл, лог) | Payload сохраняется и отображается всем пользователям, просматривающим заражённый ресурс |
| **DOM-based XSS** | В URL-фрагменте или параметре, читаемом JS | JavaScript на странице читает небезопасный источник (location.hash, document.referrer) и записывает его в DOM через innerHTML или eval |

Reflected и Stored XSS — серверные инъекции (уязвимость в том, как сервер генерирует HTML). DOM-based XSS — клиентская инъекция (уязвимость в JavaScript самой страницы).

**Защита на стороне Java-бэкенда**

1. **Контекстно-зависимое экранирование вывода (Output Encoding)** — главная мера. Правило: данные нужно экранировать непосредственно перед вставкой в HTML, в зависимости от контекста:
   - HTML-тело: экранировать `&`, `<`, `>`, `"`, `'` → HTML entities
   - HTML-атрибут: то же + пробелы
   - JavaScript-контекст: JavaScript-encoding
   - URL-параметр: URL-encoding (`%XX`)
   - CSS-значение: CSS-encoding

   Библиотеки для Java:
   - **OWASP Java Encoder** (`org.owasp.encoder:encoder`) — лёгкая, рекомендованная OWASP
   - **OWASP ESAPI** — более широкий набор функций, тяжелее
   - **Thymeleaf / FreeMarker** — шаблонизаторы с автоматическим экранированием по умолчанию

   ```java
   // OWASP Java Encoder
   String safe = Encode.forHtml(userInput);
   String safeAttr = Encode.forHtmlAttribute(userInput);
   String safeJs = Encode.forJavaScript(userInput);
   ```

2. **Валидация и санитизация входных данных** — дополнительный слой защиты. Использовать allowlist (белый список допустимых символов/форматов), а не blacklist. Для HTML-разметки (например, WYSIWYG) применять **OWASP AntiSamy** или **HtmlSanitizer**.

3. **Content Security Policy (CSP)** — HTTP-заголовок, ограничивающий источники допустимых скриптов. Устраняет риск выполнения инлайн-скриптов даже при наличии уязвимости:
   ```
   Content-Security-Policy: default-src 'self'; script-src 'self' https://trusted.cdn.com
   ```
   В Spring Boot добавляется через `HttpSecurity.headers().contentSecurityPolicy(...)`.

4. **Флаги куки** — `HttpOnly` предотвращает доступ к сессионной куке из JavaScript, что ограничивает ущерб от XSS-атак с кражей куки.

5. **DOM-based XSS специфично**: предотвращается исключительно на стороне клиента — заменой `innerHTML` на `textContent`, использованием безопасных DOM-API. Бэкенд здесь не поможет, если JS не управляется сервером.

**Итог**: главный приоритет — контекстно-зависимое экранирование вывода через OWASP Java Encoder или шаблонизатор с автоэкранированием. Валидация — дополнительный рубеж, CSP — глубокая защита.

### Вопрос 5. Что такое CSRF-атака? Как работает CSRF-токен и почему Spring Security по умолчанию включает CSRF-защиту?

**Категория:** Безопасность · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Cross Site Request Forgery (CSRF) :: Spring Security](https://docs.spring.io/spring-security/reference/servlet/exploits/csrf.html)

**Что такое CSRF-атака**

CSRF (Cross-Site Request Forgery) — атака, при которой злоумышленник обманом заставляет аутентифицированного пользователя отправить нежелательный запрос на сайт, где тот авторизован. Например, пользователь, вошедший в интернет-банк, может быть перенаправлен на вредоносную страницу, которая незаметно отправит запрос на перевод денег.

**Synchronizer Token Pattern**

Spring Security защищает от CSRF с помощью паттерна синхронизирующего токена (Synchronizer Token Pattern):
- Каждый HTTP-запрос, изменяющий состояние (POST, PUT, DELETE, PATCH), должен содержать безопасное случайное значение — CSRF-токен — в дополнение к сессионной куке.
- Сервер генерирует токен, сохраняет его (по умолчанию в `HttpSession`) и сравнивает с токеном из запроса.
- Если токены не совпадают, запрос отклоняется.

Токен должен передаваться в части запроса, которую браузер **не добавляет автоматически** — в параметре формы (`_csrf`) или HTTP-заголовке (например, `X-CSRF-TOKEN`). Именно поэтому куки не подходят для передачи CSRF-токена: браузер автоматически включает куки в каждый запрос к домену.

**Почему CSRF включена по умолчанию**

CSRF-защита включена по умолчанию потому, что небезопасные HTTP-методы (POST, PUT, DELETE) могут изменять состояние приложения, и приложение должно быть защищено от форгированных запросов без дополнительной конфигурации.

#### Ответ от Claude

**Что такое CSRF**

CSRF (Cross-Site Request Forgery, «межсайтовая подделка запроса») — атака, эксплуатирующая то, что браузер автоматически прикрепляет куки (в том числе сессионные) к каждому запросу к соответствующему домену. Атакующий создаёт вредоносную страницу, которая от имени жертвы отправляет запрос на целевой сайт. Сервер получает «легитимный» запрос с действующей сессионной кукой и выполняет нежелательное действие.

**Пример атаки**

```html
<!-- Страница злоумышленника -->
<img src="https://bank.example.com/transfer?to=attacker&amount=1000" />
<!-- Браузер автоматически добавит куку сессии к этому GET-запросу -->
```

Для POST-запросов используется скрытая форма с автосабмитом через JavaScript.

**Почему атака работает**

Браузер соблюдает same-origin policy для чтения ответов, но отправка запросов (включая куки) не ограничивается — именно этим пользуется CSRF.

**Synchronizer Token Pattern (как работает CSRF-токен)**

1. При загрузке страницы (или создании сессии) сервер генерирует криптографически случайный токен и сохраняет его на стороне сервера (в `HttpSession` — реализация по умолчанию в Spring Security через `HttpSessionCsrfTokenRepository`).
2. Токен встраивается в каждую HTML-форму как скрытое поле `_csrf` или передаётся клиенту в куке `XSRF-TOKEN` (при использовании `CookieCsrfTokenRepository`).
3. При отправке изменяющего состояние запроса (POST, PUT, DELETE, PATCH) клиент передаёт токен в параметре формы или заголовке `X-XSRF-TOKEN`.
4. Сервер сравнивает полученный токен с сохранённым. Несовпадение → HTTP 403.

Злоумышленник не может получить токен с чужой страницы из-за same-origin policy браузера — именно это делает паттерн эффективным.

**Spring Security: реализация по умолчанию**

```java
// Включена по умолчанию, явная конфигурация выглядит так:
http.csrf(Customizer.withDefaults());

// Альтернативное хранилище — для SPA-фреймворков (Angular, React):
http.csrf(csrf -> csrf
    .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse())
);

// Отключение — только для чисто stateless REST API с JWT:
http.csrf(csrf -> csrf.disable());
```

Spring Security по умолчанию использует `XorCsrfTokenRequestAttributeHandler`, который добавляет случайность к каждому токену для защиты от BREACH-атак (атак на сжатие TLS).

**Почему CSRF включена по умолчанию в Spring Security**

Spring Security следует принципу «безопасно по умолчанию» (secure by default). Браузерные приложения с сессионной аутентификацией уязвимы к CSRF по своей природе, поэтому защита включена сразу. Отключать её имеет смысл только для stateless REST API, использующих JWT или иные токены, которые браузер не отправляет автоматически (и которые не хранятся в куках).

**GET-запросы не требуют CSRF-токена**, так как по REST-соглашению они идемпотентны и не меняют состояние.

### Вопрос 6. В чём разница между сессиями и токенами (JWT) для хранения состояния аутентификации? Назови преимущества и недостатки каждого подхода.

**Категория:** Безопасность · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [JSON Web Token for Java - OWASP Cheat Sheet Series](https://cheatsheetseries.owasp.org/cheatsheets/JSON_Web_Token_for_Java_Cheat_Sheet.html)

JWT — самодостаточный формат: все необходимые данные (claims) содержатся внутри самого токена и подписаны сервером, поэтому для валидации JWT не требуется обращение к базе данных.

**Основные проблемы JWT, которые выделяет OWASP:**

- **Отзыв токена**: у пользователя нет встроенного механизма явного аннулирования токена. Если токен похищен, пользователь не может его отозвать. При явном выходе (logout) или таймауте простоя на стороне API необходимо вести denylist хэшей отозванных JWT вплоть до истечения срока их действия.
- **Stateless JWT должны быть короткоживущими**, чтобы минимизировать окно возможности для атакующего.
- **Хранение**: токен не должен храниться в localStorage из-за риска XSS; предпочтительнее httpOnly-куки.

Если приложению не требуется полная stateless-архитектура, OWASP рекомендует рассмотреть традиционные сессионные механизмы, предоставляемые веб-фреймворками.

#### Ответ от Claude

**Сессии (Session-based / Stateful)**

При сессионной аутентификации сервер хранит состояние пользователя на своей стороне. Клиент получает лишь непрозрачный идентификатор сессии (`JSESSIONID`), обычно в куке.

**Как работает:**
1. Пользователь логинится → сервер создаёт сессию в памяти / Redis / БД, возвращает `sessionId` в Set-Cookie.
2. Каждый следующий запрос браузер отправляет с этой кукой.
3. Сервер по `sessionId` достаёт данные пользователя из хранилища.

| Сессии — плюсы | Сессии — минусы |
|----------------|-----------------|
| Мгновенный отзыв: удалил сессию — пользователь разлогинен | Требует серверного хранилища (память, Redis, БД) |
| Простота реализации (стандартный механизм всех Java EE / Spring MVC) | Усложняет горизонтальное масштабирование (sticky sessions или общий session store) |
| Маленький payload на клиенте (только ID) | Не подходит для микросервисов с разными доменами без дополнительной инфраструктуры |
| Требует CSRF-защиты при использовании с куками | — |

---

**JWT-токены (Stateless)**

JWT (JSON Web Token) — это подписанный (и, опционально, зашифрованный) JSON-объект, содержащий claims (утверждения) о пользователе. Сервер не хранит состояние — достаточно проверить подпись.

**Структура JWT:** `header.payload.signature` (Base64URL-encoded).

**Как работает:**
1. Пользователь логинится → сервер генерирует и подписывает JWT (HS256 или RS256), возвращает клиенту.
2. Клиент хранит токен (httpOnly-кука или Authorization header) и передаёт его в каждом запросе.
3. Сервер проверяет подпись и срок действия (`exp`) — обращение к БД не требуется.

| JWT — плюсы | JWT — минусы |
|-------------|--------------|
| Stateless: нет серверного хранилища — просто масштабируется | Нет встроенного механизма отзыва до истечения `exp` |
| Хорошо работает в микросервисах и между доменами | Payload виден всем (если не использовать JWE) — нельзя хранить секреты |
| Самодостаточен: содержит роли, ID — не нужен доп. запрос к БД | При компрометации токена окно атаки = время до `exp` |
| Не требует CSRF-защиты (если не хранится в куке) | Требует грамотной реализации отзыва (denylist / refresh token rotation) |
| Подходит для OAuth2/OIDC и взаимодействия с third-party | Размер токена больше sessionId |

**Отзыв JWT**: обходные решения включают:
- Короткий TTL access token (15 мин) + долгоживущий refresh token с возможностью отзыва
- Denylist отозванных токенов в Redis (проверяется по `jti`)
- Refresh token rotation (каждый обмен порождает новую пару)

**Хранение на клиенте:**
- `httpOnly`-кука — защита от XSS (JS не может прочитать), но требует CSRF-защиты
- `localStorage` / `sessionStorage` — доступны JS, уязвимы к XSS; не рекомендуется OWASP

**Когда что выбирать:**

| Сценарий | Рекомендация |
|----------|-------------|
| Традиционное MVC веб-приложение, один домен | Сессии |
| REST API, микросервисы, мобильные клиенты | JWT |
| OAuth2 / OpenID Connect | JWT (стандарт) |
| Требование мгновенного отзыва | Сессии или JWT + denylist |

**В Spring Security** сессии настраиваются через `sessionManagement()`, JWT — через `oauth2ResourceServer().jwt()` или кастомный `OncePerRequestFilter`. При использовании JWT рекомендуется отключить CSRF (`csrf.disable()`), так как токен не отправляется браузером автоматически.
### Вопрос 7. Из каких трёх частей состоит JWT? Что содержит каждая часть и почему payload не является секретным?

**Категория:** Безопасность · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [JSON Web Token Introduction — jwt.io](https://jwt.io/introduction)

JWT (JSON Web Token) состоит из трёх частей, разделённых точками: `header.payload.signature`.

**Header (заголовок)** — содержит метаданные токена: тип токена (`typ: "JWT"`) и алгоритм подписи (`alg: "HS256"` или `"RS256"`). Заголовок кодируется в Base64Url.

**Payload (полезная нагрузка)** — содержит claims (утверждения): данные о пользователе и дополнительные метаданные. Стандартные claims включают `sub` (субъект), `iss` (издатель), `exp` (срок действия), `iat` (время выпуска). Payload также кодируется в Base64Url. Для подписанных токенов информация защищена от подделки, но читаема любым. Не следует помещать секретные данные в payload или header, если токен не зашифрован (JWE).

**Signature (подпись)** — создаётся как: `HMACSHA256(base64UrlEncode(header) + "." + base64UrlEncode(payload), SECRET_KEY)`. Подпись используется для верификации того, что отправитель токена — именно тот, кем он представляется, и что содержимое не было изменено в пути.

Payload не является секретным, потому что Base64Url — это лишь кодирование, а не шифрование. Любой, у кого есть токен, может декодировать и прочитать payload мгновенно. Безопасность JWT строится на подписи: нельзя подделать данные, не зная секретного ключа.

#### Ответ от Claude

JWT (JSON Web Token, RFC 7519) — это компактный, URL-безопасный способ передачи утверждений (claims) между двумя сторонами в виде JSON-объекта. Токен имеет вид:

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
```

**Часть 1 — Header (заголовок)**

JSON-объект, Base64Url-закодированный:
```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```
Содержит: алгоритм подписи (`alg`) и тип токена (`typ`). Алгоритм может быть симметричным (HMAC: HS256, HS512) или асимметричным (RSA: RS256, PS256; ECDSA: ES256).

**Часть 2 — Payload (полезная нагрузка)**

JSON-объект, Base64Url-закодированный. Содержит claims — утверждения о сущности (обычно пользователе) и дополнительные данные. Делятся на три типа:
- **Registered claims** (зарегистрированные, рекомендуемые): `iss` (issuer), `sub` (subject), `aud` (audience), `exp` (expiration time), `nbf` (not before), `iat` (issued at), `jti` (JWT ID).
- **Public claims** (публичные): определяются пользователями, должны быть зарегистрированы в IANA или содержать устойчивый к коллизиям URI.
- **Private claims** (приватные): произвольные данные, согласованные между сторонами, например: `"role": "admin"`, `"email": "user@example.com"`.

**Часть 3 — Signature (подпись)**

Вычисляется по формуле:
```
HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  secret
)
```
Для RSA это будет `RSASHA256(...)` с приватным ключом. Подпись позволяет верифицировать, что токен не был изменён после выпуска.

**Почему payload не является секретным**

Base64Url — это кодирование (encoding), а не шифрование (encryption). Любой человек, получивший токен, может декодировать header и payload без каких-либо ключей, просто применив base64url-декодирование. Это намеренное дизайнерское решение: JWT предназначен для передачи данных, а не для их сокрытия.

Безопасность JWT строится исключительно на **целостности**: злоумышленник может прочитать payload, но не может изменить его незаметно — изменение любого бита сделает подпись невалидной. Если нужна ещё и конфиденциальность payload, используется JWE (JSON Web Encryption) — зашифрованная форма токена.

**Практический вывод для собеседования:** никогда не помещайте пароли, секретные ключи или персональные данные (PII), которые не должны быть видны клиенту, в payload обычного (не JWE) токена.

Цель документа — просветительская: показать разработчикам, на какие угрозы обращать внимание в первую очередь.

**Подробное объяснение ключевых уязвимостей:**

**A01 — Broken Access Control (Нарушение контроля доступа)**

Самая распространённая уязвимость 2021 года. Происходит, когда приложение не проверяет, имеет ли пользователь право выполнять запрошенное действие. Примеры: пользователь меняет `id=123` на `id=124` в URL и получает чужие данные (IDOR — Insecure Direct Object Reference); обычный пользователь получает доступ к административным функциям; неправильно настроенные CORS-заголовки позволяют несанкционированные cross-origin запросы.

**A02 — Cryptographic Failures (Криптографические сбои)**

Ранее называлась «Sensitive Data Exposure». Охватывает ситуации, когда данные не защищены должным образом криптографически: передача паролей открытым текстом по HTTP; хранение паролей в MD5 или SHA1 без соли; использование устаревших алгоритмов (DES, RC4); жёстко прописанные ключи в коде; отсутствие TLS для передачи чувствительных данных.

**A03 — Injection (Инъекция)**

Происходит, когда непроверенные данные от пользователя передаются интерпретатору (SQL, OS, LDAP, XML). Наиболее известный вид — SQL-инъекция: `SELECT * FROM users WHERE name = '' OR '1'='1'`. В версии 2021 года сюда включён XSS (Cross-Site Scripting) — внедрение вредоносного JavaScript-кода в страницу, который выполняется в браузере другого пользователя.

**A05 — Security Misconfiguration (Неправильная конфигурация безопасности)**

Очень широкая категория: включённый `DEBUG=True` в продакшене; дефолтные логины/пароли (admin/admin); открытые облачные S3-бакеты; ненужные открытые порты; подробные stack trace в ответах об ошибках; незакрытые заголовки безопасности (CSP, HSTS).

**A07 — Identification and Authentication Failures (Сбои аутентификации)**

Уязвимости в механизмах проверки подлинности: возможность брутфорса паролей без ограничения попыток; слабые или предсказуемые пароли по умолчанию; небезопасное восстановление пароля; хранение сессионных токенов в URL; отсутствие инвалидации сессии при logout; отсутствие многофакторной аутентификации.

### Вопрос 9. Как работает интерфейс UserDetailsService в Spring Security? Что возвращает метод loadUserByUsername и где обычно хранятся данные пользователей?

**Категория:** Безопасность · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Spring Security: Database-backed UserDetailsService — Baeldung](https://www.baeldung.com/spring-security-authentication-with-a-database)

`UserDetailsService` — это интерфейс Spring Security для получения данных пользователя. Он имеет единственный метод `loadUserByUsername(String username)`, который можно переопределить для кастомизации поиска пользователя. Метод используется `DaoAuthenticationProvider` для загрузки информации о пользователе в процессе аутентификации.

Если пользователь найден, метод возвращает объект `UserDetails` (например, `MyUserPrincipal`), который содержит имя пользователя, пароль, роли и флаги статуса аккаунта. Если пользователь не найден — бросается `UsernameNotFoundException`.

Для активации кастомного `UserDetailsService` достаточно аннотировать реализацию `@Service` — Spring автоматически обнаружит её при сканировании компонентов и включит в процесс аутентификации.

Данные пользователей обычно хранятся в реляционной базе данных и загружаются через Spring Data JPA или `JdbcTemplate`.

#### Ответ от Claude

**UserDetailsService** — центральный интерфейс Spring Security для загрузки данных пользователя по имени. Определён в пакете `org.springframework.security.core.userdetails`.

**Интерфейс содержит один метод:**

```java
public interface UserDetailsService {
    UserDetails loadUserByUsername(String username)
        throws UsernameNotFoundException;
}
```

**Что возвращает loadUserByUsername**

Метод возвращает объект типа `UserDetails` — ещё один интерфейс Spring Security, описывающий пользователя с точки зрения системы безопасности. `UserDetails` содержит:

- `getUsername()` — имя пользователя (логин)
- `getPassword()` — хэшированный пароль (Spring Security само сравнит его с введённым через `PasswordEncoder`)
- `getAuthorities()` — коллекция `GrantedAuthority`, то есть набор ролей и разрешений (например: `ROLE_USER`, `ROLE_ADMIN`)
- `isAccountNonExpired()`, `isAccountNonLocked()`, `isCredentialsNonExpired()`, `isEnabled()` — флаги состояния аккаунта

Метод **никогда не должен возвращать `null`** — если пользователь не найден, нужно бросить `UsernameNotFoundException`.

**Как это работает в цепочке аутентификации**

1. Пользователь отправляет логин и пароль (например, через форму или HTTP Basic).
2. `UsernamePasswordAuthenticationFilter` перехватывает запрос и создаёт объект `UsernamePasswordAuthenticationToken`.
3. `AuthenticationManager` (обычно `ProviderManager`) делегирует обработку `DaoAuthenticationProvider`.
4. `DaoAuthenticationProvider` вызывает `UserDetailsService.loadUserByUsername(username)`.
5. Полученный `UserDetails` передаётся в `PasswordEncoder.matches(rawPassword, encodedPassword)`.
6. Если всё совпало — создаётся аутентифицированный `Authentication` объект и помещается в `SecurityContextHolder`.

**Пример реализации:**

```java
@Service
public class CustomUserDetailsService implements UserDetailsService {

    private final UserRepository userRepository;

    public CustomUserDetailsService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    @Override
    public UserDetails loadUserByUsername(String username)
            throws UsernameNotFoundException {
        User user = userRepository.findByEmail(username)
            .orElseThrow(() ->
                new UsernameNotFoundException("User not found: " + username));

        return org.springframework.security.core.userdetails.User
            .withUsername(user.getEmail())
            .password(user.getPassword())       // уже хэшированный пароль
            .roles(user.getRole().name())
            .accountExpired(!user.isActive())
            .build();
    }
}
```

**Где обычно хранятся данные пользователей**

Spring Security не навязывает конкретное хранилище. На практике встречаются:

- **Реляционная БД** (PostgreSQL, MySQL) + Spring Data JPA — самый распространённый вариант в продакшн-приложениях. Реализуется через `JpaRepository<User, Long>` и кастомный `UserDetailsService`.
- **In-memory** (`InMemoryUserDetailsManager`) — только для тестирования и прототипирования, не используется в продакшне.
- **JDBC** (`JdbcUserDetailsManager`) — встроенная реализация, работающая напрямую с БД через `JdbcTemplate`. Ожидает определённую схему таблиц.
- **LDAP / Active Directory** (`LdapUserDetailsService`) — для корпоративных систем с централизованным каталогом пользователей.
- **OAuth2 / OIDC** — когда пользователи управляются внешним провайдером (Keycloak, Okta, Google). В этом случае `UserDetailsService` может не использоваться вовсе.

**Важный нюанс:** если в контексте Spring есть ровно один бин, реализующий `UserDetailsService`, Spring Security автоматически подхватывает его без дополнительной конфигурации (начиная со Spring Boot 2.x).
### Вопрос 10. Что такое SecurityFilterChain в Spring Security? Перечисли несколько ключевых фильтров и порядок их выполнения.

**Категория:** Безопасность · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Architecture :: Spring Security](https://docs.spring.io/spring-security/reference/servlet/architecture.html)

`SecurityFilterChain` — это компонент Spring Security, используемый `FilterChainProxy` для определения того, какие фильтры безопасности должны выполняться для конкретного HTTP-запроса. `FilterChainProxy` — специальный `Filter`, предоставляемый Spring Security, который маршрутизирует входящие запросы к нужным `SecurityFilterChain`-экземплярам на основе паттернов URL или других условий запроса (`HttpServletRequest`). Если зарегистрировано несколько цепочек, срабатывает только первая совпавшая.

Полный порядок фильтров по умолчанию (из debug-вывода документации):

1. `DisableEncodeUrlFilter`
2. `WebAsyncManagerIntegrationFilter`
3. `SecurityContextHolderFilter`
4. `HeaderWriterFilter`
5. `CsrfFilter`
6. `LogoutFilter`
7. `UsernamePasswordAuthenticationFilter`
8. `DefaultLoginPageGeneratingFilter`
9. `DefaultLogoutPageGeneratingFilter`
10. `BasicAuthenticationFilter`
11. `RequestCacheAwareFilter`
12. `SecurityContextHolderAwareRequestFilter`
13. `AnonymousAuthenticationFilter`
14. `ExceptionTranslationFilter`
15. `AuthorizationFilter`

#### Ответ от Claude

**SecurityFilterChain** — это интерфейс Spring Security, представляющий упорядоченную цепочку `javax.servlet.Filter`-объектов, применяемых к HTTP-запросам. Он является центральной точкой конфигурации безопасности в современном Spring Security (начиная с версии 5.4+, где `WebSecurityConfigurerAdapter` признан устаревшим).

**Архитектура:**

- `DelegatingFilterProxy` — стандартный сервлет-фильтр, зарегистрированный в контейнере; делегирует работу Spring-бину `springSecurityFilterChain`.
- `FilterChainProxy` — сам является этим Spring-бином; содержит список `SecurityFilterChain`-экземпляров и при обработке запроса выбирает первый, чей `RequestMatcher` совпадает с запросом.
- `SecurityFilterChain` — хранит упорядоченный список фильтров безопасности для конкретного паттерна URL.

**Типичный порядок ключевых фильтров:**

| Позиция | Фильтр | Роль |
|---------|--------|------|
| 1 | `DisableEncodeUrlFilter` | Запрещает кодирование сессионного ID в URL (защита от Session Fixation через URL) |
| 3 | `SecurityContextHolderFilter` | Загружает `SecurityContext` из хранилища (сессия / токен) и очищает его по завершении |
| 5 | `CsrfFilter` | Проверяет CSRF-токен для небезопасных методов (POST, PUT, DELETE) |
| 6 | `LogoutFilter` | Обрабатывает запросы на выход, инвалидирует сессию |
| 7 | `UsernamePasswordAuthenticationFilter` | Аутентификация по логину/паролю через форму |
| 10 | `BasicAuthenticationFilter` | Аутентификация через HTTP Basic (заголовок `Authorization: Basic ...`) |
| 13 | `AnonymousAuthenticationFilter` | Если аутентификация ещё не установлена — добавляет анонимный `Authentication` |
| 14 | `ExceptionTranslationFilter` | Перехватывает `AuthenticationException` и `AccessDeniedException`; запускает `AuthenticationEntryPoint` или `AccessDeniedHandler` |
| 15 | `AuthorizationFilter` | Финальная проверка прав доступа к ресурсу |

**Конфигурация через `HttpSecurity`:**

```java
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
        .csrf(Customizer.withDefaults())
        .httpBasic(Customizer.withDefaults())
        .formLogin(Customizer.withDefaults())
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/admin/**").hasRole("ADMIN")
            .anyRequest().authenticated()
        );
    return http.build();
}
```

**Важные нюансы:**
- При нескольких `SecurityFilterChain`-бинах используется `@Order` для расстановки приоритетов; первая совпавшая цепочка выполняется, остальные игнорируются.
- Добавление собственного фильтра: `http.addFilterBefore(myFilter, UsernamePasswordAuthenticationFilter.class)`.
- `ExceptionTranslationFilter` обязательно должен стоять перед `AuthorizationFilter`, иначе исключения безопасности не будут перехвачены корректно.

### Вопрос 11. Как происходит подпись JWT? Чем отличаются алгоритмы HS256 и RS256, и в каком случае предпочтительнее использовать асимметричную подпись?

**Категория:** Безопасность · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [JWT.IO — JSON Web Tokens Introduction](https://jwt.io/introduction)

JWT состоит из трёх частей, разделённых точками: заголовка (`header`), полезной нагрузки (`payload`) и подписи (`signature`). Заголовок и payload кодируются в Base64URL, затем конкатенируются через точку, после чего вычисляется подпись. Для HS256 формула подписи: `HMACSHA256(base64UrlEncode(header) + "." + base64UrlEncode(payload), secret)`. Подпись позволяет убедиться, что содержимое токена не было изменено в пути. В случае токенов, подписанных приватным ключом, она также подтверждает личность отправителя. Важно: содержимое JWT не зашифровано, а только закодировано — любой может его прочитать.

#### Ответ от Claude

**Структура JWT**

JWT (JSON Web Token, RFC 7519) состоит из трёх частей в формате `header.payload.signature`:

- **Header** — JSON с типом токена (`"typ": "JWT"`) и алгоритмом подписи (`"alg": "HS256"` или `"alg": "RS256"`), закодированный в Base64URL.
- **Payload** — JSON с claims (утверждениями): стандартными (`sub`, `iss`, `exp`, `iat`) и пользовательскими, закодированный в Base64URL.
- **Signature** — криптографическая подпись конкатенации `base64url(header) + "." + base64url(payload)`.

**Процесс подписи**

```
1. JSON(header)  → Base64URL → H
2. JSON(payload) → Base64URL → P
3. Signature = SIGN(H + "." + P, key)  // алгоритм зависит от alg
4. JWT = H + "." + P + "." + Base64URL(Signature)
```

**HS256 vs RS256**

| Характеристика | HS256 | RS256 |
|---------------|-------|-------|
| Тип алгоритма | Симметричный (HMAC-SHA256) | Асимметричный (RSA-SHA256) |
| Ключ для подписи | Один общий секрет | Приватный ключ издателя |
| Ключ для верификации | Тот же общий секрет | Публичный ключ (доступен всем) |
| Размер ключа | Любой, рекомендуется ≥256 бит | Рекомендуется 2048+ бит RSA |
| Компрометация | Необходимо перегенерировать секрет и перевыпустить токены | Ротация ключей без перезапуска сервисов (через JWKS) |
| Производительность | Быстрее | Медленнее (RSA-операции) |

**Когда предпочтительна асимметричная подпись (RS256)?**

1. **Несколько сервисов-верификаторов**: публичный ключ можно распространить свободно — любой сервис верифицирует токен, не зная приватного ключа и не имея возможности сам создавать токены. При HS256 каждый сервис-верификатор должен иметь копию секрета, что увеличивает поверхность атаки.
2. **Разделение ответственности**: только Identity Provider (IDP), владеющий приватным ключом, может выпускать токены; остальные сервисы — только проверять.
3. **Публичное раскрытие ключей через JWKS**: Identity Provider публикует публичные ключи по стандартному эндпоинту (`/.well-known/jwks.json`), и клиенты могут получить их автоматически.
4. **Ротация ключей без downtime**: при компрометации приватного ключа публикуется новый — сервисы подтягивают обновлённые JWKS без перезапуска.
5. **Федерация и OpenID Connect**: стандарт OIDC требует использования асимметричных алгоритмов.

**Когда допустим HS256?** При наличии единственного сервиса, который и выпускает, и верифицирует токены (монолит или закрытая доверенная среда).

### Вопрос 12. Что такое OAuth 2.0? Опиши Authorization Code Flow: какие участники задействованы и какие токены выдаются в результате?

**Категория:** Безопасность · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [RFC 6749 — The OAuth 2.0 Authorization Framework](https://datatracker.ietf.org/doc/html/rfc6749)

OAuth 2.0 — это фреймворк авторизации, описанный в RFC 6749. Он позволяет сторонним приложениям (клиентам) получать ограниченный доступ к HTTP-сервисам от имени владельца ресурса путём организации процесса согласия (approval interaction) между владельцем ресурса и HTTP-сервисом, либо путём предоставления самому приложению возможности действовать от своего имени.

Authorization Code Flow включает четыре роли:
- **Resource Owner** — конечный пользователь, владеющий защищёнными ресурсами.
- **Client** — приложение, запрашивающее доступ.
- **Authorization Server** — сервер, аутентифицирующий владельца ресурса и выдающий токены.
- **Resource Server** — сервер с защищёнными ресурсами.

Последовательность шагов по RFC 6749 (раздел 4.1):
- (A) Клиент перенаправляет пользователя на Authorization Server с `response_type=code`, `client_id`, `redirect_uri`, `scope`, `state`.
- (B) Authorization Server аутентифицирует пользователя и запрашивает его согласие.
- (C) После согласия — перенаправляет обратно на `redirect_uri` с параметром `code` (authorization code) и `state`.
- (D) Клиент обменивает `code` на токены, аутентифицируя себя через `client_id` + `client_secret`.
- (E) Authorization Server возвращает Access Token (и опционально Refresh Token).
- (F) Клиент использует Access Token для обращения к Resource Server.

#### Ответ от Claude

**OAuth 2.0 — что это такое**

OAuth 2.0 (RFC 6749, 2012) — промышленный стандарт протокола авторизации. Его ключевая идея: владелец ресурса (пользователь) делегирует клиентскому приложению ограниченный доступ к своим ресурсам без передачи логина и пароля этому приложению. OAuth 2.0 — это протокол авторизации (не аутентификации); аутентификация строится поверх него через OpenID Connect.

**Участники Authorization Code Flow**

| Роль | Описание | Пример |
|------|----------|--------|
| Resource Owner | Пользователь, владеющий ресурсами | Конечный пользователь |
| Client | Приложение, запрашивающее доступ | Веб-приложение, мобильное приложение |
| Authorization Server | Выпускает токены после аутентификации и получения согласия | Keycloak, Okta, Google Auth |
| Resource Server | Хостит защищённые ресурсы, принимает Access Token | REST API |

**Пошаговый Authorization Code Flow**

```
[1] Client → Authorization Server
    GET /authorize?response_type=code&client_id=...
                   &redirect_uri=...&scope=...&state=...

[2] Authorization Server → Resource Owner
    Показывает форму входа и экран согласия (consent)

[3] Resource Owner → Authorization Server
    Вводит credentials, даёт согласие

[4] Authorization Server → Client (redirect)
    302 Location: https://client/callback?code=AUTH_CODE&state=...

[5] Client → Authorization Server (back-channel, серверный запрос)
    POST /token
    grant_type=authorization_code&code=AUTH_CODE
    &redirect_uri=...&client_id=...&client_secret=...

[6] Authorization Server → Client
    {
      "access_token": "...",
      "token_type": "Bearer",
      "expires_in": 3600,
      "refresh_token": "..."   // если запрошен scope offline_access
    }

[7] Client → Resource Server
    GET /api/resource
    Authorization: Bearer ACCESS_TOKEN
```

**Выдаваемые токены**

- **Authorization Code** — короткоживущий одноразовый код (обычно 10–60 секунд), передаётся через браузер пользователя (front-channel). Сам по себе не даёт доступа — нужен для обмена на токены по защищённому back-channel.
- **Access Token** — Bearer-токен для доступа к Resource Server. Обычно JWT, короткоживущий (минуты–часы). Resource Server проверяет его без обращения к Authorization Server (stateless).
- **Refresh Token** — долгоживущий токен (дни–недели) для получения нового Access Token без повторной аутентификации пользователя. Передаётся только между клиентом и Authorization Server, никогда не отправляется на Resource Server.
- **ID Token** (OpenID Connect) — JWT с данными об аутентификации пользователя (sub, name, email и др.); выдаётся при наличии scope `openid`. Это расширение поверх OAuth 2.0, не часть базового RFC 6749.

**Почему используется промежуточный authorization code?**

Код передаётся через браузер (front-channel), что само по себе менее безопасно. Но код бесполезен без `client_secret`, который хранится на сервере и никогда не проходит через браузер. Это исключает перехват токенов из URL или истории браузера.

**PKCE (RFC 7636)** — обязательное расширение для публичных клиентов (SPA, мобильные приложения), у которых нет возможности безопасно хранить `client_secret`. Клиент генерирует `code_verifier` и отправляет его хэш (`code_challenge`) при запросе кода, а сам `code_verifier` — при обмене на токены. OAuth 2.1 делает PKCE обязательным для всех клиентов.
### Вопрос 13. В чём разница между OAuth 2.0 и OpenID Connect? Что такое ID-токен и какие стандартные claims он содержит?

**Категория:** Безопасность · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ID Tokens vs Access Tokens — oauth.net](https://oauth.net/id-tokens-vs-access-tokens/)

OAuth 2.0 — это фреймворк делегированного доступа: он позволяет клиентскому приложению получить ограниченный доступ к HTTP-ресурсам от имени пользователя, но не определяет стандартных механизмов для передачи информации об идентичности пользователя. OpenID Connect (OIDC) — это тонкий слой поверх OAuth 2.0, добавляющий аутентификацию: он позволяет клиенту верифицировать личность пользователя и получить базовую информацию о нём.

Ключевое различие: access token (OAuth 2.0) предназначен для Resource Server — сервера ресурсов, который его проверяет и предоставляет доступ. ID token (OIDC) предназначен непосредственно для клиентского приложения (Relying Party) и несёт информацию о сессии аутентификации. Access tokens могут быть как JWT, так и непрозрачными строками; ID tokens всегда являются JWT.

ID token содержит минимально необходимые claims:
- `sub` — идентификатор пользователя (subject), уникальный в рамках issuer-а
- `iss` — идентификатор эмитента (Authorization Server / OpenID Provider)
- `aud` — аудитория; должна содержать `client_id` Relying Party
- `exp` — время истечения токена (Unix timestamp)
- `iat` — время выдачи токена (Unix timestamp)

#### Ответ от Claude

**OAuth 2.0 vs OpenID Connect: суть различия**

OAuth 2.0 (RFC 6749) — это протокол *авторизации* (делегированного доступа). Он отвечает на вопрос: «Разрешает ли владелец ресурса клиенту получить доступ к этому ресурсу?». OAuth 2.0 выдаёт access token, который клиент предъявляет Resource Server. Протокол намеренно не определяет, как Resource Server проверяет токен и что означает «пользователь вошёл в систему» — это за рамками спецификации.

OpenID Connect 1.0 (спецификация openid.net) — это протокол *аутентификации*, построенный поверх OAuth 2.0. Он отвечает на вопрос: «Кто этот пользователь и подтверждён ли факт его входа?». OIDC добавляет:
1. **ID token** — JWT, передаваемый клиенту вместе с access token.
2. **UserInfo endpoint** — защищённый OAuth-токеном эндпоинт для получения дополнительных claims о пользователе.
3. **Scope `openid`** — обязательный scope, сигнализирующий о том, что запрашивается аутентификация.

**Что такое ID token**

ID token — это подписанный JWT (и опционально зашифрованный), содержащий утверждения (claims) о результате аутентификации пользователя. Он направлен клиенту (Relying Party), а не Resource Server. Клиент должен верифицировать подпись и проверить claims перед доверием токену.

**Стандартные claims ID token (OpenID Connect Core 1.0, секция 2)**

| Claim | Обязательность | Описание |
|---|---|---|
| `iss` | REQUIRED | URI идентификатора OpenID Provider (эмитента) |
| `sub` | REQUIRED | Идентификатор пользователя; уникален в рамках `iss`; не более 255 ASCII символов |
| `aud` | REQUIRED | Аудитория; должна включать `client_id` клиента; может быть строкой или массивом |
| `exp` | REQUIRED | Время истечения (Unix timestamp); после него токен нельзя принимать |
| `iat` | REQUIRED | Время выдачи токена (Unix timestamp) |
| `auth_time` | OPTIONAL / REQUIRED при `max_age` | Время, когда пользователь был аутентифицирован |
| `nonce` | OPTIONAL / REQUIRED при наличии в запросе | Случайная строка из запроса; защищает от replay-атак |
| `acr` | OPTIONAL | Authentication Context Class Reference (уровень аутентификации, напр. `urn:mace:incommon:iap:silver`) |
| `amr` | OPTIONAL | Authentication Methods References (методы аутентификации: `pwd`, `otp`, `mfa` и т.д.) |
| `azp` | OPTIONAL | Authorized party: `client_id` стороны, которой выдан токен, если `aud` содержит несколько значений |
| `at_hash` | OPTIONAL | Хэш access token; позволяет клиенту убедиться, что access token соответствует ID token |

Дополнительные claims о пользователе (имя, email, адрес и т.п.) определены в секции 5 спецификации и могут присутствовать в ID token или возвращаться через UserInfo endpoint.

**Ключевой практический вывод:**
- Используй **access token** для авторизации запросов к API/Resource Server.
- Используй **ID token** для установления сессии пользователя в клиентском приложении.
- Никогда не передавай ID token на Resource Server как средство авторизации — он для этого не предназначен.

### Вопрос 14. Как настроить защиту от SQL-инъекций при использовании Spring Data JPA и нативных запросов? Какие риски несут @Query-аннотации с конкатенацией строк?

**Категория:** Безопасность · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Spring Data JPA @Query — Baeldung](https://www.baeldung.com/spring-data-jpa-query)

JPA и другие ORM-фреймворки абстрагируют работу с SQL, но не защищают автоматически от инъекций, если разработчик строит запросы небезопасно. Конкатенация строк при построении JPQL или нативного SQL — основной источник уязвимостей.

Правильный подход — именованные параметры через аннотацию `@Param`:

```java
// БЕЗОПАСНО: именованный параметр
@Query("SELECT u FROM User u WHERE u.firstname = :firstname OR u.lastname = :lastname")
User findByLastnameOrFirstname(@Param("lastname") String lastname,
                               @Param("firstname") String firstname);

// БЕЗОПАСНО: нативный запрос с именованным параметром
@Query(value = "SELECT * FROM users WHERE email = :email", nativeQuery = true)
User findByEmail(@Param("email") String email);
```

Позиционные параметры (`?1`, `?2`) также безопасны, но именованные параметры предпочтительнее: они нагляднее и не зависят от порядка аргументов.

#### Ответ от Claude

**Механизм защиты параметризованных запросов**

Spring Data JPA транслирует JPQL через Hibernate в нативный SQL с использованием PreparedStatement JDBC. При именованном (`:name`) или позиционном (`?1`) связывании параметров драйвер JDBC передаёт значение как отдельный параметр — БД обрабатывает его как данные, а не как часть SQL-синтаксиса. Это исключает SQL-инъекцию на уровне протокола.

**Безопасные паттерны для @Query**

```java
// 1. JPQL с именованными параметрами (рекомендуется)
@Query("SELECT u FROM User u WHERE u.status = :status AND u.age > :minAge")
List<User> findActive(@Param("status") String status, @Param("minAge") int minAge);

// 2. Нативный SQL с именованными параметрами
@Query(value = "SELECT * FROM users WHERE role = :role", nativeQuery = true)
List<User> findByRole(@Param("role") String role);

// 3. Позиционные параметры (безопасно, но менее читаемо)
@Query("SELECT u FROM User u WHERE u.email = ?1")
Optional<User> findByEmail(String email);

// 4. Derived query methods (автоматически безопасны — Spring генерирует параметризованный запрос)
List<User> findByStatusAndAgeGreaterThan(String status, int age);
```

**Риски @Query с конкатенацией строк**

Любая конкатенация пользовательского ввода в строку запроса делает приложение уязвимым:

```java
// ОПАСНО: JPQL-инъекция
@Query("SELECT u FROM User u WHERE u.name = '" + name + "'") // нельзя так писать!

// ОПАСНО: нативный запрос с конкатенацией
@Query(value = "SELECT * FROM users WHERE name = '" + name + "'", nativeQuery = true)

// ОПАСНО: динамическое построение в сервисе
String jpql = "SELECT u FROM User u WHERE u." + fieldName + " = :value";
entityManager.createQuery(jpql).setParameter("value", value).getResultList();
```

Вектор атаки во втором примере: если `fieldName` контролируется пользователем, атакующий может передать `name = '1' OR '1'='1' --` или манипулировать структурой запроса.

**Правила для нативных запросов (nativeQuery = true)**

Нативные запросы обходят JPQL-парсер Hibernate и уходят напрямую в JDBC — это делает их потенциально более опасными:
- Никогда не конкатенируй имена таблиц, колонок или ORDER BY из пользовательского ввода.
- Если нужна динамическая сортировка, используй whitelist допустимых значений с явной проверкой в коде.
- При необходимости сложных динамических запросов используй **JPA Criteria API** или **QueryDSL** — они строят параметризованные запросы программно.

**SpEL в @Query и риски**

Spring Data поддерживает SpEL в `@Query` (`#{#entityName}`, `?#{...}`). Использование `?#{[0]}` с пользовательским вводом в SpEL-выражении может привести к JPQL-инъекции:

```java
// РИСК: пользовательский ввод в SpEL
@Query("SELECT u FROM User u WHERE u.name LIKE ?#{[0]}")
// Если [0] содержит JPQL-метасимволы, возможна инъекция
```

Предпочитай обычное именованное связывание `:param` вместо SpEL с пользовательскими данными.

**Дополнительные меры защиты**

- Принцип минимальных привилегий: БД-пользователь приложения должен иметь только необходимые права (SELECT/INSERT/UPDATE/DELETE), без DDL-привилегий.
- Используй Hibernate Validator для валидации входных данных до уровня репозитория.
- Логируй и мониторируй нетипичные запросы (Hibernate SQL logging: `spring.jpa.show-sql=true` в dev-окружении).

### Вопрос 15. Что такое хранимые процедуры с точки зрения безопасности — защищают ли они от SQL-инъекций автоматически? При каком условии уязвимость всё равно возникает?

**Категория:** Безопасность · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [SQL Injection Prevention Cheat Sheet — OWASP Cheat Sheet Series](https://cheatsheetseries.owasp.org/cheatsheets/SQL_Injection_Prevention_Cheat_Sheet.html)

OWASP выделяет хранимые процедуры (Stored Procedures) как одну из четырёх основных защит от SQL-инъекций наравне с параметризованными запросами (Prepared Statements), валидацией через allowlist и экранированием.

Хранимая процедура считается «безопасно реализованной», если она не содержит небезопасной генерации динамического SQL. В этом случае вызов процедуры работает как параметризованный запрос: параметры передаются отдельно от SQL-кода и интерпретируются как данные, а не как синтаксис.

Однако хранимые процедуры **не защищают автоматически**: уязвимость возникает, если внутри хранимой процедуры на PL/SQL (Oracle) или T-SQL (SQL Server) происходит конкатенация строк и последующее выполнение через `EXECUTE IMMEDIATE` (Oracle) или `EXEC`/`sp_executesql` с конкатенацией (SQL Server). В таком случае атакующий может внедрить произвольный SQL через параметры процедуры.

OWASP рекомендует: если генерации динамического SQL внутри процедур избежать нельзя, необходимо использовать bind-переменные даже внутри динамического SQL, либо применять строгую валидацию по allowlist.

#### Ответ от Claude

**Что такое хранимая процедура с точки зрения безопасности**

Хранимая процедура (stored procedure) — это именованный блок SQL-кода, заранее скомпилированный и сохранённый в БД. С точки зрения безопасности её ключевое свойство: при вызове с параметрами (`CALL get_user(?)`) структура SQL-запроса фиксирована в момент компиляции, а параметры подставляются как данные. Это аналогично PreparedStatement в JDBC — SQL-синтаксис и данные разделены.

**Когда хранимые процедуры защищают**

Хранимая процедура безопасна, если:
1. Тело процедуры содержит статический SQL с параметрами-плейсхолдерами.
2. Параметры используются только как значения, но не как части идентификаторов (имён таблиц, колонок) или ключевых слов.

Пример безопасной процедуры (T-SQL):
```sql
CREATE PROCEDURE GetUserById
    @UserId INT
AS
BEGIN
    SELECT id, name, email FROM Users WHERE id = @UserId;
END
```

Здесь `@UserId` интерпретируется СУБД как целочисленное значение — SQL-синтаксис изменить невозможно.

**Когда уязвимость всё равно возникает**

Уязвимость возникает при **генерации динамического SQL внутри хранимой процедуры** с конкатенацией пользовательского ввода и последующим выполнением через:
- `EXECUTE IMMEDIATE` (Oracle PL/SQL)
- `EXEC` или `sp_executesql` с конкатенацией строк (SQL Server T-SQL)
- `PREPARE` / `EXECUTE` (MySQL, PostgreSQL)

Пример уязвимой хранимой процедуры (T-SQL):
```sql
CREATE PROCEDURE SearchUsers
    @OrderBy NVARCHAR(50)
AS
BEGIN
    -- ОПАСНО: конкатенация без валидации
    DECLARE @sql NVARCHAR(500);
    SET @sql = N'SELECT id, name FROM Users ORDER BY ' + @OrderBy;
    EXEC(@sql);
END
```

Атакующий передаёт `@OrderBy = 'name; DROP TABLE Users--'` — и получает выполнение произвольного SQL.

Аналогичный пример в PL/SQL (Oracle):
```sql
CREATE OR REPLACE PROCEDURE get_user(p_name IN VARCHAR2) IS
    v_sql VARCHAR2(200);
BEGIN
    v_sql := 'SELECT * FROM users WHERE name = ''' || p_name || '''';
    EXECUTE IMMEDIATE v_sql; -- УЯЗВИМОСТЬ: конкатенация
END;
```

**Правильное использование динамического SQL внутри процедур**

Если динамического SQL не избежать, нужно использовать bind-переменные и в нём:

```sql
-- БЕЗОПАСНО: sp_executesql с параметрами (T-SQL)
DECLARE @sql NVARCHAR(500);
SET @sql = N'SELECT id, name FROM Users WHERE status = @status';
EXEC sp_executesql @sql, N'@status NVARCHAR(50)', @status = @InputStatus;

-- БЕЗОПАСНО: EXECUTE IMMEDIATE с USING (Oracle)
v_sql := 'SELECT * FROM users WHERE name = :name';
EXECUTE IMMEDIATE v_sql USING p_name;
```

**Динамические идентификаторы (имена таблиц, колонок)**

Bind-переменные работают только для значений данных, но не для идентификаторов (имён таблиц, колонок, операторов ORDER BY). Для динамических идентификаторов единственный безопасный подход — **allowlist (белый список)**:

```java
// Java-сторона: allowlist для ORDER BY
private static final Set<String> ALLOWED_COLUMNS = Set.of("name", "email", "created_at");

public List<User> findSorted(String column) {
    if (!ALLOWED_COLUMNS.contains(column)) {
        throw new IllegalArgumentException("Invalid column: " + column);
    }
    // Теперь безопасно подставить column в запрос
}
```

**Итог**

| Ситуация | Защищена от SQL-инъекции? |
|---|---|
| Хранимая процедура со статическим SQL и параметрами | Да |
| Хранимая процедура + EXEC(строка с конкатенацией) | Нет |
| Хранимая процедура + sp_executesql с bind-переменными | Да |
| Хранимая процедура + EXECUTE IMMEDIATE с USING | Да |
| Хранимая процедура + динамический идентификатор без allowlist | Нет |

Главный принцип: **хранимые процедуры сами по себе не дают защиты** — защищает параметризация. Если внутри процедуры строится динамический SQL с конкатенацией, уязвимость присутствует независимо от использования хранимой процедуры.

### Вопрос 16. Как реализовать stateless-аутентификацию через JWT в Spring Security? Какой фильтр нужно добавить в цепочку и на каком этапе устанавливается SecurityContext?

**Категория:** Безопасность · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Servlet Authentication Architecture :: Spring Security](https://docs.spring.io/spring-security/reference/servlet/authentication/architecture.html)

`SecurityContextHolder` — центральное хранилище аутентификационных данных в Spring Security. Он содержит `SecurityContext` с текущим объектом `Authentication`. По умолчанию использует `ThreadLocal`, поэтому контекст доступен любому методу в том же потоке без явной передачи. После завершения запроса `FilterChainProxy` автоматически очищает контекст.

Для stateless-сценариев используется `NullSecurityContextRepository`: он ничего не сохраняет и не загружает, то есть контекст не связывается с `HttpSession`. Каждый запрос должен самостоятельно восстанавливать аутентификацию из токена.

Типичный пример установки контекста вручную:
```java
SecurityContext context = SecurityContextHolder.createEmptyContext();
Authentication authentication = new UsernamePasswordAuthenticationToken(
    username, null, authorities);
context.setAuthentication(authentication);
SecurityContextHolder.setContext(context);
```

Конфигурация stateless-режима:
```java
http.securityContext(sc -> sc
    .securityContextRepository(new NullSecurityContextRepository()));
```

#### Ответ от Claude

**Архитектура цепочки фильтров**

Spring Security реализует аутентификацию через цепочку сервлет-фильтров, управляемую `FilterChainProxy`. Каждый запрос последовательно проходит через все зарегистрированные фильтры. Для JWT-аутентификации создаётся кастомный фильтр, который вставляется **перед** `UsernamePasswordAuthenticationFilter` с помощью `addFilterBefore`.

**Почему OncePerRequestFilter**

Кастомный JWT-фильтр должен расширять `OncePerRequestFilter` — Spring гарантирует, что такой фильтр выполняется ровно один раз на запрос, даже при повторных диспетчеризациях (forward, include, error dispatch).

**Реализация фильтра**

```java
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    private final JwtTokenProvider jwtTokenProvider;

    public JwtAuthenticationFilter(JwtTokenProvider jwtTokenProvider) {
        this.jwtTokenProvider = jwtTokenProvider;
    }

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain filterChain)
            throws ServletException, IOException {
        String bearerToken = request.getHeader("Authorization");
        if (bearerToken != null && bearerToken.startsWith("Bearer ")) {
            String jwt = bearerToken.substring(7);
            if (jwtTokenProvider.validateToken(jwt)) {
                String username = jwtTokenProvider.getUsernameFromToken(jwt);
                List<GrantedAuthority> authorities =
                    jwtTokenProvider.getAuthoritiesFromToken(jwt);

                UsernamePasswordAuthenticationToken authentication =
                    new UsernamePasswordAuthenticationToken(
                        username, null, authorities);

                // Устанавливаем Authentication в SecurityContextHolder
                SecurityContext context = SecurityContextHolder.createEmptyContext();
                context.setAuthentication(authentication);
                SecurityContextHolder.setContext(context);
            }
        }
        filterChain.doFilter(request, response); // передаём управление дальше
    }
}
```

**Конфигурация Security**

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http,
                                           JwtAuthenticationFilter jwtFilter)
            throws Exception {
        http
            .csrf(AbstractHttpConfigurer::disable)
            .sessionManagement(sm -> sm
                .sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/auth/**").permitAll()
                .anyRequest().authenticated())
            // JWT-фильтр встаёт ДО стандартного фильтра форм-логина
            .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class);

        return http.build();
    }
}
```

**Этапы установки SecurityContext при stateless JWT**

1. Запрос поступает в `FilterChainProxy`.
2. `SecurityContextHolderFilter` (или `SecurityContextPersistenceFilter`) пытается загрузить контекст из `SecurityContextRepository`. При `SessionCreationPolicy.STATELESS` под капотом используется `NullSecurityContextRepository` — он возвращает пустой контекст.
3. **JWT-фильтр (`OncePerRequestFilter`)** разбирает заголовок `Authorization: Bearer <token>`, валидирует токен, извлекает `username` и права, создаёт `UsernamePasswordAuthenticationToken` и помещает его в `SecurityContextHolder`.
4. Downstream-фильтры (в частности, `AuthorizationFilter`) читают `SecurityContext` и принимают решение об авторизации.
5. После завершения запроса `FilterChainProxy` очищает `SecurityContextHolder` (`clearContext()`). Контекст **не сохраняется** — следующий запрос снова начинает с пустого контекста.

**Ключевые отличия от stateful-подхода**

| Аспект | Stateful (session) | Stateless (JWT) |
|---|---|---|
| Хранилище контекста | `HttpSessionSecurityContextRepository` | `NullSecurityContextRepository` |
| `SessionCreationPolicy` | `IF_REQUIRED` | `STATELESS` |
| Аутентификация восстанавливается | Из сессии по `JSESSIONID` | Из JWT в каждом запросе |
| Масштабирование | Требует sticky sessions или репликации | Горизонтальное масштабирование без ограничений |

**Важные нюансы**

- Всегда создавайте новый экземпляр `SecurityContext` через `SecurityContextHolder.createEmptyContext()`, а не через `getContext()` — это исключает гонки при использовании общего пула потоков.
- При `SessionCreationPolicy.STATELESS` CSRF-защита обычно отключается, так как состояние сессии отсутствует и CSRF-атака не применима (нет cookie сессии, которую браузер отправляет автоматически).
- Используйте проверенную библиотеку (например, `io.jsonwebtoken:jjwt` или `com.nimbusds:nimbus-jose-jwt`) для работы с токенами — не реализуйте разбор JWT вручную.

#### Оригинальный ответ из интернета
> Источник: [JSON Web Token for Java - OWASP Cheat Sheet Series](https://cheatsheetseries.owasp.org/cheatsheets/JSON_Web_Token_for_Java_Cheat_Sheet.html)

OWASP рекомендует делать stateless JWT короткоживущими, чтобы минимизировать окно возможности для атакующего. Для долгоживущего доступа следует использовать refresh-токены согласно стандартам OAuth.

**Хранение на клиенте:** нельзя хранить токены аутентификации (JWT, refresh tokens, session IDs) в `localStorage` или `sessionStorage`, так как эти API доступны любому JavaScript-коду, выполняющемуся в данном origin — одна XSS-уязвимость раскрывает все токены. Рекомендуемый подход: использовать куки с атрибутами `HttpOnly; Secure; SameSite=Strict` или паттерн Backend-for-Frontend (BFF).

Токен со статусом "revoked" следует помещать в deny list (чёрный список) вплоть до момента его истечения по `exp`-claim.

#### Ответ от Claude

**Access Token**

Access Token — краткоживущий токен (обычно JWT), предъявляемый клиентом при каждом обращении к защищённому ресурсу. Сервер ресурсов верифицирует его подпись и claims локально, без обращения к базе данных или auth-серверу.

Типичное время жизни: **5–15 минут** (OWASP), максимум — часы. Содержит: `sub` (идентификатор пользователя), `exp` (время истечения), `iat` (время выдачи), `scope`/роли.

**Зачем Access Token должен быть короткоживущим**

1. **JWT нельзя отозвать мгновенно.** Stateless-токен валиден до истечения `exp`. Если токен украден, атакующий будет иметь доступ ровно до момента истечения. Короткое время жизни минимизирует ущерб.
2. **Компрометация ограничена во времени.** При утечке access token атакующий имеет доступ максимум на несколько минут.
3. **Смена прав применяется быстро.** Если у пользователя отзывают роль, следующий access token будет выдан уже без неё.

Альтернатива для ситуаций, требующих немедленного отзыва — использование `jti` (JWT ID) + deny list на стороне сервера, но это нарушает stateless-природу JWT.

**Refresh Token**

Refresh Token — долгоживущий непрозрачный (opaque) токен, хранящийся на сервере. Клиент предъявляет его auth-серверу для получения нового access token, когда текущий истёк. Время жизни — часы, дни или недели.

Refresh token **не отправляется** на сервер ресурсов (Resource Server), только на Authorization Server (эндпоинт `/token`).

**Refresh Token Rotation**

При каждом обмене refresh token → новый access token сервер одновременно выдаёт **новый refresh token** и **инвалидирует старый**. Это ключевой механизм обнаружения кражи токена: если украденный refresh token уже был использован атакующим, легитимная попытка обмена старого токена должна немедленно инвалидировать всю сессию (detect reuse → revoke family).

```
Клиент                      Auth Server
  |--- POST /token (refresh_token=RT1) -->|
  |<-- { access_token: AT2,              |
  |       refresh_token: RT2 } ----------|
  # RT1 инвалидирован, RT2 активен

  # Если атакующий использует RT1 ещё раз:
  |--- POST /token (refresh_token=RT1) -->|
  |<-- 400 invalid_grant (RT1 revoked) --|
  # Сервер должен инвалидировать всю цепочку (RT2 и все дочерние)
```

**Безопасное хранение Refresh Token на клиенте**

| Хранилище | Уязвимость | Рекомендация |
|---|---|---|
| `localStorage` / `sessionStorage` | XSS (любой JS в origin может прочитать) | Нельзя использовать |
| Memory (JS-переменная) | Сбрасывается при закрытии вкладки; уязвим к XSS в рамках сессии | Допустимо для access token |
| `HttpOnly; Secure; SameSite=Strict` cookie | Защищено от XSS (JS не читает); уязвимо к CSRF без `SameSite` | **Рекомендуется** для refresh token |
| Backend-for-Frontend (BFF) | Клиент вообще не хранит токены; вся логика на сервере | **Рекомендуется** для SPA |

**Атрибуты cookie для Refresh Token:**
```
Set-Cookie: refresh_token=<value>;
  HttpOnly;          -- JS не может прочитать через document.cookie
  Secure;            -- передаётся только по HTTPS
  SameSite=Strict;   -- не отправляется при кросс-сайтовых запросах (CSRF-защита)
  Path=/auth/token;  -- ограничение пути: cookie отправляется только на эндпоинт обмена
  Max-Age=604800     -- 7 дней
```

**Ограничение `Path`** — важный дополнительный уровень защиты: refresh token cookie будет отправляться браузером только на `/auth/token`, но не на все API-запросы.

**Дополнительные меры защиты Refresh Token на сервере**

- Хранить хеш refresh token в БД (не сам токен, как и для паролей).
- Привязывать к `user_agent`, IP-адресу (опционально, с учётом мобильных сетей).
- Ограничивать количество активных сессий на пользователя.
- Логировать все события обмена с целью обнаружения аномалий.

### Вопрос 18. Как работает механизм cost factor (work factor) в bcrypt? Как выбрать подходящее значение cost и почему увеличение cost замедляет brute-force?

**Категория:** Безопасность · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Password Storage Cheat Sheet - OWASP Cheat Sheet Series](https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html)

Work factor (cost factor) bcrypt — это количество итераций хэш-алгоритма (фактически 2^cost итераций). Work factor должен быть как можно больше, насколько позволяет производительность сервера, с минимальным значением **10**. Из-за конструктивного ограничения bcrypt входной пароль обрезается до **72 байт** в большинстве реализаций — поэтому следует ограничивать максимальную длину пароля 72 байтами.

Медленные алгоритмы хэширования (bcrypt, Argon2, scrypt) делают brute-force-атаки значительно сложнее, дороже и затратнее по времени, чем быстрые алгоритмы (SHA-256), которые позволяют атакующему проверять миллиарды паролей в секунду.

При legacy-системах, где недоступны Argon2 и scrypt, следует использовать bcrypt с work factor ≥ 10.

#### Ответ от Claude

**История и основа алгоритма**

Bcrypt разработан Нильсом Провосом и Дэвидом Мазьером в 1999 году и основан на блочном шифре Blowfish. Ключевое нововведение — алгоритм "Eksblowfish" (Expensive Key Schedule Blowfish): в отличие от стандартного Blowfish, процедура расширения ключа (`key schedule`) намеренно сделана медленной и дорогостоящей.

**Внутреннее устройство алгоритма**

Процесс хэширования bcrypt состоит из следующих этапов:

1. **Генерация случайной соли.** Генерируется 16-байтовая (128-бит) криптографически стойкая случайная соль с помощью CSPRNG.

2. **Инициализация EksBlowfish (`eksblowfish_setup`).** Используя пароль (до 72 байт), соль и значение cost, инициализируются P-блоки и S-блоки шифра. Именно здесь выполняется **2^cost итераций** перемешивания подключей. Это вычислительно дорогая операция, которая и составляет суть защиты.

3. **Шифрование магической строки.** После инициализации ключа строка-константа `"OrpheanBeholderScryDoubt"` (24 байта) шифруется в режиме ECB ровно **64 раза** с полученным состоянием шифра.

4. **Формирование итогового хэша.** Cost, соль и результат шифрования упаковываются в формат Modular Crypt Format:
   ```
   $2a$12$SaltSaltSaltSaltSaltSaHashHashHashHashHashHashHashHashHashHash
   └┘ └┘ └────────────────────┘└───────────────────────────────────────┘
   ver cost     соль (22 символа)          хэш (31 символ)
   ```

**Как работает cost factor**

Значение cost задаёт степень двойки для числа итераций:

| cost | Итерации (2^cost) | Относительное время |
|------|-------------------|---------------------|
| 10   | 1 024             | 1×                  |
| 11   | 2 048             | 2×                  |
| 12   | 4 096             | 4×                  |
| 13   | 8 192             | 8×                  |
| 14   | 16 384            | 16×                 |

Каждое увеличение cost на 1 **удваивает** время вычисления. Это экспоненциальная зависимость. Алгоритм адаптивный: по мере роста производительности железа cost можно поднимать, чтобы время хэширования оставалось достаточно долгим.

**Почему увеличение cost замедляет brute-force**

Скорость brute-force-атаки ограничена скоростью генерации хэшей. Алгоритмы вроде SHA-256 или MD5 (быстрые) позволяют атакующему на современном GPU проверять **миллиарды паролей в секунду**. Bcrypt с cost=12 занимает ~250 мс на одну итерацию хэширования — то есть тот же GPU сможет проверять лишь **несколько тысяч паролей в секунду**, а не миллиарды.

Пример сравнения (примерные цифры для одной GPU NVIDIA RTX 3090):
- MD5: ~60 000 000 000 хэшей/сек
- bcrypt (cost=10): ~20 000 хэшей/сек
- bcrypt (cost=12): ~5 000 хэшей/сек

Даже если атакующий получит дамп базы паролей, стоимость перебора делает атаку практически нереализуемой для стойких паролей.

**Как выбрать подходящее значение cost**

Рекомендуемый подход: **подобрать значение, при котором хэширование одного пароля занимает ~100–300 мс на вашем production-железе.**

```java
// Программный подбор оптимального cost:
int targetMs = 250; // целевое время хэширования
int cost = 10;
while (true) {
    long start = System.currentTimeMillis();
    BCrypt.hashpw("benchmark", BCrypt.gensalt(cost));
    long elapsed = System.currentTimeMillis() - start;
    if (elapsed >= targetMs) break;
    cost++;
}
System.out.println("Recommended cost: " + cost);
```

Практические ориентиры:
- **Минимальный cost: 10** (требование OWASP).
- **Рекомендуемый для 2024–2026: 12–13** — большинство современных серверов справляются за ~100–300 мс.
- **Cost 14+** — для высокочувствительных систем, где допустима бо́льшая задержка при аутентификации.
- Проводите бенчмарк на каждом новом поколении железа и поднимайте cost при обновлении инфраструктуры.

**Ограничение 72 байта и предварительное хэширование**

Bcrypt обрезает вход до 72 байт. Если нужно поддерживать длинные пароли, распространённый паттерн — предварительно хэшировать пароль через SHA-256/SHA-512 перед подачей в bcrypt. Однако это спорная практика: она может привести к проблемам с нулевыми байтами ("") в base64-выводе SHA. Безопаснее ограничить максимальную длину пароля 72 байтами на уровне валидации.

**Spring Security: BCryptPasswordEncoder**

```java
// cost factor задаётся в конструкторе (по умолчанию — 10)
PasswordEncoder encoder = new BCryptPasswordEncoder(12);

String hash = encoder.encode("myPassword");
boolean matches = encoder.matches("myPassword", hash);
```

`BCryptPasswordEncoder` из Spring Security генерирует новую случайную соль при каждом вызове `encode()`, поэтому один и тот же пароль всегда даёт разный хэш — это корректное поведение, не баг.
### Вопрос 19. PreparedStatement защищает от SQL-инъекции, но есть случаи, когда она всё равно возможна при его использовании. Назовите конкретные сценарии и объясните, почему параметризация там не помогает.

**Категория:** Безопасность · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [SQL Injection Prevention Cheat Sheet — OWASP](https://cheatsheetseries.owasp.org/cheatsheets/SQL_Injection_Prevention_Cheat_Sheet.html)

OWASP прямо указывает: параметризованные запросы защищают только значения данных. Структурные элементы SQL — имена таблиц, столбцов, ключевые слова ORDER BY, ASC/DESC — не могут быть переданы через параметры-заглушки (`?`). Если приложение формирует такие части запроса конкатенацией строк (например, на основе пользовательского ввода), SQL-инъекция остаётся возможной. Дополнительно OWASP предупреждает, что хранимые процедуры, использующие внутри себя динамический SQL с EXECUTE IMMEDIATE или exec(), также могут быть уязвимы, даже если вызываются через PreparedStatement.

#### Ответ от Claude

PreparedStatement защищает только **значения-параметры** — те позиции в запросе, которые обозначены символом `?`. Структура самого SQL-выражения фиксируется в момент подготовки запроса и не может быть изменена через параметры. Существует несколько конкретных сценариев, в которых это ограничение делает параметризацию недостаточной.

**1. Динамические имена таблиц и столбцов**

Имена объектов БД не могут быть параметрами в SQL. Если приложение формирует имя таблицы или столбца из пользовательского ввода — возникает уязвимость:

```java
// НЕБЕЗОПАСНО: имя столбца вставляется через конкатенацию
String query = "SELECT * FROM users ORDER BY " + userInput;
PreparedStatement ps = conn.prepareStatement(query);
```

Злоумышленник может передать `id; DROP TABLE users--` или вложенный подзапрос. PreparedStatement здесь не поможет, потому что к моменту вызова `prepareStatement()` опасный текст уже встроен в SQL-строку. Защита: белый список допустимых значений (`allowlist`), проверяемый на уровне Java-кода.

**2. Клауза ORDER BY (направление сортировки и имя колонки)**

Ни имя колонки для сортировки, ни направление (`ASC`/`DESC`) не поддерживают параметризацию. OWASP и Baeldung фиксируют это как распространённую ошибку в отчётном ПО:

```java
// НЕБЕЗОПАСНО
String sql = "SELECT * FROM products ORDER BY " + sortColumn + " " + sortDirection;
```

Решение — сравнение переданного значения с фиксированным enum или `Set<String>` допустимых имён колонок.

**3. Wildcards в операторе LIKE**

PreparedStatement защищает от SQL-инъекции через `LIKE`, но **не защищает от инъекции wildcard-символов**. Символы `%` и `_` являются частью SQL-синтаксиса оператора LIKE, а не метасимволами SQL-парсера. PreparedStatement передаёт их как обычные строковые значения, и база данных интерпретирует их как шаблон поиска:

```java
// Пользователь вводит "%", получает все строки вместо ничего
String sql = "SELECT * FROM discount_codes WHERE code LIKE ?";
ps.setString(1, userInput); // userInput = "%" — вернёт все коды
```

Это не SQL-инъекция в классическом смысле, но приводит к нарушению логики доступа к данным. Исправление: экранировать `%` и `_` в пользовательском вводе перед привязкой параметра и указать символ экранирования через `ESCAPE`:

```java
String escaped = userInput.replace("\\", "\\\\").replace("%", "\\%").replace("_", "\\_");
ps.setString(1, escaped + "%");
// В SQL: ... LIKE ? ESCAPE '\'
```

**4. Хранимые процедуры с внутренним динамическим SQL**

PreparedStatement безопасно передаёт параметры в хранимую процедуру. Однако если сама процедура внутри выполняет динамический SQL с конкатенацией (`EXECUTE IMMEDIATE` в PL/SQL, `EXEC` или `sp_executesql` в T-SQL), инъекция происходит уже внутри СУБД:

```sql
-- НЕБЕЗОПАСНАЯ хранимая процедура (PL/SQL)
CREATE PROCEDURE search_users(p_name IN VARCHAR2) AS
BEGIN
  EXECUTE IMMEDIATE 'SELECT * FROM users WHERE name = ''' || p_name || '''';
END;
```

Вызов этой процедуры через PreparedStatement с параметром `' OR '1'='1` выполнит инъекцию внутри СУБД. Защита: использовать внутри процедур только параметризованные запросы (USING-клаузу в EXECUTE IMMEDIATE).

**5. Клиентская эмуляция PreparedStatement**

В MySQL Connector/J по умолчанию (без `useServerPrepStmts=true`) PreparedStatement реализован на стороне **клиента**: JDBC-драйвер просто подставляет значения в строку запроса с экранированием и отправляет готовый SQL на сервер. При наличии ошибки в логике экранирования драйвера (например, в редких edge-case с многобайтовыми кодировками) инъекция становится возможной. Кроме того, клиентские фреймворки, строящие SQL через конкатенацию и лишь имитирующие «параметризацию», также не дают защиты.

**6. Запросы с IN-клаузой и динамическим числом элементов**

PreparedStatement не поддерживает передачу массива значений в `IN (?)`. Разработчики нередко строят IN-список через конкатенацию:

```java
// НЕБЕЗОПАСНО
String ids = String.join(",", userIds); // userIds из ввода пользователя
String sql = "SELECT * FROM orders WHERE id IN (" + ids + ")";
```

Правильный подход — динамически строить строку из нужного числа `?` и привязывать каждый элемент отдельно.

**Итог:** PreparedStatement — необходимый, но недостаточный инструмент. Его ограничения охватывают все элементы SQL, не являющиеся скалярными значениями данных. Для структурных элементов (имена объектов, ключевые слова) обязателен белый список валидных значений на стороне приложения.

### Вопрос 20. Объясните механизм работы PreparedStatement на уровне протокола JDBC: в какой момент SQL парсится базой данных и почему пользовательский ввод уже не может изменить структуру запроса?

**Категория:** Безопасность · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Using Prepared Statements — Oracle Java Tutorials (JDBC Basics)](https://docs.oracle.com/javase/tutorial/jdbc/basics/prepared.html)

Официальная документация Oracle объясняет: главная особенность PreparedStatement состоит в том, что SQL-выражение передаётся базе данных **в момент создания объекта**, где немедленно компилируется. Объект PreparedStatement содержит не просто SQL-строку, а уже **предкомпилированный** SQL. При последующих вызовах execute() СУБД выполняет уже готовый план — без повторного разбора. Документация явно формулирует ключевое свойство безопасности: _"Prepared statements always treat client-supplied data as content of a parameter and never as a part of an SQL statement."_ Именно потому, что SQL-структура зафиксирована до привязки любых параметров, пользовательский ввод физически не может изменить её.

#### Ответ от Claude

**Жизненный цикл PreparedStatement на уровне протокола состоит из двух чётко разделённых фаз.**

**Фаза 1 — Подготовка (Prepare)**

При вызове `connection.prepareStatement(sql)` происходит следующее:

1. JDBC-драйвер отправляет SQL-строку с заглушками-параметрами (`?`) на сервер СУБД по сетевому протоколу (например, MySQL Protocol или PostgreSQL Extended Query Protocol).
2. СУБД выполняет **полный лексический и синтаксический разбор** запроса: проверяет синтаксис, разрешает имена таблиц и столбцов, проверяет права доступа.
3. Строится и кэшируется **план выполнения** (execution plan) — дерево операций с уже выбранными индексами и стратегией соединений.
4. Сервер возвращает драйверу **идентификатор подготовленного запроса** (statement handle, например `S_1`).

На этом этапе СУБД уже знает всё о структуре запроса. Параметров ещё нет — только их позиции в синтаксическом дереве помечены как «слоты для данных».

**Фаза 2 — Выполнение (Execute)**

При вызове `ps.executeQuery()` или `ps.executeUpdate()`:

1. Драйвер сериализует значения параметров (не SQL-текст!) и передаёт их серверу вместе с идентификатором подготовленного запроса.
2. В случае **серверного PreparedStatement** (true server-side) данные передаются в **бинарном формате** по протоколу бинарных параметров: без кавычек, без SQL-экранирования, как типизированные байтовые последовательности (int как 4 байта, строка как prefixed bytes и т.д.).
3. СУБД подставляет значения в уже скомпилированный план как **операнды**, не перепарсивая SQL.

**Почему пользовательский ввод не может изменить структуру запроса**

Принципиальная причина — **разрыв во времени и канале передачи**: SQL-структура передаётся и разбирается в фазе Prepare, тогда как пользовательские данные передаются позже, в фазе Execute, и никогда не смешиваются с SQL-текстом. СУБД получает не строку `SELECT * FROM users WHERE name = 'admin' OR '1'='1'`, а операцию `EXECUTE S_1 WITH ('admin'' OR ''1''=''1')`, где строка целиком является одним строковым значением параметра — и СУБД интерпретирует её именно так.

**Серверная vs. клиентская эмуляция**

Важное практическое уточнение: не все JDBC-драйверы по умолчанию используют протокол серверной подготовки.

- **MySQL Connector/J**: по умолчанию применяет **клиентскую эмуляцию** — драйвер подставляет экранированные значения в SQL-строку и отправляет готовый SQL на сервер как обычный Statement. Это всё ещё защищает от инъекции (при корректной реализации экранирования), но не использует бинарный протокол и не даёт двухфазного разделения. Для включения истинного серверного режима нужно указать `useServerPrepStmts=true` в строке подключения.
- **PostgreSQL pgJDBC**: использует Extended Query Protocol (двухфазный режим) начиная с того выполнения, которое превышает `prepareThreshold` (по умолчанию 5 выполнений одного запроса). После достижения порога драйвер переходит на именованный серверный стейтмент с бинарной передачей параметров.
- **Oracle JDBC**: серверная подготовка используется по умолчанию.

**Четырёхэтапная модель обработки (детально)**

Описание из технической документации фиксирует следующие этапы компиляции на стороне СУБД при подготовке запроса:

1. **Parsing** — синтаксический разбор SQL, верификация имён объектов, проверка прав.
2. **Optimization** — выбор оптимального плана выполнения (полный скан, индекс, стратегия JOIN).
3. **Storage (Plan Caching)** — сохранение плана в кэше сервера, привязанного к идентификатору запроса.
4. **Parameter Binding** — при каждом Execute: подстановка конкретных значений в слоты плана без повторения шагов 1–3.

Благодаря этой модели даже злонамеренная строка `'; DROP TABLE users; --` воспринимается СУБД как обычное строковое значение длиной N символов — она попадает в слот данных плана, а не в SQL-парсер.
