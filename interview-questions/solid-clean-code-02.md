### Вопрос 21. В чём разница между SRP и высокой связностью (high cohesion)? Можно ли нарушить SRP, не снижая cohesion?

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Single-responsibility principle — Wikipedia](https://en.wikipedia.org/wiki/Single_responsibility_principle)

Принцип единственной ответственности гласит: "Модуль должен отвечать перед одним и только одним актором". Роберт Мартин явно указывает, что SRP основан на принципе связности (cohesion) в том виде, в каком его определили Том ДеМарко и Мейлир Пейдж-Джонс. Мартин формулирует это так: "Собирай вместе то, что меняется по одним причинам. Разделяй то, что меняется по разным причинам." Это определение непосредственно перекликается с принципами связности: группировать родственную функциональность при разделении разнородных обязанностей.

#### Ответ от Claude

**SRP и high cohesion — родственные, но не тождественные концепции.**

**Определения**

*High cohesion* (высокая связность) — метрика из структурного проектирования (Том ДеМарко, Ларри Константин, 1970-е). Измеряет, насколько тесно связаны методы и поля внутри модуля. Чем больше методов работают с одними и теми же данными и решают одну задачу, тем выше cohesion. Это *измеримая характеристика*, существующая в спектре: от coincidental (случайная, низшая) до functional cohesion (функциональная, наивысшая).

*SRP* — принцип из SOLID (Роберт Мартин, 2000-е). Формулировка: "Класс должен иметь только одну причину для изменения", более поздняя версия: "Модуль должен отвечать перед одним и только одним актором". Это *нормативное правило*, бинарное: либо соблюдается, либо нет.

**Ключевые различия**

| Аспект | High Cohesion | SRP |
|--------|---------------|-----|
| Происхождение | Структурное проектирование (DeMarco, 1979) | ООП-принцип (Martin, ~2002) |
| Природа | Измеримая характеристика, спектр | Нормативный принцип, бинарный |
| Фокус | Внутренняя связность элементов модуля | Внешний актор, причина изменения |
| Вопрос | "Насколько логично сгруппированы элементы?" | "Кто диктует изменения в этом классе?" |

**Тонкое различие: "логическая связность" vs "организационная ответственность"**

Пример: класс `Employee` с методами `calculatePay()`, `reportHours()`, `save()`.

- С точки зрения *cohesion* — класс может казаться связным: все методы относятся к "сотруднику".
- С точки зрения *SRP* — это нарушение, потому что три разных актора диктуют изменения: бухгалтерия (`calculatePay`), HR-отдел (`reportHours`), DBA-команда (`save`). Три причины для изменения.

Мартин специально использует этот пример в "Clean Architecture", чтобы показать: высокая логическая связность не защищает от нарушения SRP.

**Можно ли нарушить SRP, не снижая cohesion?**

Да — и это не надуманный сценарий.

```java
// Класс Report — высокая cohesion: все методы работают с данными отчёта
// Но нарушает SRP: два разных актора
public class Report {
    private List<ReportLine> lines;
    
    public String generateHTML() { ... }   // <- веб-команда диктует формат
    public String generatePDF()  { ... }   // <- print-команда диктует формат
    public List<ReportLine> getData() { ... }
}
```

Все четыре метода тематически связаны с "отчётом" — cohesion высокая. Но изменения инициируются двумя независимыми командами с разными требованиями — SRP нарушен. Если веб-команда захочет изменить шаблон HTML, это потребует изменения того же класса, что и изменения формата PDF по инициативе print-команды.

**Обратная ситуация: соблюдение SRP при низкой cohesion**

Формально возможна (класс обслуживает одного актора, но содержит разнородные утилиты), хотя это плохой дизайн.

**Практический итог**

- *High cohesion* — необходимое, но не достаточное условие для SRP.
- SRP добавляет организационное измерение: кто именно инициирует изменения.
- Мартин сам признаёт, что SRP "основан на cohesion", но уточняет его через понятие актора — конкретного стейкхолдера или роли в организации.
- При проектировании: сначала убедитесь в высокой cohesion, затем задайте вопрос "чьи требования меняют этот класс?" — и если ответов несколько, разбейте на части.
### Вопрос 22. Классический пример нарушения LSP — Rectangle/Square. Как именно нарушается LSP и как правильно спроектировать иерархию? Чем это отличается от нарушения через выброс UnsupportedOperationException?

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [The Liskov Substitution Principle with Examples — DZone](https://dzone.com/articles/the-liskov-substitution-principle-with-examples)

Нарушение происходит из-за того, что `Square` не может полностью заменить `Rectangle`. Когда `Square` расширяет `Rectangle`, он переопределяет методы `setLength()` и `setBreadth()` так, чтобы обе стороны всегда оставались равными. Это ломает контракт `Rectangle`.

Контракт `Rectangle` гарантирует:
- «Длина всегда равна значению, переданному в `setLength`»
- «Ширина всегда равна значению, переданному в `setBreadth`»

Когда вызываются `r.setLength(3)` и `r.setBreadth(2)` на экземпляре `Square`, обе стороны становятся равными 2 (последнему установленному значению), нарушая ожидаемое поведение. Утверждения `assert r.getLength() == 3`, `assert r.getBreadth() == 2`, `assert r.getArea() == 6` — все провалятся.

Фундаментальная проблема по версии источника: «Класс `LSPDemo` вынужден знать детали подклассов `Rectangle` (в частности, `Square`), чтобы написать корректный код и не получить ошибку».

Правильное решение: не использовать наследование. `Square` и `Rectangle` — математически разные понятия, они не должны состоять в отношении IS-A. Следует использовать композицию или раздельные иерархии.

#### Ответ от Claude

**Суть нарушения LSP в паре Rectangle/Square**

LSP (принцип подстановки Лисков) требует: объекты подкласса должны быть взаимозаменяемы с объектами базового класса, не нарушая корректности программы. Нарушение в паре Rectangle/Square проявляется через разрушение поведенческого контракта.

`Rectangle` устанавливает контракт независимости измерений: можно независимо задать ширину и высоту. `Square`, будучи подклассом, вынужден переопределить оба сеттера, чтобы обе стороны оставались равными:

```java
// Rectangle
public void setWidth(double width)  { this.width = width; }
public void setHeight(double height) { this.height = height; }

// Square (переопределяет, нарушая контракт)
@Override
public void setWidth(double width) {
    this.width = width;
    this.height = width; // принудительно синхронизирует
}
@Override
public void setHeight(double height) {
    this.width = height;
    this.height = height;
}
```

Клиентский код, написанный для `Rectangle`, ломается при получении `Square`:

```java
void testArea(Rectangle r) {
    r.setWidth(5);
    r.setHeight(4);
    assert r.getArea() == 20; // ПРОВАЛ: Square вернёт 16, т.к. setHeight(4) перезапишет width
}
```

Клиент вынужден добавлять проверку типа (`instanceof Square`) — это прямое нарушение принципа: код становится зависим от знания конкретного подтипа.

**Правильное проектирование иерархии**

Решение — разорвать ложное отношение IS-A и ввести общий абстрактный тип:

```java
interface Shape {
    double area();
}

class Rectangle implements Shape {
    private double width, height;
    // независимые сеттеры
    public double area() { return width * height; }
}

class Square implements Shape {
    private double side;
    // один сеттер для стороны
    public double area() { return side * side; }
}
```

`Rectangle` и `Square` становятся «сестринскими» классами, оба реализуют `Shape`. Ни один из них не подставляется вместо другого там, где ожидается независимость измерений.

**Чем это отличается от нарушения через UnsupportedOperationException**

Нарушение через `UnsupportedOperationException` — другой паттерн, но суть та же: подкласс не выполняет контракт базового класса. Классический пример в Java — `java.util.Collections.unmodifiableList()` или `Arrays.asList()`: возвращаемый список является подтипом `List`, но при вызове `add()` или `remove()` выбрасывает `UnsupportedOperationException`, тогда как контракт `List` подразумевает, что операция поддерживается (или по меньшей мере не выдаёт неожиданного исключения при штатном использовании).

| Аспект | Rectangle/Square | UnsupportedOperationException |
|---|---|---|
| Тип нарушения | Неправильный результат (тихое изменение поведения) | Исключение вместо ожидаемого результата |
| Обнаружение | Только через семантику и тесты | Немедленный сбой в рантайме |
| Опасность | Труднее обнаружить: код «работает», но даёт неверные данные | Явный сбой, но также нарушает контракт |
| Решение | Переработать иерархию или использовать композицию | Разделить интерфейс (ISP): ввести `ReadOnlyList` и `MutableList` |

Оба случая нарушают LSP: в первом — контракт postcondition (результат операции), во втором — precondition (операция вообще не выполняется). LSP запрещает и то, и другое: нельзя ни ослаблять постусловия, ни усиливать предусловия, ни добавлять новые исключения.

### Вопрос 23. OCP требует закрытости для модификации — но как тогда исправлять баги? Как совместить OCP с необходимостью изменять существующий код?

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [SOLID series: The Open-Closed Principle — LogRocket Blog](https://blog.logrocket.com/solid-open-closed-principle/)

Статья прямо называет «OCP означает никогда не изменять код» фундаментальным заблуждением. Как поясняет автор: «Идея — минимизировать изменения, а не ставить вечный и неподвижный запрет на них». Это непосредственно отвечает на вопрос: исправление багов является законным исключением из правила «расширение вместо модификации».

OCP не поощряет *модификацию существующего кода ради новых функций*, но не запрещает необходимые изменения, такие как исправление багов. Принцип направлен против *предотвратимых* изменений через лучший архитектурный дизайн, а не против любых изменений вообще.

Статья рекомендует стратегически применять OCP, задавая вопросы:
- Отвечает ли эта гибкость реальным, текущим потребностям?
- Не было бы более простое изменение лучше добавления слоёв абстракции?
- Помог бы простой рефакторинг в данной ситуации?

Исправления багов — это законное техническое обслуживание, сохраняющее стабильность системы; они отличаются от добавления новых функций через модификацию.

#### Ответ от Claude

**Что на самом деле означает «закрыт для модификации»**

OCP в формулировке Бертрана Мейера (1988) и переработанной версии Роберта Мартина (2000) говорит: программные сущности должны быть открыты для расширения, но закрыты для модификации. «Закрыты» — не означает «никогда не трогать исходный код», это означает «не изменять уже работающее поведение для добавления новой функциональности».

Ключевое разграничение, которое необходимо понимать:

| Тип изменения | OCP разрешает? | Обоснование |
|---|---|---|
| Исправление бага | Да | Баг — это дефект уже зафиксированного поведения; исправление восстанавливает корректность, не меняет интерфейс |
| Добавление новой функции через изменение класса | Нет | Нарушает принцип; правильно — расширить через новый класс/реализацию |
| Рефакторинг без смены поведения | Да | Не затрагивает семантику, только структуру |
| Изменение логики существующего метода ради нового сценария | Нет | Именно это OCP запрещает |

**Практическое совмещение OCP с реальной разработкой**

1. **Баги исправляются напрямую.** Баг — это несоответствие между фактическим и заявленным поведением. Его исправление не расширяет систему, а возвращает её в корректное состояние. Это исключение из правила, прямо признаваемое большинством авторов (включая Мартина).

2. **Новые требования — через расширение.** Если приходит новый тип платежа, новый формат отчёта или новый алгоритм — не меняем существующий класс, а создаём новую реализацию интерфейса:

```java
// Закрытый для модификации базовый механизм
interface PaymentProcessor {
    void process(Payment payment);
}

// Расширение без модификации существующих классов
class CryptoPaymentProcessor implements PaymentProcessor {
    public void process(Payment payment) { /* новая логика */ }
}
```

3. **OCP — не абсолют, а вектор проектирования.** Роберт Мартин в «Clean Architecture» признаёт, что 100%-ное соблюдение OCP недостижимо: всегда найдётся тип изменений, к которому система не была готова. Цель — применять OCP к наиболее вероятным осям изменений.

4. **«Закрытость» достигается через абстракции.** Когда высокоуровневая политика зависит от интерфейса, а не от конкретного класса, её исходный код остаётся неизменным при добавлении новых реализаций. Именно это и называется «закрытостью» — защита от волновых изменений при добавлении новых функций.

**Резюме.** OCP — про архитектурную устойчивость к росту функциональности, а не про запрет на обслуживание кода. Баги исправляются в исходном коде. Рефакторинг допустим. Новые функции добавляются через новые классы/реализации, не через правку старых.

### Вопрос 24. Чем DIP отличается от Dependency Injection? Можно ли соблюдать DIP без IoC-контейнера?

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Introduction IoC, DIP, DI and IoC Container — TutorialsTeacher](https://www.tutorialsteacher.com/ioc/introduction)

По классификации источника:
- **DIP и IoC** — «высокоуровневые принципы проектирования»
- **DI (Dependency Injection)** — «паттерн проектирования»
- **IoC Container** — «фреймворк»

DIP гарантирует, что высокоуровневые модули не зависят от низкоуровневых; оба должны зависеть от абстракций. DI — паттерн, реализующий принцип IoC для инвертирования создания зависимых объектов. IoC Container автоматизирует DI в масштабах приложения.

Источник утверждает: «Нельзя достичь слабой связанности классов, используя только IoC. Вместе с IoC также необходимо использовать DIP, DI и IoC Container» — это указывает на то, что все три понятия взаимодополняют, а не заменяют друг друга.

#### Ответ от Claude

**DIP — принцип, DI — паттерн**

DIP (Dependency Inversion Principle) и Dependency Injection — связанные, но принципиально разные вещи.

**DIP** — это принцип SOLID, описывающий *структуру зависимостей*:
1. Высокоуровневые модули не должны зависеть от низкоуровневых модулей. Оба должны зависеть от абстракций.
2. Абстракции не должны зависеть от деталей. Детали должны зависеть от абстракций.

DIP говорит *что* должно зависеть от *чего*, но ничего не говорит о том, *как* зависимости попадают в объект.

**Dependency Injection** — это паттерн (техника), описывающий *способ передачи зависимостей*: зависимости передаются объекту извне (через конструктор, сеттер или интерфейс), а не создаются внутри.

| Понятие | Тип | Отвечает на вопрос |
|---|---|---|
| DIP | Принцип проектирования (SOLID) | На что должны опираться зависимости (абстракции, а не детали) |
| Dependency Injection | Паттерн реализации | Как зависимости попадают в объект (снаружи, не внутри) |
| IoC (Inversion of Control) | Широкий архитектурный принцип | Кто управляет потоком выполнения и жизненным циклом объектов |
| IoC Container (Spring, Guice и др.) | Фреймворк | Автоматизация создания и связывания объектов по описанным правилам |

**Можно ли соблюдать DIP без IoC-контейнера?**

Да, абсолютно. DIP соблюдается через введение абстракции между высокоуровневым и низкоуровневым модулем — это решение на уровне дизайна, не на уровне фреймворка.

Пример: DIP без Spring/Guice, только ручная инъекция:

```java
// Абстракция — зависимость описана через интерфейс (DIP выполнен)
public interface CoffeeMachine {
    Coffee brew();
}

// Высокоуровневый модуль зависит от абстракции, не от деталей
public class CoffeeApp {
    private final CoffeeMachine coffeeMachine;

    // Dependency Injection через конструктор (ручная)
    public CoffeeApp(CoffeeMachine coffeeMachine) {
        this.coffeeMachine = coffeeMachine;
    }
}

// Точка сборки — единственное место, где знают о конкретной реализации
public class Main {
    public static void main(String[] args) {
        CoffeeMachine machine = new PremiumCoffeeMachine(); // конкретика только здесь
        CoffeeApp app = new CoffeeApp(machine);             // ручная инъекция
    }
}
```

DIP здесь соблюдён: `CoffeeApp` зависит от `CoffeeMachine` (абстракция), а не от `PremiumCoffeeMachine` (деталь). Никакого IoC-контейнера нет.

**Роль IoC-контейнера**

IoC-контейнер (Spring, Guice, CDI) — это инструмент *автоматизации* ручной сборки зависимостей. Он позволяет не писать вручную цепочки `new A(new B(new C()))` в точке сборки, делегируя это фреймворку через аннотации или конфигурацию. Контейнер не является необходимым условием для соблюдения DIP — он лишь упрощает масштабируемое применение DI в больших приложениях.

**Типичная путаница и как её избежать**

- Можно использовать IoC-контейнер (Spring `@Autowired`) и при этом нарушать DIP — если инжектировать конкретный класс вместо интерфейса.
- Можно соблюдать DIP без какого-либо фреймворка — достаточно зависеть от абстракций и применять ручную инъекцию через конструктор.
- DI реализует IoC (управление созданием отдаётся внешнему коду), но DIP — это не про IoC, а про направление зависимостей в графе классов.
### Вопрос 25. Покажи пример, где нарушение ISP приводит к нарушению LSP. Как эти два принципа связаны на практике?

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Interface Segregation Principle: Everything You Need to Know — reflectoring.io](https://reflectoring.io/interface-segregation-principle/)

Статья демонстрирует, как нарушение ISP порождает нарушение LSP на примере сервиса заказов. Когда `BurgerOrderService` вынужден реализовать «толстый» интерфейс `OrderService`, содержащий метод `orderFries()`, которого в «бургерном» заказе нет, реализация выбрасывает исключение:

```java
@Override
public void orderFries(int fries) {
    throw new UnsupportedOperationException("No fries in burger only order");
}
```

Автор объясняет: «Мы выбросили `UnsupportedOperationException` в `BurgerOrderService`, что является нарушением LSP, поскольку дочерний класс не расширяет функциональность родителя, а ограничивает её». Выброс исключения в унаследованном методе ломает контракт подстановки: клиентский код, работающий с `OrderService`, не может безопасно использовать `BurgerOrderService` на его месте. Решение — разбить «толстый» интерфейс на узкоспециализированные интерфейсы, чтобы каждая реализация наследовала только те методы, которые реально поддерживает.

#### Ответ от Claude

**Механизм связи между ISP и LSP**

ISP (Interface Segregation Principle) запрещает «толстые» интерфейсы, заставляющие клиентов зависеть от методов, которые им не нужны. LSP (Liskov Substitution Principle) требует, чтобы объекты подтипов можно было подставить вместо объектов базового типа без нарушения поведения программы. Связь прямая: когда ISP нарушен, реализатор вынужден либо оставить метод пустым, либо бросить исключение — оба варианта нарушают LSP.

**Пример нарушения (Java)**

```java
// "Толстый" интерфейс — нарушение ISP
interface Worker {
    void work();
    void eat();
    void sleep();
}

// RobotWorker вынужден реализовать eat() и sleep(), которых у робота нет
class RobotWorker implements Worker {
    @Override
    public void work() {
        System.out.println("Robot working...");
    }

    @Override
    public void eat() {
        throw new UnsupportedOperationException("Robots don't eat"); // нарушение LSP
    }

    @Override
    public void sleep() {
        throw new UnsupportedOperationException("Robots don't sleep"); // нарушение LSP
    }
}

// Клиентский код ломается при подстановке RobotWorker вместо Worker
void lunchBreak(Worker worker) {
    worker.eat(); // NullPointerException или UnsupportedOperationException для RobotWorker
}
```

Клиент `lunchBreak` написан в расчёте на контракт `Worker`. Подстановка `RobotWorker` на место `Worker` нарушает этот контракт — LSP сломан.

**Правильное решение: разделить интерфейс**

```java
interface Workable  { void work(); }
interface Eatable   { void eat();  }
interface Sleepable { void sleep(); }

class HumanWorker implements Workable, Eatable, Sleepable {
    public void work()  { System.out.println("Human working"); }
    public void eat()   { System.out.println("Human eating");  }
    public void sleep() { System.out.println("Human sleeping"); }
}

class RobotWorker implements Workable {
    public void work() { System.out.println("Robot working"); }
}
```

Теперь `RobotWorker` реализует только `Workable`. Клиент, работающий с `Eatable`, просто не получит `RobotWorker` — компилятор не позволит. LSP соблюдён по конструкции: каждый объект полностью выполняет контракт своего интерфейса.

**Правило из практики:** маркер нарушения обоих принципов одновременно — `UnsupportedOperationException` или пустая реализация унаследованного метода. Это сигнал «интерфейс сказал слишком много», а наследник вынужден солгать.

**Ключевое различие принципов:**
- ISP смотрит «снизу вверх» (со стороны реализатора): «не заставляй класс зависеть от методов, которые он не использует».
- LSP смотрит «сверху вниз» (со стороны клиента): «подтип должен выполнять контракт родителя без сюрпризов».
- Нарушение ISP — структурная причина; нарушение LSP — поведенческое следствие.

### Вопрос 26. DRY говорит «не повторяй себя», но иногда дублирование лучше неправильной абстракции. Как ты определяешь, когда дублирование допустимо? Что такое «wrong abstraction»?

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [The Wrong Abstraction — Sandi Metz](https://sandimetz.com/blog/2016/1/20/the-wrong-abstraction)

Санди Метц утверждает: «Дублирование обходится намного дешевле, чем неправильная абстракция». Она описывает типичный жизненный цикл плохой абстракции: программист A замечает дублирование, извлекает его в общий метод и даёт ему имя. Позже программист B получает новое требование, которое «почти подходит» под существующую абстракцию. Чтобы не нарушать инвестицию в код, он добавляет параметр и ветку `if`. Процесс повторяется: абстракция обрастает параметрами и условиями, и в итоге её смысл теряется.

Метц рекомендует: когда абстракция сломана, «самый быстрый путь вперёд — это назад». Нужно вернуть дублирование: заинлайнить абстрагированный код обратно во всех вызывающих местах, удалить лишние ветки, применимые к каждому конкретному случаю, и только затем извлечь новые, более честные абстракции.

#### Ответ от Claude

**Что такое DRY на самом деле**

DRY (Don't Repeat Yourself) сформулирован Хантом и Томасом в «Pragmatic Programmer» как: «Каждая единица знания должна иметь единственное, однозначное, авторитетное представление в системе». Ключевое слово — «знание», а не «код». Одинаковый синтаксис не означает одинаковое знание.

**Что такое «wrong abstraction»**

Неправильная абстракция — это общий код (метод, класс, модуль), возникший из синтаксически похожего, но семантически различного дублирования. Признаки:

1. Метод принимает флаг или `enum` для переключения между принципиально разными ветками поведения.
2. Добавление нового вызывающего кода требует нового `if/switch` внутри «общей» функции.
3. Смысл параметров метода меняется в зависимости от вызывающего контекста.
4. Для понимания метода нужно знать всех его вызывателей.

**Когда дублирование допустимо (и лучше абстракции)**

| Ситуация | Почему лучше дублировать |
|---|---|
| Код похож синтаксически, но относится к разным бизнес-доменам | Домены будут эволюционировать независимо; общая абстракция создаст ненужную связанность |
| Третье вхождение ещё не появилось | «Правило трёх»: абстрагировать стоит при третьем повторении, не раньше |
| Абстракция уже обросла параметрами-флагами | Проще раздублировать и переписать честно |
| Разные команды/модули владеют своими копиями | Независимые изменения не должны ломать чужой код через общую абстракцию |

**Как определить «правильность» абстракции**

Задай три вопроса:
1. Это одно и то же **знание** или просто похожий **синтаксис**? (Формула расчёта налога — знание; цикл по массиву — синтаксис.)
2. Все потребители захотят **одних и тех же изменений**? Если нет — это случайное совпадение.
3. Что произойдёт, если один из вызывателей изменится? Если придётся менять и абстракцию — она была неверной.

**Алгоритм выхода из «wrong abstraction» по Метц**

```
1. Заинлайнить абстракцию обратно во все места вызова.
2. Для каждого вызова удалить ветки кода, которые туда не относятся
   (используя передаваемые параметры как подсказку).
3. Теперь у каждого вызывателя есть своя честная копия.
4. Если среди этих копий появляется новое реальное дублирование —
   извлечь новую, более точную абстракцию.
```

**Важно:** DRY нарушается не когда код выглядит одинаково, а когда одно и то же **решение** приходится менять в нескольких местах. Два метода с одинаковым телом, но разными причинами для изменения — это не нарушение DRY.

### Вопрос 27. Что такое code smell «Feature Envy» и чем он отличается от «Inappropriate Intimacy»? Какой рефакторинг применить в каждом случае?

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Feature Envy — sourcemaking.com](https://sourcemaking.com/refactoring/smells/feature-envy)

Feature Envy («зависть к чужим данным») — это ситуация, когда метод использует данные или методы другого объекта значительно чаще, чем собственные. Классический признак: метод «больше интересован» чужим классом, чем своим. Корневая причина — неправильно размещённая ответственность: метод физически находится не там, где находятся данные, с которыми он работает. Типичный рефакторинг — **Move Method**: перенести метод туда, где живут данные, которым он «завидует». Если лишь часть метода проявляет зависть — применяют **Extract Method** для выделения этой части, а затем Move Method для её переноса.

#### Ответ от Claude

**Feature Envy**

Метод класса A тратит большую часть своей логики на чтение полей или вызов методов класса B. Метод «завидует» данным соседнего класса.

```java
// Запах: OrderPrinter.print() завидует классу Order
class OrderPrinter {
    public String print(Order order) {
        // метод работает исключительно с данными Order,
        // не использует ничего из самого OrderPrinter
        return "Order #" + order.getId()
             + ", items: " + order.getItems().size()
             + ", total: " + order.getItems().stream()
                                  .mapToDouble(Item::getPrice)
                                  .sum();
    }
}

// Правильно: метод переезжает в Order
class Order {
    public String toDisplayString() {
        return "Order #" + id
             + ", items: " + items.size()
             + ", total: " + items.stream()
                                  .mapToDouble(Item::getPrice)
                                  .sum();
    }
}
```

**Рефакторинги для Feature Envy:**
- **Move Method** — перенести метод целиком в класс, которым «завидует».
- **Extract Method + Move Method** — если только часть метода страдает завистью, выделить эту часть и перенести.
- **Move Field** — если поле используется преимущественно в другом классе, перенести и его.

**Исключения:** паттерны **Strategy** и **Visitor** намеренно разделяют поведение и данные — там Feature Envy является дизайнерским решением, а не запахом.

---

**Inappropriate Intimacy**

Два класса слишком хорошо знают внутренние детали друг друга. В отличие от Feature Envy (односторонняя направленность: A → B), Inappropriate Intimacy — это **двусторонняя или чрезмерная связанность** между двумя классами.

```java
// Запах: Order и Customer слишком хорошо знают друг о друге
class Order {
    private Customer customer;

    public double getDiscountedTotal() {
        // Order напрямую лезет во внутренности Customer
        if (customer.loyaltyPoints > 1000) {          // прямой доступ к полю
            return total * 0.9;
        }
        if (customer.registrationDate.isBefore(       // прямой доступ к полю
                LocalDate.now().minusYears(2))) {
            return total * 0.95;
        }
        return total;
    }
}

class Customer {
    int loyaltyPoints;                    // вынужден быть package-private или public
    LocalDate registrationDate;           // вынужден быть package-private или public

    public double calculateOrderTotal(Order order) {
        // Customer тоже лезет в Order — двусторонняя связанность
        return order.getItems().stream()
                    .mapToDouble(Item::getPrice).sum();
    }
}
```

**Рефакторинги для Inappropriate Intimacy:**
- **Move Method / Move Field** — перенести поведение/данные туда, где они используются больше всего.
- **Extract Class** — если у двух классов есть общая ответственность, выделить её в третий класс.
- **Hide Delegate** — добавить метод-делегат, чтобы внешний класс не лез напрямую во внутренности.
- **Change Bidirectional Association to Unidirectional** — убрать обратную ссылку, если она не нужна.

**Сравнительная таблица**

| Критерий | Feature Envy | Inappropriate Intimacy |
|---|---|---|
| Направление | Односторонняя: метод A использует данные B | Двусторонняя или тесная взаимозависимость A ↔ B |
| Масштаб | Один метод | Два класса в целом |
| Суть проблемы | Метод находится не в том классе | Классы знают слишком много о внутренностях друг друга |
| Основной рефакторинг | Move Method | Move Method/Field, Extract Class, Hide Delegate |
| Симптом | Метод читает много чужих полей | Класс обращается к приватным/package-private деталям другого |

**Практическая связь:** многократное применение Move Method для лечения Feature Envy может постепенно перегрузить целевой класс и создать Inappropriate Intimacy. Поэтому после серии Move Method нужно проверить баланс ответственностей и при необходимости применить Extract Class.

### Вопрос 28. Чем «Shotgun Surgery» отличается от «Divergent Change»? Оба связаны с изменениями в нескольких местах — в чём принципиальная разница и какой принцип SOLID нарушается в каждом случае?

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Shotgun Surgery — refactoring.guru](https://refactoring.guru/smells/shotgun-surgery)

Shotgun Surgery возникает, когда одно изменение вынуждает вносить многочисленные мелкие правки в большое количество разных классов. Это запах противоположный Divergent Change: Divergent Change — это когда одному классу требуется много изменений, тогда как Shotgun Surgery — это когда одно изменение затрагивает много классов. Корень проблемы в том, что единственная ответственность «размазана» по большому числу классов. Это нарушает SRP: логически единая зона ответственности разорвана, и любое точечное изменение требует правок во множестве файлов. Рецепты: Move Method / Move Field — собрать связанное поведение в один класс; Inline Class — устранить ставший пустым класс.

#### Ответ от Claude

**Ключевое различие: направление и локализация изменений**

| Аспект | Divergent Change | Shotgun Surgery |
|---|---|---|
| Единица наблюдения | Один класс | Одно изменение бизнес-логики |
| Что происходит | Класс меняется по многим несвязанным причинам | Одно изменение затрагивает много разных классов |
| Метафора | «Класс на все случаи жизни» | «Выстрел дробью» по кодовой базе |
| Направление нарушения SRP | Слишком много ответственностей в одном классе | Одна ответственность расщеплена между многими классами |

**Divergent Change** — класс слишком «толстый». Добавляете новый тип продукта — меняете методы поиска, отображения и сортировки в одном и том же классе. Класс имеет слишком много причин для изменения. Нарушен **SRP** в смысле «один класс — одна причина для изменения»: класс содержит несвязанные ответственности и меняется при изменении любой из них.

**Shotgun Surgery** — ответственность слишком «тонкая» и распылённая. Меняете формат даты — правите `UserService`, `OrderService`, `ReportService`, `AuditLogger` и ещё три класса. Нарушен **SRP** в смысле «всё, что изменяется по одной причине, должно находиться вместе»: связанная логика разбросана по кодовой базе.

**Принципы SOLID:**

- Оба запаха нарушают **SRP**, но с разных сторон одного и того же принципа. SRP формулирует два требования одновременно: (1) не объединяй в классе то, что меняется по разным причинам, и (2) не разрывай то, что меняется по одной причине. Divergent Change нарушает требование (1), Shotgun Surgery — требование (2).
- Shotgun Surgery косвенно нарушает также **DRY** (знание о форматировании даты существует во многих местах) и указывает на отсутствие должной инкапсуляции.

**Мнемоника для различения:**
- Divergent Change: «один класс, много изменений» — смотришь на один файл и думаешь «зачем тут всё это?»
- Shotgun Surgery: «одно изменение, много классов» — смотришь на задачу и думаешь «где ещё надо поправить?»

**Решения:**
- Divergent Change → Extract Class: разбить класс на несколько с чёткими границами ответственности.
- Shotgun Surgery → Move Method / Move Field + Inline Class: собрать рассеянное поведение в один класс.

### Вопрос 29. YAGNI и OCP прямо противоречат друг другу: OCP призывает проектировать под расширение, YAGNI — не добавлять то, что не нужно сейчас. Как ты разрешаешь этот конфликт при проектировании?

**Категория:** SOLID/чистый код · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [OCP vs YAGNI — Enterprise Craftsmanship](https://enterprisecraftsmanship.com/posts/ocp-vs-yagni/)

Конфликт разрешается через вопрос «кто потребляет твой код?». YAGNI побеждает OCP в энтерпрайз-разработке, где ты контролируешь всех потребителей кода. Поскольку ты можешь отрефакторить код позже, когда реальные потребности возникнут, приоритизируй простоту сейчас. OCP побеждает YAGNI для публичных библиотек и фреймворков, где внешние команды зависят от твоего кода. После публичного релиза менять API становится непомерно дорого. Вывод: не закладывай точки расширения преждевременно во внутреннем коде; резервируй инвестиции в OCP для customer-facing API, сторонних библиотек и фундаментальных архитектурных решений.

#### Ответ от Claude

**Суть конфликта**

YAGNI (You Aren't Gonna Need It) говорит: «не пиши код для требований, которых ещё нет». OCP (Open/Closed Principle) говорит: «проектируй так, чтобы добавление новой функциональности не требовало изменения существующего кода». На первый взгляд OCP требует заложить абстракции заранее — то есть писать код «на будущее», что прямо противоречит YAGNI.

**Разрешение: контекст потребления определяет приоритет**

Два различных контекста требуют разных решений:

1. **Внутренний код (enterprise-приложение)**: вы — единственный потребитель, рефакторинг всегда возможен. Здесь **YAGNI важнее OCP**. Абстракции вводить только тогда, когда возникла реальная вариация (правило трёх: увидел три похожих случая — можно обобщить). Большинство «гибких точек расширения», которые закладываются заранее, в реальности никогда не используются.

2. **Публичный API / библиотека / SDK**: внешние потребители зависят от вашего кода, и вы не можете «просто отрефакторить» — ломающее изменение требует мажорной версии и миграции у всех пользователей. Здесь **OCP важнее YAGNI**. Точки расширения (через интерфейсы, abstract классы, стратегии) закладываются намеренно.

**Практические правила баланса**

- **Принцип «открытости по факту»**: сначала делай просто (YAGNI), а при втором или третьем расширении одного и того же места — вводи абстракцию (OCP). Не проектируй на трёхлетнее будущее.
- **Различай OCP как правило дизайна vs OCP как правило процесса**: не «нельзя менять существующий код никогда», а «при добавлении нового поведения хорошо бы не трогать проверенное». Это достижимо через рефакторинг-прежде-расширения, а не через заблаговременные абстракции.
- **«Seam points»**: целенаправленно оставляй «швы» (интерфейс вместо конкретного класса, внедрение зависимости вместо `new`) там, где смена реализации **достоверно возможна** (например, отправка уведомлений: email vs SMS — вот это реальная вариация). Если вариация гипотетическая — не вводи.
- **Стоимость абстракции**: каждая ненужная абстракция — это дополнительный файл, дополнительный уровень навигации, дополнительная ментальная нагрузка. Абстракция бесплатна только в теории.

**Итог**: конфликт мнимый, когда понимаешь, что OCP — не призыв проектировать под всё возможное будущее, а призыв не ломать проверенный код без причины. В энтерпрайзе YAGNI — основной принцип, OCP — локальный тактический приём там, где реальная вариативность уже проявилась.

### Вопрос 30. Как применение всех пяти принципов SOLID к небольшому классу может ухудшить код? Приведи конкретный пример чрезмерной абстракции («AbstractFactoryFactory»-проблема).

**Категория:** SOLID/чистый код · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Too Many Interfaces — blog.ttulka.com](https://blog.ttulka.com/too-many-interfaces/)

Автор утверждает, что само по себе использование интерфейсов не гарантирует хорошего дизайна, а при злоупотреблении лишь ухудшает его. Пример: класс `ProductImpl` реализует интерфейс `Product`, который содержит только геттеры — никакой реальной пользы от такой абстракции нет, зато сложность растёт. Другой пример: `ShoppingCart` зависит от `PriceCalculator`-интерфейса, реализованного единственным классом `PriceCalculatorImpl`. В результате между интерфейсом и реализацией возникает дополнительная связность, которая не убирает зависимость, а лишь маскирует её. Автор предлагает формулу: «программируй к абстракциям», а не «программируй к интерфейсам» — интерфейс не обязан быть Java-ключевым словом `interface`, а хорошая абстракция должна возникать из предметной области, а не навязываться ради соответствия принципам.

#### Ответ от Claude

**Как SOLID вредит маленькому классу: механизм**

Каждый принцип SOLID в изоляции разумен. Применённые механически все пять к простому классу, они вступают в резонанс и порождают «архитектурный взрыв»: резко растут количество файлов, уровни косвенности и когнитивная нагрузка при чтении кода.

**Конкретный пример: простой калькулятор скидок**

Исходная задача: «Применить скидку к заказу».

Простое решение (20 строк):
```java
public class OrderService {
    public BigDecimal applyDiscount(Order order, int discountPercent) {
        return order.getTotal()
                    .multiply(BigDecimal.valueOf(100 - discountPercent))
                    .divide(BigDecimal.valueOf(100));
    }
}
```

После механического применения всех пяти принципов SOLID:

```
IDiscountCalculator              <- ISP: отделили интерфейс
  AbstractDiscountCalculator     <- OCP: базовый класс для расширения
    PercentageDiscountCalculator <- реализация

IDiscountCalculatorFactory       <- DIP: не зависим от конкретного класса
  AbstractDiscountCalculatorFactory
    DiscountCalculatorFactory    <- SRP: фабрика отдельно от калькулятора

IOrderDiscountApplier            <- ISP снова: отделили применение скидки
  OrderDiscountApplier           <- LSP: реализует контракт

IDiscountValidationStrategy      <- OCP: стратегия валидации
  PositiveDiscountValidationStrategy

DiscountCalculatorFactoryProvider  <- теперь кто-то должен создавать фабрику
```

Итого: 1 метод → 10–12 классов/интерфейсов. Чтобы понять, как применяется скидка, разработчику нужно открыть 8 файлов и проследить цепочку зависимостей.

**«AbstractFactoryFactory» как апогей**

`AbstractFactoryFactory` — это метафора ситуации, когда абстракция создаётся над абстракцией без реальной необходимости. В Java-экосистеме это встречается в виде:

```java
// Реальный пример чрезмерного DIP + OCP
DataSourceFactory factory = DataSourceFactoryProvider
    .getInstance()
    .getFactory(DataSourceType.POSTGRES);
DataSource ds = factory.create(config);
```

Когда в проекте ровно одна БД и ни при каких обстоятельствах не планируется смена, весь этот код — чистый оверхед. Прямое `new PostgresDataSource(config)` было бы честнее.

**Почему это происходит: список нарушений**

- **SRP доведён до абсурда**: каждый класс содержит ровно один метод, что формально «чисто», но требует навигации по десяткам файлов для понимания одной операции. SRP про причины изменения, а не про размер.
- **OCP без реальной вариации**: заложена точка расширения для смены алгоритма скидок, хотя в продукте один алгоритм и он никогда не менялся.
- **LSP создаёт фиктивные иерархии**: наследование используется ради структуры, а не ради замещаемости.
- **ISP дробит интерфейсы** до состояния, когда каждый интерфейс содержит один метод, — и это перестаёт что-либо документировать.
- **DIP требует внедрения зависимостей** везде, включая utility-методы без состояния, где `new` абсолютно безвреден.

**Практические признаки «AbstractFactoryFactory»-проблемы**

1. Соотношение интерфейсов к реализациям близко 1:1 повсюду.
2. Для того чтобы добавить новое поле, нужно изменить 6+ файлов.
3. В репозитории есть классы с именами `*Factory`, `*Provider`, `*Builder`, `*Strategy` без единственной альтернативной реализации.
4. Новый разработчик не может понять «что делает этот код» без IDE и часа навигации.

**Правило здравого смысла**: применяй SOLID там, где сложность уже существует или достоверно возникнет. Не вводи абстракцию до того, как появятся хотя бы два разных случая, которые эта абстракция должна объединить. Принципы — инструменты управления сложностью, а не самоцель; инструмент, применённый к несуществующей проблеме, сам становится проблемой.
### Вопрос 31. java.util.Properties наследует Hashtable — это нарушение LSP в стандартной библиотеке Java. Объясни механизм нарушения и почему Sun/Oracle не исправили это в следующих версиях.

**Категория:** SOLID/чистый код · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Liskov Substitution Principle (LSP) — TU Darmstadt Lecture Notes](https://stg-tud.github.io/sedc/Lecture/ws13-14/3.3-LSP.html)

Лекционный материал приводит `java.util.Properties` как классический пример нарушения LSP прямо из стандартной библиотеки Java. Цитата из JavaDoc самого класса:

> "Because `Properties` inherits from `Hashtable`, the `put` and `putAll` methods can be applied to a `Properties` object. Their use is strongly discouraged as they allow the caller to insert entries whose keys or values are not Strings."

Механизм нарушения таков: `Hashtable` объявляет методы `put(Object key, Object value)` и `putAll(Map<?, ?> t)`, принимающие произвольные объекты. Класс `Properties` наследует эти методы, и код, работающий с переменной типа `Hashtable`, может вставить в неё нестроковые ключи или значения. Когда тот же объект передаётся туда, где ожидается `Properties`, вызов `store()`, `save()` или `getProperty()` упадёт с `ClassCastException` или вернёт `null` — поведение, которое клиентский код не ожидает от потомка `Hashtable`. Это разрушает фундаментальный контракт LSP: объект подкласса не может безопасно заменить объект суперкласса во всех контекстах.

#### Ответ от Claude

**Механизм нарушения LSP**

`java.util.Properties` объявляет семантический контракт «хранить только строковые ключи и значения» — об этом явно говорит документация, а интерфейс к строковым данным предоставляется через `setProperty(String key, String value)` и `getProperty(String key)`. Однако поскольку `Properties extends Hashtable<Object, Object>`, класс наследует сигнатуры:

```java
// Унаследовано от Hashtable — принимает произвольные типы
public Object put(Object key, Object value)
public void putAll(Map<? extends Object, ? extends Object> t)
public Object get(Object key)
```

LSP требует, чтобы подкласс S можно было подставить вместо суперкласса T без нарушения корректности программы. Здесь нарушение проявляется в нескольких формах:

1. **Нарушение инварианта подкласса через унаследованные методы.** Если клиент держит ссылку типа `Hashtable` и вызывает `put(42, new Object())`, а реальный объект — `Properties`, инвариант «все ключи и значения являются строками» будет сломан без какого-либо исключения. Последующий вызов `props.store(out, "comment")` завершится `ClassCastException`.

2. **Слабее postcondition у метода `getProperty`.** Метод `get(Object key)` из `Hashtable` может вернуть нестроковое значение, тогда как `getProperty(String key)` может вернуть `null` для того же ключа — различное поведение для одной и той же записи в зависимости от того, каким способом она была прочитана.

3. **Несовместимость поведения `entrySet()`.** Итерация через `entrySet()` из `Hashtable` может вернуть `Map.Entry<Object, Object>` с нестроковым содержимым, что нарушает ожидание любого кода, трактующего `Properties` как хранилище строк.

Корректным дизайном было бы использование **композиции вместо наследования**: `Properties` должна была содержать `Hashtable<String, String>` как приватное поле, а не расширять его.

**Почему Oracle не исправила это**

Причина одна — **обратная совместимость (backward compatibility)**:

1. **Легаси-код использует Object API.** До появления generics в Java 5 вполне допустимым (хотя и неправильным) было вызывать `properties.put(someObject, someObject)`. Такой код существует в продакшн-системах, и его смена типа сломала бы его.

2. **Невозможность безболезненного переключения на `Hashtable<String, String>`.** Если бы `Properties` стала расширять `Hashtable<String, String>`, методы `put(String, String)` перестали бы быть совместимы с унаследованным `put(Object, Object)` — это бинарная несовместимость, нарушающая один из главных контрактов Java-платформы («Write Once, Run Anywhere»).

3. **Официальная позиция в трекере OpenJDK.** Баг JDK-8157123 («java.util.Properties class is violating the Liskov's Substitution Principle») был закрыт без исправления именно по соображениям обратной совместимости. Изменение семантики класса, которому более 25 лет, несёт неприемлемые риски для экосистемы.

4. **Паллиативные меры вместо исправления.** Вместо рефакторинга архитектуры Oracle добавила предупреждение в Javadoc и инспекцию в IntelliJ IDEA (`UseOfPropertiesAsHashtable`), которая помечает вызовы `put()`/`get()` на объектах `Properties` как нежелательные и рекомендует использовать `setProperty()`/`getProperty()`.

**Урок для интервью:** этот пример демонстрирует, что нарушение LSP может быть встроено в дизайн с самого начала и оставаться «живым» годами из-за требований совместимости. В новом коде такой ситуации избегают через предпочтение композиции наследованию.

### Вопрос 32. Как OCP реализован в Spring Framework? Приведи конкретные примеры механизмов (BeanPostProcessor, HandlerInterceptor и т.п.) и объясни, почему именно такой дизайн выбран.

**Категория:** SOLID/чистый код · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Container Extension Points — Spring Framework Reference Documentation](https://docs.spring.io/spring-framework/reference/core/beans/factory-extension.html)

Официальная документация Spring описывает механизмы расширения контейнера как явные точки расширения (extension points), спроектированные так, чтобы разработчик мог изменить поведение контейнера без модификации его исходного кода.

**BeanPostProcessor** — интерфейс с двумя методами: `postProcessBeforeInitialization` и `postProcessAfterInitialization`. Контейнер автоматически обнаруживает и регистрирует все бины, реализующие этот интерфейс. Они получают колбэк до и после вызова методов инициализации каждого бина. Это позволяет добавлять поведение (проксирование, аудит, валидацию) ко всем бинам без изменения ни одной строки кода Spring.

**BeanFactoryPostProcessor** — интерфейс для модификации метаданных определений бинов (BeanDefinition) до того, как будет создан хотя бы один экземпляр. Встроенные реализации: `PropertySourcesPlaceholderConfigurer` (подстановка `${...}`-плейсхолдеров) и `PropertyOverrideConfigurer` (переопределение свойств из внешних файлов).

**FactoryBean** — интерфейс для инкапсуляции сложной логики создания объектов. Контейнер получает от него готовый объект, не зная деталей его создания.

#### Ответ от Claude

**Принцип OCP в применении к фреймворкам**

OCP гласит: программная сущность должна быть открыта для расширения, но закрыта для модификации. Применительно к фреймворку это означает: ядро фреймворка должно оставаться неизменным, а пользователи должны иметь возможность встраивать любое поведение через зафиксированные точки расширения.

Spring реализует этот принцип на нескольких уровнях:

**1. BeanPostProcessor — расширение жизненного цикла бинов**

```java
public interface BeanPostProcessor {
    Object postProcessBeforeInitialization(Object bean, String beanName);
    Object postProcessAfterInitialization(Object bean, String beanName);
}
```

Сам Spring использует этот механизм для реализации своих ключевых функций, не меняя класс `AbstractBeanFactory`:

| Реализация | Поведение |
|---|---|
| `AutowiredAnnotationBeanPostProcessor` | Обрабатывает `@Autowired` |
| `CommonAnnotationBeanPostProcessor` | Обрабатывает `@PostConstruct`, `@PreDestroy` |
| `AnnotationAwareAspectJAutoProxyCreator` | Создаёт AOP-прокси для `@Aspect` |
| `AsyncAnnotationBeanPostProcessor` | Включает поддержку `@Async` |

Разработчик добавляет собственную реализацию тем же способом — через `@Bean static` — и ядро контейнера не меняется.

**2. BeanFactoryPostProcessor — расширение метаданных конфигурации**

Позволяет изменять `BeanDefinition` (описания бинов) до создания экземпляров. Это единственная точка, где можно, например, подменить класс бина или изменить его scope декларативно. Работает на уровне метаданных, не экземпляров.

**3. HandlerInterceptor — расширение Web MVC без модификации DispatcherServlet**

```java
public interface HandlerInterceptor {
    boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler);
    void postHandle(HttpServletRequest request, HttpServletResponse response,
                    Object handler, ModelAndView modelAndView);
    void afterCompletion(HttpServletRequest request, HttpServletResponse response,
                         Object handler, Exception ex);
}
```

Документация Spring прямо формулирует OCP-цель: «Applications can register any number of existing or custom interceptors for certain groups of handlers, to add common preprocessing behavior **without needing to modify each handler implementation**.»

Встроенные реализации (`LocaleChangeInterceptor`, `WebContentInterceptor`, `UserRoleAuthorizationInterceptor`) существуют рядом с пользовательскими — все зарегистрированы декларативно, ни одна не меняет `DispatcherServlet`.

**4. ApplicationEvent / ApplicationListener — расширение через события**

Spring публикует встроенные события (`ContextRefreshedEvent`, `ContextClosedEvent`) через тот же механизм, что и пользовательские события. `ApplicationContext` закрыт для модификации, но открыт для подписки на любые события.

**5. Почему именно такой дизайн**

Альтернативой было бы предоставление богатого API для прямой манипуляции внутренним состоянием контейнера. Это дало бы больше гибкости, но породило бы хрупкость: любое изменение внутренней структуры сломало бы пользовательский код. Фиксированные интерфейсы-точки расширения позволяют Spring эволюционировать внутри, не нарушая внешние контракты. Это тот же аргумент, по которому методы ядра (`AbstractBeanFactory`, `DispatcherServlet`) часто объявлены `final` — явная «закрытость для модификации».

**Резюме для интервью:** OCP в Spring — это не один паттерн, а системная архитектурная политика: ядро реализует своё собственное поведение через те же extension points, что предоставляет пользователям, что исключает двойные стандарты и делает модель расширения надёжной.

### Вопрос 33. Cohesion и coupling — взаимосвязанные характеристики. Можно ли добиться высокой cohesion без снижения coupling? Какие паттерны проектирования намеренно идут на компромисс между ними?

**Категория:** SOLID/чистый код · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Managing Complexity with the Mediator and Facade Patterns — Embedded Artistry](https://embeddedartistry.com/blog/2020/07/06/managing-complexity-with-the-mediator-and-facade-patterns/)

Статья рассматривает паттерны Mediator и Facade именно через призму баланса cohesion/coupling.

**Mediator** снижает coupling между компонентами: они знают только о медиаторе, а не друг о друге. Но сложность не исчезает — она перемещается в медиатор, который рискует превратиться в «бога». Компромисс: каждый отдельный компонент становится более cohesive (занимается только своим делом), но медиатор получает высокий coupling со всеми участниками.

**Facade** создаёт единый интерфейс поверх подсистемы. Компоненты подсистемы могут оставаться сильно связанными между собой (высокий internal coupling) — Facade не устраняет эту связь, а лишь скрывает её от клиентов. Клиент получает низкий coupling с подсистемой, а сама подсистема может сохранять любую внутреннюю структуру.

Ключевое различие: Mediator знает обо всех участниках, а участники знают о медиаторе (двустороннее). Facade знает о подсистеме, а подсистема о Facade не знает (одностороннее).

#### Ответ от Claude

**Можно ли иметь высокую cohesion без низкого coupling?**

Да, и это важное наблюдение, которое часто упускают. Cohesion и coupling — ортогональные характеристики:

- **Cohesion** — насколько сфокусирован модуль внутри себя (все его элементы служат одной цели).
- **Coupling** — насколько сильно модуль зависит от других модулей.

Класс может быть highly cohesive (делать ровно одну вещь) и при этом tightly coupled (зависеть от десяти конкретных реализаций). Пример: сервис `OrderFulfillmentService`, который занимается исключительно выполнением заказов (высокая cohesion), но прямо инстанцирует `InventoryRepository`, `PaymentGateway`, `EmailNotifier` и `AuditLogger` (высокий coupling). Снижение coupling здесь достигается отдельным приёмом — dependency injection или DIP.

Другой пример «высокая cohesion + высокий coupling»: монолит с тщательно выделенными слоями. Каждый слой внутри себя связен, но слои жёстко сцеплены через прямые вызовы.

**Паттерны, намеренно идущие на компромисс**

**1. Mediator — снижает coupling компонентов, создавая highly coupled медиатор**

Компоненты (`Colleague` объекты) теряют прямые связи между собой, их cohesion растёт — каждый занимается только своей работой. Но медиатор получает coupling со всеми компонентами. Если медиатор становится «богом» (god object), система превращает распределённую сложность в централизованную. Это намеренный компромисс: управлять сложностью в одном месте удобнее, чем отслеживать её по всему графу зависимостей.

```
До:  A <-> B <-> C <-> D (много связей, decentralized)
После: A -> M <- B, C -> M <- D (всё через медиатор, M — highly coupled)
```

**2. Facade — скрывает высокий coupling внутри подсистемы**

Facade не устраняет coupling внутри подсистемы. `VideoConversionFacade` может внутри жёстко связывать `VideoFile`, `Codec`, `ColorSpace`, `BitrateReader`, `AudioMixer`. Клиент видит один метод `convert(file, format)` — его coupling низкий. Но если подсистема меняется, Facade нужно обновлять. Паттерн явно принимает «внутренний coupling — внешняя простота».

**3. Observer — снижает coupling, но усложняет понимание потока управления**

Observer уменьшает coupling между субъектом и наблюдателями. Но: в крупных системах высокий fanout (одно событие — сотни подписчиков) делает систему трудно предсказуемой. Cohesion каждого отдельного обработчика высока, но «системная cohesion» (понимание, что происходит при событии) падает. Это компромисс между coupling и observability.

**4. Service Locator — высокая cohesion, скрытый coupling**

Паттерн (считающийся антипаттерном в современной практике) позволяет классу оставаться cohesive, скрывая зависимости за локатором. Но coupling не исчезает — он становится неявным, что хуже явного.

**5. Event-driven architecture / Microservices**

На архитектурном уровне: сервисы стремятся к high cohesion (bounded context, DDD) и low coupling (асинхронные события). Но на практике высокая cohesion модели домена внутри сервиса часто означает, что этот сервис становится точкой притяжения функциональности, и coupling с ним со временем растёт.

**Вывод для интервью:** «высокая cohesion + низкий coupling» — желаемый идеал, но не математическая зависимость. Паттерны Mediator и Facade явно перераспределяют coupling, а не уничтожают его. Observer прячет coupling за события, теряя прозрачность. Осознанный дизайн — это не «убрать coupling», а «переместить его туда, где им проще управлять».

### Вопрос 34. При рефакторинге «God Class» в несколько небольших классов производительность системы деградировала из-за роста числа объектов и вызовов. Как сбалансировать чистоту кода и производительность? Какие техники помогают?

**Категория:** SOLID/чистый код · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [How to Refactor a God Class: Architectural Decomposition and Dependency Control — IN-COM Data Systems](https://www.in-com.com/blog/how-to-refactor-a-god-class-architectural-decomposition-and-dependency-control/)

Статья описывает God Class как компонент, который централизует в себе несколько ответственностей — бизнес-логику, представление и работу с данными — в одной структуре. При рефакторинге такого класса рекомендуется:

- Обнаруживать проблемные классы с помощью статического анализа по метрикам WMC (Weighted Methods per Class), CBO (Coupling Between Objects) и цикломатической сложности.
- Выделять связные поддомены внутри класса и извлекать независимые модули с чёткими интерфейсами.
- Вводить интерфейсы, которые описывают поведение, не раскрывая реализацию (Dependency Inversion + Interface Segregation).
- Перестраивать потоки данных между новыми компонентами и внедрять слоистую архитектуру для устранения циклических зависимостей.

Относительно производительности: статья указывает, что сама по себе разбивка God Class снижает время перекомпиляции (компилятор больше не перекомпилирует весь граф зависимостей при изменении одного метода) и ускоряет юнит-тестирование. Прямого анализа деградации рантайм-производительности из-за роста числа объектов и делегирующих вызовов статья не даёт, сосредоточившись на операционных метриках: MTTR, скорости деплоя и объёме регрессионного тестирования.

#### Ответ от Claude

**Почему производительность может деградировать после рефакторинга God Class**

Когда единый класс разбивается на десятки маленьких, возникает несколько источников накладных расходов:

1. **Рост числа объектов и давление на GC.** Каждый новый объект занимает память в куче; при высокой частоте создания/уничтожения коротко живущих объектов увеличивается нагрузка на Garbage Collector. JVM HotSpot использует TLAB (Thread-Local Allocation Buffers), что делает само выделение памяти очень дешёвым, однако сбор мусора всё равно требует пауз.

2. **Цепочки делегирующих вызовов.** Там, где раньше был один метод, теперь может быть 3–5 методов в разных классах. Каждый виртуальный вызов (`invokevirtual`) несёт накладные расходы на поиск в таблице диспетчеризации (vtable). JIT-компилятор HotSpot умеет инлайнить небольшие методы, убирая эти накладные расходы, но только после достаточного прогрева (порог — обычно 10 000 вызовов для `-client` и аналогично для `-server`).

3. **Cache miss.** God Class, несмотря на концептуальную «грязность», концентрировала данные в одном объекте. После разбивки связанные данные могут оказаться в разных объектах в разных областях кучи, что снижает локальность кэша L1/L2 процессора.

**Как измерить, а не предполагать деградацию**

Перед оптимизацией — профилировать:
- `async-profiler` или `JFR` (Java Flight Recorder) для поиска горячих точек.
- `-XX:+PrintCompilation` и JITWatch для анализа, какие методы инлайнятся, а какие — нет.
- `jmap` / `jcmd GC.heap_info` для анализа давления на GC.

**Техники балансировки чистоты кода и производительности**

| Проблема | Техника | Пояснение |
|---|---|---|
| Накладные расходы на создание объектов | **Flyweight / Object Pool** | Переиспользовать неизменяемые или дорогостоящие объекты через фабрику; `java.util.concurrent` предлагает `ArrayBlockingQueue` для пулинга |
| Цепочки делегирующих вызовов | **Доверять JIT** + делать методы `final` или `private` | JIT агрессивнее инлайнит методы, которые не переопределяются; избегать глубины цепочек > 5–6 уровней |
| Избыточное создание временных объектов | **Value Objects / Records (Java 16+)** | Компилятор и JIT оптимизируют аллокацию на стеке для мелких record-типов (scalar replacement) |
| Снижение локальности кэша | **Data-Oriented Design**: держать связанные данные в массивах, а не в объектах | Для горячих путей — рассмотреть `int[]` / `long[]` вместо `List<SomeValueObject>` |
| «Тонкие» служебные методы | **Паттерн Facade поверх новых классов** | Старый God Class превращается в фасад, делегирующий новым классам; после прогрева JIT инлайнит вызовы |
| Статические утилиты | Вынести в `static final` методы утилитного класса | Статические вызовы (`invokestatic`) дешевле виртуальных |

**Ключевые принципы принятия решений**

- **Measure first, optimize second.** Деградацию нужно подтвердить бенчмарком (JMH), а не интуицией.
- **Не жертвовать читаемостью ради гипотетической производительности.** JIT HotSpot очень хорошо оптимизирует чистый, предсказуемый код. Код, насыщенный ручными микро-оптимизациями, JIT оптимизирует хуже, чем прямолинейный объектно-ориентированный код.
- **Горячий путь vs. холодный путь.** Оптимизировать имеет смысл только «горячие» части (те самые 20% кода, которые выполняются 80% времени). Новые небольшие классы, вызываемые редко, не нуждаются в оптимизации.
- **Исследование Researchgate** (Analyzing the Harmful Effect of God Class Refactoring on Power Consumption) показало, что использование наследования вместо делегирования при рефакторинге позволяет снизить время выполнения на 77% и потребление энергии на 4%. Это означает, что выбор конкретной техники рефакторинга (наследование vs. агрегация) влияет на производительность и должен приниматься осознанно.

**Практический рецепт**

1. Разбить God Class на логические компоненты (не жертвовать SRP).
2. Оставить тонкий Facade на месте старого класса — JIT его инлайнит.
3. Запустить JMH-бенчмарк и `async-profiler` на реалистичной нагрузке.
4. Если GC-давление высокое — применить Object Pool или Records для Value Objects.
5. Если виртуальные вызовы не инлайнятся — использовать `final` классы/методы или заменить интерфейс на прямую зависимость в горячем пути.

### Вопрос 35. Объясни технику «Strangler Fig» для поэтапного рефакторинга легаси-кода. Как она связана с OCP и как избежать долгосрочного сосуществования старого и нового кода?

**Категория:** SOLID/чистый код · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Refactoring Legacy Code with the Strangler Fig Pattern — Shopify Engineering](https://shopify.engineering/refactoring-legacy-code-strangler-fig-pattern)

Shopify применил паттерн Strangler Fig для рефакторинга модели `Shop` — God Object объёмом более 3000 строк кода. Команда извлекла настройки хранилища (функционал `Capital::SettingsToLock`) в отдельный класс через семишаговый процесс.

**Связь с OCP.** Статья явно OCP не упоминает, однако паттерн выстраивает новый интерфейс рядом с легаси-системой, не изменяя существующий код `Shop` напрямую, что соответствует принципу «открыт для расширения, закрыт для изменения».

**Как Shopify избегает долгосрочного сосуществования:**

1. **Параллельная запись в транзакции.** Запись происходит одновременно в старый и новый источник данных внутри одной транзакции — гарантируется консистентность.
2. **Пессимистические блокировки** при backfill-задаче: блокируется объект `Shop` для предотвращения гонок при параллельных операциях.
3. **Мониторинг и логирование.** Команда использовала подробное логирование для диагностики сбоев персистентности в процессе миграции.
4. **Чёткий критерий завершения.** Как только все читатели переключились на новый источник и backfill завершился, легаси-код немедленно удалялся: «we remove the now-unused, 'strangled' code from the codebase».

#### Ответ от Claude

**Что такое паттерн Strangler Fig**

Термин ввёл Мартин Фаулер в 2004 году в [bliki: StranglerFigApplication](https://martinfowler.com/bliki/StranglerFigApplication.html), взяв метафору из ботаники: фиговое дерево-душитель прорастает в крону хозяина, постепенно опутывает его корнями, которые достигают земли, и в конце концов хозяин отмирает, а новое дерево стоит самостоятельно.

В программировании: **новая система строится вокруг легаси-системы, перехватывая всё больше функциональности по одному срезу за раз, пока старая система не «задушена» и не выведена из эксплуатации**.

**Механизм работы**

```
Клиент
  |
  v
[Facade / Router / API Gateway]  <-- единственная точка входа
  |                |
  v                v
[Legacy System]  [New Service(s)]
```

Facade (или HTTP-прокси, или API Gateway) перехватывает входящий запрос и маршрутизирует его либо в легаси, либо в новый сервис в зависимости от того, переведена ли данная функциональность.

**Четыре ключевые активности (по Фаулеру)**

1. Прояснить желаемые результаты — согласовать, что именно должна дать модернизация.
2. Декомпозировать проблему — найти «швы» (seams), по которым систему можно разрезать.
3. Поставлять инкрементально — заменять маленькие компоненты с ранним возвратом ценности.
4. Обеспечить эволюцию команды — менять практики разработки параллельно с кодом.

**Связь с принципом OCP (Open/Closed Principle)**

OCP гласит: программные сущности должны быть открыты для расширения, но закрыты для модификации. Strangler Fig реализует эту идею на уровне системной архитектуры:

- **Закрыто для модификации:** легаси-код в процессе перехода **не трогается** — это снижает риск регрессий.
- **Открыто для расширения:** новая функциональность добавляется как отдельные сервисы/модули, не ломая существующее поведение.
- **Facade как точка расширения:** маршрутизирующий слой позволяет добавлять новые реализации без изменения ни легаси, ни клиентского кода.

Таким образом, Strangler Fig — это OCP, применённый не к одному классу, а к целой системе на временном горизонте нескольких месяцев или лет.

**Как избежать долгосрочного сосуществования старого и нового кода**

Это главная операционная опасность паттерна — «временная» архитектура становится постоянной. Практические меры:

| Риск | Контрмера |
|---|---|
| Нет чёткого критерия завершения | Определить Definition of Done для каждого среза: 100% трафика на новый сервис + backfill завершён + мониторинг стабилен N дней |
| Потеря импульса после первых срезов | Roadmap с фиксированными дедлайнами для каждого следующего среза; видимость прогресса в команде |
| Двойная запись становится постоянной | Устанавливать таймер (например, 2–4 спринта): после переключения чтения — выключить запись в легаси |
| Facade превращается в новый God Object | Facade должен быть тонким (только маршрутизация), не накапливать логику |
| Внешние зависимости от легаси-API | Версионирование API: v1 (легаси) → v2 (новый). После переключения клиентов v1 объявляется deprecated с явным дедлайном удаления |

**Staged traffic migration (из практики Shopify/GoCodeo)**

```
Шаг 1: 0% нового   → 100% легаси   (Facade установлен)
Шаг 2: 10% нового  → 90% легаси    (canary)
Шаг 3: 50% нового  → 50% легаси    (majority test)
Шаг 4: 100% нового → 0% легаси     (cutover)
Шаг 5: Легаси-код удалён            (cleanup)
```

Шаг 5 — обязательный и должен стоять в sprint backlog сразу, иначе технический долг кристаллизуется.

**Когда Strangler Fig неприменим**

- Монолит с deeply coupled database (одна схема на всё): нужна предварительная декомпозиция данных (database-per-service).
- Когда легаси не имеет тестов и поведение неизвестно: сначала добавить characterization tests (Golden Master), потом стрэнглить.
- Когда дедлайн критически короткий: большой рефакторинг за один раз может быть быстрее (но рискованнее).

### Вопрос 36. SRP применяется не только к классам, но и к модулям, микросервисам, методам. Как ты определяешь «одну причину для изменения» на уровне микросервиса? Чем это отличается от определения bounded context в DDD?

**Категория:** SOLID/чистый код · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [SRP: The Most Important Rule in the Software World — DZone](https://dzone.com/articles/the-most-important-rule-in-software)

Статья рассматривает SRP как универсальный принцип, применимый не только к классам, но и к сервисам и системам любого масштаба. Ключевые тезисы:

**Переопределение Мартина через «актора»:** Роберт Мартин уточнил определение SRP: «Модуль должен быть ответственен перед одним и только одним актором» — группой людей или стейкхолдеров, заинтересованных в его изменении. На уровне сервиса это означает: каждый сервис обслуживает интересы одной бизнес-группы.

**Связь с Bounded Context:** «Если мы решим рассматривать Bounded Context как модуль, мы быстро заметим, что по своим определениям он полностью соответствует SRP». Оба понятия устанавливают чёткие границы бизнес-владения без зависимостей между доменами.

**Ключевое различие:** Bounded Context может содержать внутри себя несколько компонентов (Entities, Aggregates), при этом каждый компонент сам моделирует единственный доменный концепт. То есть Bounded Context — это граница контейнера, а SRP — правило для каждого компонента внутри.

#### Ответ от Claude

**SRP на уровне микросервиса: как определить «одну причину для изменения»**

На уровне класса SRP формулируется просто — «один класс, одна ответственность». На уровне микросервиса это определение нужно операционализировать, потому что «ответственность» сервиса принципиально шире ответственности класса.

**Рабочее определение:** Микросервис имеет одну причину для изменения, если все его изменения инициируются **одним и тем же бизнес-актором** (командой, продуктовым менеджером, группой стейкхолдеров) и затрагивают **одну бизнес-способность** (business capability).

**Практические вопросы для проверки границ сервиса:**

1. *Кто является заказчиком изменений в этом сервисе?* Если ответ «и команда платежей, и команда логистики» — сервис нарушает SRP.
2. *Какая бизнес-событие (domain event) служит основным поводом для деплоя?* Один сервис — один класс domain events.
3. *Если меняется бизнес-правило X, нужно ли менять этот сервис?* Если X не относится к домену сервиса — ответ должен быть «нет».
4. *Может ли сервис существовать независимо, если другие сервисы заморожены?* Связность через общую бизнес-логику нарушает SRP.

**Масштабирование SRP по уровням:**

| Уровень | «Одна причина для изменения» | Инструмент выявления |
|---|---|---|
| Метод | Одна операция / шаг алгоритма | Code review, цикломатическая сложность |
| Класс | Одна ответственность перед одним актором | SRP, LCOM-метрика |
| Модуль / пакет | Одна бизнес-функция (Common Closure Principle) | Package coupling metrics |
| Микросервис | Одна бизнес-способность, один стейкхолдер-актор | Event Storming, Conway's Law |
| Команда / bounded context | Одна субдомен DDD | Domain-Driven Design |

**Чем SRP отличается от Bounded Context в DDD**

Это центральный концептуальный вопрос, и разница здесь нетривиальная:

| Критерий | SRP | Bounded Context |
|---|---|---|
| **Происхождение** | Принцип проектирования (Роберт Мартин, SOLID) | Стратегический паттерн (Эрик Эванс, DDD) |
| **Уровень анализа** | Техническая единица (класс, модуль, сервис) | Бизнес-единица с общим языком (Ubiquitous Language) |
| **Что определяет границу** | Причина изменения (актор/стейкхолдер) | Семантическая согласованность доменных понятий и язык |
| **Направление анализа** | Снизу вверх: «что меняется вместе?» | Сверху вниз: «где одно понятие означает разное?» |
| **Содержимое** | Один класс / модуль / сервис | Может содержать несколько сервисов, агрегатов, сущностей |
| **Цель** | Минимизировать случайные изменения | Изолировать семантические конфликты (полисемия) |
| **Пример** | Сервис `OrderService` меняется только при изменении логики заказов | Bounded Context «Ordering» содержит `Order`, `OrderLine`, `Shipping` |

**Ключевое различие на практике:**

- **SRP** задаёт вопрос: *«Должен ли этот элемент знать об этом изменении?»* — это вопрос связанности по изменению.
- **Bounded Context** задаёт вопрос: *«Означает ли слово `Customer` одно и то же для отдела продаж и для склада?»* — это вопрос семантики и Ubiquitous Language.

Пример: в интернет-магазине слово `Product` в контексте «Каталог» означает описание товара с фотографиями и SEO-тегами, а в контексте «Склад» означает физическую единицу с весом и ячейкой хранения. DDD предписывает разделить их на два Bounded Context. SRP предписывает не смешивать логику публикации каталога и логику складского учёта в одном сервисе — по совпадающей, но отдельной причине.

**Как они дополняют друг друга**

На практике при проектировании микросервисов:

1. **DDD** даёт карту субдоменов и Bounded Contexts — это стратегический ориентир для декомпозиции.
2. **SRP** применяется внутри каждого Bounded Context для дополнительного разбиения: не каждый Bounded Context — один микросервис; иногда один BC содержит 2–3 сервиса, разделённых по SRP (например, «чтение» и «запись» в CQRS).
3. **Common Closure Principle** (CCP — части, изменяющиеся вместе, должны находиться вместе) — промежуточный принцип, который объединяет SRP и Bounded Context: классы, меняющиеся по одной причине, должны быть в одном модуле/сервисе.

**Антипаттерны нарушения SRP на уровне микросервиса:**

- «Нано-сервисы»: сервис настолько мал, что каждое бизнес-изменение требует изменений в 5+ сервисах одновременно — это нарушение CCP, смежного с SRP.
- «Мега-сервис»: сервис изменяется по инициативе трёх разных команд — явное нарушение SRP.
- Shared database: два сервиса пишут в одну таблицу — границы SRP обходятся на уровне данных.
### Вопрос 37. Что такое Закон Деметры (Law of Demeter / Principle of Least Knowledge)? Чем «цепочки вызовов» (train wreck) нарушают этот принцип — и почему Fluent API (Stream API, Builder) является допустимым исключением?

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Law of Demeter and How to Work With It — programmingideaswithjake.wordpress.com](https://programmingideaswithjake.wordpress.com/2015/02/28/law-of-demeter-and-how-to-work-with-it/)

Закон Деметры имеет почти математическое определение: «Любой метод `m` объекта `O` может вызывать методы только следующих категорий объектов: самого `O`, параметров `m`, объектов, созданных внутри `m`, прямых компонентов `O` (полей), а также глобальных переменных, доступных `O`».

По существу, принцип запрещает вызывать методы на объектах, возвращённых другими методами, — то есть исключает цепочки вида `a.getB().getC().doSomething()`. Это снижает связанность (coupling) и усиливает инкапсуляцию.

**Антипаттерн «train wreck» (крушение поезда):**
```java
// НАРУШЕНИЕ — каждый «вагон» — чужой объект
ReturnValue result = parameter.getA();
result.getB().doWork();
```

Или в одну строку:
```java
xml.getXML().getBooks().getBookArray(0).getBookHeader().getBookCategory();
```
Здесь код завязан сразу на пять типов: `XMLMessage`, `XML`, `Book`, `BookHeader`, `BookCategory`. Изменение любого из них ломает вызывающий код.

**Допустимое исключение — Fluent API:**
Когда API специально разработан для цепочек вызовов, нет смысла жертвовать читаемостью ради строгого соблюдения закона. Например, `Stream API` в Java 8 был бы бесполезен, если бы не разрешалось цепочечно вызывать методы. Ключевое различие: методы Fluent API возвращают *тот же самый объект* (или объект того же «конвейера»), а не чужие объекты из глубины иерархии.

#### Ответ от Claude

**Определение и история.**
Закон Деметры (LoD) сформулирован Яном Холландом в 1987 году в Northeastern University в рамках проекта Demeter. Также называется «Принципом наименьшего знания» (Principle of Least Knowledge). Цель — минимизировать осведомлённость объекта о структуре других объектов, то есть снизить связанность.

**Четыре разрешённые категории обращений (для метода `m` объекта `O`):**
1. Методы самого `O` (`this.someMethod()`).
2. Методы объектов, переданных как параметры `m`.
3. Методы объектов, созданных *внутри* `m` (`new Foo().bar()`).
4. Методы прямых полей `O` (компонентов объекта).

Запрещено: вызывать методы объектов, возвращённых геттерами или промежуточными методами — «не разговаривай с незнакомцами».

**Пример нарушения («train wreck»):**
```java
// НАРУШЕНИЕ — проходим сквозь несколько чужих объектов
public void process(Order order) {
    // order — параметр, OK
    // order.getMessage() — OK (прямой вызов на параметре)
    // .normalize() на возвращённом объекте — НАРУШЕНИЕ
    order.getMessage().normalize();

    // Ещё хуже — длинная цепочка через чужие объекты:
    String city = order.getCustomer().getAddress().getCity();
}
```

Проблема: `process` знает о существовании `Customer` и `Address`, хотя работает с `Order`. При изменении структуры `Customer` или `Address` придётся менять `process`.

**Правильный подход — делегирование:**
```java
// В классе Order:
public String getCustomerCity() {
    return customer.getCity(); // Order сам знает Customer
}

// В классе Customer:
public String getCity() {
    return address.getCity(); // Customer сам знает Address
}

// Вызывающий код:
String city = order.getCustomerCity(); // одна точка
```

**Fluent API — допустимое исключение.**
Есть два принципиально разных вида цепочек:

| Вид цепочки | Пример | Нарушение LoD? |
|---|---|---|
| Переход по чужим объектам | `order.getCustomer().getAddress().getCity()` | Да |
| Тот же объект / конвейер | `stream.filter(...).map(...).collect(...)` | Нет |

В Stream API каждый метод возвращает новый `Stream` — объект того же «конвейера обработки», созданного локально. В Builder (`new Person.Builder().name("Ivan").age(30).build()`) все вызовы идут на *один и тот же* объект-строитель, созданный локально. Вызывающий код не «проваливается» в чужую иерархию зависимостей, поэтому смысловая связанность не увеличивается.

**Связь с другими принципами.**
LoD тесно связан с принципом «Tell, Don't Ask» (не запрашивай чужое состояние, чтобы потом принимать решение) и косвенно поддерживает SRP и OCP: объект, не знающий о внутренней структуре соседей, проще изменять и тестировать.

### Вопрос 38. Почему «композиция предпочтительнее наследования» (composition over inheritance)? Объясните проблему хрупкого базового класса (Fragile Base Class Problem) на конкретном Java-примере и покажите, как её решает композиция.

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Why favor Composition over Inheritance in Java and OOP — javarevisited.blogspot.com](https://javarevisited.blogspot.com/2013/06/why-favor-composition-over-inheritance-java-oops-design.html)

Главный недостаток наследования — оно нарушает инкапсуляцию. Подкласс зависит от деталей реализации суперкласса.

**Классический пример с `HashSet`.**
Допустим, нужен `HashSet`, который подсчитывает количество добавленных элементов. Через наследование:

```java
public class InstrumentedHashSet<E> extends HashSet<E> {
    private int addCount = 0;

    @Override
    public boolean add(E e) {
        addCount++;
        return super.add(e);
    }

    @Override
    public boolean addAll(Collection<? extends E> c) {
        addCount += c.size();
        return super.addAll(c);
    }

    public int getAddCount() { return addCount; }
}
```

Использование:
```java
InstrumentedHashSet<String> s = new InstrumentedHashSet<>();
s.addAll(List.of("a", "b", "c"));
System.out.println(s.getAddCount()); // ожидаем 3, получаем 6!
```

Почему 6? Потому что `HashSet.addAll()` внутри вызывает `add()` для каждого элемента. Наш `addAll()` прибавил 3, затем через `super.addAll()` для каждого элемента вызвался наш переопределённый `add()`, прибавив ещё 3. Это деталь реализации суперкласса, не задокументированная в контракте.

**Решение через композицию:**
```java
public class InstrumentedHashSet<E> implements Set<E> {
    private final HashSet<E> container = new HashSet<>();
    private int addCount = 0;

    public boolean add(E e) {
        addCount++;
        return container.add(e);
    }

    public boolean addAll(Collection<? extends E> c) {
        addCount += c.size();
        return container.addAll(c);
    }

    public int getAddCount() { return addCount; }
    // ... делегирование остальных методов Set ...
}
```

Теперь `addAll()` вызывает `container.addAll()` напрямую (не наш `add()`), поэтому `addCount` увеличивается ровно на `c.size()`. Изменение реализации `HashSet` не влияет на поведение класса.

#### Ответ от Claude

**Проблема хрупкого базового класса (Fragile Base Class Problem).**
Это фундаментальная проблема ООП: «безопасные» на первый взгляд изменения суперкласса могут сломать подклассы. Программист не может оценить безопасность изменения базового класса, не изучив все его подклассы — иногда это невозможно (публичная библиотека).

**Механизм поломки — динамическая диспетчеризация.**

Рассмотрим пример из Wikipedia (Fragile base class):

```java
class Base {
    public void inc1() { this.inc2(); } // после рефакторинга вызывает inc2
    public void inc2() { /* ... */ }
}

class Derived extends Base {
    @Override
    public void inc2() {
        super.inc1(); // переопределение вызывает inc1 суперкласса
    }
}
```

После того как разработчик суперкласса «безобидно» изменил `inc1()` так, чтобы она вызывала `inc2()`, вызов `new Derived().inc2()` порождает бесконечную рекурсию и `StackOverflowError`. Подкласс не менялся — сломал его суперкласс.

**Ещё один классический пример (из Effective Java, Joshua Bloch):**

Задача: подсчитать, сколько элементов всего было добавлено в `HashSet`.

*Наследование (хрупкое):*
```java
// ПРОБЛЕМА: зависимость от внутренней реализации HashSet
public class CountingHashSet<E> extends HashSet<E> {
    private int addCount = 0;

    @Override
    public boolean add(E e) {
        addCount++;
        return super.add(e);
    }

    @Override
    public boolean addAll(Collection<? extends E> c) {
        addCount += c.size();
        return super.addAll(c); // HashSet.addAll вызывает this.add()!
    }
}

CountingHashSet<String> s = new CountingHashSet<>();
s.addAll(List.of("a", "b", "c"));
// addCount == 6, а не 3 — двойной учёт из-за деталей реализации
```

*Композиция (надёжное):*
```java
// РЕШЕНИЕ: оборачиваем HashSet, не наследуем его
public class CountingSet<E> implements Set<E> {
    private final Set<E> delegate = new HashSet<>();
    private int addCount = 0;

    @Override
    public boolean add(E e) {
        addCount++;
        return delegate.add(e);
    }

    @Override
    public boolean addAll(Collection<? extends E> c) {
        addCount += c.size();
        return delegate.addAll(c); // вызывает метод HashSet, не наш add()
    }

    public int getAddCount() { return addCount; }

    // Все остальные методы Set делегируются:
    @Override public int size() { return delegate.size(); }
    @Override public boolean contains(Object o) { return delegate.contains(o); }
    // ... и т.д.
}

CountingSet<String> s = new CountingSet<>();
s.addAll(List.of("a", "b", "c"));
// addCount == 3 — корректно при любой реализации HashSet
```

**Почему композиция решает проблему.**

При композиции класс взаимодействует с суперклассом через контракт (интерфейс `Set`), а не через детали реализации. Если `HashSet` изменит внутренний алгоритм `addAll()`, это не затронет `CountingSet`, потому что наш `addAll()` вызывает `delegate.addAll()` напрямую — не через динамическую диспетчеризацию на `this`.

**Когда наследование оправдано.**
Наследование подходит, когда соблюдается условие «является» (is-a), суперкласс проектировался с учётом расширения (документированные точки переопределения), и оба класса находятся в одном пакете или под контролем одной команды. Joshua Bloch в *Effective Java* формулирует правило: «Проектируй и документируй наследование — или запрещай его» (`final`-класс).

**Дополнительные преимущества композиции:**
- Возможность подменить реализацию в рантайме (стратегия).
- Лёгкость тестирования: зависимость подставляется через конструктор (mock).
- Отсутствие проблемы «ромбовидного наследования».

### Вопрос 39. Что такое принцип «Tell, Don't Ask»? Как он связан с инкапсуляцией и нарушением принципов SOLID? Приведите Java-пример кода, нарушающего этот принцип, и покажите рефакторинг.

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Tell Don't Ask — martinfowler.com](https://martinfowler.com/bliki/TellDontAsk.html)

«Tell-Don't-Ask» — это принцип, помогающий помнить, что объектно-ориентированное программирование связано с объединением данных и функций, которые ими оперируют. Вместо того чтобы запрашивать у объекта данные и принимать решение снаружи, нужно *сказать* объекту, что делать.

**Пример нарушения (Ask-стиль):**
```java
AskMonitor am = new AskMonitor("Time Vortex Hocus", 2, alarm);
am.setValue(3);
if (am.getValue() > am.getLimit()) {
    am.getAlarm().warn(am.getName() + " too high");
}
```

Внешний код запрашивает состояние монитора (`getValue()`, `getLimit()`) и сам принимает решение о срабатывании тревоги, хотя эта логика должна находиться внутри объекта.

**Рефакторинг (Tell-стиль):**
```java
TellMonitor tm = new TellMonitor("Time Vortex Hocus", 2, alarm);
tm.setValue(3); // объект сам проверяет порог и вызывает тревогу
```

Логика проверки переносится внутрь `setValue()`:
```java
public void setValue(int value) {
    this.value = value;
    if (this.value > this.limit) {
        alarm.warn(this.name + " too high");
    }
}
```

Мартин Фаулер оговаривает: принцип не следует применять догматично — иногда методы-запросы улучшают читаемость и не несут вреда.

#### Ответ от Claude

**Определение.**
«Tell, Don't Ask» (TDA) — принцип проектирования, согласно которому клиентский код должен *командовать* объекту выполнить действие, а не *запрашивать* его внутреннее состояние, принимать решение на основе этого состояния и затем снова командовать. Данные и поведение, которое на них завязано, должны находиться вместе.

**Нарушение принципа на Java-примере.**

```java
// Класс Circle с геттерами — нарушение TDA
public class Circle {
    private double x, y, r;

    public double getX() { return x; }
    public double getY() { return y; }
    public double getR() { return r; }
}

// Клиентский код — "Ask"-стиль
public boolean isPointInCircle(Circle c, double px, double py) {
    double cx = c.getX(); // запрашиваем состояние
    double cy = c.getY();
    double r  = c.getR();
    // принимаем решение снаружи — логика, которая принадлежит Circle
    return (px - cx) * (px - cx) + (py - cy) * (py - cy) <= r * r;
}
```

Проблемы:
1. **Нарушение инкапсуляции**: клиент знает внутреннюю структуру (`x`, `y`, `r`). Если `Circle` перейдёт на полярные координаты, сломается весь клиентский код.
2. **Нарушение SRP**: логика принадлежности точки окружности размазана по клиентам, а не сосредоточена в `Circle`.
3. **Нарушение OCP**: при добавлении нового типа фигуры клиентский код потребует изменения.

**Рефакторинг (Tell-стиль):**

```java
public class Circle {
    private double x, y, r;

    public Circle(double x, double y, double r) {
        this.x = x; this.y = y; this.r = r;
    }

    // Поведение, завязанное на данные, — внутри класса
    public boolean contains(double px, double py) {
        return (px - x) * (px - x) + (py - y) * (py - y) <= r * r;
    }
}

// Клиентский код — Tell-стиль
boolean inside = circle.contains(px, py); // просто командуем
```

Теперь клиент не знает, как именно `Circle` хранит данные. Изменение внутреннего представления затронет только метод `contains()`.

**Ещё один пример: мониторинг температуры.**

```java
// НАРУШЕНИЕ TDA
public class TemperatureSensor {
    private double temperature;
    public double getTemperature() { return temperature; }
}

// Клиент принимает решение
TemperatureSensor sensor = ...;
if (sensor.getTemperature() > 100.0) {
    alertSystem.sendAlert("Overheating!");
}
```

```java
// РЕФАКТОРИНГ — Tell
public class TemperatureSensor {
    private double temperature;
    private double threshold;
    private AlertSystem alertSystem;

    public void update(double newTemperature) {
        this.temperature = newTemperature;
        if (this.temperature > this.threshold) {
            alertSystem.sendAlert("Overheating!");
        }
    }
}

// Клиент просто сообщает новые данные
sensor.update(105.0);
```

**Связь с SOLID и инкапсуляцией.**

- **Инкапсуляция**: TDA — это и есть настоящая инкапсуляция. Открывать геттеры для каждого поля — значит инкапсулировать синтаксически (поле `private`), но не семантически (поведение вынесено наружу).
- **SRP**: при нарушении TDA логика, относящаяся к одному объекту, распределяется по множеству клиентов. Каждый клиент становится ответственным за то, что должен делать сам объект.
- **OCP**: если поведение инкапсулировано внутри объекта, добавление новой реализации (например, `Ellipse` вместо `Circle`) не требует изменений клиентского кода — достаточно полиморфизма.
- **Закон Деметры**: TDA и LoD взаимно усиливают друг друга. Нарушение TDA часто приводит к цепочкам запросов (`obj.getA().getB().getValue()`), что одновременно нарушает LoD.

**Когда TDA не применяется.**
Мартин Фаулер подчёркивает, что принцип не абсолютен: запросы данных (`query methods`) вполне оправданны, если они не несут с собой поведения и не ведут к дублированию логики принятия решений в клиентах. Например, методы `toString()`, `equals()`, `size()` на коллекциях — легитимные запросы.

### Вопрос 40. Что такое «Refused Bequest» (отказ от наследства) как code smell? Чем он отличается от нарушения LSP через UnsupportedOperationException, и какой рефакторинг применяется: Replace Inheritance with Delegation или Extract Superclass?

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Refused Bequest — refactoring.guru](https://refactoring.guru/smells/refused-bequest)

Refused Bequest («отказ от наследства») возникает, когда подкласс наследует методы и поля родителя, но использует лишь их часть. Ненужные методы могут просто не вызываться или быть переопределены с выбрасыванием исключений.

**Причина:** мотивация к повторному использованию кода, а не подлинная общность характеристик. Подкласс и суперкласс могут не иметь ничего общего между собой.

**Как запахло сильнее:** если подкласс переиспользует поведение, но не хочет поддерживать интерфейс суперкласса. Отказ от реализаций ещё терпим, а вот отказ от интерфейса — нет.

**Рефакторинги:**

1. **Replace Inheritance with Delegation** — применяется, когда наследование не имеет смысла и подкласс вообще не имеет ничего общего с суперклассом. Суперкласс становится полем-объектом, к которому делегируются нужные вызовы.

2. **Extract Superclass** — применяется, если наследование всё же оправдано: из родителя выносятся только нужные методы и поля в новый суперкласс, от которого наследуют оба класса.

#### Ответ от Claude

**Определение.** Refused Bequest — код-запах из категории «злоупотреблений ОО», описанный Мартином Фаулером в «Refactoring». Подкласс наследует весь контракт суперкласса, но реально использует лишь его часть. Оставшиеся унаследованные методы либо игнорируются, либо переопределяются с выбросом исключения.

**Чем отличается от нарушения LSP через UnsupportedOperationException.**

Refused Bequest и нарушение LSP — пересекающиеся, но не тождественные понятия:

- Refused Bequest — это структурный запах: иерархия «перекошена», наследование мотивировано удобством, а не отношением «является». Он может проявляться как молча (методы просто не вызываются), так и явно (бросается исключение).
- Нарушение LSP через `UnsupportedOperationException` — это явная, «громкая» форма Refused Bequest: подкласс декларирует реализацию метода, но нарушает поведенческий контракт суперкласса в рантайме. Клиентский код, работающий с суперклассом, ломается при подстановке подкласса — это и есть нарушение LSP.

Таким образом: **UnsupportedOperationException — частный, наиболее опасный случай Refused Bequest**, при котором LSP нарушено явно. Refused Bequest без исключений нарушает LSP лишь в мягкой форме (неиспользуемые члены).

Классический пример из Java: `java.util.Stack extends Vector` — Stack наследует методы вроде `add(int index, E element)`, которые нарушают семантику стека.

**Выбор рефакторинга.**

| Ситуация | Рефакторинг |
|---|---|
| Подкласс использует часть поведения суперкласса, но не хочет его интерфейса (нет отношения «является») | **Replace Inheritance with Delegation** |
| Подкласс разделяет часть поведения с суперклассом, иерархия в целом уместна, но «лишние» методы надо убрать из базового класса | **Extract Superclass** |

**Replace Inheritance with Delegation** (пример):

```java
// До: Cat наследует Animal, но не использует метод swim()
class Cat extends Animal { ... }

// После: Animal становится полем
class Cat {
    private Animal animal = new Animal();

    public String speak() {
        return animal.speak(); // делегируем только нужное
    }
}
```

**Extract Superclass** применяется, когда в иерархии есть реальная общность: создаётся новый абстрактный базовый класс только с теми членами, которые нужны обоим потомкам.

**Итог:** используйте Replace Inheritance with Delegation, когда отношение «является» ложное; Extract Superclass — когда иерархия верна, но базовый класс перегружен лишними членами.
