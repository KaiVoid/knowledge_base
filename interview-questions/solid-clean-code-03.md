### Вопрос 41. Что такое «Анемичная модель предметной области» (Anemic Domain Model)? Почему Фаулер считает её антипаттерном, как она связана с code smell «Data Class», и в каких случаях она всё же оправдана (например, в CQRS)?

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Anemic Domain Model — martinfowler.com](https://martinfowler.com/bliki/AnemicDomainModel.html)

Анемичная доменная модель состоит из доменных объектов, которые содержат практически только данные. Как описывает Фаулер: «если посмотреть на поведение, понимаешь, что его почти нет — объекты представляют собой не что иное, как мешки геттеров и сеттеров». Вся бизнес-логика вынесена в отдельные сервисные объекты.

**Почему это антипаттерн по Фаулеру:**

1. Нарушает базовый принцип ОО-дизайна — объединение данных и поведения.
2. Несёт все издержки доменной модели (сложный маппинг O/R), не давая её преимуществ.
3. По существу возвращает к процедурному стилю, лишь маскируясь под ОО.

**Связь с Data Class:** анемичные доменные объекты — это по сути коллекции Data Class-объектов: содержат только поля и методы доступа, не несут поведения, что является прямым воплощением code smell «Data Class».

#### Ответ от Claude

**Определение.** Анемичная доменная модель (Anemic Domain Model, ADM) — архитектурный подход, при котором доменные объекты содержат только данные (поля + геттеры/сеттеры), а вся бизнес-логика сосредоточена в сервисном слое (Service, Manager, Helper и т.п.).

```java
// Анемичная модель
class Order {
    private List<Item> items;
    private BigDecimal total;
    // только геттеры/сеттеры
}

class OrderService {
    public void applyDiscount(Order order, double percent) { ... }
    public BigDecimal calculateTotal(Order order) { ... }
}

// Богатая модель (Rich Domain Model)
class Order {
    private List<Item> items;

    public void applyDiscount(double percent) { ... }
    public BigDecimal calculateTotal() { ... }
}
```

**Почему Фаулер считает её антипаттерном.**

Фаулер указывает на три проблемы:
1. **Нарушение ОО-принципов** — объект должен объединять данные и поведение; ADM возвращает к процедурному программированию в ОО-обёртке.
2. **Издержки без выгод** — O/R-маппинг, доменные объекты, сложная инфраструктура — всё это присутствует, но без ключевого преимущества: инкапсуляции логики рядом с данными.
3. **Нарушение Tell, Don't Ask** — клиентский код вынужден вытягивать состояние объекта и принимать решения вместо него, что ведёт к дублированию логики.

**Связь с code smell «Data Class».**

Data Class (код-запах по Фаулеру) — класс, содержащий только поля и методы доступа без какого-либо поведения. Анемичная доменная модель — это архитектурный паттерн, при котором все доменные объекты деградируют до Data Class. Связь прямая: ADM — это намеренное, системное применение Data Class ко всему домену.

**Когда ADM всё же оправдана.**

Существуют контексты, в которых ADM является прагматичным решением:

1. **CQRS (Command Query Responsibility Segregation)** — на стороне чтения (Query side) доменные объекты часто являются плоскими DTO/Read Model без логики. Это не «ошибка», а намеренный выбор: Read Model оптимизирована под запросы, а не под бизнес-правила. Бизнес-логика сосредоточена на Command side, где может использоваться богатая модель.

2. **DTO и API-контракты** — объекты передачи данных между слоями и через сеть по определению анемичны: они несут данные, а не поведение. Это не антипаттерн, а корректная роль объекта.

3. **Простые CRUD-приложения** — если домен не несёт сложных бизнес-правил, богатая модель избыточна. Transaction Script (по Фаулеру) честнее, чем симуляция DDD.

4. **Функциональный стиль** — разделение данных (иммутабельные структуры) и функций (чистые трансформации) является основой функционального подхода, который хорошо сочетается с тестируемостью и Continuous Delivery.

**Расхождения в источниках:** Фаулер категоричен — ADM всегда антипаттерн при использовании доменной модели. Ряд практиков (Vladimir Khorikov, авторы микросервисных руководств) считают ADM приемлемой в CQRS и stateless-сервисах. Консенсус: ADM опасна при сложном домене с богатыми бизнес-правилами; при простом домене или на read-side CQRS она оправдана.

### Вопрос 42. Почему передача булевого флага (boolean parameter) в метод является code smell? Приведите пример метода вида processOrder(order, true) — покажите, как его исправить через два отдельных метода или паттерн «Стратегия».

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Flag Argument — martinfowler.com](https://martinfowler.com/bliki/FlagArgument.html)

Флаговый аргумент — параметр функции, который указывает ей выполнять разную логику в зависимости от своего значения. Мартин Фаулер объясняет: когда вы видите `book(martin, false)` — невозможно понять смысл `false` без изучения реализации метода. Флаг нарушает принцип самодокументируемого кода.

**Решение по Фаулеру:** вместо одного метода с флагом создать два отдельных метода с говорящими именами:

```java
regularBook(customer);
premiumBook(customer);
```

Если реализации сильно переплетены, публичные методы-обёртки делегируют в приватный метод:

```java
public void regularBook(Customer c) { hiddenBookImpl(c, false); }
public void premiumBook(Customer c) { hiddenBookImpl(c, true); }
private void hiddenBookImpl(Customer c, boolean isPremium) { ... }
```

#### Ответ от Claude

**Почему boolean-параметр — code smell.**

1. **Нечитаемый вызов.** `processOrder(order, true)` — что означает `true`? Обработать немедленно? Применить скидку? Пропустить валидацию? Читатель вынужден идти в реализацию за контекстом.

2. **Нарушение SRP.** Метод делает два разных дела в зависимости от флага. Если эти ветки будут изменяться по разным причинам — это нарушение Single Responsibility Principle.

3. **Масштабируемость ограничена.** Boolean-параметр кодирует ровно два состояния. При появлении третьего варианта придётся менять сигнатуру или добавлять второй флаг.

4. **Риск ошибок.** При нескольких boolean-параметрах легко перепутать порядок: `processOrder(order, true, false)` — какой флаг за что отвечает?

**Пример до рефакторинга:**

```java
// Непонятно: true — это «срочная обработка»?
orderService.processOrder(order, true);

// Реализация скрывает два разных алгоритма
public void processOrder(Order order, boolean express) {
    if (express) {
        // срочная обработка: пропускаем часть проверок, списываем приоритетно
    } else {
        // обычная обработка: полный цикл проверок
    }
}
```

**Способ 1: два отдельных метода.**

```java
// Вызов стал самодокументируемым
orderService.processExpressOrder(order);
orderService.processStandardOrder(order);

// Реализация
public void processExpressOrder(Order order) {
    // срочная обработка
}

public void processStandardOrder(Order order) {
    // обычная обработка
}
```

Если код двух методов частично пересекается, общую логику выносят в приватный метод.

**Способ 2: паттерн «Стратегия» (Strategy).**

Когда вариантов обработки больше двух или они могут меняться независимо, boolean заменяется на стратегию:

```java
// Вместо boolean — перечисление или интерфейс
public enum ProcessingStrategy {
    EXPRESS, STANDARD, DEFERRED
}

public void processOrder(Order order, ProcessingStrategy strategy) {
    strategy.process(order);
}
```

Или через интерфейс:

```java
public interface OrderProcessingStrategy {
    void process(Order order);
}

public class ExpressProcessingStrategy implements OrderProcessingStrategy {
    @Override
    public void process(Order order) { ... }
}

public class StandardProcessingStrategy implements OrderProcessingStrategy {
    @Override
    public void process(Order order) { ... }
}

// Вызов
orderService.processOrder(order, new ExpressProcessingStrategy());
```

**Когда какой способ выбирать:**

| Ситуация | Решение |
|---|---|
| Два фиксированных варианта, логика простая | Два отдельных метода |
| Вариантов 3+, или варианты могут добавляться | Перечисление или Strategy |
| Варианты определяются в рантайме или приходят извне | Интерфейс Strategy + DI |

**Исключение:** в приватных методах или внутри тестового кода boolean-флаг допустим, если область видимости мала и контекст очевиден.

### Вопрос 43. Что такое технический долг? Объясните метафору Уорда Каннингема «долг с процентами». Как классифицирует технический долг квадрант Мартина Фаулера (преднамеренный/случайный × безрассудный/благоразумный)?

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [bliki: TechnicalDebt — Martin Fowler](https://martinfowler.com/bliki/TechnicalDebt.html)

Технический долг — это накопление «мусора» (cruft): внутренних дефектов качества кода, которые со временем усложняют модификацию и расширение системы. Метафора Уорда Каннингема уподобляет некачественный код финансовому долгу: дополнительные усилия, которые требуются для добавления каждой новой функции, — это «проценты» по долгу. Выплата «основного долга» означает рефакторинг проблемного кода; иногда это оправдано, если предстоит многократно затрагивать тот же участок. Стабильный, но грязный код может оставаться нетронутым — «проценты» начисляются только в момент активной работы с ним.

Каннингем предостерегал от злоупотребления метафорой: команды часто недооценивают, насколько технический долг замедляет разработку, «исчерпывая кредитный лимит» и в итоге всё равно опаздывая с поставкой.

#### Ответ от Claude

**Что такое технический долг**

Технический долг (technical debt) — термин, введённый Уордом Каннингемом в 1992 году в докладе «The WyCash Portfolio Management System» на конференции OOPSLA. Он обозначает накопленные в коде или архитектуре недостатки качества, которые затрудняют дальнейшее развитие системы.

**Метафора Каннингема «долг с процентами»**

Каннингем описал метафору следующим образом (оригинальная цитата 1992 года):

> «Shipping first time code is like going into debt. A little debt speeds development so long as it is paid back promptly with a rewrite... The danger occurs when the debt is not repaid. Every minute spent on not-quite-right code counts as interest on that debt.»

Перевод: «Поставка кода впервые — это как взять в долг. Небольшой долг ускоряет разработку, пока его быстро возвращают через переработку кода... Опасность возникает, когда долг не возвращается. Каждая минута, потраченная на код, который "почти правильный", считается процентом по этому долгу.»

Аналогия с финансами:
- **Основной долг (principal)** — оценочное время, необходимое для исправления некачественного кода (рефакторинг).
- **Проценты (interest)** — потеря производительности при каждом касании этого кода: разработчики вынуждены работать вокруг плохо спроектированных участков. Если запутанный модуль увеличивает разработку фичи с 4 до 6 дней — 2 дня это и есть «процент».
- **Погашение долга** — рефакторинг; оправдан, когда суммарная экономия будущих «процентов» превышает стоимость рефакторинга.

Ключевой вывод: небольшой и осознанный долг допустим как инструмент ускорения; долг, который не возвращается, превращается в «долговую яму» с нарастающими процентами.

**Квадрант технического долга Мартина Фаулера**

Фаулер расширил концепцию в статье [Technical Debt Quadrant](https://martinfowler.com/bliki/TechnicalDebtQuadrant.html), добавив два измерения:
1. Был ли долг принят **осознанно (deliberate)** или возник **случайно (inadvertent)**?
2. Было ли это решение **благоразумным (prudent)** или **безрассудным (reckless)**?

```
                     Безрассудный          Благоразумный
                  ┌────────────────────┬────────────────────────────┐
Преднамеренный   │ «Нам некогда        │ «Надо поставить сейчас,    │
(deliberate)     │  думать о дизайне» │  разберёмся с последствиями│
                 │                    │  позже»                    │
                 ├────────────────────┼────────────────────────────┤
Случайный        │ «Что такое          │ «Теперь мы знаем, как надо │
(inadvertent)    │  слои?»            │  было делать»              │
                 └────────────────────┴────────────────────────────┘
```

- **Преднамеренный + безрассудный**: команда осознанно пишет плохой код, игнорируя практики качества. Фаулер считает это ошибочной стратегией: именно хороший дизайн ускоряет разработку.
- **Преднамеренный + благоразумный**: осознанный компромисс ради дедлайна при понимании последствий. Приемлем, если процентные выплаты невелики, а долг будет погашен.
- **Случайный + безрассудный**: разработчики не знают принципов хорошего проектирования и создают проблемный код неосознанно. Генерирует наибольшие «процентные» выплаты.
- **Случайный + благоразумный**: даже лучшие команды обнаруживают после года работы, что знают теперь правильный дизайн — тот, который следовало применить с самого начала. Фаулер считает это неизбежным и ожидаемым явлением.

**Практическое применение**

Приоритет погашения долга — участки с наибольшей частотой изменений (высокими «процентными выплатами»). Стабильный, редко трогаемый «грязный» код может оставаться без рефакторинга: проценты по нему фактически не начисляются.

### Вопрос 44. Каковы правила именования согласно «Чистому коду» Роберта Мартина? Объясните, что значит «имя должно выражать намерение» (reveal intent), и приведите примеры плохих и хороших имён переменных, методов и классов в Java.

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Naming Conventions from Uncle Bob's Clean Code Philosophy — DZone](https://dzone.com/articles/naming-conventions-from-uncle-bobs-clean-code-phil)

Правила именования из «Чистого кода»:

1. **Имя должно выражать намерение (reveal intent)** — имя должно отвечать на вопросы: зачем существует, что делает, как используется. Вместо `int d;` — `int elapsedTimeInDays;`.
2. **Избегать дезинформации** — не использовать `accountList`, если переменная не является настоящим `List`. Не использовать типовые суффиксы без необходимости (`nameString`).
3. **Делать осмысленные различия** — не использовать `a1, a2, a3`; избегать слов-шумов (`ProductData` vs `ProductInfo` ничем не отличаются по смыслу).
4. **Использовать произносимые имена** — `generationTimestamp` вместо `genymdhms`.
5. **Использовать поисковые имена** — избегать одиночных букв и магических чисел. Длина имени должна соответствовать размеру области видимости.
6. **Имена классов** — существительные или именные группы: `Customer`, `WikiPage`, `Account`, `AddressParser`.
7. **Имена методов** — глаголы или глагольные группы: `postPayment()`, `deletePage()`, `save()`. Аксессоры/мутаторы — `get`/`set`/`is` по стандарту JavaBean.
8. **Одно слово на концепцию** — не смешивать `fetch`, `get`, `retrieve` в разных классах для одного и того же действия.
9. **Не использовать каламбуры** — если `add()` означает конкатенацию, то операция вставки в коллекцию должна называться `insert()` или `append()`.

#### Ответ от Claude

**Принцип «имя должно выражать намерение» (Intention-Revealing Names)**

Согласно главе 2 «Clean Code» Роберта Мартина, имя переменной, метода или класса должно самостоятельно отвечать на три вопроса:
- Зачем это существует?
- Что это делает?
- Как это используется?

Если имя требует комментария — оно не выражает намерения.

**Все правила именования с примерами на Java**

**1. Intention-Revealing Names — имя выражает намерение**

```java
// Плохо
int d; // прошедшее время в днях

// Хорошо
int elapsedTimeInDays;
int daysSinceCreation;
int fileAgeInDays;
```

```java
// Плохо — что такое ячейки со значением 4?
public List<int[]> getThem() {
    List<int[]> list1 = new ArrayList<>();
    for (int[] x : theList)
        if (x[0] == 4) list1.add(x);
    return list1;
}

// Хорошо — намерение очевидно
public List<Cell> getFlaggedCells() {
    List<Cell> flaggedCells = new ArrayList<>();
    for (Cell cell : gameBoard)
        if (cell.isFlagged()) flaggedCells.add(cell);
    return flaggedCells;
}
```

**2. Avoid Disinformation — избегать дезинформации**

```java
// Плохо: accountList — не List<Account>, а Set
Set<Account> accountList = new HashSet<>();

// Хорошо
Set<Account> accountGroup = new HashSet<>();
// или просто
Set<Account> accounts = new HashSet<>();
```

**3. Make Meaningful Distinctions — осмысленные различия**

```java
// Плохо: слова-шумы, нет разницы
class ProductInfo { ... }
class ProductData { ... }

// Плохо: числовые серии
void copy(char[] a1, char[] a2) { ... }

// Хорошо
void copy(char[] source, char[] destination) { ... }
```

**4. Use Pronounceable Names — произносимые имена**

```java
// Плохо
private Date genymdhms; // generation year-month-day-hour-minute-second

// Хорошо
private Date generationTimestamp;
```

**5. Use Searchable Names — поисковые имена**

```java
// Плохо: 7 — магическое число, не найти поиском
for (int j = 0; j < 7; j++) { ... }

// Хорошо
int DAYS_IN_WEEK = 7;
for (int day = 0; day < DAYS_IN_WEEK; day++) { ... }
```

Правило: одиночные буквы (`i`, `j`) допустимы только в коротких локальных циклах, где область видимости минимальна.

**6. Имена классов**

Классы — **существительные или именные группы**. Недопустимы глаголы или размытые слова:

```java
// Хорошо
Customer, WikiPage, Account, AddressParser, OrderProcessor

// Плохо
Manager, Processor, Data, Info  // слишком расплывчато
```

**7. Имена методов**

Методы — **глаголы или глагольные группы**:

```java
// Хорошо
postPayment(), deletePage(), save(), getUserById(), isActive()

// По стандарту JavaBean
getName()    // геттер
setName()    // сеттер
isActive()   // предикат (boolean)
```

Для перегруженных конструкторов Мартин рекомендует статические фабричные методы:

```java
// Хуже
Complex c = new Complex(5.0);

// Лучше — намерение выражено явно
Complex c = Complex.fromRealNumber(5.0);
```

**8. Одно слово на концепцию (One Word Per Concept)**

```java
// Плохо: три слова для одного действия в разных классах
userRepository.fetchUser(id);
orderService.getOrder(id);
productDao.retrieveProduct(id);

// Хорошо: единообразие
userRepository.findUser(id);
orderService.findOrder(id);
productDao.findProduct(id);
```

**9. Не использовать каламбуры (Don't Pun)**

```java
// Плохо: add() используется и для конкатенации, и для добавления в коллекцию
string = string.add(suffix);    // конкатенация
list.add(element);              // добавление в коллекцию

// Хорошо: разные имена для разных действий
string = string.concat(suffix);
list.add(element);
// или insert()/append(), если add() уже занято другим смыслом
```

**Итоговый принцип**

Хорошее имя делает код самодокументируемым: читатель понимает намерение автора без дополнительных комментариев. Мартин подчёркивает: «Имена — самый мощный инструмент коммуникации между разработчиками».

### Вопрос 45. Каким должен быть размер метода и сколько аргументов он должен принимать согласно принципам Clean Code? Почему Мартин рекомендует методы до 5–20 строк, и как Introduce Parameter Object / Preserve Whole Object помогают сократить число параметров?

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Clean Code: functions — Fabrizio Duroni](https://www.fabrizioduroni.it/blog/post/2019/07/28/clean-code-functions)

Согласно главе 3 «Clean Code»:

**Размер метода**

Первое правило функций: они должны быть маленькими. Второе правило: они должны быть ещё меньше. «Functions should hardly ever be 20 lines long» — функции редко должны превышать 20 строк. Исторически экран 1980-х составлял 24 строки — функция должна была умещаться на экране. Современный стандарт строже.

Тела блоков `if`/`else`, `for`, `while` должны состоять из **одной строки** — как правило, вызова функции с описательным именем. Вложенных структур быть не должно.

**Число аргументов**

- 0 (нилада) — идеал
- 1 (монада) — приемлемо
- 2 (диада) — приемлемо
- 3 (триада) — следует избегать
- 4+ (полиада) — не должно быть вообще

Когда аргументов больше трёх и они концептуально связаны, их следует объединить в класс:

```java
// Плохо
Circle makeCircle(double x, double y, double radius);

// Хорошо
Circle makeCircle(Point center, double radius);
```

#### Ответ от Claude

**Размер метода по Clean Code**

Роберт Мартин в главе 3 «Clean Code» формулирует два правила:
1. Функции должны быть **маленькими**.
2. Функции должны быть **ещё меньше**.

Конкретные ориентиры:
- В эпоху терминалов 1980-х функция должна умещаться на экране: 24 строки × 80 символов.
- Современный стандарт: «Functions should hardly ever be 20 lines long» — функции редко должны превышать 20 строк.
- Практика Мартина: его образцовые функции занимают **2–5 строк**.
- Длина строки: не более 150 символов (лучше меньше).

**Почему маленькие методы лучше**

- **Принцип единственной ответственности (Single Responsibility)**: «FUNCTIONS SHOULD DO ONE THING. THEY SHOULD DO IT WELL. THEY SHOULD DO IT ONLY.» Маленький метод физически не может делать несколько вещей.
- **Читаемость**: человек охватывает взглядом весь метод без прокрутки.
- **Именование**: короткий метод легче назвать точно — имя описывает одно действие.
- **Тестируемость**: небольшой метод с одним ответственностью тестируется изолированно.

**Правило для управляющих структур**

```java
// Плохо: тело if — несколько строк, вложенность
if (user != null) {
    Order order = orderService.findLatest(user.getId());
    if (order != null && order.isActive()) {
        notificationService.sendReminder(user, order);
    }
}

// Хорошо: тело if — одна строка (вызов функции с описательным именем)
if (userHasActiveOrder(user)) {
    sendOrderReminder(user);
}
```

**Число аргументов**

Мартин классифицирует функции по числу аргументов:

| Термин | Аргументов | Оценка |
|--------|-----------|--------|
| Нилада (niladic) | 0 | Идеал |
| Монада (monadic) | 1 | Приемлемо |
| Диада (dyadic) | 2 | Приемлемо |
| Триада (triadic) | 3 | Использовать с осторожностью |
| Полиада (polyadic) | 4+ | Не должно быть |

Проблемы большого числа аргументов:
- Читатель должен удержать в голове смысл каждого параметра и их порядок.
- Высок риск перепутать аргументы одного типа (например, два `String` или два `boolean`).
- Сигнализирует, что метод, вероятно, делает слишком много.

**Introduce Parameter Object**

Когда несколько аргументов концептуально связаны, они объединяются в отдельный класс-объект параметра.

```java
// Плохо: 11 аргументов
public Person createPerson(
    String lastName, String firstName, String middleName,
    String salutation, String suffix,
    String streetAddress, String city, String state,
    boolean isFemale, boolean isEmployed, boolean isHomeOwner) { ... }

// Хорошо: Introduce Parameter Object
public Person createPerson(
    FullName fullName,
    Address address,
    Gender gender,
    EmploymentStatus employment,
    HomeownerStatus homeownerStatus) { ... }
```

Пример из самого «Clean Code»:

```java
// Плохо: три отдельных аргумента
Circle makeCircle(double x, double y, double radius);

// Хорошо: x и y сгруппированы в Point
Circle makeCircle(Point center, double radius);
```

Преимущества:
- Число аргументов сокращается.
- Тип параметра становится самодокументирующим.
- Валидация данных переносится внутрь объекта параметра.
- Снижается риск ошибки в порядке аргументов.

**Preserve Whole Object**

Техника из книги Мартина Фаулера «Refactoring» ([refactoring.com/catalog/preserveWholeObject](https://refactoring.com/catalog/preserveWholeObject.html)): если метод использует несколько значений из одного объекта, передавать следует весь объект, а не отдельные значения.

```java
// Плохо: из объекта извлекаются отдельные поля
int low = room.getDaysTempRange().getLow();
int high = room.getDaysTempRange().getHigh();
if (plan.withinRange(low, high)) { ... }

// Хорошо: Preserve Whole Object — передаётся весь диапазон
if (plan.withinRange(room.getDaysTempRange())) { ... }
```

Отличие от Introduce Parameter Object:
- **Introduce Parameter Object** — создаётся **новый** класс для группировки аргументов из разных источников.
- **Preserve Whole Object** — передаётся **существующий** объект целиком вместо извлечения из него отдельных значений.

Оговорка Фаулера: если передача целого объекта создаёт нежелательную зависимость между модулями, лучше этого избегать. В таких случаях альтернативой может быть перенос метода в тот класс, чьи данные он использует (Move Method).

**Итог**

Мартин рекомендует методы до 5–20 строк, потому что маленький метод естественно выполняет одно действие, легко именуется, читается и тестируется. Для сокращения числа аргументов применяются два взаимодополняющих приёма: **Introduce Parameter Object** (новый класс объединяет разрозненные аргументы) и **Preserve Whole Object** (передаётся уже существующий объект вместо его отдельных полей).
### Вопрос 46. Когда комментарий в коде является code smell, а когда — необходимостью? Приведите примеры комментариев, которые Роберт Мартин считает «дезодорантом для плохого кода», и ситуаций, где комментарий оправдан (публичный API, нетривиальный алгоритм, предупреждение о последствиях).

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Chapter 4 – Comments | Clean Code – Robert C. Martin (vivekkhatri.com)](https://vivekkhatri.com/chapter-4-comments-clean-code-robert-c-martin/)

Роберт Мартин в четвёртой главе «Clean Code» формулирует ключевой тезис: «Комментарии не компенсируют плохой код». Если приходится объяснять код комментарием — сначала нужно попытаться переписать сам код так, чтобы он говорил сам за себя.

**Плохие комментарии (code smell — «дезодорант»):**

- **Избыточные (Redundant)** — повторяют то, что и так очевидно из кода: `i++; // увеличиваем i на 1`. Такой комментарий не добавляет информации.
- **Бормотание (Mumbling)** — расплывчатые, неясные объяснения, которые сбивают с толку больше, чем помогают.
- **Вводящие в заблуждение (Misleading)** — неточно описывают код; могут привести к ошибкам при отладке.
- **Обязательные по регламенту (Mandated)** — организации требуют ставить Javadoc на каждый метод, даже очевидный: образуется шум без ценности.
- **Журнальные (Journal Comments)** — история правок в виде комментариев; эту функцию выполняет система контроля версий (Git).
- **Шумовые (Noise Comments)** — заполнители без смысловой нагрузки.
- **Закомментированный код** — «мёртвый» код, оставленный в файле; засоряет кодовую базу и вводит в замешательство.
- **Маркеры позиций** — декоративные разделители вида `////////////////////`.
- **Комментарии закрывающих скобок** — `} // end of if`, `} // end of processOrder`.
- **Нелокальная информация** — комментарий описывает не тот код, рядом с которым стоит.

Если код нуждается в объяснении — это сигнал переименовать переменные/методы, выделить функцию, улучшить структуру. Комментарий в таком случае — «дезодорант», маскирующий запах, а не устраняющий причину.

**Хорошие комментарии (необходимость):**

- **Юридические (Legal Comments)** — уведомления об авторских правах и лицензии в начале файла.
- **Информативные** — поясняют неочевидные детали: формат даты, смысл регулярного выражения.
- **Объяснение намерения (Explanation of Intent)** — почему выбрано именно такое решение, а не что оно делает.
- **Уточнение (Clarification)** — упрощают понимание действительно сложного фрагмента.
- **Предупреждение о последствиях (Warning of Consequences)** — предостерегают от опасных действий: «Внимание: этот тест занимает несколько часов» или «Не вызывать из нескольких потоков одновременно».
- **TODO-комментарии** — фиксируют известную задолженность (но не должны накапливаться бесконечно).
- **Усиление (Amplification)** — подчёркивают важность элемента, который иначе можно проглядеть.
- **Javadoc для публичного API** — документирование публичных библиотек и фреймворков через Javadoc абсолютно оправданно и необходимо.

#### Ответ от Claude

Комментарий — не самоцель, а инструмент. Роберт Мартин в «Clean Code» (глава 4) последовательно проводит границу: хороший код должен быть самодокументируемым; комментарий появляется только тогда, когда выразительных средств языка недостаточно.

**Принцип «дезодоранта»:** когда программист чувствует, что код нужно пояснить, первый импульс — написать комментарий. Правильный импульс — улучшить код. Комментарий, скрывающий плохую читаемость, — это именно «дезодорант»: он перебивает запах, не устраняя источник.

**Категории плохих комментариев (code smell):**

| Тип | Пример | Проблема |
|---|---|---|
| Избыточный | `// устанавливаем значение поля` перед `this.value = value;` | Нет новой информации |
| Вводящий в заблуждение | Комментарий описывает старое поведение, код изменился | Активно вреден |
| Журнальный | `// 2023-01-10 Ivan: добавил проверку` | Это задача Git-лога |
| Закомментированный код | `// List<String> old = getOld();` | Засоряет, пугает |
| Обязательный по шаблону | Javadoc `@param name the name` без смысла | Шум |
| Комментарий закрывающей скобки | `} // end while` | Сигнал, что метод слишком длинный |

**Категории оправданных комментариев:**

1. **Предупреждение о последствиях** — обязателен, если действие необратимо или опасно:
   ```java
   // ВНИМАНИЕ: не синхронизировано. Вызывать только из одного потока.
   public void flushCache() { ... }
   ```

2. **Нетривиальный алгоритм** — если алгоритм взят из статьи или реализует нестандартную математику, ссылка на источник (DOI, статья) ценнее любого пересказа:
   ```java
   // Алгоритм быстрой сортировки Хоара, см. Knuth TAOCP vol.3, §5.2.2
   ```

3. **Публичный API (Javadoc)** — контракт метода, допустимые диапазоны аргументов, проверяемые исключения. Без этого API нельзя безопасно использовать без чтения исходников:
   ```java
   /**
    * Возвращает текущую цену позиции с учётом скидок.
    *
    * @param quantity количество единиц, должно быть > 0
    * @return цена в копейках, никогда не отрицательная
    * @throws IllegalArgumentException если quantity <= 0
    */
   public long getPrice(int quantity) { ... }
   ```

4. **Объяснение намерения** — почему сделано именно так (обходной путь, известный баг сторонней библиотеки, бизнес-требование):
   ```java
   // Явно сравниваем через compareTo, а не equals: см. баг JDK-8130425
   if (a.compareTo(b) == 0) { ... }
   ```

5. **TODO** — допустимо при наличии трекинга задачи; не должно превращаться в долгосрочный «долг молчания».

**Итоговое правило:** перед написанием любого комментария задайте вопрос — «могу ли я переименовать, выделить метод или улучшить структуру так, чтобы этот комментарий стал не нужен?». Если ответ «нет» — комментарий оправдан.

### Вопрос 47. Почему возврат null из метода является code smell? Как паттерны Null Object и Special Case (Мартин Фаулер) позволяют его избежать? Покажите Java-пример с Optional, Collections.emptyList() и реализацией Null Object.

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Code Smells: Null (DZone)](https://dzone.com/articles/code-smells-null)

Главная проблема `null` в том, что он многозначен — и ни одно из значений не задокументировано явно:

- значение никогда не инициализировалось;
- значение некорректно;
- значение не требуется;
- такого значения не существует;
- произошла ошибка.

«Основная проблема с null — мы не знаем, что он означает». Это делает null семантически пустым и заставляет каждого вызывающего защищаться проверками.

**Три практических подхода для устранения null:**

1. **Аннотации `@NotNull` / `@Nullable`** — объявляют контракт на уровне параметра; инструменты (IntelliJ IDEA) могут применять их ко всем реализациям интерфейса сразу.

2. **`Optional` для случая «не найдено»** — когда метод намеренно возвращает «ничего», `Optional` делает это явным на уровне типа:
   ```java
   public Optional<Object> encode(Object value, MappedField field) {
       return value.equals('\0')
           ? Optional.empty()
           : Optional.of(String.valueOf(value));
   }
   ```

3. **Исключения для ошибочных случаев** — «проглатывать» исключение и возвращать `null` крайне опасно: NullPointerException всплывёт далеко от источника проблемы.

**Возврат `Collections.emptyList()`** — классический пример паттерна Null Object применительно к коллекциям: вместо `null` возвращается корректная пустая коллекция, итерирование которой не требует никаких проверок.

#### Ответ от Claude

**Почему возврат `null` — code smell:**

Возврат `null` нарушает принцип наименьшего удивления и противоречит принципу подстановки Лисков в части пост-условий: метод, объявленный возвращающим `Customer`, возвращает объект, на котором нельзя вызвать ни одного метода. Это подрывает полиморфизм — один из ключевых механизмов ООП. Вызывающий код вынужден разрывать поток выполнения защитными `if (x == null)`, которые дублируются в каждом месте использования.

**Паттерн Null Object:**

Вместо возврата `null` возвращается специальный объект, реализующий тот же интерфейс, но с нейтральным (no-op или default) поведением. Полиморфизм сохраняется: клиент вызывает методы как обычно, не подозревая об отсутствии «реального» объекта.

```java
// Абстракция
public abstract class AbstractCustomer {
    public abstract String getName();
    public abstract List<Order> getOrders();
    public abstract boolean isNull();
}

// Реальная реализация
public class RealCustomer extends AbstractCustomer {
    private final String name;
    private final List<Order> orders;

    public RealCustomer(String name, List<Order> orders) {
        this.name = name;
        this.orders = orders;
    }

    @Override public String getName()        { return name; }
    @Override public List<Order> getOrders() { return orders; }
    @Override public boolean isNull()        { return false; }
}

// Null Object — нейтральное поведение, не возвращает null
public class NullCustomer extends AbstractCustomer {
    @Override public String getName()        { return "Неизвестный клиент"; }
    @Override public List<Order> getOrders() { return Collections.emptyList(); }
    @Override public boolean isNull()        { return true; }
}

// Фабрика
public class CustomerRepository {
    public AbstractCustomer findById(long id) {
        Customer found = db.find(id);
        return found != null ? new RealCustomer(found) : new NullCustomer();
    }
}

// Клиент — без единой проверки на null
AbstractCustomer customer = repository.findById(42);
System.out.println(customer.getName());            // "Неизвестный клиент" или реальное имя
for (Order o : customer.getOrders()) { ... }       // пустой цикл или реальные заказы
```

`Collections.emptyList()` в методе `getOrders()` — тот же принцип на уровне коллекции: возвращается неизменяемый пустой список, итерирование которого безопасно без каких-либо проверок.

**Паттерн Special Case (Мартин Фаулер, «Patterns of Enterprise Application Architecture»):**

Special Case — обобщение Null Object. Там, где Null Object имеет нейтральное (no-op) поведение, Special Case реализует осмысленное бизнес-поведение для конкретного граничного случая. Fowler использует пример: клиент с неизвестным именем возвращает строку `"occupant"`, а не пустую строку и не `null`.

```java
// Special Case: неизвестный клиент с осмысленным поведением
public class UnknownCustomer extends AbstractCustomer {
    @Override
    public String getName() { return "occupant"; }   // бизнес-правило, а не просто ""

    @Override
    public BigDecimal getOutstandingBalance() {
        return BigDecimal.ZERO;   // для неизвестного клиента долг всегда 0
    }
}
```

Разница с Null Object: Null Object молча ничего не делает; Special Case делает что-то значимое с точки зрения домена.

**`Optional` как явный контракт:**

`Optional<T>` (Java 8+) кодирует факт «значение может отсутствовать» в системе типов, заставляя вызывающий код явно обработать оба случая:

```java
// Плохо — null как сигнал "не найдено"
public Customer findByEmail(String email) {
    return db.query(email); // может вернуть null
}

// Хорошо — Optional делает отсутствие явным
public Optional<Customer> findByEmail(String email) {
    return Optional.ofNullable(db.query(email));
}

// Использование без NPE
findByEmail("user@example.com")
    .map(Customer::getName)
    .orElse("Гость");
```

**Ограничения `Optional`:** не рекомендуется использовать как тип поля класса, параметр метода или в коллекциях — только как возвращаемое значение.

**Резюме — когда что применять:**

| Инструмент | Сценарий |
|---|---|
| `Optional<T>` | Метод ищет одиночный объект, результат может отсутствовать |
| `Collections.emptyList()` | Метод возвращает коллекцию, которая может быть пустой |
| Null Object | Клиентский код работает с объектом единообразно, нейтральное поведение достаточно |
| Special Case | Граничный случай имеет собственную бизнес-логику |
| Исключение | Отсутствие результата — нештатная ситуация (ошибка, нарушение инварианта) |

### Вопрос 48. Как Закон Деметры соотносится с принципом инкапсуляции и с code smell «Message Chains»? Покажите на Java-примере цепочку вида order.getCustomer().getAddress().getCity() — чем она опасна и как её устранить через «Hide Delegate» или перенос метода.

**Категория:** SOLID/чистый код · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Understanding the Law of Demeter: The "Don't Talk to Strangers" Rule (imartynov.substack.com)](https://imartynov.substack.com/p/understanding-the-law-of-demeter)

Закон Деметры гласит: метод M объекта O должен вызывать методы только у:

- самого объекта O (его собственные методы);
- объектов, переданных M в качестве аргументов;
- объектов, созданных внутри M;
- непосредственных компонентов O (его полей/атрибутов).

Эмпирическое правило: `a.m()` допустимо; `a.m().n()` уже сигнал о нарушении.

**Пример нарушения — «паровозик» (train wreck):**

```java
public class OrderProcessor {
    public void printCustomerCity(Order order) {
        // НАРУШЕНИЕ: OrderProcessor знает о Customer и Address
        String city = order.getCustomer().getBillingAddress().getCity();
        System.out.println("Город клиента: " + city);
    }
}
```

`OrderProcessor` вынужден знать о внутренней структуре `Order` (есть `Customer`), о структуре `Customer` (есть `BillingAddress`) и о структуре `Address` (есть `getCity()`). Любое изменение в цепочке требует правок во всех местах, где она использована.

**Рефакторинг Hide Delegate — решение:**

```java
// Order скрывает делегирование
class Order {
    private Customer customer;

    // Делегирующий метод — прячет внутреннюю структуру
    public String getCustomerCity() {
        return customer.getBillingAddress().getCity();
    }
}

public class OrderProcessor {
    public void printCustomerCity(Order order) {
        // Теперь OrderProcessor общается только с Order
        System.out.println("Город клиента: " + order.getCustomerCity());
    }
}
```

#### Ответ от Claude

**Закон Деметры и инкапсуляция:**

Закон Деметры (LoD, принцип наименьшего знания, «не разговаривай с незнакомцами») — прямое следствие принципа инкапсуляции: объект не должен раскрывать свою внутреннюю структуру через цепочки геттеров. Инкапсуляция скрывает данные; LoD скрывает граф зависимостей. Нарушение LoD обычно означает, что инкапсуляция нарушена де-факто: геттеры позволяют вызывающему «заглянуть внутрь» и пройти по всей объектной модели.

**Code smell «Message Chains» (Мартин Фаулер, «Refactoring»):**

Message Chains — когда клиент проходит по цепочке объектов, запрашивая у каждого следующий: `A.getB().getC().getD()`. Это хрупкий код: любое изменение в середине цепочки ломает всех клиентов. Smell также называют «train wreck» («авария поезда»).

**Чем опасна цепочка `order.getCustomer().getAddress().getCity()`:**

```java
// Типичная ситуация в сервисном слое
public class DeliveryService {
    public void scheduleDelivery(Order order) {
        // НАРУШЕНИЕ LoD: DeliveryService знает о Customer и Address
        String city = order.getCustomer().getAddress().getCity();
        String street = order.getCustomer().getAddress().getStreet();
        courier.assign(city, street);
    }
}
```

Проблемы:
1. **Высокая связность (tight coupling):** `DeliveryService` зависит от трёх классов (`Order`, `Customer`, `Address`), хотя непосредственно работает только с `Order`.
2. **Хрупкость:** если `Customer` переименует поле `address` в `billingAddress` — сломаются все вызывающие.
3. **Сложность тестирования:** для unit-теста `DeliveryService` нужно создавать/мокировать весь граф объектов.
4. **Нарушение «Tell, Don't Ask»:** сервис вытаскивает данные, чтобы делать работу за объект, вместо того чтобы попросить объект сделать это самому.

**Решение 1 — Hide Delegate (добавить делегирующий метод):**

```java
// Order берёт на себя навигацию к адресу
public class Order {
    private Customer customer;

    public String getDeliveryCity()   { return customer.getAddress().getCity(); }
    public String getDeliveryStreet() { return customer.getAddress().getStreet(); }
}

// DeliveryService общается только с Order
public class DeliveryService {
    public void scheduleDelivery(Order order) {
        courier.assign(order.getDeliveryCity(), order.getDeliveryStreet());
    }
}
```

Теперь `DeliveryService` зависит только от `Order`. Если структура адреса изменится — правится только `Order`.

**Решение 2 — перенос метода (Move Method) к данным:**

Если логика реально принадлежит другому объекту, метод переносится туда:

```java
// Customer инкапсулирует всё, что касается адреса клиента
public class Customer {
    private Address address;

    public String getCity()   { return address.getCity(); }
    public String getStreet() { return address.getStreet(); }
}

// Order делегирует Customer
public class Order {
    private Customer customer;

    public String getDeliveryCity()   { return customer.getCity(); }
    public String getDeliveryStreet() { return customer.getStreet(); }
}
```

**Компромисс и предупреждение (Middle Man):**

Злоупотребление Hide Delegate приводит к противоположному смеллу — «Middle Man»: класс, который только перенаправляет все вызовы дальше, не добавляя ценности. Правило: скрывать делегирование стоит там, где это реально уменьшает связность; если метод механически пробрасывает вызов без какой-либо логики и класс состоит из таких методов — это уже Middle Man.

**Исключение — Data Transfer Objects (DTO) и Value Objects:**

LoD, строго говоря, применяется к объектам с поведением. Для DTO и value objects, предназначенных именно для передачи данных, прямой доступ к полям через геттеры допустим — это не нарушение духа закона.

**Формальные правила LoD для метода M объекта O:**

1. Можно вызывать методы самого `O`.
2. Можно вызывать методы объектов, переданных как аргументы `M`.
3. Можно вызывать методы объектов, созданных внутри `M`.
4. Можно вызывать методы прямых полей `O`.
5. Нельзя вызывать методы на объектах, возвращённых другими методами («чужаки»).

Эмпирически: не более одной точки в цепочке вызовов — `object.method()` нормально; `object.getA().method()` — уже повод задуматься.

### Вопрос 49. Когда наследование всё же предпочтительнее композиции? Назовите критерии выбора (истинное IS-A, стабильная иерархия, переиспользование поведения через шаблонный метод) и поясните, почему Java-коллекции (Stack extends Vector) являются хрестоматийным примером неправильного наследования.

**Категория:** SOLID/чистый код · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Inheritance and Composition (Is-a vs Has-a relationship) in Java — Baeldung](https://www.baeldung.com/java-inheritance-composition)

В Java наследование реализует отношение «является» (IS-A), а композиция — отношение «содержит» (HAS-A). Несмотря на общую рекомендацию предпочитать композицию, наследование оправдано в нескольких типичных сценариях.

**Когда наследование уместно:**

1. **Подлинное IS-A отношение.** Прежде чем применять наследование, необходимо убедиться, что дочерний класс действительно является специализированным типом родительского: «Car is a Vehicle», «Cat is an Animal», «Dog is an Animal». Если отношение надуманное или носит характер «HAS-A», нужно выбрать композицию.

2. **Минимальное переопределение поведения.** Наследование особенно полезно, когда базовый класс предоставляет множество методов с поведением по умолчанию, а дочернему классу нужно переопределить лишь несколько из них. В таком случае объём кода значительно меньше, чем при делегировании.

3. **Шаблонный метод (Template Method).** Наследование оправдано, когда абстрактный базовый класс фиксирует инвариантную часть алгоритма, а подклассы реализуют только вариативные шаги. Паттерн «Шаблонный метод» из каталога GoF гласит: «Определи скелет алгоритма в одном методе, откладывая некоторые шаги на подклассы». Здесь общая структура алгоритма не меняется, но конкретные шаги варьируются — именно это делает наследование оправданным в отличие от композиции.

**Stack extends Vector — хрестоматийный пример неправильного наследования:**

`Stack` в Java унаследован от `Vector`. Это решение считается ошибочным по нескольким причинам:

- **Нарушение семантики стека.** Стек должен поддерживать только операции LIFO (push/pop/peek). Унаследовав все методы `Vector`, класс `Stack` открывает доступ к элементам по индексу (`get(i)`, `insertElementAt`, `add`), что противоречит контракту структуры данных «стек».
- **Нарушение принципа Liskov.** `Stack` не является подтипом `Vector` в поведенческом смысле — использование `Stack` вместо `Vector` может привести к непредсказуемым результатам.
- **Утечка абстракции.** Внутренняя реализация (вектор) становится частью публичного API.
- **Снижение расширяемости.** Пользователь не может предоставить собственную реализацию стека, не разрывая зависимость от `Vector`.

Правильным решением была бы композиция: класс `Stack` содержит приватный `Vector` (или `Deque`) и делегирует ему операции, не открывая ничего лишнего наружу.

#### Ответ от Claude

**Критерии выбора наследования вместо композиции**

Принцип «предпочитай композицию наследованию» (Design Patterns, GoF, 1994) — это рекомендация, а не абсолютный запрет. Наследование оправдано при одновременном выполнении следующих условий:

**1. Истинное IS-A отношение (поведенческое, а не только номинальное)**

Недостаточно, чтобы дочерний класс «называлась» разновидностью родительского — он должен быть подтипом в смысле принципа подстановки Лисков (LSP). Подкласс должен быть использован везде, где ожидается суперкласс, без изменения корректности программы. Классическая ловушка: «квадрат является прямоугольником» номинально верно, но нарушает LSP (изменение одной стороны квадрата изменяет обе, что ломает контракт `Rectangle`).

**2. Стабильная иерархия**

Наследование создаёт жёсткую связь (tight coupling): изменения в суперклассе распространяются на все подклассы. Если суперкласс не стабилен или вероятно его развитие несовместимым образом, наследование станет источником хрупкости. Правило: «Используй наследование, только если уверен, что суперкласс не изменится» (источник: DigitalOcean Composition vs Inheritance).

**3. Паттерн «Шаблонный метод» (Template Method, GoF)**

Это самый весомый технический аргумент в пользу наследования. Абстрактный суперкласс фиксирует скелет алгоритма в шаблонном методе (обычно `final`), вызывающем набор шагов. Конкретные подклассы реализуют вариативные шаги. Наследование здесь принципиально: оно позволяет повторно использовать неизменяемую последовательность шагов и при этом обеспечивает расширение в строго определённых точках.

Пример: в Spring Framework `AbstractSecurityInterceptor` фиксирует общий алгоритм проверки доступа, а конкретные реализации (`FilterSecurityInterceptor`, `MethodSecurityInterceptor`) переопределяют только специфичные для транспорта детали.

**4. Минимальное переопределение при богатом базовом классе**

Когда суперкласс содержит много методов с готовым поведением и нужно изменить лишь небольшую часть, наследование даёт экономию кода по сравнению с делегированием всех методов через обёртку.

---

**Stack extends Vector: анализ провальной конструкции**

| Проблема | Описание |
|---|---|
| Нарушение семантики LIFO | `Stack` открывает методы `add(index, element)`, `insertElementAt`, `get(i)` — всё это противоречит контракту стека |
| Нарушение LSP | `Stack` нельзя использовать как полноценный заменитель `Vector` — семантика поведения расходится |
| Утечка абстракции | Пользователи видят и могут вызывать методы реализации (Vector), а не только API стека |
| Жёсткая связь | Изменения в `Vector` автоматически влияют на `Stack`; нельзя поменять реализацию без публичных последствий |
| Ограниченная расширяемость | Написать собственную реализацию стека, совместимую по типу, невозможно — `Stack` является конкретным классом |

`Stack` — не IS-A `Vector`. Стек и вектор — разные абстракции с разными контрактами. Правильная реализация использовала бы приватное поле-делегат (`Deque<E>`) и открывала только `push`, `pop`, `peek`, `isEmpty`. Именно так сделан `ArrayDeque`, который Javadoc официально рекомендует использовать вместо `Stack`.

**Итоговое правило выбора (Three Questions Test):**

1. Является ли подкласс специализированным типом суперкласса в поведенческом смысле (LSP)? — если нет, только композиция.
2. Стабилен ли суперкласс (маловероятно нарушение обратной совместимости)? — если нет, предпочтительна композиция.
3. Нужен ли полиморфизм или шаблонный метод, требующий наследования? — если нет, рассмотрите композицию.

При положительном ответе на все три вопроса наследование оправдано.

### Вопрос 50. Как измерить и приоритизировать технический долг в реальном проекте? Опишите метрики (цикломатическая сложность, LCOM, SQALE), концепцию «процентной ставки» (interest rate = скорость деградации производительности команды) и стратегии погашения (boy scout rule, refactoring budget, tech debt sprints).

**Категория:** SOLID/чистый код · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Technical Debt — Martin Fowler's Bliki](https://martinfowler.com/bliki/TechnicalDebt.html)

Мартин Фаулер описывает технический долг через финансовую метафору. Программные системы подвержены накоплению «хлама» (cruft) — дефицитов внутреннего качества, которые затрудняют модификацию и расширение системы.

**Процентная ставка как деградация производительности.** Если запутанная структура модуля существует, добавление новых функций занимает больше времени, чем в чистом коде. Дополнительное время — это «выплата процентов». Например, функция требует 4 дня в чистом коде, но 6 дней при наличии технического долга — 2-дневная разница и есть процентный платёж. Чем дольше долг не погашается, тем больше процентов накапливается.

**Типология долга.** Фаулер ссылается на «квадрант технического долга», разделяя долг на:
- **Намеренный vs. случайный** — был ли сознательный выбор.
- **Разумный vs. безрассудный** — был ли оправдан компромисс.

Не весь хлам требует немедленного внимания: стабильный код, который почти не изменяется, требует меньшего внимания, чем активно модифицируемые секции.

**Стратегия погашения.** Фаулер рекомендует постепенное погашение основного долга, а не масштабные операции по очистке. Потратьте пару лишних дней на улучшение кода во время разработки функций, чтобы снизить будущую процентную ставку. Это естественно концентрирует рефакторинг там, где он важнее всего: в наиболее активно изменяемых областях кода.

#### Ответ от Claude

**Метрики измерения технического долга**

**1. Цикломатическая сложность (Cyclomatic Complexity, CC)**

Введена Томасом МакКейбом в 1976 году. Измеряет количество линейно независимых путей выполнения через код (формально: число рёбер в графе потока управления минус число узлов плюс 2).

- CC = 1–5: низкая сложность, код прост для понимания и тестирования.
- CC = 6–10: умеренная сложность, приемлемо.
- CC > 10: высокая сложность, высокий риск дефектов, трудно тестировать.
- CC > 20: критически высокий уровень долга, рефакторинг приоритетен.

Высокая CC напрямую коррелирует с числом unit-тестов, необходимых для полного покрытия, и вероятностью скрытых дефектов.

**2. LCOM (Lack of Cohesion of Methods)**

Метрика отсутствия связности методов в классе. Измеряет, насколько методы класса работают с одними и теми же полями (instance variables).

*Расчёт LCOM1 (Чидамбер и Кемерер):*
- P = число пар методов, не разделяющих ни одного поля.
- Q = число пар методов, разделяющих хотя бы одно поле.
- LCOM1 = max(0, P − Q).

LCOM1 = 0 означает высокую связность (хороший дизайн). Высокие значения сигнализируют о «боге-классе» (God Class), который нарушает принцип единственной ответственности (SRP) и является источником технического долга. Современные варианты (LCOM4, LCOM5) дают нормализованное значение от 0 до 1, где 1 — полное отсутствие связности.

**3. SQALE (Software Quality Assessment based on Lifecycle Expectations)**

Стандартизированный метод оценки технического долга, лежащий в основе SonarQube. Переводит нарушения кода в время ремонта (person-hours/days):

- **Debt** (основная сумма) = суммарное расчётное время исправления всех нарушений.
- **SQALE Rating** (A–E) = отношение долга к стоимости разработки проекта с нуля:
  - A: ≤5% — отличное качество;
  - B: 6–10%;
  - C: 11–20%;
  - D: 21–50%;
  - E: >50% — критическое состояние.

Оси оценки: тестируемость, надёжность, безопасность, сопровождаемость, эффективность, переносимость, дублирование. SQALE делает долг осязаемым для бизнеса: «120 человеко-дней или 180 000 €» — понятно CFO.

---

**Концепция «процентной ставки» (interest rate)**

Термин «технический долг» ввёл Уорд Каннингем (Ward Cunningham) в 1992 году. Процентная ставка — это скорость деградации производительности команды из-за накопленного долга.

**Как работает метафора:**
- *Основной долг (principal)* = усилия, необходимые для исправления существующих проблем.
- *Проценты (interest)* = дополнительное время на каждую следующую задачу из-за существующего долга.
- *Процентная ставка* = насколько быстро проценты растут, то есть насколько каждое изменение кода замедляется из-за накопленного долга.

**Ключевые наблюдения:**
- Модули с высокой частотой изменений имеют **высокую процентную ставку** — каждый коммит обходится дороже. Модули, которые годами не трогают, имеют **низкую ставку** — долг почти не влияет на производительность.
- Согласно эмпирическому исследованию (ResearchGate, 2012): рост отношения долга к объёму кода на одно стандартное отклонение соответствует снижению скорости поставки на 23% и росту плотности дефектов на 31%.
- Разработчики тратят 23–42% рабочего времени на управление техническим долгом (различные отраслевые исследования).

**Приоритизация через процентную ставку:**

Приоритет долга определяется не размером, а ставкой: *impact × frequency*. Небольшой долг в ядре системы, который затрагивается при каждой задаче, важнее крупного долга в редко изменяемом legacy-модуле. Для каждого элемента долга оценивают:
1. Насколько он замедляет команду сейчас (interest per sprint).
2. Насколько он рискует вызвать инцидент.
3. Насколько часто команда работает с этим кодом.

---

**Стратегии погашения технического долга**

**1. Boy Scout Rule (Правило бойскаута)**

Автор: Роберт Мартин (Uncle Bob), книга «Clean Code» (2008). Принцип: «Всегда оставляй код немного чище, чем нашёл его». Аналог взят из правила американских бойскаутов: «Оставь лагерную стоянку чище, чем застал её».

*Механика:* При работе над любой задачей разработчик попутно устраняет мелкие проблемы — переименовывает неудачную переменную, вытаскивает дублирующийся фрагмент в метод, добавляет тест. Без отдельного бюджета, инкрементально.

*Когда работает:* Для борьбы с медленным «гниением» кода. Не подходит для крупного архитектурного рефакторинга.

**2. Refactoring Budget (Бюджет на рефакторинг)**

Резервирование фиксированного процента ресурсов команды на работу с техническим долгом. Практическая рекомендация из *The DevOps Handbook* (Gene Kim et al.): 20% ёмкости команды отдаётся инженерии по усмотрению разработчиков (рефакторинг, устранение долга, автоматизация). Это примерно один день в неделю или одна неделя в месяц.

Преимущество: долг погашается непрерывно, без накопления до критического уровня. Метрики (SQALE Rating, CC, LCOM) обеспечивают прозрачность использования бюджета.

**3. Tech Debt Sprints (Спринты технического долга)**

Выделенные итерации, целиком посвящённые устранению долга без доставки функциональности. Используются для крупных, системных проблем, которые нельзя решить попутно.

*Когда обоснованы:* большая архитектурная реструктуризация, миграция на новую версию фреймворка, введение новых стандартов кодирования по всему проекту.

*Риски:* бизнес с трудом соглашается «потерять» спринт; необходима убедительная аргументация через метрики (снижение velocity, рост lead time, частота инцидентов).

**Комбинированная стратегия (рекомендация):**

| Тип долга | Стратегия |
|---|---|
| Мелкий локальный (опечатки, запах кода) | Boy Scout Rule |
| Умеренный системный | Refactoring Budget (20% ёмкости) |
| Крупный архитектурный | Tech Debt Sprint (квартально) |
| Критический, блокирующий | Немедленная задача в текущем спринте |

Ключевой принцип (Fowler): постепенное погашение, сконцентрированное в наиболее активно изменяемых областях, даёт наибольшую отдачу — именно там процентная ставка наиболее высока.

### Вопрос 51. Что такое рефакторинг «Извлечение метода» (Extract Method)? Когда его применяют и какие параметры/возвращаемые значения возникают при переносе фрагмента кода в новый метод?

**Категория:** SOLID/чистый код · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Extract Method — refactoring.guru](https://refactoring.guru/extract-method)

Extract Method применяется, когда фрагмент кода можно сгруппировать и вынести в отдельный метод, заменив исходный код вызовом нового метода.

**Когда применять:** чем длиннее метод, тем труднее понять, что он делает. Также извлечение метода помогает устранить дублирование — найденный фрагмент можно вызывать из нескольких мест.

**Пример (до):**

```java
void printOwing() {
    printBanner();
    System.out.println("name: " + name);
    System.out.println("amount: " + getOutstanding());
}
```

**Пример (после):**

```java
void printOwing() {
    printBanner();
    printDetails(getOutstanding());
}

void printDetails(double outstanding) {
    System.out.println("name: " + name);
    System.out.println("amount: " + outstanding);
}
```

**Преимущества:**
- Более читаемый код: описательное имя метода говорит само за себя.
- Меньше дублирования: найденный код можно использовать в нескольких местах.
- Изоляция независимых частей — ошибки менее вероятны.

#### Ответ от Claude

**Определение.** Extract Method (Извлечение метода) — одна из наиболее часто применяемых техник рефакторинга, описанных Мартином Фаулером. Суть: взять фрагмент тела метода, перенести его в отдельный метод с понятным именем, отражающим намерение фрагмента, и заменить оригинальный фрагмент вызовом нового метода.

**Когда применять:**

1. **Метод слишком длинный** — Роберт Мартин в «Clean Code» рекомендует функции до 5–20 строк. Если тело метода требует комментариев, объясняющих блоки кода, — каждый такой блок является кандидатом на извлечение.
2. **Дублирование кода** — одинаковый фрагмент встречается в нескольких методах; после Extract Method дублирование устраняется вызовом общего метода.
3. **Комментарий объясняет блок кода** — если перед блоком есть комментарий «// вычисляем скидку», значит блок можно назвать `calculateDiscount()` и извлечь.

**Как выбрать параметры и возвращаемое значение нового метода:**

```java
// ИСХОДНЫЙ МЕТОД — перегружен логикой
public void processOrder(Order order) {
    // --- валидация ---
    if (order == null) throw new IllegalArgumentException("order is null");
    if (order.getItems().isEmpty()) throw new IllegalArgumentException("no items");
    // --- расчёт итога ---
    double total = 0;
    for (Item item : order.getItems()) {
        total += item.getPrice() * item.getQuantity();
    }
    order.setTotal(total);
    // --- сохранение ---
    orderRepository.save(order);
}

// ПОСЛЕ Extract Method
public void processOrder(Order order) {
    validateOrder(order);               // нет параметров: использует только order из внешнего контекста
    calculateTotal(order);              // order — единственная зависимость
    orderRepository.save(order);
}

private void validateOrder(Order order) {
    if (order == null) throw new IllegalArgumentException("order is null");
    if (order.getItems().isEmpty()) throw new IllegalArgumentException("no items");
}

private void calculateTotal(Order order) {
    double total = 0;
    for (Item item : order.getItems()) {
        total += item.getPrice() * item.getQuantity();
    }
    order.setTotal(total);
}
```

**Правила определения параметров:**
- Переменные, которые только читаются внутри фрагмента, становятся **параметрами** нового метода.
- Переменные, которые изменяются внутри фрагмента и используются после него в исходном методе, становятся **возвращаемым значением**.
- Если изменяется несколько переменных — сначала проверьте, не является ли это сигналом к Extract Class.

**Связь с другими принципами:**
- Поддерживает **SRP**: метод начинает делать одно дело.
- Создаёт условия для **OCP**: мелкие методы легче переопределять или делегировать.
- Является фундаментальным шагом многих других рефакторингов (Extract Class, Move Method и др.).

### Вопрос 52. Что такое code smell «Примитивная одержимость» (Primitive Obsession)? Почему использование String для хранения телефона или Money как double — плохая идея? Покажите, как Value Object решает эту проблему.

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Primitive Obsession — refactoring.guru](https://refactoring.guru/smells/primitive-obsession)

Primitive Obsession — запах кода, при котором разработчик использует примитивные типы данных вместо небольших объектов для хранения доменных концепций (например, валюты, диапазонов, телефонных номеров).

**Признаки:**
- Константы для кодирования информации: `USER_ADMIN_ROLE = 1`.
- Строки в роли констант для доступа к полям массивов.
- Одни и те же группы примитивов постоянно передаются вместе.

**Рефакторинги:**
- **Replace Data Value with Object** — сгруппировать связанные поля-примитивы в выделенный класс.
- **Introduce Parameter Object** / **Preserve Whole Object** — объединить группы параметров.
- **Replace Type Code with Class/Subclasses/State-Strategy** — заменить числовые «типы» правильной системой типов.

**Преимущества устранения:**
- Гибкость кода за счёт ОО-подхода.
- Повышение читаемости: операции над данными сосредоточены в одном месте.

#### Ответ от Claude

**Почему примитивы для доменных концепций — плохая идея.**

`String phone = "+79001234567"` — синтаксически корректно, но семантически пусто:
1. **Нет валидации в точке создания.** Любая строка принимается без проверки; ошибки обнаруживаются поздно, далеко от источника.
2. **Нет инкапсуляции поведения.** Форматирование, сравнение, маскирование номера — всё это дублируется по всему коду.
3. **Нет типобезопасности.** Компилятор не различает телефон, email и имя пользователя — все они `String`. Легко перепутать аргументы при вызове `createUser(email, phone)`.

Аналогично `double price` вместо `Money`:
- Арифметика на `double` имеет ошибки округления: `0.1 + 0.2 != 0.3`.
- Нет валюты: цена в рублях или долларах?
- Не представлена операция «сложить цены разных валют».

**Value Object как решение.**

Мартин Фаулер в [bliki: ValueObject](https://martinfowler.com/bliki/ValueObject.html) определяет Value Object как небольшой объект, равенство которого определяется значением, а не идентичностью, и который является иммутабельным.

```java
// ДО: Примитивная одержимость
class Customer {
    private String phone;      // никакой валидации
    private double price;      // без валюты, проблемы с точностью
    private String email;      // строка == строка, компилятор не поможет
}

// ПОСЛЕ: Value Objects
public final class PhoneNumber {
    private final String value;

    public PhoneNumber(String value) {
        if (value == null || !value.matches("\\+\\d{7,15}")) {
            throw new IllegalArgumentException("Некорректный номер телефона: " + value);
        }
        this.value = value;
    }

    public String toMasked() {
        return value.substring(0, 3) + "***" + value.substring(value.length() - 2);
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof PhoneNumber)) return false;
        return value.equals(((PhoneNumber) o).value);
    }

    @Override
    public int hashCode() { return value.hashCode(); }

    @Override
    public String toString() { return value; }
}

public final class Money {
    private final BigDecimal amount;
    private final Currency currency;

    public Money(BigDecimal amount, Currency currency) {
        if (amount.compareTo(BigDecimal.ZERO) < 0)
            throw new IllegalArgumentException("Сумма не может быть отрицательной");
        this.amount = amount;
        this.currency = currency;
    }

    public Money add(Money other) {
        if (!this.currency.equals(other.currency))
            throw new IllegalArgumentException("Нельзя складывать суммы в разных валютах");
        return new Money(this.amount.add(other.amount), this.currency);
    }

    // equals, hashCode, toString
}

class Customer {
    private PhoneNumber phone;
    private Money price;
    private Email email;  // аналогичный Value Object
}
```

**Ключевые свойства Value Object:**
1. **Иммутабельность** — состояние не меняется после создания; вместо изменения создаётся новый объект.
2. **Равенство по значению** — два объекта `PhoneNumber("+79001234567")` равны, даже если это разные экземпляры.
3. **Самовалидация** — невозможно создать невалидный объект.
4. **Поведение рядом с данными** — метод `toMasked()` не дублируется по всему коду.

**Расхождения в источниках:** refactoring.guru называет этот запах «Primitive Obsession» и предлагает «Replace Data Value with Object»; Мартин Фаулер в bliki разграничивает Value Object и Entity (у Entity идентичность, у Value Object — значение). В DDD Value Object — фундаментальный строительный блок наряду с Entity и Aggregate.

### Вопрос 53. Что такое code smell «Длинный список параметров» (Long Parameter List)? Чем он отличается от «Data Clumps» и как техники Introduce Parameter Object и Preserve Whole Object помогают устранить оба запаха?

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Long Parameter List — refactoring.guru](https://refactoring.guru/smells/long-parameter-list)

Long Parameter List возникает, когда метод принимает более трёх-четырёх параметров. Такой список трудно понять и поддерживать.

**Причины:**
- Объединение нескольких алгоритмов в один метод.
- Стремление снизить связность классов: код создания объектов перемещается к вызывающей стороне, а объекты передаются как параметры.

**Рефакторинги:**
- **Replace Parameter with Method Call** — если аргумент является результатом вызова другого метода, объект может стать полем класса.
- **Preserve Whole Object** — передать весь объект вместо отдельных значений.
- **Introduce Parameter Object** — консолидировать параметры разного происхождения в один объект.

#### Ответ от Claude

**Long Parameter List — определение и проблемы.**

Роберт Мартин в «Clean Code» устанавливает идеальное число параметров — 0 (нилада), приемлемое — 1–2, опасное — 3+ (триада), недопустимое — 4 и более. Длинный список параметров сложен потому, что:
1. Читатель должен удерживать в голове смысл каждого параметра и их порядок.
2. При нескольких параметрах одного типа легко перепутать порядок: `createUser("Ivan", "ivan@mail.ru")` — что первым: имя или email?
3. Добавление нового параметра ломает все вызывающие стороны.

**Data Clumps — определение и отличие.**

> Источник: [Data Clumps — refactoring.guru](https://refactoring.guru/smells/data-clumps)

Data Clumps — запах, при котором в разных частях кода встречаются одни и те же группы переменных. Тест: если удалить одну из переменных группы и оставшиеся потеряют смысл — это Data Clump.

| Критерий | Long Parameter List | Data Clumps |
|---|---|---|
| Где встречается | Сигнатура одного метода | В разных методах/классах повторяется одна и та же группа |
| Проблема | Трудно понять и вызвать метод | Скрытая концепция не вынесена в отдельный класс |
| Связь | Data Clumps часто приводят к Long Parameter List | Long Parameter List может содержать Data Clump |

**Introduce Parameter Object — пример.**

```java
// ДО: Long Parameter List + Data Clump (firstName, lastName, email встречаются везде)
public void registerUser(String firstName, String lastName,
                          String email, String password,
                          String city, String country, String zipCode) {
    // 7 параметров — катастрофа
}

// ПОСЛЕ: Introduce Parameter Object
public record UserRegistrationRequest(
    PersonalInfo personalInfo,
    Credentials credentials,
    Address address
) {}

public record PersonalInfo(String firstName, String lastName, String email) {}
public record Credentials(String password) {}
public record Address(String city, String country, String zipCode) {}

public void registerUser(UserRegistrationRequest request) {
    // один параметр, самодокументирующийся
}
```

**Preserve Whole Object — пример.**

```java
// ДО: из объекта извлекаются отдельные поля
int low = thermometer.getDaysTempRange().getLow();
int high = thermometer.getDaysTempRange().getHigh();
heatingPlan.setTargetRange(low, high); // два отдельных значения

// ПОСЛЕ: Preserve Whole Object — передаётся весь объект
heatingPlan.setTargetRange(thermometer.getDaysTempRange());
```

**Предупреждение:** Preserve Whole Object создаёт зависимость между классами. Если `heatingPlan` не должен знать о `TempRange` — лучше оставить примитивные параметры или применить Move Method.

**Когда что выбирать:**

| Ситуация | Рефакторинг |
|---|---|
| Параметры из разных источников, явно связаны по смыслу | Introduce Parameter Object |
| Параметры — значения одного существующего объекта | Preserve Whole Object |
| Один из параметров — результат вызова метода на доступном объекте | Replace Parameter with Method Call |

### Вопрос 54. Как паттерн «Декоратор» (Decorator) реализует принцип открытости/закрытости (OCP)? Приведите Java-пример с BufferedInputStream или аналогичной цепочкой, покажите, как добавлять поведение без изменения исходного класса.

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [The Decorator Pattern in Java — Baeldung](https://www.baeldung.com/java-decorator-pattern)

Паттерн Decorator «прикрепляет дополнительные обязанности к объекту статически или динамически». Ключевое: вместо наследования используется **композиция**, что позволяет стекировать декораторы без взрыва числа подклассов.

Структура:
1. Базовый интерфейс (`ChristmasTree`).
2. Конкретная реализация (`ChristmasTreeImpl`).
3. Абстрактный декоратор, реализующий тот же интерфейс и содержащий ссылку на него.
4. Конкретные декораторы, расширяющие абстрактный.

```java
public abstract class TreeDecorator implements ChristmasTree {
    private ChristmasTree tree;

    @Override
    public String decorate() {
        return tree.decorate();
    }
}

public class BubbleLights extends TreeDecorator {
    public String decorate() {
        return super.decorate() + " with Bubble Lights";
    }
}
```

Практика Java: `BufferedInputStream` декорирует `FileInputStream`, добавляя буферизацию без изменения ни одного из них.

#### Ответ от Claude

**Как Decorator реализует OCP.**

OCP гласит: программные сущности должны быть открыты для расширения, но закрыты для модификации. Decorator воплощает этот принцип через следующую схему:
- **Закрыт для модификации:** исходный класс (`FileInputStream`, `ChristmasTreeImpl`) не трогается.
- **Открыт для расширения:** новое поведение добавляется через создание нового класса-декоратора.

**Полный Java-пример: логирующий и шифрующий потоки.**

```java
// Базовый интерфейс
public interface DataSource {
    void writeData(String data);
    String readData();
}

// Конкретная реализация (закрыта для модификации)
public class FileDataSource implements DataSource {
    private final String filename;

    public FileDataSource(String filename) { this.filename = filename; }

    @Override
    public void writeData(String data) {
        // запись в файл
    }

    @Override
    public String readData() {
        return "данные из файла";
    }
}

// Абстрактный декоратор
public abstract class DataSourceDecorator implements DataSource {
    private final DataSource wrappee;

    public DataSourceDecorator(DataSource source) { this.wrappee = source; }

    @Override
    public void writeData(String data) { wrappee.writeData(data); }

    @Override
    public String readData() { return wrappee.readData(); }
}

// Декоратор 1: шифрование (без изменения FileDataSource)
public class EncryptionDecorator extends DataSourceDecorator {
    public EncryptionDecorator(DataSource source) { super(source); }

    @Override
    public void writeData(String data) {
        super.writeData(encrypt(data));
    }

    @Override
    public String readData() {
        return decrypt(super.readData());
    }

    private String encrypt(String data) { return "[encrypted]" + data; }
    private String decrypt(String data) { return data.replace("[encrypted]", ""); }
}

// Декоратор 2: сжатие (добавляется без изменений предыдущих классов)
public class CompressionDecorator extends DataSourceDecorator {
    public CompressionDecorator(DataSource source) { super(source); }

    @Override
    public void writeData(String data) {
        super.writeData(compress(data));
    }

    private String compress(String data) { return "[compressed]" + data; }
}

// Использование — стекирование декораторов в рантайме
DataSource source = new FileDataSource("output.dat");
source = new CompressionDecorator(source);   // добавили сжатие
source = new EncryptionDecorator(source);    // добавили шифрование поверх сжатия
source.writeData("Hello World");
// записывается: [encrypted][compressed]Hello World
```

**Пример из стандартной библиотеки Java — java.io:**

```java
// Стандартная цепочка декораторов
InputStream raw = new FileInputStream("file.txt");      // конкретная реализация
InputStream buffered = new BufferedInputStream(raw);     // декоратор: буферизация
InputStream decompress = new GZIPInputStream(buffered);  // декоратор: распаковка

// Каждый декоратор добавляет поведение, не меняя исходные классы
```

**Преимущества Decorator над наследованием:**

| Критерий | Наследование | Decorator |
|---|---|---|
| Добавление поведения | Создать новый подкласс для каждой комбинации | Стекировать декораторы в рантайме |
| Взрыв числа классов | Да: LoggingBufferedEncryptedFileStream | Нет: 3 отдельных декоратора |
| Изменение поведения в рантайме | Невозможно | Возможно |
| OCP | Нарушается при добавлении новых комбинаций | Соблюдается |

**Когда применять:** когда нужно добавлять функциональность к объектам гибко и комбинируемо, не создавая взрыв подклассов; когда изменение базового класса нежелательно или невозможно (сторонняя библиотека).

### Вопрос 55. Что такое рефакторинг «Замена условного оператора полиморфизмом» (Replace Conditional with Polymorphism)? Покажите на Java-примере, как switch/if-else по типу объекта заменяется иерархией классов, и объясните связь с OCP.

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Replace Conditional with Polymorphism — refactoring.guru](https://refactoring.guru/replace-conditional-with-polymorphism)

Рефакторинг устраняет условные операторы, которые «выполняют различные действия в зависимости от типа объекта или свойств». Вместо этого логика каждой ветви перемещается в соответствующий подкласс.

**Пример до:**

```java
class Bird {
    double getSpeed() {
        switch (type) {
            case EUROPEAN: return getBaseSpeed();
            case AFRICAN: return getBaseSpeed() - getLoadFactor() * numberOfCoconuts;
            case NORWEGIAN_BLUE: return (isNailed) ? 0 : getBaseSpeed(voltage);
        }
    }
}
```

**Пример после:**

```java
abstract class Bird { abstract double getSpeed(); }

class European extends Bird {
    double getSpeed() { return getBaseSpeed(); }
}
class African extends Bird {
    double getSpeed() { return getBaseSpeed() - getLoadFactor() * numberOfCoconuts; }
}
class NorwegianBlue extends Bird {
    double getSpeed() { return (isNailed) ? 0 : getBaseSpeed(voltage); }
}
```

**Преимущества:** следует принципу «Tell, Don't Ask»; устраняет почти идентичные условные ветви; поддерживает OCP — добавление нового варианта не требует изменения существующего кода.

#### Ответ от Claude

**Когда switch/if-else по типу является проблемой.**

Условный оператор, переключающийся по типу объекта или дискриминатору (type code), — прямой признак того, что полиморфизм не используется. Каждый раз при добавлении нового типа нужно находить и обновлять все такие условные операторы в кодовой базе — это нарушение OCP (Open/Closed Principle).

**Полный Java-пример: расчёт стоимости доставки.**

```java
// ДО: нарушение OCP — добавление нового типа доставки требует изменения метода
public class DeliveryService {
    public double calculateCost(String type, double weight, double distance) {
        switch (type) {
            case "STANDARD":
                return weight * 0.5 + distance * 0.1;
            case "EXPRESS":
                return weight * 1.0 + distance * 0.3;
            case "OVERNIGHT":
                return weight * 2.0 + distance * 0.5 + 10.0; // базовая плата
            default:
                throw new IllegalArgumentException("Unknown type: " + type);
        }
    }
}
```

При добавлении типа `DRONE` придётся менять `DeliveryService` — нарушение OCP.

```java
// ПОСЛЕ: Replace Conditional with Polymorphism
public abstract class DeliveryMethod {
    public abstract double calculateCost(double weight, double distance);
}

public class StandardDelivery extends DeliveryMethod {
    @Override
    public double calculateCost(double weight, double distance) {
        return weight * 0.5 + distance * 0.1;
    }
}

public class ExpressDelivery extends DeliveryMethod {
    @Override
    public double calculateCost(double weight, double distance) {
        return weight * 1.0 + distance * 0.3;
    }
}

public class OvernightDelivery extends DeliveryMethod {
    private static final double BASE_FEE = 10.0;

    @Override
    public double calculateCost(double weight, double distance) {
        return weight * 2.0 + distance * 0.5 + BASE_FEE;
    }
}

// Новый тип добавляется без изменения существующего кода — OCP соблюдён
public class DroneDelivery extends DeliveryMethod {
    @Override
    public double calculateCost(double weight, double distance) {
        return distance * 0.05; // дроны дёшевы и не зависят от веса
    }
}

// Клиентский код работает с абстракцией
public class DeliveryService {
    public double calculateCost(DeliveryMethod method, double weight, double distance) {
        return method.calculateCost(weight, distance); // полиморфный вызов
    }
}
```

**Связь с OCP и Tell-Don't-Ask.**

- **OCP:** добавление `DroneDelivery` — это новый класс, а не правка существующего. `DeliveryService` закрыт для модификации.
- **Tell, Don't Ask:** вместо «спроси объект о его типе, потом прими решение» — «скажи объекту вычислить стоимость, он сам знает как».
- **SRP:** каждый класс доставки содержит ровно одну логику ценообразования.

**Когда рефакторинг не нужен:**

- Если условных ветвей всего 2–3 и они стабильны (не расширяются) — полиморфизм добавляет сложность без пользы.
- Если условие проверяет не тип, а простое бизнес-правило (например, `if (price > 1000) discount = 0.1`) — это не повод для замены.

### Вопрос 56. Что такое code smell «Группы данных» (Data Clumps)? Как определить, является ли группа переменных Data Clump, и какими рефакторингами устранить запах?

**Категория:** SOLID/чистый код · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Data Clumps — refactoring.guru](https://refactoring.guru/smells/data-clumps)

Data Clumps — запах, при котором в разных частях кода встречаются идентичные группы переменных (например, параметры для подключения к БД).

**Диагностический тест:** удалите одно из значений группы — если оставшиеся значения потеряют смысл, это надёжный признак того, что данные следует объединить в объект.

**Причины:** плохая структура программы; «copy-paste программирование».

**Рефакторинги:**
- Если повторяющиеся данные — поля класса: **Extract Class**.
- Если одни и те же данные передаются как параметры: **Introduce Parameter Object**.
- Если данные передаются в другие методы: **Preserve Whole Object**.

#### Ответ от Claude

**Определение.** Data Clumps (Группы данных) — код-запах из категории «Раздутые» по классификации Мартина Фаулера. Когда несколько переменных постоянно появляются вместе — в сигнатурах методов, в полях классов — это признак скрытой концепции, которая ещё не получила своего имени и класса.

**Как диагностировать:**

Тест Фаулера: «Если удалить одну переменную из группы, и оставшиеся перестанут иметь смысл — это Data Clump».

Например, `String host, int port, String username, String password` — уберите `host`, и `port` теряет смысл без контекста. Все четыре переменные образуют концепцию «настройки подключения к БД».

**Примеры Data Clumps и рефакторинги:**

```java
// ДО: группа (host, port, username, password) повторяется везде
public class DatabaseService {
    public Connection connect(String host, int port, String username, String password) { ... }
    public void test(String host, int port, String username, String password) { ... }
}

public class BackupService {
    public void backup(String host, int port, String username, String password) { ... }
}

// ПОСЛЕ: Extract Class + Introduce Parameter Object
public class DatabaseCredentials {
    private final String host;
    private final int port;
    private final String username;
    private final String password;

    public DatabaseCredentials(String host, int port, String username, String password) {
        if (host == null || host.isBlank()) throw new IllegalArgumentException("host required");
        if (port < 1 || port > 65535) throw new IllegalArgumentException("invalid port");
        this.host = host;
        this.port = port;
        this.username = username;
        this.password = password;
    }

    public String getConnectionUrl() {
        return "jdbc:postgresql://" + host + ":" + port;
    }
    // геттеры, equals, hashCode
}

public class DatabaseService {
    public Connection connect(DatabaseCredentials credentials) { ... }
    public void test(DatabaseCredentials credentials) { ... }
}

public class BackupService {
    public void backup(DatabaseCredentials credentials) { ... }
}
```

**Рефакторинг Extract Class** применяется, когда Data Clump встречается в полях класса:

```java
// ДО: поля класса образуют несколько групп
class Employee {
    String firstName, lastName;    // группа "имя"
    String street, city, zipCode;  // группа "адрес"
    LocalDate startDate;
}

// ПОСЛЕ: Extract Class для каждой группы
class Employee {
    FullName name;       // выделенный класс
    Address address;     // выделенный класс
    LocalDate startDate;
}
```

**Связь с другими запахами:**

Data Clumps часто сопровождаются Primitive Obsession (члены группы — примитивы) и Long Parameter List (группа расширяется случайными новыми параметрами). Устранение Data Clumps через Extract Class также устраняет оба сопутствующих запаха и создаёт естественное место для перемещения поведения (Feature Envy).

### Вопрос 57. Что такое принцип разделения команд и запросов (Command-Query Separation, CQS)? Чем он отличается от CQRS? Почему `Stack.pop()` — допустимое нарушение CQS?

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [bliki: Command Query Separation — martinfowler.com](https://martinfowler.com/bliki/CommandQuerySeparation.html)

CQS был введён Бертраном Мейером в книге «Object Oriented Software Construction». Принцип разделяет методы объекта на две категории:

- **Запросы (Queries):** возвращают результат и не изменяют наблюдаемое состояние системы (свободны от побочных эффектов).
- **Команды (Commands / Modifiers):** изменяют состояние системы, но не возвращают значения.

**Практическая ценность:** если можно чётко разделить методы, изменяющие состояние, от тех, что его не меняют, запросы можно использовать с намного большей уверенностью — вставлять в любое место, менять порядок.

**Исключения:** Мейер придерживается CQS абсолютно, Фаулер признаёт допустимые исключения: «поп стека — хороший пример запроса, модифицирующего состояние».

#### Ответ от Claude

**CQS — определение и правила.**

Command-Query Separation (CQS) — принцип проектирования методов объекта, сформулированный Бертраном Мейером (1988). Правило: каждый метод должен быть либо командой, либо запросом — но не обоими одновременно.

- **Запрос (Query):** возвращает данные, не меняет состояние. Идемпотентен — можно вызывать многократно с одним результатом. Примеры: `getBalance()`, `size()`, `isEmpty()`, `findById(id)`.
- **Команда (Command):** меняет состояние, возвращает `void`. Примеры: `save()`, `delete()`, `setName()`, `processOrder()`.

**Java-пример нарушения и исправления:**

```java
// НАРУШЕНИЕ: метод и возвращает значение, и меняет состояние
public class UserService {
    private final List<User> users = new ArrayList<>();

    // Нарушение CQS: возвращает Id и добавляет пользователя
    public Long addUser(User user) {
        users.add(user);
        return user.getId(); // побочный эффект + возврат значения
    }
}

// СООТВЕТСТВИЕ CQS: разделяем команду и запрос
public class UserService {
    private final List<User> users = new ArrayList<>();

    // Команда: меняет состояние, ничего не возвращает
    public void addUser(User user) {
        users.add(user);
    }

    // Запрос: только читает, не меняет состояние
    public Optional<User> findUserById(Long id) {
        return users.stream()
                    .filter(u -> u.getId().equals(id))
                    .findFirst();
    }
}

// Вызывающий код: явная последовательность команда → запрос
userService.addUser(newUser);
Optional<User> saved = userService.findUserById(newUser.getId());
```

**Почему `Stack.pop()` — допустимое нарушение.**

`pop()` одновременно возвращает верхний элемент (запрос) и удаляет его (команда). Разделить на две отдельные операции (`peek()` + `remove()`) теоретически возможно, но создаёт проблему безопасности в многопоточной среде: между вызовами `peek()` и `remove()` другой поток может изменить стек. Атомарность операции pop — её главное достоинство. Мейер принимает это как исключение, Фаулер — как «полезный идиом».

**Чем CQS отличается от CQRS.**

| Аспект | CQS | CQRS |
|---|---|---|
| Уровень применения | Метод объекта | Архитектурный паттерн |
| Автор | Бертран Мейер (1988) | Грег Янг (2010), развивает идею Фаулера |
| Суть | Метод — либо команда, либо запрос | Отдельные модели (и часто отдельные хранилища) для команд и запросов |
| Масштаб | Класс / интерфейс | Система, микросервис |
| Типичное использование | Любой ОО-код | Event Sourcing, высоконагруженные системы с разными требованиями к чтению и записи |

**Практические преимущества CQS:**
- Запросы безопасно вызывать в любом порядке, в цепочках, в тестах — без риска побочных эффектов.
- Тесты для запросов проще: не нужно проверять побочные эффекты.
- Логи и трассировка: команды — точки изменения состояния, запросы — точки чтения; их легко разделить в мониторинге.

### Вопрос 58. Что такое паттерн «Шаблонный метод» (Template Method)? Как он реализует OCP и DIP? Покажите Java-пример, объясните связь с наследованием и то, чем он отличается от паттерна «Стратегия».

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Template Method — refactoring.guru](https://refactoring.guru/design-patterns/template-method)

Template Method — поведенческий паттерн проектирования, который «определяет скелет алгоритма в суперклассе, позволяя подклассам переопределять конкретные шаги алгоритма, не изменяя его структуру».

**Структура:**
1. **Абстрактный класс** содержит шаблонный метод (обычно `final`), вызывающий шаги алгоритма, и объявляет абстрактные (обязательные) и конкретные (опциональные — «хуки») шаги.
2. **Конкретные классы** реализуют абстрактные шаги, при желании переопределяя хуки.

**Применение:**
- Позволяет клиентам расширять только конкретные шаги алгоритма, не затрагивая его структуру.
- Устраняет дублирование кода в классах, различающихся лишь отдельными шагами.

#### Ответ от Claude

**Template Method и OCP.**

OCP требует: открыто для расширения, закрыто для модификации. Template Method реализует это следующим образом:
- **Закрыто:** шаблонный метод (`final`) фиксирует структуру алгоритма — его не изменить без модификации суперкласса.
- **Открыто:** подклассы расширяют поведение, реализуя абстрактные шаги. Добавление нового варианта — новый подкласс без изменения базового.

**Java-пример: отчёты в разных форматах.**

```java
// Абстрактный класс с шаблонным методом
public abstract class ReportGenerator {

    // Шаблонный метод — final, структура алгоритма неизменна
    public final String generate(ReportData data) {
        String header = buildHeader(data.getTitle());
        String body   = buildBody(data.getRows());
        String footer = buildFooter();
        return header + body + footer;
    }

    // Абстрактные шаги — подклассы обязаны реализовать
    protected abstract String buildHeader(String title);
    protected abstract String buildBody(List<String[]> rows);

    // Хук (hook) — опциональный шаг с реализацией по умолчанию
    protected String buildFooter() {
        return "\n--- конец отчёта ---";
    }
}

// Конкретный подкласс 1: CSV-отчёт
public class CsvReportGenerator extends ReportGenerator {
    @Override
    protected String buildHeader(String title) {
        return "# " + title + "\n";
    }

    @Override
    protected String buildBody(List<String[]> rows) {
        StringBuilder sb = new StringBuilder();
        for (String[] row : rows) {
            sb.append(String.join(",", row)).append("\n");
        }
        return sb.toString();
    }
}

// Конкретный подкласс 2: HTML-отчёт (без изменения ReportGenerator — OCP)
public class HtmlReportGenerator extends ReportGenerator {
    @Override
    protected String buildHeader(String title) {
        return "<html><head><title>" + title + "</title></head><body>\n";
    }

    @Override
    protected String buildBody(List<String[]> rows) {
        StringBuilder sb = new StringBuilder("<table>\n");
        for (String[] row : rows) {
            sb.append("<tr>");
            for (String cell : row) sb.append("<td>").append(cell).append("</td>");
            sb.append("</tr>\n");
        }
        return sb.append("</table>\n").toString();
    }

    @Override
    protected String buildFooter() {
        return "</body></html>";
    }
}

// Spring Framework использует Template Method в JdbcTemplate, RestTemplate:
// структура алгоритма (открыть соединение → выполнить запрос → закрыть) — в базовом классе,
// конкретные действия (маппинг строк в объекты) — в переопределяемых методах или callback.
```

**Template Method vs. Strategy — ключевое различие.**

| Аспект | Template Method | Strategy |
|---|---|---|
| Механизм расширения | Наследование (подкласс переопределяет шаги) | Композиция (объект стратегии внедряется) |
| Изменение алгоритма | На этапе компиляции (выбор подкласса) | В рантайме (подмена стратегии) |
| Степень контроля | Суперкласс контролирует структуру алгоритма | Клиент полностью контролирует выбор алгоритма |
| OCP | Соблюдается при добавлении новых подклассов | Соблюдается при добавлении новых стратегий |
| Когда выбирать | Алгоритм стабилен, варьируются только шаги | Алгоритм нужно менять в рантайме или тестировать в изоляции |

**Реализация DIP в Template Method:** клиентский код зависит от абстрактного `ReportGenerator`, а не от конкретных `CsvReportGenerator` / `HtmlReportGenerator`. Конкретики (детали) зависят от абстракции (базового класса) — это и есть инверсия зависимостей.

### Вопрос 59. Что такое цикломатическая сложность (Cyclomatic Complexity)? Как она связана с тестируемостью кода? Какое значение считается приемлемым, и как рефакторинг снижает этот показатель?

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [What Is Cyclomatic Complexity? — Baeldung on Computer Science](https://www.baeldung.com/cs/cyclomatic-complexity)

Цикломатическая сложность — метрика стабильности и надёжности программного модуля. Высокие значения указывают на высокую сложность и большее число дефектов, что свидетельствует о необходимости разбить модуль на более мелкие части.

**Способы вычисления:**

1. Формула: CC = E − N + 2C, где E — рёбра, N — узлы, C — связные компоненты графа потока управления.
2. Упрощённый подсчёт: CC = 1 + количество точек ветвления (решений).

**Связь с тестируемостью:** значение CC показывает ровно столько независимых путей выполнения, сколько тест-кейсов необходимо для 100% покрытия ветвей. Код с CC > 11 считается очень сложным и трудным для тестирования.

#### Ответ от Claude

**Определение и формула.**

Цикломатическая сложность (Cyclomatic Complexity, CC) введена Томасом МакКейбом в 1976 году. Измеряет число линейно независимых путей через код. Практическая формула для метода:

**CC = 1 + количество точек ветвления**

Точки ветвления: `if`, `else if`, `for`, `while`, `do-while`, `case` в `switch`, `&&`, `||`, `? :`.

**Пример расчёта:**

```java
public String classify(int score) {       // CC начинается с 1
    if (score >= 90) {                    // +1 = 2
        return "Отлично";
    } else if (score >= 70) {             // +1 = 3
        return "Хорошо";
    } else if (score >= 50) {             // +1 = 4
        return "Удовлетворительно";
    } else {
        return "Неудовлетворительно";
    }
}                                         // CC = 4
```

Для полного покрытия ветвей этого метода нужно 4 тест-кейса — по числу CC.

**Шкала оценки (по МакКейбу):**

| CC | Оценка | Риск |
|---|---|---|
| 1–10 | Простой | Низкий |
| 11–20 | Умеренно сложный | Умеренный |
| 21–50 | Сложный | Высокий |
| > 50 | Нетестируемый | Критический |

**Связь с тестируемостью и принципами чистого кода.**

- **Прямая связь с числом тестов:** CC = точное минимальное число тестов для 100% branch coverage. CC = 15 означает 15 тестов только для одного метода.
- **Признак нарушения SRP:** высокая CC в одном методе — почти всегда признак того, что метод делает слишком много (несколько ответственностей).
- **Признак нарушения OCP:** if-else по типу объекта вместо полиморфизма — прямой вклад в CC.

**Как рефакторинг снижает CC:**

```java
// CC = 6 — нарушение SRP, сложно тестировать
public double processPayment(String type, double amount, String currency) {
    double converted;
    if ("USD".equals(currency)) {          // +1
        converted = amount;
    } else if ("EUR".equals(currency)) {   // +1
        converted = amount * 1.1;
    } else {
        converted = amount * 0.9;
    }

    if ("CARD".equals(type)) {             // +1
        if (converted > 10000) {           // +1
            applyFraudCheck(converted);
        }
        return cardGateway.charge(converted);
    } else if ("CASH".equals(type)) {      // +1
        return cashRegister.process(converted);
    }
    throw new IllegalArgumentException("Unknown payment type: " + type);
}

// ПОСЛЕ рефакторинга:
// 1. Extract Method: конвертация валют (CC = 3 отдельно)
// 2. Replace Conditional with Polymorphism: типы платежей (CC = 1 в базовом методе)
// 3. Каждый метод: CC <= 3, легко тестируется

private double convertCurrency(double amount, String currency) { // CC = 3
    if ("USD".equals(currency)) return amount;
    if ("EUR".equals(currency)) return amount * 1.1;
    return amount * 0.9;
}

public interface PaymentMethod { double charge(double amount); }

public class CardPayment implements PaymentMethod {
    public double charge(double amount) {  // CC = 2
        if (amount > 10000) applyFraudCheck(amount);
        return cardGateway.charge(amount);
    }
}

public class CashPayment implements PaymentMethod {
    public double charge(double amount) { // CC = 1
        return cashRegister.process(amount);
    }
}

// Основной метод: CC = 1 (нет ветвлений!)
public double processPayment(PaymentMethod method, double amount, String currency) {
    return method.charge(convertCurrency(amount, currency));
}
```

**Инструменты измерения CC в Java-проектах:** SonarQube, Checkstyle (метрика `CyclomaticComplexity`), IntelliJ IDEA (Analyze → Calculate Metrics), PMD.

### Вопрос 60. Что такое code smell «Умозрительная обобщённость» (Speculative Generality)? Как он связан с принципом YAGNI? Приведите примеры: неиспользуемые абстрактные классы, параметры, стратегии «на будущее».

**Категория:** SOLID/чистый код · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Speculative Generality — refactoring.guru](https://refactoring.guru/smells/speculative-generality)

Speculative Generality — запах кода из категории «Ненужное» (Dispensables): неиспользуемые классы, методы, поля или параметры, созданные «на случай» поддержки будущей функциональности, которая так и не была реализована.

**Признаки:** в коде присутствует неиспользуемый класс, метод, поле или параметр.

**Причина:** «Код создаётся "про запас" для поддержки предполагаемых будущих возможностей, которые так и не реализуются. В результате код становится трудным для понимания и поддержки».

**Рефакторинги:**
- Неиспользуемые абстрактные классы → **Collapse Hierarchy**.
- Ненужное делегирование → **Inline Class**.
- Неиспользуемые методы → **Inline Method**.
- Методы с неиспользуемыми параметрами → **Remove Parameter**.
- Неиспользуемые поля → удалить.

#### Ответ от Claude

**Определение и связь с YAGNI.**

Speculative Generality (Умозрительная обобщённость) — код-запах, описанный Мартином Фаулером в «Refactoring». Разработчик добавляет абстракции, параметры или классы, которые «могут понадобиться», не имея текущей потребности в них. Это прямое нарушение принципа **YAGNI (You Aren't Gonna Need It)** из Extreme Programming: не реализовывать функциональность, пока она реально не нужна.

**Форма 1: Неиспользуемый абстрактный класс / иерархия.**

```java
// ЗАПАХ: абстрактный класс с единственным подклассом,
// который никогда не будет расширяться
abstract class AbstractOrderProcessor {
    abstract void processOrder(Order order);
    abstract void validateOrder(Order order);
    // «на будущее», если появятся разные процессоры
}

class StandardOrderProcessor extends AbstractOrderProcessor {
    @Override void processOrder(Order order) { /* реализация */ }
    @Override void validateOrder(Order order) { /* реализация */ }
}

// Клиентский код никогда не работает с AbstractOrderProcessor напрямую

// РЕФАКТОРИНГ: Collapse Hierarchy — убираем лишний уровень
class OrderProcessor {
    void processOrder(Order order) { /* реализация */ }
    void validateOrder(Order order) { /* реализация */ }
}
```

**Форма 2: Неиспользуемые параметры метода.**

```java
// ЗАПАХ: параметр mode никогда не используется вызывающим кодом
public void exportData(List<Record> records, String format, boolean verbose) {
    // verbose добавлен «про запас», но всегда вызывается с false
    doExport(records, format);
}

// РЕФАКТОРИНГ: Remove Parameter
public void exportData(List<Record> records, String format) {
    doExport(records, format);
}
```

**Форма 3: Стратегии «на будущее» без альтернативных реализаций.**

```java
// ЗАПАХ: интерфейс с единственной реализацией, никогда не подменяемой
public interface TaxCalculationStrategy {
    BigDecimal calculate(Order order);
}

public class StandardTaxCalculationStrategy implements TaxCalculationStrategy {
    // единственная реализация; другие не планируются
}

// В конфигурации всегда хардкодится StandardTaxCalculationStrategy;
// интерфейс не используется ни в тестах, ни при инъекции

// РЕФАКТОРИНГ: Inline Class — убираем интерфейс, оставляем класс
public class TaxCalculator {
    public BigDecimal calculate(Order order) { ... }
}
```

**Допустимые исключения (когда Speculative Generality оправдана):**

1. **Публичный API / SDK:** расширяемость закладывается для внешних пользователей, которые не могут позволить себе ломающих изменений.
2. **Фреймворки:** точки расширения (`BeanPostProcessor` в Spring) создаются заранее — это намеренный дизайн.
3. **Требования тестируемости:** интерфейс может быть создан именно для того, чтобы в тестах подставлять mock — это легитимная причина.

**Как различить YAGNI-нарушение и хороший дизайн:**

| Ситуация | Оценка |
|---|---|
| Абстракция с единственной реализацией, не используется в тестах | Speculative Generality |
| Абстракция с двумя и более реализациями | Оправданная гибкость |
| Хук в публичном фреймворке | Оправданная расширяемость |
| Параметр передаётся при каждом вызове с одним и тем же значением | Remove Parameter |
| Параметр варьируется в разных вызовах | Нужный параметр |

**Связь с другими принципами:** Speculative Generality нарушает YAGNI; его устранение поддерживает KISS (Keep It Simple, Stupid) — код становится проще, понятнее и легче для сопровождения.
