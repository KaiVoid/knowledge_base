### Вопрос 1. Чем отличаются аннотации @BeforeEach и @BeforeAll в JUnit 5? Почему @BeforeAll-метод должен быть статическим?

**Категория:** Тестирование · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [BeforeAll (JUnit 5.5.2 API)](https://docs.junit.org/5.5.2/api/org/junit/jupiter/api/BeforeAll.html)

Аннотация `@BeforeAll` сигнализирует, что помеченный метод должен быть выполнен **один раз перед всеми тестами** в классе. Метод обязан иметь тип возвращаемого значения `void`, не может быть `private` и по умолчанию **должен быть `static`**. Требование статичности объясняется тем, что метод должен выполниться до создания каких-либо экземпляров тестового класса — то есть на уровне класса, а не на уровне объекта.

Исключение: аннотация не поддерживается во вложенных (`@Nested`) классах и не может быть методом по умолчанию в интерфейсе, **если только** тестовый класс не аннотирован `@TestInstance(Lifecycle.PER_CLASS)`. В этом режиме JUnit создаёт единственный экземпляр класса для всех тестов, и тогда `@BeforeAll`-метод может быть нестатическим.

`@BeforeEach`, напротив, выполняется **перед каждым отдельным тестом** (`@Test`, `@RepeatedTest`, `@ParameterizedTest` и т.д.), метод не может быть статическим.

#### Ответ от Claude

**@BeforeEach** и **@BeforeAll** — аннотации жизненного цикла тестового класса в JUnit 5, различающиеся частотой и условиями вызова.

| Критерий | `@BeforeEach` | `@BeforeAll` |
|---|---|---|
| Когда выполняется | Перед каждым тестом | Один раз перед всеми тестами класса |
| Статичность | **Запрещена** (метод не static) | **Обязательна** по умолчанию |
| Доступ к `this` | Есть (instance-метод) | Нет (static-метод) |
| Применение | Инициализация, которую нужно повторять | Дорогостоящая инициализация (БД, файлы) |

**Почему `@BeforeAll` требует `static`?**

По умолчанию JUnit 5 создаёт **новый экземпляр тестового класса для каждого теста** (lifecycle `PER_METHOD`). Это значит, что к моменту вызова `@BeforeAll` ни один объект тестового класса ещё не создан — метод выполняется на уровне класса. Статический метод не принадлежит конкретному экземпляру, поэтому JVM может вызвать его без создания объекта.

**Исключение: `@TestInstance(Lifecycle.PER_CLASS)`**

Если аннотировать класс `@TestInstance(Lifecycle.PER_CLASS)`, JUnit создаёт единственный экземпляр класса для всех тестов. В этом случае `@BeforeAll`-метод **может быть нестатическим**, так как экземпляр уже существует к моменту его вызова. Этот режим также необходим для работы `@BeforeAll` во вложенных `@Nested`-классах.

```java
// Стандартное использование (метод static)
class StandardTest {
    @BeforeAll
    static void initAll() { /* выполняется один раз */ }

    @BeforeEach
    void init() { /* выполняется перед каждым @Test */ }
}

// С @TestInstance(PER_CLASS) — метод может быть нестатическим
@TestInstance(TestInstance.Lifecycle.PER_CLASS)
class PerClassTest {
    @BeforeAll
    void initAll() { /* нестатический метод разрешён */ }
}
```

### Вопрос 2. Что делает аннотация @ExtendWith(MockitoExtension.class) и зачем она нужна при использовании Mockito с JUnit 5?

**Категория:** Тестирование · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Mockito: 3 Ways to Init Mock in JUnit 5 — Mincong Huang](https://mincong.io/2020/04/19/mockito-junit5/)

`@ExtendWith(MockitoExtension.class)` регистрирует Mockito как расширение JUnit 5, автоматизируя инициализацию моков. Расширение выполняет две ключевые функции:

1. **Инициализирует моки**, аннотированные `@Mock`, перед каждым тестом — без необходимости вручную вызывать `MockitoAnnotations.openMocks(this)`.
2. **Валидирует использование фреймворка** после каждого теста, выявляя проблемные паттерны — например, лишние стабы (unnecessary stubbings), которые объявлены, но не вызваны в тесте.

В отличие от ручного подхода через `openMocks()`, расширение автоматически включает режим **строгой проверки (strict stubbing)**: если стаб объявлен, но не используется в тесте, бросается `UnnecessaryStubbingException`. Это помогает поддерживать тесты в чистоте.

#### Ответ от Claude

**Что такое `@ExtendWith(MockitoExtension.class)`?**

В JUnit 5 механизм расширений (`Extension`) заменяет JUnit 4-й `@RunWith`. `@ExtendWith(MockitoExtension.class)` — это JUnit 5-эквивалент `@RunWith(MockitoJUnitRunner.class)` из JUnit 4.

**Что именно делает расширение:**

1. **Инициализация аннотированных полей**: автоматически создаёт моки для полей, аннотированных `@Mock`, `@Spy`, `@Captor`, и внедряет их в поле с `@InjectMocks` перед каждым тестом.

2. **Строгая проверка стабов (Strict Stubbing)**: по умолчанию включён режим `STRICT_STUBS`. Если стаб определён, но не был вызван ни в одном тестовом методе, тест упадёт с `UnnecessaryStubbingException`. Это ловит «мёртвые» стабы и помогает держать тесты актуальными.

3. **Валидация фреймворка**: после каждого теста расширение проверяет корректность использования Mockito (например, неверная последовательность вызовов).

**Сравнение трёх способов инициализации моков в JUnit 5:**

| Способ | Описание | Strict stubbing |
|---|---|---|
| `@ExtendWith(MockitoExtension.class)` | Декларативно, расширение JUnit 5 | Да (по умолчанию) |
| `MockitoAnnotations.openMocks(this)` в `@BeforeEach` | Ручной вызов | Нет |
| `Mockito.mock(...)` вручную | Программный, без аннотаций | Нет |

**Пример:**

```java
@ExtendWith(MockitoExtension.class)
class UserServiceTest {

    @Mock
    private UserRepository userRepository;

    @InjectMocks
    private UserService userService;

    @Test
    void shouldReturnUser() {
        when(userRepository.findById(1L)).thenReturn(Optional.of(new User(1L, "Alice")));
        User user = userService.getUser(1L);
        assertEquals("Alice", user.getName());
    }
}
```

Без `@ExtendWith` поля `userRepository` и `userService` остались бы `null`.

### Вопрос 3. В чём разница между @Mock и @Spy в Mockito? Когда использовать @Spy вместо @Mock?

**Категория:** Тестирование · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [@Spy — Mockito Javadoc](https://site.mockito.org/javadoc/current/org/mockito/Spy.html)

`@Spy` — сокращение для создания объекта-шпиона (spy). Шпион оборачивает **реальный экземпляр** объекта, позволяя отслеживать взаимодействия с ним, при этом по умолчанию выполняются реальные методы объекта.

Ключевые особенности `@Spy`:
- Для стабирования методов шпиона рекомендуется использовать семейство `doReturn|Answer|Throw()|CallRealMethod`, а не `when(spy.method())` — вызов через `when()` сначала выполнит реальный метод до подмены.
- Mockito **не делегирует вызовы** исходному переданному экземпляру, а создаёт его **копию**. Изменения исходного объекта после создания шпиона не повлияют на поведение шпиона.
- **Финальные методы** нельзя стабировать и верифицировать через шпиона.

#### Ответ от Claude

**@Mock** и **@Spy** — оба создают объекты, взаимодействие с которыми можно отслеживать (`verify`), но принципиально различаются по природе:

**@Mock — полная заглушка (fake object):**
- Создаёт **синтетический объект** без реальной реализации.
- Все методы по умолчанию возвращают пустые значения: `null`, `0`, `false`, пустые коллекции.
- Реальный код класса **никогда не выполняется**.
- Используется для полной изоляции тестируемого компонента от его зависимостей.

**@Spy — частичная заглушка (partial mock):**
- Оборачивает **реальный объект** (создаёт его копию).
- Все методы по умолчанию **вызывают реальную реализацию**.
- Отдельные методы можно переопределить стабами, остальные работают как обычно.
- Позволяет тестировать реальное поведение, избирательно заменяя часть методов.

**Сравнительная таблица:**

| Критерий | `@Mock` | `@Spy` |
|---|---|---|
| Основа | Синтетический объект | Копия реального объекта |
| Поведение по умолчанию | Возвращает null/0/false | Вызывает реальный метод |
| Реальный код выполняется | Никогда | Всегда (если не застаблен) |
| Когда использовать | Изоляция от зависимостей | Частичное мокирование |

**Синтаксис стабирования для `@Spy`:**

```java
// ОПАСНО: реальный метод вызовется до подмены
when(spy.getSize()).thenReturn(100);

// ПРАВИЛЬНО для @Spy: подмена без вызова реального метода
doReturn(100).when(spy).getSize();
```

**Когда использовать `@Spy` вместо `@Mock`:**

1. **Тестирование legacy-кода**, где нельзя легко разделить зависимости.
2. Нужно проверить реальное поведение объекта, но **заглушить один дорогостоящий или side-effect метод** (например, отправку email).
3. Частичное мокирование **конкретного класса**, когда большинство его методов должны работать по-настоящему.

**Важные ограничения `@Spy`:**
- Финальные методы (`final`) нельзя стабировать.
- Spy создаёт **копию** объекта, а не обёртку над оригиналом — изменения в исходном экземпляре не отражаются на шпионе.
- При использовании с `@ExtendWith(MockitoExtension.class)` класс должен иметь конструктор без аргументов, либо поле должно быть инициализировано явно.

```java
@ExtendWith(MockitoExtension.class)
class OrderServiceTest {

    @Spy
    private EmailSender emailSender = new EmailSender();

    @InjectMocks
    private OrderService orderService;

    @Test
    void shouldProcessOrderWithoutRealEmail() {
        doNothing().when(emailSender).send(anyString());

        orderService.placeOrder(new Order());

        verify(emailSender).send(anyString());
    }
}
```

### Вопрос 4. Что такое stub в контексте тестирования? Чем stub отличается от mock?

**Категория:** Тестирование · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Mocks Aren't Stubs — Martin Fowler](https://martinfowler.com/articles/mocksArentStubs.html)

Стабы (stubs) предоставляют заготовленные ответы на вызовы, сделанные во время теста; как правило, они не реагируют ни на что, кроме запрограммированного. Моки (mocks) — это объекты, заранее запрограммированные с ожиданиями, которые формируют спецификацию вызовов, которые они должны получить.

Ключевое различие: стаб использует **state verification** (проверку состояния объекта по завершении теста), а мок использует **behavior verification** (проверку факта и способа взаимодействия тестируемого кода с зависимостями). Стаб пассивно предоставляет данные; мок активно проверяет, что система под тестом взаимодействует с коллаборатором правильно.

#### Ответ от Claude

**Stub** — это тестовый двойник (test double), который возвращает заранее заданные значения при вызове его методов. Stub не проверяет, как именно и сколько раз был вызван метод — его цель исключительно в том, чтобы обеспечить тестируемый код нужными входными данными.

**Mock** — это тестовый двойник, который, помимо возможности задавать возвращаемые значения, фиксирует все взаимодействия с ним и позволяет после теста проверить: был ли метод вызван, с какими аргументами и сколько раз.

Принципиальные отличия:

| Критерий | Stub | Mock |
|---|---|---|
| Основная цель | Управление входными данными | Проверка взаимодействий |
| Тип верификации | State verification | Behavior verification |
| Проверяет вызовы | Нет | Да |
| Используется для | Изоляции зависимостей (контроль input) | Проверки поведения (control of interaction) |

**Важно:** в Mockito нет отдельного типа «stub» — всё создаётся через `mock()`. Объект, созданный `Mockito.mock()`, по умолчанию является моком. Если к нему применяется только `when(...).thenReturn(...)` без `verify()` — он фактически используется как stub. Если дополнительно вызывается `verify()` — как mock. Терминология stub/mock описывает **способ использования** объекта, а не способ его создания.

Пример stub:
```java
UserRepository stubRepo = mock(UserRepository.class);
when(stubRepo.findById(1L)).thenReturn(new User("Alice"));
// verify() не вызывается — это stub
```

Пример mock:
```java
EmailService mockEmail = mock(EmailService.class);
// ... выполнение тестируемого кода ...
verify(mockEmail).send("alice@example.com"); // проверяем взаимодействие
```

### Вопрос 5. Как работает конструкция when(...).thenReturn(...) в Mockito? Что вернёт метод mock-объекта, если для него не задан stub?

**Категория:** Тестирование · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Mockito Official Javadoc — site.mockito.org](https://site.mockito.org/javadoc/current/org/mockito/Mockito.html)

Конструкция `when(mock.method()).thenReturn(value)` настраивает мок так, чтобы при вызове указанного метода он возвращал заданное значение. Пример из документации:

```java
when(mockedList.get(0)).thenReturn("first");
```

По умолчанию для всех методов, возвращающих значение, мок вернёт: `null` — для ссылочных типов, значение по умолчанию для примитива или его обёртки (например, `0` для `int`/`Integer` и `false` для `boolean`/`Boolean`), а также пустую коллекцию — если метод возвращает коллекцию.

#### Ответ от Claude

**Как работает `when(...).thenReturn(...)`**

Это конструкция для настройки поведения mock-объекта (stubbing). При записи:

```java
when(mock.someMethod("arg")).thenReturn("result");
```

Mockito перехватывает вызов `someMethod("arg")` и запоминает его как «правило»: при следующем реальном вызове этого метода с таким аргументом вернуть `"result"`.

Механизм работы: при вызове `when(mock.someMethod(...))` Mockito переводит mock во внутренний режим записи. Вызов метода внутри `when(...)` не выполняет реальный код — он лишь регистрирует, какой метод будет настраиваться. После этого `.thenReturn(value)` завершает регистрацию правила.

Можно цепочкой задать разные значения для последовательных вызовов:

```java
when(mock.getValue())
    .thenReturn("first")
    .thenReturn("second"); // первый вызов → "first", второй и далее → "second"
```

**Что вернёт метод, для которого не задан stub?**

Mockito возвращает безопасные значения по умолчанию (never throws NullPointerException без причины):

| Тип возврата | Значение по умолчанию |
|---|---|
| `int`, `long`, `byte`, `short` | `0` |
| `double`, `float` | `0.0` |
| `boolean` | `false` |
| `char` | `''` |
| `String` и другие объекты | `null` |
| `List`, `Set`, `Map` и другие коллекции | пустая коллекция (`[]`, `{}`) |
| `Optional` | `Optional.empty()` |

Это поведение определено стратегией `RETURNS_DEFAULTS` (она же `ReturnsEmptyValues`). Его можно изменить при создании мока, передав альтернативную стратегию, например:

```java
Foo mock = mock(Foo.class, Mockito.RETURNS_SMART_NULLS);
```

`RETURNS_SMART_NULLS` возвращает не `null`, а «умные» объекты-заглушки, которые при повторном обращении бросают осмысленное исключение с подсказкой, где не настроен stub.

**Важный нюанс:** при использовании `when(spy.method())` реальный метод **вызывается** в момент записи. Чтобы избежать этого (например, для spy или void-методов), используется альтернативный синтаксис:

```java
doReturn("result").when(mock).someMethod("arg");
```

### Вопрос 6. Для чего используется Mockito.verify()? Как проверить, что метод был вызван ровно два раза?

**Категория:** Тестирование · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Mockito verify(): Test Verification Examples in Java — DigitalOcean](https://www.digitalocean.com/community/tutorials/mockito-verify)

`Mockito.verify()` используется для подтверждения того, что конкретные методы были вызваны на mock-объектах в ходе выполнения теста. Метод позволяет убедиться, что mock-методы вызываются с нужными аргументами, нужное количество раз и в нужном порядке.

Чтобы проверить, что метод вызван ровно два раза, используется режим верификации `times(2)`:

```java
verify(mockList, times(2)).add("value");
```

Если проверяется один вызов, `times(1)` можно опустить — это поведение по умолчанию. Также доступны режимы: `never()` (метод не должен был вызываться), `atLeastOnce()`, `atLeast(n)`, `atMost(n)`, `only()` (единственный метод, вызванный на моке).

#### Ответ от Claude

**Назначение `Mockito.verify()`**

`verify()` реализует **behavior verification** — проверку того, что тестируемый код действительно взаимодействовал с зависимостями нужным образом. Используется, когда нельзя проверить результат через возвращаемое значение — например, для void-методов, отправки событий, записи в лог, обращений к внешним сервисам.

```java
// Проверяем, что emailService.send() был вызван с конкретным адресом
verify(emailService).send("user@example.com");
```

**Как проверить, что метод вызван ровно два раза**

```java
import static org.mockito.Mockito.*;

verify(mock, times(2)).someMethod("arg");
```

`times(2)` — это объект типа `VerificationMode`, передаваемый вторым аргументом в `verify()`. Если метод был вызван не ровно 2 раза, тест упадёт с `WantedButNotInvoked` или `TooManyActualInvocations` с подробным сообщением.

**Полная таблица режимов верификации:**

| Режим | Смысл |
|---|---|
| `verify(mock).method()` | Ровно 1 раз (по умолчанию) |
| `verify(mock, times(n)).method()` | Ровно n раз |
| `verify(mock, never()).method()` | Никогда (0 раз) |
| `verify(mock, atLeastOnce()).method()` | Хотя бы 1 раз |
| `verify(mock, atLeast(n)).method()` | Хотя бы n раз |
| `verify(mock, atMost(n)).method()` | Не более n раз |
| `verify(mock, only()).method()` | Ровно 1 раз и это единственный вызов на моке |

**Пример полного теста:**

```java
@Test
void shouldSendTwoNotifications() {
    NotificationService notificationService = mock(NotificationService.class);
    OrderService orderService = new OrderService(notificationService);

    orderService.placeOrder(order); // внутри вызывает notify() дважды

    verify(notificationService, times(2)).notify(any(Order.class));
}
```

**Важно:** `verify()` проверяет вызовы, состоявшиеся **после создания мока**. Если нужно проверить порядок вызовов нескольких методов, используется `InOrder`:

```java
InOrder inOrder = inOrder(mock);
inOrder.verify(mock).firstMethod();
inOrder.verify(mock).secondMethod();
```

### Вопрос 7. Что такое @InjectMocks? Каким образом Mockito внедряет зависимости в аннотированный объект?

**Категория:** Тестирование · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [InjectMocks (Mockito — официальная документация)](https://site.mockito.org/javadoc/current/org/mockito/InjectMocks.html)

Аннотация `@InjectMocks` помечает поле, в которое Mockito должен автоматически внедрить моки (объекты, помеченные `@Mock` или `@Spy`). Mockito последовательно пробует три стратегии внедрения и останавливается на первой успешной:

1. **Constructor Injection (инъекция через конструктор)** — выбирается конструктор с наибольшим числом параметров; аргументы разрешаются по типу и имени из доступных моков. Если конструктор не найден или некоторые аргументы не могут быть сопоставлены, передаётся `null`.
2. **Property Setter Injection (инъекция через сеттеры)** — моки сначала сопоставляются по типу (если тип уникален — имя игнорируется), затем, при коллизии типов, — по имени поля/сеттера.
3. **Field Injection (инъекция напрямую в поле)** — аналогичная логика: сначала по типу, затем по имени.

Чтобы аннотации сработали, необходимо вызвать `MockitoAnnotations.openMocks(this)` в `@BeforeEach` либо пометить класс теста `@ExtendWith(MockitoExtension.class)`. Важное ограничение: Mockito **не сообщает об ошибке**, если ни одна стратегия не сработала — тест просто получит `null` в зависимости. Статические и финальные поля, а также абстрактные классы, интерфейсы и внутренние классы не поддерживаются.

#### Ответ от Claude

`@InjectMocks` — аннотация Mockito, указывающая, что в данное поле нужно внедрить все доступные в тесте объекты, помеченные `@Mock` или `@Spy`. Это позволяет тестировать класс в изоляции, заменяя его реальные зависимости моками.

**Порядок стратегий внедрения (строго последовательный):**

1. **Constructor Injection** — Mockito ищет конструктор с максимальным числом параметров, пытается сопоставить с доступными моками. Если подходящий конструктор найден — другие стратегии не проверяются. Несовпавшие параметры получают `null`.
2. **Setter Injection** — если конструктор не подошёл, Mockito перебирает сеттеры. При уникальном типе зависимости имя не важно; при совпадении типов происходит сопоставление по имени поля.
3. **Field Injection** — прямое присваивание в поле через рефлексию, по той же логике: тип → имя.

**Ключевые нюансы:**
- Если ни одна стратегия не завершилась успехом, Mockito **молча оставляет поле `null`** — это частая причина `NullPointerException` в тестах.
- Аннотация работает только совместно с `@ExtendWith(MockitoExtension.class)` (JUnit 5) или явным вызовом `MockitoAnnotations.openMocks(this)`.
- `@InjectMocks` **не является DI-фреймворком**: он не умеет строить граф зависимостей, работать с квалификаторами или профилями Spring. Для тестирования Spring-бинов следует использовать `@MockBean` из `spring-boot-test`.
- Поля `static` и `final`, а также абстрактные классы, интерфейсы и нестатические внутренние классы не поддерживаются.

**Пример:**
```java
@ExtendWith(MockitoExtension.class)
class OrderServiceTest {

    @Mock
    private PaymentRepository paymentRepository;

    @InjectMocks
    private OrderService orderService; // Mockito создаст экземпляр и внедрит paymentRepository

    @Test
    void shouldProcess() {
        when(paymentRepository.findById(1L)).thenReturn(Optional.of(new Payment()));
        orderService.process(1L);
        verify(paymentRepository).findById(1L);
    }
}
```

### Вопрос 8. Что такое пирамида тестирования? Назовите три уровня и объясните, почему количество тестов убывает снизу вверх.

**Категория:** Тестирование · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [TestPyramid — Martin Fowler](https://martinfowler.com/bliki/TestPyramid.html)

Пирамида тестирования — метафора, описывающая, как распределять автоматизированные тесты по уровням детализации. Мартин Фаулер формулирует главный принцип так: у основания должно быть **много мелких и быстрых юнит-тестов**, в середине — умеренное количество **сервисных/интеграционных тестов**, а на вершине — **минимум UI/end-to-end тестов**.

Три уровня (снизу вверх):
1. **Unit Tests** — низкоуровневые тесты отдельных компонентов.
2. **Service / Integration Tests** — тесты через сервисный слой приложения («подкожные тесты»); здесь проверяется взаимодействие нескольких компонентов.
3. **UI / End-to-End Tests** — тесты, управляющие приложением через пользовательский интерфейс.

Чем выше уровень, тем тесты дороже в написании и поддержке, медленнее в выполнении и более хрупки: изменение UI может сломать сразу много тестов. Фаулер предупреждает об «анти-паттерне перевёрнутого мороженого» — когда команды делают упор на UI-тестирование в ущерб юнит-тестам.

#### Ответ от Claude

**Пирамида тестирования** — концепция, введённая Майком Коном и популяризованная Мартином Фаулером. Она задаёт оптимальное соотношение автоматизированных тестов трёх уровней: много тестов в основании и мало на вершине.

**Три уровня (снизу вверх):**

| Уровень | Другие названия | Что проверяет | Типичные инструменты (Java) |
|---|---|---|---|
| **Unit (Юнит)** | Component | Отдельный класс/метод в изоляции | JUnit 5, Mockito |
| **Integration (Интеграционный)** | Service, Subcutaneous | Взаимодействие нескольких компонентов, работу с БД, брокерами | Spring Boot Test, Testcontainers |
| **End-to-End (Сквозной)** | UI, System, Broad-stack | Весь стек приложения через UI или API | Selenium, REST Assured, Cypress |

**Почему тестов становится меньше при движении вверх:**

1. **Скорость выполнения** — юнит-тест работает за миллисекунды; E2E-тест может занимать минуты. Тысячи медленных тестов делают сборку неприемлемо долгой.
2. **Стоимость написания и поддержки** — тест верхнего уровня хрупок: любое изменение UI/API может сломать десятки сценариев. Юнит-тест изолирован и изменяется только вместе с тестируемым классом.
3. **Детализация диагностики** — при падении юнит-теста сразу ясно, где именно ошибка. E2E-тест только сигнализирует «что-то сломалось», но не указывает причину.
4. **Недетерминизм** — тесты через сеть, UI или реальную БД флакуют (дают разные результаты при одинаковых данных), что подрывает доверие к тест-сьюту.
5. **Инфраструктурные затраты** — E2E-тесты требуют запущенного окружения (браузер, внешние сервисы, лицензии), что ограничивает возможности запуска на CI.

**Анти-паттерн «перевёрнутого мороженого»** — противоположность пирамиде: минимум юнит-тестов и максимум ручных/UI-тестов. Приводит к медленному циклу обратной связи и высоким затратам на поддержку.

**Примерное соотношение** (ориентировочное): 70% юнит, 20% интеграционные, 10% E2E.

### Вопрос 9. Как запустить один тест с несколькими наборами входных данных в JUnit 5? Какие аннотации-источники данных вы знаете (@ValueSource, @CsvSource, @MethodSource)?

**Категория:** Тестирование · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [CsvSource (JUnit 5.9.1 API — официальная документация)](https://docs.junit.org/5.9.1/api/org.junit.jupiter.params/org/junit/jupiter/params/provider/CsvSource.html)

`@CsvSource` — аннотация из пакета `org.junit.jupiter.params.provider`, которая передаёт CSV-данные в параметризованный тест. Значения задаются массивом строк, каждая строка — одна запись (строка CSV). По умолчанию разделитель — запятая, символ кавычки — одинарная кавычка. Поддерживаются атрибуты: `delimiter()`, `delimiterString()`, `quoteCharacter()`, `emptyValue()`, `nullValues()`, `maxCharsPerColumn()`, `useHeadersInDisplayName()`. Начиная с Java 15+ можно использовать атрибут `textBlock()` для многострочного блока данных.

Пример:
```java
@ParameterizedTest
@CsvSource({
    "apple,         1",
    "banana,        2",
    "'lemon, lime', 0xF1"
})
void test(String fruit, int rank) { ... }
```

#### Ответ от Claude

**Параметризованные тесты в JUnit 5** запускаются с помощью аннотации `@ParameterizedTest` в связке с одним из источников данных (`ArgumentsSource`). Тест выполняется столько раз, сколько наборов данных предоставляет источник.

**Подключение зависимости:**
```xml
<dependency>
    <groupId>org.junit.jupiter</groupId>
    <artifactId>junit-jupiter-params</artifactId>
    <version>5.10.2</version>
    <scope>test</scope>
</dependency>
```

---

**@ValueSource** — простейший источник: один массив литеральных значений одного типа. Подходит, когда тестовый метод принимает один аргумент.

Поддерживаемые типы: `shorts`, `bytes`, `ints`, `longs`, `floats`, `doubles`, `chars`, `booleans`, `strings`, `classes`.

```java
@ParameterizedTest
@ValueSource(strings = {"racecar", "radar", "level"})
void shouldBePalindrome(String candidate) {
    assertTrue(StringUtils.isPalindrome(candidate));
}
```

Ограничение: нельзя смешивать разные типы в одной аннотации и нельзя передать более одного аргумента за вызов.

---

**@CsvSource** — источник для нескольких аргументов разных типов. Каждая строка массива — отдельная запись CSV, значения разделяются запятой и автоматически конвертируются в типы параметров метода.

```java
@ParameterizedTest
@CsvSource({
    "Alice, 30, true",
    "Bob,   25, false",
    "'O''Brien', 40, true"   // экранирование одинарной кавычки
})
void shouldValidateUser(String name, int age, boolean active) {
    // ...
}
```

Атрибуты: `delimiter` (символ-разделитель), `quoteCharacter` (символ кавычки, по умолчанию `'`), `nullValues` (строки, трактуемые как `null`), `useHeadersInDisplayName` (первая строка как заголовки).

Вариант `@CsvFileSource` загружает данные из внешнего `.csv`-файла в classpath.

---

**@MethodSource** — наиболее гибкий источник: данные возвращает статический метод, который должен возвращать `Stream<Arguments>`, `Stream<T>`, `Iterable<T>` или массив.

```java
@ParameterizedTest
@MethodSource("provideInputs")
void shouldCalculate(int a, int b, int expected) {
    assertEquals(expected, Calculator.add(a, b));
}

static Stream<Arguments> provideInputs() {
    return Stream.of(
        Arguments.of(1, 2, 3),
        Arguments.of(0, 0, 0),
        Arguments.of(-1, 1, 0)
    );
}
```

Если имя метода не указано (`@MethodSource` без аргумента), JUnit ищет метод с тем же именем, что и тест. Метод-поставщик может находиться во внешнем классе: `@MethodSource("com.example.DataProvider#provideInputs")`.

---

**Другие аннотации-источники:**

| Аннотация | Назначение |
|---|---|
| `@EnumSource` | Все или выбранные константы перечисления |
| `@NullSource` | Передаёт `null` как единственный аргумент |
| `@EmptySource` | Передаёт пустую строку, коллекцию или массив |
| `@NullAndEmptySource` | Объединяет `@NullSource` и `@EmptySource` |
| `@ArgumentsSource` | Пользовательский провайдер, реализующий `ArgumentsProvider` |
| `@CsvFileSource` | CSV-данные из файла в classpath |
### Вопрос 10. Чем отличается @SpringBootTest от @WebMvcTest? В каком случае вы выберете каждую из них?

**Категория:** Тестирование · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [Testing :: Spring Boot Docs 2.1.5 — boot-features-testing](https://docs.spring.io/spring-boot/docs/2.1.5.RELEASE/reference/html/boot-features-testing.html)

`@WebMvcTest` автоматически конфигурирует инфраструктуру Spring MVC и ограничивает сканирование только MVC-слоем: `@Controller`, `@ControllerAdvice`, `@JsonComponent`, `Converter`, `GenericConverter`, `Filter`, `WebMvcConfigurer` и `HandlerMethodArgumentResolver`. Обычные `@Component`-бины при этом не регистрируются. Для работы контроллера нужны зависимости (сервисы и т.д.) — их необходимо объявлять через `@MockBean` или `@Import`.

`@SpringBootTest`, напротив, загружает полный контекст приложения — всё, что делает `@SpringBootApplication`. Чтобы использовать MockMvc совместно с ним, необходимо дополнительно указать `@AutoConfigureMockMvc`. Если нужно полное интеграционное тестирование с реальными слоями — выбирайте `@SpringBootTest`.

Пример с `@WebMvcTest`:
```java
@RunWith(SpringRunner.class)
@WebMvcTest(UserVehicleController.class)
public class MyControllerTests {

    @Autowired
    private MockMvc mvc;

    @MockBean
    private UserVehicleService userVehicleService;

    @Test
    public void testExample() throws Exception {
        given(this.userVehicleService.getVehicleDetails("sboot"))
            .willReturn(new VehicleDetails("Honda", "Civic"));
        this.mvc.perform(get("/sboot/vehicle").accept(MediaType.TEXT_PLAIN))
            .andExpect(status().isOk())
            .andExpect(content().string("Honda Civic"));
    }
}
```

#### Ответ от Claude

**Ключевое различие — область загружаемого Spring-контекста.**

| Характеристика | `@SpringBootTest` | `@WebMvcTest` |
|---|---|---|
| Загружаемый контекст | Полный (все бины) | Только MVC-слой |
| Скорость | Медленнее | Быстрее |
| MockMvc | Нужен `@AutoConfigureMockMvc` | Настраивается автоматически |
| Реальные сервисы | Доступны | Нет, нужно мокировать |
| Запуск сервера | Опционально (через `webEnvironment`) | Нет (mock-окружение) |

**`@WebMvcTest`** сканирует только: `@Controller`, `@ControllerAdvice`, `@JsonComponent`, `Converter`, `GenericConverter`, `Filter`, `HandlerInterceptor`, `HttpMessageConverter`, `WebMvcConfigurer`. Все зависимости контроллеров (сервисы, репозитории) необходимо объявлять через `@MockitoBean` (актуальный аналог устаревшего `@MockBean`) или подключать через `@Import`.

**`@SpringBootTest`** запускает полный контекст, аналогичный продуктивному. Поддерживает три режима через атрибут `webEnvironment`:
- `MOCK` (по умолчанию) — mock Servlet-окружение, требует `@AutoConfigureMockMvc`;
- `RANDOM_PORT` / `DEFINED_PORT` — запускает реальный встроенный сервер;
- `NONE` — контекст без веб-слоя.

**Когда выбирать `@WebMvcTest`:**
- Нужно протестировать логику конкретного контроллера в изоляции (маппинг URL, сериализацию/десериализацию, валидацию, обработку ошибок через `@ControllerAdvice`);
- Важна скорость: контекст поднимается значительно быстрее, так как не инициализируются база данных, репозитории и прочие бины.

**Когда выбирать `@SpringBootTest`:**
- Нужен сквозной интеграционный тест нескольких слоёв (контроллер → сервис → репозиторий);
- Тестируется реальное взаимодействие с базой данных или внешними системами;
- Требуется реалистичная конфигурация всего приложения.

> Начиная со Spring Boot 3.4, `@MockBean` устарел в пользу `@MockitoBean` из Spring Framework. В примерах с `@WebMvcTest` следует использовать именно `@MockitoBean`.

### Вопрос 11. Что такое TDD? Опишите цикл «Red — Green — Refactor».

**Категория:** Тестирование · **Уровень:** Junior

#### Оригинальный ответ из интернета
> Источник: [bliki: Test Driven Development — Martin Fowler](https://www.martinfowler.com/bliki/TestDrivenDevelopment.html)

Test-Driven Development (TDD) — это техника разработки программного обеспечения, при которой написание тестов управляет процессом разработки. Методика создана Кентом Беком в конце 1990-х годов в рамках Extreme Programming.

Цикл TDD состоит из трёх повторяющихся шагов:
1. **Red** — написать тест для следующего фрагмента функциональности;
2. **Green** — реализовать ровно столько кода, сколько нужно для прохождения теста;
3. **Refactor** — улучшить структуру как нового, так и существующего кода.

Ключевое достоинство TDD по Фаулеру: необходимость сначала подумать о тесте заставляет думать об интерфейсе кода раньше, чем о его реализации. Самая распространённая ошибка — пропуск шага рефакторинга, что ведёт к постепенному накоплению технического долга, несмотря на наличие тестов.

#### Ответ от Claude

**TDD (Test-Driven Development)** — это методология разработки программного обеспечения, при которой тест пишется **до** реализуемой функциональности. Суть не просто в покрытии кода тестами, а в том, что тесты становятся спецификацией: они описывают желаемое поведение ещё до того, как оно реализовано.

**Цикл Red — Green — Refactor:**

**1. Red (Красный) — написать падающий тест**
Пишется тест, описывающий желаемое поведение, которого ещё не существует. Тест обязан упасть (compile error или assertion failure). Это подтверждает, что тест реально проверяет нужное поведение, а не является ложноположительным.

```java
@Test
void shouldReturnSumOfTwoNumbers() {
    Calculator calc = new Calculator();
    assertEquals(5, calc.add(2, 3)); // Класса Calculator ещё нет — тест не компилируется
}
```

**2. Green (Зелёный) — написать минимальный код для прохождения теста**
Цель — заставить тест пройти как можно быстрее, не заботясь о качестве кода. Допускается жёсткое кодирование значений, дублирование — всё, что делает тест зелёным.

```java
public class Calculator {
    public int add(int a, int b) {
        return a + b; // Минимальная реализация
    }
}
```

**3. Refactor (Рефакторинг) — улучшить код, сохранив тест зелёным**
Устраняется дублирование, улучшается читаемость и структура — как нового кода, так и тестов. Тесты при этом остаются зелёными и выступают страховочной сетью.

**Важные принципы:**
- Один цикл — один маленький шаг. Не нужно писать сразу много тестов.
- Рефакторинг без страха: тесты гарантируют, что поведение не сломалось.
- TDD формирует дизайн «снаружи внутрь» (outside-in): сначала продумывается интерфейс, потом реализация.

**Преимущества TDD:**
- Тестируемый код с первого дня;
- Живая документация через тесты;
- Меньше регрессий при изменениях;
- Лучший дизайн за счёт принуждения к разделению ответственности.

**Распространённая ошибка** (по Фаулеру): пропуск фазы Refactor. В результате код обрастает дублированием и теряет читаемость, хотя тесты и проходят.

### Вопрос 12. Чем @MockBean в Spring Boot Test отличается от @Mock в Mockito? Когда необходимо использовать именно @MockBean?

**Категория:** Тестирование · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [MockBean (Spring Boot 3.5 API)](https://docs.spring.io/spring-boot/3.5/api/java/org/springframework/boot/test/mock/mockito/MockBean.html)

`@MockBean` — аннотация Spring Boot, которая добавляет Mockito-моки в Spring `ApplicationContext`. Её можно использовать на уровне класса или на полях в `@Configuration`-классах и тестовых классах.

Поведение при взаимодействии с контекстом:
- Если в контексте уже есть бин соответствующего типа — он **заменяется** моком;
- Если бина нет — **создаётся новый** мок-бин и регистрируется в контексте;
- При объявлении на поле мок автоматически инжектируется в это поле.

Аннотация **устарела (deprecated) начиная с Spring Boot 3.4.0** и будет удалена в 4.0.0. Вместо неё следует использовать `@MockitoBean` из Spring Framework (пакет `org.springframework.test.context.bean.override.mockito`).

#### Ответ от Claude

**Ключевое различие: `@Mock` работает вне Spring-контейнера, `@MockBean`/`@MockitoBean` — внутри него.**

| Характеристика | `@Mock` (Mockito) | `@MockBean` / `@MockitoBean` (Spring) |
|---|---|---|
| Spring ApplicationContext | Не нужен | Обязателен |
| Регистрация в контексте | Нет | Да — заменяет или создаёт бин |
| Внедрение зависимостей | Вручную через `@InjectMocks` | Автоматически через Spring DI |
| Перезапуск контекста | Не требуется | Может потребоваться (меняет контекст) |
| Использование | Чистые unit-тесты | Интеграционные тесты со Spring |

**`@Mock` (Mockito)** — создаёт объект-мок силами Mockito без участия Spring. Используется совместно с `@InjectMocks`: Mockito самостоятельно инжектирует моки в тестируемый объект. Требует либо аннотации `@ExtendWith(MockitoExtension.class)` (JUnit 5), либо вызова `MockitoAnnotations.openMocks(this)`.

```java
@ExtendWith(MockitoExtension.class)
class UserServiceTest {

    @Mock
    private UserRepository userRepository;

    @InjectMocks
    private UserService userService; // Mockito внедрит userRepository

    @Test
    void shouldFindUser() {
        when(userRepository.findById(1L)).thenReturn(Optional.of(new User()));
        assertNotNull(userService.findById(1L));
    }
}
```

**`@MockBean` / `@MockitoBean` (Spring Boot)** — регистрирует мок непосредственно в Spring ApplicationContext. Если бин такого типа уже существует — он заменяется моком на время теста; если нет — создаётся новый. Используется, когда тест поднимает Spring-контекст (`@SpringBootTest`, `@WebMvcTest` и т.д.) и нужно подменить реальный бин моком.

```java
@WebMvcTest(UserController.class)
class UserControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @MockitoBean // актуальная аннотация вместо устаревшей @MockBean
    private UserService userService;

    @Test
    void shouldReturnUser() throws Exception {
        when(userService.findById(1L)).thenReturn(new UserDto("Alice"));
        mockMvc.perform(get("/users/1"))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$.name").value("Alice"));
    }
}
```

**Когда необходимо использовать `@MockBean` / `@MockitoBean`:**
1. Тест использует Spring-контекст (`@WebMvcTest`, `@SpringBootTest`) и нужно заменить реальный бин (например, сервис, обращающийся к БД или внешнему API) моком;
2. Тестируется контроллер в изоляции от сервисного слоя — `@WebMvcTest` не загружает сервисы, но контроллер требует их для работы;
3. Нужно убедиться, что другие бины в контексте получат именно мок (а не реальную реализацию) через внедрение зависимостей Spring.

**Когда достаточно `@Mock`:**
- Тест — чистый unit-тест без Spring-контекста;
- Тестируемый класс создаётся вручную или через `@InjectMocks`;
- Нет необходимости в полноценном контексте приложения.

**Важно о deprecation:**
Начиная со Spring Boot 3.4.0, аннотация `@MockBean` помечена как устаревшая и будет удалена в Spring Boot 4.0.0. Её замена — `@MockitoBean` из Spring Framework (`org.springframework.test.context.bean.override.mockito.MockitoBean`). Переход не является полностью прозрачным: `@MockitoBean` не поддерживается на `@Configuration`-классах и `@Component`-классах, в отличие от `@MockBean`.
### Вопрос 13. Как с помощью MockMvc написать тест для GET-эндпоинта, который возвращает JSON? Какие методы используются для проверки статуса и тела ответа?

**Категория:** Тестирование · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Defining Expectations :: Spring Framework](https://docs.spring.io/spring-framework/reference/testing/mockmvc/hamcrest/expectations.html)

MockMvc позволяет определять ожидания после выполнения запроса с помощью вызовов `andExpect(..)` или `andExpectAll(..)`. Базовый синтаксис:

```java
// Одно ожидание — остановится на первой ошибке
mockMvc.perform(get("/accounts/1")).andExpect(status().isOk());

// Несколько ожиданий — проверит все и сообщит обо всех ошибках
mockMvc.perform(get("/accounts/1")).andExpectAll(
    status().isOk(),
    content().contentType("application/json;charset=UTF-8"));
```

Для проверки тела JSON-ответа используется `jsonPath()`:

```java
mockMvc.perform(get("/people").accept(MediaType.APPLICATION_JSON))
    .andExpect(jsonPath("$.links[?(@.rel == 'self')].href")
        .value("http://localhost:8080/people"));
```

Для получения объекта `MvcResult` (если нужно дополнительно обработать ответ) используется `andReturn()`.

#### Ответ от Claude

MockMvc — это инструмент из `spring-test`, позволяющий тестировать Spring MVC контроллеры без запуска реального HTTP-сервера. Тест для GET-эндпоинта, возвращающего JSON, строится по следующей схеме:

**1. Настройка теста**

```java
@WebMvcTest(UserController.class)   // загружает только слой MVC
class UserControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @MockBean
    private UserService userService;  // мокаем сервисный слой
}
```

**2. Написание теста**

```java
@Test
void getUser_returnsJsonWithStatus200() throws Exception {
    User user = new User(1L, "Alice");
    when(userService.findById(1L)).thenReturn(user);

    mockMvc.perform(
            get("/users/1")
                .accept(MediaType.APPLICATION_JSON))
        .andExpect(status().isOk())                          // HTTP 200
        .andExpect(content().contentType(
            MediaType.APPLICATION_JSON))                     // Content-Type
        .andExpect(jsonPath("$.id").value(1))                // поле id
        .andExpect(jsonPath("$.name").value("Alice"));       // поле name
}
```

**Ключевые методы:**

| Метод | Назначение |
|---|---|
| `mockMvc.perform(get("/path"))` | Выполнить GET-запрос |
| `.accept(MediaType.APPLICATION_JSON)` | Установить заголовок `Accept` |
| `.andExpect(status().isOk())` | Проверить HTTP-статус 200 |
| `.andExpect(status().is(404))` | Проверить произвольный статус |
| `.andExpect(content().contentType(...))` | Проверить `Content-Type` |
| `.andExpect(jsonPath("$.field").value(...))` | Проверить поле JSON по JSONPath |
| `.andExpect(jsonPath("$.list", hasSize(3)))` | Проверить размер массива (с Hamcrest) |
| `.andReturn()` | Получить `MvcResult` для ручной обработки |
| `.andDo(print())` | Вывести детали запроса/ответа в консоль |

**`andExpect` vs `andExpectAll`:**
- `andExpect` — прерывается на первой неудавшейся проверке;
- `andExpectAll` — выполняет все проверки и сообщает обо всех ошибках сразу (добавлен в Spring 5.3).

**JSONPath-синтаксис:** `$` — корень документа, `.field` — поле объекта, `[0]` — элемент массива, `[?(@.x == 'y')]` — фильтр. Требует зависимости `com.jayway.jsonpath:json-path` в classpath (входит в `spring-boot-starter-test`).

### Вопрос 14. Что делает @DataJpaTest? Какую базу данных он использует по умолчанию и как подключить реальную БД для этих тестов?

**Категория:** Тестирование · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [DataJpaTest (Spring Boot 4.0.6 API)](https://docs.spring.io/spring-boot/api/java/org/springframework/boot/data/jpa/test/autoconfigure/DataJpaTest.html)

`@DataJpaTest` — аннотация для тестирования JPA-компонентов. Она:
- Активирует только конфигурацию, релевантную для JPA-тестов;
- Ограничивает сканирование компонентов: подхватывает `@Entity`-классы и Spring Data-репозитории;
- Делает тесты транзакционными с откатом в конце каждого теста;
- По умолчанию заменяет любой явно настроенный или автоматически сконфигурированный `DataSource` встроенной базой данных в памяти;
- Включает логирование SQL-запросов.

Чтобы использовать реальную БД вместо встроенной, применяется аннотация `@AutoConfigureTestDatabase`:

```java
@DataJpaTest
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)
class MyRepositoryTest {
    // использует настоящий DataSource из application.properties
}
```

#### Ответ от Claude

**Что делает `@DataJpaTest`:**

Аннотация загружает урезанный контекст Spring, содержащий только компоненты JPA-слоя:
- Сканирует и регистрирует `@Entity`-классы и Spring Data JPA репозитории;
- Автоконфигурирует Hibernate, `DataSource`, `JdbcTemplate`, `TestEntityManager`;
- Исключает `@Service`, `@Controller`, `@Component` и прочие бины, не связанные с JPA;
- Каждый тест по умолчанию оборачивается в транзакцию и откатывается (`@Transactional` + rollback), что обеспечивает изоляцию тестов.

**База данных по умолчанию:**

По умолчанию `@DataJpaTest` подменяет любой настроенный `DataSource` **встроенной in-memory базой данных** — как правило, **H2**. Это позволяет запускать тесты без внешней инфраструктуры. Для работы с H2 нужна зависимость `com.h2database:h2` в `test`-scope.

**Как подключить реальную БД:**

Используется аннотация `@AutoConfigureTestDatabase(replace = Replace.NONE)`, которая запрещает подмену `DataSource`:

```java
@DataJpaTest
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)
class UserRepositoryIntegrationTest {

    @Autowired
    private UserRepository userRepository;

    @Test
    void shouldSaveAndFindUser() {
        User user = userRepository.save(new User("Bob"));
        assertThat(userRepository.findById(user.getId())).isPresent();
    }
}
```

При этом Spring использует `DataSource`, настроенный в `application.properties` (или `application-test.properties`). Для указания отдельного URL тестовой БД:

```java
@DataJpaTest
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)
@TestPropertySource(properties = {
    "spring.datasource.url=jdbc:postgresql://localhost:5432/testdb",
    "spring.datasource.username=test",
    "spring.datasource.password=test"
})
class MyRepositoryTest { ... }
```

**Альтернатива — Testcontainers:** для интеграционных тестов против реальной СУБД (PostgreSQL, MySQL и т.д.) популярен подход с Testcontainers, который поднимает Docker-контейнер с базой данных на время теста.

**`TestEntityManager`:** `@DataJpaTest` предоставляет специальный `TestEntityManager` — обёртку над `EntityManager` с удобными методами (`persistFlushFind`, `persistAndFlush`) для подготовки состояния БД в тестах.

### Вопрос 15. В чём разница между thenReturn и doReturn в Mockito? В каких случаях thenReturn не подходит и приходится использовать doReturn?

**Категория:** Тестирование · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Mockito: doReturn vs thenReturn](http://sangsoonam.github.io/2019/02/04/mockito-doreturn-vs-thenreturn.html)

**`when-thenReturn`** — предпочтительный подход для обычных моков, так как обеспечивает типобезопасность на этапе компиляции:

```java
User user = Mockito.mock(User.class);
when(user.getName()).thenReturn("John");
```

**`doReturn-when`** — не проверяет тип возвращаемого значения во время компиляции (принимает `Object`), но имеет принципиальное преимущество при работе со шпионами (spy):

```java
doReturn("foo").when(spy).get(0);
```

Ключевое различие — **side effect (побочный эффект)**. В конструкции `when(spy.get(0)).thenReturn("foo")` параметром `when()` является результат вызова реального метода `spy.get(0)`. То есть настоящий метод вызывается в момент настройки стаба. Если `spy` обёрнут над пустым списком, `get(0)` бросит `IndexOutOfBoundsException` ещё до того, как стаб будет зарегистрирован.

`doReturn("foo").when(spy).get(0)` передаёт только ссылку на объект, реальный метод не вызывается.

**Рекомендация:** для обычных моков используйте `when-thenReturn` (типобезопасно), для шпионов — `doReturn-when` (без побочных эффектов).

#### Ответ от Claude

**Синтаксическое различие:**

```java
// Стиль thenReturn (fluent, рекомендован для моков)
when(mockObject.method()).thenReturn(value);

// Стиль doReturn (для шпионов и особых случаев)
doReturn(value).when(spyObject).method();
```

**Три ситуации, где `thenReturn` не подходит:**

**1. Метод — void.**
`thenReturn` нельзя использовать с void-методами в принципе — компилятор не позволит. Для void используются `doNothing()`, `doThrow()`, `doAnswer()`:

```java
// Компилируется
doNothing().when(mockService).sendEmail(any());
doThrow(new RuntimeException()).when(mockService).sendEmail(any());
// НЕ компилируется:
// when(mockService.sendEmail(any())).thenReturn(...); — ошибка компиляции
```

**2. Шпион (`spy`) и реальный метод бросает исключение или имеет побочный эффект.**
При записи `when(spy.method()).thenReturn(x)` реальный метод `spy.method()` вызывается немедленно — в момент передачи аргумента в `when()`. Это приводит к исключению или нежелательному побочному эффекту (запись в БД, сетевой вызов и т.п.):

```java
List<String> realList = new ArrayList<>();
List<String> spy = Mockito.spy(realList);

// ОПАСНО: реальный get(0) вызван на пустом списке → IndexOutOfBoundsException
when(spy.get(0)).thenReturn("first");  // бросит исключение

// ПРАВИЛЬНО: реальный метод не вызывается
doReturn("first").when(spy).get(0);
```

**3. Переопределение стаба final-метода или метода с особым поведением.**
В редких случаях с final-методами или специфическими Mockito-расширениями `doReturn` также предпочтительнее.

**Почему `thenReturn` лучше для обычных моков:**
- **Типобезопасность:** `thenReturn(T value)` — параметр типизирован, несовпадение типов ловится компилятором. У `doReturn(Object value)` тип не проверяется, ошибка обнаруживается только в рантайме (`WrongTypeOfReturnValue`).
- **Читаемость:** стиль `when(...).thenReturn(...)` читается как естественный язык.

**Итоговое правило:**

| Ситуация | Используй |
|---|---|
| Обычный `mock`, ненулевой возврат | `when(...).thenReturn(...)` |
| Шпион (`spy`) | `doReturn(...).when(...)` |
| void-метод | `doNothing()` / `doThrow()` / `doAnswer()` |
| Нужно перехватить и вычислить результат | `when(...).thenAnswer(...)` / `doAnswer(...).when(...)` |
### Вопрос 16. Как замокировать метод, выбрасывающий исключение, с помощью Mockito? Приведите пример для void-метода и для метода с возвращаемым значением.

**Категория:** Тестирование · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Mockito Javadoc — класс Mockito (официальный сайт Mockito)](https://site.mockito.org/javadoc/current/org/mockito/Mockito.html)

Официальная документация Mockito описывает два разных синтаксиса в зависимости от того, является ли метод void или возвращает значение.

**Для метода с возвращаемым значением** используется цепочка `when(...).thenThrow(...)`:

```java
when(mockedList.get(1)).thenThrow(new RuntimeException());
// следующий вызов выбросит RuntimeException:
System.out.println(mockedList.get(1));
```

**Для void-метода** компилятор Java не допускает вызов void-метода внутри скобок `when(...)`, поэтому используется синтаксис `doThrow(...).when(...)`:

```java
doThrow(new RuntimeException()).when(mockedList).clear();
// следующий вызов выбросит RuntimeException:
mockedList.clear();
```

Документация также упоминает семейство методов `do*()` (`doThrow()`, `doAnswer()`, `doNothing()`, `doReturn()`, `doCallRealMethod()`) как альтернативу стандартному `when()`; они необходимы при заглушке void-методов, методов на spy-объектах или при изменении поведения в ходе теста.

#### Ответ от Claude

Mockito предоставляет два синтаксических варианта для настройки выброса исключений, и выбор между ними определяется возвращаемым типом метода.

**1. Метод с возвращаемым значением — `when(...).thenThrow(...)`**

Это стандартный и предпочтительный синтаксис для non-void методов:

```java
List<String> mockedList = mock(List.class);

// передача экземпляра исключения
when(mockedList.get(0)).thenThrow(new IllegalArgumentException("bad index"));

// передача класса исключения (Mockito создаёт экземпляр сам; работает с Mockito 2+)
when(mockedList.get(1)).thenThrow(RuntimeException.class);

// тест
assertThrows(IllegalArgumentException.class, () -> mockedList.get(0));
```

При передаче класса проверяемого исключения (checked) Mockito проверяет, что оно объявлено в сигнатуре метода; нарушение приводит к `MockitoException` во время настройки стаба.

**2. Void-метод — `doThrow(...).when(...)`**

Компилятор Java не разрешает использовать void-выражение как аргумент, поэтому синтаксис `when(mock.voidMethod())` не компилируется. Необходим инвертированный порядок:

```java
EmailService emailService = mock(EmailService.class);

// передача экземпляра
doThrow(new RuntimeException("SMTP down")).when(emailService).sendEmail(anyString(), anyString());

// передача класса (Mockito создаёт новый экземпляр при каждом вызове)
doThrow(IllegalStateException.class).when(emailService).sendEmail(anyString(), anyString());

// тест
assertThrows(RuntimeException.class,
        () -> emailService.sendEmail("to@example.com", "Hello"));
```

**3. Последовательные вызовы: первый проходит, второй бросает исключение**

```java
doNothing()
    .doThrow(new RuntimeException("second call fails"))
    .when(emailService).sendEmail(anyString(), anyString());
```

**4. BDD-стиль через `BDDMockito`**

```java
// non-void
given(mockedList.get(0)).willThrow(new RuntimeException());

// void
willThrow(new RuntimeException()).given(emailService).sendEmail(anyString(), anyString());
```

**Ключевые правила:**

| Тип метода | Синтаксис | Примечание |
|---|---|---|
| Non-void | `when(mock.method()).thenThrow(...)` | Стандартный, компилируется |
| Void | `doThrow(...).when(mock).method()` | Единственный вариант для void |
| Void (BDD) | `willThrow(...).given(mock).method()` | BDDMockito-псевдоним |

**Важная деталь:** При передаче класса исключения Mockito создаёт новый экземпляр для каждого вызова метода. При передаче экземпляра — повторно использует один и тот же объект. Для проверяемых исключений (checked exceptions) класс должен совпадать с объявленным в сигнатуре метода, иначе Mockito выбросит исключение ещё на этапе настройки стаба.

### Вопрос 17. Что такое ArgumentCaptor в Mockito и когда его применяют вместо обычного verify с матчерами?

**Категория:** Тестирование · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ArgumentCaptor — официальная документация Mockito (site.mockito.org)](https://site.mockito.org/javadoc/current/org/mockito/ArgumentCaptor.html)

`ArgumentCaptor` позволяет захватить аргументы, переданные мок-объекту, чтобы затем выполнить по ним утверждения (assertions). Используется совместно с `verify()`:

```java
ArgumentCaptor<Person> argument = ArgumentCaptor.forClass(Person.class);
verify(mock).doSomething(argument.capture());
assertEquals("John", argument.getValue().getName());
```

Для нескольких вызовов применяют `getAllValues()`:

```java
ArgumentCaptor<Person> peopleCaptor = ArgumentCaptor.forClass(Person.class);
verify(mock, times(2)).doSomething(peopleCaptor.capture());
List<Person> capturedPeople = peopleCaptor.getAllValues();
assertEquals("John", capturedPeople.get(0).getName());
assertEquals("Jane", capturedPeople.get(1).getName());
```

Документация явно предостерегает: `ArgumentCaptor` следует использовать только с `verify()`, **не** со стабами (`when()`), так как это снижает читаемость теста — каптор создаётся вне блока утверждений, что затрудняет понимание кода.

`ArgumentCaptor` наиболее уместен, когда кастомный матчер скорее всего не будет переиспользоваться, и нужно лишь проверить значения аргументов в блоке верификации.

#### Ответ от Claude

**Что такое ArgumentCaptor**

`ArgumentCaptor<T>` — специальный инструмент Mockito для захвата (capture) фактических аргументов, переданных мок-объекту при вызове метода. Он позволяет после верификации вызова получить точное значение аргумента и применить к нему любые assertions.

**Как работает**

1. Создать каптор: `ArgumentCaptor<T> captor = ArgumentCaptor.forClass(T.class);`  
   Или через аннотацию: `@Captor ArgumentCaptor<T> captor;` (требует `MockitoAnnotations.openMocks(this)` или `@ExtendWith(MockitoExtension.class)`).
2. Использовать `captor.capture()` как аргумент в `verify(...)`.
3. После верификации вызвать `captor.getValue()` (последний захваченный аргумент) или `captor.getAllValues()` (все захваченные значения при нескольких вызовах).

```java
@ExtendWith(MockitoExtension.class)
class OrderServiceTest {

    @Mock
    EmailService emailService;

    @Captor
    ArgumentCaptor<Email> emailCaptor;

    @InjectMocks
    OrderService orderService;

    @Test
    void shouldSendConfirmationEmailWithCorrectData() {
        orderService.placeOrder(new Order("item-1", 2));

        verify(emailService).send(emailCaptor.capture());

        Email sent = emailCaptor.getValue();
        assertEquals("order-confirmation@shop.com", sent.getFrom());
        assertTrue(sent.getSubject().contains("item-1"));
        assertEquals(2, sent.getItemCount());
    }
}
```

**Когда ArgumentCaptor предпочтительнее `verify` с матчерами**

| Ситуация | Рекомендация |
|---|---|
| Нужно проверить **несколько полей** сложного объекта-аргумента | `ArgumentCaptor` — матчер `argThat(...)` стал бы громоздким |
| Логика проверки **одноразовая**, не переиспользуется | `ArgumentCaptor` (создать кастомный `ArgumentMatcher` излишне) |
| Нужно получить аргумент для **дальнейших вычислений** или проверок | `ArgumentCaptor` — матчер не возвращает значение |
| Логика сравнения **повторяется** в нескольких тестах | Кастомный `ArgumentMatcher` через `argThat(...)` |
| **Стабирование** (`when(...)`) | Только `ArgumentMatcher`; использовать `ArgumentCaptor` со стабами — антипаттерн |

**Важные ограничения и антипаттерны**

- `capture()` вызывается **только внутри `verify()`**, не внутри `when()`. Использование каптора при стабировании — официально задокументированный антипаттерн: каптор создаётся вне блока утверждений, что делает тест трудночитаемым.
- `getValue()` возвращает значение **последнего** вызова; для нескольких вызовов используйте `getAllValues()`.
- При использовании ArgumentCaptor с varargs необходимо учитывать, что `getAllValues()` возвращает все элементы varargs-массива как плоский список.

**Кастомный ArgumentMatcher как альтернатива**

```java
// Матчер — хорошо, если логика переиспользуется:
verify(emailService).send(argThat(email ->
        email.getFrom().equals("order-confirmation@shop.com")
        && email.getSubject().contains("item-1")));
```

Матчер лаконичнее для простой проверки одного условия, но не позволяет инспектировать несколько свойств объекта так же удобно, как ArgumentCaptor.

### Вопрос 18. Как изменить жизненный цикл экземпляра тестового класса в JUnit 5 с помощью @TestInstance? Какие у этого практические последствия?

**Категория:** Тестирование · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Test Instance Lifecycle — JUnit 5 User Guide (docs.junit.org)](https://docs.junit.org/current/writing-tests/test-instance-lifecycle.html)

По умолчанию JUnit Jupiter создаёт новый экземпляр тестового класса перед выполнением каждого тестового метода. Это поведение называется режимом `PER_METHOD`.

Для изменения жизненного цикла используется аннотация `@TestInstance(Lifecycle.PER_CLASS)`. В этом режиме один экземпляр тестового класса создаётся один раз на весь класс и используется для всех тестовых методов. Если тестовые методы зависят от состояния в переменных экземпляра, это состояние может накапливаться между тестами, поэтому может потребоваться сброс в `@BeforeEach` или `@AfterEach`.

Режим `PER_CLASS` позволяет объявлять `@BeforeAll` и `@AfterAll` на **нестатических методах**, а также на методах `default` интерфейсов. Это особенно полезно для разработчиков на Kotlin.

Изменить режим по умолчанию для всего проекта можно тремя способами:
1. JVM-системное свойство: `-Djunit.jupiter.testinstance.lifecycle.default=per_class`
2. Через `LauncherDiscoveryRequest`
3. Файл конфигурации (рекомендуется): `junit-platform.properties` в корне classpath с записью `junit.jupiter.testinstance.lifecycle.default = per_class`

#### Ответ от Claude

**Аннотация @TestInstance**

`@TestInstance` — аннотация уровня класса, появившаяся в JUnit 5. Она управляет тем, как часто JUnit создаёт экземпляр тестового класса.

**Два режима**

`Lifecycle.PER_METHOD` (по умолчанию):
- Новый экземпляр создаётся **перед каждым тестовым методом**.
- Гарантирует полную изоляцию тестов: любое состояние в полях класса сбрасывается автоматически.
- `@BeforeAll` и `@AfterAll` должны быть `static` (или объявлены на `static`-методах интерфейса).

`Lifecycle.PER_CLASS`:
- Один экземпляр создаётся **один раз для всего тестового класса**.
- `@BeforeAll` и `@AfterAll` могут быть **нестатическими** — это главное практическое преимущество.
- Состояние в полях экземпляра **сохраняется между тестами**, что может приводить к взаимному влиянию тестов (test pollution), если им явно не управлять.

```java
@TestInstance(TestInstance.Lifecycle.PER_CLASS)
class DatabaseIntegrationTest {

    // не нужен static — работает в PER_CLASS
    Connection connection;

    @BeforeAll
    void initDb() throws Exception {
        // дорогостоящая операция — выполняется один раз
        connection = DriverManager.getConnection("jdbc:h2:mem:testdb");
    }

    @AfterAll
    void closeDb() throws Exception {
        connection.close();
    }

    @BeforeEach
    void cleanUp() throws Exception {
        // сброс состояния между тестами — ОБЯЗАТЕЛЕН при PER_CLASS
        connection.createStatement().execute("DELETE FROM orders");
    }

    @Test
    void testInsert() { /* ... */ }

    @Test
    void testSelect() { /* ... */ }
}
```

**Практические последствия и сравнение**

| Аспект | PER_METHOD (default) | PER_CLASS |
|---|---|---|
| Создание экземпляра | Перед каждым тестом | Один раз на класс |
| Изоляция тестов | Автоматическая | Требует ручного сброса в `@BeforeEach` |
| `@BeforeAll` / `@AfterAll` | Должны быть `static` | Могут быть нестатическими |
| Производительность | Хуже при дорогостоящей инициализации | Лучше при разделяемых ресурсах |
| Совместное состояние между тестами | Невозможно через поля экземпляра | Возможно (и опасно) |
| Параллельное выполнение | Безопасно по умолчанию | Требует `@Execution(CONCURRENT)` явно |
| Kotlin / нестатические утилиты | Проблематично | Удобно |

**Когда использовать PER_CLASS**

1. Инициализация дорогостоящего ресурса (соединение с БД, запуск контейнера, загрузка файла) — ресурс создаётся один раз в нестатическом `@BeforeAll`.
2. Тесты в стиле BDD/сценарий, где методы намеренно используют общее состояние (однако это спорная практика).
3. Kotlin-тесты, где `companion object` для статических методов нежелателен.
4. Тестовые классы, реализующие интерфейсы с `default`-методами `@BeforeAll`/`@AfterAll`.

**Предупреждение:** Режим `PER_CLASS` не отменяет необходимость думать об изоляции. Без явного `@BeforeEach`-сброса состояния тесты будут зависеть от порядка выполнения, что делает набор тестов хрупким и труднодиагностируемым.

**Глобальная настройка через `junit-platform.properties`:**

```properties
junit.jupiter.testinstance.lifecycle.default = per_class
```

Этот файл размещается в `src/test/resources/` и версионируется вместе с кодом, что делает его предпочтительным перед JVM-флагом.
### Вопрос 19. В чём разница между unit-тестом и интеграционным тестом на практике? Как в Spring Boot изолировать интеграционный тест от реальных внешних зависимостей?

**Категория:** Тестирование · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Testing · Spring Boot Reference Documentation (2.1.5)](https://docs.spring.io/spring-boot/docs/2.1.5.RELEASE/reference/html/boot-features-testing.html)

Из официальной документации Spring Boot: «Одно из главных преимуществ внедрения зависимостей заключается в том, что оно упрощает unit-тестирование. Вы можете создавать объекты оператором `new`, не задействуя Spring вовсе. Вместо реальных зависимостей можно использовать mock-объекты. Часто, однако, возникает необходимость выйти за рамки unit-тестирования и перейти к интеграционному тестированию (с `ApplicationContext` Spring).»

Интеграционные тесты в Spring Boot запускаются с помощью `@SpringBootTest`, который поднимает полный `ApplicationContext`. Атрибут `webEnvironment` позволяет контролировать среду: использовать mock-окружение (`MockMvc`, по умолчанию) или запускать реальный сервер на случайном порту (`RANDOM_PORT`). Для изоляции от внешних зависимостей применяется `@MockBean` — он подменяет реальный бин в контексте Mockito-mock-объектом.

#### Ответ от Claude

**Практическое отличие unit-теста от интеграционного теста**

| Критерий | Unit-тест | Интеграционный тест |
|---|---|---|
| Область | Один класс / метод в изоляции | Несколько компонентов или вся система |
| Spring-контекст | Не нужен | Загружается (полностью или частично) |
| Зависимости | Заменяются mock/stub вручную | Реальные или управляемые заменители |
| Скорость | Очень быстро (мс) | Медленнее (сотни мс — секунды) |
| Инструменты | JUnit + Mockito `@Mock` / `@InjectMocks` | `@SpringBootTest`, Test Slices, Testcontainers, WireMock |

*Практический признак*: если для теста нужен поднятый Spring-контекст (`@SpringBootTest` или любой slice-аннотация) — это интеграционный тест.

**Способы изоляции интеграционного теста от реальных внешних зависимостей**

1. **`@MockBean` / `@MockitoBean`** — подменяет конкретный Spring-бин Mockito-mock-объектом прямо в контексте. Подходит для сервисов, репозиториев, HTTP-клиентов. Важно: каждое уникальное сочетание `@MockBean` создаёт отдельный `ApplicationContext` в кэше, что замедляет прогон при большом числе тестов.

2. **Встроенная база данных (H2 / HSQLDB)** — `@DataJpaTest` по умолчанию заменяет настоящую БД встроенной in-memory базой. Для `@SpringBootTest` можно указать `spring.datasource.url=jdbc:h2:mem:testdb` в `application-test.properties`.

3. **Testcontainers** — запускает реальный Docker-контейнер (PostgreSQL, Redis, Kafka и т.д.) на время теста. Spring Boot 3.1+ имеет встроенную поддержку через `@ServiceConnection`. Используется, когда нужно проверить совместимость именно с конкретной СУБД.

4. **WireMock** — поднимает HTTP-сервер-заглушку, имитирующий поведение внешнего REST-сервиса. Подключается через `@AutoConfigureWireMock` (spring-cloud-contract-stub-runner) или прямым созданием `WireMockServer`. Позволяет описывать сценарии ответов через `stubFor(...)`.

5. **`@TestConfiguration` + кастомный `@Bean`** — объявляет тестовую конфигурацию, которая подменяет продакшн-бин более предсказуемой реализацией (например, stub-классом), не загрязняя основной код.

6. **`@ActiveProfiles("test")`** — активирует профиль, в котором перегрязаются свойства подключений (URL, порты) на тестовые значения.

**Типичная схема изоляции:**
```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
@ActiveProfiles("test")
class OrderServiceIntegrationTest {

    @MockBean
    private PaymentGatewayClient paymentClient; // внешний HTTP-сервис

    @Autowired
    private OrderService orderService;

    @Test
    void shouldCreateOrder() {
        given(paymentClient.charge(any())).willReturn(PaymentResult.SUCCESS);
        // ...
    }
}
```

### Вопрос 20. Что такое test slice в Spring Boot? Назовите несколько slice-аннотаций помимо @WebMvcTest и @DataJpaTest и объясните, что каждая из них загружает.

**Категория:** Тестирование · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Test Slices :: Spring Boot Reference (Appendix)](https://docs.spring.io/spring-boot/appendix/test-auto-configuration/slices.html)

Официальная документация Spring Boot перечисляет все доступные test-slice аннотации в виде таблицы. Каждый slice ограничивает сканирование компонентов и загружает только auto-configuration, относящуюся к конкретной части приложения.

- **`@JsonTest`** — загружает Jackson `ObjectMapper`, бины `@JsonComponent`, модули `JacksonModule`, а также поддержку Gson и Jsonb. Не загружает обычные `@Component`-бины. Используется для проверки сериализации/десериализации JSON.
- **`@WebFluxTest`** — загружает WebFlux-инфраструктуру, регистрирует бины `@Controller`, `@ControllerAdvice`, `@JsonComponent`, `Converter`, `WebFluxConfigurer`. Автоматически конфигурирует `WebTestClient`. Обычные `@Component` не сканируются.
- **`@DataMongoTest`** — загружает встроенный MongoDB (если доступен), `MongoTemplate`, сканирует `@Document`-классы и Spring Data MongoDB репозитории. Обычные `@Component` исключены.
- **`@RestClientTest`** — загружает поддержку Jackson/GSON/Jsonb, `RestTemplateBuilder`, `RestClient.Builder` и `MockRestServiceServer`. Позволяет тестировать HTTP-клиенты без реального сервера.

#### Ответ от Claude

**Что такое test slice**

Test slice (тестовый срез) — это механизм Spring Boot, позволяющий загружать только ту часть `ApplicationContext`, которая нужна для тестирования конкретного «слоя» приложения (web-слой, слой данных, JSON-маппинг и т.д.). Достигается за счёт специальных аннотаций `@…Test`, каждая из которых:
- ограничивает сканирование компонентов подходящими для данного слоя типами (`@Controller`, `@Repository` и т.д.);
- подключает только нужные auto-configuration классы;
- не загружает ничего лишнего — сервисы, полные конфигурации и т.д.

Главная цель: быстрые, сфокусированные тесты без поднятия полного контекста.

**Slice-аннотации помимо `@WebMvcTest` и `@DataJpaTest`**

| Аннотация | Что загружает | Для чего |
|---|---|---|
| **`@JsonTest`** | `ObjectMapper`, `@JsonComponent`-бины, Jackson/Gson/Jsonb модули | Тестирование JSON-сериализации и десериализации |
| **`@WebFluxTest`** | WebFlux-инфраструктуру, `@Controller`, `@ControllerAdvice`, `@JsonComponent`, `Converter`, `WebFluxConfigurer`; автоконфигурирует `WebTestClient` | Тестирование реактивных контроллеров (Project Reactor / WebFlux) |
| **`@RestClientTest`** | Jackson/GSON/Jsonb, `RestTemplateBuilder`, `RestClient.Builder`, `MockRestServiceServer` | Тестирование HTTP-клиентов (`RestTemplate`, `RestClient`) |
| **`@DataMongoTest`** | Встроенный MongoDB (если в classpath), `MongoTemplate`, `@Document`-классы, Spring Data MongoDB репозитории | Тестирование MongoDB-репозиториев |
| **`@DataJdbcTest`** | DataSource, `JdbcTemplate`, Spring Data JDBC репозитории, транзакции (с rollback после каждого теста) | Тестирование JDBC-репозиториев без JPA |
| **`@DataRedisTest`** | Redis-инфраструктуру, Spring Data Redis репозитории | Тестирование Redis-репозиториев |
| **`@JooqTest`** | DataSource, `DSLContext` (jOOQ), JDBC | Тестирование jOOQ-запросов |
| **`@GraphQlTest`** | GraphQL auto-configuration, GraphQL Tester | Тестирование GraphQL-обработчиков |

**Важные особенности:**
- Обычные `@Component`-бины (сервисы) не сканируются ни одним slice — нужные зависимости добавляются через `@MockBean`.
- Нельзя комбинировать несколько slice-аннотаций в одном тесте: если нужна функциональность двух срезов — берётся одна аннотация и вручную добавляются `@AutoConfigure…` от второй.
- Test slices существенно ускоряют тестовый прогон по сравнению с `@SpringBootTest`, поскольку контекст меньше и поднимается быстрее.

**Пример `@JsonTest`:**
```java
@JsonTest
class VehicleDetailsJsonTest {

    @Autowired
    private JacksonTester<VehicleDetails> json;

    @Test
    void serialize() throws Exception {
        VehicleDetails details = new VehicleDetails("Honda", "Civic");
        assertThat(this.json.write(details)).isEqualToJson("expected.json");
    }
}
```

**Пример `@RestClientTest`:**
```java
@RestClientTest(RemoteVehicleDetailsService.class)
class RemoteVehicleDetailsServiceTest {

    @Autowired
    private RemoteVehicleDetailsService service;

    @Autowired
    private MockRestServiceServer server;

    @Test
    void getVehicleDetails() {
        this.server.expect(requestTo("/vehicles/details"))
                   .andRespond(withSuccess("{...}", MediaType.APPLICATION_JSON));
        VehicleDetails result = service.getDetails("sboot");
        assertThat(result.getMake()).isEqualTo("Honda");
    }
}
```
