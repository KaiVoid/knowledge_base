### Вопрос 21. Чем отличается @Mock от @MockBean в Spring Boot тестах? В каком случае использование @MockBean может замедлить тестовый прогон?

**Категория:** Тестирование · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Mockito.mock() vs @Mock vs @MockBean | Baeldung](https://www.baeldung.com/java-spring-mockito-mock-mockbean)

По материалам Baeldung: `@Mock` — это аннотация Mockito, создающая mock-объект без какого-либо взаимодействия со Spring-контекстом. `@MockBean` — аннотация Spring Boot Test, которая добавляет mock-объект в `ApplicationContext`: если бин с таким типом уже существует — он заменяется mock-версией; если нет — новый mock-бин добавляется в контекст.

Ключевой момент по производительности: «Каждый раз, когда в тест-классе встречается `@MockBean`, кэш `ApplicationContext` помечается как "грязный" (dirty), поэтому после завершения тест-класса runner очищает кэш — это добавляет несколько лишних секунд к сборке».

#### Ответ от Claude

**Ключевые отличия `@Mock` и `@MockBean`**

| Критерий | `@Mock` (Mockito) | `@MockBean` (Spring Boot Test) |
|---|---|---|
| Пакет | `org.mockito.Mock` | `org.springframework.boot.test.mock.mockito.MockBean` |
| Spring-контекст | Не задействует | Регистрирует mock в `ApplicationContext` |
| Внедрение зависимостей | Вручную через `@InjectMocks` или конструктор | Автоматически через Spring DI |
| Применимость | Unit-тесты, без Spring | Интеграционные тесты, slice-тесты со Spring |
| Очистка mock | После каждого метода (при `MockitoExtension`) | Сбрасывается после каждого теста Spring |
| Влияние на кэш контекста | Никакого | Делает конфигурацию контекста уникальной |

**Когда использовать:**
- `@Mock` — в unit-тестах, где Spring не нужен: тестируется отдельный класс с зависимостями, заменёнными на mock. Быстро, без накладных расходов на поднятие контекста.
- `@MockBean` — когда тест требует Spring-контекста (`@SpringBootTest`, `@WebMvcTest` и т.д.) и нужно подменить один из бинов mock-реализацией.

```java
// Unit-тест — @Mock достаточно
@ExtendWith(MockitoExtension.class)
class OrderServiceTest {

    @Mock
    private PaymentRepository paymentRepository;

    @InjectMocks
    private OrderService orderService;

    @Test
    void shouldProcessOrder() {
        when(paymentRepository.save(any())).thenReturn(new Payment());
        // ...
    }
}

// Интеграционный тест — нужен @MockBean
@WebMvcTest(OrderController.class)
class OrderControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @MockBean
    private OrderService orderService; // реальный бин заменяется mock-ом

    @Test
    void shouldReturn200() throws Exception {
        when(orderService.getOrder(1L)).thenReturn(new Order());
        mockMvc.perform(get("/orders/1")).andExpect(status().isOk());
    }
}
```

**Почему `@MockBean` может замедлить тестовый прогон**

Spring Test кэширует `ApplicationContext` между тестами: если два тестовых класса используют одну конфигурацию — контекст создаётся один раз и повторно используется (это одна из важнейших оптимизаций).

Проблема: `@MockBean` является частью ключа кэша. Каждый раз, когда набор `@MockBean` в тесте уникален (другой тип, другое имя поля или другой набор подменяемых бинов), Spring вынужден поднять новый `ApplicationContext`. Старый контекст помечается как «грязный» (dirty) и уничтожается.

**Сценарии замедления:**

1. **Разные наборы `@MockBean` в разных тест-классах** — каждый класс создаёт собственный контекст.
2. **Разные имена полей с `@MockBean` одного типа** — `CustomService service` в одном классе и `CustomService customService` в другом — это два разных ключа кэша, два контекста.
3. **`@MockBean` в `@SpringBootTest`** — поднимает полный контекст, и при его «загрязнении» стоимость пересоздания максимальна.

**Рекомендации по оптимизации:**
- Использовать одинаковые имена полей `@MockBean` во всех тест-классах, мокающих один и тот же тип.
- Выносить общий набор `@MockBean` в базовый абстрактный тест-класс — все наследники будут иметь одинаковый ключ кэша и разделят один контекст.
- Предпочитать `@Mock` + unit-тесты там, где Spring не требуется.
- Применять test slices вместо полного `@SpringBootTest`, чтобы контекст был легче и пересоздавался быстрее.
- В Spring Framework 6.2+ вместо `@MockBean` используется `@MockitoBean` (нативная аннотация Spring, без Boot), которая работает по той же логике кэширования.

```java
// Хорошая практика: базовый класс с единым набором @MockBean
@SpringBootTest
abstract class BaseIntegrationTest {

    @MockBean
    protected ExternalApiClient externalApiClient;

    @MockBean
    protected EmailService emailService;
}

class OrderServiceIT extends BaseIntegrationTest {
    // использует тот же контекст, что и другие наследники BaseIntegrationTest
}

class UserServiceIT extends BaseIntegrationTest {
    // переиспользует уже поднятый контекст
}
```

### Вопрос 22. Какой порядок выполнения методов жизненного цикла JUnit 5: @BeforeAll, @BeforeEach, @AfterEach, @AfterAll — и как он меняется при использовании @TestInstance(Lifecycle.PER_CLASS)?

**Категория:** Тестирование · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [JUnit 5 Test Lifecycle: Before and After Annotations](https://www.arhohuttunen.com/junit-5-test-lifecycle/)

В стандартной конфигурации JUnit 5 создаёт новый экземпляр тестового класса перед каждым тестовым методом. Порядок выполнения следующий:

1. `@BeforeAll` — выполняется один раз перед всеми тестами (метод должен быть `static`)
2. Конструктор — вызывается перед каждым тестом
3. `@BeforeEach` — выполняется перед каждым отдельным тестом
4. Тестовый метод
5. `@AfterEach` — выполняется после каждого отдельного теста
6. `@AfterAll` — выполняется один раз после всех тестов (метод должен быть `static`)

Если в классе десять тестовых методов, то `@BeforeEach` и `@AfterEach` выполнятся десять раз, а `@BeforeAll` и `@AfterAll` — только по одному разу.

`@BeforeAll` и `@AfterAll` должны быть `static`, потому что при создании нового экземпляра перед каждым тестом иначе нет общего состояния.

При использовании `@TestInstance(Lifecycle.PER_CLASS)` создаётся единственный экземпляр тестового класса на весь класс. В этом режиме:
- Методы `@BeforeAll` и `@AfterAll` **больше не обязаны быть `static`**
- Конструктор вызывается лишь один раз
- Переменные экземпляра сохраняются между тестами
- Это удобно для дорогостоящих операций: открытие соединения с БД, запуск сервера

#### Ответ от Claude

**Порядок выполнения (по умолчанию, `PER_METHOD`)**

JUnit 5 по умолчанию использует режим `Lifecycle.PER_METHOD`: перед каждым тестовым методом создаётся **новый экземпляр** тестового класса. Полный порядок для класса с N тестами:

```
@BeforeAll          ← 1 раз (static)
  new TestClass()   ← N раз (конструктор)
  @BeforeEach       ← N раз
  @Test             ← N раз
  @AfterEach        ← N раз
@AfterAll           ← 1 раз (static)
```

Аннотации `@BeforeAll` и `@AfterAll` обязаны быть на `static`-методах (или на методах `default`/`static` в интерфейсах), потому что они вызываются до/после создания экземпляров и, значит, не могут полагаться на состояние объекта.

**Режим `@TestInstance(Lifecycle.PER_CLASS)`**

Аннотация на уровне класса меняет стратегию: JUnit создаёт ровно **один экземпляр** тестового класса и переиспользует его для всех тестов. Следствия:

| Аспект | PER_METHOD (умолч.) | PER_CLASS |
|---|---|---|
| Экземпляр класса | новый перед каждым тестом | один на весь класс |
| `@BeforeAll` / `@AfterAll` | обязательно `static` | могут быть нестатическими |
| Состояние полей между тестами | изолировано | **разделяется** (нужна осторожность) |
| `@BeforeAll` в `@Nested`-классах | не работает (нет static) | **работает** |

```java
@TestInstance(TestInstance.Lifecycle.PER_CLASS)
class DatabaseTest {

    Connection connection;

    @BeforeAll          // нестатический — разрешено в PER_CLASS
    void openConnection() {
        connection = DriverManager.getConnection(URL);
    }

    @BeforeEach
    void beginTransaction() { connection.setAutoCommit(false); }

    @AfterEach
    void rollback() { connection.rollback(); }

    @AfterAll           // нестатический — разрешено в PER_CLASS
    void closeConnection() { connection.close(); }
}
```

**Важные нюансы**

- В режиме `PER_CLASS` мутабельное состояние поля сохраняется между тестами, что может сделать тесты зависимыми друг от друга — нужно явно сбрасывать состояние в `@BeforeEach`.
- Порядок следования `@BeforeAll → @BeforeEach → @Test → @AfterEach → @AfterAll` **не меняется** — меняется только количество экземпляров и требование `static`.
- `@TestInstance` влияет и на вложенные классы (`@Nested`): с `PER_CLASS` в них становятся доступны `@BeforeAll`/`@AfterAll` без `static`.

### Вопрос 23. В чём разница между thenReturn() и doReturn() в Mockito? Почему thenReturn() не работает корректно со шпионами (spy)?

**Категория:** Тестирование · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Mockito: doReturn vs thenReturn](http://sangsoonam.github.io/2019/02/04/mockito-doreturn-vs-thenreturn.html)

Принципиальное различие проявляется при использовании `Mockito.spy()`.

- **`when(spy.method()).thenReturn(value)`** — при вычислении аргумента `when(...)` происходит **реальный вызов** метода шпиона: `spy.method()` выполняется фактически, и лишь после этого Mockito подменяет результат. Если метод бросает исключение или имеет побочный эффект, это случится до того, как стаб будет применён.
- **`doReturn(value).when(spy).method()`** — аргументом `doReturn(...)` является только само возвращаемое значение, а не результат вызова метода. Реальный метод **не вызывается**.

Пример: для пустого `List`-шпиона вызов `when(spy.get(0)).thenReturn("foo")` упадёт с `IndexOutOfBoundsException`, потому что `spy.get(0)` выполнится реально. `doReturn("foo").when(spy).get(0)` — работает корректно.

Дополнительное отличие: `doReturn()` принимает параметр типа `Object`, тогда как `thenReturn()` типизирован — компилятор проверяет тип возвращаемого значения.

#### Ответ от Claude

**Механика двух подходов**

В Mockito существуют два DSL-стиля для стаббинга:

```java
// Стиль 1: when-thenReturn
when(mock.method()).thenReturn(value);

// Стиль 2: doReturn-when
doReturn(value).when(mock).method();
```

Для **обычного мока** оба стиля функционально эквивалентны: `mock.method()` внутри `when(...)` не выполняется реально — Mockito перехватывает вызов на уровне прокси.

**Почему `thenReturn` ломается со шпионами**

`spy` — это обёртка над **реальным объектом**. Когда Mockito видит `when(spy.method())`, он **сначала** выполняет `spy.method()` (реальный код), и лишь потом регистрирует стаб. Это приводит к проблемам:

```java
List<String> list = new ArrayList<>();
List<String> spy = Mockito.spy(list);

// ОШИБКА: вызывает реальный get(0) → IndexOutOfBoundsException
when(spy.get(0)).thenReturn("foo");

// ПРАВИЛЬНО: реальный get(0) НЕ вызывается
doReturn("foo").when(spy).get(0);
```

Последовательность выполнения при `when(spy.get(0)).thenReturn(...)`:
1. JVM вычисляет аргумент `when(...)` → вызывает `spy.get(0)` → **реальный метод**
2. Если реальный метод бросает исключение — до `thenReturn` дело не доходит

При `doReturn(value).when(spy).method()`:
1. `doReturn(value)` — Mockito запоминает возвращаемое значение
2. `.when(spy)` — возвращает прокси для перехвата следующего вызова
3. `.method()` — **перехватывается без реального выполнения**

**Сводная таблица различий**

| Критерий | `when().thenReturn()` | `doReturn().when()` |
|---|---|---|
| Реальный вызов на моке | нет | нет |
| Реальный вызов на шпионе | **да** (до стаббинга) | нет |
| Типобезопасность | да (generics) | нет (`Object`) |
| Работа с `void`-методами | нет | да |
| Рекомендация для spy | избегать | **использовать** |

**Практические рекомендации**

- Для **mock**: предпочитать `when().thenReturn()` — читаемее и типобезопасно.
- Для **spy**: всегда использовать `doReturn().when()` (а также `doThrow()`, `doAnswer()`, `doCallRealMethod()`).
- Официальная документация Mockito явно указывает: для шпионов семейство `do*`-методов является **обязательным**, а не опциональным.

### Вопрос 24. Что такое пирамида тестирования? Какое соотношение unit/integration/e2e тестов считается оптимальным и почему?

**Категория:** Тестирование · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [The testing pyramid: Strategic software testing for Agile teams](https://circleci.com/blog/testing-pyramid/)

Пирамида тестирования — иерархический фреймворк, организующий тестирование программного обеспечения на три уровня: юнит-тесты в основании, интеграционные тесты посередине и E2E-тесты на вершине. Согласно статье, эта модель «направляет основные усилия по тестированию на тесты, которые быстры, надёжны и просты в сопровождении».

Рекомендуемое соотношение — приблизительно **70% юнит-тестов, 20% интеграционных тестов и 10% E2E-тестов**, хотя конкретные проценты зависят от требований проекта.

Обоснование:
- **Стоимость и скорость**: юнит-тесты дёшевы и быстры, интеграционные — ресурсоёмки, E2E — требуют наибольшего времени и инфраструктуры.
- **Обнаружение дефектов**: юнит-тесты выявляют проблемы в изолированных компонентах быстро, снижая стоимость исправления.
- **Баланс**: структура «предотвращает чрезмерную зависимость от одного типа тестов», избегая хрупких, трудных в сопровождении наборов E2E-тестов — «антипаттерна мороженого рожка».

#### Ответ от Claude

**Что такое пирамида тестирования**

Концепцию предложил Майк Кон в книге «Succeeding with Agile» (2009), а Мартин Фаулер популяризировал в статье [TestPyramid](https://martinfowler.com/bliki/TestPyramid.html). Пирамида — это метафора оптимального распределения автоматизированных тестов по уровням: чем быстрее и дешевле тест, тем больше таких тестов должно быть.

```
        /\
       /E2E\          ← мало, дорого, медленно
      /------\
     /Integr. \       ← умеренно
    /----------\
   / Unit Tests \     ← много, дёшево, быстро
  /______________\
```

**Три уровня**

| Уровень | Что проверяет | Скорость | Стоимость | Изоляция |
|---|---|---|---|---|
| Unit | отдельный класс/функцию | миллисекунды | минимальная | полная |
| Integration | взаимодействие компонентов, слоёв, БД, внешних сервисов | секунды | средняя | частичная |
| E2E / UI | сценарий с точки зрения пользователя | десятки секунд | высокая | нет |

**Классическое соотношение: 70 / 20 / 10**

Широко принятое практикой соотношение:
- **~70% — юнит-тесты**: быстрые, детерминированные, покрывают бизнес-логику
- **~20% — интеграционные тесты**: проверяют взаимодействие слоёв (сервис → репозиторий → БД), REST API
- **~10% — E2E тесты**: проверяют только критически важные пользовательские сценарии

Это соотношение **не догма**, а ориентир. Фаулер специально не приводит точных процентов, подчёркивая принцип: «больше юнит-тестов, чем интеграционных; больше интеграционных, чем E2E».

**Почему именно такая форма**

1. **Обратная связь**: юнит-тесты дают результат за миллисекунды → разработчик узнаёт о баге немедленно.
2. **Надёжность**: E2E-тесты нестабильны (flaky) из-за сети, таймаутов, UI-изменений; юнит-тесты детерминированы.
3. **Сопровождаемость**: один E2E-тест может сломаться при любом изменении стека; сотня юнит-тестов — только при изменении своего компонента.
4. **Локализация ошибки**: упавший юнит-тест точно указывает на место бага; упавший E2E — нет.

**Антипаттерны**

- **«Мороженое рожок»** (Ice Cream Cone): много E2E, мало юнит — медленная и хрупкая суита.
- **«Соты»** (Honeycomb, предложен Spotify): для микросервисов смещение в сторону интеграционных тестов (сервис-контракты), меньше E2E.

**Адаптация под контекст**

- Микросервисная архитектура: 60% unit / 30% integration / 10% E2E — больше контрактных тестов между сервисами.
- Legacy-монолит без DI: 80% unit / 15% integration / 5% E2E — интеграционные тесты сложны.
- CRUD-приложение с REST API: часто сдвиг к интеграционным тестам, покрывающим слой контроллер → сервис → БД целиком (подход, популяризованный Spring Boot Test).
### Вопрос 25. Чем отличается @WebMvcTest от @SpringBootTest при использовании MockMvc? Что загружается в контекст в каждом случае?

**Категория:** Тестирование · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Testing Spring Boot Applications :: Spring Boot](https://docs.spring.io/spring-boot/reference/testing/spring-boot-applications.html)

`@SpringBootTest` в сочетании с `@AutoConfigureMockMvc` позволяет тестировать контроллеры Spring MVC в mock-окружении сервлетов **без запуска реального HTTP-сервера**. При этом загружается **полный** `ApplicationContext`: все бины `@Component`, `@Service`, `@Repository`, `@Controller`, все `@ConfigurationProperties` и все Spring Boot auto-configurations. MockMvc при этом настраивается дополнительной аннотацией `@AutoConfigureMockMvc`.

Для изолированного тестирования только слоя MVC используется `@WebMvcTest`: он загружает **только** web-релевантные компоненты — `@Controller`, `@ControllerAdvice`, `@JsonComponent`, `Filter`, `WebMvcConfigurer` и т.д. Сервисные и репозиторные бины в контекст не попадают; MockMvc настраивается автоматически без дополнительных аннотаций.

#### Ответ от Claude

**`@WebMvcTest`** — аннотация для узкого (unit) тестирования слоя MVC. В Spring-контекст загружаются только:

- Бины, аннотированные `@Controller`, `@ControllerAdvice`, `@JsonComponent`
- Инфраструктурные компоненты: `Filter`, `FilterRegistrationBean`, `HandlerInterceptor`, `WebMvcConfigurer`, `HttpMessageConverter`, `Converter`, `GenericConverter`
- Бины безопасности: `SecurityFilterChain`, `WebSecurityConfigurer` (если Spring Security в classpath)
- Авто-конфигурации: Spring MVC, кэш, MockMvc (с поддержкой HtmlUnit и Selenium)

**Что НЕ загружается:** `@Service`, `@Repository`, `@Component` общего назначения, JPA/JDBC auto-configuration, большинство остальных Spring Boot auto-configurations. Зависимости контроллеров необходимо мокировать через `@MockitoBean`.

MockMvc **настраивается автоматически** — никаких дополнительных аннотаций не нужно.

---

**`@SpringBootTest` + `@AutoConfigureMockMvc`** — аннотация для интеграционного тестирования. В Spring-контекст загружается:

- **Весь ApplicationContext** приложения: все `@Component`, `@Service`, `@Repository`, `@Controller`
- Все `@ConfigurationProperties`
- Все Spring Boot auto-configurations (JPA, DataSource, Security и т.д.)

MockMvc **НЕ настраивается автоматически** — требуется явная аннотация `@AutoConfigureMockMvc`. По умолчанию `webEnvironment = MOCK`, то есть реальный HTTP-сервер не запускается, но поднимается mock-окружение сервлетов.

**Сравнительная таблица:**

| Характеристика | `@WebMvcTest` | `@SpringBootTest` + `@AutoConfigureMockMvc` |
|---|---|---|
| Загружаемый контекст | Только web-слой (MVC) | Полный ApplicationContext |
| Сервисные/репозиторные бины | Не загружаются | Загружаются |
| Авто-конфигурация MockMvc | Автоматически | Требует `@AutoConfigureMockMvc` |
| Скорость запуска | Быстро | Медленнее |
| Назначение | Unit-тесты контроллеров | Интеграционные тесты |
| Нужен `@MockitoBean` для зависимостей | Да | Нет (реальные бины) |

**Практическое правило:** если нужно проверить только логику маппинга URL, валидацию запросов/ответов и сериализацию — используйте `@WebMvcTest`. Если важно, что все слои приложения работают вместе — используйте `@SpringBootTest`.

### Вопрос 26. Как работает @DataJpaTest? Какая база данных используется по умолчанию, оборачиваются ли тесты в транзакцию и откатываются ли изменения после каждого теста?

**Категория:** Тестирование · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [DataJpaTest (Spring Boot 3.3 API) — docs.spring.io](https://docs.spring.io/spring-boot/3.3/api/java/org/springframework/boot/test/autoconfigure/orm/jpa/DataJpaTest.html)

Аннотация для JPA-тестов, которая фокусируется **только** на JPA-компонентах. Включает только те авто-конфигурации, которые релевантны для Data JPA тестов; сканирование компонентов ограничено JPA-репозиториями и сущностями (`@Entity`).

По умолчанию тесты, аннотированные `@DataJpaTest`, являются **транзакционными** и **откатываются** в конце каждого теста. Они также используют **встроенную базу данных в памяти** (заменяя любой явно настроенный или авто-сконфигурированный `DataSource`). SQL-запросы логируются по умолчанию (`spring.jpa.show-sql=true`).

#### Ответ от Claude

`@DataJpaTest` — специализированная аннотация Spring Boot для изолированного тестирования JPA-слоя приложения.

**Что загружается в контекст:**
- JPA-репозитории (`@Repository`, расширяющие Spring Data интерфейсы)
- `@Entity`-классы
- JPA-релевантные авто-конфигурации: Hibernate, `EntityManagerFactory`, `TransactionManager`, Spring Data JPA

**Что НЕ загружается:** `@Controller`, `@Service`, `@Component` общего назначения, web-слой, большинство остальных auto-configurations.

**База данных по умолчанию:**
Используется **встроенная in-memory база данных** — как правило H2 (если H2 присутствует в classpath). Она заменяет любой явно настроенный или авто-сконфигурированный `DataSource`. Чтобы использовать реальную базу данных, нужно применить `@AutoConfigureTestDatabase(replace = Replace.NONE)`.

**Транзакционность:**
Да, каждый тест **оборачивается в транзакцию** — аннотация `@DataJpaTest` мета-аннотирована `@Transactional`.

**Откат изменений:**
Да, **изменения автоматически откатываются** после каждого теста. Это обеспечивает изоляцию тестов: данные, созданные в одном тесте, не влияют на другие.

**Как изменить поведение:**
- Отключить откат для конкретного теста: `@Rollback(false)` или `@Commit`
- Отключить транзакционность: `@Transactional(propagation = Propagation.NOT_SUPPORTED)` на классе или методе
- Использовать реальную БД: `@AutoConfigureTestDatabase(replace = Replace.NONE)`
- Отключить логирование SQL: `@DataJpaTest(showSql = false)`

**Пример использования:**
```java
@DataJpaTest
class UserRepositoryTest {
    @Autowired
    private UserRepository userRepository;

    @Test
    void shouldFindByEmail() {
        userRepository.save(new User("test@example.com"));
        Optional<User> found = userRepository.findByEmail("test@example.com");
        assertThat(found).isPresent();
        // После теста транзакция будет откатана автоматически
    }
}
```

**Важный нюанс:** откат транзакции происходит на уровне теста, но если сам репозиторный метод вызывает `flush()` или помечен `@Transactional(propagation = REQUIRES_NEW)`, изменения могут быть зафиксированы раньше отката тестовой транзакции.

### Вопрос 27. Что такое ArgumentCaptor в Mockito и в каких ситуациях его использование предпочтительнее, чем передача матчера напрямую в verify()?

**Категория:** Тестирование · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [ArgumentCaptor (Mockito API) — site.mockito.org](https://site.mockito.org/javadoc/current/org/mockito/ArgumentCaptor.html)

`ArgumentCaptor` используется для **захвата значений аргументов** с целью последующих проверок (assertions). Mockito проверяет значения аргументов в естественном Java-стиле через метод `equals()`, что является рекомендованным способом. Однако иногда полезно проинспектировать аргументы уже после верификации.

Пример использования:
```java
ArgumentCaptor<Person> argument = ArgumentCaptor.forClass(Person.class);
verify(mock).doSomething(argument.capture());
assertEquals("John", argument.getValue().getName());
```

Когда `ArgumentCaptor` предпочтительнее: если кастомный матчер вряд ли будет переиспользоваться, или когда нужно сделать несколько проверок значения аргумента после верификации. Кастомные `ArgumentMatcher`-ы обычно лучше подходят для **стаббинга**.

Важное предупреждение: использование `ArgumentCaptor` вместе со стаббингом **не рекомендуется** — это снижает читаемость теста (captor создаётся вне блока assert/verify) и затрудняет локализацию дефектов (если застаббированный метод не был вызван, аргумент не захватывается).

#### Ответ от Claude

**ArgumentCaptor** — утилитный класс Mockito, который позволяет **захватить (capture) аргумент**, переданный в метод мока, и затем провести над ним детальные проверки в блоке assertion.

**Как работает:**
1. Создаётся экземпляр: `ArgumentCaptor<T> captor = ArgumentCaptor.forClass(T.class)` или через аннотацию `@Captor`
2. В `verify()` вместо реального значения или матчера передаётся `captor.capture()`
3. После верификации значение доступно через `captor.getValue()` (последний вызов) или `captor.getAllValues()` (все вызовы)

```java
@Captor
ArgumentCaptor<EmailMessage> emailCaptor;

// Тест:
emailService.sendWelcome(user);

verify(mailSender).send(emailCaptor.capture());
EmailMessage sent = emailCaptor.getValue();
assertThat(sent.getTo()).isEqualTo("user@example.com");
assertThat(sent.getSubject()).contains("Добро пожаловать");
assertThat(sent.getBody()).contains(user.getName());
```

**Когда ArgumentCaptor предпочтительнее матчера в verify():**

1. **Нужно проверить несколько свойств аргумента** — матчер в `verify()` может проверить только одно условие или потребует написания сложного кастомного `ArgumentMatcher`. Captor захватывает весь объект, и можно проверить любые его поля отдельными assertions с понятными сообщениями об ошибке.

2. **Аргумент недоступен снаружи тестируемого метода** — например, тестируемый метод создаёт объект внутри себя и передаёт его в зависимость. Получить этот объект иным способом невозможно.

3. **Кастомный матчер не нужен повторно** — если логика сравнения используется только в одном месте, создавать отдельный класс `ArgumentMatcher` избыточно.

4. **Нужна понятная диагностика при падении** — при использовании captor каждый `assertThat` падает с конкретным сообщением; комплексный матчер сообщит только "аргумент не совпал".

**Когда предпочесть матчер напрямую в verify():**

- Проверка простая и умещается в одну строку: `verify(repo).save(argThat(u -> u.isActive()))`
- Матчер переиспользуется в нескольких тестах — выгоднее вынести в `ArgumentMatcher`
- Нужна проверка при стаббинге (`when(...).thenReturn(...)`) — здесь **ArgumentCaptor не подходит** принципиально: captor захватывает значения только при верификации, а не при stubbing; если метод не будет вызван, захвата не произойдёт и тест не упадёт с понятной ошибкой

**Ключевое правило из официальной документации Mockito:** ArgumentCaptor рекомендуется использовать **только с verify()**, но не с `when()` (stubbing). Для stubbing используйте `ArgumentMatcher`.
### Вопрос 28. Как организовать параметризованный тест в JUnit 5? Перечислите как минимум три источника данных (@ValueSource, @CsvSource, @MethodSource) и укажите ограничения каждого.

**Категория:** Тестирование · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [A More Practical Guide to JUnit 5 Parameterized Tests – arhohuttunen.com](https://www.arhohuttunen.com/junit-5-parameterized-tests/)

Параметризованные тесты в JUnit 5 подключаются аннотацией `@ParameterizedTest` (из модуля `junit-jupiter-params`). Источник данных задаётся отдельной аннотацией.

**@ValueSource** — самый простой источник: задаёт массив литеральных значений одного примитивного или строкового типа. Поддерживаемые типы: `short`, `byte`, `int`, `long`, `float`, `double`, `char`, `boolean`, `String`, `Class`. Ограничения: можно указать только один тип за раз; тест-метод может принимать только один аргумент; передача сложных объектов или нескольких параметров одновременно невозможна.

**@CsvSource** — задаёт строки в CSV-формате, каждая строка соответствует одному вызову теста, столбцы — отдельным параметрам. JUnit автоматически конвертирует строки в объявленные типы параметров. Ограничения: все значения исходно являются строками и требуют неявной конверсии; разделитель — только одиночный символ или строка; сложные объекты передать нельзя; при большом количестве параметров строки становятся трудно читаемыми; нельзя использовать динамические данные.

**@MethodSource** — указывает фабричный метод, возвращающий `Stream<Arguments>`, `Stream<T>`, `Iterable`, `Iterator` или массив. Позволяет передавать произвольно сложные объекты. Ограничения: фабричный метод должен быть `static` (если только класс не использует `@TestInstance(Lifecycle.PER_CLASS)`); отсутствует компиляторная проверка соответствия типов и количества аргументов сигнатуре теста — ошибка обнаруживается только во время выполнения; фабричный метод не должен принимать параметров (до JUnit 5.11).

#### Ответ от Claude

**Подготовка.** Для использования параметризованных тестов нужен артефакт `junit-jupiter-params`. В Maven/Gradle он обычно транзитивно тянется вместе с `junit-jupiter`, либо подключается явно.

```java
@ParameterizedTest
@ValueSource(ints = {1, 2, 3})
void testWithInts(int value) {
    assertTrue(value > 0);
}
```

---

**@ValueSource**

Назначение: передать набор одиночных значений примитивного или строкового типа.

Поддерживаемые типы: `short[]`, `byte[]`, `int[]`, `long[]`, `float[]`, `double[]`, `char[]`, `boolean[]`, `String[]`, `Class[]`.

```java
@ParameterizedTest
@ValueSource(strings = {"racecar", "level", "madam"})
void testPalindromes(String candidate) {
    assertTrue(isPalindrome(candidate));
}
```

Ограничения:
- Одновременно можно объявить массив только одного типа — нельзя смешать `ints` и `strings` в одной аннотации.
- Тест-метод принимает ровно один параметр (исключение: неявная конверсия типа через `@ConvertWith`).
- Нельзя передать `null` через `@ValueSource` напрямую (для этого есть `@NullSource` / `@NullAndEmptySource`).
- Не подходит для сложных объектов или кортежей из нескольких значений.

---

**@CsvSource**

Назначение: задать табличные данные прямо в аннотации в виде CSV-строк — каждая строка соответствует одному вызову теста.

```java
@ParameterizedTest
@CsvSource({
    "apple,  1",
    "banana, 2",
    "'lemon, lime', 3"
})
void testFruits(String fruit, int rank) { ... }
```

Особенности:
- Разделитель по умолчанию — запятая; можно заменить через атрибут `delimiter` (одиночный символ) или `delimiterString` (строка) — взаимоисключающие.
- Пустое значение без кавычек → `null`; пустая строка в одиночных кавычках (`''`) → `""`.
- Нельзя передать `null` в примитивный тип — выбросится `ArgumentConversionException`.
- Конверсия типов выполняется неявно (строки конвертируются во `int`, `boolean` и т.д.).
- Максимум 4096 символов на колонку по умолчанию.

Ограничения:
- Только статические (константные) данные — динамическую генерацию не поддерживает.
- Читаемость снижается при большом числе параметров или длинных строках.
- Нельзя передавать произвольные объекты — только типы, поддерживаемые встроенной конверсией.

---

**@MethodSource**

Назначение: сослаться на фабричный метод, возвращающий данные для теста.

```java
@ParameterizedTest
@MethodSource("provideStringsForIsBlank")
void isBlank_ShouldReturnTrueForNullOrBlankStrings(String input, boolean expected) {
    assertEquals(expected, Strings.isBlank(input));
}

static Stream<Arguments> provideStringsForIsBlank() {
    return Stream.of(
        Arguments.of(null, true),
        Arguments.of("",   true),
        Arguments.of("  ", true),
        Arguments.of("not blank", false)
    );
}
```

Ограничения:
- Фабричный метод **должен быть `static`**, если только класс не аннотирован `@TestInstance(Lifecycle.PER_CLASS)`.
- Метод не должен принимать параметров (до JUnit 5.11; в JUnit 6 это разрешено через `ParameterResolver`).
- Отсутствует компиляторная проверка: число и типы элементов в `Arguments` не сверяются с сигнатурой теста до запуска.
- Метод должен находиться в том же классе или быть указан по полному имени (`"com.example.Utils#provideData"`).

---

**Дополнительные источники для справки:**
- `@EnumSource` — перебирает значения enum.
- `@CsvFileSource` — CSV из файла на classpath.
- `@ArgumentsSource` — произвольный `ArgumentsProvider`.

### Вопрос 29. Что произойдёт, если в Mockito создать stub на метод, который в ходе теста так и не будет вызван? Как называется исключение и как его отключить?

**Категория:** Тестирование · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: ["Unnecessary Stubbings Detected!" — What Mockito Is Trying to Tell You — codefarm0.medium.com](https://codefarm0.medium.com/unnecessary-stubbings-detected-d21df974b675)

Если в Mockito создать stub (`when(...).thenReturn(...)`) на метод, который в ходе теста ни разу не вызывается, фреймворк при использовании строгого режима выбрасывает `UnnecessaryStubbingException`. Это сигнал о «мёртвом» тестовом коде — настройке, которая не участвует в проверке.

Основные причины появления:
1. Метод был сначала задуман как нужный, но тест изменился.
2. Stub описывает путь кода, который в данном тесте не выполняется.
3. Stub определён в `@BeforeEach`, но не все тесты его используют.
4. Ошибочно заглушён не тот объект.

Способы устранить:
- **Удалить неиспользуемый stub** — самый правильный подход.
- **`lenient().when(...).thenReturn(...)`** — пометить конкретный stub как разрешённый к неиспользованию.
- **`@MockitoSettings(strictness = Strictness.LENIENT)`** на уровне класса — отключить строгость для всех stubs теста.

#### Ответ от Claude

**Что происходит при неиспользованном stub**

Поведение зависит от режима строгости (`Strictness`), с которым создаются моки:

| Режим | Поведение при неиспользованном stub |
|---|---|
| `STRICT_STUBS` (по умолчанию в `MockitoExtension`) | Тест завершается с `UnnecessaryStubbingException` |
| `LENIENT` | Неиспользованный stub игнорируется молча |
| `WARN` | Предупреждение выводится в stderr, тест не падает |

`MockitoExtension` (JUnit 5) и `MockitoJUnitRunner.StrictStubs` (JUnit 4) по умолчанию используют `Strictness.STRICT_STUBS`.

**Имя исключения**

`org.mockito.exceptions.misusing.UnnecessaryStubbingException`

Это исключение выбрасывается по завершении теста (в `afterEach`/`afterAll` фазе расширения), а не в момент объявления stub. Сообщение перечисляет все неиспользованные stubs с именами методов и местами объявления.

**Пример:**
```java
@ExtendWith(MockitoExtension.class)
class MyTest {
    @Mock UserRepository repo;

    @Test
    void test() {
        // Stub объявлен, но repo.findById() в тесте не вызывается
        when(repo.findById(1L)).thenReturn(Optional.of(new User()));
        // ... тест делает что-то другое
        // По завершении теста: UnnecessaryStubbingException
    }
}
```

**Как отключить**

*Вариант 1 — удалить ненужный stub (рекомендуется):*
```java
// Просто убрать строку when(repo.findById(1L)).thenReturn(...)
```

*Вариант 2 — пометить конкретный stub как lenient:*
```java
lenient().when(repo.findById(1L)).thenReturn(Optional.of(new User()));
```
Используется, когда stub нужен в некоторых сценариях, но не во всех (например, вынесен в `@BeforeEach`).

*Вариант 3 — отключить строгость для всего тестового класса:*
```java
@ExtendWith(MockitoExtension.class)
@MockitoSettings(strictness = Strictness.LENIENT)
class MyTest { ... }
```

*Вариант 4 — для JUnit 4 через MockitoRule:*
```java
@Rule
public MockitoRule rule = MockitoJUnit.rule().strictness(Strictness.LENIENT);
```

*Вариант 5 — для JUnit 4 через Runner:*
```java
@RunWith(MockitoJUnitRunner.Silent.class) // не рекомендуется: полностью подавляет предупреждения
```

**Важный нюанс.** Если stub вынесен в `@BeforeEach` и хотя бы в одном тесте класса он реально используется, `STRICT_STUBS` не выбрасывает исключение — проверка выполняется на уровне класса, а не метода. Исключение возникает только если ни один тест не использует данный stub.

### Вопрос 30. Объясните разницу между mock, stub и spy в терминологии Mockito. Когда применение spy является признаком плохо спроектированного кода?

**Категория:** Тестирование · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Mockito Spies and When to Use Them – ericthecoder.com](https://www.ericthecoder.com/2021/03/22/mockito-spies-and-when-to-use-them/)

Статья объясняет концепции через практическое использование в Mockito:

**Stub** — объект, возвращающий заранее заданные ответы на вызовы методов. Не проверяет взаимодействия; его задача — предоставить детерминированные данные тестируемому коду.

**Mock** — полностью «пустой» объект, созданный фреймворком. По умолчанию все методы возвращают null/0/false. Позволяет как задавать возвращаемые значения (stub-поведение), так и проверять, что методы были вызваны с нужными аргументами (верификация).

**Spy** — обёртка над реальным объектом. Реальные методы выполняются в штатном режиме, но можно отдельные из них перехватить. Это «частичный мок».

Когда spy — признак плохого дизайна: автор цитирует официальную документацию Mockito: «OOP решает сложность, разделяя её на отдельные объекты с единственной ответственностью. Частичный мок не вписывается в эту парадигму». Если вам требуется spy в новом, хорошо написанном коде — это сигнал, что класс нарушает SRP и содержит слишком много ответственностей.

#### Ответ от Claude

**Терминологическое разграничение**

В теории тестирования (Meszaros, «xUnit Test Patterns») и в документации Mockito эти понятия трактуются так:

**Stub** — объект, предоставляющий заранее настроенные ответы на вызовы. Стаб не запоминает, как его вызывали, и не проверяет взаимодействия. Его роль — обеспечить изоляцию тестируемого кода от зависимостей.

**Mock** — объект, на котором можно осуществить верификацию: убедиться, что метод был вызван, с какими аргументами, сколько раз. В Mockito `mock()` создаёт объект, который одновременно может играть роль и стаба (через `when().thenReturn()`), и мока (через `verify()`). По умолчанию нестабованные методы возвращают «нулевые» значения: `null`, `0`, `false`, пустые коллекции.

**Spy** — обёртка над реально существующим объектом. Нестабованные методы делегируются реальной реализации; отдельные методы можно перехватить. Spy создаётся через `Mockito.spy(realObject)` или аннотацию `@Spy`.

```java
// Mock — полностью синтетический объект
List<String> mockList = mock(List.class);
when(mockList.size()).thenReturn(10);   // stubbing
verify(mockList).add("foo");            // verification

// Spy — обёртка над реальным ArrayList
List<String> spyList = spy(new ArrayList<>());
spyList.add("hello");                   // вызывается реальный add()
when(spyList.size()).thenReturn(100);   // только size() перехвачен
```

**Важное предостережение для spy:** при использовании `when(spy.method()).thenReturn(...)` реальный метод всё равно вызывается в момент задания stub, что может вызвать исключение. Для spy следует использовать `doReturn(...).when(spy).method()`.

---

**Ключевые различия**

| Характеристика | Mock | Spy |
|---|---|---|
| Основа | Синтетический прокси | Реальный объект |
| Нестабованные методы | Возвращают null/0/false | Вызывают реальную реализацию |
| Создание | `mock(Clazz.class)` | `spy(new Clazz())` |
| Типичное применение | Изоляция от внешних зависимостей | Частичное переопределение поведения |
| Верификация | Поддерживается | Поддерживается |

---

**Когда spy — признак плохого дизайна**

Официальная документация Mockito прямо предупреждает:

> «Real spies should be used carefully and occasionally, for example when dealing with legacy code. [...] Object oriented programming tackles complexity by dividing it into separate, specific, SRPy objects. How does partial mock fit into this paradigm? Well, it just doesn't...»

Применение spy сигнализирует о проблемах дизайна в следующих ситуациях:

1. **Нарушение принципа единственной ответственности (SRP).** Если класс делает слишком много и его нельзя протестировать целиком, приходится «отключать» часть поведения через spy. Правильное решение — разбить класс.

2. **Тестирование private/protected методов через публичный API.** Если хочется заглушить один публичный метод, чтобы протестировать другой в том же классе — это сигнал, что методы следует вынести в отдельную зависимость.

3. **Spy на классе, который является самим SUT (System Under Test).** Мокировать объект, который тестируется, — противоречие: тест перестаёт проверять реальное поведение.

4. **Новый, «чистый» код требует spy.** В well-designed, TDD-коде spy не нужен. Если он появляется — значит, декомпозиция недостаточна.

**Законные случаи использования spy:**
- Тестирование legacy-кода, который нельзя рефакторить.
- Работа со сторонними библиотеками (third-party API), которые нельзя заменить интерфейсом.
- Тестирование абстрактных классов, когда создание конкретной реализации нецелесообразно.
- Временный приём в ходе постепенного рефакторинга.

### Вопрос 31. Как Mockito внедряет зависимости через @InjectMocks? В каком порядке он пробует стратегии (constructor injection, setter injection, field injection)? Что произойдёт, если подходящего конструктора нет?

**Категория:** Тестирование · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Mockito @InjectMocks - Mocks Dependency Injection — DigitalOcean](https://www.digitalocean.com/community/tutorials/mockito-injectmocks-mocks-dependency-injection)

Mockito поддерживает три стратегии внедрения зависимостей через `@InjectMocks`, применяемые в строго заданном порядке приоритета.

1. **Constructor-Based Injection** (инъекция через конструктор): Mockito пытается внедрить зависимости через наибольший доступный конструктор, если он определён.
2. **Setter Methods-Based Injection** (инъекция через сеттеры): если конструкторов нет, Mockito пробует внедрить зависимости через setter-методы.
3. **Field-Based Injection** (инъекция напрямую в поля): как последний вариант, Mockito напрямую внедряет объекты-моки в поля.

Механизм разрешения неоднозначности: если есть только один мок подходящего типа — он внедряется автоматически. Если моков одного типа несколько — Mockito использует имена полей для сопоставления. Необходимо явно вызвать `MockitoAnnotations.initMocks(this)` в `@Before` (или использовать `@ExtendWith(MockitoExtension.class)` для JUnit 5), иначе аннотации не обработаются и возникнет `NullPointerException`.

#### Ответ от Claude

**Порядок применения стратегий (официальная документация Mockito)**

Mockito пробует стратегии строго в следующем порядке и останавливается после первой успешно применённой:

**1. Constructor Injection (инъекция через конструктор)**

Mockito выбирает конструктор с наибольшим числом параметров («largest constructor»). Аргументы подбираются из доступных моков по типу. Если аргумент не найден — передаётся `null`. Если тип параметра не является mockable (например, примитив или final-класс, который нельзя замокать) — constructor injection не применяется вообще. Ключевое правило из документации: *«if the object is successfully created with the constructor, then Mockito won't try the other strategies»* — после успешного создания объекта через конструктор сеттеры и поля не трогаются.

Поле с `@InjectMocks` может иметь как параметрический конструктор, так и конструктор без аргументов (или оба). Видимость конструктора может быть `package-private`, `protected` или `private`.

**2. Setter Injection (инъекция через сеттеры)**

Применяется, когда constructor injection не сработал (не найден подходящий конструктор или объект уже инициализирован). Mockito ищет setter-методы и сопоставляет моки с параметрами: сначала по типу (если совпадение единственное — внедряется независимо от имени), затем — по имени свойства и имени мока при нескольких кандидатах одного типа. Если экземпляр `@InjectMocks` не был инициализирован и имеет no-arg конструктор — Mockito создаёт объект через него.

**3. Field Injection (инъекция напрямую в поля)**

Применяется как последний вариант. Логика сопоставления аналогична сеттерам: сначала по типу, затем по имени поля и имени мока. `static` и `final` поля игнорируются. Mockito использует рефлексию и может работать с `private`-полями.

**Что происходит при отсутствии подходящего конструктора**

Если Mockito не может применить constructor injection (нет параметрического конструктора, или типы параметров не соответствуют доступным мокам), он переходит к следующей стратегии — setter injection, а затем field injection. При наличии no-arg конструктора объект создаётся через него, после чего применяется setter или field injection.

**Важно:** если ни одна из стратегий не смогла внедрить зависимость, Mockito **не бросает исключение и не сигнализирует об ошибке**. Поле просто остаётся `null`. Это частая причина трудноуловимых `NullPointerException` в тестах — разработчик ожидает, что мок внедрён, но Mockito молча пропустил внедрение.

**Практические рекомендации**

- Использовать `@ExtendWith(MockitoExtension.class)` в JUnit 5 вместо ручного `MockitoAnnotations.openMocks(this)`.
- При нескольких моках одного типа обязательно именовать поля `@Mock` в соответствии с именами зависимостей в тестируемом классе.
- Предпочитать constructor injection (явный конструктор в тестируемом классе) — это делает поведение предсказуемым и избавляет от «тихих» сбоев field injection.
- Если тестируемый класс использует `final`-поля или конструкторную инъекцию Spring — убедиться, что конструктор доступен Mockito.

### Вопрос 32. Каким образом @MockBean влияет на кэширование Spring Application Context между тестами? Как минимизировать пересоздание контекста в большом тестовом модуле?

**Категория:** Тестирование · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Context Caching :: Spring Framework — docs.spring.io](https://docs.spring.io/spring-framework/reference/testing/testcontext-framework/ctx-management/caching.html)

Spring TestContext Framework кэширует `ApplicationContext` и переиспользует его для тестов с одинаковой конфигурацией. Уникальный ключ кэша формируется из комбинации параметров: `locations`, `classes`, `contextInitializerClasses`, `contextCustomizers`, `contextLoader`, `parent`, `activeProfiles`, `propertySourceDescriptors`, `propertySourceProperties`, `resourceBasePath`.

В раздел `contextCustomizers` попадают `@DynamicPropertySource`-методы, а также переопределения бинов: `@TestBean`, `@MockitoBean`, `@MockitoSpyBean` (и старый `@MockBean` из Spring Boot). Это означает, что `@MockBean` является частью ключа кэша. Если два тестовых класса объявляют одинаковые `@MockBean` — они разделяют один кэшированный контекст. Если наборы `@MockBean` различаются — создаются разные контексты.

Размер кэша по умолчанию — 32 контекста, политика вытеснения — LRU. Настраивается через JVM-параметр `spring.test.context.cache.maxSize`. Статистику кэша можно увидеть, включив `DEBUG` для `org.springframework.test.context.cache`.

#### Ответ от Claude

**Механизм кэширования контекстов**

Spring TestContext Framework кэширует `ApplicationContext` статически (в JVM-процессе). При загрузке контекста он сохраняется в кэш с ключом, уникально идентифицирующим конфигурацию теста. При запуске следующего теста фреймворк проверяет кэш: если ключ совпадает — контекст переиспользуется; если нет — создаётся новый.

**Как @MockBean влияет на ключ кэша**

`@MockBean` (Spring Boot) и `@MockitoBean` (Spring Framework 6.2+) обрабатываются как `ContextCustomizer`. Это означает, что набор `@MockBean`-объявлений в тестовом классе является **частью ключа кэша**. Изменение любого `@MockBean` — добавление, удаление, изменение типа или имени бина — создаёт новый уникальный ключ и вынуждает Spring поднять новый контекст с нуля.

Последствие: если 50 тестовых классов используют `@MockBean` с разными наборами моков, Spring создаст до 50 разных контекстов. Поднятие каждого включает сканирование classpath, создание бинов, инициализацию ресурсов, соединений с БД и т.д. — это дорогостоящая операция.

**Стратегии минимизации пересоздания контекста**

1. **Группировка тестов с одинаковыми @MockBean в один класс (или иерархию)**
   Тесты с идентичным набором `@MockBean` делят один кэшированный контекст. Выносить общую конфигурацию в абстрактный базовый класс — эффективный приём:
   ```java
   @SpringBootTest
   abstract class BaseIntegrationTest {
       @MockBean UserService userService;
       @MockBean EmailService emailService;
   }
   class OrderServiceTest extends BaseIntegrationTest { ... }
   class PaymentServiceTest extends BaseIntegrationTest { ... }
   ```

2. **Использование слайс-тестов (@WebMvcTest, @DataJpaTest и др.)**
   Слайс-тесты загружают только подмножество контекста — меньший контекст быстрее поднимается. Даже при пересоздании потери меньше, чем при полном `@SpringBootTest`.

3. **Замена @MockBean на обычный @Mock + ручная инъекция**
   Там, где полный Spring-контекст не нужен, использовать чистые юнит-тесты с `@ExtendWith(MockitoExtension.class)`. Это полностью исключает контекст Spring из теста.

4. **Единый «базовый» контекст с фиксированным набором моков**
   Определить один общий набор `@MockBean` для всего модуля и не отступать от него в отдельных тестах. Добавление мока в одном тесте разрушает кэш для всей группы.

5. **@DirtiesContext — только по необходимости**
   `@DirtiesContext` принудительно удаляет контекст из кэша после теста. Использовать исключительно когда тест действительно изменяет состояние контекста (например, модифицирует бины через `ApplicationContext.registerBean`). Бездумное применение `@DirtiesContext` — антипаттерн.

6. **ContextCustomizerFactory для модульных тестов**
   В сложных проектах (модульная архитектура, modulith) — реализовать `ContextCustomizerFactory`, где `equals()`/`hashCode()` зависят только от выбранных модулей. Это позволяет контролировать ключ кэша явно и переиспользовать контексты даже при разных наборах моков.

7. **Параллельный запуск тестов в разных JVM-процессах**
   Кэш контекстов статический и привязан к JVM-процессу. При форк-запуске (Surefire `forkCount`) каждый форк имеет свой кэш — межпроцессного переиспользования нет. Оптимальный баланс: один процесс с максимальным переиспользованием контекстов.

**Диагностика проблем с кэшем**

Включить лог: `logging.level.org.springframework.test.context.cache=DEBUG`. В логе будет видно количество попаданий и промахов кэша (`hits`, `misses`), а также текущий размер.

### Вопрос 33. Как устроена Extension Model в JUnit 5? Назовите интерфейсы расширений (BeforeEachCallback, ParameterResolver и др.) и объясните, чем она лучше механизма Rules из JUnit 4.

**Категория:** Тестирование · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [Creating custom extensions for JUnit 5 Jupiter — mscharhag.com](https://www.mscharhag.com/java/junit5-custom-extensions)

JUnit 4 предлагал два механизма расширения: **Runners** (через `@RunWith`) и **Rules** (добавлены в 4.7). Ключевое ограничение: на один тестовый класс допускался только один Runner, что делало невозможным комбинирование функциональностей разных Runners. Rules были гибче, но требовали дополнительного boilerplate-кода.

JUnit 5 заменяет оба механизма единой Extension Model. Основные интерфейсы:

| Интерфейс | Назначение |
|-----------|-----------|
| `ExecutionCondition` | Управляет тем, должен ли тест выполняться или пропускаться |
| `TestInstanceFactory` | Кастомизирует создание экземпляра тестового класса |
| `TestInstancePostProcessor` | Постобработка экземпляра после создания (инъекция зависимостей) |
| `TestInstancePreDestroyCallback` | Очистка ресурсов при уничтожении экземпляра |
| `ParameterResolver` | Разрешение параметров для конструкторов и lifecycle-методов |
| `BeforeAllCallback` / `AfterAllCallback` | Обёртка вокруг `@BeforeAll` / `@AfterAll` |
| `BeforeEachCallback` / `AfterEachCallback` | Обёртка вокруг `@BeforeEach` / `@AfterEach` |
| `BeforeTestExecutionCallback` / `AfterTestExecutionCallback` | Выполняется непосредственно до/после тестового метода |
| `InvocationInterceptor` | Позволяет вручную управлять вызовом через `proceed()` |

Ключевые преимущества: несколько расширений можно регистрировать одновременно через `@ExtendWith`; расширения получают доступ к `ExtensionContext` с информацией о тестовом классе, методе, экземпляре и аннотациях.

#### Ответ от Claude

**Архитектура Extension Model в JUnit 5**

JUnit 5 (Jupiter) строится на принципе «всё есть расширение». Базовый интерфейс — `Extension` (маркерный, без методов). Все точки расширения реализуют его и регистрируются через:
- `@ExtendWith(MyExtension.class)` — декларативно на классе или методе (повторяемая аннотация);
- `@RegisterExtension` — программно как поле тестового класса (позволяет передавать параметры в конструктор расширения);
- ServiceLoader — автоматическая регистрация через `META-INF/services`.

Один класс расширения может реализовывать несколько интерфейсов одновременно.

**Полный перечень интерфейсов расширений**

*Управление жизненным циклом тестового экземпляра:*
- `TestInstanceFactory` — создание экземпляра тестового класса (DI-фреймворк, Spring, CDI).
- `TestInstancePostProcessor` — постобработка после создания (используется `MockitoExtension`, `SpringExtension`).
- `TestInstancePreDestroyCallback` — очистка ресурсов перед уничтожением экземпляра.

*Lifecycle callbacks (порядок выполнения вокруг аннотаций):*
- `BeforeAllCallback` — выполняется **до** `@BeforeAll`-методов класса.
- `BeforeEachCallback` — выполняется **до** `@BeforeEach`-методов.
- `BeforeTestExecutionCallback` — выполняется **после** `@BeforeEach`, непосредственно перед телом теста.
- `AfterTestExecutionCallback` — выполняется **сразу после** тела теста, до `@AfterEach`.
- `AfterEachCallback` — выполняется **после** `@AfterEach`-методов.
- `AfterAllCallback` — выполняется **после** `@AfterAll`-методов.

*Разрешение параметров:*
- `ParameterResolver` — разрешает аргументы для конструктора тестового класса и любого lifecycle-метода (`@Test`, `@BeforeEach`, `@AfterEach`, `@BeforeAll`, `@AfterAll`). Метод `supportsParameter()` проверяет применимость, `resolveParameter()` возвращает значение.

*Условное выполнение:*
- `ExecutionCondition` — динамически включает/отключает тест или контейнер. На этом интерфейсе построены `@EnabledOnOs`, `@DisabledIfSystemProperty` и пр.

*Шаблоны тестов:*
- `TestTemplateInvocationContextProvider` — базис для `@ParameterizedTest` и `@RepeatedTest`.

*Обработка исключений:*
- `TestExecutionExceptionHandler` — перехватывает исключения из тела теста. Позволяет «поглотить» или трансформировать исключение.

*Перехват вызовов:*
- `InvocationInterceptor` — самый мощный интерфейс. Предоставляет `Invocation<T>` с методом `proceed()`. Позволяет оборачивать вызовы конструктора, lifecycle-методов и тела теста в произвольную логику (транзакции, смена ClassLoader, выполнение в другом потоке).

**Хранение состояния: ExtensionContext.Store**

Расширения не должны хранить состояние в полях — это нарушает изоляцию. Для хранения данных используется `ExtensionContext.Store` с иерархическим неймспейсом. Данные автоматически очищаются по завершении соответствующей области видимости (`Namespace` привязан к классу или методу).

**Сравнение с механизмом Rules из JUnit 4**

| Аспект | JUnit 4 Rules | JUnit 5 Extensions |
|--------|---------------|-------------------|
| Концепций для расширения | Два: `@Rule`/`@ClassRule` + `@RunWith` | Одна: `Extension` |
| Количество на класс | Один Runner, несколько Rules | Несколько расширений без ограничений |
| Точки перехвата | `Statement`-цепочка (декоратор) | 10+ специализированных callback-интерфейсов |
| Параметры тестовых методов | Отсутствуют в стандартной модели | Полноценный `ParameterResolver` |
| Комбинирование | `RuleChain` — явная цепочка с boilerplate | Несколько `@ExtendWith` — прозрачная композиция |
| Хранение состояния | Поля в правиле или тестовом классе | Изолированный `ExtensionContext.Store` |
| Регистрация | Только декларативно (поле с `@Rule`) | Декларативно, программно, через ServiceLoader |
| Условное выполнение | Не поддерживается | `ExecutionCondition` |
| Кастомные аннотации | Невозможны как точка интеграции | Мета-аннотации работают с `@ExtendWith` |

Ключевой недостаток JUnit 4: `@RunWith` допускает только один Runner. Это означало, что нельзя было одновременно использовать, например, `SpringJUnit4ClassRunner` и `Parameterized`. Пользователи вынуждены были идти на обходные решения (вложенные Runner-ы, специальные библиотеки). В JUnit 5 это ограничение снято: `@ExtendWith({SpringExtension.class, MockitoExtension.class})` работает «из коробки».

Механизм `Statement` (Rule как декоратор над `Statement`) в JUnit 4 неудобен для отладки и плохо поддаётся типизации. В JUnit 5 каждая точка расширения — отдельный именованный интерфейс с явной сигнатурой.

**Пример реализации расширения с несколькими интерфейсами:**

```java
public class TimingExtension
        implements BeforeTestExecutionCallback, AfterTestExecutionCallback {

    private static final String START_TIME = "start time";

    @Override
    public void beforeTestExecution(ExtensionContext ctx) {
        ctx.getStore(GLOBAL).put(START_TIME, System.currentTimeMillis());
    }

    @Override
    public void afterTestExecution(ExtensionContext ctx) {
        long start = ctx.getStore(GLOBAL).remove(START_TIME, long.class);
        long duration = System.currentTimeMillis() - start;
        System.out.printf("Test [%s] took %d ms%n",
            ctx.getRequiredTestMethod().getName(), duration);
    }
}
```

```java
@ExtendWith(TimingExtension.class)
class MyServiceTest {
    @Test void shouldDoSomething() { ... }
}
```
### Вопрос 34. Mockito по умолчанию не может мокировать final-классы и static-методы. Как это решается начиная с Mockito 2/3 (MockMaker, mockito-extensions)? Каковы ограничения этого подхода?

**Категория:** Тестирование · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [How to use the Mockito's inline mock maker — davidvlijmincx.com](https://davidvlijmincx.com/posts/writing_higher_quality_tests_with_mockitos_inline_mock_maker/)

Для мокирования final-классов и методов необходимо активировать inline MockMaker. В Mockito 4 и ниже это делается двумя способами:

1. **Через зависимость**: добавить артефакт `mockito-inline` в тестовые зависимости Maven/Gradle.
2. **Через конфигурационный файл** (без добавления зависимости): создать файл `src/test/resources/mockito-extensions/org.mockito.plugins.MockMaker` с единственной строкой `mock-maker-inline`.

После включения inline MockMaker синтаксис мокирования final-классов ничем не отличается от обычных классов — используются стандартные вызовы `mock()` и `when()`. Для мокирования статических методов применяется паттерн `MockedStatic<T>` внутри блока try-with-resources:

```java
try (MockedStatic<Instant> mocked = Mockito.mockStatic(Instant.class)) {
    mocked.when(Instant::now).thenReturn(mockInstant);
    // тест
}
```

В Mockito 5.0.0 inline MockMaker стал дефолтным — отдельная конфигурация больше не нужна.

#### Ответ от Claude

**Исторический контекст**

До Mockito 2 единственным способом мокировать final-классы и статические методы был PowerMock — отдельная библиотека с собственным раннером. Начиная с Mockito 2 появился экспериментальный `InlineMockMaker` (на базе Java Instrumentation API и Byte Buddy), а с версии 3.4.0 — поддержка статических методов через `MockedStatic`.

**Способы включения (Mockito 2–4)**

Способ 1 — отдельная зависимость (Mockito 2–4):
```xml
<dependency>
    <groupId>org.mockito</groupId>
    <artifactId>mockito-inline</artifactId>
    <version>4.x.x</version>
    <scope>test</scope>
</dependency>
```

Способ 2 — через файл расширений (без изменения зависимостей):
Создать файл `src/test/resources/mockito-extensions/org.mockito.plugins.MockMaker` с содержимым:
```
mock-maker-inline
```

**Mockito 5+**

Начиная с Mockito 5.0.0, inline MockMaker используется по умолчанию. Минимальная версия Java повышена с 8 до 11. Для сред, где inline-мокинг невозможен (например, GraalVM Native Image), доступен артефакт `mockito-subclass` с прежним поведением на основе подклассов.

**Мокирование статических методов**

```java
try (MockedStatic<MyUtils> mocked = Mockito.mockStatic(MyUtils.class)) {
    mocked.when(() -> MyUtils.compute(42)).thenReturn("result");
    assertEquals("result", MyUtils.compute(42));
}
```

Блок try-with-resources обязателен: mock остаётся активным до вызова `close()`, иначе следующие тесты получат замокированное поведение.

**Ограничения inline MockMaker**

1. **Классы bootstrap-загрузчика**: нельзя мокировать `java.lang.String`, `java.lang.Object`, примитивные обёртки (`Integer`, `Long` и т.д.) и другие классы, загружаемые bootstrap ClassLoader-ом.
2. **Thread-safety статических моков**: `MockedStatic` работает только в потоке, создавшем мок. Код, запускающий дочерние потоки, не увидит мокированного поведения.
3. **Java 21+**: JVM предупреждает о динамической загрузке агента (`Dynamic agent loading`) и в будущих версиях JDK это будет запрещено. Нужна явная настройка Maven Surefire/Failsafe для передачи Mockito-агента через `-javaagent`.
4. **GraalVM Native Image**: bytecode-инструментирование не работает в AOT-скомпилированном коде — нужен `mockito-subclass`.
5. **Производительность**: инструментирование байткода добавляет накладные расходы; для классов, мокируемых через `MockedStatic`, каждый вызов `mockStatic()` перехватывает вызовы глобально.
6. **Область видимости**: незакрытый `MockedStatic` или `MockedConstruction` приведёт к утечке мока между тестами — всегда используйте try-with-resources или аннотации JUnit-расширений.

### Вопрос 35. Что такое @DirtiesContext? Какова его стоимость с точки зрения производительности и в каких случаях без него действительно не обойтись?

**Категория:** Тестирование · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [DirtiesContext (Spring Framework API) — docs.spring.io](https://docs.spring.io/spring-framework/docs/current/javadoc-api/org/springframework/test/annotation/DirtiesContext.html)

`@DirtiesContext` — тестовая аннотация Spring, которая указывает, что `ApplicationContext` ассоциированный с тестом является «грязным» и должен быть закрыт и удалён из кэша контекстов. Используется, когда тест изменил контекст — например, изменил состояние singleton-бина или встроенной базы данных. Последующие тесты, запрашивающие тот же контекст, получат свежий экземпляр.

Аннотация может применяться как на уровне класса, так и на уровне метода.

**Атрибут `classMode`** (при использовании на классе):
- `BEFORE_CLASS` — до выполнения класса;
- `BEFORE_EACH_TEST_METHOD` — до каждого метода;
- `AFTER_EACH_TEST_METHOD` — после каждого метода;
- `AFTER_CLASS` (по умолчанию) — после всего класса.

**Атрибут `methodMode`** (при использовании на методе):
- `BEFORE_METHOD` — до теста;
- `AFTER_METHOD` (по умолчанию) — после теста.

**Атрибут `hierarchyMode`** (при `@ContextHierarchy`):
- `EXHAUSTIVE` (по умолчанию) — очищает всю иерархию;
- `CURRENT_LEVEL` — только текущий уровень.

#### Ответ от Claude

**Что делает `@DirtiesContext`**

Spring TestContext Framework кэширует `ApplicationContext` между тестами: если конфигурация одна и та же, контекст создаётся один раз и переиспользуется. `@DirtiesContext` принудительно вытесняет контекст из кэша, сигнализируя фреймворку, что данный контекст изменён и дальнейшее его переиспользование ненадёжно.

**Стоимость производительности**

Создание `ApplicationContext` — дорогостоящая операция:
- Каждый новый контекст добавляет 2–3 секунды к времени выполнения тестов (реальные цифры из производственных кодовых баз).
- При 30 дополнительных контекстах суммарная потеря — до 90 секунд, что может составлять 25% и более от общего времени прогона тестов.
- Spring кэширует до 32 контекстов по умолчанию; `@DirtiesContext` уничтожает запись в кэше принудительно, лишая последующие тесты возможности переиспользования.
- При `BEFORE_EACH_TEST_METHOD` / `AFTER_EACH_TEST_METHOD` контекст пересоздаётся на каждый тест — стоимость растёт линейно с количеством методов.

**Когда без `@DirtiesContext` действительно не обойтись**

1. **Изменение состояния singleton-бинов**: тест вручную изменяет поля singleton через `ReflectionTestUtils` или прямой инжекцией, и это состояние нельзя откатить.
2. **Изменение статического состояния JVM**: тест модифицирует системные свойства (`System.setProperty`) или переменные окружения таким образом, что они влияют на бины, уже созданные в контексте.
3. **Встроенные базы данных с миграциями**: когда тест выполняет DDL-изменения (CREATE TABLE, DROP TABLE) и механизм `@Transactional` + rollback недостаточен, например при тестировании самих миграций Flyway/Liquibase.
4. **Изменение контекста через `ConfigurableApplicationContext`**: тест вызывает `context.close()`, регистрирует/удаляет bean-определения.
5. **Тестирование `ApplicationListener` с побочными эффектами**: слушатели, изменяющие глобальное состояние при получении событий.

**Лучшие альтернативы (предпочтительны перед `@DirtiesContext`)**

| Проблема | Альтернатива |
|---|---|
| Загрязнение БД | `@Transactional` + rollback; `@Sql` с cleanup-скриптом; `@AfterEach` с удалением данных |
| Разные свойства | `@TestPropertySource`; `@DynamicPropertySource` |
| Разные бины | `@MockBean`/`@SpyBean`; `@TestConfiguration` |
| Состояние singleton | `ReflectionTestUtils` для восстановления в `@AfterEach` |

Ключевое правило: `@DirtiesContext` — последнее средство, а не способ скрыть некорректную изоляцию тестов. Появление этой аннотации в большом количестве тестов — симптом архитектурной проблемы с разделяемым изменяемым состоянием.

### Вопрос 36. Опишите цикл Red–Green–Refactor в TDD. Как TDD влияет на дизайн кода и какие антипаттерны возникают, когда разработчики пишут тесты после реализации?

**Категория:** Тестирование · **Уровень:** Senior

#### Оригинальный ответ из интернета
> Источник: [bliki: Test Driven Development — martinfowler.com](https://www.martinfowler.com/bliki/TestDrivenDevelopment.html)

По определению Мартина Фаулера, Test-Driven Development (TDD) — это техника построения ПО, разработанная Кентом Беком в конце 1990-х как часть Extreme Programming. Процесс состоит из трёх повторяющихся шагов:

1. **Red**: написать тест на следующую функциональность, которую нужно реализовать.
2. **Green**: написать ровно столько кода, чтобы тест прошёл.
3. **Refactor**: привести в порядок как новый, так и существующий код.

Фаулер подчёркивает, что TDD — это прежде всего **дисциплина проектирования**, а не только тестирования. Написание тестов до кода заставляет думать об интерфейсе и о том, как класс будет использоваться, что способствует разделению интерфейса и реализации. Наиболее распространённый антипаттерн, который он выделяет, — это **пропуск фазы Refactor**: даже при наличии тестов пренебрежение рефакторингом ведёт к деградации дизайна.

#### Ответ от Claude

**Цикл Red–Green–Refactor**

Цикл является атомарной единицей работы в TDD и намеренно очень короткий — от нескольких секунд до нескольких минут на итерацию.

**Red — думай, что нужно сделать**
- Написать один маленький тест, описывающий конкретное ожидаемое поведение.
- Тест должен компилироваться, но падать — это подтверждает, что функциональность ещё не реализована.
- Красный цвет важен: если тест зелёный сразу, либо функциональность уже есть, либо тест написан неверно.

**Green — думай, как заставить тест пройти**
- Написать минимально необходимый код для прохождения теста.
- На этом этапе допустимы «грязные» решения: copy-paste, захардкоженные значения, нарушение DRY. Главное — зелёный тест.
- Не добавлять логику, которую тест не проверяет.

**Refactor — думай, как улучшить код**
- Убрать дублирование, улучшить именование, повысить читаемость.
- Тесты остаются зелёными — они служат страховочной сетью.
- Рефакторинг касается как продакшн-кода, так и самих тестов.

**Влияние TDD на дизайн кода**

1. **Тестируемость как первый критерий дизайна**: код, написанный через TDD, изначально слабо связан (loose coupling), потому что тяжело тестируемый код — это немедленная обратная связь о плохом дизайне. Класс с 10 зависимостями болезненно тестировать — это вынуждает применять принцип единственной ответственности.
2. **Фокус на интерфейсе, а не на реализации**: тест пишется с позиции потребителя API, что естественно ведёт к разделению интерфейса и реализации.
3. **Инкрементальный дизайн**: вместо попытки спроектировать систему целиком заранее, дизайн возникает итеративно и эволюционирует под реальные требования, выраженные тестами.
4. **Живая документация**: тесты точно описывают ожидаемое поведение системы, становясь исполняемой спецификацией.

**Антипаттерны при написании тестов после реализации (test-last)**

1. **Тестирование реализации, а не поведения**: разработчик смотрит на готовый код и пишет тесты, покрывающие его внутренние детали. При рефакторинге такие тесты ломаются, даже если поведение не изменилось. Это создаёт порочный круг: тесты требуют постоянной поддержки, разработчики начинают считать тестирование «потерей времени».
2. **«Гигантский» тест (The Giant)**: попытка покрыть сразу весь пользовательский сценарий одним тестом с десятками assertions. Тест не документирует конкретное поведение и при падении неясно, что именно сломалось.
3. **Чрезмерная настройка (Excessive Setup)**: код уже написан с жёсткими зависимостями, поэтому тест вынужден инициализировать огромное количество объектов. Симптом: больше кода настройки, чем самого теста.
4. **«Лжец» (The Liar)**: тест написан так, чтобы заведомо пройти (например, проверяется не то, что нужно), с целью формально повысить покрытие. Даёт ложное ощущение надёжности.
5. **Тавтологические тесты**: тест повторяет логику реализации вместо того, чтобы проверять ожидаемый результат. Любая ошибка в реализации воспроизводится и в тесте, поэтому тест никогда не поймает баг.
6. **Низкая ценность при code review**: без TDD нет «красной» стадии — невозможно убедиться, что тест действительно проверяет нужное поведение, а не просто проходит потому что код уже написан.
7. **Игнорирование рефакторинга**: достигнув покрытия, разработчики не трогают код — нет итерации Refactor, дизайн деградирует.

**Итог**: TDD — это не про «сначала тесты», а про управляемый дизайн через короткие петли обратной связи. Тесты после реализации дают некоторую страховку от регрессий, но не обеспечивают ни дизайнового давления, ни уверенности в том, что тест проверяет именно то поведение, которое нужно.
### Вопрос 37. Что такое пять типов тестовых двойников по классификации Джерарда Месароша (Meszaros): Dummy, Fake, Stub, Spy, Mock? Чем Fake отличается от Stub?

**Категория:** Тестирование · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [bliki: Test Double — Martin Fowler](https://martinfowler.com/bliki/TestDouble.html)

Джерард Месарош ввёл термин «тестовый двойник» (Test Double) как общее обозначение любого объекта, заменяющего реальную зависимость в тестах. Fowler пересказывает классификацию Месароша следующим образом:

- **Dummy** — объекты, которые передаются, но никогда не используются. Как правило, нужны лишь для заполнения списков параметров.
- **Fake** — объекты с реальной рабочей реализацией, но использующие «срезанный путь», непригодный для продакшена (классический пример — база данных в памяти).
- **Stub** — заглушки, возвращающие заранее заданные («консервированные») ответы на вызовы, сделанные в ходе теста. На всё, что выходит за рамки запрограммированного, они не реагируют.
- **Spy** — стабы, которые дополнительно записывают информацию о том, как именно они были вызваны. Пример — сервис электронной почты, ведущий счёт отправленных сообщений.
- **Mock** — объекты, заранее запрограммированные с ожиданиями, формирующими спецификацию вызовов. Если они получают вызов, который не ожидали, — выбрасывают исключение и тем самым самостоятельно проваливают тест.

**Чем Fake отличается от Stub:** Fake содержит настоящую, пусть и упрощённую логику (например, `HashMap` вместо реальной БД). Stub лишь возвращает фиксированные («консервированные») значения и не содержит никакой деловой логики.

#### Ответ от Claude

**Контекст.** Классификация введена Джерардом Месарошем в книге «xUnit Test Patterns» (2007) и популяризирована Мартином Фаулером. Термин «тестовый двойник» аналогичен понятию «каскадёр» в кино: реальный объект заменяется дублёром для нужд теста.

**Пять типов тестовых двойников:**

| Тип | Суть | Ключевая характеристика |
|-----|------|------------------------|
| **Dummy** | Пустышка-заполнитель | Передаётся в метод, но не вызывается |
| **Stub** | Заглушка с фиксированными ответами | Возвращает «консервированные» данные, логики нет |
| **Fake** | Упрощённая рабочая реализация | Содержит реальную логику, но непригодна для продакшена |
| **Spy** | Наблюдатель | Stub + запись вызовов для проверки в фазе assert |
| **Mock** | Объект с ожиданиями | Проверяет взаимодействие сам; падает при неожиданном вызове |

**Dummy** используется, когда зависимость обязательна по сигнатуре, но реально не задействована в тестируемом пути:
```java
// EmailService нигде не вызывается, просто передаётся в конструктор
EmailService dummy = mock(EmailService.class);
OrderService service = new OrderService(dummy);
```

**Stub** возвращает фиксированное значение, не выполняя никаких вычислений:
```java
UserRepository stub = mock(UserRepository.class);
when(stub.findById(1L)).thenReturn(new User(1L, "Alice"));
```

**Fake** содержит рабочую (но упрощённую) реализацию. Поведение зависит от состояния и входных данных:
```java
class FakeUserRepository implements UserRepository {
    private final Map<Long, User> store = new HashMap<>();

    @Override
    public void save(User user) { store.put(user.getId(), user); }

    @Override
    public Optional<User> findById(Long id) { return Optional.ofNullable(store.get(id)); }
}
```

**Spy** записывает факты взаимодействия для последующей проверки:
```java
List<String> spyList = spy(new ArrayList<>());
spyList.add("hello");
verify(spyList).add("hello"); // проверяем, что метод был вызван
```

**Mock** задаёт ожидания заранее и самостоятельно верифицирует их:
```java
MailSender mock = mock(MailSender.class);
// если send() не будет вызван - тест упадёт
verify(mock).send(any(Email.class));
```

**Ключевое различие Fake vs Stub:**

- **Stub** — пассивный. Он не знает ничего о предыдущих вызовах и не вычисляет результат. Ответ задан жёстко в тесте: `when(repo.findById(1L)).thenReturn(user)`.
- **Fake** — активный. Он содержит реальную логику (хранилище, алгоритм), реагирующую на входные данные. При вызове `save()` объект действительно сохраняется, и последующий `findById()` его найдёт. Fakes обычно реализуются как отдельные классы, а не через Mockito.

Stub годится для изоляции одного конкретного вызова. Fake нужен, когда тест проверяет несколько взаимодействий с сохранением состояния между ними (например, сохранить запись — найти её — удалить).

**Верификация по источникам:** Martinfowler.com, shaunabram.com и youngju.dev единогласны в определениях. Незначительное расхождение: некоторые авторы называют Spy разновидностью Stub (Fowler), другие выделяют его в отдельный тип наравне с Mock (youngju.dev). Месарош рассматривает Spy как отдельный тип.

### Вопрос 38. Что делает Mockito.thenAnswer() и чем он отличается от thenReturn()? Для каких сценариев нужен doAnswer(), и как при его использовании получить доступ к аргументам вызова?

**Категория:** Тестирование · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Differences between thenReturn and thenAnswer in Mockito — chdr.tech](https://chdr.tech/en/2025/05/14/differences-between-thenreturn-and-thenanswer-in-mockito/)

`thenReturn` используется для возврата статического, заранее определённого значения при каждом вызове мока. Это наиболее простой и читаемый способ:

```java
List<String> mockList = Mockito.mock(List.class);
Mockito.when(mockList.get(0)).thenReturn("Hello, Mockito!");
assertEquals("Hello, Mockito!", mockList.get(0));
```

`thenAnswer` более гибок: он позволяет возвращать динамически вычисленные значения на основе аргументов вызова. Доступ к аргументам осуществляется через объект `InvocationOnMock`:

```java
Mockito.when(mockList.get(Mockito.anyInt()))
    .thenAnswer(invocation -> {
        Integer index = invocation.getArgument(0);
        return "Dynamic value for index: " + index;
    });
```

Когда ответ постоянен — используйте `thenReturn`. Когда ответ зависит от аргументов или требует вычислений — используйте `thenAnswer`.

#### Ответ от Claude

**thenReturn() — статический ответ**

`thenReturn(value)` задаёт фиксированное возвращаемое значение. Значение вычисляется один раз в момент настройки стаба и больше не изменяется:

```java
UserRepository repo = mock(UserRepository.class);
when(repo.findById(anyLong())).thenReturn(new User(1L, "Alice"));
// Всегда вернёт того же пользователя, независимо от переданного id
```

Можно задать цепочку ответов — при первом вызове вернётся первое значение, при втором — второе:
```java
when(repo.findById(anyLong()))
    .thenReturn(new User(1L, "Alice"))
    .thenReturn(new User(2L, "Bob"));
```

**thenAnswer() — динамический ответ**

`thenAnswer(Answer<T>)` принимает функцию, которая вызывается при каждом обращении к мок-методу. Через объект `InvocationOnMock` доступны аргументы текущего вызова:

```java
when(person.talk(anyString())).thenAnswer(invocation -> {
    String phrase = invocation.getArgument(0); // получаем первый аргумент
    if ("Good afternoon".equals(phrase)) {
        return "Good afternoon to you too";
    }
    return "Hello!";
});
```

Сценарии, где `thenAnswer` необходим:
- Ответ зависит от аргументов вызова.
- Нужно смоделировать side-эффект (например, заполнить коллекцию, переданную по ссылке).
- Нужно выбросить исключение по условию.
- Тестируется логика с колбэками (callback-based APIs).

**doAnswer() — для void-методов и особых случаев**

Синтаксис `when(...).thenAnswer(...)` не работает с `void`-методами: компилятор не позволяет передать вызов `void`-метода в `when()`. Для таких случаев используется альтернативный синтаксис `doAnswer(...).when(mock).method(...)`:

```java
doAnswer(invocation -> {
    String name = invocation.getArgument(0);   // первый аргумент
    int age = invocation.getArgument(1);        // второй аргумент
    System.out.println("Saving user: " + name + ", age: " + age);
    return null; // void-метод должен вернуть null
}).when(userService).saveUser(anyString(), anyInt());
```

`doAnswer` также применяется при работе со шпионами (`spy`), когда нужно предотвратить вызов реального метода, но при этом выполнить собственную логику.

**Доступ к аргументам через InvocationOnMock:**

| Метод | Описание |
|-------|----------|
| `invocation.getArgument(0)` | Аргумент по индексу с автоматическим приведением типа |
| `invocation.getArgument(0, String.class)` | Аргумент по индексу с явным указанием типа |
| `invocation.getArguments()` | Массив всех аргументов `Object[]` |
| `invocation.getMethod()` | Объект `java.lang.reflect.Method` вызванного метода |

**Сводная таблица:**

| | `thenReturn` | `thenAnswer` | `doAnswer` |
|---|---|---|---|
| Тип ответа | Фиксированный | Динамический | Динамический |
| Работает с void | Нет | Нет | Да |
| Доступ к аргументам | Нет | Да (`InvocationOnMock`) | Да (`InvocationOnMock`) |
| Синтаксис | `when(m()).thenReturn(v)` | `when(m()).thenAnswer(a)` | `doAnswer(a).when(mock).m()` |

**Расхождение между источниками:** Официальный javadoc Mockito называет `thenAnswer` «довольно противоречивой фичей, не включённой в Mockito изначально» и рекомендует по возможности использовать `thenReturn`/`thenThrow` для лучшей читаемости. Baeldung и chdr.tech напротив демонстрируют `thenAnswer` как рядовой инструмент. Оба подхода корректны; рекомендация Mockito касается именно читаемости, а не корректности.

### Вопрос 39. Для чего используются verifyNoMoreInteractions() и verifyNoInteractions() в Mockito? В чём разница между ними и почему официальная документация предостерегает от повсеместного применения verifyNoMoreInteractions()?

**Категория:** Тестирование · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [Mockito — verifyNoMoreInteractions() and verifyNoInteractions() — LogicBig](https://www.logicbig.com/tutorials/unit-testing/mockito/verify-no-more-interactions.html)

**verifyNoMoreInteractions(Object... mocks)** — проверяет, что после всех явных вызовов `verify()` с указанными моками не осталось неверифицированных взаимодействий. Метод позволяет убедиться, что никакие лишние вызовы не произошли:

```java
// Вызовы
mockedList.add("one");
mockedList.clear();

// Верификация
verify(mockedList).add("one");
verify(mockedList).clear();
verifyNoMoreInteractions(mockedList); // пройдёт, все вызовы верифицированы
```

**verifyNoInteractions(Object... mocks)** — проверяет, что с указанными моками не произошло вообще никаких взаимодействий:

```java
verifyNoInteractions(mockedList); // упадёт, если любой метод был вызван
```

Ключевое различие: `verifyNoInteractions` проверяет полное отсутствие вызовов, а `verifyNoMoreInteractions` — отсутствие вызовов, которые не были явно верифицированы.

#### Ответ от Claude

**verifyNoInteractions(mock)**

Утверждает, что с данным моком не произошло ни одного взаимодействия — ни до, ни во время теста. Используется, когда по логике теста мок вообще не должен вызываться:

```java
EmailService emailService = mock(EmailService.class);
userService.processInactiveUser(user); // не должен отправлять письмо

verifyNoInteractions(emailService); // упадёт, если send() был вызван
```

**verifyNoMoreInteractions(mock)**

Проверяет, что после всех вызовов `verify(...)` не осталось неверифицированных взаимодействий. Требует предварительной верификации интересующих вызовов:

```java
verify(orderRepo).save(any(Order.class));
verify(emailService).send(any(Email.class));
verifyNoMoreInteractions(orderRepo, emailService);
// Упадёт, если был вызван, например, orderRepo.findById(), который не верифицировали
```

**Историческая справка.** В старых версиях Mockito существовал `verifyZeroInteractions()` — он был псевдонимом `verifyNoInteractions()` и в Mockito 3.x объявлен устаревшим (`@Deprecated`). В актуальных версиях (4.x, 5.x) следует использовать `verifyNoInteractions()`.

**Сравнительная таблица:**

| | `verifyNoInteractions` | `verifyNoMoreInteractions` |
|---|---|---|
| Проверяет | Полное отсутствие любых вызовов | Отсутствие вызовов, не верифицированных через `verify()` |
| Нужны ли предварительные `verify()` | Нет | Да (иначе падает на первом же вызове) |
| Типичный сценарий | Мок не должен вызываться вообще | Убедиться, что не было лишних вызовов помимо ожидаемых |

**Почему официальная документация предостерегает от повсеместного применения verifyNoMoreInteractions()**

Официальный javadoc Mockito (все версии от 2.x до 5.x) содержит явное предупреждение:

> «Некоторые пользователи, привыкшие к классическому подходу expect-run-verify, склонны использовать `verifyNoMoreInteractions()` в каждом тестовом методе. `verifyNoMoreInteractions()` не рекомендуется использовать в каждом тестовом методе. Злоупотребление им ведёт к переспецифицированным, сложным в поддержке тестам.»

Конкретные причины:

1. **Хрупкость тестов.** Если реализация добавит новый вспомогательный вызов мока, не влияющий на результат теста, `verifyNoMoreInteractions` сломает тест. Тест должен проверять поведение, а не полный список всех взаимодействий с зависимостями.

2. **Ложные падения из setUp/@Before.** Метод учитывает взаимодействия, произошедшие до тестового метода — в `@BeforeEach`, конструкторе или фабричном методе. Это приводит к неочевидным ошибкам.

3. **Дублирование информации.** Если вы уже верифицировали все нужные вызовы через `verify()`, добавление `verifyNoMoreInteractions()` не даёт дополнительной ценности, но увеличивает связность теста с реализацией.

4. **Альтернатива.** Для явного указания на то, что метод не должен вызываться, документация рекомендует использовать `verify(mock, never()).someMethod()` — это более читаемо и точно выражает намерение.

**Правило применения:** `verifyNoMoreInteractions` уместен только в тестах, где критически важно, что никакие побочные вызовы не происходят (например, не должен вызываться внешний платёжный шлюз). В большинстве unit-тестов достаточно верифицировать только то, что важно для конкретного теста.
### Вопрос 40. Как проверить порядок вызовов методов на нескольких mock-объектах в Mockito с помощью InOrder? Что произойдёт, если порядок нарушен?

**Категория:** Тестирование · **Уровень:** Middle

#### Оригинальный ответ из интернета
> Источник: [InOrder (Mockito 2.2.7 API) — site.mockito.org](https://site.mockito.org/javadoc/current/org/mockito/InOrder.html)

Интерфейс `InOrder` позволяет выполнять проверку вызовов в заданном порядке. Объект `InOrder` создаётся передачей нужных mock-объектов в метод `Mockito.inOrder(...)`. Далее на нём вызывается `verify()` — в той последовательности, в которой ожидаются реальные вызовы.

```java
InOrder inOrder = inOrder(firstMock, secondMock);

inOrder.verify(firstMock).someMethod("was called first");
inOrder.verify(secondMock).someMethod("was called second");
```

Верификация гибкая: не нужно проверять каждый вызов — достаточно указать только те взаимодействия, порядок которых важен. Допустимо использовать режимы верификации (`times(x)`, `atLeastOnce()`):

```java
inOrder.verify(firstMock, times(2)).someMethod("called twice");
inOrder.verify(secondMock, atLeastOnce()).someMethod("called at least once");
```

Метод `inOrder.verifyNoMoreInteractions()` отличается от статического `Mockito.verifyNoMoreInteractions()`: он проверяет отсутствие вызовов только **после** последнего верифицированного вызова (с учётом порядка), а не среди всех зафиксированных взаимодействий.

Если порядок нарушен, Mockito бросает исключение `VerificationInOrderFailure` (наследует `MockitoAssertionError`).

#### Ответ от Claude

**Создание InOrder и базовое использование**

Метод `Mockito.inOrder(Object... mocks)` принимает один или несколько mock-объектов и возвращает экземпляр `InOrder`. В него передаются только те моки, взаимный порядок вызовов которых нужно проверить.

```java
// Создание двух mock-объектов
ServiceA serviceA = mock(ServiceA.class);
ServiceB serviceB = mock(ServiceB.class);

// Код под тестом вызывает сначала serviceA, потом serviceB
objectUnderTest.doSomething();

// Верификация порядка
InOrder inOrder = inOrder(serviceA, serviceB);
inOrder.verify(serviceA).methodA();
inOrder.verify(serviceB).methodB();
```

**Верификация на одном моке**

InOrder применим и к одному mock-объекту, если важен порядок вызовов его методов:

```java
InOrder inOrder = inOrder(singleMock);
inOrder.verify(singleMock).firstMethod();
inOrder.verify(singleMock).secondMethod();
```

**Гибкость верификации**

Проверять каждый вызов не обязательно — в `inOrder.verify()` указываются только те взаимодействия, порядок которых принципиален. Пропущенные между ними вызовы игнорируются. Поддерживаются все стандартные режимы верификации (`times(n)`, `atLeastOnce()`, `atLeast(n)`, `never()` и т.д.).

**Что происходит при нарушении порядка**

Если `inOrder.verify(mockB).method()` вызывается раньше, чем фактически был вызван `mockB` относительно предыдущего верифицированного вызова, Mockito выбрасывает `VerificationInOrderFailure` — подкласс `MockitoAssertionError`. Тест завершается с ошибкой и сообщением о том, что взаимодействие найдено, но не в ожидаемом порядке (либо вообще не найдено в части оставшейся последовательности).

**verifyNoMoreInteractions() в контексте InOrder**

```java
mock.foo(); // 1-й
mock.bar(); // 2-й
mock.baz(); // 3-й

InOrder inOrder = inOrder(mock);
inOrder.verify(mock).bar();
inOrder.verify(mock).baz();
inOrder.verifyNoMoreInteractions(); // проходит: после baz() вызовов нет
```

В отличие от статического `Mockito.verifyNoMoreInteractions()`, метод экземпляра `InOrder` учитывает только то, что произошло **после** последнего верифицированного взаимодействия, а не все взаимодействия с моком.
