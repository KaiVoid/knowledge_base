# Урок 4. Аннотации

**Трейл:** Learning the Java Language · **Оригинал:** [Annotations](https://docs.oracle.com/javase/tutorial/java/annotations/index.html)
**Связанные области:** [[01-core-java-syntax-oop]] · [[09-modern-java-features]] · **Вопросы:** core-java

> Перевод официального руководства Oracle (The Java Tutorials, JDK 8). Объединяет страницы
> *Lesson: Annotations*, *Annotations Basics*, *Declaring an Annotation Type*,
> *Predefined Annotation Types*, *Type Annotations and Pluggable Type Systems* и
> *Repeating Annotations*.

**Аннотации** (*annotations*) — это разновидность метаданных (*metadata*): они предоставляют
данные о программе, которые не являются частью самой программы. Аннотации не оказывают прямого
влияния на работу кода, который они аннотируют.

## Для чего применяются аннотации

У аннотаций множество применений, среди которых:

- **Информация для компилятора.** Аннотации могут использоваться компилятором для обнаружения
  ошибок или подавления предупреждений.
- **Обработка во время компиляции и развёртывания (*compile-time and deployment-time
  processing*).** Программные инструменты могут обрабатывать информацию аннотаций для генерации
  кода, XML-файлов и т. п.
- **Обработка во время выполнения (*runtime processing*).** Некоторые аннотации доступны для
  анализа во время выполнения программы.

Этот урок объясняет, где можно применять аннотации, как их применять, какие предопределённые
типы аннотаций доступны в Java Platform, Standard Edition (Java SE API), как аннотации типов
(*type annotations*) совместно с подключаемыми системами типов (*pluggable type systems*)
позволяют писать код с более строгой проверкой типов, и как реализовать повторяющиеся аннотации
(*repeating annotations*).

## Формат аннотации

В простейшей форме аннотация выглядит так:

```java
@Entity
```

Символ «собака» (`@`) указывает компилятору, что далее следует аннотация. В следующем примере
имя аннотации — `Override`:

```java
@Override
void mySuperMethod() { ... }
```

Аннотация может включать **элементы** (*elements*), которые бывают именованными и неименованными,
и эти элементы имеют значения:

```java
@Author(
   name = "Benjamin Franklin",
   date = "3/27/2003"
)
class MyClass { ... }
```

или

```java
@SuppressWarnings(value = "unchecked")
void myMethod() { ... }
```

Если есть лишь один элемент с именем `value`, то имя можно опустить, как здесь:

```java
@SuppressWarnings("unchecked")
void myMethod() { ... }
```

Если у аннотации нет элементов, то можно опустить и круглые скобки — как было показано выше
в примере с `@Override`.

Также допустимо использовать несколько аннотаций для одного и того же объявления:

```java
@Author(name = "Jane Doe")
@EBook
class MyClass { ... }
```

Если аннотации имеют один и тот же тип, то это называется **повторяющейся аннотацией**
(*repeating annotation*):

```java
@Author(name = "Jane Doe")
@Author(name = "John Smith")
class MyClass { ... }
```

Повторяющиеся аннотации поддерживаются начиная с выпуска Java SE 8. Подробнее см. раздел
«Повторяющиеся аннотации».

Тип аннотации может быть одним из типов, определённых в пакетах `java.lang` или
`java.lang.annotation` из состава Java SE API. В предыдущих примерах `Override` и
`SuppressWarnings` — это предопределённые аннотации Java. Можно также определить собственный
тип аннотации. Аннотации `Author` и `Ebook` из примера выше — это пользовательские типы
аннотаций.

### Где можно применять аннотации

Аннотации можно применять к **объявлениям** (*declarations*): объявлениям классов, полей,
методов и других программных элементов. При использовании в объявлении каждая аннотация по
соглашению часто записывается на отдельной строке.

Начиная с выпуска Java SE 8, аннотации можно применять также к **использованию типов**
(*use of types*). Вот несколько примеров:

- Выражение создания экземпляра класса (*class instance creation expression*):
    ```java
    new @Interned MyObject();
    ```

- Приведение типа (*type cast*):
    ```java
    myString = (@NonNull String) str;
    ```

- Конструкция `implements`:
    ```java
    class UnmodifiableList<T> implements
        @Readonly List<@Readonly T> { ... }
    ```

- Объявление выбрасываемого исключения (*thrown exception declaration*):
    ```java
    void monitorTemperature() throws
        @Critical TemperatureException { ... }
    ```

Эта форма аннотации называется **аннотацией типа** (*type annotation*).

## Объявление типа аннотации

Многие аннотации заменяют комментарии в коде.

Предположим, что группа разработчиков по традиции начинает тело каждого класса с комментариев,
содержащих важную информацию:

```java
public class Generation3List extends Generation2List {

   // Автор: John Doe
   // Дата: 3/17/2002
   // Текущая ревизия: 6
   // Последнее изменение: 4/12/2004
   // Кем: Jane Doe
   // Рецензенты: Alice, Bill, Cindy

   // здесь идёт код класса

}
```

Чтобы добавить те же метаданные с помощью аннотации, сначала необходимо определить **тип
аннотации** (*annotation type*). Синтаксис для этого таков:

```java
@interface ClassPreamble {
   String author();
   String date();
   int currentRevision() default 1;
   String lastModified() default "N/A";
   String lastModifiedBy() default "N/A";
   // Обратите внимание на использование массива
   String[] reviewers();
}
```

Определение типа аннотации похоже на определение интерфейса, в котором ключевому слову
`interface` предшествует символ «собака» (`@`) (@ = AT, как в *annotation type*). Типы аннотаций
являются разновидностью **интерфейса** (*interface*), о котором речь пойдёт в одном из следующих
уроков. Пока понимать интерфейсы не обязательно.

Тело приведённого выше определения аннотации содержит объявления **элементов типа аннотации**
(*annotation type element*), которые очень похожи на методы. Обратите внимание, что они могут
задавать необязательные значения по умолчанию (*default values*).

После того как тип аннотации определён, можно использовать аннотации этого типа с заполненными
значениями, например так:

```java
@ClassPreamble (
   author = "John Doe",
   date = "3/17/2002",
   currentRevision = 6,
   lastModified = "4/12/2004",
   lastModifiedBy = "Jane Doe",
   // Обратите внимание на нотацию массива
   reviewers = {"Alice", "Bob", "Cindy"}
)
public class Generation3List extends Generation2List {

// здесь идёт код класса

}
```

**Примечание.** Чтобы информация из `@ClassPreamble` попадала в документацию, генерируемую
Javadoc, нужно пометить определение `@ClassPreamble` аннотацией `@Documented`:

```java
// импортируем этот пакет, чтобы использовать `@Documented`
import java.lang.annotation.*;

@Documented
@interface ClassPreamble {

   // определения элементов аннотации

}
```

## Предопределённые типы аннотаций

В Java SE API предопределён набор типов аннотаций. Часть типов аннотаций используется
компилятором Java, а часть применяется к другим аннотациям.

### Типы аннотаций, используемые языком Java

Предопределённые типы аннотаций, определённые в `java.lang`, — это `@Deprecated`, `@Override`
и `@SuppressWarnings`.

#### @Deprecated

Аннотация `@Deprecated` указывает, что помеченный элемент **устарел** (*deprecated*) и его
больше не следует использовать. Компилятор генерирует предупреждение всякий раз, когда программа
использует метод, класс или поле с аннотацией `@Deprecated`. Когда элемент объявлен устаревшим,
его также следует задокументировать с помощью тега Javadoc `@deprecated`, как показано в
следующем примере. Использование символа «собака» (`@`) и в комментариях Javadoc, и в аннотациях
не случайно: они концептуально связаны. Обратите также внимание, что тег Javadoc начинается со
строчной буквы _d_, а аннотация — с прописной буквы _D_.

```java
// далее следует комментарий Javadoc
/**
 * @deprecated
 * объяснение, почему элемент признан устаревшим
 */
@Deprecated
static void deprecatedMethod() { }
```

#### @Override

Аннотация `@Override` сообщает компилятору, что элемент предназначен для переопределения
(*override*) элемента, объявленного в суперклассе. Переопределение методов будет обсуждаться
в уроке «Интерфейсы и наследование» (*Interfaces and Inheritance*).

```java
// помечаем метод как метод суперкласса,
// который был переопределён
@Override
int overriddenMethod() { }
```

Хотя использовать эту аннотацию при переопределении метода не обязательно, она помогает
предотвращать ошибки. Если метод, помеченный `@Override`, в действительности не переопределяет
корректно метод одного из своих суперклассов, компилятор генерирует ошибку.

#### @SuppressWarnings

Аннотация `@SuppressWarnings` сообщает компилятору, что следует подавить определённые
предупреждения, которые он иначе сгенерировал бы. В следующем примере используется устаревший
метод, и компилятор обычно сгенерировал бы предупреждение. Однако в данном случае аннотация
приводит к тому, что предупреждение подавляется.

```java
// используем устаревший метод и говорим
// компилятору не генерировать предупреждение
@SuppressWarnings("deprecation")
void useDeprecatedMethod() {
    // предупреждение об устаревании
    // — подавлено
    objectOne.deprecatedMethod();
}
```

Каждое предупреждение компилятора относится к некоторой категории. Спецификация языка Java
(*Java Language Specification*) перечисляет две категории: `deprecation` и `unchecked`.
Предупреждение `unchecked` может возникать при взаимодействии с устаревшим кодом, написанным до
появления обобщений (*generics*). Чтобы подавить сразу несколько категорий предупреждений,
используйте такой синтаксис:

```java
@SuppressWarnings({"unchecked", "deprecation"})
```

#### @SafeVarargs

Аннотация `@SafeVarargs`, применённая к методу или конструктору, утверждает, что код не
выполняет потенциально небезопасных операций над своим параметром-переменным числом аргументов
(*varargs*). При использовании этого типа аннотации предупреждения категории `unchecked`,
связанные с использованием `varargs`, подавляются.

#### @FunctionalInterface

Аннотация `@FunctionalInterface`, введённая в Java SE 8, указывает, что объявление типа
предназначено быть функциональным интерфейсом (*functional interface*) в том смысле, в каком он
определён спецификацией языка Java.

### Аннотации, применяемые к другим аннотациям

Аннотации, применяемые к другим аннотациям, называются **мета-аннотациями**
(*meta-annotations*). В `java.lang.annotation` определено несколько типов мета-аннотаций.

#### @Retention

Аннотация `@Retention` указывает, как хранится помеченная аннотация:

- `RetentionPolicy.SOURCE` — помеченная аннотация сохраняется только на уровне исходного кода
  и игнорируется компилятором.
- `RetentionPolicy.CLASS` — помеченная аннотация сохраняется компилятором во время компиляции,
  но игнорируется виртуальной машиной Java (JVM).
- `RetentionPolicy.RUNTIME` — помеченная аннотация сохраняется JVM, так что её можно
  использовать в среде выполнения.

#### @Documented

Аннотация `@Documented` указывает, что всякий раз, когда используется указанная аннотация, эти
элементы должны документироваться с помощью инструмента Javadoc. (По умолчанию аннотации в
Javadoc не включаются.)

#### @Target

Аннотация `@Target` помечает другую аннотацию, чтобы ограничить, к каким видам элементов Java
эту аннотацию можно применять. Аннотация-цель указывает в качестве своего значения один из
следующих типов элементов:

- `ElementType.ANNOTATION_TYPE` — может применяться к типу аннотации.
- `ElementType.CONSTRUCTOR` — может применяться к конструктору.
- `ElementType.FIELD` — может применяться к полю или свойству.
- `ElementType.LOCAL_VARIABLE` — может применяться к локальной переменной.
- `ElementType.METHOD` — может применяться к аннотации уровня метода.
- `ElementType.PACKAGE` — может применяться к объявлению пакета.
- `ElementType.PARAMETER` — может применяться к параметрам метода.
- `ElementType.TYPE` — может применяться к любому элементу класса.

#### @Inherited

Аннотация `@Inherited` указывает, что тип аннотации может наследоваться от суперкласса. (По
умолчанию это не так.) Когда пользователь запрашивает тип аннотации, а у класса нет аннотации
этого типа, то на наличие данного типа аннотации опрашивается суперкласс этого класса. Эта
аннотация применяется только к объявлениям классов.

#### @Repeatable

Аннотация `@Repeatable`, введённая в Java SE 8, указывает, что помеченную аннотацию можно
применять более одного раза к одному и тому же объявлению или использованию типа.

## Аннотации типов и подключаемые системы типов

До выпуска Java SE 8 аннотации можно было применять только к объявлениям. Начиная с выпуска
Java SE 8, аннотации можно применять также к любому **использованию типа** (*type use*). Это
означает, что аннотации можно использовать везде, где используется тип. Несколько примеров мест,
где используются типы, — это выражения создания экземпляра класса (`new`), приведения типов
(*casts*), конструкции `implements` и конструкции `throws`. Эта форма аннотации называется
**аннотацией типа** (*type annotation*).

Аннотации типов были созданы для поддержки улучшенного анализа Java-программ путём обеспечения
более строгой проверки типов. Выпуск Java SE 8 не предоставляет фреймворка проверки типов, но
позволяет вам написать (или скачать) такой фреймворк, реализованный в виде одного или нескольких
подключаемых модулей (*pluggable modules*), которые используются совместно с компилятором Java.

Например, вы хотите гарантировать, что некоторой переменной в вашей программе никогда не
присваивается `null`; вы хотите избежать возникновения `NullPointerException`. Вы можете написать
собственный плагин для проверки этого. После этого вы измените свой код, аннотировав ту самую
переменную, указав, что ей никогда не присваивается `null`. Объявление переменной могло бы
выглядеть так:

```java
@NonNull String str;
```

Когда вы компилируете код, включив модуль `NonNull` в командной строке, компилятор выводит
предупреждение, если обнаруживает потенциальную проблему, позволяя вам изменить код, чтобы
избежать ошибки. После того как вы исправите код так, чтобы все предупреждения исчезли, эта
конкретная ошибка не возникнет при выполнении программы.

Можно использовать несколько модулей проверки типов, где каждый модуль проверяет свой вид ошибок.
Таким образом вы можете надстраивать систему типов Java, добавляя конкретные проверки тогда и
там, где вам это нужно.

При разумном использовании аннотаций типов и наличии подключаемых средств проверки типов вы
можете писать код, который более надёжен и менее подвержен ошибкам.

Во многих случаях вам не нужно писать собственные модули проверки типов. Есть сторонние
разработчики, которые сделали эту работу за вас. Например, вы можете воспользоваться **Checker
Framework**, созданным Университетом Вашингтона (*University of Washington*). Этот фреймворк
включает модуль `NonNull`, а также модуль регулярных выражений и модуль мьютекс-блокировок
(*mutex lock module*).

## Повторяющиеся аннотации

Бывают ситуации, когда требуется применить одну и ту же аннотацию к объявлению или использованию
типа. Начиная с выпуска Java SE 8, **повторяющиеся аннотации** (*repeating annotations*)
позволяют сделать это.

Например, вы пишете код, использующий службу таймера, которая позволяет запускать метод в
заданное время или по расписанию, наподобие службы UNIX `cron`. Теперь вы хотите задать таймер
для запуска метода `doPeriodicCleanup` в последний день месяца и каждую пятницу в 23:00.
Чтобы таймер сработал, создайте аннотацию `@Schedule` и примените её к методу
`doPeriodicCleanup` дважды. Первое применение указывает последний день месяца, второе — пятницу
в 23:00, как показано в следующем примере кода:

```java
@Schedule(dayOfMonth="last")
@Schedule(dayOfWeek="Fri", hour="23")
public void doPeriodicCleanup() { ... }
```

В предыдущем примере аннотация применяется к методу. Повторяющуюся аннотацию можно применить
везде, где применяется стандартная аннотация. Например, у вас есть класс для обработки
несанкционированного доступа. Вы аннотируете класс аннотацией `@Alert` для менеджеров и
аннотацией `@Alert` для администраторов:

```java
@Alert(role="Manager")
@Alert(role="Administrator")
public class UnauthorizedAccessException extends SecurityException { ... }
```

Из соображений совместимости повторяющиеся аннотации хранятся в **аннотации-контейнере**
(*container annotation*), которая автоматически создаётся компилятором Java. Чтобы компилятор мог
это сделать, в вашем коде требуются два объявления.

### Шаг 1. Объявите повторяемый тип аннотации

Тип аннотации должен быть помечен мета-аннотацией `@Repeatable`. В следующем примере объявляется
пользовательский повторяемый тип аннотации `@Schedule`:

```java
import java.lang.annotation.Repeatable;

@Repeatable(Schedules.class)
public @interface Schedule {
  String dayOfMonth() default "first";
  String dayOfWeek() default "Mon";
  int hour() default 12;
}
```

Значение мета-аннотации `@Repeatable` в скобках — это тип аннотации-контейнера, который компилятор
Java генерирует для хранения повторяющихся аннотаций. В данном примере тип-контейнер —
`Schedules`, поэтому повторяющиеся аннотации `@Schedule` хранятся в аннотации `@Schedules`.

Применение к объявлению той же аннотации без объявления её как повторяемой приводит к ошибке
времени компиляции.

### Шаг 2. Объявите тип аннотации-контейнера

Тип аннотации-контейнера должен иметь элемент `value` с типом «массив». Тип компонента массива
должен быть повторяемым типом аннотации. Объявление типа аннотации-контейнера `Schedules`
выглядит так:

```java
public @interface Schedules {
    Schedule[] value();
}
```

### Извлечение аннотаций

В Reflection API есть несколько методов, которые можно использовать для извлечения аннотаций.
Поведение методов, возвращающих одну аннотацию, таких как
`AnnotatedElement.getAnnotation(Class<T>)`, не изменилось: они возвращают лишь одну аннотацию,
если присутствует аннотация запрошенного типа. Если присутствует более одной аннотации
запрошенного типа, то получить одну из них можно, сначала получив их аннотацию-контейнер.
Так старый код продолжает работать. В Java SE 8 введены другие методы, которые сканируют
аннотацию-контейнер и возвращают сразу несколько аннотаций, например
`AnnotatedElement.getAnnotationsByType(Class<T>)`. Полный список всех доступных методов см. в
документации класса `AnnotatedElement`.
