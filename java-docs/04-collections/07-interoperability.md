В этом разделе вы познакомитесь с двумя аспектами совместимости (*interoperability*):

- **Совместимость (Compatibility)** — как заставить коллекции работать со старыми API,
  которые появились до того, как в платформу Java были добавлены коллекции (`Collection`).
- **Проектирование API (API Design)** — как проектировать новые API так, чтобы они без
  проблем взаимодействовали друг с другом.

## Совместимость (Compatibility)

> Каркас коллекций Java (*Java Collections Framework*) был спроектирован так, чтобы обеспечить
> полную совместимость между основными интерфейсами коллекций и теми типами, которые
> использовались для представления коллекций в ранних версиях платформы Java: `Vector`,
> `Hashtable`, массив (*array*) и `Enumeration`. В этом разделе вы узнаете, как преобразовывать
> старые коллекции в коллекции каркаса коллекций Java и наоборот.

### Прямая совместимость (Upward Compatibility)

Предположим, что вы используете один API, который возвращает устаревшие (*legacy*) коллекции,
вместе с другим API, который требует объекты, реализующие интерфейсы коллекций. Чтобы эти два
API корректно взаимодействовали, вам нужно преобразовать устаревшие коллекции в современные.
К счастью, каркас коллекций Java делает это просто.

Предположим, старый API возвращает массив объектов, а новый API требует `Collection`. В каркасе
коллекций есть удобная реализация, которая позволяет рассматривать массив объектов как `List`.
Метод `Arrays.asList` используется, чтобы передать массив в любой метод, требующий `Collection`
или `List`.

```java
Foo[] result = oldMethod(arg);
newMethod(Arrays.asList(result));
```

Если старый API возвращает `Vector` или `Hashtable`, то делать вообще ничего не нужно, потому
что `Vector` был дополнен (*retrofitted*) реализацией интерфейса `List`, а `Hashtable` —
реализацией `Map`. Следовательно, `Vector` можно напрямую передать в любой метод, требующий
`Collection` или `List`.

```java
Vector result = oldMethod(arg);
newMethod(result);
```

Аналогично, `Hashtable` можно напрямую передать в любой метод, требующий `Map`.

```java
Hashtable result = oldMethod(arg);
newMethod(result);
```

Реже бывает, что API возвращает `Enumeration`, представляющий коллекцию объектов. Метод
`Collections.list` преобразует `Enumeration` в `Collection`.

```java
Enumeration e = oldMethod(arg);
newMethod(Collections.list(e));
```

### Обратная совместимость (Backward Compatibility)

Предположим, что вы используете один API, который возвращает современные коллекции, вместе с
другим API, который требует передавать устаревшие коллекции. Чтобы эти два API корректно
взаимодействовали, вам нужно преобразовать современные коллекции в старые. И снова каркас
коллекций Java делает это просто.

Предположим, новый API возвращает `Collection`, а старый API требует массив `Object`. Как вы,
вероятно, знаете, интерфейс `Collection` содержит метод `toArray`, предназначенный именно для
этой ситуации.

```java
Collection c = newMethod();
oldMethod(c.toArray());
```

Что если старый API требует массив `String` (или другого типа) вместо массива `Object`? Тогда
вы просто используете другую форму `toArray` — ту, что принимает массив на вход.

```java
Collection c = newMethod();
oldMethod((String[]) c.toArray(new String[0]));
```

Если старый API требует `Vector`, пригодится стандартный конструктор коллекции.

```java
Collection c = newMethod();
oldMethod(new Vector(c));
```

Случай, когда старый API требует `Hashtable`, обрабатывается аналогично.

```java
Map m = newMethod();
oldMethod(new Hashtable(m));
```

Наконец, что делать, если старый API требует `Enumeration`? Этот случай не распространён, но
время от времени встречается, и для него предусмотрен метод `Collections.enumeration`. Это
статический фабричный метод, который принимает `Collection` и возвращает `Enumeration` по
элементам этой `Collection`.

```java
Collection c = newMethod();
oldMethod(Collections.enumeration(c));
```

## Проектирование API (API Design)

> Руководство The Java Tutorials написано для JDK 8. Примеры и приёмы, описанные на этой
> странице, не используют улучшений, появившихся в более поздних выпусках, и могут опираться
> на технологии, которых уже нет.

В этом коротком, но важном разделе вы познакомитесь с несколькими простыми рекомендациями,
которые позволят вашему API без проблем взаимодействовать со всеми остальными API, следующими
этим рекомендациям. По сути, эти правила определяют, что значит быть хорошим «гражданином» в
мире коллекций.

### Параметры (Parameters)

Если ваш API содержит метод, который требует коллекцию на входе, крайне важно объявлять тип
соответствующего параметра как один из типов-интерфейсов коллекций. **Никогда** не используйте
тип реализации, потому что это сводит на нет саму идею каркаса коллекций, основанного на
интерфейсах, — а она в том, чтобы манипулировать коллекциями без оглядки на детали реализации.

Кроме того, всегда следует использовать наименее специфичный тип из тех, что имеют смысл.
Например, не требуйте `List` или `Set`, если достаточно `Collection`. Дело не в том, что
требовать `List` или `Set` на входе нельзя никогда: это правильно, если метод зависит от
свойства одного из этих интерфейсов. Например, многие алгоритмы, предоставляемые платформой
Java, требуют `List` на входе, потому что опираются на тот факт, что списки упорядочены. Однако
как общее правило: лучшие типы для использования на входе — самые общие: `Collection` и `Map`.

**Внимание:** Никогда не определяйте собственный самодельный (*ad hoc*) класс коллекции и не
требуйте объекты этого класса на входе. Поступив так, вы лишитесь всех преимуществ, которые даёт
каркас коллекций Java.

### Возвращаемые значения (Return Values)

С возвращаемыми значениями вы можете позволить себе гораздо больше гибкости, чем с входными
параметрами. Можно возвращать объект любого типа, который реализует или расширяет один из
интерфейсов коллекций. Это может быть один из интерфейсов или специальный тип, который расширяет
или реализует один из этих интерфейсов.

Например, можно представить пакет обработки изображений под названием `ImageList`, который
возвращает объекты нового класса, реализующего `List`. Помимо операций `List`, `ImageList` мог
бы поддерживать любые специфичные для приложения операции, которые показались бы желательными.
Например, он мог бы предоставлять операцию `indexImage`, которая возвращает изображение,
содержащее уменьшенные копии (*thumbnail*) каждого графического элемента из `ImageList`. Важно
отметить: даже если API выдаёт на выходе экземпляры `ImageList`, на входе он должен принимать
произвольные экземпляры `Collection` (или, возможно, `List`).

В каком-то смысле возвращаемые значения должны вести себя противоположно входным параметрам:
лучше возвращать наиболее специфичный применимый интерфейс коллекции, а не самый общий.
Например, если вы уверены, что всегда будете возвращать `SortedMap`, то соответствующему методу
следует задать возвращаемый тип `SortedMap`, а не `Map`. Экземпляры `SortedMap` строятся
дольше, чем обычные экземпляры `Map`, и при этом мощнее. Раз ваш модуль уже потратил время на
построение `SortedMap`, имеет смысл дать пользователю доступ к его дополнительным возможностям.
Более того, пользователь сможет передать возвращённый объект как в методы, требующие `SortedMap`,
так и в те, что принимают любой `Map`.

### Устаревшие API (Legacy APIs)

В настоящее время существует множество API, которые определяют собственные самодельные
(*ad hoc*) типы коллекций. Хотя это и прискорбно, это факт жизни, учитывая, что в первых двух
крупных выпусках платформы Java каркаса коллекций не было. Предположим, что вы владеете одним из
таких API; вот что вы можете с этим сделать.

Если возможно, доработайте свой устаревший тип коллекции так, чтобы он реализовывал один из
стандартных интерфейсов коллекций. Тогда все возвращаемые вами коллекции будут без проблем
взаимодействовать с другими API, основанными на коллекциях. Если это невозможно (например,
потому что одна или несколько уже существующих сигнатур типов конфликтуют со стандартными
интерфейсами коллекций), определите *класс-адаптер* (*adapter class*), который оборачивает один
из ваших объектов устаревших коллекций, позволяя ему работать как стандартная коллекция. (Класс
`Adapter` — пример пользовательской реализации (*custom implementation*).)

Доработайте свой API новыми вызовами, которые следуют рекомендациям для входных параметров и
принимают объекты стандартного интерфейса коллекций, если это возможно. Такие вызовы могут
сосуществовать с вызовами, принимающими устаревший тип коллекции. Если это невозможно,
предоставьте конструктор или статическую фабрику для вашего устаревшего типа, которая принимает
объект одного из стандартных интерфейсов и возвращает устаревшую коллекцию, содержащую те же
элементы (или отображения). Любой из этих подходов позволит пользователям передавать в ваш API
произвольные коллекции.
