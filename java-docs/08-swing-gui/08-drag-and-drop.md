> типичных проблем.

Перетаскивание (*drag and drop*, DnD), а также вырезание, копирование и вставка (вместе их
называют **передачей данных**, *data transfer*) — это важнейшие возможности большинства
приложений. Но какую поддержку для них предоставляет Swing и как ею воспользоваться?

Для многих компонентов при выполнении перетаскивания или вырезания и вставки Swing берёт всю
работу на себя. Для небольшого числа компонентов бо́льшая часть работы тоже выполняется за вас,
и вам остаётся лишь добавить детали импорта и экспорта данных.

Этот урок знакомит с механизмом передачи данных в Swing и подробно рассматривает, в частности,
класс `TransferHandler` — главный «рабочий инструмент» системы передачи данных.

> Если вы хотите создавать графический интерфейс на JavaFX, см.
> [Drag-and-Drop Feature in JavaFX Applications](https://docs.oracle.com/javase/8/javafx/events-tutorial/drag_drop_feature.htm).

## Введение в DnD

Если вы пишете приложение, вам наверняка захочется поддержать возможность передавать
информацию между компонентами внутри вашего приложения. Но вы также захотите, чтобы ваше
приложение хорошо «уживалось» с другими — а это значит поддерживать передачу информации между
вашим приложением и другими Java-приложениями, а также между вашим приложением и нативными
(*native*) приложениями. Возможность передавать данные принимает две формы:

- **Поддержка перетаскивания (drag and drop, DnD).** Например, перетаскивание из списка
  `JList` и сброс на текстовое поле `JTextField` (данные движутся от источника к цели).
- **Передача через буфер обмена с помощью вырезания или копирования и вставки.** Например,
  вырезание (или копирование) из `JList` и вставка в `JTextField`.

### Перетаскивание изнутри

Допустим, есть пользователь по имени Ролло (Rollo), запустивший Java-приложение. Он хочет
перетащить некоторый текст из списка и поместить его в текстовое поле. (Обратите внимание, что
процесс точно такой же при перетаскивании из нативного приложения в Java-приложение.) В двух
словах процесс перетаскивания и сброса работает так:

- Ролло выделил строку текста в **компоненте-источнике** (*source*) — в списке. Удерживая
  кнопку мыши, Ролло начинает перетаскивать текст — это инициирует **жест перетаскивания**
  (*drag gesture*).

- Когда перетаскивание начинается, список упаковывает данные для **экспорта** (*export*) и
  объявляет, какие **действия источника** (*source actions*) он поддерживает: `COPY`
  (копирование), `MOVE` (перемещение) или `LINK` (создание ссылки).

- Пока Ролло перетаскивает данные, Swing непрерывно вычисляет местоположение и обрабатывает
  отрисовку.

- Если во время перетаскивания Ролло одновременно удерживает клавишу Shift и/или Control, это
  **действие пользователя** (*user action*) тоже становится частью жеста перетаскивания. Обычно
  обычное перетаскивание запрашивает действие `MOVE`. Удержание Control при перетаскивании
  запрашивает действие `COPY`, а удержание Shift и Control одновременно запрашивает действие
  `LINK`.

- Как только Ролло перетаскивает текст за границы компонента текстового поля, **цель**
  (*target*) непрерывно опрашивается на предмет того, примет она или отвергнет потенциальный
  сброс. По мере перетаскивания цель даёт обратную связь, показывая **место сброса** (*drop
  location*) — например, курсор вставки или подсвеченное выделение. В данном случае текстовое
  поле (текущая цель) допускает как замену выделенного текста, так и вставку нового текста.

- Когда Ролло отпускает кнопку мыши, текстовый компонент анализирует объявленные действия
  источника и любое действие пользователя, после чего выбирает то, что ему нужно, из доступных
  вариантов. В данном случае текстовое поле решает вставить новый текст в точке сброса.

- Наконец, текстовое поле **импортирует** (*import*) данные.

Хотя процесс может показаться пугающим, бо́льшую часть работы Swing берёт на себя. Каркас
спроектирован так, что вы подключаете специфичные для вашего компонента детали, а остальное
«просто работает».

> **Примечание.** Мы **не рекомендуем** создавать собственную поддержку перетаскивания на
> основе классов AWT. Такая реализация потребовала бы значительной сложной поддержки внутри
> каждого компонента. До выпуска 1.4, когда система DnD была переработана, разработчики иногда
> создавали свою поддержку DnD, но она не работает со сложными компонентами вроде дерева и
> таблицы, у которых есть тонкие нюансы выделения и сброса.

## Поддержка DnD по умолчанию

Технически каркас перетаскивания поддерживает все компоненты Swing — механизм передачи данных
встроен в каждый `JComponent`. При желании вы могли бы реализовать поддержку сброса для
`JSlider`, чтобы он полноценно участвовал в передаче данных. Хотя `JSlider` по умолчанию сброс
не поддерживает, те компоненты, для которых вы хотели бы (и ожидали бы) поддержки
перетаскивания, действительно предоставляют специализированную встроенную поддержку.

Следующие компоненты распознают жест перетаскивания после того, как у компонента вызван метод
`setDragEnabled(true)`. Например, после вызова `myColorChooser.setDragEnabled(true)` вы можете
перетаскивать цвета из вашего выбора цвета:

- `JColorChooser`
- `JEditorPane`
- `JFileChooser`
- `JFormattedTextField`
- `JList`
- `JTable`
- `JTextArea`
- `JTextField`
- `JTextPane`
- `JTree`

Следующие компоненты поддерживают сброс «из коробки». Если вы используете один из этих
компонентов, ваша работа уже сделана:

- `JEditorPane`
- `JFormattedTextField`
- `JPasswordField`
- `JTextArea`
- `JTextField`
- `JTextPane`
- `JColorChooser`

Для следующих компонентов каркас сброса уже на месте, но вам нужно добавить небольшой объём
кода, чтобы настроить поддержку под свои нужды:

- `JList`
- `JTable`
- `JTree`

Для этих важнейших компонентов Swing выполняет вычисление места сброса и отрисовку; позволяет
задать режим сброса (*drop mode*); обрабатывает специфичные для компонента детали, такие как
раскрытие узлов дерева. Вашей работы остаётся совсем немного.

> **Примечание.** Поддержку сброса можно также установить на контейнеры верхнего уровня
> (*top-level containers*), такие как `JFrame` и `JDialog`. Подробнее об этом — в разделе
> «Сброс на контейнер верхнего уровня».

## Класс TransferHandler

В сердце механизма передачи данных лежит класс `TransferHandler`. Как видно из названия,
`TransferHandler` предоставляет простой механизм передачи данных в `JComponent` и из него — все
детали содержатся в этом классе и его вспомогательных классах. Большинству компонентов
предоставляется обработчик передачи по умолчанию. Вы можете создать и установить собственный
обработчик передачи на любой компонент.

Для подключения `TransferHandler` к компоненту используются три метода:

- `setDragEnabled(boolean)` — включает поддержку перетаскивания. (По умолчанию — `false`.)
  Этот метод определён для каждого компонента, поддерживающего жест перетаскивания.
- `setDropMode(DropMode)` — настраивает, как определяются места сброса. Этот метод определён
  для `JList`, `JTable` и `JTree`.
- `setTransferHandler(TransferHandler)` — используется для подключения собственного импорта и
  экспорта данных. Этот метод определён в `JComponent`, поэтому наследуется каждым компонентом
  Swing.

Как уже отмечалось, обработчики передачи Swing по умолчанию — например, используемые текстовыми
компонентами и выбором цвета — предоставляют поддержку, которую считают наиболее полезной как
для импорта, так и для экспорта данных. Однако список, таблица и дерево по умолчанию сброс не
поддерживают. Причина в том, что нет универсального способа обработать сброс на эти компоненты.
Например, что значит сбросить на конкретный узел `JTree`? Заменить узел, вставить под ним или
вставить как дочерний элемент этого узла? К тому же нам неизвестно, какая модель стоит за
деревом — она может быть неизменяемой.

Хотя Swing не может предоставить реализацию по умолчанию для этих компонентов, каркас сброса
там есть. Вам нужно лишь предоставить собственный `TransferHandler`, который управляет
непосредственным импортом данных.

> **Примечание.** Если вы устанавливаете собственный `TransferHandler` на компонент Swing,
> поддержка по умолчанию заменяется. Например, если вы замените `TransferHandler` у
> `JTextField` на такой, что обрабатывает только цвета, вы отключите его способность
> поддерживать импорт и экспорт текста.
>
> Если вам всё же нужно заменить `TransferHandler` по умолчанию — например, обрабатывающий
> текст, — вам придётся заново реализовать возможность импорта и экспорта текста. Это не обязано
> быть таким же обширным, как то, что предоставляет Swing — может быть достаточно поддержки
> «вкуса данных» (*data flavor*) `stringFlavor`, в зависимости от потребностей вашего
> приложения.

## Методы экспорта

Первый набор методов, который мы рассмотрим, используется для экспорта данных из компонента.
Эти методы вызываются для жеста перетаскивания или для действия вырезания/копирования, когда
рассматриваемый компонент является источником операции. Методы `TransferHandler` для экспорта
данных таковы:

- **`getSourceActions(JComponent)`** — этот метод используется, чтобы узнать, какие действия
  поддерживаются компонентом-источником, например `COPY`, `MOVE` или `LINK` в любой комбинации.
  Например, список клиентов может не поддерживать перемещение имени клиента из списка, но,
  скорее всего, поддерживает копирование имени клиента. Большинство наших примеров поддерживают
  и `COPY`, и `MOVE`.

- **`createTransferable(JComponent)`** — этот метод упаковывает экспортируемые данные в объект
  `Transferable` для подготовки к передаче.

- **`exportDone(JComponent, Transferable, int)`** — этот метод вызывается после завершения
  экспорта. Когда действие — `MOVE`, данные нужно удалить из источника после завершения
  передачи; именно в этом методе выполняется необходимая очистка.

Вот примеры реализаций методов экспорта:

```java
int getSourceActions(JComponent c) {
    return COPY_OR_MOVE;
}

Transferable createTransferable(JComponent c) {
    return new StringSelection(c.getSelection());
}

void exportDone(JComponent c, Transferable t, int action) {
    if (action == MOVE) {
        c.removeSelection();
    }
}
```

## Методы импорта

Теперь рассмотрим методы, используемые для импорта данных в компонент. Эти методы вызываются
для жеста сброса или для действия вставки, когда компонент является целью операции. Методы
`TransferHandler` для импорта данных таковы:

- **`canImport(TransferHandler.TransferSupport)`** — этот метод вызывается многократно во время
  жеста перетаскивания и возвращает `true`, если область под курсором может принять передачу,
  или `false`, если передача будет отклонена. Например, если пользователь перетаскивает цвет
  над компонентом, который принимает только текст, метод `canImport` обработчика передачи этого
  компонента должен вернуть `false`.

- **`importData(TransferHandler.TransferSupport)`** — этот метод вызывается при успешном сбросе
  (или вставке) и инициирует передачу данных в целевой компонент. Метод возвращает `true`, если
  импорт прошёл успешно, и `false` в противном случае.

> **Примечание о версии.** Эти методы заменяют более старые версии, не использующие класс
> `TransferSupport`. В отличие от своего метода-заменителя, `canImport(JComponent,
> DataFlavor[])` не вызывается непрерывно.

Как видите, эти методы импорта принимают аргумент `TransferHandler.TransferSupport`. Далее
рассмотрим класс `TransferSupport`, а затем — примеры методов импорта.

## Класс TransferSupport

Класс `TransferSupport` выполняет две функции. Как следует из названия, его первая функция —
поддерживать процесс передачи; для этого он предоставляет несколько вспомогательных методов,
дающих доступ к деталям передачи данных. Следующий список показывает методы, которые можно
использовать для получения информации из `TransferHandler`. Некоторые из этих методов связаны с
действиями сброса (*drop actions*), которые обсуждаются в разделе «Установка режима сброса».

- `Component getComponent()` — возвращает целевой компонент передачи.
- `int getDropAction()` — возвращает выбранное действие (`COPY`, `MOVE` или `LINK`), когда
  передача является сбросом. Если передача не является сбросом, метод выбрасывает исключение.
- `int getUserDropAction()` — возвращает действие сброса, выбранное пользователем. Например,
  если пользователь одновременно удерживает Control и Shift во время жеста перетаскивания, это
  указывает на действие `ACTION_LINK`. Подробнее о действиях сброса пользователя см. API
  `DropTargetDragEvent`. Если передача не является сбросом, метод выбрасывает исключение.
- `int getSourceDropActions()` — возвращает набор действий, поддерживаемых компонентом-
  источником. Если передача не является сбросом, метод выбрасывает исключение.
- `DataFlavor[] getDataFlavors()` — возвращает все «вкусы данных» (*data flavors*),
  поддерживаемые этим компонентом. Например, компонент может поддерживать файлы и текст или
  текст и цвет. Если передача не является сбросом, метод выбрасывает исключение.
- `boolean isDataFlavorSupported(DataFlavor)` — возвращает `true`, если указанный `DataFlavor`
  поддерживается. `DataFlavor` указывает тип представляемых данных, например изображение
  (`imageFlavor`), строку (`stringFlavor`), список файлов (`javaFileListFlavor`) и так далее.
- `Transferable getTransferable()` — возвращает данные `Transferable` для этой передачи.
  Эффективнее использовать один из приведённых выше методов для запроса информации о передаче,
  чем извлекать `Transferable` и опрашивать его, поэтому этот метод не рекомендуется, кроме
  случаев, когда нельзя получить информацию иначе.
- `DropLocation getDropLocation()` — возвращает место сброса в компоненте. Компоненты со
  встроенной поддержкой сброса — список, таблица и дерево — переопределяют этот метод, чтобы
  возвращать более полезные данные. Например, версия этого метода для компонента `JList`
  возвращает индекс в списке, куда произошёл сброс. Если передача не является сбросом, метод
  выбрасывает исключение.

Теперь, когда вы знакомы с вспомогательными методами `TransferSupport`, рассмотрим примеры
методов `canImport` и `importData`:

```java
public boolean canImport(TransferSupport supp) {
    // Проверяем поддержку «вкуса» String
    if (!supp.isDataFlavorSupported(stringFlavor)) {
        return false;
    }

    // Получаем место сброса
    DropLocation loc = supp.getDropLocation();

    // Возвращаем, принимаем ли мы это место
    return shouldAcceptDropLocation(loc);
}

public boolean importData(TransferSupport supp) {
    if (!canImport(sup)) {
        return false;
    }

    // Получаем Transferable и его данные
    Transferable t = supp.getTransferable();
    String data = t.getTransferData(stringFlavor);

    // Получаем место сброса
    DropLocation loc = supp.getDropLocation();

    // Вставляем данные в это место
    insertAt(loc, data);

    return true;
}
```

## Установка режима сброса (drop mode)

Включая сброс на компонент — например на список, — вам нужно решить, как интерпретировать место
сброса. Например, хотите ли вы ограничить пользователя заменой существующих записей? Хотите ли
вы разрешать только добавление или вставку новых записей? Хотите ли вы разрешить и то, и
другое? Чтобы настроить это поведение, класс `JList` предоставляет метод `setDropMode`,
поддерживающий следующие режимы сброса:

- Режим сброса по умолчанию для `JList` — `DropMode.USE_SELECTION`. При перетаскивании в этом
  режиме выделенный элемент списка перемещается, отражая потенциальную точку сброса. При сбросе
  выделенный элемент перемещается в место сброса. Этот режим предоставлен для обратной
  совместимости, но в остальном не рекомендуется.

- В режиме `DropMode.ON` выделенный элемент списка перемещается, отражая потенциальную точку
  сброса, но при сбросе на сам выделенный элемент это не влияет. Этот режим можно использовать
  для сброса поверх существующих элементов списка.

- В режиме `DropMode.INSERT` пользователь ограничен выбором пространства между существующими
  элементами списка, либо перед первым элементом, либо после последнего. Выбор существующих
  элементов списка не допускается.

- `DropMode.ON_OR_INSERT` — это комбинация режимов `ON` и `INSERT`.

Класс `JTree` предоставляет тот же набор режимов сброса, а у класса `JTable` есть ещё несколько
специфичных для добавления строк или столбцов.

Чтобы получить место сброса, класс `TransferSupport` предоставляет метод `getDropLocation`,
возвращающий точное место, куда произошёл сброс. Но для компонента-списка индекс сброса
полезнее, чем местоположение в пикселях, поэтому `JList` предоставляет специальный подкласс
`JList.DropLocation`. Этот класс предоставляет методы `getIndex` и `isInsert`, которые берут
вычисления на себя.

Компоненты таблицы, дерева и текста каждый предоставляют свою реализацию `DropLocation` с
методами, наиболее осмысленными для каждого компонента. У метода `JTable.setDropMode` больше
всего вариантов. Следующая таблица показывает методы для всех четырёх классов:

| `JList.DropLocation` | `JTree.DropLocation` | `JTable.DropLocation` | `JTextComponent.DropLocation` |
|---|---|---|---|
| `isInsert` | `getChildIndex` | `isInsertRow` | `getIndex` |
| `getIndex` | `getPath` | `isInsertColumn` | `getBias` |
| | | `getRow` | |
| | | `getColumn` | |

Демонстрационный пример `DropDemo` реализует собственный обработчик передачи для компонента-
списка, так что список полноценно участвует в перетаскивании. В этом примере у списка задан
режим сброса `DropMode.INSERT`, и пользователь может вставлять новые элементы между
существующими, перед первым или после последнего.

## Выбор действия сброса (drop action)

Каждый источник перетаскивания (на Java или нет) при экспорте данных объявляет набор действий,
которые он поддерживает. Если он поддерживает копирование данных, он объявляет действие `COPY`;
если поддерживает перемещение данных, он объявляет действие `MOVE` и так далее. Для компонентов
Swing действия источника объявляются через метод `getSourceActions`.

Когда перетаскивание начинается, пользователь имеет некоторый контроль над тем, какое из
действий источника выбирается для передачи, с помощью клавиш-модификаторов, используемых вместе
с жестом перетаскивания — это называется **действием пользователя** (*user action*). Например,
по умолчанию (без модификаторов) обычно подразумевается перемещение, удержание Control
указывает на копирование, а удержание Shift и Control — на создание ссылки. Действие
пользователя доступно через метод `getUserDropAction`.

Действие пользователя указывает предпочтение, но в конечном счёте именно цель решает, каким
будет действие сброса. Например, рассмотрим компонент, который принимает только скопированные
данные, и источник перетаскивания, поддерживающий и копирование, и перемещение. `TransferHandler`
цели, принимающей только копирование, можно запрограммировать так, чтобы он принимал данные от
источника с помощью метода `setDropAction` только как копирование, даже если пользователь указал
предпочтение перемещения.

Эта работа происходит в методе `canImport`, где `TransferHandler` цели решает, принять ли
поступающие данные. Реализация, которая явно выбирает действие `COPY`, если оно поддерживается
источником, может выглядеть так:

```java
public boolean canImport(TransferHandler.TransferSupport support) {
    // для демонстрации поддерживаем только сброс (не вставку из буфера обмена)
    if (!support.isDrop()) {
        return false;
    }

    // импортируем только строки (String)
    if (!support.isDataFlavorSupported(DataFlavor.stringFlavor)) {
        return false;
    }

    // проверяем, содержит ли набор действий источника
    // (побитовое ИЛИ поддерживаемых действий) действие COPY
    boolean copySupported = (COPY & support.getSourceDropActions()) == COPY;
    if (copySupported) {
        support.setDropAction(COPY);
        return true;
    }

    // COPY не поддерживается, поэтому отклоняем передачу
    return false;
}
```

Здесь запрашиваются поддерживаемые источником действия сброса. Если копирование поддерживается,
вызывается метод `setDropAction`, чтобы гарантировать, что произойдёт только копирование, и
метод возвращает `true`. Демонстрационный пример `ChooseDropAction` явно задаёт действие сброса
с помощью `setDropAction`.

## Отображение места сброса (drop location)

Обычно во время операции перетаскивания компонент даёт визуальную обратную связь, когда может
принять данные. Он может подсвечивать место сброса или показывать каретку либо горизонтальную
линию там, где произойдёт вставка. Swing отрисовывает место сброса, когда метод `canImport`
обработчика передачи компонента возвращает `true`.

Чтобы управлять этим программно, можно использовать метод `setShowDropLocation`. Вызов этого
метода со значением `true` приводит к тому, что визуальная обратная связь для места сброса
всегда отображается, даже если сброс не будет принят. Вызов со значением `false` подавляет любую
визуальную обратную связь, даже если сброс будет принят. Этот метод всегда вызывается из
`canImport`.

Демонстрационный пример `LocationSensitiveDemo` содержит выпадающий список, позволяющий выбрать:
всегда показывать место сброса, никогда не показывать или поведение по умолчанию.

## Чувствительный к месту сброс (location sensitive drop)

Иногда у вас есть сложный компонент, и вы хотите, чтобы пользователь мог сбрасывать на одни его
части, но не на другие. Возможно, это таблица, разрешающая сброс данных только в определённые
столбцы; или дерево, разрешающее сброс данных только на определённые узлы. Очевидно, вы хотите,
чтобы курсор давал точную обратную связь — он должен показывать место сброса только тогда, когда
находится над той частью компонента, которая принимает сброс.

Этого легко добиться, поместив необходимую логику в метод
`canImport(TransferHandler.TransferSupport)` класса `TransferHandler`. Это работает только с
этой конкретной версией `canImport`, потому что она вызывается непрерывно, пока жест
перетаскивания находится в границах компонента. Когда этот метод возвращает `true`, Swing
показывает курсор сброса, и место сброса визуально обозначается; когда метод возвращает `false`,
Swing показывает курсор «нельзя перетаскивать» (*no-drag*), и место сброса не отображается.

Например, представьте таблицу, которая разрешает сброс, но не в первый столбец. Метод
`canImport` может выглядеть примерно так:

```java
public boolean canImport(TransferHandler.TransferSupport info) {
    // для демонстрации поддерживаем только сброс (не вставку из буфера обмена)
    if (!info.isDrop()) {
        return false;
    }

    // импортируем только строки (String)
    if (!info.isDataFlavorSupported(DataFlavor.stringFlavor)) {
        return false;
    }

    // получаем место сброса
    JTable.DropLocation dl = (JTable.DropLocation)info.getDropLocation();

    int column = dl.getColumn();

    // не поддерживаем недопустимые столбцы и первый столбец
    if (column == -1 || column == 0) {
        return false;
    }

    return true;
}
```

Когда пользователь сбрасывает данные так, что столбец нельзя вычислить (и он, следовательно,
недопустим), или когда пользователь сбрасывает текст в первый столбец, метод `canImport`
возвращает `false` — поэтому Swing показывает курсор «нельзя перетаскивать». Как только
пользователь смещает мышь с первого столбца, `canImport` возвращает `true`, и Swing показывает
курсор перетаскивания. Демонстрационный пример `LocationSensitiveDemo` показывает дерево, в
котором реализован чувствительный к месту сброс.

## Сброс на пустую таблицу

Перетаскивание и сброс на пустую таблицу представляют особую трудность. При соблюдении всех
правильных шагов:

- создание пустой таблицы;
- создание и подключение `TransferHandler`;
- включение передачи данных вызовом `setDragEnabled(true)`;
- создание панели прокрутки и добавление таблицы в эту панель —

вы запускаете приложение, пытаетесь перетащить корректные данные в таблицу, но она отклоняет
сброс. В чём дело?

Причина в том, что пустая таблица (в отличие от пустого списка или пустого дерева) не занимает
места в панели прокрутки. `JTable` не растягивается автоматически, чтобы заполнить высоту
области просмотра (*viewport*) панели `JScrollPane` — она занимает по вертикали лишь столько
места, сколько нужно для содержащихся в ней строк. Поэтому, когда вы перетаскиваете над пустой
таблицей, вы фактически находитесь не над таблицей, и сброс не удаётся.

Вы можете настроить таблицу так, чтобы разрешить сброс в любом месте области просмотра, вызвав
`JTable.setFillsViewportHeight(boolean)`. По умолчанию это свойство равно `false` для
обеспечения обратной совместимости.

Демонстрационный пример `FillViewportHeightDemo` позволяет поэкспериментировать со сбросом на
пустую таблицу. Пример содержит пустую таблицу с пятью столбцами, у которой режим сброса
установлен на вставку строк, и источник перетаскивания, предоставляющий пять разделённых
запятыми значений, которые автоматически увеличиваются.

Главное, что нужно запомнить: как правило, следует вызывать `setFillsViewportHeight(true)` на
любой таблице, которая будет принимать сброшенные данные.

## Отрисовка места сброса

Это более продвинутая тема, и большинству людей о ней беспокоиться не нужно. Однако, если у вас
есть собственный компонент, вам придётся обрабатывать отрисовку места сброса самостоятельно.

Вы можете подписаться на уведомления о любом изменении свойства `dropLocation`. Вы слушаете это
изменение и выполняете свою отрисовку места сброса в собственном отрисовщике (*renderer*)
компонента или в методе `paintComponent`, используя метод `getDropLocation`.

Вот пример прослушивания свойства `dropLocation`:

```java
class Repainter extends PropertyChangeListener {
    public void propertyChange(PropertyChangeEvent pce) {
        repaintDropLocation(pce.getOldValue());
        repaintDropLocation(pce.getNewValue());
    }
}

comp.addPropertyChangeListener("dropLocation", newRepainter());
```

Вот пример подхода через `paintComponent`:

```java
public void paintComponent(Graphics g) {
    super.paintComponent(g);

    DropLocation loc = getDropLocation();
    if (loc == null) {
        return;
    }

    renderPrettyIndicatorAt(loc);
}
```

## Сброс на контейнер верхнего уровня (top-level drop)

До сих пор мы в основном фокусировались на подключении `TransferHandler` к одному из подклассов
`JComponent`. Но вы можете также установить `TransferHandler` непосредственно на контейнер
верхнего уровня, такой как `JFrame` и `JDialog`.

Это особенно полезно для приложений, импортирующих файлы: редакторов, сред разработки (IDE),
программ обработки изображений, программ записи компакт-дисков. Такое приложение обычно включает
меню, панель инструментов, область для редактирования документов и, вероятно, список или
механизм переключения между открытыми документами.

Демонстрационный пример `TopLevelTransferHandlerDemo` имеет меню, (нефункциональную) панель
инструментов, область (слева), отображающую список открытых документов, и область (справа),
отображающую содержимое каждого открытого документа. При старте синей области документа назначен
обработчик передачи, поддерживающий импорт файлов — поэтому это единственное место, которое
может принять сброс.

**Попробуйте:**

1. Скомпилируйте и запустите пример `TopLevelTransferHandlerDemo`.

2. Перетащите файл с рабочего стола или из файловой системы и сбросьте его на синюю область
   документа справа. Файл открывается, и появляется фрейм с его содержимым. Область документа
   — `JDesktopPane` — содержит обработчик передачи, поддерживающий импорт `javaFileListFlavor`.

3. Перетащите другой файл и попробуйте сбросить его на область документа. Вы обнаружите, что не
   можете сбросить его поверх фрейма, отображающего предыдущий файл. Вы также не можете сбросить
   его на список, меню или панель инструментов. Единственное место, куда можно сбросить — синяя
   часть области документа или строка меню ранее открытого фрейма. Внутри каждого фрейма
   содержимого есть обработчик передачи текстового компонента, который не понимает сброс файла —
   туда можно сбросить текст, но не файл.

4. В меню выберите Demo->Use Top-Level TransferHandler, чтобы установить обработчик передачи на
   контейнер верхнего уровня — `JFrame`.

5. Снова попробуйте перетаскивать над демонстрацией. Число областей, принимающих сброс,
   увеличилось. Теперь можно сбросить почти где угодно в приложении, включая строку меню, панель
   инструментов, заголовок фрейма, кроме списка (слева) и области содержимого ранее открытого
   файла. Ни обработчик передачи `JList`, ни обработчик текстовой области не умеют импортировать
   файлы.

6. Отключите обработчики передачи на оставшихся компонентах, выбрав в меню Demo->Remove
   TransferHandler from List and Text.

7. Снова перетащите над демонстрацией. Теперь вы можете сбросить файл *куда угодно* в
   приложении!

8. В меню выберите Demo->Use COPY Action.

9. Снова перетащите над демонстрацией. Обратите внимание, что курсор мыши теперь показывает
   курсор COPY — это даёт более точную обратную связь, потому что успешный сброс не удаляет файл
   из источника. Цель можно запрограммировать так, чтобы она выбирала из доступных действий
   сброса, как описано в разделе «Выбор действия сброса».

> **Примечание.** Один нежелательный побочный эффект отключения обработчика передачи по
> умолчанию на текстовом компоненте: вы больше не можете перетаскивать (или
> вырезать/копировать/вставлять) текст внутри области редактирования. Чтобы это исправить, вам
> нужно реализовать собственный обработчик передачи для текстового компонента, который принимает
> сброс файлов и при этом заново реализует отсутствующую поддержку передачи текста.

## Добавление вырезания, копирования и вставки (CCP)

До сих пор наше обсуждение в основном крутилось вокруг поддержки перетаскивания. Однако подключить
вырезание, копирование или вставку (*cut, copy, paste*, CCP) к обработчику передачи несложно. Для
этого нужны следующие шаги:

- убедиться, что на компоненте установлен обработчик передачи;
- создать способ, которым можно вызвать поддержку CCP у `TransferHandler`. Обычно это означает
  добавление привязок (*bindings*) в карты ввода и действий (*input and action maps*), чтобы
  действия CCP у `TransferHandler` вызывались в ответ на конкретные нажатия клавиш;
- создать пункты меню и/или кнопки CCP. (Этот шаг необязателен, но рекомендуется.) С текстовыми
  компонентами это делается легко, но для других компонентов требуется чуть больше работы,
  поскольку нужна логика, определяющая, на каком компоненте запустить действие. Подробнее см.
  «CCP в нетекстовом компоненте»;
- решить, где вы хотите выполнять вставку. Возможно, над или под текущим выделением. Поместите
  эту логику в метод `importData`.

### CCP в текстовом компоненте

Если вы реализуете вырезание, копирование и вставку с использованием одного из текстовых
компонентов Swing (текстовое поле, поле пароля, форматированное текстовое поле или текстовая
область), ваша работа очень проста. Эти текстовые компоненты используют `DefaultEditorKit`,
который предоставляет встроенные действия для вырезания, копирования и вставки. Редакторский
набор по умолчанию также берёт на себя задачу запоминать, какой компонент последним имел фокус.
Это значит, что если пользователь инициирует одно из этих действий через меню или клавиатурный
эквивалент, действие получает правильный компонент — дополнительный код не требуется.

Демонстрационный пример `TextCutPaste` содержит три текстовых поля. Вы можете вырезать,
копировать и вставлять в любое из этих текстовых полей и из любого. Они также поддерживают
перетаскивание.

Вот код, создающий меню Edit путём подключения встроенных действий вырезания, копирования и
вставки, определённых в `DefaultEditorKit`, к пунктам меню. Это работает с любым компонентом,
происходящим от `JComponent`:

```java
/**
 * Создаём меню Edit для поддержки вырезания/копирования/вставки.
 */
public JMenuBar createMenuBar () {
    JMenuItem menuItem = null;
    JMenuBar menuBar = new JMenuBar();
    JMenu mainMenu = new JMenu("Edit");
    mainMenu.setMnemonic(KeyEvent.VK_E);

    menuItem = new JMenuItem(new DefaultEditorKit.CutAction());
    menuItem.setText("Cut");
    menuItem.setMnemonic(KeyEvent.VK_T);
    mainMenu.add(menuItem);

    menuItem = new JMenuItem(new DefaultEditorKit.CopyAction());
    menuItem.setText("Copy");
    menuItem.setMnemonic(KeyEvent.VK_C);
    mainMenu.add(menuItem);

    menuItem = new JMenuItem(new DefaultEditorKit.PasteAction());
    menuItem.setText("Paste");
    menuItem.setMnemonic(KeyEvent.VK_P);
    mainMenu.add(menuItem);

    menuBar.add(mainMenu);
    return menuBar;
}
```

### CCP в нетекстовом компоненте

Если вы реализуете вырезание, копирование и вставку с использованием одного из компонентов
Swing, который *не* является текстовым, вам придётся выполнить дополнительную настройку.
Во-первых, нужно установить действия вырезания, копирования и вставки в карту действий (*action
map*). Следующий метод показывает, как это сделать:

```java
private void setMappings(JList list) {
    ActionMap map = list.getActionMap();
    map.put(TransferHandler.getCutAction().getValue(Action.NAME),
            TransferHandler.getCutAction());
    map.put(TransferHandler.getCopyAction().getValue(Action.NAME),
            TransferHandler.getCopyAction());
    map.put(TransferHandler.getPasteAction().getValue(Action.NAME),
            TransferHandler.getPasteAction());
```

Настраивая меню Edit, вы можете также добавить ускорители меню (*accelerators*), чтобы
пользователь мог, например, нажать Control-C для запуска копирования. В следующем фрагменте кода
показано, как задать ускоритель меню для действия вырезания:

```java
menuItem = new JMenuItem("Cut");
menuItem.setActionCommand((String)TransferHandler.getCutAction().
         getValue(Action.NAME));
menuItem.addActionListener(actionListener);
menuItem.setAccelerator(
  KeyStroke.getKeyStroke(KeyEvent.VK_X, ActionEvent.CTRL_MASK));
menuItem.setMnemonic(KeyEvent.VK_T);
mainMenu.add(menuItem);
```

Если вы задали ускорители меню для действий CCP, следующий шаг избыточен. Если вы их не задали,
нужно добавить привязки CCP в карту ввода (*input map*). Следующий фрагмент показывает, как это
делается:

```java
// требуется только если вы не задали ускорители меню
InputMap imap = this.getInputMap();
imap.put(KeyStroke.getKeyStroke("ctrl X"),
    TransferHandler.getCutAction().getValue(Action.NAME));
imap.put(KeyStroke.getKeyStroke("ctrl C"),
    TransferHandler.getCopyAction().getValue(Action.NAME));
imap.put(KeyStroke.getKeyStroke("ctrl V"),
    TransferHandler.getPasteAction().getValue(Action.NAME));
```

После установки привязок и настройки меню Edit нужно решить ещё один вопрос: когда пользователь
инициирует вырезание, копирование или вставку, какой компонент должен получить действие? В случае
текстового компонента `DefaultEditorKit` запоминает, какой компонент последним имел фокус, и
перенаправляет действие этому компоненту. Следующий класс `TransferActionListener` выполняет ту
же функцию для нетекстовых компонентов Swing. Этот класс можно встроить почти в любое приложение:

```java
public class TransferActionListener implements ActionListener,
                                              PropertyChangeListener {
    private JComponent focusOwner = null;

    public TransferActionListener() {
        KeyboardFocusManager manager = KeyboardFocusManager.
           getCurrentKeyboardFocusManager();
        manager.addPropertyChangeListener("permanentFocusOwner", this);
    }

    public void propertyChange(PropertyChangeEvent e) {
        Object o = e.getNewValue();
        if (o instanceof JComponent) {
            focusOwner = (JComponent)o;
        } else {
            focusOwner = null;
        }
    }

    public void actionPerformed(ActionEvent e) {
        if (focusOwner == null)
            return;
        String action = (String)e.getActionCommand();
        Action a = focusOwner.getActionMap().get(action);
        if (a != null) {
            a.actionPerformed(new ActionEvent(focusOwner,
                                              ActionEvent.ACTION_PERFORMED,
                                              null));
        }
    }
}
```

Наконец, вам нужно решить, как обрабатывать вставку. В случае перетаскивания вы вставляете
данные в место сброса. В случае вставки у вас нет возможности воспользоваться тем, что
пользователь указывает желаемое место вставки. Нужно решить, что имеет смысл для вашего
приложения — возможно, лучшим решением будет вставка данных перед текущим выделением или после
него.

Демонстрационный пример `ListCutPaste` показывает, как реализовать CCP в экземпляре `JList`. Там
три списка, и вы можете вырезать, копировать и вставлять в любой из них и из любого. Они также
поддерживают перетаскивание. В этом примере вставленные данные помещаются после текущего
выделения. Если текущего выделения нет, данные добавляются в конец списка.

## Использование и создание DataFlavor

Класс `DataFlavor` позволяет указать тип содержимого ваших данных. Вам нужно указать
`DataFlavor` при извлечении данных в методе `importData`. Несколько типов «вкусов» (*flavors*)
предопределены за вас:

- `imageFlavor` представляет данные в формате `java.awt.Image`. Используется при перетаскивании
  данных изображения.
- `stringFlavor` представляет данные в самой базовой форме текста — `java.lang.String`. Это
  наиболее часто используемый «вкус данных» для большинства приложений.
- `javaFileListFlavor` представляет объекты `java.io.File` в формате `java.util.List`. Полезно
  для приложений, перетаскивающих файлы, как пример `TopLevelTransferHandler`, обсуждавшийся в
  разделе «Сброс на контейнер верхнего уровня».

Для большинства приложений это всё, что вам нужно знать о «вкусах данных». Однако, если вам
требуется «вкус», отличный от этих предопределённых типов, вы можете создать свой. Если вы
создаёте собственный компонент и хотите, чтобы он участвовал в передаче данных, вам потребуется
создать собственный «вкус данных». Конструктор для указания «вкуса данных» —
`DataFlavor(Class, String)`. Например, чтобы создать «вкус данных» для класса
`java.util.ArrayList`:

```java
new DataFlavor(ArrayList.class, "ArrayList");
```

Чтобы создать «вкус данных» для массива целых чисел:

```java
new DataFlavor(int[].class, "Integer Array");
```

Передача данных с помощью этого механизма использует сериализацию объектов (*Object
serialization*), поэтому класс, используемый для передачи данных, должен реализовывать интерфейс
`Serializable`, как и всё, что сериализуется вместе с ним. Если не всё сериализуемо, вы увидите
`NotSerializableException` во время сброса или копирования в буфер обмена.

Создание «вкуса данных» с помощью конструктора `DataFlavor(Class, String)` позволяет передавать
данные между приложениями, включая нативные приложения. Если вы хотите создать «вкус данных»,
передающий данные только внутри одного приложения, используйте `javaJVMLocalObjectMimeType` и
конструктор `DataFlavor(String)`. Например, чтобы указать «вкус данных», передающий цвет из
`JColorChooser` только внутри вашего приложения, можно использовать такой код:

```java
String colorType = DataFlavor.javaJVMLocalObjectMimeType +
                   ";class=java.awt.Color";
DataFlavor colorFlavor = new DataFlavor(colorType);
```

Чтобы создать «вкус данных» для `ArrayList`, работающий только внутри вашего приложения:

```java
new DataFlavor(DataFlavor.javaJVMLocalObjectMimeType +
               ";class=java.util.ArrayList");
```

Чтобы создать «вкус данных» для массива целых чисел:

```java
new DataFlavor(DataFlavor.javaJVMLocalObjectMimeType +
               ";class=\"" + int[].class.getName() + "\"");
```

MIME-тип, содержащий специальные символы, такие как `[` или `;`, должен заключать эти символы в
кавычки.

`Transferable` можно реализовать так, чтобы он поддерживал несколько «вкусов». Например, можно
использовать вместе локальный и сериализуемый «вкусы», или две формы одних и тех же данных,
такие как «вкусы» `ArrayList` и массива целых чисел, или создать `TransferHandler`, принимающий
разные типы данных, например цвет и текст.

Когда вы создаёте массив `DataFlavor`, возвращаемый методом `getTransferDataFlavors` у
`Transferable`, «вкусы» следует вставлять в порядке предпочтения, с наиболее предпочтительным в
элементе с индексом 0. Обычно предпочтительный порядок — от самой богатой, или самой сложной,
формы данных к простому набору — форме, наиболее вероятно понимаемой другими объектами.

## Объединение всего вместе — DnD и CCP

Мы показали, как реализовать поддержку перетаскивания и как реализовать поддержку
вырезания/копирования/вставки. Как объединить и то, и другое в одном компоненте?

Вы реализуете оба в методе `importData` у `TransferHandler`, например так:

```java
if (transferSupport.isDrop()) {
    // помещаем данные в transferSupport.getDropLocation()
} else {
    // определяем, куда должна пойти вставка (например, после текущего выделения)
    // помещаем данные туда
}
```

Пример `ListCutPaste`, обсуждавшийся на странице «CCP в нетекстовом компоненте», поддерживает и
DnD, и CCP. Вот его метод `importData` (логика if-else для сброса выделена):

```java
public boolean importData(TransferHandler.TransferSupport info) {
    String data = null;

    // Если не можем обработать импорт — выходим сейчас.
    if (!canImport(info)) {
        return false;
    }

    JList list = (JList)info.getComponent();
    DefaultListModel model = (DefaultListModel)list.getModel();
    // Получаем данные — выходим, если не удалось
    try {
        data = (String)info.getTransferable().getTransferData(DataFlavor.stringFlavor);
    } catch (UnsupportedFlavorException ufe) {
        System.out.println("importData: unsupported data flavor");
        return false;
    } catch (IOException ioe) {
        System.out.println("importData: I/O exception");
        return false;
    }

    if (info.isDrop()) { // Это сброс
        JList.DropLocation dl = (JList.DropLocation)info.getDropLocation();
        int index = dl.getIndex();
        if (dl.isInsert()) {
            model.add(index, data);
            return true;
        } else {
            model.set(index, data);
            return true;
        }
    } else { // Это вставка
        int index = list.getSelectedIndex();
        // если есть корректное выделение,
        // вставляем данные после выделения
        if (index >= 0) {
            model.add(list.getSelectedIndex()+1, data);
        // иначе добавляем в конец списка
        } else {
            model.addElement(data);
        }
        return true;
    }
}
```

Это единственное место, где нужно устанавливать логику if-else для различения DnD и CCP.

## Решение типичных проблем передачи данных

В этом разделе обсуждаются проблемы, с которыми вы можете столкнуться при использовании передачи
данных.

**Проблема.** Мой распознаватель жеста перетаскивания (*drag gesture recognizer*) работает
неправильно с таблицей/списком/деревом/текстом.

Не используйте собственные распознаватели жеста перетаскивания с этими компонентами. Используйте
`setDragEnabled(true)` и `TransferHandler`.

**Проблема.** Я не могу сбросить данные на мою пустую таблицу `JTable`.

Нужно вызвать на таблице `setFillsViewportHeight(true)`. Подробнее см. раздел «Сброс на пустую
таблицу».
