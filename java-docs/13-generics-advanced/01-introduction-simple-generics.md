## Введение

В JDK 5.0 язык программирования Java получил несколько новых расширений. Одно из них —
появление **обобщений** (*generics*).

Этот трейл — введение в обобщения. Возможно, вы знакомы с похожими конструкциями из
других языков, прежде всего с шаблонами C++ (*C++ templates*). Если так, вы увидите как
сходства, так и важные различия. Если же подобные конструкции из других языков вам
незнакомы — тем лучше: можно начать с чистого листа и не переучиваться, избавляясь от
ложных представлений.

Обобщения позволяют **абстрагироваться над типами** (*abstract over types*). Самый
распространённый пример — контейнерные типы, такие как типы из иерархии коллекций
(*Collections hierarchy*).

### Зачем нужны обобщения

Рассмотрим типичный фрагмент кода:

```java
List myIntList = new LinkedList(); // 1
myIntList.add(new Integer(0)); // 2
Integer x = (Integer) myIntList.iterator().next(); // 3
```

Приведение типа (*cast*) в строке 3 слегка раздражает. Как правило, программист знает,
данные какого рода помещены в конкретный список. Однако приведение здесь необходимо.
Компилятор может гарантировать лишь то, что итератор вернёт `Object`. Чтобы присваивание
переменной типа `Integer` было типобезопасным (*type safe*), приведение требуется.

Разумеется, приведение не только добавляет «шум» в код. Оно ещё и создаёт возможность
ошибки во время выполнения (*run time error*), поскольку программист может ошибаться.

А что, если бы программисты могли выразить своё намерение и пометить список как
ограниченный определённым типом данных? Это и есть основная идея обобщений. Вот версия
приведённого выше фрагмента с использованием обобщений:

```java
List<Integer>
    myIntList = new LinkedList<Integer>(); // 1'
myIntList.add(new Integer(0)); // 2'
Integer x = myIntList.iterator().next(); // 3'
```

Обратите внимание на объявление типа переменной `myIntList`. Оно указывает, что это не
просто произвольный `List`, а `List` из `Integer` — записывается как `List<Integer>`. Мы
говорим, что `List` — это обобщённый интерфейс (*generic interface*), принимающий
**параметр типа** (*type parameter*), в данном случае — `Integer`. Параметр типа мы также
указываем при создании объекта списка.

Заметьте и то, что приведения типа в строке 3' больше нет.

Может показаться, что всё, чего мы добились, — это переместили «шум» с одного места на
другое. Вместо приведения к `Integer` в строке 3 у нас теперь `Integer` как параметр типа
в строке 1'. Однако разница здесь очень велика. Компилятор теперь может проверить
правильность типов программы во время компиляции (*at compile-time*). Когда мы говорим, что
`myIntList` объявлена с типом `List<Integer>`, это сообщает нам нечто о переменной
`myIntList`, что остаётся истинным всюду и всегда, где бы она ни использовалась, и компилятор
это гарантирует. Приведение же сообщает нам лишь то, что программист считает истинным в
одной-единственной точке кода.

Совокупный эффект — особенно в больших программах — это улучшение читаемости и надёжности
(*readability and robustness*).

## Определение простых обобщений

Вот небольшой фрагмент определений интерфейсов `List` и `Iterator` из пакета `java.util`:

```java
public interface List<E> {
    void add(E x);
    Iterator<E> iterator();
}

public interface Iterator<E> {
    E next();
    boolean hasNext();
}
```

Этот код должен быть полностью знаком, кроме того, что заключено в угловые скобки. Это —
объявления **формальных параметров типа** (*formal type parameters*) интерфейсов `List` и
`Iterator`.

Параметры типа могут использоваться по всему обобщённому объявлению практически везде, где вы
бы использовали обычные типы (хотя есть и некоторые важные ограничения; см. раздел «Тонкости»
(*The Fine Print*)).

### Параметризованные типы

Во введении мы видели **вызовы** (*invocations*) обобщённого объявления типа `List` — такие
как `List<Integer>`. В вызове (его обычно называют **параметризованным типом**, *parameterized
type*) все вхождения формального параметра типа (в данном случае `E`) заменяются **фактическим
аргументом типа** (*actual type argument*) — здесь это `Integer`.

Можно представить, что `List<Integer>` обозначает версию `List`, в которой `E` единообразно
заменён на `Integer`:

```java
public interface IntegerList {
    void add(Integer x);
    Iterator<Integer> iterator();
}
```

Эта интуиция может быть полезной, но она же и вводит в заблуждение.

Она полезна, потому что параметризованный тип `List<Integer>` действительно имеет методы,
выглядящие в точности как при таком «раскрытии».

Она вводит в заблуждение, потому что объявление обобщения на самом деле никогда так не
раскрывается. Нет нескольких копий кода — ни в исходниках, ни в двоичном виде, ни на диске,
ни в памяти. Если вы программируете на C++, вы поймёте, что это совсем не то же самое, что
шаблон C++ (*C++ template*). Обобщённое объявление типа компилируется раз и навсегда и
превращается в единственный class-файл — точно так же, как обычное объявление класса или
интерфейса.

### Аналогия с параметрами методов

Параметры типа аналогичны обычным параметрам, используемым в методах или конструкторах.
Подобно тому как метод имеет **формальные параметры-значения** (*formal value parameters*),
описывающие виды значений, с которыми он работает, обобщённое объявление имеет формальные
параметры типа. Когда метод вызывается, на место формальных параметров подставляются
**фактические аргументы** (*actual arguments*), и тело метода вычисляется. Когда вызывается
обобщённое объявление, фактические аргументы типа подставляются на место формальных параметров
типа.

### О соглашениях по именованию

Несколько слов о соглашениях по именованию. Мы рекомендуем использовать для формальных
параметров типа лаконичные (по возможности — однобуквенные), но при этом выразительные имена.
Лучше избегать в таких именах строчных букв, чтобы формальные параметры типа было легко
отличить от обычных классов и интерфейсов. Многие контейнерные типы используют `E` — от
*element* (элемент) — как в примерах выше. Некоторые дополнительные соглашения мы увидим в
последующих примерах.
