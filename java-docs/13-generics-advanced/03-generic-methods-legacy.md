## Обобщённые методы (Generic Methods)

> Рассмотрим написание метода, который принимает массив объектов и коллекцию и помещает все
> объекты из массива в коллекцию. Вот первая попытка:

```java
static void fromArrayToCollection(Object[] a, Collection<?> c) {
    for (Object o : a) {
        c.add(o); // ошибка компиляции
    }
}
```

> К этому моменту вы уже научились избегать ошибки новичка — попытки использовать в качестве типа
> параметра-коллекции `Collection<Object>`. Возможно, вы также уже поняли, что использование
> `Collection<?>` тоже не сработает. Вспомните, что нельзя просто так заталкивать объекты в
> коллекцию неизвестного типа.
>
> Способ справиться с этими проблемами — использовать **обобщённые методы** (*generic methods*).
> Точно так же, как объявления типов, объявления методов могут быть обобщёнными — то есть
> параметризованными одним или несколькими параметрами типа.

```java
static <T> void fromArrayToCollection(T[] a, Collection<T> c) {
    for (T o : a) {
        c.add(o); // Корректно
    }
}
```

> Мы можем вызвать этот метод с любой коллекцией, тип элементов которой является супертипом
> (supertype) типа элементов массива.

```java
Object[] oa = new Object[100];
Collection<Object> co = new ArrayList<Object>();

// T выводится как Object
fromArrayToCollection(oa, co);

String[] sa = new String[100];
Collection<String> cs = new ArrayList<String>();

// T выводится как String
fromArrayToCollection(sa, cs);

// T выводится как Object
fromArrayToCollection(sa, co);

Integer[] ia = new Integer[100];
Float[] fa = new Float[100];
Number[] na = new Number[100];
Collection<Number> cn = new ArrayList<Number>();

// T выводится как Number
fromArrayToCollection(ia, cn);

// T выводится как Number
fromArrayToCollection(fa, cn);

// T выводится как Number
fromArrayToCollection(na, cn);

// T выводится как Object
fromArrayToCollection(na, co);

// ошибка компиляции
fromArrayToCollection(na, cs);
```

> Обратите внимание, что нам не нужно передавать в обобщённый метод фактический аргумент типа
> (type argument). Компилятор сам выводит (infers) аргумент типа на основании типов фактических
> аргументов. Как правило, он выводит наиболее конкретный аргумент типа, при котором вызов
> остаётся типобезопасным.

### Когда обобщённые методы, а когда символы подстановки

> Возникает вопрос: когда следует использовать обобщённые методы, а когда — типы с символом
> подстановки (wildcard types)? Чтобы понять ответ, разберём несколько методов из библиотек
> `Collection`.

```java
interface Collection<E> {
    public boolean containsAll(Collection<?> c);
    public boolean addAll(Collection<? extends E> c);
}
```

> Здесь мы могли бы вместо этого использовать обобщённые методы:

```java
interface Collection<E> {
    public <T> boolean containsAll(Collection<T> c);
    public <T extends E> boolean addAll(Collection<T> c);
    // Эй, у переменных типа тоже могут быть границы (bounds)!
}
```

> Однако и в `containsAll`, и в `addAll` параметр типа `T` используется лишь однажды. Тип
> возвращаемого значения не зависит от параметра типа, как и любой другой аргумент метода (в данном
> случае аргумент вообще один). Это говорит нам о том, что аргумент типа используется для
> полиморфизма (polymorphism); его единственный эффект — позволить использовать в разных точках
> вызова различные фактические типы аргументов. Если дело обстоит именно так, следует использовать
> символы подстановки. Символы подстановки спроектированы для поддержки гибкого
> субтипирования (subtyping) — а именно это мы здесь и пытаемся выразить.
>
> Обобщённые методы позволяют использовать параметры типа, чтобы выразить зависимости между типами
> одного или нескольких аргументов метода и/или типом возвращаемого значения. Если такой
> зависимости нет, обобщённый метод использовать не следует.

### Совместное использование обобщённых методов и символов подстановки

> Можно использовать обобщённые методы и символы подстановки в связке. Вот метод
> `Collections.copy()`:

```java
class Collections {
    public static <T> void copy(List<T> dest, List<? extends T> src) {
    ...
}
```

> Обратите внимание на зависимость между типами двух параметров. Любой объект, копируемый из
> исходного списка `src`, должен быть присваиваемым (assignable) к типу элементов `T` целевого
> списка `dst`. Поэтому тип элементов `src` может быть любым подтипом `T` — каким именно, нам
> неважно. Сигнатура `copy` выражает эту зависимость с помощью параметра типа, но использует символ
> подстановки для типа элементов второго параметра.
>
> Мы могли бы записать сигнатуру этого метода иначе, вообще без символов подстановки:

```java
class Collections {
    public static <T, S extends T> void copy(List<T> dest, List<S> src) {
    ...
}
```

> Это допустимо, но, в то время как первый параметр типа используется и в типе `dst`, и в границе
> второго параметра типа `S`, сам `S` используется лишь однажды — в типе `src`, и больше ничто от
> него не зависит. Это признак того, что `S` можно заменить символом подстановки. Использование
> символов подстановки яснее и лаконичнее, чем объявление явных параметров типа, и поэтому ему
> следует отдавать предпочтение всегда, когда это возможно.
>
> У символов подстановки есть и ещё одно преимущество: их можно использовать вне сигнатур
> методов — в качестве типов полей, локальных переменных и массивов. Вот пример.
>
> Возвращаясь к нашей задаче рисования фигур, предположим, что мы хотим хранить историю запросов на
> отрисовку. Мы можем вести историю в статическом поле внутри класса `Shape`, а метод `drawAll()`
> пусть сохраняет приходящий ему аргумент в это поле истории.

```java
static List<List<? extends Shape>>
    history = new ArrayList<List<? extends Shape>>();

public void drawAll(List<? extends Shape> shapes) {
    history.addLast(shapes);
    for (Shape s: shapes) {
        s.draw(this);
    }
}
```

### Соглашение об именовании параметров типа

> Наконец, ещё раз отметим соглашение об именовании, используемое для параметров типа. Мы
> используем `T` для типа (type), когда нет ничего более конкретного, что отличало бы этот тип. Так
> часто бывает в обобщённых методах. Если параметров типа несколько, можно использовать буквы,
> соседствующие с `T` в алфавите, например `S`. Если обобщённый метод находится внутри обобщённого
> класса, желательно избегать одинаковых имён для параметров типа метода и класса — во избежание
> путаницы. То же относится и к вложенным обобщённым классам.

## Взаимодействие со старым кодом (Interoperating with Legacy Code)

> До сих пор все наши примеры предполагали идеализированный мир, где все используют новейшую версию
> языка программирования Java, поддерживающую обобщения (generics).
>
> Увы, в реальности это не так. Миллионы строк кода написаны на более ранних версиях языка, и они
> не будут переведены все разом за одну ночь.
>
> Позже, в разделе «Преобразование старого кода для использования обобщений» (Converting Legacy
> Code to Use Generics), мы возьмёмся за задачу преобразования вашего старого кода для использования
> обобщений. В этом разделе мы сосредоточимся на более простой задаче: как старый
> код (legacy code) и обобщённый код могут взаимодействовать? У этого вопроса две части:
> использование старого кода внутри обобщённого кода и использование обобщённого кода внутри старого
> кода.

### Использование старого кода в обобщённом коде

> Как можно использовать старый код, по-прежнему пользуясь преимуществами обобщений в своём
> собственном коде?
>
> В качестве примера предположим, что вы хотите использовать пакет `com.Example.widgets`. Сотрудники
> Example.com продают систему управления складскими запасами (inventory control), основные элементы
> которой показаны ниже:

```java
package com.Example.widgets;

public interface Part {...}

public class Inventory {
    /**
     * Добавляет новую сборку (Assembly) в базу данных склада.
     * Сборке присваивается имя name, и она
     * состоит из набора деталей, заданного parts.
     * Все элементы коллекции parts
     * должны поддерживать интерфейс Part.
     **/
    public static void addAssembly(String name, Collection parts) {...}
    public static Assembly getAssembly(String name) {...}
}

public interface Assembly {
    // Возвращает коллекцию деталей (Part)
    Collection getParts();
}
```

> Теперь вы хотели бы добавить новый код, использующий приведённый выше API. Было бы хорошо
> гарантировать, что вы всегда вызываете `addAssembly()` с правильными аргументами — то есть что
> передаваемая вами коллекция действительно является `Collection` из `Part`. Конечно, обобщения как
> будто специально для этого созданы:

```java
package com.mycompany.inventory;

import com.Example.widgets.*;

public class Blade implements Part {
    ...
}

public class Guillotine implements Part {
}

public class Main {
    public static void main(String[] args) {
        Collection<Part> c = new ArrayList<Part>();
        c.add(new Guillotine()) ;
        c.add(new Blade());
        Inventory.addAssembly("thingee", c);
        Collection<Part> k = Inventory.getAssembly("thingee").getParts();
    }
}
```

> Когда мы вызываем `addAssembly`, он ожидает, что второй параметр будет иметь тип `Collection`.
> Фактический аргумент имеет тип `Collection<Part>`. Это работает, но почему? Ведь большинство
> коллекций не содержат объектов `Part`, и поэтому в общем случае у компилятора нет способа узнать,
> на какую именно коллекцию ссылается тип `Collection`.
>
> В правильном обобщённом коде `Collection` всегда сопровождался бы параметром типа. Когда
> обобщённый тип вроде `Collection` используется без параметра типа, он называется **сырым
> типом** (*raw type*).
>
> Первая мысль большинства людей — что `Collection` на самом деле означает `Collection<Object>`.
> Однако, как мы видели ранее, небезопасно передавать `Collection<Part>` туда, где требуется
> `Collection<Object>`. Точнее будет сказать, что тип `Collection` обозначает коллекцию некоторого
> неизвестного типа — точно так же, как `Collection<?>`.
>
> Но погодите, это тоже не может быть верно! Рассмотрим вызов `getParts()`, который возвращает
> `Collection`. Результат затем присваивается `k`, имеющему тип `Collection<Part>`. Если бы
> результат вызова был `Collection<?>`, такое присваивание было бы ошибкой.
>
> На самом деле присваивание допустимо, но оно порождает **непроверяемое
> предупреждение** (*unchecked warning*). Предупреждение необходимо, потому что компилятор не может
> гарантировать его корректность. У нас нет способа проверить старый код в `getAssembly()`, чтобы
> убедиться, что возвращаемая коллекция действительно является коллекцией `Part`. В коде используется
> тип `Collection`, и в такую коллекцию вполне законно можно вставить объекты всех видов.
>
> Так не должно ли это быть ошибкой? Теоретически — да; но практически, если обобщённый код
> собирается вызывать старый код, это должно быть разрешено. Уже вы, программист, должны убедить
> себя в том, что в данном случае присваивание безопасно, потому что контракт `getAssembly()`
> гласит, что он возвращает коллекцию `Part`, пусть даже сигнатура типа этого не показывает.
>
> Итак, сырые типы очень похожи на типы с символом подстановки, но проверяются типами не так строго.
> Это намеренное проектное решение — позволить обобщениям взаимодействовать с уже существующим старым
> кодом.
>
> Вызов старого кода из обобщённого кода по своей природе опасен; как только вы смешиваете
> обобщённый код с необобщённым старым кодом, все гарантии безопасности, которые обычно
> предоставляет система типов обобщений, аннулируются. Тем не менее вы всё равно в лучшем положении,
> чем были бы вообще без использования обобщений. По крайней мере вы знаете, что код с вашей стороны
> согласован.
>
> На данный момент необобщённого кода вокруг гораздо больше, чем обобщённого, и неизбежно будут
> возникать ситуации, когда им придётся смешиваться.
>
> Если вы обнаружите, что вынуждены смешивать старый и обобщённый код, уделяйте пристальное внимание
> непроверяемым предупреждениям. Тщательно продумывайте, чем вы можете обосновать безопасность кода,
> порождающего предупреждение.
>
> Что произойдёт, если вы всё же допустили ошибку и код, вызвавший предупреждение, действительно не
> является типобезопасным? Давайте рассмотрим такую ситуацию. Попутно мы получим некоторое
> представление о работе компилятора.

### Стирание и трансляция (Erasure and Translation)

```java
public String loophole(Integer x) {
    List<String> ys = new LinkedList<String>();
    List xs = ys;
    xs.add(x); // непроверяемое предупреждение во время компиляции
    return ys.iterator().next();
}
```

> Здесь мы сделали алиас (aliased) для списка строк и обычного старого списка. Мы вставляем
> `Integer` в список и пытаемся извлечь `String`. Это явно неверно. Если мы проигнорируем
> предупреждение и попытаемся выполнить этот код, он сломается ровно в той точке, где мы пытаемся
> использовать неправильный тип. Во время выполнения этот код ведёт себя так:

```java
public String loophole(Integer x) {
    List ys = new LinkedList;
    List xs = ys;
    xs.add(x);
    return (String) ys.iterator().next(); // ошибка во время выполнения
}
```

> Когда мы извлекаем элемент из списка и пытаемся обращаться с ним как со строкой, приводя его к
> `String`, мы получаем `ClassCastException`. Ровно то же самое происходит и с обобщённой версией
> `loophole()`.
>
> Причина этого в том, что обобщения реализованы компилятором Java как внешнее (front-end)
> преобразование, называемое **стиранием** (*erasure*). Можно (почти) думать о нём как о
> трансляции из исходного кода в исходный код (source-to-source), при которой обобщённая версия
> `loophole()` преобразуется в необобщённую.
>
> Как следствие, **типобезопасность и целостность виртуальной машины Java никогда не оказываются
> под угрозой, даже при наличии непроверяемых предупреждений**.
>
> По сути, стирание избавляется (или *стирает*) всю информацию об обобщённых типах. Вся информация
> о типах между угловыми скобками выбрасывается, так что, например, параметризованный тип вроде
> `List<String>` преобразуется в `List`. Все оставшиеся использования переменных типа заменяются
> верхней границей (upper bound) переменной типа (обычно `Object`). И всякий раз, когда
> получающийся код оказывается некорректным по типам, вставляется приведение к подходящему типу —
> как в последней строке `loophole`.
>
> Полные подробности стирания выходят за рамки этого руководства, но только что данное простое
> описание недалеко от истины. Полезно знать об этом хотя бы немного — особенно если вы хотите
> делать более сложные вещи, например преобразовывать существующие API для использования
> обобщений (см. раздел «Преобразование старого кода для использования обобщений», Converting Legacy
> Code to Use Generics), или просто хотите понять, почему всё устроено именно так.

### Использование обобщённого кода в старом коде

> Теперь рассмотрим обратный случай. Представьте, что Example.com решили преобразовать свой API для
> использования обобщений, но некоторые их клиенты этого ещё не сделали. Тогда код выглядит так:

```java
package com.Example.widgets;

public interface Part {
    ...
}

public class Inventory {
    /**
     * Добавляет новую сборку (Assembly) в базу данных склада.
     * Сборке присваивается имя name, и она
     * состоит из набора деталей, заданного parts.
     * Все элементы коллекции parts
     * должны поддерживать интерфейс Part.
     **/
    public static void addAssembly(String name, Collection<Part> parts) {...}
    public static Assembly getAssembly(String name) {...}
}

public interface Assembly {
    // Возвращает коллекцию деталей (Part)
    Collection<Part> getParts();
}
```

> а клиентский код выглядит так:

```java
package com.mycompany.inventory;

import com.Example.widgets.*;

public class Blade implements Part {
...
}

public class Guillotine implements Part {
}

public class Main {
    public static void main(String[] args) {
        Collection c = new ArrayList();
        c.add(new Guillotine()) ;
        c.add(new Blade());

        // 1: непроверяемое предупреждение
        Inventory.addAssembly("thingee", c);

        Collection k = Inventory.getAssembly("thingee").getParts();
    }
}
```

> Клиентский код был написан до появления обобщений, но он использует пакет `com.Example.widgets` и
> библиотеку коллекций, в обоих из которых применяются обобщённые типы. Все использования объявлений
> обобщённых типов в клиентском коде — это сырые типы (raw types).
>
> Строка 1 порождает непроверяемое предупреждение, потому что сырая `Collection` передаётся туда,
> где ожидается `Collection` из `Part`, и компилятор не может гарантировать, что сырая `Collection`
> действительно является `Collection` из `Part`.
>
> В качестве альтернативы вы можете скомпилировать клиентский код с флагом `source 1.4`,
> гарантировав, что никакие предупреждения не будут сгенерированы. Однако в этом случае вы не
> сможете использовать ни одну из новых возможностей языка, появившихся в JDK 5.0.
