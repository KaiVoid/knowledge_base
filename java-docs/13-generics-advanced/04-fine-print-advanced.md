## Тонкости (The Fine Print)

### Обобщённый класс общий для всех своих инстанцирований

Что напечатает следующий фрагмент кода?

```java
List <String> l1 = new ArrayList<String>();
List<Integer> l2 = new ArrayList<Integer>();
System.out.println(l1.getClass() == l2.getClass());
```

Возможно, у вас возникнет соблазн ответить `false`, но это будет ошибкой. Код печатает `true`,
потому что все экземпляры обобщённого класса имеют один и тот же класс времени выполнения
(*run-time class*) независимо от их фактических параметров типа.

И действительно, обобщённым класс делает то, что он ведёт себя одинаково при любых допустимых
параметрах типа; один и тот же класс можно рассматривать как имеющий множество различных типов.

Как следствие, статические переменные и методы класса тоже общие для всех экземпляров. Именно
поэтому запрещено ссылаться на параметры типа объявления типа в статическом методе или
инициализаторе, а также в объявлении или инициализаторе статической переменной.

### Приведения типов и instanceof

Ещё одно следствие того, что обобщённый класс является общим для всех своих экземпляров, состоит
в том, что обычно бессмысленно спрашивать у экземпляра, является ли он экземпляром конкретного
инстанцирования обобщённого типа:

```java
Collection cs = new ArrayList<String>();
// Недопустимо.
if (cs instanceof Collection<String>) { ... }
```

аналогично, приведение типа вида

```java
// Непроверяемое предупреждение (unchecked warning).
Collection<String> cstr = (Collection<String>) cs;
```

даёт непроверяемое предупреждение (*unchecked warning*), поскольку это не та проверка, которую
система времени выполнения сделает за вас.

То же справедливо и для переменных типа (*type variables*):

```java
// Непроверяемое предупреждение.
<T> T badCast(T t, Object o) {
    return (T) o;
}
```

Переменные типа не существуют во время выполнения. Это значит, что они не влекут накладных
расходов ни по времени, ни по памяти, и это хорошо. К сожалению, это также означает, что вы не
можете надёжно использовать их в приведениях типов.

### Массивы

Тип компонентов массива не может быть переменной типа или параметризованным типом, если только
это не (неограниченный) wildcard-тип. Вы можете объявлять *типы* массивов, элементами которых
являются переменная типа или параметризованный тип, но не *объекты* массивов.

Это, конечно, неудобно. Но такое ограничение необходимо, чтобы избежать ситуаций вроде следующей:

```java
// На самом деле не разрешено.
List<String>[] lsa = new List<String>[10];
Object o = lsa;
Object[] oa = (Object[]) o;
List<Integer> li = new ArrayList<Integer>();
li.add(new Integer(3));
// Некорректно, но проходит проверку хранилища во время выполнения.
oa[1] = li;

// Ошибка времени выполнения: ClassCastException.
String s = lsa[1].get(0);
```

Если бы массивы параметризованного типа были разрешены, приведённый пример скомпилировался бы
без каких-либо непроверяемых предупреждений и тем не менее упал бы во время выполнения. Одной из
главных целей проектирования обобщённых типов была типобезопасность. В частности, язык спроектирован
так, чтобы гарантировать: **если всё ваше приложение скомпилировано без непроверяемых предупреждений
с помощью `javac -source 1.5`, оно типобезопасно**.

Тем не менее вы по-прежнему можете использовать массивы wildcard-типов. Следующий вариант предыдущего
кода отказывается и от объектов массивов, и от типов массивов с параметризованным типом элементов.
В результате нам приходится явно приводить тип, чтобы извлечь из массива `String`.

```java
// OK, массив неограниченного wildcard-типа.
List<?>[] lsa = new List<?>[10];
Object o = lsa;
Object[] oa = (Object[]) o;
List<Integer> li = new ArrayList<Integer>();
li.add(new Integer(3));
// Корректно.
oa[1] = li;
// Ошибка времени выполнения, но приведение явное.
String s = (String) lsa[1].get(0);
```

В следующем варианте, который вызывает ошибку компиляции, мы воздерживаемся от создания объекта
массива с параметризованным типом элементов, но всё же используем тип массива с параметризованным
типом элементов.

```java
// Ошибка.
List<String>[] lsa = new List<?>[10];
```

Аналогично, попытка создать объект массива, типом элементов которого является переменная типа,
вызывает ошибку компиляции:

```java
<T> T[] makeArray(T t) {
    return new T[100]; // Ошибка.
}
```

Поскольку переменные типа не существуют во время выполнения, нет способа определить, каким был бы
фактический тип массива.

Способ обойти подобные ограничения — использовать литералы классов в качестве токенов типа времени
выполнения, как описано в следующем разделе, «Литералы классов как токены типа времени выполнения».

## Литералы классов как токены типа времени выполнения (Class Literals as Runtime-Type Tokens)

Одно из изменений в JDK 5.0 — то, что класс `java.lang.Class` стал обобщённым. Это интересный пример
использования обобщённости для чего-то отличного от класса-контейнера.

Теперь, когда у `Class` есть параметр типа `T`, вы вполне можете спросить: что обозначает `T`? Он
обозначает тип, который представляет объект `Class`.

Например, тип `String.class` — это `Class<String>`, а тип `Serializable.class` — это
`Class<Serializable>`. Это можно использовать для повышения типобезопасности кода, работающего через
рефлексию (*reflection*).

В частности, поскольку метод `newInstance()` класса `Class` теперь возвращает `T`, вы можете получать
более точные типы при создании объектов через рефлексию.

Предположим, например, что вам нужно написать вспомогательный метод, выполняющий запрос к базе данных,
заданный в виде строки SQL, и возвращающий коллекцию объектов из базы данных, удовлетворяющих этому
запросу.

Один из способов — передавать объект-фабрику (*factory object*) явно, написав код вроде такого:

```java
interface Factory<T> { T make();}

public <T> Collection<T> select(Factory<T> factory, String statement) {
    Collection<T> result = new ArrayList<T>();

    /* Выполнить SQL-запрос через JDBC. */
    for (/* Перебрать результаты JDBC. */) {
        T item = factory.make();
        /* Через рефлексию заполнить все поля
         * item из результатов SQL.
         */
        result.add(item);
    }
    return result;
}
```

Вызвать его можно либо так

```java
select(new Factory<EmpInfo>(){
    public EmpInfo make() {
        return new EmpInfo();
    }}, "selection string");
```

либо объявить класс `EmpInfoFactory`, реализующий интерфейс `Factory`

```java
class EmpInfoFactory implements Factory<EmpInfo> {
    ...
    public EmpInfo make() {
        return new EmpInfo();
    }
}
```

и вызвать его

```java
select(getMyEmpInfoFactory(), "selection string");
```

Недостаток такого решения в том, что оно требует либо:

- использования многословных анонимных классов-фабрик в месте вызова, либо
- объявления класса-фабрики для каждого используемого типа и передачи экземпляра фабрики в месте
  вызова, что несколько неестественно.

Естественно использовать в качестве объекта-фабрики литерал класса (*class literal*), который затем
можно задействовать через рефлексию. Сегодня (без обобщённых типов) код можно было бы написать так:

```java
Collection emps = sqlUtility.select(EmpInfo.class, "select * from emps");
...
public static Collection select(Class c, String sqlStatement) {
    Collection result = new ArrayList();
    /* Выполнить SQL-запрос через JDBC. */
    for (/* Перебрать результаты JDBC. */ ) {
        Object item = c.newInstance();
        /* Через рефлексию заполнить все поля
         * item из результатов SQL.
         */
        result.add(item);
    }
    return result;
}
```

Однако это не дало бы нам коллекцию нужного точного типа. Теперь, когда `Class` обобщённый, мы можем
вместо этого написать следующее:

```java
Collection<EmpInfo>
    emps = sqlUtility.select(EmpInfo.class, "select * from emps");
...
public static <T> Collection<T> select(Class<T> c, String sqlStatement) {
    Collection<T> result = new ArrayList<T>();
    /* Выполнить SQL-запрос через JDBC. */
    for (/* Перебрать результаты JDBC. */ ) {
        T item = c.newInstance();
        /* Через рефлексию заполнить все поля
         * item из результатов SQL.
         */
        result.add(item);
    }
    return result;
}
```

Приведённый выше код даёт нам коллекцию точного типа типобезопасным образом.

Этот приём использования литералов классов в качестве токенов типа времени выполнения — очень полезная
вещь, которую стоит знать. Это идиома, широко применяемая, например, в новых API для работы с аннотациями.

## Ещё немного о wildcards (More Fun with Wildcards)

В этом разделе мы рассмотрим некоторые более продвинутые применения wildcards. Мы уже видели несколько
примеров, где ограниченные wildcards были полезны при чтении из структуры данных. Теперь рассмотрим
обратную ситуацию — структуру данных только для записи. Интерфейс `Sink` — простой пример такого рода.

```java
interface Sink<T> {
    flush(T t);
}
```

Можно представить себе его использование так, как показано в коде ниже. Метод `writeAll()` предназначен,
чтобы вытолкнуть (*flush*) все элементы коллекции `coll` в приёмник (*sink*) `snk` и вернуть последний
вытолкнутый элемент.

```java
public static <T> T writeAll(Collection<T> coll, Sink<T> snk) {
    T last;
    for (T t : coll) {
        last = t;
        snk.flush(last);
    }
    return last;
}
...
Sink<Object> s;
Collection<String> cs;
String str = writeAll(cs, s); // Недопустимый вызов.
```

В таком виде вызов `writeAll()` недопустим, так как нельзя вывести подходящий аргумент типа: ни
`String`, ни `Object` не подходят на роль `T`, потому что тип элемента `Collection` и тип элемента
`Sink` должны совпадать.

Мы можем исправить эту ошибку, изменив сигнатуру `writeAll()`, как показано ниже, с использованием
wildcard.

```java
public static <T> T writeAll(Collection<? extends T>, Sink<T>) {...}
...
// Вызов OK, но неверный тип возвращаемого значения.
String str = writeAll(cs, s);
```

Теперь вызов допустим, но присваивание ошибочно, поскольку выводимый тип возвращаемого значения —
`Object`, ведь `T` сопоставляется с типом элемента `s`, то есть с `Object`.

Решение — воспользоваться разновидностью ограниченного wildcard, которую мы ещё не встречали: wildcard
с *нижней границей* (*lower bound*). Синтаксис `? super T` обозначает неизвестный тип, являющийся
надтипом (*supertype*) `T` (или самим `T`; напомним, что отношение «быть надтипом» рефлексивно). Это
двойственный вариант ограниченных wildcards, которые мы использовали, где `? extends T` обозначает
неизвестный тип, являющийся подтипом (*subtype*) `T`.

```java
public static <T> T writeAll(Collection<T> coll, Sink<? super T> snk) {
    ...
}
String str = writeAll(cs, s); // Да!
```

При таком синтаксисе вызов допустим, а выводимый тип — `String`, как мы и хотели.

Теперь обратимся к более реалистичному примеру. `java.util.TreeSet<E>` представляет дерево
упорядоченных элементов типа `E`. Один из способов построить `TreeSet` — передать конструктору объект
`Comparator`. Этот компаратор будет использоваться для сортировки элементов `TreeSet` в желаемом порядке.

```java
TreeSet(Comparator<E> c)
```

Интерфейс `Comparator` по сути таков:

```java
interface Comparator<T> {
    int compare(T fst, T snd);
}
```

Предположим, мы хотим создать `TreeSet<String>` и передать подходящий компаратор. Нам нужно передать
`Comparator`, способный сравнивать строки `String`. Это можно сделать с помощью `Comparator<String>`,
но и `Comparator<Object>` подойдёт ничуть не хуже. Однако вызвать приведённый выше конструктор для
`Comparator<Object>` мы не сможем. Чтобы получить нужную гибкость, можно воспользоваться wildcard
с нижней границей:

```java
TreeSet(Comparator<? super E> c)
```

Такой код позволяет использовать любой применимый компаратор.

В качестве последнего примера использования wildcards с нижней границей рассмотрим метод
`Collections.max()`, возвращающий максимальный элемент в переданной ему коллекции. Чтобы `max()`
работал, все элементы передаваемой коллекции должны реализовывать `Comparable`. Более того, все они
должны быть сравнимы *друг с другом*.

Первая попытка обобщить сигнатуру этого метода даёт:

```java
public static <T extends Comparable<T>> T max(Collection<T> coll)
```

То есть метод принимает коллекцию некоторого типа `T`, сравнимого с самим собой, и возвращает элемент
этого типа. Однако такой код оказывается слишком ограничительным. Чтобы понять почему, рассмотрим тип,
сравнимый с произвольными объектами:

```java
class Foo implements Comparable<Object> {
    ...
}
Collection<Foo> cf = ... ;
Collections.max(cf); // Должно работать.
```

Каждый элемент `cf` сравним с любым другим элементом `cf`, поскольку каждый такой элемент — это `Foo`,
который сравним с любым объектом, и в частности с другим `Foo`. Однако при использовании приведённой
выше сигнатуры мы обнаруживаем, что вызов отвергается. Выводимый тип должен быть `Foo`, но `Foo` не
реализует `Comparable<Foo>`.

Не обязательно, чтобы `T` был сравним **в точности** с самим собой. Всё, что требуется, — чтобы `T`
был сравним с одним из своих надтипов. Это даёт нам:

```java
public static <T extends Comparable<? super T>>
        T max(Collection<T> coll)
```

Заметьте, что фактическая сигнатура `Collections.max()` сложнее. Мы вернёмся к ней в следующем разделе,
«Перевод старого кода на обобщённые типы». Это рассуждение применимо почти к любому использованию
`Comparable`, рассчитанному на работу с произвольными типами: всегда стоит использовать
`Comparable<? super T>`.

В общем случае, если у вас есть API, использующий параметр типа `T` только как аргумент, его
использования должны опираться на wildcards с нижней границей (`? super T`). И наоборот, если API
только возвращает `T`, вы дадите клиентам больше гибкости, используя wildcards с верхней границей
(`? extends T`).

### Захват wildcard (Wildcard Capture)

К этому моменту должно быть достаточно ясно, что при наличии

```java
Set<?> unknownSet = new HashSet<String>();
...
/* Добавить элемент t в множество s. */
public static <T> void addToSet(Set<T> s, T t) {
    ...
}
```

приведённый ниже вызов недопустим.

```java
addToSet(unknownSet, "abc"); // Недопустимо.
```

Не имеет значения, что фактически передаётся множество строк; важно, что выражение, передаваемое в
качестве аргумента, является множеством неизвестного типа, про которое нельзя гарантировать, что это
множество строк или вообще множество какого-либо конкретного типа.

Теперь рассмотрим следующий код:

```java
class Collections {
    ...
    <T> public static Set<T> unmodifiableSet(Set<T> set) {
        ...
    }
}
...
Set<?> s = Collections.unmodifiableSet(unknownSet); // Это работает! Почему?
```

Кажется, что это не должно быть разрешено; и всё же, если присмотреться к этому конкретному вызову,
разрешить его совершенно безопасно. В конце концов, `unmodifiableSet()` работает с любым видом `Set`
независимо от типа его элементов.

Поскольку такая ситуация возникает относительно часто, существует специальное правило, разрешающее
подобный код при очень конкретных обстоятельствах, в которых можно доказать его безопасность. Это
правило, известное как *захват wildcard* (*wildcard capture*), позволяет компилятору вывести
неизвестный тип wildcard в качестве аргумента типа для обобщённого метода.

## Перевод старого кода на обобщённые типы (Converting Legacy Code to Use Generics)

Ранее мы показали, как новый и старый код могут взаимодействовать. Теперь пора рассмотреть более
сложную задачу — «обобщение» (*generifying*) старого кода.

Если вы решили перевести старый код на обобщённые типы, нужно тщательно продумать, как вы изменяете API.

Вы должны убедиться, что обобщённый API не становится чрезмерно ограничительным; он должен продолжать
поддерживать исходный контракт API. Рассмотрим снова несколько примеров из `java.util.Collection`. До
введения обобщённых типов API выглядел так:

```java
interface Collection {
    public boolean containsAll(Collection c);
    public boolean addAll(Collection c);
}
```

Наивная попытка обобщить его была бы такой:

```java
interface Collection<E> {
    public boolean containsAll(Collection<E> c);
    public boolean addAll(Collection<E> c);
}
```

Хотя это, безусловно, типобезопасно, оно не соответствует исходному контракту API. Метод
`containsAll()` работает с любым видом входящей коллекции. Он завершится успехом, только если входящая
коллекция действительно содержит лишь экземпляры `E`, но:

- Статический тип входящей коллекции может отличаться — возможно, потому что вызывающая сторона не знает
  точного типа передаваемой коллекции, либо потому что это `Collection<S>`, где `S` — подтип `E`.
- Совершенно правомерно вызывать `containsAll()` с коллекцией другого типа. Метод должен работать,
  возвращая `false`.

В случае `addAll()` мы должны иметь возможность добавлять любую коллекцию, состоящую из экземпляров
подтипа `E`. Как корректно справиться с этой ситуацией, мы видели в разделе «Обобщённые методы»
(*Generic Methods*).

Также необходимо обеспечить, чтобы переработанный API сохранял двоичную совместимость (*binary
compatibility*) со старыми клиентами. Это означает, что стирание (*erasure*) API должно быть таким же,
как у исходного, необобщённого API. В большинстве случаев это получается естественно, но есть и
тонкие случаи. Мы разберём один из самых тонких встречавшихся нам случаев — метод `Collections.max()`.
Как мы видели в разделе «Ещё немного о wildcards», правдоподобной сигнатурой `max()` является:

```java
public static <T extends Comparable<? super T>>
        T max(Collection<T> coll)
```

Это хорошо, за исключением того, что стирание этой сигнатуры таково:

```java
public static Comparable max(Collection coll)
```

что отличается от исходной сигнатуры `max()`:

```java
public static Object max(Collection coll)
```

Можно было бы, конечно, задать `max()` именно эту сигнатуру, но этого сделано не было, и все старые
двоичные class-файлы, вызывающие `Collections.max()`, рассчитывают на сигнатуру, возвращающую `Object`.

Мы можем заставить стирание отличаться, явно указав суперкласс в границе для формального параметра типа `T`.

```java
public static <T extends Object & Comparable<? super T>>
        T max(Collection<T> coll)
```

Это пример задания *множественной границы* (*multiple bounds*) для параметра типа с помощью синтаксиса
`T1 & T2 ... & Tn`. Про переменную типа с множественной границей известно, что она является подтипом
всех типов, перечисленных в границе. Когда используется множественная граница, в качестве стирания
переменной типа берётся первый упомянутый в границе тип.

Наконец, следует вспомнить, что `max` только читает из своей входной коллекции и потому применим к
коллекциям любого подтипа `T`.

Это приводит нас к фактической сигнатуре, используемой в JDK:

```java
public static <T extends Object & Comparable<? super T>>
        T max(Collection<? extends T> coll)
```

На практике нечто столь хитрое встречается крайне редко, но опытные проектировщики библиотек должны
быть готовы думать очень внимательно при переводе существующих API.

Ещё одна проблема, которой стоит остерегаться, — *ковариантные возвращаемые типы* (*covariant returns*),
то есть уточнение типа возвращаемого значения метода в подклассе. Вам не следует пользоваться этой
возможностью в старом API. Чтобы понять почему, рассмотрим пример.

Предположим, ваш исходный API имел вид:

```java
public class Foo {
    // Фабрика. Должна создавать экземпляр того
    // класса, в котором она объявлена.
    public Foo create() {
        ...
    }
}

public class Bar extends Foo {
    // Фактически создаёт Bar.
    public Foo create() {
        ...
    }
}
```

Воспользовавшись ковариантными возвращаемыми типами, вы изменяете его на:

```java
public class Foo {
    // Фабрика. Должна создавать экземпляр того
    // класса, в котором она объявлена.
    public Foo create() {
        ...
    }
}

public class Bar extends Foo {
    // Фактически создаёт Bar.
    public Bar create() {
        ...
    }
}
```

Теперь предположим, что сторонний клиент вашего кода написал следующее:

```java
public class Baz extends Bar {
    // Фактически создаёт Baz.
    public Foo create() {
        ...
    }
}
```

Виртуальная машина Java не поддерживает напрямую переопределение методов с разными типами
возвращаемых значений. Эта возможность обеспечивается компилятором. Следовательно, пока класс `Baz`
не будет перекомпилирован, он не будет корректно переопределять метод `create()` класса `Bar`. Более
того, `Baz` придётся изменить, поскольку код в написанном виде будет отвергнут — тип возвращаемого
значения `create()` в `Baz` не является подтипом типа возвращаемого значения `create()` в `Bar`.
