## LDAP

X.500 — стандарт CCITT для служб каталогов (directory services), входящий в набор служб OSI.
Стандарт X.500 определяет (среди прочего) протокол, по которому клиентское приложение обращается
к каталогу X.500, — **протокол доступа к каталогу** (Directory Access Protocol, DAP). Он построен
поверх стека протоколов взаимодействия открытых систем (Open Systems Interconnection, OSI).

Интернет-сообщество, осознав потребность в службе наподобие X.500, но столкнувшись с иной сетевой
инфраструктурой (TCP/IP вместо OSI), разработало на основе X.500 DAP новый протокол — **облегчённый
DAP** (Lightweight DAP, или LDAP). [RFC 2251](http://www.ietf.org/rfc/rfc2251.txt) определяет то,
что теперь называется **версией 3** LDAP (LDAP v3) — улучшенную версию предшественника LDAP v2,
описанного в [RFC 1777](http://www.ietf.org/rfc/rfc1777.txt).

Целью LDAP был протокол, который можно легко реализовать, с особым упором на возможность создавать
маленькие и простые клиенты. Один из способов добиться упрощения — широко использовать строки и
по возможности минимизировать применение структур. Например, различающиеся имена (Distinguished
Names, DN) представлены в протоколе строками — как и имена типов атрибутов, и многие значения
атрибутов.

Протокол заключается в том, что клиент посылает запросы серверу, а сервер на них отвечает, хотя
и не обязательно в том же порядке, в каком запросы были отправлены. Каждый запрос помечается
идентификатором (ID), чтобы можно было сопоставить запросы с ответами. Протокол работает поверх
TCP или UDP, но чаще всего используется вариант на TCP.

Из-за ориентации на клиентов сообщество LDAP также определило стандарты строкового представления
DN ([RFC 2553](http://www.ietf.org/rfc/rfc2553.txt)), фильтров поиска
([RFC 1960](http://www.ietf.org/rfc/rfc1960.txt)) и синтаксисов атрибутов
([RFC 1778](http://www.ietf.org/rfc/rfc1778.txt)), для API на языке C
([RFC 1823](http://www.ietf.org/rfc/rfc1823.txt)) и для формата URL доступа к службам LDAP
([RFC 1959](http://www.ietf.org/rfc/rfc1959.txt)).

LDAP v3 поддерживает интернационализацию, различные механизмы аутентификации, отсылки (referrals)
и общий механизм развёртывания. Он позволяет добавлять в протокол новые возможности, не требуя при
этом изменения самого протокола, за счёт **расширений** (extensions) и **управляющих элементов**
(controls).

## LDAP v3

### Интернационализация

Интернационализация решается за счёт международного набора символов (ISO 10646) для представления
строковых элементов протокола (таких как DN). Версия 3 также отличается от версии 2 тем, что для
кодирования строк использует UTF-8.

### Аутентификация

Помимо анонимной и простой (clear-text password, незашифрованный пароль) аутентификации, LDAP v3
использует фреймворк аутентификации **Simple Authentication and Security Layer (SASL)**
([RFC 2222](http://www.ietf.org/rfc/rfc2222.txt)), позволяющий применять с LDAP различные механизмы
аутентификации. SASL задаёт протокол «запрос-ответ» (challenge-response), в котором между клиентом
и сервером обмениваются данными в целях аутентификации.

Сейчас определено несколько механизмов SASL:
[DIGEST-MD5](http://www.ietf.org/rfc/rfc2831.txt), [CRAM-MD5](http://www.ietf.org/rfc/rfc2195.txt),
[Anonymous](http://www.ietf.org/rfc/rfc2245.txt), [External](http://www.ietf.org/rfc/rfc2222.txt),
[S/Key](http://www.ietf.org/rfc/rfc2222.txt), [GSSAPI](http://www.ietf.org/rfc/rfc2222.txt) и
[Kerberos v4](http://www.ietf.org/rfc/rfc2222.txt). Клиент LDAP v3 может использовать любой из этих
механизмов SASL при условии, что сервер LDAP v3 их поддерживает. Более того, новые (ещё не
определённые) механизмы SASL можно применять без внесения изменений в LDAP.

### Отсылки (Referrals)

**Отсылка** (referral) — это информация, которую сервер отправляет клиенту, указывая, что
запрошенные сведения можно найти в другом месте (возможно, на другом сервере). В LDAP v2 серверы
должны были обрабатывать отсылки сами и не возвращать их клиенту, потому что обработка отсылок может
быть очень сложной и привела бы к усложнению клиентов. Когда серверы стали создавать и развёртывать,
оказалось, что отсылки полезны, но мало какие серверы поддерживали обработку отсылок на стороне
сервера. Поэтому был найден способ «дооснастить» протокол возможностью возвращать отсылки — отсылку
помещали внутрь сообщения об ошибке в ответе «partial result» (частичный результат).

LDAP v3 явно поддерживает отсылки и позволяет серверам возвращать их клиенту напрямую. Отсылки в
этом уроке не рассматриваются, но всегда можно обратиться к
[JNDI Tutorial](https://docs.oracle.com/javase/jndi/tutorial/ldap/referral/index.html) для управления
отсылками в вашем приложении.

### Развёртывание (Deployment)

Общий протокол, такой как LDAP, полезен тем, что обеспечивает «общий язык» для всех клиентов и
серверов каталогов. Когда в сети развёрнуто множество различных клиентских приложений и серверов
каталогов, всем этим сущностям также очень полезно «говорить» об одних и тех же объектах.

**Схема каталога** (directory schema) задаёт, среди прочего, типы объектов, которые может содержать
каталог, а также обязательные и необязательные атрибуты каждого типа объекта. LDAP v3 определяет
схему ([RFC 2252](http://www.ietf.org/rfc/rfc2252.txt) и
[RFC 2256](http://www.ietf.org/rfc/rfc2256.txt)) на основе стандарта X.500 для распространённых
объектов сети — стран, населённых пунктов, организаций, пользователей/лиц, групп и устройств. Он
также определяет способ доступа клиентского приложения к схеме сервера, чтобы узнать типы объектов
и атрибутов, поддерживаемые конкретным сервером.

LDAP v3 далее определяет набор синтаксисов для представления значений атрибутов
([RFC 2252](http://www.ietf.org/rfc/rfc2252.txt)). О написании Java-приложений, которым нужен доступ
к схеме, см. [JNDI Tutorial](https://docs.oracle.com/javase/jndi/tutorial/ldap/schema/index.html).

### Расширения (Extensions)

Помимо набора предопределённых операций, таких как «search» (поиск) и «modify» (изменение), LDAP v3
определяет **«расширенную» операцию** (extended operation). «Расширенная» операция принимает запрос
в качестве аргумента и возвращает ответ. Запрос содержит идентификатор, определяющий запрос, и
аргументы запроса. Ответ содержит результаты выполнения запроса. Пару «расширенная» операция
запрос/ответ называют **расширением** (extension). Например, может быть расширение для Start TLS —
запрос от клиента серверу активировать протокол Start TLS.

Эти расширения могут быть стандартными (определёнными сообществом LDAP) или проприетарными
(определёнными конкретным поставщиком каталога). О написании приложений, которым нужны расширения,
см. [JNDI Tutorial](https://docs.oracle.com/javase/jndi/tutorial/ldap/ext/index.html).

### Управляющие элементы (Controls)

Ещё один способ добавления новых возможностей — **управляющий элемент** (control). LDAP v3 позволяет
изменять поведение любой операции с помощью управляющих элементов. С операцией можно отправить любое
число управляющих элементов, и любое их число может быть возвращено вместе с результатами. Например,
с операцией «search» можно отправить управляющий элемент Sort, который укажет серверу отсортировать
результаты поиска по атрибуту «name».

Как и расширения, такие управляющие элементы могут быть стандартными или проприетарными. Стандартные
управляющие элементы предоставляются
[в платформе](https://docs.oracle.com/javase/8/docs/api/javax/naming/ldap/Control.html). О написании
приложений, которым нужны управляющие элементы, см.
[JNDI Tutorial](https://docs.oracle.com/javase/jndi/tutorial/ldap/ext/index.html).

## JNDI как API к LDAP

И модель JNDI, и модель LDAP определяют иерархическое пространство имён, в котором вы именуете
объекты. Каждый объект в пространстве имён может иметь атрибуты, которые можно использовать для
поиска этого объекта. На столь высоком уровне обе модели схожи, поэтому неудивительно, что JNDI
хорошо отображается на LDAP.

### Модели

Запись LDAP (LDAP entry) можно представлять как `DirContext` в JNDI. Каждая запись LDAP содержит
имя и набор атрибутов, а также необязательный набор дочерних записей. Например, запись LDAP
«o=JNDITutorial» может иметь атрибуты «objectclass» и «o», а её детьми могут быть «ou=Groups» и
«ou=People».

В JNDI запись LDAP «o=JNDITutorial» представляется контекстом с именем «o=JNDITutorial», у которого
есть два подконтекста с именами «ou=Groups» и «ou=People». Атрибуты записи LDAP представляются
интерфейсом `Attributes`, а отдельные атрибуты — интерфейсом `Attribute`.

О том, как операции LDAP доступны через JNDI, см. следующую часть этого урока.

### Имена

В результате федерации имена, которые вы передаёте методам контекста JNDI, могут охватывать
несколько пространств имён. Они называются **составными именами** (composite names). Используя JNDI
для доступа к службе LDAP, помните, что символ прямого слэша («/») в строковом имени имеет особое
значение для JNDI. Если имя записи LDAP содержит этот символ, его нужно экранировать (с помощью
обратного слэша «\\»). Например, запись LDAP с именем «cn=O/R» нужно передавать методам контекста
JNDI в виде строки «cn=O\\/R». Подробнее об именах — в JNDI Tutorial. Классы `LdapName` и `Rdn`
упрощают создание и обработку имён LDAP.

Имена LDAP в том виде, как они используются в протоколе, всегда являются полностью квалифицированными
именами, идентифицирующими записи, начиная от корня пространства имён LDAP (как он определён
сервером). Вот несколько примеров полностью квалифицированных имён LDAP:

```
cn=Ted Geisel, ou=Marketing, o=Some Corporation, c=gb
cn=Vinnie Ryan, ou=People, o=JNDITutorial
```

В JNDI, однако, имена всегда **относительны** (relative); то есть объект всегда именуется
относительно контекста. Например, запись «cn=Vinnie Ryan» можно именовать относительно контекста с
именем «ou=People, o=JNDITutorial». Или можно именовать запись «cn=Vinnie Ryan, ou=People»
относительно контекста «o=JNDITutorial». Или можно создать начальный контекст, указывающий на корень
пространства имён сервера LDAP, и именовать запись «cn=Vinnie Ryan, ou=People, o=JNDITutorial».

В JNDI для именования записей LDAP можно также использовать URL-адреса LDAP. См. обсуждение URL
LDAP в JNDI Tutorial.

## Как операции LDAP отображаются на API JNDI

LDAP определяет набор операций (запросов) — см. [RFC 2251](http://www.ietf.org/rfc/rfc2251.txt). В
JNDI они отображаются на операции интерфейсов
[`DirContext`](https://docs.oracle.com/javase/8/docs/api/javax/naming/directory/DirContext.html) и
[`LdapContext`](https://docs.oracle.com/javase/8/docs/api/javax/naming/ldap/LdapContext.html)
(которые являются субинтерфейсами
[`Context`](https://docs.oracle.com/javase/8/docs/api/javax/naming/Context.html)). Например, когда
вызывающий код обращается к методу `DirContext`, поставщик службы LDAP реализует этот метод, отправляя
LDAP-запросы серверу LDAP.

В следующей таблице показано, как операции LDAP соответствуют методам JNDI.

| Операция LDAP | Соответствующие методы JNDI |
|---|---|
| **bind** | Соответствующий способ установить начальное соединение с сервером LDAP в JNDI — это создание [`InitialDirContext`](https://docs.oracle.com/javase/8/docs/api/javax/naming/directory/InitialDirContext.html). При создании начального контекста приложение передаёт информацию аутентификации клиента через свойства окружения (environment properties). Чтобы изменить эту информацию для уже существующего контекста, используйте `Context.addToEnvironment()` и `Context.removeFromEnvironment()`. |
| **unbind** | `Context.close()` используется для освобождения ресурсов, занятых контекстом. От LDAP-операции «unbind» отличается тем, что в рамках одной реализации поставщика ресурсы могут разделяться между контекстами, поэтому закрытие одного контекста не освободит все ресурсы, если они разделяются с другим контекстом. Закрывайте все контексты, если хотите освободить все ресурсы. |
| **search** | Соответствующий метод JNDI — перегрузка `DirContext.search()`, принимающая фильтр поиска ([RFC 2254](http://www.ietf.org/rfc/rfc2254.txt)). См. пример [filter](https://docs.oracle.com/javase/tutorial/jndi/ops/filter.html). |
| **modify** | Соответствующий метод JNDI — перегрузка `DirContext.modifyAttributes()`, принимающая массив `ModificationItem`. Пример — см. раздел Modify Attributes. |
| **add** | Соответствующие методы JNDI — `DirContext.bind()` и `DirContext.createSubcontext()`. Любой из них можно использовать для добавления новой записи LDAP. С помощью `bind()` можно задать для новой записи не только набор атрибутов, но и Java-объект, добавляемый вместе с атрибутами. |
| **delete** | Соответствующие методы JNDI — `Context.unbind()` и `Context.destroySubcontext()`. Любой из них можно использовать для удаления записи LDAP. |
| **modify DN/RDN** | Соответствующий метод JNDI — `Context.rename()`. Подробнее см. раздел «Переименование объектов». |
| **compare** | Соответствующая операция JNDI — подходящим образом ограниченный `DirContext.search()`. Пример — см. раздел «LDAP Compare». |
| **abandon** | Когда вы закрываете контекст, все его незавершённые запросы прекращаются (abandoned). Аналогично, при закрытии `NamingEnumeration` прекращается соответствующий LDAP-запрос «search». |
| **extended operation** | Соответствующий метод JNDI — `LdapContext.extendedOperation()`. Подробнее см. [JNDI Tutorial](https://docs.oracle.com/javase/jndi/tutorial/ldap/ext/index.html). |

## Как коды ошибок LDAP отображаются на исключения JNDI

LDAP определяет набор кодов состояния, возвращаемых вместе с ответами сервера LDAP (см.
[RFC 2251](http://www.ietf.org/rfc/rfc2251.txt)). В JNDI ошибочные состояния обозначаются
проверяемыми исключениями — подклассами
[`NamingException`](https://docs.oracle.com/javase/8/docs/api/javax/naming/NamingException.html).

Поставщик службы LDAP транслирует полученный от сервера код состояния LDAP в подходящий подкласс
`NamingException`. В следующей таблице показано соответствие кодов состояния LDAP и исключений JNDI.

| Код LDAP | Значение | Исключение или действие |
|---|---|---|
| 0 | Success (успех) | Сообщить об успехе. |
| 1 | Operations error (ошибка операций) | `NamingException` |
| 2 | Protocol error (ошибка протокола) | `CommunicationException` |
| 3 | Time limit exceeded (превышен лимит времени) | `TimeLimitExceededException` |
| 4 | Size limit exceeded (превышен лимит размера) | `SizeLimitExceededException` |
| 5 | Compared false (сравнение дало «ложь») | Используется `DirContext.search()`. Исключение не генерируется. |
| 6 | Compared true (сравнение дало «истину») | Используется `DirContext.search()`. Исключение не генерируется. |
| 7 | Authentication method not supported (метод аутентификации не поддерживается) | `AuthenticationNotSupportedException` |
| 8 | Strong authentication required (требуется строгая аутентификация) | `AuthenticationNotSupportedException` |
| 9 | Partial results being returned (возвращаются частичные результаты) | Если свойство окружения "java.naming.referral" установлено в "ignore" или содержимое ошибки не содержит отсылки — выбросить `PartialResultException`. Иначе использовать содержимое для построения отсылки. |
| 10 | Referral encountered (встречена отсылка) | Если "java.naming.referral" = "ignore" — игнорировать. Если "throw" — выбросить `ReferralException`. Если "follow" — поставщик LDAP обрабатывает отсылку. Если превышен предел "java.naming.ldap.referral.limit" — выбросить `LimitExceededException`. |
| 11 | Administrative limit exceeded (превышен административный лимит) | `LimitExceededException` |
| 12 | Unavailable critical extension requested (запрошено недоступное критическое расширение) | `OperationNotSupportedException` |
| 13 | Confidentiality required (требуется конфиденциальность) | `AuthenticationNotSupportedException` |
| 14 | SASL bind in progress (выполняется bind по SASL) | Используется внутренне поставщиком LDAP во время аутентификации. |
| 16 | No such attribute exists (нет такого атрибута) | `NoSuchAttributeException` |
| 17 | Undefined attribute type (неопределённый тип атрибута) | `InvalidAttributeIdentifierException` |
| 18 | Inappropriate matching (неподходящее сопоставление) | `InvalidSearchFilterException` |
| 19 | Constraint violation (нарушение ограничения) | `InvalidAttributeValueException` |
| 20 | Attribute or value already in use (атрибут или значение уже используется) | `AttributeInUseException` |
| 21 | Invalid attribute syntax (неверный синтаксис атрибута) | `InvalidAttributeValueException` |
| 32 | No such object exists (нет такого объекта) | `NameNotFoundException` |
| 33 | Alias problem (проблема псевдонима) | `NamingException` |
| 34 | Invalid DN syntax (неверный синтаксис DN) | `InvalidNameException` |

## Безопасность

Служба LDAP предоставляет универсальную службу каталога. Её можно использовать для хранения
информации любого рода. У всех серверов LDAP есть та или иная система контроля того, кто может читать
и обновлять информацию в каталоге.

Чтобы получить доступ к службе LDAP, клиент LDAP сначала должен **аутентифицироваться** перед службой.
То есть он должен сообщить серверу LDAP, кто собирается обращаться к данным, чтобы сервер мог решить,
что клиенту разрешено видеть и делать. Если клиент успешно аутентифицируется на сервере LDAP, то при
последующем получении запроса от клиента сервер проверит, разрешено ли клиенту выполнять этот запрос.
Этот процесс называется **управлением доступом** (access control).

Стандарт LDAP предложил способы аутентификации клиентов LDAP на серверах LDAP
([RFC 2251](http://www.ietf.org/rfc/rfc2251.txt) и
[RFC 2829](http://www.ietf.org/rfc/rfc2829.txt)). Они в общих чертах рассмотрены в разделах
«Режимы аутентификации в LDAP» и «Механизмы аутентификации». Этот урок также описывает, как
использовать механизмы аутентификации **anonymous** (анонимный), **simple** (простой) и **SASL**.

Управление доступом поддерживается по-разному в разных реализациях серверов LDAP. В этом уроке оно
не рассматривается.

Ещё один аспект безопасности службы LDAP — поддержка защищённых каналов связи с клиентами, например
для отправки и получения атрибутов, содержащих секреты (пароли, ключи). Для этого серверы LDAP
используют SSL. Этот урок также показывает, как использовать SSL с поставщиком службы LDAP.

## Режимы аутентификации в LDAP

В LDAP информация аутентификации передаётся в операции «bind». В LDAP v2 клиент инициирует соединение
с сервером LDAP, отправляя серверу операцию «bind», содержащую информацию аутентификации.

В LDAP v3 эта операция служит той же цели, но является необязательной. Клиент, отправивший
LDAP-запрос без выполнения «bind», рассматривается как **анонимный** клиент (см. раздел Anonymous).
В LDAP v3 операцию «bind» можно отправлять в любой момент и, возможно, неоднократно в течение
соединения. Клиент может отправить запрос «bind» посреди соединения, чтобы сменить свою личность.
Если запрос успешен, то все незавершённые запросы, использовавшие старую личность в этом соединении,
отбрасываются, а соединение связывается с новой личностью.

Информация аутентификации, передаваемая в операции «bind», зависит от **механизма аутентификации**,
выбранного клиентом (см. раздел «Механизмы аутентификации»).

### Аутентификация в LDAP с помощью JNDI

В JNDI информация аутентификации задаётся в свойствах окружения. При создании начального контекста
с помощью класса
[`InitialDirContext`](https://docs.oracle.com/javase/8/docs/api/javax/naming/directory/InitialDirContext.html)
(или его суперкласса либо подкласса) вы передаёте набор свойств окружения, часть из которых может
содержать информацию аутентификации. Можно использовать следующие свойства окружения:

* **`Context.SECURITY_AUTHENTICATION`** ("java.naming.security.authentication"). Задаёт используемый
  механизм аутентификации. Для поставщика службы LDAP в JDK это может быть одна из строк: "none",
  "simple" или *sasl_mech*, где *sasl_mech* — список имён механизмов SASL, разделённых пробелами.
* **`Context.SECURITY_PRINCIPAL`** ("java.naming.security.principal"). Задаёт имя пользователя/программы,
  выполняющей аутентификацию; зависит от значения свойства `Context.SECURITY_AUTHENTICATION`.
* **`Context.SECURITY_CREDENTIALS`** ("java.naming.security.credentials"). Задаёт учётные данные
  пользователя/программы, выполняющей аутентификацию; зависит от значения свойства
  `Context.SECURITY_AUTHENTICATION`.

При создании начального контекста нижележащий поставщик службы LDAP извлекает информацию
аутентификации из этих свойств окружения и передаёт её серверу с помощью LDAP-операции «bind».

В следующем примере показано, как клиент аутентифицируется на сервере LDAP, используя простой
незашифрованный пароль.

```java
// Настраиваем окружение для создания начального контекста
Hashtable<String, Object> env = new Hashtable<String, Object>();
env.put(Context.INITIAL_CONTEXT_FACTORY, "com.sun.jndi.ldap.LdapCtxFactory");
env.put(Context.PROVIDER_URL, "ldap://localhost:389/o=JNDITutorial");

// Аутентификация как S. User с паролем "mysecret"
env.put(Context.SECURITY_AUTHENTICATION, "simple");
env.put(Context.SECURITY_PRINCIPAL,
        "cn=S. User, ou=NewHires, o=JNDITutorial");
env.put(Context.SECURITY_CREDENTIALS, "mysecret");

// Создаём начальный контекст
DirContext ctx = new InitialDirContext(env);

// ... делаем что-нибудь полезное с ctx
```

### Использование другой информации аутентификации для контекста

Если для существующего контекста нужно использовать другую информацию аутентификации, можно применить
`Context.addToEnvironment()` и `Context.removeFromEnvironment()`, чтобы обновить свойства окружения,
содержащие информацию аутентификации. Последующие вызовы методов контекста будут использовать новую
информацию аутентификации при обмене с сервером.

```java
// Аутентификация как S. User с паролем "mysecret"
env.put(Context.SECURITY_AUTHENTICATION, "simple");
env.put(Context.SECURITY_PRINCIPAL,
        "cn=S. User, ou=NewHires, o=JNDITutorial");
env.put(Context.SECURITY_CREDENTIALS, "mysecret");

// Создаём начальный контекст
DirContext ctx = new InitialDirContext(env);

// ... делаем что-нибудь полезное с ctx

// Переходим на отсутствие аутентификации
ctx.addToEnvironment(Context.SECURITY_AUTHENTICATION, "none");

// ... делаем что-нибудь полезное с ctx
```

### Сбои аутентификации

Аутентификация может завершиться неудачей по ряду причин. Например, если вы передадите неверную
информацию аутентификации (неверный пароль или имя принципала), будет выброшено исключение
[`AuthenticationException`](https://docs.oracle.com/javase/8/docs/api/javax/naming/AuthenticationException.html).
Вот вариант предыдущего примера с неверным паролем:

```java
// Аутентификация как S. User с неверным паролем
env.put(Context.SECURITY_AUTHENTICATION, "simple");
env.put(Context.SECURITY_PRINCIPAL,
        "cn=S. User, ou=NewHires, o=JNDITutorial");
env.put(Context.SECURITY_CREDENTIALS, "notmysecret");
```

Это даст вывод:

```
javax.naming.AuthenticationException: [LDAP: error code 49 - Invalid Credentials]
        ...
```

Поскольку разные серверы поддерживают разные механизмы аутентификации, можно запросить механизм,
который сервер не поддерживает. В этом случае будет выброшено
[`AuthenticationNotSupportedException`](https://docs.oracle.com/javase/8/docs/api/javax/naming/AuthenticationNotSupportedException.html):

```java
// Запрос неподдерживаемого механизма "custom"
env.put(Context.SECURITY_AUTHENTICATION, "custom");
env.put(Context.SECURITY_PRINCIPAL,
        "cn=S. User, ou=NewHires, o=JNDITutorial");
env.put(Context.SECURITY_CREDENTIALS, "mysecret");
```

Вывод:

```
javax.naming.AuthenticationNotSupportedException: custom
        ...
```

## Механизмы аутентификации

Разные версии LDAP поддерживают разные типы аутентификации. LDAP v2 определяет три типа: anonymous
(анонимный), simple (clear-text password, незашифрованный пароль) и Kerberos v4.

LDAP v3 поддерживает anonymous, simple и SASL. SASL — это Simple Authentication and Security Layer
([RFC 2222](http://www.ietf.org/rfc/rfc2222.txt)). Он задаёт протокол «запрос-ответ», в котором между
клиентом и сервером обмениваются данными в целях аутентификации и установления уровня защиты, на
котором будет вестись последующий обмен. Благодаря SASL LDAP может поддерживать любой тип
аутентификации, согласованный между клиентом и сервером LDAP.

Этот урок описывает, как аутентифицироваться с помощью Anonymous, Simple и SASL.

### Указание механизма аутентификации

Механизм аутентификации задаётся свойством окружения `Context.SECURITY_AUTHENTICATION`. Свойство
может иметь одно из следующих значений:

| Имя свойства | Значение свойства |
|---|---|
| *sasl_mech* | Список имён механизмов SASL, разделённых пробелами. Используйте один из перечисленных механизмов SASL (например, "CRAM-MD5" означает использовать механизм SASL CRAM-MD5, описанный в RFC 2195). |
| none | Без аутентификации (анонимная) |
| simple | Слабая аутентификация (незашифрованный пароль) |

### Механизм по умолчанию

Если клиент не задаёт никаких свойств окружения аутентификации, то механизм по умолчанию — "none".
Тогда клиент рассматривается как анонимный.

Если клиент задаёт информацию аутентификации, но явно не указывает свойство
`Context.SECURITY_AUTHENTICATION`, то механизм по умолчанию — "simple".

### Анонимная аутентификация (Anonymous)

Как уже сказано, механизм по умолчанию — "none", если не заданы никакие свойства окружения
аутентификации. Если клиент устанавливает `Context.SECURITY_AUTHENTICATION` в "none", то механизм —
"none", а все остальные свойства окружения аутентификации игнорируются. Делать это явно имеет смысл
лишь для того, чтобы гарантированно проигнорировать любые другие свойства аутентификации, которые
могли быть заданы. В любом случае клиент рассматривается как **анонимный**. Это означает, что сервер
не знает и не интересуется тем, кто такой клиент, и разрешит клиенту обращаться (читать и обновлять)
к любым данным, сконфигурированным как доступные любому неаутентифицированному клиенту.

Поскольку ни в одном из примеров уроков «Naming and Directory Operations» не задаются свойства
окружения аутентификации, все они используют анонимную аутентификацию.

Вот пример, явно устанавливающий `Context.SECURITY_AUTHENTICATION` в "none" (хотя это и не строго
необходимо, поскольку это значение по умолчанию):

```java
// Настраиваем окружение для создания начального контекста
Hashtable<String, Object> env = new Hashtable<String, Object>();
env.put(Context.INITIAL_CONTEXT_FACTORY, "com.sun.jndi.ldap.LdapCtxFactory");
env.put(Context.PROVIDER_URL, "ldap://localhost:389/o=JNDITutorial");

// Используем анонимную аутентификацию
env.put(Context.SECURITY_AUTHENTICATION, "none");

// Создаём начальный контекст
DirContext ctx = new InitialDirContext(env);

// ... делаем что-нибудь полезное с ctx
```

### Простая аутентификация (Simple)

**Простая** аутентификация заключается в отправке серверу LDAP полностью квалифицированного DN
клиента (пользователя) и незашифрованного пароля клиента (см.
[RFC 2251](http://www.ietf.org/rfc/rfc2251.txt) и
[RFC 2829](http://www.ietf.org/rfc/rfc2829.txt)). У этого механизма есть проблема безопасности:
пароль можно прочитать из сети. Чтобы не раскрывать пароль таким образом, можно использовать простую
аутентификацию внутри зашифрованного канала (например, SSL), если это поддерживается сервером LDAP.

Простую аутентификацию поддерживают и LDAP v2, и LDAP v3.

Чтобы использовать механизм простой аутентификации, нужно задать три свойства окружения:

* **`Context.SECURITY_AUTHENTICATION`** — установить в "simple".
* **`Context.SECURITY_PRINCIPAL`** — установить в полностью квалифицированный DN аутентифицируемой
  сущности (например, "cn=S. User, ou=NewHires, o=JNDITutorial"). Тип — `java.lang.String`.
* **`Context.SECURITY_CREDENTIALS`** — установить в пароль принципала (например, "mysecret"). Тип —
  `java.lang.String`, массив символов (`char[]`) или массив байтов (`byte[]`). Если пароль — это
  `java.lang.String` или `char[]`, то он кодируется в UTF-8 для LDAP v3 и в ISO-Latin-1 для LDAP v2
  при передаче серверу. Если пароль — `byte[]`, то он передаётся серверу как есть.

> **Примечание.** Если передать пустую строку, пустой массив байтов/символов или null в свойство
> `Context.SECURITY_CREDENTIALS`, то механизм аутентификации станет "none". Это потому, что LDAP
> требует, чтобы при простой аутентификации пароль был непустым. Протокол автоматически переключает
> аутентификацию на "none", если пароль не предоставлен.

### SASL

Протокол LDAP v3 использует [SASL](http://www.ietf.org/rfc/rfc2222.txt) для поддержки
**подключаемой** (pluggable) аутентификации. Это означает, что клиент и сервер LDAP можно настроить
на согласование и использование, возможно, нестандартных и/или настраиваемых механизмов
аутентификации в зависимости от желаемого уровня защиты. Протокол LDAP v2 SASL не поддерживает.

Сейчас определено несколько механизмов SASL:

* Anonymous ([RFC 2245](http://www.ietf.org/rfc/rfc2245.txt))
* CRAM-MD5 ([RFC 2195](http://www.ietf.org/rfc/rfc2195.txt))
* Digest-MD5 ([RFC 2831](http://www.ietf.org/rfc/rfc2831.txt))
* External ([RFC 2222](http://www.ietf.org/rfc/rfc2830.txt))
* Kerberos V4 ([RFC 2222](http://www.ietf.org/rfc/rfc2830.txt))
* Kerberos V5 ([RFC 2222](http://www.ietf.org/rfc/rfc2830.txt))
* SecurID ([RFC 2808](http://www.ietf.org/rfc/rfc2808.txt))
* S/Key ([RFC 2222](http://www.ietf.org/rfc/rfc2830.txt))

#### Механизмы SASL, поддерживаемые серверами LDAP

Из перечисленных механизмов популярные серверы LDAP (например, от Oracle, OpenLDAP и Microsoft)
поддерживают External, Digest-MD5 и Kerberos V5. [RFC 2829](http://www.ietf.org/rfc/rfc2829.txt)
предлагает использовать Digest-MD5 в качестве обязательного механизма по умолчанию для серверов
LDAP v3.

Вот простая программа, позволяющая узнать список механизмов SASL, поддерживаемых сервером LDAP:

```java
// Создаём начальный контекст
DirContext ctx = new InitialDirContext();

// Читаем supportedSASLMechanisms из корневого DSE
Attributes attrs = ctx.getAttributes(
    "ldap://localhost:389", new String[]{"supportedSASLMechanisms"});
```

Вот вывод этой программы против сервера, поддерживающего механизм External:

```
{supportedsaslmechanisms=supportedSASLMechanisms:
                         EXTERNAL, GSSAPI, DIGEST-MD5}
```

#### Указание механизма аутентификации

Чтобы использовать конкретный механизм SASL, укажите его имя, зарегистрированное в IANA (Internet
Assigned Numbers Authority), в свойстве окружения `Context.SECURITY_AUTHENTICATION`. Можно также
указать список механизмов, которые поставщик LDAP будет пробовать, — в виде упорядоченного списка
имён, разделённых пробелами. Поставщик LDAP использует первый механизм, для которого найдёт
реализацию.

Вот пример, который просит поставщика LDAP попробовать получить реализацию механизма DIGEST-MD5, а
если она недоступна — использовать GSSAPI:

```java
env.put(Context.SECURITY_AUTHENTICATION, "DIGEST-MD5 GSSAPI");
```

Этот список механизмов можно получить от пользователя приложения или спросив сервер LDAP (вызовом,
подобным показанному выше). Сам поставщик LDAP не обращается к серверу за этой информацией — он лишь
пытается найти и использовать реализацию указанных механизмов.

Поставщик LDAP в платформе имеет встроенную поддержку механизмов SASL External, Digest-MD5 и GSSAPI
(Kerberos v5). Можно добавить поддержку дополнительных механизмов.

#### Указание входных данных для механизма аутентификации

Некоторые механизмы, такие как External, не требуют дополнительного ввода — для аутентификации
достаточно одного имени механизма. Большинству других механизмов нужен какой-либо дополнительный
ввод. Тип ввода может различаться в зависимости от механизма. Вот распространённые входные данные:

* **Идентификатор аутентификации** (Authentication id). Личность сущности, выполняющей аутентификацию.
* **Идентификатор авторизации** (Authorization id). Личность сущности, для которой следует выполнять
  проверки управления доступом, если аутентификация успешна.
* **Учётные данные аутентификации** (Authentication credentials). Например, пароль или ключ.

Идентификаторы аутентификации и авторизации могут различаться, если программа (например,
прокси-сервер) аутентифицируется от имени другой сущности. Идентификатор аутентификации задаётся
свойством `Context.SECURITY_PRINCIPAL` (тип `java.lang.String`). Пароль/ключ задаётся свойством
`Context.SECURITY_CREDENTIALS` (тип `java.lang.String`, `char[]` или `byte[]`; если пароль — массив
байтов, он преобразуется в массив символов с помощью кодировки UTF-8).

Если задано свойство "java.naming.security.sasl.authorizationId", то его значение используется как
идентификатор авторизации (тип `java.lang.String`). По умолчанию используется пустая строка, что
указывает серверу вывести идентификатор авторизации из учётных данных аутентификации клиента.

Если механизм требует входных данных, отличных от описанных, нужно определить объект **callback**
для использования механизмом (см. пример callback в JNDI Tutorial). Механизмы SASL Policies, GSS-API
(Kerberos v5) и CRAM-MD5 рассмотрены в JNDI Tutorial.

### Аутентификация Digest-MD5

**Аутентификация Digest-MD5** — обязательный механизм аутентификации для серверов LDAP v3
([RFC 2829](http://www.ietf.org/rfc/rfc2829.txt)). Поскольку использование SASL — часть LDAP v3
([RFC 2251](http://www.ietf.org/rfc/rfc2251.txt)), серверы, поддерживающие только LDAP v2, Digest-MD5
не поддерживают.

Механизм Digest-MD5 описан в [RFC 2831](http://www.ietf.org/rfc/rfc2831.txt). Он основан на HTTP
Digest Authentication. В Digest-MD5 сервер LDAP отправляет клиенту данные, включающие различные
поддерживаемые им опции аутентификации, плюс специальный токен. Клиент отвечает зашифрованным
ответом, указывающим выбранные им опции. Ответ зашифрован так, что подтверждает: клиент знает свой
пароль. Затем сервер LDAP расшифровывает и проверяет ответ клиента.

Чтобы использовать механизм Digest-MD5, нужно задать свойства окружения аутентификации:

* **`Context.SECURITY_AUTHENTICATION`** — установить в строку "DIGEST-MD5".
* **`Context.SECURITY_PRINCIPAL`** — установить в имя принципала. Формат специфичен для сервера.
  Одни серверы поддерживают формат логина (как для входа в UNIX или Windows), другие принимают
  различающееся имя (DN), третьи используют форматы идентификатора авторизации из
  [RFC 2829](http://www.ietf.org/rfc/rfc2829.txt) (имя должно быть строкой "dn:" с последующим
  полностью квалифицированным DN либо строкой "u:" с последующим идентификатором пользователя). Одни
  серверы принимают несколько форматов. Примеры: "cuser", "dn: cn=C. User, ou=NewHires, o=JNDITutorial",
  "u: cuser". Тип свойства — `java.lang.String`.
* **`Context.SECURITY_CREDENTIALS`** — установить в пароль принципала (например, "mysecret"). Тип —
  `java.lang.String`, `char[]` или `byte[]`. Если пароль — `String` или `char[]`, он кодируется в
  UTF-8 при передаче серверу; если `byte[]` — передаётся как есть.

Следующий пример показывает, как клиент выполняет аутентификацию Digest-MD5 на сервере LDAP:

```java
// Настраиваем окружение для создания начального контекста
Hashtable<String, Object> env = new Hashtable<String, Object>();
env.put(Context.INITIAL_CONTEXT_FACTORY, "com.sun.jndi.ldap.LdapCtxFactory");
env.put(Context.PROVIDER_URL, "ldap://localhost:389/o=JNDITutorial");

// Аутентификация как C. User с паролем "mysecret"
env.put(Context.SECURITY_AUTHENTICATION, "DIGEST-MD5");
env.put(Context.SECURITY_PRINCIPAL,
        "dn:cn=C. User, ou=NewHires, o=JNDITutorial");
env.put(Context.SECURITY_CREDENTIALS, "mysecret");

// Создаём начальный контекст
DirContext ctx = new InitialDirContext(env);

// ... делаем что-нибудь полезное с ctx
```

> **Примечание.** Oracle Directory Server v5.2 поддерживает Digest-MD5 для пользователей с
> незашифрованными паролями. Режим шифрования пароля нужно задать **до** создания пользователя.
> Если пользователь уже создан, удалите и пересоздайте его. Сервер принимает простые имена
> пользователей (значение атрибута "uid") и формат имён "dn:". Подробности — в документации сервера.

#### Указание области (Realm)

**Область** (realm) определяет пространство имён, из которого выбирается аутентифицируемая сущность
(значение свойства `Context.SECURITY_PRINCIPAL`). У сервера может быть несколько областей. Например,
сервер университета может быть настроен на две области — для студентов и для преподавателей.
Конфигурацией области занимается администратор каталога. Некоторые каталоги имеют одну область по
умолчанию (например, Oracle Directory Server v5.2 использует в качестве области по умолчанию полностью
квалифицированное имя хоста машины).

В аутентификации Digest-MD5 нужно аутентифицироваться в конкретной области. Для указания области
можно использовать следующее свойство. Если область не задана, будет использована любая из областей,
предложенных сервером.

**java.naming.security.sasl.realm** — установить в область принципала. Это специфичная для
развёртывания и/или сервера строка, чувствительная к регистру. Она идентифицирует область или домен,
из которого должно выбираться имя принципала (`Context.SECURITY_PRINCIPAL`). Если эта область не
совпадает ни с одной из областей, предлагаемых сервером, аутентификация завершится неудачей.

```java
// Аутентификация как C. User с паролем "mysecret" в области "JNDITutorial"
env.put(Context.SECURITY_AUTHENTICATION, "DIGEST-MD5");
env.put(Context.SECURITY_PRINCIPAL,
        "dn:cn=C. User, ou=NewHires, o=JNDITutorial");
env.put(Context.SECURITY_CREDENTIALS, "mysecret");
env.put("java.naming.security.sasl.realm", "JNDITutorial");
```

Если нужны защита приватности и другие свойства SASL — они рассмотрены в JNDI Tutorial.

## SSL и пользовательские сокеты

Помимо аутентификации SASL большинство серверов LDAP позволяют обращаться к своим службам через SSL.
SSL особенно полезен для серверов LDAP v2, поскольку протокол v2 не поддерживает аутентификацию SASL.

Сервер с поддержкой SSL часто поддерживает SSL двумя способами. В самом базовом сервер поддерживает
порты SSL в дополнение к обычным (незащищённым) портам. Другой способ — через расширение Start TLS
([RFC 2830](http://www.ietf.org/rfc/rfc2830.txt)). Этот вариант доступен только серверам LDAP v3.

### Использование свойства SSL-сокета

По умолчанию поставщик службы LDAP в JDK при обмене с сервером LDAP использует обычные (plain)
сокеты. Чтобы запросить использование SSL-сокетов, установите свойство `Context.SECURITY_PROTOCOL`
в "ssl".

В следующем примере сервер LDAP предлагает SSL на порту 636. Чтобы запустить эту программу, нужно
включить SSL на порту 636 на вашем сервере LDAP (обычно это делает администратор каталога).

> **Требования к серверу.** Сервер LDAP должен быть настроен с серверным SSL-сертификатом X.509 и с
> включённым SSL. Обычно сначала нужно получить подписанный сертификат для сервера у удостоверяющего
> центра (CA), затем следовать инструкциям поставщика каталога по включению SSL. У разных поставщиков
> разные инструменты для этого.
>
> **Требования к клиенту.** Нужно убедиться, что клиент доверяет используемому серверу LDAP. Нужно
> установить сертификат сервера (или сертификат его CA) в базу доверенных сертификатов вашего JRE.
> Пример:
>
> ```
> # cd JAVA_HOME/lib/security
> # keytool -import -file server_cert.cer -keystore jssecacerts
> ```
>
> Об использовании инструментов безопасности см. трейл Security. О JSSE см. Java Secure Socket
> Extension (JSSE) Reference Guide.

```java
// Настраиваем окружение для создания начального контекста
Hashtable<String, Object> env = new Hashtable<String, Object>();
env.put(Context.INITIAL_CONTEXT_FACTORY, "com.sun.jndi.ldap.LdapCtxFactory");
env.put(Context.PROVIDER_URL, "ldap://localhost:636/o=JNDITutorial");

// Указываем SSL
env.put(Context.SECURITY_PROTOCOL, "ssl");

// Аутентификация как S. User с паролем "mysecret"
env.put(Context.SECURITY_AUTHENTICATION, "simple");
env.put(Context.SECURITY_PRINCIPAL,
        "cn=S. User, ou=NewHires,o=JNDITutorial");
env.put(Context.SECURITY_CREDENTIALS, "mysecret");

// Создаём начальный контекст
DirContext ctx = new InitialDirContext(env);

// ... делаем что-нибудь полезное с ctx
```

> **Примечание.** Если использовать SSL для подключения к серверу на порту, который SSL **не**
> использует, программа зависнет. Аналогично, если использовать обычный сокет для подключения к
> SSL-сокету сервера, приложение зависнет. Это свойство протокола SSL.

### Использование URL LDAPS

Вместо запроса SSL через свойство `Context.SECURITY_PROTOCOL` можно запросить SSL через URL LDAPS.
URL LDAPS похож на URL LDAP, но схема URL — "ldaps" вместо "ldap". Он указывает на использование SSL
при обмене с сервером LDAP.

```java
// Настраиваем окружение для создания начального контекста
Hashtable<String, Object> env = new Hashtable<String, Object>();
env.put(Context.INITIAL_CONTEXT_FACTORY, "com.sun.jndi.ldap.LdapCtxFactory");

// Указываем URL LDAPS
env.put(Context.PROVIDER_URL, "ldaps://localhost:636/o=JNDITutorial");

// Аутентификация как S. User с паролем "mysecret"
env.put(Context.SECURITY_AUTHENTICATION, "simple");
env.put(Context.SECURITY_PRINCIPAL,
        "cn=S. User, ou=NewHires, o=JNDITutorial");
env.put(Context.SECURITY_CREDENTIALS, "mysecret");

// Создаём начальный контекст
DirContext ctx = new InitialDirContext(env);

// ... делаем что-нибудь полезное с ctx
```

URL LDAPS принимаются везде, где принимаются URL LDAP. Подробности об URL LDAP и LDAPS — в JNDI
Tutorial.

### Аутентификация клиента: SSL с механизмом SASL External

SSL обеспечивает аутентификацию и другие службы безопасности на более низком уровне, чем LDAP. Если
аутентификация уже выполнена на уровне SSL, уровень LDAP может использовать эту информацию через
механизм SASL **External**. Следующий пример аналогичен предыдущему примеру с SSL, но вместо простой
аутентификации использует External. Применяя External, не нужно передавать ни принципала, ни пароль —
они берутся из SSL.

> **Требования к серверу.** Этот пример требует, чтобы сервер LDAP разрешал аутентификацию клиента на
> основе сертификата. Кроме того, сервер LDAP должен доверять (CA) получаемых клиентских сертификатов
> и уметь сопоставлять различающиеся имена владельцев в клиентских сертификатах с известными ему
> принципалами. Следуйте инструкциям поставщика каталога.
>
> **Требования к клиенту.** Этот пример требует наличия у клиента клиентского SSL-сертификата X.509.
> Более того, сертификат должен быть сохранён как первая запись ключа в файле хранилища ключей
> (keystore). Если эта запись защищена паролем, он должен совпадать с паролем хранилища ключей.

```java
// Настраиваем окружение для создания начального контекста
Hashtable<String, Object> env = new Hashtable<String, Object>(11);
env.put(Context.INITIAL_CONTEXT_FACTORY, "com.sun.jndi.ldap.LdapCtxFactory");
env.put(Context.PROVIDER_URL, "ldap://localhost:636/o=JNDITutorial");

// Принципал и учётные данные будут получены из соединения
env.put(Context.SECURITY_AUTHENTICATION, "EXTERNAL");

// Указываем SSL
env.put(Context.SECURITY_PROTOCOL, "ssl");

// Создаём начальный контекст
DirContext ctx = new InitialDirContext(env);

...
```

Чтобы запустить программу так, чтобы для аутентификации использовался сертификат клиента, нужно
передать (как системные свойства) расположение и пароль хранилища ключей с сертификатом клиента:

```
java -Djavax.net.ssl.keyStore=MyKeystoreFile \
    -Djavax.net.ssl.keyStorePassword=mysecret \
    External
```

Если хранилище ключей не предоставить, программа будет работать с анонимной аутентификацией, поскольку
на уровне SSL нет учётных данных клиента.

Этот пример показывает самый базовый способ аутентификации клиента на основе сертификата. Более
продвинутые способы возможны через написание и использование пользовательской фабрики сокетов,
обращающейся к сертификату клиента более гибким образом (например, через каталог LDAP). Следующий
раздел показывает, как использовать пользовательскую фабрику сокетов в JNDI-приложении.

### Использование пользовательских сокетов

При использовании SSL поставщик LDAP по умолчанию применяет фабрику сокетов
[`javax.net.ssl.SSLSocketFactory`](https://docs.oracle.com/javase/8/docs/api/javax/net/ssl/SSLSocketFactory.html)
для создания SSL-сокета связи с сервером, используя конфигурацию JSSE по умолчанию. JSSE можно
настроить разными способами (см. JSSE Reference Guide). Однако иногда этих настроек недостаточно и
нужен больший контроль над SSL-сокетами или сокетами вообще, используемыми поставщиком LDAP.
Например, могут понадобиться сокеты, обходящие межсетевые экраны, или JSSE-сокеты с нестандартными
политиками кэширования/получения хранилищ доверия и ключей. Чтобы задать реализацию фабрики сокетов,
используемую поставщиком LDAP, установите свойство "java.naming.ldap.factory.socket" в полностью
квалифицированное имя класса фабрики сокетов. Этот класс должен наследовать абстрактный класс
[`javax.net.SocketFactory`](https://docs.oracle.com/javase/8/docs/api/javax/net/SocketFactory.html) и
предоставлять реализацию метода `getDefault()`, возвращающего экземпляр фабрики сокетов.

Вот пример пользовательской фабрики сокетов, производящей обычные сокеты:

```java
public class CustomSocketFactory extends SocketFactory {
    public static SocketFactory getDefault() {

        System.out.println("[получение фабрики сокетов по умолчанию]");
        return new CustomSocketFactory();
    }
        ...
}
```

Обратите внимание: этот пример создаёт новый экземпляр `CustomSocketFactory` при каждом создании
нового LDAP-соединения. Для некоторых приложений и фабрик сокетов это уместно. Если же фабрику сокетов
нужно переиспользовать, метод `getDefault()` должен возвращать одиночку (singleton).

Чтобы использовать эту пользовательскую фабрику сокетов в JNDI-программе, установите свойство
"java.naming.ldap.factory.socket":

```java
// Настраиваем окружение для создания начального контекста
Hashtable<String, Object> env = new Hashtable<String, Object>();
env.put(Context.INITIAL_CONTEXT_FACTORY, "com.sun.jndi.ldap.LdapCtxFactory");
env.put(Context.PROVIDER_URL, "ldap://localhost:389/o=JNDITutorial");

// Указываем фабрику сокетов
env.put("java.naming.ldap.factory.socket", "CustomSocketFactory");

// Создаём начальный контекст
DirContext ctx = new InitialDirContext(env);

// ... делаем что-нибудь полезное с ctx
```

Свойство "java.naming.ldap.factory.socket" полезно для задания фабрики сокетов на уровне отдельного
контекста. Другой способ управлять сокетами поставщика LDAP — задать фабрику сокетов для всех сокетов
всей программы через `java.net.Socket.setSocketImplFactory()`. Этот способ менее гибок, так как
влияет на все соединения по сокетам, а не только на LDAP, и его следует применять с осторожностью.

## Дополнительные операции LDAP

Остаток урока описывает, как JNDI предоставляет возможность выполнять некоторые интересные операции
LDAP.

### Переименование объектов

Для переименования объекта в каталоге используется `Context.rename()`. В
[LDAP v2](http://www.ietf.org/rfc/rfc1777.txt) это соответствует операции «modify RDN», которая
переименовывает запись в том же контексте (то есть переименование «сиблинга»). В
[LDAP v3](http://www.ietf.org/rfc/rfc2251.txt) это соответствует операции «modify DN», аналогичной
«modify RDN», но старая и новая записи не обязаны быть в одном контексте. С помощью `Context.rename()`
можно переименовать как листовую запись, так и внутренний узел. Следующий код переименовывает
внутренний узел из "ou=NewHires" в "ou=OldHires":

```java
ctx.rename("ou=NewHires", "ou=OldHires");
```

> **Примечание.** Oracle Directory Server v5.2 не поддерживает переименование внутренних узлов. При
> запуске этого примера вы получите `ContextNotEmptyException`.

#### Переименование в другую часть DIT

В LDAP v3 запись можно переименовать в другую часть DIT (Directory Information Tree, дерево
информации каталога). Чтобы сделать это через `Context.rename()`, нужно использовать контекст,
являющийся общим предком и новой, и старой записи. Например, чтобы переименовать
"cn=C. User, ou=NewHires, o=JNDITutorial" в "cn=C. User, ou=People, o=JNDITutorial", нужно
использовать контекст с именем "o=JNDITutorial". Если запустить этот пример против сервера LDAP v2,
вы получите `InvalidNameException`, поскольку версия 2 эту возможность не поддерживает.

```java
ctx.rename("cn=C. User, ou=NewHires", "cn=C. User, ou=People");
```

> **Примечание.** Oracle Directory Server v5.2 не поддерживает переименование с разными родительскими
> узлами. При запуске этого примера на таком сервере вы получите `OperationNotSupportedException`
> (указывающее на «ошибку протокола»).

#### Сохранение старых атрибутов имени

В LDAP при переименовании записи есть опция сохранить старый RDN записи как атрибут обновлённой
записи. Например, при переименовании "cn=C. User" в "cn=Claude User" можно указать, хотите ли вы
сохранить старый RDN "cn=C. User" как атрибут.

Чтобы указать, нужно ли сохранять старый атрибут имени при `Context.rename()`, используйте свойство
окружения "java.naming.ldap.deleteRDN". Если его значение "true" (по умолчанию), старый RDN
удаляется. Если "false" — старый RDN сохраняется как атрибут обновлённой записи.

```java
// Устанавливаем свойство для сохранения RDN
env.put("java.naming.ldap.deleteRDN", "false");

// Создаём начальный контекст
DirContext ctx = new InitialDirContext(env);

// Выполняем переименование
ctx.rename("cn=C. User, ou=NewHires", "cn=Claude User,ou=NewHires");
```

### LDAP Compare

Операция LDAP «compare» позволяет клиенту спросить у сервера, есть ли у именованной записи пара
«атрибут/значение». Это позволяет серверу держать определённые пары «атрибут/значение» в секрете
(то есть не открывать их для обычного доступа через «search»), но всё же давать клиенту ограниченную
возможность их использовать. Некоторые серверы могут применять это, например, для паролей, хотя
передавать незашифрованные пароли в самой операции «compare» небезопасно.

Чтобы выполнить это в JNDI, используйте подходящим образом ограниченные аргументы методов:

* `search(Name name, String filter, SearchControls ctls)`
* `search(Name name, String filterExpr, Object[] filterArgs, SearchControls ctls)`

1. Фильтр должен иметь форму "(*name*=*value*)". Подстановочные знаки (wild cards) использовать нельзя.
2. Область поиска должна быть `SearchControls.OBJECT_SCOPE`.
3. Нужно потребовать, чтобы не возвращалось никаких атрибутов.

Если эти критерии не выполнены, методы используют LDAP-операцию «search» вместо «compare».

Вот пример, приводящий к использованию операции LDAP «compare»:

```java
// Значение атрибута
byte[] key = {(byte)0x61, (byte)0x62, (byte)0x63, (byte)0x64,
              (byte)0x65, (byte)0x66, (byte)0x67};

// Настраиваем параметры поиска
SearchControls ctls = new SearchControls();
ctls.setReturningAttributes(new String[0]);       // Не возвращать атрибуты
ctls.setSearchScope(SearchControls.OBJECT_SCOPE); // Искать только сам объект

// Вызываем search(), который использует LDAP-операцию "compare"
NamingEnumeration answer = ctx.search("cn=S. User, ou=NewHires",
                                      "(mySpecialKey={0})",
                                       new Object[]{key}, ctls);
```

Если сравнение успешно, итоговое перечисление будет содержать единственный элемент, имя которого —
пустое имя и который не содержит атрибутов.

### Результаты поиска

При использовании методов поиска интерфейса `DirContext` вы получаете обратно `NamingEnumeration`.
Каждый элемент в `NamingEnumeration` — это `SearchResult`, содержащий следующую информацию: имя,
объект, имя класса, атрибуты.

#### Имя

Каждый `SearchResult` содержит имя записи LDAP, удовлетворившей фильтру поиска. Имя записи получают
методом `getName()`. Он возвращает составное имя записи LDAP **относительно целевого контекста**
(target context). Целевой контекст — это контекст, в который разрешается параметр `name`. На языке
LDAP целевой контекст — это **базовый объект** (base object) поиска. Пример:

```java
NamingEnumeration answer = ctx.search("ou=NewHires",
    "(&(mySpecialKey={0}) (cn=*{1}))",       // Выражение фильтра
    new Object[]{key, name},                 // Аргументы фильтра
    null);                                   // Параметры поиска по умолчанию
```

Целевой контекст здесь — именованный "ou=NewHires". Имена в `SearchResult` в `answer` относительны
"ou=NewHires". Например, если `getName()` вернёт "cn=J. Duke", то его имя относительно `ctx` —
"cn=J. Duke, ou=NewHires".

Если поиск выполнен с `SearchControls.SUBTREE_SCOPE` или `SearchControls.OBJECT_SCOPE` и сам целевой
контекст удовлетворил фильтру, то возвращённое имя будет "" (пустое имя), поскольку это имя
относительно целевого контекста.

Это ещё не всё. Если поиск задействует отсылки или разыменование псевдонимов (aliases), то
соответствующие `SearchResult` будут иметь имена, не относительные целевому контексту, — это будут
URL, ссылающиеся непосредственно на запись. Чтобы определить, относительно ли возвращённое
`getName()` имя или абсолютно, используйте `isRelative()`. Если метод возвращает true, имя
относительно целевого контекста; если false — имя является URL.

Если имя — URL, и вам нужно его использовать, передайте его начальному контексту, который понимает
URL. Чтобы получить полный DN записи, используйте `NameClassPair.getNameInNamespace()`.

#### Объект

Если поиск выполнялся с запросом вернуть объект записи (`SearchControls.setReturningObjFlag()` был
вызван с true), то `SearchResult` будет содержать объект, представляющий запись. Для его получения
вызовите `getObject()`. Если ранее по этому имени LDAP был привязан объект `java.io.Serializable`,
`Referenceable` или `Reference`, то для воссоздания этого объекта используются атрибуты записи. Иначе
атрибуты записи используются для создания экземпляра `DirContext`, представляющего запись LDAP. В
любом случае поставщик LDAP вызывает `DirectoryManager.getObjectInstance()` для объекта и возвращает
результат.

#### Имя класса

Если поиск выполнялся с запросом вернуть объект записи, имя класса выводится из возвращённого объекта.
Если поиск запрашивал атрибуты, включающие получение атрибута "javaClassName" записи LDAP, то имя
класса — значение этого атрибута. Иначе имя класса — "javax.naming.directory.DirContext". Имя класса
получают методом `getClassName()`.

#### Атрибуты

При выполнении поиска возвращаемые атрибуты можно выбрать, передав параметр одному из методов
`search()` или задав параметры поиска через `SearchControls.setReturningAttributes()`. Если атрибуты
не указаны явно, возвращаются все атрибуты записи LDAP. Чтобы указать, что атрибуты возвращать не
нужно, передайте пустой массив (`new String[0]`). Для получения атрибутов записи LDAP вызовите
`getAttributes()` на `SearchResult`.

#### Управляющие элементы ответа

О том, как получить управляющие элементы ответа результата поиска, см. урок «Controls and Extensions»
в JNDI Tutorial.

### Неотправленные уведомления LDAP

LDAP v3 ([RFC 2251](http://www.ietf.org/rfc/rfc2251.txt)) определяет **неотправленное уведомление**
(unsolicited notification) — сообщение, отправляемое сервером LDAP клиенту без какого-либо запроса
со стороны клиента. В JNDI неотправленное уведомление представляется интерфейсом
`UnsolicitedNotification`.

Поскольку неотправленные уведомления отправляются сервером асинхронно, можно использовать ту же
модель событий, что и для получения уведомлений об изменениях в пространстве имён и в содержимом
объектов. Интерес к получению неотправленных уведомлений регистрируется регистрацией
`UnsolicitedNotificationListener` с `EventContext` или `EventDirContext`.

Пример реализации `UnsolicitedNotificationListener`:

```java
public class UnsolListener implements UnsolicitedNotificationListener {
    public void notificationReceived(UnsolicitedNotificationEvent evt) {
        System.out.println("received: " + evt);
    }

    public void namingExceptionThrown(NamingExceptionEvent evt) {
        System.out.println(">>> UnsolListener got an exception");
            evt.getException().printStackTrace();
    }
}
```

Пример регистрации. Релевантен только аргумент `listener` метода `EventContext.addNamingListener()`.
Параметры `name` и `scope` к неотправленным уведомлениям отношения не имеют.

```java
// Получаем контекст событий для регистрации слушателя
EventContext ctx = (EventContext)
    (new InitialContext(env).lookup("ou=People"));

// Создаём слушателя
NamingListener listener = new UnsolListener();

// Регистрируем слушателя в контексте (все цели эквивалентны)
ctx.addNamingListener("", EventContext.ONELEVEL_SCOPE, listener);
```

При запуске этой программы её нужно направить на сервер LDAP, способный генерировать неотправленные
уведомления, и подтолкнуть сервер их выдать. Иначе через минуту программа молча завершится.

Слушатель, реализующий `UnsolicitedNotificationListener`, может также реализовывать другие интерфейсы
`NamingListener`, такие как `NamespaceChangeListener` и `ObjectChangeListener`.

## Управление соединениями

JNDI предоставляет высокоуровневый интерфейс доступа к службам имён и каталогов. Соответствие между
экземпляром JNDI `Context` и нижележащим сетевым соединением может быть не «один к одному». Поставщик
службы вправе разделять и переиспользовать соединения, пока сохраняется семантика интерфейса.
Разработчику обычно не нужно знать деталей того, как экземпляры `Context` создают и используют
соединения. Эти детали полезны, когда разработчику нужно настроить (tune) программу.

Этот урок описывает, как поставщик службы LDAP использует соединения: когда соединения создаются и
как указать специальные параметры соединения (несколько серверов, тайм-ауты), как закрываются
соединения и как использовать пул соединений.

> **Примечание.** Сведения в этом уроке специфичны для поставщика службы LDAP в JDK. Поставщики LDAP
> других производителей могут использовать иные политики управления соединениями.

### Создание соединения

Есть несколько способов создания соединения. Самый распространённый — при создании начального
контекста. Когда вы создаёте `InitialContext`, `InitialDirContext` или `InitialLdapContext` с
помощью поставщика службы LDAP, соединение немедленно устанавливается с целевым сервером LDAP,
указанным в свойстве `Context.PROVIDER_URL`. Каждый раз при создании начального контекста создаётся
новое LDAP-соединение (см. раздел «Пул соединений» о том, как изменить это поведение).

Если значение свойства содержит более одного URL, то каждый URL пробуется по очереди, пока один из
них не приведёт к успешному соединению. Затем значение свойства обновляется на успешный URL.

Есть три других прямых способа создания соединения:

1. Передача URL как аргумента имени начальному контексту. Когда URL LDAP или LDAPS передаётся как
   параметр имени начальному контексту, информация из URL используется для создания нового соединения
   с сервером LDAP, независимо от того, есть ли у самого экземпляра начального контекста соединение с
   сервером LDAP. По сути, начальный контекст может быть не подключён ни к какому серверу.

2. Использование `Reference`. Когда `Reference`, содержащий URL LDAP или LDAPS, передаётся в
   `NamingManager.getObjectInstance()` или `DirectoryManager.getObjectInstance()`, новое соединение
   создаётся с использованием информации, указанной в URL.

3. При обработке отсылки (вручную или автоматически) информация из отсылки используется для создания
   нового соединения.

#### Разделяемые соединения

Экземпляры `Context` и `NamingEnumeration`, производные от одного экземпляра `Context`, разделяют
одно соединение, пока изменения в одном из экземпляров `Context` не сделают такое разделение
невозможным. Например, если вызвать `Context.lookup()`, `Context.listBindings()` или
`DirContext.search()` из начального контекста и получить обратно другие экземпляры `Context`, то все
эти экземпляры будут разделять одно соединение.

```java
// Создаём начальный контекст
DirContext ctx = new InitialDirContext(env);

// Получаем копию того же контекста
Context ctx2 = (Context)ctx.lookup("");

// Получаем дочерний контекст
Context ctx3 = (Context) ctx.lookup("ou=NewHires");
```

В этом примере `ctx`, `ctx2` и `ctx3` будут разделять одно соединение. Разделение происходит
независимо от того, как возник экземпляр `Context`. Например, экземпляр `Context`, полученный
следованием отсылке, будет разделять то же соединение, что и отсылка.

Когда вы меняете свойства окружения экземпляра `Context`, связанные с соединением (например, имя
принципала или учётные данные пользователя), то экземпляр `Context`, в котором сделаны эти изменения,
получит собственное соединение (если соединение было разделяемым). Экземпляры `Context`, производные
от него в будущем, будут разделять это новое соединение. Экземпляры, ранее разделявшие старое
соединение, не затрагиваются (они продолжают использовать старое соединение).

```java
// Создаём начальный контекст (первое соединение)
DirContext ctx = new InitialDirContext(env);

// Получаем копию того же контекста
DirContext ctx2 = (DirContext)ctx.lookup("");

// Меняем свойства аутентификации в ctx2
ctx2.addToEnvironment(Context.SECURITY_PRINCIPAL,
    "cn=C. User, ou=NewHires, o=JNDITutorial");
ctx2.addToEnvironment(Context.SECURITY_CREDENTIALS, "mysecret");

// Метод на ctx2 будет использовать новое соединение
System.out.println(ctx2.getAttributes("ou=NewHires"));
```

`ctx2` сначала разделяет соединение с `ctx`. Но когда меняются свойства принципала и пароля, он уже
не может использовать соединение `ctx`. Поставщик LDAP автоматически создаст новое соединение для
`ctx2`. Аналогично, если использовать `LdapContext.reconnect()` для смены управляющих элементов
соединения, экземпляр `Context` получит собственное соединение, если оно было разделяемым.

Если соединение экземпляра `Context` не было разделяемым (то есть от него не было производных
контекстов), то изменения его окружения или управляющих элементов соединения не приведут к созданию
нового соединения. Вместо этого все изменения, относящиеся к соединению, будут применены к
существующему соединению.

#### Тайм-ауты создания соединения

Не все создания соединений успешны. Если поставщик LDAP не может установить соединение в течение
определённого периода тайм-аута, он прерывает попытку. По умолчанию этот период равен сетевому (TCP)
тайм-ауту — порядка нескольких минут. Чтобы изменить период, используйте свойство окружения
"com.sun.jndi.ldap.connect.timeout" — строковое представление целого числа, задающего тайм-аут
соединения в миллисекундах.

```java
// Настраиваем окружение для создания начального контекста
Hashtable env = new Hashtable(11);
env.put(Context.INITIAL_CONTEXT_FACTORY,
    "com.sun.jndi.ldap.LdapCtxFactory");
env.put(Context.PROVIDER_URL, "ldap://localhost:389/o=JNDITutorial");

// Указываем тайм-аут в 5 секунд
env.put("com.sun.jndi.ldap.connect.timeout", "5000");

// Создаём начальный контекст
DirContext ctx = new InitialDirContext(env);

// делаем что-нибудь полезное с ctx
```

В этом примере, если соединение не удаётся создать за 5 секунд, будет выброшено исключение. Если
свойство `Context.PROVIDER_URL` содержит более одного URL, поставщик использует тайм-аут для каждого
URL. Например, при 3 URL и тайм-ауте 5 секунд поставщик будет ждать максимум 15 секунд суммарно.

### Закрытие соединения

Обычная сборка мусора (garbage collection) заботится об удалении экземпляров `Context`, когда они
больше не используются. Соединения, используемые собираемыми экземплярами `Context`, закрываются
автоматически. Поэтому явно закрывать соединения не нужно. Однако сетевые соединения — ограниченный
ресурс, и для некоторых программ может потребоваться контроль над их количеством и использованием.

#### Явное закрытие

Чтобы указать, что экземпляр `Context` больше не нужен, вызовите на нём `Context.close()`. Если
закрываемый экземпляр использует выделенное соединение, оно также закрывается. Если экземпляр
разделяет соединение с другими экземплярами `Context` и незавершёнными `NamingEnumeration`, то
соединение не закроется, пока `close()` не будет вызван на всех таких экземплярах `Context` и
`NamingEnumeration`.

В примере из раздела о создании соединения все три экземпляра `Context` должны быть закрыты, прежде
чем закроется нижележащее соединение:

```java
// Создаём начальный контекст
DirContext ctx = new InitialDirContext(env);

// Получаем копию того же контекста
Context ctx2 = (Context)ctx.lookup("");

// Получаем дочерний контекст
Context ctx3 = (Context) ctx.lookup("ou=NewHires");

// делаем что-нибудь полезное с ctx, ctx2, ctx3

// Закрываем контексты, когда закончили
ctx.close();
ctx2.close();
ctx3.close();
```

#### Принудительное неявное закрытие

Как упоминалось, для экземпляров `Context` и `NamingEnumeration`, вышедших из области видимости,
система времени выполнения Java со временем выполнит сборку мусора, очистив состояние, как это сделал
бы `close()`. Чтобы форсировать сборку мусора, можно использовать следующий код:

```java
Runtime.getRuntime().gc();
Runtime.getRuntime().runFinalization();
```

В зависимости от состояния программы это может вызвать серьёзную (временную) деградацию
производительности. Если нужно гарантировать закрытие соединений, отслеживайте экземпляры `Context`
и закрывайте их явно.

#### Обнаружение закрытия соединения

У серверов LDAP часто есть период тайм-аута простоя, после которого они закрывают неиспользуемые
соединения. Когда вы затем вызовете метод на экземпляре `Context`, использующем такое соединение,
метод выбросит `CommunicationException`. Чтобы обнаружить, когда сервер закрывает соединение,
используемое экземпляром `Context`, зарегистрируйте `UnsolicitedNotificationListener` с экземпляром
`Context`. Пример показан в разделе «Неотправленные уведомления LDAP». Хотя пример предназначен для
получения неотправленных уведомлений от сервера, его можно использовать и для обнаружения закрытия
соединения сервером. После запуска программы остановите сервер LDAP и наблюдайте, как вызывается
метод слушателя `namingExceptionThrown()`.

### Пул соединений (Connection Pooling)

Ещё один тип разделения соединений, поддерживаемый поставщиком службы LDAP, называется **пулом
соединений** (connection pooling). При таком разделении поставщик LDAP поддерживает пул (возможно,
ранее использованных) соединений и назначает их экземплярам `Context` по мере необходимости. Когда
экземпляр `Context` завершает работу с соединением (закрыт или собран сборщиком мусора), соединение
возвращается в пул для будущего использования. Это разделение последовательное: соединение
извлекается из пула, используется, возвращается в пул, а затем снова извлекается из пула для другого
экземпляра `Context`.

Пул соединений поддерживается отдельно для каждой системы времени выполнения Java. В некоторых
ситуациях пул может значительно повысить производительность. Например, для обработки ответа поиска с
четырьмя отсылками на один и тот же сервер LDAP при использовании пула нужно лишь одно соединение.
Без пула такой сценарий потребовал бы четырёх отдельных соединений.

#### Как использовать пул соединений

Пул запрашивается добавлением свойства "com.sun.jndi.ldap.connect.pool" к свойствам окружения,
передаваемым конструктору начального контекста:

```java
// Настраиваем окружение для создания начального контекста
Hashtable env = new Hashtable(11);
env.put(Context.INITIAL_CONTEXT_FACTORY, "com.sun.jndi.ldap.LdapCtxFactory");
env.put(Context.PROVIDER_URL, "ldap://localhost:389/o=JNDITutorial");

// Включаем пул соединений
env.put("com.sun.jndi.ldap.connect.pool", "true");

// Создаём один начальный контекст (берём соединение из пула)
DirContext ctx = new InitialDirContext(env);

// делаем что-нибудь полезное с ctx

// Закрываем контекст, когда закончили
ctx.close();   // Возвращаем соединение в пул

// Создаём другой начальный контекст (берём соединение из пула)
DirContext ctx2 = new InitialDirContext(env);

// делаем что-нибудь полезное с ctx2

// Закрываем контекст, когда закончили
ctx2.close();   // Возвращаем соединение в пул
```

Этот пример создаёт два начальных контекста подряд. Второй переиспользует соединение первого. Чтобы
наблюдать, как соединения извлекаются и возвращаются в пул, используйте командную строку:

```
#java -Dcom.sun.jndi.ldap.connect.pool.debug=fine UsePool
```

Это даст вывод вида:

```
Create com.sun.jndi.ldap.LdapClient@5d173[localhost:389]
Use com.sun.jndi.ldap.LdapClient@5d173
{ou=ou: NewHires, objectclass=objectClass: top, organizationalUnit}
Release com.sun.jndi.ldap.LdapClient@5d173
Use com.sun.jndi.ldap.LdapClient@5d173
{ou=ou: People, objectclass=objectClass: top, organizationalunit}
Release com.sun.jndi.ldap.LdapClient@5d173
```

Решать, когда и где использовать пул, можно, включая или опуская свойство
"com.sun.jndi.ldap.connect.pool", то есть управлять пулом на уровне **отдельного контекста**.
Поставщик LDAP отслеживает, используется ли соединение, по сигналам приложения. Он считает, что
приложение, удерживающее открытый хэндл контекста, использует соединение. Поэтому, чтобы поставщик
LDAP правильно управлял пулом, нужно прилежно вызывать `Context.close()` на ненужных контекстах.

Плохие соединения автоматически обнаруживаются и удаляются из пула поставщиком LDAP. Вероятность
того, что контекст получит плохое соединение, одинакова независимо от того, используется ли пул.

#### Тайм-аут создания (для пула)

Пул соединений может быть ограничен по размеру (см. раздел «Конфигурация пула соединений»). Когда пул
включён и доступного соединения в пуле нет, приложение блокируется в ожидании. Свойство окружения
"com.sun.jndi.ldap.connect.timeout" задаёт, сколько ждать соединение из пула. Если свойство опущено,
приложение будет ждать бесконечно. Это же свойство задаёт период тайм-аута установления
LDAP-соединения (см. раздел «Создание соединения»).

#### Когда НЕ использовать пул!!

Пулируемые соединения предназначены для переиспользования. Поэтому, если вы планируете выполнять на
экземпляре `Context` операции, способные изменить состояние нижележащего соединения, пул для такого
контекста использовать не следует. Например, если вы планируете вызвать расширенную операцию Start
TLS на экземпляре `Context` или изменить свойства, связанные с безопасностью (такие как
"java.naming.security.principal" или "java.naming.security.protocol"), после создания начального
контекста, пул для такого контекста использовать не следует, поскольку поставщик LDAP не отслеживает
такие изменения состояния. Использование пула в таких ситуациях может скомпрометировать безопасность
вашего приложения.

### Конфигурация пула соединений

Пул соединений настраивается и поддерживается для каждой системы времени выполнения Java отдельно.
Соединения не разделяются между разными системами времени выполнения. Чтобы использовать пул, никакой
конфигурации не требуется. Конфигурация нужна, только если нужно настроить, как именно выполняется
пул, — например, контролировать размер пулов и какие типы соединений пулируются.

Пул настраивается рядом **системных свойств** (не свойств окружения!) при запуске программы. Эти
свойства затрагивают все запросы пула. Вот пример командной строки, задающей максимальный размер
пула 20, предпочтительный размер 10 и тайм-аут простоя в минуту:

```
# java -Dcom.sun.jndi.ldap.connect.pool.maxsize=20 \
       -Dcom.sun.jndi.ldap.connect.pool.prefsize=10 \
       -Dcom.sun.jndi.ldap.connect.pool.timeout=60000 \
    UsePool
```

#### Системные свойства конфигурации пула соединений

| Имя системного свойства | Описание | По умолчанию |
|---|---|---|
| com.sun.jndi.ldap.connect.pool.authentication | Список типов аутентификации соединений (через пробел), которые можно пулировать. Допустимые типы: "none", "simple", "DIGEST-MD5". | "none simple" |
| com.sun.jndi.ldap.connect.pool.debug | Строка, указывающая уровень отладочного вывода. Допустимые значения: "fine" (трассировка создания и удаления соединений) и "all" (вся отладочная информация). | |
| com.sun.jndi.ldap.connect.pool.initsize | Строковое представление целого: число соединений на одну идентичность соединения, создаваемых при первоначальном создании соединения для этой идентичности. | 1 |
| com.sun.jndi.ldap.connect.pool.maxsize | Строковое представление целого: максимальное число соединений на одну идентичность, которые можно поддерживать одновременно. | без максимума |
| com.sun.jndi.ldap.connect.pool.prefsize | Строковое представление целого: предпочтительное число соединений на одну идентичность, которое следует поддерживать одновременно. | без предпочтительного размера |
| com.sun.jndi.ldap.connect.pool.protocol | Список типов протоколов соединений (через пробел), которые можно пулировать. Допустимые типы: "plain" и "ssl". | "plain" |
| com.sun.jndi.ldap.connect.pool.timeout | Строковое представление целого: число миллисекунд, которое простаивающее соединение может оставаться в пуле, не будучи закрытым и удалённым из пула. | без тайм-аута |

#### Что попадает в пул

Когда вы запрашиваете для экземпляра `Context` использование пула через свойство окружения
"com.sun.jndi.ldap.connect.pool", используемое соединение может пулироваться, а может и нет. Правило
по умолчанию: обычные (не-SSL) соединения, использующие простую или отсутствующую аутентификацию,
разрешено пулировать. Это правило можно изменить, включив SSL-соединения и тип аутентификации
DIGEST-MD5, через системные свойства. Чтобы разрешить пулировать и обычные, и SSL-соединения,
установите "com.sun.jndi.ldap.connect.pool.protocol" в "plain ssl". Чтобы разрешить пулировать
соединения с типами аутентификации anonymous (none), simple и DIGEST-MD5, установите
"com.sun.jndi.ldap.connect.pool.authentication" в "none simple DIGEST-MD5".

Есть пара свойств окружения, которые автоматически дисквалифицируют экземпляр `Context` от
использования пулируемого соединения. Экземпляр `Context` не может использовать пулируемое
соединение, если у него установлено свойство "java.naming.ldap.factory.socket" (пользовательский
класс фабрики сокетов), или "java.naming.security.sasl.callback" (пользовательский обработчик
callback), или "com.sun.jndi.ldap.trace.ber" (включение трассировки протокола).

#### Как пулируются соединения

Когда экземпляр `Context` запрашивает пулируемое соединение, поставщик LDAP определяет, можно ли
удовлетворить запрос существующим пулируемым соединением. Для этого он назначает каждому пулируемому
соединению **идентичность соединения** (connection identity) и проверяет, имеет ли входящий запрос ту
же идентичность, что и одно из его пулируемых соединений.

Идентичность соединения — это набор параметров, необходимых для создания (возможно,
аутентифицированного) LDAP-соединения. Её состав зависит от типа аутентификации запроса:

| Тип аутентификации | Содержимое идентичности соединения |
|---|---|
| none | • управляющие элементы соединения<br>• имя хоста, номер порта (как указано в "java.naming.provider.url", отсылке или URL, переданном начальному контексту)<br>• содержимое свойств: java.naming.security.protocol, java.naming.ldap.version |
| simple | • вся информация для none<br>• содержимое свойств: java.naming.security.principal, java.naming.security.credentials |
| DIGEST-MD5 | • вся информация для simple<br>• содержимое свойств: java.naming.security.sasl.authorizationId, java.naming.security.sasl.realm, javax.security.sasl.qop, javax.security.sasl.strength, javax.security.sasl.server.authentication, javax.security.sasl.maxbuffer, javax.security.sasl.policy.noplaintext, javax.security.sasl.policy.noactive, javax.security.sasl.policy.nodictionary, javax.security.sasl.policy.noanonymous, javax.security.sasl.policy.forward, javax.security.sasl.policy.credentials |

#### Размеры пула

Поставщик LDAP поддерживает пулы соединений; каждый пул содержит соединения (используемые или
простаивающие) с одной и той же идентичностью. На управление каждым пулом влияют три размера. Эти
размеры глобальны и затрагивают все пулы.

**Начальный размер пула** (initial pool size) — число соединений на идентичность, которое поставщик
службы LDAP создаёт при первом создании пула (когда приложение впервые запрашивает пулируемое
соединение для этой идентичности). Аутентификация каждого соединения в пуле выполняется по требованию,
по мере использования соединения. По умолчанию начальный размер — 1, изменяется системным свойством
"com.sun.jndi.ldap.connect.pool.initsize". Обычно используется при старте приложения для «прогрева»
пула определённым числом соединений к серверу.

**Максимальный размер пула** (maximum pool size) — максимальное число соединений на идентичность,
которое поставщик LDAP может поддерживать одновременно. И используемые, и простаивающие соединения
учитываются в этом числе. Когда размер пула достигает этого значения, новое соединение для
соответствующей идентичности нельзя создать, пока соединение не будет удалено из пула (то есть
физически закрыто). Когда размер пула достиг максимума и все соединения используются, запрос
приложения на соединение из этого пула блокируется, пока соединение в пуле не освободится или не будет
удалено. Максимальный размер 0 означает отсутствие максимума: запрос пулируемого соединения использует
существующее простаивающее или вновь созданное пулируемое соединение.

**Предпочтительный размер пула** (preferred pool size) — предпочтительное число соединений на
идентичность, которое поставщик LDAP должен поддерживать. И используемые, и простаивающие соединения
учитываются. Когда приложение запрашивает пулируемое соединение и размер пула меньше предпочтительного,
поставщик LDAP создаст и использует новое пулируемое соединение, независимо от наличия простаивающего.
Когда приложение завершает работу с пулируемым соединением (вызвав `Context.close()` на всех
контекстах, разделяющих соединение) и размер пула больше предпочтительного, поставщик LDAP закроет и
удалит соединение из пула. Предпочтительный размер 0 означает отсутствие предпочтительного размера:
запрос приведёт к созданию нового соединения только при отсутствии простаивающих.

Максимальный размер пула переопределяет и начальный, и предпочтительный размеры. Например, установка
предпочтительного размера больше максимального фактически равна установке его в максимальный размер.

#### Простаивающие соединения

Когда приложение завершает работу с пулируемым соединением (вызвав `Context.close()` на всех
контекстах, разделяющих соединение), нижележащее пулируемое соединение помечается как простаивающее
(idle), ожидая переиспользования. По умолчанию простаивающие соединения остаются в пуле бесконечно,
пока не будут собраны сборщиком мусора. Если установлено системное свойство
"com.sun.jndi.ldap.connect.pool.timeout", поставщик LDAP автоматически закроет и удалит из пула
соединения, простаивавшие дольше заданного периода.

## Часто задаваемые вопросы (FAQ)

Этот урок отвечает на часто задаваемые вопросы при использовании JNDI для доступа к службам LDAP.
Некоторые распространённые проблемы рассмотрены в разделе «Trouble Shooting Tips» урока «Naming and
Directory Operations».

### Контексты

**1. Безопасен ли контекст для многопоточного доступа, или мне нужно блокировать/синхронизировать
доступ к контексту?**

Ответ зависит от реализации, поскольку интерфейсы `Context` и `DirContext` не задают требований к
синхронизации. Реализация LDAP в JDK оптимизирована для однопоточного доступа. Если несколько потоков
обращаются к одному экземпляру `Context`, каждый поток должен блокировать этот экземпляр при
использовании. Это касается и любого `NamingEnumeration`, производного от того же экземпляра
`Context`. Однако несколько потоков могут обращаться к **разным** экземплярам `Context` (даже
производным от одного начального контекста) одновременно без блокировок.

**2. Почему поставщик LDAP игнорирует мои свойства окружения безопасности, если я не задаю свойство
`Context.SECURITY_CREDENTIALS` или задаю его пустой строкой?**

Если передать пустую строку, пустой массив байтов/символов или null в свойство
`Context.SECURITY_CREDENTIALS`, произойдёт анонимный bind, даже если `Context.SECURITY_AUTHENTICATION`
было установлено в "simple". Это потому, что при простой аутентификации LDAP требует непустой пароль.
Если пароль не предоставлен, протокол автоматически переключает аутентификацию на "none".

**3. Почему я постоянно получаю `CommunicationException` при попытке создать начальный контекст?**

Возможно, вы общаетесь с сервером, поддерживающим только LDAP v2. См. урок Miscellaneous в JNDI
Tutorial о том, как задать номер версии.

**4. Как мне трассировать сообщения LDAP?**

Попробуйте свойство окружения "com.sun.jndi.ldap.trace.ber". Если его значение — экземпляр
`java.io.OutputStream`, то информация трассировки о BER-буферах, отправляемых и получаемых
поставщиком LDAP, пишется в этот поток. Если значение null — трассировка не пишется. Например,
следующий код направит вывод трассировки в `System.err`:

```
env.put("com.sun.jndi.ldap.trace.ber", System.err);
```

**5. Как использовать другой механизм аутентификации, например Kerberos?**

Следуйте инструкциям раздела «GSS-API/Kerberos v5 Authentication» в JNDI Tutorial. Для других
механизмов см. раздел «Using Arbitrary SASL Mechanisms» в JNDI Tutorial.

**6. Нужно ли включать SSL при смене пароля?**

Зависит от используемого сервера каталога. Некоторые серверы не позволяют менять пароль без
включённого SSL, некоторые позволяют. Хорошо иметь SSL включённым, чтобы пароль был защищён в канале
связи.

### Атрибуты

**7. Я запрашиваю один атрибут, а получаю другой. Почему?**

Используемое имя атрибута может быть синонимом другого атрибута. В этом случае сервер LDAP может
вернуть каноническое имя атрибута вместо предоставленного вами. В возвращённых сервером `Attributes`
нужно использовать каноническое имя вместо синонима. Например, "fax" может быть синонимом
канонического "facsimiletelephonenumber".

**8. Как узнать тип значения атрибута?**

Значение атрибута может быть либо `java.lang.String`, либо `byte[]`. О том, у каких атрибутов значения
возвращаются как `byte[]`, см. раздел Miscellaneous в JNDI Tutorial. Программно можно использовать
оператор `instanceof` для проверки полученного значения.

**9. Как получить значение атрибута в форме, отличной от String или массива байтов?**

Сейчас нельзя. Поставщик LDAP возвращает значения атрибутов только как `java.lang.String` или
`byte[]`.

**10. Почему «\*» в значении атрибута не работает в моём поиске как ожидается?**

При использовании формы `search(Name name, Attributes matchingAttrs)` значения атрибутов трактуются
буквально (literals): ожидается, что атрибут в записи каталога содержит ровно это значение. Чтобы
использовать подстановочные знаки, используйте строковые формы фильтра `search()`:
`search(Name name, String filter, SearchControls ctls)` и
`search(Name name, String filterExpr, Object[] filterArgs, SearchControls ctls)`. Для последней формы
подстановочные знаки должны быть в аргументе `filterExpr`, а не в `filterArgs` (значения в
`filterArgs` тоже трактуются буквально).

### Поиск

**11. Почему подстановочные знаки в фильтрах поиска не всегда работают?**

Подстановочный знак до или после значения атрибута (как в "attr=\*I\*") указывает серверу искать
совпадающие значения по правилу подстрокового сопоставления (substring matching rule) атрибута. Если
в определении атрибута нет правила подстрокового сопоставления, сервер не сможет найти атрибут.
Придётся искать по фильтру равенства или «present».

**12. Почему я получаю только *n* записей, хотя знаю, что в каталоге их больше?**

Некоторые серверы настроены ограничивать число возвращаемых записей. Другие также ограничивают число
записей, проверяемых при поиске. Проверьте конфигурацию сервера.

**13. Как передавать управляющие элементы (controls) с поиском?**

Управляющие элементы в этом руководстве не рассматриваются. См. JNDI Tutorial.

**14. Как узнать, сколько результатов поиска я получил?**

Нужно вести подсчёт самостоятельно при перечислении результатов. LDAP эту информацию не предоставляет.

### Имена

**15. Почему я получаю пустую строку как имя в моём `SearchResult`?**

`getName()` всегда возвращает имя **относительно целевого контекста** поиска. Если целевой контекст
удовлетворяет фильтру, возвращённое имя будет "" (пустое имя), поскольку это имя относительно целевого
контекста. См. раздел «Результаты поиска».

**16. Почему я получаю строку URL как имя в моём `SearchResult`?**

Запись LDAP была получена следованием псевдониму или отсылке, поэтому её имя — это URL. См. раздел
«Результаты поиска».

**17. Какого типа аргумент `Name`, передаваемый методам `Context` и `DirContext`, — `CompoundName`
или `CompositeName`?**

Строковые формы принимают строковое представление составного имени (composite name). Использование
строкового имени эквивалентно вызову `new CompositeName(stringName)` и передаче результата методу.
Аргумент `Name` может быть любым объектом, реализующим интерфейс `Name`. Если это экземпляр
`CompositeName`, имя трактуется как составное; иначе — как составное компонентное (compound).

**18. Можно ли передать методам `Context` имя, полученное от `NameParser`?**

Да. `NameParser.parse()` возвращает compound-имя, реализующее интерфейс `Name`. Это имя можно
передать методам `Context`, которые интерпретируют его как compound-имя.

**19. Какова связь между именем для `Context.SECURITY_PRINCIPAL` и каталогом?**

Имя принципала можно считать происходящим из другого пространства имён, нежели каталог. См.
[RFC 2829](http://www.ietf.org/rfc/rfc2829.txt) и раздел «Безопасность» о механизмах аутентификации
LDAP. Поставщик LDAP в JDK принимает строковое имя принципала, которое передаёт серверу LDAP
напрямую. Одни серверы принимают DN, другие поддерживают схемы из RFC 2829.

**20. Почему в именах, читаемых из каталога, есть странные кавычки и экранирование?**

Парсер имён LDAP в JDK консервативен в отношении правил кавычек, но всё же выдаёт «корректные» имена.
Кроме того, имена записей, возвращаемые `NamingEnumeration`, — это составные имена, которые можно
передать обратно методам `Context` и `DirContext`. Поэтому, если имя содержит символ, конфликтующий с
синтаксисом составного имени (например, прямой слэш «/»), поставщик LDAP применит кодирование, чтобы
символ слэша трактовался как часть имени LDAP, а не как разделитель составного имени. Начните
использовать классы `LdapName` и `Rdn`, упрощающие манипуляцию именами.

**21. Как получить полный DN записи LDAP?**

Используйте `NameClassPair.getNameInNamespace()`.
