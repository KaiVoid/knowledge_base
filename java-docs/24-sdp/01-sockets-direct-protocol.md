# Урок 1. Sockets Direct Protocol (SDP)

**Трейл:** Sockets Direct Protocol · **Оригинал:** [Sockets Direct Protocol](https://docs.oracle.com/javase/tutorial/sdp/sockets/index.html)
**Связанные области:** [[17-rest-web]] · **Вопросы:** rest-web

Для сред высокопроизводительных вычислений (*high performance computing*) обязательным требованием
является способность быстро и эффективно перемещать данные по сети. Такие сети обычно описываются
как требующие высокой пропускной способности (*high throughput*) и низкой задержки (*low latency*).
**Высокая пропускная способность** означает окружение, способное обеспечивать большой объём
вычислительной обработки в течение длительного времени. **Низкая задержка** означает минимальную
задержку между обработкой входных данных и выдачей результата — такую, какую ожидают от приложений
реального времени.

В таких окружениях обычная работа с сетью через сокетные потоки (*socket streams*) может создавать
узкие места при перемещении данных. Технология InfiniBand (IB), представленная в 1999 году
ассоциацией [InfiniBand Trade Association](http://www.infinibandta.org/), была создана для решения
задач высокопроизводительных вычислений. Одна из важнейших возможностей IB — прямой удалённый
доступ к памяти (*Remote Direct Memory Access*, RDMA). RDMA позволяет перемещать данные напрямую
из памяти одного компьютера в память другого, минуя операционные системы обоих компьютеров, что
даёт существенный выигрыш в производительности.

**Sockets Direct Protocol** (SDP) — сетевой протокол, разработанный для поддержки потоковых
соединений (*stream connections*) поверх инфраструктуры InfiniBand. Поддержка SDP была введена в
выпуске JDK 7 платформы Java Platform, Standard Edition («платформа Java SE») для приложений,
развёртываемых в операционной системе Solaris («Solaris OS») и в операционных системах Linux.
Solaris OS поддерживает SDP и InfiniBand начиная с Solaris 10 5/08. В Linux пакет InfiniBand
называется OFED (OpenFabrics Enterprise Distribution). Выпуск JDK 7 поддерживает версии 1.4.2 и
1.5 пакета OFED.

## Обзор (Overview)

Поддержка SDP по сути является технологией обхода TCP (*TCP bypass*).

Когда SDP включён и приложение пытается открыть TCP-соединение, механизм TCP обходится, и обмен
данными идёт напрямую в сеть IB. Например, когда ваше приложение пытается привязаться (*bind*) к
TCP-адресу, нижележащее программное обеспечение, исходя из информации в конфигурационном файле,
решает, следует ли выполнить повторную привязку к протоколу SDP. Это может происходить во время
процесса привязки (*binding*) или процесса подключения (*connecting*), но происходит лишь один раз
для каждого сокета.

Чтобы воспользоваться протоколом SDP, никаких изменений API в вашем коде не требуется: реализация
прозрачна и поддерживается классическим сетевым пакетом (`java.net`) и пакетом New I/O
(`java.nio.channels`). Список классов, поддерживающих протокол SDP, см. в разделе
«Поддерживаемые API Java».

По умолчанию поддержка SDP отключена. Шаги для включения поддержки SDP:

- создать конфигурационный файл SDP;
- задать системное свойство, указывающее местоположение конфигурационного файла.

## Создание конфигурационного файла SDP (Creating an SDP Configuration File)

Конфигурационный файл SDP — это текстовый файл, и вы сами решаете, где в файловой системе он будет
находиться. Каждая строка конфигурационного файла является либо комментарием, либо правилом.
Комментарий обозначается символом решётки (`#`) в начале строки; всё, что следует за символом
решётки, игнорируется.

Существуют два типа правил:

- Правило **bind** (привязки) указывает, что транспорт протокола SDP должен использоваться, когда
  TCP-сокет привязывается к адресу и порту, соответствующим правилу.
- Правило **connect** (подключения) указывает, что транспорт протокола SDP должен использоваться,
  когда непривязанный TCP-сокет пытается подключиться к адресу и порту, соответствующим правилу.

Правило имеет следующую форму:

```text
("bind"|"connect")1*LWSP-char(hostname|ipaddress)["/"prefix])1*LWSP-char("*"|port)["-"("*"|port)]
```

**Расшифровка нотации:**

*1\*LWSP-char* означает, что любое количество символов линейного пропуска (*linear whitespace* —
табуляции или пробелы) может разделять токены. Квадратные скобки обозначают необязательный текст.
Нотация (*xxx* | *yyy*) означает, что токен будет включать либо *xxx*, либо *yyy*, но не оба сразу.
Символы в кавычках обозначают буквальный текст.

Первое ключевое слово указывает, является ли правило правилом **bind** или **connect**. Следующий
токен задаёт либо имя хоста, либо буквальный IP-адрес. Когда вы задаёте буквальный IP-адрес, можно
также указать префикс, который обозначает диапазон IP-адресов. Третий и последний токен — это номер
порта или диапазон номеров портов.

Рассмотрим следующую нотацию в этом примере конфигурационного файла:

```text
# Использовать SDP при привязке к 192.0.2.1
bind 192.0.2.1 *

# Использовать SDP при подключении ко всем прикладным службам на 192.0.2.*
connect 192.0.2.0/24     1024-*

# Использовать SDP при подключении к http-серверу или базе данных на examplecluster
connect examplecluster.example.com   80
connect examplecluster.example.com   3306
```

Первое правило в примере файла указывает, что SDP используется для любого порта (`*`) на локальном
IP-адресе `192.0.2.1`. Правило **bind** добавляется для каждого локального адреса, назначенного
адаптеру InfiniBand. (*Адаптер InfiniBand* — это эквивалент сетевой карты (*network interface card*,
NIC) для InfiniBand.) Если бы у вас было несколько IB-адаптеров, вы бы использовали отдельное
правило **bind** для каждого адреса, назначенного этим адаптерам.

Второе правило в примере файла указывает, что при подключении к `192.0.2.*` и целевом порту 1024
или выше используется SDP. Префикс `/24` у IP-адреса означает, что первые 24 бита из 32-битного
IP-адреса должны совпадать с указанным адресом. Каждая часть IP-адреса занимает 8 бит, поэтому
24 бита означают, что IP-адрес должен совпадать с `192.0.2`, а последний байт может быть любым.
Нотация `-*` у токена порта означает «и выше». Диапазон портов, например 1024—2056, также был бы
допустим и включал бы концевые точки указанного диапазона.

Последние правила в примере файла задают имя хоста (`examplecluster`): сначала с портом,
назначенным http-серверу (80), а затем с портом, назначенным базе данных (3306). В отличие от
буквального IP-адреса, имя хоста может разрешаться в несколько адресов. Когда вы задаёте имя хоста,
оно соответствует всем адресам, на которые это имя хоста зарегистрировано в службе имён.

## Включение протокола SDP (Enabling the SDP Protocol)

По умолчанию поддержка SDP отключена. Чтобы включить поддержку SDP, задайте системное свойство
`com.sun.sdp.conf`, указав местоположение конфигурационного файла. Следующий пример запускает
приложение с конфигурационным файлом по имени `sdp.conf`:

```bash
% java -Dcom.sun.sdp.conf=sdp.conf -Djava.net.preferIPv4Stack=true ExampleApplication
```

*ExampleApplication* — это клиентское приложение, пытающееся подключиться к IB-адаптеру.

Обратите внимание, что в этом примере задаётся ещё одно системное свойство —
`java.net.preferIPv4Stack`. Подробнее о том, зачем используется это свойство, см. в разделе
«Технические вопросы SDP».

## Отладка SDP (Debugging SDP)

Вы можете включить отладочные сообщения с помощью флага `-Dcom.sun.sdp.debug`. Если вы укажете
файл, сообщения будут выводиться в этот файл. В противном случае сообщения будут печататься в
стандартный вывод.

В первом примере показаны образцы сообщений, выводимых в стандартный вывод:

```bash
% java -Dcom.sun.sdp.conf=sdp.conf -Dcom.sun.sdp.debug ExampleApplicaton
BIND to 192.0.2.1:5000 (socket converted to SDP protocol)
CONNECT to 129.156.232.160:80 (no match)
CONNECT to 192.0.2.2:80 (socket converted to SDP protocol)
```

Во втором примере показан вывод, перенаправленный в файл по имени `debug.log`:

```bash
% java -Dcom.sun.sdp.conf=sdp.conf -Dcom.sun.sdp.debug=debug.log ExampleApplication
[1] 27310
% tail -f debug.log
BIND to 192.0.2.1:5000 (socket converted to SDP protocol)
```

## Технические вопросы SDP (Technical Issues with SDP)

### Несовместимость IPv4 и IPv6

Internet Protocol версии 4 (IPv4) долгое время был отраслевым стандартом версии интернет-протокола
(IP) для доставки данных по сети Интернет. Internet Protocol версии 6 (IPv6) — это интернет-протокол
сетевого уровня следующего поколения. Сегодня используются обе версии IP.

Адреса IPv4 имеют длину 32 бита, записываются в десятичном формате и разделяются точками. Адреса
IPv6 имеют длину 128 бит, записываются в шестнадцатеричном формате и разделяются двоеточиями.
Адреса IPv4 нельзя использовать в IPv6 как есть, однако IPv6 поддерживает специальный класс
адресов — адрес IPv4-mapped (*IPv4-mapped address*). В адресе IPv4-mapped первые 80 бит установлены
в ноль, следующие 16 бит установлены в 1, а последние 32 бита представляют адрес IPv4.

Например, вот один и тот же IP-адрес, выраженный в обоих форматах:

```text
Адрес IPv4                  Адрес IPv4-mapped (для использования в IPv6)
192.0.2.1                   ::ffff:192.0.2.1
```

По умолчанию, если IPv6 включён на каком-либо из IB-адаптеров, платформа Java использует IPv6.
Однако адреса IPv4-mapped в настоящее время недоступны в Solaris OS и под Linux. По этой причине,
если вы хотите использовать формат адреса IPv4 под JDK 7, необходимо задать свойство
`java.net.preferIPv4Stack`, как показано в этом примере:

```bash
% java -Dcom.sun.sdp.conf=sdp.conf -Djava.net.preferIPv4Stack=true MyApplication
```

### Ошибки (Bugs)

В ранней реализации InfiniBand было обнаружено несколько ошибок. Эти ошибки исправлены в выпуске
Solaris 10 10/09. Убедитесь, что вы используете как минимум этот выпуск.

## Поддержка в Solaris и Linux (Solaris and Linux Support)

### Solaris 10

Чтобы проверить, что SDP включён, используйте команду `sdpadm`(1M):

```bash
% /usr/sbin/sdpadm status
SDP is Enabled
```

Вы можете использовать команду `grep` для поиска строки «ibd» в файле `/etc/path_to_inst`, чтобы
просмотреть список IB-адаптеров, поддерживаемых в вашей сети.

Другие команды, которые могут оказаться полезными: `ib`(7D), `ibd`(7D) и `sdp`(7D).

### Solaris 11

Вы можете использовать следующую команду для получения сведений о канале раздела InfiniBand
(*InfiniBand partition link*):

```bash
% dladm show-port -o LINK
```

Другие команды, которые могут оказаться полезными: `ib`(7D), `ibd`(7D) и `sdp`(7D).

### Linux

Вы можете использовать следующую команду для получения списка адаптеров InfiniBand:

```bash
% egrep "^[ \t]+ib" /proc/net/dev
```

## Поддерживаемые API Java (Supported Java APIs)

Все API, использующие TCP, могут использовать SDP, в частности следующие классы:

- пакет `java.net`:
  - `Socket`
  - `ServerSocket`
- пакет `java.nio.channels`:
  - `SocketChannel`
  - `ServerSocketChannel`
  - `AsynchronousSocketChannel`
  - `AsynchronousServerSocketChannel`

Когда поддержка SDP включена, всё просто работает без каких-либо изменений в вашем коде.
Перекомпиляция не нужна. Однако важно понимать, что сокет привязывается только один раз.
Подключение является неявной привязкой (*implicit bind*). Поэтому, если сокет не был привязан
ранее и вызывается `connect`, привязка происходит именно в этот момент.

Например, рассмотрим следующий фрагмент кода:

```java
AsynchronousSocketChannel ch = AsynchronousSocketChannel.open();
ch.bind(local);
Future<Void> result = ch.connect(remote);
```

В этом фрагменте асинхронный сокетный канал привязывается к локальному TCP-адресу при вызове
`bind` на сокете. Затем код пытается подключиться к удалённому адресу, используя тот же сокет.
Если удалённый адрес использует InfiniBand, как указано в конфигурационном файле, соединение
**не** будет преобразовано в SDP, потому что сокет был привязан ранее.
