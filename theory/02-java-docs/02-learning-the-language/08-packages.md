## Создание и использование пакетов

Чтобы типы было проще находить и использовать, чтобы избегать конфликтов имён и управлять
доступом, программисты объединяют группы связанных типов в **пакеты** (*packages*).

> **Определение.** *Пакет* (*package*) — это группировка связанных типов, обеспечивающая защиту
> доступа и управление пространством имён (*name space*). Под *типами* здесь понимаются классы,
> интерфейсы, перечисления (*enumerations*) и аннотационные типы (*annotation types*).
> Перечисления и аннотационные типы — это особые виды классов и интерфейсов соответственно,
> поэтому в этом уроке *типы* часто называются просто *классами и интерфейсами*.

Типы, входящие в платформу Java, являются членами различных пакетов, которые группируют классы
по назначению: базовые классы находятся в `java.lang`, классы для чтения и записи (ввод и вывод) —
в `java.io`, и так далее. Свои типы вы тоже можете помещать в пакеты.

Предположим, вы пишете группу классов, представляющих графические объекты — окружности,
прямоугольники, линии и точки. Также вы пишете интерфейс `Draggable`, который реализуют классы,
если их можно перетаскивать мышью.

```java
// в файле Draggable.java
public interface Draggable {
    ...
}

// в файле Graphic.java
public abstract class Graphic {
    ...
}

// в файле Circle.java
public class Circle extends Graphic
    implements Draggable {
    . . .
}

// в файле Rectangle.java
public class Rectangle extends Graphic
    implements Draggable {
    . . .
}

// в файле Point.java
public class Point extends Graphic
    implements Draggable {
    . . .
}

// в файле Line.java
public class Line extends Graphic
    implements Draggable {
    . . .
}
```

Эти классы и интерфейс стоит объединить в пакет по ряду причин, среди которых:

- Вы и другие программисты легко поймёте, что эти типы связаны между собой.
- Вы и другие программисты будете знать, где искать типы, отвечающие за графику.
- Имена ваших типов не будут конфликтовать с именами типов в других пакетах, потому что пакет
  создаёт новое пространство имён.
- Вы можете разрешить типам внутри пакета неограниченный доступ друг к другу и при этом
  ограничить доступ для типов вне пакета.

## Создание пакета

Чтобы создать пакет, нужно выбрать для него имя (соглашения об именовании обсуждаются в следующем
разделе) и поместить оператор `package` с этим именем в начало *каждого исходного файла*,
содержащего типы (классы, интерфейсы, перечисления и аннотационные типы), которые вы хотите
включить в пакет.

Оператор `package` (например, `package graphics;`) должен быть первой строкой исходного файла.
В каждом исходном файле может быть только один оператор `package`, и он применяется ко всем типам
в файле.

> **Примечание.** Если вы помещаете несколько типов в один исходный файл, только один из них может
> быть `public`, и он должен иметь то же имя, что и исходный файл. Например, вы можете определить
> `public class Circle` в файле `Circle.java`, `public interface Draggable` — в файле
> `Draggable.java`, `public enum Day` — в файле `Day.java` и так далее.
>
> Вы можете включить непубличные типы в тот же файл, что и публичный тип (это настоятельно
> не рекомендуется, если только непубличные типы не малы и не тесно связаны с публичным типом),
> но извне пакета будет доступен только публичный тип. Все непубличные типы верхнего уровня будут
> *пакетно-приватными* (*package private*).

Если поместить интерфейс и классы из предыдущего раздела в пакет с именем `graphics`, потребуется
шесть исходных файлов:

```java
// в файле Draggable.java
package graphics;
public interface Draggable {
    . . .
}

// в файле Graphic.java
package graphics;
public abstract class Graphic {
    . . .
}

// в файле Circle.java
package graphics;
public class Circle extends Graphic
    implements Draggable {
    . . .
}

// в файле Rectangle.java
package graphics;
public class Rectangle extends Graphic
    implements Draggable {
    . . .
}

// в файле Point.java
package graphics;
public class Point extends Graphic
    implements Draggable {
    . . .
}

// в файле Line.java
package graphics;
public class Line extends Graphic
    implements Draggable {
    . . .
}
```

Если не использовать оператор `package`, ваш тип попадает в **безымянный пакет** (*unnamed
package*). В общем случае безымянный пакет предназначен лишь для небольших или временных
приложений либо для самого начала разработки. В остальных случаях классы и интерфейсы должны
принадлежать именованным пакетам.

## Именование пакета

Поскольку программисты по всему миру пишут классы и интерфейсы на языке Java, вполне вероятно,
что многие из них используют одно и то же имя для разных типов. По сути, в предыдущем примере так
и сделано: там определён класс `Rectangle`, хотя класс `Rectangle` уже есть в пакете `java.awt`.
Тем не менее компилятор позволяет обоим классам иметь одинаковое имя, если они находятся в разных
пакетах. **Полностью квалифицированное имя** (*fully qualified name*) каждого класса `Rectangle`
включает имя пакета. То есть полностью квалифицированное имя класса `Rectangle` в пакете
`graphics` — это `graphics.Rectangle`, а полностью квалифицированное имя класса `Rectangle`
в пакете `java.awt` — это `java.awt.Rectangle`.

Это работает хорошо до тех пор, пока два независимых программиста не используют одно и то же имя
для своих пакетов. Что предотвращает эту проблему? Соглашение.

### Соглашения об именовании

Имена пакетов пишутся целиком в нижнем регистре, чтобы избежать конфликта с именами классов или
интерфейсов.

Компании используют своё инвертированное доменное имя в интернете как начало имён своих пакетов —
например, `com.example.mypackage` для пакета с именем `mypackage`, созданного программистом из
`example.com`.

Конфликты имён, возникающие внутри одной компании, нужно разрешать соглашениями внутри этой
компании — например, добавляя название региона или проекта после имени компании (например,
`com.example.region.mypackage`).

Пакеты самого языка Java начинаются с `java.` или `javax.`

В некоторых случаях доменное имя в интернете может оказаться недопустимым именем пакета. Так
бывает, если доменное имя содержит дефис или другой специальный символ, если имя пакета начинается
с цифры или другого символа, который недопустим в начале Java-имени, или если имя пакета содержит
зарезервированное ключевое слово Java, например `int`. В этом случае рекомендуется добавлять знак
подчёркивания. Например:

#### Легализация имён пакетов

| Доменное имя | Префикс имени пакета |
|---|---|
| `hyphenated-name.example.org` | `org.example.hyphenated_name` |
| `example.int` | `int_.example` |
| `123name.example.com` | `com.example._123name` |

## Использование членов пакета

Типы, составляющие пакет, называются **членами пакета** (*package members*).

Чтобы использовать `public`-член пакета извне его пакета, нужно сделать одно из следующего:

- сослаться на член по его полностью квалифицированному имени;
- импортировать член пакета;
- импортировать весь пакет, которому принадлежит член.

Каждый способ подходит для разных ситуаций, как поясняется в следующих разделах.

### Обращение к члену пакета по квалифицированному имени

До сих пор большинство примеров в этом руководстве обращались к типам по их простым именам, таким
как `Rectangle` и `StackOfInts`. Простое имя члена пакета можно использовать, если код, который вы
пишете, находится в том же пакете, что и этот член, или если этот член был импортирован.

Однако если вы пытаетесь использовать член из другого пакета, и этот пакет не был импортирован,
вы должны использовать полностью квалифицированное имя члена, которое включает имя пакета. Вот
полностью квалифицированное имя класса `Rectangle`, объявленного в пакете `graphics` в предыдущем
примере:

```java
graphics.Rectangle
```

Это квалифицированное имя можно использовать, чтобы создать экземпляр `graphics.Rectangle`:

```java
graphics.Rectangle myRect = new graphics.Rectangle();
```

Квалифицированные имена подходят для нечастого использования. Но когда имя применяется
многократно, набирать его снова и снова утомительно, а код становится трудночитаемым. В качестве
альтернативы можно *импортировать* член или его пакет и затем использовать простое имя.

### Импорт члена пакета

Чтобы импортировать конкретный член в текущий файл, поместите оператор `import` в начало файла —
перед любыми определениями типов, но после оператора `package`, если он есть. Вот как импортировать
класс `Rectangle` из пакета `graphics`, созданного в предыдущем разделе:

```java
import graphics.Rectangle;
```

Теперь вы можете ссылаться на класс `Rectangle` по его простому имени:

```java
Rectangle myRectangle = new Rectangle();
```

Этот подход хорош, если вы используете лишь несколько членов пакета `graphics`. Но если вы
используете много типов из пакета, лучше импортировать весь пакет.

### Импорт всего пакета

Чтобы импортировать все типы, содержащиеся в определённом пакете, используйте оператор `import`
с символом-шаблоном «звёздочка» `(*)`:

```java
import graphics.*;
```

Теперь вы можете ссылаться на любой класс или интерфейс пакета `graphics` по его простому имени:

```java
Circle myCircle = new Circle();
Rectangle myRectangle = new Rectangle();
```

Звёздочку в операторе `import` можно использовать только для указания всех классов внутри пакета,
как показано здесь. Её нельзя использовать для сопоставления с подмножеством классов в пакете.
Например, следующая запись *не* сопоставляется со всеми классами пакета `graphics`, начинающимися
на `A`:

```java
// не работает
import graphics.A*;
```

Вместо этого она вызывает ошибку компиляции. С помощью оператора `import` вы, как правило,
импортируете либо один член пакета, либо весь пакет.

> **Примечание.** Существует ещё одна, менее распространённая форма `import`, позволяющая
> импортировать публичные вложенные классы (*nested classes*) объемлющего класса. Например, если
> бы класс `graphics.Rectangle` содержал полезные вложенные классы, такие как
> `Rectangle.DoubleWide` и `Rectangle.Square`, вы могли бы импортировать `Rectangle` и его
> вложенные классы двумя следующими операторами:
>
> ```java
> import graphics.Rectangle;
> import graphics.Rectangle.*;
> ```
>
> Учтите, что второй оператор импорта *не* импортирует сам `Rectangle`.

Ещё одна менее распространённая форма `import` — *оператор статического импорта* (*static import
statement*) — будет рассмотрена в конце этого раздела.

Для удобства компилятор Java автоматически импортирует два пакета целиком для каждого исходного
файла: (1) пакет `java.lang` и (2) текущий пакет (пакет текущего файла).

### Кажущиеся иерархии пакетов

На первый взгляд пакеты кажутся иерархическими, но это не так. Например, Java API включает пакет
`java.awt`, пакет `java.awt.color`, пакет `java.awt.font` и многие другие, начинающиеся с
`java.awt`. Однако пакет `java.awt.color`, пакет `java.awt.font` и другие пакеты `java.awt.xxxx`
*не входят* в пакет `java.awt`. Префикс `java.awt` (Java Abstract Window Toolkit) используется для
ряда связанных пакетов, чтобы показать их связь, но не вложенность.

Импорт `java.awt.*` импортирует все типы пакета `java.awt`, но *не импортирует* `java.awt.color`,
`java.awt.font` или любые другие пакеты `java.awt.xxxx`. Если вы планируете использовать классы и
другие типы из `java.awt.color`, а также из `java.awt`, вы должны импортировать оба пакета со
всеми их файлами:

```java
import java.awt.*;
import java.awt.color.*;
```

Структуру «кажущейся иерархии» можно представить так (вложенности на самом деле нет — это лишь
общий префикс):

```text
java.awt          ← пакет (НЕ содержит перечисленные ниже)
java.awt.color    ← отдельный пакет с общим префиксом
java.awt.font     ← отдельный пакет с общим префиксом
java.awt.xxxx     ← прочие отдельные пакеты с общим префиксом
```

### Неоднозначности имён

Если член одного пакета имеет то же имя, что и член другого пакета, и оба пакета импортированы,
вы должны ссылаться на каждый член по его квалифицированному имени. Например, в пакете `graphics`
определён класс с именем `Rectangle`. В пакете `java.awt` также содержится класс `Rectangle`. Если
импортированы и `graphics`, и `java.awt`, то следующая запись неоднозначна:

```java
Rectangle rect;
```

В такой ситуации нужно использовать полностью квалифицированное имя члена, чтобы точно указать,
какой именно класс `Rectangle` вам нужен. Например:

```java
graphics.Rectangle rect;
```

### Оператор статического импорта

Бывают ситуации, когда требуется частый доступ к статическим финальным полям (константам) и
статическим методам одного-двух классов. Постоянное добавление имени этих классов в виде префикса
может загромождать код. *Оператор статического импорта* (*static import statement*) позволяет
импортировать нужные вам константы и статические методы, чтобы не указывать перед ними имя их
класса.

Класс `java.lang.Math` определяет константу `PI` и множество статических методов, в том числе для
вычисления синусов, косинусов, тангенсов, квадратных корней, максимумов, минимумов, степеней и
многого другого. Например:

```java
public static final double PI 
    = 3.141592653589793;
public static double cos(double a)
{
    ...
}
```

Обычно, чтобы использовать эти объекты из другого класса, вы добавляете имя класса в виде
префикса:

```java
double r = Math.cos(Math.PI * theta);
```

Вы можете использовать оператор статического импорта, чтобы импортировать статические члены
`java.lang.Math` и не добавлять перед ними имя класса `Math`. Статические члены `Math` можно
импортировать как по отдельности:

```java
import static java.lang.Math.PI;
```

так и группой:

```java
import static java.lang.Math.*;
```

После импорта статические члены можно использовать без квалификации. Например, предыдущий фрагмент
кода превратится в:

```java
double r = cos(PI * theta);
```

Очевидно, вы можете писать собственные классы с константами и статическими методами, которые
используете часто, а затем применять оператор статического импорта. Например:

```java
import static mypackage.MyConstants.*;
```

> **Примечание.** Используйте статический импорт очень умеренно. Чрезмерное его использование
> может привести к коду, который трудно читать и сопровождать, потому что читающие код не будут
> знать, какой класс определяет тот или иной статический объект. При правильном использовании
> статический импорт делает код более читаемым, устраняя повторение имени класса.

## Управление исходными файлами и файлами классов

Многие реализации платформы Java для управления исходными файлами и файлами классов полагаются на
иерархические файловые системы, хотя *Спецификация языка Java* (*The Java Language Specification*)
этого не требует. Стратегия такова.

Поместите исходный код класса, интерфейса, перечисления или аннотационного типа в текстовый файл,
имя которого совпадает с простым именем типа, а расширение — `.java`. Например:

```java
// в файле Rectangle.java
package graphics;
public class Rectangle {
   ... 
}
```

Затем поместите исходный файл в каталог, имя которого отражает имя пакета, которому принадлежит
тип:

```text
.....\graphics\Rectangle.java
```

Квалифицированное имя члена пакета и путь к файлу параллельны друг другу (здесь использован
разделитель имён файлов Microsoft Windows — обратная косая черта; для UNIX используется прямая
косая черта):

- **имя класса** — `graphics.Rectangle`
- **путь к файлу** — `graphics\Rectangle.java`

Как вы помните, по соглашению компания использует для имён своих пакетов инвертированное доменное
имя в интернете. Компания Example, чьё доменное имя — `example.com`, предваряла бы все имена своих
пакетов префиксом `com.example`. Каждый компонент имени пакета соответствует подкаталогу. Так,
если бы у компании Example был пакет `com.example.graphics`, содержащий исходный файл
`Rectangle.java`, он находился бы в такой цепочке подкаталогов:

```text
....\com\example\graphics\Rectangle.java
```

Когда вы компилируете исходный файл, компилятор создаёт отдельный выходной файл для каждого
определённого в нём типа. Базовое имя выходного файла — это имя типа, а его расширение — `.class`.
Например, если исходный файл выглядит так:

```java
// в файле Rectangle.java
package com.example.graphics;
public class Rectangle {
      . . . 
}

class Helper{
      . . . 
}
```

то скомпилированные файлы будут располагаться по путям:

```text
<путь к родительскому каталогу выходных файлов>\com\example\graphics\Rectangle.class
<путь к родительскому каталогу выходных файлов>\com\example\graphics\Helper.class
```

Как и исходные файлы `.java`, скомпилированные файлы `.class` должны находиться в цепочке каталогов,
отражающей имя пакета. Однако путь к файлам `.class` не обязан совпадать с путём к исходным файлам
`.java`. Вы можете организовать каталоги исходников и классов раздельно:

```text
<путь_один>\sources\com\example\graphics\Rectangle.java

<путь_два>\classes\com\example\graphics\Rectangle.class
```

Так вы можете передать каталог `classes` другим программистам, не раскрывая своих исходников. Кроме
того, управлять исходными файлами и файлами классов подобным образом нужно для того, чтобы
компилятор и виртуальная машина Java (JVM) могли находить все типы, которые использует ваша
программа.

Полный путь к каталогу `classes`, `<путь_два>\classes`, называется **путём к классам** (*class
path*) и задаётся системной переменной `CLASSPATH`. И компилятор, и JVM строят путь к вашим файлам
`.class`, добавляя имя пакета к пути к классам. Например, если

```text
<путь_два>\classes
```

— это ваш путь к классам, а имя пакета —

```text
com.example.graphics,
```

то компилятор и JVM ищут файлы `.class` в

```text
<путь_два>\classes\com\example\graphics.
```

Путь к классам может включать несколько путей, разделённых точкой с запятой (Windows) или
двоеточием (UNIX). По умолчанию компилятор и JVM ищут в текущем каталоге и в JAR-файле, содержащем
классы платформы Java, поэтому эти каталоги автоматически входят в ваш путь к классам.

Раздельную организацию исходных файлов и файлов классов можно представить деревом:

```text
<путь_один>\
└── sources\
    └── com\
        └── example\
            └── graphics\
                └── Rectangle.java

<путь_два>\
└── classes\                ← путь к классам (CLASSPATH)
    └── com\
        └── example\
            └── graphics\
                ├── Rectangle.class
                └── Helper.class
```

### Установка системной переменной CLASSPATH

Чтобы вывести текущее значение переменной `CLASSPATH`, используйте эти команды в Windows и UNIX
(командная оболочка Bourne):

```text
В Windows:   C:\> set CLASSPATH
В UNIX:      % echo $CLASSPATH
```

Чтобы удалить текущее содержимое переменной `CLASSPATH`, используйте эти команды:

```text
В Windows:   C:\> set CLASSPATH=
В UNIX:      % unset CLASSPATH; export CLASSPATH
```

Чтобы задать переменную `CLASSPATH`, используйте такие команды (для примера):

```text
В Windows:   C:\> set CLASSPATH=C:\users\george\java\classes
В UNIX:      % CLASSPATH=/home/george/java/classes; export CLASSPATH
```

## Итог по созданию и использованию пакетов

Чтобы создать пакет для типа, поместите оператор `package` первым оператором в исходном файле,
содержащем этот тип (класс, интерфейс, перечисление или аннотационный тип).

Чтобы использовать публичный тип, находящийся в другом пакете, у вас есть три варианта: (1)
использовать полностью квалифицированное имя типа, (2) импортировать тип или (3) импортировать
весь пакет, членом которого является тип.

Пути к исходным файлам и файлам классов пакета отражают имя пакета.

Возможно, вам придётся настроить `CLASSPATH`, чтобы компилятор и JVM могли находить файлы `.class`
для ваших типов.
