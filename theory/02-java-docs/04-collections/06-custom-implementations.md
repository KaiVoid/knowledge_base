# Урок 6. Собственные реализации

**Трейл:** Collections · **Оригинал:** [Custom Implementations](https://docs.oracle.com/javase/tutorial/collections/custom-implementations/index.html)
**Связанные области:** [[03-collections]] · [[03-collections-internals]] · **Вопросы:** collections

> Перевод официального руководства Oracle (The Java Tutorials, JDK 8).

> Многим программистам никогда не понадобится реализовывать собственные классы коллекций (`Collection`).
> Используя реализации, описанные в предыдущих разделах этой главы, можно зайти достаточно далеко.
> Тем не менее, когда-нибудь вам может захотеться написать собственную реализацию. Сделать это
> довольно легко с помощью абстрактных реализаций (*abstract implementations*), которые предоставляет
> платформа Java. Прежде чем обсуждать, *как* написать реализацию, давайте обсудим, *почему* вам
> может это понадобиться.

## Зачем писать собственную реализацию

> Следующий список иллюстрирует те разновидности собственных коллекций (`Collection`), которые вам
> может захотеться реализовать. Он не претендует на полноту:

- **Постоянные (*Persistent*).** Все встроенные реализации `Collection` располагаются в основной
  памяти и исчезают при завершении программы. Если вам нужна коллекция, которая по-прежнему будет
  присутствовать при следующем запуске программы, вы можете реализовать её, построив тонкую обёртку
  (*veneer*) поверх внешней базы данных. К такой коллекции могут одновременно обращаться несколько
  программ.

- **Прикладные (*Application-specific*).** Это очень широкая категория. Один из примеров —
  неизменяемое отображение (`Map`), содержащее телеметрические данные в реальном времени. Ключами
  могли бы быть местоположения, а значения считывались бы с датчиков в этих местоположениях в ответ
  на операцию `get`.

- **Высокопроизводительные специального назначения (*High-performance, special-purpose*).** Многие
  структуры данных используют ограничения на способ применения, чтобы предложить лучшую
  производительность, чем возможна у реализаций общего назначения. Например, рассмотрим список
  (`List`), содержащий длинные серии одинаковых значений элементов. Такие списки, часто
  встречающиеся при обработке текста, могут быть закодированы методом длин серий (*run-length
  encoded*) — серия может быть представлена единственным объектом, содержащим повторяемый элемент и
  число последовательных повторений. Этот пример интересен тем, что в нём идёт компромисс между
  двумя аспектами производительности: он требует меньше места, но больше времени, чем `ArrayList`.

- **Высокопроизводительные общего назначения (*High-performance, general-purpose*).** Разработчики
  Java Collections Framework старались предоставить лучшие реализации общего назначения для каждого
  интерфейса, но могло быть использовано очень-очень много структур данных, и новые изобретаются
  каждый день. Может быть, вам удастся придумать что-то более быстрое!

- **Расширенная функциональность (*Enhanced functionality*).** Допустим, вам нужна эффективная
  реализация мешка (*bag*, также известного как *мультимножество*, *multiset*): коллекция
  (`Collection`), которая предлагает проверку вхождения за константное время, допуская при этом
  дублирующиеся элементы. Реализовать такую коллекцию поверх `HashMap` достаточно несложно.

- **Удобство (*Convenience*).** Вам могут понадобиться дополнительные реализации, предлагающие
  удобства сверх тех, что предлагает платформа Java. Например, вам может часто требоваться экземпляр
  списка (`List`), представляющий непрерывный диапазон целых чисел (`Integer`).

- **Адаптер (*Adapter*).** Допустим, вы используете устаревший API (*legacy API*), у которого есть
  собственный самодельный API коллекций. Вы можете написать реализацию-адаптер, которая позволит
  этим коллекциям работать в Java Collections Framework. *Реализация-адаптер* (*adapter
  implementation*) — это тонкая обёртка, которая оборачивает объекты одного типа и заставляет их
  вести себя как объекты другого типа, транслируя операции над вторым типом в операции над первым.

## Как написать собственную реализацию

> Написать собственную реализацию на удивление легко. Java Collections Framework предоставляет
> абстрактные реализации, специально разработанные для того, чтобы облегчить создание собственных
> реализаций.

Начнём со следующего примера реализации метода
[`Arrays.asList`](https://docs.oracle.com/javase/8/docs/api/java/util/Arrays.html#asList-T...-).

```java
public static <T> List<T> asList(T[] a) {
    return new MyArrayList<T>(a);
}

private static class MyArrayList<T> extends AbstractList<T> {

    private final T[] a;

    MyArrayList(T[] array) {
        a = array;
    }

    public T get(int index) {
        return a[index];
    }

    public T set(int index, T element) {
        T oldValue = a[index];
        a[index] = element;
        return oldValue;
    }

    public int size() {
        return a.length;
    }
}
```

Хотите верьте, хотите нет, но это очень близко к реализации, содержащейся в `java.util.Arrays`. Вот
так просто! Вы предоставляете конструктор и методы `get`, `set` и `size`, а `AbstractList` делает всё
остальное. Вы бесплатно получаете итератор списка (`ListIterator`), массовые операции (*bulk
operations*), операции поиска, вычисление хеш-кода, сравнение и строковое представление.

Допустим, вы хотите сделать реализацию немного быстрее. Документация API для абстрактных реализаций
точно описывает, как реализован каждый метод, так что вы будете знать, какие методы переопределить,
чтобы добиться нужной производительности. Производительность предыдущей реализации нормальная, но её
можно немного улучшить. В частности, метод `toArray` проходит по списку (`List`), копируя по одному
элементу за раз. Учитывая внутреннее представление, гораздо быстрее и разумнее просто клонировать
массив.

```java
public Object[] toArray() {
    return (Object[]) a.clone();
}
```

С добавлением этого переопределения и ещё нескольких подобных ему эта реализация в точности
совпадает с той, что находится в `java.util.Arrays`. Ради полной ясности отметим: другими
абстрактными реализациями пользоваться немного сложнее, потому что вам придётся написать собственный
итератор, но это всё равно не так трудно.

## Сводка абстрактных реализаций

> Следующий список суммирует абстрактные реализации:

| Класс | Назначение | Минимально необходимые методы |
|-------|------------|-------------------------------|
| [`AbstractCollection`](https://docs.oracle.com/javase/8/docs/api/java/util/AbstractCollection.html) | Коллекция (`Collection`), которая не является ни множеством (`Set`), ни списком (`List`). | Как минимум, нужно предоставить методы `iterator` и `size`. |
| [`AbstractSet`](https://docs.oracle.com/javase/8/docs/api/java/util/AbstractSet.html) | Множество (`Set`). | Используется так же, как `AbstractCollection` (`iterator` и `size`). |
| [`AbstractList`](https://docs.oracle.com/javase/8/docs/api/java/util/AbstractList.html) | Список (`List`), опирающийся на хранилище с произвольным доступом (*random-access*), например массив. | Как минимум, нужно предоставить методы позиционного доступа (`get` и, опционально, `set`, `remove` и `add`) и метод `size`. Абстрактный класс берёт на себя `listIterator` (и `iterator`). |
| [`AbstractSequentialList`](https://docs.oracle.com/javase/8/docs/api/java/util/AbstractSequentialList.html) | Список (`List`), опирающийся на хранилище с последовательным доступом (*sequential-access*), например связный список (*linked list*). | Как минимум, нужно предоставить методы `listIterator` и `size`. Абстрактный класс берёт на себя методы позиционного доступа. (Это противоположность `AbstractList`.) |
| [`AbstractQueue`](https://docs.oracle.com/javase/8/docs/api/java/util/AbstractQueue.html) | Очередь (`Queue`). | Как минимум, нужно предоставить методы `offer`, `peek`, `poll` и `size`, а также итератор (`iterator`), поддерживающий `remove`. |
| [`AbstractMap`](https://docs.oracle.com/javase/8/docs/api/java/util/AbstractMap.html) | Отображение (`Map`). | Как минимум, нужно предоставить представление `entrySet`. Обычно оно реализуется с помощью класса `AbstractSet`. Если отображение изменяемо, нужно также предоставить метод `put`. |

## Процесс написания собственной реализации

> Процесс написания собственной реализации выглядит так:

1. Выберите подходящий абстрактный класс реализации из приведённого выше списка.
2. Предоставьте реализации всех абстрактных методов класса. Если ваша собственная коллекция должна
   быть изменяемой, вам придётся переопределить также один или несколько конкретных методов.
   Документация API для абстрактного класса реализации подскажет, какие методы нужно переопределить.
3. Протестируйте и, при необходимости, отладьте реализацию. Теперь у вас есть работающая собственная
   реализация коллекции.
4. Если вас беспокоит производительность, прочитайте документацию API абстрактного класса реализации
   по всем методам, реализации которых вы наследуете. Если какой-то из них кажется слишком медленным,
   переопределите его. Если вы переопределяете какие-либо методы, обязательно измерьте
   производительность метода до и после переопределения. То, сколько усилий вы вложите в подгонку
   производительности, должно зависеть от того, насколько активно будет использоваться реализация и
   насколько критично её использование для производительности. (Часто этот шаг лучше пропустить.)
