# Урок 4. Развёртывание в деталях

**Трейл:** Deployment · **Оригинал:** [Deployment In-Depth](https://docs.oracle.com/javase/tutorial/deployment/deploymentInDepth/index.html)
**Связанные области:** [[18-build-tools]] · [[23-security]] · **Вопросы:** build-tools

> Перевод официального руководства Oracle (The Java Tutorials, JDK 8). Урок объединяет страницы
> раздела *Deployment In-Depth*: пользовательское согласие, инструментарий развёртывания

> протокол JNLP и структуру JNLP-файла, а также лучшие практики развёртывания.

Развёртывание полнофункциональных интернет-приложений (rich Internet application, RIA) задействует
различные технологии и инструменты: протокол сетевого запуска Java (Java Network Launch Protocol, JNLP),
инструментарий развёртывания (Deployment Toolkit), `pack200`, `jarsigner` и так далее. Этот урок
показывает, как эти технологии и инструменты помогают разработчикам развёртывать RIA — апплеты
(applets) и приложения Java Web Start.

В развёртывании RIA участвуют следующие основные компоненты:

- HTML-страница, на которой развёртывается RIA;
- JNLP-файл для RIA;
- JAR-файл, содержащий файлы классов и ресурсы RIA.

На эти компоненты ссылаются последующие разделы.

> **Историческая справка.** Технологии Java Plug-in, апплеты и Java Web Start, описанные в этом
> уроке, относятся к JDK 8 и в последующих выпусках устарели или удалены. Материал сохранён как
> точный перевод оригинала Oracle.

## Пользовательское согласие на запуск RIA

В целях безопасности перед первым запуском RIA у пользователя запрашивается разрешение на запуск
приложения — даже если приложение подписано или не требует доступа за пределы «песочницы»
безопасности (security sandbox). В зависимости от запускаемого RIA приглашение (prompt) содержит
следующую информацию:

- имя RIA либо уведомление о том, что приложение не подписано;
- сведения об издателе (publisher), если приложение подписано сертификатом доверенного
  удостоверяющего центра (certificate authority). Если сертификат просрочен, добавляется
  предупреждение. Если приложение самоподписано (self-signed), издатель отображается как UNKNOWN
  (неизвестный);
- предупреждения, если сертификат просрочен, отозван либо его статус отзыва не удаётся проверить;
- местоположение (location), откуда осуществляется доступ к приложению;
- уровень доступа (level of access), требуемый приложением. Ограниченный доступ (limited access)
  удерживает приложение в «песочнице» безопасности, неограниченный доступ (unrestricted access)
  предоставляет приложению доступ к ресурсам системы пользователя;
- предупреждение об отсутствующих атрибутах манифеста JAR-файла, если рекомендуемые атрибуты
  отсутствуют;
- для неподписанных или самоподписанных приложений — флажок, который пользователь должен
  установить, чтобы принять приложение;
- в некоторых случаях — возможность больше не показывать это приглашение.

Описание приглашений см. в статье
[What should I do when I see a security prompt from Java?](http://java.com/faq-securityprompts).

Пользователи также получают предупреждение, если запускают устаревшую JRE, и им предлагается
обновиться до последней версии перед запуском приложения. Пользователь может выбрать запуск
с имеющейся в системе JRE либо заблокировать запуск приложения.

Параметр уровня безопасности (security level) в панели управления Java (Java Control Panel)
определяет, будет ли пользователю предоставлена возможность запускать RIA. Значение по умолчанию
High (высокий) запрашивает у пользователя разрешение на запуск приложений, которые подписаны
действительным сертификатом и включают атрибут Permissions в манифесте главного JAR-файла. Если
статус отзыва приложения проверить невозможно, приложению тоже разрешается работать с разрешения
пользователя.

Подпись RIA обеспечивает пользователю определённый уровень доверия. Учитывайте следующее при
подготовке приложения к развёртыванию:

- наилучший пользовательский опыт обеспечивает приложение, подписанное сертификатом, выданным
  признанным удостоверяющим центром;
- самоподписанные и неподписанные приложения не разрешается запускать, если только не создан
  список исключений сайтов (exception site list) или набор правил развёртывания (deployment rule
  set), явно разрешающий запуск приложения;
- подписанные приложения могут быть либо привилегированными (privileged), либо приложениями
  «песочницы» (sandbox). Привилегированным приложениям предоставляется неограниченный доступ к
  ресурсам системы пользователя. Приложения «песочницы» ограничены «песочницей» безопасности Java.
  Неподписанные приложения ограничены «песочницей».

## Инструментарий развёртывания (Deployment Toolkit)

Скрипт инструментария развёртывания (Deployment Toolkit) — это набор JavaScript-функций, которые
помогают разработчикам единообразно развёртывать полнофункциональные интернет-приложения (RIA) в
различных сочетаниях браузеров и операционных систем. Скрипт оценивает нижележащие браузер и
операционную систему и развёртывает RIA с корректным HTML. Этот скрипт также может убедиться, что
на клиентской машине присутствует требуемая версия среды выполнения Java (Java Runtime Environment,
JRE). Скрипт инструментария развёртывания был представлен в выпуске Java Platform, Standard Edition 6
update 10.

### Расположение скрипта инструментария развёртывания

Скрипт находится по следующим веб-адресам:

- `http://www.java.com/js/deployJava.js` (см. примечание);
- `https://www.java.com/js/deployJava.js` — при развёртывании апплета на защищённой странице
  используйте скрипт из этого защищённого расположения, чтобы избежать предупреждений о смешанном
  содержимом (mixed content) при загрузке страницы.

> **Примечание.** Веб-адрес `http://www.java.com/js/deployJava.js` выводится из употребления.
> Для запуска всех приложений используйте веб-адрес `https://www.java.com/js/deployJava.js`.

JavaScript-код в этом расположении минимизирован, чтобы загружаться быстро. Человекочитаемую версию
кода с соответствующими блоками комментариев можно посмотреть по адресу
[`https://www.java.com/js/deployJava.txt`](https://www.java.com/js/deployJava.txt).

> **Примечание.** В браузере клиента должен быть включён интерпретатор JavaScript, чтобы скрипт
> инструментария развёртывания мог работать и корректно развёртывать ваш RIA.

## Развёртывание апплета

Развернуть апплеты можно с помощью функции `runApplet` скрипта инструментария развёртывания.
Функция `runApplet` убеждается, что на клиенте присутствует требуемая минимальная версия среды
выполнения Java (JRE), и затем запускает апплет. Функция `runApplet` генерирует HTML-тег
`<applet>` с переданной информацией.

> **Примечание.** В зависимости от типа браузера вы можете не увидеть HTML, сгенерированный
> скриптом инструментария развёртывания, при попытке посмотреть исходный код веб-страницы. Чтобы
> увидеть сгенерированный HTML, попробуйте сохранить HTML-страницу после её загрузки либо
> используйте инструмент вроде Firebug (дополнение к Mozilla Firefox).

Развернуть апплеты можно, указав параметры развёртывания как атрибуты и параметры тега `<applet>`.
Также можно указать параметры развёртывания в файле протокола сетевого запуска Java (JNLP), чтобы
воспользоваться расширенными возможностями. Подробнее об этом протоколе см. раздел
«Протокол сетевого запуска Java (JNLP)».

> **Примечание.** Если у клиента нет требуемой минимальной версии JRE, скрипт инструментария
> развёртывания перенаправляет браузер на `http://www.java.com`, чтобы пользователь смог загрузить
> последнюю версию JRE. На некоторых платформах пользователь может быть перенаправлен ещё до того,
> как увидит веб-страницу с апплетом.

Параметры функции `runApplet` различаются в зависимости от того, используете ли вы JNLP. Апплеты,
развёрнутые с помощью JNLP, могут работать только при наличии на клиентской машине Java Plug-in
нового поколения (next generation Java Plug-in; впервые представлен в выпуске Java Platform,
Standard Edition 6 update 10).

Возможны следующие сценарии использования:

- задание параметров развёртывания как пар «имя-значение» атрибутов и параметров;
- использование параметра `jnlp_href` для указания параметров развёртывания в JNLP-файле;
- задание пар «имя-значение» атрибутов и параметров *вместе с* JNLP-файлом (позволяет апплету
  работать как на старом, так и на новом Java Plug-in).

**Сигнатура функции:** `runApplet: function(attributes, parameters, minimumVersion)`

Параметры:

- `attributes` — имена и значения атрибутов генерируемого тега `<applet>`;
- `parameters` — имена и значения тегов `<param>` в генерируемом теге `<applet>`;
- `minimumVersion` — минимальная версия JRE, требуемая для запуска этого апплета.

### Параметры развёртывания как пары «имя-значение»

Атрибуты и параметры, переданные парами «имя-значение», записываются как атрибуты и вложенные теги
`<param>` в генерируемый тег `<applet>`. Апплеты, развёрнутые таким способом, могут запускаться
старым Java Plug-in.

```html
<script src=
    "https://www.java.com/js/deployJava.js"></script>
<script>
    // запускаем апплет Java 2D на JRE версии 1.6.0 или выше
    // с одним параметром (fontSize)
    var attributes = {code:'java2d.Java2DemoApplet.class',
        archive:'Java2Demo.jar', width:710, height:540};
    var parameters = { fontSize:16, permissions:'sandbox' };
    var version = '1.6';
    deployJava.runApplet(attributes, parameters, version);
</script>
```

### Параметры развёртывания в JNLP-файле через `jnlp_href`

Атрибуты и параметры (в данном случае `jnlp_href`), переданные парами «имя-значение», записываются
как атрибуты и вложенные теги `<param>` в генерируемый тег `<applet>`. Апплеты, развёрнутые таким
способом, могут запускаться только Java Plug-in нового поколения. Ширину и высоту апплета лучше
задавать как атрибуты:

```html
<script src="https://www.java.com/js/deployJava.js"></script>
<script>
    var attributes = { code:'java2d.Java2DemoApplet', width:710, height:540 };
    var parameters = { jnlp_href: 'java2d.jnlp' };
    deployJava.runApplet(attributes, parameters, '1.6');
</script>
```

### Пары «имя-значение» вместе с JNLP-файлом

Апплеты, развёрнутые с помощью JNLP, запускаются только у тех конечных пользователей, в браузере
которых работает Java Plug-in нового поколения. Если вы хотите, чтобы апплет работал и на старом
Java Plug-in, задайте параметры развёртывания парами «имя-значение» *вместе с* JNLP-файлом.

```html
<script src="https://www.java.com/js/deployJava.js"></script>
<script>
    var attributes = {code:'java2d.Java2DemoApplet.class',
            archive:'Java2Demo.jar', width:710, height:540};
    var parameters = { fontSize:16, jnlp_href:'java2d.jnlp' };
    var version = '1.6' ;
    deployJava.runApplet(attributes, parameters, version);
</script>
```

Если некоторые параметры развёртывания имеют разные значения в парах «имя-значение» атрибутов и в
JNLP-файле, полезны следующие рекомендации:

- задавайте `width` и `height` как пары «имя-значение» атрибутов (не в JNLP-файле);
- задавайте параметры вроде `image` и `boxbgcolor` как пары «имя-значение» параметров (не в
  JNLP-файле). Эти параметры нужны на раннем этапе запуска апплета;
- в JNLP-файле оставьте атрибут `codebase` пустым либо укажите абсолютный URL. Когда атрибут
  `codebase` оставлен пустым, по умолчанию используется каталог, содержащий JNLP-файл;
- если апплет запускается с помощью JNLP-файла, значения атрибутов `code`, `codebase` и `archive`
  берутся из JNLP-файла. Если эти атрибуты также заданы отдельно как пары «имя-значение», такие
  пары игнорируются.

## Встраивание JNLP-файла в тег апплета

Когда апплеты развёртываются с помощью протокола сетевого запуска Java (JNLP), Java Plug-in
запускает апплет после загрузки JNLP-файла из сети. Начиная с выпуска Java SE 7, можно сократить
время запуска апплетов, встроив JNLP-файл в саму веб-страницу, чтобы избежать дополнительного
сетевого запроса при первой загрузке апплета. Это позволяет апплетам быстро запускаться в браузере.

JNLP-файл, закодированный по схеме Base64, можно встроить в параметр `jnlp_embedded` при
развёртывании апплета на веб-странице. Атрибуты элемента `<jnlp>` должны удовлетворять следующим
ограничениям:

- атрибут `href` должен содержать относительный путь;
- атрибут `codebase` не должен указываться. Это означает, что codebase будет выведен из URL
  веб-страницы, на которой загружается апплет.

Следующие шаги описывают, как встроить JNLP-файл в веб-страницу для развёртывания апплета.

1. Создайте JNLP-файл для своего апплета. Пример файла приведён ниже.

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!-- атрибут href содержит относительный путь;
     атрибут codebase не указан -->
<jnlp href="dynamictree_applet.jnlp">
    <information>
        <title>Dynamic Tree Demo</title>
        <vendor>Dynamic Team</vendor>
    </information>
    <resources>
        <!-- Ресурсы приложения -->
        <j2se version="1.7+" />
        <jar href=
            "dist/applet_ComponentArch_DynamicTreeDemo/DynamicTreeDemo.jar"
             main="true" />
    </resources>
    <applet-desc
         name="Dynamic Tree Demo Applet"
         main-class="appletComponentArch.DynamicTreeApplet"
         width="300"
         height="300">
     </applet-desc>
     <update check="background"/>
</jnlp>
```

2. Закодируйте содержимое JNLP-файла по схеме Base64. Можно использовать любой инструмент
   Base64-кодирования. Сверьтесь с описанием инструмента, чтобы получить строку в кодировке Base64.
   Примеры инструментов и сайтов:
   - команды UNIX — `base64`, `uuencode`;
   - сайты — [Base64 Encode and Decode](http://base64encode.org/),
     [Base64 Encoder](http://www.opinionatedgeek.com/dotnet/tools/base64encode/).

3. При развёртывании апплета на веб-странице задайте параметр `jnlp_embedded`, присвоив ему
   значение в виде закодированной по Base64 строки JNLP. Убедитесь, что включаете только сами байты
   Base64 без заголовков и подвалов, характерных для конкретного инструмента кодирования.

```html
<script src="https://www.java.com/js/deployJava.js"></script>
<script>
    var attributes = {} ;
    <!-- закодированная по Base64 строка ниже сокращена для читаемости -->
    var parameters = {jnlp_href: 'dynamictree_applet.jnlp',
        jnlp_embedded: 'PCEtLSANCi8qDQogIb20Q ... bmxwPg=='
    } ;
    deployJava.runApplet(attributes, parameters, '1.6');
</script>
```

Некоторые инструменты кодирования переносят закодированную строку на несколько строк по 76 символов.
Чтобы использовать такое многострочное значение атрибута в коде JavaScript, задайте значение
атрибута как набор конкатенированных строк. Многострочное значение атрибута можно включать как есть,
если апплет развёртывается напрямую через HTML-тег `<applet>`.

## Развёртывание приложения Java Web Start

Развернуть приложения Java Web Start можно с помощью функции `createWebStartLaunchButton` скрипта
инструментария развёртывания. Приложения Java Web Start запускаются с использованием протокола
сетевого запуска Java (JNLP). Функция `createWebStartLaunchButton` генерирует ссылку (HTML-тег
якоря — `<a>`) на JNLP-файл приложения Java Web Start.

Сгенерированный тег якоря — это кнопка «Launch» (Запуск) приложения Java Web Start. Когда конечный
пользователь нажимает кнопку «Launch», скрипт инструментария развёртывания убеждается, что
установлена подходящая среда выполнения Java (JRE), и затем запускает приложение Java Web Start.

> **Примечание.** В зависимости от типа браузера вы можете не увидеть HTML, сгенерированный скриптом
> инструментария развёртывания, при попытке посмотреть исходный код веб-страницы. Чтобы увидеть
> сгенерированный HTML, попробуйте сохранить HTML-страницу после её загрузки либо используйте
> инструмент вроде Firebug.

> **Примечание.** Если у клиента нет требуемой минимальной версии JRE, скрипт инструментария
> развёртывания перенаправляет браузер на `http://www.java.com`, чтобы пользователь смог загрузить
> последнюю версию JRE.

**Сигнатура функции:** `createWebStartLaunchButton: function(jnlp, minimumVersion)` либо
`createWebStartLaunchButton: function(jnlp)`

Параметры:

- `jnlp` — URL JNLP-файла, содержащего информацию о развёртывании приложения Java Web Start. Этот
  URL должен быть абсолютным путём;
- `minimumVersion` — минимальная версия JRE, требуемая для запуска этого приложения.

### Указание минимальной версии JRE

```html
<script src="https://www.java.com/js/deployJava.js"></script>
<script>
    var url = "http://java.sun.com/javase/technologies/desktop/javawebstart/apps/notepad.jnlp";
    deployJava.createWebStartLaunchButton(url, '1.6.0');
</script>
```

### Запуск на любой версии JRE

Используйте функцию `createWebStartLaunchButton: function(jnlp)`, если у вашего приложения нет
требования к минимальной версии JRE.

> **Примечание.** При развёртывании с помощью любой из описанных выше функций
> `createWebStartLaunchButton` необходимо указать абсолютный codebase в JNLP-файле приложения Java
> Web Start. Это позволяет запускать приложение Java Web Start из командной строки командой
> `javaws <путь/к/локальному JNLP-файлу>`.

## Изменение кнопки запуска

Изображение кнопки «Launch» приложения Java Web Start можно изменить, если вам не нравится кнопка по
умолчанию или у вас есть другое стандартизированное изображение.

Используйте переменную `deployJava.launchButtonPNG`, чтобы указать расположение изображения кнопки
запуска.

**Переменная:** `deployJava.launchButtonPNG`

**Использование:** указание альтернативного URL изображения.

В этом примере кнопка «Launch» приложения Notepad заменена изображением машущего Дюка (Duke).

```html
<script src="https://www.java.com/js/deployJava.js"></script>
<script>
    deployJava.launchButtonPNG='https://docs.oracle.com/javase/tutorial/images/DukeWave.gif';
    var url = "https://docs.oracle.com/javase/tutorialJWS/samples/deployment/NotepadJWSProject/Notepad.jnlp";
    deployJava.createWebStartLaunchButton(url, '1.6.0');
</script>
```

> **Примечание.** Если пример не запускается, возможно, в браузере нужно включить интерпретатор
> JavaScript, чтобы скрипт инструментария развёртывания работал корректно.

## Развёртывание без codebase

Начиная с выпуска Java SE 7, не обязательно указывать абсолютный путь для атрибута `codebase` в
JNLP-файле приложения Java Web Start. Можно разрабатывать и тестировать приложения в разных
окружениях, не изменяя путь в атрибуте `codebase`. Если codebase не указан, программное обеспечение
Java Web Start считает, что codebase задан относительно веб-страницы, с которой запускается
приложение Java Web Start.

Для развёртывания приложений Java Web Start на веб-странице, когда JNLP-файл не содержит атрибута
`codebase`, можно использовать следующие функции скрипта инструментария развёртывания:

- `launchWebStartApplication` — используйте эту функцию в HTML-ссылке для развёртывания приложения
  Java Web Start;
- `createWebStartLaunchButtonEx` — используйте эту функцию для создания кнопки «Launch» приложения
  Java Web Start.

> **Примечание.** Для работы приложениям Java Web Start, развёрнутым с помощью указанных выше
> функций, требуется как минимум выпуск Java SE 7. Если у клиента нет как минимум выпуска Java SE 7,
> функции предписывают пользователю установить требуемую JRE перед запуском приложения Java Web Start.

**Сигнатура функции:** `launchWebStartApplication: function(jnlp)`

Параметр:

- `jnlp` — путь к JNLP-файлу, содержащему информацию о развёртывании приложения Java Web Start. Этот
  путь может быть относительным относительно веб-страницы, на которой развёртывается приложение.

В следующем примере функция `launchWebStartApplication` вызывается в атрибуте `href` HTML-тега
якоря (`a`). Для развёртывания приложения Dynamic Tree Demo используется JNLP-файл
`dynamictree_webstart_no_codebase.jnlp`.

```html
<script src="https://www.java.com/js/deployJava.js"></script>
<a href="javascript:deployJava.launchWebStartApplication('dynamictree_webstart_no_codebase.jnlp');">Launch</a>
```

Приложение Java Web Start запускается, когда пользователь нажимает получившуюся HTML-ссылку.

**Сигнатура функции:** `createWebStartLaunchButtonEx: function(jnlp)`

Параметр:

- `jnlp` — путь к JNLP-файлу, содержащему информацию о развёртывании приложения Java Web Start. Этот
  путь может быть относительным относительно веб-страницы, на которой развёртывается приложение.

Следующий пример показывает использование функции `createWebStartLaunchButtonEx`. Для развёртывания
приложения Dynamic Tree Demo используется JNLP-файл `dynamictree_webstart_no_codebase.jnlp`.

```html
<script src="https://www.java.com/js/deployJava.js"></script>
<script>
    var jnlpFile = "dynamictree_webstart_no_codebase.jnlp";
    deployJava.createWebStartLaunchButtonEx(jnlpFile);
</script>
```

Приложение Java Web Start запускается, когда пользователь нажимает получившуюся кнопку «Launch».

> **Примечание.** Приложение Java Web Start можно также запустить из командной строки системы,
> вызвав команду `javaws` с полным URL JNLP-файла:
>
> ```
> javaws http://example.com/dynamictree_webstart_no_codebase.jnlp
> ```

## Проверка версии JRE на клиенте

Существует много причин проверить, доступна ли на клиентской машине конкретная версия среды
выполнения Java (JRE). Например, вы можете захотеть запустить другую версию своего RIA либо
перенаправить пользователя на другую страницу в зависимости от версии JRE на клиенте.

Используйте функцию `versionCheck` скрипта инструментария развёртывания, чтобы проверить, установлена
ли на клиенте конкретная версия или диапазон версий JRE.

**Сигнатура функции:** `versionCheck: function(versionPattern)`

Параметры:

- `versionPattern` — строка, задающая версию или диапазон версий для проверки, например «1.4»,
  «1.5.0\*» (семейство 1.5.x) и «1.6.0_02+» (любая версия больше либо равная 1.6.0_02).

**Использование:** создание разного пользовательского опыта в зависимости от версии JRE на клиенте.

В этом примере кнопка «Launch» для приложения Notepad создаётся, только если версия JRE на клиенте
больше либо равна 1.6. В противном случае браузер перенаправляется на `oracle.com`.

```html
<script src="https://www.java.com/js/deployJava.js"></script>
<script>
    if (deployJava.versionCheck('1.6+')) {
        var url = "https://docs.oracle.com/javase/tutorialJWS/deployment/webstart/examples/Notepad.jnlp";

        <!-- здесь можно также вызвать deployJava.runApplet -->
        deployJava.createWebStartLaunchButton(url, '1.6.0');
    } else {
        document.location.href="http://oracle.com";
    }
</script>
```

> **Примечание.** В зависимости от операционной системы клиента и версии платформы Java вы можете
> проверить информацию о версии JRE на уровне старшей версии (major version, например 1.6) либо на
> более точном уровне обновления (update level, например 1.6.0_10).

## Протокол сетевого запуска Java (JNLP)

Протокол сетевого запуска Java (Java Network Launch Protocol, JNLP) позволяет запускать приложение
на рабочем столе клиента, используя ресурсы, размещённые на удалённом веб-сервере. Java Plug-in и
Java Web Start считаются JNLP-клиентами (JNLP clients), поскольку могут запускать удалённо
размещённые апплеты и приложения на рабочем столе клиента. Подробности см. в
[Java Network Launching Protocol and API Specification Change Log](http://www.oracle.com/technetwork/java/javase/jnlp-spec-log-139509.html).

Недавние улучшения в технологиях развёртывания позволяют запускать полнофункциональные
интернет-приложения (RIA) с помощью JNLP. По этому протоколу можно запускать как апплеты, так и
приложения Java Web Start. RIA, запущенные с помощью JNLP, также имеют доступ к JNLP API. Эти API
позволяют RIA обращаться к рабочему столу клиента с разрешения пользователя.

JNLP включается JNLP-файлом RIA. JNLP-файл описывает RIA. JNLP-файл задаёт имя главного JAR-файла,
версию JRE, требуемую для запуска RIA, имя и информацию для отображения, дополнительные пакеты
(optional packages), параметры времени выполнения, системные свойства и так далее.

## Структура JNLP-файла

Этот раздел описывает синтаксис файла протокола сетевого запуска Java (JNLP) для полнофункциональных
интернет-приложений (RIA).

### Пример JNLP-файла

```xml
<?xml version="1.0" encoding="UTF-8"?>
<jnlp spec="1.0+" codebase="" href="">
    <information>
        <title>Dynamic Tree Demo</title>
        <vendor>Dynamic Team</vendor>
        <icon href="sometree-icon.jpg"/>
        <offline-allowed/>
    </information>
    <resources>
        <!-- Ресурсы приложения -->
        <j2se version="1.6+" href=
           "http://java.sun.com/products/autodl/j2se"/>
        <jar href="DynamicTreeDemo.jar"
            main="true" />

    </resources>
    <application-desc
         name="Dynamic Tree Demo Application"
         main-class="webstartComponentArch.DynamicTreeApplication"
         width="300"
         height="300">
     </application-desc>
     <update check="background"/>
</jnlp>
```

### Часто используемые элементы и атрибуты

В следующей таблице описаны часто используемые элементы JNLP-файла. Колонка «Начиная с» (Since)
указывает версию спецификации JNLP, в которой появился элемент; колонка «Обязательный» отмечает
обязательные элементы.

| Элемент | Родитель | Описание | Начиная с | Обязательный |
|---|---|---|---|---|
| `jnlp` | (корень) | Самый верхний XML-элемент JNLP-файла. | 1.0 | Да |
| `information` | `jnlp` | Содержит другие элементы, описывающие RIA и его источник. | 1.0 | Да |
| `title` | `information` | Заголовок RIA. | 1.0 | Да |
| `vendor` | `information` | Поставщик RIA. | 1.0 | Да |
| `homepage` | `information` | Домашняя страница RIA. | 1.0 | — |
| `description` | `information` | Краткое описание RIA. | 1.0 | — |
| `icon` | `information` | Значок, по которому пользователь может идентифицировать RIA. | 1.0 | — |
| `offline-allowed` | `information` | Указывает, что данный RIA может работать, когда клиентская система отключена от сети. | 1.0 | — |
| `shortcut` | `information` | Указывает предпочтения RIA по интеграции с рабочим столом. | 1.5.0 | — |
| `association` | `information` | Подсказывает JNLP-клиенту, что RIA хочет быть зарегистрирован в операционной системе как основной обработчик определённых расширений и определённого MIME-типа. Если этот элемент включён, должен быть также включён либо элемент `offline-allowed`, либо задан атрибут `href` элемента `jnlp`. | 1.5.0 | — |
| `related-content` | `information` | Дополнительная связанная единица содержимого, которая может быть интегрирована с RIA. | 1.5.0 | — |
| `resources` | `jnlp` | Описывает все ресурсы, необходимые для RIA. | 1.0 | Да |
| `java` или `j2se` | `resources` | Версии Java, на которых запускается RIA. | 1.6.0 (`java`) | — |
| `jar` | `resources` | JAR-файл, входящий в путь к классам (classpath) RIA. | 1.0 | Да |
| `nativelib` | `resources` | JAR-файл, содержащий нативные библиотеки в своём корневом каталоге. | 1.0 | — |
| `extension` | `resources` | Указатель на дополнительный `component-desc` или `installer-desc`, используемый с этим RIA. | 1.0 | — |
| `package` | `resources` | Указывает JNLP-клиенту, какие пакеты реализованы в каких JAR-файлах. | 1.0 | — |
| `property` | `resources` | Определяет системное свойство, доступное через методы `System.getProperty` и `System.getProperties`. | 1.0 | — |
| `application-desc` | `jnlp` | Обозначает, что это JNLP-файл для приложения. | 1.0 | Да (один из четырёх) |
| `applet-desc` | `jnlp` | Обозначает, что это JNLP-файл для апплета. | 1.0 | — |
| `component-desc` | `jnlp` | Обозначает, что это JNLP-файл для компонентного расширения (component extension). | 1.0 | — |
| `installer-desc` | `jnlp` | Обозначает, что это JNLP-файл для установленного расширения (installed extension). | 1.0 | — |
| `security` | `jnlp` | Запрашивает расширенные права. Если этот элемент не включён, приложение работает в «песочнице» безопасности. | 1.0 | — |
| `update` | `jnlp` | Предпочтения по тому, как JNLP-клиент должен обрабатывать обновления RIA. | 1.6.0 | — |

> **Примечание.** JNLP-файл должен содержать один из элементов: `application-desc`, `applet-desc`,
> `component-desc` или `installer-desc`.

### Кодировка JNLP-файлов

Java Web Start поддерживает любую кодировку символов, поддерживаемую платформой Java. Кодировка
задаётся в XML-прологе:

```xml
<?xml version="1.0" encoding="utf-16"?>
```

При этом сам XML-пролог должен быть закодирован в UTF-8.

## Лучшие практики развёртывания

Улучшить пользовательский опыт вашего RIA помогут описанные ниже лучшие практики.

- Подписывайте RIA сертификатом признанного удостоверяющего центра. Убедитесь, что подписаны все
  артефакты и что сертификат не просрочен. Информацию о подписи см. в разделе
  «Signing and Verifying JAR Files» (Подпись и проверка JAR-файлов).
- Запрашивайте минимально необходимый уровень прав. Если RIA не требует неограниченного доступа к
  системе пользователя, укажите уровень прав `sandbox`. Дополнительные рекомендации по безопасности
  см. в разделе «Security in Rich Internet Applications».
- Оптимизируйте размер JAR-файлов и связанных ресурсов, чтобы RIA загружался быстро. Приёмы
  оптимизации см. в разделе «Сокращение времени загрузки».
- Включите протокол загрузки по версиям (version download protocol) и используйте фоновые проверки
  обновлений, чтобы RIA запускался быстро. Подробнее о протоколе загрузки по версиям и проверках
  обновлений см. в разделе «Избегание ненужных проверок обновлений».
- Убедитесь, что у клиента есть требуемая версия JRE. О том, как использовать для этого скрипт
  инструментария развёртывания, см. в разделе «Обеспечение наличия JRE».
- Встраивайте содержимое JNLP-файла апплета в тег `<applet>`, чтобы не загружать JNLP-файл из сети.
  Эта возможность была представлена в выпуске Java SE 7. См. раздел «Встраивание JNLP-файла в тег
  апплета».
- По возможности предзагружайте приложение Java Web Start. Если вы планируете развернуть RIA как
  приложение Java Web Start в организации, где у вас есть административный контроль, можно
  предзагрузить приложение на разные клиенты, чтобы оно было закэшировано и готово к использованию.
  Для предзагрузки используйте команду:

```
javaws -import -silent <jnlp url>
```

### Сокращение времени загрузки

RIA загружаются с веб-сайта, когда пользователь пытается обратиться к ним (после первоначальной
загрузки RIA могут кэшироваться для повышения производительности). Время загрузки RIA зависит от
размера JAR-файла RIA. Большие JAR-файлы загружаются дольше.

Сократить время загрузки RIA можно следующими приёмами:

- сжать JAR-файл RIA с помощью инструмента
  [`pack200`](https://docs.oracle.com/javase/8/docs/technotes/tools/windows/pack200.html);
- удалить лишние пробельные символы из JNLP-файла и из JavaScript-файлов;
- оптимизировать изображения и анимацию.

Следующие шаги описывают, как создать и развернуть сжатый JAR-файл для подписанного RIA.

1. Нормализуйте JAR-файл с помощью опции `--repack`. Этот шаг гарантирует, что сертификат
   безопасности и JAR-файл пройдут проверки при запуске RIA.

   ```
   pack200 --repack DynamicTreeDemo.jar
   ```

2. Подпишите нормализованный JAR-файл.

   ```
   jarsigner -keystore myKeyStore DynamicTreeDemo.jar me
   ```

   где `myKeyStore` — имя хранилища ключей (keystore), а `me` — псевдоним (alias) для хранилища ключей.

3. Упакуйте подписанный JAR-файл.

   ```
   pack200 DynamicTreeDemo.jar.pack.gz DynamicTreeDemo.jar
   ```

4. Установите свойство `jnlp.packEnabled` в `true` в JNLP-файле RIA.

   ```xml
   <resources>
       <j2se version="1.6+"
           href="http://java.sun.com/products/autodl/j2se"
                 max-heap-size="128m" />
       <jar href="DynamicTreeDemo.jar"
           main="true"/>
       <property name="jnlp.packEnabled"
           value="true"/>
       <!-- ... -->
   </resources>
   ```

Когда в JNLP-файле установлено свойство `jnlp.packEnabled`, Java Plug-in ищет сжатый JAR-файл с
расширением `.pack.gz` (например, `DynamicTreeDemo.jar.pack.gz`). Если он найден, Java Plug-in
автоматически распаковывает и загружает JAR-файл. Если файл с расширением `.pack.gz` не найден,
Java Plug-in пытается загрузить обычный JAR-файл (например, `DynamicTreeDemo.jar`).

> **Примечание.** Чтобы протестировать свойство `jnlp.packEnabled`, RIA нужно развернуть на
> веб-сервере.

### Избегание ненужных проверок обновлений

RIA кэшируются локально для улучшения времени запуска. Однако перед запуском RIA программа запуска
проверяет, что каждый JAR-файл, на который ссылается JNLP-файл RIA, актуален. Иными словами,
программа запуска убеждается, что вы запускаете последнюю версию RIA, а не более старую
закэшированную копию. Эти проверки обновлений могут занимать до нескольких сотен миллисекунд в
зависимости от количества JAR-файлов и скорости сети. Описанные ниже приёмы помогают избежать ненужных
проверок обновлений и ускорить запуск RIA.

> **Примечание.** Термин «программа запуска» (launch software) используется здесь собирательно для
> обозначения Java Plug-in и Java Web Start. Java Plug-in запускает апплеты, а Java Web Start —
> приложения Java Web Start.

#### Использование протокола загрузки по версиям

Чтобы устранить ненужные проверки версий, можно задействовать *протокол загрузки по версиям*
(version download protocol). Для его включения выполните следующие шаги.

1. Переименуйте JAR-файлы, добавив суффикс с номером версии по следующему соглашению об именовании:

   ```
   <имя JAR-файла>__V<номер версии>.jar
   ```

   Например, переименуйте `DynamicTreeDemo.jar` в `DynamicTreeDemo__V1.0.jar`.

2. В JNLP-файле укажите версию для каждого JAR-файла и установите свойство `jnlp.versionEnabled` в
   `true`.

   ```xml
   <resources>
       <!-- Ресурсы приложения -->
       <j2se version="1.6+"
           href="http://java.sun.com/products/autodl/j2se"
               max-heap-size="128m" />
       <jar href="DynamicTreeDemo.jar"
           main="true" version="1.0"/>
       <jar href="SomeOther.jar" version="2.0"/>
       <property name="jnlp.versionEnabled"
           value="true"/>
       <!-- ... -->
   </resources>
   ```

   Когда свойство `jnlp.versionEnabled` включено, программа запуска выполняет лишь *одну* проверку
   обновления, чтобы убедиться в актуальности JNLP-файла. Программа сравнивает номера версий,
   указанные в JNLP-файле, с соответствующими версиями JAR-файлов (по соглашению об именовании из
   шага 1) и обновляет только устаревшие JAR-файлы. Этот подход эффективен, потому что по сети
   выполняется только проверка обновления JNLP-файла. Все остальные проверки версий происходят
   локально.

   Если файл с правильным номером версии не найден, программа запуска пытается загрузить
   JAR-файл по умолчанию (например, `DynamicTreeDemo.jar`).

#### Выполнение проверок обновлений в фоне

Если для пользователя не критично немедленно запускать последнюю версию RIA, можно указать, что все
проверки обновлений должны выполняться в фоне. В этом случае программа запуска запускает локально
закэшированную копию для немедленного использования и загружает более новую версию RIA в фоновом
режиме. Более новая версия RIA будет запущена при следующей попытке пользователя воспользоваться RIA.
Чтобы включить фоновые проверки обновлений, добавьте в JNLP-файл следующую строку:

```xml
<update check='background'/>
```

Следующий фрагмент показывает пример JNLP-файла с включённой фоновой проверкой обновлений:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<jnlp spec="1.0+" codebase="" href="">
    <information>
        <title>Applet Takes Params</title>
        <vendor>Dynamic Team</vendor>
    </information>
    <resources>
        <!-- Ресурсы приложения -->
        <j2se version="1.6+" href=
            "http://java.sun.com/products/autodl/j2se"/>
        <jar href="applet_AppletWithParameters.jar"
            main="true" />
    </resources>
    <applet-desc
         name="Applet Takes Params"
         main-class="AppletTakesParams"
         width="800"
         height="50">
             <param name="paramStr" value="someString"/>
             <param name="paramInt" value="22"/>
     </applet-desc>
     <update check="background"/>
</jnlp>
```

### Обеспечение наличия JRE

RIA обычно требуют наличия минимальной версии JRE на клиентской машине. При развёртывании RIA нужно
убедиться, что на клиентских машинах есть требуемая версия JRE, чтобы RIA работал корректно. Со
скриптом инструментария развёртывания есть как минимум два способа выполнить это требование.

- Можно проверить версию JRE на клиенте сразу, как только пользователь заходит на ваш веб-сайт, и
  при необходимости установить последнюю версию.
- Можно позволить пользователю перемещаться по веб-сайту, а проверять и устанавливать последнюю JRE
  только тогда, когда он пытается воспользоваться вашим RIA.

#### Проверка и установка последней JRE при заходе на сайт

Следующий пример проверяет, установлена ли у пользователя хотя бы версия JRE 1.6.0_13. Если нет, код
устанавливает последнюю версию JRE. См. встроенные комментарии в коде.

```html
<script src="https://www.java.com/js/deployJava.js"></script>
<script>

    // проверяем, что текущая версия JRE больше 1.6.0
    alert("versioncheck " + deployJava.versionCheck('1.6.0_10+'));
    if (deployJava.versionCheck('1.6.0_10+') == false) {
        userInput = confirm(
            "You need the latest Java(TM) Runtime Environment. " +
            "Would you like to update now?");
        if (userInput == true) {

            // Устанавливаем deployJava.returnPage, чтобы пользователь вернулся
            // на ваш веб-сайт после установки JRE
            deployJava.returnPage = location.href;

            // Устанавливаем последнюю JRE или перенаправляем пользователя
            // на другую страницу для получения JRE
            deployJava.installLatestJRE();
        }
    }
</script>
```

#### Установка нужной JRE только при попытке использовать RIA

Когда вы указываете минимальную версию JRE в функции `runApplet` или `createWebStartLaunchButton`,
скрипт инструментария развёртывания убеждается, что требуемая версия JRE есть на клиенте, перед
запуском вашего RIA.

Используйте функцию `runApplet` для развёртывания апплета, как показано в следующем примере.
Последний параметр функции `runApplet` — минимальная версия, требуемая для запуска апплета
(версия 1.6).

```html
<script src="https://www.java.com/js/deployJava.js"></script>
<script>
    var attributes = { code:'components.DynamicTreeApplet',
        width:300, height:300};
    var parameters = {jnlp_href: 'dynamictree_applet.jnlp'};
    deployJava.runApplet(attributes, parameters, '1.6');
</script>
```

Для развёртывания приложения Java Web Start используйте функцию `createWebStartLaunchButton` с
правильным параметром минимальной версии (версия 1.6).

```html
<script src="https://www.java.com/js/deployJava.js"></script>
<script>
    var url = "dynamictree_applet.jnlp";
    deployJava.createWebStartLaunchButton(url, '1.6.0');
</script>
```

Функции `runApplet` и `createWebStartLaunchButton` проверяют версию JRE на клиенте. Если минимальная
версия не установлена, функции устанавливают последнюю версию JRE.
