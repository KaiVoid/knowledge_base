# Урок 6. Упаковка программ в JAR-файлы

**Трейл:** Deployment · **Оригинал:** [Packaging Programs in JAR Files](https://docs.oracle.com/javase/tutorial/deployment/jar/index.html)
**Связанные области:** [[18-build-tools]] · **Вопросы:** build-tools

> Перевод официального руководства Oracle (The Java Tutorials, JDK 8).

Формат Java-архива (Java™ Archive, JAR) позволяет упаковать несколько файлов в один
архивный файл. Обычно JAR-файл содержит class-файлы и вспомогательные ресурсы,
связанные с апплетами и приложениями.

Формат JAR-файла даёт множество преимуществ:

- **Безопасность (Security).** Содержимое JAR-файла можно подписать цифровой подписью.
  Пользователи, узнающие вашу подпись, могут по желанию выдать вашему ПО привилегии
  безопасности, которых оно иначе бы не имело.
- **Сокращение времени загрузки (Decreased download time).** Если апплет упакован в
  JAR-файл, class-файлы апплета и связанные с ним ресурсы можно загрузить в браузер за одну
  HTTP-транзакцию без необходимости открывать новое соединение для каждого файла.
- **Сжатие (Compression).** Формат JAR позволяет сжимать файлы для эффективного хранения.
- **Упаковка расширений (Packaging for extensions).** Механизм расширений (extensions
  framework) даёт возможность добавлять функциональность к ядру платформы Java, и формат
  JAR-файла определяет упаковку для расширений. Используя формат JAR, вы можете превращать
  своё ПО в расширения.
- **Запечатывание пакетов (Package Sealing).** Пакеты, хранящиеся в JAR-файлах, можно по
  желанию запечатать, чтобы пакет мог обеспечивать согласованность версий. Запечатывание
  пакета в JAR-файле означает, что все классы, определённые в этом пакете, должны находиться
  в одном и том же JAR-файле.
- **Версионирование пакетов (Package Versioning).** JAR-файл может хранить данные о файлах,
  которые он содержит, например информацию о поставщике и версии.
- **Переносимость (Portability).** Механизм работы с JAR-файлами — это стандартная часть
  базового API платформы Java.

Этот урок состоит из четырёх разделов:

- **Работа с JAR-файлами: основы** — базовые операции с JAR-файлами и запуск ПО,
  упакованного в JAR-файлы.
- **Работа с файлами манифеста: основы** — что такое манифест и как его настраивать,
  чтобы, например, запечатывать пакеты и задавать точку входа приложения.
- **Подпись и проверка JAR-файлов** — как ставить цифровую подпись на JAR-файлы и
  проверять подписи подписанных JAR-файлов.
- **Использование JAR-related API** — некоторые возможности платформы Java по работе с
  JAR-файлами.

## Работа с JAR-файлами: основы

JAR-файлы упакованы в формате ZIP, поэтому их можно использовать для таких задач, как
сжатие данных без потерь, архивирование, распаковка и извлечение из архива. Эти задачи —
одно из самых частых применений JAR-файлов, и многие преимущества JAR можно получить,
пользуясь только этими базовыми возможностями.

Даже если вы захотите воспользоваться продвинутой функциональностью формата JAR, например
электронной подписью, сначала вам нужно освоить основные операции.

Для базовых задач с JAR-файлами используется инструмент Java Archive Tool, поставляемый в
составе Java Development Kit (JDK). Поскольку этот инструмент вызывается командой `jar`, в
данном руководстве он называется «инструмент Jar» (the Jar tool).

В качестве краткого обзора и предварительного знакомства с темами этого раздела в
следующей таблице приведены распространённые операции с JAR-файлами:

### Распространённые операции с JAR-файлами

| Операция | Команда |
|----------|---------|
| Создать JAR-файл | `jar cf jar-file input-file(s)` |
| Просмотреть содержимое JAR-файла | `jar tf jar-file` |
| Извлечь содержимое JAR-файла | `jar xf jar-file` |
| Извлечь конкретные файлы из JAR-файла | `jar xf jar-file archived-file(s)` |
| Запустить приложение, упакованное в JAR-файл (нужен заголовок манифеста `Main-Class`) | `java -jar app.jar` |
| Вызвать апплет, упакованный в JAR-файл | `<applet code=AppletClassName.class archive="JarFileName.jar" width=width height=height></applet>` |

### Создание JAR-файла

Базовый формат команды для создания JAR-файла:

```bash
jar cf jar-file input-file(s)
```

Опции и аргументы этой команды:

- Опция `c` указывает, что вы хотите **создать** (*create*) JAR-файл.
- Опция `f` указывает, что вывод должен идти в **файл** (*file*), а не в stdout.
- `jar-file` — имя, которое вы хотите дать получившемуся JAR-файлу. Имя может быть любым.
  По соглашению JAR-файлам дают расширение `.jar`, хотя это и не обязательно.
- Аргумент `input-file(s)` — разделённый пробелами список из одного или нескольких файлов,
  которые вы хотите включить в JAR-файл. В аргументе `input-file(s)` можно использовать
  символ-шаблон `*`. Если какие-то из входных файлов являются каталогами, их содержимое
  добавляется в JAR-архив рекурсивно.

Опции `c` и `f` могут идти в любом порядке, но между ними не должно быть пробела.

Эта команда создаст сжатый JAR-файл и поместит его в текущий каталог. Команда также
сгенерирует для JAR-архива файл манифеста по умолчанию.

> **Примечание.** Метаданные в JAR-файле — такие как имена записей, комментарии и
> содержимое манифеста — должны быть закодированы в UTF-8.

Полная сводка опций команды `jar`:

| Опция | Описание |
|-------|----------|
| `c` | **Создать** (*create*) JAR-файл. |
| `f` | Направить вывод в **файл** (*file*), а не в stdout (при создании); при просмотре/извлечении — указать имя JAR-файла в командной строке. |
| `t` | Показать **оглавление** (*table of contents*) JAR-файла. |
| `x` | **Извлечь** (*extract*) файлы из JAR-архива. |
| `u` | **Обновить** (*update*) существующий JAR-файл. |
| `v` | Выводить **подробную** (*verbose*) информацию в stdout во время сборки JAR-файла. Подробный вывод сообщает имя каждого файла по мере его добавления в JAR-файл. |
| `0` (ноль) | Указывает, что JAR-файл сжимать не нужно. |
| `M` | Указывает, что файл манифеста по умолчанию создавать не нужно. |
| `m` | Включить информацию манифеста из существующего файла манифеста. Формат: `jar cmf jar-file existing-manifest input-file(s)`. См. раздел «Модификация файла манифеста». |
| `e` | Задать точку входа приложения (*entrypoint*) — создаёт или переопределяет атрибут `Main-Class` манифеста. |
| `-C` | Сменить каталог во время выполнения команды (см. пример ниже). |

> **Предупреждение.** Манифест должен заканчиваться символом новой строки или возврата
> каретки. Последняя строка не будет разобрана корректно, если она не заканчивается
> новой строкой или возвратом каретки.

> **Примечание.** При создании JAR-файла в нём сохраняется время создания. Поэтому, даже
> если содержимое JAR-файла не меняется, при многократном создании JAR-файла результирующие
> файлы не будут полностью идентичными. Учитывайте это при использовании JAR-файлов в
> сборочном окружении. Для управления версиями JAR-файла рекомендуется использовать
> информацию о версии в файле манифеста, а не время создания. См. раздел «Установка
> информации о версии пакета».

#### Пример

Чтобы упаковать демо TicTacToe в один JAR-файл с именем `TicTacToe.jar`, нужно выполнить
эту команду из каталога TicTacToe:

```bash
jar cvf TicTacToe.jar TicTacToe.class audio images
```

Аргументы `audio` и `images` — это каталоги, поэтому инструмент Jar рекурсивно поместит их
и их содержимое в JAR-файл. Сгенерированный JAR-файл `TicTacToe.jar` будет помещён в текущий
каталог. Поскольку в команде использовалась опция `v` для подробного вывода, при её
выполнении вы увидите примерно следующий вывод:

```
adding: TicTacToe.class (in=3825) (out=2222) (deflated 41%)
adding: audio/ (in=0) (out=0) (stored 0%)
adding: audio/beep.au (in=4032) (out=3572) (deflated 11%)
adding: audio/ding.au (in=2566) (out=2055) (deflated 19%)
adding: audio/return.au (in=6558) (out=4401) (deflated 32%)
adding: audio/yahoo1.au (in=7834) (out=6985) (deflated 10%)
adding: audio/yahoo2.au (in=7463) (out=4607) (deflated 38%)
adding: images/ (in=0) (out=0) (stored 0%)
adding: images/cross.gif (in=157) (out=160) (deflated -1%)
adding: images/not.gif (in=158) (out=161) (deflated -1%)
```

Из вывода видно, что JAR-файл `TicTacToe.jar` сжат. Инструмент Jar сжимает файлы по
умолчанию. Сжатие можно отключить опцией `0` (ноль):

```bash
jar cvf0 TicTacToe.jar TicTacToe.class audio images
```

Сжатия можно избегать, например, чтобы увеличить скорость загрузки JAR-файла браузером.
Несжатые JAR-файлы обычно загружаются быстрее сжатых, потому что отпадает необходимость
распаковывать файлы при загрузке. Однако есть компромисс: время скачивания по сети для
больших несжатых файлов может оказаться больше.

Инструмент Jar принимает аргументы с символом-шаблоном `*`. Если в каталоге TicTacToe не
было лишних файлов, для сборки JAR-файла можно было бы использовать такую альтернативную
команду:

```bash
jar cvf TicTacToe.jar *
```

Хотя подробный вывод этого не показывает, инструмент Jar автоматически добавляет в
JAR-архив файл манифеста с путём `META-INF/MANIFEST.MF`.

В примере выше файлы в архиве сохранили свои относительные пути и структуру каталогов.
Инструмент Jar предоставляет опцию `-C`, с помощью которой можно создать JAR-файл, в
котором относительные пути архивируемых файлов **не** сохраняются. Эта опция смоделирована
по образцу опции `-C` команды TAR.

Например, предположим, что вы хотите поместить аудиофайлы и gif-изображения, используемые
демо TicTacToe, в JAR-файл, причём все файлы должны быть на верхнем уровне, без иерархии
каталогов. Этого можно добиться, выполнив из родительского каталога каталогов `images` и
`audio` такую команду:

```bash
jar cf ImageAudio.jar -C images . -C audio .
```

Часть `-C images` указывает инструменту Jar перейти в каталог `images`, а следующая за
`-C images` точка `.` указывает заархивировать всё содержимое этого каталога. Часть
`-C audio .` делает то же самое с каталогом `audio`. Результирующий JAR-файл будет иметь
такое оглавление:

```
META-INF/MANIFEST.MF
cross.gif
not.gif
beep.au
ding.au
return.au
yahoo1.au
yahoo2.au
```

Для сравнения, если бы вы использовали команду без опции `-C`:

```bash
jar cf ImageAudio.jar images audio
```

то результирующий JAR-файл имел бы такое оглавление:

```
META-INF/MANIFEST.MF
images/cross.gif
images/not.gif
audio/beep.au
audio/ding.au
audio/return.au
audio/yahoo1.au
audio/yahoo2.au
```

### Просмотр содержимого JAR-файла

Базовый формат команды для просмотра содержимого JAR-файла:

```bash
jar tf jar-file
```

Опции и аргумент этой команды:

- Опция `t` указывает, что вы хотите просмотреть **оглавление** (*table of contents*)
  JAR-файла.
- Опция `f` указывает, что JAR-файл, содержимое которого нужно просмотреть, задан в
  командной строке.
- Аргумент `jar-file` — путь и имя JAR-файла, содержимое которого вы хотите просмотреть.

Опции `t` и `f` могут идти в любом порядке, но между ними не должно быть пробела.

Эта команда выведет оглавление JAR-файла в stdout. Можно дополнительно добавить опцию
подробного вывода `v`, чтобы вывести дополнительную информацию о размерах файлов и датах
последнего изменения.

#### Пример

Выведем содержимое файла `TicTacToe.jar`, созданного в предыдущем разделе:

```bash
jar tf TicTacToe.jar
```

Команда выводит содержимое JAR-файла в stdout:

```
META-INF/MANIFEST.MF
TicTacToe.class
audio/
audio/beep.au
audio/ding.au
audio/return.au
audio/yahoo1.au
audio/yahoo2.au
images/
images/cross.gif
images/not.gif
```

JAR-файл содержит class-файл TicTacToe, а также каталоги `audio` и `images`, как и
ожидалось. Вывод также показывает, что JAR-файл содержит файл манифеста по умолчанию
`META-INF/MANIFEST.MF`, который был автоматически помещён в архив инструментом Jar.
Подробнее см. раздел «Понимание манифеста по умолчанию».

Все пути отображаются с прямыми слешами, независимо от платформы или операционной системы.
Пути в JAR-файлах всегда относительные; вы никогда не увидите путь, начинающийся, например,
с `C:`.

Инструмент Jar выведет дополнительную информацию, если использовать опцию `v`:

```bash
jar tvf TicTacToe.jar
```

Например, подробный вывод для JAR-файла TicTacToe выглядел бы примерно так:

```

   553 Mon Sep 24 21:57:48 PDT 2012 TicTacToe.class

  9584 Mon Sep 24 21:57:48 PDT 2012 TicTacToe.java
     0 Mon Sep 24 21:57:48 PDT 2012 audio/

  2566 Mon Sep 24 21:57:48 PDT 2012 audio/ding.au
```

### Извлечение содержимого JAR-файла

Базовая команда для извлечения содержимого JAR-файла:

```bash
jar xf jar-file [archived-file(s)]
```

Опции и аргументы этой команды:

- Опция `x` указывает, что вы хотите **извлечь** (*extract*) файлы из JAR-архива.
- Опция `f` указывает, что JAR-**файл** (*file*), из которого нужно извлечь файлы, задан в
  командной строке, а не через stdin.
- Аргумент `jar-file` — имя (или путь и имя) JAR-файла, из которого нужно извлечь файлы.
- `archived-file(s)` — необязательный аргумент, состоящий из разделённого пробелами списка
  файлов, которые нужно извлечь из архива. Если этот аргумент отсутствует, инструмент Jar
  извлечёт все файлы архива.

Как обычно, порядок опций `x` и `f` не важен, но между ними не должно быть пробела.

При извлечении файлов инструмент Jar делает копии нужных файлов и записывает их в текущий
каталог, воспроизводя структуру каталогов, которую файлы имеют в архиве. Исходный JAR-файл
остаётся без изменений.

> **Осторожно.** При извлечении файлов инструмент Jar перезапишет любые существующие файлы,
> имеющие тот же путь, что и извлекаемые файлы.

#### Пример

Извлечём несколько файлов из JAR-файла TicTacToe, используемого в предыдущих разделах.
Напомним, что содержимое `TicTacToe.jar`:

```
META-INF/MANIFEST.MF
TicTacToe.class
TicTacToe.java
audio/
audio/beep.au
audio/ding.au
audio/return.au
audio/yahoo1.au
audio/yahoo2.au
example1.html
images/
images/cross.gif
images/not.gif
```

Допустим, вы хотите извлечь class-файл TicTacToe и файл изображения `cross.gif`. Для этого
можно использовать такую команду:

```bash
jar xf TicTacToe.jar TicTacToe.class images/cross.gif
```

Эта команда делает две вещи:

- Помещает копию `TicTacToe.class` в текущий каталог.
- Создаёт каталог `images`, если его ещё нет, и помещает в него копию `cross.gif`.

Исходный JAR-файл TicTacToe остаётся без изменений.

Тем же способом можно извлечь сколько угодно файлов. Если в команде не указано, какие файлы
извлекать, инструмент Jar извлекает все файлы архива. Например, все файлы архива TicTacToe
можно извлечь так:

```bash
jar xf TicTacToe.jar
```

### Обновление JAR-файла

Инструмент Jar предоставляет опцию `u`, с помощью которой можно обновлять содержимое
существующего JAR-файла, изменяя его манифест или добавляя файлы.

Базовая команда для добавления файлов имеет такой формат:

```bash
jar uf jar-file input-file(s)
```

В этой команде:

- Опция `u` указывает, что вы хотите **обновить** (*update*) существующий JAR-файл.
- Опция `f` указывает, что обновляемый JAR-файл задан в командной строке.
- `jar-file` — существующий JAR-файл, который нужно обновить.
- `input-file(s)` — разделённый пробелами список из одного или нескольких файлов, которые
  вы хотите добавить в JAR-файл.

Любые файлы, уже находящиеся в архиве и имеющие тот же путь, что и добавляемый файл, будут
перезаписаны.

При создании нового JAR-файла можно по желанию использовать опцию `-C` для смены каталога.
Подробнее см. раздел «Создание JAR-файла».

#### Примеры

Напомним, что `TicTacToe.jar` имеет такое содержимое:

```
META-INF/MANIFEST.MF
TicTacToe.class
TicTacToe.java
audio/
audio/beep.au
audio/ding.au
audio/return.au
audio/yahoo1.au
audio/yahoo2.au
example1.html
images/
images/cross.gif
images/not.gif
```

Допустим, вы хотите добавить в JAR-файл файл `images/new.gif`. Это можно сделать, выполнив
из родительского каталога каталога `images` такую команду:

```bash
jar uf TicTacToe.jar images/new.gif
```

Обновлённый JAR-файл будет иметь такое оглавление:

```
META-INF/MANIFEST.MF
TicTacToe.class
TicTacToe.java
audio/
audio/beep.au
audio/ding.au
audio/return.au
audio/yahoo1.au
audio/yahoo2.au
example1.html
images/
images/cross.gif
images/not.gif
images/new.gif
```

Опцию `-C` можно использовать для «смены каталога» во время выполнения команды. Например:

```bash
jar uf TicTacToe.jar -C images new.gif
```

Эта команда сменит каталог на `images` перед добавлением `new.gif` в JAR-файл. Каталог
`images` не будет включён в путь `new.gif` при его добавлении в архив, в результате чего
оглавление будет выглядеть так:

```
META-INF/MANIFEST.MF
TicTacToe.class
TicTacToe.java
audio/
audio/beep.au
audio/ding.au
audio/return.au
audio/yahoo1.au
audio/yahoo2.au
example1.html
images/
images/cross.gif
images/not.gif
new.gif
```

### Запуск ПО, упакованного в JAR-файлы

После того как вы научились создавать JAR-файлы, как же запустить упакованный код?
Рассмотрите такие сценарии:

- Ваш JAR-файл содержит апплет, который нужно запустить внутри браузера.
- Ваш JAR-файл содержит приложение, которое нужно запустить из командной строки.
- Ваш JAR-файл содержит код, который вы хотите использовать как расширение.

Этот раздел рассматривает первые две ситуации. Использованию JAR-файлов в качестве
расширений посвящён отдельный трейл руководства — про механизм расширений.

#### Апплеты, упакованные в JAR-файлы

Чтобы запустить любой апплет из HTML-файла для выполнения внутри браузера, используется
тег `applet`. Если апплет упакован как JAR-файл, единственное отличие — нужно использовать
параметр `archive`, чтобы указать относительный путь к JAR-файлу.

В качестве примера возьмём демо-апплет TicTacToe. Тег `applet` в HTML-файле,
отображающем апплет, можно разметить так:

```html
<applet code=TicTacToe.class
        width="120" height="120">
</applet>
```

Если демо TicTacToe упаковано в JAR-файл с именем `TicTacToe.jar`, тег `applet` можно
изменить, добавив параметр `archive`:

```html
<applet code=TicTacToe.class
        archive="TicTacToe.jar"
        width="120" height="120">
</applet>
```

Параметр `archive` указывает относительный путь к JAR-файлу, содержащему `TicTacToe.class`.
В этом примере предполагается, что JAR-файл и HTML-файл находятся в одном каталоге. Если
это не так, нужно включить относительный путь к JAR-файлу в значение параметра `archive`.
Например, если JAR-файл находится на один каталог ниже HTML-файла, в каталоге `applets`,
тег `applet` выглядел бы так:

```html
<applet code=TicTacToe.class
        archive="applets/TicTacToe.jar"
        width="120" height="120">
</applet>
```

#### JAR-файлы как приложения

Приложения, упакованные в JAR, можно запускать с помощью лаунчера Java (команда `java`).
Базовая команда:

```bash
java -jar jar-file
```

Флаг `-jar` сообщает лаунчеру, что приложение упаковано в формате JAR-файла. Можно указать
только один JAR-файл, который должен содержать весь специфичный для приложения код.

Перед выполнением этой команды убедитесь, что среда выполнения знает, какой класс внутри
JAR-файла является точкой входа приложения.

Чтобы указать, какой класс является точкой входа приложения, нужно добавить в манифест
JAR-файла заголовок `Main-Class`. Заголовок имеет вид:

```
Main-Class: classname
```

Значение заголовка `classname` — имя класса, который является точкой входа приложения.
Подробнее см. раздел «Установка точки входа приложения».

Когда `Main-Class` задан в файле манифеста, приложение можно запустить из командной строки:

```bash
java -jar app.jar
```

Чтобы запустить приложение из JAR-файла, находящегося в другом каталоге, нужно указать путь
к этому каталогу:

```bash
java -jar path/app.jar
```

## Работа с файлами манифеста: основы

JAR-файлы поддерживают широкий набор функций, включая электронную подпись, контроль версий,
запечатывание пакетов и другие. Что придаёт JAR-файлу такую гибкость? Ответ — **манифест**
JAR-файла.

Манифест (*manifest*) — это специальный файл, который может содержать информацию о файлах,
упакованных в JAR-файл. Настраивая эту «мета»-информацию, содержащуюся в манифесте, вы
позволяете JAR-файлу служить различным целям.

### Понимание манифеста по умолчанию

При создании JAR-файла он автоматически получает файл манифеста по умолчанию. В архиве
может быть только один файл манифеста, и он всегда имеет путь:

```
META-INF/MANIFEST.MF
```

При создании JAR-файла манифест по умолчанию содержит просто следующее:

```
Manifest-Version: 1.0
Created-By: 1.7.0_06 (Oracle Corporation)
```

Эти строки показывают, что записи манифеста имеют форму пар «заголовок: значение»
(*header: value*). Имя заголовка отделяется от его значения двоеточием. Манифест по
умолчанию соответствует версии 1.0 спецификации манифеста и был создан JDK версии 1.7.0_06.

Манифест также может содержать информацию о других файлах, упакованных в архив. Что именно
записывать в манифест, зависит от того, как вы намерены использовать JAR-файл. Манифест по
умолчанию не делает никаких предположений о том, какую информацию о других файлах ему
следует записывать.

Информация о дайджестах (digest) в манифест по умолчанию не включается. О дайджестах и
подписи см. урок «Подпись и проверка JAR-файлов».

### Модификация файла манифеста

Чтобы добавить пользовательскую информацию в манифест при создании JAR-файла, используется
опция командной строки `m`.

Инструмент Jar автоматически помещает в любой создаваемый JAR-файл манифест по умолчанию с
путём `META-INF/MANIFEST.MF`. Вы можете включить специальную функциональность JAR-файла,
например запечатывание пакетов, изменив манифест по умолчанию. Обычно изменение манифеста
по умолчанию заключается в добавлении в манифест специальных **заголовков** (*headers*),
позволяющих JAR-файлу выполнять определённую желаемую функцию.

Чтобы изменить манифест, сначала подготовьте текстовый файл с информацией, которую хотите
добавить в манифест. Затем используйте опцию `m` инструмента Jar, чтобы добавить информацию
из вашего файла в манифест.

> **Предупреждение.** Текстовый файл, из которого создаётся манифест, должен заканчиваться
> символом новой строки или возврата каретки. Последняя строка не будет разобрана
> корректно, если она не заканчивается новой строкой или возвратом каретки.

Базовая команда имеет такой формат:

```bash
jar cfm jar-file manifest-addition input-file(s)
```

Опции и аргументы этой команды:

- Опция `c` указывает, что вы хотите **создать** (*create*) JAR-файл.
- Опция `m` указывает, что вы хотите слить информацию из существующего файла в файл
  манифеста создаваемого JAR-файла.
- Опция `f` указывает, что вывод должен идти в **файл** (создаваемый JAR-файл), а не в
  стандартный вывод.
- `manifest-addition` — имя (или путь и имя) существующего текстового файла, содержимое
  которого нужно добавить к содержимому манифеста JAR-файла.
- `jar-file` — имя, которое вы хотите дать получившемуся JAR-файлу.
- Аргумент `input-file(s)` — разделённый пробелами список из одного или нескольких файлов,
  которые вы хотите поместить в JAR-файл.

Опции `m` и `f` должны идти в том же порядке, что и соответствующие аргументы.

> **Примечание.** Содержимое манифеста должно быть закодировано в UTF-8.

### Установка точки входа приложения

Если у вас есть приложение, упакованное в JAR-файл, нужно как-то указать, какой класс
внутри JAR-файла является точкой входа приложения. Эту информацию вы предоставляете
заголовком `Main-Class` в манифесте, который имеет общий вид:

```
Main-Class: classname
```

Значение `classname` — имя класса, который является точкой входа вашего приложения.

Напомним, что точка входа — это класс, имеющий метод с сигнатурой
`public static void main(String[] args)`.

После того как вы задали заголовок `Main-Class` в манифесте, JAR-файл запускается такой
формой команды `java`:

```bash
java -jar JAR-name
```

Выполняется метод `main` класса, указанного в заголовке `Main-Class`.

#### Пример

Мы хотим выполнить метод `main` в классе `MyClass` из пакета `MyPackage` при запуске
JAR-файла.

Сначала создаём текстовый файл с именем `Manifest.txt` со следующим содержимым:

```
Main-Class: MyPackage.MyClass
```

> **Предупреждение.** Текстовый файл должен заканчиваться символом новой строки или
> возврата каретки. Последняя строка не будет разобрана корректно, если она не
> заканчивается новой строкой или возвратом каретки.

Затем создаём JAR-файл с именем `MyJar.jar`, выполнив команду:

```bash
jar cfm MyJar.jar Manifest.txt MyPackage/*.class
```

Это создаёт JAR-файл с манифестом следующего содержания:

```
Manifest-Version: 1.0
Created-By: 1.7.0_06 (Oracle Corporation)
Main-Class: MyPackage.MyClass
```

Когда вы запускаете JAR-файл следующей командой, выполняется метод `main` класса `MyClass`:

```bash
java -jar MyJar.jar
```

#### Установка точки входа с помощью инструмента Jar

Флаг `e` (от *entrypoint* — точка входа) создаёт или переопределяет атрибут манифеста
`Main-Class`. Его можно использовать при создании или обновлении JAR-файла. Используйте его,
чтобы задать точку входа приложения без редактирования или создания файла манифеста.

Например, эта команда создаёт `app.jar`, где значение атрибута `Main-Class` в манифесте
установлено в `MyApp`:

```bash
jar cfe app.jar MyApp MyApp.class
```

Вы можете напрямую запустить это приложение, выполнив команду:

```bash
java -jar app.jar
```

Если имя класса точки входа находится в пакете, в качестве разделителя можно использовать
символ `.` (точка). Например, если `Main.class` находится в пакете `foo`, точку входа можно
задать так:

```bash
jar cfe Main.jar foo.Main foo/Main.class
```

### Добавление классов в classpath JAR-файла

Иногда требуется обращаться к классам или ресурсам, находящимся вне JAR-файла вашего
приложения. Для этого можно использовать атрибут манифеста `Class-Path`.

В атрибуте манифеста `Class-Path` JAR-файла вашего приложения можно указать один или
несколько относительных URL, ссылающихся на JAR-файлы или каталоги, содержащие классы и
ресурсы, нужные вашему приложению. Эти URL трактуются как относительные ссылки, а не
абсолютные (см. метод [`URI.isAbsolute()`](https://docs.oracle.com/javase/8/docs/api/java/net/URI.html#isAbsolute--)).
То есть эти относительные URL не содержат компонента схемы (scheme) к базе кода (code base),
из которой был загружен JAR-файл приложения, содержащий атрибут `Class-Path`.

Разделяйте каждое значение в атрибуте `Class-Path` пробелом, табуляцией, символом новой
строки, символом возврата каретки или символом перевода страницы (form-feed).

Относительные URL в значении атрибута `Class-Path`, не заканчивающиеся слешем (`/`),
считаются ссылками на JAR-файлы.

Пример атрибута манифеста `Class-Path`:

```
Class-Path: servlet.jar infobus.jar acme/beans.jar images/
```

Используя атрибут `Class-Path` в манифесте JAR-файла приложения, можно избежать
необходимости указывать длинный флаг `-classpath` при запуске `java`.

> **Примечание.** Значение атрибута манифеста `Class-Path` не ссылается на JAR-файлы или
> каталоги **внутри** JAR-файла приложения. Чтобы загружать классы и ресурсы из JAR-файлов,
> находящихся внутри JAR-файла, нужно писать собственный код. Например, если `MyJar.jar`
> содержит другой JAR-файл `MyNested.jar`, вы не можете использовать атрибут `Class-Path` в
> манифесте `MyJar.jar`, чтобы загружать классы из `MyNested.jar`.

#### Пример

Рассмотрим случай, когда из класса в `MyJar.jar` мы хотим загрузить классы и ресурсы,
находящиеся в следующих местах:

- JAR-файл `MyUtils.jar`
- Каталог `MyResources`
- JAR-файл `Lib.jar`, находящийся в дочернем каталоге `MyLibs`

Наша цель — собрать JAR-файл `MyJar.jar` с подходящим атрибутом `Class-Path`, чтобы он мог
обращаться к классам и ресурсам из этих мест. В файловой системе все эти JAR-файлы и
каталоги находятся в одном корневом каталоге. Будем считать этим корневым каталогом
`MyApp`.

Сначала из каталога `MyApp` создаём текстовый файл `Manifest.txt` со следующим содержимым:

```
Class-Path: MyUtils.jar MyLibs/Lib.jar MyResources/
```

> **Предупреждение.** Этот текстовый файл должен заканчиваться символом новой строки или
> возврата каретки. Последняя строка не будет разобрана корректно, если она не
> заканчивается новой строкой или возвратом каретки.

Затем создаём JAR-файл `MyJar.jar` следующей командой:

```bash
jar cfm MyJar.jar Manifest.txt MyPackage/*.class
```

Это создаёт JAR-файл `MyJar.jar` с манифестом следующего содержания:

```
Manifest-Version: 1.0
Class-Path: MyUtils.jar MyLibs/Lib.jar MyResources/
Created-By: 1.8.0_422 (Oracle Corporation)
```

В результате, когда приложение Java запускается из каталога `MyApp`, классы в `MyJar.jar`,
помимо доступа к классам и ресурсам, которые являются частью `MyJar.jar`, могут также
обращаться к классам и ресурсам, содержащимся в JAR-файлах `MyUtils.jar` и `MyLibs/Lib.jar`
и в каталоге `MyResources/`.

Подробнее см. [Class-Path Attribute](https://docs.oracle.com/javase/8/docs/technotes/guides/jar/jar.html#classpath)
в [JAR File Specification](https://docs.oracle.com/javase/8/docs/technotes/guides/jar/jar.html).

### Установка информации о версии пакета

В манифест JAR-файла может потребоваться включить информацию о версии пакета. Эту информацию
вы предоставляете следующими заголовками в манифесте:

#### Заголовки манифеста

| Заголовок | Определение |
|-----------|-------------|
| `Name` | Имя спецификации. |
| `Specification-Title` | Название спецификации. |
| `Specification-Version` | Версия спецификации. |
| `Specification-Vendor` | Поставщик спецификации. |
| `Implementation-Title` | Название реализации. |
| `Implementation-Version` | Номер сборки реализации. |
| `Implementation-Vendor` | Поставщик реализации. |

Один набор таких заголовков можно назначить каждому пакету. Заголовки версионирования должны
располагаться непосредственно под заголовком `Name` для пакета. Этот пример показывает все
заголовки версионирования:

```
Name: java/util/
Specification-Title: Java Utility Classes
Specification-Version: 1.2
Specification-Vendor: Example Tech, Inc.
Implementation-Title: java.util
Implementation-Version: build57
Implementation-Vendor: Example Tech, Inc.
```

Подробнее о заголовках версии пакета см. [Package Versioning specification](https://docs.oracle.com/javase/8/docs/technotes/guides/versioning/spec/versioning2.html#wp89936).

#### Пример

Мы хотим включить заголовки из примера выше в манифест `MyJar.jar`.

Сначала создаём текстовый файл `Manifest.txt` со следующим содержимым:

```
Name: java/util/
Specification-Title: Java Utility Classes
Specification-Version: 1.2
Specification-Vendor: Example Tech, Inc.
Implementation-Title: java.util
Implementation-Version: build57
Implementation-Vendor: Example Tech, Inc.
```

> **Предупреждение.** Текстовый файл должен заканчиваться символом новой строки или возврата
> каретки. Последняя строка не будет разобрана корректно, если она не заканчивается новой
> строкой или возвратом каретки.

Затем создаём JAR-файл `MyJar.jar` командой:

```bash
jar cfm MyJar.jar Manifest.txt MyPackage/*.class
```

Это создаёт JAR-файл с манифестом следующего содержания:

```
Manifest-Version: 1.0
Created-By: 1.7.0_06 (Oracle Corporation)
Name: java/util/
Specification-Title: Java Utility Classes
Specification-Version: 1.2
Specification-Vendor: Example Tech, Inc.
Implementation-Title: java.util
Implementation-Version: build57
Implementation-Vendor: Example Tech, Inc.
```

### Запечатывание пакетов в JAR-файле

Пакеты в JAR-файлах можно по желанию запечатывать (*seal*), что означает, что все классы,
определённые в этом пакете, должны быть заархивированы в одном и том же JAR-файле.
Запечатать пакет может потребоваться, например, чтобы обеспечить согласованность версий
классов в вашем ПО.

Пакет запечатывается в JAR-файле добавлением в манифест заголовка `Sealed`, имеющего общий
вид:

```
Name: myCompany/myPackage/
Sealed: true
```

Значение `myCompany/myPackage/` — имя запечатываемого пакета.

Обратите внимание, что имя пакета должно заканчиваться символом `/`.

#### Пример

Мы хотим запечатать два пакета `firstPackage` и `secondPackage` в JAR-файле `MyJar.jar`.

Сначала создаём текстовый файл `Manifest.txt` со следующим содержимым:

```
Name: myCompany/firstPackage/
Sealed: true

Name: myCompany/secondPackage/
Sealed: true
```

> **Предупреждение.** Текстовый файл должен заканчиваться символом новой строки или возврата
> каретки. Последняя строка не будет разобрана корректно, если она не заканчивается новой
> строкой или возвратом каретки.

Затем создаём JAR-файл `MyJar.jar` командой:

```bash
jar cfm MyJar.jar Manifest.txt MyPackage/*.class
```

Это создаёт JAR-файл с манифестом следующего содержания:

```
Manifest-Version: 1.0
Created-By: 1.7.0_06 (Oracle Corporation)
Name: myCompany/firstPackage/
Sealed: true
Name: myCompany/secondPackage/
Sealed: true
```

#### Запечатывание JAR-файлов

Если вы хотите гарантировать, что все классы пакета происходят из одного источника кода
(code source), используйте запечатывание JAR (JAR sealing). Запечатанный JAR указывает, что
все пакеты, определённые этим JAR, запечатаны, если только это не переопределено для
отдельных пакетов.

Чтобы запечатать JAR-файл, используйте заголовок манифеста `Sealed` со значением `true`.
Например:

```
Sealed: true
```

указывает, что все пакеты в этом архиве запечатаны, если только это явно не переопределено
для конкретных пакетов атрибутом `Sealed` в записи манифеста.

## Подпись и проверка JAR-файлов

Вы можете по желанию подписать JAR-файл своей электронной «подписью». Пользователи, которые
проверяют вашу подпись, могут выдать вашему упакованному в JAR ПО привилегии безопасности,
которых оно обычно бы не имело. И наоборот, вы можете проверять подписи подписанных
JAR-файлов, которые хотите использовать.

> **Примечание.** Это руководство написано для JDK 8. Примеры и практики, описанные на этой
> странице, не используют улучшения, появившиеся в более поздних релизах, и могут применять
> технологии, более не доступные. Актуальные руководства см. на [Dev.java](https://dev.java/learn/).

### Понимание подписи и проверки

Платформа Java™ позволяет ставить цифровую подпись на JAR-файлы. Вы подписываете файл
цифровой подписью по той же причине, по которой подписываете бумажный документ ручкой и
чернилами — чтобы читатели знали, что документ написали (или хотя бы одобрили) вы.

Когда вы подписываете письмо, каждый, кто узнаёт вашу подпись, может подтвердить, что письмо
написали вы. Аналогично, когда вы ставите цифровую подпись на файл, любой, кто «узнаёт» вашу
цифровую подпись, знает, что файл пришёл от вас. Процесс «узнавания» электронных подписей
называется **проверкой** (*verification*).

При подписании JAR-файла у вас есть также возможность поставить **отметку времени**
(*time stamp*) на подпись. Подобно дате на бумажном документе, отметка времени указывает,
когда JAR-файл был подписан. Отметку времени можно использовать, чтобы убедиться, что
сертификат, которым подписан JAR-файл, был действителен на момент подписания.

Способность подписывать и проверять файлы — важная часть архитектуры безопасности платформы
Java. Безопасность контролируется **политикой** (*policy*) безопасности, действующей во
время выполнения. Политику можно настроить так, чтобы выдавать привилегии безопасности
апплетам и приложениям. Например, можно выдать апплету разрешение выполнять обычно
запрещённые операции, такие как чтение и запись локальных файлов или запуск локальных
исполняемых программ. Если вы скачали код, подписанный доверенной сущностью, вы можете
использовать этот факт как критерий при решении, какие разрешения безопасности назначить
коду.

Платформа Java обеспечивает подпись и проверку с помощью специальных чисел, называемых
открытым и закрытым **ключами** (*keys*). Открытые и закрытые ключи существуют парами и
играют взаимодополняющие роли.

Закрытый ключ — это электронная «ручка», которой вы подписываете файл. Как следует из
названия, ваш закрытый ключ известен только вам, чтобы никто другой не мог «подделать» вашу
подпись. Файл, подписанный вашим закрытым ключом, может быть проверен только соответствующим
открытым ключом.

Однако одних только открытого и закрытого ключей недостаточно для подлинной проверки
подписи. Требуется ещё один элемент — **сертификат** (*certificate*), который подписавший
включает в подписанный JAR-файл. Сертификат — это подписанное цифровой подписью заявление от
признанного **центра сертификации** (*certification authority*), указывающее, кто владеет
конкретным открытым ключом. Центры сертификации — это сущности (обычно фирмы,
специализирующиеся на цифровой безопасности), которым доверяет вся отрасль для подписания и
выдачи сертификатов на ключи и их владельцев.

Подытожим цифровую подпись:

- Подписавший подписывает JAR-файл закрытым ключом.
- Соответствующий открытый ключ помещается в JAR-файл вместе со своим сертификатом, чтобы
  им мог воспользоваться любой, кто хочет проверить подпись.

#### Дайджесты и файл подписи

При подписании JAR-файла каждому файлу в архиве даётся запись дайджеста (*digest*) в
манифесте архива. Пример такой записи:

```
Name: test/classes/ClassOne.class
SHA1-Digest: TD1GZt8G11dXY2p4olSZPc5Rj64=
```

Значения дайджеста — это хеши или закодированные представления содержимого файлов на момент
подписания. Дайджест файла изменится тогда и только тогда, когда изменится сам файл.

При подписании JAR-файла автоматически генерируется **файл подписи** (*signature file*) и
помещается в каталог `META-INF` JAR-файла — тот же каталог, что содержит манифест архива.
Файлы подписи имеют имена с расширением `.SF`. Пример содержимого файла подписи:

```
Signature-Version: 1.0
SHA1-Digest-Manifest: h1yS+K9T7DyHtZrtI+LxvgqaMYM=
Created-By: 1.7.0_06 (Oracle Corporation)

Name: test/classes/ClassOne.class
SHA1-Digest: fcav7ShIG6i86xPepmitOVo4vWY=

Name: test/classes/ClassTwo.class
SHA1-Digest: xrQem9snnPhLySDiZyclMlsFdtM=

Name: test/images/ImageOne.gif
SHA1-Digest: kdHbE7kL9ZHLgK7akHttYV4XIa0=

Name: test/images/ImageTwo.gif
SHA1-Digest: mF0D5zpk68R4oaxEqoS9Q7nhm60=
```

Файл подписи содержит записи дайджестов файлов архива, похожие на записи дайджестов в
манифесте. Однако значения дайджеста в манифесте вычисляются из самих файлов, а значения
дайджеста в файле подписи вычисляются из соответствующих записей в манифесте. Файлы подписи
также содержат значение дайджеста для всего манифеста (см. заголовок `SHA1-Digest-Manifest`
в примере выше).

При проверке подписанного JAR-файла дайджесты каждого из его файлов вычисляются заново и
сравниваются с дайджестами, записанными в манифесте, чтобы убедиться, что содержимое
JAR-файла не изменилось с момента подписания. В качестве дополнительной проверки значения
дайджеста самого манифеста вычисляются заново и сравниваются со значениями, записанными в
файле подписи.

#### Файл блока подписи

В дополнение к файлу подписи при подписании JAR-файла в каталог `META-INF` автоматически
помещается **файл блока подписи** (*signature block file*). В отличие от манифеста или файла
подписи, файлы блока подписи не предназначены для чтения человеком.

Файл блока подписи содержит два элемента, необходимых для проверки:

- цифровую подпись JAR-файла, сгенерированную закрытым ключом подписавшего;
- сертификат, содержащий открытый ключ подписавшего, для использования любым, кто хочет
  проверить подписанный JAR-файл.

Имена файлов блока подписи обычно имеют расширение `.DSA`, указывающее, что они созданы
алгоритмом цифровой подписи по умолчанию (Digital Signature Algorithm). Возможны и другие
расширения, если для подписи используются ключи, связанные с другим стандартным алгоритмом.

### Подпись JAR-файлов

Для подписания JAR-файлов и постановки отметки времени на подпись используется инструмент
JAR Signing and Verification Tool. Он вызывается командой `jarsigner`, поэтому для краткости
будем называть его «Jarsigner».

Чтобы подписать JAR-файл, у вас сначала должен быть закрытый ключ. Закрытые ключи и связанные
с ними сертификаты открытого ключа хранятся в защищённых паролем базах данных, называемых
**хранилищами ключей** (*keystores*). Хранилище ключей может содержать ключи многих
потенциальных подписавших. Каждый ключ в хранилище может быть идентифицирован **алиасом**
(*alias*), которым обычно является имя владеющего ключом подписавшего. Например, ключ,
принадлежащий Rita Jones, мог бы иметь алиас «rita».

Базовая форма команды для подписания JAR-файла:

```bash
jarsigner jar-file alias
```

В этой команде:

- `jar-file` — путь к подписываемому JAR-файлу.
- `alias` — алиас, идентифицирующий закрытый ключ, которым подписывается JAR-файл, и
  связанный с ключом сертификат.

Инструмент Jarsigner запросит у вас пароли для хранилища ключей и алиаса.

Эта базовая форма команды предполагает, что используемое хранилище ключей находится в файле
с именем `.keystore` в вашем домашнем каталоге. Команда создаст файл подписи и файл блока
подписи с именами `x.SF` и `x.DSA` соответственно, где `x` — первые восемь букв алиаса,
приведённые к верхнему регистру. Эта базовая команда **перезапишет** исходный JAR-файл
подписанным.

На практике может потребоваться одна или несколько доступных опций команды. Например,
постановка отметки времени на подпись рекомендуется, чтобы любой инструмент развёртывания
вашего приложения мог убедиться, что сертификат, использованный для подписи JAR-файла, был
действителен на момент подписания. Если отметка времени не включена, Jarsigner выдаёт
предупреждение.

Опции предшествуют пути `jar-file`. Доступные опции описаны в таблице:

#### Опции команды Jarsigner

| Опция | Описание |
|-------|----------|
| `-keystore url` | Указывает хранилище ключей, если вы не хотите использовать базу `.keystore` по умолчанию. |
| `-sigfile file` | Задаёт базовое имя для файлов `.SF` и `.DSA`, если вы не хотите брать его из алиаса. `file` должно состоять только из заглавных букв (A–Z), цифр (0–9), дефиса (-) и подчёркивания (_). |
| `-signedjar file` | Задаёт имя генерируемого подписанного JAR-файла, если вы не хотите, чтобы исходный неподписанный файл был перезаписан подписанным. |
| `-tsa url` | Генерирует отметку времени для подписи, используя удостоверяющий центр времени (Time Stamping Authority, TSA), идентифицируемый по URL. |
| `-tsacert alias` | Генерирует отметку времени для подписи, используя сертификат открытого ключа TSA, идентифицируемый по `alias`. |
| `-altsigner class` | Указывает, что для постановки отметки времени на подпись используется альтернативный механизм подписания. Полностью квалифицированное имя класса идентифицирует используемый класс. |
| `-altsignerpath classpathlist` | Указывает путь к классу, идентифицированному опцией `altsigner`, и любым JAR-файлам, от которых этот класс зависит. |

#### Пример

Рассмотрим пару примеров подписания JAR-файла инструментом Jarsigner. В этих примерах будем
считать, что:

- ваш алиас — «johndoe»;
- хранилище ключей, которое вы хотите использовать, находится в файле «mykeys» в текущем
  рабочем каталоге;
- TSA, которым вы хотите ставить отметку времени на подпись, находится по адресу
  `http://tsa.url.example.com`.

При этих предположениях вы могли бы использовать такую команду для подписания JAR-файла
`app.jar`:

```bash
jarsigner -keystore mykeys -tsa http://tsa.url.example.com app.jar johndoe
```

У вас запросят ввод паролей и для хранилища ключей, и для вашего алиаса. Поскольку эта
команда не использует опцию `-sigfile`, создаваемые файлы `.SF` и `.DSA` получат имена
`JOHNDOE.SF` и `JOHNDOE.DSA`. Поскольку команда не использует опцию `-signedjar`,
результирующий подписанный файл перезапишет исходную версию `app.jar`.

Посмотрим, что произошло бы при другой комбинации опций:

```bash
jarsigner -keystore mykeys -sigfile SIG -signedjar SignedApp.jar \
          -tsacert testalias app.jar johndoe
```

Файлы подписи и блока подписи получили бы имена `SIG.SF` и `SIG.DSA` соответственно, а
подписанный JAR-файл `SignedApp.jar` был бы помещён в текущий каталог. Исходный неподписанный
JAR-файл остался бы без изменений. Кроме того, подпись была бы помечена отметкой времени с
помощью сертификата открытого ключа TSA, идентифицированного как `testalias`.

### Проверка подписанных JAR-файлов

Обычно проверка подписанных JAR-файлов — задача вашей среды выполнения Java™. Браузер
проверяет подписанные апплеты, которые скачивает. Подписанные приложения, запускаемые с
опцией `-jar` интерпретатора, проверяются средой выполнения.

Однако вы можете проверить подписанные JAR-файлы и сами, используя инструмент `jarsigner`.
Это может понадобиться, например, чтобы протестировать подготовленный вами подписанный
JAR-файл.

Базовая команда для проверки подписанного JAR-файла:

```bash
jarsigner -verify jar-file
```

Эта команда проверит подпись JAR-файла и убедится, что файлы в архиве не изменились с момента
подписания. При успешной проверке вы увидите сообщение:

```
jar verified.
```

Если вы попытаетесь проверить неподписанный JAR-файл, появится сообщение:

```
jar is unsigned. (signatures missing or not parsable)
```

Если проверка не удалась, выводится соответствующее сообщение. Например, если содержимое
JAR-файла изменилось после подписания, при попытке проверки появится сообщение, похожее на:

```
jarsigner: java.lang.SecurityException: invalid SHA1
signature file digest for test/classes/Manifest.class
```

> **Примечание.** JDK трактует подписанный JAR-файл как неподписанный, если он использует
> любой алгоритм, указанный в свойстве безопасности `jdk.jar.disabledAlgorithms` в файле
> `java.home/lib/security/java.security` (где `java.home` — каталог, в который установлена
> ваша JRE).

## Использование JAR-related API

Платформа Java содержит несколько классов для работы с JAR-файлами. Некоторые из этих API:

- пакет [**java.util.jar**](https://docs.oracle.com/javase/8/docs/api/java/util/jar/package-summary.html)
- класс [**java.net.JarURLConnection**](https://docs.oracle.com/javase/8/docs/api/java/net/JarURLConnection.html)
- класс [**java.net.URLClassLoader**](https://docs.oracle.com/javase/8/docs/api/java/net/URLClassLoader.html)

Чтобы дать представление о возможностях, которые открывают эти API, этот раздел проводит вас
по внутреннему устройству примера приложения под названием JarRunner.

### Пример — приложение JarRunner

JarRunner позволяет запустить приложение, упакованное в JAR-файл, указав URL JAR-файла в
командной строке. Например, если приложение `TargetApp` упаковано в JAR-файл по адресу
`http://www.example.com/TargetApp.jar`, его можно запустить такой командой:

```bash
java JarRunner http://www.example.com/TargetApp.jar
```

Чтобы JarRunner работал, он должен уметь выполнять следующие задачи, и все они достигаются с
помощью этих API:

- получить доступ к удалённому JAR-файлу и установить с ним коммуникационную связь;
- проинспектировать манифест JAR-файла, чтобы узнать, какой из классов архива является
  главным;
- загрузить классы из JAR-файла.

Приложение JarRunner состоит из двух классов — `JarRunner` и `JarClassLoader`. `JarRunner`
делегирует большинство задач по работе с JAR классу `JarClassLoader`. `JarClassLoader`
расширяет класс `java.net.URLClassLoader`.

#### Класс JarClassLoader

Класс `JarClassLoader` расширяет `java.net.URLClassLoader`, чтобы загружать классы и ресурсы
из JAR-файлов, доступных по URL.

Используемые API:

- **`URLClassLoader`** — родительский класс, предназначенный для загрузки классов из
  источников на основе URL (каталогов или JAR-файлов):

```java
public JarClassLoader(URL url) {
    super(new URL[] { url });
    this.url = url;
}
```

- **`JarURLConnection`** — представляет коммуникационную связь с JAR-файлом и даёт доступ к
  информации манифеста. Использует формат JAR URL:

```
jar:http://www.example.com/jarfile.jar!/
jar:http://www.example.com/jarfile.jar!/mypackage/myclass.class
```

  Разделитель `!/` указывает на весь JAR-файл целиком; всё, что после него, ссылается на
  конкретное содержимое.

- **`java.util.jar.Attributes`** — отображает имена заголовков в манифестах JAR на их
  строковые значения.

**Поиск главного класса.** Метод `getMainClassName()` получает точку входа:

```java
public String getMainClassName() throws IOException {
    URL u = new URL("jar", "", url + "!/");
    JarURLConnection uc = (JarURLConnection)u.openConnection();
    Attributes attr = uc.getMainAttributes();
    return attr != null
                   ? attr.getValue(Attributes.Name.MAIN_CLASS)
                   : null;
}
```

Процесс:

1. Строит JAR URL с разделителем `!/`.
2. Открывает `JarURLConnection` к JAR-файлу.
3. Получает атрибуты манифеста через `getMainAttributes()`.
4. Извлекает значение заголовка `Main-Class`.

**Вызов главного класса.** Метод `invokeClass()` загружает и выполняет главный класс с
помощью рефлексии:

```java
public void invokeClass(String name, String[] args)
    throws ClassNotFoundException, NoSuchMethodException, InvocationTargetException
{
    Class c = loadClass(name);
    Method m = c.getMethod("main", new Class[] { args.getClass() });
    m.setAccessible(true);
    int mods = m.getModifiers();
    if (m.getReturnType() != void.class || !Modifier.isStatic(mods) ||
        !Modifier.isPublic(mods)) {
        throw new NoSuchMethodException("main");
    }
    m.invoke(null, new Object[] { args });
}
```

Процесс:

1. Загружает класс унаследованным методом `loadClass()`.
2. С помощью рефлексии получает метод `main`.
3. Проверяет, что он `public`, `static` и возвращает `void`.
4. Вызывает метод с аргументами командной строки.

#### Класс JarRunner

Приложение JarRunner запускается так:

```bash
java JarRunner url [arguments]
```

`JarRunner` служит точкой входа: он берёт URL и аргументы из командной строки, передаёт их в
`JarClassLoader` и запускает упакованное в JAR приложение.

**Шаг 1. Разобрать URL из командной строки:**

```java
public static void main(String[] args) {
    if (args.length < 1) {
        usage();
    }
    URL url = null;
    try {
        url = new URL(args[0]);
    } catch (MalformedURLException e) {
        fatal("Invalid URL: " + args[0]);
    }
```

`JarRunner` проверяет, что аргумент-URL был передан, и создаёт объект `java.net.URL`.

**Шаг 2. Создать экземпляр JarClassLoader:**

```java
JarClassLoader cl = new JarClassLoader(url);
```

**Шаг 3. Получить имя класса точки входа:**

```java
String name = null;
try {
    name = cl.getMainClassName();
} catch (IOException e) {
    System.err.println("I/O error while loading JAR file:");
    e.printStackTrace();
    System.exit(1);
}
if (name == null) {
    fatal("Specified jar file does not contain a 'Main-Class'" +
          " manifest attribute");
}
```

`JarRunner` вызывает метод `getMainClassName()` класса `JarClassLoader`, чтобы определить
класс-точку входа из манифеста JAR.

**Шаг 4. Подготовить аргументы и вызвать приложение:**

```java
// Получить аргументы для приложения
String[] newArgs = new String[args.length - 1];
System.arraycopy(args, 1, newArgs, 0, newArgs.length);

// Вызвать главный класс приложения
try {
    cl.invokeClass(name, newArgs);
} catch (ClassNotFoundException e) {
    fatal("Class not found: " + name);
} catch (NoSuchMethodException e) {
    fatal("Class does not define a 'main' method: " + name);
} catch (InvocationTargetException e) {
    e.getTargetException().printStackTrace();
    System.exit(1);
}
```

`JarRunner` извлекает аргументы приложения (пропуская URL в `args[0]`) и вызывает
`invokeClass()`. Этот метод использует рефлексию (через `Method.invoke()`), чтобы загрузить
главный класс и выполнить его метод `main()` с переданными аргументами.

JAR-формат — важная часть механизма расширений платформы Java. Подробнее об этом аспекте
JAR-файлов можно узнать в трейле The Extension Mechanism этого руководства.
