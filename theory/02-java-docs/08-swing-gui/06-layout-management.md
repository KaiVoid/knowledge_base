**Менеджер компоновки** (*layout manager*) — это объект, реализующий интерфейс `LayoutManager` и
определяющий размер и положение компонентов внутри контейнера. Хотя компоненты могут давать
подсказки о своём размере и выравнивании, последнее слово о размере и положении компонентов внутри
контейнера остаётся за его менеджером компоновки.

> **Примечание.** Этот урок посвящён написанию кода компоновки вручную, что может быть непросто.
> Если вам неинтересны все детали управления компоновкой, удобнее использовать менеджер
> `GroupLayout` в связке с инструментом-построителем GUI. Один из таких инструментов — среда
> NetBeans IDE. Если же вы хотите писать код вручную и не желаете применять `GroupLayout`, то в
> качестве следующего по гибкости и мощности менеджера рекомендуется `GridBagLayout`.

## Наглядный обзор менеджеров компоновки (A Visual Guide to Layout Managers)

Несколько классов AWT и Swing предоставляют менеджеры компоновки общего назначения:

- `BorderLayout`
- `BoxLayout`
- `CardLayout`
- `FlowLayout`
- `GridBagLayout`
- `GridLayout`
- `GroupLayout`
- `SpringLayout`

Ниже кратко описано, какой графический интерфейс получается с каждым из этих менеджеров и где
найти подробную страницу-инструкцию по каждому.

### Сравнение менеджеров компоновки

| Менеджер | Краткая характеристика |
|---|---|
| **BorderLayout** | Размещает компоненты максимум в пяти областях: верх, низ, левый край, правый край и центр. Всё лишнее пространство уходит в центр. Используется по умолчанию для панели содержимого (*content pane*) любых фреймов, апплетов и диалогов. Панели инструментов на базе `JToolBar`, которые нужно перетаскивать с исходной позиции, должны создаваться в контейнере с `BorderLayout`. |
| **BoxLayout** | Выстраивает компоненты в один ряд или одну колонку. Учитывает запрошенные **максимальные** размеры компонентов и позволяет их выравнивать. |
| **CardLayout** | Позволяет реализовать область, которая в разные моменты содержит разные компоненты. Часто управляется выпадающим списком (*combo box*): его состояние определяет, какую панель (группу компонентов) показывает `CardLayout`. Альтернатива — панель с вкладками (*tabbed pane*) с тем же функционалом, но готовым GUI. |
| **FlowLayout** | Менеджер по умолчанию для каждого `JPanel`. Просто выстраивает компоненты в один ряд, начиная новый ряд, если контейнер недостаточно широк. |
| **GridBagLayout** | Изощрённый и гибкий менеджер. Выравнивает компоненты, помещая их в сетку ячеек, причём компонент может занимать несколько ячеек. Строки сетки могут иметь разную высоту, а колонки — разную ширину. |
| **GridLayout** | Делает группу компонентов одинакового размера и показывает их в заданном числе строк и колонок. |
| **GroupLayout** | Разработан для инструментов-построителей GUI, но пригоден и для ручного кода. Работает с горизонтальной и вертикальной компоновкой по отдельности; каждое измерение задаётся независимо, поэтому каждый компонент описывается в компоновке дважды. |
| **SpringLayout** | Гибкий менеджер, рассчитанный на построители GUI. Позволяет задавать точные соотношения между краями компонентов (например, что левый край одного компонента находится на некотором, в т.ч. динамически вычисляемом, расстоянии от правого края другого). Компонует дочерние элементы по набору ограничений. |

> **Примечание.** Если вы не хотите вникать во все детали управления компоновкой, предпочтительнее
> использовать `GroupLayout` вместе с построителем GUI (например, NetBeans IDE). Иначе, при ручном
> кодировании без `GroupLayout`, рекомендуется `GridBagLayout` как следующий по гибкости и мощности.

## Использование менеджеров компоновки (Using Layout Managers)

Этот раздел рассматривает типичные задачи, связанные с использованием менеджеров компоновки:

- установка менеджера компоновки;
- добавление компонентов в контейнер;
- задание подсказок о размере и выравнивании;
- расстановка пространства между компонентами;
- задание ориентации контейнера;
- советы по выбору менеджера компоновки;
- сторонние менеджеры компоновки.

> Если вы хотите создавать GUI на JavaFX, см. [Working With Layouts in JavaFX](https://docs.oracle.com/javase/8/javafx/layout-tutorial/index.html).

### Установка менеджера компоновки

Как правило, заботиться о менеджере компоновки приходится только для `JPanel` и панелей
содержимого (*content pane*). Каждый объект `JPanel` инициализируется с `FlowLayout`, если при
создании не указано иное. Панели содержимого по умолчанию используют `BorderLayout`. Менеджер
по умолчанию можно сменить. Однако, если вы не используете `JToolBar`, менеджеры `FlowLayout` и
`BorderLayout` пригодны лишь для прототипирования: любому реальному приложению потребуется
переустановить менеджер компоновки (и делать это лучше подходящим инструментом, а не вручную).

Менеджер можно задать в конструкторе `JPanel`:

```java
JPanel panel = new JPanel(new BorderLayout());
```

После создания контейнера менеджер устанавливается методом `setLayout`:

```java
Container contentPane = frame.getContentPane();
contentPane.setLayout(new FlowLayout());
```

Хотя использование менеджеров компоновки **настоятельно рекомендуется**, можно обойтись и без них.
Установив свойство layout контейнера в `null`, вы заставляете контейнер не использовать менеджер.
Эта стратегия называется **абсолютным позиционированием** (*absolute positioning*): тогда вы
обязаны указывать размер и положение каждого компонента. Минус — плохая адаптация при изменении
размера верхнеуровневого контейнера, а также при различиях между пользователями и системами
(разные размеры шрифтов, разные локали).

### Добавление компонентов в контейнер

Аргументы метода `add` зависят от используемого менеджера. Некоторые менеджеры (например,
`GroupLayout`) вообще не требуют явного добавления компонента. Например, `BorderLayout` требует
указать область (одной из констант, определённых в `BorderLayout`):

```java
pane.add(aComponent, BorderLayout.PAGE_START);
```

Некоторые менеджеры (`GridBagLayout`, `SpringLayout`) требуют сложной настройки; многие же просто
размещают компоненты в порядке их добавления в контейнер.

Swing-контейнеры, отличные от `JPanel` и панелей содержимого, обычно предоставляют собственный API,
который следует использовать вместо `add`. Например, в панель прокрутки (*scroll pane*) компонент
добавляют через конструктор `JScrollPane` или метод `setViewportView`, а не напрямую. (Для
любопытных: панели прокрутки используют менеджер `ScrollPaneLayout`.)

### Подсказки о размере и выравнивании

Иногда нужно настроить размерные подсказки компонента. Для этого задают один или несколько из
минимального, предпочтительного и максимального размеров — методами `setMinimumSize`,
`setPreferredSize`, `setMaximumSize`, либо переопределяя `getMinimumSize`, `getPreferredSize`,
`getMaximumSize`. Пример — сделать максимальный размер компонента неограниченным:

```java
component.setMaximumSize(new Dimension(Integer.MAX_VALUE,
                                       Integer.MAX_VALUE));
```

Многие менеджеры **не** учитывают запрошенный максимальный размер. Однако `BoxLayout` и
`SpringLayout` его учитывают. Кроме того, `GroupLayout` позволяет задать минимальный,
предпочтительный или максимальный размер явно, не трогая сам компонент.

Помимо размера, можно давать подсказки о выравнивании — методами `setAlignmentX` / `setAlignmentY`
или переопределением `getAlignmentX` / `getAlignmentY`. Большинство менеджеров их игнорирует, но
`BoxLayout` учитывает (примеры — в разделе про `BoxLayout`).

### Расстановка пространства между компонентами

На величину промежутков между видимыми компонентами влияют три фактора:

- **Менеджер компоновки.** Одни менеджеры автоматически добавляют пространство, другие нет;
  некоторые позволяют задать величину промежутка.
- **Невидимые компоненты.** Можно создавать лёгкие компоненты, которые ничего не рисуют, но
  занимают место в GUI. Часто их применяют в контейнерах с `BoxLayout`.
- **Пустые рамки** (*empty borders*). Независимо от менеджера, можно влиять на видимый объём
  пространства, добавляя компонентам пустые рамки. Лучшие кандидаты — компоненты без рамки по
  умолчанию: панели и метки.

### Задание ориентации контейнера

Свойство `componentOrientation` позволяет указать, что компонент должен использовать ориентацию,
отличную от привычной «слева направо, сверху вниз» (многие языки имеют другую ориентацию). Для
кнопки-переключателя ориентация может быть подсказкой look-and-feel о расположении значка и текста;
для контейнера — подсказкой менеджеру компоновки.

Ориентацию задают методом `setComponentOrientation` (только сам компонент) или
`applyComponentOrientation` (включая дочерние компоненты). Аргументом может быть константа вроде
`ComponentOrientation.RIGHT_TO_LEFT` или вызов `ComponentOrientation.getOrientation(Locale)`.
Например, ниже все `JComponent` инициализируются арабской локалью, а затем ориентация задаётся
панели содержимого и всем компонентам внутри неё:

```java
JComponent.setDefaultLocale(new Locale("ar"));
JFrame frame = new JFrame();
...
Container contentPane = frame.getContentPane();
contentPane.applyComponentOrientation(
    ComponentOrientation.getOrientation(
        contentPane.getLocale()));
```

Стандартные менеджеры, поддерживающие ориентацию компонентов: `FlowLayout`, `BorderLayout`,
`BoxLayout`, `GridBagLayout` и `GridLayout`.

> **Примечание.** Следует позаботиться о том, чтобы ориентация применялась к отрисовщикам
> (*renderers*), редакторам и прочим компонентам, недостижимым обычным обходом иерархии вложения.

### Советы по выбору менеджера компоновки

> Ещё раз: настоятельно рекомендуется создавать менеджеры компоновки построителем GUI (например,
> GUI-построителем Matisse в NetBeans IDE), а не вручную. Сценарии ниже даны для понимания, какой
> тип менеджера применяется в разных ситуациях.

- **Нужно отобразить компонент на максимально возможном пространстве.** Если он единственный в
  контейнере — `GridLayout` или `BorderLayout`. Иначе подойдут `BorderLayout` или `GridBagLayout`.
  С `BorderLayout` «прожорливый» компонент кладут в центр; с `GridBagLayout` задают
  `fill=GridBagConstraints.BOTH`; ещё вариант — `BoxLayout` с очень большими предпочтительным и
  максимальным размерами компонента.
- **Несколько компонентов в компактном ряду в их естественном размере.** Сгруппируйте их в
  `JPanel` с `FlowLayout` по умолчанию или с `BoxLayout`. Хорош и `SpringLayout`.
- **Несколько компонентов одинакового размера в строках и колонках.** Идеален `GridLayout`.
- **Несколько компонентов в ряд или колонку с разными промежутками, выравниванием или размерами.**
  Идеален `BoxLayout`.
- **Выровненные колонки, как в форме (колонка меток рядом с колонкой текстовых полей).**
  Естественный выбор — `SpringLayout`. Класс `SpringUtilities` из примеров Tutorial определяет
  метод `makeCompactGrid`, упрощающий выравнивание нескольких строк и колонок.
- **Сложная компоновка с множеством компонентов.** Либо очень гибкий менеджер (`GridBagLayout`
  или `SpringLayout`), либо группировка компонентов в `JPanel` (каждая со своим менеджером).

### Сторонние менеджеры компоновки

Сообщество Swing создало дополнительные менеджеры. Список не претендует на полноту, но наиболее
популярны:

- [MiGLayout](http://www.miglayout.com/);
- [FormLayout от Karsten Lentzsch](http://www.jgoodies.com/).

> Ещё в JDK 1.1 был введён интерфейс `LayoutManager2`, расширяющий `LayoutManager` и добавляющий
> поддержку максимального размера и выравнивания. `LayoutManager2` добавляет методы
> `addLayoutComponent`, принимающий `Object`, и `invalidateLayout`. Любому современному менеджеру
> компоновки нужны уведомления `LayoutManager2`, поэтому он его реализует.

## Как работает управление компоновкой (How Layout Management Works)

Пример последовательности управления компоновкой контейнера на базе `LayoutManager2`:

1. Менеджеры компоновки делают по сути две вещи:
   - вычисляют минимальный/предпочтительный/максимальный размеры контейнера;
   - размещают дочерние элементы контейнера.

   Это делается на основе заданных ограничений, свойств контейнера (например, внутренних отступов —
   *insets*) и минимальных/предпочтительных/максимальных размеров дочерних элементов. Если дочерний
   элемент сам контейнер, его собственный менеджер используется для получения его размеров и
   компоновки.

2. Контейнер может быть **валидным** (`isValid()` возвращает `true`) или невалидным. Чтобы
   контейнер был валиден, все его дочерние элементы должны быть уже скомпонованы и тоже валидны.
   Метод `Container.validate` валидирует невалидный контейнер: он запускает компоновку для
   контейнера и всех вложенных контейнеров вниз по иерархии и помечает контейнер как валидный.

3. После создания компонент по умолчанию невалиден. Метод `Window.pack` валидирует окно и впервые
   компонует иерархию компонентов окна.

В итоге, чтобы определить наилучший размер контейнера, система определяет размеры контейнеров в
самом низу иерархии вложения; затем эти размеры «всплывают» вверх по иерархии, в конце концов
определяя общий размер контейнера.

Если размер компонента меняется (например, после смены шрифта), компонент нужно пересчитать и
перерисовать вызовом методов `revalidate` и `repaint`. Оба метода **потокобезопасны** — их не
обязательно вызывать из потока обработки событий (*event-dispatching thread*).

Когда вы вызываете `revalidate` для компонента, запрос поднимается вверх по иерархии вложения, пока
не встретит контейнер (панель прокрутки или верхнеуровневый контейнер), которого изменение размера
компонента касаться не должно (определяется методом `isValidateRoot`). Затем этот контейнер
компонуется заново, что корректирует размер компонента и всех затронутых компонентов.

## Менеджер `BorderLayout`

Класс `BorderLayout` делит контейнер на пять областей, задаваемых константами:

- `PAGE_START`
- `PAGE_END`
- `LINE_START`
- `LINE_END`
- `CENTER`

> **Замечание о версиях.** До выпуска JDK 1.4 предпочтительные имена областей были иными — от
> сторон света (например, `BorderLayout.NORTH` для верхней области) до более многословных вариантов.
> Перечисленные выше константы предпочтительны: они стандартны и позволяют программам подстраиваться
> под языки с разной ориентацией.

При увеличении окна центральная область получает как можно больше места; остальные области
расширяются лишь настолько, чтобы заполнить всё доступное пространство. Часто используются лишь
одна-две области (только центр, или центр и низ).

Поскольку панели содержимого по умолчанию используют `BorderLayout`, устанавливать менеджер не нужно:

```java
//Container pane = aFrame.getContentPane()...
JButton button = new JButton("Button 1 (PAGE_START)");
pane.add(button, BorderLayout.PAGE_START);

//Сделать центральный компонент большим — это типичное
//применение BorderLayout.
button = new JButton("Button 2 (CENTER)");
button.setPreferredSize(new Dimension(200, 100));
pane.add(button, BorderLayout.CENTER);

button = new JButton("Button 3 (LINE_START)");
pane.add(button, BorderLayout.LINE_START);

button = new JButton("Long-Named Button 4 (PAGE_END)");
pane.add(button, BorderLayout.PAGE_END);

button = new JButton("5 (LINE_END)");
pane.add(button, BorderLayout.LINE_END);
```

Расположение указывается как один из аргументов `add`. Если компонент не виден в контейнере с
`BorderLayout`, убедитесь, что его расположение задано и в той же области нет другого компонента.

Во всех примерах руководства компонент — первый аргумент `add` (предпочтительно):

```java
add(component, BorderLayout.CENTER)  //предпочтительно
```

В других программах встречается компонент вторым аргументом:

```java
add(BorderLayout.CENTER, component)  //допустимо, но устаревший стиль
    или
add("Center", component)             //допустимо, но чревато ошибками
```

### API `BorderLayout`

| Конструктор или метод | Назначение |
|---|---|
| `BorderLayout(int horizontalGap, int verticalGap)` | Создаёт компоновку с заданными промежутками между компонентами. |
| `setHgap(int)` | Задаёт горизонтальный промежуток между компонентами. |
| `setVgap(int)` | Задаёт вертикальный промежуток между компонентами. |

## Менеджер `BoxLayout`

`BoxLayout` либо ставит компоненты друг над другом, либо в ряд — на ваш выбор. Можно считать его
улучшенной версией `FlowLayout`.

Пример из `ListDialog.java`:

```java
JScrollPane listScroller = new JScrollPane(list);
listScroller.setPreferredSize(new Dimension(250, 80));
listScroller.setAlignmentX(LEFT_ALIGNMENT);
...
//Компонуем метку и панель прокрутки сверху вниз.
JPanel listPane = new JPanel();
listPane.setLayout(new BoxLayout(listPane, BoxLayout.PAGE_AXIS));
JLabel label = new JLabel(labelText);
...
listPane.add(label);
listPane.add(Box.createRigidArea(new Dimension(0,5)));
listPane.add(listScroller);
listPane.setBorder(BorderFactory.createEmptyBorder(10,10,10,10));

//Компонуем кнопки слева направо.
JPanel buttonPane = new JPanel();
buttonPane.setLayout(new BoxLayout(buttonPane, BoxLayout.LINE_AXIS));
buttonPane.setBorder(BorderFactory.createEmptyBorder(0, 10, 10, 10));
buttonPane.add(Box.createHorizontalGlue());
buttonPane.add(cancelButton);
buttonPane.add(Box.createRigidArea(new Dimension(10, 0)));
buttonPane.add(setButton);

//Собираем всё вместе, используя BorderLayout панели содержимого.
Container contentPane = getContentPane();
contentPane.add(listPane, BorderLayout.CENTER);
contentPane.add(buttonPane, BorderLayout.PAGE_END);
```

### Оси компоновки

Первый аргумент конструктора `BoxLayout` — управляемый контейнер, второй — ось:

- **`PAGE_AXIS`** — компоненты компонуются в направлении, в котором строки текста идут по странице
  (по свойству `ComponentOrientation` контейнера). При ориентации «слева направо» это компоновка
  сверху вниз.
- **`LINE_AXIS`** — компоненты компонуются в направлении строки текста. При ориентации «слева
  направо» — слева направо.

> **Замечание о версиях.** До JDK 1.4 констант для локализуемого задания оси не было: указывали
> `X_AXIS` (слева направо) или `Y_AXIS` (сверху вниз). Константы `LINE_AXIS` и `PAGE_AXIS`
> предпочтительны, так как позволяют программам подстраиваться под языки с разной ориентацией.

При компоновке сверху вниз `BoxLayout` пытается задать каждому компоненту его предпочтительную
высоту. Если суммарная предпочтительная высота не совпадает с вертикальным пространством, компоненты
растягиваются или сжимаются, заполняя место (с учётом их минимального и максимального размеров).
Лишнее место остаётся внизу контейнера. Предпочтительная ширина контейнера равна максимальной
предпочтительной ширине дочерних элементов; если контейнер шире, ширина каждого компонента
доводится до ширины контейнера (минус отступы), а если максимальный размер компонента меньше ширины
контейнера, в игру вступает выравнивание по оси X.

Значения выравнивания по X:

- **0.0 (`Component.LEFT_ALIGNMENT`)** — выравнивание по левому краю;
- **0.5 (`Component.CENTER_ALIGNMENT`)** — по центру;
- **1.0 (`Component.RIGHT_ALIGNMENT`)** — по правому краю.

### Невидимые компоненты-заполнители

Каждый компонент в `BoxLayout` примыкает к соседям. Чтобы получить промежуток, добавляют пустую
рамку или вставляют невидимые компоненты. Их помогает создать класс `Box`, определяющий вложенный
класс `Box.Filler` — прозрачный компонент, ничего не рисующий.

| Тип заполнителя | Ограничения размера | Как создать |
|---|---|---|
| Жёсткая область (*rigid area*) | Фиксированный размер по обоим измерениям | `Box.createRigidArea(size)` |
| Клей (*glue*), горизонтальный | Растёт/сжимается по горизонтали, минимален по вертикали | `Box.createHorizontalGlue()` |
| Клей, вертикальный | Растёт/сжимается по вертикали, минимален по горизонтали | `Box.createVerticalGlue()` |
| Собственный `Box.Filler` | Как задано | `new Box.Filler(minSize, prefSize, maxSize)` |

**Жёсткая область** — фиксированный промежуток между двумя компонентами. Чтобы вставить 5 пикселей
между компонентами в горизонтальном боксе:

```java
container.add(firstComponent);
container.add(Box.createRigidArea(new Dimension(5,0)));
container.add(secondComponent);
```

> **Примечание.** Класс `Box` предоставляет ещё один вид заполнителя — распорку (*strut*),
> вертикальную или горизонтальную. К сожалению, распорки имеют неограниченную максимальную высоту
> (или ширину). Поэтому рекомендуется использовать жёсткие области вместо распорок.

**Клей** указывает, куда уходит лишнее пространство. Это «эластичный клей»: растягивается, но не
занимает места, пока компоненты не раздвинут. Чтобы лишнее место в горизонтальном боксе ушло между
двумя компонентами, а не справа от всех:

```java
container.add(firstComponent);
container.add(Box.createHorizontalGlue());
container.add(secondComponent);
```

**Собственный `Box.Filler`** — компонент с любыми минимальным, предпочтительным и максимальным
размерами. Например, не менее 5 пикселей между компонентами и минимальная высота контейнера 100:

```java
container.add(firstComponent);
Dimension minSize = new Dimension(5, 100);
Dimension prefSize = new Dimension(5, 100);
Dimension maxSize = new Dimension(Short.MAX_VALUE, 100);
container.add(new Box.Filler(minSize, prefSize, maxSize));
container.add(secondComponent);
```

### Исправление проблем выравнивания

Бывают два типа проблем: (1) у группы компонентов одинаковое выравнивание, но его хочется изменить;
(2) компоненты с разным выравниванием по умолчанию оказываются смещены (например, левый край метки
по умолчанию выровнен по центру панели).

**Общее правило.** Все компоненты под управлением `BoxLayout` сверху вниз должны иметь одинаковое
выравнивание по X; все компоненты под управлением `BoxLayout` слева направо — одинаковое
выравнивание по Y.

Пример: выровнять две кнопки по нижнему краю:

```java
button1.setAlignmentY(Component.BOTTOM_ALIGNMENT);
button2.setAlignmentY(Component.BOTTOM_ALIGNMENT);
```

У большинства компонентов выравнивание по X и Y центральное, но у кнопок, выпадающих списков,
меток и пунктов меню по X — `LEFT_ALIGNMENT`. Часто достаточно сделать «провинившийся» компонент
центрированным:

```java
label.setAlignmentX(Component.CENTER_ALIGNMENT);
```

### Задание размеров компонентов

`BoxLayout` учитывает запрошенные минимальный, предпочтительный и максимальный размеры. Размеры
меняют двумя способами — вызовом `set...Size` или переопределением `get...Size`:

```java
comp.setMinimumSize(new Dimension(50, 25));
comp.setPreferredSize(new Dimension(50, 25));
comp.setMaximumSize(new Dimension(Short.MAX_VALUE,
                                  Short.MAX_VALUE));
```

```java
//в подклассе компонента:
public Dimension getMaximumSize() {
    size = getPreferredSize();
    size.width = Short.MAX_VALUE;
    return size;
}
```

> **Примечание.** В отличие от `BoxLayout`, многие менеджеры не учитывают максимальный размер.
> Например, кнопка внизу `BorderLayout` будет шире своей предпочтительной ширины независимо от
> максимального размера. `BoxLayout` же никогда не делает кнопку шире её максимального размера.

Чтобы отследить проблемы с размером: (1) добавьте яркую линейную рамку, чтобы увидеть фактический
размер; (2) выведите через `System.out.println` минимальный/предпочтительный/максимальный размеры:

```java
comp.setBorder(BorderFactory.createCompoundBorder(
                   BorderFactory.createLineBorder(Color.red),
                   comp.getBorder()));
```

### API `BoxLayout`

| Конструктор или метод | Назначение |
|---|---|
| `BoxLayout(Container, int)` | Создаёт `BoxLayout` для указанного контейнера; целочисленный аргумент задаёт ось (`LINE_AXIS` — слева направо, `PAGE_AXIS` — сверху вниз при ориентации по умолчанию). |
| `Box(int)` | Создаёт `Box` — контейнер с `BoxLayout` по заданной оси. |
| `static Box createHorizontalBox()` | Создаёт `Box`, компонующий компоненты слева направо. |
| `static Box createVerticalBox()` | Создаёт `Box`, компонующий компоненты сверху вниз. |
| `Component createRigidArea(Dimension)` | Создаёт жёсткий (фиксированный) компонент-заполнитель. |
| `Component createHorizontalGlue()` / `createVerticalGlue()` / `createGlue()` | Создают клей. |
| `Component createHorizontalStrut()` / `createVerticalStrut()` | Создают распорку (рекомендуются жёсткие области вместо них). |
| `Box.Filler(Dimension, Dimension, Dimension)` | Создаёт компонент с заданными минимальным, предпочтительным и максимальным размерами (в этом порядке). |
| `void changeShape(Dimension, Dimension, Dimension)` *(в `Box.Filler`)* | Меняет размеры `Box.Filler`; компоновка обновляется. |

## Менеджер `CardLayout`

Класс `CardLayout` управляет двумя или более компонентами (обычно экземплярами `JPanel`),
делящими одно пространство отображения. Выбор отображаемого компонента обычно даётся пользователю
через выпадающий список. Альтернатива — панель с вкладками (*tabbed pane*), у которой свой готовый
GUI, поэтому она проще.

Концептуально каждый управляемый компонент — как игральная карта в стопке: видна только верхняя.
Карту для показа выбирают: запросив первую или последнюю (в порядке добавления); листая колоду
назад или вперёд; указав карту по имени. Демо `CardLayoutDemo` использует последний способ.

```java
//Где объявлены переменные экземпляра:
JPanel cards;
final static String BUTTONPANEL = "Card with JButtons";
final static String TEXTPANEL = "Card with JTextField";

//Где инициализируются управляемые CardLayout компоненты:
//Создаём «карты».
JPanel card1 = new JPanel();
...
JPanel card2 = new JPanel();
...

//Создаём панель, содержащую «карты».
cards = new JPanel(new CardLayout());
cards.add(card1, BUTTONPANEL);
cards.add(card2, TEXTPANEL);
```

При добавлении компонента в контейнер с `CardLayout` указывают строку-идентификатор. Эти же строки
используются в выпадающем списке.

```java
//Где собирается GUI:
//Помещаем JComboBox в JPanel для более аккуратного вида.
JPanel comboBoxPane = new JPanel(); //использует FlowLayout
String comboBoxItems[] = { BUTTONPANEL, TEXTPANEL };
JComboBox cb = new JComboBox(comboBoxItems);
cb.setEditable(false);
cb.addItemListener(this);
comboBoxPane.add(cb);
...
pane.add(comboBoxPane, BorderLayout.PAGE_START);
pane.add(cards, BorderLayout.CENTER);
...

//Метод из реализации интерфейса ItemListener;
//обрабатывает выбор элемента выпадающего списка.
public void itemStateChanged(ItemEvent evt) {
    CardLayout cl = (CardLayout)(cards.getLayout());
    cl.show(cards, (String)evt.getItem());
}
```

Первый аргумент `show` — контейнер, которым управляет `CardLayout`; второй — строка-идентификатор
компонента (та же, что при добавлении).

### API `CardLayout`

| Метод | Назначение |
|---|---|
| `first(Container parent)` | Переходит к первой карте контейнера. |
| `next(Container parent)` | Переходит к следующей карте; если текущая — последняя, переходит к первой. |
| `previous(Container parent)` | Переходит к предыдущей карте; если текущая — первая, переходит к последней. |
| `last(Container parent)` | Переходит к последней карте. |
| `show(Container parent, String name)` | Переходит к компоненту, добавленному с указанным `name` (через `addLayoutComponent`). |

## Менеджер `FlowLayout`

Класс `FlowLayout` — очень простой менеджер, используемый по умолчанию объектами `JPanel`. Он
ставит компоненты в ряд в их предпочтительном размере. Если ширины контейнера не хватает на все
компоненты в одном ряду, используются несколько рядов. Если контейнер шире нужного, ряд по
умолчанию центрируется по горизонтали. Чтобы выровнять ряд влево или вправо, используют конструктор
с аргументом выравнивания; другой конструктор задаёт вертикальные и горизонтальные отступы.

```java
FlowLayout experimentLayout = new FlowLayout();
...
    compsToExperiment.setLayout(experimentLayout);

    compsToExperiment.add(new JButton("Button 1"));
    compsToExperiment.add(new JButton("Button 2"));
    compsToExperiment.add(new JButton("Button 3"));
    compsToExperiment.add(new JButton("Long-Named Button 4"));
    compsToExperiment.add(new JButton("5"));
```

Применение ориентации «слева направо»:

```java
        compsToExperiment.setComponentOrientation(
                ComponentOrientation.LEFT_TO_RIGHT);
```

### API `FlowLayout`

| Конструктор | Назначение |
|---|---|
| `FlowLayout()` | Центрированное выравнивание, горизонтальный и вертикальный промежутки по 5 пикселей. |
| `FlowLayout(int align)` | Заданное выравнивание, промежутки по 5 пикселей. `align` может быть `FlowLayout.LEADING`, `FlowLayout.CENTER` или `FlowLayout.TRAILING`. При ориентации «слева направо» (по умолчанию) `LEADING` — выравнивание влево, `TRAILING` — вправо. |
| `FlowLayout(int align, int hgap, int vgap)` | Заданное выравнивание и указанные горизонтальный (`hgap`) и вертикальный (`vgap`) промежутки в пикселях. |

## Менеджер `GridBagLayout`

`GridBagLayout` — один из самых гибких и сложных менеджеров. Он размещает компоненты в сетке строк
и колонок, позволяя компонентам занимать несколько строк или колонок. Не все строки одинаковой
высоты, не все колонки одинаковой ширины. По сути, `GridBagLayout` помещает компоненты в
прямоугольные ячейки сетки, а затем по предпочтительным размерам компонентов определяет размеры
ячеек.

При увеличении окна нижний ряд получает всё новое вертикальное пространство, а новое горизонтальное
делится поровну между колонками. Это поведение определяется **весами** (*weights*), которые
программа назначает компонентам.

Размер и положение задаются через **ограничения** (*constraints*) для каждого компонента.
Предпочтительный способ — использовать вариант `Container.add`, передавая ему объект
`GridBagConstraints`:

```java
JPanel pane = new JPanel(new GridBagLayout());
GridBagConstraints c = new GridBagConstraints();

//Для каждого добавляемого компонента:
//...Создать компонент...
//...Задать поля экземпляра GridBagConstraints...
pane.add(theComponent, c);
```

Один и тот же экземпляр `GridBagConstraints` можно переиспользовать для нескольких компонентов, но
**не рекомендуется**: легко внести трудноуловимые ошибки, забыв сбросить поля.

> Дальнейшее обсуждение предполагает, что `GridBagLayout` управляет контейнером с ориентацией
> «слева направо».

Поля экземпляра `GridBagConstraints`:

- **`gridx`, `gridy`** — строка и колонка верхнего левого угла компонента. Левая колонка — `gridx=0`,
  верхняя строка — `gridy=0`. `GridBagConstraints.RELATIVE` (значение по умолчанию) — поместить
  компонент сразу справа (`gridx`) или ниже (`gridy`) предыдущего. Рекомендуется указывать `gridx` и
  `gridy` явно — компоновка предсказуемее.
- **`gridwidth`, `gridheight`** — число колонок (`gridwidth`) или строк (`gridheight`) в области
  компонента. Это число ячеек, а не пикселей. По умолчанию 1. `GridBagConstraints.REMAINDER` —
  компонент последний в ряду/колонке; `GridBagConstraints.RELATIVE` — предпоследний. Рекомендуется
  задавать значения явно.
  > **Примечание.** `GridBagLayout` не позволяет компонентам занимать несколько строк, если
  > компонент не в крайней левой колонке или вы не задали положительные `gridx` и `gridy`.
- **`fill`** — когда область компонента больше его запрошенного размера, определяет, как изменить
  размер. Значения: `NONE` (по умолчанию), `HORIZONTAL`, `VERTICAL`, `BOTH`.
- **`ipadx`, `ipady`** — внутренний отступ (на сколько увеличить размер компонента). По умолчанию 0.
  Ширина будет не меньше минимальной плюс `ipadx*2` (отступ с обеих сторон), аналогично высота — плюс
  `ipady*2`.
- **`insets`** — внешний отступ (минимальное пространство между компонентом и краями его области),
  задаётся объектом `Insets`. По умолчанию отсутствует.
- **`anchor`** — когда компонент меньше своей области, определяет, где его разместить. Значения:
  `CENTER` (по умолчанию), `PAGE_START`, `PAGE_END`, `LINE_START`, `LINE_END`, `FIRST_LINE_START`,
  `FIRST_LINE_END`, `LAST_LINE_END`, `LAST_LINE_START`.
  > **Замечание о версиях.** Константы `PAGE_*` и `*LINE_*` введены в 1.4; ранее использовались
  > значения по сторонам света (например, `NORTHEAST`). Рекомендуются новые константы — проще
  > локализация.
- **`weightx`, `weighty`** — веса для распределения пространства между колонками (`weightx`) и
  строками (`weighty`); важны для поведения при изменении размера. Пока хотя бы один вес не ненулевой,
  компоненты собираются в центре контейнера (при весе 0.0 лишнее место уходит между сеткой и краями).
  Обычно используют крайние значения 0.0 и 1.0. Чем больше число, тем больше места получает
  строка/колонка. Лишнее место тяготеет к правой колонке и нижнему ряду.

Код примера:

```java
JButton button;
pane.setLayout(new GridBagLayout());
GridBagConstraints c = new GridBagConstraints();
if (shouldFill) {
                //естественная высота, максимальная ширина
                c.fill = GridBagConstraints.HORIZONTAL;
}

button = new JButton("Button 1");
if (shouldWeightX) {
                   c.weightx = 0.5;
}
c.fill = GridBagConstraints.HORIZONTAL;
c.gridx = 0;
c.gridy = 0;
pane.add(button, c);

button = new JButton("Button 2");
c.fill = GridBagConstraints.HORIZONTAL;
c.weightx = 0.5;
c.gridx = 1;
c.gridy = 0;
pane.add(button, c);

button = new JButton("Button 3");
c.fill = GridBagConstraints.HORIZONTAL;
c.weightx = 0.5;
c.gridx = 2;
c.gridy = 0;
pane.add(button, c);

button = new JButton("Long-Named Button 4");
c.fill = GridBagConstraints.HORIZONTAL;
c.ipady = 40;      //сделать этот компонент высоким
c.weightx = 0.0;
c.gridwidth = 3;
c.gridx = 0;
c.gridy = 1;
pane.add(button, c);

button = new JButton("5");
c.fill = GridBagConstraints.HORIZONTAL;
c.ipady = 0;       //сброс к значению по умолчанию
c.weighty = 1.0;   //запросить всё лишнее вертикальное пространство
c.anchor = GridBagConstraints.PAGE_END; //низ области
c.insets = new Insets(10,0,0,0);  //отступ сверху
c.gridx = 1;       //выровнено с кнопкой 2
c.gridwidth = 2;   //шириной в 2 колонки
c.gridy = 2;       //третья строка
pane.add(button, c);
```

> **Примечание.** Раньше в примерах Tutorial ограничения задавались иначе — методом
> `setConstraints` объекта `GridBagLayout` (`gridbag.setConstraints(button, c); pane.add(button);`).
> Однако рекомендуется `Container.add` — код чище.

Все компоненты `pane` максимально широки в своих ячейках за счёт `fill = HORIZONTAL`. При увеличении
окна колонки растут пропорционально (у всех компонентов первого ряда `weightx = 0.5`). Высоту
набирает только последний ряд, так как лишь у кнопки 5 `weighty > 0`.

### API `GridBagLayout`

У классов `GridBagLayout` и `GridBagConstraints` по одному конструктору без аргументов. Вместо
вызова методов `GridBagConstraints` манипулируют его полями. Обычно у `GridBagLayout` вызывают лишь
метод `setConstraints`.

## Менеджер `GridLayout`

`GridLayout` размещает компоненты в сетке ячеек. Каждый компонент занимает всё доступное место в
своей ячейке, и все ячейки строго одинакового размера. При изменении размера окна `GridLayout`
меняет размер ячеек так, чтобы они были как можно больше при доступном пространстве.

```java
GridLayout experimentLayout = new GridLayout(0,2);

compsToExperiment.setLayout(experimentLayout);
compsToExperiment.add(new JButton("Button 1"));
compsToExperiment.add(new JButton("Button 2"));
compsToExperiment.add(new JButton("Button 3"));
compsToExperiment.add(new JButton("Long-Named Button 4"));
compsToExperiment.add(new JButton("5"));
```

Этот конструктор создаёт сетку с двумя колонками и нужным числом строк. Промежутки задают методами
`setHgap` и `setVgap`:

```java
applyButton.addActionListener(new ActionListener(){
    public void actionPerformed(ActionEvent e){
        //Получить значение горизонтального промежутка
        String horGap = (String)horGapComboBox.getSelectedItem();
        //Получить значение вертикального промежутка
        String verGap = (String)verGapComboBox.getSelectedItem();
        //Установить горизонтальный промежуток
        experimentLayout.setHgap(Integer.parseInt(horGap));
        //Установить вертикальный промежуток
        experimentLayout.setVgap(Integer.parseInt(verGap));
        //Перекомпоновать кнопки
        experimentLayout.layoutContainer(compsToExperiment);
    }
});
```

### API `GridLayout`

| Конструктор | Назначение |
|---|---|
| `GridLayout(int rows, int cols)` | Сетка с заданным числом строк и колонок; все компоненты одного размера. Одно (но не оба) из `rows` и `cols` может быть 0 — тогда в ряду/колонке любое число объектов. |
| `GridLayout(int rows, int cols, int hgap, int vgap)` | То же плюс горизонтальные (между колонками) и вертикальные (между строками) промежутки. |

## Менеджер `GroupLayout`

`GroupLayout` разработан для построителей GUI вроде Matisse (в NetBeans IDE), но хорошо подходит и
для ручного кода.

**Принцип проектирования: независимые измерения.** `GroupLayout` работает с горизонтальной и
вертикальной компоновкой по отдельности. Каждое измерение задаётся независимо: при описании
горизонтальной компоновки не нужно думать о вертикальной, и наоборот. Поэтому каждый компонент
описывается в компоновке **дважды**; если забыть — `GroupLayout` сгенерирует исключение.

**Организация: иерархические группы.** Используются два типа расположений — последовательное и
параллельное — в сочетании с иерархической композицией:

1. При **последовательном** (*sequential*) расположении компоненты ставятся друг за другом (как
   `BoxLayout` или `FlowLayout` вдоль одной оси).
2. При **параллельном** (*parallel*) — друг на друге в одном пространстве. По вертикальной оси они
   могут выравниваться по базовой линии (*baseline*), верху или низу; по горизонтальной — влево,
   вправо или по центру (если размеры разные).

Обычно компоненты, параллельные в одном измерении, последовательны в другом (чтобы не накладывались).
Расположения вкладываются иерархически через **группы компоновки** (*layout groups*): группа —
последовательная или параллельная — может содержать компоненты, другие группы и промежутки. Размер
последовательной группы — сумма размеров элементов; параллельной — размер наибольшего элемента.

Псевдокод для трёх компонентов в ряд и четвёртого (C4) под C3:

```
горизонтально = последовательная группа { c1, c2, параллельная группа (LEFT) { c3, c4 } }
вертикально   = последовательная группа { параллельная группа (BASELINE) { c1, c2, c3 }, c4 }
```

### Промежутки (Gaps)

Промежуток — это как невидимый компонент заданного размера. Кроме явных промежутков, `GroupLayout`
определяет **автоматические** промежутки, соответствующие *предпочтительным расстояниям* между
соседними компонентами (или между компонентом и краем контейнера). Их размер вычисляется динамически
по текущему look-and-feel (через класс `LayoutStyle`). Преимущества: не нужно указывать размеры в
пикселях, и они подстраиваются под look-and-feel.

Различают промежуток между двумя компонентами (`addPreferredGap`) и между компонентом и краем
контейнера (`addContainerGap`). Есть три типа промежутков между компонентами — **связанные**
(`RELATED`), **несвязанные** (`UNRELATED`) и **с отступом** (`INDENT`) — из перечисления
`LayoutStyle.ComponentPlacement`. Несвязанные чуть больше связанных. Автоматическую вставку
промежутков включают вызовами `setAutoCreateGaps(true)` и `setAutoCreateContainerGaps(true)`.

### Написание кода

```java
GroupLayout layout = new GroupLayout(panel);
panel.setLayout(layout);
```

Включаем автоматические промежутки:

```java
layout.setAutoCreateGaps(true);
layout.setAutoCreateContainerGaps(true);
```

Корневые группы для каждого измерения задают методами `setHorizontalGroup` и `setVerticalGroup`;
группы создают `createSequentialGroup` и `createParallelGroup`; компоненты добавляют `addComponent`:

```java
layout.setHorizontalGroup(
   layout.createSequentialGroup()
      .addComponent(c1)
      .addComponent(c2)
      .addGroup(layout.createParallelGroup(GroupLayout.Alignment.LEADING)
           .addComponent(c3)
           .addComponent(c4))
);
layout.setVerticalGroup(
   layout.createSequentialGroup()
      .addGroup(layout.createParallelGroup(GroupLayout.Alignment.BASELINE)
           .addComponent(c1)
           .addComponent(c2)
           .addComponent(c3))
      .addComponent(c4)
);
```

Выравнивание параллельных групп задаётся константами перечисления `GroupLayout.Alignment`:
`LEADING`, `TRAILING`, `CENTER`, `BASELINE`. Они зависят от ориентации компонентов (слева направо
или справа налево). По умолчанию используется `LEADING`. `BASELINE` допустимо только в вертикальном
измерении.

> **Примечание.** Выравнивание имеет смысл лишь для компонентов разного размера; компоненты
> одинакового размера выравниваются автоматически при любой константе.

Замечания: компонент не нужно добавлять в контейнер напрямую — это делает `addComponent`; метод
`addComponent` возвращает группу, на которой вызван, что позволяет составлять цепочки вызовов; код
стоит отступать по иерархии групп.

### Размер и изменяемость компонентов

Число изменяемых компонентов не ограничено. Размер каждого ограничен тремя значениями: минимальным,
предпочтительным и максимальным. Если не задано явно, компоновка спрашивает их у компонента
(`getMinimumSize()`, `getPreferredSize()`, `getMaximumSize()`). Поведение можно переопределить
методом `addComponent(Component comp, int min, int pref, int max)`.

Сделать компонент изменяемым (от нуля до бесконечности; `Short.MAX_VALUE` — «бесконечность»):

```
group.addComponent(component, 0, GroupLayout.DEFAULT_SIZE, Short.MAX_VALUE) ...
```

Сделать компонент фиксированного размера:

```
group.addComponent(component, GroupLayout.PREFERRED_SIZE, GroupLayout.DEFAULT_SIZE,
          GroupLayout.PREFERRED_SIZE) ...
```

Размер и изменяемость применимы и к промежуткам, включая предпочтительные. Например, предпочтительный
промежуток как «пружина», расталкивающая компоненты к противоположным краям:

```java
layout.createSequentialGroup()
    .addComponent(c1)
    .addPreferredGap(LayoutStyle.ComponentPlacement.RELATED,
                     GroupLayout.DEFAULT_SIZE, Short.MAX_VALUE)
    .addComponent(c2);
```

**Размеры в параллельных группах.** Изменяемые элементы в параллельной группе растягиваются до
размера наибольшего элемента, выравниваясь по одному размеру. Чтобы дать двум компонентам одинаковую
ширину (третий параметр `createParallelGroup` равен `false` — группа не растягивается):

```java
layout.createParallelGroup(GroupLayout.Alignment.LEADING, false)
  .addComponent(c3, GroupLayout.DEFAULT_SIZE, GroupLayout.DEFAULT_SIZE, Short.MAX_VALUE)
  .addComponent(c4, GroupLayout.DEFAULT_SIZE, GroupLayout.DEFAULT_SIZE, Short.MAX_VALUE);
```

Оба компонента сделаны изменяемыми ради независимости от платформы и локализации: при переводе или
смене платформы любой из них может оказаться больше, и они подстраиваются друг под друга.

**Одинаковый размер несвязанных компонентов.** Для компонентов из разных групп (например, кнопки OK
и Cancel) служит метод `linkSize`: размер связанных компонентов задаётся по наибольшему из них:

```java
layout.linkSize(SwingConstants.HORIZONTAL, c3, c4);
```

### Изменения GUI во время выполнения

Метод `replace(Component existingComponent, Component newComponent)` заменяет существующий компонент
новым без пересоздания групп. Метод `setHonorsVisibility(boolean)` глобально задаёт обработку
невидимых компонентов: `true` (по умолчанию) — невидимые компоненты как будто отсутствуют; `false` —
для них резервируется место. `setHonorsVisibility(Component, Boolean)` настраивает поведение для
конкретного компонента.

## Пример `GroupLayout` (A GroupLayout Example)

Пример строит диалог «Find» (поиск) с кроссплатформенной компоновкой без указания размеров в
пикселях. Диалог содержит метку, текстовое поле, четыре флажка (Case, Whole, Wrap, Back) и две
кнопки (Find, Cancel).

**Горизонтальная компоновка.** Слева направо три последовательных группы: метка; группа с текстовым
полем и флажками; группа с двумя кнопками.

**Вертикальная компоновка.** Сверху вниз: группа по базовой линии с меткой, текстовым полем и
кнопкой Find; группа с остальными флажками и кнопкой Cancel.

Полный код компоновки:

```java
GroupLayout layout = new GroupLayout(getContentPane());
getContentPane().setLayout(layout);
layout.setAutoCreateGaps(true);
layout.setAutoCreateContainerGaps(true);

layout.setHorizontalGroup(layout.createSequentialGroup()
    .addComponent(label)
    .addGroup(layout.createParallelGroup(GroupLayout.Alignment.LEADING)
        .addComponent(textField)
        .addGroup(layout.createSequentialGroup()
            .addGroup(layout.createParallelGroup(GroupLayout.Alignment.LEADING)
                .addComponent(caseCheckBox)
                .addComponent(wholeCheckBox))
            .addGroup(layout.createParallelGroup(GroupLayout.Alignment.LEADING)
                .addComponent(wrapCheckBox)
                .addComponent(backCheckBox))))
    .addGroup(layout.createParallelGroup(GroupLayout.Alignment.LEADING)
        .addComponent(findButton)
        .addComponent(cancelButton))
);
layout.linkSize(SwingConstants.HORIZONTAL, findButton, cancelButton);

layout.setVerticalGroup(layout.createSequentialGroup()
    .addGroup(layout.createParallelGroup(GroupLayout.Alignment.BASELINE)
        .addComponent(label)
        .addComponent(textField)
        .addComponent(findButton))
    .addGroup(layout.createParallelGroup(GroupLayout.Alignment.LEADING)
        .addGroup(layout.createSequentialGroup()
            .addGroup(layout.createParallelGroup(GroupLayout.Alignment.BASELINE)
                .addComponent(caseCheckBox)
                .addComponent(wrapCheckBox))
            .addGroup(layout.createParallelGroup(GroupLayout.Alignment.BASELINE)
                .addComponent(wholeCheckBox)
                .addComponent(backCheckBox)))
        .addComponent(cancelButton))
);
```

Ключевые особенности: нет размеров в пикселях (истинно кроссплатформенная компоновка); автоматические
промежутки через `setAutoCreateGaps`/`setAutoCreateContainerGaps`; текстовое поле автоматически
изменяет размер, а кнопки сохраняют одинаковый размер через `linkSize`; выравнивание по базовой линии
для аккуратного вида. Полный исходный код — в `Find.java`.

## Менеджер `SpringLayout`

Класс `SpringLayout` добавлен в JDK 1.4 для поддержки компоновки в построителях GUI. Это очень
гибкий, но **очень низкоуровневый** менеджер, который следует использовать только с построителем
GUI, а не писать вручную.

**Как работает.** Пружинные компоновки задают направленные соотношения, или **ограничения**
(*constraints*), между краями компонентов (например, левый край одного компонента на 5 пикселей от
правого края другого). Положение каждого края зависит лишь от положения одного другого края: новое
ограничение для края отменяет прежнее.

> **Важно.** В отличие от многих менеджеров, `SpringLayout` **не** задаёт положение управляемых
> компонентов автоматически. При ручном кодировании инициализируйте положение компонентов,
> ограничивая запад/восток и север/юг; возможно, придётся явно задать и размер контейнера.

Края компонентов соединяются экземплярами `Spring`. У каждой пружины четыре свойства: минимальное,
предпочтительное, максимальное и текущее (актуальное) значение. Пружины компонента собираются в
объект `SpringLayout.Constraints`.

**Пружина как механическая модель.** Пружина даёт корректирующую силу при сжатии или растяжении от
предпочтительного значения. Сила — линейная функция расстояния, с двумя константами (на сжатие и на
растяжение). Разность предпочтительного и минимального значений — лёгкость сжатия; разность
максимального и предпочтительного — лёгкость растяжения.

### Пример SpringDemo

**SpringDemo1** — базовая настройка без пружин положения; окно слишком маленькое, все компоненты в
точке (0,0).

```java
public class SpringDemo1 {
    public static void main(String[] args) {
        ...
        Container contentPane = frame.getContentPane();
        SpringLayout layout = new SpringLayout();
        contentPane.setLayout(layout);
        contentPane.add(new JLabel("Label: "));
        contentPane.add(new JTextField("Text field", 15));
        ...
        frame.pack();
        frame.setVisible(true);
    }
}
```

**SpringDemo2** — задаём положение компонентов методом `putConstraint`:

```java
// Метка в точке (5,5).
layout.putConstraint(SpringLayout.WEST, label,
                     5,
                     SpringLayout.WEST, contentPane);
layout.putConstraint(SpringLayout.NORTH, label,
                     5,
                     SpringLayout.NORTH, contentPane);

// Текстовое поле: (<правый край метки> + 5, 5).
layout.putConstraint(SpringLayout.WEST, textField,
                     5,
                     SpringLayout.EAST, label);
layout.putConstraint(SpringLayout.NORTH, textField,
                     5,
                     SpringLayout.NORTH, contentPane);
```

**SpringDemo3** — задаём размер контейнера (правый и нижний края на 5 пикселей дальше самого правого
и самого нижнего компонентов):

```java
layout.putConstraint(SpringLayout.EAST, contentPane,
                     5,
                     SpringLayout.EAST, textField);
layout.putConstraint(SpringLayout.SOUTH, contentPane,
                     5,
                     SpringLayout.SOUTH, textField);
```

**SpringDemo4** — то же через `SpringLayout.Constraints` и `Spring` напрямую:

```java
SpringLayout.Constraints contentPaneCons =
        layout.getConstraints(contentPane);
contentPaneCons.setX(Spring.sum(Spring.constant(5),
                          contentPaneCons
                          .getConstraint(SpringLayout.WEST)));
contentPaneCons.setY(Spring.sum(Spring.constant(5),
                          contentPaneCons
                          .getConstraint(SpringLayout.NORTH)));
```

### Константы краёв

- `SpringLayout.NORTH` — верхний край ограничивающего прямоугольника;
- `SpringLayout.SOUTH` — нижний край;
- `SpringLayout.EAST` — правый край;
- `SpringLayout.WEST` — левый край;
- `SpringLayout.BASELINE` — базовая линия компонента;
- `SpringLayout.HORIZONTAL_CENTER` — горизонтальный центр;
- `SpringLayout.VERTICAL_CENTER` — вертикальный центр.

### Пружины и размер компонента

`SpringLayout` автоматически устанавливает пружины для высоты и ширины каждого компонента — обёртки
над `getMinimumSize`, `getPreferredSize` и `getMaximumSize`, которые отслеживают эти значения. Если
`getMaximumSize` и `getPreferredSize` возвращают одно значение, компонент не растягивается (так
реализованы `JLabel` и `JButton`). Компоненты вроде `JTextField` возвращают `Integer.MAX_VALUE` для
максимального размера — они растут до любого размера. При увеличении контейнера лишнее место уходит
пружинам, способным расти.

Соотношения, поддерживаемые объектом `Constraints` (пружины есть только для *x*, *y*, *width*,
*height*):

```
west  = x
north = y
east  = x + width
south = y + height
```

### Вспомогательные методы для сеток

Поскольку настройка отдельных пружин громоздка, в `SpringUtilities.java` есть два метода — `makeGrid`
и `makeCompactGrid`. Оба группируют компоненты в строки и колонки и через `Spring.max` создают
пружину ширины/высоты, вмещающую все компоненты строки/колонки. Различие: `makeCompactGrid`
использует общую пружину ширины/высоты для всех компонентов колонки/строки (компоненты могут быть
разного размера); `makeGrid` делит пружины ширины и высоты между всеми компонентами (все одного
размера).

**SpringGrid** — `makeGrid` делает все ячейки одинаково большими (как `GridLayout`):

```java
JPanel panel = new JPanel(new SpringLayout());
for (int i = 0; i < 9; i++) {
    JTextField textField = new JTextField(Integer.toString(i));
    //при i==4 — длинный текст в поле
    panel.add(textField);
}
...
SpringUtilities.makeGrid(panel,
                         3, 3,     //строки, колонки
                         5, 5,     //initialX, initialY
                         5, 5);    //xPad, yPad
```

**SpringCompactGrid** — `makeCompactGrid`; компоненты одной колонки одной ширины, одной строки —
одной высоты:

```java
JPanel panel = new JPanel(new SpringLayout());

int rows = 10;
int cols = 10;
for (int r = 0; r < rows; r++) {
    for (int c = 0; c < cols; c++) {
        int anInt = (int) Math.pow(r, c);
        JTextField textField = new JTextField(Integer.toString(anInt));
        panel.add(textField);
    }
}

SpringUtilities.makeCompactGrid(panel,     //родитель
                                rows, cols,
                                3, 3,       //initX, initY
                                3, 3);      //xPad, yPad
```

**SpringForm** — связывает метки с компонентами (метки в одной колонке, компоненты в другой); при
изменении размера гибкие компоненты забирают лишнее место:

```java
String[] labels = {"Name: ", "Fax: ", "Email: ", "Address: "};
int numPairs = labels.length;

JPanel p = new JPanel(new SpringLayout());
for (int i = 0; i < numPairs; i++) {
    JLabel l = new JLabel(labels[i], JLabel.TRAILING);
    p.add(l);
    JTextField textField = new JTextField(10);
    l.setLabelFor(textField);
    p.add(textField);
}

SpringUtilities.makeCompactGrid(p,
                                numPairs, 2, //строки, колонки
                                6, 6,        //initX, initY
                                6, 6);       //xPad, yPad
```

### API `SpringLayout.Constraints`

| Конструктор или метод | Назначение |
|---|---|
| `SpringLayout.Constraints()` | Создаёт с пустыми (`null`) пружинами (заменяются подходящими по умолчанию). |
| `SpringLayout.Constraints(Spring, Spring)` | С заданными пружинами X и Y. |
| `SpringLayout.Constraints(Spring, Spring, Spring, Spring)` | С пружинами X, Y, ширины и высоты. |
| `Spring getConstraint(String)` | Получить пружину края (NORTH, SOUTH, EAST, WEST). |
| `Spring getHeight()` / `getWidth()` / `getX()` / `getY()` | Получить пружину высоты / ширины / X / Y. |
| `void setConstraint(String, Spring)` | Задать пружину края. |
| `void setHeight(Spring)` / `setWidth(Spring)` / `setX(Spring)` / `setY(Spring)` | Задать соответствующую пружину. |

### API `SpringLayout`

| Конструктор или метод | Назначение |
|---|---|
| `SpringLayout()` | Создаёт экземпляр `SpringLayout`. |
| `SpringLayout.Constraints getConstraints(Component)` | Получить набор пружин компонента. |
| `Spring getConstraint(String, Component)` | Получить пружину края компонента (NORTH, SOUTH, EAST, WEST). |
| `void putConstraint(String, Component, int, String, Component)` | Задать соотношение между краями двух компонентов (целое расстояние). |
| `void putConstraint(String, Component, Spring, String, Component)` | То же с объектом `Spring` вместо целого. |

### API `Spring`

| Метод | Назначение |
|---|---|
| `static Spring constant(int)` | Пружина с одинаковыми мин./предпоч./макс. значениями. |
| `static Spring constant(int, int, int)` | Пружина с заданными мин., предпоч. и макс. значениями. |
| `static Spring sum(Spring, Spring)` | Пружина-сумма двух пружин. |
| `static Spring max(Spring, Spring)` | Пружина, значение которой не меньше значений обоих аргументов. |
| `static Spring minus(Spring)` | Пружина в обратном направлении. |
| `int getMinimumValue()` / `getPreferredValue()` / `getMaximumValue()` / `getValue()` | Получить мин./предпоч./макс./текущее значение. |
| `setValue(int)` | Задать текущее значение. |

> **Примечание о постоянных пружинах.** Несмотря на название, пружины из `constant` изменяемы:
> `SpringLayout` может скорректировать «постоянную» пружину ради компоновки. Не переиспользуйте
> постоянные пружины, если только не хотите, чтобы они всегда были одинаковыми.

## Создание собственного менеджера компоновки (Creating a Custom Layout Manager)

Перед созданием собственного менеджера убедитесь, что ни один существующий не подходит: `GridBagLayout`,
`SpringLayout`, `BoxLayout` достаточно гибки для многих случаев; можно найти менеджеры из других
источников; можно упростить компоновку, группируя компоненты в контейнеры (панели).

Собственный менеджер — это класс, реализующий интерфейс `LayoutManager` (напрямую или через его
подинтерфейс `LayoutManager2`). Каждый менеджер обязан реализовать как минимум пять методов
интерфейса `LayoutManager`:

- **`void addLayoutComponent(String, Component)`** — вызывается методами `add` класса `Container`.
  Менеджеры, не связывающие строки с компонентами, обычно ничего здесь не делают.
- **`void removeLayoutComponent(Component)`** — вызывается методами `remove` и `removeAll`.
  Переопределяется для очистки внутреннего состояния, связанного с компонентом.
- **`Dimension preferredLayoutSize(Container)`** — вызывается методом `getPreferredSize`. Должен
  вычислить идеальный размер контейнера (с учётом внутренних рамок из `getInsets`).
- **`Dimension minimumLayoutSize(Container)`** — вызывается методом `getMinimumSize`. Должен
  вычислить минимальный размер контейнера (с учётом `getInsets`).
- **`void layoutContainer(Container)`** — позиционирует и масштабирует компоненты, вызывая их методы
  `setSize`, `setLocation`, `setBounds` (он не рисует компоненты). Должен учитывать внутренние рамки
  (`getInsets`) и при необходимости ориентацию (`getComponentOrientation`). Нельзя полагаться, что
  `preferredLayoutSize` или `minimumLayoutSize` будут вызваны до `layoutContainer`.

Помимо этих пяти методов, менеджеры обычно реализуют хотя бы один открытый конструктор и метод
`toString`.

Для поддержки ограничений компонентов, максимальных размеров или выравнивания менеджер должен
реализовывать интерфейс `LayoutManager2` (его реализуют почти все современные менеджеры). Он
добавляет пять методов:

- `addLayoutComponent(Component, Object)`;
- `getLayoutAlignmentX(Container)`;
- `getLayoutAlignmentY(Container)`;
- `invalidateLayout(Container)`;
- `maximumLayoutSize(Container)`.

Важнейшие — `addLayoutComponent(Component, Object)` (добавляет компонент с объектом-ограничением) и
`invalidateLayout(Container)` (сбрасывает кэшированную информацию компоновки).

> Создавая собственные менеджеры, остерегайтесь хранить ссылки на `Component`, которые более не
> являются дочерними для `Container`: переопределяйте `removeLayoutComponent`, чтобы очистить
> кэшированное состояние.

**Пример.** `CustomLayoutDemo` использует менеджер `DiagonalLayout` (исходник —
`DiagonalLayout.java`), который располагает компоненты по диагонали слева направо, по одному в строке.

## Без менеджера компоновки (абсолютное позиционирование)

Хотя обойтись без менеджера компоновки можно, по возможности менеджер следует использовать: он
упрощает адаптацию к зависящему от look-and-feel виду компонентов, к разным размерам шрифтов, к
изменению размера контейнера и к разным локалям; менеджеры легко переиспользуются.

Абсолютное позиционирование оправдано, если размер компонентов не зависит от размера контейнера,
шрифта, look-and-feel или языка. Примеры — рабочие панели (*desktop panes*) с внутренними фреймами и
панели-разделители (*split panes*), выполняющие собственные расчёты размера и положения.

Создание контейнера без менеджера компоновки:

1. Установить менеджер в `null` вызовом `setLayout(null)`.
2. Вызвать метод `setBounds` для каждого дочернего компонента.
3. Вызвать метод `repaint`.

```java
pane.setLayout(null);

JButton b1 = new JButton("one");
JButton b2 = new JButton("two");
JButton b3 = new JButton("three");

pane.add(b1);
pane.add(b2);
pane.add(b3);

Insets insets = pane.getInsets();
Dimension size = b1.getPreferredSize();
b1.setBounds(25 + insets.left, 5 + insets.top,
             size.width, size.height);
size = b2.getPreferredSize();
b2.setBounds(55 + insets.left, 40 + insets.top,
             size.width, size.height);
size = b3.getPreferredSize();
b3.setBounds(150 + insets.left, 15 + insets.top,
             size.width + 50, size.height + 20);

...//В методе main:
Insets insets = frame.getInsets();
frame.setSize(300 + insets.left + insets.right,
              125 + insets.top + insets.bottom);
```

**Недостаток.** Контейнеры с абсолютно позиционированными компонентами могут вести себя плохо при
изменении размера окна.

## Решение типовых проблем компоновки (Solving Common Layout Problems)

**Проблема: как задать точный размер компонента?**

- Некоторые более современные менеджеры позволяют переопределить размер, заданный компонентом —
  проверьте, поддерживает ли это ваш менеджер.
- Убедитесь, что точный размер действительно нужен: у каждого Swing-компонента свой предпочтительный
  размер, зависящий от шрифта и look-and-feel, поэтому задавать точный размер часто нет смысла.
- Если компонент не управляется менеджером, задайте размер методом `setSize` или `setBounds`. Иначе
  дайте размерные подсказки и используйте менеджер, который их учитывает.
- При наследовании Swing-компонента давайте подсказки, переопределяя `getMinimumSize`,
  `getPreferredSize`, `getMaximumSize` (можно получить значения по умолчанию через `super.get...Size()`
  и скорректировать). Удобно для текстовых компонентов: фиксированная ширина, высота — от содержимого.
  Иногда с `GridBagLayout` и текстовыми полями возникают проблемы: если контейнер меньше
  предпочтительного размера, используется минимальный, и поля могут сильно сжиматься.
- Другой способ дать подсказки — методы `setMinimumSize`, `setPreferredSize`, `setMaximumSize`.
- Задав новые подсказки уже видимому компоненту, вызовите `revalidate` (чтобы иерархия вложения была
  скомпонована заново), затем `repaint`.

**Проблема: компонент не появляется после добавления в контейнер.**

- После добавления компонента нужно вызвать `revalidate` и `repaint`, иначе он не отобразится.

**Проблема: собственный компонент слишком мал.**

- Реализует ли компонент `getPreferredSize` и `getMinimumSize`? Возвращают ли они правильные значения?
- Используете ли вы менеджер, способный занять всё доступное место? См. «Советы по выбору менеджера
  компоновки».
