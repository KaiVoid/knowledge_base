# Урок 4. Всё о датаграммах

**Трейл:** Custom Networking · **Оригинал:** [All About Datagrams](https://docs.oracle.com/javase/tutorial/networking/datagrams/index.html)
**Связанные области:** [[17-rest-web]] · **Вопросы:** rest-web

> Перевод официального руководства Oracle (The Java Tutorials, JDK 8). Объединяет страницы
> *All About Datagrams*, *What Is a Datagram?*, *Writing a Datagram Client and Server*
> и *Broadcasting to Multiple Recipients*.

Некоторым приложениям, которые вы пишете для взаимодействия по сети, не требуется надёжный
канал «точка-точка» (*point-to-point*), предоставляемый протоколом TCP. Напротив, вашим
приложениям может пригодиться режим связи, при котором доставляются независимые порции
информации, причём ни сам факт их доставки, ни порядок прибытия не гарантируются.

Протокол UDP обеспечивает режим сетевого взаимодействия, при котором приложения отправляют
друг другу пакеты данных — **датаграммы** (*datagrams*). Датаграмма — это независимое,
самодостаточное сообщение, отправленное по сети, доставка, время доставки и содержимое которого
не гарантируются. Классы `DatagramPacket` и `DatagramSocket` из пакета `java.net` реализуют
системно-независимое взаимодействие через датаграммы на основе UDP.

> **Примечание.** Многие межсетевые экраны (*firewalls*) и маршрутизаторы настроены так, чтобы
> не пропускать UDP-пакеты. Если у вас возникают трудности с подключением к сервису за пределами
> вашего межсетевого экрана или если клиенты не могут подключиться к вашему сервису, узнайте у
> системного администратора, разрешён ли UDP.

## Что такое датаграмма?

Клиенты и серверы, которые взаимодействуют через надёжный канал, такой как TCP-сокет, имеют
выделенный канал «точка-точка» между собой — или по крайней мере иллюзию такого канала. Чтобы
обмениваться данными, они устанавливают соединение, передают данные, а затем закрывают
соединение. Все данные, отправленные по каналу, принимаются в том же порядке, в котором они были
отправлены. Это гарантируется самим каналом.

В отличие от этого, приложения, которые взаимодействуют через датаграммы, отправляют и принимают
полностью независимые пакеты информации. Таким клиентам и серверам не нужен — и у них нет —
выделенного канала «точка-точка». Доставка датаграмм по назначению не гарантируется. Не
гарантируется и порядок их прибытия.

> **Определение.** *Датаграмма* (*datagram*) — это независимое, самодостаточное сообщение,
> отправленное по сети, доставка, время доставки и содержимое которого не гарантируются.

Пакет `java.net` содержит три класса, которые помогают писать программы на Java, использующие
датаграммы для отправки и приёма пакетов по сети: `DatagramSocket`, `DatagramPacket` и
`MulticastSocket`. Приложение может отправлять и принимать объекты `DatagramPacket` через
`DatagramSocket`. Кроме того, объекты `DatagramPacket` можно широковещательно рассылать
(*broadcast*) множеству получателей, которые все слушают один `MulticastSocket`.

## Написание клиента и сервера на датаграммах

Пример, представленный в этом разделе, состоит из двух приложений: клиента и сервера. Сервер
непрерывно принимает пакеты-датаграммы через датаграммный сокет. Каждый принятый сервером
пакет-датаграмма означает запрос клиента на цитату. Когда сервер получает датаграмму, он
отвечает, отправляя клиенту пакет-датаграмму, содержащую однострочную «цитату момента»
(*quote of the moment*).

Клиентское приложение в этом примере довольно простое. Оно отправляет серверу один
пакет-датаграмму, сообщая, что клиент хотел бы получить «цитату момента». Затем клиент ждёт,
пока сервер пришлёт пакет-датаграмму в ответ.

Серверное приложение реализуют два класса: `QuoteServer` и `QuoteServerThread`. Клиентское
приложение реализует один класс: `QuoteClient`.

Рассмотрим эти классы, начиная с класса, который содержит метод `main` серверного приложения.

### Класс QuoteServer

Класс `QuoteServer`, показанный здесь полностью, содержит единственный метод — метод `main`
приложения-сервера цитат. Метод `main` просто создаёт новый объект `QuoteServerThread` и
запускает его:

```java
import java.io.*;

public class QuoteServer {
    public static void main(String[] args) throws IOException {
        new QuoteServerThread().start();
    }
}
```

Класс `QuoteServerThread` реализует основную логику сервера цитат.

### Класс QuoteServerThread

При создании `QuoteServerThread` создаёт `DatagramSocket` на порту 4445 (выбран произвольно).
Именно через этот `DatagramSocket` сервер взаимодействует со всеми своими клиентами.

```java
public QuoteServerThread() throws IOException {
    this("QuoteServer");
}

public QuoteServerThread(String name) throws IOException {
    super(name);
    socket = new DatagramSocket(4445);

    try {
        in = new BufferedReader(new FileReader("one-liners.txt"));
    }   
    catch (FileNotFoundException e){
        System.err.println("Couldn't open quote file.  Serving time instead.");
        // Не удалось открыть файл с цитатами. Вместо этого отдаём время.
    }
}  
```

Помните, что некоторые порты выделены под общеизвестные сервисы (*well-known services*), и
использовать их нельзя. Если вы укажете порт, который уже занят, создание `DatagramSocket`
завершится неудачей.

Конструктор также открывает `BufferedReader` для файла с именем `one-liners.txt`, который
содержит список цитат. Каждая цитата в файле находится на отдельной строке.

Теперь самое интересное в `QuoteServerThread` — его метод `run`. Метод `run` переопределяет
метод `run` класса `Thread` и предоставляет реализацию для потока. Информацию о потоках
(*threads*) см. в разделе «Defining and Starting a Thread».

Метод `run` содержит цикл `while`, который продолжается, пока в файле есть ещё цитаты. На каждой
итерации цикла поток ждёт прибытия объекта `DatagramPacket` через `DatagramSocket`. Этот пакет
означает запрос от клиента. В ответ на запрос клиента `QuoteServerThread` получает цитату из
файла, помещает её в `DatagramPacket` и отправляет через `DatagramSocket` тому клиенту, который
её запросил.

Сначала посмотрим на участок, который принимает запросы от клиентов:

```java
byte[] buf = new byte[256];
DatagramPacket packet = new DatagramPacket(buf, buf.length);
socket.receive(packet);
```

Первая инструкция создаёт массив байтов, который затем используется для создания объекта
`DatagramPacket`. Этот `DatagramPacket` будет использоваться для приёма датаграммы из сокета —
из-за выбранного конструктора. Данный конструктор требует только два аргумента: массив байтов,
содержащий специфичные для клиента данные, и длину этого массива. При создании `DatagramPacket`
для отправки через `DatagramSocket` вы также должны указать интернет-адрес и номер порта
назначения пакета. Вы увидите это позже, когда мы обсудим, как сервер отвечает на запрос клиента.

Последняя инструкция в приведённом фрагменте кода принимает датаграмму из сокета (информация,
полученная от клиента, копируется в пакет). Метод `receive` ждёт бесконечно, пока не будет
получен пакет. Если ни один пакет не получен, сервер не продвигается дальше и просто ждёт.

Теперь предположим, что сервер получил от клиента запрос на цитату. Теперь сервер должен
ответить. Этот участок кода в методе `run` формирует ответ:

```java
String dString = null;
if (in == null)
    dString = new Date().toString();
else
    dString = getNextQuote();
buf = dString.getBytes();
```

Если файл с цитатами по какой-то причине не был открыт, то `in` равно `null`. В этом случае
сервер цитат вместо цитаты отдаёт текущее время суток. Иначе сервер цитат получает следующую
цитату из уже открытого файла. Наконец, код преобразует строку в массив байтов.

Теперь метод `run` отправляет ответ клиенту через `DatagramSocket` следующим кодом:

```java
InetAddress address = packet.getAddress();
int port = packet.getPort();
packet = new DatagramPacket(buf, buf.length, address, port);
socket.send(packet);
```

Первые две инструкции этого фрагмента получают соответственно интернет-адрес и номер порта из
пакета-датаграммы, принятого от клиента. Интернет-адрес и номер порта указывают, откуда пришёл
пакет-датаграмма. Именно туда сервер и должен отправить свой ответ. В этом примере массив байтов
пакета-датаграммы не содержит никакой значимой информации. Само прибытие пакета означает запрос
от клиента, которого можно найти по интернет-адресу и номеру порта, указанным в пакете-датаграмме.

Третья инструкция создаёт новый объект `DatagramPacket`, предназначенный для отправки
датаграммного сообщения через датаграммный сокет. О том, что новый `DatagramPacket` предназначен
для отправки данных через сокет, можно судить по использованному конструктору. Этот конструктор
требует четыре аргумента. Первые два аргумента — те же, что требует конструктор для создания
принимающих датаграмм: массив байтов, содержащий сообщение от отправителя к получателю, и длину
этого массива. Следующие два аргумента — другие: интернет-адрес и номер порта. Эти два аргумента
составляют полный адрес назначения пакета-датаграммы и должны быть указаны отправителем
датаграммы. Последняя строка кода отправляет `DatagramPacket` в путь.

Когда сервер прочитает все цитаты из файла цитат, цикл `while` завершается, и метод `run`
выполняет очистку:

```java
socket.close();
```

### Класс QuoteClient

Класс `QuoteClient` реализует клиентское приложение для `QuoteServer`. Это приложение отправляет
запрос серверу `QuoteServer`, ждёт ответа и, когда ответ получен, выводит его в стандартный
поток вывода. Рассмотрим код подробно.

Класс `QuoteClient` содержит один метод — метод `main` клиентского приложения. В начале метода
`main` объявляется несколько локальных переменных для его работы:

```java
int port;
InetAddress address;
DatagramSocket socket = null;
DatagramPacket packet;
byte[] sendBuf = new byte[256];
```

Сначала метод `main` обрабатывает аргументы командной строки, использованные при запуске
приложения `QuoteClient`:

```java
if (args.length != 1) {
    System.out.println("Usage: java QuoteClient <hostname>");
    return;
}
```

Приложению `QuoteClient` требуется один аргумент командной строки: имя машины, на которой
работает `QuoteServer`.

Далее метод `main` создаёт `DatagramSocket`:

```java
DatagramSocket socket = new DatagramSocket();
```

Клиент использует конструктор, который не требует номера порта. Этот конструктор просто
привязывает `DatagramSocket` к любому доступному локальному порту. Неважно, к какому порту
привязан клиент, потому что объекты `DatagramPacket` содержат адресную информацию. Сервер
получает номер порта из объектов `DatagramPacket` и отправляет свой ответ на этот порт.

Затем программа `QuoteClient` отправляет запрос серверу:

```java
byte[] buf = new byte[256];
InetAddress address = InetAddress.getByName(args[0]);
DatagramPacket packet = new DatagramPacket(buf, buf.length, 
                                address, 4445);
socket.send(packet);
```

Этот фрагмент кода получает интернет-адрес для хоста, имя которого указано в командной строке
(предположительно — имя машины, на которой работает сервер). Этот `InetAddress` и номер порта
4445 (номер порта, который сервер использовал для создания своего `DatagramSocket`) затем
используются для создания `DatagramPacket`, предназначенного для этого интернет-адреса и номера
порта. Таким образом, `DatagramPacket` будет доставлен серверу цитат.

Обратите внимание, что код создаёт `DatagramPacket` с пустым массивом байтов. Массив байтов
пуст, потому что этот пакет-датаграмма — просто запрос информации у сервера. Всё, что серверу
нужно знать для отправки ответа — адрес и номер порта, на которые отвечать, — автоматически
является частью пакета.

Далее клиент получает ответ от сервера и выводит его:

```java
packet = new DatagramPacket(buf, buf.length);
socket.receive(packet);
String received = new String(packet.getData(), 0, packet.getLength());
System.out.println("Quote of the Moment: " + received);
```

Чтобы получить ответ от сервера, клиент создаёт «принимающий» пакет и использует метод `receive`
класса `DatagramSocket`, чтобы получить ответ от сервера. Метод `receive` ждёт, пока через сокет
не придёт пакет-датаграмма, предназначенный клиенту. Обратите внимание: если ответ сервера
каким-то образом потеряется, клиент будет ждать вечно из-за политики «без гарантий» (*no-guarantee*)
датаграммной модели. Обычно клиент устанавливает таймер, чтобы не ждать ответа бесконечно; если
ответ не приходит, таймер срабатывает, и клиент повторно отправляет запрос.

Когда клиент получает ответ от сервера, он использует метод `getData`, чтобы извлечь эти данные
из пакета. Затем клиент преобразует данные в строку и выводит их.

### Запуск сервера и клиента

После того как вы успешно скомпилировали программы сервера и клиента, вы их запускаете. Сначала
нужно запустить программу-сервер. Просто используйте интерпретатор Java и укажите имя класса
`QuoteServer`.

Как только сервер запущен, можно запустить программу-клиент. Не забудьте запустить программу-клиент
с одним аргументом командной строки — именем хоста, на котором работает `QuoteServer`.

После того как клиент отправит запрос и получит ответ от сервера, вы должны увидеть вывод,
похожий на этот:

```
Quote of the Moment:
Good programming is 99% sweat and 1% coffee.
```

## Широковещательная рассылка нескольким получателям

Помимо `DatagramSocket`, который позволяет программам отправлять пакеты друг другу, `java.net`
включает класс `MulticastSocket`. Этот вид сокета используется на стороне клиента, чтобы слушать
пакеты, которые сервер широковещательно рассылает (*broadcast*) множеству клиентов.

Перепишем сервер цитат так, чтобы он широковещательно рассылал объекты `DatagramPacket`
нескольким получателям. Вместо отправки цитат конкретному клиенту, который сделал запрос, новый
сервер теперь должен рассылать цитаты через равные промежутки времени. Клиента нужно изменить
так, чтобы он пассивно слушал цитаты и делал это на `MulticastSocket`.

Этот пример состоит из трёх классов, которые являются модификациями трёх классов из предыдущего
примера: `MulticastServer`, `MulticastServerThread` и `MulticastClient`. В этом обсуждении
выделены наиболее интересные части этих классов.

Вот новая версия главной программы сервера. Отличия от предыдущей версии, `QuoteServer`, в
оригинале выделены жирным шрифтом:

```java
import java.io.*;

public class MulticastServer {
    public static void main(String[] args) throws IOException {
        new MulticastServerThread().start();
    }
}
```

По сути, сервер получил новое имя и создаёт `MulticastServerThread` вместо `QuoteServerThread`.
Теперь посмотрим на `MulticastServerThread`, который содержит ядро сервера. Вот его объявление
класса:

```java
public class MulticastServerThread extends QuoteServerThread {
    // ...
}
```

Мы сделали этот класс подклассом `QuoteServerThread`, чтобы он мог использовать конструктор и
унаследовать некоторые переменные-члены и метод `getNextQuote`. Напомним, что `QuoteServerThread`
создаёт `DatagramSocket`, привязанный к порту 4445, и открывает файл цитат. Номер порта
`DatagramSocket` в этом примере фактически не имеет значения, потому что клиент никогда ничего не
отправляет серверу.

Единственный метод, явно реализованный в `MulticastServerThread`, — это его метод `run`. Отличия
между этим методом `run` и методом из `QuoteServerThread` в оригинале выделены жирным шрифтом:

```java
public void run() {
    while (moreQuotes) {
        try {
            byte[] buf = new byte[256];
            // не ждём запроса... просто отправляем цитату

            String dString = null;
            if (in == null)
                dString = new Date().toString();
            else
                dString = getNextQuote();
            buf = dString.getBytes();

            InetAddress group = InetAddress.getByName("203.0.113.0");
            DatagramPacket packet;
            packet = new DatagramPacket(buf, buf.length, group, 4446);
            socket.send(packet);

            try {
                sleep((long)Math.random() * FIVE_SECONDS);
            } 
            catch (InterruptedException e) { }
        }
        catch (IOException e) {
            e.printStackTrace();
            moreQuotes = false;
        }
    }
    socket.close();
}
```

Интересное изменение — то, как конструируется `DatagramPacket`, в частности, какие `InetAddress`
и порт используются при создании `DatagramPacket`. Напомним, что в предыдущем примере `InetAddress`
и номер порта извлекались из пакета, отправленного серверу клиентом. Это было потому, что сервер
должен был отвечать непосредственно клиенту. Теперь же серверу нужно адресоваться к множеству
клиентов. Поэтому на этот раз и `InetAddress`, и номер порта заданы жёстко в коде (*hard-coded*).

Жёстко заданный номер порта — 4446 (у клиента должен быть `MulticastSocket`, привязанный к этому
порту). Жёстко заданный `InetAddress` пакета `DatagramPacket` — «203.0.113.0»; это идентификатор
группы (*group identifier*), а не интернет-адрес машины, на которой работает один конкретный
клиент. Этот конкретный адрес был произвольно выбран из диапазона, зарезервированного для данной
цели.

Созданный таким образом `DatagramPacket` предназначается всем клиентам, слушающим порт с номером
4446, которые являются членами группы «203.0.113.0».

Чтобы слушать порт с номером 4446, новая клиентская программа просто создаёт свой
`MulticastSocket` с этим номером порта. Чтобы стать членом группы «203.0.113.0», клиент вызывает
метод `joinGroup` объекта `MulticastSocket`, передавая `InetAddress`, который идентифицирует
группу. Теперь клиент настроен на приём объектов `DatagramPacket`, предназначенных для указанных
порта и группы. Вот соответствующий код из новой клиентской программы (которая также была
переписана так, чтобы пассивно принимать цитаты, а не активно их запрашивать). Инструкции,
взаимодействующие с `MulticastSocket`, в оригинале выделены жирным шрифтом:

```java
MulticastSocket socket = new MulticastSocket(4446);
InetAddress group = InetAddress.getByName("203.0.113.0");
socket.joinGroup(group);

DatagramPacket packet;
for (int i = 0; i < 5; i++) {
    byte[] buf = new byte[256];
    packet = new DatagramPacket(buf, buf.length);
    socket.receive(packet);

    String received = new String(packet.getData());
    System.out.println("Quote of the Moment: " + received);
}

socket.leaveGroup(group);
socket.close();
```

Обратите внимание, что сервер использует `DatagramSocket`, чтобы широковещательно рассылать
пакет, принимаемый клиентом через `MulticastSocket`. В качестве альтернативы он мог бы
использовать и `MulticastSocket`. Сокет, который сервер использует для отправки `DatagramPacket`,
не важен. Что действительно важно при широковещательной рассылке пакетов — это адресная
информация, содержащаяся в `DatagramPacket`, и сокет, который клиент использует для прослушивания.

> **Попробуйте.** Запустите `MulticastServer` и несколько клиентов. Понаблюдайте, как все клиенты
> получают одни и те же цитаты.
