# Урок 2. Обеспечение безопасности расширений

**Трейл:** The Extension Mechanism · **Оригинал:** [Making Extensions Secure](https://docs.oracle.com/javase/tutorial/ext/security/index.html)
**Связанные области:** [[18-build-tools]] · [[23-security]] · **Вопросы:** build-tools

> Перевод официального руководства Oracle (The Java Tutorials, JDK 8).
>
> Теперь, когда вы увидели, как использовать расширения (extensions), вам, возможно, интересно,
> какими привилегиями безопасности обладают расширения. Например, если вы разрабатываете
> расширение, выполняющее ввод-вывод в файлы (file I/O), вам нужно знать, как ваше расширение
> получает соответствующие разрешения на чтение и запись файлов. И наоборот, если вы собираетесь
> использовать расширение, разработанное кем-то другим, вы захотите чётко понимать, какими
> привилегиями безопасности это расширение обладает и как изменить эти привилегии, если вы того
> пожелаете.
>
> Этот урок показывает, как архитектура безопасности платформы Java™ обращается с расширениями.
> Вы увидите, как определить, какие привилегии предоставлены программному расширению, и научитесь
> изменять привилегии расширений, выполнив несколько простых шагов. Кроме того, вы научитесь
> запечатывать пакеты (sealing packages) внутри своих расширений, чтобы ограничить доступ к
> определённым частям вашего кода.

> **Примечание об актуальности.** Механизм расширений Java (Extension Mechanism), описанный в этом
> уроке, был объявлен устаревшим (deprecated) в JDK 8 (см. JEP 220 и заметки о выпуске) и
> **полностью удалён начиная с JDK 9** в рамках перехода на модульную систему (Java Platform Module
> System, проект Jigsaw). Каталоги установленных расширений (`lib/ext`), системное свойство
> `java.ext.dirs` и связанные с ними записи политики безопасности (`grant codeBase
> "file:${{java.ext.dirs}}/*"`) в современных версиях Java не поддерживаются. Материал ниже
> сохраняет историческую точность оригинала Oracle для JDK 8; на актуальных JDK вместо механизма
> расширений используйте модули и обычный путь к классам (classpath).

## Установка привилегий для расширений (Setting Privileges for Extensions)

Если в системе действует [менеджер безопасности](../../essential/environment/security.html)
(security manager), то для того, чтобы любое программное обеспечение, включая программные
расширения, могло выполнять операции, чувствительные с точки зрения безопасности
(security-sensitive operations), должны быть выполнены следующие условия:

- Код расширения, чувствительный к безопасности, должен быть обёрнут в объект `PrivilegedAction`.
- Политика безопасности (security policy), реализуемая менеджером безопасности, должна
  предоставлять расширению соответствующее разрешение (permission). По умолчанию установленным
  расширениям предоставляются все разрешения безопасности, как если бы они были частью базового
  API платформы. Разрешения, предоставляемые политикой безопасности, применяются только к коду,
  обёрнутому в экземпляр `PrivilegedAction`.

Рассмотрим каждое из этих условий немного подробнее на примерах.

### Использование класса `PrivilegedAction`

Предположим, вы хотите изменить класс `RectangleArea` из примера расширения предыдущего урока так,
чтобы он записывал площади прямоугольников в файл, а не в стандартный вывод (stdout). Запись в файл,
однако, является операцией, чувствительной к безопасности, поэтому, если ваша программа будет
работать под менеджером безопасности, вам потребуется пометить ваш код как привилегированный
(privileged). Чтобы сделать это, нужно выполнить два шага:

1. Нужно поместить код, выполняющий чувствительные к безопасности операции, внутрь метода `run`
   объекта типа `java.security.PrivilegedAction`.
2. Нужно использовать этот объект `PrivilegedAction` в качестве аргумента при вызове метода
   `doPrivileged` класса `java.security.AccessController`.

Если применить эти рекомендации к классу `RectangleArea`, его определение будет выглядеть примерно так:

```java
import java.io.*;
import java.security.*;

public final class RectangleArea {
    public static void
    writeArea(final java.awt.Rectangle r) {
        AccessController.
          doPrivileged(new PrivilegedAction() {
            public Object run() {
                try {
                    int area = r.width * r.height;
                    String userHome = System.getProperty("user.home");
                    FileWriter fw = new FileWriter( userHome + File.separator
                        + "test" + File.separator + "area.txt");
                    fw.write("The rectangle's area is " + area);
                    fw.flush();
                    fw.close();
                } catch(IOException ioe) {
                    System.err.println(ioe);
                }
                return null;
            }
        });
    }
}
```

Единственный метод этого класса, `writeArea`, вычисляет площадь прямоугольника и записывает её
в файл с именем `area.txt` в каталоге `test`, расположенном в домашнем каталоге пользователя.

Чувствительные к безопасности инструкции, работающие с выходным файлом, помещены внутрь метода
`run` нового экземпляра `PrivilegedAction`. (Обратите внимание, что `run` требует, чтобы был
возвращён экземпляр `Object`. Возвращаемый объект может быть `null`.) Затем новый экземпляр
`PrivilegedAction` передаётся в качестве аргумента при вызове `AccessController.doPrivileged`.

Дополнительную информацию об использовании `doPrivileged` см. в разделе
[API for Privileged Blocks](https://docs.oracle.com/javase/8/docs/technotes/guides/security/doprivileged.html)
документации JDK™.

Обёртывание чувствительного к безопасности кода в объект `PrivilegedAction` таким образом — это
первое требование для того, чтобы расширение могло выполнять чувствительные к безопасности операции.
Второе требование — добиться, чтобы менеджер безопасности предоставил привилегированному коду
соответствующие разрешения.

### Задание разрешений через политику безопасности

Политика безопасности, действующая во время выполнения, задаётся *файлом политики* (policy file).
Политика по умолчанию устанавливается файлом `lib/security/java.policy` в составе JRE.

Файл политики назначает привилегии безопасности программам с помощью записей *grant* (grant
entries). Файл политики может содержать любое количество записей grant. Файл политики по умолчанию
содержит следующую запись grant для установленных расширений:

```
grant codeBase "file:${{java.ext.dirs}}/*" {
    permission java.security.AllPermission;
};
```

Эта запись указывает, что файлам в каталогах, заданных шаблоном `file:${{java.ext.dirs}}/*`,
предоставляется разрешение под названием `java.security.AllPermission`. (Обратите внимание, что
начиная с Java 6, `java.ext.dirs` ссылается на путь из каталогов, подобный пути классов (classpath),
каждый из которых может содержать установленные расширения.) Нетрудно догадаться, что
`java.security.AllPermission` предоставляет установленным расширениям все привилегии безопасности,
какие только возможно предоставить.

Таким образом, по умолчанию установленные расширения не имеют ограничений безопасности.
Программное расширение может выполнять чувствительные к безопасности операции так, как если бы
менеджер безопасности не был установлен, при условии что чувствительный к безопасности код
содержится в экземпляре `PrivilegedAction`, переданном в качестве аргумента при вызове
`doPrivileged`.

Чтобы ограничить привилегии, предоставляемые расширениям, нужно изменить файл политики. Чтобы
лишить все расширения всех привилегий, можно просто удалить приведённую выше запись grant.

Не все разрешения столь же всеобъемлющи, как предоставляемое по умолчанию
`java.security.AllPermission`. После удаления записи grant по умолчанию вы можете добавить новую
запись grant для конкретных разрешений, в том числе:

- `java.awt.AWTPermission`
- `java.io.FilePermission`
- `java.net.NetPermission`
- `java.util.PropertyPermission`
- `java.lang.reflect.ReflectPermission`
- `java.lang.RuntimePermission`
- `java.security.SecurityPermission`
- `java.io.SerializablePermission`
- `java.net.SocketPermission`

Документация [Permissions in the JDK](https://docs.oracle.com/javase/8/docs/technotes/guides/security/permissions.html)
содержит подробные сведения о каждом из этих разрешений. Рассмотрим разрешения, необходимые для
использования `RectangleArea` в качестве расширения.

Методу `RectangleArea.writeArea` требуются два разрешения: одно — чтобы определить путь к домашнему
каталогу пользователя, другое — чтобы записать в файл. Предполагая, что класс `RectangleArea`
упакован в файл `area.jar`, вы можете предоставить привилегии на запись, добавив в файл политики
такую запись:

```
grant codeBase "file:${java.home}/lib/ext/area.jar" {
    permission java.io.PropertyPermission "user.home",
        "read";
    permission java.io.FilePermission
        "${user.home}${/}test${/}*", "write";
};
```

Часть `codeBase "file:${java.home}/lib/ext/area.jar"` этой записи гарантирует, что любые разрешения,
заданные данной записью, будут применяться только к `area.jar`. Разрешение
`java.io.PropertyPermission` позволяет доступ к свойствам (properties). Первый аргумент,
`"user.home"`, называет свойство, а второй аргумент, `"read"`, указывает, что свойство можно читать.
(Другой вариант — `"write"`.)

Разрешение `java.io.FilePermission` позволяет доступ к файлам. Первый аргумент,
`"${user.home}${/}test${/}*"`, указывает, что `area.jar` предоставляется разрешение на доступ ко
всем файлам в каталоге `test`, находящемся в домашнем каталоге пользователя. (Обратите внимание,
что `${/}` — это платформо-независимый разделитель файлов (file separator).) Второй аргумент
указывает, что предоставляется доступ к файлу только на запись. (Другие варианты второго аргумента —
`"read"`, `"delete"` и `"execute"`.)

### Подписывание расширений (Signing Extensions)

Вы можете использовать файл политики, чтобы наложить на разрешения, предоставляемые расширениям,
дополнительные ограничения, потребовав, чтобы расширения были подписаны доверенной стороной (trusted
entity). (Для обзора подписывания и проверки JAR-файлов см. урок
[Signing JAR Files](../../deployment/jar/signing.html) в этом руководстве.)

Чтобы разрешить проверку подписи (signature verification) расширений или другого программного
обеспечения в связке с предоставлением разрешений, файл политики должен содержать *запись хранилища
ключей* (keystore entry). Запись keystore указывает, какое хранилище ключей (keystore) должно
использоваться при проверке. Записи keystore имеют вид:

```
keystore "keystore_url";
```

URL *keystore_url* — это либо абсолютный, либо относительный адрес. Если он относительный, URL
указывается относительно расположения файла политики. Например, чтобы использовать хранилище ключей
по умолчанию, применяемое утилитой keytool, добавьте в `java.policy` такую запись:

```
keystore "file://${user.home}/.keystore";
```

Чтобы указать, что расширение должно быть подписано для получения привилегий безопасности,
используется поле `signedBy`. Например, следующая запись указывает, что расширению `area.jar`
перечисленные привилегии предоставляются только в том случае, если оно подписано пользователями,
идентифицированными в хранилище ключей псевдонимами (aliases) Robert и Rita:

```
grant signedBy "Robert,Rita",
    codeBase "file:${java.home}/lib/ext/area.jar" {
        permission java.io.PropertyPermission
            "user.home", "read";
        permission java.io.FilePermission
            "${user.home}${/}test${/}*", "write";
};
```

Если поле `codeBase` опущено, как в следующем `grant`, разрешения предоставляются *любому*
программному обеспечению, включая установленные или загружаемые расширения, подписанному «Robert»
или «Rita»:

```
grant signedBy "Robert,Rita" {
    permission java.io.FilePermission "*", "write";
};
```

Дополнительные сведения о формате файла политики см. в разделе 3.3.1 документа
[Security Architecture Specification](https://docs.oracle.com/javase/8/docs/technotes/guides/security/spec/security-spec.doc3.html#20131)
в документации JDK.

## Запечатывание пакетов в расширениях (Sealing Packages in Extensions)

Вы можете при желании *запечатать* (seal) пакеты в JAR-файлах расширений в качестве дополнительной
меры безопасности. Если пакет запечатан, все классы, определённые в этом пакете, должны происходить
из одного и того же JAR-файла.

Без запечатывания «враждебная» (hostile) программа могла бы создать класс и объявить его членом
одного из пакетов вашего расширения. Тогда враждебное программное обеспечение получило бы свободный
доступ к защищённым на уровне пакета (package-protected) членам вашего пакета расширения.

Запечатывание пакетов в расширениях ничем не отличается от запечатывания любых классов, упакованных
в JAR. Чтобы запечатать пакеты вашего расширения, нужно добавить заголовок `Sealed` в манифест
(manifest) JAR-файла, содержащего ваше расширение. Вы можете запечатывать отдельные пакеты, связывая
заголовок `Sealed` с заголовками `Name` соответствующих пакетов. Заголовок `Sealed`, не связанный с
отдельным пакетом архива, сигнализирует, что запечатаны все пакеты. Такой «глобальный» заголовок
`Sealed` переопределяется любыми заголовками `Sealed`, связанными с отдельными пакетами. Значение,
связанное с заголовком `Sealed`, — это либо `true`, либо `false`.

### Примеры

Рассмотрим несколько примеров файлов манифеста. Для этих примеров предположим, что JAR-файл содержит
такие пакеты:

```
com/myCompany/package_1/
com/myCompany/package_2/
com/myCompany/package_3/
com/myCompany/package_4/
```

Предположим, что вы хотите запечатать все пакеты. Это можно сделать, просто добавив в манифест
заголовок `Sealed` уровня архива:

```
Manifest-Version: 1.0
Sealed: true
```

Все пакеты в любом JAR-файле, имеющем такой манифест, будут запечатаны.

Если бы вы захотели запечатать только `com.myCompany.package_3`, вы могли бы сделать это с таким
манифестом:

```
Manifest-Version: 1.0

Name: com/myCompany/package_3/
Sealed: true
```

В этом примере единственный заголовок `Sealed` — это заголовок, связанный с заголовком `Name`
пакета `com.myCompany.package_3`, поэтому запечатан только этот пакет. (Заголовок `Sealed` связан
с заголовком `Name`, потому что между ними нет пустых строк.)

В качестве последнего примера предположим, что вы захотели запечатать все пакеты, *кроме*
`com.myCompany.package_2`. Этого можно добиться таким манифестом:

```
Manifest-Version: 1.0
Sealed: true

Name: com/myCompany/package_2/
Sealed: false
```

В этом примере заголовок уровня архива `Sealed: true` указывает, что все пакеты в JAR-файле должны
быть запечатаны. Однако в манифесте также есть заголовок `Sealed: false`, связанный с пакетом
`com.myCompany.package_2`, и этот заголовок переопределяет запечатывание уровня архива для данного
пакета. Поэтому такой манифест приведёт к тому, что будут запечатаны все пакеты, кроме
`com.myCompany.package_2`.
