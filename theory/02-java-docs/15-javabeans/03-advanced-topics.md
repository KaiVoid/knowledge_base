# Урок 3. Продвинутые темы JavaBeans

**Трейл:** JavaBeans · **Оригинал:** [Advanced JavaBeans Topics](https://docs.oracle.com/javase/tutorial/javabeans/advanced/index.html)
**Связанные области:** [[01-core-java-syntax-oop]] · **Вопросы:** core-java

> Перевод официального руководства Oracle (The Java Tutorials, JDK 8). Урок объединяет
> страницы *Advanced JavaBeans Topics*, *Bean Persistence*, *Long Term Persistence* и
> *Bean Customization* (включая разделы о редакторах свойств и кастомайзерах).

> Начать разработку компонентов JavaBeans легко, но у бинов (*beans*) есть неожиданная
> глубина. Этот урок охватывает несколько более продвинутых тем, в том числе то, как бины
> могут сохраняться (*persisted*) и как можно предоставить собственные редакторы для
> пользовательских типов данных.

Урок состоит из трёх частей:

- **Сохраняемость бина** (*Bean Persistence*) — механизмы сохранения и восстановления бина.
- **Долговременная сохраняемость** (*Long Term Persistence*) — представление бина в виде XML.
- **Настройка бина** (*Bean Customization*) — обзор создания редактирующих компонентов для
  пользовательских типов данных.

## Сохраняемость бина (Bean Persistence)

> Бин обладает свойством сохраняемости (*persistence*), когда его свойства (*properties*),
> поля (*fields*) и информация о состоянии (*state information*) сохраняются в хранилище и
> извлекаются из него. Компонентные модели предоставляют механизм сохраняемости, позволяющий
> хранить состояние компонентов в энергонезависимом месте для последующего извлечения.

> Механизм, делающий сохраняемость возможной, называется **сериализацией** (*serialization*).
> Сериализация объекта означает преобразование объекта в поток данных (*data stream*) и запись
> его в хранилище. Любой апплет, приложение или инструмент, использующий этот бин, может затем
> «воссоздать» (*reconstitute*) его путём **десериализации** (*deserialization*). После этого
> объект восстанавливается в своё исходное состояние.

> Например, Java-приложение может сериализовать окно `Frame` на машине с Microsoft Windows,
> сериализованный файл можно отправить по электронной почте на машину с Solaris, а затем
> Java-приложение может восстановить окно `Frame` ровно в том состоянии, в котором оно
> существовало на машине с Microsoft Windows.

> Любой апплет, приложение или инструмент, использующий этот бин, может затем «воссоздать»
> (*reconstitute*) его путём **десериализации** (*deserialization*).

> Все бины должны быть сохраняемыми. Чтобы обладать сохраняемостью, ваши бины должны
> поддерживать сериализацию, реализуя либо интерфейс
> [`java.io.Serializable`](https://docs.oracle.com/javase/8/docs/api/java/io/Serializable.html),
> либо интерфейс
> [`java.io.Externalizable`](https://docs.oracle.com/javase/8/docs/api/java/io/Externalizable.html).
> Эти интерфейсы предоставляют выбор между автоматической сериализацией и настраиваемой
> сериализацией. Если какой-либо класс в иерархии наследования класса реализует `Serializable`
> или `Externalizable`, то этот класс является сериализуемым.

### Классы, которые сериализуемы

> Любой класс является сериализуемым, если он сам или его класс-предок реализует интерфейс
> `java.io.Serializable`. Примеры сериализуемых классов: `Component`, `String`, `Date`,
> `Vector` и `Hashtable`. Таким образом, любой подкласс класса `Component`, включая `Applet`,
> может быть сериализован. Среди классов, не поддерживающих сериализацию, можно отметить
> `Image`, `Thread`, `Socket` и `InputStream`. Попытка сериализовать объекты этих типов
> приведёт к исключению `NotSerializableException`.

> API сериализации объектов Java (*Java Object Serialization API*) автоматически сериализует
> большинство полей сериализуемого объекта в поток хранилища. Сюда входят примитивные типы,
> массивы и строки. API не сериализует и не десериализует поля, помеченные `transient`
> (временные) или `static` (статические).

### Управление сериализацией

> Вы можете управлять тем, в какой степени сериализуются ваши бины. Существуют три способа
> управления сериализацией:

> - Автоматическая сериализация, реализуемая через интерфейс `Serializable`. Программное
>   обеспечение сериализации Java сериализует весь объект целиком, за исключением полей
>   `transient` и `static`.
> - Настраиваемая сериализация. Избирательно исключайте поля, которые вы не хотите
>   сериализовать, помечая их модификатором `transient` (или `static`).
> - Настраиваемый формат файла, реализуемый через интерфейс `Externalizable` и два его метода.
>   Бины записываются в определённом формате файла.

### Сериализация по умолчанию: интерфейс Serializable

> Интерфейс `Serializable` обеспечивает автоматическую сериализацию с помощью инструментов
> сериализации объектов Java. `Serializable` не объявляет ни одного метода; он выступает как
> маркер, сообщающий инструментам сериализации объектов, что ваш класс-бин является
> сериализуемым. Пометив свой класс как `Serializable`, вы тем самым сообщаете виртуальной
> машине Java (JVM), что вы убедились, что ваш класс будет работать с сериализацией по
> умолчанию. Вот несколько важных моментов при работе с интерфейсом `Serializable`:

> - Классы, реализующие `Serializable`, должны иметь доступ к **конструктору без аргументов**
>   (*no-argument constructor*) супертипа. Этот конструктор будет вызван, когда объект
>   «воссоздаётся» из файла `.ser`.
> - Вам не нужно реализовывать `Serializable` в своём классе, если он уже реализован в
>   суперклассе.
> - Сериализуются все поля, кроме статических и временных (`static` и `transient`). Используйте
>   модификатор `transient`, чтобы указать поля, которые вы не хотите сериализовать, а также
>   чтобы указать классы, которые не являются сериализуемыми.

### Избирательная сериализация с помощью ключевого слова transient

> Чтобы исключить поля из сериализации в объекте `Serializable`, пометьте эти поля модификатором
> `transient`.

```java
transient int status;
```

> Сериализация по умолчанию не будет сериализовать поля `transient` и `static`.

### Избирательная сериализация: writeObject и readObject

> Если ваш сериализуемый класс содержит любой из двух следующих методов (сигнатуры должны быть
> точными), то сериализация по умолчанию выполняться не будет.

```java
private void writeObject(java.io.ObjectOutputStream out)
    throws IOException;
private void readObject(java.io.ObjectInputStream in)
    throws IOException, ClassNotFoundException;
```

> Вы можете управлять тем, как сериализуются более сложные объекты, написав собственные
> реализации методов `writeObject` и `readObject`. Реализуйте `writeObject`, когда вам нужно
> осуществлять более тщательный контроль над тем, что сериализуется, когда вам нужно
> сериализовать объекты, с которыми не справляется сериализация по умолчанию, или когда вам
> нужно добавить в поток сериализации данные, не являющиеся данными-членами объекта. Реализуйте
> `readObject`, чтобы восстановить поток данных, который вы записали с помощью `writeObject`.

### Интерфейс Externalizable

> Используйте интерфейс `Externalizable`, когда вам нужен полный контроль над сериализацией
> вашего бина (например, при записи и чтении определённого формата файла). Чтобы использовать
> интерфейс `Externalizable`, вам нужно реализовать два метода: `readExternal` и `writeExternal`.
> Классы, реализующие `Externalizable`, должны иметь конструктор без аргументов.

## Долговременная сохраняемость (Long Term Persistence)

> **Долговременная сохраняемость** (*long-term persistence*) — это модель, позволяющая
> сохранять бины в формате XML.

> Сведения о формате XML и о том, как реализовать долговременную сохраняемость для не-бинов,
> можно найти в материалах
> [XML Schema](http://www.oracle.com/technetwork/java/persistence3-139471.html) и
> [Using XMLEncoder](http://www.oracle.com/technetwork/java/persistence4-140124.html).

### Кодировщик и декодировщик (Encoder и Decoder)

> Класс [`XMLEncoder`](https://docs.oracle.com/javase/8/docs/api/java/beans/XMLEncoder.html)
> предназначен для записи выходных файлов с текстовым представлением объектов `Serializable`.
> Следующий фрагмент кода — пример записи Java-бина и его свойств в формате XML:

```java
XMLEncoder encoder = new XMLEncoder(
           new BufferedOutputStream(
           new FileOutputStream("Beanarchive.xml")));

encoder.writeObject(object);
encoder.close();
```

> Класс [`XMLDecoder`](https://docs.oracle.com/javase/8/docs/api/java/beans/XMLDecoder.html)
> читает XML-документ, созданный с помощью `XMLEncoder`:

```java
XMLDecoder decoder = new XMLDecoder(
    new BufferedInputStream(
    new FileInputStream("Beanarchive.xml")));

Object object = decoder.readObject();
decoder.close();
```

### Что находится в XML?

> XML-архив бина имеет собственный специфический синтаксис, который включает следующие теги для
> представления каждого элемента бина:

> - XML-преамбулу для описания версии XML и типа кодировки;
> - тег `<java>`, охватывающий все объектные элементы бина;
> - тег `<object>` для представления набора вызовов методов, необходимых для восстановления
>   объекта из его сериализованной формы:

```xml
<object class="javax.swing.JButton" method="new">
    <string>Ok</string>
</object>
```

> либо операторы:

```xml
<object class="javax.swing.JButton">
    <void method="setText">
        <string>Cancel</string>
    </void>
</object>
```

> - теги для определения соответствующих примитивных типов:
>   - `<boolean>`
>   - `<byte>`
>   - `<char>`
>   - `<short>`
>   - `<int>`
>   - `<long>`
>   - `<float>`
>   - `<double>`

```xml
<int>5555</int>
```

> - тег `<class>` для представления экземпляра `Class`:

```xml
<class>java.swing.JFrame</class>
```

> - тег `<array>` для определения массива:

```xml
<array class="java.lang.String" length="5">
</array>
```

> Следующий код представляет XML-архив, который будет сгенерирован для компонента `SimpleBean`:

```xml
<?xml version="1.0" encoding="UTF-8" ?>
<java>
  <object class="javax.swing.JFrame">
    <void method="add">
      <object class="java.awt.BorderLayout" field="CENTER"/>
      <object class="SimpleBean"/>
    </void>
    <void property="defaultCloseOperation">
      <object class="javax.swing.WindowConstants" field="DISPOSE_ON_CLOSE"/>
    </void>
    <void method="pack"/>
    <void property="visible">
      <boolean>true</boolean>
    </void>
  </object>
</java>
```

## Настройка бина (Bean Customization)

> **Настройка** (*customization*) предоставляет средство для изменения внешнего вида и
> поведения бина в инструменте сборки приложений (*application builder*) так, чтобы он
> удовлетворял вашим конкретным потребностям. Для разработчика бина доступно несколько уровней
> настройки, позволяющих другим разработчикам извлекать максимальную выгоду из потенциальных
> возможностей бина.

> Следующие ссылки полезны для изучения редакторов свойств (*property editors*) и кастомайзеров
> (*customizers*):

- интерфейс [`PropertyEditor`](https://docs.oracle.com/javase/8/docs/api/java/beans/PropertyEditor.html);
- класс [`PropertyEditorSupport`](https://docs.oracle.com/javase/8/docs/api/java/beans/PropertyEditorSupport.html);
- класс [`PropertyEditorManager`](https://docs.oracle.com/javase/8/docs/api/java/beans/PropertyEditorManager.html);
- интерфейс [`Customizer`](https://docs.oracle.com/javase/8/docs/api/java/beans/Customizer.html);
- интерфейс [`BeanInfo`](https://docs.oracle.com/javase/8/docs/api/java/beans/BeanInfo.html).

> Внешний вид и поведение бина можно настроить на этапе проектирования (*design time*) внутри
> инструментов сборки (*builder tools*), совместимых с JavaBeans. Существует два способа
> настройки бина:

> - С помощью **редактора свойств** (*property editor*). У каждого свойства бина есть
>   собственный редактор свойств. Графический построитель NetBeans (*NetBeans GUI Builder*)
>   обычно отображает редакторы свойств бина в окне свойств (*Properties window*). Редактор
>   свойств, связанный с конкретным типом свойства, редактирует свойства этого типа.
> - С помощью **кастомайзеров** (*customizers*). Кастомайзеры дают вам полный контроль над
>   настройкой бина через графический интерфейс. Кастомайзеры применяются там, где редакторы
>   свойств непрактичны или неприменимы. В отличие от редактора свойств, который связан со
>   свойством, кастомайзер связан с бином.

### Редакторы свойств (Property Editors)

> Редактор свойств — это инструмент для настройки конкретного типа свойства. Редакторы свойств
> активируются в окне свойств (*Properties window*). Это окно определяет тип свойства, ищет
> подходящий редактор свойств и отображает текущее значение свойства соответствующим образом.

> Редакторы свойств должны реализовывать интерфейс `PropertyEditor`, который предоставляет
> методы для указания того, как свойство должно отображаться в листе свойств (*property sheet*).
> На следующем рисунке показано окно свойств, содержащее свойства `myBean1`:

![Рисунок: свойства myBean1 в окне свойств (Properties window) — оригинал Oracle](../../figures/javabeans/customization.gif)

*Подпись (перевод оригинала): этот рисунок представляет свойства `myBean1` в окне свойств.*

> Процесс редактирования этих свойств начинается со щелчка по записи свойства. Щелчок по
> большинству этих записей вызывает отдельные панели. Например, чтобы задать `foreground`
> (передний план) или `background` (фон), используйте окна выбора с вариантами цветов или
> нажмите кнопку «...», чтобы работать со стандартным окном `ColorEditor`. Щелчок по свойству
> `toolTipText` открывает окно `StringEditor`.

> Вспомогательный класс `PropertyEditorSupport` предоставляет реализацию интерфейса
> `PropertyEditor` по умолчанию. Унаследовав свой редактор свойств от `PropertyEditorSupport`,
> вы можете просто переопределить только те методы, которые вам нужны.

> Чтобы отображать в окне свойств текущий «образец» значения свойства, вам нужно переопределить
> метод `isPaintable`, чтобы он возвращал `true`. Затем вы должны переопределить метод
> `paintValue`, чтобы нарисовать текущее значение свойства в прямоугольнике в листе свойств.
> Вот как класс `ColorEditor` реализует `paintValue`:

```java
public void paintValue(java.awt.Graphics gfx, java.awt.Rectangle box) {
    Color oldColor = gfx.getColor();
    gfx.setColor(Color.black);
    gfx.drawRect(box.x, box.y, box.width-3, box.height-3);
    gfx.setColor(color);
    gfx.fillRect(box.x+1, box.y+1, box.width-4, box.height-4);
    gfx.setColor(oldColor);
}
```

> Чтобы поддержать собственный (*custom*) редактор свойств, переопределите ещё два метода.
> Переопределите `supportsCustomEditor`, чтобы он возвращал `true`, а затем переопределите
> `getCustomEditor`, чтобы он возвращал экземпляр собственного редактора. `ColorEditor.getCustomEditor`
> возвращает `this`.

> Кроме того, класс `PropertyEditorSupport` ведёт список `PropertyChangeListener` и рассылает
> уведомления о событиях изменения свойства (*property change event*) этим слушателям при
> изменении связанного свойства (*bound property*).

### Как редакторы свойств связываются со свойствами

> Редакторы свойств обнаруживаются и связываются с заданным свойством следующими способами:

> - Явная привязка через объект `BeanInfo`. Редактор свойства `title` задаётся следующей
>   строкой кода:

```java
pd.setPropertyEditorClass(TitleEditor.class);
```

> - Явная регистрация через метод `java.beans.PropertyEditorManager.registerEditor`. Этот метод
>   принимает два аргумента: тип класса бина и класс редактора, который должен быть связан с
>   этим типом.
> - Поиск по имени. Если у класса нет явно связанного редактора свойств, то
>   `PropertyEditorManager` ищет редактор свойств для этого класса следующими способами:
>   - Добавляя «Editor» к полному (квалифицированному) имени класса. Например, для класса
>     `my.package.ComplexNumber` менеджер редакторов свойств будет искать класс
>     `my.package.ComplexNumberEditor`.
>   - Добавляя «Editor» к имени класса и выполняя поиск по пути классов (*class path*).

### Кастомайзеры (Customizers)

> Вы узнали, что инструменты сборки предоставляют поддержку для создания собственных редакторов
> свойств. Какие ещё потребности должны удовлетворять визуальные построители для сложных,
> промышленного уровня бинов? Часто нежелательно, чтобы все свойства бина были раскрыты на одном
> (иногда огромном) листе свойств. Что, если один-единственный корневой выбор о типе бина делает
> половину свойств неактуальными? Спецификация JavaBeans предусматривает определяемые
> пользователем кастомайзеры, с помощью которых вы можете определить более высокий уровень
> настройки свойств бина, чем доступно с редакторами свойств.

> Когда вы используете **кастомайзер** (*Customizer*) бина, вы получаете полный контроль над
> тем, как конфигурировать или редактировать бин. Кастомайзер — это приложение, специально
> нацеленное на настройку бина. Иногда свойств недостаточно для представления конфигурируемых
> атрибутов бина. Кастомайзеры применяются там, где для изменения бина потребовались бы
> изощрённые инструкции, и где редакторы свойств слишком примитивны, чтобы достичь настройки
> бина.

> Все кастомайзеры должны:

> - Наследоваться от `java.awt.Component` или одного из его подклассов.
> - Реализовывать интерфейс `java.beans.Customizer`. Это означает реализацию методов для
>   регистрации объектов `PropertyChangeListener` и рассылку событий изменения свойства этим
>   слушателям, когда происходит изменение целевого бина.
> - Реализовывать конструктор по умолчанию (*default constructor*).
> - Связывать кастомайзер с его целевым классом через `BeanInfo.getBeanDescriptor`.
