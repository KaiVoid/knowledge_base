# Урок 5. Streaming API for XML (StAX)

**Трейл:** JAXP · **Оригинал:** [Streaming API for XML](https://docs.oracle.com/javase/tutorial/jaxp/stax/index.html)
**Связанные области:** [[17-rest-web]] · **Вопросы:** rest-web

> Перевод официального руководства Oracle (The Java Tutorials, JDK 8). Объединяет страницы
> *Streaming API for XML* (index), *Why StAX?*, *StAX API*, *Using StAX*,
> *Oracle's Streaming XML Parser Implementation*, *Example Code* и *Further Information*.

Этот урок посвящён потоковому API для XML (Streaming API for XML, **StAX**) — потоковому,
событийно-ориентированному API на базе технологий Java с разбором по принципу «вытягивания»
(pull-parsing) для чтения и записи XML-документов. StAX позволяет создавать двунаправленные
XML-парсеры, которые быстры, относительно просты в программировании и потребляют мало памяти.

## Зачем нужен StAX?

Проект StAX возглавляла компания BEA при поддержке Sun Microsystems; в марте 2004 года
[спецификация JSR 173](http://jcp.org/en/jsr/detail?id=173) прошла финальное голосование по
утверждению в рамках Java Community Process. Главная цель API StAX — передать «управление
разбором программисту, предоставив простой API на основе итератора. Это позволяет программисту
запрашивать следующее событие (вытягивать событие) и хранить состояние процедурным образом».
StAX был создан, чтобы устранить ограничения двух самых распространённых API разбора — SAX и DOM.

### Потоковая обработка против DOM

Вообще говоря, для работы с информационными множествами (infoset) XML существуют две модели
программирования: **потоковая** (streaming) и **объектная модель документа** (Document Object
Model, DOM).

Модель DOM предполагает создание в памяти объектов, представляющих всё дерево документа и полное
состояние infoset XML-документа. После размещения в памяти деревья DOM можно свободно обходить и
разбирать произвольным образом, что обеспечивает разработчику максимальную гибкость. Однако
платой за эту гибкость становится потенциально большой объём занимаемой памяти и значительные
требования к процессору, поскольку всё представление документа должно удерживаться в памяти в
виде объектов на протяжении всей обработки документа. Для небольших документов это может не быть
проблемой, но требования к памяти и процессору могут стремительно расти с увеличением размера
документа.

Потоковая обработка относится к модели программирования, в которой infoset-множества XML
передаются и разбираются последовательно во время выполнения приложения, часто в реальном времени
и нередко из динамических источников, содержимое которых заранее точно неизвестно. Более того,
потоковые парсеры могут начать выдавать результат немедленно, а элементы infoset можно отбрасывать
и собирать сборщиком мусора сразу после их использования. Обеспечивая меньший объём занимаемой
памяти, сниженные требования к процессору и более высокую производительность в определённых
ситуациях, потоковая обработка имеет главный компромисс: вы можете видеть состояние infoset
только в одном месте документа за раз. Вы, по сути, ограничены видом документа «сквозь картонную
трубу» (cardboard tube) — а значит, вам нужно знать заранее, какую обработку вы хотите выполнить,
прежде чем читать XML-документ.

Потоковые модели обработки XML особенно полезны, когда у приложения жёсткие ограничения по памяти —
например, у сотового телефона, работающего на платформе Java Platform, Micro Edition (платформа
Java ME), — либо когда приложению нужно обрабатывать несколько запросов одновременно, как у сервера
приложений. Более того, можно утверждать, что большинство бизнес-логики работы с XML выигрывает от
потоковой обработки и не требует поддержания в памяти целых деревьев DOM.

### Pull-разбор против push-разбора

Потоковый **pull-разбор** (вытягивающий, pull parsing) относится к модели программирования, в
которой клиентское приложение вызывает методы библиотеки разбора XML, когда ему нужно
взаимодействовать с infoset XML, — то есть клиент получает (вытягивает, pull) XML-данные только
тогда, когда явно их запрашивает.

Потоковый **push-разбор** (проталкивающий, push parsing) относится к модели программирования, в
которой XML-парсер отправляет (проталкивает, push) XML-данные клиенту по мере того, как
встречает элементы в infoset XML, — то есть парсер отправляет данные независимо от того, готов
клиент использовать их в этот момент или нет.

Pull-разбор даёт несколько преимуществ перед push-разбором при работе с потоками XML:

* При pull-разборе клиент управляет потоком выполнения приложения и может вызывать методы парсера,
  когда это необходимо. При push-обработке, напротив, потоком выполнения управляет парсер, а
  клиент может лишь принимать вызовы от парсера.
* Библиотеки pull-разбора могут быть значительно меньше, а клиентский код для взаимодействия с
  ними — значительно проще, чем у push-библиотек, даже для более сложных документов.
* Pull-клиенты могут читать несколько документов одновременно в одном потоке выполнения.
* Pull-парсер StAX может фильтровать XML-документы так, что ненужные клиенту элементы могут
  игнорироваться, и он может поддерживать XML-представления данных, не являющихся XML.

### Сценарии использования StAX

Спецификация StAX определяет ряд сценариев использования API:

* Связывание данных (data binding):
  * демаршалинг (unmarshalling) XML-документа;
  * маршалинг (marshalling) XML-документа;
  * параллельная обработка документа;
  * беспроводная связь.
* Обработка SOAP-сообщений (Simple Object Access Protocol):
  * разбор простых предсказуемых структур;
  * разбор графовых представлений с прямыми ссылками (forward references);
  * разбор Web Services Description Language (WSDL).
* Виртуальные источники данных:
  * просмотр в виде XML данных, хранящихся в базах данных;
  * просмотр данных в Java-объектах, созданных связыванием данных XML;
  * навигация по дереву DOM как по потоку событий.
* Разбор конкретных XML-словарей.
* Конвейерная обработка XML (pipelined XML processing).

Полное обсуждение всех этих сценариев выходит за рамки данного урока. За дополнительной
информацией обращайтесь к спецификации StAX.

### Сравнение StAX с другими API из семейства JAXP

Будучи API из семейства JAXP, StAX можно сравнить, среди прочих, с SAX, TrAX и JDOM. По
сравнению с двумя последними StAX не столь мощен и гибок, как TrAX или JDOM, но и не требует
столько памяти и процессорной нагрузки, чтобы быть полезным; во многих случаях StAX может
превзойти по производительности API на основе DOM. Здесь применимы те же аргументы, изложенные
выше, которые взвешивают издержки и выгоды модели DOM против потоковой модели.

С учётом этого ближе всего можно сравнивать StAX и SAX, и именно тут StAX предлагает возможности,
полезные во многих случаях; вот некоторые из них:

* Клиенты на базе StAX обычно проще писать, чем клиенты SAX. Хотя можно утверждать, что парсеры
  SAX чуть проще написать, код парсера StAX может быть меньше, а код, необходимый клиенту для
  взаимодействия с парсером, — проще.
* StAX — двунаправленный (bidirectional) API, то есть он может как читать, так и писать
  XML-документы. SAX работает только на чтение, поэтому для записи XML-документов нужен ещё один API.
* SAX — это push-API, тогда как StAX — pull. Здесь применимы изложенные выше компромиссы между
  push- и pull-API.

В следующей таблице сводятся сравнительные характеристики StAX, SAX, DOM и TrAX. (Таблица
адаптирована из статьи [Does StAX Belong in Your XML Toolbox?](http://www.developer.com/xml/article.php/3397691)
Джеффа Райана.)

#### Сводка характеристик API XML-парсеров

| Характеристика | StAX | SAX | DOM | TrAX |
|---|---|---|---|---|
| Тип API | Pull, потоковый | Push, потоковый | Дерево в памяти | Правила XSLT |
| Простота использования | Высокая | Средняя | Высокая | Средняя |
| Поддержка XPath | Нет | Нет | Да | Да |
| Эффективность по CPU и памяти | Хорошая | Хорошая | Зависит | Зависит |
| Только вперёд (forward only) | Да | Да | Нет | Нет |
| Чтение XML | Да | Да | Да | Да |
| Запись XML | Да | Нет | Да | Да |
| Create, Read, Update, Delete | Нет | Нет | Да | Нет |

## API StAX

API StAX предоставляет методы для итеративной, событийно-ориентированной обработки XML-документов.
XML-документы рассматриваются как отфильтрованная последовательность событий, а состояния infoset
можно хранить процедурным образом. Кроме того, в отличие от SAX, API StAX двунаправленный, что
позволяет как читать, так и записывать XML-документы.

API StAX — это фактически два различных набора API: **курсорный** (cursor) API и **итераторный**
(iterator) API. Эти два набора API подробнее объясняются далее в уроке, но их основные особенности
кратко описаны ниже.

### Курсорный API

Как следует из названия, курсорный (cursor) API StAX представляет курсор, с помощью которого можно
пройти по XML-документу от начала до конца. Этот курсор может указывать на одну вещь за раз и всегда
движется только вперёд, никогда назад, обычно по одному элементу infoset за шаг.

Два основных курсорных интерфейса — `XMLStreamReader` и `XMLStreamWriter`. Интерфейс
`XMLStreamReader` включает методы доступа ко всей информации, которую можно извлечь из
информационной модели XML, включая кодировку документа, имена элементов, атрибуты, пространства
имён, текстовые узлы, открывающие теги, комментарии, инструкции по обработке, границы документа и
так далее; например:

```java
public interface XMLStreamReader {
    public int next() throws XMLStreamException;
    public boolean hasNext() throws XMLStreamException;

    public String getText();
    public String getLocalName();
    public String getNamespaceURI();
    // ... другие методы не показаны
}
```

Можно вызывать методы `XMLStreamReader`, такие как `getText` и `getName`, чтобы получить данные в
текущей позиции курсора. `XMLStreamWriter` предоставляет методы, соответствующие типам событий
`StartElement` и `EndElement`; например:

```java
public interface XMLStreamWriter {
    public void writeStartElement(String localName) throws XMLStreamException;
    public void writeEndElement() throws XMLStreamException;
    public void writeCharacters(String text) throws XMLStreamException;
    // ... другие методы не показаны
}
```

Курсорный API во многом повторяет SAX. Например, доступны методы для прямого обращения к строковой
и символьной информации, а целочисленные индексы можно использовать для доступа к информации об
атрибутах и пространствах имён. Как и в SAX, методы курсорного API возвращают информацию XML в
виде строк, что минимизирует затраты на выделение объектов.

### Итераторный API

Итераторный (iterator) API StAX представляет поток XML-документа как набор дискретных объектов
событий. Эти события вытягиваются приложением и предоставляются парсером в том порядке, в котором
они читаются в исходном XML-документе.

Базовый интерфейс итератора называется `XMLEvent`, и для каждого типа событий, перечисленного в
таблице типов `XMLEvent`, есть свой подынтерфейс. Основной интерфейс парсера для чтения событий
итератора — `XMLEventReader`, а основной интерфейс для записи событий итератора — `XMLEventWriter`.
Интерфейс `XMLEventReader` содержит пять методов, важнейший из которых — `nextEvent`, возвращающий
следующее событие в потоке XML. `XMLEventReader` реализует `java.util.Iterator`, что означает: то,
что возвращает `XMLEventReader`, можно кэшировать или передавать в процедуры, способные работать со
стандартным `Iterator` Java; например:

```java
public interface XMLEventReader extends Iterator {
    public XMLEvent nextEvent() throws XMLStreamException;
    public boolean hasNext();
    public XMLEvent peek() throws XMLStreamException;
    // ...
}
```

Аналогично на стороне вывода итераторного API имеется:

```java
public interface XMLEventWriter {
    public void flush() throws XMLStreamException;
    public void close() throws XMLStreamException;
    public void add(XMLEvent e) throws XMLStreamException;
    public void add(Attribute attribute) throws XMLStreamException;
    // ...
}
```

#### Типы событий итератора

Типы `XMLEvent`, определённые в итераторном API событий:

| Тип события | Описание |
|---|---|
| StartDocument | Сообщает о начале набора XML-событий, включая свойства кодировки, версии XML и standalone. |
| StartElement | Сообщает о начале элемента, включая любые атрибуты и объявления пространств имён; также даёт доступ к префиксу, URI пространства имён и локальному имени открывающего тега. |
| EndElement | Сообщает о закрывающем теге элемента. Здесь можно повторно получить пространства имён, вышедшие из области видимости, если они были явно заданы у соответствующего `StartElement`. |
| Characters | Соответствует секциям XML CData и сущностям CharacterData. Обратите внимание, что игнорируемые и значимые пробельные символы также сообщаются как события Character. |
| EntityReference | Символьные сущности могут сообщаться как отдельные события, которые разработчик приложения затем может разрешить или пропустить нерасрешёнными. По умолчанию сущности разрешаются. Кроме того, если вы не хотите сообщать сущность как событие, вместо неё можно подставить замещающий текст и сообщить его как Characters. |
| ProcessingInstruction | Сообщает цель (target) и данные для соответствующей инструкции по обработке. |
| Comment | Возвращает текст комментария. |
| EndDocument | Сообщает об окончании набора XML-событий. |
| DTD | Сообщает в виде `java.lang.String` информацию о DTD (если она есть), связанную с потоком, и предоставляет метод для возврата пользовательских объектов, найденных в DTD. |
| Attribute | Атрибуты обычно сообщаются как часть события `StartElement`. Однако бывают случаи, когда желательно вернуть атрибут как самостоятельное событие Attribute; например, когда пространство имён возвращается как результат выражения XQuery или XPath. |
| Namespace | Как и атрибуты, пространства имён обычно сообщаются как часть `StartElement`, но бывают случаи, когда желательно сообщить пространство имён как отдельное событие Namespace. |

Обратите внимание, что события `DTD`, `EntityDeclaration`, `EntityReference`, `NotationDeclaration`
и `ProcessingInstruction` создаются только в том случае, если обрабатываемый документ содержит DTD.

#### Пример отображения событий

В качестве примера того, как итераторный API событий отображает поток XML, рассмотрим следующий
XML-документ:

```xml
<?xml version="1.0"?>
<BookCatalogue xmlns="http://www.publishing.org">
    <Book>
        <Title>Yogasana Vijnana: the Science of Yoga</Title>
        <ISBN>81-40-34319-4</ISBN>
        <Cost currency="INR">11.50</Cost>
    </Book>
</BookCatalogue>
```

Этот документ был бы разобран на восемнадцать первичных и вторичных событий, как показано в
следующей таблице. Обратите внимание, что вторичные события, показанные в фигурных скобках (`{}`),
обычно доступны через первичное событие, а не напрямую.

Пример отображения событий итераторного API:

| № | Элемент/атрибут | Событие |
|---|---|---|
| 1 | `version="1.0"` | StartDocument |
| 2 | `isCData = false data = "\n" IsWhiteSpace = true` | Characters |
| 3 | `qname = BookCatalogue:http://www.publishing.org attributes = null namespaces = {BookCatalogue" -> http://www.publishing.org"}` | StartElement |
| 4 | `qname = Book attributes = null namespaces = null` | StartElement |
| 5 | `qname = Title attributes = null namespaces = null` | StartElement |
| 6 | `isCData = false data = "Yogasana Vijnana: the Science of Yoga\n\t" IsWhiteSpace = false` | Characters |
| 7 | `qname = Title namespaces = null` | EndElement |
| 8 | `qname = ISBN attributes = null namespaces = null` | StartElement |
| 9 | `isCData = false data = "81-40-34319-4\n\t" IsWhiteSpace = false` | Characters |
| 10 | `qname = ISBN namespaces = null` | EndElement |
| 11 | `qname = Cost attributes = {"currency" -> INR} namespaces = null` | StartElement |
| 12 | `isCData = false data = "11.50\n\t" IsWhiteSpace = false` | Characters |
| 13 | `qname = Cost namespaces = null` | EndElement |
| 14 | `isCData = false data = "\n" IsWhiteSpace = true` | Characters |
| 15 | `qname = Book namespaces = null` | EndElement |
| 16 | `isCData = false data = "\n" IsWhiteSpace = true` | Characters |
| 17 | `qname = BookCatalogue:http://www.publishing.org namespaces = {BookCatalogue" -> http://www.publishing.org"}` | EndElement |
| 18 | | EndDocument |

В этом примере есть несколько важных моментов:

* События создаются в том порядке, в котором соответствующие XML-элементы встречаются в документе,
  включая вложенность элементов, открытие и закрытие элементов, порядок атрибутов, начало и конец
  документа и так далее.
* Как и в правильном синтаксисе XML, все контейнерные элементы имеют соответствующие события начала
  и конца; например, у каждого `StartElement` есть соответствующий `EndElement`, даже для пустых
  элементов.
* События Attribute рассматриваются как вторичные события и доступны из соответствующего им события
  `StartElement`.
* Подобно событиям Attribute, события Namespace рассматриваются как вторичные, но появляются дважды
  и доступны дважды в потоке событий: сначала из соответствующего `StartElement`, а затем из
  соответствующего `EndElement`.
* События Character указаны для всех элементов, даже если у этих элементов нет символьных данных.
  Аналогично, события Character могут быть разбиты на несколько событий.
* Парсер StAX поддерживает стек пространств имён, в котором хранится информация обо всех
  пространствах имён XML, определённых для текущего элемента и его предков. Стек пространств имён,
  доступный через интерфейс `javax.xml.namespace.NamespaceContext`, можно адресовать по префиксу
  пространства имён или по URI.

#### Цели разработки

Авторы спецификации StAX ориентировались на три типа разработчиков:

* **Разработчики библиотек и инфраструктуры**: создают серверы приложений, реализации JAXM, JAXB,
  JAX-RPC и подобные; им нужны высокоэффективные, низкоуровневые API с минимальными требованиями
  к расширяемости.
* **Разработчики Java ME**: им нужны небольшие простые библиотеки pull-разбора с минимальными
  потребностями в расширяемости.
* **Разработчики Java Platform, Enterprise Edition (Java EE) и Java Platform, Standard Edition
  (Java SE)**: им нужны чистые, эффективные библиотеки pull-разбора, а также гибкость для чтения
  и записи потоков XML, создания новых типов событий и расширения элементов и атрибутов
  XML-документа.

Учитывая эти разнообразные категории разработки, авторы StAX сочли более полезным определить два
небольших эффективных API, чем перегружать один более крупный и неизбежно более сложный API.

#### Сравнение курсорного и итераторного API

Прежде чем выбирать между курсорным и итераторным API, отметьте несколько вещей, которые можно
сделать с помощью итераторного API, но нельзя с помощью курсорного:

* Объекты, созданные из подклассов `XMLEvent`, неизменяемы (immutable) и могут использоваться в
  массивах, списках и отображениях, а также передаваться по приложению даже после того, как парсер
  перешёл к последующим событиям.
* Можно создавать подтипы `XMLEvent`, которые являются либо совершенно новыми элементами
  информации, либо расширениями существующих элементов, но с дополнительными методами.
* Можно добавлять и удалять события из потока XML-событий гораздо более простыми способами, чем с
  курсорным API.

Аналогично, держите в уме несколько общих рекомендаций при выборе:

* Если вы программируете для особенно ограниченного по памяти окружения, как Java ME, то с курсорным
  API можно получить меньший и более эффективный код.
* Если производительность для вас в высшем приоритете — например, при создании низкоуровневых
  библиотек или инфраструктуры — курсорный API эффективнее.
* Если вы хотите создавать конвейеры обработки XML, используйте итераторный API.
* Если вы хотите изменять поток событий, используйте итераторный API.
* Если вы хотите, чтобы приложение могло обрабатывать поток событий подключаемым (pluggable)
  образом, используйте итераторный API.
* В целом, если у вас нет сильного предпочтения в ту или иную сторону, рекомендуется использовать
  итераторный API, поскольку он более гибок и расширяем, тем самым «защищая ваши приложения на
  будущее» (future-proofing).

## Использование StAX

В общем случае программисты StAX создают потоковые читатели, писатели и события XML с помощью
классов `XMLInputFactory`, `XMLOutputFactory` и `XMLEventFactory`. Настройка выполняется заданием
свойств у фабрик: специфичные для реализации параметры можно передавать в нижележащую реализацию с
помощью метода `setProperty` у фабрик. Аналогично специфичные для реализации параметры можно
запросить с помощью фабричного метода `getProperty`.

Классы `XMLInputFactory`, `XMLOutputFactory` и `XMLEventFactory` описаны ниже, затем следуют
обсуждения выделения ресурсов, управления пространствами имён и атрибутами, обработки ошибок и,
наконец, чтения и записи потоков с использованием курсорного и итераторного API.

### Фабричные классы StAX

Фабричные классы StAX — `XMLInputFactory`, `XMLOutputFactory` и `XMLEventFactory` — позволяют
определять и настраивать экземпляры реализаций классов потокового читателя, потокового писателя и
событий XML.

#### XMLInputFactory

Класс `XMLInputFactory` позволяет настраивать экземпляры реализаций процессоров-читателей потока
XML, создаваемых фабрикой. Новые экземпляры абстрактного класса `XMLInputFactory` создаются вызовом
метода `newInstance` у класса. Затем для создания нового экземпляра фабрики используется
статический метод `XMLInputFactory.newInstance`.

Наследуя от JAXP, метод `XMLInputFactory.newInstance` определяет, какой именно класс реализации
`XMLInputFactory` загрузить, с помощью следующей процедуры поиска:

1. Использовать системное свойство `javax.xml.stream.XMLInputFactory`.
2. Использовать файл `lib/xml.stream.properties` в каталоге среды выполнения Java (JRE) платформы
   Java SE.
3. Использовать Services API, если он доступен, чтобы определить имя класса, заглядывая в файлы
   `META-INF/services/javax.xml.stream.XMLInputFactory` в JAR-файлах, доступных JRE.
4. Использовать экземпляр `XMLInputFactory` платформы по умолчанию.

Получив ссылку на подходящую `XMLInputFactory`, приложение может использовать фабрику для настройки
и создания экземпляров потоков. В следующей таблице перечислены свойства, поддерживаемые
`XMLInputFactory`. Более подробный список см. в спецификации StAX.

Свойства `javax.xml.stream.XMLInputFactory`:

| Свойство | Описание |
|---|---|
| isValidating | Включает специфичную для реализации валидацию. |
| isCoalescing | **(Обязательно)** Требует от процессора объединять (coalesce) смежные символьные данные. |
| isNamespaceAware | Отключает поддержку пространств имён. Все реализации обязаны поддерживать пространства имён. Поддержка документов без пространств имён необязательна. |
| isReplacingEntityReferences | **(Обязательно)** Требует от процессора заменять внутренние ссылки на сущности их замещающим значением и сообщать их как символы либо как набор событий, описывающих сущность. |
| isSupportingExternalEntities | **(Обязательно)** Требует от процессора разрешать внешние разбираемые сущности. |
| reporter | **(Обязательно)** Задаёт и возвращает реализацию интерфейса `XMLReporter`. |
| resolver | **(Обязательно)** Задаёт и возвращает реализацию интерфейса `XMLResolver`. |
| allocator | **(Обязательно)** Задаёт и возвращает реализацию интерфейса `XMLEventAllocator`. |

#### XMLOutputFactory

Новые экземпляры абстрактного класса `XMLOutputFactory` создаются вызовом метода `newInstance` у
класса. Затем для создания нового экземпляра фабрики используется статический метод
`XMLOutputFactory.newInstance`. Алгоритм получения экземпляра тот же, что и для `XMLInputFactory`,
но ссылается на системное свойство `javax.xml.stream.XMLOutputFactory`.

`XMLOutputFactory` поддерживает только одно свойство — `javax.xml.stream.isRepairingNamespaces`.
Это свойство обязательно, и его назначение — создавать префиксы по умолчанию и связывать их с URI
пространств имён. Дополнительную информацию см. в спецификации StAX.

#### XMLEventFactory

Новые экземпляры абстрактного класса `XMLEventFactory` создаются вызовом метода `newInstance` у
класса. Затем для создания нового экземпляра фабрики используется статический метод
`XMLEventFactory.newInstance`. Эта фабрика ссылается на свойство `javax.xml.stream.XMLEventFactory`
для инстанцирования фабрики. Алгоритм получения экземпляра тот же, что и для `XMLInputFactory` и
`XMLOutputFactory`, но ссылается на системное свойство `javax.xml.stream.XMLEventFactory`.

Свойств по умолчанию для `XMLEventFactory` нет.

### Ресурсы, пространства имён и ошибки

Спецификация StAX обрабатывает разрешение ресурсов, атрибуты и пространства имён, а также ошибки и
исключения, как описано ниже.

#### Разрешение ресурсов

Интерфейс `XMLResolver` предоставляет средство для задания метода, который разрешает ресурсы во
время обработки XML. Приложение задаёт интерфейс у `XMLInputFactory`, которая затем устанавливает
этот интерфейс у всех процессоров, создаваемых этим экземпляром фабрики.

#### Атрибуты и пространства имён

Атрибуты сообщаются процессором StAX с помощью методов поиска и строк в курсорном интерфейсе и с
помощью событий Attribute и Namespace в итераторном интерфейсе. Заметьте здесь, что пространства
имён трактуются как атрибуты, хотя пространства имён сообщаются отдельно от атрибутов как в
курсорном, так и в итераторном API. Заметьте также, что обработка пространств имён для процессоров
StAX необязательна. Полную информацию о связывании пространств имён и необязательной обработке
пространств имён см. в спецификации StAX.

#### Сообщение об ошибках и обработка исключений

Все фатальные ошибки сообщаются посредством интерфейса `javax.xml.stream.XMLStreamException`. Все
нефатальные ошибки и предупреждения сообщаются с помощью интерфейса `javax.xml.stream.XMLReporter`.

### Чтение потоков XML

Как было описано ранее в этом уроке, способ чтения потоков XML процессором StAX — и, что важнее,
то, что вы получаете обратно, — существенно различается в зависимости от того, используете вы
курсорный API StAX или итераторный API событий. В следующих двух разделах описано, как читать
потоки XML каждым из этих API.

#### Использование XMLStreamReader

Интерфейс `XMLStreamReader` в курсорном API StAX позволяет читать потоки или документы XML только
в направлении вперёд, по одному элементу infoset за раз. Для вытягивания данных из потока или
пропуска нежелательных событий доступны следующие методы:

* получить значение атрибута;
* прочитать содержимое XML;
* определить, есть ли у элемента содержимое или он пуст;
* получить индексированный доступ к коллекции атрибутов;
* получить индексированный доступ к коллекции пространств имён;
* получить имя текущего события (если применимо);
* получить содержимое текущего события (если применимо).

Экземпляры `XMLStreamReader` имеют в любой момент времени единственное текущее событие, над которым
работают их методы. Когда вы создаёте экземпляр `XMLStreamReader` на потоке, начальное текущее
событие — состояние `START_DOCUMENT`. Затем для перехода к следующему событию в потоке можно
использовать метод `XMLStreamReader.next`.

##### Чтение свойств, атрибутов и пространств имён

Метод `XMLStreamReader.next` загружает свойства следующего события в потоке. Затем к этим свойствам
можно обращаться, вызывая методы `XMLStreamReader.getLocalName` и `XMLStreamReader.getText`.

Когда курсор `XMLStreamReader` находится над событием `StartElement`, он читает имя и любые атрибуты
события, включая пространство имён. Ко всем атрибутам события можно обращаться по значению индекса,
а также искать их по URI пространства имён и локальному имени. Заметьте, однако, что доступны только
пространства имён, объявленные у текущего `StartEvent`; ранее объявленные пространства имён не
сохраняются, а переобъявленные пространства имён не удаляются.

##### Методы XMLStreamReader

`XMLStreamReader` предоставляет следующие методы для получения информации о пространствах имён и
атрибутах:

```java
int getAttributeCount();
String getAttributeNamespace(int index);
String getAttributeLocalName(int index);
String getAttributePrefix(int index);
String getAttributeType(int index);
String getAttributeValue(int index);
String getAttributeValue(String namespaceUri, String localName);
boolean isAttributeSpecified(int index);
```

К пространствам имён также можно обращаться с помощью трёх дополнительных методов:

```java
int getNamespaceCount();
String getNamespacePrefix(int index);
String getNamespaceURI(int index);
```

##### Инстанцирование XMLStreamReader

Этот пример, взятый из спецификации StAX, показывает, как инстанцировать входную фабрику, создать
читатель и проитерировать элементы потока XML:

```java
XMLInputFactory f = XMLInputFactory.newInstance();
XMLStreamReader r = f.createXMLStreamReader( ... );
while(r.hasNext()) {
    r.next();
}
```

#### Использование XMLEventReader

API `XMLEventReader` в итераторном API событий StAX даёт средство отображать события в потоке XML
на выделенные объекты событий, которые можно свободно переиспользовать, а сам API можно расширять
для обработки пользовательских событий.

`XMLEventReader` предоставляет четыре метода для итеративного разбора потоков XML:

* `next`: возвращает следующее событие в потоке;
* `nextEvent`: возвращает следующее типизированное `XMLEvent`;
* `hasNext`: возвращает `true`, если в потоке есть ещё события для обработки;
* `peek`: возвращает событие, но не переходит к следующему событию.

Например, следующий фрагмент кода иллюстрирует объявления методов `XMLEventReader`:

```java
package javax.xml.stream;
import java.util.Iterator;

public interface XMLEventReader extends Iterator {
    public Object next();
    public XMLEvent nextEvent() throws XMLStreamException;
    public boolean hasNext();
    public XMLEvent peek() throws XMLStreamException;
    // ...
}
```

Чтобы прочитать все события в потоке, а затем напечатать их, можно использовать следующее:

```java
while(stream.hasNext()) {
    XMLEvent event = stream.nextEvent();
    System.out.print(event);
}
```

##### Чтение атрибутов

К атрибутам можно обращаться из их связанного `javax.xml.stream.StartElement` следующим образом:

```java
public interface StartElement extends XMLEvent {
    public Attribute getAttributeByName(QName name);
    public Iterator getAttributes();
}
```

Можно использовать метод `getAttributes` интерфейса `StartElement`, чтобы получить `Iterator` по
всем атрибутам, объявленным у этого `StartElement`.

##### Чтение пространств имён

Аналогично чтению атрибутов, пространства имён читаются с помощью `Iterator`, создаваемого вызовом
метода `getNamespaces` у интерфейса `StartElement`. Возвращается только пространство имён текущего
`StartElement`, а приложение может получить текущий контекст пространства имён с помощью
`StartElement.getNamespaceContext`.

### Запись потоков XML

StAX — двунаправленный API, и как курсорный, так и итераторный API событий имеют собственный набор
интерфейсов для записи потоков XML. Как и в случае интерфейсов для чтения потоков, между API-
писателями для курсора и итератора событий есть существенные различия. В следующих разделах
описано, как записывать потоки XML каждым из этих API.

#### Использование XMLStreamWriter

Интерфейс `XMLStreamWriter` в курсорном API StAX позволяет приложениям записывать обратно в поток
XML или создавать совершенно новые потоки. У `XMLStreamWriter` есть методы, которые позволяют:

* записывать правильно сформированный (well-formed) XML;
* сбрасывать (flush) или закрывать вывод;
* записывать квалифицированные имена.

Заметьте, что реализации `XMLStreamWriter` не обязаны выполнять проверки правильности формирования
или валидности ввода. Хотя некоторые реализации могут выполнять строгую проверку ошибок, другие
могут этого не делать. Реализуемые вами правила применяются к свойствам, определённым в классе
`XMLOutputFactory`.

Метод `writeCharacters` используется для экранирования таких символов, как `&`, `<`, `>` и `"`.
Связывание префиксов можно выполнить либо передачей фактического значения для префикса, либо с
помощью метода `setPrefix`, либо заданием свойства для объявлений пространств имён по умолчанию.

Следующий пример, взятый из спецификации StAX, показывает, как инстанцировать выходную фабрику,
создать писатель и записать вывод XML:

```java
XMLOutputFactory output = XMLOutputFactory.newInstance();
XMLStreamWriter writer = output.createXMLStreamWriter( ... );

writer.writeStartDocument();
writer.setPrefix("c","http://c");
writer.setDefaultNamespace("http://c");

writer.writeStartElement("http://c","a");
writer.writeAttribute("b","blah");
writer.writeNamespace("c","http://c");
writer.writeDefaultNamespace("http://c");

writer.setPrefix("d","http://c");
writer.writeEmptyElement("http://c","d");
writer.writeAttribute("http://c", "chris","fry");
writer.writeNamespace("d","http://c");
writer.writeCharacters("Jean Arp");
writer.writeEndElement();

writer.flush();
```

Этот код генерирует следующий XML (переводы строк не нормативны):

```xml
<?xml version='1.0' encoding='utf-8'?>
<a b="blah" xmlns:c="http://c" xmlns="http://c">
<d:d d:chris="fry" xmlns:d="http://c"/>Jean Arp</a>
```

#### Использование XMLEventWriter

Интерфейс `XMLEventWriter` в итераторном API событий StAX позволяет приложениям записывать обратно
в поток XML или создавать совершенно новые потоки. Этот API можно расширять, но основной API таков:

```java
public interface XMLEventWriter {
    public void flush() throws XMLStreamException;
    public void close() throws XMLStreamException;
    public void add(XMLEvent e) throws XMLStreamException;
    // ... другие методы не показаны.
}
```

Экземпляры `XMLEventWriter` создаются экземпляром `XMLOutputFactory`. События потока добавляются
итеративно, и событие нельзя изменить после того, как оно было добавлено в экземпляр писателя
событий.

##### Атрибуты, экранирование символов, связывание префиксов

Реализации StAX обязаны буферизовать последний `StartElement`, пока в поток не будет добавлено или
встречено событие, отличное от Attribute или Namespace. Это означает, что когда вы добавляете
Attribute или Namespace в поток, он присоединяется к текущему событию `StartElement`.

Можно использовать метод `Characters` для экранирования таких символов, как `&`, `<`, `>` и `"`.

Метод `setPrefix(...)` можно использовать, чтобы явно связать префикс для использования при выводе,
а метод `getPrefix(...)` — чтобы получить текущий префикс. Заметьте, что по умолчанию
`XMLEventWriter` добавляет связывания пространств имён в свою внутреннюю карту пространств имён.
Префиксы выходят из области видимости после соответствующего `EndElement` для события, в котором
они были связаны.

## Реализация потокового XML-парсера от Oracle

Application Server 9.1 включает реализацию JSR 173 (StAX) от Sun Microsystems под названием Sun
Java Streaming XML Parser (далее — Streaming XML Parser). Streaming XML Parser — это
высокоскоростной, невалидирующий, совместимый с W3C XML 1.0 и Namespace 1.0 потоковый pull-парсер
XML, построенный на кодовой базе Xerces2.

В реализации Streaming XML Parser от Sun нижние уровни Xerces2, в частности Scanner и связанные с
ним классы, были переработаны так, чтобы вести себя в pull-манере. Помимо изменений в нижних
уровнях, Streaming XML Parser включает дополнительную функциональность, связанную со StAX, и
множество улучшений, повышающих производительность. Streaming XML Parser реализован в файлах
`appserv-ws.jar` и `javaee.jar`, оба из которых расположены в каталоге `install_dir/lib/`.

В состав эталонной реализации JAXP входят примеры кода StAX, расположенные в каталоге
`INSTALL_DIR/jaxp-version/samples/stax`, которые иллюстрируют, как работает реализация Streaming
XML Parser от Sun. Эти примеры описаны в разделе [Примеры кода](#примеры-кода).

Прежде чем переходить к примерам кода, вам следует знать о двух аспектах Streaming XML Parser:

* Сообщение о событиях CDATA;
* Реализация фабрик в Streaming XML Parser.

Эти темы обсуждаются ниже.

### Сообщение о событиях CDATA

`javax.xml.stream.XMLStreamReader`, реализованный в Streaming XML Parser, не сообщает о событиях
CDATA. Если у вас есть приложение, которому нужно получать такие события, настройте
`XMLInputFactory`, установив следующее специфичное для реализации свойство `report-cdata-event`:

```java
XMLInputFactory factory = XMLInptuFactory.newInstance();
factory.setProperty("report-cdata-event", Boolean.TRUE);
```

### Реализация фабрик в Streaming XML Parser

Большинству приложений не нужно знать имя класса реализации фабрики. Для большинства приложений
достаточно просто добавить файлы `javaee.jar` и `appserv-ws.jar` в classpath, поскольку эти два
JAR-файла поставляют имя класса реализации фабрики для различных свойств Streaming XML Parser в
каталоге `META-INF/services` — например, `javax.xml.stream.XMLInputFactory`,
`javax.xml.stream.XMLOutputFactory` и `javax.xml.stream.XMLEventFactory`, — что является третьим
шагом операции поиска, когда приложение запрашивает экземпляр фабрики. Дополнительную информацию о
механизме поиска см. в Javadoc для метода `XMLInputFactory.newInstance`.

Однако могут возникнуть сценарии, когда приложение хотело бы знать имя класса реализации фабрики и
задать свойство явно. К таким сценариям могут относиться случаи, когда в classpath есть несколько
реализаций JSR 173, и приложение хочет выбрать одну из них — возможно, ту, что имеет более высокую
производительность, содержит важное исправление ошибки и тому подобное.

Если приложение устанавливает SystemProperty, это первый шаг операции поиска, и поэтому получение
экземпляра фабрики будет быстрым по сравнению с другими вариантами; например:

```
javax.xml.stream.XMLInputFactory -->
  com.sun.xml.stream.ZephyrParserFactory

javax.xml.stream.XMLOutputFactory -->
  com.sun.xml.stream.ZephyrWriterFactor

javax.xml.stream.XMLEventFactory -->
  com.sun.xml.stream.events.ZephyrEventFactory
```

## Примеры кода

В состав эталонной реализации JAXP входят примеры кода StAX, расположенные в каталоге
`INSTALL_DIR/jaxp-version/samples/stax`. Предоставляется шесть каталогов примеров StAX:

1. **Пример курсора** (каталог `cursor/`) — файл `CursorParse.java`. Демонстрирует использование
   курсорного API `XMLStreamReader` для чтения XML-файла. Приложение вызывает `next()` для чтения
   событий, возвращающего целочисленные константы; методы доступа предоставляют информацию о каждом
   событии.
2. **Пример «курсор в событие»** (каталог `cursor2event/`) — файл `CursorApproachEventObject.java`.
   Показывает, как получать объекты `XMLEvent`, используя при этом курсорный API. Использует
   `XMLEventAllocator`, чтобы обернуть результаты курсора в объекты событий; объединяет
   эффективность курсорного API с удобством объектов событий.
3. **Пример событий** (каталог `event/`) — файл `EventParse.java`. Демонстрирует использование
   итераторного API событий `XMLEventReader`. Создаёт `XMLInputFactory` и `XMLEventReader`,
   итерирует события циклом `while` с `hasNext()` и `nextEvent()`.
4. **Пример фильтра** (каталог `filter/`) — файл `MyStreamFilter.java`. Показывает, как использовать
   API потоковых фильтров StAX. Фильтрует так, чтобы принимать только события `StartElement` и
   `EndElement`; реализует интерфейс `javax.xml.stream.StreamFilter`.
5. **Пример «чтение и запись»** (каталог `readnwrite/`) — файл `EventProducerConsumer.java`.
   Демонстрирует одновременное чтение и запись потоков XML. Использует `XMLEventReader`
   (производитель) и `XMLEventWriter` (потребитель); показывает, как изменять потоки и динамически
   добавлять события.
6. **Пример писателя** (каталог `writer/`) — файл `CursorWriter.java`. Демонстрирует использование
   `XMLStreamWriter` для программной записи XML. Единственный пример, не использующий
   `BookCatalog.xml`.

### Пример курсора

**Расположение:** `INSTALL_DIR/jaxp-version/samples/stax/cursor/`

Курсорный API использует `XMLStreamReader` для последовательного чтения XML. Метод `next()`
возвращает целочисленные константы, представляющие типы событий. Приложения должны вызывать методы
доступа, чтобы получить информацию о каждом событии.

Базовый шаблон:

```java
XMLStreamReader xmlr = xmlif.createXMLStreamReader(filename,
                           new FileInputStream(filename));

int eventType = xmlr.getEventType();
printEventType(eventType);
printStartDocument(xmlr);

while(xmlr.hasNext()) {
    eventType = xmlr.next();
    printEventType(eventType);

    printStartElement(xmlr);
    printEndElement(xmlr);
    printText(xmlr);
    printPIData(xmlr);
    printComment(xmlr);
}
```

Константы типов событий:

```java
public final static String getEventTypeString(int eventType) {

            return "PROCESSING_INSTRUCTION";
        case XMLEvent.CHARACTERS:
            return "CHARACTERS";
        case XMLEvent.COMMENT:
            return "COMMENT";
        case XMLEvent.START_DOCUMENT:
            return "START_DOCUMENT";
        case XMLEvent.END_DOCUMENT:
            return "END_DOCUMENT";
        case XMLEvent.ENTITY_REFERENCE:
            return "ENTITY_REFERENCE";
        case XMLEvent.ATTRIBUTE:
            return "ATTRIBUTE";
        case XMLEvent.DTD:
            return "DTD";

            return "CDATA";
```

### Пример «курсор в событие»

**Расположение:** `INSTALL_DIR/javaeetutorial5/examples/stax/cursor2event/`

Связывает курсорный и событийный API с помощью `XMLEventAllocator`.

Настройка:

```java
XMLInputFactory xmlif = XMLInputFactory.newInstance();

                           new FileInputStream(filename));
```

Итерация по событиям:

```java
int eventType = xmlr.getEventType();

while (xmlr.hasNext()) {
    eventType = xmlr.next();
    // Получить все элементы "Book" как объект XMLEvent
    if (eventType == XMLStreamConstants.START_ELEMENT
        && xmlr.getLocalName().equals("Book")) {
        // получить неизменяемый XMLEvent
        StartElement event = getXMLEvent(xmlr).asStartElement();
        System.out.println ("EVENT: " + event.toString());
    }
}
```

Метод аллокатора:

```java
private static XMLEvent getXMLEvent(XMLStreamReader reader)
    throws XMLStreamException {
    return allocator.allocate(reader);
}
```

Настройка входной фабрики:

```java
XMLInputFactory xmlif = null ;

try {
    xmlif = XMLInputFactory.newInstance();
    xmlif.setProperty(
        XMLInputFactory.IS_REPLACING_ENTITY_REFERENCES,

        Boolean.FALSE);

    xmlif.setProperty(
        XMLInputFactory.IS_NAMESPACE_AWARE,
        Boolean.TRUE);

    xmlif.setProperty(
        XMLInputFactory.IS_COALESCING,
        Boolean.TRUE);

    ex.printStackTrace();
}

System.out.println("FACTORY: " + xmlif);
System.out.println("filename = "+ filename);
```

```java
FileInputStream fis = new FileInputStream(filename);
XMLStreamReader xmlr = xmlif.createFilteredReader(
                           xmlif.createXMLStreamReader(fis),
                           new MyStreamFilter());

int eventType = xmlr.getEventType();
printEventType(eventType);

while (xmlr.hasNext()) {
    eventType = xmlr.next();
    printEventType(eventType);
    printName(xmlr,eventType);
    printText(xmlr);

    if (xmlr.isStartElement()) {

}
```

```java
public boolean accept(XMLStreamReader reader) {
```

### Пример «чтение и запись»

**Расположение:** `INSTALL_DIR/jaxp-version/samples/stax/readnwrite/`

Демонстрирует одновременное чтение и запись. `XMLEventReader` выступает **производителем**
(producer), а `XMLEventWriter` — **потребителем** (consumer).

Настройка производителя/потребителя событий:

```java
XMLEventFactory m_eventFactory = XMLEventFactory.newInstance();

public EventProducerConsumer() {
    // ...
    try {
        EventProducerConsumer ms = new EventProducerConsumer();

        XMLEventReader reader = XMLInputFactory.newInstance().
        createXMLEventReader(new java.io.FileInputStream(args[0]));

        XMLEventWriter writer =
            XMLOutputFactory.newInstance().createXMLEventWriter(System.out);
    }
    // ...
}
```

Итерация по событиям:

```java
while (reader.hasNext()) {
    XMLEvent event = (XMLEvent)reader.next();
    if (event.getEventType() == event.CHARACTERS) {
        writer.add(ms.getNewCharactersEvent(event.asCharacters()));
    }
    else {
        writer.add(event);
    }
}
writer.flush();
```

Изменение события Characters:

```java
Characters getNewCharactersEvent(Characters event) {
    if (event.getData().equalsIgnoreCase("Name1")) {
        return m_eventFactory.createCharacters(
                   Calendar.getInstance().getTime().toString());
    }
    // иначе вернуть то же самое событие
    else {
        return event;
    }
}
```

Вывод: при запуске `EventProducerConsumer` выводит содержимое `BookCatalog.xml`, как описано в
разделе «Пример XML-документа».

Запуск:

```bash
javac -classpath ../lib/jaxp-ri.jar stax/readnwrite/*.java
java stax/readnwrite/EventProducerConsumer stax/data/BookCatalogue.xml
```

### Пример писателя

**Расположение:** `INSTALL_DIR/jaxp-version/samples/stax/writer/`

Демонстрирует использование `XMLStreamWriter` для программного создания XML.

Выходная фабрика:

```java
XMLOutputFactory xof =  XMLOutputFactory.newInstance();
```

Потоковый писатель:

```java
XMLStreamWriter xtw = null;
```

Запись XML:

```java
xtw = xof.createXMLStreamWriter(new FileWriter(fileName));

xtw.writeComment("all elements here are explicitly in the HTML namespace");
xtw.writeStartDocument("utf-8","1.0");
xtw.setPrefix("html", "http://www.w3.org/TR/REC-html40");
xtw.writeStartElement("http://www.w3.org/TR/REC-html40","html");
xtw.writeNamespace("html", "http://www.w3.org/TR/REC-html40");
xtw.writeStartElement("http://www.w3.org/TR/REC-html40", "head");
xtw.writeStartElement("http://www.w3.org/TR/REC-html40", "title");
xtw.writeCharacters("Frobnostication");
xtw.writeEndElement();
xtw.writeEndElement();

xtw.writeStartElement("http://www.w3.org/TR/REC-html40", "body");
xtw.writeStartElement("http://www.w3.org/TR/REC-html40", "p");
xtw.writeCharacters("Moved to");
xtw.writeStartElement("http://www.w3.org/TR/REC-html40", "a");
xtw.writeAttribute("href","http://frob.com");
xtw.writeCharacters("here");
xtw.writeEndElement();
xtw.writeEndElement();
xtw.writeEndElement();
xtw.writeEndElement();
xtw.writeEndDocument();

xtw.flush();
xtw.close();
```

Вывод (`dist/CursorWriter-Output`):

```xml
<!--all elements here are explicitly in the HTML namespace-->
<?xml version="1.0" encoding="utf-8"?>
<html:html xmlns:html="http://www.w3.org/TR/REC-html40">
<html:head>
<html:title>Frobnostication</html:title></html:head>
<html:body>
<html:p>Moved to <html:a href="http://frob.com">here</html:a>
</html:p>
</html:body>
</html:html>
```

(Примечание: в фактическом файле переводов строк нет; разрывы добавлены здесь для удобочитаемости.)

Запуск:

```bash
javac -classpath ../lib/jaxp-ri.jar stax/writer/*.java
java stax/writer/CursorWriter -f output_file
```
