# Урок 3. Настройка программного обеспечения

**Трейл:** JNDI · **Оригинал:** [Software Setup](https://docs.oracle.com/javase/tutorial/jndi/software/index.html)
**Связанные области:** [[01-core-java-syntax-oop]] · **Вопросы:** core-java

> Перевод официального руководства Oracle (The Java Tutorials, JDK 8). Руководство написано для JDK 8.
> Примеры и практики, описанные на этой странице, не используют улучшения, появившиеся в более
> поздних выпусках, и могут опираться на технологии, которые более недоступны.

## Необходимое программное обеспечение

Ниже приведён список программ и систем, которые вам понадобятся:

- программное обеспечение платформы Java (Java platform software);
- программное обеспечение поставщика услуг (service provider software);
- программное обеспечение сервера имён и каталогов (naming and directory server software).

### Программное обеспечение платформы Java

JNDI входит в состав платформы Java SE.

> Чтобы запускать апплеты (applets), используйте режим IE (IE mode) в Microsoft Edge.
> См. руководство «Microsoft Edge + Internet Explorer mode: Getting Started guide».

### Программное обеспечение поставщика услуг

JNDI API — это универсальный (generic) интерфейс для доступа к любой службе имён (naming service)
или службе каталогов (directory service). Фактический доступ к службе имён или каталогов
обеспечивается за счёт подключения **поставщика услуг (service provider)** к JNDI. Обзор архитектуры
JNDI и роли поставщиков услуг приведён в уроке «Обзор JNDI» (JNDI Overview).

**Поставщик услуг (service provider)** — это программное обеспечение, которое отображает (maps)
вызовы JNDI API в фактические вызовы сервера имён или каталогов. Как правило, роли поставщика услуг
и сервера имён/каталогов различаются. В терминологии клиент-серверного (client/server) программного
обеспечения JNDI и поставщик услуг — это **клиент** (называемый **JNDI-клиентом**, JNDI client),
а сервер имён/каталогов — это **сервер** (server).

Клиенты и серверы могут взаимодействовать множеством способов. В одном из распространённых вариантов
они используют сетевой протокол (network protocol), так что клиент и сервер могут существовать
автономно в сетевой среде. Сервер обычно обслуживает множество разных клиентов, а не только
JNDI-клиентов, при условии что клиенты соответствуют заданному протоколу. JNDI не диктует какого-либо
конкретного стиля взаимодействия между JNDI-клиентами и серверами. Например, в крайнем случае клиент
и сервер могут быть одной и той же сущностью.

Вам потребуется получить классы тех поставщиков услуг, которые вы будете использовать. Например,
если вы планируете применять JNDI для доступа к серверу каталогов LDAP, то вам понадобится
программное обеспечение поставщика услуг для LDAP.

В состав JDK входят поставщики услуг для:

- Light Weight Directory Protocol (LDAP) — облегчённый протокол доступа к каталогам;
- CORBA Common Object Services naming (COS naming) — служба именования общих объектных служб CORBA;
- реестра RMI (RMI registry);
- Domain Name Service (DNS) — служба доменных имён.

Если вас интересуют другие поставщики услуг, проверьте информацию о загрузке на странице JNDI.

В этом руководстве используется только поставщик услуг LDAP. При использовании поставщика услуг LDAP
вам нужно либо настроить собственный сервер, либо иметь доступ к существующему серверу, как описано
далее.

### Программное обеспечение сервера имён и каталогов

После того как вы получили программное обеспечение поставщика услуг, вам нужно настроить
соответствующий сервер имён/каталогов или иметь к нему доступ. Настройка сервера имён/каталогов
обычно является задачей сетевого системного администратора. У разных производителей разные процедуры
установки своих серверов имён/каталогов. Некоторые из них требуют специальных привилегий на машине,
прежде чем сервер можно будет установить. Следует обращаться к инструкциям по установке программного
обеспечения сервера имён/каталогов.

Для примеров с каталогами в этом руководстве вам нужен доступ к серверу LDAP. Если вы хотите кратко
ознакомиться с тем, что такое LDAP, посмотрите [статью в Википедии](http://en.wikipedia.org/wiki/LDAP).
Вы можете использовать любой LDAP-совместимый сервер по вашему выбору. Oracle Directory Server,
работающий на многих платформах, включая Windows, доступен для ознакомления по адресу
[Oracle Directory Server](http://www.oracle.com/technetwork/testcontent/index-085178.html).

Вы также можете загрузить бесплатный сервер LDAP по ссылкам ниже:

- [389 Directory Server](http://directory.fedoraproject.org/);
- [Apache Directory Server](http://directory.apache.org);
- [OpenLDAP](http://www.OpenLDAP.org/).

Публично доступный сервер находится по адресу `ldap://ldap.openldap.org`. Контекст имён (naming
context): `dc=OpenLDAP,dc=org`.

## Настройка LDAP

Ниже приведены шаги по созданию Java-приложения, обращающегося к серверу каталогов LDAP.

1. Установить программное обеспечение платформы Java.
2. Получить программное обеспечение сервера каталогов (Directory Server), как обсуждалось ранее.
3. Настроить сервер каталогов с нужной схемой (schema). Для использования примеров из этого
   руководства на сервере нужно настроить специальную схему.
4. Наполнить сервер каталогов нужным содержимым. Для использования примеров из этого руководства
   на сервере нужно разместить специальное содержимое.
5. Написать JNDI-приложение для доступа к каталогу, скомпилировать и запустить его относительно
   сервера каталогов, чтобы получить желаемые результаты. JNDI-примеры рассматриваются в следующем
   уроке.

**Первые два** шага рассмотрены в предыдущем разделе. Остальная часть этого урока обсуждает шаг
**три** и часть шага **четыре**. Шаг **пять**, связанный с написанием JNDI-приложения, рассмотрен
в следующем уроке, где показано, как писать JNDI-приложения для выполнения различных операций над
каталогом.

После того как вы настроили каталог (либо направили программу на взаимодействие с существующим
каталогом), какого рода информацию можно ожидать там обнаружить?

Каталог можно рассматривать как состоящий из **связываний «имя — объект» (name-to-object bindings)**.
То есть каждому объекту в каталоге соответствует имя. Объект в каталоге можно получить, выполнив поиск
по его имени.

Также в каталоге хранятся **атрибуты (attributes)**. Объект в каталоге, помимо имени, имеет ещё
и необязательный набор атрибутов. У каталога можно запросить атрибуты объекта, а также попросить его
найти объект, обладающий определёнными атрибутами.

## Шаг 3. Схема каталога

**Схема (schema)** задаёт типы объектов, которые может содержать каталог. Это руководство наполняет
каталог записями (entries), часть из которых требует особых определений схемы. Чтобы разместить эти
записи, необходимо либо отключить проверку схемы (schema-checking) на сервере, либо добавить на сервер
файлы схемы, прилагаемые к этому руководству. Обе эти задачи обычно выполняются администратором сервера
каталогов.

К этому руководству прилагаются два файла схемы, которые необходимо установить:

- [`Схема для объектов Java`](https://docs.oracle.com/javase/tutorial/jndi/software/config/java.schema);
- [`Схема для объектов CORBA`](https://docs.oracle.com/javase/tutorial/jndi/software/config/corba.schema).

Формат этих файлов — формальное описание, которое, возможно, нельзя напрямую скопировать и вставить
в конфигурационные файлы сервера. В частности, синтаксисы атрибутов (attribute syntaxes) описаны
в терминах [RFC 2252](http://www.ietf.org/rfc/rfc2252.txt).

У разных серверов каталогов разные способы настройки их схемы. Это руководство включает некоторые
инструменты для установки схем Java и CORBA на серверы каталогов, позволяющие изменять свою схему
через LDAP. Ниже приведён список задач, которые могут выполнять эти инструменты.

1. [`Создать схему Java`](https://docs.oracle.com/javase/tutorial/jndi/software/config/CreateJavaSchema.java)
   (`CreateJavaSchema.java`);
2. [`Создать схему CORBA`](https://docs.oracle.com/javase/tutorial/jndi/software/config/CreateCorbaSchema.java)
   (`CreateCorbaSchema.java`).

Чтобы запустить эти программы, следуйте инструкциям в прилагаемом
[`файле README`](https://docs.oracle.com/javase/tutorial/jndi/software/config/README-SCHEMA.TXT).

> **Примечание: Windows Active Directory.** Active Directory управляет своей схемой во внутреннем
> формате. Чтобы обновить схему, можно использовать либо оснастку (snap-in) консоли управления
> Active Directory — ADSIEdit, либо утилиту `CreateJavaSchema`, следуя инструкциям для Active Directory.

## Шаг 4. Подготовка содержимого каталога для этого руководства

В примерах этого трейла показанные результаты отражают то, как каталог LDAP был настроен с помощью
конфигурационного файла ([`tutorial.ldif`](https://docs.oracle.com/javase/tutorial/jndi/software/config/tutorial.ldif)),
прилагаемого к этому руководству. Если вы используете существующий сервер или сервер с другой
настройкой, то можете увидеть иные результаты. Прежде чем вы сможете загрузить конфигурационный файл
(`tutorial.ldif`) в сервер каталогов, необходимо выполнить инструкции по обновлению схемы сервера,
либо вы можете воспользоваться командой `ldapadd` или `ldapmodify`, если она доступна в вашей системе
UNIX.

Например, с помощью `ldapmodify` можно сделать следующее (подставив подходящие значения для имени узла,
DN администратора (опция `-D`) и пароля):

```text
ldapmodify -a -c -v -h hostname -p 389\
        -D "cn=Administrator, cn=users, dc=xxx, dc=xxx"\
        -w passwd -f tutorial.ldif
```

Структура файла `tutorial.ldif` начинается с корневой записи `o=JNDITutorial` и далее содержит
организационные единицы (organizational units) и записи групп. Фрагмент начала файла:

```text
dn: o=JNDITutorial
o: JNDITutorial
objectclass: top
objectclass: organization

dn: ou=Groups, o=JNDITutorial
ou: Groups
objectclass: top
objectclass: organizationalunit

dn: cn=Directory Administrators, ou=Groups, o=JNDITutorial
cn: Directory Administrators
objectclass: top
objectclass: groupofuniquenames
ou: Groups
uniquemember: cn=Rosanna Lee, ou=People, o=JNDITutorial
uniquemember: cn=Scott Seligman, ou=People, o=JNDITutorial
uniquemember: cn=Jon Ruiz, ou=People, o=JNDITutorial
uniquemember: cn=Vinnie Ryan, ou=People, o=JNDITutorial
```

### Примечание по установке: управление доступом (Access Control)

Разные серверы каталогов по-разному обрабатывают управление доступом (access control). Некоторые
примеры в этом руководстве выполняют обновления каталога. Кроме того, часть пространства имён, куда
вы установили материалы руководства, может иметь ограничения на чтение. Поэтому вам нужно выполнить
специфичные для сервера действия, чтобы сделать каталог доступным для чтения и/или обновления — иначе
эти примеры не будут работать. Для [Oracle Directory Server](http://www.oracle.com/technetwork/testcontent/index-085178.html)
добавьте запись `aci`, предложенную в файле
[`sunds.aci.ldif`](https://docs.oracle.com/javase/tutorial/jndi/software/config/sunds.aci.ldif),
к записи `dn: o=JNDITutorial`, чтобы сделать весь каталог доступным для чтения и обновления.
В качестве альтернативы вы можете изменить примеры так, чтобы они выполняли аутентификацию (authenticate)
в каталоге. Подробности о том, как это сделать, описаны в уроке «Безопасность» (Security).

### Примечание по установке: настройка пространства имён (Namespace Setup)

Записи в файле `tutorial.ldif` используют отличительное имя (distinguished name, DN) `o=JNDITutorial`
для корневого контекста имён (root naming context). Если вы не настроили свой сервер каталогов так,
чтобы `o=JNDITutorial` был корневым контекстом имён, то ваша попытка импортировать `tutorial.ldif`
завершится неудачей. Проще всего обойти эту проблему, добавив DN существующего корневого контекста имён
к каждой строке `dn:` в файле `tutorial.ldif`. Например, если на вашем сервере уже есть корневой
контекст имён `dc=imc,dc=org`, то вам следует изменить строку

```text
dn: o=JNDITutorial
```

на

```text
dn: o=JNDITutorial, dc=imc, dc=org
```

Внесите это изменение для каждой строки, начинающейся с `dn:` в файле. Затем во всех примерах этого
руководства везде, где используется `o=JNDITutorial`, используйте вместо неё `o=JNDITutorial,dc=imc,dc=org`.

### Примечание по установке: формат файла (File Format)

В зависимости от платформы операционной системы, которую вы используете, вам может потребоваться
отредактировать `tutorial.ldif`, чтобы он содержал корректные символы перевода строки (newline characters)
для этой платформы. Например, если вы обнаружили, что `tutorial.ldif` содержит символы перевода строки
в стиле Windows (CRLF), а импортируете этот файл в сервер каталогов, работающий на платформе UNIX, то
вам нужно отредактировать файл и заменить CRLF на LF. Симптом этой проблемы — сервер каталогов отвергает
все записи из `tutorial.ldif`.

### Примечание по установке: Windows Active Directory

1. Корневой контекст имён не будет `o=jnditutorial`. Он будет иметь вид `dc=x,dc=y,dc=z`. Вам нужно
   следовать предыдущему примечанию «Настройка пространства имён».
2. Добавьте объектные классы (object classes) и связанные с ними атрибуты для объектных классов
   `inetOrgPerson` и `groupOfUniqueNames` в схему Active Directory с помощью оснастки консоли управления
   Active Directory — ADSIEdit. `groupOfUniqueNames` определён в [RFC 2256](http://www.ietf.org/rfc/rfc2256.txt),
   а `inetOrgPerson` — в [RFC 2798](http://www.ietf.org/rfc/rfc2798.txt).
3. Некоторые иерархические отношения, используемые в руководстве, по умолчанию не допускаются в Active
   Directory. Чтобы включить эти отношения, добавьте их с помощью оснастки консоли управления Active
   Directory — ADSIEdit.

```text
objectclass: organizationalUnit
possible superiors: domainDNS
                    inetOrgPerson
                    organization
                    organizationalPerson
                    organizationalUnit
                    person
                    top

objectclass: groupOfUniqueNames
possible superiors: top

objectclass: inetOrgPerson
possible superiors: container
                    organizationalPerson
                    person
                    top
```

4. Удалите один из двух атрибутов `sn` из записи Mark Twain в `tutorial.ldif`. Active Directory
   определяет `sn` как однозначный (single-valued) атрибут, вопреки [RFC 2256](http://www.ietf.org/rfc/rfc2256.txt).
5. Используйте утилиту командной строки `ldifde`, чтобы загрузить изменённый файл `tutorial.ldif`.

```text
# ldifde -i -v -k -f tutorial.ldif
```

6. Большинство примеров предполагают, что каталог настроен для разрешения неаутентифицированного
   (unauthenticated) доступа на чтение и обновление. Ваша настройка Active Directory может этого не
   позволять. См. примечание по установке «Управление доступом».
7. Чтение записи иногда даёт больше атрибутов, чем показано в руководстве, потому что Active Directory
   часто возвращает некоторые внутренние атрибуты.
8. Создание записей может потребовать указания дополнительных специфичных для Active Directory атрибутов
   или использования других объектных классов.

## Настройка Java-приложения

Чтобы использовать JNDI в своей программе, нужно настроить среды её компиляции и выполнения.

### Импорт классов JNDI

Ниже перечислены пакеты JNDI:

- [`javax.naming`](https://docs.oracle.com/javase/8/docs/api/javax/naming/package-summary.html);
- [`javax.naming.directory`](https://docs.oracle.com/javase/8/docs/api/javax/naming/directory/package-summary.html);
- [`javax.naming.event`](https://docs.oracle.com/javase/8/docs/api/javax/naming/event/package-summary.html);
- [`javax.naming.ldap`](https://docs.oracle.com/javase/8/docs/api/javax/naming/ldap/package-summary.html);
- [`javax.naming.spi`](https://docs.oracle.com/javase/8/docs/api/javax/naming/spi/package-summary.html).

Примеры в этом трейле используют классы и интерфейсы из первых двух пакетов. Эти два пакета нужно
импортировать в свою программу или импортировать отдельные используемые вами классы и интерфейсы.
Следующие две строки импортируют все классы и интерфейсы из двух пакетов `javax.naming` и
`javax.naming.directory`.

```java
import javax.naming.*;
import javax.naming.directory.*;
```

### Среда компиляции

Чтобы скомпилировать программу, использующую JNDI, вам нужен доступ к классам JNDI. Java SE 6 уже
включает классы JNDI, поэтому, если вы используете её, никаких дополнительных действий не требуется.

### Среда выполнения

Чтобы запустить программу, использующую JNDI, вам нужен доступ к классам JNDI и классам всех
поставщиков услуг, которые использует программа. Среда выполнения Java (Java Runtime Environment, JRE) 6
уже включает классы JNDI и поставщиков услуг для LDAP, COS naming, реестра RMI и DNS.

Если вы используете каких-либо других поставщиков услуг, вам нужно загрузить и установить их
архивные файлы в каталог `JAVA_HOME/jre/lib/ext`, где `JAVA_HOME` — это каталог, содержащий JRE.
На странице JNDI перечислены некоторые поставщики услуг. Вы можете загрузить этих поставщиков или
использовать поставщиков от других производителей.
