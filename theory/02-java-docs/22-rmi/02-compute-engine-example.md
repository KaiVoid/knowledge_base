# Урок 2. Пример: вычислительный движок (Compute Engine)

**Трейл:** RMI · **Оригинал:** [Writing an RMI Server](https://docs.oracle.com/javase/tutorial/rmi/designing.html)
**Связанные области:** [[20-microservices]] · **Вопросы:** microservices

> Перевод официального руководства Oracle (The Java Tutorials, JDK 8).

Этот урок проходит сквозной пример RMI-приложения — **вычислительный движок** (compute engine).
Это удалённый объект на сервере, который принимает от клиентов задания (tasks), выполняет их и
возвращает результаты. Урок собран из четырёх страниц оригинала: проектирование удалённого
интерфейса, его реализация, написание клиента, а также сборка и запуск примера.

## Проектирование удалённого интерфейса (Designing a Remote Interface)

В основе вычислительного движка лежит протокол, который позволяет передавать задания движку,
выполнять их движком и возвращать результаты клиенту. Этот протокол выражается через интерфейсы,
поддерживаемые вычислительным движком.

Каждый интерфейс содержит один метод. Удалённый интерфейс движка — `Compute` — позволяет
передавать задания движку. Клиентский интерфейс — `Task` — определяет, как движок выполняет
переданное задание.

### Интерфейс `Compute`

Интерфейс `Compute` описывает удалённо доступную часть — сам вычислительный движок. Расширяя
интерфейс `java.rmi.Remote`, интерфейс `Compute` обозначает себя как интерфейс, методы которого
можно вызывать из другой виртуальной машины Java (Java virtual machine). Любой объект, реализующий
этот интерфейс, может быть удалённым объектом (remote object).

```java
package compute;

import java.rmi.Remote;
import java.rmi.RemoteException;

public interface Compute extends Remote {
    <T> T executeTask(Task<T> t) throws RemoteException;
}
```

Как член удалённого интерфейса, метод `executeTask` является удалённым методом (remote method).
Поэтому этот метод должен быть объявлен способным выбрасывать исключение
`java.rmi.RemoteException`. Это исключение выбрасывается системой RMI при вызове удалённого метода,
чтобы сообщить о сбое связи (communication failure) либо об ошибке протокола (protocol error).
`RemoteException` — проверяемое исключение (checked exception), поэтому любой код, вызывающий
удалённый метод, обязан обработать это исключение: либо перехватить его, либо объявить в своём
выражении `throws`.

### Интерфейс `Task`

Второй интерфейс, нужный вычислительному движку, — это `Task`, тип параметра метода `executeTask`
интерфейса `Compute`. Интерфейс `Task` определяет связь между движком и работой, которую тот должен
выполнить, предоставляя способ запустить эту работу.

```java
package compute;

public interface Task<T> {
    T execute();
}
```

Интерфейс `Task` определяет единственный метод `execute`, у которого нет параметров и который не
выбрасывает исключений. Поскольку интерфейс **не** расширяет `Remote`, метод этого интерфейса не
обязан перечислять `java.rmi.RemoteException` в выражении `throws`.

У интерфейса `Task` есть параметр типа (type parameter) `T`, который представляет тип результата
вычисления задания. Метод `execute` этого интерфейса возвращает результат вычисления, поэтому его
возвращаемый тип — `T`.

В свою очередь, метод `executeTask` интерфейса `Compute` возвращает результат выполнения экземпляра
`Task`, переданного ему. Таким образом, у метода `executeTask` есть собственный параметр типа `T`,
который связывает его возвращаемый тип с типом результата переданного экземпляра `Task`.

### Требования к сериализации

RMI использует механизм сериализации объектов Java (object serialization) для передачи объектов по
значению между виртуальными машинами Java. Чтобы объект считался сериализуемым, его класс должен
реализовывать интерфейс-маркер `java.io.Serializable`. Поэтому классы, реализующие интерфейс
`Task`, обязаны также реализовывать `Serializable`, как и классы объектов, используемых в качестве
результатов заданий.

### Динамическая загрузка классов и гибкость заданий

Объект `Compute` может выполнять разные виды заданий, лишь бы они были реализациями типа `Task`.
Классы, реализующие этот интерфейс, могут содержать любые данные, нужные для вычисления задания, и
любые другие методы, необходимые для вычисления.

Поскольку RMI может полагать, что объекты `Task` написаны на языке Java, реализации объекта `Task`,
ранее неизвестные вычислительному движку, загружаются (download) системой RMI в виртуальную машину
Java движка по мере надобности. Эта возможность позволяет клиентам движка определять новые виды
заданий для выполнения на серверной машине без явной установки кода на этой машине.

Вычислительный движок, реализованный классом `ComputeEngine`, реализует интерфейс `Compute`,
позволяя передавать ему различные задания вызовами его метода `executeTask`. Эти задания
выполняются с помощью их реализации метода `execute`, а результаты возвращаются удалённому клиенту.

## Реализация удалённого интерфейса (Implementing a Remote Interface)

В этом разделе обсуждается задача реализации класса вычислительного движка. В общем случае класс,
реализующий удалённый интерфейс, должен как минимум:

- объявить реализуемые удалённые интерфейсы;
- определить конструктор для каждого удалённого объекта;
- предоставить реализацию каждого удалённого метода удалённых интерфейсов.

Серверной программе RMI нужно создать начальные удалённые объекты и экспортировать (export) их в
среду выполнения RMI (RMI runtime), которая делает их доступными для приёма входящих удалённых
вызовов. Эта процедура настройки может быть инкапсулирована в методе самого класса-реализации
удалённого объекта либо вынесена в отдельный класс. Процедура настройки должна:

- создать и установить менеджер безопасности (security manager);
- создать и экспортировать один или несколько удалённых объектов;
- зарегистрировать хотя бы один удалённый объект в реестре RMI (RMI registry) — либо в другой
  службе именования, например доступной через Java Naming and Directory Interface, — для целей
  начальной загрузки (bootstrapping).

Полный код класса `ComputeEngine`:

```java
package engine;

import java.rmi.RemoteException;
import java.rmi.registry.LocateRegistry;
import java.rmi.registry.Registry;
import java.rmi.server.UnicastRemoteObject;
import compute.Compute;
import compute.Task;

public class ComputeEngine implements Compute {

    public ComputeEngine() {
        super();
    }

    public <T> T executeTask(Task<T> t) {
        return t.execute();
    }

    public static void main(String[] args) {
        if (System.getSecurityManager() == null) {
            System.setSecurityManager(new SecurityManager());
        }
        try {
            String name = "Compute";
            Compute engine = new ComputeEngine();
            Compute stub =
                (Compute) UnicastRemoteObject.exportObject(engine, 0);
            Registry registry = LocateRegistry.getRegistry();
            registry.rebind(name, stub);
            System.out.println("ComputeEngine bound");
        } catch (Exception e) {
            System.err.println("ComputeEngine exception:");
            e.printStackTrace();
        }
    }
}
```

### Объявление реализуемых удалённых интерфейсов

Класс-реализация вычислительного движка объявлен так:

```java
public class ComputeEngine implements Compute
```

Это объявление утверждает, что класс реализует удалённый интерфейс `Compute` и потому может быть
использован для удалённого объекта.

Класс `ComputeEngine` определяет класс-реализацию удалённого объекта, который реализует единственный
удалённый интерфейс и никаких других. Класс `ComputeEngine` также содержит два исполняемых элемента
программы, которые можно вызывать только локально. Первый из них — конструктор экземпляров
`ComputeEngine`. Второй — метод `main`, используемый для создания экземпляра `ComputeEngine` и
предоставления его клиентам.

### Определение конструктора удалённого объекта

У класса `ComputeEngine` есть единственный конструктор без аргументов. Хотя конструктор суперкласса
вызывается, даже если опущен в конструкторе `ComputeEngine`, он включён для наглядности.

### Реализация каждого удалённого метода

Класс удалённого объекта предоставляет реализации каждого удалённого метода, указанного в удалённых
интерфейсах. Интерфейс `Compute` содержит единственный удалённый метод `executeTask`, который
реализует протокол между удалённым объектом `ComputeEngine` и его клиентами. Каждый клиент передаёт
`ComputeEngine` объект `Task` с конкретной реализацией метода `execute` интерфейса `Task`. Движок
`ComputeEngine` выполняет задание каждого клиента и возвращает результат метода `execute` этого
задания непосредственно клиенту.

### Передача объектов в RMI

Аргументы и возвращаемые значения удалённых методов могут быть почти любого типа, включая локальные
объекты, удалённые объекты и примитивные типы данных. Точнее, любая сущность любого типа может быть
передана в удалённый метод или из него, если эта сущность — экземпляр типа, который является
примитивным типом данных, удалённым объектом или сериализуемым объектом, то есть реализует интерфейс
`java.io.Serializable`.

Некоторые типы объектов не удовлетворяют ни одному из этих критериев и потому не могут быть переданы
в удалённый метод или возвращены из него. Большинство таких объектов — например, потоки (threads)
или файловые дескрипторы — инкапсулируют информацию, имеющую смысл только в пределах одного
адресного пространства. Многие из базовых классов, включая классы пакетов `java.lang` и `java.util`,
реализуют интерфейс `Serializable`.

Правила, определяющие, как передаются аргументы и возвращаемые значения, таковы:

- **Удалённые объекты по сути передаются по ссылке** (by reference). Ссылка на удалённый объект — это
  заглушка (stub), то есть клиентский прокси (client-side proxy), который реализует полный набор
  удалённых интерфейсов, реализуемых удалённым объектом.
- **Локальные объекты передаются по копии** (by copy) с помощью сериализации объектов. По умолчанию
  копируются все поля, кроме помеченных `static` или `transient`. Поведение сериализации по умолчанию
  можно переопределить для каждого класса в отдельности.

Передача удалённого объекта по ссылке означает, что любые изменения состояния объекта удалёнными
вызовами методов отражаются в исходном удалённом объекте. Когда передаётся удалённый объект,
получателю доступны только те интерфейсы, которые являются удалёнными. Любые методы, определённые в
классе-реализации или в неудалённых интерфейсах, реализуемых классом, получателю недоступны.

Например, если бы вы передали ссылку на экземпляр класса `ComputeEngine`, получатель имел бы доступ
только к методу `executeTask` движка. Этот получатель не увидел бы ни конструктор `ComputeEngine`,
ни его метод `main`, ни его реализацию каких-либо методов `java.lang.Object`.

В параметрах и возвращаемых значениях удалённых вызовов объекты, не являющиеся удалёнными, передаются
по значению (by value). Таким образом, в принимающей виртуальной машине Java создаётся копия объекта.
Любые изменения состояния объекта получателем отражаются только в копии получателя, а не в исходном
экземпляре отправителя. Любые изменения состояния объекта отправителем отражаются только в исходном
экземпляре отправителя, а не в копии получателя.

### Реализация метода `main` сервера

Самый сложный метод реализации `ComputeEngine` — метод `main`. Он используется для запуска
`ComputeEngine` и потому должен выполнить необходимую инициализацию и подготовительную работу, чтобы
сервер был готов принимать вызовы клиентов. Это не удалённый метод, то есть его нельзя вызвать из
другой виртуальной машины Java. Поскольку метод `main` объявлен как `static`, он вообще не связан с
объектом, а связан с классом `ComputeEngine`.

#### Создание и установка менеджера безопасности

Первая задача метода `main` — создать и установить менеджер безопасности (security manager), который
защищает доступ к системным ресурсам от недоверенного загруженного кода, выполняющегося в виртуальной
машине Java. Менеджер безопасности определяет, имеет ли загруженный код доступ к локальной файловой
системе и может ли он выполнять любые другие привилегированные операции.

Если программа RMI не устанавливает менеджер безопасности, RMI не будет загружать классы (кроме как
из локального пути классов) для объектов, полученных в качестве аргументов или возвращаемых значений
удалённых вызовов. Это ограничение гарантирует, что операции, выполняемые загруженным кодом, подчинены
политике безопасности (security policy).

#### Предоставление удалённого объекта клиентам

Далее метод `main` создаёт экземпляр `ComputeEngine` и экспортирует его в среду выполнения RMI.
Статический метод `UnicastRemoteObject.exportObject` экспортирует переданный удалённый объект так,
чтобы он мог принимать вызовы своих удалённых методов от удалённых клиентов. Второй аргумент,
типа `int`, задаёт TCP-порт, на котором будут прослушиваться входящие запросы удалённых вызовов для
объекта. Обычно используется значение ноль, означающее использование анонимного порта (anonymous
port). Тогда фактический порт выбирается во время выполнения системой RMI или нижележащей
операционной системой. Однако можно использовать и ненулевое значение, чтобы указать конкретный порт
для прослушивания. После успешного возврата вызова `exportObject` удалённый объект `ComputeEngine`
готов обрабатывать входящие удалённые вызовы.

Метод `exportObject` возвращает заглушку (stub) для экспортированного удалённого объекта. Обратите
внимание: тип переменной `stub` должен быть `Compute`, а не `ComputeEngine`, потому что заглушка
удалённого объекта реализует только удалённые интерфейсы, реализуемые экспортируемым удалённым
объектом.

Метод `exportObject` объявляет, что может выбросить `RemoteException` — проверяемое исключение.
Метод `main` обрабатывает это исключение блоком `try/catch`. Если бы исключение не обрабатывалось
таким образом, `RemoteException` пришлось бы объявить в выражении `throws` метода `main`. Попытка
экспортировать удалённый объект может выбросить `RemoteException`, если недоступны нужные ресурсы
связи — например, если запрошенный порт занят для других целей.

Прежде чем клиент сможет вызвать метод удалённого объекта, он должен сначала получить ссылку на этот
удалённый объект. Получить ссылку можно так же, как получают любую другую ссылку на объект в
программе, — например, получив её как часть возвращаемого значения метода или как часть структуры
данных, содержащей такую ссылку.

Система предоставляет особый вид удалённого объекта — **реестр RMI** (RMI registry) — для поиска
ссылок на другие удалённые объекты. Реестр RMI — это простая служба именования удалённых объектов,
позволяющая клиентам получать ссылку на удалённый объект по имени. Обычно реестр используется лишь
для нахождения первого удалённого объекта, который нужен RMI-клиенту. Этот первый удалённый объект
затем может поддерживать поиск других объектов.

Удалённый интерфейс `java.rmi.registry.Registry` — это API для привязки (binding, или регистрации) и
поиска (looking up) удалённых объектов в реестре. Класс `java.rmi.registry.LocateRegistry`
предоставляет статические методы для синтеза удалённой ссылки на реестр по определённому сетевому
адресу (хост и порт). Эти методы создают объект удалённой ссылки, содержащий указанный сетевой адрес,
без выполнения какого-либо удалённого взаимодействия. `LocateRegistry` также предоставляет
статические методы для создания нового реестра в текущей виртуальной машине Java, хотя в этом примере
они не используются. После регистрации удалённого объекта в реестре RMI на локальном хосте клиенты с
любого хоста могут найти удалённый объект по имени, получить его ссылку и затем вызывать его удалённые
методы. Реестр может быть общим для всех серверов, работающих на хосте, либо отдельный серверный
процесс может создать и использовать собственный реестр.

Класс `ComputeEngine` создаёт имя для объекта и затем добавляет это имя в реестр RMI, работающий на
сервере. Этот шаг выполняется следующими операторами:

```java
Registry registry = LocateRegistry.getRegistry();
registry.rebind(name, stub);
```

Этот вызов `rebind` выполняет удалённый вызов реестра RMI на локальном хосте. Как и любой удалённый
вызов, он может привести к выбросу `RemoteException`, который обрабатывается блоком `catch` в конце
метода `main`.

Обратите внимание на следующее относительно вызова `Registry.rebind`:

- Перегрузка `LocateRegistry.getRegistry` без аргументов синтезирует ссылку на реестр на локальном
  хосте и на порту реестра по умолчанию — 1099. Если реестр создан на порту, отличном от 1099, нужно
  использовать перегрузку с параметром `int`.
- При выполнении удалённого вызова реестра вместо копии самого удалённого объекта передаётся его
  заглушка. Объекты-реализации удалённых объектов, например экземпляры `ComputeEngine`, никогда не
  покидают виртуальную машину Java, в которой были созданы. Поэтому, когда клиент выполняет поиск в
  реестре удалённых объектов сервера, возвращается копия заглушки. Таким образом, удалённые объекты в
  таких случаях фактически передаются по (удалённой) ссылке, а не по значению.
- По соображениям безопасности приложение может выполнять привязку (bind), отвязку (unbind) или
  повторную привязку (rebind) ссылок на удалённые объекты только в реестре, работающем на том же
  хосте. Это ограничение не позволяет удалённому клиенту удалять или перезаписывать записи в реестре
  сервера. Однако запрос на поиск (lookup) можно выполнять с любого хоста — локального или удалённого.

После того как сервер зарегистрировался в локальном реестре RMI, он печатает сообщение о готовности
обрабатывать вызовы. Затем метод `main` завершается. Не нужно держать поток, который бы поддерживал
сервер «живым». Пока существует ссылка на объект `ComputeEngine` в другой виртуальной машине Java —
локальной или удалённой, — объект `ComputeEngine` не будет остановлен или собран сборщиком мусора
(garbage collected). Поскольку программа привязывает ссылку на `ComputeEngine` в реестре, объект
достижим из удалённого клиента — самого реестра. Система RMI поддерживает процесс `ComputeEngine`
работающим. `ComputeEngine` доступен для приёма вызовов и не будет освобождён, пока его привязка не
будет удалена из реестра и пока ни один удалённый клиент не удерживает удалённую ссылку на объект
`ComputeEngine`.

Последний фрагмент кода в методе `ComputeEngine.main` обрабатывает любое возможное исключение.
Единственный проверяемый тип исключения, который может быть выброшен в этом коде, — `RemoteException`,
выбрасываемый либо вызовом `UnicastRemoteObject.exportObject`, либо вызовом `rebind` реестра. В любом
случае программа мало что может сделать, кроме как завершиться, напечатав сообщение об ошибке. В
некоторых распределённых приложениях восстановление после неудачного удалённого вызова возможно.
Например, приложение могло бы попытаться повторить операцию или выбрать другой сервер для её
продолжения.

## Создание клиентской программы (Creating a Client Program)

Вычислительный движок — относительно простая программа: он выполняет задания, которые ему передают.
Клиенты для движка сложнее. Клиенту нужно вызвать движок, но он также должен определить задание,
которое движок выполнит.

Клиент в нашем примере состоит из двух отдельных классов. Первый класс, `ComputePi`, находит и
вызывает объект `Compute`. Второй класс, `Pi`, реализует интерфейс `Task` и определяет работу,
которую должен выполнить движок. Задача класса `Pi` — вычислить значение числа π с некоторым числом
десятичных знаков.

Неудалённый интерфейс `Task` определён так:

```java
package compute;

public interface Task<T> {
    T execute();
}
```

Код, вызывающий методы объекта `Compute`, должен получить ссылку на этот объект, создать объект
`Task` и затем запросить выполнение задания. Определение класса задания `Pi` показано далее. Объект
`Pi` конструируется с единственным аргументом — желаемой точностью результата. Результат выполнения
задания — объект `java.math.BigDecimal`, представляющий π, вычисленное с указанной точностью.

### Клиентский класс `ComputePi`

```java
package client;

import java.rmi.registry.LocateRegistry;
import java.rmi.registry.Registry;
import java.math.BigDecimal;
import compute.Compute;

public class ComputePi {
    public static void main(String args[]) {
        if (System.getSecurityManager() == null) {
            System.setSecurityManager(new SecurityManager());
        }
        try {
            String name = "Compute";
            Registry registry = LocateRegistry.getRegistry(args[0]);
            Compute comp = (Compute) registry.lookup(name);
            Pi task = new Pi(Integer.parseInt(args[1]));
            BigDecimal pi = comp.executeTask(task);
            System.out.println(pi);
        } catch (Exception e) {
            System.err.println("ComputePi exception:");
            e.printStackTrace();
        }
    }
}
```

Как и сервер `ComputeEngine`, клиент начинает с установки менеджера безопасности. Этот шаг необходим,
потому что процесс получения заглушки удалённого объекта сервера может потребовать загрузки
определений классов с сервера. Чтобы RMI мог загружать классы, должен действовать менеджер
безопасности.

После установки менеджера безопасности клиент конструирует имя для поиска удалённого объекта
`Compute`, используя то же имя, под которым `ComputeEngine` привязал свой удалённый объект. Также
клиент использует API `LocateRegistry.getRegistry` для синтеза удалённой ссылки на реестр на хосте
сервера. Значение первого аргумента командной строки, `args[0]`, — это имя удалённого хоста, на
котором работает объект `Compute`. Затем клиент вызывает метод `lookup` реестра, чтобы найти
удалённый объект по имени в реестре хоста сервера. Использованная перегрузка
`LocateRegistry.getRegistry` — с единственным параметром `String` — возвращает ссылку на реестр на
указанном хосте и на порту реестра по умолчанию, 1099. Если реестр создан на порту, отличном от 1099,
нужно использовать перегрузку с параметром `int`.

Далее клиент создаёт новый объект `Pi`, передавая конструктору `Pi` значение второго аргумента
командной строки, `args[1]`, разобранное как целое число. Этот аргумент указывает число десятичных
знаков для вычисления. Наконец, клиент вызывает метод `executeTask` удалённого объекта `Compute`.
Этот вызов возвращает объект типа `BigDecimal`, который программа сохраняет в переменной `result`. В
завершение программа печатает результат.

### Класс `Pi`

Класс `Pi` реализует интерфейс `Task` и вычисляет значение π с заданным числом десятичных знаков. Для
этого примера сам алгоритм не важен. Важно то, что алгоритм вычислительно затратен (computationally
expensive), а значит, его желательно выполнять на достаточно мощном сервере.

```java
package client;

import compute.Task;
import java.io.Serializable;
import java.math.BigDecimal;

public class Pi implements Task<BigDecimal>, Serializable {

    private static final long serialVersionUID = 227L;

    /** константы, используемые при вычислении pi */
    private static final BigDecimal FOUR =
        BigDecimal.valueOf(4);

    /** режим округления, используемый при вычислении pi */
    private static final int roundingMode =
        BigDecimal.ROUND_HALF_EVEN;

    /** число значащих цифр после десятичной точки */
    private final int digits;

    /**
     * Создать задание для вычисления pi с указанной
     * точностью.
     */
    public Pi(int digits) {
        this.digits = digits;
    }

    /**
     * Вычислить pi.
     */
    public BigDecimal execute() {
        return computePi(digits);
    }

    /**
     * Вычислить значение pi с указанным числом цифр
     * после десятичной точки. Значение вычисляется
     * по формуле Мэчина (Machin's formula):
     *
     *          pi/4 = 4*arctan(1/5) - arctan(1/239)
     *
     * и разложению arctan(x) в степенной ряд с
     * достаточной точностью.
     */
    public static BigDecimal computePi(int digits) {
        int scale = digits + 5;
        BigDecimal arctan1_5 = arctan(5, scale);
        BigDecimal arctan1_239 = arctan(239, scale);
        BigDecimal pi = arctan1_5.multiply(FOUR).subtract(
                                  arctan1_239).multiply(FOUR);
        return pi.setScale(digits,
                           BigDecimal.ROUND_HALF_UP);
    }
    /**
     * Вычислить значение (в радианах) арктангенса
     * обратной величины переданного целого с указанным
     * числом цифр после десятичной точки. Значение
     * вычисляется по разложению арктангенса в
     * степенной ряд:
     *
     * arctan(x) = x - (x^3)/3 + (x^5)/5 - (x^7)/7 +
     *     (x^9)/9 ...
     */
    public static BigDecimal arctan(int inverseX,
                                    int scale)
    {
        BigDecimal result, numer, term;
        BigDecimal invX = BigDecimal.valueOf(inverseX);
        BigDecimal invX2 =
            BigDecimal.valueOf(inverseX * inverseX);

        numer = BigDecimal.ONE.divide(invX,
                                      scale, roundingMode);

        result = numer;
        int i = 1;
        do {
            numer =
                numer.divide(invX2, scale, roundingMode);
            int denom = 2 * i + 1;
            term =
                numer.divide(BigDecimal.valueOf(denom),
                             scale, roundingMode);
            if ((i % 2) != 0) {
                result = result.subtract(term);
            } else {
                result = result.add(term);
            }
            i++;
        } while (term.compareTo(BigDecimal.ZERO) != 0);
        return result;
    }
}
```

Обратите внимание: все сериализуемые классы — реализующие интерфейс `Serializable` прямо или
косвенно — должны объявлять `private static final` поле с именем `serialVersionUID`, чтобы
гарантировать совместимость сериализации между версиями. Если ни одна предыдущая версия класса не
выпускалась, значением этого поля может быть любое значение `long`, например `227L`, использованное в
`Pi`, — лишь бы это значение последовательно использовалось в будущих версиях. Если предыдущая версия
класса была выпущена без явного объявления `serialVersionUID`, но совместимость сериализации с этой
версией важна, то в качестве значения явного объявления новой версии нужно использовать вычисленное
по умолчанию значение для предыдущей версии. Чтобы определить значение, вычисленное по умолчанию для
предыдущей версии, можно запустить против неё инструмент `serialver`.

Класс `Pi` вычисляет значение π по формуле Мэчина (Machin's formula):

```
pi/4 = 4*arctan(1/5) - arctan(1/239)
```

и разложению arctan(x) в степенной ряд с достаточной точностью.

### Динамическая загрузка кода

Самая интересная особенность этого примера в том, что объекту-реализации `Compute` определение класса
`Pi` не нужно до тех пор, пока объект `Pi` не будет передан как аргумент методу `executeTask`. В этот
момент код класса загружается системой RMI в виртуальную машину Java объекта `Compute`, вызывается
метод `execute`, и код задания выполняется. Результат — в случае задания `Pi` это объект
`BigDecimal` — возвращается вызывающему клиенту, где используется для печати результата вычисления.

Тот факт, что переданный объект `Task` вычисляет значение π, не имеет значения для объекта
`ComputeEngine`. Вы могли бы также реализовать задание, которое, например, генерирует случайное
простое число с помощью вероятностного алгоритма. Такое задание тоже было бы вычислительно
интенсивным и потому хорошим кандидатом на передачу `ComputeEngine`, но потребовало бы совсем другого
кода. Этот код тоже мог бы быть загружен при передаче объекта `Task` объекту `Compute`. Точно так же,
как алгоритм вычисления π подгружается при необходимости, код, генерирующий случайное простое число,
был бы подгружен при необходимости. Объект `Compute` знает лишь, что каждый получаемый им объект
реализует метод `execute`. Объект `Compute` не знает — и ему не нужно знать, — что делает реализация.

## Сборка и запуск примера (Compiling and Running the Example)

### Замечание о безопасности

Серверная и клиентская программы работают с установленным менеджером безопасности. При запуске любой
из программ нужно указать файл политики безопасности (security policy file), чтобы коду были выданы
необходимые для работы разрешения.

#### Примеры файлов политики

Пример файла политики `server.policy` для серверной программы на Solaris или Linux:

```
grant codeBase "file:///home/ann/src/" {
    permission java.security.AllPermission;
};
```

Пример файла политики `client.policy` для клиентской программы на Solaris или Linux:

```
grant codeBase "file:///home/jones/src/" {
    permission java.security.AllPermission;
};
```

Пример файла `server.policy` для Windows:

```
grant codeBase "file://C:/home/ann/src/" {
    permission java.security.AllPermission;
};
```

Пример файла `client.policy` для Windows:

```
grant codeBase "file://C:/home/jones/src/" {
    permission java.security.AllPermission;
};
```

**Примечание.** Точный смысл значения `codeBase` зависит от символов в конце. `codeBase` с
завершающим `/` соответствует всем файлам классов (`.class`, но не JAR-файлам) в указанном каталоге.
`codeBase` с завершающим `/*` соответствует всем файлам (и класс-файлам, и JAR-файлам) в этом
каталоге. `codeBase` с завершающим `/-` соответствует всем файлам (и класс-файлам, и JAR-файлам) в
каталоге и рекурсивно всем файлам в подкаталогах, содержащихся в этом каталоге.

В этих примерах файлов политики все разрешения выданы классам в локальном пути классов программы,
потому что локальный код приложения считается доверенным, но коду, загруженному из других мест,
никакие разрешения не выдаются. Поэтому сервер вычислительного движка ограничивает выполняемые им
задания (чей код не известен как доверенный и может быть враждебным) от выполнения любых операций,
требующих разрешений безопасности. Заданию `Pi` из примера клиента для выполнения никакие разрешения
не требуются.

### Запуск сервера

Перед запуском вычислительного движка нужно запустить реестр RMI. Реестр RMI — это простая серверная
служба именования для начальной загрузки (bootstrap), позволяющая удалённым клиентам получить ссылку
на начальный удалённый объект. Его можно запустить командой `rmiregistry`. Прежде чем выполнять
`rmiregistry`, убедитесь, что в оболочке или окне, где вы будете запускать `rmiregistry`, либо вообще
не установлена переменная окружения `CLASSPATH`, либо переменная `CLASSPATH` не включает путь к
классам, которые вы хотите загружать клиентам ваших удалённых объектов.

Чтобы запустить реестр на сервере, выполните команду `rmiregistry`. Эта команда не выводит ничего и
обычно запускается в фоне. В этом примере реестр запускается на хосте `mycomputer`.

**Microsoft Windows:**

```bash
javaw rmiregistry -J-Djava.rmi.server.codebase=file://C:/home/ann/public_html/classes/
```

**Solaris или Linux:**

```bash
rmiregistry -J-Djava.rmi.server.codebase=file:///home/ann/public_html/classes/ &
```

Как показывают эти примеры, при запуске реестра на сервере нужно указать, где классы сервера доступны
по сети, с помощью свойства `java.rmi.server.codebase`. Можно также использовать HTTP-URL:

```bash
javaw rmiregistry -J-Djava.rmi.server.codebase=http://mycomputer.example.com/~ann/classes/
```

По умолчанию реестр работает на порту 1099. Чтобы запустить реестр на другом порту, укажите номер
порта в командной строке. Не забудьте сбросить (unset) переменную окружения `CLASSPATH`.

**Microsoft Windows:**

```bash
javaw rmiregistry -J-Djava.rmi.server.codebase=file://C:/home/ann/public_html/classes/ 2001
```

**Solaris или Linux:**

```bash
rmiregistry -J-Djava.rmi.server.codebase=file:///home/ann/public_html/classes/ 2001 &
```

После запуска реестра можно запускать сервер. Убедитесь, что и файл `compute.jar`, и класс-реализация
удалённого объекта находятся в вашем пути классов. При запуске вычислительного движка нужно с помощью
свойства `java.rmi.server.codebase` указать, где классы сервера доступны по сети. В этом примере
серверные классы, которые делаются доступными для загрузки, — это интерфейсы `Compute` и `Task`,
доступные в файле `compute.jar` в каталоге `public_html\classes` пользователя `ann`. Сервер
вычислительного движка запускается на хосте `mycomputer` — том же хосте, на котором запущен реестр.

**Microsoft Windows:**

```bash
java -cp C:\home\ann\src;C:\home\ann\public_html\classes\compute.jar ^
     -Djava.rmi.server.codebase=file://C:/home/ann/public_html/classes/ ^
     -Djava.rmi.server.hostname=mycomputer.example.com ^
     -Djava.security.policy=server.policy ^
        engine.ComputeEngine
```

**Solaris или Linux:**

```bash
java -cp /home/ann/src:/home/ann/public_html/classes/compute.jar \
     -Djava.rmi.server.codebase=file:///home/ann/public_html/classes/ \
     -Djava.rmi.server.hostname=mycomputer.example.com \
     -Djava.security.policy=server.policy \
        engine.ComputeEngine
```

Приведённая команда `java` определяет следующие системные свойства:

- Свойство `java.rmi.server.codebase` задаёт расположение (codebase URL), откуда можно загружать
  определения классов, происходящих от этого сервера. Если codebase указывает на иерархию каталогов
  (а не на JAR-файл), нужно добавить завершающий слеш в конец codebase URL.
- Свойство `java.rmi.server.hostname` задаёт имя хоста или адрес, который помещается в заглушки
  удалённых объектов, экспортированных в этой виртуальной машине Java. Это значение — имя хоста или
  адрес, используемый клиентами при попытке выполнить удалённые вызовы методов. По умолчанию
  реализация RMI использует IP-адрес сервера, как его указывает API
  `java.net.InetAddress.getLocalHost`. Однако иногда этот адрес подходит не всем клиентам, и
  полностью квалифицированное имя хоста было бы эффективнее. Чтобы RMI использовал для сервера имя
  хоста (или IP-адрес), маршрутизируемое от всех потенциальных клиентов, задайте свойство
  `java.rmi.server.hostname`.
- Свойство `java.security.policy` используется для указания файла политики, содержащего разрешения,
  которые вы намерены выдать.

### Запуск клиента

Когда реестр и вычислительный движок запущены, можно запустить клиент, указав следующее:

- расположение, откуда клиент предоставляет свои классы (класс `Pi`), с помощью свойства
  `java.rmi.server.codebase`;
- свойство `java.security.policy`, используемое для указания файла политики безопасности, содержащего
  разрешения, которые вы намерены выдать различным частям кода;
- в качестве аргументов командной строки — имя хоста сервера (чтобы клиент знал, где найти удалённый
  объект `Compute`) и число десятичных знаков для вычисления π.

Запустите клиент на другом хосте (например, на хосте с именем `mysecondcomputer`) так:

**Microsoft Windows:**

```bash
java -cp C:\home\jones\src;C:\home\jones\public_html\classes\compute.jar ^
     -Djava.security.policy=client.policy ^
        client.ComputePi mycomputer.example.com 45
```

**Solaris или Linux:**


### Как классы получаются во время выполнения программы

Когда сервер `ComputeEngine` привязывает ссылку на свой удалённый объект в реестре, реестр загружает
интерфейсы `Compute` и `Task`, от которых зависит класс заглушки. Эти классы загружаются либо с
веб-сервера, либо из файловой системы сервера `ComputeEngine` — в зависимости от типа codebase URL,
использованного при запуске сервера.

Поскольку у клиента `ComputePi` есть оба интерфейса — `Compute` и `Task` — в его пути классов, он
загружает их определения из своего пути классов, а не из codebase сервера.

Наконец, класс `Pi` загружается в виртуальную машину Java сервера `ComputeEngine`, когда объект `Pi`
передаётся в удалённом вызове `executeTask` объекту `ComputeEngine`. Класс `Pi` загружается сервером
либо с веб-сервера, либо из файловой системы клиента — в зависимости от типа codebase URL,
использованного при запуске клиента.
