# Урок 3. Генерация и проверка цифровых подписей

**Трейл:** Security · **Оригинал:** [Generating and Verifying Signatures](https://docs.oracle.com/javase/tutorial/security/apisign/gensig.html)
**Связанные области:** [[23-security]] · **Вопросы:** security

> Перевод официального руководства Oracle (The Java Tutorials, JDK 8).

> Этот урок проведёт вас через шаги, необходимые для использования API безопасности JDK
> (JDK Security API) для генерации цифровой подписи (digital signature) данных и для проверки
> того, что подпись подлинная. Урок предназначен для разработчиков, которые хотят встроить в
> свои программы функциональность безопасности, в том числе криптографические сервисы.
>
> Этот урок демонстрирует использование API безопасности JDK применительно к подписыванию
> документов. Урок показывает, что должна сделать одна программа, выполняемая человеком,
> у которого есть исходный документ, чтобы сгенерировать ключи (keys), сгенерировать цифровую
> подпись документа с помощью закрытого ключа (private key) и экспортировать открытый ключ
> (public key) и подпись в файлы.
>
> Затем показывается пример другой программы, выполняемой получателем документа, подписи и
> открытого ключа. Показывается, как программа может импортировать открытый ключ и проверить
> подлинность подписи. В уроке также обсуждаются и демонстрируются возможные альтернативные
> подходы и методы передачи и импорта ключей, в том числе в составе сертификатов (certificates).

> Дополнительные сведения о понятиях и терминологии (цифровые подписи, сертификаты, хранилища
> ключей (keystores)) см. в уроке «API and Tools Use for Secure Code and File Exchanges».

> В этом уроке вы создадите два простых приложения: одно для генерации цифровой подписи, другое —
> для её проверки. Далее следует обсуждение и демонстрация возможных улучшений. Урок состоит из
> трёх разделов.
>
> - **Генерация цифровой подписи** показывает использование API для генерации ключей и цифровой
>   подписи данных с помощью закрытого ключа и для экспорта открытого ключа и подписи в файлы.
>   Приложение получает имя файла с данными из командной строки.
>
> - **Проверка цифровой подписи** показывает использование API для импорта открытого ключа и
>   подписи, которая предположительно является подписью заданного файла данных, и для проверки
>   подлинности подписи. Имена файлов данных, открытого ключа и подписи задаются в командной строке.
>
> - **Слабые места и альтернативы** обсуждает потенциальные слабые места подхода, использованного
>   в базовых программах. Затем представляются и демонстрируются возможные альтернативные подходы и
>   методы передачи и импорта ключей, включая использование файлов с закодированными байтами ключа
>   и использование сертификатов, содержащих открытые ключи.

## Генерация цифровой подписи (Generating a Digital Signature)

> Программа `GenSig`, которую вы вот-вот создадите, будет использовать API безопасности JDK для
> генерации ключей и цифровой подписи данных с помощью закрытого ключа, а также для экспорта
> открытого ключа и подписи в файлы. Приложение получает имя файла данных из командной строки.

> Создание программы-примера `GenSig` состоит из следующих шагов:
>
> 1. **Подготовка начальной структуры программы** — создать текстовый файл `GenSig.java`. Ввести
>    начальную структуру программы (инструкции импорта, имя класса, метод `main` и т. д.).
> 2. **Генерация открытого и закрытого ключей** — сгенерировать пару ключей (key pair) (открытый и
>    закрытый ключи). Закрытый ключ нужен для подписывания данных. Открытый ключ будет использован
>    программой `VerSig` для проверки подписи.
> 3. **Подписывание данных** — получить объект `Signature` и инициализировать его для подписывания.
>    Передать ему данные для подписывания и сгенерировать подпись.
> 4. **Сохранение подписи и открытого ключа в файлы** — сохранить байты подписи в один файл, а
>    байты открытого ключа — в другой.
> 5. **Компиляция и запуск программы.**

### Подготовка начальной структуры программы (Prepare Initial Program Structure)

> Вот базовая структура программы `GenSig`. Поместите её в файл с именем `GenSig.java`.

```java
import java.io.*;
import java.security.*;

class GenSig {

    public static void main(String[] args) {

        /* Генерируем подпись DSA */

        if (args.length != 1) {
            System.out.println("Usage: GenSig nameOfFileToSign");
        }
        else try {

        // остальной код располагается здесь

        } catch (Exception e) {
            System.err.println("Caught exception " + e.toString());
        }
    }
}
```

> **Примечания:**
>
> - Методы для подписывания данных находятся в пакете `java.security`, поэтому программа
>   импортирует всё из этого пакета. Программа также импортирует пакет `java.io`, который содержит
>   методы, нужные для ввода данных файла, подлежащих подписыванию.
> - Ожидается единственный аргумент, задающий файл данных для подписывания.
> - Код, написанный на последующих шагах, будет располагаться между блоками `try` и `catch`.

### Генерация открытого и закрытого ключей (Generate Public and Private Keys)

> Чтобы иметь возможность создать цифровую подпись, вам нужен закрытый ключ. (Соответствующий ему
> открытый ключ понадобится для проверки подлинности подписи.)

> В некоторых случаях пара ключей (*key pair*) (закрытый ключ и соответствующий открытый ключ) уже
> доступна в файлах. В этом случае программа может импортировать и использовать закрытый ключ для
> подписывания, как показано в разделе «Слабые места и альтернативы».

> В других случаях программе нужно сгенерировать пару ключей. Пара ключей генерируется с помощью
> класса `KeyPairGenerator`.

> В этом примере вы сгенерируете пару открытый/закрытый ключ для алгоритма цифровой подписи
> (Digital Signature Algorithm, DSA). Вы сгенерируете ключи длиной 1024 бита.

> Генерация пары ключей требует нескольких шагов:

#### Создание генератора пары ключей (Create a Key Pair Generator)

> Первый шаг — получить объект-генератор пары ключей для генерации ключей для алгоритма подписи DSA.

> Как и для всех движковых классов (engine classes), способ получить объект `KeyPairGenerator` для
> определённого типа алгоритма — вызвать статический фабричный метод `getInstance` у класса
> `KeyPairGenerator`. Этот метод имеет две формы, обе из которых имеют первый аргумент
> `String algorithm`; у одной из форм также есть второй аргумент `String provider`.

> Таким образом, вызывающая сторона может опционально указать имя провайдера, что гарантирует, что
> реализация запрошенного алгоритма берётся именно из указанного провайдера. Код-пример этого урока
> всегда указывает провайдера SUN по умолчанию, встроенного в JDK.

> Поместите следующую инструкцию после строки
>
> `else try {`
>
> в файле, созданном на предыдущем шаге, «Подготовка начальной структуры программы»:

```java
KeyPairGenerator keyGen = KeyPairGenerator.getInstance("DSA", "SUN");
```

#### Инициализация генератора пары ключей (Initialize the Key Pair Generator)

> Следующий шаг — инициализировать генератор пары ключей. Все генераторы пар ключей разделяют
> понятия размера ключа (keysize) и источника случайности (source of randomness). У класса
> `KeyPairGenerator` есть метод `initialize`, принимающий эти два типа аргументов.

> Размер ключа для генератора ключей DSA — это длина ключа (в битах), которую вы установите равной
> 1024.

> Источником случайности должен быть экземпляр класса `SecureRandom`, который предоставляет
> криптографически стойкий генератор случайных чисел (random number generator, RNG). Дополнительные
> сведения о `SecureRandom` см. в спецификации API `SecureRandom` и в «Java Cryptography
> Architecture Reference Guide».

> В следующем примере запрашивается экземпляр `SecureRandom`, использующий алгоритм SHA1PRNG, как
> его предоставляет встроенный провайдер SUN. Затем пример передаёт этот экземпляр `SecureRandom`
> методу инициализации генератора пары ключей.

```java
SecureRandom random = SecureRandom.getInstance("SHA1PRNG", "SUN");
keyGen.initialize(1024, random);
```

> Некоторые ситуации требуют стойких случайных значений — например, при создании ценных и
> долгоживущих секретов, таких как открытые и закрытые ключи RSA. Чтобы помочь приложениям выбрать
> подходящую стойкую реализацию `SecureRandom`, начиная с JDK 8 дистрибутивы Java включают список
> известных стойких реализаций `SecureRandom` в свойстве `securerandom.strongAlgorithms` класса
> `java.security.Security`. Когда вы создаёте подобные данные, вам следует рассмотреть
> использование `SecureRandom.getInstanceStrong()`, так как этот метод получает экземпляр одного из
> известных стойких алгоритмов.

#### Генерация пары ключей (Generate the Pair of Keys)

> Финальный шаг — сгенерировать пару ключей и сохранить ключи в объектах `PrivateKey` и `PublicKey`.

```java
KeyPair pair = keyGen.generateKeyPair();
PrivateKey priv = pair.getPrivate();
PublicKey pub = pair.getPublic();
```

### Подписывание данных (Sign the Data)

> Теперь, когда вы создали открытый и закрытый ключи, вы готовы подписать данные. В этом примере вы
> подпишете данные, содержащиеся в файле. `GenSig` получает имя файла из командной строки. Цифровая
> подпись создаётся (или проверяется) с помощью экземпляра класса `Signature`.

> Подписывание данных, то есть генерация цифровой подписи для этих данных, выполняется следующими
> шагами.

> **Получение объекта `Signature`**: следующий код получает объект `Signature` для генерации или
> проверки подписей с помощью алгоритма DSA — того же алгоритма, для которого программа
> сгенерировала ключи на предыдущем шаге, «Генерация открытого и закрытого ключей».

```java
Signature dsa = Signature.getInstance("SHA1withDSA", "SUN");
```

> Примечание: указывая имя алгоритма подписи, вам следует также включать имя алгоритма выработки
> дайджеста сообщения (message digest algorithm), используемого алгоритмом подписи. `SHA1withDSA` —
> это способ указать алгоритм подписи DSA, использующий алгоритм дайджеста сообщения SHA-1.

> **Инициализация объекта `Signature`**
>
> Прежде чем объект `Signature` можно будет использовать для подписывания или проверки, его нужно
> инициализировать. Метод инициализации для подписывания требует закрытого ключа. Используйте
> закрытый ключ, помещённый в объект `PrivateKey` с именем `priv` на предыдущем шаге.

```java
dsa.initSign(priv);
```

> **Передача объекту `Signature` данных для подписывания**: эта программа использует данные из
> файла, имя которого задано в качестве первого (и единственного) аргумента командной строки.
> Программа читает данные по одному буферу за раз и передаёт их объекту `Signature`, вызывая метод
> `update`.

```java
FileInputStream fis = new FileInputStream(args[0]);
BufferedInputStream bufin = new BufferedInputStream(fis);
byte[] buffer = new byte[1024];
int len;
while ((len = bufin.read(buffer)) >= 0) {
    dsa.update(buffer, 0, len);
};
bufin.close();
```

> **Генерация подписи**
>
> После того как все данные переданы объекту `Signature`, вы можете сгенерировать цифровую подпись
> этих данных.

```java
byte[] realSig = dsa.sign();
```

### Сохранение подписи и открытого ключа в файлы (Save the Signature and the Public Key in Files)

> Теперь, когда вы сгенерировали подпись для некоторых данных, вам нужно сохранить байты подписи в
> один файл, а байты открытого ключа — в другой, чтобы вы могли отправить (по модему, на дискете,
> по почте и т. д.) кому-то другому
>
> - данные, для которых была сгенерирована подпись,
> - подпись и
> - открытый ключ.

> Получатель может проверить, что данные пришли от вас и не были изменены при передаче, запустив
> программу `VerSig`, которую вы сгенерируете на предстоящих шагах раздела «Проверка цифровой
> подписи». Эта программа использует открытый ключ, чтобы проверить, что полученная подпись является
> истинной подписью полученных данных.

> Вспомните, что подпись была помещена в байтовый массив с именем `realSig`. Вы можете сохранить
> байты подписи в файл с именем `sig` следующим образом.

```java
/* сохраняем подпись в файл */
FileOutputStream sigfos = new FileOutputStream("sig");
sigfos.write(realSig);
sigfos.close();
```

> Вспомните из шага «Генерация открытого и закрытого ключей», что открытый ключ был помещён в объект
> `PublicKey` с именем `pub`. Вы можете получить закодированные байты ключа, вызвав метод
> `getEncoded`, а затем сохранить закодированные байты в файл. Файл можно назвать как угодно. Если,
> например, вас зовут Сьюзан (Susan), вы можете назвать его как-нибудь вроде `suepk` (от «Sue's
> public key» — «открытый ключ Сью»), как показано ниже:

```java
/* сохраняем открытый ключ в файл */
byte[] key = pub.getEncoded();
FileOutputStream keyfos = new FileOutputStream("suepk");
keyfos.write(key);
keyfos.close();
```

### Компиляция и запуск программы (Compile and Run the Program)

> Вот полный исходный код программы `GenSig.java` с добавленными комментариями. Скомпилируйте и
> запустите его. Помните, вам нужно указать имя файла, подлежащего подписыванию, например:

```
java GenSig data
```

> Вы можете скачать и использовать этот пример файла с именем `data` или любой другой файл по вашему
> желанию. Файл не будет изменён. Он будет прочитан, чтобы для него можно было сгенерировать подпись.

> После выполнения программы вы должны увидеть сохранённые файлы `suepk` (открытый ключ) и `sig`
> (подпись).

## Проверка цифровой подписи (Verifying a Digital Signature)

> Если у вас есть данные, для которых была сгенерирована цифровая подпись, вы можете проверить
> подлинность подписи. Для этого вам нужны
>
> - данные,
> - подпись,
> - открытый ключ, соответствующий закрытому ключу, которым были подписаны данные.

> В этом примере вы напишете программу `VerSig` для проверки подписи, сгенерированной программой
> `GenSig`. Это демонстрирует шаги, необходимые для проверки подлинности предполагаемой подписи.

> `VerSig` импортирует открытый ключ и подпись, которая предположительно является подписью заданного
> файла данных, а затем проверяет подлинность подписи. Имена файлов открытого ключа, подписи и
> данных задаются в командной строке.

> Шаги создания программы-примера `VerSig`, которая импортирует файлы и проверяет подпись, следующие.
>
> 1. **Подготовка начальной структуры программы** — создать текстовый файл `VerSig.java`. Ввести
>    начальную структуру программы (инструкции импорта, имя класса, метод `main` и т. д.).
> 2. **Ввод и преобразование закодированных байтов открытого ключа** — импортировать закодированные
>    байты открытого ключа из файла, заданного первым аргументом командной строки, и преобразовать
>    их в `PublicKey`.
> 3. **Ввод байтов подписи** — ввести байты подписи из файла, заданного вторым аргументом командной
>    строки.
> 4. **Проверка подписи** — получить объект `Signature` и инициализировать его открытым ключом для
>    проверки подписи. Передать ему данные, чью подпись нужно проверить (из файла, заданного третьим
>    аргументом командной строки), и проверить подпись.
> 5. **Компиляция и запуск программы.**

### Подготовка начальной структуры программы (Prepare Initial Program Structure)

> Вот базовая структура программы `VerSig`, создаваемой в следующих частях этого урока. Поместите эту
> структуру программы в файл с именем `VerSig.java`.

```java
import java.io.*;
import java.security.*;
import java.security.spec.*;

class VerSig {

    public static void main(String[] args) {

        /* Проверяем подпись DSA */

        if (args.length != 3) {
            System.out.println("Usage: VerSig " +
                "publickeyfile signaturefile " + "datafile");
        }
        else try {

        // остальной код располагается здесь

        } catch (Exception e) {
            System.err.println("Caught exception " + e.toString());
        }
    }

}
```

> **Примечания:**
>
> - Методы для проверки данных находятся в пакете `java.security`, поэтому программа импортирует всё
>   из этого пакета. Программа также импортирует пакет `java.io` ради методов, нужных для ввода
>   данных файла, подлежащих подписыванию, а также пакет `java.security.spec`, содержащий класс
>   `X509EncodedKeySpec`.
> - Ожидаются три аргумента, задающие файлы открытого ключа, подписи и данных.
> - Код, написанный на последующих шагах этого урока, будет располагаться между блоками `try` и
>   `catch`.

### Ввод и преобразование закодированных байтов открытого ключа (Input and Convert the Encoded Public Key Bytes)

> Далее `VerSig` нужно импортировать закодированные байты открытого ключа из файла, заданного первым
> аргументом командной строки, и преобразовать их в `PublicKey`. `PublicKey` необходим, потому что
> именно его требует метод `initVerify` класса `Signature`, чтобы инициализировать объект
> `Signature` для проверки.

> Сначала прочитайте закодированные байты открытого ключа.

```java
FileInputStream keyfis = new FileInputStream(args[0]);
byte[] encKey = new byte[keyfis.available()];
keyfis.read(encKey);

keyfis.close();
```

> Теперь байтовый массив `encKey` содержит закодированные байты открытого ключа.

> Вы можете использовать класс `KeyFactory`, чтобы создать открытый ключ DSA из его кодировки. Класс
> `KeyFactory` обеспечивает преобразования между непрозрачными ключами (opaque keys) (типа `Key`) и
> спецификациями ключей (key specifications) — прозрачными представлениями исходного ключевого
> материала. С непрозрачным ключом вы можете получить имя алгоритма, имя формата и закодированные
> байты ключа, но не сам ключевой материал, который, например, может состоять из самого ключа и
> параметров алгоритма, использованных для вычисления ключа. (Обратите внимание, что `PublicKey`,
> поскольку он расширяет `Key`, сам является `Key`.)

> Итак, сначала вам нужна спецификация ключа. Вы можете получить её следующим образом, предполагая,
> что ключ был закодирован по стандарту X.509, как, например, в случае, если ключ был сгенерирован
> встроенным генератором пар ключей DSA, поставляемым провайдером SUN:

```java
X509EncodedKeySpec pubKeySpec = new X509EncodedKeySpec(encKey);
```

> Теперь вам нужен объект `KeyFactory` для выполнения преобразования. Этот объект должен быть таким,
> который работает с ключами DSA.

```java
KeyFactory keyFactory = KeyFactory.getInstance("DSA", "SUN");
```

> Наконец, вы можете использовать объект `KeyFactory`, чтобы сгенерировать `PublicKey` из
> спецификации ключа.

```java
PublicKey pubKey =
    keyFactory.generatePublic(pubKeySpec);
```

### Ввод байтов подписи (Input the Signature Bytes)

> Далее введите байты подписи из файла, заданного вторым аргументом командной строки.

```java
FileInputStream sigfis = new FileInputStream(args[1]);
byte[] sigToVerify = new byte[sigfis.available()];
sigfis.read(sigToVerify);
sigfis.close();
```

> Теперь байтовый массив `sigToVerify` содержит байты подписи.

### Проверка подписи (Verify the Signature)

> Вы добавили в программу `VerSig` код, чтобы
>
> - ввести закодированные байты ключа и преобразовать их в `PublicKey` с именем `pubKey`,
> - ввести байты подписи в байтовый массив с именем `sigToVerify`.
>
> Теперь вы можете перейти к самой проверке.

#### Инициализация объекта `Signature` для проверки (Initialize the Signature Object for Verification)

> Как и при генерации подписи, подпись проверяется с помощью экземпляра класса `Signature`. Вам нужно
> создать объект `Signature`, использующий тот же алгоритм подписи, что был использован для генерации
> подписи. Алгоритм, использованный программой `GenSig`, — это алгоритм SHA1withDSA от провайдера SUN.

```java
Signature sig = Signature.getInstance("SHA1withDSA", "SUN");
```

> Далее вам нужно инициализировать объект `Signature`. Метод инициализации для проверки требует
> открытого ключа.

```java
sig.initVerify(pubKey);
```

#### Передача объекту `Signature` данных для проверки (Supply the Signature Object With the Data to be Verified)

> Теперь вам нужно передать объекту `Signature` данные, для которых была сгенерирована подпись. Эти
> данные находятся в файле, имя которого было задано третьим аргументом командной строки. Как и при
> подписывании, читайте данные по одному буферу за раз и передавайте их объекту `Signature`, вызывая
> метод `update`.

```java
FileInputStream datafis = new FileInputStream(args[2]);
BufferedInputStream bufin = new BufferedInputStream(datafis);

byte[] buffer = new byte[1024];
int len;
while (bufin.available() != 0) {
    len = bufin.read(buffer);
    sig.update(buffer, 0, len);
};

bufin.close();
```

#### Проверка подписи (Verify the Signature)

> После того как вы передали все данные объекту `Signature`, вы можете проверить цифровую подпись
> этих данных и сообщить результат. Вспомните, что предполагаемая подпись была прочитана в байтовый
> массив с именем `sigToVerify`.

```java
boolean verifies = sig.verify(sigToVerify);

System.out.println("signature verifies: " + verifies);
```

> Значение `verifies` будет `true`, если предполагаемая подпись (`sigToVerify`) является настоящей
> подписью заданного файла данных, сгенерированной закрытым ключом, соответствующим открытому ключу
> `pubKey`.

### Компиляция и запуск программы (Compile and Run the Program)

> [Здесь](https://docs.oracle.com/javase/tutorial/security/apisign/examples/VerSig.java) находится
> полный исходный код программы `VerSig.java` с добавленными комментариями.

> Скомпилируйте и запустите программу. Помните, вам нужно указать три аргумента в командной строке:
>
> - имя файла, содержащего закодированные байты открытого ключа;
> - имя файла, содержащего байты подписи;
> - имя файла данных (того, для которого была сгенерирована подпись).

> Поскольку вы будете тестировать вывод программы `GenSig`, имена файлов, которые вам следует
> использовать, такие:
>
> - `suepk`
> - `sig`
> - `data`

> Вот пример запуска; жирным выделено то, что вы вводите.

```
%java VerSig suepk sig data
signature verifies: true
```

## Слабые места и альтернативы (Weaknesses and Alternatives)

> Программы `GenSig` и `VerSig` из этого урока иллюстрируют использование API безопасности JDK для
> генерации цифровой подписи данных и для проверки того, что подпись подлинная. Однако сам сценарий,
> изображённый этими программами, — в котором отправитель использует API безопасности JDK для
> генерации новой пары открытый/закрытый ключ, сохраняет закодированные байты открытого ключа в файл,
> а получатель считывает байты ключа, — не обязательно реалистичен и имеет потенциальный серьёзный
> изъян.

> Во многих случаях ключи не нужно генерировать; они уже существуют — либо как закодированные ключи
> в файлах, либо как записи в хранилище ключей (keystore).

> Потенциальный серьёзный изъян в том, что ничто не гарантирует подлинность открытого ключа, который
> получает получатель, а программа `VerSig` корректно проверяет подлинность подписи только в том
> случае, если переданный ей открытый ключ *сам по себе* подлинный!

### Работа с закодированными байтами ключа (Working with Encoded Key Bytes)

> Иногда закодированные байты ключа уже существуют в файлах для пары ключей, которую нужно
> использовать для подписывания и проверки. Если это так, программа `GenSig` может импортировать
> закодированные байты закрытого ключа и преобразовать их в `PrivateKey`, нужный для подписывания,
> следующим образом, предполагая, что имя файла, содержащего байты закрытого ключа, находится в
> строке `privkeyfile` типа `String`, и что байты представляют ключ DSA, закодированный по стандарту
> PKCS #8.

```java
FileInputStream keyfis = new FileInputStream(privkeyfile);
byte[] encKey = new byte[keyfis.available()];
keyfis.read(encKey);
keyfis.close();

PKCS8EncodedKeySpec privKeySpec = new PKCS8EncodedKeySpec(encKey);

KeyFactory keyFactory = KeyFactory.getInstance("DSA");
PrivateKey privKey = keyFactory.generatePrivate(privKeySpec);
```

> `GenSig` больше не нужно сохранять байты открытого ключа в файл, так как они уже находятся в файле.

> В этом случае отправитель отправляет получателю
>
> - уже существующий файл, содержащий закодированные байты открытого ключа (если у получателя его
>   ещё нет), и
> - файл данных и файл подписи, экспортированные программой `GenSig`.

> Программа `VerSig` остаётся неизменной, поскольку она и так ожидает закодированные байты открытого
> ключа в файле.

> Но как насчёт потенциальной проблемы, когда злоумышленник перехватывает файлы и заменяет их все
> так, что подмену нельзя обнаружить? В некоторых случаях это не проблема, потому что люди уже
> обменялись открытыми ключами лицом к лицу или через доверенную третью сторону, которая выполняет
> обмен лицом к лицу. После этого множество последующих обменов файлами и подписями могут
> выполняться удалённо (то есть между двумя людьми в разных местах), а открытые ключи могут
> использоваться для проверки их подлинности. Если злоумышленник попытается изменить данные или
> подпись, это будет обнаружено программой `VerSig`.

> Если обмен ключами лицом к лицу невозможен, вы можете попробовать другие методы повышения
> вероятности корректного получения. Например, вы можете отправить свой открытый ключ наиболее
> безопасным из возможных способов до последующих обменов файлами данных и подписей, возможно,
> используя менее безопасные средства.

> В целом отправка данных и подписи отдельно от вашего открытого ключа значительно снижает
> вероятность атаки. Если только все три файла не изменены, причём определённым образом,
> обсуждаемым в следующем абзаце, `VerSig` обнаружит любое искажение.

> Если все три файла (документ с данными, открытый ключ и подпись) были перехвачены
> злоумышленником, этот человек мог бы заменить документ чем-то иным, подписать его закрытым ключом
> и переслать вам заменённый документ, новую подпись и открытый ключ, соответствующий закрытому
> ключу, использованному для генерации новой подписи. Тогда `VerSig` сообщит об успешной проверке, и
> вы будете думать, что документ пришёл от исходного отправителя. Поэтому вам следует предпринять
> шаги, чтобы хотя бы открытый ключ был получен в целости (`VerSig` обнаруживает любое искажение
> остальных файлов), либо вы можете использовать сертификаты, чтобы облегчить аутентификацию
> открытого ключа, как описано в следующем разделе.

### Работа с сертификатами (Working with Certificates)

> В криптографии чаще обмениваются *сертификатами* (certificates), содержащими открытые ключи, чем
> самими ключами.

> Одно из преимуществ в том, что сертификат подписан одной стороной (издателем (issuer)), чтобы
> подтвердить, что вложенный открытый ключ действительно является открытым ключом другой стороны
> (субъекта (subject) или владельца (owner)). Обычно доверенный сторонний удостоверяющий центр
> (certification authority, CA) проверяет личность субъекта, а затем удостоверяет, что субъект
> является владельцем открытого ключа, подписывая сертификат.

> Ещё одно преимущество использования сертификатов в том, что вы можете проверить действительность
> полученного сертификата, проверив его цифровую подпись с помощью открытого ключа его издателя
> (подписавшего), который сам может храниться в сертификате, чью подпись можно проверить с помощью
> открытого ключа издателя того сертификата; этот открытый ключ сам может храниться в сертификате, и
> так далее, пока вы не достигнете открытого ключа, которому вы уже доверяете.

> Если вы не можете установить цепочку доверия (trust chain) (например, потому что необходимые
> сертификаты издателей вам недоступны), можно вычислить **отпечаток(ки) (fingerprint)** сертификата.
> Каждый отпечаток — это относительно короткое число, которое уникально и надёжно идентифицирует
> сертификат. (Технически это хеш-значение информации сертификата, использующее дайджест сообщения,
> также известный как односторонняя хеш-функция (one-way hash function).) Вы можете позвонить
> владельцу сертификата и сравнить отпечатки полученного вами сертификата с отправленными. Если они
> одинаковы, то сертификаты одинаковы.

> Было бы безопаснее, чтобы `GenSig` создавала сертификат, содержащий открытый ключ, а `VerSig`
> затем импортировала сертификат и извлекала открытый ключ. Однако в JDK нет публичных API для
> сертификатов, которые позволили бы вам создать сертификат из открытого ключа, поэтому программа
> `GenSig` не может создать сертификат из сгенерированного ею открытого ключа. (Тем не менее
> публичные API для извлечения открытого ключа из сертификата *есть*.)

> При желании вы можете использовать различные инструменты безопасности (security tools), а не API,
> чтобы подписать ваши важные документы и работать с сертификатами из хранилища ключей, как было
> сделано в уроке «Exchanging Files».

> Как вариант, вы можете использовать API, чтобы изменить ваши программы для работы с уже
> существующим закрытым ключом и соответствующим открытым ключом (в сертификате) из вашего хранилища
> ключей. Для начала измените программу `GenSig`, чтобы она извлекала закрытый ключ из хранилища
> ключей, а не генерировала новые ключи. Сперва предположим следующее:
>
> - имя хранилища ключей находится в строке `ksName` типа `String`;
> - тип хранилища ключей — «JKS», проприетарный тип от Oracle;
> - пароль хранилища ключей находится в массиве `char` `spass`;
> - псевдоним (alias) записи хранилища ключей, содержащей закрытый ключ и сертификат открытого
>   ключа, находится в строке `alias` типа `String`;
> - пароль закрытого ключа находится в массиве `char` `kpass`.

> Тогда вы можете извлечь закрытый ключ из хранилища ключей следующим образом.

```java
KeyStore ks = KeyStore.getInstance("JKS");
FileInputStream ksfis = new FileInputStream(ksName);
BufferedInputStream ksbufin = new BufferedInputStream(ksfis);

ks.load(ksbufin, spass);
PrivateKey priv = (PrivateKey) ks.getKey(alias, kpass);
```

> Вы можете извлечь сертификат открытого ключа из хранилища ключей и сохранить его закодированные
> байты в файл с именем `suecert` следующим образом.

```java
java.security.cert.Certificate cert = ks.getCertificate(alias);
byte[] encodedCert = cert.getEncoded();

// Сохраняем сертификат в файл с именем "suecert"

FileOutputStream certfos = new FileOutputStream("suecert");
certfos.write(encodedCert);
certfos.close();
```

> Затем вы отправляете получателю файл данных, подпись и сертификат. Получатель проверяет подлинность
> сертификата, сначала получив отпечатки сертификата с помощью команды `keytool -printcert`.

```
keytool -printcert -file suecert
Owner: CN=Susan Jones, OU=Purchasing, O=ABC, L=Cupertino, ST=CA, C=US
Issuer: CN=Susan Jones, OU=Purchasing, O=ABC, L=Cupertino, ST=CA, C=US
Serial number: 35aaed17
Valid from: Mon Jul 13 22:31:03 PDT 1998 until:
Sun Oct 11 22:31:03 PDT 1998
Certificate fingerprints:
MD5:  1E:B8:04:59:86:7A:78:6B:40:AC:64:89:2C:0F:DD:13
SHA1: 1C:79:BD:26:A1:34:C0:0A:30:63:11:6A:F2:B9:67:DF:E5:8D:7B:5E
```

> Затем получатель проверяет отпечатки, возможно, позвонив отправителю и сравнив их с отпечатками
> сертификата отправителя или посмотрев их в публичном репозитории.

> Программа проверки получателя (модифицированная `VerSig`) затем может импортировать сертификат и
> извлечь из него открытый ключ следующим образом, предполагая, что имя файла сертификата (например,
> `suecert`) находится в строке `certName` типа `String`.

```java
FileInputStream certfis = new FileInputStream(certName);
java.security.cert.CertificateFactory cf =
    java.security.cert.CertificateFactory.getInstance("X.509");
java.security.cert.Certificate cert =  cf.generateCertificate(certfis);
PublicKey pub = cert.getPublicKey();
```

### Обеспечение конфиденциальности данных (Ensuring Data Confidentiality)

> Предположим, вы хотите сохранить содержимое данных конфиденциальным, чтобы люди, случайно или
> злонамеренно пытающиеся просмотреть их при передаче (или на вашей собственной машине или диске), не
> могли этого сделать. Чтобы сохранить данные конфиденциальными, вам следует зашифровать их и
> сохранять и отправлять только результат шифрования (называемый *шифртекстом (ciphertext)*).
> Получатель может расшифровать шифртекст, чтобы получить копию исходных данных.
