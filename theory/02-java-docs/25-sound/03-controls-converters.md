# Урок 3. Обработка аудио элементами управления и конвертеры форматов

**Трейл:** Sound · **Оригинал:** [Processing Audio with Controls](https://docs.oracle.com/javase/tutorial/sound/controls.html)
**Связанные области:** [[01-core-java-syntax-oop]] · **Вопросы:** core-java

> Перевод официального руководства Oracle (The Java Tutorials, JDK 8). Объединяет страницы
> *Processing Audio with Controls* и *Using Files and Format Converters* трейла *Sound*.

В предыдущих разделах обсуждалось, как воспроизводить или захватывать аудиосэмплы. Неявной целью
было доставить сэмплы как можно более точно, без изменений (за исключением, возможно, смешивания
сэмплов с сэмплами из других аудиолиний). Однако иногда нужно иметь возможность изменять сигнал.
Пользователь может захотеть, чтобы звук был громче, тише, полнее, более реверберирующим, выше или
ниже по высоте тона и так далее. Этот раздел рассматривает возможности Java Sound API, которые
обеспечивают подобную обработку сигнала.

Применять обработку сигнала можно двумя способами:

* Использовать любую обработку, поддерживаемую микшером (*mixer*) или его компонентами-линиями,
  запрашивая объекты [`Control`](https://docs.oracle.com/javase/8/docs/api/javax/sound/sampled/Control.html)
  и затем настраивая элементы управления (*controls*) так, как захочет пользователь. Типичные
  элементы управления, поддерживаемые микшерами и линиями, включают регуляторы усиления (*gain*),
  панорамы (*pan*) и реверберации (*reverberation*).
* Если нужный вид обработки не предоставляется микшером или его линиями, ваша программа может
  работать напрямую с аудиобайтами, манипулируя ими как угодно.

Первый способ рассматривается ниже подробнее, поскольку для второго специального API нет.

## Введение в элементы управления (Controls)

Микшер может иметь различные виды элементов управления обработкой сигнала на некоторых или всех
своих линиях. Например, микшер, используемый для захвата аудио, может иметь входной порт с
регулятором усиления, а целевые линии данных (*target data lines*) — с регуляторами усиления и
панорамы. Микшер, используемый для воспроизведения аудио, может иметь регуляторы частоты
дискретизации (*sample-rate*) на своих исходных линиях данных (*source data lines*). В каждом
случае элементы управления доступны через методы интерфейса `Line`.

Поскольку интерфейс `Mixer` расширяет `Line`, сам микшер может иметь собственный набор элементов
управления. Они могут выступать в роли мастер-регуляторов, влияющих на все исходные или целевые
линии микшера. Например, у микшера может быть мастер-регулятор усиления, значение которого в
децибелах добавляется к значениям отдельных регуляторов усиления на его целевых линиях.

Другие собственные элементы управления микшера могут влиять на особую линию — ни исходную, ни
целевую, — которую микшер использует внутренне для своей обработки. Например, глобальный регулятор
реверберации может выбирать вид реверберации, применяемый к смеси входных сигналов, и этот «влажный»
(реверберированный) сигнал затем подмешивался бы обратно в «сухой» сигнал перед доставкой на целевые
линии микшера.

Если у микшера или у любой из его линий есть элементы управления, вы можете захотеть отобразить их
через графические объекты в пользовательском интерфейсе вашей программы, чтобы пользователь мог
настраивать аудиохарактеристики по своему усмотрению. Сами элементы управления не являются
графическими; они лишь позволяют получать и изменять свои настройки. Какой вид графического
представления (ползунки, кнопки и т. д.) использовать — и использовать ли вообще — решаете вы.

Все элементы управления реализованы как конкретные подклассы абстрактного класса `Control`. Многие
типичные элементы управления обработкой аудио могут быть описаны абстрактными подклассами `Control`
на основе типа данных (например, логического, перечислимого или с плавающей точкой). Логические
элементы управления, например, представляют элементы с двумя состояниями, такие как переключатели
вкл/выкл для отключения звука (*mute*) или реверберации. Элементы управления с плавающей точкой,
напротив, хорошо подходят для представления непрерывно изменяемых регуляторов, таких как панорама,
баланс или громкость.

Java Sound API определяет следующие абстрактные подклассы `Control`:

* [`BooleanControl`](https://docs.oracle.com/javase/8/docs/api/javax/sound/sampled/BooleanControl.html)
  — представляет элемент управления с двумя состояниями (`true` или `false`). Например, отключение
  звука (*mute*), соло (*solo*) и переключатели вкл/выкл — хорошие кандидаты на роль `BooleanControl`.
* [`FloatControl`](https://docs.oracle.com/javase/8/docs/api/javax/sound/sampled/FloatControl.html)
  — модель данных, обеспечивающая управление диапазоном значений с плавающей точкой. Например,
  громкость и панорама — это `FloatControl`, которыми можно управлять через регулятор или ползунок.
* [`EnumControl`](https://docs.oracle.com/javase/8/docs/api/javax/sound/sampled/EnumControl.html)
  — предлагает выбор из набора объектов. Например, вы можете связать набор кнопок в пользовательском
  интерфейсе с `EnumControl`, чтобы выбирать одну из нескольких предустановленных настроек
  реверберации.
* [`CompoundControl`](https://docs.oracle.com/javase/8/docs/api/javax/sound/sampled/CompoundControl.html)
  — предоставляет доступ к коллекции связанных элементов, каждый из которых сам является экземпляром
  подкласса `Control`. `CompoundControl` представляет составные модули с несколькими элементами
  управления, такие как графические эквалайзеры. (Графический эквалайзер обычно изображается набором
  ползунков, каждый из которых влияет на отдельный `FloatControl`.)

У каждого подкласса `Control`, перечисленного выше, есть методы, соответствующие его базовому типу
данных. Большинство классов включают методы, которые устанавливают и получают текущее значение
(значения) элемента управления, получают метку (метки) элемента управления и так далее.

Разумеется, у каждого класса есть методы, специфичные именно для него и для модели данных, которую
он представляет. Например, `EnumControl` имеет метод, позволяющий получить набор его возможных
значений, а `FloatControl` позволяет получить его минимальное и максимальное значения, а также
точность (шаг приращения) элемента управления.

У каждого подкласса `Control` есть соответствующий подкласс `Control.Type`, который включает
статические экземпляры, идентифицирующие конкретные элементы управления.

В следующей таблице показан каждый подкласс `Control`, соответствующий ему подкласс `Control.Type` и
статические экземпляры, обозначающие конкретные виды элементов управления:

| `Control` | `Control.Type` | Экземпляры `Control.Type` |
|-----------|----------------|---------------------------|
| `BooleanControl` | `BooleanControl.Type` | `MUTE` — состояние отключения звука линии; `APPLY_REVERB` — реверберация вкл/выкл |
| `CompoundControl` | `CompoundControl.Type` | (нет) |
| `EnumControl` | `EnumControl.Type` | `REVERB` — доступ к настройкам реверберации (каждая — экземпляр `ReverbType`) |
| `FloatControl` | `FloatControl.Type` | `AUX_RETURN` — усиление возврата вспомогательной шины на линии; `AUX_SEND` — усиление посыла на вспомогательную шину; `BALANCE` — баланс громкости лево-право; `MASTER_GAIN` — общее усиление на линии; `PAN` — позиция лево-право; `REVERB_RETURN` — усиление сигнала после реверберации; `REVERB_SEND` — усиление сигнала до реверберации; `SAMPLE_RATE` — частота дискретизации при воспроизведении; `VOLUME` — громкость на линии |

Реализация Java Sound API может предоставлять любые или все эти типы элементов управления на своих
микшерах и линиях. Она также может предоставлять дополнительные типы элементов управления, не
определённые в Java Sound API. Такие типы могут быть реализованы через конкретные подклассы любого
из этих четырёх абстрактных подклассов либо через дополнительные подклассы `Control`, не наследующие
ни от одного из этих четырёх абстрактных подклассов. Прикладная программа может опросить каждую
линию, чтобы узнать, какие элементы управления она поддерживает.

## Получение линии с нужными элементами управления

Во многих случаях прикладная программа просто отображает те элементы управления, которые
оказываются поддерживаемыми данной линией. Если у линии нет элементов управления — пусть так. Но что
если важно найти линию с определёнными элементами управления? В этом случае можно использовать
`Line.Info`, чтобы получить линию с нужными характеристиками, как ранее описывалось в разделе
*Getting a Line of a Desired Type*.

Например, предположим, вы предпочитаете входной порт, позволяющий пользователю настраивать громкость
звукового входа. Следующий фрагмент кода показывает, как можно опросить микшер по умолчанию, чтобы
определить, есть ли у него нужный порт и элемент управления:

```java
Port lineIn;
FloatControl volCtrl;
try {
  mixer = AudioSystem.getMixer(null);
  lineIn = (Port)mixer.getLine(Port.Info.LINE_IN);
  lineIn.open();
  volCtrl = (FloatControl) lineIn.getControl(
      FloatControl.Type.VOLUME);

  // Если вызов getControl успешен,
  // у нас теперь есть регулятор VOLUME для LINE_IN.
} catch (Exception e) {
  System.out.println("Failed trying to find LINE_IN"
    + " VOLUME control: exception = " + e);
}
if (volCtrl != null)
  // ...
```

## Получение элементов управления от линии

Прикладной программе, которой нужно выставить элементы управления в своём пользовательском
интерфейсе, может быть достаточно просто опросить доступные линии и элементы управления, а затем
отобразить подходящий элемент пользовательского интерфейса для каждого элемента управления на каждой
интересующей линии. В таком случае единственная задача программы — предоставить пользователю «ручки»
для управления, а не знать, что эти элементы делают с аудиосигналом. Пока программа знает, как
отобразить элементы управления линии в элементы пользовательского интерфейса, архитектура Java Sound
API из `Mixer`, `Line` и `Control` в общем случае позаботится об остальном.

Например, предположим, ваша программа воспроизводит звук. Вы используете `SourceDataLine`, которую
получили, как описывалось ранее в разделе *Getting a Line of a Desired Type*. Вы можете получить
доступ к элементам управления линии, вызвав следующий метод `Line`:

```java
Control[] getControls()
```

Затем для каждого из элементов управления в возвращённом массиве вы используете следующий метод
`Control`, чтобы получить тип элемента управления:

```java
Control.Type getType()
```

Зная конкретный экземпляр `Control.Type`, ваша программа может отобразить соответствующий элемент
пользовательского интерфейса. Разумеется, выбор «соответствующего элемента пользовательского
интерфейса» для конкретного `Control.Type` зависит от подхода, принятого в вашей программе. С одной
стороны, вы можете использовать один и тот же вид элемента для представления всех экземпляров
`Control.Type` одного класса. Это потребовало бы запроса *класса* экземпляра `Control.Type`,
используя, например, метод `Object.getClass`. Допустим, результат совпал с `BooleanControl.Type`. В
этом случае ваша программа могла бы отобразить обычный флажок или кнопку-переключатель, а если бы его
класс совпал с `FloatControl.Type`, то вы могли бы отобразить графический ползунок.

С другой стороны, ваша программа могла бы различать разные типы элементов управления — даже одного
класса — и использовать свой элемент пользовательского интерфейса для каждого. Это потребовало бы
проверки *экземпляра*, возвращённого методом `getType` класса `Control`. Тогда если, например, тип
совпал с `BooleanControl.Type.APPLY_REVERB`, ваша программа могла бы отобразить флажок; а если тип
совпал с `BooleanControl.Type.MUTE`, вы вместо этого могли бы отобразить кнопку-переключатель.

## Использование элемента управления для изменения аудиосигнала

Теперь, когда вы знаете, как получить доступ к элементу управления и определить его тип, этот раздел
опишет, как использовать `Control` для изменения свойств аудиосигнала. Раздел охватывает не все
доступные элементы управления; скорее, он приводит несколько примеров в этой области, чтобы показать,
с чего начать. Эти примеры включают:

* Управление состоянием отключения звука (*mute*) линии;
* Изменение громкости линии;
* Выбор среди различных предустановок реверберации.

Предположим, что ваша программа получила доступ ко всем своим микшерам, их линиям и элементам
управления на этих линиях, и что у неё есть структура данных для управления логическими связями
между элементами управления и соответствующими им элементами пользовательского интерфейса. Тогда
перевод манипуляций пользователя с этими элементами управления в соответствующие методы `Control`
становится довольно простым делом.

В следующих подразделах описаны некоторые методы, которые нужно вызвать, чтобы внести изменения в
конкретные элементы управления.

### Управление состоянием отключения звука (Mute) линии

Управление состоянием отключения звука любой линии — это просто вызов следующего метода
`BooleanControl`:

```java
void setValue(boolean value)
```

(Предположительно, программа знает, обращаясь к своим структурам данных управления элементами, что
отключение звука — это экземпляр `BooleanControl`.) Чтобы отключить сигнал, проходящий через линию,
программа вызывает приведённый выше метод, указывая `true` в качестве значения. Чтобы выключить
отключение звука, позволив сигналу проходить через линию, программа вызывает метод с параметром,
установленным в `false`.

### Изменение громкости линии

Допустим, ваша программа связывает определённый графический ползунок с регулятором громкости
конкретной линии. Значение регулятора громкости (то есть `FloatControl.Type.VOLUME`)
устанавливается с помощью следующего метода `FloatControl`:

```java
void setValue(float newValue)
```

Обнаружив, что пользователь передвинул ползунок, программа получает текущее значение ползунка и
передаёт его как параметр `newValue` в приведённый выше метод. Это изменяет громкость сигнала,
проходящего через линию, которой «принадлежит» этот элемент управления.

### Выбор среди различных предустановок реверберации

Предположим, что у нашей программы есть микшер с линией, у которой есть элемент управления типа
`EnumControl.Type.REVERB`. Вызов метода `EnumControl`:

```java
java.lang.Objects[] getValues()
```

на этом элементе управления возвращает массив объектов `ReverbType`. При желании к конкретным
настройкам параметров каждого из этих объектов можно получить доступ с помощью следующих методов
`ReverbType`:

```java
int getDecayTime()
int getEarlyReflectionDelay()
float getEarlyReflectionIntensity()
int getLateReflectionDelay()
float getLateReflectionIntensity()
```

Например, если программе нужна единственная настройка реверберации, звучащая как пещера, она может
перебрать объекты `ReverbType`, пока не найдёт тот, для которого `getDecayTime` возвращает значение
больше 2000. Подробное объяснение этих методов, включая таблицу типичных возвращаемых значений,
смотрите в справочной документации API по `javax.sound.sampled.ReverbType`.

Однако обычно программа создаёт элемент пользовательского интерфейса, например радиокнопку, для
каждого из объектов `ReverbType` внутри массива, возвращённого методом `getValues`. Когда
пользователь щёлкает по одной из этих радиокнопок, программа вызывает метод `EnumControl`:

```java
void setValue(java.lang.Object value)
```

где `value` устанавливается равным `ReverbType`, соответствующему вновь нажатой кнопке. Аудиосигнал,
отправляемый через линию, которой «принадлежит» этот `EnumControl`, тогда будет реверберирован
согласно настройкам параметров, составляющим текущий `ReverbType` элемента управления (то есть
конкретный `ReverbType`, указанный в аргументе `value` метода `setValue`).

Таким образом, с точки зрения нашей прикладной программы, дать пользователю возможность переходить от
одной предустановки реверберации (то есть `ReverbType`) к другой — это просто вопрос связывания
каждого элемента массива, возвращённого `getValues`, с отдельной радиокнопкой.

## Прямое манипулирование аудиоданными

API `Control` позволяет реализации Java Sound API или стороннему поставщику микшера предоставлять
произвольные виды обработки сигнала через элементы управления. Но что если ни один микшер не
предлагает нужный вам вид обработки сигнала? Потребуется больше работы, но вы, возможно, сумеете
реализовать обработку сигнала в своей программе. Поскольку Java Sound API даёт вам доступ к
аудиоданным в виде массива байтов, вы можете изменять эти байты любым выбранным способом.

Если вы обрабатываете входящий звук, вы можете прочитать байты из `TargetDataLine` и затем
манипулировать ими. Алгоритмически тривиальный пример, способный дать звуково интересные результаты,
— воспроизвести звук задом наперёд, расположив его кадры (*frames*) в обратном порядке. Этот
тривиальный пример может оказаться не особенно полезным для вашей программы, но существуют
многочисленные изощрённые методы цифровой обработки сигнала (*digital signal processing, DSP*),
которые могут оказаться более подходящими. Некоторые примеры — эквализация, компрессия динамического
диапазона, ограничение пиков и растяжение или сжатие во времени, а также спецэффекты, такие как
задержка (*delay*), хорус (*chorus*), флэнджер (*flanging*), искажение (*distortion*) и так далее.

Чтобы воспроизвести обработанный звук, вы можете поместить ваш изменённый массив байтов в
`SourceDataLine` или `Clip`. Разумеется, массив байтов не обязательно должен быть получен из
существующего звука. Вы можете синтезировать звуки с нуля, хотя это требует определённых знаний
акустики или доступа к функциям синтеза звука. Для обработки или синтеза вы можете обратиться к
учебнику по аудио-DSP за интересующими вас алгоритмами либо импортировать в свою программу стороннюю
библиотеку функций обработки сигнала. Для воспроизведения синтезированного звука рассмотрите,
удовлетворяет ли вашим потребностям вместо этого API `Synthesizer` из пакета `javax.sound.midi`. Об
`javax.sound.midi` вы узнаете больше позже в разделе *Synthesizing Sound*.

# Использование файлов и конвертеров форматов

Большинству прикладных программ, работающих со звуком, нужно читать звуковые файлы или аудиопотоки.
Это типичная функциональность, независимо от того, что программа впоследствии делает с прочитанными
данными (например, воспроизводит, смешивает или обрабатывает их). Аналогично, многим программам нужно
записывать звуковые файлы (или потоки). В некоторых случаях прочитанные (или подлежащие записи)
данные нужно преобразовать в другой формат.

Как кратко упоминалось в разделе *Accessing Audio System Resources*, Java Sound API предоставляет
разработчикам приложений различные средства для файлового ввода-вывода и преобразований форматов.
Прикладные программы могут читать, записывать и преобразовывать данные между различными форматами
звуковых файлов и форматами аудиоданных.

Раздел *Overview of the Sampled Package* представил основные классы, связанные со звуковыми файлами и
форматами аудиоданных. Для повторения:

- Поток аудиоданных, который может читаться из файла или записываться в него, представлен объектом
  `AudioInputStream`. (`AudioInputStream` наследует от `java.io.InputStream`.)
- Формат этих аудиоданных представлен объектом `AudioFormat`.

Этот формат определяет, как устроены сами аудиосэмплы, но не структуру файла, в котором они могут
храниться. Иными словами, `AudioFormat` описывает «сырые» аудиоданные, такие, какие система может
передать вашей программе после захвата их с микрофонного входа или после разбора их из звукового
файла. `AudioFormat` включает такую информацию, как кодировка (*encoding*), порядок байтов (*byte
order*), число каналов, частота дискретизации (*sampling rate*) и число бит на сэмпл.

- Существует несколько известных стандартных форматов звуковых файлов, таких как WAV, AIFF или AU.
  Разные типы звуковых файлов имеют разную структуру для хранения аудиоданных, а также для хранения
  описательной информации об аудиоданных. Формат звукового файла представлен в Java Sound API
  объектом `AudioFileFormat`. `AudioFileFormat` включает объект `AudioFormat` для описания формата
  аудиоданных, хранящихся в файле, а также информацию о типе файла и длине данных в файле.
- Класс `AudioSystem` предоставляет методы для (1) сохранения потока аудиоданных из
  `AudioInputStream` в звуковой файл определённого типа (иными словами, записи файла), (2) извлечения
  потока аудиобайтов (`AudioInputStream`) из звукового файла (иными словами, чтения файла) и (3)
  преобразования аудиоданных из одного формата данных в другой. Этот раздел, разделённый на три
  части, объясняет эти три вида деятельности.

**Примечание:** реализация Java Sound API не обязательно предоставляет всеобъемлющие средства для
чтения, записи и преобразования аудио в разных форматах данных и файлов. Она может поддерживать лишь
наиболее распространённые форматы данных и файлов. Однако поставщики служб (*service providers*) могут
разрабатывать и распространять службы преобразования, расширяющие этот набор, как вы позже увидите в
разделе *Providing Sampled-Audio Services*. Класс `AudioSystem` предоставляет методы, позволяющие
прикладным программам узнать, какие преобразования доступны, как описано далее в разделе
«Преобразование форматов файлов и данных».

## Чтение звуковых файлов

Класс `AudioSystem` предоставляет два типа служб чтения файлов:

- Информация о формате аудиоданных, хранящихся в звуковом файле;
- Поток форматированных аудиоданных, который можно прочитать из звукового файла.

Первая из них предоставляется тремя вариантами метода `getAudioFileFormat`:

```java
static AudioFileFormat getAudioFileFormat (java.io.File file)
static AudioFileFormat getAudioFileFormat(java.io.InputStream stream)
static AudioFileFormat getAudioFileFormat (java.net.URL url)
```

Как упоминалось выше, возвращённый объект `AudioFileFormat` сообщает вам тип файла, длину данных в
файле, кодировку, порядок байтов, число каналов, частоту дискретизации и число бит на сэмпл.

Второй тип функциональности чтения файлов предоставляется этими методами `AudioSystem`:

```java
static AudioInputStream getAudioInputStream (java.io.File file)
static AudioInputStream getAudioInputStream (java.net.URL url)
static AudioInputStream getAudioInputStream (java.io.InputStream stream)
```

Эти методы дают вам объект (`AudioInputStream`), который позволяет читать аудиоданные файла,
используя один из методов чтения `AudioInputStream`. Пример мы вскоре увидим.

Предположим, вы пишете приложение для редактирования звука, которое позволяет пользователю загружать
звуковые данные из файла, отображать соответствующую форму волны или спектрограмму, редактировать
звук, воспроизводить отредактированные данные и сохранять результат в новый файл. Или, возможно, ваша
программа будет читать данные, хранящиеся в файле, применять некоторую обработку сигнала (например,
алгоритм, замедляющий звук без изменения его высоты тона), а затем воспроизводить обработанное аудио.
В любом случае вам нужно получить доступ к данным, содержащимся в звуковом файле. Полагая, что ваша
программа предоставляет некоторое средство для пользователя, чтобы выбрать или указать входной
звуковой файл, чтение аудиоданных этого файла включает три шага:

1. Получить объект `AudioInputStream` из файла.
2. Создать массив байтов, в котором вы будете хранить последовательные порции данных из файла.
3. Многократно читать байты из аудиопотока в массив. На каждой итерации делать что-то полезное с
   байтами в массиве (например, можно воспроизводить их, фильтровать, анализировать, отображать или
   записывать в другой файл).

Следующий фрагмент кода обрисовывает эти шаги:

```java
int totalFramesRead = 0;
File fileIn = new File(somePathName);
// somePathName — заранее существующая строка, значение которой
// основано на выборе пользователя.
try {
  AudioInputStream audioInputStream =
    AudioSystem.getAudioInputStream(fileIn);
  int bytesPerFrame =
    audioInputStream.getFormat().getFrameSize();
    if (bytesPerFrame == AudioSystem.NOT_SPECIFIED) {
    // у некоторых аудиоформатов размер кадра может быть не указан,
    // в этом случае мы можем читать любое количество байтов
    bytesPerFrame = 1;
  }
  // Задаём произвольный размер буфера в 1024 кадра.
  int numBytes = 1024 * bytesPerFrame;
  byte[] audioBytes = new byte[numBytes];
  try {
    int numBytesRead = 0;
    int numFramesRead = 0;
    // Пытаемся прочитать numBytes байтов из файла.
    while ((numBytesRead =
      audioInputStream.read(audioBytes)) != -1) {
      // Вычисляем количество фактически прочитанных кадров.
      numFramesRead = numBytesRead / bytesPerFrame;
      totalFramesRead += numFramesRead;
      // Здесь сделать что-то полезное с аудиоданными, которые
      // теперь находятся в массиве audioBytes...
    }
  } catch (Exception ex) {
    // Обработать ошибку...
  }
} catch (Exception e) {
  // Обработать ошибку...
}
```

Давайте посмотрим, что происходит в приведённом выше примере кода. Сначала внешний блок `try`
создаёт экземпляр объекта `AudioInputStream` через вызов метода
`AudioSystem.getAudioInputStream(File)`. Этот метод прозрачно выполняет все проверки, необходимые
для определения того, действительно ли указанный файл является звуковым файлом типа, поддерживаемого
Java Sound API. Если проверяемый файл (`fileIn` в этом примере) не является звуковым файлом или
является звуковым файлом неподдерживаемого типа, выбрасывается исключение
`UnsupportedAudioFileException`. Это поведение удобно тем, что программисту приложения не нужно
утруждать себя проверкой атрибутов файла или соблюдением каких-либо соглашений об именовании файлов.
Вместо этого метод `getAudioInputStream` берёт на себя весь низкоуровневый разбор и проверку,
необходимые для валидации входного файла. Затем внешний блок `try` создаёт массив байтов,
`audioBytes`, произвольной фиксированной длины. Мы обеспечиваем, чтобы его длина в байтах равнялась
целому числу кадров, чтобы не прочитать в итоге лишь часть кадра или, что ещё хуже, лишь часть
сэмпла. Этот массив байтов послужит буфером для временного хранения порции аудиоданных по мере её
чтения из потока. Если бы мы знали, что будем читать только очень короткие звуковые файлы, мы могли
бы сделать этот массив той же длины, что и данные в файле, выведя длину в байтах из длины в кадрах,
возвращаемой методом `getFrameLength` класса `AudioInputStream`. (На самом деле мы, вероятно, просто
использовали бы вместо этого объект `Clip`.) Но чтобы избежать нехватки памяти в общем случае, мы
вместо этого читаем файл порциями, по одному буферу за раз.

Внутренний блок `try` содержит цикл `while`, где мы читаем аудиоданные из `AudioInputStream` в массив
байтов. Вам следует добавить в этот цикл код для обработки аудиоданных в этом массиве любым способом,
подходящим для нужд вашей программы. Если вы применяете какой-либо вид обработки сигнала к данным,
вам, вероятно, потребуется дополнительно запросить `AudioFormat` у `AudioInputStream`, чтобы узнать
число бит на сэмпл и так далее.

Обратите внимание, что метод `AudioInputStream.read(byte[])` возвращает число прочитанных *байтов* —
а не число сэмплов или кадров. Этот метод возвращает -1, когда больше нет данных для чтения. Обнаружив
это условие, мы выходим из цикла `while`.

## Запись звуковых файлов

В предыдущем разделе были описаны основы чтения звукового файла с использованием конкретных методов
классов `AudioSystem` и `AudioInputStream`. Этот раздел описывает, как записать аудиоданные в новый
файл.

Следующий метод `AudioSystem` создаёт дисковый файл указанного типа. Файл будет содержать
аудиоданные, находящиеся в указанном `AudioInputStream`:

```java
static int write(AudioInputStream in,
  AudioFileFormat.Type fileType, File out)
```

Обратите внимание, что второй аргумент должен быть одним из типов файлов, поддерживаемых системой
(например, AU, AIFF или WAV), иначе метод `write` выбросит `IllegalArgumentException`. Чтобы избежать
этого, вы можете проверить, можно ли записать конкретный `AudioInputStream` в файл конкретного типа,
вызвав этот метод `AudioSystem`:

```java
static boolean isFileTypeSupported
  (AudioFileFormat.Type fileType, AudioInputStream stream)
```

который вернёт `true`, только если конкретная комбинация поддерживается.

В более общем виде вы можете узнать, какие типы файлов система может записывать, вызвав один из этих
методов `AudioSystem`:

```java
static AudioFileFormat.Type[] getAudioFileTypes()
static AudioFileFormat.Type[] getAudioFileTypes(AudioInputStream stream)
```

Первый из них возвращает все типы файлов, которые система может записывать, а второй возвращает только
те, которые система может записать из данного аудиопотока.

Следующий фрагмент демонстрирует один из приёмов создания выходного файла из `AudioInputStream` с
использованием метода `write`, упомянутого выше.

```java
File fileOut = new File(someNewPathName);
AudioFileFormat.Type fileType = fileFormat.getType();
if (AudioSystem.isFileTypeSupported(fileType,
    audioInputStream)) {
  AudioSystem.write(audioInputStream, fileType, fileOut);
}
```

Первый оператор выше создаёт новый объект `File`, `fileOut`, с указанным пользователем или программой
путём. Второй оператор получает тип файла из заранее существующего объекта `AudioFileFormat` по имени
`fileFormat`, который мог быть получен из другого звукового файла, например того, что был прочитан в
разделе «Чтение звуковых файлов» выше. (Вместо этого вы могли бы предоставить любой поддерживаемый
тип файла, который захотите, вместо получения типа файла откуда-то ещё. Например, вы могли бы удалить
второй оператор и заменить два других вхождения `fileType` в коде выше на
`AudioFileFormat.Type.WAVE`.)

Третий оператор проверяет, можно ли записать файл указанного типа из желаемого `AudioInputStream`.
Как и формат файла, этот поток мог быть получен из ранее прочитанного звукового файла. (Если так, то,
предположительно, вы каким-то образом обработали или изменили его данные, потому что иначе существуют
более простые способы просто скопировать файл.) Или, возможно, поток содержит байты, только что
захваченные с микрофонного входа.

Наконец, поток, тип файла и выходной файл передаются методу `write` класса `AudioSystem`, чтобы
достичь цели записи файла.

## Преобразование форматов файлов и данных

Вспомните из раздела *What is Formatted Audio Data?*, что Java Sound API различает форматы аудио*файлов*
и форматы аудио*данных*. Эти две вещи более или менее независимы. Грубо говоря, формат данных
относится к способу, которым компьютер представляет каждую исходную точку данных (сэмпл), тогда как
формат файла относится к организации звукового файла, хранимого на диске. Каждый формат звукового
файла имеет особую структуру, которая определяет, например, информацию, хранящуюся в заголовке файла.
В некоторых случаях формат файла также включает структуры, содержащие некоторую форму метаданных, в
дополнение к собственно «сырым» аудиосэмплам. Остаток этого раздела рассматривает методы Java Sound
API, обеспечивающие разнообразные преобразования форматов файлов и форматов данных.

### Преобразование из одного формата файла в другой

Этот раздел охватывает основы преобразования типов аудиофайлов в Java Sound API. И вновь мы выдвигаем
гипотетическую программу, чья цель на этот раз — прочитать аудиоданные из произвольного входного
файла и записать их в файл типа AIFF. Разумеется, входной файл должен быть типа, который система
способна читать, а выходной файл — типа, который система способна записывать. (В этом примере мы
полагаем, что система способна записывать файлы AIFF.) Программа-пример не выполняет никакого
преобразования формата данных. Если формат данных входного файла не может быть представлен как файл
AIFF, программа просто уведомляет пользователя об этой проблеме. С другой стороны, если входной
звуковой файл уже является файлом AIFF, программа уведомляет пользователя, что преобразовывать его
нет необходимости.

Следующая функция реализует только что описанную логику:

```java
public void ConvertFileToAIFF(String inputPath,
  String outputPath) {
  AudioFileFormat inFileFormat;
  File inFile;
  File outFile;
  try {
    inFile = new File(inputPath);
    outFile = new File(outputPath);
  } catch (NullPointerException ex) {
    System.out.println("Error: one of the
      ConvertFileToAIFF" +" parameters is null!");
    return;
  }
  try {
    // запрашиваем тип файла
    inFileFormat = AudioSystem.getAudioFileFormat(inFile);
    if (inFileFormat.getType() != AudioFileFormat.Type.AIFF)
    {
      // inFile не является AIFF, поэтому попробуем его преобразовать.
      AudioInputStream inFileAIS =
        AudioSystem.getAudioInputStream(inFile);
      inFileAIS.reset(); // перемотка в начало
      if (AudioSystem.isFileTypeSupported(
             AudioFileFormat.Type.AIFF, inFileAIS)) {
         // inFileAIS можно преобразовать в AIFF,
         // поэтому записываем AudioInputStream в
         // выходной файл.
         AudioSystem.write(inFileAIS,
           AudioFileFormat.Type.AIFF, outFile);
         System.out.println("Successfully made AIFF file, "
           + outFile.getPath() + ", from "
           + inFileFormat.getType() + " file, " +
           inFile.getPath() + ".");
         inFileAIS.close();
         return; // Всё готово
       } else
         System.out.println("Warning: AIFF conversion of "
           + inFile.getPath()
           + " is not currently supported by AudioSystem.");
    } else
      System.out.println("Input file " + inFile.getPath() +
          " is AIFF." + " Conversion is unnecessary.");
  } catch (UnsupportedAudioFileException e) {
    System.out.println("Error: " + inFile.getPath()
        + " is not a supported audio file type!");
    return;
  } catch (IOException e) {
    System.out.println("Error: failure attempting to read "
      + inFile.getPath() + "!");
    return;
  }
}
```

Как упоминалось, цель этой примерной функции `ConvertFileToAIFF` — опросить входной файл, чтобы
определить, является ли он звуковым файлом AIFF, и если нет — попытаться преобразовать его в таковой,
создав новую копию, путь к которой указан вторым аргументом. (В качестве упражнения вы можете
попробовать сделать эту функцию более общей, чтобы вместо постоянного преобразования в AIFF функция
преобразовывала в тип файла, указанный новым аргументом функции.) Обратите внимание, что формат
аудиоданных копии — то есть нового файла — повторяет формат аудиоданных исходного входного файла.

Бо́льшая часть этой функции говорит сама за себя и не специфична для Java Sound API. Однако есть
несколько методов Java Sound API, используемых процедурой, которые критически важны для
преобразований типов звуковых файлов. Эти вызовы методов все находятся во втором блоке `try` выше и
включают следующее:

- `AudioSystem.getAudioFileFormat`: используется здесь для определения того, является ли входной файл
  уже типом AIFF. Если да, функция быстро завершается; иначе попытка преобразования продолжается.
- `AudioSystem.isFileTypeSupported`: указывает, может ли система записать файл указанного типа,
  содержащий аудиоданные из указанного `AudioInputStream`. В нашем примере этот метод возвращает
  `true`, если указанный входной аудиофайл может быть преобразован в формат аудиофайла AIFF. Если
  `AudioFileFormat.Type.AIFF` не поддерживается, `ConvertFileToAIFF` выдаёт предупреждение, что
  входной файл не может быть преобразован, а затем завершается.
- `AudioSystem.write`: используется здесь для записи аудиоданных из `AudioInputStream` `inFileAIS` в
  выходной файл `outFile`.

Второй из этих методов, `isFileTypeSupported`, помогает определить заранее, до записи, может ли
конкретный входной звуковой файл быть преобразован в конкретный тип выходного звукового файла. В
следующем разделе мы увидим, как с помощью нескольких изменений этой примерной процедуры
`ConvertFileToAIFF` мы можем преобразовать формат аудиоданных, а также тип звукового файла.

### Преобразование аудио между разными форматами данных

В предыдущем разделе было показано, как использовать Java Sound API для преобразования файла из одного
*формата файла* (то есть одного типа звукового файла) в другой. Этот раздел исследует некоторые
методы, обеспечивающие преобразования *форматов аудиоданных*.

В предыдущем разделе мы читали данные из файла произвольного типа и сохраняли их в файл AIFF. Заметьте,
что хотя мы изменили тип файла, используемого для хранения данных, мы не изменили формат самих
аудиоданных. (Большинство распространённых типов аудиофайлов, включая AIFF, могут содержать
аудиоданные различных форматов.) Поэтому, если исходный файл содержал аудиоданные качества CD
(размер сэмпла 16 бит, частота дискретизации 44,1 кГц и два канала), то и наш выходной файл AIFF
содержал бы их.

Теперь предположим, что мы хотим указать *формат данных* выходного файла, а также тип файла. Например,
возможно, мы сохраняем много длинных файлов для использования в интернете и обеспокоены объёмом
дискового пространства и временем загрузки, требуемыми нашими файлами. Мы могли бы выбрать создание
файлов AIFF меньшего размера, содержащих данные более низкого разрешения, — например, данные с
размером сэмпла 8 бит, частотой дискретизации 8 кГц и единственным каналом.

Не вдаваясь в такие же детали кодирования, как раньше, давайте исследуем некоторые методы,
используемые для преобразования формата данных, и рассмотрим изменения, которые нам нужно было бы
внести в функцию `ConvertFileToAIFF`, чтобы достичь новой цели.

Основной метод для преобразования аудиоданных, опять же, находится в классе `AudioSystem`. Этот метод
— вариант `getAudioInputStream`:

```java
AudioInputStream getAudioInputStream(AudioFormat
    format, AudioInputStream stream)
```

Эта функция возвращает `AudioInputStream`, который является результатом преобразования
`AudioInputStream`, `stream`, с использованием указанного `AudioFormat`, `format`. Если преобразование
не поддерживается `AudioSystem`, эта функция выбрасывает `IllegalArgumentException`.

Чтобы избежать этого, мы можем сначала проверить, может ли система выполнить требуемое преобразование,
вызвав этот метод `AudioSystem`:

```java
boolean isConversionSupported(AudioFormat targetFormat,
    AudioFormat sourceFormat)
```

В этом случае мы передали бы `stream.getFormat()` в качестве второго аргумента.

Чтобы создать конкретный объект `AudioFormat`, мы используем один из двух конструкторов `AudioFormat`,
показанных ниже, либо:

```java
AudioFormat(float sampleRate, int sampleSizeInBits,
    int channels, boolean signed, boolean bigEndian)
```

который создаёт `AudioFormat` с линейной PCM-кодировкой и заданными параметрами, либо:

```java
AudioFormat(AudioFormat.Encoding encoding,
    float sampleRate, int sampleSizeInBits, int channels,
    int frameSize, float frameRate, boolean bigEndian)
```

который также создаёт `AudioFormat`, но позволяет указать кодировку, размер кадра и частоту кадров в
дополнение к остальным параметрам.

Теперь, вооружившись приведёнными выше методами, давайте посмотрим, как мы могли бы расширить нашу
функцию `ConvertFileToAIFF`, чтобы выполнить желаемое преобразование формата аудиоданных в
«низкое разрешение». Сначала мы сконструировали бы объект `AudioFormat`, описывающий желаемый формат
выходных аудиоданных. Следующего оператора было бы достаточно, и его можно было бы вставить ближе к
началу функции:

```java
AudioFormat outDataFormat = new AudioFormat((float) 8000.0,
(int) 8, (int) 1, true, false);
```

Поскольку приведённый выше конструктор `AudioFormat` описывает формат с 8-битными сэмплами, последний
параметр конструктора, указывающий, являются ли сэмплы big- или little-endian, не имеет значения.
(Big- против little-endian важно только если размер сэмпла больше одного байта.)

Следующий пример показывает, как мы использовали бы этот новый `AudioFormat`, чтобы преобразовать
`AudioInputStream`, `inFileAIS`, который мы создали из входного файла:

```java
AudioInputStream lowResAIS;
  if (AudioSystem.isConversionSupported(outDataFormat,
    inFileAIS.getFormat())) {
    lowResAIS = AudioSystem.getAudioInputStream
      (outDataFormat, inFileAIS);
  }
```

Не имело бы большого значения, куда мы вставим этот код, лишь бы он был после конструирования
`inFileAIS`. Без проверки `isConversionSupported` вызов завершился бы неудачей и выбросил бы
`IllegalArgumentException`, если бы запрашиваемое конкретное преобразование было неподдерживаемым. (В
этом случае управление перешло бы к соответствующему блоку `catch` в нашей функции.)

Итак, к этому моменту процесса мы создали бы новый `AudioInputStream`, получившийся в результате
преобразования исходного входного файла (в его форме `AudioInputStream`) в желаемый формат
аудиоданных низкого разрешения, как определено `outDataFormat`.

Последним шагом для получения желаемого звукового файла AIFF низкого разрешения было бы заменить
параметр `AudioInputStream` в вызове `AudioSystem.write` (то есть первый параметр) на наш
преобразованный поток `lowResAIS`, как показано ниже:

```java
AudioSystem.write(lowResAIS, AudioFileFormat.Type.AIFF,
  outFile);
```

Эти несколько изменений нашей более ранней функции дают нечто, преобразующее как аудиоданные, так и
формат файла любого указанного входного файла, при условии, конечно, что система поддерживает
преобразование.

## Какие преобразования доступны

Несколько методов `AudioSystem` проверяют свои параметры, чтобы определить, поддерживает ли система
конкретное преобразование формата данных или операцию записи файла. (Обычно каждый такой метод
сопряжён с другим, который выполняет преобразование данных или записывает файл.) Один из этих методов
запроса, `AudioSystem.isFileTypeSupported`, был использован в нашей примерной функции
`ConvertFileToAIFF`, чтобы определить, способна ли система записать аудиоданные в файл AIFF. Связанный
с ним метод `AudioSystem`, `getAudioFileTypes(AudioInputStream)`, возвращает полный список
поддерживаемых типов файлов для данного потока в виде массива экземпляров `AudioFileFormat.Type`.
Метод:

```java
boolean isConversionSupported(AudioFormat.Encoding encoding,
AudioFormat format)
```

используется для определения того, может ли аудиопоток указанной кодировки быть получен из
аудиопотока, имеющего указанный формат аудио. Аналогично, метод:

```java
boolean isConversionSupported(AudioFormat newFormat,
                              AudioFormat oldFormat)
```

сообщает нам, может ли `AudioInputStream` с указанным форматом аудио, `newFormat`, быть получен через
преобразование `AudioInputStream`, имеющего формат аудио `oldFormat`. (Этот метод вызывался в
фрагменте кода из предыдущего раздела, который создавал аудиопоток низкого разрешения, `lowResAIS`.)

Эти запросы, связанные с форматом, помогают предотвратить ошибки при попытке выполнить преобразования
форматов с помощью Java Sound API.
